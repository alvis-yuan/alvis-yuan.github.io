iAP2 认证与用户数据交互 · 完整指南

本文档基于本站全部 MFi PDF 图解（Link、Authentication、Identification、App Match/Discovery/Launch、EA Protocol、Control Session Messages、Auth 3.0 CP） 以及 Apple AIS 公开结构，将从物理连接到 App 私有数据收发的完整流程串联为一条可读路径。 适用于 MFi 配件固件、iOS Companion App 与系统集成工程师。

一句话总结

Link 建链 → CP 硬件认证 → 能力声明（含 EA Match）→ 可选 Discovery/Launch → EA Session 传用户数据。 前三个阶段失败则无法进入业务态；用户数据永远在 Link 层 Session 之上，而非裸物理管道。

图 2 · iAP2 从连接到用户数据的主时间线（P0–P5）

Phase 0 物理连接

  • 常见传输：Bluetooth（RFCOMM + iAP2 Service）、USB Device Mode（HID 或 Bulk）、UART/Serial。
  • CarPlay / 高带宽场景可能涉及 USB Host Mode、Native Bulk 端点（与 iAP2 EA Multiplex 并行）。
  • Identification 中的 TransportComponent 必须描述真实链路（如 BluetoothTransportComponent 含 MAC）。
门禁：物理层不通 → 无 Detect/SYN，后续全部无法进行。

Phase 1 Link 建链

  1. Detect：周期发送 FF 55 02 00 EE 10，对端回应确认 iAP2 能力。
  2. SYN + LSP：交换 Link Version、Outstanding Packets、Max Packet Length、超时/重传参数、Session 列表（Control / FileTransfer / EA 及各自 Session ID）。
  3. ACK：双方确认 PSN，进入 Normal 态。
  4. 稳定态下：Data+ACK、累计 ACK、EAK 选择性重传、窗口流控。

Link PDU 头 9 字节（0xFF5A + Len + Control + Seq + Ack + SessionID + HC），Payload 路由到对应 Session。详见 iAP2 Protocol 图解。

门禁：SYN 协商失败 / RST → 无 Control Session，不能发 0xAA。

Phase 2 Authentication（硬件信任）

在 Control Session 上完成；MCU 通过 I²C 驱动 Auth 3.0 CP（写 0x20 / 读 0x21，7-bit 地址 0x10）。

步骤iAP2 CSMCP / MCU 动作
10xAA00 RequestAuthenticationCertificate读 0x30 长度 + 0x31… DER 证书（607–609 B）
20xAA01 AuthenticationCertificateiOS 校验 Apple 根 CA 证书链
30xAA02 RequestAuthenticationChallengeResponse写 challenge → 0x21；写 0x10←0x01 + I²C STOP
40xAA03 AuthenticationResponse轮询 0x10；读 0x11/0x12 签名
50xAA05 AuthenticationSucceeded可选 0xAA06 CP 序列号（0x4E）

CP 互斥（79.5.7）

读完 Challenge Response 之前，不得并行读证书或启动新签名；否则 Error Code 0x05 Sequence error。

门禁：0xAA04 AuthenticationFailed → 禁止 Identification；ATS 失败。

Phase 3 Identification（能力声明 + App Match）

  1. iOS 发 0x1D00 StartIdentification
  2. 配件回 0x1D01 IdentificationInformation（TLV 参数集）
  3. iOS 回 0x1D02 Accepted 或 0x1D03 Rejected（带 rejection 标志）

IdentificationInformation 关键字段（与用户数据相关）

match_action / AppMatchTeamID控制 iOS 如何匹配已安装 App（BK42CH45）
MessagesSentByAccessory / MessagesReceivedFromDevice后续允许收发的 CSM ID 白名单（含 0xEA02、0xAD00 等）

均在 IdentificationAccepted 之后；Match 本身发生在 Phase 3 的 0x1D01 内。

能力Control Session 消息说明

Phase 5 EA Session · 用户数据交互

用户业务数据不走 Control Session TLV，而走 EA Session（Link 层 Session Type 2）上的字节流；Control Session 仅协调生命周期。

Control Session 协调（0xEAxx）

消息ID作用
StartExternalAccessoryProtocolSession0xEA00按 Identification 中的 protocol_id 建立 EA Session，分配 session_id
StopExternalAccessoryProtocolSession0xEA01关闭 session_id
StatusExternalAccessoryProtocolSession0xEA03OK / CLOSE；Last In Wins 时通知旧 Session 关闭

数据面（用户 payload）

  • over iAP2：私有协议字节装在 Link PDU 的 EA Session Payload 中；可多个 EA Session 并行（不同 protocol_id / session_id）。
  • 常见格式：首 2 字节 stream handle + 业务帧；或 ExternalAccessorySession Datagram（见 iap2_protocol.pdf）。
  • iOS App 通过 EAAccessory / EASession 的 readData: / writeData: 收发（ExternalAccessory.framework）。

EA over Native（高带宽分支）

USB Host Mode（如 CarPlay）下，除 iAP2 Link 上的 Control/EA Multiplex 外，可存在独立 USB Bulk 端点承载 EA 数据，吞吐更高；Control Session 仍负责协调。详见 EA Protocol · 两种承载方式。

iOS Companion App 侧要点

  1. Info.plist：UISupportedExternalAccessoryProtocols 数组，字符串与配件 SupportedExternalAccessoryProtocol.name 完全一致。
  2. ExternalAccessory.framework：监听 EAAccessoryConnected / Disconnected；为每个 protocol 创建 EASession。
  3. 数据读写：[session readData:] / writeData: — 对应 Phase 5 EA Session 用户 payload。
  4. Background：EA 通信通常需 App 在前台或具备相应 Background Modes；Launch 由配件 0xEA02 或用户手动打开触发。

理解要点（总结）

  1. 两层信任：Authentication = 硬件可信（CP）；Identification = 能力可信（参数与 EA 声明）。
  2. 三层数据：Link 可靠投递 → Session 类型分流 → CSM 协调 / EA payload 承载用户数据。
  3. Match 静态、Discovery 动态：Match 在 0x1D01；Discovery/Launch 在 Accepted 后。
  4. 用户数据不在 CSM：业务 bytes 只走 EA Session（或 Native Bulk）；0xEA00 只是「开管道」。
  5. 白名单机制：MessagesSent/Recv 是 ATS 硬约束；少声明或多发送都会失败。
  6. 版本敏感：App Discovery ID（0xAD vs 0x4E）、Cancel/Update ID 以当前 AIS 为准。
