App Match · Discovery · Launch 总览

依据 App_discovery_lauch_match.pdf（Accessory Interface Specification），三者共同解决 MFi 配件与 iOS App 的 配对声明、安装状态感知、主动拉起问题。它们都发生在 Authentication / Identification 成功之后，属于 External Accessory（EA）生态。

特性PDF 章节发生时机核心问题
App Match BK42CH45 Identification 阶段（0x1D01） 配件声明支持哪些 EA 协议、如何匹配 App、是否绑定 Team ID
App Discovery BK42CH43 IdentificationAccepted 之后（运行时） 配件订阅「哪些匹配 App 已安装 / 卸载」的状态更新
App Launch BK42CH44 IdentificationAccepted 之后（通常需用户操作） 配件请求 iOS 拉起指定 Bundle ID 的 App，随后建立 EA Session

图 1 · Match（声明）→ Discovery（感知）→ Launch（拉起）→ EA Session（通信）

前置条件（Authentication & Identification）

  • 必须先完成 Authentication 与 Identification，收到 0x1D02 IdentificationAccepted。
  • IdentificationInformation 的 MessagesSentByAccessory 须声明配件实际会发送的消息 ID（含 Discovery / Launch 相关 ID）。
  • SupportedExternalAccessoryProtocol 须正确填写——这是 App Match 的载体。

ATS 常见错误

发送了 RequestAppLaunch (0xEA02) 但未在 Identification 的 MessagesSentByAccessory 中声明该 ID，会触发 Section 28.2.2 错误（Apple Developer Forums 已确认）。

App Match 概述

App Match 不是独立的 Control Session 握手，而是嵌入在 IdentificationInformation (0x1D01) 中的 EA 协议匹配策略。PDF 章节 BK42CH45 从 Requirements / Usage 两个维度说明 External Accessory Protocol 与 Team ID 的配置规则。

理解 · Match 解决什么问题

iOS 需要知道：配件支持哪个 EA 协议字符串、当 App 未安装时是否提示用户去 App Store、是否限定特定开发者 Team 的 App。这些在 Identification 阶段一次性声明，后续 Discovery / Launch 都依赖这里的 protocol name 与 match_action。

External Accessory Protocol（Param 10 子结构）

每个 SupportedExternalAccessoryProtocol 条目包含：

子字段说明
id (Uint8)协议 ID，后续 0xEA00 建 EA Session 时使用
name (string)EA 协议名，须与 App Info.plist 中 UISupportedExternalAccessoryProtocols 条目完全一致
match_action (Uint8)App 未安装 / 不匹配时的 iOS 行为（见下表）
native_transport_component_identifier可选；绑定到 Identification 中某 Transport Component
match_actionInfineon / AIS 命名行为理解
0NO_PROMPT不提示、不引导找 App
1DEVICE_MAY_PROMPT_AND_FIND_APP_BUTTON可弹出提示，并显示「查找 App」类引导（Settings + Prompt）
2DEVICE_NO_PROMPT_AND_FIND_APP_BUTTON不弹提示，但仍可显示找 App 入口（Settings Only）

Team ID（App Match Requirements / Usage）

PDF BK42CH45 单独说明 Team ID：通过 AppMatchTeamID（Identification Param 11）限定可匹配的 App 必须来自指定 Apple Developer Team。用于防止第三方 App 冒用同一 EA 协议字符串。

  • Requirements：Team ID 字符串格式须符合 Apple Developer 账号规范。
  • Usage：当配件只应与特定厂商 App 配对时填写；通用配件可省略。

App Match · Requirements & Usage（PDF 摘要）

  • Requirements · External Accessory Protocol：至少声明一个 EA 协议；protocol name 命名规范；match_action 取值合法；id 在同一 Identification 内唯一。
  • Requirements · Team ID：若使用 AppMatchTeamID，须与 MFi 产品计划中的开发者团队一致。
  • Usage · External Accessory Protocol：match_action=1 时常配合 App Launch，在用户连接配件后引导安装/打开 App。
  • Usage · Team ID：OEM 定制场景下绑定自有 App，避免 App Store 中同名协议冲突。

App Discovery 概述

App Discovery 让配件在运行期感知 iOS 上「与已声明 EA 协议匹配的 App」的安装/卸载变化。PDF BK42CH43 包含 Requirements 与 Usage；对应 iAP2 Control Session 消息组 App Discovery（iAP2_Control_Session_Messages.pdf）。

理解 · 与 App Match 的分工

Match 在 Identification 中声明「我支持哪些协议、匹配策略是什么」。Discovery 在连接建立后动态查询「这些协议对应的 App 当前是否已安装」。配件据此决定 UI 提示、是否发送 RequestAppLaunch 等。

App Discovery · Requirements & Usage（PDF 摘要）

  • Requirements：仅在 IdentificationAccepted 后发送；Start/Stop 成对使用；MessagesSent/Recv 中须声明 Discovery 相关 ID。
  • Usage：配件在需要感知 App 是否存在时订阅；不需要时 Stop 以节省 Control Session 带宽；可与 App Launch 决策联动。

App Launch 概述

App Launch 允许 MFi 配件通过 RequestAppLaunch (0xEA02) 请求 iOS 前台启动指定 Bundle ID 的 App。PDF BK42CH44 含 Requirements、Usage、Test Procedures 与 iAP2 Tests。

理解 · Launch 不是 Identification 的一部分

Launch 是连接建立后的 Control Session 命令。iOS 会校验：Bundle ID 对应 App 是否安装、是否声明了相同 EA protocol、配件是否在 Identification 中声明会发送 0xEA02。

EA Session 管理（Launch 之后）

Launch 成功后，配件与 App 通过 External Accessory Session（iAP2 Link 层上的 EA Session）交换业务数据。Control Session 协调消息（External_Accessory_Protocol.pdf）：

消息ID方向说明
StartExternalAccessoryProtocolSession0xEA00双向按 protocol_id 建立 EA Session，分配 session_id
StopExternalAccessoryProtocolSession0xEA01双向关闭指定 session_id
StatusExternalAccessoryProtocolSession0xEA03双向报告 Session OK / CLOSE 状态

EA 数据面在 Link 层 EA Session 上传输；App 侧使用 EASession 的 input/output stream 读写，首 2 字节通常为 session handle（big-endian）。

App Launch · Test Procedures & iAP2 Tests（PDF 摘要）

  • Requirements：仅 IdentificationAccepted 后发送；MessagesSentByAccessory 须含 0xEA02；Bundle ID 与 EA 协议须匹配已安装 App。
  • Usage：典型场景为用户在配件 UI 上按下「打开 App」后发送 RequestAppLaunch。
  • Test Procedures / iAP2 Tests：ATS 脚本验证 Launch 时序；测试步骤要求 RequestAppLaunch 在用户直接操作之后发送（与 Table 85-33 「无用户提示启动」的解读存在社区争议，过认证以 ATS 为准）。

认证注意

不应假设「Lightning 一插入就自动 Launch」一定能通过 MFi Test——即使 AppLaunchMethod 允许较少提示，Test Procedures 仍可能要求用户动作触发。

iOS App 侧配置（与三者配合）

图 5 · 配件 Control Session 与 App External Accessory API 协作

  • UISupportedExternalAccessoryProtocols：数组元素必须与配件 EA Protocol name 一致。
  • UIBackgroundModes → external-accessory：若声明后台 EA，必须同时有 supported protocols（App Store 审核要求）。
  • App 通过 EAAccessoryManager.shared().connectedAccessories 获取配件，用 EASession(accessory:forProtocol:) 打开数据流。
  • 配件不能代替 App 做 Match——Team ID / protocol 匹配由 iOS 在 Identification 解析阶段完成。

理解要点（总结）

  1. Match 是静态契约：在 Identification 中声明 EA 协议名、match_action、Team ID；决定 iOS 如何匹配 App、是否提示用户找 App。
  2. Discovery 是动态感知：订阅后 iOS 推送 App 安装状态变化；适合 UI 显示「已安装 / 未安装」，本身不启动 App。
  3. Launch 是显式请求：配件发 0xEA02 拉起指定 App；须事先在 MessagesSentByAccessory 中声明；MFi Test 通常要求用户操作触发。
  4. 三者顺序：Match（Identification）→ 可选 Discovery（订阅）→ 用户触发 Launch → EA Session 业务数据。
  5. 端到端一致：配件 protocol name ↔ App Info.plist ↔ Bundle ID ↔ Team ID 必须全链路透传，ATS 会逐环校验。
