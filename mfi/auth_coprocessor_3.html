Auth 3.0 Coprocessor 概述

auto_coprocessor_3.pdf（AIS Section 79）描述 MFi 配件内的 Apple Authentication 3.0 Coprocessor（Auth 3.0 CP）： 一颗内嵌 X.509 配件证书与私钥的硬件芯片，配件 MCU 通过 I²C 读取证书、提交 challenge 并取回签名。 iOS 通过 iAP2 Control Session（0xAA 消息）验证该 CP 是否真实存在。

图 1 · Auth 3.0 CP 是 MFi 硬件信任根，MCU 仅通过 I²C 寄存器访问

理解 · 为何需要独立 CP

私钥存储在 CP 内部，配件软件只能请求「签名」与「读证书」，无法导出密钥。iOS 用 Apple 根 CA 验证证书链并对 challenge 验签，从而确认配件为合法 MFi 硬件而非纯软件伪造。

I²C · Setup & Addressing

上电与启动（79.5.1 Setup）

  • I²C 接口在 VCC 上电或 Sleep 模式下 SDA 任意边沿时启动。
  • 上电/唤醒后 I²C 输出缓冲清空；首次传输须等待 t_CMD（SCL 变高后，见 Figure 79-10 时序）。

I²C · 读写流程

79.5.4–79.5.6 规定标准 I²C 事务；NACK 时等待 500 µs 后重试。

写入 Auth 3.0 CP（79.5.5）

  1. 发送 I²C 写地址 0x20，等待 ACK
  2. 发送寄存器地址（起始偏移）
  3. 发送数据字节
  4. 最后一字节后必须发送 STOP，CP 才开始执行操作

STOP 必不可少

若在 STOP 之前发送 Repeated START，行为未定义。写 0x10 启动签名等控制操作尤其依赖完整 STOP。

读取 Auth 3.0 CP（79.5.6）

  1. 写地址 + 寄存器地址（指针）
  2. Repeated START + 读地址 0x21
  3. 读数据字节；缓冲区不足时可重复读地址继续读
  4. 寄存器数据读完后 CP 返回 0xFF 直至选择新寄存器

Authentication Control and Status（0x10）

读：反映最近一次 CP 进程状态（Table 79-18 ~ 79-20）；写：启动进程（Table 79-21）。

写（Procedure Control · bit 0）

写入值动作
0x01启动 Challenge Response Generation（对 0x21 中 challenge 签名）

读（状态位）

位域名称含义
bit 7Error Set0 = Error Code 为 0；1 = 有错误（读 Error Code 后清除）
bit 6–4Procedure Results1 = Challenge Response 成功生成；0 = 无效或未请求
bit 3–0Reserved保留

配件固件应轮询 0x10 直至 Procedure Results = 1 或超时；同时检查 Error Set。

读配件证书流程

对应 iAP2 0xAA00 → 0xAA01：MCU 从 CP 取出 DER X.509 证书发给 iOS。

  1. 读 0x30 得证书长度（607–609 字节）
  2. 从 0x31 连续流式读，或按 128 字节分页读 0x31–0x35
  3. 组装完整 DER，作为 AuthenticationCertificate blob 发送

图 4 · 读证书与 iAP2 AuthenticationCertificate 对应关系

签 Challenge 流程

对应 iAP2 0xAA02 → 0xAA03：iOS 下发 32 字节 challenge，CP 私钥签名后回传。

  1. 将 32 字节 challenge 写入 0x21（Challenge Data 流寄存器）
  2. 写 0x10 = 0x01 启动签名，发送 I²C STOP
  3. 轮询 0x10：Procedure Results = 1；若 Error Set 则读 0x05
  4. 读 0x11 得响应长度，从 0x12 读签名数据
  5. 作为 AuthenticationResponse blob 回复 iOS

图 5 · Challenge 签名状态机（MCU 视角）

Mutual Exclusivity（79.5.7）

若配件支持多设备/多传输/多功能认证，各 challenge 生成过程必须互斥：

  • 在前一操作完成且 Challenge Response Data 已被读取 之前，不得读取 Accessory Certificate、写入新 Challenge Data，或再次写 0x10 启动签名。
  • 违反顺序会触发 Error Code 0x05（Sequence error）或内部错误 0x06。

理解要点（总结）

  1. 硬件信任根：Auth 3.0 CP 存私钥；软件只能读证书、写 challenge、取签名。
  2. I²C 从地址：写 0x20 / 读 0x21（7-bit 0x10）；100/400 kHz，12 kΩ 上拉。
  3. STOP 触发执行：写控制寄存器后必须 I²C STOP，否则行为未定义。
  4. 轮询 0x10：签名异步完成；Procedure Results = 1 表示成功，Error Set 需读 0x05。
  5. 互斥：读完 Response 前不得启动下一认证操作（79.5.7）。
  6. 与 AIS 关系：本文档 = Section 79 硬件手册；iAP2 消息层见 Section 101 与 Authentication 章节。
