External Accessory Protocol 概述

External_Accessory_Protocol.pdf 定义 MFi 配件与 iOS App 之间的自定义业务数据通道。 EA 不是 Control Session 本身，而是 iAP2 Link 层上的一种 Session 类型（或与 Native USB 端点绑定的并行通道）， 由 Control Session 的 0xEAxx 消息协调生命周期。

与 Identification / App Launch 的关系

配件在 Identification 中声明 SupportedExternalAccessoryProtocol（protocol name + id）； 可选通过 RequestAppLaunch 拉起 App； 随后双方通过 StartExternalAccessoryProtocolSession (0xEA00) 建立 EA Session，在数据面传输私有协议。

iAP2 协议栈中的 EA Session

iAP2 Link 层之上分为三类 Session（wiomoc / AIS）：Control、File Transfer、External Accessory。单个 iAP2 Link 可并发 multiplex 多个 EA Session，各自绑定不同 protocol_id。

两种 EA 承载方式（PDF 核心区分）

文档 Test Procedures 明确区分 External Accessory Protocol over iAP2 与 over Native 两类实现路径：

方式数据通道典型场景吞吐
over iAP2 经 iAP2 Link 层 EA Session Multiplex Bluetooth、USB Device Mode (HID)、Serial 受 Link 层 MTU / 窗口限制
over Native USB Host Mode 下独立 Bulk IN/OUT 端点，绕过 iAP2 EA Session Multiplex CarPlay / 高带宽 OEM 配件、USB Host Mode 更高（直连 Bulk 管道）

图 2 · EA over iAP2（Multiplex）vs EA over Native（USB Bulk 直连）

理解 · Native 不是「不用 iAP2」

USB Host Mode 下仍需要 iAP2 完成 Authentication / Identification 及 Control Session 协调；Native 仅指 EA 业务 payload 走独立 Bulk 端点，以降低 Link 层 Multiplex 开销。CarPlay 即典型：iAP2 引导 + NCM/AirPlay 大数据通道。

Requirements · iAP2 EA Session

  • Identification 中必须声明至少一个 SupportedExternalAccessoryProtocol，且 protocol id 与 name 唯一、合法。
  • Link 层协商时必须包含 EA Session 的 Session Identifier；配件固件须能按 Session ID 路由 EA 数据包。
  • Control Session 须实现 0xEA00 / 0xEA01 / 0xEA03（及若使用 Launch 则含 0xEA02）。
  • MessagesSentByAccessory / MessagesReceivedFromAccessory 须覆盖实际使用的 EA 协调消息。
  • 同一 Link 上多个 EA Session 时，每个 session_id 与 protocol_id 映射须保持一致直至 Stop。

Requirements · EA Native Transport (USB Host Mode)

  • 配件 USB 描述符须提供 Bulk IN/OUT 端点，并在 Identification 的 USBHostTransportComponent 中声明接口号等信息。
  • 完成 USB Role Switch，使 iPhone 切换为 USB Host（Control Transfer 请求，见 iAP2 USB 规范）。
  • Native EA 数据格式仍须与 App 侧 External Accessory 框架期望一致（含 session 语义）。
  • Native 与 iAP2 EA Session 并存时，Control Session 仍负责 session 生命周期与状态同步。

EA Session Management

当多个 App 或同一 App 多次请求同一 EA protocol 时，配件须声明并实现会话仲裁策略。PDF Examples 给出两种模式：

策略行为适用理解
First In Wins 首个成功建立 EA Session 的 App 占用 protocol；后续 Start 请求被拒绝或收到 CLOSE 状态 独占型配件（单用户、单控制端）
Last In Wins 新 Session 建立时，配件主动关闭旧 Session（Status CLOSE），新连接取代旧连接 允许 App 重连 / 切换；与 iOS「同 protocol 仅单 Session」框架行为相近

iOS Application Usage

App 侧通过 ExternalAccessory.framework 访问 EA 通道（Apple External Accessory Programming Topics）：

  1. 监听 EAAccessoryDidConnectNotification / DidDisconnect。
  2. 从 EAAccessoryManager.shared().connectedAccessories 获取配件对象。
  3. 检查 accessory.protocolStrings 是否包含目标 protocol（与 Identification 中 name 一致）。
  4. EASession(accessory:forProtocol:) 创建 Session——同一 accessory + 同一 protocol 同时仅允许一个 Session；新建会关闭旧 Session。
  5. 对 inputStream / outputStream 设置 delegate、加入 RunLoop、open 后开始读写。
  6. 关闭时先 close streams，再释放 Session（否则 dealloc 报错）。

扩展 Usage 场景（PDF 摘要）

  • Firmware Updates：App 经 EA Session 向配件发送固件镜像；配件 Bootloader 校验后刷写；须定义分块、ACK、超时与断点续传。
  • Background Operation：App 声明 external-accessory 后台模式且 Info.plist 含 supported protocols；系统允许在后台维持 EA 流。
  • Notification Center / File Provider / Shortcuts：在 EA 连接之上，由 App 扩展触发数据同步或用户快捷操作；配件侧仍通过标准 EA 读写，扩展本身不直接绕过 MFi 认证。
  • Bluetooth Pairing：EA over iAP2 前须完成 BT 配对与 RFCOMM iAP2 服务发现（SDP UUID …decacaff / …decacafe）；未配对设备无法进入 EA 数据面。

EA 数据面格式与读写

  • over iAP2：每个 Link PDU 带 Session ID，路由到 EA Session；payload 为 App 自定义二进制协议。
  • over Native：Bulk 传输同样承载 App 协议；Identification 中 Native Transport 绑定决定走哪条物理管道。
  • 部分协议栈（如 Infineon iAP2 库）在 EA 数据前附加 2 字节 big-endian session handle，与 Control Session 分配的 session_id 对应——实现时须与 App 侧期望一致。
  • App 使用 NSStream 事件驱动：hasBytesAvailable 读、缓冲满时写；配件侧对应非阻塞 send/recv 与 Link 层流控。

实现陷阱

  • 未在 RunLoop 上 schedule stream 导致后台无法收数据。
  • 未 close stream 就释放 EASession，系统报 unable to close session。
  • protocol name 大小写或拼写与 Identification 不一致，EASession 创建失败。

理解要点（总结）

  1. EA = 数据 Session，不是 Control Message：0xEA 消息只负责协调；真实业务字节在 EA Session（或 Native Bulk）上传输。
  2. 两条路：over iAP2 通用；over Native 为 USB Host 高带宽优化；Control / Auth / ID 仍走 iAP2。
  3. protocol 是契约：Identification 声明 → App Info.plist 声明 → Start EA 时 protocol_id 三者必须一致。
  4. Session 仲裁：First In Wins / Last In Wins 是配件策略；iOS 框架层对同 protocol 本身也倾向单 Session。
  5. 全链路：Match（ID）→ Launch（可选）→ Start EA → 数据 → Stop/CLOSE；Background / FW Update 等是同一通道上的用法扩展。
