MFi / iAP2 Control Session

Accessory Authentication 概述

在 iAP2 Link 层建链完成后、Identification 开始之前，Apple 设备（Client）通过 Control Session 对配件（Accessory）执行 挑战-应答认证，验证配件内嵌的 Apple Authentication Coprocessor 是否真实存在且未被伪造。

与 Identification 的分工

Authentication 回答「你是不是合法的 MFi 硬件」；Identification 回答「你是什么设备、支持什么传输与 EA 协议」。必须先认证成功，iOS 才会接受后续的识别信息。

Requirements（规范要求）

依据 Authentication_Identification.pdf 中 Accessory Authentication 章节（BK42CH37），配件侧需满足：

  • 内置经 Apple 供应的 Authentication Coprocessor（如 Auth 2.0 / 3.0 CP），私钥不可导出。
  • 在 Control Session 上正确实现 0xAA 系列消息，并按 TLV 参数格式组包（Start of Message = 0x4040）。
  • 收到挑战后，将随机数写入协处理器、触发签名、读回响应，在超时内返回给 iOS。
  • 认证失败时不得进入 Identification；iOS 可发送 0xAA04 AuthenticationFailed 并断开或重试。
  • 通过 ATS（Accessory Test System）等 MFi 测试工具验证典型流程与失败场景。

Typical Accessory Authentication（Control Session Version 1）

Version 1 为经典五步握手：请求证书 → 返回证书 → 请求挑战 → 返回签名 → 成功/失败通知。

理解 · 为何必须走协处理器

证书中的公钥与协处理器内私钥配对；挑战由 iOS 现场随机生成，只有真实 CP 能在时限内返回有效签名。软件模拟证书无法通过验签，这是 MFi 计件与反盗版的核心机制。

Typical Accessory Authentication（Control Session Version 2）

Version 2 在 Version 1 成功路径上增加一步：配件向 iOS 上报协处理器的序列号，便于 Apple 追溯硬件批次。

Accessory Authentication Failure Due To Invalid Certificate

典型触发条件

  • 证书不是 Apple MFi 根签发的合法链。
  • 证书格式错误、截断或 CP 读取出错。
  • 使用已吊销、克隆或非本配件 CP 的证书。

iOS 在收到 0xAA01 后立即校验；失败则发送 0xAA04，不会下发 challenge。ATS 会报证书相关错误。

图 6 · 无效证书失败路径

Accessory Authentication Failure Due To Invalid Response

典型触发条件

  • 签名与 challenge 不匹配（软件伪造响应、错误 CP、数据损坏）。
  • 协处理器超时未完成签名（未轮询 status、I²C 错误）。
  • 响应长度异常或算法/版本与 iOS 期望不符。

证书已通过但 0xAA03 验签失败时，iOS 同样回复 0xAA04。

理解要点（总结）

  1. 安全模型：基于 Apple 签发的 X.509 证书 + 私钥不可导出的 CP；等价于「硬件持有的私钥」对 iOS 随机挑战签名。
  2. 时序位置：Link 层会话协商完成后立即执行；失败则整个 iAP2 连接无法进入业务态。
  3. Version 1 vs 2：v2 额外交换 AccessoryAuthenticationSerialNumber，便于制造与合规追溯。
  4. 实现要点：I²C 访问需处理 CP 较慢的响应；必须在规范超时内完成轮询，否则落入 Invalid Response。
  5. 与 Wi‑Fi/BT/TLS 对比：这是 MFi 特有的硬件信任根，不能仅用软件证书或 TLS 客户端证书替代。
