iAP2 Protocol 概述

iap2_protocol.pdf 是 Accessory Interface Specification 中关于 iAP2（iPod Accessory Protocol 2） 的核心规范： 定义 Link 层可靠传输（类 TCP 滑动窗口）、三种 Session Multiplex，以及 Control / File Transfer / EA 各 Session 上的消息与 Datagram 格式。 上层 Authentication、Identification、App Launch 等 Control Session 消息均承载于本规范之上。

图 1 · iAP2 协议栈：Link 层 + 三种 Session（PDF 顶层结构）

理解 · Link 层在 TCP/IP 模型中的位置

wiomoc 等实现指出：iAP2 Link 层职责接近传输层——在不可靠物理链路上提供按消息计数的 seq/ack、重传与窗口流控；Session 层再在其上 Multiplex 不同业务。Authentication 之前必须先完成 Link 初始化（Detect → SYN 协商）。

iAP2 Connection

Connection 章节（BK42CH74SLK1065）描述一次 iAP2 逻辑连接的属性：

属性说明
Type连接类型标识（与传输介质、角色相关）
VersioniAP2 Link / Session 协议版本，LSP 中 Link Version 字段协商
Attributes连接级元数据（由 AIS 定义的具体属性集）

一条 iAP2 Connection 建立后，Control Session 上跑 Authentication / Identification；File Transfer / EA Session 在 Link 稳定后并行传输数据。

iAP2 Link · Packet Structure

每个 Link PDU 固定 9 字节头（含 Header Checksum），可选 Payload + Payload Checksum。Start 魔数 0xFF 0x5A（16-bit 大端 0xFF5A）。

Offset字段长度说明
0Start of Packet MSB10xFF
1Start of Packet LSB10x5A
2–3Packet Length2整包长度（含头与校验），大端；无 Payload 时为 9
4Control Byte1SYN / ACK / EAK / RST / SLP 标志位
5Packet Sequence Number (PSN)1发送方序号 0–255 循环
6Packet Acknowledgement Number1累计确认的对端 PSN
7Session Identifier1路由到 Control / FileTransfer / EA Session
8Header Checksum1使 header 9 字节之和 ≡ 0 (mod 256)
9…N-1Payload Data变长Session 数据或 LSP / EAK 列表
NPayload Checksum1使 Payload+校验 之和 ≡ 0 (mod 256)

Checksum Calculation

Header 与 Payload 各自独立校验：所有字节（含 checksum 自身）8 位无符号相加，结果低 8 位为 0。实现上常用「负累加和」生成校验字节（见开源实现 gen_checksum）。

Control Byte 标志位

Control Byte 可组合；数据包通常带 ACK (0x40)，SYN 协商带 0x80。

位标志值含义
SYN0x80Link Synchronization Payload；用于参数协商
ACK0x40确认字段有效；纯 ACK 或数据+ACK
EAK0x20Extended ACK；Payload 列出缺失 PSN，请求重传
RST0x10Reset；对端应终止 Link（Device Reset of Transport）
SLP（Sleep）进入低功耗 Sleep 状态（PDF Operation · Sleep）

Link Synchronization Payload (LSP)

SYN 包 Payload 携带 LSP，双方交换 Link 参数与 Session 列表（Link Synchronization Payload · BK42CH74SLK1081）。

字段说明
Link VersionLink 协议版本（如 0x01）
Maximum Number of Outstanding Packets未确认窗口大小（滑动窗口上限）
Maximum Received Packet Length单包最大长度（含头；常见 65535）
Retransmission Timeout重传超时（ms）
Cumulative Acknowledgement Timeout累计 ACK 延迟发送阈值（ms）
Maximum Number of Retransmissions单包最大重传次数，超限则 RST/断链
Maximum Cumulative Acknowledgements收到多少包后必须回 ACK
ZeroACK / ZeroRetransmit可选 Link 配置（低延迟 / 无重传场景）
Session 列表每项 3 字节：Session ID + Session Type + Version
Session Type值用途
Control0Control Session · CSM 消息
File Transfer1File Transfer Session · Datagram
External Accessory2EA Session · App 数据

理解 · Session ID 是协商出来的

Session ID 不是全局固定值；由 LSP 中双方声明的列表决定（开源实现常见 Control=10、EA=11，但以 SYN 协商结果为准）。Link 层根据 offset 7 的 Session Identifier 将 Payload 投递到对应 Session 状态机。

理解要点（总结）

  1. 两层分离：Link 层管可靠投递与 Multiplex；Session 层管消息语义（CSM / Datagram / 原始字节）。
  2. 先 Link 后业务：Detect → SYN(LSP) → ACK 成功后，才能在 Control Session 发 Authentication。
  3. PSN 不是字节序号：seq/ack 按包计数，窗口单位是 Link PDU 个数，与 TCP 字节序不同。
  4. EAK 是选择性重传：乱序时发 EAK 列出缺失 PSN，比盲目超时重传更高效（PDF Examples）。
  5. Session Type 决定解析器：同一 Link 头上 Session ID 不同，Payload 分别进入 Control CSM 解析、File Datagram 状态机或 EA 字节流。
  6. 与其他 AIS 文档的关系：Authentication/Identification/App Launch/EA 均是本规范 Control/EA Session 上的具体 Message ID 与用法。
