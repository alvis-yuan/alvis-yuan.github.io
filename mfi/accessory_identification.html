Accessory Identification 概述

Authentication 成功后，iOS 通过 Identification 流程收集配件的身份、能力与支持的服务。 配件在 IdentificationInformation (0x1D01) 中一次性声明名称、序列号、传输组件、 EA 协议列表、供电能力等；iOS 校验后回复 Accepted 或 Rejected。

与 Authentication 的关系

Authentication 证明「硬件可信」；Identification 声明「软件能力与元数据」。Rejected 时 iOS 会在消息中标记哪些参数字段不合法，配件可修正后重试（PDF 示例：最多两次尝试）。

Requirements（规范要求）

  • 仅在 AuthenticationSucceeded 之后响应 Identification。
  • IdentificationInformation 必须包含规范要求的字符串字段（Name、ModelIdentifier、Manufacturer、SerialNumber 等，以零结尾 UTF-8 字符串传递）。
  • 声明的 Transport Component 必须与配件实际提供的物理/逻辑传输一致（Bluetooth MAC、USB 接口号等）。
  • MessagesSentByAccessory / MessagesReceivedFromAccessory 必须准确列出配件在 Identification 之后可能收发的 Control Session 消息 ID 列表（ATS 会校验，例如 RequestAppLaunch 必须出现在 Sent 列表中若配件会使用该功能）。
  • SupportedExternalAccessoryProtocol 中每个 EA 协议需分配唯一 ID，并与后续 StartExternalAccessoryProtocolSession 一致。
  • CarPlay / Wireless CarPlay 配件需包含对应的 WirelessCarPlayTransportComponent 等扩展组件（PDF：iAP2 Transport Components、Product Plan UID）。

重试场景（PDF Examples）

Successful Accessory Identification With Two Tries

首次 IdentificationInformation 存在参数错误 → iOS 回复 0x1D03 IdentificationRejected（各字段带 rejection 标志）→ 配件修正后再次发送 0x1D01 → 0x1D02 Accepted。

Unsuccessful Accessory Identification After Two Tries

两次均被 Reject 后，iOS 终止 Identification，iAP2 连接无法完成建立。

理解 · Rejected 参数位图

IdentificationRejected (0x1D03) 镜像 IdentificationInformation 的参数 ID；某字段出现 rejection 标志即表示 iOS 要求配件修正该字段。常见错误：SerialNumber 格式、EA 协议名与 Bundle ID 不匹配、MessagesSent 未包含实际会发送的消息 ID。

Accessory Identification With Information Update

Identification 完成后，若配件能力变化（例如蓝牙 MAC 变更、语言切换），可发送 0x1D05 IdentificationInformationUpdate 推送增量更新，无需完整重走 StartIdentification。

配件也可发送 0x1D04 CancelIdentification 主动取消进行中的 Identification。

iAP2 Transport Components（PDF 专节）

每种物理传输在 Identification 中声明一个 Transport Component，包含 id、name、supports_iap2_connection 及传输特有字段：

组件Param ID特有字段
SerialTransportComponent14串口传输（57600 bps 等）
USBDeviceTransportComponent15audio_sample_rate 等
USBHostTransportComponent16car_play_interface_number（USB Host CarPlay）
BluetoothTransportComponent17bluetooth_transport_mac（6 字节 MAC）
WirelessCarPlayTransportComponent24supports_car_play

理解 · 为何必须声明 Transport

iOS 用 Transport Component 将逻辑 EA 协议绑定到实际链路（例如 Wireless CarPlay 先走 BT iAP2，再切 Wi‑Fi）。MAC / 接口号不一致会导致 ATS 失败或运行时无法建 EA Session。

控制会话消息一览（0x1D）

消息ID方向说明
StartIdentification0x1D00iOS → 配件启动识别；配件应回复 IdentificationInformation
IdentificationInformation0x1D01配件 → iOS完整 TLV 参数集
IdentificationAccepted0x1D02iOS → 配件识别成功，iAP2 连接建立完成
IdentificationRejected0x1D03iOS → 配件带 rejection 标志，指示需修正的字段
CancelIdentification0x1D04配件 → iOS取消进行中的 Identification
IdentificationInformationUpdate0x1D05配件 → iOSAccepted 后的增量更新

Manufacturing Information · ATS · Sent/Received Messages

PDF 在 Identification 章节还包含以下与量产/测试相关的要求（BK42CH38）：

  • Manufacturing Information：制造阶段写入的元数据需与 Identification 中 SerialNumber、Model 等一致。
  • Sent/Received iAP2 Control Session Messages（Section 28.2.2）：配件声明的消息 ID 集合必须覆盖其实际行为；ATS 会检测「发送了未声明的消息」或「声明了从未发送的消息」。
  • Test Procedures：使用 Accessory Test System 模拟典型/失败/重试/Update 场景。

ATS 常见 Identification 错误

  • 使用 RequestAppLaunch 但未在 MessagesSentByAccessory 中声明 0xEA02。
  • EA 协议 name 与 iOS App 的 protocolString 不一致。
  • BluetoothTransportComponent 的 MAC 与当前 RFCOMM 链路不符。

理解要点（总结）

  1. 目的：让 iOS 知道配件是谁、能做什么、走哪条传输、提供哪些 EA 协议——从而启用 HID、媒体库、CarPlay、App 通信等 Control Session 功能。
  2. Rejected 即反馈：不是简单布尔失败，而是逐字段告知修正方向，支持两次尝试的典型量产流程。
  3. Messages Sent/Recv 是契约：可视为配件对 iOS 的「Control Session API 声明」；违反会导致 ATS 或运行时拒收。
  4. Update 而非重认证：运行期能力变化用 0x1D05，无需重新走 Authentication（除非 Link 重建）。
  5. CarPlay 扩展：Wireless CarPlay 在 Identification 中声明 Wi‑Fi/CarPlay 组件后，才进入 Wi‑Fi 凭证交换（0x5702/0x5703 等，属后续 Control Session 消息）。
