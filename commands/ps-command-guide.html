📋 Chapter 01

Linux ps 命令 — 概述与简介

ps（Process Status）是 Linux 系统中用于报告当前进程快照的核心命令，隶属于 procps-ng 工具集。它能显示系统中活跃进程的详细信息，是系统管理、性能分析和故障排查的第一工具。

📖 官方手册页：man7.org/linux/man-pages/man1/ps.1.html

基本语法

SYNOPSIS 
ps [option ...]

不带任何选项运行 ps 时，默认只显示与当前终端关联、且与当前用户相同有效用户 ID（EUID）的进程：

默认输出示例 
$ ps
  PID TTY          TIME CMD
11845 pts/1    00:00:00 bash
17636 pts/1    00:00:00 ps

默认输出字段含义

字段全称含义
PIDProcess ID进程唯一标识符
TTYTerminal进程所关联的终端设备
TIMECPU Time进程累计消耗的 CPU 时间（[DD-]HH:MM:SS）
CMDCommand可执行文件名称（不含参数）

ps vs top 的定位区别

图1：ps（快照）与 top（实时）的定位对比

🔬 第一性原理
进程是操作系统的基本执行单元，管理员需要一种可靠手段来"观察"这些单元的状态。ps 的本质是：在某一时刻读取内核维护的进程元数据，并以人类可读的格式呈现。它不修改任何进程状态，只做只读查询。这种"快照"语义使其天然适合可重复、可审计的诊断场景。
⚙️ Chapter 02

工作原理 — /proc 虚拟文件系统

ps 的实现完全依赖 Linux 内核提供的 /proc 虚拟文件系统（procfs）。它不需要 setuid 或任何特殊权限即可运行，通过读取 /proc/[PID]/ 目录下的文件获取所有进程信息。

🔗 参考来源：unix.stackexchange.com — How does the ps command work?

数据流路径

图2：ps 命令数据流 — 从内核 task_struct 到终端输出

/proc/[PID] 目录结构

每个运行中的进程在 /proc 下都有一个以 PID 命名的目录，包含该进程的所有元数据：

文件内容ps 使用场景
/proc/[PID]/stat进程状态、CPU 时间、优先级、内存等核心字段PID, STAT, TIME, VSZ, RSS, PRI, NI
/proc/[PID]/status人类可读的状态信息，UID/GID、线程数等USER, GROUP, NLWP
/proc/[PID]/cmdline完整命令行参数（NUL 分隔）CMD/COMMAND 列
/proc/[PID]/exe指向可执行文件的符号链接可执行路径
/proc/[PID]/fd/进程打开的文件描述符辅助诊断
/proc/uptime系统运行时间（计算进程启动时间）START 时间
/proc/meminfo系统内存信息%MEM 计算

用 strace 验证数据来源

strace 追踪 ps 的系统调用（来源：unix.stackexchange.com） 
$ strace -e open ps
...（省略库加载）
open("/proc/self/stat", O_RDONLY)      = 3
open("/proc/uptime", O_RDONLY)         = 3
open("/proc/sys/kernel/pid_max", O_RDONLY) = 4
open("/proc/meminfo", O_RDONLY)        = 4
open("/proc/1/stat", O_RDONLY)         = 6
open("/proc/1/status", O_RDONLY)       = 6
open("/proc/1/cmdline", O_RDONLY)      = 6
open("/proc/2/stat", O_RDONLY)         = 6
... 依次读取每个 PID 目录
ℹ️
procfs 的特殊性：这些文件并非真实存在于磁盘上，而是内核在每次 read() 系统调用时动态生成的内容。因此文件大小通常显示为 0，但可以正常读取。这也解释了为何 ps 不需要特殊权限——它只是普通的文件读取操作。
🔬 第一性原理
操作系统的本质是资源管理器，必须向外提供观测接口。Linux 内核选择"一切皆文件"作为哲学，将内核数据结构（如 task_struct）映射为 procfs 中的虚拟文件。ps 的实现因此无需特权、无需内核补丁——只需读文件。这是 Unix 设计哲学的完美体现：简单接口，强大组合。

三种选项风格

这是 Linux ps 最容易让人困惑的地方。由于历史原因，ps 同时支持三种不同风格的选项，不同风格可以自由混用，但可能产生冲突。

📖 来源：man7.org ps(1) 官方手册页

图3：ps 命令的三种选项风格

典型示例对比

风格命令示例效果说明
Unixps -ef显示所有进程，全格式（含 UID, PID, PPID, STIME, CMD）
Unixps -ely显示所有进程，长格式（用 RSS 替代 ADDR）
BSDps aux所有用户，用户格式（含 %CPU, %MEM, VSZ, RSS, STAT）
BSDps axjf所有进程，树形显示（forest）
GNUps --forest -e所有进程，树形（等价于 BSD 的 f）
混合ps -eo pid,comm --sort=-%cpu自定义字段，按 CPU 降序（Unix + GNU 混合）
⚠️
注意区分：ps aux（BSD，无横线）与 ps -aux（Unix，有横线）行为不同！BSD 的 aux 中 a 代表"所有终端进程"，而 Unix 的 -u 后接用户名。混用可能产生意外输出，建议在脚本中明确使用一种风格。
🔬 第一性原理
ps 的三种风格并存，根本原因是 Unix 操作系统的历史分裂（System V vs BSD）。Linux 为了兼容性，将三者都实现了。从第一性原理看：任何工具的接口设计都是其历史约束的产物。理解这个分裂，才能避免在脚本中因风格混用而踩坑。在正式环境中应优先使用 POSIX/Unix 风格（带 -），保证跨平台兼容性。
🎯 Chapter 04

进程选择选项

进程选择决定"显示哪些进程"，是 ps 最重要的功能之一。选择条件叠加生效（任意条件满足即显示），默认选择会在指定选项时被丢弃。

📖 来源：man7.org ps(1) SIMPLE PROCESS SELECTION & PROCESS SELECTION BY LIST

简单选择开关

选项风格含义
-A / -eUnix选择所有进程（等价）
aBSD解除"仅自身"限制，列出所有有终端的进程
xBSD解除"必须有 TTY"限制，包含无终端进程（如守护进程）
rBSD仅显示正在运行（R 状态）的进程
-aUnix所有进程，但排除会话领导者及无终端进程
-dUnix所有进程，但排除会话领导者
-N / --deselectUnix/GNU取反选择（排除满足条件的进程）
TBSD仅当前终端关联的进程

按列表精确选择

以下选项接受逗号或空格分隔的列表，可多次使用（如 ps -p "1 2" -p 3,4）：

选项选择依据示例
-p / p / --pid进程 ID（PID）ps -p 1234,5678
-q / q / --quick-pidPID（快速模式，不排序不过滤）ps -q 1234
--ppid父进程 ID（PPID），显示某进程的所有子进程ps --ppid 1
-C命令名（executable name，非命令行）ps -C nginx
-u / U / --user有效用户名或 EUIDps -u www-data
-U / --User真实用户名或 RUIDps -U root
-g / --group有效组名或 EGIDps -g staff
-G / --Group真实组名或 RGIDps -G 1000
-t / t / --tty终端设备名ps -t pts/0
-s / --sid会话 IDps -s 1234

实用选择组合

# 查看所有进程（两种等价写法）
ps -e            # Unix 风格
ps ax            # BSD 风格

# 查看特定 PID 的进程
ps -p 1234
ps -q 1234       # 快速模式，仅读取该 PID 信息

# 查看某命令名的所有进程
ps -C nginx

# 查看某用户的所有进程
ps -u alice

# 查看某进程的所有子进程
ps --ppid 1234

# root 账户的所有进程，包含无终端的
ps -U root -u root u

# 取反：显示非 root 的进程
ps -U root --deselect
🔬 第一性原理
进程选择本质上是集合过滤——从系统所有进程的全集中，按某个属性维度（PID、用户、终端、命令名）进行筛选。选择条件默认叠加（OR 语义），这与数据库中 WHERE IN 子句类似。理解这个数学逻辑，就能灵活组合选项而不出错。

输出格式控制

ps 提供多种预设格式和完全自定义的输出控制，能精确指定要显示哪些列、列头如何命名、列宽如何控制。

📖 来源：man7.org ps(1) OUTPUT FORMAT CONTROL

预设格式选项

选项风格等价 -o 字段适用场景
-fUnixuid,pid,ppid,c,stime,tty,time,cmd全格式，含 PPID 和启动时间
-FUnixuid,pid,ppid,c,sz,rss,psr,stime,tty,time,cmd扩展全格式，额外含内存和 CPU 信息
-lUnixf,uid,pid,ppid,c,pri,ni,addr,sz,wchan,tty,time,ucmd长格式，含优先级、等待通道
uBSDuser,pid,pcpu,pmem,vsz,rss,tty,stat,start_time,bsdtime,args用户友好格式，含资源使用百分比
lBSDf,uid,pid,ppid,pri,ni,vsz,rss,wchan,stat,tty,time,cmdBSD 长格式
jBSDppid,pid,pgid,sid,tty,tpgid,stat,uid,time,cmd作业控制格式

sBSDuid,pid,pending,blocked,ignored,caught,stat,tty,bsdtime,cmd信号格式

自定义格式：-o / --format

使用 -o 可以精确指定输出列，是生产环境脚本的首选方式：

# 只显示 PID 和命令名
ps -eo pid,comm

# 显示资源使用情况
ps -eo pid,user,%cpu,%mem,vsz,rss,stat,comm

# 自定义列头名称
ps -o pid,ruser=RealUser -o comm=Command

# 不打印列头（适合脚本解析）
ps -o pid= -o comm=

# 控制列宽（WCHAN 列加宽）
ps -o pid,wchan=WIDE-WCHAN-COLUMN -o comm

# 显示进程完整启动时间
ps -eo pid,lstart,comm

# 获取 syslogd 的 PID（脚本友好）
ps -C syslogd -o pid=

# 获取 PID 42 的进程名
ps -q 42 -o comm=

树形（Forest）格式

# 显示进程树（Unix 风格）
ps -ejH
ps -eH

# BSD 风格 forest
ps axjf
ps axf

# GNU 长选项
ps -e --forest

# 示例输出
  PID TTY      STAT   TIME COMMAND
    1 ?        Ss     0:03 /sbin/init
  890 ?        Ss     0:00  \_ sshd
 1234 ?        S      0:00      \_ sshd: alice
 1235 pts/0    Ss     0:00          \_ -bash
 2001 pts/0    R+     0:00              \_ ps axf
🔬 第一性原理
输出格式是"信息呈现"的问题，与"信息获取"（进程选择）相互独立。-o 的存在体现了关注点分离原则：选择哪些进程、展示哪些字段是两个正交维度，可以任意组合。在脚本中使用 -o field=（空列头）返回纯数据，让 ps 作为数据源而非人类阅读工具，是工程上最简洁的选择。
🚦 Chapter 06

进程状态码（STAT / S）

进程状态是理解系统健康的关键。ps 的 STAT 列（多字符）或 S 列（单字符）显示进程当前所处的状态，来源于 Linux 内核调度器的内部状态机。

📖 来源：man7.org ps(1) PROCESS STATE CODES

基本状态码（第一个字符）

图4：Linux 进程状态转换关系图

状态码名称含义常见原因
R Running 正在 CPU 上运行，或在运行队列（就绪） 正常计算密集型任务
S Sleeping（可中断） 等待某事件完成（如用户输入、网络） 绝大多数守护进程的正常状态
D Disk Sleep（不可中断） 等待 IO（通常是磁盘 IO），不响应信号 大量 D 状态 → 磁盘 IO 瓶颈/NFS 挂起
T Stopped 被作业控制信号停止（SIGSTOP/SIGTSTP）或正被调试 kill -STOP PID，或 gdb 调试中
Z Zombie（僵尸） 已退出，但父进程未调用 wait() 回收 父进程 Bug，大量僵尸进程消耗 PID 资源
X Dead 已死亡（正常情况下不可见） 理论状态，实际几乎不出现

BSD 附加状态修饰符（第二个字符起）

字符含义
<高优先级（nice 值较低，不谦让其他进程）
N低优先级（nice 值较高，更谦让）
L页面被锁定在内存（实时进程或自定义 IO）
s会话领导者（Session Leader）
l多线程（使用 CLONE_THREAD，如 NPTL 线程）
+位于前台进程组
I空闲内核线程（Kernel idle thread）
常见 STAT 组合示例 
STAT    含义
Ss      可中断睡眠 + 会话领导者（典型：sshd）
Ssl     可中断睡眠 + 会话领导者 + 多线程（典型：数据库）
R+      运行中 + 前台（当前交互进程）
D       不可中断 IO 等待（磁盘问题告警信号）
Z       僵尸（应尽快排查父进程）
T       已停止（可能被 Ctrl+Z 挂起）
🔬 第一性原理
进程状态是内核调度器的内部状态机的用户空间投影。从硬件视角，CPU 同一时刻只能执行一个进程（单核），调度器需要跟踪哪些进程"可运行"（R）、哪些在"等待资源"（S/D/T）。D 状态不可被 kill，因为进程在等待内核临界区完成，这是操作系统设计的必然——你不能强制中断正在写磁盘的操作，否则会数据损坏。理解状态码，就是理解操作系统如何在资源不足时调度竞争。

常用命令速查表

这里汇总了日常运维和开发中最常用的 ps 命令组合，按场景分类整理。

🔍 基础查看

# 查看当前终端进程
ps

# 查看所有进程（最常用，两种风格）
ps aux                        # BSD 风格，含资源使用率
ps -ef                        # Unix 风格，含 PPID 和启动时间

# 全格式（更多字段）
ps -eF                        # 含 SZ, RSS, PSR 等
ps -ely                       # 长格式，RSS 替代 ADDR

🎯 精确定位进程

# 按进程名查找
ps -C nginx
ps -C python3

# 按 PID 查找
ps -p 1234
ps -p 1,2,3                   # 多个 PID

# 按用户查找
ps -u www-data
ps -U root -u root u          # root 真实+有效 UID，用户格式

# 查看某进程的子进程
ps --ppid 1234

# 配合 grep（注意过滤自身）
ps aux | grep nginx | grep -v grep
ps aux | grep [n]ginx         # 技巧：用字符类避免匹配 grep 自身

📊 资源监控

# 按 CPU 降序排列（找高 CPU 进程）
ps aux --sort=-%cpu | head -15
ps -eo pid,user,%cpu,comm --sort=-%cpu | head -15

# 按内存降序排列（找内存占用大的进程）
ps aux --sort=-%mem | head -15
ps -eo pid,user,%mem,rss,vsz,comm --sort=-%mem | head -15

# 查看进程内存详情
ps -eo pid,user,rss,vsz,%mem,comm

# 查看运行时间最长的进程
ps -eo pid,user,etime,comm --sort=-etime | head -10

🌳 进程树

# 以树形显示所有进程
ps axjf
ps -e --forest
ps -ejH

# 从指定 PID 开始的进程树
ps --ppid 1 f

🔧 脚本专用（纯数据输出）

# 获取 nginx 的 PID（无列头）
ps -C nginx -o pid=

# 获取 PID 1234 的命令名
ps -q 1234 -o comm=

# 检查进程是否存在
ps -p $PID > /dev/null 2>&1 && echo "running" || echo "not found"

# 输出所有进程 PID 列表
ps -eo pid=

# 自定义分隔符输出（GNU）
ps -eo pid,comm --delimiter=','
🔬 第一性原理
命令行工具的价值在于可组合性。ps 通过 -o field= 输出纯数据，再通过管道（|）与 grep、awk、sort、head 组合，实现了复杂的进程分析。这是 Unix 哲学"做好一件事，然后配合其他工具"的体现。脚本中应避免依赖 ps 的格式化输出，而是使用 -o 的精确字段，保证跨系统一致性。
🔀 Chapter 09

排序与过滤

ps 支持多级排序，并可通过组合选项实现精确过滤，是进程分析的强大工具。

📖 来源：man7.org ps(1) OUTPUT MODIFIERS

--sort 排序语法

排序语法：[+|-]key[,[+|-]key[,...]] 
# + 升序（默认），- 降序

# 单字段排序
ps -eo pid,user,%cpu,comm --sort=-%cpu   # CPU 降序
ps -eo pid,user,rss,comm --sort=-rss    # 内存降序
ps -eo pid,user,etime,comm --sort=pid   # PID 升序

# 多字段排序
ps jax --sort=uid,-ppid,+pid
ps -eo user,pid,%cpu,comm --sort=user,-%cpu  # 先按用户升序，再按 CPU 降序

# 等价写法（k 与 --sort 相同）
ps -eo pid,comm k -%cpu

实用过滤模式

# 只显示 R 状态（正在运行）的进程
ps r
ps aux | awk '$8 ~ /^R/'

# 找出所有僵尸进程
ps aux | awk '$8 == "Z"'
ps -eo stat,pid,ppid,comm | grep '^Z'

# 找出不可中断睡眠（D）进程（IO 问题预警）
ps aux | awk '$8 ~ /^D/'

# 找高 CPU 占用（超过 10%）
ps aux | awk 'NR==1 || $3>10'

# 找高内存占用（超过 5%）
ps aux | awk 'NR==1 || $4>5'

# 找运行超过 1 小时的进程
ps -eo pid,etime,comm | awk -F: '$2 > 60'

# 按名称过滤并显示详细信息
ps -eo pid,ppid,stat,user,%cpu,%mem,vsz,rss,lstart,comm | grep nginx

场景推荐排序字段命令
查找 CPU 消耗大的进程-%cpups -eo pid,user,%cpu,comm --sort=-%cpu | head -10
查找内存占用大的进程-rss 或 -%memps -eo pid,user,rss,comm --sort=-rss | head -10

🔬 第一性原理
排序和过滤是信息降维的手段。当系统有数百个进程时，人脑无法直接处理全量数据，必须通过排序（突出异常值）和过滤（缩小问题范围）来快速定位问题。--sort 在 ps 内部完成排序（效率更高），而 | awk/grep 则是外部过滤。在进程数量不多时两者等价；进程极多时优先用 ps 内置排序。

应用场景实战

以下是生产环境中最常见的 ps 使用场景，每个场景均附有完整命令和解释。

场景一：系统性能排查

CPU 飙高排查 
# 步骤1：找 CPU 使用率最高的前10个进程
ps -eo pid,user,%cpu,stat,lstart,comm --sort=-%cpu | head -11

# 步骤2：若是多线程程序，找具体是哪个线程
ps -p <PID> -L -o tid,lwp,%cpu,state,wchan:20

# 步骤3：持续监控该进程（配合 watch）
watch -n 1 'ps -p <PID> -o pid,%cpu,%mem,stat,wchan:20,comm'
内存泄漏排查 
# 查找内存占用最大的进程（按 RSS 实际物理内存）
ps -eo pid,user,rss,vsz,%mem,comm --sort=-rss | head -15

# 记录快照用于对比（诊断内存泄漏）
ps aux --sort=-%mem > /tmp/mem_before.txt
# ... 执行某些操作 ...
ps aux --sort=-%mem > /tmp/mem_after.txt
diff /tmp/mem_before.txt /tmp/mem_after.txt

场景二：服务状态检查

# 检查 nginx 是否在运行，获取 PID 和 worker 数
ps -C nginx -o pid,ppid,stat,user,comm

# 检查所有 Java 应用（按用户分组）
ps -u app -o pid,ppid,%cpu,%mem,lstart,comm | sort -k5

# 验证服务进程及其子进程树
ps --ppid $(ps -C nginx -o pid= | head -1) -o pid,stat,comm

# 脚本中检查服务是否存活
if ps -C myservice -o pid= | grep -q .; then
    echo "Service is running"
else
    echo "Service is DOWN - alerting..."
fi

场景三：僵尸进程处理

# 找出所有僵尸进程及其父进程
ps -eo stat,pid,ppid,user,comm | grep '^Z'

# 查找僵尸进程的父进程（准备 kill 父进程）
ps -eo stat,pid,ppid,comm | awk '$1 ~ /^Z/ {print "Zombie PID:", $2, "Parent PID:", $3, "Cmd:", $4}'

# 向父进程发 SIGCHLD 信号，让其回收僵尸
kill -SIGCHLD <PPID>

# 若父进程也失控，则 kill 父进程（init 会接管子进程并回收）
kill -SIGTERM <PPID>

场景四：安全审计

# 查看所有以 root 运行的网络服务
ps -U root -u root -o pid,stat,user,comm

# 查找可疑进程（无终端、高权限）
ps -eo pid,user,stat,tty,comm | awk '$4 == "?" && $2 == "root"'

# 显示 SELinux 上下文（安全合规）
ps -eM | grep -v "^system_u"

# 查看进程完整命令行（检测混淆进程名）
ps -eo pid,user,comm,args | awk '$3 != $4'

场景五：容器/DevOps 环境

# 在容器内查看所有进程（含 init）
ps -ef

# 查看命名空间中的进程（排查 PID 命名空间）
ps -eo pid,timens,pidns,comm

# 输出 JSON 格式（配合 jq 处理）
ps -eo pid,user,%cpu,%mem,comm | awk 'NR>1 {printf "{\"pid\":%s,\"user\":\"%s\",\"cpu\":%s,\"mem\":%s,\"cmd\":\"%s\"}\n",$1,$2,$3,$4,$5}'

# 监控关键指标并写入日志
while true; do
    ps -eo pid,user,%cpu,%mem,rss,comm --sort=-%cpu | head -5 >> /var/log/proc_monitor.log
    sleep 60
done &
🔬 第一性原理
所有应用场景都指向同一个核心：可观测性（Observability）。系统出现问题时，解决问题的第一步永远是"看清楚现在发生了什么"。ps 提供了进程维度的可观测性——它读取内核状态、以人类可理解的格式呈现，成为系统管理员的"眼睛"。只有先看清进程状态，才能做出正确的干预决策（kill、renice、重启服务）。

注意事项、标准符合与参考

使用 ps 时需要了解的重要限制、注意事项，以及官方标准依据。

📖 来源：man7.org ps(1) NOTES & STANDARDS

重要注意事项

⚠️
CPU 使用率的含义：官方手册明确指出，%CPU 表示进程整个生命周期内的 CPU 时间占比，而非近期瞬时值。因此刚启动的高 CPU 进程在这里可能显示很低，而长期运行但最近空闲的进程可能显示较高。需要实时 CPU 率请使用 top 或 pidstat。
⚠️
SIZE 和 RSS 的不完整性：官方注明 SIZE 和 RSS 字段不包含页表、内核栈、thread_info 和 task_struct 本身的内存，通常至少低估 20 KiB 的常驻内存。
ℹ️
无需特殊权限：这个版本的 ps 通过读取 /proc 虚拟文件工作，不需要 setuid kmem 或任何特殊权限。请不要给 ps 任何特殊权限。
ℹ️
WCHAN 显示：正确显示 WCHAN 需要内核符号表。对于 2.6 之前的内核，需要安装 System.map 文件。
🚫
僵尸进程标记：进程被标记为 <defunct> 表示是僵尸进程。它们已退出，但父进程未正确销毁它们。若父进程退出，init(8) 会最终回收这些进程。大量僵尸进程会耗尽 PID 空间，是严重的 Bug 信号。

🔬 第一性原理 · 总结
ps 的本质是内核状态的只读镜像工具。从最底层看：CPU 执行指令 → 产生 task_struct（进程描述符）→ 内核通过 procfs 映射为文件 → ps 读取文件并格式化 → 管理员看到进程状态。整个链路没有任何"魔法"，只有内核数据结构的逐层抽象。

理解 ps 的三个关键洞察：
1. 无副作用：ps 只读取，不修改任何进程状态，可以随时随意运行；
2. 时间切片：ps 是快照，不是连续监控，两次 ps 之间发生的变化不可见；
3. 权限边界：普通用户只能看到自己的进程细节，root 可以看到所有进程——这是 procfs 的访问控制机制，与 ps 本身无关。
