GitHub Copilot CLI
常用命令详解

基于 GitHub 官方文档整理，涵盖 gh copilot（CLI 扩展）与新一代独立 copilot CLI 的核心命令、底层原理及真实使用场景。

🔗 官方参考文档：cli-command-reference   🔗 gh-copilot GitHub 仓库

0两代 CLI 架构对比

GitHub Copilot CLI 经历了两代演进。理解两代差异是正确使用命令的前提。

⚠️
重要说明： gh copilot 扩展已于 2025 年 10 月 25 日正式弃用，官方建议迁移至新一代独立 copilot CLI。本文两代命令均有详细介绍，方便平滑迁移。
🔭 第一性原理

CLI 工具的本质是将人类意图翻译为机器指令。第一代 gh copilot 是「查词典」式翻译——你描述需求，AI 返回一条命令；新一代 copilot 是「智能助理」式协作——AI 理解上下文、执行多步骤任务，具备主动规划与代码修改能力。这一演进反映了 AI 从「补全」到「代理（Agent）」的范式转变。

1安装与认证

使用 Copilot CLI 前需要一个有效的 GitHub Copilot 订阅，并完成认证流程。

📦 安装方式

🔧 方式一：gh 扩展（旧）
# 需先安装 GitHub CLI
gh auth login
gh extension install github/gh-copilot
gh extension upgrade gh-copilot
⚡ 方式二：独立 CLI（推荐）
# 参考官方安装说明
# github.com/github/copilot-cli
copilot login
# 或通过环境变量认证
COPILOT_GITHUB_TOKEN=xxx copilot

🔐 认证原理

认证方式 变量 / 方法 适用场景 安全性
浏览器 OAuth（默认） copilot login 交互式开发环境 ⭐⭐⭐⭐⭐
细粒度 PAT COPILOT_GITHUB_TOKEN CI/CD 自动化 ⭐⭐⭐⭐
gh CLI Token GH_TOKEN 已有 gh 认证环境 ⭐⭐⭐⭐
GitHub Actions Token GITHUB_TOKEN Actions 工作流 ⭐⭐⭐
ℹ️
注意：不支持经典 PAT（ghp_ 前缀）。优先级：COPILOT_GITHUB_TOKEN > GH_TOKEN > GITHUB_TOKEN
🔭 第一性原理

认证的本质是「身份证明 + 权限授权」。OAuth Device Flow 的设计哲学是：终端无法安全存储 client_secret，因此将浏览器作为可信环境完成授权，再将令牌回流到 CLI。这避免了密码泄露风险，且令牌范围可精确控制（仅 Copilot Requests 权限），符合最小权限原则。

2gh copilot suggest — 命令建议

💡
gh copilot suggest [query] [-t target]
根据自然语言描述，AI 生成对应的 shell / git / gh 命令建议
核心命令 自然语言 命令生成

📖 用法与参数

Shell 
# 基础用法（交互式引导）
gh copilot suggest

# 直接传入查询（自动引导选择命令类型）
gh copilot suggest "列出所有超过 5GB 的文件"

# 指定目标类型 -t / --target: shell | git | gh
gh copilot suggest -t shell "Kill processes holding onto deleted files"
gh copilot suggest -t git   "Undo the most recent local commits"
gh copilot suggest -t gh    "List pull requests waiting for my review"

# 其他实用示例
gh copilot suggest -t shell "Convert MOV to animated PNG"
gh copilot suggest -t git   "Setup LFS for images"
gh copilot suggest -t gh    "Create pull request"

🎯 交互流程

📊 -t 参数对比

类型命令示例适用场景命令前缀
shell find / -size +5G 文件操作、系统管理、进程管理 bash/zsh 原生命令
git git log --oneline --graph 版本控制、分支管理、历史查询 git ...
gh gh pr list --reviewer @me PR 管理、Issue 操作、GitHub API gh ...
示例输出 
# 执行：gh copilot suggest -t shell "列出所有超过5GB的文件"

version 1.0.0 (2024-03-21)

Suggestion:

  find / -type f -size +5G 2>/dev/null

? Select an option
❯ Execute command
  Copy command to clipboard
  Revise command
  Explain command
  Rate response
  Exit
🔭 第一性原理

gh copilot suggest 的核心是自然语言到命令语言的语义映射。人类语言描述的是「目标状态」，shell 命令描述的是「操作步骤」。LLM 通过在海量命令文档、StackOverflow 问答、man pages 上训练，习得了这种映射关系。-t 参数的作用是约束命令空间，相当于告诉 AI「请只从 git 命令集合里找答案」，减少歧义、提高精准度。这本质上是一种有监督的「意图消歧」机制。

3gh copilot explain — 命令解释

🔍
gh copilot explain [command]
用自然语言逐段解释任意 shell / git 命令的含义与作用
理解命令 安全审查 学习工具

📖 用法

Shell 
# 交互式输入
gh copilot explain

# 直接解释指定命令
gh copilot explain 'du -sh | sort -h'

# 解释复杂管道命令
gh copilot explain 'git log --oneline --graph --decorate --all'

# 解释危险命令（先解释，再决策）
gh copilot explain 'rm -rf /'

# 解释 bfg 工具命令
gh copilot explain 'bfg --strip-blobs-bigger-than 50M'

# 解释 make 命令参数
gh copilot explain 'make -j 5'

# 查看帮助
gh copilot explain -h

💡 典型示例输出

示例输出：make -j 5
Explanation:

  • make is a build automation tool used to compile and build projects.
  • -j 5 specifies the maximum number of jobs (parallel tasks)
    that make can run simultaneously.
    In this case, it is set to 5.

🛡️ 核心使用场景

🔴 安全审查

在执行来自网络的陌生脚本前，先通过 explain 了解每一步的作用，避免「无知执行」高危命令。

📚 学习提升

遇到复杂的正则、管道、重定向、参数组合时，explain 相当于实时的 man page + 专家注释。

👥 代码审查

CR 时遇到运维脚本不熟悉的命令，快速获取自然语言解释，提高 review 质量与效率。

🔧 调试定位

日志中出现的命令输出不符合预期，explain 帮助快速定位命令语义与副作用。

🔭 第一性原理

gh copilot explain 的本质是「逆向翻译」——将机器语言翻译为人类语言。命令行语法是高度压缩的符号系统，每个 flag 背后都是一套设计决策。LLM 能够执行这种翻译，是因为命令文档、注释、讨论区大量存在于训练数据中，形成了「符号 → 语义」的隐式映射。此命令本质上是对「黑盒命令」的信任验证机制，让开发者在执行前建立认知理解，是「信任但验证」原则在 CLI 层面的具体实现。

4gh copilot alias — 快捷别名

🔗
gh copilot alias -- [shell]
为 suggest / explain 生成 shell 别名配置，支持命令执行并自动加入 history
效率提升 Shell 集成

📖 安装别名

Bash 
# 添加到 ~/.bashrc
echo 'eval "$(gh copilot alias -- bash)"' >> ~/.bashrc
source ~/.bashrc
Zsh 
# 添加到 ~/.zshrc
echo 'eval "$(gh copilot alias -- zsh)"' >> ~/.zshrc
source ~/.zshrc
PowerShell 
$GH_COPILOT_PROFILE = Join-Path -Path $(Split-Path -Path $PROFILE -Parent) `
  -ChildPath "gh-copilot.ps1"
gh copilot alias -- pwsh | Out-File ( New-Item -Path $GH_COPILOT_PROFILE -Force )
echo ". `"$GH_COPILOT_PROFILE`"" >> $PROFILE

⚡ 别名速查

别名等价命令用途特性
ghcs gh copilot suggest 快速建议命令 支持执行 + 加入 history
ghce gh copilot explain 快速解释命令 简化输入

🚀 使用别名示例

Shell 
# suggest 别名 — 建议并可执行
ghcs "print Hello world"
# Suggestion: echo "Hello world"
# ? Select an option > Execute command > Yes
# Hello world

# explain 别名
ghce 'chmod 755 /usr/local/bin'

# 自定义别名（手动定义，无需 alias 子命令）
alias '??'='gh copilot suggest -t shell'
alias 'git?'='gh copilot suggest -t git'
alias 'gh?'='gh copilot suggest -t gh'
💡
亮点：使用 ghcs 执行建议命令后，该命令会自动加入 shell history，方便 ↑ 调取复用。
🔭 第一性原理

别名机制的本质是降低认知摩擦成本。每次需要输入 gh copilot suggest（18个字符）会产生微小但真实的「使用阻力」，最终影响工具的使用频率。alias 子命令将摩擦降到最低（2~4个字符），符合「工具越顺手越会被使用」的行为经济学规律。eval 配合 shell 初始化文件的设计，则确保了别名在每个会话中「零感知加载」，实现了工具集成的自动化。

6copilot — 新一代交互模式

新一代独立 copilot CLI 提供命令行 + 交互模式两种入口，支持 Agentic 任务、多轮对话、代码编辑、文件访问等能力。

📋 命令行命令总览

命令作用使用场景
copilot 启动交互式界面（REPL） 日常开发、代码问答、任务规划
copilot completion SHELL 生成 Tab 补全脚本（bash/zsh/fish） 提升 CLI 命令输入效率
copilot help [TOPIC] 显示帮助（billing/config/commands/permissions…） 查阅命令文档
copilot init 为当前 repo 初始化 Copilot 自定义指令 项目级 AI 个性化配置
copilot login [--host] OAuth 设备流认证 初次配置或多账户切换
copilot mcp 管理 MCP（Model Context Protocol）服务器 集成外部工具和数据源
copilot plugin 管理插件和插件市场 扩展 Copilot 能力
copilot update 下载并安装最新版本 保持版本更新
copilot version 显示版本信息并检查更新 版本确认

⚡ 三种工作模式

Shell 
# 启动交互模式
copilot

# 非交互式：直接传入 prompt（用于脚本/管道）
copilot -p "显示本周 git commits 并总结" --allow-tool 'shell(git)'

# 管道输入
./generate-prompt.sh | copilot

# 继续上次会话
copilot --continue

# Tab 补全（zsh 示例）
copilot completion zsh > "${fpath[1]}/_copilot"

# 初始化项目指令
copilot init
🔭 第一性原理

新一代 copilot CLI 的交互 REPL 模式是将 AI 从「工具」升级为「同事」的关键设计。REPL（Read-Eval-Print Loop）的本质是持续上下文保持——每轮对话都在同一个上下文窗口内累积，AI 能够理解「这个」、「上面提到的」等指代，无需用户每次都重复背景。三种模式（Standard/Plan/Autopilot）则体现了人机信任度的可调节性：用户可以根据任务风险级别决定 AI 的自主程度，这是负责任 AI 设计的核心原则之一。

7Slash 命令 — 交互界面控制命令

在 copilot 交互界面中，输入以 / 开头的命令可控制会话行为、切换模型、管理上下文等。

📌 会话管理

命令作用使用场景
/clear / /new / /reset 开始全新对话，清除所有历史 对话跑偏、重新开始任务
/compact [focus ...] 压缩对话历史为摘要，释放上下文窗口 长会话上下文不足时
/context 显示当前上下文窗口 Token 占用与可视化 监控上下文使用量
/resume 打开会话选择器，恢复历史会话 继续上次未完成任务
/exit / /quit 退出 CLI 结束工作会话

🧠 规划与代码

命令作用使用场景
/plan 在编码前生成结构化实现计划（Plan 模式入口） 复杂功能开发前
/diff 查看当前目录变更摘要（含分支 diff） 提交前 review 变更
/delegate 委托远程仓库执行任务，AI 生成 PR 复杂多文件修改
/fleet 并行子 Agent 执行任务的多个部分 大型并行任务
/pr 管理当前分支的 Pull Request PR 创建、review、合并

⚙️ 模型与配置

命令作用使用场景
/model / /models 选择 AI 模型 切换推理深度/速度平衡
/mcp 管理 MCP 服务器配置 添加/移除外部工具集成
/permissions 查看/清除当前会话工具和路径授权 审查 AI 访问权限
/allow-all / /yolo 启用所有权限（工具、路径、URL） 信任环境快速开发（慎用）
/instructions 查看和切换自定义指令文件 项目级 AI 行为定制

📊 信息与导航

命令作用
/help显示所有可用命令帮助
/copy复制最后一个响应到剪贴板
/env显示当前环境（指令、MCP、插件、扩展等）
/chronicle会话历史分析（standup 报告、提示、改进）
/changelog查看 CLI 变更日志
/login / /logout会话内认证状态管理
/add-dir PATH允许 AI 访问指定目录
/cwd / /cd切换或显示当前工作目录

⏰ 定时与调度（实验性）

Slash Commands
# 单次定时任务（30分钟后提醒）
/after 30m remind me the time

# 周期性任务（每小时运行测试）
/every 1h run tests

# 防止机器休眠（开发时保持活跃）
/keep-alive 2h
🔬
/after、/every、/ask 等命令目前处于实验性（Experimental）功能状态，需通过 /experimental 命令开启后才能使用。
🔭 第一性原理

Slash 命令系统的设计哲学是「命令即意图的精确表达」。普通自然语言对话存在歧义（「帮我看看变更」可能是 diff 也可能是 review），Slash 命令通过特殊前缀实现了精确意图路由，减少 LLM 理解歧义。上下文管理命令（/compact、/context）的存在揭示了 LLM 的核心限制——有限 Token 窗口，这些命令本质上是在有限内存约束下最大化工作效率的工程解法。

熟练使用键盘快捷键可以大幅提升在交互界面中的操作效率。

快捷键功能

! COMMAND在 Copilot 之外执行本地 shell 命令（单次）

🗂️ Diff 模式快捷键

快捷键功能
↑ / k向上移动一行
↓ / j向下移动一行
← / h跳到上一个文件
→ / l跳到下一个文件
c对选中行添加/编辑注释

🔭 第一性原理

键盘快捷键的价值在于「肌肉记忆替代认知决策」。每次需要思考「我该怎么操作」都会打断工作流（心流状态）。快捷键将操作内化为肌肉记忆后，认知负担趋近于零。! 命令的设计尤其体现了「不出 Copilot 环境即可做任何事」的设计哲学——本质上是在 AI 对话与系统操作之间建立了无缝切换通道。

📐 上下文窗口管理

上下文管理命令

/context

# 压缩对话历史（保留要点，释放空间）
/compact

🔒 权限管理

权限控制命令 / 参数 
# 查看当前会话工具和路径授权
/permissions

# 允许访问指定目录
/add-dir ./src

# 查看所有已授权目录
/list-dirs

# 非交互式：预授权特定工具（避免每次询问）
copilot -p "备份数据库" --allow-tool 'shell(pg_dump)'

copilot --allow-all-tools

# 在交互界面中开启所有权限
/allow-all
⚠️
安全提示：使用 --allow-all-tools 或 /yolo 时，Copilot 拥有与当前用户相同的系统访问权限。生产环境请勿使用，仅适合隔离的开发/测试环境。
🔭 第一性原理

权限系统的存在是因为「AI 的能力边界应与其信任边界对齐」。Copilot CLI 能执行文件读写、shell 命令等高危操作，如果不加限制，一个错误的 AI 决策可能导致不可逆损失。最小权限原则要求：AI 仅获得当前任务所需的最小权限集。/add-dir 和 --allow-tool 的粒度设计，让用户能精确控制 AI 的「触手范围」，这是「人类主导、AI 辅助」协作模式的技术保障。

10MCP / 插件 / 模型切换

🔌 MCP 服务器（Model Context Protocol）

MCP 是一个开放协议，允许 Copilot 连接外部数据源和工具（如数据库、API、IDE 等），扩展其上下文获取能力。

MCP 管理命令 
# 交互界面内管理 MCP
/mcp

# 命令行管理 MCP
copilot mcp --help

🤖 模型切换

模型选择 
# 查看并切换可用模型
/model
# 或
/models

# 降级到特定版本（Team 账户可用）

模型特性适用场景
高推理深度模型 复杂算法设计、架构分析、难以调试的 bug
高速模型 简单代码补全、快速问答、格式转换
Auto（自动选择） 让 Copilot 根据任务复杂度自动选择合适模型

🧩 插件管理

插件命令 
# 在交互界面中管理插件
/plugin

# 命令行中管理插件

🔭 第一性原理

MCP 的设计遵循「关注点分离」原则：LLM 负责推理，MCP 服务器负责数据访问。这避免了将所有能力都耦合在 AI 模型中的反模式。模型切换机制则体现了「工具应服务于任务，而非任务迁就工具」——不同模型有不同的能力-成本-速度权衡，提供切换能力让用户根据实际需求做最优决策，符合资源最优分配原则。

🔭 总结：第一性原理视角

GitHub Copilot CLI 的所有命令，归根结底都在解决一个问题：「降低人类在计算机前的认知负担」。

• suggest / explain：解决「不知道命令 / 看不懂命令」的知识壁垒问题
• alias / 快捷键：解决「知道命令但操作繁琐」的效率摩擦问题
• 上下文管理（/compact, /context）：解决 LLM 有限记忆与无限任务之间的资源矛盾
• 权限系统（/permissions, /add-dir）：解决 AI 能力强大但需要可控的信任边界问题
• MCP / 插件 / 模型切换：解决单一工具能力不足、需要扩展集成的问题

所有设计的终极目标是同一件事：让开发者专注于创造，而非被工具本身分心。
