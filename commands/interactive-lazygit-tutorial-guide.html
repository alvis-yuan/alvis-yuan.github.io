什么是 Lazygit

入门

Lazygit 是由 Jesse Duffield 开发的开源终端 UI（TUI），用 Go 语言编写，让你在终端里以可视化方式操作 Git。它将复杂的 Git 命令行转化为直观的键盘驱动界面，让交互式 Rebase、行级暂存、Cherry-pick 等操作只需一次按键即可完成。

🔗 官方原文
"If you're a mere mortal like me and you're tired of hearing how powerful git is when in your daily life it's a powerful pain in your ass, lazygit might be for you." —— Jesse Duffield (lazygit 作者)

为什么选择 Lazygit？

🚀 效率极高

交互式 Rebase、Cherry-pick、行级暂存，全部一键完成，无需记忆复杂命令

👁️ 所见即所得

实时 Diff 预览，分支图可视化，commit 颜色区分状态

⌨️ Vim 风格导航

j/k 上下移动，h/l 切换面板，适合终端深度用户

🔧 高度可定制

YAML 配置文件，自定义命令、快捷键、主题、pager

🧭 第一性原理

Git 的本质是一个内容寻址对象存储，所有操作（commit、branch、rebase）本质都是对 DAG（有向无环图）的操作。Lazygit 的核心价值在于：将这些抽象的图操作可视化为直观的面板交互，把认知负担从「记住命令语法」转移到「看界面做决策」，从而让 Git 强大的能力真正为普通开发者所用。

安装与启动

入门

Lazygit 支持所有主流平台，使用包管理器一键安装。以下来自官方 README。

平台包管理器安装命令
macOSHomebrewbrew install lazygit
WindowsScoopscoop bucket add extras && scoop install lazygit
WindowsWinGetwinget install -e --id=JesseDuffield.lazygit
Arch Linuxpacmansudo pacman -S lazygit
Fedoradnfsudo dnf install lazygit
Debian/Ubuntu ≥13aptsudo apt install lazygit
任意平台Gogo install github.com/jesseduffield/lazygit@latest

启动与别名设置

# 在任意 Git 仓库目录下运行
lazygit

# 推荐添加 lg 别名（加入 ~/.bashrc 或 ~/.zshrc）
echo "alias lg='lazygit'" >> ~/.zshrc
source ~/.zshrc

# 指定仓库目录启动
lazygit -p ~/projects/myapp

退出时自动 cd 到仓库（高级别名）

官方 README 提供了一个 shell 函数，退出 lazygit 时自动切换到当前操作仓库的目录：

# 加入 ~/.zshrc 或 ~/.bashrc
lg() {
  export LAZYGIT_NEW_DIR_FILE=~/.lazygit/newdir
  lazygit "$@"
  if [ -f $LAZYGIT_NEW_DIR_FILE ]; then
    cd "$(cat $LAZYGIT_NEW_DIR_FILE)"
    rm -f $LAZYGIT_NEW_DIR_FILE > /dev/null
  fi
}
💡 提示
使用 shift+Q 退出可跳过目录切换，只用 q 退出才会自动 cd。

🧭 第一性原理

安装工具的核心原则是零摩擦：越快进入工作流，学习曲线越低。Lazygit 用系统包管理器一行命令安装，符合「最小认知负担」原则，让开发者立刻专注于 Git 操作本身，而非环境配置。

界面总览

入门

Lazygit 启动后呈现 5 个主要面板，通过数字键 1–5 或 Tab/Shift+Tab 切换。右侧大面板为 Diff 预览区，实时跟随选中内容变化。

面板快捷键内容常用操作
Status1当前分支、远程状态查看仓库整体状态
Files2已修改文件（已/未暂存）暂存、提交、丢弃
Branches3本地分支、远程、Tag切换、新建、删除、Rebase
Commits4提交历史图Squash、Drop、Cherry-pick
Stash5暂存箱列表Apply、Pop、Drop

Commit 颜色含义（官方 FAQ）

🟢 绿色 commit

该 commit 已包含在 master/main 分支中

🟡 黄色 commit

该 commit 不在 master/main 分支中

🔴 红色 commit

该 commit 尚未推送到远程 upstream 分支

⚫ 暗色 commit

已推送且已合并的历史提交

🧭 第一性原理

界面设计遵循「信息密度最优化」原则：5 个面板对应 Git 工作流的 5 个核心状态维度（工作区、分支、历史、暂存箱、当前仓库），颜色编码直接映射 commit 在 DAG 中的拓扑位置（是否合并、是否推送），减少了用户需要进行的心理推理步骤。

2

暂存与提交

初级

这是 Lazygit 最基础也是最常用的工作流。从「修改文件」到「推送远程」全流程只需几个按键。

基本工作流

1
按 2 进入 Files 面板，看到红色（未暂存）文件列表
2
用 j/k 选中文件，按 Space 暂存；或按 a 暂存所有文件
3
按 c 弹出提交信息输入框，输入信息后按 Enter 提交
4
按 P（大写）推送到远程
按键操作等价 Git 命令
Space暂存/取消暂存文件git add / git restore --staged
a暂存所有/取消所有git add -A / git restore --staged .
c提交git commit -m "..."
C用外部编辑器提交git commit（打开 $EDITOR）
w跳过 pre-commit hook 提交git commit --no-verify
A追加到上一个 commitgit commit --amend --no-edit
d丢弃文件变更git checkout -- <file>
D查看丢弃选项菜单-
e在编辑器中打开文件-
i忽略该文件（加入 .gitignore）手动编辑 .gitignore

场景示例：修复 bug 并快速提交

# 假设你修改了 3 个文件，只想提交其中 1 个
# 1. 进入 Files 面板（按 2）
# 2. 用 j/k 移动到目标文件
# 3. 按 Space 暂存该文件（变为绿色）
# 4. 按 c → 输入 "fix: resolve null pointer in auth"
# 5. 按 P 推送
# 等价命令序列：
git add src/auth.go
git commit -m "fix: resolve null pointer in auth"
git push

Nuke 工作区（紧急清理）

当你想彻底丢弃所有未提交的变更（包括子模块），按 Shift+D 打开重置菜单，选择 nuke 选项。官方称之为 "Nuke the working tree"，还会有动画特效 🎆。

⚠️ 注意
nuke 操作不可逆（未暂存的变更会丢失），使用前请确认你真的不需要这些变更。

🧭 第一性原理

暂存区（Index）是 Git 设计中最核心的创新之一——它是「工作区快照」与「仓库快照」之间的缓冲层。Lazygit 的 Space 键将这个抽象的两阶段提交过程具象化为「单次按键即时反馈」，消除了因命令语法错误导致的操作失误。

行级精确暂存

初级

这是 Lazygit 最受开发者喜爱的特性之一。相比 git add -p 的繁琐交互，Lazygit 让你像编辑器一样直观地选择要暂存的代码行。

操作步骤

1
在 Files 面板选中文件，按 Enter 进入 hunk（代码块）视图
2
用 j/k 上下移动到目标代码行
3
按 Space 暂存当前行；或按 v 进入范围选择模式后再按 Space 暂存多行
4
按 a 暂存/取消整个 hunk（代码块）
5
按 Tab 在「未暂存区」和「已暂存区」之间切换查看
按键操作说明
Space暂存/取消暂存当前行精准到单行的暂存控制
v开启可视范围选择类似 Vim 的 Visual 模式，选中多行后按 Space
a暂存/取消整个 hunk一次性操作一个代码块
d删除该行（丢弃变更）用于移除不需要的调试代码
E在编辑器中编辑 hunk需要对 hunk 做更复杂的修改
Tab切换暂存/未暂存区在两个视图之间切换
💡 典型场景
你在一个文件里同时修复了 bug 和添加了调试日志（console.log/fmt.Println）。使用行级暂存，可以只提交 bug 修复代码，将调试代码排除或删除，保持提交历史干净。

🧭 第一性原理

代码提交的本质是「意图的记录」。一个好的 commit 应该只包含一个逻辑变更。行级暂存实现了「一次编辑，多次提交」——你不需要在写代码时就考虑提交粒度，事后可以任意拆分。这体现了 Git Index 层的核心价值：解耦「编辑」和「提交」两个行为的时序。

分支管理

初级

在 Branches 面板（3）可以完成所有分支操作，包括本地分支、远程分支和 Tag。

按键操作等价 Git 命令
Space切换到选中分支git checkout <branch>
n新建分支git checkout -b <name>
d删除分支git branch -d <branch>
D强制删除分支git branch -D <branch>
r将当前分支 Rebase 到选中分支git rebase <branch>
M将选中分支合并到当前分支git merge <branch>
f快速拉取（Fast-forward）git fetch && git merge --ff-only
R重命名分支git branch -m <newname>
u设置 upstreamgit branch --set-upstream-to
o在浏览器打开 PR 页面-（需 GitHub CLI 支持）
-切换到上一个分支git checkout -

场景：Feature 分支完整开发流程

1
在 Branches 面板按 n，输入 feature/user-auth 创建并切换新分支
2
开发完成后，在 Files 面板暂存并提交（Space → c）
3
回到 Branches，切换到 main，选中 feature/user-auth，按 r 执行 Rebase
4
切换回 feature 分支，按 P 推送，按 o 在浏览器打开 PR

🧭 第一性原理

Git 的分支本质是一个指向 commit 对象的可移动指针，切换分支只是移动 HEAD 指针和更新工作区文件。Lazygit 将这个廉价的操作暴露为直观的列表选择，降低了开发者对「频繁切换分支」的心理抵抗——在 Git 中分支操作是 O(1) 的，完全可以随心创建和切换。

推送与拉取

初级
按键操作说明
pPull（拉取）从远程拉取当前分支变更；若无 upstream 会提示配置
PPush（推送）推送到 upstream；若无配置会提示
R刷新状态手动刷新 git 状态（不执行 fetch）
f（Branches）Fetch 远程分支仅拉取选中分支的远程数据
ℹ️ 自动 Fetch
Lazygit 默认每 60 秒自动从远程 fetch（git.autoFetch: true），无需手动触发。可在配置文件中调整间隔或关闭。

🧭 第一性原理

Push 和 Pull 是分布式版本控制的核心——每个本地仓库都是完整的副本（DVCS），远程同步是可选操作而非必须。Lazygit 通过大小写区分 Push（大写 P）和 Pull（小写 p），符合「大写=副作用更强」的记忆规律。

Stash 暂存箱

初级

Stash（暂存箱）让你临时保存未提交的变更，切换任务后再恢复。

位置按键操作等价命令
Files 面板sStash 所有未暂存变更git stash push
Files 面板S打开 Stash 选项菜单（含命名、仅暂存 staged 等）
Stash 面板Space应用 stash（保留 stash）git stash apply
Stash 面板g弹出 stash（应用并删除）git stash pop
Stash 面板d删除 stashgit stash drop
Stash 面板n从 stash 创建新分支git stash branch <name>

场景：紧急切换任务

1
正在 feature 分支开发，突然需要处理紧急 bug
2
Files 面板按 S → 选 "Stash all changes including untracked"，给 stash 起个名字（如 "wip: feature login"）
3
切换到 main 分支，修复 bug 并提交推送
4
切回 feature 分支，按 5 进入 Stash 面板，选中 stash 按 g 弹出恢复工作

🧭 第一性原理

Stash 的本质是一个特殊的 commit 对象链表（存储在 refs/stash），它将工作区和暂存区的差异临时序列化为 Git 对象。Lazygit 将 Stash 独立为第 5 个面板，体现了「最近最常用的操作最易触达」的 UI 设计原则。

交互式 Rebase

中级

Interactive Rebase 是 Git 中最强大的历史整理工具，但在命令行下极为繁琐（需要编辑 Vim TODO 文件）。Lazygit 将其简化为直观的键盘操作。

⚠️ 官方原文背景
"To do something as simple as amending an old commit requires following these steps: 1. Stash changes 2. Copy SHA 3. git rebase --interactive 4. Edit TODO in Vim 5. Save 6. git stash pop 7. git commit --amend 8. git rebase --continue" —— 作者 Jesse Duffield 描述了 CLI 的噩梦

Rebase 操作快捷键（Commits 面板）

按键操作说明
i开启交互式 Rebase 模式所有 commit 变为可操作状态
sSquash（合并到前一个 commit）合并 commit 并保留所有提交信息
fFixup（合并，丢弃当前信息）合并 commit 并丢弃当前的提交信息
rRename（重命名提交信息）内联编辑提交信息
R用编辑器重命名在外部编辑器中修改提交信息
dDrop（删除该 commit）从历史中移除此 commit
eEdit（在此暂停 rebase）可以修改代码再继续
ctrl+j向下移动 commit改变 commit 顺序
ctrl+k向上移动 commit改变 commit 顺序
m（全局）打开 merge/rebase 选项菜单包含 continue/abort/skip

场景 1：Squash 多个 WIP 提交

# Lazygit 操作步骤：
# 1. 进入 Commits 面板（4）
# 2. 按 i 开启 interactive rebase 模式
# 3. 移动到第一个 wip commit，按 s（squash）
# 4. 移动到第二个 wip commit，按 s
# 5. 移动到第三个 wip commit，按 f（fixup）
# 6. 按 m → Continue 开始执行

# 等价的 Git 命令（繁琐版本）：
git rebase -i HEAD~4
# 然后在 Vim 里手动改：
# pick a1b2c3 feat: add user authentication
# squash d4e5f6 wip: fix typo
# squash 789abc wip: forgot to add test
# fixup 012def wip: debug logging

场景 2：从标记的 Base Commit 开始 Rebase（官方特性）

官方 README 描述的进阶场景：你在 feature 分支上开发，该分支原本从 develop 切出，现在想改为从 master 切出，只需将 feature 上的 commit 重新 rebase 到 master。

1
在 Commits 面板找到 develop 分支上的最后一个 commit，按 Shift+B 标记为 base commit（显示为黄色背景）
2
切换到 Branches 面板，选中 master 分支
3
按 r（rebase onto），lazygit 自动执行 git rebase --onto master <base>
4
按 P 推送（可能需要 force push）

🧭 第一性原理

Rebase 的本质是提交的重新应用：将一系列 commit（diff + 元数据）在新的 base commit 之上重放，生成新的 commit 对象（SHA 会变化）。Lazygit 将这个多步骤操作简化为「选 commit → 按单键」，背后原理是 lazygit 自动构建并执行 git rebase --interactive 的 TODO 序列，相当于把 Vim 编辑 TODO 文件的过程内化为 UI 交互。

🔗 官方：Interactive Rebase 🔗 官方：从标记 Base Commit Rebase

Cherry-pick

中级

Cherry-pick 允许你将某个分支上的特定 commit 复制到另一个分支。Lazygit 将其设计为直觉性的「复制-粘贴」操作。

按键操作说明
Shift+C复制（标记）commit已复制的 commit 显示蓝色背景
Shift+V粘贴（cherry-pick）标记的 commits将复制的 commit 应用到当前分支
Ctrl+R重置（取消）cherry-pick 选择清除已标记的 commits

场景：将 hotfix 从 main 移植到 release 分支

1
在 Commits 面板，浏览 main 分支的历史，找到 hotfix commit
2
按 Shift+C 标记该 commit（变为蓝色）；可标记多个
3
切换到 Branches 面板，切换到 release/2.0 分支（按 Space）
4
在 Commits 面板按 Shift+V，标记的 commit 被 cherry-pick 到 release 分支
💡 批量 Cherry-pick
可以连续按 Shift+C 标记多个 commit（显示为蓝色），然后一次性用 Shift+V 按顺序粘贴所有标记的 commit。

🧭 第一性原理

Cherry-pick 的本质是将 diff 的应用：Git 计算出 commit C 相对于其父节点的差异，然后将这个差异应用到目标分支的最新 commit 上，生成新的 commit C'（SHA 不同，但内容等价）。Lazygit 的「复制-粘贴」隐喻非常准确地反映了这个「搬运 diff」的本质。

🔗 官方：Cherry-pick 特性

修改历史提交

中级

方法一：Amend 到任意历史 commit

官方 README 核心特性：Shift+A 在任意 commit 上，将当前已暂存的变更追加到该 commit（在后台自动执行 interactive rebase）。

1
在 Files 面板暂存你的修改（Space）
2
切换到 Commits 面板，用 j/k 导航到目标 commit
3
按 Shift+A，lazygit 自动将暂存区的变更合并到该 commit

方法二：自定义 Patch 精确修改（进阶）

这是 lazygit 独有的「Rebase Magic」——可以从历史 commit 中提取特定代码行组成 patch，然后进行各种操作：

1
在 Commits 面板，按 Enter 进入某个 commit 的文件列表
2
再按 Enter 进入文件的 diff 视图
3
按 Space 将特定代码行添加到自定义 patch（Custom Patch）
4
按 Ctrl+P 打开 Patch 操作菜单（从 commit 移除、创建新 commit、反向应用等）
📌 官方场景示例
"In this example we have a redundant comment that we want to remove from an old commit. We hit Enter on the commit to view its files, then Enter on a file to focus the patch, then Space to add the comment line to our custom patch, and then ctrl+p to view the custom patch options; selecting to remove the patch from the current commit." —— 官方 README
按键操作
A（Commits 面板）用当前暂存区内容 amend 最新 commit
Shift+A（Commits 面板）将暂存区内容 amend 到选中的任意历史 commit
Ctrl+P（全局）查看自定义 patch 操作菜单

🧭 第一性原理

Git 的 commit 对象一旦创建就不可变（SHA 由内容决定）。「修改历史提交」的实质是：创建新的替代 commit，然后将后续 commit 以新 commit 为 base 重放（rebase）。Lazygit 的 Shift+A 封装了这个完整的 rebase 流程，让它看起来像原地编辑，降低了用户对「历史不可变」这个 Git 底层概念的认知负担。

合并冲突处理

中级

当 merge 或 rebase 发生冲突时，lazygit 会提示你解决冲突，并在解决后自动继续操作。

冲突解决流程

1
执行 merge/rebase 后出现冲突，Files 面板中冲突文件显示 UU 标记（橙色）
2
选中冲突文件，按 Enter 进入冲突视图，看到 <<< / === / >>> 标记
3
在冲突视图中：Space 选择当前 hunk（ours 或 theirs），b 选择保留双方
4
按 e 用外部编辑器手动解决复杂冲突
5
所有冲突解决后（git.autoStageResolvedConflicts: true），lazygit 自动暂存并提示继续 rebase/merge
按键操作
Space选择当前冲突块（ours/theirs 交替）
b保留双方的改动（pick both）
e在外部编辑器中手动编辑
m打开 merge/rebase 继续菜单
💡 自动继续
配置 git.autoStageResolvedConflicts: true（默认开启），当所有冲突文件解决完毕后，lazygit 会自动询问是否继续 merge/rebase，大幅简化操作流程。

🧭 第一性原理

合并冲突本质是两个 diff 在同一位置的语义冲突，Git 无法自动决定「保留哪方的意图」，因此交由人来判断。Lazygit 的冲突视图将 ours 和 theirs 以高亮块呈现，将抽象的「三方合并决策」转化为「选哪块」的视觉问题，符合人类「选择」比「编辑」更直觉的认知规律。

自定义补丁 (Custom Patch)

高级

Custom Patch 是 Lazygit 独有的「Rebase Magic」功能。你可以从历史 commit 中精确选择某几行代码组成一个 patch，然后执行各种操作。

Patch 可执行的操作

操作说明
从当前 commit 移除将选中行从历史 commit 中删除（等价于针对该行的反向 patch）
拆分为新 commit将选中行单独拆分为一个新的 commit
反向应用到 Index将 patch 取反后应用到工作区（用于撤销特定改动）
应用到 Index将 patch 直接应用到暂存区

场景：从老 commit 中移除一个调试注释

1
Commits 面板，按 Enter 进入目标 commit 的文件列表
2
选中含有冗余注释的文件，按 Enter 进入 diff 视图
3
用 j/k 定位到注释行，按 Space 添加到自定义 patch（该行高亮）
4
按 Ctrl+P 打开 Patch 菜单，选择「从该 commit 中移除」
5
Lazygit 在后台自动执行 interactive rebase，完成后历史 commit 中的注释已消失
🎬 官方教程
官方提供了 YouTube 视频详解此功能：Rebase Magic Tutorial (youtu.be/4XaToVut_hs)

🧭 第一性原理

Git 的 patch（差异）是一等公民——Git 的底层操作（apply、cherry-pick、rebase）本质都是 patch 的应用。Custom Patch 功能直接暴露了这个底层原语，让用户能以「行」为粒度操控 Git 历史。这体现了「工具的能力边界应该与底层机制的能力边界对齐」的设计哲学。

🔗 官方：Custom Patches

Bisect 二分排查

高级

Git Bisect 用二分搜索在历史提交中找出「引入 bug 的那个 commit」。Lazygit 将命令行的多步骤操作简化为菜单驱动的流程。

操作步骤

步骤Lazygit 操作等价 Git 命令
开启 BisectCommits 面板，按 b，选 "Start bisect"git bisect start
标记坏的 commit选中已知有 bug 的 commit，按 b → "Mark as bad"git bisect bad HEAD
标记好的 commit选中已知正常的 commit，按 b → "Mark as good"git bisect good v1.0.0
自动导航Lazygit 自动 checkout 中间点，在新 commit 上测试后再标记git bisect 自动 checkout
找到 culpritLazygit 高亮显示第一个 bad commit输出：first bad commit
结束 Bisect按 b → "Reset bisect"git bisect reset

🧭 第一性原理

Bisect 基于「二分搜索」算法：在 N 个 commit 中找到第一个 bad commit 只需 log₂(N) 次测试。例如 1000 个 commit 只需约 10 次。Git 维护一个内部状态机跟踪搜索进度。Lazygit 将这个「状态机 + 测试循环」可视化，让开发者专注于「测试当前代码是否有 bug」这一本质问题，而非「记住执行哪个 git bisect 子命令」。

🔗 官方：Bisect 特性

Worktree 工作树

高级

Git Worktree 允许你同时 checkout 多个分支到不同目录，无需 stash 或 WIP commit 就能快速切换任务。Lazygit v0.40.0 新增了 Worktrees 视图。

Worktree 操作

位置按键操作
Branches 面板w打开 Worktree 选项菜单（创建 worktree）
Worktrees 面板w查看 Worktree 操作选项
全局Ctrl+R切换到最近的 repo/worktree

场景：同时开发两个功能不互相干扰

1
Branches 面板，选中 feature/auth，按 w → "Create worktree"
2
输入 worktree 路径（如 ../project-feat-auth），lazygit 自动创建并切换
3
用 Ctrl+R 在不同 worktree（仓库）之间快速切换，无需 stash
💡 适用场景
  • 同时开发 feature 分支和修复 hotfix，不相互干扰
  • 在一个 worktree 里运行测试，另一个里继续开发
  • Review PR 时不影响当前工作进度

🧭 第一性原理

Git Worktree 的本质是：一个 .git 目录（对象库和引用）可以对应多个工作目录，每个工作目录各自 checkout 一个独立的 HEAD。这消除了「切换分支」必须序列化的假设——分支切换的成本不是"保存工作"，而是"管理多个上下文"，Worktree 让这些上下文并行共存。

🔗 官方：Worktrees 特性

Undo / Redo（撤销/重做）

高级

Lazygit 提供基于 reflog 的撤销机制，让你可以反悔大多数 Git 操作。

按键操作说明
zUndo（撤销）通过 reflog 确定撤销的 Git 命令并执行
ZRedo（重做）重做上一次撤销的操作
⚠️ 重要限制
官方说明："The reflog will be used to determine what git command to run to undo the last git command. This does not include changes to the working tree; only commits are taken into consideration."
Undo/Redo 只针对 commit 层面的操作（通过 reflog），不包含工作区文件的变更。

场景：误 drop 了一个 commit

1
在 Commits 面板不小心按了 d（drop commit）
2
立即按 z（undo），lazygit 通过 reflog 找到前一个状态并恢复
3
Commit 恢复出现，可以继续工作

🧭 第一性原理

Git reflog 是所有 HEAD 变化的本地日志，每次 checkout、commit、rebase、reset 都在 reflog 中留下记录。Lazygit 的 Undo 本质是「读取 reflog 的 N-1 条目，对比当前状态，生成反向操作命令」，这是 Git「一切皆对象，对象不删除」设计哲学的最好体现——即使你 drop 了 commit，对象依然存在于 .git/objects 中，reflog 帮你找回指针。

自定义命令 (Custom Commands)

专家

Lazygit 提供灵活的自定义命令系统（Custom Command Keybindings），让你绑定任意 shell 命令到快捷键，支持 Go 模板语法获取上下文变量。

配置字段（来自官方 Custom_Command_Keybindings.md）

字段必填说明
key否触发快捷键，单字母或特殊键（如 <c-r>）
command是要执行的 shell 命令，支持 Go 模板语法
context是生效的面板上下文（commits/files/branches 等）
description否在快捷键菜单（?）中显示的说明
output否输出方式：none/log/popup/terminal
prompts否执行前的用户输入提示

示例配置

# ~/.config/lazygit/config.yml
customCommands:
  # 示例1：在浏览器打开选中 commit 的 GitHub 页面
  - key: '<c-g>'
    context: commits
    command: 'hub browse -- "commit/{{.SelectedLocalCommit.Hash}}"'
    description: 'Open commit in GitHub'

  # 示例2：切换文件的 staged/unstaged 状态
  - key: 'a'
    context: files
    command: "git {{if .SelectedFile.HasUnstagedChanges}} add {{else}} reset {{end}} {{.SelectedFile.Name | quote}}"
    description: 'Toggle file staged'

  # 示例3：交互式 fixup（自动查找对应的 commit）
  - key: '<ctrl+f>'
    context: files
    command: 'git commit --fixup={{.SelectedLocalCommit.Hash}}'
    description: 'Create fixup commit'
    output: log

可用的模板变量（部分）

变量说明
.SelectedLocalCommit.Hash选中 commit 的 SHA
.SelectedFile.Name选中文件名
.SelectedFile.HasUnstagedChanges文件是否有未暂存变更
.SelectedLocalBranch.Name选中分支名
.CheckedOutBranch.Name当前 checkout 的分支名

🧭 第一性原理

自定义命令体现了「可扩展性优于完备性」的设计原则：不可能在工具中内置所有操作，但可以提供一个「元编程」机制让用户自由扩展。Go 模板语法 + 上下文变量的组合，实质上是将 lazygit 的内部状态暴露为一个「可编程的外壳」，让用户将 lazygit 与任何外部工具（hub、gh、jira、slack...）集成。

🔗 官方：Custom_Command_Keybindings.md

🧭 总结：第一性原理

Lazygit 的所有快捷键设计遵循一个核心原则：「操作成本应该与操作的使用频率成反比，与操作的危险性成正比」。最常用的操作（Space 暂存、c 提交、P 推送）只需 1 个键；危险操作（D 强制删除、nuke）需要确认对话框或大写键；批量/高级操作（interactive rebase、bisect）有专属菜单保护。这种设计让新手安全入门，专家高效飞速。
