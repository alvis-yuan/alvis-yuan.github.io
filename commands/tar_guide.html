INTRODUCTION

Linux tar 命令
原理、场景与示例深度解析

📄 原文来源：man7.org — tar(1) Linux Manual Page 📄 原文来源：GNU tar 官方手册 1.35

tar（Tape Archiver）是 GNU/Linux 系统中历史最悠久、使用最广泛的归档工具。根据 GNU tar 官方手册：

"GNU tar is an archiving program designed to store multiple files in a single file (an archive), and to manipulate such archives. The archive can be either a regular file or a device (e.g., a tape drive, hence the name of the program, which stands for tape archiver)."
— man7.org/linux/man-pages/man1/tar.1.html

tar 本身不进行压缩——它是一个纯粹的归档工具，将多个文件打包为单一字节流。压缩功能由外部程序（gzip、bzip2、xz、zstd 等）通过管道实现。这种"单一职责"的设计正是 Unix 哲学的体现。

📌 命名溯源： tar = Tape Archiver，最初专为磁带备份设计。如今更多用于磁盘文件、网络传输，但名称沿用至今。

📝 本节小结

  • tar 是 Tape Archiver 的缩写，源自磁带备份时代，现广泛用于文件打包
  • tar 核心职责：将多个文件/目录打包为单一归档流
  • 压缩是可选的，通过管道委托给外部程序（gzip、bzip2、xz 等）
  • 遵循 Unix "单一职责" 哲学
PRINCIPLE

归档格式底层原理

📄 原文来源：GNU tar 1.35 — Basic Tar Format

根据 GNU tar 官方手册，tar 归档文件由一系列 512 字节的块（block）组成：

"A tar archive file contains a series of blocks. Each block contains BLOCKSIZE bytes... Each file archived is represented by a header block which describes the file, followed by zero or more blocks which give the contents of the file. At the end of the archive file there are two 512-byte blocks filled with binary zeros as an end-of-file marker."
— gnu.org/software/tar/manual/html_node/Standard.html

📦 归档文件物理结构

🔢 typeflag 文件类型字段

官方源代码 src/tar.h 中定义的文件类型标志：

typeflag 值宏定义含义
'0'REGTYPE普通文件（regular file）
'1'LNKTYPE硬链接（hard link）
'2'SYMTYPE符号链接（symbolic link）
'3'CHRTYPE字符设备（character special）
'4'BLKTYPE块设备（block special）
'5'DIRTYPE目录（directory）
'6'FIFOTYPEFIFO 特殊文件
'x'XHDTYPEPAX 扩展头（下一个文件）
'g'XGLTYPEPAX 全局扩展头

📐 校验和算法

根据 GNU tar 官方手册：

"The chksum field represents the simple sum of all bytes in the header block. Each 8-bit byte in the header is added to an unsigned integer, initialized to zero, the precision of which shall be no less than seventeen bits. When calculating the checksum, the chksum field is treated as if it were filled with spaces (ASCII 32)."
— gnu.org/software/tar/manual/html_node/Standard.html

🧱 阻塞因子（Blocking Factor）

tar 引入"记录（Record）"的概念以优化磁带 I/O：

"Each record of n blocks (where n is set by the --blocking-factor=512-size (-b 512-size) option to tar) is written with a single write() operation."
— gnu.org/software/tar/manual/html_node/Standard.html

默认阻塞因子为 20，即每次写 20×512 = 10240 字节。

🗄 归档格式版本

根据 man7.org tar(1) 手册，GNU tar 支持多种格式：

格式选项说明
v7--format=v7最古老的 Unix V7 格式
ustar--format=ustarPOSIX 1003.1-1988 标准格式
gnu--format=gnuGNU tar 1.13.x 扩展格式（默认）
pax / posix--format=paxPOSIX 1003.1-2001，最新标准，支持长文件名、扩展属性
oldgnu--format=oldgnutar 1.12 及更早版本的 GNU 格式

📝 本节小结 — 第一性原理

  • 一切皆字节流：tar 的本质是将多个文件的元数据（header）和内容（data）按 512B 块顺序拼接为一条字节流
  • 自描述结构：每个文件的 Header Block 携带文件名、权限、大小、时间戳、checksum，使 tar 无需外部索引
  • 校验和保障完整性：header 级别的 checksum 可检测归档损坏
  • 向后兼容：POSIX ustar → GNU → PAX 逐步演进，GNU tar 可读取所有旧格式
COMPRESSION

压缩器集成原理

📄 原文来源：man7.org — tar(1) Linux Manual Page

tar 通过 管道（pipe） 与外部压缩程序协作，官方手册中描述：

"Use external compression program PROG... when invoked without additional options, it should read data from standard input, compress it and output it on standard output."
— computerhope.com/unix/utar.htm（-I/--use-compress-program 选项说明）

压缩算法对比

算法选项扩展名速度压缩率特点
gzip-z.tar.gz / .tgz⚡⚡⚡★★★最广泛兼容，速度与压缩率均衡
bzip2-j.tar.bz2⚡⚡★★★★压缩率优于 gzip，速度稍慢
xz-J.tar.xz⚡★★★★★最高压缩率，适合发行包分发
zstd--zstd.tar.zst⚡⚡⚡⚡★★★★高速高压缩率，现代优选
自定义-I prog任意依程序依程序pigz(多线程gzip)、lz4 等

📝 本节小结

  • tar 通过 stdin/stdout 管道调用外部压缩器，完全解耦归档与压缩两个功能
  • gzip(-z) 速度快兼容好；bzip2(-j) 压缩率更高；xz(-J) 压缩率最佳；zstd 兼顾速度与压缩率
  • -I/--use-compress-program 允许接入任意压缩器（如 pigz 实现多核并行压缩）
  • --auto-compress 或 -a 可根据文件扩展名自动选择压缩器
SCENARIO 01

#01 创建归档与压缩

📄 原文来源：man7.org — tar(1) -c, --create 操作模式

根据官方手册："-c, --create: Create a new archive. Arguments supply the names of the files to be archived. Directories are archived recursively, unless the --no-recursion option is given."

🔬 底层原理

-c 模式下，tar 遍历所有指定路径，对每个文件/目录： ① 构造 512 字节的 POSIX Header Block（含文件名、权限、大小、时间戳等） ② 将文件内容按 512B 对齐写入数据块 ③ 最后写入两个全零 512B 的 EOF 标记 若指定压缩选项，整个字节流通过管道传给压缩器。

💻 示例

# 最基础：创建不压缩的 tar 归档（官方示例格式）
tar -cf archive.tar foo bar
# GNU 官方示例：归档 /etc 目录，详细输出
tar -cvf etc.tar /etc

# 使用 gzip 压缩（-z 选项）
tar -czvf archive.tar.gz /path/to/dir

# 使用 bzip2 压缩（-j 选项，压缩率更高）
tar -cjvf archive.tar.bz2 /path/to/dir

# 使用 xz 压缩（-J 选项，最高压缩率）
tar -cJvf archive.tar.xz /path/to/dir

# 使用 zstd（现代高速压缩）
tar --zstd -cvf archive.tar.zst /path/to/dir

# 使用 pigz 并行压缩（-I 自定义压缩程序）
tar --use-compress-program=pigz -cf archive.tar.gz /path/to/dir

# 自动根据扩展名选择压缩器（-a / --auto-compress）
tar -cavf archive.tar.xz /path/to/dir
💡 选项解释（来自官方手册）： -c 创建归档 | -v 详细输出（verbose）| -f 指定归档文件名 | -z gzip | -j bzip2 | -J xz

📝 本场景小结

  • -c（--create）是 tar 的核心操作：顺序写入 header+data 块序列
  • 目录默认递归归档，--no-recursion 可关闭
  • 压缩选项（-z/-j/-J/--zstd）只是管道过滤器，不改变归档结构
  • -v 增加详细输出，生产脚本中可用 --totals 替代，避免日志冗余
SCENARIO 02

#02 解压与提取文件

📄 原文来源：man7.org — tar(1) -x, --extract

官方手册："-x, --extract, --get: Extract files from an archive. Arguments are optional. When given, they specify names of the archive members to be extracted."

🔬 底层原理

提取时，tar 顺序读取归档字节流，解析每个 Header Block，根据 typeflag 字段恢复文件类型（普通文件、目录、符号链接等），并将数据块内容写入磁盘。若检测到压缩格式（通过文件扩展名或 magic bytes），先通过反向管道解压再归档读取。

默认情况下，tar 会将当前工作目录中已存在的文件 覆盖；使用 -k（--keep-old-files）可防止覆盖。

💻 示例

# 提取所有文件到当前目录（官方快速参考示例）
tar -xf archive.tar

# 提取并显示文件列表
tar -xvf archive.tar

# 提取到指定目录（-C 切换目录）
tar -xvf archive.tar -C /path/to/destination

# 解压 gzip 压缩的归档
tar -xzvf archive.tar.gz

# 解压 bzip2 压缩的归档
tar -xjvf archive.tar.bz2

# 只提取特定文件（指定 archive member 名称）
tar -xvf archive.tar config/settings.conf

# 不覆盖已有文件（-k / --keep-old-files）
tar -xvf archive.tar -k

# 自动识别压缩格式提取（无需手动指定 -z/-j 等）
tar -xf archive.tar.xz

# 提取时不还原文件权限（以当前用户 umask 为准）
tar -xf archive.tar --no-same-permissions
⚠️ 安全注意：提取前建议先用 -t 列出内容，避免 "tar slip" 攻击（恶意归档文件使用 ../ 路径逃逸）。

📝 本场景小结

  • -x 顺序读取归档块，依 header 中的 typeflag 恢复文件类型
  • -C dir 本质是 chdir(dir) 系统调用，在提取前改变工作目录
  • 现代 tar 可自动检测压缩格式（通过文件头 magic bytes），无需显式指定 -z/-j/-J
  • -k（--keep-old-files）防止覆盖，适合幂等部署场景
SCENARIO 03

#03 查看归档内容（不解压）

📄 原文来源：man7.org — tar(1) -t, --list

官方手册："-t, --list: List the contents of an archive. Arguments are optional. When given, they specify the names of the members to list."

官方快速参考："tar -tvf archive.tar # List all files in archive.tar verbosely."

🔬 底层原理

-t 模式下，tar 仅读取并解析每个文件的 Header Block，直接输出元数据，完全跳过数据块读取（seek 跳过），因此速度极快，也不消耗额外磁盘空间。

💻 示例

# 快速列出所有成员（官方示例）
tar -tvf archive.tar

# 列出压缩归档内容（无需解压）
tar -tvf archive.tar.gz

# 只列出匹配模式的成员
tar -tvf archive.tar '*.conf'

# 详细格式输出（类似 ls -l）
tar -tvvf archive.tar

# 结合 grep 过滤特定文件
tar -tvf archive.tar | grep '.conf'

# 列出时显示完整时间（--full-time）
tar --full-time -tvf archive.tar

# 验证归档卷标（--test-label）
tar --test-label -f archive.tar -v
📌 -tvf 输出格式解读： 每行格式类似 -rw-r--r-- user/group 1234 2024-01-01 12:00 path/to/file，依次为权限、用户/组、大小、时间、路径。

📝 本场景小结

  • -t 只解析 Header Block，跳过数据块，速度快、安全无副作用
  • 是提取操作前的"预检"最佳实践，防止误解压覆盖文件
  • -v 添加详细信息，-vv 输出更完整的元数据（类 ls -l 格式）
  • 可配合 grep/awk 做归档内容搜索与统计
SCENARIO 04

#04 排除文件与目录

📄 原文来源：man7.org — tar(1) --exclude, --exclude-from

官方手册："--exclude=PATTERN: Exclude files matching PATTERN, a glob(3)-style wildcard pattern."

以及："--exclude-vcs: Exclude version control system directories." （包括 .git/, .svn/, .hg/ 等）

🔬 底层原理

tar 在遍历文件树时，对每个路径应用 glob 风格的匹配规则。匹配成功的路径被跳过，不会写入归档。--exclude-from 本质是批量读取排除规则文件，逐行应用相同的匹配逻辑。

💻 示例

# 排除所有 .log 文件（glob 通配符匹配）
tar -czvf archive.tar.gz --exclude='*.log' /path/to/dir

# 排除特定目录
tar -czvf backup.tar.gz --exclude='./tmp' --exclude='./cache' .

# 排除版本控制目录（.git/.svn/.hg 等），来自官方 --exclude-vcs
tar -czvf project.tar.gz --exclude-vcs /path/to/project

# 从排除列表文件读取规则（--exclude-from / -X）
cat exclude_list.txt
## *.log
## tmp/
## *.cache
tar -czvf archive.tar.gz -X exclude_list.txt /path/to/dir

# 排除备份文件和锁文件（--exclude-backups）
tar -czf backup.tar.gz --exclude-backups /path/to/dir

# 排除含 CACHEDIR.TAG 的目录（--exclude-caches-all）
tar -czf backup.tar.gz --exclude-caches-all /home/user

# 排除 node_modules（前端项目常用）
tar -czvf webapp.tar.gz --exclude='node_modules' --exclude='.git' ./webapp
💡 路径匹配技巧： --exclude='./tmp'（相对路径）与 --exclude='tmp'（任意位置）行为不同。建议使用 --exclude='./目录名' 精确匹配顶层目录。

📝 本场景小结

  • --exclude 使用 glob(3) 通配符模式，默认匹配路径任意位置（--no-anchored 行为）
  • --exclude-vcs 是预置的 VCS 目录排除快捷方式，覆盖 .git/.svn/.hg/.bzr 等
  • --exclude-from 支持规则文件，适合复杂项目的可维护排除配置
  • --exclude-caches-all 识别 CACHEDIR.TAG 标准标记文件，适合系统备份
SCENARIO 05

#05 管道流式传输

📄 原文来源：man7.org — tar(1) -O, --to-stdout; -f - (stdin/stdout)

tar 支持用 -f - 指定标准输入/输出作为归档设备，实现流式归档传输。官方手册说明 -O, --to-stdout："Extract files to standard output."

🔬 底层原理

Unix 设计哲学：一切皆文件。tar 将归档视为字节流，-f - 使其读写 stdin/stdout，从而可以通过 shell 管道与 ssh、dd、nc 等工具无缝组合，实现零中间文件的流式传输。

💻 示例

# 通过 SSH 远程复制目录（无中间文件，官方推荐用法）
tar -cf - /path/to/dir | ssh user@remote 'tar -xvf -'

# 本地目录复制（比 cp -r 更高效，保留权限）
tar -cf - src_dir/ | tar -xf - -C dst_dir/

# 提取文件内容直接输出到 stdout（-O）
tar -xOf archive.tar config.conf

# 流式创建 gzip 压缩并写入文件（管道）
tar -cvf - /path/to/dir | gzip > archive.tar.gz

# 通过 nc（netcat）网络发送归档
# 接收端：
nc -l 9999 | tar -xvf -
# 发送端：
tar -cf - /data | nc remote-host 9999

# 归档同时用 tee 保存副本并计算 md5
tar -czf - /data | tee backup.tar.gz | md5sum

📝 本场景小结 — 第一性原理

  • tar 字节流本质上是一维序列，完美契合 Unix 管道模型
  • -f - 将归档设备指向 stdin/stdout，零中间文件，节省 I/O 和磁盘空间
  • SSH + tar 管道是系统管理员跨机器迁移数据的经典方案
  • -O（--to-stdout）可用于将归档成员直接"流"给下游命令
SCENARIO 06

#06 增量备份

📄 原文来源：man7.org — tar(1) -g, --listed-incremental

官方手册："-g, --listed-incremental=FILE: Handle new GNU-format incremental backups. FILE is the name of a snapshot file, where tar stores additional information which is used to decide which files changed since the previous incremental dump and, consequently, must be dumped again. If FILE does not exist when creating an archive, it will be created and all files will be added to the resulting archive (the level 0 dump)."

🔬 底层原理

GNU tar 通过 快照文件（snapshot file） 实现增量备份。快照文件记录上次备份时每个目录的成员列表和文件 mtime/ctime。下次备份时，tar 比对当前状态与快照，只归档变化的文件，并更新快照。

💻 示例

# 第一步：Level 0 全量备份（snap.snar 不存在时自动创建）
tar -czf backup-full.tar.gz --listed-incremental=snap.snar /data

# 第二步：Level 1 增量备份（使用相同快照文件）
tar -czf backup-incr-1.tar.gz --listed-incremental=snap.snar /data

# 强制 Level 0（重置快照，--level=0）
tar -czf backup-full-new.tar.gz --listed-incremental=snap.snar --level=0 /data

# 恢复：先恢复全量，再按序恢复增量
tar -xzf backup-full.tar.gz --listed-incremental=/dev/null -C /restore
tar -xzf backup-incr-1.tar.gz --listed-incremental=/dev/null -C /restore

# 实际脚本：每天增量备份
DATE=$(date +%Y%m%d)
tar -czf "/backup/data-${DATE}.tar.gz" \
    --listed-incremental=/backup/snap.snar \
    /data
📌 官方说明：恢复增量备份时，--listed-incremental 的参数通常指定为 /dev/null（因为不需要检查快照内容，仅满足语法要求）。

📝 本场景小结

  • 快照文件（.snar）记录目录结构和 mtime，是增量判断的核心依据
  • Level 0 = 全量备份（快照不存在时默认行为），Level N = 增量备份
  • --level=0 强制重置快照，实现周期性全量备份
  • 恢复时必须按备份顺序依次应用：Level 0 → Level 1 → Level 2 → …
SCENARIO 07

#07 追加与更新归档

📄 原文来源：man7.org — tar(1) -r, --append; -u, --update

官方手册：

"-r, --append: Append files to the end of an archive."
"-u, --update: Append files which are newer than the corresponding copy in the archive. Notice, that newer files don't replace their old archive copies, but instead are appended to the end of archive. The resulting archive can thus contain several members of the same name, corresponding to various versions of the same file."
— man7.org/linux/man-pages/man1/tar.1.html

🔬 底层原理

由于 tar 是顺序字节流格式，无法原地修改已归档的文件。-r 和 -u 的实现是将新文件追加到归档末尾，覆盖原来的 EOF 标记，再写入新的 EOF。因此同名文件可能存在多个版本，提取时取最后一个。

🚫 重要限制：官方手册明确指出 --append 和 --update 均 不支持压缩归档（compressed archives cannot be concatenated/updated）。这是因为压缩数据是整体流，无法在中间插入。

💻 示例

# 追加新文件到现有归档（-r / --append）
tar -rvf archive.tar newfile.txt

# 追加整个目录
tar -rvf archive.tar new_directory/

# 只追加比归档中版本更新的文件（-u / --update）
tar -uvf archive.tar /path/to/dir

# 将两个未压缩 tar 合并（-A / --concatenate）
tar -Avf archive1.tar archive2.tar
# 结果：archive2.tar 内容追加到 archive1.tar

# 从归档中删除特定文件（--delete，不支持压缩归档）
tar --delete -f archive.tar path/to/old_file.txt

# 验证归档完整性（-W / --verify）
tar -cWf archive.tar /data

📝 本场景小结

  • -r（追加）直接在 EOF 标记处继续写入，结构上是简单的字节流延伸
  • -u（更新）通过比较 mtime 决定哪些文件需要追加，旧版本不删除而是被压在底层
  • -A（--concatenate）合并两个 tar 文件，是删除第一个的 EOF 标记后追加第二个的内容
  • --delete 需要重写整个归档（内部创建临时文件），对大文件代价较高
  • 以上操作均不支持压缩归档（因为压缩格式的整体性约束）
SCENARIO 08

#08 保留权限与文件属性

📄 原文来源：man7.org — tar(1) -p, --preserve-permissions; --atime-preserve

官方手册："-p, --preserve-permissions: Extract information about file permissions (default for superuser)."

以及 "--atime-preserve[=METHOD]: Preserve access times on dumped files, either by restoring the times after reading (METHOD=replace; default) or by not setting the times in the first place (METHOD=system)."

🔬 底层原理

tar 的 Header Block 存储了 mode（权限）、uid/gid、uname/gname、mtime 等元数据。提取时，tar 通过 chmod()、chown()、utimes() 系统调用还原这些属性。 普通用户默认受 umask 影响；超级用户（root）默认启用 -p。

💻 示例

# 创建时保留 atime（--atime-preserve，避免备份改变访问时间）
tar --atime-preserve -czf backup.tar.gz /home/user

# 提取时强制保留原始权限（-p，非 root 用户需显式指定）
tar -xpvf archive.tar -C /restore

# 创建时强制指定属主（--owner 和 --group）
tar --owner=deploy --group=www-data -czf app.tar.gz ./app

# 提取时不还原属主（--no-same-owner，容器/CI 常用）
tar -xf archive.tar --no-same-owner

# 跟随符号链接（-h / --dereference，归档实际文件内容而非链接本身）
tar -chvf archive.tar /path/with/symlinks

# 设置归档时的 mtime（所有文件统一时间戳，可重现构建）
tar --mtime='2024-01-01 00:00:00' -czf reproducible.tar.gz ./dist

# PAX 格式保留完整时间戳精度（纳秒级）
tar --format=pax -czf archive.tar.gz /data
💡 可重现构建：使用 --mtime 固定时间戳 + --sort=name 固定文件顺序，可生成内容完全一致的归档，适合软件发布包的哈希验证。

📝 本场景小结

  • tar Header Block 的 mode/uid/gid/mtime 字段是权限与属性保留的数据载体
  • -p（--preserve-permissions）让 tar 在提取时调用 chmod/chown 还原权限，root 默认开启
  • --atime-preserve 避免备份读取文件时污染 atime，对监控系统透明
  • --mtime + --sort=name 是实现 "可重现归档"（reproducible builds）的关键

#09 路径转换与选择性提取

📄 原文来源：man7.org — tar(1) --strip-components; --transform

官方手册："--strip-components=NUMBER: Strip NUMBER leading components from file names on extraction."

以及："--transform=EXPRESSION, --xform=EXPRESSION: Use sed replace EXPRESSION to transform file names."

🔬 底层原理

提取时，tar 对从 Header Block 中读到的文件名路径进行字符串变换，然后再实际创建文件。--strip-components=N 移除路径前 N 级前缀；--transform 应用 sed 风格正则替换。这些操作在内存中完成，不影响归档文件本身。

💻 示例

# 去掉最外层目录，直接提取内部内容（--strip-components）
# 例如归档中是 project-v1.0/src/main.c
# 提取后变成 src/main.c（去掉了 project-v1.0/ 前缀）
tar --strip-components=1 -xvf archive.tar

# 去掉两级目录前缀
tar --strip-components=2 -xvf archive.tar -C /dest

# 用 sed 表达式重命名提取路径（--transform）
# 将路径中的 v1.0 替换为 current
tar --transform='s/v1\.0/current/' -xvf archive.tar

# 只提取特定路径下的文件
tar -xvf archive.tar --wildcards '*/config/*'

# 从文件列表中批量提取（-T / --files-from）
cat extract_list.txt
## data/file1.csv
## data/file2.csv
tar -xvf archive.tar -T extract_list.txt

# 提取时跳过开头的成员（-K / --starting-file）
tar -xvf archive.tar -K data/target_start_file.txt

# 同一个文件有多个版本时，只提取第 N 次出现（--occurrence）
tar -xvf archive.tar --occurrence=2 path/to/file.txt
📌 常见用法：GitHub/GitLab 下载的源码包通常包含顶层目录（如 repo-main/），使用 --strip-components=1 可直接解压内容到当前目录，无需手动移动文件。

📝 本场景小结

  • --strip-components=N 在内存中裁剪路径前缀，是 "去顶层目录" 的标准方法
  • --transform 应用 sed 正则，可实现任意路径重命名，支持多个 -s 表达式链
  • -T（--files-from）与 find 命令配合，可实现精细的选择性归档/提取
  • --occurrence 处理同名多版本成员，结合 -u 操作后特别有用
SCENARIO 10

#10 稀疏文件高效处理

📄 原文来源：GNU tar 1.35 — Basic Tar Format（Sparse Files 章节）

GNU tar 官方手册对稀疏文件的描述：

"Sparse files are often dbm files, or other database-type files which have data at some points and emptiness in the greater part of the file. Such files can appear to be very large when an 'ls -l' is done on them, when in truth, there may be a very small amount of important data contained in the file. It is thus undesirable to have tar think that it must back up this entire file, as great quantities of room are wasted on empty blocks..."
— gnu.org/software/tar/manual/html_node/Standard.html

🔬 底层原理

稀疏文件（Sparse File）在文件系统层面由"数据区段"（extents）和"空洞"（holes，未分配块，读取为零）组成。tar 默认会将这些零字节逐一写入归档，导致归档文件远大于实际数据量。

使用 -S（--sparse）时，GNU tar 通过 lseek/FIEMAP 系统调用探测文件的 hole 区段，在归档中只记录非零区段的偏移量和长度（struct sparse 描述符），还原时用 lseek 跳过空洞区域创建稀疏文件。

💻 示例

# 创建包含稀疏文件的归档（-S / --sparse）
tar -Sczf database.tar.gz /var/lib/database/

# 演示：先创建一个 1GB 的稀疏文件
dd if=/dev/zero of=sparse.img bs=1 count=0 seek=1G
ls -lsh sparse.img
## 显示：1.0G（逻辑大小）vs 0（实际占用）

# 不使用 -S：归档会包含大量零字节（非常大）
tar -czf non-sparse.tar.gz sparse.img

# 使用 -S：归档极小（只记录描述符）
tar -Sczf sparse-efficient.tar.gz sparse.img

# 指定探测方法（--hole-detection，默认 seek 方法）
tar -S --hole-detection=seek -czf output.tar.gz /data

# 指定稀疏格式版本（--sparse-version，默认 1.0）
tar -S --sparse-version=1.0 -czf output.tar.gz /data
💡 适用场景：虚拟机磁盘镜像（qcow2/vmdk 原始格式）、数据库文件（SQLite/DBM）、Docker overlay 层等，均是典型稀疏文件，使用 -S 可节省数十倍的归档空间。

📝 本场景小结 — 第一性原理

  • 稀疏文件的本质：文件系统层"空洞"（未分配块），逻辑大小远大于物理占用
  • -S 通过 lseek/FIEMAP 内核接口定位空洞，只归档实际数据区段
  • struct sparse 描述符（offset + numbytes，各 12 字节）记录每段空洞位置，提取时通过 lseek 重建稀疏结构
  • --hole-detection=seek（默认）或 raw（回退），seek 方式更快（内核级探测）

FIRST PRINCIPLE

第一性原理总结

从最底层的物理原理出发，理解 tar 命令的全貌：

🔑 核心第一性原理

原理 1：字节流即归档
tar 的本质是将多个文件的元数据（header）和内容拼接为一维有序字节流。这个约束决定了 tar 的所有设计决策：追加是延伸末尾，删除需重建，随机访问受限。
原理 2：自描述性
每个文件的 Header Block 包含还原该文件所需的全部元信息（名称、权限、大小、类型、时间戳、checksum），使 tar 归档在没有外部索引的情况下完全自包含。
原理 3：单一职责与管道组合
tar 不压缩，只归档。压缩通过 -z/-j/-J 选项调用外部程序，通过 stdin/stdout 管道传递。这使得任意压缩器都能接入，无需修改 tar 本身。
原理 4：向后兼容的格式演进
v7 → ustar → GNU → PAX 格式逐步扩展，GNU tar 能读取所有旧格式。PAX 扩展头（XHDTYPE 'x'）用于记录旧格式字段容纳不下的扩展信息（长文件名、纳秒时间戳、ACL 等）。

📝 最终总结

  • tar = Tape Archiver：从磁带时代走来的顺序字节流工具，核心不变，应用场景从未停止扩展
  • 一切皆字节流：理解了这个约束，tar 的所有行为（追加、更新、管道传输、不能随机修改压缩归档）都能从第一性原理推导
  • 组合而非集成：与外部压缩器、SSH、netcat 的管道组合，是 tar 强大生命力的根源
  • 官方文档是权威：man7.org tar(1) 和 GNU tar 1.35 手册是一切行为的最终定义来源
