01 什么是 rsync

根据 Linux 官方手册页：

官方定义

Rsync is a fast and extraordinarily versatile file copying tool. It can copy locally, to/from another host over any remote shell, or to/from a remote rsync daemon. It is famous for its delta-transfer algorithm, which reduces the amount of data sent over the network by sending only the differences between the source files and the existing files in the destination.

rsync 的核心特点：

  • 增量传输（Delta Transfer）：只传输源文件与目标文件之间的差异部分，大幅减少网络流量
  • 多场景支持：本地复制、SSH 远程传输、直接连接 rsync daemon
  • 元数据保留：可保留符号链接、硬链接、文件权限、所有者、时间戳等
  • 无需超级权限：普通用户即可使用
  • 流水线传输：内部流水线机制降低多文件传输的延迟
  • 灵活过滤：类似 GNU tar 的 exclude/include 过滤规则

rsync 广泛应用于备份、镜像和日常文件复制替代方案。它由 Andrew Tridgell 和 Paul Mackerras 最初编写，作者同时也是 Samba 项目的核心贡献者。

图1：rsync 与传统复制工具的对比 —— rsync 只传输差量数据

📌 本节小结

rsync 是 Linux/Unix 系统上事实标准的文件同步工具，其核心价值在于"只传输差异"的 Delta 传输算法。它不仅是一个网络工具，也是一个功能强大的本地文件管理工具。从第一性原理看：最小化数据传输量 = 减少网络 I/O = 提高效率。

02 历史与起源

rsync 的诞生源于一个真实的工程问题。1996 年，rsync.samba.org 记载：

起源背景

Andrew Tridgell 在澳大利亚国立大学攻读博士学位期间（1996年），面临在慢速网络链路上高效同步文件的问题。他设计了 rsync 算法，并以此为基础撰写了博士论文：「Efficient Algorithms for Sorting and Synchronization」（1999）。

发展里程碑

时间事件
1996年Andrew Tridgell 发布 rsync 1.0，基于其创新的滚动校验和差量同步算法
1999年博士论文正式发表，算法理论完善
2003年rsync 2.6.x 系列发布，引入重大功能改进
2008年rsync 3.0.0 发布，带来增量递归、更快的协议
2022年~Wayne Davison 维护的 rsync 3.2.x，引入 xxhash、zstd 支持
2026年5月rsync 3.4.3 发布（当前版本，来源：man7.org）

rsync 的地位

根据 rsync.samba.org/features.html，rsync 的灵魂算法（rsync algorithm）奠定了 Andrew Tridgell 博士论文的基础，描述了一种高效利用校验和来确定两个文件差异的方法，无需两个文件同时在同一台机器上存在。

📌 本节小结

rsync 算法的诞生不是偶然的工程积累，而是对「慢速网络下文件同步」这一具体工程问题的第一性原理思考：既然带宽有限，就不要传输已经存在的数据。这一思想至今仍是现代版本控制（Git）、云存储同步等系统的核心原则。

03 Delta 差量传输算法

这是 rsync 最核心的创新。官方手册页描述其为 "delta-transfer algorithm"。根据 Michael Stapelberg 对 rsync 协议的深度分析（基于官方源码），算法分为三个步骤：

算法三步骤

  1. 签名生成（Signature Generation）：接收端（Receiver）将本地旧文件分割为固定大小的数据块（block size = 文件长度的平方根取整），对每个块计算两种校验和：弱校验和（rolling checksum）和强校验和（MD4/MD5），将校验和列表发送给发送端。
  2. 签名搜索（Signature Search）：发送端（Sender）接收到校验和列表后，对本地新文件的每一个字节位置进行滑动窗口扫描，查找与接收端数据块匹配的内容，生成"匹配令牌"和"字面量字节"的混合指令序列。
  3. 文件重建（Reconstruction）：接收端根据发送端发来的指令序列——"在旧文件第 N 块取 K 字节" + "插入新的字面量字节"——重建出与发送端完全相同的文件。

图2：rsync Delta 传输算法完整流程（接收端→校验和→发送端→差量→重建）

块大小的选择

根据 Michael Stapelberg 的分析：

官方源码行为

块大小（block size）是文件长度的平方根取整（rounded square root of file length）。这是一个经典的权衡：块越小，匹配精度越高，但校验和数量越多；块越大，校验和少，但对小改动的匹配效率降低。

📌 本节小结

Delta 算法的第一性原理是：两个相似文件的大多数数据是共同的。因此，可以用"定位差异"代替"传输全量"。三个步骤（分块→签名→搜索→重建）是对这一思想的最小化实现。块大小的平方根选择则是对"校验和数量 vs 匹配精度"权衡的数学最优解。

04 校验和机制详解

rsync 使用两层校验和组合来平衡性能与准确性，这是其算法高效性的关键。根据 官方源码分析：

弱校验和（Weak Checksum / sum1）

来源：rsync 官方源码 checksum.c

rsync 首先计算称为 "sum1" 或 "fast signature" 的弱校验和。这是一个小型校验和（两个 uint16），可以对文件数据的滑动窗口以极小的计算代价进行计算。

弱校验和基于 Mark Adler 的 adler-32，由 Andrew Tridgell 针对 rsync 的需求改造。其关键特性是：

  • 滚动性（Rolling）：当窗口滑动一个字节时，新的校验和可以从旧校验和以常数时间 O(1) 推导出来，无需重新计算整个窗口
  • SIMD 加速：现代 rsync 提供 x86_64 SIMD 优化实现，进一步提速
# 滚动校验和的核心思想（O(1) 滑动）
# 当窗口从 data[i..i+L] 滑动到 data[i+1..i+L+1]
sum = sum - data[i] + data[i+L+1]   # 常数时间更新

强校验和（Strong Checksum / sum2）

来源：rsync 官方源码 checksum.c

只有在 sum1（弱校验和）匹配时，才会计算 sum2（强校验和）。sum2 是一个 16 字节的 MD4 哈希。更新的协议版本允许协商哈希算法，并支持更快的 xxhash 算法。

双层校验和的设计逻辑

图3：双层校验和过滤机制 —— 先用快速弱校验和初筛，再用强校验和精确验证

校验和算法演进

校验和类型早期版本现代版本（rsync 3.x）
弱校验和Adler-32 变体（滚动）同，增加 SIMD 加速
强校验和128-bit MD4MD4 / MD5 / xxhash（可协商）
文件级校验MD4/MD5xxhash / sha1 / sha256（--checksum）

📌 本节小结

双层校验和是性能与正确性的最优权衡：弱校验和快但不可靠，强校验和可靠但计算代价高。通过"先用廉价筛选器初筛，再用精确验证器确认"的两阶段设计，rsync 在保持高性能的同时确保数据一致性。这是工程设计中"分层过滤（Layered Filtering）"思想的经典应用。

05 传输阶段与工作流

根据 Michael Stapelberg 的 rsync 协议深度分析（基于协议版本 27），rsync 传输分为两个主要阶段：

官方协议说明

在协议层面，rsync 区分两种角色："sender"（发送端） 和 "receiver"（接收端）；以及 TCP 层面的 "client"（客户端） 和 "server"（服务端）。这两组角色可以任意组合：客户端和服务端都可以是发送端或接收端。

第一阶段：文件列表构建

发送端遍历本地文件树，生成并发送完整的文件列表（file list）给接收端。由于双方需要对文件列表排序（按文件名），该阶段必须完整传输后才能进入第二阶段（早期协议版本）。

  • 此阶段对发送端是随机 I/O（查询文件系统元数据）
  • 接收端在此阶段从网络读取数据写入 RAM

第二阶段：并发传输（三进程流水线）

来源：rsync 协议架构分析

第二阶段的架构使其可以用 3 个独立进程实现，每个进程都尽可能快地向网络发送数据（重度流水线）。这充分利用了发送端和接收端的硬件资源（I/O、CPU、网络）。

三进程并发工作：

  • 接收端比较进程：对文件列表中每个文件，若目标不存在则请求完整文件；若存在则发送块校验和列表给发送端
  • 发送端传输进程：根据请求，发送足以重建文件所需的最少数据（delta）
  • 接收端写入进程：接收 delta 数据并重建文件写入磁盘

图4：rsync 两阶段传输流程与三进程并发架构

📌 本节小结

rsync 的两阶段三进程架构体现了流水线（Pipeline）的第一性原理：将一个串行任务拆解为可并发执行的子任务，使各硬件资源（CPU/磁盘/网络）同时满载运行，从而最大化吞吐量。这与现代 CPU 指令流水线、工厂生产线的设计哲学完全相同。

06 Quick Check 快速检查

根据 Linux 官方手册页（rsync 3.4.3）：

官方说明

Rsync finds files that need to be transferred using a "quick check" algorithm (by default) that looks for files that have changed in size or in last-modified time. Any changes in the other preserved attributes are made on the destination file directly when the quick check indicates that the file's data does not need to be updated.

Quick Check 的判断逻辑

默认情况下，rsync 不会对每个文件都执行完整的 Delta 算法。它首先用"快速检查"决定文件是否需要传输：

图5：Quick Check 决策树 —— 默认基于文件大小和 mtime 判断是否需要 Delta 传输

影响 Quick Check 行为的关键选项

选项行为说明
-c / --checksum不用 mtime/size 判断，改为计算文件级校验和比较（更准确，更慢）
-I / --ignore-times忽略时间戳，始终更新文件（对每个文件都运行 Delta）
--size-only只比较文件大小，忽略 mtime
-u / --update跳过目标端 mtime 比源更新的文件

📌 本节小结

Quick Check 是 rsync 高性能的第二个关键：在执行昂贵的 Delta 算法前，先用廉价的元数据比较（文件大小+mtime）淘汰不需要传输的文件。这是"不做不必要的工作"原则的体现。只有当文件确实需要同步时，才启动完整的 Delta 流程。

07 传输模式

根据 官方手册页，rsync 有两种联系远程系统的方式：

官方说明

There are two different ways for rsync to contact a remote system: using a remote-shell program as the transport (such as ssh or rsh) or contacting an rsync daemon directly via TCP.

语法速查

# 本地模式
rsync [OPTIONS] SRC... DEST

# 远程 Shell 模式（单冒号，通过 SSH/RSH）
# Pull（从远程拉取）
rsync [OPTIONS] user@host:SRC... DEST
# Push（推送到远程）
rsync [OPTIONS] SRC... user@host:DEST

# Daemon 模式（双冒号，直接 TCP 873 端口）
# Pull
rsync [OPTIONS] user@host::module/SRC... DEST
rsync [OPTIONS] rsync://user@host[:port]/SRC... DEST
# Push
rsync [OPTIONS] SRC... user@host::DEST
模式语法标志端口/传输典型用途
本地无冒号本地文件系统本地目录备份/整理
SSH 远程单冒号 :SSH（默认22）安全加密传输到远程主机
RSH 远程单冒号 :RSH内网老系统
Daemon 模式双冒号 ::TCP 873公共镜像、匿名同步
Daemon URLrsync://TCP 873脚本友好的 Daemon 访问
注意：不支持双端远程传输

Rsync copies files either to or from a remote host, or locally on the current host. It does not support copying files between two remote hosts.（官方手册原文）

📌 本节小结

rsync 的传输模式设计遵循单一职责：rsync 本身专注于文件差量计算与传输，而网络安全由 SSH 负责、认证由 rsyncd.conf 负责。这种"关注点分离"使 rsync 既可在高安全需求场景（SSH加密）使用，也可在公共镜像场景（Daemon 匿名）高效运作。

08 常用使用场景

以下场景均基于 rsync(1) 官方手册页 和 rsync.samba.org/examples.html 的官方示例，并附加原理说明。

📁
场景① 本地文件复制 Local
替代 cp 命令，更强大的本地复制工具

根据官方手册："You can also use rsync in local-only mode, where both the source and destination don't have a ':' in the name. In this case it behaves like an improved copy command."

# 复制整个目录（含目录本身）
rsync -av /src/foo /dest
# 结果：/dest/foo/...

# 复制目录内容（注意末尾斜线）
rsync -av /src/foo/ /dest/foo
# 结果与上面等价

# 显示进度的大文件复制（替代 cp 并显示进度条）
rsync -av --progress largefile.iso /backup/

# 重命名复制：将 foo.c 复制为 bar.c
# 来源：官方手册 COPYING TO A DIFFERENT NAME
rsync -ai src/foo.c save/bar.c

⚙️ 原理说明

末尾斜线（trailing slash）是核心语义：根据官方手册，rsync -av /src/foo /dest 会在 /dest 下创建 foo 目录；而 rsync -av /src/foo/ /dest/foo 则将 foo 的内容复制到 /dest/foo，两者结果等价但语义不同。"A trailing / on a source means 'copy the contents of this directory'"。

⚠️ 排序行为

rsync 总会对传输文件列表排序。根据官方手册"SORTED TRANSFER ORDER"章节，这可能导致文件传输顺序与命令行指定顺序不同。如需控制顺序，应分多次调用 rsync。

🔐
场景② 通过 SSH 远程备份 SSH
加密传输远程文件，最常用的生产场景

官方手册示例："rsync -avz foo:src/bar /data/tmp — This would recursively transfer all files from the directory src/bar on the machine foo into the /data/tmp/bar directory on the local machine. The files are transferred in archive mode..."

# 从远程拉取（Pull）：-a 归档模式，-v 详细，-z 压缩
rsync -avz user@remote-host:src/bar /data/tmp

# 推送本地到远程（Push）
rsync -avz /local/dir/ user@remote-host:/remote/backup/

# 指定 SSH 端口 2222（-e 指定远程 shell）
rsync -avz -e "ssh -p 2222" /local/data/ user@host:/backup/

# 官方示例：rsync -Cavz . arvidsjaur:backup
# Tridgell 本人使用：通过 PPP 链路每晚备份家目录
rsync -Cavz . arvidsjaur:backup

⚙️ 原理说明

单冒号语法表示通过远程 shell（默认 SSH）传输。rsync 会在远端启动另一个 rsync 进程，两端的 rsync 通过 SSH 加密隧道通信。-z 选项在网络层面压缩数据，与 Delta 算法叠加进一步减少传输量。

-a（archive）选项 等价于 -rlptgoD，即递归 + 保留符号链接 + 保留权限 + 保留时间戳 + 保留所有者 + 保留组 + 保留设备文件，是生产环境的标准选项组合。

🪞
场景③ 镜像同步（精确副本）Mirror
--delete 使目标成为源的精确镜像

根据 rsync 官方示例页，Samba 项目本身就用匿名 rsync 维护镜像：

# 镜像同步：目标与源保持完全一致（含删除目标多余文件）
rsync -av --delete /source/ /mirror/

# 官方 Samba 镜像示例（带 SSH 加密）
# 来源：rsync.samba.org/examples.html 和官方 man page
rsync -az -e ssh --delete ~ftp/pub/samba/ nimbus:"~ftp/pub/tridge/samba"

# 本地磁盘镜像（rsync 官方备份磁盘示例）
rsync -ax --exclude fstab --delete /usr/ /backup/usr/

# 先 dry-run 测试（-n 不实际执行）
rsync -av --delete -n /source/ /mirror/

⚙️ 原理说明

--delete 的工作机制：根据官方手册，--delete 会删除接收端有而发送端没有的文件（extraneous files）。它只作用于被同步的目录。重要安全机制："If the sending side detects any I/O errors, then the deletion of any files at the destination will be automatically disabled." 即当发送端出现 I/O 错误时自动禁用删除，防止临时故障（如 NFS 错误）导致目标端文件被大量删除。

⚠️ 危险操作警告

官方手册明确警告："This option can be dangerous if used incorrectly! It is a very good idea to run first using the --dry-run option (-n) to see what files would be deleted." 错误指定目标路径可能导致大量文件被删除！

💾
场景④ 7天滚动增量备份 Backup
--backup + --backup-dir 实现增量存储

这是 rsync.samba.org/examples.html 的官方示例，由 Tridgell 本人提供：

#!/bin/sh
# 官方来源：rsync.samba.org/examples.html
# 7 天滚动增量备份脚本

# 备份目录
BDIR=/home/$USER
EXCLUDES=$HOME/cron/excludes
BSERVER=owl
export RSYNC_PASSWORD=XXXXXX

BACKUPDIR=`date +%A`   # Monday ~ Sunday

OPTS="--force --ignore-errors --delete-excluded \
      --exclude-from=$EXCLUDES \
      --delete --backup --backup-dir=/$BACKUPDIR -a"

# 清空上周同一天的增量目录
mkdir $HOME/emptydir
rsync --delete -a $HOME/emptydir/ $BSERVER::$USER/$BACKUPDIR/
rmdir $HOME/emptydir

# 执行备份：最新全量在 current，变更文件进入 $BACKUPDIR
rsync $OPTS $BDIR $BSERVER::$USER/current

⚙️ 原理说明

这里组合了三个关键选项：

  • --backup：在覆盖目标文件前，将旧版本重命名为备份（加 ~ 后缀）
  • --backup-dir=/$BACKUPDIR：将备份文件统一存到以星期命名的子目录（如 Monday、Tuesday）
  • --delete：删除源端已删除的文件，保持 current 目录是精确镜像

最终效果：current/ 始终是最新完整备份，Monday/~Sunday/ 存储对应那天被修改/删除的文件历史版本，实现 7 天滚动增量备份，且总空间占用远小于 7 份全量备份。

⏸️
场景⑤ 大文件断点续传 Resume
--partial / --append 处理中断传输
# --partial：保留部分传输文件，下次继续（等同于 -P 的一半）
rsync -av --partial user@host:/huge-file.tar.gz /local/

# -P 等价于 --partial --progress（最常用）
rsync -avP user@host:/data/bigfile.iso /backup/

# --append：仅追加新增数据（适用于只增长的文件，如日志）
# 来源：官方手册 --append 选项说明
rsync --append user@host:/var/log/app.log /backup/app.log

# --inplace：直接在目标文件上修改（适合大文件小改动，避免临时文件）
# 来源：官方手册 --inplace 选项
rsync -av --inplace user@host:/vm/disk.img /backup/

⚙️ 原理说明

官方手册对 --partial 的说明：正常情况下，如果传输中断，rsync 会删除部分传输的文件。使用 --partial 后，这些文件会被保留，下次传输时 rsync 的 Delta 算法会基于这个部分文件计算差量，只传输剩余部分。

官方手册对 --append 的警告："The use of --append can be dangerous if you aren't 100% sure that all the files in the transfer are shared, growing files." —— 该选项信任接收端现有内容与发送端一致，只追加新增部分，不验证已有内容，适合日志等只增长文件。

官方手册对 --inplace 的说明：该选项改变 rsync 更新文件的方式，直接写入目标文件而非创建临时文件后移动替换。适合块存储设备变更、copy-on-write 文件系统快照等场景。

场景⑥ rsync Daemon 匿名同步 Daemon
公共镜像站、内网分发的标准方案

根据 rsync.samba.org/features.html："supports anonymous rsync which is ideal for mirroring"。

# 列出远程 Daemon 的可用模块
rsync somehost.mydomain.com::

# 从 Daemon 拉取模块 src 内容
rsync -av host::src /dest

# 官方 vger CVS 树镜像脚本（来源：rsync.samba.org/examples.html）
rsync -az --delete --force vger.rutgers.edu::cvs/ /var/www/cvs/vger/

# 使用 rsync:// URL 语法
rsync -av rsync://host/module/path /local/dest

# 需要认证的 Daemon 连接（避免密码提示）
export RSYNC_PASSWORD=mypassword
rsync -av user@host::module /dest

⚙️ 原理说明

Daemon 模式下，rsync 作为独立服务运行，监听 TCP 873 端口，通过 /etc/rsyncd.conf 配置访问模块（module）。

与 SSH 模式的核心区别：Daemon 模式不加密（明文传输），但适合公共镜像（无需 SSH 账号）。若需要加密，可通过 SSH 隧道代理：

# 通过 SSH 隧道加密 Daemon 连接（官方手册 RSYNC_CONNECT_PROG 示例）
export RSYNC_CONNECT_PROG='ssh proxyhost nc %H 873'
rsync -av targethost::module/src/ /dest/

环境变量 RSYNC_PROXY 还支持通过 Web 代理（HTTP CONNECT）连接到 Daemon 端口 873。

场景⑦ 排除与过滤规则 Filter
精确控制哪些文件参与同步

官方手册描述 rsync 拥有类似 GNU tar 的过滤规则系统。根据官方示例：

# 排除特定模式（-C 选项使用 CVS 忽略规则）
# 官方示例：同步 samba 代码，排除编辑器备份文件
rsync -avuzb --exclude '*~' samba:samba/ .

# 使用 --exclude-from 文件指定排除列表
rsync -av --exclude-from=/home/user/excludes.txt /src/ /dest/

# 包含/排除组合：只同步 .jpg 文件
rsync -av --include='*.jpg' --exclude='*' /src/ /dest/

# --cvs-exclude（-C）：自动排除 CVS/.gitignore 等版本控制忽略文件
rsync -Cavze ssh . samba:backup   # 官方示例

# 不跨越文件系统边界（-x）
rsync -ax --delete / /backup/root/

# --delete-excluded：同时删除目标端被排除规则过滤掉的文件
rsync -av --exclude='*.tmp' --delete --delete-excluded /src/ /dest/

⚙️ 原理说明

rsync 的过滤规则按顺序优先级匹配（first match wins）：

  • 规则从上到下依次检查，第一条匹配的规则决定文件是否包含或排除
  • --include 和 --exclude 的顺序非常重要，通常先写 include 再写全局 exclude
  • -x / --one-file-system 根据官方手册："tells rsync to avoid crossing a filesystem boundary when recursing"，这是系统备份的关键参数，防止意外同步 /proc、/sys 等虚拟文件系统
⚡
场景⑧ 带宽限速与性能调优 Perf
生产环境中控制 rsync 对网络的影响
# 限制带宽为 1MB/s（单位：KB/s）
rsync -avz --bwlimit=1000 /src/ user@host:/dest/

# --whole-file：禁用 Delta 算法，整体传输
# 官方手册：当本地带宽远高于磁盘速度时更快
rsync -av --whole-file /src/ /nfs/dest/

# 压缩传输（-z），尤其对文本文件有效
rsync -avz /src/ user@host:/dest/

# Dry run：模拟执行，不实际传输（生产前必做）
rsync -avn /src/ user@host:/dest/

# -i (itemize)：显示每个文件的变更类型
rsync -aiv /src/ /dest/
# 输出格式：YXcstpoguax  (Y=更新类型, X=文件类型)
# 例：>f+++++++++ new_file.txt  (新文件)
# 例：.f..t...... unchanged.txt  (仅时间更新)

⚙️ 原理说明

何时应该禁用 Delta 算法（--whole-file）：官方手册说明："With this option the incremental rsync algorithm is not used and the whole file is sent as-is instead. The transfer may be faster if this option is used when the bandwidth between the source and destination machines is higher than the bandwidth to disk (especially when the 'disk' is actually a networked filesystem). This is the default when both the source and destination are specified as local paths."

即：当网络带宽 >> 磁盘 I/O 时（如 NFS），Delta 算法的计算和磁盘读写开销反而比直接传输更慢，此时关闭 Delta 更高效。本地复制时 rsync 默认就会自动关闭 Delta 算法。

10 常见陷阱与最佳实践

陷阱① 末尾斜线混淆

官方手册原文说明

"A trailing / on a source as meaning 'copy the contents of this directory' as opposed to 'copy the directory by name'"

# /dest/foo 下会有 foo 目录，即 /dest/foo/foo/...
rsync -av /src/foo /dest/foo   # ❌ 通常不是你想要的

# /dest/foo 下直接是内容，即 /dest/foo/file...
rsync -av /src/foo/ /dest/foo  # ✅ 正确

陷阱② --delete 误删

官方警告

必须始终先用 -n / --dry-run 模拟执行，确认将被删除的文件列表无误后，再去掉 -n 正式执行。

陷阱③ 数据一致性问题

根据 Michael Stapelberg 的分析：

官方协议行为

rsync 不保证数据集的一致性视图。文件列表生成后，如果文件内容发生变化，rsync 会传输它读取到的内容，可能是不一致的中间状态。对于 PostgreSQL 等数据库，不应直接使用 rsync 同步数据文件，应使用 pg_dump 等快照工具。

陷阱④ 不支持两端均为远程

# ❌ 不支持！rsync 不能在两个远程主机之间直接传输
rsync user@host1:/src/ user@host2:/dest/

# ✅ 变通方案：在本地中转
rsync user@host1:/src/ /tmp/transfer/
rsync /tmp/transfer/ user@host2:/dest/

# ✅ 或在 host1 上执行推送
ssh user@host1 rsync -av /src/ user@host2:/dest/

陷阱⑤ SSH 端口密钥问题

# rsync 默认用系统配置的 SSH，非标准端口需明确指定
rsync -av -e "ssh -p 2222 -i ~/.ssh/mykey" /src/ user@host:/dest/

# 也可在 ~/.ssh/config 配置，rsync 会自动使用

📌 生产环境最佳实践清单

  1. 首次执行新命令时，始终先加 -n dry-run
  2. 理解末尾斜线的语义差异，源目录是否加 / 效果截然不同
  3. 使用 --delete 前先用 --dry-run 确认删除列表
  4. 同步活跃数据库时，用 dump 工具生成一致性快照再 rsync
  5. 定期用 rsync --stats 评估实际传输效率
  6. 生产脚本中处理退出码：0=成功，23/24=部分传输（文件消失）

11 第一性原理总结

核心命题

rsync 解决的根本问题是：在受限带宽下，如何以最小代价使两个文件系统保持一致？

五个第一性原理

图6：rsync 六大第一性原理设计要素

① 最小传输原则

只传输两端的差异（delta），而非完整文件。这是 rsync 存在的根本理由。Delta 算法是这一原则的数学实现。

② 分层过滤原则

用廉价操作（mtime/size 比较）先淘汰大多数不需要处理的文件；用中间代价操作（rolling checksum）筛选可能匹配的块；用高代价操作（MD4 强校验）精确验证。永远不做超出必要的计算。

③ 流水线并发原则

三进程架构使 CPU、磁盘、网络同时满载，避免串行等待造成的资源浪费。这是所有高性能 I/O 系统的共同哲学。

④ 元数据优先原则

文件元数据（大小、时间戳）的比较成本远低于内容比较。Quick Check 默认用元数据判断，将完整 Delta 算法作为"最后手段"。

⑤ 关注点分离原则

rsync 的 Delta 算法与传输通道完全解耦——无论通过 SSH、RSH、TCP Daemon 还是本地文件系统，同样的算法都能工作。安全性交由 SSH，访问控制交由 rsyncd.conf，rsync 只专注于"高效同步"本身。

⑥ 原子性安全原则

发送端检测到 I/O 错误时，自动禁用目标端删除操作，防止临时故障（如 NFS 错误）触发大规模误删。默认通过临时文件 + rename 替换目标文件，保证写入的原子性。

📌 终极总结：rsync 的本质

rsync 是对"两个相似文件集合的最小差异同步"问题的一个完备工程解答。从第一性原理看，它回答了：

  • 传什么：只传差异（Delta 算法）
  • 怎么找差异：分块 + 滚动弱校验 + 强校验（双层过滤）
  • 传哪些文件：先用元数据 Quick Check 快速筛选
  • 怎么传：三进程流水线，充分利用硬件
  • 传输通道：可插拔（SSH/RSH/Daemon/Local）
  • 安全性：错误自动回退，原子写入

这六点共同构成了 rsync 自 1996 年至今仍是业界标准的根本原因：它是该问题的第一性最优解，不存在根本性的改进空间。
