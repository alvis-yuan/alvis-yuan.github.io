🧩 Chapter 00

什么是 tmux？

📄 官方文档: tmux/wiki/Getting-Started 📄 man7.org: tmux(1)

tmux（Terminal MUltipleXer，终端多路复用器）是一个在单一终端窗口中运行多个终端程序的工具。它由 Nicholas Marriott 开发，官方定义为：

"tmux is a terminal multiplexer: it enables a number of terminals to be created, accessed, and controlled from a single screen. tmux may be detached from a screen and continue running in the background, then later reattached." —— man tmux(1)

为什么需要 tmux？

以下是 tmux 官方文档列出的三大核心使用场景：

1
🛡️ 防止远程连接断开丢失工作

SSH 连接远程服务器时，若网络中断，所有进程会随之终止。tmux 让程序在后台持续运行，重连后可无缝恢复。

2
🖥️ 多设备共享同一会话

可以从家里、公司等不同计算机连接到同一个服务器会话，甚至多人实时协作共享同一终端。

3
📐 单终端内类窗口管理器

在一个终端中同时运行编辑器、编译器、Shell 等多个程序，自由切换，类似图形界面的窗口管理。

tmux 架构概览

图 0-1：tmux 客户端/服务端架构（来源：man7.org tmux(1)）
💡 核心机制
tmux 以 C/S 架构运行：tmux server 在后台持续存在，client 通过 /tmp 下的 socket 文件与之通信。即使所有 client 断开，server 和其中的 sessions 依然存活。
🔬 第一性原理

tmux 解决的本质问题是：进程生命周期与终端会话生命周期的解耦。

传统 shell 中，进程是终端（tty）的子进程，终端关闭则进程收到 SIGHUP 信号而终止。tmux 通过在 server 进程内维护 伪终端（pty），让程序的父进程变为 tmux server 而非用户的终端——从根本上切断了"网络断线 = 进程死亡"的强绑定。

常用启动命令

# 启动默认会话（名称为数字 0,1,2...）
tmux

# 指定会话名启动
tmux new -s mysession

# 后台启动（不进入终端）
tmux new -d -s background

# 启动并指定第一个窗口名称
tmux new -s dev -n editor

# 启动并运行特定命令（如 top）
tmux new -s monitor -n top top
💡 小贴士
运行 tmux new -As mysession 可实现：若 mysession 已存在则 attach，否则新建——非常适合写入 ~/.bashrc 作为自动恢复脚本。
🔬 第一性原理

前缀键设计源于信号与数据的命名空间隔离问题：所有按键默认被透传到 pty 中的程序，必须有一个"逃逸序列"来区分"给程序的输入"和"给 tmux 的指令"。前缀键就是这个逃逸符，其本质与终端 escape 字符（^[）的设计思想一致。

核心概念：Server · Session · Window · Pane

📄 官方文档: Basic Concepts

理解 tmux 的层级结构是掌握一切操作的基础。官方文档将其总结为四层嵌套关系：

图 2-1：tmux 四层嵌套结构（基于 github.com/tmux/tmux/wiki）

官方术语表

术语中文说明
Server服务端后台单一主进程，管理所有状态，通过 socket 通信
Client客户端连接到 Server 的终端实例（如 xterm、SSH 连接）
Session会话将多个 Window 组织在一起，有唯一名称
Window窗口占据整个终端面积，可包含多个 Pane，有 index 编号

Window Layout窗格布局Window 内所有 Pane 的尺寸与位置

层级关系一句话总结

📐 结构公式
Server ⊃ Session(s) ⊃ Window(s) ⊃ Pane(s)
一个 Server，多个 Session，每个 Session 多个 Window（有 index），每个 Window 多个 Pane。
🔬 第一性原理

tmux 的层级设计本质是命名空间的嵌套隔离：Session 提供"工作上下文"隔离（项目A/项目B），Window 提供"任务"隔离（编辑/运行/调试），Pane 提供"视图"的空间共存。每一层都是对"注意力"的管理粒度——越粗粒度的层越适合语义隔离，越细粒度的层越适合同时可见。

📋 Chapter 03

会话管理（Session）

📄 官方文档: Creating Sessions

场景：多项目并行开发

假设你同时维护三个项目：前端、API 后端、基础设施。用 Session 管理可以做到"一键切换工作上下文"：

# 分别创建三个项目会话（后台运行）
tmux new-session -d -s frontend
tmux new-session -d -s api-backend
tmux new-session -d -s infra

# 查看所有会话
tmux ls
# 输出：
# frontend: 1 windows (created Mon Jan 1 10:00:00 2026)
# api-backend: 1 windows (created Mon Jan 1 10:00:01 2026)
# infra: 1 windows (created Mon Jan 1 10:00:02 2026)

# 进入前端会话
tmux attach -t frontend

实战：SSH 断线恢复

1
SSH 登录服务器，创建命名会话 
ssh user@server
tmux new -s work
2
启动长时间任务，然后 detach 
# 在 tmux 内启动耗时任务
./build.sh && ./test.sh

# 分离（网络断线也等效于此）
# 按 Prefix + d
3
网络恢复后重新连接 
ssh user@server
tmux attach -t work
# 所有任务状态和输出完整保留！

attach-or-create 模式

在 ~/.bashrc 或 ~/.zshrc 中加入以下内容，SSH 登录后自动恢复工作会话：

# 若存在 work 会话则 attach，否则新建
tmux new-session -A -s work
📖 官方说明
来自官方文档：new-session -A 的 -A 标志会在同名会话已存在时 attach，不存在时新建。配合 -D 可踢掉其他客户端独占该会话。
🔬 第一性原理

Session 的本质是工作上下文的持久化单元。传统 shell 的上下文（工作目录、运行进程、历史输出）随 tty 关闭而消失。tmux Session 将这些状态序列化到 server 内存中，实现了上下文持久性——这与操作系统的进程持久化（daemon）思想同根同源。

🪟 Chapter 04

窗口管理（Window）

📄 官方文档: Creating new windows

Window 是 tmux 中的"标签页"，每个 Window 有一个 index 编号和可选名称，显示在状态栏的窗口列表中。

场景：开发环境的多标签管理

一个典型的开发会话包含：代码编辑、服务运行、数据库访问、日志查看等多个独立任务，用 Window 来组织：

# 在 dev 会话中创建各功能窗口
# 已在 dev 会话中...

# 新建窗口（自动命名为当前进程名）
# 按 Prefix + c

# 新建并指定名称
# 在命令提示符输入（Prefix + :）：
:neww -dn server

# 在 index 5 创建窗口
:neww -t 5

# 新建窗口并运行命令
:neww top

💡 专业技巧
按 Prefix w 打开交互式窗口/会话选择器，可用方向键浏览、Enter 切换，支持跨 Session 选择窗口——这是管理大量窗口最高效的方式。
🔬 第一性原理

Window 的 index 编号（而非名称）作为主要标识符，原因是序数寻址比名称寻址快捷——Prefix + 2 比输入窗口名称跳转快一个数量级。这是 tmux 将"频繁操作的成本最小化"这一工程哲学的体现。Window 名称仅用于人类可读标注，不作为系统标识符。

实战：开发调试四宫格

用命令行一次性搭建"编辑器 + 服务 + 测试 + 日志"四窗格布局：

# 在已有会话 dev 的窗口 0 中：

# 左右分割：左边编辑器
tmux split-window -h -t dev:0

# 对右侧再上下分割：右上服务
tmux split-window -v -t dev:0.1

# 对左侧再上下分割：左下日志
tmux split-window -v -t dev:0.0

# 在各 Pane 中发送命令
tmux send-keys -t dev:0.0 'vim .' Enter
tmux send-keys -t dev:0.1 'npm start' Enter
tmux send-keys -t dev:0.2 'npm test -- --watch' Enter
tmux send-keys -t dev:0.3 'tail -f app.log' Enter
⚠️ 注意
官方文档特别警告：synchronize-panes 选项开启后，所有 Pane 同步接收键盘输入，操作时需格外小心，避免误操作破坏性命令。
🔬 第一性原理

Pane 的本质是对 伪终端（PTY） 的空间复用。每个 Pane 都有独立的 master/slave PTY 对，程序看不到其他 Pane 的存在——这是操作系统级别的进程隔离。tmux 的工作仅是将多个 PTY 的输出绘制到屏幕的不同矩形区域，并将用户的键盘输入路由到活动 Pane 的 PTY master 端。分割 = 新建 PTY；关闭 = 销毁 PTY。

实战：配置 Vi 复制模式（推荐）

在 ~/.tmux.conf 中加入以下配置，实现类 vim 的复制体验：

# 使用 vi 模式
set -g mode-keys vi

# v 开始选择，y 复制（类 vim）
bind -T copy-mode-vi v send -X begin-selection
bind -T copy-mode-vi y send -X copy-selection-and-cancel

# 矩形选择（类 vim 块选择）
bind -T copy-mode-vi 'C-v' send -X rectangle-toggle
💡 系统剪贴板集成
在支持的终端中，可通过 set -g set-clipboard on 让 tmux 自动同步到系统剪贴板（X11 clipboard 或 macOS pasteboard）。Linux 下可配合 xclip 或 xsel；macOS 下配合 pbcopy。
🔬 第一性原理

复制模式的本质是：tmux server 为每个 Pane 维护了一个 滚动缓冲区（scrollback buffer），默认保存 2000 行历史输出（由 history-limit 控制）。进入复制模式后，tmux 冻结屏幕渲染到历史快照，并允许光标移动，本质上是对这块内存区域的只读随机访问。这完全绕开了终端模拟器自身的滚动机制，实现了跨终端的统一行为。

⚙️ Chapter 07

配置文件（~/.tmux.conf）

📄 官方文档: Common configuration changes 📄 man tmux(1): FILES

tmux 的配置文件默认路径为 ~/.tmux.conf（或 ~/.config/tmux/tmux.conf），在 server 首次启动时读取。配置文件本质上是一组 tmux 命令序列。

加载与重载

# 重新加载配置（不需要重启 tmux）
tmux source-file ~/.tmux.conf

# 在 tmux 内用快捷键 Prefix + : 输入命令
:source-file ~/.tmux.conf

# 绑定快捷键 Prefix + r 重载配置
bind r source-file ~/.tmux.conf \; display '配置已重载!'

完整推荐配置

以下配置均来自官方文档和 man page 中的示例：

# ═══════════════════════════════════════════
#  ~/.tmux.conf — 推荐入门配置
#  参考：https://github.com/tmux/tmux/wiki/Getting-Started
# ═══════════════════════════════════════════

# ── 前缀键：改为 C-a（更符合手指习惯）──
set -g prefix C-a
unbind C-b
bind C-a send-prefix   # 双击 C-a 传递给程序

# ── 索引从 1 开始（键盘布局更直觉）──
set -g base-index 1
set -g pane-base-index 1
set -g renumber-windows on  # 关闭窗口后自动重编号

# ── 性能与响应优化 ──
set -s escape-time 20        # Esc 响应时间 20ms
set -g history-limit 10000   # 历史缓冲 10000 行
set -g repeat-time 600       # 重复键超时 600ms

# ── 鼠标支持 ──
set -g mouse on

# ── Vi 键绑定 ──
set -g mode-keys vi
set -g status-keys vi

# ── 更直觉的分割键 ──
bind | split-window -h -c '#{pane_current_path}'
bind - split-window -v -c '#{pane_current_path}'
unbind '"'
unbind %

# ── Alt+方向键切换 Pane（无需前缀）──
bind -n M-Left  select-pane -L
bind -n M-Right select-pane -R
bind -n M-Up    select-pane -U
bind -n M-Down  select-pane -D

# ── Vi 复制模式 ──
bind -T copy-mode-vi v send -X begin-selection
bind -T copy-mode-vi y send -X copy-selection-and-cancel
bind -T copy-mode-vi C-v send -X rectangle-toggle

# ── 重载配置 ──
bind r source-file ~/.tmux.conf \; display '✅ 配置已重载'

# ── 真彩色支持 ──
set -g default-terminal 'tmux-256color'
set -sa terminal-overrides ',xterm-256color:RGB'

🔬 第一性原理

配置文件的本质是启动时批量执行的命令序列，与运行时 Prefix+: 交互式输入命令完全等价。tmux 不区分"配置"和"命令"——所有选项设置都通过 set-option，所有键绑定都通过 bind-key，这体现了 tmux "一切皆命令"的设计哲学，使配置可组合、可脚本化。

set -g status-left-length 40

颜色与样式

官方文档描述的颜色体系：支持标准色名、colour0-255（256色）和 #RRGGBB（真彩色）。

# 状态栏整体样式：深蓝背景
set -g status-style 'bg=colour235,fg=colour250'

# 当前窗口标签样式
set -g window-status-current-style 'fg=colour81,bold'
set -g window-status-current-format ' #I:#W '

# 活动 Pane 边框颜色
set -g pane-active-border-style 'fg=colour81'
set -g pane-border-style 'fg=colour238'

# 状态栏刷新间隔（秒）
set -g status-interval 5

# 状态栏移到顶部
set -g status-position top
🔬 第一性原理

状态栏的本质是一个以终端最后一行为载体的信息渲染引擎。Formats 系统将运行时状态（session/window/pane 属性）映射为字符串，#(cmd) 将外部进程输出集成进来，#[] 嵌入 ANSI 样式序列——三者叠加构成了一个轻量级的"状态仪表盘"。更新间隔由 status-interval 控制，本质是定时重新执行渲染，与网页的 polling 刷新机制等价。

🤖 Chapter 09

脚本自动化

📄 man tmux(1): send-keys, new-session, new-window

tmux 所有命令均可从 shell 调用，这使得一键复原完整工作环境成为可能。

send-keys 命令

官方 man page 中 send-keys 命令可以向指定 Pane 发送任意按键序列：

# 语法：tmux send-keys -t target 'text' Enter

# 向 dev 会话，窗口 0，Pane 0 发送命令
tmux send-keys -t dev:0.0 'vim .' Enter

# 向当前窗口的所有 Pane 广播（需先开启 synchronize-panes）
tmux set-window-option -t dev:0 synchronize-panes on

实战：一键启动开发环境脚本

#!/bin/bash
# dev-workspace.sh
# 使用：bash dev-workspace.sh
# 参考：tmux wiki & man tmux send-keys

SESSION="dev"
PROJECT="$HOME/projects/myapp"

# 已存在则先销毁
tmux kill-session -t "$SESSION" 2>/dev/null

# 创建主会话（不 attach）
tmux new-session -d -s "$SESSION" -n editor -c "$PROJECT"

# Window 1：编辑器
tmux send-keys -t "$SESSION:editor" 'vim .' Enter

# Window 2：后端服务
tmux new-window -t "$SESSION" -n server -c "$PROJECT"
tmux send-keys -t "$SESSION:server" 'python manage.py runserver' Enter

# Window 3：测试 + 日志（左右分割）
tmux new-window -t "$SESSION" -n tools -c "$PROJECT"
tmux split-window -h -t "$SESSION:tools"

# 左侧：测试
tmux send-keys -t "$SESSION:tools.0" 'pytest -v --tb=short' Enter

# 右侧：日志
tmux send-keys -t "$SESSION:tools.1" 'tail -f logs/app.log' Enter

# 回到编辑器窗口
tmux select-window -t "$SESSION:editor"

# 最终 attach 到会话
tmux attach-session -t "$SESSION"

Target 语法说明

# 示例：
# dev:editor.0   → dev会话, editor窗口, 第0个Pane
# dev:1.{right}  → dev会话, index 1窗口, 右侧Pane
# mysess:2       → mysess会话, index 2窗口
# {current}      → 当前 Pane（特殊写法）
💡 进阶：tmuxinator / tmux-resurrect
对于更复杂的会话自动化，社区工具 tmuxinator（Ruby YAML 配置）和 tmux-resurrect（会话持久化插件）可以进一步简化多项目管理——但其底层均基于本节介绍的原生 tmux 命令。
🔬 第一性原理

脚本自动化的核心是 send-keys 实现了键盘输入的程序化替代：它将字符串写入目标 PTY 的输入缓冲区，与人手动敲键完全等价。配合 new-session/new-window/split-window，整个会话拓扑结构（Session→Window→Pane 树）可以通过声明式脚本精确重建——本质是基础设施即代码（IaC）在终端环境的应用。

SSH 持久会话与多端协作

📄 man tmux(1): attach-session, -d flag 📄 官方文档: Attaching and detaching

场景一：防止 SSH 断线

图 10-1：SSH 断线后 tmux 进程持续运行，重连即可恢复

场景二：多人协作共享会话

tmux 允许多个 client 同时 attach 到同一 session，实现实时协作或远程指导：

# 服务器端：创建共享会话
tmux new-session -s pair-programming

# 协作者 SSH 连接到同一服务器后：
tmux attach -t pair-programming
# 现在两人看到同样的屏幕，输入都有效！

# 踢掉其他 client（独占）
tmux attach -d -t pair-programming

场景三：多设备不同尺寸适配

当多个 client 尺寸不同时，tmux 默认以最小尺寸显示（避免信息不一致）：

# 查看所有 client
tmux list-clients

# 选择要断开的 client（Prefix + D 交互选择）
# 或命令行指定
tmux detach-client -t /dev/pts/1

最佳实践：开机/登录自动恢复

# 在 ~/.bashrc 或 ~/.zshrc 末尾添加：
# 若当前在 SSH 连接中且 tmux 未运行，则自动 attach 或新建
if [ -n "$SSH_CONNECTION" ] && [ -z "$TMUX" ]; then
  tmux new-session -A -s main
fi
⚠️ 安全提示
共享 tmux Session 意味着对方拥有与你完全相同的权限。在生产服务器上谨慎使用多人共享会话，优先考虑创建只读观察者模式或使用单独用户账号。
🔬 第一性原理

SSH + tmux 的组合解决了网络连接的不稳定性与进程生命周期的稳定性需求之间的矛盾。SSH 是面向流的传输层协议（TCP），任何中断都意味着会话终止（SIGHUP）。tmux 通过将进程托管在服务器本地的 daemon 进程中，完全切断这一依赖——网络连接仅作为"渲染通道"，随时可接、随时可断，核心状态不受影响。

⌨️ Chapter 11

键绑定进阶

📄 man tmux(1): bind-key, KEY TABLES 📄 官方文档: vi key bindings

bind-key 语法

# 完整语法
bind-key [-nr] [-T key-table] key command [args]

# -n：不需要前缀键（直接触发）
# -r：标记为可重复键（在 repeat-time 内无需再按前缀）
# -T：指定键表（prefix/root/copy-mode-vi 等）

# 示例：绑定 Prefix+r 重载配置
bind r source-file ~/.tmux.conf \; display 'Reloaded!'

# 示例：不用前缀键，Alt+h/j/k/l 切换 Pane
bind -n M-h select-pane -L
bind -n M-j select-pane -D
bind -n M-k select-pane -U
bind -n M-l select-pane -R

# 可重复键：Prefix+方向键 调整大小，重复按方向键
bind -r H resize-pane -L 5
bind -r J resize-pane -D 5
bind -r K resize-pane -U 5
bind -r L resize-pane -R 5

键表（Key Tables）

官方 man page 定义了多个键表，不同上下文使用不同键表：

键表名触发条件用途
prefix按下前缀键后默认 tmux 命令键
root任意时刻（-n 绑定）无前缀的全局快捷键
copy-modeEmacs 复制模式中复制/搜索操作
copy-mode-viVi 复制模式中Vi 风格复制/搜索

实用进阶绑定

# 快速 SSH 到指定主机（命令提示符）
bind S command-prompt -p "SSH to:" "new-window -n %1 'ssh %1'"

# 快速打开 man 手册页
bind / command-prompt "split-window 'exec man %%'"

# 同步/取消同步所有 Pane 的输入
bind y set-window-option synchronize-panes \; \
    display-message "同步输入: #{?synchronize-panes,on,off}"

# 在新窗口中运行 htop
bind t new-window -n htop htop

# 在新 Pane 中打开当前目录（保持路径）
bind '"' split-window -v -c '#{pane_current_path}'
bind % split-window -h -c '#{pane_current_path}'

# 查看并列出所有键绑定
# Prefix + ?  （或 tmux lsk -N | more）

查看与删除绑定

# 列出所有键绑定（带描述）
tmux list-keys -N

# 列出指定键表的绑定
tmux list-keys -T copy-mode-vi

# 删除绑定
tmux unbind-key -T prefix Space
🔬 第一性原理

键绑定系统的本质是有限状态机 + 事件路由器：按键事件根据当前所处的"键表状态"（正常/前缀按下后/复制模式中）路由到不同的命令处理器。-T 参数就是状态机的"当前状态"选择符。前缀键本质上是一个状态切换触发器（root → prefix 状态），让同一按键在不同上下文表达不同语义——这与 Vim 的模态编辑哲学完全一致。

🔬 综合第一性原理回顾

tmux 的所有功能都围绕一个核心公理："进程状态的持久化与访问的灵活性"。

• Server/Client 分离 → 解耦进程生命周期与终端连接

• Session/Window/Pane 层级 → 解决注意力的命名空间管理

• PTY 复用 → 以最小系统代价实现多程序并存

• 键绑定系统 → 以最小击键数实现最大操作覆盖

• 脚本化接口 → 将人工操作转化为可重复的自动化流程

掌握 tmux 的本质是掌握如何在有限的终端界面内最大化工作流效率。
