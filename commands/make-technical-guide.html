📖 第一章

GNU Make 概述

自动化构建工具的设计哲学与核心能力

🔗 官方文档 §1 Overview of make
💡Make 是什么？

make 实用程序能自动判断一个大型程序中哪些部分需要重新编译，并发出命令来执行重新编译。

虽然示例通常展示 C 程序，但 make 不限于任何特定语言或任务——只要某些文件需要根据其他文件的变化而自动更新，就可以使用 make。

🔑核心工作原理（一句话）
📌
make 读取 Makefile 构建依赖图，通过对比文件的最后修改时间，决定哪些目标需要重建，然后按序执行相应的命令（Recipe）。
——来源：GNU Make Manual §1 Preparing and Running Make
🗺️整体工作流程图
📚规则的基本语法

来源：GNU Make Manual §2.1 What a Rule Looks Like

# 规则基本格式 目标(target) … : 前置条件(prerequisites) … 配方(recipe) … # 示例：编译 C 程序 edit : main.o kbd.o command.o display.o cc -o edit main.o kbd.o command.o display.o main.o : main.c defs.h cc -c main.c clean : rm -f edit *.o
⚠️
每个配方行必须以 Tab 字符开头，而不是空格。这是 make 区分配方行与其他指令的方式。可通过设置 .RECIPEPREFIX 变量更改此默认字符。

⚡ 第一性原理解析

make 的本质是一个时间戳驱动的依赖推断引擎。它解决的核心问题是：在一组相互依赖的文件中，当部分文件发生变化时，如何以最小的计算代价重建受影响的输出。这与函数式编程中的"增量计算"本质相同——只重新计算依赖发生变化的结果，而不是全量重算。Makefile 就是这个依赖函数图的声明式描述，make 本身是图的执行引擎。

组成部分作用示例

📎include 指令与 MAKEFILES 变量

来源：GNU Make Manual §3.3 Including Other Makefiles / §3.4 The Variable MAKEFILES

include 指令

⚠️
MAKEFILES 变量（环境）：若设置，make 会将其值视为额外的 makefile 名称列表，在处理其他 makefile 前先读入。官方文档不推荐使用此功能，因为它会导致不同用户环境的行为差异。
🗂️include 搜索路径

来源：GNU Make Manual §3.3 Including Other Makefiles

当被 include 的文件名不以 / 开头，且在当前目录未找到时，make 按以下顺序搜索：

  1. 通过 -I 或 --include-dir 选项指定的目录
  2. prefix/include（通常为 /usr/local/include）
  3. /usr/gnu/include
  4. /usr/local/include
  5. /usr/include

⚡ 第一性原理解析

配置文件查找顺序的设计逻辑是兼容性优先、明确性优先。GNUmakefile 放在最前表示"我明确声明我只用 GNU make"；Makefile（大写）之所以被推荐，是因为大写字母在目录列表中靠前显示，便于人类发现。include 机制实现了关注点分离——把公共规则、平台适配、依赖声明等拆分到多个文件中，本质上是模块化思想在构建系统中的体现。

🌍 第三章

环境变量

shell 环境如何影响 make 的行为

🔗 官方文档 §6.10 Variables from the Environment
📥环境变量的导入机制

来源：GNU Make Manual §6.10 Variables from the Environment

📌
make 启动时，会将当前 shell 中所有的环境变量转换为同名的 make 变量，默认值与环境变量相同。但 Makefile 中的显式赋值或命令行参数会覆盖环境变量值。
——GNU Make Manual §6.10
📊重要的 make 专属环境变量

来源：GNU Make Manual §3.4, §5.7, §6.14

变量名类型作用
MAKEFILES 环境 make 启动时额外读入的 makefile 列表（空格分隔）。官方不推荐依赖此变量
MAKE 特殊 make 程序自身路径，用于递归调用 make 时保持一致性
MAKELEVEL 特殊 当前 make 的递归深度，顶层为 0，每递归一层加 1
MAKEFLAGS 特殊 传递给子 make 的命令行选项字符串
SHELL 特殊 执行配方使用的 shell，默认 /bin/sh。make 特意忽略同名环境变量，避免用户的交互式 shell 影响构建
PATH 环境 shell 搜索可执行文件的路径，影响配方中命令的查找
CFLAGS 环境 C 编译器额外参数，如设置在 shell 环境中可影响大多数 makefile 的编译行为
📤export：向子 make 传递变量

来源：GNU Make Manual §5.7.2 Communicating Variables to a Sub-make

# 将变量传递给递归调用的子 make export BUILD_DIR = /tmp/build # 禁止传递某变量 unexport PRIVATE_VAR # 导出所有变量（慎用） export all: $(MAKE) -C subdir # 子 make 能读到 BUILD_DIR
✅
默认情况下，只有来自环境或命令行的变量会自动传入子 make 的环境。Makefile 内部定义的变量需要显式 export。
⚡MAKE_TERMOUT / MAKE_TERMERR

来源：GNU Make Manual §6.14 Other Special Variables

make 启动时检查 stdout/stderr 是否连接终端，若是则设置 MAKE_TERMOUT/MAKE_TERMERR 为终端设备名（或 true）。这可用于让配方在终端下输出彩色，在重定向时禁用颜色。

⚡ 第一性原理解析

环境变量的引入解决了配置复用问题：不修改 Makefile 就能改变构建行为。但官方文档明确警告不要过度依赖环境变量，因为这违背了 make 的核心价值——确定性与可重复性。不同用户相同 Makefile 应产生相同输出；而依赖环境变量则打破了这一保证。SHELL 变量被特意忽略正是这一哲学的体现：构建环境必须与用户的交互式偏好隔离。

🔬内置变量在隐式规则中的使用示例

来源：GNU Make Manual §10.3（内置隐式规则中的实际命令）

# C 编译的内置隐式规则实际等价于： %.o: %.c $(CC) -c $(CFLAGS) $(CPPFLAGS) -o $@ $< # C++ 编译的内置隐式规则实际等价于： %.o: %.cpp $(CXX) -c $(CXXFLAGS) $(CPPFLAGS) -o $@ $< # 修改这些变量即可定制编译行为，无需重写规则 CC = clang CFLAGS = -Wall -Wextra -O2 -std=c11 CPPFLAGS = -I./include -DDEBUG=1

⚡ 第一性原理解析

内置变量体现了开放/封闭原则：隐式规则的行为逻辑是封闭的（内置于 make），但其配置参数是开放的（通过变量暴露）。用户只需修改 CC=clang 或 CFLAGS=-O3，就能驱动整个内置规则体系切换工具链或优化级别，而无需了解规则的内部实现。这是策略与机制分离的经典设计。

🤖 第五章

自动变量

每次规则执行时动态计算的上下文变量

🔗 官方文档 §10.5.3 Automatic Variables
💡什么是自动变量？
📌
自动变量的值在每次规则执行时重新计算，基于该规则的目标和前置条件。它们只在配方（recipe）中有效，不能在目标列表或前置条件列表中使用。
——GNU Make Manual §10.5.3
📊核心自动变量速查
变量含义使用场景
$@当前规则的目标文件名输出文件名，如 -o $@
$<第一个前置条件的名称隐式规则中的源文件，如 $(CC) -c $<
$^所有前置条件（去重，含空格分隔）链接时传入所有 .o 文件
$?比目标更新的所有前置条件增量更新 archive：ar r lib $?
$+所有前置条件（含重复）需要按序重复传库名的链接命令
$|所有仅顺序前置条件目录创建等无时间戳依赖
$*模式规则匹配的茎（stem）构造相关文件名，如 $*.h
$%archive 成员名处理 lib.a(foo.o) 格式目标
📁目录/文件名变体（D 和 F 后缀）

来源：GNU Make Manual §10.5.3（变体列表）

每个单文件名自动变量都有 D（目录部分）和 F（文件名部分）两个变体：

# 若目标为 src/main.o $(@D) → src # 目录部分（无尾部斜杠） $(@F) → main.o # 文件名部分，等价于 $(notdir $@) $(<D) → 源文件目录 $(<F) → 源文件名 $(^D) → 所有前置条件目录列表 $(^F) → 所有前置条件文件名列表

⚡ 第一性原理解析

自动变量解决的是参数化规则问题：同一个模式规则需要应用到无数不同的文件，规则体内不能硬编码文件名。自动变量本质上是对"当前规则执行上下文"的动态绑定——在执行每条规则时，make 自动将目标和前置条件绑定到这些保留名称。这是元编程的一种形式：规则描述的是变换模式，而非具体操作。

🔄 第六章

执行阶段

make 内部分为两个截然不同的阶段

🔗 官方文档 §3.7 How make Reads a Makefile
📌两阶段概览
📌
GNU make 以两个截然不同的阶段完成工作：第一阶段读入所有 makefile 并内化变量、规则，构建依赖图；第二阶段利用这些内化数据决定哪些目标需要更新，并运行相应配方。
——GNU Make Manual §3.7 How make Reads a Makefile
🔄详细阶段流程
1
📖 第一阶段：读入阶段（Read-in Phase）

make 读取所有 makefile（包括 include 的文件）。此阶段进行立即展开（Immediate Expansion）。

操作展开时机
VAR := value（简单赋值）✅ 立即展开（Immediate）
VAR = value（递归赋值）⏳ 延迟展开（Deferred）
VAR ?= value（条件赋值）⏳ 延迟展开（Deferred）
规则的目标和前置条件✅ 立即展开（Immediate）
规则的配方（recipe）⏳ 延迟展开（Deferred）
条件指令（ifdef/ifeq 等）✅ 立即展开（Immediate）
📥 输入
  • GNUmakefile / makefile / Makefile
  • 被 include 的所有子 makefile
  • Shell 环境变量
  • 命令行参数（变量、目标）
  • make 内置隐式规则库
📤 输出（内部数据结构）
  • 变量数据库（Variable Database）
  • 显式规则列表
  • 模式规则列表
  • 目标依赖关系图（DAG）
  • 默认目标确定
2
⚡ 第二阶段：目标更新阶段（Target-Update Phase）

make 遍历依赖图，对比文件时间戳，决定哪些目标需要重建，展开并执行配方。

步骤描述
1. 确定构建目标命令行指定目标，或使用第一个非特殊目标作为默认目标
2. 递归解析依赖深度优先遍历 DAG，先确保前置条件是最新的
3. 时间戳比较若前置条件比目标更新（或目标不存在），则目标需重建
4. 展开配方此时才展开配方中的延迟变量（包括自动变量）
5. 执行配方将每行配方传给 shell 执行（默认每行一个子 shell）
📥 输入
  • 第一阶段产生的依赖图
  • 变量数据库
  • 文件系统中各文件的时间戳
  • 目标列表（命令行或默认）
📤 输出
  • 已更新的目标文件（.o、可执行等）
  • 配方执行的副作用（安装、清理等）
  • 退出码（0=成功，非0=失败）
⚙
🐚 配方执行细节（Shell Execution）

默认情况下，配方的每一行由独立的子 shell 执行。

# 每行独立 shell：变量不共享 bad-example: cd /tmp # 子 shell 1：cd 有效 ls # 子 shell 2：仍在原目录！ # 正确做法：用 && 或 ; 合并 good-example: cd /tmp && ls # 或开启 .ONESHELL（GNU make 3.82+） .ONESHELL: oneshell-example: cd /tmp ls # 有效，同一 shell
⏱️立即展开 vs 延迟展开

来源：GNU Make Manual §3.7（Variable Assignment 小节）

⚡ 第一性原理解析

两阶段设计源于一个根本约束：要执行一个配方，必须先知道其所有依赖是否最新；要知道依赖是否最新，必须先有完整的依赖图。因此，"构建图"与"执行图"必须分离为两个不可交叉的阶段。立即/延迟展开的区别则来自于此：构图时需要确定结构（目标/前置条件立即），执行时才知道上下文（配方延迟）。理解这一点可以解释大量 make 中令人困惑的行为——比如为什么自动变量不能出现在前置条件列表中。

🌳 第七章

依赖图与规则类型

make 如何构建和遍历依赖关系

🔗 官方文档 §4 Writing Rules
🌳依赖图（DAG）示例

来源：GNU Make Manual §2.3 How make Processes a Makefile

箭头表示"依赖于"关系：make 从目标出发，递归向下检查所有前置条件。

📋规则类型
规则类型语法示例特点
显式规则 main.o: main.c defs.h 明确指定目标及前置条件，优先于隐式规则
模式规则（隐式） %.o: %.c 使用 % 通配符，匹配一类文件
静态模式规则 $(OBJS): %.o: %.c 限定目标集合的模式规则，比普通模式规则更精确
Phony 目标 .PHONY: clean all 非文件目标，总是执行，不做时间戳比较
双冒号规则 target:: dep1 同一目标的多个独立规则，各自独立执行
⏩前置条件类型（普通 vs 仅顺序）

来源：GNU Make Manual §4.3 Types of Prerequisites

# 普通前置条件（Normal Prerequisites） # 若 dep 比 target 更新 → 重建 target target: dep1 dep2 # 仅顺序前置条件（Order-only Prerequisites）| 后 # 只要求 dep 已存在，不触发重建，用于创建目录等 target: dep1 | mkdir-first $(OBJS): | build_dir # 确保 build_dir 先创建 build_dir: mkdir -p build

⚡ 第一性原理解析

依赖图（DAG）是 make 的数据核心。有向无环图保证了依赖关系有明确的拓扑顺序，不会陷入循环。make 对图的遍历本质上是后序遍历（Post-order Traversal）：先递归确保所有子节点（前置条件）是最新的，再处理当前节点（目标）。时间戳比较是一种增量计算的近似手段——如果前置条件的修改时间晚于目标，则认为目标需要重建。这是以牺牲极少数边界情况（如时钟错误）换取高效增量构建的工程妥协。

📐 第八章

隐式规则

make 内置的推断规则与搜索算法

🔗 官方文档 §10 Using Implicit Rules
💡什么是隐式规则？
📌
隐式规则告诉 make 如何根据文件名模式推断出构建方法，无需为每个文件显式编写规则。例如，当 make 看到需要 foo.o 时，它自动知道可以从 foo.c 编译得到。
📋常用内置隐式规则

来源：GNU Make Manual §10.2 Catalogue of Built-In Rules

规则从到实际命令
编译 C.c.o$(CC) -c $(CFLAGS) $(CPPFLAGS)
编译 C++.cpp/.cc.o$(CXX) -c $(CXXFLAGS) $(CPPFLAGS)
汇编.s.o$(AS) $(ASFLAGS)
链接.o可执行$(CC) $(LDFLAGS) ... $(LDLIBS)
Lex.l.c$(LEX) $(LFLAGS)
Yacc.y.c$(YACC) $(YFLAGS)
🔍隐式规则搜索算法

来源：GNU Make Manual §10.8 Implicit Rule Search Algorithm

1
分解目标：将目标名分为目录部分 d 和文件名部分 n
2
找候选规则：列出所有目标模式能与 n 匹配的模式规则
3
验证前置条件：检查对应前置条件文件是否存在或可被构建
4
选择并应用：使用第一个满足条件的规则；若无匹配规则，报错
⛔禁用隐式规则
# 禁用所有内置隐式规则 make -r # 或 --no-builtin-rules # 禁用所有内置变量 make -R # 或 --no-builtin-variables # 在 Makefile 中取消特定后缀规则 .SUFFIXES: # 清空所有后缀规则 .SUFFIXES: .c .o # 只保留 .c → .o

⚡ 第一性原理解析

隐式规则是 make 对约定优于配置（Convention Over Configuration）理念的实现。通过观察到"大多数 C 文件都用 cc -c 编译"这一事实，make 将其内置为默认行为，让用户只需声明异常情况。隐式规则的搜索算法是一个模式匹配引擎，本质上是在从当前文件系统状态到目标状态的转换图中寻找可行路径。-r 选项则允许用户完全掌控构建——在安全性或可移植性要求极高的场景下，明确胜于约定。

🔗 官方文档 §16 Makefile Conventions
📂文件职责全景
📋Makefile 规范标准目标

来源：GNU Make Manual §16.6 Standard Targets for Users

🔄自动依赖生成（.d 文件）

来源：GNU Make Manual §4.14 Generating Prerequisites Automatically

# 使用 -MMD -MP 让编译器自动生成 .d 文件 CFLAGS += -MMD -MP SRCS := main.c kbd.c command.c OBJS := $(SRCS:.c=.o) DEPS := $(SRCS:.c=.d) all: edit edit: $(OBJS) $(CC) $(LDFLAGS) -o $@ $^ $(LDLIBS) # 包含自动生成的依赖文件（-include 忽略不存在的） -include $(DEPS)
✅
.d 文件由编译器自动生成，用户只需在 Makefile 中 -include *.d 即可。这些文件记录了每个 .c 文件依赖的所有头文件，使 make 能在头文件变化时精确触发重编译。
📌完整的现代 Makefile 骨架
# ── 工具链变量（内置变量，用户可覆盖）── CC ?= gcc CFLAGS += -Wall -Wextra -std=c11 LDLIBS += -lm # ── 目录结构 ── SRC_DIR := src BUILD := build # ── 文件列表（自动发现）── SRCS := $(wildcard $(SRC_DIR)/*.c) OBJS := $(patsubst $(SRC_DIR)/%.c,$(BUILD)/%.o,$(SRCS)) DEPS := $(OBJS:.o=.d) TARGET := myapp # ── 默认目标 ── .PHONY: all clean all: $(TARGET) $(TARGET): $(OBJS) $(CC) $(LDFLAGS) -o $@ $^ $(LDLIBS) $(BUILD)/%.o: $(SRC_DIR)/%.c | $(BUILD) $(CC) $(CFLAGS) $(CPPFLAGS) -MMD -MP -c -o $@ $< $(BUILD): mkdir -p $@ clean: $(RM) -r $(BUILD) $(TARGET) -include $(DEPS)

⚡ 第一性原理解析

用户维护的 Makefile 本质上是在做声明式编程：你声明"edit 依赖于这些 .o"、"main.o 依赖于 main.c"，而不是命令式地告诉计算机"先编译 main.c，再编译 kbd.c……"。这种声明式风格让 make 可以自行决定并行度（-j）和执行顺序，因为它拥有完整的依赖信息。用户只需维护"是什么"，make 负责"怎么做"。

变量优先级与赋值类型

变量来源的优先顺序与不同赋值运算符

🔗 官方文档 §6.4 How Variables Get Their Values
🏆变量优先级（高→低）

来源：GNU Make Manual §6.4 / §6.7 / §9.5

  1. 命令行赋值
    make CC=clang CFLAGS="-O3" — 覆盖 Makefile 中的赋值（除非用 override）
  2. override 指令
    override CFLAGS += -g — 即使命令行指定了值，也会生效
  3. Makefile 中的显式赋值
    CC = gcc — 覆盖环境变量（无 -e 时）
  4. 环境变量（-e 标志启用时提升）
    来自 shell 的环境变量，默认被 Makefile 赋值覆盖；指定 -e 后提升到第3级
  5. 内置默认值
    CC=cc, CFLAGS= 等内置初始值，优先级最低
⚙️赋值运算符详解

来源：GNU Make Manual §6.2 The Two Flavors of Variables / §3.7

运算符名称展开时机特点
= 递归展开赋值 引用时（Deferred） 每次引用都重新展开，可引用后定义的变量
:= / ::= 简单展开赋值 赋值时（Immediate） 立即展开并固化，高效，不能引用后定义变量
?= 条件赋值 引用时（Deferred） 仅在变量未定义时赋值，已有值则跳过
+= 追加赋值 依上下文 在现有值后追加，类型继承原变量风格
!= Shell 赋值 赋值时（Immediate） 右侧传给 shell 执行，结果存入变量
:::= 立即展开并转义 赋值时（Immediate） 展开后将 $ 替换为 $$，避免二次展开
# = 递归展开：引用时才求值 X = hello A = $(X) world # A 尚未求值 X = hi # $(A) → "hi world"（使用最新的 X） # := 立即展开：赋值时固化 X := hello B := $(X) world # B 立即求值 = "hello world" X := hi # $(B) → "hello world"（X 的改变不影响 B） # ?= 条件赋值：已有值时不覆盖 CC ?= gcc # 若 CC 未定义则设为 gcc # += 追加 CFLAGS := -Wall CFLAGS += -O2 # → "-Wall -O2"
🎯目标特定变量（Target-specific Variables）

来源：GNU Make Manual §6.11 Target-specific Variable Values

# 仅对特定目标生效的变量 debug: CFLAGS += -g -DDEBUG debug: myapp release: CFLAGS += -O3 -DNDEBUG release: myapp # 对匹配模式的目标生效 %.test.o: CFLAGS += -fprofile-arcs CFLAGS = -Wall # 全局值
📌
目标特定变量在该目标及其所有前置条件的配方中有效，是全局变量的独立副本，不影响全局值。

⚡ 第一性原理解析

变量优先级的设计遵循最小惊讶原则：越靠近"调用时刻"的赋值，优先级越高。命令行参数是用户的即时意图，理应覆盖文件中的静态配置；文件中的显式赋值代表项目的默认决策，理应覆盖外部环境的隐式设置；内置默认值只是最后的兜底。override 的存在则是为了让 Makefile 作者能够声明某些变量不可被外部覆盖，用于保护安全关键的构建选项。= 与 := 的区别本质是惰性求值 vs 急性求值——两种编程范式在构建语言中的体现。
