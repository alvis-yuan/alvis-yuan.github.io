Chapter 0

什么是 Autotools（GNU 构建系统）

📖 GNU Autoconf Manual 2.73 (2026)

Autotools（GNU Build System）是一套由 GNU 项目维护的工具链，用于使 C/C++ 等软件包能够在几乎所有类 POSIX 系统上自动化配置和编译。 它由三个核心工具组成：

⚙️ Autoconf

将 configure.ac 转换成可移植的 configure shell 脚本，在用户系统上探测编译器、头文件、库的存在性。

📝 Automake

将 Makefile.am 转换为 Makefile.in 模板，由 configure 最终生成 Makefile。

🔗 Libtool

抽象不同平台的共享库差异（.so / .dylib / .dll），让库的构建和安装跨平台可移植。

以 curl 为例：curl 源码仓库（github.com/curl/curl）使用完整的 Autotools 体系。 从 Git 克隆后，开发者需要运行 autoreconf -fi 来生成 configure；而普通用户下载发行 tarball 时，configure 已经预生成，只需执行三步经典咒语： 

# 来源：curl/GIT-INFO.md（官方仓库）
autoreconf -fi           # 仅 Git 克隆需要，发行包不需要
./configure --with-openssl
make
make install

Autoconf 的核心设计哲学来自官方手册： "The primary goal of Autoconf is making the user's life easier; making the maintainer's life easier is only a secondary goal." — GNU Autoconf Manual

🔬 第一性原理

核心问题：同一份 C 代码，如何在 Linux、macOS、BSD、Solaris 等数百种环境下都能编译？

根本矛盾：不同系统的头文件路径、库名称、编译器行为各不相同。写死路径必然失败。

最简解：在用户的机器上运行探测脚本，动态发现当前环境的特征，再据此生成 Makefile。Autotools 就是这套探测-生成机制的工业级实现。 configure.ac 声明"我需要什么"，configure 脚本在用户机器上回答"这里有没有"。

💡 关键原则：开发者只需维护 configure.ac、Makefile.am 和自定义 m4/*.m4 宏文件，其余文件均由工具链自动生成。发行 tarball（make dist 生成）会包含预生成的 configure、Makefile.in 等，使终端用户无需安装 Autotools。
🔬 第一性原理

版本控制系统的基本原则：只追踪无法从其他文件重新生成的文件。 configure 完全可以由 configure.ac 重新生成，因此 Git 仓库不提交 configure。 但发行 tarball 需要包含 configure，因为用户机器上没有 Autotools。 这就是两套文件集合（Git vs tarball）存在差异的根本原因。

Chapter 2

configure.ac 详解

📖 GNU Automake — configure.ac Explained 📖 curl configure.ac 实际源码

configure.ac 是 Autotools 体系中最核心的手写文件。它用 M4 宏语言编写，描述了软件包的编译需求：需要什么编译器、哪些库、哪些头文件，以及最终要生成哪些输出文件。 autoconf 工具读取它，展开所有宏，输出一个可移植的 shell 脚本 configure。

标准结构布局

根据 GNU Autoconf 手册，configure.ac 的标准结构从上到下依次为： 

dnl  Process this file with autoconf to produce a configure script.
dnl  curl/configure.ac — 简化演示版本

# ① 初始化 Autoconf
AC_PREREQ([2.59])
AC_INIT([curl], [-], [a suitable curl mailing list: https://curl.se/mail/])
AC_CONFIG_SRCDIR([lib/urldata.h])   # 验证源码目录合法性
AC_CONFIG_HEADERS([lib/curl_config.h])  # 声明 C 头文件输出
AC_CONFIG_MACRO_DIR([m4])            # 自定义宏所在目录

# ② 初始化 Automake
AM_INIT_AUTOMAKE
AM_MAINTAINER_MODE

# ③ curl 自定义选项检测
CURL_CHECK_OPTION_DEBUG
CURL_CHECK_OPTION_OPTIMIZE
CURL_CHECK_OPTION_WARNINGS

# ④ 检测编译器
AC_PROG_CC
AC_PROG_CPP

# ⑤ 检测库和头文件
AC_CHECK_LIB([z], [gzopen])
AC_CHECK_HEADERS([sys/socket.h])
CURL_CHECK_LIB_OPENSSL             # curl 自定义宏

# ⑥ 类型/结构检测
AC_TYPE_SIZE_T
AC_TYPE_SSIZE_T

# ⑦ 声明输出文件
AC_CONFIG_FILES([
  Makefile
  src/Makefile
  lib/Makefile
  libcurl.pc
])
AC_CONFIG_FILES([curl-config], [chmod +x curl-config])

# ⑧ 生成所有输出
AC_OUTPUT

curl 特有的自定义 M4 宏

curl 项目在 m4/ 目录下维护了大量自定义宏（如 CURL_CHECK_LIB_OPENSSL、CURL_CHECK_OPTION_DEBUG 等）， 这些宏定义在 m4/*.m4 文件中，由 aclocal 收集后生成 aclocal.m4，再供 autoconf 使用。

AC_OUTPUT
🔬 第一性原理

configure.ac 的本质：一份用 M4 宏语言写的"需求声明"。 它回答的问题是："我这个软件需要哪些外部条件，以及当这些条件满足/不满足时，该设置什么变量？"

M4 宏展开后得到 shell 代码，运行这段 shell 代码就是在用户机器上做探测。 AC_CHECK_LIB([openssl], [SSL_connect]) 展开后的 shell 代码会尝试编译并链接一个使用 SSL_connect 的小程序——成功则记录 HAVE_OPENSSL，失败则不记录。这就是可移植性的底层机制。

Chapter 3

Makefile.am 详解

📖 GNU Automake — Makefile.am Explained

Makefile.am 是 Automake 的输入文件，语法与 Makefile 相同，但特殊变量会被 Automake 解析、展开成完整的 Makefile.in。 它解决的问题是："需要编译什么，生成什么目标，安装到哪里？"

变量命名规则：目录前缀 + 主类型

Automake 变量遵循 [目录前缀]_[主类型] 的模式：

bin_PROGRAMS     = curl        # 安装到 $bindir 的可执行程序
lib_LTLIBRARIES  = libcurl.la  # 安装到 $libdir 的 Libtool 库
include_HEADERS  = curl.h      # 安装到 $includedir 的头文件
noinst_LIBRARIES = libfoo.a    # 仅内部使用，不安装
pkgconfig_DATA   = libcurl.pc  # 安装到 $pkgconfigdir 的数据文件
SUBDIRS          = lib src docs # 递归构建的子目录（顺序有意义！）
目录前缀安装路径变量含义
bin$bindir用户可执行文件 (默认 /usr/local/bin)
sbin$sbindir系统管理员可执行文件
lib$libdir库文件 (默认 /usr/local/lib)
include$includedir头文件 (默认 /usr/local/include)
pkgconfig$pkgconfigdirpkg-config .pc 文件
noinst不安装仅在构建过程中使用
check不安装仅在 make check 时构建

主类型（Primaries）

主类型含义示例
_PROGRAMS可执行程序（需编译链接）bin_PROGRAMS = curl
_LTLIBRARIESLibtool 共享/静态库lib_LTLIBRARIES = libcurl.la
_LIBRARIES静态库（ar 打包）noinst_LIBRARIES = libinternal.a
_HEADERS头文件include_HEADERS = curl.h
_DATA任意数据文件pkgconfig_DATA = libcurl.pc
_SCRIPTS可执行脚本bin_SCRIPTS = curl-config

目标特定变量

每个构建目标可以配套一组变量，以目标的规范化名称为前缀：

# 以 libcurl 为例（lib/Makefile.am 中的简化示意）

lib_LTLIBRARIES = libcurl.la

libcurl_la_SOURCES = \       # 构成该库的源文件
  url.c transfer.c connect.c \
  http.c ftp.c ssl.c

libcurl_la_CFLAGS  = @CURL_CFLAG_EXTRAS@  # 编译标志
libcurl_la_LDFLAGS = -version-info \       # 链接标志（库版本信息）
  @CURL_LT_SHLIB_VERSIONED_FLAVOUR@

libcurl_la_LIBADD  = @LIBS@               # 链接时追加的库

# curl 可执行程序（src/Makefile.am 简化示意）
bin_PROGRAMS = curl
curl_SOURCES = tool_main.c tool_operate.c tool_cb_hdr.c
curl_LDADD   = ../lib/libcurl.la @LIBS@    # 链接 libcurl

SUBDIRS 与递归构建

SUBDIRS 变量控制 make 递归进入哪些子目录，顺序决定编译顺序。 curl 的 SUBDIRS = lib src docs tests 确保先编译 lib（生成 libcurl.la）， 再编译 src（链接 libcurl.la），顺序不能颠倒。

dist_ 和 nodist_ 前缀

dist_doc_DATA   = README CHANGES  # 分发并安装
nodist_curl_SOURCES = curl_config.h  # 不分发（构建时生成）
🔬 第一性原理

Makefile.am 解决的核心问题：完整的 Makefile 有数百行规则（编译、链接、安装、清理、分发、测试……），手写极易出错且不可移植。

Automake 的方法是：开发者只声明"我有哪些目标，源文件是什么"（约 10 行），Automake 自动展开成完整的 Makefile.in（可能数百行）。 规范化目标名称的算法（把 libcurl.la → libcurl_la_SOURCES）保证了变量命名的唯一性和可预测性。

Chapter 4

其他辅助文件

自定义 M4 宏文件（m4/*.m4）

curl 项目在 m4/ 目录下维护了数十个 .m4 文件，如 curl-openssl.m4、curl-bearssl.m4 等， 这些宏封装了复杂的特定检测逻辑（如检测 OpenSSL 版本、BearSSL API 兼容性等），使 configure.ac 保持简洁。

📁
m4/*.m4
自定义 Autoconf 宏定义。开发者维护，入版本库。aclocal 会扫描此目录，将其内容汇入 aclocal.m4。

aclocal.m4

🤖
aclocal.m4
由 aclocal 自动生成。汇总了系统安装的第三方宏（如 libtool 宏）和项目自己的 m4/*.m4 中的宏。autoconf 读取此文件以识别所有宏定义。

config.h / config.h.in

🔧
lib/curl_config.h（curl 中的 config.h）
config.h.in 由 autoheader 从 configure.ac 生成，是模板。 configure 运行后填充实际值，生成 config.h（或 curl 中的 lib/curl_config.h）。 源代码通过 #include "curl_config.h" 利用这些宏，如 HAVE_OPENSSL、HAVE_ZLIB。 
/* lib/curl_config.h 示例内容（configure 运行后生成）*/
#define HAVE_OPENSSL  1
#define HAVE_ZLIB     1
#define SIZEOF_INT    4
#define HAVE_SEND     1
/* 如果 zstd 不存在则不定义 HAVE_ZSTD */

libcurl.pc（pkg-config 文件）

📦
libcurl.pc
由 configure 从 libcurl.pc.in 模板生成（@VAR@ 被替换）。安装后， 其他项目可通过 pkg-config --libs libcurl 获取链接标志。 libcurl.pc.in 由开发者维护并入版本库。

config.status

🔄
config.status
由 configure 自动生成的 shell 脚本。功能是"记忆" configure 运行的参数和结果， 可在不重新运行 configure 的情况下重新生成 Makefile 和 config.h。 Makefile 中的某些规则会自动调用 config.status 来保持配置文件最新。

辅助脚本（install-sh、depcomp、compile 等）

🛠️
install-sh / depcomp / compile / missing
由 automake --add-missing 或 autoreconf --install 自动添加的辅助脚本。 这些脚本提供了在某些系统上缺失的工具的可移植替代实现（如 install 命令的兼容版本）。 一般放在项目根目录或 build-aux/ 目录（由 AC_CONFIG_AUX_DIR 指定）。
🔬 第一性原理

为何有这么多"中间"文件？这是分层抽象的代价： configure.ac（高层声明）→ M4展开 → configure（可执行 shell 脚本）→ 运行探测 → config.h + Makefile（最终制品）。

每一层的存在都有其合理性：M4 提供宏复用，shell 脚本确保无依赖运行，config.h 让 C 代码条件编译。 config.status 是"增量更新"的关键——只重新生成受影响的文件，而无需重跑整个 configure。

Chapter 5

编译的五个阶段

📖 GNU Build System: Automake, Autoconf, Libtool 📖 curl GIT-INFO.md 官方构建说明

从 Git 克隆 curl 到最终安装，整个过程分为 5 个明确的阶段，每个阶段有清晰的输入和输出。 普通用户从发行 tarball 安装时，跳过阶段 1，直接从阶段 2 开始。

1

Bootstrap 引导阶段 — autoreconf -fi

仅开发者/Git 克隆需要 需安装 Autotools

autoreconf 是一个协调工具，按正确顺序调用以下所有工具：

工具输入输出作用
aclocalconfigure.ac, m4/*.m4aclocal.m4收集第三方和自定义 M4 宏
libtoolize—ltmain.sh, m4/libtool.m4添加 Libtool 支持文件
autoheaderconfigure.ac, aclocal.m4config.h.in生成 config.h 模板
automake --add-missingMakefile.am, configure.acMakefile.in生成 Makefile 模板
autoconfconfigure.ac, aclocal.m4configure生成配置脚本
 
# curl 官方 GIT-INFO.md 指定的命令
autoreconf -fi
# -f: 强制重建所有文件（即使已存在）
# -i: 安装缺失的辅助文件（install-sh 等）

输入：configure.ac、Makefile.am（所有目录）、m4/*.m4

输出：configure、Makefile.in（所有目录）、config.h.in、aclocal.m4

2

配置阶段 — ./configure

普通用户从此开始 无需 Autotools

在用户机器上运行探测逻辑，检测编译器、库、头文件等环境特征，将结果填入模板文件中生成实际的构建文件。 

# 基础配置
./configure --prefix=/usr/local --with-openssl

# curl 开发者常用配置（来自 GIT-INFO.md）
./configure --disable-shared --enable-debug --enable-maintainer-mode

# configure 完成时打印的摘要（简化示意）
# curl version:    8.x.x
# SSL:             OpenSSL
# zlib:            enabled
# Build libcurl:   Shared=yes, Static=no
# Install prefix:  /usr/local

输入：configure（脚本）、Makefile.in、config.h.in、用户参数（--with-openssl 等）、系统环境

输出：Makefile（所有目录）、lib/curl_config.h、config.status、config.log、libcurl.pc、curl-config

configure 的工作原理：对每个检测项，编写一小段测试 C 代码，用探测到的编译器编译并链接，根据成败决定是否定义对应的宏（写入 config.h）或追加链接标志（写入 Makefile 的 LIBS 变量）。
3

编译阶段 — make

读取 Makefile，调用编译器和链接器，将 C 源文件编译为目标文件（.o），再链接成库或可执行文件。

# 单线程编译
make

# 并行编译（推荐，N 为 CPU 核数）
make -j$(nproc)

# 实际发生的事情（简化）：
# [CC]  lib/url.o         ← 编译 lib/url.c
# [CC]  lib/connect.o     ← 编译 lib/connect.c
# [CCLD] lib/.libs/libcurl.so.4  ← 链接共享库
# [CC]  src/tool_main.o   ← 编译 src/tool_main.c
# [CCLD] src/curl         ← 链接可执行文件

输入：Makefile、所有 .c 源文件、lib/curl_config.h

输出：lib/.libs/libcurl.so（共享库）、lib/.libs/libcurl.a（静态库）、src/curl（可执行文件）

4

测试阶段 — make check

可选阶段。运行项目自带的测试套件，验证编译结果的正确性。

make check
# 对于 curl：运行 tests/ 目录下的集成测试和单元测试

输入：Makefile、编译好的二进制文件、测试脚本

输出：测试通过/失败报告

5

安装阶段 — make install

将编译产物复制到 --prefix 指定的安装目录（默认 /usr/local）。

sudo make install

# 安装内容：
# /usr/local/bin/curl           ← 可执行文件
# /usr/local/lib/libcurl.so.4   ← 共享库
# /usr/local/include/curl/curl.h ← 头文件
# /usr/local/lib/pkgconfig/libcurl.pc ← pkg-config 文件
# /usr/local/bin/curl-config    ← 配置脚本

# 也可以安装到临时目录（常用于打包）
make install DESTDIR=/tmp/staging

输入：Makefile、编译产物、--prefix 配置

输出：安装到目标系统上的文件

附加：发行打包 — make dist

开发者使用此命令生成供用户下载的 tarball，其中包含预生成的 configure 和 Makefile.in，使用户无需安装 Autotools。

make dist
# 生成 curl-8.x.x.tar.gz，包含：
# configure（预生成）
# Makefile.in（所有目录，预生成）
# config.h.in（预生成）
# 所有源码文件
🔬 第一性原理

为何分这五步？每一步都是"关注点分离"的体现：

1️⃣ Bootstrap = 生成探测工具；2️⃣ Configure = 探测当前环境；3️⃣ Make = 执行编译；4️⃣ Check = 验证正确性；5️⃣ Install = 部署到系统。

每步都是幂等的、可重复的。用户可以在任意步骤停下来检查中间产物，也可以用不同参数重新运行 configure 而无需重新 bootstrap。 这种"流水线"设计使得调试、打包、交叉编译等复杂场景都变得可控。

🔬 第一性原理

整个 Autotools 流程可以用"声明 → 编译 → 运行"三层模型来理解：

第一层（声明）：configure.ac + Makefile.am 是开发者对构建需求的声明式描述。

第二层（编译）：Autotools 工具链将这些声明"编译"成可执行的探测脚本（configure）和 Makefile 模板（Makefile.in）。

第三层（运行）：在用户机器上"运行"这些脚本（./configure && make），探测环境并构建最终产物。

这与高级语言→字节码→执行的三层结构同构。分层的好处是：用户只需要运行第三层，不需要安装第二层工具（Autotools）。

🔬 第一性原理

宏命名约定的本质：Autoconf 宏用前缀区分来源（AC_=Autoconf、AM_=Automake、LT_=Libtool）；Automake 变量用"目录前缀+下划线+主类型"的组合实现了一套声明式的安装路径语言。

这种命名约定使得构建系统自文档化：看到 bin_PROGRAMS = curl，就能立刻知道"curl 这个程序编译后安装到 $bindir"，无需查阅额外的安装脚本。 规范化命名（特殊字符→下划线）则保证了变量名的唯一性和可预测性，是整个系统自动化的基础。
