1curl 与 HTTP 概述

根据官方文档 everything.curl.dev/http：

"In all user surveys and during all curl's lifetime, HTTP has been the most important and most frequently used protocol that curl supports."

curl 是一款通过 URL 传输数据的命令行工具，由 Daniel Stenberg 于 1998 年发起维护至今，底层依赖 libcurl 实现所有传输功能。它支持 HTTP/0.9、1.0、1.1、2、3 全版本，并对 HTTPS 提供原生 TLS 支持。

HTTP 方法与 curl 选项的映射

根据官方文档："Normally however you do not specify the method in the command line, but instead the exact method used depends on the specific options you use."

HTTP 方法 触发方式 说明
GET默认（无额外选项）读取资源，幂等安全
POST-d 或 -F提交数据，含请求体
HEAD-I只获取响应头，不含 Body
PUT-T上传文件，整体替换
DELETE-X DELETE删除资源
PATCH-X PATCH部分修改资源

📌 本章总结

  • curl 是 HTTP 交互的通用"手术刀"，底层通过 libcurl 完成 TCP 连接、TLS 握手、HTTP 封包全流程
  • HTTP 方法不是靠 -X 硬编码，而是由具体数据选项（-d/-F/-T/-I）自动推断
  • HTTPS 与 HTTP 本质相同，只是在 TCP 之上加了 TLS 层
🔬 第一性原理：HTTP 是一种无状态的请求-响应文本协议。curl 将命令行参数翻译为符合 RFC 规范的字节流，通过 TCP（或 QUIC）发送给服务器，再将响应字节流输出到 stdout 或文件。一切高级特性（Cookie、认证、重定向）都是在此基础上的约定扩展。

2GET 请求：获取资源

GET 是 curl 的默认 HTTP 方法。直接提供 URL 即发起 GET 请求，curl 将响应内容打印到标准输出。

2.1 基本 GET 请求

场景：获取网页内容 GET
# 获取网页主页（来源：curl/docs/MANUAL.md）
curl https://www.example.com/

# 指定端口
curl http://www.example.com:8000/

2.2 保存到文件

场景：下载资源到本地
# 自定义保存文件名（-o 小写）
curl -o thatpage.html https://www.example.com/

# 使用服务器文件名（-O 大写）
curl -O https://www.example.com/index.html

# 同时下载多个文件，保留远端文件名
curl -O www.haxx.se/index.html -O curl.se/download.html

2.3 查看响应头（HEAD 请求）

场景：仅获取 HTTP 响应头 HEAD
# -I 发送 HEAD 请求，只显示响应头
curl -I https://www.example.com/

# -i 在 GET 响应中同时显示响应头和 body
curl -i https://www.example.com/

# 将响应头保存到单独文件
curl --dump-header headers.txt curl.se

2.4 范围请求（Range）

根据 MANUAL.md："HTTP 1.1 introduced byte-ranges. Using this, a client can request to get only one or more sub-parts of a specified document. curl supports this with the -r flag."

图 2-1：HTTP 范围请求示意（断点续传基础）
场景：部分下载
# 获取文档前 100 字节（对应 HTTP Range: bytes=0-99）
curl -r 0-99 https://www.example.com/

# 获取文档最后 500 字节
curl -r -500 https://www.example.com/

2.5 断点续传

场景：恢复中断的下载
# -C - 告诉 curl 自动续传（-C - 中的 - 表示自动判断偏移量）
curl -C - -o file https://www.example.com/large.zip

# 配合限速，模拟弱网续传
curl -C - --limit-rate 100K -o file https://example.com/big.iso

📌 本章总结

  • GET 是 curl 的默认行为，不需要任何方法选项
  • -o（小写）自定义文件名，-O（大写）沿用服务器文件名
  • -I 发 HEAD 请求，获取元信息而不下载 body，常用于检测资源是否存在
  • 范围请求（-r）映射到 HTTP Range: bytes= 请求头，是断点续传的底层机制
  • -C - 实现续传，原理是向服务器发送 Range: bytes=已下载量-

3POST 请求：提交数据

根据官方文档："POST is the HTTP method that was invented to send data to a receiving web application, and it is how most common HTML forms on the web work."

图 3-1：三种 POST 数据格式对比

3.1 表单数据 POST（-d）

根据 MANUAL.md："While -d uses the application/x-www-form-urlencoded mime-type, generally understood by CGI's and similar..."

场景：提交 HTML 表单 POST
# 基本 POST（-d 自动设置 Content-Type: application/x-www-form-urlencoded）
curl -d "name=Rafael%20Sagula&phone=3320780" https://www.example.com/guest.cgi

# 多个 -d 会自动用 & 拼接
curl -d name=admin -d shoesize=12 http://example.com/

# 自动 URL 编码（无需手动 %20）
curl --data-urlencode "name=Rafael Sagula&phone=3320780" https://www.example.com/guest.cgi

# 从文件读取 POST 数据（@ 前缀）
curl -d @filename https://example.com

# 发送以 @ 开头的字面字符串（用 --data-raw 避免文件解释）
curl --data-raw '@string' https://example.com

3.2 发送 JSON（--json）

根据 everything.curl.dev："curl 7.82.0 introduced the --json option... This option works as a shortcut and provides a single option that replaces these three: --data-binary [arg], --header "Content-Type: application/json", --header "Accept: application/json""

场景：REST API JSON 请求 POST
# --json 自动设置 Content-Type 和 Accept（curl >= 7.82.0）
curl --json '{"tool": "curl"}' https://example.com/

# 从文件读取 JSON
curl --json @json.txt https://example.com/

# 从 stdin 读取 JSON
echo '{"a":"b"}' | curl --json @- https://example.com/

# 兼容旧版 curl 的手动方式
curl -X POST \
     -H "Content-Type: application/json" \
     -H "Accept: application/json" \
     -d '{"username":"admin","password":"secret"}' \
     https://api.example.com/login

3.3 多部件表单上传（-F）

根据 MANUAL.md："-F accepts parameters like -F "name=contents". If you want the contents to be read from a file, use @filename as contents."

场景：文件上传（multipart/form-data）POST
# 上传单个文件（@ 表示文件内容）
curl -F "file=@cooltext.txt" \
     -F "yourname=Daniel" \
     -F "filedescription=Cool text" \
     https://www.example.com/postit.cgi

# 指定 Content-Type（追加 ;type=）
curl -F "web=@index.html;type=text/html" url.com

# 一个字段发送多个文件
curl -F "pictures=@dog.gif,cat.gif" https://example.com/

# 发送密码文件的例子（来自官方 man page）
curl -F "password=@/etc/passwd" www.mypasswords.com
⚠️
根据 MANUAL.md："To send a field value literally without interpreting a leading @ or <, or an embedded ;type=, use --form-string instead of -F." 当字段值来自用户输入时，应使用 --form-string 防止被解释为文件路径。

3.4 二进制数据 POST

场景：发送二进制或保留换行的数据
# --data-binary 保留换行符和回车符（-d 会去除）
curl --data-binary @filename https://example.com

# 发送二进制内容时指定 Content-Type
curl --data-binary @photo.jpg \
     -H "Content-Type: image/jpeg" \
     https://upload.example.com/

📌 本章总结

  • -d：表单编码 POST，自动去除换行，适合普通表单
  • --data-urlencode：自动处理 URL 特殊字符编码
  • --json（≥7.82.0）：REST API 最简洁的 JSON POST 方式
  • -F：multipart/form-data，用于文件上传；--form-string 安全版本
  • --data-binary：逐字节发送，不做任何处理，适合二进制或需保留换行的数据
🔬 第一性原理：POST 的本质是向服务器发送一个含 Body 的 HTTP 请求。Body 的格式（Content-Type）决定了服务器如何解析。curl 的三个 POST 选项分别对应 Web 的三种场景：浏览器表单（form-urlencoded）、API 调用（JSON）、文件上传（multipart）。

4PUT / DELETE / PATCH 请求

4.1 PUT 上传文件

根据 MANUAL.md："Note that the HTTP server must have been configured to accept PUT before this can be done successfully."

场景：HTTP PUT 文件上传 PUT
# 通过 stdin 上传到 HTTPS（-T - 表示从 stdin 读取）
curl -T - https://www.example.com/myfile

# 上传本地文件
curl -T upload.tar.gz https://example.com/files/upload.tar.gz

# 同时上传多个文件
curl -T local1 https://example.com/file1 \
     -T local2 https://example.com/file2

4.2 DELETE 请求

场景：删除资源 DELETE
# 发送 DELETE 请求
curl -X DELETE https://api.example.com/users/123

# 携带认证 token 的 DELETE
curl -X DELETE \
     -H "Authorization: Bearer YOUR_TOKEN" \
     https://api.example.com/resources/456

4.3 PATCH 请求

场景：部分更新资源 PATCH
# PATCH 请求更新部分字段
curl -X PATCH \
     -H "Content-Type: application/json" \
     -d '{"email":"new@example.com"}' \
     https://api.example.com/users/123
图 4-1：RESTful HTTP 方法语义对比

📌 本章总结

  • -T 文件名 触发 PUT，用于整体替换资源；服务器需预先配置允许 PUT
  • -X DELETE/PATCH 显式指定方法，常配合 -H "Authorization: ..." 使用
  • PUT 是幂等的（多次提交结果相同），PATCH 不一定幂等，POST 不幂等

5自定义请求头与 User-Agent

根据 MANUAL.md："When using curl in your own programs, you may end up needing to pass on your own custom headers when getting a webpage. You can do this by using the -H flag."

5.1 自定义请求头

场景：添加自定义 HTTP 头
# 添加自定义请求头
curl -H "X-you-and-me: yes" love.example.com

# 多个请求头（多次 -H）
curl -H "Content-Type: application/json" \
     -H "X-API-Key: abc123" \
     -H "Accept: application/json" \
     https://api.example.com/data

# 覆盖 curl 默认的 Host 头（发送空值即禁用）
curl -H "Host:" server.example.com

# Bearer Token 认证
curl -H "Authorization: Bearer eyJhbGciOiJIUzI1NiJ9..." \
     https://api.example.com/protected

5.2 User-Agent

根据 MANUAL.md："An HTTP request has the option to include information about the browser that generated the request. curl allows it to be specified on the command line. It is especially useful to fool or trick stupid servers or CGI scripts that only accept certain browsers."

场景：模拟浏览器或自定义标识
# -A 设置 User-Agent（官方示例）
curl -A 'Mozilla/3.0 (Win95; I)' https://www.bank.example.com/

# 模拟 Chrome 浏览器
curl -A 'Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 Chrome/120.0.0.0' \
     https://example.com/

# 清空 User-Agent（发送空字符串）
curl -A '' https://example.com/

5.3 Referer（来源页）

根据 MANUAL.md："An HTTP request has the option to include information about which address referred it to the actual page. curl allows you to specify the referrer to be used on the command line."

场景：设置来源页 URL
# -e 设置 Referer 头
curl -e www.example.org https://www.example.com/

# 等价方式（使用 -H）
curl -H "Referer: https://www.google.com/" https://example.com/
图 5-1：HTTP 请求报文中的请求头位置

📌 本章总结

  • -H "Name: Value" 可任意添加 HTTP 请求头，多次使用可添加多个
  • 用 -H "Name:"（空值）可禁用 curl 默认添加的某个请求头
  • -A 设置 User-Agent，-e 设置 Referer，均是 -H 的快捷方式
  • 自定义头中若与 curl 内置头同名，以自定义为准

6HTTP 认证

根据 everything.curl.dev/http/auth："Each HTTP request can be made authenticated. If a server or a proxy want the user to provide proof that they have the correct credentials... it sends back a 401 response code and an associated WWW-Authenticate: header."

图 6-1：HTTP Basic 认证的两次握手流程

6.1 Basic 认证

根据官方文档："The Basic authentication method sends the username and password in clear text over the network (base64 encoded) and should be avoided for HTTP transport."

场景：用户名密码认证
# -u 提供用户名和密码（默认使用 Basic 方式）
curl --user daniel:secret http://example.com/

# 等价写法
curl -u name:passwd https://http.server.example/full/path/to/file

# URL 内嵌方式（注意：不适用于代理场景）
curl https://name:passwd@http.server.example/full/path/to/file

# 显式指定 Basic 方法
curl -u name:password --basic https://example.com

6.2 Digest / NTLM / Negotiate 认证

根据官方文档："curl typically speaks several other authentication methods as well, including Digest, Negotiate and NTLM."

场景：使用安全认证方式
# Digest 认证（MD5 散列，不传明文密码）
curl --digest --user daniel:secret http://example.com/

# NTLM 认证（Windows 域认证）
curl --ntlm --user daniel:secret http://example.com/

# Negotiate/SPNEGO 认证（Kerberos）
curl --negotiate --user daniel:secret http://example.com/

# anyauth：让 curl 自动选择最安全的方式
curl --anyauth --user daniel:secret http://example.com/

6.3 AWS SigV4 认证

根据官方文档："The defacto authentication standard AWS SigV4 is a little different than the other HTTP authentication mechanisms."

场景：AWS API 认证
# --aws-sigv4 参数格式：provider1[:provider2[:region[:service]]]
curl --aws-sigv4 "aws:amz:us-east-2:es" \
     --user "key:secret" \
     https://example.com
认证方式curl 选项安全性说明
Basic--basic（默认）⭐Base64 编码，未加密，需配合 HTTPS
Digest--digest⭐⭐MD5 散列密码，不传明文
NTLM--ntlm⭐⭐⭐微软 Windows 域认证
Negotiate--negotiate⭐⭐⭐⭐Kerberos/SPNEGO
Bearer Token-H "Authorization: Bearer ..."⭐⭐⭐OAuth2/JWT 常用方式
AWS SigV4--aws-sigv4⭐⭐⭐⭐⭐HMAC 签名，AWS 专用

📌 本章总结

  • 服务器通过 401 + WWW-Authenticate 告知客户端需要认证及支持的方式
  • Basic 认证只是 Base64 编码，本质是明文，必须配合 HTTPS 使用
  • --anyauth 让 curl 先发无认证请求探测，再用服务器支持的最安全方式
  • 代理认证须使用 -U/--proxy-user，不能在 URL 内嵌用户名密码
🔬 第一性原理：HTTP 认证的本质是在请求头中携带凭证证明身份。各种认证方式的区别在于"如何证明你知道密码"——Basic 是直接告诉密码，Digest 是证明你能用密码算出正确散列，Negotiate 是持有 Kerberos 票据。越高级的方式越难被中间人攻击。

7Cookie 管理

根据官方文档："HTTP cookies are key/value pairs that a client stores on the behalf of a server. They are sent back in subsequent requests to allow clients to keep state between requests. Remember that the HTTP protocol itself has no state..."

图 7-1：Cookie 的发送与接收流程

7.1 发送 Cookie

根据 MANUAL.md："Example, get a page that wants my name passed in a cookie: curl -b "name=Daniel" www.example.com"

场景：携带 Cookie 请求
# 直接在命令行发送 Cookie 字符串
curl -b "name=Daniel" www.example.com

# 从文件读取 Cookie（Netscape 格式或 HTTP 头格式）
curl -b cookies.txt www.example.com

7.2 保存与持久化 Cookie

根据 MANUAL.md："make curl save the incoming cookies using the well-known Netscape cookie format like this: curl -c cookies.txt www.example.com"

场景：模拟登录会话（Cookie 持久化）
# -c 保存服务器返回的 Cookie 到文件
curl -c cookies.txt www.example.com

# 完整的会话场景：读取并写入同一文件（跨请求保持登录态）
curl -b cookies.txt -c cookies.txt www.example.com

# 保存响应头到文件（传统方式，但不推荐）
curl --dump-header headers www.example.com
curl -b headers www.example.com  # 下次读取头文件中的 Cookie

# 配合重定向跟随，使用空文件启用 Cookie 引擎
curl -L -b empty.txt www.example.com
💡
根据官方文档："Note that by specifying -b you enable the cookie engine..." — -b 不仅是"发送 Cookie"，更是"启动 Cookie 引擎"，让 curl 开始追踪和管理 Cookie 状态。

📌 本章总结

  • HTTP 协议本身无状态，Cookie 是在客户端维护状态的约定机制
  • -b 发送/读取 Cookie，-c 保存 Cookie，同时使用实现完整会话
  • Cookie 文件格式为 Netscape 格式（curl 自动识别）
  • 会话 Cookie（无过期时间）在 curl 退出后丢失；持久 Cookie 有明确过期时间
🔬 第一性原理：HTTP 的无状态性意味着每次请求都是全新的，服务器没有"记忆"。Cookie 解决了这个问题——服务器把一个标识符"盖"在客户端，客户端每次请求都带上这个戳，服务器靠这个戳识别用户。curl 的 -b/-c 就是模拟浏览器这个"存印章、带印章"的行为。

8重定向跟随

根据 everything.curl.dev/http/redirects："The 'redirect' is a fundamental part of the HTTP protocol. The concept was present and is documented already in the first spec (RFC 1945), published in 1996."

图 8-1：HTTP 重定向状态码对比（来源：everything.curl.dev/http/redirects）

8.1 跟随重定向

根据官方文档："In curl's tradition of only doing the basics unless you tell it differently, it does not follow HTTP redirects by default. Use the -L, --location option to tell it to do that."

场景：自动跟随重定向
# -L 跟随重定向（默认最多跟随 30 次）
curl -L https://example.com/old-url

# 自定义最大重定向次数
curl -L --max-redirs 5 https://example.com/

# 跟随重定向，同时发送 Cookie
curl -L -b cookies.txt -c cookies.txt https://example.com/login

# POST 重定向后仍保持 POST（用于非标准服务器）
curl -L --post301 -d "data=value" https://example.com/api

8.2 重定向中的凭证处理

根据官方文档："curl limits what it sends away to other hosts than the original within the same transfer."

场景：跨域重定向携带凭证
# 允许向重定向目标主机发送凭证（需信任目标主机）
curl -L --location-trusted -u user:pass https://example.com/

📌 本章总结

  • curl 默认不跟随重定向，必须加 -L
  • 301/302/303 重定向后 curl 将 POST 转换为 GET；307/308 保持原方法
  • 如需重定向后仍用 POST，使用 --post301/--post302/--post303
  • 跨域重定向默认不传递认证凭证，需 --location-trusted 显式信任

9HTTPS 与 TLS/SSL

根据官方文档："HTTPS is in effect Secure HTTP. The secure part means that the TCP transport layer is enhanced to provide authentication, privacy (encryption) and data integrity by the use of TLS."

图 9-1：TLS 握手简化流程（每次 HTTPS 连接的前置步骤）

9.1 基本 HTTPS 请求

场景：HTTPS 安全请求
# 直接使用 HTTPS（curl 自动进行 TLS 握手和证书验证）
curl https://secure.example.com

# 指定 TLS 版本（对旧服务器兼容）
curl --tlsv1.0 https://secure.example.com/
curl --tlsv1.2 https://secure.example.com/
curl --tlsv1.3 https://secure.example.com/

9.2 证书管理

根据 MANUAL.md："curl is also capable of using client certificates to get/post files from sites that require valid certificates. The only drawback is that the certificate needs to be in PEM-format."

场景：客户端证书认证
# 使用客户端证书（PEM 格式，格式：证书文件:密码）
curl -E /path/to/cert.pem:password https://secure.example.com/

# 指定自定义 CA 证书（验证服务器证书）
curl --cacert CA-file.txt https://example.com

# 指定 CA 证书目录
curl --capath /etc/ssl/certs/ https://example.com

# 使用操作系统原生证书存储（curl >= 8.2.0）
curl --ca-native https://example.com

9.3 跳过证书验证（仅测试用）

场景：自签名证书开发环境
# -k/--insecure 跳过证书验证（警告：生产环境绝对不要使用！）
curl -k https://self-signed.example.com/

# 同等写法
curl --insecure https://self-signed.example.com/
⚠️
-k/--insecure 会禁用所有证书验证，使连接容易受到中间人攻击。仅在受控的开发/测试环境中使用。

9.4 HTTP 版本选择

根据官方文档："curl defaults to HTTP/1.1 for HTTP servers but if you connect to HTTPS and you have a curl that has HTTP/2 abilities built-in, it attempts to negotiate HTTP/2 automatically."

场景：指定 HTTP 协议版本
# 使用 HTTP/2（需要 curl 编译时支持）
curl --http2 https://example.com

# 跳过升级协商，直接用 HTTP/2（需确认服务器支持）
curl --http2-prior-knowledge https://example.com

# 使用 HTTP/3（实验性，基于 QUIC）
curl --http3 https://example.com

# 强制 HTTP/1.1
curl --http1.1 https://example.com

# 强制 HTTP/1.0
curl -0 https://example.com
HTTP 版本curl 选项传输层多路复用主要特性
HTTP/1.0-0TCP✗每请求单独连接
HTTP/1.1--http1.1（默认）TCP✗持久连接、分块传输
HTTP/2--http2TCP+TLS✓二进制帧、头压缩、服务器推送
HTTP/3--http3（实验）QUIC/UDP✓0-RTT、无队头阻塞

📌 本章总结

  • HTTPS = HTTP + TLS，TLS 提供：服务器身份认证（证书）、传输加密、数据完整性
  • curl 默认验证服务器证书，使用系统或指定的 CA 束文件
  • -E 用于客户端证书认证（双向 TLS），需 PEM 格式
  • HTTP/2 在 HTTPS 上通过 ALPN 自动协商；HTTP/3 基于 QUIC（UDP），彻底解决队头阻塞
🔬 第一性原理：HTTP 在设计上是明文协议，不解决身份认证和窃听问题。TLS 通过非对称加密交换密钥、对称加密传输数据、数字证书证明身份，在 TCP 之上构建了"安全隧道"。curl 的 TLS 选项本质上是配置这个隧道的参数。

10代理与网络控制

根据 MANUAL.md："curl supports both HTTP and SOCKS proxy servers, with optional authentication."

10.1 使用 HTTP 代理

场景：通过代理访问
# -x 指定代理服务器（来自官方示例）
curl -x my-proxy:888 ftp://ftp.example.com/README

# 通过代理访问 HTTPS，带认证
curl -u user:passwd -x my-proxy:888 https://www.example.com/

# 代理本身需要认证（-U 是代理用户名密码）
curl -U user:passwd -x my-proxy:888 https://www.example.com/

# 绕过代理的主机列表
curl --noproxy example.com -x my-proxy:888 https://www.example.com/

# SOCKS5 代理（常用于 Tor 或内网穿透）
curl --socks5 127.0.0.1:1080 https://www.example.com/

10.2 环境变量代理配置

根据 MANUAL.md 列举的环境变量：http_proxy、HTTPS_PROXY、FTP_PROXY、ALL_PROXY、NO_PROXY

场景：通过环境变量全局配置代理
# 为 HTTP 设置代理
export http_proxy=http://proxy.company.com:8080
curl http://example.com/

# 为 HTTPS 设置代理
export HTTPS_PROXY=http://proxy.company.com:8080
curl https://example.com/

# 全协议代理
export ALL_PROXY=socks5://127.0.0.1:1080

# 不走代理的域名列表
export NO_PROXY=localhost,127.0.0.1,.company.internal

10.3 速率与超时控制

根据 MANUAL.md："Forcing curl not to transfer data faster than a given rate is also possible, which might be useful if you are using a limited bandwidth connection..."

场景：限速与超时
# 限制下载速度不超过 10KB/s
curl --limit-rate 10K www.far-away.example.com

# 限制速度不超过 1MB/s
curl --limit-rate 1M https://example.com/bigfile.iso

# 整个操作最多 30 分钟（1800 秒）
curl -m 1800 https://example.com/

# 低速保护：如果速度低于 3000B/s 持续 60s 则中止
curl -Y 3000 -y 60 www.far-away.example.com

# 组合：30分钟上限 + 低速保护
curl -m 1800 -Y 3000 -y 60 www.far-away.example.com

📌 本章总结

  • -x 指定 HTTP 代理，--socks5 指定 SOCKS 代理，-U 代理认证
  • 环境变量 http_proxy/HTTPS_PROXY/ALL_PROXY 实现全局代理，NO_PROXY 排除白名单
  • --limit-rate 带宽限速，防止占满网络；-m 整体超时；-Y/-y 低速保护
  • 代理认证使用 -U，不同于目标站点认证的 -u

11调试与诊断

根据 MANUAL.md："If curl fails where it is not supposed to, if the servers do not let you in, if you cannot understand the responses: use the -v flag to get verbose fetching."

图 11-1：curl -v 详细输出格式说明（* > < 前缀含义）

11.1 详细输出与追踪

场景：调试 HTTP 请求
# -v 详细模式（* 连接信息，> 发出请求，< 收到响应）
curl -v https://example.com/

# --trace 完整追踪（包括二进制数据），输出到文件
curl --trace my-trace.txt www.haxx.se

# --trace-ascii 追踪（ASCII 可读格式）
curl --trace-ascii trace.txt https://example.com/

11.2 自定义输出格式（-w）

根据 MANUAL.md："To better allow script programmers to get to know about the progress of curl, the -w/--write-out option was introduced."

场景：提取响应指标
# 显示 HTTP 状态码
curl -w '%{response_code}\n' -o /dev/null -s https://example.com

# 显示下载字节数
curl -w 'We downloaded %{size_download} bytes\n' www.example.com

# 完整性能诊断（各阶段耗时）
curl -w "time_namelookup: %{time_namelookup}s\ntime_connect: %{time_connect}s\ntime_appconnect: %{time_appconnect}s\ntime_pretransfer: %{time_pretransfer}s\ntime_starttransfer: %{time_starttransfer}s\ntime_total: %{time_total}s\n" \
     -o /dev/null -s https://example.com/

# 只显示状态码，静默其余输出（-s 静默，-o 丢弃 body）
curl -s -o /dev/null -w "%{http_code}" https://example.com/

11.3 时间条件请求

根据 MANUAL.md："HTTP allows a client to specify a time condition for the document it requests. It is If-Modified-Since or If-Unmodified-Since. curl allows you to specify them with the -z/--time-cond flag."

场景：条件下载（节省带宽）
# 只在远端文件比本地文件新时才下载（If-Modified-Since）
curl -z local.html https://remote.example.com/remote.html

# 只在本地文件比远端更新时下载（前加 -）
curl -z -local.html https://remote.example.com/remote.html

# 指定具体日期（2012年1月12日之后更新才下载）
curl -z "Jan 12 2012" https://remote.example.com/remote.html

📌 本章总结

  • -v 显示请求/响应全过程，* 连接元信息，> 发出请求头，< 收到响应头
  • --trace 是比 -v 更深层的完整追踪，包括二进制帧数据
  • -w 提取自定义指标，配合 -s -o /dev/null 可实现纯性能测量
  • -z 对应 HTTP 条件请求（If-Modified-Since），用于增量更新，节省带宽

12综合实战场景

12.1 模拟完整的 Web 登录会话

场景：模拟浏览器登录，维持会话
# 第一步：发 POST 登录请求，保存 Cookie
curl -c session.txt \
     -d "username=admin&password=secret" \
     https://example.com/login

# 第二步：携带 Cookie 访问受保护页面
curl -b session.txt \
     -L \
     https://example.com/dashboard

12.2 REST API 完整 CRUD 示例

场景：RESTful API 测试
# 创建（POST）
curl --json '{"name":"Alice","role":"admin"}' \
     -H "Authorization: Bearer TOKEN" \
     https://api.example.com/users

# 读取（GET）
curl -H "Authorization: Bearer TOKEN" \
     https://api.example.com/users/1

# 更新（PUT 整体替换）
curl -X PUT \
     --json '{"name":"Alice","role":"superadmin"}' \
     -H "Authorization: Bearer TOKEN" \
     https://api.example.com/users/1

# 部分更新（PATCH）
curl -X PATCH \
     --json '{"role":"superadmin"}' \
     -H "Authorization: Bearer TOKEN" \
     https://api.example.com/users/1

# 删除（DELETE）
curl -X DELETE \
     -H "Authorization: Bearer TOKEN" \
     https://api.example.com/users/1

12.3 批量下载（URL 通配）

根据 man7.org curl(1) 手册："You can specify multiple URLs or parts of URLs by writing lists within braces or ranges within brackets. We call this 'globbing'."

场景：URL 通配批量请求
# 下载 file1.txt 到 file100.txt
curl -O "ftp://ftp.example.com/file[1-100].txt"

# 每隔 10 个下一个
curl -O "http://example.com/file[1-100:10].txt"

# 访问多个子域名
curl "http://site.{one,two,three}.com"

# 多文件多 URL，每个指定不同输出文件名
curl -O https://example.com/file.txt ftp://example.com/moo.exe -o moo.jpg

12.4 管道与脚本集成

场景：与其他工具协作
# 通过管道安装 GPG key（来自官方示例）
curl -L https://apt.example.org/llvm-snapshot.gpg.key | sudo apt-key add -

# 结合 jq 解析 JSON 响应
curl -s --json '{"q":"curl"}' https://api.example.com/search | jq '.results[].name'

# 检测网站可用性（仅返回状态码）
STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://example.com/)
echo "HTTP 状态码: $STATUS"

# 从配置文件读取选项（.curlrc）
echo "user = user:passwd" | curl -K - https://that.secret.example.com

📌 全文总结

  • curl 的设计哲学：默认只做最基础的事（GET），所有高级特性都需显式声明——这让命令意图清晰可审查
  • HTTP 方法由数据选项隐式决定，或用 -X 显式指定
  • 认证：从 Basic（Base64 明文）到 Digest（散列）再到 Negotiate（Kerberos），安全性逐级提升
  • Cookie：-b 发送、-c 保存，组合使用实现完整会话模拟
  • HTTPS：TLS 在 TCP 上构建安全隧道，curl 自动处理证书验证，支持 mTLS
  • 调试：-v 是诊断利器，-w 是脚本化监控的关键
🔬 终极第一性原理：HTTP 的一切复杂性都源于一个简单事实——网络通信的双方相互不信任、通道不可靠、状态需要额外维护。curl 的各种选项本质上是在解决这三个问题：认证解决"信任"，TLS 解决"可靠/私密"，Cookie/Session 解决"状态"。理解了这三点，所有 curl HTTP 选项的存在意义就一目了然。
