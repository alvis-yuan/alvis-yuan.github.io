第一章Linux perf 概述与工作原理

perf（又称 Performance Counters for Linux / perf_events）是 Linux 官方内置的性能分析与追踪工具，源码位于 Linux 内核树 tools/perf 目录下。它通过内核提供的 perf_events 接口，将 CPU 硬件性能监控单元（PMU）、内核软件计数器和 ftrace 追踪点统一抽象成一套命令行界面。

参考来源： perf 官方 Wiki 教程 · Brendan Gregg perf 示例页 · Linux 内核源码 tools/perf

1.1 核心工作原理

perf 的工作核心是 perf_event_open() 系统调用（定义于 include/uapi/linux/perf_event.h）。该系统调用返回一个文件描述符，用户态通过该 fd 控制 PMU 寄存器的读写、采样缓冲区的 mmap 映射，以及事件的开启/关闭。

四大工作模式

根据 perf 官方 Wiki 和 Brendan Gregg 的 perf 示例页，perf 有四种基本工作模式：

模式代表命令工作原理开销
计数（Counting） perf stat 在内核上下文累计事件计数，程序结束时汇总打印，不写文件 极低
采样（Sampling） perf record 每 N 个事件触发一次 PMU 中断，捕获 IP/调用栈写入 perf.data 中等
报告（Reporting） perf report / script 读取 perf.data 文件，汇总/解析已记录的样本 事后
实时录制（Live） perf top 实时录制并在终端展示，无 perf.data 文件 低~中

1.2 PMU 与采样机制

CPU 内部有一组性能监控计数器（PMC）寄存器。每次发生特定微架构事件（如一个时钟周期、一次 L1 缓存缺失、一次分支指令），对应计数器自动递增。当计数器溢出时，CPU 触发一次 NMI（不可屏蔽中断），内核的中断处理程序此时捕获当前的指令指针（IP）和调用栈，写入 环形缓冲区（Ring Buffer），用户态通过 mmap 读出。

📌 采样的统计含义 采样并非捕获每一个事件，而是每隔 N 个事件取一个样本（默认频率如 99Hz）。样本越多，统计越准确。这是一种概率性方法，低开销换取高粒度观测。

📋 第一章总结

第一性原理：perf 的本质是通过 perf_event_open() 系统调用，将 CPU 硬件 PMU 寄存器（数量有限，通常 4~8 个）映射为软件可控的文件描述符。当硬件计数器溢出时触发 NMI，内核捕获上下文写入环形缓冲区，用户态 mmap 读出。这一机制让 perf 得以在极低干扰下观测系统行为——相比 ptrace 类工具，开销降低数个数量级。

perf 的四种模式（计数/采样/报告/实时）本质上都是该机制的不同配置：计数模式只累加不复制数据；采样模式按频率触发中断并记录上下文；实时模式在内存中处理不落盘。

第二章perf 事件类型体系

perf 支持多种来源的事件。理解事件类型是高效使用 perf 的基础。根据 perf Wiki 事件章节 和内核源码 include/uapi/linux/perf_event.h，事件分为以下四大类型。

2.1 事件类型全景

2.2 硬件 PMU 事件

由 CPU 的性能监控单元（PMU）提供，直接反映微架构行为。通过 perf list 可以看到当前 CPU 支持的具体事件列表。常见通用硬件事件：

事件名含义诊断场景
cyclesCPU 时钟周期数（受频率缩放影响）整体 CPU 使用量
instructions已退休指令数与 cycles 计算 IPC
branch-instructions分支指令数代码分支密度
branch-misses分支预测失败次数分支预测优化
cache-referencesLLC（末级缓存）访问次数缓存利用率
cache-missesLLC 缺失次数内存访问瓶颈
stalled-cycles-frontend前端停滞周期（指令获取/解码）代码体积/I-Cache
stalled-cycles-backend后端停滞周期（执行单元等待）内存/计算延迟

2.3 软件事件

由内核软件计数器实现，不依赖 PMU 硬件，在任何平台（含虚拟机）均可用：

# 查看软件事件列表
perf list software

# 常见软件事件
context-switches    # 上下文切换次数
cpu-migrations      # 进程跨 CPU 迁移次数
page-faults         # 页错误总数
minor-faults        # 次要页错误（无需 I/O）
major-faults        # 主要页错误（需要 I/O）
alignment-faults    # 内存对齐错误
cpu-clock           # per-CPU 高分辨率定时器
task-clock          # per-task 高分辨率定时器

2.4 Tracepoint 事件

由内核 ftrace 基础设施实现，是内核中预埋的静态探针点（TRACE_EVENT 宏定义）。覆盖调度、块设备、网络、文件系统、系统调用等几乎所有子系统：

# 列出所有 tracepoint
perf list tracepoint

# 常用分组
sched:*          # 调度器事件（switch, wakeup, migrate...）
block:*          # 块设备 I/O 事件
syscalls:*       # 系统调用进入/退出
net:*            # 网络收发事件
ext4:*, xfs:*   # 文件系统事件
kmem:*           # 内核内存分配事件
vmscan:*         # 内存回收事件
raw_syscalls:*   # 底层系统调用追踪

2.5 原始 PMC 事件（Raw Events）

当 perf 的通用事件名不能满足需求时，可直接使用 CPU 手册中的原始事件编码（Event ID + Umask）：

# 格式：rNNN（十六进制编码）
perf stat -e r003c -a sleep 5

# 或使用完整格式
perf stat -e cpu/event=0x0e,umask=0x01,inv,cmask=0x01/ -a sleep 5
💡 如何查看所有事件 perf list — 列出当前系统支持的所有事件（包括 PMU、软件、tracepoint）。输出可能数千行，建议配合 grep 过滤。

📋 第二章总结

第一性原理：perf 的事件体系反映了"可观测性信号来源的分层"——硬件层（PMU）→ 内核层（软件事件、tracepoint）→ 用户层（uprobe）。每向下一层，观测精度更高，但对系统特定性依赖也越强（硬件事件不同 CPU 各异；tracepoint 需内核版本支持）。选择事件时，应优先使用能直接回答问题的最高层事件，再按需下探。

PMU 事件数量受限于 CPU 的物理计数器（一般 4~8 个），当监控事件数超出时，内核使用多路复用（Multiplexing）轮转测量，结果会按时间比例缩放（scaling），perf stat 输出中的 # 行会注明这一情况。

第三章perf stat — 性能计数统计

perf stat 运行一个命令并收集其性能计数器统计信息，是最常用的入门命令。它的原理是计数模式：在内核累加事件计数，程序结束后一次性打印汇总，开销极低。

来源：perf Wiki Tutorial · Brendan Gregg perf Examples

3.1 基础用法与输出解读

perf stat ./my_program

典型输出示例：

 Performance counter stats for './my_program':

       3,246.80 msec task-clock         #  3.937 CPUs utilized
            127      context-switches   #   39.117 /sec
              4      cpu-migrations     #    1.232 /sec
          1,234      page-faults        #  380.083 /sec
  9,847,532,124      cycles             #  3.032 GHz
 12,891,045,678      instructions       #  1.31  insn per cycle
  2,741,887,890      branches           #  843.9M /sec
     51,234,567      branch-misses      #  1.87% of all branches

       0.824561234 seconds time elapsed
       3.245876543 seconds user
       0.001234567 seconds sys
📌 关键指标解读
  • IPC（insn per cycle）：越高越好。现代 CPU 通常 1.5~4，低于 1 说明存在瓶颈
  • branch-misses %：超过 2~5% 需关注，影响流水线效率
  • cache-misses %：超过 5~10% 说明内存访问模式较差
  • CPUs utilized：有效并行度，越接近核心数越好

3.2 常用场景

场景一：系统全局统计（5 秒）

perf stat -a sleep 5
# -a: 所有 CPU，system-wide 模式

场景二：追踪特定进程

perf stat -p 12345     # 附加到已运行进程（Ctrl-C 结束）
perf stat -t 67890     # 指定线程 TID

场景三：自定义事件组合

# CPU-bound 程序的关键指标
perf stat -e cycles,instructions,cache-references,cache-misses,branch-misses ./app

# L1 数据缓存分析
perf stat -e L1-dcache-loads,L1-dcache-load-misses,L1-dcache-stores ./app

# TLB 分析
perf stat -e dTLB-loads,dTLB-load-misses,dTLB-prefetch-misses ./app

# 末级缓存（LLC）分析
perf stat -e LLC-loads,LLC-load-misses,LLC-stores,LLC-prefetches ./app

场景四：重复运行取平均（性能对比）

# 运行 5 次，输出均值和标准差
perf stat -r 5 ./my_program

# 表格形式展示每次耗时
perf stat --null -r 5 --table perf bench sched pipe

场景五：系统调用频率监控

# 每秒统计系统调用次数
perf stat -e raw_syscalls:sys_enter -I 1000 -a

# 统计特定进程的所有系统调用类型
perf stat -e 'syscalls:sys_enter_*' -p 1234

# 统计文件系统事件（10 秒）
perf stat -e 'ext4:*' -a sleep 10

场景六：每 CPU 分析

# 系统级，每 CPU 单独展示（不聚合）
perf stat -a -A sleep 5

# 只监控特定 CPU 核
perf stat -C 0,1,2,3 sleep 5

3.3 完整示例：诊断 CPU-bound 程序

🔧 实战场景：一个矩阵乘法程序运行很慢，用 perf stat 诊断是 CPU 算力不足还是缓存效率低。 
## 步骤 1：整体概览
perf stat -d ./matrix_multiply
# -d: detailed，显示更多缓存事件

## 输出关键部分（示例）：
#  2.31  insn per cycle              ← IPC 较好
# 48.2%  of all cache refs are misses ← 缓存命中率低！需优化访问模式

## 步骤 2：针对缓存深入分析
perf stat -e L1-dcache-loads,L1-dcache-load-misses,\
LLC-loads,LLC-load-misses ./matrix_multiply

## 步骤 3：对比优化前后（转置矩阵改善局部性）
perf stat -r 3 -e cache-misses,instructions ./matrix_v1
perf stat -r 3 -e cache-misses,instructions ./matrix_v2

📋 第三章总结

perf stat 是性能分析的第一道关卡。它通过计数模式（零采样开销）给出程序的宏观性能画像。核心指标 IPC（指令/周期）、cache-miss 率、branch-miss 率可以快速定位问题类型：计算密集、内存密集还是分支预测低效。

第一性原理：CPU 的效率由流水线利用率决定。IPC 是综合指标，cache miss 和 branch miss 是拉低 IPC 的主要原因。perf stat 让你在不改动代码、不重启服务的前提下，秒级确认性能瓶颈的性质。

第四章perf record — 采样录制

perf record 是 perf 的核心录制命令，通过 PMU 中断采样的方式，捕获程序运行时各函数的 CPU 时间分布及调用栈，并写入 perf.data 文件供后续分析。

来源：perf Wiki Tutorial · Brendan Gregg perf Examples

4.1 采样原理

4.2 核心用法

# 对命令采样（默认事件 cycles，默认频率 4000Hz）
perf record ./my_program

# 指定采样频率 99Hz（推荐，避免与 100Hz 定时器产生共振）
perf record -F 99 ./my_program

# 附加到已有进程
perf record -F 99 -p 1234 -- sleep 30

# 系统全局采样，30 秒
perf record -F 99 -a -- sleep 30

# 捕获调用栈（frame pointer 方式）
perf record -F 99 -g ./my_program

# 捕获调用栈（DWARF 方式，适用于无帧指针编译的程序）
perf record -F 99 --call-graph dwarf ./my_program

# 使用 LBR (Last Branch Record)，Intel 支持
perf record -F 99 --call-graph lbr ./my_program

4.3 常用场景

场景一：CPU 热点函数分析

# 录制
perf record -F 99 -g ./cpu_bound_app

# 分析（交互 TUI）
perf report

# 文本输出
perf report --stdio

场景二：按 L1 缓存缺失采样

# 每 10000 次 L1 缺失采样一次
perf record -e L1-dcache-load-misses -c 10000 -ag -- sleep 5

# 按 LLC（末级缓存）缺失采样
perf record -e LLC-load-misses -c 100 -ag -- sleep 5

场景三：追踪磁盘 I/O 请求

# 追踪所有块设备 I/O 发起事件，带调用栈
perf record -e block:block_rq_insert -ag
perf report

# 同时追踪 I/O 发起和完成（计算延迟）
perf record -e block:block_rq_issue -e block:block_rq_complete -a sleep 60

场景四：追踪网络连接

# 追踪所有出站 TCP 连接，带调用栈
perf record -e syscalls:sys_enter_connect -ag

# 追踪所有入站连接
perf record -e 'syscalls:sys_enter_accept*' -ag

场景五：内存页错误分析

# 采样页错误（RSS 增长）带调用栈
perf record -e minor-faults -ag
perf record -e page-faults -ag

场景六：容器/cgroup 级别采样

# 对特定 Docker 容器的 cgroup 采样
perf record -F 99 -e cpu-clock --cgroup=docker/<容器ID> -a -- sleep 10

4.4 调用栈展开方式对比

方式选项优点缺点
Frame Pointer -g 低开销，速度快 需要编译时保留帧指针 (-fno-omit-frame-pointer)，现代发行版默认优化掉
DWARF --call-graph dwarf 无需特殊编译，适配度高 数据量大，overhead 较高
LBR --call-graph lbr 硬件支持，低开销，精确 仅 Intel 部分 CPU 支持，栈深度有限
⚠️ 99Hz 而非 100Hz 采样频率选择 99Hz 而非 100Hz 是为了避免与系统 100Hz 定时器产生共振（Nyquist 效应），防止采样点总落在同一类事件上导致偏差。这是 Brendan Gregg 在其 perf 文档中明确推荐的做法。

📋 第四章总结

perf record 的本质是统计采样：以可接受的开销（通常 <5%）换取对程序执行热点的统计推断。样本越多（更高频率或更长时间），结论越可靠。

第一性原理：每次 PMU 中断捕获一个瞬时快照（IP + 调用栈），大量快照的统计分布即反映 CPU 时间在各函数间的分配。这与随机抽样的统计学原理完全一致——只要样本量足够，不需要完整追踪每条指令。perf.data 是这些快照的压缩存储，后续所有分析工具（report/script/flamegraph）都是对这份数据的不同视角展现。

第五章perf report — 分析报告

perf report 读取 perf.data 文件，将采样数据汇总、符号解析、调用栈还原，以交互式 TUI 或纯文本方式呈现性能热点分布。

来源：perf Wiki Tutorial

5.1 输出格式解读

perf report --stdio

# Samples: 22K of event 'cpu-clock'
# Event count (approx.): 22751
#
# Overhead  Command      Shared Object          Symbol
# ........  ...........  .....................  ...............................
#
   38.42%   my_program   my_program             [.] compute_hash
   21.07%   my_program   libc.so.6              [.] __memcpy_avx_unaligned
    9.14%   my_program   my_program             [.] parse_record
    8.91%   my_program   [kernel.kallsyms]      [k] copy_user_enhanced_fast
    ...
📌 符号标记说明 [.] = 用户空间符号  |  [k] = 内核符号  |  [.] in lib = 动态库

5.2 常用选项

# 交互式 TUI（默认）
perf report

# 文本模式，方便脚本处理
perf report --stdio

# 显示样本计数
perf report -n

# 折叠调用栈格式（火焰图准备）
perf report --stdio -n -g folded

# 按特定字段排序
perf report --sort comm,dso,symbol

# 只显示特定进程
perf report --comm nginx

# 显示 Children（包含子调用的占比，v3.16+ 默认开启）
perf report                   # Children 列 = 包含所有被调者的开销
perf report --no-children    # 只看 Self（函数自身代码的开销）

5.3 Children vs Self 的含义

根据 perf Wiki，perf report 中有两个核心列：

列含义用途
Self (Overhead) 该函数自身代码消耗的 CPU 时间比例（样本落在该函数指令上） 找到真正的执行热点指令
Children 该函数及其所有子调用消耗的 CPU 时间总比例 找到调用链中的性能瓶颈入口 
# 示例输出（带 Children 列）：
# Children  Self    Command  Symbol
#   100.00%  0.00%  foo      __libc_start_main
#    99.47% 10.87%  foo      main
#    86.22% 85.68%  foo      f1        ← Self 高：f1 本身是热点
#    60.82%  2.77%  foo      f2        ← Children 高：f2 调用了耗时子函数

5.4 完整分析示例

## 1. 录制数据（30秒，带调用栈）
perf record -F 99 -ag -- sleep 30

## 2. 交互式分析
perf report
# TUI 快捷键：
#   Enter  → 展开调用树
#   a      → 汇编注解（进入 perf annotate）
#   /      → 搜索符号
#   s      → 切换排序
#   ESC/q  → 返回/退出

## 3. 按 DSO（动态库）排序，定位哪个库最耗 CPU
perf report --sort dso,symbol --stdio | head -30

## 4. 导出为脚本处理
perf script | head -100
# 输出每条原始样本，可用 awk/grep 进一步过滤

📋 第五章总结

perf report 将 perf.data 中的原始地址样本转化为人类可读的函数级分析。理解 Self vs Children 是关键：Self 告诉你"在哪里花时间"，Children 告诉你"哪条调用路径导致的"。实践中应先看 Children 定位问题模块，再通过 Self 和 annotate 找到具体热点指令。

第六章perf top — 实时性能监控

perf top 类似 top 命令，但显示的是函数级的 CPU 占用，实时刷新，无需生成 perf.data 文件。它是定位当前系统 CPU 热点的最快方式。

来源：perf Wiki: Live analysis with perf top

6.1 基础用法

# 系统级实时监控（默认 cycles 事件）
perf top

# 指定采样频率
perf top -F 49

# 监控特定进程
perf top -p 1234

# 按进程名和 DSO 显示
perf top -F 49 -ns comm,dso

# 展示系统调用热点（按进程）
perf top -e raw_syscalls:sys_enter -ns comm

场景一：快速定位 CPU 热点进程和函数

# 生产环境排查 CPU 飙高的首选命令

📋 第六章总结

perf top 是实时采样模式的极简化身：同样利用 PMU 中断采样，但将统计窗口滑动展示，不落盘。对于运维场景中的"当前 CPU 为何高？"这类紧急问题，perf top 是秒级定位的利器。它的局限是样本不持久、不支持事后生成火焰图，但这正是其低开销的代价。

第七章perf annotate — 源码/汇编级注解

perf annotate 读取 perf.data，将采样命中的比例标注到每一条汇编指令（以及对应的源代码行），帮助精确定位热点在哪个机器指令上。

来源：perf Wiki: Source level analysis with perf annotate

7.1 基础用法

# 录制数据
perf record -F 99 -g ./my_program

# 汇编级注解（TUI）
perf annotate

# 文本模式
perf annotate --stdio

# 注解特定函数
perf annotate -d my_program --symbol compute_hash

# 注解内核函数（需指定 vmlinux）
perf annotate -k /tmp/vmlinux -d symbol_name

第八章Flame Graph — 火焰图可视化

火焰图（Flame Graph）由 Netflix 的 Brendan Gregg 发明，是将 perf record 采集的调用栈数据可视化的最强工具。横轴代表 CPU 时间占比，纵轴代表调用深度，宽度越大的函数条代表消耗 CPU 越多。

来源：Brendan Gregg: perf Flame Graphs · USENIX LISA 2016 Workshop Notes

8.1 火焰图原理

8.2 生成火焰图的三步法

根据 Brendan Gregg 的 LISA 2016 Workshop 笔记，火焰图生成分三步：

## 步骤 0：准备工具
git clone --depth 1 https://github.com/brendangregg/FlameGraph /opt/FlameGraph

## 步骤 1：采集调用栈数据（30秒，99Hz）
perf record -F 99 -a -g -- sleep 30

## 步骤 2：展开调用栈为可处理格式
perf script | /opt/FlameGraph/stackcollapse-perf.pl > out.folded

## 步骤 3：生成 SVG 火焰图
/opt/FlameGraph/flamegraph.pl out.folded > flamegraph.svg

## 在浏览器中打开（可交互）
firefox flamegraph.svg

8.3 高级用法

过滤 idle 线程

perf script | /opt/FlameGraph/stackcollapse-perf.pl | \
  grep -v cpu_idle | /opt/FlameGraph/flamegraph.pl > flamegraph.svg

差异火焰图（对比优化前后）

# 采集 v1（优化前）
perf record -F 99 -g ./app_v1; mv perf.data perf_v1.data
# 采集 v2（优化后）
perf record -F 99 -g ./app_v2; mv perf.data perf_v2.data

# 生成折叠格式
perf script -i perf_v1.data | /opt/FlameGraph/stackcollapse-perf.pl > v1.folded
perf script -i perf_v2.data | /opt/FlameGraph/stackcollapse-perf.pl > v2.folded

# 差异图（红=变慢，蓝=变快）
/opt/FlameGraph/difffolded.pl v1.folded v2.folded | \
  /opt/FlameGraph/flamegraph.pl > diff.svg

Off-CPU 火焰图（分析等待/阻塞时间）

perf record -e sched:sched_stat_sleep -e sched:sched_switch \
  -e sched:sched_process_exit -g -o ~/perf.data.raw ./myapp
perf inject -v -s -i ~/perf.data.raw -o ~/perf.data
perf script -i ~/perf.data | /opt/FlameGraph/stackcollapse-perf.pl | \
  /opt/FlameGraph/flamegraph.pl --color=io --title="Off-CPU" > offcpu.svg
💡 阅读火焰图的要点
  • 关注宽平台顶部（plateau）——宽且在顶端的函数是 Self 热点
  • 关注宽塔底部——说明某调用路径整体耗时长
  • 搜索特定函数名（Ctrl+F in SVG）快速定位
  • 点击任意函数可以 zoom in，只看该子树

📋 第八章总结

火焰图是 perf 数据可视化的巅峰形态，将 perf script 输出的原始调用栈，经过折叠（stackcollapse）、聚合、SVG 渲染，产生既直观又可交互的热点地图。其核心洞察是：宽度 = 时间，高度 = 深度，颜色 = 标识（非语义）。Off-CPU 火焰图将"被阻塞的时间"也纳入视野，完整呈现程序的时间去向。这是 Brendan Gregg 对性能工程界的重要贡献。

第九章perf probe — 动态追踪

perf probe 利用内核的 kprobes（内核动态探针）和 uprobes（用户态动态探针）技术，在无需重编译的情况下，在任意内核/用户态函数的入口、返回点、指定行号处插入探针，实现精细化追踪。

来源：Brendan Gregg: Dynamic Tracing

9.1 工作原理

9.2 内核动态探针（kprobe）

## 添加 tcp_sendmsg() 函数入口探针
perf probe --add tcp_sendmsg

## 添加函数返回探针（reutnr point）
perf probe 'tcp_sendmsg%return'

## 添加探针并捕获参数（需要 debuginfo）
perf probe 'tcp_sendmsg size'          # 捕获 size 参数
perf probe 'tcp_sendmsg size sk->__sk_common.skc_state'  # 深层结构体字段

## 在特定行添加探针（需 debuginfo）
perf probe 'tcp_sendmsg:81 seglen'   # 第 81 行，捕获 seglen 变量

## 添加 do_sys_open 的 filename 字符串探针
perf probe 'do_sys_open filename:string'

## 使用探针录制（带条件过滤）
perf record -e probe:tcp_sendmsg --filter 'size > 100' -a

## 查看当前已定义的动态探针
perf probe -l

## 删除探针
perf probe -d tcp_sendmsg

9.3 用户态动态探针（uprobe）

## 追踪 libc 的 malloc 函数
perf probe -x /lib64/libc.so.6 malloc

## 追踪用户态静态探针 (USDT/SDT)
perf probe -x /usr/lib64/libpthread.so %sdt_libpthread:mutex_entry

## Node.js USDT (Linux 4.10+)
perf buildid-cache --add `which node`
perf record -e sdt_node:http__server__request -a

9.4 实战：追踪 TCP 发包大小分布

## 场景：分析系统中 TCP 发包的大小分布，找出小包过多的调用路径

## 1. 添加探针并捕获 size 参数
perf probe 'tcp_sendmsg size'

## 2. 录制 30 秒，带调用栈
perf record -e probe:tcp_sendmsg -ag -- sleep 30

## 3. 用 perf script 导出原始数据
perf script

## 4. 过滤小包（size < 1500）的调用栈
perf record -e probe:tcp_sendmsg --filter 'size < 1500 && size > 0' -ag -- sleep 10
perf report

## 5. 清理探针
perf probe -d tcp_sendmsg

📋 第九章总结

perf probe 是 perf 中最强大但也最复杂的能力。它利用 kprobes/uprobes 将断点插入任意函数，在捕获参数、局部变量时，开销仅限于命中该探针的时刻，不改变系统整体行为。

第一性原理：kprobe 的本质是将目标指令的第一个字节替换为 INT3（断点指令），断点触发后由内核的 kprobe 处理程序接管，记录数据后恢复正常执行。这是一种"外科手术式"的动态插桩，不需要重启服务，是生产环境黑盒问题诊断的利器。代价是需要内核调试符号（debuginfo）来解析变量名。

第十章静态追踪 — Tracepoint

Tracepoint 是内核开发者预埋在代码中的稳定追踪点，通过 TRACE_EVENT 宏定义，提供结构化、有语义的事件数据。相比 kprobe 的动态插桩，tracepoint 是稳定的 ABI，不依赖 debuginfo，但只能在预定义位置观测。

来源：Brendan Gregg: Static Tracing

10.1 查看可用 Tracepoint

# 列出所有 tracepoint（可能数千个）
perf list tracepoint

# 过滤调度器相关
perf list 'sched:*'

# 过滤块设备
perf list 'block:*'

# 过滤系统调用
perf list 'syscalls:*'

10.2 常用追踪场景

场景一：追踪新进程启动

perf record -e sched:sched_process_exec -a
perf script
# 实时看到哪个进程在启动，由谁启动（调用栈）

场景二：上下文切换分析

# 采样上下文切换（含调用栈），找出是什么导致切换
perf record -e context-switches -ag -- sleep 10
perf report

# 追踪所有上下文切换（-c 1 表示每次都记录）
perf record -e context-switches -c 1 -ag -- sleep 5

场景三：磁盘 I/O 请求追踪与过滤

# 追踪所有块设备 I/O 请求（带调用栈）
perf record -e block:block_rq_insert -ag

# 只追踪大于 100KB 的写请求
perf record -e block:block_rq_complete --filter 'nr_sector > 200'

# 只追踪同步写
perf record -e block:block_rq_complete --filter 'rwbs == "WS"'

# I/O 延迟分析：同时记录发出和完成时间
perf record -e block:block_rq_issue -e block:block_rq_complete -a sleep 120
perf script | awk '...'  # 配对计算延迟

场景四：调度器事件追踪

# 采样调度切换，找出哪些任务频繁被切换
perf record -e sched:sched_switch -a sleep 10
perf report

# 追踪 ext4 文件系统所有事件
perf record -e 'ext4:*' -o /tmp/perf.data -a

# 追踪 kswapd 内存回收唤醒
perf record -e vmscan:mm_vmscan_wakeup_kswapd -ag

场景五：网络事件

# 按进程统计网络发包（实时）
stdbuf -oL perf top -e net:net_dev_xmit -ns comm | strings

# 记录所有 TCP 连接建立
perf record -e syscalls:sys_enter_connect -ag -- sleep 30
perf report
📌 Tracepoint 过滤器语法 --filter 'expr' 支持：字段比较（==、!=、>、<）、字符串匹配（~ 通配符）、逻辑组合（&&、||）。字段名来自内核 tracepoint 定义的 fields。

📋 第十章总结

Tracepoint 的优势在于稳定性和结构化：内核开发者为每个 tracepoint 定义了精确的字段格式，不依赖 debuginfo，跨内核版本 ABI 相对稳定，且支持内核过滤器减少数据量。相比 kprobe，tracepoint 是"已知问题域"的首选：知道要看什么（调度、I/O、网络）时，直接用 tracepoint；需要在任意位置观测时，才用 kprobe。两者组合使用，覆盖了绝大多数内核层面的性能分析场景。

第十一章perf trace — 低开销系统调用追踪

perf trace 是 strace 的低开销替代品（内核 3.7+ 引入）。它利用内核 tracepoint 而非 ptrace 机制追踪系统调用，对目标程序的干扰远小于 strace，适合在性能敏感场景中使用。

来源：Brendan Gregg perf Examples — perf trace 章节

11.1 perf trace vs strace

特性straceperf trace
追踪机制ptrace（每次系统调用暂停目标进程）syscalls tracepoint（内核侧记录，异步）
开销极高（100x 以上减速）低（通常 <10%）
生产环境适用❌ 不推荐✅ 可使用
输出格式每次调用详细参数时间戳、耗时、调用名
过滤能力基础支持 tracepoint filter
系统全局追踪❌✅ -a

11.2 基础用法

# 追踪命令的系统调用（类 strace）
perf trace ls

# 追踪特定进程
perf trace -p 1234

# 系统全局追踪系统调用（5秒）
perf trace -a sleep 5

# 只追踪特定系统调用
perf trace -e open,read,write ./my_program

# 统计系统调用次数（不显示明细）
perf trace -s ./my_program

# 统计全系统的系统调用（10秒）
perf trace -s -a sleep 10

11.4 实战场景

场景：定位服务响应慢是否因系统调用延迟

## 1. 追踪 nginx 进程的系统调用耗时分布
perf trace -p $(pgrep -n nginx) -- sleep 30 2>&1 | head -100

## 2. 统计汇总，找出最耗时的调用类型
perf trace -s -p $(pgrep -n nginx) -- sleep 10

## 3. 配合 perf record 深入分析高延迟 read() 调用
perf record -e syscalls:sys_enter_read -e syscalls:sys_exit_read \
  -ag -p $(pgrep -n nginx)

📋 第十一章总结

perf trace 的核心价值在于以极低开销实现系统调用级别的追踪。相比 strace 的 ptrace 机制（每次系统调用都要暂停目标进程、上下文切换 4 次），perf trace 通过内核 tracepoint 异步记录，理论上与普通 perf record 开销相当。这使其成为线上服务系统调用诊断的可行工具，而 strace 在生产环境中几乎是禁忌。

第十二章调度、锁与内存分析

perf 提供了专门针对调度延迟（perf sched）、锁竞争（perf lock）、内存访问（perf mem）的专用子命令，以及用于分析睡眠/等待时间的场景化追踪方案。

来源：perf Wiki: Other Scenarios · Linux Kernel Labs: Kernel Profiling

12.1 perf sched — 调度延迟分析

perf sched 专门用于追踪和分析调度器行为，找出延迟来源。

## 录制调度事件
perf sched record -a sleep 10

## 显示调度延迟统计（每个任务的 wakeup→run 延迟）
perf sched latency

## 按时间顺序显示调度事件

12.4 睡眠/等待时间分析（Off-CPU）

根据 perf Wiki: Profiling sleep times，分析程序等待在哪里需要组合追踪调度事件：

## 步骤 1：录制睡眠和切换事件
perf record -e sched:sched_stat_sleep \
             -e sched:sched_switch \
             -e sched:sched_process_exit \
             -g -o ~/perf.data.raw ./my_app

## 步骤 2：注入合并（将 sleep 与对应的 switch 配对）
perf inject -v -s -i ~/perf.data.raw -o ~/perf.data

## 步骤 3：分析报告
perf report --stdio --show-total-period -i ~/perf.data

## 示例输出：
#   100.00%  502408738  foo  __schedule

perf kmem stat  # 分配大小分布统计

📋 第十二章总结

专项子命令是 perf 将通用采样能力封装为特定领域解决方案的体现。perf sched 关注"任务何时能运行"，perf lock 关注"等锁等多久"，perf mem/c2c 关注"内存访问是否引发缓存行竞争"。睡眠时间分析（Off-CPU）通过将 sched_stat_sleep 与 sched_switch 事件配对，完整呈现程序"不在 CPU 上的时间"的分布，弥补了 CPU 采样看不到等待时间的缺陷。

以下 Top 10 命令整理自 Brendan Gregg 的 perf 示例页和 perf 官方 Wiki，覆盖最常见的性能诊断场景，是日常工作的第一参考。

第十四章权限控制与常见问题

使用 perf 时经常遇到权限错误、符号缺失、调用栈截断等问题。本章整理了最常见的障碍及解决方案，内容来自 perf Wiki 访问控制章节。

14.1 perf_event_paranoid 权限控制

Linux 通过 /proc/sys/kernel/perf_event_paranoid 控制非 root 用户的 perf 使用范围：

值含义安全性
-1所有用户可使用所有事件（包括原始 tracepoint）❌ 极不安全
0非 root 用户可使用原始 tracepoint（含 perf trace）⚠️ 开发环境
1非 root 用户可使用 CPU 硬件事件（stat/record）✅ 推荐开发
2只允许用户空间 profiling，无内核事件✅ 生产默认
3完全禁止非 root 使用 perf✅ 高安全
# 临时修改（重启后恢复）
sudo sysctl -w kernel.perf_event_paranoid=1

# 永久修改（写入 sysctl.conf）
echo "kernel.perf_event_paranoid = 1" | sudo tee -a /etc/sysctl.conf
sudo sysctl -p

# 允许非 root 看到内核符号
sudo sysctl -w kernel.kptr_restrict=0

14.2 符号缺失问题

现象原因解决方案
显示 [unknown] 或十六进制地址缺少调试符号安装 -dbgsym / -debuginfo 包
内核符号显示 [kernel.kallsyms] 地址kptr_restrict=1sysctl -w kernel.kptr_restrict=0

Node.js 函数缺失JIT 无符号--perf_basic_prof_only_functions

14.3 调用栈截断/丢失

# 问题：调用栈只有 1~2 层
# 原因 1：编译优化省略了帧指针（默认 -O2 行为）
# 解决：改用 DWARF 模式
perf record --call-graph dwarf ./my_program

# 解决：重新编译时保留帧指针
gcc -O2 -fno-omit-frame-pointer -o myapp myapp.c

# 或 Java 应用
java -XX:+PreserveFramePointer -jar myapp.jar

14.4 文件描述符限制

根据 perf Wiki 文件限制章节，perf 每个事件每个线程/CPU 消耗一个文件描述符。大规模监控时可能触发限制：

# 查看当前限制
ulimit -n

# 临时增大（需 root）
ulimit -n 65536

# 计算示意：16 核系统，100 线程进程，监控 1 个事件
# = 100 × 1 × 16 = 1600 个 fd

14.5 容器/虚拟机限制

⚠️ 容器环境注意 Docker 容器默认 seccomp profile 会阻止 perf_event_open，硬件计数器不可用。只有软件事件（task-clock、page-faults、context-switches）可用。测试环境可以 --privileged 运行，生产环境不推荐。 
# 容器内软件事件（无需特权）
perf stat -e task-clock,context-switches,page-faults ./workload

# 特权容器（仅开发/测试）
docker run --privileged --pid=host my-image perf stat ./workload

14.6 开销评估

使用方式典型开销说明
perf stat（计数模式）<1%只累加计数，无采样
perf record -F 991~5%99Hz 采样，生产可用
perf record -F 9995~15%高频采样，短时间使用
perf record -e tracepoint -c 1事件相关全量追踪，高频事件开销大
perf probe + 高频事件高谨慎在生产使用

📋 第十四章总结（全书总结）

perf 的第一性原理综述：

perf 的核心是通过 perf_event_open() 将 CPU PMU 硬件能力和内核 ftrace/kprobe 追踪能力统一抽象。它以最小干扰换取全栈可观测性——从硬件微架构（cycles、cache-miss），到内核调度/I/O/网络，到用户态任意函数，无一不可观测。

使用原则：先宏观后微观（stat → top → record → annotate），先假设后验证采样而非全量追踪（降低开销），火焰图为终态可视化。掌握这一工具，意味着能在不修改代码、不重启服务的前提下，对任何 Linux 系统进行深度、低干扰的性能诊断。
