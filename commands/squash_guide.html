概述：什么是 SquashFS

SquashFS 是一个高度压缩的 只读文件系统，专为 Linux 设计。它使用 gzip、xz、lzo、lz4、zstd 等算法对文件数据、inode 及目录信息进行压缩。 inode 紧凑存储，所有数据块紧密排列以最小化额外开销。 支持最大 1 MiB 的数据块（默认 128 KiB），最大文件系统体积可达 264 字节。

官方定义（Debian Manpage）：
"Squashfs is intended for general read-only filesystem use, for archival use (i.e. in cases where a .tar.gz file may be used), and in constrained block device/memory systems (e.g. embedded systems) where low overhead is needed."

SquashFS 工具链由 Phillip Lougher 编写，源代码托管于 github.com/plougher/squashfs-tools。 核心工具为 mksquashfs（创建镜像）和 unsquashfs（解压/提取镜像）。 SquashFS 于 2009 年随 Linux 2.6.29 合并进内核主线。

🔧 mksquashfs

将目录或文件打包为 .squashfs / .sqfs / .sfs 镜像文件，支持多源合并、文件排除、伪文件定义等高级功能。

📦 unsquashfs

从 SquashFS 镜像中列举、提取文件，支持通配符过滤、正则提取、查看超级块元数据等功能。

📌 本节小结 · 第一性原理

SquashFS 的核心设计哲学是：以空间换速度的逆向策略——用大块压缩（128 KiB 默认块）换取高压缩比，用只读约束换取零碎片化和确定性布局。文件系统本质是一个带索引的压缩归档，内核在访问时按需解压，读取代价换存储收益。

SquashFS 文件系统内部结构与原理

根据 Linux 内核官方文档（Documentation/filesystems/squashfs.rst）及 SquashFS 二进制格式规范， 一个 SquashFS 文件系统镜像由最多 九个部分 顺序拼接而成，字节对齐。

📎 内核文档：github.com/torvalds/linux/.../squashfs.rst
📎 格式规范：dr-emann.github.io/squashfs/

1.1 磁盘布局（九段结构）

1.2 超级块（Superblock）

超级块是 SquashFS 镜像的第一个 96 字节 结构，记录文件系统的所有关键元信息。 格式规范来源：dr-emann.github.io/squashfs/

字段类型说明
magicu32必须为 0x73717368（"sqsh"）
inode_countu32inode 总数
modification_timeu32创建时间戳（无符号，2106年才过期）
block_sizeu32数据块大小，必须为 4096~1048576 之间的 2 的幂
compression_idu161=GZIP 2=LZMA 3=LZO 4=XZ 5=LZ4 6=ZSTD
version_major/minoru16当前版本为 4.0
root_inode_refu64根目录 inode 引用
bytes_usedu64有效字节数（不含填充）
inode_table_startu64inode 表偏移
directory_table_startu64目录表偏移
fragment_table_startu64碎片表偏移

1.3 数据块与碎片（Datablocks & Fragments）

文件内容被切割为固定大小的数据块（默认 128 KiB），每块独立压缩后写入磁盘。 文件大小不是块大小整数倍时，尾部剩余数据（tail-end）可以：

  • 作为一个不完整块（short block）独立存储
  • 或与其他文件的尾部合并到一个 碎片块（Fragment Block） 中共同压缩，进一步提升压缩率
重复文件去重： mksquashfs 默认检测内容相同的文件（MD5/SHA1 哈希），只写入一份物理数据，通过硬链接引用，显著减小镜像体积。

1.4 元数据压缩（Inode / Directory）

元数据（inode 和目录）以 8 KiB 为单位组织成元数据块，每块单独压缩。 每个压缩块前缀一个 u16 长度头：最高位若置 1 表示该块未压缩（压缩后反而更大时自动回退）。 inode 不按块边界对齐，可横跨两个元数据块，用 48 位引用（块偏移 + 块内偏移）定位。

1.5 支持的压缩算法对比

算法ID压缩率速度典型场景
gzip1中快通用，默认选项
lzma2高慢已废弃，内核不支持
lzo3低极快嵌入式低功耗设备
xz4极高慢发行版 Live CD、固件
lz45低极快对解压速度要求极高场景
zstd6高快现代推荐，压缩/解压均衡
📌 本节小结 · 第一性原理

SquashFS 的设计核心是最小化存储开销：① 大块压缩（128KiB 块比 4KiB 块拥有更多上下文，压缩率更高）；② 元数据高密度打包（inode 平均 8 字节，不按块边界对齐）；③ 重复内容去重；④ 只读约束消除了写时复杂性，使文件系统结构完全确定性，可直接 mmap 使用。

mksquashfs 工作原理与核心选项

📎 官方 Man Page：manpages.debian.org → mksquashfs(1)

2.1 处理流水线

mksquashfs 采用多线程流水线架构（默认使用所有 CPU 核心）：

2.2 基础语法

mksquashfs source1 [source2 ...] FILESYSTEM [OPTIONS] [-e exclude_files]

2.3 核心常用选项

选项说明示例
-comp COMP选择压缩算法（gzip/xz/zstd/lzo/lz4/lzma）-comp zstd
-b BLOCK_SIZE数据块大小，默认 128K，最大 1M-b 1M
-processors N使用 N 个 CPU 核心并行压缩-processors 4
-e file1 file2排除指定文件/目录-e /proc /sys
-wildcards启用通配符匹配（配合 -e）-wildcards -e "*.log"
-noappend覆盖已有镜像而非追加-noappend
-all-root将所有文件属主改为 root-all-root
-reproducible生成可重现构建镜像（默认开启）-reproducible
-p "pseudo-def"定义伪文件（设备节点/目录/命令输出）见下方示例
-action "ACTION@EXPR"条件动作（按属性匹配文件执行操作）见下方示例
-no-duplicates禁用重复文件检测-no-duplicates
-info打印每个文件的压缩信息-info

2.4 使用示例

# 基础打包：将目录打包为镜像
mksquashfs /some/dir image.sqfs

# 使用 zstd 算法 + 1MiB 块（更高压缩率）
mksquashfs /some/dir image.sqfs -b 1M -comp zstd

# 排除特定文件
mksquashfs /some/dir image.sqfs -e /some/dir/tmp /some/dir/.git

# 通配符排除（注意加引号防 shell 展开）
mksquashfs /some/dir image.sqfs -wildcards -e "*.log" "... *.tmp"

# 伪文件：创建设备节点（无需实际文件）
mksquashfs /rootfs firmware.sqfs -p "dev/console c 0600 0 0 5 1"

# 动作：jpg/mpg 文件不压缩（已是压缩格式）
mksquashfs /dir image.sqfs -action "uncompressed@(name(*.jpg)||name(*.mpg))"

# 动作：修复不可被其他用户读取的文件权限
mksquashfs /dir image.sqfs -action "chmod(o+r)@! perm(o+r)"

# 追加新文件到已有镜像
mksquashfs /new/files image.sqfs
📌 本节小结

mksquashfs 本质是一个流水线式压缩打包器：先遍历源目录建立文件树，再多线程并行压缩数据块，最后串行写出元数据表并回写超级块。理解其流水线结构有助于优化参数选择：大块 + 强压缩适合存储受限场景，小块 + 快速压缩适合随机访问场景。

unsquashfs 工作原理与核心选项

📎 官方 Man Page：manpages.debian.org → unsquashfs(1)

3.1 解压流程

3.2 基础语法

unsquashfs [OPTIONS] FILESYSTEM [files/dirs to extract or exclude]

3.3 核心选项

选项说明
-d PATHNAME指定解压目标目录（默认 ./squashfs-root）
-l / -ls列出文件系统内容，不提取文件
-ll / -lls以 ls -l 格式列出（含属性），不提取
-lln / -llnumeric同 -lls 但显示数字 uid/gid
-s / -stat显示超级块信息（压缩算法、块大小、版本等）
-f / -force若文件已存在则覆盖
-excludes将命令行文件参数作为排除列表
-r / -regex使用 POSIX 正则表达式匹配
-p N / -processors N使用 N 个 CPU 核心解压
-max-depth N最大递归深度
-cat将指定文件内容输出到 stdout
-pf FILE输出等效伪文件定义（用于 mksquashfs 重建）
-o BYTES跳过文件开头 N 字节（处理嵌入偏移）

3.4 使用示例

# 提取到默认目录 squashfs-root
unsquashfs image.sqfs

# 指定目标目录
unsquashfs -d /tmp/extracted image.sqfs

# 列出内容（不提取）
unsquashfs -ll image.sqfs | head -30

# 查看镜像元信息
unsquashfs -s image.sqfs

# 只提取特定目录
unsquashfs image.sqfs fs/squashfs

# 通配符提取（引号保护）
unsquashfs image.sqfs "[Tt]est/example*"

# 排除特定模式（提取除 *.gz 之外所有文件）
unsquashfs -excludes image.sqfs "test/*data*.gz"

# 正则提取
unsquashfs -r image.sqfs "^lib/.*\.so$"

# 将文件内容输出到 stdout（无需解压）
unsquashfs -cat image.sqfs etc/os-release

3.5 退出状态码

0 正常列举或提取完成
1 致命错误（镜像损坏/IO错误），已中止
2 非致命错误（如 xattr 不支持），继续执行
📌 本节小结

unsquashfs 的本质是带索引的随机访问解压器：借助超级块中的偏移量表，可直接定位到任意文件的 inode，进而找到其数据块，无需顺序扫描整个镜像。这使得"只提取某个子目录"的操作非常高效，远优于 tar -xf | grep 的线性扫描方式。

Top 10 使用场景详解

以下场景选取依据：使用频率广泛、具有代表性、在官方文档/权威资料中有据可查。覆盖嵌入式固件、发行版制作、现代软件打包、HPC 存储优化等主流方向。

1
嵌入式系统固件制作（OpenWrt / DD-WRT / 路由器）
场景类型：固件存储 | 压缩要求：极致 | 典型算法：lzma / xz
📎 参考：Wikipedia: SquashFS — 嵌入式发行版 ｜ SquashFS HOWTO §4.3

背景与原理

嵌入式设备（路由器、IoT 网关）的存储介质通常只有 4~32 MiB 的 Flash，且为只读。SquashFS 是 OpenWrt、DD-WRT 等路由器固件的标准根文件系统格式。 Wikipedia 明确指出："Squashfs is used on embedded distributions such as the OpenWrt and DD-WRT router firmware."

Flash 固件分区通常由 bootloader → kernel → squashfs 根文件系统三部分组成。squashfs 镜像直接写入 Flash 分区，内核挂载时按需解压到内存，节省宝贵的 Flash 空间。

工作流程图

命令示例

# 打包嵌入式根文件系统，所有文件属主强制为 root
mksquashfs /path/to/rootfs firmware.sqfs \
    -comp xz -b 256K -all-root \
    -e /path/to/rootfs/proc \
    -e /path/to/rootfs/sys

# 写入 Flash 设备（mtd 分区）
dd if=firmware.sqfs of=/dev/mtdblock2

# 逆向：从路由器固件提取 squashfs（通常有偏移）
unsquashfs -o 0x40000 firmware.bin
原理说明

第一性原理： Flash 存储不可随机写入（必须整块擦除），只读文件系统天然契合 Flash 特性。xz 压缩在 ARM 嵌入式 CPU 上解压速度满足要求，而压缩比极高，可在 4 MiB 内塞入完整 Linux 系统。-all-root 确保跨用户构建环境的权限一致性，避免安全漏洞。

2
Live CD / Live USB 发行版制作
场景类型：系统分发 | 压缩要求：高 | 典型算法：xz / zstd
📎 参考：Wikipedia: SquashFS — Live CD ｜ SquashFS HOWTO §4.3.2

背景与原理

Ubuntu、Debian、Arch Linux、Kali Linux 等发行版的 Live CD/USB 均使用 SquashFS 作为根文件系统载体。 Wikipedia 记载："Squashfs is used by the Live CD versions of Arch Linux, Clonezilla, Debian, Fedora, Gentoo Linux, KDE neon, Kali Linux, Linux Mint, NixOS, Ubuntu, openSUSE..."

Live 系统通常将 filesystem.squashfs 存储在 ISO 9660 光盘或 FAT32 USB 设备上，内核启动后通过 loop 设备挂载，再配合 OverlayFS（或 UnionFS/aufs）叠加一个内存中的读写层，实现完整的可用系统体验。

命令示例

# 制作 Live 系统根文件系统（排除伪文件系统）
mksquashfs /live/chroot filesystem.squashfs \
    -comp xz -b 1M \
    -e /live/chroot/proc \
    -e /live/chroot/sys \
    -e /live/chroot/dev \
    -e /live/chroot/run

# 挂载 Live 系统（验证）
mount -t squashfs -o loop filesystem.squashfs /mnt/squash

# 叠加 OverlayFS（Live 系统启动脚本中）
mount -t tmpfs tmpfs /run/overlay/rw
mount -t overlay overlay \
    -o lowerdir=/mnt/squash,upperdir=/run/overlay/rw/upper,workdir=/run/overlay/rw/work /

# 检查已有 Live 镜像的内容
unsquashfs -ll filesystem.squashfs | grep "^squashfs-root/etc"
原理说明

第一性原理： ISO/USB 介质本身不可修改（或用户不应修改），SquashFS 只读特性完美匹配。OverlayFS 的写时复制（Copy-on-Write）机制将"写到 tmpfs"，关机即丢弃，实现了"用完即走"的 Live 体验。xz 在 ISO 中选择 1MiB 块可获得接近极限的压缩率，适合网络下载发行场景。

3
Snap 包管理系统（Ubuntu Snap）
场景类型：软件分发 | 压缩要求：中 | 典型算法：xz / zstd
📎 参考：Wikipedia: SquashFS — "The snap package system also uses Squashfs as its file container format"

背景与原理

Canonical 的 Snap 包格式本质上是一个 SquashFS 镜像（.snap 文件）。每个 snap 包含应用程序的完整依赖环境，安装时通过 loop 设备挂载到 /snap/<name>/<revision>/，以只读方式运行。这提供了应用隔离、版本回滚、原子更新等特性。

snap 守护进程（snapd）自动管理镜像的挂载和卸载。开发者使用 snapcraft 工具最终调用 mksquashfs 生成 .snap 文件。

命令示例

# 查看已安装的 snap 包（均为 squashfs loop 设备）
losetup -l | grep snap

# 检查 snap 包内容（不需要 root）
unsquashfs -ll /var/lib/snapd/snaps/firefox_*.snap | head -30

# 提取 snap 包以检查内容
unsquashfs -d /tmp/snap-content /var/lib/snapd/snaps/vlc_*.snap

# 查看 snap 包压缩元信息
unsquashfs -s /var/lib/snapd/snaps/core22_*.snap

# 手动创建 snap 包（snapcraft 底层调用）
mksquashfs prime/ my-app.snap \
    -noappend -comp xz -no-xattrs \
    -no-fragments
原理说明

第一性原理： Snap 用 SquashFS 实现"不可变软件包"——每个版本的 snap 是一个内容哈希确定的镜像，无法被篡改（只读挂载）。多版本共存、原子切换通过切换 loop 挂载点实现，回滚只需重新挂载旧版本镜像。这避免了传统包管理的"依赖地狱"。

4
AppImage 便携应用打包
场景类型：便携软件分发 | 压缩要求：中 | 典型算法：zstd / gzip
📎 参考：Wikipedia: SquashFS — "The AppImage project, which aims to create portable Linux applications, uses Squashfs for creating AppImages."

背景与原理

AppImage 是一种"一文件即应用"的 Linux 软件分发方式。每个 AppImage 文件本质是一个 ELF 运行时（Runtime）+ SquashFS 镜像 的拼接体：文件开头是一段可执行的 ELF stub，当用户执行该文件时，stub 将文件中的 SquashFS 部分通过 FUSE（squashfuse）挂载到临时目录，然后运行其中的应用程序。

命令示例

# 查看 AppImage 内部内容（利用 -offset 跳过 ELF 头）
unsquashfs -o $(./MyApp.AppImage --appimage-offset) -ll MyApp.AppImage

# 提取 AppImage 内容
unsquashfs -o $(./MyApp.AppImage --appimage-offset) \
    -d extracted_app MyApp.AppImage

# 或：AppImage 自解压（标准方式）
./MyApp.AppImage --appimage-extract

# 手动制作 AppImage（appimagetool 底层调用 mksquashfs）
mksquashfs AppDir/ my.squashfs -root-owned -noappend -comp zstd
cat runtime my.squashfs > MyApp.AppImage
chmod +x MyApp.AppImage
原理说明

第一性原理： AppImage 利用 SquashFS 的两个关键特性：① 偏移挂载（-offset）使 squashfs 镜像可以拼接在任意二进制文件之后；② FUSE 挂载（squashfuse）无需 root 权限即可挂载，用户态直接访问压缩内容，运行完卸载。这实现了真正的"下载即用、无需安装"。

5
只读归档备份（替代 .tar.gz）
场景类型：数据归档 | 压缩要求：可选 | 典型算法：zstd / xz
📎 参考：mksquashfs man page ｜ SquashFS HOWTO §4.2

背景与原理

SquashFS HOWTO 明确指出："For archival use (i.e. in cases where a .tar.gz file may be used)。" 与 tar.gz 相比，SquashFS 作为归档格式有以下优势：

✅ SquashFS 优势
  • 可直接 mount 挂载，无需解压
  • 支持随机访问单个文件
  • 保留完整文件系统语义（权限、符号链接、硬链接、xattr）
  • 重复文件自动去重
  • 可并行解压
⚠️ tar.gz 局限
  • 访问任意文件需顺序解压
  • 无法直接挂载
  • 不支持随机访问
  • 大文件访问 I/O 开销大

命令示例

# 将大型归档目录制作为可挂载的 squashfs
mksquashfs /var/arch /var/arch.sqsh

# 挂载以访问（无需解压）
mkdir -p /mnt/arch
mount -t squashfs -o loop /var/arch.sqsh /mnt/arch
ls /mnt/arch

# 写入 /etc/fstab 实现开机自动挂载
echo "/var/arch.sqsh  /var/arch  squashfs  ro,defaults  0 0" >> /etc/fstab

# 从已有归档中直接提取特定文件（快速随机访问）
unsquashfs -d /tmp/out /var/arch.sqsh "project/src/main.c"

# 对比压缩率（-info 显示每个文件）
mksquashfs /var/arch test.sqfs -info -comp zstd 2>&1 | tail -5
原理说明

第一性原理： SquashFS 是"可寻址的压缩归档"——借助 inode 表和目录索引，访问任意文件的时间复杂度为 O(depth)，而 tar 为 O(n)。对于包含海量小文件的代码库或媒体库，SquashFS 的访问效率远优于 tar 解压后操作，且节省磁盘空间。

6
容器镜像层与不可变根文件系统（Docker / OCI / Talos Linux）
场景类型：云原生 | 压缩要求：中 | 典型算法：zstd
📎 参考：Talos Linux SquashFS Root Filesystem 分析 ｜ LUMI HPC SquashFS + Singularity

背景与原理

Talos Linux 等不可变操作系统将整个根文件系统打包为 SquashFS 镜像，内核启动后挂载为只读根，再用 OverlayFS 叠加可写层到特定目录（如 /var）。这实现了：

  • 完整性保证：根文件系统 SHA256 可在启动时验证（Secure Boot 信任链）
  • 原子更新：新版本镜像替换旧镜像，重启后生效
  • 系统扩展：额外的 SquashFS 层叠加在基础镜像之上

在容器场景，OCI 镜像层（layer）也可以 SquashFS 格式存储（eStargz/nydus 等格式）以支持懒加载。

命令示例

# Talos Linux 风格：制作不可变根文件系统
mksquashfs /rootfs talos-rootfs.sqsh \
    -comp zstd -Xcompression-level 19 \
    -all-root -reproducible \
    -mkfs-time 0

# 验证镜像完整性
sha256sum talos-rootfs.sqsh

# 启动时挂载（initramfs 脚本）
mount -t squashfs -o ro,loop talos-rootfs.sqsh /sysroot
mount -t overlay overlay \
    -o lowerdir=/sysroot,upperdir=/run/writable,workdir=/run/work /

# 检查不可变根文件系统内容
unsquashfs -ll talos-rootfs.sqsh | grep "^squashfs-root/usr/bin"

# 系统扩展：叠加额外 SquashFS 层
mount -t overlay overlay \
    -o lowerdir=/sysroot/extension.sqfs:/sysroot,upperdir=... /
原理说明

第一性原理： 不可变基础设施（Immutable Infrastructure）要求"基础层不可修改"——SquashFS 只读特性使其成为理想载体。镜像哈希唯一确定内容，配合 Secure Boot 实现从固件到操作系统的完整信任链。-reproducible 选项（默认启用）消除时间戳等非确定性因素，使相同输入永远产生相同字节序列的镜像。

7
HPC 超算系统大文件聚合（减轻 Lustre 元数据压力）
场景类型：高性能计算 | 压缩要求：中 | 典型算法：zstd / lz4
📎 参考：LUMI Supercomputer: Squashfs and FUSE 官方文档

背景与原理

LUMI 超算官方文档明确记载："On HPC systems like LUMI, it reduces the load on parallel filesystems like Lustre by packaging numerous small files into a single, compressed file."

HPC 并行文件系统（如 Lustre）对元数据操作（stat、open、readdir）有严格限制。大量小文件（如 Python 环境、深度学习数据集）的 import 操作会产生海量元数据请求，造成系统瓶颈。将小文件打包为单个 SquashFS 镜像，再通过 squashfuse_ll（FUSE 用户态挂载）或 Singularity 容器挂载，可将成千上万次元数据请求减少为一次文件打开。

命令示例

# 将 Python 环境打包（LUMI 官方示例）
mksquashfs data data.sqsh -processors 4

# 通过 squashfuse 挂载（无需 root）
squashfuse_ll data.sqsh $MOUNT_DIR &

# Singularity 容器中使用（最常见方式）
singularity exec --bind data.sqsh:/data:image-src=/ mycontainer.sif python train.py

# 打包大型 Python 虚拟环境
mksquashfs /path/to/venv pytorch_env.sqfs \
    -comp zstd -Xcompression-level 3 \
    -processors 16 -mem 8G

# 脚本：节点本地挂载（SLURM 作业脚本片段）
# 使用 $SLURM_LOCALID 确保每节点只挂载一次
if [ $SLURM_LOCALID -eq 0 ]; then
    squashfuse_ll pytorch_env.sqfs /mnt/env
fi
原理说明

第一性原理： Lustre 等并行文件系统的元数据服务是全局共享的稀缺资源，每次文件系统调用（stat/open）都有网络往返开销。SquashFS 将元数据索引（inode 表、目录表）打包在单个文件内，访问时本地解压，将 N 次分布式元数据请求压缩为 1 次顺序 I/O + 本地内存操作，吞吐量提升可达数量级。

8
内核 initramfs / initrd 构建
场景类型：系统启动 | 压缩要求：速度优先 | 典型算法：lz4 / xz
📎 参考：SquashFS HOWTO §2.2 — initrd 支持 ｜ Linux kernel docs: squashfs.rst

背景与原理

initramfs（初始 RAM 文件系统）是内核启动后最先挂载的文件系统，包含挂载真正根文件系统所需的最小工具集（udev、busybox、驱动模块等）。SquashFS HOWTO 中指出："It is only obligatory to compile SquashFS inside the kernel if you plan using squashed initial RAM disks (initrd)."

将 initramfs 制作为 SquashFS 格式可显著减小 initramfs 体积（相比 cpio.gz 可小 30~50%），减少从 Flash/SSD 读取到内存的数据量，加快启动速度。内核会在加载时自动解压 SquashFS 格式的 initramfs。

命令示例

# 制作最小化 initramfs（floppy 示例，来自官方 HOWTO）
# 创建根文件系统，排除 /usr 目录
mksquashfs floppylinux root.sqsh -e usr

# 制作 /usr 独立分区
mksquashfs floppylinux/usr usr.sqsh

# 写入启动磁盘
cp root.sqsh /mnt/floppy/boot
dd if=usr.sqsh of=/dev/fd0

# 现代场景：构建嵌入式最小 initramfs
mksquashfs /tmp/initramfs_root initramfs.sqfs \
    -comp lz4 -Xhc \
    -no-exports -no-xattrs \
    -all-root

# 将 squashfs 格式 initramfs 嵌入内核构建
# 在 Linux kernel .config 中：
# CONFIG_INITRAMFS_SOURCE="/path/to/initramfs.sqfs"

# 提取现有系统的 initramfs 内容以分析
unsquashfs -d /tmp/initrd-content /boot/initrd.img-$(uname -r)
原理说明

第一性原理： initramfs 的关键约束是"启动时间"——压缩越小，从存储读取越快；但解压也消耗时间。lz4 以极快的解压速度（数 GB/s）换取中等压缩率，适合 CPU 速度充裕但存储 I/O 是瓶颈的场景；xz 则适合 Flash 存储极为有限的嵌入式场景。

9
固件逆向工程与修改（解包 → 修改 → 重新打包）
场景类型：安全研究 / 固件修改 | 工具：unsquashfs + mksquashfs
📎 参考：unsquashfs man page: -offset 选项 ｜ mksquashfs man page

背景与原理

路由器、NAS、IP 摄像头等嵌入式设备固件通常以二进制文件形式发布，内部包含 SquashFS 文件系统，但可能有特定的起始偏移（前面有文件头、内核镜像等）。unsquashfs -offset 选项允许跳过指定字节数直接读取 SquashFS 分区，-stat 可在不解压的情况下验证是否为有效 SquashFS。

命令示例

# 用 binwalk 找到 squashfs 偏移量
binwalk firmware.bin | grep -i squash
# 假设输出：DECIMAL: 2621440 (0x280000)  SquashFS filesystem

# 验证偏移处是否为有效 SquashFS
unsquashfs -s -o 2621440 firmware.bin

# 提取 SquashFS 内容
unsquashfs -o 2621440 -d extracted firmware.bin

# 修改提取的内容...
echo "my_backdoor" >> extracted/etc/passwd  # 仅示意

# 重新打包为 SquashFS（需与原始参数匹配）
mksquashfs extracted/ new_rootfs.sqfs \
    -comp lzma -b 131072 -all-root -noappend

# 将新镜像写回固件二进制（替换原有 squashfs 区域）
dd if=new_rootfs.sqfs of=firmware_modified.bin bs=1 seek=2621440 conv=notrunc
原理说明

第一性原理： 固件逆向利用了 SquashFS 格式的自描述性——超级块中的魔数 0x73717368 可在二进制中精确定位，-offset 使工具具备对任意偏移的容错性。理解 SquashFS 内部结构是安全研究和固件自定义的基础，但需注意法律和道德边界。

10
可重现构建（Reproducible Builds）
场景类型：CI/CD 安全 | 关键选项：-reproducible / SOURCE_DATE_EPOCH
📎 参考：mksquashfs man page: -reproducible / SOURCE_DATE_EPOCH ｜ reproducible-builds.org: SOURCE_DATE_EPOCH

背景与原理

可重现构建（Reproducible Builds）要求：给定相同的源代码和构建环境，任意时间的构建产物字节完全一致。这对于安全审计至关重要——任何人都可以验证发布的二进制与源代码严格对应，排除供应链攻击。

mksquashfs 的 -reproducible 选项（Debian Bookworm 版本默认开启）消除了构建时间戳等不确定因素。环境变量 SOURCE_DATE_EPOCH 可统一所有文件时间戳，使跨时间、跨机器构建产出完全一致的镜像。

命令示例

# 设置 SOURCE_DATE_EPOCH（Git 最后提交时间）
export SOURCE_DATE_EPOCH=$(git log -1 --format=%ct)

# mksquashfs 会自动读取 SOURCE_DATE_EPOCH
# 所有超过此时间的文件时间戳被截断至 SOURCE_DATE_EPOCH
mksquashfs /source image.sqfs -reproducible

# 等效显式写法（-reproducible 默认已开启）
mksquashfs /source image.sqfs \
    -reproducible \
    -mkfs-time $SOURCE_DATE_EPOCH \
    -all-time $SOURCE_DATE_EPOCH

# 验证两次构建结果一致
sha256sum image1.sqfs image2.sqfs
# 输出：两行 SHA256 应完全相同

# 使用 unsquashfs 验证时间戳已统一
unsquashfs -ll image.sqfs | awk '{print $6, $7}' | sort -u
原理说明

第一性原理： 不确定性的来源包括：① 文件时间戳（每次构建不同）；② 文件处理顺序（取决于目录遍历顺序）；③ 构建主机信息。-reproducible 固定文件排序为确定性顺序，SOURCE_DATE_EPOCH 统一时间戳，使 SquashFS 镜像成为纯粹的"内容哈希"——相同内容 → 相同字节 → 相同 SHA256，这是供应链安全的基础。

📌 全文总结 · 第一性原理

SquashFS 的设计可以用一个核心原则概括："将存储成本转化为计算成本，以确定性换灵活性"。

存储永远是嵌入式系统、发行版分发和网络传输的稀缺资源。SquashFS 通过： ① 大块压缩（128KiB 块提供足够上下文，压缩率远超小块）； ② 元数据紧凑化（inode 平均 8 字节、不对齐存储）； ③ 内容去重（相同文件只存一份）； ④ 只读约束（消除写时复杂性，实现确定性镜像）； 这四个机制，将存储成本压缩至极致，再以 CPU 解压成本（通常极低）换回运行时的透明访问。

mksquashfs 是"压缩打包器"，unsquashfs 是"带索引的随机访问解压器"。两者共同构成了现代 Linux 系统中从嵌入式固件到云原生基础设施不可或缺的工具链。
