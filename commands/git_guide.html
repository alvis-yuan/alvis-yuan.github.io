01 · 设计背景与历史

Git 是如何诞生的？

官方文档：A Short History of Git — git-scm.com

正如很多伟大的事物一样，Git 始于一场创造性的破坏与激烈的争议。Linux 内核是一个相当大型的开源软件项目。在 Linux 内核维护的早期（1991–2002），软件变更以补丁（patch）和归档文件的形式在开发者之间流通。

1991

Linux 内核诞生

Linus Torvalds 发布 Linux 内核第一版。变更通过邮件列表以 patch 文件流通，没有版本控制系统。

2002

采用 BitKeeper

Linux 内核项目开始使用商业分布式版本控制系统 BitKeeper。对开源项目免费授权，但本质上是私有软件。

2005

危机爆发 → Git 诞生

BitKeeper 与 Linux 社区的合作关系破裂，免费授权被撤销。Linus 在思考了数月后，仅用约 10 天时间编写了 Git 的初始版本。2005年4月7日，Git 第一次提交，提交信息为"Initial revision of 'git', the information manager from hell."

2008

GitHub 上线

GitHub 的出现将 Git 推广至更广泛的开发者社区，Git 开始主导整个版本控制市场。

2025

Git 20周年

GitHub 对 Linus 进行了专访。Linus 表示："Git 席卷整个 SCM 世界让我颇感意外。我本以为我只是在解决我自己的问题。"

🎯 初始设计目标

官方文档：A Short History of Git

根据官方 Pro Git 文档，新系统（Git）的目标如下：

  • 速度（Speed）：操作必须极快，哪怕是超大型项目
  • 简洁的设计（Simple design）：底层是内容寻址文件系统，逻辑清晰
  • 对非线性开发的强力支持：支持数千个并行分支
  • 完全分布式（Fully distributed）：每个开发者都拥有完整的仓库副本
  • 高效处理大型项目：能够高效管理 Linux 内核这样规模的代码库
官方原文引用 "Git was initially a toolkit for a version control system rather than a full user-friendly VCS. Git is fundamentally a content-addressable filesystem with a VCS user interface written on top of it." — Pro Git Book, git-scm.com

03 · 工作原理

快照而非差异

官方文档：What is Git? — git-scm.com

Git 与其他版本控制系统（包括 Subversion）最主要的区别在于 Git 对数据的思考方式。概念上，大多数其他系统以基于变更列表（delta-based）的方式存储信息——即把每个文件以及随时间对文件所做的变更列表存储起来。

图2：Delta-based vs Git 快照模型 — 来源：Pro Git Book, git-scm.com

官方原文 "Git thinks of its data more like a stream of snapshots of a miniature filesystem. With Git, every time you commit, or save the state of your project, Git basically takes a picture of what all your files look like at that moment and stores a reference to that snapshot. To be efficient, if files have not changed, Git doesn't store the file again, just a link to the previous identical file it has already stored." — Pro Git, What is Git?, git-scm.com

🔑 Git 的四大核心特性

官方文档：What is Git?

1. 几乎所有操作都是本地执行

Git 的大部分操作只需要本地文件和资源，通常不需要网络上其他计算机的任何信息。因为你的本地磁盘上有项目的整个历史，所以大多数操作看起来几乎是即时完成的。

2. Git 保证完整性

Git 中所有数据在存储前都会计算校验和，然后以校验和来引用。这意味着不可能在 Git 不知情的情况下更改任何文件或目录的内容。Git 用于计算校验和的机制叫做 SHA-1 哈希——一个由 40 个十六进制字符（0-9 和 a-f）组成的字符串。

3. Git 一般只添加数据

Git 几乎不会擦除任何数据，执行的操作几乎都只是往数据库中添加数据。一旦你提交快照到 Git，就很难再丢失数据，尤其是如果你定期将数据库推送到另一个仓库的话。

4. 完全分布式

每个开发者的本地都有完整的仓库副本，包括全部历史记录。这既是数据安全保障，也是协作灵活性的基础。

04 · 工作原理

Git 对象数据库

官方文档：Git Objects — git-scm.com

Git 的核心是一个内容寻址文件系统。这意味着 Git 的核心是一个简单的键值（key-value）数据存储。你可以向 Git 仓库中插入任意类型的内容，它会返回一个唯一的键，之后你可以用这个键来检索该内容。

官方原文 "Git is fundamentally a content-addressable filesystem. It means that at the core of Git is a simple key-value data store." — Pro Git, Git Objects, git-scm.com

📦 三种核心对象类型

图3：Git 三种对象类型（blob / tree / commit）及其引用关系 — 来源：Pro Git, git-scm.com

对象类型说明对应 Unix 概念
blob 存储文件内容，不存文件名。每个唯一内容只存一份（内容寻址）。 文件 inode / 内容
tree 存储目录结构：包含多个条目，每条是 blob 或子 tree 的 SHA-1 + 文件名 + 模式。 目录（directory）
commit 指向一个 tree 对象（根目录快照），包含父提交、作者、提交时间、提交消息。 版本节点
tag（附注标签） 指向某个 commit 对象，包含标签名、附注、签名等额外元数据。 带注释的书签
05 · 工作原理

三区域与三状态

官方文档：What is Git? — Three States

这是关于 Git 最重要的概念之一。Git 有三种状态，你的文件可能处于其中之一：已修改（modified）、已暂存（staged）和已提交（committed）。

图4：Git 三区域工作流 — 来源：Pro Git, What is Git?, git-scm.com

官方描述的基本工作流

  1. 在工作目录中修改文件
  2. 有选择地将你想包含在下次提交中的更改暂存（git add），这样只有那些更改会被添加到暂存区
  3. 执行提交（git commit），这会将暂存区中的文件快照永久地存储到 Git 目录中

06 · 工作原理

分支与 HEAD 指针

官方文档：Branches in a Nutshell — git-scm.com

几乎所有的版本控制系统都以某种形式支持分支。Git 的分支模型被称为它的"杀手级特性"（killer feature）。Git 分支的方式极为轻量，使分支操作几乎是瞬间完成的，并且在分支之间的来回切换也同样便捷。

官方原文 "A branch in Git is simply a lightweight movable pointer to one of these commits. Because a branch in Git is actually a simple file that contains the 40 character SHA-1 checksum of the commit it points to, branches are cheap to create and destroy. Creating a new branch is as quick and simple as writing 41 bytes to a file." — Pro Git, Branches in a Nutshell, git-scm.com

图6：Git 分支与 HEAD 指针模型 — 来源：Pro Git, Branches in a Nutshell, git-scm.com

为什么 Git 分支如此轻量？

在旧式版本控制系统中，创建分支意味着复制整个项目文件目录，对大型项目可能需要数秒甚至数分钟。而 Git 的分支本质上只是一个包含 40 字符 SHA-1 校验和的简单文件（41 字节：40字符+换行符），创建和销毁都极其廉价，几乎是瞬间完成的。

HEAD 是一个特殊指针，指向你当前所在的本地分支。当你运行 git commit 时，当前分支指针会向前移动，始终指向最新提交。

07 · 常用命令

初始化与配置

官方文档：First-Time Git Setup — git-scm.com
git config 配置 Git 用户信息与偏好设置 ▼
🔧 使用场景：首次安装 Git 后必须配置

在使用 Git 之前，你需要配置你的用户名和邮件地址。每次 Git 提交都会用到这些信息，并且会嵌入到你的提交中，不可更改。

# 全局配置（推荐，对所有仓库生效）
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

# 仅对当前仓库配置（覆盖全局）
git config user.name "Work Name"

# 设置默认编辑器
git config --global core.editor vim

# 查看所有配置
git config --list
配置优先级 系统级（/etc/gitconfig）→ 用户级（~/.gitconfig）→ 仓库级（.git/config），优先级从低到高，高优先级覆盖低优先级。
git init 在当前目录初始化一个新的 Git 仓库 ▼
🆕 使用场景：开始一个全新项目时

在一个新的或已有的目录中运行 git init，Git 会创建 .git 子目录，这是 Git 存储和操作几乎所有内容的地方。

git init
# Initialized empty Git repository in /your/path/.git/

# .git 目录初始结构
config        # 项目配置
description   # 仓库描述（GitWeb 使用）
HEAD          # 当前分支指针
hooks/        # 钩子脚本目录
info/         # 全局 exclude 规则
objects/      # 对象数据库（blob/tree/commit）
refs/         # 引用（branches/tags/remotes）
git clone 克隆一个已有的远程仓库 ▼
📥 使用场景：加入已有项目，或备份远程仓库

克隆会获取远程仓库的完整历史，包括所有 commit、分支和 tag，你将获得一个完整的本地副本。

# 克隆到同名目录
git clone https://github.com/user/repo.git

# 克隆到指定目录名
git clone https://github.com/user/repo.git my-project

# 通过 SSH 克隆
git clone git@github.com:user/repo.git
08 · 常用命令

基础工作流命令

官方文档：Recording Changes to the Repository
git status 查看工作目录和暂存区的状态 ▼
🔍 使用场景：随时了解当前工作状态，是最常用命令之一 
git status
On branch master
Your branch is up-to-date with 'origin/master'.
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
    new file:   README          ← 已暂存
    modified:   CONTRIBUTING.md ← 已暂存

Changes not staged for commit:
  modified:   CONTRIBUTING.md  ← 已修改但未暂存

Untracked files:
  LICENSE.txt                  ← 未追踪

# 简短输出
git status -s
M  README        ← 暂存区有修改
MM Rakefile      ← 暂存+工作区均有修改
A  lib/git.rb   ← 新加入暂存区
?? LICENSE.txt  ← 未追踪
git add 将文件变更添加到暂存区 ▼
➕ 使用场景：选择要包含在下次提交中的具体变更

官方文档指出，可以将 git add 理解为"精确地将这些内容添加到下次提交"，而非简单地"把这个文件加入项目"。

# 添加单个文件
git add README.md

# 添加目录（递归添加）
git add src/

# 添加所有变更（新增+修改+删除）
git add -A
# 或
git add .

# 交互式选择要暂存的内容（逐个 hunk）
git add -p
注意 git add 暂存的是执行时文件的快照。如果暂存后再次修改文件，必须再次运行 git add 才能包含最新内容。
git commit 将暂存区的快照永久记录到仓库 ▼
💾 使用场景：完成一个有意义的工作单元后永久记录 
# 打开编辑器写提交消息
git commit

# 内联提交消息（最常用）
git commit -m "fix: resolve login timeout issue"

# 自动暂存已追踪的修改文件并提交（跳过 git add）
git commit -a -m "chore: update dependencies"

# 修改最近一次提交（消息或内容）
git commit --amend

[master 463dc4f] fix: resolve login timeout issue
 2 files changed, 2 insertions(+)
 create mode 100644 README
好的提交消息格式 推荐使用 Conventional Commits 格式：type(scope): subject，如 feat: add user authentication，让历史记录清晰可读。
git diff 查看具体的变更内容（行级别） ▼
🔎 使用场景：在提交前检查确切修改了哪些内容 
# 查看工作区与暂存区的差异（尚未暂存的修改）
git diff

# 查看暂存区与最近提交的差异（将进入下次提交的内容）
git diff --staged
# 同义词
git diff --cached

# 比较两个提交之间的差异
git diff abc123 def456

# 比较两个分支
git diff main..feature-branch
git log 查看提交历史 ▼
📜 使用场景：查看项目历史、追溯变更来源 
# 基本历史
git log

# 每提交一行（简洁模式）
git log --oneline

# 图形化显示分支历史
git log --oneline --decorate --graph --all

# 查看特定文件的修改历史
git log --follow src/auth.js

# 搜索提交消息
git log --grep="login"

# 查看某人的提交
git log --author="Alice"

# 显示最近3次
git log -3

* c2b9e (HEAD -> master) Make other changes
| * 87ab2 (testing) Make a change
|/
* f30ab Add feature #32
* 34ac2 Fix bug #1328
git rm / git mv 从 Git 追踪中删除或移动文件 ▼
🗑️ 使用场景：删除或重命名受追踪的文件 
# 删除文件（同时从工作区和暂存区移除）
git rm PROJECTS.md

# 仅从暂存区移除，保留工作区文件（忘记加入 .gitignore 时）
git rm --cached debug.log

# 重命名文件（等价于 mv + git rm + git add）
git mv old_name.js new_name.js
09 · 常用命令

分支操作命令

官方文档：Basic Branching and Merging — git-scm.com
git branch 列出、创建或删除分支 ▼
🌿 使用场景：管理功能分支、维护分支列表 
# 列出所有本地分支
git branch

# 列出所有分支（含远程）
git branch -a

# 创建新分支（不切换）
git branch feature/login

# 删除已合并的分支
git branch -d feature/login

# 强制删除（未合并的分支）
git branch -D feature/login

# 重命名当前分支
git branch -m new-name
git checkout / git switch 切换分支或恢复文件 ▼
🔀 使用场景：在不同功能分支之间切换工作

从 Git 2.23 版本起，推荐使用 git switch 替代 git checkout 来切换分支，语义更清晰。

# 切换到已有分支
git checkout main
# 新语法（Git 2.23+）
git switch main

# 创建并立即切换（最常用的工作流）
git checkout -b feature/auth
# 新语法
git switch -c feature/auth

# 切换回上一个分支
git switch -
官方提示 切换分支时，Git 会将工作目录中的文件还原为该分支最后一次提交时的状态。如果 Git 无法干净地完成此操作，它将拒绝切换。
git merge 将另一个分支的历史合并到当前分支 ▼
🔗 使用场景：功能开发完成，合并回主干 
# 将 feature 分支合并到当前分支（如 main）
git switch main
git merge feature/auth

# 禁止 fast-forward，强制生成合并提交
git merge --no-ff feature/auth

# 发生冲突时，解决后标记
# 1. 编辑冲突文件
git add conflicted-file.js
git commit  # 完成合并提交
git rebase 变基：将一系列提交移植到另一个基点上 ▼
🔄 使用场景：保持线性提交历史，整理 feature 分支 
# 将当前分支变基到 main
git rebase main

# 交互式变基（整理最近3次提交）
git rebase -i HEAD~3
# 在编辑器中可进行 squash / reword / drop 等操作
黄金规则 不要对已推送到远程的公共分支执行 rebase！Rebase 会重写提交历史，会给他人造成混乱。仅在私有 feature 分支上使用。
git stash 临时保存工作区的未提交变更 ▼
📦 使用场景：需要紧急切换分支处理 bug，但当前工作未完成 
# 保存当前工作现场
git stash
# 等同于
git stash push

# 查看 stash 列表
git stash list
stash@{0}: WIP on feature: abc1234 partial work
stash@{1}: WIP on main: def5678 old stash

# 恢复最近的 stash（保留记录）
git stash apply

# 恢复并删除记录
git stash pop

# 删除特定 stash
git stash drop stash@{1}

远程协作命令

官方文档：Working with Remotes — git-scm.com

图7：Git 分布式协作模型 — 所有开发者与远程仓库双向同步

git remote 管理远程仓库连接 ▼
🌐 使用场景：配置和查看远程仓库地址 
# 查看远程仓库
git remote -v
origin  https://github.com/user/repo.git (fetch)
origin  https://github.com/user/repo.git (push)

# 添加远程仓库
git remote add origin https://github.com/user/repo.git

# 修改远程 URL
git remote set-url origin git@github.com:user/repo.git

# 删除远程
git remote remove old-remote
git fetch 从远程获取最新数据但不合并 ▼
📡 使用场景：想查看远程有何变化，但暂不合并到本地 
# 获取 origin 的所有分支更新
git fetch origin

# 获取所有远程的更新
git fetch --all

# 查看远程分支内容（不影响本地）
git log origin/main
fetch vs pull git fetch 仅下载数据到本地仓库，不会自动合并或修改你当前的工作。git pull = git fetch + git merge。
git pull 拉取远程变更并合并到本地 ▼
⬇️ 使用场景：同步团队最新代码到本地工作分支 
# 拉取并合并当前追踪分支
git pull

# 指定远程和分支
git pull origin main

# 使用 rebase 方式合并（保持线性历史）
git pull --rebase
git push 将本地提交推送到远程仓库 ▼
⬆️ 使用场景：分享你的工作，或备份到远程仓库 
# 推送到 origin 的同名分支
git push

# 首次推送并建立追踪关系
git push -u origin feature/auth

# 推送标签
git push origin v1.0.0

# 删除远程分支
git push origin --delete feature/old
git tag 为特定提交打标签（通常用于版本发布） ▼
🏷️ 使用场景：标记版本发布节点，如 v1.0.0 
# 创建轻量标签（仅指针）
git tag v1.0.0

# 创建附注标签（推荐，包含完整元数据）
git tag -a v1.0.0 -m "Release version 1.0.0"

# 查看所有标签
git tag

# 推送标签到远程
git push origin --tags
11 · 常用命令

撤销与回退命令

官方文档：Undoing Things — git-scm.com
Git 的安全性保证 官方文档指出："当你做完操作后，几乎所有的 Git 操作都只是向数据库中添加数据。很难让 Git 执行任何不可撤销的操作，或者让它删除任何数据。一旦你提交了快照，就很难再丢失。"
命令作用范围是否安全典型场景
git restore <file> 工作区 ⚠️ 丢弃未暂存修改 放弃某文件的改动
git restore --staged <file> 暂存区 ✅ 安全 取消暂存（不影响工作区）
git commit --amend 最近提交 ⚠️ 重写历史 修改最近提交的消息或内容
git revert <commit> 历史提交 ✅ 安全（产生新提交） 撤销已推送的提交
git reset --soft HEAD~1 HEAD 指针 ✅ 安全 撤销提交，保留暂存区变更
git reset --mixed HEAD~1 HEAD + 暂存区 ✅ 安全 撤销提交，变更退回工作区
git reset --hard HEAD~1 HEAD + 暂存区 + 工作区 ❌ 危险，丢弃变更 彻底放弃最近提交及所有修改
git revert 通过新提交来安全撤销历史变更 ▼
↩️ 使用场景：撤销已推送到远程的错误提交 
# 撤销指定提交（生成一个新的"撤销提交"）
git revert abc1234

# 撤销最近一次提交
git revert HEAD

# 仅生成撤销变更，不自动提交
git revert --no-commit HEAD
git reset 移动 HEAD 指针，可选地重置暂存区和工作区 ▼
⏪ 使用场景：在私有分支上整理提交历史 
# 软重置：撤销提交，保留暂存区（可重新提交）
git reset --soft HEAD~1

# 混合重置（默认）：撤销提交和暂存，保留工作区修改
git reset HEAD~1

# 硬重置：彻底丢弃一切，慎用！
git reset --hard HEAD~1
警告 不要对已推送到共享远程分支的提交执行 git reset —— 这会重写历史，导致团队其他人的仓库出现分歧。对公共历史只使用 git revert。

13 · 第一性原理总结

从第一性原理理解 Git

第一性原理方法 抛开所有关于版本控制、命令行工具的先入之见，从最基本的问题出发：软件协作开发本质上在解决什么问题？

⚛️ 本质问题分解

01

核心问题：如何安全、高效地管理随时间变化的文件集合？

软件开发的本质是持续修改文件。最原始的方法是手动复制备份（v1, v2, v2-final）——这是所有 VCS 要解决的根本问题。Git 的回答是：内容寻址的不可变对象存储。每次变更不修改原来的对象，而是创建新对象，用 SHA-1 哈希寻址。

02

存储模型：快照 = 效率 + 完整性

Git 不存差异（patch），而是存快照（snapshot）。未变化的文件复用同一 blob 对象，变化的文件创建新对象。这使得任意版本的访问代价恒定，无需逐级回放差异。SHA-1 校验确保数据完整性——任何字节的损坏都会被立即检测到。

03

分布式 = 去除单点瓶颈

集中式 VCS（SVN）的根本缺陷：所有操作依赖中央服务器，网络延迟、服务器宕机都会中断工作。Git 的回答：每个节点都是完整的仓库。"远程仓库"本质上只是另一个 Git 仓库，没有特殊地位。这使离线工作、并行开发、去中心化协作成为自然而然的事。

04

分支 = 指针，而非目录拷贝

旧式 VCS 的分支是物理复制，代价昂贵。Git 的洞察：分支只需是指向某个 commit 的指针（41字节文件）。Commit 构成一个有向无环图（DAG），分支是图中某节点的别名。合并是图的连接操作，变基是图的拓扑重构。极低代价使"频繁分支、小步迭代"成为可行的工程实践。

05

暂存区 = 精确控制提交边界

为什么需要三个区域？因为"修改的文件"和"想提交的内容"不总是相同的。暂存区（Index）是两者之间的缓冲层，让开发者可以精确选择每次提交的原子内容，使提交历史有意义、可读。这是 Git 相比简单备份工具的关键工程决策。

💡 一句话总结

Git 的本质

Git 是一个以内容寻址的不可变对象图为核心数据结构，以 SHA-1 哈希保证完整性，以轻量指针实现分支，以分布式副本消除单点瓶颈，并通过暂存区赋予开发者精确表达提交意图能力的版本控制系统。

理解了这一点，Git 所有看似复杂的命令都变成了对这个数据结构的自然操作：add 是写入暂存区，commit 是将暂存区快照持久化为对象并移动指针，branch 是创建新指针，merge 是连接两个图节点，rebase 是重构图的拓扑结构，push/pull 是在不同仓库副本之间同步对象图。
