🖥️SSH 简介

SSH（Secure Shell） 是用于在两台不可信主机之间建立加密安全通信的程序。OpenSSH 是其最广泛使用的实现版本。

📋

官方定义（来源：man7.org ssh(1)）：
"ssh (SSH client) is a program for logging into a remote machine and for executing commands on a remote machine. It is intended to provide secure encrypted communications between two untrusted hosts over an insecure network."

SSH 工具族

🔌 ssh

OpenSSH 远程登录客户端，核心程序

🔑 ssh-keygen

生成、管理和转换 SSH 认证密钥

📋 ssh-copy-id

将公钥安全地复制到远程服务器

🔐 ssh-agent

保存私钥的认证代理，避免重复输入密码

➕ ssh-add

向 ssh-agent 添加或移除私钥

📁 scp / sftp

基于 SSH 的安全文件复制与传输

⚛ 第一性原理

SSH 的本质是：在不安全的网络上，用密码学手段解决「我怎么知道对方是我想连的那台机器」和「传输的数据不被窃听」两个核心问题。其余所有功能——端口转发、文件传输、代理转发——都是建立在这条加密隧道之上的扩展应用。

🏗️工作原理

SSH 连接建立经历以下阶段，每个阶段都有对应的安全机制：

认证方式对比

认证方式安全性便利性适用场景
公钥认证 推荐 ⭐⭐⭐⭐⭐ 配置后极方便 日常登录、自动化脚本
密码认证 ⭐⭐⭐ 开箱即用 临时访问、初始配置
GSSAPI / Kerberos ⭐⭐⭐⭐ 企业环境中透明 企业 AD 域环境
基于主机认证 ⭐⭐ 较方便 信任内网主机（不推荐）
⚛ 第一性原理

认证的本质是「证明你拥有某个秘密而不泄露它」。公钥认证用数字签名做到这一点：用私钥签名，服务端用公钥验签，私钥永不离开本地。密码认证则是在加密隧道内传输密码，其安全性依赖于隧道本身——这也是为什么必须先建立加密连接，再做认证。

🔌连接与登录

📖 来源：man7.org — ssh(1) OpenSSH 9.9

基本语法

# 语法格式（来自官方 man page）
ssh [-46AaCfGgKkMNnqsTtVvXxYy] [-p port] [user@]hostname [command]
ssh ssh://[user@]hostname[:port]

常用连接场景

# 1. 最基本的连接（使用默认端口 22）
ssh user@hostname

# 2. 指定端口（-p port）
ssh -p 2222 user@hostname

# 3. 指定私钥文件（-i identity_file）
ssh -i ~/.ssh/my_key user@hostname

# 4. 登录后直接执行远程命令
ssh user@hostname "df -h"

# 5. 强制分配伪终端（-t），用于执行需要 tty 的命令
ssh -t user@hostname "sudo vim /etc/ssh/sshd_config"

# 6. 详细调试输出（-v / -vv / -vvv 级别递增）
ssh -vv user@hostname

# 7. 仅使用 IPv4（-4）或 IPv6（-6）
ssh -4 user@hostname

# 8. 启用数据压缩（-C），适合慢速网络
ssh -C user@hostname

# 9. 静默模式，抑制警告信息（-q）
ssh -q user@hostname

关键参数速查表

参数全称/说明典型使用场景
-p port指定远程端口（默认 22）非标准端口服务器
-i file指定私钥文件路径多密钥环境
-l user指定登录用户名等同于 user@host
-t强制分配伪终端（PTY）sudo、vim 等需要终端的命令
-T禁止分配伪终端非交互式脚本调用
-v/-vv/-vvv详细/调试输出（最大 3 级）连接问题排查
-C启用 gzip 压缩低带宽网络、X11 转发
-q安静模式，抑制警告脚本中使用
-A开启 agent 转发跳板机场景（谨慎使用）
-X开启 X11 转发远程图形界面
-Y开启可信 X11 转发受信任环境的 X11
-f认证后进入后台运行隧道维持、X11 程序启动

-G打印配置并退出调试 ssh_config 配置

-F file指定配置文件使用非默认配置
⚠️

当第一次连接到一台服务器时，SSH 会显示服务器主机公钥的指纹（fingerprint），必须人工确认以防范中间人攻击。确认后，服务器密钥将记录到 ~/.ssh/known_hosts。

⚛ 第一性原理

连接命令的所有参数，本质上都在回答一个问题：「在这次连接中，安全性（加密强度、认证方式）和效率（压缩、复用）如何权衡？」 — 理解了这个权衡，你就能在任何场景选出正确的参数组合。

🔑密钥管理

📖 来源：man7.org — ssh-keygen(1) | ssh-copy-id(1) | ssh-agent(1)

密钥类型

类型密钥长度安全性推荐度
ed25519固定 256 bit最高（现代椭圆曲线）强烈推荐
ecdsa256/384/521 bit高推荐
rsa≥2048 bit（推荐4096）中高（兼容性好）兼容场景
dsa1024 bit（固定）低（已弃用）不推荐

ssh-keygen — 生成与管理密钥

# 生成 Ed25519 密钥（推荐，现代算法）
ssh-keygen -t ed25519 -C "your_email@example.com"

# 生成 RSA 4096 位密钥（兼容性好）
ssh-keygen -t rsa -b 4096 -C "comment"

# 指定密钥文件保存路径（-f filename）
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519_work

# 查看公钥指纹（-l -f）
ssh-keygen -l -f ~/.ssh/id_ed25519.pub

# 查看所有已知主机的指纹（含随机艺术图）
ssh-keygen -lv -f ~/.ssh/known_hosts

# 修改私钥的 passphrase（-p）
ssh-keygen -p -f ~/.ssh/id_ed25519

# 从 known_hosts 中删除特定主机记录（-R）
ssh-keygen -R hostname

# 生成主机密钥指纹（用于 SSHFP DNS 记录）
ssh-keygen -r host.example.com.

ssh-copy-id — 部署公钥到远程服务器

# 将默认公钥复制到远程服务器（最常用）
ssh-copy-id user@hostname

# 指定公钥文件（-i identity_file）
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@hostname

# 指定目标服务器端口（-p port）
ssh-copy-id -p 2222 -i ~/.ssh/id_ed25519.pub user@hostname

# 演习模式，仅打印将要安装的密钥，不实际安装（-n）
ssh-copy-id -n user@hostname
ℹ️

ssh-copy-id 原理：该工具通过 ssh 登录远程机器，将公钥追加到远程用户的 ~/.ssh/authorized_keys 文件（如不存在则创建目录和文件）。—— 来源：man7.org ssh-copy-id(1)

ssh-agent + ssh-add — 密钥代理

# 启动 ssh-agent（在当前 shell 中）
eval "$(ssh-agent -s)"

# 将私钥添加到 agent（只需输入一次 passphrase）
ssh-add ~/.ssh/id_ed25519

# 列出 agent 中已加载的所有密钥
ssh-add -l

# 列出已加载密钥的公钥内容
ssh-add -L

# 从 agent 中移除指定密钥
ssh-add -d ~/.ssh/id_ed25519

# 移除 agent 中所有密钥
ssh-add -D

# 添加密钥时设置有效期（秒），3600 = 1小时后自动删除
ssh-add -t 3600 ~/.ssh/id_ed25519
⚛ 第一性原理

密钥管理的本质是「私钥的安全存储与安全使用」。生成密钥 = 创建数学秘密；passphrase = 加密存储这个秘密；ssh-agent = 在内存中持有解密后的秘密，代替用户完成签名，私钥内容本身永远不跨网络传输。

📁文件传输

📖 来源：man7.org — scp(1) | sftp(1)

scp — 安全复制

# 从本地复制到远程
scp /local/file.txt user@host:/remote/path/

# 从远程复制到本地
scp user@host:/remote/file.txt /local/path/

# 递归复制目录（-r）
scp -r /local/dir/ user@host:/remote/dir/

# 指定端口（-P，注意大写）
scp -P 2222 file.txt user@host:/tmp/

# 显示进度（-v）
scp -v file.txt user@host:/tmp/

# 启用压缩（-C）
scp -C largefile.tar.gz user@host:/tmp/

# 两台远程主机之间复制
scp user1@host1:/file.txt user2@host2:/path/

sftp — 交互式 SSH 文件传输

# 连接到远程 SFTP 服务器
sftp user@hostname

# 连接后可用的交互命令：
sftp> ls               # 列出远程目录
sftp> lls              # 列出本地目录
sftp> cd /remote/path  # 切换远程目录
sftp> lcd /local/path  # 切换本地目录
sftp> get file.txt     # 下载文件
sftp> put file.txt     # 上传文件
sftp> mget *.log       # 批量下载
sftp> mput *.txt       # 批量上传
sftp> mkdir newdir     # 远程创建目录
sftp> rm file.txt      # 删除远程文件
sftp> pwd              # 显示远程当前目录
sftp> bye              # 退出
⚛ 第一性原理

scp/sftp 的本质是：把文件读写操作封装成 SSH 协议消息，在已有的加密隧道内传输。scp 适合一次性任务（命令即传完），sftp 适合会话式操作（需要浏览、多次传输）。选择哪个，取决于你的操作是「一条路走到底」还是「需要交互探索」。

🔧端口转发（隧道）

📖 来源：man7.org — ssh(1) TCP FORWARDING 章节

SSH 支持三种端口转发模式，可将任意 TCP 流量通过加密隧道传输：

① 本地端口转发（-L）

本地指定 TCP 端口上的连接，通过加密隧道转发到远端主机的指定端口。来源：man7.org ssh(1)

# 语法：-L [bind_address:]port:host:hostport

# 场景：访问远程内网的 MySQL 数据库
# 本地 3307 → SSH 服务端 → db-server:3306
ssh -L 3307:db-server:3306 user@ssh-host

# 仅转发端口，不打开 shell（-N -f 后台运行）
ssh -N -f -L 8080:localhost:80 user@ssh-host

# 多个本地转发同时使用
ssh -L 3307:localhost:3306 \
    -L 6380:localhost:6379 \
    -L 8080:localhost:80 \
    user@ssh-host

# 连接后通过本地访问：
# mysql -h 127.0.0.1 -P 3307 -u dbuser -p

② 远程端口转发（-R）反向隧道

远端主机监听指定端口，将连接转发回本地。来源：man7.org ssh(1)

# 语法：-R [bind_address:]port:host:hostport

# 场景：将本地开发服务器 3000 暴露到公网 8080
ssh -R 8080:localhost:3000 user@public-server
# 此后访问 public-server:8080 即访问你的 localhost:3000

# 绑定所有接口（需服务端 GatewayPorts yes）
ssh -R 0.0.0.0:8080:localhost:3000 user@public-server

# 动态分配端口（port 填 0），运行时报告实际端口
ssh -R 0:localhost:3000 user@public-server

③ 动态端口转发（-D）SOCKS 代理

本地创建 SOCKS4/SOCKS5 代理，所有请求动态转发。来源：man7.org ssh(1)

# 语法：-D [bind_address:]port

# 在本地 1080 端口创建 SOCKS5 代理
ssh -D 1080 -N -f user@ssh-host

# 使用 curl 通过 SOCKS5 代理访问
curl --socks5 localhost:1080 https://internal.example.com
⚛ 第一性原理

端口转发本质是「把 TCP 连接的一端搬到 SSH 隧道的另一头」。本地转发（-L）：把远端服务拉到本地来；远程转发（-R）：把本地服务推到远端去；动态转发（-D）：让 SSH 充当出口路由器，目的地在连接时才决定。三者方向相反，但底层机制相同——都是在 SSH 加密信道上复用 TCP 流。

🪂跳板机（ProxyJump）

📖 来源：man7.org — ssh(1) -J 选项

官方描述（-J 选项）："Connect to the target host by first making an ssh connection to the jump host described by destination and then establishing a TCP forwarding to the ultimate destination from there."

# -J 选项：通过跳板机连接目标（-J destination）
ssh -J user@bastion.example.com user@internal-server

# 多级跳板（逗号分隔）
ssh -J hop1,hop2,hop3 user@final-target

# 通过跳板机做端口转发
ssh -J user@bastion \
    -L 5433:10.0.0.5:5432 \
    user@internal-server

# 在 ~/.ssh/config 中配置 ProxyJump（推荐）
# Host internal-server
#   HostName 10.0.0.20
#   User deploy
#   ProxyJump bastion.example.com

# 配置后直接使用：
ssh internal-server
ℹ️

官方说明："This is a shortcut to specify a ProxyJump configuration directive. Note that configuration directives supplied on the command-line generally apply to the destination host and not any specified jump hosts. Use ~/.ssh/config to specify configuration for jump hosts." — man7.org

⚛ 第一性原理

跳板机模式的本质是网络可达性的组合传递：你能到达 A，A 能到达 B，所以你通过 A 可以到达 B。-J 选项让 SSH 在 A 上建立透明 TCP 转发，使你的本地 SSH 客户端直接与 B 的 sshd 通信。跳板机看到的只是两段 TCP 连接，无法解密端到端的 SSH 内容——这比传统的二次 SSH 登录更安全且更高效。

⚡连接复用（ControlMaster）

📖 来源：man7.org — ssh(1) -M / -S / -O 选项

SSH 连接复用允许多个会话共享一条已建立的 TCP 连接，大幅减少重复认证开销。

# -M：进入 master 模式（ControlMaster）
ssh -M -S ~/.ssh/ctrl-%h-%p-%r user@host

# 后续连接复用上面的连接（自动认证，无需密码）
ssh -S ~/.ssh/ctrl-%h-%p-%r user@host

# -O ctl_cmd：控制复用 master 进程
ssh -O check -S ~/.ssh/ctrl-%h user@host   # 检查是否运行
ssh -O exit  -S ~/.ssh/ctrl-%h user@host   # 退出 master
ssh -O stop  -S ~/.ssh/ctrl-%h user@host   # 停止接受新连接

在 ~/.ssh/config 中配置（推荐）

Host myserver
    HostName       server.example.com
    User           alice
    ControlMaster  auto
    ControlPath    ~/.ssh/ctrl-%h-%p-%r
    ControlPersist 600     # 最后一个会话断开后保持 600 秒
⚛ 第一性原理

TCP 连接建立和 SSH 握手认证是有代价的（时延 + CPU）。连接复用的本质是摊平这个固定成本：第一次连接付出全部代价，后续连接复用已有信道——就像高速公路上共享一条车道而非每次重建道路。对于频繁连接同一主机（如 CI/CD 脚本）效果显著。

⌨️SSH 转义序列（快捷键）

📖 来源：man7.org — ssh(1) ESCAPE CHARACTERS 章节

⚠️

重要规则（官方原文）："Note that escapes are only recognized immediately after newline."
转义序列必须在新行开头输入。操作步骤：先按 Enter，再输入 ~，再输入命令字符。

官方完整的转义序列列表（来源：ssh -? 或 ~?）：

~.
终止连接（含所有多路复用会话）。网络冻结时最有用，无需等待超时。
~^Z
挂起 SSH（后台暂停）。可用 fg 恢复，相当于 Ctrl+Z 但作用于 SSH 本身。
~C
打开命令行（ssh> 提示符）。可在不断开连接的情况下动态添加/取消端口转发。
注：新版 OpenSSH 需 -o EnableEscapeCommandline=yes
~B
向远端发送 BREAK 信号。串口控制台服务器场景下触发设备中断。
~R
请求重新密钥协商（Rekey）。对端支持时刷新会话密钥，提升长连接安全性。
~V
降低日志详细度（LogLevel）。减少写入 stderr 的错误信息量。
~v
增加日志详细度（LogLevel）。实时增加 SSH 调试输出，用于连接问题诊断。
~#
列出所有已转发的连接。查看当前会话中活跃的端口转发清单。
~&
后台化 SSH（等待转发连接关闭后）。将 SSH 进程放入后台，继续维持转发。
~?
显示转义序列帮助列表。忘记命令时随时查看。
~~
发送转义字符本身。发送字面量 ~ 字符到远端（连按两次）。

~C 命令行动态端口转发

# 在活跃的 SSH 会话中按 Enter，然后输入 ~C
ssh>

# 添加本地端口转发（-L）
ssh> -L 8080:192.168.10.5:80
Forwarding port.

# 添加远程端口转发（-R）
ssh> -R 9090:localhost:3000

# 添加动态转发（-D）
ssh> -D 1080

# 取消本地端口转发（-KL）
ssh> -KL8080

# 查看可用命令
ssh> help

多级跳转时的转义处理

# 连接两层 SSH 后：
# ~.  → 断开第二层（最内层）连接
# ~~. → 断开第一层连接（发送 ~. 到外层 SSH）
# 每多一层，多加一个 ~ 前缀
⚛ 第一性原理

转义序列是 SSH 客户端在 PTY（伪终端）层面拦截的带外信号——在数据流之外插入一个控制通道。这解决了一个根本问题：当 SSH 连接卡死时，没有任何「在隧道内」的方式可以修复它，因为隧道本身不通了。转义字符让客户端在本地直接终止连接，完全绕过远端，这也是为什么 ~. 即使在网络完全冻结时也能生效。

⚙️配置文件（~/.ssh/config）

📖 来源：man7.org — ssh_config(5)

用户级配置文件 ~/.ssh/config 是最强大的 SSH 效率工具，可为不同主机设置独立配置。

# 权限必须正确，否则 SSH 会忽略此文件
chmod 600 ~/.ssh/config

# ─── 示例配置文件 ~/.ssh/config ───

# 全局默认配置
Host *
    ServerAliveInterval  60      # 每 60 秒发送心跳
    ServerAliveCountMax  3       # 心跳失败 3 次则断开
    AddKeysToAgent       yes     # 自动将密钥加入 agent
    IdentityFile         ~/.ssh/id_ed25519

# 为特定主机定义别名
Host prod
    HostName    192.168.1.100
    User        deploy
    Port        2222
    IdentityFile ~/.ssh/id_rsa_prod

# 跳板机配置
Host internal
    HostName    10.0.0.50
    User        admin
    ProxyJump   bastion.example.com

# 数据库隧道（连接后自动建立端口转发）
Host db-tunnel
    HostName       db.example.com
    User           tunnel
    LocalForward   3307 localhost:3306
    LocalForward   6380 localhost:6379

# 连接复用配置
Host fast-server
    HostName       fast.example.com
    ControlMaster  auto
    ControlPath    ~/.ssh/ctrl-%h-%p-%r
    ControlPersist 600

# 使用后直接：ssh prod / ssh internal / ssh db-tunnel

⚛ 第一性原理

配置文件的本质是「把重复的命令行参数固化为声明式配置」。每次输入 ssh -p 2222 -i ~/.ssh/id_rsa deploy@192.168.1.100，都是在重复「告诉 ssh 应该怎么做」。配置文件将这些决策只做一次，永久复用。这是 DRY（Don't Repeat Yourself）原则在运维领域的直接体现。

🛡️安全加固实践

📖 来源：man7.org — sshd(8) | ssh(1)

主机密钥验证流程

官方说明："When connecting to a server for the first time, a fingerprint of the server's public key is presented to the user (unless the option StrictHostKeyChecking has been disabled)." — man7.org ssh(1)

# 查看服务器主机密钥指纹（在服务端执行）
ssh-keygen -l -f /etc/ssh/ssh_host_ed25519_key

# 用随机艺术图方式显示（更直观）
ssh-keygen -lv -f /etc/ssh/ssh_host_ed25519_key

# 生成 SSHFP DNS 记录（用于 DNS 验证主机密钥）
ssh-keygen -r host.example.com.

# 通过 DNS 验证连接（需 DNS 配置 SSHFP 记录）
ssh -o "VerifyHostKeyDNS ask" host.example.com

# 主机密钥变更时（如重装系统）删除旧记录
ssh-keygen -R hostname

推荐的安全配置

安全措施说明操作
禁用密码认证 仅允许公钥登录 PasswordAuthentication no（sshd_config）
禁用 root 登录 避免直接 root SSH PermitRootLogin no
修改默认端口 降低扫描噪音 Port 2222
使用 Ed25519 密钥 现代安全算法 ssh-keygen -t ed25519
开启 StrictHostKeyChecking 防范 MITM 攻击 默认 ask，不要改为 no
谨慎使用 ForwardAgent 转发的 agent 可被滥用 官方建议：使用 ProxyJump 替代
私钥设置 passphrase 防止密钥文件泄露 生成时设置，用 ssh-agent 管理
known_hosts 哈希 保护主机名信息 HashKnownHosts yes
🚫

官方警告（ForwardAgent）："Agent forwarding should be enabled with caution. Users with the ability to bypass file permissions on the remote host (for the agent's Unix-domain socket) can access the local agent through the forwarded connection." — man7.org ssh(1)

⚛ 第一性原理

SSH 安全的核心命题是：「最小权限 + 最短路径 + 最强密码学」。最小权限 = 禁用 root/密码登录；最短路径 = 用 ProxyJump 而非多级代理；最强密码学 = Ed25519、禁用弱算法。每一条安全建议都能归结到这三个原则上。

📂关键文件与目录

📖 来源：man7.org — ssh(1) FILES 章节

文件/目录权限说明
~/.ssh/700SSH 配置目录，仅用户可读写执行
~/.ssh/config600用户级客户端配置，不可对他人可写
~/.ssh/id_*600私钥文件，SSH 会拒绝权限过宽的私钥
~/.ssh/id_*.pub644公钥文件，可公开
~/.ssh/authorized_keys600服务端允许登录的公钥列表
~/.ssh/known_hosts644已验证的服务器公钥记录
/etc/ssh/sshd_config600sshd 全局配置文件
🔒

官方说明（私钥权限）："ssh will simply ignore a private key file if it is accessible by others." — man7.org ssh(1)
私钥文件权限必须为 600（-rw-------），否则 SSH 客户端会直接忽略它。

⚛ 第一性原理

SSH 的文件权限机制体现了「最小可信面」原则：系统通过检查文件权限来判断「这些秘密是否被妥善保管」。私钥对他人可读 = 秘密已经不是秘密 = SSH 拒绝使用它。这是在操作系统层面强制执行安全策略，而不是依赖用户自律。
