第 1 章

ELF 简介与历史

原文：Linux man-pages elf(5) 原文：Wikipedia ELF

ELF（Executable and Linkable Format，可执行与可链接格式）是 Linux/Unix 系统上可执行文件、目标代码、共享库及核心转储（core dump）的标准二进制文件格式。

ELF 最初由 Unix System Laboratories 在 System V Release 4（SVR4）的应用二进制接口（ABI）规范中发布，后经 Tool Interface Standard（TIS）标准化（1995 年，Version 1.2）。1999 年，86open 项目将其确定为 x86 Unix 及类 Unix 系统的标准二进制格式。

设计哲学

ELF 的核心设计目标是：灵活性（支持多种 CPU 架构、字节序、地址宽度）、可扩展性（允许平台特定扩展）和跨平台性（不与任何特定指令集绑定）。

ELF 的适用场景

文件扩展名典型用途
（无扩展名）可执行程序（如 /bin/ls）
.o可重定位目标文件（编译器输出）
.so共享库（动态链接库）
.koLinux 内核模块
.elf嵌入式系统固件
core程序崩溃时的内存转储

魔数标识

每个 ELF 文件的前 4 字节为固定魔数，用于快速识别文件类型：

字节[0]: 0x7F
字节[1]: 0x45  'E'
字节[2]: 0x4C  'L'
字节[3]: 0x46  'F'
图1：ELF 头部字段总览

📌 本章小结

ELF 是 Linux/Unix 生态系统的基础格式规范，诞生于 Unix System V，被几乎所有现代 Unix-like 系统采用。其设计核心是将一个二进制文件同时服务于两类受众：链接器/调试器（以 Section 为单位）和操作系统加载器（以 Segment 为单位），这种双视图架构是 ELF 最根本的设计思想。

第 2 章

ELF 文件类型

原文：Linux man-pages elf(5) — e_type 字段

ELF 头部的 e_type 字段标识文件的对象类型。根据 Linux man-pages（elf.5），定义了以下类型：

常量名数值中文名称说明典型工具/场景
ET_NONE0x00未知类型类型未定义—
ET_REL0x01可重定位文件包含代码和数据，需要链接后才能执行。所有地址均为相对偏移，链接器负责填写最终地址。编译器输出的 .o 文件
ET_EXEC0x02可执行文件包含可直接执行的程序，具有固定的入口地址（e_entry）。/bin/ls、a.out
ET_DYN0x03共享对象文件动态链接库（.so）或位置无关可执行文件（PIE）。地址在加载时由动态链接器决定。libc.so.6、PIE 可执行
ET_CORE0x04核心转储文件程序异常终止时由内核生成，包含进程的内存快照和寄存器状态，用于事后调试。core 文件，gdb
图2：ELF 文件类型及其来源关系
关于 PIE

现代 Linux 系统下，为了支持地址空间随机化（ASLR），普通可执行文件也常被编译为位置无关可执行文件（PIE），其 e_type 为 ET_DYN 而非 ET_EXEC。这是安全增强的重要手段。

📌 本章小结

ELF 的四种主要文件类型对应程序生命周期的不同阶段：ET_REL（编译期产物）→ ET_EXEC/ET_DYN（链接期产物，运行时载入）→ ET_CORE（运行时崩溃记录）。第一性原理：文件格式的类型划分，本质上反映了程序从"源码"到"运行"再到"事后分析"这一完整生命周期的不同状态。

第 3 章

ELF 头部（ELF Header）

原文：Linux man-pages elf(5) — ELF header (Ehdr)

ELF 头部始终位于文件偏移 0 处，是整个文件的"目录入口"。32 位系统下占 52 字节，64 位系统下占 64 字节。

typedef struct {
    unsigned char e_ident[EI_NIDENT];  /* 16字节：魔数+分类信息 */
    uint16_t      e_type;              /* 文件类型 */
    uint16_t      e_machine;           /* 目标架构 */
    uint32_t      e_version;           /* ELF 版本（恒为1） */
    ElfN_Addr     e_entry;             /* 程序入口虚拟地址 */
    ElfN_Off      e_phoff;             /* 程序头表文件偏移 */
    ElfN_Off      e_shoff;             /* 节头表文件偏移 */
    uint32_t      e_flags;             /* 处理器特定标志 */
    uint16_t      e_ehsize;            /* ELF 头本身大小 */
    uint16_t      e_phentsize;         /* 程序头表每项大小 */
    uint16_t      e_phnum;             /* 程序头表项数 */
    uint16_t      e_shentsize;         /* 节头表每项大小 */
    uint16_t      e_shnum;             /* 节头表项数 */
    uint16_t      e_shstrndx;          /* 节名字符串表的索引 */
} ElfN_Ehdr;

e_ident 数组（16 字节）

这 16 个字节独立于处理器和文件内容，用于基本识别。各字节含义（来源：Linux man-pages elf.5）：

宏名偏移含义取值说明
EI_MAG0–30–3魔数0x7F 'E' 'L' 'F'，固定不变
EI_CLASS4文件类ELFCLASS32(1) = 32位；ELFCLASS64(2) = 64位
EI_DATA5数据编码ELFDATA2LSB(1) = 小端序；ELFDATA2MSB(2) = 大端序
EI_VERSION6ELF 版本必须为 EV_CURRENT（值为 1）
EI_OSABI7目标 OS/ABIELFOSABI_NONE(0)=System V；ELFOSABI_LINUX(3)=Linux；等
EI_ABIVERSION8ABI 版本依赖 EI_OSABI，通常为 0
EI_PAD9–15填充字节保留，填 0，读取时忽略

关键字段说明

字段作用
e_entry程序开始执行的虚拟地址（即 _start 函数地址）。若文件无入口，此值为 0。
e_phoff程序头表（Segment 描述）在文件中的字节偏移量。
e_shoff节头表（Section 描述）在文件中的字节偏移量。
e_phnum程序头表的表项数量。若 ≥ PN_XNUM(0xffff)，实际数量存入节头表首项的 sh_info。
e_shnum节头表的表项数量。若 ≥ SHN_LORESERVE(0xff00)，实际数量存入节头表首项的 sh_size。
e_shstrndx节名字符串表（.shstrtab）在节头表中的索引，用于查找各节的名称。
图3：ELF 文件整体布局 — 头部作为导航中枢

📌 本章小结

ELF 头部是整个文件的"目录页"，记录了文件的基本属性（架构、位宽、字节序）和两张关键索引表的位置（程序头表 e_phoff、节头表 e_shoff）。第一性原理：文件格式的核心问题是"如何在不知道内容的情况下定位内容"，ELF 头用固定位置+偏移量的方式优雅地解决了这一问题。

第 4 章

两种视图：Section vs Segment

参考：ELF File Format: Sections vs Segments — linuxvox.com

ELF 最重要的设计是同一份文件数据，提供两种不同的解读视角：

维度节视图（Section View）段视图（Segment View）
目的链接、调试、分析运行时加载与执行
描述结构节头表（Section Header Table）程序头表（Program Header Table）
服务对象链接器 ld、调试器 gdb、readelfOS 内核加载器、动态链接器 ld.so
内容粒度细粒度逻辑分组（代码、数据、符号…）粗粒度内存映射（一组 Section 合并为一个 Segment）
权限控制无直接权限（通过 sh_flags 提示）显式 R/W/X 权限（p_flags）
内存映射不直接映射到内存直接 mmap 到进程虚拟地址空间
典型示例.text、.data、.symtabPT_LOAD（R-X）、PT_LOAD（RW-）
图4：同一份文件数据的两种视角——多个 Section 组合成一个 Segment
关键关系

一个 Segment（段）可以包含多个 Section（节）；一个 Section 也可以属于多个 Segment。两者是"多对多"的覆盖关系，不是层级关系。文件中有些字节（如 ELF 头本身）不属于任何 Section，称为"孤儿字节"。

📌 本章小结

ELF 的双视图设计是其精妙之处：Section 是逻辑概念（给工具看的），Segment 是物理概念（给操作系统看的）。第一性原理：程序在"构建时"和"运行时"的需求完全不同——构建时需要细粒度符号信息，运行时只需知道"哪块内存要以什么权限映射到哪个地址"，ELF 用一套格式同时满足了两种需求。

第 5 章

节头表（Section Header Table）

原文：Linux man-pages elf(5) — Section header (Shdr)

节头表（SHT）是一个结构体数组，每项描述文件中的一个 Section。其位置由 ELF 头的 e_shoff 给出。

typedef struct {
    uint32_t   sh_name;      /* 节名，.shstrtab 中的偏移索引 */
    uint32_t   sh_type;      /* 节类型（SHT_*） */
    ElfN_Xword sh_flags;     /* 节属性标志（SHF_*） */
    ElfN_Addr  sh_addr;      /* 运行时虚拟地址（不加载则为0）*/
    ElfN_Off   sh_offset;    /* 节在文件中的字节偏移 */
    ElfN_Xword sh_size;      /* 节的字节大小 */
    uint32_t   sh_link;      /* 链接到其他节（类型相关） */
    uint32_t   sh_info;      /* 附加信息（类型相关） */
    ElfN_Xword sh_addralign; /* 内存对齐要求 */
    ElfN_Xword sh_entsize;   /* 若节含固定大小条目，每项大小 */
} ElfN_Shdr;

节类型（sh_type）

常量值含义
SHT_NULL0无效节，节头表第一项固定为此类型
SHT_PROGBITS1程序自定义内容（代码、数据等），由程序决定含义
SHT_SYMTAB2完整符号表（供链接/调试使用）
SHT_STRTAB3字符串表
SHT_RELA4带显式加数的重定位表
SHT_HASH5符号哈希表（用于快速动态符号查找）
SHT_DYNAMIC6动态链接信息
SHT_NOTE7注释信息（厂商或系统构建器附加信息）
SHT_NOBITS8不占文件空间的节（如 .bss）
SHT_REL9无显式加数的重定位表
SHT_DYNSYM11动态链接符号表（运行时最小符号集）

节标志（sh_flags）

标志含义
SHF_WRITE (0x1)节在进程执行时可写
SHF_ALLOC (0x2)节在进程执行时占用内存（需要被加载）
SHF_EXECINSTR (0x4)节包含可执行的机器指令
SHF_MERGE (0x10)节中相同数据可被合并以节省空间
SHF_STRINGS (0x20)节包含以 null 结尾的字符串
SHF_INFO_LINK (0x40)sh_info 存放节头表索引
SHF_LINK_ORDER (0x80)链接时需保持特定顺序
特殊节头表索引

SHN_UNDEF(0)：未定义节；SHN_ABS(0xfff1)：绝对地址符号（不受重定位影响）；SHN_COMMON(0xfff2)：公共块（C 语言未初始化全局变量的暂存区）。

📌 本章小结

节头表是链接器理解文件内容的核心索引。每个节头项告诉链接器该节的类型、在文件中的位置、大小、对齐要求及其与其他节的关联关系。注意：在最终的可执行文件中，节头表可以被完全剥离（strip 命令），不影响运行，因为运行时只需程序头表。

第 6 章

代码与数据节

原文：Oracle/Sun — ELF Predefined Sections

以下是 ELF 规范定义的核心用户节（Predefined User Sections），这些是构成可执行程序的基本组成部分：

节名类型标志详细说明
.text SHT_PROGBITS AX 可执行指令：程序的机器码，包含所有函数体。只读+可执行。是程序执行的核心，对应 CPU 取指操作的来源区域。
.data SHT_PROGBITS WA 已初始化读写数据：在源代码中已赋初值的全局变量和静态变量（如 int x = 5;）。初始值存在文件中，加载时原样复制到内存。
.bss SHT_NOBITS WA 未初始化数据："Block Started by Symbol"。未赋初值的全局/静态变量（如 int y;）。在文件中不占空间（SHT_NOBITS），加载时由 OS 分配内存并清零。
.rodata / .rodata1 SHT_PROGBITS A 只读数据：字符串字面量（如 "Hello"）、const 全局变量、switch 跳转表等。加载到只读内存页，写入会触发段错误（Segfault）。
.init SHT_PROGBITS AX 运行时初始化代码：程序 main() 执行前由运行时库（CRT）调用的初始化指令，执行 C++ 全局构造函数等。
.fini SHT_PROGBITS AX 运行时终止代码：程序退出时执行的清理代码，执行 C++ 全局析构函数、atexit() 注册的函数等。
.data1 SHT_PROGBITS WA .data 的补充节，某些链接器会生成此节，语义与 .data 相同。
图5：C 语言变量类型与 ELF 节的映射关系
.bss 的特殊性

.bss 类型为 SHT_NOBITS，意味着它在文件中不实际占用空间，只存储大小信息。这是一个重要的优化：如果一个程序有 1MB 的未初始化全局数组，ELF 文件中只需几个字节的描述，OS 在加载时负责分配内存并清零（ISO C 标准保证未初始化全局变量为 0）。

📌 本章小结

代码与数据节的划分体现了程序内存布局的基本原则：分离只读与可写（安全性）、分离代码与数据.bss 不占文件体积的设计是一个经典的空间优化例子——用"描述"代替"实际内容"。第一性原理：程序本质上是"指令"+"数据"，而数据又按"是否已知初值"+"是否可修改"进一步细分，ELF 节的划分正是这一分类的直接体现。

第 7 章

符号与字符串节

原文：Linux man-pages elf(5) — Symbol table

.symtab — 符号表

.symtab（类型 SHT_SYMTAB）是完整的符号表，包含程序中所有函数名、全局变量名及其地址信息。每项是一个 ElfN_Sym 结构：

typedef struct {
    uint32_t      st_name;   /* 符号名在 .strtab 中的偏移 */
    unsigned char st_info;   /* 符号类型和绑定属性 */
    unsigned char st_other;  /* 符号可见性 */
    uint16_t      st_shndx;  /* 所属节索引 */
    ElfN_Addr     st_value;  /* 符号值（地址或偏移量） */
    uint64_t      st_size;   /* 符号大小（字节数） */
} ElfN_Sym;

符号绑定类型（st_info 高4位）

绑定说明
STB_LOCAL本地符号，仅在目标文件内可见（如 static 函数）
STB_GLOBAL全局符号，对所有链接的目标文件可见
STB_WEAK弱符号，优先级低于同名全局符号，可被覆盖

符号类型（st_info 低4位）

类型说明
STT_NOTYPE类型未指定
STT_OBJECT数据对象（变量、数组等）
STT_FUNC函数或其他可执行代码
STT_SECTION与节关联的符号（主要用于重定位）
STT_FILE源文件名，与目标文件关联

符号可见性（st_other）

可见性说明
STV_DEFAULT默认可见性：全局/弱符号对外部模块可见，可被插值
STV_HIDDEN对外部模块不可见，本模块内部引用始终解析到本地定义
STV_PROTECTED对外部模块可见，但本模块内部引用不可被外部插值
STV_INTERNAL处理器特定的隐藏类

.dynsym — 动态符号表

.dynsym（类型 SHT_DYNSYM）是 .symtab 的子集，仅包含动态链接所需的最小符号集合。在运行时，动态链接器通过此表解析共享库函数。.symtab 可以被 strip，但 .dynsym 不能，否则动态链接失败。

.strtab — 字符串表

.strtab（类型 SHT_STRTAB）存储以 null 结尾的字符串序列。符号表（.symtab）的 st_name 字段是此表中的字节偏移，而非直接存储字符串，这避免了重复存储相同字符串。

.shstrtab — 节名字符串表

专门存储各节名称（".text"、".data" 等）的字符串表。ELF 头的 e_shstrndx 指向此节，节头表的 sh_name 是此表中的索引。

图6：.symtab 通过偏移索引引用 .strtab 中的字符串

📌 本章小结

符号表是链接器和调试器的"名称目录"，通过间接引用字符串表的设计避免了数据冗余。.symtab 服务于开发工具，.dynsym 服务于运行时动态链接器。第一性原理：名称（字符串）与值（地址）的分离存储，是数据库设计中"规范化"思想的体现——减少冗余，集中管理。

第 8 章

重定位节（Relocation Sections）

原文：Linux man-pages elf(5) — Relocation entries (Rel & Rela)

重定位（Relocation）是将代码中的符号引用（如函数调用地址）连接到其实际定义的过程。重定位节告诉链接器"在文件的哪个位置，需要填入什么值"。

两种重定位结构

Elf_Rel — 无显式加数

typedef struct {
    ElfN_Addr r_offset;  /* 需要修改的位置（文件偏移或虚拟地址）*/
    uintN_t   r_info;    /* 符号索引（高位）+ 重定位类型（低位） */
} ElfN_Rel;

Elf_Rela — 带显式加数（更常用于 x86-64）

typedef struct {
    ElfN_Addr r_offset;  /* 需要修改的位置 */
    uintN_t   r_info;    /* 符号索引 + 重定位类型 */
    intN_t    r_addend;  /* 显式加数常量 */
} ElfN_Rela;
字段说明
r_offset需要被修改的存储单元在文件（可重定位文件）或内存（可执行文件）中的位置
r_info高位：符号表索引（指向哪个符号）；低位：重定位类型（如何计算最终值）
r_addend计算最终地址时加上的常数偏移量（仅 Rela 有）

命名规则

节名示例说明
.rel.text针对 .text 节的 Rel 类型重定位表
.rela.text针对 .text 节的 Rela 类型重定位表（含加数）
.rel.data针对 .data 节的重定位表
.rela.plt针对 PLT（过程链接表）的重定位表，用于延迟绑定
图7：链接器根据重定位表将占位符替换为实际地址

📌 本章小结

重定位节解决了编译时无法确定最终地址的问题——编译器生成"占位符"，重定位节记录"哪里需要填什么"，链接器（静态）或动态链接器（动态）负责在适当时机填入实际值。第一性原理：这是"延迟绑定"哲学的体现——将不确定性推迟到信息可知的时刻解决。

第 9 章

动态链接节

原文：Linux man-pages elf(5) — Dynamic tags (Dyn) 原文：Oracle — Predefined Non-User Sections
节名类型说明
.dynamic SHT_DYNAMIC 动态链接信息核心节，包含一系列 Elf_Dyn 结构，列出所需共享库（DT_NEEDED）、符号表位置、重定位表位置等。动态链接器通过此节了解程序的所有动态链接需求。
.dynstr SHT_STRTAB 动态链接专用字符串表，存储动态链接所需的字符串（如库名 "libc.so.6"、符号名等）。
.dynsym SHT_DYNSYM 动态链接符号表，是 .symtab 的子集，仅含运行时所需符号。
.got SHT_PROGBITS 全局偏移表（Global Offset Table）：存储全局变量和外部函数的实际地址。由于共享库加载地址不固定，代码通过 GOT 间接访问外部符号，动态链接器在加载时填写 GOT 表项。
.got.plt SHT_PROGBITS GOT 中专门用于 PLT 的部分，存储外部函数的地址（延迟绑定时初始指向 PLT 存根，首次调用后更新为真实地址）。
.plt SHT_PROGBITS 过程链接表（Procedure Linkage Table）：为每个外部函数提供一个跳板（stub）。首次调用时触发延迟绑定，通过动态链接器解析函数地址并更新 GOT，后续调用直接通过 GOT 跳转，无需再次解析。
.hash SHT_HASH 符号哈希表，用于动态链接器快速查找符号（O(1) 平均复杂度）。GNU 扩展的 .gnu.hash 性能更优。
.interp SHT_PROGBITS 存储动态链接器（解释器）的路径字符串，如 /lib64/ld-linux-x86-64.so.2。内核在加载程序时首先读取此节，找到并启动动态链接器。

DT_STRTAB动态字符串表（.dynstr）的地址
DT_SYMTAB动态符号表（.dynsym）的地址
DT_PLTGOTPLT/GOT 地址

DT_RELA / DT_REL重定位表地址及大小
DT_INIT / DT_FINI初始化/终止函数地址

📌 本章小结

动态链接节构成了运行时软件生态的基础设施：.interp 指定解释器、.dynamic 描述依赖关系、.got/.plt 实现延迟绑定。这套机制让多个程序共享同一份库代码，大幅节省内存和磁盘空间。第一性原理：动态链接的核心是"间接性"——不直接写死地址，而通过一层跳转表实现运行时可修改的绑定。

第 11 章

程序头表（Program Header Table）

原文：Linux man-pages elf(5) — Program header (Phdr)

程序头表（PHT）是供操作系统加载器使用的 Segment 描述数组。只有可执行文件（ET_EXEC）和共享对象（ET_DYN）才有程序头表，纯粹的可重定位文件（ET_REL）没有。

typedef struct {
    uint32_t   p_type;    /* Segment 类型（PT_*） */
    Elf32_Off  p_offset;  /* Segment 在文件中的偏移 */
    Elf32_Addr p_vaddr;   /* Segment 的虚拟内存地址 */
    Elf32_Addr p_paddr;   /* Segment 的物理地址（通常不用）*/
    uint32_t   p_filesz;  /* Segment 在文件中的字节数 */
    uint32_t   p_memsz;   /* Segment 在内存中的字节数 */
    uint32_t   p_flags;   /* Segment 权限标志 */
    uint32_t   p_align;   /* 对齐要求 */
} Elf32_Phdr;
/* 64位版本字段顺序不同：p_flags 移至第2位 */

关键字段说明

字段说明
p_offset该 Segment 第一个字节在文件中的偏移量
p_vaddr该 Segment 在进程虚拟地址空间中的起始地址
p_paddr物理地址（对大多数 OS 不相关，BSD 下必须为 0）
p_fileszSegment 在文件中的实际字节数（.bss 贡献 0）
p_memszSegment 在内存中的字节数（≥ p_filesz，差值用 0 填充，即 .bss 部分）
p_flagsPF_X(1)=可执行；PF_W(2)=可写；PF_R(4)=可读
p_align内存和文件的对齐值，通常为页大小（4096），0/1 表示无要求
p_memsz > p_filesz 的含义

当 p_memsz > p_filesz 时，额外的字节在内存中被初始化为 0。这正是 .bss 节的处理方式：文件中无内容（p_filesz 不含 BSS 部分），但内存中需要分配并清零（p_memsz 包含 BSS 大小）。

📌 本章小结

程序头表是 OS 加载器的"施工图纸"，精确描述了"从文件哪里取多少字节，映射到内存什么地址，赋予什么权限"。p_filesz 与 p_memsz 的差值优雅地处理了 BSS 的零初始化问题。第一性原理：操作系统加载程序的本质是"建立虚拟地址到物理内容的映射"，程序头表正是这个映射的规格说明书。

段类型详解（Segment Types）

原文：Linux man-pages elf(5) — p_type 字段
类型常量值说明典型内容
PT_NULL 0 无效项，忽略此程序头表项 —
PT_LOAD 1 可加载段：最重要的类型。从文件的 p_offset 处读取 p_filesz 字节，映射到 p_vaddr，内存大小为 p_memsz（超出部分清零）。所有 PT_LOAD 段按 p_vaddr 升序排列。 .text+.rodata（R-X），.data+.bss（RW-）

第 13 章

动态链接原理

参考：OSDev Wiki — ELF Dynamic Linking

动态链接是在程序运行时（而非编译时）解析外部符号的机制。它使多个程序共享同一份库代码的内存映射，大幅节省资源。

动态链接的启动过程

执行流程（内核到 main()）

① 内核加载：内核读取 ELF 头，找到 PT_INTERP，获取动态链接器路径（如 /lib64/ld-linux-x86-64.so.2）。
② 启动动态链接器：内核将动态链接器映射到进程空间，控制权转交给它（而非 e_entry）。
③ 处理 .dynamic：动态链接器读取 PT_DYNAMIC，获取 DT_NEEDED 列表。
④ 加载依赖库：递归加载所有依赖的共享库（.so）到进程地址空间。
⑤ 符号解析：在各库的 .dynsym 中查找未定义符号，填写 GOT 表项。
⑥ 重定位：处理 .rela.plt、.rela.dyn 等重定位节。
⑦ RELRO 保护：若有 PT_GNU_RELRO，调用 mprotect() 将已完成重定位的 GOT 区域设为只读。
⑧ 跳转到 e_entry：控制权交给可执行文件的入口（通常是 _start，再调用 main()）。

静态链接 vs 动态链接对比

维度静态链接动态链接
链接时机编译/链接期运行期
文件大小大（包含所有库代码）小（只含引用，不含库代码）
内存共享每个进程独立一份库代码多进程共享库代码的物理内存页
启动速度快（无需运行时解析）稍慢（有链接器启动开销）
库更新需重新编译替换 .so 文件即可（API兼容时）
可移植性高（无外部依赖）依赖目标系统有相应库
安全安全漏洞需重编所有程序修复库后所有程序立即受益
图11：多进程共享 libc 物理内存页（各进程拥有独立虚拟地址映射）

📌 本章小结

动态链接的精髓是"共享"——共享代码页（节省内存）、共享维护成本（库更新即修复）、共享安全补丁（一处修复全局受益）。PLT/GOT 机制通过一层间接性（indirection）实现了运行时地址绑定。第一性原理：动态链接是对"重复"的系统级消除——代码只需在物理内存中存在一份，所有需要者通过虚拟地址映射共享访问。

第 14 章

重定位机制深入

原文：Linux man-pages elf(5) — Relocation entries

重定位的两个阶段

阶段执行者时机处理内容
静态重定位链接器 ld链接期（生成可执行文件时）合并多个 .o 文件，解析目标文件内部符号引用，填写段内地址
动态重定位动态链接器 ld.so运行期（程序加载时）解析外部共享库符号，填写 GOT 表项（即时或延迟绑定）

重定位类型（x86-64 常见示例）

重定位类型决定了链接器如何计算填入目标位置的值。r_info 的低位字段指定类型：

类型名计算公式典型用途
R_X86_64_64S + A64位绝对地址重定位
R_X86_64_PC32S + A - P32位PC相对地址（call/jmp指令）
R_X86_64_PLT32L + A - P通过PLT的函数调用
R_X86_64_GLOB_DATS填写GOT中全局变量地址
R_X86_64_JUMP_SLOTS填写GOT.PLT中函数地址（PLT延迟绑定）
R_X86_64_RELATIVEB + A位置无关可执行文件（PIE）的基址相对重定位

注：S=符号值，A=加数，P=被修改位置的地址，L=PLT表项地址，B=加载基地址

RELRO 安全机制

RELRO（Relocation Read-Only）是将完成重定位后不再需要修改的内存区域设为只读的安全特性：

Partial RELRO vs Full RELRO

Partial RELRO（默认）：仅将 .got（非PLT部分）设为只读，.got.plt 仍可写（支持延迟绑定）。

Full RELRO（-Wl,-z,relro,-z,now）：禁用延迟绑定（DT_BIND_NOW），启动时解析所有符号，然后将整个 GOT（含 .got.plt）设为只读。有效防御 GOT 覆写攻击，但增加启动时间。

📌 本章小结

重定位机制解决了"编译时不知道最终地址"的根本问题，分两阶段完成：链接期（静态）解析内部引用，运行期（动态）解析外部引用。RELRO 安全特性利用了"重定位完成后 GOT 不再需要写入"这一事实，通过 mprotect 将其保护起来。第一性原理：重定位是"引用"与"定义"的最终绑定——将抽象的名称映射到具体的地址。

总结与第一性原理

第一性原理总结

🔑 核心问题

如何在一个文件格式中，同时满足：链接器（需要细粒度符号信息）、调试器（需要源码映射）、操作系统（需要内存映射规格）、动态链接器（需要运行时依赖信息）等多种工具的不同需求？

ELF 的答案：双视图架构（Section/Segment）+ 可选表结构（不需要的表可省略）+ 间接引用（字符串表、符号表、GOT）

📐 设计原则映射

分离关注点：代码节 / 数据节 / 符号节 / 重定位节各司其职，没有一个节身兼多职。

延迟绑定：编译期 → 链接期 → 加载期 → 首次调用，每个阶段只解决当时可知的信息。

间接性即灵活性：GOT/PLT 的一层间接使运行时地址可变；字符串表的间接引用消除冗余。

用描述代替数据：.bss 不存零字节只存大小；重定位表只存"哪里需要填什么"而非直接填好。

可选即可裁剪：节头表可 strip、调试节可剥离、静态可执行无需程序头表以外的动态信息。ELF 在最小化和完整化之间提供了良好的连续谱。

📌 终章总结

ELF 格式的本质是一套描述语言，而非数据格式。它描述的是：

① 程序是什么（类型、架构、入口）→ ELF Header
② 程序由什么组成（代码、数据、符号）→ Sections + Section Header Table
③ 程序如何加载（内存布局、权限、地址）→ Segments + Program Header Table
④ 程序需要什么（依赖库、外部符号）→ .dynamic + .interp + .dynsym
⑤ 程序怎样修复自己（未知地址的填写）→ Relocation Sections

从第一性原理看：程序执行需要三样东西——指令、数据、地址。ELF 用最优雅的方式把这三者在不同阶段、对不同受众的表达方式统一在一个文件格式中。这是 ELF 成为 30 年来 Unix 生态事实标准的根本原因。
