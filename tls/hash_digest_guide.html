核心 API · Chapter 01

EVP 摘要接口详解

EVP（Envelope）是 OpenSSL 提供的高层密码学接口。摘要（Digest）模块通过统一的 Init→Update→Final 三步模型， 屏蔽了底层算法实现细节，使代码与具体算法解耦。

📖
导读：为什么使用 EVP 而非低层接口？

OpenSSL 官方文档明确指出："The EVP interface to message digests should almost always be used in preference to the low level interfaces."（应始终优先使用 EVP 接口而非低层接口。）原因是代码变得与算法无关，更加灵活。

🔑核心 API 一览

EVP_MD_CTX_create / EVP_MD_CTX_new

/* OpenSSL 1.0.x */ EVP_MD_CTX *EVP_MD_CTX_create(void); void EVP_MD_CTX_init(EVP_MD_CTX *ctx); /* OpenSSL 1.1.x+ */ EVP_MD_CTX *EVP_MD_CTX_new(void);

分配并返回一个摘要上下文。在 OpenSSL 1.1.0 中，EVP_MD_CTX_create() 被重命名为 EVP_MD_CTX_new()。两者功能等价。

1.0.x: EVP_MD_CTX_create 1.1.x+: EVP_MD_CTX_new

EVP_DigestInit_ex — 初始化摘要操作

int EVP_DigestInit_ex(EVP_MD_CTX *ctx, const EVP_MD *type, ENGINE *impl);
  • ctx已分配的摘要上下文，调用前须已初始化
  • type摘要类型，通常由 EVP_sha256() 等函数返回
  • implENGINE 实现，通常传 NULL 使用默认实现

返回值：成功返回 1，失败返回 0。

✅

优先使用 EVP_DigestInit_ex() 而非 EVP_DigestInit()，前者可复用上下文，性能更好，且允许指定非默认实现。

EVP_DigestUpdate — 输入数据

int EVP_DigestUpdate(EVP_MD_CTX *ctx, const void *d, size_t cnt);
  • ctx已初始化的摘要上下文
  • d指向待哈希数据的指针
  • cnt数据字节长度

可以在同一个上下文上多次调用本函数，以流式处理大文件或分块数据。每次调用都将数据累积进内部状态。

返回值：成功返回 1，失败返回 0。

EVP_DigestFinal_ex — 获取摘要值

int EVP_DigestFinal_ex(EVP_MD_CTX *ctx, unsigned char *md, unsigned int *s);
  • ctx完成 Update 的摘要上下文
  • md输出缓冲区，至少 EVP_MAX_MD_SIZE (64) 字节
  • s输出实际摘要字节数（可为 NULL）

调用后，上下文不可再调用 Update，但可调用 Init_ex 重新初始化以复用。

返回值：成功返回 1，失败返回 0。

EVP_MD_CTX_destroy / EVP_MD_CTX_free — 释放上下文

/* OpenSSL 1.0.x */ void EVP_MD_CTX_destroy(EVP_MD_CTX *ctx); int EVP_MD_CTX_cleanup(EVP_MD_CTX *ctx); /* OpenSSL 1.1.x+ */ void EVP_MD_CTX_free(EVP_MD_CTX *ctx);

清理并释放摘要上下文内存。在 1.1.0 中 EVP_MD_CTX_destroy() 被重命名为 EVP_MD_CTX_free()。必须在上下文不再使用后调用，否则发生内存泄漏。

1.0.x: EVP_MD_CTX_destroy 1.1.x+: EVP_MD_CTX_free

EVP_Digest — 一次性便捷接口 1.1.x+

int EVP_Digest(const void *data, size_t count, unsigned char *md, unsigned int *size, const EVP_MD *type, ENGINE *impl);

Init_ex + Update + Final_ex 的封装。适合一次性哈希短数据，无需手动管理上下文。

EVP_MD_CTX_copy_ex — 复制上下文

int EVP_MD_CTX_copy_ex(EVP_MD_CTX *out, const EVP_MD_CTX *in);

将 in 的消息摘要状态复制到 out。out 必须已初始化。适用于大量数据只在末尾几字节不同的场景（如 Merkle 树计算）。

🔍辅助信息 API

EVP_MD_size(md) 返回摘要字节长度（如 SHA-256 返回 32） int，字节数

核心 API · Chapter 02

上下文生命周期与调用序列

理解 EVP_MD_CTX 的生命周期是正确使用 OpenSSL 摘要 API 的关键。 错误的调用顺序或遗漏清理操作会导致内存泄漏或程序崩溃。

⏱️完整生命周期流程图

图 2-1：EVP_MD_CTX 完整生命周期 MSC 时序图（虚线表示可选的重用路径）

📝调用规则总结

  1. 创建上下文 1.0.x 使用 EVP_MD_CTX_create()，1.1.x+ 使用 EVP_MD_CTX_new()。返回指针必须检查是否为 NULL。
  2. 初始化算法 调用 EVP_DigestInit_ex(ctx, EVP_sha256(), NULL)。第三个参数 ENGINE 通常传 NULL。
  3. 输入数据（可多次） EVP_DigestUpdate() 可被反复调用，适合流式处理文件或网络数据包。
  4. 输出摘要 EVP_DigestFinal_ex() 输出摘要，之后不能再调用 Update。可调用 Init_ex 重新初始化后复用。
  5. 释放上下文 必须调用 EVP_MD_CTX_free()（1.1.x+）或 EVP_MD_CTX_destroy()（1.0.x）。即使中途出错也要释放。

核心 API · Chapter 03

支持的摘要算法

OpenSSL EVP 提供了从经典 MD5、SHA-1 到现代 SHA-2、SHA-3 的全系列摘要算法， 以及获取算法对象的两种方式：类型函数（如 EVP_sha256()）和名称查找（EVP_get_digestbyname）。

📖
导读：算法选择的重要性

官方文档建议："New applications should use the SHA2 digest algorithms such as SHA256."（新应用应使用 SHA-2，如 SHA-256。）MD2 自 OpenSSL 1.0 起已从默认配置中移除。

🔍按名称获取算法

使用 EVP_get_digestbyname() 可通过字符串名称动态获取摘要算法，无需在编译时硬编码：

#include <openssl/evp.h>
#include <openssl/opensslv.h>

/* 在 1.0.x 中，使用前需要调用 OpenSSL_add_all_digests() */
#if OPENSSL_VERSION_NUMBER < 0x10100000L
    OpenSSL_add_all_digests();
#endif

const EVP_MD *md = EVP_get_digestbyname("sha256");
if (md == NULL) {
    /* 算法不存在或未初始化 */
}
⚠️
OpenSSL 1.0.x 注意事项

在 OpenSSL 1.0.x 中，使用 EVP_get_digestbyname() 前必须先调用 OpenSSL_add_all_digests()（或 OpenSSL_add_all_algorithms()）来初始化摘要表。OpenSSL 1.1.0+ 会自动完成这一步。

进阶用法 · Chapter 04

HMAC 接口 — 带密钥的消息认证

HMAC（Hash-based Message Authentication Code）是基于哈希函数的消息认证码， 需要密钥参与，可同时提供完整性验证和来源认证。OpenSSL 提供了专用的 HMAC API。

📖
导读：HMAC vs 普通 Hash

普通摘要（如 SHA-256）只验证完整性，任何人都能重新计算。HMAC 增加了密钥，使得攻击者在不知道密钥的情况下无法伪造认证码，遵循 RFC 2104 规范。

🔐HMAC API 列表

HMAC — 一次性计算

unsigned char *HMAC( const EVP_MD *evp_md, const void *key, int key_len, const unsigned char *d, int n, unsigned char *md, unsigned int *md_len);

对 n 字节数据 d 使用密钥 key 和哈希函数 evp_md 计算 HMAC。结果写入 md（至多 EVP_MAX_MD_SIZE 字节），长度写入 md_len。

⚠️

如果 md 传 NULL，结果存入静态数组，非线程安全，生产环境不应使用。

流式 HMAC API（推荐用于大数据）

/* 1.0.x */ void HMAC_CTX_init(HMAC_CTX *ctx); /* 初始化栈上结构 */ void HMAC_CTX_cleanup(HMAC_CTX *ctx); /* 清理（不释放指针本身）*/ /* 1.1.x+ */ HMAC_CTX *HMAC_CTX_new(void); void HMAC_CTX_free(HMAC_CTX *ctx); /* 通用（1.0.x ~ 1.1.x） */ int HMAC_Init_ex(HMAC_CTX *ctx, const void *key, int key_len, const EVP_MD *md, ENGINE *impl); int HMAC_Update(HMAC_CTX *ctx, const unsigned char *data, int len); int HMAC_Final(HMAC_CTX *ctx, unsigned char *md, unsigned int *len);

⚡HMAC 工作原理图

图 4-1：HMAC 双哈希结构（RFC 2104），ipad=0x36，opad=0x5C

📋HMAC_Init_ex 详细说明

官方文档说明："HMAC_Init_ex() initializes or reuses a HMAC_CTX structure to use the function evp_md and key key. Either can be NULL, in which case the existing one will be reused."

⚠️
关键限制（官方文档原文警告）

如果以 key=NULL 调用 HMAC_Init_ex()，且 evp_md 与上一次使用的摘要算法不同，则返回错误——因为不支持以不同摘要复用已有密钥。

🔢HMAC_Init_ex 返回值变化（版本历史）

ℹ️
版本差异（官方文档记录）

在 OpenSSL 1.0.0 之前，HMAC_Init_ex()、HMAC_Update() 和 HMAC_Final() 没有返回值（void）。自 1.0.0 起才返回 int（成功=1，失败=0）。在兼容 1.0.x 时务必注意此差异。

官方原文：OpenSSL 1.0.2 hmac(3) man page

使用场景分析

哈希摘要 API 在实际工程中有多种典型应用场景，每种场景对算法选择和 API 使用方式都有不同要求。

🎯场景全景图

图 5-1：EVP 哈希 API 的主要使用场景分类

📂场景详解

① 文件完整性校验

  • 推荐算法：SHA-256 / SHA-512
  • 流式读取文件，多次调用 Update
  • 输出 hex 字符串供用户比对
  • 典型：软件包下载校验、固件完整性

② 密码存储

  • 不应直接哈希密码（彩虹表攻击）
  • 必须加盐（随机盐，每用户唯一）
  • 推荐使用 PBKDF2（OpenSSL PKCS5_PBKDF2_HMAC）
  • MD5/SHA1 绝对不能用于密码存储

③ 消息认证码 (HMAC)

  • API 请求签名（HMAC-SHA256）
  • JWT token 签名验证
  • 网络通信数据防篡改
  • 必须保护密钥安全

④ 数字签名预哈希

  • RSA/ECDSA 只能对固定长度数据签名
  • 先用 SHA-256 哈希，再对摘要签名
  • 使用 EVP_DigestSign* 系列一体化接口
  • 不要手动拆分：Init/Update/Final 一步完成

⑤ 上下文复制加速

  • 使用 EVP_MD_CTX_copy_ex()
  • 多文件共同前缀只哈希一次
  • Merkle Tree 节点计算
  • 适合大数据末尾不同场景

⑥ 内容寻址 / 去重

  • Git 用 SHA-1（已迁移 SHA-256）
  • 对象存储内容唯一 Key 生成
  • 缓存 Key 生成（参数哈希）
  • 数据去重（块级哈希比较）

📊算法选择指南

场景推荐算法禁止使用原因
通用数据完整性 SHA-256 MD5, SHA-1 抗碰撞性，SHA-1 已被攻破
API 签名 / HMAC HMAC-SHA256 HMAC-MD5 安全强度，互操作性
TLS 1.2 协议 SHA-256/384 MD5, SHA-1 RFC 强制要求
密码存储 PBKDF2+SHA256 直接 SHA256 需要慢速哈希抗暴力破解
非安全校验（速度优先） MD5 / SHA-1 — 内部不安全场景如缓存 Key
进阶用法 · Chapter 06

版本兼容策略

OpenSSL 1.0.x 到 1.1.0 有重大 API 变更，EVP_MD_CTX 从可栈分配的结构体变为不透明指针。 本章提供兼容 1.0.1 及后续所有版本的编写策略。

📖
导读：为何 1.1.0 是重要分水岭？

OpenSSL 1.1.0 将所有公开头文件中的结构体变为不透明指针（opaque structs），包括 EVP_MD_CTX。这意味着不能再在栈上声明 EVP_MD_CTX ctx;，只能通过指针和 API 操作。同时函数被重命名。

📋API 变更对照表

功能OpenSSL 1.0.xOpenSSL 1.1.x+说明
创建上下文（堆） EVP_MD_CTX_create() EVP_MD_CTX_new() 重命名，功能相同
初始化（栈上结构） EVP_MD_CTX_init(&ctx) ❌ 不支持栈分配 1.1.x 结构体不透明
清理（栈上结构） EVP_MD_CTX_cleanup(&ctx) ❌ 已移除 改用 EVP_MD_CTX_free
销毁（堆指针） EVP_MD_CTX_destroy(ctx) EVP_MD_CTX_free(ctx) 重命名，功能相同
重置上下文 — EVP_MD_CTX_reset(ctx) 1.1.x 新增
上下文结构 可栈分配 EVP_MD_CTX ctx; 仅指针 EVP_MD_CTX *ctx; 1.1.x 起不透明
HMAC 创建上下文 栈上 HMAC_CTX_init() HMAC_CTX_new() 类似 EVP_MD_CTX
摘要表初始化 OpenSSL_add_all_digests() 自动完成 1.1.x 无需手动调用

🛠️兼容性宏策略

以下是官方社区推荐的兼容宏写法（来自 liboauth、nmap 等项目的实践）：

#include <openssl/opensslv.h>
#include <openssl/evp.h>
#include <openssl/hmac.h>

/* ── EVP_MD_CTX 兼容层 ─────────────────────────────── */
#if OPENSSL_VERSION_NUMBER < 0x10100000L
#  define EVP_MD_CTX_new()      EVP_MD_CTX_create()
#  define EVP_MD_CTX_free(ctx)  EVP_MD_CTX_destroy(ctx)
#  define EVP_MD_CTX_reset(ctx) EVP_MD_CTX_cleanup(ctx)
#endif

/* ── HMAC_CTX 兼容层 ───────────────────────────────── */
#if OPENSSL_VERSION_NUMBER < 0x10100000L
static inline HMAC_CTX *HMAC_CTX_new(void) {
    HMAC_CTX *ctx = (HMAC_CTX *)OPENSSL_malloc(sizeof(HMAC_CTX));
    if (ctx) HMAC_CTX_init(ctx);
    return ctx;
}
static inline void HMAC_CTX_free(HMAC_CTX *ctx) {
    if (ctx) {
        HMAC_CTX_cleanup(ctx);
        OPENSSL_free(ctx);
    }
}
#endif

/* ── 使用方式（1.0.x 和 1.1.x+ 完全一致）────────────── */
EVP_MD_CTX *ctx = EVP_MD_CTX_new();   /* 两版本均可用 */
EVP_DigestInit_ex(ctx, EVP_sha256(), NULL);
EVP_DigestUpdate(ctx, data, len);
EVP_DigestFinal_ex(ctx, out, &out_len);
EVP_MD_CTX_free(ctx);                  /* 两版本均可用 */

示例 · Chapter 07

完整代码示例

以下四个示例均基于官方文档原始代码，扩展了兼容层和错误处理， 可在 OpenSSL 1.0.1 及之后版本直接编译运行。

💻示例一：SHA-256 哈希（版本兼容）

基于 OpenSSL 官方示例，增加了版本兼容宏和完整错误处理：

#include <stdio.h>
#include <string.h>
#include <openssl/evp.h>
#include <openssl/opensslv.h>
#include <openssl/err.h>

/* ── 版本兼容宏（兼容 1.0.1+） ─────────────── */
#if OPENSSL_VERSION_NUMBER < 0x10100000L
#  define EVP_MD_CTX_new()      EVP_MD_CTX_create()
#  define EVP_MD_CTX_free(p)    EVP_MD_CTX_destroy(p)
#endif

int sha256_digest(const unsigned char *data, size_t data_len,
                   unsigned char *out, unsigned int *out_len)
{
    EVP_MD_CTX *ctx = NULL;
    int ret = 0;

#if OPENSSL_VERSION_NUMBER < 0x10100000L
    OpenSSL_add_all_digests(); /* 1.0.x 必须调用 */
#endif

    ctx = EVP_MD_CTX_new();
    if (!ctx) goto err;

    if (!EVP_DigestInit_ex(ctx, EVP_sha256(), NULL))
        goto err;

    if (!EVP_DigestUpdate(ctx, data, data_len))
        goto err;

    if (!EVP_DigestFinal_ex(ctx, out, out_len))
        goto err;

    ret = 1;

err:
    EVP_MD_CTX_free(ctx);   /* NULL 安全：传 NULL 无操作 */
    if (!ret)
        ERR_print_errors_fp(stderr);
    return ret;
}

int main(void)
{
    const char *mess1 = "Test Message\n";
    const char *mess2 = "Hello World\n";
    unsigned char  buf[EVP_MAX_MD_SIZE];
    unsigned int   len = 0;
    EVP_MD_CTX    *ctx = EVP_MD_CTX_new();
    unsigned int   i;

    EVP_DigestInit_ex(ctx, EVP_sha256(), NULL);
    EVP_DigestUpdate(ctx, mess1, strlen(mess1)); /* 分两次输入 */
    EVP_DigestUpdate(ctx, mess2, strlen(mess2));
    EVP_DigestFinal_ex(ctx, buf, &len);
    EVP_MD_CTX_free(ctx);

    printf("SHA-256 Digest: ");
    for (i = 0; i < len; i++)
        printf("%02x", buf[i]);
    printf("\n");
    return 0;
}
/* 编译：gcc -o sha256_demo sha256_demo.c -lssl -lcrypto */

💻示例二：流式文件哈希

适合大文件场景，使用 Update 分块处理：

#include <stdio.h>
#include <openssl/evp.h>
#include <openssl/opensslv.h>

#if OPENSSL_VERSION_NUMBER < 0x10100000L
#  define EVP_MD_CTX_new()   EVP_MD_CTX_create()
#  define EVP_MD_CTX_free(p) EVP_MD_CTX_destroy(p)
#endif

int hash_file(const char *path)
{
    FILE         *fp  = fopen(path, "rb");
    EVP_MD_CTX   *ctx = NULL;
    unsigned char buf[4096];         /* 读取缓冲区 */
    unsigned char digest[EVP_MAX_MD_SIZE];
    unsigned int  digest_len, i;
    size_t        n;

    if (!fp) { perror(path); return -1; }

    ctx = EVP_MD_CTX_new();
    EVP_DigestInit_ex(ctx, EVP_sha256(), NULL);

    while ((n = fread(buf, 1, sizeof(buf), fp)) > 0)
        EVP_DigestUpdate(ctx, buf, n); /* 反复调用 Update */

    EVP_DigestFinal_ex(ctx, digest, &digest_len);
    EVP_MD_CTX_free(ctx);
    fclose(fp);

    printf("%s  %s\n",
           ({ static char h[65];
              for(i=0;i<digest_len;i++) sprintf(h+i*2,"%02x",digest[i]);
              h; }),
           path);
    return 0;
}

💻示例三：HMAC-SHA256（跨版本兼容）

#include <stdio.h>
#include <string.h>
#include <openssl/hmac.h>
#include <openssl/evp.h>
#include <openssl/opensslv.h>

/* HMAC_CTX 兼容层（适用于 1.0.x） */
#if OPENSSL_VERSION_NUMBER < 0x10100000L
static inline HMAC_CTX *HMAC_CTX_new(void) {
    HMAC_CTX *c = (HMAC_CTX *)OPENSSL_malloc(sizeof(*c));
    if (c) HMAC_CTX_init(c);
    return c;
}
static inline void HMAC_CTX_free(HMAC_CTX *c) {
    HMAC_CTX_cleanup(c);
    OPENSSL_free(c);
}
#endif

int main(void)
{
    const unsigned char key[]  = "supersecretkey";
    const unsigned char data[] = "Hello, HMAC!";
    unsigned char       result[EVP_MAX_MD_SIZE];
    unsigned int        result_len = 0;
    unsigned int        i;
    HMAC_CTX           *ctx;

    ctx = HMAC_CTX_new();
    if (!ctx) return 1;

    /* 初始化：密钥 + 算法 */
    HMAC_Init_ex(ctx, key, (int)strlen((char*)key),
                 EVP_sha256(), NULL);

    /* 输入消息（可多次调用） */
    HMAC_Update(ctx, data, strlen((char*)data));

    /* 获取 MAC */
    HMAC_Final(ctx, result, &result_len);
    HMAC_CTX_free(ctx);

    printf("HMAC-SHA256: ");
    for (i = 0; i < result_len; i++)
        printf("%02x", result[i]);
    printf("\n");
    return 0;
}

💻示例四：上下文复制（批量哈希加速）

/* 场景：计算 N 个文件的哈希，文件共享一个公共前缀头部 */
#include <openssl/evp.h>

#if OPENSSL_VERSION_NUMBER < 0x10100000L
#  define EVP_MD_CTX_new()   EVP_MD_CTX_create()
#  define EVP_MD_CTX_free(p) EVP_MD_CTX_destroy(p)
#endif

void batch_hash_with_prefix(
    const unsigned char *prefix, size_t prefix_len,
    const unsigned char **msgs,  size_t *msg_lens, int n)
{
    EVP_MD_CTX   *base_ctx = EVP_MD_CTX_new();
    EVP_MD_CTX   *work_ctx = EVP_MD_CTX_new();
    unsigned char digest[EVP_MAX_MD_SIZE];
    unsigned int  digest_len;
    int i, j;

    /* 公共前缀只哈希一次 */
    EVP_DigestInit_ex(base_ctx, EVP_sha256(), NULL);
    EVP_DigestUpdate(base_ctx, prefix, prefix_len);

    for (i = 0; i < n; i++) {
        /* 复制已包含前缀的状态，避免重复计算 */
        EVP_MD_CTX_copy_ex(work_ctx, base_ctx);
        EVP_DigestUpdate(work_ctx, msgs[i], msg_lens[i]);
        EVP_DigestFinal_ex(work_ctx, digest, &digest_len);

        printf("msg[%d] digest: ", i);
        for (j = 0; j < (int)digest_len; j++)
            printf("%02x", digest[j]);
        printf("\n");
    }

    EVP_MD_CTX_free(base_ctx);
    EVP_MD_CTX_free(work_ctx);
}
注意事项 · Chapter 08

注意事项与常见陷阱

在工程实践中，OpenSSL 哈希 API 有多个容易踩坑的地方， 涵盖内存管理、线程安全、算法安全性和版本差异等方面。

💾内存管理

⚠️
陷阱①：忘记释放上下文导致内存泄漏

官方文档原文："In OpenSSL 0.9.7 and later if digest contexts are not cleaned up after use memory leaks will occur."
必须在所有代码路径（包括错误路径）上释放上下文，推荐使用 goto err 模式。

/* ❌ 错误：错误路径遗漏释放 */
EVP_MD_CTX *ctx = EVP_MD_CTX_new();
if (!EVP_DigestInit_ex(ctx, EVP_sha256(), NULL))
    return -1; /* ctx 泄漏！ */

/* ✅ 正确：使用 goto 统一清理 */
EVP_MD_CTX *ctx = EVP_MD_CTX_new();
if (!EVP_DigestInit_ex(ctx, EVP_sha256(), NULL))
    goto cleanup;
/* ... */
cleanup:
    EVP_MD_CTX_free(ctx); /* 传 NULL 时安全（1.1.x+）*/

🔢输出缓冲区大小

⚠️
陷阱②：输出缓冲区过小

官方文档："at most EVP_MAX_MD_SIZE bytes will be written"。EVP_MAX_MD_SIZE 为 64（SHA-512 的字节长度）。应始终使用此宏而非硬编码长度，或使用 EVP_MD_size() 精确获取。

/* ❌ 错误：硬编码缓冲区（将来切换算法会溢出）*/
unsigned char buf[32]; /* SHA-256 刚好，SHA-512 则溢出！*/

/* ✅ 正确：使用宏保证足够大 */
unsigned char buf[EVP_MAX_MD_SIZE]; /* 64 字节，永远安全 */
unsigned int  buf_len;
EVP_DigestFinal_ex(ctx, buf, &buf_len); /* 实际长度存入 buf_len */

🧵线程安全

⚠️
陷阱③：HMAC() 的静态数组非线程安全

一次性 HMAC() 函数：若 md 参数传 NULL，结果存入内部静态数组，官方文档注明 "Note: passing a NULL value for md to use the static array is not thread safe"。多线程场景必须传入自己分配的缓冲区。

/* ❌ 非线程安全 */
unsigned char *result = HMAC(EVP_sha256(), key, klen,
                              data, dlen,
                              NULL, NULL); /* NULL → 静态数组 */

/* ✅ 线程安全 */
unsigned char result[EVP_MAX_MD_SIZE];
unsigned int  result_len;
HMAC(EVP_sha256(), key, klen, data, dlen,
     result, &result_len);  /* 使用自己的缓冲区 */

🔒算法安全性警告

🚫 禁止场景

  • MD5 用于安全哈希（碰撞已实现）
  • SHA-1 用于数字证书签名（已废弃）
  • 直接哈希密码（无盐）
  • 短密钥 HMAC（密钥短于摘要长度）

✅ 推荐实践

  • 安全场景用 SHA-256 或更强
  • 密码使用 PBKDF2/bcrypt/scrypt
  • HMAC 密钥长度 ≥ 摘要输出长度
  • 新项目使用 SHA-3 系列（1.1.x+）

🚧调用顺序错误

⚠️
陷阱④：Final 后再调用 Update

官方文档明确："After calling EVP_DigestFinal_ex() no additional calls to EVP_DigestUpdate() can be made"。若需继续，必须先调用 EVP_DigestInit_ex() 重新初始化。

第一性原理 · Chapter 09

第一性原理总结

回归本质：从哈希函数的数学性质出发，推导出 OpenSSL EVP API 的设计决策， 理解「为什么是这样」而不仅仅是「是什么」。

🧱第一性原理：哈希函数的本质

图 9-1：哈希函数的数学本质 — 从任意长度输入映射到固定长度输出

🔬从本质推导 API 设计

①

输入任意长度 → Init/Update/Final 流式设计

哈希函数处理任意长度数据，而现实中数据可能来自文件流、网络包，无法全部载入内存。因此 API 被设计为流式：Init 设定算法，Update 分批输入，Final 产出固定输出。这不是设计喜好，而是数学性质的直接映射。

②

确定性 → 上下文（Context）状态机设计

同样的输入必须产生同样的输出（确定性）。为了在多次 Update 调用中维持「已处理多少数据、当前中间状态是什么」，必须有一个承载状态的对象——这就是 EVP_MD_CTX。状态机模式正是为了保证确定性而不得不引入的结构。

③

单向性 + 不可逆 → API 没有「解哈希」接口

哈希函数的核心性质之一：不可逆（无法从 h 推导 M）。这决定了 OpenSSL 摘要 API 从设计上就没有任何逆操作——没有解密函数，没有还原接口。验证只能通过重新计算哈希比对，不能反向推算。

④

抗碰撞性 → 算法替换不改变接口（EVP 抽象层）

随着算法被攻破（MD5 碰撞 2004 年实现，SHA-1 2017 年），需要替换更安全的算法。EVP 的抽象层设计使得只需修改传入的算法类型（如 EVP_sha256()），业务代码无需变动——这正是「抗碰撞性失效时的工程应对方案」。

⑤

HMAC 是「密钥化的哈希」— 来自安全需求的最小扩展

纯哈希无法防伪造（任何人都能重算）。HMAC 的本质是：在不改变哈希函数数学结构的前提下，通过双层嵌套结合密钥引入「只有持有密钥者才能生成正确 MAC」的性质。API 设计（Init_ex 增加 key 参数）是安全需求对原始哈希接口的最小扩展。

⑥

EVP_MAX_MD_SIZE = 64 — 来自物理上界约束

SHA-512 输出 512 bit = 64 字节，目前 EVP 支持的最长摘要。EVP_MAX_MD_SIZE 是 OpenSSL 对当前所有支持算法输出长度的物理上界约束，是一个工程安全边界，不是随意定义的常量。使用它而非硬编码，就是在遵循这个物理事实。

🎯一句话总结

OpenSSL EVP 哈希 API 的所有设计，
都是哈希函数三大数学性质的工程化映射：

确定性
→ 上下文状态机
不可逆
→ 无解密接口
抗碰撞性
→ EVP 算法抽象
