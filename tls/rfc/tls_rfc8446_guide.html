TLS 1.3 协议图解导读

RFC 8446 定义 TLS 第三版：1-RTT 完整握手、可选 0-RTT、HKDF 密钥调度、加密握手消息，并移除 RSA 密钥传输与大量弱密码套件。

HKDFAEAD-onlykey_sharepre_shared_key0-RTT

1 · 简介

TLS 1.3 在 TLS 1.2（RFC 5246）基础上大幅简化并强化安全：握手更快、默认前向保密、握手中间消息加密，协议状态机更清晰。

主要变化（§1.2）

  • 完整握手通常 1-RTT（ServerHello 后即可发应用数据）
  • 支持 0-RTT 早期数据（基于 PSK 恢复）
  • 仅 AEAD 密码套件；移除 CBC、压缩、重协商、RSA 密钥传输
  • 密钥派生统一为 HKDF，取代 TLS 1.2 PRF + master_secret
  • 握手消息使用 handshake_traffic_secret 加密

2 · 相对 TLS 1.2 的差异

方面TLS 1.2TLS 1.3

3 · 记录层（§5）

TLS 记录将上层数据分片、压缩（1.3 禁用）、添加 MAC/加密、封装为 TLSPlaintext / TLSCiphertext。

TLSPlaintext: type(1) + legacy_version(2) + length(2) + fragment
TLSCiphertext (AEAD): opaque_type(1)=23 + legacy_version(2) + length(2) + encrypted{ inner.type + inner.version + inner.length + fragment + padding }

TLS 1.3 在 AEAD 记录内使用内部明文类型（handshake / application_data / alert），外层 opaque_type 固定为 23（application_data）以隐藏真实类型，防流量分析。

5 · 握手三阶段

握手逻辑阶段
  • 密钥交换：ClientHello / ServerHello 确定 (EC)DHE 共享秘密和/或 PSK
  • 服务器参数：EncryptedExtensions 回应非密码学扩展；CertificateRequest 可选
  • 认证：服务器（及可选客户端）证书 + CertificateVerify + Finished 确认握手完整性

6 · 三种密钥交换模式（§1.3）

模式组成特点
(EC)DHEkey_share 扩展默认，提供前向保密（PFS）
PSK-onlypre_shared_key无 PFS，适合会话恢复或预共享密钥
PSK + (EC)DHE两者兼有恢复加速 + PFS，推荐组合

7 · 0-RTT 早期数据（§2.3 · §8）

若客户端持有来自先前握手的 PSK（NewSessionTicket），可在第一个 ClientHello 之后立即发送应用数据，无需等待 ServerHello（Figure 4）。

0-RTT 消息流（简化）
安全 weaker：0-RTT 数据无前向保密，且可能被重放。服务器须实现反重放（§8），应用层须幂等处理 0-RTT 请求。
  • 密钥：client_early_traffic_secret（来自 Early Secret + "c e traffic"）
  • 服务器可在 EncryptedExtensions 中拒绝 early_data
  • EndOfEarlyData 标记 0-RTT 结束，切换到 handshake traffic keys

8 · 会话恢复与 PSK（§2.2 · §4.6.1）

TLS 1.3 用 PSK 模式替代 TLS 1.2 的 session ID / session ticket 恢复机制：

  • 握手结束时服务器发 NewSessionTicket，内含 PSK 标识与 lifetime
  • 下次连接 ClientHello 带 pre_shared_key 扩展引用该 ticket
  • resumption_master_secret 从 Master Secret 派生，用于生成恢复 PSK
  • PSK binder 用 binder_key（"res binder" label）绑定 ClientHello，防 ticket 篡改
带 (EC)DHE 的 PSK 恢复 = 1-RTT + PFS；纯 PSK 恢复更快但无 PFS。

9 · §7.1 Key Schedule（HKDF）

TLS 1.3 从两个输入秘密派生全部密钥：PSK（外部或恢复）与 (EC)DHE 共享秘密。使用 HKDF-Extract + Derive-Secret + HKDF-Expand-Label。

HKDF-Expand-Label(Secret, Label, Context, Length) = HKDF-Expand(Secret, HkdfLabel, Length)
HkdfLabel = uint16(Length) || "tls13 " + Label || Context
Derive-Secret(Secret, Label, Messages) = HKDF-Expand-Label(Secret, Label, Transcript-Hash(Messages), Hash.length)
密钥调度总览（RFC 8446 §7.1）

若某秘密不可用，用 Hash.length 字节的零串代替（仍执行完整 HKDF 轮次）。派生完成后应擦除中间秘密。

KeyUpdate（§7.2）

握手完成后可用 KeyUpdate 消息轮换 application traffic secret：application_traffic_secret_N+1 = HKDF-Expand-Label(N, "traffic upd", "", Hash.length)

10 · ContentType 与 AEAD 封装（§5.1）

类型值用途
invalid0保留
change_cipher_spec20仅兼容模式
alert21告警
handshake22握手消息
application_data23应用数据；AEAD 外层 opaque_type 也固定为此值

AEAD 加密前将真实 ContentType 写入明文末尾，并填充至块边界，使长度不泄露消息类型。

11 · 与 RFC 5705 Exporter 的关系（§7.5）

RFC 8446 更新 RFC 5705：不再从 master_secret 做 PRF，而是从 exporter_master_secret（或 0-RTT 的 early_exporter_master_secret）经 HKDF 导出。

TLS-Exporter(label, context_value, length) = HKDF-Expand-Label(Derivation-Secret, label, Hash(context_value), length) Derivation-Secret = early_exporter_master_secret (0-RTT) or exporter_master_secret (1-RTT+)

详见 RFC 5705 导读 中的 label/context 语义；TLS 1.3 要求 label 以 EXPORTER 开头（实验除外）。

12 · 安全考虑摘要

  • 降级保护：supported_versions、downgrade_canary（RFC 8446 §4.1.3）防 TLS 1.3 → 1.2 降级
  • 0-RTT 重放：服务器必须实现 §8 反重放策略；应用须假设 0-RTT 可被重放
  • 中间盒兼容：Appendix D 定义「兼容模式」伪造 ChangeCipherSpec，非安全必需
  • 证书验证：与 1.2 相同原则 — 须验证链、主机名（SNI，见 RFC 6066）、用途
  • 密钥擦除：实现应在用完后清零 PSK、ECDH 共享秘密与各阶段 secret
