TLS 1.2 完整握手抓包分析

基于 tls_full_flow.pcap（13 帧 · 约 1.6 s）还原 TCP 建连 → TLS 1.2 双向认证握手 → 加密告警关闭。端口 8883 表明为 MQTTS / 通用 TLS 服务场景。

TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384 curve: x25519 mTLS extended_master_secret

1 · 抓包摘要

本抓包为典型 TLS 1.2 完整握手（非会话恢复）：服务器要求客户端证书（CertificateRequest），双方交换 ECDHE 公钥后派生主密钥，再经 ChangeCipherSpec 切换至 AEAD 记录保护。

阶段帧Δt内容
TCP#1–#30 – 0.02 ms三次握手
TLS协商#41.68 msClientHello
TLS协商#62.35 msServerHello + Certificate + SKE + CertReq + SHD（同帧 5 条记录）
TLS认证#84.23 msClient Cert + CKE + CertVerify + CCS + Finishedenc
TLS完成#94.57 msNewSessionTicket + CCS + Finishedenc
加密#121603 msAlert（加密 close_notify）
TCP#11,#13~1603 msFIN / ACK 挥手
协商套件 0xC030 = ECDHE_RSA + AES-256-GCM + SHA384 PRF；SKE 选用 X25519（NamedCurve 29，RFC 8422）。帧 #8/#9 中 40 字节 Handshake 记录为已加密的 Finished，Wireshark 在未解密时无法展开类型字节。

6 · ClientHello（帧 #4）

记录层封装

字段抓包值处理说明
content_type22 (handshake)交付握手协议解析器
legacy_version0x0301记录头兼容值；真实版本看 ClientHello.client_version
length183后续 fragment 字节数

握手体（HandshakeType=1，179 字节）

/* 结构 RFC 5246 §7.4.1.2 */ 01 HandshakeType = client_hello 00 00 b3 length = 179 03 03 client_version = TLS 1.2 ae29658d... client_random (32B) → 参与 master_secret 00 session_id length = 0（新会话） 00 38 cipher_suites length = 56 → 28 个套件 首选 0xC02C, 0xC030, … 含 0x00FF (SCSV) 01 00 compression: null(0) only 00 77 extensions length = 119

扩展字段（收到后服务器如何用）

ExtensionID本抓包内容对端处理
elliptic_curves10x25519, secp256r1, x448, secp521r1, secp384r1选 SKE 曲线时取交集；本例选 x25519
ec_point_formats11uncompressed, 1, 2确定 EC 点编码格式
signature_algorithms13RSA-PSS, ECDSA, ed25519 等 20 种约束 CertificateVerify / SKE 签名算法
extended_master_secret23空协商后 master_secret 绑定完整握手 transcript（RFC 7627）
session_ticket35空客户端愿意接受 RFC 5077 票据（帧 #9 下发）
encrypt_then_mac22空对 CBC 套件有意义；GCM 套件忽略

服务器收到后：校验版本 ≥ 1.2、过滤密码套件、检查扩展合法性；若无匹配套件则 Alert handshake_failure。本抓包进入完整握手（Session ID 为空，非恢复）。

7 · ServerHello（帧 #6 · 记录 1/5）

参数值含义
server_random4dfa2305…4752440132 字节熵；末 4 字节可见 ASCII GRE0 模式（实现自定义）
session_id(empty)未分配恢复 ID
cipher_suite0xC030TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
compression0 (null)必须 null
extensionsrenegotiation_info, ec_point_formats, session_ticket, extended_master_secret确认客户端扩展

客户端收到后：锁定套件与 PRF 哈希（SHA-384）；保存 server_random；状态机进入等待 Certificate / SKE。

8 · Certificate（帧 #6 · 记录 2/5）

握手长度 1637 B：3 字节 cert_list 长度 + 若干 DER X.509 证书（通常叶子 + 中间 CA）。

struct { 00 06 62 certificates length = 1634 00 04 xx cert[0] length + DER ... 00 02 xx cert[1] length + DER ... } Certificate;

客户端收到后：按信任锚验证链、检查 EKU（serverAuth）、匹配主机名（若用 SNI）；失败 → bad_certificate / unknown_ca。本抓包为 RSA/EC 签名的服务器链，供后续验证 SKE 签名与可选客户端认证参考。

9 · ServerKeyExchange（帧 #6 · 记录 3/5）

ECDHE_RSA 套件必须携带 SKE（RFC 4492 §2.4）。本抓包 296 字节：

字段抓包值
curve_type3 (named_curve)
named_curve29 → x25519 (0x001D)
pubkey_len32
server ECDH pubkey31e732ba2b97b80f…09ae33
signature length2052（RSA-2048/4096 + PKCS#1）
/* ServerECDHParams + digitally-signed */ 03 curve_type = named_curve 00 1d named_curve = x25519 20 pubkey len = 32 31e732ba... 服务器临时公钥 Y_s 08 04 SignatureAndHashAlgorithm (示例) ... Signature = RSA-Sign( ClientHello||ServerHello||SKE_params )

客户端收到后：

  1. 确认曲线在 ClientHello 扩展列表内
  2. 用服务器证书公钥验证 SKE 签名（transcript 至当前所有握手消息）
  3. 暂存 Y_s，待 ClientKeyExchange 发送 Y_c 后计算 premaster = X25519(Y_s, x_c)

10 · CertificateRequest + ServerHelloDone（帧 #6 · 记录 4–5）

CertificateRequest（48 B）

字段值
certificate_typesRSA(1), DSS(2), ECDSA(64)
supported_signature_algorithms与 ClientHello ext#13 类似列表
certificate_authorities长度 0（不限制 CA DN）

客户端收到后：若本地有符合类型的客户端证书则后续发送 Certificate + CertificateVerify；否则可发空 Certificate 消息（本抓包选择发送 1631 B 客户端链）。

ServerHelloDone（0 B body）

标志服务器 Hello 阶段结束。客户端收到后必须开始发送客户端认证与密钥交换消息（帧 #8 五条记录）。

11 · 客户端认证（帧 #8）

Certificate（1631 B）

客户端证书链，供服务器验证 mTLS 身份（MQTT 8883 常见配置）。

ClientKeyExchange（33 B）

20 ECPoint length = 32 59e3a7638ec57ad2b31fd10801d19b58f657a314cf9e3db4cd77b1b28640a20b 客户端临时公钥 Y_c

服务器收到后：ECDH 得 premaster；与客户端应算出相同 48 字节 premaster secret（TLS 1.2 仍用 48B 结构，ECDH 输出嵌入其中）。

CertificateVerify（260 B）

结构：SignatureAndHashAlgorithm + Signature。签名对象为从 ClientHello 到 CertificateVerify 之前所有握手消息的连续哈希（算法由套件 PRF 决定，此处 SHA-384）。

服务器收到后：用客户端证书公钥验证签名；失败 → decrypt_error / 握手失败。证明客户端持有私钥。

12 · ChangeCipherSpec 与 Finished（帧 #8–#9）

ChangeCipherSpec（双方各 1 字节）

01 opaque value 必须为 1 · 独立 TLS 子协议 · RFC 5246 §6.2.1

收到后：读/写状态机切换至协商好的 AES_256_GCM 密钥；后续记录使用 AEAD 保护。注意 CCS 不是 Handshake 类型，但顺序上夹在握手消息流中。

Finished（各 40 字节记录，已加密）

明文语义（加密前）：

verify_data = PRF(master_secret, "client finished" | "server finished", Hash(handshake_messages))[0..verify_length-1] /* GCM 套件 verify_length = 12（SHA-384 仍截断 12B）*/

处理流程：

  • 发送方：用新 write 密钥封装 Handshake 记录（内含 Finished）
  • 接收方：解密后校验 verify_data；失败 → decrypt_error
抓包中 Handshake 类型字节不可读（显示为伪类型 198/250），说明 Finished 已用 AES-GCM 加密；需会话密钥或 SSLKEYLOGFILE 才能在 Wireshark 中展开。

13 · NewSessionTicket 与连接关闭（帧 #9、#12）

NewSessionTicket（998 B，帧 #9，仍明文）

在服务器 CCS 之前发送（部分栈顺序如此）。含 ticket_lifetime_hint + ticket opaque，供后续会话恢复（ClientHello session_ticket 扩展）。

客户端收到后：缓存 ticket；下次连接可尝试 PSK/会话恢复 abbreviated handshake。

Alert（帧 #12，26 B 加密记录）

连接空闲约 1.6 s 后客户端 FIN；服务器回复加密 Alert，通常为 close_notify（level=warning, description=0）。

收到后：TLS 状态进入关闭；应用层（如 MQTT）应停止读写并释放会话。

本抓包在 Finished 之后未出现 Application Data（无 MQTT CONNECT）；仅完成 TLS 层协商与拆除，适合作为纯握手教学样本。
