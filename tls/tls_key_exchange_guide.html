TLS 密钥交换算法原理与演化

从 RSA 加密 premaster 到 DH/DHE 有限域 再到 ECDHE 椭圆曲线：说明数学原理、握手交互、数值示例，以及为何业界最终推荐前向保密（PFS）的 ECDHE。

premaster_secret → master_secret PFS = 长期私钥泄露不危及旧会话 TLS 1.3 仅 (EC)DHE 套件名：RSA / DHE / ECDHE

1 · 密钥交换在 TLS 中的共同目标

TLS 记录层使用对称密钥（如 AES-GCM）加密业务数据。对称密钥来自 master_secret，而 master_secret 由双方协商出的 premaster_secret（48 字节，TLS 1.2）与 ClientHello/ServerHello 随机数经 PRF 导出：

master_secret = PRF(premaster_secret, "master secret", client_random + server_random)[0..47] key_block = PRF(master_secret, "key expansion", server_random + client_random)

密钥交换算法的职责：让客户端与服务器在不可信网络上，各自算出相同的 premaster_secret，且使被动窃听者无法从截获的报文恢复该值。不同算法实现这一目标的数学机制不同。

3 · 演化脉络：为何走向 ECDHE

  • RSA 密钥交换：实现简单，但与证书密钥绑定；私钥一旦泄露，历史密文可被解密（无 PFS）。
  • 静态 DH：共享秘密固定，泄露 DH 私钥同样危及所有会话；且缺少类似 DHE 的 SKE 签名绑定，易配置错误。
  • DHE：引入临时密钥实现 PFS，但 2048-bit 模幂运算慢，且依赖安全素数参数（弱参数可导致 Logjam）。
  • ECDHE：更短密钥达到相当安全强度（如 P-256 ≈ 3072-bit RSA），握手更快，成为 TLS 1.2 事实标准；TLS 1.3 强制使用。

5 · RSA 密钥交换

5.1 原理

客户端生成 48 字节的 premaster_secret（前两字节为客户端支持的 TLS 版本，用于降级检测）。用服务器证书中的 RSA 公钥加密后，放入 ClientKeyExchange。仅持有对应私钥的服务器能解密得到 premaster。

密文 c = premaster^e mod n （服务器公钥 (e,n)） premaster = c^d mod n （服务器私钥 d）
RSA 密钥交换 — 数学关系（示意）

5.2 数值示例（极小素数，仅教学）

设教学用 RSA：p=61, q=53, n=3233, φ=3120, e=17, d=2753。premaster 简化为整数 m=1234。

c = 1234^17 mod 3233 = 2423 m' = 2423^2753 mod 3233 = 1234 ✓

真实 TLS 使用 2048+ 位模数；premaster 为 48 字节随机数，而非小整数。

5.3 TLS 1.2 握手交互（RSA 密钥交换）

优点

  • 握手消息少，无 ServerKeyExchange
  • 实现与理解简单
  • 早期硬件加速成熟

缺点

  • 无 PFS：私钥泄露危及历史流量
  • 与证书密钥强绑定，无法独立轮换加密能力
  • PKCS#1 v1.5 _padding 历史攻击面（Bleichenbacher → ROBOT）
  • TLS 1.3 已删除；RFC 7568 建议禁用

6 · DH（静态 Diffie-Hellman）

6.1 原理

经典 Diffie-Hellman：公开大素数 p、生成元 g。双方各自有长期私钥 a、b，公钥 A=g^a mod p、B=g^b mod p。共享秘密：

K = B^a mod p = A^b mod p = g^(ab) mod p

静态 DH：DH 公钥写在 X.509 证书里（密钥用法 keyAgreement），握手时不再发送临时公钥。premaster 由 K 派生。

6.2 教学数值示例

设 p=23, g=5。Alice 私钥 a=6 → A=5^6 mod 23=8。Bob 私钥 b=15 → B=5^15 mod 23=19。

K = 19^6 mod 23 = 8^15 mod 23 = 2

6.3 TLS 握手（概念性，静态 DH 罕见）

优点

  • 概念简洁，无需 RSA 加密 premaster
  • 证书同时承载身份与 DH 公钥

缺点

  • 无 PFS（长期 DH 私钥）
  • 证书与密钥交换耦合，部署不灵活
  • 参数 p,g 必须安全且足够大
  • 实践中几乎被 DHE/ECDHE 取代

7 · DHE（Ephemeral Diffie-Hellman）

7.1 原理

每建立一次 TLS 会话，服务器与客户端各自生成临时 DH 私钥 a_s, a_c（握手完成后丢弃）。服务器在 ServerKeyExchange 中发送 p, g, B，并用证书私钥对握手 transcript 签名，防止中间人替换参数或公钥。客户端回送 A。

K = B^{a_c} mod p = A^{a_s} mod p premaster = 按 RFC 5246 将 K 编码为 48 字节结构
Logjam (2015)： 若服务器使用弱/预计算 512-bit 素数，攻击者可降级并批量预计算，破坏 DHE 安全性。应使用 ≥2048-bit 安全素数或改用 ECDHE。

7.2 TLS 1.2 握手交互（DHE_RSA）

优点

  • 前向保密：会话私钥销毁后，历史密文不可解密
  • 证书仅用于认证/签名，可与加密能力解耦
  • 不依赖 RSA PKCS#1 加密 premaster

缺点

  • 模幂运算比 ECDHE 慢（尤其 4096-bit）
  • SKE 体积大（p,g,B + 签名）
  • 依赖高质量 DH 参数，配置错误风险高
  • 移动/高并发场景 CPU 压力大

8 · ECDHE（Ephemeral Elliptic Curve DH）

8.1 原理

在椭圆曲线群上，私钥为整数 d，公钥为点 Q = d·G（G 为基点）。共享点为：

P = d_c · Q_s = d_s · Q_c = d_s · d_c · G premaster = P 的 x 坐标（及 RFC 4492 / 8446 规定填充）

相比有限域 DHE，相同安全强度下密钥更短（如 P-256 约等价 3072-bit RSA），计算更快，握手更轻。

8.2 与 X25519

X25519 是 Curve25519 上的 ECDH 函数，常作为 named_curve（如 TLS 扩展中 curve_id=29）。本仓库抓包 tls12_decode 即使用 x25519 临时密钥 + RSA 证书签名 SKE（套件 TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384）。

8.3 TLS 1.2 握手交互（ECDHE_RSA）

8.4 TLS 1.3 变化

TLS 1.3（RFC 8446）仅保留 (EC)DHE：密钥协商通过 key_share 扩展完成，握手后半段加密，无独立 ServerKeyExchange 消息；共享秘密经 HKDF 派生，不再使用 TLS 1.2 式 premaster 48 字节结构。

优点

  • PFS + 更短密钥、更快速度
  • 曲线标准化（P-256、X25519、P-384）
  • TLS 1.2/1.3 主流默认
  • 与 RSA 签名证书组合灵活（ECDHE_RSA / ECDHE_ECDSA）

缺点

  • 实现需防侧信道、无效曲线点攻击
  • 错误曲线/参数实现可导致完全失效
  • 仍需可信证书链与 SKE/签名验证（TLS 1.2）
  • 量子计算长期威胁（需 PQC 迁移）
