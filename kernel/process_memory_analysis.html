Linux 进程物理内存精确排查

如何准确测量某个进程占用的物理内存及各部分的构成 —— 基于 Linux 内核官方 proc 文档与 man-pages

📎 Linux Kernel Documentation: /proc Filesystem 📎 man7.org: proc_pid_smaps(5) 📎 man7.org: proc_pid_status(5)

📖背景与第一性原理

在 Linux 中，进程使用虚拟内存（Virtual Memory）。内核通过页表（Page Table）将虚拟地址映射到物理页框。 物理内存（RAM）中实际驻留的那部分内存才是我们真正关心的"占用"量。

"一个进程占多少物理内存"并没有唯一答案，原因是：多个进程可以共享同一份物理页（如共享库、共享内存段）。 因此，内核提供了多种视角的统计指标，从不同维度反映内存使用情况。

虚拟地址空间 → 物理内存 映射关系示意

📌 第一性原理小结

  • 虚拟内存 ≠ 物理内存：进程的地址空间远大于其实际占用的物理 RAM。
  • 共享页是核心难题：同一物理页可被多个进程映射，如何"分摊"决定了 RSS/PSS/USS 的差异。
  • 唯一精确来源：/proc/<pid>/smaps 逐段记录每个内存映射区域 (VMA) 的实际居留量，是最权威的数据源。
  • 内核文档警告：/proc/<pid>/status 中的 RSS 值因内核异步更新，可能不够精确；要获取精确快照，需扫描 smaps 文件（速度较慢但精准）。

📊核心指标总览

以下五个指标是 Linux 进程内存分析的核心，理解它们的区别是排查的基础：

VSZ 虚拟内存大小
进程可访问的全部虚拟地址空间大小，包含已分配但未使用的区域、已交换出去的页、共享库全部映射大小。严重高估实际内存用量，通常没有实际排查意义。
≈ VmSize in /proc/pid/status
RSS 驻留集大小
进程当前驻留在物理 RAM 中的页面总量，包含共享库中当前已加载的所有页。会重复计数共享页，多进程共享同一库时 RSS 被高估。
= RssAnon + RssFile + RssShmem
PSS 比例分摊大小
将共享页按共享进程数均摊后的内存量。私有页 100% 计入，共享页按 1/N 计入（N 为共享该页的进程数）。最能反映单进程实际内存代价，所有进程 PSS 之和等于系统总内存用量。
Σ (private_pages + shared_pages / N)
USS 独占集大小
进程完全私有的内存页，即只有该进程映射的页面（Private_Clean + Private_Dirty）。若该进程退出，USS 就是系统实际回收量，也是判断内存泄漏的最佳指标。
= Σ (Private_Clean + Private_Dirty)
Swap 交换空间
进程的匿名私有内存被换出到 swap 分区的量。不占用物理 RAM，但属于进程"逻辑内存"的一部分。SwapPss 是按比例分摊的 swap 大小。
VmSwap in /proc/pid/status
VSZ / RSS / PSS / USS 大小关系
USS ≤ PSS ≤ RSS ≤ VSZ

📌 本章小结

  • 不要用 VSZ 判断实际内存占用，它包含大量未使用的虚拟地址空间。
  • RSS 会高估：若进程链接了 libssl（10MB），而系统中还有 99 个进程也使用它，每个进程的 RSS 都包含这 10MB，但物理上只有一份。
  • PSS 最均衡：适合统计整体内存压力；USS 最保守：适合定位内存泄漏（退出后真实释放量）。
  • 所有精确数据来自 /proc/<pid>/smaps，RSS 快速估算来自 /proc/<pid>/status。

📄/proc/<pid>/status

内核接口 since Linux 2.x 最快速的内存信息查看入口，以人类可读格式提供进程内存统计。 内核文档说明：RSS 相关信息以异步方式处理，数值可能不够精确，若需精确快照应扫描 smaps 文件。

SHELL查看进程内存状态
# 替换 <PID> 为实际进程ID，例如 1234 cat /proc/<PID>/status # 只过滤内存相关字段 grep -E '^(Vm|Rss|Huge)' /proc/<PID>/status # 示例输出（内核文档 proc.txt 原文） VmPeak: 5004 kB # 虚拟内存峰值 VmSize: 5004 kB # 当前虚拟内存大小 (VSZ) VmLck: 0 kB # 锁定在内存中的页（mlock） VmPin: 0 kB # 固定页（since Linux 3.2） VmHWM: 476 kB # RSS 历史峰值（high water mark） VmRSS: 476 kB # 当前驻留物理内存 (RSS) RssAnon: 352 kB # 匿名内存（堆、栈、私有mmap） RssFile: 120 kB # 文件映射内存（代码段、共享库） RssShmem: 4 kB # 共享内存（SysV shm, tmpfs, 匿名共享映射） VmData: 156 kB # 私有数据段大小 VmStk: 88 kB # 栈段大小 VmExe: 68 kB # 代码段（text）大小 VmLib: 1412 kB # 共享库代码大小 VmPTE: 20 kB # 页表项大小 VmSwap: 0 kB # 匿名私有数据被换出的大小 HugetlbPages: 0 kB # 大页内存大小

VmRSS 三组分详解

自 Linux 4.5 起，内核将 VmRSS 分解为三个子字段（由 Jerome Marchand 提交的 内核补丁引入）：

字段含义典型来源内核版本
RssAnon 居住于 RAM 的匿名内存页 malloc 分配的堆、栈、私有 mmap(MAP_ANONYMOUS) since 4.5
RssFile 居住于 RAM 的文件映射页 可执行文件代码段、共享库 .so 文件 since 4.5
RssShmem 居住于 RAM 的共享内存页 SysV shm、tmpfs 映射、匿名共享映射 since 4.5
VmRSS = RssAnon + RssFile + RssShmem
VmRSS 三组分示意（以 bash 进程为例）
⚠️ 精度说明（内核文档原文）
对于 SMP（多处理器）配置，RSS 相关信息以异步方式处理，因此数值可能不够精确。 若需获取某一时刻的精确快照，应读取 /proc/<pid>/smaps 文件并扫描页表（速度较慢但非常精确）。
来源: kernel.org Documentation/filesystems/proc.txt

📌 本章小结

  • /proc/<pid>/status 是快速估算的首选，命令简单，字段人类可读。
  • VmRSS 是最常用的物理内存指标，但含重复计数的共享页；其三组分（RssAnon/RssFile/RssShmem）可帮助定位内存类型。
  • 该文件的 RSS 值由内核异步更新，不保证实时精确，生产环境精确排查应配合 smaps。

📄/proc/<pid>/smaps

内核接口 since Linux 2.6.14 CONFIG_PROC_PAGE_MONITOR smaps 是内存分析的最权威精确数据源，为进程的每个虚拟内存区域（VMA / 内存映射段）单独列出物理内存使用情况。

根据 man7.org: proc_pid_smaps(5) 和 kernel.org: proc.rst： "此文件展示进程每个内存映射的内存消耗情况"。

SHELL查看 smaps（以 /bin/bash 为例）
cat /proc/<PID>/smaps # 典型输出（来自 Linux man-pages 6.18 官方示例）： 00400000-0048a000 r-xp 00000000 fd:03 960637 /bin/bash Size: 552 kB # 该VMA虚拟地址空间大小 Rss: 460 kB # 当前驻留物理RAM的量 Pss: 100 kB # 比例分摊后的物理内存量 Shared_Clean: 452 kB # 共享+未修改（通常是共享库代码） Shared_Dirty: 0 kB # 共享+已修改（被多进程共用的写入页） Private_Clean: 8 kB # 私有+未修改 Private_Dirty: 0 kB # 私有+已修改（CoW后的私有副本） Referenced: 460 kB # 被标记为已引用/访问的量 Anonymous: 0 kB # 不属于任何文件的匿名内存 AnonHugePages: 0 kB # 透明大页（THP）支持的内存 ShmemHugePages: 0 kB # 大页支持的共享内存 ShmemPmdMapped: 0 kB # PMD级映射的共享大页 Swap: 0 kB # 该VMA被换出到swap的匿名内存量 SwapPss: 0 kB # 按比例分摊的swap使用量 KernelPageSize: 4 kB # 内核backing此VMA时使用的页大小 MMUPageSize: 4 kB # MMU使用的页大小 Locked: 0 kB # 是否被mlock锁定在内存中 THPeligible: 0 # 是否符合THP分配条件(since 5.4) VmFlags: rd ex mr mw me dw # 内核VMA标志位（since Linux 3.8）

核心字段详解（来自 kernel.org proc.rst 原文）

字段含义（基于官方文档）用于计算
Size 该内存映射段的虚拟地址范围大小（不代表物理占用） VSZ 的组成部分
Rss 该映射当前驻留在 RAM 中的量，包含共享页全额 ∑Rss ≈ 进程 RSS
Pss 进程对此映射的比例分摊量。私有页全算，共享页除以共享进程数 N。官方文档示例：1000个私有页 + 1000个与另1人共享 → PSS = 1500页 ∑Pss = 进程 PSS
Shared_Clean 被多个进程共享且未被修改的页（典型：共享库代码段，只读映射） 共享部分 = Shared_Clean + Shared_Dirty
Shared_Dirty 被多个进程共享且已被修改的页
Private_Clean 仅本进程使用且未被修改的私有页 USS = ∑(Private_Clean + Private_Dirty)
Private_Dirty 仅本进程使用且已修改的私有页（CoW 产生的副本、堆分配等）
Referenced 当前被内核标记为已引用/已访问的内存量（用于页面回收判断） -
Anonymous 不属于任何文件的匿名内存。即使是 MAP_PRIVATE 的文件映射，被修改后产生的 CoW 副本也在此计入 -
LazyFree 被 madvise(MADV_FREE) 标记的内存，在内存压力时才真正释放（如页是干净的） -
AnonHugePages 由透明大页（THP）支持的匿名内存量 -
Swap 本 VMA 中被换出到 swap 的匿名内存量。对 shmem 映射，也包含底层 shmem 对象换出量 ∑Swap = 进程总换出量
SwapPss 按比例分摊的 swap 使用量（不含 shmem 底层对象，与 Swap 的统计范围有差别） -
Locked 该映射是否被 mlock 锁定在物理内存中（不可被换出） -
VmFlags 内核内部 VMA 标志位的两字母缩写（since Linux 3.8），如 rd=可读、wr=可写、ex=可执行、sh=共享等 -

关键关系式

RSS 的构成：

RSS = Shared_Clean + Shared_Dirty + Private_Clean + Private_Dirty

PSS 的计算：

PSS = Private_pages + Σ(Shared_page_i / N_i)

USS 的计算（非内核直接输出，需手工汇总）：

USS = Σ(Private_Clean + Private_Dirty)  across all VMAs

Clean vs Dirty / Private vs Shared

Clean 页（干净页）
自映射以来未被修改的页面。典型例子：共享库的代码段（只从磁盘读取，从不写入），内核可随时丢弃并在需要时重新从文件加载。
Dirty 页（脏页）
已被进程修改的页面（相对于磁盘上的原始内容）。不能被简单丢弃，必须写回文件或 swap 才能释放，内存压力下代价更高。
Private 页（私有页）
只有当前进程映射到的页，其他进程无法访问。注意：一个"可共享"的页在实际被共享之前，内核仍将其计为私有页。
Shared 页（共享页）
被多个进程同时映射的页面，如共享库代码。RSS 对每个进程全额计入，PSS 则按进程数均摊。
SHELL用 awk 从 smaps 汇总各项指标
# 汇总进程总 RSS（单位 kB） awk '/^Rss:/{sum+=$2} END{print "RSS:", sum, "kB"}' /proc/<PID>/smaps # 汇总进程总 PSS（比例分摊，最准确的单进程内存占用） awk '/^Pss:/{sum+=$2} END{print "PSS:", sum, "kB"}' /proc/<PID>/smaps # 汇总 USS（独占私有内存，等于退出后真实释放量） awk '/^Private_(Clean|Dirty):/{sum+=$2} END{print "USS:", sum, "kB"}' /proc/<PID>/smaps # 汇总 Swap awk '/^Swap:/{sum+=$2} END{print "Swap:", sum, "kB"}' /proc/<PID>/smaps # 一次性输出所有关键指标 awk ' /^Rss:/ { rss+=$2 } /^Pss:/ { pss+=$2 } /^Private_(Clean|Dirty):/ { uss+=$2 } /^Swap:/ { swap+=$2 } END { printf "RSS: %8d kB\n", rss printf "PSS: %8d kB\n", pss printf "USS: %8d kB\n", uss printf "Swap: %8d kB\n", swap } ' /proc/<PID>/smaps

📌 本章小结

  • /proc/<pid>/smaps 是最精确的内存数据源，每个 VMA（虚拟内存区域）单独列出，可精确计算 RSS、PSS、USS。
  • Clean 共享页（如共享库代码）多进程复用，RSS 被高估；Private Dirty 页是进程独占并修改的，是内存泄漏排查的焦点。
  • 该文件需要 CONFIG_PROC_PAGE_MONITOR 内核配置选项开启（现代发行版默认开启）。
  • 读取 smaps 速度较慢（需扫描所有页表项），生产环境可用 smaps_rollup 代替。

📄/proc/<pid>/smaps_rollup

内核接口 since Linux 4.14 (2017年8月) 由 Daniel Colascione 贡献，文档位于 Documentation/ABI/testing/procfs-smaps_rollup。

smaps_rollup 提供的是整个进程所有 VMA 的汇总值，等价于手工 awk 累加 smaps 中每一行，但速度快得多。 格式与 smaps 几乎相同，只是只有一条记录（标注为 [rollup]），并额外提供三个 smaps 中没有的分组字段。

SHELLsmaps_rollup 典型输出（来自内核 ABI 文档原文）
cat /proc/<PID>/smaps_rollup 00100000-ff709000 ---p 00000000 00:00 0 [rollup] Size: 1192 kB KernelPageSize: 4 kB MMUPageSize: 4 kB Rss: 884 kB Pss: 385 kB Pss_Dirty: 68 kB # PSS 中脏页部分 Pss_Anon: 301 kB # ★ 匿名内存的比例分摊 (smaps中无此字段) Pss_File: 80 kB # ★ 文件映射的比例分摊 (smaps中无此字段) Pss_Shmem: 4 kB # ★ 共享内存的比例分摊 (smaps中无此字段) Shared_Clean: 696 kB Shared_Dirty: 0 kB Private_Clean: 120 kB Private_Dirty: 68 kB Referenced: 884 kB Anonymous: 68 kB LazyFree: 0 kB AnonHugePages: 0 kB ShmemPmdMapped: 0 kB Shared_Hugetlb: 0 kB Private_Hugetlb: 0 kB Swap: 0 kB SwapPss: 0 kB Locked: 385 kB
💡 smaps_rollup 独有的三个字段
内核 ABI 文档说明：Pss_Anon、Pss_File、Pss_Shmem 在 smaps 中不存在（因为 smaps 中每段 VMA 的内存类型可以直接从映射信息判断）， 但在 smaps_rollup 的汇总条目中提供这三个分类汇总，便于快速了解 PSS 的构成。
来源: Documentation/ABI/testing/procfs-smaps_rollup

smaps vs smaps_rollup 对比

特性/proc/pid/smaps/proc/pid/smaps_rollup
数据粒度每个 VMA 单独一条记录全进程汇总，单条记录
速度慢（VMA 多时尤其慢）快（内核直接汇总，无需用户空间 awk）
Pss_Anon/File/Shmem❌ 无✅ 有
定位具体映射段✅ 可以❌ 不能

SHELL快速获取进程内存摘要（推荐生产环境使用）
# 一行命令获取 RSS/PSS/USS/Swap（优先用 smaps_rollup，回退到 smaps） PID=<PID> FILE=/proc/$PID/smaps_rollup [ -f "$FILE" ] || FILE=/proc/$PID/smaps awk '/^(Rss|Pss|Private_Clean|Private_Dirty|Swap):/{ if($1=="Rss:") rss+=$2 if($1=="Pss:") pss+=$2 if($1~/Private/) uss+=$2 if($1=="Swap:") swap+=$2 } END { printf "RSS: %6d kB\nPSS: %6d kB\nUSS: %6d kB\nSwap: %6d kB\n", rss,pss,uss,swap }' "$FILE"

📌 本章小结

  • smaps_rollup 是 smaps 的高性能汇总版本，适合监控脚本和自动化工具（如 psutil 就从 smaps 改用 smaps_rollup 以提升性能）。
  • 独有的 Pss_Anon / Pss_File / Pss_Shmem 字段可直接分辨 PSS 中匿名内存与文件映射各占多少比例。
  • 若需了解哪个具体库或堆段占用高，仍需使用完整 smaps 文件。

📄/proc/<pid>/statm

内核接口 机器可读格式的内存统计，以页面数为单位（而非 kB），适合程序解析。 内容等同于 status 文件中的 VmSize 和 VmRSS，但格式更紧凑。

SHELL/proc/pid/statm 格式说明（来自 kernel.org proc.txt）
cat /proc/<PID>/statm 49829626 5213229 1212275 1 0 5773980 0 # 七列含义： # 1. size - 虚拟内存总页数 (= VmSize in status) # 2. resident- 驻留物理内存页数 (= VmRSS in status) # 3. shared - 共享页数 (= RssFile+RssShmem in status) # 4. trs - 代码页数（已废弃，包含数据段，不准确） # 5. lrs - 库页数（2.6后始终为0） # 6. drs - 数据+栈页数（包含库，不准确） # 7. dt - 脏页数（2.6后始终为0） # 转换页数为 kB（Linux 默认页大小 4096 字节） pagesize=$(getconf PAGESIZE) resident=$(awk '{print $2}' /proc/<PID>/statm) echo "RSS: $((resident * pagesize / 1024)) kB"
⚠️ 注意事项
根据内核文档（proc.txt Table 1-3），statm 的第 4 列（trs，代码页数）和第 6 列（drs，数据+栈页数）的统计范围已知存在缺陷（broken）， 不能准确反映各段实际大小。第 5 列（lrs，库页数）在 Linux 2.6+ 上始终为 0。 建议仅使用前 3 列（size、resident、shared）。

📌 本章小结

  • statm 是最轻量的内存查询接口，适合程序解析，但精度低于 smaps。
  • 仅第 1-3 列（虚拟大小、驻留量、共享量）可靠；其余列在现代内核中已废弃或数据不准确。
  • 单位是页面数，需乘以页大小（通常 4096 字节）转换为字节数。

🔧用户空间工具：ps / top

pstophtop ps 和 top 从 /proc/<pid>/stat 和 /proc/<pid>/status 读取数据， 只能看到 VSZ（虚拟内存）和 RSS，不能直接显示 PSS 和 USS。

SHELLps / top 内存相关命令
# ps 查看 VSZ 和 RSS（单位 kB） ps -p <PID> -o pid,comm,vsz,rss PID COMMAND VSZ RSS 1234 nginx 10240 4096 # 以人类可读格式显示 ps -p <PID> -o pid,comm,vsz=VSZ_kB,rss=RSS_kB # top 交互模式：按 M 按内存排序，RES 列即 RSS # VIRT = VSZ（虚拟），RES = RSS（驻留物理），SHR = 共享内存 top -p <PID> # top 批处理模式输出一次结果 top -b -n 1 -p <PID> # htop（如已安装）支持更丰富的内存列显示 htop -p <PID>
工具列名对应指标说明
ps/topVSZ / VIRTVSZ虚拟内存大小，极度高估
ps/topRSS / RESRSS驻留物理内存，含共享页全额
topSHRRssFile+RssShmem共享内存量
ps/top-PSS/USS❌ 不支持，需用 smem 或手工 awk

📌 本章小结

  • ps / top 是快速概览工具，适合初步判断进程内存量级，不适合精确排查。
  • RSS 可能因共享库被高估；若多个进程使用同一个大型库，每个进程的 RSS 都会包含该库的全部内存。
  • 精确排查需要 smaps 或 smem 工具。

🔧用户空间工具：pmap

pmap pmap 命令以更易于人类阅读的格式展示 /proc/<pid>/smaps 中的内容， 可逐段查看每个内存映射的 RSS。根据 man-pages 官方文档，pmap 显示的信息与 smaps 来源相同。

SHELLpmap 命令用法

pmap 输出字段说明

列含义

📌 本章小结

  • pmap -x 是查看进程各内存段 RSS 分布的便捷工具，适合定位哪个共享库或堆段占用高。
  • pmap 数据直接来源于 smaps，无需自行解析 smaps 文件。
  • pmap -XX 可展示更多 smaps 字段（PSS、Swap 等），但格式较宽，终端可能需要横向滚动。

🔧用户空间工具：smem

smem smem 由 Matt Mackall 开发，是专门用于展示 PSS 和 USS 的工具，直接读取 /proc/<pid>/smaps。 它是第一个将 PSS/USS 以友好界面展示的工具，为 Linux 内存分析领域做出了重要贡献。

SHELLsmem 命令用法
# 安装（各发行版） apt install smem # Debian/Ubuntu yum install smem # RHEL/CentOS # 查看指定进程的 VSS/RSS/PSS/USS smem -P <进程名> smem -p <PID> PID User Command Swap USS PSS RSS 1234 root nginx: master process 0 1024 1280 5120 # 显示系统所有进程的 PSS/USS（按 USS 降序排列） smem -s uss -r # 显示共享库各自的 PSS 占用 smem -m # 生成饼图（需要 matplotlib） smem --pie name -s pss
smem 列含义说明
VSS虚拟内存大小= VSZ，最大，最不准
RSS驻留集大小含共享页全额，偏高
PSS比例分摊大小最能反映单进程内存代价
USS独占集大小进程退出后真实释放量，排查泄漏首选

📌 本章小结

  • smem 是 Linux 上直接展示 PSS 和 USS 的最方便工具，无需手工 awk。
  • USS（独占集）= 进程退出后真实释放的内存，是排查内存泄漏的最佳指标。
  • PSS 汇总 = 系统所有进程 PSS 之和 ≈ 系统实际物理内存总占用量。

🔧awk 手工汇总（无需额外工具）

在没有 smem 等工具的环境中，可直接用 awk/grep 从 smaps 或 smaps_rollup 提取各项指标：

SHELL完整内存排查脚本
#!/bin/bash # 用法: ./memcheck.sh <PID> PID=${1:?需要提供PID} # 1. 快速概览（来自 status，可能略不精确） echo "=== 快速概览 (/proc/$PID/status) ===" grep -E '^(Name|Vm(RSS|Peak|Size|HWM|Swap)|Rss(Anon|File|Shmem))' /proc/$PID/status # 2. 精确汇总（来自 smaps_rollup 或 smaps） echo "" echo "=== 精确汇总 (smaps_rollup/smaps) ===" SRC="/proc/$PID/smaps_rollup" [ -f "$SRC" ] || SRC="/proc/$PID/smaps" awk ' /^Rss:/ { rss+=$2 } /^Pss:/ { pss+=$2 } /^Pss_Anon:/ { pss_anon+=$2 } /^Pss_File:/ { pss_file+=$2 } /^Pss_Shmem:/ { pss_shmem+=$2 } /^Private_(Clean|Dirty):/ { uss+=$2 } /^Shared_(Clean|Dirty):/ { shared+=$2 } /^Swap:/ { swap+=$2 } /^Anonymous:/ { anon+=$2 } END { printf "RSS: %8d kB (驻留物理内存，含共享页全额)\n", rss printf "PSS: %8d kB (比例分摊，最准确单进程占用)\n", pss printf "USS: %8d kB (独占私有，进程退出真实释放量)\n", uss printf "Shared: %8d kB (共享部分总量)\n", shared printf "Swap: %8d kB (换出量)\n", swap if(pss_anon+pss_file+pss_shmem > 0) { printf " PSS_Anon: %6d kB (匿名内存比例分摊)\n", pss_anon printf " PSS_File: %6d kB (文件映射比例分摊)\n", pss_file printf " PSS_Shmem:%6d kB (共享内存比例分摊)\n", pss_shmem } } ' "$SRC" # 3. 查找 RSS 最大的前10个内存段 echo "" echo "=== RSS 最大的内存段 TOP 10 (/proc/$PID/smaps) ===" awk '/^[0-9a-f]/{vma=$0} /^Rss:/{print $2, vma}' /proc/$PID/smaps \ | sort -rn | head -10 \ | awk '{printf "%8d kB %s\n", $1, substr($0, index($0,$2))}'

📌 本章小结

  • 不依赖任何外部工具，只用系统自带的 awk/grep/bash 即可从 smaps 提取全部关键指标。
  • 优先读取 smaps_rollup（速度快）；若内核版本较旧（<4.14）则回退到 smaps。
  • "找出 RSS 最大的内存段"功能可快速定位哪个共享库或堆分配段是内存大户。

📌 本章小结 —— 第一性原理总结

  • 物理内存排查的根本问题是："共享页如何归属？" 不同指标给出了不同的答案：RSS（全归自己）→ PSS（按比例分摊）→ USS（只算私有）。
  • 没有完美的唯一答案：根据排查目的选择合适指标。泄漏排查用 USS，系统压力分析用 PSS，粗略概览用 RSS。
  • 所有数据最终来源于内核维护的页表（page table）和 VMA 链表，smaps 是对这些内核数据结构的直接暴露。

🗺️完整排查流程

以下是一个系统化的进程内存排查流程，从快到精逐步深入：

  1. 第一步：用 ps/top 获取初始概览

    ps aux | grep <进程名> # 查看 VSZ（第5列）和 RSS（第6列），单位 kB # 若 RSS 异常大（如超过可用内存的 10%），进入下一步精查

  2. 第二步：用 /proc/pid/status 查看 RSS 三组分

    grep -E '^(VmRSS|RssAnon|RssFile|RssShmem|VmSwap|VmHWM)' /proc/<PID>/status # 判断内存类型：RssAnon 大 → 堆泄漏？RssShmem 大 → 共享内存问题？

  3. 第三步：用 smaps_rollup 获取精确 PSS/USS 汇总

    awk '/^(Rss|Pss|Private_|Swap):/{print}' /proc/<PID>/smaps_rollup # PSS = 真实物理内存代价；USS = 私有独占内存量

  4. 第四步：用 pmap 或完整 smaps 定位具体内存段

    pmap -x <PID> | sort -k3 -rn | head -20 # 找出 RSS 最大的前20个内存段 # 若 [anon] 段持续增长 → 堆内存泄漏 # 若共享库段很大但只有1个进程用 → 考虑共享

  5. 第五步（可选）：监控 USS 随时间的变化趋势

    # 每5秒采样一次 USS，观察是否持续增长 while true; do USS=$(awk '/^Private_/{s+=$2} END{print s}' /proc/<PID>/smaps) echo "$(date +%H:%M:%S) USS: $USS kB" sleep 5 done # USS 持续上涨且不回落 → 内存泄漏的强烈信号
排查决策流程图

📌 排查总结 —— 一图记忆

  • 初步判断：ps aux 或 top → 看 RSS 量级
  • 分类定位：/proc/pid/status → RssAnon / RssFile / RssShmem
  • 精确测量：/proc/pid/smaps_rollup → PSS（公平代价）、USS（独占量）
  • 段级分析：pmap -x 或 /proc/pid/smaps → 找出最大内存段
  • 泄漏监控：周期性采样 USS，持续增长则为泄漏信号
