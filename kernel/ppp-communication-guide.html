01

PPP协议概述

📄 官方定义（RFC 1661，1994年7月）

"The Point-to-Point Protocol (PPP) provides a standard method for transporting multi-protocol datagrams over point-to-point links."
——《点对点协议（PPP）》，Simpson，IETF，1994

PPP（点对点协议，Point-to-Point Protocol）是一种数据链路层协议，由 IETF 于 1994 年正式发布为 RFC 1661（互联网标准 STD 51）。它专为两个对等端之间的直连链路传输多协议数据报而设计，支持全双工双向同时通信。

PPP 的三大核心组件

根据 RFC 1661 原文，PPP 由以下三个主要组件构成：

图1-1：PPP 三大核心组件（依据 RFC 1661）

设计目标

PPP 被设计为运行在简单链路上——两个对等端之间传输数据包的点对点链路。这些链路须满足：

设计要求说明
全双工同时支持双向数据传输
有序投递假定底层链路按序交付数据包
多协议同时承载 IP、IPX、NetBEUI 等多种网络层协议
自动协商无需人工干预，通过 LCP 自动协商链路参数
动态 IP 分配通过 IPCP 在连接时协商 IP 地址
⚛ 第一性原理解读

通信的本质是：两个节点如何就"说什么语言、用什么方式通话、谁来开口"达成共识。PPP 的出发点正是此：它不假设底层物理媒介（串口、电话线、光纤均可），只要求"点到点"的直连；它不固定网络层协议（IP、IPX 均支持）。三个组件分别解决了三个最基本的问题：
①"如何打包数据"（封装）→ ②"如何建立共识"（LCP 协商）→ ③"如何映射到具体网络协议"（NCP）。

💡 字节转义（Byte Stuffing）

由于 0x7E 用作帧起始/结束标志，数据中若出现 0x7E 或 0x7D，需用控制转义字符 0x7D 前缀并将原字节 XOR 0x20。例如：0x7E → 0x7D 0x5E，0x7D → 0x7D 0x5D。这是 PPPoS 在串口上运行时的关键机制。

⚛ 第一性原理解读

帧结构解决的是最本质的问题：接收端如何在连续的比特流中分辨"一个包的开始和结束"。0x7E 标志字节如同信封的封口，告诉接收端"从这里到那里"是一个完整的数据单元。Protocol 字段解决了"收到的是什么类型的数据"——这等同于快递单上的货物类型说明，使得一条物理链路可以同时承载多种"货物"（协议）。

03

PPP 连接建立阶段

根据 RFC 1661 第 3.2 节，PPP 链路建立经历明确定义的五个阶段（Phase）：

图3-1：PPP 五阶段状态机（RFC 1661 原文 Phase Diagram）
1
Dead 阶段 — 物理层未就绪
链路从此阶段开始和结束。当物理层就绪时（如检测到载波、串口打开），触发 UP 事件进入建立阶段。LCP 自动机处于 Initial 或 Starting 状态。
2
Establish 阶段 — LCP 协商
双方互发 LCP Configure-Request 报文，协商 MRU、认证方式、Magic-Number 等链路参数。收到 Configure-Ack 后 LCP 进入 Opened 状态。

协商报文类型：Configure-Request(1) → Configure-Ack(2) / Configure-Nak(3) / Configure-Reject(4)
3
Authenticate 阶段 — 可选身份认证
若 LCP 协商时指定了认证协议，则进行认证。支持两种方式：
• PAP（0xC023）：明文两次握手，用户名/密码明文传输
• CHAP（0xC223）：MD5 三次握手挑战响应，更安全
4
Network 阶段 — NCP 协商（IPCP）
发送 IPCP Configure-Request（0x8021），协商 IP 地址、DNS 服务器地址等。PPP 端通常从对端请求一个 IP 地址，对端分配后发送 Configure-Ack。链路此后可传输 IP 数据报。
5
Terminate 阶段 — 链路终止
任意一端发送 LCP Terminate-Request，对端回 Terminate-Ack 后链路关闭，回到 Dead 阶段。也可由不活动定时器超时或外部事件触发。
⚛ 第一性原理解读

PPP 的五阶段本质上回答了通信建立的最基本问题顺序：物理通路是否通畅？→ 双方参数能否对齐？→ 你是谁？→ 我给你什么地址？→ 通话结束如何挂断？

这与打电话极为相似：先拨通（Dead→Establish），再验证身份（Authenticate），再分配资源（Network），再挂断（Terminate）。这种分层的状态机设计使 PPP 可以在任何物理链路上运行。

04

MCU 与 4G 模组通信架构

在嵌入式物联网系统中，4G 模组作为 MCU 的外设接入。MCU 通过 UART 串口发送 AT 指令控制模组，并在数据模式下通过 PPPoS（PPP over Serial） 建立网络连接。

图4-1：MCU 通过 4G 模组连接运营商核心网的完整架构（PPPoS + LTE EPC）

关键链路说明

链路段协议/接口说明
MCU ↔ 4G模组UART + PPPoSMCU 的 TCP/IP 栈通过 PPP 帧与模组通信；AT*99#指令触发数据模式
4G模组 ↔ 基站LTE Uu 空口射频层，RRC/NAS 层负责信令，PDCP/RLC 层负责数据
基站 ↔ 核心网GTP-U/S1-U基站通过 S1-U 接口将用户面数据发往 SGW
SGW ↔ PGWGTP-U/S5S5 接口传输 GTP 隧道数据，PGW 分配 UE IP 地址
PGW ↔ InternetSGi 接口用户 IP 流量经此接口路由至公网或专用数据网
🔑 PPP 的"两个端点"在哪里？

在 MCU+4G 模组场景下，PPP 的本地端是 MCU（运行 PPPoS 客户端，如 lwIP/pppd），PPP 的远端是 4G 模组内部的操作系统（内嵌 PPP 服务端，负责 LCP/IPCP 协商并代理连接核心网）。模组获取的 IP 地址实际上来自核心网 PGW/GGSN，再通过 IPCP 下发给 MCU。

⚛ 第一性原理解读

MCU 本身没有无线射频能力，4G 模组提供的是"射频链路抽象"。PPP 在此场景下的本质作用是：将串口字节流这条"哑管道"，通过协议协商，抽象成一条虚拟的网络接口（ppp0），使 MCU 的 TCP/IP 栈像使用以太网一样使用它。模组对 MCU 而言就是一个"PPP 路由器"。

05

PPPoS 拨号过程详解

PPPoS（PPP over Serial）是 PPP 运行在串口上的形式（参考 RFC 1662）。完整拨号过程分为两个大阶段：AT 命令初始化阶段和 PPP 数据模式阶段。

图5-1：PPPoS 完整拨号时序图（AT命令阶段 + PPP协商阶段）

AT 命令关键指令说明

AT 命令作用典型响应
ATE0关闭命令回显OK
AT+CPIN?查询 SIM 卡状态+CPIN: READY
AT+CREG?查询网络注册状态+CREG: 0,1（已注册本地）
AT+CGDCONT=1,"IP","cmnet"配置 PDP 上下文（APN）OK
ATD*99***1#拨号进入数据模式（PPP）CONNECT
+++退出数据模式，回到 AT 命令模式OK
📌 PPPoS vs PPPoE 串口差异

PPPoS 没有发现阶段（因为串口是专用点对点连接），帧内容与标准 PPP 相同，但字节转义（0x7D）处理是关键。每帧以 0x7E 定界，数据内容中的 0x7E 和 0x7D 均需转义，低于 0x20 的控制字符默认也需转义（ACCM 协商可减少转义范围）。

⚛ 第一性原理解读

PPPoS 拨号的本质是一次"角色切换"：串口在 ATD*99***1# 之前是"命令通道"，之后变为"数据通道"。这是因为物理串口只有一对引脚，不能同时做两件事。切换后，MCU 的 PPP 栈接管串口，模组的 AT 解析器让位给 PPP 引擎。整个 IPCP 协商的核心是一次"IP 地址租赁"：MCU 说"我要一个地址"，模组向核心网申请后转告 MCU。

模组内部系统与核心网

4G 模组内部运行一个完整的嵌入式操作系统（如 ThreadX、RTOS 或厂商定制系统），对外提供 AT 命令接口，对内管理完整的 LTE 协议栈。理解模组内部架构和核心网结构，才能完整理解 PPP 两端的角色。

PPP 在核心网中的终止位置

在 4G/LTE 场景下，PPP 的"远端"逻辑上终止于 PGW（Packet Data Network Gateway），但物理上 PPP 协商发生在 MCU 与模组之间：

🏛 核心网 PGW 的角色（官方参考：3GPP TS 23.401）

PGW 负责：为 UE（User Equipment，即模组+SIM）分配 IP 地址；作为 SGi 接口与外网连接的锚点；执行 QoS、计费策略。

模组从 PGW 获得 IP 后，通过 IPCP Configure-Nak 将该 IP 传递给 MCU 的 PPP 栈，MCU 最终使用该 IP 进行网络通信。

⚛ 第一性原理解读

核心网的本质是一套"身份验证 + 地址分配 + 路由"系统。PGW 是"你的家门口"——所有进出 MCU 的 IP 流量都经过它。模组是"快递中转站"：它把 MCU 的 PPP 数据包解封装、转化为 GTP 隧道包，在核心网内传输，到 PGW 处再还原成普通 IP 包送往 Internet。整条链路中，PPP 只存在于 MCU↔模组之间的串口段，核心网内使用的是 GTP-U 隧道。

与家庭 PPPoE 拨号对比

家庭宽带拨号使用的是 PPPoE（PPP over Ethernet，RFC 2516），而 4G 模组使用的是 PPPoS（PPP over Serial，RFC 1662）。两者都是 PPP 的变体，但承载方式和网络结构有显著差异。

图7-1：PPPoE（家庭宽带）与 PPPoS（4G物联网）对比架构图

全面对比表

对比维度家庭 PPPoE（RFC 2516）4G 模组 PPPoS（RFC 1662）
承载介质以太网（Ethernet II 帧）UART 串口（异步串行字节流）
发现阶段需要（PADI→PADO→PADR→PADS）不需要（点对点直连）
PPP 两端路由器（客户端）↔ BRAS（服务端）MCU（客户端）↔ 模组（服务端）
认证依据ISP 账号/密码（用户身份）SIM 卡 IMSI + APN 配置
IP 来源BRAS 分配（Radius 服务器）PGW 分配（3GPP 核心网）
帧额外开销PPPoE 头 6B + PPP 2B = 8BHDLC 帧 6B + 字节转义
最大 MTU1492 字节（1500-8）通常 1500 字节（LCP 协商）
物理连接RJ45/SFP（固定有线接入）UART（本地）+ LTE（无线）
移动性无（固定接入）支持（随 SIM 卡移动）
典型应用家庭/企业宽带上网嵌入式 IoT、工业 M2M、车载
⚛ 第一性原理解读

PPPoE 和 PPPoS 的本质区别在于"链路发现"：以太网是共享广播介质（一个口可能连多个服务器），所以需要 Discovery 阶段来"选择"服务端；而串口是专用的点对点连接，两端固定，无需发现，省略此步骤。这体现了 PPP 协议族的灵活性——核心协商（LCP/NCP）不变，只是承载"容器"不同。

08

PPP 适用场景分析

PPP 凭借其协议无关性、动态协商能力和对多种物理媒介的支持，在多个领域有明确的适用场景：

📡
嵌入式 IoT / M2M 通信
MCU 通过 4G/NB-IoT/2G 模组上网。PPPoS 提供轻量级 IP 接入，无需 MAC 地址，适合资源受限设备。代表：工业传感器、智能表计、追踪器。
🏠
家庭/企业宽带接入
PPPoE 用于 ADSL/光纤宽带，用户账号密码认证，BRAS 分配 IP 并计费。支持按需上线、可溯源管理。国内三大运营商均广泛使用。
🚗
车载/移动终端
车载 T-BOX 通过 4G 模组 PPPoS 连接云平台。支持 SIM 卡身份认证，无需固定 IP，适合移动场景的动态地址分配。
🔗
WAN 专线接入
路由器间通过 PPP over Serial（如 V.35/E1）建立 WAN 链路。支持 CHAP 认证和链路质量监控（LQM/LQR），用于企业专线。
🛰️
卫星/低速链路接入
PPP 的 Van Jacobson 头部压缩可大幅减少 TCP/IP 头开销，在带宽昂贵的卫星链路（56Kbps）上效益显著。
🔒
VPN 隧道底层协议
L2TP/PPTP 将 PPP 帧封装在 UDP/GRE 隧道中传输，实现远程访问 VPN。PPP 提供认证和加密协商能力（EAP、MPPE）。

PPP 适用场景的决策矩阵

场景推荐方案理由不适用场景
资源受限 MCU + 4GPPPoS ✅无 MAC 地址需求，轻量，库（lwIP ppp）成熟高吞吐视频流（串口带宽限制）
家庭/小企业宽带PPPoE ✅账号管理、按需计费、现有 ISP 基础设施组播业务（PPP 不支持广播）
企业内网大量设备DHCP/IPoE ✅PPP 不支持广播/组播，扩展性差需要用户级计费场景
WAN 路由器互联PPP over Serial ✅标准化、支持 LCP 认证和 QoS 协商高速以太网 WAN（改用 L3 路由）
大规模 LAN 接入802.1X / RADIUS ✅PPP 是点对点协议，不支持多点广播需要 PPP 特定功能（压缩、MLPPP）

PPP 的局限性

✅ PPP 的优势
  • 协议无关，支持 IP/IPX 等多协议
  • 动态协商，无需预配置 IP
  • 内置认证（PAP/CHAP）和加密协商
  • 支持链路质量监控（LQM）
  • 头部压缩（Van Jacobson）
  • 适用多种物理媒介（串口/以太网/ATM）
  • 标准化程度高（RFC 1661 国际标准）
❌ PPP 的局限
  • 点对点，不支持广播/组播
  • 不适合 LAN 多接入场景
  • PPPoS 字节转义降低吞吐效率
  • LTE 核心网已逐步用 GTP 替代 PPP
  • PPPoE 额外头部增加 MTU 压力
  • 无内生安全加密（需配合 IPSec）
  • 5G SA 场景 PPP 使用进一步减少
⚛ 第一性原理解读

PPP 的核心价值在于它解决了一个最基本的问题：如何在一条"哑管道"（串口/电话线）上建立有状态的、经过认证的、多协议的网络连接。它适用的场景本质是"两个端点、一条独占链路、需要 IP 接入"。当场景变成"多个端点共享一条链路"时，PPP 就不再适用——这时需要以太网、Wi-Fi 等广播型媒介协议。PPP 的价值在于其不可替代性：在串口这种最底层、最通用的接口上，PPP 仍然是嵌入式设备联网的最佳协议选择。

📋 关键 RFC 原文摘录

"The Point-to-Point Protocol (PPP) provides a standard method for transporting multi-protocol datagrams over point-to-point links. PPP is comprised of three main components: 1. A method for encapsulating multi-protocol datagrams. 2. A Link Control Protocol (LCP) for establishing, configuring, and testing the data-link connection. 3. A family of Network Control Protocols (NCPs) for establishing and configuring different network-layer protocols."

—— RFC 1661, Abstract, Simpson, IETF, July 1994

📋 PPP 链路阶段原文摘录

"In order to establish communications over a point-to-point link, each end of the PPP link MUST first send LCP packets to configure and test the data link. After the link has been established, the peer MAY be authenticated. Then, PPP MUST send NCP packets to choose and configure one or more network-layer protocols."

—— RFC 1661, §3.1 Overview, Simpson, IETF, July 1994

⚛ 总结：第一性原理视角下的 PPP

PPP 存在的根本原因：任何两个设备要通过一条物理链路通信，都必须解决三个最基本的问题——①"怎么分包"（帧定界/封装）②"用什么语言"（协议协商）③"你是谁"（认证）。PPP 将这三个问题的解决方案标准化，使任何两端的设备都能自动完成协商，而无需事先手动配置。

PPP 的不可替代性：在点对点链路（串口、电话线、拨号链路）上，没有以太网 MAC 地址，没有广播域，PPP 是唯一能提供完整"链路建立+认证+IP分配"功能的标准协议。这使它在 4G 模组这类嵌入式外设场景中至今不可替代。

历史演进：PPP 诞生于 1994 年，贯穿了拨号上网时代（调制解调器）、ADSL 时代（PPPoE）、3G 时代（PPPoS over AT commands）直到 4G 时代（PPPoS + GTP）。5G SA 时代开始逐步被纯 IP 接入取代，但在嵌入式领域仍是主流。
