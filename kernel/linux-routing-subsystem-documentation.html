Linux内核路由子系统
架构全景

路由子系统是Linux网络栈中最核心的模块之一，决定每一个IP数据报的命运：本地投递、转发还是丢弃。从家用路由器到互联网骨干核心路由器，Linux路由子系统在各种场景中都扮演着关键角色。

什么是路由子系统？

Linux IP栈中，负责转发数据包并维护转发数据库的层叫做路由子系统（Routing Subsystem）。其核心任务是：给定一个目标IP地址，快速决定该数据包应如何处理。可能的决策包括：

🏠
RTN_LOCAL — 本地投递
数据包目标地址属于本机，向上传递到传输层协议栈
➡️
RTN_UNICAST — 单播转发
数据包需转发，通过指定下一跳（next hop）和出口设备发送
🚫
RTN_BLACKHOLE — 静默丢弃
匹配黑洞路由，静默丢弃，不发送任何错误通知
⚠️
RTN_UNREACHABLE — 不可达
返回ICMP Destination Unreachable错误给发送方
🔒
RTN_PROHIBIT — 管控拒绝
返回ICMP Communication Administratively Prohibited
📡
RTN_BROADCAST — 广播
目标是广播地址，本地接受并处理广播包

🧬 第一性原理

路由的本质是"决策函数"：给定一个五元组（源IP、目的IP、TOS、入接口、标记），映射到一个"动作"（投递/转发/丢弃）和"路径"（出接口+下一跳）。

一切设计复杂性，无论是Trie结构、多路由表、策略规则，都来自于同一个问题：如何在数百万条路由中，以纳秒级速度找到最长前缀匹配（LPM）？

路由子系统就是这个决策函数的高效实现：对正确性（最长前缀语义）和速度（数据结构压缩、缓存）二者的同时极致追求。

FIB 转发信息库

FIB（Forwarding Information Base，转发信息库）是路由子系统的核心数据存储。它不只是一张"路由表"，而是一套多层次的数据结构，支持多路由表、策略路由、等价多路径（ECMP）等高级特性。

FIB 的职责边界

FIB负责存储和检索路由条目，可以用一句话描述其核心逻辑：

"对于目标地址 D，在哪张路由表中，通过哪个下一跳，经过哪个接口，以何种方式转发？"

多路由表体系

默认情况下，Linux存在三张内建路由表：

表名ID (tb_id)用途管理者
local 255 (RT_TABLE_LOCAL) 本机地址、广播地址的投递路由，由内核自动维护 内核自动
main 254 (RT_TABLE_MAIN) 正常路由的存放位置，ip route默认操作此表 管理员 / 路由守护进程
default 253 (RT_TABLE_DEFAULT) 历史遗留，通常为空 历史遗留
自定义表 1–252 策略路由使用，通过ip rule选择 管理员
💡
当配置IP地址时（如 ip addr add 192.168.1.100/24 dev eth0），内核会自动在 local 表中添加三条路由：本地单播路由、网络广播路由、网络地址路由。同时在 main 表添加链路范围的直连路由。

local 表示例

local     127.0.0.0/8 dev lo proto kernel scope host src 127.0.0.1

FIB 层级结构

图2：FIB数据结构层级 — fib_table → LC-Trie → fib_alias → fib_info

FIB Alias 优化

fib_alias 是一个关键优化：多个路由条目可以指向同一个 fib_info 实例（共享下一跳信息），从而避免重复分配内存。所有 fib_alias 按 TOS降序、metric升序 排列，确保语义上的正确匹配顺序。

🧬 第一性原理

FIB的本质是"键值映射"的高效实现：键是(前缀, 掩码长度, TOS)，值是(下一跳, 出接口, 指标)。

多路由表体系源于同一需求：不同的流量（依据来源、标记等）可能需要走不同的路径。这是"路由策略"与"路由数据"的解耦——RPDB负责选择"看哪张表"，FIB负责"在表中找什么"。

fib_alias的存在，是"共享不变量"原则的体现：当多条路由的TOS或metric不同但下一跳相同时，复用同一个fib_info实例，内存使用降低，修改时只需改一处。

路由查找算法
从哈希表到 LC-Trie

路由查找的本质是"最长前缀匹配（LPM）"。在百万条路由规模下，算法效率至关重要。Linux的路由查找算法经历了从哈希表（fib_hash）到LC-Trie（LPC-Trie）的根本性演进。

最长前缀匹配（LPM）

给定目标地址，路由表中可能有多个前缀都与之匹配，应选择最精确（掩码最长）的那条：

目标IP匹配的前缀最终选择
192.0.2.490.0.0.0/0, 192.0.2.0/25, 192.0.2.49/32192.0.2.49/32 (最长)
192.0.2.510.0.0.0/0, 192.0.2.0/25192.0.2.0/25
192.0.2.2000.0.0.0/00.0.0.0/0 (默认路由)

LPC-Trie 核心机制

LPC-Trie（Level Path Compressed Trie，也称 LC-trie）结合了两种压缩技术：

📏
路径压缩（Path Compression）
跳过只有单一子节点的中间节点，记录跳过的位数（pos字段）。需要在叶节点验证完整匹配。
📐
层级压缩（Level Compression）
密集区域将多个层级合并为一个节点，用一个 2^k 大小的子节点向量代替多层二叉分支。
📈
动态平衡（inflate/halve）
当填充率>50%时扩展（inflate），<25%时收缩（halve）。保持节点数组合理规模。
🔄
trie_rebalance()
每次路由插入/删除后触发，从修改点向根回溯，依次对每个tnode执行resize()优化。

查找流程（fib_table_lookup）

// net/ipv4/fib_trie.c
int fib_table_lookup(struct fib_table *tb,
                     const struct flowi4 *flp,
                     struct fib_result *res,
                     int fib_flags)
{
    // 1. 从根节点开始向下遍历
    // 2. 每个节点取出 key 的 bits 位用于索引子节点数组
    // 3. 到达叶节点：通过 check_leaf() 进行语义匹配
    //    - 遍历 fib_alias hlist，按 TOS/metric 排序
    //    - 验证完整前缀（路径压缩跳过的位）
    // 4. 若未匹配：进入"最长前缀回溯模式"
    //    - 截断最低有效"1"位向上回溯
    //    - 继续寻找次长匹配前缀
    // 5. 成功则填充 fib_result，返回0
}

性能基准（官方数据）

1000条~20 ns~3<1 MiB
10万条~35 ns~5~13 MiB

✅
LPC-Trie的关键洞见：当路由条目密集时（如大量/32主机路由集中在一个/24网段），内部节点可以处理多达20位，路由查找退化为一次向量数组访问，延迟极低。内存效率也非常高：每百万条路由约使用128 MiB（IPv4）。
🧬 第一性原理

LPM查找的本质是"二叉决策树"的极致压缩：IP前缀天然形成一棵二叉树结构（每一位决定走左还是走右）。

路径压缩解决了"稀疏长链"问题——当某段路径上没有分叉，就不需要逐位检查，直接跳过；层级压缩解决了"密集短链"问题——当某个区域有大量分叉，用数组索引比指针链接效率高得多。

两者的结合，使得任意分布的路由集合都能以接近最优的树深进行查找，时间复杂度趋近于 O(1)（对于固定位宽的IP地址而言），而非朴素的 O(N)。

路由表结构
fib_table 与 key_vector

每张路由表由 struct fib_table 表示，其内嵌的 LC-Trie 由 struct key_vector 组成——既可以是内部节点（tnode），也可以是叶节点（leaf）。

fib_table 结构

struct fib_table { /* include/net/ip_fib.h */
  struct hlist_node tb_hlist; /* 挂在全局哈希表中 */
  u32 tb_id; /* 路由表ID: 254=main, 255=local */
  int tb_num_default; /* 默认路由条数 */
  struct rcu_head rcu; /* RCU保护 */
  unsigned long *tb_data; /* 指向LC-Trie根节点 */
};

key_vector：LC-Trie 节点

struct key_vector { /* net/ipv4/fib_trie.c */
  t_key key; /* 节点代表的IP前缀 */
  unsigned char pos; /* 从哪一位开始处理（路径压缩） */
  unsigned char bits;/* 处理多少位（层级压缩），0=叶节点 */
  unsigned char slen;/* 叶节点的前缀长度 */
  /* 内部节点: child[0..2^bits-1] 子节点向量 */
  /* 叶节点: hlist of struct fib_alias */
};
📌
当 bits == 0 时，该节点是叶节点（leaf），其下挂着一个 fib_alias hlist；
当 bits > 0 时，该节点是内部节点（tnode），持有 2^bits 个子指针的向量数组。

fib_info：路由条目的核心

struct fib_info { /* include/net/ip_fib.h */
  struct hlist_node fib_hash; /* 在全局fib_info_hash中 */
  struct net *fib_net; /* 所属网络命名空间 */
  atomic_t fib_clntref; /* 引用计数 */
  unsigned char fib_protocol;/* 路由来源: RTPROT_KERNEL/STATIC/OSPF... */
  unsigned char fib_scope; /* 范围: global/link/host */
  unsigned char fib_type; /* 类型: RTN_UNICAST/LOCAL/... */
  __be32 fib_prefsrc; /* 首选源地址 */
  u32 fib_priority;/* 路由优先级/metric */
  int fib_nhs; /* 下一跳数量(ECMP) */
  struct fib_nh fib_nh[0]; /* 下一跳数组（柔性数组） */
#define fib_dev fib_nh[0].nh_dev /* 快捷访问第一个下一跳设备 */
};

fib_nh：下一跳信息

struct fib_nh { /* include/net/ip_fib.h */
  struct net_device *nh_dev; /* 出口网络接口 */
  unsigned int nh_flags; /* 标志 */
  unsigned char nh_scope; /* 范围 */
  int nh_weight; /* ECMP权重 */
  __be32 nh_gw; /* 下一跳网关地址 */
  int nh_oif; /* 出口接口索引 */
  /* struct fib_nh_exception *fnhe_hash[] — PMTU/redirect异常表 */
};

fib_result：查找结果

struct fib_result { /* include/net/ip_fib.h */
  __be32 prefix; /* 匹配的前缀 */
  unsigned char prefixlen; /* 前缀长度 */
  unsigned char nh_sel; /* 选中的下一跳索引(ECMP) */
  unsigned char type; /* 路由类型 RTN_* */
  unsigned char scope; /* 作用域 */
  struct fib_info *fi; /* 指向fib_info */
  struct fib_table *table; /* 查找成功的路由表 */
  struct list_head *fa_head; /* fib_alias链表头 */
};
图4：从查找请求到路由决策的数据流 — flowi4 → fib_result → rtable
🧬 第一性原理

数据结构的设计体现了"关注点分离"：flowi4 描述"我要找什么"；fib_table + key_vector 描述"数据怎么存"；fib_result 描述"找到了什么"；rtable/dst_entry 描述"如何处理这个包"。

每一层都是纯粹的：查找层不关心路由如何应用，应用层不关心路由如何查找。dst.input / dst.output 函数指针是这种分离的极致——路由查找只需确定"该调用哪个函数"，其余留给包流处理层执行。

🚦 第4章 · 核心设计

策略路由（RPDB）
不只是目的地址

策略路由（Policy-Based Routing, PBR）允许基于源地址、标记（mark）、入接口、TOS等多种属性选择路由表，而非仅依靠目的地址。RPDB（Routing Policy Database）是其实现基础。

RPDB — 路由策略数据库

RPDB 是一组有序规则的列表，每条规则包含：优先级（PRIORITY）、选择器（SELECTOR）、动作（ACTION）。 内核按优先级从小到大依次评估规则，直到找到匹配并有路由结果。 

$ ip rule show
0:      from all lookup local    # 优先级0：所有包查local表
32766:  from all lookup main     # 优先级32766：所有包查main表
32767:  from all lookup default  # 优先级32767：查default表（通常为空）

CONFIG_IP_MULTIPLE_TABLES 编译选项

fib_lookup() 有两个实现，编译时由 CONFIG_IP_MULTIPLE_TABLES 决定：

配置行为适用场景
未设置 只有main表，fib_lookup()直接调用fib_table_lookup()，无RPDB开销 嵌入式系统、极简路由器
已设置（默认） 支持多路由表，fib_lookup()实现完整RPDB规则评估 大多数Linux发行版默认配置
🔑
优化：自Linux 3.0起（commit f4530fa574df），当RPDB中只有默认规则（未添加自定义规则）时，策略路由几乎无性能损耗。只有在添加了自定义规则后，才会产生额外的每包CPU消耗（约每条规则+30ns）。自Linux 4.1起（commit 0ddcf43d5d4a），在默认规则下，local表和main表被合并为一张表进行查找，进一步减少开销。
🧬 第一性原理

策略路由的本质是"查找哪张表"的元决策层：普通路由解决"在这张表里怎么走"，而策略路由解决"应该查哪张表"。

RPDB是一个有序规则的线性扫描器。其设计遵循"最优情形免费"原则：若无自定义规则，运行时无额外代价；复杂性只在真正需要时才引入。这体现了Linux内核设计的一贯哲学——为最常见的情况优化，为罕见情况提供逃生通道。

Linux路由子系统的优雅之处，在于各数据结构职责明确、相互分离。本章汇总所有核心结构体，展示它们之间的关联关系。

路由指标（Metrics）

每条路由可携带多种指标（metrics），存储在 fib_info.fib_metrics 数组中：

指标符号含义TCP相关

🧬 第一性原理

数据结构的分层反映了职责的分层：存储（key_vector/fib_alias）→ 语义（fib_info/fib_nh）→ 决策（fib_result）→ 应用（dst_entry/rtable）→ 执行（函数指针input/output）。

每一层只做自己的事。这种垂直分工使得修改某一层（如换用新查找算法）不会影响其他层。这是软件工程中"单一职责原则"在操作系统内核中的体现。

📦 第6章 · 数据结构

目标缓存 dst_entry
路由决策的载体

路由查找完成后，结果不会直接作用于数据包，而是封装成一个 struct dst_entry 对象，挂附到 sk_buff 上。这个对象包含函数指针，决定了数据包后续的处理路径。

dst_entry 结构

struct dst_entry { /* include/net/dst.h */
  struct net_device *dev; /* 出口网络接口 */
  struct dst_ops *ops; /* 操作集（IPv4/IPv6/其他协议） */
  unsigned long _metrics; /* 路由指标 */
  unsigned long expires; /* 过期时间 */
  int (*input)(struct sk_buff *); /* 入路径处理函数 */
  int (*output)(struct sk_buff *); /* 出路径处理函数 */
  unsigned short flags; /* DST_HOST等标志 */
  short obsolete; /* 是否已失效 */
  struct rcu_head rcu_head;
};

input / output 函数指针赋值

路由决策通过设置 input 和 output 函数指针来实现"多态"：

函数指针赋值情景
ip_local_deliver()input：接收到的包，目标为本机（RTN_LOCAL）
ip_forward()input：接收到的包，需要转发（RTN_UNICAST）
ip_output()output：本地发送的单播包
ip_mc_output()output：多播包
xfrm4_output()output：需经IPsec加密
ip_error()input：RTN_UNREACHABLE/PROHIBIT等错误路由
dst_discard_out()默认初始值，调用时表示BUG

rtable：IPv4路由决策的完整形态

struct rtable { /* include/net/route.h */
  struct dst_entry dst; /* 必须是第一个成员！*/
  int rt_genid; /* 路由代际ID，用于失效检测 */
  unsigned int rt_flags; /* RTCF_LOCAL/BROADCAST/MULTICAST... */
  __u16 rt_type; /* RTN_UNICAST/LOCAL/... */
  __u8 rt_is_input;/* 是否为输入路由 */
  __u8 rt_uses_gateway; /* 是否经过网关 */
  int rt_iif; /* 入接口索引 */
  __be32 rt_gateway; /* 下一跳网关IP */
  u32 rt_pmtu; /* 路径MTU */
};

缓存机制（Nexthop条目缓存）

自Linux 3.6移除路由缓存后，路由决策仍有缓存，但粒度不同：

💾
fib_nh级别缓存：每个 fib_nh（下一跳条目）缓存一个 dst_entry（接收路径）和一个（发送路径）。当路由条目首次被使用时，分配并初始化 rtable；后续相同路由命中时，直接复用缓存的 rtable 而无需重新分配。这比原来按"每个流"缓存更加安全（不可被DoS攻击填满），代价是粒度稍粗（不含流级信息如PMTU）。
🔌
Socket级别缓存：对于TCP等有状态协议，路由决策被缓存在 sock.sk_dst_cache 中，同一连接的后续数据包无需重新执行 fib_lookup()。
🧬 第一性原理

dst_entry 是"决策结果"与"执行策略"的分离点：查找层产生 fib_result，构建层将其翻译成 dst_entry（含函数指针），执行层调用函数指针而不关心结果从何而来。

函数指针机制（input/output）实现了路由子系统与协议栈其余部分的解耦：当路由变化时，只需更新 dst_entry 中的函数指针，协议栈其余代码无需任何修改。这是"开闭原则"在内核C语言中的实现方式。

📥 第7章 · 包流路径

接收路径（Rx Path）
数据包的入栈之旅

数据包从网卡接收到最终被本机应用层处理（或转发），要经过多层处理函数。路由查找在接收路径中处于 Prerouting 钩子之后、Transport 层之前。

完整接收路径

图7：IPv4单播包接收路径 — 从网卡到本地投递/转发的完整函数调用栈

关键函数调用链

// 网卡中断 → SoftIRQ → netif_receive_skb()
// → deliver_skb() → ip_rcv()          [链路层→网络层]
// → NF_HOOK(PREROUTING)
// → ip_rcv_finish() → ip_rcv_finish_core()
//     → ip_route_input_noref()         [路由查找入口]
//         → ip_route_input_slow()
//             → fib_lookup(net, &fl4, &res)  [实际查找]
//             → __mkroute_input()      [构建rtable]
//             → skb_dst_set(skb, &rt->dst)   [挂附到skb]
// → dst_input(skb)                    [按决策分叉]
//     → (*dst->input)(skb)
//       若RTN_LOCAL: ip_local_deliver()
//       若RTN_UNICAST: ip_forward()
//           → NF_HOOK(FORWARD)
//           → ip_forward_finish()
//               → dst_output(skb)     [发送路径入口]
//                   → (*dst->output)(skb)
//                       → ip_output()
//                           → NF_HOOK(POSTROUTING)
//                           → ip_finish_output()
⚠️
ICMPv4 重定向（Redirect）：当入接口和出接口相同时，内核会设置 RTCF_DOREDIRECT 标志，并在 ip_forward() 中通过 ip_rt_send_redirect() 向发送方发送 ICMP Redirect 消息，告知更优路径。收到 Redirect 消息的主机通过 update_or_create_fnhe() 创建 FIB 下一跳异常（nexthop exception）记录。
🧬 第一性原理

接收路径是一条流水线：每个函数只做一件事，结果传给下一个。路由查找是这条流水线的"分流器"——它是唯一一个根据数据内容改变后续处理函数的地方。

dst_input() 的函数指针调用，是这条流水线中最关键的一步：一个间接调用，决定了数据包进入本机处理栈还是走向另一个网络接口。这种"延迟绑定"模式让路由逻辑与协议处理逻辑完全解耦。

📤 第8章 · 包流路径

发送路径（Tx Path）
本地生成包的出栈之旅

本机应用层（TCP/UDP）产生的数据包，同样需要经过路由查找才能确定出口接口和下一跳。与接收路径不同，发送路径的路由查找在传输层出口处进行，且路由决策会缓存在socket中。

TCP 发送路径

图8：IPv4本地发送路径 — 从应用层socket到网卡发送的函数调用栈

UDP 路径差异

UDP 在 udp_sendmsg() 中直接进行路由查找（对于已连接的socket同样有socket级路由缓存）：

// net/ipv4/udp.c
int udp_sendmsg(struct sock *sk, struct msghdr *msg, size_t len) {
    // 1. 检查已连接UDP的路由缓存
    rt = (struct rtable *)sk_dst_check(sk, 0);
    if (!rt) {
        // 2. 执行路由查找
        rt = ip_route_output_flow(net, fl4, sk);
        // 3. 缓存到socket
        sk_dst_set(sk, &rt->dst);
    }
    // 4. 构建IP头，调用 ip_send_skb()
}

转发包与本地包的合并点

🔀
dst_output() 是转发路径和本地发送路径的汇合点。无论数据包来自转发（ip_forward_finish()）还是本地生成（ip_local_out()），最终都调用 dst_output(skb)，进而调用 dst->output 函数指针（通常是 ip_output()），经过 Postrouting 钩子后送达网卡驱动。
🧬 第一性原理

发送路径揭示了"缓存换速度"的核心权衡：路由查找代价高（即使LPC-Trie已很快），对于长寿命连接（TCP），每包都查找是巨大浪费。socket级路由缓存将这个代价摊薄到连接建立时一次。

但缓存有失效问题：网络拓扑变化（路由更新）时，缓存的路由可能过时。Linux用 rt_genid 代际ID（每次路由表变化递增）来检测失效，无需主动通知每个socket——这是"懒惰失效"（lazy invalidation）策略，在实践中效率极高。

🕰️ 第9章 · 进化历程

历史演进时间线

Linux路由子系统历经三十年持续演进，从最初的简单哈希表，到策略路由、LC-Trie、路由缓存消亡，再到VRF的引入，每次变化都有明确的工程动机。

Linux 2.1.15 (1996)
首次引入路由分类（Classes）
Alexey Kuznetsov 引入路由"类别"概念（对应今天的路由表），最初设计支持255个类别。策略路由思想的雏形。
Linux 2.1.68 (1997)
策略路由（Policy Routing）正式成形
RPDB（Routing Policy Database）引入，三张内建路由表（local/main/default）确立。ip rule命令对应的内核支持正式完整。原 default 表保留至今主要为历史兼容。
Linux 2.6.13 (2005)
LC-Trie (LPC-Trie) 首次引入
commit 19baf839ff4a：Robert Olsson 将 LC-Trie 算法引入内核作为 fib_trie，但此时只是可选编译项（CONFIG_IP_FIB_TRIE），默认仍使用 fib_hash。基于 Nilsson & Karlsson 1999年IEEE论文。
Linux 2.6.23 (2007)
IPv4 多路径路由缓存移除
IPv4 Multipath Routing Cache 从未稳定工作，以实验状态存在多年后正式移除。多路径路由（ECMP）本身继续存在。
Linux 2.6.39 (2011)
fib_hash 移除，LPC-Trie 成为唯一查找算法
commit 3630b7c050d9：David Miller 移除哈希表实现，LPC-Trie 成为IPv4路由查找的默认且唯一算法。这是为后续移除路由缓存奠基的关键步骤。
Linux 3.6 (2012)
IPv4 路由缓存完全移除
commit 5e9965c15ba8：路由缓存（大哈希表，按源+目的IP+TOS索引）因DoS漏洞（可任意填满）被完全移除。LPC-Trie的高性能使这一移除成为可能。同年，IPv6路由缓存也被标记为弃用。
Linux 3.0 (2011)
默认规则下多路由表零开销
commit f4530fa574df：未配置自定义规则时，CONFIG_IP_MULTIPLE_TABLES 不产生任何运行时开销。策略路由与非策略路由场景的性能差距消除。
Linux 4.0 (2015)
LC-Trie 查找大幅优化
commit 9f9e636d4f89（Alexander Duyck）：对 Trie 查找算法进行多项优化，性能显著提升（在基准测试中降低约40%延迟）。
Linux 4.1 (2015)
local表与main表合并
commit 0ddcf43d5d4a（Alexander Duyck）：默认规则集下，local表和main表合并为一张表进行查找，减少一次 fib_table_lookup() 调用。
Linux 4.3 (2015)
VRF (Virtual Routing and Forwarding) 引入
commit 193125dbd8eb：VRF设备驱动引入，实现L3路由隔离。Linux 4.8进一步优化性能并引入 l3mdev-table 特殊规则，支持用单条IP规则管理整个VRF域的路由。
Linux 5.x+ (持续)
Nexthop 对象化、多路径改进
Linux 5.1引入 nexthop 对象（CONFIG_IP_ROUTE_MULTIPATH），将下一跳作为独立内核对象管理，支持更灵活的ECMP策略和快速更新。持续的性能优化和BPF集成使路由子系统更加可编程。
🧬 第一性原理

进化的驱动力始终是同一组矛盾：性能 vs 安全性、简单性 vs 功能性、内存效率 vs 查找速度。

路由缓存的兴衰是最典型案例：为提速而引入缓存，为安全（DoS防御）而移除缓存，为弥补性能损失而将LPC-Trie设为默认。每一步都是在约束条件变化后重新寻找最优解，而非无目的的改变。

这印证了软件演化的本质：没有永久的最优解，只有在当前约束下的最优解。约束（硬件、威胁模型、使用场景）改变，最优设计随之改变。

路由缓存的消亡
DoS、性能与权衡

IPv4路由缓存在Linux中存在了十多年，于2012年（内核3.6）被完全移除。这是一个关于安全性、性能权衡和技术债务的典型案例。

路由缓存的工作原理（≤ Linux 3.5）

路由缓存是一个全局哈希表（rt_hash_table），以 （源IP, 目的IP, TOS, 入接口） 作为键：

图9：路由缓存工作原理与DoS攻击面

为什么移除路由缓存？

💣
DoS 攻击漏洞
缓存按"流"（源IP+目的IP+TOS）存储。攻击者发送随机目的地址包，每包产生新缓存条目，轻易耗尽内核内存。
🐢
GC 开销
垃圾回收器（定时器+阈值触发）在高负载下对性能有显著影响，代码复杂度高且难以调优。
⚡
LPC-Trie 的成熟
2.6.39引入LPC-Trie后，直接FIB查找已足够快（50ns量级），不再需要缓存弥补性能差距。
🧹
代码简化
移除缓存同时清理了fib_hash表和大量GC相关代码，代码库显著精简，可维护性提升。

移除后的缓存替代机制

✅
路由缓存消亡 ≠ 缓存消亡。移除的是"按流缓存"，保留的是"按路由条目缓存"（fib_nh级别缓存）和"按socket缓存"（sk_dst_cache）。前者粒度更粗（不记录流级细节如PMTU异常），但完全免疫DoS；后者针对长寿命连接优化，同样高效。 
# IPv4路由缓存移除相关关键 commits (Linux 3.6)
5e9965c15ba8  net: Remove route cache related code
                (David S. Miller, 2012年7月)
                
# LPC-Trie设为唯一默认（使移除缓存成为可能）
3630b7c050d9  ipv4: Remove fib_hash
                (David S. Miller, 2011年, Linux 2.6.39)
🧬 第一性原理

缓存的存在有其前提假设：缓存只在"查找成本远高于缓存管理成本"时才有价值。当LPC-Trie使查找本身足够快，这个假设不再成立。

路由缓存的消亡揭示了一个普遍规律：用缓存解决性能问题，需先确保缓存本身不会引入更严重的问题（如安全漏洞、GC抖动、一致性问题）。有时，足够快的算法比缓存更好——因为它没有缓存带来的所有附加复杂性。

路由规则数量对性能的影响

⚠️
每条自定义IP规则大约增加30ns的每包开销（Linux 4.11测试数据）。规则数量与延迟呈线性关系（1-100条范围内），超过100条后斜率显著增大。因此，在高性能场景中，应尽量减少IP规则数量，优先使用VRF（单条 l3mdev-table 规则代替多条接口规则）。
🧬 第一性原理

性能优化的本质是减少"无效工作"：每一次重大性能提升，都对应消除了某类不必要的计算——fib_hash的O(N/bucket)退化、GC的随机抖动、规则评估的固定开销、重复的fib_table_lookup调用。

LPC-Trie的性能洞见在于：对于固定位宽(32位)的键，树的最大深度有上界。密集路由时，层级压缩让多个层次合并，深度趋近于1-2，查找退化为数组索引——这是所有数据结构中的性能天花板。

IPv6路由子系统
与IPv4的异同

IPv6路由子系统在2.1.8（1996）引入，早于基于LPC-Trie的IPv4实现。两者结构相似但算法不同，且IPv6支持源地址特定路由等额外特性。

IPv6 vs IPv4：核心差异

特性IPv4IPv6
查找算法 LPC-Trie（层级+路径压缩）
fib_trie.c		 Radix Tree（Patricia Trie，路径压缩）
ip6_fib.c
节点结构 struct key_vector (内部节点 + 叶节点统一) struct fib6_node（分开）+ struct rt6_info
路由缓存 3.6版本完全移除 3.12+版本移除（稍晚）

为何IPv6不使用LPC-Trie？

🤔
Vincent Bernat在其2017年分析中指出三个原因：
1. 历史：IPv6实现（1996）早于LPC-Trie引入IPv4（2005），彼时代码路径已固化。
2. 地址空间密度：IPv4地址空间（32位）在实际部署中较密集，层级压缩效益高；IPv6（128位）地址极其稀疏，层级压缩几乎无效。
3. 源地址路由：IPv6的源地址特定路由需要先按目的地址查找，再按源地址匹配，两次查找的结构更适合Radix Tree。

IPv6源地址特定路由

# IPv6 支持源地址特定路由（Source-Specific Routing）
$ ip -6 route add 2001:db8:1::/64 \
    from 2001:db8:3::/64 \      # 源地址约束
    via fe80::1 \
    dev eth0

# 查找流程：
# 1. 先按目的地址(2001:db8:1::x)在Radix Tree中查找
# 2. 再按源地址(from 2001:db8:3::y)过滤匹配的路由条目
# 只有源+目的都满足的条目才被采用

IPv6 Radix Tree 节点

struct fib6_node { /* net/ipv6/ip6_fib.c */
  struct fib6_node *parent; /* 父节点 */
  struct fib6_node *left; /* 左子节点 (bit=0) */
  struct fib6_node *right; /* 右子节点 (bit=1) */
  struct fib6_node *subtree; /* 源地址子树 */
  struct rt6_info *leaf; /* 路由条目链表 */
  __be32 fn_bit; /* 检查的bit位 */
  __u16 fn_flags; /* RTN_ROOT等 */
};
🧬 第一性原理

IPv6与IPv4路由的差异，根源在于地址空间特性：IPv4 32位地址空间在实际中高度密集（CIDR汇总后仍有大量/24-/28路由），层级压缩能将多层合并为数组访问；IPv6 128位地址空间在实际中极度稀疏，每个前缀在空间中都"孤立"，层级压缩几乎无法工作。

"一种算法适合所有情况"是一个危险的假设。IPv6路由保留Radix Tree而非强行使用LPC-Trie，是理性的工程决策——用正确的工具解决正确的问题，而不是教条地统一实现。

🏗️ 第13章 · 高级特性

VRF 虚拟路由转发
L3级网络隔离

VRF（Virtual Routing and Forwarding）允许在单台Linux主机上创建相互隔离的L3路由域，每个域拥有独立的路由表。这是数据中心和运营商场景中网络隔离的关键技术。

VRF 架构

图11：VRF架构 — 两个隔离的路由域，一条RPDB规则管理

VRF 配置示例

# 创建VRF设备并绑定路由表
ip link add vrf-A type vrf table 10
ip link set dev vrf-A up
ip link add vrf-B type vrf table 20
ip link set dev vrf-B up

# 将接口绑定到VRF（本地/直连路由自动迁移到VRF路由表）
ip link set dev eth1 master vrf-A
ip link set dev eth2 master vrf-A
ip link set dev eth3 master vrf-B

# 添加VRF默认路由（高metric防止覆盖正常默认路由）
ip route add table 10 unreachable default metric 4278198272
ip route add table 20 unreachable default metric 4278198272

# 查看VRF规则（Linux 4.8+自动添加 l3mdev-table 规则）
$ ip rule show
0:      from all lookup local
1000:   from all lookup [l3mdev-table]   # 自动添加
32766:  from all lookup main
32767:  from all lookup default

VRF 与网络命名空间的比较

维度网络命名空间 (netns)VRF
隔离层次L2+L3（设备层隔离）仅L3（路由层隔离）
LLDP等L2协议隔离不隔离（L2工具无感知）
嵌套使用可与VRF嵌套使用可嵌套于netns内
管理复杂度较高（独立栈）较低（共享内核栈）
适用场景完全隔离的网络栈多租户L3路由隔离
🔑
关键设计决策：VRF的 l3mdev-table 规则（Linux 4.8）将"多条per-接口规则"合并为"一条VRF感知规则"。这不仅减少了RPDB规则数量（降低每包开销），也让VRF设备成为路由域的"一等公民"，与策略路由框架无缝集成。
🧬 第一性原理

VRF的本质是"路由表的命名空间"：正如网络命名空间是"网络接口+路由+socket的命名空间"，VRF只隔离L3路由维度，其余（L2、socket绑定）共享。

VRF设计遵循"最小隔离原则"：只隔离需要隔离的东西（L3路由），其余保持共享（这使得LLDP等L2协议不受影响，减少了运营复杂度）。l3mdev-table规则是"隐式约定优于显式配置"的体现——系统自动根据接口归属推断应查哪张路由表，而非要求管理员逐一配置。
