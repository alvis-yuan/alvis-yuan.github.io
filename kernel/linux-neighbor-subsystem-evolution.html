Linux Kernel Networking · 邻居子系统

Linux 内核邻居子系统
演进史

从 1994 年 Florian La Roche 的第一行 ARP 代码，到 2024 年支持 SDN/云原生的现代邻居子系统——30 年间，Linux 如何解决"IP 地址到 MAC 地址"这一看似简单却极度复杂的问题。

🌍 为什么需要邻居子系统？

RFC 826（ARP，1982）开篇一句话道出网络的本质："The world is a jungle in general, and the networking game contributes many animals."（世界本是一片丛林，而网络游戏让其中的动物更加多样。）

在以太网环境中，IP 层看到的是 IP 地址（L3），而网卡驱动需要的是 MAC 地址（L2）。每当内核要将一个 IP 数据包发送给同一链路上的邻居时，必须先解析其 MAC 地址。邻居子系统就是负责维护这一 L3→L2 映射关系的核心模块。

在 IPv4 中，该映射通过 ARP（Address Resolution Protocol）完成；在 IPv6 中，则由 NDP（Neighbor Discovery Protocol，RFC 4861）承担，同时还集成了路由器发现、无状态地址自动配置等功能。

🔬 第一性原理概述
网络通信的物理本质是：数据帧在链路层流动，而链路层只认硬件地址（MAC），不认 IP。因此，凡是基于 IP 的通信，都必须在发送前解决"这个 IP 对应哪个 MAC"的问题。邻居子系统的存在，正是对这一物理约束的最小化响应——它是内核中 L3 意图与 L2 现实之间唯一的"翻译官"。
第一纪元 · 奠基期 · 1994
Linux 1.x — ARP 的诞生
以太网假设、单一协议、全局锁
🔴 历史背景与痛点

1994 年，Florian La Roche 为 Linux 编写了第一版 ARP 实现（net/ipv4/arp.c），直接遵循 RFC 826。代码深度绑定以太网假设，硬编码了 6 字节 MAC 地址长度，无法支持 AX25、ATM、FDDI 等其他链路类型。

彼时整个网络栈运行在一个单一的"大内核锁（BKL）"保护下。ARP 缓存是一个简单的固定大小数组，既无高效的查找结构，也无超时回收机制。当网络设备接口 down 掉时，残留的 ARP 条目可能指向已失效的内存。

🔧 Alan Cox 的早期修复（1994–1996）

Alan Cox 对原始实现进行了大量修复：

去除以太网假设
移除 Florian 代码中对 6 字节 MAC 的硬编码，支持可变长度硬件地址
设备通知机制
改用 notifier 链，当网络设备 down 时自动清除相关 ARP 条目（避免悬挂指针）
定时器初始化
引入 init_timer()，为 ARP 条目增加超时回收机制
双重锁修复
修复多处 double-lock 竞态条件
📋 其他贡献者修复
Andrew Tridgell — Proxy ARP
新增 ARP netmask 代码，重新整理代理 ARP（Proxy ARP）处理逻辑
Mike Shaver — RFC1122 合规
增加 RFC 1122 要求的检查：重传节流、缓存超时可配置、保存未决包
Jonathan Layes — arpd 支持
通过 kerneld 消息队列支持用户空间 ARP 守护进程（1996.03.14）
Manuel Rodriguez — Gratuitous ARP
新增免费 ARP（GARP）支持，用于主动宣告自身 MAC 地址变更
⚠ 核心痛点
早期实现将"以太网"视为唯一链路类型，无法运行在 AX25（业余无线电）、FDDI、ATM 等网络上。ARP 缓存是一个哑巴数组：满了就扔旧条目，不论对方是否仍然活跃；也没有"对方已下线"的探测机制，导致包会持续发往已消失的邻居。
🔬 第一性原理
链路层寻址的本质需求是：用最小代价维护一张 IP→MAC 的映射表。Linux 1.x 的实现选择了"最简单可行"路径——固定数组 + 全局锁——在网络规模小、硬件单一的时代是合理的，但它把"以太网"和"IPv4"作为了隐含的公理，埋下了日后所有扩展工作的债务。
第一纪元 · 奠基期 · 1997
Linux 2.1 — 邻居子系统的诞生
Alexey Kuznetsov 的革命性重构：NUD 状态机 + net/core/neighbour.c
🚀 背景：互联网规模爆炸

1996–1997 年，互联网用户数量从数百万跃升至数千万。Linux 开始被部署在真实的服务器和路由器上。原有的 ARP 实现面临严峻挑战：大型园区网中的 ARP 缓存动辄需要维护数千条目，而固定大小的数组完全不够用。更严重的是，当一台主机下线后，内核无法主动检测——只能等待条目超时，期间所有发往该主机的包都会静默丢失。

Alexey Kuznetsov（kuznet@ms2.inr.ac.ru）在内核 2.1.x 开发周期中，将 ARP 的状态管理逻辑完全从 net/ipv4/arp.c 中剥离，创建了全新的 net/core/neighbour.c——这正是现代邻居子系统的起点。

⚙️ 核心创新：NUD 状态机

NUD（Neighbor Unreachability Detection，邻居不可达检测）状态机是这次重构的核心。它借鉴了 RFC 2461（后被 RFC 4861 替代）的概念，为每个邻居条目定义了明确的生命周期状态：

📦 struct neighbour 核心字段
struct neighbour {
  struct neighbour __rcu *next; /* 哈希链表 */
  struct neigh_table  *tbl;     /* 所属表 */
  struct neigh_parms  *parms;   /* 可调参数 */
  unsigned long confirmed;      /* 最后确认时间 */
  rwlock_t      lock;           /* 读写锁 */
  __u8          nud_state;      /* NUD 状态 */
  __u8          ha[MAX_ADDR_LEN]; /* 硬件地址 */
  struct hh_cache  hh;          /* 头部缓存 */
  int (*output)(...);           /* 输出函数指针 */
  struct sk_buff_head arp_queue; /* 待发队列 */
  struct timer_list timer;      /* 状态定时器 */
};

output() 函数指针随状态变化动态切换：未解析时为 neigh_resolve_output()，已解析时为 neigh_connected_output()。

🧠 L4 确认机制

NUD 的关键创新：上层协议（TCP/UDP）可以直接向邻居子系统反馈"可达性确认"，无需重新发 ARP 请求：

/* tcp_input.c */
tcp_ack()
  → dst_confirm()
    → neigh_confirm()
      /* 更新 confirmed 时间戳
         但不改变 NUD 状态 —
         状态只由 neigh_timer_handler()
         负责切换 */

这意味着：只要 TCP 连接正常，邻居条目就不会被标记为不可达，减少了不必要的 ARP 探测风暴。

🗑️ 垃圾回收机制（GC）

Kuznetsov 引入了两级 GC 机制，这一设计一直沿用至今：

周期性 GC（异步）

neigh_periodic_timer() 定期扫描表，删除 NUD_FAILED 状态和过期条目。仅在表内条目数超过 gc_thresh1 时才真正删除，避免空表时的无谓扫描。

强制 GC（同步）

neigh_forced_gc() 在 neigh_alloc() 发现条目数超过 gc_thresh3 时被调用，属于"背压"机制：立即释放引用计数为 1 的非永久条目。

⚙️ 可调参数（/proc/sys/net/ipv4/neigh/）
gc_thresh1（默认128）· gc_thresh2（默认512）· gc_thresh3（默认1024）
base_reachable_time · retrans_time · delay_probe_time
🔬 第一性原理
现实网络中，主机会上线、下线、迁移。任何"认为邻居永远在线"的假设都会导致包的静默丢失。NUD 状态机的本质是：将"邻居是否可达"从一个二元静态标记，升级为一个带时间衰减的动态概率估计。通过定时器驱动的状态转换，内核能够以最小的探测代价（单播 ARP）主动发现邻居失效，而不是被动等待上层报告错误。
第二纪元 · 整合期 · 2001
Linux 2.4 — IPv6 与 NDP 的整合
一张表统治一切：ARP 与 NDP 共享同一套邻居框架
🌐 背景：IPv6 时代的来临

IPv6 标准化工作在 1990 年代末基本完成，RFC 2461（1998，后被 RFC 4861 取代）定义了 NDP（Neighbor Discovery Protocol）。NDP 与 IPv4 的 ARP 相比，设计上有重要区别：

ARP（IPv4）

  • 独立协议（EtherType 0x0806）
  • 广播查询（整个链路段）
  • 仅地址解析功能
  • 无路由器发现
  • 无重复地址检测（DAD）

NDP（IPv6）

  • 嵌入 ICMPv6（类型 133-137）
  • 组播查询（仅发给目标节点组播组）
  • 集成路由器发现（RA/RS）
  • 集成 DAD（重复地址检测）
  • 集成无状态地址自动配置（SLAAC）
🔀 架构整合：neigh_table 的统一抽象

为了避免重复实现状态机，Linux 内核巧妙地让 IPv6 NDP 复用了 Kuznetsov 的邻居框架。ARP 和 NDP 共享同一套 struct neigh_table 机制，各自只需实现协议特定的操作：

/* IPv4 ARP 表 */
struct neigh_table arp_tbl = {
    .family     = AF_INET,
    .key_len    = 4,          /* IPv4 地址 4 字节 */
    .constructor = arp_constructor,
    .solicit     = arp_solicit,       /* 发 ARP 请求 */
    .error_report = arp_error_report,
    ...
};

/* IPv6 NDP 表 */
struct neigh_table nd_tbl = {
    .family     = AF_INET6,
    .key_len    = 16,         /* IPv6 地址 16 字节 */
    .constructor = ndisc_constructor,
    .solicit     = ndisc_solicit,     /* 发 NS 消息 */
    .error_report = ndisc_error_report,
    ...
};

两张表的 GC、状态机、定时器逻辑完全共用 net/core/neighbour.c，只有 solicit/constructor 等几个钩子不同。这是"开闭原则"在内核中的典范应用。

🔍 DAD（重复地址检测）

NDP 集成了 DAD 机制（RFC 4862），在接口配置 IPv6 地址前，先以源地址 ::（全零）发送 NS，探测该地址是否已被占用。IPv4 也通过 arping -D 实现了类似功能（RFC 2131），内核 arp_process() 中有特殊处理：

/* net/ipv4/arp.c */
/* IPv4 DAD: 源 IP 为 0.0.0.0 时的特殊处理 */
if (sip == 0) {
    if (arp->ar_op == htons(ARPOP_REQUEST) &&
        inet_addr_type(tip) == RTN_LOCAL &&
        !arp_ignore(in_dev, dev, sip, tip))
        arp_send(ARPOP_REPLY, ETH_P_ARP, tip, dev,
                 tip, sha, dev->dev_addr, dev->dev_addr);
    goto out;
}
🔬 第一性原理
ARP 广播是一个"广而告之"的设计，在小型网络中无问题；但在拥有数千台主机的数据中心，每秒数百次 ARP 广播会让所有主机的 CPU 都产生中断，这是带宽和算力的双重浪费。NDP 的组播设计是对"最小必要通知范围"原则的回归——只通知真正需要知道的节点。内核通过统一的 neigh_table 抽象，使得这两种协议的实现代码共享超过 80%，完美体现了"机制与策略分离"的设计哲学。
第二纪元 · 整合期 · 2003–2006
Linux 2.6 — Netlink 接口与管理能力
从 ioctl 到 rtnetlink，从 /proc 到 sysctl，从静态配置到动态管理
📡 Netlink 接口：取代 ioctl

早期内核使用 ioctl 管理 ARP 表，功能受限且难以扩展。Linux 2.6 中，邻居子系统完全接入 rtnetlink 消息框架，ip neigh 命令（iproute2）由此取代了传统的 arp 命令：

# 旧方式 (arp 命令)
arp -s 192.168.0.1 aa:bb:cc:dd:ee:ff
arp -d 192.168.0.1
arp -n  # 显示（无状态信息）
# 新方式 (ip neigh)
ip neigh add 192.168.0.1 dev eth0 \
  lladdr aa:bb:cc:dd:ee:ff nud permanent
ip neigh del 192.168.0.1 dev eth0
ip neigh show  # 带完整 NUD 状态

rtnetlink 消息类型：RTM_NEWNEIGH / RTM_DELNEIGH / RTM_GETNEIGH，支持 IPv4 和 IPv6 统一接口。

📊 /proc 接口优化

Arnaldo Carvalho de Melo 将 /proc/net/arp 从传统的单次 dump 实现转换为 seq_file 接口，解决了大型 ARP 表在 /proc 读取时的内存分配问题（原来需要一次性分配足以容纳整个表的缓冲区，> 4K 时会失败，这是 Alan Cox 早期就注意到的 bug）。

统计接口也同步完善：

cat /proc/net/stat/arp_cache    # IPv4 邻居统计
cat /proc/net/stat/ndisc_cache  # IPv6 NDP 统计
# 统计字段：allocs, destroys, hash_grows, lookups, hits,
#           res_failed, rcv_probes_mcast, rcv_probes_ucast,
#           periodic_gc_runs, forced_gc_runs, table_fulls
🔐 arp_ignore / arp_announce：精细化 ARP 行为

在 LVS（Linux Virtual Server）等负载均衡场景中，多个网卡可能配置相同的 VIP（虚拟 IP）。当 ARP 广播来临时，所有网卡都会响应，导致负载均衡失效。内核引入了 arp_ignore 和 arp_announce 两个 sysctl 参数解决此问题：

arp_ignore（控制回复）

0: 任何接口都可以回复（默认）
1: 只有目标 IP 配置在
   接收接口上才回复
2: 目标 IP 必须配置在
   接收接口的同一子网内

arp_announce（控制发送）

0: 使用任何本地地址（默认）
1: 尽量用属于目标子网的
   源地址
2: 严格使用目标接口的
   最优本地地址

LVS/DR 标准配置：在实际服务器的 lo 接口上配置 VIP，同时设置 arp_ignore=1 和 arp_announce=2，防止 VIP 被泄露到 ARP 广播中。

🔬 第一性原理
ARP 的原始设计假设"每个 IP 只绑定一个接口"，但现实的负载均衡、多宿主（Multi-homing）、容器网络都在打破这一假设。arp_ignore 和 arp_announce 的本质是：允许管理员在策略层面重新定义"哪个接口有资格代表某个 IP"，这是对 ARP 设计假设的受控例外——而非修改协议本身。
第三纪元 · 扩展期 · 2011–2014
Linux 3.x — RCU 无锁化改造
多核时代的性能革命：读写锁 → RCU，哈希表线性链 → 动态扩展
⚡ 背景：多核 CPU 的锁争用危机

2010 年代初，x86 服务器普遍拥有 16–64 个 CPU 核心，10GbE/40GbE 网卡已经普及。邻居子系统的 rwlock_t tbl->lock 成为了严重的性能瓶颈：每次查找邻居条目都需要获取读锁，在高并发下多核之间的锁竞争导致 CPU 缓存行反弹（cache line bouncing），每秒数百万次的查找操作中，大量时间浪费在锁等待上。

Eric Dumazet 主导了一系列 RCU（Read-Copy-Update）改造，提交消息记录为："neigh: RCU conversion of struct neighbour"（commit 767e97e1e0db）。

📐 动态哈希表：neigh_hash_table

Linux 3.x 还将邻居表的哈希表从静态固定大小改为动态扩展。哈希表本身通过 RCU 保护，扩容时无需停止读者：

struct neigh_hash_table {
    struct neighbour __rcu **hash_heads; /* RCU 保护的桶数组 */
    unsigned int    hash_shift;          /* 桶数 = 1 << hash_shift */
    __u32           hash_rnd[4];         /* 随机哈希种子（防 DoS） */
    struct rcu_head rcu;                 /* 延迟释放 */
};

/* 当条目数超过桶数时，自动扩容 */
if (atomic_read(&tbl->entries) > (1 << nht->hash_shift))
    nht = neigh_hash_grow(tbl, nht->hash_shift + 1);

哈希函数同步从简单取模改为 Jenkins Hash（Harald Welte 贡献），有效防止了恶意构造的 IP 地址导致哈希碰撞攻击（哈希洪水 DoS）。

🔏 hh_cache 与 seqlock

头部缓存（hh_cache）存储了预构建的链路层帧头，用于加速快速路径（fast path）的包发送。在 RCU 时代，它使用 seqlock 保护：读者在读取前后检查序列号，如果不一致则重试，避免了写者持有锁时阻塞高频读者：

static inline int neigh_hh_output(const struct hh_cache *hh,
                                   struct sk_buff *skb) {
    unsigned int seq;
    do {
        seq = read_seqbegin(&hh->hh_lock);
        hh_len = READ_ONCE(hh->hh_len);
        memcpy(skb->data - hh_alen, hh->hh_data, hh_alen);
    } while (read_seqretry(&hh->hh_lock, seq)); /* 写者修改期间重试 */
    return dev_queue_xmit(skb);
}
🔬 第一性原理
邻居表查找是 IP 转发路径上的热点操作（每个 IP 包都可能触发一次）。在多核系统中，锁的竞争代价与核心数成正比，而 RCU 的读侧代价是常数（仅禁抢占，无原子操作）。RCU 的本质是用"空间换时间"——保留旧版本数据供并发读者使用，同时写者在副本上操作——这是计算机科学中"版本控制"思想在内核并发中的极致应用。
第三纪元 · 扩展期 · 2015–2018
Linux 4.x — 网络命名空间与容器化
每个网络命名空间拥有独立的邻居表，容器网络隔离的基础
🐳 背景：Docker 与容器网络爆发

2013–2015 年，Docker 的普及使 Linux 网络命名空间（network namespace）从一个"存在但鲜少使用"的特性变成了基础设施的核心。每个容器拥有独立的网络栈，理应拥有独立的邻居表——但早期实现中，邻居表是全局的，这导致：

  • 同一主机上的两个容器可能共享 ARP 缓存，造成信息泄露
  • 容器删除时残留的邻居条目污染全局表
  • gc_thresh 参数无法按命名空间隔离配置
  • VXLAN/overlay 网络中，内外层 ARP 条目混杂
🔧 neigh_parms 的 per-device 配置

struct neigh_parms 允许为每个网络设备（dev）定制邻居行为参数，覆盖全局默认值：

struct neigh_parms {
    possible_net_t  net;                    /* 所属网络命名空间 */
    struct net_device *dev;                  /* 绑定设备，NULL=默认 */
    int reachable_time;                      /* 动态计算的可达时间 */
    int data[NEIGH_VAR_DATA_MAX];            /* 所有可调参数 */
    DECLARE_BITMAP(data_state, NEIGH_VAR_DATA_MAX); /* 哪些被显式设置 */
};

这使得管理员可以针对不同接口设置不同的 ARP/NDP 行为：

# 为 eth0 单独设置较短的 reachable time（高流量接口）
ip link set eth0 ... 
sysctl net.ipv4.neigh.eth0.base_reachable_time_ms=15000

# 为 eth1 设置更大的 gc_thresh（大规模 ARP 表）
sysctl net.ipv4.neigh.eth1.gc_thresh3=8192
🔬 第一性原理
隔离的本质是：不同租户的状态不应相互影响。网络命名空间为邻居子系统带来的隔离，是将"全局单例"分解为"每实例独立副本"的工程实践。其代价是每个命名空间需要独立维护一张表（内存开销），收益是完全的故障隔离和安全边界——这正是云计算多租户模型的物理基础。
第四纪元 · 云原生期 · 2018–2021
Linux 5.x — SDN 支持与外部学习条目
NTF_EXT_LEARNED、NTF_MANAGED：控制平面协同的新范式
🌩️ 背景：SDN 与 EVPN 时代

在 BGP EVPN（RFC 7432）、VXLAN Overlay、Kubernetes CNI 等场景中，邻居表条目不再仅由内核的 ARP/NDP 过程自动学习，而是由用户空间的控制平面（如 FRR 路由套件、Calico、Cilium）通过 rtnetlink 注入内核。

传统问题：控制平面注入的条目会被内核 GC 当作普通动态条目处理——在超时或 GC 触发时被删除，导致控制平面需要不断重新注入，造成振荡。

🏷️ NTF_EXT_LEARNED：外部学习标志

Roopa Prabhu（Cumulus Networks）在 2018 年提交了 NTF_EXT_LEARNED 支持（"neighbour: support for NTF_EXT_LEARNED flag"），专为 EVPN/SDN 场景设计：

/* 添加外部学习条目 */
ip neigh add 192.168.178.30 dev eth0 \
  lladdr aa:bb:cc:dd:ee:ff extern_learn

/* 内核 GC 跳过此类条目 */
/* net/core/neighbour.c: neigh_periodic_work() */
if ((state & (NUD_PERMANENT | NUD_IN_TIMER)) ||
    (n->flags & NTF_EXT_LEARNED)) {  /* ← 新增判断 */
    write_unlock(&n->lock);
    goto next_elt;  /* GC 不删除外部学习条目 */
}

这使得 BGP/EVPN 控制平面可以将远端 VTEP 学习到的 MAC/IP 映射注入内核，而不用担心被 GC 清除。

🤖 NTF_MANAGED：自动刷新的托管条目

进一步地，NTF_MANAGED（内核扩展标志）允许控制平面声明"此条目应保持 REACHABLE 状态"——内核会周期性地主动探测该条目，确保其始终有效，而无需控制平面轮询：

/* 托管条目：内核自动维护 REACHABLE 状态 */
ip neigh add 192.168.1.100 dev eth0 managed

/* 对应内核行为：
   - 条目不存在时：内核主动发起 ARP 解析
   - 条目 STALE 时：内核自动发起单播探测
   - 解析失败时：通知用户空间（via netlink）*/
🔔 邻居事件通知（Netlink 异步）

Linux 5.x 增强了邻居状态变化的异步通知机制。用户空间的控制平面可以通过 RTMGRP_NEIGH 多播组订阅邻居表事件，实时感知条目的创建、更新、删除和状态变化：

/* 用户空间监听邻居事件 */
nl_socket_add_memberships(sk, RTNLGRP_NEIGH);
// 收到事件：RTM_NEWNEIGH / RTM_DELNEIGH
// 包含：IP地址、MAC地址、NUD状态、接口索引

/* 典型事件流：
   INCOMPLETE → REACHABLE (ARP 解析完成)
   REACHABLE  → STALE     (超时)
   STALE      → DELAY     (有新流量)
   DELAY      → PROBE     (需要单播确认)
   PROBE      → FAILED    (无法联系)
   FAILED     → (删除)    */
🔬 第一性原理
在 SDN 时代，邻居发现不再是一个自包含的分布式协议，而成为了"控制平面意图"和"数据平面状态"之间的接口。NTF_EXT_LEARNED 的本质是：承认内核 ARP/NDP 的学习能力有边界，允许外部更高层知识（BGP EVPN、SDN 控制器）注入内核数据平面，而不是让内核重新发明 BGP。这是"单一职责原则"在系统设计层面的体现——内核只负责维护和转发，策略由外部控制平面决定。
第四纪元 · 云原生期 · 2022–2024
Linux 6.x — 现代化精炼
去除 rcu_bh、lockdep 注解、RFC 9131 支持、数据竞争修复
🧹 去除 rcu_bh：简化并发模型

Linux 6.1 中，Eric Dumazet 提交了 "neighbour: switch to standard rcu, instead of rcu_bh"（ChangeLog-6.1.57），移除了对 rcu_bh（底半部 RCU）的依赖：

📖 技术细节
rcu_bh 是一种特殊的 RCU 变体，它通过"禁止软中断"来定义宽限期。随着内核向 PREEMPT_RCU 演进，rcu_bh 不再比标准 rcu 更高效——尤其是邻居条目使用标准 call_rcu() 释放时，混用 rcu_bh 反而会带来不必要的底半部禁用开销。

同步提交 "neighbour: annotate lockless accesses to n->nud_state" 为所有无锁访问的 nud_state 字段加上 READ_ONCE()/WRITE_ONCE() 注解，使 KCSAN（内核并发检测器）能够检测数据竞争。

🌐 RFC 9131：更新 IPv6 邻居广告处理

RFC 9131（2021）更新了 RFC 4861 中关于接收未请求邻居广告（Unsolicited Neighbor Advertisement）的处理规则。Linux 6.x 在 net/ipv6/ndisc.c 中实现了这一更新：

/* RFC 9131: 路由器可以对无对应条目的 NA 创建 STALE 条目
 * 条件：accept_untracked_na=1 且本机配置为路由器
 *
 *   entry  accept  fwding  solicited   behavior
 *   ------  ------  ------  ---------  ----------------
 *   absent    1       1         X      Add new STALE entry ← 新增
 *   absent    1       0         X      Do nothing
 *   absent    0       X         X      Do nothing
 */
if (!neigh && lladdr && idev &&
    READ_ONCE(idev->cnf.forwarding)) {
    if (accept_untracked_na(dev, saddr)) {
        neigh = neigh_create(&nd_tbl, &msg->target, dev);
        new_state = NUD_STALE;
    }
}

此改进使路由器能够更积极地学习网络拓扑，减少后续通信建立时的延迟。

🛡️ NTF_EXT_VALIDATED：外部验证条目

Linux 6.x 引入了 NTF_EXT_VALIDATED（NDA_FLAGS_EXT 扩展标志），代表"由外部控制平面验证为有效"的条目：

/**
 * NTF_EXT_EXT_VALIDATED flagged neighbor entries were externally
 * validated by a user space control plane. The kernel will not
 * remove or invalidate them, but it can probe them and notify
 * user space when they become reachable.
 * The kernel will return the entry to stale state if it did not
 * receive a confirmation after probing the entry.
 */

与 NTF_EXT_LEARNED 的区别：

NTF_EXT_LEARNED

GC 不删除，内核不主动探测，适合"相信控制平面"的场景

NTF_EXT_VALIDATED

内核仍可探测并通知，控制平面可感知实际可达性变化，适合需要验证的场景

🔬 第一性原理
软件的演进总是在回应外部约束的变化：CPU 从单核变多核（→ RCU），网络从局域网变云原生（→ 命名空间、SDN 支持），协议从 IPv4 变双栈（→ 统一框架），规模从百台变万台（→ 动态哈希、GC 优化）。Linux 邻居子系统 30 年的演进史，本质上是一部对"环境假设改变"的持续适应史——每一次重构，都是当旧假设无法再解释新现实时的必然响应。
总结
🎯 第一性原理总结
从最基本的物理约束，推导 Linux 邻居子系统的所有设计决策
🧱 公理：物理世界的不可绕过约束

以太网帧必须携带目标 MAC 地址，这是物理层的硬性要求，无法更改。IP 协议工作在逻辑层，只认 IP 地址。因此：

不可变公理： 在共享链路上发送 IP 数据包之前，必须已知目标 IP 对应的链路层地址。
📐 推论一：必须有缓存

每次发包都广播查询 MAC 地址的代价是不可接受的（O(n×m) 网络开销，n=发包率，m=主机数）。因此必须维护一个本地缓存——这是 neigh_table 存在的根本原因。

📐 推论二：缓存必须有时效

网络是动态的，主机会宕机、迁移、更换网卡。永不过期的缓存会导致数据发往错误目标。因此必须有 TTL 和主动探测——这是 NUD 状态机的根本原因。

📐 推论三：探测必须节制

如果每个主机的每条缓存条目都频繁广播探测，网络会被 ARP 淹没（ARP Storm）。因此探测必须是单播优先、有退避策略的——这是 DELAY→PROBE 转换的根本原因。

📐 推论四：上层可以辅助

TCP ACK 本身就是"对方可达"的证明。利用上层确认信息来延长邻居条目有效期，比重新发 ARP 代价更低——这是 neigh_confirm() 的根本原因。

🏆 30 年演进的核心张力

Linux 邻居子系统的整个演进史，本质上是在三组张力之间持续寻找平衡：

正确性 vs 性能
加锁保证正确性，但代价是串行化；RCU 用更复杂的内存模型换取并发性能。两者的平衡点随 CPU 核心数的增加而不断移动。
主动发现 vs 被动等待
主动 ARP/NDP 探测能及时发现邻居失效，但会产生网络流量；被动等待流量触发不产生额外开销，但响应延迟高。NUD 的 DELAY 状态就是这一权衡的具体实现。
内核自治 vs 外部控制
传统 ARP 完全自治（无需人工干预）；现代 SDN 场景要求内核接受外部注入的 MAC/IP 映射。NTF_EXT_LEARNED 等机制是这一张力在 5.x 时代的解决方案。
内存效率 vs 响应速度
条目保留越久，后续通信越快（无需重新 ARP）；但内存有限，条目太多会耗尽 gc_thresh3。GC 机制是这一张力的动态调节器。
✅ 终极第一性原理

邻居子系统解决的根本问题是：在一个动态、异构、不可信的分布式环境中，以最小的全局代价，维护最新的局部通信路径信息。

所有 30 年的改进——从 NUD 状态机到 RCU，从命名空间隔离到 SDN 支持——都是在回答同一个问题的不同约束版本：当"动态"变得更动态（虚拟机迁移），当"异构"变得更异构（容器/VXLAN），当"不可信"变得更不可信（多租户），当"全局代价"的边界变了（多核/高速网络），系统如何演化以适应？
