什么是 Linux Bridge？

根据 Linux 内核官方文档，IEEE 802.1Q-2022 标准（桥接与桥接网络）定义了网络中桥接器的操作。 桥接器在本标准上下文中，是一种连接两个或多个网段并在 OSI 模型数据链路层（第二层）运行的设备。 桥接器的目的是根据目标 MAC 地址在不同网段之间过滤和转发帧。

🔍 官方定义
Bridge（网桥）以协议无关的方式连接两个以太网网段。数据包基于以太网地址（而非 IP 地址）进行转发，因此所有协议（IPX、NetBEUI、IP、IPv6 等）均可透明地通过桥接。 Linux bridge 代码实现了 ANSI/IEEE 802.1d 标准的子集。

来源：Linux Foundation Bridge Wiki

核心价值与定位

Linux 网桥不仅仅是纯硬件网桥的软件副本。其最重要的特征在于灵活性：

  • 可与完整的 netfilter（iptables/ebtables/nftables） 能力结合，实现桥接级防火墙
  • 支持作为虚拟化平台（KVM/QEMU/LXC/Docker）的 虚拟交换机
  • 通过 Switchdev/DSA 实现对硬件交换芯片的卸载
  • 是 Linux 网络命名空间隔离的重要组件
Linux Bridge 在网络栈中的位置

Bridge 子系统的本质是：将 IEEE 802.1d 所定义的"二层交换"能力在操作系统内核中以软件形式实现，并通过统一的 net_device 抽象对上层完全透明。其第一性原理是：网络转发的最小原语是"查表→决策→转发"三步，而所有附加功能（STP、VLAN、组播、硬件卸载）均是对这一原语的扩展与优化。

历史

发展历史时间线

Linux Bridge 的演进历史横跨近三十年，从一个简单的二层转发模块成长为支持硬件卸载、VLAN 感知、组播智能分发的完整交换子系统。 以下时间线基于 Linux Foundation Bridge Wiki、 内核官方文档及源码提交记录整理。

Linux 2.2 (1999)
原始实现：第一版桥接代码

Linux 2.2 内核中首次引入桥接功能。该版本实现较为基础，转发表使用 AVL 树存储，存在性能局限。 配置接口仅有 ioctl 方式。

⚠️ 局限性
不支持多网桥实例；AVL 树查找 O(log n) 性能有限；不支持 STP 生成树；无 VLAN 感知。
Linux 2.4 (2001)
Lennert Buytenhek 重写：现代桥接架构奠基

由 Lennert Buytenhek（buytenh@gnu.org）对桥接代码进行完全重写并集成至 2.4 内核。 这是现代 bridge 代码的直接祖先，至今所有文件仍保留 Buytenhek 的版权署名。

  • 转发表改为 哈希表（BR_HASH_SIZE），查找性能提升至 O(1)
  • 支持多桥实例共存
  • 引入 STP（802.1D） 内核实现
  • 引入 net_bridge、net_bridge_port、net_bridge_fdb_entry 三大核心数据结构
  • 通过 ioctl 配置（brctl 工具）
📄 br_private.h 源码
Linux 2.6 (2003)
ebtables 与 bridge-nf 集成

Lennert Buytenhek 与 Bart de Schuymer 合作将 bridge-nf 补丁合入 2.6 主线内核，实现了桥接与 netfilter 的融合：

  • 引入 ebtables（二层以太网帧过滤框架）
  • 引入 br_netfilter 模块，使 iptables 规则可作用于桥接流量
  • 引入 NF_BR_* 钩子点（PRE_ROUTING、LOCAL_IN、FORWARD、LOCAL_OUT、POST_ROUTING）
  • 通过 /proc/sys/net/bridge/bridge-nf-* 控制桥接过滤行为
Linux 2.6.18 (2006)
Netlink 接口原型引入

开始为桥接子系统添加 Netlink 配置接口原型，为后续取代 ioctl 奠定基础。sysfs 接口也在此阶段逐步完善。

Linux 2.6.36 (2010)
rx_handler 机制：关键解耦点

引入 netdev_rx_handler_register() 机制，将桥接的数据包接收逻辑与通用网络栈彻底解耦。 每个桥接端口通过注册独立的 rx_handler 回调（br_handle_frame）来捕获入站帧， 避免了对核心协议栈的侵入式修改。

✨ 设计突破
这是 bridge 架构史上最重要的解耦节点：桥接逻辑以"钩子"方式挂载到网络设备，而不是修改通用收包路径。
Linux 3.9 (2013)
VLAN 感知桥接（802.1Q VLAN Filtering）

引入 VLAN 过滤支持（commit 来自 Vlad Yasevich 等人）。 通过 vlan_filtering 开关，网桥可从"VLAN 透传"升级为"VLAN 感知"模式：

  • 每端口维护 VLAN 位图（net_bridge_vlan_group）
  • 支持 PVID（Port VLAN ID）和 Tagged/Untagged 出口策略
  • 引入 net_bridge_vlan 结构
📄 VLAN filtering 补丁
Linux 3.8 / 3.12 (2013-2014)
IGMP Snooping / MDB 组播嗅探

引入完整的 IGMP/MLD 嗅探功能，内核桥接可维护组播成员数据库（MDB），实现组播流量的精准分发而非洪泛广播。支持 IGMPv2/IGMPv3 与 MLDv1/MLDv2。

Linux 4.4 (2016)
Switchdev 框架：软硬件解耦

引入 Switchdev 框架（由 Jiri Pirko 与 Scott Feldman 设计），允许将桥接转发平面卸载到硬件交换芯片。 通过 switchdev_notifier 通知链，FDB 条目可双向同步于软件表和硬件表。

📄 Switchdev 官方文档
Linux 4.9 (2016)
SR-IOV eSwitch 与 Switchdev 深度整合

MLX5（Mellanox）等厂商完成 Switchdev 集成，SR-IOV 虚函数（VF）代表接口可作为桥接端口，实现 VF 之间的零拷贝硬件转发。

Linux 5.x / 6.x (2019-现在)
802.1ad QinQ、MSTP、EHT、MAB、802.1X

持续演进：

  • 802.1ad（QinQ）：双层 VLAN 标签支持
  • per-VLAN 组播嗅探：每 VLAN 独立的 MDB
  • IGMPv3/MLDv2 EHT（显式主机追踪）：精准组播离开延迟控制
  • 802.1X 端口认证与 MAC Authentication Bypass（MAB）
  • MSTP（多生成树）：802.1Q MST 实例支持
  • MRP（媒体冗余协议）支持
  • per-VLAN per-port 统计计数
📄 最新内核桥接文档
Bridge 子系统演进时间轴

Bridge 子系统的演进遵循一条清晰的第一性原理：每次架构突破都来自于找到并消除一个新的耦合点。 从 ioctl→Netlink 是控制面与内核实现的解耦；rx_handler 是数据面与协议栈的解耦；Switchdev 是软件实现与硬件实现的解耦； VLAN filtering 是转发域与物理端口的解耦。解耦不是目的，而是使系统在不修改既有代码的前提下支持新场景的手段。

架构

抽象层次与核心结构

Linux Bridge 的架构设计精髓在于将"逻辑网桥"、"桥端口"、"转发条目"这三个概念抽象为独立的数据结构， 并通过 net_device 统一接口对外呈现。这种设计使得桥接器对上层应用和用户工具完全透明。

核心数据结构

1. struct net_bridge — 逻辑桥实例

每个桥实例对应一个 net_bridge 结构体，它作为 net_device 的私有数据（netdev_priv(dev)）存在：

/* net/bridge/br_private.h (来源：https://codebrowser.dev/linux/linux/net/bridge/br_private.h.html) */
struct net_bridge {
    spinlock_t              lock;
    spinlock_t              hash_lock;
    struct hlist_head       frame_type_list;
    struct net_device       *dev;           /* 桥的虚拟网络设备 */
    unsigned long           options;        /* 桥选项位图 */
    /* STP 相关 */
    bridge_id               designated_root;
    bridge_id               bridge_id;
    unsigned char           topology_change;
    unsigned char           topology_change_detected;
    struct timer_list       hello_timer;
    struct timer_list       tcn_timer;
    struct timer_list       topology_change_timer;
    struct delayed_work     gc_work;        /* FDB 老化垃圾回收 */
    /* 端口列表 */
    struct list_head        port_list;
    /* 转发表 */
    struct hlist_head       hash[BR_HASH_SIZE];  /* FDB 哈希表 */
    /* VLAN 相关 */
    struct net_bridge_vlan_group __rcu *vlgrp;
    /* 组播相关 */
    struct net_bridge_mcast         multicast_ctx;
};

2. struct net_bridge_port — 桥端口

每个加入桥的网络设备（端口）对应一个 net_bridge_port 结构：

struct net_bridge_port {
    struct net_bridge       *br;        /* 指回所属桥实例 */
    struct net_device       *dev;       /* 该端口对应的物理/虚拟网络设备 */
    struct list_head        list;       /* 链接在 br->port_list 中 */
    unsigned char           priority;
    unsigned char           state;      /* STP 状态：BLOCKING/LEARNING/FORWARDING等 */
    u16                     port_no;
    unsigned long           flags;      /* 端口标志：BR_LEARNING|BR_FLOOD 等 */
    /* VLAN 组 */
    struct net_bridge_vlan_group __rcu *vlgrp;
    /* 组播上下文 */
    struct net_bridge_mcast_port        multicast_ctx;
    struct kobject          kobj;       /* sysfs 节点 */
};

3. struct net_bridge_fdb_entry — 转发表条目

FDB（Forwarding Data Base）是网桥的"记忆"，记录 MAC 地址到端口的映射：

struct net_bridge_fdb_entry {
    struct net_bridge_port __rcu *dst;   /* 目标端口（NULL 表示本地） */
    struct net_bridge_fdb_key    key;    /* {MAC地址, VLAN ID} 复合键 */
    struct hlist_node            fdb_node;
    unsigned long                flags;  /* 永久/本地/卸载等标志 */
    unsigned long                used;   /* 最后使用时间（老化用） */
    unsigned long                updated;
    struct rhash_head            rhnode; /* rhashtable 成员 */
    struct rcu_head              rcu;
};

4. struct net_bridge_vlan — per-VLAN 条目（内核 3.9+）

根据 官方文档：

struct net_bridge_vlan {
    struct rhash_head               vnode;   /* 全局桥 VLAN 哈希 */
    struct rhash_head               tnode;   /* 隧道哈希 */
    u16 vid;                                 /* VLAN ID */
    u16 flags;                               /* MASTER/PVID/UNTAGGED 等 */
    u16 priv_flags;
    u8  state;                               /* STP 状态（per-VLAN） */
    struct pcpu_sw_netstats __percpu *stats; /* per-CPU VLAN 统计 */
    union {
        struct net_bridge       *br;         /* MASTER 标志：指向桥 */
        struct net_bridge_port  *port;       /* 非MASTER：指向端口 */
    };
    union {
        refcount_t refcnt;                   /* MASTER：引用计数 */
        struct net_bridge_vlan  *brvlan;     /* 非MASTER：指向全局VLAN */
    };
    struct br_tunnel_info           tinfo;
    union {
        struct net_bridge_mcast         br_mcast_ctx;   /* 桥级组播上下文 */
        struct net_bridge_mcast_port    port_mcast_ctx; /* 端口级组播上下文 */
    };
    u16 msti;                                /* MST 实例（MSTP） */
    struct list_head                vlist;
    struct rcu_head                 rcu;
};

核心抽象设计遵循单一职责原则：net_bridge 是"逻辑域"，net_bridge_port 是"物理连接点"，net_bridge_fdb_entry 是"转发决策依据"，net_bridge_vlan 是"隔离边界"。这四层抽象对应现实中的"交换机"、"物理端口"、"MAC 地址表"、"VLAN"四个概念，实现了一比一的直觉映射，使代码可读性和可维护性达到最优。

数据面

数据包接收路径

数据包接收路径是 Bridge 子系统中最关键的设计，它体现了"如何在不修改网络核心代码的前提下，将数据包劫持到桥接逻辑"的精妙设计。

rx_handler：解耦神器

自 Linux 2.6.36 起，netdev_rx_handler_register() 成为实现端口捕获的标准机制。 当一个网络设备被加入桥时，br_add_if() 会为其注册 br_handle_frame 作为 rx_handler： 

/* net/bridge/br_if.c */
int br_add_if(struct net_bridge *br, struct net_device *dev,
              struct netlink_ext_ack *extack)
{
    struct net_bridge_port *p;
    /* ... 前置检查 ... */

    /* 不允许桥接非以太网设备 */
    if ((dev->flags & IFF_LOOPBACK) ||
        dev->type != ARPHRD_ETHER || ...)
        return -EINVAL;

    /* 不允许桥套桥 */
    if (dev->netdev_ops->ndo_start_xmit == br_dev_xmit) {
        NL_SET_ERR_MSG(extack, "Can not enslave a bridge to a bridge");
        return -ELOOP;
    }

    p = new_nbp(br, dev);  /* 创建 net_bridge_port */

    /* 【关键】注册 rx_handler，接管该设备的入站数据包 */
    err = netdev_rx_handler_register(dev, br_get_rx_handler(dev), p);

    dev->priv_flags |= IFF_BRIDGE_PORT;

    /* 建立 upper/lower 设备关系链 */
    err = netdev_master_upper_dev_link(dev, br->dev, NULL, NULL, extack);

    list_add_rcu(&p->list, &br->port_list);
    /* ... */
}
🔍 rx_handler 机制
在 __netif_receive_skb_core() 中，如果设备注册了 rx_handler，则在所有 ptype 处理之前先调用它。 rx_handler 返回 RX_HANDLER_CONSUMED 表示数据包已被处理（不再向上传递），或 RX_HANDLER_PASS 表示继续正常流程。

数据包接收完整流程

/* net/bridge/br_input.c — br_handle_frame() */
rx_handler_result_t br_handle_frame(struct sk_buff **pskb)
{
    struct net_bridge_port *p = br_port_get_rcu(skb->dev);

    /* 1. 过滤非以太网类型帧 */
    if (!is_valid_ether_addr(eth_hdr(skb)->h_source))
        goto drop;

    /* 2. 处理 STP BPDU（01:80:c2:00:00:00-0f 链路本地地址） */
    if (unlikely(is_link_local_ether_addr(dest))) {
        /* 转发给 STP/LLC 处理 */
        return RX_HANDLER_PASS;
    }

    /* 3. 检查端口 STP 状态 */
    switch (p->state) {
    case BR_STATE_FORWARDING:
    case BR_STATE_LEARNING:
        break;
    case BR_STATE_DISABLED:
    case BR_STATE_BLOCKING:
        goto drop;
    }

    /* 4. 进入帧处理主逻辑 */
    return nf_hook_bridge_pre(skb, pskb);  /* 调用 NF_BR_PRE_ROUTING 钩子 */
}
/* br_handle_frame_finish() — 核心转发决策 */
int br_handle_frame_finish(struct net *net, struct sock *sk, struct sk_buff *skb)
{
    struct net_bridge_port *p = br_port_get_rcu(skb->dev);
    struct net_bridge *br = p->br;
    enum br_pkt_type pkt_type = BR_PKT_UNICAST;
    struct net_bridge_fdb_entry *dst = NULL;
    u16 vid = 0;

    /* 5. VLAN 处理（若开启 vlan_filtering） */
    if (!br_allowed_ingress(br, nbp_vlan_group_rcu(p), skb, &vid, ...))
        goto out;  /* VLAN 过滤丢弃 */

    /* 6. MAC 地址学习 */
    if (p->flags & BR_LEARNING)
        br_fdb_update(br, p, eth_hdr(skb)->h_source, vid, 0);

    /* 7. 组播处理 */
    if (is_multicast_ether_addr(eth_hdr(skb)->h_dest)) {
        /* IGMP/MLD 嗅探 + MDB 查找 */
        pkt_type = BR_PKT_MULTICAST;
        /* ... */
    }

    /* 8. 单播 FDB 查找 */
    if (pkt_type == BR_PKT_UNICAST)
        dst = br_fdb_find_rcu(br, eth_hdr(skb)->h_dest, vid);

    /* 9. 转发决策 */
    if (dst) {
        if (test_bit(BR_FDB_LOCAL, &dst->flags))
            return br_pass_frame_up(skb, false);  /* 本地接收 */
        br_forward(dst->dst, skb, local_rcv, false);  /* 单播转发 */
    } else {
        br_flood(br, skb, pkt_type, local_rcv, false, vid);  /* 洪泛 */
    }
    /* ... */
}
数据包在 Bridge 中的完整流转路径

数据包接收路径体现了"最早过滤，最晚决策"的第一性原理：STP 状态过滤（硬规则）→ Netfilter 钩子（可插拔规则）→ VLAN 过滤（域隔离规则）→ MAC 学习（副作用）→ FDB 查找（决策）→ 转发/洪泛（执行）。这种流水线设计使得无效帧尽早丢弃，减少无谓开销，同时钩子点的设置使各功能模块可独立插拔，互不干扰。

核心机制

FDB 转发数据库

FDB（Forwarding Data Base）是网桥的"大脑"，存储了 {MAC 地址, VLAN ID} → 端口 的映射关系。 它是桥接器实现"学习"和"智能转发"（区别于集线器的洪泛转发）的核心。

数据结构演进

版本FDB 存储结构查找复杂度特点
Linux 2.2 AVL 树 O(log n) 有序，但查找较慢
Linux 2.4 固定桶哈希表（BR_HASH_SIZE=256） O(1) 平均 快速，链表解决冲突
Linux 5.x+ rhashtable（可伸缩哈希表） O(1) 平均，自动扩缩容 RCU 友好，支持并发读

FDB 的三种条目类型

📋 条目分类
  • 动态学习条目（Dynamic）：由桥自动从入站帧的源 MAC 地址学习，有老化时间（默认 300 秒），到期后自动删除。
  • 静态条目（Static/Permanent）：由用户通过 bridge fdb add 命令手动添加，不会老化。
  • 本地条目（Local）：桥自身端口的 MAC 地址，用于识别发往本机的数据包，触发 br_pass_frame_up()。

MAC 地址学习流程

/* net/bridge/br_fdb.c */
void br_fdb_update(struct net_bridge *br, struct net_bridge_port *source,
                   const unsigned char *addr, u16 vid, unsigned long flags)
{
    struct net_bridge_fdb_entry *fdb;

    /* RCU 读侧查找 */
    fdb = fdb_find_rcu(&br->fdb_hash_tbl, addr, vid);

    if (likely(fdb)) {
        /* 条目已存在：更新使用时间（防止老化），可能更新端口（移动检测） */
        if (READ_ONCE(fdb->dst) != source) {
            /* MAC 地址在新端口出现：更新转发目标 */
            WRITE_ONCE(fdb->dst, source);
        }
        WRITE_ONCE(fdb->used, jiffies);
    } else {
        /* 新 MAC：创建新条目 */
        fdb = fdb_create(br, source, addr, vid, flags);
        /* 通知 switchdev（若有硬件卸载） */
        br_switchdev_fdb_notify(br, fdb, RTM_NEWNEIGH);
    }
}

FDB 老化机制

FDB 使用延迟工作队列（delayed_work → br->gc_work）定期扫描表项， 将超过 ageing_time（默认 300s）未使用的动态条目删除： 

/* br_fdb.c — gc_work 回调 */
static void br_fdb_cleanup(struct work_struct *work)
{
    struct net_bridge *br = container_of(work, struct net_bridge, gc_work.work);
    /* 遍历 rhashtable，删除过期条目 */
    rhashtable_walk_enter(&br->fdb_hash_tbl, &iter);
    while ((fdb = rhashtable_walk_next(&iter)) != NULL) {
        if (!test_bit(BR_FDB_STATIC, &fdb->flags) &&
            time_after(jiffies, fdb->used + hold_time(br))) {
            fdb_delete(br, fdb, true);
        }
    }
    /* 调度下次 GC */
    mod_delayed_work(system_long_wq, &br->gc_work, HZ/10);
}
FDB 学习与老化流程

FDB 与 Switchdev 协同

当桥接端口被硬件 switchdev 驱动接管时，FDB 条目的学习和通知流程被扩展：

  • 硬件学习到新 MAC → switchdev 驱动调用 call_switchdev_notifiers(SWITCHDEV_FDB_ADD, ...)
  • 桥接收通知 → 在软件 FDB 中创建条目，标记 NTF_EXT_LEARNED（硬件已学习，软件不老化）
  • 软件 FDB 新增静态条目 → 桥向 switchdev 驱动发送 SWITCHDEV_FDB_ADD_TO_DEVICE 通知 → 驱动将条目下发到硬件 ASIC

FDB 的核心第一性原理是：网桥通过"观察"而非"控制"来学习拓扑。MAC 学习是接收数据包的副作用，而非主动探测。这种被动学习机制使得网桥无需任何配置即可工作，天然实现了"即插即用"。老化机制则解决了"主机移动"问题——当一台主机迁移到另一端口，旧条目会超时失效，新条目通过学习重建，整个过程对应用完全透明。

STP 生成树协议

根据 内核官方文档， STP（Spanning Tree Protocol）是 Linux 桥接驱动中防止环路和广播风暴的关键功能，通过识别和禁用冗余链路来维护网络稳定性。

端口 STP 状态机

官方文档定义了以下端口状态：

状态值状态名行为
0DISABLED完全停用，不参与 STP，不转发数据帧
1LISTENING监听 BPDU，丢弃所有其他流量帧
2LEARNING接受流量仅用于学习 MAC 地址，不转发数据帧
3FORWARDING完全活跃，转发 BPDU 和数据帧
4BLOCKINGSTP 选举过程中，仅处理 STP BPDU
STP 端口状态机转换图

内核 STP vs 用户空间 STP

Linux bridge 支持两种 STP 实现模式，通过 IFLA_BR_STP_MODE 或 /sbin/bridge-stp 控制：

🔧 BR_STP_MODE_AUTO（默认）
内核调用 /sbin/bridge-stp <bridge> start。若返回 0，则启用用户空间 STP（如 mstpd）；否则启用内核 STP。 仅限初始网络命名空间。
✅ BR_STP_MODE_KERNEL
直接启用内核 STP（BR_KERNEL_STP），支持 IEEE 802.1D 基础 STP。适用于简单场景。
⚙️ BR_STP_MODE_USER
直接启用用户空间 STP（BR_USER_STP），不调用 helper。适用于需要 RSTP/MSTP 的场景（配合 mstpd）。 可在任意网络命名空间使用（内核 6.x 新增）。

配置示例：

# 启用内核 STP（基础 802.1D）
ip link set dev br0 type bridge stp_state 1

# 启用用户空间 STP（RSTP/MSTP，配合 mstpd）
ip link set dev br0 type bridge stp_mode user stp_state 1

STP 根桥选举过程

根据官方文档描述的选举流程：

  1. 所有桥初始假设自己是根桥，向外发送 BPDU（含 Bridge ID = Priority + MAC）
  2. 收到 BPDU 后比较 Bridge ID，最低 Bridge ID 的桥成为根桥
  3. 根桥确定后，其他桥计算到根的最短路径（根据 Path Cost）
  4. 每个非根桥选出一个根端口（到根路径最短）和若干指定端口
  5. 其余端口进入 BLOCKING 状态，防止环路

STP 体现的第一性原理是：在动态拓扑中，任意两点之间的通信只需一条无环路径。STP 通过分布式共识（BPDU 交换）找到生成树，将图（可能有环）变为树（无环）。内核 STP vs 用户空间 STP 的分离则体现了另一个原理：策略（路径选择算法）应与机制（端口状态控制）分离——机制在内核，策略可以在用户空间由更智能的 daemon 实现（如 RSTP/MSTP），两者通过状态接口（IFLA_BRPORT_STATE）解耦。

VLAN 过滤子系统

根据 官方文档， VLAN 过滤于 Linux 3.9 引入，可通过 IFLA_BR_VLAN_FILTERING 属性开启。 启用后，桥将根据目标 MAC 地址和 VLAN 标签共同做转发决策。

VLAN 感知 vs VLAN 透传

模式vlan_filtering行为适用场景
VLAN 透传 0（默认） 桥完全忽略 VLAN 标签，原样转发带标签或不带标签的帧 简单桥接，不需要 VLAN 隔离
VLAN 感知 1 严格按 VLAN ID 进行入口检查，不在白名单的 VID 丢弃 企业交换机模拟，容器/VM 隔离

核心 VLAN 概念

  • PVID（Port VLAN ID）：端口的默认 VLAN，未标记帧进入时添加此 VID。设置方法：bridge vlan add vid 10 dev eth0 pvid
  • Egress Untagged：帧从该端口发出时剥掉 VLAN 标签。对应 Access 口。
  • Egress Tagged：帧保留 VLAN 标签发出。对应 Trunk 口。

VLAN 数据结构设计

net_bridge_vlan 结构被桥级和端口级共用（通过 MASTER flag 区分）， 这是一个精妙的设计——避免了重复存储，通过 union + refcount 实现了两者的关联：

VLAN 数据结构共享设计

VLAN 过滤流程（数据面）

/* net/bridge/br_vlan.c */
/* 入口 VLAN 检查：br_allowed_ingress() */
bool br_allowed_ingress(const struct net_bridge *br,
                        struct net_bridge_vlan_group *vg,
                        struct sk_buff *skb, u16 *vid, ...)
{
    /* 1. 提取/确定 VID */
    if (!br_vlan_get_tag(skb, vid)) {
        /* 无 VLAN 标签：使用 PVID */
        if (pvid) {
            *vid = pvid;
            /* 在 skb 上打标（内部） */
        } else {
            return false;  /* 无 PVID 配置时丢弃无标签帧 */
        }
    }

    /* 2. 在端口 VLAN 组中查找该 VID */
    v = br_vlan_find(vg, *vid);
    if (!v || !br_vlan_should_use(v))
        return false;  /* VID 不在白名单 → 丢弃 */

    return true;
}

/* 出口 VLAN 处理：br_allowed_egress() + br_vlan_untag() */
/* 若端口对该 VID 标记为 UNTAGGED，则去除 VLAN 标签再发出 */

协议支持

  • IEEE 802.1Q：标准 VLAN，默认协议（EtherType 0x8100）
  • IEEE 802.1ad（QinQ）：双层 VLAN 栈（EtherType 0x88A8），通过 IFLA_BR_VLAN_PROTOCOL 配置
  • per-VLAN STP 状态：net_bridge_vlan.state 存储每 VLAN 独立的 STP 状态
  • per-VLAN per-port 统计：通过 pcpu_sw_netstats 实现 per-CPU 原子统计

VLAN 子系统的第一性原理是：隔离域的引入不能破坏已有功能，必须以"零侵入"方式叠加。因此 VLAN filtering 默认关闭，开启时通过在转发路径的入口和出口插入检查点实现，而不修改 FDB 学习、STP、组播等独立模块。net_bridge_vlan 结构的桥级/端口级共用设计，则体现了"共享不变量（VID 语义），隔离可变量（端口行为）"的原则。

组播嗅探 MDB

根据 官方文档， Linux bridge 驱动支持完整的组播功能，可处理 IGMP/MLD 消息并精准转发组播数据包， 避免将组播流量洪泛到所有端口。创建时默认启用组播嗅探。

核心机制：MDB（Multicast Forwarding Database）

MDB 维护 {组播组 IP/MAC, VLAN ID} → 端口集合 的映射，是实现精准组播转发的关键。 与 FDB 不同，MDB 的映射是"一对多"的（一个组播地址可对应多个端口）。

IGMP Snooping 工作流程

协议支持矩阵

协议版本支持状态说明
IGMPv2✅ 完整支持默认版本，简单 join/leave
IGMPv3✅ 完整支持支持 Source-Specific Multicast（SSM）
MLDv1✅ 完整支持IPv6 组播，类似 IGMPv2
MLDv2✅ 完整支持IPv6 SSM 支持

IGMPv3/MLDv2 EHT 显式主机追踪

根据官方文档，EHT（Explicit Host Tracking）由 commit 474ddb37fa3a 引入。 它使桥能够追踪每个加入特定组/信道的独立主机，实现最短离开延迟：

  • 当某组的最后一个主机发送 Leave，桥可立即停止该组的流量转发
  • 无需等待 Last Member Query Interval（默认 1 秒）
  • 端口 EHT 主机数限制：IFLA_BRPORT_MCAST_EHT_HOSTS_LIMIT（默认 512）

per-VLAN 组播嗅探

由 Nikolay Aleksandrov 引入， 允许每个 VLAN 维护独立的 MDB 和查询器，实现更精细的多租户场景下的组播控制： 

# 为 br0 上的 VLAN 10 单独开启组播嗅探和查询器
bridge vlan set vid 10 dev br0 mcast_snooping 1 mcast_querier 1

组播嗅探的第一性原理是：将网络交换机从"不理解内容"升级为"理解内容意图"。传统交换机将组播当广播处理（因为 MAC 层没有订阅概念），IGMP snooping 则让交换机"偷听"上层协议的组播订阅信号（IGMP/MLD），从而在不修改端到端协议的前提下实现精准分发。这是经典的"中间层理解上层语义以优化传输"模式。

Netfilter 集成

根据 官方文档， bridge netfilter 模块（br_netfilter）是一个遗留功能，允许使用 iptables/ip6tables 过滤桥接数据包。 官方不建议继续使用，推荐使用 nftables。

Bridge Netfilter 钩子点

桥接框架定义了完整的 NF 钩子链，类比 IP 层的 netfilter 钩子：

Bridge Netfilter 钩子点（NF_BR_*）

br_netfilter 的工作原理

根据官方文档的详细描述：

  • br_netfilter 在桥接层拦截进入桥的数据包
  • 对 IPv4/IPv6 数据包做基础合法性检测
  • 然后伪装这些数据包"好像是被路由的"，而非桥接的
  • 从桥接层调用 ip/ip6 netfilter 钩子，使 iptables 规则集也能"看到"这些包
  • 这就是为何 iptables 需要 physdev match 才能区分路由和桥接包

推荐的现代替代方案

功能需求遗留方案现代推荐
二层帧过滤ebtablesnftables（bridge family）
三层流量过滤（桥接场景）iptables + br_netfilternftables（无需 br_netfilter）
ARP 过滤arptablesnftables（arp family）
连接追踪（conntrack）iptables + br_netfilternftables（内建 conntrack 支持）
⚠️ 重要说明
ebtables 和 nftables 无需 br_netfilter 模块即可正常工作。 iptables/ip6tables/arptables 对桥接流量不起作用（它们挂载在路由栈）。 nftables 的 ip/ip6/inet/arp family 规则也看不到桥接转发流量——但这是正确的行为，因为桥接本质上不经过路由层。

Netfilter 集成体现了内核设计中"向后兼容的代价"这一第一性原理：br_netfilter 是在已有 iptables 生态和新桥接子系统之间的"适配层"，它用伪装技术解决了"两套不同钩子体系之间的语义鸿沟"。这种适配层短期有效，长期则成为技术债务。nftables 的设计从一开始就考虑了桥接场景，使适配层不再必要——这正是正确的演进方向：新基础设施应从第一性原理出发，而非在旧基础上打补丁。

用户态接口演进

Bridge 子系统的用户态接口经历了三个时代：ioctl → sysfs → Netlink，这一演进过程完美体现了 "控制面与数据面分离"以及"接口标准化"的设计哲学。

第一代：ioctl（brctl 时代）

最早的 bridge 配置接口通过 ioctl() 系统调用实现，工具为 brctl（bridge-utils 包）： 

# brctl 基本命令（ioctl 接口）
brctl addbr br0                      # 创建网桥
brctl addif br0 eth0                 # 添加端口
brctl stp br0 on                     # 开启 STP
brctl setageing br0 300              # 设置老化时间
brctl showmacs br0                   # 查看 FDB
brctl showstp br0                    # 查看 STP 状态
⚠️ ioctl 接口的局限性
  • 功能有限，无法配置 VLAN、组播等高级特性
  • 不支持事件通知（无法监听拓扑变化）
  • 二进制协议，不易扩展
  • 与网络命名空间（network namespace）配合困难

第二代：sysfs（过渡期）

随着 IGMP Snooping 等功能的加入，通过 sysfs 暴露了大量配置参数。 官方文档明确指出：sysfs 接口已废弃，不应再扩展新选项。 

# sysfs 配置示例（已废弃）
echo 1 > /sys/class/net/br0/bridge/multicast_snooping
echo 1 > /sys/class/net/br0/bridge/vlan_filtering
cat /sys/class/net/eth0/brport/state     # 查看端口 STP 状态

第三代：Netlink（现代标准）

现代 bridge 配置通过 Netlink 接口（RTM_NEWLINK/RTM_SETLINK 等）实现， 工具为 iproute2 中的 ip link 和 bridge 命令。

Bridge 用户态接口演进对比

接口演进的第一性原理是：接口的设计应与实现解耦，且应能随需求增长而演进而不破坏已有用户。ioctl 的固定结构无法在不改变接口的情况下新增属性；sysfs 虽解决了扩展性问题，但每个属性一个文件导致大量系统调用且无原子性保证；Netlink 的 TLV（Type-Length-Value）编码天然支持前向兼容扩展，且同一消息可携带完整配置，是正确的终局接口设计。

Switchdev 硬件卸载

根据 Switchdev 官方文档（版权 © 2014 Jiri Pirko、Scott Feldman）， Switchdev 是内核中用于支持可将转发（数据）平面卸载到硬件的交换设备驱动模型。

🎯 核心目标
将 L2 数据转发路径从内核卸载到 switchdev 设备，通过将桥接 FDB 条目镜像到设备来实现。 一个 FDB 条目即 {端口, MAC, VLAN} 三元组转发目标。

Switchdev 架构

Switchdev 软硬件协同架构（来源：官方文档 Figure 1）

FDB 双向同步机制

方向1：软件 FDB → 硬件（静态条目下发）

/* 用户添加静态 FDB 条目 */
bridge fdb add AA:BB:CC:DD:EE:FF dev sw1p1 master static

/* 桥生成通知 → 驱动处理 */
SWITCHDEV_FDB_ADD_TO_DEVICE

/* 驱动将条目写入 ASIC 硬件表 */
asic_write_fdb_entry(port=1, mac=AA:BB:..., vid=0);

方向2：硬件学习 → 软件 FDB（动态同步）

/* 硬件在 p1 口学到新 MAC，驱动通知内核 */

/* info = { mac=AA:BB..., vid=100, offloaded=true } */

/* 桥接收通知，在软件 FDB 创建条目 */
br_fdb_update(br, p, mac, vid, BR_FDB_ADDED_BY_EXT_LEARN);
/* 标记 NTF_EXT_LEARNED：软件不老化，由硬件负责老化 */

/* iproute2 显示 "offload" 标志 */
# bridge fdb
AA:BB:CC:DD:EE:FF dev sw1p1 master br0 offload

VLAN 添加/删除SWITCHDEV_OBJ_ID_PORT_VLAN将桥 VLAN 配置同步到硬件
IGMP 组播组SWITCHDEV_OBJ_ID_PORT_MDB将 MDB 条目同步到硬件

Switchdev 的第一性原理是：软件和硬件共享"同一个转发模型的两个实现"，而非软件控制硬件这种主从关系。FDB 双向同步（软件→硬件、硬件→软件）保证了两个实现的一致性，而 NTF_EXT_LEARNED 标记则优雅地划定了"谁负责老化"的责任边界。这种"双镜像+事件驱动同步"的设计使得硬件加速对上层完全透明，应用程序无需感知转发在哪里发生。

DSA 分布式交换架构

根据 DSA 官方文档， DSA（Distributed Switch Architecture）是一个主要为嵌入式 Marvell 以太网交换芯片设计的子系统， 后来扩展支持其他厂商。其核心哲学是： 使用未经修改的 Linux 工具（bridge、iproute2、ifconfig）透明地工作， 无论配置的是交换机端口网络设备还是普通网络设备。

DSA vs Switchdev 对比

DSA Tag 协议：关键创新

DSA 使用特殊的帧标签协议在 CPU 与交换芯片之间传递元数据（端口号、VLAN、优先级等）。 常见实现：

  • Marvell DSA：在以太网帧头部插入 4 字节专有 DSA 标签
  • 802.1Q：复用标准 VLAN 标签传递端口信息
  • Ocelot：使用 NPI（Node Processor Interface）标签
  • Seville（Microchip）：使用专有标签格式
DSA 整体架构（来源：官方文档）

DSA 的第一性原理与 Switchdev 相同，但在一个更受限的环境下实现：嵌入式交换芯片通过专有 DSA 标签协议与 CPU 通信，这一"带内控制信道"使得标准 netdev 接口在物理上只有一条链路（CPU 端口）的情况下，逻辑上表现为多个独立端口。DSA 框架的核心贡献是将这一标签的注入/提取过程标准化，使得各厂商驱动可以专注于 ASIC 操作，而无需重复实现与 Linux bridge 的集成逻辑。

解耦与分离原则

Linux Bridge 子系统在漫长的演进过程中形成了一套清晰的解耦哲学。 理解这些解耦点，有助于深入理解为何这个子系统能在不断变化的需求下保持架构清晰。

六大解耦维度

Bridge 子系统解耦全景图

解耦机制对应表

解耦层次问题解耦机制效果
控制面 配置工具与内核实现绑定 Netlink TLV 接口标准化 brctl、iproute2、nmcli 均可独立演进

关键设计模式

Observer 模式（观察者）

Bridge 通过 register_netdevice_notifier() 订阅网络设备事件（NETDEV_UP/DOWN/CHANGEADDR 等）。 当端口链路状态变化时，通知链驱动 STP 状态机重新计算，而无需轮询。

Mediator 模式（调停者）

Bridge 核心充当 Mediator：端口之间不直接通信，而是通过桥的 FDB 查找和转发逻辑进行协调。 Switchdev 扩展了这一模式——硬件也通过 switchdev notifier 与桥（Mediator）通信，而不是绕过它直接操作其他端口。

Strategy 模式（策略）

STP 的内核/用户空间分离是 Strategy 模式的典型应用：接口（端口状态控制）固定，算法（STP/RSTP/MSTP）可替换。 同样，br_get_rx_handler(dev) 根据设备类型（普通端口 vs 其他）返回不同的 rx_handler 函数，也是 Strategy 模式。

Bridge 子系统解耦的终极第一性原理是：找到系统中变化速度不同的部分，在它们之间建立接口，使得各自可以独立演进而不相互破坏。硬件技术日新月异（Switchdev 应对），用户需求多样（Netlink 扩展），网络规模扩大（VLAN/组播精细化），安全要求提升（Netfilter/nftables）——这些都是不同速度的变化。接口（interface）的价值不在于今天的功能，而在于为明天的变化留出空间。

总结

演进路径总结

站在今天回望 Linux Bridge 子系统近三十年的演进，可以清晰地看到几条贯穿始终的设计主线。

演进主线一：从单体到模块化

早期的桥接代码是一个相对单体的实现。随着时间推移，不同关注点（STP、VLAN、组播、Netfilter）被抽取为独立模块， 通过清晰的内部接口（钩子、回调、通知链）相互协作。

Bridge 子系统模块化演进

演进主线二：从软件交换到软硬协同

最初的 Bridge 是纯软件实现，性能受 CPU 限制。Switchdev 和 DSA 的引入使得：

  • 控制面仍然在 Linux 内核（STP、VLAN、MDB 数据库）
  • 数据面可选择由硬件 ASIC 接管（线速转发，零 CPU 开销）
  • 两者之间通过事件通知机制保持一致性

演进主线三：接口标准化

从 ioctl 到 Netlink 的迁移代表了"接口质量"的系统性提升。 Netlink 的 TLV 编码不仅允许向后兼容扩展，还支持：

  • 原子操作：在一条 Netlink 消息中完成多属性配置
  • 事件订阅：用户空间通过 Netlink socket 订阅 FDB 变更、端口状态变更等事件
  • 命名空间隔离：不同网络命名空间的桥接配置完全隔离
  • 向后兼容：旧工具可忽略新属性，新内核可处理旧格式消息

未来方向

根据现有技术趋势和社区讨论，Bridge 子系统的未来演进方向包括：

  • MSTP 深度支持：Multiple Spanning Tree 在多 VLAN 环境中的进一步优化
  • P4-based 卸载：通过可编程数据面（P4）实现更灵活的硬件卸载
  • eBPF 集成：在桥接路径上支持 eBPF 程序，实现自定义转发逻辑
  • EVPN/VXLAN 深化：进一步完善 VLAN-aware bridge + VXLAN 隧道的集成
  • Switchdev/DSA 融合：官方文档已提出两者代码库的融合方向

Linux Bridge 子系统演进的终极启示是：好的架构不是一开始就设计完美的，而是在每次遇到真实约束时，找到最小化修改的解耦方案，逐步演进到当前状态的。 每一次引入新机制（rx_handler、switchdev notifier、Netlink attr、VLAN group），都是在已有设计中找到了一个"接缝"，沿接缝切开，插入新的抽象层。 这种"在现有复杂系统中不破坏稳定性地引入新能力"的能力，是 Linux 内核架构设计最值得学习的精髓。

第一性原理的终极表达：网络中的数据转发只需要三件事——知道"从哪来"（MAC学习），知道"去哪里"（FDB查找），知道"能不能去"（STP/VLAN/ACL）。Bridge 子系统三十年所有的演进，都是在更高效、更灵活、更安全、更可扩展地回答这三个问题。
