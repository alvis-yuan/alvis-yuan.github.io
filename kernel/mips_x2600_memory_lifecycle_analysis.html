01
MIPS 32 位虚拟地址空间
kseg0 / kseg1 / kseg2 / kuseg 分区原理

MIPS 32 位 CPU 在上电时不存在"实模式"，硬件天生将虚拟地址空间分为四个固定段（无需操作系统参与）， 这决定了整个内存管理的基础结构。

参考： MIPS Memory Management – kseg0/kseg1 Virtual and Physical Address Mapping； MIPS32 Architecture Manual

核心要点 MIPS 上电时 kseg1（0xA0000000）是唯一可用区域，因为此时 TLB 未初始化、Cache 未使能。 BootROM 在 kseg1 中执行，一旦 Cache 初始化完成后可迁移到 kseg0 运行。 这是 MIPS 平台与 x86"实模式"最本质的区别。

阶段总结

MIPS 32 位虚拟地址空间的固定分区是整个内存管理的硬件基础。kseg0/kseg1 直接映射物理内存， 无需 TLB，这使得内核在未建立页表前即可访问 DDR；kseg2 和 kuseg 依赖 TLB，分别服务内核虚拟地址 和用户进程地址空间。128MB DDR 完整落在 kseg0/kseg1 的直接映射窗口（0~512MB）内，无需 HIGHMEM。

02
阶段一：BootROM 执行
上电 → DDR 初始化前 · 极少量 SRAM 可用
⬡ ROM 阶段

执行位置与硬件状态

X2600 上电后，CPU（XBurst®2 MIPS 核）从片内 BootROM（固化在芯片内部）开始执行。 此时 DDR3 尚未初始化，DRAM 控制器未就绪，整个外部物理内存空间不可访问。

资源状态地址（kseg1 视角）
BootROM可用（只读）0xBFC00000（MIPS 标准复位向量）
TCSM（32KB SRAM）可用片内，由厂商 BSP 决定地址
8KB SRAM可用片内辅助 SRAM
DDR3 128MB不可用控制器未初始化
Cache未初始化所有操作通过 kseg1（uncached）
MMU / TLB空置kseg0/kseg1 不需要 TLB

BootROM 主要工作

复位向量跳转 (0xBFC00000)
CPU 从 MIPS 标准复位向量 0xBFC00000 开始取指（kseg1 Uncached），BootROM 代码在此运行
栈指针设置
将 SP 指向片内 TCSM（约 32KB），作为 C 代码运行的临时栈。此时无 DDR，所有栈操作在 SRAM 上
PLL / 时钟配置
初始化 PLL，设定 CPU/DDR 工作频率（X2600 参考值：CPU 800MHz，DDR 400MHz）
DDR3 控制器初始化
运行 DDR 训练序列，完成 PHY 初始化，此后 PA 0x00000000~0x07FFFFFF 可访问
加载下级 Bootloader (SPL)
从 SFC Flash / eMMC 读取 SPL，拷贝到 DDR 或 TCSM，跳转执行

阶段一：内存状态
DDR 初始化前后的物理内存使用量

DDR 初始化前（BootROM 早期）

物理内存视图（128MB DDR 未就绪）
TCSM/SRAM 40KB
DDR 128MB ✗ 不可用

DDR 初始化完成后（BootROM 末期）

128MB DDR 物理内存视图（总计 0x8000000 字节）
已用
~126MB 空闲（SPL 拷贝区域极小）
内存管理方式 BootROM 阶段没有内存管理器。所有"分配"都是硬编码地址（由芯片厂商固化）。 TCSM 用作临时栈，DDR 初始化后 BootROM 直接将 SPL 拷贝到预定义的 DDR 地址（通常是 DDR 起始处或某固定偏移）。

阶段总结

管理机制：无操作系统级内存管理，完全硬编码地址访问。

使用量：仅 TCSM 约 40KB 栈空间 + BootROM 代码本身（ROM 中，不占 DDR）。

剩余量：DDR 初始化后 128MB 全部空闲（仅 SPL 加载占少量，约几十 KB）。

交接方式：BootROM 将控制权（寄存器 PC）跳转至 SPL 入口；无数据结构传递，完全依赖硬件状态。

第一性原理：在没有操作系统、没有虚拟内存的最原始状态下，"内存管理"退化为"你知道什么地址可以用"——这完全由芯片硬件设计决定，是不可逾越的物理约束。

04
阶段二：U-Boot SPL 阶段
Secondary Program Loader · 精简初始化
⬡ U-Boot SPL

SPL（Secondary Program Loader）是 U-Boot 的精简版，其职责是在最小环境下完成必要初始化， 将完整的 U-Boot 镜像加载到 DDR，然后跳转执行。

SPL 执行环境

SPL 被 BootROM 加载到 DDR 低端（或 TCSM）运行。在 MIPS 平台，SPL 入口同样在 kseg1（uncached）或 kseg0（cached，需先初始化 Cache）段运行。 

/* U-Boot MIPS 平台：物理地址无 MMU 映射，使用 kseg0/kseg1 直接访问 DDR */
/* arch/mips/cpu/start.S 简化流程 */
reset:
    /* 清理 CP0 Status / Config */
    mfc0   k0, CP0_STATUS
    mtc0   zero, CP0_CAUSE

    /* 初始化 Cache（使能 kseg0 cached 访问）*/
    jal    mips_init_icache
    jal    mips_init_dcache

    /* 设置栈指针到 DDR 区域 */
    li     sp, CONFIG_SYS_INIT_SP_ADDR
    jal    board_init_f    /* U-Boot board_init_f */

参考：u-boot-ingenic README · OpenIPC GitHub

阶段二：U-Boot 主体内存管理
物理地址直接访问 · 无 MMU · malloc 堆
⬡ U-Boot 主体

U-Boot 的内存管理哲学在官方 README 中有明确描述：

U-Boot 官方 README 原文 "U-Boot runs in system state and uses physical addresses, i.e. the MMU is not used either for address mapping nor for memory protection. The available memory is mapped to fixed addresses using the memory controller."

"After booting and sizing and initializing DRAM, the code relocates itself to the upper end of DRAM. Immediately below the U-Boot code some memory is reserved for use by malloc() [see CONFIG_SYS_MALLOC_LEN configuration setting]. 阶段二：交接给内核
bootm / bootz · DTB 传递 · 内存 Map 传递

U-Boot 向内核传递的信息

U-Boot 通过 bootm 命令启动内核，MIPS Linux 的启动协议规定： 通过 设备树 DTB（Device Tree Blob） 向内核传递所有硬件信息，包括物理内存布局。

传递方式内容在 DTB 中的节点
a0 寄存器内核 argc—
a1 寄存器命令行字符串指针—
a2 寄存器DTB 指针（物理地址）—
DTB /memory 节点物理内存起始地址 + 大小memory { reg = <0x0 0x8000000>; }
DTB /chosen 节点bootargs，initrd 地址/大小chosen { bootargs = "..."; }
DTB /reserved-memory预留内存区域各厂商自定义
/* 典型 DTB /memory 节点（X2600 128MB）*/
memory {
    device_type = "memory";
    reg = <0x0 0x08000000>;  /* 起始 0x0，大小 128MB */
};

/* /chosen 节点 */
chosen {
    bootargs = "console=ttyS3,115200n8 mem=128M root=/dev/mmcblk0p2";
    linux,initrd-start = <0x01000000>;
    linux,initrd-end   = <0x01800000>;
};
关键：U-Boot 不做内存范围校验 U-Boot 自身的内存管理非常原始，不检测内核/DTB 是否超出边界，不保护已用区域。 交接时仅保证：内核镜像在目标地址可访问，DTB 指针有效，CPU 跳转到内核入口。 之后 U-Boot 自身被内核完全覆盖（不再需要）。

阶段总结

管理机制：物理地址直接访问，简单 dlmalloc 堆，无虚拟内存，无 TLB 映射。

使用量：U-Boot 自身约 2-4MB（顶端），内核+ramdisk ~8-16MB，DTB ~64KB，堆/栈约 2-8MB。总计占用约 10-20MB。

剩余量：约 108-118MB 未被任何管理器追踪。

交接方式：通过 CPU 寄存器 a2 传递 DTB 物理地址，由 DTB /memory 节点告知内核物理内存布局。内核从此接管全部物理内存管理权。

阶段三：内核早期启动
kernel_entry → start_kernel → setup_arch · 建立 Memblock
⬡ 内核早期

MIPS 内核入口流程

内核 ELF 入口为 kernel_entry（arch/mips/kernel/head.S）， 随后跳入 C 语言的 start_kernel()。整个内存初始化调用链如下： 

/* arch/mips/kernel/head.S → init/main.c */
kernel_entry:                    /* MIPS 汇编入口 */
    jal   start_kernel          /* 跳入 C 入口 */

start_kernel():
  → setup_arch(&cmdline_p)
      → cpu_probe()             /* 探测 CPU 类型 */
      → prom_init()             /* 平台早期初始化 */
      → arch_mem_init(cmdline_p)
          → plat_mem_setup()    /* 板级内存描述 */
          → bootmem_init()      /* memblock 初始化 */
          → sparse_init()
          → paging_init()       /* 建立页表/Zone */
  → mm_init()
      → mem_init()             /* memblock → buddy */
      → kmem_cache_init()      /* SLUB 分配器 */
      → vmalloc_init()

参考：the mem_init on MIPS (martinbj2008.github.io)

DTB 扫描与物理内存注册

在 setup_arch() 中，内核通过扫描 U-Boot 传来的 DTB，将物理内存布局注册进 memblock： 

/* kernel/lib/drivers/of/fdt.c */
early_init_dt_scan(dtb_pointer)
  → early_init_dt_scan_chosen()   /* 解析 /chosen 节点（bootargs 等）*/
  → early_init_dt_scan_root()     /* 解析地址单元格式 */
  → early_init_dt_scan_memory()   /* 解析 /memory 节点 */
      → memblock_add(0x00000000, 0x08000000)
        /* 将 128MB 注册为可用物理内存 */

阶段三：Memblock 分配器详解
Linux 官方启动期内存管理器

Linux 内核文档对 Memblock 有明确定义：

Linux Kernel 官方文档原文 "Memblock is a method of managing memory regions during the early boot period when the usual kernel memory allocators are not up and running. Memblock views the system memory as collections of contiguous regions. There are several types of these collections: memory – describes the physical memory available to the kernel; reserved – describes the regions that were allocated."

"As the system boot progresses, the architecture specific mem_init() function frees all the memory to the buddy page allocator."

"Unless an architecture enables CONFIG_ARCH_KEEP_MEMBLOCK, the memblock data structures (except 'physmem') will be discarded after the system initialization completes."

来源：Boot time memory management — The Linux Kernel documentation (docs.kernel.org)

Memblock 数据结构

/* include/linux/memblock.h */
struct memblock_region {
    phys_addr_t  base;          /* 区域起始物理地址 */
    phys_addr_t  size;          /* 区域大小 */
    enum memblock_flags flags;  /* NONE / HOTPLUG / NOMAP 等 */
};

struct memblock_type {
    unsigned long       cnt;         /* 当前区域数量 */
    unsigned long       max;         /* 最大区域容量 */
    phys_addr_t         total_size;  /* 总大小 */
    struct memblock_region *regions; /* 区域数组 */
    char               *name;
};

struct memblock {
    bool              bottom_up;      /* 分配方向（MIPS 为 bottom-up）*/
    phys_addr_t       current_limit;  /* 当前分配上限 */
    struct memblock_type memory;      /* 可用内存列表 */
    struct memblock_type reserved;    /* 已保留内存列表 */
};

来源：Boot time memory management — The Linux Kernel documentation

各类保留区域详解与估算

在 MIPS 平台的 arch_mem_init() → bootmem_init() 流程中， 以下区域被相继调用 memblock_reserve() 标记为已保留：

MIPS 特有：memblock_set_bottom_up(true)

在 MIPS Linux 5.10 的 arch_mem_init() 中，明确设置 memblock 分配方向为从底端向上： 

/* arch/mips/kernel/setup.c */
static void __init arch_mem_init(char **cmdline_p)
{
    extern void plat_mem_setup(void);
    plat_mem_setup();

    memblock_set_bottom_up(true);  /* MIPS 从低地址向上分配 */

    bootmem_init();
    ...
    sparse_init();
    plat_swiotlb_setup();
    paging_init();
}

参考：MIPS: cmdline: Clean up boot_command_line initialization — spinics.net

128MB 内存的特殊考量 128MB 对于嵌入式 Linux 来说较为紧张。若 CMA/reserved-memory 配置不当， 可能导致内核可用内存 < 64MB，进而引发 OOM。建议 CMA 配置不超过 32MB， 厂商 reserved 区域尽量精简。

阶段总结

管理机制：Memblock 分配器接管全部物理内存，维护 memory[] 和 reserved[] 两张表。分配通过 memblock_alloc() / memblock_reserve()，释放到 Buddy 通过 memblock_free_all()。

使用量：内核约 4-8MB + DTB ~64KB + initramfs 数MB + 页表 ~64KB + 厂商预留（可变）= 总计约 8-30MB 被 reserved。

剩余量：约 98-120MB 在 memblock.memory 中且未 reserved，等待移交 Buddy。

交接方式：调用 mem_init() → memblock_free_all() 将所有未保留页移交给 Buddy 分配器；Memblock 数据结构在内核初始化完成后通常被丢弃（CONFIG_ARCH_KEEP_MEMBLOCK 未开启时）。

MIPS paging_init() 流程

/* arch/mips/mm/init.c (Linux 5.10) */
void __init paging_init(void)
{
    unsigned long max_zone_pfns[MAX_NR_ZONES];

    pagetable_init();          /* 建立内核页表 */

    /* 设置各 Zone 的 PFN 上限 */
#ifdef CONFIG_ZONE_DMA
    max_zone_pfns[ZONE_DMA] = MAX_DMA_PFN;
#endif
    max_zone_pfns[ZONE_NORMAL] = max_low_pfn;  /* 128MB: PFN=0x8000 */
#ifdef CONFIG_HIGHMEM
    max_zone_pfns[ZONE_HIGHMEM] = highend_pfn;
#endif
    free_area_init_nodes(max_zone_pfns);       /* 初始化 struct page 数组 */
}

阶段四：Slab/SLUB 分配器初始化
小对象分配器 · kmalloc 体系

在 mm_init() 中，kmem_cache_init() 建立 SLUB 分配器， 这是内核小对象分配（kmalloc）的基础：

kmem_cache_init() 建立核心 cache
建立 kmem_cache（cache 描述符）和 kmem_cache_node（NUMA 节点信息）的基础 slab cache
kmalloc_caches 初始化
为 8B / 16B / 32B / 64B / 96B / 128B … 8MB 各尺寸创建 kmalloc-N cache，每个 cache 对应一种固定大小的小对象分配
per-CPU slab 结构分配
为每个 CPU 分配 kmem_cache_cpu 结构，XBurst®2 为双核，故分配 2 份
vmalloc_init()

SLUB 与 Buddy 的关系

/* SLUB 向 Buddy 申请整页，再切割为小对象 */
kmalloc(512) → kmem_cache_alloc(kmalloc-512) 
    → [若 cpu slab 无空闲] → get_page_from_freelist()

阶段四：vmalloc 区域
MIPS kseg2 · 虚拟地址连续但物理不连续

阶段总结（Buddy + Slab + vmalloc）

管理机制：Buddy 系统按 order 管理 4KB 页帧，SLUB 分配器切割整页为小对象，vmalloc 在 kseg2 提供大块虚拟连续分配。三层分配器协同工作。

使用量估算（128MB）：内核代码/数据 ~6MB，struct page 数组 ~2MB，SLUB caches ~1-4MB，内核线程栈等 ~1MB，init 释放前 initramfs ~4-8MB。总计约 14-22MB 被内核使用。

剩余量：约 106-114MB 可通过 Buddy 分配，由 kswapd 在低水位时回收。

交接方式：内核 do_basic_setup() → initcalls 链调用驱动初始化，各驱动通过 kmalloc / alloc_pages 从 Buddy/SLUB 分配内存。

第一性原理：Buddy 系统的本质是"二叉树+分裂/合并"策略，用最少的元数据（仅 struct page）管理最多的物理页帧，是空间与时间效率的最优折中。

阶段五：init 进程内存管理
进程地址空间 · VMA · mmap · fork/exec

进程地址空间（mm_struct）

每个用户进程拥有独立的 mm_struct，通过 TLB 映射到物理页帧。 MIPS 32 位用户地址空间为 0x00000000~0x7FFFFFFF（kuseg，2GB）， 实际在 128MB 物理内存约束下，每个进程实际可映射的物理页受限。 

/* 进程地址空间典型分布（MIPS 32-bit，小地址在下）*/
0x7FFFFFFF ─────────────────────────
            内核映射区（kseg0/kseg2，进程共享）
0x7F000000 ─────────────────────────
            栈（向下增长，RLIMIT_STACK 限制）
            ...
            共享库映射（mmap 区域）
            ...
0x40000000 ─────────────────────────
            堆（向上增长，brk）
            ...
0x10000000 ─────────────────────────
            可执行文件映射（.text / .data / .bss）
0x00010000 ─────────────────────────
            （保留低地址空间，NULL 陷阱）
0x00000000 ─────────────────────────

物理内存分配路径（用户进程）

mmap / brk → VMA 建立（虚拟）
sys_mmap() 仅建立 VMA 描述符（vm_area_struct），不立即分配物理页，实现延迟分配
缺页异常 → do_page_fault()
用户进程访问虚拟地址触发 TLB miss → MIPS TLB refill → 若页不存在则 do_page_fault() → handle_mm_fault()
alloc_page() → Buddy 分配物理页
从对应 Zone 的 Buddy 链表中分配物理页，写入 PTE，TLB 更新，继续执行
匿名页 / 文件映射
匿名页（堆/栈）直接从 Buddy 分配；文件映射通过 page cache，复用已缓存页

阶段五：稳态内存使用分布
init 进程启动后的 128MB 内存全貌

阶段总结（用户进程稳态）

管理机制：Buddy（物理页） + SLUB（小对象） + VMA（虚拟地址） + TLB（地址翻译） + kswapd（内存回收）协同工作。用户进程通过缺页异常按需分配物理页。

使用量：内核固定占用约 12-20MB，驱动/预留约 8-16MB，用户进程约 2-6MB，页缓存 5-10MB。总计约 28-52MB 被使用。

剩余量：约 76-100MB 在 Buddy 空闲链表中（/proc/meminfo MemFree），可随时分配。

第一性原理：操作系统内存管理的终极目标是"让有限的物理内存高效服务尽可能多的进程"。在 128MB 嵌入式场景下，这要求每一层（Bootloader、Memblock、Buddy、SLUB、VMA）都必须精确记账、按需分配、及时回收。

第一性原理总结
内存管理的本质逻辑

核心命题

第一性原理 物理内存是有限的稀缺资源（此处 128MB）。内存管理的终极目标是： 在任意时刻，每一字节的物理内存都被"知道在哪儿、谁在用、何时可回收"。 实现这一目标的方式随系统状态演进而不断精化。
最原始：硬编码（BootROM）
在没有任何软件基础设施时，只有一种"管理"方式：提前知道哪个地址可以用。这是确定性的极限。代价：零灵活性。
线性堆（U-Boot）
引入简单的线性记账（dlmalloc）。优点：能动态分配。代价：无法感知物理内存全貌，无 Zone，无 GC。
区域表（Memblock）
用两张有序区间列表（memory[] / reserved[]）描述全部物理内存。这是"知道全貌"的最小代价实现。代价：无法高效分配小对象。
分级分配（Buddy + SLUB）
Buddy 解决页级外碎片（伙伴合并），SLUB 解决小对象内碎片（按尺寸切割）。这是空间和时间效率的工程最优解。代价：元数据（struct page，kmem_cache）占用固定内存。
虚拟化（VMA + TLB + Swap）
通过"虚实分离"让每个进程以为自己拥有全部地址空间，物理页按需延迟分配，冷数据换出换入。这是多进程共享有限物理内存的终极机制。代价：TLB miss、缺页异常、换页延迟。

最终结论

在君正 X2600（MIPS）+ Linux 5.10.186 + 128MB DDR3 的完整启动链中， 物理内存经历了从"无管理"（BootROM）→ "线性堆"（U-Boot）→ "区域表"（Memblock） → "伙伴系统"（Buddy）→ "虚实分离"（VMA）五个层次的演进。

每一次交接都是前一阶段使命完成、后一阶段能力更强的体现。 最终进入稳态时，128MB 物理内存中约 76-104MB 以空闲页形式在 Buddy 系统中待命， 随时响应内核和用户进程的分配请求。

排查内存问题的根本方法是：找到对应生命周期阶段的记账工具，读取其状态—— 这与分层的管理机制完全对应，不存在"统一视图"，只有层层递进的精确描述。
