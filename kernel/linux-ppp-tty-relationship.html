§0 · 引言

问题背景：为什么要研究这个问题？

当你拨号上网时，pppd 守护进程打开串口（如 /dev/ttyS0），在 AT 命令握手之后，拨号连接升级成了 PPP 网络接口（ppp0）。这背后发生了什么？

PPP（Point-to-Point Protocol，点对点协议）是 RFC 1661 定义的标准互联网协议，用于在串行链路上传输 IP 帧。它为什么要和 TTY 子系统扯上关系？难道不能独立实现一个网络驱动？

答案隐藏在 Linux 内核的历史设计哲学中：串行硬件的唯一内核入口是 TTY 子系统。要使用串口，就必须通过 TTY；要在 TTY 上跑网络协议，就必须使用「线路规程」（Line Discipline）这个插件机制。

📌 原始文献来源 本文内容严格基于以下权威来源： 🔗 kernel.org · ppp_generic.txt 🔗 kernel.org · TTY Line Discipline 🔗 The TTY demystified 🔗 GitHub · ppp_async.c 🔗 GitHub · pppd/sys-linux.c
⚛ 第一性原理概述

串行硬件（UART）的字节流需要一个内核层来管理；Linux 将所有串行设备统一抽象为 TTY，并通过可插拔的"线路规程"模块扩展其行为。PPP 不是第一个，也不是最后一个复用这个机制的网络协议（SLIP 更早）。问题的本质是：如何在不修改底层驱动的前提下，改变串行数据的处理方式？答案是线路规程。

§1 · TTY 子系统

TTY 的历史与整体架构

历史起源

TTY（Teletype，电传打字机）子系统的历史可以追溯到 1869 年的股票报价机。电传打字机是早期 UNIX 计算机的标准输入输出设备。当物理电传打字机逐渐消失后，它的软件抽象却以 TTY 的形式保留了下来，成为 Linux 与 UNIX 系统设计的核心组件。

如 Linus Åkesson 在「The TTY demystified」中所述：

"In present time, we find ourselves in a world where physical teletypes and video terminals are practically extinct... But the legacy from the old cast-iron beasts is still lurking beneath the surface."

三层架构

TTY 子系统由三个层次组成，从底层到顶层：

图1：Linux TTY 子系统三层架构
层次 核心文件 主要职责
UART 驱动 drivers/tty/serial/ 物理字节传输、波特率、奇偶校验、流控
线路规程 drivers/tty/n_tty.c 等 数据解释与处理（行编辑、协议封帧等）
TTY 驱动 drivers/tty/tty_io.c 会话管理、作业控制、信号派发
⚛ 第一性原理

TTY 的核心价值是关注点分离：硬件只管传字节（UART 驱动），内核只管会话与信号（TTY 驱动），而数据的语义解释留给可插拔的线路规程。这种分层使得在不改任何底层代码的情况下，只需替换中间层就能改变数据的行为——这正是 PPP 所利用的设计契机。

§2 · 线路规程

线路规程（Line Discipline）详解

核心概念

官方内核文档（kernel.org · tty_ldisc）定义：

"TTY line discipline process all incoming and outgoing character from/to a tty device. The default line discipline is N_TTY. It is also a fallback if establishing any other discipline for a tty fails."

线路规程是一个可插拔的内核模块，插在 UART 驱动与 TTY 驱动之间，负责处理所有进出数据的语义。每个 TTY 设备在任何时刻只能使用一种线路规程。

注册与切换

/* 注册新线路规程 - 来自 include/linux/tty_ldisc.h */
int tty_register_ldisc(const struct tty_ldisc_ops *new_ldisc);

/* 切换线路规程 - 用户空间通过 ioctl 触发 */
int disc = N_PPP;
ioctl(tty_fd, TIOCSETD, &disc);  /* 将 tty_fd 切换到 PPP 线路规程 */

tty_ldisc_ops 接口结构

来自官方内核文档，线路规程通过 tty_ldisc_ops 结构提供接口：

struct tty_ldisc_ops {
    char  *name;           /* 显示在 /proc/tty/ldiscs 中 */
    int    num;            /* N_* 编号（如 N_TTY=0, N_PPP=3） */

    /* 来自 TTY 核心的调用（"上方"） */
    int     (*open)(struct tty_struct *tty);
    void    (*close)(struct tty_struct *tty);
    ssize_t (*read)(struct tty_struct *tty, ...);
    ssize_t (*write)(struct tty_struct *tty, ...);
    int     (*ioctl)(struct tty_struct *tty, unsigned int cmd, ...);
    void    (*hangup)(struct tty_struct *tty);

    /* 来自底层驱动的调用（"下方"） */
    void    (*receive_buf)(struct tty_struct *tty,
                           const u8 *cp, const u8 *fp, size_t count);
    void    (*write_wakeup)(struct tty_struct *tty);
    /* ... */
};

已知线路规程编号

编号宏名协议用途
0N_TTY默认标准终端行编辑
1N_SLIPSLIP串行线路网络协议（PPP 前身）
3N_PPPPPP点对点协议（异步）
12N_HDLCHDLC同步数据链路控制
14N_SYNC_PPPPPP 同步同步串行 PPP
27N_NULLNULL丢弃所有数据（最终兜底）
💡 来自内核文档的警告 "Do not reuse ldisc numbers as they are part of the userspace ABI and writing over an existing ldisc will cause demons to eat your computer." —— kernel.org/doc/driver-api/tty/tty_ldisc
⚛ 第一性原理

线路规程本质上是一个策略插件——它回答"这些字节的含义是什么？"这个问题。N_TTY 说"这是可编辑的文本"，N_PPP 说"这是 PPP 帧的字节流"。通过运行时替换策略，内核实现了"同一块硬件，不同语义"的多态能力，而无需为每个协议重写驱动。

§3 · PPP 架构

PPP 在 Linux 内核中的整体结构

三层 PPP 架构

根据 Linux 内核官方文档（ppp_generic.txt），Linux PPP 实现分为三个层次：

图2：Linux PPP 内核架构（基于 ppp_generic.txt）

PPP 通用层的职责

官方文档（ppp_generic.txt）描述 PPP 通用层提供以下功能：

  • 网络接口单元（ppp0 等）
  • 与 Linux 网络栈的接口
  • PPP 多链路（Multilink）：在多个链路间分片传输数据报
  • 通过 /dev/ppp 字符设备与 pppd 的接口
  • 数据包压缩与解压（Deflate、BSD-Compress）
  • TCP/IP 头压缩与解压
  • 按需拨号检测与空闲超时
  • 简单数据包过滤

PPP 通道（Channel）抽象

文档原文：

"A PPP channel encapsulates a mechanism for transporting PPP frames from one machine to another. A PPP channel implementation can be arbitrarily complex internally but has a very simple interface with the generic PPP code: it merely has to be able to send PPP frames, receive PPP frames, and optionally handle ioctl requests." 
/* PPP 通道接口 - include/linux/ppp_channel.h */
struct ppp_channel_ops {
    /* 发送 PPP 帧：返回 1 表示接受，0 表示拒绝（流控） */
    int  (*start_xmit)(struct ppp_channel *chan, struct sk_buff *skb);
    /* 处理 ioctl */
    int  (*ioctl)(struct ppp_channel *chan, unsigned int cmd,
                  unsigned long arg);
};
⚛ 第一性原理

PPP 通用层与 PPP 通道之间的接口极度精简——只有"发帧"和"收帧"两个动作。这符合接口最小化原则：通道层不需要知道上层在做多链路还是压缩，通用层也不需要知道下面跑的是异步串口还是以太网（PPPoE）。分层解耦使得每一层都可以独立演化。

§4 · N_PPP

N_PPP：PPP 线路规程的实现

异步 PPP 线路规程

来源：drivers/net/ppp/ppp_async.c

ppp_async.c 实现了异步 PPP 的 TTY 线路规程，本质上是在 PPP 通用驱动和底层 TTY 设备之间插入了一个中间层，负责 HDLC 帧的封装与解封。

/* ppp_async.c 注册 N_PPP 线路规程 */
static int __init ppp_async_init(void)
{
    int err = tty_register_ldisc(&ppp_ldisc);
    /* N_PPP = 3 */
    return err;
}

/* 线路规程操作集 */
static struct tty_ldisc_ops ppp_ldisc = {
    .owner      = THIS_MODULE,
    .num        = N_PPP,
    .name       = "ppp",
    .open       = ppp_asynctty_open,
    .close      = ppp_asynctty_close,
    .hangup     = ppp_asynctty_hangup,
    .read       = ppp_asynctty_read,     /* 返回 -EAGAIN（不直接读）*/
    .write      = ppp_asynctty_write,    /* 返回 -EAGAIN（不直接写）*/
    .ioctl      = ppp_asynctty_ioctl,
    .receive_buf = ppp_asynctty_receive, /* 核心：从 UART 接收数据 */
    .write_wakeup = ppp_asynctty_wakeup, /* 通知可以继续发送 */
};

read/write 返回 -EAGAIN 的设计

代码注释原文（来自 ppp_async.c）揭示了一个重要设计决策：

/*
 * Read does nothing - no data is ever available this way.
 * Pppd reads and writes packets via /dev/ppp instead.
 */
static ssize_t ppp_asynctty_read(...)  { return -EAGAIN; }

/*
 * Write on the tty does nothing, the packets all come in
 * from the ppp generic stuff.
 */
static ssize_t ppp_asynctty_write(...) { return -EAGAIN; }
⚠️ 关键设计点 pppd 不通过 tty 的 read()/write() 接口读写 PPP 数据！它通过 /dev/ppp 字符设备操作数据，而 tty 的 receive_buf（由底层 UART 驱动调用）才是真正的数据入口。

asyncppp 状态结构

/* ppp_async.c 内部状态 */
struct asyncppp {
    struct tty_struct *tty;        /* 关联的 TTY 设备 */
    unsigned int      flags;
    unsigned int      state;
    unsigned int      rbits;
    int               mru;         /* 最大接收单元 */
    spinlock_t        xmit_lock;
    spinlock_t        recv_lock;
    /* ... */
    struct sk_buff    *rpkt;       /* 正在组装的接收包 */
    struct sk_buff_head rqueue;    /* 接收队列 */
    struct tasklet_struct tsk;
    struct ppp_channel chan;       /* 通向 PPP 通用层的接口 */
};
图3：asyncppp 结构体与相邻层的关系
⚛ 第一性原理

N_PPP 线路规程的关键设计是双向解耦：向下，它通过 receive_buf 钩子被动接收 UART 字节流，通过 tty->ops->write() 主动发出字节流；向上，它通过 ppp_channel 接口与 PPP 通用层交换完整 PPP 帧。两端都不需要知道对方的实现细节——这是标准的"端口/适配器"架构（Hexagonal Architecture）在内核中的体现。

§5 · 建立连接

建立 PPP 连接的完整流程

pppd 的实际操作步骤

来源：pppd/sys-linux.c · tty_establish_ppp() 与 kernel.org 文档

图4：pppd 建立 PPP 连接的时序图（基于 pppd/sys-linux.c）

关键代码：tty_establish_ppp()

以下是 pppd 源码（pppd/sys-linux.c）中的核心函数，展示了线路规程切换的精确过程：

/* pppd/sys-linux.c - tty_establish_ppp() */
int tty_establish_ppp(int tty_fd)
{
    /* 1. 独占串口 */
    ioctl(tty_fd, TIOCEXCL, 0);

    /* 2. 选择线路规程：同步或异步 */
    ppp_disc = ppp_sync_serial() ? N_SYNC_PPP : N_PPP;

    /* 3. 🔑 关键操作：切换线路规程 */
    if (ioctl(tty_fd, TIOCSETD, &ppp_disc) < 0) {
        error("Couldn't set tty to PPP discipline: %m");
        return -1;
    }

    /* 4. 建立通用 PPP 层连接 */
    ret_fd = ppp_generic_establish(tty_fd);

    return ret_fd;
}

int ppp_generic_establish(int fd)
{
    /* 获取通道编号 */
    ioctl(fd, PPPIOCGCHAN, &chindex);

    /* 打开 /dev/ppp 控制接口 */
    fd = open("/dev/ppp", O_RDWR);

    /* 绑定通道 */
    ioctl(fd, PPPIOCATTCHAN, &chindex);

    /* ... 创建或连接 PPP 网络单元 ... */
}
📖 官方文档原文（ppp_generic.txt） "Generally a user-level process will initialize the underlying communications medium and prepare it to do PPP. For example, with an async tty, this can involve setting the tty speed and modes, issuing modem commands, and then going through some sort of dialog with the remote system to invoke PPP service there. We refer to this process as 'discovery'."
⚛ 第一性原理

整个建立过程体现了 UNIX 哲学中的渐进式配置：先用已有工具（N_TTY 文本模式）做准备（拨号），再用一个 ioctl 原子性地切换语义（TIOCSETD），最后用标准的 fd 传递机制把不同层次连接起来。没有新的系统调用，没有新的特权接口——一切都通过现有机制组合完成。

流量控制

官方文档（ppp_generic.txt）描述了 PPP 的流控机制：

  • 当通道的 start_xmit() 返回 0 时，表示拒绝接受数据包（流控）
  • 通道在可以接受更多数据时调用 ppp_output_wakeup()
  • PPP 通用层会重新尝试发送被拒绝的帧
  • TTY 层的 write_wakeup 回调在发送缓冲区有空间时触发
⚛ 第一性原理

数据路径揭示了协议栈的本质是函数调用链：每一层接收下层的原始数据，添加/去除自己的语义（帧边界、错误检测、压缩），再传递给相邻层。PPP 通过线路规程将这条调用链的一个环节（字节流 ↔ PPP 帧）插入到 TTY 与 UART 之间，优雅地避免了对现有代码的任何修改。

§8 · 深层原因

为什么 PPP 必须与 TTY 子系统关联？

原因一：串行硬件的唯一内核入口

在 Linux 内核中，所有串行硬件（UART）的访问都通过 TTY 子系统。UART 驱动注册为 TTY 驱动，其接口对外只暴露 /dev/ttyS0、/dev/ttyUSB0 等 TTY 设备文件。没有绕过 TTY 的"原始 UART 接口"。

如 O'Reilly《Linux Network Administrator's Guide》所述（原文链接）：

"The kernel doesn't really treat the tty device as a network device that you can manipulate like an Ethernet device, using commands such as ifconfig. However, it does treat tty devices as places where network devices can be bound. To do this, the kernel changes what is called the 'line discipline' of the tty device. Both SLIP and PPP are line disciplines that may be enabled on tty devices."

原因二：线路规程提供了完美的插入点

线路规程机制在 PPP 出现之前就存在（为 N_TTY 设计），它在 UART 驱动和 TTY 驱动之间提供了一个天然的拦截点：

  • 不需要修改 UART 驱动（硬件相关，难以修改）
  • 不需要修改 TTY 驱动（通用，修改影响面广）
  • 只需注册一个新的线路规程，拦截字节流即可

原因三：历史路径依赖（SLIP 先例）

SLIP（Serial Line Internet Protocol）是 PPP 的前身，它同样通过线路规程（N_SLIP）实现。PPP 沿用了这一成熟模式。

图8：串行网络协议的历史演进

原因四：复用现有的设备管理基础设施

通过 TTY 子系统，PPP 可以免费获得：

  • 设备文件：/dev/ttyS0 等已经存在，无需为 PPP 创建新设备
  • modem 控制：DCD（载波检测）、HUP 信号处理
  • 独占访问：TIOCEXCL 防止多个进程竞争串口
  • 波特率和线路配置：通过 termios API 设置，pppd 和 chat 都使用
  • 流量控制：硬件 RTS/CTS 流控，软件 XON/XOFF 流控

原因五：PPPoE 说明了问题的反面

PPPoE（PPP over Ethernet）不经过 TTY 子系统——它运行在以太网上，直接与网络层交互。这恰好证明了：PPP 需要 TTY 的原因是载体（串行线路），而不是 PPP 协议本身的需要。当载体是以太网时，就不需要 TTY 了。

🎯 关键洞察 PPP 需要 TTY 的原因是：它的物理载体是串行线路，而 Linux 内核中串行线路的唯一抽象是 TTY。线路规程机制是这个约束下最优雅的扩展方式。
⚛ 第一性原理

网络协议的实现路径由其物理介质决定。PPP 的物理介质是串行线（UART），而 Linux 对串行硬件的抽象就是 TTY。因此 PPP 必须通过 TTY。这不是 PPP 的特殊性，而是普遍规律：以太网协议通过网络驱动框架，USB 协议通过 USB 子系统，串行协议通过 TTY 子系统——每类硬件在内核中只有一个抽象入口。

⚛ 终极第一性原理

一切的根源在于一个物理事实：信息在串行线路上以字节流的形式传输，没有天然的"包"边界。

为了在这种介质上运行网络协议，必须解决两个问题：①如何把字节流切割成有意义的帧（成帧）；②如何把这个成帧逻辑插入现有的内核体系（集成）。

Linux 的 TTY 线路规程机制完美解决了问题②：它是内核预留的、可替换的"字节流处理策略插槽"。PPP 通过注册 N_PPP 线路规程解决了问题①，并借用了 TTY 已有的全套基础设施。

结论：PPP 与 TTY 的关联，是物理约束（串行介质）、工程约束（复用已有内核基础设施）和设计哲学（可插拔、关注点分离）共同作用的必然结果。
