Linux sk_buff
网络协议栈设计原理

深入解析 Linux 内核中最核心的网络数据结构 —— Socket Buffer（sk_buff）的设计理念、内存布局，以及数据包在各协议层发送与接收过程中的完整流转机制。

基于 Linux Kernel 官方文档 oldvger.kernel.org/~davem/skb.html wiki.linuxfoundation.org

背景：为什么需要 sk_buff？

Linux 网络实现被设计为协议无关的（protocol-independent）。这既适用于网络层与传输层协议（TCP/IP、IPX/SPX 等），也适用于网络适配器协议（以太网、令牌环等）。Socket Buffer（sk_buff） 是 Linux 内核中用于表示和管理数据包的核心数据结构。

在其他操作系统中，数据包穿越协议层时通常需要频繁内存拷贝。Linux 通过 sk_buff 的 headroom 预留机制，使各层可直接在已有缓冲区前端写入头部，极大减少拷贝开销。载荷（payload）仅被拷贝两次：一次从用户空间到内核空间，一次从内核空间到网络适配器。

核心设计目标： 在整个内核网络处理期间，始终通过同一个 sk_buff 来寻址和管理数据包，避免不必要的内存拷贝。

图1：sk_buff 作为统一数据结构贯穿所有协议层

🔬 第一性原理

网络协议栈的本质是：同一份数据在传递过程中被不同层的处理逻辑"包装"和"解包"。最高效的实现是零拷贝原则：不移动数据本身，只移动指针。sk_buff 就是这一原则的具体实现 —— 它是数据的"地址簿"，而非数据本身的容器。

数据结构

sk_buff 内存布局

sk_buff 由两个独立部分组成：元数据头（struct sk_buff）和数据缓冲区。元数据中包含 4 个关键指针：head、data、tail、end，共同描述缓冲区的几何形状。

图2：sk_buff 内存布局与四大指针关系

四大核心指针

指针含义说明
head缓冲区起始地址分配的内存块最开始位置，固定不变
data有效数据起始地址skb_push() 使其向左移（添加头部），skb_pull() 使其向右移（消费头部）
tail有效数据结束地址skb_put() 使其向右移（追加尾部数据）
end缓冲区结束地址固定不变，end 之后紧跟 skb_shared_info
关键设计： head 到 data 之间的空间称为 headroom，data 到 tail 之间为有效数据，tail 到 end 之间为 tailroom。发送时通过 skb_reserve() 预留 headroom，各层依次通过 skb_push() 写入本层头部。

🔬 第一性原理

数据包头部的"封装"本质上是在数据前方加前缀。最高效的方式不是把原数据后移（O(n) 拷贝），而是把指针前移（O(1)）。headroom 机制正是这一原理：事先在缓冲区前端保留足够空间，各层只需移动 data 指针即可完成封装，代价恒为常数时间。

数据结构

关键字段解析

以下字段解析基于 David S. Miller 在 oldvger.kernel.org/~davem/skb.html 的权威说明以及 Linux 基金会 Wiki。

// include/linux/skbuff.h
struct sk_buff {
    /* 链表：数据包可存在于多种队列（如 TCP 发送队列）*/
    struct sk_buff     *next, *prev;
    struct sk_buff_head *list;  // 所在队列

    struct sock        *sk;    // 关联的 socket（路由器转发时为 NULL）
    struct timeval     stamp;  // 时间戳（数据包到达/发送时间）
    struct net_device  *dev;   // 当前处理网络设备

    /* 三层协议头指针 */
    union { struct tcphdr *th; struct udphdr *uh; ... } h; // L4
    union { struct iphdr  *iph; ...                     } nh;// L3
    union { unsigned char *raw;                         } mac;// L2

    struct dst_entry   *dst;   // 路由信息（输入/输出均使用）

    /* 控制块：各层私有存储区（48字节）*/
    char cb[48] __aligned(8);

    /* 长度字段 */
    unsigned int len;       // 总长度（线性+分片）
    unsigned int data_len;  // 分片数据长度
    unsigned int mac_len;   // MAC 头部长度
    unsigned int csum;      // 校验和

    /* 标志位 */
    unsigned char cloned:1; // 是否为克隆（共享数据区）
    unsigned char nohdr:1;  // TSO支持：不允许修改头部
    unsigned char pkt_type; // PACKET_HOST/BROADCAST/MULTICAST
    unsigned char ip_summed;// CHECKSUM_NONE/HW/UNNECESSARY

    unsigned short protocol;// 以太网协议类型 ETH_P_*
    __u32          priority; // QoS 优先级（来自 IP TOS 字段）

    /* 引用计数与内存核算 */
    atomic_t     users;     // 引用计数
    unsigned int truesize;  // 计入 socket 内存的实际大小

    /* 四大核心指针（见图2）*/
    unsigned char *head, *data, *tail, *end;
};

dev 字段的双重角色

接收时：dev 指向接收网卡；路由决策完成后，dev 更新为出口网卡。input_dev 始终保持真正的物理接收设备（即使经过 bonding/虚拟设备层层解封装）。

protocol 字段

由 eth_type_trans() 等函数初始化，取 ETH_P_* 值（如 ETH_P_IP=0x0800）。即使非以太网设备也使用这套值体系，用于标识应接收该包的上层协议。

ip_summed 校验和

CHECKSUM_NONE：网卡未提供校验；CHECKSUM_HW：csum 字段含全包两补码校验；CHECKSUM_UNNECESSARY：无需校验（如 loopback）。

truesize 内存核算

记入 sk->sk_rmem_alloc 的实际内存量。若 socket 接收缓冲区满（超过 tcp_rmem 上限），新到达的数据包将被丢弃。

🔬 第一性原理

网络协议层之间需要传递的不仅是数据，还有关于数据的"元信息"（协议类型、校验状态、路由信息等）。传统方法是为每层单独分配结构体并建立关联。sk_buff 的设计将所有元信息嵌入同一结构体，消除了跨层查找的间接寻址开销，以空间换取了时间效率。

数据结构

数据指针操作函数

Linux 内核提供了一套标准函数来操作 sk_buff 数据区，这些操作通过移动指针而非移动数据来实现高效的头部增删。

图3：skb_push / skb_pull / skb_put 操作对指针的影响

// 向前添加头部数据（发送时各层封装头部）
void *skb_push(struct sk_buff *skb, unsigned int len) {
    skb->data -= len;
    skb->len  += len;
    return skb->data;
}

// 从前端消费头部（接收时各层剥除头部）
void *skb_pull(struct sk_buff *skb, unsigned int len) {
    skb->data += len;
    skb->len  -= len;
    return skb->data;
}

// 在尾部追加数据（接收时网卡DMA写入数据）
void *skb_put(struct sk_buff *skb, unsigned int len) {
    void *tmp = skb_tail_pointer(skb);
    skb->tail += len;
    skb->len  += len;
    return tmp;
}

// 获取各层头部指针
struct ethhdr  *eth_hdr(skb)   { return (ethhdr *)skb_mac_header(skb); }
struct iphdr   *ip_hdr(skb)    { return (iphdr  *)skb_network_header(skb); }
struct tcphdr  *tcp_hdr(skb)   { return (tcphdr *)skb_transport_header(skb); }

🔬 第一性原理

所有 skb 指针操作的时间复杂度均为 O(1)。这遵循了"让操作代价与数据大小无关"的设计准则。发送时 skb_push 把 data 左移，接收时 skb_pull 把 data 右移 —— 两者都是寄存器级整数运算，与数据包大小完全无关。

数据结构

cb[] 控制块 —— 各层私有存储

sk_buff 中的 cb[48]（或现代内核的 cb[48]，__aligned(8)）是一块不透明的"暂存区"，供协议层和驱动在处理数据包期间存储私有的每包信息（per-packet info）。

来源： David S. Miller 官方说明："This is the SKB control block. It is an opaque storage area usable by protocols, and even some drivers, to store private per-packet information. TCP uses this, for example, to store sequence numbers and retransmission state for the frame." 
// TCP 使用 cb[] 存储 TCP 私有元信息
struct tcp_skb_cb {
    union {
        struct inet_skb_parm  h4; // IPv4 选项信息
        struct inet6_skb_parm h6; // IPv6 选项信息
    } header;
    __u32  seq;       // 起始序列号
    __u32  end_seq;   // SEQ + FIN + SYN + datalen
    __u8   tcp_flags; // TCP 头部标志
    __u8   sacked;    // SACK 位状态
    // ...
};
// 访问宏：直接将 cb[] 强制转型为目标结构
#define TCP_SKB_CB(__skb) ((struct tcp_skb_cb *)&((__skb)->cb[0]))

// 使用示例
TCP_SKB_CB(skb)->seq = tcp_seq;
TCP_SKB_CB(skb)->tcp_flags = TCPHDR_SYN;

为什么不用单独分配？

每收/发一个包都需要分配 sk_buff。若为每层的私有元数据单独 kmalloc，高速网络下（百万包/秒）会产生巨大分配压力。内嵌 cb[] 使得所有元数据随 sk_buff 一次性分配和释放。

各层如何协作？

cb[] 在不同协议层之间是互斥使用的：当包处于 IP 层时，TCP 的 cb 数据无意义；反之亦然。各层约定各自的解释方式，通过类型转换宏访问，无运行时开销。

🔬 第一性原理

cb[] 体现了"按需复用，约定优于运行时检查"的设计哲学。固定大小的内联存储区消除了动态分配，协议层通过编译期类型约定而非运行时标签来区分，这是用静态知识替代动态开销的典型范式。

数据结构

分片支持与 skb_shared_info

现代 NIC 支持 Scatter-Gather DMA，可将数据包 DMA 写入不连续的物理页。内核用"线性区 + 分片数组"的混合模型来表示这种情况。

图4：sk_buff 线性区 + 分片页混合内存模型

/* 分片存储在 skb_shared_info 中（位于线性区末尾 skb→end 处）*/
struct skb_shared_info {
    skb_frag_t frags[MAX_SKB_FRAGS]; // 分片数组
    __u8       nr_frags;             // 实际分片数
    __u16      gso_size;             // GSO 段大小
    __u16      gso_segs;             // GSO 段数量
    // ...
};

/* 访问：通过内联函数将 skb→end 强转为 skb_shared_info* */
static inline struct skb_shared_info *skb_shinfo(const struct sk_buff *skb) {
    return (struct skb_shared_info *)skb_end_pointer(skb);
}

/* 为何 skb_shared_info 放在 skb→end 处？
 * 避免额外的 kmalloc：线性区和分片元数据一次性分配，
 * 减少高速路径上的内存分配次数。*/
TSO（TCP Segmentation Offload）： 支持 TSO 的 NIC 可接受一个大 sk_buff（例如 64KB payload 在分片数组中），由硬件自行切割成 MTU 大小的帧。内核无需软件分片，显著降低 CPU 负载。

🔬 第一性原理

skb_shared_info 紧跟线性区末尾，是"布局即语义"原则：将相关数据物理上放在一起，利用指针偏移代替单独的堆分配。一次 kmalloc 同时获得线性缓冲区和分片元数据，最小化热路径（每包路径）上的内存分配次数。

🔬 第一性原理

发送路径体现了"逐层封装，头部前置，零移动数据"的设计。每层只做两件事：①向 headroom 写入本层头部（skb_push）；②将指针传给下层。数据载荷在内核态全程不移动，直到 DMA 将其复制到网卡缓冲区（第 2 次也是最后一次必要的拷贝）。

协议栈流程

接收路径（RX）—— 从网卡到应用层

接收路径与发送路径对称：从硬件 → L2 → L3 → L4 → Socket，每层通过 skb_pull() 消费本层头部，将剩余数据传给上层。

🔬 第一性原理

接收路径的本质是"逐层剥壳"：每层读取并验证本层头部后，通过 skb_pull 将 data 指针右移，向上层呈现"更小的视图"。注意头部字节并未被清除，只是不再"可见"。这种惰性删除策略避免了清零开销，mac_header/network_header/transport_header 偏移量允许任意层随时回溯访问已处理的头部。

高级特性

克隆与共享 —— skb_clone / skb_copy

图7：skb_clone 与 skb_copy 对比

skb_clone() —— 共享克隆

创建一个新的 sk_buff 头部，但与原始 skb 共享相同的数据缓冲区。cloned 位在两者中均置 1，dataref 引用计数增加。适用于需要将同一数据包发给多个消费者的场景（如：packet socket 监听 + 正常转发路径）。

任何需要修改头部的层必须先调用 skb_unshare() 或 pskb_expand_head() 获取独立副本（写时复制语义）。

skb_copy() —— 完整拷贝

创建完全独立的 sk_buff，包括所有数据区的深拷贝。适用于需要独立修改内容的场景，代价是全量内存拷贝。

kfree_skb()：引用计数减 1，为 0 时才真正释放（丢包路径）。consume_skb()：正常完成路径的释放（不计入丢包统计）。

🔬 第一性原理

克隆机制体现了"共享不可变数据，隔离可变状态"的并发设计原则。数据载荷在同一个流内是不变的（只会被读取、转发），元数据（当前处理层、校验状态）则是变化的。cloned + dataref 的引用计数设计，以极低代价（一个整数加法）实现了无拷贝的多播分发。

高级特性

Netfilter 钩子点

Netfilter 在数据包路径的关键节点插入钩子（hooks），内核模块可在这些点注册回调，实现防火墙、NAT、连接跟踪等功能。sk_buff 会在这些点被检查或修改。

图8：Netfilter 五大钩子点与数据包路径

钩子位置典型用途
NF_INET_PRE_ROUTINGIP 层入口，路由决策前DNAT、连接跟踪
NF_INET_LOCAL_IN路由到本机后，交给 socket 前INPUT 链过滤
NF_INET_FORWARD转发路径中间FORWARD 链过滤
NF_INET_LOCAL_OUT本机发出，路由决策前OUTPUT 链过滤
NF_INET_POST_ROUTING路由后，发出前SNAT、MASQUERADE

🔬 第一性原理

Netfilter 钩子是"开放-封闭原则"在内核中的应用：核心路径对扩展开放，对修改封闭。防火墙、NAT、连接跟踪模块通过注册钩子函数插入处理逻辑，而无需修改核心转发代码。每个钩子都接收同一个 sk_buff 指针，模块可读取、修改或丢弃数据包。

高级特性

NAPI 与 GRO —— 高速接收优化

NAPI（New API）

在旧内核中，每个数据包到达都触发一次硬件中断。高速网络下（百万包/秒）中断开销极大。NAPI 将接收改为"中断+轮询"混合模式：

  1. 第一个包触发硬件中断 → 调用 netif_rx_schedule() → 禁用该设备中断
  2. 注册设备到 softnet_data.poll_list，触发 NET_RX_SOFTIRQ
  3. ksoftirqd 运行 net_rx_action() → 调用驱动 poll() 批量处理
  4. 处理完 budget 后重启中断（或超时重启）

GRO（Generic Receive Offload）

GRO 是软件实现的接收聚合：将多个属于同一 TCP 流的小 sk_buff 合并为一个大 sk_buff，再交给协议栈处理。

效果： 1Gbps 流量在 1460 字节分段下约 85,000 包/秒。经 GRO 聚合后可减少 10-20 倍的协议栈调用次数，显著降低 CPU 负载。

dev_gro_receive() 在 NAPI poll 中匹配同一 5 元组的连续包，合并后通过 netif_receive_skb() 上送协议栈。

图9：NAPI 接收流程与 GRO 效果对比

🔬 第一性原理

NAPI 和 GRO 都遵循"批处理摊销固定开销"原则。每次协议栈调用（ip_rcv、tcp_v4_rcv）都有固定的调度、锁、状态机开销。通过批量处理（NAPI poll）和聚合（GRO），将这些固定成本摊销到更多字节上，使得边际处理成本趋近于零，等效于软件层面的分段卸载。

设计总结

sk_buff 设计哲学总结

图10：sk_buff 六大设计原则总览

两次不可避免的拷贝

Linux 网络栈通过 sk_buff 将内核态的数据拷贝次数压缩到理论最小值 —— 2次：

  1. 第1次（发送）：用户空间 → 内核空间（skb_add_data() / skb_copy_datagram_iovec()）。这是跨越地址空间边界的必要拷贝。
  2. 第2次（发送）：内核空间 → 网卡 DMA 缓冲区。这是将数据交给硬件的必要拷贝（零拷贝 NIC 可消除此步）。

接收方向同样 2 次：① DMA 写入 → sk_buff 数据区；② sk_buff 数据区 → 用户空间（sendfile/splice 等零拷贝技术可消除接收侧的第2次拷贝）。

🔬 第一性原理 —— 终极总结

从第一性原理看，网络传输的本质是：在两个不信任的地址空间之间，通过可靠的协议约定安全地传递字节序列。内核的根本任务是：① 以最小拷贝次数搬运数据；② 在协议层间高效传递描述信息；③ 以公平的资源核算防止内存耗尽。

sk_buff 是这三个目标的统一解：指针而非数据的移动满足①；内联 cb[] 与头部偏移量满足②；truesize 与 users 引用计数满足③。整个设计围绕"避免不必要的工作"这一极简原则展开。
