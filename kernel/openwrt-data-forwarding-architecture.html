00 架构总览

基于 OpenWrt 的家庭路由器在 MCU（运行 Linux 内核）与一颗 8 端口交换芯片之间， 建立了一套完整的软硬件协同数据转发框架。 交换芯片的 7 个端口分别连接 LAN 设备与 WAN 上行，第 8 个端口为 CPU 口， 通过以太网 MAC（如 RGMII/SGMII）与 SoC 相连，是所有需要 CPU 参与的流量的唯一通路。

OpenWrt 从 21.02 起全面采用 Linux 内核的 DSA（Distributed Switch Architecture） 框架取代旧有的 swconfig， 每个物理交换端口在内核中被抽象为独立的 netdev（如 lan1、wan）， 再通过 Linux 网桥（br-lan）将 LAN 口汇聚，WAN 口独立暴露给上层路由/NAT 模块。

图1：OpenWrt 路由器整体软硬件架构（MCU + 8端口交换芯片）
🔬 第一性原理

路由器的本质是一台有两张以上网络接口的计算机，通过软件决策将数据从一个接口转发到另一个接口。 交换芯片解决了"同一子网内多设备高速互联"的问题（二层转发，不经 CPU）； CPU 运行的 Linux 解决了"跨越不同子网"的问题（三层路由，必须经 CPU 决策）。 DSA 架构将二者有机整合，使 CPU 既能控制交换芯片，又能高效处理需要三层介入的流量。

01 硬件拓扑与交换芯片

典型物理连接拓扑

家庭路由器的硬件核心是一颗集成了多个以太网 PHY 的交换芯片（如 Qualcomm QCA8337、MediaTek MT7531、Realtek RTL8370 等）。 在 8 端口配置中，端口分配通常如下：

端口号连接对象VLAN 分组角色
Port 0WAN（光猫/上行）VLAN 2外网口
Port 1LAN 设备 1VLAN 1局域网
Port 2LAN 设备 2VLAN 1局域网
Port 3LAN 设备 3VLAN 1局域网
Port 4LAN 设备 4VLAN 1局域网
Port 5LAN 设备 5VLAN 1局域网
Port 6LAN 设备 6VLAN 1局域网
Port 7CPU (SoC ETH MAC)VLAN 1+2 TaggedCPU 口

CPU 口的特殊性

CPU 口（Port 7）是交换芯片与 SoC 之间的唯一物理通道，工作在较高速率（通常 1Gbps RGMII 或更高速 SGMII/SerDes）。 所有需要 CPU 参与的帧（三层路由、防火墙检查、PPPoE 封包等）都必须经此口进出。 为了让 CPU 区分帧来自哪个逻辑端口，交换芯片在帧中插入 DSA Tag（厂商专属标记）， CPU 驱动据此识别源端口并路由到对应的虚拟网卡。

图2：物理连接拓扑 — 交换芯片 Port 7 通过 RGMII 连接 SoC ETH MAC
🔬 第一性原理

交换芯片的本质是一个高速硬件哈希表：它记录了每个 MAC 地址对应的端口（FDB，Forwarding Database）。 收到帧后查表，找到目标端口就直接在硬件层面转发，速度远超 CPU 软件处理。 CPU 口只是交换芯片众多端口中的一个，特殊之处在于它连接的不是外部设备而是运行 OS 的处理器。 当交换芯片发现帧的目的地需要三层决策（跨网段）时，就将帧送往 CPU 口，交由操作系统处理。

02 DSA 分布式交换架构

DSA（Distributed Switch Architecture）是 Linux 内核中的交换机管理子系统，最初于 2008 年为 Marvell 系列交换芯片设计， 后扩展支持 Qualcomm、Broadcom、MediaTek、Realtek 等主流厂商。 OpenWrt 21.02 起以 DSA 全面取代 swconfig，成为主流方案。

内核官方文档：Documentation/networking/dsa/dsa.txt The Good Penguin: An Overview of DSA

核心设计理念

DSA 将每个交换机前面板端口抽象为独立的 Linux 网络设备（slave netdev）， 使标准工具（ip link、bridge、ethtool）可以直接操作每个端口， 无需了解底层硬件细节。

术语含义Linux 设备名示例
master netdevSoC 侧 ETH MAC 对应的主网络设备，DSA 在此注册 hooketh0
CPU port交换芯片连接 CPU 的物理端口，DSA 不为此创建 netdev（无独立 netdev）
slave netdev每个前面板交换端口对应的虚拟网络设备lan1 ~ lan6、wan
DSA tag交换芯片在帧中插入的厂商专属标记，标识源/目的端口QCA tag (2字节), Broadcom tag (4字节)

DSA Tag 机制（接收方向）

如内核文档所述，DSA 在 eth_type_trans() 中注册钩子，当检测到设备上启用了 DSA（netdev_uses_dsa(dev)）， 将 skb->protocol 设置为 ETH_P_XDSA，触发 DSA 专用的收包处理函数： 

/* net/ethernet/eth.c */
__be16 eth_type_trans(struct sk_buff *skb, struct net_device *dev)
{
    if (unlikely(netdev_uses_dsa(dev)))
        return htons(ETH_P_XDSA);
    /* ... 普通以太网处理 ... */
}

DSA 接收链路（以 Qualcomm QCA tag 为例）：

/* net/dsa/tag_qca.c */
static struct sk_buff *qca_tag_rcv(struct sk_buff *skb, struct net_device *dev)
{
    /* 从 DSA tag 中提取源端口号 */
    port = FIELD_GET(QCA_HDR_RECV_SOURCE_PORT, hdr);
    /* 将 skb->dev 更新为对应的 slave netdev（如 lan2） */
    skb->dev = dsa_conduit_find_user(dev, 0, port);
    /* 剥离 DSA tag，恢复原始以太网帧 */
    skb_pull_rcsum(skb, QCA_HDR_LEN);
    dsa_strip_etype_header(skb, QCA_HDR_LEN);
    return skb;
}
图3：CPU 口传输的以太网帧中嵌入 DSA Tag，驱动接收时自动剥离
🔬 第一性原理

DSA 本质上解决了"多路复用"问题：一条物理链路（CPU 口）需要承载来自多个逻辑端口的流量。 Tag 就是这条共享通道上的"地址标签"，让接收方（CPU 驱动）能区分每个帧的逻辑来源。 这与 VLAN 802.1Q 的理念相同，只是 Tag 格式由各厂商自定义以承载更多控制信息。

03 驱动层与数据帧接收

Linux 内核处理数据包接收采用 NAPI（New API） 机制， 将高频中断转为软中断轮询，大幅降低每包处理开销。 完整的接收路径如下：

Linux Kernel Internals: Life of a Packet (receive)
图4：Linux 内核完整数据包接收路径（NIC → 协议栈）

GRO（Generic Receive Offload）

GRO 在 NAPI 轮询阶段将同一 TCP 流的多个小包合并为一个大 sk_buff， 再一次性送往协议栈，显著降低每包的 CPU 处理开销。 这对于大流量场景（千兆宽带）尤为重要。

NAPI 预算机制： 每次软中断最多处理 64 个数据包（budget），超出时让出 CPU， 防止单个高负载网卡独占 softirq，保证系统响应性。
🔬 第一性原理

网络接收的性能瓶颈在于"中断风暴"：每收到一个帧就触发一次 CPU 中断， 在 Gbps 速率下每秒可产生数百万次中断，开销远大于数据处理本身。 NAPI 的解法是：中断只负责"告知有事发生"，真正的处理改为批量轮询。 这是从中断驱动转向轮询驱动的思路——以轻微延迟换取吞吐量的巨大提升。

04 Linux 网桥与 VLAN

DSA 为每个 LAN 物理端口创建独立的 slave netdev（lan1~lan6）， OpenWrt 通过 Linux 内核的网桥（bridge）将它们汇聚为 br-lan， 模拟传统二层交换机的行为：同一网桥内的设备可以直接二层通信。

OpenWrt Network Interfaces — DSA VLAN configuration

典型 /etc/config/network 配置（DSA 模式）

# 定义网桥设备，将 DSA slave ports 汇聚
config device
    option name   br-lan
    option type   bridge
    list ports    lan1
    list ports    lan2
    list ports    lan3
    list ports    lan4
    list ports    lan5
    list ports    lan6

# LAN 接口绑定到网桥
config interface 'lan'
    option device   br-lan
    option proto    static
    option ipaddr   192.168.1.1
    option netmask  255.255.255.0

# WAN 接口独立（PPPoE 或 DHCP）
config interface 'wan'
    option device   wan      # DSA slave netdev
    option proto    pppoe

VLAN 隔离原理

交换芯片通过 VLAN 将 LAN 端口（VLAN 1）与 WAN 端口（VLAN 2）在硬件层面隔离。 CPU 口以 Tagged 模式同时加入 VLAN 1 和 VLAN 2，DSA 驱动通过解析 Tag 区分两种流量：

图5：VLAN 隔离 — LAN（VLAN 1）与 WAN（VLAN 2）由交换芯片硬件隔离
🔬 第一性原理

网桥的本质是软件模拟的二层交换机，维护 MAC 地址表（FDB），按 MAC 学习和转发帧。 DSA 将硬件交换芯片的 FDB 同步到 Linux 软件 bridge，使内核 bridge 框架能作为统一控制面， 实际的帧转发尽量由硬件完成（Switchdev 卸载），内核 bridge 仅在硬件无法处理时介入。 VLAN 的本质是在共享介质上添加"颜色标签"实现逻辑隔离，节省物理端口。

05 Netfilter / nftables

Netfilter 是 Linux 内核的数据包处理框架，在 IP 协议栈的关键节点注册 hook 点， 供防火墙、NAT、连接跟踪（conntrack）等模块挂载处理函数。 OpenWrt 22.03+ 默认使用 nftables（由 fw4 生成规则），旧版使用 iptables。

Linux Networking Stack — Netfilter hooks

五个核心 Hook 点

图6：Netfilter 五个 Hook 点与数据包流向
Hook 点触发时机典型功能
PREROUTING路由决策前，帧刚进入 IP 栈conntrack、DNAT（端口映射目标地址转换）
INPUT路由决策后，目标为本机防火墙过滤入站请求（SSH、LuCI）
FORWARD路由决策后，目标为其他主机防火墙过滤转发流量（LAN→WAN 规则）
OUTPUT本机产生数据包，路由决策前过滤路由器自身发出的流量
POSTROUTING离开路由器之前，路由决策后SNAT/MASQUERADE（源地址伪装，LAN→WAN NAT）

连接跟踪（conntrack）

Netfilter 的连接跟踪模块（nf_conntrack）为每条 TCP/UDP 连接维护状态（NEW、ESTABLISHED、RELATED）。 WAN 侧返回的响应包被识别为 ESTABLISHED 状态后，可以直接通过防火墙， 无需逐条匹配规则，这是 OpenWrt 默认防火墙策略（WAN DROP）下能正常上网的关键。

🔬 第一性原理

防火墙的本质是在数据通路上设置"检查站"，按规则决定放行或丢弃。 Netfilter 的 hook 点设计遵循了最小权限原则： 默认阻断所有来自 WAN 的主动连接（DROP），只允许由 LAN 侧发起的连接的响应回包通过（ESTABLISHED）， 从根本上防止外网主动攻击内部设备，同时不影响正常上网。

06 IP 路由决策

数据包经过 PREROUTING hook 后，进入 ip_rcv_finish()， 调用 ip_route_input_noref() 查询 FIB（Forwarding Information Base，转发信息库）， 决定数据包的命运：

Kernel Internals: ip_rcv → ip_rcv_finish → routing decision
图7：IP 路由决策树 — FIB 查询决定本地交付、转发或丢弃

OpenWrt 路由表示例

# ip route show
default via 202.x.x.1 dev ppp0          # WAN 默认路由（PPPoE拨号）
192.168.1.0/24 dev br-lan proto kernel  # LAN 直连路由
# 或 DHCP 模式：
default via 1.2.3.1 dev wan             # WAN 默认路由
🔬 第一性原理

路由的本质是最长前缀匹配（Longest Prefix Match）： 对目标 IP 地址，在路由表中找最精确的网段匹配项，按其指示的出口转发。 FIB 是路由表的优化存储形式（Trie 树），将查找从 O(n) 降至 O(k)（k 为地址位数）。 OpenWrt 路由器的 FIB 只需维护极简路由：直连 LAN（192.168.1.0/24）和默认路由（0.0.0.0/0 → WAN）。

07 LAN → LAN 转发路径

当同一局域网内两台设备互相通信时（如 PC 访问 NAS），属于二层转发。 交换芯片直接在硬件 FDB 中查找目标 MAC 地址对应的端口，无需经过 CPU，延迟极低。

图8：LAN → LAN 二层转发（硬件交换，CPU 不参与）

完整步骤

步骤位置动作
1PC（LAN1/Port 1）构造以太网帧，目标 MAC = NAS 的 MAC，发出
2交换芯片入口在 FDB 中查找目标 MAC，命中 → Port 4
3交换芯片内部硬件直接将帧从 Port 4 输出，全程不经 CPU 口
4NAS（LAN4/Port 4）接收到以太网帧
MAC 学习：交换芯片通过监听每个端口收到的帧的源 MAC 地址，自动更新 FDB 表（MAC Learning）。 若 FDB 中没有目标 MAC（未知单播），则泛洪（Flood）到同 VLAN 的所有端口。
🔬 第一性原理

LAN 内部通信不需要路由（不跨网段），只需要二层寻址（MAC 地址）。 交换芯片硬件 FDB 的存在意义是：避免广播风暴——如果每次都泛洪，带宽和 CPU 资源会被大量消耗。 学习→记录→精准转发，是从广播到单播的演化，也是以太网交换机相对于早期集线器（Hub）的本质进步。

08 LAN → WAN 转发路径

当 LAN 内设备访问互联网时（如 PC 请求 www.example.com）， 数据包需要跨越 LAN（192.168.1.0/24）和 WAN（公网 IP）两个不同网段， 必须经过 CPU 进行三层路由 + NAT（网络地址转换）处理。

图9：LAN → WAN 完整三层转发路径（含 NAT/MASQUERADE）

NAT（网络地址转换）详解

家庭路由器通常只有一个公网 IP（WAN IP），而内部可能有数十台设备。 MASQUERADE（地址伪装）是 SNAT 的动态版本： 自动将所有出站包的源 IP 改为 WAN 接口的当前 IP，并记录映射关系： 

内网包出去：
  src: 192.168.1.2:1234 → dst: 8.8.8.8:80
         ↓ MASQUERADE
  src: 202.x.x.1:56789 → dst: 8.8.8.8:80

conntrack 记录：
  (192.168.1.2:1234 → 8.8.8.8:80) ←→ (202.x.x.1:56789 → 8.8.8.8:80)
🔬 第一性原理

NAT 本质上是一种地址空间映射，解决了 IPv4 地址稀缺的工程问题。 其代价是破坏了网络的端对端透明性——外部无法主动连接内部设备（反而提供了安全隔离）。 连接跟踪（conntrack）是 NAT 能正确工作的基础：它记录了"谁先发起连接"， 保证响应包能被正确逆映射回原始内网设备，否则路由器无法知道该把响应包转发给哪台内网机器。

09 WAN → LAN 响应路径

互联网返回的响应数据包（如 HTTP 响应）从 WAN 口进入路由器， 经过 conntrack 反向映射后，找到对应的内网设备并转发回去。 这是 LAN→WAN 路径的逆向过程，但逻辑有所不同。

图10：WAN → LAN 响应路径（conntrack 逆映射 + 三层转发回内网）

流量类型conntrack 状态防火墙默认动作
LAN 发起的响应回包ESTABLISHED / RELATEDACCEPT（放行）

端口转发规则匹配NEW（有 DNAT 规则）DNAT → 内网转发

🔬 第一性原理

WAN → LAN 响应路径的核心是 conntrack 的有状态检测（Stateful Inspection）。 与无状态防火墙（只看单包规则）不同，有状态防火墙记住了"谁先主动发起连接"， 因此能区分"合法的响应包"和"外部主动攻击包"——前者放行，后者阻断。 这是现代网络安全的基础思想：信任由内到外的主动请求，拒绝由外到内的主动连接。

10 硬件卸载加速

对于千兆宽带场景，所有流量都经 CPU 处理会成为瓶颈。 Linux 内核的 Flow Offload（流卸载）和 HW NAT 机制 允许将已建立连接的数据包处理卸载到硬件（交换芯片或 SoC NPU），绕过软件协议栈，大幅提升吞吐量。

OpenWrt Forum: Netfilter "Flow offload" / HW NAT

软件 Flow Offload

当一条连接的前几个包经过完整软件路径（PREROUTING → FORWARD → POSTROUTING）确认放行后， Netfilter 将该连接的五元组信息（src IP/port、dst IP/port、协议）加入 Flow Table， 后续同一连接的数据包在 PREROUTING 阶段命中 Flow Table 后，直接跳过防火墙规则链， 减少 CPU 指令路径约 50%。

硬件 Flow Offload

支持 Switchdev 的交换芯片可以将 Flow Table 下推到硬件： 建立连接后，交换芯片直接在硬件执行 MAC 地址替换、IP/端口改写（NAT）、TTL 递减， 数据包完全不经过 CPU，实现接近线速的转发。

图11：慢速路径（完整协议栈）vs. 软件/硬件卸载快速路径

与 QoS 的冲突

流量卸载绕过了 Linux 的流量控制（TC/QDisc）层，导致 QoS 规则失效。 如需精细的带宽管理（限速、优先级队列），应禁用硬件卸载。 这是性能与控制能力之间的典型工程取舍。

🔬 第一性原理

卸载的本质是用硬件专用逻辑替代通用 CPU 指令。 通用 CPU 的优势是灵活，劣势是每条指令都有开销；专用硬件逻辑的优势是极低延迟，劣势是功能固化。 Flow Offload 在两者之间寻找平衡：用 CPU 处理"初次决策"（哪些流量允许通过）， 用硬件执行"重复性动作"（大量数据的实际转发），本质上是缓存（caching）思想的体现。
