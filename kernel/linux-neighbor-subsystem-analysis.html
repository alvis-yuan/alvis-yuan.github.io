Linux 5.10.186 内核邻居子系统深度解析

从设计原理、模式到具体实现的完整剖析  ·  源码参考： net/core/neighbour.c、 net/ipv4/arp.c、 include/net/neighbour.h

0引言与背景

在现代计算机网络体系中，数据包从发送方到接收方需要跨越多个协议层。当一个 IP 数据包被路由到下一跳时，内核必须知道目标节点在链路层（L2）上的物理地址（如 MAC 地址），才能构建正确的二层帧头。邻居子系统（Neighbouring Subsystem）正是承担这一职责的内核核心组件。

邻居子系统提供了一个协议无关的基础设施，用于完成 L3（网络层）地址到 L2（链路层）地址的映射。对于 IPv4 使用 ARP（地址解析协议），对于 IPv6 使用 NDISC（邻居发现协议），二者均构建在统一的邻居核心框架之上。

图 0-1：邻居子系统在 Linux 网络栈中的位置
📌 核心文件
邻居子系统核心代码位于 net/core/neighbour.c 和 include/net/neighbour.h。ARP 实现在 net/ipv4/arp.c，NDISC 在 net/ipv6/ndisc.c。
⚡ 第一性原理

网络通信的本质是"把数据从 A 送到 B"。在共享链路上，物理传输需要明确的硬件地址。邻居子系统解决的根本问题是：给定一个 IP 地址，如何找到与之对应的硬件地址？这一需求不依赖于具体协议（IPv4/IPv6），因此内核抽象出统一框架，协议仅作为"插件"注入。

1设计目标与架构

1.1 设计目标

Linux 邻居子系统的设计需要同时满足以下关键目标：

目标 说明 实现机制
协议无关性 同一套基础设施支持 ARP、NDISC 等不同协议 neigh_table + 函数指针 constructor/ops
高性能 快速路径下避免重复地址解析，直接使用缓存 hh_cache 硬件头缓存 + output() 函数指针
并发安全 多核环境下的读写并发访问 RCU + rwlock + seqlock 分层锁
可达性检测 自动探测邻居的可达性，维护状态机 NUD 状态机 + 定时器
内存管理 自动清理过期/失效条目，防止内存泄漏 垃圾回收（同步 + 异步）
可配置性 通过 sysctl / procfs 暴露调优参数 neigh_parms + sysctl 表

1.3 设计模式分析

邻居子系统采用了以下经典的软件设计模式：

  • 策略模式（Strategy Pattern）： neigh_ops 结构体封装了不同状态下的输出函数（output、connected_output），运行时可动态切换。
  • 模板方法模式（Template Method Pattern）： 通用核心定义算法骨架（查找、创建、更新、GC），具体协议通过 constructor、solicit、hash 等回调注入协议细节。
  • 工厂模式（Factory Pattern）： neigh_create() / ___neigh_create() 统一负责邻居对象的分配与初始化。
  • 观察者模式： 通过 netlink 的 RTM_NEWNEIGH/RTM_DELNEIGH 消息向用户空间通知状态变化。
⚡ 第一性原理

系统设计的最根本约束是：变化与不变的分离。地址解析的流程是稳定的（查找→未找到→发送探测→等待应答→缓存），而协议细节（报文格式、探测方式）是变化的。邻居子系统将不变的部分固化为核心框架，将变化的部分通过函数指针暴露为"插槽"，这是所有优秀框架设计的核心逻辑。

2核心数据结构

2.1 struct neighbour — 邻居条目

struct neighbour 是邻居子系统的最基本数据结构，代表同一链路上的一个网络节点。定义于 include/net/neighbour.h：

/* include/net/neighbour.h */
struct neighbour {
    struct neighbour __rcu  *next;        /* 哈希表同桶中下一个邻居 */
    struct neigh_table     *tbl;         /* 所属邻居表 */
    struct neigh_parms     *parms;        /* 配置参数 */
    unsigned long           confirmed;    /* 最后一次可达性确认时间 */
    unsigned long           updated;      /* 最后一次状态更新时间 */
    rwlock_t                lock;         /* 保护 nud_state 等字段 */
    refcount_t              refcnt;       /* 引用计数 */
    unsigned int            arp_queue_len_bytes;
    struct sk_buff_head     arp_queue;    /* 待解析报文队列 */
    struct timer_list       timer;        /* NUD 状态机定时器 */
    unsigned long           used;         /* 最后一次使用时间 */
    atomic_t                probes;       /* 已发送探测次数 */
    __u8                    flags;        /* NTF_xxx 标志位 */
    __u8                    nud_state;    /* NUD 状态（核心状态机） */
    __u8                    type;         /* RTN_xxx 类型 */
    __u8                    dead;         /* 标记为待删除 */
    u8                      protocol;
    seqlock_t               ha_lock;      /* 保护硬件地址 */
    unsigned char           ha[ALIGN(MAX_ADDR_LEN, sizeof(unsigned long))]
                                __aligned(8); /* 硬件地址 (MAC) */
    struct hh_cache         hh;           /* L2 帧头缓存（快速路径） */
    int (*output)(struct neighbour *, struct sk_buff *); /* 发送函数指针 */
    const struct neigh_ops *ops;         /* 操作函数集合 */
    struct list_head        gc_list;      /* GC 链表节点 */
    struct rcu_head         rcu;          /* RCU 回调头 */
    struct net_device      *dev;         /* 关联的网络设备 */
    u8                      primary_key[]; /* IP 地址（变长，查找键） */
} __randomize_layout;

字段类型说明
nud_state__u8NUD 状态机当前状态（INCOMPLETE/REACHABLE/STALE/DELAY/PROBE/FAILED/NOARP/PERMANENT）
ha[]unsigned char[]硬件地址（MAC），用 seqlock 保护，允许无锁读取

output函数指针依据 nud_state 动态切换，connected→neigh_connected_output，suspect→neigh_resolve_output

2.3 struct neigh_ops — 操作集合

neigh_ops 定义了邻居的行为接口，类似于面向对象的"虚函数表"：

struct neigh_ops {

⚡ 第一性原理

数据结构的本质是"把相关的数据放在一起"。struct neighbour 把一个邻居的全部知识（地址、状态、缓存、定时器、统计）封装为一个完整的认知单元，使得对邻居的所有操作都围绕一个对象展开，这正是"高内聚、低耦合"的直接体现。primary_key[] 使用零长数组做变长扩展，是 C 语言对"按需分配"的优雅处理。

3NUD 状态机

NUD（Network Unreachability Detection，网络不可达检测）是邻居子系统的核心协议机制，定义了邻居条目的生命周期状态转换。

3.1 NUD 状态定义

状态十六进制说明L2 地址有效？
NUD_NONE 0x00 初始状态，条目刚分配，未开始解析 ❌
NUD_INCOMPLETE 0x01 地址解析进行中，已发送请求，等待应答 ❌
NUD_REACHABLE 0x02 邻居可达，条目有效，在 reachable_time 内有效 ✅
NUD_STALE 0x04 条目有效但可能过期，下次使用时触发验证 ✅（可用）
NUD_DELAY 0x08 发包到 STALE 邻居后，等待上层协议确认 ✅（可用）
NUD_PROBE 0x10 DELAY 超时后，主动发送单播探测 ✅（可用）
NUD_FAILED 0x20 解析失败或探测无响应，邻居不可达 ❌
NUD_NOARP 0x40 设备无需 ARP（如 loopback），地址始终有效 ✅
NUD_PERMANENT 0x80 静态配置，永不过期，仅管理员可删除 ✅
📌 组合宏定义
NUD_VALID = NUD_PERMANENT | NUD_NOARP | NUD_REACHABLE | NUD_PROBE | NUD_STALE | NUD_DELAY
NUD_CONNECTED = NUD_PERMANENT | NUD_NOARP | NUD_REACHABLE
NUD_IN_TIMER = NUD_INCOMPLETE | NUD_REACHABLE | NUD_DELAY | NUD_PROBE

3.3 状态转换关键路径

路径一：正常解析流程

NUD_NONE → NUD_INCOMPLETE → NUD_REACHABLE

当内核需要发送数据包但无对应 L2 地址时，触发 __neigh_event_send()，将数据包压入 arp_queue，发送 ARP 请求（arp_solicit()），状态变为 NUD_INCOMPLETE。收到 ARP 应答后，调用 neigh_update()，状态变为 NUD_REACHABLE，排队数据包批量发出。

路径二：可达性验证流程

NUD_REACHABLE → NUD_STALE → NUD_DELAY → NUD_PROBE → NUD_REACHABLE (或 NUD_FAILED)

可达性定时器到期后转入 NUD_STALE。下次使用时转入 NUD_DELAY，等待 delay_first_probe_time（默认 5s）内的上层确认。若无确认，转入 NUD_PROBE 发送单播探测。收到应答则恢复 NUD_REACHABLE，探测耗尽则变为 NUD_FAILED。

✅ TCP 可以提供可达性"暗示"
TCP 在双向通信时会调用 dst_confirm()，更新 neigh->confirmed 时间戳，使 DELAY 状态直接跳回 REACHABLE，无需发送额外的探测报文，这是上层协议辅助邻居子系统的典型设计。
⚡ 第一性原理

NUD 状态机的本质是"不信任假设 + 代价最小化"。内核不能永久相信一个 MAC-IP 映射是正确的（设备可能下线、IP 可能迁移），但频繁验证开销极高。状态机是这两个约束的最优解：缓存结果（REACHABLE），允许脏读（STALE/DELAY），惰性验证（PROBE），最终一致（FAILED 或 REACHABLE）。这与数据库的读未提交/最终一致性哲学完全相同。

4定时器机制

4.1 per-neighbour 定时器

每个 struct neighbour 都有一个 timer_list，回调函数为 neigh_timer_handler()。该定时器负责驱动 NUD 状态机的时间相关转换。

/* net/core/neighbour.c */
static void neigh_timer_handler(struct timer_list *t)
{
    struct neighbour *neigh = from_timer(neigh, t, timer);
    unsigned int state, now = jiffies;
    int notify = 0;

    write_lock(&neigh->lock);
    state = neigh->nud_state;

    if (state & NUD_REACHABLE) {
        /* REACHABLE 超时 → STALE */
        if (time_before_eq(neigh->confirmed + neigh->parms->reachable_time, now)) {
            neigh->nud_state = NUD_STALE;
            notify = 1;
        } else {
            /* 重新设定定时器到 confirmed + reachable_time */
            neigh_add_timer(neigh, neigh->confirmed + neigh->parms->reachable_time);
        }
    } else if (state & NUD_DELAY) {
        /* DELAY 超时 → PROBE，开始发送探测 */
        neigh->nud_state = NUD_PROBE;
        atomic_set(&neigh->probes, 0);
        notify = 1;
        neigh_add_timer(neigh, jiffies + NEIGH_VAR(neigh->parms, RETRANS_TIME));
        neigh_probe(neigh);
    } else if (state & NUD_PROBE) {
        /* PROBE：检查探测次数是否耗尽 */
        if (atomic_read(&neigh->probes) >= neigh_max_probes(neigh)) {
            neigh->nud_state = NUD_FAILED;
            notify = 1;
            neigh_invalidate(neigh); /* 清空 arp_queue，报告不可达 */
        } else {
            neigh_add_timer(neigh, jiffies + NEIGH_VAR(neigh->parms, RETRANS_TIME));
            neigh_probe(neigh);
        }
    }
    /* ... */
}

4.3 neigh_max_probes() 探测次数计算

static __inline__ int neigh_max_probes(struct neighbour *n)
{
    struct neigh_parms *p = n->parms;
    /* 单播探测次数 + 应用层探测 + 多播探测（或重探测）*/
    return NEIGH_VAR(p, UCAST_PROBES) +
           NEIGH_VAR(p, APP_PROBES) +
           (n->nud_state & NUD_PROBE ?
               NEIGH_VAR(p, MCAST_REPROBES) :
               NEIGH_VAR(p, MCAST_PROBES));
}

默认值：UCAST_PROBES=3，APP_PROBES=0，MCAST_PROBES=3，因此最多发送 6 次探测后进入 NUD_FAILED。

⚡ 第一性原理

定时器的本质是把"时间"作为一等公民纳入状态机。网络状态天然具有时效性——"5 秒前可达"和"现在可达"是两件不同的事情。定时器将隐式的时间维度显式化，每个状态的有效期受时间约束，这是网络协议对"不确定性的时间管理"的工程实践。

5邻居表初始化

5.1 neigh_table_init() 流程

IPv4 在 arp_init() 中调用 neigh_table_init() 初始化 ARP 表：

/* net/ipv4/arp.c */
void __init arp_init(void)
{
    neigh_table_init(NEIGH_ARP_TABLE, &arp_tbl); /* 初始化邻居表 */
    dev_add_pack(&arp_packet_type);              /* 注册 ARP 报文处理器 */
    arp_proc_init();                             /* /proc/net/arp */
    neigh_sysctl_register(NULL, &arp_tbl.parms, NULL); /* sysctl 条目 */
    register_netdevice_notifier(&arp_netdev_notifier); /* 网卡事件 */
}

/* net/core/neighbour.c */
void neigh_table_init(int index, struct neigh_table *tbl)
{
    /* 1. 初始化链表和引用计数 */
    INIT_LIST_HEAD(&tbl->parms_list);
    INIT_LIST_HEAD(&tbl->gc_list);
    list_add(&tbl->parms.list, &tbl->parms_list);
    write_pnet(&tbl->parms.net, &init_net);
    refcount_set(&tbl->parms.refcnt, 1);
    
    /* 2. 计算随机化的可达时间 */
    tbl->parms.reachable_time =
        neigh_rand_reach_time(NEIGH_VAR(&tbl->parms, BASE_REACHABLE_TIME));
    
    /* 3. 分配 per-CPU 统计 */
    tbl->stats = alloc_percpu(struct neigh_statistics);
    
    /* 4. 初始化哈希表（初始 2^3=8 个桶）*/
    RCU_INIT_POINTER(tbl->nht, neigh_hash_alloc(3));
    
    /* 5. 分配代理 ARP 哈希桶 */
    tbl->phash_buckets = kzalloc(phsize, GFP_KERNEL);
    
    /* 6. 计算 entry_size（包含 primary_key 的变长部分）*/
    if (!tbl->entry_size)
        tbl->entry_size = ALIGN(offsetof(struct neighbour, primary_key)
                                + tbl->key_len, NEIGH_PRIV_ALIGN);
    
    /* 7. 启动异步 GC 工作队列 */
    INIT_DEFERRABLE_WORK(&tbl->gc_work, neigh_periodic_work);
    queue_delayed_work(system_power_efficient_wq, &tbl->gc_work,
                       tbl->parms.reachable_time);
    
    /* 8. 初始化 Proxy ARP 定时器 */
    timer_setup(&tbl->proxy_timer, neigh_proxy_process, 0);
    
    /* 9. 注册到全局表数组 */
    neigh_tables[index] = tbl;
}

⚡ 第一性原理

初始化的本质是"建立已知世界的边界"。初始化函数确定了：哈希表初始容量（8桶，可动态扩张）、GC 阈值（128/512/1024）、默认超时（30s 可达时间）。这些数字不是随意的，而是基于典型局域网规模（百台主机以内）和网络稳定性假设精心选取的经验值，可通过 sysctl 覆盖。

6邻居创建与查找

6.1 查找流程

/* 查找邻居，不存在时可选择创建 */
static inline struct neighbour *
__neigh_lookup(struct neigh_table *tbl, const void *pkey,
               struct net_device *dev, int creat)
{
    struct neighbour *n = neigh_lookup(tbl, pkey, dev);
    if (n || !creat)
        return n;
    n = neigh_create(tbl, pkey, dev);
    return IS_ERR(n) ? NULL : n;
}

neigh_lookup() 通过哈希表快速定位：

struct neighbour *neigh_lookup(struct neigh_table *tbl,
                              const void *pkey,
                              struct net_device *dev)
{
    struct neighbour *n;
    NEIGH_CACHE_STAT_INC(tbl, lookups);
    rcu_read_lock();               /* RCU 读临界区 */
    n = __neigh_lookup_noref(tbl, pkey, dev);
    if (n) {
        if (!refcount_inc_not_zero(&n->refcnt))
            n = NULL;
        NEIGH_CACHE_STAT_INC(tbl, hits);
    }
    rcu_read_unlock();
    return n;
}

6.2 创建流程

___neigh_create() 是核心创建函数：

static struct neighbour *___neigh_create(struct neigh_table *tbl,
                                         const void *pkey,
                                         struct net_device *dev,
                                         bool exempt_from_gc, bool want_ref)
{
    /* Step 1: 分配内存并初始化基本字段 */
    n = neigh_alloc(tbl, dev, exempt_from_gc);
    
    /* Step 2: 复制 L3 地址到 primary_key */
    memcpy(n->primary_key, pkey, key_len);
    n->dev = dev;
    
    /* Step 3: 调用协议特定构造函数（如 arp_constructor）*/
    if (tbl->constructor && (error = tbl->constructor(n)) < 0)
        goto out_neigh_release;
    
    /* Step 4: 设置初始时间戳 */
    n->confirmed = jiffies - (NEIGH_VAR(n->parms, BASE_REACHABLE_TIME) << 1);
    
    /* Step 5: 插入哈希表（RCU 写侧）*/
    write_lock_bh(&tbl->lock);
    nht = rcu_dereference_protected(tbl->nht, lockdep_is_held(&tbl->lock));
    
    /* 检查是否已有相同键的条目（race 防护）*/
    neigh_for_each_in_bucket(n1, &nht->hash_heads[hash_val]) {
        if (dev == n1->dev && tbl->key_eq(n1, pkey)) { ... }
    }
    
    /* Step 6: 若哈希表过载，触发扩容 */
    if (atomic_read(&tbl->entries) > (1 << nht->hash_shift))
        nht = neigh_hash_grow(tbl, nht->hash_shift + 1);
    
    /* Step 7: 头插法插入哈希桶 */
    hlist_add_head_rcu(&n->hash, &nht->hash_heads[hash_val]);
    n->dead = 0;
    write_unlock_bh(&tbl->lock);
    
    return n;
}

6.3 neigh_alloc() — 内存分配与 GC 阈值检查

static struct neighbour *neigh_alloc(struct neigh_table *tbl,
                                     struct net_device *dev,
                                     bool exempt_from_gc)
{
    if (!exempt_from_gc) {
        entries = atomic_inc_return(&tbl->gc_entries) - 1;
        /* 超过 gc_thresh3 或 gc_thresh2 且 5s 内未 flush → 强制 GC */
        if (entries >= tbl->gc_thresh3 ||
            (entries >= tbl->gc_thresh2 &&
             time_after(now, tbl->last_flush + 5 * HZ))) {
            if (!neigh_forced_gc(tbl) && entries >= tbl->gc_thresh3)
                goto out_entries;  /* 分配失败 */
        }
    }
    /* kzalloc: entry_size(含primary_key) + dev->neigh_priv_len */
    n = kzalloc(tbl->entry_size + dev->neigh_priv_len, GFP_ATOMIC);
    
    /* 初始化各字段 */
    __skb_queue_head_init(&n->arp_queue);
    rwlock_init(&n->lock);
    seqlock_init(&n->ha_lock);
    n->updated = n->used = now;
    n->nud_state = NUD_NONE;
    n->output    = neigh_blackhole;      /* 初始：丢包 */
    n->parms     = neigh_parms_clone(&tbl->parms);
    timer_setup(&n->timer, neigh_timer_handler, 0);
    refcount_set(&n->refcnt, 1);
    n->dead = 1;  /* 未完成初始化，标记为 dead */
    atomic_inc(&tbl->entries);
    return n;
}

⚡ 第一性原理

查找是最频繁的操作，必须极快；创建是稀有操作，可以稍慢。内核用 RCU 读锁（几乎零开销）保护查找，用 rwlock 写锁保护创建，体现了"读多写少时优化读路径"的基本原则。neigh_alloc() 在分配前检查 GC 阈值，是"创建即承诺资源"的防御性编程——在接受新请求之前先确认资源上限。

7.2 arp_process() 关键逻辑

/* net/ipv4/arp.c - 简化版 */
static int arp_process(struct net *net, struct sock *sk,
                       struct sk_buff *skb)
{
    arp = arp_hdr(skb);
    sha = arp->ar_sha;   /* Sender Hardware Address (MAC) */
    sip = arp->ar_sip;   /* Sender IP */
    tip = arp->ar_tip;   /* Target IP */

    /* 1. 更新发送方的邻居条目（学习 MAC-IP 映射）*/
    n = __neigh_lookup(&arp_tbl, &sip, dev, 0);
    if (n) {
        neigh_update(n, sha, NUD_STALE, NEIGH_UPDATE_F_OVERRIDE, 0);
        neigh_release(n);
    }

    /* 2. 若为 ARP 请求且目标 IP 是本机 → 发送应答 */
    if (arp->ar_op == htons(ARPOP_REQUEST)) {
        if (inet_addr_type(net, tip) == RTN_LOCAL) {
            arp_send_dst(ARPOP_REPLY, ETH_P_ARP, sip, dev,
                         tip, sha, dev->dev_addr, sha, ...);
        }
    }

    /* 3. 若为 ARP 应答 → 更新目标邻居 */
    if (arp->ar_op == htons(ARPOP_REPLY)) {
        n = __neigh_lookup(&arp_tbl, &sip, dev, 0);
        if (n) {
            neigh_update(n, sha, NUD_REACHABLE,
                         NEIGH_UPDATE_F_OVERRIDE | NEIGH_UPDATE_F_ADMIN, 0);
            neigh_release(n);
        }
    }
    return NET_RX_SUCCESS;
}

7.3 ARP 发送路径 — arp_solicit()

/* 发送 ARP 探测请求（由 neigh_probe() 调用）*/
static void arp_solicit(struct neighbour *neigh, struct sk_buff *skb)
{
    /* 1. 确定源 IP：优先与目标同子网的 IP */
    saddr = inet_select_addr(dev, target, RT_SCOPE_LINK);

    probes -= NEIGH_VAR(neigh->parms, UCAST_PROBES);
    if (probes < 0) {
        /* 还在单播探测阶段：使用已知的目标 MAC */
        neigh_ha_snapshot(dst_ha, neigh, dev);
        dst_hw = dst_ha;
    } else {
        probes -= NEIGH_VAR(neigh->parms, APP_PROBES);
        if (probes < 0) {
            neigh_app_ns(neigh); /* 应用层处理 */
            return;
        }
        /* 广播 ARP 请求：dst_hw = NULL (广播) */
    }
    /* 2. 发送 ARP 请求帧 */
    arp_send_dst(ARPOP_REQUEST, ETH_P_ARP, target, dev, saddr,
                 dst_hw, dev->dev_addr, NULL, dst);
}

7.4 arp_constructor() — ARP 邻居初始化

static int arp_constructor(struct neighbour *neigh)
{
    __be32 addr = *(__be32 *)neigh->primary_key;
    struct net_device *dev = neigh->dev;
    struct in_device *in_dev = in_dev_get(dev);

    neigh->type = inet_addr_type(dev_net(dev), addr);

    if (dev->header_ops == NULL) {
        /* 无 L2 头的设备 (PPP 等) → 使用 arp_direct_ops */
        neigh->ops = &arp_direct_ops;
    } else if (dev->header_ops->cache) {
        /* 支持硬件头缓存的设备 → arp_hh_ops */
        neigh->ops = &arp_hh_ops;
    } else {
        neigh->ops = &arp_generic_ops;
    }

    if (neigh->nud_state & NUD_VALID)
        neigh->output = neigh->ops->connected_output;
    else
        neigh->output = neigh->ops->output;
    return 0;
}
⚡ 第一性原理

ARP 的本质是"广播询问 + 单播回答"的分布式查询协议。内核实现遵循了该协议最简约的语义：不问不答，一问一答，顺带学习（看到别人发 ARP 请求时，也把发送方的 MAC-IP 记录下来）。arp_constructor() 中按设备能力选择不同的 neigh_ops，是"装备按需分配"的工程哲学——不是所有邻居都需要相同的处理流程。

/* net/ipv6/ndisc.c - 报文处理分发 */
static int ndisc_rcv(struct sk_buff *skb)
{
    struct nd_msg *msg = (struct nd_msg *)skb_transport_header(skb);

    switch (msg->icmph.icmp6_type) {
    case NDISC_NEIGHBOUR_SOLICITATION:
        ndisc_recv_ns(skb);  /* 邻居请求 → 应答 + 更新 */
        break;
    case NDISC_NEIGHBOUR_ADVERTISEMENT:
        ndisc_recv_na(skb);  /* 邻居应答 → 更新邻居条目 */
        break;
    case NDISC_ROUTER_SOLICITATION:
        ndisc_recv_rs(skb);  /* 路由器请求（主机→路由器）*/
        break;
    case NDISC_ROUTER_ADVERTISEMENT:
        ndisc_router_discovery(skb);  /* 路由器应答 */
        break;
    case NDISC_REDIRECT:
        ndisc_redirect_rcv(skb);  /* 重定向 */
        break;
    }
}

8.3 ndisc_recv_na() — 处理邻居通告

static void ndisc_recv_na(struct sk_buff *skb)
{
    neigh = neigh_lookup(&nd_tbl, &msg->target, dev);
    if (neigh) {
        if (neigh->nud_state & NUD_FAILED)
            goto out;
        
        /* icmp6_solicited=1 → NUD_REACHABLE，否则 NUD_STALE */
        neigh_update(neigh, lladdr,
                     msg->icmph.icmp6_solicited ? NUD_REACHABLE : NUD_STALE,
                     NEIGH_UPDATE_F_WEAK_OVERRIDE |
                     (msg->icmph.icmp6_override ? NEIGH_UPDATE_F_OVERRIDE : 0) |
                     NEIGH_UPDATE_F_OVERRIDE_ISROUTER |
                     (msg->icmph.icmp6_router ? NEIGH_UPDATE_F_ISROUTER : 0));
    out:
        neigh_release(neigh);
    }
}

8.4 重复地址检测 (DAD)

IPv6 在接口配置新地址时，通过 NDISC 发送 NS 报文（源 IP 为全零），若收到 NA 回应则说明地址冲突，触发 addrconf_dad_failure()：

⚠️ DAD 流程
  1. 为新 IPv6 地址分配 IFA_F_TENTATIVE 标志
  2. 发送 NS，目标为 Solicited-Node 组播地址，源 IP 为 ::
  3. 若在 DAD_TIMEOUT 内收到 NA → 地址冲突，调用 addrconf_dad_failure()
  4. 若无响应 → DAD 成功，地址标志从 TENTATIVE 变为 PREFERRED
⚡ 第一性原理

NDISC 相对于 ARP 的改进体现了"用专门机制替代广播"的网络设计哲学。广播（ARP）让所有节点处理每个解析请求，组播（NDISC）让只关心目标地址的节点处理，大幅降低了 IPv6 网络中的"解析噪音"。DAD 则将"我想使用某地址"的声明转化为可验证的协议过程，而不是依赖外部的人工配置保证——这是协议自治性的体现。

9.2 neigh_resolve_output() — 慢速路径

int neigh_resolve_output(struct neighbour *neigh, struct sk_buff *skb)
{
    if (!neigh_event_send(neigh, skb)) {
        /* 邻居已解析，构建 L2 头 */
        rc = dev_hard_header(skb, dev, ntohs(skb->protocol),
                             neigh->ha, NULL, skb->len);
        if (rc >= 0)
            rc = dev_queue_xmit(skb);
    }
    return rc;
}

9.3 neigh_connected_output() — 快速路径（hh_cache）

int neigh_connected_output(struct neighbour *neigh, struct sk_buff *skb)
{
    struct hh_cache *hh = &neigh->hh;

    if (likely(hh->hh_len)) {
        /* 快速路径：直接从缓存 memcpy L2 头 */
        neigh_hh_output(hh, skb);
    } else {
        read_lock_bh(&neigh->lock);
        err = dev_hard_header(skb, dev, ntohs(skb->protocol),
                              neigh->ha, NULL, skb->len);
        read_unlock_bh(&neigh->lock);
        if (err >= 0)
            dev_queue_xmit(skb);
    }
}

9.4 neigh_connect() 与 neigh_suspect()

/* 邻居变为 connected 状态（REACHABLE/NOARP/PERMANENT）*/
static inline void neigh_connect(struct neighbour *neigh) {
    neigh->output = neigh->ops->connected_output;  /* 切换到快速路径 */
}

/* 邻居状态不确定（STALE/DELAY/PROBE）*/
static inline void neigh_suspect(struct neighbour *neigh) {
    neigh->output = neigh->ops->output;            /* 切换到慢速路径 */
}
⚡ 第一性原理

发送路径的设计核心是"零分支的快速路径"。函数指针 output() 把"根据状态决定如何发送"这一判断从热路径移到状态转换时刻。当邻居处于 REACHABLE 状态时，每次发送调用 neigh_connected_output()，内部仅做 memcpy + xmit，没有任何 if-else 判断——这就是把代价最大化地前置到稀有的状态转换时刻，而非付出在最频繁的发送时刻的典型优化思想。

10.1 两种 GC 模式

模式函数触发条件特点

entries ≥ gc_thresh2 且 5s 未 flush		 在 neigh_alloc() 内同步执行，阻塞分配路径

10.2 neigh_forced_gc() — 同步强制回收

static int neigh_forced_gc(struct neigh_table *tbl)
{
    int shrunk = 0;
    NEIGH_CACHE_STAT_INC(tbl, forced_gc_runs);

    write_lock_bh(&tbl->lock);
    /* 遍历所有哈希桶 */
    nht = rcu_dereference_protected(tbl->nht, ...);
    for (i = 0; i < (1 << nht->hash_shift); i++) {
        neigh_for_each_in_bucket_safe(n, tmp, &nht->hash_heads[i]) {
            write_lock(&n->lock);
            /* 删除条件：非永久、引用计数为 1（仅哈希表持有）*/
            if ((n->nud_state == NUD_FAILED) ||
                (!(n->nud_state & NUD_PERMANENT) &&
                  refcount_read(&n->refcnt) == 1)) {
                hlist_del_rcu(&n->hash);
                neigh_mark_dead(n);
                shrunk = 1;
            }
            write_unlock(&n->lock);
        }
    }
    tbl->last_flush = jiffies;
    write_unlock_bh(&tbl->lock);
    return shrunk;
}

10.3 neigh_periodic_work() — 异步周期回收

static void neigh_periodic_work(struct work_struct *work)
{
    NEIGH_CACHE_STAT_INC(tbl, periodic_gc_runs);

    /* 条目数低于 gc_thresh1 → 不执行 GC */
    if (atomic_read(&tbl->entries) < READ_ONCE(tbl->gc_thresh1))
        goto out;

    /* 每 300s 重新随机化 reachable_time */
    if (time_after(jiffies, tbl->last_rand + 300 * HZ)) {
        list_for_each_entry(p, &tbl->parms_list, list)
            p->reachable_time = neigh_rand_reach_time(
                NEIGH_VAR(p, BASE_REACHABLE_TIME));
    }

    for (i = 0; i < (1 << nht->hash_shift); i++) {
        neigh_for_each_in_bucket_safe(n, tmp, &nht->hash_heads[i]) {
            state = n->nud_state;
            /* 跳过永久条目和计时中的条目 */
            if (state & (NUD_PERMANENT | NUD_IN_TIMER)) continue;

            /* 删除条件：refcnt==1 且（FAILED 或超过 gc_staletime）*/
            if (refcount_read(&n->refcnt) == 1 &&
                (state == NUD_FAILED ||
                 !time_in_range_open(jiffies, n->used,
                     n->used + NEIGH_VAR(n->parms, GC_STALETIME)))) {
                hlist_del_rcu(&n->hash);
                hlist_del_rcu(&n->dev_list);
                neigh_mark_dead(n);
                neigh_cleanup_and_release(n);
            }
        }
        /* 每桶后让出 CPU */
        write_unlock_bh(&tbl->lock);
        cond_resched();
        write_lock_bh(&tbl->lock);
    }
out:
    /* 下次 GC：BASE_REACHABLE_TIME/2 后 */
    queue_delayed_work(system_power_efficient_wq, &tbl->gc_work,
        NEIGH_VAR(&tbl->parms, BASE_REACHABLE_TIME) >> 1);
}

⚡ 第一性原理

垃圾回收是"有限内存与无限需求之间的博弈"。内核用三级阈值实现了渐进式压力响应：压力低时不打扰（低于 128 不 GC），压力中等时周期清理（128~1024 定期 GC），压力极高时阻塞分配强制清理（超过 1024 同步 GC），分配失败则拒绝（保证系统不崩溃）。这是一种典型的"反应式"资源管理策略，把 GC 代价与资源压力直接挂钩。

11锁与 RCU 并发

11.1 并发挑战

邻居子系统的并发需求极为复杂：

  • 读操作（数据包发送时查找邻居）：极高频，必须无锁或近无锁
  • 写操作（NUD 状态更新、MAC 地址更新）：低频，可承受短暂锁竞争
  • 哈希表结构变更（扩容、插入、删除）：极低频，可用写锁

11.2 三层锁策略

保护对象锁类型读操作写操作
哈希表结构 (nht) RCU + tbl->lock (rwlock) rcu_read_lock() write_lock_bh(&tbl->lock)
邻居状态 (nud_state 等) neigh->lock (rwlock) read_lock_bh(&neigh->lock) write_lock_bh(&neigh->lock)
硬件地址 (ha[]) neigh->ha_lock (seqlock) read_seqbegin / read_seqretry write_seqlock(&neigh->ha_lock)
邻居引用计数 neigh->refcnt (atomic) refcount_read() neigh_hold() / neigh_release()

11.3 RCU 读取路径示例

/* 读取邻居：使用 RCU 保护哈希表，无需持有任何锁 */
rcu_read_lock();
nht = rcu_dereference(tbl->nht);   /* RCU 安全的指针读取 */
hash_val = tbl->hash(pkey, dev, nht->hash_rnd) >> (32 - nht->hash_shift);
neigh_for_each_in_bucket(n, &nht->hash_heads[hash_val]) {
    if (dev == n->dev && tbl->key_eq(n, pkey)) {
        neigh_hold(n);   /* 增加引用计数，防止 RCU 退出后被释放 */
        rcu_read_unlock();
        return n;        /* 安全返回，引用计数已持有 */
    }
}

11.4 seqlock 保护 ha[] 的无锁读取

/* 读取 MAC 地址（seqlock 实现无阻塞读）*/
static inline void neigh_ha_snapshot(char *dst,
                                      const struct neighbour *n,
                                      const struct net_device *dev)
{
    unsigned int seq;
    do {
        seq = read_seqbegin(&n->ha_lock);
        memcpy(dst, n->ha, dev->addr_len);
    } while (read_seqretry(&n->ha_lock, seq));
    /* 若写者在读取中间修改了 ha[]，seq 不匹配则重试 */
}
图 11-1：RCU 读写并发时序（哈希表查找 vs 哈希表修改）
⚡ 第一性原理

并发控制的最基本矛盾是一致性 vs 性能。强锁（mutex）保证一致性但串行化性能；无锁（lock-free）有极高性能但实现复杂且对业务语义要求严格。RCU 是内核为"读多写少"场景找到的黄金中间点：读者无等待、无锁、极快；写者承担略高代价（等待 grace period）但可以异步进行。seqlock 则是对"读不可忍受锁竞争但必须保证原子读"的专用解答——用序列号检测并重试比持锁读代价更低。

12.2 常见调优场景

⚠️ 大规模云环境（邻居表溢出）
在 Kubernetes 集群或 OpenStack 等大规模场景中，容器/虚拟机数量可能超过默认 1024 的 gc_thresh3 阈值，导致 neighbour: arp_cache: neighbor table overflow! 错误。 
# 扩大邻居表容量（适合大规模节点场景）
sysctl -w net.ipv4.neigh.default.gc_thresh1=4096
sysctl -w net.ipv4.neigh.default.gc_thresh2=8192
sysctl -w net.ipv4.neigh.default.gc_thresh3=16384
📌 高可用场景（快速检测主机下线）
# 加快邻居失效检测
sysctl -w net.ipv4.neigh.default.delay_first_probe_time=2
sysctl -w net.ipv4.neigh.default.ucast_solicit=3
sysctl -w net.ipv4.neigh.default.retrans_time_ms=500
✅ 查看 ARP/NDISC 统计
# ARP 缓存统计（allocs, destroys, hash_grows, res_failed, lookups...）
cat /proc/net/stat/arp_cache

# 查看当前邻居表
ip -s neigh show

# 查看邻居表详细状态
ip neigh show dev eth0

12.3 arp_announce 和 arp_ignore

参数值行为
arp_ignore0（默认）对任何本地 IP 的 ARP 请求均回复
1仅回复接收接口上配置的目标 IP
2回复条件：目标 IP 在接收接口配置，且与发送方同子网
arp_announce0（默认）使用任意本地地址作为 ARP 源 IP
1尽量使用与目标同子网的本地地址
2（LVS推荐）使用最佳本地地址（与目标同子网的主 IP）
⚡ 第一性原理

sysctl 参数的存在是对"没有放之四海而皆准的最优值"这一事实的承认。局域网规模、网络稳定性、故障检测要求因场景而异。内核选择了保守的默认值（面向普通场景），同时通过 sysctl 允许运维人员根据实际约束调整——这是"机制与策略分离"（Mechanism vs Policy）原则在配置层面的体现。

✅ 全文总结

Linux 5.10.186 邻居子系统的核心设计可以归纳为三点：

  1. 协议无关的抽象框架：通过 neigh_table + 函数指针，将 ARP/NDISC 的共性抽象为统一基础设施，差异通过构造函数注入。
  2. 状态机驱动的生命周期管理：NUD 状态机精确描述了邻居可达性的所有可能状态，结合 per-neighbour 定时器实现了完全自治的可达性探测和维护。
  3. 读性能优化的并发设计：RCU 保护热路径查找，seqlock 保护硬件地址读取，函数指针 output() 实现零分支快速发送路径，将并发开销和决策代价降至最低。
