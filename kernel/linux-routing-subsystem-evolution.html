引言
为什么 Linux 路由子系统变得如此复杂？
第一性原理：网络的本质需求驱动架构演化

路由的最原始使命，是回答一个问题：「这个数据包该往哪里发？」。在最朴素的模型里，答案只需查一张以目的 IP 为键的转发表，找到出口即可。

然而，互联网的真实世界远比这复杂。企业需要安全隔离、运营商需要 QoS 差异化、云平台需要多租户隔离、家庭宽带用户需要多线接入均衡负载——每一个新场景都向路由子系统提出新的维度要求。

图 0-1：路由子系统 Key/Value 维度随内核版本演化示意

本文将沿着内核的真实演化轨迹，逐步拆解每一次「加字段」背后的工程动机，并在每章结尾用第一性原理提炼出本质。

第一性原理

路由的本质是一个映射函数：f(包特征) → 转发决策。复杂性不是设计失误，而是现实世界对「包特征」和「转发决策」两端维度扩展的自然响应。每一次内核演化，都是对某类新场景需求的精确回应。

①
Linux 0.99 – 1.x · 约 1992–1994
最初的路由模型：一张表、一个键
目的地址驱动的单一转发表

Linux 最早的路由实现直接继承自 BSD Unix 的思想：维护一张转发信息库（FIB，Forwarding Information Base），以目的 IP 网段为键，查找时取最长匹配前缀（Longest Prefix Match）。

📌 场景：单宿主小型网络
一台 PC 连接一个局域网，通过单一出口路由器访问互联网。所有数据包只需问：「目的地在哪个网段？走哪个接口出去？」
图 1-1：Linux 1.x 路由模型 — 单张 FIB，纯目的地址匹配

这个阶段的路由表结构极为简洁，每条表项仅包含：

维度字段说明
Key 输入dst (目的网段)唯一匹配键，最长前缀匹配
Value 输出dev (出接口)数据包从哪个网卡发出
Value 输出gateway (下一跳)下一跳路由器 IP，可为空（直连）

用户通过 route add 命令（来自 net-tools 包）管理路由，内核通过 ioctl（SIOCADDRT/SIOCDELRT）接收配置。数据结构在 net/ipv4/route.c 中以简单的哈希表或链表实现。

第一性原理

在网络只有「一张出口」的世界里，路由的全部信息量就是「目的地属于哪个网段」。单一 FIB + 最长前缀匹配，是在信息最小化前提下的最优解。复杂性出现的唯一理由，是「单一目的地址」不再能区分所有需要差异化处理的流量。

②
Linux 2.0 – 2.1 · 约 1996–1997
路由缓存、TOS 与 Realm
性能优化与首个「附加字段」：服务类型匹配

随着 Linux 作为网关路由器的使用场景增多，性能瓶颈出现了：每个数据包都需要做最长前缀匹配的全表查找，开销较大。内核在 2.0 系列引入了路由缓存（Routing Cache），并同时增加了对 IP 头部 TOS（Type of Service）字段的匹配能力。

🚀 路由缓存（dst_entry cache）

以 {src, dst, TOS, iif} 为键缓存路由决策结果（struct rtable）。后续匹配相同流的包直接命中缓存，跳过全表查找，大幅提升吞吐量。

🎯 TOS 字段匹配

RFC 1349 定义的 IP 头 TOS 字节可表达优先级/延迟/吞吐等偏好。Linux 允许路由规则按 TOS 值差异化转发，为 QoS 打下基础。

📌 场景：多媒体流量 QoS
同一目的 IP（如视频服务器）的普通 HTTP 请求与实时视频流，TOS 不同，可以通过不同路径（带宽大 vs 延迟低）转发，实现初步的服务质量差异化。
图 2-1：引入路由缓存后的查找流程，TOS 作为额外匹配维度

同期还引入了 Realm（域） 的概念（ip route add ... realms src/dst）。Realm 是路由条目上的整数标签，用于流量聚合和统计（如 TC 子系统对不同 ISP 流量的计费统计），这是路由「输出值」字段的第一次真正扩展。

维度新增字段解决的问题
Key 输入 tos (服务类型) 同一目的地按流量类型差异化路由，支持初步 QoS
Key 输入 src (源地址) 路由缓存以四元组为键，区分同一目的的不同源
Value 输出 realm (域标签) 流量聚合统计，TC 子系统可按 realm 计费
Value 输出 prefsrc (首选源 IP) 指定该路由下外发包的源 IP，解决多 IP 接口的源选择问题
第一性原理

性能与功能的矛盾是这个阶段的核心张力。路由缓存是一个经典的「以空间换时间」方案，但它必须用足够细粒度的键来保证缓存正确性——这迫使 Key 从纯 dst 扩展到 {src, dst, TOS}。TOS 的引入则揭示了一个规律：只要有两类流量需要「同目的，不同路径」，Key 就必须增加维度。

③
Linux 2.1 – 2.2 · 约 1997–1999
策略路由与路由策略数据库（RPDB）
多路由表 + 规则引擎：将「目的地」决策权交给管理员

Linux 2.1 系列由 Alexey Kuznetsov（俄罗斯核物理研究所）主导实现了策略路由（Policy-Based Routing, PBR），引入了革命性的路由策略数据库（RPDB, Routing Policy Database）。这是路由子系统最重要的架构变革之一。

📌 场景：多线接入 / ISP 选择
企业接入两条 ISP 线路（ISP-A 和 ISP-B）。规定：来自 192.168.1.0/24 的流量走 ISP-A；来自 192.168.2.0/24 的流量走 ISP-B；管理流量（标记了 fwmark）始终走 ISP-A。单张路由表无法表达这种「谁发的决定走哪里」的语义。

RPDB 的架构思路是：将路由决策分成两层——先用「规则（Rules）」选择路由表，再在选定的路由表内做最长前缀匹配。

图 3-1：RPDB 两级路由决策架构 — 规则选表 → 表内最长匹配

RPDB 在内核中由 CONFIG_IP_MULTIPLE_TABLES 选项控制。启用后，内核默认创建 3 张路由表（local=255、main=254、default=253），由 3 条默认规则依次查找，兼容原有行为。

# ip rule show（默认规则集）
0:     from all lookup local    # 优先处理本地/广播地址
32766: from all lookup main     # 普通路由表
32767: from all lookup default  # 空表，历史保留

规则的选择器（SELECTOR）可匹配以下字段——这是路由 Key 的重大扩展：

维度新增字段解决的问题
Key 输入 from (源地址) 按数据包来源 IP 段选择不同路由表，实现多宿主源路由
Key 输入 to (目的地址) 规则层可再次按目的地址分流，与表内匹配叠加
Key 输入 iif (入接口) 按流量从哪个接口进入选择路由表，实现接口级隔离
Key 输入 tos (服务类型) 规则层按 TOS 选表，与表内 TOS 匹配叠加
Value 输出 规则动作类型 unicast / blackhole / unreachable / prohibit / nat，路由决策不再只是「转发」
Value 输出 scope (作用域) host / link / global，精确描述下一跳可达性，影响 ARP/邻居子系统行为
📖 权威来源
Linux 2.2.12 支持 255 张路由表、255 个聚合 Realm、2³²个规则优先级。— Matthew G. Marsh，《Policy Routing in Linux》，Dr.Dobb's Journal，约 1999 年（原文链接）
第一性原理

RPDB 揭示了一个深刻的设计原理：把「选择哪张表」和「表内查什么」分离，是 Policy/Mechanism 分离思想的体现。规则是策略层，路由表是机制层。这种分层使得系统可以在不改变查找算法的情况下，无限扩展选路策略的复杂度。RPDB 在架构上等价于在「路由函数」外再套一层「路由函数选择器」，实现了元路由（Meta-routing）能力。

④
Linux 2.1.68 · 1997 → 2.6 → 4.4
多路径路由（ECMP）
Value 扩展：一个目的，多个等价下一跳

1997 年 11 月，Linux v2.1.68 首次引入多路径路由支持（IPv4），允许一条路由条目携带多个等价下一跳（ECMP, Equal-Cost Multipath），实现 L3 层的流量负载均衡。

📌 场景：数据中心多上行链路负载均衡
服务器接入两条 10G 上行链路，目的网段相同。希望流量按哈希分摊到两条链路，提升出口带宽利用率，同时实现链路故障时的自动切换。
图 4-1：ECMP 多路径路由结构与哈希选路演化

内核数据结构 struct fib_info 持有一个 fib_nh[] 数组，每个元素代表一个下一跳。多路径路由的哈希策略经历了多次演化：

内核版本ECMP 哈希策略说明
v2.1.68随机（per-packet）用 jiffies 作随机数，可能乱序
v2.6.12–2.6.22基于目的地址路由缓存保证同目的走同路径
v3.6路由缓存移除，退化为近随机缓存移除导致乱序风险重现
NEW v4.4L3 哈希（src+dst+proto）commit 0e884c78，流级别一致性
NEW v4.12L4 哈希（含 src/dst port）fib_multipath_hash_policy=1
NEW v5.2+自定义哈希字段fib_multipath_hash_fields sysctl
维度新增字段解决的问题
Value 输出 fib_nh[] 数组（多下一跳） 单条路由条目表达多条等价路径，实现 L3 负载均衡
Value 输出 nh_weight（权重） 加权 ECMP，按比例分配流量到不同带宽的链路
Value 输出 fib_metrics（链路参数） MTU / RTT / rttvar / cwnd 等 TCP 参数，路由级别的链路质量描述
第一性原理

ECMP 的本质是将路由的「1 对 1 映射」泛化为「1 对 N 映射」。但 N 条路径的一致性哈希问题随即出现：同一连接的多个包必须走同一条路径，否则 TCP 收包乱序。这迫使哈希的粒度从「per-packet」演进到「per-flow」（L3 hash），再到「per-micro-flow」（L4 hash）。负载均衡粒度与有序性保证永远是一对矛盾，ECMP 的演化史就是在这对矛盾中寻找平衡的历史。

⑤
Linux 2.4 · 约 2001
Netfilter 集成与 fwmark 路由
防火墙标记成为路由键：将安全策略与转发决策融合

Linux 2.4 带来了 Netfilter 框架（取代了 2.2 的 ipchains），并通过 iptables -j MARK（及后来的 nftables meta mark set）为数据包打上任意 32 位整数标记 fwmark。

关键创新在于：RPDB 规则可以用 fwmark 作为匹配键，将防火墙的分类决策转化为路由决策。这彻底打通了 L3/L4 层信息与路由子系统之间的通道。

📌 场景：基于应用类型的策略路由
企业需要将所有 HTTP(s) 流量（TCP:80/443）走代理出口，将 SSH 和管理流量走专用管理线路，将 VoIP 流量（UDP:5060）走低延迟线路。这些需求无法仅用 IP 地址区分，必须结合 L4 端口信息。
图 5-1：fwmark 打通 Netfilter 与路由子系统，实现应用感知路由
维度新增字段解决的问题
Key 输入 fwmark（防火墙标记） 将 L4~L7 信息（端口、协议、连接状态）映射为路由键，实现应用感知路由
Value 输出 路由类型扩展 blackhole/unreachable/prohibit/throw，路由即安全策略执行点

fwmark 是路由子系统与安全子系统之间的解耦接口。它使两个子系统可以独立演化，同时通过一个 32 位整数共享决策信息，是 Linux 网络架构模块化思想的典范。

第一性原理

fwmark 体现了「关注点分离 + 抽象接口」的设计哲学。安全子系统（Netfilter）负责「分类」，路由子系统负责「转发」，两者通过一个整数标记解耦。这个设计允许任何能写 skb→mark 的子系统（Netfilter、eBPF、tc/BPF 等）影响路由决策，而无需修改路由子系统本身。路由 Key 维度的增加，本质上是「外部信息注入路由决策平面」能力的增加。

⑥
Linux 2.6.39 → 3.6 · 2011–2012
LPC-Trie 算法与路由缓存的移除
查找算法革新：用更快的树换掉有安全隐患的缓存

路由缓存从 Linux 2.0 时代服务了十多年，但它有一个致命弱点：DoS 攻击。攻击者只需向随机目标 IP 发包，就能用海量伪流填满路由缓存，耗尽内存。同时，路由缓存以 {src, dst, TOS} 为键，粒度极细，缓存条目可能达到数百万个。

📌 场景：路由缓存 DoS 与性能问题
2010年前后，有研究者发现通过发送大量随机目的 IP 包，可以在数秒内让 Linux 路由器内存耗尽。随着 Internet 路由表条目突破 40 万条，传统哈希表查找的扩展性也开始成为瓶颈。

解决方案分为两步：

🌳 v2.6.39：LPC-Trie 成为默认

引入 LC-Trie（Level Compressed Trie，也称 FIB Trie），取代旧有哈希表。LC-Trie 通过路径压缩和层压缩优化前缀树，在大型路由表（数十万条）中查找复杂度接近 O(1)，远优于哈希表的最坏情况。代码位于 net/ipv4/fib_trie.c。

🗑️ v3.6：移除路由缓存

commit 89aef8921bfb：正式删除全局路由缓存（dst_cache）。有了更快的 LPC-Trie，直接查 FIB 已足够高效。缓存改为每条路由条目级别的 nexthop 缓存（fib_nh.nh_rth_output/input），不再以流为粒度，彻底消除了 DoS 攻击面。

图 6-1：路由缓存移除与 LPC-Trie 替换后的查找架构

这次变化对「Key/Value 维度」没有直接增加新字段，而是重塑了路由查找的性能基础设施，为后续更复杂的功能（VRF、nexthop 对象）奠定了基础。移除缓存后，路由子系统的行为更加确定性——每次查找结果完全由当前路由表决定，不再受历史缓存状态影响。

📖 参考：Routing Decisions in the Linux Kernel（thermalcircle.de） | LPC-Trie: kernel 2.6.39 commit 3630b7c050d9 | 路由缓存移除: kernel 3.6
第一性原理

路由缓存的移除是一次「用算法换缓存」的经典工程权衡。当查找算法（LPC-Trie）快到一定程度时，缓存的边际收益（节省查找时间）小于其代价（内存占用、DoS 风险、一致性复杂度）。这揭示了一个普适原则：缓存是对慢速操作的补偿，当底层操作变快到足够程度，缓存的存在反而成为负担。安全性（DoS 免疫）往往是打破这个权衡的最终驱动力。

⑦
Linux 4.3 → 4.8 · 2015–2016
VRF 与 L3MDev：路由隔离的原生支持
Key 扩展：将「属于哪个 VRF」纳入路由决策

云计算和 SDN 的兴起带来了多租户路由隔离的强烈需求。同一台物理路由器（或 Linux 服务器）上，需要为不同客户维护完全独立的路由表，即使它们使用相同的 IP 地址段也互不干扰——这就是 VRF（Virtual Routing and Forwarding）的核心需求。

📌 场景：多租户云路由器 / 企业多 VPN 并存
同一台 Linux 服务器上运行 VPN 客户 A 和客户 B，两者均使用 10.0.0.0/8 网段，但路由完全隔离。客户 A 的 10.1.1.1 和客户 B 的 10.1.1.1 路由到不同的物理接口，彼此不可见。

Linux 4.3 引入 l3mdev（Layer 3 Master Device）框架，4.8 引入统一的 l3mdev-table FIB 规则，使 VRF 实现更简洁：

图 7-1：VRF 设备与 l3mdev-table 规则实现路由域隔离
# 创建 VRF 设备并绑定路由表
ip link add vrf-blue type vrf table 10
ip link set dev vrf-blue up
ip link set dev eth1 master vrf-blue   # eth1 归属 vrf-blue

# v4.8+ 自动插入 l3mdev 规则（单条规则覆盖所有 VRF）
# ip rule show:
#  1000: from all lookup [l3mdev-table]

# VRF 内添加路由
ip route add table 10 10.0.0.0/8 dev eth1
ip route add table 10 unreachable default metric 4278198272
维度新增字段解决的问题
Key 输入 l3mdev / VRF 设备 数据包进入哪个 VRF 域决定查哪张路由表，实现租户级路由隔离
Key 输入 网络命名空间（net ns） 每个 netns 拥有独立路由表集合，提供更强的隔离（进程级）
Value 输出 l3mdev-table 查找 单条 RPDB 规则取代 N 个 VRF 各自的 iif/oif 规则，极大简化配置
📖 参考：kernel.org: VRF Documentation | v4.3 l3mdev 引入 | v4.8 l3mdev-table 规则统一（"Prior to v4.8 iif and oif rules were needed for each VRF device"）
第一性原理

VRF 的本质是将「路由域归属」提升为一个一等公民的 Key 维度。在此之前，隔离只能通过 网络命名空间（整个网络栈复制，重量级）或 policy rules（配置复杂、难以扩展）模拟。l3mdev 的设计选择了「最小侵入性」原则：只在 L3 层做隔离，L2 工具（LLDP 等）、TC 子系统对 VRF 透明。一个设计是否优雅，往往体现在它对相邻子系统的透明度——VRF 只影响路由，不影响其他层，是模块化的典范。

⑧
Linux 5.1 – 5.3 · 2019
Nexthop 对象独立化（RTA_NHID）
Value 重构：下一跳从路由条目内联到独立对象，实现规模化管理

在 v5.3 之前，每条路由条目（fib_info）内嵌一个 fib_nh[] 数组表示下一跳。这在路由表规模达到数十万条、下一跳可能被大量路由共享时，造成严重的内存浪费和更新开销：修改一个下一跳（如链路变更）需要重新插入所有引用它的路由条目。

David Ahern 在 2019 年（最终合入 v5.3）引入了独立的 Nexthop 对象，通过属性 RTA_NHID 让路由条目引用外部 nexthop 对象，而非内嵌数据：

图 8-1：Nexthop 对象独立化前后对比 — 引用替代内嵌
# 创建独立 nexthop 对象
ip nexthop add id 10 via 10.0.0.1 dev eth0
ip nexthop add id 11 via 10.0.0.2 dev eth1
ip nexthop add id 100 group 10/11   # ECMP 组

# 路由条目引用 nexthop 对象（而非内嵌）
ip route add 10.0.0.0/8    nhid 100
ip route add 172.16.0.0/12 nhid 100

# 原子替换 nexthop（所有引用路由同时更新）
ip nexthop replace id 100 group 10/12
维度新增字段解决的问题
Value 输出 nhid（Nexthop 对象 ID） 路由条目通过 ID 引用外部 nexthop 对象，消除数据冗余
Value 输出 nexthop 组（nh_grp） 一个 nexthop 对象可包含多个 leaf nexthop 并分配权重，ECMP 管理更灵活
Value 输出 IPv6 nexthop for IPv4 route 支持 RFC 5549（IPv6 下一跳 for IPv4 路由），统一 IPv4/IPv6 nexthop 结构

第一性原理

Nexthop 对象独立化是一次数据规范化（Database Normalization）：将重复存储的 nexthop 信息提取为独立实体，用 ID 引用。这是数据库设计中消除冗余的基本原则在内核数据结构中的应用。规范化的代价是多一次间接引用（pointer dereference），但带来的好处是：原子性更新（改一个对象，所有引用生效）、内存节省（消除重复）、路由注入速度提升。这个演化表明，随着系统规模增长，数据组织方式本身也需要演化——性能瓶颈往往不在算法，而在数据结构的冗余。

从全景视角看，路由子系统的演化遵循几个清晰的规律：

  • Key 维度扩展驱动因素：新的流量分类需求（TOS→安全→VRF），每次都是「某类流量需要差异化处理但原有 Key 无法区分」。
  • Value 维度扩展驱动因素：转发语义丰富化（单路径→多路径→规范化引用），以及链路参数精细化（MTU、RTT 等）。
  • 架构层次演化：从单表→多表+规则→VRF+命名空间，每一层都是「隔离粒度」的增加。
  • 性能基础设施演化相对独立：LPC-Trie 和 nexthop 对象是「怎么存」的优化，不影响「存什么」（Key/Value 语义）。
第一性原理：路由子系统演化的本质规律

路由子系统的复杂性遵循一个简洁的元模式：每当现实世界出现「同一组现有 Key 值，需要两种不同 Value」的场景时，就必须向 Key 增加新的维度；每当「多个路由条目需要表达同一个语义差异」的场景出现时，就需要向 Value 增加新的结构。

这不是软件腐化，而是对信息论的遵从：区分 N 种情况，最少需要 log₂N 位信息。现实世界的流量分类需求（QoS、安全、租户、ISP 选择……）不断增加分类维度，路由 Key 的位数必然随之增长。复杂性是需求的忠实镜像。
