00

引言与设计哲学

Linux 内核的网络子系统遵循一条清晰的分层设计原则：物理链路（Layer 2）与网络地址（Layer 3）被显式分离。一个网络接口（network interface）代表的是硬件或虚拟的数据收发通道，而 IP 地址则是附加在这个通道上的"门牌号"——两者是一对多的关系。

📘 核心设计思想
来自 ip-address(8) man page（iproute2 官方文档）：

"The address is a protocol (IPv4 or IPv6) address attached to a network device. Each device must have at least one address to use the corresponding protocol. It is possible to have several different addresses attached to one device."

在 Linux 2.0 时代，一块网卡只能绑定一个 IP 地址，若需多个 IP 则要使用"别名接口"（如 eth0:1）。自 Linux 2.2 起，内核彻底重构了地址管理模型，引入了 in_ifaddr 链表，使得单个接口可以原生挂载多个 IP 地址，无需别名设备。这一变化通过 rtnetlink(7) 消息类型 RTM_NEWADDR 对外暴露。

📜 第一性原理概述
网络通信的本质是：标识节点 + 路由数据。IP 地址解决"你是谁、在哪里"的问题，而网络接口解决"数据从哪个物理出口进出"的问题。内核将两者解耦，是因为同一个出口可以服务多个"门牌号"（多 IP），而"门牌号"的有效范围（scope）和生存时间（lifetime）也因场景而异。
01

网络接口的内核表示：net_device

每个网络接口在内核中由 struct net_device（定义于 <linux/netdevice.h>）唯一表示。它既是驱动层（L2）的抽象，也是 IP 层（L3）配置的挂载点。

关键字段（精简版）

/* include/linux/netdevice.h */
struct net_device {
    char              name[IFNAMSIZ];   /* 接口名, e.g. "eth0", "lo" */
    int               ifindex;          /* 全局唯一索引号 */
    unsigned int      flags;            /* IFF_UP, IFF_LOOPBACK 等 */
    unsigned int      mtu;              /* 最大传输单元 */
    unsigned char     dev_addr[MAX_ADDR_LEN]; /* MAC 地址 */

    /* ▼ 关键：指向 IPv4 配置层 */
    struct in_device  __rcu *ip_ptr;   /* IPv4 协议配置 */

    /* ▼ IPv6 配置层 */
    struct inet6_dev  *ip6_ptr;        /* IPv6 协议配置 */

    /* 统计、队列、驱动操作等... */
};
⚠️ 关键设计点
net_device 本身不直接存储 IP 地址。IP 地址通过 ip_ptr 指针间接关联到 struct in_device，再经由 ifa_list 链表访问每一个具体的 IP 地址（struct in_ifaddr）。这种三层解耦设计保证了协议无关性。

接口标志（flags）

来源：netdevice(7) man page

标志含义
IFF_UP接口已激活（运行中）
IFF_BROADCAST支持广播，广播地址有效
IFF_LOOPBACK是回环接口（lo）
IFF_POINTOPOINT点对点链路（如 PPP）
IFF_RUNNING资源已分配，链路就绪
IFF_PROMISC混杂模式，接收所有帧
IFF_MULTICAST支持组播
IFF_NOARP无 ARP 协议（如隧道接口）
IFF_DYNAMIC接口 down 时地址丢失
IFF_LOWER_UP驱动信号 L1 链路 up（Linux 2.6.17+）
🔬 第一性原理
net_device 是"物理通道"的抽象——它描述的是数据如何进出（MTU、队列、驱动操作）。IP 地址描述的是"谁在这个通道上"。内核通过 ip_ptr 这个指针将两者连接，体现了"机制与策略分离"的 UNIX 哲学：驱动层不需要理解 IP，IP 层也不需要关心物理介质。
02

接口与 IP 地址的关系：一对多模型

一个网络接口可以同时关联多个 IP 地址（IPv4 和 IPv6 分别管理）。这一设计自 Linux 2.2 起取代了旧的"别名接口"机制（eth0:0、eth0:1…）。

历史演进

时代机制限制
Linux 2.0别名接口 eth0:N每个别名视为独立设备，管理复杂
Linux 2.2+in_ifaddr 链表无需别名，多地址原生支持
现代内核rtnetlink RTM_NEWADDR通过 Netlink 动态增删，支持 IPv4/IPv6
📘 来源：rtnetlink(7) man page
"In Linux 2.2, an interface can carry multiple IP addresses, this replaces the alias device concept in Linux 2.0."

主地址与辅助地址（Primary / Secondary）

当同一网段的多个地址被分配到一个接口时，内核将第一个地址标记为主地址（primary），后续同网段地址标记为辅助地址（secondary，IFA_F_SECONDARY）。主地址被删除时，辅助地址会被提升或一同删除。

不同网段的地址彼此独立，均视为主地址。

🔬 第一性原理
"一接口多地址"的本质需求是：在同一条物理链路上提供多个逻辑端点。这在虚拟主机（Web hosting）、负载均衡（anycast）、地址迁移（不停服切换 IP）等场景中至关重要。内核用链表而非固定数组存储地址，是因为地址数量在运行时动态变化，链表是最自然的动态集合。
03

IP 地址在内核中的表示：struct in_ifaddr

IPv4 地址在内核中由 struct in_ifaddr 表示，定义于 <linux/inetdevice.h>。每个 in_ifaddr 实例描述一个 IPv4 地址条目，通过 ifa_next 单链表组织。

/* include/linux/inetdevice.h */
struct in_ifaddr {
    struct hlist_node   hash;          /* 哈希表节点（地址快速查找） */
    struct in_ifaddr    *ifa_next;      /* 同接口下一个地址（链表） */
    struct in_device    *ifa_dev;       /* 所属 in_device 反向引用 */
    struct rcu_head     rcu_head;      /* RCU 延迟释放 */

    /* ── 地址值 ── */
    __be32              ifa_local;     /* 本地地址（单播，即"本机 IP"） */
    __be32              ifa_address;   /* 对端地址（PPP）；广播接口同 local */
    __be32              ifa_mask;      /* 子网掩码 */
    __be32              ifa_broadcast; /* 广播地址 */

    /* ── 路由优先级 ── */
    __u32               ifa_rt_priority; /* 关联前缀路由优先级 */

    /* ── 范围与长度 ── */
    unsigned char       ifa_scope;     /* 地址有效范围（RT_SCOPE_*） */
    unsigned char       ifa_prefixlen; /* 前缀长度（CIDR），如 24 */

    /* ── 标志位 ── */
    __u32               ifa_flags;     /* IFA_F_SECONDARY 等 */

    /* ── 标签 ── */
    char                ifa_label[IFNAMSIZ]; /* 地址标签（兼容旧别名） */

    /* ── 生命周期（秒，相对 tstamp） ── */
    __u32               ifa_valid_lft;     /* 有效生命周期 */
    __u32               ifa_preferred_lft; /* 首选生命周期 */
    unsigned long       ifa_cstamp;    /* 创建时间戳（jiffies） */
    unsigned long       ifa_tstamp;    /* 更新时间戳（jiffies） */
};

来源：Stack Overflow / Linux kernel source（include/linux/inetdevice.h），及 kernel.org devinet.c 源码。

📘 ifa_local 与 ifa_address 的区别
  • 普通以太网/广播接口：ifa_local == ifa_address，均为本机 IP。
  • 点对点接口（PPP）：ifa_local 是本端地址，ifa_address 是对端（peer）地址。这与 ip-address(8) man page 中 local / peer 参数对应。
🔬 第一性原理
一个 IP 地址不只是"一串数字"，它还携带了它在哪个范围内有意义（scope）、是否还有效（lifetime）、与谁共享子网（mask）、如何广播（broadcast）等完整的网络语义。struct in_ifaddr 是这些信息的最小完备集合——去掉任何一个字段，路由或地址选择就会失去依据。
04

IPv4 地址属性详解

ifa_local — 本地地址

__be32 类型，网络字节序（Big Endian）的 32 位整数。这是接口实际使用的单播地址，即通常所说的"IP 地址"。内核在发送数据包选择源地址、响应 ARP 请求时，以 ifa_local 为准。

ifa_mask — 子网掩码

与 ifa_local 按位与（AND）运算，得到所属网络地址。内核通过它判断目的地是否在同一子网（是否需要 ARP，还是走网关）。ifa_prefixlen 是其 CIDR 等价表达（如 24 对应 0xFFFFFF00）。

ifa_broadcast — 广播地址

该子网的定向广播地址（如 192.168.1.255）。内核向该地址发送的数据包会在链路层被扩散给所有主机。ip addr add 支持用 +/- 符号自动推导（设置/清除主机位）。

ifa_prefixlen — 前缀长度

CIDR 表示法中斜杠后的数字（0-32）。与 ifa_mask 冗余但互为补充：内核内部用 mask 计算，外部接口用 prefixlen 展示。

🔬 第一性原理
子网掩码的本质是一个"范围声明"：它告诉内核"哪些目的地址不需要经过网关就能直接到达"。前缀长度越大，子网越小，本地范围越窄；越小则覆盖范围越大。内核路由决策的核心就是：目的 IP AND mask == ifa_local AND mask？是则直连，否则走网关。
05

地址范围（Scope）：地址在哪里有效

ifa_scope 描述了地址的可达域——这个地址对哪些网络节点有意义。来源：ip-address(8) man page 及 rtnetlink(7)。

常量值含义典型示例
RT_SCOPE_UNIVERSE0 全局有效，可跨互联网路由 203.0.113.1（公网 IP）
RT_SCOPE_SITE200 站点内有效（IPv6 已废弃） IPv6 fec0::/10（已弃用）
RT_SCOPE_LINK253 仅链路本地有效，不可路由 169.254.x.x、IPv6 fe80::
RT_SCOPE_HOST254 仅在本机内部有效 127.0.0.1（loopback）
RT_SCOPE_NOWHERE255 目的地不存在 内核内部使用
📘 来源：ip-address(8) man page（iproute2 官方）
"scope - the scope of the area where this address is valid:
global - the address is globally valid.
link - the address is link local, i.e. it is valid only on this device.
host - the address is valid only inside this host."

Scope 对路由的影响

内核在生成前缀路由时会参考地址的 scope。当你给接口添加一个 global scope 的地址时，内核自动在路由表中创建该网段的 link scope 路由。link scope 的地址（如链路本地地址）对应的路由不会被通告给外部路由协议。

🔬 第一性原理
Scope 的本质是"可见性约束"。网络协议必须区分"只有我自己知道的地址"、"本链路上的设备知道的地址"和"全互联网都能寻址到的地址"。不加区分，路由器可能会把 127.0.0.1 通告到互联网，或者把链路本地地址放入路由表导致路由混乱。Scope 是地址语义的一部分，不是可选附属信息。
06

地址标志（Flags）：地址的状态与属性

ifa_flags 是一个位掩码，描述地址的状态、来源和特殊属性。来源：rtnetlink(7)、ip-address(8) man pages。

标志适用含义
IFA_F_SECONDARY IPv4 辅助地址（与某主地址同网段）。旧别名接口的替代表示。
IFA_F_PERMANENT IPv4IPv6 由用户手动配置的永久地址（非 DHCP 或自动配置）。
IFA_F_TENTATIVE IPv6 地址处于 DAD（重复地址检测）阶段，尚未确认唯一性，不可用于通信。
IFA_F_DEPRECATED IPv6 首选生命周期已过期，地址仍有效但不再优先用于新连接（RFC 4862）。
IFA_F_DADFAILED IPv6 DAD 失败，检测到地址冲突。地址被禁用。
IFA_F_TEMPORARY IPv6 隐私扩展（RFC 3041）自动生成的临时地址，定期更换以保护隐私。
IFA_F_NODAD IPv6 添加地址时跳过 DAD 检测（如手动配置时已知唯一）。
IFA_F_HOMEADDRESS IPv6 移动 IPv6（RFC 6275）的家庭地址。
IFA_F_MANAGETEMPADDR IPv6 内核自动管理基于此地址派生的临时地址（隐私扩展）。
IFA_F_NOPREFIXROUTE IPv4IPv6 添加地址时不自动创建对应前缀路由。

DAD（重复地址检测）状态机

🔬 第一性原理
标志位解决了"地址在任意时刻处于什么状态"的问题。网络是动态的——地址可能正在被验证（TENTATIVE）、正在老化（DEPRECATED）、已冲突（DADFAILED）、是临时隐私地址（TEMPORARY）。如果没有这些状态，内核就无法实现"地址平滑切换"（先配新地址再删旧地址）和"隐私保护"（临时地址自动轮换）等高级功能。
07

地址生命周期（Lifetime）

生命周期机制主要用于 IPv6 无状态地址自动配置（SLAAC，RFC 4862），IPv4 也支持（DHCP 租期管理）。来源：ip-address(8) man page。

两种生命周期

字段含义过期后行为
ifa_preferred_lft 首选生命周期（preferred lifetime） 地址进入 DEPRECATED 状态，不再用于新建连接的源地址选择，已有连接继续使用
ifa_valid_lft 有效生命周期（valid lifetime） 地址从接口彻底删除，内核发起 RTM_DELADDR 通知
📘 来源：ip-address(8) man page
"valid_lft LFT — the valid lifetime of this address; see section 5.5.4 of RFC 4862. When it expires, the address is removed by the kernel."

"preferred_lft LFT — the preferred lifetime of this address. When it expires, the address is no longer used for new outgoing connections."

rtnetlink 的 ifa_cacheinfo

通过 Netlink 查询地址时，内核返回 struct ifa_cacheinfo（来源：rtnetlink(7)）：

struct ifa_cacheinfo {
    __u32 ifa_prefered; /* 首选生命周期剩余秒数 */
    __u32 ifa_valid;    /* 有效生命周期剩余秒数 */
    __u32 cstamp;       /* 创建时间戳（百毫秒） */
    __u32 tstamp;       /* 更新时间戳（百毫秒） */
};
/* ifa_valid 不能为 0；ifa_prefered ≤ ifa_valid */
/* UINT32_MAX = 无限生命周期（永久地址） */
🔬 第一性原理
生命周期机制的本质是"地址的租约"思想。IP 地址不是永恒的，它可能由 DHCP 服务器分配（有租期）、由 SLAAC 自动生成（有 RA 中指定的寿命）。两阶段生命周期（preferred → valid）实现了"软老化"：旧地址先停止接受新连接，再彻底消失，避免连接中断的硬切换。这和操作系统内存页面的"aging"机制如出一辙。
08

IPv6：inet6_dev 与 inet6_ifaddr

IPv6 地址管理使用独立的结构体，以支持 IPv6 特有的特性（SLAAC、DAD、隐私扩展、链路本地地址等）。来源：include/net/if_inet6.h，net/ipv6/addrconf.c。

struct inet6_ifaddr（精简）

/* net/ipv6/addrconf.c / include/net/if_inet6.h */
struct inet6_ifaddr {
    struct in6_addr     addr;          /* 128 位 IPv6 地址 */
    __u32               prefix_len;    /* 前缀长度（0-128） */
    __u32               rt_priority;   /* 前缀路由优先级 */
    __u32               flags;         /* IFA_F_TENTATIVE 等 */
    __u8                dad_probes;    /* DAD 探针发送次数 */
    __u8                stable_privacy_prefix; /* 稳定隐私地址前缀 */

    /* 生命周期（与 in_ifaddr 类似） */
    __u32               valid_lft;
    __u32               prefered_lft;  /* RFC 原文拼写 */
    unsigned long       cstamp;
    unsigned long       tstamp;

    struct inet6_dev    *idev;         /* 反向引用所属 inet6_dev */
    struct list_head    if_list;       /* 链表节点（作用域有序） */
    struct list_head    tmp_list;      /* 隐私扩展临时地址链表 */
};

IPv6 与 IPv4 地址管理对比

特性IPv4（in_ifaddr）IPv6（inet6_ifaddr）
地址宽度32 位（__be32）128 位（in6_addr）
链表组织单链表（ifa_next）双向链表（list_head，按 scope 排序）
地址来源手动/DHCP手动/DHCPv6/SLAAC/链路本地自动生成
重复检测ARP（可选）DAD（RFC 4862，必须）
生命周期支持（扩展）核心特性（RA 下发）
隐私地址不支持IFA_F_TEMPORARY（RFC 3041）
链路本地169.254.x.x（可选）fe80::/10（自动生成，必须）
📘 来源：Stack Overflow / net/ipv6/addrconf.c
"Each device address list is sorted in order of scope - global before linklocal." （inet6_ifaddr 链表按 scope 降序排列：全局地址在前，链路本地地址在后）
🔬 第一性原理
IPv6 地址管理的复杂性来源于其设计目标：零配置、可扩展、隐私保护。链路本地地址必须自动生成（设备开机即可通信），SLAAC 让设备无需 DHCP 服务器即可获得全球地址，DAD 保证地址唯一性，临时地址保护用户不被长期追踪。每一个机制都对应一个真实的网络工程问题——内核数据结构的复杂度正是这些需求的直接映射。
09

配置接口：ioctl 与 rtnetlink

Linux 提供了两套用户空间配置 IP 地址的接口，均有官方 man page 记录。

传统接口：ioctl

通过 Socket 的 ioctl(2) 系统调用，传入 struct ifreq 结构体操作接口地址。来源：netdevice(7) man page。

/* include/net/if.h — struct ifreq */
struct ifreq {
    char ifr_name[IFNAMSIZ];  /* 接口名，如 "eth0" */
    union {
        struct sockaddr ifr_addr;      /* SIOCGIFADDR: 接口地址 */
        struct sockaddr ifr_netmask;   /* SIOCGIFNETMASK: 子网掩码 */
        struct sockaddr ifr_broadaddr; /* SIOCGIFBRDADDR: 广播地址 */
        struct sockaddr ifr_hwaddr;    /* SIOCGIFHWADDR: MAC 地址 */
        short           ifr_flags;     /* SIOCGIFFLAGS: 接口标志 */
        int             ifr_ifindex;   /* 接口索引 */
        int             ifr_mtu;       /* MTU */
        /* ... */
    };
};

/* IPv6 使用独立结构 */
struct in6_ifreq {
    struct in6_addr ifr6_addr;
    __u32           ifr6_prefixlen;
    int             ifr6_ifindex;
};
ioctl 命令操作备注
SIOCGIFADDR获取接口 IP 地址仅返回 AF_INET（IPv4）
SIOCSIFADDR设置接口 IP 地址接受 AF_INET / AF_INET6
SIOCDIFADDR删除接口 IP 地址仅删除 AF_INET6
SIOCGIFNETMASK获取子网掩码仅 AF_INET
SIOCSIFNETMASK设置子网掩码特权操作
SIOCGIFBRDADDR获取广播地址仅 AF_INET
SIOCGIFFLAGS获取接口标志IFF_UP 等
SIOCGIFCONF枚举所有接口地址返回 AF_INET 地址列表

现代接口：rtnetlink

通过 Netlink socket（AF_NETLINK，协议 NETLINK_ROUTE）进行全双工通信，支持事件通知。iproute2 的 ip addr 命令即基于此。来源：rtnetlink(7) man page。

/* 地址相关消息类型（rtnetlink.h） */
RTM_NEWADDR   /* 添加 IP 地址 */
RTM_DELADDR   /* 删除 IP 地址 */
RTM_GETADDR   /* 查询 IP 地址 */

/* 消息携带 ifaddrmsg 结构 */
struct ifaddrmsg {
    unsigned char ifa_family;    /* AF_INET 或 AF_INET6 */
    unsigned char ifa_prefixlen; /* 前缀长度 */
    unsigned char ifa_flags;     /* IFA_F_SECONDARY 等 */
    unsigned char ifa_scope;     /* RT_SCOPE_* */
    unsigned int  ifa_index;     /* 接口索引（ifindex）*/
};

/* 后接 rtattr 属性（TLV 编码） */
IFA_ADDRESS    /* 接口地址（原始协议地址）*/
IFA_LOCAL      /* 本地地址 */
IFA_LABEL      /* 标签字符串 */
IFA_BROADCAST  /* 广播地址 */
IFA_ANYCAST    /* 任播地址 */
IFA_CACHEINFO  /* 生命周期信息（ifa_cacheinfo）*/
🔬 第一性原理
ioctl 是"我问你答"的同步模型，适合简单查询，但无法感知内核主动发出的地址变更事件（如 DHCP 租期到期）。rtnetlink 是"发布-订阅"模型，进程可以订阅地址变更事件（RTNLGRP_IPV4_IFADDR），实现实时响应。这个演进反映了操作系统接口从"轮询"到"事件驱动"的设计进步。

内核数据流全景

下图展示了从用户空间配置 IP 地址到内核数据结构的完整路径，以及内核如何利用这些结构进行路由决策。

代码遍历示例（内核模块）

来源：Stack Overflow 内核模块示例 / Linux netdev 邮件列表

/* 遍历接口的所有 IPv4 地址（内核模块代码片段） */
struct net_device *dev = dev_get_by_name(&init_net, "eth0");
if (!dev) return;

rcu_read_lock();
struct in_ifaddr *ifa;
for (ifa = rcu_dereference(dev->ip_ptr->ifa_list);
     ifa;
     ifa = rcu_dereference(ifa->ifa_next)) {
    printk("address: %pI4, mask: %pI4\n",
           &ifa->ifa_address, &ifa->ifa_mask);
}
rcu_read_unlock();
🔬 第一性原理
整个数据流体现了"分层抽象 + 延迟绑定"：驱动层（net_device）不感知 IP；IP 层（in_device/in_ifaddr）不感知硬件；路由层仅通过前缀和 scope 决策，不关心地址的具体来源。每一层只需了解上下层的接口契约。RCU（Read-Copy-Update）锁机制确保了在高并发场景下，读路径（数据包转发）无锁高效，写路径（地址增删）安全一致。

第一性原理总结

🔬 整体第一性原理
网络通信的根本目标是：让任意两个节点能够可靠地交换数据。为此，需要解决三个基本问题：

① 标识（谁是谁） → IP 地址（in_ifaddr）
② 定位（在哪里） → Scope + 路由表
③ 传输（怎么发） → net_device（驱动 / 物理接口）

Linux 内核将这三个问题拆解为三层数据结构，通过指针链接。每层独立演进（驱动层换 NIC 不影响 IP 层，IP 层改地址不影响驱动），符合"开闭原则"。

核心设计决策回顾

设计决策背后的第一性原理
一接口多 IP（链表） 物理出口数量 << 逻辑地址数量；虚拟化、多宿主场景要求解耦
Scope 分层 地址的"有效范围"是其语义的一部分，不是可选属性；路由决策依赖 scope
地址标志位 网络是动态系统，地址有状态机（TENTATIVE→PREFERRED→DEPRECATED）
双生命周期 "软老化"避免硬断连接；preferred_lft 控制新连接，valid_lft 控制删除
RCU 保护地址链表 读（转发）远多于写（配置）；RCU 优化读路径，消除锁开销
rtnetlink 事件通知 网络状态变化应主动推送，而非轮询；NetworkManager 等需要实时感知
ifa_local / ifa_address 分离 PPP 等点对点链路的本端/对端地址语义不同，统一表达必须支持
