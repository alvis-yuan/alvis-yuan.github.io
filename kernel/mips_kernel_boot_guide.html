01引言与架构背景

在嵌入式Linux开发中，内核编译完成后并非立即可用——特别是在MIPS这类非x86架构上， 编译产物需要经过一系列平台相关的打包步骤，才能被引导加载程序（Bootloader） 识别和引导。本文以君正（Ingenic）X2600系列芯片为蓝本，系统梳理从 vmlinux到uImage的完整链路，以及U-Boot如何完成内核的加载与跳转。

02MIPS内存模型——理解地址的根基

MIPS架构的地址空间设计与x86截然不同：它没有实模式， 上电即处于内核模式，并且通过固定地址段映射区分不同的内存访问语义。 这直接决定了内核镜像应该加载到哪个地址。

32位MIPS虚拟地址分区

💡 关键规律

MIPS上电后，程序计数器（PC）被设置到 0xBFC0_0000（kseg1段，非缓存）， 这是BootROM / U-Boot SPL的入口。kseg0和kseg1都直接映射到物理地址0起始的512 MB， 差别仅在于Cache属性。Linux内核始终在kseg0段运行（带缓存）， 典型虚拟起始地址为 0x8001_0000，对应物理地址 0x0001_0000。

X2600典型内存布局

# 君正Ingenic XBurst系列典型DDR物理地址布局
0x0000_0000  ←  DDR3起始
0x0001_0000  ←  Linux内核物理加载地址（kseg0虚拟: 0x8001_0000）
   ...          内核代码/数据/BSS
0x0400_0000  ←  initramfs / rootfs（部分方案）
   ...
高地址端     ←  U-Boot重定位位置（RAM顶端以下）
             ←  U-Boot malloc区
             ←  Board Info / Global Data
             ←  Stack（向下增长）

🔑 本节要点 · 第一性原理

MIPS地址空间的本质是：地址本身携带访问语义——哪段地址对应哪种Cache策略 和是否经过TLB，完全由硬件固定规定，无需软件配置页表即可使用kseg0/kseg1。 这是嵌入式MIPS系统能在缺乏MMU初始化的情况下运行早期启动代码的根本原因。 内核镜像打包时必须指定的"加载地址"，本质上就是内核在kseg0段的链接地址。

03Linux内核编译全流程

Linux内核使用Kbuild构建系统，通过递归调用Makefile 完成从源码到二进制的转换。在MIPS交叉编译场景中，需额外指定架构和工具链前缀。

基本编译步骤

# 1. 设置环境变量
export ARCH=mips
export CROSS_COMPILE=mips-linux-gnu-    # 或 mipsel-linux-gnu-（小端）

# 2. 加载开发板默认配置（X2600 Ingenic BSP）
make x2600_defconfig                   # 或 make menuconfig 手动配置

# 3. 编译内核（并行4线程）
make -j4 uImage                        # 直接产出uImage格式

# 等效分步操作:
make -j4 vmlinux                       # 步骤A: ELF格式内核
make -j4 arch/mips/boot/uImage         # 步骤B: 打包成uImage

编译产物层级关系

Kbuild关键Makefile节点

# arch/mips/Makefile — 顶层MIPS架构入口
load-y    = 0x80010000         # 内核链接/加载地址（kseg0）
all       := vmlinux.bin uImage zImage

# arch/mips/boot/Makefile — MIPS启动镜像构建
LOADADDR = 0x80010000         # 与arch/mips/Makefile保持一致

vmlinux.bin: $(VMLINUX)
        $(OBJCOPY) -O binary -S $< $@

uImage: vmlinux.bin
        gzip -9 vmlinux.bin -c > vmlinux.bin.gz
        mkimage -A mips -O linux -T kernel -C gzip \
            -a $(LOADADDR) \
            -e $(shell $(NM) vmlinux | grep kernel_entry | ...) \
            -n 'Linux-$(KERNELRELEASE)' \
            -d vmlinux.bin.gz uImage

内核编译的核心链路是：Kbuild递归Make → vmlinux（ELF）→ objcopy裸二进制 → 压缩 → mkimage封装。 MIPS平台的特殊性在于加载地址固定为kseg0段（0x80010000）， 该值需要在arch/mips/Makefile和mkimage调用时严格保持一致。

04MIPS平台特有的打包步骤

不同于x86的bzImage或ARM的zImage，MIPS平台在 arch/mips/boot/目录下实现了一套独特的镜像生成机制。

步骤一：vmlinux → vmlinux.bin（objcopy）

# 从ELF格式去除调试符号、section头、注释等，输出纯二进制
mips-linux-gnu-objcopy \
    -O binary \
    -R .comment \
    -R .note \
    -S \                    # strip all 符号信息
    vmlinux \
    arch/mips/boot/vmlinux.bin

vmlinux.bin是可以直接映射到MIPS物理地址空间运行的原始二进制， 但其尺寸通常在5–10 MB，需压缩后存入容量有限的Flash。

步骤二：压缩（gzip / lzma）

# 方式A：gzip（通用，解压较快）
gzip -9 arch/mips/boot/vmlinux.bin
# 输出: arch/mips/boot/vmlinux.bin.gz

# 方式B：lzma（压缩率更高，Ingenic SDK常用）
lzma -9 < arch/mips/boot/vmlinux.bin > arch/mips/boot/vmlinux.bin.lzma
ℹ️ 君正Ingenic的压缩格式选择

从实际捕获的U-Boot日志来看，君正旗下设备（T10、T20、T31、X2600等） 普遍使用lzma压缩的uImage（如uImage.lzma）， 因为lzma在嵌入式设备的Flash空间约束下能获得更好的压缩比。 参考：U-Boot日志显示 Image Type: MIPS Linux Kernel Image (lzma compressed)。

步骤三：入口点获取

# 使用nm工具从ELF符号表获取内核真实入口地址
mips-linux-gnu-nm vmlinux | grep " kernel_entry"

# 典型输出示例（因内核版本和配置而异）:
8039a050 T kernel_entry   # Ingenic T10/T31 真实示例
803f58b0 T kernel_entry   # 另一版本示例

入口点（Entry Point）与加载地址（Load Address）可以不同： 加载地址是内核数据放入RAM的起始位置，而入口点是CPU开始执行的指令地址（ arch/mips/kernel/head.S中的kernel_entry标号）。

MIPS kernel_entry与head.S

# arch/mips/kernel/head.S（片段示意）
NESTED(kernel_entry, 16, sp)
    .set push
    .set mips3
    # 初始化CP0寄存器
    mtc0    zero, CP0_COUNT
    # 设置状态寄存器，关闭中断
    li      t0, STATUSF_IP0 | ...
    # 清零BSS段
    la      a0, __bss_start
    la      a1, __bss_stop
    # 跳转到C语言入口
    j       start_kernel       # init/main.c::start_kernel()
END(kernel_entry)

retrofw/kernel README-JZ : "The U-Boot format kernel image is linux-2.6.24.3/arch/mips/boot/uImage. The compressed raw kernel image is linux-2.6.24.3/arch/mips/boot/compressed/zImage."

🔑 本节要点

MIPS平台打包链路的核心三步：objcopy二进制化 → lzma/gzip压缩 → mkimage封装头部。 君正设备倾向于lzma压缩以节省Flash空间。入口点通过nm工具从ELF符号kernel_entry读取， 对应arch/mips/kernel/head.S中的汇编起始标号。

05uImage格式详解

uImage是U-Boot识别Linux内核的标准"旧版格式"（Legacy Image Format）， 由64字节固定头部加上有效载荷（压缩内核）组成。

image_header_t结构体（来自U-Boot include/image.h）

/* Legacy format image header
 * all data in network byte order (big-endian)
 * Source: U-Boot include/image.h
 */
#define IH_MAGIC  0x27051956   /* Image Magic Number */
#define IH_NMLEN  32           /* Image Name Length */

typedef struct image_header {
    uint32_t  ih_magic;          /* [0x00] 魔数: 0x27051956 */
    uint32_t  ih_hcrc;           /* [0x04] 头部CRC32校验和 */
    uint32_t  ih_time;           /* [0x08] 创建时间戳(Unix) */
    uint32_t  ih_size;           /* [0x0C] 数据载荷字节数 */
    uint32_t  ih_load;           /* [0x10] 数据加载地址 */
    uint32_t  ih_ep;             /* [0x14] 内核入口点地址 */
    uint32_t  ih_dcrc;           /* [0x18] 数据CRC32校验和 */
    uint8_t   ih_os;             /* [0x1C] 操作系统类型 */
    uint8_t   ih_arch;           /* [0x1D] CPU架构 */
    uint8_t   ih_type;           /* [0x1E] 镜像类型 */
    uint8_t   ih_comp;           /* [0x1F] 压缩方式 */
    uint8_t   ih_name[IH_NMLEN]; /* [0x20] 镜像名称(32字节) */
} image_header_t;                /* 总计: 64字节 = 0x40 */

🔑 本节要点 · 第一性原理

uImage格式的本质是一个"自描述信封"：64字节头部告诉U-Boot "如何处置我的载荷"——包括放到哪（ih_load）、 从哪里开始跑（ih_ep）、如何解压（ih_comp）， 以及如何验证完整性（两个CRC32）。 魔数0x27051956是U-Boot能识别该文件的唯一凭证。

06mkimage工具详解

mkimage是U-Boot源码树中tools/mkimage.c编译产生的工具， 专门用于封装各类固件镜像。它是连接Linux内核构建系统与U-Boot引导的关键桥梁。

完整参数说明

mkimage -A arch      # 目标CPU架构: mips | arm | ppc | ...
        -O os        # 目标OS: linux | vxworks | ...
        -T type      # 镜像类型: kernel | ramdisk | multi | ...
        -C comp      # 压缩方式: none | gzip | bzip2 | lzma | lzo
        -a load_addr # 数据加载到RAM的地址（十六进制）
        -e entry_pt  # CPU执行入口点地址（十六进制）
        -n name      # 镜像名称（最多31字符）
        -d data_file # 输入数据文件
        output_file  # 输出uImage文件

MIPS平台（君正X2600类）标准调用

# gzip压缩版本
mkimage -A mips -O linux -T kernel -C gzip \
        -a 0x80010000 \
        -e 0x8039a050 \
        -n 'Linux-5.10.x-x2600' \
        -d arch/mips/boot/vmlinux.bin.gz \
        arch/mips/boot/uImage

# lzma压缩版本（Ingenic BSP常用）
mkimage -A mips -O linux -T kernel -C lzma \
        -a 0x80010000 \
        -e $(mips-linux-gnu-nm vmlinux | awk '/\bkernel_entry\b/{print "0x"$1}') \
        -n 'Linux-5.10.x' \
        -d arch/mips/boot/vmlinux.bin.lzma \
        arch/mips/boot/uImage
📌 参考来源

U-Boot doc/README.qemu-mips : "tools/mkimage -A mips -O linux -T kernel -C gzip -a 0x80010000 -e 0x80245650 -n 'Linux 2.6.24.y' -d vmlinux.bin.gz uImage"

多文件镜像（Multi-File Image）

# 将内核与initrd打包为一个multi-file uImage
mkimage -A mips -O linux -T multi -C none \
        -a 0x80010000 -e 0x80010000 \
        -n 'Linux+initrd' \
        -d vmlinux.bin:initrd.gz \
        uImage.multi
⚠️ 常见错误：mkimage版本不匹配

系统自带的/usr/bin/mkimage版本可能不支持某些架构或压缩格式。 应优先使用对应版本U-Boot源码中编译的mkimage（tools/mkimage）， 或通过内核Makefile调用scripts/mkuboot.sh包装脚本（会自动检测mkimage路径）。

🔑 本节要点

mkimage的核心职责是添加64字节自描述头部。 对MIPS Linux而言，-A mips -O linux -T kernel是固定组合， -a和-e必须与内核链接脚本和kernel_entry符号精确对应， 否则U-Boot可能将内核加载到错误位置或跳转到非法地址。

07FIT镜像格式（新一代）

除传统Legacy uImage外，U-Boot还支持FIT（Flattened Image Tree）格式， 基于设备树（FDT）语法描述，支持多镜像组合、多配置选择和强哈希验证。

FIT与Legacy格式对比

特性Legacy uImageFIT Image (.itb)
头部格式固定64字节C结构体FDT格式（DTS源文件描述）
多镜像支持有限（Multi-File）原生支持（kernel+dtb+initrd）
哈希算法CRC32CRC32 / SHA-1 / SHA-256
签名验证不支持支持RSA签名
配置选择不支持支持多配置切换
君正X2600使用✅ 常见（BSP默认）较新SDK版本

FIT镜像源文件示例（MIPS）

/* kernel.its — FIT镜像描述源文件 */
/dts-v1/;
/ {
    description = "X2600 Linux Kernel FIT";
    #address-cells = <1>;

    images {
        kernel@1 {
            description = "Linux kernel for X2600";
            data = /incbin/("./vmlinux.bin.lzma");
            type = "kernel";
            arch = "mips";
            os = "linux";
            compression = "lzma";
            load = <0x80010000>;
            entry = <0x803f58b0>;
            hash@1 { algo = "crc32"; };
            hash@2 { algo = "sha1";  };
        };
        fdt@1 {
            description = "X2600 Device Tree";
            data = /incbin/("./x2600.dtb");
            type = "flat_dt";
            arch = "mips";
            compression = "none";
            hash@1 { algo = "crc32"; };
        };
    };

    configurations {
        default = "config@1";
        config@1 {
            description = "X2600 Boot Config";
            kernel = "kernel@1";
            fdt = "fdt@1";
        };
    };
};
# 从.its源文件生成.itb二进制FIT镜像
mkimage -f kernel.its kernel.itb

09bootm命令深度解析

bootm是U-Boot中将已加载镜像引导执行的核心命令， 分为多个子阶段，每一步都有明确职责。

bootm执行子阶段

# bootm完整子命令序列（可分步执行）
bootm start  [addr]    # 解析uImage头部，验证魔数和架构
bootm loados           # 将OS镜像解压到ih_load地址
bootm ramdisk          # （可选）处理initrd/ramdisk
bootm fdt              # （可选）重定位设备树Blob
bootm cmdline          # 构造内核命令行（bootargs）
bootm prep             # OS特定的准备工作
bootm go               # 跳转到ih_ep，控制权交给内核

bootm的MIPS具体处理

# U-Boot环境变量配置示例（君正Ingenic典型）

# 从SPI Flash加载内核并引导
setenv bootcmd 'sf probe 0; sf read 0x80600000 0x40000 0x280000; bootm 0x80600000'

# 从NAND Flash加载
setenv bootcmd 'nand read 0x80600000 0x100000 0x500000; bootm 0x80600000'

# 从TFTP（开发调试用）
setenv bootcmd 'tftp 0x80600000 uImage; bootm 0x80600000'

# bootargs — 传递给Linux内核的命令行
setenv bootargs 'console=ttyS1,115200n8 mem=128M root=/dev/mtdblock3 rw \
    rootfstype=squashfs mtdparts=nand:256k(boot),2m(kernel),-(rootfs)'

ℹ️ bootm地址的含义

bootm 0x80600000中的0x80600000是uImage文件在RAM中的 暂存地址（"读取位置"），与ih_load（内核最终运行地址） 是不同的两个地址。

U-Boot会将暂存地址处的压缩内核数据解压到ih_load（如0x80010000）， 然后跳转到ih_ep（如0x803f58b0）执行。 这种两步分离设计确保了解压时不会覆盖源数据。

🔑 本节要点 · 第一性原理

bootm的本质是一个受控的"跳转代理"：它从uImage头部读取元数据， 完成解压和参数注入后，通过函数指针（或汇编jr指令）放弃CPU控制权， 跳转到内核入口点。这一设计模式体现了固件分层的核心思想： 每一层负责初始化自己所需的环境，然后将控制权精确移交给下一层。

10U-Boot → Linux内核的控制权交接

控制权交接是整个引导链路中最关键的一步，涉及CPU状态保存、MMU/Cache清理、 以及参数传递约定。

交接前的CPU状态准备

/* U-Boot在bootm go之前执行的MIPS特定清理工作 */

/* 1. 关闭所有中断 */
mtc0  t0, CP0_STATUS          /* 清除IE位 */

/* 2. Cache刷新（确保内核代码被写入内存）*/
/* 调用 flush_cache(0, ~0) */

/* 3. 禁用MMU（MIPS无需，kseg0/kseg1天然直接映射）*/
/* MIPS TLB在内核初始化前不会被内核使用 */

/* 4. 设置内核参数寄存器（MIPS o32 ABI）*/
move  a0, zero               /* argc */
la    a1, cmdline_start       /* 指向bootargs字符串 */

/* 5. 跳转 */
jr    s0                      /* s0 = ih_ep（kernel_entry地址）*/

12完整构建示例

将前面章节的知识点整合成一个完整的构建脚本，涵盖从交叉编译环境准备 到最终生成可被U-Boot引导的uImage的全过程。

构建环境准备

# 安装MIPS交叉编译工具链（Ubuntu/Debian）
apt-get install gcc-mipsel-linux-gnu binutils-mipsel-linux-gnu

# 或使用Ingenic官方SDK工具链（mips-linux-gnu，大端）
# 君正X2600通常使用小端（mipsel），需确认芯片字节序
export ARCH=mips
export CROSS_COMPILE=mipsel-linux-gnu-

# 安装mkimage（u-boot-tools包）
apt-get install u-boot-tools
# 或从U-Boot源码编译:
cd u-boot-ingenic && make tools && cp tools/mkimage /usr/local/bin/

内核配置与编译

# 克隆君正内核源码（以Ingenic community为例）
git clone https://github.com/Ingenic-community/linux.git
cd linux

# 使用X2600对应的defconfig
make ARCH=mips x2600_defconfig
# 或手动配置
make ARCH=mips menuconfig

# 编译内核（直接产出uImage）
make ARCH=mips \
     CROSS_COMPILE=mipsel-linux-gnu- \
     -j$(nproc) \
     uImage

# 成功时输出类似：
#   UIMAGE  arch/mips/boot/uImage
# Image Name:   Linux-5.10.x
# Image Type:   MIPS Linux Kernel Image (lzma compressed)
# Data Size:    2092002 Bytes = 2.0 MiB
# Load Address: 80010000
# Entry Point:  803xxxxx
# Image arch/mips/boot/uImage is ready

手动打包（若Makefile不直接支持）

# 步骤1: ELF → 裸二进制
mipsel-linux-gnu-objcopy -O binary -S -R .comment \
    vmlinux arch/mips/boot/vmlinux.bin

# 步骤2: 压缩
lzma -9 < arch/mips/boot/vmlinux.bin \
        > arch/mips/boot/vmlinux.bin.lzma

# 步骤3: 获取入口点
EP=$(mipsel-linux-gnu-nm vmlinux | \
     awk '/\bkernel_entry\b/{print "0x"$1}')
echo "Entry Point: $EP"

# 步骤4: 封装uImage
mkimage -A mips \
        -O linux \
        -T kernel \
        -C lzma \
        -a 0x80010000 \
        -e ${EP} \
        -n "Linux-5.10.x-X2600" \
        -d arch/mips/boot/vmlinux.bin.lzma \
        arch/mips/boot/uImage

# 步骤5: 验证
mkimage -l arch/mips/boot/uImage

烧录到Flash（eMMC示例）

# 在U-Boot命令行，通过TFTP传输并烧录
# 1. 设置网络
setenv ipaddr 192.168.1.100
setenv serverip 192.168.1.1

# 2. 下载uImage到RAM暂存
tftp 0x80600000 uImage

# 3. 烧录到eMMC内核分区
mmc write 0x80600000 0x800 0x1000    # 扇区偏移和大小（示例）

# 或 写入SPI Flash
sf probe 0
sf erase 0x40000 0x280000           # 先擦除
sf write 0x80600000 0x40000 ${filesize}

# 4. 设置启动命令并保存
setenv bootcmd 'sf probe 0; sf read 0x80600000 0x40000 0x280000; bootm 0x80600000'
setenv bootargs 'console=ttyS1,115200n8 mem=128M root=/dev/mmcblk0p3 rootfstype=squashfs ro'
saveenv

第一性原理总结

🌟 从第一性原理看整个过程

整个Linux MIPS内核镜像打包与引导的本质，是解决一个核心问题： "如何让一个裸CPU安全地找到并执行操作系统？"

MIPS的答案是：①用硬连线的地址段（kseg）保证上电后总有代码可以执行； ②用分层的Bootloader（BootROM→SPL→U-Boot）逐步建立复杂的硬件环境（DDR、Flash、外设）； ③用自描述的镜像格式（uImage）让最后一级Bootloader无需额外配置即可知道"把什么放到哪里、从哪里开始跑"； ④用CRC32校验确保传输和存储过程中的数据完整性。

uImage = 压缩内核 + 元数据信封，这个信封（64字节头部）携带的 负载地址（ih_load）和入口点（ih_ep）是连接 编译时链接脚本与运行时Bootloader的唯一契约。违反这个契约，系统就无法启动。
