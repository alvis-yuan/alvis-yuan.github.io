第一章 · 概述

引言与背景

Linux 内核热插拔机制与 mdev 应用层交互的技术全景

🔌什么是热插拔（Hotplug）

热插拔（Hotplug）是指在系统运行过程中，无需重启即可动态地添加或移除硬件设备的机制。当一个设备插入或拔出系统时，内核必须能够：

  • 检测硬件状态变化（通过总线中断或轮询）
  • 在内核数据结构中注册/注销设备
  • 通知用户空间，以便自动创建设备节点、加载驱动或执行自定义脚本

Linux 2.6.x 内核通过 sysfs 导出设备树，配合 uevent 热插拔机制，使用户空间工具（如 udev 或 mdev）能够动态维护 /dev 目录下的设备节点。

📚 参考来源

📄 LWN: sysfs/hotplug 官方文档 (Rob Landley, 2007) 📄 Linux Kernel Labs: Device Model
🗂️mdev 的定位

mdev 是 BusyBox 提供的一个轻量级设备管理器，是 udev 的精简替代品，专为嵌入式 Linux 系统设计。

特性mdev（BusyBox）udev（完整 Linux）
运行模式由内核直接 fork（无守护进程）守护进程 udevd 常驻内存
事件接收uevent_helper（/proc/sys/kernel/hotplug）Netlink 套接字
适用场景嵌入式系统、内存受限环境桌面/服务器 Linux
体积约 5KB较大
配置文件/etc/mdev.conf/etc/udev/rules.d/*.rules
系统层次架构总览
🔭 第一性原理解读

热插拔的本质是一个状态变化通知系统：硬件状态从"不存在"变为"存在"（或反之），这个变化必须沿"硬件→内核→用户空间"的链路传播，使系统各层级保持一致的认知。mdev 是这个链路在用户空间的末端执行者——它的核心工作是将内核描述的设备信息（设备号、路径、子系统）物化为 /dev 下的节点文件，让应用程序能够通过文件系统接口访问硬件。

第二章 · 内核层

Linux 设备模型基础

kobject、kset、bus_type、device、class 的层次结构

🏗️Linux 设备模型（Linux Device Model）

Linux 2.6 引入了统一设备模型（LDM），其核心目标是维护内部数据结构，反映系统的状态与结构。该模型通过 sysfs 虚拟文件系统（通常挂载于 /sys）向用户空间暴露内核数据结构。

模型中的核心实体包括：

  • device（设备）：连接到总线的物理设备
  • driver（驱动）：与设备关联并操作设备的软件实体
  • bus（总线）：设备连接到处理器的通信信道
  • class（类）：按功能分组的设备类型（磁盘、串口等）
Linux 设备模型层次结构

sysfs 目录结构

sysfs 的子目录与内核设备模型直接对应：

/sys/
├── block/          # 块设备（磁盘、分区）
├── bus/            # 总线类型（pci, usb, i2c, platform...）
│   └── usb/devices/1-1/dev   # 包含 major:minor
├── class/          # 设备类（net, input, tty, block...）
│   └── tty/ttyUSB0/dev       # 例: "188:0\n"
├── devices/        # 设备树层次结构
├── firmware/       # 固件信息（ACPI）
├── kernel/         # 内核状态（含 uevent_helper）
└── module/         # 已加载模块列表
关键文件： /sys/class/*/*/dev 文件包含设备的主次设备号（格式 MAJOR:MINOR），mdev 通过读取此文件来调用 mknod() 创建 /dev 下的设备节点。
例：cat /sys/class/mem/zero/dev → 1:5

device_create() 触发 uevent

驱动调用 device_create() 后，内核会在 sysfs 中创建 dev 属性文件（包含设备号），并自动触发 KOBJ_ADD uevent，通知用户空间新设备已就绪：

/* 驱动注册设备（典型代码路径） */
struct device *my_dev;
my_dev = device_create(my_class, NULL,
                        MKDEV(major, minor),
                        NULL, "mydevice%d", minor);
/* 内部调用链：
 * device_create()
 *   → device_register()
 *     → device_add()
 *       → kobject_uevent(&dev->kobj, KOBJ_ADD)  ← 触发热插拔通知
 */
🔭 第一性原理解读

Linux 设备模型的本质是用统一数据结构抽象差异硬件：不管是 USB、PCI 还是 I2C 设备，都通过 struct device（内嵌 kobject）表示，并以树形结构存储于 sysfs。这个"统一视图"是整个热插拔机制的信息源头——所有后续的 uevent 通知、设备节点创建，都依赖于 sysfs 中准确的设备信息（特别是 dev 文件的设备号）。

第三章 · 内核层

kobject 与 uevent 机制

kobject_uevent_env() 源码解析与环境变量构建

⚙️kobject_uevent() 核心函数

uevent 框架实现于 lib/kobject_uevent.c，核心 API 为：

  • kobject_uevent(kobj, action) — 发送基础 uevent
  • kobject_uevent_env(kobj, action, envp_ext[]) — 发送带额外环境变量的 uevent

kobject_action 事件类型

/* include/linux/kobject.h */
enum kobject_action {
    KOBJ_ADD,      /* 设备/驱动添加 → "add"    */
    KOBJ_REMOVE,   /* 设备/驱动移除 → "remove" */
    KOBJ_CHANGE,   /* 设备状态变化  → "change" */
    KOBJ_MOVE,     /* kobject 重命名 → "move"  */
    KOBJ_ONLINE,   /* CPU 上线      → "online" */
    KOBJ_OFFLINE,  /* CPU 下线      → "offline"*/
};

kobject_uevent_env() 执行流程

/* lib/kobject_uevent.c（简化） */
int kobject_uevent_env(struct kobject *kobj,
                       enum kobject_action action,
                       char *envp_ext[])
{
    struct kobj_uevent_env *env;
    struct kset *kset;
    const struct kset_uevent_ops *uevent_ops;

    /* 1. 向上遍历父节点，找到所属的 kset（即 subsystem） */
    top_kobj = kobj;
    while (!top_kobj->kset && top_kobj->parent)
        top_kobj = top_kobj->parent;
    kset = top_kobj->kset;
    uevent_ops = kset->uevent_ops;

    /* 2. 过滤检查（uevent_suppress 或 filter 函数） */
    if (kobj->uevent_suppress)  return 0;
    if (uevent_ops && uevent_ops->filter)
        if (!uevent_ops->filter(kset, kobj)) return 0;

    /* 3. 构建默认环境变量 */
    env = kzalloc(sizeof(*env), GFP_KERNEL);
    devpath = kobject_get_path(kobj, GFP_KERNEL);
    add_uevent_var(env, "ACTION=%s", action_string);  /* "add"/"remove" 等 */
    add_uevent_var(env, "DEVPATH=%s", devpath);       /* /sys 中的路径 */
    add_uevent_var(env, "SUBSYSTEM=%s", subsystem);   /* kset 名（子系统）*/

    /* 4. 附加调用者传入的额外环境变量 */
    for (i = 0; envp_ext && envp_ext[i]; i++)
        add_uevent_var(env, "%s", envp_ext[i]);

    /* 5. 调用 kset->uevent_ops->uevent() 添加子系统专属变量
     *    例如 bus/class 的 uevent() 会添加 MAJOR、MINOR、DEVNAME 等 */
    if (uevent_ops && uevent_ops->uevent)
        uevent_ops->uevent(kset, kobj, env);

    /* 6. 添加全局序列号 */
    add_uevent_var(env, "SEQNUM=%llu", ++uevent_seqnum);

    /* 7a. 通过 Netlink 广播给监听守护进程（如 udevd）
     *     需要 CONFIG_NET 支持 */
    kobject_uevent_net_broadcast(kobj, env, action_string, devpath);

    /* 7b. 通过 uevent_helper（call_usermodehelper）调用 mdev */
    if (uevent_helper[0]) {
        char *argv[3];
        argv[0] = uevent_helper;    /* 通常是 "/sbin/mdev" */
        argv[1] = (char *)subsystem; /* 子系统名称 */
        argv[2] = NULL;
        call_usermodehelper(argv[0], argv, envp, UMH_NO_WAIT);
    }
}
kobject_uevent_env() 执行流程图
🔭 第一性原理解读

kobject_uevent_env() 的本质是内核内部状态变化的"广播者"：它将内核数据结构中已经确定的事实（某设备已被注册，其设备号为 X:Y，路径为 P）翻译成一组键值对（环境变量），然后通过两条并行通道广播出去。这个设计体现了"生产者-消费者"模型：内核只负责产生事件和数据，用户空间工具（mdev/udev）负责消费和响应。

第四章 · 内核层

uevent 两种通知路径

Usermode Helper（uevent_helper）与 Netlink 的对比与实现细节

官方文档原文（LWN.net 2007，Rob Landley）：
"Linux provides two interfaces to hotplug; the kernel can spawn a usermode helper process, or it can send a message to an existing daemon listening to a netlink socket."

路径一：Usermode Helper（uevent_helper）

🚀调用机制
  • 内核为每个热插拔事件单独 fork 一个进程，运行 uevent_helper 指定的程序
  • 子进程属于 root 用户（UID 0），是 init 进程（PID 1）的子进程
  • 热插拔事件信息通过环境变量传递给子进程
  • 默认调用 /sbin/hotplug，可通过写入 /proc/sys/kernel/hotplug 修改
注意：此机制需要内核配置项 CONFIG_UEVENT_HELPER 启用。现代内核中，该选项有时默认禁用，倾向于使用 Netlink 方式。

设置 mdev 为 uevent_helper

# 方法1：运行时动态设置（最常用）
echo /sbin/mdev > /proc/sys/kernel/hotplug

# 方法2：sysctl 接口（等价于方法1）
sysctl -w kernel.hotplug=/sbin/mdev

# 方法3：内核编译时配置（Kconfig）
CONFIG_UEVENT_HELPER=y
CONFIG_UEVENT_HELPER_PATH="/sbin/mdev"

# 禁用 uevent_helper（让 udevd 通过 Netlink 处理）
echo "" > /proc/sys/kernel/hotplug

call_usermodehelper() 内核实现

/* lib/kobject_uevent.c */
if (uevent_helper[0]) {
    char *argv[3];
    argv[0] = uevent_helper;    /* "/sbin/mdev" */
    argv[1] = (char *)subsystem; /* 子系统名，如 "block", "usb" */
    argv[2] = NULL;

    /* call_usermodehelper 将任务放入 system_unbound_wq 工作队列 */
    retval = call_usermodehelper(argv[0], argv, env->envp, UMH_NO_WAIT);
}
/* 内核环境变量（env->envp）包含：
 *   HOME=/
 *   PATH=/sbin:/bin:/usr/sbin:/usr/bin
 *   ACTION=add
 *   DEVPATH=/class/xxx/yyy
 *   SUBSYSTEM=xxx
 *   MAJOR=189
 *   MINOR=0
 *   SEQNUM=1234
 */

路径二：Netlink Socket（NETLINK_KOBJECT_UEVENT）

🔗Netlink 广播机制
  • 需要 CONFIG_NET 内核配置
  • 内核调用 netlink_broadcast_filtered() 向 NETLINK_KOBJECT_UEVENT family 广播
  • 数据包包含一组以 \0 分隔的文本行（第一行为 ACTION@DEVPATH）
  • 监听守护进程（udevd）在 socket 上 recv() 接收，无需每次 fork
  • udev/mdev 的 netlink 模式需要 busybox 的 uevent 守护进程支持

Netlink 数据包格式

/* Netlink 数据包：以 \0 分隔的文本行 */
"add@/class/input/input9/mouse2\0"  ← 第一行：ACTION@DEVPATH
"ACTION=add\0"
"DEVPATH=/class/input/input9/mouse2\0"
"SUBSYSTEM=input\0"
"SEQNUM=1064\0"
"MAJOR=13\0"
"MINOR=34\0"
/* 来源：Stack Overflow "uevent sent from kernel to user space" */

用户空间监听 Netlink uevent（示例）

/* 来源：LWN.net Articles/242046（Rob Landley, 2007） */
int fd = socket(PF_NETLINK, SOCK_DGRAM, NETLINK_KOBJECT_UEVENT);
struct sockaddr_nl nls = {
    .nl_family = AF_NETLINK,
    .nl_pid    = getpid(),
    .nl_groups = -1   /* 订阅所有 multicast group */
};
bind(fd, (void*)&nls, sizeof(nls));

char buf[512];
while (1) {
    int len = recv(fd, buf, sizeof(buf), MSG_DONTWAIT);
    int i = 0;
    while (i < len) {
        printf("%s\n", buf+i);   /* 打印每一行键值对 */
        i += strlen(buf+i) + 1;
    }
}

对比项Netlink 路径uevent_helper 路径

🔭 第一性原理解读

两种通知路径的根本差异在于进程模型的选择：Netlink 是"长连接多路复用"——一个守护进程处理所有事件；uevent_helper 是"短连接一次性"——每事件一个进程。mdev 选择后者，代价是每次 fork 的开销，换来的是极简的部署（无需守护进程）和在 init 前工作的能力——这是嵌入式系统对资源与简单性的折中。

第五章 · 用户空间

mdev 工作原理

BusyBox mdev 源码解析：mdev_main() 与 make_device()

📦mdev 两种工作模式
  1. 热插拔模式：由内核通过 call_usermodehelper() 直接调用（无参数），通过读取环境变量处理单个设备事件
  2. 冷插拔扫描模式：系统启动时执行 mdev -s，递归扫描 /sys/class 和 /sys/block，为已存在设备补全 /dev 节点

mdev_main() 源码解析

/* busybox/util-linux/mdev.c（基于 busybox-1.23.1）*/
int mdev_main(int argc, char **argv)
{
    xchdir("/dev");   /* 切换工作目录到 /dev */

    if (argv[1] && strcmp(argv[1], "-s") == 0) {
        /* ===== 冷插拔模式：mdev -s ===== */
        putenv((char*)"ACTION=add");

        /* 若 /sys/class/block 不存在，才扫描旧的 /sys/block */
        if (access("/sys/class/block", F_OK) != 0)
            recursive_action("/sys/block",
                ACTION_RECURSE | ACTION_FOLLOWLINKS | ACTION_QUIET,
                fileAction, dirAction, temp, 0);

        /* 递归扫描 /sys/class，对每个含 "dev" 的目录调用 fileAction */
        recursive_action("/sys/class",
            ACTION_RECURSE | ACTION_FOLLOWLINKS,
            fileAction, dirAction, temp, 0);
    }
    else {
        /* ===== 热插拔模式：由内核直接 fork 调用 ===== */
        /* 读取内核通过 call_usermodehelper 设置的环境变量 */
        env_devname = getenv("DEVNAME");   /* 可为 NULL */
        G.subsystem = getenv("SUBSYSTEM");
        action      = getenv("ACTION");    /* "add" 或 "remove" */
        env_devpath = getenv("DEVPATH");  /* /sys 中的路径 */

        /* 构建 sysfs 完整路径：/sys + DEVPATH */
        snprintf(temp, PATH_MAX, "/sys%s", env_devpath);

        /* 核心函数：根据 sysfs 信息创建/删除 /dev 节点 */
        make_device(env_devname, temp, op);
    }
}

make_device() — 核心设备节点创建

/* make_device() 关键逻辑（简化）*/
static void make_device(char *devname, char *path, int op)
{
    /* 1. 读取 /etc/mdev.conf 配置文件 */
    fd = open("/etc/mdev.conf", O_RDONLY);

    /* 2. 读取 sysfs dev 文件获取设备号
     *    /sys/class/tty/ttyUSB0/dev → "188:0"
     */
    fd = open(path "/dev", O_RDONLY);
    read(fd, dev_info, ...);
    major = ...; minor = ...;

    /* 3. 匹配 mdev.conf 规则，获取权限/用户/组/命令 */
    match_mdev_conf_rule(devname, &uid, &gid, &mode, &command);

    /* 4. 创建设备节点（ACTION=add 时） */
    if (op == OP_add) {
        mknod(devname, mode | dev_type, makedev(major, minor));
        chown(devname, uid, gid);

        /* 5. 执行 mdev.conf 中 @ 标记的命令（创建后脚本）*/
        if (command && command[0] == '@')
            system(command + 1);
    }
    /* 6. 删除设备节点（ACTION=remove 时） */
    else if (op == OP_remove) {
        /* 执行 $ 标记命令（删除前脚本）*/
        if (command && command[0] == '$')
            system(command + 1);
        unlink(devname);
    }
}
mdev 内部处理流程
🔭 第一性原理解读

mdev 的本质是一个翻译器：它将内核用"文件路径+键值对"描述的设备（sysfs 树 + 环境变量）翻译为操作系统文件系统中的特殊文件（字符/块设备节点）。这个翻译工作的核心是两个系统调用：从 /sys/class/xxx/dev 读取设备号，然后调用 mknod() 以该设备号创建节点。设备节点的本质是内核设备驱动的文件系统入口点。

第七章 · 深入分析

冷插拔：mdev -s

系统启动时的设备节点批量创建机制

🔄冷插拔（Cold Plug）问题

系统启动时，内核在 init 进程启动之前就已经完成了大量设备的检测和注册。由于此时 mdev 还未运行，这些设备的 uevent 事件已经错过。

解决方案：mdev -s（scan 模式）— 主动扫描 sysfs，补全所有已存在设备的 /dev 节点。

另一种方法（较新内核）： 通过向 /sys/*/uevent 文件写入 "add" 重新触发 uevent 事件：
for i in $(find /sys -name uevent); do echo add > $i; done

系统初始化标准脚本

#!/bin/sh
# /etc/init.d/rcS — 系统启动初始化脚本
# 来源：BusyBox mdev 文档（busybox-mdev.sh）

# 1. 挂载必要的虚拟文件系统
mount -t tmpfs -o size=64k,mode=0755 tmpfs /dev  # /dev 使用 tmpfs
mount -t sysfs sysfs /sys                          # sysfs 暴露内核数据
mount -t proc proc /proc                           # procfs（含 /proc/sys）

# 2. 创建常用设备目录
mkdir /dev/pts
mount -t devpts devpts /dev/pts                    # 伪终端支持

# 3. 注册 mdev 为内核热插拔 helper
#    写入后，内核 uevent_helper 变量被更新
#    /sys/kernel/uevent_helper 也同步变化
echo /sbin/mdev > /proc/sys/kernel/hotplug

# 4. 冷插拔扫描：为已存在设备创建 /dev 节点
#    扫描 /sys/class 和 /sys/block 下所有含 "dev" 的目录
mdev -s

# 5. 启动用户空间服务...
exec /sbin/init "$@"
mdev -s 冷插拔扫描流程

典型初始化顺序

  • 1
    挂载 tmpfs 到 /dev
    使用内存文件系统，设备节点在每次启动时重新创建，不持久化到磁盘
  • 2
    挂载 sysfs 到 /sys
    sysfs 是 mdev 读取设备信息（设备号、路径）的数据源，必须在 mdev -s 前就绪
  • 3
    注册 mdev 为热插拔 helper
    echo /sbin/mdev > /proc/sys/kernel/hotplug — 此后内核每次 uevent 都会 fork mdev
  • 4
    执行 mdev -s 冷插拔扫描
    一次性为所有已注册设备创建 /dev 节点，弥补启动阶段丢失的 uevent
  • 5
    后续热插拔自动处理
    通过 uevent_helper 机制，每次设备插入/拔出都自动触发 mdev
🔭 第一性原理解读

冷插拔问题的根源是时序依赖：用户空间工具（mdev）必须先于它需要处理的事件启动，但在启动早期，内核已经在没有用户空间的情况下独立注册了设备。mdev -s 的本质是一次状态同步操作——将内核当前的设备状态（sysfs 树）与用户空间的期望状态（/dev 目录）强制对齐。这类似于数据库的"全量同步"，之后的热插拔则是"增量同步"。

第八章 · 深入分析

环境变量详解

内核传递给 mdev 的完整环境变量说明

官方文档原文（LWN.net 2007，Rob Landley）：
"Every hotplug event should provide at least the following variables: ACTION, DEVPATH, SUBSYSTEM. The following variables are also provided for some devices: MAJOR, MINOR, DRIVER, INTERFACE, IFINDEX, FIRMWARE."

必须存在的环境变量

变量名示例值说明
ACTION add / remove / change / move 热插拔动作类型，对应 kobject_action 枚举值转换的字符串
DEVPATH /class/tty/ttyUSB0 设备在 /sys 下的路径（不含 /sys 前缀），以 /block/ 开头为块设备，否则为字符设备
SUBSYSTEM usb / block / input / net 设备所属子系统（kset 的名称），"block" 表示块设备，其他为字符设备
SEQNUM 1064 全局递增序列号，用于事件排序（各 namespace 独立计数，不保证连续）

可选环境变量

变量名示例值说明
MAJOR 188 设备主设备号，与 MINOR 一起用于 mknod() 创建设备节点
MINOR 0 设备次设备号
DEVNAME ttyUSB0 建议的设备节点名称，mdev 优先使用此名称创建 /dev 节点
DRIVER ftdi_sio 建议使用的驱动模块名，与设备是否有驱动无关
INTERFACE eth0 当 SUBSYSTEM=net 时，网络接口名称
IFINDEX 2 当 SUBSYSTEM=net 时，接口唯一整数标识
FIRMWARE dvb-usb-vp7045-01.fw 固件文件名，内核请求用户空间提供固件时设置（触发固件加载流程）
MODALIAS usb:v0403pFA00d0600... 设备别名，用于 modprobe 自动加载驱动模块

🔭 第一性原理解读

环境变量是内核与用户空间之间的语义契约：内核承诺提供 ACTION、DEVPATH、SUBSYSTEM 这三个最小集合，使用户空间能够确定"什么事件、发生在哪个设备上、属于哪个子系统"。MAJOR/MINOR 是设备节点创建的充分条件，其余变量提供可选的附加语义。这种基于键值对的接口设计具有良好的前向兼容性——新增变量不会破坏旧的处理程序。

第九章 · 深入分析

mdev.conf 配置详解

规则格式、权限设置与自动挂载脚本

📋mdev.conf 规则格式

来源：BusyBox 官方文档。mdev.conf 每行定义一条规则：

# 格式（BusyBox mdev 文档）：
# <设备名正则> <uid>:<gid> <权限八进制> [=路径] [<@$*><命令>]
#
# 命令触发时机：
#   @ 命令：设备节点创建 之后 执行
#   $ 命令：设备节点删除 之前 执行
#   * 命令：创建后和删除前都执行

# 示例：USB 串口设备
ttyUSB[0-9]+    0:0 0660 @/etc/hotplug/usb/serial_insert

# 示例：SD 卡插入/拔出
mmcblk[0-9]p[0-9]  0:0 0660  @/etc/hotplug/sd/sd_insert
mmcblk[0-9]        0:0 0660  $/etc/hotplug/sd/sd_remove

# 示例：U 盘插入/拔出
sd[a-z][0-9]       0:0 0660  @/etc/hotplug/usb/udisk_insert
sd[a-z]            0:0 0660  $/etc/hotplug/usb/udisk_remove

# 示例：设置特定设备的权限（允许普通用户访问视频设备）
video[0-9]+        0:44 0660

自动挂载 U 盘脚本示例

#!/bin/sh
# /etc/hotplug/usb/udisk_insert — U 盘插入后自动挂载

MOUNT_POINT=/mnt/usb

# 等待设备节点就绪
sleep 1

# 自动检测文件系统类型并挂载
mkdir -p $MOUNT_POINT
mount -t auto /dev/${MDEV} $MOUNT_POINT && \
    echo "U盘已挂载到 $MOUNT_POINT" > /dev/console
#!/bin/sh
# /etc/hotplug/usb/udisk_remove — U 盘拔出前自动卸载

sync
umount /mnt/usb 2>/dev/null
echo "U盘已安全弹出" > /dev/console
mdev.conf 规则匹配与执行流程
🔭 第一性原理解读

mdev.conf 是一个策略引擎：它将"当什么设备出现时，做什么操作"的策略与机制分离。机制由 mdev 内核代码实现（固定的 mknod/unlink 流程），策略由管理员通过配置文件定制（权限、自动挂载脚本）。这种分离使得嵌入式系统可以在不重新编译 mdev 的情况下，灵活定制设备管理行为，体现了"机制与策略分离"的 Unix 哲学。

第十章 · 深入分析

固件加载流程

内核请求固件时的 uevent 交互与用户空间响应

官方文档原文（LWN.net 2007，Rob Landley）：
"If the hotplug variable FIRMWARE is set, the kernel is requesting firmware for a device (identified by $DEVPATH). To provide the firmware to the kernel, do the following:
echo 1 > /sys/$DEVPATH/loading
cat /path/to/$FIRMWARE > /sys/$DEVPATH/data
echo 0 > /sys/$DEVPATH/loading"
📡固件加载触发条件

当设备驱动在 probe 阶段调用 request_firmware() 时，内核会通过 uevent 机制向用户空间发出固件请求，环境变量中设置 FIRMWARE=固件文件名。

  • 固件加载超时时间在 /sys/class/firmware/timeout 中配置（单位：秒）
  • 可通过 echo -1 > /sys/$DEVPATH/loading 取消固件加载
  • uevent_helper 的一个优势：可在 initramfs 中的 init 启动前就处理固件请求（Netlink 方式无法做到）

固件加载交互流程

固件加载 MSC 序列图

固件加载用户空间脚本示例

#!/bin/sh
# 固件加载脚本（由 mdev.conf 或直接由 mdev 调用）
# 环境变量由内核 kobject_uevent_env() 设置

FIRMWARE_DIR=/lib/firmware

if [ "$ACTION" = "add" ] && [ -n "$FIRMWARE" ]; then
    if [ -f "$FIRMWARE_DIR/$FIRMWARE" ]; then
        # 告知内核开始接收固件
        echo 1 > /sys/$DEVPATH/loading
        # 将固件文件写入内核接口
        cat $FIRMWARE_DIR/$FIRMWARE > /sys/$DEVPATH/data
        # 告知内核固件传输完成
        echo 0 > /sys/$DEVPATH/loading
    else
        # 取消固件加载（触发 -ECANCELED）
        echo -1 > /sys/$DEVPATH/loading
    fi
fi
🔭 第一性原理解读

固件加载流程揭示了一个优雅的双向交互模式：通常 uevent 是内核→用户空间的单向通知，但固件加载建立了一个请求-响应对话——内核通过 uevent 提问（"我需要固件 X"），用户空间通过 sysfs 文件接口回答（写入固件数据）。sysfs 此时充当了内核与用户空间之间的共享内存缓冲区，实现了跨越内核/用户边界的数据传输，而无需额外的系统调用接口。

🔭 综合第一性原理解读

Linux 热插拔机制的整体设计体现了几个核心原则：

1. 分层解耦：每层只关心自己的职责——硬件检测交给总线驱动，设备模型维护交给 kobject 子系统，事件传播交给 uevent 框架，节点创建交给 mdev。

2. 事件驱动：整个系统由状态变化驱动，而非轮询。uevent 是内核对状态变化的"广播"，mdev 是订阅者的"响应"。

3. sysfs 作为真相源：所有设备信息最终存储在 sysfs 中，用户空间工具总是从 sysfs 读取权威数据（设备号、路径），而非依赖事件消息本身——这使得系统即使在错过事件时也能通过扫描 sysfs 恢复正确状态（mdev -s）。

4. 机制与策略分离：内核提供机制（uevent、sysfs、call_usermodehelper），用户空间定义策略（mdev.conf 规则、挂载脚本）——这是 Unix 设计哲学的直接体现。
