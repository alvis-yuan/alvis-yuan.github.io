🗺 整体架构分层

Linux 内核路由子系统是 IPv4/IPv6 网络栈的核心决策中枢。其核心职责只有一个：对于每一个经过内核的数据包，决定它应该去哪里。该子系统围绕"查找"这一基本操作展开，形成了从管理接口到快速数据路径的完整分层体系。

图1：Linux 路由子系统层次架构（基于 v5.10 源码）
🔬 第一性原理

路由子系统的本质是一个查找系统：给定目标 IP 地址，找到最合适的下一跳（nexthop）和输出设备。所有的数据结构、算法、缓存都围绕这一核心问题展开。分层架构的目的是将"控制面"（如何管理路由表）与"数据面"（如何快速查找路由）分离，以达到高效率和可维护性的平衡。

📋 FIB 转发信息库（Forwarding Information Base）

FIB（转发信息库）是 Linux 内核路由子系统的核心数据库，存储所有路由条目。与用户可见的"路由表"不同，FIB 是内核内部经过优化的数据结构，专为高速查找而设计。

RTN_THROW9抛出继续查找下一张表
🔬 第一性原理

FIB 存在的根本原因是：路由决策需要极低延迟（纳秒级），而路由表可能有数十万条目。将路由信息组织为 FIB（而非简单列表）是从第一性原理推导的必然结果——查找时间复杂度必须独立于路由表大小。fib_info 将"如何转发"（nexthop）与"匹配什么前缀"（FIB trie）分离，实现了数据最小化存储和引用共享。

🗄 路由表结构与多表机制

Linux 支持多张路由表（最多 2^32 张，ID 0~4294967295），这是策略路由的基础。内核在启动时默认创建三张表，通过编译选项 CONFIG_IP_MULTIPLE_TABLES 决定是否支持多表。

路由表 Hash 数组定义于 struct netns_ipv4.fib_table_hash（include/net/netns/ipv4.h），每个 network namespace 拥有独立的 FIB 表集合。表的注册入口为 fib_magic()（net/ipv4/fib_frontend.c），在接口地址变更时自动调用。

🔬 第一性原理

多路由表的存在是因为：单一路由表无法表达"对不同流量使用不同路径"这一需求。第一性原理告诉我们，路由决策的本质是一个函数 f(packet) → nexthop。当 f 的输入维度超出"目标IP"时（如需要考虑源IP、TOS、防火墙标记），就必须有多张表和策略规则来表达更复杂的映射关系。

🎯 路由决策对象：rtable 与 dst_entry

路由查找的结果不是一个简单的下一跳地址，而是一个路由决策对象（routing decision object）。该对象被附加到数据包（sk_buff）上，并通过函数指针 input/output 驱动数据包后续的处理路径。

🔬 第一性原理

使用函数指针（input/output）而非 if-else 分支来驱动数据包流转，体现了策略模式（Strategy Pattern）的第一性原理：将"决定做什么"与"如何做"解耦。路由查找时一次性设置好函数指针，后续处理无需重复判断——这是将查找代价摊销到单次操作的最优解。

🌳 LC-Trie 算法：FIB 查找核心

自 Linux 2.6.39 起，IPv4 FIB 的最长前缀匹配算法使用 LC-Trie（Level Compressed Trie，层次压缩字典树），也称 Patricia Trie。该算法由 S. Nilsson 和 G. Karlsson 于 1999 年在 IEEE JSAC 发表的论文 "IP-address lookup using LC-tries" 中提出。

📌 官方文档引用

内核官方文档 Documentation/networking/fib_trie.rst 对此算法有详细说明，以下内容严格基于该文档和 net/ipv4/fib_trie.c 源码。

节点类型与核心概念

🍃 叶节点（leaf）

终端节点，包含实际路由数据。存储完整的键（IP地址前缀），并有按前缀长度排序的路由条目 hlist。对应 struct key_vector（bits=0）。

🔵 内部节点（tnode）

中间节点，持有子节点（leaf 或 tnode）的指针数组，通过键的某个子集来索引子数组。对应 struct key_vector（bits>0）+ struct tnode。

📍 Pos（位置）

当前 tnode 使用键的哪一位段作为子数组索引的起始位置。实现"路径压缩"——跳过只有单一路径的位段。

📏 Bits（位宽）

用于索引子数组的键位段的宽度。子数组大小为 2^bits。实现"层次压缩"——允许每个节点持有多个子节点，而非二叉树。

/* 键向量：leaf(bits=0) 和 tnode(bits>0) 的统一表示 */
struct key_vector {
    t_key   key;          /* 该节点关联的键（IP地址，大端序） */
    unsigned char pos;   /* 键段的起始位位置（路径压缩）*/
    unsigned char bits;  /* 键段的位宽（0=leaf, >0=tnode） */
    unsigned char slen;  /* 该节点下最长前缀长度 */
    union {
        /* tnode: 子节点指针数组，大小为 2^bits */
        struct key_vector __rcu *tnode[0];
        /* leaf: 路由条目 hlist */
        struct hlist_head leaf;
    };
};

/* 获取子节点索引：XOR 键差后右移到 pos 位 */
#define get_cindex(key, kv)  (((key) ^ (kv)->key) >> (kv)->pos)

/* fib_table_lookup 核心逻辑（简化） */
int fib_table_lookup(struct fib_table *tb,
                     const struct flowi4 *flp,
                     struct fib_result *res, int fib_flags)
{
    struct trie *t = (struct trie *)tb->tb_data;
    struct key_vector *n, *pn;
    t_key key = ntohl(flp->daddr); /* 目标地址转主机字节序 */
    unsigned long index;
    t_key cindex;

    pn = t->kv;         /* 从根开始 */
    cindex = 0;

    for (;;) {
        index = get_index(key, pn); /* 计算子节点索引 */
        if (index >= child_length(pn))
            break;

        n = get_child_rcu(pn, index); /* 取子节点（RCU 安全）*/
        if (!n)
            break;

        if (IS_LEAF(n)) {
            /* 到达叶节点，做语义匹配（TOS/scope等）*/
            if (check_leaf(tb, t, n, key, flp, res, fib_flags) > 0)
                goto found;
        }
        pn = n; /* 继续下降 */
    }

    /* 未直接命中，进入回溯最长前缀匹配模式 */
    do {
        pn = node_parent_rcu(pn);
        /* 逐步截短键，回溯查找更短匹配前缀 */
        /* t->stats.backtrack++ */
    } while (pn);

found:
    return err;
}
🔬 第一性原理

LC-Trie 的设计哲学是用空间换时间，用自适应压缩换均匀性能。从第一性原理看，IP 地址的前缀查找本质是一个 32 位整数的前缀比较问题。Trie 是这一问题的自然数据结构，而"路径压缩"消除了无效的单路径节点，"层次压缩"将多个串行决策合并为一次数组索引——这是在算法信息论层面的最优设计。resize() 的动态调整保证了 Trie 随路由表变化始终保持最优形态。

🎯 最长前缀匹配（LPM）机制

最长前缀匹配（Longest Prefix Match）是路由查找的核心算法原则：当多条路由都能匹配目标 IP 时，选择前缀最长（最具体）的那条。

图6：最长前缀匹配示例——192.168.0.0/24 在三个匹配中胜出

回溯查找流程

当 LC-Trie 下降到叶节点发现无精确匹配时，触发回溯（backtrack）机制，逐步减少前缀长度向上查找：

伪代码 — Documentation/networking/fib_trie.rst 描述的查找流程 
/* LPM 查找逻辑（基于内核官方文档描述）*/
1. 下降 trie，键段逐段匹配，直到叶节点
2. 对叶节点执行 check_leaf()（语义匹配：TOS/scope/type 等）
   - 匹配成功 → Done
   - 匹配失败 → 进入前缀匹配模式

3. 前缀长度从 32 开始递减
   - 在每个 tnode 内，对子数组索引逐步截零最低有效 "1" 位
   - 直到找到匹配或索引全为零

4. 若当前 tnode 无匹配，回溯到父节点（backtrack++）
   继续截短键，查找更短的匹配前缀

5. 最终找到最长匹配的叶节点，返回 fib_result
   或返回错误（无路由）

🔬 第一性原理

LPM 存在的原因是路由的特异性原则：更具体的路由应优先于通用路由，这样管理员才能精确控制特定网段的流量路径，同时保留"兜底"的默认路由。从数学角度看，IP 路由是一个前缀包含关系的偏序集上的最大元查找问题——LC-Trie 正是为高效解决这一问题而设计的。

📥 接收路径路由查找

当网卡接收到一个 IP 数据包，内核需要决定：这个包是给我的，还是需要转发？ 路由查找在 ip_rcv_finish_core() 中触发，经过一系列函数调用到达 fib_table_lookup()。

/* ip_route_input_slow() 核心逻辑（简化，基于 v5.10 源码） */
static int ip_route_input_slow(struct sk_buff *skb, __be32 daddr,
                               __be32 saddr, u8 tos,
                               struct net_device *dev,
                               struct fib_result *res)
{
    struct in_device *in_dev = __in_dev_get_rcu(dev);
    struct flowi4 fl4 = {
        .flowi4_iif  = dev->ifindex,
        .daddr       = daddr,
        .saddr       = saddr,
        .flowi4_tos  = tos,
        .flowi4_mark = skb->mark,
    };

    /* Step 1: 特殊地址检查（回环/多播/广播/"火星地址"）*/
    if (ipv4_is_multicast(daddr))  goto mc_input;
    if (ipv4_is_lbcast(daddr))     goto brd_input;

    /* Step 2: FIB 查找 */
    err = fib_lookup(net, &fl4, res, 0);
    if (err != 0) goto no_route;

    /* Step 3: 根据路由类型决定处理方式 */
    if (res->type == RTN_LOCAL) {
        /* 目的地是本机 */
        err = fib_validate_source(skb, saddr, daddr, ...);
        rth = rt_dst_alloc(l_dev, flags | RTCF_LOCAL, res->type, ...);
        rth->dst.input = ip_local_deliver; /* 设置本机投递函数 */
        goto out;
    }

    if (res->type == RTN_UNICAST || res->type == RTN_MULTICAST) {
        /* 需要转发 */
        err = ip_mkroute_input(skb, res, in_dev, daddr, saddr, tos, flkeys);
        /* 在 ip_mkroute_input 中：*/
        /* rth->dst.input  = ip_forward; */
        /* rth->dst.output = ip_output;  */
    }

    /* Step 4: 将路由决策附加到 skb */
    skb_dst_set(skb, &rth->dst); /* skb->_skb_refdst = &rth->dst */
}
🔬 第一性原理

接收路径路由查找的本质是"这个包应该交给谁"这一问题的算法化。内核设计的关键洞察是：路由查找结果通过函数指针（dst->input）"编译"进了数据包，后续处理无需任何分支判断——只需调用 dst_input(skb) 即可。这是将控制流决策转化为数据驱动调用的典型设计。

📤 发送路径路由查找

本地发送的数据包（如 TCP/UDP socket 发送的数据）同样需要路由查找，以确定输出接口和下一跳。发送路径的路由查找在 ip_route_output_key_hash() 中完成。

/* TCP 发送路径（已建立连接，net/ipv4/ip_output.c）*/
int __ip_queue_xmit(struct sock *sk, struct sk_buff *skb, struct flowi *fl)
{
    struct rtable *rt;

    /* Step 1: 检查 socket 路由缓存（避免每包查找）*/
    rt = (struct rtable *)__sk_dst_check(sk, 0);

    if (!rt) {
        /* Step 2: 缓存未命中，执行完整路由查找 */
        struct flowi4 *fl4 = &fl->u.ip4;
        rt = ip_route_output_ports(net, fl4, sk,
                                   daddr, inet->inet_saddr, ...);
        if (IS_ERR(rt))
            goto no_route;

        /* Step 3: 将结果存入 socket 缓存 */
        sk_setup_caps(sk, &rt->dst);
        /* → sk->sk_dst_cache = &rt->dst */
    }

    /* Step 4: 将路由决策附加到 skb */
    skb_dst_set_noref(skb, &rt->dst);

    /* Step 5: 通过 Netfilter OUTPUT 钩子后调用 dst_output() */
    err = ip_local_out(net, sk, skb);
    /* ip_local_out → __ip_local_out → NF_HOOK(OUTPUT) → dst_output(skb) */
    /* dst_output 调用 rt->dst.output(net, sk, skb) */
    /* 通常是 ip_output() → ip_finish_output() → 邻居子系统 */
}

/* 核心查找函数（net/ipv4/route.c）*/
struct rtable *ip_route_output_key_hash(struct net *net,
                                          struct flowi4 *fl4,
                                          const struct sk_buff *skb)
{
    struct fib_result res = { .type = RTN_UNSPEC, .fi = NULL };
    struct rtable *rth;

    fl4->flowi4_iif = LOOPBACK_IFINDEX; /* 发送包 iif=loopback */

    rcu_read_lock();
    rth = ip_route_output_key_hash_rcu(net, fl4, &res, skb);
    rcu_read_unlock();
    return rth;
}
🔬 第一性原理

发送路径使用 socket 级路由缓存（sk_dst_cache），体现了"局部性原理"的工程应用：同一个 TCP 连接的所有数据包目的地相同，路由结果也相同，因此只需查找一次。这将路由查找的摊销开销降低到接近零。与接收路径不同，发送路径的路由决策在传输层就确定，这是因为源 IP 地址的选择（取决于路由）必须在构建 IP 头之前完成。

⚖ 策略路由 RPDB

策略路由（Policy-Based Routing，PBR）通过路由策略数据库（RPDB，Routing Policy Database）实现，允许基于源地址、TOS、防火墙标记等多维度属性来选择不同的路由表。

🔬 第一性原理

策略路由的根本出发点是：现实网络中"目标地址"不足以唯一确定最优路径。当同一目标需要根据流量来源、服务类型或安全策略走不同路径时，单一路由表成为瓶颈。RPDB 用"规则优先级链"替代"单一查找表"，将路由决策从函数升格为策略，实现了任意维度的流量分流——这是对"路由=查找函数"这一基本模型的自然泛化。

⚡ ECMP 等价多路径路由

ECMP（Equal Cost Multipath）允许将流量分散到多条等价路径上，实现负载均衡。Linux 5.10 通过 fib_info.fib_nhs > 1 和 fib_nh.nh_weight 实现。

/* 内核选择下一跳（net/ipv4/fib_trie.c）*/
static struct fib_nh_common *fib_select_multipath(
    const struct fib_result *res,
    int hash)
{
    struct fib_info *fi = res->fi;
    int total = 0;

    /* 根据哈希值和权重选择下一跳 */
    for_each_fib_nh(fi, nhsel, nh) {
        if (!fib_nh_is_v6(nh) && !(nh->nh_flags & RTNH_F_DEAD)) {
            total += nh->nh_weight;
            if (total > hash)
                return &nh->nh_common;
        }
    }
    return &fi->fib_nh->nh_common; /* fallback */
}

/* 哈希计算（L4 模式下基于 5 元组）*/
static int fib_multipath_hash(const struct net *net,
                              const struct flowi4 *fl4,
                              const struct sk_buff *skb)
{
    switch (net->ipv4.sysctl_fib_multipath_hash_policy) {
    case 0:
        return jhash_2words(fl4->daddr, fl4->saddr, 0);
    case 1:
        /* 5 元组哈希 */
        return fib_multipath_hash_5tuple(...);
    }
}
图9：ECMP 按权重哈希分流（同一连接始终选相同路径）
🔬 第一性原理

ECMP 的本质是负载与状态的权衡：纯粹的轮询（round-robin）最大化了负载均衡，但破坏了流的连续性（同一 TCP 连接可能走不同路径，导致乱序）。L4 哈希策略在两者间找到了最优平衡点——相同 5 元组的包始终映射到同一路径（保证流的有序性），而不同流则被分散到多条路径（实现负载均衡）。

💾 路由缓存机制

Linux 3.6 后移除了全局路由缓存（route cache），改为下一跳缓存（nexthop cache）和 socket 缓存两级缓存架构，避免了旧版缓存在 DDoS 攻击下的哈希碰撞问题。

C — net/ipv4/route.c（缓存有效性检查） 
/* 下一跳缓存有效性检查 */
static inline bool rt_cache_valid(const struct rtable *rt)
{
    return rt &&
           rt->dst.obsolete == DST_OBSOLETE_FORCE_CHK &&
           !dst_check(&rt->dst, 0); /* genid 比对 */
}

/* 路由表变更时，全局 genid 递增，使所有缓存失效 */
/* rt_genid_bump_ipv4(net) 在路由增删时调用 */
/* 每个 rtable 存储创建时的 genid，不匹配则视为过期 */

/* PMTU 异常缓存（fnhe）的查找与绑定 */
static struct fib_nh_exception *find_exception(
    struct fib_nh_common *nhc, __be32 daddr)
{
    struct fnhe_hash_bucket *hash = rcu_dereference(nhc->nhc_exceptions);
    u32 hval = fnhe_hashfun(daddr);   /* 对目标地址哈希 */

    for (fnhe = rcu_dereference(hash[hval].chain); fnhe; ...) {
        if (fnhe->fnhe_daddr == daddr) {
            if (fnhe->fnhe_expires &&
                time_after(jiffies, fnhe->fnhe_expires)) {
                ip_del_fnhe(nhc, daddr); /* 过期删除 */
                break;
            }
            return fnhe; /* 返回 PMTU 异常 */
        }
    }
    return NULL;
}
🔬 第一性原理

缓存的存在是为了避免重复计算相同问题的答案。Linux 3.6 移除全局路由缓存，是因为第一性原理分析后发现：全局缓存在大量不同目标 IP（如 DDoS）时会因哈希碰撞退化为 O(n) 查找，性能比直接 LC-Trie 查找更差。新的 Per-CPU nexthop 缓存将缓存粒度从"每个(src,dst)对"提升到"每个nexthop"，适应了现代高速路由的实际访问模式。

🤝 邻居子系统（ARP）

路由查找给出下一跳 IP 地址，但发送数据包还需要知道下一跳的 MAC 地址。邻居子系统（net/core/neighbour.c）负责 IP→MAC 的解析，IPv4 使用 ARP 协议。

邻居子系统的存在源于网络分层的基本约束：IP 层只知道逻辑地址，链路层需要物理地址。ARP 是解决这一地址空间映射问题的最简协议。NUD 状态机是对"如何高效维护 IP→MAC 映射"这一问题的工程化回答——它通过状态机管理 ARP 条目的生命周期，在"避免过度 ARP 请求"和"及时感知邻居不可达"之间取得最优平衡。

static int inet_rtm_newroute(struct sk_buff *skb, struct nlmsghdr *nlh, ...)
{

    struct fib_table *tb;

/* 内部调用 fib_magic() 向 LOCAL 和 MAIN 表添加对应路由 */
static void fib_magic(int cmd, int type, __be32 dst, int dst_len,
                      struct in_ifaddr *ifa, u32 rt_priority)
{
    struct fib_config cfg = {
        .fc_protocol = RTPROT_KERNEL,  /* 来源：内核自动 */
        .fc_type     = type,           /* RTN_LOCAL 或 RTN_UNICAST */
        .fc_dst      = dst,
        .fc_dst_len  = dst_len,
        .fc_prefsrc  = ifa->ifa_local,
        .fc_oif      = ifa->ifa_dev->dev->ifindex,
    };
    /* 示例：ip addr add 192.168.1.100/24 dev eth0 后，内核自动添加：
     * LOCAL 表: 192.168.1.100/32 type local
     * LOCAL 表: 192.168.1.255/32 type broadcast
     * MAIN  表: 192.168.1.0/24  dev eth0 type link */
    if (cmd == RTM_NEWROUTE)
        fib_table_insert(net, tb, &cfg, NULL);
    else
        fib_table_delete(net, tb, &cfg, NULL);
}
🔬 第一性原理

Netlink 接口的设计体现了异步事件驱动与消息传递的系统设计原则。与 IOCTL 的同步阻塞模式相比，Netlink 允许内核主动通知用户空间路由变化（路由监控），实现了真正的双向通信。fib_magic() 的存在则体现了内核的一致性自动化原则：用户只需管理 IP 地址，内核自动维护对应的路由条目，避免了用户空间管理的不一致风险。

🔬 第一性原理总结

回归本质，Linux 5.10 路由子系统的所有设计决策都可以从少数几个基本问题推导：

原理1：路由的本质是函数映射

f(packet) → nexthop。所有复杂性（LC-Trie、RPDB、多表）都是为了让这个函数在各种条件下（大路由表、策略规则、ECMP）仍然高效可计算。

原理2：查找代价决定数据结构

路由查找在数据路径上每包发生，毫秒级的代价都是不可接受的。LC-Trie 的 O(W/bits) 查找时间（接近O(1)）是满足高速网络吞吐量的必然选择。路由表大小不影响查找时间——这是硬性约束，不是优化选项。

原理3：控制面与数据面分离

路由表的增删改（控制面）通过 Netlink 异步进行，可以接受相对高的延迟；而路由查找（数据面）通过 RCU 无锁读取，必须是零等待的。两者之间通过 RCU + genid 机制解耦，互不阻塞。

原理4：决策与执行解耦（策略模式）

路由查找的结果不是"下一步 if-else"，而是函数指针（dst.input/output）。决策（查找）只做一次，执行（转发/投递/丢弃）由函数指针驱动，消除了数据路径上的分支预测开销。

原理5：层次化抽象实现最大复用

dst_entry 作为所有协议路由决策的基类，rtable（IPv4）和 rt6_info（IPv6）继承它。邻居子系统统一处理 ARP/NDISC，与路由子系统解耦。这种分层使每层只处理本层关心的问题，同时通过接口边界进行组合。

原理6：缓存基于局部性，失效基于版本号

路由缓存（socket cache、nexthop cache）利用时间局部性（同一连接重复发包）减少查找次数。genid 版本号机制保证路由表变更时缓存立即失效，实现了性能与正确性的最优平衡。
