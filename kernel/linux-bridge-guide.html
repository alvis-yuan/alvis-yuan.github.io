什么是网桥（Bridge）

网桥是连接两个或多个网络段的设备，工作在 OSI 模型第二层（数据链路层）， 依据目标 MAC 地址对帧进行过滤和转发。Linux 内核原生实现了软件网桥功能。

📖 官方定义
IEEE 802.1Q-2022 标准定义：网桥（Bridge）是连接两个或多个网络段、 运行在 OSI 数据链路层（L2）的设备，其目的是基于目标 MAC 地址对帧进行过滤和转发。

📄 kernel.org/networking/bridge.html

现实中的类比

可以把网桥想象成一台"智能"交换机：它记住了每个端口背后有哪些 MAC 地址（学习），并根据目标 MAC 把帧只发送到正确的端口（转发），而不是像集线器那样广播给所有人。

Linux Bridge 的四大核心组件

🔌 端口集合（Network Ports）
用于在网络段之间转发流量。每个加入网桥的物理或虚拟网络接口都成为一个"端口"。
🎛️ 控制平面（Control Plane）
运行 STP（生成树协议），计算最小生成树，防止环路导致广播风暴。
✈️ 转发平面（Forwarding Plane）
处理来自端口的入站帧，根据 MAC 学习数据库做出转发决策。
📚 MAC 学习数据库（FDB）
记录局域网中主机的位置信息（MAC → Port 映射），是转发决策的核心依据。

三种配置接口

接口用途示例工具
ioctl创建/删除网桥，添加/删除接口brctl, iproute2
sysfs管理网桥和端口参数（已弃用）/sys/class/net/br0/
Netlink现代异步通信接口（推荐）ip link, bridge命令
🔬 第一性原理
网桥的本质：将多块物理网卡虚拟成一台"软件交换机"。其核心洞察是： 帧转发只需要 L2 头部（MAC 地址），与上层协议（IP/IPv6/IPX）无关。 这种协议无关性使得网桥能透明地连接不同 L3 协议的网络段， 而不需要了解它们的内容。

L2 转发的核心逻辑

图2-3：L2 转发决策流程图 —— 包含 MAC 学习、FDB 查找与泛洪逻辑
🔬 第一性原理
L2 转发的本质是地址学习 + 最短路径。网桥工作在 L2 的原因： 以太网帧中的 MAC 地址是唯一标识符，帧长度固定、处理快速。 与 L3 路由不同，L2 转发不需要 TTL 递减、不需要分片、不需要校验和重算， 因此延迟极低。代价是 MAC 地址不具备层次性，无法像 IP 那样聚合路由， 所以网桥不适合大规模网络（广播域问题）。

net_bridge — 网桥主结构体

struct net_bridge 是内核中表示一个网桥实例的核心数据结构， 定义于 net/bridge/br_private.h。它聚合了网桥的所有状态信息。

📄 net/bridge/br_private.h · GitHub

核心字段解析

/* net/bridge/br_private.h */
struct net_bridge {
    /* === 基础设施 === */
    spinlock_t              lock;           // 保护 bridge 状态的自旋锁
    struct list_head        port_list;      // 所有端口链表头
    struct net_device      *dev;           // 对应的虚拟网络设备
    
    /* === 转发数据库 (FDB) === */
    struct rhashtable       fdb_hash_tbl;   // FDB 哈希表（RCU 保护）
    struct list_head        fdb_list;       // FDB 有序链表
    
    /* === STP 生成树协议 === */
    bridge_id               designated_root; // 根桥 ID
    bridge_id               bridge_id;       // 本桥 ID（优先级+MAC）
    u32                     root_path_cost;  // 到根桥路径代价
    unsigned char           topology_change; // 拓扑变更标志
    struct timer_list       hello_timer;     // Hello 消息定时器
    struct timer_list       tcn_timer;       // TCN 定时器
    struct timer_list       topology_change_timer;
    struct delayed_work     gc_work;         // FDB 老化工作队列
    u8                      stp_enabled;     // STP 开关
    
    /* === VLAN 过滤 === */
    struct net_bridge_vlan_group __rcu *vlgrp; // VLAN 组
    u16                     default_pvid;    // 默认 PVID
    
    /* === 组播/IGMP Snooping === */
    struct net_bridge_mcast  multicast_ctx;  // 组播上下文
    
    /* === Netfilter 调用选项 === */
    unsigned long           options;         // BROPT_* 标志位
    
    /* === 统计信息 === */
    struct pcpu_sw_netstats __percpu *stats;
};

单一数据结构承载全部状态：struct net_bridge 将网络设备、 端口管理、FDB、STP、VLAN、组播等所有子系统都组合在一起。 这是 Linux 内核"一切皆结构体"设计哲学的体现——通过嵌入 struct net_device 指针， 网桥虚拟设备能无缝融入内核现有的设备管理框架。 RCU（读-拷贝-更新）锁机制保证了高并发下无锁读性能。

net_bridge_port — 端口结构体

每个加入网桥的网络接口都由 struct net_bridge_port 表示， 它存储了端口的 STP 状态、STP 计时器、VLAN 信息及对父网桥的引用。

📄 net_bridge_port 结构体参考 
/* net/bridge/br_private.h (简化版) */
struct net_bridge_port {
    /* === 关联关系 === */
    struct net_bridge      *br;        // 指向父网桥
    struct net_device      *dev;       // 对应的物理/虚拟网络接口
    struct list_head        list;      // 挂入 br->port_list
    
    /* === STP 端口状态 === */
    u8                      priority;   // 端口优先级 (0-255)
    u8                      state;      // BR_STATE_DISABLED/LISTENING/
                                        // LEARNING/FORWARDING/BLOCKING
    u16                     port_no;    // 端口编号
    
    /* === STP 计算字段 === */
    bridge_id               designated_root;   // 该端口看到的根桥
    bridge_id               designated_bridge; // 指定桥 ID
    port_id                 designated_port;   // 指定端口 ID
    u32                     path_cost;         // 路径代价
    u32                     designated_cost;   // 指定代价
    
    /* === STP 定时器 === */
    struct timer_list       forward_delay_timer; // 转发延迟定时器
    struct timer_list       hold_timer;          // 发送 BPDU 限速
    struct timer_list       message_age_timer;   // BPDU 老化定时器
    
    /* === VLAN === */
    struct net_bridge_vlan_group __rcu *vlgrp;   // 端口 VLAN 组
    
    /* === 端口标志 === */
    unsigned long           flags;      // BR_LEARNING, BR_FLOOD, ...
    u32                     group_fwd_mask; // 组播转发掩码
    
    struct kobject          kobj;       // sysfs 支持
    struct rcu_head         rcu;        // RCU 延迟释放
};

端口标志位（flags）

标志含义默认
BR_LEARNING允许从入站流量学习源 MAC开启
BR_FLOOD允许对未知单播泛洪开启

🔬 第一性原理
端口结构体是网桥与物理世界的接口：net_bridge_port 将一个真实的 net_device（网卡）"嫁接"到网桥逻辑中，同时维护 STP 状态机所需的全部信息。 这种分离设计（网桥逻辑 vs. 设备驱动）使得任何以太网类设备都可以加入网桥， 体现了内核的面向接口编程哲学。
📚 核心数据结构 · 第⑤章

FDB — 转发数据库

FDB（Forwarding DataBase）是网桥的"记忆"，记录每个 MAC 地址从哪个端口可达。 内核通过哈希表实现快速查找，并通过老化机制清除过期条目。

📄 net/bridge/br_private.h 📄 net/bridge/br_fdb.c

FDB 条目结构体

/* net/bridge/br_private.h */
struct net_bridge_fdb_key {
    mac_addr    addr;       // MAC 地址（6字节）
    u16         vlan_id;    // VLAN ID（支持 VLAN 感知桥）
};

struct net_bridge_fdb_entry {
    struct rhash_head       rhnode;  // rhashtable 成员
    struct net_bridge_port *dst;     // 目标端口（NULL 表示本机）
    
    struct net_bridge_fdb_key key;   // MAC + VLAN 查找键
    
    unsigned long           updated; // 最后更新时间（用于老化）
    unsigned long           used;    // 最后使用时间
    
    /* 条目类型标志 */
    unsigned char           is_local:1;  // 本地地址（桥自身端口 MAC）
    unsigned char           is_static:1; // 静态条目（不老化）
    unsigned char           added_by_user:1;  // 用户手动添加
    unsigned char           offloaded:1;      // 已卸载到硬件
    
    struct rcu_head         rcu;     // RCU 延迟释放
    struct list_head        list;    // 有序链表（老化遍历）
};

关键 FDB 函数

函数作用调用时机

🔬 第一性原理
FDB 是网桥智能的核心：没有 FDB，网桥就退化为集线器（全端口广播）。 FDB 的设计取舍是空间换时间：用内存存储 MAC→Port 映射， 换取 O(1) 的转发查找。老化机制解决了"MAC 漂移"问题（同一 MAC 出现在不同端口）， 保证 FDB 与网络实际拓扑保持一致。 内核使用 rhashtable（可自动调整大小的并发安全哈希表）而非简单哈希链表， 是在规模性与性能间的精心权衡。

VLAN — net_bridge_vlan 结构体

Linux Bridge 支持 IEEE 802.1Q 和 802.1AD（QinQ）VLAN 标记， 通过 struct net_bridge_vlan 实现每 VLAN 的独立状态管理。 该结构体在官方内核文档中有明确定义。

📄 官方文档 kernel.org/networking/bridge.html 
/* 来自官方文档 docs.kernel.org/networking/bridge.html */
struct net_bridge_vlan {
    struct rhash_head   vnode;       // 全局 rhashtable 成员
    struct rhash_head   tnode;       // tunnel rhashtable 成员
    u16                 vid;         // VLAN ID
    u16                 flags;       // 桥 VLAN 标志
    u16                 priv_flags;  // 内核私有标志
    u8                  state;       // STP 状态
    struct pcpu_sw_netstats __percpu *stats; // per-CPU 统计
    
    union {
        struct net_bridge      *br;   // MASTER flag: 指向网桥
        struct net_bridge_port *port; // 否则: 指向端口
    };
    union {
        refcount_t          refcnt;   // MASTER: 引用计数
        struct net_bridge_vlan *brvlan; // 端口: 指向全局 VLAN
    };
    
    struct br_tunnel_info   tinfo;   // tunnel 信息
    union {
        struct net_bridge_mcast      br_mcast_ctx;   // 桥级组播上下文
        struct net_bridge_mcast_port port_mcast_ctx; // 端口级组播上下文
    };
    u16                 msti;        // MST 实例（MSTP）
    struct list_head    vlist;       // 有序 VLAN 链表
    struct rcu_head     rcu;         // RCU 延迟释放
};
⚠️ 注意
内核 kAPI 是不稳定的，可能随时更改（unstable）。此结构体定义来自官方文档， 实际字段可能随内核版本变化。

VLAN 过滤工作流

步骤说明
1. 启用过滤设置 BROPT_VLAN_ENABLED，默认 PVID=1

🔬 第一性原理
VLAN 是广播域隔离的最小代价方案：通过在帧中插入 4 字节标签（802.1Q）， 一个物理交换机（或 Linux Bridge）可以虚拟成多个独立交换机。 net_bridge_vlan 的 Union 设计使同一结构体能同时表示全局 VLAN 和端口本地 VLAN，通过 MASTER 标志区分，避免了代码重复。 这是 Linux 内核中"多态性"的经典实现模式。
🚀 初始化流程 · 第⑦章

模块初始化 — br_init()

Linux Bridge 作为内核模块加载时，br_init() 是第一个被调用的函数， 它完成所有子系统的注册与初始化。

📄 net/bridge/br.c 
/* net/bridge/br.c */
static int __init br_init(void)
{
    int err;
    
    /* 1. 注册 STP 协议处理器，接收 BPDU 帧 */
    err = stp_proto_register(&br_stp_proto);
    
    /* 2. 初始化 FDB 缓存（kmem_cache）*/
    err = br_fdb_init();
    
    /* 3. 注册网络命名空间子系统 */
    err = register_pernet_subsys(&br_net_ops);
    
    /* 4. 初始化 netfilter 核心 */
    err = br_nf_core_init();
    
    /* 5. 注册网络设备通知链 */
    err = register_netdevice_notifier(&br_device_notifier);
    
    /* 6. 注册 FDB 通知链 */
    err = register_fib_notifier(&init_net, &br_fib_notifier_ops, ...);
    
    /* 7. 初始化 Netlink 接口 */
    err = br_netlink_init();
    
    /* 8. 设置 ioctl 处理钩子 */
    brioctl_set(br_ioctl_deviceless_stub);
    
    return 0;
}
module_init(br_init);
MODULE_ALIAS_RTNL_LINK("bridge");

🔬 第一性原理
模块化初始化的本质是依赖注入：br_init() 通过向内核各子系统 "注册钩子"来集成 Bridge 功能，而不是修改内核核心代码。 这体现了 Linux 内核的微内核精神——功能通过钩子点（hook points） 和通知链（notifier chain）插入，内核主路径代码保持精简不变， 可加载模块只在必要时介入。

用户执行 ip link add br0 type bridge 或 brctl addbr br0 时， 内核通过 ioctl 调用 br_add_bridge()，创建虚拟网络设备和 net_bridge 实例。

br_dev_setup() 核心内容

/* net/bridge/br_device.c */
void br_dev_setup(struct net_device *dev)
{
    struct net_bridge *br = netdev_priv(dev);
    
    /* 设置设备操作函数指针 */
    dev->netdev_ops = &br_netdev_ops;  // 包含 br_dev_xmit
    dev->ethtool_ops = &br_ethtool_ops;
    
    /* 设置以太网设备属性 */
    ether_setup(dev);
    dev->priv_flags |= IFF_EBRIDGE;    // 标识这是网桥设备
    
    /* 初始化 net_bridge 私有数据 */
    br->dev = dev;
    spin_lock_init(&br->lock);
    INIT_LIST_HEAD(&br->port_list);
    INIT_HLIST_HEAD(&br->fdb_list);
    
    br->bridge_id.prio[0] = 0x80;     // 默认优先级 32768
    br->bridge_id.prio[1] = 0x00;
    
    /* 初始化 FDB 老化定时器 */
    INIT_DELAYED_WORK(&br->gc_work, br_fdb_cleanup);
}

✅ IFF_EBRIDGE 标志的意义
内核通过检查 dev->priv_flags & IFF_EBRIDGE 来区分网桥设备与普通网卡。 这个标志告诉内核：该设备有自己的 L2 转发逻辑，不应该像普通设备那样处理帧。
🔬 第一性原理
一切皆 net_device：Linux 内核将网桥也实现为一个 net_device， 这意味着网桥可以有 IP 地址（作为管理接口），可以被路由器路由到， 可以运行 DHCP 等。这种"万物皆设备"的统一抽象大大降低了代码复杂度—— 网桥设备无需特殊处理，就能自然融入内核的设备管理框架。

添加端口 — br_add_if()

将物理或虚拟网卡加入网桥是实现 L2 转发的关键步骤。 br_add_if() 不仅创建 net_bridge_port， 还通过注册 rx_handler 将网卡的收包路径"劫持"到网桥处理逻辑。

📄 net/bridge/br_if.c · GitHub 📄 Linux Bridge 分析 (goyalankit.com) 
/* net/bridge/br_if.c (简化版) */
int br_add_if(struct net_bridge *br, struct net_device *dev)
{
    struct net_bridge_port *p;
    int err;
    
    /* 验证：只允许以太网类设备 */
    if (dev->type != ARPHRD_ETHER || dev->flags & IFF_LOOPBACK)
        return -EINVAL;
    /* 不能把网桥加入另一个网桥 */
    if (dev->priv_flags & IFF_EBRIDGE)
        return -ELOOP;

    /* 创建 net_bridge_port 对象 */
    p = new_nbp(br, dev);
    
    /* 通知其他内核组件有设备加入 */
    call_netdevice_notifiers(NETDEV_JOIN, dev);
    
    /* 设置混杂模式：接收所有帧 */
    err = dev_set_promiscuity(dev, 1);
    
    /* 将端口加入 sysfs */
    err = kobject_init_and_add(&p->kobj, &brport_ktype,
                               &(dev->dev.kobj), "brport");
    
    /* 【关键】注册 rx_handler，截获该网卡所有入站帧！*/
    err = netdev_rx_handler_register(dev, br_handle_frame, p);
    
    /* 将端口 MAC 加入 FDB（is_local=1） */
    if (br_fdb_insert(br, p, dev->dev_addr, 0))
        dev_err(...);
    
    /* 将端口添加到网桥端口链表 */
    list_add_rcu(&p->list, &br->port_list);
    
    return 0;
}
图9-1：br_add_if() 前后对比 —— rx_handler 注册是截获帧的关键
💡 rx_handler 机制
netdev_rx_handler_register(dev, br_handle_frame, p) 将 br_handle_frame 注册为 eth0 的接收处理函数。此后，每当 eth0 收到帧， 内核的 __netif_receive_skb() 会首先调用这个 handler， 将帧"拦截"进入网桥逻辑，而不是直接送往协议栈。
🔬 第一性原理
rx_handler 是"钩子模式"的典型应用：网桥不修改网卡驱动代码， 而是通过注册回调函数（rx_handler）"插入"收包流程。混杂模式是必要的—— 因为网桥需要接收目标 MAC 不是自己的帧（转发给其他端口）， 这突破了普通网卡只接收"自己的包"的限制。

br_handle_frame() 核心逻辑

/* net/bridge/br_input.c — 简化 */
rx_handler_result_t br_handle_frame(struct sk_buff **pskb)
{
    struct net_bridge_port *p;
    const unsigned char *dest = eth_hdr(skb)->h_dest;

    /* 丢弃回环包 */
    if (unlikely(skb->pkt_type == PACKET_LOOPBACK))
        return RX_HANDLER_PASS;

    /* 检查源 MAC 是否为广播（错误帧） */
    if (!is_valid_ether_addr(eth_hdr(skb)->h_source))
        goto drop;

    /* 获取端口，检查 STP 状态 */
    p = br_port_get_rcu(skb->dev);
    if (!(br->dev->flags & IFF_UP) || p->state == BR_STATE_DISABLED)
        goto drop;

    /* 处理链路本地地址 (01:80:C2:00:00:XX)
     * 包含 STP BPDUs, 802.1X PAE, LLDP 等 */
    if (unlikely(is_link_local_ether_addr(dest))) {
        // STP/控制平面处理
        return br_handle_local_finish(skb);
    }

    /* 处于 LEARNING 或 FORWARDING 状态时，进入 pre_routing hook */
    switch (p->state) {
    case BR_STATE_FORWARDING:
    case BR_STATE_LEARNING:
        return nf_hook_bridge_pre(skb, pskb);
        // 最终调用 br_handle_frame_finish()
    default:
        kfree_skb(skb);  // BLOCKING 等状态丢弃数据帧
    }
}
🔬 第一性原理
收包路径是"竞争性截获"：__netif_receive_skb() 按优先级调用 各注册处理器，rx_handler 优先级最高。一旦返回 RX_HANDLER_CONSUMED， 帧就被网桥消费，不再传递给其他处理器（如 IP 协议栈）。 这是 Linux 内核"快速路径"设计的体现——最先匹配的处理器赢得控制权。
🗺️ 收发流程 · 第⑪章

转发决策 — br_handle_frame_finish()

br_handle_frame_finish() 是网桥的"大脑"， 它完成 MAC 学习、FDB 查找，并根据目标 MAC 类型做出转发决策。

📄 br_input.c · codebrowser 
/* net/bridge/br_input.c — 简化展示核心逻辑 */
int br_handle_frame_finish(struct net *net, struct sock *sk,
                             struct sk_buff *skb)
{
    struct net_bridge_port *p = br_port_get_rcu(skb->dev);
    const unsigned char *dest = eth_hdr(skb)->h_dest;
    struct net_bridge_fdb_entry *dst = NULL;
    struct net_bridge *br;
    u16 vid = 0;

    /* 1. VLAN 检查 */
    if (!br_allowed_ingress(p->br, nbp_vlan_group_rcu(p), skb, &vid, ...))
        goto out;

    /* 2. MAC 学习 —— 从源 MAC 学习端口映射 */
    br = p->br;
    br_fdb_update(br, p, eth_hdr(skb)->h_source, vid, 0);

    /* 3. LEARNING 状态：只学习，不转发 */
    if (p->state == BR_STATE_LEARNING)
        goto drop;

    /* 4. 判断目标 MAC 类型 */
    if (is_broadcast_ether_addr(dest)) {
        /* 广播：泛洪所有端口 + 送本机 */
        skb2 = skb;
    } else if (is_multicast_ether_addr(dest)) {
        /* 组播：查 MDB 或泛洪 */
        mdst = br_mdb_get(brmctx, skb, vid);
        ...
    } else {
        /* 单播：查 FDB */
        dst = br_fdb_find_rcu(br, dest, vid);
        if (dst) {
            if (dst->is_local) {
                skb2 = skb;  // 目标是本机
                skb = NULL;
            }
        }
        /* FDB 未命中：泛洪 */
    }

    /* 5. 转发 */
    if (skb) {
        if (dst)
            br_forward(dst->dst, skb, skb2);     // 单播转发
        else
            br_flood(br, skb, pkt_type, skb2);  // 泛洪
    }

    /* 6. 递交给本机协议栈 */
    if (skb2)
        return br_pass_frame_up(skb2);
    return 0;
}
图11-1：br_handle_frame_finish() 详细决策流程图

三种目标 MAC 处理路径

目标 MAC 类型判断条件处理函数
广播帧FF:FF:FF:FF:FF:FFbr_flood() + br_pass_frame_up()
组播帧最高位 bit 为 1查 MDB，命中则定向，否则泛洪
单播帧（已知）FDB 命中br_forward(dst->dst, skb)
单播帧（未知）FDB 未命中br_flood() 除源端口外所有端口
本地帧is_local=1br_pass_frame_up() 交给协议栈
🔬 第一性原理
转发决策的本质是信息不对称的处理：当网桥"知道"目标在哪里（FDB 命中）， 就精准转发（unicast）；当它"不知道"时，只能广播（flood）让所有端口自己判断。 这与人类社会的信息传播规律完全一致：信息越完整，通信越精准； 信息缺失时，只能广播浪费带宽。FDB 的存在将帧从"广播"向"单播"演进， 这是网桥智能化的唯一路径。

发送流程 — TX Path

网桥的发送路径有两种触发方式： 本机应用程序向网桥 IP 地址发送数据（br_dev_xmit）， 以及网桥转发其他端口收到的帧（br_forward/__br_forward）。

发送路径 1：本机发送（br_dev_xmit）

/* net/bridge/br_device.c */
netdev_tx_t br_dev_xmit(struct sk_buff *skb, struct net_device *dev)
{
    struct net_bridge *br = netdev_priv(dev);
    const unsigned char *dest = skb->data;  // 目标 MAC
    struct net_bridge_fdb_entry *dst;
    u16 vid = 0;

    br_multicast_count(br, NULL, skb, ...);

    if (is_broadcast_ether_addr(dest)) {
        br_flood(br, skb, BR_PKT_BROADCAST, false, true);
    } else if (is_multicast_ether_addr(dest)) {
        if (br_multicast_rcv(...))
            goto out;
        br_multicast_flood(mdst, skb, ...);
    } else {
        /* 单播：查 FDB */
        dst = br_fdb_find_rcu(br, dest, vid);
        if (dst)
            br_forward(dst->dst, skb, false, true);
        else
            br_flood(br, skb, BR_PKT_UNICAST, false, true);
    }
    return NETDEV_TX_OK;
}

发送路径 2：帧转发（__br_forward）

/* net/bridge/br_forward.c */
static void __br_forward(const struct net_bridge_port *to,
                          struct sk_buff *skb, bool local_orig)
{
    struct net_bridge_vlan_group *vg;
    struct net_device *indev;

    /* VLAN 出站处理（添加/删除 tag） */
    skb = br_handle_vlan(to->br, to, vg, skb);
    if (!skb) return;

    indev = skb->dev;
    skb->dev = to->dev;  // 切换出口设备
    skb_forward_csum(skb);

    /* 通过 Netfilter POST_ROUTING 钩子发送 */
    NF_HOOK(NFPROTO_BRIDGE, NF_BR_POST_ROUTING, net, NULL,
            skb, indev, skb->dev, br_forward_finish);
}

static int br_forward_finish(..., struct sk_buff *skb)
{
    return dev_queue_xmit(skb);  // 最终通过网卡发送
}

br_flood() — 泛洪实现

/* net/bridge/br_forward.c — 简化 */
void br_flood(struct net_bridge *br, struct sk_buff *skb, ...)
{
    struct net_bridge_port *p;
    
    /* 遍历所有端口 */
    list_for_each_entry_rcu(p, &br->port_list, list) {
        if (should_deliver(p, skb)) {  // 过滤源端口 + 状态检查
            if (prev) {
                struct sk_buff *skb2 = skb_clone(skb, GFP_ATOMIC);
                deliver_clone(prev, skb2, ...);
            }
            prev = p;
        }
    }
    if (prev)
        __br_forward(prev, skb, ...);  // 最后一个端口直接发送
}
🔬 第一性原理
发送路径体现了"最小代价克隆"原则：泛洪时，内核只在需要时才 skb_clone()。 对于 n 个端口，只需要 n-1 次克隆（最后一个端口直接使用原始 skb）， 节省了一次内存分配。这种优化看似微小，但在高流量场景（广播风暴时每秒数十万帧） 下能显著减少内存压力。Linux 内核中类似的"延迟做"（lazy）优化随处可见。

br_netfilter 的工作原理

/* net/bridge/br_netfilter_hooks.c — 简化 */
/* PRE_ROUTING hook: 拦截桥接的 IP 包 */
static unsigned int br_nf_pre_routing(void *priv,
    struct sk_buff *skb, const struct nf_hook_state *state)
{
    if (!IS_IP(skb) && !is_vlan_ip(skb, state->net))
        return NF_ACCEPT;

    nf_bridge_pull_encap_header_rcsum(skb);
    
    /* 模拟路由设备，调用 IPv4 Netfilter 的 PRE_ROUTING */
    NF_HOOK(NFPROTO_IPV4, NF_INET_PRE_ROUTING, state->net,
            state->sk, skb, skb->dev, NULL,
            br_nf_pre_routing_finish);
            
    return NF_STOLEN;  // 帧被"借走"，不继续桥接处理
}
🔬 第一性原理
Netfilter 是"截面式"安全机制：通过在数据流的固定截面点插入钩子， 而不是在每个函数中嵌入安全逻辑，实现了安全策略与转发逻辑的解耦。 Bridge 拥有独立的 hook 体系（NFPROTO_BRIDGE）是因为 L2 帧和 L3 包 在生命周期中的关键决策点不同。br_netfilter 通过"伪装路由"打通两套 hook 体系， 是一种"胶水层"设计，但代价是复杂性增加，这也是官方不推荐的原因。

🔬 终极第一性原理
Linux Bridge 的设计哲学：整个网桥子系统建立在三个核心原则之上：
  1. 一切皆 net_device：网桥是设备，端口是设备，让所有功能自然复用内核基础设施。
  2. 钩子（Hook）而非修改：通过 rx_handler、Netfilter hook、通知链插入逻辑，主路径代码保持纯净。
  3. RCU 保护高并发：FDB 查找（读）是热路径，RCU 使其无锁；写操作（学习/老化）可以慢一些。
这三条原则使 Linux Bridge 在不修改任何现有网卡驱动的情况下， 将任意以太网设备组合成功能完整的软件交换机，并与 iptables/nftables 无缝集成。
