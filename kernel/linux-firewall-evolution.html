总览：为何防火墙需要不断演进？

防火墙的本质是网络的门卫——它在数据包流经内核网络协议栈的关键节点上设卡拦截，根据规则决定"放行"还是"丢弃"。然而，这个看似简单的任务背后，隐藏着三十年来始终未曾停歇的技术演进张力：

图1：Linux 防火墙五代演进时间线（数据来源：netfilter.org、LWN.net）

正如 Rusty Russell（ipchains 与 netfilter/iptables 之父）在官方 HOWTO 中所写：

"Linux kernels have had packet filtering since the 1.1 series. The first generation, based on ipfw from BSD, was ported by Alan Cox in late 1994."
— netfilter.org 官方 Packet Filtering HOWTO

驱动每次演进的核心矛盾，始终是以下三对张力：

三大演进驱动力
  • 安全模型的提升：从无状态→有状态→可编程的策略控制
  • 性能与可扩展性：从线性规则匹配→哈希/树结构→JIT 编译字节码
  • 可维护性与架构整洁：从内核硬编码协议知识→模块化框架→用户空间编译器
第一性原理
防火墙的本质：在正确位置以正确的代价检查正确的信息

防火墙演进的根本逻辑只有一条：内核能力的边界随硬件与网络复杂度而扩张，旧的抽象层无法跟上，新的抽象层就必须出现。每一代更替都是在问同一个问题——"我们用最小的成本，在最合适的位置，检查了最准确的信息吗？"

当答案是"否"，演进就发生了。这不是工程师的任性，而是物理约束与安全需求共同施加的必然压力。

01
第一代 · 1994 年 · Linux 1.1 ~ 2.0

ipfw / ipfwadm — 从 BSD 移植的初代守门人

历史背景

1994 年，互联网方兴未艾，Linux 仅发布数年，完全没有内建的网络安全机制。彼时 BSD 系统已有简单的 IP 包过滤工具 ipfw，Linux 内核开发者 Alan Cox 在 1994 年底将其移植到 Linux 1.1 系列，这就是 Linux 防火墙的起点。

随后，Jos Vos 等人对其进行增强并发布了配套的用户空间工具 ipfwadm（IP Firewall Administration），用于管理 Linux 2.0 内核中的过滤规则。这一组合一直服务到 1999 年 Linux 2.2.0 发布。

图2：ipfwadm 架构示意——三条独立链，线性规则匹配

解决的核心问题

历史贡献
  • 首次为 Linux 引入数据包过滤能力，实现基于源/目标 IP、端口、协议的基础访问控制
  • 提供 input、forward、output 三条过滤链，覆盖包的三种流向
  • 建立了"规则列表 + 匹配动作"的防火墙基础范式，延续至今
  • 用户空间工具 ipfwadm 提供了可操作的规则管理界面

根本性痛点（驱动下一代演进的力量）

致命缺陷
  • 完全无状态：无法追踪 TCP 连接状态，无法区分"已建立连接的响应包"与"新连接尝试"
  • 无 NAT 支持：无法做网络地址转换，局域网共享上网需额外机制
  • 规则语义模糊：input/forward/output 链的地址语义在路由前后不同，极易配置错误
  • 无模块化架构：所有过滤逻辑硬编码在内核，扩展极其困难
  • IPv6/ARP 不支持：仅针对 IPv4 设计

典型用法示例

# ipfwadm 示例：禁止来自 10.0.0.1 的数据包
ipfwadm -I -a deny -S 10.0.0.1 -W eth0

# 允许 TCP 80 端口
ipfwadm -I -a accept -P tcp -D 0.0.0.0/0 80
第一性原理
无状态过滤的根本局限：网络连接天然是有状态的

TCP/IP 协议本身是有状态的——一个连接由三次握手建立，由四次挥手拆除，中间的每一个包都属于特定的会话上下文。

ipfwadm 把每个包都当成独立的个体处理，就像门卫只认"证件上写着什么"，不认"这个人是不是我刚才放进去的那批人的随行"。这导致管理员要么把规则写得极其繁琐（为每个方向都配置规则），要么被迫放松限制留下安全缺口。

无状态防火墙无法回答"这个包属于哪个连接？"这个最基本的问题，而这个问题恰恰是现代网络安全的核心。

第二代 · 1998 年 · Linux 2.2

ipchains — 链式规则的崛起

历史背景与改进动机

1998 年，随着互联网商业化浪潮席卷全球，Linux 服务器开始大规模部署于企业边界，ipfwadm 的种种局限暴露无遗。 Rusty Russell 与 Michael Neuling 对内核进行了重大重写，引入了 ipchains 并将其合并进 Linux 2.2 内核。

"In mid-1998, for Linux 2.2, I reworked the kernel quite heavily, with the help of Michael Neuling, and introduced the userspace tool 'ipchains'."
— Rusty Russell，netfilter.org 官方 Packet Filtering HOWTO

核心改进

图3：ipchains 架构——支持用户自定义链，但 NAT 与过滤代码仍耦合

ipchains 带来的改进
  • 用户自定义链（User-defined chains）：管理员可创建自己的规则链，规则可"跳转（JUMP）"到自定义链，实现层次化组织
  • 改进的分段包处理：通过 -f 选项可对 IP 分片的后续片段应用规则
  • 更清晰的链语义：INPUT/FORWARD/OUTPUT 三链语义比 ipfwadm 更为明确
  • 日志记录：支持使用 -l 标志记录匹配的包
  • Masquerading（IP 伪装）：初步支持 NAT 的子集——IP 伪装（即 SNAT 的一种形式）

典型用法示例

# 允许已建立连接的响应包（但仍是伪"有状态"，靠 TCP 标志位判断）
ipchains -A input -p tcp ! --syn -j ACCEPT

# 创建自定义链
ipchains -N mychain
ipchains -A input -s 192.168.0.0/24 -j mychain

# IP 伪装（简单 NAT）
ipchains -A forward -s 192.168.1.0/24 -j MASQUERADE

根本性痛点（驱动第三代演进的力量）

ipchains 发布后不久，其架构缺陷就被社区识别。官方 Netfilter Hacking HOWTO 列出了多达 11 条致命问题：

ipchains 的根本缺陷（来源：netfilter.org 官方文档）
  • 仍然无状态：仅靠 TCP SYN 标志位模拟状态判断，可被 ACK 扫描绕过（Nmap ACK scan）
  • 透明代理实现混乱：REDIRECT 目标不处理 UDP 响应，且破坏了 TCP/UDP 端口分配协调
  • 规则必须绑定接口地址：无法不依赖接口 IP 来区分"本地产生"与"转发"的包
  • Masquerade 与过滤代码耦合：NAT 与过滤逻辑混在一起，代码复杂且难以扩展
  • TOS 操作、重定向等附着在过滤代码中：不是模块化的，扩展极困难
  • 无模块化、不可扩展：添加 MAC 地址过滤、选项过滤等需修改内核核心
  • 计数器无法原子读取：并发场景下规则计数器存在竞争条件
  • CONFIG_IP_ALWAYS_DEFRAG 是编译期选项：发行版无法用同一内核支持所有场景
"The netfilter motto is (stolen shamelessly from Cort Dougan): 'So... how is this better than KDE?' (This motto narrowly edged out 'Whip me, beat me, make me use ipchains')."
— Rusty Russell，Netfilter Hacking HOWTO
第一性原理
耦合是架构腐化的根源：功能越多，边界越模糊

ipchains 的最大问题不是它"能做什么"，而是它把太多不相关的事情混在了一个代码层：过滤判断、NAT 地址转换、重定向、TOS 标记……这些逻辑在 ipchains 中彼此嵌套，形成了一个难以推理的整体。

从第一性原理看，任何系统一旦将"策略（Policy）"与"机制（Mechanism）"混合编码，就失去了独立演进的能力。当你想升级 NAT，你必须同时理解过滤；当你想新增过滤条件，你可能破坏 NAT。这就是为什么 Rusty Russell 必须从头设计 Netfilter——不是功能不够，而是架构从根基上就错了。

03
第三代 · 1999~2001 年 · Linux 2.4

Netfilter / iptables — 模块化框架的时代

历史背景与设计动机

1999 年中期，Rusty Russell 开始了对 Linux 内核防火墙子系统的彻底重写。这次不仅仅是修修补补，而是引入了一套全新的框架架构——Netfilter。

"The netfilter project was founded by Paul 'Rusty' Russell to re-design and to heavily improve the previous Linux 2.2.x ipchains and Linux 2.0.x ipfwadm systems."
— netfilter.org 官方介绍

在 1999 年 11 月，Rusty 前往蒙特利尔与 Marc Boucher 整晚讨论，共同设计了"多表框架（multiple tables framework）"，这导致了早期独立 NAT 控制工具 ipnatctl 的消亡，以及 iptable_{filter, nat, mangle} 模块的诞生。Netfilter 最终并入 Linux 2.4 内核（2001 年 1 月 4 日正式发布）。

核心架构创新

图4：Netfilter 五大 Hook 点架构——实现策略与机制的彻底分离

Netfilter / iptables 的革命性改进
  • 五大 Hook 点（钩子）：PREROUTING、INPUT、FORWARD、OUTPUT、POSTROUTING，覆盖数据包流经协议栈的所有关键节点，模块可注册回调
  • 有状态连接跟踪（Conntrack）：ip_conntrack 模块独立维护连接状态表，可识别 NEW/ESTABLISHED/RELATED/INVALID 四种连接状态——这是最革命性的改进
  • 多表架构：filter（过滤）、nat（地址转换）、mangle（包修改）分离，后续又增加 raw（绕过 conntrack）和 security 表
  • 完整 NAT 支持：SNAT（源地址转换）、DNAT（目标地址转换）、MASQUERADE（IP 伪装）均以独立模块实现
  • 完全模块化：新的匹配条件（matches）和目标动作（targets）无需修改内核核心，以内核模块形式动态加载
  • IPv6 支持：通过 ip6tables 独立支持 IPv6 过滤（架构相同）
  • 向后兼容：提供 ipchains.ko 和 ipfwadm.ko 兼容层

有状态连接跟踪的革命

# iptables 有状态防火墙示例
# 允许已建立/相关的连接（真正的有状态检测）
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# 允许新的 SSH 连接
iptables -A INPUT -p tcp --dport 22 -m state --state NEW -j ACCEPT

# DNAT：将外部 80 端口请求转发到内部服务器
iptables -t nat -A PREROUTING -p tcp --dport 80 -j DNAT --to-destination 192.168.1.10:8080

# 默认策略：丢弃所有未匹配的入站包
iptables -P INPUT DROP

根本性痛点（驱动第四代演进的力量）

iptables 统治 Linux 防火墙长达 13 年（2001-2014），在此期间其局限性逐渐积累：

iptables 的根本性架构缺陷
  • 规则更新需替换全表（原子性问题）：添加/删除单条规则时，iptables 工具必须把整张表从内核 dump 出来，修改后再整体写回内核。内核看不到"哪条规则变了"，只知道"来了一张新表"。在 Kubernetes 等容器编排场景中，10 万条规则 + 15 秒 iptables-restore 是真实的生产痛点。
  • 线性规则匹配，性能随规则数 O(N) 退化：每个包要从链头逐条匹配，规则越多性能越差。在 DDoS 防护、大型集群场景下成为瓶颈。
  • 协议知识硬编码在内核：TCP 端口提取模块与 UDP 端口提取模块是不同的内核代码，大量重复。Patrick McHardy 估计即使经过四年整合，内核仍有约 10,000 行重复过滤代码。
  • IPv4/IPv6 需要两套工具：iptables（IPv4）、ip6tables（IPv6）、arptables（ARP）、ebtables（桥接），配置同一策略需重复编写。
  • ABI 僵化，阻碍扩展：iptables 的内核 ABI 包含协议特定字段，内核内部数据结构与 ABI 结构直接共用，改一个必须两边同步，风险极高。
  • 全局写锁争用：多 CPU 场景下，规则更新需要跨所有 CPU 同步，CPU 越多，锁等待越长。
"The iptables ABI has become a serious impediment to further progress in packet filtering... Patrick estimates that, even after four years of unification work, the kernel contains some 10,000 lines of duplicated filtering code."
— Jonathan Corbet，LWN.net《Nftables: a new packet filtering engine》2009
第一性原理
策略语言与执行机制不应共享同一个 ABI

iptables 最大的架构错误是：用户空间的规则表示格式（ABI）与内核内部的匹配数据结构直接复用。这意味着每次规则更新都必须以"全量替换"而非"增量修改"的方式进行——就像每次修改一个文件，都必须重写整个磁盘。

从第一性原理看，内核需要的是"执行引擎"，用户需要的是"策略描述语言"。这两者的演进速度和关注点完全不同，将它们绑定在同一 ABI 中，最终会让双方都无法自由演进。

nftables 的根本解法是：将规则"编译"成字节码再送入内核的虚拟机执行，ABI 只暴露"加载字节码"这一个操作——内核不需要理解规则语义，只需要执行字节码。

第四代 · 2009~2014 年 · Linux 3.13

nftables — 虚拟机字节码的统一框架

历史背景

2008 年，在 Netfilter Workshop 上，Patrick McHardy 首次公开提出 nftables 的构想。2009 年 3 月 18 日，第一个公开预览版发布。2013 年 10 月，nftables 合并进 Linux 主线；2014 年 1 月，随 Linux 3.13 正式发布。

"nftables: a new packet filtering engine... this could be the biggest change to Linux firewalling since the introduction of iptables in 2001."
— Jonathan Corbet，LWN.net, 2009年3月

核心架构：内核虚拟机

nftables 最根本的创新是将内核变成了一台简单的虚拟机，执行由用户空间编译器生成的字节码，内核本身不再内置任何协议知识：

图5：nftables 架构——用户空间编译器 + 内核虚拟机 + Sets/Maps 数据结构

nftables 的核心改进（来源：LWN.net、Red Hat 官方博客、netfilter.org）
  • 内核虚拟机（nf_tables VM）：规则在用户空间被编译成字节码，内核只执行通用指令（载荷加载、寄存器比较、集合查找），完全去除协议特定代码
  • 统一的 nft 工具：单一命令替代 iptables、ip6tables、arptables、ebtables 四个工具
  • inet 地址族：同一条规则可同时匹配 IPv4 和 IPv6，无需重复编写
  • Sets 和 Maps 数据结构：将 N 条"匹配不同 IP"的规则合并为一次哈希/红黑树查找，性能从 O(N) 降为 O(1) 或 O(log N)
  • 原子性增量更新：通过 Netlink 接口单独增删规则，内核知道"哪条规则变了"，无需全表替换
  • 无锁读路径（RCU）：读路径使用 RCU 机制，不需要在所有 CPU 上同步，高并发性能显著提升
  • 更强大的表达能力：支持字典（dictionaries）、映射（maps）、拼接（concatenations）等高级数据结构
  • 兼容层：提供 iptables-nft（iptables-legacy 语法 → nftables 字节码），渐进迁移

nftables 语法示例

# nftables 示例：统一 IPv4/IPv6 防火墙
table inet filter {
    # 使用 Set 存储黑名单 IP（哈希查找，O(1)）
    set blacklist {
        type ipv4_addr
        elements = { 10.0.0.1, 192.168.99.5, 203.0.113.0/24 }
    }

    chain input {
        type filter hook input priority 0; policy drop;

        # 一条规则匹配所有黑名单 IP（无论列表有多长）
        ip saddr @blacklist drop

        # 允许已建立连接（有状态）
        ct state established,related accept

        # 允许 SSH（同时适用 IPv4 和 IPv6）
        tcp dport 22 accept
    }
}

iptables vs nftables 性能对比

图6：规则数量对处理延迟的影响——nftables Sets 使延迟近乎不随规则数增长（示意图，基于架构原理）

第一性原理
将"编译"从运行时移到配置时，是性能优化的普适原则

nftables 的核心洞见是：规则匹配的大部分"理解工作"可以在规则加载时（配置时）完成，而不是在每个数据包到达时（运行时）重复计算。

这与编程语言中"解释器 vs 编译器"的经典权衡如出一辙：iptables 是"解释器"——每个包都触发一次对规则的线性解读；nftables 是"编译器"——规则被预先编译成高效的字节码，执行时只做最少量的工作。

Sets/Maps 则进一步把"N 次比较"压缩为"1 次哈希查找"，这是算法层面的根本性降维：让数据结构承载策略的复杂性，而不是让执行路径承载。

05
前沿探索 · 2014年至今 · Linux 3.18 / 4.8+

eBPF / XDP / bpfilter — 可编程数据平面

历史背景

即便 nftables 解决了 iptables 的诸多问题，一个更根本的问题仍然存在：Netfilter 框架本身在内核协议栈的"中层"工作，数据包已经历过了若干处理才到达过滤点。对于 DDoS 防护、高频交易、云原生网络等超高性能场景，这仍然太慢。

与此同时，eBPF（extended Berkeley Packet Filter）在 2014 年经 David S. Miller 重写后进入主线内核（Linux 3.18），成为一个通用的安全内核可编程框架。2016 年，XDP（eXpress Data Path）在 Linux 4.8 中出现，将 eBPF 程序的执行点下沉到网卡驱动层，在数据包被内核协议栈处理之前就进行过滤。

图7：XDP 将过滤点下沉至网卡驱动层，数据包无需经过完整协议栈即可被丢弃或转发

eBPF 关键里程碑（来源：Wikipedia eBPF 词条）

eBPF / XDP 发展时间线
  • 2011年4月：Linux 内核首个 BPF JIT 编译器合并
  • 2014年3月：David S. Miller 重写 BPF 解释器，引入 eBPF（Linux 3.18）
  • 2015年3月：eBPF 可附着到流量控制（TC）层，用于网络过滤
  • 2016年7月：XDP（eXpress DataPath）合并，eBPF 程序可挂载到网卡驱动核心收包路径（Linux 4.8）
  • 2018年1月：AF_XDP socket 发布，支持用户空间零拷贝 XDP 处理
  • 2018年2月：bpfilter 原型发布——将 iptables 规则翻译成 eBPF 程序（David Miller、Alexei Starovoitov、Daniel Borkmann），但因争议未合并主线
  • 2020年3月：Google 将 BPF LSM 支持合并进内核，实现可编程 Linux 安全模块

bpfilter：夭折的第五代尝试

2018 年 2 月，Linux 网络子系统维护者 David S. Miller 提出了 bpfilter 的 RFC 补丁：其核心思路是在内核中引入一种新的"用户模式驱动（User Mode Helper）"模块，将 iptables 的规则集翻译成 eBPF 程序并通过 XDP 钩子执行，从而在不改变用户空间 iptables 接口的情况下，将底层执行引擎换成 eBPF/JIT。

"The bpfilter prototype has been published, allowing translation of a subset of iptables rulesets into eBPF via a newly developed user mode driver. The work has caused controversies due to the ongoing nftables development effort and has not been merged into mainline."
— Wikipedia eBPF 词条，引用 LWN.net 2018
bpfilter 为何未进入主线
  • 与 nftables 路线竞争：netfilter 社区认为应继续推进 nftables，而非为遗留的 iptables 接口"续命"
  • 100% iptables 行为复现几乎不可能：iptables 的边界行为、副作用过于复杂，eBPF 翻译层无法完全等效
  • 新型内核模块机制的安全顾虑：bpfilter 使用的"用户空间 ELF 模块在内核线程中运行"机制引发安全审查
  • 实际结果：社区最终选择以 nftables 为未来方向，同时 eBPF/XDP 作为专项高性能场景的独立技术栈并行发展

当前格局：nftables + eBPF/XDP 双轨并行

# eBPF/XDP 程序示例（C 语言，以 libbpf 方式编写）
// 在网卡驱动层丢弃来自特定 IP 的数据包
SEC("xdp")
int xdp_drop_ip(struct xdp_md *ctx) {
    void *data = (void *)(long)ctx->data;
    void *data_end = (void *)(long)ctx->data_end;
    
    struct ethhdr *eth = data;
    if ((void *)(eth + 1) > data_end) return XDP_PASS;
    if (eth->h_proto != htons(ETH_P_IP)) return XDP_PASS;
    
    struct iphdr *ip = (void *)(eth + 1);
    if ((void *)(ip + 1) > data_end) return XDP_PASS;
    
    // 丢弃来自 10.0.0.1 的包（在驱动层，零拷贝）
    if (ip->saddr == htonl(0x0a000001))
        return XDP_DROP;  // 丢弃，连内核协议栈都不进入
    
    return XDP_PASS;
}
第一性原理
可编程性比固定框架更持久：让内核成为执行平台而非功能集合

eBPF 的根本哲学是：与其不断往内核里加功能，不如让内核成为一个安全可编程的执行平台。用户可以在运行时向内核注入经过验证器校验的字节码，内核提供的是"执行保障"而非"功能清单"。

XDP 将这一逻辑推向极致：不但可编程，而且把执行点下沉到硬件-软件的最边界——网卡 DMA 之后、协议栈之前。此时过滤成本趋近于零（数据包尚未被内核"看见"就被处理），这是从架构上消灭不必要工作的最彻底方式。

这与"第一代"的 ipfwadm 形成了完美的回环：ipfwadm 是"把过滤功能硬编码进内核"，eBPF 是"把内核变成可运行过滤逻辑的安全容器"——方向完全相反，但都在回答同一个问题：如何最高效地在数据路径上做决策？

各代技术的现状与使用建议

2024~2026 年现状
  • iptables：2026 年已进入"官方遗留维护模式"，Netfilter 项目正式引导新部署迁移至 nftables。大量发行版已将 iptables 命令的底层实现替换为 iptables-nft（iptables 语法 + nftables 后端）
  • nftables：现为 Debian 10+、RHEL 8+、Ubuntu 20.04+、Fedora 等主流发行版的默认防火墙框架。firewalld（Red Hat）已将 nftables 作为默认后端
  • eBPF/XDP：Cloudflare、Meta（Facebook Katran）、Cilium/Kubernetes 网络已大规模生产应用。不替代 nftables，而是在超高性能场景互补
  • 采用建议：通用防火墙配置 → nftables；Kubernetes/容器网络 → Cilium/eBPF；DDoS 防护/超高 PPS 场景 → XDP
第一性原理 · 终篇
所有演进都在回答同一个问题：用最少的资源，在最早的时机，做最正确的决定

回顾三十年来 Linux 防火墙的五代演进，每一次更替都不是"为了技术而技术"，而是在回答一个永恒的工程问题：

①何时检查（When）——执行点从 IP 层前移到驱动层，越早越好，代价越小。

②检查什么（What）——从无状态包头匹配，到有状态连接追踪，到任意可编程逻辑，匹配能力随安全需求扩张。

③如何描述检查（How to express）——从命令行规则，到模块化框架，到编译器+虚拟机，到可编程字节码，表达能力与执行效率持续解耦。

底层逻辑是第一性原理的三个层面：物理约束（网络带宽 × 包速率）、安全需求（攻击模型的复杂度）、工程约束（可维护性与可扩展性）。当三者中任意一个超越了当前抽象层的能力边界，新的抽象层就必须出现。Linux 防火墙的历史，就是不断突破这些边界的历史。
