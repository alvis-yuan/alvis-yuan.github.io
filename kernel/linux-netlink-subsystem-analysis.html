Linux Netlink 子系统深度解析

基于内核版本 5.10.186 (LTS) · 源码路径：net/netlink/

内核空间 用户空间 IPC 网络子系统 RFC 3549

§1 引言与背景

Netlink 是 Linux 内核提供的一种 套接字（socket）基础的进程间通信（IPC）机制， 专门用于内核空间与用户空间进程之间，以及不同用户空间进程之间交换信息。 它诞生于对 ioctl() 接口的不满——ioctl 使用固定格式的 C 结构体， 难以扩展，而 Netlink 则引入了可扩展的 TLV（Type-Length-Value）属性格式。

1.2 Netlink 与 ioctl 的对比

维度ioctlNetlink
消息格式固定结构体TLV 可扩展
异步通知不支持支持（多播组）
双向通信请求-响应只能单次全双工，可异步
多播不支持支持（最多32组/协议）
版本兼容难以扩展可向后兼容地添加新属性
用户接口系统调用标准 socket API
📌 Netlink 不是可靠协议。内存不足时消息可能被丢弃。发送方可设置 NLM_F_ACK 标志请求确认，接收方需处理 ENOBUFS 错误进行重同步。

本质：一切内核配置接口的问题，都归结于「如何在不破坏既有 ABI 的前提下传递结构化数据」。 ioctl 用固定宽度结构体，每次扩展都要新增 ioctl 号，是「静态合同」； Netlink 用带长度字段的 TLV，相当于「自描述的动态合同」——双方只需约定 Type， 不理解的 Type 直接跳过，从而天然支持向前和向后兼容。

2.3 通信模式

模式方向标志典型场景
单播 Do用户→内核→用户NLM_F_REQUEST | NLM_F_ACK设置路由、添加地址
Dump用户→内核→用户（多消息）NLM_F_REQUEST | NLM_F_DUMP枚举所有路由、接口
多播通知内核→多个用户无（异步推送）链路状态变化、路由更新

本质：Netlink 的核心设计原理是「协议族复用 + 消息驱动 + 发布订阅」。 用一个 AF_NETLINK 地址族承载多达 32 个不同协议子系统（用 protocol 字段区分）， 每个子系统内部再用消息类型（nlmsg_type）路由到具体处理函数； 多播组机制则把「主动推送」变成标准能力，无需轮询。 这是「事件驱动架构」在内核 IPC 层的体现。

§3 消息格式

3.1 nlmsghdr — 通用消息头

定义于 include/uapi/linux/netlink.h，是所有 Netlink 消息的统一入口，长度固定 16 字节： 

struct nlmsghdr {
    __u32  nlmsg_len;   /* 消息总长度，含本头部 */
    __u16  nlmsg_type;  /* 消息类型/操作类型 */
    __u16  nlmsg_flags; /* 附加标志位 */
    __u32  nlmsg_seq;   /* 序列号，用于匹配请求/响应 */
    __u32  nlmsg_pid;   /* 发送端 Port ID（非进程 PID）*/
};

3.2 nlmsg_type 保留类型

类型值常量含义
0x01NLMSG_NOOP忽略，无实际用途
0x02NLMSG_ERROR错误响应，携带 nlmsgerr

3.3 nlmsg_flags 标志位

/* 标准标志 */
#define NLM_F_REQUEST   0x01  /* 所有请求必须设置 */
#define NLM_F_MULTI     0x02  /* 多部分消息 */
#define NLM_F_ACK       0x04  /* 成功时也请求 ACK */
#define NLM_F_ECHO      0x08  /* 回显通知到请求者 */

/* GET 请求额外标志 */
#define NLM_F_ROOT      0x100 /* 返回完整表 */
#define NLM_F_MATCH     0x200 /* 返回匹配项 */
#define NLM_F_DUMP      /* = NLM_F_ROOT|NLM_F_MATCH */

/* NEW 请求额外标志 */
#define NLM_F_REPLACE   0x100 /* 替换已有对象 */
#define NLM_F_EXCL      0x200 /* 对象已存在则失败 */
#define NLM_F_CREATE    0x400 /* 不存在则创建 */
#define NLM_F_APPEND    0x800 /* 追加到列表末尾 */

3.4 错误消息格式（nlmsgerr）

struct nlmsgerr {
    int            error;  /* 负 errno，成功为 0 */
    struct nlmsghdr msg;   /* 导致错误的原始请求头 */
    /* 可选：Extended ACK TLV 属性（需 NETLINK_EXT_ACK setsockopt）*/
};

图 4：NLMSG_ERROR 消息内存布局

本质：消息格式的核心约束是「头部固定 + 载荷可变」。 固定头部提供路由信息（去哪、谁发、类型是什么）， 可变载荷承载内容。对齐到 4 字节边界（NLMSG_ALIGN）是性能与 ABI 稳定的工程权衡—— 统一对齐让内核解析路径可以零拷贝地直接操作指针，无需字段拷贝。

§4 协议模型与寻址

4.1 地址格式（sockaddr_nl）

struct sockaddr_nl {
    sa_family_t  nl_family;  /* AF_NETLINK */
    unsigned short nl_pad;  /* 置零 */
    pid_t        nl_pid;    /* Port ID（非进程PID）*/
    __u32        nl_groups; /* 多播组掩码 */
};
nl_pid 规则
  • nl_pid = 0：目标为内核；或让内核自动分配
  • 内核自动分配：第一个 socket 分配为进程 PID，后续分配唯一 ID
  • 应用可 bind() 时手动指定，需保证唯一性
  • 注意：nl_pid 标识 socket，不是进程

4.3 多播组机制

每个协议族最多支持 32 个多播组（经典 Netlink）。订阅通过 bind() 设置 nl_groups 位掩码，或 setsockopt(NETLINK_ADD_MEMBERSHIP)：

/* 订阅组，需 CAP_NET_ADMIN 或 root */
unsigned int group_id = RTNLGRP_LINK; /* 链路状态变化组 */
setsockopt(fd, SOL_NETLINK, NETLINK_ADD_MEMBERSHIP,
           &group_id, sizeof(group_id));

本质：Netlink 的寻址模型本质上是「协议族 × 端口ID × 多播组」三元组。 协议族做一级分流（路由到正确的内核子系统），端口ID做单播寻址，多播组位掩码做一对多分发。 这与 UDP 多播类似，只不过「网络」换成了内核内部的消息总线， 无需额外的网络栈开销，直接操作 sk_buff。

§5 内核核心数据结构

5.1 netlink_sock — 扩展 socket 结构

定义于 net/netlink/af_netlink.h，在通用 struct sock 基础上扩展：

struct netlink_sock {
    /* struct sock 必须是第一个成员，用于 container_of */
    struct sock           sk;

    unsigned long         flags;        /* NETLINK_F_* 标志位 */
    u32                   portid;       /* 本 socket 的 Port ID */
    u32                   dst_portid;   /* 对端 Port ID */
    u32                   dst_group;    /* 目标多播组 */
    u32                   subscriptions;/* 已订阅组数量 */
    u32                   ngroups;      /* 总多播组数 */
    unsigned long        *groups;       /* 已订阅组位图 */
    unsigned long         state;        /* 状态位 */
    size_t                max_recvmsg_len;

    wait_queue_head_t     wait;
    bool                  bound;        /* 是否已 bind() */
    bool                  cb_running;   /* dump 回调运行中 */
    int                   dump_done_errno;
    struct netlink_callback cb;         /* dump 回调上下文 */
    struct mutex          nl_cb_mutex;

    /* 内核 socket 回调钩子 */
    void  (*netlink_rcv)(struct sk_buff *skb);
    int   (*netlink_bind)(struct net *net, int group);
    void  (*netlink_unbind)(struct net *net, int group);
    void  (*netlink_release)(struct sock *sk,

    struct module        *module;
    struct rhash_head    node;         /* rhashtable 节点 */
    struct rcu_head      rcu;          /* RCU 释放 */

5.3 锁机制

锁类型保护对象
nl_table_lockrwlock_tnl_table 数组整体读写

本质：内核数据结构设计遵循「继承复用 + 控制块内联」原则。 netlink_sock 把 struct sock 作为第一个成员， 通过 container_of 在通用 sock 指针和 netlink_sock 之间零开销转换。 NETLINK_CB(skb) 把控制信息内联进 skb->cb[] 48 字节控制块， 避免额外内存分配——这是在「性能」和「抽象」之间取得极致平衡的经典内核技法。

§6 Socket 操作层

6.1 协议族注册与初始化

Netlink 通过 netlink_proto_init() 在内核启动时注册，位于 net/netlink/af_netlink.c 底部：

/* Netlink 注册到网络协议栈 */
static const struct net_proto_family netlink_family_ops = {
    .family = PF_NETLINK,
    .create = netlink_create,  /* 创建 socket 时调用 */
    .owner  = THIS_MODULE,
};

static int __init netlink_proto_init(void)
{
    int i;
    int err = proto_register(&netlink_proto, 0);
    if (err != 0) goto out;

    /* 分配并初始化 MAX_LINKS(32) 个协议槽的哈希表 */
    nl_table = kcalloc(MAX_LINKS, sizeof(*nl_table), GFP_KERNEL);

    for (i = 0; i < MAX_LINKS; i++)
        rhashtable_init(&nl_table[i].hash,
                         &netlink_rhashtable_params);

    netlink_add_usersock_entry();
    sock_register(&netlink_family_ops); /* 注册 AF_NETLINK */
    register_pernet_subsys(&netlink_net_ops);
    /* ... */
}
core_initcall(netlink_proto_init);

6.2 内核侧创建 Netlink Socket

内核子系统调用 netlink_kernel_create() 创建内核侧的 netlink socket：

struct netlink_kernel_cfg cfg = {
    .input  = rtnetlink_rcv,  /* 消息接收回调 */
    .groups = RTNLGRP_MAX,    /* 多播组数 */
    .cb_mutex = &rtnl_mutex,   /* 串行化锁 */

/* 示例：rtnetlink 在 net/core/rtnetlink.c 中初始化 */
struct sock *sk = netlink_kernel_create(net, NETLINK_ROUTE, &cfg);

§7 消息收发路径

7.1 发送路径（用户→内核）

/* net/netlink/af_netlink.c: netlink_sendmsg() 关键路径 */
static int netlink_sendmsg(struct socket *sock, struct msghdr *msg,
                           size_t len)
{
    struct sock *sk = sock->sk;
    struct netlink_sock *nlk = nlk_sk(sk);
    struct sockaddr_nl *addr = ...;
    struct sk_buff *skb;

    /* 1. 分配 skb，将用户态数据拷贝进来 */
    skb = netlink_alloc_large_skb(len, ...);
    if (memcpy_from_msg(skb_put(skb, len), msg, len) < 0) ...

    /* 2. 设置 NETLINK_CB 控制块 */
    NETLINK_CB(skb).portid    = nlk->portid;
    NETLINK_CB(skb).dst_group = dst_group;

    /* 3. 单播/多播分发 */
    if (dst_group)
        return netlink_broadcast(sk, skb, ...);
    else
        return netlink_unicast(sk, skb, dst_portid, ...);
}

本质：内核的消息收发路径本质上是「生产者-消费者 + 回调」模式。 发送方把 sk_buff 放入目标 socket 的接收队列（生产），通过 sk_data_ready 唤醒消费者； 内核侧 socket 无需轮询，回调函数在有数据时被直接调用（消费）。 整个流程无需系统调用上下文切换的额外开销，是零拷贝异步 IPC 的体现。

图 8：netlink_broadcast 多播机制 — 按组掩码匹配，per-subscriber skb_clone

8.3 通知链（Notification Chain）与 Netlink 联动

许多内核子系统（如网络设备、路由）用 内核通知链（notifier_call_chain） 通知内部模块，rtnetlink 则将这些事件"转换"为 Netlink 多播消息发给用户态： 

/* net/core/rtnetlink.c — 网络设备变化通知 */
static int rtnetlink_event(struct notifier_block *this,
                            unsigned long event, void *ptr)
{
    struct net_device *dev = netdev_notifier_info_to_dev(ptr);
    switch (event) {
    case NETDEV_UP:
    case NETDEV_DOWN:
        rtmsg_ifinfo(RTM_NEWLINK, dev, 0, GFP_KERNEL);
        /* 最终调用 nlmsg_multicast 发给 RTNLGRP_LINK 组 */
        break;
    }
    return NOTIFY_DONE;
}

本质：Netlink 多播的核心是「发布-订阅 + 按需克隆」。 内核事件只生成一个 skb（生产一次）， netlink_broadcast 遍历订阅者列表，对每个匹配者做 skb_clone（引用计数+1）而非完整复制， 最后每个消费者释放自己的 clone，当所有引用归零时原 skb 才被释放。 这是写时复制（COW）思想在 IPC 多播场景中的经典实现。

§9 经典 Netlink 协议

9.1 特点与局限

早期 Netlink（经典 Netlink）的特点是：

  • 静态 ID 分配：协议号在头文件中硬编码，最多 32 个
  • 无内省能力：用户无法查询内核支持哪些消息类型
  • 混合格式：消息中大量使用固定宽度结构体（如 ifinfomsg、rtmsg），与 TLV 混用
  • 严格检查选项：NETLINK_GET_STRICT_CHK setsockopt 选项（5.2 引入）用于向后兼容地开启字段校验

9.2 请求类型模式（NETLINK_ROUTE）

NETLINK_ROUTE 将操作分为四类，消息类型低 2 bit 编码操作类型（步长为 4）：

操作低2位示例
NEW00RTM_NEWLINK, RTM_NEWROUTE
DEL01RTM_DELLINK, RTM_DELROUTE
GET10RTM_GETLINK, RTM_GETROUTE
SET11RTM_SETLINK

9.3 经典使用示例（获取路由表）

/* 用户空间：发起路由表 dump 请求 */
struct {
    struct nlmsghdr  nlh;
    struct rtmsg     rtm;
} request;

request.nlh.nlmsg_type  = RTM_GETROUTE;
request.nlh.nlmsg_flags = NLM_F_REQUEST | NLM_F_DUMP;
request.nlh.nlmsg_len   = sizeof(request);
request.nlh.nlmsg_seq   = time(NULL);
request.rtm.rtm_family  = AF_INET;

send(nl_sock, &request, sizeof(request), 0);

本质：经典 Netlink 的设计教训在于「协议 ID 空间是有限资源，静态分配会枯竭」。 MAX_LINKS=32 在 2005 年前后就已接近满额，迫使引入 Generic Netlink 进行二级复用。 这正是「过早优化资源会阻碍扩展性」的工程案例—— Generic Netlink 本质上就是在 Netlink 协议 ID 16 上实现了一个「元协议」（协议的协议）。

§10 Generic Netlink（NETLINK_GENERIC）

10.1 设计动机与架构

Generic Netlink（2005 年，2.6.15）是对经典 Netlink 协议 ID 耗尽问题的解决方案。 它在 NETLINK_GENERIC（ID=16） 上实现了一个动态多路复用层， 允许任意数量的子系统（Family）注册，Family ID 由内核动态分配。

图 9：Generic Netlink 多路复用架构 — 单一协议号承载无限 Family

10.4 Family ID 发现流程

用户程序需要先查询 CTRL_CMD_GETFAMILY 获取动态 Family ID：

/* 向 GENL_ID_CTRL（固定 ID=16）发送查询 */
struct nlmsghdr → nlmsg_type = GENL_ID_CTRL
struct genlmsghdr → cmd = CTRL_CMD_GETFAMILY
struct nlattr → nla_type = CTRL_ATTR_FAMILY_NAME
                 data = "DOC_EXMPL\0"

/* 内核返回：*/
struct nlattr → nla_type = CTRL_ATTR_FAMILY_ID
                 data = 123  /* 动态分配的 Family ID */

10.5 genl_ops 结构（5.10.186）

/* include/net/genetlink.h */
struct genl_ops {
    int  (*doit)(struct sk_buff *skb, struct genl_info *info);
    int  (*start)(struct netlink_callback *cb);
    int  (*dumpit)(struct sk_buff *skb, struct netlink_callback *cb);
    int  (*done)(struct netlink_callback *cb);
    const struct nla_policy *policy;
    unsigned int  maxattr;
    u8            cmd;
    u8            internal_flags;
    u8            flags;
    u8            validate;
};

本质：Generic Netlink 的本质是「在有限资源上实现无限扩展性」的元编程技术。 用一个固定协议号（16）承载"协议注册协议"（CTRL）， 其余所有 Family 动态申请 ID，并通过"名字解析"（CTRL_CMD_GETFAMILY）实现运行时发现。 这与 DNS 将域名解析为 IP 的思想完全一致—— 字符串名称比数字 ID 更具可读性和可扩展性，数字 ID 只在通信时才有意义。

Dump 用于枚举内核中所有某类对象（如所有路由、所有接口）。 由于对象可能很多，无法在单个消息中返回，Netlink 使用多部分消息（multipart）机制：

图 10：Netlink Dump 多部分消息序列

12.2 netlink_callback 结构

/* include/linux/netlink.h — Dump 状态上下文 */
struct netlink_callback {
    struct sk_buff    *skb;      /* 原始请求 skb */
    const struct nlmsghdr *nlh;  /* 原始请求头 */
    int (*start)(struct netlink_callback *);
    int (*dump)(struct sk_buff *skb, struct netlink_callback *);
    int (*done)(struct netlink_callback *);
    void    *data;                /* 子系统私有数据 */
    long    args[6];             /* 状态跟踪（已枚举到哪里）*/
};

/* dumpit 回调模式：每次调用填充尽可能多的消息，
   返回 0 表示完成，返回负值或 skb 被填满则继续 */

12.3 Dump 一致性

⚠️ 某些内核数据结构（如路由表）在 dump 期间可能被修改，导致 dump 不一致。内核会在消息头设置 NLM_F_DUMP_INTR 标志，用户空间收到此标志应重新发起 dump。

本质：Dump 机制本质上是「有状态的分页迭代器」。 netlink_callback.args[] 保存迭代器状态（已遍历到第几个对象）， 每次内核被唤醒时从上次断点继续，类似数据库游标。 不同之处在于内核无法阻止迭代期间的写入，所以 NLM_F_DUMP_INTR 实现了 「乐观并发」——先读，发现脏了再重试，避免长时间持锁阻塞写路径。

本质：TLV 属性系统的本质是「自描述的二进制 JSON」。 Type 提供语义标识，Length 提供边界信息，Value 承载数据， 不理解某个 Type 的接收方可以直接跳过（按 Length 跳转），实现天然的前向兼容。 嵌套属性对应 JSON 对象，数组属性对应 JSON 数组。 相比 protobuf 等格式，Netlink TLV 没有 schema 文件， 内核通过 nla_policy 在运行时自动校验——这是运行时类型安全的体现。

§14 安全与网络命名空间

14.1 权限控制

操作权限要求
发送/接收多播某些协议族需要 CAP_NET_ADMIN 或 effective UID 0
修改路由/地址CAP_NET_ADMIN
读取 dump通常无需特权（协议族可配置 NL_CFG_F_NONROOT_SEND）
内核 socket（NETLINK_F_KERNEL_SOCKET）内核内部创建，用户不可直接访问

14.2 网络命名空间（Net Namespace）

Linux 5.10 中，Netlink socket 与创建它的网络命名空间（net ns）绑定。 nl_table 通过 struct net 隔离，不同 ns 中的 socket 不能互相通信： 

/* sockaddr_nl 的 portid 在 net namespace 中唯一 */
static struct sock *__netlink_lookup(struct netlink_table *table,
                                        u32 portid,
                                        struct net *net)
{
    struct netlink_compare_arg arg;
    netlink_compare_arg_init(&arg, net, portid);
    /* 以 (net, portid) 二元组作为哈希键 */
    return rhashtable_lookup_fast(&table->hash, &arg,
                                    netlink_rhashtable_params);
}

图 13：Netlink 与网络命名空间隔离 — (net, portid) 二元组唯一寻址

14.3 扩展 ACK（Extended ACK）

5.2 版本引入，通过 setsockopt(NETLINK_EXT_ACK, 1) 启用， 在错误时返回更详细的诊断信息： 

/* 启用扩展 ACK */
int val = 1;
setsockopt(fd, SOL_NETLINK, NETLINK_EXT_ACK, &val, sizeof(val));

/* 内核在 NLMSG_ERROR 后追加：
   NLMSGERR_ATTR_MSG  → 人类可读错误字符串
   NLMSGERR_ATTR_OFFS → 出错属性在消息中的偏移
   NLMSGERR_ATTR_MISS_TYPE → 缺少的必需属性类型 */

本质：Netlink 安全模型遵循「能力最小化原则」—— 读操作通常不需要特权（任何进程都可以查询路由表）， 写操作才需要 CAP_NET_ADMIN。 网络命名空间隔离是「容器化」的基石，Netlink 的 (net, portid) 二元键设计确保不同 ns 中相同 portid 不会冲突， 是「地址空间分离」原则在 IPC 层的应用。

15.3 5.10.186 重要特性

5.10 LTS 中 Netlink 的关键特性
  • rhashtable：portid 查找使用并发友好的 RCU 哈希表，替代旧的 nl_pid_hash
  • Extended ACK：详细的错误诊断 TLV（NLMSGERR_ATTR_MSG/OFFS/MISS）
  • NETLINK_GET_STRICT_CHK：NETLINK_ROUTE 严格输入校验模式
  • Net Namespace 完整支持：(net, portid) 二元键寻址
  • BPF 集成：BPF 迭代器可遍历 netlink socket 状态
  • NLM_F_DUMP_INTR：Dump 一致性中断检测

终极第一性原理：Netlink 解决的根本问题是 「如何在内核保护域与用户保护域之间建立一个可扩展、异步、类型安全的通信通道」。 它的每一个设计决策都回归到这个问题： socket API 复用（利用已有的 VFS/epoll 基础设施）、TLV 格式（可扩展性）、 多播组（异步推送）、nla_policy（类型安全）、网络命名空间（隔离性）。 没有任何一个设计是偶然的——它们都是对「用最小新增机制，满足最大功能需求」这一原则的实践。
