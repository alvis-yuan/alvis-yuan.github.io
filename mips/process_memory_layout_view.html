应用进程视角的 Linux 内存布局

本文从“一个用户态应用启动后，它自己眼中的内存世界是什么样”这个角度出发，梳理代码段、数据段、堆、mmap 区、用户栈、内核栈、VDSO 等对象的角色与形成过程。讲解重点放在原理、时机和行为，不以源码细节为主；源码仅作为轻量锚点，帮助把概念落到当前仓库的实现上。

一、先建立整体直觉

从应用视角看，看到的是“虚拟地址空间”，不是物理内存。

进程读写的每一个地址，首先都是虚拟地址。CPU 和内核再把这些虚拟地址翻译成物理地址，或者在需要时触发缺页异常去补齐映射。进程通常并不知道某个地址背后是否真的已经分配了物理页。

从内核视角看，这些地址空间由 VMA 组织。

Linux kernel 官方文档强调：用户地址空间由一个个 vm_area_struct 管理，每个 VMA 描述一段“虚拟上连续、属性一致”的范围。一个进程的所有 VMA 共同构成它的地址空间。

因此，“进程内存布局”不应理解为一块简单的连续 RAM，而应理解为：

  • 一张虚拟地址地图。
  • 地图上的每一块区域都有用途和权限，例如可执行、只读、私有、共享。
  • 地图上的块与块之间，可能还没真正装入物理页，只是先把“允许访问的范围”定义好了。

二、一个典型进程会“看见”哪些区域

以一个动态链接的普通用户程序为例，启动后它的虚拟地址空间里，通常会看到以下几类区域：

区域 主要用途 增长方向 / 特征
代码段 / 只读数据 程序指令、常量、ELF 装载内容 启动时由内核与动态链接器建立，通常位置相对稳定但可能受 ASLR 影响
.data / .bss 全局变量、静态变量 属于程序映像的一部分，和堆相邻但概念上不同
堆 heap 动态分配，小块 malloc() 常见来源 传统上由 brk/sbrk 增长，通常向高地址扩展
mmap 区 共享库、匿名映射、文件映射、大块内存 通常由内核在空闲地址中挑选，现代分配器大量依赖它
用户栈 函数调用栈、返回地址、局部变量、启动参数 通常向低地址增长，并带 guard page
VDSO 内核暴露给用户态的少量辅助代码页 例如某些时间相关调用的快速路径

三、这些区域是怎么来的

execve() 装载 ELF → 建立代码/数据相关映射 → 建立初始用户栈 → 动态链接器继续映射共享库 → 程序运行中通过 brk/mmap 扩展地址空间

1. 程序映像

当进程通过 execve() 启动时，内核读取 ELF 文件，把代码段、只读数据段、可写数据段等映射进新地址空间。对于动态链接程序，还会再装入动态链接器和所需共享库。

2. 初始用户栈

内核在 exec 阶段会建立临时栈并最终定位到合适的用户栈地址，随后把 argv、envp、辅助向量等压入栈中。当前仓库里的锚点是 [exec.c](/home/xiaoyuan/sourceCodes/x2600kernel/fs/exec.c#L747) 的 setup_arg_pages()。

3. 堆

堆与“program break”相关。传统上，进程通过 brk()/sbrk() 改变数据段末端，从而扩大或收缩堆。当前仓库里真正建立这类匿名区的内核路径是 [mmap.c](/home/xiaoyuan/sourceCodes/x2600kernel/mm/mmap.c#L3080) 的 do_brk_flags()。

4. mmap 区

mmap() 用于建立文件映射和匿名映射。共享库、JIT 代码、分配器的大块内存、线程栈等，经常都来自这里。man7 对 mmap(2) 的描述是最直接的权威来源。

四、堆和 mmap 区，为什么经常会让人混淆

很多人把“动态分配内存”简单等同于“堆”，但在现代 Linux 进程里，malloc() 背后可能同时使用两套来源：

  • 小块、频繁分配：常常来自 brk 扩展的传统堆。
  • 大块、特殊对齐、线程相关分配：常常直接来自 mmap 的匿名映射。
堆是一个概念，brk 是一种实现路径。

从应用员角度说“堆”没有问题；但从内核角度，heap 往往对应 program break 附近那段 VMA，而很多动态内存并不一定来自那里。

为什么现代分配器喜欢 mmap

mmap 更容易独立释放大块区域、减少碎片，也更适合线程化程序。于是你会看到某些程序的 /proc/pid/maps 中，真正的 [heap] 并不大，但匿名映射很多。

示意例子：
malloc(8 KB)    -> 可能来自已有 heap
malloc(64 MB)   -> 很可能直接触发匿名 mmap
dlopen()        -> 触发新的文件映射与匿名映射
pthread_create() -> 新线程通常需要新用户栈，常见实现也会借助 mmap

五、用户栈是什么，和内核栈有什么区别

用户栈

用户栈位于用户地址空间中，保存函数调用链、返回地址、局部变量、参数等。它是应用代码直接使用的栈。主线程启动时，argv、envp 和辅助向量也是从这里开始交给用户程序的。

内核栈

内核栈不在用户地址空间里。每个线程都会有自己的内核栈，用于系统调用、中断、异常和内核态执行。用户程序既看不到，也不能直接访问它。

当前 MIPS 仓库里可以看到：

  • 用户地址空间上限由 [processor.h](/home/xiaoyuan/sourceCodes/x2600kernel/arch/mips/include/asm/processor.h#L37) 一带的 TASK_SIZE / STACK_TOP_MAX 定义。
  • 内核栈基础访问接口在 [task_stack.h](/home/xiaoyuan/sourceCodes/x2600kernel/include/linux/sched/task_stack.h#L1)；它直接说明 task->stack 表示 kernel stack。
  • MIPS 上线程寄存器现场通常放在内核栈顶部附近，可从 [processor.h](/home/xiaoyuan/sourceCodes/x2600kernel/arch/mips/include/asm/processor.h#L438) 的 task_pt_regs(tsk) 看出。
一句话记忆

用户栈是“应用自己用的栈”，内核栈是“线程进内核以后临时落脚的栈”。二者都叫栈，但不在同一片地址空间，也不由同一段代码直接操作。

六、为什么说进程“看到”的布局并不等于已经分配好的物理内存

虚拟地址空间的一个核心价值，就是允许“先定义范围，再按需分配”。Linux kernel 官方文档对匿名内存的描述非常关键：匿名映射通常先只建立“允许访问”的地址范围；读访问可能映射到零页，写访问才真正分配普通物理页。

现象 进程看到什么 内核可能实际做了什么
刚 mmap 了一大块匿名内存 地址空间里多出一块可访问区域 可能只建立了 VMA 和部分页表，没有为每页都分配物理页
第一次读取某页 读到零值或文件内容 可能通过缺页异常补齐页表，匿名页可能先映射零页
第一次写匿名页 写成功 分配真正物理页，建立可写映射
fork 后写共享私有页 写成功，父子互不影响 触发 COW，分配新页并复制旧内容

七、一个具体例子：程序启动后，内存是如何逐步“长出来”的

int main(void)
{
    char local[4096];              // 局部变量：用户栈
    void *p1 = malloc(1024);       // 小块分配：常见来自 heap
    void *p2 = malloc(64 * 1024 * 1024); // 大块分配：常见直接走 mmap
    int fd = open("data.bin", O_RDONLY);
    void *p3 = mmap(NULL, 1 << 20, PROT_READ, MAP_PRIVATE, fd, 0); // 文件映射
    ...
}
对象 程序员的感受 更接近内核真实情况的解释
local[4096] 我在栈上放了一个数组 该地址位于用户栈 VMA 中，首次触碰时可能通过缺页建立真实页映射
malloc(1024) 堆上分配 1KB 分配器可能从已有 heap 切出 1KB，底层不一定新增系统调用
malloc(64MB) 堆上分配很大一块 分配器往往直接用匿名 mmap，在 /proc/pid/maps 中体现为新的匿名映射
文件 mmap 拿到一段像数组一样可读的内存 这段地址背后是文件页缓存和文件映射，第一次触碰时可能产生文件缺页
很好用的观察窗口

/proc/<pid>/maps 会把当前进程的映射区域列出来，包括地址范围、权限、偏移、设备、inode 和路径名。man7 的 proc_pid_maps(5) 对字段含义有权威说明。

理解顺序建议

先把“进程看见的是虚拟空间”吃透，再理解“VMA 是地图块”“页表是翻译表”“TLB 是翻译缓存”“缺页是按需兑现机制”。这样再回头看源码，就不会被细节先压住。
