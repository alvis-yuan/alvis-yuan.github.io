IPv4 邻居子系统：从路由下一跳到 ARP 状态机

邻居子系统位于三层路由和二层发送之间。FIB 决定下一跳 IP 与输出设备，邻居子系统负责把这个下一跳解析成链路层地址，并用 NUD 状态机判断这个映射是否仍可信。IPv4 的协议适配是 ARP；通用缓存、定时器、GC、队列和状态迁移在 `net/core/neighbour.c`。

官方依据

主题官方链接本文使用的事实
ARP 协议RFC 826ARP 用于把协议地址动态映射到链路层地址；发送前先查表，缺失时广播 request，收到 request/reply 后学习 sender 映射。
Linux ARP 模块arp(7)Linux ARP 维护有限大小的缓存，permanent 项不被 GC 删除；stale 后会重新探测；上层可通过正反馈延长可达性。
IPv4 sysctlip-sysctl`neigh/default/gc_thresh*`、`unres_qlen_bytes`、`mcast_solicit`、`ucast_solicit`、`arp_ignore`、`arp_announce`、`arp_accept` 等控制缓存、探测和 ARP 策略。
第一性原理概述：路由回答“下一跳是谁”，邻居子系统回答“这个下一跳当前对应哪个二层地址，以及这个答案还能不能信”。没有邻居层，IP 路由无法落到设备发送队列；没有状态机，旧 MAC 会长时间污染数据面。

核心模型：通用缓存 + 协议适配 + 设备发送

图 1：邻居子系统把路由选择出的下一跳 IP 转换为二层可发送目标。

Linux 把邻居缓存做成通用框架：`include/net/neighbour.h` 定义 `struct neigh_table`、`struct neighbour`、`struct neigh_ops`、`NUD_*` 状态；`net/core/neighbour.c` 实现 hash、创建、状态机、GC、未解析 skb 队列、netlink 通知和 sysctl；`net/ipv4/arp.c` 注册 `arp_tbl`，提供 IPv4 特有的 key、constructor、probe、ARP 收包和 proxy ARP。

第一性原理概述：IPv4 ARP 和 IPv6 ND 都需要“邻居可达性检测”。把状态机抽成通用 neighbour 层，可以让协议只负责如何发探测包和如何解释收到的链路层地址。

核心对象：一张表、一条邻居项、一组协议回调

对象关键字段/函数作用
`struct neigh_table``arp_tbl`、`hash`、`arp_constructor`、`parms`、`gc_thresh1/2/3`、`gc_work`一类邻居表。IPv4 ARP 的表 ID 是 `NEIGH_ARP_TABLE`，key 长度为 4 字节 IPv4 地址。
`struct neighbour``primary_key`、`dev`、`ha`、`nud_state`、`arp_queue`、`timer`、`confirmed/used/updated`、`output`一条 IP 到链路层地址映射，附带状态机、队列和发送函数。
`struct neigh_parms``MCAST_PROBES`、`UCAST_PROBES`、`RETRANS_TIME`、`BASE_REACHABLE_TIME`、`QUEUE_LEN_BYTES`全局或 per-device 参数。`devinet.c` 为 IPv4 设备分配 `in_dev->arp_parms`。
`struct neigh_ops``solicit`、`error_report`、`output`、`connected_output`协议适配层。IPv4 ARP 把 `solicit` 指向 `arp_solicit()`。
`hh_cache``neigh->hh`、`dev->header_ops->cache`、`neigh_hh_output()`缓存二层头，`NUD_CONNECTED` 且 header cache 有效时绕过慢速构造。

初始化：`arp_init()` 注册 ARP 表、包入口和设备事件

IPv4 协议族初始化 `inet_init()` 会调用 `arp_init()`。`arp_init()` 做四件事：`neigh_table_init(NEIGH_ARP_TABLE, &arp_tbl)` 初始化通用邻居表；`dev_add_pack(&arp_packet_type)` 注册 `ETH_P_ARP` 收包入口；创建 proc 节点；注册 `neigh_sysctl_register(NULL, &arp_tbl.parms, NULL)` 和 `arp_netdev_notifier`。

inet_init()
  -> arp_init()
       -> neigh_table_init(NEIGH_ARP_TABLE, &arp_tbl)
            -> 初始化 hash、proxy hash、gc_work、proxy_timer、默认 neigh_parms
       -> dev_add_pack(ETH_P_ARP -> arp_rcv)
       -> neigh_sysctl_register(NULL, &arp_tbl.parms, NULL)
       -> register_netdevice_notifier(&arp_netdev_notifier)

devinet: inetdev_init(dev)
  -> in_dev->arp_parms = neigh_parms_alloc(dev, &arp_tbl)

`arp_tbl` 的默认参数与 man page 对应：`MCAST_PROBES=3`、`UCAST_PROBES=3`、`RETRANS_TIME=1*HZ`、`BASE_REACHABLE_TIME=30*HZ`、`DELAY_PROBE_TIME=5*HZ`、`GC_STALETIME=60*HZ`、`QUEUE_LEN_BYTES=SK_WMEM_MAX`、`gc_thresh1/2/3=128/512/1024`。

NUD 状态：缓存项可信度的有限状态机

图 2：`NUD_NOARP` 和 `NUD_PERMANENT` 是伪状态，普通动态项围绕 INCOMPLETE/REACHABLE/STALE/DELAY/PROBE/FAILED 迁移。
状态语义代码行为
`NUD_INCOMPLETE`还不知道链路层地址。`__neigh_event_send()` 排队 skb，启动 timer，`neigh_probe()` 调 `arp_solicit()`。
`NUD_REACHABLE`最近确认过，映射可信。`neigh_output()` 可走 `neigh_hh_output()` 或 `connected_output` 快速发送。
`NUD_STALE`有地址但没有近期确认。下一次发送把状态改成 `NUD_DELAY`，先不立刻探测。
`NUD_DELAY`给上层正反馈一个窗口。`neigh_timer_handler()` 到期后若无确认则进入 `NUD_PROBE`。
`NUD_PROBE`主动确认旧地址是否仍有效。先用已有 MAC 单播探测，再可经应用探测、广播探测。
`NUD_FAILED`探测失败。`neigh_invalidate()` 调 `error_report` 并清空 `arp_queue`。

定时器：`neigh_timer_handler()` 驱动状态迁移

每个 `struct neighbour` 在 `neigh_alloc()` 中设置 `timer_setup(&n->timer, neigh_timer_handler, 0)`。状态进入 `NUD_IN_TIMER` 集合时会挂定时器，集合定义在 `include/net/neighbour.h`：`NUD_INCOMPLETE | NUD_REACHABLE | NUD_DELAY | NUD_PROBE`。

neigh_timer_handler()
  -> REACHABLE:
       confirmed 仍在 reachable_time 内: 继续保持
       recently used: 转 DELAY，等待上层确认
       否则: 转 STALE，通知用户态
  -> DELAY:
       confirmed 足够新: 回 REACHABLE
       否则: 转 PROBE，probes 清零
  -> INCOMPLETE / PROBE:
       probes >= neigh_max_probes(): 转 FAILED，清队列
       否则: 重新挂 retrans timer，调用 neigh_probe()

`neigh_max_probes()` 把 `UCAST_PROBES`、`APP_PROBES`、`MCAST_PROBES/MCAST_REPROBES` 相加。IPv4 的 probe 实现是 `arp_solicit()`：先决定 ARP sender IP，再根据 probes 次数选择单播旧 MAC、通知用户态 ARP daemon，或广播 ARP request。

GC 与容量控制：防止邻居表无限增长

`neigh_table_init()` 给每张表初始化 hash、proxy hash、`gc_work` 和 `proxy_timer`。周期任务 `neigh_periodic_work()` 会定期重算随机 reachable_time，并在表项数量达到 `gc_thresh1` 后扫描 hash。普通动态项如果 refcnt 只有表自身持有，且处于 `NUD_FAILED` 或超过 `GC_STALETIME` 未使用，就会被删除。

创建新项时 `neigh_alloc()` 也会看 `gc_thresh2/gc_thresh3`：超过软阈值并且距离上次 flush 足够久时尝试 forced GC；超过硬阈值且 GC 不能释放空间时返回失败并记录 table overflow。`NUD_PERMANENT`、`NTF_EXT_LEARNED` 等特殊项会被排除在普通 GC 外。

第一性原理概述：邻居表缓存的是“局部链路上的近期通信对象”。它必须有容量边界，否则扫描、hash 冲突和未解析队列都可能被异常流量拖垮。

发送解析：从 `ip_finish_output2()` 到 ARP request

图 3：没有有效邻居项时，发送路径不会忙等，而是排队 skb 并触发 ARP 探测。

`ip_finish_output2()` 拿到 `rtable` 后调用 `ip_neigh_for_gw(rt, skb, &is_v6gw)`。route.c 的 `ipv4_neigh_lookup()` 会根据 `rt->rt_gw_family` 选择 gateway：有 IPv4 gateway 时查 `rt_gw4`，有 IPv6 gateway 时走 IPv6 stub，否则用报文目的地址作为直接下一跳。

`neigh_output()` 是真正的分流点：如果 `n->nud_state & NUD_CONNECTED` 且 `hh_cache` 可用，直接复制缓存二层头并 `dev_queue_xmit()`；否则调用 `n->output`。IPv4 普通 ARP 项的 `output` 是 `neigh_resolve_output()`，连接态的 `connected_output` 是 `neigh_connected_output()`；前者会先调用 `neigh_event_send()`，后者直接构造硬件头并发送。

ARP 接收：先校验协议头，再学习 sender，再决定是否回复

`arp_init()` 注册 `arp_packet_type` 后，设备收到 `ETH_P_ARP` 会进 `arp_rcv()`。`arp_rcv()` 丢弃 `IFF_NOARP`、OTHERHOST、LOOPBACK 包，检查 ARP 头长度与设备地址长度，再经 `NF_HOOK(NFPROTO_ARP, NF_ARP_IN, ..., arp_process)` 进入处理。

图 4：ARP 接收既可能驱动回复，也可能只更新已有邻居项。

`arp_process()` 严格过滤硬件类型、协议类型、opcode、loopback/multicast 目标和 gratuitous ARP 策略。对 request，它会用 `ip_route_input_noref()` 判断 `tip` 是否本机 `RTN_LOCAL`，或者是否可代理；本机地址还要经过 `arp_ignore()` 和 `arp_filter()`。如果需要回复，先通过 `neigh_event_ns(&arp_tbl, sha, &sip, dev)` 学习 sender，再 `arp_send_dst(ARPOP_REPLY, ...)`。

对 reply 或可接受的 gratuitous ARP，`arp_process()` 会查 `sip` 对应的邻居项；如果存在，调用 `neigh_update(n, sha, state, flags, 0)`。单播 reply 且目标是本机时状态为 `NUD_REACHABLE`；广播 reply 或 request 不证明双向可达，通常只更新为 `NUD_STALE`。

`neigh_update()`：学习地址、切换 output、释放队列

`__neigh_update()` 是所有学习路径的核心：ARP 收包、ioctl 设置、netlink 设置、探测失败都会进入这里。它先检查 dead、`NUD_NOARP/PERMANENT`、override 规则和 link-layer 地址合法性；然后根据新状态更新 timer、`ha`、`confirmed`、`updated`，并调用 `neigh_connect()` 或 `neigh_suspect()` 改变 `neigh->output`。

__neigh_update(neigh, lladdr, new, flags)
  -> 若 new 不再 NUD_VALID: 删除 timer，必要时 neigh_invalidate()
  -> 若 lladdr 变化且允许 override: 更新 neigh->ha 和 hh_cache
  -> new & NUD_CONNECTED: neigh_connect()    // output = connected_output
     否则: neigh_suspect()                   // output = output / resolve_output
  -> old 非 VALID 且 new 变 VALID:
       while skb = arp_queue.pop():
         dst_neigh_lookup_skb(dst, skb)
         n1->output(n1, skb)
  -> 发送 RTM_NEWNEIGH / NETEVENT_NEIGH_UPDATE 通知

这里有一个关键细节：释放队列时不是简单地直接用当前 `neigh` 输出，而是通过 `dst_neigh_lookup_skb()` 重新查 top-level neighbour。源码注释说明，shaper、eql、sch_teql 等路径可能在输出路径中使用不同 neighbour 对象，因此重新查一次避免把包送到错误对象。

用户管理：ioctl、rtnetlink、sysctl 三条控制面

控制面入口作用
ioctl`arp_ioctl()`、`SIOCSARP/SIOCDARP/SIOCGARP`设置、删除、查询 IPv4 ARP 映射；`ATF_PERM` 会变成 `NUD_PERMANENT`，`ATF_PUBL` 走 proxy ARP。
rtnetlink`RTM_NEWNEIGH/DELNEIGH/GETNEIGH` 在通用 rtnetlink/neighbour 层`ip neigh` 使用的主接口，状态和属性来自 `include/uapi/linux/neighbour.h`。
sysctl`/proc/sys/net/ipv4/neigh/*/*`调节探测次数、超时、未解析队列、GC 阈值、proxy delay 等。
设备事件`arp_netdev_event()``NETDEV_CHANGEADDR` 调 `neigh_changeaddr()`；carrier down 调 `neigh_carrier_down()`；IP down 后 `arp_ifdown()` 调 `neigh_ifdown()`。

用户态设置 permanent 项并不意味着它绕开 neighbour 框架；它仍是 `struct neighbour`，只是状态为 `NUD_PERMANENT`，不会被普通状态机和 GC 删除。相反，动态项必须接受 timer 和 GC 管理。

与路由子系统交互：FIB 定下一跳，neighbour 定二层地址

路由文档中的输出路径到 `__mkroute_output()` 后，会通过 `rt_set_nexthop()` 在 `rtable` 中记录 gateway 语义：如果 `fib_nh_common` 有 gateway 且 scope 是 link，就设置 `rt_uses_gateway`、`rt_gw_family`、`rt_gw4/rt_gw6`；否则 direct route 用目的地址作为邻居 key。随后 `ip_finish_output2()` 根据 `rtable` 查邻居并发送。

交互点源码意义
输出邻居查找`ip_finish_output2()`、`ip_neigh_for_gw()`、`ipv4_neigh_lookup()`把 `rtable` 中的 gateway 或目的地址映射到 `arp_tbl` 的 neighbour。
路径确认`dst_confirm_neigh()`、`ipv4_confirm_neigh()`、`sock_confirm_neigh()`上层有正反馈时更新 neighbour confirmed 时间，延长 REACHABLE。
多路径健康`fib_detect_death()`、`fib_good_nh()`、`sysctl_fib_multipath_use_neigh`ECMP 选择可参考 neighbour 状态，避免选择已经失败的 nexthop。
设备 down`fib_disable_ip()` 调 `arp_ifdown()`路由失效时同步清邻居缓存，避免旧二层映射继续使用。
地址/设备参数`devinet.c:inetdev_init()` 分配 `arp_parms`每个 IPv4 设备有自己的 ARP/neighbor 参数视图。
第一性原理概述：FIB 不能缓存 MAC，因为 MAC 属于输出设备所在链路；neighbor 也不能决定路由，因为它只知道本链路可达对象。两者通过 `rtable` 和 `dst_ops.neigh_lookup` 组合。

典型场景：从代码看行为

场景主路径设计含义
第一次发往直连 IP`ip_finish_output2 -> neigh_output -> neigh_resolve_output -> __neigh_event_send -> arp_solicit`先排队 skb，再发 ARP request；不会阻塞发送者等待 reply。
收到 ARP reply`arp_rcv -> arp_process -> neigh_update(REACHABLE)`更新 `ha`，切换到 connected output，并释放未解析队列。
收到 ARP request for 本机`arp_process -> ip_route_input_noref -> RTN_LOCAL -> neigh_event_ns -> arp_send_dst(REPLY)`回复前学习请求方 sender 映射，符合 RFC 826 的“先合并 sender 信息”思想。
STALE 项再次发送`neigh_event_send` 把 `NUD_STALE` 转 `NUD_DELAY`先允许使用旧地址发送，等待 TCP ACK/MSG_CONFIRM 等正反馈，避免每次 stale 都立即广播。
探测耗尽`neigh_timer_handler -> NUD_FAILED -> neigh_invalidate`丢弃 queued skb，调用 `arp_error_report()` / `dst_link_failure()` 通知上层。
设备 MAC 改变`arp_netdev_event(NETDEV_CHANGEADDR) -> neigh_changeaddr -> rt_cache_flush`二层头缓存与邻居项都要失效，否则会继续使用旧硬件地址。
设备 carrier down`arp_netdev_event(NETDEV_CHANGE) -> neigh_carrier_down`链路不可用时清理动态 neighbour，防止短时间内复用无效映射。
手工永久项`arp_ioctl(SIOCSARP) -> neigh_update(..., NUD_PERMANENT, ADMIN)`permanent 项不会走普通 NUD 与 GC，但仍通过 neighbour output 发送。

设计原则：为什么邻居层这样拆

按需解析

RFC 826 本身强调需要通信时再解析。Linux 在 `neigh_event_send()` 中只在有 skb 要发时进入 INCOMPLETE/PROBE，避免无意义探测。

状态表达可信度

`REACHABLE`、`STALE`、`DELAY`、`PROBE` 不是协议包类型，而是内核对映射可信度的估计。

队列解耦等待

解析未完成时 skb 进入 `arp_queue`，发送路径返回；ARP reply 到来后由 `neigh_update()` 重放队列。

正反馈优先

TCP ACK、`MSG_CONFIRM`、`dst_confirm_neigh()` 等能延长 confirmed 时间，减少 ARP 探测流量。

协议适配最小化

ARP 只实现 key、constructor、solicit、receive；hash、GC、timer、netlink 通知都复用通用 neighbour。

容量有边界

GC 阈值、未解析队列字节数、proxy 队列长度共同限制异常流量对内存和队列的影响。

整体总结

邻居子系统是三层路由结果落到二层发送的执行层。`arp_init()` 注册 IPv4 的 `arp_tbl`、ARP 包入口、sysctl 和设备事件；`neigh_table_init()` 建立通用 hash、GC、proxy queue 和默认参数；`devinet.c` 为每个 IPv4 设备分配自己的 `arp_parms`。

发送时，`ip_finish_output2()` 根据 `rtable` 找到下一跳 neighbour，`neigh_output()` 在 `NUD_CONNECTED + hh_cache` 条件下走最快路径，否则进入 `neigh_resolve_output()`。如果没有有效二层地址，`__neigh_event_send()` 把 skb 排入 `arp_queue`，转 `NUD_INCOMPLETE` 并调用 `arp_solicit()`。收到 ARP request/reply 后，`arp_process()` 学习 sender 并调用 `neigh_update()`；一旦状态变成有效，队列中的 skb 被重新送入输出路径。

状态机的核心是延迟确认和主动探测：`REACHABLE` 到期后先变 `STALE`，下一次使用变 `DELAY`，仍没有正反馈才进入 `PROBE`；探测耗尽后转 `FAILED` 并清队列。与 路由子系统文档 合起来看，FIB 选择下一跳，neighbour 证明下一跳当前是否能在链路层抵达。
