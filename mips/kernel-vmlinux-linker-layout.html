从链接脚本还原 MIPS Linux 内核内存布局与启动路径

这份文档分析当前 `vmlinux.lds.S` 在当前 `include/generated/autoconf.h` 配置下如何排布 `vmlinux`，并说明这个布局如何被 `head.S` 使用：ELF 入口指向 `kernel_entry`，早期汇编根据链接地址修正执行位置、选择 DTB、清零 `.bss`、建立初始栈，最后跳到 `start_kernel`。

官方依据

本文先获取 Linux 官方文档，再结合当前本地源码、生成配置和反汇编编写。外部链接只作为来源入口，页面本身不加载任何外部资源。

主题官方原文链接本文使用的事实
MIPS 启动约定BMIPS DeviceTree Booting官方文档说明 MIPS 启动可通过 ELF entry 或镜像起始入口进入，并描述 DT-aware bootloader 传递 DTB 的寄存器约定。
Kbuild 与链接脚本Linux Kernel Makefiles官方文档说明 `arch/$(SRCARCH)/kernel/vmlinux.lds.S` 会被预处理成 `vmlinux.lds`，并用于最终 `vmlinux` 链接。
启动期内存管理Boot time memory management官方文档说明早期启动不能使用普通页分配器，`setup_arch()` 建立 memblock，之后 `mem_init()` 释放给 buddy。
MIPS 文档入口MIPS-specific Documentation官方 MIPS 文档索引；本文中 X2600/MIPS32 地址细节以当前本地源码和构建参数为准。
第一性原理概述：链接脚本不是“格式描述文件”，而是内核启动契约：它决定入口符号、段顺序、运行地址、早期可释放区域、BSS 边界和调试/丢弃段。早期汇编依赖这些符号才能把 CPU 从加载状态带到 C 内核。

当前 autoconf 配置

用户给出的文件是 `include/generated/autoconf.h`。当前链接脚本中的 `#ifdef` 路径要按这个文件判断，而不是按通用 MIPS 内核推测。

/* include/generated/autoconf.h */
#define CONFIG_PAGE_SIZE_4KB 1
#define CONFIG_MIPS_NO_APPENDED_DTB 1
#define CONFIG_BUILTIN_DTB 1
#define CONFIG_USE_OF 1
#define CONFIG_LOADADDR 0x80010000
#define CONFIG_32BIT 1
#define CONFIG_SMP 1
#define CONFIG_CPU_MIPSR5 1
#define CONFIG_MIPS_L1_CACHE_SHIFT 5
#define CONFIG_INGENIC_BUILTIN_DTB 1
#define CONFIG_MIPS_CMDLINE_FROM_DTB 1
#define CONFIG_SOC_X2600 1
#define CONFIG_CPU_LITTLE_ENDIAN 1
/* compile_commands.json 中实际传入的构建宏 */
-DVMLINUX_LOAD_ADDRESS=0xffffffff80010000
-DLINKER_LOAD_ADDRESS=0x80010000
-DDATAOFFSET=0
配置/宏当前值对链接布局的影响
`CONFIG_32BIT` + `CONFIG_CPU_LITTLE_ENDIAN`启用`jiffies = jiffies_64`，采用 32 位 MIPS KSEG 地址模型，小端。
`CONFIG_LOADADDR``0x80010000`构建命令把它展开为 `LINKER_LOAD_ADDRESS=0x80010000`，链接脚本从该 VMA 开始排布。
`CONFIG_SMP`启用链接脚本包含 `PERCPU_SECTION(1 << 5)`，并且 `head.S` 编译出 `smp_bootstrap`。
`CONFIG_MIPS_L1_CACHE_SHIFT``5`cacheline 对齐粒度是 `1 << 5 = 32` 字节，用于 data/percpu/read_mostly 对齐。
`CONFIG_BUILTIN_DTB` + `CONFIG_USE_OF`启用`.init.data` 内会包含 `KERNEL_DTB()` 的 `.dtb.init.rodata`，`head.S` 会检查 `__dtb_start/__dtb_end`。
`CONFIG_MIPS_NO_APPENDED_DTB`启用`CONFIG_MIPS_ELF_APPENDED_DTB` 和 `CONFIG_MIPS_RAW_APPENDED_DTB` 未启用，`.appended_dtb` 和预留 1 MiB raw appended DTB 路径不进入当前布局。
`CONFIG_RELOCATABLE` / `CONFIG_MAPPED_KERNEL`未在当前 `autoconf.h` 定义不生成 `.data.reloc`；`DATAOFFSET=0`，`.data` 不为 mapped kernel 增加偏移。
第一性原理概述：同一份链接脚本包含很多体系结构变体。真正的内存布局是“链接脚本 + 预处理配置 + 构建命令宏”的结果，必须先把当前分支收窄。

地址模型：链接 VMA、物理加载与 KSEG0

/* arch/mips/include/asm/mach-generic/spaces.h */
#ifdef CONFIG_32BIT
#define CAC_BASE        _AC(0x80000000, UL)
#define UNCAC_BASE      _AC(0xa0000000, UL)
#define MAP_BASE        _AC(0xc0000000, UL)
#define HIGHMEM_START   _AC(0x20000000, UL)
#endif

#ifndef PAGE_OFFSET
#define PAGE_OFFSET     (CAC_BASE + PHYS_OFFSET)
#endif
/* arch/mips/include/asm/addrspace.h */
#define KUSEG           0x00000000
#define KSEG0           0x80000000
#define KSEG1           0xa0000000
#define KSEG2           0xc0000000
#define KSEG3           0xe0000000

#define KSEG0ADDR(a)    (CPHYSADDR(a) | KSEG0)
#define KSEG1ADDR(a)    (CPHYSADDR(a) | KSEG1)
/* arch/mips/include/asm/page.h */
if (!IS_ENABLED(CONFIG_EVA)) {
    /* standard MIPS32 legacy memory map, kseg0 or kseg1 */
    return CPHYSADDR(x);
}

#define __pa(x)     ___pa((unsigned long)(x))
#define __va(x)     ((void *)((unsigned long)(x) + PAGE_OFFSET - PHYS_OFFSET))

当前是 32 位非 EVA 的传统 MIPS 地址模型：`0x80000000..0x9fffffff` 是 KSEG0 cached direct map，`0xa0000000..0xbfffffff` 是 KSEG1 uncached direct map。链接脚本使用 `LINKER_LOAD_ADDRESS=0x80010000` 作为内核运行虚拟地址起点；按 KSEG0 规则，它对应低 512 MiB 物理地址窗口中的 `0x00010000` 偏移。

构建命令同时传入 `VMLINUX_LOAD_ADDRESS=0xffffffff80010000`，这是带符号扩展的 64 位形式；当前 `vmlinux.lds.S` 实际用于 `.` 的是 `LINKER_LOAD_ADDRESS=0x80010000`。这也解释了为什么 `head.disassemble` 中 `kernel_entry`、`start_kernel`、`__dtb_start`、`__bss_start` 都位于 `0x80xxxxxx` 的 KSEG0 虚拟地址范围。

第一性原理概述：链接脚本描述的是 CPU 执行时看到的虚拟地址；bootloader 或 objcopy 镜像关心的是加载地址。MIPS KSEG0 把二者用固定高位映射连接起来，早期启动还没有完整页表也能取指和访问内核数据。

入口与 PHDR：ELF 从哪里进入内核

/* arch/mips/kernel/vmlinux.lds.S */
#undef mips
#define mips mips
OUTPUT_ARCH(mips)
ENTRY(kernel_entry)
PHDRS {
    text PT_LOAD FLAGS(7);  /* RWX */
#ifndef CONFIG_CAVIUM_OCTEON_SOC
    note PT_NOTE FLAGS(4);  /* R__ */
#endif
}

`OUTPUT_ARCH(mips)` 指定 ELF 目标架构。`ENTRY(kernel_entry)` 把 ELF entry 设为 `head.S` 中的 `kernel_entry`，所以支持 ELF entry 的 bootloader 会直接跳到这个符号。当前 `CONFIG_BOOT_RAW` 没有在 `autoconf.h` 中定义，因此 `head.S` 中给 raw image 起始地址准备的 `__kernel_entry` 分支不属于当前构建主路径。

`text PT_LOAD FLAGS(7)` 表示主要内核镜像段是可读、可写、可执行的 load segment。`CONFIG_CAVIUM_OCTEON_SOC` 未定义，因此保留 `PT_NOTE`。这个 note program header 是给 ELF 元信息使用，不是内核运行时代码路径。

/* arch/mips/kernel/head.disassemble */
806caafc <kernel_entry>:
806caafc: 40086000  mfc0    t0,c0_status
...
806cabd0: 0822d9fd  j       808b67f4 <start_kernel>
第一性原理概述：启动入口必须是链接脚本和启动汇编共同约定的符号。链接脚本让 ELF header 指向 `kernel_entry`，`head.S` 则保证从这里能建立最早的 CPU 状态并进入 C 世界。

.text 与异常表：可执行主体和修复表

/* arch/mips/kernel/vmlinux.lds.S */
. = LINKER_LOAD_ADDRESS;
/* read-only */
_text = .;  /* Text and read-only data */
.text : {
    TEXT_TEXT
    SCHED_TEXT
    CPUIDLE_TEXT
    LOCK_TEXT
    KPROBES_TEXT
    IRQENTRY_TEXT
    SOFTIRQENTRY_TEXT
    *(.text.*)
    *(.fixup)
    *(.gnu.warning)
} :text = 0
_etext = .; /* End of text section */

EXCEPTION_TABLE(16)

__dbe_table : {
    __start___dbe_table = .;
    KEEP(*(__dbe_table))
    __stop___dbe_table = .;
}

`_text` 从 `LINKER_LOAD_ADDRESS` 开始，是内核链接布局的基准符号。`.text` 汇聚普通代码、调度器特殊代码、cpuidle、锁、kprobes、IRQ/softirq entry 代码以及 `.fixup`。`_etext` 标记可执行主体结束。

`EXCEPTION_TABLE(16)` 来自通用 `vmlinux.lds.h`，会生成 `__start___ex_table/__stop___ex_table`。异常表用于内核访问用户地址或可能 fault 的内存访问时做 fixup。紧随其后的 `__dbe_table` 是 MIPS 数据总线错误修复表。

第一性原理概述：`.text` 提供“CPU 要执行什么”，异常表提供“执行中发生可预期 fault 时怎么修复”。二者放在早期、连续、可加载区域，是为了启动和运行期都能快速定位。

rodata/data/gp：只读数据、可写数据和小数据窗口

/* arch/mips/kernel/vmlinux.lds.S */
_sdata = .;         /* Start of data section */
RO_DATA(4096)

.data : {           /* Data */
    . = . + DATAOFFSET;     /* for CONFIG_MAPPED_KERNEL */

    INIT_TASK_DATA(THREAD_SIZE)
    NOSAVE_DATA
    PAGE_ALIGNED_DATA(PAGE_SIZE)
    CACHELINE_ALIGNED_DATA(1 << CONFIG_MIPS_L1_CACHE_SHIFT)
    READ_MOSTLY_DATA(1 << CONFIG_MIPS_L1_CACHE_SHIFT)
    DATA_DATA
    CONSTRUCTORS
}
BUG_TABLE
_gp = . + 0x8000;
.lit8 : { *(.lit8) }
.lit4 : { *(.lit4) }
.sdata : { *(.sdata) }
_edata =  .;
/* include/asm-generic/vmlinux.lds.h */
#define INIT_TASK_DATA(align)                      \
    . = ALIGN(align);                              \
    __start_init_task = .;                         \
    init_thread_union = .;                         \
    init_stack = .;                                \
    KEEP(*(.data..init_task))                      \
    KEEP(*(.data..init_thread_info))               \
    . = __start_init_task + THREAD_SIZE;           \
    __end_init_task = .;

`RO_DATA(4096)` 生成 `.rodata`、`.rodata1`、PCI fixup 表、builtin firmware、符号表等只读或准只读区域。随后 `.data` 放可写数据，并且第一项就是 `INIT_TASK_DATA(THREAD_SIZE)`。这解释了 `head.S` 为什么能用 `init_thread_union` 建立启动栈。

当前 `DATAOFFSET=0`，所以 `.data` 不被额外平移。`CONFIG_MIPS_L1_CACHE_SHIFT=5`，cacheline 对齐是 32 字节。`_gp = . + 0x8000` 给 MIPS small data 访问建立 `$gp` 基准，使 `.sdata/.lit4/.lit8` 能在短偏移窗口中访问。

第一性原理概述：启动早期最重要的可写对象是初始 task/thread_info/stack。链接脚本把它固定放入 `.data` 的明确位置，`head.S` 才能在没有调度器、没有普通内存分配器时获得第一条内核栈。

init/percpu/DTB：启动后可释放区域和每 CPU 区域

/* arch/mips/kernel/vmlinux.lds.S */
. = ALIGN(PAGE_SIZE);
__init_begin = .;
INIT_TEXT_SECTION(PAGE_SIZE)
INIT_DATA_SECTION(16)

. = ALIGN(4);
.mips.machines.init : AT(ADDR(.mips.machines.init) - LOAD_OFFSET) {
    __mips_machines_start = .;
    KEEP(*(.mips.machines.init))
    __mips_machines_end = .;
}

.exit.text : { EXIT_TEXT }
.exit.data : { EXIT_DATA }
#ifdef CONFIG_SMP
PERCPU_SECTION(1 << CONFIG_MIPS_L1_CACHE_SHIFT)
#endif
/* include/asm-generic/vmlinux.lds.h */
#define KERNEL_DTB()                    \
    STRUCT_ALIGN();                     \
    __dtb_start = .;                    \
    KEEP(*(.dtb.init.rodata))           \
    __dtb_end = .;

#define INIT_DATA                       \
    ...                                 \
    CLK_OF_TABLES()                     \
    RESERVEDMEM_OF_TABLES()             \
    TIMER_OF_TABLES()                   \
    CPU_METHOD_OF_TABLES()              \
    CPUIDLE_METHOD_OF_TABLES()          \
    KERNEL_DTB()                        \
    IRQCHIP_OF_MATCH_TABLE()            \
    ...

`__init_begin` 到 `__init_end` 之间是启动初始化区域。`.init.text/.init.data` 中包含 initcall、early param、OF match table、reserved memory table、内置 DTB 等启动阶段需要但运行期可释放的数据。当前启用 `CONFIG_BUILTIN_DTB` 与 `CONFIG_INGENIC_BUILTIN_DTB`，所以 `__dtb_start/__dtb_end` 非空；反汇编显示 `__dtb_start=0x808ec300`，`__dtb_end=0x808f1640`。

当前启用 `CONFIG_SMP`，所以 `PERCPU_SECTION(32)` 进入布局。通用宏会按 `PAGE_SIZE` 对齐，生成 `.data..percpu`，并定义 `__per_cpu_load`。这块区域提供每 CPU 变量的初始模板，SMP 初始化阶段会据此建立每个 CPU 的实例。

第一性原理概述：init 区域服务“把系统带起来”，percpu 区域服务“多 CPU 运行时隔离”。链接脚本把两者分开，是为了 boot 后能释放 init 内存，同时保留运行期必须存在的 per-CPU 数据。

bss 与调试段：零初始化边界和非运行期信息

/* arch/mips/kernel/vmlinux.lds.S */
. = ALIGN(0x10000);
__init_end = .;
/* freed after init ends here */

BSS_SECTION(0, 0x10000, 8)

_end = . ;

.mdebug.abi32 : { KEEP(*(.mdebug.abi32)) }
.mdebug.abi64 : { KEEP(*(.mdebug.abi64)) }
.mdebug : { *(.mdebug) }

STABS_DEBUG
DWARF_DEBUG
ELF_DETAILS

/DISCARD/ : {
    *(.MIPS.abiflags)
    *(.MIPS.options)
    *(.options)
    *(.pdr)
    *(.reginfo)
}
/* include/asm-generic/vmlinux.lds.h */
#define BSS_SECTION(sbss_align, bss_align, stop_align)  \
    . = ALIGN(sbss_align);                              \
    __bss_start = .;                                    \
    SBSS(sbss_align)                                    \
    BSS(bss_align)                                      \
    . = ALIGN(stop_align);                              \
    __bss_stop = .;

当前链接脚本在 BSS 前强制 `ALIGN(0x10000)`，并且 `BSS_FIRST_SECTIONS` 把 `.bss..swapper_pg_dir` 放在 `.bss` 的最前面。注释说明这样做是为了让 `swapper_pg_dir` 拿到 64 KiB 对齐。当前 `head.disassemble` 显示 `__bss_start=0x80900000`，`__bss_stop-4=0x8092151c`，启动汇编会逐 word 清零这个范围。

`_end` 之后是调试和 ELF 元信息段，大量 DWARF/STABS 段被放到地址 0 的调试空间，`.MIPS.abiflags/.reginfo` 等 ABI 元信息被丢弃。这些段帮助调试或满足工具链，但不是内核运行期内存主体。

第一性原理概述：BSS 不占镜像文件中的初始化数据，但运行时必须是 0。链接脚本给出 `__bss_start/__bss_stop`，`head.S` 在进入 C 代码前清零它，从而让所有未初始化全局变量满足 C 语言语义。

head.S 如何使用链接布局

/* arch/mips/kernel/head.S */
NESTED(kernel_entry, 16, sp)            # kernel entry point

    kernel_entry_setup                  # cpu specific setup

    setup_c0_status_pri

    /* We might not get launched at the address the kernel is linked to,
       so we jump there.  */
    PTR_LA  t0, 0f
    jr      t0
0:

#ifdef CONFIG_USE_OF
    li      t1, -2
    move    t2, a1
    beq     a0, t1, dtb_found

#ifdef CONFIG_BUILTIN_DTB
    PTR_LA  t2, __dtb_start
    PTR_LA  t1, __dtb_end
    bne     t1, t2, dtb_found
#endif

    li      t2, 0
dtb_found:
#endif
    PTR_LA  t0, __bss_start     # clear .bss
    LONG_S  zero, (t0)
    PTR_LA  t1, __bss_stop - LONGSIZE
1:
    PTR_ADDIU   t0, LONGSIZE
    LONG_S      zero, (t0)
    bne         t0, t1, 1b
/* arch/mips/kernel/head.S */
    LONG_S  a0, fw_arg0         # firmware arguments
    LONG_S  a1, fw_arg1
    LONG_S  a2, fw_arg2
    LONG_S  a3, fw_arg3

#ifdef CONFIG_USE_OF
    LONG_S  t2, fw_passed_dtb
#endif

    MTC0    zero, CP0_CONTEXT   # clear context register
    PTR_LA  $28, init_thread_union
    /* Set the SP after an empty pt_regs.  */
    PTR_LI  sp, _THREAD_SIZE - 32 - PT_SIZE
    PTR_ADDU    sp, $28
    back_to_back_c0_hazard
    set_saved_sp    sp, t0, t1
    PTR_SUBU    sp, 4 * SZREG   # init stack pointer

#ifdef CONFIG_RELOCATABLE
    jal     relocate_kernel
    jr.hb   v0
#else
    j       start_kernel
#endif

当前 `CONFIG_RELOCATABLE` 未定义，因此反汇编中直接跳 `start_kernel`，没有 `.data.reloc` 和 `relocate_kernel` 过程。当前 `CONFIG_BUILTIN_DTB` 启用，反汇编中确实加载 `__dtb_start=0x808ec300`、`__dtb_end=0x808f1640` 并保存到 `fw_passed_dtb`。

/* arch/mips/kernel/head.disassemble */

806cab3c: 254ac300  addiu   t2,t2,-15616  # __dtb_start = 0x808ec300
806cab40: 3c09808f  lui     t1,0x808f
806cab44: 25291640  addiu   t1,t1,5696    # __dtb_end   = 0x808f1640

806cab54: 3c088090  lui     t0,0x8090     # __bss_start = 0x80900000
806cab60: 3c098092  lui     t1,0x8092
806cab64: 2529151c  addiu   t1,t1,5404    # __bss_stop - 4 = 0x8092151c

806caba4: 3c1c8085  lui     gp,0x8085
806caba8: 279c2000  addiu   gp,gp,8192    # init_thread_union = 0x80852000
806cabac: 241d1f28  li      sp,7976       # 8192 - 32 - PT_SIZE(184)

第一性原理概述：`head.S` 中每个关键地址都来自链接脚本符号。没有这些符号，汇编代码无法知道 BSS 在哪里、内置 DTB 在哪里、第一条内核栈在哪里，也无法安全进入 `start_kernel`。

/* arch/mips/xburst2/core/prom.c */
void __init prom_init(void)
{
    prom_init_cmdline((int)fw_arg0, (char **)fw_arg1);

    if (fw_arg0 == 0 && fw_arg1 == 0xffffffffUL)
        _fw_fdt_addr = phys_to_virt(fw_arg2);
    else if ((int)fw_arg0 == -2)    /*UHI*/
        _fw_fdt_addr = (void *)fw_arg1;
    else if ((void *)__dtb_start != (void *)__dtb_end)
        _fw_fdt_addr = __dtb_start;
    else
        panic("no dtb found!\n");

    mips_machtype = MACH_INGENIC_X2000;
#ifdef CONFIG_SMP
    register_smp_ops(&xburst2_smp_ops);
#endif
}

官方 memblock 文档说明，早期启动不能使用普通页分配器，体系结构代码要在 `setup_arch()` 中建立 memblock，后续 `mem_init()` 再把空闲页交给 buddy。链接脚本提供的是“内核镜像自身占用的静态边界”，memblock 则把这些边界与 DT/firmware 提供的物理内存范围结合起来，形成启动期内存分配视图。

这也是为什么内置 DTB 放在 init 区域、BSS 在进入 C 前清零、`__init_begin/__init_end` 明确标出：启动期需要先使用这些静态对象完成探测和初始化，随后释放不再需要的 init 内存，把系统从静态链接布局过渡到动态页分配。

第一性原理概述：链接脚本解决“内核自己在哪里”；memblock 解决“可用物理内存在哪里、哪些已经被占用”。`start_kernel` 之后，系统逐步从链接器静态世界切换到运行期内存管理世界。

总结

当前 X2600/MIPS32 内核的链接布局从 `LINKER_LOAD_ADDRESS=0x80010000` 开始，在 KSEG0 cached direct map 中按 `.text -> exception tables -> rodata -> data -> init -> percpu -> bss -> debug/discard` 排列。由于 `CONFIG_SMP=y`，存在 percpu 初始模板；由于 `CONFIG_BUILTIN_DTB=y` 且 `CONFIG_MIPS_NO_APPENDED_DTB=y`，当前 DTB 走 `.dtb.init.rodata` 的 `__dtb_start/__dtb_end` 路径，而不是 appended DTB；由于 `CONFIG_RELOCATABLE` 未定义，早期入口直接跳 `start_kernel`。

内核启动依赖这套布局：`ENTRY(kernel_entry)` 决定 ELF 入口，`head.S` 跳到链接地址，使用 `__dtb_start/__dtb_end` 选择 DTB，使用 `__bss_start/__bss_stop` 清零 BSS，使用 `init_thread_union` 和 `_THREAD_SIZE/PT_SIZE` 建立初始栈，最后进入 `start_kernel`。随后 `setup_arch()` 和 memblock 根据 DT/firmware 内存信息建立启动期内存管理，再进入分页、SMP 和普通内核初始化。

第一性原理概述：链接脚本把“文件中的内核”变成“CPU 可执行的地址空间”；启动汇编把“链接好的地址空间”变成“可运行的 C 内核”；memblock 再把“内核自身占用”放进整机物理内存视图中。三者连起来，就是 Linux 内核从镜像到运行系统的第一段生命线。
