IPv4 路由子系统设计：初始化、查询与跨子系统协作

本文关注当前源码树中的 IPv4 路由/FIB。它不是简单的“路由表数组”，而是一套控制面和数据面分离的索引系统：控制面通过 rtnetlink、地址事件和设备事件维护 FIB；数据面把 packet/socket 转成 `flowi4`，查 FIB 得到 `fib_result`，再构造 `rtable/dst` 给 IP 层、邻居层、XFRM 和上层协议使用。

核心模型：控制面维护 FIB，数据面消费 rtable

图 1：控制面只更新 FIB；数据面查 FIB 后构造 `rtable/dst`，后续转发/本地投递依赖 dst 回调。

当前树的顶层入口在 `net/ipv4/route.c:ip_rt_init()`。它先初始化 `rtable` 的 slab、dst 计数、uncached list 等数据面设施，然后调用 `devinet_init()` 和 `ip_fib_init()`。这说明 IPv4 路由初始化不是孤立行为：地址子系统、FIB、dst 缓存、proc/sysctl、XFRM 都在 IPv4 route 初始化序列里被接起来。

第一性原理概述：FIB 是慢变索引，`rtable/dst` 是一次查找后的快速执行对象。内核把“如何查”与“查完怎么发/收”拆开，可以让查表结构、邻居解析、PMTU、XFRM 和轻量隧道各自演进。

核心对象：从表到路径的四级抽象

对象所在层级作用
`struct net` 的 `net->ipv4`网络命名空间保存本 netns 的 `fib_table_hash`、`fib_main`/`fib_default`、notifier ops、FIB 序列号、route genid、sysctl 等。
`struct fib_table`路由表一张逻辑表，如 `RT_TABLE_LOCAL`、`RT_TABLE_MAIN`、`RT_TABLE_DEFAULT` 或 VRF 表；表内数据由 trie 承载。
`fib_trie` / leaf / `fib_alias`前缀索引trie 负责最长前缀匹配；同一个前缀下的 `fib_alias` 区分 TOS、type、scope、priority 与 `fib_info`。
`struct fib_info` / `fib_nh_common`路径语义保存 prefsrc、metric、nexthop、设备、gateway、lwtunnel、multipath 等真正影响转发的语义。
`struct fib_result`查询结果把表、route type、scope、prefixlen、selected nexthop、`fib_info` 返回给 route.c。
`struct rtable` / `struct dst_entry`执行对象承载 `dst.input`、`dst.output`、PMTU、cache、XFRM/lwtunnel 指针，挂到 skb 或 socket 路径上。

这四级抽象的关键收益是复用：多条前缀可以共享 `fib_info`，同一前缀可以挂多个 `fib_alias`，一次 FIB 查找可以被缓存为 `rtable`，而 netns 让容器/VRF 场景不共享路由表状态。

初始化顶层：`ip_rt_init()` 先搭数据面，再拉 FIB

`ip_rt_init()` 的顺序可以概括为五步：创建 IP ID hash；初始化 per-cpu uncached `rtable` 链表；创建 `ip_dst_cache` slab 与 `ipv4_dst_ops` 计数；初始化地址与 FIB；最后注册 route proc、XFRM、`RTM_GETROUTE`、route sysctl、route genid 和 inetpeer。

ip_rt_init()
  -> kmem_cache_create("ip_dst_cache", sizeof(struct rtable), ...)
  -> dst_entries_init(&ipv4_dst_ops)
  -> devinet_init()
  -> ip_fib_init()
  -> ip_rt_proc_init()
  -> xfrm_init() / xfrm4_init()                 // CONFIG_XFRM
  -> rtnl_register(PF_INET, RTM_GETROUTE, inet_rtm_getroute, ...)
  -> register_pernet_subsys(sysctl_route_ops / rt_genid_ops / inetpeer)
注意这里的 `RTM_GETROUTE` 注册在 `route.c`，而 `RTM_NEWROUTE`、`RTM_DELROUTE`、FIB dump 注册在 `fib_frontend.c:ip_fib_init()`。这反映了控制面分工：修改 FIB 属于 FIB frontend，按实际转发语义解析一条路由属于 route resolver。

每网络命名空间：`ip_fib_net_init()` 创建表和通知能力

图 2：`ip_fib_init()` 负责全局 trie slab、per-netns FIB、事件订阅和 route netlink 注册。

`ip_fib_net_init()` 先调用 `fib4_notifier_init()`，让本 netns 具备 FIB 变更通知能力；随后分配 `net->ipv4.fib_table_hash`，再调用 `fib4_rules_init()`。在这份源码中，未发现独立的 IPv4 `fib_rules.c` 实现文件，`fib_frontend.c` 中的非 multiple-tables 分支直接创建 `RT_TABLE_MAIN` 与 `RT_TABLE_LOCAL` 两张 trie 表并挂入 hash。头文件 `include/net/ip_fib.h` 同时保留了 `CONFIG_IP_MULTIPLE_TABLES` 分支：若启用策略路由，`fib_lookup()` 会在有自定义规则时转入 `__fib_lookup()`。

`fib_net_init()` 还创建 `NETLINK_FIB_LOOKUP` socket 和 proc 展示节点；退出时 `ip_fib_net_exit()` 会在 rtnl 锁下逆序销毁表，特别保证 local 表在 main 表之前销毁，因为 local 表可能引用 main 表中的数据。

表与 trie：逻辑表是容器，LC-trie 是查找结构

`fib_trie_init()` 只创建两类 slab：`ip_fib_alias` 和 `ip_fib_trie`。真正的表由 `fib_trie_table(id, alias)` 创建：`struct fib_table` 后面跟一块 trie 数据，或者在 local/main alias 场景中共享已有 trie 数据。新 trie 的根 `key_vector` 初始化为 `pos = KEYLENGTH`、`slen = KEYLENGTH`，这是空表的根。

表常见内容谁维护
`RT_TABLE_LOCAL``RTN_LOCAL` 主机路由、`RTN_BROADCAST` 广播路由内核地址事件自动维护，通常不建议用户直接改。
`RT_TABLE_MAIN`普通 unicast、connected prefix、默认路由用户 `ip route`、地址自动直连路由、路由守护进程。
`RT_TABLE_DEFAULT`默认后处理表策略路由配置可能使用；当前树的简单路径先查 main，再查 default。
VRF/L3 master 表属于某个 L3 domain 的路由`l3mdev_fib_table()` 参与表选择，设备上下级变化触发 flush。
第一性原理概述：路由表是隔离边界，trie 是算法结构。表决定“哪些策略/命名空间能看见这批 route”，trie 决定“怎么快速找到最长匹配前缀”。

控制面入口：rtnetlink、地址事件、设备事件

`ip_fib_init()` 注册三个重要入口：`RTM_NEWROUTE -> inet_rtm_newroute()`、`RTM_DELROUTE -> inet_rtm_delroute()`、`RTM_GETROUTE dump -> inet_dump_fib()`。`route.c` 另外注册 `RTM_GETROUTE doit -> inet_rtm_getroute()`，用于模拟一次实际 lookup。

手工 route 修改大致是：rtnetlink 消息转成 `fib_config`，校验 route type、scope、table、nexthop，再由 `fib_table_insert()` 或 `fib_table_delete()` 修改 trie。地址自动路由则来自 `fib_inetaddr_event()`；设备 up/down、carrier 变化、MTU 变化、VRF 上下级变化则来自 `fib_netdev_event()`。

事件源进入函数影响
用户 `ip route add/del``inet_rtm_newroute()` / `inet_rtm_delroute()`显式增删 FIB 表项，可能触发 FIB notifier 和 route cache/genid 刷新。
用户 `ip route get``inet_rtm_getroute()`构造一次输入或输出 lookup，返回解析后的 `dst/rtable` 语义。
地址新增/删除`fib_inetaddr_event()`调用 `fib_add_ifaddr()` / `fib_del_ifaddr()`，维护 local、broadcast、connected route。
设备 up/down/change/MTU/VRF`fib_netdev_event()`同步 nexthop dead/linkdown、flush 受影响路由、更新 MTU 与 ARP 状态。
FIB 监听者`fib_notifier.c``call_fib4_notifiers()` 维护 `fib_seq`，向关注者广播 route/rule dump 与变更。

查询模型：`flowi4` 是查询请求，`fib_result` 是答案

路由查询不会只看目的地址。输入路径会填 `flowi4_iif`、mark、TOS、隧道 ID、源/目的地址，必要时通过 `fib4_rules_early_flow_dissect()` 提取 L4 端口；输出路径还会带 `flowi4_oif`、socket 选择的源地址、uid、protocol、端口等。策略路由启用时这些字段会参与 rule 匹配；简单路径下主要用于 FIB trie 的 TOS/scope/nexthop 过滤和后续源地址选择。

图 3：FIB 只回答“匹配到什么路径”；route.c 负责把路径变成可执行的 `rtable`。

`include/net/ip_fib.h` 中的 `fib_lookup()` 在当前简单路径下直接查 `net->ipv4.fib_main`，失败后再查 `fib_default`；如果返回 `-EAGAIN` 则转成 `-ENETUNREACH`。在 `CONFIG_IP_MULTIPLE_TABLES` 分支中，只有 `net->ipv4.fib_has_custom_rules` 为真时才进入 `__fib_lookup()`，否则仍走 main/default 的快速路径。

输入路径：`ip_route_input_slow()` 决定本地接收还是转发

输入路径的核心入口是 `ip_route_input_rcu()`，它先特殊处理 multicast，再进入 `ip_route_input_slow()`。slow path 会执行一系列 martian 源/目的检查，填充 `flowi4`，然后调用 `fib_lookup()`。

ip_route_input_rcu(skb, daddr, saddr, tos, dev, res)
  -> multicast? ip_route_input_mc()
  -> ip_route_input_slow()
       -> 检查 in_device、loopback、zeronet、limited broadcast
       -> flowi4_iif = dev->ifindex, mark = skb->mark, daddr/saddr/tos
       -> fib_lookup(net, &fl4, res, 0)
       -> RTN_BROADCAST: 本地广播输入或受控转发
       -> RTN_LOCAL: fib_validate_source() 后 local_input
       -> RTN_UNICAST: 若允许转发，ip_mkroute_input()
       -> 无路由/非法源: RTN_UNREACHABLE 或 martian

`fib_validate_source()` 是路由与反向路径/源地址校验的交汇点。它会再次基于源地址构造 lookup，用来判断源是否合理、是否满足 `rp_filter`、是否应该标记 martian。这就是为什么源地址合法性不是地址子系统单独能决定的：它必须结合入接口、路由策略和反向可达性。

输出路径：`ip_route_output_key_hash()` 选择源地址、nexthop 和 dst

本地发送路径从 socket、ICMP、隧道、netfilter 等处构造 `flowi4`，进入 `ip_route_output_key_hash()` / `ip_route_output_key_hash_rcu()`。它会先处理显式源地址、显式 oif、multicast/broadcast、缺省目的地址等边界，再调用 `fib_lookup()`。

查到 `RTN_LOCAL` 时，输出设备会切到 loopback 或 L3 master，因为目的在本机；查到普通 unicast 时调用 `fib_select_path()`。`fib_select_path()` 在 multipath 下根据 hash 选择 nexthop，在多个默认路由下调用 `fib_select_default()`，最后如果 `fl4->saddr` 为空则用 `fib_result_prefsrc()` 选择首选源地址。

ip_route_output_key_hash_rcu(net, fl4, res, skb)
  -> 校验/补全 saddr、oif、daddr
  -> fib_lookup(net, fl4, res, 0)
  -> RTN_LOCAL: dev_out = loopback 或 l3 master
  -> RTN_UNICAST: fib_select_path()
  -> __mkroute_output()
       -> rt_dst_alloc()
       -> rt_set_nexthop()
       -> lwtunnel_set_redirect()
第一性原理概述：输出查找不只是“下一跳是谁”，还要回答“用哪个源地址、是否本地回环、是否走 VRF、是否可缓存、是否需要 lwtunnel/XFRM”。这些答案最终都被折叠进 `rtable/dst`。

LC-trie 查询：最长前缀匹配加语义过滤

`fib_table_lookup()` 的三步与官方 LC-trie 文档一致：先沿 key 的比特片段下降到候选最长匹配节点；如果失败则回溯并逐步剥掉 child index 的最低有效 1，寻找更短前缀；找到 leaf 后遍历 `fib_alias`，按 prefix、TOS、dead flag、scope、route type、nexthop 状态等做语义匹配。

阶段代码意图为什么需要
下降`get_cindex(key, n)` 选择 child，遇到 leaf 则准备匹配。快速逼近最长可能前缀，利用 path/level compression 减少节点层数。
回溯prefix mismatch 或 child 缺失时，向上寻找更短匹配。最长前缀匹配要求 `/24` 不存在时能退到 `/16`、`/8`、`0/0`。
alias 语义匹配检查 `fib_alias` 的 TOS、scope、type、`fib_info`、nexthop。同一前缀可以有多条语义不同的路由，最长前缀不是唯一决策条件。
结果填充设置 `res->prefix`、`prefixlen`、`nh_sel`、`nhc`、`type`、`scope`、`fi`、`table`。route.c 不需要知道 trie 内部结构，只消费标准化结果。

nexthop 选择本身有两层：trie 阶段用 `fib_lookup_good_nhc()` 排除 dead/linkdown 或与 `flowi4_oif` 不匹配的 nexthop；输出阶段再用 `fib_select_path()` 处理 multipath hash、默认路由选择和 prefsrc。

与其他子系统的交互边界

地址子系统

`devinet.c` 维护 `in_ifaddr`，通过 `inetaddr_chain` 触发 `fib_inetaddr_event()`。这部分已在 ipv4-address-route-interaction.html 详细分析：地址是事实源，FIB 是派生索引。

netdevice

`fib_netdev_event()` 监听 `NETDEV_UP/DOWN/CHANGE/CHANGEMTU/UNREGISTER/CHANGEUPPER`，同步 nexthop linkdown/dead、flush route cache、更新 MTU，VRF 上下级变化会强制禁用相关 IP 路由。

rtnetlink

用户态 `ip route` 通过 `RTM_NEWROUTE`、`RTM_DELROUTE` 修改 FIB，通过 `RTM_GETROUTE` dump 或 resolver 获取路由。rtnl 锁提供控制面串行化。

netns / VRF

pernet ops 为每个 `struct net` 创建独立 FIB；`l3mdev_fib_table()`、`l3mdev_master_dev_rcu()` 把 VRF/L3 master 变成表选择和本地回环语义的一部分。

邻居/ARP

route 选择输出设备和 gateway，真正的二层解析交给 ARP/neighbor。`fib_disable_ip()` 在设备禁用时调用 `arp_ifdown()`，避免邻居缓存继续相信旧链路状态。

XFRM / lwtunnel

`ip_rt_init()` 初始化 XFRM；`rtable` 的 `dst` 可携带 lwtunnel state。路由查找先决定基础路径，再由 XFRM/lwtunnel 扩展封装、安全策略或重定向。

dst/rtable 缓存

FIB 是配置索引，`rtable` 是执行缓存。地址、设备、sysctl、route 修改会调用 `rt_cache_flush()` 或提升 genid，使旧 dst 失效或不再复用。

sysctl 与策略

`rp_filter` 影响 `fib_validate_source()`，`ignore_routes_with_linkdown` 影响 nexthop 可用性，`fib_multipath_*` 影响 ECMP hash 和邻居健康过滤。

完整 MSC：一次路由从配置到转发

图 4：控制面修改的是 FIB；数据面通过 `flowi4` 查询 FIB，再由 route.c 生成 `dst`。

如果配置来自地址变化，时序中的第一步不是 `RTM_NEWROUTE`，而是 `RTM_NEWADDR -> __inet_insert_ifa() -> inetaddr_chain -> fib_inetaddr_event()`。FIB frontend 仍然是路由派生和表项更新的归口。

典型场景：从设计看行为

场景主路径设计含义
添加默认路由`RTM_NEWROUTE -> inet_rtm_newroute -> fib_create_info -> fib_table_insert`默认路由是 `0/0` 前缀，trie 查找时只有更长前缀缺失才会回退到它。
网卡 down`fib_netdev_event(NETDEV_DOWN) -> fib_disable_ip -> fib_sync_down_dev/fib_flush -> arp_ifdown`路由、rtable 和邻居缓存必须一起失效，否则会继续向不可用链路发包。
carrier 变化`NETDEV_CHANGE -> fib_sync_up/fib_sync_down_dev -> rt_cache_flush`linkdown 不是删除 route，而是标记 nexthop 状态；查询可按 sysctl 忽略或接受 linkdown nexthop。
MTU 变化`NETDEV_CHANGEMTU -> fib_sync_mtu -> rt_cache_flush`PMTU 和 advmss 等执行属性依赖设备 MTU，旧 `rtable` 不能继续无条件复用。
网关路由校验`fib_check_nh_v4_gw -> fib_table_lookup/fib_lookup`添加 gateway route 时，gateway 自身必须可达；`onlink` 是用户显式声明“当作直连”。
多路径路由`fib_table_lookup -> fib_select_path -> fib_select_multipath`trie 找到 route，真正 nexthop 由 hash、权重、邻居健康和 sysctl 决定。
VRF 设备加入/退出`NETDEV_CHANGEUPPER -> fib_disable_ip(force)`设备所属 L3 domain 改变后，原表中的派生路由不再可信，需要强制刷新。
源地址校验`ip_route_input_slow -> fib_validate_source -> fib_lookup`反向路径校验使用 FIB 本身，因此路由策略也会影响收包合法性判断。

设计原则：为什么这样拆

控制面/数据面分离

rtnetlink、地址、设备事件更新 FIB；收发包只消费 `fib_lookup()` 与 `rtable`。这样更新路径可以加锁、通知和校验，而 fast path 只保留必要判断。

表隔离，trie 加速

netns、VRF 和策略路由需要多张逻辑表；每张表内的最长前缀匹配由 LC-trie 处理，避免线性扫描。

语义晚绑定

前缀匹配只确定候选集合；type、scope、TOS、nexthop 状态、multipath、prefsrc 在后续阶段过滤和选择。

事件驱动一致性

地址和设备不是 FIB 的内部字段，但它们变化会使 route 语义失效，所以通过 notifier 与 flush/genid 维持一致性。

第一性原理概述：路由系统的本质是缓存和索引一组不断变化的网络事实。设计重点不是把事实复制到一个地方，而是在事实变化时准确失效，在包到来时快速求解。

整体总结

从顶层看，IPv4 路由子系统由 `ip_rt_init()` 接入 IPv4 栈，`ip_fib_init()` 拉起 FIB trie、per-netns 表、netdevice/inetaddr notifier 和 route netlink 修改入口。表项来自三类来源：用户/路由协议通过 rtnetlink 显式配置，地址子系统自动派生 local/connected/broadcast route，设备事件驱动 nexthop 与 cache 状态同步。

从查询看，输入和输出路径都先构造 `flowi4`，再调用 `fib_lookup()`。当前源码的普通路径查 main/default 表；若启用 multiple tables 且存在自定义规则，则进入策略路由分支。表内由 `fib_table_lookup()` 执行 LC-trie 最长前缀匹配，并用 `fib_alias`、`fib_info`、nexthop 状态做语义过滤。最终 `route.c` 把 `fib_result` 变成 `rtable/dst`，交给 IP 输入、输出、转发、邻居解析、XFRM 和轻量隧道继续处理。

与 地址与路由交互文档 对照看：地址页解释“地址事实如何派生 route”，本文解释“route 子系统如何存储、查询、失效并服务数据面”。二者合起来就是 IPv4 可达性从配置到转发的完整闭环。
