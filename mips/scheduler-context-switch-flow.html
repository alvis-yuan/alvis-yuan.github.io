进程调度与上下文切换：从调度时机到底层寄存器保存

调度不是一个单点动作，而是“产生重新调度需求 -> 在安全边界进入 `schedule()` -> 在 runqueue 上选择 next -> 切换 mm 和 CPU 现场 -> 从 next 的内核栈继续执行”。本文以当前 X2600 MIPS 内核源码为准，说明什么时候会调度、`__schedule()` 如何走到 `context_switch()`，以及切换时哪些信息保存在哪里。

架构切换约束sched-arch默认 `switch_to` 在 runqueue lock 持有期间被调用；架构代码必须实现保存 prev 与装载 next 的低层上下文切换。
第一性原理概述：调度核心只决定“下一个谁运行”和维护 runqueue 一致性；真正让 CPU 变成另一个任务现场的是架构层 `switch_to()`。

调度时机：什么时候会进入调度器

当前源码在 `__schedule()` 的注释里直接列出三类驱动方式：显式阻塞、返回路径检查 `TIF_NEED_RESCHED`、以及 wakeup 导致的重新调度标志。wakeup 本身只是把任务放回运行队列；如果它应抢占当前任务，才设置 `TIF_NEED_RESCHED`，等最近的安全点调用 `schedule()`。

时机本地源码入口发生了什么
主动睡眠/阻塞`mutex`、`waitqueue`、`schedule()`当前任务先把 `state` 设为非运行态，然后显式调用 `schedule()`。`__schedule(false)` 会把它从 runqueue 摘下。
时钟 tick 抢占`scheduler_tick()`调用 `curr->sched_class->task_tick()` 更新当前任务运行时间；调度类可调用 `resched_curr()` 标记需要调度。
wakeup 抢占`wake_up_new_task()`/唤醒路径新 runnable task 入队后，`check_preempt_curr()` 判断是否应该抢占当前 task，必要时设置 `TIF_NEED_RESCHED`。
返回用户态`arch/mips/kernel/entry.S: resume_userspace`返回前检查 `_TIF_WORK_MASK`，其中包含 `_TIF_NEED_RESCHED`，命中后跳到 `work_resched` 调用 `schedule()`。
内核抢占`preempt_schedule()`/`preempt_schedule_irq()`在可抢占内核中，最外层 `preempt_enable()` 或 IRQ 返回到可抢占上下文时进入 `__schedule(true)`。
idle 循环`schedule_idle()`idle 任务发现 `need_resched()` 后让出 CPU，调度真正 runnable 的任务。
图 1：抢占类调度先设置标志，阻塞类调度直接进入 `schedule()`。
第一性原理概述：内核不能在任意指令中间随意换栈换寄存器。抢占先变成一个标志，再在中断返回、系统调用返回、preempt enable、显式睡眠这些边界真正切换。

从顶层到底层的总流程

顶层 C 代码负责 runqueue、任务状态、调度类和 mm 切换；底层汇编负责通用寄存器、栈指针、返回地址、CP0 状态和每线程特殊寄存器。两层的交界点是 `context_switch()` 里的 `switch_to(prev, next, prev)`。

第一性原理概述：调度算法决定 CPU 应该给谁；上下文切换机制保证 CPU 能从那个人上次停下的地方继续执行。

`need_resched`：抢占请求保存在线程标志里

/* kernel/sched/core.c */
void resched_curr(struct rq *rq)
{
        struct task_struct *curr = rq->curr;
        int cpu;

        lockdep_assert_held(&rq->lock);

        if (test_tsk_need_resched(curr))
                return;

        cpu = cpu_of(rq);

        if (cpu == smp_processor_id()) {
                set_tsk_need_resched(curr);
                set_preempt_need_resched();
                return;
        }

        if (set_nr_and_not_polling(curr))
                smp_send_reschedule(cpu);
}

`set_tsk_need_resched(curr)` 最终修改当前任务 `thread_info.flags` 里的 `_TIF_NEED_RESCHED`。如果目标 CPU 不是本 CPU，SMP 会发 reschedule IPI；如果是本 CPU，只设置 task 标志和 preempt 标志。

/* arch/mips/kernel/entry.S */
resume_userspace:
        local_irq_disable
        LONG_L  a2, TI_FLAGS($28)
        andi    t0, a2, _TIF_WORK_MASK
        bnez    t0, work_pending
        j       restore_all

work_pending:
        andi    t0, a2, _TIF_NEED_RESCHED
        beqz    t0, work_notifysig
work_resched:
        TRACE_IRQS_OFF
        jal     schedule

MIPS 返回用户态之前读取 `$28` 指向的 `thread_info`，如果 `_TIF_NEED_RESCHED` 置位，就先调用 `schedule()`，调度完成后再重新检查工作标志。

第一性原理概述：`need_resched` 是“现在运行的 task 已不该继续独占 CPU”的延迟通知，保存在线程自己的低层标志中，等返回边界或抢占边界消费。

`schedule()` 与 `__schedule()`：摘下 prev，选出 next

/* kernel/sched/core.c */
asmlinkage __visible void __sched schedule(void)
{
        struct task_struct *tsk = current;

        sched_submit_work(tsk);
        do {
                preempt_disable();
                __schedule(false);
                sched_preempt_enable_no_resched();
        } while (need_resched());
        sched_update_worker(tsk);
}
/* kernel/sched/core.c */
static void __sched notrace __schedule(bool preempt)
{
        struct task_struct *prev, *next;
        unsigned long prev_state;
        struct rq *rq;

        rq = cpu_rq(smp_processor_id());
        prev = rq->curr;

        local_irq_disable();
        rq_lock(rq, &rf);
        update_rq_clock(rq);

        prev_state = prev->state;
        if (!preempt && prev_state) {
                if (signal_pending_state(prev_state, prev))
                        prev->state = TASK_RUNNING;
                else
                        deactivate_task(rq, prev, DEQUEUE_SLEEP | DEQUEUE_NOCLOCK);
                switch_count = &prev->nvcsw;
        }

        next = pick_next_task(rq, prev, &rf);
        clear_tsk_need_resched(prev);
        clear_preempt_need_resched();

        if (likely(prev != next)) {
                rq->nr_switches++;
                RCU_INIT_POINTER(rq->curr, next);
                ++*switch_count;
                trace_sched_switch(preempt, prev, next);
                rq = context_switch(rq, prev, next, &rf);
        } else {
                rq_unlock_irq(rq, &rf);
        }
}

关键分支是 `!preempt && prev_state`：主动睡眠时，`prev->state` 已经不是 `TASK_RUNNING`，所以 `deactivate_task()` 会将它从运行队列移除；抢占时 `prev` 仍 runnable，一般不摘队，只是重新排序后选择 next。

第一性原理概述：`__schedule()` 在 runqueue 锁内做原子决策：当前任务是否还 runnable，谁是下一位，`rq->curr` 指向谁。只有这一步完成后，才允许真正换 CPU 现场。

选择 next：调度类决定策略

核心调度器不把 CFS、RT、Deadline 策略写死在 `__schedule()`，而是调用 `pick_next_task()`，再由调度类选择。官方文档称这些策略被封装在 `sched_class` 钩子中。

/* kernel/sched/rt.c */
static struct task_struct *pick_next_task_rt(struct rq *rq)
{
        struct task_struct *p;

        if (!sched_rt_runnable(rq))
                return NULL;

        p = _pick_next_task_rt(rq);
        set_next_task_rt(rq, p, true);
        return p;
}

CFS 的普通任务从 CFS runqueue 选择合适的 `sched_entity`，本质是让运行较少的实体获得 CPU；RT 类则按实时优先级队列选最高优先级 runnable task。若高优先级类无任务，才逐级落到 CFS、idle。

第一性原理概述：调度公平性、实时优先级、空闲任务不是上下文切换机制的一部分；它们只是回答“next 是谁”。一旦 next 决定，后面的保存恢复流程相同。

`context_switch()`：切 mm，再切寄存器和栈

/* kernel/sched/core.c */
static __always_inline struct rq *
context_switch(struct rq *rq, struct task_struct *prev,
               struct task_struct *next, struct rq_flags *rf)
{
        prepare_task_switch(rq, prev, next);
        arch_start_context_switch(prev);

        if (!next->mm) {                         /* to kernel thread */
                enter_lazy_tlb(prev->active_mm, next);
                next->active_mm = prev->active_mm;
                if (prev->mm)
                        mmgrab(prev->active_mm);
                else
                        prev->active_mm = NULL;
        } else {                                 /* to user task */
                membarrier_switch_mm(rq, prev->active_mm, next->mm);
                switch_mm_irqs_off(prev->active_mm, next->mm, next);
                if (!prev->mm) {
                        rq->prev_mm = prev->active_mm;
                        prev->active_mm = NULL;
                }
        }

        prepare_lock_switch(rq, next, rf);
        switch_to(prev, next, prev);
        barrier();

        return finish_task_switch(prev);
}

`switch_mm_irqs_off()` 处理地址空间切换；内核线程没有自己的 `mm`，所以借用前一个用户进程的 `active_mm` 并进入 lazy TLB。真正的寄存器与内核栈切换发生在 `switch_to(prev, next, prev)`。

注意 `finish_task_switch(prev)` 是在切到 next 的内核栈之后执行的：代码文本相同，但执行者已经是 next task。它释放 runqueue lock、清理 `prev->on_cpu`、处理死亡 task 的栈释放、做 membarrier 后续同步。

第一性原理概述：进程上下文切换至少包含两种上下文：内存上下文 `mm/active_mm` 和 CPU 执行上下文。前者决定虚拟地址如何翻译，后者决定下一条指令、栈和寄存器属于谁。

MIPS `switch_to`：协处理器、TLS、watch 与 `resume`

/* arch/mips/include/asm/switch_to.h */
#define switch_to(prev, next, last)                                    \
do {                                                                  \
        __mips_mt_fpaff_switch_to(prev);                              \
        lose_fpu_inatomic(1, prev);                                   \
        if (tsk_used_math(next))                                      \
                __sanitize_fcr31(next);                               \
        if (cpu_has_dsp) {                                            \
                __save_dsp(prev);                                     \
                __restore_dsp(next);                                  \
        }                                                             \
        if (cpu_has_mxu) {                                            \
                if (KSTK_STATUS(prev) & ST0_CU2) __save_mxu(prev);     \
                if (KSTK_STATUS(next) & ST0_CU2) __restore_mxu(next);  \
        }                                                             \
        if (cop2_present) {                                           \
                ... cop2_save(prev); cop2_restore(next);              \
        }                                                             \
        __clear_r5_hw_ll_bit();                                       \
        __clear_software_ll_bit();                                    \
        if (cpu_has_userlocal)                                        \
                write_c0_userlocal(task_thread_info(next)->tp_value); \
        __restore_watch(next);                                        \
        (last) = resume(prev, next, task_thread_info(next));           \
} while (0)

`switch_to` 在调用 `resume` 前处理那些不属于普通 GPR 的架构状态：FPU 延迟丢弃、DSP/MXU/COP2 保存恢复、LL bit 清除、TLS 写 `CP0_USERLOCAL`、硬件 watch 寄存器恢复。注释还指出：新建线程会从 `ret_from_kernel_thread` 或 `ret_from_fork` 走，`resume()` 后面的语句对新线程会被跳过，所以重要工作必须放在 `resume()` 之前。

/* arch/mips/include/asm/processor.h */
struct thread_struct {
        unsigned long reg16;

第一性原理概述：通用寄存器只是 CPU 现场的一部分。MIPS/XBurst 还要处理协处理器、DSP/MXU、TLS 和 watch 寄存器，否则切换后用户线程可能看到上一个线程的数学状态、TLS 或调试状态。

`resume` 汇编：真正换栈和返回地址

/* arch/mips/kernel/r4k_switch.S */
LEAF(resume)
        mfc0    t1, CP0_STATUS
        LONG_S  t1, THREAD_STATUS(a0)
        cpu_save_nonscratch a0
        LONG_S  ra, THREAD_REG31(a0)

        move    $28, a2
        cpu_restore_nonscratch a1

        PTR_ADDU        t0, $28, _THREAD_SIZE - 32
        set_saved_sp    t0, t1, t2
        mfc0    t1, CP0_STATUS
        li      a3, 0xff01
        and     t1, a3
        LONG_L  a2, THREAD_STATUS(a1)
        nor     a3, $0, a3
        and     a2, a3
        or      a2, t1
        mtc0    a2, CP0_STATUS
        move    v0, a0
        jr      ra
END(resume)
/* arch/mips/include/asm/asmmacro-32.h */
.macro cpu_save_nonscratch thread
        LONG_S  s0, THREAD_REG16(\thread)
        LONG_S  s1, THREAD_REG17(\thread)
        LONG_S  s2, THREAD_REG18(\thread)
        LONG_S  s3, THREAD_REG19(\thread)
        LONG_S  s4, THREAD_REG20(\thread)
        LONG_S  s5, THREAD_REG21(\thread)
        LONG_S  s6, THREAD_REG22(\thread)
        LONG_S  s7, THREAD_REG23(\thread)
        LONG_S  sp, THREAD_REG29(\thread)
        LONG_S  fp, THREAD_REG30(\thread)
.endm

.macro cpu_restore_nonscratch thread
        LONG_L  s0, THREAD_REG16(\thread)
        LONG_L  s1, THREAD_REG17(\thread)
        ...
        LONG_L  sp, THREAD_REG29(\thread)
        LONG_L  fp, THREAD_REG30(\thread)
        LONG_L  ra, THREAD_REG31(\thread)
.endm

调用约定上，`a0=prev`、`a1=next`、`a2=task_thread_info(next)`。`resume` 先把 prev 的 CP0 status、`$s0-$s7`、`$sp`、`$fp`、`$ra` 保存到 `prev->thread`，再把 `$28` 切成 next 的 `thread_info`，从 `next->thread` 装回非易失寄存器、`$sp` 和 `$ra`，最后 `jr ra`。从这条 `jr ra` 开始，CPU 已经在 next 的内核栈和返回地址上运行。

第一性原理概述：上下文切换的最小魔法就是保存旧栈指针和返回地址，装入新栈指针和返回地址。其余寄存器保存都是为了让被切走的内核执行流下次回来时像没有离开过。

切换回来后从哪里继续

如果 next 是以前运行过的任务，`jr ra` 会回到它上次在 `switch_to()` 之后的位置，然后执行 `finish_task_switch(prev)`。如果 next 是新建任务，`copy_thread()` 已把 `thread.reg31` 设成 `ret_from_fork` 或 `ret_from_kernel_thread`，所以第一次被调度时会走新线程专用返回路径。

/* arch/mips/kernel/entry.S */
FEXPORT(ret_from_kernel_thread)
        jal     schedule_tail
        move    a0, s1
        jal     s0
        j       syscall_exit

FEXPORT(ret_from_fork)
        jal     schedule_tail

FEXPORT(syscall_exit)
        local_irq_disable
        LONG_L  a2, TI_FLAGS($28)
        li      t0, _TIF_ALLWORK_MASK
        and     t0, a2, t0
        bnez    t0, syscall_exit_work

restore_all:
        RESTORE_TEMP
        RESTORE_AT
        RESTORE_STATIC
        RESTORE_SOME
        RESTORE_SP_AND_RET

`schedule_tail(prev)` 与 `finish_task_switch(prev)` 配对，释放上一个任务留下的 runqueue lock。之后用户线程沿 `syscall_exit -> restore_all` 恢复内核栈中的 `pt_regs`，最终回到用户态；内核线程则调用创建时保存在 `$s0/$s1` 的函数和参数。

第一性原理概述：老任务恢复的是“上次被切走的内核执行点”，新任务恢复的是“构造好的首次执行点”。两者共用同一个 `switch_to/resume` 机制，只是 `thread.reg31` 初值不同。

整体总结

调度时机

显式阻塞直接调用 `schedule()`；tick/wakeup/跨 CPU 事件先设置 `_TIF_NEED_RESCHED`，在返回边界或抢占边界调度。

调度核心

`__schedule()` 在 rq lock 内更新时钟、处理睡眠任务出队、调用调度类选 next、更新 `rq->curr` 和切换计数。

上下文切换

`context_switch()` 先处理 `mm/active_mm`，再调用 MIPS `switch_to()`，最终 `resume()` 换 `$sp/$ra` 和非易失寄存器。

保存位置

调度状态在 `task_struct/rq`，地址空间在 `mm`，内核续跑现场在 `thread_struct`，用户态完整现场在内核栈 `pt_regs`。

一句话：调度的顶层是“谁该运行”的 runqueue 决策，底层是“如何让 CPU 从那个人的栈和寄存器继续”的架构切换。
