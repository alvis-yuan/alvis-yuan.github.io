进程 A 向进程 B 发送信号，内核到底做了什么

本文只按当前仓库源码解释：A 通过 kill(pid, sig) 发给 B，内核完成权限检查、目标解析、pending 队列标记和唤醒；B 在返回用户态时可能执行用户态信号回调，也可能由 signalfd 的 read() 同步取走信号。

第一性原理概述：信号不是一次“函数远程调用”，而是内核在目标任务状态里设置一个待处理事实。目标任务何时响应该事实，取决于它的屏蔽字、调度状态、返回用户态边界，或是否主动通过同步接口读取。

MSC 总览

图中 handler 分支由返回用户态触发，signalfd 分支由 B 主动 read/poll 触发。
第一性原理概述：内核把“事件发生”与“用户态处理方式”解耦。发送方只负责制造事件；接收方在安全边界上决定是被异步改写用户态执行流，还是通过 fd 同步拉取。

数据模型：位图、队列、处理表

/* include/linux/sched.h */
struct task_struct {
	...
	struct signal_struct		*signal;
	struct sighand_struct __rcu	*sighand;
	sigset_t			blocked;
	struct sigpending		pending;
	...
};

sigpending.signal 是“这个信号号是否 pending”的位图，sigpending.list 保存对应 siginfo。标准信号只需要知道“至少有一个”，实时信号和带数据的场景需要队列保存更多信息。task_struct.pending 是线程私有 pending，signal_struct.shared_pending 是线程组共享 pending；sighand_struct.action[] 是进程级 disposition 表。

这种设计解决两个问题：一是快速判断是否存在未屏蔽信号，只需位运算；二是只有需要用户可见细节时才走 sigqueue 列表。siglock 作为 sighand 的锁，也覆盖共享的 signal_struct 信号状态，避免发送、读取、默认动作之间互相踩踏。

第一性原理概述：信号系统的最小状态是“号、来源信息、目标范围、屏蔽关系、处理策略”。位图负责快，链表负责准，锁负责让多个 CPU 对同一事实达成一致。

发送入口：kill 系统调用

/* kernel/signal.c */
static inline void prepare_kill_siginfo(int sig, struct kernel_siginfo *info)
{
	clear_siginfo(info);
	info->si_signo = sig;
	info->si_errno = 0;
	info->si_code = SI_USER;
	info->si_pid = task_tgid_vnr(current);
	info->si_uid = from_kuid_munged(current_user_ns(), current_uid());
}

SYSCALL_DEFINE2(kill, pid_t, pid, int, sig)
{
	struct kernel_siginfo info;

	prepare_kill_siginfo(sig, &info);

	return kill_something_info(sig, &info, pid);
}

kill() 入口首先把用户态传入的 sig 转成内核内部的 kernel_siginfo。SI_USER 表示这是普通用户发送路径，si_pid 和 si_uid 记录发送者身份。官方 kill(2) 也说明，sig=0 不实际发送信号，但仍会做存在性和权限检查；当前源码后面通过 if (!ret && sig) 保留这个语义。

设计上，入口不立刻改目标进程，而是先构造一份不可歧义的事件描述。这样权限检查、命名空间转换、队列入队和用户态 siginfo_t 暴露都能沿同一份信息传播。

第一性原理概述：发送信号的第一步不是“打断 B”，而是把 A 的意图转成内核可验证、可审计、可复制的事件记录。

权限检查：谁可以给谁发

/* kernel/signal.c */
static bool kill_ok_by_cred(struct task_struct *t)
{
	const struct cred *cred = current_cred();
	const struct cred *tcred = __task_cred(t);

	return uid_eq(cred->euid, tcred->suid) ||
	       uid_eq(cred->euid, tcred->uid) ||
	       uid_eq(cred->uid, tcred->suid) ||
	       uid_eq(cred->uid, tcred->uid) ||
	       ns_capable(tcred->user_ns, CAP_KILL);
}

static int check_kill_permission(int sig, struct kernel_siginfo *info,
				 struct task_struct *t)
{
	struct pid *sid;
	int error;

	if (!valid_signal(sig))
		return -EINVAL;

	if (!si_fromuser(info))
		return 0;

	error = audit_signal_info(sig, t);
	if (error)
		return error;

	if (!same_thread_group(current, t) &&
	    !kill_ok_by_cred(t)) {
		switch (sig) {
		case SIGCONT:
			sid = task_session(t);
			if (!sid || sid == task_session(current))
				break;
			fallthrough;
		default:
			return -EPERM;
		}
	}

	return security_task_kill(t, info, sig, NULL);
}

这里对应 kill(2) 的权限语义：发送方需要匹配目标的真实/保存 UID，或具备目标 user namespace 中的 CAP_KILL；SIGCONT 有同 session 的例外。源码还把 audit 和 LSM 钩子纳入路径，说明信号发送不仅是调度事件，也是安全边界。

si_fromuser() 用来区分用户来源与内核特权来源。内核内部强制信号不受普通用户权限规则限制，因为它们通常来自页错误、非法指令、OOM、退出等内核事实。

第一性原理概述：信号能改变目标进程控制流甚至终止它，所以它必须按“谁有权影响谁”的身份关系先筛选，再进入调度和递送逻辑。

目标解析：pid 的几种语义

/* kernel/signal.c */
static int kill_something_info(int sig, struct kernel_siginfo *info, pid_t pid)
{
	int ret;

	if (pid > 0)
		return kill_proc_info(sig, info, pid);

	if (pid == INT_MIN)
		return -ESRCH;

	read_lock(&tasklist_lock);
	if (pid != -1) {
		ret = __kill_pgrp_info(sig, info,
				pid ? find_vpid(-pid) : task_pgrp(current));
	} else {
		int retval = 0, count = 0;
		struct task_struct * p;

		for_each_process(p) {
			if (task_pid_vnr(p) > 1 &&
					!same_thread_group(p, current)) {
				int err = group_send_sig_info(sig, info, p,
							      PIDTYPE_MAX);
				++count;
				if (err != -EPERM)
					retval = err;
			}
		}
		ret = count ? retval : -ESRCH;
	}
	read_unlock(&tasklist_lock);

	return ret;
}

int kill_pid_info(int sig, struct kernel_siginfo *info, struct pid *pid)
{
	int error = -ESRCH;
	struct task_struct *p;

	for (;;) {
		rcu_read_lock();
		p = pid_task(pid, PIDTYPE_PID);
		if (p)
			error = group_send_sig_info(sig, info, p, PIDTYPE_TGID);
		rcu_read_unlock();
		if (likely(!p || error != -ESRCH))
			return error;
	}
}

官方 kill(2) 对 pid 的语义在这里直接落地：正数找单个进程，0 找当前进程组，负数找指定进程组，-1 遍历可发送的进程。当前源码还跳过 PID 1 和当前线程组，避免 kill(-1, sig) 把自己和 init 也纳入。

kill_pid_info() 用 RCU 查 pid_task()，如果遇到任务正在退出或 de_thread() 竞争，就重试。这个循环解决的是 PID 到 task 的生命周期问题：PID 号可能仍在，但 task 指针的领导线程身份可能正在切换。

第一性原理概述：用户给的是数字 PID，内核要把它映射成受生命周期保护的 task_struct。只有目标对象稳定，后续加锁和入队才有意义。

入队：pending 位图与 siginfo 队列

/* kernel/signal.c */
int group_send_sig_info(int sig, struct kernel_siginfo *info,
			struct task_struct *p, enum pid_type type)
{
	int ret;

	rcu_read_lock();
	ret = check_kill_permission(sig, info, p);
	rcu_read_unlock();

	if (!ret && sig)
		ret = do_send_sig_info(sig, info, p, type);

	return ret;
}

int do_send_sig_info(int sig, struct kernel_siginfo *info, struct task_struct *p,
			enum pid_type type)
{
	unsigned long flags;
	int ret = -ESRCH;

	if (lock_task_sighand(p, &flags)) {
		ret = send_signal(sig, info, p, type);
		unlock_task_sighand(p, &flags);
	}

	return ret;
}

外层先权限检查，再在 sig != 0 时真正发送。do_send_sig_info() 锁住目标 sighand->siglock，因为它马上要同时查看 action、pending、shared_pending 和线程组状态。这个锁是信号状态的一致性边界。

/* kernel/signal.c */
static int __send_signal(int sig, struct kernel_siginfo *info, struct task_struct *t,
			enum pid_type type, bool force)
{
	struct sigpending *pending;
	struct sigqueue *q;
	int override_rlimit;
	int ret = 0, result;

	assert_spin_locked(&t->sighand->siglock);

	result = TRACE_SIGNAL_IGNORED;
	if (!prepare_signal(sig, t, force))
		goto ret;

	pending = (type != PIDTYPE_PID) ? &t->signal->shared_pending : &t->pending;
	result = TRACE_SIGNAL_ALREADY_PENDING;
	if (legacy_queue(pending, sig))
		goto ret;

	result = TRACE_SIGNAL_DELIVERED;
	if ((sig == SIGKILL) || (t->flags & PF_KTHREAD))
		goto out_set;

	if (sig < SIGRTMIN)
		override_rlimit = (is_si_special(info) || info->si_code >= 0);
	else
		override_rlimit = 0;

	q = __sigqueue_alloc(sig, t, GFP_ATOMIC, override_rlimit);
	if (q) {
		list_add_tail(&q->list, &pending->list);
		switch ((unsigned long) info) {
		case (unsigned long) SEND_SIG_NOINFO:
			clear_siginfo(&q->info);
			q->info.si_signo = sig;
			q->info.si_errno = 0;
			q->info.si_code = SI_USER;
			q->info.si_pid = task_tgid_nr_ns(current,
							task_active_pid_ns(t));
			q->info.si_uid = from_kuid_munged(task_cred_xxx(t, user_ns),
							 current_uid());
			break;
		case (unsigned long) SEND_SIG_PRIV:
			clear_siginfo(&q->info);
			q->info.si_signo = sig;
			q->info.si_errno = 0;
			q->info.si_code = SI_KERNEL;
			q->info.si_pid = 0;
			q->info.si_uid = 0;
			break;
		default:
			copy_siginfo(&q->info, info);
			break;
		}
	} else if (!is_si_special(info) &&
		   sig >= SIGRTMIN && info->si_code != SI_USER) {
		result = TRACE_SIGNAL_OVERFLOW_FAIL;
		ret = -EAGAIN;
		goto ret;
	} else {
		result = TRACE_SIGNAL_LOSE_INFO;
	}

out_set:
	signalfd_notify(t, sig);
	sigaddset(&pending->signal, sig);
	complete_signal(sig, t, type);
ret:
	trace_signal_generate(sig, info, t, type != PIDTYPE_PID, result);
	return ret;
}

pending 的选择体现“进程定向”和“线程定向”的区别：PIDTYPE_PID 放到线程私有 t->pending，否则放到线程组共享 signal->shared_pending。这正对应 signal(7) 对 process-directed 与 thread-directed signal 的说明。

legacy_queue() 使标准信号在同一个 pending 集里只保留一个实例；实时信号则尽量分配 sigqueue 保存多个实例和附加数据。即使 sigqueue 分配失败，普通 kill() 路径也尽量保留 pending 位，只是丢失详细 siginfo，这是“至少递送事件”和“尽量保留原因”的折中。

第一性原理概述：入队的本质是把事件写进目标状态：位图回答“有没有”，链表回答“为什么”。标准信号偏事件合并，实时信号偏消息队列。

唤醒：选择哪个线程来响应

/* kernel/signal.c */
static inline bool wants_signal(int sig, struct task_struct *p)
{
	if (sigismember(&p->blocked, sig))
		return false;

	if (p->flags & PF_EXITING)
		return false;

	if (sig == SIGKILL)
		return true;

	if (task_is_stopped_or_traced(p))
		return false;

	return task_curr(p) || !task_sigpending(p);
}

static void complete_signal(int sig, struct task_struct *p, enum pid_type type)
{
	struct signal_struct *signal = p->signal;
	struct task_struct *t;

	if (wants_signal(sig, p))
		t = p;
	else if ((type == PIDTYPE_PID) || thread_group_empty(p))
		return;
	else {
		t = signal->curr_target;
		while (!wants_signal(sig, t)) {
			t = next_thread(t);
			if (t == signal->curr_target)
				return;
		}
		signal->curr_target = t;
	}

	if (sig_fatal(p, sig) &&
	    !(signal->flags & SIGNAL_GROUP_EXIT) &&
	    !sigismember(&t->real_blocked, sig) &&
	    (sig == SIGKILL || !p->ptrace)) {
		if (!sig_kernel_coredump(sig)) {
			signal->flags = SIGNAL_GROUP_EXIT;
			signal->group_exit_code = sig;
			signal->group_stop_count = 0;
			t = p;
			do {
				task_clear_jobctl_pending(t, JOBCTL_PENDING_MASK);
				sigaddset(&t->pending.signal, SIGKILL);
				signal_wake_up(t, 1);
			} while_each_thread(p, t);
			return;
		}
	}

	signal_wake_up(t, sig == SIGKILL);
}

对于进程定向信号，Linux 可以把它递送给任一未屏蔽该信号的线程。wants_signal() 排除 blocked、退出中、停止/被 ptrace 的线程；complete_signal() 先看主线程，再轮询线程组的 curr_target。这避免所有信号总打到同一线程，也避免唤醒一个无法处理它的线程。

如果信号默认动作是致命且无需 core dump，当前源码会直接设置 SIGNAL_GROUP_EXIT 并给所有线程加 SIGKILL pending。这解决“一个线程已收到 fatal signal，其他线程还继续破坏共享进程状态”的问题。

第一性原理概述：信号响应需要一个执行载体。多线程进程里，事件属于进程，但最终必须落到某个线程的返回用户态路径或等待路径上。

返回用户态检查：MIPS do_notify_resume

/* arch/mips/kernel/signal.c */
asmlinkage void do_notify_resume(struct pt_regs *regs, void *unused,
		__u32 thread_info_flags)
{
	local_irq_enable();

	user_exit();

	if (thread_info_flags & _TIF_UPROBE)
		uprobe_notify_resume(regs);

	/* deal with pending signal delivery */
	if (thread_info_flags & (_TIF_SIGPENDING | _TIF_NOTIFY_SIGNAL))
		do_signal(regs);

	if (thread_info_flags & _TIF_NOTIFY_RESUME) {
		tracehook_notify_resume(regs);
		rseq_handle_notify_resume(NULL, regs);
	}

	user_enter();
}

static void do_signal(struct pt_regs *regs)
{
	struct ksignal ksig;

	if (get_signal(&ksig)) {
		handle_signal(&ksig, regs);
		return;
	}

	if (regs->regs[0]) {
		switch (regs->regs[2]) {
		case ERESTARTNOHAND:
		case ERESTARTSYS:
		case ERESTARTNOINTR:
			regs->regs[2] = regs->regs[0];
			regs->regs[7] = regs->regs[26];
			regs->cp0_epc -= 4;
			break;
		case ERESTART_RESTARTBLOCK:
			regs->regs[2] = current->thread.abi->restart;
			regs->regs[7] = regs->regs[26];
			regs->cp0_epc -= 4;
			break;
		}
		regs->regs[0] = 0;
	}

	restore_saved_sigmask();
}

发送方完成入队后，B 不一定立刻运行 handler。真正检查发生在 MIPS 返回用户态前的 do_notify_resume()：只要线程标志带 _TIF_SIGPENDING 或 _TIF_NOTIFY_SIGNAL，就进入 do_signal()。这和 signal(7) 的描述一致：内核在从 kernel-mode 返回 user-mode 时检查 pending unblocked signal。

regs 是保存的用户态寄存器。若没有可递送信号，代码会处理被中断系统调用的 restart 语义；若有可递送信号，则进入 handle_signal() 改写用户态返回现场。

第一性原理概述：异步信号不能在任意内核临界区里直接跳进用户 handler；它必须等到“准备回用户态”这个可控边界，才能安全改写用户态 PC、SP 和寄存器。

取出信号：get_signal 与 dequeue_signal

/* kernel/signal.c */
int dequeue_signal(struct task_struct *tsk, sigset_t *mask, kernel_siginfo_t *info)
{
	bool resched_timer = false;
	int signr;

	signr = __dequeue_signal(&tsk->pending, mask, info, &resched_timer);
	if (!signr) {
		signr = __dequeue_signal(&tsk->signal->shared_pending,
					 mask, info, &resched_timer);
	}

	recalc_sigpending();
	if (!signr)
		return 0;

	if (unlikely(sig_kernel_stop(signr)))
		current->jobctl |= JOBCTL_STOP_DEQUEUED;

	return signr;
}

bool get_signal(struct ksignal *ksig)
{
	struct sighand_struct *sighand = current->sighand;
	struct signal_struct *signal = current->signal;
	int signr;

	if (unlikely(current->task_works))
		task_work_run();

	if (!IS_ENABLED(CONFIG_GENERIC_ENTRY)) {
		if (test_thread_flag(TIF_NOTIFY_SIGNAL))
			tracehook_notify_signal();
		if (!task_sigpending(current))
			return false;
	}

	try_to_freeze();

relock:
	spin_lock_irq(&sighand->siglock);

	if (signal_group_exit(signal)) {
		ksig->info.si_signo = signr = SIGKILL;
		sigdelset(&current->pending.signal, SIGKILL);
		recalc_sigpending();
		goto fatal;
	}

	for (;;) {
		struct k_sigaction *ka;

		signr = dequeue_synchronous_signal(&ksig->info);
		if (!signr)
			signr = dequeue_signal(current, &current->blocked, &ksig->info);

		if (!signr)
			break;

		ka = &sighand->action[signr-1];

		if (ka->sa.sa_handler == SIG_IGN)
			continue;
		if (ka->sa.sa_handler != SIG_DFL) {
			ksig->ka = *ka;
			if (ka->sa.sa_flags & SA_ONESHOT)
				ka->sa.sa_handler = SIG_DFL;
			break;
		}

		if (sig_kernel_ignore(signr))
			continue;

		if (sig_kernel_stop(signr)) {
			if (likely(do_signal_stop(ksig->info.si_signo)))
				goto relock;
			continue;
		}

fatal:
		spin_unlock_irq(&sighand->siglock);
		do_group_exit(ksig->info.si_signo);
	}
	spin_unlock_irq(&sighand->siglock);
	ksig->sig = signr;
	return ksig->sig > 0;
}

dequeue_signal() 先取线程私有 pending，再取线程组共享 pending。传入的 mask 是当前 blocked 集，next_signal() 会找“pending 但未被 mask 屏蔽”的信号。同步硬件异常类信号优先于普通 pending 信号，因为 handler 的栈帧必须指向触发异常的那条指令现场。

get_signal() 处理四类结果：忽略则继续找下一个；有用户 handler 则填充 ksig 返回给体系结构代码；默认忽略也继续；默认停止或默认致命则执行 job control 或进程退出。这里完成“pending 事实”到“用户可见动作”的选择。

第一性原理概述：响应信号就是在锁保护下从候选事件中选出一个未屏蔽事件，再按 disposition 转换成 ignore、stop、terminate、core dump 或用户 handler。

默认动作：忽略、停止、终止

官方 signal(7) 把默认动作分为 Term、Ign、Core、Stop、Cont。当前源码里这些动作被封装为 sig_kernel_ignore()、sig_kernel_stop()、sig_kernel_coredump()、sig_fatal() 等判断。SIGKILL 和 SIGSTOP 不能被 catch、block 或 ignore，这一点也在官方文档中明确。

为什么默认动作在取信号阶段执行，而不是发送阶段执行？因为发送阶段并不知道目标线程什么时候能安全停止或退出，也不知道信号到达时 action 是否被 sigaction() 改过。递送阶段拿着 siglock 和当前线程现场，才能做最终决定。

第一性原理概述：默认动作是接收方策略，不是发送方策略。发送只制造事实，响应时才用接收方当时的 disposition 把事实转成行为。

用户回调：MIPS 构造 rt signal frame

/* arch/mips/kernel/signal.c */
static void handle_signal(struct ksignal *ksig, struct pt_regs *regs)
{
	sigset_t *oldset = sigmask_to_save();
	int ret;
	struct mips_abi *abi = current->thread.abi;
	void *vdso = current->mm->context.vdso;

	dsemul_thread_rollback(regs);

	if (regs->regs[0]) {
		switch(regs->regs[2]) {
		case ERESTART_RESTARTBLOCK:
		case ERESTARTNOHAND:
			regs->regs[2] = EINTR;
			break;
		case ERESTARTSYS:
			if (!(ksig->ka.sa.sa_flags & SA_RESTART)) {
				regs->regs[2] = EINTR;
				break;
			}
			fallthrough;
		case ERESTARTNOINTR:
			regs->regs[7] = regs->regs[26];
			regs->regs[2] = regs->regs[0];
			regs->cp0_epc -= 4;
		}
		regs->regs[0] = 0;
	}

	rseq_signal_deliver(ksig, regs);

	if (sig_uses_siginfo(&ksig->ka, abi))
		ret = abi->setup_rt_frame(vdso + abi->vdso->off_rt_sigreturn,
					  ksig, regs, oldset);
	else
		ret = abi->setup_frame(vdso + abi->vdso->off_sigreturn,
				       ksig, regs, oldset);

	signal_setup_done(ret, ksig, 0);
}

static int setup_rt_frame(void *sig_return, struct ksignal *ksig,
			  struct pt_regs *regs, sigset_t *set)
{
	struct rt_sigframe __user *frame;
	int err = 0;

	frame = get_sigframe(ksig, regs, sizeof(*frame));
	if (!access_ok(frame, sizeof (*frame)))
		return -EFAULT;

	err |= copy_siginfo_to_user(&frame->rs_info, &ksig->info);
	err |= __put_user(0, &frame->rs_uc.uc_flags);
	err |= __put_user(NULL, &frame->rs_uc.uc_link);
	err |= __save_altstack(&frame->rs_uc.uc_stack, regs->regs[29]);
	err |= setup_sigcontext(regs, &frame->rs_uc.uc_mcontext);
	err |= __copy_to_user(&frame->rs_uc.uc_sigmask, set, sizeof(*set));

	if (err)
		return -EFAULT;

	regs->regs[ 4] = ksig->sig;
	regs->regs[ 5] = (unsigned long) &frame->rs_info;
	regs->regs[ 6] = (unsigned long) &frame->rs_uc;
	regs->regs[29] = (unsigned long) frame;
	regs->regs[31] = (unsigned long) sig_return;
	regs->cp0_epc = regs->regs[25] = (unsigned long) ksig->ka.sa.sa_handler;

	return 0;
}

当 B 安装了 handler，MIPS 不是在内核里调用那个函数，而是修改将要返回的用户态寄存器：a0 放信号号，a1 放 siginfo，a2 放 ucontext，sp 指向新建的 rt_sigframe，ra 指向 vDSO 的 rt_sigreturn trampoline，cp0_epc 指向 handler。这样 eret 回用户态后，B 看起来就像自然调用了 handler。

signal_setup_done() 成功后调用 signal_delivered()，把 sa_mask 和当前信号加入 blocked 集，除非设置 SA_NODEFER。这对应 sigaction(2)：handler 执行期间会临时屏蔽指定信号，避免默认重入。

/* kernel/signal.c */
static void signal_delivered(struct ksignal *ksig, int stepping)
{
	sigset_t blocked;

	clear_restore_sigmask();

	sigorsets(&blocked, &current->blocked, &ksig->ka.sa.sa_mask);
	if (!(ksig->ka.sa.sa_flags & SA_NODEFER))
		sigaddset(&blocked, ksig->sig);
	set_current_blocked(&blocked);
	tracehook_signal_handler(stepping);
}
第一性原理概述：handler 不是内核回调用户函数，而是内核重写用户态返回现场。保存旧现场、跳到 handler、再通过 trampoline 恢复，是异步控制流能继续原程序的根本机制。

handler 返回：rt_sigreturn 恢复现场

/* arch/mips/kernel/signal.c */
asmlinkage void sys_rt_sigreturn(void)
{
	struct rt_sigframe __user *frame;
	struct pt_regs *regs;
	sigset_t set;
	int sig;

	regs = current_pt_regs();
	frame = (struct rt_sigframe __user *)regs->regs[29];
	if (!access_ok(frame, sizeof(*frame)))
		goto badframe;
	if (__copy_from_user(&set, &frame->rs_uc.uc_sigmask, sizeof(set)))
		goto badframe;

	set_current_blocked(&set);

	sig = restore_sigcontext(regs, &frame->rs_uc.uc_mcontext);
	if (sig < 0)
		goto badframe;
	else if (sig)
		force_sig(sig);

	if (restore_altstack(&frame->rs_uc.uc_stack))
		goto badframe;

	__asm__ __volatile__(
		"move\t$29, %0\n\t"
		"j\tsyscall_exit"
		: /* no outputs */
		: "r" (regs));

badframe:
	force_sig(SIGSEGV);
}

int restore_sigcontext(struct pt_regs *regs, struct sigcontext __user *sc)
{
	unsigned long treg;
	int err = 0;
	int i;

	current->restart_block.fn = do_no_restart_syscall;

	err |= __get_user(regs->cp0_epc, &sc->sc_pc);
	err |= __get_user(regs->hi, &sc->sc_mdhi);
	err |= __get_user(regs->lo, &sc->sc_mdlo);

	for (i = 1; i < 32; i++)
		err |= __get_user(regs->regs[i], &sc->sc_regs[i]);

	return err ?: protected_restore_fp_context(sc);
}

handler 普通返回后会跳到 regs[31] 指定的 vDSO trampoline，trampoline 发起 rt_sigreturn 系统调用。这里内核从用户栈上的 rt_sigframe 读回保存的信号屏蔽字、通用寄存器、PC、FPU 上下文和 altstack 状态。

如果用户栈帧损坏，代码进入 badframe 强制 SIGSEGV。这不是惩罚，而是保护：内核不能相信用户态给出的任意恢复现场，否则用户程序可以伪造不受约束的内核返回状态。

第一性原理概述：异步 handler 要能回到被打断处，就必须把旧世界压到用户栈，再由一个受内核校验的系统调用恢复它。恢复失败说明控制流完整性已经丢失，只能转为故障信号。

signalfd 创建：把信号变成可读 fd

/* fs/signalfd.c */
struct signalfd_ctx {
	sigset_t sigmask;
};

static int do_signalfd4(int ufd, sigset_t *mask, int flags)
{
	struct signalfd_ctx *ctx;

	BUILD_BUG_ON(SFD_CLOEXEC != O_CLOEXEC);
	BUILD_BUG_ON(SFD_NONBLOCK != O_NONBLOCK);

	if (flags & ~(SFD_CLOEXEC | SFD_NONBLOCK))
		return -EINVAL;

	sigdelsetmask(mask, sigmask(SIGKILL) | sigmask(SIGSTOP));
	signotset(mask);

	if (ufd == -1) {
		ctx = kmalloc(sizeof(*ctx), GFP_KERNEL);
		if (!ctx)
			return -ENOMEM;

		ctx->sigmask = *mask;

		ufd = anon_inode_getfd("[signalfd]", &signalfd_fops, ctx,
				       O_RDWR | (flags & (O_CLOEXEC | O_NONBLOCK)));
		if (ufd < 0)
			kfree(ctx);
	} else {
		struct fd f = fdget(ufd);
		if (!f.file)
			return -EBADF;
		ctx = f.file->private_data;
		if (f.file->f_op != &signalfd_fops) {
			fdput(f);
			return -EINVAL;
		}
		spin_lock_irq(&current->sighand->siglock);
		ctx->sigmask = *mask;
		spin_unlock_irq(&current->sighand->siglock);

		wake_up(&current->sighand->signalfd_wqh);
		fdput(f);
	}

	return ufd;
}

signalfd(2) 说 SIGKILL 和 SIGSTOP 不能通过 signalfd 接收；源码用 sigdelsetmask() 删除它们。随后 signotset(mask) 转成内核内部 dequeue_signal() 使用的 mask 形式。

anon_inode_getfd() 说明 signalfd 不对应磁盘 inode，而是匿名文件对象。它的核心状态只有一个 sigmask：这个 fd 愿意同步接收哪些信号。官方建议通常先用 sigprocmask() block 这些信号，否则它们可能按默认动作或 handler 被处理，而不是留给 fd 读取。

第一性原理概述：signalfd 没有创造第二套信号系统；它只是给同一 pending 集合提供了一个 fd 视角，让事件循环能用 read/poll/epoll 管理信号。

signalfd read：同步消费 pending 信号

/* fs/signalfd.c */
static ssize_t signalfd_dequeue(struct signalfd_ctx *ctx, kernel_siginfo_t *info,
				int nonblock)
{
	ssize_t ret;
	DECLARE_WAITQUEUE(wait, current);

	spin_lock_irq(&current->sighand->siglock);
	ret = dequeue_signal(current, &ctx->sigmask, info);
	switch (ret) {
	case 0:
		if (!nonblock)
			break;
		ret = -EAGAIN;
		fallthrough;
	default:
		spin_unlock_irq(&current->sighand->siglock);
		return ret;
	}

	add_wait_queue(&current->sighand->signalfd_wqh, &wait);
	for (;;) {
		set_current_state(TASK_INTERRUPTIBLE);
		ret = dequeue_signal(current, &ctx->sigmask, info);
		if (ret != 0)
			break;
		if (signal_pending(current)) {
			ret = -ERESTARTSYS;
			break;
		}
		spin_unlock_irq(&current->sighand->siglock);
		schedule();
		spin_lock_irq(&current->sighand->siglock);
	}
	spin_unlock_irq(&current->sighand->siglock);

	remove_wait_queue(&current->sighand->signalfd_wqh, &wait);
	__set_current_state(TASK_RUNNING);

	return ret;
}

static ssize_t signalfd_read(struct file *file, char __user *buf, size_t count,
			     loff_t *ppos)
{
	struct signalfd_ctx *ctx = file->private_data;
	struct signalfd_siginfo __user *siginfo;
	int nonblock = file->f_flags & O_NONBLOCK;
	ssize_t ret, total = 0;
	kernel_siginfo_t info;

	count /= sizeof(struct signalfd_siginfo);
	if (!count)
		return -EINVAL;

	siginfo = (struct signalfd_siginfo __user *) buf;
	do {
		ret = signalfd_dequeue(ctx, &info, nonblock);
		if (unlikely(ret <= 0))
			break;
		ret = signalfd_copyinfo(siginfo, &info);
		if (ret < 0)
			break;
		siginfo++;
		total += ret;
		nonblock = 1;
	} while (--count);

	return total ? total: ret;
}

signalfd_read() 与 handler 路径共享 dequeue_signal()，所以它会真正消费 pending 信号。官方 signalfd(2) 明确说，read 的结果是这些信号不再 pending，也不会再被 handler 捕获或被 sigwaitinfo() 接收。

如果没有匹配信号，阻塞 fd 会把当前任务挂到 sighand->signalfd_wqh，等待发送路径的 signalfd_notify() 唤醒；非阻塞 fd 返回 -EAGAIN。循环读取允许一次 read 返回多个 struct signalfd_siginfo。

/* fs/signalfd.c */
static int signalfd_copyinfo(struct signalfd_siginfo __user *uinfo,
			     kernel_siginfo_t const *kinfo)
{
	struct signalfd_siginfo new;

	BUILD_BUG_ON(sizeof(struct signalfd_siginfo) != 128);
	memset(&new, 0, sizeof(new));

	new.ssi_signo = kinfo->si_signo;
	new.ssi_errno = kinfo->si_errno;
	new.ssi_code  = kinfo->si_code;
	switch (siginfo_layout(kinfo->si_signo, kinfo->si_code)) {
	case SIL_KILL:
		new.ssi_pid = kinfo->si_pid;
		new.ssi_uid = kinfo->si_uid;
		break;
	case SIL_RT:
		new.ssi_pid = kinfo->si_pid;
		new.ssi_uid = kinfo->si_uid;
		new.ssi_ptr = (long) kinfo->si_ptr;
		new.ssi_int = kinfo->si_int;
		break;
	case SIL_CHLD:
		new.ssi_pid    = kinfo->si_pid;
		new.ssi_uid    = kinfo->si_uid;
		new.ssi_status = kinfo->si_status;
		new.ssi_utime  = kinfo->si_utime;
		new.ssi_stime  = kinfo->si_stime;
		break;
	default:
		break;
	}

	if (copy_to_user(uinfo, &new, sizeof(struct signalfd_siginfo)))
		return -EFAULT;

	return sizeof(*uinfo);
}

signalfd_copyinfo() 把内核 kernel_siginfo_t 转成固定 128 字节的用户 ABI 结构。不同 si_code 布局只填充有意义字段，避免把无关联合字段暴露给用户态。

第一性原理概述：signalfd 的 read 是“拉模型”：用户态在自己选择的时刻，把 pending 事件从内核状态中取出并转成普通字节流。

signalfd poll/epoll：事件循环如何感知信号

/* fs/signalfd.c */
static __poll_t signalfd_poll(struct file *file, poll_table *wait)
{
	struct signalfd_ctx *ctx = file->private_data;
	__poll_t events = 0;

	poll_wait(file, &current->sighand->signalfd_wqh, wait);

	spin_lock_irq(&current->sighand->siglock);
	if (next_signal(&current->pending, &ctx->sigmask) ||
	    next_signal(&current->signal->shared_pending,
			&ctx->sigmask))
		events |= EPOLLIN;
	spin_unlock_irq(&current->sighand->siglock);

	return events;
}

poll_wait() 把当前任务注册到 signalfd_wqh，随后在锁内检查私有 pending 和共享 pending 中是否存在与 fd mask 匹配的信号。若存在就返回 EPOLLIN。这就是官方文档所说“signalfd fd 可被 select/poll/epoll 监控”的内核实现。

它只检查当前线程可读的信号：线程私有 pending 加线程组共享 pending。官方 signalfd(2) 也说明，多线程程序中，一个线程不能通过 signalfd 读到定向给其他线程的信号。

第一性原理概述：poll 不消费事件，只回答“现在 read 会不会有东西”。消费仍由 read 完成，这让事件循环可以先统一调度，再决定处理顺序。

handler 与 signalfd 对比

维度signal handlersignalfd
触发时机目标线程返回用户态时，内核改写用户态 PC/SP/寄存器。目标线程调用 read() 或通过 poll/epoll 发现可读后主动读取。
控制流异步插入 handler，之后通过 rt_sigreturn 恢复。同步函数调用，像读取普通 fd 一样返回结构体。
pending 消费get_signal() 递送时 dequeue。signalfd_read() 调用 dequeue_signal() 消费。
适合场景传统 UNIX 信号、崩溃处理、必须捕获同步硬件异常如 SIGSEGV。事件循环、服务进程、希望避免异步 handler 限制的程序。
限制handler 内只能安全调用 async-signal-safe 函数，且要考虑重入。通常需先 block 对应信号；不能接收 SIGKILL/SIGSTOP；同步硬件异常不能靠 signalfd 接收。
第一性原理概述：handler 和 signalfd 不是两个来源，而是同一 pending 状态的两个出口。为了让 signalfd 稳定接收，必须阻止 handler/default action 抢先消费。

全流程总结

  1. A 调用 kill(pid, sig)，内核构造 kernel_siginfo，记录发送者 PID/UID 和 SI_USER。
  2. 内核按 PID 语义找到目标进程或进程组，执行 UID、CAP_KILL、SIGCONT session 例外、audit 和 LSM 检查。
  3. 目标 sighand->siglock 下，内核选择线程私有 pending 或线程组共享 pending，设置 pending 位，必要时挂入 sigqueue 保存 siginfo。
  4. complete_signal() 选择一个未屏蔽且可响应的线程，设置 pending 标志并唤醒；fatal signal 可直接推动整个线程组退出。
  5. B 返回用户态时，MIPS do_notify_resume() 调用 do_signal()，get_signal() 从 pending 中取出未屏蔽信号，按 disposition 决定忽略、停止、终止或构造 handler。
  6. 若执行 handler，MIPS 把 rt_sigframe 写到用户栈，设置 a0/a1/a2/sp/ra/epc，用户态 handler 返回后通过 rt_sigreturn 恢复原现场。
  7. 若 B 使用 signalfd，read() 在同一 siglock 下调用 dequeue_signal() 消费 pending，并把 siginfo 转成 128 字节 signalfd_siginfo 返回。
第一性原理概述：整个信号系统围绕一个核心不变量：信号先成为目标任务中受锁保护的 pending 状态，再由接收方可处理的执行上下文消费。异步 handler 是“内核改写用户返回现场”，signalfd 是“用户同步读取同一 pending 队列”。
