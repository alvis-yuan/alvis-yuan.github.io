线程创建：从 pthread_create/clone 到 MIPS 首次返回

Linux 内核没有一个独立于进程的“线程对象”：线程也是 `task_struct`。所谓线程，是用 `clone()` 创建一个新 task，并用 `CLONE_VM|CLONE_THREAD|CLONE_SIGHAND|CLONE_FILES|CLONE_FS...` 选择共享地址空间、信号处理、文件表和线程组。本文按当前源码说明用户线程、`clone3()` 和内核线程如何汇入同一条 `kernel_clone() -> copy_process() -> copy_thread() -> wake_up_new_task()` 路径。

官方依据

主题官方链接本文使用的事实
pthread_createpthread_create(3)在调用进程内启动新线程，新线程从 `start_routine(arg)` 开始；默认 joinable；新线程有自己的栈和线程 ID。
pthreadspthreads(7)同一进程的线程共享全局内存、PID、文件描述符、信号 dispositions 等；每个线程有自己的栈、信号 mask、errno、调度属性等。
clone/clone3clone(2)`clone()`/`clone3()` 创建子 task，并通过 `CLONE_*` 精确控制 VM、文件表、信号处理、线程组、TLS、TID 写回等共享关系。
第一性原理概述：线程创建不是“复制一个完整进程”，而是在调度器里新增一个执行流，并让它共享足够多的进程级资源，使多个执行流看起来属于同一个进程。

线程模型：同 TGID，多 TID，共享 mm

第一性原理概述：进程是资源容器视角，线程是执行流视角。Linux 用 `task_struct` 表示执行流，用 `signal_struct/mm_struct/files_struct` 等引用关系表达“这些执行流属于同一进程”。

系统调用入口：`clone()` 构造 `kernel_clone_args`

/* kernel/fork.c */
SYSCALL_DEFINE5(clone, unsigned long, clone_flags, unsigned long, newsp,
                 int __user *, parent_tidptr,
                 int __user *, child_tidptr,
                 unsigned long, tls)
{
        struct kernel_clone_args args = {
                .flags          = (lower_32_bits(clone_flags) & ~CSIGNAL),
                .pidfd          = parent_tidptr,
                .child_tid      = child_tidptr,
                .parent_tid     = parent_tidptr,
                .exit_signal    = (lower_32_bits(clone_flags) & CSIGNAL),
                .stack          = newsp,
                .tls            = tls,
        };

        return kernel_clone(&args);
}

逐段解释：传统 `clone()` 的低 8 位 `CSIGNAL` 被拆成 `exit_signal`，其余位作为共享/命名空间/线程组 flags。用户线程库通常传入用户栈顶 `newsp`、TLS 值 `tls`、父/子 TID 写回地址和一组 `CLONE_*`。

MIPS 的原始 clone 参数顺序属于 man page 提到的“部分架构不同”类别；但当前源码最终都会归一化到 `kernel_clone_args`。

第一性原理概述：线程库在用户态决定“新执行流共享什么、从哪个栈运行、TLS 在哪里”，内核入口只把这些策略压缩成统一的创建参数。

`kernel_clone()`：复制 task，唤醒前仍不可运行

/* kernel/fork.c */
pid_t kernel_clone(struct kernel_clone_args *args)
{
        u64 clone_flags = args->flags;
        struct completion vfork;
        struct pid *pid;
        struct task_struct *p;
        int trace = 0;
        pid_t nr;

        if ((args->flags & CLONE_PIDFD) &&
            (args->flags & CLONE_PARENT_SETTID) &&
            (args->pidfd == args->parent_tid))
                return -EINVAL;

        if (!(clone_flags & CLONE_UNTRACED)) {
                if (clone_flags & CLONE_VFORK)
                        trace = PTRACE_EVENT_VFORK;
                else if (args->exit_signal != SIGCHLD)
                        trace = PTRACE_EVENT_CLONE;
                else
                        trace = PTRACE_EVENT_FORK;
                ...
        }

        p = copy_process(NULL, trace, NUMA_NO_NODE, args);
        add_latent_entropy();
        if (IS_ERR(p))
                return PTR_ERR(p);

        trace_sched_process_fork(current, p);
        pid = get_task_pid(p, PIDTYPE_PID);
        nr = pid_vnr(pid);

        if (clone_flags & CLONE_PARENT_SETTID)
                put_user(nr, args->parent_tid);
        ...
        wake_up_new_task(p);
        ...
        put_pid(pid);
        return nr;
}

`kernel_clone()` 负责 ptrace 事件分类、调用 `copy_process()` 构造新 task、给父线程写回 TID、处理 `vfork` completion，最后用 `wake_up_new_task()` 把新 task 放入调度器。

在 `copy_process()` 返回和 `wake_up_new_task()` 之前，新线程已经可被系统看见一部分信息，但还处于 `TASK_NEW`，不会真正运行。

第一性原理概述：创建线程需要先把调度实体完整构造好，再一次性发布到调度器。否则半初始化 task 被运行会破坏调度、信号、PID 和资源引用一致性。

`copy_process()`：线程创建的主装配线

/* kernel/fork.c */
static __latent_entropy struct task_struct *copy_process(
                                        struct pid *pid,
                                        int trace,
                                        int node,
                                        struct kernel_clone_args *args)
{
        struct task_struct *p;
        u64 clone_flags = args->flags;
        struct nsproxy *nsp = current->nsproxy;

        if ((clone_flags & (CLONE_NEWNS|CLONE_FS)) == (CLONE_NEWNS|CLONE_FS))
                return ERR_PTR(-EINVAL);
        if ((clone_flags & CLONE_THREAD) && !(clone_flags & CLONE_SIGHAND))
                return ERR_PTR(-EINVAL);
        if ((clone_flags & CLONE_SIGHAND) && !(clone_flags & CLONE_VM))
                return ERR_PTR(-EINVAL);
        if (clone_flags & CLONE_THREAD) {
                if ((clone_flags & (CLONE_NEWUSER | CLONE_NEWPID)) ||
                    (task_active_pid_ns(current) != nsp->pid_ns_for_children))
                        return ERR_PTR(-EINVAL);
        }
        ...
        p = dup_task_struct(current, node);
        if (!p)
                goto fork_out;
        p->set_child_tid = (clone_flags & CLONE_CHILD_SETTID) ? args->child_tid : NULL;
        p->clear_child_tid = (clone_flags & CLONE_CHILD_CLEARTID) ? args->child_tid : NULL;
        ...
        retval = copy_creds(p, clone_flags);
        ...
        retval = sched_fork(clone_flags, p);
        ...
        retval = copy_files(clone_flags, p);
        retval = copy_fs(clone_flags, p);
        retval = copy_sighand(clone_flags, p);
        retval = copy_signal(clone_flags, p);
        retval = copy_mm(clone_flags, p);
        retval = copy_namespaces(clone_flags, p);
        retval = copy_io(clone_flags, p);
        retval = copy_thread(clone_flags, args->stack, args->stack_size, p, args->tls);
        ...
}

逐段解释：开头先验证非法组合：线程必须共享 sighand，sighand 必须共享 VM；线程不能同时进入新 PID/user namespace；`CLONE_NEWNS` 不能和 `CLONE_FS` 同时共享根/cwd。

`dup_task_struct()` 复制当前 task 的主体和内核栈相关结构，然后按顺序复制或共享凭据、调度实体、文件表、fs 视图、信号处理、线程组信号结构、地址空间、命名空间、IO context，最后调用架构相关 `copy_thread()` 设置首次运行现场。

第一性原理概述：线程创建是资源图复制问题：哪些边指向父进程同一个对象，哪些边指向新对象，由 `CLONE_*` 决定。

`CLONE_*`：线程与进程的分界线

/* kernel/fork.c */
static int copy_mm(unsigned long clone_flags, struct task_struct *tsk)
{
        oldmm = current->mm;
        if (!oldmm)
                return 0;
        if (clone_flags & CLONE_VM) {
                mmget(oldmm);
                mm = oldmm;
                goto good_mm;
        }
        mm = dup_mm(tsk, current->mm);
good_mm:
        tsk->mm = mm;
        tsk->active_mm = mm;
        return 0;
}

static int copy_files(unsigned long clone_flags, struct task_struct *tsk)
{
        oldf = current->files;
        if (clone_flags & CLONE_FILES) {
                atomic_inc(&oldf->count);
                goto out;
        }
        newf = dup_fd(oldf, NR_OPEN_MAX, &error);
        tsk->files = newf;
        return error;
}

static int copy_sighand(unsigned long clone_flags, struct task_struct *tsk)
{
        if (clone_flags & CLONE_SIGHAND) {
                refcount_inc(&current->sighand->count);
                return 0;
        }
        sig = kmem_cache_alloc(sighand_cachep, GFP_KERNEL);
        memcpy(sig->action, current->sighand->action, sizeof(sig->action));
        ...
}

static int copy_signal(unsigned long clone_flags, struct task_struct *tsk)
{
        if (clone_flags & CLONE_THREAD)
                return 0;
        sig = kmem_cache_zalloc(signal_cachep, GFP_KERNEL);
        tsk->signal = sig;
        sig->nr_threads = 1;
        ...
}
flag语义线程创建中的意义
`CLONE_VM`共享 `mm_struct`同一进程地址空间，堆/全局变量互相可见。
`CLONE_THREAD`加入同一线程组共享 TGID，`getpid()` 返回线程组 leader 的 PID。
`CLONE_SIGHAND`共享信号处理表signal disposition 是进程级属性。
`CLONE_FILES`共享文件描述符表一个线程 close/open 影响同进程其它线程。
`CLONE_FS`共享 root/cwd/umask`chdir()` 等影响同进程线程。
`CLONE_SETTLS`设置新线程 TLS支撑 `errno`、线程局部变量和 pthread 自身控制块。
`CLONE_CHILD_SETTID/CLEARTID`写入/退出清零 TID 并唤醒 futex支撑 `pthread_create()` 返回、`pthread_join()` 和线程退出同步。
第一性原理概述：线程的“轻量”来自引用共享而不是少创建对象。内核仍创建完整 task，但大量资源指针指向同一份对象。

逐段解释：线程有自己的 `pid`，这是内核 TID；但 `CLONE_THREAD` 时 `tgid` 继承当前线程组的 TGID，`group_leader` 指向同一个 leader。非 leader 线程不作为普通子进程加入父进程 children 列表，而是加入 leader 的 `thread_group` 和共享 `signal->thread_head`。

`exit_signal = -1` 表示普通线程退出时不向父进程发送 SIGCHLD；只有整个线程组结束时父进程才观察到进程级退出。

第一性原理概述：用户看到的“进程 PID”是 TGID，内核调度的“线程 ID”是 PIDTYPE_PID。线程创建同时要维护这两套命名视图。

MIPS `copy_thread()`：构造子线程第一次运行的寄存器现场

/* arch/mips/kernel/process.c */
int copy_thread(unsigned long clone_flags, unsigned long usp,
                unsigned long kthread_arg, struct task_struct *p,
                unsigned long tls)
{
        struct thread_info *ti = task_thread_info(p);
        struct pt_regs *childregs, *regs = current_pt_regs();
        unsigned long childksp;

        childksp = (unsigned long)task_stack_page(p) + THREAD_SIZE - 32;
        childregs = (struct pt_regs *) childksp - 1;
        childksp = (unsigned long) childregs;
        p->thread.cp0_status = (read_c0_status() & ~(ST0_CU2|ST0_CU1)) | ST0_KERNEL_CUMASK;
        if (unlikely(p->flags & (PF_KTHREAD | PF_IO_WORKER))) {
                memset(childregs, 0, sizeof(struct pt_regs));
                ti->addr_limit = KERNEL_DS;
                p->thread.reg16 = usp; /* fn */
                p->thread.reg17 = kthread_arg;
                p->thread.reg29 = childksp;
                p->thread.reg31 = (unsigned long) ret_from_kernel_thread;
                ...
                return 0;
        }

        *childregs = *regs;
        childregs->regs[7] = 0; /* Clear error flag */
        childregs->regs[2] = 0; /* Child gets zero as return value */
        if (usp)
                childregs->regs[29] = usp;
        ti->addr_limit = USER_DS;

        p->thread.reg29 = (unsigned long) childregs;
        p->thread.reg31 = (unsigned long) ret_from_fork;
        ...
        if (clone_flags & CLONE_SETTLS)
                ti->tp_value = tls;
        return 0;
}

用户线程路径复制父线程的 `pt_regs`，然后把返回值寄存器 `$v0` 即 `regs[2]` 改为 0，清错误标志 `regs[7]`。这就是 clone/fork 在子线程中“返回 0”的根源。若用户传入新栈 `usp`，则把 `$sp` 即 `regs[29]` 改成该栈顶。

`p->thread.reg31 = ret_from_fork` 让调度器第一次切到该 task 时从 `ret_from_fork` 继续；返回用户态时恢复 `childregs`。`CLONE_SETTLS` 会写 `thread_info.tp_value`，支撑 MIPS TLS。

第一性原理概述：新线程并不是从 C 函数里“被调用”出来，而是拥有一份伪造好的陷入现场。调度器第一次恢复它时，看起来就像系统调用刚刚在子线程中返回。

/* kernel/fork.c */
pid_t kernel_thread(int (*fn)(void *), void *arg, unsigned long flags)
{
        struct kernel_clone_args args = {
                .flags          = ((lower_32_bits(flags) | CLONE_VM |
                                    CLONE_UNTRACED) & ~CSIGNAL),
                .exit_signal    = (lower_32_bits(flags) & CSIGNAL),
                .stack          = (unsigned long)fn,
                .stack_size     = (unsigned long)arg,
        };

        return kernel_clone(&args);
}

内核线程由 `kthreadd` 代创建。`__kthread_create_on_node()` 把请求挂到 `kthread_create_list`，唤醒 `kthreadd_task`，由 `create_kthread()` 调 `kernel_thread(kthread, create, ...)`。

在 MIPS `copy_thread()` 的内核线程分支中，`usp` 被解释为函数指针 `fn`，`kthread_arg` 被解释为参数，首次返回地址是 `ret_from_kernel_thread`。这和用户线程的 `ret_from_fork` 分支不同。

第一性原理概述：用户线程有用户地址空间和用户栈；内核线程没有用户态执行现场，只需要一个内核函数、参数和内核栈。但调度器看到的依旧是 task。

`clone3()`：结构化参数与栈方向修正

/* kernel/fork.c */
SYSCALL_DEFINE2(clone3, struct clone_args __user *, uargs, size_t, size)
{
        int err;
        struct kernel_clone_args kargs;
        pid_t set_tid[MAX_PID_NS_LEVEL];

        kargs.set_tid = set_tid;
        err = copy_clone_args_from_user(&kargs, uargs, size);
        if (err)
                return err;

        if (!clone3_args_valid(&kargs))
                return -EINVAL;

        return kernel_clone(&kargs);
}

static inline bool clone3_stack_valid(struct kernel_clone_args *kargs)
{
        if (kargs->stack == 0) {
                if (kargs->stack_size > 0)
                        return false;
        } else {
                if (kargs->stack_size == 0)
                        return false;
                if (!access_ok((void __user *)kargs->stack, kargs->stack_size))
                        return false;
#if !defined(CONFIG_STACK_GROWSUP) && !defined(CONFIG_IA64)
                kargs->stack += kargs->stack_size;
#endif
        }
        return true;
}

`clone3()` 把传统 clone 零散参数整理成 `struct clone_args`，并支持 `stack_size`、`set_tid`、`cgroup` 等扩展。当前 MIPS 配置声明了 `__ARCH_WANT_SYS_CLONE3`，系统调用表中也有 `__sys_clone3`。

对向下增长栈的架构，`clone3_stack_valid()` 会把用户传入的栈底地址加上 `stack_size`，转换成传统 clone 所需的栈顶。

第一性原理概述：clone3 不改变创建线程的内核核心，只是把 ABI 从“位置参数 + 架构差异”升级成“版本化结构体 + 明确尺寸”。

void sched_post_fork(struct task_struct *p, struct kernel_clone_args *kargs)
{
        raw_spin_lock_irqsave(&p->pi_lock, flags);
        rseq_migrate(p);
        __set_task_cpu(p, smp_processor_id());
        if (p->sched_class->task_fork)
                p->sched_class->task_fork(p);
        raw_spin_unlock_irqrestore(&p->pi_lock, flags);
        uclamp_post_fork(p);
}

`sched_fork()` 初始化调度实体，并把新 task 标为 `TASK_NEW`，确保外部事件无法在半初始化期间把它插入运行队列。`copy_process()` 完成后，`kernel_clone()` 调 `wake_up_new_task(p)`，这才正式进入可调度状态。

第一性原理概述：线程创建完成的标志不是分配了 `task_struct`，而是该 task 的资源图、寄存器现场和调度实体都一致，并被安全放入运行队列。

共享与私有属性对照

属性用户线程通常状态源码依据
地址空间共享`copy_mm()` 中 `CLONE_VM` 执行 `mmget(oldmm)`。
线程组/TGID共享 TGID，各自有 TID`CLONE_THREAD` 时 `p->tgid = current->tgid`，仍 `attach_pid(PIDTYPE_PID)`。
信号处理表共享 dispositions，私有 pending/mask 细节`copy_sighand()` 共享 `sighand`，每 task 仍有 `pending/blocked`。
文件描述符表共享`copy_files()` 中 `CLONE_FILES` 增加 `oldf->count`。
root/cwd/umask共享`copy_fs()` 中 `CLONE_FS` 增加 `fs->users`。
用户栈私有，由 pthread 分配后传给 clone`copy_thread()` 用 `usp` 更新 MIPS `$sp`。
TLS私有`CLONE_SETTLS` 时 `ti->tp_value = tls`。
内核栈/pt_regs私有`dup_task_struct()` 和 `copy_thread()` 为新 task 构造自己的内核栈现场。
第一性原理概述：线程共享的是进程语义上应当一致的资源，私有的是执行流推进所必需的状态：栈、寄存器、TID、调度实体和 TLS。

整体总结

入口统一

`pthread_create()` 在用户态准备栈/TLS 后调用 clone；`clone()`、`clone3()`、`kernel_thread()` 最终都进入 `kernel_clone()`。

核心构造

`copy_process()` 先校验 `CLONE_*` 组合，再复制 `task_struct`，按 flag 共享或复制 mm、files、fs、sighand、signal 等资源。

线程身份

`CLONE_THREAD` 让新 task 继承 TGID 和 group leader，同时保留自己的 TID、内核栈、调度实体和 TLS。

首次运行

MIPS `copy_thread()` 构造 `pt_regs`：子线程 clone 返回 0，`$sp` 指向新用户栈，首次调度从 `ret_from_fork` 返回用户态。

一句话：Linux 线程创建就是创建一个新 `task_struct`，但让它共享足够多的父线程资源，并为它准备一份独立的调度现场和用户栈。
