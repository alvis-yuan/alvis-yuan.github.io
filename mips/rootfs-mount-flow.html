根文件系统从 rootfs 到真实根设备的挂载过程

本文按当前仓库源码解释根文件系统挂载。官方文档说明通用 rootfs/initramfs/initrd 机制；当前源码树没有 `init/initramfs.c` 和 `do_mounts_rd.c`，配置也未启用 `CONFIG_BLK_DEV_INITRD`，所以实际主线是：先创建空 rootfs，再由 `prepare_namespace()` 挂载真实块设备根到 `/root`，最后把它移动为 `/`。

VFS 初始化先于真实根挂载

/* fs/dcache.c */

        names_cachep = kmem_cache_create_usercopy("names_cache", PATH_MAX, 0,

        files_maxfiles_init();
        mnt_init();

逐段解释：`vfs_caches_init()` 初始化路径名、dentry、inode、file、mount、块设备和字符设备相关缓存。根文件系统挂载依赖这些 VFS 基础设施，因为挂载的结果要表现为 dentry、superblock、vfsmount 和 mount namespace。

`mnt_init()` 是本文第一个关键点，它会初始化 mount hash、sysfs、shmem/rootfs，并创建初始 mount tree。

第一性原理概述：文件系统不是块设备本身，而是 VFS 对超级块、目录项、inode 和挂载关系的统一视图。真实根挂载前必须先有这些抽象容器。

rootfs 与初始 mount namespace

/* fs/namespace.c */
static void __init init_mount_tree(void)
{
        struct vfsmount *mnt;
        struct mount *m;
        struct mnt_namespace *ns;
        struct path root;

        mnt = vfs_kern_mount(&rootfs_fs_type, 0, "rootfs", NULL);
        if (IS_ERR(mnt))
                panic("Can't create rootfs");

        ns = alloc_mnt_ns(&init_user_ns, false);
        if (IS_ERR(ns))
                panic("Can't allocate initial namespace");
        m = real_mount(mnt);
        m->mnt_ns = ns;
        ns->root = m;
        ns->mounts = 1;
        list_add(&m->mnt_list, &ns->list);
        init_task.nsproxy->mnt_ns = ns;
        get_mnt_ns(ns);

        root.mnt = mnt;
        root.dentry = mnt->mnt_root;
        mnt->mnt_flags |= MNT_LOCKED;

        set_fs_pwd(current->fs, &root);
        set_fs_root(current->fs, &root);
}

void __init mnt_init(void)
{
        ...
        shmem_init();
        init_rootfs();
        init_mount_tree();
}

逐段解释：`vfs_kern_mount(&rootfs_fs_type, ...)` 创建 rootfs 的内核挂载。随后分配初始 `mnt_namespace`，把 rootfs mount 放入 namespace，并设置 `init_task.nsproxy->mnt_ns`。

`set_fs_pwd()` 和 `set_fs_root()` 把当前内核线程的当前目录和根目录都设为 rootfs 根。此后 `/dev/console`、`/root` 等路径才能被内核初始化代码正常解析。

第一性原理概述：路径解析总要从某个根开始。rootfs 是内核给自己的最小根，使后续挂载真实根之前已有可工作的 VFS 命名空间。

当前仓库的默认 rootfs 内容

/* init/noinitramfs.c */
static int __init default_rootfs(void)
{
        int err;

        err = init_mkdir("/dev", 0755);
        if (err < 0)
                goto out;

        err = init_mknod("/dev/console", S_IFCHR | S_IRUSR | S_IWUSR,
                        new_encode_dev(MKDEV(5, 1)));
        if (err < 0)
                goto out;

        err = init_mkdir("/root", 0700);
        if (err < 0)
                goto out;

        return 0;

out:
        printk(KERN_WARNING "Failed to create a rootfs\n");
        return err;
}
rootfs_initcall(default_rootfs);

逐段解释：当前源码没有 initramfs 解包文件，只有 `noinitramfs.c`。它通过 `rootfs_initcall()` 在 rootfs 阶段创建 `/dev`、`/dev/console` 和 `/root`。

`/dev/console` 供 `console_on_rootfs()` 打开标准输入输出错误；`/root` 是 `do_mount_root()` 后续挂载真实根的临时挂载点。这个默认 rootfs 非常小，只满足内核继续启动所需的最小路径。

第一性原理概述：临时根不需要完整用户空间，只需要为内核后续操作提供几个确定路径：控制台设备和真实根挂载点。

`kernel_init_freeable()` 决定是否交给早期用户态

/* init/main.c */

{

逐段解释：`do_basic_setup()` 会运行各级 initcall，包括 `rootfs_initcall(default_rootfs)`。`console_on_rootfs()` 打开 `/dev/console` 并复制为 fd 0、1、2。

`ramdisk_execute_command` 默认是 `"/init"`。如果 rootfs 中 `/init` 可执行，内核不调用 `prepare_namespace()`，而是在后续 `kernel_init()` 中直接执行它，让早期用户态自己挂载真实根。当前默认 rootfs 不创建 `/init`，所以 `init_eaccess()` 失败，内核进入 `prepare_namespace()`。

第一性原理概述：如果早期用户态存在，根设备选择属于策略问题，交给用户态更灵活；如果没有 `/init`，内核必须走内置的根设备挂载逻辑。

...
__setup("rootflags=", root_data_setup);
__setup("rootfstype=", fs_names_setup);
__setup("rootdelay=", root_delay_setup);

逐段解释：默认根挂载参数是只读且安静：`MS_RDONLY | MS_SILENT`。`ro` 保持只读，`rw` 清除只读标志。

`root=` 保存到 `saved_root_name`，稍后在 `prepare_namespace()` 中解析为 `ROOT_DEV`。`rootfstype=` 限制尝试的文件系统类型，`rootflags=` 传给具体文件系统，`rootdelay=` 在挂载前固定等待，`rootwait` 则在设备尚未出现时循环等待。

第一性原理概述：根挂载需要四类输入：设备在哪里、用什么文件系统解释、用什么挂载参数、设备没准备好时等不等。命令行就是早期内核获取这些策略的通道。

`prepare_namespace()`：等待设备、解析 root、挂载并切根

/* init/do_mounts.c */
void __init prepare_namespace(void)
{
        if (root_delay) {
                printk(KERN_INFO "Waiting %d sec before mounting root device...\n",
                       root_delay);
                ssleep(root_delay);
        }

        wait_for_device_probe();

        md_run_setup();

        if (saved_root_name[0]) {
                root_device_name = saved_root_name;
                if (!strncmp(root_device_name, "mtd", 3) ||
                    !strncmp(root_device_name, "ubi", 3)) {
                        mount_block_root(root_device_name, root_mountflags);
                        goto out;
                }
                ROOT_DEV = name_to_dev_t(root_device_name);
                if (strncmp(root_device_name, "/dev/", 5) == 0)
                        root_device_name += 5;
        }

        if (initrd_load())
                goto out;

        if ((ROOT_DEV == 0) && root_wait) {
                printk(KERN_INFO "Waiting for root device %s...\n",
                        saved_root_name);
                while (driver_probe_done() != 0 ||
                        (ROOT_DEV = name_to_dev_t(saved_root_name)) == 0)
                        msleep(5);
                async_synchronize_full();
        }

        mount_root();
out:
        devtmpfs_mount();
        init_mount(".", "/", NULL, MS_MOVE, NULL);
        init_chroot(".");
}

逐段解释：函数先处理 `rootdelay`，再调用 `wait_for_device_probe()` 等待已知设备探测完成，随后启动 md RAID 设置。

如果指定了 `root=`，`mtd`/`ubi` 直接按名字挂载；普通块设备通过 `name_to_dev_t()` 转成 `dev_t`。当前配置中 `initrd_load()` 是 false，所以会继续走 `mount_root()`。

如果 `ROOT_DEV` 仍为 0 且指定 `rootwait`，内核会等驱动探测结束并反复解析 root 设备名，直到设备出现。挂载完成后，`devtmpfs_mount()` 挂载设备节点文件系统，然后 `MS_MOVE` 把当前目录挂载移动到 `/`，最后 `chroot(".")` 让进程根目录指向新根。

第一性原理概述：根挂载必须在设备发现和路径命名之间建立同步点。否则内核可能在块设备或分区还没注册时就错误地认定根设备不存在。

`name_to_dev_t()`：把 root= 字符串变成设备号

/* init/do_mounts.c */
/*
 *      Convert a name into device number.  We accept the following variants:
 *
 *      1) <hex_major><hex_minor> device number in hexadecimal represents itself
 *      2) /dev/nfs represents Root_NFS (0xff)
 *      3) /dev/<disk_name> represents the device number of disk
 *      4) /dev/<disk_name><decimal> represents the device number
 *      6) PARTUUID=00112233-4455-6677-8899-AABBCCDDEEFF
 *      8) <major>:<minor> major and minor number of the device separated by
 *         a colon.
 *      9) PARTLABEL=<name> with name being the GPT partition label.
 *      10) /dev/cifs represents Root_CIFS (0xfe)
 */
dev_t name_to_dev_t(const char *name)
{
        char s[32];
        char *p;
        dev_t res = 0;
        int part;

#ifdef CONFIG_BLOCK
        if (strncmp(name, "PARTUUID=", 9) == 0) {
                name += 9;
                res = devt_from_partuuid(name);
                if (!res)
                        goto fail;
                goto done;
        } else if (strncmp(name, "PARTLABEL=", 10) == 0) {
                struct device *dev;

                dev = class_find_device(&block_class, NULL, name + 10,
                                        &match_dev_by_label);
                if (!dev)
                        goto fail;

                res = dev->devt;
                put_device(dev);
                goto done;
        }
#endif
        ...
done:
        return res;
}

逐段解释：这个函数接受多种 `root=` 形式：数字设备号、`major:minor`、`/dev/sda1`、`PARTUUID=`、`PARTLABEL=`、`/dev/nfs`、`/dev/cifs` 等。

当前启用 `CONFIG_BLOCK`，所以 PARTUUID/PARTLABEL 和块设备名查找路径可用。若找不到，返回 0；`rootwait` 场景下会稍后重试。

第一性原理概述：内核最终挂载块设备需要 `dev_t`，而用户和 bootloader 提供的是稳定性不同的名字。`name_to_dev_t()` 是从人类/固件命名到内核设备号的转换层。

`mount_root()`：选择 NFS、CIFS 或块设备根

/* init/do_mounts.c */
void __init mount_root(void)
{
#ifdef CONFIG_ROOT_NFS
        if (ROOT_DEV == Root_NFS) {
                if (!mount_nfs_root())
                        printk(KERN_ERR "VFS: Unable to mount root fs via NFS.\n");
                return;
        }
#endif
#ifdef CONFIG_CIFS_ROOT
        if (ROOT_DEV == Root_CIFS) {
                if (!mount_cifs_root())
                        printk(KERN_ERR "VFS: Unable to mount root fs via SMB.\n");
                return;
        }
#endif
#ifdef CONFIG_BLOCK
        {
                int err = create_dev("/dev/root", ROOT_DEV);

                if (err < 0)
                        pr_emerg("Failed to create /dev/root: %d\n", err);
                mount_block_root("/dev/root", root_mountflags);
        }
#endif
}

逐段解释：源码保留了 NFS 和 CIFS 根的条件分支，但当前配置没有启用它们。实际路径进入 `CONFIG_BLOCK` 分支。

`create_dev("/dev/root", ROOT_DEV)` 在 rootfs 上创建一个临时块设备节点，随后 `mount_block_root("/dev/root", root_mountflags)` 尝试把它作为真实根挂载。

第一性原理概述：根文件系统可以来自网络或块设备，但 VFS 挂载最终都需要一个 source、一个 fstype、一个 mountpoint 和一组 flags。`mount_root()` 负责选择 source 类别。

块设备根：枚举文件系统并挂载到 `/root`

/* init/do_mounts.c */
static int __init do_mount_root(const char *name, const char *fs,
                                 const int flags, const void *data)
{
        struct super_block *s;
        struct page *p = NULL;
        char *data_page = NULL;
        int ret;

        if (data) {
                /* init_mount() requires a full page as fifth argument */
                p = alloc_page(GFP_KERNEL);
                if (!p)
                        return -ENOMEM;
                data_page = page_address(p);
                strncpy(data_page, data, PAGE_SIZE);
        }

        ret = init_mount(name, "/root", fs, flags, data_page);
        if (ret)
                goto out;

        init_chdir("/root");
        s = current->fs->pwd.dentry->d_sb;
        ROOT_DEV = s->s_dev;
        printk(KERN_INFO
               "VFS: Mounted root (%s filesystem)%s on device %u:%u.\n",
               s->s_type->name,
               sb_rdonly(s) ? " readonly" : "",
               MAJOR(ROOT_DEV), MINOR(ROOT_DEV));

out:
        if (p)
                put_page(p);
        return ret;
}

void __init mount_block_root(char *name, int flags)
{
        struct page *page = alloc_page(GFP_KERNEL);
        char *fs_names = page_address(page);
        char *p;
        char b[BDEVNAME_SIZE];

        scnprintf(b, BDEVNAME_SIZE, "unknown-block(%u,%u)",
                  MAJOR(ROOT_DEV), MINOR(ROOT_DEV));
        get_fs_names(fs_names);
retry:
        for (p = fs_names; *p; p += strlen(p)+1) {
                int err = do_mount_root(name, p, flags, root_mount_data);
                switch (err) {
                        case 0:
                                goto out;
                        case -EACCES:
                        case -EINVAL:
                                continue;
                }
                ...
        }
        if (!(flags & SB_RDONLY)) {
                flags |= SB_RDONLY;
                goto retry;
        }
        ...
out:
        put_page(page);
}

逐段解释：`mount_block_root()` 先取得文件系统候选列表。如果指定了 `rootfstype=`，候选来自命令行；否则来自已注册文件系统列表。

它逐个调用 `do_mount_root()`。`-EACCES` 或 `-EINVAL` 通常表示该文件系统类型不适合当前设备或参数错误，可以继续尝试下一个类型。若读写挂载失败，代码会强制加上只读标志重试一次。

`do_mount_root()` 真正调用 `init_mount(name, "/root", fs, flags, data_page)`，成功后切换当前目录到 `/root`，读取 superblock 的 `s_dev` 更新 `ROOT_DEV`，并打印 “VFS: Mounted root ...”。

第一性原理概述：块设备只提供字节块，文件系统驱动才解释目录树。内核必须枚举可能的文件系统类型，直到某个驱动能从该设备读出合法 superblock。

VFS 挂载调用链：`init_mount → path_mount → do_new_mount → vfs_get_tree`

/* fs/init.c */
int __init init_mount(const char *dev_name, const char *dir_name,
                const char *type_page, unsigned long flags, void *data_page)
{
        struct path path;
        int ret;

        ret = kern_path(dir_name, LOOKUP_FOLLOW, &path);
        if (ret)
                return ret;
        ret = path_mount(dev_name, &path, type_page, flags, data_page);
        path_put(&path);
        return ret;
}

/* fs/namespace.c */
int path_mount(const char *dev_name, struct path *path,
                const char *type_page, unsigned long flags, void *data_page)
{
        ...
        if (flags & MS_MOVE)
                return do_move_mount_old(path, dev_name);

        return do_new_mount(path, type_page, sb_flags, mnt_flags, dev_name,

}

逐段解释：`init_mount()` 是早期内核专用版本，不从用户地址空间取参数，也不走文件描述符。它先把挂载点路径解析成 `struct path`，再调用 `path_mount()`。

`path_mount()` 解析 mount flags，把 superblock flags 和 per-mount flags 分离。普通新挂载进入 `do_new_mount()`；后续切根的 `MS_MOVE` 则进入 `do_move_mount_old()`。

/* fs/namespace.c */
static int do_new_mount(struct path *path, const char *fstype, int sb_flags,
                        int mnt_flags, const char *name, void *data)
{
        struct file_system_type *type;
        struct fs_context *fc;
        const char *subtype = NULL;
        int err = 0;

        if (!fstype)
                return -EINVAL;

        type = get_fs_type(fstype);
        if (!type)
                return -ENODEV;
        ...
        fc = fs_context_for_mount(type, sb_flags);
        put_filesystem(type);
        if (IS_ERR(fc))
                return PTR_ERR(fc);
        ...
        if (!err && name)
                err = vfs_parse_fs_string(fc, "source", name, strlen(name));
        if (!err)
                err = parse_monolithic_mount_data(fc, data);
        if (!err && !mount_capable(fc))
                err = -EPERM;
        if (!err)
                err = vfs_get_tree(fc);
        if (!err)
                err = do_new_mount_fc(fc, path, mnt_flags);

        put_fs_context(fc);
        return err;
}

逐段解释：`do_new_mount()` 根据文件系统名查找 `file_system_type`，创建 `fs_context`，把 source 设为 `/dev/root`，解析挂载参数，然后调用 `vfs_get_tree()` 让具体文件系统获得 mountable root。

/* fs/super.c */
int vfs_get_tree(struct fs_context *fc)
{
        struct super_block *sb;
        int error;

        if (fc->root)
                return -EBUSY;

        error = fc->ops->get_tree(fc);
        if (error < 0)
                return error;

        if (!fc->root) {
                pr_err("Filesystem %s get_tree() didn't set fc->root\n",
                       fc->fs_type->name);
                BUG();
        }

        sb = fc->root->d_sb;
        WARN_ON(!sb->s_bdi);

        smp_wmb();
        sb->s_flags |= SB_BORN;

        error = security_sb_set_mnt_opts(sb, fc->security, 0, NULL);
        if (unlikely(error)) {
                fc_drop_locked(fc);
                return error;
        }
        ...
        return 0;
}

逐段解释：`vfs_get_tree()` 调用具体文件系统的 `get_tree()` 创建或取得 superblock，并要求文件系统把根 dentry 放在 `fc->root`。成功后设置 `SB_BORN`，应用安全挂载选项，再由 `do_new_mount_fc()` 把新 mount 加到 namespace 树。

第一性原理概述：挂载的本质是把“某个文件系统实例的根 dentry”接到“当前命名空间中的某个挂载点”。设备、文件系统名和参数只是构造这个实例的输入。

把 `/root` 移到 `/`：`MS_MOVE` 与 `chroot`

/* init/do_mounts.c */
out:
        devtmpfs_mount();
        init_mount(".", "/", NULL, MS_MOVE, NULL);
        init_chroot(".");

逐段解释：`do_mount_root()` 成功后已经 `init_chdir("/root")`，所以此处的 `"."` 就是新挂载的真实根。`init_mount(".", "/", NULL, MS_MOVE, NULL)` 触发 `path_mount()` 的 `MS_MOVE` 分支，把当前目录对应的挂载移动到 `/`。

`devtmpfs_mount()` 在切根前后确保 `/dev` 设备节点文件系统可用。`init_chroot(".")` 把当前进程根目录改到新根，后续执行 `/sbin/init` 等路径时就从真实根解析。

第一性原理概述：切根改变的是 mount namespace 中“/ 指向哪个 mount”的拓扑关系。文件没有搬家，块设备没有复制，只是 VFS 的根引用换到真实根。

根挂载后执行第一个用户态 init

/* init/main.c */
static int run_init_process(const char *init_filename)
{
        const char *const *p;

        argv_init[0] = init_filename;
        pr_info("Run %s as init process\n", init_filename);
        ...
        return kernel_execve(init_filename, argv_init, envp_init);
}

static int __ref kernel_init(void *unused)
{
        int ret;

        kernel_init_freeable();
        ...
        if (ramdisk_execute_command) {
                ret = run_init_process(ramdisk_execute_command);
                if (!ret)
                        return 0;
                pr_err("Failed to execute %s (error %d)\n",
                       ramdisk_execute_command, ret);
        }
        ...
        if (!try_to_run_init_process("/sbin/init") ||
            !try_to_run_init_process("/etc/init") ||
            !try_to_run_init_process("/bin/init") ||
            !try_to_run_init_process("/bin/sh"))
                return 0;

        panic("No working init found.  Try passing init= option to kernel. "
              "See Linux Documentation/admin-guide/init.rst for guidance.");
}

逐段解释：如果早期 `/init` 存在，`ramdisk_execute_command` 保持有效，`kernel_init()` 会先执行它。当前默认 rootfs 没有 `/init`，所以前面已把它置空并挂载真实根。

真实根就位后，`kernel_init()` 按 `init=`、`CONFIG_DEFAULT_INIT`、`/sbin/init`、`/etc/init`、`/bin/init`、`/bin/sh` 的顺序尝试执行。成功后 PID 1 进入用户态，内核启动阶段完成。

第一性原理概述：挂载根文件系统只是为了让第一个用户态程序有可解析的路径和可读取的文件。最终目标不是“挂载”本身，而是成功 `execve()` PID 1。

整体总结

临时根

`mnt_init()` 创建 rootfs 和初始 mount namespace。当前 `noinitramfs.c` 只创建 `/dev`、`/dev/console`、`/root`。

决策点

`kernel_init_freeable()` 检查 `/init`。存在则早期用户态接管；不存在则内核调用 `prepare_namespace()`。

真实根

`prepare_namespace()` 等待设备、解析 `root=`、调用 `mount_root()`。当前配置实际走块设备根，挂载到 `/root`。

切换

成功后 `MS_MOVE` 把当前 `/root` 挂载移动到 `/`，`chroot(".")` 更新根目录，然后 `kernel_init()` 执行真正的 init。

从第一性原理看，根挂载过程就是把“可启动的最小 VFS 根”替换成“包含用户态 init 的真实目录树”。所有等待、解析、枚举文件系统和移动 mount 的代码，都是为了让 PID 1 的路径解析落在正确的根上。
