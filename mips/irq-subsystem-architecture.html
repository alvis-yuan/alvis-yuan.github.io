Linux generic IRQ handling / irq_domain / DeviceTree

中断子系统设计架构

这份文档只基于 Linux 官方文档和当前仓库代码编写。它解释每个概念为什么被引入：先从硬件事实出发，再看 DTS 如何描述中断拓扑，SoC irqchip 如何注册控制器，最后看 generic IRQ core 如何把一次电平变化交给设备驱动的 handler。

概念解决的问题

DTS解决硬件描述硬编码

第一性原理：内核需要知道设备接在哪个中断控制器、用哪根线、触发类型是什么。如果这些都写死在板级 C 文件里，一个 SoC 多块板子会产生大量重复代码。DT 把硬件事实变成数据，让驱动按数据初始化。

irq_domain解决编号命名空间冲突

每个中断控制器都可能从 0 开始编号自己的输入线。多个控制器共存时，硬件号会重复。irq_domain 给每个控制器建立局部 hwirq 到全局 Linux IRQ 的映射，并提供反向查找。

irq_chip解决寄存器差异

中断核心知道“需要 mask/ack/unmask/eoi”，但不知道每个芯片寄存器地址和位语义。irq_chip 把这些硬件动作封装成回调，flow handler 只调用语义化操作。

flow handler解决触发语义差异

电平中断、边沿中断、per-CPU 中断的正确处理顺序不同。generic IRQ 把“流程”从硬件寄存器操作中拆出来，例如 level flow 会先 mask/ack，再调用 action，最后按条件 unmask。

irq_desc解决每条 IRQ 的状态聚合

一次中断不只是一个号码，还包含状态、锁、flow handler、irq_data、action 链、统计等。irq_desc 是 Linux IRQ 的中心状态对象。

irqaction解决驱动 handler 安装

设备驱动通过 request_irq() 或 request_threaded_irq() 把 handler 安装到 desc 的 action 链。共享中断时，一条 IRQ 可以有多个 action。

X2600 控制器拓扑

官方 binding 规定：能产生中断的 client 节点使用 interrupts 或 interrupts-extended；interrupt-parent 指向中断控制器，且可以从父节点继承；中断控制器节点必须有空布尔属性 interrupt-controller，并用 #interrupt-cells 定义一个 interrupt specifier 需要几个 cell。

当前 X2600 DTS 中有两个早期中断控制器节点：CPU interrupt controller 是根控制器，core intc 级联到 CPU 控制器的 CORE_INTC_IRQ 上。

cpuintc: interrupt-controller {
    #interrupt-cells = <1>;
    interrupt-controller;
    compatible = "ingenic,cpu-interrupt-controller";
};

core_intc: core-intc@0x12300000 {
    compatible = "ingenic,core-intc";
    reg = <0x12300000 0x1000>, <0x10001000 0x1000>;
    interrupt-controller;
    #interrupt-cells = <1>;
    cpu-intc-map = <0 0x000>, <1 0x100>;
    interrupt-parent = <&cpuintc>;
    interrupts = <CORE_INTC_IRQ>;
};

uart0: serial@0x10030000 {
    interrupt-parent = <&core_intc>;
    interrupts = <IRQ_UART0>;
};

图 · X2600 实际中断控制器拓扑：core_intc 级联到 cpuintc，UART/GPIO/WDT 等外设使用 core_intc 的本地 hwirq 编号。

irqchip 驱动如何被 DTS 找到

include/linux/irqchip.h 定义 IRQCHIP_DECLARE(name, compat, fn)，用于建立 DT compatible 字符串与初始化函数的关联。drivers/of/irq.c 的 of_irq_init() 扫描匹配节点，过滤没有 interrupt-controller 或 disabled 的节点，并按父控制器先于子控制器的顺序调用初始化函数。

IRQCHIP_DECLARE(cpu_intc,
    "ingenic,cpu-interrupt-controller",
    ingenic_cpu_irq_of_init);

IRQCHIP_DECLARE(ingenic_intc,
    "ingenic,core-intc",
    ingenic_intc_init);

这一步解决的是“早期启动阶段还没有普通 platform driver 绑定，根中断控制器却必须先工作”的问题。中断控制器是许多驱动的前置条件，所以它通过 irqchip early init 表优先初始化。

irq_domain 如何建立映射

官方 irq_domain 文档明确说：irq_domain 处理给定中断域内硬件号和虚拟中断号之间的映射，domain callback 负责在映射发生时设置对应 irq_desc 的 irq_chip。

当前仓库 CPU intc 和 core intc 都使用 legacy domain：CPU domain 把 8 条 MIPS CPU IRQ 固定映射到 MIPS_CPU_IRQ_BASE 起始的 Linux IRQ；core intc 把 INTC_NR_IRQS 条 SoC 外设中断固定映射到 IRQ_INTC_BASE 起始的 Linux IRQ。

/* CPU intc */
irq_domain_add_legacy(of_node, 8, MIPS_CPU_IRQ_BASE, 0,
    &ingenic_cpu_intc_irq_domain_ops, NULL);

/* core intc */
irq_domain_add_legacy(node, INTC_NR_IRQS, 8, 0,
    &ingenic_irq_domain_ops, chip);

映射时，domain core 调用 domain->ops->map()。X2600 的 map() 做了两件决定性事情：设置 chip 私有数据；给该 Linux IRQ 绑定 irq_chip 和 flow handler。

设备驱动申请 IRQ

设备驱动通常不直接理解 DTS 中的硬件号，而是调用 OF API 或 platform resource API 取得 Linux IRQ。官方 DeviceTree Kernel API 对 irq_of_parse_and_map() 的描述是：它把 of_irq_parse_one() 与 irq_create_of_mapping() 串起来，方便调用者。

request_threaded_irq() 创建 irqaction，检查共享中断规则、触发类型和 ONESHOT 约束，然后把 action 挂到 desc->action 链上。第一次安装 action 时还会激活并启动 IRQ，使 chip 的 enable/unmask 路径生效。

UART0 外部中断时序

运行时路径从体系结构入口开始。X2600/MIPS 的 plat_irq_dispatch() 读取 CP0 status/cause，按 CPU pending 位选择 Linux IRQ，并调用 do_IRQ()。当命中 core intc 的 CPU IRQ 时，CPU IRQ 的 action 是 xburst2_irq_handler()，它读取 core intc 的 IPR 寄存器，找到具体外设 hwirq，再调用 do_IRQ(irq_num + IRQ_INTC_BASE)。

图 · UART0 外部中断时序：先经过 core_intc 级联到 CPU IRQ，再由级联 handler 解复用到 UART0 的 Linux IRQ，最后 generic IRQ core 调用已注册的 irqaction。

X2600 SoC 相关 irq 驱动

CPU interrupt controller

irq-ingenic-cpu.c 封装 MIPS CP0 status 位：unmask_ingenic_irq() 设置 c0_status 对应 bit，mask_ingenic_irq() 清除对应 bit。它的 irq_chip 名为 XBurst2，domain map 中把每条 CPU IRQ 设置为 per-CPU devid，并绑定 handle_percpu_devid_irq。

core intc

irq-ingenic-chip.c 负责 SoC 外设中断控制器寄存器：IMSR 用于 mask，IMCR 用于 unmask，IPR 用于读取 pending。它的 xburst2_irq_chip 提供 irq_mask、irq_mask_ack、irq_unmask、irq_ack、irq_eoi、irq_set_wake，SMP 下还提供 irq_set_affinity。

static struct irq_chip xburst2_irq_chip = {
    .name           = "XBurst2-irqchip",
    .irq_mask       = xburst2_irq_mask,
    .irq_mask_ack   = xburst2_irq_mask,
    .irq_unmask     = xburst2_irq_unmask,
    .irq_ack        = xburst2_irq_mask,
    .irq_eoi        = xburst2_irq_unmask,
    .irq_set_wake   = xburst2_irq_set_wake,
};

这段实现体现了 generic IRQ 的分层初衷：core intc driver 不需要实现整个中断调度流程，只需要描述硬件能做的基本动作；handle_level_irq 决定这些动作何时调用。

核心对象的职责边界

对象第一性原理解释在当前仓库中的落点
irq_desc每条 Linux IRQ 必须有独立状态：是否 disabled、是否 pending、flow handler 是谁、action 链是谁。kernel/irq/chip.c 和 kernel/irq/handle.c 围绕 desc 操作。
irq_datachip 回调只需要当前 IRQ 的硬件号、domain、chip_data 和状态位，不应直接依赖完整 desc。include/linux/irq.h 定义 hwirq、chip、domain、chip_data。
irq_chip硬件控制器能力集合：mask、unmask、ack、eoi、set_type、set_wake、set_affinity。xburst2_irq_chip 和 ingenic_cpu_irq_controller。
irq_domain一个控制器一套本地编号空间；domain 是这套空间和 Linux IRQ 空间之间的翻译器。irq_domain_add_legacy()、irq_domain_xlate_*()、irq_create_of_mapping()。
irqaction设备驱动真正想注册的是“这个 IRQ 来了调用我的函数”。action 是这个函数、dev_id、线程函数、flags 的载体。request_threaded_irq() 最终安装到 desc->action。

flow handler 与 irq_chip 的分工

官方 generic IRQ 文档把旧式 __do_IRQ() super-handler 的问题归结为：flow logic 和 low-level hardware logic 混在一起，导致复杂路径和重复代码。新的模型把中断类型处理流程抽象为 flow handler，把硬件寄存器动作抽象为 irq_chip。

flow handler适用硬件语义典型动作顺序X2600 使用情况
handle_level_irq电平有效期间中断线保持 activemask/ack，调用 action，条件满足后 unmaskcore intc 外设 IRQ 使用
handle_edge_irq边沿事件被控制器锁存ack，处理 pending/running 状态，避免丢边沿本文所读 X2600 core intc 未使用
handle_fasteoi_irq控制器透明处理大部分流程，只需结束 EOI调用 action，最后 eoi本文所读 X2600 core intc 未使用
handle_percpu_devid_irq每 CPU 本地 IRQ，dev_id 是 per-CPU 变量按当前 CPU action 处理CPU intc 使用

因此，irq_chip 的 irq_ack 不应该“顺便调用驱动 handler”，flow handler 也不应该知道 X2600 的 IMCR_OFF 寄存器。边界清晰后，generic IRQ core 才能复用同一套 flow handler 支持不同 SoC。

架构总结

把整个链路压缩成一句话：DTS 描述“谁连到谁、用哪个本地号”；irqchip early init 根据 compatible 注册控制器；irq_domain 把本地 hwirq 映射成 Linux IRQ 并安装 chip/flow；设备驱动通过 OF/platform API 得到 Linux IRQ 并 request action；硬件触发后，体系结构入口进入 desc 的 flow handler，flow handler 调用 irq_chip 完成硬件控制，再调用 action 进入设备驱动。

阅读当前仓库时的最短路径：

module_drivers/dts/x2600.dtsi -> module_drivers/drivers/irqchip/irq-ingenic-cpu.c -> module_drivers/drivers/irqchip/irq-ingenic-chip.c -> drivers/of/irq.c -> kernel/irq/irqdomain.c -> kernel/irq/chip.c -> kernel/irq/handle.c -> kernel/irq/manage.c。
