GPIO + pinctrl subsystem architecture

Linux GPIO 与 pinctrl 子系统设计架构

这份文档沿用 IRQ 子系统那份分析方法：先从硬件问题出发，再解释每个内核概念被引入是为了解决什么，最后落到 X2600 的 DTS、Ingenic pinctrl 驱动和 gpiolib 调用链。

官方 GPIO binding 明确要求 GPIO 控制器节点包含 gpio-controller 和 #gpio-cells；pinctrl binding 明确把客户端设备的状态写成 pinctrl-0、pinctrl-names，这些状态引用 pin controller 子节点。

为什么要分 GPIO 与 pinctrl

第一性原理上，一个 SoC 引脚至少有三个互相独立的问题：

归属

这根线是谁的？

同一颗 pad 可以接 UART、I2C、PWM、LCD，也可以作为普通 GPIO。这个“把 pad 交给哪个硬件功能”的问题由 pinmux 解决。

电气

这根线怎么驱动？

上下拉、驱动强度、施密特触发、slew rate、开漏等不是逻辑值，而是 pad 电气属性。这个问题由 pinconf 解决。

值

这根线当前是 0 还是 1？

当 pad 被当作通用输入输出时，软件需要 request、direction、get、set、to_irq。这个问题由 GPIO 子系统解决。

所以 pinctrl 的中心问题是“物理引脚如何被复用和配置”，GPIO 的中心问题是“把一条可用作 GPIO 的线抽象成可申请、可读写、可转中断的描述符”。二者相邻，但边界不同。

概念解决的问题

概念解决的问题X2600 落点
pinctrl_dev注册一个 pin controller，让内核知道它有哪些 pin、group、function、配置能力。pinctrl_register() 注册 ingenic pinctrl，pin 名称形如 GPA-0、GPE-12。
pin group把一组必须一起复用的 pin 命名，避免客户端逐个写寄存器。uart0_pe 表示 GPE9..12 这一组 UART0 引脚。
function描述硬件外设功能，例如 UART、I2C、SFC。一个 function 可以有多个可选 group。uart0-pin 下有 uart0_pc 和 uart0_pe 两组。
pinmux_ops把 “function + group” 翻译为硬件 mux 寄存器写入。ingenic_pinmux_enable() 调 ingenic_gpio_set_func() 写 PxINT/PxMSK/PxPAT1/PxPAT0。
pinconf_ops把 bias、drive strength、debounce 等电气属性翻译为寄存器。ingenic_pinconf_set() 写 PxPU/PxPD/PXDSx/PxSMT/PxSR 等。
gpio_chip抽象一个 GPIO bank 的 get/set/direction/to_irq 等能力。GPA..GPE 每个 bank 一个 gpio_chip，每个最多 32 条线。
gpio_desc用稳定描述符取代全局整数 GPIO，保存 active-low、open-drain、requested 等状态。gpio-keys 的 gpios = <&gpb 30 ...> 最终得到一个 descriptor。
gpio range告诉 pinctrl：GPIO bank 的 offset 与 pinctrl pin number 如何对应。驱动为每个 bank 调 pinctrl_add_gpio_range()，base 与 pin_base 都使用该 bank 的全局 GPIO base。
gpio irqchip当 GPIO 线也能触发中断时，需要映射 GPIO offset 到 Linux IRQ。X2600 驱动自建 irq_domain_add_linear()，父中断 IRQ_GPIO0..4 作为 chained IRQ。

pinctrl 选择路径

pinctrl 的典型入口不是外设驱动手写寄存器，而是设备模型在 probe 前后按 state 选择。DTS 中客户端设备写 pinctrl-names 和 pinctrl-0；pinctrl core 解析 phandle，调用控制器驱动的 dt_node_to_map 生成 mux/config map，再选择 state。

&uart0 {
    status = "okay";
    pinctrl-names = "default";
    pinctrl-0 = <&uart0_pe>;
};

uart0_pe: uart0-pe {
    ingenic,pinmux = <&gpe 9 12>;
    ingenic,pinmux-funcsel = <PINCTL_FUNCTION1>;
};

X2600 的特殊点是：pinctrl 子节点使用私有属性 ingenic,pinmux 表达 bank + 起止 offset，ingenic,pinmux-funcsel 表达功能号。驱动中 ingenic_dt_node_to_map() 找到 group 和 function，生成 PIN_MAP_TYPE_MUX_GROUP；真正切 mux 时由 ingenic_pinmux_enable() 写寄存器。

GPIO consumer 路径

GPIO 的典型入口是消费者驱动调用 devm_gpiod_get() 或 gpiod_get()。在 OF 场景下，gpiolib 会按 <name>-gpios 或兼容旧属性名查找 phandle，调用 GPIO controller 的 of_xlate 得到 bank 内 offset，再返回 gpio_desc。

gpio_keys {
    compatible = "gpio-keys";
    KEY_VOL_1 {
        label = "key_vol-";
        linux,code = <0x93>;
        gpios = <&gpb 30 GPIO_ACTIVE_LOW INGENIC_GPIO_NOBIAS>;
    };
};

在本仓库的 gpiolib 中，gpiod_get() 转到 gpiod_get_index()；设备树路径进入 of_find_gpio() 和 of_get_named_gpiod_flags()；后者通过 of_parse_phandle_with_args_map() 找到 &gpb，再让 ingenic_of_gpio_xlate() 解析第 0 个 cell 作为 pin offset、第 1 个 cell 作为 active-low 等通用 flag、第 2 个 cell 作为 Ingenic 私有电气配置。

GPIO IRQ 路径

GPIO 线如果能作为中断源，它同时进入 IRQ 子系统。X2600 的 GPIO bank DTS 同时声明 gpio-controller 和 interrupt-controller，每个 bank 有一个父中断接到 core_intc。

gpb: gpb {
    gpio-controller;
    #gpio-cells = <3>;
    interrupts = <IRQ_GPIO1>;
    interrupt-parent = <&core_intc>;
    interrupt-controller;
    #interrupt-cells = <3>;
};

驱动初始化时，ingenic_gpio_irq_init() 使用 irq_of_parse_and_map() 取得父 IRQ，再创建 bank 内的 linear irq_domain，并把父 IRQ 设置为 chained handler。硬件中断到来后，ingenic_gpio_irq_handler() 读 PxFLG 与 PxMSK，逐 bit 找到 pending GPIO line，通过 irq_find_mapping() 得到 Linux virq，最后 generic_handle_irq() 进入通用 IRQ flow。

X2600 拓扑

UART0 pinctrl 案例

UART0 本身的中断仍然是外设 IRQ，走 interrupts = <IRQ_UART0> 到 core_intc。但 UART0 的 TX/RX/CTS/RTS 引脚能不能连到 UART0 外设，是 pinctrl 要解决的问题。板级 DTS 选择 uart0_pe，含义是让 GPE9..12 进入 function1。

DTS &uart0 引用 &uart0_pe

pinctrl map ingenic_dt_node_to_map() 生成 mux group map

set_mux ingenic_pinmux_enable() 把 PINCTL_FUNCTION1 转为 GPIO_FUNC_1

register ingenic_gpio_set_func() 写 X2600 GPIO bank 寄存器

GPIO key 案例

gpio-keys 不需要把引脚复用成 UART/SPI 这样的外设功能，它要的是一条可读的 GPIO 输入线，可能再转成 IRQ。以 KEY_VOL_1 为例，DTS 中 &gpb 30 GPIO_ACTIVE_LOW INGENIC_GPIO_NOBIAS 同时表达三层信息：GPB bank、30 号线、逻辑低有效和 Ingenic 私有电气配置。

当消费者调用 gpiod_get() 后，gpiolib 返回 descriptor；当它调用 gpiod_direction_input()，会进入 ingenic_gpio_direction_input()，该函数再调用 pinctrl_gpio_direction_input()，最终由 ingenic_pinmux_gpio_set_dir() 把该 pin 设为 GPIO_INPUT。这就是 GPIO 和 pinctrl 的边界衔接。

设计边界

什么时候写 pinctrl

选择外设功能、设置 pad 电气属性、定义 default/sleep/idle 状态时写 pinctrl。比如 UART0 选 GPE9..12，SFC 选 GPD6..11。

什么时候写 GPIO

驱动需要控制一条板级信号，例如 reset、enable、detect、按键输入时写 GPIO consumer 属性，并在驱动里使用 descriptor API。

什么时候转 IRQ

GPIO 线既是输入又要触发事件时，消费者在拿到 descriptor 后调用 gpiod_to_irq()，再 request_irq()。

不要混淆的点

硬件手册里的“GPIO mode”常常是 pinmux 的一个 function，不等于 Linux GPIO consumer API。Linux 里“申请 GPIO”还包含所有权、方向、逻辑极性和生命周期管理。
