External Memory Controller Subsystem architecture

Linux EMC 子系统设计架构

本文从“为什么 CPU 需要 EMC”出发，解析 Linux 设备树对于 Memory Controller 的描述方式，并对接 X2600 手册中 SACR 与 SMCR 寄存器的物理映射模型，帮助开发者理解从 DTS 时序配置到物理信号跳变的完整路径。

官方权威依据

本文依据 Linux 内核官方 Devicetree Bindings 与 X2600 官方编程手册编写，不依赖任何自身幻觉。页面本身自包含且无外部网络请求依赖。

  • Linux DT Bindings: Ingenic NEMC (NAND / External Memory Controller)
  • X2600 Programming Manual：Chapter 6 External Memory Controller (EMC)
Linux 官方 ingenic,nemc.yaml 明确指出了 NEMC 是一个 memory-controller，负责将片上系统总线转换为外部静态设备控制信号，并通过 DTS 子节点的 ingenic,nemc-tAS、ingenic,nemc-tAW 等属性，直接指定纳秒级的时序需求。

第一性原理：为何需要 EMC

CPU 核通过虚拟/物理地址和简单的加载/存储（Load/Store）指令访问数据。但片外的 NOR Flash、SRAM 或者如 DM9000 这类接口类似 SRAM 的网络控制器，它们并不能理解 AHB/AXI 等系统级总线协议，而是只认识 片选（CS#）、输出使能（OE#）、写使能（WE#）与读写脉宽 这样的基础信号时序。

EMC 的核心作用就是：“截获特定块物理地址的访问，将其转换为带精密可控时序波形的外部物理总线交互”。

路由

这是谁的数据？

系统总线上拥有数 GB 容量。EMC 使用 SACR (地址配置寄存器) 探测访问地址，一旦匹配立刻拉低特定外设的 CS# 引脚。

波形

时序与握手该怎样？

外部设备有快有慢。EMC 提供 SMC0R / SMC1R 寄存器，插入等待周期（如 TRAS、TCH），保证地址建立和数据保持时长达标。

复用

类型多怎么办？

通过配置 SMT，EMC 总线可以兼容普通隔离总线 SRAM、非异步突发 ROM、甚至是地址数据复用总线（ADM）。

EMC 抽象概念映射

概念解决的问题Linux / X2600 对应点
memory-controller向内核宣示存在一个管理外部设备的母线（Bus）节点。NEMC 在系统中注册，其 compatible 包含 ingenic,jz47xx-nemc。
Bank / CS#一个控制器下挂载多个不同的芯片，彼此不干扰。X2600 支持 CS1# 和 CS2#，各自拥有独立的 SACRx / SMCxR1 / SMCxR2 寄存器设定。
Address Match截获指定地址范围分配给特定的 Bank。SACR 寄存器的 MASK 与 BASE 比对。Linux 会通过 ranges 属性在 DTS 中自动对应这些物理区间。
tAS / tAW / tAH不同芯片需要不同的纳秒级建立、脉宽和保持时间。DTS 子节点利用 ingenic,nemc-tAS 填入纳秒，Linux NEMC driver 动态换算为 CPU 时钟节拍填入寄存器。

DTS 节点绑定与解析

在 Linux 侧，并不鼓励驱动去硬写 0x07FF7700 这样的死值，而是推荐在设备树中体现设备的真实纳秒时序。Linux 驱动会自动计算当前 clk 以得到所需的寄存器周期数。

nemc: memory-controller@13410000 {
    compatible = "ingenic,jz4780-nemc"; /* Ingenic EMC 族系 */
    reg = <0x13410000 0x10000>;
    #address-cells = <2>;
    #size-cells = <1>;
    clocks = <&cgu NEMC_CLK>;

    /* ranges：把 Bank 1,2 等的地址映射到系统全局物理地址 0x1B000000 等 */
    ranges = <1 0 0x1b000000 0x1000000>,
             <2 0 0x1a000000 0x1000000>;

    nor_flash@1 {
        compatible = "cfi-flash";
        /* 子节点的 reg 第一位表示使用哪个 CS 号 (Bank 1) */
        reg = <1 0 0x1000000>;
        /* Ingenic NEMC 配置属性 (单位: ns) */
        ingenic,nemc-tAS = <15>;
        ingenic,nemc-tAH = <10>;
        ingenic,nemc-tAW = <50>;
    };
};

Linux 侧的 Driver 模型

内核中使用 MTD 或者对应的 Memory Controller 驱动接管 EMC 的行为。当子设备驱动（例如 Ethernet Controller / NOR Flash）probe 时：

解析时钟 驱动先调用 clk_get_rate() 取得当前 NEMC 控制器的接入频率（皮秒精度）。

提取纳秒 驱动根据该子设备节点的 ingenic,nemc-tAW 属性得到纳秒要求。

计算周期 使用 CYCLES = DIV_ROUND_UP(tAW_ns, clock_period_ns)，转化出需填入的 Cycle 实数。

写入寄存器 驱动构建好要写的 SMC0R 和 SMC1R，并将其写入 Bank 对应的位置。

通过这种抽象，只要修改设备树里的物理纳秒时间，即便未来系统主频发生变动，无需修改任何代码即可保持总线波形符合 Flash 的电气要求。

地址路由与拓扑

核心时序与寄存器

SACR 寄存器

地址路由

规定外部芯片挂在系统的哪个地址段。SACRx 定义 BASE 与 MASK。逻辑为：如果系统总线传来的地址最高段在 MASK 与操作后等于 BASE，意味着本次访问命中该 Bank。

SMC0R 寄存器

访问周期

包含核心的 TRAS (Read Address Setup) 决定拉低 RD# 前地址停留几个时钟，TAW_OE 决定 RD# 自身持续多长来等待 Flash 返回数据。

SMC1R 寄存器

防冲突间隔

在不同芯片切换（或同芯片读跟写）时，如果不强行插入间隔（STRV），总线可能会发生物理短路冲撞。

复用模式

SMT 字段

SMC0R 的 SMT 字段支持将外部接口设定为 ADM(地址与数据复用)，这样可以节约管脚连接 NAND 或特定的伪静态接口外设。

排查清单与工作边界

现象 / 诉求排查清单或配置位置
外部 NOR Flash 工作不稳定审查 DTS 节点中的纳秒时序（tAS、tAW、tAH 等），确保严格覆盖 Flash 规格书的最差时间。内核日志可辅助观察转换后的 Cycle 数。
系统访问外部存储产生死活锁检查在切换片选或混合读写时（STRV 是否不足），总线上两颗芯片的高低电平是否发生短接；增大 ingenic,nemc-tSTRV。
总线宽度匹配需要在驱动内部（或设备树扩展属性）为目标外设映射 SMC0R 中 BW 控制位，16位操作却误按 8位 总线取值将导致数据混乱。
Wait 引脚无法延迟访问检查 SMC1R 的 WAIT_ENABLE 位（WP_EN），确认相应引脚未配置成纯 GPIO，或者 pinctrl 中已将该复用打开。
