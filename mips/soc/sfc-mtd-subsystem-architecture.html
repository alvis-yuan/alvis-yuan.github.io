MTD / SPI-NOR Subsystem architecture

Linux MTD/SFC 子系统设计架构

本文从“为什么我们需要一套统一且高效的闪存管理体系”出发，解析 Linux MTD (Memory Technology Device) 子系统对于 Flash 闪存设备的抽象方式，并对接 X2600 中 SFC (SPI Flash Controller) 硬件控制器的实现原理。揭示内部软件栈（MTD -> SPI NOR -> SFC Driver）与底层硬件机制（CDT 表、DMA）如何共同协作完成高效的数据吞吐。

第一性原理：为何需要 MTD / SFC？

MTD 的诉求 统一各类闪存

痛点：闪存（NAND, NOR）不同于常规块设备（例如硬盘），不能直接随机进行覆盖写。需要先"擦除(Erase)"一个 Block，再"编程(Program)"相应的 Page，最后进行读取(Read)。并且还会有"坏块"。因此不能直接使用传统的 Block 层进行挂载映射。

解法：Linux 引入了 MTD 层，对各种 Flash 硬件的共性操作进行了抽象和封装，提供统一的 API (mtd_read, mtd_write, mtd_erase)。并在之上架构了支持闪存特性的文件系统（如 JFFS2, UBIFS, UBIFS, SPIFFS），以屏蔽底层物理差异。

SFC 的诉求 专用的 SPI 加速

痛点：传统的通用 SPI（Serial Peripheral Interface）通过简单的 TX/RX FIFO 进行字节流传输。当需要极高速和长期的数据吞吐（如引导加载，XIP等）时，通用 SPI 协议的 CPU 干预过于频繁和复杂，成为性能瓶颈。

解法：引入专门定制的 SFC (SPI Flash Controller)，具备硬件直接执行指令表的功能（如 CDT, Command Descriptor Table）。支持直接发送命令（Command）、地址（Address）、和数据格式，并通过 DMA Controller 进行数据搬运，实现极高的吞吐而近乎零的 CPU 开销。

DTS 节点绑定与解析

在 Linux MTD 架构中，SPI Flash 通常挂载在具体的控制器下作为其子节点。以下是典型配置模式：

sfc: sfc@13440000 {
    compatible = "ingenic,sfc-v2";
    reg = <0x13440000 0x1000>;
    interrupts = <21>;
    clocks = <&cgu SFC>;
    dmas = <&dma 20 0xffffffff>, <&dma 20 0xffffffff>;
    dma-names = "rx", "tx";
    
    /* 挂载的 Flash 设备 */
    flash: spi-flash@0 {
        #address-cells = <1>;
        #size-cells = <1>;
        compatible = "jedec,spi-nor"; // 或 "spi-nand"
        spi-max-frequency = <100000000>; // 100MHz
        
        partition@0 {
            label = "bootloader";
            reg = <0x0 0x40000>; // Offset: 0, Size: 256K
        };
        partition@40000 {
            label = "kernel";
            reg = <0x40000 0x400000>;
        };
    };
};

在上述结构中，第一层 sfc 节点给定了该 SFC 模块的控制寄存器基址（0x13440000）、硬件中断以及 DMA 信道。子节点 spi-flash@0 定义了实际的闪存，操作系统将会使用 SPI NOR Framework 检测名为 jedec,spi-nor 的设备，解析 partitions 以便在 /dev 下生成块映射（如 /dev/mtd0, /dev/mtd1）。

SFC 硬件拓扑与 CDT

在 X2600 SoC 中，相较于让 CPU 控制每一个 GPIO/SPI 的拉高拉低并搬运字节（即所谓的 Bit Banging），SFC 实现了极大的硬件减负。

其核心机制为：通过一片寄存器区域（Command Descriptor Table, CDT），让 CPU 或 DMA 来配置"一次完整的 Flash IO 会话需要哪些步骤"。硬件一旦激发，自动完成整个传输过程并触发中断。

SFC_CMD_IDX

指定该事务的指令编号。比如 0x03 为 NOR 的标准读，0xEB 可能是 4x IO 的快速读指令。SFC 硬件会自动在前 8 个串行时钟周期将该索引发送出去。

SFC_COL/ROW_ADDR

指定操作的目标页和块地址。SFC 硬件自动结合配置里的地址宽度（例如 3 或者 4 字节），组装地址字段紧跟在指令周期的后面输给外部存储。

DMA Desc Payload

利用系统内的外部 DMA 控制器，将 SRAM 中准备好的写数据、或者接收进的读数据直接搬运到主存中，整个接收过程不占据 CPU 周期。这是高吞吐量运转的关键。

驱动层至物理层的生命周期

当文件系统调用读写请求时，一个典型的数据流动过程如下：

文件系统/MTD 层调用 sfc_do_read() ➜ 驱动层构建出 struct sfc_cdt_xfer 结构实例 ➜ 设置对应 cmd_index, rowaddr，并判断是否需要使用 DMA 数据模式 ➜ 调用同步下发函数 sfc_sync_cdt() ➜ SFC 控制器接管物理引脚 CS/CLK/D[3:0] 发送时序 ➜ SFC 利用 DMA 将收取的数据放入主存，发送硬件中断 ➜ CPU 从主存取得数据对象并层层向上返回。

总结

由于 MTD 的良好抽象，绝大多数驱动侧的工作在于将复杂的 MTD / SPI NOR Framework 内的一条命令转化为 平台特定的操作寄存器流程。熟悉 X2600 SFC 的 CDT 使用方式，能够让我们在分析高并发大容量读写的挂死或超时问题时，立刻锁定是“上层驱动封装错误”，还是“SFC和DMA时序断层”的问题。
