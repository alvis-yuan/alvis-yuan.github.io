IPv4 地址子系统与路由子系统的交互：配置 IP 到自动直连路由

本文关注 IPv4。用户配置地址时，内核不是只把地址挂到网卡上；地址提交成功后会通过 notifier 驱动 FIB 自动生成本地路由、直连 prefix 路由、广播路由，并在地址删除、设备 up/down、secondary 地址提升、metric 变化时保持这些派生路由一致。这里少贴代码，重点解释控制点、流程和设计背景。

官方依据

主题官方链接本文使用的事实
地址管理ip-address(8)`ip address add` 给设备添加协议地址；`noprefixroute` 表示不自动创建/删除该地址关联的 prefix route；`metric` 是地址关联 prefix route 的优先级。
路由管理ip-route(8)Linux 有 main/local 等路由表；local 表由内核维护，包含本地和广播地址；直连路由通常是 scope link、protocol kernel。
IPv4 sysctlip-sysctl`promote_secondaries`、`arp_filter`、`rp_filter`、`ignore_routes_with_linkdown` 等配置会影响地址删除、源校验、邻居和路由行为。
第一性原理概述：地址是“本机在哪些链路上拥有哪些 IP”的事实，路由是“给定目的地址应如何转发或本地接收”的索引。地址变化必须同步更新路由索引，否则收包、发包、ARP 与源地址选择都会看到过期事实。

整体设计模型：地址事实源，FIB 派生索引

图 1：地址配置通过 notifier 让 FIB 维护派生路由，数据面只查 FIB。

本地源码中的核心分工很清楚：`net/ipv4/devinet.c` 管理 `struct in_ifaddr` 链表、生命周期、netlink/ioctl 地址入口；`net/ipv4/fib_frontend.c` 监听地址和设备事件，并把它们转换为 FIB route；`net/ipv4/fib_trie.c` 保存路由前缀和同前缀多个 alias；`net/ipv4/route.c` 在收包/发包时执行 FIB lookup 并构造 `rtable`。

第一性原理概述：地址链表适合表达接口属性，FIB trie 适合最长前缀匹配。Linux 不让数据面直接扫地址链表找路由，而是把地址事件投影到 FIB。

配置入口：netlink 与 ioctl 最终都提交 in_ifaddr

现代 `ip addr add 192.168.1.10/24 dev eth0` 走 rtnetlink：`RTM_NEWADDR -> inet_rtm_newaddr() -> rtm_to_ifaddr() -> __inet_insert_ifa()`。老接口或启动期自动配置可以走 ioctl：`SIOCSIFADDR -> devinet_ioctl() -> inet_set_ifa() -> inet_insert_ifa()`。

入口关键函数产生的地址字段
`RTM_NEWADDR``inet_rtm_newaddr()`、`rtm_to_ifaddr()`解析 `IFA_LOCAL`、`IFA_ADDRESS`、`IFA_BROADCAST`、`IFA_LABEL`、`IFA_FLAGS`、`IFA_RT_PRIORITY`、lifetime。
`SIOCSIFADDR``devinet_ioctl()`、`inet_set_ifa()`使用 classful 默认前缀推导 mask/broadcast；点到点设备设 /32；生命周期为 forever。
启动期 IP 配置`net/ipv4/ipconfig.c` 调 `devinet_ioctl()`早期网络启动同样复用 IPv4 地址子系统，不绕开 FIB 派生逻辑。

`rtm_to_ifaddr()` 是 netlink 属性到内核地址对象的边界：`IFA_ADDRESS` 缺省等于 `IFA_LOCAL`，`IFA_RT_PRIORITY` 保存到 `ifa_rt_priority`，`IFA_FLAGS` 可携带 `IFA_F_NOPREFIXROUTE`、`IFA_F_SECONDARY` 等语义。

第一性原理概述：入口可以不同，但必须归一到同一种地址对象。只有这样，后续通知、路由派生、生命周期、地址删除和设备事件才能共享一套一致规则。

新增地址：先提交地址，再通知 FIB

`__inet_insert_ifa()` 的顺序非常关键：先判断 primary/secondary、调用 validator、插入 `in_dev->ifa_list` 与地址 hash，随后发 `RTM_NEWADDR`，最后调用 `blocking_notifier_call_chain(&inetaddr_chain, NETDEV_UP, ifa)`。FIB 不是地址函数直接硬编码维护，而是作为 inetaddr notifier 的订阅者被调用。

流程摘要：
inet_rtm_newaddr()
  -> rtm_to_ifaddr()              // netlink 属性转 in_ifaddr
  -> find_matching_ifa()          // 判断新增还是替换
  -> __inet_insert_ifa()
       -> 插入 in_device 地址链表与 hash
       -> rtmsg_ifa(RTM_NEWADDR)  // 先通知地址变化
       -> inetaddr_chain NETDEV_UP
            -> fib_inetaddr_event()
                 -> fib_add_ifaddr()

这解释了为什么抓 netlink 事件时通常能看到地址事件和路由事件：地址子系统先宣布“地址存在”，FIB 随后宣布“由这个地址派生出的路由存在”。这样用户态路由守护进程看到路由事件时，地址对象已经在内核里可查。

第一性原理概述：地址是原因，路由是结果。Linux 用 notifier 把二者解耦：地址子系统不需要知道 FIB trie 细节，FIB 也不负责解析 `ip addr` 的用户参数。

`fib_magic()`：把地址事件翻译成内核路由

`fib_magic()` 是地址子系统和路由子系统之间最集中的翻译点。它构造 `fib_config`，使用 `RTPROT_KERNEL` 标记这些路由来自内核自动配置，设置 `prefsrc = ifa->ifa_local`、`oif = ifindex`，再根据 route type 选择表和 scope。

路由类型默认表scope含义
`RTN_LOCAL``RT_TABLE_LOCAL``RT_SCOPE_HOST`目的 IP 是本机地址，收包应交给本地协议栈，发往本机也可 loopback。
`RTN_BROADCAST``RT_TABLE_LOCAL``RT_SCOPE_LINK`目的 IP 是链路广播或网络广播，按广播输入/输出规则处理。
`RTN_UNICAST``RT_TABLE_MAIN``RT_SCOPE_LINK`直连 prefix，无网关，下一跳在本链路，通过 ARP/邻居解析。

如果设备属于 L3 master/VRF，`l3mdev_fib_table()` 会让这些自动路由进入对应 VRF 表，而不是普通 main/local 语义的全局表。这是地址配置与策略路由/VRF 的交汇点。

第一性原理概述：自动直连路由不是用户态“顺手加了一条 route”，而是内核从地址、prefix、设备和 scope 推导出的最小可达性事实。

自动生成哪些路由：local、prefix、broadcast

`fib_add_ifaddr()` 的策略可以按三层理解。

派生项触发条件例子：`192.168.1.10/24 dev eth0`
本地主机路由每个 IPv4 地址都会添加`local 192.168.1.10/32 dev eth0 scope host table local`。
直连 prefix 路由设备 up、非 secondary、非 zero prefix、非纯 /32 host、未设置 `IFA_F_NOPREFIXROUTE``192.168.1.0/24 dev eth0 proto kernel scope link src 192.168.1.10 metric ifa_rt_priority`。
显式广播地址`ifa_broadcast` 存在且不是 255.255.255.255`broadcast 192.168.1.255/32 table local`。
网络号/全 1 广播prefixlen < 31 且 prefix 有意义`broadcast 192.168.1.0/32` 与 `broadcast 192.168.1.255/32`。
图 2：一个地址对象会派生多类路由，分别服务本地接收、链路可达和广播语义。

`noprefixroute` 只抑制 prefix route，不抑制 local route。因为“这个地址属于本机”与“这个 prefix 通过该接口直连”是两条不同事实；前者影响本地投递，后者影响对同网段其它主机的转发/发送。

第一性原理概述：本机地址必须能被 local 表识别；直连网络是否自动可达则是 prefix route 策略。`noprefixroute` 关掉的是后者，不是本机地址身份。

删除地址：不要删多，也不能留脏

删除比新增复杂。`__inet_del_ifa()` 先处理 primary/secondary 关系：删除 primary 时，如果没有 `promote_secondaries`，同一子网 secondary 会一起删除；如果允许提升，则选择一个 secondary 变为 primary。为了让路由 prefsrc 正确，提升前会先对相关 secondary 的路由做静默删除，提升后再添加回来。

`fib_del_ifaddr()` 删除路由时会扫描同设备地址链表，判断 local/broadcast/prefix 是否仍被其它地址使用。只有没有其它地址还需要这些 FIB 项时才真正删除。这就是源码注释说的“not to delete too much”。

场景路由处理原因
删除普通 primary删 prefix route、local route、相关 broadcast route该地址和直连网络事实消失。
仍有相同 local IP 或广播被其它地址使用保留对应 local/broadcast route避免删除其它地址仍依赖的本地或广播可达性。
secondary 提升先删旧 prefsrc 关联路由，再用新 primary 加回直连路由的 `prefsrc` 应指向新的 primary。
最后一个地址删除`fib_disable_ip()`、flush route/cache、ARP ifdown接口 IPv4 语义整体关闭，关联路由和邻居状态要失效。
第一性原理概述：路由删除不能只按“这个地址没了”机械删除，因为多个地址可能共享广播、prefix 或 prefsrc 关系。删除逻辑必须证明某个 FIB 事实真的不再被任何地址支持。

设备事件：地址存在不等于链路可用

`fib_netdev_event()` 处理网卡层事件。设备 `NETDEV_UP` 时，会遍历已有 `in_ifaddr` 调 `fib_add_ifaddr()`，这解释了“先配地址、后 up 设备”时路由也会出现。设备 `NETDEV_DOWN` 或 unregister 时，调用 `fib_disable_ip()` 使相关 nexthop/路由失效，刷新 route cache，并执行 `arp_ifdown()`。

设备 carrier 变化不会删除地址，但可能标记/解除 `RTNH_F_LINKDOWN`，并刷新路由缓存。MTU 变化会同步到 FIB nexthop metrics，避免数据面继续使用旧 MTU。

第一性原理概述：地址说明“逻辑身份”，设备状态说明“链路是否可用”。FIB 必须同时受二者约束：有地址但设备 down，不能继续把该链路当成正常可达路径。

手工路由：走同一个 FIB，但来源不同

`ip route add` 走 `RTM_NEWROUTE -> inet_rtm_newroute() -> rtm_to_fib_config() -> fib_table_insert()`。这条路径和地址自动路由最终写入同一个 FIB trie，但 `fib_config` 的来源、protocol、scope、nexthop 语义不同。

来源入口protocol典型表生命周期
地址自动派生`fib_magic()``RTPROT_KERNEL`local/main/VRF 表随地址和设备事件自动增删。
管理员手工添加`inet_rtm_newroute()`用户指定或 iproute2 默认默认 main，也可指定 table/VRF不随地址对象自动删除，除非 nexthop/device 失效触发同步。
网关路由`fib_create_info()`/nexthop check通常 static/bootmain 或策略表下一跳必须可解析，除非 `onlink` 强制认为在链路上。

`fib_check_nh_v4_gw()` 的设计背景很重要：网关路由的 gateway 本身也要能通过 FIB 找到可达链路。也就是说，默认路由 `default via 192.168.1.1 dev eth0` 依赖此前由地址生成的 `192.168.1.0/24 dev eth0 scope link`，除非用户指定 `onlink` 绕过该递归可达性证明。

第一性原理概述：直连路由是网关路由的地基。没有“网关在本链路可达”的证明，内核不能安全地把任意地址当下一跳，否则 ARP/邻居解析没有边界。

数据面查找：收包和发包都消费 FIB 结果

收包路径 `ip_route_input_slow()` 构造 `flowi4`，调用 `fib_lookup()`。如果结果是 `RTN_LOCAL`，走本地输入；如果是 `RTN_BROADCAST`，走广播输入；如果是 `RTN_UNICAST` 且开启转发，才构造转发路由。源地址还会经过 `fib_validate_source()`，这就是 `rp_filter` 等配置影响收包的地方。

发包路径 `ip_route_output_key_hash_rcu()` 同样调用 `fib_lookup()`。如果目的命中 `RTN_LOCAL`，输出设备会变成 loopback 或 L3 master loopback；如果命中普通 unicast，`fib_select_path()` 选择 nexthop，然后构造 `rtable`。如果应用指定 `oif` 且查不到路由，内核在一些场景下仍可按 on-link 方式构造输出路由，但这属于显式绑定接口的特殊兼容逻辑。

第一性原理概述：控制面维护 FIB 的正确性，数据面只做高频 lookup。把复杂地址语义提前折叠进 FIB，是性能和一致性的共同要求。

典型场景汇总

场景流程结果/设计背景
`ip addr add 192.168.1.10/24 dev eth0``RTM_NEWADDR -> __inet_insert_ifa -> fib_add_ifaddr`生成 local 192.168.1.10/32、main 192.168.1.0/24、广播路由；prefix route 的 `src` 为 192.168.1.10。
加 `noprefixroute`地址照常插入，`fib_add_ifaddr` 跳过 prefix route本机仍拥有该 IP，但不自动认为该 prefix 直连；适合手工/策略路由控制。
同子网第二个地址`__inet_insert_ifa` 标记 `IFA_F_SECONDARY`仍添加 local route，但不重复添加 prefix route；prefix 的 prefsrc 归 primary 控制。
修改地址 metric匹配已有 ifa，调用 `fib_modify_prefix_metric()`先添加新 metric prefix route，再删除旧 metric route，避免短窗口无路由。
删除 primary`__inet_del_ifa -> fib_del_ifaddr`，必要时提升 secondary保证 prefsrc 和派生路由与新的 primary/secondary 关系一致。
设备 down`fib_netdev_event NETDEV_DOWN -> fib_disable_ip`地址对象可能还在，但 FIB nexthop/route/cache/ARP 状态被同步失效。
添加默认路由 via 网关`RTM_NEWROUTE -> fib_check_nh_v4_gw -> fib_lookup(gw)`网关必须通过已有直连路由可达；`onlink` 是显式强制例外。
VRF/L3 master`l3mdev_fib_table()` 改变自动路由表同样的地址配置会进入 VRF 对应表，避免不同 L3 域互相污染。
第一性原理概述：大多数“为什么加 IP 会多出路由”的现象，都来自一个规则：接口地址会派生本地身份、链路 prefix 可达性和广播语义。

设计原则与背景

事实与索引分离

`in_ifaddr` 是事实源，FIB 是 lookup 索引。这样地址生命周期和路由查找性能可以分别优化。

自动与手工共用 FIB

内核自动路由和管理员路由最终都写入 `fib_table_insert()`，区别在 protocol、scope、table、nexthop 和生命周期。

地址事件驱动一致性

新增、删除、secondary 提升、metric 变化、设备 up/down 都通过 notifier 或 netdev event 维护派生路由。

scope 约束可达性

local、link、global scope 防止把本机地址、直连地址、网关地址混为一谈，是网关递归校验的基础。

常见误解：直连路由不是“iproute2 帮你偷偷加的”。它是内核根据 IPv4 地址对象自动生成的 `RTPROT_KERNEL` FIB 项，除非使用 `noprefixroute` 或设备/地址状态让条件不成立。

整体总结

IPv4 地址子系统负责“设备拥有哪些地址、地址主从关系、生命周期、label、metric、flags”；路由子系统负责“哪些目的地址如何匹配、走哪个设备或本地输入、是否广播、网关是否可达”。两者通过 `inetaddr_chain` 和 `netdev` notifier 耦合，而不是互相嵌入实现细节。

配置 IP 后自动生成直连路由的本质，是把 `local/prefix/broadcast` 三类网络事实写入 FIB。删除地址和设备状态变化则是反向操作：只删除真正失去事实支撑的 FIB 项，并刷新缓存、邻居或 nexthop 状态，让数据面不继续使用过期路径。
