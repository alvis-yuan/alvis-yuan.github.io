从 kernel_entry 到 start_kernel 前，入口汇编到底做了什么

这份文档以当前反汇编为主线，只解释 `806caafc <kernel_entry>` 到 `j start_kernel` 之间已经发生的处理。每段都附上当前仓库本地源码节选，不把本地源码作为外部引用处理；外部背景只使用已获取到的 Linux 官方文档链接。

当前配置事实

/* include/generated/autoconf.h */
#define CONFIG_BUILTIN_DTB 1
#define CONFIG_USE_OF 1

因此，当前反汇编里没有出现 appended DTB 检查分支的实际指令，而是直接进入“检查 a0 是否为 -2，否则尝试 builtin DTB”的路径。

第一性原理概述：入口汇编阶段还没有 C 运行环境、没有普通内存分配器、也不能假设固件以唯一格式传参。它必须先把 CPU 放到内核可控状态，并把后续 C 代码需要的最小事实保存下来。

当前机器包含的是 generic `kernel-entry-init.h`，所以 `kernel_entry_setup` 是空宏。反汇编中 `kernel_entry` 标签后的第一条真实指令就是 `mfc0 t0,c0_status`，也证明此处没有额外 CPU 私有入口指令。

设计上保留这个空钩子，是为了让某些 MIPS 机器在进入通用 `head.S` 流程前做必要的 CPU 或平台修正；当前 X2600 路径不需要这个修正，所以空宏就是严格的当前行为。

第一性原理概述：公共入口代码必须允许少数平台在“能安全执行通用代码”之前补一小段机器私有动作；当前平台不需要时，最小动作就是不做任何事。

这段宏先读 CP0 Status，再把 `ST0_KERNEL_CUMASK` 和低 5 位临时置 1，随后用 `xor 0x1f` 把低 5 位翻回 0。当前 32 位配置下 `setup_c0_status_pri` 没有传入 `ST0_KX`，因此反汇编里出现的是 `0x1000001f`：高位 `0x10000000` 对应 `ST0_CU0`，低 5 位是 `IE/EXL/ERL/KSU` 相关早期状态清理。

结果是：内核可使用 CP0，CPU 处在内核态相关状态，早期中断被关闭。`ehb` 是 hazard barrier，用来让刚写入 CP0 Status 的效果在后续指令前完成。这一阶段还没有异常向量、栈、IRQ 子系统和调度器，允许异步中断进入只会制造不可恢复的不确定性。

第一性原理概述：在没有栈和异常处理上下文前，CPU 只能做线性、可预测的初始化。先关中断并取得 CP0 控制权，是为了避免外部事件在系统还不会处理它们时打断启动。

跳到链接地址：修正入口 PC 的不确定性

/* arch/mips/kernel/head.S */
    /* We might not get launched at the address the kernel is linked to,
       so we jump there.  */
    PTR_LA  t0, 0f
    jr      t0
0:
806cab18: 3c08806d  lui     t0,0x806d
806cab1c: 2508ab28  addiu   t0,t0,-21720
806cab20: 01000408  jr.hb   t0
806cab24: 00000000  nop

806cab28: 2409fffe  li      t1,-2

源码注释已经给出原因：内核不一定从链接地址被启动。`PTR_LA t0, 0f` 取的是标签 `0` 的链接地址，随后 `jr t0` 强制把 PC 带回链接视角下的地址。当前反汇编显示目标是 `806cab28`，也就是后续 DTB 选择逻辑开始处。

官方 MIPS booting 文档也说明，不同 bootloader 可能只支持镜像开头入口，也可能跳 ELF start address；这类差异会让“最初进入内核的位置”存在平台差别。这里用一次显式跳转，把后续所有绝对符号访问统一到链接地址语义。

第一性原理概述：早期汇编会访问 `__bss_start`、`fw_arg0`、`init_thread_union` 等链接符号。如果 PC 还停在不确定的加载地址语义里，符号地址就可能不可信；先跳回链接地址，是为了让后续每一次取符号地址都有同一个坐标系。

/* arch/mips/xburst2/core/prom.c */
void __init prom_init(void)
{
    prom_init_cmdline((int)fw_arg0, (char **)fw_arg1);

    if (fw_arg0 == 0 && fw_arg1 == 0xffffffffUL)
        _fw_fdt_addr = phys_to_virt(fw_arg2);
    else if ((int)fw_arg0 == -2)    /*UHI*/
        _fw_fdt_addr = (void *)fw_arg1;
    else if ((void *)__dtb_start != (void *)__dtb_end)
        _fw_fdt_addr = __dtb_start;
    else
        panic("no dtb found!\n");
}

当前配置没有 `CONFIG_MIPS_RAW_APPENDED_DTB` 或 `CONFIG_MIPS_ELF_APPENDED_DTB` 的实际反汇编指令，所以入口汇编没有先检查 appended DTB magic。它先把 `t1` 设为 `-2`，把 `a1` 复制到 `t2`，如果 `a0 == -2`，就认为这是 UHI 形式，`t2 = a1` 作为 DTB 指针进入 `dtb_found`。

如果不是 UHI，且 `CONFIG_BUILTIN_DTB` 打开，代码把 `t2 = __dtb_start`、`t1 = __dtb_end`。只要二者不同，说明内核镜像中确实有 builtin DTB，于是接受 `t2`。如果二者相等，最后把 `t2` 清成 0。

官方 DT 使用模型说明，DT 是内核获取硬件拓扑、运行时配置和设备创建信息的数据结构；MIPS booting 文档说明 legacy bootloader 可能不传 DTB，此时 Linux 会寻找 builtin DTB。当前 X2600 后续 `prom_init()` 也再次按 firmware DTB、UHI、builtin DTB 的顺序确定 `_fw_fdt_addr`。

第一性原理概述：内核必须知道硬件是什么，但不同固件传 DTB 的方式不同。入口汇编先把“可能的 DTB 地址”压缩成一个寄存器值，后续 C 代码再用它扫描硬件描述，这样可以同时兼容新旧启动方式。

`.bss` 保存未显式初始化的全局变量和静态变量。C 语言语义要求这些对象在程序开始时为 0，但裸机入口并没有运行用户态程序那种加载器来替内核清零，所以 `head.S` 自己用 `sw zero` 从 `__bss_start` 清到 `__bss_stop - LONGSIZE`。

当前反汇编显示 `LONGSIZE` 是 4 字节，说明这是 32 位路径。后面马上要写入的 `fw_arg0..3`、`fw_passed_dtb`、`kernelsp` 都定义在 C 文件中；如果不先清 BSS，后续 C 代码会看到随机旧内存。

第一性原理概述：C 世界默认“未初始化全局变量等于 0”，但 CPU 从固件跳进内核时没有人为这个承诺兜底。清 BSS 是入口汇编把硬件世界转换成 C 语言世界的第一项契约。

/* arch/mips/xburst2/core/prom.c */
static __init void prom_init_cmdline(int argc, char *argv[])
{
    unsigned int count = COMMAND_LINE_SIZE - 1;
    int i;
    char *dst = &(arcs_cmdline[0]);
    char *src;

    for (i = 1; i < argc && count; ++i) {
        src = argv[i];
        while (*src && count) {
            *dst++ = *src++;
            --count;
        }
        *dst++ = ' ';
    }
    if (i > 1)
        --dst;

    *dst = 0;
}

MIPS 调用约定中的 `a0-a3` 是参数寄存器。入口汇编之后会建立栈并跳 C 函数，寄存器很快会被普通调用约定覆盖；因此必须把固件传来的原始参数保存到全局变量。X2600 的 `prom_init()` 会用 `fw_arg0/fw_arg1` 解析命令行，也会用 `fw_arg0/fw_arg1/fw_arg2` 判断 DTB 来源。

`fw_passed_dtb` 保存的是入口汇编筛选出的 `t2`。在当前 X2600 C 代码中，真正传给 `__dt_setup_arch()` 的是 `get_fdt_addr()` 返回的 `_fw_fdt_addr`；而 `_fw_fdt_addr` 又由 `prom_init()` 根据固件参数和 builtin DTB 决定。也就是说，入口阶段保存这些参数，是为了让 C 阶段可以重新做更完整的判断。

第一性原理概述：寄存器是短期载体，全局变量是跨阶段载体。启动入口必须把固件交接的信息从易失寄存器搬到固定内存地址，否则 C 初始化阶段就无法知道命令行、DTB 和平台参数。

清 CP0 Context：丢弃旧地址空间上下文

/* arch/mips/kernel/head.S */
    MTC0        zero, CP0_CONTEXT   # clear context register
#ifdef CONFIG_64BIT
    MTC0        zero, CP0_XCONTEXT
#endif
/* arch/mips/include/asm/mipsregs.h */
#define CP0_CONTEXT $4
806caba0: 40802000  mtc0    zero,c0_context

这里把 CP0 Context 写 0。当前是 32 位反汇编，所以没有 `CP0_XCONTEXT` 指令。Context 寄存器属于 MIPS 地址转换/TLB 相关状态；入口阶段把它清零，是为了不继承固件或早期异常留下的旧上下文。

紧接着建立栈时，`set_saved_sp` 在 SMP 配置下会从 CP0 读取 CPU ID 相关信息，反汇编表现为读取 `c0_context` 后移位并索引 `kernelsp`。这说明 CP0 状态在进入 C 前已经成为后续 per-CPU 栈记录的一部分。

第一性原理概述：地址空间上下文是“上一任执行环境”的状态。内核启动要建立自己的地址空间和异常处理模型，就必须先清除会影响 TLB/上下文判断的旧状态。

`$28` 在 MIPS ABI 中是 `gp`，这里被装入 `init_thread_union`。随后 `sp = init_thread_union + (_THREAD_SIZE - 32 - PT_SIZE)`，也就是把栈顶放到初始线程栈空间的高地址附近，并预留一个空的 `pt_regs` 区域。当前反汇编中 `_THREAD_SIZE - 32 - PT_SIZE` 被折叠成立即数 `7976`。

`set_saved_sp` 把这个栈指针写入 `kernelsp`。反汇编中的 `mfc0 t0,c0_context; srl t0,0x17; ... sw sp,kernelsp+offset` 对应 SMP 分支的“按 CPU ID 保存栈指针”。最后 `sp -= 4 * SZREG`，当前 32 位 `SZREG` 为 4，所以减 16，为进入 C 代码预留调用帧空间。

图 · 入口汇编把 `init_thread_union` 当作初始内核栈所在的线程存储区，为异常帧和调用帧留出空间。

第一性原理概述：C 函数调用需要栈；异常处理需要可保存寄存器的空间；SMP 需要每个 CPU 能找到自己的内核栈。建立 `sp` 和 `kernelsp`，就是把“裸寄存器执行”切换成“可调用 C 函数、可处理异常”的执行模型。

跳转 start_kernel：入口汇编交棒给通用 C 初始化

806cabd0: 0822d9fd  j       808b67f4 <start_kernel>
806cabd4: 00000000  nop

当前反汇编直接是 `j start_kernel`，说明这份镜像没有走 `CONFIG_RELOCATABLE` 的运行时重定位路径。到这个跳转点为止，入口汇编已经完成了 `start_kernel()` 所需的最低条件：CPU 处于受控状态，PC 已回到链接地址语义，BSS 已清零，固件参数已保存，DTB 候选信息已保存，CP0 Context 已清，初始栈已建立。

`start_kernel()` 之后才会进入通用内核初始化；对当前 X2600 来说，后续 `setup_arch()` 中会调用 `prom_init()`、`plat_mem_setup()`，再进入 `__dt_setup_arch(get_fdt_addr())` 扫描 DT。官方 early boot memory 文档说明，普通内存分配器此时仍不可用，体系结构初始化需要在 `setup_arch()` 建立早期内存布局，后续再逐步过渡到完整内存管理。

第一性原理概述：`start_kernel()` 是 C 世界的总入口，但它不是魔法入口。入口汇编必须先把硬件状态整理到 C 代码能依赖的最小不变量集合，然后通用内核初始化才能安全展开。

架构总结

把当前反汇编压缩成一句话：X2600 的 MIPS 入口先关闭早期不该进入的异步性，修正 PC 到链接地址，按当前配置选择 UHI 或 builtin DTB 候选地址，清零 `.bss`，保存固件参数和 DTB 指针，清 CP0 Context，建立 `init_thread_union` 上的初始栈和 `kernelsp`，最后直接跳到 `start_kernel()`。

确定 CPU解决不可控执行状态

CP0 Status 的早期设置让内核先拥有 CP0，并暂时拒绝中断进入。

确定 PC解决加载入口差异

跳到链接地址让后续符号寻址使用同一坐标系。

确定数据解决 C 初值契约

清 BSS 和保存固件参数，让 C 代码看到稳定的全局状态。

确定栈解决 C 调用前提

建立 `sp` 和 `kernelsp`，让函数调用、异常保存和 per-CPU 栈查找有落点。

第一性原理概述：启动不是“初始化很多功能”，而是先消除不确定性。只有当执行地址、CPU 状态、全局内存、启动参数和栈都稳定后，内核才有资格进入 `start_kernel()` 去初始化更复杂的子系统。
