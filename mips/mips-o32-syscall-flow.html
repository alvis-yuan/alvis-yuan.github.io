MIPS O32 系统调用如何进入内核、执行并返回用户态

本文按当前源码解释 `handle_sys`：用户态执行 `syscall` 指令后，CPU 进入 MIPS SYS 异常，异常向量跳到 `handle_sys`；汇编保存 `pt_regs`、跳过 syscall 指令、复制 O32 第 5 到第 8 个参数，必要时进入 trace/seccomp/audit，再通过 `sys_call_table` 调用真正的 `sys_*` 函数。系统调用完成后返回 `entry.S` 的 `syscall_exit_partial`，处理 syscall exit trace、调度、信号、notify-resume，最后恢复寄存器并通过 `eret` 回到用户态。

第一性原理概述：系统调用本质是受控异常。用户态主动执行 `syscall`，CPU 切换到内核异常上下文；内核把用户寄存器固化成 `pt_regs`，用系统调用号选择服务函数，再把结果写回同一个 `pt_regs`，最后恢复状态并 `eret`。

第一性原理概述：进入路径和返回路径共享同一个 `pt_regs`。入口把用户寄存器快照保存下来；内核只修改其中的 EPC、返回值和错误标志；退出时按这个快照恢复用户态机器状态。

异常向量注册：syscall 指令如何找到 handle_sys

/* arch/mips/kernel/traps.c */
extern asmlinkage void handle_sys(void);

set_except_vector(EXCCODE_SYS, handle_sys);

MIPS CPU 执行用户态 `syscall` 指令时，会产生异常码 `EXCCODE_SYS`。内核初始化异常向量时把这个异常码绑定到 `handle_sys`。因此 `scall32-o32.S` 中的 `handle_sys` 是当前 O32 syscall 的体系结构入口。

这里和普通函数调用完全不同：用户态没有直接跳转到内核函数，也不会传入 C ABI 栈帧。CPU 保存异常原因到 CP0 寄存器，进入内核异常入口，随后汇编自己构造 `pt_regs`。

第一性原理概述：系统调用必须通过异常向量进入内核，是为了让 CPU 硬件完成权限级切换，并让内核获得可审计、可恢复的异常现场。

#define PT_R2      8
#define PT_R4      16
#define PT_R7      28
#define PT_R26     136
#define PT_R29     148
#define PT_STATUS  160
#define PT_CAUSE   176

#define PT_SIZE    184
#define TI_FLAGS   4
#define TI_ADDR_LIMIT 20

`SAVE_SOME` 做两件核心工作：第一，判断异常来自用户态还是内核态，来自用户态时切换到当前线程的内核栈；第二，在栈上预留 `PT_SIZE=184` 字节，把用户态的通用寄存器、`CP0_STATUS`、`CP0_CAUSE`、`CP0_EPC` 等写入 `pt_regs`。

`STI` 随后把 CPU 置为内核态并打开中断。这样系统调用主体运行时可以被中断抢占，但返回用户态前会重新关中断检查 pending work，避免错过调度或信号。

第一性原理概述：内核不能信任用户栈，也不能依赖用户 ABI 栈帧。它必须先切换到内核栈，并用 `pt_regs` 保存一份完整、内核可控的返回现场。

EPC 和 O32 参数：跳过 syscall，复制第 5 到第 8 参数

/* arch/mips/kernel/scall32-o32.S */
    lw  t1, PT_EPC(sp)      # skip syscall on return
    addiu   t1, 4           # skip to next instruction
    sw  t1, PT_EPC(sp)

    sw  a3, PT_R26(sp)      # save a3 for syscall restarting

    lw  t0, PT_R29(sp)      # get old user stack pointer
    lw  t5, TI_ADDR_LIMIT($28)
    addu    t4, t0, 32
    and t5, t4
    bltz    t5, bad_stack   # -> sp is bad

load_a4: user_lw(t5, 16(t0))    # argument #5 from usp
load_a5: user_lw(t6, 20(t0))    # argument #6 from usp
load_a6: user_lw(t7, 24(t0))    # argument #7 from usp
load_a7: user_lw(t8, 28(t0))    # argument #8 from usp
loads_done:
    sw  t5, 16(sp)      # argument #5 to ksp
    sw  t6, 20(sp)      # argument #6 to ksp
    sw  t7, 24(sp)      # argument #7 to ksp
    sw  t8, 28(sp)      # argument #8 to ksp

系统调用返回时不应该再次执行同一条 `syscall` 指令，所以入口立即把 `PT_EPC` 加 4。MIPS 普通指令长度是 4 字节，返回用户态时 `eret` 会跳到更新后的 EPC，也就是 `syscall` 的下一条指令。

O32 ABI 的前四个参数在 `a0-a3`，更多参数放在用户栈。`handle_sys` 为了让后续 C 函数按内核栈上的参数访问第 5 到第 8 个参数，会从旧用户栈 `PT_R29` 读取 `16/20/24/28(t0)`，再写到当前内核栈 `16/20/24/28(sp)`。

第一性原理概述：系统调用入口必须同时满足 CPU 返回语义和 C 函数调用语义：EPC 要指向下一条用户指令，参数要被整理成内核后续调用可以消费的形式。

用户栈异常修复：为什么有 __ex_table

/* arch/mips/kernel/scall32-o32.S */
    .section __ex_table,"a"
    PTR load_a4, bad_stack_a4
    PTR load_a5, bad_stack_a5
    PTR load_a6, bad_stack_a6
    PTR load_a7, bad_stack_a7
    .previous

bad_stack:
    li  v0, EFAULT
    sw  v0, PT_R2(sp)
    li  t0, 1           # set error flag
    sw  t0, PT_R7(sp)
    j   o32_syscall_exit

bad_stack_a4:
    li  t5, 0
    b   load_a5

用户栈指针可能非法，也可能检查通过后在实际 load 时 fault。`__ex_table` 把每个 `user_lw` 的 fault 地址映射到对应修复标签。修复路径把读取失败的参数置 0，然后继续读取后续参数；如果初始用户栈范围检查失败，则直接设置 `EFAULT` 和错误标志。

前面链接脚本文档中已经说明 `EXCEPTION_TABLE(16)` 会把这些表项放入内核镜像。这里就是该异常表的一个实际使用场景。

第一性原理概述：内核访问用户地址必须允许失败。异常表让 fault 不变成内核崩溃，而是跳转到受控修复代码，把失败转换为系统调用返回错误。

trace/seccomp/audit 入口：系统调用执行前的策略点

/* arch/mips/kernel/scall32-o32.S */
    lw  t0, TI_FLAGS($28)       # syscall tracing enabled?
    li  t1, _TIF_WORK_SYSCALL_ENTRY
    and t0, t1
    bnez    t0, syscall_trace_entry # -> yes

syscall_trace_entry:
    SAVE_STATIC
    move    a0, sp
    move    a1, v0
    subu    t2, v0,  __NR_O32_Linux
    bnez    t2, 1f /* __NR_syscall at offset 0 */
    lw  a1, PT_R4(sp)

1:  jal syscall_trace_enter

    bltz    v0, 1f          # seccomp failed? Skip syscall
    RESTORE_STATIC
    lw  v0, PT_R2(sp)       # Restore syscall (maybe modified)
    lw  a0, PT_R4(sp)
    lw  a1, PT_R5(sp)
    lw  a2, PT_R6(sp)
    lw  a3, PT_R7(sp)
    j   syscall_common

1:  j   syscall_exit
/* arch/mips/include/asm/thread_info.h */
#define _TIF_WORK_SYSCALL_ENTRY (_TIF_NOHZ | _TIF_SYSCALL_TRACE | \
                 _TIF_SYSCALL_AUDIT | \
                 _TIF_SYSCALL_TRACEPOINT | _TIF_SECCOMP)

/* arch/mips/kernel/ptrace.c */
asmlinkage long syscall_trace_enter(struct pt_regs *regs, long syscall)
{
    user_exit();
    current_thread_info()->syscall = syscall;

    if (test_thread_flag(TIF_SYSCALL_TRACE)) {
        if (tracehook_report_syscall_entry(regs))
            return -1;
        syscall = current_thread_info()->syscall;
    }

#ifdef CONFIG_SECCOMP
    if (unlikely(test_thread_flag(TIF_SECCOMP))) {
        struct seccomp_data sd;
        sd.nr = syscall;
        sd.arch = syscall_get_arch(current);
        syscall_get_arguments(current, regs, args);
        ret = __secure_computing(&sd);
        if (ret == -1)
            return ret;
        syscall = current_thread_info()->syscall;
    }
#endif

    if (unlikely(test_thread_flag(TIF_SYSCALL_TRACEPOINT)))
        trace_sys_enter(regs, regs->regs[2]);

    audit_syscall_entry(syscall, regs->regs[4], regs->regs[5],
                regs->regs[6], regs->regs[7]);
    if (syscall < 0)
        syscall_set_return_value(current, regs, -ENOSYS, 0);
    return syscall;
}

如果 `_TIF_WORK_SYSCALL_ENTRY` 任一位存在，入口不会直接分发，而是先调用 `syscall_trace_enter()`。这条 C 路径完成 ptrace syscall-entry hook、seccomp 过滤、sys_enter tracepoint 和 audit 记录。官方 seccomp 文档强调，过滤器看到的是系统调用号、参数和 arch 信息，不能解引用用户指针，从而避免 TOCTOU 类问题。

`syscall_trace_enter()` 返回负数时，汇编跳到 `syscall_exit`，跳过真正系统调用。否则恢复可能被 ptrace/seccomp 修改过的 syscall number 和参数，进入 `syscall_common`。

第一性原理概述：系统调用入口不仅是函数分发点，也是安全、审计和调试策略点。把策略点放在真实 `sys_*` 之前，才能做到“拒绝时不执行服务函数”。

系统调用表分发：v0 如何变成 sys_read/sys_write

/* arch/mips/kernel/scall32-o32.S */
syscall_common:
    subu    v0, v0, __NR_O32_Linux  # check syscall number
    sltiu   t0, v0, __NR_O32_Linux_syscalls
    beqz    t0, illegal_syscall

    sll t0, v0, 2
    la  t1, sys_call_table
    addu    t1, t0
    lw  t2, (t1)        # syscall routine

    beqz    t2, illegal_syscall

    jalr    t2          # Do The Real Thing (TM)
/* arch/mips/kernel/scall32-o32.S */
#define __SYSCALL(nr, entry)    PTR entry
    .align  2
    .type   sys_call_table, @object
EXPORT(sys_call_table)
#include <asm/syscall_table_32_o32.h>
#undef __SYSCALL

/* arch/mips/include/generated/asm/syscall_table_32_o32.h */
__SYSCALL(4000,sys_syscall)
__SYSCALL(4001,sys_exit)
__SYSCALL(4002,__sys_fork)
__SYSCALL(4003,sys_read)
__SYSCALL(4004,sys_write)
__SYSCALL(4005,sys_open)
__SYSCALL(4006,sys_close)

用户态把 O32 syscall number 放在 `v0`。因为 O32 的编号从 `4000` 开始，汇编先 `v0 -= __NR_O32_Linux` 得到数组下标，再检查它小于 `441`。每个表项是 32 位函数指针，所以左移 2 字节寻址，取出 `sys_call_table[index]` 到 `t2`，最后 `jalr t2` 调用真正服务函数。

如果系统调用号越界，或者表项为 0，则进入 `illegal_syscall`，写回 `ENOSYS`。生成表中的 `sys_ni_syscall` 也表达了“有编号但当前内核不实现”的情况。

第一性原理概述：系统调用表把“不可信的用户态数字”变成“内核中受控的函数指针”。边界检查必须在取表项之前完成，否则 syscall number 就会变成任意内核地址索引。

结果与 errno：MIPS O32 如何向用户态表达错误

/* arch/mips/kernel/scall32-o32.S */
    li  t0, -EMAXERRNO - 1  # error?
    sltu    t0, t0, v0
    sw  t0, PT_R7(sp)       # set error flag
    beqz    t0, 1f

    lw  t1, PT_R2(sp)       # syscall number
    negu    v0              # error
    sw  t1, PT_R0(sp)       # save it for syscall restarting
1:  sw  v0, PT_R2(sp)       # result

o32_syscall_exit:
    j   syscall_exit_partial

illegal_syscall:
    li  v0, ENOSYS          # error
    sw  v0, PT_R2(sp)
    li  t0, 1               # set error flag
    sw  t0, PT_R7(sp)
    j   o32_syscall_exit

Linux 内核 C 函数通常返回负 errno，例如 `-EFAULT`。O32 用户态约定不是直接把负数原样当 errno，而是用 `a3` 表示错误标志：`a3=0` 表示成功，`v0` 是结果；`a3=1` 表示失败，`v0` 是正 errno。这里 `PT_R7` 对应用户态 `a3`，`PT_R2` 对应用户态 `v0`。

`sltu t0, -EMAXERRNO-1, v0` 用无符号比较检测 `v0` 是否落在内核 errno 返回范围。检测到错误后，汇编对 `v0` 取负，保存正 errno，并把原 syscall number 写到 `PT_R0`，用于后续 syscall restart 逻辑。

第一性原理概述：内核内部错误码和用户 ABI 错误码不一定同形。返回路径要把 C 函数返回值转换成 O32 ABI 承诺的寄存器语义。

返回用户态：syscall_exit_partial 到 eret

/* arch/mips/kernel/entry.S */
FEXPORT(syscall_exit_partial)
    local_irq_disable       # make sure need_resched doesn't
                    # change between and return
    LONG_L  a2, TI_FLAGS($28)   # current->work
    li  t0, _TIF_ALLWORK_MASK
    and t0, a2
    beqz    t0, restore_partial
    SAVE_STATIC
syscall_exit_work:
    LONG_L  t0, PT_STATUS(sp)       # returning to kernel mode?
    andi    t0, t0, KU_USER
    beqz    t0, resume_kernel
    li  t0, _TIF_WORK_SYSCALL_EXIT
    and t0, a2          # a2 is preloaded with TI_FLAGS
    beqz    t0, work_pending    # trace bit set?
    local_irq_enable
    TRACE_IRQS_ON
    move    a0, sp
    jal syscall_trace_leave
    b   resume_userspace
/* arch/mips/kernel/entry.S */
resume_userspace:
    local_irq_disable
    LONG_L  a2, TI_FLAGS($28)   # current->work
    andi    t0, a2, _TIF_WORK_MASK
    bnez    t0, work_pending
    j   restore_all

work_pending:
    andi    t0, a2, _TIF_NEED_RESCHED
    beqz    t0, work_notifysig
work_resched:
    TRACE_IRQS_OFF
    jal schedule
    ...
work_notifysig:
    move    a0, sp
    li  a1, 0
    jal do_notify_resume
    j   resume_userspace_check

    .macro  RESTORE_SOME docfi=0

    LONG_L  v0, PT_STATUS(sp)
    ...

    cfi_ld  $5,  PT_R5, \docfi
    cfi_ld  $4,  PT_R4, \docfi

    .macro  RESTORE_SP_AND_RET docfi=0

#ifdef CONFIG_CPU_MIPSR6
    eretnc
#else

#endif

`syscall_exit_partial` 是 `handle_sys` 的正常出口。它先关中断，读取当前线程 `TI_FLAGS`。如果没有 `_TIF_ALLWORK_MASK` 相关工作，则走 `restore_partial`，只恢复 syscall 入口保存的必要寄存器，最后 `eret` 回用户态。

如果有 pending work，`syscall_exit_work` 会区分是否返回用户态。如果存在 `_TIF_WORK_SYSCALL_EXIT`，就调用 `syscall_trace_leave()` 处理 audit、`trace_sys_exit`、ptrace syscall-exit。随后 `resume_userspace` 再检查 `_TIF_WORK_MASK`，必要时调度或处理信号、uprobes、notify-resume，直到没有工作，才恢复寄存器返回。

/* arch/mips/kernel/ptrace.c */
asmlinkage void syscall_trace_leave(struct pt_regs *regs)
{
    user_exit();
    audit_syscall_exit(regs);

    if (unlikely(test_thread_flag(TIF_SYSCALL_TRACEPOINT)))
        trace_sys_exit(regs, regs_return_value(regs));

    if (test_thread_flag(TIF_SYSCALL_TRACE))
        tracehook_report_syscall_exit(regs, 0);

    user_enter();
}
第一性原理概述：返回用户态不是简单恢复寄存器。内核必须在最后一刻检查调度、信号、trace、audit 等工作，因为这些状态可能在系统调用执行期间被中断或其他 CPU 改变。

sys_syscall 特例：用户态 syscall(__NR_xxx, ...)

/* arch/mips/kernel/scall32-o32.S */
LEAF(sys_syscall)
    subu    t0, a0, __NR_O32_Linux  # check syscall number
    sltiu   v0, t0, __NR_O32_Linux_syscalls
    beqz    t0, einval      # do not recurse
    sll t1, t0, 2
    beqz    v0, einval
    lw  t2, sys_call_table(t1)      # syscall routine

    move    a0, a1          # shift argument registers
    move    a1, a2
    move    a2, a3
    lw  a3, 16(sp)
    lw  t4, 20(sp)
    lw  t5, 24(sp)
    lw  t6, 28(sp)
    sw  t4, 16(sp)
    sw  t5, 20(sp)
    sw  t6, 24(sp)
    jr  t2

einval: li  v0, -ENOSYS
    jr  ra
END(sys_syscall)

O32 表的第 0 项是 `sys_syscall`，用于支持用户态调用通用 `syscall(nr, arg1, arg2, ...)`。这里真正的 syscall number 在 `a0`，所以 `sys_syscall` 先检查 `a0`，禁止递归调用 `syscall` 自己，然后把参数整体左移一位：`a1` 变成真实 `a0`，`a2` 变成真实 `a1`，以此类推。

它最后 `jr t2` 而不是 `jalr` 后返回处理，因为外层 `handle_sys` 已经建立了返回路径。真实 `sys_*` 返回后仍会回到外层系统调用出口，按同样的 errno 和 `syscall_exit_partial` 规则返回用户态。

第一性原理概述：`sys_syscall` 是 ABI 兼容层。它不提供新服务，只把“间接系统调用”的参数重新排列成普通系统调用函数期望的形态。

总结

当前 O32 系统调用路径可以分为五步：第一，用户态 `syscall` 指令触发 `EXCCODE_SYS`，异常向量进入 `handle_sys`；第二，`SAVE_SOME` 切换到内核栈并保存 `pt_regs`，入口把 EPC 加 4 并复制 O32 栈参数；第三，如果 `_TIF_WORK_SYSCALL_ENTRY` 置位，进入 `syscall_trace_enter()` 做 ptrace、seccomp、tracepoint、audit；第四，`v0 - 4000` 得到表下标，通过 `sys_call_table` 调用真正 `sys_*`；第五，把返回值转换成 O32 的 `v0/a3` 语义，进入 `syscall_exit_partial`，处理 exit work、调度、信号，最终恢复 `CP0_STATUS/CP0_EPC` 和寄存器并 `eret` 回用户态。

和前面的 entry-return-path.html 合起来看，`scall32-o32.S` 负责“如何从用户态 syscall 指令进入内核并调用服务函数”，`entry.S` 负责“服务函数结束后是否能立刻返回，以及如何安全返回”。两者以栈上的 `pt_regs` 为接口。

第一性原理概述：系统调用不是普通跨边界函数调用，而是一条完整的状态机：硬件异常建立权限边界，汇编建立寄存器快照，C 层执行策略和服务，退出汇编在最后一刻处理 pending work，再用 `eret` 恢复用户态。
