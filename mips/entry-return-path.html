entry.S 返回路径：从异常栈帧回到用户态或内核态

这份文档分析当前 `entry.disassemble` 中 `80016220 <ret_from_irq>` 到 `80016470 <mips_ihb>` 的路径。它解释的不是最初异常入口保存现场，而是中断、异常、系统调用和新任务第一次运行之后，如何检查延迟工作、触发调度、处理信号/trace，并最终恢复 `pt_regs` 执行 `eret`。

官方依据

本文先获取 Linux 官方文档，再结合当前仓库源码和反汇编编写。外部链接只作为原文入口，页面本身不加载任何外部资源。

主题官方原文链接本文使用的事实
generic IRQLinux generic IRQ handling低层体系结构代码在中断触发后进入 generic IRQ；文档明确区分 IRQ flow 和 chip details，并说明 IRQ 处理发生在体系结构入口建立的上下文中。
ftrace / latencyftrace - Function Tracer官方 trace 格式解释了 `irqs-off`、`need-resched`、hardirq/softirq、preempt-depth 等状态，这些正是本文件返回路径反复采样和保护的状态。
preempt/irq latencyftrace irqsoff官方文档说明关中断期间 CPU 不能响应普通外部事件；因此返回路径在采样 thread flags 时短暂关中断，是为了避免检查与返回之间丢失工作。
scheduler wakeupftrace wakeup官方文档用 wakeup tracer 说明 `need-resched` 代表有调度需求；本文件通过 `_TIF_NEED_RESCHED` 决定是否进入 `schedule()` 或 `preempt_schedule_irq()`。

当前本地常量

/* include/generated/asm-offsets.h */
#define PT_R29 148 /* offsetof(struct pt_regs, regs[29]) */
#define PT_R31 156 /* offsetof(struct pt_regs, regs[31]) */
#define PT_LO 168 /* offsetof(struct pt_regs, lo) */
#define PT_HI 164 /* offsetof(struct pt_regs, hi) */
#define PT_EPC 180 /* offsetof(struct pt_regs, cp0_epc) */
#define PT_STATUS 160 /* offsetof(struct pt_regs, cp0_status) */
#define PT_SIZE 184 /* sizeof(struct pt_regs) */

#define TI_FLAGS 4 /* offsetof(struct thread_info, flags) */
#define TI_PRE_COUNT 16 /* offsetof(struct thread_info, preempt_count) */
#define TI_REGS 24 /* offsetof(struct thread_info, regs) */

这些 offset 与反汇编完全对应：例如 `lw t0,160(sp)` 是读 `PT_STATUS`，`lw a2,4(gp)` 是读 `TI_FLAGS`，`sw s0,24(gp)` 是保存 `TI_REGS`。

第一性原理概述：返回路径的核心问题是“现在能不能直接回去”。答案取决于返回目标、是否有调度需求、是否有信号/通知/syscall trace 工作，以及保存的寄存器帧是否已经完整恢复。

路径总览

图 · `entry.S` 不是每种事件都单独返回，而是把 IRQ、异常、syscall、fork/kthread 的出口汇聚到同一套检查与恢复逻辑。

第一性原理概述：入口可以来自很多事件，但返回前要回答的问题高度相同：目标态是什么、是否还能抢占、用户态前是否有必须处理的工作、寄存器帧如何原样恢复。

ret_from_irq：从中断处理回到统一返回路径

#else

#endif

80016220 <ret_from_irq>:
80016220: af900018  sw      s0,24(gp)

中断入口在 `genex.S` 中调用 `plat_irq_dispatch`，恢复被切到 IRQ stack 前保存的 `sp`，然后跳 `ret_from_irq`。当前配置打开 `CONFIG_PREEMPTION`，所以 `__ret_from_irq` 被定义为 `ret_from_exception`，反汇编中 `ret_from_irq` 保存 `s0` 到 `thread_info.regs` 后立刻落到 `80016224 <ret_from_exception>`。

`$28` 是当前 `thread_info` 指针；`TI_REGS=24`，所以 `sw s0,24(gp)` 是记录当前任务的 pt_regs 指针。这样 C 代码或调试路径需要访问“当前任务的寄存器帧”时，可以从 thread_info 找到它。

第一性原理概述：中断处理完不能直接 `eret`，因为中断可能设置了调度、信号或通知工作。先把当前 pt_regs 位置挂到 thread_info，再进入统一返回检查，是为了让后续路径有共同的现场视图。

`PT_STATUS=160`，这里读取异常入口保存到 pt_regs 中的 CP0 Status。`KU_USER=0x10`，如果该位为 0，说明返回目标不是用户态，于是走 `resume_kernel`；否则进入 `resume_userspace`。

源码注释强调一种特殊情况：可能来自内核空间执行的 syscall，例如失败的 `kernel_execve()`。所以不能简单认为“走到 ret_from_exception 就一定回用户态”，必须用保存的 status 判断返回目标。

第一性原理概述：用户态返回前需要处理信号、notify、uprobes、syscall trace；内核态返回前主要考虑抢占。两者规则不同，所以第一步必须从硬件保存的 status 中确认原始上下文。

返回用户态前，代码先 `di; ehb`，然后读取 `thread_info.flags`。源码注释解释了原因：不能在采样 flags 和真正返回之间漏掉一个中断设置的 `need_resched`。官方 ftrace 文档也把 `irqs-off` 和 `need-resched` 作为判断延迟与调度状态的关键字段。

当前 `_TIF_WORK_MASK` 编译成 `0xe6`，对应 signal、need_resched、notify_resume、uprobe、notify_signal 这些“返回用户态前必须处理”的工作。如果没有这些位，直接进入 `restore_all`。

第一性原理概述：用户态没有内核栈和内核锁上下文，一旦 `eret` 出去，内核就失去立即处理这些工作的机会。因此返回用户态前要在关中断的小窗口里做最后一次一致性检查。

resume_kernel：内核态返回时的抢占检查

/* arch/mips/kernel/entry.S */
#ifdef CONFIG_PREEMPTION
resume_kernel:
    local_irq_disable
    lw      t0, TI_PRE_COUNT($28)
    bnez    t0, restore_all
    LONG_L  t0, TI_FLAGS($28)
    andi    t1, t0, _TIF_NEED_RESCHED
    beqz    t1, restore_all
    LONG_L  t0, PT_STATUS(sp)       # Interrupts off?
    andi    t0, 1
    beqz    t0, restore_all
    PTR_LA  ra, restore_all
    j       preempt_schedule_irq
#endif
/* kernel/sched/core.c */
asmlinkage __visible void __sched preempt_schedule_irq(void)
{
    enum ctx_state prev_state;

    /* Catch callers which need to be fixed */
    BUG_ON(preempt_count() || !irqs_disabled());

    prev_state = exception_enter();

    do {
        preempt_disable();
        local_irq_enable();
        __schedule(true);
        local_irq_disable();
        sched_preempt_enable_no_resched();
    } while (need_resched());

    exception_exit(prev_state);
}

内核态返回不处理用户信号，但在抢占式内核中要考虑“是否应该切换任务”。这段代码要求三个条件同时满足：`preempt_count == 0`，`_TIF_NEED_RESCHED` 已置位，保存的 `PT_STATUS` 显示被打断处中断是开的。任何一个不满足，都直接恢复现场。

如果条件满足，设置 `ra = restore_all` 后跳 `preempt_schedule_irq`。C 函数注释说明它从 IRQ context 做内核抢占调度，调用和返回时都保持关中断；它内部临时开中断调用 `__schedule(true)`，然后关中断回来。返回地址设成 `restore_all`，意味着调度结束后继续恢复被打断的寄存器帧。

第一性原理概述：内核抢占只能发生在安全点：没有嵌套 preempt 禁止、确实需要调度、原上下文允许中断。这个三重门槛防止在持锁或关中断临界区中切走当前任务。

`copy_thread()` 为新任务设置 `thread.reg31`：内核线程第一次被切入时跳 `ret_from_kernel_thread`，普通用户子进程第一次被切入时跳 `ret_from_fork`。两者第一步都调用 `schedule_tail(prev)` 完成 scheduler 的切换尾处理。

内核线程路径随后把 `s1` 放入 `a0`，调用 `s0`，也就是 `process.c` 中设置的内核线程函数和参数；函数返回后进入 `syscall_exit`。普通用户子进程路径则落到紧随其后的 `syscall_exit`，让“fork 返回用户态前的工作检查”复用系统调用返回逻辑。

第一性原理概述：新任务不是从 C 函数正常返回出来的，它是被 scheduler 第一次切换进来的。给 `ra/reg31` 预置一个架构返回桩，就能把“第一次运行”接到统一的返回检查和寄存器恢复路径上。

`syscall_exit` 比普通异常返回检查更多位，因为系统调用返回还要考虑 syscall trace、audit、tracepoint、NOHZ 等工作。当前 `_TIF_ALLWORK_MASK` 编译成 `0x840800ee`，反汇编通过 `lui/ori` 组装这个 32 位掩码。

和 `resume_userspace` 一样，这里也先关中断再采样 flags。原因相同：如果在“看起来无工作”和“真正返回用户态”之间有中断设置了 resched 或 signal，直接返回会延迟甚至漏掉必须在用户态前完成的处理。

第一性原理概述：系统调用返回是内核与用户态的高频边界。这个边界不仅要返回寄存器，还要兑现 tracing、audit、信号、调度等内核承诺，所以它使用比普通异常返回更大的工作掩码。

`restore_all` 先恢复临时寄存器、`at`、callee-saved 静态寄存器；`restore_partial` 则从 `RESTORE_SOME` 开始，适用于系统调用快路径已经不需要完整恢复静态寄存器的情况。当前反汇编里 `PT_LO=168`、`PT_HI=164`、`PT_STATUS=160`、`PT_EPC=180`、`PT_R29=148` 都对上本地 offset。

`RESTORE_SOME` 对 CP0 Status 做两步处理：先把当前 Status 的关键硬件位清到安全状态，再从 pt_regs 恢复大部分原状态，但保留当前 Status 中的 `ST0_CU1 | ST0_FR | ST0_IM` 位。这避免把浮点/中断屏蔽等敏感位用过时栈帧完全覆盖。最后写 CP0_EPC，恢复通用寄存器和 sp，执行 `eret`。

第一性原理概述：异常返回的本质是让 CPU 相信“从未发生过中断/异常”。这要求寄存器、Status、EPC、sp 都回到保存现场；但硬件敏感位还要按当前内核判断合并，不能盲目整字恢复。

/* arch/mips/kernel/signal.c */
asmlinkage void do_notify_resume(struct pt_regs *regs, void *unused,
    __u32 thread_info_flags)
{
    local_irq_enable();

    user_exit();

    if (thread_info_flags & _TIF_UPROBE)
        uprobe_notify_resume(regs);

    /* deal with pending signal delivery */
    if (thread_info_flags & (_TIF_SIGPENDING | _TIF_NOTIFY_SIGNAL))
        do_signal(regs);

    if (thread_info_flags & _TIF_NOTIFY_RESUME) {
        tracehook_notify_resume(regs);
        rseq_handle_notify_resume(NULL, regs);
    }

    user_enter();
}

如果 `_TIF_NEED_RESCHED` 置位，先调用 `schedule()`。调度返回后重新关中断并重新读取 flags，因为调度期间可能产生新的信号、notify 或再次需要调度。只要仍有 `NEED_RESCHED`，就循环回 `work_resched`。

没有调度需求但仍有 `_TIF_WORK_MASK` 时，进入 `do_notify_resume(regs, NULL, flags)`。C 侧会开中断，处理 uprobe、signal、notify_resume、rseq 等返回用户态前工作。处理完跳回 `resume_userspace_check`，重新判断返回目标和工作位，避免处理过程中状态改变。

第一性原理概述：返回用户态前的工作不是一次性快照，而是一个可能自我产生新工作的闭环。调度可能带来信号，信号处理也可能改变返回状态，所以每轮处理后必须重查。

第一性原理概述：快路径的价值是避免不必要的保存恢复；慢路径的要求是现场必须完整。`syscall_exit_partial` 用一次 flags 判断，在性能和正确性之间切换。

`syscall_exit_work` 先再次确认是否返回用户态。如果不是，走内核态抢占检查。若返回用户态但没有 `_TIF_WORK_SYSCALL_EXIT`，说明剩余工作属于普通 `work_pending`。如果有 syscall exit 工作，则开中断调用 `syscall_trace_leave()`，执行 audit、sys_exit tracepoint、ptrace tracehook。

`mips_ihb` 是一个很小的叶子函数，核心就是 `jr.hb ra`。它不是返回路径主体的一部分，而是同一个汇编文件里导出的 MIPS instruction hazard barrier 辅助函数。源码注释说明 C 代码通常使用内联版本 `instruction_hazard()`。

结合前面多处 `di/e i` 后的 `ehb` 可以看出，MIPS 在修改 CP0 状态或控制流相关状态后，需要显式 hazard barrier 来保证后续指令看到正确状态。

第一性原理概述：现代 CPU 不保证控制寄存器修改立刻对后续取指/执行可见。hazard barrier 的作用是把“我刚改了执行环境”变成对流水线可观察的边界。

架构总结

当前 `entry.disassemble` 的主线可以概括为：中断或异常处理结束后，先根据保存的 `PT_STATUS` 判断返回用户态还是内核态；用户态返回前在关中断窗口采样 `thread_info.flags`，必要时调度、处理信号/notify、执行 syscall trace；内核态返回则只在安全条件满足时走 `preempt_schedule_irq`；所有路径最终通过 `restore_all` 或 `restore_partial` 恢复 pt_regs，写回 CP0 Status/EPC，并用 `eret` 回到被打断的位置。

统一出口减少重复

IRQ、exception、syscall、fork/kthread 最终汇聚到同一套返回检查。

关中断采样避免竞态

检查 flags 与真正返回之间不能被新中断插入并丢失 `need_resched`。

用户态边界兑现内核承诺

signal、notify、uprobes、audit、trace 都必须在离开内核前处理。

精确恢复维持异常透明性

`restore_all` 把寄存器和 CP0 返回状态恢复到异常入口保存的现场。

第一性原理概述：异常返回的目标不是“尽快返回”，而是“只在没有欠账时返回”。`entry.S` 用最小汇编逻辑把调度、信号、trace 和硬件现场恢复串成一个闭环，保证返回点看到一致的 CPU 和任务状态。
