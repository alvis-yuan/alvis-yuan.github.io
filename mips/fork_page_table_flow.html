fork 创建进程时页表的创建与更新

本文把 Linux 中 fork() 时进程地址空间和页表的创建、复制、写时复制（COW）与 TLB 关系串起来说明，并结合本仓库代码路径给出可对照的实现落点。重点覆盖：fork -> kernel_clone -> copy_process -> copy_mm -> dup_mm -> dup_mmap -> copy_page_range。

先说结论

fork 不是立刻把所有物理页都复制一份。

Linux 的常规 fork() 语义是：先创建一个新的 task_struct 和新的 mm_struct，复制 VMA 元数据和页表层级；对私有可写页，把父子两边的 PTE 都改成只读，随后在第一次写入时再通过缺页异常分配新页并完成 COW。

fork 的主要开销是什么？

权威表述可直接引用 fork(2) 的 NOTE：Linux 中 fork() 使用 copy-on-write，因此主要代价是复制父进程的页表，以及创建子进程自身的内核对象，而不是复制全部用户物理页。

  • 新进程的顶层页表会被创建出来，且内核映射部分通常直接继承自内核模板。
  • 用户态各个 VMA 会被复制，必要时还会复制页表层级与叶子 PTE。
  • 私有可写映射在 fork() 后通常会降级为父子共享的只读映射。
  • 真正分配新的物理页，通常发生在后续写缺页，而不是发生在 fork() 这个系统调用本身。

二、新进程的顶层页表是怎么来的

在普通 fork() 中，子进程不是共享父进程的 mm，而是创建新的 mm_struct。顶层页表的分配发生在 mm_init() 里面触发的 pgd_alloc()。对于这棵 MIPS 树，pgd_alloc() 位于 arch/mips/mm/pgtable.c。

pgd_t *pgd_alloc(struct mm_struct *mm)
{
    pgd_t *ret, *init;

    ret = (pgd_t *) __get_free_pages(GFP_KERNEL, PGD_ORDER);
    if (ret) {
        init = pgd_offset(&init_mm, 0UL);
        pgd_init((unsigned long)ret);
        memcpy(ret + USER_PTRS_PER_PGD, init + USER_PTRS_PER_PGD,
               (PTRS_PER_PGD - USER_PTRS_PER_PGD) * sizeof(pgd_t));
    }
    return ret;
}

这一段代码说明了什么

  • 先为子进程分配一份新的顶层 PGD 页。
  • pgd_init() 先把新 PGD 初始化成“空/无效”的用户态部分。
  • 再把 init_mm 的内核半区 PGD 直接拷贝给子进程。
MIPS 树里的一个关键点

这意味着子进程从一开始就拥有与内核一致的 kernel mapping，而用户半区映射则需要随后在 dup_mmap() / copy_page_range() 阶段按 VMA 和 PTE 逐步补齐。

三、VMA 何时复制，页表何时真正遍历

dup_mmap() 在持有父子 mmap_write_lock 的情况下遍历父进程 VMA，为子进程建立对应 VMA，并在满足条件时调用 copy_page_range(tmp, mpnt)。这一步是“地址空间结构复制”的核心。

/* Link in the new vma and copy the page table entries. */
*pprev = tmp;
...
mm->map_count++;
if (!(tmp->vm_flags & VM_WIPEONFORK))
    retval = copy_page_range(tmp, mpnt);

哪些 VMA 可能不立即复制 PTE

copy_page_range() 开头有一个很重要的优化：

/* Don't copy ptes where a page fault will fill them correctly. */
if (!(src_vma->vm_flags & (VM_HUGETLB | VM_PFNMAP | VM_MIXEDMAP)) &&
    !src_vma->anon_vma)
    return 0;
含义

如果这段 VMA 没有匿名内存状态，又不是 hugetlb、PFNMAP、MIXEDMAP 这类特殊映射，那么 Linux 可能选择不去复制叶子 PTE，而让以后真正访问时由 page fault 正确补齐。这样可以显著降低 fork() 在大只读映射场景下的成本。

四、真正复制页表项时，内核做了什么

copy_page_range() 按页表层级向下走，直到 PTE 级别。每一级逻辑基本都是：

  • 为子进程按需分配下级页表页，例如 p4d_alloc()、pud_alloc()、pmd_alloc()。
  • 遍历父进程对应范围的条目。
  • 在 PTE 级别根据条目类型分别处理 present / nonpresent / huge / swap / device-private 等情况。

非 present 条目

copy_nonpresent_pte() 会处理 swap entry、migration entry、device private entry 等情况。它不是“分配一页新物理内存再拷一份”，而是复制/调整描述这些特殊状态的页表项，并维护引用与统计。

present 条目

copy_present_pte() 是常见路径。它会先通过 vm_normal_page() 找到父进程当前 PTE 对应的物理页，再决定是直接共享，还是因为 pinned page 等原因需要提前复制一页给子进程。

五、COW 在 fork 期间是如何准备好的

对于私有可写映射，fork() 并不会马上复制物理页，而是在 copy_present_pte() 中把父子两边的 PTE 都改成只读：

if (is_cow_mapping(vm_flags) && pte_write(pte)) {
    ptep_set_wrprotect(src_mm, addr, src_pte);
    pte = pte_wrprotect(pte);
}
...
set_pte_at(dst_vma->vm_mm, addr, dst_pte, pte);

fork 刚结束时

  • 父进程原来可写的私有页，PTE 会被改成只读。
  • 子进程得到的是同一个物理页的只读映射。
  • 两边共享同一个物理页，页引用计数与反向映射计数增加。
关键认识

COW 的本质不是“fork 后马上复制页”，而是“fork 后把可写共享先降成只读共享，等谁先写谁触发缺页，再复制”。这也是 fork() 能足够轻量的根本原因。

六、第一次访问某个 VA 时，页表、缺页和 TLB 分别做什么

从使用者角度，地址转换可分成三种情况：

场景 TLB 页表 是否分配新物理页
TLB 命中 已有缓存翻译 不需要再走主页表 否
TLB miss，但页表已有有效映射 未命中 页表查得到 PTE 否，只需要回填 TLB
TLB miss，且页表也无有效映射或发生写保护 未命中 进入缺页异常处理 视具体 fault 类型而定：可能分配新页、读文件页、执行 COW，或报错
和 fork 的关系

fork 阶段最常做的是“建立页表条目并准备 COW 语义”，不是“为每个用户 VA 立即分配一个新物理页”。真正分配新页通常发生在后续 page fault，例如匿名页首次写入或 COW 写缺页。

七、结合官方资料，如何理解 fork 的“权威语义”

官方手册 fork(2)

man-pages 的权威表述是：父子进程在 fork() 时拥有不同的地址空间，但起始内容相同；Linux 借助 copy-on-write 实现，因此 fork() 的主要惩罚是复制页表和创建新的进程结构，而不是复制全部物理页。

Linux kernel 文档

docs.kernel.org 的 MM 文档强调：页表是分层结构；叶子条目可以指向真正的数据页，也可以是 swap/migration 等特殊条目；安装新的页表项时必须保证 VMA 稳定并在相应层级持有正确的页表锁。

因此可以得到一条稳定的工程判断

分析 fork() 时，不要把“复制地址空间”误解成“复制每一页物理内存”。在 Linux 里，这通常意味着：复制 mm_struct、复制 VMA 拓扑、按需复制/共享页表层级、对私有可写页建立 COW 保护。

八、结合本仓库代码，可抓住的几个观察点

  1. kernel/fork.c 里的 copy_mm() 是地址空间复制与共享的分叉点。CLONE_VM 共享地址空间；普通 fork() 则创建新 mm。
  2. arch/mips/mm/pgtable.c 说明了 MIPS 上顶层页表不是完全从父进程原样拷贝，而是新分配一份，再继承内核半区模板。
  3. dup_mmap() 先复制 VMA，再决定是否复制页表。这意味着“虚拟地址范围定义”和“PTE 具体复制”是两层逻辑。
  4. copy_page_range() 对没有匿名页状态的普通映射会尽量偷懒，让缺页路径后补，这是一项重要的 fork 优化。
  5. copy_present_pte() 是理解 COW 的最佳入口：它清晰展示了父子共享同一物理页、同时降为只读的关键动作。
