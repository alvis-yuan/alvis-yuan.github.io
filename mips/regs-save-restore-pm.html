regs_save_restore.S：低功耗路径中的 CPU 状态保存与恢复

这个文件不是普通进程上下文切换，也不是异常返回路径。它是 X2600 进入 standby / mem suspend 前后使用的一段极小汇编桥：`save_goto(func)` 保存当前 C 调用点所需的寄存器和关键 CP0 状态，然后跳到搬运到 TCSM 的睡眠代码；唤醒后，TCSM 恢复代码跳到 `restore_goto`，它恢复保存的状态，并通过原来的 `ra` 回到 `x2600_pm_enter()` 中 `save_goto()` 调用之后。

使用场景说明

当前文件的实际使用点在 `arch/mips/xburst2/soc-x2600/pm.c`。`late_initcall(pm_init)` 注册 `platform_suspend_ops`，其中 `.enter = x2600_pm_enter`；当系统进入 `PM_SUSPEND_STANDBY` 或 `PM_SUSPEND_MEM` 时，`x2600_pm_enter()` 准备 TCSM 代码和参数，最后调用 `save_goto()`。

/* arch/mips/xburst2/soc-x2600/pm.c */
static const struct platform_suspend_ops x2600_pm_ops = {
    .valid      = ingenic_pm_valid,
    .begin      = x2600_pm_begin,
    .enter      = x2600_pm_enter,
    .end        = x2600_pm_end,
};

static int __init pm_init(void)
{
    suspend_set_ops(&x2600_pm_ops);
    return 0;
}

late_initcall(pm_init);
/* arch/mips/xburst2/soc-x2600/pm.c */
int x2600_pm_enter(suspend_state_t state)
{
    sleep_param->uart_base = UART2_IOBASE + 0xa0000000;
    sleep_param->state = state;

    load_func_to_tcsm((unsigned int *)SLEEP_CPU_RESUME_BOOTUP_TEXT,
        (unsigned int *)cpu_resume_bootup, SLEEP_CPU_RESUME_BOOTUP_LEN);
    load_func_to_tcsm((unsigned int *)SLEEP_CPU_RESUME_TEXT,
        (unsigned int *)cpu_resume, SLEEP_CPU_RESUME_LEN);
    load_func_to_tcsm((unsigned int *)SLEEP_CPU_SLEEP_TEXT,
        (unsigned int *)cpu_sleep, SLEEP_CPU_SLEEP_LEN);

    if (state == PM_SUSPEND_STANDBY) {
        soc_pm_idle();
    } else if (state == PM_SUSPEND_MEM) {
        soc_pm_sleep();
    } else {
        printk("WARNING : unsupport pm suspend state\n");
    }

    /* set reset entry */
    *(volatile unsigned int *)0xb2200f00 = SLEEP_CPU_RESUME_BOOTUP_TEXT;

    mb();
    save_goto((unsigned int)SLEEP_CPU_SLEEP_TEXT);
    mb();

    soc_post_wakeup();
    show_wakeup_sources();
    return 0;
}

PM_SUSPEND_MEM深度睡眠

DDR 进入 self refresh，DDR PHY PLL 可被关闭，CPU 执行 `wait`。唤醒后需要从 TCSM 跑早期恢复代码，再回到内核。

PM_SUSPEND_STANDBY浅睡/待机

仍复用同一条保存恢复桥。平台代码根据 `sleep_param->state` 决定 DDR/PLL 操作深度。

TCSM不依赖 DDR

`SLEEP_MEMORY_START=0xb2400000`，sleep/resume 代码被复制到这段片上存储执行，避免 DDR 不可访问时取指失败。

返回 C继续 PM 收尾

`restore_goto` 恢复原 `ra` 后跳回 `save_goto()` 的返回地址，继续执行 `soc_post_wakeup()` 和 wakeup source 打印。

第一性原理概述：这个文件的使用场景是“内核自己把 CPU 送进低功耗，再从硬件唤醒后回到原来的内核调用栈”。它不是给 scheduler 保存 task，也不是给异常入口保存 pt_regs。

执行总览

图 · 保存点和恢复点不是对称函数调用，而是一次跨越低功耗状态的控制流接力。

第一性原理概述：只要 DDR 或 CPU 时钟可能被改变，恢复代码就不能假设正常内核执行环境仍然存在。把一小段可独立运行的代码放进 TCSM，再用保存好的寄存器快照接回内核，是这个文件的核心设计。

CP0 寄存器定义：明确要跨睡眠保存哪些控制状态

/* arch/mips/xburst2/soc-x2600/regs_save_restore.S */
#define CP0_PAGEMASK    $5,0
#define CP0_STATUS      $12,0
#define CP0_INTCTL      $12,1
#define CP0_CAUSE       $13,0
#define CP0_EBASE       $15,1
#define CP0_CONFIG      $16,0
#define CP0_CONFIG1     $16,1
#define CP0_CONFIG2     $16,2
#define CP0_CONFIG3     $16,3
#define CP0_CONFIG4     $16,4
#define CP0_CONFIG5     $16,5
#define CP0_CONFIG7     $16,7
#define CP0_LLADDR      $17,0
#define CP0_WATCHLo     $18,0
#define CP0_WATCHHI     $19,0
#define CP0_ERRCTL      $26,0
#define CP0_CONTEXT     $4,0
#define CP0_KSCRATCH1   $31,2
#define CP0_KSCRATCH2   $31,3
#define CP0_KSCRATCH3   $31,4
#define CP0_KSCRATCH4   $31,5
#define CP0_KSCRATCH5   $31,6
#define CP0_KSCRATCH6   $31,7

这组宏把 CP0 register/selector 写成易读名字。保存范围集中在地址转换、异常/中断控制、CPU 配置、watch/debug、上下文和 kernel scratch。它没有保存所有 CP0 寄存器，例如没有保存 EPC/BadVAddr，因为这条路径不是异常返回，不需要从异常点 `eret`。

寄存器组作用为什么低功耗后要恢复
`PAGEMASK/CONTEXT/KSCRATCH`TLB refill、页表或异常快速路径辅助状态。恢复后内核马上可能访问内存或处理异常，不能让这些状态停留在 boot/resume 临时值。
`STATUS/CAUSE/INTCTL/EBASE`中断使能、异常向量基址、异常原因和中断控制。睡眠唤醒常会改 reset entry 和中断状态，返回内核前必须还原。
`CONFIG1..7`MIPS CPU 配置能力和实现相关控制。低功耗或早期恢复代码可能依赖硬件默认状态，回到内核前要恢复原运行配置。
`LLADDR/WATCH/ERRCTL`LL/SC、watchpoint、错误控制相关状态。这些属于 CPU 执行环境的一部分，不能让 suspend/resume 路径破坏调试或原子操作相关状态。
第一性原理概述：睡眠恢复要保存的不是“所有硬件状态”，而是返回内核后立刻影响取指、访存、中断和异常处理的 CPU 控制状态。这个文件保存的是一组面向“继续执行内核”的最小关键集。

_regs_stack：固定 140 字节保存槽

/* arch/mips/xburst2/soc-x2600/regs_save_restore.S */
    .data
    .global _regs_stack
_regs_stack:
    .align 5
    .space 140,0

`_regs_stack` 是一个全局静态保存区，32 字节对齐，大小 140 字节。后续保存表明它正好存放 35 个 32 位值：前 12 个是通用寄存器，后 23 个是 CP0 寄存器。

图 · `_regs_stack` 是单槽保存区，适合单 CPU suspend 路径，不是可重入的多任务上下文栈。

设计边界：这个保存区只有一份。如果在保存和恢复之间发生重入，或多 CPU 同时使用同一份 `_regs_stack`，会互相覆盖。当前调用场景是平台 suspend enter 路径，符合单次保存、单次恢复的假设。
第一性原理概述：固定保存槽牺牲了通用性，换来极小、确定、无需分配内存的低功耗桥。睡眠边界越靠近硬件，越需要这种没有依赖的静态状态保存。

save_goto：保存当前内核调用现场并跳入 TCSM

/* arch/mips/xburst2/soc-x2600/regs_save_restore.S */
    .text
    .global save_goto
    .align 2
    .ent save_goto,0
    LEAF(save_goto)
save_goto:
    .set    push
    .set    noreorder
    .set    noat

    la  k0, _regs_stack
    sw  s0,0(k0)
    sw  s1,4(k0)
    sw  s2,8(k0)
    sw  s3,12(k0)
    sw  s4,16(k0)
    sw  s5,20(k0)
    sw  s6,24(k0)
    sw  s7,28(k0)
    sw  gp,32(k0)
    sw  sp,36(k0)
    sw  fp,40(k0)
    sw  ra,44(k0)

函数入口先关闭汇编器重排并禁止使用 `$at`。这里保存的是 MIPS ABI 中 C 调用点继续执行所需的 callee-saved 通用寄存器，以及 `gp/sp/fp/ra`。它没有保存 `a0/v0/t*` 等 caller-saved 寄存器，因为从 C 语言调用者视角，调用 `save_goto()` 后这些寄存器本来就不能假设保持不变。

/* arch/mips/xburst2/soc-x2600/regs_save_restore.S */
    mfc0    k1,CP0_PAGEMASK
    sw  k1,48(k0)
    mfc0    k1,CP0_STATUS
    sw  k1,52(k0)
    mfc0    k1,CP0_INTCTL
    sw  k1,56(k0)
    mfc0    k1,CP0_CAUSE
    sw  k1,60(k0)
    mfc0    k1,CP0_EBASE
    sw  k1,64(k0)
    mfc0    k1,CP0_CONFIG
    sw  k1,68(k0)
    ...
    mfc0    k1,CP0_KSCRATCH6
    sw  k1,136(k0)

    jr.hb   a0
    nop

`a0` 是 `save_goto((unsigned int)SLEEP_CPU_SLEEP_TEXT)` 传入的目标地址。保存完成后不是 `jalr`，而是 `jr.hb a0`：它不建立新的返回地址，而是把控制流直接交给 TCSM 中的 `cpu_sleep` 代码。原始返回地址已经在 `_regs_stack + 44` 中保存，稍后由 `restore_goto` 恢复。

`k0/k1` 是内核保留临时寄存器，适合这种极短的底层汇编桥使用。由于当前处在平台 suspend enter 路径，不是在普通可抢占 C 代码中任意调用，所以它可以直接使用这两个寄存器。

第一性原理概述：`save_goto` 做的是“保存后单向跳转”，不是函数调用。它把 C 调用栈冻结在 `_regs_stack` 中，然后让 CPU 去执行可能关闭 DDR/PLL 的睡眠代码。

restore_goto：恢复 CPU 状态并返回原 C 调用点

/* arch/mips/xburst2/soc-x2600/regs_save_restore.S */
    .text
    .global restore_goto
    .align 2
    .ent restore_goto,0
    LEAF(restore_goto)
restore_goto:
    .set    push
    .set    noreorder
    .set    noat

    la  k0, _regs_stack
    lw  s0,0(k0)
    lw  s1,4(k0)
    lw  s2,8(k0)
    lw  s3,12(k0)
    lw  s4,16(k0)
    lw  s5,20(k0)
    lw  s6,24(k0)
    lw  s7,28(k0)
    lw  gp,32(k0)
    lw  sp,36(k0)
    lw  fp,40(k0)
    lw  ra,44(k0)
/* arch/mips/xburst2/soc-x2600/regs_save_restore.S */
    lw  k1,48(k0)
    mtc0    k1,CP0_PAGEMASK
    lw  k1,52(k0)
    mtc0    k1,CP0_STATUS
    lw  k1,56(k0)
    mtc0    k1,CP0_INTCTL
    lw  k1,60(k0)
    mtc0    k1,CP0_CAUSE
    lw  k1,64(k0)
    mtc0    k1,CP0_EBASE
    ...
    lw  k1,136(k0)
    mtc0    k1,CP0_KSCRATCH6

    jr.hb   ra
    nop

`restore_goto` 的恢复顺序与保存顺序一致。关键点是它先恢复 `ra`，最后 `jr.hb ra`。这个 `ra` 不是 `cpu_resume()` 调用 `restore_goto` 的返回地址，而是 `save_goto()` 被 C 代码调用时硬件写入的返回地址，因此跳回的位置是 `x2600_pm_enter()` 中 `save_goto()` 之后。

CP0 用 `mtc0` 写回后立即通过 `jr.hb` 离开，是为了把控制流跳转和 instruction hazard barrier 结合起来。恢复 Status/EBase/Config/KScratch 这类控制状态后，不能让流水线继续在不确定状态下执行大量普通指令。

第一性原理概述：恢复动作的最终目的不是回到 `cpu_resume()`，而是回到睡眠前的内核调用现场。`restore_goto` 用保存的 `ra/sp/gp` 把“唤醒后的早期代码”重新接到“睡眠前的 C 栈”。

x2600_pm_enter 调用链：从 Linux PM core 到汇编桥

`pm.c` 中的 `ingenic_pm_valid()` 只接受 `PM_SUSPEND_ON`、`PM_SUSPEND_STANDBY`、`PM_SUSPEND_MEM`。真正进入低功耗时，`.enter` 回调执行 `x2600_pm_enter()`。这对应官方 PM 文档中的平台/驱动参与系统级 sleep 的模型：Linux PM core 负责阶段化调用，平台代码负责 SoC 特定的寄存器和电源状态。

/* arch/mips/xburst2/soc-x2600/pm.c */
static int ingenic_pm_valid(suspend_state_t state)
{
    switch (state) {
        case PM_SUSPEND_ON:
        case PM_SUSPEND_STANDBY:
        case PM_SUSPEND_MEM:
            return 1;

        default:
            return 0;
    }
}

`save_goto` 前后各有 `mb()`，用于在跨越低功耗边界前后建立内存访问顺序。`save_goto` 返回后执行 `soc_post_wakeup()` 和 `show_wakeup_sources()`，说明这条路径期望恢复后继续走正常内核 C 代码收尾，而不是重启内核。

第一性原理概述：Linux PM core 只能组织系统进入和离开 sleep 的阶段；真正把 CPU、DDR、PLL 送入低功耗并恢复，是 SoC 平台代码负责。`regs_save_restore.S` 正是这个平台层的最低级状态桥。

TCSM 与 DDR/PLL：为什么不能直接在原地执行

/* arch/mips/xburst2/soc-x2600/pm.c */
#define SLEEP_MEMORY_START          0xb2400000
#define SLEEP_MEMORY_END            0xb2401FFC
#define SLEEP_RESUME_SP             SLEEP_MEMORY_END
#define SLEEP_RESUME_BOOTUP_TEXT    SLEEP_MEMORY_START

#define SLEEP_CPU_RESUME_BOOTUP_TEXT SLEEP_RESUME_BOOTUP_TEXT
#define SLEEP_CPU_RESUME_BOOTUP_LEN  256
#define SLEEP_CPU_RESUME_TEXT        (SLEEP_CPU_RESUME_BOOTUP_TEXT + SLEEP_CPU_RESUME_BOOTUP_LEN)
#define SLEEP_CPU_RESUME_LEN         (2596)
#define SLEEP_CPU_SLEEP_TEXT         (SLEEP_CPU_RESUME_TEXT + SLEEP_CPU_RESUME_LEN)
#define SLEEP_CPU_SLEEP_LEN          (3072)
#define NORMAL_PARAM_ADDR            (SLEEP_CPU_SLEEP_TEXT + SLEEP_CPU_SLEEP_LEN)

#define sleep_param ((struct sleep_param *)NORMAL_PARAM_ADDR)
/* arch/mips/xburst2/soc-x2600/pm.c */
static noinline void cpu_sleep(void)
{
    blast_dcache32();
    blast_scache64();
    __sync();
    __fast_iob();

    if ((sleep_param->state == PM_SUSPEND_MEM) ||
        (sleep_param->state == PM_SUSPEND_STANDBY )) {
        unsigned int tmp,pll;
        unsigned int ddrc_ctrl = ddr_readl(DDRC_CTRL);

        sleep_param->pdt = ddrc_ctrl & DDRC_CTRL_PDT_MASK;
        sleep_param->dpd = ddrc_ctrl & DDRC_CTRL_DPD;
        sleep_param->dlp = ddr_readl(DDRC_DLP);
        sleep_param->autorefresh = ddr_readl(DDRC_AUTOSR_EN);

        ddrc_ctrl |= 1 << 5;
        ddr_writel(ddrc_ctrl, DDRC_CTRL);
        while(!(ddr_readl(DDRC_STATUS) & (1<<2)));

        pll = ddr_readl(DDRP_INNOPHY_PLL_CTRL);
        pll |= 1 << 3;
        ddr_writel(pll,DDRP_INNOPHY_PLL_CTRL);
    }

    __asm__ volatile(
        ".set push  \n\t"
        ".set mips32r2  \n\t"
        "wait       \n\t"
        "nop        \n\t"
        "nop        \n\t"
        ".set pop   \n\t"
    );
}

`load_func_to_tcsm()` 把 `cpu_sleep`、`cpu_resume`、`cpu_resume_bootup` 复制到 `0xb2400000` 开始的 SLEEP memory。`cpu_sleep` 中会刷 cache、设置 DDR self refresh、关闭 DDR PHY PLL 并执行 `wait`。如果此时代码仍从 DDR 取指，或者栈/参数仍依赖 DDR 可访问状态，风险就很高。

/* arch/mips/xburst2/soc-x2600/pm.c */
static noinline void cpu_resume(void)
{
    if ((sleep_param->state == PM_SUSPEND_MEM) ||
        (sleep_param->state == PM_SUSPEND_STANDBY)) {
        unsigned int ddrc_ctrl = ddr_readl(DDRC_CTRL);

        /* enable pll */
        tmp = ddr_readl(DDRP_INNOPHY_PLL_CTRL);
        tmp &= ~(1 << 3);
        ddr_writel(tmp,DDRP_INNOPHY_PLL_CTRL);

        while(!(ddr_readl(DDRP_INNOPHY_PLL_LOCK) & (1 << 2))) {
        }

        /* exit sr */
        ddrc_ctrl = ddr_readl(DDRC_CTRL);
        ddrc_ctrl &= ~(1<<5);
        ddr_writel(ddrc_ctrl, DDRC_CTRL);
        while(ddr_readl(DDRC_STATUS) & (1<<2));

        ddrp_auto_calibration();
        ddr_writel(sleep_param->autorefresh, DDRC_AUTOSR_EN);
        ddr_writel(sleep_param->dlp, DDRC_DLP);
    }

    __asm__ volatile(
        ".set push  \n\t"
        ".set mips32r2  \n\t"
        "jr.hb %0   \n\t"
        "nop        \n\t"
        ".set pop   \n\t"
        :
        : "r" (restore_goto)
        :
    );
}

唤醒早期路径先由 reset entry 进入 `cpu_resume_bootup`，设置临时栈并跳 `cpu_resume`；`cpu_resume` 恢复 DDR/PLL 后才 `jr.hb restore_goto`。这就是为什么 `regs_save_restore.S` 保存的是“返回正常内核世界所需的 CPU 状态”，而 TCSM 代码负责“让硬件世界重新可用”。

第一性原理概述：低功耗恢复分两层：先用 TCSM 代码恢复硬件执行环境，再用 `restore_goto` 恢复 CPU/ABI 执行环境。两层都完成后，内核 C 代码才可以继续。

jr.hb 与 hazard：跳转时建立流水线边界

#define ___ehb                              \
    sll $0, $0, 3

/*
 * MIPSR2 defines ehb for hazard avoidance
 */

#define __irq_enable_hazard                 \
    ___ehb

#define __irq_disable_hazard                \
    ___ehb

#define __back_to_back_c0_hazard            \
    ___ehb

#define instruction_hazard()                \
do {                                        \
    unsigned long tmp;                      \
                                            \
    __asm__ __volatile__(                   \
    "   .set    push                \n"     \
    "   .set "MIPS_ISA_LEVEL"      \n"     \
    "   dla %0, 1f                 \n"     \
    "   jr.hb   %0                 \n"     \
    "   .set    pop                \n"     \
    "1:                            \n"     \
    : "=r" (tmp));                          \
} while (0)

本地 `hazards.h` 明确把 MIPSR2/R5/R6 的 hazard avoidance 映射到 `ehb`，并用 `jr.hb` 实现 `instruction_hazard()`。这说明 `regs_save_restore.S` 里的 `jr.hb a0` 和 `jr.hb ra` 不是普通跳转的随意写法，而是在跨越 CP0 状态修改和取指环境变化时建立指令 hazard 边界。

`save_goto` 末尾的 `jr.hb a0` 让 CPU 跳到 TCSM sleep 代码；`restore_goto` 末尾的 `jr.hb ra` 在写回 CP0 后跳回原 C 返回地址。两处都处在执行环境切换边界，因此使用 hazard-barrier 版本的跳转更合适。

第一性原理概述：修改 CP0 或跳到新执行环境后，流水线中可能仍有旧状态下取到或解释的指令。hazard barrier 的作用是让“状态改变”和“后续取指执行”之间形成硬件可见边界。

边界和限制：它刻意不做什么

不是 task switch不保存全部通用寄存器

只保存 ABI callee-saved 和 C 调用点关键寄存器；caller-saved 寄存器不保证跨函数调用保持。

不是 exception return不用 eret

没有 pt_regs，没有 EPC/Status 的异常返回语义；最终是 `jr.hb ra` 回到普通 C 调用返回地址。

不是可重入栈只有一个全局保存槽

`_regs_stack` 是单实例静态区，适合 suspend enter 的单次低功耗桥，不适合并发调用。

不是完整 SoC 状态只管 CPU 侧桥接

DDR、PLL、GPIO、电压、wakeup source 由 `pm.c` 的 TCSM sleep/resume 代码处理。

第一性原理概述：底层低功耗代码越小越可靠。这个文件只负责最难被 C 语言安全表达的部分：在硬件环境变化前后保存和恢复 CPU 执行上下文，并把控制流接回原调用点。

总结

`regs_save_restore.S` 的整体逻辑可以概括为：`x2600_pm_enter()` 将 sleep/resume 代码复制到 TCSM 并设置唤醒入口，随后 `save_goto(SLEEP_CPU_SLEEP_TEXT)` 保存 ABI callee-saved 寄存器和关键 CP0 状态，跳进 TCSM 的 `cpu_sleep`；硬件唤醒后，TCSM 的 `cpu_resume_bootup/cpu_resume` 恢复 DDR、PLL 和早期执行环境，再跳到 `restore_goto`；`restore_goto` 写回 `_regs_stack` 中的 CP0 和通用寄存器，最后 `jr.hb ra` 回到 `save_goto()` 调用之后，继续执行 PM 收尾。

它解决的问题是：当 CPU 睡眠期间正常 DDR/PLL/异常控制状态可能被改变时，如何不用完整内核运行环境，也不用重新启动内核，就把控制流精确接回睡眠前的 C 调用栈。

第一性原理概述：低功耗保存恢复的本质是跨越“不连续的硬件时间”。睡眠前的内核调用栈要在唤醒后继续成立，就必须保存足够的 CPU 和 CP0 状态，并用一条确定的恢复路径把硬件恢复和软件返回连接起来。
