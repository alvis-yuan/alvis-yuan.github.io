📦 概述与历史

SquashFS 是一种面向 Linux 的高度压缩、只读文件系统。它能对文件、Inode 和目录进行压缩，支持最大 1 MiB 的数据块（默认 128 KiB），同时将 Inode 及目录数据紧密打包以最小化磁盘开销。

来自 Linux 内核官方文档的定义：
"Squashfs is intended for general read-only filesystem use, for archival use (i.e. in cases where a .tar.gz file may be used), and in constrained block device/memory systems (e.g. embedded systems) where low overhead is needed."
— Linux Kernel Documentation: Squashfs 4.0 Filesystem

🔤 名称含义

"Squash"（压榨）形象地体现了这种文件系统的核心特征：将文件系统内容极度压缩打包成紧凑的只读镜像，就像将内容"榨干"一样。

💡 章节总结

SquashFS 由 Phillip Lougher 于 2002 年创建，2009 年并入 Linux 2.6.29 主线，是目前 Linux 下最主流的压缩只读文件系统。其核心定位是：在存储/内存受限环境中，以只读方式高效存取大量文件，兼顾归档与系统镜像用途。

第一性原理：文件系统的本质是组织和索引数据。SquashFS 的设计哲学是"写入时一次性最优压缩，读取时按需解压"，以牺牲写能力换取极致的空间效率和挂载后的快速只读访问。

🗂️ 整体布局

根据 Linux 内核官方文档及 SquashFS Binary Format 规范，一个 SquashFS 归档由最多九个部分组成，按字节对齐紧密排列：

来源：Linux Kernel Docs: Squashfs Filesystem Design；SquashFS Binary Format Specification

所有整数以小端序（little-endian）存储，确保跨架构兼容。归档本身必须填充至底层设备块大小（1K 或 4K）的倍数。尽管超级块记录了各区段的精确位置，Linux 内核实现仍要求上述顺序严格不变。

💡 章节总结

SquashFS 的整体布局遵循"头部先行，数据居中，元数据索引居后"的原则。超级块是整个归档的唯一入口点；数据块存放实际文件内容；各类查找表在挂载时被部分加载入内存，实现 O(1) 或 O(log n) 级别的快速检索。这种布局使得顺序构建、随机读取两种操作模式都能高效运作。

🔑 超级块（Superblock）

超级块是 SquashFS 归档的第一个区段，固定 96 字节，包含归档的关键元信息及其他所有区段的位置信息。

字段类型说明
magicu32魔数，固定为 0x73717368（磁盘上为 "hsqs"）
inode_countu32归档中存储的 inode 总数
mod_timeu32归档最后修改时间（Unix 时间戳，无符号，可用到 2106 年）
block_sizeu32数据块大小（字节），必须是 4096 到 1048576 之间的 2 的幂
frag_countu32碎片表中的条目数
compressoru16压缩算法 ID（1=GZIP, 2=LZMA, 3=LZO, 4=XZ, 5=LZ4, 6=ZSTD）
block_logu16块大小的 log2 值，与 block_size 不符则视为损坏
flagsu16标志位（见下表），控制压缩选项、导出表等
id_countu16UID/GID 查找表条目数
version_majoru16格式主版本，必须为 4
version_minoru16格式次版本，必须为 0
root_inodeu64根目录 inode 的引用（48 位块地址 + 16 位偏移）
bytes_usedu64归档实际使用字节数（可小于文件大小，因填充对齐）

来源：SquashFS Binary Format: The Superblock

💡 章节总结

超级块是整个文件系统的"目录页"，96 字节固定大小使其解析极为高效。魔数校验、版本检查和 block_size/block_log 双重冗余验证共同保障归档完整性。所有区段地址均存于此，实现了文件系统元数据的中心化管理。

💾 数据块与碎片块

文件内容被分割为固定大小的数据块（Datablocks），每个块独立压缩后顺序写入归档。若压缩后块大小反而超过原始大小，则直接存储未压缩数据，保证压缩不会带来额外膨胀。

碎片块（Fragment Blocks）

文件的"尾端"（不足一个完整块大小的部分）可以与其他文件的尾端打包进一个碎片块（Fragment Block），大小不超过 block_size。小于一个块的文件同样按此处理。这种机制显著减少了对小文件的空间浪费。

来源：SquashFS Binary Format: Packing File Data；Linux Kernel Docs: File Data

大文件加速机制：对于 256 MiB 以上的"大文件"，SquashFS 实现了一个块索引缓存（index cache），缓存从块索引到磁盘位置的映射，最多可处理 1.75 TiB 的文件（使用 128 KiB 块时）。缓存默认仅使用 16 KiB 内存，极为节省。

💡 章节总结

数据块机制是 SquashFS 支持"随机访问"的关键：每块独立压缩，读取任意文件只需解压对应块，无需解压整个归档。碎片块机制则解决了大量小文件导致的空间浪费问题。块大小的可调性（4K–1M）允许在压缩率与随机访问性能之间灵活权衡：块越大压缩率越高，但随机访问需解压更多数据。

🔖 Inode 表

元数据（inode 和目录）被压缩进 8 KiB 的元数据块。每个压缩块以一个 2 字节的长度头为前缀，最高位置 1 表示未压缩存储。Inode 紧密打包，不按块边界对齐，因此一个 inode 可以跨越两个压缩块。

Inode 由 48 位编码的引用地址标识：高 48 位为包含该 inode 的压缩元数据块的起始位置，低 16 位为块内字节偏移（<block, offset> 对）。这是 SquashFS 高效寻址的核心机制。

Inode 类型

为了最大化压缩效率，SquashFS 针对不同文件类型定义了不同的 inode 结构，每种类型的长度和内容各异：

📄
普通文件 Inode
包含数据块列表、碎片索引、文件大小。分基础型（常见文件）和扩展型（含稀疏文件支持）两种。
📁
目录 Inode
含目录表的块地址与偏移、硬链接数、父目录 inode 编号。扩展型含目录索引。
🔗
符号链接 Inode
存储目标路径（非 null 结尾字节串），硬链接计数器（符号链接也支持硬链接）。
💽
设备文件 Inode
块设备/字符设备，含设备号（major/minor）。IPC（FIFO、Socket）类似，仅存硬链接数。

UID/GID 压缩策略

为节省 inode 空间，UID 和 GID 并不直接存入 inode，而是存储为一个 16 位的 ID 查找表索引。查找表集中存储所有唯一的 32 位 UID/GID 值，每个 inode 仅引用其索引。这将每个 inode 中 UID/GID 的存储从 8 字节降至 4 字节。

来源：Linux Kernel Docs: Inodes；SquashFS Binary Format: Inodes

💡 章节总结

Inode 设计遵循"按类型特化，按需打包"原则：不同文件类型用不同 inode 结构，避免通用结构导致的浪费；Inode 不按块对齐、可跨块存储，实现最紧密打包。平均每个 inode 仅 8 字节，是 ext4 等通用文件系统的几分之一。UID/GID 去重进一步节省了空间。

📂 目录表

目录条目按字典序（ASCIIbetically）排序，打包进压缩的 8 KiB 元数据块，存储于目录表区段。每个目录的访问通过 <block 地址, 偏移> 对定位。

两级列表结构

目录并非简单的文件名列表，而采用一种精巧的两级结构：一个目录头（Directory Header）存储共享的 inode 块起始地址，后跟多个目录条目（Directory Entry），每个条目仅存储相对该共享块地址的偏移量和 inode 编号差值。

目录索引（Directory Index）

对于含有大量条目的目录，扩展目录 inode 可以携带一个目录索引：每个元数据块存储一条索引记录（块地址 + 该块第一个条目的文件名），查找时先线性扫描索引找到目标所在块，再只解压那一个元数据块即可完成查找。

核心设计思想（来自内核文档）："The general idea of the index is to ensure only one metadata block needs to be decompressed to do a lookup irrespective of the length of the directory."
无论目录有多大，查找一个文件名最多只需解压一个元数据块。

来源：Linux Kernel Docs: Directories；SquashFS Binary Format: Directory Table

💡 章节总结

目录表的两级结构充分利用了"同目录下 inode 往往在同一元数据块"这一局部性特征，通过共享块地址大幅压缩目录条目体积。字典序排序配合目录索引，将任意大小目录的文件名查找代价限定为"最多一次元数据块读取"，兼顾紧凑性与查找性能。

📋 查找表体系

SquashFS 设计了统一的查找表（Lookup Table）存储模式，适用于碎片表、Export 表、UID/GID 表和 Xattr 表：

🧩
碎片表（Fragment Table）
记录每个碎片块的磁盘偏移（u64）、磁盘大小（u32），每块可存 512 条记录（8192/16字节）。
📤
Export 表
inode 编号到 inode 引用的平坦数组（64位），支持 NFS 文件句柄到磁盘 inode 的重建。每块可存 1024 条。
👤
UID/GID 查找表（ID Table）
存储所有唯一 32 位 UID/GID，每块可存 2048 个 ID（8192/4字节）。inode 中存 16 位索引。
🏷️
Xattr 表
键值对打包入压缩元数据块，支持值去重（out-of-line 引用），inode 存 32 位 xattr_id 索引，每块 512 条目。

来源：SquashFS Binary Format: Storing Lookup Tables；Linux Kernel Docs: Fragment/Export/UID/Xattr Tables

💡 章节总结

查找表体系是 SquashFS 在"压缩存储"与"高效随机访问"之间求得平衡的核心设计：两级索引（位置列表 + 压缩元数据块）让每次查找的 I/O 代价恒为常数，小型位置列表在挂载时整体载入内存，进一步消除了磁盘寻道开销。这是以少量（通常仅数KB）内存换取常数时间随机查找的经典工程权衡。

🗜️ 压缩机制

SquashFS 支持六种压缩算法，整个归档只能使用一种（由超级块中的 compressor 字段指定），数据块和元数据块共用同一算法。

算法ID特点适用场景
gzip (zlib)1 经典，压缩率适中，兼容性最好（注：使用原始 zlib 流，无 gzip 文件头） 通用场景，工具链广泛支持
LZMA2 高压缩率，已废弃 已由 XZ 取代
LZO3 极快解压，压缩率较低 嵌入式设备、CPU 弱的场景
XZ (LZMA2)4 最高压缩率，解压慢，内存消耗大；必须使用 CRC32 校验 空间极度受限、网络分发
LZ45 最快解压速度，压缩率低，适合内存带宽受限场景 高速读取优先、RAM 容量小
Zstd6 现代算法，在压缩率和速度之间取得良好平衡，推荐用于新项目 Snap、容器、现代 Linux 发行版

关键压缩行为

压缩失败回退：若压缩后大小超过原始数据，SquashFS 自动以未压缩方式存储该块，确保块的存储大小绝不超过其原始输入大小。这一机制对已压缩的媒体文件（JPEG、MP4 等）尤为重要。

来源：SquashFS Binary Format: Supported Compressors；Linux Kernel Docs: Squashfs 4.0

💡 章节总结

SquashFS 的压缩机制遵循"写时最优、读时按需"原则：构建时对每个块独立尝试压缩，失败则存原文；运行时仅按需解压被访问的块。算法选择上，Zstd 是现代场景的最佳默认选择；XZ 适合极致存储优化；LZ4/LZO 适合 CPU 弱或解压延迟敏感的嵌入式场景。

📐 元数据压缩

元数据（inode、目录条目等）以连续流的形式切成 8 KiB 块后独立压缩，并在每个压缩块前加 16 位头部（包含大小和压缩标志）。条目可以跨越块边界，此时需要读取并解压两个相邻块。

元数据压缩的高效性（来自 squashfs-tools 维护者回复）：元数据具有极高的冗余性和重复性，压缩通常可以将信息密度提升 4 到 8 倍——即一次磁盘 I/O（读取一个元数据块）可获取原本需要 4~8 次 I/O 才能取得的信息量。元数据压缩的解压开销相对于 I/O 节省而言微乎其微。

💡 章节总结

元数据压缩以固定 8 KiB 输入块（独立于数据块大小）统一处理 inode、目录等高重复性数据，通常实现 4–8 倍信息密度提升。这是 SquashFS 能够在 inode 平均仅 8 字节的同时维持完整 POSIX 语义的根本原因。元数据块的头部双编码（大小 + 压缩标志），也是流式解析时的自描述容错设计。

⚡ 内部缓存机制

压缩块的重复解压会造成性能损耗。SquashFS 为此实现了两层缓存策略：

🗃️
页面缓存（Page Cache）
文件数据块直接解压后进入 Linux 内核页面缓存，由 VFS 统一管理，与其他文件系统共享缓存基础设施，无需 SquashFS 自行维护。
🔬
内部元数据/碎片缓存
元数据块和碎片块不经过页面缓存，SquashFS 维护专用的小型缓存（两个），用于临时缓存最近读取的元数据和碎片块，利用局部性原理避免重复读取和解压。
📌
块索引缓存（Index Cache）
针对 256 MiB 以上的大文件，缓存块索引到磁盘位置的映射，默认 16 KiB，最多支持 1.75 TiB 文件（8个槽位）。
🧠
挂载时内存缓存
碎片表、Export 表和 UID/GID 表的索引列表在挂载时整体加载入内存，消除这些小型查找表的磁盘 I/O 延迟。

多线程解压（Mount 选项）

模式说明
threads=single默认单线程，内存和 CPU 占用最低，但多核并行 I/O 时性能受限
threads=multi每核最多 2 个并行解压器，按需动态分配，适合并行 I/O 负载
threads=percpu每核 1 个解压器，使用 percpu 变量确保解压负载在核心间均衡
threads=N指定线程数，上限为 num_online_cpus() × 2

来源：Linux Kernel Docs: Squashfs Internal Cache；Linux Kernel Docs: Mount Options

💡 章节总结

SquashFS 的缓存策略遵循分层原则：热数据（文件内容）交由 VFS 页面缓存统一管理；冷数据（元数据、碎片）由专用内部缓存处理，利用局部性降低重复 I/O；大文件块索引以极低内存（16 KiB）换取对 TiB 级别文件的快速定位。多线程解压选项允许在 CPU 资源充足时突破单核解压瓶颈。

♻️ 去重与稀疏文件

数据去重（Deduplication）

mksquashfs 在构建时会检测重复的数据块。若两个文件含有相同内容的块，这些块在归档中只存储一份，其他文件的 inode 引用同一磁盘位置。超级块标志位 0x0040 标记数据已去重。

利用追加模式的去重能力，SquashFS 可实现简单的版本化归档：将 linux-2.4.19 源代码创建成镜像，再追加 linux-2.4.20 源代码时，只有真正变更的文件才会占用额外空间，未变更文件被识别为重复块而共享存储。

稀疏文件（Sparse Files）

SquashFS 支持稀疏文件——文件中的"空洞"（全零区域）不会实际写入磁盘。与 CramFS 相比，这是 SquashFS 的重要改进之一（见下方特性对比表）。

Xattr 值去重

在 Xattr 表中，若多个 inode 拥有相同的 Xattr 值，该值只存储一次，其他引用通过 out-of-line 指针共享，进一步节省空间。

💡 章节总结

去重与稀疏文件是 SquashFS 在算法压缩之外的两个正交节省空间的机制。数据块去重消除文件间的冗余（对软件仓库、OS 镜像等效果显著）；稀疏文件支持消除文件内的冗余（对数据库文件、虚拟磁盘镜像等效果显著）；Xattr 值去重消除元数据层面的冗余。三者相互独立、叠加增益，使得实际压缩效果往往超出单纯算法压缩所能达到的水平。

✅ 核心优势

🗜️
极高压缩效率
同等数据与 ext3 文件系统相比，大小约为 1/3。支持 gzip、XZ、LZ4、LZO、Zstd 多种算法，适应不同压缩率与速度需求。
🔀
随机访问支持
数据块独立压缩，无需解压整个归档即可访问任意文件，与 .tar.gz 等纯流式归档有本质区别。
🐧
内核原生支持
自 Linux 2.6.29（2009年）起并入主线，无需 FUSE，在内核态运行，syscall 开销小，可用于 initramfs 早期启动阶段。
📏
支持超大规模
最大文件系统 2^64 字节（约 16 EiB），最大单文件约 1.75 TiB，文件/目录数量不限，远超 CramFS（256 MiB / 16 MiB）。
🔒
只读安全保证
只读属于数据格式本身特性，而非挂载选项——即使获得 root，也无法 remount 为读写，为系统提供强不可篡改性保证。
🧮
完整 POSIX 语义
支持真实 inode 编号、硬链接、32位 UID/GID、文件创建时间、稀疏文件、扩展属性（Xattr）、NFS 导出，优于 CramFS 的简陋实现。
🏎️
读取性能优于直觉
压缩减少了实际 I/O 量，在慢速存储（NOR Flash、机械硬盘、网络）上，解压 CPU 开销往往小于节省的 I/O 等待，总体读速反而更快。
💡
极低内存开销
设计理念是最小化内存使用：块索引缓存默认 16 KiB，位置列表通常仅数 KB，元数据/碎片缓存极小，适合内存受限的嵌入式设备。

💡 章节总结

SquashFS 的优势可以从第一性原理归结为：它将"写入"操作的所有代价（压缩计算、去重检测、布局优化）集中在镜像构建阶段，而"读取"路径则因数据已经最优化存储而变得极为高效。内核原生实现消除了 FUSE 的用户态切换开销，只读格式约束消除了写入同步、日志、崩溃恢复等复杂性，使实现保持简洁。

⚠️ 局限性与不足

❌ 核心限制
  • 格式层面只读，无法运行时写入，remount 为读写不可能
  • 修改任何文件须重建整个镜像，成本极高
  • 不支持 ACL（访问控制列表），内核文档列为 TODO
  • 单一压缩算法约束：数据块和元数据必须用同一算法
  • 单线程解压默认配置在多核并行 I/O 时性能受限
  • 大块大小（如 1 MiB）时随机访问需解压更多无关数据
⚡ 性能权衡
  • 每次读取需解压，CPU 存在一定开销（XZ 尤甚）
  • 首次访问延迟高于直接读取（特别是 XZ 压缩时）
  • 压缩文件（JPEG/MP4）几乎无压缩收益，徒增 CPU 开销
  • 不适合写入密集型工作负载（日志、数据库）
  • 仅限 Linux 内核原生支持，其他平台需 FUSE 或专用工具

只读的根本原因（来自 Stack Overflow 的技术解释）："Because the entire filesystem is compressed and zipped, before it is written to the storage media. And so making any changes to any files, it will have to be zipped and rewritten to the entire partition again — which is a great waste of effort if done repeatedly."
压缩存储的本质决定了就地修改的不可行性：任何修改都破坏了原有压缩上下文，必须重压整个受影响区域。
— Stack Overflow: How to make squash file system read-write

方面SquashFS 局限替代/解决方案

💡 章节总结

SquashFS 的所有局限几乎都可以归因于其核心设计决策：写时全局压缩优化。这个决策使读路径极度高效，却使写路径完全不可能。这是一个清醒的工程取舍，而非设计缺陷。在实践中，通过 OverlayFS 叠加可写层的方式，已经优雅地解决了绝大多数"既要压缩又要可写"的需求（Live CD、容器、Snap 均采用此方案）。

🎯 典型应用场景

🖥️ Live CD / Live USB

SquashFS 被 Ubuntu、Debian Live、Fedora、Arch Linux、Kali Linux、Linux Mint、Gentoo、openSUSE 等几乎所有主流 Linux 发行版用于 Live 媒体的根文件系统存储。整个操作系统压缩为一个 .squashfs 镜像，启动时挂载该镜像作为只读底层，再通过 OverlayFS / aufs / UnionFS 叠加一个可写的 tmpfs 层，提供完整的读写体验。

UbuntuDebian LiveFedora Kali LinuxArch LinuxPuppy Linux

📱 嵌入式设备 / 路由器固件

OpenWrt、DD-WRT 等路由器固件广泛使用 SquashFS 作为根文件系统，存储于 NOR Flash（因为 SquashFS 不感知坏块，NOR Flash 的高可靠性是前提；在 NAND Flash 上需配合 UBI 层）。Chromecast、部分 Android 版本（Android 7.0 Nougat 起的系统分区）也采用 SquashFS。

OpenWrtDD-WRTChromecast Android Nougat+

📦 应用打包格式

AppImage 项目使用 SquashFS 作为便携式 Linux 应用的容器格式，整个应用及其依赖被压缩为一个单文件，运行时自动挂载。Snap（Canonical 的软件包系统）同样使用 SquashFS 作为其包格式（.snap 文件本质是 SquashFS 镜像），提供原子更新和强隔离。

AppImageSnapTiny Core Linux 扩展

🐳 容器与不可变基础设施

Talos Linux 使用 SquashFS 作为整个操作系统的根文件系统，容器节点 OS 镜像以 SquashFS 形式分发，配合安全启动（Secure Boot）和镜像完整性校验（checksum 验证），提供强安全保证。SquashFS 不可变的格式特性使得镜像完整性验证极为简单——对单个文件算哈希即可。

Talos LinuxNixOS

🗄️ 归档与软件分发

SquashFS 可作为 .tar.gz 的替代方案用于归档——在提供相近压缩率的同时，挂载后可像普通文件系统一样随机访问，无需先完整解压。这对大型软件包（Linux 内核源码、大型 SDK 等）尤为便利。Linux Terminal Server Project（LTSP）和 Splashtop 也使用 SquashFS 进行网络根文件系统分发。

💡 章节总结

SquashFS 的适用场景有一个共同特征：写入操作极少或不存在，而读取操作频繁。从 Live CD 到路由器固件、从 Snap 包到容器节点 OS，SquashFS 都扮演着"一次写入，多次读取"的角色。与 OverlayFS 的组合则优雅地解决了"需要写入"的少数场景，使 SquashFS 能够覆盖从嵌入式到云原生的广泛应用。

⚖️ 与其他方案对比

方案 类型 压缩率 随机读 写入 内核原生 适用场景
SquashFS 压缩只读 FS 高 良好 ✘ ✔ (Linux) Live CD、嵌入式、容器
CramFS 压缩只读 FS 中 差 ✘ ✔ 已废弃，被 SquashFS 取代
EROFS 压缩只读 FS 高 优秀 ✘ ✔ Android 系统分区（新世代）
DwarFS 压缩只读 FS (FUSE) 极高 优秀 ✘ ✘（FUSE） 归档数据集、桌面/服务器
ext4 通用读写 FS 无 优秀 ✔ ✔ 通用场景，日志、数据库
UBIFS Flash 读写 FS 有（LZO/zlib） 良好 ✔ ✔ NAND Flash 嵌入式设备
.tar.gz 流式归档 高 ✘（需全解压） ✘ ✘ 文件备份传输

SquashFS vs DwarFS（来自实际用户讨论）：SquashFS 适合运行时加载压缩应用（AppImage、系统库等），因为其内核态实现 syscall 开销低，应用启动快；DwarFS 适合归档大型数据集（ML 数据、备份），提供更高压缩率和值去重，但 FUSE 层带来的 syscall 延迟不适合频繁小文件读取场景。
— Porteus Forum: DwarFS vs SquashFS

💡 章节总结

在压缩只读文件系统赛道上，SquashFS 的核心竞争力在于：内核原生 + 成熟稳定 + 生态广泛。EROFS 代表下一代（主攻 Android），DwarFS 代表用户态高压缩方向，但都无法完全取代 SquashFS 在引导环境（initramfs）、路由器固件等对内核原生支持有强依赖的场景中的地位。选择 SquashFS 通常是在"需要在 Linux 内核环境下高效挂载压缩只读文件系统"时的最优解。

🛠️ 工具链

🔨
mksquashfs
创建 SquashFS 镜像。支持指定压缩算法、块大小、去重、伪文件定义、追加模式等。squashfs-tools 4.7.4 (2025-11)。
📂
unsquashfs
从 SquashFS 镜像中提取文件，或列出内容。支持正则过滤、并发解压。
🔌
squashfuse
用户态 FUSE 驱动，可在无内核模块的环境（如 macOS、非特权容器）中挂载 SquashFS 镜像。
📦
7-Zip
7-Zip 支持读取（解压）SquashFS 归档，已移植至 Windows NT – Windows 8.1。

常用命令速查

# 创建镜像（默认 gzip）
mksquashfs /path/to/dir image.sqsh

# 使用 Zstd 压缩，块大小 256K
mksquashfs /path/to/dir image.sqsh -comp zstd -b 256K

# 使用 XZ 最高压缩
mksquashfs /path/to/dir image.sqsh -comp xz

# 挂载镜像（需 squashfs 内核模块）
sudo mount -t squashfs -o loop image.sqsh /mnt/target

# 挂载并叠加可写层（OverlayFS）
sudo mount -t overlay overlay \
  -o lowerdir=/mnt/target,upperdir=/tmp/upper,workdir=/tmp/work \
  /tmp/rw_view

# 解压镜像内容
unsquashfs image.sqsh

# 列出内容不解压
unsquashfs -l image.sqsh
💡 章节总结

SquashFS 工具链简洁而完整：mksquashfs 负责构建（一次性写入），unsquashfs 负责提取，mount 命令配合内核模块负责透明挂载。squashfuse 则将适用范围延伸至非 Linux 环境。配合 OverlayFS，只需五行命令即可构建一个"SquashFS 只读底层 + 可写覆盖层"的完整可写环境。
