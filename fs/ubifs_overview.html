UBIFS 深度解析

UBI File System — 专为裸 NAND/NOR Flash 设计的下一代日志型文件系统，涵盖结构、原理、优劣势、适用场景与工具链

内核原生支持 掉电安全 B+树索引 写回支持 透明压缩 对数扩展性

1. 什么是 UBIFS

UBIFS（UBI File System）是由诺基亚工程师与匈牙利塞格德大学联合开发的闪存文件系统，在某种意义上可视为 JFFS2 的下一代继任者。UBIFS 专门设计用于裸闪存（Raw Flash），与运行在块设备（eMMC、SD 卡、SSD、USB 闪存）之上的传统文件系统（Ext4、XFS 等）有着根本性的区别。

⚠️

重要警告：UBIFS 不能运行在块设备上（如硬盘、MMC/SD 卡、SSD）。块设备内部通过硬件 FTL（Flash Translation Layer）向外暴露块接口，已隐藏了裸 Flash 的细节。UBIFS 只能运行在 UBI 卷之上，而 UBI 运行在 MTD 设备上，MTD 直接驱动原始 Flash 芯片。

📎 原文：linux-mtd.infradead.org/doc/ubifs.html

MTD 设备 vs. 块设备

理解 UBIFS 必须先理解 MTD 设备的独特操作模型，内核官方文档明确列出以下关键差异：

特性MTD 设备（裸 Flash）块设备（HDD/eMMC/SSD）
基本单元擦除块（Eraseblock），典型约 128 KiB块（Block），典型 512 字节
操作读（偏移）/ 写（偏移）/ 擦除（整块）读块 / 写块
覆盖写必须先擦除整块才能重写可直接覆盖写
磨损擦除次数有限（SLC NAND: 100K~1G 次，MLC: 1K~10K 次）无磨损限制（HW 透明处理）
坏块NAND 闪存存在坏块，软件必须处理现代 LBA 磁盘自动处理坏块
📎 原文：kernel.org/doc/Documentation/filesystems/ubifs.txt
📌 章节总结 · 第一性原理

闪存的物理约束（必须整块擦除、有磨损次数、可能出现坏块）决定了传统块设备文件系统无法直接适用。UBIFS 的根本设计动机是：在接受这些物理约束的前提下，提供类 POSIX 的文件系统语义，同时兼顾磨损均衡、掉电安全和性能。

2. 软件层次结构

UBIFS 并非直接操作硬件，而是构建在完整的软件栈之上。内核文档描述了三个核心子系统：

🗂️ UBIFS 文件系统层 — 实现文件语义、B+树索引、日志、压缩
📦 UBI 层 — 磨损均衡、卷管理、坏块屏蔽
⚡ MTD 子系统 — 统一 Flash 接口 (/dev/mtd0 …)
🔩 硬件驱动 — NAND 驱动 / NOR 驱动
图1：Linux 内核中处理原始 Flash 的子系统层次（来源：kernel.org ubifs-authentication 文档）
📎 原文：kernel.org/doc/html/v5.8/filesystems/ubifs-authentication.html

UBI 层为 UBIFS 提供了以下保证，使文件系统层无需处理底层 Flash 的原始复杂性：

🔄 磨损均衡

UBI 在物理擦除块（PEB）之间均匀分布写入，延长整体寿命

🚫 坏块屏蔽

UBI 透明地将逻辑擦除块（LEB）映射到正常的物理擦除块，坏块对 UBIFS 不可见

📚 卷管理

UBI 支持在一个 MTD 分区上创建多个命名的逻辑卷，类似于分区

🔍 位翻转清理

UBI 检测并纠正可修正的位翻转（Bit Flip），防止累积为不可修正错误

📌 章节总结

分层设计是 UBIFS 的核心工程哲学：MTD 负责硬件抽象，UBI 负责可靠性（磨损均衡、坏块），UBIFS 专注于文件系统语义。每一层职责清晰，UBIFS 因此得以在不关心 Flash 物理细节的情况下提供高性能文件系统服务。

3. 节点类型（On-Flash Nodes）

UBIFS 在闪存上存储的基本实体称为节点（Node）。几乎所有节点类型共享一个公共头部（struct ubifs_ch），包含节点类型、长度、序列号等基础信息（见内核源码 fs/ubifs/ubifs-media.h）。

📎 原文：kernel.org/doc/Documentation/filesystems/ubifs-authentication.rst
节点类型内核结构体用途
超级块节点 (Superblock)ubifs_sb_node存储文件系统全局参数（LEB 大小、日志大小、压缩类型等），位于 LEB 0，很少变化
主节点 (Master Node)ubifs_mst_node指向 B+树根节点的最新位置，存于 LEB 1（LEB 2 备份），每次 commit 更新
索引节点 (Index Node)ubifs_idx_nodeB+树的中间节点，存储子节点的键和物理位置
数据节点 (Data Node)ubifs_data_node存储文件内容的压缩数据块（最大 4096 字节数据）
Inode 节点 (Inode Node)ubifs_ino_node对应 VFS inode，存储文件元数据（权限、时间戳、大小等）
目录项节点 (Dentry Node)ubifs_dent_node目录条目，关联文件名与 inode 编号
引用节点 (Reference Node)ubifs_ref_node日志区内使用，指向日志条目（bud）所在的物理位置

填充节点 (Padding Node)—填充 LEB 末尾无法利用的空间，使数据对齐
图2：UBIFS 节点通用布局及典型节点示例
📌 章节总结

节点是 UBIFS 闪存存储的原子单元。所有节点携带 CRC 校验，保证数据完整性。节点不跨越 LEB 边界，这一设计使垃圾回收成为以 LEB 为粒度的简单操作。

4. LEB 空间布局

UBIFS 将 UBI 卷中的逻辑擦除块（LEB）按照固定用途进行划分。以下是一个典型 UBIFS 卷的 LEB 区域分配（摘自 mkfs.ubifs 输出）：

图3：UBIFS 卷 LEB 区域分配示意图
ℹ️

UBIFS 节点不跨越 LEB 边界。这一约束简化了垃圾回收（GC 以整个 LEB 为单位操作）和崩溃恢复。这也意味着 LEB 末尾可能有少量空间浪费（padding）。

📎 原文：linux-mtd.infradead.org/faq/ubifs.html
📌 章节总结

UBIFS 将 UBI 卷的 LEB 按功能分区：超级块、双主节点（互备）、日志区、LPT 区、孤儿区、主数据区。这种固定职责划分保证了每个区域语义清晰，挂载时只需读 LEB 1/2（主节点）即可定位整个文件系统状态，避免了全盘扫描。

5. B+ 树索引（Wandering Tree）

UBIFS 在闪存上维护一棵 B+ 树作为文件系统索引（Index）。叶节点存储实际的文件系统数据（数据节点、inode 节点等），中间节点存储路由信息。

为了避免每次修改都重写整棵 B+ 树，UBIFS 将其实现为漂移树（Wandering Tree）：只有被修改的节点才会被重新写入，旧版本节点被标记为过时（Obsolete）但不立即擦除。结果是索引并不固定在 Flash 的某一位置，而是随着写入不断"漂移"。

📎 原文：kernel.org/doc/Documentation/filesystems/ubifs-authentication.rst
图4：UBIFS B+ 漂移树示意图，修改的节点写入新位置，旧节点标记过时

主节点（Master Node）

为了找到最新版本的 B+ 树根节点，UBIFS 在 LEB 1 中存储一个特殊的主节点（Master Node），它始终指向最新的 B+ 树根。为了可恢复性，主节点同时备份于 LEB 2。挂载 UBIFS 只需读取 LEB 1 和 LEB 2 即可获取主节点，再从主节点找到最新的索引根——无需全盘扫描。

📎 原文：kernel.org/doc/Documentation/filesystems/ubifs-authentication.rst
📌 章节总结 · 第一性原理

闪存写入只能追加（Out-of-Place Write），不能原地修改。B+ 漂移树正是对这一约束的直接映射：每次修改写到新位置，旧位置由垃圾回收异步清理。主节点的双备份机制保证了即使在修改索引途中掉电，也总能找到一致的文件系统状态。

6. 日志系统（Journal）

为避免频繁更新闪存上的 B+ 树（每次 commit 的开销），UBIFS 使用日志（Journal）来缓冲文件系统变更。日志记录 commit 之间的所有修改，挂载时回放日志即可重建最新状态。

ℹ️

UBIFS 日志是"漂移"的，没有固定位置。提交后 UBIFS 选择新的 LEB 作为新日志，旧日志 LEB 被回收。这与 ext3 固定位置的日志完全不同。

📎 原文：linux-mtd.infradead.org/faq/ubifs.html（"I need more space" 章节）

日志区组成

日志区（Log Area）由 mkfs.ubifs 创建时配置，只包含两种节点：

📌 引用节点（Reference Node）

每次日志更新时写入，指向实际变更数据（Bud 节点）所在的 Flash 位置

🏁 提交开始节点（Commit Start Node）

每次索引 commit 时写入，挂载时从最新的 Commit Start Node 开始回放，忽略之前的引用节点

图5：UBIFS 日志区结构与挂载时回放流程

后台线程（ubifs_bgt）

UBIFS 为每个挂载的文件系统创建一个后台线程，命名为 ubifs_bgtX_Y（X=UBI设备号，Y=卷号）。其主要职责包括：

  • 日志使用率达约 80% 时发起后台提交（Background Commit），确保提交完成前日志不会满
  • 定时刷新写缓冲区（Write Buffer）
📎 原文：linux-mtd.infradead.org/faq/ubifs.html（"ubifs_bgt" 章节）
📌 章节总结

日志是 UBIFS 掉电安全的核心机制，也是快速挂载的关键：挂载时只需扫描固定大小的日志（不随文件系统容量增长），而非全盘扫描。日志大小是在 mkfs.ubifs 创建时配置的，并存储在超级块节点中。

7. LPT — 逻辑擦除块属性树

LPT（LEB Properties Tree）是一棵存储在闪存上的 B+ 树，记录每个 UBI 逻辑擦除块（LEB）的空间使用信息（空闲空间、脏空间大小）。这是 UBIFS 相比 JFFS2 的关键改进之一。

特性JFFS2UBIFS (LPT)
空间统计存储位置 不存储，每次挂载重新扫描全盘计算 持久化存储在 LPT B+树中
挂载后全盘 CRC 检查 需要，挂载后后台进行，消耗大量 CPU 不需要，直接读 LPT 即可
内存占用 与 Flash 大小线性相关 LPT 可收缩（Shrinkable）
📎 原文：linux-mtd.infradead.org/doc/ubifs.html（Scalability 章节对比表）

LPT 的根节点地址存储在主节点（Master Node）中，每次 commit 时 LPT 被原子性地整体写入，无需对单个 LPT 节点做单独认证。

📌 章节总结

LPT 的本质是将"空间记账"持久化到 Flash。由于空间信息总是最新且随时可读，UBIFS 既不需要全盘扫描（像 JFFS2 挂载时那样），也不需要在内存中为每个节点保留数据结构。这是实现"挂载时间不依赖文件系统内容"的根本保障。

8. TNC — 树节点缓存

TNC（Tree Node Cache）是闪存上 B+ 索引树的内存表示。它包含每个节点的运行时额外属性（不持久化），最重要的是一个脏标志（Dirty Flag）——标记需要在下次 commit 时写回闪存的节点。

📝 写回缓存

TNC 充当写回缓存，所有对 B+ 树的修改都经过 TNC。这使 UBIFS 支持写回（Write-Back）模式，大幅提升写入吞吐量

💾 可收缩

TNC 不需要将完整的 B+ 树镜像到内存中，按需从 Flash 读取部分节点，并可在内存不足时被内核收缩（Shrink）

Commit 操作是将 TNC 中所有脏节点写回闪存、更新 B+ 树的过程。触发 commit 的条件包括：fsync(2)、日志将满（约80%时由后台线程触发）、sync() 系统调用等。

📎 原文：kernel.org/doc/Documentation/filesystems/ubifs-authentication.rst（TNC 章节）
📌 章节总结

TNC 是内存与闪存之间的桥梁。它不仅提升了读性能（缓存热索引节点），更通过延迟写（写回）减少了闪存擦写次数，延长设备寿命。这是 UBIFS 在写性能上显著优于 JFFS2（写穿）的直接原因。

9. 写回机制（Write-Back）

与 JFFS2 的写穿（Write-Through）不同，UBIFS 支持写回（Write-Back）。数据先写入页缓存（Page Cache），由内核 pdflush/writeback 线程异步刷写到 Flash。这极大地提升了写入吞吐量。

📎 原文：linux-mtd.infradead.org/doc/ubifs.html（Write-back support 章节）

写缓冲区（Write Buffer）

UBIFS 维护一个内存中的写缓冲区（Write Buffer），对应 Flash 的最小 I/O 单元大小（通常为 2KiB NAND 页）。写缓冲区使 UBIFS 可以将小写操作合并，避免频繁的 Flash 写入操作。

写缓冲区由后台线程定时刷新（超时后强制写回），也可通过 fsync()/sync() 显式触发。

⚠️

写回模式意味着：文件创建或截断后若发生掉电，文件可能丢失或变为零字节（创建是同步的，数据写入是异步的）。应用程序应使用 fsync() 或原子文件更新技术来保证关键数据安全。

📎 原文：linux-mtd.infradead.org/faq/ubifs.html（"Why is my file empty" 章节）
📌 章节总结 · 第一性原理

写回是以牺牲"掉电前最后数据必然持久化"为代价，换取显著的写入性能。对于嵌入式系统，这是合理的取舍——关键数据可通过 fsync 显式持久化，一般文件则享受批量写入带来的性能红利与闪存寿命延长。

10. 垃圾回收（Garbage Collection）

UBIFS 无法原地覆盖 Flash 数据。文件内容被修改时，旧数据节点变为脏空间（Dirty Space）。脏空间必须通过垃圾回收（GC）整理后才能重新利用。

图6：UBIFS 垃圾回收流程——移动有效节点、擦除受害 LEB、释放空间

GC 的效率受最小 I/O 单元大小影响：当脏空间小于最小 I/O 单元时，GC 无法高效回收所有脏空间。UBIFS GC 目前不在后台自动运行（不像 JFFS2），只在需要空间时被动触发。

📎 原文：linux-mtd.infradead.org/doc/ubifs.html（"Dirty space" 章节）
📌 章节总结

GC 是裸 Flash 文件系统的必要机制，因为 Flash 的写入本质上是追加式（Out-of-Place）的。UBIFS GC 以 LEB 为粒度，被动触发，无后台 GC。对于接近写满的文件系统，GC 压力增大，性能可能下降。

11. 透明压缩（On-the-Fly Compression）

UBIFS 支持对数据进行透明压缩，数据以压缩形式存储在 Flash 上，读取时自动解压，对上层应用完全透明。这使得同样大小的 Flash 可以存储更多数据。

📎 原文：linux-mtd.infradead.org/doc/ubifs.html（Compression 章节）
⚡ LZO（默认）

速度快，压缩比适中。适合嵌入式 CPU 性能有限的场景

🗜️ zlib

压缩比更高，但速度较慢（压缩、解压均慢）。数据敏感、存储紧张时使用

🔀 Favor LZO

mkfs.ubifs 1.1+ 新增：同时尝试两种算法，若 zlib 压缩率比 LZO 高 20%（可配置）则用 zlib，否则用 LZO

UBIFS 支持按 inode 设置压缩方式，可全局开启，也可针对特定文件（如多媒体文件）关闭压缩，避免无效压缩计算。

挂载时覆盖压缩器

# 挂载时指定压缩器
mount -t ubifs ubi0:rootfs /mnt/ubifs -o compr=lzo
mount -t ubifs ubi0:rootfs /mnt/ubifs -o compr=zlib
mount -t ubifs ubi0:rootfs /mnt/ubifs -o compr=none
📎 原文：kernel.org/doc/Documentation/filesystems/ubifs.txt（Mount options 章节）
📌 章节总结

透明压缩对嵌入式系统意义重大：它等效地扩大了 Flash 容量，且对应用层完全透明。LZO 的选择平衡了速度与压缩比；Favor LZO 模式进一步优化了这一平衡。

12. 掉电容忍（Power-Cut Tolerance）

UBIFS 被设计为对意外掉电和不正常重启具有充分的容忍能力，无需像 fsck.ext2 这样的工具进行修复。

📎 原文：linux-mtd.infradead.org/doc/ubifs.html（Power-cuts tolerance 章节）
✅

UBIFS 在 OneNAND Flash 上经历了超过 100,000 次真实掉电测试（使用 Power Node 设备通过串口控制电源），证明了其掉电安全性。

恢复机制

📖 日志回放

挂载时找到最新的 Commit Start 节点，回放其后的引用节点和 Bud，重建一致的文件系统状态

🔄 主节点双备份

LEB 1 和 LEB 2 互为备份，即使更新主节点途中掉电，也总有一个有效副本可用

🗑️ 孤儿区清理

孤儿区记录已删除但尚有引用的 inode，重启时清理，防止空间泄漏

⚡ CRC 完整性

每个节点携带 CRC32，UBIFS 可检测到部分写入（Partial Write）产生的损坏节点并跳过

MLC NAND 的特殊问题

官方文档明确指出，MLC NAND 存在配对页（Paired Pages）问题：对某页写入时掉电，可能同时损坏与之配对的另一页。UBIFS 当前尚未完全解决此问题（标注为 NEED WORK）：

📎 原文：linux-mtd.infradead.org/doc/ubifs.html（UBIFS and MLC NAND flash 章节）
📌 章节总结 · 第一性原理

掉电安全的本质是保证"原子性"：每次提交要么完整写入（可通过日志回放恢复），要么被检测为损坏并丢弃。UBIFS 通过日志、CRC、主节点双备份三重机制实现这一保证。但 MLC NAND 的配对页问题是硬件约束，当前仍是 UBIFS 的已知局限。

13. 优势

🚀 快速挂载

挂载时间不依赖 Flash 大小（UBIFS 层面），只需读取主节点（LEB 1/2）和回放固定大小的日志。百毫秒级挂载。

📈 对数扩展性

UBIFS 使用 B+ 树，文件访问时间、挂载时间均为 O(log N)，不随 Flash 容量线性增长（JFFS2 是线性的）。

⚡ 写回支持

支持 Write-Back，显著提升写入吞吐量（尤其是小文件密集写场景）。JFFS2 是同步写穿，每次写操作直接落盘。

🗜️ 透明压缩

LZO/zlib/Favor LZO 按需压缩，可大幅减少 Flash 占用，支持按 inode 精细控制。

⚡ 掉电安全

日志型文件系统，支持 100,000+ 次真实掉电测试验证。自动回放日志，无需 fsck。

🔍 数据完整性

所有节点携带 CRC32，元数据损坏可被检测。文件系统可通过扫描全盘重建索引（类似 JFFS2）。

💾 内存高效

内存占用不依赖文件系统内容量（JFFS2 为每个节点保留内存数据结构）。TNC 可收缩。

🔒 无关坏块/磨损

借助 UBI 层，坏块处理和磨损均衡对 UBIFS 完全透明，文件系统层无需关心硬件细节。

📎 原文：linux-mtd.infradead.org/doc/ubifs.html（Overview 章节特性列表）

14. 劣势与局限

❌ 不支持块设备

UBIFS 不能用于 eMMC、SD 卡、SSD、USB 存储等块设备。只能用于裸 NAND/NOR Flash 的 UBI 卷。

❌ UBI 线性扩展

虽然 UBIFS 本身对数扩展，但 UBI 层是线性扩展的（扫描所有 PEB 以构建映射表）。整体栈在大容量 Flash 上仍有扩展瓶颈。

❌ 空间开销较大

相比 JFFS2，UBIFS 在闪存上存储更多索引信息（LPT、B+树），对小 Flash（如 64MiB 以下）空间开销相对较高。官方建议小 Flash 考虑 JFFS2。

❌ 空间统计不精确

UBIFS 的 df 报告可能偏保守（悲观估计）：因节点不跨 LEB 存在末尾浪费、脏空间不可完全回收、索引大小不精确估算。文件系统 sync 后报告更准确。

❌ MLC NAND 配对页问题

MLC NAND 的配对页（Paired Pages）掉电损坏问题尚未完全解决，UBIFS 官方文档标注为 NEED WORK。

❌ 最小 LEB 数限制

UBIFS 要求至少 17 个 LEB。对极小 Flash（如 ≤8MiB）不适用，会报 "too few LEBs" 错误。

❌ 写回带来文件一致性风险

写回模式下，掉电可能导致文件内容为空（创建后未 fsync）。应用程序需配合使用 fsync 或原子文件更新模式。

❌ 不支持循环挂载

UBIFS 无法挂载为 loopback 设备（文件驱动的块设备），必须通过 UBI 卷挂载。调试时需借助 nandsim 模拟器。

📎 原文：linux-mtd.infradead.org/faq/ubifs.html
📌 章节总结

UBIFS 的局限大多源自其设计哲学：专为裸 Flash 优化。当使用场景超出这一前提（块设备、极小 Flash、MLC NAND 高可靠需求），局限就变得明显。这些不是 BUG，而是设计取舍的结果。

16. 适用场景

官方文档指出，裸 Flash + UBI/UBIFS 的组合主要用于嵌入式世界，适合以下场景：

📱 嵌入式 Linux 根文件系统

路由器、工控设备、智能家电、机顶盒等搭载裸 NAND Flash 的嵌入式系统，使用 UBIFS 作为根文件系统，兼顾掉电安全与性能

🏭 工业级设备

对掉电安全要求极高的工控设备，在已知 Flash 可靠参数的情况下，UBIFS 的掉电容忍机制比 FTL 黑盒更可控、可验证

🌐 网络设备

使用裸 NAND/NOR Flash 的网络交换机、无线接入点等，UBIFS 提供压缩存储以节省 Flash 空间，同时支持快速重启后恢复

💾 中大容量 Flash（64MiB~数 GiB）

🔧 SLC NAND 场景

📌 章节总结

UBIFS 的黄金使用场景：裸 SLC NAND Flash + 中大容量 + 嵌入式 Linux + 需要掉电安全。在这个场景下，UBI/UBIFS 是目前 Linux 生态中最成熟、可靠的选择。

17. 不适用场景

❌ eMMC / SD / SSD / USB 存储

这些是块设备（内部有 FTL），UBIFS 无法运行于其上。应使用 ext4、f2fs 等块设备文件系统

❌ 极小 Flash（<17 LEB）

UBIFS 自身的元数据开销较大，官方建议极小 Flash 使用 JFFS2，其元数据开销更低

❌ MLC NAND 高可靠需求

MLC NAND 配对页问题在 UBIFS 尚未完全解决，对掉电安全有极高要求时需谨慎评估

❌ FAT 兼容需求

若需要与 Windows/相机/消费电子兼容（如相机存储卡），应使用 FTL 设备 + FAT，UBIFS 不具备 FAT 兼容性

18. 打包工具：mkfs.ubifs

mkfs.ubifs 是创建 UBIFS 镜像的工具，来自 mtd-utils 项目（git://git.infradead.org/mtd-utils.git）。它生成的镜像可写入 UBI 卷，或进一步用 ubinize 打包为 UBI 镜像。

📎 原文：linux-mtd.infradead.org/doc/ubifs.html（User-space tools 章节） 📎 原文：linux-mtd.infradead.org/faq/ubifs.html（"How do I create an UBIFS image" 章节）
mkfs.ubifs
从目录树创建 UBIFS 镜像文件（.img）
# 基本用法：从 root-fs 目录创建 UBIFS 镜像
# 针对 256MiB SLC OneNAND，128KiB PEB，2048B 页，512B 子页
mkfs.ubifs -q -r root-fs -m 2048 -e 129024 -c 2047 -o ubifs.img

# 参数说明：
# -r root-fs    从 root-fs 目录构建文件系统内容
# -m 2048       最小 I/O 单元大小（NAND 页大小，字节）
# -e 129024     逻辑擦除块（LEB）大小（字节）
#               = 物理擦除块 - UBI 开销
# -c 2047       最大文件系统大小（以 LEB 为单位，约 251MiB）
# -o ubifs.img  输出文件
# -q            安静模式

# 使用 zlib 压缩（压缩比更高）
mkfs.ubifs -q -r root-fs -m 2048 -e 129024 -c 2047 -x zlib -o ubifs.img

# 使用 Favor LZO 压缩（threshold=5%，若 zlib 压缩率高5%则用zlib）
mkfs.ubifs -q -r root-fs -m 2048 -e 129024 -c 2047 -x favor_lzo -X 5 -o ubifs.img

# 不压缩
mkfs.ubifs -q -r root-fs -m 2048 -e 129024 -c 2047 -x none -o ubifs.img

# 启用 free space fixup（某些平台如 AM335x 必须加 -F）
mkfs.ubifs -q -r root-fs -m 2048 -e 129024 -c 2047 -F -o ubifs.img

# NOR Flash（32MiB，128KiB PEB，min IO=1字节）
mkfs.ubifs -q -r root-fs -m 1 -e 130944 -c 255 -o ubifs.img

关键参数详解

参数含义如何获取
-m最小 I/O 单元大小（NAND 页大小）mtdinfo -u 输出的 "Minimum input/output unit size"
-e逻辑擦除块（LEB）大小 = 物理擦除块大小 - UBI 开销mtdinfo -u 计算；UBI overhead = ALIGN(VID_hdr_offset + 64, min_io)
-c最大文件系统大小（LEB 数量），决定 LPT 大小根据 MTD 分区大小计算；建议用最大可能值保证兼容性
-r源目录（文件系统内容）你的 rootfs 目录路径
-x压缩算法：lzo（默认）/ zlib / favor_lzo / none按需选择
-F空闲空间修复标志（某些平台首次挂载时触发自动修复）某些 SoC（如 TI AM335x）必须
📎 原文：linux-mtd.infradead.org/faq/ubifs.html（创建镜像章节）
📌 章节总结

mkfs.ubifs 最关键的参数是 -m（页大小）和 -e（LEB 大小），必须与目标硬件完全匹配。错误的 LEB 大小会导致超级块 mismatch 错误，文件系统无法挂载。

19. 打包工具：ubinize

ubinize 将一个或多个 UBIFS 镜像打包为 UBI 镜像（.ubi），该 UBI 镜像可直接刷写到裸 Flash。UBI 镜像中包含 UBI 头部信息（EC header、VID header）以及 UBI 卷。

📎 原文：linux-mtd.infradead.org/faq/ubifs.html（"How do I create an UBIFS image" 章节）
ubinize
将 UBIFS 镜像封装为 UBI 镜像（可直接刷写到裸 Flash）
# 第一步：准备 ubinize.cfg 配置文件
cat ubinize.cfg
# [ubifs]
# mode=ubi
# image=ubifs.img          # mkfs.ubifs 生成的 UBIFS 镜像
# vol_id=0                 # 卷 ID
# vol_size=200MiB          # 卷大小（初始值；autoresize 时会自动扩展）
# vol_type=dynamic         # 动态卷（可读写）
# vol_name=rootfs          # 卷名称
# vol_flags=autoresize     # 首次运行时自动扩展到最大可用空间

# 第二步：运行 ubinize
# 针对 256MiB SLC OneNAND，128KiB PEB，512B 子页
ubinize -o ubi.img -m 2048 -p 128KiB -s 512 ubinize.cfg

# 参数说明：
# -o ubi.img    输出 UBI 镜像文件
# -m 2048       最小 I/O 单元大小（NAND 页大小，字节）
# -p 128KiB     物理擦除块大小
# -s 512        子页大小（支持子页时 EC/VID 头共用同一 NAND 页，节省空间）

# 针对 512MiB MLC NAND，128KiB PEB，2048B 页，无子页
ubinize -o ubi.img -m 2048 -p 128KiB ubinize.cfg

# 针对 32MiB NOR Flash，128KiB PEB，min_io=1
ubinize -o ubi.img -m 1 -p 128KiB ubinize.cfg

参数详解

参数含义说明
-m最小 I/O 单元大小与 mkfs.ubifs 的 -m 参数一致
-p物理擦除块大小Flash 芯片规格，通常为 128KiB 或 256KiB
-s子页大小SLC NAND 典型支持子页；设置后 VID header 与 EC header 放同一 NAND 页，节省一个子页
vol_flags=autoresize自动扩容配合 -c 2047，首次挂载时卷和文件系统自动扩展到最大可用空间
vol_type=static静态卷只读，带 CRC；与 dynamic（可读写）对应
📎 原文：linux-mtd.infradead.org/faq/ubifs.html

📌 总结 · 第一性原理回顾

UBIFS 的整个设计链条可以从一个第一性原理推导：闪存只能顺序追加写、不能原地修改、有磨损限制。由此衍生出：分层架构（UBI 处理磨损，UBIFS 处理文件语义）→ Out-of-Place Write（写到新位置）→ B+ 漂移树（追踪最新数据位置）→ 日志（缓冲变更，减少索引更新频率）→ LPT（持久化空间统计，避免全盘扫描）→ 垃圾回收（整理旧版本数据）→ 透明压缩（最大化闪存利用率）。每一个设计决策都能溯源到这一物理约束。
