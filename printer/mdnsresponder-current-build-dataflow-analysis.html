mDNSResponder 当前配置、原理与场景数据流

本文基于当前工程 server/mDNSResponder-1310.80.1 的 Makefile、已编译对象文件和源码控制路径，结合 IETF 官方 RFC 6762/6763，分析本设备上 mDNSResponder 的实际作用、编译边界、mDNS/DNS-SD 原理，以及打印机场景下的数据流。

结论先行：当前顶层安装目标不是完整 mdnsd daemon，也不是 libdns_sd.so 客户端库，而是 embedded standalone responder：mDNSPosix/build/prod/mDNSResponderPosix，安装时复制为 mDNSResponder。它在本设备上注册一个打印相关服务，使局域网内客户端可以通过 mDNS/DNS-SD 发现这台打印设备。
第一性原理：服务发现的根问题是“客户端不知道服务在哪台机器、哪个端口、有哪些属性”。mDNSResponder 的价值是把这些信息用标准 DNS 记录发布到本地链路，而不依赖中心 DNS 服务器。

官方依据：RFC 6762 与 RFC 6763

来源原文要点本文使用方式
RFC 6762: Multicast DNSmDNS 在没有传统单播 DNS 服务器时，在本地链路上执行 DNS-like 查询和响应；.local. 是本地链路特殊域；查询发往 IPv4 224.0.0.251 或 IPv6 FF02::FB，UDP 端口 5353；包含 probing、announcing、conflict resolution、Known-Answer Suppression、cache TTL、cache-flush 等规则。解释当前代码为何绑定 5353、加入 multicast group、设置 TTL 255、用 probe/announce 注册唯一记录，并在收到 query 时按 mDNS 规则响应。
RFC 6763: DNS-Based Service DiscoveryDNS-SD 用标准 DNS 记录命名和组织服务发现：浏览阶段查询 <Service>.<Domain> 的 PTR；解析阶段查询服务实例名的 SRV/TXT；SRV 提供目标主机和端口，TXT 提供少量 key/value 附加属性；服务实例名形式为 <Instance>.<Service>.<Domain>。解释当前 mDNS_RegisterService() 为什么生成 RR_PTR、RR_SRV、RR_TXT 和 RR_ADV，以及局域网客户端如何发现设备。
第一性原理：mDNS 是“运输方式”：把 DNS 包放到本地 multicast；DNS-SD 是“数据模型”：规定服务用哪些 DNS 名字和记录表达。

mDNS 原理：把 DNS 从中心服务器改成本地协作

概念官方规则当前代码落点
.local.RFC 6762 定义 .local. 为本地链路特殊域，名字只在发起查询的链路上有意义。默认服务域 kDefaultServiceDomain = "local."；当前服务实例注册在本地链路域。
多播地址/端口IPv4 使用 224.0.0.251，IPv6 使用 FF02::FB，UDP 端口 5353。SetupSocket() 为接口 socket 加入 AllDNSLinkGroup_v4/v6，绑定 MulticastDNSPort。
IP TTL响应建议 TTL/hop limit 为 255，用于判断来源在本地链路。SetupSocket() 设置 IP_TTL、IP_MULTICAST_TTL、IPv6 hop limit 为 255。
响应来源responder 只能发送自己权威拥有的记录，不能把 cache 中学到的他人记录作为权威答案转发。ProcessQuery() 遍历 m->ResourceRecords 选出本机可回答记录。
第一性原理：mDNS 把“谁是 DNS 服务器”这个中心角色拆给每个服务提供者。每台设备只对自己拥有的信息负责，所有设备通过 multicast 共享观察结果。

DNS-SD 记录模型：PTR 浏览，SRV/TXT 解析

RFC 6763 规定服务发现不新增 DNS 报文格式，也不新增新的 op code 或 RR type。它复用 PTR、SRV、TXT、A/AAAA：PTR 用来列出服务实例，SRV 给出目标主机与端口，TXT 给出少量附加属性。当前核心函数 mDNS_RegisterService() 正是按这个模型构造记录集。

第一性原理：服务发现分两步：先枚举“有哪些服务实例”，再解析“这个实例怎么连接”。PTR 负责枚举，SRV/TXT/A/AAAA 负责连接参数。

模块组合：Responder + Posix 平台层 + mDNSCore

层次当前职责关键函数
应用壳决定本设备发布什么服务，何时启动、重载、退出。initArgs()、RegisterOurServices()、main()
核心协议维护 mDNS 全局状态、AuthRecord、ServiceRecordSet、probe/announce/query/response/cache。mDNS_Init()、mDNS_RegisterService()、mDNS_Execute()、mDNSCoreReceive()
平台适配枚举接口、创建 socket、加入 multicast group、收发 UDP、向核心投递包。mDNSPlatformInit()、SetupSocket()、mDNSPlatformSendUDP()、SocketDataReady()
第一性原理：清晰分层来自职责分离：业务层只说“我要发布什么服务”，协议层决定“何时发什么 DNS 记录”，平台层负责“如何在这个 OS 上发出去”。

启动注册流：从 SN 到 PTR/SRV/TXT

阶段源码路径数据变化
参数初始化Responder.c:initArgs()由 sys_global_var()->sn 组成 Printer_<sn>，端口设为 9100，TXT 写入 t=printer。
核心初始化mDNS_Init() → mDNS_InitStorage() → mDNSPlatformInit()初始化 mDNSStorage，设置主机名/FQDN，创建 unicast socket，枚举接口并注册到核心。
接口加入SetupInterfaceList() → SetupOneInterface() → SetupSocket()为接口建立 multicast socket，加入 224.0.0.251/FF02::FB，绑定 UDP 5353。
服务注册RegisterOurServices() → RegisterOneService() → mDNS_RegisterService()生成 DNS-SD 的 PTR/SRV/TXT/ADV 记录，放入核心 ResourceRecords，等待 probe/announce。
第一性原理：启动注册的本质是把设备内部状态转换成可被网络发现的 DNS 记录；记录未发布前只是内存状态，经过 probe/announce 后才成为局域网事实。

事件循环：select 驱动核心定时器和 socket

当前 standalone responder 没有 daemon 的客户端 IPC 主循环。它在 Responder.c:main() 中循环调用 mDNSPosixGetFDSet()、select()、mDNSPosixProcessFDSet()。平台层注释明确说明 Posix locking 是 no-op，因为该路径只在主线程进入 mDNSCore。

第一性原理：mDNS 是事件驱动协议：有些动作由网络包触发，有些动作由时间触发。事件循环把“等包”和“到点发包”统一成一个 select 调度问题。

查询响应流：客户端如何找到打印服务

网络动作当前 responder 路径输出
客户端浏览服务类型收到 _afpovertcp._tcp.local. PTR 查询后，SocketDataReady() 调 mDNSCoreReceive()。核心识别标准查询，进入 mDNSCoreReceiveQuery()。
匹配本机记录ProcessQuery() 遍历 m->ResourceRecords，用 AnyTypeRecordAnswersQuestion() 判断可回答记录。标记 PTR，并通过 additional 机制带出 SRV/TXT。
发送响应SendResponses() 或即时 unicast response 路径调用 mDNSSendDNSMessage()，最终 mDNSPlatformSendUDP()。向本地链路发送 DNS response，客户端和其他设备可更新 cache。
客户端连接服务DNS-SD 解析到 SRV 目标 host 和端口 9100，再查 A/AAAA。后续打印数据流不再属于 mDNS；mDNS 只负责发现和解析连接参数。
第一性原理：发现协议不搬运业务数据。它只把“连接业务服务需要的信息”发布出去，真正打印数据走 SRV 记录指向的业务端口。

冲突、通告与缓存：让本地视图保持一致

探测与冲突

RFC 6762 要求唯一记录在声明前先 probe。当前 mDNS_RegisterService() 把 SRV/TXT 注册为 unique 记录；如果注册回调收到 mStatus_NameConflict，RegistrationCallback() 调 mDNS_RenameAndReregisterService() 自动改名重注册。

通告

mDNS_Execute() 在定时器到期时执行 SendQueries() 与 SendResponses()。probe 完成后的记录会进入 announce 流程，对外发布 PTR/SRV/TXT/A 等信息。

Known-Answer Suppression

ProcessQuery() 会解析 query 的 Answer Section，若 querier 已经带了足够新鲜的 Known Answer，则抑制重复响应，符合 RFC 6762 的流量降低规则。

缓存与 POOF

核心维护 cache record TTL，并在看到其他主机查询但没有看到期望响应时，可用 Passive Observation Of Failures 加速清理陈旧记录。当前本设备主要是 responder，但核心机制仍在 mDNS.c 中。

第一性原理：没有中心服务器时，一致性只能靠“先声明前探测、声明后旁路观察、过期前刷新、离开时 goodbye”。这些规则共同把临时网络状态变成可接受的一致视图。

基于打印机场景的数据流

场景数据流代码路径用户可见结果
设备开机入网枚举接口 → 绑定 UDP 5353 → 加入 mDNS multicast group → 注册 Printer_<sn> 服务 → probe/announce。main()、mDNS_Init()、mDNSPlatformInit()、RegisterOurServices()、mDNS_Execute()同网段客户端的服务浏览列表出现这台打印设备。
手机/PC 搜索打印机客户端发 PTR 查询到 224.0.0.251:5353 → 本机匹配 PTR → 回复实例名，并附加 SRV/TXT/A。SocketDataReady()、mDNSCoreReceive()、ProcessQuery()、SendResponses()客户端获得实例名、目标主机、端口 9100、TXT t=printer。
客户端开始打印DNS-SD 发现完成后，客户端连接 SRV 指向的主机和 TCP 9100。不再由 mDNSResponder 处理，进入打印业务服务。打印数据发往设备的打印端口。
网络接口变化平台层收到 netlink/接口变化事件 → 刷新接口列表 → 重新注册/通告。WatchForInterfaceChange()、mDNSPlatformPosixRefreshInterfaceList()、SIGHUP 处理也可触发刷新。设备切网、IP 改变后，局域网服务记录重新收敛。
重名冲突其他设备使用相同服务名并被检测到 → 注册回调收到冲突 → 自动重命名并重注册。RegistrationCallback()、mDNS_RenameAndReregisterService()服务名可能变为带数字后缀的唯一名称，避免列表中两个不可区分的同名服务。
第一性原理：打印机场景里，mDNS/DNS-SD 只负责“让客户端找到入口”。入口找到后，可靠传输、打印协议、任务状态都属于后续打印服务，不应混入 mDNS 职责。

eth0 + wlan0 同时联网时的行为

结论：当前 Posix standalone responder 是按接口工作的。eth0 和 wlan0 都处于 IFF_UP 且不是 point-to-point 时，SetupInterfaceList() 会分别调用 SetupOneInterface() 注册接口，并为每个接口建立 multicast socket。服务记录本身用 mDNSInterface_Any 注册，且本平台 mDNSPlatformValidRecordForInterface() 返回 true，所以这组打印服务记录可在两个接口上发布和回答。
问题当前实现结论代码依据抓包预期
1. 设备开启时，会分别往 eth0 和 wlan0 都发送组播吗？会。只要两个接口都被枚举为活动接口，核心在 probe/announce/send response 时会遍历 m->HostInterfaces，通过对应接口的 multicast socket 发送到 224.0.0.251:5353。SetupInterfaceList() 枚举 UP 接口；SetupOneInterface() 设置 InterfaceID、McastTxRx、接口 multicast socket；SendResponses() 遍历接口并设置 SendRNow = GetNextActiveInterfaceID(...)；mDNSPlatformSendUDP() 根据 InterfaceID 选择 thisIntf->multicastSocket4/6。在 eth0 所在网络能看到开机 probe/announcement；在 wlan0 所在网络也能看到对应 probe/announcement。若两个接口连接同一个二层网络，Windows 可能在同一个抓包视图中看到来自设备两个源 IP/MAC 的 mDNS 包。
2. 对端发送组播查询时，会两个接口都回复吗？通常不会“两个接口一起回复”。对端的 multicast query 从哪个接口进入，SocketDataReady() 就把那个接口的 InterfaceID 传给 mDNSCoreReceive()，核心把即时响应标记到该接口，响应从该接口发回。若 eth0、wlan0 分属两个不同二层网络，另一个接口根本收不到这次组播查询，也不会回复。mDNSPosixProcessFDSet() 对每个接口 socket 调 SocketDataReady(m, info, ...)；SocketDataReady() 校验包的实际接收接口，传入 InterfaceID = intf->coreIntf.InterfaceID；SendResponses() 对 rr->ImmedAnswer 的单接口情况设置 rr->SendRNow = rr->ImmedAnswer。Windows 在 eth0 网络发查询，只应看到设备 eth0 地址回应；在 wlan0 网络发查询，只应看到设备 wlan0 地址回应。若两个网口桥接到同一广播域，可能因为网络拓扑导致两个接口都收到同一查询，这时可能看到两个接口各自响应，这属于二层拓扑造成的“两个接口都收到了查询”。
3. 其中一个网口变动时，会主动发布更新吗？会触发接口刷新，并对变化后的接口重新注册/通告。Linux 当前宏包含 USES_NETLINK，平台层注册接口变化通知；有变化时会重建整个接口列表。旧接口 deregister 可能产生 goodbye/撤销，新接口 register 后会触发 AdvertiseInterfaceIfNeeded() 和后续 announce。WatchForInterfaceChange() 把 netlink fd 加入事件循环；InterfaceChangeCallback() 处理路由通知后调用 mDNSPlatformPosixRefreshInterfaceList()；该函数先 ClearInterfaceList() 再 SetupInterfaceList()；mDNS_RegisterInterface() 在 set->Advertise 时调用 AdvertiseInterfaceIfNeeded()。拔掉/断开其中一个网口时，可能看到该接口上的 goodbye 或记录撤销；重新连上、获取地址或接口 up 后，会看到 probe/announcement。另一个未变动接口也可能因当前实现“整表重建”出现重新注册相关包。

查询从哪个接口来，就从哪个接口回复在 Windows 上打开 Wireshark 抓对应网卡，然后触发一次 DNS-SD 查询。可用 Bonjour Browser、支持 DNS-SD 的打印发现界面，或安装 Bonjour SDK 后用 dns-sd -B _afpovertcp._tcp local。分别在有线和无线网络测试。udp.port == 5353 && (dns.qry.name contains "_afpovertcp._tcp.local" || dns.resp.name contains "_afpovertcp._tcp.local")有线侧查询后看到设备 eth0 地址回复；无线侧查询后看到设备 wlan0 地址回复。若只在有线抓包，不应凭空看到 wlan0 网络的独立回复，除非两个网口处在同一个二层广播域或桥接环境。
单个网口变动会主动更新保持 Wireshark 抓包，断开/重新连接设备 eth0 或 wlan0，或切换 Wi-Fi，等待接口 up/down 和地址变化。udp.port == 5353 && (dns.flags.response == 1 || dns.qry.name contains "_afpovertcp._tcp.local")接口 down 时可能看到 TTL 为 0 的 goodbye/撤销记录；接口 up 或地址变化后看到新的 probe/announcement。因为当前实现刷新时重建整个接口列表，另一个未变接口也可能出现重新通告。
确认包的接口归属在 Wireshark 的包详情中查看 Frame/Interface 或源 MAC/IP；也可以同时打开两个抓包窗口分别抓 Ethernet 和 Wi-Fi。eth.addr == 设备MAC
ip.addr == 设备eth0地址 || ip.addr == 设备wlan0地址		同一个服务实例会出现在不同接口的链路上，但源 MAC/IP 应对应各自接口；这能证明 responder 是按接口发布，而不是只在一个全局 socket 上发布。
抓包注意：Windows 的 Wi-Fi 网卡通常只能看到自己所在 BSS 内的普通数据帧；不需要 monitor mode 也能抓到本机收到/发送的 UDP 5353 mDNS。若 Windows 只连有线或只连无线，就只能证实它所在那一条链路。要同时证实 eth0 和 wlan0，需要两块网卡同时抓，或分两次分别接入两张网络。
第一性原理：mDNS 的传播边界是“接口所在的本地链路”，不是设备整体。多网口设备等价于同时站在多个本地链路上，每条链路都要独立发布、独立接收查询、独立维护缓存一致性。

边界与风险：当前实现应如何理解

边界当前事实影响
Services.txt当前 SUNMI 路径不读取它。它仍描述了 -f 文件格式：服务名、类型/协议、端口、TXT 字符串。修改 Services.txt 不会改变当前产品注册内容，除非关闭 SUNMI 路径并走命令行参数。
服务类型选择当前注册 _afpovertcp._tcp.，端口却是 9100。文档按源码事实描述，不推断它是否应为 IPP 或 pdl-datastream；客户端必须按当前服务类型理解。
安全模型RFC 6762 明确 mDNS 冲突解决假设本地链路参与者合作；陌生网络中 .local. 名字可被伪造。mDNS 适合本地发现，不提供强身份认证；敏感业务仍需后续连接层的安全机制。
多 responder 共存RFC 6762 建议使用 SO_REUSEADDR/SO_REUSEPORT 支持多个进程共享 5353 multicast；当前 SetupSocket() 在 Linux 优先设置 SO_REUSEADDR。多实例可共享 multicast，但 unicast response 可能只被第一个绑定者收到，代码用 CanReceiveUnicastOn5353 标记能力。
本地链路范围.local. 与 224.0.0.251/FF02::FB 都是 link-local 语义。跨路由/跨 VLAN 通常不会直接发现，除非网络另有代理或中继机制。
第一性原理：边界不是缺陷。mDNS 的设计目标就是低配置、本地链路、轻量发现；超出这个目标的跨网段发现和强认证，应由其他机制补足。

总结

当前设备使用 Apple mDNSResponder 的嵌入式 standalone responder 路径，在 MIPS Linux 上编译成 mDNSResponderPosix。它通过 SUNMI 定制的 initArgs() 注册 Printer_<sn>._afpovertcp._tcp.local.，端口 9100，TXT t=printer。mDNSCore 负责把这组服务记录按 RFC 6762/6763 的规则探测、通告、响应和维护一致性；mDNSPosix 负责 Linux socket、接口、multicast group 和 select 事件循环。
问题答案
mDNS 的作用是什么？在没有中心 DNS 的本地链路上，让设备能用 DNS 报文发现名字和服务。
DNS-SD 的作用是什么？规定服务如何用 PTR/SRV/TXT/A/AAAA 记录表达，使客户端能浏览和解析服务。
当前产品发布什么？一个打印相关服务实例，名称来自设备 SN，端口为 9100，TXT 为 t=printer。
数据流关键路径是什么？main() → mDNS_Init() → mDNSPlatformInit() → mDNS_RegisterService() → mDNS_Execute() / mDNSCoreReceive() → mDNSPlatformSendUDP()。
第一性原理：mDNSResponder 是“把设备自我描述转换成本地链路 DNS 事实”的组件。它不打印、不传输打印数据，只让正确的客户端用正确的名字、端口和属性找到打印入口。
