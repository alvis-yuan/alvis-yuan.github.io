lib 目录各文件如何组合完成 MQTT 协议

本文聚焦 dmp/package/mosquitto/src/lib 中参与当前 libmosquitto.a 的 C 源文件，按 MQTT 协议主题解释文件职责、关键函数和组合链路。它不是逐行注释，也不展开 broker、apps、clients、tests 或 C++ wrapper。

当前工程把 Mosquitto 2.0.14 的 lib 目录编成静态库 libmosquitto.a。这批文件共同完成三件事：把公开 API 调用转成 MQTT 控制报文；把 socket 字节流拆回控制报文并分发给 handler；用消息状态机维持 QoS、Keep Alive、重连、属性和回调。
第一性原理：MQTT 客户端库的最小模型是“状态对象 + 字节流编解码 + 网络循环 + 报文状态机”。lib 目录的文件分工基本就是围绕这四个不变量展开。

官方协议依据

官方原文关键事实映射到 lib 文件
OASIS MQTT 5.0MQTT 控制报文由 Fixed Header、可选 Variable Header、可选 Payload 按顺序组成；控制报文类型包含 CONNECT、CONNACK、PUBLISH、PUBACK、PUBREC、PUBREL、PUBCOMP、SUBSCRIBE、SUBACK、UNSUBSCRIBE、UNSUBACK、PINGREQ、PINGRESP、DISCONNECT、AUTH。packet_mosq.c、packet_datatypes.c、read_handle.c、send_*.c、handle_*.c
OASIS MQTT 3.1.1PUBLISH 的 QoS 0/1/2 分别表示最多一次、至少一次、恰好一次；QoS 1 需要 PUBACK，QoS 2 需要 PUBREC/PUBREL/PUBCOMP 四步。messages_mosq.c、send_publish.c、handle_publish.c、handle_pubackcomp.c、handle_pubrec.c、handle_pubrel.c
Mosquitto APImosquitto_loop() 必须频繁调用，用于处理输入、输出以及 QoS>0 流程中剩余命令；可用 mosquitto_loop_forever() 或 mosquitto_loop_start() 管理循环。loop.c、thread_mosq.c、packet_mosq.c
Mosquitto API线程应用若自建线程而不使用 mosquitto_loop_start()，需通过 mosquitto_threaded_set() 告知库；TLS、Will、用户名密码、属性均应在连接前配置。thread_mosq.c、options.c、tls_mosq.c、will_mosq.c、send_connect.c
第一性原理：协议规范定义“线上必须是什么字节和状态迁移”，Mosquitto API 定义“应用如何驱动这些字节和状态迁移”。源码分析必须同时满足这两层契约。

归档对象边界

当前 lib/Makefile 的 MOSQ_OBJS 决定了静态库边界，规则为 libmosquitto.a : ${MOSQ_OBJS} 后调用 $(AR) cr 归档。下表只覆盖这些对象对应的 C 文件。

主题文件说明
公开 API 与上下文mosquitto.c、callbacks.c、options.c、actions.c接收应用意图，初始化 struct mosquitto，设置回调/选项，发起发布订阅等动作。
连接与循环connect.c、loop.c、net_mosq.c、net_mosq_ocsp.c、time_mosq.c建立 TCP/TLS/SOCKS 连接，select/pselect 等待读写，维护 Keep Alive 与重连。
报文编解码packet_mosq.c、packet_datatypes.c、property_mosq.c、read_handle.c构造 Fixed Header/Remaining Length/Variable Header/Payload，拆包后按 command 分发。
发送端报文send_connect.c、send_disconnect.c、send_publish.c、send_subscribe.c、send_unsubscribe.c、send_mosq.c把 CONNECT/PUBLISH/SUBSCRIBE/UNSUBSCRIBE/PING/ACK 编成待发送 packet。
接收端 handlerhandle_connack.c、handle_publish.c、handle_pubackcomp.c、handle_pubrec.c、handle_pubrel.c、handle_suback.c、handle_unsuback.c、handle_ping.c、handle_disconnect.c、handle_auth.c解析 broker 返回报文，更新状态机并触发回调或下一步 ACK。
支撑能力messages_mosq.c、tls_mosq.c、socks_mosq.c、srv_mosq.c、thread_mosq.c、will_mosq.c、utf8_mosq.c、util_topic.c、util_mosq.c、strings_mosq.c、memory_mosq.c、misc_mosq.c、logging_mosq.c、helpers.cQoS 消息队列、TLS、SOCKS、线程、Will、UTF-8/topic 校验、错误/日志/内存工具。
cpp/、CMakeLists.txt、linker.version 不在当前静态库 C 对象链路中；srv_mosq.c 虽然归档，但当前包构建传入 WITH_SRV=no，SRV 功能路径不应按启用状态解释。
第一性原理：分析“参与实现”的文件时，Makefile 对象清单比目录树更重要；目录树回答“有哪些文件”，归档规则回答“哪些文件进入了当前产品”。

组合总图

第一性原理：MQTT 是字节流协议，不是函数调用协议。API 层只负责表达“我要连接/发布/订阅”，真正协议正确性由报文编码、网络循环、handler 状态迁移共同保证。

上下文与公开 API

mosquitto.c

实现 mosquitto_lib_init()、mosquitto_lib_cleanup()、mosquitto_new()、mosquitto_reinitialise()、mosquitto_destroy()。它分配并重置 struct mosquitto，初始化 socket、keepalive、协议版本、回调指针、in/out packet、消息队列配额和线程 mutex。

mosquitto_internal.h

定义内部核心状态：enum mosquitto_client_state、enum mosquitto_msg_state、struct mosquitto__packet、struct mosquitto_message_all、struct mosquitto_msg_data、struct mosquitto。这是所有 lib 文件共享的状态语言。

callbacks.c

只做一件事：把应用提供的函数指针写入 struct mosquitto。connect、disconnect、publish、message、subscribe、unsubscribe、log 都是后续 handler 回调应用的出口。

options.c / will_mosq.c

配置连接前参数：Will、用户名密码、重连延迟、TLS CA/cert/key、TLS insecure、PSK、协议版本、send/receive maximum、bind address 等。Will 的实际内存复制与校验落在 will__set()。

API 调用内部效果之后由谁消费
mosquitto_new()分配 struct mosquitto，调用 mosquitto_reinitialise() 建立默认协议 mqtt311、keepalive=60、最大 QoS=2、inflight=20。connect.c、loop.c、messages_mosq.c、handler 全部依赖该上下文。
mosquitto_username_pw_set()保存 username/password。send_connect.c 在 CONNECT payload 中设置 User Name/Password 标志并写入字段。
mosquitto_will_set()创建 mosquitto_message_all 形式的 will。send_connect.c 在 CONNECT 中写 Will Flag、Will QoS、Will Retain、Will Topic/Payload。
mosquitto_publish/sub/unsub先校验参数、生成 mid、检查 topic/UTF-8/属性，再进入 send 或消息队列。send_publish.c、send_subscribe.c、send_unsubscribe.c。
第一性原理：公开 API 不是直接“发包函数集合”，而是把应用意图写进一个可恢复、可回调、可并发保护的客户端上下文。

报文字节层

文件关键实现对应协议事实
packet_mosq.cpacket__alloc() 把 remaining_length 编成 MQTT Variable Byte Integer；packet__queue() 把待发 packet 追加到 out_packet 链表，并通过 socketpair 唤醒 select；packet__write() 处理部分写和 EAGAIN；packet__read() 负责从字节流累计一帧完整 MQTT packet。MQTT 控制报文固定头中的 Remaining Length 是变长整数，TCP/TLS 是字节流，不能假设一次 read/write 就是一个完整报文。
packet_datatypes.c提供 packet__read/write_byte、uint16、uint32、string、binary、varint。字符串读取后调用 mosquitto_validate_utf8()。MQTT 的 UTF-8 Encoded String、Binary Data、Two Byte Integer、Variable Byte Integer 都有明确线格式。
read_handle.chandle__packet() 按 in_packet.command & 0xF0 分发到 handle__connack/publish/pubrec/suback/...。控制报文类型位于 Fixed Header 第一个字节高 4 位。
send_mosq.c封装 PINGREQ/PINGRESP/PUBACK/PUBREC/PUBREL/PUBCOMP 等小报文；send__command_with_mid() 写 packet id、可选 reason code 和属性。ACK 类报文通常由固定头 + Packet Identifier + MQTT v5 可选 reason/properties 组成。
第一性原理：只要字节流边界可靠，协议层就能被拆成“类型分发 + 字段读写 + 状态迁移”。这就是 packet_* 与 handle_* 分开的原因。

连接与会话

阶段文件/函数实现动作协议语义
准备连接参数connect.c / mosquitto__connect_init()校验 host/port/keepalive，必要时生成随机 client id，保存 host/port/keepalive，重置 in/out inflight quota 和 retain capability。CONNECT 需要 Client Identifier、Keep Alive、Clean Start/Clean Session 等会话参数。
建立网络连接connect.c / mosquitto__reconnect()，net_mosq.c清理旧 in/out packet，调用 message__reconnect_reset()，关闭旧 socket，通过 net__socket_connect() 建立 TCP/TLS/SOCKS 连接。MQTT 会话建立前必须先有有序、可靠、双向的网络连接。
发送 CONNECTsend_connect.c / send__connect()根据协议版本写协议名、协议级别、Connect Flags、Keep Alive、MQTT v5 properties、client id、will、username、password。CONNECT 是客户端在网络连接建立后发给 broker 的第一个 MQTT 控制报文。
处理 CONNACKhandle_connack.c / handle__connack()读取 connect flags 和 reason code；MQTT v5 读取 properties；保存 Assigned Client Identifier、Retain Available、Maximum QoS、Receive Maximum、Server Keep Alive、Maximum Packet Size；成功后设置状态为 mosq_cs_active 并触发 connect 回调。CONNACK 决定连接是否被接受，也把 broker 能力反馈给客户端。
第一性原理：连接不是“socket 成功”就完成。MQTT 客户端只有在收到成功 CONNACK，并把 broker 能力写回上下文后，才进入可发布/订阅的 active 状态。

事件循环与网络 IO

loop.c 是库内部的驱动器。它不理解业务语义，只负责何时读、何时写、何时检查 Keep Alive、何时触发断线回调或重连。

函数作用关联文件
mosquitto_loop()建立 read/write fdset；如果 out packet 存在则监听写；如果 socketpair 有数据则唤醒写；select/pselect 后调用 mosquitto_loop_read()、mosquitto_loop_write()，最后调用 mosquitto_loop_misc()。packet_mosq.c、net_mosq.c、socks_mosq.c、tls_mosq.c
mosquitto_loop_read()按 in/out QoS 队列长度估算本轮处理包数；TLS 仍在握手时先推进 TLS；普通路径进入 packet__read() 或 SOCKS 读状态机。packet_mosq.c、read_handle.c、socks_mosq.c
mosquitto_loop_write()多次调用 packet__write()，处理 EAGAIN/EWOULDBLOCK 和错误回调。packet_mosq.c、net_mosq.c
mosquitto_loop_misc()调用 mosquitto__check_keepalive()。util_mosq.c、send_mosq.c
mosquitto_loop_forever()在循环出错后区分致命错误和可重连错误，按 reconnect delay/backoff 睡眠并调用 mosquitto_reconnect()。connect.c、loop.c
当前业务 mqtt.c 自己实现了一个类似 loop_forever 的循环，周期调用 mosquitto_loop() 并自行做重连退避。lib 内部原生 mosquitto_loop_forever() 也存在，但当前业务没有直接使用它。
第一性原理：事件循环的价值是把异步网络变成可重复的小步：能读就读，能写就写，没事件也要检查时间驱动的协议义务。

发布与 QoS

PUBLISH 是 MQTT 的核心报文。lib 目录把它拆成三层：actions.c 接收应用调用，messages_mosq.c 管 QoS 状态，send_publish.c/handle_publish.c/handle_pub*.c 编解码协议。

协议场景发送端实现接收端实现状态/回调
QoS 0actions.c 校验后直接 send__publish()；send_publish.c 设置 command 中 DUP/QoS/RETAIN 位并写 topic/payload。handle_publish.c 解析 topic/payload 后立即调用 on_message，无 ACK。packet__write() 写完 QoS0 PUBLISH 时触发 on_publish。
QoS 1生成非零 mid，messages_mosq.c 将消息设为 mosq_ms_wait_for_puback 后发送 PUBLISH。收到 PUBLISH 后 send__puback()，再触发 on_message。handle_pubackcomp.c 收到 PUBACK 后 message__delete() 并触发 on_publish。
QoS 2先发 PUBLISH 并等待 PUBREC；收到 PUBREC 后 message__out_update(...wait_for_pubcomp) 并 send__pubrel()。收到 PUBLISH 后发送 PUBREC，保存入站消息为 mosq_ms_wait_for_pubrel；收到 PUBREL 后发送 PUBCOMP，再交给 on_message。handle_pubackcomp.c 收到 PUBCOMP 后删除出站消息并触发 on_publish。
第一性原理：QoS 的本质是“所有权转移何时完成”。QoS0 把所有权交给网络，QoS1 交给 PUBACK，QoS2 交给 PUBREC/PUBREL/PUBCOMP 协议状态机。

订阅与退订

文件订阅路径协议语义
actions.cmosquitto_subscribe_multiple() 校验 qos、MQTT v5 options、属性、topic filter、UTF-8 和 packet size；然后调用 send__subscribe()。mosquitto_unsubscribe_multiple() 对退订 topic filter 做同类校验后调用 send__unsubscribe()。SUBSCRIBE payload 必须是一组 Topic Filter/QoS；UNSUBSCRIBE payload 必须至少包含一个 Topic Filter。
send_subscribe.c生成 mid，构造 CMD_SUBSCRIBE|2，写 Packet Identifier、MQTT v5 properties、每个 topic filter 和 requested QoS/options。SUBSCRIBE 固定头低 4 位必须是 0010。
send_unsubscribe.c构造 CMD_UNSUBSCRIBE|2，写 Packet Identifier、属性、topic filter 列表。UNSUBSCRIBE 固定头低 4 位也必须是 0010。
handle_suback.c读取 mid、MQTT v5 properties、granted QoS 列表，触发 on_subscribe 或 on_subscribe_v5。SUBACK 中 reason/granted QoS 的顺序对应 SUBSCRIBE 请求中的 Topic Filter 顺序。
handle_unsuback.c读取 mid 和 MQTT v5 reason code/properties，触发 unsubscribe 回调。UNSUBACK 确认退订请求处理完成，即使没有删除任何订阅，broker 也会响应。
第一性原理：订阅不是本地过滤表，而是向 broker 注册兴趣。客户端库只负责把兴趣表达成合法 Topic Filter 报文，并把 broker 授权结果回调给应用。

MQTT v5 属性

property_mosq.c 是 MQTT v5 支撑的核心文件。它把规范中的 Property Length、Property Identifier、不同类型属性值实现为链表，并负责合法性检查。

函数族作用参与报文
property__read_all()先读 property length varint，再循环读取单个 property，最后调用 mosquitto_property_check_all(command,...) 校验该属性是否允许出现在当前 command。CONNACK、PUBLISH、PUBACK/PUBREC/PUBREL/PUBCOMP、SUBACK、UNSUBACK、DISCONNECT、AUTH 等。
property__write_all()按链表顺序写属性；可选择是否写 property length。CONNECT、PUBLISH、SUBSCRIBE、UNSUBSCRIBE、DISCONNECT、ACK 类报文。
mosquitto_property_add_*对外 API 创建 byte/int16/int32/varint/binary/string/string pair 属性。应用构造 MQTT v5 properties 后传入 connect/publish/subscribe 等 API。
mosquitto_property_read_*从回调收到的属性链表中读取指定属性。应用处理 on_message_v5、on_connect_v5 等。
当前工程业务代码主要按默认 MQTT 3.1.1 使用，但静态库中已包含 MQTT v5 属性实现。文档中把它归为“归档存在、API 可用，但当前业务未重点使用”的能力。
第一性原理：MQTT v5 的扩展能力不是新增一套报文系统，而是在既有控制报文中增加“受 command 约束的键值字段”。所以属性模块必须同时懂类型编码和命令合法性。

Keep Alive 与重连

util_mosq.c

mosquitto__check_keepalive() 读取 last_msg_in、next_msg_out、ping_t。如果到期且处于 active 状态，先 send__pingreq()；如果已经等待 PINGRESP 仍超时，则关闭 socket 并触发 disconnect 回调。

send_mosq.c / handle_ping.c

send__pingreq() 发送 PINGREQ 并记录 ping_t；handle__pingresp() 收到 PINGRESP 后把 ping_t 清零。

loop.c

mosquitto_loop_misc() 每轮事件循环都会检查 Keep Alive；mosquitto_loop_forever() 在连接丢失后按 reconnect_delay、reconnect_delay_max、reconnect_exponential_backoff 执行重连。

connect.c / messages_mosq.c

重连前清理 packet，调用 message__reconnect_reset()。QoS1/QoS2 出站消息根据状态恢复为需要重发或重发 PUBREL，符合持久会话的未确认消息语义。

第一性原理：Keep Alive 是时间驱动的协议义务：没有业务数据时也必须证明连接仍活着；重连则是把未完成的可靠投递状态重新接回网络连接。

TLS/SOCKS/线程

能力文件实现要点当前配置状态
TLS 与证书校验options.c、net_mosq.c、tls_mosq.c、net_mosq_ocsp.c选项层保存 cafile/capath/cert/key/PSK/insecure/OCSP；网络层创建 SSL_CTX、加载 CA、做 TLS 握手；tls_mosq.c 做证书主机名匹配。WITH_TLSWITH_TLS_PSK
SOCKS5 代理socks_mosq.c、connect.c、loop.cmosquitto_socks5_set() 保存代理配置；连接时先连代理，再由 socks5__send/read() 推进 SOCKS5 握手状态。WITH_SOCKS
线程接口thread_mosq.c、mosquitto_internal.hmosquitto_loop_start() 创建内部线程，mosquitto_loop_stop() 停止，mosquitto_threaded_set() 标记外部线程使用模式；结构体中多个 mutex 保护状态、队列、mid、回调。WITH_THREADING
SRVsrv_mosq.c提供 SRV 连接入口，但只有定义 WITH_SRV 才有完整依赖路径。WITH_SRV=no
当前业务代码自建 MQTT 线程并循环调用 mosquitto_loop()，但官方 API 明确要求这种模式应调用 mosquitto_threaded_set()。这不是本页的代码修改范围，但属于 lib 组合方式分析中必须指出的线程契约。
第一性原理：TLS、代理、线程不是 MQTT 报文本身，但它们改变“字节如何安全到达”和“状态由谁并发访问”。协议正确性依赖这些横切能力不破坏底层有序字节流和状态一致性。

校验与工具文件

文件职责为什么协议需要它
utf8_mosq.cmosquitto_validate_utf8() 校验 MQTT UTF-8 字符串。Client Identifier、Topic Name、Topic Filter、User Property 等都受 UTF-8 与长度约束。
util_topic.c发布 topic 禁止 +/#，订阅 filter 校验通配符位置；还提供 topic/filter 匹配函数。MQTT 区分 Topic Name 与 Topic Filter，前者不能包含通配符，后者通配符必须在合法层级位置。
util_mosq.cmid 生成、Keep Alive、send/receive quota、随机字节、状态读写。Packet Identifier 必须非零且在未完成确认前不可复用；Receive Maximum/Send Maximum 需要配额控制。
memory_mosq.c统一封装 malloc/calloc/realloc/free/strdup。属性链表、消息队列、packet payload 都是动态内存，统一封装便于错误处理和可移植性。
logging_mosq.c、strings_mosq.c、misc_mosq.c、helpers.c日志、错误码/原因码字符串、消息辅助函数、一次性订阅 helper 等。不直接改变协议状态机，但为诊断、API 体验和辅助用法提供支撑。
Topic Alias当前 MOSQ_OBJS 没有独立 alias_mosq.o；Topic Alias 属于 MQTT v5 属性语义，应从 property_mosq.c 与 PUBLISH handler 的当前代码路径判断。对象边界优先于目录想象；不能把未归档对象写成当前静态库能力模块。
第一性原理：协议实现的可靠性常常来自“边缘校验”：长度、UTF-8、通配符、mid、quota 任一处放松，最终都会变成线上报文错误或状态机错乱。

第一性原理：文件索引的价值不是记住文件名，而是能从任一协议现象反推控制点：连接看 connect/send_connect/handle_connack，字节问题看 packet/net，QoS 问题看 messages/handle_pub。

总结

协议层

packet_mosq.c 与 packet_datatypes.c 保证 MQTT 控制报文按规范线格式进出 socket。

状态层

struct mosquitto、messages_mosq.c、util_mosq.c 保存连接状态、mid、quota、inflight 和 Keep Alive。

语义层

send_*.c 与 handle_*.c 把 CONNECT/PUBLISH/SUBSCRIBE/PING/DISCONNECT/AUTH 的协议语义落到具体函数。

所以，lib 目录完成 MQTT 协议的方式可以概括为：公开 API 把应用动作写入 struct mosquitto；发送文件把动作编码为 MQTT 控制报文；事件循环把报文写入网络并读取响应；接收 handler 按报文类型更新状态机和触发回调；工具文件保证 topic、UTF-8、属性、mid、quota、TLS 与线程边界不破坏协议契约。

第一性原理：一个 MQTT 客户端库不是“会发 PUBLISH”的函数集合，而是一台小型协议机器：它必须在不可靠的时间、分片的字节流、重连和并发调用之间，持续维护同一个会话状态。
