struct mosquitto 的结构设计背景与多锁设计

本文结合 MQTT 协议和当前 Mosquitto 2.0.14 源码，解释 mosquitto_internal.h 中 struct mosquitto 为什么聚合了网络连接、会话状态、收发包、QoS 队列、TLS、回调和线程同步字段，以及为什么 WITH_THREADING 下使用多把锁而不是一把全局大锁。

核心结论：struct mosquitto 是 libmosquitto 的“客户端连接上下文”。它不是普通 DTO，而是 MQTT Client 在一个会话中的运行时对象：既要满足协议对 Session、Packet Identifier、QoS、Keep Alive、重连、订阅确认的状态要求，又要作为 Reactor loop 线程和应用线程之间的共享边界。
第一性原理：对象结构来自协议不变量。MQTT 要求客户端持续维护连接、会话、未确认消息、包序号和心跳，所以实现必须有一个能跨 API 调用保存这些不变量的上下文对象。

协议背景：为什么客户端需要一个大上下文

协议要求官方依据落到 struct mosquitto 的字段
Client 打开 Network Connection，并 publish、subscribe、unsubscribe、disconnect。MQTT 5.0 Terminology 对 Client 的定义；MQTT 3.1.1 Terminology 同样说明 Client 建立网络连接并执行发布/订阅/取消订阅/断开。sock、address、host、port、bind_address、state。
Session 是 Client 与 Server 的有状态交互，可跨多个 Network Connection。MQTT 5.0 4.1 Session State；MQTT 3.1.1 4.1 Storing state。id、clean_start、session_expiry_interval、session_expiry_time、msgs_in、msgs_out。
QoS 1/2 需要保存未确认 PUBLISH/PUBREL，收到 ACK 后才能释放 Packet Identifier。MQTT 5.0 2.2.1 Packet Identifier、4.3 QoS flows、4.4 retry；MQTT 3.1.1 2.3.1、4.3、4.4。last_mid、struct mosquitto_msg_data msgs_out、struct mosquitto_msg_data msgs_in。
Keep Alive 要求在没有其他控制包时发送 PINGREQ，并监控 PINGRESP/输入输出时间。MQTT 5.0 3.1.2.10 Keep Alive；MQTT 3.1.1 3.1.2.10 Keep Alive。keepalive、last_msg_in、next_msg_out、ping_t。
MQTT 5 增加 Receive Maximum、Maximum Packet Size、Topic Alias、Maximum QoS、Retain Available 等连接级协商。MQTT 5.0 CONNECT/CONNACK properties 与 4.9 Flow Control。maximum_packet_size、aliases、alias_count、max_qos、retain_available、inflight_quota。
协议运行在有序、无损、双向字节流上，客户端需要拆包/组包。MQTT 5.0 Network Connection；MQTT 3.1.1 4.2 Network Connections。in_packet、current_out_packet、out_packet、out_packet_last、out_packet_count。
第一性原理：协议是状态机，不是一次函数调用。只要协议要求“发送后等待 ACK”“断线后重发”“按心跳时间做动作”，实现就必须把这些状态保存到一个长期存在的上下文。

结构形状：一个客户端对象承载五类状态

第一性原理：上下文对象要把“协议状态”和“程序边界”放在一起：协议字段保证 MQTT 正确，回调和锁保证 C API 能被应用安全驱动。

生命周期：创建、重置、运行、销毁

阶段源码行为设计含义
创建mosquitto_new() 分配对象后调用 mosquitto_reinitialise()。把 API 可见的 client handle 转成内部完整上下文。
重置mosquitto_reinitialise() 先 mosquitto__destroy()，再 memset，设置默认协议 mosq_p_mqtt311、socket、keepalive、状态、QoS quota、回调指针、线程状态，并初始化 mutex。同一个对象可以复用，但复用动作是“先清理再重新构造”，不是运行期任意重写字段。
运行应用 API 构造 packet/message，loop 线程执行 mosquitto_loop()、packet__write()、handle__packet()、mosquitto_loop_misc()。运行期核心是 Reactor：一个 loop 推进网络事件，其他 API 把工作投递进共享队列。
销毁mosquitto__destroy() 在内部 loop 线程模式下可能 cancel/join，然后关闭 socket、清理消息、Will、TLS、属性、packet，并销毁 mutex。销毁是生命周期边界，不是普通并发 API。其他线程仍访问对象时销毁，锁无法兜底。
第一性原理：mutex 保护运行期字段一致性，不保护对象生死。对象生命周期必须由调用者建立更外层的所有权规则。

字段分组：每组字段对应一个协议问题

字段组代表字段协议/实现问题是否运行期共享
网络连接sock、sockpairR/W、host、port、bind_address、want_write、want_connectMQTT 运行在有序双向字节流上；TLS pending 和 socketpair 唤醒都属于 loop 事件源。是，loop 线程和应用线程都可能影响写需求或唤醒。
身份与认证id、username、password、TLS/PSK/ALPN 字段CONNECT 需要 ClientID、用户名密码、TLS 配置；ClientID 与 Session State 绑定。主要连接前配置，不应作为任意并发可写字段。
会话与连接状态state、clean_start、session_expiry_interval、reconnect_delayCONNECT/CONNACK/DISCONNECT/重连决定会话是否延续、是否处于连接中。state 是运行期共享；其他多为配置/策略。
协议时间keepalive、last_msg_in、next_msg_out、ping_tKeep Alive 要求按时间发送 PINGREQ，并在超时后关闭连接。last_msg_in/next_msg_out 共享并加锁；ping_t 由 loop 路径主导。
Packet 流in_packet、current_out_packet、out_packet、out_packet_last、out_packet_countTCP 是字节流，读包/写包可能分片；出站控制包要排队保持顺序。出站队列和当前包共享；入站包通常由 loop 线程独占。
QoS 会话队列msgs_in、msgs_out、last_midPacket Identifier、QoS1/2 ACK、Receive Maximum、重连重发都需要未完成消息集合。是，多线程 publish 与 ACK handler 并发。
MQTT 5 能力maximum_packet_size、aliases、max_qos、retain_availableCONNACK 中服务器能力限制影响后续 PUBLISH、Topic Alias、包大小。多由连接协商后 loop 路径更新/读取，应用不应随意并发修改。
应用边界userdata、in_callback、各类 on_* 回调libmosquitto 通过回调把协议事件转交给应用。是，setter 与 loop 触发回调并发。
第一性原理：字段不是随意堆叠，而是协议职责分层后的结果：连接、身份、会话、时间、包、QoS、能力、应用出口。

状态机落地：为什么有这么多状态枚举

enum mosquitto_client_state 覆盖了从 mosq_cs_new、mosq_cs_connect_pending、mosq_cs_active、mosq_cs_connected 到 mosq_cs_disconnecting、mosq_cs_disconnected 的连接生命周期，还包含 SOCKS5、认证、重认证、broker 上下文复用等状态。

状态枚举的设计收益是把协议合法性检查从“散落的布尔判断”变成“显式状态”。例如 packet__write() 写包前读取 state，如果仍在 connect pending 或 TLS connect 阶段，就不继续写应用层 packet。

第一性原理：协议正确性依赖状态可判定。枚举状态比多个布尔变量更容易表达“当前允许哪些控制包、哪些行为必须延后”。

队列与 QoS：为什么既有 packet 队列又有 message 队列

结构保存什么生命周期协议意义
struct mosquitto__packet已经编码好的 MQTT Control Packet 字节流，包括 command、remaining length、payload、pos、to_process。进入 out_packet 后等待写 socket；写完即释放。处理 TCP 非阻塞写和 MQTT 控制包顺序。
current_out_packet当前正在写的 packet。可能跨多次 net__write()；EAGAIN 时保留写入进度。保证一个 MQTT packet 不会被写到一半就丢失上下文。
msgs_out.inflightQoS1/2 出站 application message、state、dup、timestamp、properties。从 publish 后直到 PUBACK/PUBREC/PUBCOMP 完成。满足“未确认前保留消息、重连时按原 Packet Identifier 重发”。
msgs_in.inflight入站 QoS2 消息状态。收到 PUBLISH QoS2 到 PUBREL/PUBCOMP 流程完成。满足 QoS2 不重复交付与 ACK 流程。
packet 是“线上的字节包”，message 是“协议会话里的未完成业务消息”。两者生命周期不同，所以拆成两个结构。QoS0 通常只需要 packet；QoS1/2 还需要 message state。
第一性原理：网络发送缓冲解决“怎么把字节写出去”，会话消息队列解决“协议是否已经完成”。这两个不变量不同，不能混成一个队列。

回调与应用边界

libmosquitto 是 C 库，不能把协议事件主动推给上层对象系统，所以 struct mosquitto 保存 userdata 和一组 on_* 函数指针。loop 线程收到 CONNACK、PUBLISH、SUBACK、PUBACK、DISCONNECT 等协议事件后，通过这些回调把事件交给应用。

回调字段协议事件为什么放在 struct mosquitto
on_connect / on_connect_v5CONNACK。连接结果属于该客户端实例；回调需要带回该实例和 userdata。
on_message / on_message_v5入站 PUBLISH。消息交付依赖该实例的订阅、协议版本、properties 处理。
on_publish / on_publish_v5QoS0 发送完成或 QoS1/2 ACK 完成。mid 和 QoS 状态来自该实例的出站消息队列。
on_subscribe / on_unsubscribeSUBACK / UNSUBACK。ACK 的 Packet Identifier 对应该实例发出的订阅控制包。
on_log库内部日志。日志需要携带实例 userdata，但不应与业务回调用同一把锁。
第一性原理：回调是协议事件到应用代码的出口。出口必须挂在客户端上下文上，才能保证事件、userdata、协议状态属于同一个连接实例。

为什么使用很多锁，而不是一把大锁

减少无关竞争

应用线程设置 log callback 不应阻塞 loop 线程处理 QoS ACK；生成 mid 不应阻塞出站 packet 写 socket。多锁让独立不变量独立竞争。

锁保护范围清晰

out_packet_mutex 只保护出站链表，mid_mutex 只保护 last_mid，state_mutex 只保护连接状态。调试时能快速定位哪个不变量被破坏。

贴合协议并行性

MQTT 允许同时存在多个不同 Packet Identifier 的未确认交换；应用线程也可能同时 publish。细粒度锁支持这种并行，而不是把所有 API 串行化。

避免长临界区

网络 I/O、select、TLS pending、用户回调都可能慢。多锁使大多数临界区只覆盖链表修改、计数更新、指针读取这些短操作。

多锁不是越多越好。这里能成立，是因为 Mosquitto 的共享数据天然按协议不变量分域：packet 队列、QoS 队列、mid、state、msgtime、callback。每把锁都有明确数据域，且嵌套很少。
第一性原理：锁应该保护不变量，而不是保护代码块。不同不变量用不同锁，可以降低竞争，并让死锁分析变成锁图分析。

锁保护表：每把锁保护什么

锁保护对象为什么不能共享一把锁典型临界区长度
callback_mutex业务回调指针和回调执行期的 in_callback。业务回调和日志回调语义不同；业务回调可能触发 publish，不应被日志路径拖住。相对长，因为覆盖用户回调调用。
log_callback_mutexon_log 指针与日志回调。日志可能来自很多路径，独立锁避免与业务事件互相阻塞。取决于用户日志回调。
state_mutexstate。状态读取高频且只是一枚枚举，不应等待队列锁或回调锁。极短。
mid_mutexlast_mid。Packet Identifier 生成是独立计数器，使用专锁能允许并发 publish/subscribe 快速拿号。极短。
msgtime_mutexlast_msg_in、next_msg_out。Keep Alive 判断和收发包时间更新不应阻塞 QoS 队列操作。极短。
out_packet_mutexout_packet、out_packet_last、out_packet_count。保护生产者-消费者队列；与 QoS message 队列生命周期不同。短，链表入队/出队。
current_out_packet_mutexcurrent_out_packet 及当前写包所有权。当前写包跨多次非阻塞写，需要与待发送队列分离。可能覆盖一次 packet__write() 写循环。
msgs_in.mutex入站 QoS message 链表、长度、quota。入站 QoS2 与出站 QoS1/2 是两个协议方向，Packet Identifier 空间也独立。短到中等，链表与状态更新。
msgs_out.mutex出站 QoS message 链表、长度、quota、message state。出站 publish 与 ACK handler 会并发访问，必须与入站队列分开以降低竞争。短到中等，链表与状态更新。
第一性原理：一把锁对应一个被保护的不变量集合。保护集合越小，等待关系越少，死锁图越简单，性能退化越可控。

如何保证这些锁不会造成死锁

这里的“保证”不是语言层面的自动保证，而是源码通过锁域拆分、固定嵌套顺序、短临界区和调用约束降低死锁条件。死锁需要同时满足互斥、持有并等待、不可抢占、循环等待。Mosquitto 内部主要破坏的是“循环等待”和“持有并等待”。

死锁条件源码中的规避方式仍需调用者遵守的边界
循环等待绝大多数锁不嵌套；出站 packet 相关嵌套统一为 current_out_packet_mutex 后 out_packet_mutex，没有反向加锁路径。新增代码不能反向持有 out_packet_mutex 再等待 current_out_packet_mutex。
持有并等待packet__queue() 入队后释放 out_packet_mutex，再写 socketpair 唤醒；mosquitto_loop() select 前只短暂检查队列，不持锁进入 select。应用不能在外部持有自己的业务锁后调用可能回调再反拿业务锁的逻辑。
长时间阻塞大多数 mutex 只保护链表指针、计数、枚举、时间戳；线程 join/cancel 不在这些内部 mutex 保护区中执行。callback_mutex 覆盖用户回调，用户回调必须短小，避免调用会等待同一回调锁的 setter。
跨线程等待mosquitto_loop_stop() 用 socketpair 唤醒 loop，再 pthread_join()；不是持 packet/message 锁等待线程退出。停止/销毁必须在对象生命周期外层串行化，不能让其他线程继续访问已销毁对象。
隐藏共享运行期高频共享字段都有专锁：state、mid、msgtime、packet queue、message queue、callback。配置类字段多数要求 connect 前设置，不应被当作任意并发可写状态。
重要边界：Mosquitto 内部锁图可以避免库内部常规路径形成死锁，但不能保证任意用户回调重入都安全。特别是回调执行期间持有 callback_mutex，如果回调里调用会再次获取同一锁的 callback setter，NORMAL mutex 会自死锁。
第一性原理：死锁来自等待环。多锁系统要能安全运行，必须让锁图接近 DAG：锁域独立，必要嵌套有固定方向，等待线程退出或阻塞 I/O 不发生在内部锁持有期间。

总结

结构背景

struct mosquitto 是 MQTT Client 的运行时上下文，承载连接、会话、QoS、Packet Identifier、Keep Alive 和回调。

多锁收益

多锁把不同协议不变量拆开保护，降低无关竞争，缩短临界区，并让应用线程和 loop 线程通过队列协作。

死锁边界

内部依赖锁域拆分、固定嵌套顺序、短临界区和 socketpair 唤醒避免等待环；用户回调和生命周期仍需遵守调用约束。

从设计模式看，struct mosquitto 是 Reactor 上下文对象：网络 loop 独占协议推进，应用 API 只生产动作或设置回调。多把锁不是为了让所有字段任意并发改写，而是让必须跨线程共享的少数字段各自拥有清晰保护者。

第一性原理：好的并发结构不是“共享一个大对象再加一把大锁”，而是先按协议不变量拆分状态，再只给真正跨线程读写的状态配最小必要同步。
