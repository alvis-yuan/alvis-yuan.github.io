打印四大领域术语：协议、排版、图像与硬件心智模型

打印模块里的术语多，不是因为代码故意复杂，而是因为一次打印跨过四个知识域：协议负责理解字节，排版负责确定内容放在哪里，图像负责把内容变成点阵，硬件负责让热敏头、电机、切刀和传感器执行动作。

一条最短主线：外部输入一串字节；协议层把它解释为命令和数据；排版层把文字、条码、图片放到行或页面上；图像层把这些内容变成每一行的点阵；硬件层把点阵写给驱动并读取状态。
第一性原理：打印的最终目标是控制热敏头的每个点在正确时间发热。所有术语都可以追问：它是在决定“字节是什么意思”、“内容在哪里”、“哪些点发热”，还是“硬件是否能执行”。

顶层心智模型

领域回答的问题本仓库主要代码
协议这串字节代表什么命令？参数多长？是普通命令还是实时命令？Instruct_Proc.c、*_Instruct.c、Realtime.c、TSPL/tspl_parser.c
排版文字、条码、图片应该放在哪里？什么时候换行？标准模式还是页模式？Runtime_Data.c、CharBuffer.c、WordSet.c、HarfBuzz.c、LineBuffer.c、PageBuffer.c
图像内容如何变成黑白点阵或灰度点阵？位图/二维码/条码如何落到点行？GS_Instruct.c、Util.c、LineBuffer.c、PageBuffer.c
硬件点阵如何写给驱动？纸张、开盖、过热、切刀、缓冲满怎么处理？printer.c、printer.h、print_queue.c
第一性原理：每个领域的输出都是下一个领域的输入。哪里输出不清晰，哪里就会出现“术语听懂了但代码读不动”的感觉。

领域交接物

最容易混淆的是“数据”和“数据的意义”。同一个字节在协议层是 opcode，在排版层可能变成字符，在图像层可能变成像素，在硬件层只剩发热点。

交接点传递的东西例子
外部入口 -> 协议原始字节流1B 40、1D 56 00、UTF-8 文本、TSPL 文本命令。
协议 -> 排版语义事件和运行时设置初始化、设置行距、选择字体、追加字符、换行。
排版 -> 图像版面上的对象和缓冲当前行字符点阵、页模式画布、下划线、对齐结果。
图像 -> 硬件点阵行数据BYTES_PER_LINE 宽度的 mono 数据，或灰度数据。
硬件 -> 调度/应用状态快照打印中、缺纸、开盖、过热、纸未取、切刀状态。
第一性原理：跨领域调试时，先问“我现在看到的是哪一种数据形态”。原始字节、语义命令、版面对象、点阵行、硬件状态不能混为一谈。

协议领域

协议领域关注“字节是什么意思”。Epson ESC/POS 官方把命令按代码顺序和功能分类；命令格式里常见 n、nL/nH、pL/pH 等参数记法。本项目普通 ESC/POS 走 Cmd_Proc() 状态机，实时命令走 Realtime_Proc()，标签模式走 TSPL parser。

输入

USB/TCP/BLE/SPP/业务侧传入的字节流，可能是 ESC/POS、TSPL、私有命令、普通文本或实时状态命令。

要完成的工作

识别命令前缀，计算参数长度，收集可变 payload，处理模式差异，决定要更新设置、输出字符、打印图像、回复状态还是执行控制动作。

输出

对排版层的调用、对图像层的调用、对硬件层的控制请求、对主机的回复数据。

读代码入口

Cmd_Start()、Cmd_Proc()、ESC_Proc()、GS_Proc()、Realtime_Proc()、tspl_parser()。

术语解释记忆方式
ESC/POSPOS 打印机命令体系。它不是文件格式，而是持续到来的控制字节和数据字节。把它当作“打印机能听懂的字节语言”。
TSPL标签打印语言，常用文本命令描述标签尺寸、坐标、文本、条码和打印份数。像“描述一张标签页面的脚本”。
Opcode命令码。决定当前命令是哪一种操作。ESC @ 中 @ 是 opcode。
Prefix前缀字节，表示后面进入某个命令族。ESC、FS、GS、DLE。
n1 字节参数，范围通常是 0 到 255。一个小旋钮。
nL/nH低字节/高字节组成的 16 位数值。两个字节拼成一个位置或长度。
pL/pH表示后续参数长度的低/高字节。先告诉你后面还有多少数据。
fnfunction number，多功能命令里的子功能号。GS ( k 下二维码、PDF417 都靠 fn 区分。
Realtime command实时命令，要求接收时尽快处理，常用于状态查询。控制面插队，但不能破坏普通命令。
State machine状态机。当前状态决定下一个字节怎么解释。pNextFun 就是“下一字节处理函数”。
第一性原理：协议领域的目标不是打印，而是消除歧义：当前字节到底是字符、命令码、长度、参数，还是图片 payload。

排版领域

排版领域关注“内容放在哪里”。它处理字体、编码、字号、对齐、旋转、行距、页模式、方向、下划线、反白等概念。ESC/POS 里的很多命令并不直接打印，而是修改这些运行时设置。

输入

协议层传来的字符、换行、设置命令、定位命令、页模式命令，以及当前 Settings。

要完成的工作

查字库，把编码变成字形点阵；计算字符宽高、行高、间距、对齐；把内容放入 LineBuffer 或 PageBuffer。

输出

已经排列好的行缓冲或页面缓冲，等待图像/硬件层输出点阵。

读代码入口

Char_Proc()、ASC_Proc()、Unicode_Proc()、AppendToLineBuffer()、lineBuffer->printLine()。

术语解释代码直觉
Code Page代码页。单字节编码到字符的映射表。同一个 0x80 在不同代码页可能是不同字符。
Unicode统一字符编码。描述“字符是什么”。UTF-8 最终会转成 Unicode 再找字形。
CJK中文、日文、韩文。常涉及多字节编码和专用字库。FS 命令常和 Kanji/CJK 有关。
Glyph字形。字符在某个字体中的具体外观。“A” 是字符，字体里的 A 图形是 glyph。
HarfBuzz文本 shaping 库，处理复杂文字、连字、从右到左排版等。不是翻译文字，而是决定字形组合方式。
Baseline基线。文字排版时字符对齐的参考线。不同高度字符要在同一行看起来齐。
Line spacing行距。打印一行后额外走纸的距离。ESC 2、ESC 3 类命令会修改。
Justification对齐方式。左对齐、居中、右对齐。ESC a 修改。
Standard mode标准模式。按行顺序打印小票。大多数收据打印走这个模式。
Page mode页模式。先在一个页面区域内定位，再统一输出。需要坐标和页面缓冲。
LineBuffer行缓冲。一行内容的临时画布。文字、下划线、反白、对齐最终汇到这里。
PageBuffer页缓冲。页模式下的二维画布。先画页面，再整体打印。
第一性原理：排版领域的目标是把“内容”变成“有位置的内容”。如果还没确定位置和尺寸，就还不能直接驱动热敏头。

图像领域

图像领域关注“哪些点应该黑”。文字排版完成后会变成点阵；条码、二维码、位图、NV 图像、下载图像也都要变成点阵。对热敏打印机来说，最终不是打印“图片文件”，而是打印一行行黑白或灰度点。

输入

排版层输出的行/页缓冲、ESC/POS 位图命令 payload、条码/二维码数据、TSPL 页面对象。

要完成的工作

栅格化、缩放、旋转、裁剪、二值化或灰度转换，把对象整理成驱动能接受的点阵行。

输出

BYTES_PER_LINE 宽度的点阵行，或多行点阵块，交给 PrnDotLine()。

读代码入口

LineBuffer.c 的 print()，GS_Instruct.c 中 bit image/raster image/barcode/QR 相关处理，Util.c。

术语解释代码直觉
Dot打印头上的一个发热点，对应纸面上的一个点。最终控制对象。
DPIDots Per Inch，每英寸点数。决定物理尺寸和像素数换算。
Raster栅格，按行排列的像素/点阵数据。GS v 0 是典型栅格位图命令。
Bit image位图，通常 1 bit 表示一个点黑或白。节省空间，适合热敏黑白打印。
Grayscale灰度。一个点不只是黑/白，可有深浅。PRN_DATA_GRAYSCALE。
Mono单色黑白点阵。PRN_DATA_MONO。
Dithering抖动。用黑白点模拟灰度效果。图片打印常见预处理。
Barcode一维条码。GS k、高度/宽度/HRI 相关命令。
QR Code / 2D Code二维码/二维条码。GS ( k 多功能命令常处理。
HRIHuman Readable Interpretation，条码下方可读文本。条码领域术语，但会影响排版。
NV graphics非易失存储里的图像。存到设备里，下次可按 key code 打印。
Download graphics下载到工作区的图像。通常掉电或重置后可能失效。
第一性原理：图像领域的目标是把“看起来应该是什么”变成“每一行哪些 bit 是 1”。所有图片、字体、条码最终都要服从打印头宽度和点阵格式。

硬件领域

硬件领域关注“设备能否按要求动作”。它处理热敏头、电机、传感器、切刀、钱箱、驱动缓冲、温度、电压、缺纸、开盖等问题。本仓库通过 PrnDotLine() 写点阵，通过 PrnIoctl() 做控制，通过 PrnGetStatus() 读取状态。

输入

图像层生成的点阵行、控制命令、模式设置、切刀/走纸/钱箱动作请求。

要完成的工作

写内核驱动，处理缓冲满，执行 ioctl，采样状态，把底层状态映射成共享内存状态。

输出

纸面打印结果、纸张移动、切刀动作、钱箱脉冲，以及 print_status_t 状态快照。

读代码入口

PrnDotLine()、write_to_kernel()、PrnIoctl()、PrnGetStatus()、PrnStatusToShmStatus()。

术语解释代码直觉
Thermal head热敏打印头，靠加热点让热敏纸显色。最终由点阵决定哪些点发热。
Stepper motor步进电机，控制走纸。行距、走纸、回退都要靠它。
Cutter切刀。PRN_CMD_CUT、GS V。
Platen胶辊/压纸结构，开盖会影响打印。platen_opened 状态。
Paper sensor纸张传感器。缺纸、纸将尽、黑标/标签定位。
Black mark黑标纸上的定位标记。标签/票据定位用。
Gap标签间隙。TSPL 标签定位常见术语。
Buffer full内核或设备缓冲满。PRN_BUF_FULL，需要等待或暂停。
ioctlLinux 设备控制调用。不是写打印内容，而是发控制命令或查状态。
Embedded command写给底层驱动的内部命令。PRN_CMD_SEND_EMBEDDED_COMMAND。
Status snapshot状态快照。shm->status 保存给应用读。
Reprint重打。缺纸后根据任务和 buffer id 从头或从切刀点重发。
第一性原理：硬件领域的目标是让物理动作可靠完成。软件说“已经发完”不等于纸上已经打印完，必须看驱动状态和硬件进度。

输入工作输出对照

领域输入要完成的工作输出典型失败
协议原始字节流解析命令、参数、payload、实时命令语义事件误判长度、payload 被当命令、模式不匹配
排版语义事件、字符、设置编码转换、字体选择、定位、换行、页模式布局行/页缓冲乱码、错位、换行异常、字体缺失
图像行/页缓冲、图像/条码数据栅格化、缩放、裁剪、二值/灰度处理点阵行图片变形、宽度越界、二维码不可扫
硬件点阵行、控制命令写驱动、执行 ioctl、读传感器、异常恢复纸面结果和状态缺纸、开盖、过热、缓冲满、切刀异常
定位问题时不要跳层。例如“打印乱码”通常先看协议/编码/字库，不要先怀疑热敏头；“打印一半停住”通常先看硬件状态、缓冲满、缺纸和队列进度。
第一性原理：每类 bug 都有天然归属。先归域，再查文件，效率会高很多。

新人阅读法

  1. 从协议领域入手：读 Cmd_Start()，知道 ESC/POS 字节怎样进入 ESC/FS/GS/DLE。
  2. 选一个最短命令闭环：例如 LF，看它如何触发行打印和走纸。
  3. 再选一个排版命令：例如 ESC a 对齐或 ESC 3 行距，看它如何修改 Settings。
  4. 再选一个图像命令：例如 GS v 0 栅格位图，看 payload 如何变成 PrnDotLine()。
  5. 最后看硬件边界：PrnDotLine()、PrnIoctl()、PrnGetStatus()。
学习顺序不要从术语表开始背，而是拿一条真实数据流，把沿途遇到的词放回四个领域里。这样术语会自然成网，不会变成碎片。
第一性原理：术语是压缩过的经验。只有把它还原到输入、工作、输出，才真正理解。

总结

协议、排版、图像、硬件四个领域对应四次转换：字节到语义，语义到版面，版面到点阵，点阵到物理动作。理解这四次转换，就能把大量术语和文件名放到同一张地图里。
