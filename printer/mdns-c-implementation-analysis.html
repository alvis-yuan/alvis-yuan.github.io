mDNS C 开源项目深度解析 mjansson/mdns

⭐ github.com/mjansson/mdns 📄 RFC 6762 mDNS 📄 RFC 6763 DNS-SD

mjansson/mdns 是由 Mattias Jansson 于 2017 年编写并开放公有领域的 C 语言 mDNS/DNS-SD 库。 该库以 单文件头文件 方式实现，无任何外部依赖，跨平台支持 Linux、macOS、Windows， 基于 RFC 6762（Multicast DNS）和 RFC 6763（DNS-Based Service Discovery）。

特性说明
许可证公有领域（Public Domain）
实现方式Header-Only，仅 mdns.h 一个文件
内存分配库内部零 malloc，全部由调用方提供 buffer
协议标准RFC 6762 (mDNS) + RFC 6763 (DNS-SD)
网络协议IPv4 + IPv6 双栈
平台支持Linux / macOS / Windows
编程语言纯 C（C99 兼容，C++ extern "C" 友好）
功能范围DNS-SD 发现、mDNS 查询/响应、服务公告与注销
⚛ 第一性原理

网络上的设备如何在没有中心化 DNS 服务器的前提下互相找到对方？本质就是将 DNS 消息广播到局域网多播地址，让所有人都能"听到"并回应。mjansson/mdns 把这一协议需求翻译为最精简的 C 代码：一个头文件 + 调用方提供的 buffer = 完整的无服务器本地服务发现。

01 · RFC 协议基础

mDNS — RFC 6762 核心原理

传统 DNS 需要中心化服务器，而 mDNS 将查询发送到 链路本地多播地址， 网络内所有设备都能收到并各自决定是否响应。

⚛ 第一性原理

为何要有 .local. 域和多播地址这两个约定？根本原因是：任意两台设备要通信，必须先有个共同的"频道"。 固定的多播地址就是这个频道，.local. 域名是这个频道上的命名空间。 RFC 6762/6763 把"零配置服务发现"分解成：命名空间约定 + 多播频道 + 链式记录语义，三层各司其职。

⚛ 第一性原理

为什么只需要一个头文件？因为整个库的功能本质是：把字节数组按协议格式读写，并通过 UDP socket 发送或接收。 这两件事完全可以通过 static inline 函数内联到调用方，无需编译单元隔离，消除了链接依赖， 使库的集成成本趋近于零。

03 · 设计模式与哲学

设计模式总览

图4：六大设计模式

Header-Only 的实现机制

通过将所有函数声明为 static inline，编译器在每个包含 mdns.h 的编译单元中独立内联，避免了多重定义的链接错误：

// mdns.h 中所有实现函数均为 static inline
static inline int mdns_socket_open_ipv4(const struct sockaddr_in* saddr) {
    int sock = (int)socket(AF_INET, SOCK_DGRAM, IPPROTO_UDP);
    // ... 多播加入 + 绑定 + 非阻塞
    return sock;
}

static inline uint16_t mdns_ntohs(const void* data) {
    uint16_t aligned;
    memcpy(&aligned, data, sizeof(uint16_t));
    return ntohs(aligned); // 通过 memcpy 解决对齐问题
}
对齐安全性：直接指针转换（如 (uint16_t*)ptr）可能在 ARM 等严格对齐平台上触发总线错误。库通过 memcpy 中间变量来安全读写非对齐数据。

回调驱动与用户数据透传

所有解析结果通过统一的回调函数签名传递，void* user_data 是调用方状态的不透明载体：

typedef int (*mdns_record_callback_fn)(
    int sock,
    const struct sockaddr* from,  // 发送方地址
    size_t addrlen,
    mdns_entry_type_t entry,       // QUESTION/ANSWER/AUTHORITY/ADDITIONAL
    uint16_t query_id,
    uint16_t rtype,                // 记录类型: A/PTR/SRV/TXT/AAAA
    uint16_t rclass,
    uint32_t ttl,
    const void* data,              // 原始报文 buffer 指针
    size_t size,
    size_t name_offset,            // 记录名称在 buffer 中的偏移
    size_t name_length,
    size_t record_offset,          // 记录数据在 buffer 中的偏移
    size_t record_length,
    void* user_data                // 调用方自定义状态，原样透传
);
⚛ 第一性原理

为什么采用回调而非返回数据结构？因为 DNS 报文可以包含任意数量的记录， 流式回调消除了中间缓冲区的需求，parse 一条就 callback 一条， 既适配 zero-alloc 设计，也让调用方完全控制何时停止（返回非零值即停止解析）。

核心数据结构

// 字符串（非 NUL 结尾，持有指向 buffer 的引用）
struct mdns_string_t {
    const char* str;
    size_t      length;
};

// 字符串对（用于 DNS 压缩指针解析中间状态）
struct mdns_string_pair_t {
    size_t offset;  // 子串在 buffer 中的偏移
    size_t length;  // 子串长度
    int    ref;     // 是否为压缩指针引用
};

// 字符串压缩表（最多 16 个表项，环形使用）
struct mdns_string_table_t {
    size_t offset[16]; // 记录每个已写入子串的偏移位置
    size_t count;      // 当前记录数
    size_t next;       // 下一个写入位置（环形）
};

// SRV 记录解析结果
struct mdns_record_srv_t {
    uint16_t       priority;
    uint16_t       weight;
    uint16_t       port;
    mdns_string_t  name;  // 目标主机名
};

// 通用记录（含 Union 覆盖所有类型数据）
struct mdns_record_t {
    mdns_string_t     name;
    mdns_record_type_t type;
    union {
        mdns_record_ptr_t  ptr;
        mdns_record_srv_t  srv;
        mdns_record_a_t    a;
        mdns_record_aaaa_t aaaa;
        mdns_record_txt_t  txt;
    } data;
    uint16_t rclass;
    uint32_t ttl;
};
Union 的妙用：mdns_record_t 用 Union 覆盖所有记录类型的数据字段， 确保结构体大小只由最大成员决定，内存使用紧凑，且通过 type 字段区分活跃成员， 是 C 语言中实现"代数类型"（Tagged Union）的经典手法。
⚛ 第一性原理

类型系统的本质是：用编译器检查约束替代运行时错误。 枚举替代魔法数字，typedef 提供语义名称，Union 保证内存紧凑且类型安全。 mdns_string_t 持有指针而非拷贝字符串，使零分配成为可能—— 字符串活在 buffer 里，结构体只是"窗口"。

05 · DNS 报文结构

DNS 消息格式（RFC 6762 §18）

图5：DNS/mDNS 报文格式（字节级）

mdns_header_t 与报文对应关系

struct mdns_header_t {
    uint16_t query_id;       // 查询标识符（mDNS 通常为 0）
    uint16_t flags;          // 0x0000=查询, 0x8400=权威响应
    uint16_t questions;      // 问题区记录数
    uint16_t answer_rrs;     // 答案区记录数
    uint16_t authority_rrs;  // 权威区记录数
    uint16_t additional_rrs; // 附加区记录数
};
特殊位的 mDNS 语义：
  • QU bit（QCLASS bit 15）= 0x8000：请求单播响应（Unicast Response）
  • Cache-Flush bit（RCLASS bit 15）= 0x8000：通知接收方刷新缓存中的旧记录
  • Flags = 0x8400：QR(1) + AA(1)，表示这是权威响应包

DNS 名称在报文中以"标签序列"存储，每个标签前有1字节长度前缀。当长度字节高两位为 11 时，表示这是一个指向报文前部的压缩指针：

⚛ 第一性原理

DNS 报文为什么要压缩名称？因为同一 DNS 消息中同一域名会重复出现多次（问题区、答案区都有）。 压缩 = 用 2 字节指针替代重复的变长字符串。mDNS 多播报文更需要紧凑， 因为 UDP 报文有大小限制（通常 <1500 字节）。

多接口枚举（mdns.c 实现）

// Linux/macOS: 通过 getifaddrs 枚举所有网络接口
struct ifaddrs* ifaddr = 0;
getifaddrs(&ifaddr);
for (struct ifaddrs* ifa = ifaddr; ifa; ifa = ifa->ifa_next) {
    // 过滤：必须是 UP + MULTICAST，排除 LOOPBACK 和 POINTOPOINT
    if (!(ifa->ifa_flags & IFF_UP) || !(ifa->ifa_flags & IFF_MULTICAST)) continue;
    if ((ifa->ifa_flags & IFF_LOOPBACK) || (ifa->ifa_flags & IFF_POINTOPOINT)) continue;

    // 对每个有效接口分别开一个 socket（发送时每 socket 只能用一个接口）
    struct sockaddr_in* saddr = (struct sockaddr_in*)ifa->ifa_addr;
    saddr->sin_port = htons(port);
    int sock = mdns_socket_open_ipv4(saddr);
    if (sock >= 0) sockets[num_sockets++] = sock;
}
freeifaddrs(ifaddr);
设计意图：发送方每个 socket 只能绑定一个接口（IP_MULTICAST_IF）， 因此对多个接口发送需要多个 socket。接收方则相反——一个绑定到 INADDR_ANY 的 socket 即可从所有接口接收多播数据。
⚛ 第一性原理

多播的本质是：OS 网络栈负责把 UDP 包复制到所有订阅该组的接口上。 调用方只需"加入组"（IP_ADD_MEMBERSHIP），OS 就把多播包投递到 socket 的接收队列。 非阻塞模式 + select/poll 是事件驱动的最小实现——不需要线程，不需要异步框架。

07 · 字符串压缩算法

字符串表（String Table）设计

在构建 DNS 报文时，同一个域名标签（如 local、_tcp）会在多条记录中重复出现。 mdns_string_table_t 记录已写入的子串偏移，后续写入时优先使用 2 字节压缩指针替代重复内容： 

struct mdns_string_table_t {
    size_t offset[16]; // 环形表，记录最近 16 个子串的起始偏移
    size_t count;      // 已存储数量（最大 16）
    size_t next;       // 下一个写入槽位（环形推进）
};

图8：mdns_string_make 字符串写入流程

mdns_string_extract 解压缩流程

读取时遇到压缩指针需要递归跳转，库限制最大递归深度为 16 次，防止恶意数据造成无限循环：

static inline mdns_string_pair_t
mdns_get_next_substring(const void* rawbuffer, size_t size, size_t offset) {
    const uint8_t* buffer = (const uint8_t*)rawbuffer;
    int recursion = 0;

    // 跟随压缩指针（最多跳 16 次，防止循环）
    while (mdns_is_string_ref(buffer[offset])) {
        if (size < offset + 2) return pair; // 越界保护
        pair.ref = 1;
        if (++recursion > 16) return pair;  // 递归深度保护
        // 取低 14 位作为跳转偏移
        size_t ref = (size_t)((buffer[offset] & 0x3F) << 8) | buffer[offset + 1];
        offset = ref;
    }

    size_t length = buffer[offset++];
    pair.offset = offset;
    pair.length = length;
    return pair;
}
安全设计：
  • 每次访问前检查 offset < size，防止越界读
  • 最大子串数量限制 MDNS_MAX_SUBSTRINGS = 64
  • 压缩指针跳转最多 16 次，防止循环引用
⚛ 第一性原理

字符串压缩的本质是空间换时间的逆运算——时间换空间。 在构建时多花 O(n) 查表时间，换取报文字节数的减少。 环形 16 槽位表是对"常用后缀有限"这一事实的近似——.local.、._tcp. 等 后缀会被反复命中，16 个槽位在实践中足够。

08 · 记录解析引擎

解析管线总体流程

图9：记录解析管线（从 socket 到 callback）

核心解析函数：mdns_records_parse

static inline size_t
mdns_records_parse(int sock, const struct sockaddr* saddr, socklen_t addrlen,
                   const void* buffer, size_t data_size, size_t* offset,
                   mdns_entry_type_t type, uint16_t query_id, size_t records,
                   mdns_record_callback_fn callback, void* user_data) {
    size_t parsed = 0;
    for (size_t i = 0; i < records; ++i) {
        size_t name_offset = *offset;
        // 1. 跳过/解析压缩域名（记录偏移以便 callback 调用 extract）
        mdns_string_skip(buffer, data_size, offset);

        size_t name_length = (*offset) - name_offset;
        const uint16_t* data = (const uint16_t*)MDNS_POINTER_OFFSET(buffer, *offset);

        // 2. 按网络字节序读取 TYPE CLASS TTL RDLENGTH（共10字节）
        uint16_t rtype  = mdns_ntohs(data++);
        uint16_t rclass = mdns_ntohs(data++);
        uint32_t ttl    = mdns_ntohl(data); data += 2;
        uint16_t length = mdns_ntohs(data++);
        *offset += 10;

        // 3. 边界检查
        if (length > (data_size - *offset)) return parsed;

        // 4. 触发回调，传入原始 buffer 和偏移（不拷贝数据）
        ++parsed;
        if (callback && callback(sock, saddr, addrlen, type, query_id,
                                 rtype, rclass, ttl, buffer, data_size,
                                 name_offset, name_length,
                                 *offset, length, user_data))
            return parsed; // 回调返回非 0 则提前终止

        *offset += length; // 推进游标到下条记录
    }
    return parsed;
}

各类型记录解析函数

函数输入输出关键操作
mdns_record_parse_ptr buffer, offset, length mdns_string_t 调用 mdns_string_extract 解压缩域名
mdns_record_parse_srv buffer, offset, length mdns_record_srv_t 读 priority(2B) + weight(2B) + port(2B) + 解压主机名
mdns_record_parse_a buffer, offset, length struct sockaddr_in* length==4 时 memcpy 4 字节 IPv4
mdns_record_parse_aaaa buffer, offset, length struct sockaddr_in6* length==16 时 memcpy 16 字节 IPv6
mdns_record_parse_txt buffer, offset, length mdns_record_txt_t[] 循环解析 len + key=value 键值对
⚛ 第一性原理

解析器的核心约束是：不能分配内存，不能修改 buffer。 因此所有解析结果要么是值类型（integer），要么是持有 buffer 内部指针的 mdns_string_t。 游标（offset）在 buffer 上线性滑动，每解一条记录向前推进一段——这是零拷贝流式解析的本质。

                 size_t name_offset, size_t name_length,
                 size_t record_offset, size_t record_length,
                 void* user_data) {

    // 只处理问题（入站查询），忽略其他区段
    if (entry != MDNS_ENTRYTYPE_QUESTION) return 0;

    // 从 buffer 解压记录名称
    mdns_string_t name = mdns_string_extract(
        data, size, &name_offset, namebuffer, sizeof(namebuffer));

    // 判断是否为 DNS-SD 全服务枚举查询
    uint16_t unicast = (rclass & MDNS_UNICAST_RESPONSE);
    if (strncmp(name.str, "_services._dns-sd._udp.local.", name.length) == 0) {
        // 用 mdns_discovery_answer 回应服务列表
        mdns_discovery_answer(sock, from, addrlen, buffer,
                               sizeof(buffer), &service_record, 1);
    } else if (strncmp(name.str, service->service.str, name.length) == 0) {
        // 按 unicast/multicast 选择响应方式
        if (unicast) {
            mdns_query_answer_unicast(sock, from, addrlen, ...);
        } else {
            mdns_query_answer_multicast(sock, ...);
        }
    }
    return 0; // 返回 0 继续解析，返回非 0 停止
}

回调中 rclass 的双重含义

场景bit 15 含义处理方式
问题区（Question）QU bit：请求单播响应根据 rclass & MDNS_UNICAST_RESPONSE 决定用 unicast 还是 multicast 回答
答案区（Answer）Cache-Flush bit：通知刷新缓存接收方应删除同名旧记录
⚛ 第一性原理

回调模型把"解析策略"与"响应策略"完全分离： 库只负责把字节流翻译成结构化参数并调用你的函数，至于怎么响应完全由调用方决定。 这是开放/封闭原则在 C 语言中的纯粹体现——扩展开放（实现任何回调逻辑），修改封闭（不改库代码）。

硬编码的发现查询包

库内置了一个静态字节数组，直接 sendto 即可，无需动态构建：

// _services._dns-sd._udp.local. 的 PTR/QU 查询，硬编码字节
static const uint8_t mdns_services_query[] = {
    0x00, 0x00,  // Query ID = 0
    0x00, 0x00,  // Flags = 0 (Query)
    0x00, 0x01,  // 1 Question
    0x00, 0x00,  0x00, 0x00, 0x00, 0x00, // Ans/Auth/Add = 0
    0x09, '_','s','e','r','v','i','c','e','s', // "_services"
    0x07, '_','d','n','s','-','s','d',        // "_dns-sd"
    0x04, '_','u','d','p',                  // "_udp"
    0x05, 'l','o','c','a','l', 0x00,        // "local."
    0x00, MDNS_RECORDTYPE_PTR,               // QTYPE = PTR
    0x80, MDNS_CLASS_IN                      // QU bit + CLASS IN
};
⚛ 第一性原理

为什么用硬编码字节而不动态构建？因为这个查询包的内容永远不变（RFC 6763 §9 规定了固定查询名）。 固定输入 = 固定输出 = 编译时常量。硬编码消除了运行时字符串编码的开销和出错可能，是极致的零成本抽象。

单播 vs 多播响应过滤

mdns_query_recv 支持 only_query_id 参数过滤响应， 但 RFC 6762 不鼓励依赖 query_id（mDNS query_id 通常为 0）， 传 0 则接受所有响应： 

// 过滤逻辑：只处理匹配 query_id 的响应（0 则不过滤）
if ((only_query_id > 0) && (query_id != only_query_id))
    return 0; // 不是目标查询的响应，跳过
⚛ 第一性原理

一次查询为什么能包含多个问题（multiquery）？因为 DNS 报文头中的 questions 字段就是计数器，后面可以跟任意多条 QNAME+QTYPE+QCLASS 三元组。 批量查询 = 把多个独立问题压缩进同一 UDP 包，减少网络包数量，减少响应延迟。

图12：mDNS 响应报文结构（以 PTR 查询响应为例）

TXT 记录合并优化

库将多个 mdns_record_t 类型为 TXT 的记录自动合并到同一个 TXT 记录中， 减少报文大小： 

// mdns_answer_get_record_count: 计算实际记录数，TXT 记录合并计1条
static inline uint16_t
mdns_answer_get_record_count(const mdns_record_t* records, size_t record_count) {
    uint16_t total_count = 0;
    uint16_t txt_record  = 0;
    for (size_t irec = 0; irec < record_count; ++irec) {
        if (records[irec].type == MDNS_RECORDTYPE_TXT)
            txt_record = 1;  // 多个 TXT 只计 1 条
        else
            ++total_count;
    }
    return total_count + txt_record;
}

// mdns_answer_add_txt_record: 将所有 TXT 记录写入同一条 RDATA
// 格式: [len][key=value][len][key=value]...

公告与注销（Announce / Goodbye）

函数TTL语义RFC 依据
mdns_announce_multicast正常值（如 3600s）服务上线，主动广播服务记录RFC 6762 §8.3
mdns_goodbye_multicast0服务下线，TTL=0 通知所有缓存失效RFC 6762 §11.3
⚛ 第一性原理

为什么 Goodbye 包的 TTL=0？因为 mDNS 没有中心服务器维护服务列表， 每台设备各自缓存记录。服务终止时不广播 TTL=0， 对方的缓存会在 TTL 超期前一直认为服务存在，导致尝试连接已下线的服务。 TTL=0 = 立即失效信号，是分布式缓存一致性的最小实现。

14 · 零内存分配设计

设计约束

核心原则：库的所有函数内部不调用 malloc/free，所有需要的 buffer 均由调用方在函数调用时传入。 这保证了：嵌入式友好、可预测内存使用、无内存泄漏风险。

buffer 传递模式

图13：零分配 buffer 传递模式

调用方示例

// 调用方负责分配和管理内存
size_t capacity = 2048;
void* buffer = malloc(capacity); // 调用方 malloc

// 库函数使用调用方的 buffer
mdns_query_send(sock, MDNS_RECORDTYPE_PTR, name, len, buffer, capacity, 0);

// 接收时也使用同一 buffer
mdns_query_recv(sock, buffer, capacity, query_callback, user_data, query_id);

free(buffer); // 调用方负责释放

字符串缓冲区同样由调用方提供

static char namebuffer[256]; // 调用方静态分配，解压缩字符串用

// mdns_string_extract 将解压缩结果写入调用方的 strbuffer
mdns_string_t name = mdns_string_extract(
    data, size, &offset, namebuffer, sizeof(namebuffer));
⚛ 第一性原理

零分配的根本出发点是：内存分配是隐含的全局状态依赖。 嵌入式系统可能没有堆，RTOS 环境 malloc 可能不是线程安全的， 服务器程序追求精确控制分配策略。把内存管理权归还给调用方， 库变成了纯粹的"算法盒子"：给我数组，我给你结果——完全确定性，完全可测试。

16 · 第一性原理总览

为什么 mDNS 以这种方式设计？

图14：mjansson/mdns 设计的八大第一性原理

各章节第一性原理汇总

章节核心问题第一性原理答案
RFC 协议基础如何无服务器发现服务？共享多播频道 = 固定地址 + 固定端口，.local. = 链路命名空间
整体架构为什么只有一个头文件？static inline 内联消除链接依赖，集成成本趋零
设计模式为什么用回调而非返回值？流式解析 + 零分配，记录数不确定，callback = 惰性消费者
类型系统Union 为什么合适？同一内存，不同语义，type 字段做 tag，比继承更轻量
报文结构为什么要压缩名称？UDP 包有大小限制，重复出现的域名标签可以用 2B 指针替代
Socket 层为什么需要多接口枚举？IP_MULTICAST_IF 每 socket 只绑一个接口，多接口需多 socket
字符串压缩16 槽位够用吗？mDNS 报文中重复后缀有限（.local./_tcp.等），16 近似最优
记录解析为何不拷贝数据？零拷贝 = offset + len 传给回调，回调按需 parse，最小内存
零分配为什么不在库内 malloc？内存是全局状态，归还调用方保证可预测性和嵌入式兼容
Goodbye TTL=0为什么用 TTL 而非通知协议？分布式缓存无法主动失效，TTL=0 是最小的显式失效信号
模糊测试为何要 1G 次随机数据？网络输入天然不可信，随机 = 最坏情况攻击者，找到未检查边界
📌 终极总结：

mjansson/mdns 的整个设计可以归结为一句话： "用最少的约束（一个头文件、调用方提供 buffer、回调驱动）实现 RFC 6762/6763 的完整协议语义， 同时在任何平台上都不崩溃。"

每一个设计选择都直接由某个第一原则推导出来，没有任何多余的抽象层—— 这正是优秀 C 语言库的本质：协议的忠实翻译 + 最小可信代码。
