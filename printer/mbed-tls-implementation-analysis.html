mbedTLS 深度解析

从设计原理、架构模式到具体实现的系统性讲解。mbedTLS（原 PolarSSL）是 ARM/Mbed 维护的开源 C 语言密码学与 TLS 库，以极小代码体积（最低 45 KB）、高可移植性、模块化设计著称，广泛用于嵌入式系统、IoT 设备及服务器。

Apache-2.0 许可 支持 TLS 1.2 / 1.3 DTLS 支持 PSA Crypto API

原文参考： github.com/Mbed-TLS/mbedtls · mbed-tls.readthedocs.io · TF-PSA-Crypto

🧠 第一性原理

安全通信的本质是在不可信信道上建立可信通道。这需要三件事：验证对方身份（X.509）、协商共享秘密（TLS 握手）、保护数据完整性与机密性（密码学原语）。mbedTLS 的三库结构直接对应这三个本质需求，层次分明，职责单一。

⚗️密码学原语层

libmbedcrypto（libtfpsacrypto）提供所有密码学原语实现，通过统一的上下文（context）模式管理状态。

上下文（Context）设计模式

所有模块遵循统一的生命周期：init → setup/seed → use → free，使用不透明结构体封装内部状态。

/* 以 AES 为例，所有模块均遵循此模式 */
mbedtls_aes_context ctx;

mbedtls_aes_init(&ctx);                          /* 1. 初始化（置零/初始状态）*/
mbedtls_aes_setkey_enc(&ctx, key, 256);           /* 2. 装载密钥（展开轮密钥）*/
mbedtls_aes_crypt_cbc(&ctx, MBEDTLS_AES_ENCRYPT,  /* 3. 使用 */
                      len, iv, input, output);
mbedtls_aes_free(&ctx);                           /* 4. 释放（安全清零）*/

通用密码接口（Generic Cipher API）

mbedtls_cipher_context_t 提供统一的密码操作接口，通过 mbedtls_cipher_info_t 描述符动态选择算法，实现了策略模式：

mbedtls_cipher_context_t ctx;
const mbedtls_cipher_info_t *info;

info = mbedtls_cipher_info_from_type(MBEDTLS_CIPHER_AES_256_GCM);
mbedtls_cipher_init(&ctx);
mbedtls_cipher_setup(&ctx, info);       /* 通过描述符配置算法 */
mbedtls_cipher_setkey(&ctx, key, 256, MBEDTLS_ENCRYPT);
mbedtls_cipher_update(&ctx, input, ilen, output, &olen);
mbedtls_cipher_finish(&ctx, output + olen, &olen2);
mbedtls_cipher_free(&ctx);
图2：密码学原语模块结构与统一 API 接口层

🧠 第一性原理

密码学的本质是数学变换的不可逆性与计算复杂度假设。mbedTLS 将每种算法封装为独立上下文（Context），遵循"初始化—使用—释放"的确定性生命周期，确保任何路径（包括错误路径）都不会泄漏密钥材料。统一接口（Generic Cipher API / PSA）将算法选择从使用逻辑中解耦，使上层代码无需感知底层算法变化。

🛡️PSA Crypto API

从 mbedTLS v3 开始，底层密码学操作统一通过 PSA Cryptography API（Platform Security Architecture）实现。mbedTLS 4.0 更进一步，TLS/X.509 层完全使用 PSA API，移除了传统 RNG 函数参数。 ↗ TF-PSA-Crypto

PSA 核心概念：密钥标识符（Key ID）

PSA 使用整数型密钥 ID（psa_key_id_t）代替指针或原始字节，密钥材料由 PSA 核心内部管理（可存于安全区或 HSM 中）：

psa_key_attributes_t attributes = PSA_KEY_ATTRIBUTES_INIT;
psa_key_id_t key_id;

/* 1. 设置密钥属性 */
psa_set_key_type(&attributes, PSA_KEY_TYPE_AES);
psa_set_key_bits(&attributes, 256);
psa_set_key_usage_flags(&attributes, PSA_KEY_USAGE_ENCRYPT | PSA_KEY_USAGE_DECRYPT);
psa_set_key_algorithm(&attributes, PSA_ALG_GCM);
psa_set_key_lifetime(&attributes, PSA_KEY_LIFETIME_VOLATILE); /* 易失性 */

/* 2. 导入密钥，获得 ID */
psa_import_key(&attributes, key_bytes, key_len, &key_id);

/* 3. 使用密钥 ID 进行操作（不暴露密钥材料）*/
psa_aead_encrypt(key_id, PSA_ALG_GCM, nonce, nonce_len,
                 ad, ad_len, input, input_len,
                 output, output_size, &output_len);

/* 4. 销毁密钥 */
psa_destroy_key(key_id);

💡

PSA API 的核心安全优势：密钥材料永远不需要暴露给调用者。调用者只持有整数型 Key ID，实际密钥可存于 TEE / HSM 中，硬件保证密钥不离开安全边界。

🧠 第一性原理

密钥管理的本质问题是：如何防止密钥材料在不需要暴露时被读取？PSA 的答案是将密钥转换为不透明标识符（Key ID），调用者操作的是 ID 而非字节，密码学操作在 PSA Core 内部完成，就像数据库用外键而非直接暴露数据一样。这是最小权限原则在密钥管理上的具体体现。

🎲随机数生成：熵池 + CTR-DRBG

mbedTLS 实现了 NIST SP 800-90A 规范的 CTR-DRBG（Counter mode Deterministic Random Bit Generator），以 熵收集模块 为种子源，两者协同工作。

图4：熵收集 + CTR-DRBG 随机数生成流水线（来源：mbedtls-docs）

CTR-DRBG 内部机制

CTR-DRBG 使用 AES-256 以 ECB 模式加密递增计数器，每次生成 16 字节随机块（与 NIST SP 800-90A §10.2.1.5.2 对应）：

/* CTR-DRBG 生成核心（简化版）
   对应 NIST SP 800-90A §10.2.1.5.2 */
while (output_len > 0) {
    /* 递增 128-bit 大端计数器 */
    for (i = MBEDTLS_CTR_DRBG_BLOCKSIZE; i > 0; i--)
        if (++ctx->counter[i - 1] != 0) break;

    /* AES-ECB 加密计数器产生随机块 */
    mbedtls_aes_crypt_ecb(&ctx->aes_ctx, MBEDTLS_AES_ENCRYPT,
                          ctx->counter, tmp);
    /* 复制到输出缓冲区 */
    use_len = (output_len > MBEDTLS_CTR_DRBG_BLOCKSIZE)
              ? MBEDTLS_CTR_DRBG_BLOCKSIZE : output_len;
    memcpy(p, tmp, use_len);
    p += use_len; output_len -= use_len;
}
/* 生成后更新内部状态（反向预测抵抗）*/
ctr_drbg_update_internal(ctx, add_input);
ctx->reseed_counter++;

重播种策略

场景重播种条件配置宏
普通模式计数器超过间隔值（默认 10000）MBEDTLS_CTR_DRBG_RESEED_INTERVAL
预测抵抗模式每次生成前必须重播种MBEDTLS_CTR_DRBG_PR_ON

标准集成代码模式

mbedtls_entropy_context entropy;
mbedtls_ctr_drbg_context ctr_drbg;
const char *pers = "my_application";   /* 个性化字符串，防止启动时熵不足 */

mbedtls_entropy_init(&entropy);
mbedtls_ctr_drbg_init(&ctr_drbg);

/* 播种：熵池作为源，加入个性化字符串 */
mbedtls_ctr_drbg_seed(&ctr_drbg, mbedtls_entropy_func, &entropy,
                      (const unsigned char *) pers, strlen(pers));

/* 生成随机字节 */
mbedtls_ctr_drbg_random(&ctr_drbg, output, 32);

/* 硬件 TRNG 可通过回调接入熵池 */
mbedtls_entropy_add_source(&entropy, my_hw_rng_poll,
                           NULL, 32, MBEDTLS_ENTROPY_SOURCE_STRONG);

🧠 第一性原理

随机数的本质需求是不可预测性。真随机数来自物理熵源（热噪声、中断抖动），但速率慢且平台相关；伪随机数速率快但依赖种子质量。mbedTLS 的两层设计正是这一张力的最优解：用熵池聚合所有可用物理熵，用 CTR-DRBG 将有限熵"拉伸"为所需长度的高质量伪随机序列，同时通过定期重播种维持前向安全性。

🧠 第一性原理

身份验证的本质是：你如何知道你在和谁通信？ X.509 将这个问题转化为信任传递：只要你信任根 CA，根 CA 信任中间 CA，中间 CA 信任服务器证书，就建立了完整的信任链。mbedTLS 的链式验证严格遵循这一逻辑，并通过安全配置文件约束"信任"的最低算法强度下限，防止弱算法破坏信任传递。

🤝TLS 握手状态机

mbedTLS 的 TLS 握手以状态机实现。核心函数 mbedtls_ssl_handshake() 循环调用 mbedtls_ssl_handshake_step() 直至握手完成。状态存储在 ssl->state，每个状态由专属处理函数负责。 ↗ ssl_tls.c

状态机核心实现模式

int mbedtls_ssl_handshake(mbedtls_ssl_context *ssl) {
    int ret = 0;
    while (mbedtls_ssl_is_handshake_over(ssl) == 0) {
        ret = mbedtls_ssl_handshake_step(ssl);
        if (ret != 0) break;
    }
    return ret;
}

int mbedtls_ssl_handshake_step(mbedtls_ssl_context *ssl) {
    /* 根据协议版本分发到对应状态机 */
#if defined(MBEDTLS_SSL_PROTO_TLS1_3)
    if (ssl->tls_version == MBEDTLS_SSL_VERSION_TLS1_3)
        return mbedtls_ssl_tls13_handshake_client_step(ssl);
#endif
    return mbedtls_ssl_handshake_client_step(ssl); /* TLS 1.2 */
}

/* TLS 1.2 客户端状态机 */
int mbedtls_ssl_handshake_client_step(mbedtls_ssl_context *ssl) {
    switch (ssl->state) {
        case MBEDTLS_SSL_CLIENT_HELLO:
            return ssl_write_client_hello(ssl);
        case MBEDTLS_SSL_SERVER_HELLO:
            return ssl_parse_server_hello(ssl);
        case MBEDTLS_SSL_SERVER_CERTIFICATE:
            return mbedtls_ssl_parse_certificate(ssl);
        /* ... 更多状态 ... */
        case MBEDTLS_SSL_HANDSHAKE_OVER:
            return 0;
    }
}
⚠️

握手步骤可能返回 MBEDTLS_ERR_SSL_WANT_READ 或 MBEDTLS_ERR_SSL_WANT_WRITE，表示 I/O 未就绪。调用方需循环重试直至握手完成或真正出错。这是非阻塞 I/O 的设计体现。

🧠 第一性原理

网络协议的本质是有序的消息交换以达成双方状态同步。TLS 握手必须保证：在任意网络条件（包括丢包、延迟、分片）下，双方都能从同一状态出发、经过相同消息序列、到达完全一致的安全参数。状态机是解决这一问题的最自然数学模型：每个状态精确描述"我知道什么/期望什么"，每次状态转换对应一个已验证的消息，任何异常均可映射到确定的错误状态。

📦记录层与加密

TLS 记录层负责将上层数据封装为 TLS 记录（Record），并进行加密/解密、MAC 计算和序列号管理。核心结构是 mbedtls_ssl_transform，它由握手层导出的密钥材料填充。 ↗ ssl_msg.c

mbedtls_ssl_context 核心字段

typedef struct mbedtls_ssl_context {
    const mbedtls_ssl_config  *conf;          /* 共享配置（不可变）*/
    int                        state;          /* 当前握手状态 */
    mbedtls_ssl_session       *session;        /* 当前会话 */
    mbedtls_ssl_handshake_params *handshake;  /* 握手临时参数 */

    /* 记录层 Transform（握手完成后切换）*/
    mbedtls_ssl_transform    *transform_in;    /* 入站解密 */
    mbedtls_ssl_transform    *transform_out;   /* 出站加密 */
    mbedtls_ssl_transform    *transform_negotiate; /* 协商中 */

    /* I/O 回调（网络层抽象）*/
    mbedtls_ssl_send_t        *f_send;         /* 发送函数指针 */
    mbedtls_ssl_recv_t        *f_recv;         /* 接收函数指针 */
    void                      *p_bio;          /* I/O 上下文（如 socket fd）*/

    /* 缓冲区指针（零拷贝设计）*/
    unsigned char             *in_buf;         /* 接收缓冲区 */
    unsigned char             *out_buf;        /* 发送缓冲区 */
} mbedtls_ssl_context;

   PSA AEAD 加密：AAD = 记录头，plaintext = 数据 */

);

🧠 第一性原理

记录层解决的核心问题是：如何保证每一条消息的机密性、完整性和抗重放性？ AEAD（Authenticated Encryption with Associated Data）是现代密码学对这三个需求的统一解答：一次操作同时完成加密（机密性）和认证（完整性），序列号纳入 IV 计算防止重放。mbedTLS 的 Transform 结构是这一数学保证在代码中的物化。

🚀TLS 1.3 新机制

mbedTLS 完整实现了 TLS 1.3（RFC 8446），相比 TLS 1.2 有根本性改进：握手仅需 1-RTT（0-RTT 可选），移除所有不安全算法，密钥派生完全基于 HKDF。 ↗ tls13-support.md

TLS 1.3 状态机特点

TLS 1.3 状态机由 mbedtls_ssl_tls13_handshake_client_step() 和 mbedtls_ssl_tls13_handshake_server_step() 实现，每个状态处理分为五个阶段：

阶段职责可选
Coordination（协调）判断是否跳过此状态（如 PSK 无需证书状态）可选
Fetching（获取）从网络接收预期的握手消息必须
Pre-processing（预处理）解析前的上下文准备可选
Parsing（解析）纯粹解析消息字段，最小化上下文修改必须
Post-processing（后处理）派生密钥、更新校验和、切换密钥可选
ℹ️

TLS 1.3 移除了所有已知不安全特性：无 RSA 密钥交换、无 CBC 模式、无 RC4/DES/3DES、无 MD5/SHA-1、无重协商（Renegotiation）。所有握手消息在 ServerHello 之后即开始加密。

🧠 第一性原理

TLS 1.3 的设计哲学是：从安全缺陷的根本原因入手消除问题，而非打补丁。RTT 减少不只是性能优化，而是因为旧版本额外的消息往返本是为了支持已被淘汰的 RSA 密钥交换；早期加密不只是隐私改进，而是因为 TLS 1.2 握手中传输明文的元数据被证明可被利用于流量分析攻击。mbedTLS 对 TLS 1.3 的实现忠实反映了这种"化繁为简"的安全设计哲学。

🧠 第一性原理

代码体积最小化的本质是：只为你需要的功能付出代价。C 预处理器宏在编译时（而非运行时）进行功能选择，未使用的代码路径由编译器完全消除，没有任何运行时开销。这与面向对象语言中运行时 feature flag 或插件系统有根本区别——mbedTLS 的裁剪是真正的零成本抽象。

🌐平台抽象层（PAL）

mbedTLS 通过平台抽象层将可移植的密码学核心与平台相关的 I/O、内存、时钟、线程等系统服务解耦，保证同一套代码可在 Linux、RTOS（FreeRTOS、Zephyr）、裸机（Bare Metal）等任意平台运行。

🧠 第一性原理

可移植性的本质是：将变化的部分与不变的部分分离。mbedTLS 的"不变部分"是密码学正确性（AES 的数学变换在任何处理器上相同），"变化部分"是 I/O、内存、时钟（每个平台完全不同）。平台抽象层通过函数指针和宏替换将二者解耦，使核心代码零修改就能跑在任意平台上。这正是依赖反转原则（DIP）在嵌入式 C 中的实现形式。

🧠内存管理策略

mbedTLS 支持三种内存策略，满足从 Linux 服务器到无操作系统裸机的全部场景。 ↗ 官方文档

① 标准堆（默认）

使用 calloc() 和 free()，适用于有完整 libc 的平台。

② 自定义分配器

通过函数指针替换，适用于需要内存池、追踪或隔离的场景。

③ 静态缓冲分配器

mbedTLS 内置缓冲分配器，用静态数组模拟堆，无需 OS 支持。

自定义内存分配器

/* 启用内存层（mbedtls_config.h）*/
#define MBEDTLS_PLATFORM_C
#define MBEDTLS_PLATFORM_MEMORY

/* 注册自定义分配器 */
mbedtls_platform_set_calloc_free(my_calloc, my_free);

/* IoT 设备示例：静态内存池 */
static unsigned char memory_pool[65536];
static size_t pool_used = 0;

void *my_calloc(size_t n, size_t size) {
    size_t total = n * size;
    if (pool_used + total > sizeof(memory_pool)) return NULL;
    void *ptr = &memory_pool[pool_used];
    pool_used += total;
    memset(ptr, 0, total);
    return ptr;
}

内置缓冲分配器

/* 内置缓冲分配器（无需 OS 堆）
   启用：MBEDTLS_MEMORY_BUFFER_ALLOC_C */
unsigned char buf[100000];

mbedtls_memory_buffer_alloc_init(buf, sizeof(buf));

/* 之后所有 mbedtls_calloc/free 都使用这个缓冲区 */
/* 支持调试：统计分配/释放次数和峰值使用量 */
size_t cur_used, max_used;
mbedtls_memory_buffer_alloc_cur_get(&cur_used, NULL);
mbedtls_memory_buffer_alloc_max_get(&max_used, NULL);

内存安全：敏感数据清零

mbedTLS 使用 mbedtls_platform_zeroize() 而非 memset() 清零敏感数据，确保编译器不会优化掉此操作：

/* 握手结束后安全清零随机数（防止内存泄漏密钥材料）*/
mbedtls_platform_zeroize(ssl->handshake->randbytes,
                         sizeof(ssl->handshake->randbytes));

/* CTR-DRBG 生成后清零临时缓冲区 */
mbedtls_platform_zeroize(add_input, sizeof(add_input));
mbedtls_platform_zeroize(tmp, sizeof(tmp));
🚨

切勿用 memset(ptr, 0, len) 清零密钥等敏感数据——编译器优化（如 dead-store elimination）可能将其完全删除。mbedtls_platform_zeroize() 通过编译器屏障（barrier）保证清零操作不被省略。

🧠 第一性原理

内存管理的安全本质有两个维度：生存期控制（密钥材料用完即销毁）和隔离性（密钥所在内存不被非授权代码读取）。mbedtls_platform_zeroize() 解决生存期问题；自定义内存池和静态分配器解决隔离问题（同时消除了堆碎片这一嵌入式系统的典型故障模式）。

示例：STM32 硬件 AES 接入

/* mbedtls_config.h */
#define MBEDTLS_AES_ALT                /* 替换整个 AES 模块 */
#define MBEDTLS_ENTROPY_HARDWARE_ALT   /* 替换熵源（使用硬件 RNG）*/

/* aes_alt.c：使用 STM32 HAL 驱动 */
int mbedtls_aes_crypt_ecb(mbedtls_aes_context *ctx,
                          int mode,
                          const unsigned char input[16],
                          unsigned char output[16]) {
    if (mode == MBEDTLS_AES_ENCRYPT)
        return HAL_CRYP_Encrypt(&hcryp, input, 16, output, 100);
    else
        return HAL_CRYP_Decrypt(&hcryp, input, 16, output, 100);
}

/* entropy_hardware_alt.c：使用硬件 TRNG */
int mbedtls_hardware_poll(void *data, unsigned char *output,
                          size_t len, size_t *olen) {
    for (size_t i = 0; i < len/4; i++) {
        uint32_t rnd;
        HAL_RNG_GenerateRandomNumber(&hrng, &rnd);
        memcpy(output + i*4, &rnd, 4);
    }
    *olen = len;
    return 0;
}
💡

Context 可移动性要求：替换实现的 Context 结构体必须支持内存移动（不能持有指向自身内部的指针）。这与 Rust 的 Pin<T> 约束类似，是 C 语言中的重要规范。

🧠 第一性原理

硬件加速的本质是：相同数学变换（AES 轮函数）既可由软件循环实现，也可由专用硬件电路在一个时钟周期内完成。_ALT 机制将"什么"（AES 加密接口）和"如何"（软件/硬件实现）分离，使上层代码无需感知加速细节。这是开放-封闭原则（OCP）在 C 语言中的精准体现：对扩展开放（支持任意 ALT 实现），对修改封闭（上层接口不变）。

💻实战：TLS 客户端完整实现

以下展示一个完整的 TLS 1.3 客户端，涵盖从初始化到安全通信的全流程，注释对应各模块的设计原理。 ↗ TLS Guide

上下文生命周期总览

图13：TLS 客户端上下文生命周期

完整代码实现

#include "mbedtls/ssl.h"
#include "mbedtls/net_sockets.h"
#include "mbedtls/entropy.h"
#include "mbedtls/ctr_drbg.h"
#include "mbedtls/x509_crt.h"
#include "mbedtls/error.h"

int tls_client_connect(const char *host, const char *port,
                        const char *ca_cert_pem) {
    int ret;
    char err_buf[256];

    /* ① 声明所有上下文 */
    mbedtls_net_context      server_fd;
    mbedtls_ssl_context      ssl;
    mbedtls_ssl_config       conf;
    mbedtls_entropy_context  entropy;
    mbedtls_ctr_drbg_context ctr_drbg;
    mbedtls_x509_crt         cacert;

    /* ② 初始化（置零/安全状态）*/
    mbedtls_net_init(&server_fd);
    mbedtls_ssl_init(&ssl);
    mbedtls_ssl_config_init(&conf);
    mbedtls_entropy_init(&entropy);
    mbedtls_ctr_drbg_init(&ctr_drbg);
    mbedtls_x509_crt_init(&cacert);

    /* ③ 初始化 RNG：熵池 → CTR-DRBG 播种 */
    ret = mbedtls_ctr_drbg_seed(&ctr_drbg, mbedtls_entropy_func,
              &entropy, (const unsigned char*)"tls_client", 10);
    if (ret != 0) goto cleanup;

    /* ④ 加载 CA 根证书（PEM 格式，含 '\0'）*/
    ret = mbedtls_x509_crt_parse(&cacert,
              (const unsigned char *)ca_cert_pem,
              strlen(ca_cert_pem) + 1);
    if (ret != 0) goto cleanup;

    /* ⑤ 建立 TCP 连接 */
    ret = mbedtls_net_connect(&server_fd, host, port,
                              MBEDTLS_NET_PROTO_TCP);
    if (ret != 0) goto cleanup;

    /* ⑥ 配置 TLS 参数 */
    mbedtls_ssl_config_defaults(&conf,
        MBEDTLS_SSL_IS_CLIENT,
        MBEDTLS_SSL_TRANSPORT_STREAM,    /* TCP；DATAGRAM = DTLS */
        MBEDTLS_SSL_PRESET_DEFAULT);

    /* 强制验证服务器证书（生产必须）*/
    mbedtls_ssl_conf_authmode(&conf, MBEDTLS_SSL_VERIFY_REQUIRED);
    mbedtls_ssl_conf_ca_chain(&conf, &cacert, NULL);
    /* 注入 RNG（mbedTLS 4.0 之前需要，4.0 后 PSA 内置）*/
    mbedtls_ssl_conf_rng(&conf, mbedtls_ctr_drbg_random, &ctr_drbg);

    /* ⑦ 设置 SSL 上下文 */
    mbedtls_ssl_setup(&ssl, &conf);
    mbedtls_ssl_set_hostname(&ssl, host);  /* SNI + CN 验证 */

    /* ⑧ 注入 I/O 回调（网络层与 SSL 层解耦）*/
    mbedtls_ssl_set_bio(&ssl, &server_fd,
        mbedtls_net_send, mbedtls_net_recv, NULL);

    /* ⑨ TLS 握手（状态机循环直至完成）*/
    while ((ret = mbedtls_ssl_handshake(&ssl)) != 0) {
        if (ret != MBEDTLS_ERR_SSL_WANT_READ &&
            ret != MBEDTLS_ERR_SSL_WANT_WRITE) goto cleanup;
    }

    /* ⑩ 验证服务器证书（握手后检查结果）*/
    uint32_t flags = mbedtls_ssl_get_verify_result(&ssl);
    if (flags != 0) {
        mbedtls_x509_crt_verify_info(err_buf, sizeof(err_buf),
                                     "  ! ", flags);
        /* err_buf 包含具体失败原因 */
        ret = -1; goto cleanup;
    }

    /* ⑪ 发送/接收加密应用数据 */
    const char *req = "GET / HTTP/1.1\r\nHost: example.com\r\n\r\n";
    mbedtls_ssl_write(&ssl, (const unsigned char*)req, strlen(req));

    unsigned char resp[4096];
    int n = mbedtls_ssl_read(&ssl, resp, sizeof(resp) - 1);

    /* ⑫ 关闭通知（close_notify alert）*/
    mbedtls_ssl_close_notify(&ssl);

cleanup:
    /* ⑬ 释放所有资源（顺序无关，各自独立）*/
    mbedtls_net_free(&server_fd);
    mbedtls_ssl_free(&ssl);
    mbedtls_ssl_config_free(&conf);
    mbedtls_x509_crt_free(&cacert);
    mbedtls_ctr_drbg_free(&ctr_drbg);
    mbedtls_entropy_free(&entropy);
    return ret;
}

🧠 总结：第一性原理回顾

回到最本质的问题：两台机器如何在不可信网络上安全通信？ 答案需要三样东西：①身份（X.509 证书链）、②秘密协商（ECDHE 密钥交换，满足前向安全）、③数据保护（AEAD 加密，同时保证机密性与完整性）。mbedTLS 的每一行代码都是这三个需求的具体落地。其模块化、可配置、可替换的设计不是美学偏好，而是对"在资源受限的嵌入式设备上安全通信"这一工程约束的最优响应。理解了这个目标，mbedTLS 的每一个设计决策都变得顺理成章。
