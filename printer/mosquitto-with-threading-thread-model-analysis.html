WITH_THREADING 开启后的线程模型与锁保护边界

本文聚焦当前 libmosquitto.a 中 WITH_THREADING 相关实现：对外线程接口、各接口在哪类线程执行、哪些 struct mosquitto 数据会跨线程共享、使用什么同步手段，以及这些共享数据在什么场景下会被并发访问。

结论先行：开启 WITH_THREADING 后，libmosquitto 不是把所有逻辑改成多线程，而是允许一个网络 loop 执行线程与一个或多个应用调用线程共享同一个 struct mosquitto。它用 9 类 mutex 保护回调指针、日志回调、状态、消息时间、mid、出站 packet 队列、当前发送 packet、入站 QoS 队列、出站 QoS 队列。
第一性原理：线程模型要从“谁拥有事件循环”开始。只要网络循环和业务 API 可能在不同线程同时访问同一个客户端对象，状态对象就必须把共享字段和同步边界显式化。

官方线程契约

官方原文关键事实本文如何使用
mosquitto.h Threadslibmosquitto 提供 thread safe operation，但 mosquitto_lib_init() 不是线程安全的。初始化必须在并发使用前完成；本文不把 mosquitto_lib_init() 归入可并发 API。
mosquitto_loop_start()这是 threaded client interface 的一部分；调用一次后创建新线程处理网络流量，是反复手动调用 mosquitto_loop() 的替代方案。将它定义为“库内部 loop 线程模型”。
mosquitto_loop_stop()停止由 mosquitto_loop_start() 创建的网络线程；调用会阻塞直到网络线程结束；若 force=false，需要先调用 mosquitto_disconnect()。将它归为应用线程发起、等待 loop 线程退出的同步接口。
mosquitto_threaded_set()用于告诉库应用正在使用线程，但没有使用 mosquitto_loop_start()；如果自管线程却不调用它，会因 race condition 导致崩溃；使用 loop_start 时会自动设置。将它定义为“外部线程模型”的必要标记接口。
mosquitto_loop()主网络循环必须频繁调用，不能在 callback 内调用；它会处理输入、输出以及 QoS>0 流程中剩余命令。将它视为 loop 线程执行函数，手动模型下由应用指定线程调用。
第一性原理：官方契约给出两个线程安全前提：全局初始化不并发；每个客户端对象的网络循环必须有清晰 owner，其他线程只能通过受保护 API 向这个 owner 投递工作。

当前构建事实

构建开关

当前包规则传入 WITH_THREADING=yes，lib 编译命令包含 -DWITH_THREADING，并链接 -lpthread。因此 thread_mosq.c、mutex 字段、加锁路径都进入当前静态库语义。

初始化位置

mosquitto_reinitialise() 在 #ifdef WITH_THREADING 下初始化 callback_mutex、log_callback_mutex、state_mutex、out_packet_mutex、current_out_packet_mutex、msgtime_mutex、msgs_in.mutex、msgs_out.mutex、mid_mutex。

销毁位置

mosquitto__destroy() 会在已初始化对象上销毁这些 mutex；如果 threaded == mosq_ts_self 且当前线程不是 loop 线程，在支持 pthread cancel 时会先取消并 join loop 线程。

socketpair 唤醒

mutex 之外，库还用 sockpairR/sockpairW 做跨线程唤醒：应用线程 enqueue 出站 packet 后写一个字节，打断 loop 线程的 select()。

第一性原理：线程安全不是一个宏名，而是一组对象生命周期动作：创建锁、按共享字段加锁、跨线程唤醒、停止线程、销毁锁。

三种运行模式

模式mosq->threaded网络 loop 在哪里运行应用 API 在哪里运行典型用途
单线程手动 loopmosq_ts_none同一个应用线程反复调用 mosquitto_loop() 或拆分调用 loop_read/write/misc。同一线程调用 connect/publish/subscribe/disconnect。简单进程或已有单线程事件循环。
库内部 loop 线程mosq_ts_selfmosquitto_loop_start() 创建的新 pthread，入口是 mosquitto__thread_main()，内部调用 mosquitto_loop_forever()。调用者线程继续运行其他业务，可调用 publish/subscribe/disconnect/stop。官方推荐的非阻塞客户端接口。
应用自建线程mosq_ts_external应用自己创建某个线程调用 mosquitto_loop() 或 loop_forever()。其他应用线程调用 publish/subscribe 等；必须先 mosquitto_threaded_set(mosq,true)。工程已有线程模型，不希望 lib 自己创建线程。
第一性原理：三种模式的差别不是锁是否存在，而是“谁拥有网络循环”。一旦 loop 与业务 API 分属不同线程，packet 队列就是生产者-消费者结构。

对外接口与运行线程

接口类别代表 API在哪个线程调用内部线程效果共享数据影响
全局初始化mosquitto_lib_init()、mosquitto_lib_cleanup()应用初始化/退出线程。不创建客户端线程；官方说明 lib_init 非线程安全。全局 refcount 与网络库初始化，不应并发调用。
实例生命周期mosquitto_new()、mosquitto_reinitialise()、mosquitto_destroy()应用线程，通常在并发前/并发后调用。new/reinitialise 初始化 mutex；destroy 可能取消并 join 内部 loop 线程。生命周期必须与其他 API 串行，否则销毁对象与其他线程访问对象会冲突。
内部线程控制mosquitto_loop_start()、mosquitto_loop_stop()应用线程调用。loop_start 创建 loop 线程并设置 mosq_ts_self；loop_stop 写 sockpair 唤醒并 join。改变 threaded/thread_id，要求 start/stop 调用序列清晰。
外部线程标记mosquitto_threaded_set()应用线程，在自建 loop 线程模式下尽早调用。不创建线程，只设置 mosq_ts_external。使 packet__queue() 不走单线程立即写优化，避免应用线程和 loop 线程同时写 socket/packet。
网络 loopmosquitto_loop()、mosquitto_loop_forever()、mosquitto_loop_read/write/misc()loop owner 线程：可能是应用主线程、库内部线程，或应用自建网络线程。处理 socket read/write、Keep Alive、QoS ACK、回调触发。读写 in_packet/current_out_packet/out_packet/msgtime/state/msgs_*。
业务动作mosquitto_publish()、mosquitto_subscribe()、mosquitto_unsubscribe()、mosquitto_disconnect()应用线程，也可能从回调线程内调用部分 API。构造 packet 或 QoS message，入队后用 sockpair 唤醒 loop。访问 last_mid/out_packet/msgs_out/state/msgtime。
回调配置mosquitto_*_callback_set()应用线程，通常连接前设置，也可运行期改。不创建线程，只写回调函数指针。callback_mutex 或 log_callback_mutex 保护函数指针。
第一性原理：对外 API 可以按“生命周期、线程控制、loop owner、业务投递、回调配置”分层；只有业务投递和 loop owner 同时运行时，锁才真正承载并发同步。

回调运行线程

回调触发文件运行线程加锁方式并发含义
on_connect / on_connect_v5handle_connack.c处理 CONNACK 的 loop 线程。调用前锁 callback_mutex，设置 in_callback=true，回调后恢复。如果应用线程正在修改 connect 回调指针，会被互斥。
on_message / on_message_v5handle_publish.c、handle_pubrel.c收到 PUBLISH/PUBREL 的 loop 线程。callback_mutex 覆盖回调指针读取和回调调用。回调内不能调用 mosquitto_loop()；消息内存回调后由库释放，应用要复制。
on_publish / on_publish_v5packet_mosq.c、handle_pubackcomp.c、handle_pubrec.cQoS0 可能在写 packet 的线程；QoS1/2 通常在处理 ACK 的 loop 线程。callback_mutex。publish 回调与应用线程更新回调指针互斥，但回调内再发布会进入 packet 队列。
on_subscribe / on_unsubscribehandle_suback.c、handle_unsuback.c处理 SUBACK/UNSUBACK 的 loop 线程。callback_mutex。回调反映 broker 响应，不是在应用调用 subscribe/unsubscribe 的线程同步返回。
on_disconnect / on_disconnect_v5loop.c、util_mosq.c、connect.c检测断线或 Keep Alive 超时的 loop 线程；某些连接失败路径也会触发。callback_mutex。应用线程调用 mosquitto_disconnect() 后，最终回调仍由处理网络/状态变化的路径触发。
on_loglogging_mosq.c任何调用 log__printf() 的线程。log_callback_mutex。日志回调单独用锁，避免与业务回调共用大锁。
源码里回调调用期间仍持有 callback_mutex。这保护了回调指针一致性，但也意味着回调内如果调用会反向等待同一把锁的 API，存在设计层面的死锁风险；当前 Mosquitto 通过 in_callback 和 packet 入队路径降低部分重入问题，但应用回调仍应保持短小。
第一性原理：回调不是“应用线程异步通知”，而是 loop 线程在协议状态推进点执行的用户代码。它天然继承 loop 线程的时序和约束。

当前工程落点

当前业务客户端没有使用 mosquitto_loop_start()，而是在自己的 MQTT 工作线程中循环调用 mosquitto_loop() 并做重连。UBUS 线程/主事件循环可能通过业务封装调用 publish、subscribe、reconnect。

线程当前工程中的职责对应 lib 线程模型注意点
MQTT 工作线程创建 mosquitto 实例，设置 TLS/Will/callback，connect 后循环 mosquitto_loop()。外部 loop owner 线程。如果其他线程会同时调用同一 mosq 的 publish/subscribe，应属于 mosq_ts_external 模式。
UBUS/业务线程接收 publish/subscribe/reconnect 请求，通过业务封装触达 Mosquitto API。应用 API 调用线程。与 MQTT 工作线程共享 struct mosquitto，依赖 out_packet、msgs_out、mid、state 等锁。
消息缓存线程业务层 QoS2 缓存与重发。lib 外部的应用线程。它不直接改变 lib 的内部队列，但最终仍通过 mqtt_publish() 进入 lib。
官方 API 明确说：应用自己使用线程且不使用 mosquitto_loop_start() 时，需要调用 mosquitto_threaded_set()。当前业务自管 loop 线程，如果存在其他线程并发调用同一实例，应检查是否补上该调用。
第一性原理：工程线程模型和库线程模型必须对齐。库不知道你的线程拓扑，只有 threaded 状态能告诉它“不要再按单线程优化处理出站队列”。

共享数据总图

第一性原理：共享数据不是“整个 struct 都一样危险”。真正需要锁的是跨线程读写且会破坏链表、计数、状态或函数指针一致性的字段。

第一性原理：锁的保护范围要按数据不变量定义：链表锁保护 next/prev/head/count，状态锁保护枚举状态，时间锁保护 Keep Alive 判断输入，mid 锁保护单调生成。

第一性原理：并发访问不是“两个线程都存在”就发生，而是两个线程在同一时间读写同一个可变不变量。表中的锁对应的就是这些不变量。

边界与风险

mosquitto_lib_init()

官方说明它不是线程安全的。它维护静态 init_refcount 并调用 net__init()，应在创建并发线程前调用。

生命周期 API

mosquitto_destroy() 会释放整个对象和 mutex。它不是给其他线程仍在调用 API 时并发使用的“安全关闭开关”；应先停止 loop、断开连接、确保不再有外部线程访问。

threaded 状态

mosq->threaded 和 thread_id 没有独立 mutex。源码依赖 loop_start/stop/threaded_set 的调用序列约束，而不是把线程模式当作高频共享数据。

配置类字段

TLS、Will、username/password、协议版本、SOCKS 等配置通常要求 connect 前设置。很多字段没有专门锁保护，不能假定连接运行中任意并发修改都安全。

回调内重入

callback_mutex 在调用用户回调时仍持有。回调内应避免长时间阻塞或调用可能反向等待同一锁的接口，尤其不要调用 mosquitto_loop()。

锁顺序

出站 packet 清理和写路径按 current_out_packet_mutex 再 out_packet_mutex 的顺序加锁。新增代码不能反向拿锁，否则会制造死锁。

第一性原理：线程安全总有边界：锁保护的是运行期共享数据，不负责纠正错误的对象生命周期、错误的线程模式声明，或连接后随意修改配置。

总结

线程模型

WITH_THREADING 提供三种模式：单线程、库内部 loop 线程、应用自建 loop 线程。关键状态是 mosq_ts_none/self/external。

接口分层

对外接口分为初始化、生命周期、线程控制、loop、业务动作、回调配置。真正多线程交汇点是业务动作线程和 loop 线程。

锁保护

9 类 mutex 保护回调、日志、状态、时间、mid、出站 packet、当前 packet、入站 QoS、出站 QoS。socketpair 负责跨线程唤醒。

因此，开启 WITH_THREADING 后的 libmosquitto 可以理解为一个 Reactor 客户端：loop 线程独占 socket 事件推进，应用线程通过 API 生产控制报文和 QoS 消息，内部用队列锁与状态锁维持一致性。如果应用不使用 mosquitto_loop_start() 而自己创建 loop 线程，就必须通过 mosquitto_threaded_set() 告诉库关闭单线程即时写优化。

第一性原理：并发安全的核心不是“有很多锁”，而是每个共享不变量只有一个清晰的保护者，并且线程之间只通过这些受保护的不变量交接工作。
