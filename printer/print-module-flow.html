打印模块数据结构、共享内存分布与打印流程分析

本文分析本仓库 libcommon/print_api.c、libcommon/include/print_api.h 与 server/print/src 中 PrintServer 的数据通路：应用侧如何把打印数据写入共享内存，服务端如何把多通道任务收敛到全局队列，最终如何进入 ESC/POS 或 TSPL 解析、渲染并写入内核打印设备。

一句话概括：打印数据走共享内存里的“任务池 + 小块 buffer 链表”，控制命令走 POSIX 消息队列，互斥和状态通知走命名信号量；PrintServer 是唯一真正接触打印硬件的进程。
第一性原理：打印系统的最小问题是“多个生产者如何把有序字节流交给一个独占硬件消费者”。因此设计上必须先隔离生产者，再定义可恢复的队列边界，最后让唯一消费者串行化硬件访问。

官方依据

本文先获取并核对了外部官方资料，再结合本仓库源码解释实现。外部资料只用于解释 IPC 和打印指令体系的通用语义；具体结构字段、队列策略和异常处理以本仓库代码为准。

运行时边界

打印模块分成两个层次：应用侧 print_api 是客户端库，负责把数据组织进共享内存；PrintServer 是守护进程，负责初始化 IPC 对象、接收控制消息、扫描任务队列、解析打印指令并访问设备。

客户端库

典型入口是 print_channel_send_data()、print_channel_send_data_with_flag()、print_get_status()、print_set_density() 等。数据类 API 写共享内存；控制类 API 发送 print_msg_t 到消息队列。

服务端进程

main() 初始化全局变量、定时器、打印设备和队列，启动 USB/TCP/串口入口，启动 TSPL parser 线程，最后进入 print_queue_run() 主循环。

/* 服务端启动顺序的关键片段 */
if (printer_init())
    exit(1);

if (print_queue_init())
    exit(1);

usb_init();
tcp_server_init();
tcp_init();
serial_init();

tspl_parser_run();
print_queue_run();
第一性原理：硬件打印头是串行资源，不能让多个进程各自解释、各自写设备。把 PrintServer 作为唯一硬件消费者，能把并发问题从“多进程抢设备”降维成“多生产者排队”。

共享内存总图

print_shm_t 是整个打印模块的共享状态根对象。它不是单纯的数据缓冲区，而是把内存池、任务池、通道、全局打印队列、状态、设置、请求和检测结果放在一个固定布局里。客户端与服务端用同一头文件解释这块内存。

typedef struct {
    struct { print_buffer_t mem[PRINT_BUFFER_POOL_SIZE]; ... } buffer_pool;
    struct { print_task_t mem[PRINT_TASK_POOL_SIZE]; ... } task_pool;
    print_channel_t channels[PRINT_CHANNELS_MAX];
    struct { uint16_t head; uint16_t tail; uint16_t buf_to_send; ... } task_queue;
    print_status_t status;
    print_settings_t settings;
    struct { uint8_t status_change_subscribers; ... } request;
    struct { uint8_t result[72]; uint32_t size; int state; } bad_points;
    struct { uint32_t black_mark_location; ... } paper_location;
    struct { char version[16]; char language[16]; } fontlib;
    struct { uint8_t enabled; char seqnum[16]; ... } data_collection;
    struct { uint8_t enabled; uint64_t channel_allow; ... } third_party_data_collection;
    print_log_t print_log_mem[PRINT_TASK_POOL_SIZE];
    struct { uint8_t result[32]; int state; } gap_detect;
} print_shm_t;
第一性原理：共享内存里的对象必须能被多个进程用“相同偏移”解释，所以它不能保存进程私有指针。该设计用 16 位索引把 buffer 和 task 串起来，天然适合跨进程共享。

池化结构

打印数据被拆成固定大小的 print_buffer_t，任务只保存 buffer 链表的头尾索引。这样做避免了跨进程堆内存和碎片问题，也让服务端在任务完成、删除或不可重打时可以逐块归还。

结构关键字段设计含义
print_buffer_tnext、id、len、data[250]固定小块数据节点。next 是共享内存索引，id 用于打印过程中的 buffer 确认和重打定位。
print_task_thead、tail、chn_id、id、copies、bytes_received、buffer_sent、flags描述一个可打印任务，管理数据链、份数、接收/发送进度和异常策略。
buffer_poolmem[]、head、tail、free_count所有空闲数据块组成单链表。
task_poolmem[]、head、tail、next_task_id、last_completed_task_id任务描述符池，同时维护对外可见的任务编号。
struct _print_buffer_t {
    uint16_t next;
    uint16_t id;
    uint16_t len;
    uint8_t data[PRINT_BUFFER_SIZE];
};

struct _print_task_t {
    uint16_t prev;
    uint16_t next;
    uint16_t head;
    uint16_t tail;
    int chn_id;
    int id;
    uint32_t copies;
    uint32_t bytes_received;
    uint32_t bytes_handled;
    uint32_t bytes_to_receive;
    uint16_t next_buffer_id;
    uint16_t buffer_id_resend_count;
    uint16_t buffer_sent;
    uint16_t buffer_total;
    uint16_t flags;

第一性原理：如果数据大小不可预测，最稳定的共享表示不是“大 malloc”，而是固定块链表。固定块让分配、归还、重打定位和跨进程解释都变得确定。

通道与队列

系统有两级队列。第一级是每个输入通道的当前任务和通道待打印队列，避免 USB、MQTT、HTTPS、蓝牙、TCP 等来源的数据互相混合。第二级是全局 task_queue，PrintServer 只从这里按顺序发送给解析器和硬件。

struct _print_channel_t {
    uint32_t flags;
    int task_end_timeout;
    int no_data_received;
    int last_task_id;
    uint16_t task;
    struct {
        uint16_t head;
        uint16_t tail;
    } task_queue;
};

struct {
    uint16_t head;
    uint16_t tail;
    uint16_t buf_to_send;
    uint16_t buf_offset;
    int last_channel_id;
} task_queue;
第一性原理：不同来源的字节流不能交错，因为 ESC/POS/TSPL 是有状态协议。通道先隔离输入流，全局队列再统一硬件顺序，这是保持协议状态正确的最低成本做法。

客户端写入

应用调用打印 API 时，库先确保共享内存和信号量已打开。普通打印数据会进入 append_data_to_channel() 或创建独立任务；实时命令会先被识别，转为小型消息交给服务端立即处理。

  1. 检查通道和参数。
  2. 如果当前通道没有未完成任务，先尝试识别实时命令。
  3. 需要打印的数据从 buffer_pool 取块，追加到 print_task_t 的链表。
  4. 更新 bytes_received、buffer_total、last_task_id 和任务标志。
  5. copies != 0 时直接创建完整任务并放入通道任务队列；copies == 0 时允许同一通道连续追加，直到超时或长度达成后由服务端收口。
/* 客户端追加数据的核心意图：取 buffer，复制数据，挂到 task 链表 */
while (len > 0) {
    b = TASK_AT(p->task)->tail;
    free_bytes = IS_VALID_BUFFER_INDEX(b) ?
        (PRINT_BUFFER_SIZE - BUFFER_AT(b)->len) : 0;
    if (free_bytes == 0) {
        b = buffer_new();
        append_buffer_to_task(p->task, b);
        free_bytes = PRINT_BUFFER_SIZE;
    }
    l = (len > free_bytes) ? free_bytes : len;
    memcpy(BUFFER_AT(b)->data + BUFFER_AT(b)->len,
           data + ignored_bytes + bytes_copied, l);
    BUFFER_AT(b)->len += l;
    len -= l;
    bytes_copied += l;
}
第一性原理：打印数据通常大、连续、需要回放，适合放共享内存；控制命令小、离散、需要及时唤醒服务端，适合走消息队列。两类数据分流能避免把共享内存当万能邮箱。

服务端初始化

print_queue_init() 是共享内存生命周期的创建者。它创建命名信号量，创建并截断共享内存对象到 sizeof(print_shm_t)，用 MAP_SHARED 映射，再创建服务端消息队列，最后初始化池、通道和全局队列。

fd = shm_open(PRINT_SERVER_SHM, O_RDWR|O_CREAT, 0600);
ftruncate(fd, sizeof(print_shm_t));

shm = (print_shm_t *)mmap(NULL, sizeof(print_shm_t),
        PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);
close(fd);

mq = mq_open(PRINT_SERVER_MQ, O_RDONLY|O_CREAT, 0600, &attr);

buffer_pool_init();
task_pool_init();
channels_init();
task_queue_init();

pthread_create(&m_tid, &g_pthread_attr, print_mqueue_thread, NULL);
第一性原理：共享内存创建必须先有名字，再有大小，最后才有映射地址。初始化池链表的本质是把一段零散结构化内存变成可分配资源。

调度循环

print_queue_run() 是 PrintServer 的主调度器。它周期性读取内核打印状态，回写 shm->status，通知订阅者，处理缺纸/开盖/过热/待取纸等状态，然后把通道中的任务推进全局队列，并在条件允许时发送下一个 buffer。

while (g_process_running) {
    if (PrnGetStatus(&status) == 0) {
        PrnStatusToShmStatus(&status);
        if (status changed) {
            for (posts = 0; posts < shm->request.status_change_subscribers; posts++)
                sem_post(sem_status);
        }
    }

    flush_channels(msec, shm->status.printing);

    if (shm->status.printing == 0)
        check_task_print_status();

    if ((shm->status.step_motor_overheated == 0) &&
        (shm->status.printing != 0 || task_delay_ms == 0))
        send_next_buffer(curr_paper_status);
}
第一性原理：调度循环要回答三个问题：现在能不能打印，哪个任务该打印，当前任务是否结束。所有状态更新、异常恢复和游标推进都围绕这三个问题展开。

解析与渲染

send_next_buffer() 是共享内存队列到打印解释器的桥。普通 ESC/POS 模式下，它把当前 buffer 交给 printer_send_data()，内部逐字节进入 Cmd_Proc() 状态机；TSPL 模式下，它把 buffer 保存为脚本文件，任务接收完成后通过 /tspl_mqueue 通知 TSPL parser 线程解析。

/* ESC/POS 状态机入口 */
void Cmd_Start(void)
{
    switch (Cmd_Data) {
    case 0x10: pNextFun = DLE_Proc; break;
    case 0x1B: pNextFun = ESC_Proc; break;
    case 0x1C: pNextFun = FS_Proc;  break;
    case 0x1D: pNextFun = GS_Proc;  break;
    default:   Char_Proc();         break;
    }
}

void Cmd_Proc(uint32_t data)
{
    Cmd_Data = data;
    (*pNextFun)();
}
/* 最终写入内核打印缓冲的核心路径 */
static inline int write_to_kernel(void)
{
    int ret = write(PrnFd, PrnDivBuf, SizeOfPrnDivBuf);
    if (ret == PRN_BUF_FULL) {
        while (1) {
            SysDelay(100);
            ret = write(PrnFd, PrnDivBuf, 0);
            if (ret != PRN_BUF_FULL)
                break;
        }
    }
    return ret;
}
第一性原理：打印指令不是“数据包”，而是会改变解释器状态的字节流。解析层必须保持顺序和状态连续性，渲染层才知道当前字节应该变成字符、图像、走纸、切刀还是状态回复。

状态与控制

状态通过共享内存直接暴露，状态变化通过 sem_status 唤醒订阅者。控制命令则通过 print_msg_t 进入 print_mqueue_thread()，由服务端执行实际硬件操作或回填共享内存结果。

类别通路例子
状态读取客户端读取 shm->status缺纸、开盖、过热、打印中、任务编号、切刀状态。
状态订阅客户端增加订阅计数，服务端状态变化时 sem_post(sem_status)print_status_change_subscribe()。
控制消息mq_send() 到 /mq.print_server设置浓度、速度、阈值、走纸、切刀、获取设置、坏点检测、缝隙检测。
实时命令客户端预识别后发送 PRINT_CMD_HANDLE_REALTIME_COMMANDESC/POS 实时状态查询、部分商米私有指令。
typedef union {

第一性原理：状态是“当前事实”，适合共享读取；控制是“事件请求”，适合排队处理。把事实和事件分开，可以减少同步冲突，也让硬件操作集中在服务端。

重打与异常

任务设计保留了完整 buffer 链、buffer id、发送进度和任务标志，所以服务端能在缺纸、开盖、删除任务、多份打印和不允许重打之间做不同策略。

  • 缺纸或开盖时，普通模式下如果任务允许重打，服务端清空打印机缓冲并把 buf_to_send 和 buf_offset 复位到任务开头，或者复位到最近切刀位置。
  • 标记 TASK_FLAG_DONT_REPRINT 的任务，发送过的 buffer 可以提前释放，节省共享内存。
  • 标记 TASK_FLAG_RECV_NOT_COMPLETE 的任务可以边接收边打印，但发送时保留至少一个 buffer 余量，避免解释器吃到尚未稳定的尾部。
  • 多份打印通过 copies 递减实现；当前任务所有 buffer 发送并确认后，如果还有份数，游标回到任务头重新发送。
  • 删除任务并不一定立即从所有位置摘除，而是设置 TASK_FLAG_DONT_PRINT，调度到安全点后丢弃或释放。
第一性原理：热敏打印是不可回滚的物理过程，软件只能回滚“尚可重发的输入”。因此任务数据链就是恢复能力的上限；一旦为了省内存提前释放，就必须接受不可重打。

整体总结

数据结构模型

print_shm_t 是共享内存根对象。打印数据由 print_task_t 描述，实际字节保存在 print_buffer_t 链表中；通道用于隔离输入，全局队列用于确定硬件发送顺序。

IPC 分工

大块、可回放的数据走共享内存；小型、即时的控制命令走 POSIX 消息队列；互斥和订阅通知走命名信号量。

打印主路径

应用写入任务链，服务端按通道超时或长度完成收口，推进全局队列，再按 ESC/POS 或 TSPL 分流解析，最终渲染为点行或控制 ioctl 写入内核打印设备。

可靠性策略

任务保留 buffer 链用于缺纸重打、多份打印和删除延迟处理；不需要重打的任务可提前释放已发送 buffer，换取共享内存容量。

第一性原理：这个模块的核心不是“打印 API 有多少函数”，而是“一个有状态字节协议如何在多输入、单硬件、可恢复的约束下保持顺序”。共享内存保存可回放事实，消息队列传递事件，服务端把事件和事实折叠成唯一的硬件时间线。
