打印模块 IPC：信号量与共享内存分析

基于 libcommon/print_api.c、libcommon/include/print_api.h、server/print/src/print_queue.c、server/print/src/print_server.c 的本地代码整理。本文档为单独 HTML 文件，样式内联，不依赖外部网络、图片或脚本。

1个命令消息队列 /mq.print_server
总体结论：打印服务进程是 IPC 对象的创建者和主要消费者，业务进程通过 print_api_init() 打开已存在的命名信号量和共享内存，再直接修改共享队列或通过消息队列发送控制命令。信号量按资源维度拆分，主要保护 buffer pool、task pool、各 channel 队列、全局打印队列和状态通知。

2. 主共享内存布局

print_shm_t in /shm.print_server
buffer_poolPRINT_BUFFER_POOL_SIZE 个 print_buffer_t，保存打印数据块链表。
task_poolPRINT_TASK_POOL_SIZE 个 print_task_t，保存任务元信息、链表头尾、计数、标志位。
channels[]每个输入来源一个 channel，暂存正在接收的任务和 channel 私有待打印队列。
task_queue服务端全局打印队列，含当前待发送 buffer、偏移和最近 channel。
status打印状态快照：缺纸、开盖、过热、切刀、打印距离等。
settingsGET_SETTINGS 的返回缓存，使用 DataReady 作为完成标志。
request订阅数量、日志开关、打印策略、纸未取动作等请求配置。
bad_points / paper_location坏点检测和黑标/纸张位置查询结果。
data_collection订单补打、数据采集所需的 seqnum、时间戳、任务 ID、任务索引。
print_log_mem[]每个任务索引对应订单号、接收时间、完成时间。
memory model

访问模型

  • 服务端和客户端都以 PROT_READ|PROT_WRITE、MAP_SHARED 映射同一块共享内存。
  • 共享结构中的索引使用 uint16_t，无效值统一为 0xFFFF。
  • 数据块、任务、channel 和队列通过索引链表组织，避免跨进程保存裸指针。
  • 共享内存本身不提供同步；同步依赖命名信号量和约定的访问顺序。
注意：ftruncate(fd, sizeof(print_shm_t)) 只调整共享内存大小，当前代码随后通过各 init 函数初始化 pool、channel、queue 和日志字段，但并未整体 memset(shm, 0, sizeof(*shm))。如果新增字段，需要确认是否有显式初始化路径。

4. 主要数据流

print data path

业务进程提交打印数据

  1. 业务侧调用 print_channel_send_data_with_flag() 或 print_channel_send_data()。
  2. IS_INITIALIZED() 确保信号量和共享内存已打开。
  3. 按 channel 加锁：channels_lock(chn_id)。
  4. 若是新任务，从 task_pool 分配 print_task_t，从 buffer_pool 分配一个或多个 print_buffer_t，把业务数据拷贝进共享内存。
  5. 任务进入 channel 的私有队列或保持为正在接收任务；服务端主循环 flush_channels() 根据超时或完整接收条件把它转入全局 task_queue。
  6. send_next_buffer() 从全局队列取 buffer，调用底层打印机接口发送数据。
  7. 打印完成后 remove_task_from_queue() 更新 last_completed_task_id，task_free() 归还 task 和 buffers。
command path

控制命令路径

print_set_density()、print_get_settings()、print_get_bad_points() 等不是直接改复杂状态，而是构造 print_msg_t 经 /mq.print_server 发送到服务端 print_mqueue_thread()。服务端线程执行硬件调用，再把结果写回共享内存字段。

status path

状态变化通知

客户端调用 print_status_change_subscribe() 后创建状态线程，线程先增加 shm->request.status_change_subscribers，再阻塞在 sem_wait(sem_status)。服务端 print_queue_run() 检测硬件状态变化后按订阅数量执行 sem_post(sem_status)。

collection path

数据采集同步

服务端在采集条件满足时写入 shm->data_collection 或 third_party_data_collection，随后调用 print_data_share_notify()。第三方应用通过 print_data_share_wait() 获取通知和互斥，读取共享字段后调用 print_data_share_complete() 通知服务端完成。

5. 锁顺序与同步边界

场景典型锁顺序说明
业务进程向 channel 追加数据channels_lock(i) → task_pool_lock() / buffer_pool_lock()append_data_to_channel() 持有 channel 锁期间会申请 task 和 buffer。
服务端 flush channelchannels_lock(i) → task_queue_lock()flush_channels() 持有 channel 锁，把完成任务转入全局队列。
任务完成释放task_queue_lock() → 后续 task_pool_lock() / buffer_pool_lock()remove_task_from_queue() 只短暂持有全局队列锁，释放资源在外层继续进行。
状态订阅计数task_queue_lock()当前使用全局队列锁保护 request.status_change_subscribers，语义上可工作，但名称和保护对象不完全匹配。
状态字段读取多数路径无互斥shm->status 作为状态快照直接读写，可能读到更新中的字段组合；状态变化通知只负责唤醒，不保护拷贝。
实现特征：SEM_TIMEDWAIT() 调用 sem_timedwait() 后没有检查返回值。如果 100ms 超时或被信号中断，宏仍会继续执行临界区，锁保护会失效。这是整个同步设计里最值得优先核查的问题。

6. 代码风险与改进建议

risk

sem_timedwait() 返回值被忽略

主打印服务和客户端 API 都用相同的 SEM_TIMEDWAIT 宏，但未判断返回值。超时后继续改共享链表，会破坏互斥语义。

risk

部分共享字段无明确锁

status、settings.DataReady、bad_points、paper_location 等字段多为直接读写。若要求一致快照，应增加锁、序列号或单写多读约束说明。

warn

IPC 对象未见清理路径

当前检索到创建和打开，未看到主打印对象的 sem_unlink() / shm_unlink()。重启后依赖重新初始化共享内存内容，信号量历史计数也需要关注。

warn

sem_open() 未用 O_EXCL

服务端使用 O_CREAT 打开命名信号量。若旧对象残留，初值参数不会重置已有信号量，可能保留非预期计数。

warn

request 信号量语义弱

sem_request 创建并在退出时 post，但等待宏没有实际调用。若保留，应补充用途；若废弃，应删除或改名避免误导。

good

共享内存使用索引链表

print_task_t 和 print_buffer_t 保存索引而不是进程内指针，这是跨进程共享内存里正确且稳妥的设计。

优先级建议：若后续要修复，第一优先级是让锁宏返回错误并让调用者在获取锁失败时退出当前操作；第二优先级是梳理服务端启动时旧命名信号量的残留处理策略；第三优先级是给状态类共享字段定义一致性模型。
