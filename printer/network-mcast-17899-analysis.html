server/network/src/mcast.c 组播发现实现分析

本文分析当前 server/network/src/mcast.c 的 UDP 17899 发现服务：它如何接收发往 224.0.0.1:17899 的组播请求，为什么代码里没有显式 IP_ADD_MEMBERSHIP 仍可能收到，以及在 eth0/wlan0 同时联网时回复会从哪个接口发出。

结论先行：这个实现本质上是“监听所有本地 IPv4 地址的 UDP 17899，然后把设备网络信息单播回请求来源”。它不是 mDNS，也不是一个严格按接口加入自定义组的 multicast responder。收到请求后，sendto() 的目的地址是 recvfrom() 得到的对端单播地址，回复出接口由 Linux 路由表选择，不由接收接口强绑定。
第一性原理：组播负责“把一个请求送到同一链路上的多个主机”；这个程序负责“把收到的发现请求转成一份单播设备信息响应”。

官方依据

来源关键原文含义对本实现的解释
RFC 1112: Host Extensions for IP MulticastingIP multicast 是把 IP 数据报发送给一个 host group；224.0.0.1 是 all-hosts 永久组，用于直接连接网络上的所有 multicast hosts；Level 2 主机在每个接口初始化时都保持 all-hosts 组成员关系。客户端把请求发到 224.0.0.1:17899，目的不是某一台设备 IP，而是链路上所有 IPv4 multicast 主机。设备内核本身通常已经是 224.0.0.1 成员。
Linux ip(7)Linux IPv4 实现包含符合 RFC 1112 的 Level 2 multicasting；INADDR_ANY 绑定表示 socket 绑定到所有本地接口；224.0.0.0/4 是 multicast 地址。bind(0.0.0.0:17899) 不是只监听某个 IP，而是监听所有本地接口上的 UDP 17899。
Linux IP_MULTICAST_ALLIP_MULTICAST_ALL 默认值为 1：socket 会接收系统全局已加入的所有 multicast group 的消息；设为 0 后才只接收这个 socket 自己显式加入的 group。mcast.c 没有 IP_ADD_MEMBERSHIP，但在 Linux 默认策略下，绑定 INADDR_ANY:17899 的 UDP socket 仍可能收到系统全局组成员 224.0.0.1 上的 UDP 17899 报文。
Linux IP_ADD_MEMBERSHIPIP_ADD_MEMBERSHIP 是 socket 显式加入 multicast group 的接口。当前文件没有调用它，因此对自定义组播地址不具备显式、可移植的接收保证；它依赖 all-hosts 特性和 Linux 默认交付策略。
第一性原理：“收到组播”需要两层条件：IP 层愿意接收这个组，UDP 层有 socket 匹配端口。本实现的特殊点是 IP 层的 224.0.0.1 成员关系不是由应用显式建立的。

1. 组播是如何工作的

层发送请求时设备接收时设备回复时
IP 层目的 IP 是 224.0.0.1，表示 all-hosts multicast group。内核判断本机是否应接收该 multicast group，再继续交给 UDP。目的 IP 变成请求方的单播 IP，不再是 multicast。
UDP 层目的端口是 17899。bind(0.0.0.0:17899) 的 socket 匹配该端口。sendto() 使用 recvfrom() 填回的 addr，即请求方 IP 和源端口。
应用层请求内容没有被解析，任意 payload 都可以触发响应。收到后填充设备信息结构体。返回固定二进制结构 host_cmd_dev_info_t。
第一性原理：组播请求解决“客户端不知道设备 IP”的问题；单播响应解决“设备把自己的可连接信息直接告诉发起者”的问题。

2. 本地只是监听 17899，为什么能收到发给 224.0.0.1 的报文

因为 socket 监听的是 0.0.0.0:17899，而不是监听某个单播 IP。Linux 先在 IP 层决定这台主机是否接收 224.0.0.1 这个 multicast datagram；通过后，UDP 层再按端口 17899 找 socket。mcast.c 恰好有一个绑定到所有本地接口的 UDP 17899 socket。
条件当前是否满足说明
目的地址是本机愿意接收的 multicast group通常满足224.0.0.1 是 all-hosts 永久组，RFC 1112 要求 Level 2 主机在每个接口保持成员关系。
socket 绑定所有本地地址满足addr.sin_addr.s_addr = INADDR_ANY，Linux ip(7) 说明 INADDR_ANY bind 绑定所有本地接口。
UDP 端口匹配满足addr.sin_port = htons(17899)，请求必须发到 UDP 17899。
socket 自己显式加入 group没有代码没有 setsockopt(IP_ADD_MEMBERSHIP)。在 Linux 默认 IP_MULTICAST_ALL=1 时，socket 可接收系统全局已加入组的 multicast 消息；这解释了为什么仍能收到 224.0.0.1。
重要边界：如果客户端发到一个普通自定义组播地址，例如 239.x.x.x:17899，当前代码没有显式加入该组，不能按可移植方式保证收到。若要做严格的应用组播发现，应显式对目标组和接口调用 IP_ADD_MEMBERSHIP，并视需求设置 IP_MULTICAST_ALL=0、IP_PKTINFO 或 SO_BINDTODEVICE。
第一性原理：UDP socket 匹配的是“本地地址 + 端口”；multicast 是否进入这台主机，是 IP 层 group membership 和系统交付策略先决定的。

3. 以太网发送请求，会从 wlan0 回复吗

当前 mcast.c 没有记录接收接口，也没有把回复绑定到接收接口。它使用同一个未连接 UDP socket 调 sendto(fd, &info, ..., &addr)，目的地址是请求方的单播 IP。因此回复出接口由 Linux 路由表按“到请求方 IP 的最佳路由”决定。
场景请求如何进来回复可能从哪里出去结论
eth0 和请求方在同一有线网段，wlan0 在另一个网段请求方发到 224.0.0.1:17899，设备 eth0 收到，recvfrom() 得到请求方有线 IP。路由表通常选择 eth0，因为请求方 IP 在 eth0 直连网段。通常从 eth0 回复，不会从 wlan0 回复。
eth0 收到请求，但路由表到请求方 IP 的最佳路径是 wlan0可能出现在策略路由、metric 异常、两个接口同网段、默认路由偏向 wlan0 等配置下。sendto() 可能选择 wlan0 源地址/出接口。有可能从 wlan0 回复；这是当前实现没有绑定接收接口的直接后果。
eth0/wlan0 在同一个二层网络或同一 IP 网段同一个 224.0.0.1 请求可能被两个接口都收到，或者只有一个接口收到，取决于 AP/交换机/内核过滤。单 socket 无法区分接收接口；回复仍按路由选路。抓包可能看到看似“错接口”的回复，需要同时看路由表、源 IP、源 MAC。
想保证从收到请求的接口回复当前代码做不到严格保证。需要改为 recvmsg(IP_PKTINFO) 获取入接口和目的地址，再用 sendmsg(IP_PKTINFO) 或 SO_BINDTODEVICE/IP_MULTICAST_IF 控制出接口。这是实现增强点，不是当前行为。
第一性原理：接收路径由“包从哪块网卡进来”决定；发送路径由“目标单播地址如何路由”决定。当前代码没有把这两者绑定在一起。

抓包验证方法

目标操作过滤器预期
证明请求是组播Windows 或 Linux 客户端向 224.0.0.1:17899 发送任意 UDP payload。udp.port == 17899 && ip.dst == 224.0.0.1看到目的 IP 为 224.0.0.1，目的端口为 17899 的请求。
证明响应是单播继续抓同一次请求后的回包。udp.port == 17899 && ip.src == 设备IP响应目的 IP 是客户端单播 IP，不是 224.0.0.1；UDP 负载长度接近 host_cmd_dev_info_t 结构大小。
证明出接口由路由决定设备 eth0/wlan0 同时联网时，分别查看设备路由：ip route get 客户端IP。同时在 eth0/wlan0 两侧抓包。udp.port == 17899 && (ip.addr == 客户端IP || ip.addr == 设备eth0IP || ip.addr == 设备wlan0IP)回复出现在哪个接口，应与 ip route get 客户端IP 的 dev 一致，而不是必然等于收到请求的接口。
证明没有显式加入自定义组客户端改发到某个自定义组播地址，例如 239.1.2.3:17899，不改设备代码。udp.port == 17899 && ip.dst == 239.1.2.3通常不会收到设备响应；若收到，要检查系统是否有其他进程全局加入了该组，以及 IP_MULTICAST_ALL 策略。
第一性原理：抓包要同时看三件事：目的组播地址证明“请求如何扩散”，响应目的地址证明“回复是单播”，设备路由表证明“单播从哪块网卡出去”。

实现边界与改进建议

问题当前状态风险改进方向
没有显式加入组没有 IP_ADD_MEMBERSHIP。依赖 Linux all-hosts 与 IP_MULTICAST_ALL 默认行为；对其他组或其他系统不稳。为目标组播地址和目标接口显式调用 IP_ADD_MEMBERSHIP。
不知道接收接口使用 recvfrom()，没有启用 IP_PKTINFO。无法保证以太网收到就从以太网回复。改用 recvmsg() 获取 in_pktinfo.ipi_ifindex，回复时用 sendmsg() 指定出接口。
任意 payload 触发响应没有校验请求 magic、版本、长度或认证。局域网任意主机可获取 SN、型号、eth0/wlan0 配置。增加协议头、版本、最小长度检查；必要时增加认证或限制响应字段。
线程无退出机制while(1) 阻塞在 recvfrom()。服务退出/重启时不可控。增加停止标志、超时或事件 fd。
fd_set 未使用循环中设置 FD_ZERO/FD_SET，但没有 select()。代码噪声，不影响行为。删除无用 fd_set，或真正使用 select() 做退出/超时。
第一性原理：发现协议越底层，越要明确“谁能问、问什么、从哪里问、从哪里答”。当前实现足够简单，但接口归属和信息暴露边界不够明确。

总结

问题答案
组播是如何工作的？客户端向 224.0.0.1:17899 发送 UDP 包；设备内核接收 all-hosts 组播后，把端口 17899 的包交给 bind(0.0.0.0:17899) 的 socket；应用单播回复设备信息。
为什么只监听 17899 也能收到 224.0.0.1？224.0.0.1 是 all-hosts 组，Linux 主机通常全局加入；INADDR_ANY 绑定所有本地接口；IP_MULTICAST_ALL 默认允许 socket 接收系统全局已加入组的 multicast 消息。
以太网请求会从 wlan0 回复吗？通常不会，如果请求方 IP 在 eth0 直连网段，路由会选 eth0。但当前代码没有绑定接收接口，若路由表认为到请求方 IP 应走 wlan0，就可能从 wlan0 回复。
第一性原理：这个文件实现的是“简化版局域网设备发现”：组播只用于找到设备，真正响应是路由控制的单播。要做到严格多网口语义，需要把接收接口信息显式带入发送路径。
