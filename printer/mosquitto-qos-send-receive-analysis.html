QoS 0/1/2 报文发送与接收如何处理

本文基于当前工程内 Mosquitto 2.0.14 的 libmosquitto 源码，围绕 struct mosquitto、mosquitto_publish_v5()、send__publish()、handle__publish()、handle__pubackcomp()、handle__pubrec()、handle__pubrel() 分析 QoS 0/1/2 的发送和接收状态机。

阅读边界：这里只分析当前客户端库路径，不展开 broker-only 的 WITH_BROKER 数据库投递实现；表中涉及 msgs_in/msgs_out、packet 队列、callback 的部分，均对应当前 WITH_THREADING 开启后的 libmosquitto 客户端行为。
第一性原理：QoS 的差异不是“发送函数不同”，而是消息所有权转移时机不同。QoS0 发出即结束，QoS1 等 PUBACK，QoS2 通过 PUBREC/PUBREL/PUBCOMP 两阶段确认来避免重复交付。

协议语义：三种 QoS 的确认模型

QoS协议含义Packet Identifier确认包客户端必须保存的状态
QoS 0At most once，最多一次。底层网络尽力发送，不做确认和重发。没有。MQTT 要求 QoS0 PUBLISH 不带 Packet Identifier。无。不保存会话消息，只需要临时 packet 缓冲。
QoS 1At least once，至少一次。发送方保存消息直到收到 PUBACK，可能重复。必须有非零 mid，ACK 必须带相同 mid。PUBLISH → PUBACK。msgs_out 中保存未确认消息；接收方收到后发 PUBACK。
QoS 2Exactly once，正好一次。通过两阶段握手转移所有权，避免重复投递。必须有非零 mid，PUBREC/PUBREL/PUBCOMP 都绑定同一 mid。PUBLISH → PUBREC → PUBREL → PUBCOMP。发送方保存 msgs_out 状态；接收方保存 msgs_in 等 PUBREL。
第一性原理：QoS 越高，确认链越长，需要保存的中间状态越多。实现复杂度来自“断线、重复、乱序边界下仍能判断消息是否已经交付”。

源码入口：发送和接收各自从哪里分流

发送入口

mosquitto_publish() 直接转 mosquitto_publish_v5()。该函数校验 topic、payload、properties、max_qos、maximum_packet_size，然后生成 local_mid，按 QoS 分流：QoS0 直接 send__publish()；QoS1/2 构造 mosquitto_message_all 后进入 msgs_out。

接收入口

handle__packet() 按 MQTT Control Packet 高 4 位分派。CMD_PUBLISH 进入 handle__publish()；CMD_PUBACK/CMD_PUBCOMP 进入 handle__pubackcomp()；CMD_PUBREC 进入 handle__pubrec()；CMD_PUBREL 进入 handle__pubrel()。

API 发送: mosquitto_publish_v5()
  qos == 0 -> send__publish() -> send__real_publish() -> packet__queue()
  qos == 1 -> msgs_out(state=invalid) -> message__release_to_inflight() -> PUBLISH -> wait PUBACK
  qos == 2 -> msgs_out(state=invalid) -> message__release_to_inflight() -> PUBLISH -> wait PUBREC

网络接收: handle__packet()
  PUBLISH -> handle__publish()  // 按入站 QoS 分流
  PUBACK/PUBCOMP -> handle__pubackcomp()
  PUBREC -> handle__pubrec()
  PUBREL -> handle__pubrel()
第一性原理：发送路径按“应用想发什么”分流，接收路径按“线上收到什么控制包”分流。两者通过 Packet Identifier 和 msgs_in/out 状态表闭合。

发送总览：从 API 到 wire packet

第一性原理：发送路径先决定是否需要“会话级持久状态”。不需要确认的 QoS0 只排 packet；需要确认的 QoS1/2 先排 message，再由 inflight quota 释放成 packet。

QoS 0 发送：只编码 PUBLISH，不进入 msgs_out

步骤当前源码行为关键差异
API 校验mosquitto_publish_v5() 校验 QoS 范围、max_qos、topic、payload、properties、包大小。QoS0 仍会生成 local_mid 并可回填给调用者，但线上 PUBLISH 不带 Packet Identifier。
编码send__real_publish() 计算 packet length；if(qos > 0) packetlen += 2，QoS0 不加 mid 字段。packet->command = CMD_PUBLISH | dup<<3 | qos<<1 | retain，QoS0 的 QoS bits 为 0。
排队发送packet__queue() 将编码好的 packet 放入 out_packet，loop 线程写 socket。不进入 msgs_out.inflight，没有 ACK handler。
完成通知写完 QoS0 PUBLISH 后，库可触发 on_publish 表示 packet 已交给网络写路径。这不是 broker 确认，只是本地发送完成语义。
第一性原理：QoS0 没有远端确认，因此实现不能等待不存在的 ACK。它只需要保证 packet 格式正确并进入网络输出队列。

QoS 1 发送：保存消息，等待 PUBACK

步骤状态/函数作用
保存消息message = mosquitto__calloc(...)，复制 topic/payload/properties，message->msg.qos = 1，message->state = mosq_ms_invalid。发送方必须在收到 PUBACK 前保存消息。
释放到 inflightmessage__queue() 后调用 message__release_to_inflight()；如果 inflight_quota > 0，状态变 mosq_ms_wait_for_puback。实现 MQTT 5 Receive Maximum / send quota；没有额度时消息留在队列中。
发送 PUBLISHsend__publish() 编码 PUBLISH，写入 mid，排入 packet 队列。QoS1 PUBLISH 的 Variable Header 必须包含 Packet Identifier。
收到 PUBACKhandle__pubackcomp(mosq, "PUBACK") 读取 mid，校验 reason code，message__delete(... qos=1)。删除出站会话消息，mid 可复用。
通知应用删除成功后持 callback_mutex 调用 on_publish/on_publish_v5。应用只收到一次“发布完成”通知。
第一性原理：QoS1 的可靠性来自发送方保留副本直到 ACK。重复可以发生，但丢失被 ACK 前的本地副本所约束。

QoS 2 发送：两阶段确认，等待 PUBREC 再等待 PUBCOMP

阶段当前源码行为状态意义
发送 PUBLISHmessage__release_to_inflight() 将 QoS2 的 mosq_ms_invalid 改为 mosq_ms_wait_for_pubrec，然后 send__publish()。等待接收方确认已接管消息。
收到 PUBREChandle__pubrec() 读取 mid。正常 reason code 下调用 message__out_update(... mosq_ms_wait_for_pubcomp, 2)，再 send__pubrel()。发送方不再重发原 PUBLISH，转为等待最终 PUBCOMP。
收到 PUBCOMPhandle__pubackcomp(mosq, "PUBCOMP") 以 qos=2 删除 msgs_out 中的消息。QoS2 交换完成，Packet Identifier 可复用。
失败 reason codeMQTT 5 下 handle__pubrec() 若 reason code >= 0x80，会删除消息、通知 on_publish_v5，并增加 send quota。失败 ACK 也结束这条 PUBLISH 的会话状态。
第一性原理：QoS2 的关键不是多发两个包，而是把“消息是否可交付”和“Packet Identifier 是否可释放”拆成两个阶段。

接收总览：handle__publish() 按 QoS 分流

第一性原理：入站 QoS 的核心是“什么时候把消息交给应用”。QoS0/1 可以立即回调；QoS2 要等 PUBREL 后才回调，以阻止重复 PUBLISH 导致重复交付。

QoS 0 接收：解析后立即回调

步骤源码行为协议意义
解析固定头message->dup、message->msg.qos、message->msg.retain 从 in_packet.command 位域解析。QoS0 没有 mid，也没有 ACK 流程。
解析 topic/payloadpacket__read_string() 读取 topic，剩余长度减当前位置得到 payloadlen，再读 payload。topic 不能为空；payload 可以为 0 长度。
应用回调持 callback_mutex，设置 in_callback=true，调用 on_message 和/或 on_message_v5。消息立即交给应用，不保存到 msgs_in。
清理message__cleanup()、mosquitto_property_free_all()。没有会话状态需要保留。
第一性原理：QoS0 接收方不承诺可靠交付，所以实现只做“解析成功则交付”，没有必要保存中间状态。

QoS 1 接收：发 PUBACK 并交付应用

当前源码中 QoS1 接收的顺序是：先 util__decrement_receive_quota()，再 send__puback()，随后回调应用，最后清理临时 message。send__puback() 内部会 util__increment_receive_quota() 并排队 PUBACK。这个实现把 QoS1 入站消息视为“收到即可确认”，应用处理不是 ACK 的前置条件。

第一性原理：QoS1 的 ACK 表示接收方已经接管消息，不表示应用业务已经处理成功。否则网络协议会被应用处理时间拖住。

QoS 2 接收：先存入 msgs_in，等 PUBREL 后交付

阶段源码行为为什么这样做
收到 QoS2 PUBLISHhandle__publish() 读 mid 和 payload，util__decrement_receive_quota()，send__pubrec()。PUBREC 表示接收方开始 QoS2 两阶段处理。
保存入站消息message->properties = properties，持 msgs_in.mutex 设置 state = mosq_ms_wait_for_pubrel，再 message__queue(... mosq_md_in)。重复 PUBLISH 或断线恢复时，库知道这个 mid 已经进入 QoS2 中间态。
收到 PUBRELhandle__pubrel() 先 send__pubcomp()，然后 message__remove(mosq, mid, mosq_md_in, &message, 2)。PUBCOMP 完成协议交换；移除成功才允许应用回调。
回调应用移除 msgs_in 成功后，调用 on_message/on_message_v5，再清理 message。“只回调一次”的保障来自 message__remove() 成功条件。
未知 midhandle__pubrel() 收到不存在的 mid 时返回成功，不重复回调。允许处理重发 PUBREL，不把重复协议包变成重复业务消息。
第一性原理：QoS2 接收方必须把“已经收到 PUBLISH”持久到会话状态里，直到 PUBREL 到达后才交付应用；否则重复 PUBLISH 会造成重复业务投递。

状态表：enum mosquitto_msg_state 如何对应 QoS

状态方向触发点下一步
mosq_ms_invalid出站QoS1/2 message 刚进入 msgs_out，还未获得 inflight quota。QoS1 → wait_for_puback；QoS2 → wait_for_pubrec。
mosq_ms_wait_for_puback出站 QoS1message__release_to_inflight() 发送 QoS1 PUBLISH 后。收到 PUBACK 后 message__delete() 并回调 on_publish。
mosq_ms_wait_for_pubrec出站 QoS2发送 QoS2 PUBLISH 后。收到 PUBREC 后改为 wait_for_pubcomp 并发送 PUBREL。
mosq_ms_wait_for_pubcomp出站 QoS2收到 PUBREC 并发送 PUBREL 后。收到 PUBCOMP 后删除 message 并回调 on_publish。
mosq_ms_wait_for_pubrel入站 QoS2收到 QoS2 PUBLISH，发出 PUBREC，并保存到 msgs_in。收到 PUBREL 后发 PUBCOMP，移除 message，并回调 on_message。
mosq_ms_publish_qos1/mosq_ms_publish_qos2重连/重发message__reconnect_reset() 或 message__retry_check() 将未完成消息恢复为可重发状态。重新发送 PUBLISH，DUP 可置位。
mosq_ms_resend_pubrel出站 QoS2 重连断线前已进入等待 PUBCOMP，重连后需要重发 PUBREL。发送 PUBREL，继续等待 PUBCOMP。
第一性原理：状态枚举记录的不是“函数执行到哪里”，而是“协议所有权转移到了哪一步”。这让断线重连和重复 ACK 都能按同一张状态表恢复。

配额、重连与重试

发送配额

msgs_out.inflight_quota 限制 QoS>0 的未确认 PUBLISH 数量。message__release_to_inflight() 只有在 quota 大于 0 时才把 mosq_ms_invalid 消息变为等待 ACK 状态并发送。

接收配额

handle__publish() 在 MQTT 5 且 QoS>0 时检查 msgs_in.inflight_quota；收到 QoS1/2 PUBLISH 后 decrement，发送 PUBACK/PUBCOMP 或失败 PUBREC 时 increment。

重连恢复

message__reconnect_reset() 重置 quota，并将未完成出站消息恢复为 publish_qos1、publish_qos2 或 resend_pubrel 等状态。

重试检查

message__retry_check() 遍历 msgs_out，对需要重发的 PUBLISH 设置 dup=true 并重新 send__publish()，对 QoS2 中间态可重发 PUBREL/PUBREC。

MQTT 5 规范要求 Receive Maximum 达到上限后不能继续发送 QoS>0 PUBLISH，但仍应继续处理其他控制包。当前实现把这条规则落在 inflight_quota 和 message__release_to_inflight() 上。
第一性原理：可靠消息不是无限并发。配额把“可靠性状态”限制在可控窗口内，避免一个连接积累无限未确认消息。

总结：三种 QoS 的实现差异

QoS发送侧接收侧完成条件回调时机
0直接编码 PUBLISH，排入 packet 队列，不进 msgs_out。解析 PUBLISH 后立即 on_message。本地 packet 写完；无远端 ACK。发送侧可在写完后 on_publish；接收侧解析后立即 on_message。
1保存到 msgs_out，发送 PUBLISH，状态 wait_for_puback。收到 PUBLISH 后发 PUBACK，再 on_message。发送侧收到 PUBACK 并删除 message。发送侧收到 PUBACK 后 on_publish；接收侧发送 PUBACK 后 on_message。
2保存到 msgs_out，PUBLISH 后等 PUBREC，发 PUBREL 后等 PUBCOMP。收到 PUBLISH 后发 PUBREC 并保存到 msgs_in；收到 PUBREL 后发 PUBCOMP、移除并回调。发送侧收到 PUBCOMP；接收侧成功移除 msgs_in。发送侧 PUBCOMP 后 on_publish；接收侧 PUBREL 后 on_message。
把这三条路径放回 struct mosquitto 看：in_packet/current_out_packet/out_packet 解决字节流收发，msgs_in/msg_out 解决 QoS 会话状态，callback_mutex 解决应用回调边界，mid_mutex 解决 Packet Identifier 分配。
第一性原理：QoS 实现的本质是“消息状态机 + 线性字节流编码”。packet 队列负责把控制包发出去，message 队列负责记住协议是否完成。
