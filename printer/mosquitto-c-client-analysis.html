设计哲学

🎯 极简主义

单一 struct mosquitto 对象承载全部状态，所有 API 以此为第一参数，无隐式全局状态（除引用计数外）。

⚡ 事件驱动

基于 select()/pselect() 的 I/O 多路复用，回调函数驱动业务逻辑，与 Reactor 模式高度吻合。

🔀 可选多线程

loop_start() 启动后台线程自动处理 I/O，socketpair 机制实现线程安全的跨线程唤醒。

🔌 可扩展传输

TCP/TLS/WebSocket/SOCKS5 通过枚举 mosq_t_* 统一切换，核心逻辑与传输无关。

🧠 第一性原理

libmosquitto 的本质是：将 MQTT 协议的有限状态机（FSM）映射到 C 结构体字段上，以 socket fd + select() 作为事件感知机制，以函数指针数组（回调）作为用户扩展点，以 out_packet 链表作为发送缓冲区。 一切复杂性都源于需要在单一 C 进程中，跨状态、跨时间地维护"连接"这个抽象概念的完整性。

MQTT v3.1.1 · v5.0 OASIS 标准

libmosquitto 实现了 MQTT v3.1、v3.1.1 和 v5.0 协议。理解协议包格式是理解客户端实现的前提。

🧠 第一性原理

MQTT 本质上是一个面向连接、基于发布/订阅的应用层协议，底层依赖 TCP 的可靠传输。 libmosquitto 将 MQTT 的控制包序列转化为枚举状态转换，让每个网络字节都在明确的状态上下文中被解析，消除了歧义。 协议的复杂性（QoS 握手、保活、遗嘱）被封装为状态机变迁，用户只需关注回调事件。

设计要点
struct mosquitto 通过 mosquitto_calloc(1, sizeof(struct mosquitto)) 在堆上分配，所有指针字段在 mosquitto__destroy() 中逐一释放，遵循严格的所有权语义。 
/* 来源: lib/libmosquitto.c - mosquitto_new() */
struct mosquitto *mosquitto_new(const char *id, bool clean_start, void *userdata) {
    struct mosquitto *mosq = NULL;
    
    /* clean_start=false 且没有 id 时，必须返回错误（MQTT 规范要求） */
    if(clean_start == false && id == NULL){ errno = EINVAL; return NULL; }
    
    mosq = (struct mosquitto *)mosquitto_calloc(1, sizeof(struct mosquitto));
    if(mosq){
        mosq->sock = INVALID_SOCKET;
        mosq->sockpairR = INVALID_SOCKET;
        mosq->sockpairW = INVALID_SOCKET;
        mosquitto_reinitialise(mosq, id, clean_start, userdata);
    }
    return mosq;
}

🧠 第一性原理

struct mosquitto 是「上下文对象」模式的典型 C 实现。在没有类和 this 指针的 C 语言中，将所有状态打包进一个结构体，并把结构体指针作为所有函数的第一个参数，等价于面向对象语言中的实例方法。 这种设计允许多个客户端实例并发存在，每个实例完全隔离，因为它们的状态不共享任何全局变量。

数据包结构设计

packet_mosq.c / packet_mosq.h mosquitto_internal.h

libmosquitto 将网络数据包抽象为两套结构：出站的 mosquitto__packet（链表节点）和入站的 mosquitto__packet_in（单个填充缓冲区）。

struct mosquitto__packet {

};

入站数据包结构

/* 来源: lib/mosquitto_internal.h */
struct mosquitto__packet_in {
    uint8_t  *payload;              /* 动态分配的载荷缓冲区 */
    uint32_t  remaining_mult;       /* 变长解码乘数 */
    uint32_t  remaining_length;     /* 累计解码得到的剩余长度 */
    uint32_t  packet_length;        /* 完整包总长 */
    uint32_t  to_process;           /* 还需读多少字节 */
    uint32_t  pos;                  /* payload 写指针 */
    uint16_t  packet_buffer_pos;    /* 预读缓冲区消费指针 */
    uint16_t  packet_buffer_size;   /* 4096 字节预读缓冲区大小 */
    uint8_t   command;              /* 已读到的命令字节 */
    uint8_t  *packet_buffer;        /* 4096 字节预读缓冲区 */
    int8_t    remaining_count;      /* 负值=正在读；正值=已完成 */
};
关键设计：4096 字节预读缓冲区
libmosquitto 使用一个 4096 字节的 packet_buffer 做预读（在 alloc_packet_buffer() 中分配），避免频繁 syscall。每次 read() 可能读取多个包的数据，通过 packet_buffer_pos 和 packet_buffer_to_process 追踪消费进度。

变长剩余长度编码实现

/* 来源: lib/packet_mosq.c - packet__alloc() */
int packet__alloc(struct mosquitto__packet **packet,
                  uint8_t command, uint32_t remaining_length)
{
    uint8_t remaining_bytes[5] = {0}, byte;
    int8_t  remaining_count = 0;
    do{
        byte = remaining_length % 128;
        remaining_length = remaining_length / 128;
        if(remaining_length > 0){
            byte = byte | 0x80;   /* 设置最高位为继续标志 */
        }
        remaining_bytes[remaining_count] = byte;
        remaining_count++;
    }while(remaining_length > 0 && remaining_count < 4);
    
    /* 一次 malloc 包含头部 + payload，避免额外指针 */
    packet_length = 1 + remaining_count + remaining_length_stored;
    *packet = mosquitto_malloc(
        sizeof(struct mosquitto__packet) + packet_length + WS_PACKET_OFFSET);
    ...
}

🧠 第一性原理

数据包设计遵循「流式、零拷贝友好」原则：出站包通过链表延迟写出，支持部分写（to_process/pos跟踪进度），一次 write() 失败不丢数据； 入站包通过有限状态机（command → remaining_length → payload）逐步填充，一次 read() 不足时保存进度下次继续，完全适应非阻塞 socket 的边沿触发模型。

库级初始化

在调用任何其他 API 之前，必须调用 mosquitto_lib_init()。该函数使用引用计数，支持多次调用。

/* 来源: lib/libmosquitto.c */
static unsigned int init_refcount = 0;  /* 唯一全局变量 */

int mosquitto_lib_init(void) {
    if(init_refcount == 0){
        mosquitto_time_init();  /* 初始化时钟 */
        net__init();            /* 网络初始化（WSAStartup on Windows）*/
    }
    init_refcount++;
    return MOSQ_ERR_SUCCESS;
}

int mosquitto_lib_cleanup(void) {
    if(init_refcount == 1){ net__cleanup(); }
    if(init_refcount > 0){ --init_refcount; }
    return MOSQ_ERR_SUCCESS;
}

socketpair 的作用

关键机制：线程间唤醒
net__socketpair(&mosq->sockpairR, &mosq->sockpairW) 创建一对互连的 socket。 当在多线程模式下，应用线程调用 mosquitto_publish() 时，会向 sockpairW 写入一个字节， 使 sockpairR 在 select() 中变得可读，从而立即唤醒后台 I/O 线程处理新数据包，消除不必要的 keepalive 超时等待。

🧠 第一性原理

初始化的本质是「确定性初始状态」：mosquitto_calloc 用零初始化内存（等价于 memset(0)），然后有选择地设置非零默认值。 这保证了在最坏情况（如部分初始化后内存不足返回失败）下，mosquitto__destroy() 也能安全地检查 NULL 并跳过释放，防止 double-free。 引用计数确保网络子系统精确地初始化和清理一次，适应库被多次动态加载的场景。

断开连接

/* 来源: lib/connect.c */
int mosquitto_disconnect_v5(struct mosquitto *mosq,
                             int reason_code,
                             const mosquitto_property *properties) {
    /* 1. 先设置状态，防止自动重连 */
    mosquitto__set_state(mosq, mosq_cs_disconnected);
    mosquitto__set_request_disconnect(mosq, true);
    
    if(!net__is_connected(mosq)){ return MOSQ_ERR_NO_CONN; }
    
    /* 2. 发送 DISCONNECT 控制包（MQTT v5 支持 reason code）*/
    return send__disconnect(mosq, (uint8_t)reason_code, outgoing_properties);
    /* 3. packet__write() 发送完 DISCONNECT 后自动 shutdown socket */
}
注意：优雅断开 vs 强制断开
mosquitto_disconnect() 发送 DISCONNECT 包后等待写完才关闭 socket（优雅）， 而 do_client_disconnect()（内部使用）直接 net__socket_close()，用于错误场景的强制断开，然后触发 on_disconnect 回调。

🧠 第一性原理

连接的本质是在两个端点间建立带状态的信道。libmosquitto 将连接过程分为「网络层建立」（connect() syscall）和「协议层握手」（CONNECT/CONNACK 交换）两个阶段， 前者由操作系统处理，后者由状态机处理。 request_disconnect 标志的存在使得「想要断开」和「实际断开」解耦，让 loop_forever 的重连逻辑能正确判断是否应该重试。

/* 来源: lib/loop.c */
int mosquitto_loop_read(struct mosquitto *mosq, int max_packets) {
    /* max_packets 由在途消息数决定，保证及时处理 PUBACK 等 */
    if(max_packets < 1){
        COMPAT_pthread_mutex_lock(&mosq->msgs_out.mutex);
        max_packets = mosq->msgs_out.queue_len;
        COMPAT_pthread_mutex_unlock(&mosq->msgs_out.mutex);
        ...
        if(max_packets < 1) max_packets = 1;
    }
    for(i=0; i < max_packets; i++){
        rc = packet__read(mosq);   /* 尝试读取一个完整 MQTT 包 */
        if(rc || errno == EAGAIN || ...) {
            return mosquitto__loop_rc_handle(mosq, rc);
        }
    }
    return rc;
}

/* 错误处理：关闭 socket，根据状态决定是否触发 on_disconnect */
static int mosquitto__loop_rc_handle(struct mosquitto *mosq, int rc) {
    if(rc){
        net__socket_close(mosq);
        state = mosquitto__get_state(mosq);
        if(state == mosq_cs_disconnecting || state == mosq_cs_disconnected){
            rc = MOSQ_ERR_SUCCESS;  /* 正常断开，不算错误 */
        }
        callback__on_disconnect(mosq, rc, NULL);
    }
    return rc;
}

loop_forever 重连逻辑

/* 来源: lib/loop.c - mosquitto_loop_forever() 简化版 */
while(mosq->run){
    /* 正常运行循环 */
    do{
        rc = mosquitto_loop(mosq, timeout, max_packets);
    }while(mosq->run && rc == MOSQ_ERR_SUCCESS);
    
    /* 致命错误直接退出 */
    switch(rc){
        case MOSQ_ERR_NOMEM: case MOSQ_ERR_PROTOCOL: ... return rc;
    }
    
    /* 重连延迟（支持指数退避）*/
    if(mosq->reconnect_exponential_backoff){
        reconnect_delay = mosq->reconnect_delay *
                         (mosq->reconnects+1) * (mosq->reconnects+1);
    }else{
        reconnect_delay = mosq->reconnect_delay * (mosq->reconnects+1);
    }
    reconnect_delay = MIN(reconnect_delay, mosq->reconnect_delay_max);
    
    interruptible_sleep(mosq, reconnect_delay); /* 可被 sockpair 唤醒 */
    mosquitto_reconnect(mosq);
}

🧠 第一性原理

事件循环的本质是「把 CPU 时间精确地分配给有事发生的 socket」。 select() 是操作系统提供的多路 I/O 通知机制，libmosquitto 在其上构建了 MQTT 语义层： 「可读」→ 解析协议包 → 触发回调；「可写」→ 冲刷出站队列。 mosquitto_loop_misc() 处理时间驱动的事件（keepalive），与 I/O 驱动的事件（read/write）正交组合，覆盖了所有 MQTT 通信场景。

void mosquitto_message_callback_set(struct mosquitto *mosq,

回调触发（callback_depth 防止递归死锁）

/* 来源: lib/callbacks.c - callback__on_message() */
void callback__on_message(struct mosquitto *mosq,
                          const struct mosquitto_message *message,
                          const mosquitto_property *properties) {
    LIBMOSQ_CB_message on_message;
    LIBMOSQ_CB_message_v5 on_message_v5;
    
    /* 1. 加锁读取函数指针（而非持锁调用），最小化锁竞争 */
    COMPAT_pthread_mutex_lock(&mosq->callback_mutex);
    on_message    = mosq->on_message;
    on_message_v5 = mosq->on_message_v5;
    COMPAT_pthread_mutex_unlock(&mosq->callback_mutex);
    
    /* 2. callback_depth 跟踪嵌套深度
     *    防止用户在回调内调用 publish()，触发 packet__write()
     *    导致在非线程模式下重入问题 */
    mosq->callback_depth++;
    
    if(on_message){
        on_message(mosq, mosq->userdata, message);
    }
    if(on_message_v5){
        on_message_v5(mosq, mosq->userdata, message, properties);
    }
    
    mosq->callback_depth--;
}
callback_depth 的精妙之处
在 packet__queue() 中检查：若 callback_depth == 0 且非线程模式，则立即调用 packet__write()。 若 callback_depth > 0（即在回调中调用了 publish），则只加入队列不立即写，防止在 read 回调中 re-enter write 路径造成栈溢出或锁死。

PUBLISH 包构建与写出

/* lib/send_publish.c - send__publish() 简化 */
int send__publish(struct mosquitto *mosq, uint16_t mid,
                  const char *topic, uint32_t payloadlen,
                  const void *payload, uint8_t qos, bool retain,
                  bool dup, ...) {
    
    /* 计算可变头 + 载荷长度 */
    tlen = strlen(topic);
    remaining_length = 2 + tlen + payloadlen;
    if(qos > 0) remaining_length += 2; /* + packet identifier */
    
    /* 分配数据包内存（一次 malloc） */
    command = CMD_PUBLISH;
    if(dup)    command |= 0x08;      /* bit3: DUP */
    command |= (qos << 1);           /* bit2-1: QoS */
    if(retain) command |= 0x01;     /* bit0: RETAIN */
    
    packet__alloc(&packet, command, remaining_length);
    
    /* 填充可变头：topic length(2) + topic string */
    packet__write_string(packet, topic, tlen);
    if(qos > 0) packet__write_uint16(packet, mid);
    
    /* 填充载荷 */
    if(payloadlen) packet__write_bytes(packet, payload, payloadlen);
    
    /* 入队，触发写出 */
    return packet__queue(mosq, packet);
}

QoS 0 / 1 / 2 发布流程对比

三种 QoS 发布时序对比
inflight 配额机制
msgs_out.inflight_quota（默认 20）控制最大在途消息数。当 QoS 1/2 消息入队时，若配额耗尽，消息进入 mosq_ms_queued 状态等待。 每收到一个 PUBACK/PUBCOMP，配额+1，下一个排队消息被发送。这防止了内存无限增长和网络拥塞。

🧠 第一性原理

发布的本质是「可靠地将字节序列从一端传输到另一端」。QoS 的三个等级对应「尽力而为」、「至少一次」、「恰好一次」三种不同的可靠性承诺。 实现可靠性的代价是：在客户端保留消息直到得到对端确认，这要求持久化 mid 和消息状态。 inflight_quota 是流量控制的实现，本质是令牌桶：发出一个消息消耗一个令牌，收到确认归还一个令牌。

SUBACK 处理

/* lib/handle_suback.c */
int handle__suback(struct mosquitto *mosq) {
    uint16_t mid;
    int qos_count, i;
    int *granted_qos;
    
    /* 解析 SUBACK: Packet ID + 每个主题的授予 QoS */
    packet__read_uint16(&mosq->in_packet, &mid);
    qos_count = mosq->in_packet.remaining_length - 2;
    granted_qos = mosquitto_malloc(qos_count * sizeof(int));
    
    for(i = 0; i < qos_count; i++){
        packet__read_byte(&mosq->in_packet, &byte);
        granted_qos[i] = byte; /* 0, 1, 2 或 0x80 (拒绝) */
    }
    
    /* 触发 on_subscribe 回调 */
    callback__on_subscribe(mosq, mid, qos_count, granted_qos, NULL);
    
    mosquitto_free(granted_qos);
    return MOSQ_ERR_SUCCESS;
}

Message ID 生成

/* lib/util_mosq.c */
uint16_t mosquitto__mid_generate(struct mosquitto *mosq) {
    uint16_t mid;
    COMPAT_pthread_mutex_lock(&mosq->mid_mutex);
    
    if(mosq->last_mid == 65535){
        mosq->last_mid = 0;   /* 环绕，跳过 0（MQTT 规范要求 mid != 0）*/
    }
    mosq->last_mid++;
    mid = mosq->last_mid;
    
    COMPAT_pthread_mutex_unlock(&mosq->mid_mutex);
    return mid;
}

QoS 消息持久化与重发

重发机制
QoS 1/2 消息在收到确认前保存在 msgs_out.inflight 链表中。 mosquitto__check_keepalive()（在 loop_misc() 中调用）扫描该链表， 对超时未确认的消息设置 DUP=1 重新发送。重连时 message__reconnect_reset() 将已发送但未确认的消息重置为待发送状态（而非丢弃）。
QoS可靠性重发消息排重适用场景
QoS 0 至多一次不重发不需要 传感器高频数据，丢几个无所谓
QoS 1 至少一次超时重发(DUP=1)Broker/接收端处理 命令下发，不能丢但可以重复
QoS 2 恰好一次4次握手保障两阶段提交 计费、事务性消息

🧠 第一性原理

QoS 的本质是「在不可靠信道上构建不同等级的可靠性保证」。 QoS 0 信任 TCP 本身的最大努力；QoS 1 用 ACK 确认语义检测丢失并重传，接受重复； QoS 2 通过两阶段提交（类似数据库 2PC）彻底消除重复，代价是 4 次往返时延。 mid 是这套机制的关键：它是消息的「事务 ID」，使双方能在乱序/重传场景中唯一识别一次投递。

/* 来源: lib/thread_mosq.c */
int mosquitto_loop_stop(struct mosquitto *mosq, bool force) {
    mosq->run = false;  /* 设置退出标志 */
    
    /* 向 sockpairW 写一字节，打破 select() 的等待 */
    if(mosq->sockpairW != INVALID_SOCKET){
        write(mosq->sockpairW, &sockpair_data, 1);
    }
    
    if(force){ COMPAT_pthread_cancel(mosq->thread_id); }
    COMPAT_pthread_join(mosq->thread_id, NULL);  /* 等待线程结束 */
    mosq->threaded = mosq_ts_none;
    return MOSQ_ERR_SUCCESS;
}

🧠 第一性原理

多线程的本质挑战是「共享可变状态的安全访问」。libmosquitto 通过细粒度互斥锁（每个字段组独立的 mutex）最小化锁竞争， 以 socketpair 作为跨线程的异步信号机制（避免轮询消耗 CPU），以 callback_depth 防止回调中再次进入写路径导致的锁序问题。 这三者合力解决了「高并发、低延迟、无死锁」的三角矛盾。

/* 来源: lib/loop.c - interruptible_sleep() */
/* 使用 pselect() 实现可中断的重连等待
 * 当 loop_stop() 被调用时，sockpairW 写入唤醒，立即退出等待 */
static int interruptible_sleep(struct mosquitto *mosq, time_t reconnect_delay) {
    FD_ZERO(&readfds);
    FD_SET(mosq->sockpairR, &readfds);  /* 监听 sockpairR */
    
    local_timeout.tv_sec = reconnect_delay;
    
    fdcount = pselect(mosq->sockpairR+1, &readfds, NULL, NULL,
                      &local_timeout, NULL);
    
    if(fdcount > 0 && FD_ISSET(mosq->sockpairR, &readfds)){
        /* 被唤醒（loop_stop 调用），清空管道数据 */
        read(mosq->sockpairR, &pairbuf, 1);
    }
    return MOSQ_ERR_SUCCESS;
}

重连时的消息保障

message__reconnect_reset 的行为
重连时，message__reconnect_reset(mosq, false) 将 msgs_out.inflight 中所有处于 wait_for_puback/wait_for_pubrec/wait_for_pubcomp 状态的消息重置回待发状态，而非丢弃。 这保证 QoS 1/2 消息在重连后会被重新发送（DUP=1），实现断线续传语义。 QoS 0 消息若尚未写入 OS，则丢失（符合 QoS 0 规范）。
重连配置项设置方式默认值说明
reconnect_delaymosquitto_reconnect_delay_set()1s基础重连延迟
reconnect_delay_maxmosquitto_reconnect_delay_set()1s最大重连延迟上限
reconnect_exponential_backoffmosquitto_reconnect_delay_set()false是否开启指数退避

🧠 第一性原理

重连机制的本质是「在瞬态故障下维持连接抽象的连续性」。 指数退避是信息论中的退让策略（Backoff）：在网络拥塞时，指数增长的等待时间让网络有机会恢复，同时避免大量客户端同时重连形成的「惊群效应」（Thundering Herd）。 reconnect_delay_max 的截断确保了系统最终一定会重连，不会因退避时间无限增长而永久放弃。

常见陷阱与注意事项

⚠️ 回调内不要调用 loop_*

在回调中调用 mosquitto_loop_forever() 会导致递归，应使用 mosquitto_publish()（安全，由 callback_depth 保护）。

⚠️ 消息载荷非 null 结尾

message->payload 是原始字节，message->payloadlen 是长度，打印时用 %.*s 而非 %s。

⚠️ clean_start=false 需要固定 clientID

mosquitto_new(NULL, false, ...) 会返回 EINVAL，必须提供稳定的客户端 ID 才能使用持久会话。

⚠️ loop_start 后调用 connect

loop_start() 后台线程会等待 state 离开 mosq_cs_new，需先调用 mosquitto_connect() 或 connect_async()。

🧠 第一性原理 — 终极总结

libmosquitto 解决的根本问题是：「如何用最小的系统资源，在不可靠的网络上，为受限设备提供可靠的发布/订阅通信」。

它的答案是： ① 单结构体上下文（极小内存足迹）+ ② select() 事件驱动（避免为每连接创建线程）+ ③ 分层有限状态机（协议复杂性被局部化）+ ④ 函数指针回调（用户逻辑与库逻辑解耦）+ ⑤ 指数退避重连（网络韧性）。

这五个设计决策互相正交、各司其职，共同实现了一个在嵌入式 MCU 和服务器端均可使用的轻量级、高可靠 MQTT 客户端库。 这正是「第一性原理」工程设计的典范：从问题本质出发，选择最简单而足够强大的工具，不引入任何不必要的复杂度。
