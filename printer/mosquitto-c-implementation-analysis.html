第一章

Eclipse Mosquitto 项目概览

Eclipse Mosquitto 是一个开源（EPL/EDL 双许可）的 MQTT 消息代理（Broker），由 Roger Light 创建并于 2009 年发布，现为 Eclipse IoT 项目的核心组件之一。它以 纯 C 语言实现，支持 MQTT 协议 v5.0、v3.1.1 和 v3.1，定位为轻量、高效、可嵌入的消息中间件。

核心特性一览

  • 协议支持：MQTT v5.0 / v3.1.1 / v3.1 完整实现
  • 传输层：TCP、TLS/SSL、WebSocket、Unix Domain Socket
  • QoS 等级：0（最多一次）、1（至少一次）、2（精确一次）
  • 持久化：可选的磁盘消息与会话持久化
  • 安全性：TLS 加密、用户名/密码认证、ACL 访问控制、插件扩展
  • 桥接：Broker-to-Broker 连接，构建复杂网络拓扑
  • $SYS 树：内置系统监控主题层次结构

Mosquitto 源代码托管于 GitHub：eclipse/mosquitto，官方网站为 mosquitto.org。本文所有架构分析均基于其 master 分支（commit d01524，2026年2月）及 deepwiki 权威文档。

⬡ 第一性原理

Mosquitto 存在的第一性原理是：在资源受限的物联网场景中，消息的可靠传递必须以最小的内存与 CPU 代价实现。因此，一切设计决策都指向"足够简单，刚好够用"——纯 C 无运行时开销、单线程事件循环无锁竞争、轻量协议 MQTT 省去 HTTP 头部冗余。

第二章

整体架构设计

Mosquitto 采用 分层模块化架构，以一个 全局单例 mosquitto_db 结构体作为所有子系统的核心枢纽。各模块通过良定义的接口互相协作，但共享同一进程地址空间，从而避免进程间通信开销。

⬡ 第一性原理

分层架构的本质是关注点分离（Separation of Concerns）：每个模块只知道自己该做什么，通过最小接口交互。Mosquitto 用全局 mosquitto_db 代替进程间消息传递，用同一进程内函数调用代替 RPC，这是在"简单与可维护性之间取得最优解"的体现。

第三章

事件循环与 I/O 多路复用

Mosquitto Broker 采用 单线程事件驱动模型，核心是 src/loop.c 中的 mosquitto_main_loop() 函数。这个函数在全局变量 run 为 true 时持续循环，每次迭代顺序执行一组固定操作。

I/O 多路复用抽象层（mux）

Mosquitto 使用一个抽象的 mux 层（src/mux.c）屏蔽底层 I/O 多路复用机制的差异，根据编译选项和操作系统自动选择最优实现：

实现文件底层机制适用平台特点
mux_epoll.cepoll_wait()LinuxO(1) 就绪检测，高并发首选，由 Tatsuzo Osawa 贡献后 CPU 使用从 80-90% 降至 15-20%
mux_poll.cpoll()其他 POSIX 平台标准接口，跨平台兼容
（kqueue）kqueue()BSD/macOS类 epoll 的事件通知

mux 层对外暴露统一函数集：mux__add_in()、mux__add_out()、mux__delete()、mux__handle()，使上层代码完全无需关心底层 I/O 实现。

/* mux_epoll.c 核心：等待事件，分发读写 */
int mux_epoll__handle(void) {
    sigset_t origsig;
    sigprocmask(SIG_SETMASK, &my_sigblock, &origsig);
    event_count = epoll_wait(db.epollfd, ep_events, MAX_EVENTS, 100);
    sigprocmask(SIG_SETMASK, &origsig, NULL);
    db.now_s = mosquitto_time();  /* 统一时间戳 */
    /* 遍历就绪事件，分发读/写处理 */
    for(int i = 0; i < event_count; i++) {
        context = ep_events[i].data.ptr;
        loop_handle_reads_writes(context, ep_events[i].events);
    }
    return MOSQ_ERR_SUCCESS;
}

WebSocket 集成

WebSocket 连接通过 libwebsockets 库集成。mosquitto_websockets_init()（src/websockets.c）在 Listener 初始化时创建 lws_context，其文件描述符通过回调 LWS_CALLBACK_ADD_POLL_FD 注入 mux 层，实现与普通 TCP 连接统一管理。

⬡ 第一性原理

单线程事件循环的本质是：共享状态不需要加锁，因为任意时刻只有一个代码路径在运行。这以串行化换来了无锁简单性。代价是单个慢操作（如阻塞系统调用）会卡住所有连接——因此所有 I/O 均采用非阻塞模式 + 多路复用，确保循环每次迭代都快速完成。

第四章

网络层实现

网络层由 src/net.c 实现，负责监听器（Listener）的创建、客户端套接字的接受与管理，以及 TLS 会话的协商。

连接建立流程

Packet 读写机制

所有套接字操作均为非阻塞。报文读取通过 packet__read(context) 实现，每次读取尽可能多的数据后返回，未读完的报文状态保存在 context->in_packet 中，下次事件触发时继续读取（流式解析）。

/* 非阻塞读取示意（packet_mosq.c）*/
int packet__read(struct mosquitto *context) {
    /* 读固定头部（1字节命令 + 变长剩余长度）*/
    if(context->in_packet.pos == 0) {
        rc = net__read(context, &byte, 1);
        context->in_packet.command = byte;
    }
    /* 读完整报文体后，派发到 handle__packet() */
    if(context->in_packet.pos == context->in_packet.packet_length) {
        rc = handle__packet(context);
        packet__cleanup(&context->in_packet);
    }
    return rc;
}

TLS 支持

TLS 通过 OpenSSL 实现，封装在 net__read()/net__write() 内部，上层代码对 TLS 完全透明。在 epoll 模式下，want_write 标志解决了 TLS 握手期间可能出现的"读事件触发写操作"的特殊情形：

#ifdef WITH_TLS
    if(events & EPOLLOUT
        || context->want_write
        || (context->ssl && context->state == mosq_cs_new)) {
        packet__write(context);  /* TLS 握手可能需要在读事件时写 */
    }
#endif

⬡ 第一性原理

网络层的本质问题是：如何用一个线程同时服务成千个连接？答案是：将阻塞等待的时间交还给 OS（epoll），自己只在数据到来时才工作。非阻塞 I/O + 流式解析 + 事件驱动，三者缺一不可，共同实现"零等待、零浪费"。

第五章

核心数据结构

所有核心数据结构定义于 src/mosquitto_broker_internal.h。Mosquitto 大量使用 uthash（头文件 Hash 库）和 utlist（双向链表宏库）实现高效的内存数据结构，无需额外依赖。

mosquitto_db —— 全局单例

struct mosquitto_db 是整个 Broker 的根结构，保存着所有运行时状态：
  • contexts_by_id — 按 client-id 索引的客户端哈希表
  • contexts_by_sock — 按 socket fd 索引的客户端哈希表
  • msg_store — 全局消息存储哈希表（mosquitto__base_msg）
  • subs — 订阅树根节点（mosquitto__subhier）
  • config — 运行时配置
  • now_s — 当前时间戳（统一时间源，避免频繁 syscall）

⬡ 第一性原理

数据结构是算法的物质基础。Mosquitto 选择 uthash 宏库的原因在于：零内存分配开销——哈希句柄 UT_hash_handle 直接内嵌在结构体中，无需为哈希表节点额外分配内存，实现了"把查找索引与数据合二为一"，节省了每次 malloc 的时间与内存碎片。

CONNECT 处理（最复杂的处理器）

handle__connect()（src/handle_connect.c:334-771）是所有处理器中最复杂的，它需要：

  1. 解析 CONNECT 报文，验证协议版本（v3.1/v3.1.1/v5.0）
  2. 调用 mosquitto_acl_check() 执行认证（插件/密码文件）
  3. 处理 会话接管：若已有同 client-id 连接，强制断开旧连接
  4. 恢复持久会话（转移 msgs_in/msgs_out/subs）
  5. 发送 CONNACK 并注册到 mux 层

PUBLISH 处理流水线

/* handle__publish() 核心流程（src/handle_publish.c）*/
int handle__publish(struct mosquitto *context) {
    /* 1. 权限校验 */
    rc = mosquitto_acl_check(context, MOSQ_ACL_WRITE, topic, ...);
    /* 2. 存入消息存储，生成 store_id，引用计数=1 */
    rc = db__message_store(context, &stored, ...);
    /* 3. 路由到匹配订阅者 */
    rc = sub__messages_queue(context->id, topic, qos, retain, &stored);
    /* 4. QoS 1: 回复 PUBACK，QoS 2: 回复 PUBREC */
    if(qos == 1) send__puback(context, mid, reason_code, NULL);
    if(qos == 2) send__pubrec(context, mid, reason_code, NULL);
}

⬡ 第一性原理

报文处理的本质是协议状态机的映射：每个 MQTT 报文类型对应一个状态转换。分发器 + 处理器模式是开闭原则（OCP）在 C 语言中的体现——新增报文类型只需添加新 case 和新处理函数，无需修改分发逻辑。

第七章

QoS 消息流状态机

QoS 保障是 MQTT 的核心价值。Mosquitto 使用 mosquitto_client_msg 结构体中的 state 字段跟踪每条消息的传递状态，实现精确的状态机管理。

流量控制（Flow Control）

消息入队前，Broker 调用 db__ready_for_flight() 和 db__ready_for_queue() 检查是否超出配额限制：

  • inflight_maximum：同时在途（未确认）的最大 QoS 1/2 消息数（MQTT v5 通过 Receive Maximum 属性协商）
  • max_queued_messages：每个客户端的最大离线消息队列长度
  • 超出限制时，QoS 0 消息直接丢弃，QoS 1/2 消息进入等待队列

📄 src/handle_publish.c:36-390 📄 src/database.c:38-1002 📄 lib/messages_mosq.c

⬡ 第一性原理

QoS 状态机的本质是：用磁盘（或内存）状态换取网络不可靠性的代偿。当网络中断时，Broker 并不需要重新知道消息内容，只需知道消息所处的状态，就能在恢复连接后从正确的状态点继续。状态机让"刚好一次传递"从概念变为可实现的工程。

第八章

订阅树与主题匹配

Mosquitto 使用 层次化哈希树（Hierarchical Hash Tree）实现 MQTT 主题匹配。每个 mosquitto__subhier 节点代表主题中以 / 分隔的一个层级，子节点通过 uthash 哈希表组织，实现 O(1) 的精确字符串查找。

主题树结构示意

订阅匹配算法

sub__search()（src/subs.c）递归遍历订阅树，对于每个消息，沿主题层级路径匹配：

/* 伪代码：订阅树遍历（src/subs.c）*/
void sub__search(subhier, topics[], level) {
    /* 精确匹配：查当前层字符串 */
    HASH_FIND(hh, subhier->children, topics[level], len, branch);
    if(branch) sub__search(branch, topics, level+1);

    /* + 通配：匹配单层任意字符串 */
    HASH_FIND(hh, subhier->children, "+", 1, branch);
    if(branch) sub__search(branch, topics, level+1);

    /* # 通配：匹配剩余所有层 */
    HASH_FIND(hh, subhier->children, "#", 1, branch);
    if(branch) subs__process(branch, msg);  /* 直接投递 */
}

⬡ 第一性原理

主题匹配的本质是前缀树（Trie）的变体，用 / 作为分隔符构造多叉树。哈希表替代传统 Trie 的数组子节点，将每层查找从 O(字符数) 降低到 O(1)。通配符节点只是特殊名称（"+"、"#"）的普通树节点——用最简单的统一结构表达复杂语义，这是"奥卡姆剃刀"原则在数据结构设计中的体现。

第九章

消息存储与引用计数

Mosquitto 使用 引用计数（Reference Counting）机制管理消息内存，实现消息在多个订阅者之间的零拷贝共享。核心结构是 mosquitto__base_msg。

引用计数实现

/* src/database.c — 引用计数接口 */
void db__msg_store_ref_inc(struct mosquitto__base_msg *base_msg) {
    base_msg->ref_count++;
}

void db__msg_store_ref_dec(struct mosquitto__base_msg **base_msg) {
    (*base_msg)->ref_count--;
    if((*base_msg)->ref_count == 0) {
        db__msg_store_remove(*base_msg, true);  /* 触发持久化删除事件 */
        *base_msg = NULL;  /* 防悬空指针 */
    }
}

/* 消息入队时（每个订阅者+1）*/
db__msg_store_ref_inc(stored);
/* 消息投递确认后（-1）*/
db__msg_store_ref_dec(&cmsg->base_msg);

消息存储流程

  1. PUBLISH 到达 → db__message_store() 分配 mosquitto__base_msg，ref_count=1，注册到 db.msg_store 哈希表
  2. 路由到订阅者 → 每个匹配的订阅者调用 db__msg_store_ref_inc()，ref_count 累加
  3. QoS 确认完成 → 调用 db__msg_store_ref_dec()，ref_count 递减
  4. ref_count 归零 → 自动从 db.msg_store 移除，mosquitto__free(payload) 释放内存

📄 src/database.c:38-1002

⬡ 第一性原理

引用计数解决的根本问题是：谁是内存的最后使用者？在多订阅者场景中，消息不知道哪个客户端最后确认。引用计数让每个消费者"声明自己还需要"，让内存在"所有人都不再需要时"自动回收——这是 所有权转移思想的 C 语言实现，与 Rust 的借用检查器在哲学上一脉相承。

第十章

会话管理与持久化

MQTT 协议允许客户端在断开连接后保留会话状态（持久会话，clean_start=false），Broker 需要在客户端重新连接时恢复其订阅列表和未投递消息。

客户端上下文生命周期

会话接管（Session Takeover）

当同一 client-id 的新连接到来时，Broker 执行会话接管：

/* src/handle_connect.c:96 */
HASH_FIND(hh_id, db.contexts_by_id, context->id,
          strlen(context->id), found_context);
if(found_context) {
    /* 将旧上下文的消息队列和订阅迁移到新连接 */
    new_context->msgs_in  = found_context->msgs_in;
    new_context->msgs_out = found_context->msgs_out;
    new_context->subs     = found_context->subs;
    new_context->sub_count = found_context->sub_count;
    /* 强制断开旧连接 */
    do_disconnect(found_context, MOSQ_ERR_CONN_LOST);
}

持久化（Persistence）

Mosquitto 提供两种持久化方案：

  • 内置文件持久化：通过 persist__backup() 定期将消息存储和会话状态写入 mosquitto.db（二进制格式）。配置项 persistence true、autosave_interval
  • 插件持久化（SQLite）：plugins/persist-sqlite/ 通过插件 API 实现 SQLite 后端持久化，可在 SIGHUP 热重载

⬡ 第一性原理

会话持久化的本质是：将客户端的"身份"与"连接"解耦。连接是临时的（网络可能断），而身份（client-id）是持久的。Broker 维护的不是连接，而是客户端状态的快照，连接只是访问该快照的通道——这正是 MQTT 在不稳定网络上依然可靠的根本原因。

第十一章

插件系统架构

Mosquitto 2.0 引入了全新的插件接口，显著降低了插件开发门槛。插件只需实现 三个固定函数，然后通过回调注册机制订阅所需事件。

动态安全插件（Dynamic Security）

plugins/dynamic-security/ 是最强大的官方插件，通过 $CONTROL/dynamic-security/v1 主题进行动态配置，无需重启 Broker：

  • Client：动态创建/删除用户，设置密码
  • Group：用户组管理，客户端可属于多个组
  • Role：角色定义 ACL 规则集，支持优先级
  • ACL 类型：publishClientSend、publishClientReceive、subscribe、unsubscribe

⬡ 第一性原理

插件系统的本质是控制反转（IoC）：Broker 不调用具体安全实现，而是在事件发生时"回调"插件。这让核心代码对安全实现零依赖，插件可以热插拔。Mosquitto 2.0 将插件接口从"必须实现所有功能"简化为"只注册我需要的事件"，是接口隔离原则（ISP）的精确体现。

第十二章

桥接（Bridge）机制

Bridge 功能允许 Mosquitto Broker 作为 MQTT 客户端连接到另一个 Broker，实现 Broker-to-Broker 的消息转发，构建多级或地理分布式 MQTT 拓扑。

Bridge 配置示例

# mosquitto.conf — Bridge 配置
connection bridge_to_cloud
address cloud.broker.example.com:8883
remote_username bridge_user
remote_password secret_pass
remote_clientid local_to_cloud

# 出向：本地 sensors/# → 远端 local/sensors/#
topic sensors/# out 1 "" "local/"
# 入向：远端 commands/# → 本地 commands/#
topic commands/# in 1

bridge_attempt_unsubscribe true
cleansession false   # 持久 Bridge 会话
try_private true     # 使用私有 Bridge 协议标志

Bridge 实现要点

  • 桥接连接复用 struct mosquitto：Bridge 连接作为特殊的客户端上下文存储，context->bridge 指向 mosquitto_bridge 配置
  • 自动重连：检测到 Bridge 断连后，通过主循环中的 bridge__connect() 定期尝试重连
  • 去环路：try_private 标志通知对端这是 Bridge 连接，对端不会再将消息回传，防止消息无限循环
  • 主题映射：支持本地前缀 / 远端前缀双向重写

⬡ 第一性原理

Bridge 的第一性原理是：网络的网络。单个 Broker 是物理约束的（带宽、连接数、地域）。Bridge 让 Broker 既是服务端又是客户端，实现了去中心化的消息网络——每个 Broker 只需理解自己局部的拓扑，通过 Bridge 连接构成更大的整体，这正是互联网路由协议的哲学。

第十三章

设计模式汇总

Mosquitto 的 C 代码中蕴含着大量经典设计模式，以函数指针、宏、结构体组合实现，而非面向对象语言的类继承体系。

条件编译——C 的"多态"

Mosquitto 的报文处理代码在 Broker 端（src/）和客户端库（lib/）之间共享，通过 #ifdef WITH_BROKER 宏区分不同行为，这是 C 语言在没有类继承机制下实现"同一代码，不同行为"的惯用手法：

/* lib/handle_publish.c — 客户端与 Broker 共享代码 */
int handle__publish(struct mosquitto *mosq) {
#ifdef WITH_BROKER
    /* Broker 端：路由消息到订阅者 */
    rc = sub__messages_queue(mosq->id, topic, qos, retain, &stored);
#else
    /* 客户端库：触发 on_message 回调 */
    pthread_mutex_lock(&mosq->callback_mutex);
    if(mosq->on_message) mosq->on_message(mosq, mosq->userdata, &message);
    pthread_mutex_unlock(&mosq->callback_mutex);
#endif
}
⬡ 第一性原理

设计模式不是目的，而是解决具体约束下的具体问题的经验提炼。Mosquitto 在 C 语言中用宏、函数指针、结构体组合实现了与面向对象语言相同的关注点分离——这说明好的设计思想与语言无关，真正重要的是识别问题的本质：耦合在哪？变化在哪？然后用当前语言最自然的方式解耦。

第十四章

总结与参考资料

核心设计哲学总结

Mosquitto 的 C 实现集中体现了以下核心工程哲学：

  • 🎯 单一职责：每个 .c 文件专注一件事（loop.c 做循环、subs.c 做订阅、database.c 做存储）
  • ⚡ 零拷贝优先：消息引用计数共享 payload，不为每个订阅者复制数据
  • 🔒 无锁单线程：用单线程事件循环规避竞态条件，用 epoll 实现高并发
  • 🧩 可组合性：插件系统、Bridge、持久化均以模块方式可选启用
  • 📏 MQTT 协议即规格：代码结构与 MQTT 规范一一对应，降低理解成本

⬡ 全局第一性原理

Mosquitto 整体设计的第一性原理是：消息传递的本质是"状态在时间和空间上的转移"。时间维度：QoS 状态机和持久会话确保消息"穿越"网络中断。空间维度：订阅树和 Bridge 确保消息"穿越"拓扑边界。引用计数确保同一份数据"穿越"多个消费者而无需复制。所有复杂性，都是为了在不可靠的物理世界中，可靠地完成这一次"转移"。
