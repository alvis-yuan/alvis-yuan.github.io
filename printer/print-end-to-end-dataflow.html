打印全链路数据流分析：从外部数据到驱动状态

本文沿着真实代码路径，把外部发送来的打印数据如何进入系统、如何按通道排队、如何进入 ESC/POS/TSPL 解析、如何栅格化并写入底层打印驱动，以及应用如何查询底层状态串起来。核心代码位于 server/print/src 与 libcommon/print_api.c。

一句话概括：外部入口只负责收数据并调用 print_channel_send_data()；共享内存保存任务和 buffer；PrintServer 主循环定时把通道任务刷到全局队列；普通 ESC/POS 数据由 printer_send_data() 逐字节解析，文本经 LineBuffer 栅格化，图片类命令可直接输出点行；最终 PrnDotLine() 和 PrnIoctl() 通过 write/ioctl 触达内核驱动。
第一性原理：打印系统要同时解决两个问题：数据面按顺序消费字节，控制面及时处理状态和设备动作。这个项目用共享内存承载数据面，用消息队列和 ioctl 承载控制面。

全链路地图

阶段关键函数/结构职责
外部接入usb.c、tcpserver.c、BLE/SPP 模块、业务应用收外部字节或订单数据，选择通道 ID。
API 写入print_channel_send_data()、append_data_to_channel()创建任务、分配 buffer、写入共享内存。
队列调度flush_channels()、append_task_to_queue()把各通道当前任务收敛到全局打印队列。
发送执行send_next_buffer()按打印状态、缺纸、TSPL/ESC/POS 模式发送 buffer。
解析渲染printer_send_data()、Cmd_Proc()、LineBuffer逐字节解释 ESC/POS，生成点阵行。
驱动边界PrnDotLine()、PrnIoctl()、PrnGetStatus()通过 write()/ioctl() 和内核打印驱动交互。
第一性原理：端到端数据流的稳定性来自边界清晰：入口不碰硬件，客户端不调度，调度器不理解业务订单，驱动层不理解 ESC/POS。

外部入口

外部数据来源分两类：一种是物理/网络输入，例如 USB、TCP、串口、BLE、SPP；另一种是业务侧输入，例如 HTTPS/MQTT/CGI/内部测试。它们最终都收敛到 print_channel_send_data() 或带 flag/orderNo 的变体。

来源调用点通道特点
USBserver/print/src/usb.cPRINT_CHN_ID_USB / PRINT_CHN_ID_USB_IAP2外部主机直接下发 ESC/POS 字节流。
TCPserver/print/src/tcpserver.cPRINT_CHN_ID_TCP_BEGIN + index网络 socket 收到多少写多少，按连接映射通道。
BLE/SPPserver/bluetooth/**PRINT_CHN_ID_BLE / PRINT_CHN_ID_SPP...蓝牙数据分片到达，通常 copies=0 追加到当前任务。
云订单/业务application/app0、cgi/servicePRINT_CHN_ID_HTTPS、PRINT_CHN_ID_MQTT、PRINT_CHN_ID_CGI一般有明确份数，创建完整任务并返回 task id。
内部任务PrintInfopage.c、工厂测试、模式提示小票PRINT_CHN_ID_INT系统自己生成打印内容，也走同一队列。
// 典型入口形式
print_channel_send_data(PRINT_CHN_ID_USB, data, len, 0);
print_channel_send_data(PRINT_CHN_ID_TCP_BEGIN + index, rcvbuf, ret, 0);
print_channel_send_data_cloud(PRINT_CHN_ID_HTTPS, data, len, copies, orderNo);
第一性原理：入口层只做“把外部字节归属到哪个通道”。只要所有来源都收敛到统一 API，后面的排队、缺纸、重打、状态逻辑就能复用。

写入共享内存

libcommon/print_api.c 是应用侧写共享内存的核心。它先判断是否为实时命令；普通数据则创建或复用通道当前任务，把数据拆成 PRINT_BUFFER_SIZE 大小的小块写入 buffer_pool，并挂到 print_task_t 的 buffer 链表。

if (copies == 0)
    return append_data_to_channel(chn_id, data, len, flag);

t = task_new();
TASK_AT(t)->chn_id = chn_id;
TASK_AT(t)->id = next_task_id();
TASK_AT(t)->copies = copies;

while (len > 0) {
    b = buffer_new();
    append_buffer_to_task(t, b);
    memcpy(BUFFER_AT(b)->data, data + bytes_copied, l);
    BUFFER_AT(b)->len = l;
}

append_task_to_channel_task_queue(chn_id, t);

copies == 0 的入口表示“追加到当前通道任务”，适合 USB/TCP/蓝牙这种流式到达的数据；copies > 0 的入口直接创建完整任务并放到该通道的任务队列，适合订单打印。

第一性原理：外部数据到达粒度不可控，驱动消费粒度也不可控。中间用固定小块链表解耦两端，可以避免大块内存搬移，并支持跨多次接收拼成一个任务。

通道队列到全局队列

共享内存里有两级队列。第一级是每个通道自己的当前任务和通道任务队列，第二级是 PrintServer 唯一消费的全局打印队列。flush_channels() 周期性扫描所有通道，把已经完成的任务追加到全局队列。

触发条件代码行为适用场景
通道任务队列已有完整任务取 channel->task_queue.head，调用 append_task_to_queue(t)业务侧一次性下发完整订单。
当前流式任务超时无新数据no_data_received >= task_end_timeout 后认为任务完整USB/TCP/BLE/SPP 字节流没有显式结束符。
指定接收长度已满足bytes_received >= bytes_to_receive某些命令或任务知道期望长度。
边接收边打印策略队列空、打印机空闲、buffer 大于 1 时可提前入全局队列大任务减少首张输出延迟。
if (complete) {
    TASK_FLAG_CLEAR(TASK_AT(channel->task), TASK_FLAG_RECV_NOT_COMPLETE);
    append_task_to_queue(channel->task);
    channel->task = INDEX_NULL;
} else if (task_print_strategy == PRN_START_PRINT_DURING_RECEIVING && ... ) {
    TASK_FLAG_SET(TASK_AT(channel->task), TASK_FLAG_RECV_NOT_COMPLETE);
    append_task_to_queue(channel->task);
}
第一性原理：多入口系统必须先在入口侧分流，再在执行侧串行化。通道队列保护来源之间的边界，全局队列保证硬件只按一个顺序消费。

主循环调度

print_queue_run() 是 PrintServer 的核心调度循环。启动时 main() 先初始化 printer_init()、print_queue_init()、USB/TCP/serial，再启动 TSPL 线程并进入 print_queue_run()。循环里同时做四件事：查询驱动状态、刷新通道、检查当前任务完成/重打、发送下一批 buffer。

while (g_process_running) {
    PrnGetStatus(&status);
    PrnStatusToShmStatus(&status);

    flush_channels(msec, shm->status.printing);

    if (shm->status.printing == 0)
        check_task_print_status();

    if (!shm->status.step_motor_overheated)
        send_next_buffer(curr_paper_status);
}

send_next_buffer() 会根据缺纸、开盖、过热、TSPL 解析标志、任务是否仍在接收等条件决定是否发送。普通 ESC/POS 每次最多发送 128 个 buffer；标签模式可放宽到 2048 个 buffer。

第一性原理：调度循环不是简单 while-print，它同时是“硬件状态采样器、队列推进器、异常恢复器”。打印能否稳定，取决于这三类职责是否按同一个节拍协调。

解析与渲染

普通 ESC/POS 分支中，send_next_buffer() 调用 printer_send_data()。后者加 Cmd_Mutex，设置当前 buffer id 和 offset，然后逐字节调用 Cmd_Proc()。Cmd_Start() 根据控制字节进入 DLE/ESC/FS/GS 子解析器，否则走字符渲染。

for (Cmd_BufferOffset = offset; Cmd_BufferOffset < buf->len; Cmd_BufferOffset++) {
    Cmd_Proc(buf->data[Cmd_BufferOffset]);
    if (RUNTIME_FLAG_IS_SET(RTF_KERN_BUF_FULL)) {
        rc = PRN_BUF_FULL;
        break;
    }
}

文本路径大致是：Char_Proc() 判断 UTF-8/CJK/ASCII/code page，调用 ASC_Proc()、CodePage_Proc() 或 Unicode_Proc()，再通过 AppendToLineBuffer() 写入行缓冲。遇到 LF、缓冲满、显式打印命令时，lineBuffer->printLine() 把这一行变成点阵输出。

不是所有内容都必须先进入 LineBuffer。位图、栅格图、下载图像等 GS 指令在 GS_Instruct.c 中可以直接构造点阵并调用 PrnDotLine()。页模式下，LineBuffer 会先 paint 到 PageBuffer，再由页模式逻辑输出。

第一性原理：解析器负责把字节变成“打印意图”，渲染器负责把意图变成“点阵行”。文本适合先排版成行，图片类命令本来就是点阵，直接写点行更自然。

写入底层驱动

底层写入集中在 server/print/src/printer.c。PrnDotLine() 接收单色或灰度点阵数据，把数据装入 PrnDivBuf，再调用 write_to_kernel()。write_to_kernel() 最终通过 write(PrnFd, PrnDivBuf, SizeOfPrnDivBuf) 写内核驱动。

static inline int write_to_kernel(void)
{
    int ret = write(PrnFd, PrnDivBuf, SizeOfPrnDivBuf);
    if (ret == PRN_BUF_FULL) {
        while (1) {
            SysDelay(100);
            ret = write(PrnFd, PrnDivBuf, 0);
            if (ret != PRN_BUF_FULL)
                break;
        }
    }
    return ret;
}

PrnDotLine(NULL, 0, PRN_DATA_MONO) 是一个特殊 flush/探测调用，会执行 write(PrnFd, PrnDivBuf, 0)。send_next_buffer() 在发普通数据前会先调用一次 printer_send_data(NULL, 0)，用于确认内核缓冲是否可继续接收。

底层接口系统调用用途
PrnDotLine()write()写点阵打印数据；处理内核缓冲满。
PrnIoctl(PRN_CMD_GET_STATUS)ioctl()查询驱动状态。
PrnIoctl(PRN_CMD_CUT)ioctl()切刀等机械动作。
PrnIoctl(PRN_CMD_SEND_EMBEDDED_COMMAND)write()发送嵌入式底层命令，并可能 flush ring buffer。
第一性原理：驱动层只认点阵、控制码和 ioctl 命令。ESC/POS 的语义必须在进入驱动前被消化成这些更底层的动作。

底层状态查询

状态链路方向和打印数据相反：PrintServer 周期性从驱动取状态，转换成共享内存里的 print_status_t；应用侧查询只是读取共享内存快照。这样应用不会直接碰驱动，也不会让多个进程同时 ioctl 打印设备。

int PrnGetStatus(PRN_STATUS *status)
{
    uint32_t dataOutLength = sizeof(PRN_STATUS);
    return PrnIoctl(PRN_CMD_GET_STATUS, NULL, 0, status, &dataOutLength);
}

int print_get_status(print_status_t *status)
{
    memcpy(status, &shm->status, sizeof(print_status_t));
    return 0;
}

PrnStatusToShmStatus() 做字段映射，例如 status->state & 0x01 映射为 printing，纸张位和异常位映射为 no_paper、paper_jam、paper_not_taken，开盖映射为 platen_opened，温度和电压字段映射为过热/电压异常。

如果应用调用 print_status_change_subscribe()，库里会启动 status_thread() 等待 sem_status。PrintServer 发现状态变化后按订阅数 sem_post(sem_status)，应用线程再复制 shm->status 并回调。

第一性原理：状态查询要避免“每个应用都打到硬件”。由唯一硬件 owner 采样并发布快照，可以降低驱动竞争，也让实时命令、任务调度和应用查询看到同一份状态。

控制命令与实时命令

控制面不走大块打印数据队列，而是通过 POSIX 消息队列进入 print_mqueue_thread()。例如设置走纸、切刀、打印模式、浓度、获取纸张位置、坏点检测、清除开盖状态等，最终由 PrintServer 调用 PrnIoctl() 或更新共享内存。

case PRINT_CMD_DO_CUT:
    PrnIoctl(PRN_CMD_CUT, NULL, 0, NULL, NULL);
    break;

case PRINT_CMD_HANDLE_REALTIME_COMMAND:
    Realtime_Proc(msg.handle_realtime_command.chn_id,
                  msg.handle_realtime_command.data,
                  msg.handle_realtime_command.len);
    break;

实时命令有一条特殊路径：应用侧 handle_realtime_command() 在写共享内存前识别 DLE EOT、GS I、ESC p、GS ( T、GS ( E 等子集，并把它们封装成 PRINT_CMD_HANDLE_REALTIME_COMMAND。PrintServer 收到后调用 Realtime_Proc()，再由 ReplyToHost() 回到 USB/TCP/BLE/SPP 等来源。

第一性原理：打印数据是顺序流，控制命令是事件。把事件从数据流中抽出来可以提升响应速度，但必须保持命令规格一致，避免普通解析器和实时解析器行为漂移。

异常与重打

队列层和驱动层都参与异常处理。缺纸/开盖时，send_next_buffer() 会停止普通任务发送；主循环检测到从有纸到缺纸时调用 out_of_paper_detected()。任务完成不是以“buffer 全发完”立刻释放，而是结合驱动 buffer id 判断最后一个 buffer 是否真正打印完成。

异常点处理位置作用
缺纸/开盖print_queue_run()、send_next_buffer()停止继续发送，必要时标记重打。
内核缓冲满PrnDotLine()、printer_send_data()等待驱动释放，或设置 RTF_KERN_BUF_FULL 暂停当前 buffer。
任务删除check_task_print_status()标记 TASK_FLAG_DONT_PRINT 后清理队列、恢复运行时状态。
多份打印check_task_print_status()当前份完成后重置 buf_to_send 和 buf_offset 从头再发。
不可重打任务send_next_buffer()已发送 buffer 可提前释放，节省共享内存。
这里的关键不是“发完共享内存 buffer 就完成”，而是“驱动确认对应 buffer id 已打印完成”。这也是代码里 send_buffer_id()、compare_buffer_id()、PRN_CMD_GET_BUFFER_ID 存在的原因。
第一性原理：打印是物理过程，软件队列完成不等于纸面完成。可靠任务生命周期必须以硬件进度为准，而不是以内存消费为准。

结论

数据流主线

外部入口调用 print_channel_send_data()，数据进入共享内存 task/buffer，PrintServer 把通道任务刷入全局队列，再由 send_next_buffer() 推进打印。

渲染主线

普通 ESC/POS 经 printer_send_data() 逐字节进入 Cmd_Proc()，文本写入 LineBuffer 后输出点阵，图片类命令可以直接 PrnDotLine()。

驱动主线

PrnDotLine() 通过 write() 写打印点行，PrnIoctl() 通过 ioctl() 做切刀、状态、纸张位置、模式等控制。

状态主线

PrintServer 周期性 PrnGetStatus()，映射到 shm->status；应用 print_get_status() 读快照，订阅者通过 sem_status 收变化通知。

print_channel_send_data append_data_to_channel flush_channels send_next_buffer printer_send_data Cmd_Proc LineBuffer PrnDotLine PrnGetStatus

第一性原理：这条链路的核心设计是“单硬件 owner + 共享内存队列 + 流式解析 + 状态快照”。它牺牲了一些结构简洁性，但换来了多来源输入、异常重打和底层驱动隔离。
