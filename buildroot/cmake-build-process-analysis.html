第 0 章

背景与历史

📎 cmake.org/history 📎 cmake.org/overview

CMake 是什么

CMake 是一个开源、跨平台的软件构建工具家族，被设计用来构建、测试和打包软件。它通过简单的、独立的配置文件来控制软件的编译过程。与许多跨平台系统不同，CMake 被设计为与原生构建环境协同工作，而不是替代它。

官方说明（cmake.org/overview）：

  • 可按需扩展以支持新特性
  • 生成原生构建环境——编译源代码、创建库、生成包装器和构建可执行文件
  • 支持从单一源码树进行多种不同构建
  • 支持静态库和动态库构建
  • 生成适合图形编辑器使用的缓存文件（CMakeCache.txt）
  • 支持复杂的目录层次结构和依赖多个库的应用程序
  • 处理"先构建可执行文件、再用其生成代码、最后编译链接进最终应用"的场景

每月下载量超过 200 万次。2022 年，Bryce Adelstein Lelbach 在 C++Now 大会上指出："我们实际上已经有了一个标准构建系统——它就是 CMake"。

发展时间线

CMake 的核心设计原则（源自原作者）

  • 支持 Visual Studio IDE 等原生构建工具
  • 所有平台只需一份输入文件
  • 仅依赖 C++ 编译器，易于在新系统上移植和启动
  • 支持代码生成器，将 C++ 包装到其他语言

💡 本章总结 & 第一性原理

CMake 诞生于 2000 年，核心出发点：解决大型跨平台 C++ 项目的构建难题。

第一性原理：编译软件的本质需求是"告诉编译器哪些文件用什么选项编译、如何链接"。CMake 提供一套平台无关的高级描述语言，将构建意图翻译为各平台的原生指令，从根本上解决了跨平台的 N×M 矩阵问题（N 个平台 × M 个 IDE/构建工具）。

第 1 章

目录结构

📎 cmake-language(7) Organization

典型项目布局

现代 CMake 推荐将源码树与构建树严格分离。每个子目录中若需参与构建，必须包含自己的 CMakeLists.txt。

构建树 vs 源码树

官方文档（cmake-language(7)）指出："当 CMake 处理项目源码树时，入口点是顶级源码目录中名为 CMakeLists.txt 的文件。对于每个处理了 CMakeLists.txt 的源码目录，CMake 会在构建树中生成对应的目录，作为默认的工作目录和输出目录。"

特性源码树（Source Tree）构建树（Build Tree）
根目录变量CMAKE_SOURCE_DIR
PROJECT_SOURCE_DIR		CMAKE_BINARY_DIR
PROJECT_BINARY_DIR
当前目录变量CMAKE_CURRENT_SOURCE_DIRCMAKE_CURRENT_BINARY_DIR
版本控制✅ 必须纳入 VCS❌ 通常 .gitignore 忽略
可删除重建❌ 核心资产✅ 可安全删除重建
主要内容CMakeLists.txt、源文件、头文件、资源Makefile、.o 文件、CMakeCache.txt、最终产物
⚠️ 避免 In-Source Build
永远不要在源码根目录直接运行 cmake .（In-Source Build）。正确做法是创建独立 build/ 目录后运行 cmake ..，这样删除 build 目录即可完全清理，源码树保持干净。

语言文件类型：Directories / Scripts / Modules

📎 官方原文

官方文档将 CMake 语言源文件组织为三类：

类型文件名规则用途
目录文件 (Directories)CMakeLists.txt项目构建规范的主入口，通过 add_subdirectory() 组织层次结构
脚本文件 (Scripts)<name>.cmake用 cmake -P 以脚本模式运行，不生成构建系统，不定义构建目标
模块文件 (Modules)<module>.cmake被 include() 命令加载，提供可复用功能；位置由 CMAKE_MODULE_PATH 控制

💡 本章总结

CMake 项目目录结构遵循"源码与构建分离"的核心原则。每个参与构建的子目录必须有自己的 CMakeLists.txt，顶层文件通过 add_subdirectory() 串联整个构建描述树。构建树可以完全从源码树重新生成，这保证了构建的可重复性与洁净性。

第一性原理：文件系统层次结构天然映射软件的模块层次。CMake 让每个模块"自描述"其构建规则（每个目录一个 CMakeLists.txt），再由顶层统一组装，实现了构建描述的高内聚、低耦合。

第 2 章

CMake 语言基础

📎 官方原文：cmake-language(7)

命令与语法

官方文档（cmake-language(7)）定义："CMake 语言源文件由零个或多个命令调用组成，命令调用之间用换行和可选的空格分隔。"

命令调用的语法为：名称后跟括号括起来的参数，参数间用空格分隔：

# 命令名不区分大小写，参数区分大小写
add_executable(hello world.c)
if(FALSE AND (FALSE OR TRUE))  # 括号可嵌套
endif()

参数有三种类型：

参数类型语法特点
方括号参数 (Bracket)[[...]] 或 [==[...]==]不转义，不展开变量，整体作为一个参数
引号参数 (Quoted)"..."支持转义序列和变量引用 ${VAR}，整体为一个参数
非引号参数 (Unquoted)直接书写支持转义和变量引用，遇到 ; 会分割为多个参数
 
# 方括号参数示例（原文来自官方文档）
message([=[
  This is a bracket argument.
  No \-escape or ${variable} evaluation here.
]=])

# 引号参数示例
message("Line1\nLine2 with ${MY_VAR}")

# 变量引用
set(srcs a.c b.c c.c)  # 内部存储为 "a.c;b.c;c.c"

变量与作用域

📎 官方原文

官方文档："变量是 CMake 语言的基本存储单元。其值始终为字符串类型。变量名区分大小写，具有动态作用域。"

变量作用域（由内向外查找）：

变量引用方式：

# 普通变量引用
message(${MY_VAR})

# 环境变量引用（进程级，不缓存）
message($ENV{PATH})

# Cache 变量直接引用（跳过普通变量查找）
message($CACHE{CMAKE_BUILD_TYPE})

# 嵌套引用（由内向外求值）
set(prefix "MY")
message(${${prefix}_VAR})   # 等价于 ${MY_VAR}
📌 注意
CMake 保留以 CMAKE_、_CMAKE_ 或 _<命令名> 开头的标识符。项目变量命名应避免使用这些前缀。

列表与环境变量

官方文档："所有 CMake 值均以字符串存储，但字符串在某些上下文（如非引号参数求值时）可被视为列表——以 ; 作为分隔符进行分割。"

set(srcs a.c b.c c.c)   # 存储为 "a.c;b.c;c.c"
foreach(f IN LISTS srcs)
  message(${f})          # 依次输出 a.c、b.c、c.c
endforeach()

# 注意：嵌套列表会被展开，; 不可嵌套
set(x a "b;c")           # x = "a;b;c"，不是 "a;b\;c"

💡 本章总结

CMake 语言是一种面向构建描述的专用语言（DSL），有三个关键特点：①命令不区分大小写，②变量值始终为字符串，③作用域按块→函数→目录→Cache 四层动态查找。理解变量的动态作用域和 Cache 机制，是编写正确 CMakeLists.txt 的基础。

第 3 章

配置详解

📎 cmake-buildsystem(7) 📎 CMake Tutorial

核心 CMakeLists.txt 结构

每个 CMakeLists.txt 描述了该目录层级的构建规则。以下是一个典型顶层文件和子目录文件的示例：

# 顶层 CMakeLists.txt

# 1. 声明最低 CMake 版本（必须第一行）
cmake_minimum_required(VERSION 3.20)

# 2. 定义项目名称、版本和使用的语言
project(MyProject
  VERSION 1.0.0
  DESCRIPTION "My awesome project"
  LANGUAGES CXX
)

# 3. 设置全局选项（C++ 标准等）
set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

# 4. 添加子目录（每个子目录有自己的 CMakeLists.txt）
add_subdirectory(libs/mylib)
add_subdirectory(src)
add_subdirectory(tests)

# src/CMakeLists.txt

# 定义可执行文件目标
add_executable(MyApp main.cpp)

# 链接库（同时传播使用需求）
target_link_libraries(MyApp PRIVATE mylib)

# 设置包含目录（仅本目标使用 PRIVATE）
target_include_directories(MyApp
  PRIVATE ${CMAKE_CURRENT_SOURCE_DIR}/../include
)

Target 目标系统

📎 官方原文：Binary Targets

官方文档："基于 CMake 的构建系统被组织为一组高级逻辑目标（Target）。每个目标对应一个可执行文件或库，或是包含自定义命令的自定义目标。"

命令目标类型说明
add_executable(name ...)可执行文件通过链接对象文件生成，其中一个包含程序入口（main）
add_library(name STATIC ...)静态库 (.a / .lib)对象文件的档案集，链接时符号被复制到消费者中
add_library(name SHARED ...)动态库 (.so / .dll)运行时由动态加载器解析，消费者记录引用
add_library(name MODULE ...)模块库插件，运行时按需 dlopen 加载，不作为链接依赖
add_library(name OBJECT ...)对象库编译后的对象文件集合，不归档不链接，可被其他目标使用
add_library(name INTERFACE)接口库无编译产物，仅传播使用需求，常用于头文件库
 
# 官方示例（来自 cmake-buildsystem(7)）
add_library(archive archive.cpp zip.cpp lzma.cpp)
add_executable(zipapp zipapp.cpp)
target_link_libraries(zipapp archive)

# archive 为静态库，zipapp 链接 archive 时获得所有符号

PUBLIC / PRIVATE / INTERFACE 作用域

📎 官方原文：Target Commands

这是 Modern CMake 最核心的概念之一。官方文档定义了三个作用域关键字：

# 官方文档示例（cmake-buildsystem(7)）
add_library(archive SHARED archive.cpp zip.cpp)

target_compile_definitions(archive
  PRIVATE   BUILDING_WITH_LZMA    # 仅编译 archive 时定义
  INTERFACE USING_ARCHIVE_LIB     # 仅传播给消费者
)

add_executable(consumer consumer.cpp)
# consumer.cpp 编译时自动获得 -DUSING_ARCHIVE_LIB
target_link_libraries(consumer archive)

Cache 变量与预设（CMakePresets.json）

📎 官方：Cache Variables

Cache 变量存储在 CMakeCache.txt 中，跨多次 cmake 运行持久化，用于用户可配置选项：

# 定义 Cache 变量（用户可通过 -D 或 cmake-gui 修改）
option(ENABLE_TESTS "Build tests" ON)
set(MY_OPTION "default" CACHE STRING "My option description")

# 命令行传入 Cache 变量
# cmake -DENABLE_TESTS=OFF -DCMAKE_BUILD_TYPE=Release ..

CMakePresets.json（CMake 3.19+ 引入）让构建配置版本化、可共享：

{
  "version": 3,
  "configurePresets": [
    {
      "name": "release",
      "displayName": "Release Build",
      "generator": "Ninja",
      "binaryDir": "${sourceDir}/build/release",
      "cacheVariables": {
        "CMAKE_BUILD_TYPE": "Release",
        "ENABLE_TESTS": "OFF"
      }
    }
  ]
}
# 使用预设：cmake --preset release

💡 本章总结

CMake 配置的核心是以 Target 为中心描述构建意图。每个 Target 是一个构建单元（可执行文件或库），通过 PUBLIC/PRIVATE/INTERFACE 精确控制属性的传播范围。Cache 变量和 CMakePresets.json 则提供了灵活的用户可配置性。

第一性原理：软件由模块组成，模块间有依赖关系。CMake 的 Target + 属性传播机制，将"如何构建我"和"如何使用我"这两个关注点分离并精确表达，是现代 CMake 区别于旧式 CMake 的本质所在。

第 4 章

编译流程

📎 cmake(1) 手册 📎 cmake-generators(7)

完整三阶段流程概览

步骤 1：配置阶段（Configure）

配置阶段读取 CMakeLists.txt 并执行其中的命令，包括系统自省（检测编译器、操作系统等），最终生成 CMakeCache.txt。

# 指定源码目录和构建目录
cmake -S . -B build

# 指定生成器
cmake -S . -B build -G "Ninja"

# 传入 Cache 变量
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release -DENABLE_TESTS=ON

# 使用预设（CMake 3.19+）
cmake --preset release

# 配置阶段输出示例
-- The CXX compiler identification is GNU 13.1.0
-- Detecting CXX compiler ABI info - done
-- Detecting CXX compile features - done
-- Configuring done (2.1s)
-- Generating done (0.1s)
-- Build files have been written to: /project/build

步骤 2：生成阶段（Generate）

生成阶段在配置完成后自动执行。CMake 根据选定的 Generator，将配置阶段收集的信息转化为原生构建系统的输入文件：

  • Unix Makefiles → Makefile
  • Ninja → build.ninja
  • Visual Studio → .sln + .vcxproj
  • Xcode → .xcodeproj

生成器表达式（Generator Expressions）在此阶段求值，可根据配置类型（Debug/Release）产生不同结果：

# 生成器表达式示例（配置时不求值，生成时才求值）
target_compile_definitions(exe1 PRIVATE
  $<$<CONFIG:Debug>:DEBUG_BUILD>
)
# Debug 模式编译时定义 DEBUG_BUILD，Release 时不定义

步骤 3：构建阶段（Build）

构建阶段调用原生构建工具来完成实际的编译和链接：

# 通用方式（CMake 抽象，内部调用 make/ninja/msbuild）
cmake --build build

# 指定构建类型（多配置生成器如 VS 在此阶段选择）
cmake --build build --config Release

# 构建特定目标
cmake --build build --target MyApp

# 并行构建
cmake --build build --parallel 8

# 先清理再构建
cmake --build build --clean-first

# 构建阶段输出示例
[ 10%] Building CXX object src/CMakeFiles/MyApp.dir/main.cpp.o
[ 60%] Building CXX object libs/CMakeFiles/mylib.dir/mylib.cpp.o
[100%] Linking CXX executable MyApp
[100%] Built target MyApp

生成器类型

📎 官方原文：cmake-generators(7)

官方文档："CMake 生成器负责为原生构建系统写入输入文件。对于每个构建树，必须选择且只能选择一个生成器。"

类别生成器平台配置时机
命令行工具Unix MakefilesLinux / macOS配置时指定（单配置）
Ninja跨平台配置时指定（单配置）
Ninja Multi-Config跨平台构建时选择（多配置）
IDEVisual Studio 17 2022Windows构建时选择（多配置）
XcodemacOS构建时选择（多配置）
NMake MakefilesWindows配置时指定（单配置）
💡 推荐
现代项目推荐使用 Ninja 生成器——它的增量构建速度比 Make 快，且跨平台支持好。在 CI/CD 环境中，Ninja 是首选。

构建类型（Build Configurations）

📎 官方原文：Build Configurations

官方文档说明：对于单配置生成器（如 Makefile、Ninja），配置类型在 cmake 阶段通过 CMAKE_BUILD_TYPE 指定；对于多配置生成器（如 Visual Studio、Xcode），CMAKE_BUILD_TYPE 被忽略，配置类型在构建时选择。

构建类型典型标志用途
Debug-g -O0调试，包含调试符号，无优化
Release-O3 -DNDEBUG发布，高优化，关闭断言
RelWithDebInfo-O2 -g -DNDEBUG带调试信息的优化构建
MinSizeRel-Os -DNDEBUG最小体积优化
 
# 单配置生成器（Makefile/Ninja）指定构建类型
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build

# 多配置生成器（VS/Xcode）在构建时指定
cmake -S . -B build -G "Visual Studio 17 2022"
cmake --build build --config Release

# 正确处理多配置的方式（生成器表达式）
target_compile_definitions(exe1 PRIVATE
  $<$<CONFIG:Debug>:DEBUG_BUILD>   # 不依赖 CMAKE_BUILD_TYPE
)

💡 本章总结

CMake 的编译流程分三步：配置（读取描述，产生 Cache）→ 生成（将描述转为原生构建文件）→ 构建（调用 make/ninja 实际编译）。这种两级抽象设计使得同一套描述能生成多种原生构建系统的文件。

第一性原理：CMake 是构建描述语言，而非构建执行工具。它的职责是忠实翻译意图，具体执行交给最熟悉本平台的原生工具。这种职责分离使得 CMake 的跨平台能力能够随着原生工具进化而进化。

第 5 章

构建系统原理

📎 官方原文：cmake-buildsystem(7)

二进制目标类型详解

官方文档："可执行文件和库通过 add_executable() 和 add_library() 命令定义。生成的二进制文件具有适合目标平台的前缀、后缀和扩展名。"

依赖传播机制

📎 官方原文：Build Spec & Usage Requirements

官方文档："目标根据自身的构建规范（Build Specification）与链接依赖传播的使用需求（Usage Requirements）共同构建。两者均通过目标专属命令指定。"

CMake 目标命令（Target Commands）控制两类属性：

  • 构建规范（Build Specification）：定义如何构建本目标（编译选项、包含目录、链接库）
  • 使用需求（Usage Requirements）：定义消费者使用本目标时需要满足的条件（接口包含目录、接口编译定义）
命令控制内容传播范围
target_include_directories()头文件搜索路径通过 INTERFACE 传播 INTERFACE_INCLUDE_DIRECTORIES
target_compile_definitions()预处理器宏定义通过 INTERFACE 传播 INTERFACE_COMPILE_DEFINITIONS
target_compile_options()编译器选项通过 INTERFACE 传播 INTERFACE_COMPILE_OPTIONS
target_link_libraries()链接库和传递依赖通过 INTERFACE 传播 INTERFACE_LINK_LIBRARIES
target_compile_features()所需编译器特性（如 C++17）通过 INTERFACE 传播
target_sources()源文件列表文件集支持 INTERFACE 传播（头文件集）

传递性依赖示例

# 官方文档示例
add_library(archive SHARED archive.cpp zip.cpp)

if (LZMA_FOUND)
  target_sources(archive PRIVATE lzma.cpp)
  # PRIVATE: BUILDING_WITH_LZMA 仅用于编译 archive
  target_compile_definitions(archive PRIVATE BUILDING_WITH_LZMA)
endif()

# INTERFACE: 消费者编译时获得 USING_ARCHIVE_LIB
target_compile_definitions(archive INTERFACE USING_ARCHIVE_LIB)

add_executable(consumer consumer.cpp)
# consumer.cpp 自动以 -DUSING_ARCHIVE_LIB 编译
target_link_libraries(consumer archive)

伪目标（Pseudo Targets）

📎 官方原文：Pseudo Targets

官方文档："有些目标类型不代表构建系统的输出，而只是输入——如外部依赖、别名或其他非构建产物。伪目标不在生成的构建系统中表示。"

伪目标类型说明典型用法
IMPORTED 目标代表预存在的依赖（如系统库）通过 find_package() 引入，如 OpenSSL::SSL
ALIAS 目标为现有目标创建可互换的别名Upstream::lib1 作为 lib1 的别名
INTERFACE 库无编译产物，仅传播使用需求头文件库（header-only），如 Eigen、{fmt}
 
# INTERFACE 库示例（官方文档，cmake-buildsystem(7)）
# 用于头文件库（Since CMake 3.23 支持 FILE_SET HEADERS）
add_library(Eigen INTERFACE)

target_sources(Eigen PUBLIC
  FILE_SET HEADERS
    BASE_DIRS src
    FILES src/eigen.h src/vector.h src/matrix.h
)

add_executable(exe1 exe1.cpp)
target_link_libraries(exe1 Eigen)
# exe1 自动获得 src/ 作为包含目录

💡 本章总结

CMake 构建系统以有向无环图（DAG）组织目标（Target）及其依赖关系。每个目标维护两类属性：构建规范（自用）和使用需求（传播给消费者）。CMake 通过 PUBLIC/PRIVATE/INTERFACE 精确控制属性的可见性，实现了传递性依赖的自动管理。

第一性原理：软件模块化的本质是"实现细节隐藏，接口公开"。CMake 的 Target + 使用需求机制，正是将这一面向对象的封装思想应用于构建系统，让每个库"知道"它的消费者需要什么，无需消费者手动配置。

第 6 章

适用场景

跨平台项目

CMake 最核心的适用场景是需要在多个平台和编译器上构建的 C/C++ 项目。官方概述页明确指出，CMake 支持：

  • Linux（GCC、Clang）
  • Windows（MSVC、MinGW、Clang）
  • macOS（Apple Clang、Xcode）
  • 嵌入式系统（通过工具链文件 CMAKE_TOOLCHAIN_FILE 支持交叉编译）

生态系统集成

CMake 已成为众多重要开源项目和工具链的核心构建系统：

项目/工具采用原因
LLVM / Clang2016 年完全迁移，跨平台和多配置需求
KDE2006 年迁移，快速移植到 Windows/macOS
Qt作为官方支持的构建系统
OpenCV跨平台视觉库，多平台一致构建
VTK / ITKCMake 的发源地
vcpkg / ConanC++ 包管理器，集成 CMake 包配置
Android NDK官方支持 CMake 构建原生代码

与其他构建系统对比

构建系统优势劣势适用场景
CMake生态最广、IDE 支持好、跨平台强学习曲线较陡、历史包袱C/C++ 中大型跨平台项目

CMake 适合的场景清单

  • ✅ 需要支持多个操作系统（Linux / Windows / macOS）
  • ✅ 需要与多种 IDE 集成（VS / CLion / VSCode / Xcode）
  • ✅ 有复杂的依赖关系（第三方库、子项目）
  • ✅ 需要发布可被其他项目引用的库（通过 find_package）
  • ✅ 嵌入式/交叉编译项目
  • ✅ 需要 CI/CD 集成的项目
  • ❌ 纯 Python/JS/Rust 项目（有更好的原生工具）
  • ❌ 极简单的单文件 C 程序（make 更直接）

💡 本章总结

CMake 最适合中大型 C/C++ 跨平台项目，尤其是需要同时支持多个 IDE、多个编译器或多个操作系统的场景。得益于强大的生态系统（LLVM、KDE、Qt、OpenCV 等都在用），CMake 已成为 C++ 社区的共同语言。

核心设计哲学

从第一性原理出发，理解 CMake 的一切决策：

现代 CMake 最佳实践

基于官方文档和社区共识，现代 CMake（3.x+）的最佳实践：

✅ 应该做

# 1. 始终声明最低版本
cmake_minimum_required(VERSION 3.20)

# 2. 使用 target_* 命令，而非全局命令
target_include_directories(mylib PUBLIC include/)  # ✅
# include_directories(include/)  # ❌ 全局污染

# 3. 为库正确指定 PUBLIC/PRIVATE/INTERFACE
target_link_libraries(app PRIVATE mylib)

# 4. 永远使用 Out-of-Source Build
# cmake -S . -B build  ✅

# 5. 用生成器表达式处理配置相关逻辑
target_compile_definitions(app PRIVATE
  $<$<CONFIG:Debug>:ENABLE_LOGGING>
)

# 6. 用 find_package() 而非硬编码路径查找依赖
find_package(OpenSSL REQUIRED)
target_link_libraries(app PRIVATE OpenSSL::SSL)

# 7. 用 CMakePresets.json 版本化构建配置
# cmake --preset release

❌ 避免的反模式

# 不要使用已废弃的全局命令
# include_directories(...)   ← 全局作用域，污染所有目标
# link_directories(...)      ← 同上
# add_definitions(...)       ← 同上

# 不要用 CMAKE_BUILD_TYPE 做构建时判断（多配置生成器无效）
# if(CMAKE_BUILD_TYPE STREQUAL "Debug")  ← 对 VS/Xcode 无效
# 正确方式：使用生成器表达式 $<CONFIG:Debug>

# 不要把 build/ 目录提交到版本控制
# .gitignore 中添加 /build

CMake 知识体系思维导图

全局总结：从第一性原理看 CMake

💡 全局第一性原理总结

问题本质：软件构建的本质需求是"以正确的选项将源文件转化为目标代码，并正确链接"。在多平台、多工具链环境下，这个问题演变成 N×M 矩阵——每个平台需要不同的命令和工具。

CMake 的解法：引入一个元构建系统（Meta Build System）作为中间层：

  • 向上：提供平台无关的高级声明式语言（CMakeLists.txt），让开发者描述"构建意图"而非"构建步骤"
  • 向下：翻译为各平台原生构建工具可理解的指令（Makefile / build.ninja / .sln 等）

核心抽象：Target（目标）是 CMake 最重要的抽象。Target 封装了两个关注点：①如何构建自身（PRIVATE 属性）；②消费者使用时需要什么（INTERFACE 属性）。这种"接口与实现分离"的思想，使得依赖关系可以自动传播，项目规模扩大时构建描述仍然可维护。

历史必然性：随着 C++ 生态从单一平台走向跨平台，从个人工具走向工业级工程（KDE、LLVM、Qt、OpenCV），需要一个能覆盖所有场景的统一工具。CMake 凭借其设计哲学（平台无关描述 + 原生生成 + 强大生态）成为了这个统一工具。
