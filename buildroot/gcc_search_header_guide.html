1编译链概述与工作流程

GCC（GNU Compiler Collection）并不是一个单一程序，而是一个驱动程序（Driver），它通过调用一系列子程序（cpp、cc1、as、ld 等）完成编译过程。在交叉编译场景下，这些子程序生成的代码针对与宿主机（Host）不同的目标机（Target）。

1.1 从源码到可执行文件的四个阶段

图1.1 GCC 编译四阶段示意。头文件搜索发生在预处理阶段，库文件搜索发生在链接阶段。

交叉编译的核心区别：交叉编译工具链（如 arm-linux-gnueabihf-gcc）在配置阶段就固化了目标架构相关的路径，这些路径与宿主机 gcc 完全隔离，避免误用宿主机头文件和库文件。

📌 本节总结

  • GCC 是驱动程序，分四阶段调用 cpp → cc1 → as → ld
  • 头文件搜索：预处理（cpp）阶段，与编译无关
  • 库文件搜索：链接（ld）阶段，由链接器 ld 执行
  • 交叉编译工具链名称前缀（如 arm-linux-gnueabihf-）标识目标三元组
参考文档：GCC Directory Options · CPP Search Path

2GCC 如何驱动子程序

GCC 驱动程序通过 Spec 字符串（spec strings）控制自身行为。每个子程序（cpp、cc1、as、ld）对应一个或多个 spec 字符串，驱动程序解析命令行参数并将其转换为各子程序的调用参数。

2.2 -B 选项对头文件和库文件的连带影响

根据官方文档，-B prefix 不仅影响子程序查找，还会：

  • 将 -B prefix/include 作为 -isystem 传递给预处理器（影响头文件搜索）
  • 将 -B prefix 转换为 -L 选项传递给链接器（影响库文件搜索）
Shell
# 使用 -B 指定工具链前缀，会同时影响头文件和库文件搜索
arm-linux-gnueabihf-gcc -B/opt/arm-toolchain/ main.c -o main

# 等效地，使用 GCC_EXEC_PREFIX 环境变量
export GCC_EXEC_PREFIX=/opt/arm-toolchain/
arm-linux-gnueabihf-gcc main.c -o main

📌 本节总结

  • GCC 是驱动程序，Spec 字符串控制每个阶段子程序的调用参数
  • -B prefix：指定子程序查找前缀，同时影响头文件（→ -isystem prefix/include）和库文件（→ -L prefix）搜索
  • 查找子程序顺序：-B 指定 → 标准系统前缀 → PATH
  • GCC_EXEC_PREFIX 环境变量与 -B 作用类似，但 -B 优先级更高

第一性原理：GCC 本质上是一个参数翻译器，它将用户的高层意图翻译成各子工具链程序的精确调用命令。

参考文档：GCC Directory Options — -B prefix · GCC Environment Variables

3头文件搜索顺序详解

根据 GCC 官方文档 Directory Options 和 CPP Search Path，头文件的搜索顺序如下：

3.1 #include "file" 引号形式（Quote Form）

  1. 当前源文件所在目录 例如 /src/myapp/main.c 中的 #include "config.h" → 先找 /src/myapp/config.h
  2. -iquote 指定的目录（按命令行从左到右） 仅对引号形式有效，不影响尖括号形式
  3. -I 指定的目录（按命令行从左到右） 同时适用于引号和尖括号两种 include 形式
  4. -isystem 指定的目录（按命令行从左到右） 标记为系统目录，编译器对其中的警告会有特殊处理
  5. 标准系统目录（内置） 编译时固化，与目标架构相关，例如 sysroot/usr/include 等
  6. -idirafter 指定的目录（按命令行从左到右） 最低优先级，用于在系统目录之后追加搜索路径

3.2 #include <file> 尖括号形式（Angle-bracket Form）

官方文档原文："For the angle-bracket form #include <file>, the preprocessor's default behavior is to look only in the standard system directories."（不会先查当前文件目录，直接从第②步 -I 开始）
  1. -I 指定的目录（按命令行从左到右） 尖括号形式跳过 -iquote 目录，直接从 -I 开始
  2. -isystem 指定的目录（按命令行从左到右） 同引号形式
  3. 标准系统目录（内置） 同引号形式的第5步
  4. -idirafter 指定的目录（按命令行从左到右） 同引号形式

3.3 完整顺序流程图

图3.1 头文件搜索完整顺序（引号形式从①开始；尖括号形式从③开始）

3.4 重要规则：-I 与系统目录重叠时

官方文档注意事项：如果某个标准系统目录（或 -isystem 指定的目录）也被 -I 显式指定，则 -I 指定的那个实例被忽略，该目录仍以"系统目录"的身份在正常位置被搜索。这样做是为了确保 GCC 修复有问题的系统头文件的机制和 #include_next 指令的顺序不被意外改变。若确实需要调整系统目录顺序，应使用 -nostdinc 配合 -isystem。

📌 本节总结

  • #include "..."：①当前目录 → ②-iquote → ③-I → ④-isystem → ⑤系统内置 → ⑥-idirafter
  • #include <...>：跳过①② → 从③-I 开始
  • -I 在系统目录之前，可覆盖系统头文件（谨慎使用）
  • -isystem 的目录被视为系统目录，编译器抑制其中头文件的警告
  • -idirafter 优先级最低，适合"最后备用"路径
原文来源：Using the GNU Compiler Collection — Directory Options · The C Preprocessor — 2.3 Search Path

4控制头文件路径的选项

选项 适用形式 优先级 说明
-I dir 两种 高（在系统目录之前） 最常用，同时适用于 "..." 和 <...>
-iquote dir 仅 "..." 低于 -I 仅影响引号形式，更精细的控制
-isystem dir 两种 低于 -I，高于内置系统目录 将目录标记为系统目录，抑制其中的警告
-idirafter dir 两种 最低 在所有其他目录之后搜索，用作兜底路径
-isysroot dir 两种 影响系统目录基础 仅影响头文件的 sysroot（不影响库）
--sysroot=dir 两种（含库） 影响系统目录基础 同时影响头文件和库文件搜索基础路径
-nostdinc 两种 禁用系统目录 不搜索任何标准系统目录，常用于裸机/内核编译
-nostdinc++ C++ 专用 禁用 C++ 系统目录 仅禁用 C++ 特有的标准目录，保留其他系统目录

4.1 典型使用示例

Shell — 交叉编译头文件路径控制
# 示例1：最常见的交叉编译头文件指定方式
arm-linux-gnueabihf-gcc \
  -I/opt/sysroot/usr/include \
  -I/opt/sysroot/usr/include/arm-linux-gnueabihf \
  -c main.c -o main.o

# 示例2：使用 --sysroot 自动重定向所有系统路径
# 编译器会把 /usr/include → sysroot/usr/include
arm-linux-gnueabihf-gcc \
  --sysroot=/opt/sysroot \
  -c main.c -o main.o

# 示例3：裸机编译，完全禁用标准系统头文件
arm-none-eabi-gcc \
  -nostdinc \
  -I/opt/arm-none-eabi/include \
  -c startup.c -o startup.o

# 示例4：使用 -isysroot 仅影响头文件（库文件用 --sysroot 单独控制）
arm-linux-gnueabihf-gcc \
  -isysroot /opt/headers-root \
  --sysroot=/opt/libs-root \
  -c main.c -o main.o
-isysroot 与 --sysroot 的区别：
• --sysroot=dir：同时影响头文件和库文件搜索的基础路径
• -isysroot dir：仅影响头文件（Darwin 目标除外），不影响链接库搜索
两者同时使用时，-isysroot 管头文件，--sysroot 管库文件。

📌 本节总结

  • 选项优先级从高到低：-I > -isystem > 系统内置 > -idirafter
  • -iquote 只对 "..." 有效，不影响 <...>
  • 交叉编译推荐使用 --sysroot，它是最干净的路径重定向方式
  • 裸机/内核编译：使用 -nostdinc + 精确的 -I 路径
原文来源：GCC Directory Options 完整说明

5环境变量对头文件的影响

根据 GCC 官方文档 Environment Variables Affecting GCC，以下环境变量影响头文件搜索：

环境变量 适用语言 等效选项 说明
CPATH 所有语言 -I（在命令行 -I 之后） 冒号分隔的目录列表，等效为 -I 但优先级低于命令行 -I
C_INCLUDE_PATH C -isystem（在命令行 -isystem 之后） 仅在预处理 C 文件时生效，以系统目录方式添加
CPLUS_INCLUDE_PATH C++ -isystem（在命令行 -isystem 之后） 仅在预处理 C++ 文件时生效
OBJC_INCLUDE_PATH Obj-C -isystem（在命令行 -isystem 之后） 仅在预处理 Objective-C 文件时生效
官方优先级声明："options such as -B, -I and -L … take precedence over places specified using environment variables, which in turn take precedence over those specified by the configuration of GCC."

即：命令行选项 > 环境变量 > GCC 内置配置

5.1 CPATH 的特殊语义

官方文档说明：CPATH 的目录列表等效为 -I，但添加在命令行所有 -I 选项之后。空元素（:: 或首尾的 :）表示当前工作目录。

例如：CPATH=:/special/include 等效于 -I. -I/special/include

Shell
# 设置 C 头文件搜索路径（等效于 -isystem，但优先级低于命令行 -isystem）
export C_INCLUDE_PATH=/opt/sysroot/usr/include:/opt/mylibs/include

# 设置所有语言头文件搜索路径（等效于 -I）
export CPATH=/opt/extra/include

# 分隔符：Linux/macOS 用冒号，Windows 用分号
arm-linux-gnueabihf-gcc main.c -o main

📌 本节总结

  • 总优先级：命令行选项 > 环境变量 > GCC 内置配置
  • CPATH 等效为 -I，对所有语言生效，但在命令行 -I 之后
  • C_INCLUDE_PATH/CPLUS_INCLUDE_PATH 等效为 -isystem，在命令行 -isystem 之后
  • 在交叉编译中，建议优先用命令行选项或 Makefile 变量，环境变量可能污染全局构建环境

第一性原理：越具体（命令行）的配置优先于越通用（环境变量）的配置，越通用的优先于最抽象（内置默认值）的配置。

原文来源：GCC — 3.21 Environment Variables Affecting GCC

6库文件搜索顺序详解

库文件搜索由链接器 ld 执行。GCC 将 -l、-L 等选项传递给 ld。搜索顺序与头文件类似，也是从最高优先级到最低：

  1. -L 选项指定的目录（按命令行从左到右） gcc -L/opt/sysroot/lib -L/opt/mylibs/lib -lfoo ... 优先于所有其他目录
  2. LIBRARY_PATH 环境变量指定的目录 等效于 -L，但在命令行 -L 之后；仅原生编译器适用（交叉编译时通常不生效）
  3. GCC 编译时配置的内置目录（-B/GCC_EXEC_PREFIX 转换而来的 -L） GCC 自动将 -B 前缀转换为 -L 传递给链接器
  4. 链接器脚本中 SEARCH_DIR() 定义的目录 在 binutils 配置时固化，可用 ld --verbose 查看；与 sysroot 配合使用
  5. 标准系统库目录（ld 内置） 例如 /lib, /usr/lib, sysroot/lib, sysroot/usr/lib 等，编译 binutils 时确定

6.1 静态库 vs 动态库的选择

根据 Red Hat Developer Guide — Using Libraries with GCC：

  • 当遇到 -lfoo 时，链接器先寻找 libfoo.so（动态库），再寻找 libfoo.a（静态库）
  • 若两者都找到，默认使用动态库（除非加 -static 强制使用静态库）
  • 命令行顺序很重要：foo.o -lz bar.o 中，-lz 仅解决 foo.o 的符号，不解决 bar.o 的符号

图6.1 链接器处理 -lfoo 的搜索与决策流程

📌 本节总结

  • 库搜索顺序：-L 指定 → LIBRARY_PATH → GCC 内置 → ld 脚本 SEARCH_DIR → ld 系统默认
  • 默认优先使用动态库（.so），不存在才使用静态库（.a）；-static 强制静态
  • -l 选项的位置影响符号解析：应放在用到该库符号的目标文件之后
  • 交叉编译时 LIBRARY_PATH 通常无效（仅原生编译器使用）
原文来源：GCC Link Options · Red Hat Developer Guide — Using Libraries with GCC

7控制库文件路径的选项

选项 作用 说明
-L dir 添加库搜索目录 最常用；可多次使用，按从左到右顺序搜索；必须在 -l 之前指定
-l library 链接指定库 寻找 libXXX.so 或 libXXX.a；顺序影响符号解析
--sysroot=dir 重定向系统路径基础 将所有系统库路径前缀替换为 dir（需 GNU ld 2.16+）
-static 强制静态链接 阻止动态库被链接
-shared 生成动态库 生成 .so 文件
-Wl,-rpath=dir 设置运行时库路径 在二进制文件中嵌入运行时库搜索路径（与链接时路径不同）
-nostdlib 禁用标准库 不链接任何标准启动文件和库，常用于裸机/内核
-nodefaultlibs 禁用默认库 不链接标准系统库，但保留启动文件

7.1 典型交叉编译链接示例

Shell — 交叉编译库文件路径控制
# 示例1：指定库目录并链接库
arm-linux-gnueabihf-gcc \
  -L/opt/sysroot/lib \
  -L/opt/sysroot/usr/lib \
  main.o -o myapp \
  -lfoo -lbar

# 注意：-L 必须在 -l 之前！

# 示例2：使用 --sysroot 自动重定向库路径
# 编译器将 /usr/lib → sysroot/usr/lib
arm-linux-gnueabihf-gcc \
  --sysroot=/opt/sysroot \
  main.o -o myapp -lfoo

# 示例3：嵌入运行时库路径（rpath）
arm-linux-gnueabihf-gcc \
  -L/opt/sysroot/lib \
  -Wl,-rpath=/usr/local/lib \
  main.o -o myapp -lfoo

# 示例4：裸机，完全不使用标准库
arm-none-eabi-gcc \
  -nostdlib \
  -L/opt/arm-none-eabi/lib \
  startup.o main.o -lgcc -o firmware.elf
-L 选项顺序要求（官方文档）："The order of options matters: GCC cannot link against a library foo unless it knows the directory of this library. Therefore, use the -L options to specify library directories before using the -l options for linking against libraries."

📌 本节总结

  • -L dir 必须出现在对应 -l 之前，否则链接失败
  • --sysroot 是交叉编译最推荐的方式，自动处理所有系统路径重定向
  • 链接时路径（-L）和运行时路径（-rpath）是两个独立概念
  • 裸机编译使用 -nostdlib，通常还需手动链接 -lgcc
原文来源：GCC Link Options · GCC Directory Options — -L

8环境变量对库文件的影响

环境变量 影响阶段 说明
LIBRARY_PATH 链接时（GCC） 等效为 -L，在命令行 -L 之后添加；官方注明仅原生编译器使用，交叉编译时通常无效
LD_LIBRARY_PATH 运行时（动态链接器） 影响程序运行时加载动态库的路径，与编译链接无关
GCC_EXEC_PREFIX 链接时（通过 -L 转换） GCC 将此前缀转换为 -L 传递给链接器，用于查找 libgcc 等 GCC 内部库
LIBRARY_PATH 与 LD_LIBRARY_PATH 的本质区别：
  • LIBRARY_PATH：编译时，GCC 驱动程序使用，指定链接器搜索路径
  • LD_LIBRARY_PATH：运行时，动态链接器（ld.so）使用，与 GCC 无关
交叉编译时，两者都可能需要正确设置，但作用阶段完全不同。
Shell
# 官方文档说明：LIBRARY_PATH 仅在原生编译时有效
# 对于交叉编译，请使用 -L 选项
export LIBRARY_PATH=/opt/mylibs/lib
gcc main.c -o main -lfoo  # 原生编译有效

# 运行时动态库路径（与编译无关）
export LD_LIBRARY_PATH=/opt/mylibs/lib
./main  # 运行时生效

# 交叉编译请明确使用 -L
arm-linux-gnueabihf-gcc \
  -L/opt/sysroot/usr/lib \
  main.c -o main -lfoo

📌 本节总结

  • LIBRARY_PATH：编译时库路径，仅原生编译有效，交叉编译应显式使用 -L
  • LD_LIBRARY_PATH：运行时库路径，与编译无关，由动态链接器在程序运行时使用
  • 整体优先级：命令行 -L > LIBRARY_PATH > GCC 内置路径
  • 交叉编译环境下不要依赖环境变量，所有路径应在构建系统（Makefile/CMake）中明确指定
原文来源：GCC — Environment Variables — LIBRARY_PATH

9Specs 文件机制：内置 vs 外部配置

GCC 的行为由 Spec 字符串（spec strings）控制——这正是"内置配置"的实体。

9.1 什么是 Spec 字符串

根据官方文档 GCC Spec Files：

"gcc is a driver program. It performs its job by invoking a sequence of other programs to do the work of compiling, assembling and linking. GCC interprets its command-line parameters and uses these to deduce which programs it should invoke, and which command-line options it ought to place on their command lines. This behavior is controlled by spec strings."

GCC 内置了一套 Spec 字符串，包含以下关键条目：

  • cpp：传递给 C 预处理器的选项（包含头文件路径）
  • cc1：传递给 C 编译器的选项
  • link：传递给链接器的选项
  • lib：要链接的库列表
  • startfile：链接开头的目标文件（如 crt0.o）
  • endfile：链接末尾的目标文件

9.2 是内置的还是外部文件？

图9.1 Specs 配置的三个层次及其优先级关系

9.3 查看和操作 Specs

Shell
# 1. 查看当前 GCC 使用的内置 specs（所有配置的根源）
arm-linux-gnueabihf-gcc -dumpspecs

# 2. 将内置 specs 输出到文件（以便修改）
arm-linux-gnueabihf-gcc -dumpspecs > \
  $(dirname $(arm-linux-gnueabihf-gcc -print-libgcc-file-name))/specs

# 3. 验证 GCC 是否正在读取外部 specs 文件
arm-linux-gnueabihf-gcc -v 2>&1 | head -5
# 输出 "Using built-in specs." 或 "Reading specs from /path/to/specs"

# 4. 编译时临时使用指定 specs 文件
arm-linux-gnueabihf-gcc -specs=/path/to/my.specs main.c -o main

9.4 路径是内置的还是外部配置的？

路径类型 内置/外部 确定时机 修改方式
GCC 内置头文件目录（gcc-lib-dir/include） 内置 编译 GCC 时 --sysroot 或 -I 覆盖
sysroot/usr/include 内置（由 --with-sysroot 决定） 配置 GCC 时 --sysroot 运行时覆盖
链接器 SEARCH_DIR 内置（链接脚本中） 编译 binutils 时 -L 覆盖或修改链接脚本
-I 指定目录 外部（命令行） 每次编译时 直接修改编译命令
-L 指定目录 外部（命令行） 每次链接时 直接修改链接命令
CPATH / C_INCLUDE_PATH 外部（环境变量） Shell 环境 export 设置

📌 本节总结

  • GCC 的默认搜索路径是编译时内置的，固化在 GCC 二进制文件的 Spec 字符串中
  • 可通过在 libgcc 目录放置 specs 文件来覆盖内置配置
  • -specs=file 命令行选项优先级最高，仅影响本次编译
  • 用 gcc -dumpspecs 可导出当前全部 Spec 配置，这是内置配置的唯一可见方式

第一性原理：GCC 的"内置配置"本质上是编译 GCC 本身时 ./configure 参数的固化结果。所有运行时能改变的路径，都是对这些固化值的覆盖和补充。

原文来源：GCC 14.2.0 — Spec Files · CrossGCC 邮件列表：where is built-in specs located?

10Sysroot 与交叉编译路径机制

交叉编译中，sysroot 是最核心的路径控制机制，它定义了目标系统的虚拟根目录。

10.1 Sysroot 的作用原理

根据官方文档 --sysroot=dir 的说明：

"Use dir as the logical root directory for headers and libraries. For example, if the compiler normally searches for headers in /usr/include and libraries in /usr/lib, it instead searches dir/usr/include and dir/usr/lib."

图10.1 Sysroot 机制将编译器的搜索基础从宿主机重定向到目标机根文件系统

10.2 交叉编译工具链的内置 sysroot

交叉编译工具链通常在 --with-sysroot 配置时就设定了默认 sysroot。运行 gcc -v 可以看到工具链的配置参数，例如：

实际输出示例（来自 Debian aarch64-linux-gnu-gcc）

  --with-pkgversion='Debian 10.2.1-6'
  --target=aarch64-linux-gnu
  --prefix=/usr
  --with-sysroot=/         ← 配置时设定的默认 sysroot
  --enable-multiarch
  ...
  
# 头文件搜索路径（aarch64-linux-gnu-gcc --sysroot=/sysroot -v -E /dev/null）
#include <...> search starts here:
 /usr/lib/gcc-cross/aarch64-linux-gnu/10/include    ← GCC 内置
 /usr/lib/gcc-cross/aarch64-linux-gnu/10/../../../aarch64-linux-gnu/include ← 工具链包含
 /sysroot/usr/include/aarch64-linux-gnu             ← sysroot + 架构特定
 /sysroot/usr/include                               ← sysroot + 通用
End of search list.

10.3 crosstool-NG 的默认搜索路径

根据 crosstool-NG 官方文档，默认搜索路径如下：

crosstool-NG 工具链默认头文件路径
#include <...> search starts here:
 /lib/gcc/<target>/<version>/include          ← GCC 自身的 include
 /lib/gcc/<target>/<version>/include-fixed   ← 修正过的系统头文件
 /lib/gcc/<target>/<version>/../../../<target>/include  ← 目标三元组 include
 /<sysroot>/usr/include                       ← sysroot 中的系统头文件
End of search list.

📌 本节总结

  • --sysroot=dir：交叉编译的核心，将所有系统路径搜索基础重定向到目标机根目录
  • --with-sysroot（配置时）：在编译 GCC 时固化默认 sysroot，运行时可用 --sysroot 覆盖
  • 交叉编译工具链通常已经配置了正确的 sysroot，-v 可查看
  • sysroot 目录结构必须与目标机的文件系统布局一致（含 usr/include, usr/lib 等）

第一性原理：交叉编译的本质是在宿主机上模拟目标机的文件系统视角。Sysroot 就是这个模拟视角的物理实现——给编译器一个"假"的根目录，让它以为自己在目标机上运行。

原文来源：GCC — --sysroot 选项 · crosstool-NG — Using the toolchain

11如何查看当前配置

以下命令均有官方文档支持，是查看 GCC 实际搜索路径的权威方式。

11.1 查看头文件搜索路径

Shell — 查看头文件搜索路径
# 方法1：最权威方式 - 让 CPP 在预处理空文件时输出搜索路径
# 官方文档原文："You can find the default search directory list for your
# version of CPP by invoking it with the -v option."
cpp -v /dev/null -o /dev/null

# 方法2：使用 gcc -E -v（等效，更常用）
arm-linux-gnueabihf-gcc -E -v -x c /dev/null 2>&1

# 方法3：使用 gcc -v 加参数以精确控制显示内容
echo | arm-linux-gnueabihf-gcc -E -x c - -v 2>&1 \
  | sed -n '/#include/,/End of search list/p'

# 方法4：使用 -H 参数追踪头文件实际包含路径
arm-linux-gnueabihf-gcc -H -c main.c 2>&1

# 典型输出结构：
# #include "..." search starts here:
# #include <...> search starts here:
#  /usr/lib/gcc/arm-linux-gnueabihf/11/include
#  /usr/local/include
#  /usr/include/arm-linux-gnueabihf
#  /usr/include
# End of search list.

11.2 查看库文件搜索路径

Shell — 查看库文件搜索路径
# 方法1：gcc -print-search-dirs（查看 GCC 配置的搜索路径）
arm-linux-gnueabihf-gcc -print-search-dirs

# 输出中的 libraries: 行即为库搜索路径（格式化显示）
arm-linux-gnueabihf-gcc -print-search-dirs \
  | grep libraries \
  | tr -s ' ;=:' '\n'

# 方法2：查看链接器内置 SEARCH_DIR（链接脚本中固化的路径）
arm-linux-gnueabihf-ld --verbose | grep SEARCH_DIR

# 输出示例：
# SEARCH_DIR("=/usr/local/lib/arm-linux-gnueabihf");
# SEARCH_DIR("=/usr/local/lib");
# SEARCH_DIR("=/lib/arm-linux-gnueabihf");
# SEARCH_DIR("=/lib");
# SEARCH_DIR("=/usr/lib/arm-linux-gnueabihf");
# SEARCH_DIR("=/usr/lib");

# 方法3：查看 GCC 完整的 verbose 链接过程
arm-linux-gnueabihf-gcc -v main.c -o main 2>&1

11.3 查看 Specs 配置

Shell — 查看 Specs 配置
# 查看 GCC 内置 specs 全文（这是所有内置配置的来源）

# 查看 GCC 是否读取外部 specs 文件（输出头几行）

# 查看 GCC 配置参数（--with-sysroot 等在此可见）

📌 全文总结（第一性原理视角）

GCC 的搜索机制可以从三个第一性原理来理解：

  • 原理1 — 编译时固化：GCC 和 binutils 在 ./configure && make 时将目标架构信息、sysroot 路径、标准目录列表等"烧录"进 Spec 字符串和链接脚本。这些是不可见的默认值。
  • 原理2 — 分层覆盖：命令行选项 > 环境变量 > 内置配置。每一层都可以精确覆盖下层。理解这个层次关系，就理解了所有路径问题的调试思路。
  • 原理3 — 构建与运行分离：编译时搜索路径（-I/-L）和运行时搜索路径（LD_LIBRARY_PATH/rpath）是完全独立的两套机制。前者由 GCC 驱动，后者由动态链接器（ld.so）管理。

交叉编译的核心挑战：确保编译时所有路径指向目标机环境（sysroot），而非宿主机环境。--sysroot 是最优雅的解决方案。
