📖总览与背景

在 Linux 交叉编译工具链（如 arm-linux-gnueabihf、aarch64-linux-gnu 等）中， 目标系统的 usr/include 目录包含了用户空间程序编译所需的全部头文件。 这些头文件来源不一、职责各异，按来源可分为四大类：

ISO C 标准（C89～C11）：由 ISO/IEC 9899 标准规定，任何合规 C 编译器都必须提供
POSIX / IEEE 1003.1：POSIX.1-2008（IEEE Std 1003.1-2017）规定的可移植接口
Linux 内核 UAPI：内核导出给用户空间的接口（make headers_install）
glibc/系统特定扩展：GNU C Library 或特定平台提供的非标准扩展

权威来源参考：

• C 标准头文件列表： cppreference.com — C Standard Library header files （依据 ISO/IEC 9899:2018, 9899:2011, 9899:1999）

• 内核头文件导出机制： kernel.org — Exporting kernel headers for use by userspace

• POSIX 头文件规范： man7.org — POSIX.1-2017 manual pages (IEEE Std 1003.1-2017)

• LFS 内核头安装说明： Linux From Scratch — Linux API Headers

🔵C 标准规定的头文件

权威来源：ISO/IEC 9899（C 语言标准），各版本分别规定了以下头文件。 参考：C Standard Library header files — cppreference.com

C 标准库的接口由一组头文件定义。引用 C17 标准（ISO/IEC 9899:2018）第 7.1.2 节："Standard headers"。 C 标准头文件由 libc 实现（如 glibc、musl、newlib）提供，与内核无直接关联。

📌 C89/C90 规定的头文件（ISO/IEC 9899:1990）

以下 15 个头文件由 ANSI C89/ISO C90 标准首次规定，是最基础的 C 标准库接口：

<assert.h> C89
提供 assert() 宏，用于运行时断言检查。断言失败时打印诊断信息并调用 abort()。可通过定义 NDEBUG 宏禁用。
来源：ISO/IEC 9899:1990 §4.2 | 实现：glibc
<ctype.h> C89
字符分类函数：isalpha()、isdigit()、isspace() 等，以及大小写转换 toupper()/tolower()。
来源：ISO/IEC 9899:1990 §4.3 | 实现：glibc
<errno.h> C89
定义 errno 宏及错误号常量（EDOM、ERANGE 等）。Linux 下实际错误码由内核定义，通过 glibc 封装暴露。
来源：ISO/IEC 9899:1990 §4.1.3 | 实现：glibc（部分来自内核）
<float.h> C89
定义浮点类型的实现限制常量，如 FLT_MAX、DBL_EPSILON、FLT_DIG 等，依赖于具体硬件的浮点单元（FPU）规格。
来源：ISO/IEC 9899:1990 §4.1.4 | 实现：GCC/glibc
<limits.h> C89
定义整数类型的实现限制，如 INT_MAX、LONG_MIN、CHAR_BIT 等。与目标架构的 ABI 相关，交叉编译时尤为重要。
来源：ISO/IEC 9899:1990 §4.1.5 | 实现：GCC/glibc
<locale.h> C89
本地化工具：setlocale()、localeconv()，以及 struct lconv。用于处理语言、货币、数字格式等本地化信息。
来源：ISO/IEC 9899:1990 §4.4 | 实现：glibc
<math.h> C89
数学函数库：sin()、cos()、sqrt()、pow()、log() 等，以及 HUGE_VAL、NAN、INFINITY 等常量。
来源：ISO/IEC 9899:1990 §4.5 | 实现：glibc（libm）
<setjmp.h> C89
非本地跳转：setjmp() 和 longjmp()，用于跨函数边界的控制流跳转。底层依赖架构相关的寄存器保存机制。
来源：ISO/IEC 9899:1990 §4.6 | 实现：glibc（架构相关汇编）
<signal.h> C89
信号处理：signal()、raise()，以及信号宏（SIGINT、SIGTERM 等）。Linux 实际信号机制由内核实现，此处是标准化封装。
来源：ISO/IEC 9899:1990 §4.7 | 实现：glibc（封装内核信号）
<stdarg.h> C89
可变参数支持：va_list 类型，va_start()、va_arg()、va_end()、va_copy() 宏。通常由编译器内建实现（GCC built-ins）。
来源：ISO/IEC 9899:1990 §4.8 | 实现：GCC built-in
<stddef.h> C89
基础类型和宏定义：NULL、size_t、ptrdiff_t、offsetof()，以及 C11 引入的 max_align_t。几乎所有头文件都间接依赖此文件。
来源：ISO/IEC 9899:1990 §4.1.5 | 实现：GCC/glibc
<stdio.h> C89
标准输入/输出：printf()、scanf()、fopen()、fread()、fwrite()、fclose()，以及 FILE 类型和标准流 stdin/stdout/stderr。
来源：ISO/IEC 9899:1990 §4.9 | 实现：glibc
<stdlib.h> C89
通用工具：内存管理（malloc()/free()/realloc()）、程序控制（exit()/abort()）、字符串转换（atoi()/strtol()）、随机数（rand()）等。
来源：ISO/IEC 9899:1990 §4.10 | 实现：glibc
<string.h> C89
字符串和内存操作函数：strcpy()、strcmp()、strlen()、memcpy()、memset()、memmove() 等。
来源：ISO/IEC 9899:1990 §4.11 | 实现：glibc（含SIMD优化）
<time.h> C89
日期时间工具：time()、clock()、mktime()、strftime()、struct tm、time_t。Linux 实际时钟由内核维护。
来源：ISO/IEC 9899:1990 §4.12 | 实现：glibc

📌 C95（Amendment 1）新增头文件

<iso646.h> C95
定义 C 运算符的替代拼写宏：and（&&）、or（||）、not（!）等，为不支持某些字符的键盘提供兼容性。
来源：ISO/IEC 9899:1994 Amendment 1 | 实现：glibc
<wchar.h> C95
宽字符（wchar_t）和多字节字符工具：wprintf()、wcscpy()、wcslen()、mbstowcs()、wcstombs() 等。
来源：ISO/IEC 9899:1994 Amendment 1 | 实现：glibc
<wctype.h> C95
宽字符分类和转换函数：iswalpha()、iswdigit()、towupper()、towlower()，以及可扩展的字符类 wctype_t。
来源：ISO/IEC 9899:1994 Amendment 1 | 实现：glibc

📌 C99 新增头文件（ISO/IEC 9899:1999）

<complex.h> C99
复数算术运算：creal()、cimag()、cabs()、cexp()、csin() 等。依赖硬件浮点支持，交叉编译时需注意 FPU 配置。
来源：ISO/IEC 9899:1999 §7.3 | 实现：glibc（libm）
<fenv.h> C99
浮点环境控制：fegetround()/fesetround()（舍入模式）、fetestexcept()/feclearexcept()（浮点异常标志）。
来源：ISO/IEC 9899:1999 §7.6 | 实现：glibc（架构相关）
<inttypes.h> C99
固定宽度整数类型的格式化宏：PRId32、PRIu64、SCNd8 等，以及 imaxabs()、strtoimax() 等函数。
来源：ISO/IEC 9899:1999 §7.8 | 实现：glibc
<stdbool.h> C99
定义布尔类型宏：bool（_Bool）、true（1）、false（0）。C23 起 bool/true/false 成为内置关键字，此头文件变为兼容层。
来源：ISO/IEC 9899:1999 §7.16 | 实现：GCC/glibc
<stdint.h> C99
固定宽度整数类型定义：int8_t、uint32_t、int64_t、intptr_t、uintmax_t，以及对应的极值宏。跨平台代码的基础。
来源：ISO/IEC 9899:1999 §7.18 | 实现：GCC/glibc
<tgmath.h> C99
类型泛型数学宏，封装 math.h 和 complex.h，根据参数类型自动选择正确的数学函数（如对 float 调用 sinf()，double 调用 sin()）。
来源：ISO/IEC 9899:1999 §7.22 | 实现：glibc

📌 C11 新增头文件（ISO/IEC 9899:2011）

<stdalign.h> C11
提供 alignas 和 alignof 便利宏（对应 _Alignas/_Alignof 关键字），用于控制对象的内存对齐。C23 起成为关键字。
来源：ISO/IEC 9899:2011 §7.15 | 实现：GCC
<stdatomic.h> C11
原子操作库：atomic_int、atomic_load()、atomic_store()、atomic_compare_exchange_strong() 等，基于内存模型提供无锁并发原语。
来源：ISO/IEC 9899:2011 §7.17 | 实现：GCC built-ins + glibc
<stdnoreturn.h> C11
定义 noreturn 便利宏（对应 _Noreturn 关键字），标记不会正常返回的函数（如 exit()、abort()），帮助编译器进行代码优化。
来源：ISO/IEC 9899:2011 §7.23 | 实现：GCC
<threads.h> C11
线程库：thrd_create()、mtx_lock()/mtx_unlock()、cnd_wait()/cnd_signal() 等。Linux 下底层由 POSIX pthreads（pthread.h）实现。
来源：ISO/IEC 9899:2011 §7.26 | 实现：glibc（封装 pthreads）
<uchar.h> C11
Unicode 字符工具：char16_t、char32_t 类型，以及 mbrtoc16()/mbrtoc32()（多字节转 UTF-16/UTF-32）和对应逆向函数。
来源：ISO/IEC 9899:2011 §7.28 | 实现：glibc

📋 C 标准头文件小结

  • 总计 29 个头文件由 ISO C 标准历次版本规定（C89:15 + C95:3 + C99:6 + C11:5）
  • 实现者是 libc（Linux 下通常是 glibc 或 musl），而非内核。编译器（GCC）也直接提供少量宏/built-in（如 stdarg.h、stddef.h）
  • 与内核的关系：C 标准头文件在逻辑上独立于内核，但某些功能（如信号、时间、内存映射）的底层实现通过系统调用触达内核
  • 第一性原理：C 标准的核心目标是可移植性——用同一套头文件在不同平台上编译出行为一致的程序。因此这些头文件定义的是规范，而非实现
  • 交叉编译注意：limits.h、float.h 中的值是目标架构相关的，必须来自目标 sysroot，而非宿主机

🟡POSIX 规定的扩展头文件

权威来源：IEEE Std 1003.1-2017（POSIX.1-2017），即 The Open Group Base Specifications Issue 7。 参考手册页：man7.org POSIX man pages (0p section)

POSIX 扩展了 C 标准，增加了文件系统、进程管理、网络、线程等操作系统级接口。这些头文件由 glibc 在 Linux 上实现，与内核 UAPI 存在协作关系，但不属于内核头文件。

📁 sys/ 子目录（系统调用封装）

<sys/types.h> POSIX
定义基础系统类型：pid_t（进程ID）、uid_t（用户ID）、gid_t（组ID）、off_t（文件偏移）、size_t、ssize_t、dev_t、ino_t、mode_t 等。几乎所有 POSIX 接口的基础。
来源：POSIX.1-2017 §sys_types.h | 实现：glibc
<sys/stat.h> POSIX
文件状态接口：struct stat（含 st_mode/st_size/st_mtime 等成员），stat()/fstat()/lstat() 函数，以及文件类型宏 S_ISREG()/S_ISDIR() 和权限常量 S_IRUSR 等。
来源：POSIX.1-2017 §sys_stat.h | 实现：glibc（封装 stat 系统调用）
<sys/socket.h> POSIX
BSD 套接字接口：socket()、bind()、connect()、listen()、accept()、send()/recv()，struct sockaddr，AF_INET/SOCK_STREAM 等常量。
来源：POSIX.1-2017 §sys_socket.h | 实现：glibc（封装内核网络栈）
<sys/mman.h> POSIX
内存映射接口：mmap()/munmap()（内存映射）、mprotect()（权限保护）、msync()（同步）、mlock()/munlock()（内存锁定）。
来源：POSIX.1-2017 §sys_mman.h | 实现：glibc（封装 mmap 系统调用）
<sys/wait.h> POSIX
进程等待：wait()/waitpid()/waitid()，以及退出状态宏 WIFEXITED()/WEXITSTATUS()/WIFSIGNALED() 等。
来源：POSIX.1-2017 §sys_wait.h | 实现：glibc
<sys/select.h> POSIX
I/O 多路复用：select()/pselect()，fd_set 类型，FD_SET()/FD_CLR()/FD_ISSET()/FD_ZERO() 宏。
来源：POSIX.1-2017 §sys_select.h | 实现：glibc
<sys/time.h> POSIX
时间结构体：struct timeval（微秒精度）、struct timezone，以及 gettimeofday()/settimeofday()、getitimer()/setitimer()。
来源：POSIX.1-2017 §sys_time.h | 实现：glibc
<sys/resource.h> POSIX
资源限制与使用统计：getrlimit()/setrlimit()（资源限制）、getrusage()（CPU/内存使用量），以及 RLIMIT_STACK/RLIMIT_NOFILE 等常量。
来源：POSIX.1-2017 §sys_resource.h | 实现：glibc
<sys/ipc.h> POSIX
System V IPC 基础：key_t 类型、struct ipc_perm、ftok() 函数，为消息队列（sys/msg.h）、共享内存（sys/shm.h）、信号量（sys/sem.h）提供基础。
来源：POSIX.1-2017 §sys_ipc.h | 实现：glibc
<sys/msg.h> POSIX
System V 消息队列：msgget()/msgsnd()/msgrcv()/msgctl()，以及 struct msqid_ds。用于进程间通信。
来源：POSIX.1-2017 §sys_msg.h | 实现：glibc
<sys/shm.h> POSIX
System V 共享内存：shmget()/shmat()/shmdt()/shmctl()，以及 struct shmid_ds。允许多个进程共享同一块物理内存。
来源：POSIX.1-2017 §sys_shm.h | 实现：glibc
<sys/sem.h> POSIX
System V 信号量：semget()/semop()/semctl()，以及 struct semid_ds、union semun。注意区别于 POSIX 信号量（semaphore.h）。
来源：POSIX.1-2017 §sys_sem.h | 实现：glibc
<sys/uio.h> POSIX
分散/聚集 I/O：struct iovec，readv()/writev()（向量化读写），允许单次系统调用操作多个不连续的缓冲区，减少系统调用开销。
来源：POSIX.1-2017 §sys_uio.h | 实现：glibc
<sys/un.h> POSIX
Unix 域套接字地址：struct sockaddr_un，配合 sys/socket.h 使用 AF_UNIX 协议族，实现同主机进程间的高效 IPC。
来源：POSIX.1-2017 §sys_un.h | 实现：glibc
<sys/utsname.h> POSIX
系统信息：struct utsname（含 sysname/nodename/release/version/machine 字段），uname() 函数获取操作系统和内核版本信息。
来源：POSIX.1-2017 §sys_utsname.h | 实现：glibc
<sys/statvfs.h> POSIX
文件系统统计：struct statvfs（含块大小、总块数、可用块等），statvfs()/fstatvfs() 查询文件系统使用情况。
来源：POSIX.1-2017 §sys_statvfs.h | 实现：glibc
<sys/times.h> POSIX
进程时间：struct tms（含用户CPU时间、内核CPU时间），times() 函数获取进程及其子进程的 CPU 消耗时间。
来源：POSIX.1-2017 §sys_times.h | 实现：glibc

📡 网络相关头文件

<arpa/inet.h> POSIX
网络地址转换函数：htonl()/htons()/ntohl()/ntohs()（字节序转换），inet_ntop()/inet_pton()（IP地址格式转换），INET_ADDRSTRLEN/INET6_ADDRSTRLEN。
来源：POSIX.1-2017 §arpa_inet.h（源自 XNS Issue 5.2）| 实现：glibc
<netinet/in.h> POSIX
IPv4/IPv6 地址结构：struct sockaddr_in/struct sockaddr_in6，struct in_addr，in_port_t/in_addr_t 类型，INADDR_ANY/INADDR_LOOPBACK，IPPROTO_TCP/IPPROTO_UDP 等。
来源：POSIX.1-2017 §netinet_in.h | 实现：glibc
<netinet/tcp.h> POSIX
TCP 协议选项常量：TCP_NODELAY（禁用 Nagle 算法）、TCP_KEEPALIVE，以及 struct tcphdr（TCP 头部结构），配合 setsockopt() 使用。
来源：POSIX.1-2017 §netinet_tcp.h | 实现：glibc
<net/if.h> POSIX
网络接口：struct ifreq（接口请求结构）、IF_NAMESIZE（接口名最大长度），if_nametoindex()/if_indextoname()，用于获取网络接口信息。
来源：POSIX.1-2017 §net_if.h | 实现：glibc
<netdb.h> POSIX
网络数据库接口：getaddrinfo()/freeaddrinfo()（地址解析，取代过时的 gethostbyname()）、getnameinfo()、struct addrinfo、gai_strerror()。
来源：POSIX.1-2017 §netdb.h | 实现：glibc（NSS）

📄 其他常用 POSIX 头文件

<unistd.h> POSIX
POSIX API 核心：fork()/exec() 族（进程创建执行）、read()/write()/close()（文件描述符 I/O）、pipe()/dup()/dup2()、getcwd()/chdir()、getpid()/getppid()、sleep()/usleep() 等，是 POSIX 系统编程最常用头文件。
来源：POSIX.1-2017 §unistd.h | 实现：glibc
<fcntl.h> POSIX
文件控制：open()/creat()（文件打开），fcntl()（文件描述符控制），以及标志常量 O_RDONLY/O_WRONLY/O_CREAT/O_NONBLOCK/O_CLOEXEC 等。
来源：POSIX.1-2017 §fcntl.h | 实现：glibc（封装内核 fcntl）
<pthread.h> POSIX
POSIX 线程库：线程创建/管理（pthread_create()/pthread_join()/pthread_exit()），互斥锁（pthread_mutex_t），条件变量（pthread_cond_t），读写锁（pthread_rwlock_t）等。链接时需加 -lpthread。
来源：POSIX.1-2017 §pthread.h | 实现：glibc（NPTL）
<dirent.h> POSIX
目录操作：opendir()/readdir()/closedir()/seekdir()，以及 struct dirent（含 d_name、d_type 等）。用于遍历目录内容。
来源：POSIX.1-2017 §dirent.h | 实现：glibc
<poll.h> POSIX
I/O 多路复用（select() 的现代替代）：poll()/ppoll()，struct pollfd（含 fd/events/revents），POLLIN/POLLOUT/POLLERR 等事件常量。
来源：POSIX.1-2017 §poll.h | 实现：glibc
<termios.h> POSIX
终端 I/O 控制：struct termios，tcgetattr()/tcsetattr()（终端属性），cfsetispeed()/cfsetospeed()（波特率设置），B9600/B115200 等常量，用于串口编程。
来源：POSIX.1-2017 §termios.h | 实现：glibc
<semaphore.h> POSIX
POSIX 信号量（区别于 System V 信号量）：sem_t 类型，sem_init()/sem_destroy()（无名信号量），sem_open()/sem_close()（有名信号量），sem_wait()/sem_post()。
来源：POSIX.1-2017 §semaphore.h | 实现：glibc
<pwd.h> POSIX
用户账户数据库访问：struct passwd（含 pw_name/pw_uid/pw_gid/pw_dir/pw_shell），getpwuid()/getpwnam()/getpwent()，读取 /etc/passwd。
来源：POSIX.1-2017 §pwd.h | 实现：glibc（NSS）
<grp.h> POSIX
用户组数据库访问：struct group（含 gr_name/gr_gid/gr_mem），getgrgid()/getgrnam()/getgrent()，读取 /etc/group。
来源：POSIX.1-2017 §grp.h | 实现：glibc（NSS）
<glob.h> POSIX
文件名通配符展开（shell glob）：glob()/globfree()，glob_t 结构，支持 */?/[...] 等通配符，实现 shell 文件名扩展功能。
来源：POSIX.1-2017 §glob.h | 实现：glibc
<regex.h> POSIX
POSIX 正则表达式：regcomp()/regexec()/regfree()/regerror()，regex_t 类型，支持扩展正则（ERE）和基本正则（BRE）语法。
来源：POSIX.1-2017 §regex.h | 实现：glibc
<fnmatch.h> POSIX
文件名模式匹配：fnmatch() 函数，支持 */?/[...] 通配符，常用于 .gitignore 规则匹配等场景。
来源：POSIX.1-2017 §fnmatch.h | 实现：glibc
<iconv.h> POSIX
字符编码转换：iconv_open()/iconv()/iconv_close()，在 UTF-8、GBK、Latin-1 等字符集之间转换文本。
来源：POSIX.1-2017 §iconv.h | 实现：glibc
<aio.h> POSIX
异步 I/O 接口：aio_read()/aio_write()/aio_error()/aio_return()，struct aiocb。允许程序在 I/O 操作进行中继续执行其他任务。
来源：POSIX.1-2017 §aio.h | 实现：glibc
<mqueue.h> POSIX
POSIX 消息队列（区别于 System V 消息队列）：mq_open()/mq_close()/mq_send()/mq_receive()/mq_notify()，基于文件描述符，支持 select()/poll()。
来源：POSIX.1-2017 §mqueue.h | 实现：glibc
<sched.h> POSIX
进程调度控制：sched_setscheduler()/sched_getscheduler()，调度策略 SCHED_FIFO/SCHED_RR/SCHED_OTHER，sched_yield()（主动让出CPU），cpu_set_t（CPU 亲和性）。
来源：POSIX.1-2017 §sched.h | 实现：glibc
<spawn.h> POSIX
进程派生（fork/exec 的高层封装）：posix_spawn()/posix_spawnp()，用于在资源受限（如嵌入式）环境中创建新进程，避免 fork 的内存开销。
来源：POSIX.1-2017 §spawn.h | 实现：glibc
<dlfcn.h> POSIX
动态链接接口：dlopen()/dlsym()/dlclose()/dlerror()，用于在运行时加载共享库（.so），实现插件机制。链接时需加 -ldl。
来源：POSIX.1-2017 §dlfcn.h | 实现：glibc（ld-linux.so）
<ftw.h> POSIX
文件树遍历：ftw()/nftw()（递归遍历目录树），对每个文件/目录调用用户提供的回调函数。nftw() 是现代推荐接口（支持 FTW_PHYS 等标志）。
来源：POSIX.1-2017 §ftw.h | 实现：glibc
<utmpx.h> POSIX
用户登录记录访问：struct utmpx，getutxent()/getutxid()/getutxline()/pututxline()，访问系统登录/注销记录（通常存储在 /var/run/utmpx）。
来源：POSIX.1-2017 §utmpx.h | 实现：glibc

📋 POSIX 头文件小结

  • POSIX.1-2017 规定了超过 80 个头文件，覆盖文件系统、进程、网络、IPC、线程、终端等操作系统接口
  • 实现者是 glibc：POSIX 定义规范，glibc 负责在 Linux 上实现。每个 POSIX 接口最终通过系统调用与内核交互
  • 与内核 UAPI 的关系：glibc 的 POSIX 实现内部会包含内核 UAPI 头文件（如 linux/socket.h），但对用户程序呈现标准化的 POSIX 接口
  • 第一性原理：POSIX 的核心是操作系统抽象——将 Unix-like 操作系统的各种服务（文件、进程、网络）标准化，使应用程序可以在不同系统间移植
  • 交叉编译注意：部分 POSIX 接口（如 pthread.h）需要额外链接参数（-lpthread），且嵌入式系统可能仅支持部分 POSIX 接口（如 musl libc 的精简实现）

🟢Linux 内核提供的 UAPI 头文件

权威来源：Linux kernel 官方文档 — Exporting kernel headers for use by userspace — kernel.org

Linux From Scratch 安装说明： Linux API Headers — LFS Book

内核 UAPI（User API）头文件来自 Linux 内核源码的 include/uapi/ 目录，通过 make headers_install 命令导出到 usr/include。 这些头文件描述了用户空间程序直接使用内核服务时所需的数据结构、常量和 ioctl 命令。

关键机制说明：内核 UAPI 头文件经过"消毒"（sanitization）处理——删除了内核内部使用的宏、类型和结构，只保留用户空间 ABI 接口。来自 Linux kernel COPYING 文件的注记明确指出："使用正常系统调用访问内核服务的用户程序不受 GPL 限制，这仅被视为内核的正常使用。"

📁 linux/ 子目录（内核核心 UAPI）

<linux/types.h> 内核 UAPI
内核 ABI 基础类型：__u8/__u16/__u32/__u64，__s8/__s16 等固定宽度整数类型，以及 __be32/__le32（大端/小端标注类型）。这些类型是所有其他内核 UAPI 头文件的基础。
来源：kernel include/uapi/linux/types.h | 用途：内核数据结构类型基础
<linux/ioctl.h> 内核 UAPI
ioctl 命令编码宏：_IO()/_IOR()/_IOW()/_IOWR()，用于构造设备驱动 ioctl 命令号（编码魔数、命令编号、数据方向和大小）。是所有设备驱动程序用户态接口的基础。
来源：kernel include/uapi/linux/ioctl.h | 用途：设备控制接口
<linux/fs.h> 内核 UAPI
文件系统 UAPI：文件系统相关的 ioctl 命令（FIOCLEX/FIONREAD/FS_IOC_GETFLAGS 等），以及 struct file_clone_range 等文件操作结构体。
来源：kernel include/uapi/linux/fs.h | 用途：文件系统操作
<linux/stat.h> 内核 UAPI
文件状态标志：S_IFMT/S_IFREG/S_IFDIR/S_IFLNK/S_IFBLK/S_IFCHR/S_IFIFO/S_IFSOCK，文件权限位，以及 struct statx（扩展文件状态，Linux 4.11+）。
来源：kernel include/uapi/linux/stat.h | 用途：文件类型和权限
<linux/fcntl.h> 内核 UAPI
文件控制常量：O_RDONLY/O_WRONLY/O_CREAT/O_NONBLOCK/O_CLOEXEC 等 open() 标志，以及 F_DUPFD/F_GETFL/F_SETFL 等 fcntl() 命令。glibc 的 fcntl.h 间接引用此文件。
来源：kernel include/uapi/linux/fcntl.h | 用途：文件描述符控制
<linux/socket.h> 内核 UAPI
套接字地址族常量：AF_INET/AF_INET6/AF_UNIX/AF_NETLINK/AF_PACKET 等，套接字类型 SOCK_STREAM/SOCK_DGRAM/SOCK_RAW，以及 MSG_* 标志。
来源：kernel include/uapi/linux/socket.h | 用途：网络套接字接口
<linux/if.h> 内核 UAPI
网络接口标志和结构：IFF_UP/IFF_BROADCAST/IFF_LOOPBACK/IFF_PROMISC 等接口标志，struct ifreq（接口请求，用于 SIOCGIFADDR 等 ioctl），IFNAMSIZ。
来源：kernel include/uapi/linux/if.h | 用途：网络接口管理
<linux/errno.h> 内核 UAPI
Linux 特有的错误码扩展，补充 POSIX/C 标准的 errno.h 中未定义的 Linux 特有错误号（如 ENOKEY/EKEYEXPIRED/EMEDIUMTYPE/EREMOTEIO 等）。
来源：kernel include/uapi/linux/errno.h | 用途：错误号扩展
<linux/netlink.h> 内核 UAPI
Netlink 套接字协议接口：struct nlmsghdr/struct nlattr（Netlink 消息格式），NETLINK_ROUTE/NETLINK_NETFILTER/NETLINK_GENERIC 等协议族，用于内核与用户空间的异步通信。
来源：kernel include/uapi/linux/netlink.h | 用途：内核通信机制
<linux/rtnetlink.h> 内核 UAPI
路由/网络配置 Netlink 接口：RTM_NEWROUTE/RTM_DELROUTE/RTM_GETROUTE 等消息类型，struct rtmsg/struct ifinfomsg，用于 ip route / ip link 等工具的底层实现。
来源：kernel include/uapi/linux/rtnetlink.h | 用途：路由表管理
<linux/if_ether.h> 内核 UAPI
以太网协议常量：ETH_P_IP/ETH_P_IPV6/ETH_P_ARP 等协议类型，ETH_ALEN（MAC地址长度=6），ETH_HLEN（以太网头部长度=14），struct ethhdr（以太网帧头）。
来源：kernel include/uapi/linux/if_ether.h | 用途：链路层协议
<linux/if_packet.h> 内核 UAPI
原始数据包套接字接口（AF_PACKET）：struct sockaddr_ll（链路层地址），struct tpacket_req/struct tpacket2_hdr（零拷贝包捕获 mmap 接口），用于 tcpdump/Wireshark 底层。
来源：kernel include/uapi/linux/if_packet.h | 用途：原始数据包捕获
<linux/input.h> 内核 UAPI
输入子系统（evdev）接口：struct input_event（输入事件：time/type/code/value），EV_KEY/EV_ABS/EV_REL 事件类型，KEY_* 键码定义，用于读取 /dev/input/eventN 设备。
来源：kernel include/uapi/linux/input.h | 用途：输入设备事件
<linux/fb.h> 内核 UAPI
帧缓冲设备（framebuffer）接口：struct fb_var_screeninfo（可变参数：分辨率/色深/时序）、struct fb_fix_screeninfo（固定参数：内存基址/行长度），以及 FBIOGET_VSCREENINFO 等 ioctl。
来源：kernel include/uapi/linux/fb.h | 用途：显示缓冲区控制
<linux/i2c-dev.h> 内核 UAPI
I2C 设备用户空间接口：I2C_SLAVE/I2C_RDWR/I2C_SMBUS 等 ioctl 命令，struct i2c_rdwr_ioctl_data，用于从用户空间直接操作 /dev/i2c-N 设备，嵌入式开发常用。
来源：kernel include/uapi/linux/i2c-dev.h | 用途：I2C 总线访问
<linux/spi/spidev.h> 内核 UAPI
SPI 设备用户空间接口：SPI_IOC_RD_MODE/SPI_IOC_WR_MODE/SPI_IOC_MESSAGE() 等 ioctl，struct spi_ioc_transfer（全双工 SPI 传输描述符），用于访问 /dev/spidevX.Y 设备。
来源：kernel include/uapi/linux/spi/spidev.h | 用途：SPI 总线访问
<linux/gpio.h> 内核 UAPI
GPIO 字符设备接口（Linux 4.8+ 新 API）：struct gpio_v2_line_request/struct gpio_v2_line_event，GPIO_V2_GET_LINEINFO_IOCTL 等，通过 /dev/gpiochipN 访问 GPIO，替代旧的 sysfs 接口。
来源：kernel include/uapi/linux/gpio.h | 用途：GPIO 控制
<linux/watchdog.h> 内核 UAPI
硬件看门狗接口：WDIOC_GETSUPPORT/WDIOC_SETTIMEOUT/WDIOC_KEEPALIVE 等 ioctl，struct watchdog_info，用于从用户空间控制 /dev/watchdog，嵌入式可靠性设计常用。
来源：kernel include/uapi/linux/watchdog.h | 用途：看门狗管理
<linux/can.h> 内核 UAPI
CAN 总线套接字接口（SocketCAN）：struct can_frame/struct canfd_frame（CAN 帧格式），canid_t 类型，CAN_RAW/CAN_BCM/CAN_ISOTP 等协议，汽车/工控嵌入式常用。
来源：kernel include/uapi/linux/can.h | 用途：CAN 总线通信
<linux/v4l2-controls.h> / <linux/videodev2.h> 内核 UAPI
Video4Linux2（V4L2）视频设备接口：摄像头/视频采集 ioctl（VIDIOC_QUERYCAP/VIDIOC_QBUF/VIDIOC_DQBUF），缓冲区管理，像素格式 V4L2_PIX_FMT_*，嵌入式视觉应用核心接口。
来源：kernel include/uapi/linux/videodev2.h | 用途：视频设备控制
<linux/usbdevice_fs.h> 内核 UAPI
USB 设备文件系统接口：struct usbdevfs_urb（USB 请求块），USBDEVFS_SUBMITURB/USBDEVFS_REAPURB 等 ioctl，用于 libusb 底层实现和自定义 USB 驱动。
来源：kernel include/uapi/linux/usbdevice_fs.h | 用途：USB 原始访问
<linux/perf_event.h> 内核 UAPI
性能监控计数器（PMU）接口：struct perf_event_attr（性能事件配置），perf_event_open() 系统调用，PERF_TYPE_HARDWARE/PERF_COUNT_HW_CPU_CYCLES 等常量，用于性能分析工具（perf/BPF）。
来源：kernel include/uapi/linux/perf_event.h | 用途：性能分析
<linux/bpf.h> 内核 UAPI
eBPF 程序接口：bpf() 系统调用命令（BPF_PROG_LOAD/BPF_MAP_CREATE 等），struct bpf_insn（BPF 字节码指令），映射类型（BPF_MAP_TYPE_HASH/BPF_MAP_TYPE_ARRAY）等，是现代内核可编程性的基础。
来源：kernel include/uapi/linux/bpf.h | 用途：eBPF 程序编写

📁 asm/ 子目录（架构相关 UAPI）

说明：此目录内容因目标架构不同而不同（如 ARM、AArch64、x86、RISC-V 等）。 交叉编译时，此目录对应目标架构的平台特定接口。根据 kernel.org 文档， ARCH 参数决定安装哪个架构的 asm/ 头文件。

<asm/unistd.h> 内核 UAPI
系统调用号定义：架构特定的系统调用编号（如 AArch64 的 __NR_read=63, __NR_write=64），这些编号用于通过 syscall() 直接调用内核，每种架构不同。
来源：arch/*/include/uapi/asm/unistd.h | 用途：系统调用号
<asm/signal.h> 内核 UAPI
架构特定的信号定义：信号编号（SIGKILL=9, SIGTERM=15 等，部分信号号在不同架构不同），struct sigaction，sigset_t 的架构特定实现。
来源：arch/*/include/uapi/asm/signal.h | 用途：信号处理
<asm/mman.h> 内核 UAPI
架构特定的内存映射常量：PROT_READ/PROT_WRITE/PROT_EXEC，MAP_SHARED/MAP_PRIVATE/MAP_ANONYMOUS/MAP_FIXED 等，部分常量值因架构而异。
来源：arch/*/include/uapi/asm/mman.h | 用途：内存映射标志
<asm/ptrace.h> 内核 UAPI
进程跟踪（ptrace）架构接口：目标架构的寄存器集结构（如 AArch64 的 struct user_pt_regs），ptrace 请求码（PTRACE_GETREGS/PTRACE_SETREGS），用于调试器（GDB）。
来源：arch/*/include/uapi/asm/ptrace.h | 用途：进程调试

📁 asm-generic/ 子目录（通用架构 UAPI）

<asm-generic/fcntl.h> 内核 UAPI
通用文件控制标志：架构无关的 O_*/F_* 常量定义，供架构特定的 asm/fcntl.h 引用。
来源：include/uapi/asm-generic/fcntl.h | 用途：文件控制通用定义
<asm-generic/ioctls.h> 内核 UAPI
通用 tty ioctl 命令：TCGETS/TCSETS/TIOCGWINSZ/TIOCSWINSZ（终端窗口大小）等，适用于大多数架构的终端设备控制。
来源：include/uapi/asm-generic/ioctls.h | 用途：终端 ioctl 命令
<asm-generic/mman-common.h> 内核 UAPI
通用内存映射常量：大多数架构共用的 PROT_*/MAP_*/MADV_*（内存使用建议，如 MADV_DONTNEED/MADV_SEQUENTIAL）等宏定义。
来源：include/uapi/asm-generic/mman-common.h | 用途：内存映射通用常量

视频/帧缓冲扩展接口：edid.h（EDID 显示器信息格式），uvesafb.h（VESA 帧缓冲接口），用于显示子系统的高级控制。

来源：kernel include/uapi/misc/ | 用途：杂项设备

📋 内核 UAPI 头文件小结

  • 来源明确：内核 UAPI 头文件来自 Linux 内核源码 include/uapi/ 目录，通过 make headers_install ARCH=<arch> INSTALL_HDR_PATH=/usr 安装
  • 安装位置：/usr/include/linux/、/usr/include/asm/、/usr/include/asm-generic/、/usr/include/drm/ 等子目录
  • 向后兼容性：内核头文件向后兼容但不向前兼容——旧头文件新内核可运行，新头文件旧内核可能不行。交叉编译时头文件版本应与目标内核版本匹配
  • License：内核 COPYING 文件明确声明，通过正常系统调用使用内核服务的用户程序不受 GPL 约束，UAPI 头文件可自由包含于任意许可证的用户空间代码
  • 第一性原理：内核 UAPI 的本质是内核-用户空间 ABI的文档化——它定义了通过系统调用和 ioctl 与内核交互的所有数据格式，是操作系统接口契约的直接体现
  • glibc 的角色：glibc 实现的 POSIX 接口（如 socket()）内部引用内核 UAPI（如 linux/socket.h），为用户提供标准化封装层

🟣glibc 提供的扩展头文件

权威来源：GNU C Library（glibc）文档： The GNU C Library Reference Manual

这些头文件不属于 C 标准或 POSIX 规范，而是 GNU C Library 提供的 Linux/GNU 平台特有扩展。 在 Fedora glibc 包的文件列表（glibc-headers-x86）中可以完整看到这些文件。 使用它们可能降低程序的可移植性，但往往能提供更丰富的系统功能。

📄 常见 glibc 扩展头文件

<features.h> glibc
glibc 功能特性开关控制中心：根据 _GNU_SOURCE/_POSIX_C_SOURCE/_XOPEN_SOURCE/__USE_MISC 等宏决定暴露哪些接口。几乎所有 glibc 头文件都首先包含此文件，是 glibc 可配置性的核心机制。
来源：glibc | 用途：特性开关控制
<bits/*.h> glibc
glibc 内部实现细节头文件目录，包含 bits/types.h（底层类型）、bits/socket.h（套接字底层）、bits/stat.h（stat 结构底层）等。理论上不应被用户代码直接包含，通过公共头文件间接使用。
来源：glibc | 用途：内部实现（不应直接使用）
<elf.h> glibc
ELF（可执行与可链接格式）数据结构：Elf32_Ehdr/Elf64_Ehdr（ELF 文件头），Elf64_Phdr（程序头/段），Elf64_Shdr（节头），Elf64_Sym（符号表），用于开发链接器、调试器、二进制分析工具。
来源：glibc | 用途：ELF 二进制格式分析
<endian.h> glibc
字节序检测宏：__BYTE_ORDER/__BIG_ENDIAN/__LITTLE_ENDIAN，以及字节序转换函数（be32toh()/htobe32()/le64toh()/htole64() 等），简化跨平台字节序处理。
来源：glibc | 用途：字节序处理
<byteswap.h> glibc
字节交换宏：bswap_16()/bswap_32()/bswap_64()，无条件翻转整数字节序，通常由编译器优化为单条机器指令（如 x86 的 bswap，ARM 的 rev）。
来源：glibc | 用途：字节序翻转
<error.h> glibc
GNU 风格错误报告：error()/error_at_line()，在打印错误消息后可选择性地终止程序，比手动格式化 strerror(errno) 更简洁，是 GNU coreutils 风格错误报告的标准方式。
来源：glibc | 用途：错误消息输出
<err.h> glibc
BSD 风格错误报告：err()/errx()/warn()/warnx()，在程序名+错误描述格式下报告错误并可退出，是 BSD 传统和 Linux 工具常用的错误处理模式。
来源：glibc（BSD 兼容）| 用途：BSD 风格错误处理
<syslog.h> glibc
系统日志接口：openlog()/syslog()/closelog()，日志级别（LOG_ERR/LOG_WARNING/LOG_INFO/LOG_DEBUG），设施（LOG_DAEMON/LOG_USER），发送消息到 syslogd/journald。
来源：POSIX + glibc 实现 | 用途：系统日志
<alloca.h> glibc
栈上动态内存分配：alloca()，在栈帧上分配内存（函数返回时自动释放，无需 free()），通常由编译器内联为调整栈指针的单条指令。注意：不可用于大型或递归场景。
来源：glibc（通常由 GCC built-in 实现）| 用途：栈内存分配
<execinfo.h> glibc
栈回溯（backtrace）：backtrace()/backtrace_symbols()/backtrace_symbols_fd()，用于在程序崩溃时（如信号处理器中）打印调用栈，是崩溃诊断工具的基础。
来源：glibc | 用途：调试/崩溃分析
<getopt.h> glibc
命令行参数解析：标准 getopt()（短选项），GNU 扩展 getopt_long()/getopt_long_only()（长选项，如 --help），struct option，是 GNU 命令行工具的标准解析方式。
来源：glibc（GNU 扩展）| 用途：命令行解析
<argp.h> glibc
GNU argp 参数解析框架（比 getopt 更高级）：argp_parse()，struct argp/struct argp_option，自动生成标准化的帮助文本（--help/--usage），用于开发具有完整帮助功能的 GNU 风格命令行工具。
来源：glibc（GNU 特有）| 用途：高级命令行解析
<gnu/libc-version.h> glibc
glibc 版本查询：gnu_get_libc_version()（返回版本字符串，如 "2.38"），gnu_get_libc_release()（返回发行类型），用于在运行时检测 glibc 版本，实现版本兼容性判断。
来源：glibc | 用途：运行时版本检测
<fts.h> glibc
文件层次结构遍历（比 ftw.h 更强大的 BSD API）：fts_open()/fts_read()/fts_close()，FTS/FTSENT 结构，支持物理/逻辑遍历、循环检测、文件排序等。
来源：glibc（BSD 来源）| 用途：目录树遍历

📋 glibc 扩展头文件小结

  • 非标准但实用：这些头文件不属于 ISO C 或 POSIX 规范，但在 GNU/Linux 生态中被广泛使用（如 elf.h/endian.h/execinfo.h）
  • 可移植性代价：使用 glibc 扩展会降低代码在非 Linux 平台（如 FreeBSD/macOS/Windows）或非 glibc 实现（如 musl）上的可移植性
  • features.h 控制：通过定义 _GNU_SOURCE 宏可以同时开启 GNU 扩展、POSIX 和 BSD 扩展，这是大多数 Linux 程序的默认做法
  • 第一性原理：glibc 扩展头文件体现了实用主义——当标准不足以满足系统编程需求时，提供平台特有的高效解决方案。这是可移植性与功能性之间永恒的权衡

🔬 第一性原理：为什么 usr/include 里有这么多头文件？

根本问题：C 语言仅定义了语言语法和一小组标准库，远不足以支撑现代系统程序开发。 操作系统需要向程序员暴露文件系统、网络、进程、硬件等抽象，这些接口必须有机制加以文档化和规范化。 头文件（.h 文件）就是这一机制——它是 C 语言的接口契约。

层次 核心问题 解决方案 头文件体现
语言层 程序需要基本输入/输出、内存、字符串操作 ISO C 标准库 stdio.h, stdlib.h, string.h...
操作系统层 程序需要操作文件、进程、网络等 OS 服务 POSIX 标准接口 unistd.h, sys/socket.h, pthread.h...
内核 ABI 层 程序需要直接访问 Linux 特有的内核功能 内核 UAPI 导出 linux/bpf.h, linux/input.h, drm/*.h...
实现优化层 程序需要平台特有的高效工具（ELF解析、调试等） glibc GNU 扩展 elf.h, endian.h, execinfo.h...

🎯 全章第一性原理总结

  • 分层的必然性：计算机系统天然分层——硬件→内核→系统库→应用。每一层都需要向上层暴露接口，头文件是这些接口在 C 语言中的表达形式。
  • 标准化的价值：ISO C 和 POSIX 的价值在于解耦——让应用程序不依赖特定的内核版本或 libc 实现，提高可移植性。这是"面向接口编程"在系统层面的体现。
  • 内核 UAPI 的独特地位：内核 UAPI 是唯一一类直接由操作系统内核作者维护的头文件，代表了硬件抽象和内核服务的规范化文档。它向后兼容但不向前兼容，体现了内核维护稳定 ABI 的承诺。
  • 交叉编译的意义：usr/include 在交叉编译中属于目标 sysroot，必须完全代表目标系统（不同架构、不同内核版本），不能使用宿主机的头文件，这正是为什么需要理解每个头文件的来源。
  • glibc 的桥梁作用：glibc 是连接 C/POSIX 标准与 Linux 内核 UAPI 的桥梁，将标准接口翻译为内核系统调用，同时提供超越标准的 GNU 扩展，构成了 Linux 用户空间程序开发的完整基础。
