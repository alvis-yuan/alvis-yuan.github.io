第一章

君正 X2600 平台概述

1.1 公司背景与芯片系列

君正半导体（Ingenic Semiconductor，全称：北京君正集成电路股份有限公司）成立于 2005 年，是国内知名的 无晶圆厂（Fabless） 半导体公司，专注于 MIPS 架构 CPU 和 SoC 设计。2009 年获得 MIPS 架构授权，此后围绕 XBurst 微架构持续演进，广泛应用于平板、可穿戴、IoT、安防摄像头等领域。

第二章

嵌入式启动链的由来

2.1 为什么不能直接运行 U-Boot？

嵌入式 SoC 上电后，CPU 只能从片内固化的 Boot ROM 开始执行代码。Boot ROM 通常极小（典型仅数 KB 至十余 KB），且仅提供最基本的硬件检测和存储介质访问能力。

完整的 U-Boot（俗称 U-Boot Proper）二进制体积通常在 300 KB ~ 1 MB 甚至更大。要运行 U-Boot Proper，必须先有可用的 RAM（DDR）——而 DDR 初始化本身需要运行代码。这就形成了一个 "先有鸡还是先有蛋" 的问题：

核心矛盾
DDR 初始化需要程序运行 → 程序运行需要内存 → 内存（DDR）尚未初始化

解决方案是分阶段启动：在 DDR 可用之前，先在少量的 片上 SRAM（TCSM） 中运行一段极小的代码，完成 DDR 初始化，再从外置 Flash 将完整 U-Boot 复制到 DDR 并跳转执行。这段极小的过渡代码，就是 SPL。

2.2 MIPS 平台的启动特殊性

与 ARM 平台相比，MIPS 平台的启动有以下特点：

  • MIPS CPU 上电后从固定的 Reset Exception Vector（通常为 0xBFC00000） 开始执行，该地址映射到 Boot ROM 区域。
  • MIPS 采用 KSEG0/KSEG1 地址空间划分，Boot ROM 通常位于 KSEG1（uncached）区域，保证上电后不依赖缓存即可正确取指。
  • 君正 XBurst/XBurst2 的 Boot ROM 负责检测启动介质（SFC NOR、SFC NAND、MSC 等），然后将外部 Flash 中的 SPL 镜像（通常头部包含君正专有的 NandBoot/SFCBoot Header）加载到 TCSM，并跳转执行。

📌 本章总结

多级启动链的根本原因是 资源约束：Boot ROM 空间极小，DDR 上电后不可用，而运行完整 U-Boot 需要大量内存。SPL 作为"过渡层"，在最受限的环境中完成最小必要初始化，使系统具备运行完整软件的条件。

从第一性原理看：任何计算系统在启动初期都面临"引导问题"——没有足够资源却需要初始化资源本身。解法永远是分层：用更简单、更小的程序去准备更复杂程序运行所需的环境。

第三章

SPL 是什么

3.1 官方定义

U-Boot SPL（Secondary Program Loader，二级程序加载器）是 U-Boot 官方通用 SPL 框架中的核心概念。

来自 U-Boot 官方文档 doc/README.SPL
🔗 doc/README.SPL（Google Git 镜像）

"To unify all existing implementations for a secondary program loader (SPL) and to allow simply adding of new implementations this generic SPL framework has been created. With this framework almost all source files for a board can be reused. No code duplication or symlinking is necessary anymore."

（为了统一所有现有的二级程序加载器实现，并允许简单地添加新实现，创建了这个通用 SPL 框架。使用该框架，板级的几乎所有源文件都可以复用，不再需要代码重复或符号链接。）
来自 U-Boot 官方文档 usage/spl_boot.html
🔗 Booting from TPL/SPL（U-Boot 官方）

SPL — Secondary Program Loader："Sets up SDRAM and loads U-Boot proper. It may also load other firmware components."
（初始化 SDRAM，加载正式 U-Boot。也可以加载其他固件组件。）

3.2 SPL 的输出产物

构建 SPL 后会生成以下文件（来自官方 doc/README.SPL）：

文件说明
u-boot-splELF 格式 SPL（含调试符号）
u-boot-spl.bin裸二进制 SPL，由 BootROM 或工具写入 Flash

3.3 SPL 与完整 U-Boot 的区别

SPL 和 U-Boot Proper 共享同一份源代码树，通过编译宏 CONFIG_SPL_BUILD 区分：

# 来自 U-Boot 官方 doc/README.SPL
ifeq ($(CONFIG_SPL_BUILD),y)
obj-y += board_spl.o      # SPL 专用代码
else
obj-y += board.o          # U-Boot Proper 代码
endif

# 在 C 源码中同样适用
#ifdef CONFIG_SPL_BUILD
    foo();   /* 只在 SPL 构建时编译 */
#endif
对比维度U-Boot SPLU-Boot Proper
体积极小（受 TCSM 限制，通常 <64KB）较大（300KB~1MB+）
运行位置片上 SRAM / TCSM外置 DDR
DDR 状态初始化前 → 初始化后DDR 已初始化
命令行无（通常）有完整 Shell
驱动支持最小子集完整驱动框架
主要任务PLL/DDR 初始化 + 加载 U-Boot加载操作系统内核
📌 本章总结

SPL 是 U-Boot 官方提供的通用二级程序加载器框架，其核心设计原则是 "代码最大复用，体积极致精简"。它与 U-Boot Proper 共源，通过 CONFIG_SPL_BUILD 宏实现差异化编译，能够在片上 SRAM 等极度受限环境中运行，完成最关键的硬件初始化。

SPL 主要完成的工作

4.2 各步骤详解

① 汇编入口 _start

SPL 的执行入口是 _start（MIPS 平台对应 arch/mips/cpu/start.S），与 U-Boot Proper 共享同一入口文件，通过 CONFIG_SPL_BUILD 宏进行差异化处理。主要工作：

  • 关闭全局中断（CP0 Status 寄存器）
  • 设置异常向量基地址（CP0 EBase）
  • 关闭 MMU、TLB 初始化（MIPS 无 MMU 关闭操作，但需清 TLB）
  • 初始化 CPU 寄存器到安全初始值

② lowlevel_init() — 最早期平台初始化

在具备 C 运行环境之前执行（无栈），完成不得不在汇编阶段处理的硬件配置：

  • 关闭看门狗定时器（防止复位）
  • GPIO 方向和复用功能基本配置
  • CPM（Clock/Power Management）寄存器的最低要求配置
来源
U-Boot 官方 README（Board init flow）：
🔗 xilinx-uboot/README（Board Initialisation Flow）

"lowlevel_init(): purpose: essential init to permit execution to reach board_init_f(). There is no stack… must only do the bare minimum to allow execution to continue."

④ DDR 初始化（最核心任务）

DDR 初始化是 SPL 最复杂也最重要的任务，需要按严格时序配置 DDR PHY 和 DDRC：

  • DDR PHY 初始化：配置物理层 PLL、DQ 宽度、CAS/CWL 延迟、ZQ 阻抗校准
  • DDR Training（训练）：写/读 DQ 延迟 Training，确保信号完整性
  • DDRC 配置：填写 TIMING1~TIMING6、MMAP、REMAP 等时序寄存器
  • 初始化序列：发送 MRS（Mode Register Set）命令，激活 DDR 颗粒

⑤ C 运行环境建立（board_init_f）

SPL 在完成最早期汇编初始化后，进入 C 语言环境。U-Boot 官方规定了以下初始化函数调用链：

来源：U-Boot Board Initialization Flow
🔗 xilinx-uboot/README

"board_init_f(): purpose: set up the machine ready for running board_init_r(). SDRAM and serial UART. Global data is available. Stack is in SRAM." 
# arch/mips/lib/crt0.S (或对应 MIPS SPL crt)
ldr sp, =(CONFIG_SPL_STACK)     # (1) 设置初始堆栈（在 TCSM/SRAM）
bl  board_init_f_alloc_reserve   # (2) 分配 GD + early malloc 区域
bl  board_init_f_init_reserve    # (3) 初始化 GD
bl  board_init_f                 # (4) 板级初始化（DDR 等）
bl  spl_relocate_stack_gd        # (5) DDR 就绪后迁移 stack/GD 到 DDR
bl  board_init_r                 # (6) 加载 U-Boot Proper 并跳转

📌 本章总结

U-Boot SPL 在 X2600 平台依次完成六项工作：CPU/中断初始化 → 最早期平台初始化 → PLL/时钟配置 → DDR 初始化（最核心）→ C 环境建立 → 加载并跳转 U-Boot Proper。

从第一性原理看：SPL 的所有工作都服务于同一个目标——让系统从"无法运行复杂程序"的初始状态，过渡到"具备完整 RAM 的可运行状态"。DDR 初始化是这一过程的关键里程碑。

5.2 Boot ROM → SPL 的交接机制

君正 BootROM 在加载 SPL 时，会解析 SPI Flash 开头的 君正专有启动头（Ingenic Boot Header），从中获取 SPL 的加载地址（TCSM 地址）、长度、入口地址等信息，完成搬运后跳转到 SPL 入口。

  • NOR Flash 启动：BootROM 通过 SFC 以 XIP（Execute-In-Place）或 DMA 读方式将 SPL 复制到 TCSM
  • NAND Flash 启动：BootROM 读取 NAND 首若干页（需处理坏块），解析头部后将 SPL 加载到 TCSM
  • MSC（eMMC/SD）启动：BootROM 初始化 MMC 控制器，读取固定扇区偏移处的 SPL

📌 本章总结

7.3 X2600 安全启动与 EFUSE

X2600 支持安全特性，影响 SPL 的烧录和启动流程：

  • EFUSE（一次性可编程）：存储芯片密钥、启动模式、加密配置等，SPL 在安全启动模式下需通过签名验证
  • 写保护寄存器：Ingenic Cloner 工具专门为 X2600 添加了"烧录清写保护寄存器"功能（🔗 changelog）
  • 安全烧录（Safe Burn）：Cloner 工具支持 X2600 安全烧录，包括 2ms 上下电延迟等时序保护
📌 本章总结

X2600 相比老款 XBurst1 SoC，DDR PHY 更复杂（支持更高频率和更精确的 Training），SPL 体积上限更大，并增加了 EFUSE 安全启动支持。这些特性使得 X2600 的 SPL 在核心流程不变的前提下，初始化代码量和复杂度显著提升。

Flash 布局上，SPL 始终位于 Flash 起始地址，以确保 BootROM 能够不加任何预判地直接读取并加载它。

第八章

第一性原理与全文总结

8.1 从第一性原理理解 SPL

第一性原理（First Principles）

物理约束：SRAM 速度快但面积贵，容量极小（KB 级）；DDR 容量大但需主动初始化才可用；非易失性存储（Flash）上电后可读取但不能直接执行复杂程序。

逻辑推导：任何程序的运行都需要内存。在 DDR 可用之前，只能使用片上极小的 SRAM 运行程序。这段程序必须足够小（适应 SRAM），但又必须完成足够多的工作（初始化 DDR）。

最优解：设计一个"刚好够用"的最小引导程序（SPL），在 SRAM 中运行，完成 DDR 初始化，然后将控制权交给更完整的程序。这是唯一可行的解法。

8.2 SPL 存在的必然性

SPL 不是君正或 U-Boot 的特有设计，而是 嵌入式 SoC 启动的普遍规律：

  • ARM 平台：TF-A BL2（DDR 初始化）→ BL3 → U-Boot
  • x86 平台：BIOS/UEFI 内置 Memory Reference Code（MRC）完成相同工作
  • MIPS 君正平台：Boot ROM → SPL（DDR init） → U-Boot Proper → Linux

只要存在"外置动态内存需要初始化"这一硬件约束，就必然存在一个类似 SPL 的启动阶段。

8.4 核心要点回顾

问题答案
SPL 是什么？Secondary Program Loader，U-Boot 官方通用二级程序加载器框架，源码与 U-Boot Proper 共享，通过 CONFIG_SPL_BUILD 区分编译。
为什么需要 SPL？Boot ROM 容量极小，完整 U-Boot 无法直接加载；DDR 上电不可用，需要中间层先完成初始化。
SPL 运行在哪里？X2600 片上 TCSM/SRAM，不依赖外置 DDR。
SPL 最核心的工作？DDR（SDRAM）初始化，包括 DDR PHY 配置、时序寄存器设置、Training 校准。
SPL 执行结束后？将 U-Boot Proper 从 SFC Flash 加载到 DDR，跳转执行（image entry point: 0x80100000）。
X2600 SPL 的特殊性？XBurst2 更复杂的 DDR PHY Training；支持 EFUSE 安全启动；CPU/DDR 频率 800/400 MHz。
📌 全文总结（第一性原理视角）

任何计算系统启动都面临一个根本悖论：需要程序来初始化运行程序所需的资源。解决这一悖论的唯一方法是 分层启动。

U-Boot SPL 是这一分层启动策略在嵌入式 Linux 领域的标准化实现。对于君正 X2600 这样的 MIPS SoC，SPL 在片上 TCSM 这块极小的"暂存地"中，以最精简的代码完成了最关键的工作——让外置 DDR 颗粒"醒来"并可用，然后从 SPI Flash 搬来完整的 U-Boot Proper，实现了从"一穷二白"到"百废俱兴"的系统启动破局。

SPL 的存在，本质上是硬件约束（小 SRAM、外置 DDR、非易失存储）与软件目标（运行完整 Linux 系统）之间矛盾的最优解。
