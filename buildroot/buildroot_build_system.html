概述与简介

Buildroot 是一款简化并自动化嵌入式 Linux 系统完整构建过程的工具，采用交叉编译方式工作。 它能够生成交叉编译工具链、根文件系统、Linux 内核镜像及目标平台的 Bootloader。 以上功能可按需自由组合使用。

Buildroot 主要面向嵌入式开发人员。嵌入式系统通常使用非 x86 架构处理器，如 PowerPC、MIPS、ARM 等。 Buildroot 支持众多处理器及其变体，并为多种现成开发板提供了默认配置。

🔧 工具链生成

自动构建交叉编译工具链，支持 uClibc-ng、glibc、musl

📦 软件包管理

超过 3000 个开源软件包，Kconfig 界面选择

🐧 内核构建

支持主线 Linux 内核及自定义内核配置

💾 文件系统

支持 ext2/3/4、squashfs、jffs2、ubifs 等多种格式

官方原文来源：Chapter 1. About Buildroot

系统要求

Buildroot 设计在 Linux 系统上运行。尽管 Buildroot 自身会构建大多数所需的宿主工具包，但某些标准 Linux 工具仍须预先安装在宿主系统上。

2.1 必须安装的软件包

在 Debian/Ubuntu 上快速安装

sudo apt-get install \
  which sed make binutils build-essential \
  gcc g++ bash patch gzip bzip2 perl tar \
  cpio unzip rsync file bc findutils wget \
  git python3 ncurses-dev
📘 重要提示 不需要以 root 身份运行 Buildroot。以普通用户身份运行所有命令，可保护系统免受编译安装过程中异常包的影响。

获取 Buildroot

Buildroot 每 3 个月发布一个版本，分别在 2 月、5 月、8 月和 11 月。 版本号格式为 YYYY.MM，例如 2024.02、2024.11。

方式一：下载发布压缩包（推荐）

发布压缩包可从官方下载页面获取：https://buildroot.org/downloads/

# 下载最新稳定版
wget https://buildroot.org/downloads/buildroot-2026.02.tar.gz
tar xzvf buildroot-2026.02.tar.gz
cd buildroot-2026.02

方式二：克隆 Git 仓库（跟踪最新开发版）

git clone https://gitlab.com/buildroot.org/buildroot.git
cd buildroot

方式三：使用 Vagrant 快速设置隔离环境

在 Buildroot 源码树的 support/misc/Vagrantfile 中提供了 Vagrantfile，可快速搭建带所有依赖的虚拟机。

# Linux / macOS
curl -O https://buildroot.org/downloads/Vagrantfile
vagrant up

# Windows PowerShell
(new-object System.Net.WebClient).DownloadFile(
  "https://buildroot.org/downloads/Vagrantfile","Vagrantfile")
vagrant up

📅 版本支持策略

  • 季度版本：2025.02、2025.05、2025.08、2025.11
  • 补丁版本：2025.02.1、2025.02.2… 直到下个季度的第一个补丁版本为止
  • LTS（长期支持）版本：每年的第一个季度版本（如 2026.02），支持到下一个 LTS 的第一个补丁版本发布

快速开始

使用 Buildroot 的第一步是创建配置。Buildroot 有一套类似 Linux 内核或 BusyBox 的 Kconfig 配置工具。

配置界面选择

menuconfig

基于 curses 的经典配置器

make menuconfig

nconfig

新版 curses 配置器

make nconfig

xconfig

基于 Qt 的图形配置器

make xconfig

gconfig

基于 GTK 的图形配置器

make gconfig

标准构建流程

  1. 运行配置工具，选择目标架构和软件包：make menuconfig
  2. 配置完成后，工具生成 .config 文件，由顶层 Makefile 读取
  3. 执行构建（以普通用户身份）：make
  4. 等待编译完成，在 output/images/ 目录找到所有镜像文件
⚠️ 注意 默认情况下，Buildroot 不支持顶层并行构建，无需运行 make -jN。若需要并行构建，请参阅"Top-level parallel build"实验性支持章节。

# 使用 QEMU aarch64 预定义配置
make qemu_aarch64_virt_defconfig

# 然后进一步自定义

# 开始构建

6.1 交叉编译工具链

编译工具链是用于编译系统代码的工具集，包括编译器（gcc）、 二进制工具（binutils，含汇编器和链接器）以及 C 标准库（如 glibc、uClibc-ng、musl）。

由于嵌入式系统使用不同于宿主系统的处理器架构，需要交叉编译工具链——即 在宿主系统（如 x86）上运行、但为目标系统（如 ARM）生成代码的工具链。

内部工具链可配置项

  • Linux 内核头文件版本（需 ≤ 目标内核版本）
  • GCC 编译器版本、binutils 版本、C 库版本
  • uClibc 特有选项：RPC 支持、宽字符支持、locale 支持、C++ 支持、线程支持等
⚠️ 警告 修改工具链配置选项后，整个工具链和系统必须重新构建。请参阅"何时需要完整重建"章节。

6.2 /dev 管理

Buildroot 支持以下几种 /dev 设备文件管理方式：

方式说明
devtmpfs（推荐）由内核在启动时自动填充 /dev，使用 udev 进行动态管理

mdevBusyBox mdev，较轻量的动态设备管理
systemd-udev配合 systemd 使用 udev 进行完整动态管理

高级用法

8.2 理解何时需要完整重建

Buildroot 不能自动检测所有需要重建的场景。以下情况必须手动执行 make clean 后重新构建：

  • 更改目标架构（CPU 类型、变体、ABI）
  • 更改工具链配置（C 库类型、版本、选项）
  • 向构建树添加新软件包时，已构建的其他包可能未感知到这个新包的存在

8.3 理解 Buildroot 如何构建包

8.4 使用 OVERRIDE_SRCDIR 加速开发

在开发过程中，可使用 OVERRIDE_SRCDIR 指向本地源码目录，避免每次修改都重新下载：

# 在 local.mk 中（推荐）或命令行中设置
# 让 Buildroot 使用本地源码目录而非下载
echo "BUSYBOX_OVERRIDE_SRCDIR = /path/to/my/busybox" >> local.mk

# 然后使用 rsync 方式构建，修改本地源码后执行：
make busybox-rebuild

8.5 Top-level 并行构建（实验性）

# 启用实验性顶层并行构建（需在配置中启用 BR2_PER_PACKAGE_DIRECTORIES）
make -jN        # N 为并行数
📘 提示 顶层并行构建目前仍是实验性功能，可能存在问题。 更稳定的方案是单包内的并行构建（大多数包自动利用多核）。

添加包（Packages）

Buildroot 提供多种包基础结构（package infrastructure）来简化软件包的添加。 每种基础结构对应不同的构建系统类型。

包文件结构

每个包由以下文件组成（以 libfoo 为例）：

package/libfoo/
├── Config.in          # Kconfig 配置条目（定义 BR2_PACKAGE_LIBFOO）
├── libfoo.mk          # 包描述（版本、源码、构建步骤）
├── libfoo.hash        # 源码校验哈希
└── 0001-fix-xxx.patch # 补丁文件（可选）

典型 .mk 文件示例（generic-package）

################################################################################
#
# libfoo
#
################################################################################

LIBFOO_VERSION = 1.2.3
LIBFOO_SITE = https://example.com/downloads
LIBFOO_SOURCE = libfoo-$(LIBFOO_VERSION).tar.gz
LIBFOO_LICENSE = GPL-2.0+
LIBFOO_LICENSE_FILES = COPYING

LIBFOO_INSTALL_STAGING = YES      # 安装到 staging/ 供其他包使用

define LIBFOO_BUILD_CMDS
    $(MAKE) $(TARGET_CONFIGURE_OPTS) -C $(@D)
endef

define LIBFOO_INSTALL_TARGET_CMDS
    $(INSTALL) -D -m 0755 $(@D)/libfoo $(TARGET_DIR)/usr/bin/libfoo
endef

$(eval $(generic-package))

Config.in 示例

config BR2_PACKAGE_LIBFOO
    bool "libfoo"
    depends on BR2_PACKAGE_LIBBAR
    help
      This is a description of the libfoo package.

      http://example.com/libfoo/

钩子（Hooks）机制

可在构建各阶段注入自定义操作：

# 可用钩子列表
LIBFOO_PRE_DOWNLOAD_HOOKS
LIBFOO_PRE_EXTRACT_HOOKS
LIBFOO_PRE_PATCH_HOOKS
LIBFOO_PRE_CONFIGURE_HOOKS
LIBFOO_POST_CONFIGURE_HOOKS
LIBFOO_POST_BUILD_HOOKS
LIBFOO_POST_INSTALL_HOOKS
LIBFOO_POST_INSTALL_TARGET_HOOKS

# 示例：在安装后删除不需要的文件
define LIBFOO_REMOVE_UNNEEDED
    rm -f $(TARGET_DIR)/usr/share/libfoo/*.xml
endef
LIBFOO_POST_INSTALL_TARGET_HOOKS += LIBFOO_REMOVE_UNNEEDED

自定义 Buildroot（BR2_EXTERNAL）

为了在 Buildroot 升级时保持定制化内容，强烈建议将自定义内容放在 Buildroot 源码树之外， 通过 BR2_EXTERNAL 机制管理。

BR2_EXTERNAL 树结构

my-br2-external/
├── external.desc          # 必须：定义外部树名称和描述
├── Config.in              # 自定义包的顶层 Kconfig 入口
├── external.mk            # 包含自定义包 .mk 文件
├── configs/               # 自定义 defconfig 文件
│   └── myboard_defconfig
├── package/               # 自定义包定义
│   └── mypkg/
│       ├── Config.in
│       └── mypkg.mk
├── board/                 # 开发板相关文件
│   └── myboard/
│       ├── linux.config
│       ├── busybox.config
│       ├── post-build.sh  # 构建后脚本
│       └── rootfs-overlay/ # 根文件系统覆盖层
└── patches/               # 包补丁（覆盖官方补丁）

external.desc 格式

name: MY_CUSTOM_TREE
desc: My custom Buildroot extension

使用外部树

# 指定单个外部树
make BR2_EXTERNAL=/path/to/my-br2-external menuconfig

# 指定多个外部树（用冒号分隔）
make BR2_EXTERNAL=/path/to/tree1:/path/to/tree2 menuconfig

# 保存后，BR2_EXTERNAL 被记录在 output/.br2-external.mk
# 后续 make 不再需要指定

根文件系统覆盖层（Rootfs Overlay）

覆盖层中的文件在构建结束时会被复制到 target/ 目录，覆盖同名文件：

# 在 Buildroot 配置中设置
BR2_ROOTFS_OVERLAY = "board/myboard/rootfs-overlay"

# 覆盖层示例结构
board/myboard/rootfs-overlay/
├── etc/
│   ├── network/interfaces  # 自定义网络配置
│   └── hostname            # 自定义主机名
└── usr/
    └── share/
        └── myapp/          # 应用资源文件

构建后脚本（Post-build Script）

# post-build.sh 示例
#!/bin/sh
set -e

# $1 = TARGET_DIR
# $2 = BINARIES_DIR （等同 images/ 目录）

# 修改目标文件系统中的文件
echo "myboard" > ${1}/etc/hostname

# 复制额外文件
cp ${BR2_EXTERNAL_MY_CUSTOM_TREE_PATH}/board/myboard/myconfig \
   ${1}/etc/myconfig
📘 提示：多个外部树（Buildroot ≥ 2016.11） 从 2016.11 版本起支持同时使用多个外部树。每个外部树通过 external.desc 中的 name 字段标识， 对应变量为 BR2_EXTERNAL_<NAME>_PATH。

版本迁移指南

通用迁移步骤

  1. 在旧版 Buildroot 环境中为所有配置执行构建，运行 make graph-size 并保存 graphs/file-size-stats.csv
  2. 查阅对应版本的迁移说明，为外部包和构建脚本做必要调整
  3. 更新 Buildroot 到新版本
  4. 从现有 .config 开始运行 make menuconfig
  5. 检查 Legacy 菜单中的条目，按帮助文本提示处理后取消选中并保存
  6. 在新版 Buildroot 环境中执行构建
  7. 修复外部包的构建错误（通常由更新的依赖引起）
  8. 运行 make graph-size 并与旧版结果对比

关键版本迁移说明

版本主要变化
2016.11 支持多个 BR2_EXTERNAL 树。旧树需添加 external.desc 文件并替换 BR2_EXTERNAL 变量名
2017.08 宿主包安装路径从 $(HOST_DIR)/usr 改为直接 $(HOST_DIR)。需更新 RPATH 设置
2023.11 SVN 下载后端默认不再获取外部引用（svn:externals）；移除架构特定补丁支持
2024.05 tarball 生成更可重现；归档后缀变更：git→-git4，svn→-svn5，cargo→-cargo2，go→-go2；需要 tar ≥ 1.35
2025.02 Mender 需要 bootstrap artifact 文件替代旧版 artifact_info
2025.05 SysV-like 系统的 /etc/resolv.conf 符号链接指向 /run/resolv.conf；Cargo tarball 后缀从 -cargo2 改为 -cargo4
2025.08 systemd-networkd 需要内核 ≥ 5.4（systemd ≥ 257）
2025.11 golang 包的 FOO_INSTALL_BINS 变量废弃，改从 FOO_BUILD_TARGETS 自动推导
2026.02 JavaScript 库不再随 Buildroot 包提供（改用 CDN/overlay/外部包）；network-manager 停止自动选择 gnutls

2024.05 哈希文件迁移脚本

# git 和 svn 包（原有 -br2 / -br3 后缀）
sed -r -i -e 's/-br2/-git4/; s/-br3/-svn5/' $(
    git grep -l -E -- '-br2|-br3' -- '*.hash'
)

# go 包（原本无后缀）
sed -r -i -e 's/(\.tar\.gz)$/-go2\1/' $(
    git grep -l -E '\$\(eval \$\((host-)?golang-package\)\)' -- '*.mk' \
    | sed -r -e 's/\.mk$/.hash/' | sort -u
)

# cargo 包（原本无后缀）
sed -r -i -e 's/(\.tar\.gz)$/-cargo2\1/' $(
    git grep -l -E '\$\(eval \$\((host-)?cargo-package\)\)' -- '*.mk' \
    | sed -r -e 's/\.mk$/.hash/' | sort -u
)

发布工程

发布策略

Buildroot 项目采用季度发布模式，并有月度补丁版本：

开发周期

每个发布周期由以下阶段组成：

  • 开发阶段（2个月）：在 master 分支上开发新功能
  • 稳定化阶段（1个月）：仅接受 bugfix，不引入新功能。以打 -rc1 标签开始，每周打一个 RC
  • 稳定化阶段期间，新功能和版本升级在 next 分支进行，正式发布后合并回 master

版本支持生命周期

  • 普通版本：支持到下一个版本的第一个补丁版本发布（例如 2025.05.x 在 2025.08.1 发布时 EOL）
  • LTS 版本：支持到下一个 LTS 的第一个补丁版本发布（例如 2025.02.x 支持到 2026.02.1 发布）
