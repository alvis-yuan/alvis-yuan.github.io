BusyBox init 与 Buildroot 平台

嵌入式 Linux 用户态第一个进程是如何将一台"裸机"引导成完整可用系统的

什么是 BusyBox init？

BusyBox 是嵌入式 Linux 领域最常用的"多合一"工具集，将数百个常见 Unix 命令整合进单一二进制文件。其中的 init 子程序扮演着用户态 PID 1 的角色——它是 Linux 内核在完成内核初始化后，第一个启动的用户空间程序。

ℹ️
官方来源： BusyBox init 的行为完整定义在源码 init/init.c 中，inittab 示例位于 git.busybox.net/busybox/tree/examples/inittab。Buildroot 的默认 inittab 位于 package/busybox/inittab，可在 github.com/buildroot/buildroot 查阅。

Buildroot 中的 init 选择

Buildroot 支持三种 init 系统，可在 System configuration → Init system 中选择：

Init 系统配置项适用场景复杂度
BusyBox init BR2_INIT_BUSYBOX 大多数嵌入式系统（默认） 轻量
sysvinit BR2_INIT_SYSV 需要运行级别的传统系统 中等
systemd BR2_INIT_SYSTEMD 复杂嵌入式系统（D-Bus、socket 激活等） 重量
✅
Buildroot 官方文档明确指出："The solution recommended by Buildroot developers is to use the BusyBox init as it is sufficient for most embedded systems."（推荐使用 BusyBox init，对大多数嵌入式系统已足够）。

🔭 第一性原理

系统启动的本质问题是：如何从一段冷启动代码，逐步构建出一个能运行任意程序的完整运行环境？ BusyBox init 的答案是：从最小可信根出发（内核 + 单一二进制），通过读取配置文件、顺序执行命令，逐步将资源（文件系统、设备节点、网络、服务）一层一层地添加到系统中，最终达到用户期望的稳定状态。

第 01 章

内核移交控制权

Linux 内核如何将执行权交给用户态第一个进程

内核初始化的最后一步

当 Linux 内核完成所有内部初始化（内存管理、驱动注册、VFS 初始化等）后，在 init/main.c 的 kernel_init() 函数中，通过以下顺序尝试启动用户态 init 进程：

C init/main.c — kernel_init() 关键片段 
/* 1. 优先使用 initramfs 中的 /init */
if (ramdisk_execute_command) {
    ret = run_init_process(ramdisk_execute_command);
    /* ramdisk_execute_command 默认为 "/init" */
}

/* 2. 若上面失败，尝试命令行指定的 init= 参数 */
if (execute_command) {
    ret = run_init_process(execute_command);
}

/* 3. 最后依次尝试标准路径 */
if (!try_to_run_init_process("/sbin/init") ||
    !try_to_run_init_process("/etc/init")  ||
    !try_to_run_init_process("/bin/init")  ||
    !try_to_run_init_process("/bin/sh"))
    return 0;
🔗
来源参考：Booting to User Space: The Init Process，以及 Linux 内核源码 init/main.c

run_init_process 的实现

run_init_process() 调用 kernel_execve()，将内核线程原地替换为指定的用户态程序，完成从内核空间到用户空间的跨越：

C init/main.c — run_init_process() 
static int run_init_process(const char *init_filename)
{
    argv_init[0] = init_filename;
    pr_info("Run %s as init process\n", init_filename);
    return kernel_execve(init_filename, argv_init, envp_init);
    /* execve 成功则永不返回，内核线程被替换为用户进程 */
}

Buildroot 中 /sbin/init 的真实身份

在 Buildroot 生成的根文件系统中，/sbin/init 是一个指向 /bin/busybox 的软链接。BusyBox 通过检查 argv[0] 来决定运行哪个子程序：当 argv[0] 为 init 时，执行 init 逻辑。

PID 1 的特殊地位

PID 1 在 Linux 系统中具有特殊地位：

  • 内核对 PID 1 发送任何信号的默认动作是忽略（包括 SIGKILL），必须显式安装信号处理器才能响应
  • 所有孤儿进程（父进程已退出的进程）都会被重新挂靠到 PID 1 下
  • PID 1 若退出，内核将触发 kernel panic
  • 因此 BusyBox init 进入主循环后永远不会主动退出
🔭 第一性原理

内核移交控制权的本质是：通过 execve 系统调用，用一段用户空间程序替换掉当前内核线程的地址空间。这个操作不创建新进程，而是原地替换——进程号保持为 1，但代码、数据段全部换成了 BusyBox init 的映像。这是内核与用户空间之间唯一的"边界穿越"时刻：之后所有操作都必须通过系统调用请求内核服务。

第 02 章

init_main() 初始化

BusyBox init 进程启动后的第一批关键操作

init_main() 执行序列

BusyBox init 进入 init_main() 后，在解析 inittab 之前，依次完成以下自我初始化工作（源自 init/init.c）：

1

PID 检验

确认自身 PID 为 1，否则报错退出（getpid() != 1）。

2

禁用 Ctrl-Alt-Del 硬重启

调用 reboot(RB_DISABLE_CAD)，接管 Ctrl-Alt-Del，改由 init 自己优雅处理（发送 SIGINT）。

3

设置信号屏蔽集

将 SIGINT、SIGQUIT、SIGTERM、SIGUSR1/2、SIGHUP、SIGCHLD 等关键信号加入 delayed_sigset，使用 sigtimedwait() 在主循环中集中处理。

4

控制台初始化 console_init()

探测并打开默认控制台设备（通常为 /dev/console），设置合理的终端参数（set_sane_term()），将 stdin/stdout/stderr 绑定到控制台。

5

切换工作目录 & 创建新会话

xchdir("/") 确保工作目录在根目录；setsid() 创建新会话，使 init 成为会话组长，脱离任何控制终端。

6

设置基本环境变量

设置 PATH、SHELL=/bin/sh、USER=root 等基本环境，为后续子进程提供运行环境。

7

检查单用户模式

检查内核命令行参数是否包含 single/-s/1，若是则仅启动一个 shell，跳过完整 inittab 解析。

源码关键片段

C init/init.c — init_main() 核心逻辑 
/* 验证 PID */
if (getpid() != 1)
    bb_simple_error_msg_and_die("must be run as PID 1");

/* 禁用硬件 Ctrl-Alt-Del */
reboot(RB_DISABLE_CAD);

/* 设置延迟信号集 */
sigaddset(&G.delayed_sigset, SIGINT);   /* Ctrl-Alt-Del */
sigaddset(&G.delayed_sigset, SIGQUIT);  /* re-exec another init */
sigaddset(&G.delayed_sigset, SIGUSR1);  /* halt */
sigaddset(&G.delayed_sigset, SIGTERM);  /* reboot */
sigaddset(&G.delayed_sigset, SIGUSR2);  /* poweroff */
sigaddset(&G.delayed_sigset, SIGHUP);   /* reread /etc/inittab */
sigaddset(&G.delayed_sigset, SIGCHLD);  /* child 退出通知 */
sigprocmask(SIG_BLOCK, &G.delayed_sigset, NULL);

/* 初始化控制台 */
console_init();
set_sane_term();
xchdir("/");
setsid();

/* 设置基本环境 */
putenv((char *) bb_PATH_root_path);
putenv((char *) "SHELL=/bin/sh");
putenv((char *) "USER=root");

/* 检查单用户模式 */
if (argv && (strcmp(argv, "single") == 0 || ...)) {
    new_init_action(RESPAWN, bb_default_login_shell, "");
} else {
    /* 正常启动：解析 inittab */
    parse_inittab();
}

/* 执行动作序列 */
run_actions(SYSINIT);  /* 等待完成 */
run_actions(WAIT);     /* 等待完成 */
run_actions(ONCE);     /* 不等待 */

/* 进入无限主循环 */
while (1) {
    run_actions(RESPAWN);    /* 重生进程 */
    run_actions(ASKFIRST);   /* 按键激活终端 */
    /* 等待信号或子进程退出 */
}

动作类型说明

action 类型等待完成含义
sysinit✅ 是系统初始化，最优先执行，必须等待每条命令完成
wait✅ 是sysinit 之后执行，同样等待完成
once❌ 否执行一次，不等待，不重启
respawn❌ 否在主循环中执行；进程退出后自动重启
askfirst❌ 否同 respawn，但等待用户按 Enter 再启动
ctrlaltdel✅ 是收到 SIGINT（Ctrl-Alt-Del）时执行
shutdown✅ 是系统关机/重启时执行
restart—收到 SIGQUIT 时 exec 替换 init 自身
🔭 第一性原理

init_main() 的初始化阶段解决了一个根本问题：如何在没有任何运行环境的前提下，建立最基本的运行环境？控制台需要先打开（否则无法输出错误），信号需要先屏蔽（否则意外信号会干扰初始化），工作目录需要固定为根（保证相对路径可预测）。这些操作的顺序是经过精心设计的，缺少任何一步都会导致后续操作失败。

第 03 章

/etc/inittab 解析

Buildroot 默认 inittab 的格式、内容与解析机制

inittab 格式

BusyBox inittab 的格式与 sysvinit 不同，每行格式为：

格式/etc/inittab
# <id>:<runlevels>:<action>:<process>
#
# id        == 控制 tty（附加到 /dev/ 后面），空表示 /dev/console
# runlevels == BusyBox init 完全忽略此字段
# action    == 动作类型（见下表）
# process   == 要执行的命令
⚠️
重要提示：BusyBox inittab 与 sysvinit 的 inittab 语法不兼容！BusyBox 官方明确警告"do not use a random inittab documentation from the Internet to learn about BusyBox inittab"。

parse_inittab() 工作原理

BusyBox init 使用 parse_inittab() 函数读取并解析 /etc/inittab，构建一个动作链表（init_action_list）：

C init/init.c — parse_inittab() 简化版 
static void parse_inittab(void)
{
    #if ENABLE_FEATURE_USE_INITTAB
    parser_t *parser = config_open2("/etc/inittab", fopen_for_read);

    if (parser == NULL)  /* inittab 不存在 → 使用内置默认值 */
    #endif
    {
        new_init_action(SYSINIT,    INIT_SCRIPT,  "");
        new_init_action(ASKFIRST,   bb_default_login_shell, "");
        new_init_action(CTRLALTDEL, "reboot",   "");
        new_init_action(SHUTDOWN,   "umount -a -r", "");
        new_init_action(SHUTDOWN,   "swapoff -a",   "");
        new_init_action(RESTART,    "init",         "");
        return;
    }

    /* 逐行解析：tty:runlevel:action:command */
    while (config_read(parser, token, 4, 0, "#:", ...)) {
        action = index_in_strings(actions, token[2]);
        tty    = concat_path_file("/dev/", skip_dev_pfx(token[0]));
        new_init_action(1 << action, token[3], tty);
    }
}

有无 inittab 的行为对比

情形行为
有 /etc/inittab（Buildroot 默认） 按文件定义的动作列表执行，灵活可配置
无 /etc/inittab 使用内置默认值：执行 /etc/init.d/rcS，然后在 4 个虚拟控制台上启动 askfirst shell
🔭 第一性原理

inittab 解决的核心问题是：如何将"要做什么"与"怎么做"分离？init 程序本身只负责解析配置、调度执行，而具体操作（挂载文件系统、加载设备管理器等）被描述在配置文件中。这种"策略与机制分离"的设计使 init 保持简单，同时让系统行为完全可配置——更改 inittab 即可改变启动行为，无需重新编译 init。

第 04 章

sysinit 阶段详解

run_actions(SYSINIT) 的逐步执行过程

sysinit 阶段的特性

解析完 inittab 后，init 首先调用 run_actions(SYSINIT)。sysinit 动作有以下特性：

  • 顺序执行：按照 inittab 中出现的顺序，一条一条执行
  • 同步等待：每条命令必须执行完成（waitpid()）才执行下一条
  • 执行一次：完成后该动作从链表中删除，不再重复

run_actions() 实现

C init/init.c — run_actions() 核心逻辑 
static void run_actions(int action)
{
    struct init_action *a, *tmp;
    for (a = init_action_list; a; a = tmp) {
        tmp = a->next;
        if (a->action_type & action) {
            if (a->action_type & (SYSINIT | WAIT | CTRLALTDEL | SHUTDOWN | RESTART)) {
                waitfor(a);            /* fork + waitpid，同步等待 */
                delete_init_action(a); /* 从链表中删除 */
            } else if (a->action_type & ONCE) {
                run(a);               /* fork，不等待 */
                delete_init_action(a);
            } else if (a->action_type & (RESPAWN | ASKFIRST)) {
                /* 只启动 pid==0 的（尚未运行的）动作 */
                if (a->pid == 0)
                    a->pid = run(a);
            }
        }
    }
}

Buildroot sysinit 步骤逐条解析

1

/bin/mount -t proc proc /proc

挂载 procfs 伪文件系统到 /proc。procfs 是内核向用户空间暴露进程信息、系统状态的接口。许多后续命令（如读取内核参数、查看进程列表）都依赖此挂载点。

💡
procfs 不使用真实磁盘，数据在读取时由内核动态生成。
2

/bin/mount -o remount,rw /

将根文件系统从只读重新挂载为读写。内核启动时根文件系统通常以只读方式挂载（用于 fsck 安全检查），此步骤允许后续写入操作。

3

/bin/mkdir -p /dev/pts /dev/shm

创建 /dev/pts（伪终端从设备目录）和 /dev/shm（POSIX 共享内存）目录，为后续挂载 devpts 和 tmpfs 做准备。

4

/bin/mount -a

读取 /etc/fstab 并挂载其中所有标记为自动挂载（没有 noauto 选项）的文件系统。通常包括 sysfs、devpts、tmpfs 等。

5

/bin/mkdir -p /run/lock/subsys

创建运行时数据目录结构，/run 通常挂载为 tmpfs，用于存储守护进程的 PID 文件、套接字等临时数据。

6

/sbin/swapon -a

激活 /etc/fstab 中定义的所有交换分区/文件。对于内存受限的嵌入式系统尤为重要。

7

创建标准 fd 软链接

将 /dev/fd、/dev/stdin、/dev/stdout、/dev/stderr 链接到 /proc/self/fd 对应条目，使脚本中的标准 I/O 重定向正常工作。

8

/bin/hostname -F /etc/hostname

从 /etc/hostname 文件读取主机名并设置系统主机名，影响提示符显示和网络标识。

9

/etc/init.d/rcS

执行 rc 脚本总入口，按数字顺序调用 /etc/init.d/S??* 脚本，完成设备管理、网络配置、服务启动等全部用户态初始化。详见第 07 章。

🔭 第一性原理

sysinit 阶段的设计体现了依赖排序原则：每一步操作都依赖前一步的结果。procfs 必须先挂载（后续命令需要读取 /proc）；根文件系统必须先变成读写（后续 mkdir 需要写权限）；/dev/pts 目录必须先创建（mount -a 挂载 devpts 时才有目标）。这种严格的顺序依赖不允许并行化，因此使用同步的 waitpid()——这是嵌入式启动速度慢的根本原因，也是 systemd 引入并行化的动机。

第 05 章

文件系统挂载体系

/etc/fstab 与虚拟文件系统的挂载机制

虚拟文件系统概述

Linux 嵌入式系统在启动过程中需要挂载多个虚拟文件系统（VFS），这些文件系统不对应真实磁盘，而是内核提供的接口：

文件系统挂载点类型作用
procfs/proc虚拟进程信息、内核参数（/proc/sys）、设备信息
sysfs/sys虚拟设备/驱动拓扑结构，mdev/udev 依赖此文件系统
devtmpfs/dev虚拟设备节点，内核自动创建基本设备节点
devpts/dev/pts虚拟伪终端从端（ssh、screen 等依赖）
tmpfs/dev/shm内存POSIX 共享内存
tmpfs/run内存运行时数据（PID 文件、套接字）

Buildroot 默认 /etc/fstab

Buildroot 的根文件系统骨架（system/skeleton/etc/fstab）提供如下默认内容：

fstab /etc/fstab（Buildroot 默认骨架） 
# <file system> <mount pt>  <type>   <options>          <dump> <pass>
/dev/root       /           ext2     rw,noauto          0      1
proc            /proc       proc     defaults            0      0
devpts          /dev/pts    devpts   defaults,gid=5,mode=620  0  0
tmpfs           /dev/shm    tmpfs    mode=0777           0      0
tmpfs           /tmp        tmpfs    mode=1777           0      0
tmpfs           /run        tmpfs    mode=0755,nosuid,nodev 0   0
sysfs           /sys        sysfs    defaults            0      0

挂载顺序与依赖关系

devtmpfs 与 mdev 的关系

Buildroot 支持多种 /dev 管理策略（在 System configuration → /dev management 中选择）：

策略配置项机制
静态设备表 BR2_ROOTFS_DEVICE_CREATION_STATIC 构建时预先创建设备节点，不支持热插拔
devtmpfs + mdev BR2_ROOTFS_DEVICE_CREATION_DYNAMIC_MDEV 内核自动创建基本设备节点；mdev 处理热插拔（推荐）
devtmpfs + eudev BR2_ROOTFS_DEVICE_CREATION_DYNAMIC_EUDEV 功能更强的 udev 替代，适合需要复杂规则的场景
💡
Buildroot 官方推荐从 devtmpfs only 开始，当需要热插拔通知或固件加载时，升级到 devtmpfs + mdev。
🔭 第一性原理

文件系统挂载的本质是：将内核的数据结构"映射"到用户可见的路径空间中。procfs/sysfs 不是真实的磁盘，而是内核以文件形式暴露的 API。没有这些挂载点，用户空间程序就无法与内核通信——mdev 不能扫描设备（需要 /sys），hostname 命令无法读取内核参数（需要 /proc）。因此文件系统挂载是用户态建立内核通信通道的核心操作，是一切后续活动的基础。

第 06 章

mdev 设备管理

BusyBox mdev 如何填充 /dev 目录并处理热插拔

S10mdev 脚本

当 Buildroot 选择 BR2_ROOTFS_DEVICE_CREATION_DYNAMIC_MDEV=y 时，会安装 package/busybox/S10mdev 到目标系统的 /etc/init.d/S10mdev。它由 rcS 调用，是 init.d 中最先执行的脚本之一：

Shell package/busybox/S10mdev ↗ 原文 
#!/bin/sh

DAEMON="mdev"

start() {
    # 注册 mdev 为内核热插拔处理器
    echo "/sbin/mdev" > /proc/sys/kernel/hotplug

    # 扫描 /sys 目录，为当前所有设备创建 /dev 节点
    mdev -s
}

stop() {
    # 取消热插拔注册
    echo "" > /proc/sys/kernel/hotplug
}

case "$1" in
    start)   start;;
    stop)    stop;;
    restart) stop; start;;
    *)       echo "Usage: $0 {start|stop|restart}"; exit 1;;
esac

mdev 的两个工作阶段

1

静态扫描阶段：mdev -s

递归扫描 /sys 中的所有 dev 文件（包含设备主次设备号），在 /dev 下创建对应的设备节点。这确保系统启动时所有已探测的设备都有对应的 /dev 节点。

示例mdev -s 效果
# 读取 /sys/class/tty/ttyS0/dev → "4:64"
# 创建 /dev/ttyS0，主设备号 4，次设备号 64
mknod /dev/ttyS0 c 4 64
2

动态热插拔阶段：注册为内核 hotplug 处理器

将 /sbin/mdev 写入 /proc/sys/kernel/hotplug，此后内核每次检测到设备插入/拔出事件时，都会调用 mdev，并传入相关环境变量（ACTION、DEVPATH 等），mdev 根据 /etc/mdev.conf 的规则创建或删除设备节点。

mdev.conf 规则示例

conf /etc/mdev.conf（Buildroot 默认） 
# 格式: <设备名正则> <uid>:<gid> <权限> [=别名] [@|$|!命令]

# 所有用户可读写的空设备

# 控制台
console          0:5 600

# 随机数设备

urandom          0:0 444

# 串口

# 块设备
sd[a-z][0-9]*    0:6 660

# 驱动自动加载（通过 MODALIAS）

# @modprobe $MODALIAS: 在设备创建前自动加载对应驱动

mdev 工作流程图

驱动模块自动加载

mdev 支持通过 MODALIAS 环境变量自动加载内核模块：当内核热插拔一个新设备时，会在环境变量中携带 MODALIAS（设备的模块别名），mdev 的规则文件可配置在设备节点创建前/后执行 modprobe $MODALIAS，从而实现驱动按需加载。

ℹ️
模块文件通常位于 /lib/modules/$(uname -r)/，由 Buildroot 构建内核时生成并安装到目标系统。modprobe 会读取 modules.dep 处理依赖关系。
🔭 第一性原理

设备管理的根本问题是：如何将内核中的设备对象映射为用户空间可访问的文件？Linux 的"一切皆文件"哲学要求所有设备必须通过 /dev 下的特殊文件访问。mdev 通过读取 sysfs（内核设备拓扑的投影）并调用 mknod 系统调用，完成这个映射。热插拔机制则进一步让这个映射在设备生命周期内保持同步——设备插入时创建节点，拔出时删除节点。这是用户空间对内核硬件抽象层的"实例化"过程。

第 07 章

rcS 与 S?? 初始化脚本

Buildroot 的 SysV 风格服务启动框架

rcS 脚本：服务启动总调度器

inittab 中最后一条 sysinit 动作是 /etc/init.d/rcS。这是整个用户态服务初始化的调度中心：

Shell package/initscripts/init.d/rcS ↗ 原文 
#!/bin/sh
#
# Start all init scripts in /etc/init.d
# executing them in numerical order.
#

for i in /etc/init.d/S??* ; do

    # 忽略悬空的软链接
    [ ! -f "$i" ] && continue

    case "$i" in
        *.sh)
            # .sh 脚本：在子 shell 中 source（更快，避免 fork/exec）
            (
                trap - INT QUIT TSTP
                set start
                . $i
            )
            ;;
        *)
            # 无 .sh 后缀：fork 子进程执行，传入 start 参数
            $i start
            ;;
    esac
done
ℹ️
S??* 是 shell glob 模式，匹配以大写 S 开头、后跟两位数字的文件名。Shell 对文件名排序是字典序，因此 S10 在 S20 之前，S09 在 S10 之前。

Buildroot 默认 init.d 脚本

Buildroot 的 package/initscripts/ 提供以下默认脚本，加上 busybox 包的 S10mdev：

脚本来源包功能
S10mdev package/busybox 启动 mdev 设备管理器，填充 /dev 节点，注册热插拔处理器
S20urandom package/initscripts 从上次保存的种子恢复随机数熵池（/dev/urandom），保存新种子
S40network package/initscripts 读取 /etc/network/interfaces，调用 ifup -a 配置网络接口
S??syslog busybox/syslogd 启动 syslogd/klogd 系统日志守护进程

S20urandom 详解

Shell package/initscripts/init.d/S20urandom（关键部分） 
start() {
    # 将上次保存的随机种子输入内核熵池
    if [ -f /etc/random-seed ]; then
        cat /etc/random-seed > /dev/urandom
        rm -f /etc/random-seed
    fi
    # 立即保存新种子供下次启动使用
    umask 077
    dd if=/dev/urandom of=/etc/random-seed count=1 > /dev/null 2&1
    umask 022
}

stop() {
    # 关机时保存当前熵池状态
    umask 077
    dd if=/dev/urandom of=/etc/random-seed count=1 > /dev/null 2&1
}

自定义服务脚本模板

在 Buildroot 中添加自定义服务，遵循以下模板（数字越大越晚启动）：

Shell /etc/init.d/S50myservice（模板） 
#!/bin/sh

DAEMON="myservice"
DAEMON_PATH="/usr/bin/myservice"
PIDFILE="/run/myservice.pid"

start() {
    printf "Starting %s... " "$DAEMON"
    start-stop-daemon -S -b -p "$PIDFILE" -x "$DAEMON_PATH"
    echo "OK"
}

stop() {
    printf "Stopping %s... " "$DAEMON"
    start-stop-daemon -K -p "$PIDFILE"
    echo "OK"
}

case "$1" in
    start)   start;;
    stop)    stop;;
    restart) stop; start;;
    *)       echo "Usage: $0 {start|stop|restart}"; exit 1;;
esac
🔭 第一性原理

rcS + S?? 框架体现了分层解耦的设计思想：init 只负责触发 rcS，rcS 只负责按顺序调用 S?? 脚本，每个 S?? 脚本只负责一类服务的启停逻辑。这种"职责单一原则"使得：添加新服务只需添加新脚本（无需修改 init 或 rcS），删除服务只需删除脚本，调整顺序只需改变文件名前缀数字。代价是严格的顺序执行——每个脚本必须等待前一个完成，这正是 systemd 并行化要解决的核心问题。

第 08 章

respawn 与 getty：登录终端

sysinit 完成后的主循环与终端守护

sysinit 完成后的主循环

当所有 sysinit 动作执行完毕（包括 rcS），init 进入 主循环（main loop），轮流处理 RESPAWN 和 ASKFIRST 类型的动作：

C init/init.c — main loop 
/* 进入无限循环，永不退出 */
while (1) {
    /* 1. 重启所有已退出的 respawn/askfirst 进程 */
    run_actions(RESPAWN);
    run_actions(ASKFIRST);

    /* 2. 使用 sigtimedwait() 等待信号或子进程退出
     *    最多等待 ~2 秒（避免 busy-loop）*/
    sig = sigtimedwait(&G.delayed_sigset, &info, &G.delay);

    /* 3. 处理信号（SIGCHLD → 标记已退出进程；
     *    SIGHUP → 重新读 inittab；等） */
    if (sig > 0)
        process_signals(sig);
}

getty：提供登录提示符

getty（Get TTY）是提供登录终端的程序。在 Buildroot 的 inittab 中，串口 getty 通常以 respawn 方式运行：

inittab /etc/inittab — getty 配置 
# 串口控制台 getty（由 Buildroot 根据配置自动取消注释）
# 格式: tty::respawn:/sbin/getty -L <port> <baudrate> <termtype>
ttyS0::respawn:/sbin/getty -L ttyS0 115200 vt100

# askfirst 方式（需按 Enter 激活，避免无人值守时的串口噪音）
# ::askfirst:-/bin/sh

getty 工作流程

1

init fork → getty 进程

init 通过 run(action) fork 出 getty 子进程，记录其 PID。

2

getty 打开终端设备

getty 打开对应的 tty 设备（如 /dev/ttyS0），设置波特率、终端参数，输出登录提示符 login:。

3

用户输入 → execve(login)

用户输入用户名后，getty exec 替换为 /bin/login（或直接替换为 /bin/sh），完成认证。

4

login 退出 → init 重启 getty

用户注销后，login 退出，init 的 SIGCHLD 处理器收到通知，在下一轮主循环中自动重启 getty（因为是 respawn），恢复登录提示符。

Buildroot 中自动配置 getty

Buildroot 在 System configuration → Run a getty (login prompt) after boot 中提供图形化配置，自动修改 inittab 中的 getty 行：

  • BR2_TARGET_GENERIC_GETTY=y
  • BR2_TARGET_GENERIC_GETTY_PORT（如 ttyS0）
  • BR2_TARGET_GENERIC_GETTY_BAUDRATE（如 115200）
🔭 第一性原理

respawn 机制体现了"故障自愈"原则：init 不假定任何子进程会永久存活，而是将"进程退出后重启"作为首要策略。这对于嵌入式系统至关重要——若 getty 因某种原因崩溃，系统不会失去控制台访问能力。这是一种简单但有效的可靠性设计：不试图预防故障，而是确保快速恢复。

第 09 章

信号处理机制

init 如何响应系统事件与外部控制信号

PID 1 的信号特殊性

对于 PID 1，Linux 内核在启动时将所有信号的默认动作设置为"忽略"（而不是 kill）。这意味着：除非 init 主动安装信号处理器，否则任何信号（包括 SIGKILL！）都不会终止 init 进程。

⚠️
从 Linux 3.9 起，在命名空间（namespace）容器中，SIGKILL 和 SIGSTOP 对 PID 1 的行为有所不同，但在宿主机上 PID 1 对 SIGKILL 仍然免疫（除非进程主动处理）。

BusyBox init 的信号表

信号触发方式init 的响应处理方式
SIGCHLD 子进程退出 调用 waitpid() 回收，标记 respawn 进程以便重启 延迟（主循环）
SIGHUP kill -HUP 1 重新读取 /etc/inittab，更新动作列表 延迟（主循环）
SIGINT Ctrl-Alt-Del（CAD） 执行 inittab 中的 ctrlaltdel 动作（通常为 reboot） 延迟（主循环）
SIGTERM kill -TERM 1 执行 shutdown + reboot 序列（RB_AUTOBOOT） 立即
SIGUSR1 kill -USR1 1 halt（停机，不关电） 立即
SIGUSR2 kill -USR2 1 poweroff（关电） 立即
SIGQUIT kill -QUIT 1 执行 restart 动作（re-exec init 自身） 延迟（主循环）
SIGTSTP kill -TSTP 1 暂停重生（停止 respawn），等待 SIGCONT 立即

两类信号处理器

BusyBox init 将信号处理器分为两类：

SIGHUP 重读 inittab

发送 SIGHUP 给 init 进程（kill -HUP 1）会触发 parse_inittab() 重新执行，允许在不重启系统的情况下修改运行服务。但需注意：

  • 新增的 sysinit/wait/once 动作不会重新执行（它们已被从链表中删除）
  • 新增的 respawn/askfirst 动作会在下次主循环中启动
  • 删除的 respawn 动作在当前周期进程退出后不再重启
🔭 第一性原理

信号处理机制解决的是 init 作为系统协调者的控制接口问题：外部如何安全地向 init 传达意图（重启、关机、重读配置）？将信号分为"立即"和"延迟"两类，是在"响应及时性"和"操作安全性"之间的权衡——shutdown 信号必须立即处理（不能等到 sysinit 完成），而修改动作链表的操作（SIGHUP）必须延迟到主循环中安全执行（避免在遍历链表时修改链表导致数据竞争）。这是一个经典的"中断安全"设计模式。

第 10 章

关机流程与 rcK 脚本

系统优雅关机的完整序列

关机触发方式

BusyBox init 的关机可由以下方式触发：

  • reboot 命令 → 向 init 发送 SIGTERM
  • halt 命令 → 向 init 发送 SIGUSR1
  • poweroff 命令 → 向 init 发送 SIGUSR2
  • Ctrl-Alt-Del → 内核转换为 SIGINT 发给 init

run_shutdown_and_kill_processes()

C init/init.c — 关机序列 
static void run_shutdown_and_kill_processes(void)
{
    /* 1. 先执行 inittab 中的 shutdown 动作（rcK、swapoff、umount）*/
    run_actions(SHUTDOWN);

    /* 2. 广播 SIGTERM，给所有进程机会优雅退出 */
    message(L_CONSOLE | L_LOG, "The system is going down NOW!");
    kill(-1, SIGTERM);
    message(L_CONSOLE | L_LOG, "Sent SIG%s to all processes", "TERM");
    sync();     /* 刷新所有文件系统缓冲区 */
    sleep(1);   /* 等待 1 秒让进程响应 */

    /* 3. 强制杀死所有剩余进程 */
    kill(-1, SIGKILL);
    message(L_CONSOLE, "Sent SIG%s to all processes", "KILL");
    sync();
}

/* 4. 调用 reboot(RB_AUTOBOOT/RB_HALT_SYSTEM/RB_POWER_OFF) */

rcK 脚本：服务优雅停止

/etc/init.d/rcK 是关机时执行的脚本，负责按逆序停止所有服务（K 代表 Kill/Stop）：

Shell package/initscripts/init.d/rcK ↗ 原文 
#!/bin/sh
#
# Stop all init scripts in /etc/init.d
# executing them in REVERSE numerical order.
#

for i in $(ls -r /etc/init.d/S??*) ; do
    [ ! -f "$i" ] && continue

    case "$i" in
        *.sh)
            (
                trap - INT QUIT TSTP
                set stop
                . $i
            )
            ;;
        *)
            $i stop
            ;;
    esac
done
ℹ️
rcK 使用 ls -r（逆序）遍历 S?? 脚本，这与 rcS 的正序形成镜像：先启动的服务最后停止（如 S40network 先于 S10mdev 停止）。

完整关机序列图

inittab 中的 shutdown 动作

inittabBuildroot /etc/inittab — shutdown 部分
# 关机时执行（顺序执行，等待完成）
::shutdown:/etc/init.d/rcK   # 停止所有用户服务
::shutdown:/sbin/swapoff -a   # 停用交换空间
::shutdown:/bin/umount -a -r  # 卸载文件系统（-r：失败则只读）
🔭 第一性原理

优雅关机的核心原则是：以正确的逆序撤销所有已完成的操作。启动时先挂载文件系统再启动服务，因此关机时必须先停止服务再卸载文件系统——违反此顺序会导致数据丢失或文件系统损坏。SIGTERM → sync() → sleep(1) → SIGKILL 的序列，则体现了"先礼后兵"原则：给进程机会优雅退出（刷新缓冲区、释放资源），超时后才强制终止。

🔭 最终第一性原理总结

BusyBox init 解决的根本问题是：如何用最小的代码，在最少的假设下，将一个裸的 Linux 内核引导成一个完整可用的系统？

它的答案体现了几个核心原则：

① 最小可信根：init 本身只是一个几 KB 的程序，但它能启动任意复杂的系统，因为它通过配置文件（inittab）和 shell 脚本（rcS/S??）将工作委托出去。

② 依赖驱动的顺序：每一步操作都在其依赖满足后才执行——这不是任意约定，而是对底层物理约束（文件系统必须先挂载才能访问、设备节点必须先创建才能打开）的直接映射。

③ 故障容忍：respawn 机制确保关键服务（getty）崩溃后自动恢复；PID 1 永不退出确保孤儿进程有人收割；shutdown 序列确保数据完整性。

④ 策略与机制分离：init 是"机制"（如何执行动作），inittab 是"策略"（执行什么动作）。这种分离使同一个 init 二进制可以驱动无数不同配置的嵌入式系统。
