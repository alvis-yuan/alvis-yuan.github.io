📖 概述 · 第 0 章

引言与工具链概览

什么是 GNU Autotools？

GNU Autotools（也称 GNU Build System）是一套由 GNU 项目开发的构建工具链，它使开发者能够编写出可在各种类 POSIX 系统上自动配置、编译和安装的软件包。其核心目标是：让用户的生活更轻松——无论用户使用何种操作系统、编译器版本或目录布局，都能通过统一的三条命令完成安装： 

# 用户安装软件包的标准三步骤 (GNU Coding Standards)
./configure && make && make install
📌 官方来源
本文档内容严格依据 GNU Autoconf Manual (v2.73, 2026年3月)、 GNU Automake Manual (v1.18.1, 2025年6月) 及 GNU Libtool Manual 编写。

工具链组成

Autotools 是多个工具的集合体，每个工具各司其职：

工具作用输入输出
autoconf生成可移植的 configure shell 脚本configure.ac + aclocal.m4configure
automake生成符合 GNU 标准的 Makefile 模板Makefile.am + configure.acMakefile.in
aclocal收集 M4 宏，生成 aclocal.m4configure.ac + 系统宏库aclocal.m4
autoheader生成配置头文件模板configure.acconfig.h.in
libtoolize添加 Libtool 共享库支持configure.ac辅助脚本 + M4 宏
autoreconf按正确顺序自动调用上述所有工具整个源码包完整构建系统
autoscan扫描源码，生成初始 configure.ac 草稿源码目录configure.scan
图 1：GNU Autotools 工具链全景图（开发者阶段 → 用户阶段）

M4 宏处理器

Autotools 的核心语言是 M4——一个通用宏处理器。Autoconf 使用 M4 将 configure.ac 中的高层宏调用展开成可移植的 shell 脚本。在 M4 中，方括号 [] 用于括起含空白字符的参数（类似其他语言的双引号），这是编写 configure.ac 时必须了解的基础规则。

Automake 则要求 perl 来生成 Makefile.in 文件。但重要的是：由 Automake 创建的分发包无需 perl 即可构建——最终用户只需标准的 shell 和 make 工具。

📋 本章总结

GNU Autotools 是 GNU Build System 的工具集，由 autoconf、automake、aclocal、autoheader、libtoolize 共同协作，将开发者手写的 configure.ac 与 Makefile.am 转化为可在任意 POSIX 系统上运行的构建脚本。从第一性原理看：可移植性是核心目标——在编译时探测系统能力，而非在代码中硬编码平台假设。

autoconfconfigure.ac → configure 脚本
automakeMakefile.am → Makefile.in 模板
aclocal收集 M4 宏 → aclocal.m4
autoreconf自动按序调用所有工具

项目目录结构

典型目录布局

一个采用 Autotools 管理的典型 C 项目的目录结构如下。文件分为三类：手写文件（开发者维护）、生成文件（工具自动产生）和辅助文件（autoreconf 复制）。

myproject/
├── configure.ac # ✏️ 手写：Autoconf 输入，M4 宏脚本
├── configure # 🔧 生成：由 autoconf 产生的 shell 脚本
├── aclocal.m4 # 🔧 生成：由 aclocal 收集的 M4 宏
├── config.h.in # 🔧 生成：由 autoheader 产生的头文件模板
├── Makefile.am # ✏️ 手写：顶层 Automake 输入
├── Makefile.in # 🔧 生成：由 automake 产生的 Makefile 模板
├── Makefile # 🔧 生成：由 ./configure 从 Makefile.in 产生
├── config.h # 🔧 生成：由 ./configure 从 config.h.in 产生
├── config.status # 🔧 生成：记录 configure 选项，可重新生成文件
├── config.log # 🔧 生成：configure 过程的详细日志
├── autom4te.cache/ # 🔧 生成：autom4te 缓存目录（发布时可删除）
│
├── build-aux/ # 📂 约定：辅助构建脚本目录
│ ├── install-sh # 📋 辅助：可移植安装脚本
│ ├── missing # 📋 辅助：处理缺失工具
│ ├── compile # 📋 辅助：编译器包装脚本
│ ├── depcomp # 📋 辅助：依赖跟踪脚本
│ ├── config.guess # 📋 辅助：猜测系统类型
│ └── config.sub # 📋 辅助：规范化系统类型
│
├── m4/ # 📂 约定：自定义 M4 宏目录
│ ├── libtool.m4 # 📋 辅助：libtoolize 复制的宏文件
│ └── my_macros.m4 # ✏️ 手写：项目自定义宏

│ ├── Makefile.am # ✏️ 手写：子目录 Automake 输入

│ ├── Makefile # 🔧 生成：由 ./configure 产生
│ ├── main.c # ✏️ 手写：源码
│ └── .deps/ # 🔧 生成：自动依赖跟踪文件
│
├── include/ # 📂 头文件目录（可选）
│ └── mylib.h # ✏️ 手写：公共头文件
│
├── tests/ # 📂 测试目录（可选）
│ ├── Makefile.am # ✏️ 手写
│ └── test_foo.c # ✏️ 手写：测试源码
│
├── README # ✏️ 手写：GNU 标准要求的文件
├── NEWS # ✏️ 手写：GNU 标准要求的文件
├── AUTHORS # ✏️ 手写：GNU 标准要求的文件
├── ChangeLog # ✏️ 手写：GNU 标准要求的文件
└── INSTALL # 🔧 生成：automake 安装说明（可覆盖）
💡 提示：AC_CONFIG_AUX_DIR 和 AC_CONFIG_MACRO_DIRS
在 configure.ac 中使用 AC_CONFIG_AUX_DIR([build-aux]) 可将辅助脚本（install-sh、config.guess 等）放入 build-aux/ 子目录，而非散落在根目录。使用 AC_CONFIG_MACRO_DIRS([m4]) 可将自定义 M4 宏集中在 m4/ 目录。这是现代 Autotools 项目的推荐实践。

📋 本章总结

Autotools 项目目录结构清晰分层：手写文件是不变的"意图声明"，生成文件是工具从意图推导出的"产物"。build-aux/ 集中辅助脚本，m4/ 集中自定义宏，src/ 存放每一级都有独立 Makefile.am 的源码树。从第一性原理看：每一层目录都对应一个作用域，每个 Makefile.am 只描述本目录内的构建规则，递归结构使大型项目模块化。

⚙️ 核心文件 · 第 2 章

configure.ac 详解

configure.ac（旧版称 configure.in）是 Autoconf 的输入文件，本质上是嵌入了 M4 宏调用的 shell 脚本模板。Autoconf 将其展开为可独立运行的 configure 脚本。
参考：Autoconf Manual §3.1 Writing configure.ac

标准布局（Standard configure.ac Layout）

Autoconf 手册（§3.1.3）给出了推荐的 configure.ac 编写顺序：

configure.ac
#                                               -*- Autoconf -*-
# Process this file with autoconf to produce a configure script.

# ① 初始化：必须第一个调用
AC_PREREQ([2.69])
AC_INIT([mypackage], [1.0], [bug@example.com])

# ② 可选：指定辅助文件目录和宏目录
AC_CONFIG_AUX_DIR([build-aux])
AC_CONFIG_MACRO_DIRS([m4])

# ③ Automake 初始化
AM_INIT_AUTOMAKE([-Wall -Werror foreign])

# ④ 检查编译器
AC_PROG_CC          # 查找 C 编译器，设置 CC 变量
AC_PROG_CXX         # 查找 C++ 编译器（如需要）
AC_PROG_RANLIB       # 查找 ranlib（构建静态库）

# ⑤ 生成 config.h 头文件（汇集 #define）
AC_CONFIG_HEADERS([config.h])

# ⑥ 检查库文件
AC_CHECK_LIB([m], [sin])   # 检查 libm 中是否有 sin()

# ⑦ 检查头文件
AC_CHECK_HEADERS([stdlib.h string.h])

# ⑧ 检查类型定义和编译器特性
AC_TYPE_SIZE_T

# ⑨ 检查库函数
AC_CHECK_FUNCS([malloc realloc])

# ⑩ 声明要生成的文件（*.in → 去掉 .in 的文件）
AC_CONFIG_FILES([
  Makefile
  src/Makefile
  tests/Makefile
])

# ⑪ 生成输出，必须最后调用
AC_OUTPUT

核心宏详解

AC_INIT — 项目初始化

AC_INIT([PACKAGE], [VERSION], [BUG-REPORT], [TARNAME], [URL])

必须是 configure.ac 中第一个实质性宏调用。它定义包名、版本号等基本元数据，这些信息会作为预设变量（如 PACKAGE_NAME、PACKAGE_VERSION）注入到 config.h 中。

示例（来自 GNU Automake amhello 示例）
AC_INIT([amhello], [1.0], [bug-automake@gnu.org])
来源：Automake Manual §2.4.2

AM_INIT_AUTOMAKE — Automake 初始化

AM_INIT_AUTOMAKE([-Wall -Werror foreign])

告知 Automake 以指定选项工作。-Wall 和 -Werror 让 Automake 把所有警告报告为错误（注意：这是 Automake 自身的警告，与 C 编译器的 -Wall 无关）。foreign 选项表明此包不遵循 GNU 编码标准，Automake 不会因缺少 NEWS、AUTHORS、ChangeLog 等文件而报错。

AC_PROG_CC — 编译器检测

AC_PROG_CC              # 使用默认搜索顺序
AC_PROG_CC([gcc clang cc])  # 指定搜索顺序

根据 Autoconf Manual §5.10.3：如果环境变量 CC 未设置，依次检查 gcc、cc 及其他 C 编译器，将找到的编译器名称设置到输出变量 CC 中。如果使用 GCC，则将 CFLAGS 默认设为 -g -O2。

AC_CONFIG_HEADERS — 配置头文件

AC_CONFIG_HEADERS([config.h])

指示 configure 脚本创建 config.h（从模板 config.h.in 生成），将各项系统能力探测的结果汇集为 #define。源码通过 #include "config.h" 访问这些信息。根据 Automake Manual §2.4.2： 

/* config.h 内容示例 */
#define PACKAGE_BUGREPORT "bug-automake@gnu.org"
#define PACKAGE_STRING    "amhello 1.0"
#define PACKAGE_VERSION   "1.0"
#define HAVE_STDLIB_H     1

AC_CONFIG_FILES — 声明输出文件

AC_CONFIG_FILES([Makefile src/Makefile tests/Makefile])

声明 configure 应从 *.in 模板生成的文件列表。根据 Automake Manual §2.4.2：Automake 也会扫描此列表，找到所有需要处理的 Makefile.am 文件。当你向项目添加新目录时，务必在此处添加对应的 Makefile，否则 Automake 永远不会处理新目录下的 Makefile.am。

可选特性与可选包

# --enable/--disable 开关
AC_ARG_ENABLE([myfeature],
  AS_HELP_STRING([--enable-myfeature], [启用 myfeature 支持]),
  [enable_myfeature=${enableval}], [enable_myfeature=yes])

if test "x${enable_myfeature}" = "xyes"; then
  AC_DEFINE([MYFEATURE], [1], [myfeature 已启用])
fi

# --with/--without 开关
AC_ARG_WITH([zlib],
  AS_HELP_STRING([--with-zlib=PATH], [zlib 路径]),
  [with_zlib=${withval}], [with_zlib=yes])

📋 本章总结

configure.ac 是整个构建系统的"需求规格说明书"——它描述软件需要什么（编译器、库、头文件、特性），而不描述如何构建。从第一性原理看：在运行时检测能力，而非在源码中假设能力。每个 AC_CHECK_* 调用实际上编译并运行一个测试程序，将结果缓存到 config.cache 以加速后续运行。

AC_INIT项目元数据初始化
AM_INIT_AUTOMAKEAutomake 选项配置
AC_PROG_*编译器和工具检测
AC_CHECK_*库/头文件/函数探测
AC_CONFIG_FILES声明输出文件列表
AC_OUTPUT触发文件生成
📝 核心文件 · 第 3 章

Makefile.am 详解

Makefile.am 是 Automake 的输入文件，本质上是 make 变量定义的集合（有时附带规则）。Automake 读取它并生成符合 GNU Makefile 标准的 Makefile.in，再由 configure 最终生成 Makefile。
参考：Automake Manual §3 General ideas

统一命名方案（Uniform Naming Scheme）

根据 Automake Manual §3.3，变量名由两部分构成：

PREFIX_PRIMARY
# PREFIX：安装目录前缀（bin、lib、include、data...）
# PRIMARY：对象类型（PROGRAMS、LIBRARIES、HEADERS、SCRIPTS、DATA...）
PRIMARY 类型含义示例
PROGRAMS可执行程序bin_PROGRAMS = hello
LIBRARIES静态库lib_LIBRARIES = libfoo.a
LTLIBRARIESLibtool 库（静态或共享）lib_LTLIBRARIES = libfoo.la
HEADERS头文件（安装时）include_HEADERS = foo.h
SCRIPTS可执行脚本bin_SCRIPTS = myscript.sh
DATA数据文件data_DATA = config.dat
MANSman 手册页man_MANS = foo.1

完整 Makefile.am 示例

顶层 Makefile.am

Makefile.am (顶层)
# 指定递归进入的子目录（顺序重要：先构建被依赖的目录）
SUBDIRS = src tests

# 额外分发的文件（不在默认分发列表中）
EXTRA_DIST = README.md autogen.sh

src/Makefile.am — 构建可执行程序

src/Makefile.am
# 要安装到 $(bindir) 的程序列表
bin_PROGRAMS = hello

# hello 程序的源文件
hello_SOURCES = main.c util.c util.h

# hello 程序的额外编译标志
hello_CFLAGS = $(AM_CFLAGS) -DLOCAL_BUILD

# hello 程序链接时附加的库
hello_LDADD = -lm

# 安装到 $(includedir) 的公共头文件
include_HEADERS = myapi.h

# 不安装（noinst）的私有库
noinst_LIBRARIES = libinternal.a
libinternal_a_SOURCES = internal.c internal.h
💡 变量名中的非字母字符规则
根据 Automake Manual §3.5（Canonicalization）：变量名中不允许出现非字母数字字符。所有非字母数字字符（如 .、-、/）都要替换成下划线 _。例如：库 libmylib.a 的源文件变量名为 libmylib_a_SOURCES。

src/Makefile.am — 构建库（含多子目录示例）

src/Makefile.am（库项目）
# 安装到 $(libdir) 的静态库
lib_LIBRARIES = libmylib.a

# 非字母字符用下划线替换：libmylib.a → libmylib_a
libmylib_a_SOURCES = mylib.c myclass.cpp
libmylib_a_CPPFLAGS = -I$(top_srcdir)/include

# 提供给用户的公共头文件
pkginclude_HEADERS = mylib.h  # 安装到 $(includedir)/$(PACKAGE)/

编译标志变量

根据 Automake Manual §26.5 Flag Variables Ordering，编译标志分为三层：

变量设置者说明
CFLAGS用户用户保留变量，Makefile.am 不应覆盖
AM_CFLAGSMakefile.am 维护者包级别的编译标志
target_CFLAGSMakefile.am 维护者针对特定目标的编译标志
# 正确做法：使用 AM_CFLAGS 而非直接设置 CFLAGS
AM_CFLAGS = -Wall -Wextra -std=c99
AM_CXXFLAGS = -std=c++17
AM_LDFLAGS = -L$(top_builddir)/lib

# 错误做法（会覆盖用户的 CFLAGS）：
# CFLAGS = -Wall  ← 不要这样做

标准 make 目标

根据 Automake Manual §2.2.2 Standard Makefile Targets，Automake 自动生成以下标准目标：

目标说明
make all构建所有目标（默认目标）
make install安装程序和数据文件
make install-strip安装并去除调试符号
make uninstall卸载已安装的文件
make clean删除构建产物（.o、可执行文件等）
make distclean还原到 configure 之前的状态
make maintainer-clean还原到 autoreconf 之前的状态（慎用）
make dist创建源码发布包（.tar.gz 等）
make distcheck创建并验证发布包（推荐发布前运行）
make check运行测试套件

📋 本章总结

Makefile.am 通过统一命名方案（PREFIX_PRIMARY）声明"什么文件安装到哪里"，而无需手写复杂的 Makefile 规则。从第一性原理看：Automake 是 GNU Makefile 标准的规则编译器——它将高层意图（"这是要安装到 bin 目录的程序"）翻译成底层 make 规则（编译命令、依赖跟踪、安装路径等）。

🔨 构建流程 · 第 4 章

开发者侧：生成构建系统

开发者需要先"自举"（bootstrap）构建系统，即从 configure.ac 和 Makefile.am 生成 configure、Makefile.in 等文件。
参考：Automake Manual §2.4.1

使用 autoreconf（推荐方式）

# 一条命令完成所有操作（推荐）
autoreconf --install

# --install (-i)：将 Autotools 所需的辅助文件（install-sh 等）复制到项目中
# --force  (-f)：强制重新生成所有文件（修改了 configure.ac 后使用）
# --verbose(-v)：显示执行的每条命令
autoreconf --install --force --verbose

根据 Automake Manual §2.4.1，运行 autoreconf --install 后，会看到类似输出：

configure.ac: installing './install-sh'
configure.ac: installing './missing'
configure.ac: installing './compile'
src/Makefile.am: installing './depcomp'

autoreconf 内部执行顺序

1
aclocal — 收集 M4 宏
扫描 configure.ac，从系统宏库和 m4/ 目录收集所需的 M4 宏，生成 aclocal.m4。
2
libtoolize（如需要）— 添加 Libtool 支持
如果 configure.ac 包含 LT_INIT，则复制 ltmain.sh 和 Libtool M4 宏文件到 build-aux/ 和 m4/。
3
autoheader — 生成 config.h.in
分析 configure.ac 中的 AC_DEFINE 等宏调用，生成 config.h.in 模板文件。
4
automake — 生成 Makefile.in
读取所有 Makefile.am（通过 configure.ac 中的 AC_CONFIG_FILES 确定），为每个目录生成对应的 Makefile.in。
5
autoconf — 生成 configure
将 configure.ac 与 aclocal.m4 一起通过 M4 宏处理器展开，生成最终的可执行 configure shell 脚本。
图 3：autoreconf 内部执行流程与文件依赖关系

手动分步执行（深入理解）

# 等价于 autoreconf --install 的手动步骤：
aclocal                    # 生成 aclocal.m4
libtoolize --copy          # （如用 libtool）复制 ltmain.sh 等
autoheader                 # 生成 config.h.in
automake --add-missing     # 生成 Makefile.in，复制缺失的辅助文件
autoconf                   # 生成 configure
⚠️ 注意顺序
各工具调用有严格的顺序依赖：aclocal 必须在 autoconf 之前（生成 aclocal.m4），autoheader 必须在 autoconf 之前（生成 config.h.in）。autoreconf 会自动处理这些依赖，因此通常应优先使用 autoreconf。

📋 本章总结

开发者侧的工作是一次性的"自举"操作，将高级声明（configure.ac、Makefile.am）转化为可分发的构建系统。autoreconf --install 是推荐的单一入口点，它按正确顺序调用所有底层工具。从第一性原理看：这是一个元编程过程——用工具生成工具（用 Autotools 生成构建系统）。

🚀 构建流程 · 第 5 章

用户侧：配置与编译

用户收到源码包（包含 configure 脚本和 Makefile.in 文件）后，只需运行以下命令即可构建和安装软件。
参考：Autoconf Manual §16.1 Basic Installation、Automake Manual §2.2.1

标准安装流程

# 第一步：解压源码包
tar xzf mypackage-1.0.tar.gz
cd mypackage-1.0

# 第二步：配置（探测系统并生成 Makefile）
./configure

# 第三步：编译
make

# 第四步：安装（通常需要 root 权限）
make install

./configure 详解

configure 脚本是一个独立的 shell 脚本，执行一系列系统能力探测，最终生成适合当前系统的 Makefile 和 config.h。

图 4：./configure 执行流程与输出

configure 常用选项

以下选项摘自 Autoconf Manual §16.10 及 Automake Manual §2.2 的标准选项描述：

# 安装前缀（默认 /usr/local）
./configure --prefix=/usr
./configure --prefix=$HOME/.local    # 安装到用户目录（无需 root）

# 精细调整安装位置
./configure --bindir=/opt/myapp/bin
./configure --libdir=/opt/myapp/lib
./configure --includedir=/opt/myapp/include

# 指定编译器
CC=clang ./configure
CXX=g++-12 ./configure

# 启用/禁用可选功能
./configure --enable-debug
./configure --disable-nls          # 禁用国际化

# 指定外部包路径
./configure --with-zlib=/usr/local
./configure --without-openssl

# 查看所有可用选项
./configure --help

# 使用缓存加速（-C 是 --config-cache 的简写）
./configure -C

环境变量控制

环境变量说明示例
CCC 编译器CC=clang ./configure
CXXC++ 编译器CXX=g++ ./configure
CFLAGSC 编译选项CFLAGS="-O0 -g" ./configure
CXXFLAGSC++ 编译选项CXXFLAGS="-std=c++17" ./configure
LDFLAGS链接器选项LDFLAGS="-L/usr/local/lib" ./configure
CPPFLAGS预处理器选项CPPFLAGS="-I/usr/local/include" ./configure

并行构建

# 使用多核并行编译（-j 后跟核心数，推荐）
make -j$(nproc)          # Linux：自动获取 CPU 核心数
make -j4                 # 指定 4 个并行任务

# 安装到临时目录（用于打包 RPM/DEB 等）
make DESTDIR=/tmp/staging install
💡 DESTDIR 暂存安装
根据 Automake Manual §2.2.10：DESTDIR 变量将所有安装路径的根临时改为指定目录，而不修改 --prefix。这对打包二进制发布包非常有用——可以先"安装"到临时目录，然后打包该目录。

📋 本章总结

用户侧的 ./configure && make && make install 三步流程是整个 GNU Build System 设计的核心价值体现。configure 通过实际编译和运行测试程序来检测系统能力（而非靠平台判断），生成与系统完全匹配的 Makefile。从第一性原理看：能力检测优于平台检测——检测"是否有 sin()"比检测"是否是 Linux"更可靠，因为能力可直接验证。

📂 进阶专题 · 第 6 章

标准安装目录变量

GNU Coding Standards 定义了一套标准安装目录变量。Automake 和 Autoconf 完整支持这套体系，确保软件按约定安装到正确位置。
参考：Automake Manual §2.2.3 Standard Directory Variables

目录变量默认值用途
prefix/usr/local安装根目录（所有其他目录的基准）
exec_prefix${prefix}与体系结构相关文件的根目录
bindir${exec_prefix}/bin用户可执行程序
sbindir${exec_prefix}/sbin系统管理员可执行程序
libexecdir${exec_prefix}/libexec由其他程序调用的可执行程序
libdir${exec_prefix}/lib库文件
includedir${prefix}/includeC/C++ 头文件
datarootdir${prefix}/share只读体系结构无关数据根目录
datadir${datarootdir}只读体系结构无关数据文件
mandir${datarootdir}/manman 手册页
infodir${datarootdir}/infoinfo 文档
docdir${datarootdir}/doc/${PACKAGE}包文档
sysconfdir${prefix}/etc机器相关的配置文件
localstatedir${prefix}/var程序运行时修改的数据
pkglibdir${libdir}/@PACKAGE@包私有库目录（Automake 扩展）
pkgincludedir${includedir}/@PACKAGE@包私有头文件目录（Automake 扩展）
pkgdatadir${datadir}/@PACKAGE@包私有数据目录（Automake 扩展）
图 5：GNU 标准安装目录层次结构与 Makefile.am 前缀对应关系

特殊前缀

前缀含义
noinst_构建但不安装（内部使用的库、测试辅助程序等）
check_仅在 make check 时构建和使用
dist_包含在 make dist 的发布包中
nodist_不包含在发布包中（如生成的文件）
nobase_安装时保留相对目录结构（如 nobase_include_HEADERS）
EXTRA_声明可能被构建的额外对象（configure 决定是否实际构建）

📋 本章总结

GNU 标准安装目录体系是 Unix FHS（文件系统层次标准）在构建系统层面的抽象。通过 --prefix 一个选项即可迁移整个安装根，这是因为所有目录都基于 prefix 派生。从第一性原理看：用变量代替硬编码路径——这让同一个软件包可以安装到 /usr、/usr/local、$HOME/.local 等任意位置而无需修改任何源码。

🔗 进阶专题 · 第 7 章

Libtool 共享库支持

GNU Libtool 是 Autotools 生态中负责构建可移植共享库的工具。不同操作系统的共享库机制差异极大（Linux 的 .so、macOS 的 .dylib、Windows 的 .dll），Libtool 将这些差异封装起来，提供统一的接口。
参考：Libtool Manual §5 Integrating libtool with your package

在项目中集成 Libtool

configure.ac 中的必要声明

AC_CONFIG_MACRO_DIRS([m4])   # 必须：指定 M4 宏目录
AM_PROG_AR                   # 必须：在 LT_INIT 之前
LT_INIT                      # 初始化 Libtool（替代旧的 AC_PROG_LIBTOOL）

# LT_INIT 添加 --enable-shared / --disable-shared 配置选项
# 默认：共享库 ON，静态库 ON（如不冲突）

Makefile.am 中使用 LTLIBRARIES

# 使用 lib_LTLIBRARIES 而非 lib_LIBRARIES
lib_LTLIBRARIES = libfoo.la     # .la 是 Libtool 库描述文件

libfoo_la_SOURCES = foo.c bar.c
libfoo_la_CFLAGS  = -fPIC       # 位置无关代码（通常 Libtool 自动处理）

# Libtool 版本信息（current:revision:age）
libfoo_la_LDFLAGS = -version-info 3:2:1

Libtool 版本信息

-version-info current:revision:age 控制共享库的 ABI 版本：

参数含义何时更新
current当前接口版本号接口发生任何变化时 +1
revision当前接口的修订号仅实现改变（接口不变）时 +1；接口变化时重置为 0
age向后兼容的旧接口数量新增接口（向后兼容）时 +1；删除接口时重置为 0

📋 本章总结

Libtool 通过 .la 描述文件和运行时 libtool 封装脚本，将各平台共享库构建的差异封装为统一接口。在 Makefile.am 中将 lib_LIBRARIES 改为 lib_LTLIBRARIES，在 configure.ac 中添加 LT_INIT，即可获得跨平台共享库支持。从第一性原理看：在工具层面解决可移植性问题，而非让每个软件包重新发明各平台的共享库构建逻辑。

🏗️ 进阶专题 · 第 8 章

VPATH 构建与分离目录

VPATH（Virtual PATH）构建允许将源码目录和构建目录分开，避免构建产物污染源码树。这在同时为多个平台构建时特别有用。
参考：Automake Manual §2.2.6 Parallel Build Trees

分离构建目录

# 在源码目录之外创建构建目录
mkdir build-linux
cd build-linux

# 调用源码目录中的 configure（使用相对或绝对路径）
../configure --prefix=/usr/local

# 构建（所有 .o、可执行文件生成在 build-linux/ 中）
make
图 7：VPATH 构建——源码树与构建树完全分离

递归子目录

根据 Automake Manual §7.1，在多目录项目中，SUBDIRS 变量控制 make 递归进入哪些子目录：

# 顶层 Makefile.am
SUBDIRS = lib src tests   # 顺序重要：lib 先于 src（src 依赖 lib）

# DIST_SUBDIRS：用于 make dist 的目录列表（比 SUBDIRS 更广）
DIST_SUBDIRS = lib src tests docs   # docs 不参与构建但要分发
📌 SUBDIRS vs DIST_SUBDIRS
SUBDIRS 决定 make all/make install 递归的目录；DIST_SUBDIRS 决定 make dist 递归的目录。DIST_SUBDIRS 应包含所有需要分发的目录，即使某些目录在当前配置下不参与构建。

自动依赖跟踪

根据 Automake Manual §2.2.12，Automake 会自动启用依赖跟踪：编译 .c 文件时，同时生成 .deps/ 目录下的 .Po 文件，记录头文件依赖关系。当头文件更改时，make 会自动重新编译依赖它的源文件。

# .deps/ 目录内容示例
src/.deps/main.Po:
  src/main.o: src/main.c include/myapi.h config.h /usr/include/stdio.h ...
  # 当任何头文件更新时，自动触发重新编译

📋 本章总结

VPATH 构建是 Autotools 的高级特性，让源码目录保持只读纯净，所有构建产物隔离在单独目录中。这支持了同时为多个目标平台构建（交叉编译）、CI/CD 中的并行构建等场景。从第一性原理看：输入（源码）与输出（构建产物）应该分离——这是一个普适的工程原则，确保构建过程的可重复性和幂等性。

🎯 完整示例 · 第 9 章

从零搭建完整示例

本章通过一个完整的示例演示从零开始创建一个 Autotools 项目的全过程，该示例基于 GNU Automake 官方手册的 amhello 示例（§2.4）扩展而来，包含一个可执行程序和一个库。

项目结构目标

myproject/
├── configure.ac
├── Makefile.am
├── build-aux/ # AC_CONFIG_AUX_DIR 指定
├── m4/ # AC_CONFIG_MACRO_DIRS 指定
├── src/
│ ├── Makefile.am
│ ├── main.c
│ ├── greeting.c
│ └── greeting.h
└── tests/
├── Makefile.am
└── test_greeting.c

第一步：手写源文件

src/greeting.h

src/greeting.h
#ifndef GREETING_H
#define GREETING_H

void greet(const char *name);

#endif /* GREETING_H */

src/greeting.c

src/greeting.c
#include "config.h"    /* 使用 configure 生成的 config.h */
#include "greeting.h"
#include <stdio.h>

void greet(const char *name) {
    printf("Hello, %s! (from %s)\n", name, PACKAGE_STRING);
}

src/main.c

src/main.c
#include "config.h"
#include "greeting.h"

int main(int argc, char *argv[]) {
    greet("World");
    return 0;
}

第二步：编写 configure.ac

configure.ac
#                                               -*- Autoconf -*-
# Process this file with autoconf to produce a configure script.

# 必须最低版本（对应系统 autoconf 版本）
AC_PREREQ([2.69])

# 项目元数据：包名、版本、bug 报告地址
AC_INIT([myproject], [1.0], [me@example.com])

# 辅助文件目录（install-sh、config.guess 等）
AC_CONFIG_AUX_DIR([build-aux])

# 自定义 M4 宏目录
AC_CONFIG_MACRO_DIRS([m4])

# Automake 初始化
# -Wall: Automake 警告全开
# -Werror: 警告视为错误
# foreign: 不强制要求 NEWS/AUTHORS/ChangeLog/COPYING
AM_INIT_AUTOMAKE([-Wall -Werror foreign subdir-objects])

# 检测 C 编译器
AC_PROG_CC

# 生成 config.h（聚合所有 #define）
AC_CONFIG_HEADERS([config.h])

# 检查标准头文件
AC_CHECK_HEADERS([stdlib.h stdio.h])

# 声明要生成的 Makefile（Automake 也通过此列表找到 Makefile.am）
AC_CONFIG_FILES([
  Makefile
  src/Makefile
  tests/Makefile
])

# 执行输出（必须最后调用）
AC_OUTPUT

第三步：编写 Makefile.am

顶层 Makefile.am

Makefile.am（顶层）
# 递归进入的子目录（先 src，再 tests）
SUBDIRS = src tests

src/Makefile.am

src/Makefile.am
# 安装到 $(bindir) 的可执行程序
bin_PROGRAMS = myproject

# myproject 的源文件（main 和 greeting 模块）
myproject_SOURCES = main.c greeting.c greeting.h

# 编译时传入 include 路径（包含 config.h 所在的 top_builddir）
myproject_CPPFLAGS = -I$(top_builddir)

tests/Makefile.am

tests/Makefile.am
# 仅用于 make check 的程序（不安装）
check_PROGRAMS = test_greeting

test_greeting_SOURCES = test_greeting.c ../src/greeting.c
test_greeting_CPPFLAGS = -I$(top_srcdir)/src -I$(top_builddir)

# 注册到 make check 的测试列表
TESTS = test_greeting

第四步：运行 autoreconf

# 在项目根目录运行：
autoreconf --install

# 预期输出（类似）：
# configure.ac: installing './build-aux/install-sh'
# configure.ac: installing './build-aux/missing'
# configure.ac: installing './build-aux/compile'
# src/Makefile.am: installing './build-aux/depcomp'

第五步：配置、编译、安装

# 使用分离构建目录（推荐）
mkdir _build && cd _build

# 配置（安装到 $HOME/.local）
../configure --prefix=$HOME/.local

# 编译（利用多核）
make -j$(nproc)

# 运行测试
make check

# 安装
make install

# 验证安装
$HOME/.local/bin/myproject
# 输出: Hello, World! (from myproject 1.0)

第六步：创建发布包

# 创建源码发布包（默认 .tar.gz）
make dist
# 生成: myproject-1.0.tar.gz

# 创建并验证发布包（强烈推荐发布前运行）
make distcheck
# distcheck 会：解压 → configure → make → make check → make install → make uninstall
# 如果全部通过，说明发布包完整可用

📋 本章总结 · 全局第一性原理回顾

GNU Autotools 的设计遵循几个核心第一性原理：

能力检测 > 平台检测 检测"有 sin()"比检测"是 Linux"更可靠、更可移植
意图与实现分离 configure.ac/Makefile.am 描述意图，工具推导实现细节
开发者负责，用户受益 复杂性由 Autotools 承担，用户仅需三条命令
变量代替硬编码 所有路径通过变量抽象，一个 --prefix 改变安装根
元编程 用工具生成工具（autoreconf 生成 configure 和 Makefile.in）
输入输出分离 VPATH 构建保持源码树纯净，支持多目标并行构建
