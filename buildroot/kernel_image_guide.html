第一章

概览与背景

Linux 内核编译后会产生多种格式的输出产物。理解它们的差异，是嵌入式开发、发行版打包和系统运维的基础。

📄 官方文档：kernel.org/doc/makehelp.txt 📄 Kbuild Makefiles 文档

🏗️为什么存在多种格式？

Linux 内核从源代码到可运行的系统，需要经历：编译链接 → 生成原始 ELF 内核 → 根据目标平台和使用场景， 转换为特定格式的启动镜像或安装包。多种格式的存在，源自以下几个根本需求：

🏛️ 历史演进

x86 平台从早期 zImage（低内存 512K 限制）演进到 bzImage（高内存），格式本身携带着历史痕迹。

🔌 硬件差异

ARM、ARM64、x86、MIPS 等架构的 bootloader 能力各不相同，需要不同格式与之配合。

📦 发行版需求

不同 Linux 发行版（Debian/Ubuntu、RHEL/Fedora、Arch）有各自的包管理体系，需要对应的打包格式。

🗜️ 存储与传输

嵌入式设备存储空间有限，需要压缩格式；而调试场景需要未压缩的符号信息。

ℹ️

根据官方内核文档（make help）， 内核打包目标分为两大类：启动镜像格式（由各架构 Makefile 定义，与 bootloader 直接交互） 和发行版/归档打包格式（由 scripts/Makefile.package 统一管理）。

📋 本章小结

  • Linux 内核格式多样性是历史演进、硬件差异、发行版生态共同作用的结果
  • 所有格式最终都源自同一个 vmlinux ELF 文件
  • 不同格式服务于不同的链条节点：bootloader、包管理器、部署工具
第二章

编译过程与输出产物

了解内核从源码到最终镜像的完整链路，是理解各种格式存在意义的前提。

📄 Kbuild Makefiles — The Linux Kernel documentation

⚙️构建流水线

根据官方 Kbuild 文档，顶层 Makefile 负责构建两个主要产物：vmlinux（常驻内核镜像）和 modules（可加载模块）。之后由各架构 Makefile 完成平台特定的后处理， 生成可引导的镜像文件。

图1：Linux 内核编译产物关系图（基于官方 Kbuild 文档）

📋核心构建步骤

根据官方 Kbuild Makefiles 文档，内核构建遵循以下步骤：

  1. 配置内核 → 生成 .config
  2. 递归编译所有子目录中的 .o 文件
  3. 使用 $(AR) rcSTP 将目标文件合并为 built-in.a
  4. 通过 scripts/link-vmlinux.sh 链接生成 vmlinux（ELF 格式，静态链接，未压缩）
  5. 各架构 Makefile 调用 objcopy、gzip 等工具对 vmlinux 进行处理，生成可引导镜像
  6. （可选）调用 scripts/Makefile.package 中的打包目标生成发行版包
# x86 架构的 bzImage 构建规则（来自 arch/x86/Makefile）
boot := arch/x86/boot
bzImage: vmlinux
        $(Q)$(MAKE) $(build)=$(boot) $(boot)/$@

# 执行命令示例
$ make bzImage              # x86 下默认目标
$ make ARCH=arm64 Image.gz  # ARM64 压缩镜像
$ make bindeb-pkg           # 生成 Debian 二进制包
$ make binrpm-pkg           # 生成 RPM 二进制包

📋 本章小结

  • vmlinux 是所有格式的唯一源头，由 Kbuild 的顶层 Makefile 生成
  • 平台相关镜像（bzImage/Image 等）由各架构的 Makefile 在 arch/$(ARCH)/boot/ 中生成
  • 发行版打包格式由 scripts/Makefile.package 统一管理
  • 从第一性原理看：不同阶段的消费者（bootloader vs 包管理器）决定了格式需求
第三章

格式分类总览

Linux 内核打包格式按用途可分为两大类：启动镜像格式 和 发行版/归档打包格式。

📄 官方参考：make help 输出 (kernel.org)

🗂️两大类别

🖥️ 启动镜像格式

直接由 bootloader 加载、与硬件交互的内核镜像。格式由目标架构决定。

  • vmlinux — ELF 原始内核
  • Image — 二进制裸镜像（ARM/ARM64）
  • zImage — 低内存压缩镜像（x86/ARM 旧版）
  • bzImage — 大内核压缩镜像（x86 标准）
  • vmlinuz — 压缩镜像通用别名
  • uImage — U-Boot 专用格式
  • Image.gz / Image.bz2 等

📦 发行版/归档打包格式

用于跨机器分发、安装管理的打包格式，由 make <target> 生成。

  • deb-pkg / bindeb-pkg — Debian/Ubuntu
  • rpm-pkg / binrpm-pkg — RHEL/Fedora/SUSE
  • snap-pkg — Ubuntu Core Snap
  • pacman-pkg — Arch Linux
  • tar-pkg / targz-pkg 等 — 通用 Tarball
  • dir-pkg — 目录结构
  • modules-cpio-pkg — 模块 CPIO 归档

📊快速对比：启动镜像格式

格式 目标架构 是否压缩 文件格式 加载内存位置 make 命令
vmlinux 全架构 ❌ 否 ELF N/A（需 bootloader 支持） make vmlinux
Image ARM / ARM64 ❌ 否 Raw Binary text_offset 偏移处 make Image
zImage x86 / ARM ✅ gzip 自解压二进制 低内存（≤640KB） make zImage
bzImage x86 / x86_64 ✅ gzip 等 自解压二进制 高内存（≥1MB） make bzImage
Image.gz ARM64 ✅ gzip 外部解压二进制 由 bootloader 处理 make Image.gz
uImage ARM（嵌入式） ✅ 可选 zImage + 64字节头 由头部指定 make uImage

📋 本章小结

  • 两大分类的消费者完全不同：启动镜像 → bootloader；打包格式 → 包管理器
  • 启动镜像格式由目标硬件架构决定，不可随意替换
  • 发行版打包格式由目标 Linux 发行版决定，与硬件无关
第四章 · 启动镜像格式

vmlinux — ELF 原始内核镜像

所有内核格式的起点，未压缩的完整内核 ELF 文件，包含调试符号。

📄 Kbuild Makefiles Documentation
vmlinux
ELF 格式静态链接内核
所有架构 · 路径：内核源码根目录/vmlinux
未压缩 含调试符号 ELF 格式 全架构

根据 Kbuild 官方文档，vmlinux 是顶层 Makefile 负责构建的两个主要产物之一 （另一个是 modules）。它是一个静态链接的、非剥离的 ELF 格式可执行文件， 包含了完整的内核代码、数据段和调试符号信息。

生成原理

所有 obj-y 列表中的 .o 文件通过 $(AR) rcSTP 合并为 built-in.a，再由 scripts/link-vmlinux.sh 将各架构的 built-in.a、lib.a 等链接为最终的 vmlinux。 

# 简化的链接过程
$ make vmlinux
  LD      vmlinux
  SYSMAP  System.map
# 输出：源码根目录下的 vmlinux（ELF 文件，通常数百 MB）
$ file vmlinux
vmlinux: ELF 64-bit LSB executable, x86-64, statically linked, not stripped

适用场景

  • 🔍 内核调试：GDB、KGDB 等调试工具需要未压缩的 ELF 文件及其符号表
  • 📊 符号分析：nm vmlinux 生成 System.map 符号表
  • 🔧 crash 分析：使用 crash 工具分析 core dump 时需要对应的 vmlinux
  • ⚡ eBPF/BTF：从 vmlinux 提取 BTF 类型信息用于 CO-RE 技术
⚠️

不能直接用于引导：vmlinux 是 ELF 格式，大多数 bootloader（如 GRUB）期望特定的启动镜像格式（bzImage/Image 等），无法直接加载 ELF vmlinux。需要先将其转换为平台对应的启动镜像。

第五章 · 启动镜像格式

Image — ARM/ARM64 裸二进制镜像

ARM 和 ARM64 架构最基础的可引导格式，通过 objcopy 从 vmlinux 提取纯二进制内核代码。

📄 Booting AArch64 Linux — kernel.org
Image
ARM/ARM64 裸二进制可引导内核
ARM / ARM64 架构 · 路径：arch/arm64/boot/Image
未压缩 Raw Binary ARM64 标准

Image 是通过 objcopy -O binary 从 vmlinux 提取的纯二进制文件， 去除了 ELF 头部、符号表和重定位信息，只保留实际可执行的内核数据。 根据官方 ARM64 引导文档，Image 文件包含一个 64 字节的头部结构： 

/* ARM64 Image 头部结构（来自官方内核文档 arm64/booting.txt）*/
u32 code0;          /* 可执行代码（跳转到 stext） */
u32 code1;          /* 可执行代码 */
u64 text_offset;    /* 镜像加载偏移量（小端序）*/
u64 image_size;     /* 有效镜像大小（小端序）*/
u64 flags;          /* 内核标志（小端序）*/
u64 res2 = 0;       /* 保留 */
u64 res3 = 0;       /* 保留 */
u64 res4 = 0;       /* 保留 */
u32 magic = 0x644d5241; /* 魔数："ARM\x64"（小端序）*/
u32 res5;           /* 保留（PE/COFF 偏移）*/

生成原理

# arch/s390/Makefile 中的典型规则（Kbuild 文档示例）
OBJCOPYFLAGS := -O binary
$(obj)/image: vmlinux FORCE
        $(call if_changed,objcopy)

# 生成 Image
$ make ARCH=arm64 Image
  OBJCOPY arch/arm64/boot/Image

加载要求（来自官方文档）

ℹ️

根据 arm64/booting.txt： Image 必须放置在距 2MB 对齐基地址 text_offset 字节处的系统 RAM 中。 对于不实现解压的 bootloader，应使用未压缩的 Image 而非 Image.gz。

适用场景

  • ARM64 服务器（UEFI/GRUB 环境，bootloader 可直接加载）
  • 存储空间充足且 bootloader 不支持解压的场景
  • 作为生成 Image.gz、Image.bz2 等压缩格式的源文件
第六章 · 启动镜像格式

zImage — 低内存压缩内核镜像

历史上 x86 和 ARM 的标准压缩格式，受限于 640KB 低内存，已逐渐被 bzImage 取代。

📄 Linux Kernel 2.4 Internals: Booting — TLDP 📄 The Linux/x86 Boot Protocol — kernel.org
zImage
低内存自解压压缩内核（已过时）
x86 / ARM 旧架构 · 路径：arch/x86/boot/zImage
gzip 压缩 低内存加载（≤640KB） 已过时 自解压

zImage 是将 vmlinux 进行 gzip 压缩，并在前面附加一段自解压代码（decompression stub） 的可引导格式。文件头包含实模式代码（bootsect + setup），主体是压缩后的内核数据。

生成原理（来自 TLDP 官方文档）

根据 Linux Kernel 2.4 Internals 官方文档，zImage 的构建过程如下：

# 1. C/汇编源文件编译为 .o，链接为 ELF vmlinux
# 2. 进入 arch/i386/boot/compressed 目录
#    - 将 vmlinux 转为裸二进制（去掉 .note/.comment ELF 节）
#    - 对裸二进制 gzip 压缩 → tmppiggy.gz
#    - 链接 tmppiggy.gz 为 ELF 可重定位目标 piggy.o
#    - 编译 head.S + misc.c（解压代码）为 head.o、misc.o
#    - 链接 head.o + misc.o + piggy.o → vmlinux（临时，加载地址 0x1000）
# 3. 拼接：bbootsect + bsetup + compressed/vmlinux.out → zImage
$ make zImage
# 输出：arch/x86/boot/zImage

关键限制

🚫

根据 官方 x86 引导协议文档： zImage 将受保护模式内核解压到低内存（0x10000 起，最大 512KB）， 使用 0x90000+ 的内存段用于实模式代码。 现代内核体积远超 512KB，zImage 已在 Protocol 2.02 中被标记为过时（deprecated）， 并在随后的版本中不再推荐使用。

图2：zImage x86 内存布局（解压到低内存）

📋 本章小结

  • zImage 的本质是：[boot sector + setup] + [自解压代码 + gzip(vmlinux)]
  • 其 512KB 限制源于 x86 实模式内存模型（1MB 以内，BIOS 区域占据大部分）
  • 现代内核体积动辄数十 MB，zImage 已无法使用于正式生产环境
  • 第一性原理：格式的限制来自 x86 硬件内存分区的历史约束
第七章 · 启动镜像格式

bzImage — x86 标准大内核压缩镜像

"big zImage" — 将内核解压到高内存（≥1MB），是现代 x86/x86_64 系统的标准格式。

📄 The Linux/x86 Boot Protocol — docs.kernel.org 📄 Kbuild Makefiles — docs.kernel.org
bzImage
Big zImage — x86/x86_64 标准可引导内核
x86 / x86_64 架构 · 路径：arch/x86/boot/bzImage
gzip/xz/bzip2/lzma/lzo/zstd 压缩 高内存加载（≥1MB） x86 默认格式 自解压
💡

命名常见误解：根据官方文档及 TLDP，bzImage 中的 "bz" 不代表 bzip2， 而是 "big zImage"（大 zImage）。bzImage 和 zImage 默认都使用 gzip 压缩， 区别在于解压到内存的位置不同。

与 zImage 的核心区别（来自官方文档）

根据 官方 x86 引导协议文档： 当使用 bzImage 时，受保护模式内核被重定位到 0x100000（高内存）。 判断逻辑如下： 

/* 来自 x86 引导协议官方文档 */
is_bzImage = (protocol >= 0x0200) && (loadflags & 0x01);
load_address = is_bzImage ? 0x100000 : 0x10000;

/* Image/zImage 最大 512KB：使用 0x10000 ~ 0x90000 范围
   bzImage 可以远超此限制：加载到 1MB 以上高内存 */

生成原理（来自 TLDP 与 Kbuild 文档）

# arch/x86/Makefile（Kbuild 文档示例）
boot := arch/x86/boot
bzImage: vmlinux
        $(Q)$(MAKE) $(build)=$(boot) $(boot)/$@

# 内部流程（arch/i386/boot/compressed/）：
# 1. vmlinux → 裸二进制（objcopy 去除 ELF 节）
# 2. 裸二进制 gzip 压缩 → piggy.gz
# 3. 链接 head.S + misc.c + piggy.gz → bvmlinux（-Ttext 0x100000）
# 4. 拼接 bbootsect + bsetup + bvmlinux → bzImage
$ make bzImage     # 等同于 x86 下直接 make
  AS      arch/x86/boot/header.o
  CC      arch/x86/boot/compressed/vmlinux.o
  GZIP    arch/x86/boot/compressed/vmlinux.bin.gz
  LD      arch/x86/boot/compressed/vmlinux
  OBJCOPY arch/x86/boot/vmlinux.bin
  BUILD   arch/x86/boot/bzImage
Kernel: arch/x86/boot/bzImage is ready

bzImage 内部结构

图3：bzImage 文件结构与内存加载位置（基于 x86 引导协议文档）

协议版本历史（官方 x86 引导协议文档）

协议版本内核版本关键变化
旧版<1.3.73仅支持 zImage/Image，zImage 是唯一选择
2.001.3.73添加 bzImage 和 initrd 支持；setup.S 可重定位
2.022.4.0-test3新命令行协议；zImage 被标记为过时

适用场景

  • 🖥️ 所有现代 x86/x86_64 PC 和服务器（BIOS/UEFI 均支持）
  • 📀 x86 下 make 的默认输出目标
  • 🔄 与 GRUB、LILO、syslinux 等主流 bootloader 兼容
  • ☁️ 虚拟机（KVM/QEMU/VMware）环境

📋 本章小结

  • bzImage = zImage + 突破 640KB 低内存限制（解压到 1MB+ 高内存）
  • "bz" ≠ bzip2，是 "big"（大） 的意思，默认仍用 gzip 压缩
  • 现代 x86 系统唯一推荐的启动内核格式，协议版本 2.02 起 zImage 已弃用
  • 第一性原理：高内存解压是为了突破 x86 实模式/保护模式切换时的内存布局限制
第八章 · 启动镜像格式

vmlinuz — 压缩内核的通用别名

大多数 Linux 发行版中 /boot 目录下的内核文件名，通常是 bzImage 或 Image.gz 的符号链接或拷贝。

vmlinuz
压缩内核的标准命名别名
全架构发行版 · 路径：/boot/vmlinuz-$(KERNELRELEASE)
命名惯例 非独立格式 发行版标准

vmlinuz 并非一种独立的内核格式，而是压缩内核镜像的命名惯例。 名称由 vmlinux（虚拟内存 Linux 内核）+ z（表示压缩，最初指 zlib/gzip）组成。

在大多数 Linux 发行版中，/boot/vmlinuz-$(uname -r) 是一个指向实际内核镜像的符号链接或拷贝：

  • 在 x86/x86_64 系统上：指向 bzImage
  • 在 ARM64 系统上：可能指向 Image 或 Image.gz
# 查看实际格式
$ file /boot/vmlinuz-$(uname -r)
vmlinuz-6.8.0-generic: Linux kernel x86 boot executable bzImage,
version 6.8.0-47-generic, RO-rootFS, swap_dev 0xB, ...

# 在 /boot 目录中通常存在
$ ls /boot/
vmlinuz          → vmlinuz-6.8.0-47-generic  (当前内核符号链接)
vmlinuz-6.8.0-47-generic                      (实际 bzImage 文件)
vmlinuz.old      → vmlinuz-6.7.0-41-generic  (旧内核备份)
✅

对于用户而言，vmlinuz 就是"压缩内核"的同义词。在 x86 桌面/服务器上，运行 file vmlinuz 通常会显示它是一个 bzImage 格式的文件。

第九章 · 启动镜像格式

uImage — U-Boot 专用格式

在 zImage/Image 基础上添加 64 字节 U-Boot 头部，专为嵌入式 bootloader U-Boot 设计。

uImage
U-Boot Legacy Image Format
ARM 嵌入式 · U-Boot bootloader 专用
U-Boot 专用 64字节头部 嵌入式设备 Legacy 格式

uImage 是在 zImage（或裸 Image）之前添加一个 64 字节的 U-Boot 专有头部 所形成的格式，使用 U-Boot 自带的 mkimage 工具生成。头部包含内核类型、 加载地址、入口点、时间戳、校验和等元数据，使 U-Boot 能够验证和正确加载内核。

64 字节头部结构（image_header_t）

/* U-Boot image_header_t 结构 */
typedef struct image_header {
    uint32_t ih_magic;   /* Image Header Magic Number (0x27051956) */
    uint32_t ih_hcrc;    /* Image Header CRC Checksum */
    uint32_t ih_time;    /* Image Creation Timestamp */
    uint32_t ih_size;    /* Image Data Size */
    uint32_t ih_load;    /* Data Load Address */
    uint32_t ih_ep;      /* Entry Point Address */
    uint32_t ih_dcrc;    /* Image Data CRC Checksum */
    uint8_t  ih_os;      /* Operating System */
    uint8_t  ih_arch;    /* CPU Architecture */
    uint8_t  ih_type;    /* Image Type */
    uint8_t  ih_comp;    /* Compression Type */
    uint8_t  ih_name[IH_NMLEN]; /* Image Name */
} image_header_t;

生成方式

# 方法1：直接 make uImage
$ make ARCH=arm CROSS_COMPILE=arm-linux-gnueabihf- uImage

# 方法2：使用 mkimage 工具手动包装
$ mkimage -A arm -O linux -T kernel -C gzip \
    -a 0x80008000 -e 0x80008000 \
    -n "Linux Kernel" -d zImage uImage

# 验证 uImage
$ mkimage -l uImage
Image Name:   Linux Kernel

第十章 · 启动镜像格式

Image.gz / Image.* — ARM64 压缩变体

ARM64 下 Image 的压缩版本，与 x86 的 bzImage 不同：它需要 bootloader 负责解压，内核本身不含解压代码。

📄 Booting AArch64 Linux — kernel.org
Image.gz
ARM64 外部解压压缩内核镜像
ARM64 架构 · 路径：arch/arm64/boot/Image.gz
ARM64 专用 需 Bootloader 解压 gzip 压缩 节省存储空间

ARM64 内核官方文档明确说明： "AArch64 内核目前不提供解压器，因此如果使用压缩 Image 目标（如 Image.gz）， 需要由 bootloader 执行解压（gzip 等）。对于不实现此要求的 bootloader， 可使用未压缩的 Image 目标代替。"

关键差异：与 bzImage 的对比

特性bzImage (x86)Image.gz (ARM64)
解压责任✅ 内核自包含解压代码❌ 必须由 bootloader 解压
bootloader 要求无特殊要求必须支持 gzip 解压
未解压能否运行❌ 否（格式不同）❌ 否（需先解压为 Image）
替代方案无（同架构无替代）Image（未压缩，bootloader 直接加载）

  OBJCOPY arch/arm64/boot/Image

📋 启动镜像格式总结（第四至十章）

  • vmlinux：调试用，不可直接引导
  • Image：ARM/ARM64 裸二进制，bootloader 直接加载
  • zImage：已过时，512KB 低内存限制，现代系统禁止使用
  • bzImage：x86/x86_64 唯一推荐格式，自解压到高内存，支持任意大小内核
  • vmlinuz：发行版对压缩内核的命名惯例，通常就是 bzImage
  • uImage：U-Boot 嵌入式专用，带 64 字节元数据头
  • Image.gz 等：ARM64 压缩变体，需 bootloader 解压
  • 第一性原理：格式的核心区别是谁负责解压（内核自身 vs bootloader）以及加载到内存的哪个位置

第十八章 · 总结

第一性原理总结

从最基本的原则出发，理解 Linux 内核格式多样性的根本逻辑。

🧠 第一性原理：内核镜像格式的本质问题

Linux 内核源码经过编译链接，最终产生一个静态的 ELF 二进制（vmlinux）。 但 ELF 是操作系统格式，而 bootloader 存在于操作系统之前——它无法依赖 OS 加载器解析 ELF。 因此，必须将内核转换为 bootloader 能够理解的格式。

这里存在三个核心问题，不同格式是对这三个问题的不同回答：

① 谁来解压？
→ bzImage/zImage：内核自带解压 stub，bootloader 不需关心
→ Image.gz/uImage：bootloader 负责解压，内核自身不含解压代码

→ 结论：嵌入式 bootloader 能力弱 → 让内核自解压；嵌入式 bootloader 能力强 → 外部解压效率更高

② 解压到内存的哪里？
→ zImage：0x10000（640KB 以下），受 x86 实模式限制
→ bzImage：0x100000（1MB 以上），突破 BIOS 低内存区限制
→ Image (ARM64)：text_offset 指定位置，2MB 对齐

→ 结论：内存布局限制（硬件约束）决定了加载位置，进而决定了格式设计

③ 谁来管理内核的生命周期？
→ 手动管理：tar/dir 格式，用户自行处理
→ 包管理器：deb/rpm/pacman/snap，利用发行版生态（依赖解析、触发器、回滚）

→ 结论：发行版生态的存在使打包格式成为必要，不同发行版的选择产生了多种格式

📋 终极总结

  • 格式多样的根本原因：硬件差异（x86/ARM/嵌入式）+ 发行版生态 + 历史演进，三者共同作用
  • 所有格式的共同起点：vmlinux（ELF 静态链接内核），唯一源头
  • 选择启动镜像格式：x86→bzImage，ARM64→Image/Image.gz，U-Boot→uImage，调试→vmlinux
  • 选择打包格式：目标发行版决定（deb/rpm/pacman），跨平台用 targz/tarzst，模块补充用 cpio-pkg
  • 第一性原理：任何格式都是在回答"谁负责解压 + 加载到哪里 + 谁来管理生命周期"这三个基本问题
