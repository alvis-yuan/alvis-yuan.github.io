基础概念：交叉编译与三元组

参考来源： crosstool-NG How a toolchain is constructed； Gentoo Embedded Handbook: Creating a cross-compiler； gcc.gnu.org mailing list: sysroot 说明

什么是交叉编译工具链

交叉编译工具链（Cross-Compilation Toolchain）是一组协同工作的软件工具，运行在宿主机（Host）上， 却能为目标平台（Target）生成可执行代码。本文以典型场景为例：

# 典型场景：在 x86_64 Linux 上，为 MIPS Linux 生成代码 Build = x86_64-linux-gnu # 构建机器（编译工具链本身的平台） Host = x86_64-linux-gnu # 工具链运行的平台 Target = mips-linux-gnu # 生成代码运行的目标平台
注意：此处 host 指运行工具链的机器（x86_64），target 指将要执行编译产物的机器（MIPS）。 在 autotools configure 脚本中，--host 参数表示的是目标系统（即程序的运行平台），这与直觉相反，需特别注意。

来源：crosstool-NG: Using the toolchain

GNU 目标三元组（Target Tuple）

GNU 工具链用一个形如 arch-vendor-os-abi 的字符串唯一标识一个目标平台，称为"目标三元组" （实际上通常有 3 或 4 段）。对于 MIPS Linux，常见的三元组有：

三元组说明字节序
mips-linux-gnuMIPS 大端，glibc，Linux大端 (BE)
mipsel-linux-gnuMIPS 小端，glibc，Linux小端 (LE)
mips-unknown-linux-gnuMIPS 大端，vendor=unknown大端 (BE)
mips64-linux-gnuMIPS64 大端，glibc，Linux大端 (BE)
mips-linux-muslMIPS 大端，musl libc大端 (BE)

本文以 mips-linux-gnu 为标准三元组进行分析，安装前缀记为 /opt/mips-toolchain。

📌 本章总结

  • 交叉编译工具链用三元组唯一标识目标平台（如 mips-linux-gnu）。
  • 核心组件按顺序依赖：Binutils → Linux Headers → GCC (core) → glibc → GCC (final)。
  • Host（宿主机）运行工具链，Target（目标机）运行编译产物，两者硬件架构不同。
  • MIPS 有大端（BE）和小端（LE）两种变体，三元组明确区分。

⚙ 第一性原理

交叉编译的本质：将程序的编译环境（Host）与运行环境（Target）分离。 工具链目录结构的一切设计，都服务于这个分离：需要明确区分"在宿主机上执行的工具"与"为目标机生成的库/头文件"， 防止两者互相污染。

第一章

工具链顶层目录结构

参考来源： gcc-help mailing list: sysroot 路径设计； Stack Overflow: crosstool-ng directory structure and sysroot； Gentoo Embedded Handbook

顶层目录全貌

一个通过 crosstool-NG 或发行版包管理器安装的 MIPS 交叉工具链，其顶层目录（$PREFIX）通常如下所示。 本文以 /opt/mips-toolchain 为安装前缀：

/opt/mips-toolchain/ ├── bin/ # 宿主机上运行的可执行工具（交叉编译器本体） │ ├── mips-linux-gnu-gcc │ ├── mips-linux-gnu-g++ │ ├── mips-linux-gnu-ld │ ├── mips-linux-gnu-as │ ├── mips-linux-gnu-ar │ ├── mips-linux-gnu-nm │ ├── mips-linux-gnu-objdump │ ├── mips-linux-gnu-strip │ ├── mips-linux-gnu-ranlib │ └── mips-linux-gnu-readelf │ ├── lib/ # 工具链主机侧库（GCC 插件、libstdc++ 等） │ └── gcc/ │ └── mips-linux-gnu/ │ └── 12.3.0/ # GCC 版本号 │ ├── libgcc.a # 目标机 GCC 运行时（静态） │ ├── libgcc_s.so # 目标机 GCC 运行时（共享） │ ├── crtbegin.o # C++ 构造函数链接起点 │ ├── crtend.o # C++ 析构函数链接起点 │ ├── include/ # GCC 内置头文件 │ └── plugin/ # LTO/插件接口 │ ├── libexec/ # GCC 内部可执行文件（用户不直接调用） │ └── gcc/ │ └── mips-linux-gnu/ │ └── 12.3.0/ │ ├── cc1 # C 编译器后端（C -> 汇编） │ ├── cc1plus # C++ 编译器后端 │ ├── lto-wrapper # 链接时优化包装器 │ └── collect2 # 链接包装器，处理构造/析构 │ ├── include/ # （部分工具链）工具链自身的头文件 │ ├── share/ # man 手册页、locale 数据、文档 │ └── mips-linux-gnu/ # 目标机专属子树（核心！） ├── bin/ # 目标三元组命名的 binutils（无前缀版） │ ├── as → ../../bin/mips-linux-gnu-as │ ├── ld → ../../bin/mips-linux-gnu-ld │ └── ar ... ├── lib/ # 目标机库（部分工具链放此处，另有 sysroot） ├── include/ # 目标机头文件（传统方案） └── sysroot/ # 目标系统根（现代 sysroot 方案） ├── lib/ └── usr/
关键设计原则：顶层 bin/ 中的工具均以目标三元组为前缀（如 mips-linux-gnu-gcc）， 确保多个架构的工具链可共存于同一 $PREFIX 下，互不干扰。
来源：gcc-help mailing list

bin/ 目录：主机可执行工具

$PREFIX/bin/ 目录存放运行于宿主机的所有交叉编译工具。 每个工具均以目标三元组为前缀命名，以便与本机工具区分，同时支持多套工具链共存。

文件名来源组件职能
mips-linux-gnu-gccGCCC 编译器驱动程序，协调 cc1、as、ld 完成完整编译流程
mips-linux-gnu-g++GCCC++ 编译器驱动程序
mips-linux-gnu-cppGCCC 预处理器（独立调用）
mips-linux-gnu-asBinutilsGNU 汇编器，将汇编代码转换为 ELF 目标文件
mips-linux-gnu-ldBinutilsGNU 链接器，将目标文件链接为可执行文件或共享库
mips-linux-gnu-arBinutils创建、修改和提取静态库（.a 文件）
mips-linux-gnu-nmBinutils列出目标文件的符号表
mips-linux-gnu-objdumpBinutils反汇编和检查目标文件、可执行文件
mips-linux-gnu-readelfBinutils显示 ELF 文件格式详细信息
mips-linux-gnu-stripBinutils删除可执行文件中的调试符号，减小文件大小
mips-linux-gnu-ranlibBinutils为静态库生成索引（符号表），加速链接
mips-linux-gnu-sizeBinutils显示目标文件各段的大小
mips-linux-gnu-stringsBinutils从二进制文件中提取可打印字符串
mips-linux-gnu-gdbGDB（可选）GNU 调试器，支持远程调试 MIPS 目标

mips-linux-gnu/ 子树：目标机专属目录

$PREFIX/mips-linux-gnu/ 是以目标三元组命名的子目录， 是整个工具链中与目标平台直接相关内容的集中存放处。 根据 gcc-help 邮件列表中的权威说明：

官方解释："binutils（在宿主机上运行的可执行文件）保留在其传统位置， 但纯粹依赖目标平台的 C 库则放入 sysroot。$prefix/$target 目录中的 bin 存放宿主机执行、面向目标的工具； include 和 lib 存放目标机的头文件和库二进制文件。"
来源：gcc-help mailing list
$PREFIX/mips-linux-gnu/ ├── bin/ # Binutils 工具的符号链接（无前缀版本） │ ├── as → ../../bin/mips-linux-gnu-as │ ├── ld → ../../bin/mips-linux-gnu-ld │ ├── ar → ../../bin/mips-linux-gnu-ar │ └── ... ├── lib/ # 某些工具链方案在此存放目标机库（传统方案） ├── include/ # 传统方案的目标机头文件位置 │ └── sys-include/ # C 库系统头文件（传统方案） └── sysroot/ # 现代 sysroot 方案（目标系统根文件系统的镜像） ├── lib/ # 关键运行时库（libc, ldso 等） └── usr/ ├── include/ # 开发头文件 └── lib/ # 非关键运行时库及开发库

📌 本章总结

  • 顶层 bin/ 存放宿主机上运行的工具，以目标三元组为前缀区分多架构。
  • lib/gcc/ 存放 GCC 自身使用的内部库（libgcc、crtbegin/end.o 等）。
  • libexec/gcc/ 存放 GCC 调用的子程序（cc1、lto-wrapper 等），用户不直接调用。
  • mips-linux-gnu/ 是目标机专属子树，包含目标机头文件、库及 sysroot。
  • 两种方案并存：传统方案（$prefix/$target/include）和现代 sysroot 方案（$prefix/$target/sysroot/）。

⚙ 第一性原理

目录结构的核心矛盾：宿主机工具 vs 目标机文件。 所有设计围绕这个分离展开：前缀命名防止冲突，目标三元组子目录隔离目标相关内容， sysroot 则用统一接口模拟目标系统的根文件系统布局。

第二章

GCC 编译器内部目录

参考来源： Linux From Scratch: GCC-6.3.0 安装说明； gcc.gnu.org: Cross Compiler target include directories； Stack Overflow: COLLECT_LTO_WRAPPER 路径示例

lib/gcc/ 目录：编译器私有库

$PREFIX/lib/gcc/mips-linux-gnu/<版本号>/ 是 GCC 最核心的私有目录之一。 该目录包含的内容既有在目标机上运行的代码，也有编译器在宿主机上查找的辅助文件。 根据 Linux From Scratch 官方文档，该目录内容如下：

lib/gcc/mips-linux-gnu/12.3.0/ │ ├── libgcc.a # GCC 运行时支持库（静态版）→ 目标机 ├── libgcc_s.so # GCC 运行时支持库（动态版）→ 目标机 ├── libgcc_s.so.1 # GCC 运行时共享库真实文件 ├── libgcov.a # 代码覆盖率分析库（-pg 或 --coverage 时使用） │ ├── crtbegin.o # C++ 全局构造函数链接器入口 ├── crtbeginS.o # 用于共享对象/PIE 的 crtbegin ├── crtbeginT.o # 用于静态可执行文件的 crtbegin ├── crtend.o # C++ 全局析构函数链接器入口 ├── crtendS.o # 用于共享对象/PIE 的 crtend │ ├── include/ # GCC 内置头文件（由 GCC 自身提供，与 libc 无关） │ ├── stdarg.h # 可变参数支持（GCC 内置） │ ├── stddef.h # size_t、NULL、ptrdiff_t 定义 │ ├── limits.h # 数值范围限制 │ ├── float.h # 浮点数特性 │ └── ... │ ├── include-fixed/ # 经 fixincludes 修补的系统头文件副本 │ └── plugin/ # GCC 插件接口（LTO 插件等） └── libcc1.so

libgcc：GCC 运行时支持库

libgcc 是 GCC 的运行时支持库，为目标架构提供编译器隐式调用的底层操作实现。 根据 Linux From Scratch 官方文档的描述：

libgcc（来自 LFS 官方文档）："Contains run-time support for gcc"（包含 GCC 的运行时支持代码）。
来源：Linux From Scratch: GCC Contents
库文件形式典型用途
libgcc.a静态库软件浮点运算（MIPS soft-float 场景）、整数运算辅助函数、异常处理基础设施
libgcc_s.so.1共享库动态链接版本，减小最终可执行文件体积，多个程序共享
libgcov.a静态库代码覆盖率数据收集（使用 --coverage 或 -fprofile-arcs 时链接）
libssp.a/.so静态/共享栈溢出保护（Stack Smashing Protector，-fstack-protector 时使用）

libexec/gcc/：编译器内部子进程

$PREFIX/libexec/gcc/mips-linux-gnu/<版本>/ 存放 GCC 驱动程序调用的内部子程序。 用户通过 mips-linux-gnu-gcc 调用，gcc 驱动自动派发到这些子程序。

# gcc -v 输出示例（真实工具链） COLLECT_LTO_WRAPPER=/opt/mips-toolchain/libexec/gcc/mips-linux-gnu/12.3.0/lto-wrapper # 编译阶段：gcc 调用 cc1 /opt/mips-toolchain/libexec/gcc/mips-linux-gnu/12.3.0/cc1 \ -quiet -v test.c -o /tmp/ccXXXXXX.s # 链接阶段：gcc 调用 collect2，collect2 再调用 ld /opt/mips-toolchain/libexec/gcc/mips-linux-gnu/12.3.0/collect2 \ -plugin .../liblto_plugin.so ...
子程序职能
cc1C 编译器真正的后端：将 C 预处理后代码翻译为 MIPS 汇编代码
cc1plusC++ 编译器后端
collect2链接包装器：在调用真正的 ld 前后，处理 C++ 全局构造/析构函数的注册
lto-wrapper链接时优化（LTO）的协调包装器，在链接阶段执行跨编译单元优化
liblto_plugin.soGCC LTO 插件，供 binutils (ar, ld) 在处理包含 LTO 信息的 .o 文件时调用

📌 本章总结

  • lib/gcc/mips-linux-gnu/<ver>/ 包含 GCC 私有库（libgcc、crtbegin/end.o）和内置头文件，这些文件在目标机上运行（libgcc）或用于链接。
  • GCC 头文件搜索有 4 层优先级：内置头 → 修补头 → 传统目标头 → sysroot 头。
  • libexec/gcc/ 中的子程序（cc1、collect2 等）是 GCC 驱动程序的内部机制，用户无需直接调用。
  • MIPS 的 libgcc 在 soft-float 场景下尤为关键，提供软件浮点运算支持。

⚙ 第一性原理

GCC 被设计为一个"编译器驱动"架构：顶层的 mips-linux-gnu-gcc 仅是一个协调者， 真正的工作由 cc1（编译）、as（汇编）、collect2/ld（链接）完成。 这种分离使得 GCC 可以通过替换后端支持任意新的目标架构，同时保持前端不变。

第三章

Sysroot：目标系统根文件系统镜像

参考来源： Gentoo Embedded Handbook: Sysroot 布局； crosstool-NG: Toolchain Usage； gcc.gnu.org mailing list: sysroot 解释

Sysroot 的概念

Sysroot 是现代交叉工具链最核心的概念之一。Gentoo Embedded Handbook 给出了权威定义：

官方定义（Gentoo Embedded Handbook）：
"sysroot 告诉 GCC 将 dir 视为包含目标操作系统根文件系统（子集）的目录树的根。 目标系统的头文件、库文件和运行时目标文件都将在该目录中搜索。"
顶层目录通常在 /usr/$CTARGET。
来源：Gentoo Embedded Handbook
gcc-help 邮件列表补充：
"cross GCC 需要将目标 C 库放入宿主机上一个独立的 $SYSROOT 中。 不能放在其原生位置（通常是 /lib*、/usr/include 和 /usr/lib*）， 而是放在如 /opt/freebsd_toolchain/sysroot 这样的路径下，用 sysroot 替代原生的 /。"
来源：gcc.gnu.org mailing list

sysroot/lib/：关键运行时库

根据 Gentoo Embedded Handbook，sysroot/lib/ 存放关键运行时库， 包括 libc、ldso（动态链接器）等。对于 MIPS + glibc 组合：

文件说明
ld.so.1MIPS 专用的 glibc 动态链接器/加载器（ELF interpreter）。在 x86 上对应 ld-linux.so.2，MIPS 上为 ld.so.1。程序启动时内核加载此文件，由它负责解析 .so 依赖并完成动态链接。
libc-2.38.soglibc 主库的真实 ELF 共享库文件，包含 printf、malloc、open 等数千个 C 标准函数实现
libc.so.6符号链接，指向 libc-2.38.so，应用程序在构建时通过 -lc 链接此文件
libm.so.6数学函数库（sin、cos、sqrt 等）共享版本
libpthread.so.0POSIX 线程（pthreads）库，glibc NPTL 实现
libdl.so.2动态加载接口（dlopen、dlsym、dlclose）
librt.so.1POSIX 实时扩展（时钟、消息队列等）
libresolv.so.2DNS 解析库
MIPS 动态链接器路径： MIPS Linux ELF 可执行文件的 PT_INTERP 段硬编码了动态链接器路径 /lib/ld.so.1（大端） 或 /lib/ld-linux.so.3（某些配置）。在运行时，内核据此路径找到动态链接器并执行。 这与 x86_64 的 /lib64/ld-linux-x86-64.so.2 和 x86 的 /lib/ld-linux.so.2 类似， 但路径不同。

sysroot/usr/：开发层级

sysroot/usr/ 模拟目标系统的 /usr 目录，包含开发所需的头文件和非关键库。 这个设计与 Gentoo Embedded Handbook 指出的意图一致： "这种布局并非偶然，而是有意为之——方便将文件从 /usr/$CTARGET 迁移到目标板的 /。"

子目录/文件来源说明

usr/include/linux/Linux KernelLinux 内核用户空间 API 头文件（socket、ioctl、系统调用号等）
usr/include/asm/Linux Kernel (MIPS)MIPS 架构特定头文件（寄存器定义、信号数量等）

📌 本章总结

  • Sysroot 是目标机根文件系统在宿主机上的结构镜像，GCC 将其视为目标系统的"/"。
  • sysroot/lib/ 存放关键运行时：动态链接器（ld.so.1）和各核心共享库。
  • sysroot/usr/include/ 存放所有开发头文件（glibc + Linux 内核 UAPI）。
  • sysroot/usr/lib/ 存放 CRT 启动文件和开发库，这些在链接时使用。
  • MIPS 的动态链接器路径为 /lib/ld.so.1（不同于 x86 的 /lib/ld-linux.so.2）。

⚙ 第一性原理

Sysroot 的本质：路径替换。目标机上程序运行时搜索 /lib/libc.so.6， 宿主机编译时 GCC 自动将绝对路径前缀替换为 sysroot 路径，从 $SYSROOT/lib/libc.so.6 中找到。 这个机制使得一份工具链可以在宿主机上完整模拟目标机的编译环境，而无需实际运行目标机。

共享库与动态链接器

glibc 的共享库版本管理遵循 SONAME 机制，确保 ABI 兼容性。 动态链接器（ld.so.1 on MIPS）在程序运行时承担以下工作：

  • 读取 ELF 文件的 PT_DYNAMIC 段，找到所有 DT_NEEDED 依赖库。
  • 按照 DT_RPATH、LD_LIBRARY_PATH、/etc/ld.so.conf 中的路径搜索并加载依赖库。
  • 进行符号重定位（GOT/PLT 表的填写），将符号引用绑定到实际地址。
  • 执行各库的初始化函数（.init_array 节），然后调用程序入口 _start。
# MIPS ELF 文件中动态链接器路径示例（readelf -l 输出） [Requesting program interpreter: /lib/ld.so.1] # 在交叉工具链 sysroot 中对应的实际路径： $SYSROOT/lib/ld.so.1 # → libc-2.38.so（动态链接器就是 glibc 的一部分） # 构建时使用 --sysroot，GCC/ld 自动将 /lib/ld.so.1 → $SYSROOT/lib/ld.so.1 mips-linux-gnu-gcc --sysroot=$SYSROOT hello.c -o hello

📌 本章总结

  • CRT 文件分两类来源：glibc 提供（crt1.o、crti.o、crtn.o）和 GCC 提供（crtbegin.o、crtend.o）。
  • 链接顺序固定：crt1 crti crtbegin … [用户对象] … crtend crtn。
  • MIPS 动态链接器为 /lib/ld.so.1（ELF PT_INTERP），由 glibc 安装，负责运行时动态链接。
  • glibc 头文件安装于 sysroot/usr/include/，涵盖 C 标准库、POSIX 和 Linux 特定 API。

⚙ 第一性原理

CRT 文件解决的是"程序启动"的引导问题：操作系统只能跳转到 ELF 入口点（_start）， 但 C 标准要求 main() 调用前运行时环境已就绪（argc/argv 已传递、全局变量已初始化、atexit 已注册）。 crt1.o 等文件就是这个从"内核跳转点"到"用户 main()"之间的桥梁代码。 GCC 自动在链接命令中插入这些文件，用户无需手动指定。

循环依赖与引导过程

构建交叉工具链面临一个经典的"先有鸡还是先有蛋"问题。 根据 crosstool-NG 官方文档的详细分析：

# 来源：crosstool-NG 官方文档 - 完整构建顺序 # https://crosstool-ng.github.io/docs/toolchain-construction/ 1. GMP # GNU 多精度运算库（GCC 依赖） 2. MPFR # 多精度浮点运算库（GCC 依赖） 3. MPC # 复数算术库（GCC 依赖） 4. 编译流程详解

理解一个 C 文件从源码到 MIPS ELF 可执行文件的完整流程，有助于理解各目录的职责：

$ mips-linux-gnu-gcc -v hello.c -o hello # 第1步：预处理（gcc 驱动调用 cc1 with -E 或 cpp） $PREFIX/libexec/gcc/mips-linux-gnu/12.3.0/cc1 -E \ -I$SYSROOT/usr/include \ # 在此搜索 stdio.h 等 hello.c -o hello.i # 第2步：编译（C → MIPS 汇编） $PREFIX/libexec/gcc/mips-linux-gnu/12.3.0/cc1 \ -I$PREFIX/lib/gcc/mips-linux-gnu/12.3.0/include \ # stdarg.h 等 -I$SYSROOT/usr/include \ hello.c -o hello.s # 生成 MIPS 汇编 # 第3步：汇编（汇编 → ELF 目标文件） $PREFIX/bin/mips-linux-gnu-as \ hello.s -o hello.o # 生成 MIPS ELF .o 文件 # 第4步：链接（目标文件 → 最终 ELF 可执行文件） $PREFIX/libexec/gcc/mips-linux-gnu/12.3.0/collect2 \ --sysroot=$SYSROOT \ $SYSROOT/usr/lib/crt1.o \ # _start $SYSROOT/usr/lib/crti.o \ # _init 开始 $PREFIX/lib/gcc/.../crtbegin.o \ # C++ 构造入口 hello.o \ -L$PREFIX/lib/gcc/mips-linux-gnu/12.3.0 \ -L$SYSROOT/lib -L$SYSROOT/usr/lib \ -lgcc -lgcc_s -lc \ # 链接 libgcc, libc $PREFIX/lib/gcc/.../crtend.o \ # C++ 析构入口 $SYSROOT/usr/lib/crtn.o \ # _init 结束 -o hello

📌 本章总结

  • 工具链构建的核心挑战是循环依赖：GCC 需要 glibc，glibc 需要 GCC。
  • 解决方案是分阶段引导：先构建精简版 GCC（Pass 1）→ 用它构建 glibc 基础 → 再构建完整 GCC（Pass 2）→ 最后构建完整 glibc → GCC Final。
  • 完整构建顺序（crosstool-NG 官方）共 12 步，包括 GMP/MPFR/MPC 等前置依赖库。
  • 编译一个 C 文件经过 4 步：预处理 → 编译（cc1）→ 汇编（as）→ 链接（collect2+ld）。

⚙ 第一性原理

工具链构建的引导（Bootstrap）问题本质上是一个自指问题： 要用 C 编译器构建 C 库，而 C 库又是 C 编译器运行所必需的。 解决之道是"最小可行路径"：只要有一个能生成目标机目标文件的 C 编译器， 就能编译出 glibc，再有了 glibc 就能构建功能完整的 GCC。 这是整个 Unix/Linux 系统自举（bootstrap）哲学的微缩版本。

Staging 目录策略（crosstool-NG 官方推荐）

crosstool-NG 官方文档强调：不应直接向 sysroot 安装用户软件包， 而应使用独立的 staging 目录。文档列出了 4 种方案：

方案说明推荐度
Option 1: 直接安装到 sysroot默认行为，无需额外 flag。但会污染工具链，使其无法复用。❌ 不推荐
Option 2: 复制 sysroot 到 staging先 cp -a sysroot → staging，再 --sysroot=staging。Buildroot 使用此方案。✅ 可用
Option 3: 独立 staging + sysroot通过 CPPFLAGS/LDFLAGS 指向 staging，sysroot 保持干净。✅ 推荐
Option 4: Union mount 混合staging=union(sysroot+自定义目录)，适合高级场景。⚠ 复杂
# Option 3 示例（推荐方式，来源：crosstool-NG 官方文档） ./configure \ --build=x86_64-linux-gnu \ --host=mips-linux-gnu \ --prefix=/usr \ CPPFLAGS="-I/path/to/staging/usr/include" \ LDFLAGS="-L/path/to/staging/lib -L/path/to/staging/usr/lib" make make DESTDIR=/path/to/staging install

⚙ 第一性原理与整体总结

交叉编译工具链的目录结构，是围绕一个核心矛盾展开的所有设计决策的集合：

宿主机（Host）需要运行工具，目标机（Target）需要运行产物—— 两者必须严格分离，但又必须无缝协作。

三元组前缀（mips-linux-gnu-）解决了命名冲突； lib/gcc/ 封装了编译器私有运行时； sysroot 机制提供了目标文件系统的路径模拟； CRT 文件（crt1.o、crtbegin.o）完成了程序启动的引导； 多阶段构建（Pass 1 → glibc → Pass 2 → Final）打破了循环依赖的死锁。

理解了这些第一性原理，工具链的每一个目录和文件都不再神秘—— 它们都是在回答同一个问题：如何在一台机器上，为另一台机器生成正确的可执行代码？
