# GCC 扩展与编译期检查

*23高级主题 · 静态分析与契约*

参考来源：[compile_checks.h](compile_checks.h)、[GCC Function Attributes](https://gcc.gnu.org/onlinedocs/gcc/Common-Function-Attributes.html)、[Linux kernel build_bug.h](https://github.com/torvalds/linux/blob/master/include/linux/build_bug.h)、[第9章 参数检查机制](ch9.md)

> 本章规范适用于 **GCC/Clang + gnu99** 用户态项目。宏定义见同目录 [compile_checks.h](compile_checks.h)；`compiletime_assert` / `BUILD_BUG_ON` 需 **-O1+** 优化才能可靠生效。

## 23.1 使用场景与选型

### [建议] 能在编译期发现的约束，优先用 GCC 扩展在编译期检查，而非留到运行期

**适用条件（满足任一类即应考虑本章机制）：**

1. **函数接口契约**可在声明中表达：指针非空、格式字符串类型、变参 NULL 终止、返回值不可忽略。
2. **不变量/布局约束**在编译期可确定：结构体大小、字段偏移、类型兼容性、常量表达式真假。
3. **API 演进**需引导调用方迁移：废弃函数/类型/变量。
4. 项目使用 **GCC/Clang** 且可接受 **gnu99** 扩展（与内核 `build_bug.h` / `compiler_attributes.h` 一致）。

**优先顺序：**

| 场景 | 推荐方案 | 说明 |
| --- | --- | --- |
| 公共 API 指针参数语义上非空 | `__nonnull` / `__nonnull(n…)` | 见 23.4；配合 `-Wnonnull` |
| 自定义 `printf`/`scanf` 风格日志 | `__printf` / `__scanf` | 见 23.5；配合 `-Wformat` |
| 变参列表必须以 NULL 结尾 | `__sentinel` | 见 23.6 |
| 错误码/关键返回值不可忽略 | `__must_check` | 见 23.7；见 [第10章](ch10.md) 错误处理 |
| 全局/文件作用域不变量 | `static_assert` / `_Static_assert` | 见 23.15；[第9章](ch9.md) 9.1 基础 |
| 函数体内或宏内布局/类型检查 | `BUILD_BUG_ON` / `BUILD_BUG_ON_ZERO` | 见 23.13–23.14 |
| 类型安全宏（防指针当数组） | `typeof` + `__same_type` + `ARRAY_SIZE` | 见 23.11–23.12 |
| 仅编译期可见的 NULL/常量误用 | `nonnull` + 静态可见 NULL | 运行时 NULL 仍须运行期检查，见 [第9章](ch9.md) 9.2 |

**反面示例（契约只写在注释里 — 编译器无法 enforcement）**

```c
/* 调用者必须检查返回值，path 不能为 NULL —— 仅文档，无检查 */
int open_file(const char *path, int flags);

void caller(void)
{
    open_file(NULL, 0);   /* ❌ 运行时才崩溃，编译无警告 */
    open_file("/tmp/x", 0);  /* ❌ 忽略返回值，静默失败 */
}
```

**反面示例（运行期 assert 检查本可在编译期捕获的不变量）**

```c
void init(void)
{
    assert(sizeof(struct wire_hdr) == 8);  /* ❌ Release 构建 -DNDEBUG 后消失 */
}
```

**正面示例（声明期契约 + 编译期布局 — 零运行开销）**

```c
#include "compile_checks.h"

static_assert(sizeof(struct wire_hdr) == 8, "协议头大小变更");

int open_file(const char *path, int flags)
    __nonnull(1) __must_check;

void send_msg(struct wire_hdr *h)
{
    BUILD_BUG_ON(offsetof(struct wire_hdr, flags) != 6);
    (void)h;
}
```

**正面示例（格式字符串与 sentinel — 调用点静态诊断）**

```c
void log_msg(int level, const char *fmt, ...)
    __printf(2, 3);

void build_cmd(const char *cmd, ...)
    __sentinel;

void ok(void)
{
    log_msg(1, "%d", 42);
    build_cmd("ls", "-l", "/tmp", NULL);
}
```

> 编译期属性与断言 **不能替代** 对用户输入、外部数据的运行期校验；二者互补，见 [第9章 参数检查机制](ch9.md)。

## 23.2 为什么需要编译期检查

C 语言的一大痛点是：大量错误只在**运行时**才暴露——空指针解引用、格式字符串不匹配、整数溢出……这些错误在内核与系统编程中往往是致命的。

**编译期检查**将错误发现时机前移到编译阶段，达到"zero-cost abstraction"：既不产生运行时开销，又让代码的契约显式地写在声明里，由编译器强制执行。

**示意图要点**

- 编写
- source.c
- 预处理
- cpp
- 编译期
- ✓ 检查在此
- gcc -Wall
- 链接
- 运行
- 崩溃?
- 编译期捕获 = 零运行开销 + 最早发现

GCC 提供了两类机制：

- **属性（Attribute）**：通过 `__attribute__((...)))` 注解声明，让编译器对调用点做静态检查并发出警告。

- **内置函数（Built-in）**：如 `__builtin_constant_p`、`__builtin_types_compatible_p`，在常量折叠期间求值，配合宏构成静态断言。

Linux 内核将这两类机制高度封装在 `include/linux/build_bug.h` 和 `include/linux/compiler_types.h` 中，并在用户空间工具中通过 `tools/include/linux/compiler.h` 复用。

> **📌 本章小结**
>
> - 运行时错误的代价远高于编译期错误，编译期检查是系统编程的防线。
> - GCC 扩展在 C99 语言标准下即可使用，不依赖 C11/C23 新语法。
> - 内核的 `build_bug.h` / `compiler_types.h` 是这些扩展的权威参考实现。

## 23.3 GCC __attribute__ 机制概述

- [📄 GCC 官方文档 — Attribute Syntax](https://gcc.gnu.org/onlinedocs/gcc/Attribute-Syntax.html)
- [Attribute Syntax](https://gcc.gnu.org/onlinedocs/gcc/Attribute-Syntax.html)

GCC 提供的 `__attribute__` 扩展语法，允许在声明时向编译器传递额外语义信息。语法形式为：

```c
__attribute__((attribute-name))
__attribute__((attribute-name(arg1, arg2)))
```

关键说明（来自官方文档）：

- 属性名可以加前后双下划线：`__nonnull__` 等价于 `nonnull`，防止与用户宏冲突。

- 属性可作用于：**函数声明**、**变量声明**、**类型（struct/union/enum）**、**标签**、**语句**。

- 属性是**声明期信息**，不产生任何运行时代码。

**示意图要点**

- __attribute__
- ((...)) 扩展
- 函数属性
- nonnull, format
- warn_unused_result
- deprecated, error
- 变量属性
- aligned, unused
- deprecated
- 类型属性
- packed, aligned
- 语句/标签属性
- unused (标签)
- 编译器静态分析引擎
- warning / error 诊断输出

### 属性放置位置

根据 GCC 官方文档（[Attribute Syntax](https://gcc.gnu.org/onlinedocs/gcc/Attribute-Syntax.html)），属性可放在以下位置：

```c
/* 函数声明 —— 放在声明最后 */
void my_func(void *p) __attribute__((nonnull(1)));

/* 也可以放在函数名之前 */
__attribute__((noreturn)) void die(const char *msg);

/* 类型属性 —— struct 关键字后或右花括号后 */
struct __attribute__((packed)) Point { int x; int y; };

/* 变量属性 */
static int ctx __attribute__((unused));
```

> **📌 关于双下划线写法**
>
> 推荐在头文件中使用 `__nonnull__`、`__format__` 等形式，可避免用户代码中同名宏的冲突。这是内核 `compiler_attributes.h` 的一贯做法。

> **📌 本章小结**
>
> - `__attribute__((...)))` 是 GCC 提供的纯编译器元信息注解，不生成任何指令。
> - 属性可作用于函数、变量、类型，放置位置灵活，推荐使用双下划线形式。
> - 所有功能在 C99（`-std=gnu99`）模式下完整支持，无需 C11/C23。

## 23.4 nonnull — 函数参数空指针检查

- [📄 GCC 官方文档 — nonnull attribute](https://gcc.gnu.org/onlinedocs/gcc-7.1.0/gcc/Common-Function-Attributes.html)
- [Function Attributes GCC 4.4](https://gcc.gnu.org/onlinedocs/gcc-4.4.7/gcc/Function-Attributes.html)
- [📄 kernel: compiler_attributes.h](https://github.com/torvalds/linux/blob/master/include/linux/compiler_attributes.h)

`nonnull` 属性告知编译器：某些指针参数**不得为 NULL**。当编译器在调用点能静态确定传入了 NULL，并且启用了 `-Wnonnull`（被 `-Wall` 隐含），则发出警告。

### 语法

```c
/* 指定第 1、2 个参数非空 */
extern void *my_memcpy(void *dest, const void *src, size_t len)
    __attribute__((nonnull(1, 2)));

/* 不带参数 —— 所有指针参数均非空 */
extern void *my_memcpy2(void *dest, const void *src, size_t len)
    __attribute__((nonnull));
```

来自 GCC 4.4 官方文档的原文（[Function Attributes GCC 4.4](https://gcc.gnu.org/onlinedocs/gcc-4.4.7/gcc/Function-Attributes.html)）：

> "The nonnull attribute specifies that some function parameters should be non-null pointers. […] causes the compiler to check that, in calls to my_memcpy, arguments dest and src are non-null. If the compiler determines that a null pointer is passed in an argument slot marked as non-null, and the -Wnonnull option is enabled, a warning is issued."

### 实战示例

**反面示例（编译器将发出警告/错误）**

```c
#include <stddef.h>

void process(const char *path, void *buf)
    __attribute__((nonnull(1, 2)));

void caller(void)
{
    process("/etc/passwd", (void *)0);
    /* ↑ warning: argument 2 null where non-null expected [-Wnonnull] */
}
```

### 内核宏封装

```c
/*
 * compiler_attributes.h — Linux kernel
 *   gcc: https://gcc.gnu.org/onlinedocs/gcc/Common-Function-Attributes.html
 */
#define __nonnull(...)  __attribute__((__nonnull__(__VA_ARGS__)))
```

> **⚠️ 注意事项**
>
> - 编译器只能检查**静态可见的 NULL**，运行时传入 NULL 不会报错（需运行时检查补充）。
> - 参数编号从 **1** 开始计数。
> - 编译器可能基于 nonnull 信息做优化（假设指针不为空），若运行时违反则为 UB。

> **📌 本章小结**
>
> - `nonnull(idx...)` 将"非空"契约写入函数声明，编译器在调用点执行静态检查。
> - 与 `-Wnonnull`（`-Wall` 包含）配合生效。
> - 属于零开销的接口契约文档化手段，也给编译器提供优化信息。

## 23.5 format — 格式字符串类型检查

- [📄 GCC 官方文档 — format attribute](https://gcc.gnu.org/onlinedocs/gcc-7.1.0/gcc/Common-Function-Attributes.html)
- [📄 kernel: compiler_attributes.h](https://github.com/torvalds/linux/blob/master/include/linux/compiler_attributes.h)

`format` 属性告诉编译器：某函数接受类似 `printf`/`scanf`/`strftime` 风格的格式字符串，并对后续变参按格式字符串做类型匹配检查。

### 语法

```c
extern int my_printf(void *obj, const char *fmt, ...)
    __attribute__((format(printf, 2, 3)));
/*                          ↑       ↑  ↑
 *                        类型  fmt参数位  可变参数起始位
 */
```

- 第一个参数：格式类型，可为 `printf`、`scanf`、`strftime`、`strfmon`。

- 第二个参数：格式字符串参数的位置（1-indexed）。

- 第三个参数：可变参数开始的位置（`...` 的位置）；对于 `vprintf` 风格接受 `va_list` 的函数，写 `0`。

### 实战示例

**反面示例（编译器将发出警告/错误）**

```c
#include <stdio.h>

void log_msg(int level, const char *fmt, ...)
    __attribute__((format(printf, 2, 3)));

void caller(void)
{
    log_msg(1, "%d items", "hello");
    /* ↑ warning: format '%d' expects argument of type 'int',
     *   but argument 3 has type 'char *' [-Wformat=] */
}
```

### vprintf 风格（接受 va_list）

```c
#include <stdarg.h>

void vlog(const char *fmt, va_list ap)
    __attribute__((format(printf, 1, 0)));
/*                                  ↑ 0 表示 va_list 不是 ... */
```

### 内核中的应用

```c
/*
 * kernel: __printf, __scanf 宏
 */
#define __printf(a, b)   __attribute__((__format__(__printf__, a, b)))
#define __scanf(a, b)    __attribute__((__format__(__scanf__,  a, b)))
```

**示意图要点**

- 函数声明
- __attribute__((format(...)))
- 调用点
- log_msg("%d", "hello")
- 编译警告
- -Wformat= 报告不匹配
- 启用 -Wformat（包含于 -Wall）时生效

> **📌 本章小结**
>
> - `format(archetype, fmt-idx, first-arg-idx)` 让编译器对自定义格式化函数做与 printf 相同的类型检查。
> - 内核封装为 `__printf(a,b)`/`__scanf(a,b)`，可直接在用户态复用。
> - `-Wformat`（含于 `-Wall`）是激活此检查的开关。

## 23.6 sentinel — 变参函数哨兵检查

- [📄 GCC 官方文档 — sentinel attribute](https://gcc.gnu.org/onlinedocs/gcc-4.7.2/gcc/Function-Attributes.html)

`sentinel` 属性用于变参函数（`...`），要求其参数列表**必须以 NULL（或指定位置的 NULL）结尾**。如果调用时没有提供 NULL 哨兵，编译器发出警告。

### 语法

```c
void exec_args(const char *prog, ...)
    __attribute__((sentinel));

/* 可选参数：哨兵距最后一个参数的偏移（默认 0，即最后一个参数是 NULL）*/
void exec_args2(const char *prog, ...)
    __attribute__((sentinel(1)));
/* sentinel(1): 倒数第 2 个参数必须是 NULL */
```

### 实战示例

**反面示例**

```c
#include <stddef.h>

void build_cmd(const char *cmd, ...)
    __attribute__((sentinel));

void caller(void)
{
    build_cmd("ls", "-l", "/tmp");
    /* ↑ warning: missing sentinel in function call [-Wformat] */
```

**正面示例**

```c
build_cmd("ls", "-l", "/tmp", NULL);
/* ✓ 正确 */
}
```

> **📌 说明**
>
> GCC 文档说明：有效的 NULL 定义为"零值的任意指针类型"。若系统将 NULL 宏定义为整数 0（而非 `(void*)0`），可能需要显式转换。GCC 自带的 `stddef.h` 已正确处理此问题。

> **📌 本章小结**
>
> - `sentinel` 确保变参函数的 NULL 终止符不被遗漏，常用于 execv 风格接口。
> - 通过 `-Wformat` 触发（`-Wall` 包含）。
> - 可指定哨兵偏移量，应对非最后一位的哨兵场景。

## 23.7 warn_unused_result — 返回值不可忽略

- [📄 GCC 官方文档 — warn_unused_result](https://gcc.gnu.org/onlinedocs/gcc-7.1.0/gcc/Common-Function-Attributes.html)
- [📄 kernel: compiler_attributes.h](https://github.com/torvalds/linux/blob/master/include/linux/compiler_attributes.h)

`warn_unused_result` 属性使得：若调用者未使用该函数的返回值，编译器发出警告。适用于那些忽略返回值意味着"肯定是 bug 或安全漏洞"的函数，如 `realloc`、`read`、错误码返回函数。

### 语法

```c
int fn(void) __attribute__((warn_unused_result));
```

### 实战示例

**反面示例**

```c
#include <stdlib.h>

int do_something(void) __attribute__((warn_unused_result));
```

**正面示例**

```c
int caller(void)
{
    if (do_something() < 0) return -1;
/* ✓ 使用了返回值 */
    do_something();
    /* ↑ warning: ignoring return value of 'do_something',
     *   declared with attribute warn_unused_result [-Wunused-result] */
    return 0;
}
```

### 内核封装

```c
/*
 *   gcc: https://gcc.gnu.org/onlinedocs/gcc/Common-Function-Attributes.html
 *        #index-warn_005funused_005fresult-function-attribute
 */
#define __must_check  __attribute__((__warn_unused_result__))
```

> **💡 提示**
>
> 内核中大量使用 `__must_check` 修饰返回 `int`（错误码）的函数，强制调用者处理错误。这是内核错误处理规范的重要组成部分。

> **📌 本章小结**
>
> - `warn_unused_result` 在函数声明层面声明"返回值必须被使用"。
> - 内核宏 `__must_check` 是其封装，对错误码函数强制要求错误处理。
> - 与 `-Wunused-result`（默认启用）配合，无需额外编译选项。

## 23.8 deprecated — 废弃声明警告

- [📄 GCC 官方文档 — deprecated attribute](https://gcc.gnu.org/onlinedocs/gcc-7.1.0/gcc/Common-Function-Attributes.html)

`deprecated` 属性可用于**函数、变量和类型**。当废弃的实体被使用（不是声明时），编译器发出警告，并包含废弃声明的位置，帮助用户找到替代方案。

### 语法

```c
/* 函数废弃，附带信息（GCC 4.5+） */
int old_func(void) __attribute__((deprecated("请使用 new_func() 替代")));

/* 类型废弃 */
typedef int old_type_t __attribute__((deprecated));

/* 变量废弃 */
extern int old_global __attribute__((deprecated));
```

### 行为说明（来自官方文档）

> "The deprecated attribute results in a warning if the function is used anywhere in the source file. This is useful when identifying functions that are expected to be removed in a future version of a program. The warning also includes the location of the declaration of the deprecated function, to enable users to easily find further information about why the function is deprecated, or what they should do instead. Note that the warnings only occurs for uses […]"

### 类型废弃的精细行为

```c
typedef int T1 __attribute__((deprecated));
T1 x;        /* 警告：行 2 —— 使用了废弃类型 T1 */
typedef T1 T2;   /* 警告：行 3 —— T2 间接使用 T1 */
T2 y;        /* 无警告：T2 本身没被标记为 deprecated */
typedef T1 T3 __attribute__((deprecated));  /* T3 自己也 deprecated */
T3 z __attribute__((deprecated));  /* 声明自身也 deprecated，无警告 */
```

> **📌 关于消息参数**
>
> GCC 4.5+ 支持 `deprecated("message")` 的字符串参数，该字符串将出现在警告信息中，引导开发者找到替代方案。

> **📌 本章小结**
>
> - `deprecated` 可作用于函数、变量、类型三类实体，警告出现在**使用点**而非声明点。
> - 附带消息字符串（GCC 4.5+）能给出迁移指引，是 API 演进的标准手段。
> - 不影响编译成功，仅产生警告（除非 `-Werror`）。

## 23.9 error / warning 属性 — 编译期强制错误与警告

- [📄 GCC 官方文档 — error / warning function attributes](https://gcc.gnu.org/onlinedocs/gcc-7.1.0/gcc/Common-Function-Attributes.html)
- [📄 kernel: compiler_attributes.h](https://github.com/torvalds/linux/blob/master/include/linux/compiler_attributes.h)

`error("message")` 和 `warning("message")` 属性作用于函数声明，当该函数被调用且调用**未被死代码消除优化掉**时：

- `error`：编译器产生包含 message 的**编译错误**。

- `warning`：编译器产生包含 message 的**编译警告**。

> "If the error or warning attribute is used on a function declaration and a call to such a function is not eliminated through dead code elimination or other optimizations, an error or warning (respectively) that includes message is diagnosed. This is useful for compile-time checking, especially together with __builtin_constant_p and inline functions where checking the inline function arguments is not possible through extern char [(condition) ? 1 : -1]; tricks"
>     — GCC 7.1 官方文档

### 关键：与 __builtin_constant_p 配合

这是内核 `compiletime_assert`/`BUILD_BUG_ON` 的核心机制：

```c
/* include/linux/compiler-gcc.h (历史版本，GCC >= 4.3) */
#define __compiletime_error(message) \
    __attribute__((error(message)))

/* include/linux/compiler_types.h (当前内核) */
#ifdef __OPTIMIZE__
#define __compiletime_assert(condition, msg, prefix, suffix)    \
    do {                                                        \
        extern void prefix##suffix(void)                      \
            __compiletime_error(msg);  /* __attribute__((error)) */ \
        if (!(condition))                                       \
            prefix##suffix();    /* 若条件假，调用此函数 → 编译报错 */  \
    } while (0)
#else
#define __compiletime_assert(condition, msg, prefix, suffix) do {} while(0)
#endif
```

### 机制流程图

**示意图要点**

- if (!(condition))
- 编译器求值
- 常量真
- prefix##suffix()
- 调用 error 标记函数
- 编译错误 (error)
- 含 msg 字符串
- 条件为真（假设成立）
- DCE 消除 if 分支
- error 函数调用被消除
- 编译通过 ✓
- ⚠️ 需要 -O1 或更高优化级别才能激活 DCE

### 用户态简化封装

```c
/* tools/include/linux/compiler.h (内核用户态工具) */
#ifndef __compiletime_error
# define __compiletime_error(message)
#endif

#ifdef __OPTIMIZE__
# define __compiletime_assert(condition, msg, prefix, suffix)        \
    do {                                                               \
        __noreturn extern void prefix##suffix(void)                \
            __compiletime_error(msg);                                  \
        if (!(condition))                                              \
            prefix##suffix();                                          \
    } while (0)
#else
# define __compiletime_assert(condition, msg, prefix, suffix) do {} while(0)
#endif

#define compiletime_assert(condition, msg) \
    _compiletime_assert(condition, msg, __compiletime_assert_, __COUNTER__)
```

> **⚠️ 重要约束**
>
> - `__attribute__((error(...)))` 需要 GCC ≥ 4.3，且**必须在优化模式**（`__OPTIMIZE__`，即 `-O1` 以上）下才能通过 DCE 正确工作。
> - `-O0` 时 DCE 不工作，条件即便为真，error 函数也可能不被消除，导致误报。内核通过 `#ifdef __OPTIMIZE__` 保护。

> **📌 本章小结**
>
> - `__attribute__((error("msg")))` 将函数标记为"调用即编译错误"，配合 DCE 实现条件编译期断言。
> - 这是内核 `compiletime_assert` → `BUILD_BUG_ON_MSG` 链路的核心机制。
> - 需要 `-O1+` 优化级别才能正确激活 DCE，内核用 `#ifdef __OPTIMIZE__` 保护此路径。

## 23.10 __builtin_constant_p — 编译期常量判断

- [📄 GCC 官方文档 — __builtin_constant_p](https://gcc.gnu.org/onlinedocs/gcc-5.3.0/gcc/Other-Builtins.html)

GCC 内置函数 `__builtin_constant_p(exp)` 用于判断表达式在编译期是否可求值为常量，以便进行常量折叠。

> "You can use the built-in function __builtin_constant_p to determine if a value is known to be constant at compile time and hence that GCC can perform constant-folding on expressions involving that value. The function returns the integer 1 if the argument is known to be a compile-time constant and 0 if it is not known to be a compile-time constant. A return of 0 does not indicate that the value is not a constant, but merely that GCC cannot prove it is a constant with the specified value of the -O option."
>     — GCC 5.3 官方文档

### 返回值语义

| 返回值 | 含义 |
| --- | --- |
| `1` | GCC 确定该值在编译期是常量（可做常量折叠） |
| `0` | GCC 无法证明是常量（不代表一定不是常量） |

### 典型用法一：条件静态断言

```c
/* 内核：仅当参数是常量时才做 BUILD_BUG_ON */
#define MAYBE_BUILD_BUG_ON(cond)                    \
    do {                                            \
        if (__builtin_constant_p((cond)))       \
            BUILD_BUG_ON(cond);                 \
        else                                    \
            BUG_ON(cond);                       \
    } while (0)

/* 内核：检查 nr 必须等于 1 */
BUILD_BUG_ON(__builtin_constant_p(nr) && nr != 1);
```

### 典型用法二：基于常量选择代码路径

```c
/* 内核 udelay 经典用法（参考 LKML 历史讨论 2001） */
extern void __bad_udelay(void);    /* 故意未定义，调用会链接报错 */
extern void __const_udelay(unsigned long);
extern void __udelay(unsigned long);

#define udelay(n) (__builtin_constant_p(n) ?    \
    ((n) > 20000 ? __bad_udelay() :             \
                    __const_udelay((n) * 0x10c6ul)) \
    : __udelay(n))
/* 若 n 是常量且超过 20000，链接报错（编译期发现问题） */
```

> **⚠️ 依赖优化级别**
>
> `__builtin_constant_p` 的行为与优化级别密切相关：**`-O0` 时几乎总返回 0**，因为常量折叠需要优化器支持。内核编译通常使用 `-O2`。

> **📌 本章小结**
>
> - `__builtin_constant_p(x)` 判断 x 是否在编译期为已知常量，返回 0/1。
> - 与 `BUILD_BUG_ON`、`__attribute__((error))` 配合，构建"常量参数的编译期检查"。
> - 需要 `-O1+` 才能可靠返回 1，`-O0` 下不可依赖。

## 23.11 __builtin_types_compatible_p — 编译期类型兼容性检查

- [📄 GCC 官方文档 — __builtin_types_compatible_p](https://gcc.gnu.org/onlinedocs/gcc-5.3.0/gcc/Other-Builtins.html)
- [📄 Exploring GNU Extensions in Linux Kernel — MaskRay](https://maskray.me/blog/2024-05-12-exploring-gnu-extensions-in-linux-kernel)

GCC 内置函数 `__builtin_types_compatible_p(type1, type2)` 在编译期判断两个类型（忽略 qualifier 如 const/volatile）是否兼容，返回整数常量 0 或 1。

> "This built-in function returns 1 if the unqualified versions of the types type1 and type2 (which are types, not expressions) are compatible, 0 otherwise. The result of this built-in function can be used in integer constant expressions."
>     — GCC 官方文档

### 语法

```c
__builtin_types_compatible_p(typeof(x), int)   /* 1 if x is int */
__builtin_types_compatible_p(double, float)    /* 0 */
__builtin_types_compatible_p(int, const int)   /* 1 (忽略 const) */
```

### 内核用法一：类型安全宏

```c
/* tools/include/linux/compiler.h */
#define __same_type(a, b) \
    __builtin_types_compatible_p(typeof(a), typeof(b))

/* 使用示例：检测参数不能是 struct page* */
#define kunmap_atomic(__addr)                           \
    do {                                                \
        BUILD_BUG_ON(__same_type((__addr), struct page *)); \
        __kunmap_atomic(__addr);                         \
    } while (0)
```

### 内核用法二：类型分派（type dispatch）

```c
/* 内核 abs() 宏：根据类型选择不同处理路径 */
#define abs(x)  __abs_choose_expr(x, long long,                     \
                __abs_choose_expr(x, long,                            \
                __abs_choose_expr(x, int,                             \
                __abs_choose_expr(x, short,                           \
                __abs_choose_expr(x, char, ...)))))

#define __abs_choose_expr(x, type, other) __builtin_choose_expr(   \
    __builtin_types_compatible_p(typeof(x), signed type) ||           \
    __builtin_types_compatible_p(typeof(x), unsigned type),          \
    ({ signed type __x = (x); __x < 0 ? -__x : __x; }), other)
```

> **📌 本章小结**
>
> - `__builtin_types_compatible_p` 返回编译期整数常量，可用于 `#if` 之外的条件表达式。
> - 配合 `typeof` 和 `BUILD_BUG_ON`，实现类型级别的编译期断言。
> - 在 C99 中无法通过其他方式实现，是类型泛型宏的重要基础。

## 23.12 typeof 与类型推导扩展

- [📄 GCC 官方文档 — typeof](https://gcc.gnu.org/onlinedocs/gcc/Typeof.html)

`typeof(expr)` 是 GCC 扩展，在**编译期**提取表达式或类型的类型，不对表达式求值。与类型兼容性检查、静态断言结合时极为强大。

### 语法

```c
typeof(expr)   /* 提取表达式类型 */
typeof(int)    /* 也可以直接用类型 */
typeof(*ptr)   /* 指针所指类型 */
```

### 经典用法：类型安全的 swap 宏

```c
/* 内核风格的类型安全宏 */
#define swap(a, b)                              \
    do {                                        \
        typeof(a) __tmp = (a);                  \
        (a) = (b); (b) = __tmp;                 \
    } while (0)
```

### 用于编译期类型断言

```c
/* 确保两个变量类型相同 */
#define ASSERT_SAME_TYPE(a, b) \
    BUILD_BUG_ON(!__same_type(a, b))

int x; long y;
ASSERT_SAME_TYPE(x, y);
/* 编译错误：int 和 long 不是同一类型 */
```

### typeof 与 sizeof 配合检查数组

```c
/* 安全的数组大小宏，防止对指针误用 */
#define ARRAY_SIZE(arr)                                          \
    (sizeof(arr) / sizeof((arr)[0])                           \
     + BUILD_BUG_ON_ZERO(__builtin_types_compatible_p(          \
           typeof(arr), typeof(&(arr)[0]))))
/* 若 arr 是指针而非数组，BUILD_BUG_ON_ZERO 触发编译错误 */
```

> **📌 本章小结**
>
> - `typeof` 在编译期提取类型，不产生运行时开销，是类型安全宏的基础。
> - 与 `__builtin_types_compatible_p` 配合，可做"类型不匹配则编译错误"的断言。
> - C23 标准化了 `typeof`，但在 C99+GCC 扩展下早已可用。

## 23.13 负宽度位域技巧 — BUILD_BUG_ON_ZERO 原理

- [📄 kernel: include/linux/build_bug.h](https://github.com/torvalds/linux/blob/master/include/linux/build_bug.h)

在 `_Static_assert` / GCC 4.3 的 `__attribute__((error))` 出现之前，内核使用了一个精妙的 C 技巧：**负宽度位域**。这是 `BUILD_BUG_ON_ZERO` 的经典实现基础。

### 核心原理

C 语言规定，位域宽度必须为非负整数；若宽度为负数，则是**约束违规（constraint violation）**，编译器必须报错。

```c
/* 来自 include/linux/build_bug.h (旧版实现) */

/*
 * Force a compilation error if condition is true, but also produce a
 * result (of value 0 and type int), so the expression can be used
 * e.g. in a structure initializer (or where-ever else comma expressions
 * aren't permitted).
 */
#define BUILD_BUG_ON_ZERO(e) ((int)(sizeof(struct { int:(-!!(e)); })))
```

### 逐步解析

**示意图要点**

- BUILD_BUG_ON_ZERO(e)
- e = 条件表达式
- !!(e)
- 布尔化：0 或 1
- -!!(e)
- 0 或 -1（取负）
- int: (-1)
- 负宽度位域
- e = false (0)
- int:0  → 位域宽 0
- sizeof = 0, 整体 = 0 ✓
- e = true (1)
- int:-1 → 负宽度位域
- 编译错误 ✗
- sizeof(struct { ... }) → 结果类型为 int，整体表达式值为 0，可用于初始化器

```c
/* 实用示例 */
/* 1. 用在结构体初始化器中（可以用逗号表达式的地方之外） */
int arr[10 + BUILD_BUG_ON_ZERO(sizeof(int) != 4)];

/* 2. 作为 ARRAY_SIZE 的类型检查子表达式 */
#define ARRAY_SIZE(arr)  \
    (sizeof(arr) / sizeof((arr)[0])            \
     + BUILD_BUG_ON_ZERO(__builtin_types_compatible_p( \
           typeof(arr), typeof(&(arr)[0]))))
```

> **📌 关键优势**
>
> `BUILD_BUG_ON_ZERO` 的巧妙之处：结果是一个**整数表达式（值为 0）**，因此可以出现在**不允许语句的地方**，如结构体成员初始化、`sizeof` 参数的一部分等。这是 `BUILD_BUG_ON`（语句式）做不到的。

> **📌 本章小结**
>
> - 负宽度位域 `int:(-!!(e))` 在条件为真时产生编译约束违规（错误），为假时宽度为 0 合法。
> - `BUILD_BUG_ON_ZERO(e)` 将此包装为值为 0 的整数表达式，可用于任意表达式上下文。
> - 这是内核在 C99 下、不依赖 `_Static_assert` 的最古老的编译期断言手段。

## 23.14 BUILD_BUG_ON 全解析 — 内核静态断言宏族

- [📄 kernel: include/linux/build_bug.h (torvalds/linux master)](https://github.com/torvalds/linux/blob/master/include/linux/build_bug.h)

内核 `build_bug.h` 提供了一组完整的编译期断言宏，以下为官方源码注释引用：

### 完整宏族

```c
/* ── build_bug.h 完整节选（torvalds/linux master 分支）──
 *
 * SPDX-License-Identifier: GPL-2.0
 */

/*
 * Force a compilation error if condition is true, but also produce a
 * result (of value 0 and type int), so the expression can be used
 * e.g. in a structure initializer (or where-ever else comma expressions
 * aren't permitted).
 */
#define BUILD_BUG_ON_ZERO(e) ((int)(sizeof(struct { int:(-!!(e)); })))

/* Force a compilation error if a constant expression is not a power of 2 */
#define BUILD_BUG_ON_NOT_POWER_OF_2(n)        \
    BUILD_BUG_ON((n) == 0 || (((n) & ((n) - 1)) != 0))

/*
 * BUILD_BUG_ON_INVALID() permits the compiler to check the validity of the
 * expression but avoids the generation of any code, even if that expression
 * has side-effects.
 */
#define BUILD_BUG_ON_INVALID(e) ((void)(sizeof((long)(e))))

/**
 * BUILD_BUG_ON_MSG - break compile if a condition is true & emit supplied
 *                    error message.
 * @condition: the condition which the compiler should know is false.
 *
 * See BUILD_BUG_ON for description.
 */
#define BUILD_BUG_ON_MSG(cond, msg) compiletime_assert(!(cond), msg)

/**
 * BUILD_BUG_ON - break compile if a condition is true.
 * @condition: the condition which the compiler should know is false.
 *
 * If you have some code which relies on certain constants being equal, or
 * some other compile-time-evaluated condition, you should use BUILD_BUG_ON to
 * detect if someone changes it.
 */
#define BUILD_BUG_ON(condition) \
    BUILD_BUG_ON_MSG(condition, "BUILD_BUG_ON failed: " #condition)

/**
 * BUILD_BUG - break compile if used.
 *
 * If you have some code that you expect the compiler to eliminate at
 * build time, you should use BUILD_BUG to detect if it is
 * unexpectedly used.
 */
#define BUILD_BUG() BUILD_BUG_ON_MSG(1, "BUILD_BUG failed")
```

### 宏族依赖关系图

**示意图要点**

- BUILD_BUG_ON
- 条件真则报错
- BUILD_BUG_ON_MSG
- 附带自定义消息
- BUILD_BUG
- 总是报错（死代码标记）
- BUILD_BUG_ON_ZERO
- 值为0的表达式
- compiletime_assert
- 条件假则报错
- __compiletime_assert
- __attribute__((error)) 机制
- 需要 __OPTIMIZE__
- 负宽度位域技巧
- int:(-!!(e))
- 适用表达式上下文

### 使用对比

| 宏 | 触发条件 | 可用位置 | 产生值 |
| --- | --- | --- | --- |
| `BUILD_BUG_ON(c)` | c 为真 | 函数体内（语句） | 无 |
| `BUILD_BUG_ON_MSG(c, msg)` | c 为真 | 函数体内（语句） | 无 |
| `BUILD_BUG_ON_ZERO(c)` | c 为真 | 任意表达式上下文 | int 0 |
| `BUILD_BUG_ON_NOT_POWER_OF_2(n)` | n 非 2 的幂 | 函数体内 | 无 |
| `BUILD_BUG()` | 总是（死代码检测） | 函数体内 | 无 |

```c
/* 实战用法 */
#include <stddef.h>

struct msg_hdr {
    uint32_t magic;
    uint16_t len;
    uint8_t  flags;
    uint8_t  pad;
};

void send_msg(struct msg_hdr *h)
{
    /* 确保结构体大小符合协议预期 */
    BUILD_BUG_ON(sizeof(struct msg_hdr) != 8);

    /* 确保 flags 字段在第 7 字节 */
    BUILD_BUG_ON(offsetof(struct msg_hdr, flags) != 6);
}
```

> **📌 本章小结**
>
> - 内核 `build_bug.h` 提供了 5 种静态断言宏，覆盖语句上下文和表达式上下文两类场景。
> - 底层实现：`BUILD_BUG_ON_MSG` → `compiletime_assert` → `__attribute__((error))`；`BUILD_BUG_ON_ZERO` → 负宽度位域。
> - `BUILD_BUG_ON` 失败语义是"条件为真则报错"，与 `assert()` 相同，`_Static_assert` 也相同（条件为假则报错）。注意语义方向。

## 23.15 _Static_assert / static_assert — C11 静态断言

- [📄 kernel: include/linux/build_bug.h](https://github.com/torvalds/linux/blob/master/include/linux/build_bug.h)
- [📄 LKML: [PATCH v2] build_bug.h: add wrapper for _Static_assert](https://lore.kernel.org/lkml/20190208203015.29702-1-linux@rasmusvillemoes.dk/)

`_Static_assert` 是 C11 引入的关键字，但 GCC 从 **4.6 版本**起在所有 C 方言（包括 gnu89、gnu99）中支持它，使得内核可以在 C99 模式下使用。

### 内核封装

来自 Rasmus Villemoes 于 2019 年提交的补丁（内核 LKML）：

> "static_assert() is a wrapper for the C11 _Static_assert, with a little macro magic to make the message optional (defaulting to the stringification of the tested expression). Contrary to BUILD_BUG_ON(), static_assert() can be used at global scope, but requires the expression to be an integer constant expression."

```c
/* include/linux/build_bug.h — 当前 torvalds/linux master */

/**
 * static_assert - check integer constant expression at build time
 *
 * static_assert() is a wrapper for the C11 _Static_assert, with a
 * little macro magic to make the message optional (defaulting to the
 * stringification of the tested expression).
 *
 * Contrary to BUILD_BUG_ON(), static_assert() can be used at global
 * scope, but requires the expression to be an integer constant
 * expression (i.e., it is not enough that __builtin_constant_p() is
 * true for expr).
 *
 * Also note that BUILD_BUG_ON() fails the build if the condition is
 * true, while static_assert() fails the build if the expression is
 * false.
 */
#define static_assert(expr, ...) __static_assert(expr, ##__VA_ARGS__, #expr)
#define __static_assert(expr, msg, ...) _Static_assert(expr, msg)
```

### 与 BUILD_BUG_ON 的关键区别

| 特性 | BUILD_BUG_ON(c) | static_assert(expr) |
| --- | --- | --- |
| 失败条件 | c 为**真**时报错 | expr 为**假**时报错 |
| 作用域 | 仅函数体内 | 全局/函数/块 均可 |
| 依赖优化 | 是（需要 -O1+） | 否（纯 C11/GCC 关键字） |
| 需求 | 编译期常量（宽松） | 整数常量表达式（严格） |
| 可选消息 | 通过 _MSG 变体 | 可选（内核宏封装后） |

### 使用示例

```c
/* 全局作用域 —— BUILD_BUG_ON 做不到 */
static_assert(sizeof(int) == 4, "int 必须是 4 字节");
static_assert(sizeof(long) == 8); /* 消息默认为表达式字符串化 */

struct proto_pkt {
    uint32_t seq;
    uint16_t len;
};
static_assert(sizeof(struct proto_pkt) == 6,
              "proto_pkt 大小改变，需更新协议文档");

void func(void)
{
    /* 函数内也可用 */
    static_assert(sizeof(char) == 1);
}
```

### 消息可选的宏魔法解析

```c
/* 原始调用：static_assert(expr) */
/* 展开步骤：
 * 1. static_assert(expr) → __static_assert(expr, ##__VA_ARGS__, #expr)
 *    由于 __VA_ARGS__ 为空且 ## 删除前置逗号：
 *    → __static_assert(expr, #expr)
 *
 * 2. __static_assert(expr, msg, ...) → _Static_assert(expr, msg)
 *    → _Static_assert(expr, "expr")
 *
 * 有消息调用：static_assert(expr, "my msg")
 * → __static_assert(expr, "my msg", #expr)
 * → _Static_assert(expr, "my msg")
 */
```

> **📌 本章小结**
>
> - `_Static_assert` 自 GCC 4.6 起在所有 C 方言（含 gnu99）下可用，不是 C11 专属。
> - 内核的 `static_assert` 宏让消息参数可选，是对 `_Static_assert` 的轻量封装。
> - **全局作用域**是 static_assert 相对 BUILD_BUG_ON 的最大优势；但它要求严格的整数常量表达式。

## 23.16 compiletime_assert 实现剖析

- [📄 kernel: tools/include/linux/compiler.h](https://github.com/torvalds/linux/blob/master/tools/include/linux/compiler.h)

`compiletime_assert` 是内核提供的最核心的编译期断言接口，被 `BUILD_BUG_ON_MSG` 调用，其实现完整地展示了 GCC 扩展的组合使用。

### 完整实现（tools/include/linux/compiler.h）

```c
/* tools/include/linux/compiler.h — Linux kernel
 *
 * SPDX-License-Identifier: GPL-2.0
 */

#ifndef __compiletime_error
# define __compiletime_error(message)
#endif

#ifdef __OPTIMIZE__
# define __compiletime_assert(condition, msg, prefix, suffix)        \
    do {                                                               \
        __noreturn extern void prefix##suffix(void)                \
            __compiletime_error(msg);                                  \
        if (!(condition))                                              \
            prefix##suffix();                                          \
    } while (0)
#else
# define __compiletime_assert(condition, msg, prefix, suffix) \
    do {} while(0)
#endif

#define _compiletime_assert(condition, msg, prefix, suffix) \
    __compiletime_assert(condition, msg, prefix, suffix)

/**
 * compiletime_assert - break build and emit msg if condition is false
 * @condition: a compile-time constant condition to check
 * @msg:       a message to emit if condition is false
 *
 * In tradition of POSIX assert, this macro will break the build if the
 * supplied condition is *false*, emitting the supplied error message if the
 * compiler has support to do so.
 */
#define compiletime_assert(condition, msg) \
    _compiletime_assert(condition, msg, __compiletime_assert_, __COUNTER__)
```

### 五个关键 GCC 扩展的组合

**示意图要点**

- compiletime_assert
- 静态断言核心
- __attribute__((error))
- 调用即编译错误
- __noreturn
- 告知编译器不会返回
- __COUNTER__
- 唯一函数名后缀
- DCE 优化
- 条件真时消除死分支
- ## 标记粘合
- prefix##suffix 构造唯一名
- 五种扩展缺一不可，共同实现"条件假则编译错误"

### 展开追踪

```c
/* 调用：compiletime_assert(sizeof(int) == 4, "int 必须 4 字节") */

/* 第 1 步展开 → */
_compiletime_assert(sizeof(int) == 4, "int 必须 4 字节",
                    __compiletime_assert_, __COUNTER__)

/* 第 2 步展开（假设 __COUNTER__ = 42） → */
__compiletime_assert(sizeof(int) == 4, "int 必须 4 字节",
                     __compiletime_assert_, 42)

/* 第 3 步（在 __OPTIMIZE__ 下）展开 → */
do {
    __noreturn extern void __compiletime_assert_42(void)
        __attribute__((error("int 必须 4 字节")));
    if (!(sizeof(int) == 4))
        __compiletime_assert_42();
    /* ↑ 若 sizeof(int) != 4，DCE 不能消除此调用，
     *   编译器遇到 error 属性函数调用 → 编译错误 */
} while (0)
```

> **📌 本章小结**
>
> - `compiletime_assert` 组合了 `__attribute__((error))`、`__noreturn`、`__COUNTER__`、DCE 优化、`##` 标记粘合五种 GCC 扩展。
> - `__COUNTER__` 保证每次展开产生唯一函数名，避免多次使用时命名冲突。
> - 整个机制在 `-O1+` 下工作，通过 `#ifdef __OPTIMIZE__` 在 `-O0` 下降级为空操作。

## 23.17 用户态移植指南 — 完整实用头文件

基于内核实现，以下是一份可直接用于用户态 C99 项目的头文件，严格基于 GCC 官方文档和内核源码，**不依赖 C11/C23 新特性**（`_Static_assert` 除外，但 GCC 4.6+ 的 gnu99 均支持）。

### 完整头文件

```c
/*
 * compile_checks.h — 用户态编译期检查工具集
 *
 * 基于：
 *   - GCC 官方文档: https://gcc.gnu.org/onlinedocs/gcc/Common-Function-Attributes.html
 *   - Linux kernel: include/linux/build_bug.h
 *                   include/linux/compiler_attributes.h
 *                   tools/include/linux/compiler.h
 *
 * 要求：gcc -std=gnu99 -O1 (或 -O2)
 *       对于 static_assert：gcc >= 4.6
 *       对于 compiletime_assert：gcc >= 4.3 且 __OPTIMIZE__
 */

#ifndef COMPILE_CHECKS_H
#define COMPILE_CHECKS_H

#include <stddef.h>  /* offsetof, size_t */

/* ── 1. 函数属性检查 ─────────────────────────────────── */

/* 指定参数不得为 NULL（1-indexed，不带参数=所有指针参数）
 * GCC: https://gcc.gnu.org/onlinedocs/gcc/Common-Function-Attributes.html
 */
#define __nonnull(...)    __attribute__((__nonnull__(__VA_ARGS__)))
#define __nonnull_all     __attribute__((__nonnull__))

/* 格式字符串检查
 * GCC: https://gcc.gnu.org/onlinedocs/gcc/Common-Function-Attributes.html
 */
#define __printf(fmt, args)  __attribute__((__format__(__printf__, fmt, args)))
#define __scanf(fmt, args)   __attribute__((__format__(__scanf__,  fmt, args)))

/* 变参哨兵检查
 * GCC: https://gcc.gnu.org/onlinedocs/gcc/Common-Function-Attributes.html
 */
#define __sentinel          __attribute__((__sentinel__))
#define __sentinel_at(pos)  __attribute__((__sentinel__(pos)))

/* 返回值必须被使用
 * kernel: include/linux/compiler_attributes.h
 * GCC: https://gcc.gnu.org/onlinedocs/gcc/Common-Function-Attributes.html
 */
#define __must_check        __attribute__((__warn_unused_result__))

/* 废弃声明
 * GCC: https://gcc.gnu.org/onlinedocs/gcc/Common-Function-Attributes.html
 */
#define __deprecated        __attribute__((__deprecated__))
#define __deprecated_msg(m) __attribute__((__deprecated__(m)))

/* 编译期错误属性（GCC >= 4.3）
 * kernel: include/linux/compiler_attributes.h
 * GCC: https://gcc.gnu.org/onlinedocs/gcc/Common-Function-Attributes.html
 */
#if defined(__GNUC__) && \
    (__GNUC__ > 4 || (__GNUC__ == 4 && __GNUC_MINOR__ >= 3))
# define __compiletime_error(msg) __attribute__((__error__(msg)))
# define __compiletime_warning(msg) __attribute__((__warning__(msg)))
#else
# define __compiletime_error(msg)
# define __compiletime_warning(msg)
#endif

#ifndef __noreturn
#define __noreturn __attribute__((__noreturn__))
#endif

/* ── 2. 类型检查工具 ──────────────────────────────────── */

/* 两个变量/表达式是否是同一类型（忽略 const/volatile）
 * kernel: tools/include/linux/compiler.h
 * GCC: https://gcc.gnu.org/onlinedocs/gcc/Other-Builtins.html
 */
#define __same_type(a, b) \
    __builtin_types_compatible_p(__typeof__(a), __typeof__(b))

/* ── 3. 静态断言（compiletime_assert 机制）─────────────── */

/* kernel: tools/include/linux/compiler.h */
#ifdef __OPTIMIZE__
# define __compiletime_assert(condition, msg, prefix, suffix) \
    do {                                                        \
        __noreturn extern void prefix##suffix(void)           \
            __compiletime_error(msg);                           \
        if (!(condition))                                       \
            prefix##suffix();                                   \
    } while (0)
#else
# define __compiletime_assert(condition, msg, prefix, suffix) \
    do {} while(0)
#endif

#define _compiletime_assert(condition, msg, prefix, suffix) \
    __compiletime_assert(condition, msg, prefix, suffix)

#define compiletime_assert(condition, msg) \
    _compiletime_assert(condition, msg, __compiletime_assert_, __COUNTER__)

/* ── 4. BUILD_BUG_ON 宏族 ────────────────────────────── */

/* kernel: include/linux/build_bug.h */

/* 条件为真时编译错误，结果为 int 0（可用于表达式上下文）*/
#define BUILD_BUG_ON_ZERO(e) \
    ((int)(sizeof(struct { int:(-!!(e)); })))

/* 条件为真时编译错误（带自定义消息，语句形式）*/
#define BUILD_BUG_ON_MSG(cond, msg) compiletime_assert(!(cond), msg)

/* 条件为真时编译错误（语句形式）*/
#define BUILD_BUG_ON(condition) \
    BUILD_BUG_ON_MSG(condition, "BUILD_BUG_ON failed: " #condition)

/* 总是编译错误（标记不应执行的死代码路径）*/
#define BUILD_BUG() BUILD_BUG_ON_MSG(1, "BUILD_BUG failed")

/* n 必须是 2 的幂 */
#define BUILD_BUG_ON_NOT_POWER_OF_2(n) \
    BUILD_BUG_ON((n) == 0 || (((n) & ((n) - 1)) != 0))

/* ── 5. static_assert 封装 ───────────────────────────── */

/* kernel: include/linux/build_bug.h
 * 注：_Static_assert 自 GCC 4.6 在 gnu99 下即可用
 */
#if defined(__GNUC__) && \
    (__GNUC__ > 4 || (__GNUC__ == 4 && __GNUC_MINOR__ >= 6))
# define static_assert(expr, ...) __static_assert(expr, ##__VA_ARGS__, #expr)
# define __static_assert(expr, msg, ...) _Static_assert(expr, msg)
#endif

/* ── 6. 结构体字段偏移检查 ───────────────────────────── */

/* kernel: include/linux/build_bug.h */
#define ASSERT_STRUCT_OFFSET(type, field, expected_offset)       \
    BUILD_BUG_ON_MSG(offsetof(type, field) != (expected_offset),  \
        "Offset of " #field " in " #type " has changed.")

/* ── 7. 安全的 ARRAY_SIZE ────────────────────────────── */
#define ARRAY_SIZE(arr) \
    (sizeof(arr) / sizeof((arr)[0])                             \
     + BUILD_BUG_ON_ZERO(__same_type(arr, &(arr)[0])))
/* 若 arr 是指针（非数组），编译报错 */

#endif /* COMPILE_CHECKS_H */
```

### 使用示例

**反面示例**

```c
#include "compile_checks.h"
#include <stdio.h>

/* 格式检查 */
void mylog(const char *fmt, ...) __printf(1, 2);

/* 非空参数 + 必须检查返回值 */
int open_file(const char *path, int flags)
    __nonnull(1) __must_check;

/* 废弃函数 */
void old_api(void) __deprecated_msg("请用 new_api() 替代");

/* 全局静态断言 */
static_assert(sizeof(int) == 4, "需要 32 位 int");

struct wire_hdr {
    uint32_t magic;
    uint16_t len;
    uint8_t  ver;
    uint8_t  flags;
};

/* 协议结构体布局检查 */
static_assert(sizeof(struct wire_hdr) == 8);

void process_buf(char *buf, size_t n)
{
    char local[64];
    BUILD_BUG_ON(sizeof(local) < 32);  /* 编译期断言 */

    /* 函数体内类型检查 */
    BUILD_BUG_ON(__same_type(buf, int));  /* buf 不是 int */
}
```

**正面示例**

```c
int main(void)
{
    char arr[16];
    size_t n = ARRAY_SIZE(arr);
/* ✓ 16 */

    /* 以下三行会产生编译期警告/错误 */
    /* mylog("%d", "string");     → format 类型不匹配 */
    /* open_file(NULL, 0);        → nonnull 警告      */
    /* open_file("/tmp/f", 0);    → 未使用返回值警告  */
    /* old_api();                 → deprecated 警告   */

    return 0;
}
```

### 从第一性原理理解

> **🎯 第一性原理**
>
> 编译期检查的本质是：**将运行时约束提前编码为编译器可验证的类型/属性信息**。

### 决策树：选哪种机制？

**示意图要点**

- 我需要什么样的检查？
- 函数接口契约
- 参数/返回值
- 类型/大小约束
- 结构体/常量
- 废弃通知
- API演进
- nonnull / format
- sentinel / must_check
- 全局作用域？
- 函数体内？
- static_assert
- _Static_assert
- BUILD_BUG_ON
- 或 BUILD_BUG_ON_ZERO
- deprecated
- deprecated("msg")

> **📌 最终总结 — 第一性原理**
>
> - **核心矛盾**：C 的类型系统不能表达接口语义（"这个指针不能为空"、"这是格式串"），而错误只在运行时可见。
> - **解决路径**：GCC 的 `__attribute__` 是*类型系统外的语义注解*，由静态分析引擎在编译期检验；静态断言则将约束条件直接内嵌为编译流程的一部分。
> - **零成本**：所有机制在编译后不产生任何运行时代码，是真正的"零开销抽象"。
> - **防御纵深**：将属性注解（接口契约）+ 静态断言（不变式）+ `-Wall -Werror`（强制执行）三者结合，构成系统编程的编译期安全防线。
> - **可移植性**：通过宏封装 + 版本检测，退化为空宏时不影响语义，保持与旧编译器的兼容性。
