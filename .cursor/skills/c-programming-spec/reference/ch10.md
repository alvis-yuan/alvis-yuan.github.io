# 错误处理机制

*10函数与接口*

参考来源：[SEI CERT ERR00-C to ERR08-C](https://wiki.sei.cmu.edu/confluence/display/c/ERR00-C)、
    [Linux Kernel §7 Centralized Exiting](https://www.kernel.org/doc/Documentation/process/coding-style.rst)

## 10.1 错误码设计规范

**示例：统一错误码体系**

```c
/* 模块级错误码定义 */
typedef enum net_err {
    NET_OK           =  0,   /* 成功 */
    NET_ERR_PARAM    = -1,  /* 无效参数 */
    NET_ERR_NOMEM    = -2,  /* 内存不足 */
    NET_ERR_TIMEOUT  = -3,  /* 超时 */
    NET_ERR_REFUSED  = -4,  /* 连接拒绝 */
    NET_ERR_INTERNAL = -99, /* 内部错误（不应发生）*/
} net_err_e;

/* 错误码转字符串（便于日志）*/
const char *net_err_str(net_err_e err);
```


## 10.2 错误传播模式

**反面示例（不规范（忽略错误））**

```c
void bad_example(void)
{
    uint8_t *buf = malloc(1024);
    /* 未检查 NULL！*/
    memset(buf, 0, 1024); /* UB: buf可能NULL */

    int fd = open("file", O_RDONLY);
    read(fd, buf, 1024);  /* fd可能=-1 */
    /* 忘记 free(buf)! */
}
```

**正面示例（规范（goto 清理模式））**

完整写法（含异常返回点日志）见 [10.4 节](#104-异常返回点日志)。

> 💡 **Linux 内核 §7 明确推荐 goto 清理模式：**集中退出点使资源清理逻辑清晰可靠，避免了在多个 return 点分别处理清理的遗漏。
>       标签命名应表达语义（如 `out_free_buffer:`），而非 `err1:`/`err2:`。


## 10.3 errno 使用规范

### [必须] 调用系统函数后立即检查返回值和 errno

`errno` 只在函数返回失败时有意义。成功调用后 `errno` 可能被下一次调用覆盖。

      多线程环境下 `errno` 是线程局部的（C99/POSIX 保证）。

```c
#include <errno.h>
#include <string.h>

ssize_t n = read(fd, buf, len);
if (n < 0) {
    /* 立即保存 errno 防止被覆盖 */
    int saved_errno = errno;
    log_error("read failed: %s (errno=%d)",
              strerror(saved_errno), saved_errno);
    return -saved_errno;
}
```

> 函数存在**多处退出**且需释放 fd/内存/锁时，除 goto 清理链外，可优先使用 [第24章 RAII 风格资源管理](ch24.md) 与 [raii.h](raii.h)（`autofd`、`autofree`、`WITH_LOCK`、`REQUIRE`）；**仅单一 return** 且无资源泄漏风险时，使用原生 `close`/`free` 即可。


## 10.4 异常返回点日志

参考来源：[Linux Kernel §16 Return Values](https://www.kernel.org/doc/Documentation/process/coding-style.rst)、
    [raii.h](raii.h) 中 `LogError` / `REQUIRE`

### [必须] 每个异常返回点必须打印错误日志

函数内凡表示**失败或异常**的退出路径，在 `return` 或 `goto` 转向清理/退出标签**之前**，必须调用错误级日志宏记录原因与关键上下文。
无日志的静默失败会掩盖现场，使线上问题无法追溯。

#### 何谓「异常返回点」

| 属于异常返回（必须打日志） | 不属于异常返回（无需打日志） |
| --- | --- |
| 返回负错误码（`-EINVAL`、模块 `NET_ERR_*` 等） | 谓词型函数正常回答「否」（`is_`/`has_` 返回 `false`） |
| 指针函数失败返回 `NULL` | 查找/探测函数将「未找到」作为正常业务结果（须在接口文档标明） |
| `goto out` / `goto err_*` 且最终返回失败 | 成功路径的正常 `return`（`0`、`NET_OK`、有效指针等） |
| `REQUIRE` 条件失败（宏内已含 `LogError`，见 [raii.h](raii.h)） | `noreturn` 致命退出（若另有统一崩溃日志则可豁免单次路径） |

#### 日志 API 与内容要求

1. **优先使用**项目统一的错误日志宏：`LogError`（[raii.h](raii.h)）或团队约定的 `LOG_ERROR` / `log_error`；**禁止**用 `printf` 直接替代。
2. 日志须写在该分支**判定失败处、返回之前**（紧贴 `return`/`goto` 之前），不要延迟到无关的成功路径。
3. 日志内容至少包含：
   - **失败原因**（哪项检查未通过、调用了什么操作）；
   - **可定位上下文**（相关参数值、长度、句柄、状态枚举、`errno` 或模块错误码）。
4. 系统调用失败后若使用 `errno`，须**立即**保存再写入日志（见 10.3 节），避免被后续调用覆盖。
5. 同一函数内不同失败分支**各自**打日志，禁止仅在最外层 `out:` 用笼统一句带过（除非各分支已在检测点打过且 `out:` 不再表示新错误）。

**反面示例（不规范）**

```c
int32_t cfg_open(const char *path, cfg_t *out)
{
    if (path == NULL || out == NULL) {
        return -EINVAL;                    /* ❌ 无日志 */
    }

    int fd = open(path, O_RDONLY);
    if (fd < 0) {
        return -errno;                     /* ❌ 未记录 errno/strerror */
    }

    if (read_header(fd, out) < 0) {
        close(fd);
        return -EIO;                       /* ❌ 静默失败 */
    }
    close(fd);
    return 0;
}
```

**正面示例（规范）**

```c
#include <errno.h>
#include <string.h>
#include "raii.h"

int32_t cfg_open(const char *path, cfg_t *out)
{
    int32_t ret = 0;
    int     fd  = -1;

    if (path == NULL || out == NULL) {
        LogError("invalid argument: path=%p out=%p", (const void *)path, (void *)out);
        return -EINVAL;
    }

    fd = open(path, O_RDONLY);
    if (fd < 0) {
        int saved_errno = errno;
        LogError("open(%s) failed: %s (errno=%d)",
                 path, strerror(saved_errno), saved_errno);
        return -saved_errno;
    }

    if (read_header(fd, out) < 0) {
        LogError("read_header(%s) failed", path);
        ret = -EIO;
        goto out_close;
    }

out_close:
    if (fd >= 0) {
        close(fd);
    }
    return ret;
}
```

**正面示例：goto 清理链 + REQUIRE（宏内已打日志）**

```c
int32_t good_example(void)
{
    int32_t   ret = NET_OK;
    uint8_t *buf = NULL;
    int32_t   fd  = -1;

    buf = (uint8_t *)malloc(1024);
    if (buf == NULL) {
        LogError("malloc(1024) failed");
        ret = NET_ERR_NOMEM;
        goto out;
    }

    fd = open("file", O_RDONLY);
    if (fd < 0) {
        int saved_errno = errno;
        LogError("open(file) failed: %s (errno=%d)",
                 strerror(saved_errno), saved_errno);
        ret = NET_ERR_INTERNAL;
        goto out_free;
    }
    /* ... 业务逻辑 ... */

out_free_fd:
    close(fd);
out_free:
    free(buf);
out:
    return ret;
}

/* REQUIRE：条件失败时宏内已 LogError 并 goto out；须在 out: 前设置 ret */
int32_t load_item(const char *path)
{
    int32_t ret = NET_OK;
    int     fd  = -1;

    if (path == NULL) {
        LogError("path is NULL");
        ret = -EINVAL;
        goto out;
    }

    fd = open(path, O_RDONLY);
    if (fd < 0) {
        int saved_errno = errno;
        LogError("open(%s) failed: %s (errno=%d)",
                 path, strerror(saved_errno), saved_errno);
        ret = -saved_errno;
        goto out;
    }

    /* ... 成功路径 ... */
out:
    if (fd >= 0) {
        close(fd);
    }
    return ret;
}

```

> 💡 `REQUIRE(cond, msg)`（[raii.h](raii.h)）在条件失败时已调用 `LogError(msg)` 并 `goto out`；使用方仍须在 `out:` 返回正确的错误码。
> **审查清单：**遍历函数内每个 `return`（非成功值）、`goto` 错误标签、`REQUIRE` 失败分支，确认上一行（或宏内）均有 `LogError`/`LOG_ERROR`，且消息足以复现失败条件。
