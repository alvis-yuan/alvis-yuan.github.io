蓝牙 Core 6.0 规范 - L2CAP 控制信令帧传输机制简述

L2CAP 控制信令通过特定的控制逻辑通道（C-Frames）承载，参数均为小端 (Little-Endian)。本索引对齐 Core 6.0 L2CAP §4 与 Table 4.2：现行信令 20 条（Code 0x01–0x1A，不含 0x0C–0x11 已废弃 AMP）；左侧按功能分组（连接/配置/断开等 REQ–RSP 配对）并可折叠。

离线使用：单文件自包含，双击即可打开，无需同目录其他资源。

通用信令头：Code(1) + Identifier(1) + Data Length(2) + Data(n)。ACL-U 信令 CID 0x0001（可多命令拼接）；LE-U 信令 CID 0x0005（每 C-frame 仅一条命令）。

经典蓝牙信令信道 (CID 0x0001)

工作于 BR/EDR (ACL-U) 逻辑链路上。支持多条控制指令级联打包 (Concatenated)。也就是说，一个 L2CAP PDU 控制数据区内可顺序排布多个 Code 各异的信令块，以提升信道吞吐。

低功耗蓝牙信令信道 (CID 0x0005)

工作于 LE (LE-U) 逻辑链路。出于低开销和轻量化考量，规范限制 CID 0x0005 每次通信仅允许封装单条控制信令，不能将多条指令级联，且命令大小不能超出信道 MTUsig。

📘 L2CAP 信令包格式定义

⚙️ 配置选项 TLV 详解 (§5.1 – §5.7)

用于 L2CAP_CONFIGURATION_REQ / RSP 的 Config_Options 字段。 每项为 Type(1) + Length(1) + Data(n)。 QoS (0x03) 与 Extended Flow Specification (0x06) 互斥，且均不可与对端已选 EFS 时的 Flush/QoS 混用。
