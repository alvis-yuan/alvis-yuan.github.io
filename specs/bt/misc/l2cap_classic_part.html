📡 经典蓝牙 (BR/EDR) L2CAP 层全景解析

架构定位：在经典蓝牙中，L2CAP 是真正的“网络调度中心”。它屏蔽了底层基带（Baseband）极其丑陋的物理限制，为上层应用（音频、文件、键鼠）提供协议解复用、重传流控、服务质量协商（QoS）以及分片重组功能。它是等价于计算机网络中 TCP/UDP 层的存在。

一、 L2CAP 报文结构进化史 (Packet Formats)

经典蓝牙根据业务需求，将 L2CAP 报文分为以下三种核心结构：

1. 基础信息帧 (B-Frame / Basic L2CAP Mode) - "傻瓜透传模式"
Length
2 Bytes
Channel ID (CID)
2 Bytes
Information (应用载荷)
0 ~ 65535 Bytes
  • CID (通道号): 核心路由标志。动态分配（如 0x0040 为音乐，0x0041 为控制）。0x0001 永远固定给 L2CAP 信令通道。
  • 特点: 无流控，无重传校验，全靠底层基带 ARQ 硬抗。
2. 增强重传帧 (I-Frame / ERTM Mode) - "TCP级别的顺丰保价快递"
Length
2 Bytes
Channel ID (CID)
2 Bytes
Control
2 Bytes
Information (载荷)
Variable
FCS
2 Bytes
  • Control (控制字段): 包含 TxSeq (发送序号) 和 ReqSeq (请求确认号)，实现了滑动窗口（一口气发一堆包）和选择性重传（SREJ）。
  • FCS (帧校验序列): 为什么加 FCS？因为底层基带把包切碎了，L2CAP 重新拼装时可能内存出错，加 FCS 保证端到端 100% 绝对正确。
  • 注: 还有不带 Information 的 S-Frame (监控帧)，专用来发 ACK 或 RNR(内存满流控)。
3. 无连接广播帧 (Group Frame / CID = 0x0002) - "群发大喇叭"
Length
2 Bytes
CID = 0x0002
2 Bytes
PSM
2 Bytes
Information (广播数据)
Variable
  • PSM (协议服务复用器): 因为广播通道 0x0002 大家都在用，所以要加个 PSM 告诉对方这是什么业务的广播。
  • 特点: 单向发射，无需建连，底层通过 Active Broadcast 物理广播实现。

二、 L2CAP 动态建连与协商交互图 (MSC)

与 BLE 的固定 CID（0x0004）不同，经典蓝牙每次通信前，必须在 CID=0x0001 (信令通道) 上进行极其繁琐的“打电话协商”，分配动态 CID，并协商 MTU 与 QoS 参数。

Host A (手机)
Host B (耳机)
底层基带 ACL 物理连接已建立
L2CAP_ConnectReq (PSM = 0x0019 AVDTP)
L2CAP_ConnectRsp (Success, 你的CID=0x0040, 我的CID=0x0045)
至此，逻辑通道打通。进入参数配置阶段
L2CAP_ConfigReq (我的要求: MTU=672, 模式=Basic)
L2CAP_ConfigRsp (Success, 我同意)
L2CAP_ConfigReq (我的要求: MTU=1000, 模式=Basic)
L2CAP_ConfigRsp (Success, 我同意)
===== 协商完毕，全速传输阶段 =====
Host A 往 CID 0x0045 发包 | Host B 往 CID 0x0040 发包

三、 四大工作模式总结 (Operation Modes)

模式名称 重传机制 流控与确认 核心应用场景
Basic (基础模式) 无 (依赖底层基带) 无 L2CAP 层流控 服务发现(SDP)、RFCOMM透传、基础指令控制。
ERTM (增强重传模式) L2CAP 层完美 SREJ 补发 滑动窗口 + RNR (内存满拒收) FTP文件传输、OBEX 电话本同步 (要求 100% 绝对不丢一个字节)。
Streaming (流模式) 绝对不重传 (过时即扔) 无确认，按序组装跳号忽略 A2DP 高品质音乐流 (保证实时性，宁可有杂音也绝不卡顿)。
FCS Option (可选项) ERTM和流模式尾部带有 16-bit 校验码，防止底层拼装内存错位。 高质量数据通道。
