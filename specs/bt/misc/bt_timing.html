场景 1：设备发现 - Inquiry (类似 BLE 的扫描/广播)

⏱️ 时序硬性规定：625 µs 时隙内的极速跳频

  • 经典蓝牙没有广播包！ 主设备想要找人，必须主动在 32 个唤醒频点上狂发包含“通用查询访问码 (GIAC)”的 ID 包。
  • 半时隙跳频： 在 Inquiry 状态下，主设备的速度极快，它在 1个时隙 (625µs) 内发 2 个包，跳 2 次频（相当于 3200次/秒 的跳频速度）。
  • 多久回复？ 从设备（平时处于交替睡眠和唤醒的 Inquiry Scan 状态）一旦在某个频点抓到了 ID 包，它必须在 625µs 之后 (下一个从机发送时隙) 回复一个 FHS (Frequency Hop Synchronization) 包。FHS 包含了从机的 MAC 地址和当前时钟，这是下一步建立连接的钥匙。

⏱️ 时序硬性规定：严格的 4 步握手 (Slot by Slot)

  • 主设备利用刚才获取的从机时钟，预判从机现在的频点，发送带有从机专属访问码 (DAC) 的 ID 包。
  • 经典蓝牙建链必须经过 “我叫你 -> 你答应 -> 我把我的时钟和跳频规则给你(FHS) -> 你确认收到” 这严格的 4 个时隙（即 4 * 625µs = 2.5 毫秒）。
  • 完成这 4 步后，主设备发送一个 POLL 包轮询，如果从设备回复了，代表物理层 ACL 链路彻底建立！双方开始使用主设备的时钟进行每秒 1600 次的同步跳频。

⏱️ 时序硬性规定：TDD (时分双工) 与 625µs 铁律

  • 偶数发，奇数收： 主设备 (Master) 只能在偶数时隙 (Slot 0, 2, 4...) 开始发送数据，从设备 (Slave) 只能在奇数时隙 (Slot 1, 3, 5...) 回复数据。
  • 必须回复吗？ 是的。如果 Master 在 Slot 0 发了包，Slave 必须在紧接着的 Slot 1 给出回应（哪怕是没有数据的 NULL 包，里面也包含了 ACK 确认）。
  • 大数据怎么办 (多时隙包)？ 如果发大文件（如音频或 SPP 传文件），1个时隙 (625µs) 塞不下。经典蓝牙允许使用 DH3 (占3个槽 = 1.875ms) 或 DH5 (占5个槽 = 3.125ms) 包。在发长包期间，射频不跳频，一波发完。
  • 注意对齐： 即使 Master 占用了 Slot 2, 3, 4 发送长包，Slave 下一次合法的发送时机必定顺延到下一个奇数槽，也就是 Slot 5。

(继续休眠) 若无数据

⏱️ 时序硬性规定：Sniff Interval (与 BLE 极度相似的祖师爷)

  • 在没有数据传输时，如果每隔 1.25ms (一偶一奇两个时隙) 就要互相问候一次，那经典蓝牙的电量不到半天就会耗尽。
  • Sniff Mode (嗅探模式)： 主从双方协商一个 Sniff Interval (嗅探间隔)。从设备可以关闭射频彻底休眠，只在规定的“锚点 (Anchor Point)”醒来。
  • Sniff Attempt (嗅探尝试时长)： 从机醒来后，如果在指定的 N 个时隙内没听到 Master 叫自己，就可以立刻断电继续睡到下一个周期。这正是后来 BLE (低功耗蓝牙) 连接间隔机制 灵感的直接来源！
