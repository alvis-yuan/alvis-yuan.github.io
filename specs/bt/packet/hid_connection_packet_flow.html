阶段 1：从 SDP 获取 HID 连接参数 31.523s – 31.719s

HID 不能硬编码 PSM，必须通过 SDP 的 ProtocolDescriptorList (0x0004) 和 HIDDescriptorList (0x0206) 获取。

SDP 会话 2（广搜）· CSV 第 52 行 · 31.522704

Service Search Attribute 已包含 HID 服务摘要：

  • Class UUID 0x1124（Human Interface Device）
  • 服务名 "Ifx HID Device"
  • Profile HID v1.17，HIDP v1.1.1
  • Protocol：L2CAP → PSM 0x0011（Control）、0x0013（Interrupt）
ProtocolDescriptorList 片段（Data Element 解码）: 35 06 19 01 00 09 00 11 → L2CAP, PSM = 0x0011 (HID Control) 35 06 19 01 00 09 00 13 → L2CAP, PSM = 0x0013 (HID Interrupt)
必要性：客户端据此知道要连哪两个 PSM，而不是 SDP 的 0x0001。
SDP 会话 4（精查）· CSV 第 64–66 行 · 31.704 – 31.719

Service Search

31.703957 · HID Handle 0x00010001
03 00 00 00 09 ... → UUID 0x1124，返回 Handle 0x00010001

Service Attribute

31.708956 · 响应 ParamLen ≈ 0x0181 (385 字节)

含完整 HID Report Descriptor（二进制，见 Data 列末尾 05 01 09 06 A1 01… 等 HID 类描述符）。

必要性：主机 HID 驱动需 Report Descriptor 解析键盘/鼠标/自定义报告格式； 无 Descriptor 无法正确解析 Interrupt 通道上的 Input Report。

SDP Disconnect · CID 0x0054

31.718957
SDP 任务结束；随后立即建立 HID 业务通道（不再使用 PSM 0x0001）。

HID Profile 要求两条独立 L2CAP 连接：Control（命令）与 Interrupt（输入报告，低延迟）。

PSM 0x0013

阶段 3：HIDP 业务（规范阶段 · 本 CSV 未导出具体 PDU）

L2CAP Configure 完成后，主机通常在 Control 上发送 HIDP 初始化，Interrupt 上接收报告。 本 honor50_sunmi_bt_ota.csv 解码层未列出 HID GET_REPORT / INPUT 等行，以下按 HID 1.1 / HIDP 规范补充典型顺序。

通道典型 HIDP 消息方向目的
ControlSET_PROTOCOL (Boot/Report)Host → Device选择报告协议模式
ControlGET_REPORT (Descriptor/Input)Host → Device拉取报告（若 SDP 未给全）
ControlSET_IDLE / SET_REPORTHost → Device空闲率、LED/输出报告等
InterruptDATA (Input Report)Device → Host按键、触摸、自定义数据
ControlHANDSHAKE (Success/Err)Device → Host对控制事务的应答
抓包文件名含 ota，片段侧重链路建立与维持（AFH、Sniff、Ping），HID 报告可能因解码过滤或未触发大量输入而未出现在 CSV。 CloudPrint 侧 SDP 显示设备类为键盘相关（"Keyboard"/"Infineon"），实际 OTA 可能主要走 SPP，HID 为系统配对时的标准枚举路径。
