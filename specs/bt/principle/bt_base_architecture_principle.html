深入理解蓝牙底层通信架构：从物理信道到 LMP 信令

从物理链路/物理信道的通用概念出发，推演经典蓝牙基带四层架构，并落地到 LMP 控制面信令时序。

第一部分：物理链路与物理信道的通用概念

在计算机网络中，物理链路和物理信道是两个紧密相关但截然不同的概念。理解它们的区别是理解现代通信技术（如多路复用）的基础。

  • 物理链路（Physical Link）：指连接两个网络节点之间的实体物理媒介（如网线、光纤、空气介质）。它关注硬件的连通性。可以类比为“高速公路的实体路面”。
  • 物理信道（Physical Channel）：指在物理链路上，用于传输信号的通信资源通道（按频率、时间、波长划分）。可以类比为“高速公路上划分出的车道”。

为何要区分？ 为了实现资源的最大化利用（多路复用），实现全双工通信，以及在无线环境中避免所有设备产生射频干扰。如果没有信道划分，一条链路一次只能供两个设备单向通信。

第二部分：蓝牙协议中的物理信道与物理链路

在蓝牙这种采用跳频扩频（FHSS）的无线通信技术中，这两个概念有了更特殊的定义：

概念 本质定义 通俗类比
物理信道
(Physical Channel)		 一套射频跳频规则（跳频序列 + 主设备时钟 + 访问码）。解决抗干扰和频道同步问题。 舞池里播放着特定节奏的音乐。所有在这个微微网的设备都在按同一节奏跳跃。
物理链路
(Physical Link)		 物理信道上两个具体设备之间的底层连接状态。负责发送功率控制、链路加密、超时监控。 同一个舞池里，你和具体的某个舞伴牵手跳舞。代表具体的实体连接状态。

第三部分：命名误区——为何称之为“物理”？

在OSI模型中，“物理层”通常指纯射频天线。但在蓝牙协议中，物理信道和物理链路实际上是基带层（Baseband Layer）的概念，包含了复杂的跳频算法和状态机管理。

为什么要加“物理(Physical)”二字？
这是为了与上层协议（如L2CAP层）的“逻辑(Logical)”概念形成严格对立：
- Physical： 强依赖于时钟节拍、硬件射频资源、跳频算法的底层连接（属于Controller）。
- Logical： 纯粹通过软件数据包拆分/重组来复用的数据通道（属于Host/L2CAP）。

思维转换建议：看到“物理信道”请自动翻译为“基带跳频图案”；看到“物理链路”请自动翻译为“基带底层硬件状态控制机”。

第四部分：物理层概念在蓝牙基带报文中的映射

经典蓝牙基带报文的结构极为精简，抽象概念被隐式地嵌入到了头部字段中：

[ 接入码 (Access Code) ] + [ 报文头 (Header) ] + [ 净荷 (Payload) ]
  • 如何体现物理信道？ 看 Access Code。它由主设备地址生成，扮演射频硬件的“过滤器”。如果不匹配，直接丢弃（说明不是同一信道的包）。
  • 如何体现具体的连接对象？ 看 Header 中的 AM_ADDR（后改名为 LT_ADDR）。它占3 bits，用于区分当前物理信道内不同的活动设备和逻辑通道。

第五部分：第一性原理推演——蓝牙基带的四层架构

基于“一对多（复用）且上层属性不同时必须分层”的第一性原理，可以完美推导出蓝牙基带极其严密的四层架构。每次向上分层，都是为了解决“既要...又要...”的需求冲突：

  1. 物理信道 (Physical Channel)：底层的唯一舞台。所有的电磁波都在这个基于时间/频率的信道里飞。

  4. 逻辑链路 (Logical Link, LL)：决定数据内容属性。通过 Payload 头部的 LLID 区分是交给 LMP（底层控制平面）还是 L2CAP（数据业务平面）。
灾难推演：为何不能取消物理链路（并入逻辑传输）？
如果强行合并，当 ACL 逻辑传输为了重传要求“增大发射功率”，而 SCO 逻辑传输为了省电要求“保持发射功率”时，底层射频放大器将陷入精神分裂。因此，无论上层跑多少种逻辑业务，纯物理空间的控制（功率、加密、断连监控）必须由独立的物理链路层统一接管。

第六部分：数据面的“平级假象”与控制面的“严格上下级”

在蓝牙报文中，我们能看到物理信道（Access Code）和逻辑传输（LT_ADDR）的字段，却找不到“物理链路”的专属字段。这是否说明物理链路和逻辑传输是平级关系？

答案是：在数据面看似平级，在控制面是绝对的上下层。

  • 数据面视角（报文处理）：追求极致高效。寻址字段存在的唯一意义是解决 1:N 的复用歧义。在蓝牙中，特定的“逻辑传输 (LT)”必然唯一对应于一条“物理链路 (PL)”，两者是完美的 1:1 关系。因此，解析出 LT_ADDR 就等同于锁定了特定的物理链路，无需多余字段。
  • 控制面视角（状态机生命周期）：必须先建立物理信道，再建立物理链路（对齐时钟、完成握手），最后才能在上面开辟逻辑传输管道。如果物理链路发生超时断开（Supervision Timeout），其上承载的所有逻辑传输将瞬间同时瘫痪。它们存在绝对的寄生关系。

第七部分：协议落地——LMP业务场景与信令时序

理论的终局是控制面的信令交互。LMP (Link Manager Protocol) 就是底层的大管家，它通过 ACL 逻辑传输通道（LLID=11）发送指令，动态调控物理链路和逻辑传输。以下是典型场景：

1. 建立基础物理链路 (Connection Setup)

Master -> Slave : LMP_host_connection_req  (请求建立连接)
Slave -> Master : LMP_accepted             (同意建立)
Master <-> Slave: LMP_features_req/res     (协商支持的蓝牙特性)
Master <-> Slave: LMP_setup_complete       (物理链路及ACL逻辑传输就绪)

2. 开启物理链路硬件加密 (Encryption)

Master -> Slave : LMP_au_rand              (发送随机数挑战，验证身份)
Slave -> Master : LMP_sres                 (响应计算结果)
Master -> Slave : LMP_encryption_mode_req  (请求开启加密)
Master -> Slave : LMP_start_encryption_req (正式启动)
// 此时起，空口 Payload 数据全部受硬件加密保护

3. 物理信道主从角色切换 (Role Switch)

Slave -> Master : LMP_switch_req           (从设备请求篡位)
Master -> Slave : LMP_accepted             (同意切换)
// [纯硬件动作]：摧毁旧物理信道，按新Master的跳频序列重新建立新信道
New Master -> New Slave : LMP_slot_offset  (对齐新时钟)
New Master -> New Slave : LMP_switch_req   (确认角色互换完毕)

4. 建立同步语音逻辑传输 (Setup eSCO)

Master -> Slave : LMP_eSCO_link_req        (我要在原链路上开一条发语音的新通道)
Slave -> Master : LMP_accepted             (同意)
// [状态机变更]：在同一条物理链路上，ACL 和 eSCO 逻辑传输开始并行运作
