Security Manager Specification
蓝牙安全管理规范

定义 LE-only 及 BR/EDR/LE 设备之间的配对、认证与加密协议。规定安全管理器 (SM) 的密钥分发方法、协议流程与密码学工具箱。

Section 1

1. Introduction — 简介

安全管理器 (Security Manager, SM) 定义了 LE-only 或 BR/EDR/LE 设备之间进行配对、认证和加密的协议与行为。它处于蓝牙协议栈的 Host 层，向上为 GAP/应用层提供安全服务，向下通过 HCI 与 Controller 通信。

图 1.1 — 安全管理器在 LE 蓝牙架构中的位置

1.1 Scope — 范围

安全管理器规范定义了以下三类内容：

🔗
配对方法

Just Works、数字比较、Passkey Entry、OOB 四种关联模型

📡
配对协议

Security Manager Protocol (SMP)，运行于 L2CAP 之上，使用固定通道 0x0006

🧰
密码学工具箱

e、ah、c1、s1、f4-f6、g2、h6、h7 等密码学函数

规范描述了 Central（中心设备） 和 Peripheral（外围设备） 的协议角色，与 LE 设备角色或 BR/EDR 设备角色对应。

Section 2

2. Security Manager — 安全管理器

2.1 简介 — 配对三阶段流程

SM 采用密钥分发方式实现身份验证与加密：每台设备独立生成并控制其分发的密钥，其他设备无法干预该密钥的生成过程。密钥强度取决于分发设备内部实现的算法质量。

配对 (Pairing) 是建立密钥的过程，分为固定的三个阶段：

Phase 1
功能交换
Phase 2A
Legacy STK生成
Phase 2B
SC LTK生成
Phase 3
密钥分发(可选)
图 2.1 — LE 配对阶段总览
Phase 2 关联模式选择：在 Phase 1 的功能交换后，根据设备的 IO 能力与认证需求，系统会自动选择以下配对方法之一：
Just Works — 无用户交互，无认证保护，存在 MITM 风险
Numeric Comparison — 双方显示 6 位数，用户确认（仅 SC）
Passkey Entry — 用户在一或两台设备上输入 6 位 PIN 码
Out Of Band (OOB) — 通过 NFC 等带外信道传输数据

2.2 Cryptographic Toolbox — 密码学工具箱

SM 提供一套密码学函数工具箱，用于支持随机地址、配对过程和其他操作。

密码学函数全局关系图

2.2.1 安全函数 e — AES-128 加密

函数 e 是整个 LE Legacy 密码学体系的基础构建块，使用 AES-128 分组密码（符合 FIPS-197），将 128 位密钥和 128 位明文生成 128 位密文。

encryptedData = e(key, plaintextData)

其中 key[0] 对应密钥的最高有效字节，in[0] 对应明文的最高有效字节，out[0] 对应密文的最高有效字节。

💡函数 e 既可在 Host 中以软件实现，也可通过 HCI_LE_Encrypt 命令委托 Controller 的硬件实现，以节省 Host 的计算资源。

2.2.2 随机地址哈希函数 ah

函数 ah 生成用于可解析私有地址（RPA）的 24 位哈希值。蓝牙设备使用 RPA 隐藏真实 MAC 地址，防止被跟踪。只有拥有对应 IRK（身份解析密钥）的设备才能解析 RPA。

ah(k, r) = e(k, r') mod 2²⁴
计算步骤：
  1. 将 24 位随机数 r 与 104 位零填充拼接为 r'（128 位）
  2. 以 IRK (k) 为密钥，对 r' 进行 AES-128 加密
  3. 取加密结果的最低 24 位作为哈希值 hash
  4. RPA = prand[22:0] || hash（共 48 位）

2.2.3 确认值生成函数 c1（LE Legacy Pairing）

在 Legacy 配对过程中，Central 和 Peripheral 互相交换确认值 (Confirm) 来防止 MITM 攻击。函数 c1 基于配对请求/响应报文、设备地址及随机数生成该确认值。

c1(k, r, preq, pres, iat, rat, ia, ra) = e(k, e(k, r ⊕ p1) ⊕ p2)
c1 函数计算流程

2.2.4 密钥生成函数 s1（LE Legacy Pairing）

函数 s1 在 LE Legacy 配对中用于生成短期密钥 STK (Short Term Key)，STK 用于加密配对阶段的链路。注意 STK 仅用于配对过程，配对完成后应换用 LTK。

s1(k, r1, r2) = e(k, r1'[63:0] || r2'[63:0])

取 r1 和 r2 各自的低 64 位拼接为 128 位输入，再以临时密钥 k 加密，生成 128 位 STK。

2.2.5 AES-CMAC 函数

AES-CMAC（基于 AES-128 的消息认证码，定义于 RFC 4493）是 LE Secure Connections 中所有高级密码学函数（f4/f5/f6/g2/h6/h7）的共同基础构建块。

MAC = AES-CMACₖ(m)

输入：128 位密钥 k、可变长度数据 m；输出：128 位 MAC 值。与函数 e 类似，可在 Host 软件实现或通过 HCI_LE_Encrypt 委托 Controller。

2.2.6 确认值生成函数 f4（LE Secure Connections）

SC 配对中用于生成确认值 (Commit)，防止 MITM 攻击。以设备的 ECDH 公钥坐标 (U, V)、随机数 X 和标志位 Z 为输入。

f4(U, V, X, Z) = AES-CMACₓ(U || V || Z)
参数位宽说明
U256 bit公钥 PKa 的 x 坐标
V256 bit公钥 PKb 的 x 坐标
X128 bit用作 AES-CMAC 密钥的随机数（nonce）
Z8 bitNumeric Comparison/OOB 时为 0x00；Passkey 时 MSb=1，LSb=passkey 当前位

2.2.7 密钥生成函数 f5（LE Secure Connections）

SC 配对中用于从 ECDH 共享密钥 DHKey 派生出 LTK 和 MacKey。MacKey 用于后续的 f6 校验，LTK 用于链路加密。

MacKey || LTK = f5(DHKey, Na, Nb, BD_ADDR_C, BD_ADDR_P)
f5 内部两步计算：
// Step 1: 使用固定 SALT 和 DHKey 派生中间密钥 T T = AES-CMAC_SALT(W) // SALT = 0x6C888391_AAF5A538_60370BDB_5A6083BE // Step 2: 两次 AES-CMAC 生成 MacKey (Counter=0) 和 LTK (Counter=1) MacKey = AES-CMAC_T(0 || keyID || N1 || N2 || A1 || A2 || 256) LTK = AES-CMAC_T(1 || keyID || N1 || N2 || A1 || A2 || 256) // keyID = "btle" = 0x62746C65

2.2.8 校验值生成函数 f6（LE Secure Connections）

SC 配对 认证阶段 2 中，双方互换 DHKey Check 值，验证对方确实拥有相同的 DHKey。函数 f6 以 MacKey 为密钥生成该校验值。

f6(W, N1, N2, R, IOcap, A1, A2) = AES-CMACᵥᵥ(N1 || N2 || R || IOcap || A1 || A2)

2.2.9 数字比较函数 g2（LE Secure Connections）

SC 的 Numeric Comparison 模式中，双方各自计算一个相同的 6 位数字展示给用户，用户目视确认两台设备显示的数字是否相同。

g2(U, V, X, Y) = AES-CMACₓ(U || V || Y) mod 10⁶

取 AES-CMAC 输出的最低 32 位，对 10⁶ 取模，得到 000000–999999 范围内的 6 位数字。

2.2.10 密钥转换函数 h6

用于在 LE LTK 与 BR/EDR Link Key 之间进行跨传输密钥派生 (Cross-Transport Key Derivation)。允许 LE Secure Connections 配对生成的密钥直接用于 BR/EDR 连接（反之亦然），无需重新配对。

h6(W, keyID) = AES-CMACᵥᵥ(keyID)

2.2.11 密钥转换函数 h7

与 h6 配合使用的中间密钥转换函数，在跨传输密钥派生流程中生成中间密钥。h7 使用固定 SALT（0x746D7031 或 0x6F77634B）作为 AES-CMAC 密钥。

h7(SALT, W) = AES-CMACsalt(W)

2.3 Pairing Algorithms — 配对算法详解

2.3.1 Phase 1 — Pairing Feature Exchange（功能交换）

配对的第一阶段。Central 发送 Pairing Request，Peripheral 回复 Pairing Response，双方交换以下关键信息以确定配对方法：

字段说明影响
IO Capability设备的输入/输出能力决定关联模型
OOB Data Flag是否有 OOB 数据是否使用 OOB
Auth RequirementsMITM、Bonding、SC、Keypress 标志决定 Legacy 还是 SC
Max Encryption Key Size7–16 字节协商密钥长度
Initiator/Responder Key DistributionLTK/IRK/CSRK 分发意愿Phase 3 内容
IO Capability 与关联模型映射

2.3.2 Phase 2 — LE Legacy Pairing（短期密钥生成）

Legacy 配对使用临时密钥 TK（通过用户交互或 OOB 获取）和双方的随机数生成 STK，用于加密链路。Legacy 配对不提供前向保密性。

  • 1TK 交换：根据关联模型确定 TK（Just Works: TK=0；Passkey: TK=6位数；OOB: TK来自带外数据）
  • 2Confirm 值交换：Central 生成随机数 Mrand，用 c1() 计算 Mconfirm 并发送。Peripheral 同样生成 Srand、Sconfirm。
  • 3随机数交换与验证：双方互换随机数，重新计算对方的 Confirm，与收到的比较，不一致则发送 Pairing Failed。
  • 4STK 生成：两侧均执行 STK = s1(TK, Srand, Mrand)，得到相同的 STK，用于链路层加密。

2.3.3 Phase 2 — LE Secure Connections（长期密钥生成）

SC 配对基于 ECDH（椭圆曲线 Diffie-Hellman，P-256 曲线） 密钥交换，提供前向保密性，是更安全的配对方式。

SC 配对阶段 2 流程（以 Numeric Comparison 为例）

2.3.4 Phase 3 — Transport Specific Key Distribution（密钥分发）

在链路加密后（使用 STK 或 LTK），双方可按 Phase 1 协商的结果分发长期使用的密钥：

LTK

Long Term Key：128 位，用于未来重连时直接加密链路，无需重新配对。

IRK

Identity Resolving Key：128 位，用于解析对方的可解析私有地址 (RPA)。

CSRK

Connection Signature Resolving Key：128 位，用于验证未加密连接上的已签名数据。

Identity Address

设备的公共地址或静态随机地址，与 IRK 配合用于稳定标识设备。

2.3.5 关联模型（Association Models）

关联模型决定 Phase 2 中的安全等级与用户交互方式。共有四种模型：

模型MITM 防护用户交互适用场景Legacy/SC
Just Works 无 无需交互 耳机、键盘等无屏幕设备 两者均可
Numeric Comparison 有 双方显示6位数，用户比较确认 两台设备均有显示屏 仅 SC
Passkey Entry 有 在一台或两台设备上输入6位PIN 键盘+显示屏设备组合 两者均可
Out Of Band (OOB) 有（取决于OOB信道） 依赖 NFC 等带外信道 NFC 触碰配对 两者均可

2.4 密钥分发策略

密钥分发遵循以下原则：每台设备仅生成并控制自己分发的密钥；安全架构设计使响应设备（Responder/Peripheral）的内存与计算需求低于发起设备（Initiator/Central）。

⚠️Phase 3 密钥分发必须在加密链路上进行（使用 Phase 2 生成的 STK 或 LTK 加密）。在非加密链路上分发密钥会导致严重安全漏洞。

2.5 密钥大小协商

在 Pairing Feature Exchange 中双方协商最大加密密钥大小，范围为 7–16 字节（56–128 位）。实际密钥大小取双方支持的最小值。若协商结果不满足任一方要求，配对失败并返回 "Encryption Key Size" 错误。

安全警告：短密钥（如 7 字节）在理论上可被暴力破解。建议所有实现要求至少 16 字节（128 位）密钥。

3. Security Manager Protocol (SMP)

SMP 是安全管理器的协议层，运行于 L2CAP 之上，使用固定信道 CID 0x0006（LE）或 0x0007（BR/EDR）。所有 SMP PDU 均以 1 字节命令码开头，后跟参数。

3.2 Pairing Request / Response 格式

Pairing Request (0x01) 和 Pairing Response (0x02) 的 PDU 结构相同（6 字节参数），用于交换双方能力：

Pairing Request / Response PDU 结构

3.3 Pairing Confirm & Pairing Random

Pairing Confirm (0x03)：携带 128 位确认值（Legacy 用 c1() 计算，SC 用 f4() 计算）。交换顺序：Confirm 先交换，Random 后交换，防止一方在看到对方随机数后作弊。

Pairing Random (0x04)：携带 128 位随机数（Legacy: Mrand/Srand，SC: Na/Nb）。接收方可用随机数重新计算对方的 Confirm 值并验证。

3.4 Public Key & DHKey Check（SC 专用）

Pairing Public Key (0x0C)：携带 512 位 P-256 椭圆曲线公钥（256 位 x 坐标 + 256 位 y 坐标）。双方各自发送，用于 ECDH 计算。

Pairing DHKey Check (0x0D)：携带 128 位 DHKey Check 值（由 f6() 计算），用于认证阶段 2 的相互验证，确认双方拥有相同的 DHKey 和 MacKey。

3.6 密钥交换命令（Phase 3）

Phase 3 密钥分发在加密链路上依次进行，顺序严格按照 Pairing Request/Response 中协商的 Key Distribution 字段。Initiator 先分发（如果需要），然后 Responder 分发。

🔑密钥分发顺序建议：Encryption Information → Central Identification → Identity Information → Identity Address → Signing Information

3.7 Security Request（外围设备主动请求）

Peripheral 可通过发送 Security Request (0x0B) 命令，请求 Central 发起加密或配对流程。这不是配对流程的一部分，而是独立命令。Central 可选择响应（发起配对或加密）或忽略。

Security Request 包含 Auth Requirements 字段，告知 Central 外围设备所需的最低安全级别。

3.8 Keypress Notification（按键通知）

仅用于 SC Passkey Entry 模式，用于在用户输入 Passkey 期间向对端发送输入进度通知，改善用户体验。包含以下通知类型：

0 — Passkey entry started（开始输入）
1 — Passkey digit entered（输入一位）
2 — Passkey digit erased（删除一位）
3 — Passkey cleared（清空输入）
4 — Passkey entry completed（完成输入）

4. Security Modes — 安全模式

蓝牙规范定义了多个安全模式，用于描述设备的安全能力和服务访问控制策略。GAP 和各 Profile 可依据安全模式要求对连接设备进行授权。

4.2 BR/EDR 安全模式与跨传输

BR/EDR 设备同样有安全模式定义（Security Mode 1–4）。当设备同时支持 BR/EDR 和 LE 时，可通过跨传输密钥派生（CTKD, Cross-Transport Key Derivation）在 LE 配对后自动生成 BR/EDR Link Key（或反之），避免用户在两种连接上各自配对一次。

跨传输密钥派生流程：
  • LE LTK → BR/EDR Link Key：h7(SALT="tmp1") 得中间密钥，再 h6(key,"lebr") 得 Link Key
  • BR/EDR Link Key → LE LTK：h7(SALT="owcK") 得中间密钥，再 h6(key,"brle") 得 LTK
  • CT2 标志位（Auth Requirements 的 bit 6）控制是否使用 CT2（h7）方式或旧方式（直接 h6）

C.5 失败场景

规范附录 C.5 描述了多种典型失败场景：

Pairing Not Supported

Peripheral 不支持配对时，向 Central 的 Pairing Request 回应 Pairing Failed (0x05)。

Key Size Rejected

Phase 1 协商的密钥大小不可接受，任一方发送 Pairing Failed (0x06)。

Confirm Value Failed

Confirm 值验证失败（存在 MITM 或输入错误 Passkey），发送 Pairing Failed (0x04)，须等待随机退避时间后才可重试。

Numeric Comparison Failed

用户确认数字不匹配，发送 Pairing Failed (0x0C)，终止 SC 配对流程。

DHKey Check Failed

SC 认证阶段 2 的 f6() 校验值不匹配，可能遭受中间人攻击，发送 Pairing Failed (0x0B)。

Repeated Attempts

连续配对失败后触发，设备必须拒绝新的配对请求并进入退避状态，防止暴力破解。
