TLS 密钥材料导出（Exporter）

允许上层协议在 TLS/DTLS 握手完成后，从主密钥派生导出密钥材料（EKM），典型应用：DTLS-SRTP、EAP-TLS。

PRF / HKDFLabelContextEXPORTER*

1 · 简介与需求

许多协议希望借助 TLS 完成密钥建立，再将部分密钥材料用于自身用途。RFC 5705 定义从 TLS 会话导出 EKM（Exported Keying Material） 的通用机制（曾称 TLS Extractors）。

设计需求（§1）

  • 客户端与服务器导出相同 EKM
  • 不知 master_secret 的攻击者无法将 EKM 与随机数据区分
  • 同一会话可导出多组 EKM
  • 已知一组 EKM 不得泄露 master_secret 或其他 EKM

机制与 TLS 各版本兼容（TLS 1.2 使用 PRF；TLS 1.3 见 §8446 的 exporter_master_secret）。

2 · 绑定应用上下文（§3）

除调用 exporter 外，应用还须安全约定 EKM 的使用上下文（算法、端点标识、会话、生命周期等）。

上下文协商方式（可组合）
  • Context 可作为 exporter 的 context_value 混入 PRF
  • ClientHello/ServerHello 扩展（如 DTLS-SRTP）— 受 Finished 保护
  • 上层协议自有握手，用导出密钥保护

3 · Exporter 定义（§4）

输入三项：label（消歧标签）、context_value（每关联上下文）、length（输出字节数）。

无 context 时

PRF(master_secret, label, client_random + server_random)[0..length-1]

有 context 时

PRF(master_secret, label, client_random + server_random + uint16(context_length) + context_value)[0..length-1]

PRF 为当前 TLS 会话使用的伪随机函数（TLS 1.2）；context 长度为大端 uint16。输出为从 master_secret 派生的 length 字节伪随机串。

同一 label + 同一 master_secret 重复调用得到相同 EKM — 因此不同用途必须使用不同 label，须 IANA 注册（实验可用 EXPERIMENTAL 前缀）。

4 · Label 与 IANA 注册（§4 · §6）

Label 示例说明
EXPORTER*新 label 建议此前缀，避免与 TLS 内部 PRF label 冲突
EXPERIMENTAL...私有实验，无需注册
client finished / server finished / master secret / key expansionRFC 5246 保留，禁止用于 RFC 5705 exporter
client EAP encryptionRFC 5216 EAP-TLS
ttls keying material / ttls challengeRFC 5281 EAP-TTLS

IANA 维护 TLS Exporter Label 注册表；新 label 按 Specification Required 分配，且不得互为前缀。

5 · 与 TLS 1.3（RFC 8446）的关系

TLS 1.3 不再使用 master_secret 术语，改用 HKDF 密钥调度；RFC 8446 更新了 exporter 机制：

  • 从 exporter_master_secret（或 early exporter）经 HKDF-Expand-Label 导出
  • API 概念仍为 label + context + length
  • RFC 8446 §7.5 定义 EXPORTER 类 label 的 HKDF 用法

阅读 TLS 1.3 实现时，应同时参考 8446 的密钥调度图与 5705 的安全语义。

6 · 安全考虑（§5）

  • EKM 之间须独立：攻击者即使获得多组 (label, context) 的输出，仍不能区分新一组的输出与随机数
  • 同一 label 的 EKM 不可用于两个不同目的
  • 不同 label 的 EKM 之间，因 PRF 性质，已知一个通常不泄露另一个
  • RSA 密钥交换下 master_secret 可能对两个会话相同（恶意中间人场景）— 应用需评估是否强制 (EC)DHE

5705 只提供导出原语；如何协商上下文、防替换攻击需由具体应用（SRTP、EAP 等）规范定义。
