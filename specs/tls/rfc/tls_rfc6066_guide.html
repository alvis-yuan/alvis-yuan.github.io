TLS 扩展定义图解导读

RFC 5246（TLS 1.2）定义扩展框架；RFC 6066 规范六种早期扩展，其中 SNI 与 status_request 至今广泛使用。

ExtensionType 0–5ClientHelloServerHello向后兼容

1 · 简介

RFC 6066 是 TLS 1.2 的扩展定义伴侣文档，将 RFC 4366 材料适配至 TLS 1.2。扩展用于在不新增密码套件的前提下扩展握手消息格式能力。

支持扩展的客户端与不支持扩展的服务器（及反向）应能正常互通 — 未知扩展由对端忽略（TLS 1.2 §7.4.1.4）。

2 · 扩展握手框架（§2）

扩展出现在 ClientHello、ServerHello 等握手中的 extensions 块：

struct { Extension extensions<8..2^16-1>; } ExtensionBlock;

struct { ExtensionType extension_type; opaque extension_data<0..2^16-1>; } Extension;

TLS 1.3（RFC 8446）保留 SNI 等概念，但许多扩展被新机制替代或重定义；6066 仍是理解 SNI、OCSP Stapling 的经典参考。

4 · server_name — SNI（§3）

TLS 本身不携带「用户想访问哪个主机名」。多虚拟主机共用 IP 时，客户端在 ClientHello 中发送 ServerNameList：

ServerName { NameType name_type; select(name_type) { case host_name: HostName; } name; }
NameType: host_name(0)
ServerNameList: ServerName server_name_list<1..2^16-1>
  • 同类型 name 不得重复；当前仅定义 host_name(0)（DNS 主机名，非 IP 字面量）
  • 服务器无法识别名称：fatal unrecognized_name(112) 或继续握手（由应用层校验证书）
  • 会话恢复：ClientHello 中 SNI 应与首次握手一致，否则服务器可拒绝恢复
  • 服务器不得在 ServerHello 中回显 server_name（除非 RFC 6066 允许的特定场景）

5 · max_fragment_length（§4）

协商 TLS 记录层最大明文片段长度，适用于内存/带宽受限设备。

编码最大长度
12^9 = 512
22^10 = 1024
32^11 = 2048
42^12 = 4096

客户端在 ClientHello 提议；服务器可在 ServerHello 接受相同值。

6 · client_certificate_url（§5）

客户端证书链过大时，可在 ClientHello 声明使用 HTTP/HTTPS/FTP URL 引用证书，而非在 Certificate 消息中内联。服务器在 CertificateRequest 中指示是否支持 URL 类型。

定义 application/pkix-pkipath MIME 类型承载 PKI 路径。

7 · trusted_ca_keys（§6）

客户端声明其信任的 CA（DN 或 Key SHA-1 Hash），服务器据此选择客户端能验证的证书链，减少因客户端 CA 存储有限导致的握手失败。

8 · truncated_hmac（§7）

双方协商使用截断的 HMAC（仅 TLS 1.2 及更早 MAC-then-encrypt 套件有意义）。客户端提议、服务器在 ServerHello 确认。输出长度由密码套件决定（如 10 字节）。

9 · status_request — OCSP Stapling（§8）

客户端在 ClientHello 请求服务器在握手时提供证书 OCSP 响应，避免客户端另行查询 CRL/OCSP。

Certificate Status Request 流程
  • CertificateStatusType：ocsp(1) 等
  • 服务器若发送 status_request 扩展，则 Certificate 后必须跟 CertificateStatus
  • 未收到 status_request 则不应发送 CertificateStatus

11 · 安全考虑摘要

  • SNI：明文传输主机名，敏感场景需 ESNI/ECH（后续 RFC）；客户端可对不同 SNI 呈现不同值
  • client_certificate_url：URL 获取证书需防篡改与重放
  • status_request：OCSP 响应须有效且与证书匹配
  • truncated_hmac：降低 MAC 长度会减弱完整性强度
