X.509 证书内容与 OpenSSL 验证实战

以 tls12_decode.pcapng 中 PDU #2 的 Certificate（1637 B） 为例：说明一张证书包含什么、如何用 openssl 查看与验证，以及 TLS 客户端收到证书链后的完整处理步骤。

CN=172.18.25.17 RSA 2048 · SHA-256 2 张 DER（叶 + 自签 CA） X.509 v1（无扩展） mTLS 双向证书

1 · 证书在 TLS 握手中的位置

在 TLS 1.2 完整握手中，服务器在 ServerHello 之后发送 Certificate 握手消息（类型 0x0B），载荷为一条或多条 DER 编码的 X.509 证书，通常顺序为叶证书 → 中间 CA → … → 根 CA（根可不在链内，只要客户端信任库里有）。

证书不加密传输（在 ChangeCipherSpec 之前均为明文握手）。它用于身份认证与（在 RSA 密钥交换套件中）加密 premaster；本例为 ECDHE_RSA，premaster 由 ECDH 产生，但服务器仍用证书私钥对 ServerKeyExchange 签名以证明拥有该公钥。

2 · X.509 证书逻辑结构（ASN.1）

一张 X.509 证书在 ASN.1 上是一个 SEQUENCE，包含待签名数据 tbsCertificate 与签名算法 + 签名值。下图对应 RFC 5280 的 Certificate 结构（本实例为 Version 1，无 extensions 字段）。

3 · 本抓包服务器叶证书：字段含义对照

下列取值来自 certs_from_pcap/server_chain_0.der（PDU #2 证书链第一张，819 字节）。

字段本实例取值含义与用途
version1 (0x0) — X.509 v1证书格式版本；v3 才有扩展（SAN、EKU、basicConstraints 等）。本证书无扩展，主机名校验依赖 CN=172.18.25.17 与 ClientHello 的 SNI 一致。
serialNumber71:F4:4C:93:…:CC:8B颁发者赋予的唯一编号；用于 CRL/OCSP 吊销查询与审计，须在颁发者范围内唯一。
signature (tbs 内)sha256WithRSAEncryption声明颁发者用什么算法对 tbs 签名（须与底部 signatureAlgorithm 一致）。
issuerCN=ca, OU=NSP, O=LY, …, C=CN签发者可分辨名称（DN）；客户端用issuer 的公钥验证本证书签名。叶证书 issuer 必须匹配链中下一张的 subject。
validity2025-05-29 — 2035-05-27 (UTC)notBefore / notAfter；客户端必须检查当前时间在有效期内。
subjectCN=172.18.25.17, …证书绑定的实体身份。TLS 服务器身份校验：与 SNI/连接主机名匹配（v3 通常看 subjectAltName DNS 名）。
subjectPublicKeyInfoRSA 2048 bit, e=65537实体公钥；用于验证 ServerKeyExchange 签名、（RSA 套件中）加密 premaster、以及后续 CertificateVerify 等。
signatureAlgorithmsha256WithRSAEncryption对 tbsCertificate 做签名的算法标识（外层）。
signatureValue256 字节 RSA 签名颁发者私钥对 DER(tbsCertificate) 的签名；客户端用 CA 公钥解密/验证。
注意： 生产环境应使用 X.509 v3 并包含 subjectAltName、keyUsage、extendedKeyUsage (serverAuth)、basicConstraints 等扩展。本实验室证书为简化 v1，OpenSSL 对主机名的匹配落在 CN 上，恰好与 SNI 172.18.25.17 一致。

4 · 从 tls12_decode.pcapng 提取证书

方法一：本仓库脚本（推荐）

$ cd tls
$ python3 extract_tls_certs.py tls12_decode.pcapng -o certs_from_pcap
wrote certs_from_pcap/server_chain_0.der (819 bytes)
wrote certs_from_pcap/server_chain_1.der (809 bytes)
wrote certs_from_pcap/client_chain_0.der (813 bytes)
wrote certs_from_pcap/client_chain_1.der (809 bytes)

脚本在文件中扫描 TLS 记录 0x16 0x03，解析握手类型 0x0B (Certificate)，按 3 字节长度前缀拆出每条 DER。第一条 Certificate 为服务器（PDU #2），第二条为客户端（PDU #3，mTLS）。

方法二：Wireshark

  1. 打开 tls12_decode.pcapng → 展开 PDU #2 → TLS → Handshake → Certificate。
  2. 右键某条 Certificate → Export Packet Bytes，或对单张证书使用 OpenSSL 从十六进制粘贴（较繁琐）。

DER 转 PEM（便于 openssl 子命令）

$ openssl x509 -in certs_from_pcap/server_chain_0.der -inform DER \
    -out certs_from_pcap/server_leaf.pem -outform PEM
$ openssl x509 -in certs_from_pcap/server_chain_1.der -inform DER \
    -out certs_from_pcap/server_ca.pem -outform PEM

已提取文件目录：tls/certs_from_pcap/（server_leaf.pem、server_ca.pem、server_fullchain.pem）。

6 · 证书链验证（OpenSSL verify）

客户端逻辑：用叶证书的 issuer 找中间/根 CA，逐级验证签名，直到信任锚；同时检查有效期、名称、密钥用途（若有扩展）、吊销状态等。

$ openssl verify -CAfile certs_from_pcap/server_ca.pem \
    certs_from_pcap/server_leaf.pem
server_leaf.pem: OK

含义：使用 server_ca.pem 作为信任锚，验证叶证书上的 RSA-SHA256 签名是否由该 CA 公钥验证通过，且 issuer/subject 链接正确。

与系统 CA 包对比（可选）

$ openssl verify -CAfile /etc/ssl/certs/ca-certificates.crt \
    certs_from_pcap/server_leaf.pem
# 自签 CA 通常不在系统信任库 → 会报错，除非导入 CA

7 · 客户端收到 Certificate 后的处理（RFC 5246）

以下对应本抓包 TLS 栈在 PDU #2 收到 1637 B Certificate 之后、发送 PDU #3 之前的行为（结合 tls12_decode 分析）。

1解析 Certificate 消息 — 读取 certificate_list_length，循环读取每张 ASN.1 DER 证书，得到有序链（本例 2 张）。

2建立与验证证书链 — 从叶开始：检查 issuer 是否与下一级 subject 匹配；用上级公钥验证下级 signatureValue；直到自签根或命中本地信任锚（CAfile / 系统 store）。失败 → bad_certificate / unknown_ca alert。

3有效期 — notBefore ≤ now ≤ notAfter；失败 → certificate_expired / certificate_not_yet_valid。

4主机名 / 身份匹配 — ClientHello 中 SNI 为 172.18.25.17；叶证书 subject CN 相同则通过（v3 应优先检查 SAN DNS 名）。失败 → bad_certificate。

5密钥用途与策略 — 若有 EKU，须含 id-kp-serverAuth；若有 basicConstraints，CA 须标记 cA。本 v1 证书无扩展，依赖部署策略放宽。

6保存服务器公钥 — 从叶证书 subjectPublicKeyInfo 取得 RSA 2048 公钥，供下一步验证 ServerKeyExchange 签名（ECDHE 参数 + 曲线 + 临时公钥的 transcript 哈希）。

7处理 CertificateRequest（mTLS） — 服务器请求客户端证书；客户端从本地 store 选 CN=client 叶证书（PDU #3，813 B + 同一 CA），若无合适证书可发空 Certificate 消息。

8密钥交换与后续 — 生成 X25519 临时密钥，发 ClientKeyExchange；用客户端私钥对 transcript 做 CertificateVerify；派生 master_secret（本抓包协商 extended_master_secret）；CCS + Finished。

未过期openssl x509 -checkend 0未过期

SKE 绑定应用层：RSA_verify(transcript_hash, sig, server_pubkey)296 B SKE，RSA 签 2052 B
与加密的区别： Certificate 消息本身在握手中是明文；验证的是 X.509 逻辑与后续握手签名，而非解密证书。记录层 AES-GCM 在 ChangeCipherSpec 之后才保护应用数据。

8 · 证书公钥与 ServerKeyExchange 的关系

套件 TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384 (0xC030) 中：

  • ECDHE：premaster secret 来自 X25519 临时密钥交换，不用 RSA 加密 premaster。
  • RSA：服务器用证书中的 RSA 私钥对 ServerKeyExchange 参数签名，客户端用步骤 6 保存的公钥验证，防止中间人替换 ECDH 公钥。

客户端证书（mTLS，PDU #3）

客户端发送 CN=client 的叶证书 + CertificateVerify（260 B），服务器执行对称的链验证与签名验证，确认客户端持有私钥。提取文件：client_chain_0.der。

$ openssl x509 -in certs_from_pcap/client_chain_0.der -inform DER \
    -noout -subject -issuer
subject=... CN = client
issuer=... CN = ca
