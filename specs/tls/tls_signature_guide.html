数字签名算法原理与 TLS / 证书应用

说明非对称数字签名如何工作、RSA / ECDSA / EdDSA 等算法差异，以及在 X.509 证书与 TLS 握手中各自承担的身份认证、参数绑定与私钥持有证明。

Hash-then-Sign sha256WithRSAEncryption SKE / CertificateVerify Finished ≠ 数字签名

1 · 数字签名的原理

数字签名用签名者私钥对消息（或其哈希）产生一段比特串；任何人可用对应的公钥验证该签名是否由该私钥持有者生成，且消息未被篡改。

1.1 提供的安全属性

属性含义在 TLS/证书中的体现
完整性内容被改则验证失败证书字段、握手 transcript 被篡改会被发现
认证签名对应某一公钥身份确认「是某 CA / 某服务器 / 某客户端」签发或认可
不可否认（弱）签名者难以否认曾签过审计、合规场景；TLS 更侧重实时认证

1.2 Hash-then-Sign（先哈希再签名）

对长消息直接做 RSA 等运算是低效且不安全的。标准做法是先用密码学哈希函数 H 压缩，再对摘要签名：

M ——H()——> h = Hash(M) （如 SHA-256，32 字节） Sign: σ = Sign_private(h) Verify: Verify_public(h, σ) 是否为真

1.3 RSA 签名（概念）

签名：σ = h^d mod n （d 为私钥指数） 验证：h' = σ^e mod n，检查 h' 是否等于预期哈希（经 PKCS#1 填充编码）

实际使用 PKCS#1 v1.5 或 RSA-PSS 定义哈希与填充方式；二者不可混用。

1.4 ECDSA / EdDSA（概念）

  • ECDSA：在椭圆曲线群上运算，签名 (r,s) 与消息哈希、临时随机数 k 相关；k 重用会泄露私钥。
  • EdDSA（Ed25519）：确定性签名，无 k 重用问题；速度快，TLS 1.3 广泛支持。

2 · 数字签名 vs MAC（易混淆点）

数字签名（非对称）MAC（对称）
密钥公钥 / 私钥对双方共享的 master_secret 派生密钥
典型算法RSA-SHA256, ECDSA, Ed25519HMAC-SHA256, AES-GCM（AEAD 内置 MAC）
谁可验证任何持有公钥者仅持有相同对称密钥者
TLS 中的例子证书签名、ServerKeyExchange 签名、CertificateVerifyFinished 的 verify_data（PRF）、记录层 AEAD 认证标签
Finished 不是数字签名。 TLS 1.2 的 Finished 为 PRF(master_secret, "client finished", Hash(transcript)) 的 12 字节，双方用已协商的对称密钥验证，证明「持有相同 master_secret」，而非用证书公钥验证。

4 · 哈希与签名算法演化

  • MD5 / SHA-1：碰撞攻击可行 → 禁止用于新证书与 TLS（RFC 9155 等）。
  • SHA-256 / SHA-384：当前主流哈希，与 RSA/ECDSA 组合。
  • RSA-PSS：替代 PKCS#1 v1.5 用于 TLS 1.3 与新 CA 策略。
  • Ed25519：TLS 1.3、HTTP 证书透明度等场景增长迅速。

5 · X.509 证书中的签名

场景

证书颁发机构（CA）用自己的私钥对 tbsCertificate（待签名证书主体）的 DER 编码做哈希并签名，结果写入证书的 signatureValue 字段。

作用

  • 证明证书内容（身份、公钥、有效期）由该 CA 认可，未被篡改。
  • 将服务器/客户端身份绑定到某一公钥（SPKI）。
  • 为 TLS 提供信任锚：客户端信任 store 中的 CA → 验证链 → 使用叶证书公钥做后续握手验证。

本仓库实例

tls12_decode 叶证书：sha256WithRSAEncryption，由 CN=ca 签发。详见 证书导读。

项目值
签名算法sha256WithRSAEncryption
被签内容DER(tbsCertificate)
验证密钥上级 CA 证书中的 RSA 公钥

6 · TLS 1.2 中的签名场景

TLS 1.2 在握手中多处使用非对称签名，但用途各不相同。下表与序列图对应 RFC 5246。

场景谁签签什么谁验证作用
① 证书链 CA / 上级 CA 各张证书的 tbsCertificate 客户端（或服务器，mTLS） 身份与公钥可信
② ServerKeyExchange 服务器 参数 + 曲线 + 临时 DH/ECDH 公钥 + 客户端随机数 + 服务器随机数（按套件构造） 客户端 证明 ECDHE/DHE 参数未被 MITM 替换；绑定到证书公钥（RSA/ECDSA 签）
③ CertificateVerify 客户端（或服务器，若请求） 截至本消息前所有握手消息的哈希 对端 证明持有证书私钥（mTLS 客户端认证）

6.1 ServerKeyExchange 签名（ECDHE_RSA 为例）

signed_params = SHA256(ClientHello.random + ServerHello.random + ServerECDHParams) signature = RSA_sign(server_private_key, signed_params)

套件名 ECDHE_RSA_WITH_… 中 RSA 指签名算法，不是 RSA 密钥交换。客户端用证书中的 RSA 公钥验证，确认临时 ECDH 公钥确由该服务器签发。

6.2 CertificateVerify

对握手 transcript（从 ClientHello 到 CertificateVerify 之前所有握手消息）做哈希，再用实体私钥签名。证明「我拥有证书里的公钥对应的私钥」，与仅发送证书不同。

CertificateVerify 的哈希算法必须与 signature_algorithms 扩展协商结果一致；TLS 1.2 若未协商该扩展，对 TLS 1.2 连接有默认规则（见 RFC 5246）。

7 · TLS 1.3 中的签名变化

  • 无 ServerKeyExchange 消息：临时公钥在 key_share 扩展中交换；服务器身份通过加密的 CertificateVerify 等方式证明。
  • CertificateVerify：签名覆盖专用 transcript 哈希（Transcript-Hash），使用 signature_algorithms / 证书扩展中协商的算法（如 rsa_pss_pss_sha256、ecdsa_secp256r1_sha256、ed25519）。
  • 证书要求：支持 RSA-PSS、ECDSA、EdDSA；逐步淘汰 SHA-1、RSA PKCS#1 v1.5 用于 TLS 1.3。

TLS 1.3 握手后半段在流量密钥加密下传输，但 CertificateVerify 的密码学本质仍是「私钥对 transcript 哈希签名」。

8 · signature_algorithms 扩展

ClientHello / CertificateRequest 中可携带 signature_algorithms（及 TLS 1.3 的 signature_algorithms_cert），列出客户端支持的 (hash, signature) 对，例如：

TLS 表示含义
rsa_pkcs1_sha256RSASSA-PKCS1-v1_5 + SHA-256
rsa_pss_rsae_sha256RSA-PSS（RSAE）+ SHA-256
ecdsa_secp256r1_sha256ECDSA P-256 + SHA-256
ed25519Ed25519 纯签名（无单独 hash 枚举）

服务器选 SKE / CertificateVerify 算法时须落在交集中；证书本身的签名算法（如 sha256WithRSAEncryption）也须在 TLS 1.3 的 signature_algorithms_cert 支持范围内。

本抓包 ClientHello 声明 RSA-PSS、ECDSA 等，实际 SKE 使用 RSA PKCS#1 + SHA-384（与套件 SHA-384 PRF 一致）。

9 · tls12_decode 中的三处签名

#类型算法（推断）大小验证方
①服务器叶证书sha256WithRSAEncryption256 B 签名客户端用 CA 公钥
②ServerKeyExchangeRSA + SHA-384（套件 0xC030）~2052 B客户端用叶证书 RSA 公钥

10 · OpenSSL 验证示例

10.1 验证证书链签名（CA 是否签发叶证书）

$ openssl verify -CAfile certs_from_pcap/server_ca.pem \
    certs_from_pcap/server_leaf.pem
server_leaf.pem: OK

内部即：用 CA 公钥验证叶证书 signatureValue 与 tbsCertificate 哈希是否匹配。

10.2 查看证书签名算法

$ openssl x509 -in certs_from_pcap/server_leaf.pem -noout -text \
    | grep -A1 "Signature Algorithm"
    Signature Algorithm: sha256WithRSAEncryption
    Signature Algorithm: sha256WithRSAEncryption

10.3 从证书取出公钥（用于理解 SKE 验证）

$ openssl x509 -in certs_from_pcap/server_leaf.pem -noout -pubkey \
    -out server_pubkey.pem

TLS 栈在运行时对 SKE 中的签名调用等价于 RSA_verify / EVP_DigestVerify，输入为构造的 transcript 哈希与签名比特串。

11 · 一图总览：签名在信任链中的位置

层次机制解决的问题
PKI / 证书CA 对 tbs 的签名「这个公钥属于谁」
TLS 握手（ECDHE）SKE 签名「这次临时密钥确由该公钥持有者认可」
TLS 握手（mTLS）CertificateVerify「对端确实掌握证书私钥」
TLS 记录层AEAD / Finished「后续流量来自同一 master_secret」
规范： RFC 5280（证书）、 RFC 8017（RSA）、 RFC 5246 §7.4.3 / §7.4.8（TLS 1.2）、 RFC 8446 §4.4.3（TLS 1.3 CertificateVerify）。
