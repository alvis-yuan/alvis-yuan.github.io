TLS 1.2 解密抓包协议分析

基于 Wireshark 导出的 tls12_decode.pcapng（6 个 PDU），结合 RFC 5246 解读完整握手、双向证书与密钥切换。

TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384X25519SNI 172.18.25.17extended_master_secret

1 · 抓包摘要

本文件为 Wireshark「Export PDU」格式：每条 PDU 含 TCP 元数据 + TLS 记录明文（Finished 在导出中仍为 AEAD 密文，见下文标注）。

PDUΔt方向内容
#10 msC→SClientHello（204B）
#20.82 msS→CServerHello + Certificate + SKE + CertRequest + ServerHelloDone
#32.27 msC→SCertificate + ClientKeyExchange + CertificateVerify + CCS + Finishedenc
#42.62 msS→CNewSessionTicket + CCS + Finishedenc
#5–#63.3 ms双向Alert（加密记录）
协商套件 0xC030：ECDHE 临时密钥（曲线 x25519）+ RSA 证书签名 SKE + AES-256-GCM 记录保护 + SHA-384 PRF。

3 · 与 RFC 5246 协议对照

RFC 5246 要求本抓包体现
ClientHello 携带版本、随机数、套件列表、扩展TLS 1.2、28 个套件、SNI/curves/sig_algs/EMS
ECDHE 套件须 ServerKeyExchange296B SKE，named_curve x25519，RSA 签名 2052B
可选 CertificateRequest → 客户端证书服务器请求证书，客户端回 1631B 证书链
premaster → master_secret → key_blockCKE 含 32B 客户端 ECDH 公钥，双方本地 ECDH
ChangeCipherSpec 后发送 Finished双方各 CCS(1) + 40B 加密 Finished 记录
扩展 extended_master_secret双方 ClientHello/ServerHello 均协商 EMS

密钥派生公式见 RFC 5246 导读 §8；ECC 字段见 RFC 4492 导读。

5 · 服务器 Hello 阶段（PDU #2）

ServerHello

参数值
cipher_suite0xC030 TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
server_random043c3b6c…47524401（末 4 字节含 ASCII 特征）
extensionsrenegotiation_info, ec_point_formats, session_ticket, EMS

客户端处理：锁定 PRF=SHA-384、AEAD=AES-256-GCM；保存 server_random；等待后续握手消息。

Certificate（1637 B）

DER X.509 证书链。客户端验证 serverAuth EKU、链信任、与 SNI 主机名匹配。

ServerKeyExchange（296 B）

字段值
curve_typenamed_curve (3)
named_curve29 → x25519
server_pubkey32 字节临时公钥
signature2052 字节（RSA 签 transcript）

客户端处理：验证 SKE 签名 → 存 Y_s → 待 CKE 计算 premaster = X25519(Y_s, x_c)。

CertificateRequest + ServerHelloDone

请求 RSA/DSS/ECDSA 类型客户端证书；ServerHelloDone 触发客户端发送认证与密钥交换消息。

6 · 客户端认证（PDU #3）

消息长度处理
Certificate1631 B服务器验证 mTLS 客户端链
ClientKeyExchange33 B32B ECDH 公钥 28d5f876…ef75415；双方得相同 premaster
CertificateVerify260 BRSA/ECDSA 签 handshake transcript；证明持有客户端私钥

随后派生：master_secret = PRF(premaster, "master secret", randoms)，key_block = PRF(master, "key expansion", randoms)，切分 client/server write key（GCM）。

7 · ChangeCipherSpec 与 Finished

ChangeCipherSpec：单字节 0x01，激活 AEAD 写密钥（RFC 5246 §6.2.1，独立记录类型）。

Finished（40 B 记录）：导出 PDU 中仍为 AES-GCM 密文（首字节非 HandshakeType=20）。语义上为 PRF(master, "client finished", Hash(transcript)) 的 12 字节 verify_data + AEAD 封装与填充。

服务器在 PDU #4 回送 NewSessionTicket（998B 明文）→ CCS → 加密 Finished；双方互验 verify_data 成功后进入应用数据阶段。

8 · NewSessionTicket 与 Alert

NewSessionTicket（PDU #4，998B）：RFC 5077 会话票据，供后续 ClientHello session_ticket 扩展恢复会话。

Alert（PDU #5–#6）：导出中为 26B 加密 alert 记录；按时序推断为连接关闭相关的 close_notify 交换。
