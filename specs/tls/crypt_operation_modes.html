01
简介与文档来源
Block Cipher Mode of Operation — Definition

官方定义（NIST CSRC）：分组密码工作模式（简称"模式"）是一种使用分组密码来提供信息服务（如机密性或认证）的方法。

分组密码本身每次只能加密固定长度的数据块（AES 为 128 位）。当需要加密任意长度的消息时，就必须通过"工作模式"将多次单块加密操作组合为一次连贯的整体加密操作。

NIST 在 SP 800-38 系列标准中规定了所有经批准的工作模式，每一种模式都针对不同的应用场景和安全需求。

📌 历史背景 最早的四种模式（ECB、CBC、CFB、OFB）由 NBS（现 NIST 前身）于 1980 年在 FIPS Pub. 81 中为 DES 算法标准化。CTR 模式在 2001 年由 SP 800-38A 正式标准化，GCM 则在 2007 年由 SP 800-38D 标准化，反映了数十年来密码学界对"模式"安全属性认知的不断深化。

02
NIST SP 800-38 系列标准
官方链接与规范内容

NIST 在 SP 800-38 系列规范中定义了所有经批准的分组密码工作模式。以下按发布顺序列出：

SP 800-38A
五种机密性模式（2001年，M. Dworkin）
规定 ECB、CBC、CFB、OFB、CTR 五种模式，适用于任何 FIPS 批准的分组密码算法（如 AES）。附录还规定了 CBC 密文窃取（Ciphertext Stealing）的三个变体：CBC-CS1、CBC-CS2、CBC-CS3。
→ 官方页面
SP 800-38B
CMAC — 基于密码的消息认证码（2005年）
规定 CMAC 认证模式，适用于任何 FIPS 批准的分组密码，本质上是 One-Key CBC-MAC（OMAC）算法，类似于基于哈希的 HMAC。
→ 官方页面
SP 800-38C
CCM — 带 CBC-MAC 的计数器模式（2004年）
规定 CCM 模式（AES 专用）：结合 CTR 模式提供机密性与 CBC-MAC 提供认证，属于带关联数据的认证加密（AEAD）。最初与 IEEE 802.11 无线局域网草案兼容。
→ 官方页面
SP 800-38D
GCM / GMAC — 伽罗瓦/计数器模式（2007年）
规定 GCM（带关联数据的认证加密）及其特化版本 GMAC（仅生成消息认证码，不加密数据）。GCM 基于通用哈希函数，专为高吞吐量硬件实现而设计，亦支持软件优化。NIST 已决定修订本规范（2024年3月公告）。
→ 官方页面
SP 800-38E
XTS-AES — 存储设备机密性模式
批准 XTS-AES 模式用于存储设备的机密性保护（XTS = XEX Tweakable block cipher with ciphertext Stealing）。算法规范收录于 IEEE Std 1619-2007。
→ 官方页面
SP 800-38F
密钥封装模式（2012年）
规定 AES 密钥封装（KW）、带填充的 AES 密钥封装（KWP）及 TDEA 密钥封装（TKW）模式，用于密码密钥的机密性与完整性保护。
→ 官方页面
SP 800-38G
FPE — 格式保留加密（2016年）
规定 FF1 和 FF3 格式保留加密模式，专为非二进制数据（如信用卡号、社会安全号码）设计，便于在无法使用传统加密模式的现有系统中进行改造加密。
→ 官方页面
03
ECB — 电子密码本模式
Electronic Codebook · SP 800-38A § 6.1
ECB
电子密码本 Electronic Codebook
规范来源：NIST SP 800-38A，第 6.1 节

机制：ECB 是最简单的工作模式。每个明文块独立地用同一密钥直接加密，块与块之间没有任何依赖关系。

// ECB 加密（SP 800-38A, §6.1） C_j = CIPH_K(P_j), j = 1, 2, ..., n // ECB 解密 P_j = CIPH_K⁻¹(C_j), j = 1, 2, ..., n // 其中: P_j = 明文块j, C_j = 密文块j, K = 密钥 // 无需 IV；最后一块必须补齐到块大小
ECB 加密流程图
⚠ 安全警告（SP 800-38A） ECB 模式不提供语义安全性：相同的明文块总是产生相同的密文块，会泄露明文的模式和结构信息。NIST 文档明确指出，ECB 不适用于超过一个块长度的数据加密。
不推荐用于多块数据 可并行加密/解密 需填充 无需 IV 确定性（相同输入→相同输出）
04
CBC — 密码块链接模式
Cipher Block Chaining · SP 800-38A § 6.2
CBC
密码块链接 Cipher Block Chaining
规范来源：NIST SP 800-38A，第 6.2 节

机制：每个明文块在加密之前，先与前一个密文块进行 XOR 运算。第一个块使用随机初始化向量（IV）代替"前一个密文块"，使得 IV 成为加密链的起点。

// CBC 加密（SP 800-38A, §6.2） C₀ = IV C_j = CIPH_K(P_j ⊕ C_{j-1}), j = 1, ..., n // CBC 解密 C₀ = IV P_j = CIPH_K⁻¹(C_j) ⊕ C_{j-1}, j = 1, ..., n // IV: 随机且不可预测的初始化向量（长度等于块大小） // ⊕: 按位 XOR；解密可并行，加密不可并行
CBC 加密流程图（块间链式依赖）
📌 SP 800-38A 要求 IV 必须是随机且不可预测的（unpredictable）。若 IV 可预测，CBC 将面临选择明文攻击（BEAST 攻击即利用了 TLS 1.0 中 IV 可预测的问题）。此外，SP 800-38A 文档还指出，CBC 模式的加密过程无法并行，但解密过程可以并行。
广泛使用 解密可并行 加密不可并行 需填充 需随机 IV 错误传播限两块
05
CFB — 密文反馈模式
Cipher Feedback · SP 800-38A § 6.3
CFB
密文反馈 Cipher Feedback
规范来源：NIST SP 800-38A，第 6.3 节

机制：CFB 将分组密码转变为流密码。加密的是前一个密文块（而非明文块），产生密钥流段，再与当前明文段进行 XOR。SP 800-38A 定义了通用的 CFB 模式，支持 1 到 b 位的数据段（b 为块大小）。

// CFB 加密（SP 800-38A, §6.3） I₁ = IV I_j = LSB_{b-s}(I_{j-1}) || C*_{j-1}, j = 2, ..., n O_j = CIPH_K(I_j) C*_j = P*_j ⊕ MSB_s(O_j) // CFB 解密 P*_j = C*_j ⊕ MSB_s(O_j) // s: 数据段位数(1≤s≤b); MSB_s: 取最高 s 位 // 加密的是密文（移位寄存器），再与明文 XOR
CFB 加密流程图（密文反馈到移位寄存器）
📌 SP 800-38A 说明 CFB 模式将分组密码转换为自同步流密码（self-synchronizing stream cipher）。错误会传播到当前段和随后数个段（取决于段大小 s）。NIST 文档指出，CFB-8（s=8）常用于处理字节流。
流密码，无需填充 解密可并行 加密不可并行 需 IV 误差扩散
06
OFB — 输出反馈模式
Output Feedback · SP 800-38A § 6.4
OFB
输出反馈 Output Feedback
规范来源：NIST SP 800-38A，第 6.4 节

机制：OFB 将分组密码转变为同步流密码。密钥流独立于明文和密文生成，通过反复加密 IV 产生。明文与密钥流 XOR 得到密文。与 CFB 的关键区别在于：OFB 反馈的是加密函数的输出（非密文），因此密钥流可提前独立生成。

// OFB 加密（SP 800-38A, §6.4） I₁ = IV I_j = O_{j-1}, j = 2, ..., n // 以上一轮输出作为输入 O_j = CIPH_K(I_j) C_j = P_j ⊕ O_j // 明文 ⊕ 密钥流块 // OFB 解密（结构相同） P_j = C_j ⊕ O_j // 密钥流完全独立于明文/密文 // IV 必须对每次加密唯一（不可重用）
OFB 加密流程图（输出反馈，密钥流独立生成）
✅ 误差不扩散 由于密钥流完全独立于密文，某一位密文的损坏只影响对应位置的明文解密，不会传播到其他块。这使 OFB 在有损信道（如卫星通信）中更有优势。
⚠ IV 唯一性要求（SP 800-38A） 相同的密钥与 IV 组合必须绝不重用——否则两次加密会产生相同的密钥流，使攻击者能通过 XOR 两个密文直接恢复明文的 XOR 值，完全破坏机密性。
流密码，无需填充 误差不扩散 加解密均不可并行 需唯一 IV
07
CTR — 计数器模式
Counter · SP 800-38A § 6.5
CTR
计数器 Counter Mode
规范来源：NIST SP 800-38A，第 6.5 节

机制：CTR 模式通过加密一组逐块递增的计数器值来产生密钥流，再与明文 XOR。与 OFB 类似，密钥流独立于明文；但 CTR 使用计数器（而非链式输出），使得每个密钥流块可以完全独立计算，支持真正的并行化和随机访问。

// CTR 加密（SP 800-38A, §6.5） O_j = CIPH_K(T_j), j = 1, ..., n // T_j: 计数器块 j C_j = P_j ⊕ O_j // CTR 解密（结构相同） O_j = CIPH_K(T_j) P_j = C_j ⊕ O_j // T_j: 计数器序列，每块递增（Nonce || Counter） // 各块完全独立 → 可完全并行 → 支持随机访问 // 计数器值（Nonce）每次加密必须唯一，绝不重用
CTR 加密流程图（独立计数器，完全可并行）
✅ SP 800-38A 推荐特性 CTR 是五种机密性模式中并行化最佳的模式：加密和解密均可完全并行；支持任意位置的随机访问解密；不需要填充（任意长度数据均可处理）；解密只需要正向加密函数，无需实现解密函数，简化了硬件设计。
加解密完全并行 无需填充 支持随机访问 高性能 需唯一 Nonce 仅机密性，无完整性
08
CMAC — 基于密码的消息认证码
Cipher-based MAC · SP 800-38B
CMAC
基于密码的消息认证码 Cipher-based MAC
规范来源：NIST SP 800-38B

机制：CMAC 是一种认证模式（不提供加密），用于验证数据的完整性与真实性。其本质是 One-Key CBC-MAC（OMAC）算法：使用分组密码的 CBC 模式处理消息，最终块的输出经过特殊子密钥派生处理后作为 MAC 标签。

// CMAC 生成（SP 800-38B） // Step 1: 派生子密钥 K1, K2（从 CIPH_K(0^b) 推导） K1 = CIPH_K(0^b) << 1 [若最高位=1, 则 XOR Rb] K2 = K1 << 1 [若最高位=1, 则 XOR Rb] // Step 2: 处理消息块 X₀ = 0^b X_j = CIPH_K(M_j ⊕ X_{j-1}), j = 1,...,n-1 // 最后一块加入子密钥 T = CIPH_K(M*_n ⊕ K_t ⊕ X_{n-1}) // T: MAC 标签 // 仅提供完整性/认证，不提供机密性 // 类似于基于哈希的 HMAC，但使用分组密码
📌 SP 800-38B 说明 CMAC 是 OMAC（One-Key CBC-MAC）的改进版本，由 Iwata 和 Kurosawa 提出。NIST 文档指出 CMAC 在安全性上优于原始 CBC-MAC，因为它对不同长度的消息使用了域分离（通过子密钥 K1/K2）。
完整性验证 真实性保证 不提供机密性 类似 HMAC
09
CCM — 带 CBC-MAC 的计数器模式
Counter with CBC-MAC · SP 800-38C
CCM
带 CBC-MAC 的计数器模式 Counter with CBC-MAC
规范来源：NIST SP 800-38C

机制：CCM 是一种 AEAD（带关联数据的认证加密）模式，结合了两种已有技术：CTR 模式提供机密性，CBC-MAC 提供认证。CCM 采用 MtE（MAC-then-Encrypt）结构：先用 CBC-MAC 计算认证标签，再用 CTR 模式加密明文和标签。

// CCM 加密（SP 800-38C，AES 专用） // Phase 1: CBC-MAC 认证（生成 MAC 标签 T） T = CBC-MAC_K(flags || Nonce || len(P) || A || P) // Phase 2: CTR 模式加密 C = CTR_K(P) // 加密明文 S₀ = CIPH_K(A₀) Tag = T ⊕ MSBₜ(S₀) // 加密 MAC 标签 // 输出: C || Tag // A: 关联数据（仅认证不加密）; Nonce 必须唯一 // NIST 最初为 IEEE 802.11 无线局域网设计
📌 SP 800-38C 说明 CCM 模式是 AES 专用的 AEAD 模式。其主要限制是：必须提前知道明文长度才能开始处理，因此不适合流式处理。GCM 在这方面更灵活。
AEAD — 同时加密+认证 防篡改 关联数据保护 需提前知道消息长度 AES 专用
10
GCM / GMAC — 伽罗瓦/计数器模式
Galois/Counter Mode · SP 800-38D
GCM
伽罗瓦/计数器模式 Galois/Counter Mode
规范来源：NIST SP 800-38D（2007年，已决定修订）

机制：GCM 是目前最广泛使用的 AEAD 模式（如 TLS 1.3）。它将 CTR 模式的机密性与基于通用哈希函数（GHASH）的认证相结合，采用 EtM（Encrypt-then-MAC）结构。NIST 文档指出 GCM 专为高吞吐量硬件实现而设计，同时也支持软件优化（通过预计算查找表）。

// GCM 加密（SP 800-38D） // 1. 生成哈希子密钥 H H = CIPH_K(0^128) // 2. 生成初始计数器块 J₀ J₀ = IV || 0^31 || 1 // 当 len(IV)=96 时 // 3. CTR 模式加密明文 C = GCTR_K(inc₃₂(J₀), P) // 4. GHASH 认证（覆盖关联数据 A 和密文 C） S = GHASH_H(A || 0* || C || 0* || len(A) || len(C)) T = MSBₜ(GCTR_K(J₀, S)) // T: 认证标签 // GHASH: GF(2^128) 上的多项式哈希，硬件可用 CLMUL 指令加速 // GMAC: 当 P=空 时，GCM 退化为 GMAC（仅认证，不加密）
GCM 架构图（CTR 加密 + GHASH 认证）
✅ SP 800-38D 推荐原因 GCM 同时保证数据机密性与完整性（防篡改），支持额外的关联数据（AAD）认证（如报头），且加密与认证均可并行。NIST 文档明确指出 GCM 专为高吞吐量硬件实现而设计，是目前 TLS 1.3、IPsec、SSH 等协议的首选模式。
⚠ Nonce 重用的灾难性后果 相同密钥与 Nonce 的重用不仅泄露明文 XOR 值，还会彻底暴露 GHASH 认证密钥 H，导致攻击者可以伪造任意合法密文。GCM 的 Nonce（推荐 96 位）必须对每次加密操作保持唯一。
AEAD — 业界首选 加解密完全并行 防篡改 TLS 1.3 必选 关联数据保护 Nonce 绝不可重用

12
选型建议
基于 NIST 标准的现代应用推荐

🔐 现代系统首选：AES-GCM (SP 800-38D)

对于绝大多数需要机密性的场景，NIST 推荐使用 GCM 模式（带关联数据的认证加密）。它同时满足：机密性、完整性、真实性，且可并行处理，是 TLS 1.3、IPsec、WireGuard 等现代安全协议的核心加密原语。

✅ 推荐使用场景 GCM：网络传输加密（TLS 1.3）、API 数据保护、云存储、区块链
CTR：需要高性能流加密但另有完整性机制时（磁盘加密结合 HMAC）
CCM：IoT 设备、内存受限场景（IEEE 802.11i/WPA2、ZigBee）
CBC：遗留系统兼容（必须配合 Encrypt-then-MAC，如 HMAC-SHA256）
⛔ 避免使用 ECB：绝对不应用于加密超过一个块的实际数据（NIST 文档明确指出其不安全性）
OFB/CFB：在现代系统中已被 CTR/GCM 取代，较少使用
单独使用 CBC/CTR：不提供完整性保护，必须配合 MAC 使用
加密模式选型决策树
GCM = 首选 AEAD CTR = 高性能流加密 CCM = IoT 受限环境 CBC = 遗留兼容（需配合 MAC） ECB = 避免（多块数据）
