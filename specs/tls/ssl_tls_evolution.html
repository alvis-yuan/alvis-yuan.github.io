从 SSL 到 TLS 1.3：协议演化全景

梳理 Netscape SSL、IETF TLS 各主版本的技术演进：每一代解决了什么、留下了什么隐患，以及最终被哪些攻击与标准淘汰。官方规范以 RFC Editor 为准。

1 · 引言：为何需要不断演进

传输层安全协议的目标始终是：在不可信网络上为应用层提供机密性、完整性、身份认证。三十年来，密码分析进步、算力增长、实现缺陷与部署误用不断暴露旧协议弱点，推动标准从 SSL 走向 TLS 1.3。

命名说明：1999 年起 IETF 将 SSL 3.0 之后的发展称为 TLS（Transport Layer Security），但习惯上仍统称「SSL/TLS」。本文按时间线列出各主版本；SSL 1.0 从未公开发布。

2 · 演化时间线

演进主线：更强原语（MD5/SHA-1 → AEAD/HKDF）→ 更少选项（协商面缩小）→ 更快握手（1-RTT / 0-RTT）→ 默认前向保密。

5 · SSL 2.0（1995，已禁止）

Netscape 早期商用协议，无正式 IETF RFC，现已被所有合规实现禁止。

当时解决的问题

  • 首次在浏览器与服务器间提供加密通道（相对明文 HTTP）
  • 引入会话标识、密码套件协商概念

遗留/固有缺陷（导致淘汰）

  • 消息无绑定：同一密文可出现在不同上下文中（跨连接攻击）
  • 使用弱 MD5/RC4 组合；支持 EXPORT 级弱加密
  • 握手可被截断重放；无可靠的版本降级保护
  • 与 SSL 3.0 不兼容，且存在 RFC 6176 级禁止部署指导

→ 被 SSL 3.0 完全取代；2016 年 RFC 7568 明确禁止 TLS 实现兼容 SSLv2。

6 · SSL 3.0（1996）— RFC 6101

现代 TLS 记录层与握手结构的真正奠基者，许多概念沿用至今。

相对 SSL 2 解决的问题

  • 重新设计记录格式：HMAC 完整性 + 分片
  • 主密钥派生、密钥块切分、Finished 验证机制
  • 支持 RSA、Diffie-Hellman 等多种密钥交换
  • Alert、ChangeCipherSpec、扩展协商雏形

仍存在的问题（后被 TLS 1.x 修补）

  • CBC + MAC-then-Encrypt → POODLE（2014）后必须禁用
  • 无 TLS 1.1 的显式随机 IV，CBC 易受 BEAST 类攻击
  • 支持压缩 → 催生 CRIME
  • 无 AEAD；套件过多导致「最差公共套件」降级
  • 无标准扩展框架（后续 TLS 扩展补入）

RFC 7568（2015）禁止在 TLS 实现中降级到 SSL 3.0。

7 · TLS 1.0（1999）— RFC 2246

IETF 标准化后的首个 TLS 版本，与 SSL 3.0 差异较小，版本号 0x0301。

相对 SSL 3.0 解决的问题

  • 统一 IETF 标准流程与实现互操作测试
  • HMAC 替代 SSL 3.0 的 ad-hoc MAC 构造
  • 正式定义 TLS 扩展框架（后由 RFC 6066 等扩展）
  • 消除 FORTEZZA 等过时算法

仍存在的问题

  • 默认 CBC 模式 + 隐式 IV → BEAST（2011）
  • 仍广泛支持 RC4、MD5、NULL、EXPORT
  • PRF 基于 MD5⊕SHA-1，已不符合现代密码强度
  • RSA 密钥传输无 PFS → 长期私钥泄露可解密历史流量
  • Renegotiation 不安全（后由 RFC 5746 修补）

RFC 8996（2020）宣布 TLS 1.0/1.1 不再适合一般用途。

8 · TLS 1.1（2006）— RFC 4346

版本号 0x0302，改动相对保守，主要为密码细节修补。

相对 TLS 1.0 解决的问题

  • 显式随机 IV 写入每个 CBC 记录 → 缓解 BEAST 类攻击
  • 禁止弱 IV 的 CBC 套件组合
  • 对 IMPLAUSIBLE 的 alert 与错误处理更清晰

仍存在的问题

  • 仍依赖 MAC-then-Encrypt CBC；未引入 AEAD
  • 仍允许 RC4、RSA 密钥传输、压缩、重协商
  • 握手仍为 2-RTT，性能无质变
  • 部署窗口短，很快被 1.2 超越

9 · TLS 1.2（2008）— RFC 5246

版本号 0x0303，长期作为互联网主力协议；本仓库有 RFC 5246 导读 与 解密抓包分析。

相对 TLS 1.1 解决的问题

  • AEAD 密码套件（GCM、CCM）— 统一加密与 MAC，消除 CBC 填充 oracle
  • PRF 与哈希解耦：每套件指定 SHA-256 等（告别 MD5⊕SHA-1）
  • 扩展 AES-GCM、ECC（RFC 4492）、签名算法扩展
  • 更安全默认可通过配置实现：ECDHE + AEAD
  • 密钥导出 Exporter（RFC 5705）

仍存在的问题（推动 TLS 1.3）

  • 仍允许 RSA 密钥传输（无默认 PFS）→ ROBOT 等
  • 仍允许 CBC、RC4、SHA-1 套件（需运维手动禁用）
  • 握手明文暴露证书与参数 → 指纹与隐私问题
  • 2-RTT 握手；重协商复杂且有历史漏洞
  • 压缩、重协商、ChangeCipherSpec 增加实现复杂度
  • 套件与扩展组合爆炸 → 互操作与安全配置困难
典型现代配置：TLS_ECDHE_*_WITH_AES_*_GCM_* + 禁用 1.0/1.1/RC4/3DES/RSA-kex

10 · TLS 1.3（2018）— RFC 8446

版本通过 supported_versions 扩展协商为 0x0304；本仓库有 RFC 8446 导读 与 解密抓包分析。

相对 TLS 1.2 解决的问题

  • 仅 AEAD：移除 CBC、RC4、NULL、3DES、EXPORT
  • 移除 RSA 密钥传输：仅 (EC)DHE 与 PSK → 默认前向保密
  • 1-RTT 握手（ServerHello 后即可应用数据）；可选 0-RTT
  • 握手加密（除 ClientHello/ServerHello 外）— EncryptedExtensions 起
  • HKDF 统一密钥调度，取代 PRF + master_secret 术语
  • 移除压缩、重协商、ChangeCipherSpec（兼容模式除外）、多余套件
  • 降级保护：ServerHello.random 末字节 sentinel
  • 更清晰的 0-RTT 反重放与 Exporter 语义（更新 RFC 5705）

仍须关注的权衡 / 新问题

  • 0-RTT 无前向保密且可重放 → 应用须幂等
  • 中间盒对 TLS 1.3 的干扰（需兼容模式 CCS）
  • 实现复杂度高（多阶段 traffic secret）
  • 与仅支持 1.2 的旧客户端/设备互操作需降级至 1.2
  • 配置错误（允许 1.0/弱套件回退）仍是运维风险
部署建议（RFC 8446 / BCP）：优先 TLS 1.3；必要时保留 TLS 1.2（ECDHE + AEAD）；禁用 TLS 1.0/1.1 与 SSL；使用 RFC 7525 / RFC 9325 推荐算法套件。
