1.1 端到端拓扑总览

一个典型的嵌入式Linux以太网系统由以下层次构成：应用层 → 内核网络协议栈 → MAC 控制器 → PHY 芯片 → RJ45/介质。Linux内核使用 PHY 抽象层（PHYLIB）将PHY设备与MAC驱动解耦，实现代码最大复用。

📌 第一性原理 以太网的本质是：将比特流可靠地从一个MAC地址传输到另一个MAC地址。MAC负责"帧"的边界与完整性，PHY负责"信号"的物理传输。两者之间通过标准化的电气接口（MII族）解耦，使得不同厂商的MAC和PHY可以自由组合。

2.2 MAC 子层功能

MAC（Media Access Control）子层是数据链路层的下半部分，主要负责：

  • 帧封装/解封装：添加/去除 Preamble、SFD、FCS
  • 地址识别：判断目标MAC是否为本机（单播、组播、广播过滤）
  • CRC生成与校验：发送时计算FCS，接收时校验，错误则丢弃
  • 帧间隙控制：保证12字节IPG，防止冲突
  • CSMA/CD（半双工）：载波侦听、碰撞检测、退避重传（IEEE 802.3规定，全双工模式下不需要）
  • 流量控制：暂停帧（Pause Frame，802.3x）的收发
  • 硬件卸载：TSO（TCP分段卸载）、GSO、校验和卸载（CSUM offload）

4.1 MII（Media Independent Interface）

MII 由 IEEE 802.3u 标准定义，用于连接 10/100Mbps MAC 与 PHY。最核心的特点是"介质无关"——不管使用双绞线还是光纤，MAC硬件无需改变。

信号名方向说明
TX_CLKPHY → MAC发送时钟：100Mbps=25MHz，10Mbps=2.5MHz
TXD[3:0]MAC → PHY4位发送数据，随TX_CLK上升沿有效
TX_ENMAC → PHY发送使能，高电平表示TXD有效
TX_ERMAC → PHY发送错误，高电平发送错误符号
RX_CLKPHY → MAC接收时钟，频率同TX_CLK
RXD[3:0]PHY → MAC4位接收数据
RX_DVPHY → MAC接收数据有效指示
RX_ERPHY → MAC接收错误指示
CRSPHY → MAC载波侦听（半双工用）
COLPHY → MAC冲突检测（半双工用）
MDCMAC → PHY管理时钟，最高2.5MHz
MDIO双向管理数据，串行，可总线共享
📌 MII 信号计数 标准MII共18条信号：7条发送（TX_CLK、TXD0-3、TX_EN、TX_ER）+ 9条接收（RX_CLK、RXD0-3、RX_DV、RX_ER、CRS、COL）+ 2条管理（MDC、MDIO）。8口交换机MII需要 8×16+2=130 条信号线，这推动了RMII的出现。

RGMII 时序分析
关键时序约束、延迟配置与常见问题诊断（源自 kernel.org 官方文档）

5.1 核心时序约束（官方原文）

原文引用（kernel.org phy.txt）： "The Reduced Gigabit Medium Independent Interface (RGMII) is a 12-pin electrical signal interface using a synchronous 125MHz clock signal and several data lines. Due to this design decision, a 1.5ns to 2ns delay must be added between the clock line (RXC or TXC) and the data lines to let the PHY (clock sink) have enough setup and hold times to sample the data lines correctly."
图5-1 RGMII 时序约束示意（基于 kernel.org phy.txt 官方说明）

5.2 官方推荐延迟配置原则

✅ 官方建议（kernel.org）：尽量使用 PHY 侧 RGMII 延迟，原因：
  • PHY 设备可提供亚纳秒精度的延迟配置（如 0.5/1.0/1.5ns），能补偿不同PCB走线长度
  • PHY 通常经过工业/医疗/汽车级温压电压范围认证，延迟值在极端条件下仍可靠
  • PHYLIB 中的 PHY 驱动具有复用性，正确延迟配置可使更多设计受益

5.3 延迟不匹配的常见症状（官方原文）

⚠️ 官方描述（kernel.org phy.txt）延迟不匹配症状：
  • 传输/接收部分可用，存在频繁或偶发的丢包
  • MAC 报告部分或全部收到的帧存在 FCS/CRC 错误，或直接丢弃
  • 切换到 10/100Mbps 后问题消失（因为更低速度下建立/保持时间裕量更大）

MDIO 管理总线
MDC/MDIO 协议时序、Clause 22/45、帧格式与Linux mii_bus

6.1 MDIO 协议特性

MDIO（Management Data Input/Output）是 MAC 与 PHY 之间的管理总线，类似 I²C：

  • MDC：管理时钟，由 MAC 驱动，最高 2.5MHz（IEEE 802.3 规定）
  • MDIO：双向串行数据，总线共享（多PHY共享MDC/MDIO）
  • Clause 22：5bit PHY地址 + 5bit寄存器地址，32寄存器，16bit数据
  • Clause 45：5bit PHY地址 + 5bit设备类型 + 16bit寄存器地址，扩展寄存器空间
图6-1 MDIO Clause 22 读帧格式与 Linux mii_bus 接口

6.2 Linux MDIO 总线驱动要点

根据 kernel.org/doc/Documentation/networking/phy.txt 官方文档：

  • 每个MDIO总线接口需作为独立设备注册
  • 必须实现 read() 和 write() 函数
  • 可选实现 reset() 函数
  • 通过 mdiobus_register() 注册到PAL，mdiobus_unregister() 注销
  • 这些函数保证不在中断上下文调用，可以阻塞等待操作完成
/* MDIO 总线驱动框架示例（参考 drivers/net/ethernet/freescale/fsl_pq_mdio.c） */
static int my_mdio_read(struct mii_bus *bus, int mii_id, int regnum)
{
    struct my_mdio_priv *priv = bus->priv;
    /* 向硬件寄存器写PHY地址和寄存器地址，触发读操作 */
    writel(MDIO_READ | (mii_id << 21) | (regnum << 16), priv->base + MDIO_CMD);
    /* 等待操作完成（可sleep，非中断上下文） */
    return wait_for_mdio(priv);
}

static int my_mdio_probe(struct platform_device *pdev)
{
    struct mii_bus *bus = mdiobus_alloc();
    bus->read  = my_mdio_read;
    bus->write = my_mdio_write;
    bus->name  = "my-mdio";
    snprintf(bus->id, MII_BUS_ID_SIZE, "%s", dev_name(&pdev->dev));
    return mdiobus_register(bus);
}

7.1 PHYLIB 设计目标（官方原文）

原文（kernel.org phy.txt）： "While these devices are distinct from the network devices, and conform to a standard layout for the registers, it has been common practice to integrate the PHY management code with the network driver. This has resulted in large amounts of redundant code."

PHYLIB 目标：1) 增加代码复用；2) 提高整体可维护性；3) 加快新网络驱动和新系统的开发速度。

7.3 MAC 驱动连接 PHY 的方式

根据官方文档，PAL提供4种连接方式：

方式描述适用场景
方式1PAL处理一切，链路变化时回调驱动绝大多数情况（推荐）
方式2PAL处理除中断外的一切MAC控制器自带中断寄存器
方式3PAL每秒回调驱动，驱动先响应需要更精细控制

/* 推荐的 PHY 连接方式（来源：kernel.org phy.txt 方式1） */

/* 1. 链路状态变化回调 */
static void adjust_link(struct net_device *dev)
{
    struct phy_device *phydev = dev->phydev;
    if (phydev->link) {
        /* 更新MAC速度/双工配置 */
        my_mac_set_speed(dev, phydev->speed, phydev->duplex);
    }
}

/* 2. 连接 PHY */
phydev = phy_connect(dev, phy_name, &adjust_link, PHY_INTERFACE_MODE_RGMII_ID);

/* 3. 过滤不支持的模式（如10/100MAC连GbPHY） */
phy_set_max_speed(phydev, SPEED_1000);

/* 4. 启动 PHY */
phy_start(phydev);   /* 触发自动协商，启动状态机 */

/* 断开 PHY */
phy_stop(phydev);
phy_disconnect(phydev);

8.2 stmmac DMA 描述符（官方文档）

原文（stmmac.rst）： "stmmac supports DMA descriptor to operate both in dual buffer (RING) and linked-list (CHAINED) mode. In RING each descriptor points to two data buffer pointers whereas in CHAINED mode they point to only one data buffer pointer. RING mode is the default." 
/* DMA 描述符结构示例（stmmac，来源：drivers/net/ethernet/stmicro/stmmac/descs.h） */
struct dma_desc {
    __le32 des0;  /* 状态/控制位（OWN bit, TX_EN, FS, LS等）*/
    __le32 des1;  /* 缓冲区1/2大小 */
    __le32 des2;  /* 缓冲区1物理地址 */
    __le32 des3;  /* 缓冲区2物理地址 / 下一个描述符地址 */
};

/* OWN bit：
 * des0 bit31 = 1 → DMA 拥有此描述符（硬件处理中）
 * des0 bit31 = 0 → CPU 拥有此描述符（软件可操作）
 * TX: CPU设置完成后置OWN=1交给DMA; DMA发完后清OWN=0
 * RX: CPU分配好buffer后置OWN=1; DMA收到帧后清OWN=0通知CPU
 */

9.1 RGMII 延迟配置（最高频问题）

⚠️ 最重要！ RGMII 延迟配置错误是嵌入式以太网驱动中最常见的问题。必须根据原理图和PHY手册确认：由MAC侧、PHY侧还是PCB走线提供1.5~2ns延迟，然后正确设置 phy-mode。

四种配置逻辑（来源：kernel.org phy.txt，从PHY自身角度理解）：

  • rgmii：PHY不插入任何延迟，MAC或PCB走线负责
  • rgmii-txid：PHY在TX数据线插入内部延迟
  • rgmii-rxid：PHY在RX数据线插入内部延迟
  • rgmii-id：PHY在TX和RX数据线都插入延迟（最推荐）

9.2 DMA 内存管理

  • 一致性DMA内存：描述符环用 dma_alloc_coherent()，CPU和DMA都可见且强一致
  • 流式DMA映射：sk_buff数据用 dma_map_single()，使用后必须 dma_unmap_single()
  • DMA屏障：写描述符后需确保内存屏障（wmb()），防止CPU乱序写导致DMA读错数据
  • 零拷贝：预分配RX sk_buff，DMA直接写入，避免memcpy

9.3 时钟管理

  • MAC控制器通常需要多个时钟：clk_stmmaceth（主时钟）、pclk（APB总线时钟）
  • RGMII 1000Mbps 模式：MAC需要125MHz参考时钟
  • RMII 模式：需要外部50MHz参考时钟（可由MAC输出或外部晶振提供）
  • 设备树中通过 clock-names 和 clocks 属性描述

9.4 PHY Board Fixup

💡 官方原文（kernel.org phy.txt）： "Sometimes the specific interaction between the platform and the PHY requires special handling. For instance, to change where the PHY's clock input is, or to add a delay to account for latency issues in the data path." 
/* PHY Fixup 注册（用于特殊板级硬件限制） */
phy_register_fixup_for_uid(0x01234567, 0xfffffff0, my_phy_fixup);

static int my_phy_fixup(struct phy_device *phydev)
{
    /* 例：强制PHY关闭绿色以太网节能功能 */
    phy_write(phydev, 0x1f, 0x0003);
    phy_modify(phydev, 0x10, BIT(2), 0);
    phy_write(phydev, 0x1f, 0x0000);
    return 0;
}

9.5 流量控制（Pause Frames）

官方说明（kernel.org phy.txt）： "The PHY does not participate directly in flow control/pause frames except by making sure that the SUPPORTED_Pause and SUPPORTED_AsymPause bits are set in MII_ADVERTISE."

流量控制的处理在 MAC 驱动中，PHY 只负责在自动协商时广播暂停帧支持能力。驱动需通过 ethtool::set_pauseparam 实现用户态配置。

# phytool：直接读写PHY寄存器（来源：wiki.analog.com, https://github.com/wkz/phytool）
# 格式：phytool read  <iface>/<phy_addr>/<reg>
#       phytool write <iface>/<phy_addr>/<reg> <value>

# 读取 PHY 基本控制寄存器（BMCR，寄存器0）
phytool read eth0/0/0

# 读取 PHY 基本状态寄存器（BMSR，寄存器1）
phytool read eth0/0/1
# 若 bit2=1 表示 Link Up，bit5=1 表示 Auto-Negotiation Complete

# 读取 PHY ID（寄存器2和3）
phytool read eth0/0/2   # OUI高16位
phytool read eth0/0/3   # OUI低6位 + 型号 + 版本

11.3 驱动开发 Checklist

✅ 嵌入式以太网驱动开发核查清单
  • □ 确认接口类型（MII/RMII/RGMII/SGMII）及时钟来源
  • □ 根据原理图确定 RGMII 延迟由谁提供，正确设置 phy-mode
  • □ 正确配置 PHY MDIO 地址（DT 中 reg 属性）
  • □ 实现完整的 net_device_ops（open/stop/xmit/timeout/set_rx_mode）
  • □ DMA 描述符环：分配/初始化/OWN bit 管理/内存屏障
  • □ RX 路径：预分配 sk_buff，零拷贝，正确释放/重填
  • □ TX 路径：队列停止（stop_queue）与恢复（wake_queue）逻辑
  • □ 实现 NAPI：napi_schedule + poll 函数
  • □ 正确处理 DMA 一致性（dma_alloc_coherent 描述符 vs dma_map 数据）
  • □ 时钟管理：probe时获取并enable，remove时disable和put
  • □ 实现 ethtool_ops：至少 get_link/get_drvinfo/get_ethtool_stats
  • □ PHY 特殊初始化（errata/fixup）是否需要 board fixup 或 config_init
  • □ 流量控制（Pause Frame）正确配置 SUPPORTED_Pause 位
  • □ 电源管理：suspend/resume 时保存/恢复 MAC 状态
