GNU Make 内部处理流程详解

当你执行 make 命令后，背后究竟发生了什么？本文档基于 GNU Make 官方手册，深入解析其两阶段工作机制及每步可观测方法。

两阶段处理 依赖图构建 变量展开 隐式规则 时间戳比较 Recipe 执行
官方原文：GNU Make Manual — gnu.org

GNU Make 的两阶段工作模型

GNU Make 官方手册（§3.7 How make Reads a Makefile）明确指出：

官方原文：
"GNU make does its work in two distinct phases. During the first phase it reads all the makefiles, included makefiles, etc. and internalizes all the variables and their values and implicit and explicit rules, and builds a dependency graph of all the targets and their prerequisites. During the second phase, make uses this internalized data to determine which targets need to be updated and run the recipes necessary to update them."
图1：GNU Make 两阶段工作模型概览（来源：GNU Make Manual §3.7）

步骤 1：启动与初始化

Phase 1 · 第一阶段第一步

官方原文：§9.8 Summary of Options — gnu.org
1
启动与初始化
初始化

在 Make 读取任何 Makefile 之前，它首先完成一系列初始化动作：

  1. 解析命令行参数：处理 -f, -j, -d 等选项
  2. 加载内置规则和变量：Make 自带数十条隐式规则（如 .c → .o）和预定义变量（如 CC, CFLAGS）
  3. 读取环境变量：将当前 shell 环境变量导入为 Make 变量
  4. 处理 MAKEFILES 环境变量：若设置了此变量，先读取其中列出的文件
  5. 处理 -E / --eval 选项：在读取任何 Makefile 前，先对命令行指定的字符串执行 eval

内置变量与规则

Make 启动时加载的内置规则示例（GNU Make Manual §10.2）：

# 内置的 C 编译隐式规则（简化示意）
%.o : %.c
    $(CC) $(CPPFLAGS) $(CFLAGS) -c -o $@ $<

# 内置的 C++ 编译隐式规则
%.o : %.cc
    $(CXX) $(CPPFLAGS) $(CXXFLAGS) -c -o $@ $<

# 内置预定义变量
CC     = cc
CXX    = g++
CFLAGS = (空)
内置规则示例

查找 Makefile 文件
Phase 1

当没有使用 -f 选项指定 Makefile 时，GNU Make 按照固定顺序在当前目录查找：

  1. GNUmakefile — 仅 GNU Make 会识别此名称

"When make processes an include directive, it suspends reading of the containing makefile and reads from each listed file in turn. When that is finished, make resumes reading the makefile in which the directive appears."

步骤 3：逐行解析 Makefile

Phase 1 · §3.8 How Makefiles Are Parsed

官方原文：§3.8 How Makefiles Are Parsed — gnu.org

官方手册明确给出了解析的六步流程：

3
逐行解析 Makefile — 官方定义的6个子步骤
Phase 1
官方原文（§3.8）：
"GNU make parses makefiles line-by-line. Parsing proceeds using the following steps:"
  1. 读取完整的逻辑行，包括反斜杠续行（Splitting Long Lines）
  2. 移除注释（# 开头到行尾）
  3. 若行以 recipe 前缀字符（默认为 Tab）开头，且当前处于规则上下文中，则将该行添加到当前 recipe 并读取下一行
  4. 展开该行中属于 immediate 展开上下文的元素
  5. 扫描行中的分隔符（: 或 =），判断这行是宏赋值还是规则
  6. 内化操作结果并读取下一行

解析顺序的重要性

官方警告：
"An important consequence of this is that a macro can expand to an entire rule, if it is one line long."
宏展开 不会 重新分割行——展开后的换行符会被当作普通空白处理。 
# 这个可以工作：单行宏展开为完整规则
myrule = target : ; echo built
$(myrule)

# 这个不能工作：多行 define 宏，展开后换行被忽略
define myrule
target:
        echo built
endef
# 结果：target 被解析为有 echo 和 built 两个先决条件的规则，而非 recipe
# 正确做法：
$(eval $(myrule))  # 使用 eval 函数

Makefile 的基本语法元素

元素格式说明
规则target: prerequisites
⇥ recipe		目标、先决条件、配方（Tab 缩进）
变量赋值VAR = value递归展开赋值
立即赋值VAR := value立即展开赋值
条件ifdef / ifeq条件解析（第一阶段立即处理）
指令include / define包含文件或定义多行变量
注释# ...解析时被丢弃

步骤 4：变量内化与展开

Phase 1 · §3.7 How make Reads a Makefile · §6 How to Use Variables

官方原文：§3.7 Reading Makefiles · §6 Variables — gnu.org
4
变量内化与展开时机
Phase 1 关键

官方手册定义了两种展开时机：

  • Immediate（立即展开）：在第一阶段 Makefile 被解析时立即展开
  • Deferred（延迟展开）：推迟到被引用时（immediate 上下文中），或在第二阶段执行时展开

变量赋值的展开规则

官方手册 §3.7 给出了所有赋值运算符的展开规则：

赋值语法左侧（名称）右侧（值）说明
VAR = value立即延迟递归展开变量（每次引用时展开）
VAR ?= value立即延迟条件赋值（未定义时才赋值）
VAR := value立即立即简单展开变量（赋值时立即展开）
VAR ::= value立即立即POSIX 标准的立即展开赋值
VAR :::= value立即立即后转义展开后将 $ 替换为 $$
VAR += value立即视情况而定若之前是 := 则立即；否则延迟
VAR != cmd立即立即（shell）Shell 赋值，右侧交给 shell 执行

规则定义的展开规则

官方原文（§3.7 Rule Definition）：
"A rule is always expanded the same way, regardless of the form:
immediate : immediate ; deferred
⇥ deferred
That is, the target and prerequisite sections are expanded immediately, and the recipe used to build the target is always deferred." 
# 目标和先决条件：立即展开
$(TARGET) : $(DEPS)
    # recipe 内容：延迟展开（第二阶段执行前才展开）
    $(CC) -o $@ $^

条件指令

官方原文：
"Conditional directives are parsed immediately. This means, for example, that automatic variables cannot be used in conditional directives, as automatic variables are not set until the recipe for that rule is invoked." 
# 正确：条件指令可以使用普通变量
ifdef DEBUG
CFLAGS += -g
endif

# 错误！自动变量 $@ 在条件指令中不可用（解析时尚未设置）
# ifeq ($@, target)   ← 这是错误用法
#     ...             ← 应在 recipe 中用 shell 条件语句代替
# endif

如何查看变量展开结果

make -p | grep "^VAR"

在 make 数据库中查看特定变量最终值

$(info VAR=$(VAR))

在 Makefile 中插入 info 函数，解析时打印变量值

$(warning ...)

输出警告信息（含文件名和行号），不中断执行

$(error ...)

输出错误并终止 make（用于调试时验证变量）

实用调试示例

# 在 Makefile 中插入，查看变量展开时的值
SRCS := $(wildcard *.c)
$(info SRCS is: $(SRCS))   # 第一阶段解析时立即打印

all: $(SRCS:.c=.o)
    $(info Building $@)    # 第二阶段执行时打印
    $(CC) -o $@ $^

步骤 6：重建 Makefile 自身

Phase 1 特殊步骤 · §3.5 How Makefiles Are Remade

官方原文：§3.5 How Makefiles Are Remade — gnu.org
6
重建 Makefile 自身（Remaking Makefiles）
Phase 1 特殊
官方原文（§3.5）：
"To this end, after reading in all makefiles make will consider each as a goal target, in the order in which they were processed, and attempt to update it. [...] After all makefiles have been checked, if any have actually been changed, make starts with a clean slate and reads all the makefiles over again."

这是一个很多人不知道的隐藏步骤：Make 读取所有 Makefile 后，会将每个 Makefile 文件本身视作一个构建目标，并尝试更新它。如果任何 Makefile 被重建，Make 会从头开始重新读取所有 Makefile。

典型用途：自动依赖生成

# 常见模式：从编译器生成依赖文件后包含
-include $(SRCS:.c=.d)

# 规则：生成 .d 依赖文件
%.d: %.c
    $(CC) -MM $< -MF $@

# 第一次 make 时：
# 1. 读取 Makefile（.d 文件不存在，-include 忽略）
# 2. 步骤6：尝试更新 .d 文件 → 执行 cc -MM 生成
# 3. 发现 .d 文件改变 → 重新读取整个 Makefile
# 4. 这次 include .d 文件成功 → 正确依赖

避免无限循环

官方警告：
若 Makefile 有双冒号规则（::）且带有 recipe 但无先决条件，会在每次 make 时重建，导致无限循环。Make 特别处理此情况——不会尝试重建标记为 phony 或双冒号无先决条件的 Makefile。 
# 优化：告知 Make 不需要重建 Makefile，提升启动速度
.PHONY: Makefile
# 或：
Makefile::;

重建时的特殊规则

官方手册特别指出：-n、-t、-q 选项不影响 Makefile 自身的重建——即使处于 dry-run 模式，Makefile 也会被真实更新。

可观测方法：
使用 --debug=m 选项，可以在重建 Makefile 阶段看到详细调试输出。 
# 查看 Makefile 重建过程
$ make --debug=m

步骤 8：隐式规则搜索算法

Phase 2 · §10.8 Implicit Rule Search Algorithm

官方原文：§10.8 Implicit Rule Search Algorithm — gnu.org
8
隐式规则搜索算法
Phase 2

当一个目标没有显式 recipe 时，Make 会按照以下算法搜索适用的隐式规则（§10.8 完整算法）：

  1. 将目标 t 分为目录部分 d 和其余部分 n
  2. 列出所有匹配 t 或 n 的模式规则
  3. 若列表中有非通配规则，移除所有通配规则（%: %）
  4. 移除所有无 recipe 的规则
  5. 对每条候选规则，找出 stem（% 匹配的部分），计算先决条件名，检查先决条件是否存在或应该存在
  6. 若步骤5未找到，尝试递归——检查先决条件是否可由其他隐式规则生成（规则链）
  7. 若仍未找到，再次以放宽的"应该存在"定义重试步骤5和6（向后兼容）
  8. 若仍无适用规则，使用 .DEFAULT 规则（若存在）

常见内置隐式规则

模式触发条件默认 Recipe
%.o: %.c存在 foo.c$(CC) $(CPPFLAGS) $(CFLAGS) -c -o $@ $<
%.o: %.cc存在 foo.cc$(CXX) $(CPPFLAGS) $(CXXFLAGS) -c -o $@ $<
%.o: %.f存在 foo.f$(FC) $(FFLAGS) -c -o $@ $<
%: %.o存在 foo.o$(CC) $(LDFLAGS) $^ $(LOADLIBES) $(LDLIBS) -o $@
%.c: %.y存在 foo.y (yacc)$(YACC) $(YFLAGS) -o $@ $<

隐式规则链（Chained Rules）

Make 可以自动创建规则链。例如，若存在 foo.y（Yacc 文件），Make 可以：

foo.y → foo.c  (yacc 规则)
foo.c → foo.o  (cc 编译规则)
foo.o → foo    (链接规则)
官方说明（§10.4）： 中间文件（intermediate files）会在构建完成后自动删除，除非用 .SECONDARY 或 .PRECIOUS 标记保留。

步骤 9：时间戳比较（mtime）

Phase 2 · §2.3 · §4.3 Types of Prerequisites

官方原文：§2.3 · §4.3 Types of Prerequisites — gnu.org
9
时间戳比较（mtime 检查）
Phase 2 核心

Make 决定是否重建目标的核心依据是文件的最后修改时间（mtime）。以下任一条件成立，目标就需要重建：

  1. 目标文件不存在
  2. 任意先决条件的 mtime 比目标文件的 mtime 更新
官方原文：
"The recompilation must be done if the source file, or any of the header files named as prerequisites, is more recent than the object file, or if the object file does not exist."

两类先决条件

官方手册 §4.3 定义了两种先决条件类型：

类型语法触发条件说明
普通先决条件
(Normal Prerequisites)		target: P1 P2P 比目标新 → 重建目标影响构建顺序 + 触发重建
仅序先决条件
(Order-only Prerequisites)		target: P1 | P2P 比目标新 → 不触发重建仅保证 P 先被处理，不触发时间戳检查
# 普通先决条件：config.mk 更新会触发重建
output.txt: input.txt config.mk
    process $< > $@

# 仅序先决条件：obj/ 目录必须先存在，但目录时间戳不触发重建
main.o: main.c | obj/
    $(CC) -c main.c -o obj/main.o

obj/:
    mkdir -p obj/

基本调试：显示哪些目标被认为过期（out-of-date）

返回码：0=已最新，1=需更新。不执行任何命令

make -d 2>&1 | grep "newer"

步骤 10：执行 Recipe（配方）

Phase 2 · §5 Writing Recipes · §5.3 Recipe Execution

官方原文：§5.3 Recipe Execution — gnu.org
10
执行 Recipe —— Sub-Shell 调用
Phase 2 最终
官方原文（§5.3）：
"When it is time to execute recipes to update a target, they are executed by invoking a new sub-shell for each line of the recipe, unless the .ONESHELL special target is in effect."

关键要点：

  • 每行 recipe 在独立的子 shell 中执行（默认 /bin/sh）
  • 因此，cd、设置环境变量等操作不会影响后续行
  • 使用 .ONESHELL 可让整个 recipe 在同一 shell 中运行

Recipe 执行机制详解

# 错误示例：cd 不影响下一行
wrong-target:
    cd /tmp          # sub-shell 1
    pwd              # sub-shell 2，还是原目录！

# 正确做法1：同一行用 && 连接
correct1:
    cd /tmp && pwd   # 同一 sub-shell

# 正确做法2：续行
correct2:
    cd /tmp; \
    pwd              # 同一 sub-shell（\ 续行）

# 正确做法3：使用 .ONESHELL
.ONESHELL:
oneshell-target:
    cd /tmp
    pwd              # 同一 shell 中执行，正确

Recipe 前缀字符的含义

前缀效果示例
@不回显命令（静默执行）@echo "Done" → 只输出 Done，不显示 echo 命令
-忽略命令错误，继续执行-rm *.o → 删除失败也继续
+即使 -n 模式也执行递归 make 中使用

并行执行（Parallel Execution）

官方手册 §5.4 介绍了并行构建：

官方说明： 使用 -j N 选项，Make 会同时启动最多 N 个 recipe。互不依赖的目标可以并发构建，显著加速大型项目。 
# 使用所有 CPU 核心并行构建
$ make -j$(nproc)

# 指定最多 4 个并行任务
$ make -j4

# 并行时保证各目标输出不混杂
$ make -j4 --output-sync=target

--debug 选项的各级别

官方手册 §9.8 完整定义了 --debug 的各个标志（仅第一个字母有效，可组合）：

标志全名含义典型用途
a all 所有调试输出（等同于 -d） 全量分析
b basic 打印每个被认为过期的目标，以及构建是否成功 快速了解哪些目标需要重建
v verbose basic 基础上增加：解析了哪些 Makefile、哪些先决条件不需要重建等 了解解析过程
i implicit 描述每个目标的隐式规则搜索过程 诊断为何找不到规则
j jobs 显示子命令调用的详细信息（PID、状态等） 并行构建调试
m makefile 在重建 Makefile 阶段也启用调试消息 调试 Makefile 自身重建
p print 即使 recipe 被 @ 或 .SILENT 静默，也打印出来（含文件名和行号） 查看所有实际执行的命令
w why 解释为什么每个目标需要重建（哪个先决条件更新了） 诊断意外重建原因
n none 禁用所有之前启用的调试标志 重置调试状态

其他关键诊断选项

选项作用使用场景
-d等同于 --debug=a，最全量输出深度调试
-p打印内部数据库（规则 + 变量值），然后正常执行查看最终变量值和规则
-nDry-run，打印命令但不执行预览构建操作
-qQuestion 模式，只返回退出码脚本中检查是否需要构建
--trace等同于 --debug=print,why追踪每个目标的处置
--warn-undefined-variables引用未定义变量时发出警告调试变量错误
-w打印进入/离开目录的消息递归 make 追踪

组合使用示例

# 最常用的调试组合：查看为什么重建 + 隐式规则搜索
$ make --debug=wi target

# 查看数据库，不执行构建
$ make -qp 2>/dev/null | grep -A5 "^target:"

# Dry-run + 显示静默命令
$ make -n --debug=p

# 仅查看内置规则（不读取任何 Makefile）
$ make -p -f /dev/null 2>/dev/null

# 追踪完整流程
$ make --debug=a 2>&1 | tee make.log
