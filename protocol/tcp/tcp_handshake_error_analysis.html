概述：TCP 三次握手与 Linux 内核

Linux 内核 TCP 协议栈严格实现 RFC 793、RFC 1122 等规范，在三次握手阶段为每个报文定义了精确的超时、重传和错误处理机制。本文基于官方 Linux 内核文档与 man7.org 手册页进行深度解析。

1.3 三次握手中的网络异常分类

异常类型分类

  • 报文静默丢失：防火墙 DROP、网络拥塞，无任何反馈信号
  • ICMP 目的不可达：路由器无路由 (NET_UNREACH)、ARP 失败 (HOST_UNREACH)、端口关闭 (PORT_UNREACH)、管理过滤 (PKT_FILTERED)
  • TCP RST：目标端口未监听、连接不合法
  • 服务端队列满：SYN Backlog 溢出、Accept Queue 溢出

📌 第一章总结

  • Linux 内核 TCP 通过三个关键内核参数控制三次握手阶段的重传行为
  • SYN 重传使用指数退避（Exponential Backoff），初始 RTO 为 1 秒
  • 网络异常分为"静默丢失"和"有反馈信号"两类，处理策略截然不同
  • 不同 ICMP 类型对 connect() 的影响不同：硬错误立即返回，软错误继续重试
第二章

第①报文：SYN 的发送与异常处理

客户端调用 connect() 后进入 SYN_SENT 状态，内核发送 SYN 报文并启动定时器。以下分析各种网络异常下的内核处理行为。

📄 tcp(7) — tcp_syn_retries 📄 Linux Kernel IP Sysctl

2.1 SYN 超时重传机制（无响应）

当 SYN 被静默丢弃（防火墙 DROP、网络中断）时，内核以指数退避方式重传 SYN：

官方文档原文（ip-sysctl.html）： 
tcp_syn_retries - INTEGER
    Number of times initial SYNs for an active TCP connection attempt
    will be retransmitted. Should not be higher than 127.
    Default value is 6, which corresponds to 67 seconds till the
    last retransmission with the current initial RTO of 1 second.
    With this the final timeout for an active TCP connection attempt
    will happen after 131 seconds.
tcp_syn_linear_timeouts（新内核参数）： 
tcp_syn_linear_timeouts - INTEGER
    The number of times for an active TCP connection to retransmit
    SYNs with a linear backoff timeout before defaulting to an
    exponential backoff timeout.
    Default: 4

2.3 各类网络异常下的 SYN 处理

① 网关不通 / 本地无路由

现象：本地路由表无匹配项（ENETUNREACH），或网关不可达。

内核行为：connect() 立即返回 ENETUNREACH，不发送 SYN，不重试。

条件内核行为错误码重试次数
本地路由表无路由 内核查 FIB 失败，立即拒绝 ENETUNREACH 0（立即返回）

来源：RFC 1122 Section 3.2.2.1 规定 NET_UNREACH 为软错误，但本地无路由属于立即可知的硬错误。

② 收到 ICMP 目的不可达（中间路由器返回）

根据官方来源 mechpen.github.io 及内核代码分析：

ICMP 类型触发场景connect() 行为返回错误码
NET_UNREACH 路由器无到目标网络路由 视为软错误，继续重试 SYN ENETUNREACH（最终超时后）
HOST_UNREACH 路由器 ARP 失败（约3次后），无法解析下一跳 MAC 视为软错误，继续重试 SYN EHOSTUNREACH（最终超时后）
PORT_UNREACH 目标端口未监听（TCP 通常不产生此消息，而是 RST） 视为硬错误，立即返回 ECONNREFUSED
PKT_FILTERED 防火墙 REJECT（管理过滤） 视为硬错误，立即返回 EHOSTUNREACH

⚠️ 关键设计原则：RFC 1122 §3.2.2.1 规定，NET_UNREACH 和 HOST_UNREACH 属于"瞬态"软错误（可能由临时路由问题引起），不应立即终止连接，应继续重试等待恢复。

③ 收到 TCP RST（目标端口拒绝）

服务端目标端口未监听，发回 RST 报文。

内核处理（来自 net/ipv4/tcp_input.c）：

switch (sk->sk_state) {
case TCP_SYN_SENT:
    sk->sk_err = ECONNREFUSED;  /* SYN_SENT 状态收到 RST → ECONNREFUSED */
    break;
...
}

在 SYN_SENT 状态收到 RST：立即终止连接，connect() 返回 ECONNREFUSED，不重试。

④ SYN 被防火墙静默丢弃（DROP）

无任何反馈，内核进入重传逻辑：

  • 初始 RTO = 1 秒（TCP_TIMEOUT_INIT，定义于 include/net/tcp.h）
  • 每次重传 RTO 翻倍（指数退避）
  • 默认重传 6 次（加首次发送共 7 个 SYN）
  • 总等待约 127~131 秒后，connect() 返回 ETIMEDOUT

2.4 SYN 重传参数汇总表

场景重试次数间隔模式总超时返回错误
无响应（报文丢失） 6次（共7个SYN） 1→2→4→8→16→32→64s ~131s ETIMEDOUT
收到 RST 0（立即失败） — 即时 ECONNREFUSED
ICMP NET/HOST_UNREACH 继续重试直至超时 指数退避 ~131s ENETUNREACH / EHOSTUNREACH
ICMP PKT_FILTERED 0（立即失败） — 即时 EHOSTUNREACH
本地无路由 0（立即失败） — 即时 ENETUNREACH

2.5 自定义参数

# 查看当前 SYN 重传次数
sysctl net.ipv4.tcp_syn_retries
# net.ipv4.tcp_syn_retries = 6

# 修改（减少以快速失败）
sysctl -w net.ipv4.tcp_syn_retries=3

# 单连接级别控制（通过 socket option）
setsockopt(fd, IPPROTO_TCP, TCP_SYNCNT, &val, sizeof(val));

📌 第二章总结

  • SYN 无响应时，内核以指数退避重传，默认共 7 次，约 131 秒超时
  • RST 和 PKT_FILTERED（管理过滤）属于硬错误，立即终止，不重试
  • NET_UNREACH / HOST_UNREACH 属于软错误，继续重试（符合 RFC 1122 §3.2.2.1）
  • tcp_syn_linear_timeouts 参数（新内核）控制前几次线性退避，之后转为指数退避

🔬 第一性原理分析

SYN 超时重传的设计基于"网络不可靠，但错误可能是临时的"这一根本假设。指数退避算法通过逐步增大等待时间，在快速响应短暂故障与避免网络拥塞之间取得平衡——这正是 RFC 6298（计算 TCP 的 RTT 和 RTO）的核心思想。

第三章

第②报文：SYN-ACK 的发送与异常处理

服务端收到 SYN 后，进入 SYN_RCVD 状态，发送 SYN-ACK，并等待客户端的 ACK 完成握手。这是被动连接的核心阶段。

3.1 服务端 SYN 队列结构

Linux 内核维护两个关键队列处理连接建立：

图3：Linux TCP 握手双队列（SYN Queue + Accept Queue）

3.2 SYN-ACK 超时重传机制

官方文档原文（ip-sysctl.html v5.8）： 
tcp_synack_retries - INTEGER
    Number of times SYNACKs for a passive TCP connection attempt
    will be retransmitted. Should not be higher than 255.
    Default value is 5, which corresponds to 31 seconds till the
    last retransmission with the current initial RTO of 1 second.
    With this the final timeout for a passive TCP connection will
    happen after 63 seconds.

重传计算：初始 RTO = 1s，指数退避：

第N次 SYN-ACK发送时刻等待间隔
第1次（原始）t = 0s等待 1s
第2次（重传#1）t = 1s等待 2s
第3次（重传#2）t = 3s等待 4s
第4次（重传#3）t = 7s等待 8s
第5次（重传#4）t = 15s等待 16s
第6次（重传#5）t = 31s等待 32s
t = 63s：超时，从 SYN 队列删除半连接

3.3 各类异常下 SYN-ACK 的处理

① 客户端无响应（最终 ACK 丢失）

  • 服务端重传 SYN-ACK，共 tcp_synack_retries（默认5）次
  • 每次间隔指数退避，从 1s 开始
  • 约 63 秒后，内核从 SYN 队列删除该半连接，释放资源
  • 无需应用层干预

② SYN 队列溢出（SYN Flood 或高并发）

根据 阿里云博客 及内核源码：

/* 内核源码注释（net/ipv4/tcp_ipv4.c）：
 * Accept backlog is full. If we have already queued enough
 * of warm entries in syn queue, drop request. It is better than
 * clogging syn queue with openreqs with exponentially increasing timeout.
 */
if (sk_acceptq_is_full(sk) && inet_csk_reqsk_queue_young(sk) > 1) {
    NET_INC_STATS_BH(sock_net(sk), LINUX_MIB_LISTENOVERFLOWS);
    goto drop;  /* 静默丢弃 SYN，不发送 RST */
}

默认行为：队列满时静默丢弃 SYN，客户端会超时重试（tcp_abort_on_overflow=FALSE）。

若 tcp_abort_on_overflow=TRUE：发送 RST，客户端立即收到 ECONNREFUSED。

③ SYN Cookie 机制（防 SYN Flood）

官方文档（ip-sysctl.html）： 
tcp_syncookies - INTEGER (default: 1)
    Enable TCP syncookies. The kernel must be compiled with
    CONFIG_SYN_COOKIES. The syncookies feature attempts to protect
    a socket from a SYN flood attack.
    Set to 1: send out syncookies when the syn backlog queue overflows.

启用 SYN Cookie 时，服务端不在 SYN 队列存储状态，而是将连接信息编码进 SYN-ACK 的序列号，规避队列溢出问题。

④ Accept 队列溢出（应用层处理缓慢）

  • Accept 队列满（min(backlog, net.core.somaxconn)）
  • 内核将完成三次握手的连接重置回 SYN_RCVD 状态（制造"最后一个 ACK 丢失"的假象）
  • 客户端会在超时后重发 ACK，再次触发握手完成
  • 若 tcp_abort_on_overflow=TRUE，直接发 RST

3.4 SYN-ACK 异常汇总

异常场景服务端行为重传次数总超时最终结果
最终 ACK 未到达 重传 SYN-ACK 5次 ~63s 删除半连接
SYN 队列满（默认） 静默丢弃 SYN 0 — 客户端超时重试
SYN 队列满（overflow=1） 发送 RST 0 即时 客户端 ECONNREFUSED
SYN Cookie 启用 编码序列号发 SYN-ACK 0（不重传） — 无状态处理
Accept 队列满（默认） 模拟 ACK 丢失，等待重试 0 — 客户端重发 ACK

📌 第三章总结

  • SYN-ACK 默认重传 5 次，指数退避，约 63 秒后删除半连接
  • SYN 队列（半连接队列）由 tcp_max_syn_backlog 控制，满时默认静默丢弃 SYN
  • Accept 队列（全连接队列）由 somaxconn 和 listen() backlog 的最小值控制
  • SYN Cookie 是防 SYN Flood 的兜底机制，不应作为常规负载处理手段

🔬 第一性原理分析

SYN 队列的设计体现了"资源有限，必须防御"原则。内核为每个半开连接分配状态（struct inet_request_sock），若无限接受 SYN 会被 SYN Flood 耗尽内存。双队列设计（SYN Queue + Accept Queue）实现了握手过程与应用接收的解耦，允许内核独立管理连接生命周期。

第四章

第③报文：ACK 的发送与异常处理

客户端收到 SYN-ACK 后发送 ACK（三次握手最后一步），同时进入 ESTABLISHED 状态。服务端收到 ACK 后也进入 ESTABLISHED。此阶段的异常相对少见但值得关注。

📄 tcp(7) — Linux man page 📄 Linux Kernel IP Sysctl

4.1 ACK 报文的特殊性

三次握手的最后一个 ACK 有以下特点：

  • 客户端发出 ACK 后立即进入 ESTABLISHED 状态，不等待确认
  • ACK 本身没有重传机制：TCP 不对纯 ACK 进行重传
  • 若 ACK 丢失，服务端会重传 SYN-ACK，客户端收到后重新发送 ACK
  • 连接建立完成后，若数据发送遇到问题，进入已建立连接的重传机制（tcp_retries2）

4.2 ACK 丢失的处理流程

4.3 各类异常下 ACK 阶段的处理

① ACK 被静默丢弃

  • 客户端已进入 ESTABLISHED 状态
  • 服务端等待 ACK 超时后，重传 SYN-ACK（由 tcp_synack_retries 控制）
  • 客户端收到 SYN-ACK，自动重发 ACK
  • 握手自愈恢复，无需应用层干预

② 服务端收到 ACK 时 Accept 队列已满

内核对 Accept Queue 满时的处理（来源：内核代码 + 阿里云博客）：

  • 默认（tcp_abort_on_overflow=FALSE）：内核将连接回退至 SYN_RCVD，忽略该 ACK，等待客户端重发
  • 开启（tcp_abort_on_overflow=TRUE）：发送 RST，客户端得到 ECONNRESET
tcp_abort_on_overflow - BOOLEAN
    If listening service is too slow to accept new connections,
    reset them. Default state is FALSE. It means that if overflow
    occurred due to a burst, connection will recover.
    Enable this option _only_ if you are really sure that listening
    daemon cannot be tuned to accept connections faster.

③ ACK 发送时遇到路由问题

客户端发送 ACK 时若路由暂时不可用：

  • 客户端已进入 ESTABLISHED，若 ACK 发送失败（ENETUNREACH），报告给应用层
  • 若后续有数据发送，进入已建立连接的重传机制
  • 若只是 ACK 丢失，等待服务端重发 SYN-ACK 即可触发重新发送 ACK

④ TCP_DEFER_ACCEPT 对 ACK 的影响

官方文档（tcp(7)）： 
TCP_DEFER_ACCEPT (since Linux 2.4)
    Allow a listener to be awakened only when data arrives on the
    socket. On a client socket, if nonzero, delays sending the final
    ACK of the three-way handshake until the socket has data to send,
    in which case the ACK is piggybacked on the first data segment,
    or a delayed-ACK timeout is reached.

启用此选项后，客户端延迟发送最终 ACK，直到有数据发送，ACK 随数据一起发出（捎带确认），减少报文数量。

4.4 ACK 阶段异常汇总

异常场景客户端状态服务端行为最终结果
ACK 静默丢失 ESTABLISHED（已建立） 重传 SYN-ACK（最多5次） 客户端重发 ACK，自愈
Accept 队列满（默认） ESTABLISHED 忽略 ACK，等待重试 最终队列腾空后接受
Accept 队列满（overflow=1） ESTABLISHED 发送 RST 客户端 ECONNRESET
TCP_DEFER_ACCEPT 延迟发 ACK 等待数据到来 ACK 随数据捎带发送

📌 第四章总结

  • 最终 ACK 没有重传机制，依赖服务端重传 SYN-ACK 触发客户端重发 ACK
  • ACK 丢失后系统自愈：服务端重传 SYN-ACK → 客户端重发 ACK，约 1 秒内恢复
  • Accept 队列溢出的默认策略是等待，而非立即报错，允许突发流量恢复
  • TCP_DEFER_ACCEPT 优化可减少纯 ACK 报文，提升效率

ICMP 错误与 TCP 握手：深度分析

ICMP 目的不可达消息是网络故障的重要反馈信号。Linux 内核对不同类型的 ICMP 错误有截然不同的处理策略，理解这些策略对于正确诊断网络问题至关重要。

5.3 为何已建立连接忽略 ICMP 错误？

来源：RFC 5927 - ICMP Attacks against TCP

TCP 连接建立后，内核不再将 ICMP 错误立即通知应用层，而是继续重试，原因：

  • 防御 ICMP 攻击（Off-path 攻击）：攻击者可伪造 ICMP 错误消息，若立即中断连接会被轻松攻击
  • 网络临时故障容忍：ICMP 错误可能是路由抖动等瞬态问题，连接最终可恢复
  • 重传直至超时（tcp_retries2=15，约 13~30 分钟），保证应用数据不因临时故障丢失

若需立即感知 ICMP 错误，可启用 IP_RECVERR socket 选项。

5.4 HOST_UNREACH 详解（网关无法解析 ARP）

HOST_UNREACH 通常由网关/路由器发出，触发条件：

  • 下一跳路由器尝试 ARP 解析目标 IP，发送约 3 次 ARP 请求（约 3 秒）
  • 未收到 ARP 回复后，路由器返回 ICMP HOST_UNREACH 给源主机
  • Linux 内核收到后保存错误，继续重传 SYN（软错误处理）
  • 若反复收到 HOST_UNREACH 且 SYN 超时，最终返回 EHOSTUNREACH

📌 第五章总结

  • ICMP 错误分软硬两类：硬错误立即终止连接，软错误继续重试至超时
  • TCP 握手阶段（SYN_SENT）会响应 ICMP 错误；已建立连接后忽略 ICMP，防止伪造攻击
  • HOST_UNREACH 触发于路由器 ARP 解析失败，属于软错误，TCP 继续重试
  • 需要即时 ICMP 感知时，可使用 IP_RECVERR socket 选项绕过默认行为

🔬 第一性原理分析

ICMP 软错误/硬错误的划分体现了"错误是否可恢复"的判断：PORT_UNREACH 说明对端进程不存在（硬性事实），而 NET_UNREACH 说明路由不稳定（可能恢复）。已建立连接忽略 ICMP 则是安全性优先的折中：用增加故障恢复时间换取对路径外攻击（Off-path Attack）的免疫。

6.3 实用诊断命令

# 查看当前参数
sysctl net.ipv4.tcp_syn_retries
sysctl net.ipv4.tcp_synack_retries
sysctl net.ipv4.tcp_abort_on_overflow

# 查看 SYN 队列溢出统计
netstat -s | grep -i listen
# 或
nstat | grep ListenDrop

# 查看连接状态
ss -tn state syn-sent
ss -tn state syn-recv

# TCP 重传统计
netstat -s | grep retransmit

# 抓包验证重传
tcpdump -i eth0 "tcp[tcpflags] & tcp-syn != 0" -nn

6.4 调优建议场景

✅ 快速失败场景（内部服务、容器环境）

# 减少 SYN 重试，快速感知故障
sysctl -w net.ipv4.tcp_syn_retries=2
# 减少 SYN-ACK 重试
sysctl -w net.ipv4.tcp_synack_retries=2

⚠️ 高并发服务器（防 SYN Flood）

# 增大 SYN 队列
sysctl -w net.ipv4.tcp_max_syn_backlog=4096
# 确保 SYN Cookie 启用
sysctl -w net.ipv4.tcp_syncookies=1
# 增大 Accept 队列
sysctl -w net.core.somaxconn=4096

📌 第六章总结

  • 内核参数提供了精细控制，但默认值基于 RFC 规范，适合大多数场景
  • tcp_syn_retries 和 tcp_synack_retries 是最常用的调优参数
  • 生产环境修改前应充分测试，避免过激调整影响正常连接容错性
  • 诊断问题时，结合 ss/netstat/tcpdump 和内核统计计数器可精准定位

🔬 整体第一性原理总结

TCP 三次握手的所有异常处理机制都源于一个根本原则：在不可靠网络上构建可靠通信。具体体现为：

①「重试」应对随机丢包；②「指数退避」避免重试风暴加重网络拥塞；③「软/硬错误区分」平衡故障容忍性与快速失败；④「状态机+定时器」保证资源不会无限占用；⑤「双队列设计」解耦握手与应用接收，防止服务端被耗尽。这些机制共同构成了 TCP 在各种网络异常下的自适应防御体系。
