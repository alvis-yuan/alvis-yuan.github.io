邻居子系统概述

什么是邻居子系统，它解决哪个核心问题
🧭 导读：本节回答"邻居子系统是什么"。网络层（IP 层）只知道 IP 地址，而数据帧在二层（以太网）传输时需要 MAC 地址。邻居子系统就是在这两层之间架起桥梁的通用框架——它不仅支持 IPv4 的 ARP，也支持 IPv6 的 NDP，以及其他协议族。读完本节你将了解子系统在整个协议栈中的定位。

Linux 内核的邻居子系统（Neighbouring Subsystem）是网络子系统的核心组成部分，其主要职责是将网络层（L3）地址映射到链路层（L2）地址，即维护"谁在哪里"的信息。

邻居子系统被设计为一个通用框架，位于 net/core/neighbour.c，而具体的协议实现（ARP 在 net/ipv4/arp.c，NDP 在 net/ipv6/ndisc.c）则通过函数指针（neigh_ops）插入该框架。

图1-1：邻居子系统在协议栈中的位置及架构总览
子系统职责
  • L3 地址 → L2 地址（MAC）解析
  • 邻居缓存（Neighbour Cache）维护
  • 邻居不可达检测（NUD）
  • 待发包队列缓存（解析期间）
  • 通过 Netlink 向用户态通知变更

ARP 协议基础

地址解析协议的工作原理与报文格式
🧭 导读：本节介绍 ARP（Address Resolution Protocol）的核心工作机制。ARP 是邻居子系统 IPv4 端最直接的实现。理解 ARP 的请求/应答、免费 ARP、代理 ARP 等概念，是后续深入内核实现的前提。

ARP 工作流程

当主机需要向同一二层网络内的 IP 地址发送数据包，但本地 ARP 缓存中没有对应的 MAC 地址时，会触发 ARP 解析过程：

图2-1：ARP 请求/应答流程（arp_solicit → arp_process）

内核中，ARP 请求由 arp_solicit()（位于 net/ipv4/arp.c）发出，ARP 应答由 arp_rcv() → arp_process() 处理。

特殊 ARP 类型

免费 ARP（Gratuitous ARP）

主机将自己的 IP 作为目标 IP 发送 ARP 请求，用于：

  • 网卡上线时通知邻居更新缓存
  • IP 冲突检测
  • VRRP/HSRP 主备切换

由 arp_accept sysctl 控制是否接受

代理 ARP（Proxy ARP）

路由器代替另一个网段的主机回应 ARP，让两个不同网段的主机像在同一网段通信。

  • 用于 LVS 场景（DR 模式）
  • Kubernetes 容器网络

内核参数：net.ipv4.conf.eth0.proxy_arp=1

03

NUD 状态机

邻居不可达检测（Neighbour Unreachability Detection）完整状态转换
🧭 导读：NUD（Neighbour Unreachability Detection）是邻居子系统最核心的机制之一。每个邻居缓存条目都有一个 nud_state 字段，记录当前解析状态。理解状态机转换，是判断"邻居为什么不通"的关键。这些状态定义在 include/uapi/linux/neighbour.h 中。
图3-1：NUD 状态机完整转换图（来源：include/uapi/linux/neighbour.h）

各状态详解

状态值含义内核行为
INCOMPLETE 0x01 正在解析，ARP/NS 请求已发出，等待回复 数据包缓存在 arp_queue 中，等待解析完成后发送
REACHABLE 0x02 已确认可达，MAC 地址有效 快速路径（neigh_connect()），直接使用缓存的 MAC 发帧
STALE 0x04 条目过期，但 MAC 仍可用 有流量时进入 DELAY，继续使用旧 MAC 发帧
DELAY 0x08 发出数据包，等待上层协议（TCP ACK 等）确认 等待 delay_first_probe_time（默认5s）内的上层确认
PROBE 0x10 主动发 ARP/NS 探测，重确认 MAC 发送 ucast_solicit 次单播探测，失败则 → FAILED
FAILED 0x20 解析失败，邻居不可达 调用 arp_error_report() → ipv4_link_failure() → 发 ICMP 目标不可达
NOARP 0x40 无需 ARP 解析（如 loopback） 伪状态，永不变化，不做 NUD
PERMANENT 0x80 管理员静态配置，不会被 GC 删除 伪状态，永不变化，不做 NUD；ip neigh add ... nud permanent 创建
内核源码注释 NUD_VALID = NUD_PERMANENT | NUD_NOARP | NUD_REACHABLE | NUD_PROBE | NUD_STALE | NUD_DELAY — 这些状态下链路层地址是有效的，可以直接用于发包。NUD_CONNECTED = NUD_PERMANENT | NUD_NOARP | NUD_REACHABLE 才是真正的快速路径。
📚 参考来源： include/uapi/linux/neighbour.h（NUD 状态定义）　|　 Linux-IP.Net: ip neigh 输出格式　|　 rtnetlink(7): ndm_state 说明

核心数据结构

struct neighbour / neigh_table / neigh_parms / neigh_ops
🧭 导读：本节深入 include/net/neighbour.h，逐字段解析邻居子系统的四大核心数据结构。理解它们是阅读内核代码、分析内核 crash dump、以及理解 sysctl 调优参数的必要前提。

struct neighbour — 单个邻居条目

每个邻居（即一个 IP→MAC 映射）在内核中以 struct neighbour 表示，存储在哈希表中：

struct neighbour {
    struct neighbour  __rcu *next;      /* 同一 hash 桶的链表 */
    struct neigh_table      *tbl;       /* 所属的邻居表（arp_tbl/nd_tbl）*/
    struct neigh_parms      *parms;     /* 可调参数（超时、探测次数等）*/
    unsigned long            confirmed; /* 最后一次"上层"确认时间戳 */
    unsigned long            updated;   /* nud_state 最后更新时间 */
    rwlock_t                 lock;      /* 保护 ha / nud_state / timer */
    atomic_t                 refcnt;    /* 引用计数，防止被 GC 提前释放 */
    struct sk_buff_head      arp_queue; /* INCOMPLETE 期间的缓存包队列 */
    struct timer_list        timer;     /* NUD 定时器 */
    unsigned long            used;      /* 最后使用时间（影响 GC）*/
    atomic_t                 probes;    /* 已发探测次数 */
    __u8                     nud_state; /* NUD 状态（NUD_REACHABLE 等）*/
    __u8                     type;      /* RTN_UNICAST / RTN_LOCAL 等 */
    __u8                     dead;      /* 已被标记为销毁 */
    unsigned char            ha[MAX_ADDR_LEN]; /* 链路层地址（MAC），关键字段 */
    struct hh_cache          hh;        /* 预计算的硬件头部缓存，加速快速路径 */
    int (*output)(struct neighbour *, struct sk_buff *); /* 发包函数指针 */
    const struct neigh_ops  *ops;      /* 协议相关操作集 */
    struct net_device       *dev;      /* 关联的网络设备 */
    u8                       primary_key[]; /* IP 地址（用于哈希查找）*/
};
关键设计：output 函数指针 neigh->output 是性能关键。处于 CONNECTED 状态时，它指向 neigh_ops->connected_output（快速路径，直接用缓存 MAC 发包）；其他状态指向 neigh_ops->output（慢速路径，每次检查状态）。状态切换时由 neigh_connect() 和 neigh_suspect() 更新此指针。

struct neigh_table — 邻居表

struct neigh_table {
    int         family;       /* AF_INET 或 AF_INET6 */
    int         entry_size;   /* sizeof(struct neighbour) + key_len */
    int         key_len;      /* 主键长度（IPv4=4, IPv6=16）*/
    /* 哈希函数：key(ip) + dev → bucket index */
    __u32 (*hash)(const void *pkey, const struct net_device *dev, __u32 *rnd);
    /* 构造函数：初始化新 neighbour 条目 */
    int (*constructor)(struct neighbour *);
    /* 代理条目列表 */
    struct pneigh_entry    **phash_buckets;
    char                    *id;         /* "arp_cache" 或 "ndisc_cache" */
    struct neigh_parms      parms;      /* 默认参数 */
    /* GC 参数 */
    int        gc_interval;  /* 30s */
    int        gc_thresh1;   /* 128 — GC 不运行的下限 */
    int        gc_thresh2;   /* 512 — 软上限，超过5s触发 */
    int        gc_thresh3;   /* 1024 — 硬上限，立即触发 */
    struct delayed_work    gc_work;    /* 周期 GC 工作队列项 */
    atomic_t               entries;    /* 当前条目数 */
    rwlock_t               lock;       /* 保护哈希表 */
    struct neigh_hash_table __rcu *nht; /* RCU 保护的哈希表指针 */
};

struct neigh_ops — 协议操作集

struct neigh_ops {
    int   family;
    /* 发送 ARP Request / NS（邻居请求报文）*/
    void (*solicit)(struct neighbour *, struct sk_buff *);
    /* 解析失败时报告错误 → ICMP Dest Unreachable */
    void (*error_report)(struct neighbour *, struct sk_buff *);
    /* 慢速路径（非 CONNECTED 状态） */
    int  (*output)(struct neighbour *, struct sk_buff *);
    /* 快速路径（CONNECTED 状态） */
    int  (*connected_output)(struct neighbour *, struct sk_buff *);
};

对于 ARP（net/ipv4/arp.c），solicit → arp_solicit()，error_report → arp_error_report()。对于 NDP，solicit → ndisc_solicit()。

关键函数解析

neigh_lookup / neigh_create / neigh_update / neigh_timer_handler
🧭 导读：本节梳理邻居子系统的核心函数调用链路。从数据包触发 ARP 查询，到条目创建、状态更新、定时器驱动的状态机流转，每个环节对应特定的内核函数。理解调用链是内核调试、eBPF tracing 的基础。

数据包发送调用链

1
IP 层路由查找后调用 ip_finish_output2()
确定 nexthop IP 后，调用 neigh_output()，此时 neigh 可能还未解析。
2
neigh_lookup() — 哈希表查询
以 (IP, net_device) 为 key 查找 neigh_table.nht，命中则返回 neighbour 指针并增加 refcnt；未命中则 __neigh_create() 创建新条目（状态 = INCOMPLETE）。
3
neigh->output() — 通过函数指针发包
CONNECTED 状态 → connected_output()（直接填 MAC 头发出）；INCOMPLETE/STALE 等 → output()（慢路径，检查状态、可能入队等待）。
4
neigh_ops->solicit() — 触发 ARP 请求
若需要解析，调用 arp_solicit()（IPv4）构造并发送 ARPOP_REQUEST 广播包，包被缓存在 arp_queue 中等待回复。
5
neigh_update() — 收到 ARP 回复后更新状态
从 arp_process() 调用，更新 ha[]（MAC地址）、nud_state → REACHABLE，然后发送 arp_queue 中缓存的包。通过 Netlink 发送 RTM_NEWNEIGH 通知用户态。

neigh_update() 逻辑摘要

/* net/core/neighbour.c — neigh_update() 简化逻辑 */
int neigh_update(struct neighbour *neigh, const u8 *lladdr, u8 new_state, ...)
{
    old_state = neigh->nud_state;

    if (lladdr != NULL) {
        /* 比较新旧 MAC，若不同则更新 ha[] */
        neigh_update_hhs(neigh); /* 同步更新 hh_cache */
    }

    /* 状态转换 */
    neigh->nud_state = new_state;
    neigh->updated   = jiffies;

    if (new_state & NUD_CONNECTED)
        neigh_connect(neigh); /* output = connected_output (快速路径) */
    else
        neigh_suspect(neigh); /* output = ops->output (慢速路径) */

    /* 发送 arp_queue 中积压的包 */
    while ((skb = __skb_dequeue(&neigh->arp_queue)) != NULL)
        neigh->output(neigh, skb);

    neigh_update_notify(neigh, nlmsg_pid); /* Netlink 通知 */
    return 0;
}

neigh_timer_handler() — 定时器驱动 NUD

/* 每个 neighbour 条目的定时器回调，驱动 NUD 状态机 */
static void neigh_timer_handler(struct timer_list *t)
{
    state = neigh->nud_state;
    now   = jiffies;

    if (state == NUD_REACHABLE) {
        if (time_before(now, neigh->confirmed + neigh->parms->reachable_time))
            return; /* 尚未超时，重置定时器 */
        neigh_update(neigh, NULL, NUD_STALE, ...); /* → STALE */
    }
    else if (state == NUD_DELAY) {
        if (time_before(now, neigh->confirmed + delay_probe_time))
            return; /* 等待上层确认中 */
        neigh_update(neigh, NULL, NUD_PROBE, ...); /* → PROBE */
    }
    else if (state == NUD_PROBE) {
        if (atomic_read(&neigh->probes) >= neigh_max_probes(neigh)) {
            neigh_invalidate(neigh); /* → FAILED, ICMP error */
            return;
        }
        neigh->ops->solicit(neigh, NULL); /* 发 ARP 单播探测 */
        atomic_inc(&neigh->probes);
    }
}

垃圾回收机制

neigh_periodic_work / neigh_forced_gc 与 gc_thresh 参数
🧭 导读：邻居缓存不能无限增长，内核通过两种 GC 机制控制条目数量：周期性 GC（neigh_periodic_work）和强制 GC（neigh_forced_gc）。理解 GC 三档阈值对排查"Neighbour table overflow"类故障至关重要。
图6-1：ARP 缓存 GC 三档阈值示意图（来源：RedHat / kernel.org 文档）
周期 GC — neigh_periodic_work()
  • 条目数 < gc_thresh1（128）：跳过，直接返回
  • 每 base_reachable_time / 2 触发一次
  • 扫描所有桶，删除 refcnt==1 且 used + gc_staletime 已超时的条目
  • NUD_PERMANENT 和 NUD_IN_TIMER 不会被删除
强制 GC — neigh_forced_gc()
  • 条目数 > gc_thresh3（1024）：新条目分配失败（ENOBUFS）
  • 条目数在 thresh2~thresh3 之间超过 5 秒：触发强制 GC
  • 内核日志：neighbour: arp_cache: neighbor table overflow!

核心 GC 代码逻辑

/* neigh_periodic_work：周期 GC，来自 net/core/neighbour.c */
static void neigh_periodic_work(struct work_struct *work)
{
    if (atomic_read(&tbl->entries) < tbl->gc_thresh1)
        goto out; /* 条目数低于 thresh1，GC 不运行 */

    /* 遍历 hash 桶，删除超时/FAILED 且无引用的条目 */
    if (atomic_read(&n->refcnt) == 1 &&
        (state == NUD_FAILED ||
         time_after(jiffies, n->used + n->parms->gc_staletime))) {
        neigh_cleanup_and_release(n);
    }
out:
    /* 下次 GC 在 base_reachable_time/2 之后 */
    schedule_delayed_work(&tbl->gc_work, tbl->parms.base_reachable_time >> 1);
}
生产环境注意 大规模 Kubernetes 集群（节点多、Pod 密集）经常遇到 ARP 表溢出问题。建议将 gc_thresh1/2/3 调大（如 1024/4096/8192），并通过 ip neigh show | wc -l 实时监控当前条目数。

IPv6 邻居发现（NDP）

NDP 与 ARP 的对比，NS/NA 报文机制
🧭 导读：IPv6 用 NDP（Neighbor Discovery Protocol，RFC 4861）替代了 ARP。NDP 基于 ICMPv6，使用 邻居请求（NS）/ 邻居通告（NA） 代替 ARP Request/Reply，关键改进是使用组播（solicited-node multicast）而非广播，大幅减少不相关主机的处理负担。NDP 与 ARP 共用相同的 NUD 状态机框架。
对比维度ARP（IPv4）NDP（IPv6）
协议层 独立协议（EtherType 0x0806） ICMPv6 子集（Type 135/136）
请求方式 广播（FF:FF:FF:FF:FF:FF） 组播（FF02::1:FFxx:xxxx，Solicited-Node）
安全性 无内置安全（易受 ARP 欺骗） Hop Limit=255 防止远程攻击；支持 SEcure ND（SEND）
内核实现 net/ipv4/arp.c
arp_tbl（AF_INET）		 net/ipv6/ndisc.c
nd_tbl（AF_INET6）
请求报文 ARPOP_REQUEST ICMPv6 Type=135（Neighbor Solicitation）
应答报文 ARPOP_REPLY ICMPv6 Type=136（Neighbor Advertisement）
溢出参数 /proc/sys/net/ipv4/neigh/ /proc/sys/net/ipv6/neigh/
统计查看 /proc/net/stat/arp_cache /proc/net/stat/ndisc_cache

NDP 地址解析流程

# 抓取 IPv6 邻居请求/通告（NS/NA）
$ sudo tcpdump -i eth0 -v "icmp6 and (ip6[40] == 135 or ip6[40] == 136)"

# 查看 IPv6 邻居缓存
$ ip -6 neigh show
2001:db8::1 dev eth0 lladdr 00:11:22:33:44:55 REACHABLE
fe80::1     dev eth0 lladdr 00:11:22:33:44:55 STALE

# 查看 IPv6 邻居统计
$ cat /proc/net/stat/ndisc_cache

# 调整 IPv6 邻居参数（与 IPv4 类似）
$ sysctl net.ipv6.neigh.eth0.base_reachable_time_ms
$ sysctl net.ipv6.neigh.default.gc_thresh3
NDP Solicited-Node 组播 IPv6 地址 2001:db8::aabb:ccdd 对应的 solicited-node 组播地址为 ff02::1:ffbb:ccdd（取后24位）。只有 IP 后缀相同的主机才加入该组播组，因此 NS 请求仅影响极少数主机，而非所有人——这是比 ARP 广播更高效的设计。

参数默认值含义
ucast_solicit3PROBE 状态发送单播探测的最大次数（失败后 → FAILED）
mcast_solicit3INCOMPLETE 状态发送组播/广播探测的最大次数
app_solicit0发向用户态 ARP 守护进程（arpd）的最大探测次数

ARP 行为控制参数（来自 kernel.org ip-sysctl.txt）

参数（conf.<iface>.*）默认含义
arp_announce0 控制 ARP 请求中源 IP 的选取：
0=使用任意接口地址 1=避免非目标子网地址 2=始终使用最佳本地地址
arp_ignore0 控制如何响应 ARP 请求：
0=应答任意本地 IP 1=只应答入接口配置的 IP 2=+要求发送方与本接口同子网 8=拒绝所有
arp_filter0 基于路由表决定哪张网卡响应 ARP（多网卡同子网时有用）
arp_accept0 是否接受免费 ARP（GARP）更新 ARP 表：0=不接受 1=接受 2=同子网才接受（内核 5.x 新增）
arp_notify0 1=网卡上线或 MAC 变更时主动发送免费 ARP
proxy_arp0 1=启用代理 ARP，路由器代替远端主机回应 ARP 请求

快速调优示例

# 扩大 ARP 表容量（Kubernetes 大集群推荐）
$ sysctl -w net.ipv4.neigh.default.gc_thresh1=1024
$ sysctl -w net.ipv4.neigh.default.gc_thresh2=2048
$ sysctl -w net.ipv4.neigh.default.gc_thresh3=4096

# 写入 /etc/sysctl.d/99-neigh.conf 持久化
net.ipv4.neigh.default.gc_thresh1 = 1024
net.ipv4.neigh.default.gc_thresh2 = 2048
net.ipv4.neigh.default.gc_thresh3 = 4096

# 延长 REACHABLE 时间（减少重复解析）
$ sysctl -w net.ipv4.neigh.eth0.base_reachable_time_ms=60000

# LVS DR 模式：禁止 real server 回应 VIP 的 ARP
$ sysctl -w net.ipv4.conf.eth0.arp_ignore=1
$ sysctl -w net.ipv4.conf.eth0.arp_announce=2

用户态调试命令工具集

ip neigh / arping / tcpdump / arpwatch / nstat
🧭 导读：本节按场景分组整理所有常用调试命令。ip neigh 是现代 Linux 的首选工具（替代已弃用的 arp 命令），tcpdump 用于抓包分析，arping 用于主动触发和检测，arpwatch 用于持续监控。

ip neigh — 邻居缓存管理

# ——— 查看 ———
$ ip neigh show                        # 显示所有邻居条目
$ ip -4 neigh show                     # 只显示 IPv4
$ ip -6 neigh show                     # 只显示 IPv6
$ ip neigh show dev eth0               # 指定接口
$ ip neigh show 192.168.1.1            # 指定 IP
$ ip neigh show nud reachable          # 按 NUD 状态过滤
$ ip neigh show nud failed             # 查看失败条目
$ ip neigh show nud stale              # 查看过期条目
$ ip -s neigh show                     # 显示统计信息
$ watch -n 2 ip neigh show            # 实时监控变化

# ——— 添加静态条目 ———
$ ip neigh add 192.168.1.100 lladdr aa:bb:cc:dd:ee:ff dev eth0 nud permanent
$ ip neigh replace 192.168.1.100 lladdr aa:bb:cc:dd:ee:ff dev eth0

# ——— 删除 / 刷新 ———
$ ip neigh del 192.168.1.100 dev eth0  # 删除单个条目
$ ip neigh flush dev eth0             # 刷新接口所有动态条目
$ ip neigh flush all                   # 刷新所有条目（谨慎！）

# ——— 触发重解析 ———
$ ping -c 1 192.168.1.100             # 触发 ARP 解析后查看
$ ip neigh show 192.168.1.100

arping — 主动 ARP 探测

# 发送 ARP 请求，查看目标 MAC 和响应时间
$ arping -I eth0 -c 3 192.168.1.1

# 检测 IP 冲突（-D 模式：如果收到回复说明 IP 已被占用）
$ arping -D -I eth0 192.168.1.100
$ echo $?   # 0=IP 唯一，1=IP 冲突

# 发送免费 ARP，通知邻居更新 MAC 映射
$ arping -A -I eth0 192.168.1.10      # -A: ARP reply（Gratuitous）
$ arping -U -I eth0 192.168.1.10      # -U: ARP request（Gratuitous）

tcpdump — 抓包分析

# 抓取所有 ARP 流量
$ sudo tcpdump -i eth0 arp
$ sudo tcpdump -i eth0 -n arp         # 不解析主机名

# 只抓 ARP 回复（arp[6:2]=2 表示 ARPOP_REPLY）
$ sudo tcpdump -i eth0 "arp and arp[6:2] = 2"

# 检测针对特定 IP 的重复 ARP 回复（防 ARP 欺骗排查）
$ sudo tcpdump -i eth0 "arp[6:2]=2 and arp[14:4]=0xC0A80101"
# 0xC0A80101 = 192.168.1.1（目标 IP 的十六进制）

# 抓 IPv6 NS/NA（Type 135/136）
$ sudo tcpdump -i eth0 "icmp6 and (ip6[40] == 135 or ip6[40] == 136)"

nstat / /proc — 统计与 proc 接口

# 查看 ARP 缓存统计（allocs, destroys, hash_grows, res_failed 等）
$ cat /proc/net/stat/arp_cache
$ cat /proc/net/stat/ndisc_cache      # IPv6

# 统计当前条目数，对比 gc_thresh3
$ ip neigh show | wc -l
$ cat /proc/sys/net/ipv4/neigh/default/gc_thresh3

# 内核日志：检查 ARP 相关告警
$ dmesg | grep -i "neighbour\|arp"
$ journalctl -k | grep -i "neighbour table overflow\|duplicate"

arpwatch — 持续 MAC/IP 变更监控

# 启动 arpwatch 监控（需要先 mkdir /var/lib/arpd/）
$ sudo arpwatch -i eth0

# arpwatch 检测到 MAC 变更时会输出如下日志：
# changed ethernet address: 192.168.1.1 00:11:22:33:44:55 (old: aa:bb:cc:dd:ee:ff)

# 查看 arpwatch 数据库
$ cat /var/lib/arpwatch/arp.dat

典型故障排查

ARP 表溢出 / ARP 欺骗 / 邻居不可达 / Kubernetes 场景
🧭 导读：本节以"问题 → 现象 → 根因 → 排查步骤 → 解决方案"的结构，系统梳理生产环境中最常见的邻居子系统相关故障。每种故障都给出可直接使用的命令序列。

故障一：邻居表溢出（Neighbour table overflow）

图10-1：邻居表溢出故障排查决策树
【现象】内核日志：neighbour: arp_cache: neighbor table overflow!
        网络连接随机中断，部分 IP 无法通信

【Step 1】确认当前条目数与阈值
$ ip neigh show | wc -l
$ cat /proc/sys/net/ipv4/neigh/default/gc_thresh3

【Step 2】查看失败条目
$ ip neigh show nud failed | head -20

【Step 3】如需立即缓解，扩大阈值
$ sysctl -w net.ipv4.neigh.default.gc_thresh1=1024
$ sysctl -w net.ipv4.neigh.default.gc_thresh2=4096
$ sysctl -w net.ipv4.neigh.default.gc_thresh3=8192

【Step 4】定位 ARP 洪泛根因（扫描工具、僵尸 IP 等）
$ sudo tcpdump -i eth0 -n arp | awk '{print $NF}' | sort | uniq -c | sort -rn | head

故障二：ARP 欺骗 / IP 冲突

【现象】内核日志：arp: ip address x.x.x.x is not on eth0!
        或 duplicate IP address detected

【Step 1】用 arping 检测 IP 冲突
$ arping -D -I eth0 192.168.1.100
# 返回值 1 = 检测到冲突（有其他机器回应了此 IP 的 ARP）

【Step 2】抓 ARP 确认哪台机器在仿冒
$ sudo tcpdump -i eth0 -n arp &
$ sudo journalctl -k | grep -i "duplicate"

【Step 3】查看 ARP 条目 MAC 是否频繁变化
$ watch -n 1 "ip neigh show 192.168.1.1"

【Step 4】添加静态 ARP（临时防御）
$ ip neigh replace 192.168.1.1 lladdr aa:bb:cc:dd:ee:ff dev eth0 nud permanent

【Step 5】长期：部署 arpwatch 监控 MAC 变化
$ arpwatch -i eth0

故障三：邻居 FAILED —— 目标不可达

【现象】ping 报 "Destination Host Unreachable"
        ip neigh show 显示目标 IP 为 FAILED 状态

【Step 1】确认状态
$ ip neigh show 192.168.1.100
# 192.168.1.100 dev eth0  FAILED

【Step 2】确认路由与接口选择
$ ip route get 192.168.1.100
$ ip link show eth0                    # 确认接口 UP

【Step 3】抓包确认 ARP 请求是否发出，是否有回复
$ sudo tcpdump -i eth0 arp
# 若只有 Request 没有 Reply → 目标主机不在线 / 防火墙拦截
# 若有 Reply → 检查 MAC/VLAN 配置

【Step 4】手动刷新并重试
$ ip neigh del 192.168.1.100 dev eth0  # 清除 FAILED 条目
$ ping -c 3 192.168.1.100             # 重新触发解析
$ ip neigh show 192.168.1.100

故障四：Kubernetes / 容器网络 ARP 问题

【现象】Pod 间通信偶发不通，节点 ARP 表满，
        Calico/Flannel 接口 proxy_arp 配置错误

【Step 1】查看节点 Pod IP 相关 ARP 条目
$ ip neigh show | grep -E "10\.244\.|10\.0\."

【Step 2】确认 Calico 接口的 proxy_arp 配置
$ cat /proc/sys/net/ipv4/conf/cali*/proxy_arp
# Calico 要求为 1

【Step 3】检查并扩大邻居表阈值
$ ip neigh show | wc -l
$ sysctl net.ipv4.neigh.default.gc_thresh3

【Step 4】必要时刷新节点 ARP 表
$ sudo ip neigh flush all

第一性原理总结

从最基本的事实出发，重新理解邻居子系统的设计本质
🧭 导读：本节不再罗列技术细节，而是从第一性原理（First Principles）出发，追问邻居子系统存在的根本原因，以及每一个设计决策背后的物理和逻辑约束。理解"为什么"比"是什么"更能帮助你在新场景中举一反三。

🧱 第一原则：物理约束决定协议分层

以太网硬件只认识 MAC 地址，不认识 IP 地址。这是一个无法改变的物理事实。因此，任何在以太网上传输 IP 数据包的系统，都必须维护一张 IP → MAC 的映射表。邻居子系统的存在，完全是由这个物理约束决定的，而非软件设计的主观选择。

🔄 第二原则：状态机是对不确定性的最优建模

网络是不可靠的——主机可能宕机、MAC 可能变更、链路可能抖动。面对这种不确定性，有两种极端策略：① 每次发包都重新 ARP（代价极高）；② 永久缓存（可能过时导致断连）。NUD 状态机是两者之间的最优折中：乐观使用缓存（REACHABLE），惰性验证（STALE→DELAY），主动探测（PROBE），最终放弃（FAILED）。每个状态对应一种对"邻居当前可达性"的置信度假设。

⚡ 第三原则：快速路径与慢速路径的分离是性能优化的核心

neigh->output 函数指针的设计揭示了内核性能优化的第一法则：将判断逻辑从热路径（数据包转发）移出去。CONNECTED 状态时，函数指针指向 connected_output，发包路径零条件判断，直接填 MAC 头；其他状态时才走慢路径检查。状态变更时一次性更新函数指针，而非每包判断。

🗑️ 第四原则：缓存必须有淘汰机制，否则是内存泄漏

gc_thresh 三级阈值不是随意设定的数字，而是对"缓存无限增长"这个根本问题的工程答案。gc_thresh1 定义了"正常负载"，低于此无需清理；gc_thresh2 是"警戒线"，给 GC 5 秒窗口；gc_thresh3 是"熔断值"，超过则拒绝新条目——这与任何资源池设计（线程池、连接池）的思路完全一致。调优时，根据你的网络规模（同一 L2 域的主机数量）来设定这三个值，不是越大越好，而是刚好够用。

🔌 第五原则：通用框架 + 协议插件 = 可扩展设计

net/core/neighbour.c 是框架，neigh_ops 是插件接口。这正是"面向接口编程"原则在内核 C 代码中的体现：核心 GC、状态机、定时器、哈希表全部通用；solicit()、error_report() 由具体协议（ARP/NDP）实现。新增协议族只需实现这几个函数指针，无需修改框架代码——这也是 Linux 内核能在不动核心代码的情况下支持越来越多协议的原因。

📊 综合推论：排查邻居问题的第一性思路

任何邻居相关故障，都可以用以下三个问题来定位：

  1. 条目是否存在？（ip neigh show）→ 不存在则是解析问题
  2. 状态是什么？（REACHABLE/STALE/PROBE/FAILED）→ 不是 REACHABLE 则是可达性问题
  3. 容量是否够？（条目数 vs gc_thresh3）→ 接近阈值则是容量问题

三个问题对应三类根因：协议/二层问题（ARP 请求/回复）、可达性/拓扑问题（主机宕机、VLAN 隔离）、容量/规模问题（大规模集群调优）。从第一性原理出发，所有复杂故障都归属于这三类。
