TCP 可靠传输深度解析

基于 RFC 9293 · RFC 6298 · RFC 5681 · RFC 2018 权威规范

"TCP provides a reliable, in-order, byte-stream service to applications. TCP reliability consists of detecting packet losses (via sequence numbers) and errors (via per-segment checksums), as well as correction via retransmission." — RFC 9293, Section 2.2（2022年8月，TCP 当前权威规范）

可靠传输从第一性原理来看，本质上只有四件事：丢包重传、收到确认、按序传输、乱序处理。TCP 作为互联网最重要的传输层协议，通过一系列精密设计的机制将这四件事在复杂的网络环境中落地实现。本文档将逐一解析每个机制的原理与规范。

🔢 序列号机制

每字节唯一编号，实现精确确认与检测丢包

🔁 重传机制

RTO 超时重传 + 快速重传，双保险修复丢包

🪟 滑动窗口

流量控制，按序交付，缓冲乱序到达的数据

🎯 SACK 选项

精确告知已收到的非连续数据块，避免盲目重传

📈 拥塞控制

慢启动、拥塞避免、快重传、快恢复四大算法

🔒 校验和

16位校验和覆盖首部与数据，检测传输错误

核心机制

🔢 序列号与确认机制

序列号（Sequence Number）是 TCP 可靠传输的基石。RFC 9293 Section 3.4 明确指出，TCP 连接发送的每一个字节都有一个序列号，这使得每个字节都可以被单独确认。

"A fundamental notion in the design is that every octet of data sent over a TCP connection has a sequence number. Since every octet is sequenced, each of them can be acknowledged. The acknowledgment mechanism employed is cumulative so that an acknowledgment of sequence number X indicates that all octets up to but not including X have been received. This mechanism allows for straightforward duplicate detection in the presence of retransmission." — RFC 9293, Section 3.4（原文引用）

序列号空间

序列号空间是有限的，范围 0 到 232-1，所有序列号运算均为模 232 的无符号运算。TCP 规范中定义了以下关键发送端变量：

/* RFC 9293 Section 3.4 —— 发送序列空间关键变量 */
SND.UNA  = oldest unacknowledged sequence number   // 最旧未确认序号
SND.NXT  = next sequence number to be sent          // 下一个待发送序号
SEG.ACK  = acknowledgment from the receiving TCP peer // 对端确认号
RCV.NXT  = next sequence number expected on incoming  // 接收端期望的下一序号
RCV.WND  = receive window size                       // 接收窗口大小

/* 可接受 ACK 条件（新确认） */
SND.UNA < SEG.ACK <= SND.NXT

/* 判断报文段已被完全确认（可从重传队列移除） */
SEG.SEQ + SEG.LEN <= SEG.ACK

图2：TCP 发送序列空间划分，SND.UNA 为最旧未确认序号，SND.NXT 为下一发送序号

初始序列号（ISN）的选择

"To avoid confusion, we must prevent segments from one incarnation of a connection from being used while the same sequence numbers may still be present in the network from an earlier incarnation. ... TCP implementations MUST use the following type of ISN generator: The ISN generator MUST be initialized to a pseudo-random value at startup." — RFC 9293, Section 3.4.1
ℹ️
RFC 9293 要求 ISN 必须使用伪随机数生成，原因有二：一是防止旧连接残留报文干扰新连接（避免序号混淆）；二是安全性考虑，防止攻击者预测序列号伪造 TCP 报文。

累积确认（Cumulative ACK）

TCP 使用累积确认：ACK 号为 X 表示 X 之前的所有字节均已正确收到，期望收到序号为 X 的字节。这种设计简洁地解决了：

✅ 重复检测

重传的数据若已收到，累积 ACK 会自然跳过，接收方丢弃重复数据

✅ 丢包确认

ACK 号不前进 → 发送方知道对应序号数据丢失或未到达

✅ 持久性

即使 ACK 本身丢失，后续更大的 ACK 号也能覆盖，不会错误触发重传

核心机制

🔁 丢包重传机制

TCP 通过两种互补的机制检测并修复丢包：超时重传（Retransmission Timeout, RTO）和快速重传（Fast Retransmit）。两者分别应对不同的丢包场景。

"TCP reliability consists of detecting packet losses (via sequence numbers) and errors (via per-segment checksums), as well as correction via retransmission." — RFC 9293, Section 2.2

1. 超时重传（Timer-Based Retransmission）

"When the retransmission timer expires, do the following: (5.4) Retransmit the earliest segment that has not been acknowledged by the TCP receiver. (5.5) The host MUST set RTO ← RTO × 2 ('back off the timer')." — RFC 6298, Section 5（Computing TCP's Retransmission Timer）

超时重传的工作流程：发送方每次发出含数据的报文段时，若计时器未运行则启动它（设置超时为 RTO）。若在 RTO 时间内未收到 ACK，则重传最早的未确认报文段，并将 RTO 翻倍（指数退避）。

/* RFC 6298 Section 5 —— RTO 定时器管理算法 */

// (5.1) 发出数据时，若计时器未运行，启动计时器（超时=RTO）
if (!timer_running) start_timer(RTO);

// (5.2) 所有未确认数据均被 ACK 后，关闭计时器
if (all_data_acked) stop_timer();

// (5.3) 收到确认新数据的 ACK，重启计时器
if (new_ack_received) restart_timer(RTO);

// 计时器超时时：
// (5.4) 重传最旧的未确认报文段
retransmit(SND.UNA);
// (5.5) RTO 指数退避（Exponential Backoff）
RTO = RTO * 2;  // MUST, 最大值不低于 60 秒
// (5.6) 重启计时器
start_timer(RTO);

2. 快速重传（Fast Retransmit）

"When the third duplicate ACK is received, a TCP MUST set ssthresh to no more than the value given in equation (4). ... The lost segment starting at SND.UNA MUST be retransmitted and cwnd set to ssthresh plus 3*SMSS." — RFC 5681, Section 3.2（TCP Congestion Control）

快速重传不等待 RTO 超时，而是利用重复 ACK（Duplicate ACK）作为丢包信号。当接收方收到乱序报文段时，会立刻发送一个 ACK，重复上一个正确接收的序号。当发送方连续收到 3 个重复 ACK 时，认为对应报文段已丢失，立即重传：

图3：快速重传流程 —— 收到第3个重复 ACK 时立即重传，无需等待 RTO 超时

两种重传机制对比

特性超时重传（RTO）快速重传
触发条件RTO 计时器超时收到 3 个重复 ACK
响应速度较慢（等待整个 RTO）快速（约 1 RTT 内）
适用场景严重丢包、网络断路偶发性单包丢失

⚠️
Karn 算法：RFC 6298 Section 3 规定，TCP MUST use Karn's algorithm for taking RTT samples. That is, RTT samples MUST NOT be made using segments that were retransmitted。因为对重传报文段的 ACK 无法区分是对第一次还是第二次发送的确认，使用时间戳选项（RFC 7323）时除外。

超时计算

⏱️ RTO 计算（RFC 6298）

RTO（Retransmission Timeout，重传超时）的准确计算至关重要。设得太小会导致不必要的重传，设得太大则增加丢包恢复延迟。RFC 6298 Computing TCP's Retransmission Timer 定义了标准算法。

SRTT 与 RTTVAR 计算

"(2.2) When the first RTT measurement R is made, the host MUST set: SRTT ← R; RTTVAR ← R/2; RTO ← SRTT + max(G, K×RTTVAR) where K = 4.
(2.3) When a subsequent RTT measurement R' is made: RTTVAR ← (1−β)×RTTVAR + β×|SRTT−R'|; SRTT ← (1−α)×SRTT + α×R'; then RTO ← SRTT + max(G, K×RTTVAR). The above SHOULD be computed using alpha=1/8 and beta=1/4." — RFC 6298, Section 2（Computing TCP's Retransmission Timer, June 2011）

这套算法同时追踪 RTT 的均值（SRTT）和方差（RTTVAR），RTO 设为均值加 4 倍方差，既能跟踪 RTT 变化趋势，又能容纳 RTT 抖动：

SRTT ← (1 − α) × SRTT + α × R'　　（α = 1/8）
RTTVAR ← (1 − β) × RTTVAR + β × |SRTT − R'|　　（β = 1/4）
RTO ← SRTT + max(G, 4 × RTTVAR)　　（G 为时钟粒度，min RTO = 1s）

图4：RTO 自适应计算 —— SRTT 追踪均值，RTTVAR 追踪方差，超时后指数退避

RTO 约束条件（RFC 6298）

规则要求级别
初始 RTO（无 RTT 样本时）设为 1 秒SHOULD
最小 RTO向上取整到 1 秒SHOULD
最大 RTO至少 60 秒MAY
超时后退避RTO ← RTO × 2MUST
SYN 段特殊处理若 RTO < 3s，数据传输开始后重置为 3sMUST
RTT 采样（重传段）不得使用重传段进行 RTT 采样（Karn 算法）MUST
💡
为什么 α=1/8，β=1/4？这是 Van Jacobson 1988年论文（RFC 6298 参考 [JK88]）中建议的值。α 控制 SRTT 的平滑程度（1/8 = 指数加权移动平均），β 控制方差跟踪速度（1/4 使方差对变化更敏感），二者使 RTO 既稳定又能快速适应网络变化。

流量控制

🪟 滑动窗口与流量控制

TCP 的滑动窗口机制解决两个问题：一是流量控制（防止发送方淹没接收方）；二是实现按序交付（将数据正确送达应用层）。

"The window size MUST be treated as an unsigned number, or else large window sizes will appear like negative windows and TCP will not work. ... The amount of data that can be sent is limited by the minimum of cwnd and rwnd." — RFC 9293, Section 3.1 & RFC 5681, Section 1

接收窗口（rwnd）—— 流量控制

接收方在每个 ACK 中通过 Window 字段通告其当前可用缓冲空间（rwnd）。发送方不得让网络中未确认的数据量超过 rwnd 与 cwnd（拥塞窗口）的最小值。

图5：滑动窗口机制 —— 窗口随 ACK 向右滑动，实现流量控制与按序发送

零窗口探测（Zero Window Probing）

"A TCP implementation MUST include a SWS avoidance algorithm in the sender (MUST-38). ... When the receiver's window closes to zero, the sender MUST probe the window (MUST-36). The probe SHOULD be sent after no more than one RTO (SHLD-29) and use exponential backoff (SHLD-30)." — RFC 9293, Section 3.8.6

当接收方通告零窗口（rwnd=0）时，发送方停止发送数据，但必须定期发送窗口探测报文（1字节），以防止接收方窗口恢复后的 ACK 丢失导致双方死锁（死锁场景：接收方 rwnd 恢复但 ACK 丢失，发送方永久等待）。

糊涂窗口综合症（SWS）避免

RFC 9293 Section 3.8.6 要求发送方和接收方都必须实现 SWS 避免算法：

发送方 SWS 避免

当可用窗口小于 MSS 且没有数据可以完整发送时，发送方应等待，避免发送大量小报文段（Nagle 算法）

接收方 SWS 避免

接收方不应通告极小的窗口增量；应等到窗口增长到至少 min(MSS, 接收缓冲区/2) 时才更新窗口

乱序处理

🎯 乱序处理与 SACK 选项

IP 层是无序的，TCP 报文段可能乱序到达。TCP 必须将乱序数据缓存，等待缺口填满后按序交付应用层。RFC 2018 定义了选择性确认（SACK）选项，使发送方能精确知道哪些数据已被接收方缓存，从而只重传真正丢失的段。

"TCP SHOULD queue segments that arrive out of order (SHLD-31). ... The SACK option is to be used to convey extended acknowledgment information from the receiver to the sender over an established TCP connection. ... With selective acknowledgments, the data receiver can inform the sender about all segments that have arrived successfully, so the sender need retransmit only the segments that have actually been lost." — RFC 9293 & RFC 2018（TCP Selective Acknowledgment Options, Oct 1996）

SACK 选项格式（RFC 2018）

| Kind=5 | Length |
+--------+--------+--------+--------+
|    Left Edge of 1st Block       |   // 第1个连续块左边界
+--------+--------+--------+--------+
|    Right Edge of 1st Block      |   // 第1个连续块右边界
+--------+--------+--------+--------+
/          . . .                    /
+--------+--------+--------+--------+
|    Left Edge of nth Block        |
+--------+--------+--------+--------+
|    Right Edge of nth Block       |

// TCP 选项区最多 40 字节，SACK 最多携带 4 个块
// 每个块 8 字节（左右边界各 4 字节）

图6：SACK 精确重传 —— 接收方通过 SACK 块告知已收到的非连续数据，发送方只重传真正丢失的段

SACK 接收方行为规则（RFC 2018）

"The first SACK block MUST specify the contiguous block of data containing the segment which triggered this ACK. ... The data receiver SHOULD include as many distinct SACK blocks as possible in the SACK option. ... The SACK option SHOULD be filled out by repeating the most recently reported SACK blocks so that in normal operation, any segment remaining part of a non-contiguous block is reported in at least three successive SACK options." — RFC 2018, Section 4

D-SACK（重复 SACK，RFC 2883）

RFC 2883 扩展了 SACK，允许接收方用 SACK 的第一个块报告重复收到的数据（D-SACK）。这让发送方能识别出：

虚假重传检测

ACK 丢失导致的重传 → 接收方用 D-SACK 告知，发送方知道该次重传是多余的

网络乱序检测

网络中数据包重新排序，接收方可区分乱序与丢包，避免不必要的快速重传

累积 ACK vs SACK 对比

特性纯累积 ACKSACK（RFC 2018）
重传精度只知道第一个缺口位置精确知道所有缺口
多包丢失处理每次 RTO 只能恢复一个丢包一次 RTO 可恢复多个丢包
带宽效率可能重传已到达的数据只重传真正丢失的数据
协商方式无需协商，默认支持SYN 中协商 SACK-Permitted
实现复杂度简单较复杂（需维护记分板）

进阶机制

📈 拥塞控制（RFC 5681）

拥塞控制是 TCP 可靠传输的重要保障 —— 虽然它不直接属于"可靠性"本身，但若没有拥塞控制，网络会因过载而大量丢包，从根本上破坏可靠传输。RFC 5681 定义了四个互相配合的拥塞控制算法。

"This document defines TCP's four intertwined congestion control algorithms: slow start, congestion avoidance, fast retransmit, and fast recovery. ... TCP MUST implement the slow start and congestion avoidance algorithms (MUST-19)." — RFC 5681（TCP Congestion Control, September 2009）

核心变量

/* RFC 5681 —— 拥塞控制核心变量 */
cwnd     = congestion window    // 拥塞窗口：限制网络中在途数据量
ssthresh = slow start threshold // 慢启动阈值：决定使用哪种算法
SMSS     = sender max segment size // 发送方最大报文段大小

// 实际可发送数据量受双重限制：
FlightSize ≤ min(cwnd, rwnd)

// 初始窗口（IW）设置（RFC 5681 Section 3.1）：
// SMSS > 2190 bytes: IW ≤ 2*SMSS，不超过2段
// 1095 < SMSS ≤ 2190: IW ≤ 3*SMSS，不超过3段
// SMSS ≤ 1095 bytes: IW ≤ 4*SMSS，不超过4段

图7：TCP 拥塞控制状态机 —— 慢启动、拥塞避免、快速重传/恢复四种状态转换

四大算法详解（RFC 5681）

① 慢启动（Slow Start）

连接建立后（或 RTO 超时后），cwnd 初始化为 IW（约 2~4 个 SMSS）。每收到一个 ACK，cwnd 增加一个 SMSS，即每个 RTT 内窗口翻倍（指数增长），快速探测可用带宽。当 cwnd 达到 ssthresh 时切换到拥塞避免。

cwnd ← cwnd + SMSS　（每个 ACK）

② 拥塞避免（Congestion Avoidance）

当 cwnd ≥ ssthresh 时，每个 RTT 内 cwnd 仅增加 1 个 SMSS（线性增长），谨慎探测额外带宽。RFC 5681 给出的近似公式：

cwnd ← cwnd + SMSS × SMSS / cwnd　（每个 ACK，等效每 RTT 增加 1 SMSS）

③ 快速重传（Fast Retransmit）

收到第 3 个重复 ACK 时，立即重传最旧未确认的报文段，同时：

ssthresh ← max(FlightSize / 2, 2 × SMSS)

④ 快速恢复（Fast Recovery）

"The reason for not performing slow start is that the receipt of the duplicate ACKs not only indicates that a segment has been lost, but also that segments are most likely leaving the network... since the receiver can only generate a duplicate ACK when a segment has arrived, that segment has left the network and is in the receiver's buffer." — RFC 5681, Section 3.2

快速重传后，每收到一个重复 ACK，cwnd 增加 1 SMSS（窗口膨胀，利用这段时间传输新数据）。当收到覆盖重传数据的新 ACK 后，cwnd 设为 ssthresh，退出快速恢复，切换到拥塞避免。

cwnd = ssthresh + 3 × SMSS　（进入快速恢复）
cwnd ← cwnd + SMSS　（每个后续 dupACK）
cwnd ← ssthresh　（新 ACK 到来，退出快速恢复）

连接管理

🤝 连接建立与关闭

TCP 的可靠性从连接建立就开始保障 —— 三次握手（Three-Way Handshake）确保双方序列号同步，四次挥手（Four-Way Teardown）确保所有数据被可靠交付后才关闭连接。

"The three-way handshake requires each side of the connection to learn the initial sequence number for the other side. This handshake is referred to as a 'three-way handshake' because a minimum of three messages are required to complete the exchange. ... A fundamental design choice was the use of a finite sequence number space. ... To avoid confusion between old and new connections, each TCP connection has a unique pair of (source port, destination port) and an initial sequence number." — RFC 9293, Section 3.5

图8：三次握手确保双方 ISN 同步；四次挥手确保双向数据均被可靠传输后才关闭连接

TIME-WAIT 状态的意义

"A TCP implementation MUST stay in the TIME-WAIT state for a time of 2×MSL (Maximum Segment Lifetime) (MUST-13). ... TIME-WAIT serves two purposes: (1) allow the full-duplex connection to reliably close, and (2) allow old duplicate segments to expire." — RFC 9293, Section 3.6

防止最终 ACK 丢失

若最后的 ACK 丢失，服务端会重传 FIN，TIME-WAIT 使客户端能重新发送 ACK

防止旧报文干扰新连接

等待 2×MSL 确保网络中所有旧连接的报文段都已超时消失，避免被新连接误收

数据完整性

🔒 校验和（Checksum）

TCP 的校验和（Checksum）是可靠传输在数据完整性层面的保障 —— 它检测传输过程中数据位错误，确保接收到的数据与发送的数据完全一致。

"The checksum field is the 16-bit ones' complement of the ones' complement sum of all 16-bit words in the header and text. ... The checksum also covers a pseudo-header conceptually prefixed to the TCP header. The pseudo-header is 96 bits for IPv4 and 320 bits for IPv6. Including the pseudo-header in the checksum gives the TCP connection protection against misrouted segments." — RFC 9293, Section 3.1
"TCP MUST implement generating and checking TCP checksums (MUST-2, MUST-3). ... The sender MUST compute the checksum and the receiver MUST check it." — RFC 9293, Appendix B（要求级别: MUST）

图9：TCP 校验和覆盖范围 —— 伪首部 + TCP首部 + TCP数据，防路由错误与数据位错误

校验和的局限性

⚠️
TCP 16位校验和是弱校验，可检测大多数单字节错误，但不能检测所有双字节错误。对于高可靠性数据传输，应在应用层额外使用 CRC-32 或更强的校验机制（如 TLS 的消息认证码）。

第一性原理

💡 第一性原理总结

回到最开始的问题：可靠传输的本质只有四件事。让我们用第一性原理视角，看 TCP 如何将每一件事落地：

① 丢包重传

问题：网络是不可靠的，报文段可能丢失。

TCP 解法：发送方维护重传队列，所有已发未确认的段都保留副本。双保险机制：RTO 超时重传（RFC 6298，兜底）+ 快速重传（3×dupACK，RFC 5681，快速响应）。RTO 用指数退避避免在拥塞时雪上加霜。

② 收到确认

问题：发送方需要知道数据是否被收到。

TCP 解法：累积 ACK（RFC 9293 §3.4）—— 每字节编号，ACK=X 表示 X 之前全部收到。SACK 选项（RFC 2018）扩展为选择性确认，精确告知非连续接收块，避免盲目重传。ACK 号由接收方在每段回复中携带，连接建立后始终有效。

③ 按序传输

问题：IP 不保证顺序，应用层需要有序数据。

TCP 解法：32位序列号给每字节唯一编号（RFC 9293 §3.4）。接收方用 RCV.NXT 追踪期望序号，只有连续数据才向应用层交付。滑动窗口控制发送速率，ISN 用伪随机数防旧连接干扰。

④ 乱序处理

问题：网络中报文段可能乱序到达，不能丢弃后续数据。

TCP 解法：接收方将乱序段缓存（SHLD-31，RFC 9293），等待缺口填满后一并交付。SACK 块精确描述已缓存的非连续区间。D-SACK（RFC 2883）区分重复段与乱序段，防止误触发重传。

图10：第一性原理全景 —— 四个核心目标与 TCP 对应机制的映射关系

一句话总结

TCP 可靠传输的本质是：用序列号给每字节编号以追踪和排序，用累积 ACK + SACK 精确反馈收到了什么，用重传队列 + RTO/快速重传 修复丢失，用接收缓冲区 临时存放乱序到达的数据并最终按序交付。拥塞控制、流量控制、校验和、握手协议则是这一系列核心机制的外围保障，共同构筑起在不可靠网络上实现可靠字节流传输的完整体系。
