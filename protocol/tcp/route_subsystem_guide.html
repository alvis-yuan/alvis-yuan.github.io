1. 路由基本概念

📖导读：理解路由，首先要理解"包应该往哪走"这个核心问题。本节介绍路由决策的三种结果、路由条目的组成要素，以及 Linux 内核如何在宏观上组织路由子系统。

1.1 路由的核心问题

对于每一个到达或发出的数据包，内核的路由子系统都需要回答一个问题：这个包应该去哪里？答案只有三种可能：

内核路由决策的三种结果（来源：kernel-internals.org）

1.2 路由条目组成要素

字段说明示例
目标前缀 (Destination)匹配目的 IP 地址的网络前缀192.168.1.0/24
网关 (Gateway / via)下一跳路由器 IP，若在同链路则为 0via 10.0.0.1

1.3 路由作用域 (Scope)

作用域定义了路由的可达范围，内核在 include/uapi/linux/rtnetlink.h 中定义了以下值：

  • RT_SCOPE_HOST (254)：仅本机可达，如 loopback 地址
  • RT_SCOPE_LINK (253)：仅同链路可达，直连路由
  • RT_SCOPE_SITE (200)：站点内部范围（IPv6）
  • RT_SCOPE_UNIVERSE (0)：全球可达，普通单播路由

2. 路由表与 RPDB（路由策略数据库）

📋导读：Linux 支持多路由表并存，通过"规则（Rules）"决定查哪张表。本节介绍内置路由表、RPDB（Routing Policy Database）的工作机制，以及规则扫描算法。这是理解策略路由的基础。

2.1 内置路由表

根据内核官方文档 policy-routing.txt，不启用策略路由（CONFIG_IP_MULTIPLE_TABLES 未设置）时，系统有两张内置路由表：

表 ID名称内容
255RT_TABLE_LOCAL本地接口地址、广播地址，由内核自动维护
254RT_TABLE_MAIN所有普通路由默认存放于此
253RT_TABLE_DEFAULT默认表，通常为空或存放默认网关
0RT_TABLE_UNSPEC保留，不可使用

2.2 RPDB 规则扫描算法

Linux 在查找路由时，并不直接查路由表，而是先扫描 RPDB（Routing Policy Database） 中的规则列表。规则按优先级（pref 值）从小到大扫描：

RPDB 默认规则扫描流程（来源：kernel.org policy-routing.txt）

默认三条规则（通过 ip rule show 查看）：

0:      from all lookup local    # 最高优先级，查本地表
32766:  from all lookup main     # 查主路由表
32767:  from all lookup default  # 查默认表（通常为空）

📌 规则的匹配条件可以是：源地址（from）、目的地址（to）、入接口（iif）、出接口（oif）、防火墙标记（fwmark）、TOS/DSCP 字段。

3. FIB 与 LC-Trie 算法

🌳导读：FIB（Forwarding Information Base，转发信息库）是内核路由表的实现。Linux 使用 LC-Trie（Level Compressed Trie，层压缩前缀树）数据结构来实现高效的最长前缀匹配。本节将解析其算法原理与实现文件。

3.1 FIB 是什么

FIB 是路由表在内核中的具体实现形式，存储了所有路由条目，支持最长前缀匹配（Longest Prefix Match, LPM）查找。主要实现文件：

  • net/ipv4/fib_trie.c：LC-Trie 数据结构与查找算法
  • net/ipv4/fib_frontend.c：FIB 对外接口，处理 netlink 消息
  • net/ipv4/fib_semantics.c：路由语义处理（nexthop、scope 等）
  • include/net/ip_fib.h：核心数据结构定义

3.2 LC-Trie 结构

根据内核官方文档 Documentation/networking/fib_trie.txt，LC-Trie 有两种节点类型：

节点类型说明
leaf（叶子节点）终端节点，保存相关 key 的副本，以及按前缀长度排序的路由条目 hlist。对应 struct leaf 和 struct leaf_info。
tnode（内部节点）内部节点，持有子节点（leaf 或 tnode）指针数组，通过 key 的某个子集索引。

关键概念

  • Bits（tnode）：用于索引子数组的 key 段位数，即"子索引"。
  • Pos（tnode）：key 中用于索引子数组的起始位置（路径压缩相关）。
  • 路径压缩（Path Compression）：若某 tnode 的 pos 与父节点不相邻，跳过的位称为"skipped bits"，查找时直接跳过。
  • 层压缩（Level Compression）：将满子节点的子树上移一层，使内部节点持有更大的子数组，避免纯二叉树；反之若子数组稀疏则"折半（halve）"。

LC-Trie 结构示意（源自 fib_trie.txt）

3.3 查找算法

根据 Documentation/networking/fib_trie.txt 的描述，fn_trie_lookup() 是主查找函数，查找过程分两个阶段：

  1. 精确匹配阶段：从根节点向下遍历，逐段提取 key，直到找到叶子节点，调用 check_leaf() 做语义匹配（fib_semantic_match）。
  2. 最长前缀匹配阶段：若精确匹配失败，前缀长度逐步减小，从 trie 中向上回溯，每次在 child array 中将子索引末尾最低位的 1 清零，寻找匹配的更短前缀，直到索引全为零再继续向上回溯。

📌 resize() 函数 是动态 trie 的核心，每次 trie 变化后被调用，通过 inflate()（倍增子数组）和 halve()（减半子数组）实现层压缩优化，保证查找深度最小化。

📎 原文引用：Documentation/networking/fib_trie.txt (v3.14) | net/ipv4/fib_trie.c (GitHub)

4. 关键内核数据结构

🏗️导读：阅读内核路由代码，必须先掌握核心数据结构。本节介绍 fib_table、fib_result、rtable、dst_entry 等结构，它们是路由子系统的"骨骼"。

4.1 fib_table — 路由表

定义于 include/net/ip_fib.h，每张路由表对应一个实例：

// include/net/ip_fib.h
struct fib_table {
    struct hlist_node   tb_hlist;      // 挂载在哈希表上
    u32                 tb_id;         // 表 ID: 254=main, 255=local
    int                 tb_num_default;// 默认路由数
    struct rcu_head     rcu;           // RCU 保护
    unsigned long      *tb_data;       // 指向 LC-Trie 根
};

4.2 fib_result — 查找结果

fib_lookup() 成功后将结果填入此结构（定义于 include/net/ip_fib.h）：

struct fib_result {
    __be32          prefix;     // 匹配的网络前缀
    unsigned char   prefixlen;  // 前缀长度
    struct fib_info *fi;        // 下一跳信息：网关、设备、metrics
    struct fib_nh_common *nhc; // 选中的下一跳（ECMP 时选出一个）
    u8              type;       // RTN_UNICAST / RTN_LOCAL / …
    u8              scope;
    struct fib_table *table;   // 命中的路由表
};

4.3 dst_entry — 路由缓存条目

定义于 include/net/dst.h，路由决策结果被封装在此结构中，附着到 sk_buff：

// include/net/dst.h
struct dst_entry {
    struct net_device *dev;     // 输出设备
    struct dst_ops    *ops;     // ip_dst_ops 等操作集
    void (*output)(struct net *, struct sock *, struct sk_buff *);
    // → ip_output() 本地发出，ip_forward() 转发
    unsigned long     _metrics; // MTU, RTT 等指标
    unsigned long     expires;  // 缓存过期时间
};

4.4 rtable — IPv4 路由表项

struct rtable 包含 dst_entry，是 IPv4 路由查找的最终结果：

struct rtable {
    struct dst_entry    dst;
    int                 rt_genid;
    unsigned int        rt_flags;    // RTCF_LOCAL, RTCF_BROADCAST…
    __be32              rt_gw4;      // IPv4 网关地址
    int                 rt_iif;
    u32                 rt_mtu_locked:1; // Path MTU 是否锁定
};

5. 路由查找全流程

🔍导读：数据包是如何经过路由子系统的？本节将追踪内核调用链，分别从接收路径（RX）和发送路径（TX）两个方向，解析路由决策的完整执行流程。

5.1 接收路径（RX）

当一个 IP 数据包从网卡到达时，经 ip_rcv() → ip_rcv_finish() 进入路由子系统：

// net/ipv4/route.c
int ip_route_input_noref(struct sk_buff *skb, __be32 daddr, __be32 saddr,
                          dscp_t dscp, struct net_device *dev)
{
    // 1. 执行 FIB 查找（自 Linux 3.6 起无路由缓存）
    err = fib_lookup(net, &fl4, &res, 0);

    // 2. 根据结果类型决定动作
    if (res.type == RTN_LOCAL) {
        // 包是给本机的：设置 input = ip_local_deliver
        // skb->_skb_refdst = &rt->dst
    } else if (res.type == RTN_UNICAST) {
        // 需要转发：调用 ip_route_input_slow()
        ip_route_input_slow(skb, daddr, saddr, ...);
    } else {
        // RTN_UNREACHABLE / RTN_BLACKHOLE / RTN_PROHIBIT
        goto martian_destination;
    }
}

5.2 发送路径（TX）

在 ip_queue_xmit() 中，先检查 socket 缓存的路由，未命中则重新查找：

// 发送路径路由查找（net/ipv4/ip_output.c）
rt = (struct rtable *)__sk_dst_check(sk, 0);
if (!rt) {
    rt = ip_route_output_ports(net, fl4, sk, daddr, saddr, ...);
    sk_setup_caps(sk, &rt->dst);  // 缓存到 socket
}

RX/TX 路由查找调用链（来源：kernel-internals.org & net/ipv4/route.c）

5.3 路由缓存的变化

⚠️ 注意：自 Linux 3.6 起，全局路由缓存（route cache）被移除，改为依赖 dst_entry 附着在 sk_buff 和 sock 上的 per-socket/per-packet 缓存机制，以避免缓存污染和 DoS 攻击风险。

6. 邻居子系统与 ARP

🤝导读：路由子系统找到下一跳 IP 后，还需要知道其 MAC 地址才能在链路层发送数据帧。这就是邻居子系统（Neighbour Subsystem）的职责——通过 ARP 协议完成 IP→MAC 的映射。

6.1 struct neighbour — 邻居表项

定义于 net/core/neighbour.c，每个已知的二层邻居对应一个实例：

// net/core/neighbour.c
struct neighbour {
    struct neighbour __rcu *next;
    struct neigh_table  *tbl;      // IPv4: arp_tbl
    struct neigh_parms  *parms;
    unsigned long       confirmed; // 最近确认可达时间
    u8                  flags;
    u8                  nud_state; // NUD 状态机
    u8                  ha[ALIGN(MAX_ADDR_LEN, sizeof(unsigned long))]; // MAC 地址
    struct hh_cache     hh;        // 缓存的 L2 帧头（快速路径用）
    int (*output)(struct neighbour *, struct sk_buff *);
};

7. 策略路由（Policy Routing）

🎯导读：标准路由仅基于目的地址做决策，策略路由则允许根据源地址、接口、防火墙标记、TOS 等多种因素路由数据包。本节介绍策略路由的规则类型、内核实现与典型应用场景。

7.1 规则类型

根据内核文档 policy-routing.txt（作者 Alexey Kuznetsov），规则类型包括：

规则类型说明
RTP_ROUTE在关联的路由表中查找路由
RTP_DROP静默丢弃数据包
RTP_REJECT丢弃并回复 ICMP NET UNREACHABLE
RTP_PROHIBIT丢弃并回复 ICMP COMM. ADM. PROHIBITED
RTP_NAT查找后若命中则做源地址转换

7.2 典型配置：多出口按源路由

场景：主机有两个出口，来自不同源 IP 的流量走不同的上行链路：

# 1. 创建自定义路由表（编辑 /etc/iproute2/rt_tables）
echo '100 isp1' >> /etc/iproute2/rt_tables
echo '200 isp2' >> /etc/iproute2/rt_tables

# 2. 在各表中添加默认路由
ip route add default via 10.0.1.1 table isp1
ip route add default via 10.0.2.1 table isp2

# 3. 添加规则：按源地址选表
ip rule add from 192.168.1.0/24 table isp1
ip rule add from 192.168.2.0/24 table isp2

# 4. 验证规则
ip rule show
ip route get 8.8.8.8 from 192.168.1.10

7.3 规则匹配维度

  • from PREFIX：源地址前缀
  • to PREFIX：目的地址前缀
  • iif IFNAME：入接口名
  • oif IFNAME：出接口名
  • fwmark MARK：iptables 防火墙标记（结合 mangle 表使用）
  • tos TOS：TOS/DSCP 字段
  • pref PRIO：规则优先级（数值越小越优先）

📌 策略路由查找算法（来自 policy-routing.txt）：扫描规则列表；若规则匹配，则执行对应动作；若找到路由则返回；若未找到或遇 THROW 节点，则继续扫描下一条规则。

📎 原文引用：kernel.org/doc/Documentation/networking/policy-routing.txt

8. ECMP 等价多路径路由

⚖️导读：ECMP（Equal-Cost Multipath）允许同一目的地存在多条等价路径，内核通过哈希算法将流量分散到多个下一跳，实现负载均衡。本节介绍 ECMP 的内核实现机制与新一代 Nexthop Objects。

8.1 基本配置

多路径路由通过 nexthop 关键字指定多个下一跳，流量按哈希策略分配：

# 添加双路径等价路由（流量均分到 eth0 和 eth1）
ip route add 10.0.0.0/8 \
    nexthop via 192.168.1.1 dev eth0 weight 1 \
    nexthop via 192.168.2.1 dev eth1 weight 1

# weight 可设置不等权，如 2:1 分配
ip route add 10.0.0.0/8 \
    nexthop via 192.168.1.1 dev eth0 weight 2 \
    nexthop via 192.168.2.1 dev eth1 weight 1

8.2 哈希策略

内核使用流哈希（flow hash）保证同一流（5-tuple）始终走同一路径，避免乱序：

# 查看/设置 ECMP 哈希策略
cat /proc/sys/net/ipv4/fib_multipath_hash_policy
# 0 = L3（仅源/目的 IP）
# 1 = L4（5 元组：src/dst IP + src/dst port + proto）
# 2 = L3+L4

sysctl -w net.ipv4.fib_multipath_hash_policy=1  # 推荐用 L4

8.3 Nexthop Objects（内核 5.x+）

根据 LWN.net 763950，内核引入了独立的 Nexthop Object 机制，允许多条路由共享同一 nexthop 对象，更新 nexthop 时无需逐一更新路由：

# 创建独立 nexthop 对象
ip nexthop add id 1 via 10.99.1.2 dev veth1
ip nexthop add id 2 via 10.99.3.2 dev veth3

# 创建 ECMP nexthop 组
ip nexthop add id 1001 group 1/2

# 路由引用 nexthop 对象 ID
ip route add 10.1.1.0/24 nhid 1
ip route add 10.1.2.0/24 nhid 1001  # ECMP 组

# 查看
ip nexthop show
ip route show

ECMP 流量分发示意（来源：kernel-internals.org & LWN.net）

9. VRF — 虚拟路由与转发

🏝️导读：VRF（Virtual Routing and Forwarding）让一台 Linux 主机同时运行多个独立的路由域，每个 VRF 拥有独立的路由表。这是多租户网络环境的核心技术。本节内容严格来自 Linux 内核官方文档。

9.1 VRF 设计原理

根据 docs.kernel.org/networking/vrf.html，VRF 设备与 ip rules 结合，提供 VRF-lite 虚拟路由转发能力。每个 VRF 设备关联一张路由表，网络接口被 "enslaved" 到 VRF 设备上：

+-----------------------------+
|           vrf-blue          |  ==> route table 10
+-----------------------------+
   |        |            |
+------+ +------+     +-------------+
| eth1 | | eth2 | ... |    bond1    |
+------+ +------+     +-------------+
                         |       |
                     +------+ +------+
                     | eth8 | | eth9 |
                     +------+ +------+

9.2 VRF 配置步骤

# 步骤1：创建 VRF 设备并关联路由表
ip link add vrf-blue type vrf table 10
ip link set dev vrf-blue up

# 步骤2：内核 4.8+ 自动添加 l3mdev 规则（覆盖所有 VRF）
# 旧内核需手动添加 oif/iif 规则：
ip ru add oif vrf-blue table 10
ip ru add iif vrf-blue table 10

# 步骤3：设置 VRF 默认不可达路由（防止路由泄漏）
ip route add table 10 unreachable default metric 4278198272

# 步骤4：将网络接口划入 VRF
ip link set dev eth1 master vrf-blue
# 接口的 connected/local 路由自动迁移到表 10

# 步骤5：在 VRF 路由表中添加路由
ip route add table 10 10.2.1.0/24 via 10.2.1.254 dev eth1

9.3 应用程序绑定 VRF

// 应用程序使用 setsockopt 绑定到 VRF 设备
setsockopt(sd, SOL_SOCKET, SO_BINDTODEVICE, dev, strlen(dev)+1);

// 或者允许默认 VRF 的 socket 跨 VRF 工作
# 允许默认 VRF 的 TCP/UDP 服务跨所有 VRF 域工作
sysctl -w net.ipv4.tcp_l3mdev_accept=1
sysctl -w net.ipv4.udp_l3mdev_accept=1

9.4 常用 VRF 查询命令

# 显示所有 VRF 设备
ip -d link show type vrf

# 显示 VRF red 的路由表
ip route show vrf red

# 在 VRF 中测试路由
ip route get 10.2.1.40 vrf red

# 显示 VRF 的邻居表
ip neigh show vrf red

# 将接口从 VRF 中移除
ip link set dev eth1 nomaster

VRF 路由隔离示意（来源：docs.kernel.org/networking/vrf.html）

10. iproute2 工具集

🔧导读：iproute2 是操作 Linux 路由子系统的现代标准工具集，替代了已废弃的 net-tools（ifconfig、route 等）。本节介绍 iproute2 的核心命令及其与旧工具的对应关系。

10.1 net-tools vs iproute2 对照

net-tools（旧）iproute2（新）说明
ifconfigip addr显示/配置网络接口
routeip route显示/操作路由表
arpip neigh显示/操作 ARP 缓存
netstat -rnip route show数字格式显示路由表
netstat -tulnss -tuln显示监听 socket

10.2 ip route — 路由表操作

# 显示主路由表
ip route show

# 显示所有表中的所有路由
ip route show table all

# 显示特定目的地的路由决策（最实用的调试命令）
ip route get 8.8.8.8
# 输出示例：8.8.8.8 via 192.168.1.1 dev eth0 src 192.168.1.100

# 指定源地址查询路由
ip route get 8.8.8.8 from 192.168.1.10

# 添加路由
ip route add 10.0.0.0/8 via 192.168.1.254 dev eth0

# 添加 blackhole 路由（静默丢弃）
ip route add blackhole 10.100.0.0/16

# 删除路由
ip route del 10.0.0.0/8

# 实时监控路由表变化
ip monitor route

10.3 ip rule — 策略路由规则

# 显示当前所有规则
ip rule show

# 添加规则：来自 10.0.0.0/24 的流量查表 100
ip rule add from 10.0.0.0/24 table 100

# 添加规则：带防火墙标记 0x64 的包查表 200
ip rule add fwmark 0x64 table 200

# 删除规则
ip rule del from 10.0.0.0/24 table 100

10.4 ss — socket 统计

# 显示所有 socket（TCP + UDP，含监听，数字格式）
ss -tunapl

# 显示 TCP 内部统计（RTT、拥塞窗口等）
ss -ti

# 查看指定端口
ss -tunapl | grep :80

11.3 策略路由调试流程

ip rule show

# 2. 检查指定路由表内容
ip route show table 100

# 4. 如果用了 fwmark，检查 iptables mangle 规则

tcpdump -ni eth0 host 10.0.0.1
tcpdump -ni eth1 host 10.0.0.1

SUMMARY

12. 第一性原理总结

⚛️第一性原理要求我们回到最基本的事实，从根本公理出发推导结论，而不是类比或经验。本章试图将 Linux 路由子系统还原到最本质的问题：为什么它是这个样子的？

12.1 根本问题只有一个

「这个比特序列，应该从哪根线路出去？」

路由的本质是一个映射函数：f(packet) → (output_interface, next_hop_mac)。所有复杂的数据结构、算法、规则系统，都是为了高效、灵活地计算这个函数的结果。

12.2 为什么需要 FIB？

互联网有约 90 万条路由前缀（BGP 全表）。朴素的线性扫描时间复杂度为 O(n)，每秒数百万包时延不可接受。

第一性推导：IP 地址是 32 位整数，最长前缀匹配等价于在二叉树中查找。LC-Trie 通过"路径压缩"去掉单链节点，通过"层压缩"让内部节点一次索引多位，将查找深度从 32 压缩到平均 4~7 层，实现接近 O(1) 的查找。这不是魔法，是对数据结构和查找目标的精确匹配。

12.3 为什么需要多路由表与策略路由？

第一性推导：路由决策的输入不只是目的地址。现实中的约束有：

  • 同一目的地，不同运营商的流量需走不同出口（多 ISP）
  • 安全策略要求某源 IP 的流量必须经过特定网关审计
  • 多租户环境中不同客户的路由表必须严格隔离

最小化的解法是：规则（Rules）→ 表（Tables）→ 路由。规则是函数选择器，表是子函数，RPDB 是函数组合器。ECMP 和 VRF 是这个框架的自然延伸。

12.4 为什么路由缓存被移除？

第一性推导：缓存的价值 = 命中率 × 节省的计算量。当 LC-Trie 查找已经足够快（纳秒级），而缓存本身带来内存压力、并发锁竞争、以及可被随机目的地址 DoS 攻击的风险时，缓存的"净价值"变为负数。Linux 3.6 移除全局路由缓存，是对"成本 vs 收益"的理性权衡，而非妥协。

12.5 为什么需要邻居子系统独立存在？

第一性推导：路由（L3）和地址解析（L2.5）是两个不同抽象层的职责。路由解决"去哪里"，邻居子系统解决"怎么到达"。分离这两层让路由可以复用于 IPv4、IPv6、MPLS，ARP 可以独立演化为 NDP（IPv6）或 InfiniBand 的等价协议，符合关注点分离（Separation of Concerns）原则。

12.6 设计哲学的收敛

第一性原理视角下的路由子系统各模块 —— 所有复杂性都服务于"映射函数"这一核心

12.7 终极结论

Linux 路由子系统的每一个设计决策，都可以从以下三条公理推导：

  1. 公理1：路由是映射。 输入是包的属性，输出是出口和下一跳。所有机制都是为了准确、快速地计算这个映射。
  2. 公理2：速度来自预计算。 LC-Trie 是对 IP 前缀空间的预计算索引，邻居表是对 MAC 的缓存，sk_dst_cache 是对 per-flow 决策的缓存。
  3. 公理3：灵活性来自抽象层次。 规则、表、路由三层解耦，让同一框架可以支持简单默认路由、复杂策略路由、VRF 隔离和 ECMP 负载均衡，而不需要特例处理。
