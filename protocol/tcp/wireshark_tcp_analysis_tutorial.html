0TCP 协议概述与报文结构

📄 原文来源：RFC 793 – Transmission Control Protocol（1981年9月，IETF）

TCP（传输控制协议）是一种面向连接的、端到端的可靠协议，设计目标是在数据包交换网络中提供可靠的进程间通信。RFC 793 开篇定义：

RFC 793 原文："TCP is a connection-oriented, end-to-end reliable protocol designed to fit into a layered hierarchy of protocols which support multi-network applications."

（译）TCP 是一种面向连接的端到端可靠协议，设计为分层协议体系结构的一部分，支持多网络应用。

TCP 的核心能力包括六个方面（RFC 793，Section 1.5）：基本数据传输、可靠性、流量控制、多路复用、连接管理、优先级与安全。

TCP 报文头格式（RFC 793，Section 3.1）

TCP 段头部最小为 20 字节（无选项），最大 60 字节：

各字段含义（RFC 793 原文定义）

字段位宽RFC 793 原文说明
Source Port16 bits源端口号，标识发送进程
Destination Port16 bits目的端口号，标识接收进程
Sequence Number32 bits本段首字节在数据流中的位置；SYN 存在时为初始序列号 ISN，首个数据字节为 ISN+1
Acknowledgment Number32 bitsACK 置位时有效；表示期望收到的下一个字节序列号，同时确认之前所有字节
Data Offset4 bitsTCP 头部长度（以 32 位字为单位），指示数据起始位置
Reserved6 bits保留，必须为 0
URG / ACK / PSH / RST / SYN / FIN各 1 bit6 个控制位，管理连接状态与数据处理
Window16 bits接收方愿意接受的从确认号起的字节数（流量控制）
Checksum16 bits头部与数据的 16 位反码校验和，含 96 位伪头部
Urgent Pointer16 bitsURG 置位时有效，指向紧急数据末尾之后一个字节的序列号
Options可变可选字段，8 位对齐，含 MSS、时间戳、窗口缩放等

1TCP 状态机

📄 原文来源：RFC 793，Section 3.2 – Terminology & State Machine

RFC 793 定义了 TCP 连接的完整状态机。连接的状态记录在 TCB（传输控制块，Transmission Control Block）中。共有 11 个状态：

各状态说明

状态含义（RFC 793）持方
CLOSED无连接存在，TCB 已删除两端
LISTEN等待来自任意远端的连接请求（被动打开）服务器
SYN-SENT已发送 SYN，等待对端确认客户端

2三次握手建立连接

📄 原文来源：RFC 793，Section 3.4 – Establishing a Connection

RFC 793 对三次握手的目的做了明确说明：

RFC 793 原文："The principle reason for the three-way handshake is to prevent old duplicate connection initiations from causing confusion."

（译）三次握手的主要原因是防止旧的重复连接发起报文导致混乱。

关键要点解析

  • 1
    第一次握手（SYN）：客户端选择初始序列号 ISN=x，发送 SYN 段。SYN 消耗 1 个序列号（RFC 793：SYN occupies one sequence number）。状态变为 SYN-SENT。
  • 2
    第二次握手（SYN-ACK）：服务器选择 ISN=y，回复 SYN-ACK，ACK=x+1（期望下一个字节），同时携带自己的 SYN。状态变为 SYN-RECEIVED。
  • 3
    第三次握手（ACK）：客户端发送 ACK=y+1，确认服务器 SYN。RFC 793 明确说明："ACK does not occupy sequence number space"（ACK 不占用序列号空间）。两端进入 ESTABLISHED。
Wireshark 中的相对序列号：Wireshark 默认显示相对序列号（0 开始），实际抓包中真实 ISN 是随机选取的 32 位数。这是防止历史报文干扰的安全机制（RFC 793，Section 3.3）。
## Wireshark 过滤三次握手 tcp.flags == 0x002 # 第1步：仅 SYN tcp.flags == 0x012 # 第2步：SYN+ACK tcp.flags == 0x010 # 第3步：仅 ACK（握手完成）

RFC 793 Glossary 原文："FIN: A control bit (finis) occupying one sequence number, which indicates that the sender will send no more data or control occupying sequence space."

TIME-WAIT 存在的原因

RFC 793 规定 TIME-WAIT 持续 2×MSL（MSL = 2 分钟，最大报文生命期）：
1. 确保最后一个 ACK 能到达对端（若丢失，对端会重传 FIN，TIME-WAIT 可以重新回应）
2. 让网络中可能残留的旧连接报文消散，防止它们干扰新连接（避免序列号混淆）

4RST 复位重置

📄 原文来源：RFC 793，Section 3.4 & Glossary – RST Reset
RFC 793 Glossary 原文："RST: A control bit (reset), occupying no sequence space, indicating that the receiver should delete the connection without further interaction."

（译）RST 是一个不占序列号空间的控制位，指示接收方应立即删除连接，无需进一步交互。

5序列号与确认号

📄 原文来源：RFC 793，Section 3.3 – Sequence Numbers
RFC 793 原文（Section 3.3）："The TCP must recover from data that is damaged, lost, duplicated, or delivered out of order by the internet communication system. This is achieved by assigning a sequence number to each octet transmitted, and requiring a positive acknowledgment (ACK) from the receiving TCP."

（译）TCP 必须从互联网通信系统可能造成的数据损坏、丢失、重复或乱序中恢复。这通过为每个发送字节分配序列号并要求接收方发送确认应答来实现。

6流量控制与滑动窗口

📄 原文来源：RFC 793，Section 3.7 – Data Communication & Flow Control
RFC 793 原文（Section 1.5 Flow Control）："TCP provides a means for the receiver to govern the amount of data sent by the sender. This is achieved by returning a 'window' with every ACK indicating a range of acceptable sequence numbers beyond the last segment successfully received. The window indicates an allowed number of octets that the sender may transmit before receiving further permission."

（译）TCP 为接收方提供控制发送方数据量的手段。通过在每个 ACK 中携带"窗口"值，指示最后一个成功接收字节之后可接受的序列号范围。窗口值表示发送方在获得进一步许可前可以发送的最大字节数。

7超时重传机制

📄 原文来源：RFC 793，Section 3.7 & Section 3.9（RETRANSMISSION TIMEOUT）
RFC 793 原文："If the ACK is not received within a timeout interval, the data is retransmitted."

（译）如果在超时间隔内未收到确认，数据将被重传。
RFC 793 Section 3.9 原文："RETRANSMISSION TIMEOUT: For any state if the retransmission timeout expires on a segment in the retransmission queue, send the segment at the front of the retransmission queue again, reinitialize the retransmission timer, and return."

（译）对于任何状态，如果重传队列中某段报文的重传计时器到期，则重新发送重传队列队首的报文，重新初始化重传计时器，并返回。

RTO 计算原理

TCP 使用自适应的重传超时（RTO，Retransmission Timeout），基于 RTT（往返时间）测量动态调整：

## RTO 计算（RFC 6298 建议算法） # RTT 测量 SRTT = (1 - α) × SRTT + α × RTTsample # α = 1/8 RTTVAR = (1 - β) × RTTVAR + β × |SRTT - RTT| # β = 1/4 RTO = SRTT + max(G, 4 × RTTVAR) # G=时钟粒度，最小1秒 # 超时后指数退避 RTO_new = RTO × 2 # Karn算法：重传时不更新RTT样本

8快速重传与快速恢复

📄 原文来源：RFC 5681，Section 3.2 – Fast Retransmit/Fast Recovery
RFC 5681 原文（Section 3.2）："The TCP sender SHOULD use the 'fast retransmit' algorithm to detect and repair loss, based on incoming duplicate ACKs. The fast retransmit algorithm uses the arrival of 3 duplicate ACKs (as defined in section 2, without any intervening ACKs which move SND.UNA) as an indication that a segment has been lost. After receiving 3 duplicate ACKs, TCP performs a retransmission of what appears to be the missing segment, without waiting for the retransmission timer to expire."

（译）TCP 发送方应使用"快速重传"算法基于重复 ACK 检测和修复丢包。该算法将收到 3 个重复 ACK（没有任何推进 SND.UNA 的 ACK 间隔）视为报文段丢失的信号。收到 3 个重复 ACK 后，TCP 立即重传看似丢失的报文，无需等待重传计时器超时。

快速重传流程

快速恢复（Fast Recovery）

RFC 5681 原文："After the fast retransmit algorithm sends what appears to be the missing segment, the 'fast recovery' algorithm governs the transmission of new data until a non-duplicate ACK arrives. The reason for not performing slow start is that the receipt of the duplicate ACKs not only indicates that a segment has been lost, but also that segments are most likely leaving the network."

（译）快速重传后，"快速恢复"算法管理新数据的发送，直到收到非重复 ACK。不执行慢启动的原因是：收到重复 ACK 不仅说明丢包，也说明网络中仍有数据在传输（接收方收到了后续数据），因此无需降至 1 MSS。

9慢启动与拥塞避免

📄 原文来源：RFC 5681 – TCP Congestion Control（2009年9月，IETF）
RFC 5681 原文（Section 3.1）："The slow start and congestion avoidance algorithms MUST be used by a TCP sender to control the amount of outstanding data being injected into the network. To implement these algorithms, two variables are added per TCP connection: a congestion window (cwnd) and a slow start threshold (ssthresh)."

（译）TCP 发送方必须使用慢启动和拥塞避免算法来控制注入网络的未确认数据量。为此，每个 TCP 连接增加两个变量：拥塞窗口（cwnd）和慢启动阈值（ssthresh）。

10半开连接异常场景

📄 原文来源：RFC 793，Section 3.4 – Half-Open Connections and Other Anomalies
RFC 793 原文："An established connection is said to be 'half-open' if one of the TCPs has closed or aborted the connection at its end without the knowledge of the other, or if the two ends of the connection have become desynchronized owing to a crash that resulted in loss of memory. Such connections will automatically become reset if an attempt is made to send data in either direction."

（译）如果某一端在对方不知情的情况下关闭或中止了连接，或因崩溃导致内存丢失使两端失去同步，则该连接被称为"半开连接"。若任意一方尝试发送数据，这类连接将自动被重置。

11乱序与重复报文

📄 原文来源：RFC 793，Section 3.3 & RFC 5681，Section 3.2

12第一性原理总结

📄 综合来源：RFC 793（1981）& RFC 5681（2009），IETF

💡 第一性原理：TCP 的本质问题与解法

根本问题：如何在一个不可靠、无序、有损的网络（IP 层）之上，为应用层提供可靠的、有序的、全双工的字节流通信？

答案：TCP 通过以下五个核心机制的组合来解决这个问题——

机制一
序列号 + ACK
每字节编号，确认到达，超时重传
机制二
滑动窗口
接收方通告容量，防止发送过快
机制三
三次握手/四次挥手
同步状态，防止旧报文干扰
机制四
RST 复位
快速清除异常半开或失效连接
机制五

🎯 一句话总结

TCP 的一切设计都源自同一个第一原理：在不可靠的网络上，通过"编号 + 确认 + 重传"三要素建立可靠性，再用"窗口"控制速率（流量控制 + 拥塞控制），用"握手/挥手"管理状态，用"RST"处理异常——所有机制协同工作，共同守护端到端的有序字节流。

Wireshark 让这些协议细节从抽象变为可见：每一个报文的标志位、序列号、窗口值，都是 RFC 793 / RFC 5681 规范在真实网络中的具体体现。
