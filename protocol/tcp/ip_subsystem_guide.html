§1 基本概念
导读：本章建立整体认知框架。Linux 内核网络栈专注于 OSI 第 2~4 层（数据链路、网络、传输），IP 层作为第 3 层是连接硬件驱动与传输协议的枢纽。理解数据包的生命周期、协议族注册机制，是后续深入内核源码的前提。

Linux 网络栈分层模型

Linux 内核网络子系统仅负责 OSI 的 L2（数据链路层）、L3（网络层）、L4（传输层）。更高层由用户空间实现，L1（物理层）由各 NIC 驱动自行处理。

图 1-1：Linux 网络栈分层示意（IP 层为本文核心）

NAPI（New API）机制

Linux 2.4 之前每收到一个数据包就触发一次中断，高流量时 CPU 开销极大。NAPI 将中断模式改为混合模式：NIC 收到首包时触发中断，随后禁用中断转为轮询，批量处理后再重新启用。这大幅减少中断次数，提升吞吐量。

关键结构：每个 NIC 由 net_device 结构体表示，包含 IRQ 编号、MTU、MAC 地址、设备名（如 eth0）、flags、混杂计数器（promiscuity counter）、GRO/GSO offload 能力等核心字段。

数据包的内核生命周期

参考来源：Linux IP Networking: A Guide to the Implementation — Glenn Herrin, UNH TR 00-04
  • 接收路径： NIC → DMA 写入 Ring Buffer → 硬中断 → NAPI 软中断 → __netif_receive_skb → IP 层 → 传输层 → Socket
  • 发送路径： 应用 write() → Socket → 传输层 → IP 层封装头部 → Netfilter → 队列规则(qdisc) → NIC 驱动 → 物理发送
  • 转发路径： NIC 接收 → IP 层检查目的地址 → 查路由表 → ip_forward() → Netfilter FORWARD → 出口 NIC
§2 IP 报头结构
导读：IPv4 报头是 IP 层所有决策的信息来源——路由依赖目的地址，TTL 防止路由环路，Protocol 字段决定上交哪个传输层，Flags/Fragment Offset 控制分片重组。搞清楚每个字段的语义，才能理解内核为何在特定位置丢包。
图 2-1：IPv4 报头结构（RFC 791）

关键字段说明

字段位宽内核决策意义
TTL8 bits每次转发减 1；为 0 时内核调用 icmp_send(ICMP_TIME_EXCEEDED) 并丢弃
Protocol8 bits决定解复用目标：6=TCP, 17=UDP, 1=ICMP, 50=ESP, 47=GRE
Flags.DF1 bit禁止分片（Don't Fragment）；超 MTU 时内核返回 ICMP Frag Needed
Flags.MF1 bit更多分片（More Fragments）；最后一片为 0
Frag Offset13 bits当前分片在原始数据报中的偏移，单位 8 字节，用于重组
TOS8 bits影响 pfifo_fast 队列 band（映射到 Linux SKB 优先级）
参考来源：kernel.org — IP Sysctl Documentation
§3 sk_buff 核心数据结构
导读：sk_buff（SKB）是 Linux 网络栈的"血液"——所有协议层共享同一块内存，通过移动 header 指针而非拷贝数据来实现零拷贝协议栈。理解 SKB 的内存布局，是理解为何某些操作会导致丢包（如内存不足）的关键。
图 3-1：sk_buff 结构体与内存布局（各协议层共享零拷贝）

关键操作

/* IP 层获取头部指针的典型操作 */
struct iphdr *iph = ip_hdr(skb);     // 获取 IP 头指针
skb_pull(skb, sizeof(struct iphdr)); // data 指针后移（去头）
skb_push(skb, sizeof(struct iphdr)); // data 指针前移（加头）
skb_reserve(skb, NET_SKB_PAD);      // 发送时预留 headroom

/* 路由缓存绑定 */
skb_dst_set(skb, dst);               // 绑定路由表项
dst = skb_dst(skb);                  // 获取路由表项
dst_input(skb);                      // 调用 dst->input (ip_local_deliver/ip_forward)
丢包风险：当系统内存紧张，alloc_skb() 或 dev_alloc_skb() 返回 NULL，内核直接丢弃该包，计数器 IpInDiscards / IpOutDiscards 递增。
§4 收包路径：ip_rcv
导读：ip_rcv() 是 IP 层的统一入口，负责报头合法性验证、Netfilter PRE_ROUTING 钩子、路由决策（本机/转发），最终交给 ip_local_deliver() 或 ip_forward()。每一个验证失败都是一个潜在丢包点。
图 4-1：ip_rcv() 收包完整路径与丢包点

关键源码逻辑（net/ipv4/ip_input.c）

/* 简化版 ip_rcv 流程 — 来自内核源码注释 */
int ip_rcv(struct sk_buff *skb, ...) {
    /* 1. 更新计数器 */
    __IP_UPD_PO_STATS(net, IPSTATS_MIB_IN, skb->len);  // IpInReceives++

    /* 2. 目的 MAC 地址检查：非本机且非混杂模式则丢弃 */
    if (skb->pkt_type == PACKET_OTHERHOST)
        goto drop;

    /* 3. 报头合法性校验：版本、长度、checksum */
    iph = ip_hdr(skb);
    if (iph->ihl < 5 || iph->version != 4)  // IpInHdrErrors++
        goto inhdr_error;

    /* 4. 进入 Netfilter PRE_ROUTING */
    return NF_HOOK(NFPROTO_IPV4, NF_INET_PRE_ROUTING, ..., ip_rcv_finish);
}

int ip_rcv_finish(struct sk_buff *skb, ...) {
    /* 5. 路由查找 */
    if (!skb_valid_dst(skb)) {
        err = ip_route_input_noref(skb, iph->daddr, iph->saddr, ...);
        if (err) goto drop_error;  // IpInAddrErrors / IpExtInNoRoutes
    }
    /* 6. dst_input 分发：ip_local_deliver 或 ip_forward */
    return dst_input(skb);
}

§6 Netfilter 钩子
导读：Netfilter 是 Linux 内核的包过滤框架，在 IP 层设置了 5 个 hook 点，iptables/nftables 规则就挂在这些点上。任何一个 hook 返回 NF_DROP，数据包就会被无声地丢弃。这是用户态最常配置、也最容易造成丢包的位置。

iptables 与 nftables 规则调试

# 查看现有 iptables 规则（含计数器）
iptables -nvL --line-numbers

# 在 INPUT 链插入日志规则（调试丢包）
iptables -I INPUT -j LOG --log-prefix "IP_DROP_DEBUG: " --log-level 4

# nftables 追踪特定流量（替代 iptables LOG）

conntrack -L -p tcp --dport 80

§7 发包路径：ip_output
导读：发包路径从传输层的 ip_queue_xmit() 开始，到 ip_output() 完成 IP 头组装和 MTU 检查后交给设备驱动。发包路径也有多个丢包点，尤其是路由查找失败和 MTU 超出时的分片处理。
图 7-1：IP 发包路径与 MTU 处理分支
/* ip_output 核心逻辑 — net/ipv4/ip_output.c */
int ip_output(struct net *net, struct sock *sk, struct sk_buff *skb) {
    struct net_device *dev = skb_dst(skb)->dev;

    /* 设置协议字段，更新统计 */
    skb->dev = dev;
    skb->protocol = htons(ETH_P_IP);

    /* 进入 POST_ROUTING 钩子 */
    return NF_HOOK_COND(NFPROTO_IPV4, NF_INET_POST_ROUTING,
                        net, sk, skb, NULL, dev,
                        ip_finish_output,
                        !(IPCB(skb)->flags & IPSKB_REROUTED));
}

int ip_finish_output(struct net *net, struct sock *sk, struct sk_buff *skb) {
    unsigned int mtu = ip_skb_dst_mtu(sk, skb);

    /* GSO 处理 */
    if (skb_is_gso(skb))
        return ip_finish_output_gso(net, sk, skb, mtu);

    /* MTU 超限则分片 */
    if (skb->len > mtu || IPCB(skb)->frag_max_size)
        return ip_fragment(net, sk, skb, mtu, ip_finish_output2);

    return ip_finish_output2(net, sk, skb);
}
§8 IP 分片与重组
导读：IP 分片是 IP 层处理大包的机制，但分片重组失败会造成静默丢包（对应用层看起来像超时）。Linux 通过内存限额和超时来防止分片队列耗尽资源，这些参数都可通过 sysctl 调整。

关键 sysctl 参数（来源：kernel.org ip-sysctl）

参数默认值说明

§10 SNMP/nstat 计数器
导读：Linux 内核通过 /proc/net/snmp 和 /proc/net/netstat 暴露数百个精细计数器。这些计数器是诊断 IP 层问题的"听诊器"——不需要重现问题，只需对比前后差值即可定位。
计数器名递增位置含义与排查建议
IpInReceives ip_rcv() 入口 IP 层收到包总数（含之后被丢弃的）
IpInHdrErrors 报头校验失败 版本/长度/校验和错误；可能是硬件/驱动 bug 或链路问题
IpInAddrErrors 地址无效或转发被禁 目的 IP 无效，或 IP forwarding 未开启却收到转发包
IpExtInNoRoutes ip_route_input_slow() 路由表中找不到目的地路由；检查 ip route
IpInUnknownProtos 解复用失败 IP Protocol 字段指向内核不认识的协议
IpInDiscards 内存不足等内部错误 收包路径内核内部资源耗尽
IpInDelivers ip_local_deliver() 成功交付上层协议的包数
IpOutNoRoutes 发包路由查找失败 本机发包找不到路由；检查路由表
IpReasmReqds 分片重组触发 收到需要重组的分片数
IpReasmFails 重组超时/内存 重组失败；检查 ipfrag_time 和 ipfrag_high_thresh
IpExtInNoECTPkts ECN 计数 无 ECN 标记的包数（基线参考）

nstat 使用示例

# 重置计数，复现问题，再查看增量
nstat -n                          # 重置基线
# ... 在此期间复现问题 ...
nstat -az | grep -v "0.0$"       # 只显示有变化的计数器

# 示例：IpExtInNoRoutes 增加 = 路由表缺少该目的地
$ nstat | grep NoRoute
IpExtInNoRoutes                 3          0.0

# 对应内核日志
$ dmesg | grep "martian\|net_ratelimit"

§11 IP 层丢包排查方法论
导读：这是本文最核心的实操章节。IP 层丢包排查遵循"由宏到微"的方法论：先用统计计数器缩小范围，再用抓包和路由工具验证假设，最后用内核追踪精确定位。本章提供一个完整的决策树。
图 11-1：IP 层丢包排查决策树

常见丢包场景与处置

场景 A：路由缺失（IpExtInNoRoutes / IpOutNoRoutes）

如 kernel.org SNMP 文档中的测试案例：删除默认路由后发包，服务器返回 ICMP Destination Unreachable，IpExtInNoRoutes 计数增加。 
# 验证路由
ip route get 8.8.8.8
# 如果返回 "RTNETLINK answers: Network is unreachable" → 路由缺失

# 修复：添加路由
ip route add 0.0.0.0/0 via 192.168.1.1 dev eth0   # 默认路由
ip route add 8.8.8.8/32 via 10.0.0.1               # 主机路由

场景 B：iptables 规则丢包

# 找出有 DROP 动作且计数器非零的规则
iptables -nvL | grep -E "DROP|REJECT" | awk '$1 > 0'

# 动态追踪：添加临时 LOG 规则（调试后删除）
iptables -I FORWARD 1 -j LOG --log-prefix "[FWD_DROP]: "
journalctl -k -f | grep FWD_DROP

场景 C：MTU/PMTUD 问题

# 测试 PMTUD（用 DF 位）
ping -M do -s 1472 目标IP    # 1472+28(IP+ICMP头)=1500
ping -M do -s 1400 目标IP

# 查看路径 MTU
ip route get 8.8.8.8
# 输出中 mtu 字段即路径 MTU

# 如怀疑 PMTUD black hole（防火墙丢弃 ICMP Frag Needed）
# 临时绕过：手动钳制 TCP MSS
iptables -A FORWARD -p tcp --tcp-flags SYN,RST SYN \
  -j TCPMSS --clamp-mss-to-pmtu

场景 D：内核精确定位（kfree_skb + reason）

# 使用 perf 抓取丢包 reason（Linux 5.17+ 支持 reason 字段）

perf script
# 输出示例：
# curl  883  340.79: skb:kfree_skb: ... location=tcp_v4_rcv+0x157 reason: NO_SOCKET

# 使用 dropwatch（需要 CONFIG_NET_DROP_MONITOR）

§12 sysctl 关键参数
导读：Linux 内核通过 /proc/sys/net/ipv4/ 暴露大量可调参数，这些参数直接影响 IP 层的行为——从 TTL 到分片内存，从 PMTUD 到反向路径过滤。错误的配置可能导致意想不到的丢包。
参数（来源：kernel.org ip-sysctl）默认值作用与调优建议
net.ipv4.ip_forward 0 路由器必须设为 1；改变此值会重置所有相关参数为 RFC 默认值
net.ipv4.ip_default_ttl 64 出站包的默认 TTL；RFC1700 推荐值 64
net.ipv4.ip_no_pmtu_disc 0 0=启用 PMTUD；2=强制忽略所有 PMTU 消息（适合 DNS 服务器防攻击）
net.ipv4.ipfrag_high_thresh 4MB 分片重组队列内存上限；高流量场景可适当调大
net.ipv4.ipfrag_time 30s 分片等待重组超时时间
net.ipv4.conf.all.rp_filter 2 反向路径过滤：0=关，1=严格，2=宽松；误配置会丢弃非对称路由的包
net.ipv4.conf.all.accept_source_route 0 是否接受源路由选项（安全考虑通常禁用）
net.core.netdev_max_backlog 1000 NIC 驱动送往协议栈的包队列长度；高吞吐场景增大（32768+）
net.ipv4.fib_multipath_hash_policy 0 多路径路由哈希策略：0=L3，1=L4；ECMP 场景用 L4 哈希更均衡

rp_filter 导致的丢包（典型案例）

# rp_filter=1（严格模式）：内核对每个入包验证反向路径
# 如果回程路由出口与入口 NIC 不一致 → 丢包（静默）
# 症状：单向通信正常，反向不通

# 诊断
sysctl net.ipv4.conf.eth0.rp_filter
cat /proc/net/snmp | grep IpInAddrErrors   # 可能增长

# 临时关闭验证
sysctl -w net.ipv4.conf.eth0.rp_filter=0

# 查看当前 rp_filter 配置
sysctl -a | grep rp_filter

    资源是有限的。 内存、CPU、队列都是稀缺资源。当数据包速率超过处理能力，系统必须丢弃多余的包。这不是 bug，而是物理约束的必然结果。Ring Buffer、backlog 队列、分片重组内存池，都是对这一约束的工程化应对。调优本质是重新分配有限资源。
  • 2
    数据包必须有明确的目的地。 IP 地址是逻辑坐标系，路由表是从逻辑坐标到物理接口的映射。没有映射（路由缺失）或映射到黑洞，包必然被丢弃。FIB Trie、ECMP、策略路由都是在解同一个问题：从哪条路把这个包送出去？
  • 3
    网络是不可信的，安全策略必须在内核执行。 Netfilter 是策略执行点，而非性能瓶颈。每条 iptables 规则都是一个决策节点，DROP 就是在该节点终止包的生命。凡是无法解释的丢包，先查防火墙。
  • 4
    物理链路有容量上限（MTU）。 超过 MTU 的包要么被分片（开销大），要么被丢弃并触发 ICMP 反馈。PMTUD 是动态发现这个上限的机制。当应用层超时且没有其他异常，考虑 MTU/PMTUD black hole。
  • 5
    数据在传输中可能被损坏，完整性必须验证。 TTL 防止路由环路，checksum 检测数据损坏，长度字段防止解析错误。任何验证失败，包必须被丢弃，否则错误会向上传播造成更大破坏。IpInHdrErrors 增长时，优先排查链路硬件质量。

调试的第一性原理：信息守恒

核心洞察：数据包不会"无缘无故"消失。内核丢弃每一个包时，都会在某个计数器中留下痕迹（kfree_skb tracepoint、SNMP 计数器、netfilter 规则计数器）。调试的本质是找到那个计数器，然后反推丢包原因。工具只是计数器的读取界面：nstat 读内核 MIB，perf 读 tracepoint，dropwatch 读 drop_monitor 内核基础设施。

最终总结：Linux IP 子系统是一台有限状态机，每个数据包从进入到离开，经历的每个决策节点都遵循"接受或拒绝"的二元逻辑。理解了 ip_rcv → 路由决策 → Netfilter → ip_output 这条主干路径，以及每个节点的"拒绝条件"，就掌握了 90% 的 IP 层问题排查能力。其余 10% 需要内核追踪工具（perf/dropwatch/bpftrace）来覆盖那些没有对应计数器的边缘 case。
