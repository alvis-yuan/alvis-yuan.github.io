Linux DNS 解析机制与缓存原理

以 glibc 为核心，系统梳理 Linux 下域名解析的完整流程与缓存体系

在 Linux 系统中，DNS 解析不是单一过程，而是由多个层次的组件协同完成。理解这一体系对于排查网络问题、优化服务性能至关重要。本文以 glibc（GNU C Library） 的实现为核心，深入解析各种域名解析方式的原理流程与缓存机制。

第 1 章 · 解析方式

NSS（Name Service Switch）框架

glibc 域名解析的核心调度引擎

Name Service Switch (NSS) 是 GNU C Library (glibc) 的一部分，是支撑 getaddrinfo(3) API 的核心框架，用于解析域名。NSS 允许系统数据库由独立的服务提供，管理员可以在 nsswitch.conf(5) 中配置查询顺序。

权威原文
"The Name Service Switch (NSS) facility is part of the GNU C Library (glibc) and backs the getaddrinfo(3) API, used to resolve domain names. NSS allows system databases to be provided by separate services, whose search order can be configured by the administrator in nsswitch.conf(5)."
— ArchWiki, Domain name resolution

NSS 工作原理

负责域名解析的数据库是 hosts 数据库。每次调用 getaddrinfo()，glibc 会按以下流程工作：

①
尝试连接 NSCD Unix Socket
getaddrinfo() 首先尝试连接 /var/run/nscd/socket 这个 Unix socket。这一步是固定写死（hardcoded）的，每次调用都会执行。若 NSCD 未运行，连接失败后继续后续步骤。
②
读取 /etc/nsswitch.conf
读取 NSS 配置文件，找到 hosts: 行，从左到右依次列出各个 NSS 模块（数据源）。
③
按序调用 NSS 模块
NSS 模块是以 /lib/libnss_<name>.so.2 命名的共享库，遵循 NSS Modules Interface。glibc 从左到右依次加载并调用每个模块，直到查询成功或所有模块用尽。
④
排序返回地址
依照 RFC 6724 的 10 条规则对返回的目标地址进行排序，优先选择最优路径。
权威原文（biriukov.dev）
"① The first step in the process might come as a surprise: getaddrinfo() attempts to connect to a hardcoded path of a Unix socket each time it is called. ② The next step involves reading the Name Service Switch configuration file, /etc/nsswitch.conf. ③ NSS modules are shared libraries that adhere to the NSS Modules Interface and are named accordingly to a template. ④ The final step is sorting destination addresses according to ten rules from RFC 6724."
— biriukov.dev §4.1 Internals and design

nsswitch.conf 配置格式

# /etc/nsswitch.conf (节选)
# hosts 行控制主机名解析的顺序
hosts:      files  dns  myhostname

# 错误处理动作示例：
# [NOTFOUND=return]  当前模块返回未找到时，立即终止（不继续下一个模块）
# [SUCCESS=return]   成功时终止
hosts:      mymachines  resolve [!UNAVAIL=return]  files  dns  myhostname

glibc 内置的 NSS 模块（hosts 数据库）

模块名对应 .so 文件数据来源说明
fileslibnss_files.so.2/etc/hosts读取静态本地主机映射文件

图 2 — NSS 模块按序查询流程（左向右，NOTFOUND 则继续，SUCCESS 则返回）

🧩 本章总结

  • NSS 是 glibc 的调度中间层，通过 /etc/nsswitch.conf 配置查询顺序，实现多数据源的统一调度。
  • 从第一性原理看：NSS 的本质是"策略与机制分离"——策略（顺序）由配置文件决定，机制（查询方式）由各模块（.so 动态库）实现，系统因此具备极高的可扩展性。
  • getaddrinfo() 每次调用都会重新读取 nsswitch.conf，无内置配置缓存（但 /etc/hosts 变更可由 NSCD 监测）。
  • NSCD Unix socket 尝试是 getaddrinfo() 内固定行为，即使不使用 NSCD 也会执行此连接尝试（约耗时 0.00005 秒）。
第 2 章 · 解析方式

/etc/hosts 静态解析（nss-files）

最古老、优先级最高的本地域名映射方式

/etc/hosts 是操作系统用于将主机名映射到 IP 地址的纯文本文件。通过 NSS 的 files 模块（libnss_files.so.2）提供服务。

权威原文
"The well-known /etc/hosts (man 5 hosts) is a plain text file used by operating systems to map hostnames to IP addresses."
— biriukov.dev §4.6 /etc/hosts

文件格式

# /etc/hosts 格式：
# IP地址  规范主机名  [别名...]
127.0.0.1   localhost
127.0.1.1   myhost.example.com   myhost
::1         localhost   ip6-localhost   ip6-loopback
192.168.1.10  internal-server   server
注意：规范主机名
行中第一个主机名是 canonical_hostname（规范主机名），这会包含在 getaddrinfo() 的返回结果中；其余的是别名（aliases）。
— biriukov.dev §4.6

🧩 本章总结

  • /etc/hosts 是最基础的本地 DNS 覆盖机制，优先级由 nsswitch.conf 配置，通常最高。
  • 从第一性原理看：它是最简单的"键值对映射"——以文本文件存储 hostname→IP 的静态绑定，无需网络，无需守护进程，是 DNS 存在之前的历史遗留，至今仍不可或缺（尤其在容器、开发环境中）。
  • 无缓存开销：每次 nss-files 查询都直接读文件，依赖操作系统的 Page Cache 加速。
第 3 章 · 解析方式

glibc Stub Resolver（存根解析器）

glibc 的核心 DNS 查询引擎，基于 /etc/resolv.conf 配置

glibc resolver（存根解析器）是一组低级别的 DNS 查询函数库，构成 nss-dns 模块的内核。它通过 /etc/resolv.conf 获取 DNS 服务器地址和配置，向上游递归 DNS 服务器发出 UDP/TCP 查询。

权威原文（ArchWiki）
"The glibc resolver reads /etc/resolv.conf for every resolution to determine the nameservers and options to use."
"The glibc resolver does not cache queries."
— ArchWiki, Glibc resolver

核心 API：getaddrinfo()

/* POSIX 标准接口，glibc 实现 */
int getaddrinfo(
    const char *node,          /* 主机名或 IP 字符串 */
    const char *service,       /* 服务名或端口字符串 */
    const struct addrinfo *hints, /* 查询条件（协议族、socket类型等）*/
    struct addrinfo **res      /* 返回：addrinfo 链表 */
);
⚠️ 线程安全警告
getaddrinfo() 内部调用 getenv()（用于读取 LOCALDOMAIN、RES_OPTIONS 环境变量），而 setenv() 是 MT-Unsafe 的。若多线程中同时调用 setenv() 和 getaddrinfo()，可能导致段错误（segfault）。
— biriukov.dev §4.4 Thread safety issues

底层 Resolver API

glibc resolver 库（libresolv）提供了更底层的 DNS 函数，nss-dns 模块在其上构建：

函数说明
res_ninit()初始化 resolver 状态（线程安全版本），读取 /etc/resolv.conf
res_nquery()向 nameserver 发起单个 DNS 查询，等待响应
res_nsearch()在 res_nquery 基础上，实现 search/ndots 搜索域逻辑
res_nsend()发送预构造的 DNS 消息，返回响应
res_nmkquery()构造 DNS 查询消息
↗ man3 resolver — Linux manual page

🧩 本章总结

  • glibc stub resolver 是纯粹的"中转站"：自身不做递归解析，只是将查询转发给 /etc/resolv.conf 中配置的递归 DNS 服务器。
  • 从第一性原理看：stub resolver 的本质是 DNS 协议的最小客户端实现——构造 DNS 请求报文、通过 UDP（默认）发送、解析响应报文，不存储任何状态（无缓存）。
  • 关键限制：glibc resolver 无内置缓存（ArchWiki 明确指出），每次 getaddrinfo() 调用都会重新读取 resolv.conf 并发出 DNS 查询。
  • glibc 2.9+ 默认并行发出 A 和 AAAA 查询，可通过 single-request 选项改为串行，以兼容问题 DNS 服务器。
第 4 章 · 解析方式

nss-dns 模块详解

NSS DNS 模块的内部机制与 /etc/resolv.conf 配置项

nss-dns（libnss_dns.so.2）是 glibc 实现的 NSS hosts 模块，内部调用 glibc resolver 库（libresolv）进行 DNS 查询。它是在 nsswitch.conf 中写 dns 时实际加载的模块。

权威原文（ArchWiki）
"dns: the glibc resolver which reads /etc/resolv.conf, see resolv.conf(5)"
— ArchWiki, Domain name resolution

/etc/resolv.conf 关键配置项详解

# /etc/resolv.conf — glibc resolver 配置文件

# 指定 DNS 服务器（最多 3 个，按序尝试）
nameserver   8.8.8.8
nameserver   8.8.4.4

# 搜索域：查询少于 ndots 个点的名称时，追加此域名后重试
search       example.com

# ndots：触发搜索域追加的最小点数阈值，默认 1
# ndots:5 是 Kubernetes 中的常见配置
options      ndots:1

# 等待响应超时（秒），默认 5，上限 30
options      timeout:5

# 每个 nameserver 的重试次数，默认 2，上限 5
options      attempts:2

# 对 nameserver 列表做轮询（round-robin），分散负载
options      rotate

# glibc 2.9+: 并行发 A+AAAA；此选项改为串行（避免某些 DNS 兼容问题）
options      single-request

# 启用 EDNS0 扩展（RFC 2671），自 glibc 2.6
options      edns0

# glibc 2.36+: 抑制 AAAA 查询（仅保留 A 查询，用于诊断）
options      no-aaaa
↗ man5 resolv.conf — Linux manual page

ndots 与搜索域的工作原理

权威原文（biriukov.dev）
"Resolver queries having fewer than ndots dots (default is 1) in them will be attempted using each component of the search path in turn until a match is found. This means that if the domain name used in getaddrinfo() contains fewer than the specified ndots, the resolver will append the search domain from the list and attempt to resolve it first."
— biriukov.dev §4.5 /etc/resolv.conf

图 5 — search / ndots 触发搜索域的查询展开过程

nameserver 重试算法

权威原文（resolv.conf man page）
"The algorithm used is to try a name server, and if the query times out, try the next, until out of name servers, then repeat trying all the name servers until a maximum number of retries are made."
"glibc uses up to 3 nameserver entries; additional entries are ignored."
— man5 resolv.conf

🧩 本章总结

  • nss-dns 是 glibc resolver 的 NSS 适配器，将 DNS 查询能力暴露给 NSS 框架。
  • 从第一性原理看：resolv.conf 是 glibc resolver 的"运行时参数"——nameserver 决定"问谁"，search/ndots 决定"怎么问"，timeout/attempts 决定"失败容忍度"。
  • 注意 glibc 每次解析都重新读取 resolv.conf，这意味着配置文件的变化会立即对新的 DNS 查询生效，但也带来额外的文件 I/O。
  • glibc 2.9+ 默认并行 A+AAAA 查询，提升 IPv6 性能；但某些有缺陷的 DNS 服务器（如部分家用路由器）可能只回复一个，导致超时，可用 single-request 解决。
第 5 章 · 解析方式

nss-resolve 与 systemd-resolved

现代 Linux 的带缓存缓存存根解析器

systemd-resolved 是一个实现了缓存和 DNSSEC 验证的 DNS/DNSSEC 存根解析器，同时支持 LLMNR 和 MulticastDNS 协议。本地应用程序可通过三种接口访问它。

权威原文（man8 systemd-resolved）
"systemd-resolved is a system service that provides network name resolution to local applications. It implements a caching and validating DNS/DNSSEC stub resolver, as well as an LLMNR and MulticastDNS resolver and responder."
— man8 systemd-resolved.service

接口协议适用场景
D-Bus API D-Bus IPC 推荐接口，异步、功能完整（含 DNSSEC 验证状态），org.freedesktop.resolve1
Varlink API AF_UNIX socket /run/systemd/resolve/io.systemd.Resolve，无需 D-Bus broker 即可使用
glibc NSS (nss-resolve) Unix socket → D-Bus 兼容 getaddrinfo()，同步，通过 NSS 模块 libnss_resolve.so.2 接入
本地 DNS Stub DNS over UDP/TCP 监听 127.0.0.53:53（完整功能）和 127.0.0.54:53（代理模式）

🧩 本章总结

  • systemd-resolved 是现代 Linux 的"智能 DNS 中间层"，解决了 glibc resolver 无缓存、无状态、无加密的三大短板。
  • 从第一性原理看：systemd-resolved 将"DNS 解析"从一个无状态的函数调用升级为一个有状态的系统服务——维护服务器健康状态、缓存响应、支持多协议（DoT、LLMNR、mDNS），是解析能力的质变。
  • nss-resolve 模块是 glibc NSS 与 systemd-resolved 之间的桥梁，使 getaddrinfo() 调用透明地受益于 systemd-resolved 的缓存和功能。
  • multi-label 名称不追加 search 域（与 nss-dns 的重要区别），避免了因搜索域导致的额外无效查询。
第 6 章 · 缓存机制

glibc 无内置缓存

理解 glibc resolver 缓存缺失的本质与影响

glibc resolver 自身不缓存任何 DNS 查询结果。每次调用 getaddrinfo()，如果需要通过 nss-dns 解析，都会重新读取 /etc/resolv.conf，并向 DNS 服务器发起新的网络查询。

权威原文（ArchWiki）
"The glibc resolver does not cache queries. To improve query lookup time you can set up a caching resolver."
— ArchWiki, Glibc resolver

为什么 glibc 不缓存？

从设计哲学上看，glibc 的 NSS 框架遵循"机制与策略分离"原则。glibc 只负责"如何查询"（机制），不负责"何时缓存"（策略）。缓存这一策略交由专门的守护进程（NSCD、systemd-resolved）或本地缓存 DNS 服务器（Unbound、dnsmasq）来实现。

🧩 本章总结

  • glibc resolver 无缓存是设计决策，不是缺陷——缓存职责由外部组件承担。
  • 从第一性原理看：缓存引入了一致性问题（stale data），glibc 选择把这个权衡交给明确的缓存层（NSCD/systemd-resolved）来管理，而不是在低层库中隐式决策。
  • 实际后果：高频 DNS 解析的程序（如微服务）在无外部缓存时，会对上游 DNS 服务器造成显著压力，并增加应用延迟。
第 7 章 · 缓存机制

NSCD（Name Service Cache Daemon）

glibc 配套的名称服务缓存守护进程

nscd 是一个守护进程，为最常见的名称服务请求（passwd、group、hosts、services、netgroup）提供缓存。它通过 Unix socket（/var/run/nscd/socket）与 glibc 的 getaddrinfo() 等函数交互。

权威原文（man8 nscd）
"nscd provides caching for accesses of the passwd(5), group(5), hosts(5) services(5) and netgroup databases through standard libc interfaces, such as getpwnam(3), getpwuid(3), getgrnam(3), getgrgid(3), gethostbyname(3), and others. There are two caches for each database: a positive one for items found, and a negative one for items not found. Each cache has a separate TTL (time-to-live) period for its data."
— man5 nscd.conf

NSCD 缓存架构

图 8 — NSCD 缓存架构：正向缓存与负向缓存

/etc/nscd.conf 关键配置

# /etc/nscd.conf (hosts 数据库配置)

# 启用 hosts 缓存
enable-cache            hosts   yes

# 正向缓存 TTL（成功查询，秒）
# 注：对于 DNS，nscd 会优先使用 DNS 响应中的 TTL 值，此配置被忽略
positive-time-to-live   hosts   3600

# 负向缓存 TTL（失败查询，秒）
negative-time-to-live   hosts   20

# 内部哈希表大小（推荐质数）
suggested-size          hosts   211

# 监测 /etc/hosts 和 /etc/resolv.conf 文件变化，自动刷新
check-files             hosts   yes

# 跨重启持久化缓存
persistent              hosts   yes

# 共享内存映射（客户端直接查缓存，无需经过 socket）
shared                  hosts   yes

# 数据库文件最大字节数
max-db-size             hosts   33554432
↗ man5 nscd.conf — Linux manual page

NSCD 缓存流程详解

正向缓存（Positive Cache）
存储查询成功的结果。TTL 较大（默认 3600 秒），提升缓存命中率。注意：man5 nscd.conf 明确指出，对于某些名称服务（包括特别是 DNS），nscd 会使用 DNS 响应中的 TTL 值而忽略 positive-time-to-live 配置。
— man5 nscd.conf: "Note that for some name services (including specifically DNS) the TTL returned from the name service is used and this attribute is ignored."
负向缓存（Negative Cache）
存储查询失败的结果（NXDOMAIN 等）。TTL 较小（默认 20 秒），避免因长时间缓存失败结果而导致实际上已存在的记录无法解析。
文件变化监测（check-files）
"The daemon will try to watch for changes in configuration files appropriate for each database (e.g., /etc/passwd for the passwd database or /etc/hosts and /etc/resolv.conf for the hosts database), and flush the cache when these are changed. However, this will happen only after a short delay (unless the inotify(7) mechanism is available and glibc 2.9 or later is available)..."
— man8 nscd

NSCD 的 shared 模式（共享内存）

当 shared hosts yes 时，NSCD 的缓存数据库通过内存映射（mmap）与客户端共享。客户端进程可直接在共享内存中查找缓存条目，无需通过 Unix socket 与 NSCD 通信，仅在缓存未命中时才向 NSCD 发送请求。

权威原文（man5 nscd.conf）
"The memory mapping of the nscd databases for service is shared with the clients so that they can directly search in them instead of having to ask the daemon over the socket each time a lookup is performed. The default is no. Note that a cache miss will still result in asking the daemon over the socket."
— man5 nscd.conf

NSCD 操作命令

# 使 hosts 缓存失效（强制清空）
$ nscd -i hosts

# 查看缓存统计信息
$ nscd -g

# 重启 NSCD 服务
$ systemctl restart nscd

# 关闭 NSCD
$ nscd -K
⚠️ NSCD 已知问题
"The UNIX socket at /var/run/nscd/socket is part of glibc and the Name Service Cache Daemon (nscd), which is known for its instability and frequent bugs. As a result, it is often avoided."
— biriukov.dev §4.3 NSCD

🧩 本章总结

  • NSCD 是 glibc 的配套缓存方案，通过 Unix socket 拦截 NSS 查询，避免重复的 DNS 网络请求。
  • 正向/负向双缓存设计：分别处理"命中"和"未命中"两种场景，各有独立 TTL，是 DNS 缓存设计的普遍模式。
  • shared 模式的性能优化：通过 mmap 共享内存让客户端直接读缓存，消除了 socket 通信的 IPC 开销，但代价是缓存命中率统计不准确。
  • 从第一性原理看：NSCD 是"进程外缓存代理"模式——将缓存逻辑与查询逻辑隔离到独立进程，通过 IPC 协作。这种模式的优势是缓存跨进程共享，劣势是引入进程通信开销和稳定性风险（NSCD 因历史上的不稳定性而常被禁用）。

systemd-resolved 内置 DNS 缓存

现代 Linux 的推荐缓存解决方案

systemd-resolved 内置基于 DNS TTL 的缓存机制，是当前 Ubuntu 17.10+、Fedora 33+ 等现代 Linux 发行版的默认 DNS 缓存方案。

权威原文（man8 systemd-resolved）
"systemd-resolved is a system service that provides network name resolution to local applications. It implements a caching and validating DNS/DNSSEC stub resolver..."
— man8 systemd-resolved.service

缓存自动失效时机

权威原文（man8 systemd-resolved）
"Upon reception of the SIGUSR2 process signal systemd-resolved will flush all caches it maintains. Note that it should normally not be necessary to request this explicitly – except for debugging purposes – as systemd-resolved flushes the caches automatically anyway any time the host's network configuration changes."
— man8 systemd-resolved, SIGNALS section

systemd-resolved 会在以下情况下自动刷新缓存：

  • DNS 响应的 TTL 到期
  • 主机网络配置发生变化（如切换网络接口）
  • 接收到 SIGUSR2 信号（等同于 resolvectl flush-caches）
  • 接收到 SIGHUP 信号（同时重载配置并关闭 TCP 连接）

🧩 本章总结

  • systemd-resolved 的缓存是 TTL-based：严格遵循 DNS 响应中的 TTL 值，保证缓存数据与 DNS 权威数据的一致性。
  • 自动感知网络变化：当网络接口切换时自动刷新缓存，避免旧缓存在新网络环境中造成错误解析。
  • 从第一性原理看：systemd-resolved 的缓存是"进程内缓存"模式——缓存在服务进程内存中，通过 IPC 为所有客户端提供服务。与 NSCD 相比，它的设计更现代、更稳定，且天然与网络状态管理集成。
  • Cache 统计监控：可通过 resolvectl statistics 实时查看缓存命中率，这是调优 DNS 性能的关键工具。
第 9 章 · 缓存机制

本地缓存 DNS 服务器

Unbound、dnsmasq 等独立缓存方案

除 NSCD 和 systemd-resolved 外，还可以部署专用的本地缓存 DNS 服务器。这类服务器监听本地端口（如 127.0.0.1:53），并在 /etc/resolv.conf 中将其配置为 nameserver，使 glibc resolver 的 DNS 查询先经过本地缓存。

权威原文（ArchWiki）
"The Glibc resolver does not cache queries. To implement local caching, use systemd-resolved or set up a local caching DNS server and use it as the name server by setting 127.0.0.1 and ::1 as the name servers in /etc/resolv.conf or in /etc/resolvconf.conf if using openresolv."
— ArchWiki, Resolver performance

集成方式

# unbound 示例配置（/etc/unbound/unbound.conf）
server:
    interface: 127.0.0.1
    cache-min-ttl: 60
    prefetch: yes   # 在 TTL 过期前预取

🧩 本章总结

  • 本地缓存 DNS 服务器是"外部缓存代理"模式：在 glibc resolver 和外部 DNS 之间插入一层，通过标准 DNS 协议（UDP 53）提供缓存，对 glibc 完全透明。
  • 适用场景：需要 DNSSEC 验证、DNS over TLS、细粒度缓存策略（如预取、持久化）时，本地 DNS 服务器比 NSCD/systemd-resolved 提供更强的控制能力。
  • 从第一性原理看：这是最"正交"的缓存方案——glibc 不需要任何改动，只需修改 nameserver 配置，缓存完全由独立进程管理，关注点分离最彻底。

第 12 章 · 总结

第一性原理总结

从根本原理理解 Linux DNS 解析体系

核心问题的本质

DNS 解析需要回答一个基本问题："给定一个人类可读的名称，如何找到对应的网络地址？" Linux 的答案是构建一个分层、可扩展的解析栈。

图 11 — Linux DNS 解析五层架构（从第一性原理推导）

六个核心设计原则

①
策略与机制分离
NSS 框架是"策略"（配置顺序），各 .so 模块是"机制"（具体查询方式）。这使得系统高度可扩展——任何人都可以编写新的 NSS 模块（如 biriukov.dev 示例中用 Rust 编写的自定义模块）。
②
缓存职责外包
glibc resolver 无内置缓存，这是有意为之。缓存引入了时效性（TTL）和一致性问题，由专门的外部组件（NSCD、systemd-resolved、Unbound）统一管理，各自在适当的层次处理这一权衡。
③
最小化 stub resolver
glibc resolver 是"存根"——只负责构造 DNS 请求并转发给递归 DNS 服务器，不做递归解析。这符合"单一职责"原则，将复杂的 DNS 递归留给专业服务器完成。
④
正负双缓存模式
无论是 NSCD 还是 systemd-resolved，均维护独立的正向缓存（成功查询）和负向缓存（失败查询），各有不同 TTL。负缓存防止对不存在域名的重复查询，但 TTL 较短以保证及时更新。
⑤
从无状态到有状态的进化
glibc nss-dns 每次查询从第一个 nameserver 开始（无状态）；systemd-resolved 维护服务器状态，持续使用同一服务器直到出错（有状态）。有状态设计减少了协议探测开销（feature negotiation），但要求各 DNS 服务器提供等价服务。
⑥
/etc/resolv.conf 的"最大公约数"角色
resolv.conf 不属于任何单一组件（glibc 原文：technically not part of glibc），但被所有 resolver 遵循。它是整个 DNS 解析栈的"公共合同"——无论上层策略如何演化（NSCD → systemd-resolved），这份合同始终有效。

🏁 全文总结

  • NSS (nsswitch.conf)：解析请求的调度引擎，决定查询顺序（hosts: files dns myhostname）。每次 getaddrinfo() 都会重新读取配置，首先尝试连接 NSCD socket。
  • /etc/hosts (nss-files)：最简单的静态键值映射，无缓存，通常最高优先级，实时生效。
  • glibc Stub Resolver (nss-dns)：DNS 协议客户端，每次查询重读 resolv.conf，无内置缓存，默认并行 A+AAAA 查询（glibc 2.9+）。
  • systemd-resolved (nss-resolve)：现代 Linux 的推荐方案，带 TTL-based 缓存、DNSSEC 验证、DoT 支持和 LLMNR/mDNS，通过 127.0.0.53 提供 DNS stub 接口。
  • NSCD：glibc 的传统配套缓存守护进程，双缓存（正/负向）设计，支持 mmap 共享内存，但历史上以不稳定著称。
  • 本地缓存 DNS 服务器（Unbound/dnsmasq）：最正交的缓存方案，通过标准 DNS 协议为 glibc 提供缓存，控制能力最强。
