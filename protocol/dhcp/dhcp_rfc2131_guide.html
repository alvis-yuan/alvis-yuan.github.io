DHCP 协议图解导读

基于 RFC 2131：在 BOOTP 之上扩展自动 IP 地址分配、租约管理与丰富配置选项，运行于 UDP/67–68。

1 · 协议简介

DHCP（Dynamic Host Configuration Protocol）为 TCP/IP 主机提供配置参数下发框架，基于 BOOTP（RFC 951）扩展，增加可重用网络地址的自动分配与更多选项（RFC 1533）。

设计目标（§1.6 摘要）

  • 客户端无需手工配置；网络管理员无需逐台配置
  • 不要求每个子网一台服务器 — 经 BOOTP Relay Agent 跨路由器工作
  • 与静态配置主机、现有 BOOTP 客户端互操作
  • 保证同一 IP 不会同时分配给多个 DHCP 客户端；支持租约（Lease）

DHCP 不负责 DNS 注册，也不用于配置路由器本身。

2 · 架构与 BOOTP 关系

Client · Relay Agent · DHCP Server

客户端视角下 DHCP 是 BOOTP 的扩展：op=BOOTREQUEST(1) / BOOTREPLY(2)。原 BOOTP 的 vendor extensions 字段在 DHCP 中称为 options。

字段字节说明
op11=BOOTREQUEST，2=BOOTREPLY
htype / hlen1+1硬件类型与长度（如以太网 1/6）
hops1客户端置 0；Relay 可递增

yiaddr4「你的」地址 — 服务器分配给客户端的 IP

giaddr4Relay Agent IP；非 0 时服务器经 Relay 回包
chaddr16客户端硬件地址

options变长Magic Cookie + TLV 选项列表
DHCP 引入 client identifier 选项，避免 chaddr 兼作标识与链路层地址的歧义；同一客户端在所有报文中须使用相同 client identifier。

4 · DHCP 消息类型（Table 2 · Option 53）

每条 DHCP 报文的 options 中必须包含 DHCP Message Type（Option 53）。

Option 53名称方向用途
1DHCPDISCOVERC→广播发现可用服务器
2DHCPOFFERS→C提供 IP 与配置参数
3DHCPREQUESTC→广播/单播选择 Offer / 确认地址 / 续租
4DHCPDECLINEC→S地址冲突（ARP 检测到已被占用）
5DHCPACKS→C确认配置与租约
6DHCPNAKS→C客户端地址无效或租约过期
7DHCPRELEASEC→S主动释放租约
8DHCPINFORMC→S已有 IP，仅请求其他配置参数

5 · DORA — 分配新网络地址（§3.1 · Figure 3）

典型四步交互：Discover → Offer → Request → Ack。

Figure 3 · 多服务器竞争与选定
  1. 客户端广播 DHCPDISCOVER（ciaddr=0，可带参数请求列表）
  2. 各服务器回复 DHCPOFFER（yiaddr 为提供的 IP，含 server identifier 选项）
  3. 客户端广播 DHCPREQUEST，选定一台（server identifier），requested IP = 所选 offer 的 yiaddr
  4. 被选服务器发送 DHCPACK 提交 binding；失败则 DHCPNAK
  5. 客户端 ARP 自检；正常则进入 BOUND；冲突则 DHCPDECLINE

客户端记得先前 IP 时可省略 DISCOVER/OFFER，直接广播 DHCPREQUEST（含 requested IP，ciaddr 仍为 0）。

7 · 租约续期 T1 / T2（§3.3 · §4.4.5）

租约时间以秒为单位，在报文中为相对时间；0xFFFFFFFF 表示无限租约。

客户端维护两个时刻（可由 Option 58/59 配置，默认 T1 = 0.5×租约，T2 = 0.875×租约）：

时刻状态行为
T1RENEWING向原服务器单播 DHCPREQUEST（ciaddr=当前 IP，无 server identifier）
T2REBINDING向任意服务器广播 DHCPREQUEST
租约到期—停止使用该 IP；须重新 DORA
租约生命周期时间轴

8 · DHCPINFORM（§3.4）

客户端已通过手工等方式配置 IP 时，发送 DHCPINFORM（ciaddr 已填）仅获取 DNS、路由等参数。服务器回复 DHCPACK，不分配新地址、不含 yiaddr/租约字段，通常单播至 ciaddr。

10 · Options 与 Magic Cookie（§3 · §4.1）

options 字段前 4 字节为 Magic Cookie：99, 130, 83, 99（十六进制 63 82 53 63，ASCII "cBsC"）。其后为 TLV 选项，最后一项必须为 Option 255 End。

Option名称说明
1Subnet Mask子网掩码
3Router默认网关列表
6Domain Name ServerDNS 服务器
15Domain Name域名后缀
50Requested IP Address客户端请求的 IP
51IP Address Lease Time租约时长（秒）
53DHCP Message Type1–8，见上表
54Server Identifier服务器 IP，REQUEST 中选定服务器
55Parameter Request List客户端想要的 option 编号列表

61Client Identifier客户端唯一标识
完整选项定义见 RFC 1533（及后续 RFC 如 3442 Classless Static Route 等）。options 过长时可设置标志将 sname/file 字段也用于承载选项。

11 · UDP 端口与报文投递（§4.1）

项目值
传输协议UDP
客户端 → 服务器目的端口 67（DHCP Server）
服务器 → 客户端目的端口 68（DHCP Client）
未获 IP 前客户端源地址IP 头源地址 = 0.0.0.0
广播地址255.255.255.255（或子网广播，视 flags B 位）

服务器如何选择回包方式

  • giaddr ≠ 0：发往 Relay 的 67 端口
  • giaddr = 0 且 ciaddr ≠ 0：单播至 ciaddr
  • giaddr = ciaddr = 0 且广播位置位：广播至 255.255.255.255
  • 否则：尝试硬件单播（chaddr）

12 · 主机配置参数（Appendix A 摘要）

DHCP 可下发 RFC 1122 Host Requirements 中的 IP 层、链路层、TCP 等参数，例如：

  • IP 地址、子网掩码、MTU、默认路由器
  • 静态路由、Perform router discovery
  • ARP 缓存超时、Ethernet 封装类型
  • TCP TTL、Keep-alive 参数

多数参数在 Host Requirements RFC 中有默认值；客户端可在 DISCOVER/REQUEST 中用 Parameter Request List 指定需要的选项子集。

13 · 安全考虑（§7）

RFC 2131 明确指出：DHCP 基于 UDP/IP，本身不安全。风险包括：

  • 未授权 DHCP 服务器可下发错误 IP、网关、DNS，实施中间人攻击
  • 恶意客户端可伪装合法客户端获取配置，或耗尽地址池

后续 RFC（如 DHCP Authentication）及网络层 ACL、DAI、802.1X 等用于缓解；部署生产环境应结合交换机端口安全与可信 DHCP 服务器。

相关更新 RFC：3396（长选项编码）、4361（Client Identifier）、6842（服务器应答中的 Client Identifier）等，见 RFC Editor 页面。
