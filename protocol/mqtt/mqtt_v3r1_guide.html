MQTT V3.1 协议图解导读

轻量级基于 Broker 的发布/订阅消息协议，运行于 TCP/IP 之上。固定头最小 2 字节，适合带宽受限与嵌入式场景。

Pub/Sub QoS 0/1/2 Last Will Retain UTF-8 Topic

1 · 协议简介

MQTT（MQ Telemetry Transport）采用发布/订阅模式，发布者与订阅者通过Broker（服务器）解耦，载荷内容对传输层透明。

V3.1 相对 V3 的主要变化

  • CONNECT 可携带用户名与密码
  • CONNACK 增加安全相关返回码
  • 未授权的 PUBLISH/SUBSCRIBE 对客户端不可见，但 MQTT 流程仍正常完成
  • 字符串支持完整 UTF-8（不再限于 US-ASCII 子集）
  • 协议版本号仍为 3（与 V3 兼容）

规范结构：① 通用报文格式 → ② 各控制报文细节 → ③ 客户端与服务器交互流程。

2 · 发布/订阅架构

Publisher / Subscriber 经 Broker 解耦

所有控制交互经 TCP 连接上的默认控制管道（单连接、双向），使用 14 种 MQTT 控制报文。

3 · 报文结构（§2）

每条 MQTT 控制报文 = 固定头 Fixed Header + 可选 可变头 Variable Header + 可选 载荷 Payload。

MQTT 控制报文组成

固定头 Byte 1

bit 7bit 0
7654 3210
Message Type (4 bit)DUPQoSRETAIN

Remaining Length（变长 1–4 字节）

表示当前报文中可变头 + 载荷剩余字节数（不含固定头本身）。每字节低 7 位为数据，最高位为延续位；最大约 256 MB。

字节数范围
10 – 127 (0x7F)
2128 – 16 383
316 384 – 2 097 151
42 097 152 – 268 435 455

可变头常见字段

  • Protocol Name / Level（CONNECT）
  • Connect Flags / Keep Alive（CONNECT）
  • Topic Name、Message ID（PUBLISH、SUBSCRIBE 等）

多字节整数均为大端序（MSB 在前）。MQTT 字符串 = 2 字节长度 + UTF-8 数据。

Message ID（§2.4）

QoS > 0 的 PUBLISH、PUBREL，以及 SUBSCRIBE / UNSUBSCRIBE 必须携带 16 位 Message ID（1–65535，同一方向不可复用直至完成）。

6 · CONNECT — 建立 MQTT 会话（§3.1）

TCP 连接建立后，客户端发送 CONNECT 创建协议层会话。固定头 Message Type = 1；DUP/QoS/RETAIN 未使用。

连接建立流程

Connect Flags（可变头）

标志含义
Clean Session1=清除旧会话状态；0=恢复会话（若服务器有）
Will Flag / Will QoS / Will Retain异常断开时服务器代发遗嘱消息
Username / Password是否携带用户名、密码字段（V3.1 新增）

Protocol Name 为 MQIsdp（6 字符）或规范中的 MQTT 变体；Protocol Level = 3 表示 V3.1。Client Identifier 长度 1–23 字符。

7 · CONNACK — 连接应答（§3.2）

服务器对 CONNECT 的响应。可变头 2 字节：Reserved + Connect Return Code。

CodeHEX含义
00x00Connection Accepted
10x01不可接受的协议版本
20x02Client ID 被拒绝（长度须 1–23）
30x03服务器不可用
40x04用户名或密码错误
50x05未授权

8 · Last Will and Testament（遗嘱）

CONNECT 中若 Will Flag=1，客户端预置 Will Topic、Will Message、Will QoS 与 Will Retain。当连接非正常断开（未发 DISCONNECT 且 Keep Alive 超时）时，Broker 向 Will Topic 发布遗嘱。

正常 DISCONNECT 不会触发遗嘱。Will 用于传感器离线告警、设备异常掉线通知等场景。

9 · Keep Alive 与 PING（§3.12–3.13）

CONNECT 中的 Keep Alive（秒）约定：若在此间隔内无报文，客户端应发 PINGREQ，服务器回复 PINGRESP。超时未收到则 Broker 可关闭连接并可能发布 Will。

10 · PUBLISH — 发布消息（§3.3）

客户端→Broker 或 Broker→订阅客户端。可变头含 Topic Name；QoS>0 时含 Message ID。载荷为应用数据（二进制透明）。

  • RETAIN=1：Broker 保留该 Topic 最后一条；新订阅匹配时下发（Retain 标志置位）
  • 空载荷 + RETAIN=1：删除该 Topic 保留消息

11 · SUBSCRIBE / SUBACK / UNSUBSCRIBE（§3.8–3.11）

SUBSCRIBE 载荷为多个「Topic Filter + Requested QoS」对；固定使用 QoS 1（需 SUBACK）。SUBACK 逐条返回授予的 QoS（0/1/2 或 0x80 失败）。

UNSUBSCRIBE 取消订阅，UNSUBACK 确认。二者均带 Message ID。

12 · QoS 0 — 最多一次（§4.1）

Fire and Forget · 无 Message ID

适用于可丢失的遥测采样等场景。

13 · QoS 1 — 至少一次（§4.1）

PUBLISH → PUBACK · 可能重复

SUBSCRIBE / UNSUBSCRIBE 也使用 QoS 1 级别传输。

14 · QoS 2 — 恰好一次（§4.1）

四步握手 · PUBLISH → PUBREC → PUBREL → PUBCOMP

15 · Topic 通配符（Appendix A）

层级分隔符 /。通配符仅用于订阅 Filter，发布时 Topic 不得含 # 或 +。

符号名称规则示例
#多级通配匹配零级或多级；须单独或跟在 / 后，且为最后一级finance/# 匹配 finance/stock/ibm
+单级通配只匹配一级；须单独或跟在 / 后home/+/temp 匹配 home/kitchen/temp

Topic 命名要点

  • 至少 1 字符；区分大小写
  • 可含空格；勿含 Unicode NUL
  • 前导 / 产生独立层级（/finance ≠ finance）
  • 树深度与根节点数量无硬性限制（单 Topic 最长约 64 KB）
