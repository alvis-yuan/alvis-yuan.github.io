MQTT v5.0 协议图解导读

OASIS 标准版 MQTT，在 v3.1.1 核心之上引入 Properties、统一 Reason Code、增强认证、共享订阅、请求/响应等能力，面向大规模 IoT 与 M2M 场景。

Protocol Level 5 Properties Reason Code AUTH Shared Sub Topic Alias

1 · 协议简介

MQTT v5.0 是 OASIS 发布的正式标准，与 ISO/IEC 20922 对齐。协议仍采用发布/订阅模型，客户端与服务器（Broker）通过 TCP/TLS 或 WebSocket 交换MQTT 控制报文。

规范结构（7 章）

章内容
1Introduction — 术语、数据表示、变更历史
2MQTT Control Packet format — 固定头、可变头、Properties、Reason Code
3MQTT Control Packets — CONNECT…AUTH 各报文细节
4Operational behavior — QoS 流程、会话、错误处理
5Security — 认证、授权、传输安全
6WebSocket 传输
7Conformance Targets

协议名称为 MQTT，Protocol Level = 5（0x05）。v5 不再兼容 v3.1.1 报文格式（需协商版本）。

2 · 相对 MQTT v3.1.1 的主要变化

规范 §1.8.2 列出 v5.0 设计目标：可扩展性、错误报告、常见模式形式化、性能与小客户端支持。

v5.0 新增能力一览
  • Clean Session → Clean Start + Session Expiry Interval，会话生命周期更精细
  • CONNACK / PUBACK 等多数应答带 Reason Code 与 Reason String
  • 服务器可发 DISCONNECT 主动断开客户端（v3 仅客户端发起）
  • 遗嘱消息支持 Will Delay Interval、独立 Will Properties
  • SUBACK 失败码 0x80；服务器可在 CONNACK 声明能力（如是否支持共享订阅）

3 · 发布/订阅架构

Publisher / Subscriber 经 Broker 解耦（与 v3 相同模型）

4 · 报文结构（§2.1）

控制报文仍由三部分组成；v5 在可变头末尾普遍增加 Properties 块。

MQTT v5 控制报文组成

固定头 Byte 1

bit 7bit 0
7654 3210
Message Type (4 bit)各报文专用 Flags

Remaining Length 为变长整数（Variable Byte Integer，最多 4 字节），表示可变头 + Properties + 载荷的总长度。

6 · Properties 属性机制v5

Properties = Property Length（变长整数）+ 零或多个 Property（Identifier 变长整数 + 按类型编码的值）。无属性时 Property Length 必须为 0。

8 · CONNECT / CONNACK（§3.1–3.2）

TCP 连接后客户端必须首先发送 CONNECT；同一连接上第二次 CONNECT 视为 Protocol Error。Protocol Name = MQTT，Protocol Level = 5。

v5 连接建立（含 Properties）

Connect Flags（相对 v3.1.1）

标志v5 说明
Clean Start (bit1)1=新会话（丢弃旧状态）；0=恢复已有 Session（若存在）
Will Flag / Will QoS / Will Retain遗嘱；Will 消息另有独立 Will Properties
Username / Password同 v3.1.1

CONNACK 可变头

  • Connect Acknowledge Flags：bit0 = Session Present
  • Connect Reason Code：见上表
  • Properties：Session Expiry、Assigned Client Identifier、Server Keep Alive、Maximum QoS、Retain Available 等能力声明

9 · AUTH 增强认证v5

当 CONNECT/CONNACK 中的 Authentication Method 无法一次完成认证时，双方可交换 AUTH 报文（Type=15）。Reason Code：0x18 Continue authentication、0x19 Re-authenticate。

挑战-响应式 AUTH 交换（示意）

适用于 SCRAM、OAuth 等需要多轮数据交换的认证机制；也支持连接后 Re-authenticate。

10 · 会话：Clean Start 与 Session Expiry

Clean Start=1：连接时清除旧 Session；CONNACK 的 Session Present 必为 0。

Clean Start=0：尝试恢复 Session；若服务器有状态则 Session Present=1 并恢复订阅与 QoS 1/2 未确认消息。

Session Expiry Interval（Property 0x11）：会话在连接断开后仍保留的秒数；0 表示断开即结束；0xFFFFFFFF 表示不过期（直到 Clean Start=1 清除）。可在 DISCONNECT 中修改。

v3.1.1 的 Clean Session 被拆分为 Clean Start + Session Expiry，便于「断线重连保留订阅但限制离线时长」等场景。

11 · PUBLISH 与 Topic Alias

PUBLISH 可变头：Topic Name 或 Topic Alias（二选一）、Packet Identifier（QoS>0）、Properties、Payload。

Topic Aliasv5

CONNECT/CONNACK 协商 Topic Alias Maximum 后，后续 PUBLISH 可用 2 字节 Alias 代替完整 Topic 字符串，降低带宽。Alias 仅在当前连接内有效，且 Topic Name 为空时 Alias 必须已建立映射。

其他 Publish Properties

  • Message Expiry Interval：消息过期时间（秒）
  • Payload Format Indicator：0=未指定，1=UTF-8 文本
  • Content Type：如 application/json
  • Subscription Identifier：Broker 转发时带回，标识匹配到的订阅

12 · Request / Response 模式v5

规范 §4.10 形式化请求/响应：请求方在 PUBLISH Properties 中设置 Response Topic 与可选 Correlation Data；响应方将回复 PUBLISH 到 Response Topic，并携带相同 Correlation Data 以便关联。

13 · 共享订阅 Shared Subscriptionv5

多个客户端订阅同一共享订阅时，每条匹配消息仅投递给其中一个 Session，实现消费端负载均衡。

Topic Filter 格式：

$share/{ShareName}/{TopicFilter}

示例：$share/group1/sensors/+/temp — 名为 group1 的共享组竞争消费 sensors/+/temp 消息。CONNACK 中 Shared Subscription Available 声明 Broker 是否支持。

共享订阅负载均衡

14 · SUBSCRIBE / SUBACK

SUBSCRIBE 固定头 QoS 字段必须为 1。载荷为 Topic Filter + Options 对；Options 含 Requested QoS、No Local、Retain As Published、Retain Handling 等（§3.8.3.1）。

SUBACK 载荷为与订阅一一对应的 Reason Code 列表（0x00–0x02 授予 QoS，0x80 失败）。

15 · QoS 0 / 1 / 2（§4.3）

QoS 语义与 v3.1.1 相同；v5 在 PUBACK/PUBREC 等应答中增加 Reason Code（如 0x10 No matching subscribers）。

QoS 2 四步握手（不变）

16 · DISCONNECT（§3.14）

v5 中服务器和客户端均可发送 DISCONNECT。可变头含 Reason Code 与 Properties（Session Expiry Interval、Reason String、Server Reference 等）。

  • 正常断开：Reason Code 0x00
  • 带遗嘱：0x04 Disconnect with Will Message
  • 管理踢出：0x93 Administrative action

发送 DISCONNECT 后必须关闭网络连接；收到 DISCONNECT 后也必须关闭连接。

17 · Topic 通配符

与 v3.1.1 相同：# 多级通配、+ 单级通配；仅用于订阅 Filter。共享订阅使用 $share/ 前缀（非通配符，为命名空间约定）。

符号规则示例
#匹配零级或多级，须为最后一级home/#
+匹配单级home/+/temp
$share/g/t共享订阅（v5）$share/workers/jobs/#
