About This Book

When the first edition was written, the RS-232 serial port was the workhorse of PC interfaces. Modems and scores of peripherals connected to PCs via serial ports present on every machine. When USB took hold in the late 1990s, many predicted serial ports would become obsolete — but they remained essential for many embedded systems, long-cable applications, and RS-485 networks.

While most PCs no longer have built-in COM ports, they are easy to add via USB converters. Microsoft's .NET Framework SerialPort class demonstrates that PC applications continue to find COM-port communications useful.

When to Use a Serial Port

A serial port is a computer interface that transmits data one bit at a time. In common use, the term refers to ports using a particular asynchronous protocol — these include RS-232 ports on PCs and many ports in embedded systems. Most serial ports are bidirectional: they can both send and receive data.

Advantages

  • Inexpensive hardware — readily available; PCs without built-in ports can use USB/serial converters
  • Long cable distances — RS-232 supports 130 ft or more; RS-485 can exceed 4,000 ft
  • Low-cost cables — unshielded cables with 3–9 conductors for many links
  • Universal microcontroller support — virtually every microcontroller family includes built-in serial ports
  • Flexibility — can exchange nearly any type of information

Limits

  • Maximum bit rate for RS-232 is specified at 20 kbps (though many chips exceed this)
  • Windows does not guarantee real-time performance for serial communications
  • USB maximum device-to-host distance is 16 ft (98 ft with five hubs), limiting comparison scenarios
  • RS-485 supports up to 10 Mbps — much faster than RS-232 for networked applications

Figure 1.1 — Maximum cable distance comparison across common PC interfaces (source: Jan Axelson, Chapter 1)

System Components

The Computers

A serial link connects two computers or a computer and a peripheral. One end is often a PC running Windows; the other may be another PC, a microcontroller-based embedded system, or a dedicated peripheral such as a GPS receiver or industrial sensor.

The Physical Link

The physical link includes the cable and connectors as well as interface hardware on each device. The interface hardware converts logic levels to the voltages required by the standard (e.g., ±12V for RS-232, differential pairs for RS-485).

Programming

Applications access serial ports through operating system drivers. On Windows, .NET's SerialPort class is the modern programming interface. Embedded systems access UARTs directly through hardware registers.

Applications

Example Systems

Serial ports are well suited for reading sensors, switches, and other inputs, or controlling motors, relays, displays, and other outputs. Typical applications include:

  • PC-to-PC data transfer links
  • PC-to-embedded-controller links
  • RS-485 industrial monitoring and control networks
  • GPS and GNSS receiver interfaces
  • Barcode scanner connections
  • Wireless sensor network gateways

Formats and Protocols

Asynchronous serial data, bit formats, flow control, and error checking

Chapter 2 of 16

Sending Serial Data

Asynchronous and Synchronous Communications

In asynchronous serial communication, each data word is framed with start and stop bits, and no shared clock line is needed. Sender and receiver must agree on the bit rate in advance. This is the most common form for RS-232 and many embedded systems.

In synchronous communication, a clock signal accompanies the data, allowing higher speeds but requiring an additional wire. Protocols like SPI and I²C use synchronous communication.

Word Formats

A serial word (frame) consists of:

Figure 2.1 — Asynchronous serial word format: Start bit → Data bits (LSB first) → optional Parity → Stop bit(s)

Bit Rate and Baud Rate

The bit rate (bits per second, bps) is the number of bits transmitted per second. The baud rate technically refers to the number of signal changes per second. For simple binary signaling, baud rate equals bit rate. Common bit rates include 9600, 19200, 38400, 57600, and 115200 bps.

Sending Bits

The Format

Before sending, the line is held at the mark (logic 1, high voltage) state. A start bit pulls the line to the space (logic 0) state, signaling the start of a word. Data bits follow, LSB first. One or two stop bits return the line to the mark state.

The Need for Accurate Timing

Because asynchronous communication uses no shared clock, both devices must use the same bit rate with sufficient accuracy. Typical UARTs tolerate clock errors of up to about ±5% before errors occur.

Data Formats

  • Binary Data — raw bytes; compact and efficient
  • Text Data — ASCII or Unicode strings; human-readable
  • ASCII Hex — binary data encoded as hexadecimal ASCII characters; useful for debugging
  • Application-specific Protocols — custom packet formats with headers, checksums, and command codes

Preventing Missed Data

Flow Control

Flow control prevents the sender from transmitting faster than the receiver can process data. Two common methods:

  • Hardware (RTS/CTS) — uses dedicated control signals. The receiver asserts CTS when ready; the sender waits when CTS is de-asserted.
  • Software (XON/XOFF) — embeds control characters in the data stream. XOFF (Ctrl+S, 0x13) pauses; XON (Ctrl+Q, 0x11) resumes.

Buffers

Transmit and receive buffers allow temporary storage of data when the application cannot immediately process received bytes. Windows COM port drivers provide software buffers; UARTs typically have small hardware FIFOs.

Error Checking

Methods for detecting transmission errors include:

  • Parity bit — simple; detects single-bit errors
  • Checksum — sum of transmitted bytes, appended to the packet
  • CRC (Cyclic Redundancy Check) — more robust polynomial-based error detection

Chapter 3

COM Ports on PCs

Port architecture, drivers, and accessing COM ports in Windows

Chapter 3 of 16

Port Architecture

Device Manager

Windows Device Manager lists available COM ports and reports their resources. Each port has an assigned COM number (COM1, COM2, etc.) and is associated with a device driver. Physical serial ports, USB-to-serial adapters, and Bluetooth virtual COM ports all appear here.

Port Resources

Physical COM ports on legacy PCs use an I/O address range and an IRQ (interrupt request line). The UART chip (typically an 8250/16550 derivative) handles the hardware. Modern systems access COM ports exclusively through the driver abstraction.

Serial Servers

A serial server (also called a device server) allows serial devices to be accessed over a network. The server presents itself as a COM port to the host, while the actual physical serial port may be connected elsewhere on the LAN.

Accessing Ports

Drivers

Windows uses kernel-mode drivers (serial.sys for physical ports, vendor-supplied drivers for USB and other virtual ports). Applications communicate with the driver through standard Windows file I/O and DeviceIoControl calls, or through higher-level APIs like .NET's SerialPort class.

Identifying Ports

Applications can enumerate available COM ports via the Windows Registry (HKEY_LOCAL_MACHINE\HARDWARE\DEVICEMAP\SERIALCOMM) or by using the Setup API with the appropriate GUID.

GUIDs for COM Ports

The GUID for COM ports is {4D36E978-E325-11CE-BFC1-08002BE10318}. Applications use the Setup API functions with this GUID to enumerate all COM port devices, including USB virtual COM ports.

COM Port Numbering

Windows assigns COM numbers starting at COM1. Numbers above COM9 require a special path prefix (\\.\COM10) when opening the port. USB-to-serial adapters and Bluetooth serial ports receive assigned numbers based on the Registry.

INF Files

INF files instruct Windows how to install a device driver for a new device. They specify the driver files, Registry entries, and associations between hardware IDs and drivers. Virtual COM port devices require properly crafted INF files.

  • Win32 API (CreateFile / ReadFile / WriteFile) — Direct access; more control but more complexity
  • ActiveX / COM components — Legacy third-party serial port controls
  • Winsock for Serial Servers — TCP/IP socket connection to a serial server device

Inside RS-232

Hardware signals, voltage levels, interface chips, and port-powered circuits

Chapter 4 of 16

The Hardware Interface

Signals

RS-232 defines a set of signals for serial communication. The primary data signals are:

  • TD (Transmitted Data) — carries data from DTE to DCE
  • RD (Received Data) — carries data from DCE to DTE

Handshaking signals include:

  • RTS (Request to Send) / CTS (Clear to Send) — hardware flow control pair
  • DTR (Data Terminal Ready) / DSR (Data Set Ready) — device readiness signals
  • DCD (Data Carrier Detect) — modem has detected a carrier
  • RI (Ring Indicator) — modem detects incoming call
  • SG (Signal Ground) — reference for all signal voltages

Voltages

RS-232 uses higher voltages than TTL/CMOS logic:

Signal TypeLogic 0 / SpaceLogic 1 / Mark
Data (TD, RD)+3V to +15V−3V to −15V
Control (RTS, CTS, etc.)+3V to +15V (asserted)−3V to −15V (de-asserted)
Note: The RS-232 voltage convention is inverted relative to TTL logic: a positive voltage represents a logic 0 (space) on data lines, while a negative voltage represents a logic 1 (mark). This is important when designing interface circuits.

Figure 4.1 — RS-232 voltages compared with TTL/CMOS logic; an interface chip (e.g., MAX232) translates between them

Converting Voltages

Interface Chips

The most common way to convert between RS-232 and TTL/CMOS logic is with a dedicated interface chip such as the Maxim MAX232 (for 5V systems) or MAX3232 (for 3.3V systems). These chips include built-in charge pumps to generate the ±12V supply from a single +5V or +3.3V source.

Short-range Circuits

For very short cable connections (e.g., within an enclosure), simplified circuits using transistors or dedicated low-voltage chips can replace full RS-232 level converters, reducing cost and component count.

Port-powered Circuits

Some small peripherals can draw their operating power from RS-232 signal lines rather than requiring a separate power supply. The RS-232 output signals can source limited current.

  • Using Outputs as a Power Source — multiple RS-232 outputs can be combined to supply a small amount of power
  • Regulating the Voltage — zener diodes or linear regulators ensure a stable operating voltage for the peripheral circuit

Null Modem Cables

A null modem cable connects two DTE devices (e.g., two PCs) directly, crossing the transmit and receive lines and looping back handshaking signals. This is used for direct PC-to-PC serial links without modems.

Figure 5.1 — Null modem wiring: TX and RX lines are crossed between the two DTE devices

Cables

Length Limits

The RS-232 standard specifies a maximum cable capacitance of 2500 pF (not a specific length). At common bit rates, cables up to 50–100 feet work reliably with standard cables. Shielded cable allows longer runs.

Surge Protection

Adding transient-voltage-suppression (TVS) diodes to signal lines protects against voltage spikes from electrostatic discharge (ESD) and electrical surges, especially for long outdoor cable runs.

Isolated Lines

Electrical isolation eliminates ground-loop currents that can cause errors or damage equipment when the two communicating devices are at different ground potentials (common in industrial settings).

  • Optoisolators — use LEDs and phototransistors to transmit signals with no electrical connection
  • Isolated DC-DC converters — provide isolated power for the remote circuit

Debugging Tools

  • Breakout box — passively monitors signal lines and allows reconnection for testing different wiring configurations
  • Voltmeter — verifies that RS-232 signal voltages are within range
  • Oscilloscope — displays signal waveforms and timing for detailed analysis
  • Logic analyzer — captures and decodes serial data at the bit level

Inside RS-485

Balanced lines, differential signaling, interfacing options, and driver control

Chapter 6 of 16

About RS-485

Balanced and Unbalanced Lines

RS-232 uses unbalanced (single-ended) signaling — each signal is measured relative to a shared ground. RS-485 uses balanced (differential) signaling — each signal is carried on a pair of wires (A and B), and the receiver measures the difference between them. This dramatically improves noise immunity, especially over long distances.

Figure 6.1 — RS-485 differential signaling; noise that affects both wires equally (common-mode) is cancelled at the receiver

Voltage Requirements

RS-485 receivers detect the difference between the A and B wires. A difference of at least +200 mV indicates a logic 1; −200 mV or less indicates a logic 0. The wide common-mode input range (typically −7V to +12V) allows the devices on the bus to have different ground potentials.

Speed

RS-485 supports data rates up to 10 Mbps on short cables, and 100 kbps or more on long cables (up to 4,000 ft). The cable length and bit rate are inversely related: longer cables require lower bit rates.

Interfacing Options

Chips

Popular RS-485 transceiver ICs include the MAX485 (Maxim), SN75176 (Texas Instruments), and ADM485 (Analog Devices). These chips convert between UART-level TTL signals and RS-485 differential signals.

Adding a Port on a PC

A PC can add RS-485 capability via a PCI/PCIe card, a USB-to-RS-485 adapter, or a USB-to-RS-232 adapter combined with an RS-232-to-RS-485 converter.

Controlling the Driver Enable

Unlike RS-232, RS-485 is typically half-duplex: the same pair of wires carries traffic in both directions, but only one driver transmits at a time. The driver must be enabled only during transmission and disabled afterward:

  • Software-assisted control — the application asserts the RTS line before transmitting and de-asserts it after the last byte is sent
  • Hardware control — specialized UART chips automatically enable/disable the driver based on transmit activity

Chapter 7

Designing RS-485 Links and Networks

Line terminations, biasing, cable types, and topologies

Chapter 7 of 16

Long and Short Lines

Whether a line is considered "long" or "short" depends on the relationship between the cable length, the propagation delay, and the bit rate. When the cable's propagation delay exceeds about one-quarter of the bit period, the line must be treated as a transmission line and properly terminated.

Rule of thumb: Terminate RS-485 lines when the cable length (in feet) exceeds 50 / (bit rate in Mbps). For example, at 1 Mbps, terminate cables longer than 50 ft.

Line Terminations

Characteristic Impedance

Twisted-pair cables for RS-485 typically have a characteristic impedance of 100–120 Ω. The termination resistor value should match this impedance to prevent signal reflections.

Adding a Termination

Place a termination resistor (typically 120 Ω) across the A and B wires at each end of the bus. For a two-node point-to-point link, both ends are terminated. For multidrop networks, only the two physical ends of the bus segment are terminated.

Figure 7.1 — RS-485 multidrop bus: termination resistors at both ends; up to 32 standard unit loads (or more with low-load chips)

Biasing the Line

When all drivers are disabled (no node is transmitting), the RS-485 bus is in an undefined state. Bias resistors pull the A wire to a high voltage and the B wire to a low voltage, ensuring that receivers see a valid idle state and preventing false data detection.

Cable Types

  • Twisted-pair cable — the twisting causes noise to be induced equally on both wires (common-mode), which the differential receiver cancels out
  • Shielded twisted pair — adds a grounded shield for additional noise protection in high-EMI environments
  • Select cable with a characteristic impedance matching your termination resistors (typically 100–120 Ω)

Grounds and Differential Lines

Even though RS-485 is differential, a signal ground connection between nodes is still needed to keep the common-mode voltage within the receiver's input range. A separate ground wire in the cable, or connecting the shield to ground at one end, typically provides this reference.

Going Wireless

Media, modulation, infrared, and RF wireless serial links

Chapter 8 of 16

Media and Modulation

Wireless serial communication transmits data without physical cables, using electromagnetic waves. Serial data that would normally travel on a wire instead modulates a carrier signal — radio frequency (RF) or infrared (IR) light.

Spread Spectrum Technology

Many wireless systems use spread spectrum techniques, which spread the transmitted power over a wider frequency band than necessary. This provides resistance to interference and eavesdropping, and allows multiple systems to share the same frequency band.

Ensuring Reliable Transfers

Wireless links are more prone to errors than wired connections. Reliable protocols use acknowledgments and retransmission, error-correcting codes, and adaptive frequency hopping (used in Bluetooth) to maintain data integrity.

Infrared

Infrared communication uses invisible light (wavelength ~880 nm) to transmit serial data over short distances (typically less than 1 meter for consumer devices).

IrDA

The Infrared Data Association (IrDA) standard defines a protocol for short-range, point-to-point infrared serial communication. IrDA ports were common in laptops and PDAs, and some embedded devices still include IrDA capability. IrDA presents itself to the OS as a virtual COM port.

Radio Frequency (RF)

Complying with Regulations

RF devices must comply with regulations that specify allowed frequency bands, power levels, and modulation methods. In the USA, the FCC governs unlicensed RF devices under Part 15 rules.

Choosing an RF Band

Common unlicensed RF bands used for serial wireless links include:

Accessing COM ports in Windows using the .NET Framework

Gaining Access to a Port

Finding Ports

The SerialPort.GetPortNames() static method returns an array of available COM port names. This is the easiest way to populate a port-selection combo box in a GUI application.

' VB.NET example — enumerate available COM ports Dim portNames() As String = SerialPort.GetPortNames() For Each name As String In portNames ComboBox1.Items.Add(name) Next

Opening a Port

Create a SerialPort object and set its properties (port name, baud rate, parity, data bits, stop bits), then call Open():

// C# example SerialPort myPort = new SerialPort("COM3", 9600, Parity.None, 8, StopBits.One); myPort.Open();

Timeouts

The ReadTimeout and WriteTimeout properties (in milliseconds) determine how long a read or write operation waits before throwing a TimeoutException. Setting ReadTimeout = SerialPort.InfiniteTimeout makes reads block indefinitely.

Closing a Port

Always close the port when done: myPort.Close(). Closing releases the port so other applications can use it. Using a using block in C# ensures the port is closed even if an exception occurs.

Transferring Data

Transferring Bytes

Use Write(byte[], offset, count) to send binary data and Read(byte[], offset, count) to receive it. The BytesToRead property reports how many bytes are waiting in the receive buffer.

Transferring Text

Use WriteLine(string) and ReadLine() for line-by-line text exchange. The NewLine property controls the line terminator (default: \n). Use WriteChar() or ReadChar() for single characters.

Using Stream Objects

BinaryReader and BinaryWriter

SerialPort.BaseStream provides a Stream object that can be wrapped with BinaryReader and BinaryWriter for typed read/write operations (e.g., reading Int32, Single, etc.) directly from/to the serial port.

StreamReader and StreamWriter

Similarly, StreamReader and StreamWriter can wrap the base stream for convenient text I/O with encoding support, including Unicode.

Saving a Port and Parameters

The Application Settings Architecture

.NET's application settings system (Properties.Settings) allows applications to save and restore user-selected COM port settings between sessions. Settings are stored in a user-specific XML file.

Chapter 10

Managing Ports and Transfers in .NET

Receiving data, flow control, error handling, and application structure

Chapter 10 of 16

Receiving Data

Detecting Received Data

The DataReceived event fires on a secondary thread when bytes arrive in the receive buffer. Because it runs on a different thread than the UI, accessing controls requires invoking through the main thread:

// C# — DataReceived event handler void myPort_DataReceived(object sender, SerialDataReceivedEventArgs e) { string received = myPort.ReadExisting(); // Must invoke UI updates on the main thread: textBox1.Invoke(new Action(() => textBox1.AppendText(received))); }

Setting Timeouts

Set ReadTimeout appropriately for the application. Short timeouts allow the application to remain responsive; longer timeouts suit slow or intermittent devices. Catching TimeoutException prevents the application from hanging.

Sending Data

Sending without Blocking

For applications that must remain responsive, perform serial writes on a background thread or use asynchronous write operations. Avoid lengthy serial writes on the UI thread.

Preventing Buffer Overflows

If the receiver cannot keep up, the transmit buffer fills and Write() may block or throw an exception. Use flow control (RTS/CTS or XON/XOFF) to prevent overflows automatically.

Flow Control

Selecting a Method

Set the Handshake property:

  • Handshake.None — no flow control
  • Handshake.RequestToSend — hardware RTS/CTS flow control
  • Handshake.XOnXOff — software XON/XOFF flow control
  • Handshake.RequestToSendXOnXOff — both hardware and software

Monitoring Control Signals

The CtsHolding, DsrHolding, DcdHolding, and BreakState properties let applications read the current state of RS-232 control signals. The PinChanged event fires when a control signal changes state.

Handling Errors

Exceptions

Wrap port operations in try/catch blocks to handle common exceptions: UnauthorizedAccessException (port in use), TimeoutException (read/write timeout), and IOException (device disconnected).

The ErrorReceived Event

The ErrorReceived event fires when the UART reports an error: framing error (invalid stop bit), parity error, or receive buffer overflow. Check SerialError in the event args to determine the error type.

Structuring an Application

The book recommends encapsulating all serial port code in a dedicated class (e.g., ComPorts). This class exposes high-level methods and events, hiding the details of port management from the application's UI code.

Recommended Application Structure

  • ComPorts class handles all SerialPort interactions
  • UI thread only updates controls via Invoke
  • Background thread or event-driven model for receiving
  • Application-specific events defined for received packets
  • Settings saved with .NET application settings

Chapter 12

Network Programming

Traffic management, protocols, addressing, and message formats for serial networks

Chapter 12 of 16

Managing Traffic

Steps in Exchanging a Message

A typical message exchange on a serial network follows these steps:

  1. The primary (master) node selects a target address
  2. Primary sends a command or request packet
  3. Secondary (slave) node recognizes its address
  4. Secondary sends a response packet
  5. Primary validates the response (checksum, ACK)

Protocols

Serial networks typically use a master/slave (primary/secondary) polling protocol, where a single master controls all bus access. This prevents collisions, which cannot be detected on RS-485 half-duplex networks.

Debugging Tips

  • Use a logic analyzer or second serial port to monitor bus traffic
  • Test each node individually before connecting all to the network
  • Verify address byte reception before implementing full protocol
  • Use loopback testing to verify hardware before software integration

Addressing

Assigning Addresses

Each secondary node on the network needs a unique address. Addresses can be assigned by:

  • Hardware switches (DIP switches or jumpers) — set at installation time
  • Firmware programmed at manufacture — convenient but inflexible
  • Discovery protocols — automatic address assignment through software

9-bit Format

The 9-bit UART mode is designed for multidrop networks. The 9th bit distinguishes address bytes (9th bit = 1) from data bytes (9th bit = 0). Secondary nodes can monitor only address bytes and wake up only when their address is detected, saving power.

An RS-485 Network

Complete example: hardware, protocol, commands, polling, and responses

Chapter 13 of 16
Complete Example: Chapter 13 presents a fully worked RS-485 network with a PC primary node and PIC microcontroller secondary nodes. Hardware schematics and complete firmware code are provided.

Connecting the Nodes

Transceivers

Each node (PC and each microcontroller) connects to the RS-485 bus through an RS-485 transceiver chip (e.g., MAX485). The PC connects via a USB-to-RS-485 adapter or a PC RS-232-to-RS-485 converter.

Terminating and Biasing

120 Ω termination resistors at each end of the bus, plus bias resistors on the primary node to hold the bus in a defined state when no driver is active.

Example Protocol

Addresses

Each secondary node is assigned a unique address (1–254). Address 0 is reserved for broadcast; address 255 is reserved.

Message Format

Byte PositionFieldDescription
0AddressTarget node address (1–254)
1CommandCommand code (e.g., Read=0x01, Write=0x02)
2Data (optional)Data byte for write commands
LastChecksumXOR or sum of preceding bytes

Commands

Reading a Byte

The primary sends a packet with the target address and a "Read" command. The addressed secondary receives the command, reads the requested data (e.g., a sensor value), and sends a response packet containing the data byte.

Writing a Byte

The primary sends a packet with the target address, a "Write" command, and the data byte to write. The secondary applies the data (e.g., sets an output) and optionally sends an acknowledgment.

Polling the Nodes

The primary cycles through each secondary node, sending a request and waiting for a response within a timeout period. Nodes that do not respond within the timeout are flagged as offline. The polling loop runs continuously during normal operation.

Configuring the Driver-enable Line

On the PC side, the RTS line of the serial port controls the RS-485 driver-enable pin. Before transmitting, the software asserts RTS; after transmitting (waiting for the last byte to send), it de-asserts RTS to allow the bus to be released for the responding node.

Hosts and Devices

Host Responsibilities

The USB host (typically a PC) manages all communications on the bus. The host:

  • Initiates all data transfers (devices never send unsolicited data)
  • Enumerates devices on connection and assigns addresses
  • Assigns a driver to each device based on its Vendor ID, Product ID, and class
  • Manages power distribution to downstream devices

Speed

USB defines multiple speed tiers:

SpeedData RateTypical Use
Low Speed1.5 MbpsKeyboards, mice

Hi-Speed480 MbpsMass storage, webcams
SuperSpeed (USB 3.x)5+ GbpsHigh-performance storage

Endpoints

A USB endpoint is a unidirectional data buffer in the device. Each device has a required bidirectional control endpoint (endpoint 0). Additional endpoints handle data transfers. Virtual COM port devices typically have two bulk endpoints (IN and OUT) for data plus additional endpoints for CDC notifications.

USB Transfers

Transfer Types

  • Control — configuration and command; used by all devices for enumeration
  • Bulk — large, error-corrected data; used by virtual COM ports for serial data
  • Interrupt — small, periodic data with bounded latency; used for CDC notifications
  • Isochronous — real-time streaming data (audio/video); no error correction

Chapter 15

Using Special-function USB Controllers

FTDI FT232R/FT245R chips, drivers, and implementing virtual COM ports

Chapter 15 of 16
Focus: FTDI's FT232R and FT245R are dedicated USB-to-serial/parallel bridge chips that handle all USB protocol complexity in hardware, allowing microcontrollers with only UART or parallel ports to appear as USB virtual COM ports.

Inside the Chips

FT232R — Serial Interface

The FTDI FT232R is a USB-to-UART bridge. It connects to a microcontroller via a standard UART (TX, RX, and optional handshake signals) and presents as a USB virtual COM port to the host. The chip handles all USB enumeration, protocol, and data transfer internally.

  • UART data rates up to 3 Mbaud
  • Integrated EEPROM for custom Vendor ID / Product ID / serial number
  • Supports RS-232 handshaking signals (RTS, CTS, DTR, DSR, DCD, RI)
  • Single 3.3V or 5V supply

FT245R — Parallel Interface

The FTDI FT245R provides a parallel (byte-wide) FIFO interface rather than UART. This is faster for applications that can utilize byte-parallel data transfer directly.

Prototyping Modules

FTDI and third-party suppliers offer ready-made modules containing the FT232R with USB connector, decoupling capacitors, and optional RS-232 level shifter. These modules allow rapid prototyping of USB serial devices.

Using the Controllers

Drivers

FTDI provides two types of Windows drivers:

  • VCP (Virtual COM Port) driver — presents the device as a standard COM port accessible via the SerialPort class
  • D2XX driver — provides direct access to the FTDI chip via a dedicated API, with access to chip-specific features

Implementing a Virtual COM Port

With the FT232R, no USB firmware programming is required. Connect the microcontroller's UART TX/RX to the FT232R's RXD/TXD pins, connect the FT232R to the USB connector, and install the FTDI driver on the PC. The device immediately appears as a virtual COM port.

Converting from RS-232 to USB

Legacy RS-232 equipment can be converted to USB by adding an FT232R with an RS-232 level converter (e.g., MAX232). This allows older equipment with RS-232 ports to connect to modern PCs without built-in serial ports.

Using Generic USB Controllers

The Communication Devices Class (CDC)

The USB Communications Device Class (CDC) is an official USB standard that defines how serial communication devices interface with USB hosts. CDC virtual COM port devices are supported by Windows, Linux, and macOS without vendor-specific drivers (though an INF file may be needed on older Windows versions).

Device Controllers

Any USB microcontroller with sufficient firmware can implement CDC. Common choices include:

  • Microchip PIC18F2550 / PIC18F4550 (on-chip USB Full-Speed)
  • Atmel AT90USB162 / ATmega32U4
  • Silicon Labs CP2102/CP2104
  • ST Microelectronics STM32 with USB CDC library

Host Drivers

Windows includes a built-in CDC driver (usbser.sys) that creates a virtual COM port when a CDC device is connected. Modern Windows (10 and later) loads this driver automatically without requiring an INF file.

Using the Abstract Control Model (ACM)

The CDC Abstract Control Model (ACM) is the subclass used for virtual COM ports. It defines a set of class-specific requests and notifications that allow the host to set serial parameters (baud rate, data bits, parity, stop bits) and to receive status information from the device.

POTS Models

The ACM was originally designed for modems ("Plain Old Telephone Service" devices). For virtual COM ports, most CDC class requests relating to modem control can be implemented as no-ops, as they are not meaningful for direct serial connections.

Requests and Notifications

Descriptors and INF Files

Device Descriptor

Contains the Vendor ID (VID), Product ID (PID), device class (0x02 for CDC), device protocol, and maximum packet size for endpoint 0.

Configuration Descriptor

Defines the device's interfaces and their endpoints. A CDC virtual COM port device has two interfaces: the Communications Class interface (with an interrupt endpoint for notifications) and the Data Class interface (with bulk IN and OUT endpoints for data).

The INF File

On Windows, a vendor-provided INF file links the device's VID/PID to the usbser.sys driver. The INF file contains the device's VID, PID, a description string, and instructions to load the CDC driver. Windows 10 and later can match many CDC devices without a custom INF.
