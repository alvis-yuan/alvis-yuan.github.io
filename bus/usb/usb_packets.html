USB2.0 令牌、事务及传输 — 抓包分析总结

力科分析仪实录

概述

本文基于 力科（LECROY）USB 总线分析仪的真实抓包数据，系统归纳 USB2.0 协议中的令牌（Token）、事务（Transaction）与传输（Transfer）三层结构，帮助读者从波形到协议逻辑形成完整认知。

🎯 核心三层模型
  • 包（Packet）：总线上传输的最小物理单元，包含 PID、数据字段、CRC 等
  • 事务（Transaction）：由令牌包 + 数据包（可选）+ 握手包组成的最小可靠传输单元，不可跨帧
  • 传输（Transfer）：由一个或多个事务组成的完整业务操作（控制 / 批量 / 中断 / 同步）

包的层次结构

USB2.0 的数据传输是分层架构的，从下到上依次是物理层信号 → 包 → 事务 → 传输。

图1：USB2.0 包-事务-传输层次结构
关键约束：一个事务的各个包必须在同一帧内完成，不得跨帧传输，也不允许插入其他包。

令牌包概述

令牌包全部由 USB 主机发出，用于指明后续数据包的目标地址和端点。令牌包内容包括：

PID 包标识  |  ADDR 设备地址 7位  |  ENDP 端点地址 4位  |  CRC5 校验

ADDR 为 7 位 → 最多支持 127 个设备（地址 0 用于枚举初始化）；ENDP 为 4 位 → 每设备最多 16 个双向通道。
PID（低4位）令牌类型说明PID 值
0001OUT主机发送数据到设备0xE1
1001IN主机从设备接收数据0x69
0101SOF帧/微帧开始标志0xA5
1101SETUP主机向设备发送配置信息0x2D
1000SPLIT高速与低/全速转换（特殊）—
0100PING高速模式探测设备接收能力0xB4
1100PRE低速数据先导包—
图2：令牌包格式（共 24bit = 3字节）

SETUP 令牌包

SETUP 令牌包用于主机向 USB 设备发送配置信息，是控制传输的起始令牌。

SETUP 令牌包结构
  • PID 低4位：1101，高4位补码 0010，完整PID = 0x2D
  • ADDR：7位，指明目标设备地址
  • ENDP：4位，始终指向端点0
  • CRC5：5位校验字段
SETUP 令牌 只用于控制传输的 Setup 阶段，始终向端点0发送，后续紧跟携带请求数据的 DATA0 数据包。

IN 令牌包

IN 令牌包用于主机通知设备"我要从你这里读取数据"，数据流向为 设备 → 主机。

IN 令牌包结构
  • PID 低4位：1001，高4位补码 0110，完整PID = 0x69
  • ADDR：7位设备地址
  • ENDP：4位端点号（读方向，如 0x81 = 端点1 IN）
  • CRC5：5位校验

OUT 令牌包

OUT 令牌用于主机向设备发送数据，数据流向为 主机 → 设备，令牌后跟数据包，设备接收后返回 ACK。

OUT 令牌包结构
  • PID 低4位：0001，高4位补码 1110，完整PID = 0xE1
  • ADDR：7位，目标设备地址（由 SET_ADDRESS 请求分配）
  • ENDP：4位，目标端点号
  • CRC5：5位校验
OUT 令牌之后紧跟 DATA0 或 DATA1 数据包，设备接收成功后回复 ACK；如设备缓冲区满，则回复 NAK（全速/低速）或先经 PING 探测（高速）。

🔗 原文：USB2.0 OUT令牌包

SOF 令牌包

SOF（Start Of Frame）令牌包由主机定期发送，用于标识帧或微帧的开始，是最广泛存在的令牌包。

SOF 令牌包特殊结构
  • PID：0101，PID值 0xA5
  • 帧号字段：11位（取代 ADDR+ENDP）
  • CRC5：5位校验
  • 全速设备每 1ms 一帧，高速设备每 125µs 一微帧
SOF 包不需要握手包，因此无法保证设备一定接收到。抓包工具默认会隐藏 SOF 和 NAK 包以减少噪声，但它们在物理层大量存在。

PING 令牌包

PING 事务是 USB2.0 高速模式专有功能，用于主机在发送 OUT 数据之前，先探测设备是否有空间接收，从而节约总线带宽。

PING 令牌包结构
  • PID 低4位：0100，高4位补码 1011，完整PID = 0xB4
  • 结构与 IN/OUT 令牌相同：PID + ADDR + ENDP + CRC5
图4：PING 事务两种流程对比
PING 节约带宽的原理：OUT 令牌后面紧跟数据，一旦 NAK 则数据已经在总线上浪费了。PING 只发令牌（3字节），探测成功再发大包，大幅减少总线浪费。

🔗 原文：USB2.0 PING令牌包

SPLIT 令牌包

SPLIT 令牌包是 USB2.0 为了解决高速主机与低速/全速设备互通问题而设计的特殊令牌，由 USB 集线器（Hub）在中间进行速度转换。

类型SC位说明
SSPLIT（启动 SPLIT）SC=0主机→Hub：以高速启动一次低/全速事务
CSPLIT（完成 SPLIT）SC=1主机→Hub：查询上一次低/全速事务的结果
SPLIT 令牌额外字段
  • ADDR 7位：Hub 的地址（非目标设备）
  • PORT 7位：Hub 上的端口号（最多128端口）
  • S/E：同步传输中指明全速数据在高速传输中的位置
  • ET 2位：传输类型（00=控制，01=同步，10=批量，11=中断）

🔗 原文：USB2.0 令牌包

数据包

数据包用于承载实际传输数据，由 PID（8bit） + 数据字段 + CRC16（16bit） 组成。

数据包类型PID低4位使用场景
DATA00011标准数据传输（偶数包序列）
DATA11011标准数据传输（奇数包序列）
DATA20111高速同步传输专用
MDATA1111高速同步/SPLIT 传输专用
DATA0 与 DATA1 交替翻转是 USB 可靠传输的核心机制（DATA TOGGLE）。同步传输（Isochronous）没有握手包，不使用数据翻转机制。

握手包（ACK / NAK / STALL / NYET / ERR）

握手包是最简单的包格式，只包含一个 8 位的 PID 字段，用于反馈事务的执行结果。

握手包PID低4位PID值发送方含义
ACK00100xD2主机 / 设备数据正确接收，可进行下一次事务
NAK10100x5A设备设备无数据发送(IN) / 无法接收数据(OUT)
STALL11100x1E设备不支持该请求或端点已停止（Halt）
NYET01100x96设备高速 OUT 事务：本次接收成功但下一包暂不接受
ERR1100—HubSPLIT 事务中下游发生错误
图5：握手包格式（共 8bit = 1字节）
STALL 与 NAK 的区别：NAK 是暂时状态（设备忙），主机可以重试；STALL 是错误状态（端点挂起或请求不支持），主机必须先发 CLEAR_FEATURE 请求清除挂起才能继续。

事务结构

一个 USB 事务是数据传输的最小可靠单元，通常由三个包组成：

  • 令牌包：指定事务类型、目标设备地址和端点
  • 数据包（可选）：承载数据，长度不超过端点最大包长
  • 握手包：确认数据传输的结果（同步传输无握手包）
事务 ≠ 传输：一个传输（Transfer）往往包含多个事务。例如控制传输的数据阶段，若端点最大包长为8字节而需传输18字节，则需要3个IN事务（8+8+2字节）。

🔗 原文：USB2.0 事务

SETUP 事务

SETUP 事务是控制传输的第一阶段，用于主机向设备发送 8 字节的 USB 标准请求（bmRequestType + bRequest + wValue + wIndex + wLength）。

图6：SETUP 事务时序（主机→设备，设备返回ACK）
SETUP 事务中的数据包始终使用 DATA0，设备必须接受（不能 NAK）。接收后设备必须返回 ACK。

IN 事务

IN 事务由主机发起，用于从设备读取数据。设备可以响应 DATA、NAK 或 STALL。

设备响应含义主机后续动作
DATA0/DATA1设备有数据，正常返回发送 ACK，切换数据序列
NAK设备暂无数据重传 IN 令牌，数据序列不变
STALL端点已停止上报错误，需 CLEAR_FEATURE 后重试

🔗 原文：USB2.0 IN事务

OUT 事务

OUT 事务由主机发起，向设备写入数据。设备可以响应 ACK、NAK、STALL 或（高速时）NYET。

设备响应含义主机后续动作
ACK数据正确接收切换数据序列，发送下一包
NAK设备缓冲区满（全速/低速）重传 OUT 事务，数据序列不变
NYET本包接收成功，下包暂不接受（高速）下一包使用 PING 令牌探测
STALL端点已停止上报错误

🔗 原文：USB2.0 OUT令牌包

PING 事务

PING 事务仅由令牌包 + 握手包组成，没有数据包，是 USB2.0 高速模式（480Mbps）专用机制。

PING 事务特点
  • 仅用于高速模式的 Bulk OUT 传输前
  • 主机先发 PING 令牌（3字节），比发 OUT+DATA 代价低得多
  • 设备返回 ACK → 主机发完整 OUT+DATA；返回 NAK → 主机继续 PING
  • 如果 OUT 事务后设备返回 NYET，主机下次先发 PING

DATA TOGGLE（数据翻转）机制原理

USB 使用数据翻转机制在主机和设备之间同步多个连续事务，防止数据重复或丢失。

核心原理
  • 主机和设备各自维护一个数据包序列位（Data Toggle Bit）
  • 数据包使用 DATA0 / DATA1 交替发送
  • 数据接收方收到数据后根据握手（ACK/NAK）决定是否翻转自身序列位
  • 数据发送方收到 ACK 后翻转自身序列位
  • 同步传输（Isochronous）无握手包，不使用数据翻转机制

ACK 成功时的数据翻转

当 ACK 正确接收时，主机和设备均翻转数据包序列位，双方同步。

阶段主机数据序列设备数据序列说明
事务 N 开始前DATA1DATA1双方同步，均为 DATA1
设备发 DATA1—DATA1（发送）设备用当前序列发数据
主机发 ACKDATA1→DATA0（翻转）—主机确认成功，预备下一包
设备收到 ACKDATA0DATA1→DATA0（翻转）双方序列再次同步为 DATA0
事务 N+1 开始DATA0DATA0使用新序列传输

ACK 丢失时的数据翻转（错误恢复）

当 ACK 在传输中丢失时，USB 的数据翻转机制能自动检测并恢复，防止数据重复。

IN 事务 ACK 丢失的恢复流程
  • 主机收到 DATA1 后翻转序列为 DATA0，发出 ACK → ACK 丢失
  • 设备未收到 ACK，序列保持 DATA1 不变
  • 主机发下一个 IN 令牌，设备用 DATA1（旧序列）重发数据
  • 主机收到 DATA1，但预期是 DATA0 → 判断这是重传包
  • 主机丢弃该数据，发 ACK 让设备同步序列位
  • 设备收到 ACK，翻转为 DATA0，双方序列同步
OUT 事务 ACK 丢失的恢复流程
  • 主机发 DATA0，设备接收后翻转为 DATA1，发出 ACK → ACK 丢失
  • 主机未收到 ACK，序列保持 DATA0 不变，超时后重传
  • 主机再次发 OUT + DATA0（旧序列）
  • 设备预期 DATA1，收到 DATA0 → 判断是重传包，忽略数据
  • 设备返回 ACK，主机收到后翻转为 DATA1，双方序列同步
关键洞察：数据序列位是检测重传的唯一手段。正是通过比较"期望序列"和"实际序列"，接收方才能区分新数据与重传数据，实现零重复的可靠传输。

传输 vs 事务 vs 包

三个核心概念的层次关系总结：

概念英文最小单元可靠性说明
包Packet位/字节CRC 校验物理层传输的最小单元
事务Transaction包握手包确认不可跨帧，不可插入其他包
传输Transfer事务多事务协同完整的业务操作（请求/响应）

控制传输（Control Transfer）

控制传输是最复杂的传输类型，用于设备枚举和控制命令发送。它由三个阶段组成：

图9：控制传输三阶段结构
跨帧规则：控制传输的不同事务可以在不同帧中进行，但同一个事务内的包不可以跨帧。

抓包实例：获取配置描述符（端点0最大包长=8字节）

  • 事务169：SETUP 事务 → 主机发 GET_DESCRIPTOR 请求（DATA0，8字节）
  • 事务170：IN 事务 → 设备返回配置描述符前 8 字节
  • 事务171：IN 事务 → 设备返回第 9 字节（最后 1 字节）
  • 事务224：IN 事务 → 零字节 DATA（空 Status 响应，用于确认完成）

批量 / 中断 / 同步传输
