Introduction & Abstract

The widespread integration of USB into embedded applications presents many developers with the challenge of debugging problems that are difficult to detect and isolate when a USB device misbehaves.

This paper discusses the various USB debugging techniques, including debugging at the host PC, at the device, and in the cable, and discusses when each of these techniques will be most useful.

The paper also covers the various facilities provided within the Linux kernel to aid in USB debugging — for example sysfs, debugfs, trace points — as well as the various user-space tools available to help USB debugging, such as USBMON. Profiling techniques at various levels in the USB stack are discussed as well.

Target Audience
This paper is of interest to Software Developers and Testers who develop and debug USB drivers on Linux-based embedded platforms.

§1 · USB Generic Linux System Architecture

The USB Linux system involves both hardware and software components on both the host and device sides. Understanding this layered architecture is essential for locating where a bug might originate.

Fig 1 — USB Generic Linux System Architecture: Host (left) and Device (right) software/hardware layers with debug tool positions

Key Components

Class DriverHostE.g. smsc95xx, usbhid, usb-storage
libusb / ApplicationHostUser-space library for vendor-defined device access
UDC (USB Device Controller)DeviceHardware controller on the device side; e.g. dwc3
Gadget DriverDeviceImplements USB functions like mass storage, HID, serial on device

§2 · USB Mass Storage Architecture

Mass Storage is one of the most commonly debugged USB device classes. Understanding the layered architecture from the USB bus through to the backing file system is key to isolating performance or correctness issues.

Fig 2 — USB Mass Storage Architecture: software stacks on host (left) and device (right)

The SCSI layer on the host side issues commands (e.g. READ, WRITE) that travel via USB bulk transfers to the device. Understanding which layer is at fault — VFS, block layer, USB transport, or the gadget/device — determines the appropriate debugging tool.

Documentation
Device information from cat /sys/kernel/debug/usb/devices can be decoded using the documentation at Documentation/usb/proc_usb_info.txt in the kernel source tree, and at linux-usb.org/FAQ.html#i6.

§3 · Challenges in Debugging USB

USB debugging is inherently difficult due to the nature of the protocol and the multi-layered software stack involved. The primary challenges include:

Fig 3 — Common challenges when debugging USB issues in embedded Linux
  • Encoded data: USB uses NRZI (Non-Return-to-Zero Inverted) encoding with bit stuffing, making raw bus signals non-trivial to interpret without a hardware analyzer.
  • Timing issues: USB is a polled bus — the host initiates all transfers. Race conditions between the host polling schedule and device readiness are a common source of bugs.
  • Out-of-spec signaling errors: Voltage levels, cable impedance, and connector quality can cause intermittent failures that are extremely difficult to reproduce.
  • Protocol errors: Stall conditions, broken-pipe errors, CRC failures, and incorrect descriptor responses are common protocol-level bugs.
  • Intermittent failures: Many USB bugs only appear under specific conditions (e.g. after resume from suspend, on particular USB ports, or at certain speeds).

§4 · sysfs & debugfs

4.1 sysfs — USB Device Information

The sysfs entry for enumerated USB devices is /sys/bus/usb/devices/. Each entry contains all the information about its USB device.

Shell
root@am335x-evm:~# ls /sys/bus/usb/devices/
1-0:1.0  1-1      1-1:1.0  2-0:1.0  usb1     usb2

Naming conventions:

  • Names beginning with usb refer to USB controllers (root hubs).
  • X-0:1.0 is a special case referring to the root hub's interface.
  • Devices are named as Bus-Port:Config.Interface

A more complete view of all attached USB devices can be obtained via debugfs:

Shell
cat /sys/kernel/debug/usb/devices

4.2 debugfs — Controller Register Dump

debugfs exposes debug interfaces for USB controllers (e.g. dwc3). The typical structure for a DWC3 controller:

debugfs layout
/sys/kernel/debug/
└── dwc3.0/
    ├── regdump        ← full register dump
    ├── testmode       ← USB test modes
    ├── mode           ← host, device, or otg
    └── link_state     ← link speed state

Mounting debugfs (if not already mounted):

Shell
# mount -t debugfs none /sys/kernel/debug

Dumping registers before and after a suspend/resume cycle:

Shell
# cat /sys/kernel/debug/dwc3.0/regdump > before_suspend.txt
# (trigger system suspend/resume)
# cat /sys/kernel/debug/dwc3.0/regdump > after_resume.txt
# diff before_suspend.txt after_resume.txt

§5 · usbmon — USB Bus Monitor

usbmon is a kernel facility used to collect traces of I/O on the USB bus. It is analogous to a packet socket used by network monitoring tools such as tcpdump or Wireshark. usbmon reports requests made by peripheral-specific drivers to Host Controller Drivers (HCD).

Important
Since usbmon intercepts at the usbcore/HCD boundary, it captures URBs (USB Request Blocks) — not actual wire-level packets. If the HCD itself is buggy, traces may not correspond exactly to bus transactions.
Fig 4 — usbmon interception point in the Linux USB host stack

5.1 Setting Up usbmon

Shell — Step 1: Mount debugfs and load module
# mount -t debugfs none_debugs /sys/kernel/debug
# modprobe usbmon

# Verify bus sockets are present:
# ls /sys/kernel/debug/usb/usbmon
0s  0u  1s  1t  1u  2s  2t  2u  3s  3t  3u  4s  4t  4u

File naming: the number is the bus number (0 = all buses), the letter indicates format:

FilenameDescription
XsGeneric event overview (bus X)
XuStream trace data — current preferred format (binary-compatible URB)
XtStream trace data — deprecated older format (since 2.6.21)
0uCapture from all buses simultaneously

5.2 Finding the Target Device's Bus

Shell — Step 2: Identify bus
# cat /sys/kernel/debug/usb/devices
T:  Bus=03 Lev=01 Prnt=01 Port=00 Cnt=01 Dev#=  2 Spd=12  MxCh=  0
D:  Ver= 1.10 Cls=00(>ifc ) Sub=00 Prot=00 MxPS= 8 #Cfgs=  1
P:  Vendor=0557 ProdID=2004 Rev= 1.00
S:  Manufacturer=ATEN
S:  Product=UC100KM V2.00

Bus=03 means the device is on bus 3. Alternatively, use lsusb:

Shell
# lsusb
Bus 003 Device 002: ID 0557:2004 ATEN UC100KM V2.00

5.3 Capturing Traces

Shell — Step 3: Start capture (single bus)
# cat /sys/kernel/debug/usb/usbmon/3u > /tmp/bus3.mon.out
Shell — Capture from all buses
# cat /sys/kernel/debug/usb/usbmon/0u > /tmp/all.mon.out

After performing the desired USB operation, stop capture with Ctrl+C.

5.4 usbmon Text Format (1u)

Each event is a text line of whitespace-separated words. Fields from left to right:

FieldDescription
URB TagUnique identifier for the URB (usually in-kernel address in hex)
TimestampMicroseconds since epoch
Event TypeS=submission, C=callback, E=submission error
AddressType:Bus:Dev:Endpoint — e.g. Bi:003:002:01
URB StatusReturn status code (negative = error)
Setup packet5 words for control transfers (bmRequestType, bRequest, wValue, wIndex, wLength)
Data LengthRequested (submission) or actual (callback) length
Data tag= if data words follow
Data wordsByte stream in big-endian hex

Transfer type codes in the Address field:

CodeMeaning
Ci / CoControl input / output
Zi / ZoIsochronous input / output
Ii / IoInterrupt input / output
Bi / BoBulk input / output

5.5 Analyzing with Virtual USB Analyzer

The paper recommends Virtual USB Analyzer (vusb-analyzer), a free and open-source GUI tool capable of displaying usbmon text-format data, as well as XML data from hardware analyzers such as the Ellisys USB Explorer 200.

Tool Download
Virtual USB Analyzer: vusb-analyzer.sourceforge.net

§6 · Dynamic Debug Interface

The Dynamic Debug Interface provides runtime turning on/off of kernel debug statements — specifically dev_dbg() and dev_vdbg() calls. This interface is not limited to USB; it can be used to debug any driver ported to the dev_dbg() framework.

Advantage
No kernel recompilation is needed. Debug verbosity is controlled at runtime via debugfs, allowing fine-grained control over which kernel modules and source files emit debug output.

Enabling Dynamic Debug

The interface is accessible through debugfs:

Shell
# mount -t debugfs none /sys/kernel/debug

# Enable all dev_dbg() messages in usb/core:
# echo "file drivers/usb/core/*.c +p" > /sys/kernel/debug/dynamic_debug/control

# Enable debug for a specific file:
# echo "file drivers/usb/dwc3/core.c +p" > /sys/kernel/debug/dynamic_debug/control

# Enable for a specific function:
# echo "func dwc3_gadget_suspend +p" > /sys/kernel/debug/dynamic_debug/control

# Disable again:
# echo "file drivers/usb/core/*.c -p" > /sys/kernel/debug/dynamic_debug/control

Control flags:

FlagDescription
pEnable the message (print)
fInclude function name in output
lInclude line number
mInclude module name
tInclude timestamp

§7 · Tracepoints & Ftrace

7.1 Ftrace

Ftrace is part of the kernel tracing infrastructure. It has zero overhead when disabled and minimal overhead when enabled. It is a very useful tool to debug USB issues as well as for performance analysis.

Shell — Basic Ftrace for USB
# cd /sys/kernel/debug/tracing

# List available tracers:
# cat available_tracers

# Enable function tracing:
# echo function > current_tracer

# Filter to USB-related functions:
# echo "dwc3_*" > set_ftrace_filter
# echo "xhci_*" >> set_ftrace_filter

# Start tracing:
# echo 1 > tracing_on

# Perform USB operation...

# Stop and read trace:
# echo 0 > tracing_on
# cat trace

7.2 USB Tracepoints

The Linux kernel provides dedicated USB tracepoints — another tool in the kernel tracing infrastructure for kernel debugging and performance diagnosis. Tracepoints are static instrumentation points placed throughout the USB subsystem.

Shell — Enable USB tracepoints
# cd /sys/kernel/debug/tracing/events/usb

# List USB tracepoints:
# ls
usb_submit_urb  usb_urb_complete  usb_giveback_urb
usb_hub_exit    usb_hub_irq

# Enable all USB events:
# echo 1 > /sys/kernel/debug/tracing/events/usb/enable

# Read trace output:
# cat /sys/kernel/debug/tracing/trace
Fig 5 — Comparison of kernel debug facilities for USB troubleshooting

§8 · USB Protocol Analyzer

A USB protocol analyzer is a USB bus sniffer that captures, decodes, filters, and displays USB bus activities at the wire level — including actual signal states, packets, descriptors, and timing. It operates externally in the cable, capturing everything the bus carries.

Hardware vs. Software
Hardware analyzers (e.g. Ellisys, LeCroy, Total Phase) capture wire-level packets with precise timing. Software tools like vusb-analyzer only see URBs as presented to the HCD — they are higher-level and cannot see signaling issues or out-of-spec timing.

Recommended Tools

ToolTypeDescription
Ellisys USB Explorer 200 Hardware Hardware analyzer; captures wire-level packets with precise timing; exports XML for use with vusb-analyzer
LeCroy / Teledyne LeCroy Hardware Professional USB protocol and signal analyzer
Total Phase Beagle Hardware Non-intrusive USB protocol analyzer; supports USB 2.0 and USB 3.0
Virtual USB Analyzer (vusb-analyzer) Software / Free Free, open-source GUI tool; displays usbmon output and XML from hardware analyzers. ↗
Wireshark Software / Free Can read usbmon binary captures via /dev/usbmon or pcap format

When to Use a Hardware Protocol Analyzer

  • When the device fails to enumerate and software tools give no output
  • When diagnosing out-of-spec signaling voltages or impedance
  • When timing-sensitive issues (e.g. SOF timing) are suspected
  • When usbmon shows no errors but the device misbehaves

§9 · Gadget Zero — USB Testing Driver

Gadget Zero is a simple USB gadget driver used only for testing controller drivers. It is a reference implementation in the Linux kernel and is very useful for verifying that the UDC (USB Device Controller) and gadget framework are functioning correctly.

Configurations

Gadget Zero implements two configurations:

ConfigurationDescription
Source/Sink The device generates (sources) data or absorbs (sinks) data. Useful for throughput and correctness testing of bulk and isochronous transfers.
Loopback Data written to the device is sent back to the host. Tests full round-trip transfer correctness.

As of kernel 3.12, Gadget Zero supports Control, Bulk, and Isochronous transfers.

Loading Gadget Zero

Shell
# Load the g_zero gadget (source/sink test)
# modprobe g_zero

# Or with loopback:
# modprobe g_zero loopback=1

# Check USB device state:
# cat /sys/class/udc/*/state
Test Script
The kernel provides a test utility in tools/usb/testusb.c designed to exercise the Gadget Zero driver with various transfer types and sizes.

§10 · Profiling Tools

Beyond debugging, the paper discusses profiling techniques to measure USB performance at various levels in the USB stack.

Fig 6 — Profiling tools applied at different levels of the USB mass storage stack

10.1 IOzone — Mass Storage Profiling

IOzone is a filesystem benchmark tool used to measure USB mass storage performance. It generates and measures file I/O at the application level.

Shell — Basic IOzone test
# Run a basic 1GB sequential read/write test:
iozone -a -n 512m -g 1g -i 0 -i 1 -f /mnt/usb/testfile

# Flags:
# -a : auto mode (all tests)
# -n : minimum file size
# -g : maximum file size
# -i 0 : write test
# -i 1 : read test
# -f : test file path

10.2 FFSB — Flexible Filesystem Benchmark

FFSB (Flexible Filesystem Benchmark) is a cross-platform filesystem performance analysis tool. It allows configuring workloads that simulate real-world mixed read/write patterns.

10.3 dd — Simple Transfer Benchmark

dd is a simple utility for benchmarking sequential read/write speed of a USB storage device.

Shell — Measure USB write speed
# Write speed test (to USB device):
dd if=/dev/zero of=/dev/sdb bs=1M count=1024 oflag=direct

# Read speed test (from USB device):
dd if=/dev/sdb of=/dev/null bs=1M count=1024 iflag=direct

§11 · Case Study 1: Device Not Enumerating After Suspend/Resume

Platform
Texas Instruments OMAP platform with DWC3 USB controller, running Linux 3.7.0-rc2.

Symptoms

After adding PM (Power Management) support for DWC3, it was found that after the system enters off mode and resumes, the USB device was not enumerating. Adding debug prints revealed that no interrupts were being received from DWC3 after resume.

Fig 7 — Timeline of the suspend/resume enumeration failure (Case Study 1)

Debugging Steps

  1. Take register dump before suspend:
    Shell
    cat /sys/kernel/debug/dwc3.0/regdump > before_suspend.txt
  2. Trigger system to OFF mode and resume (platform-specific suspend trigger).
  3. Take register dump after resume:
    Shell
    cat /sys/kernel/debug/dwc3.0/regdump > after_resume.txt
  4. Diff the register dumps:
    Shell
    diff before_suspend.txt after_resume.txt
  5. The diff revealed that the DEVTEN register (Device Event Enable register) had lost its contents after OFF mode — its bits were all zeroed out, explaining why no USB interrupts were generated.

Resolution

The fix was to save and restore the DWC3 register context (including DEVTEN) during PM suspend and resume callbacks. After the fix, the device enumerated correctly after resume from OFF mode.

§12 · Case Study 2: USB Stick Kernel Regression

Symptoms
A USB storage stick worked fine on older kernels (tested on 2.6.34 and 2.6.37) but stopped working on kernel 3.0 and 3.1.

Debugging Steps

  1. Obtained usbmon traces for both the working (2.6.37) and non-working (3.1) setups.
  2. Used device sysfs info to identify the USB bus and device number for the stick.
  3. Filtered the usbmon traces to only the target device's URBs.
  4. Diffed the two trace logs — the non-working log contained additional commands compared to the working log.

Finding

From the diff, the non-working trace contained additional commands including UDISK commands. These were followed by URBs returning a broken pipe error. After that, the host repeatedly sent CLEAR ENDPOINT HALT commands — but the device continued to stall.

Fig 8 — Comparison of usbmon traces: working vs. non-working kernel (Case Study 2)

Root Cause & Resolution

It was concluded that a user-space program (or kernel regression in the SCSI layer) was sending bogus commands (like UDISK) that this particular device did not support, causing it to stall. The fix involved identifying and stopping the rogue user-space application from issuing non-standard SCSI commands to the device.

§13 · Case Study 3: Webcam Works on USB 3 but Not USB 2

Symptoms
A webcam functioned correctly when connected to a USB 3.0 port but failed when plugged into a USB 2.0 port on the same system.

Debugging Methodology

  1. Obtained usbmon traces for both working (USB 3 port) and non-working (USB 2 port) scenarios.
  2. Identified the device on each bus using /sys/kernel/debug/usb/devices.
  3. Filtered and compared the traces for the webcam's device address on each bus.
Fig 9 — usbmon trace comparison for webcam on USB 3 vs USB 2 port (Case Study 3)

Root Cause

The usbmon trace comparison revealed that on the USB 2.0 bus, the device was being suspended by the host (USB selective suspend / autosuspend) shortly after enumeration — whereas on USB 3.0, the device stayed active. The webcam's firmware did not handle resume from suspend correctly on USB 2.0, so it never began streaming.

Resolution

The resolution involved either disabling USB autosuspend for this device or fixing the device firmware to correctly handle USB resume. The power management behavior difference between xHCI (USB 3) and EHCI (USB 2) host controllers was the root environmental cause.

Lesson Learned
Always capture usbmon traces on both the working and non-working setup. A diff of the two traces often immediately reveals the behavioral difference. Pay particular attention to power state transitions (suspend/resume events).
