第一章UART 基础概述

1.1 什么是 UART

UART（Universal Asynchronous Receiver-Transmitter，通用异步收发器）是一种硬件通信协议，负责将并行数据转换为串行位流进行异步传输，并在接收端完成反向转换。它是计算机与外设之间最古老、最简单也最广泛使用的通信接口之一。

在 Linux 内核中，UART 设备通过 TTY 子系统统一管理。每个串行端口在用户空间表现为 /dev/ttyS*（板载 UART）、/dev/ttyUSB*（USB 转串口）或 /dev/ttyAMA*（ARM PL011 UART）等设备节点。

图 1-1：UART 帧格式（8N1：8位数据、无奇偶校验、1位停止位）

1.2 历史演进

Linux 早期的串行驱动代码集中在 drivers/char/serial.c，包含大量 #ifdef 硬件分支，代码极难维护。以 ARM Linux 维护者 Russell King 为首的开发者，将其重构为通用 serial core 框架加多个小型硬件特定驱动，该改动随 Linux 2.5.28 合入主线。

📌 原文引用
"Thankfully, a number of developers, led by ARM Linux maintainer Russell King, restructured the serial driver into a generic serial core and a number of smaller, hardware-specific drivers. That code showed up in the main kernel tree sometime around the 2.5.28 release." — Greg Kroah-Hartman, Linux Journal #104, Dec 2002
🔗 Linux Journal: The Serial Driver Layer 🔗 Kernel Doc: Low Level Serial API

📝 章节总结 · 第一性原理

  • UART 的本质是时序控制的异步串行总线：发送方和接收方靠预约的波特率同步，无需额外时钟线。
  • Linux 内核将硬件多样性封装在驱动层，对上提供统一的 TTY 接口——这是"分层抽象"原则的典型实践。
  • 历史上的代码重构（2.5.x 时代）正是为了解决关注点分离（Separation of Concerns）问题：通用逻辑与硬件细节分离。

第二章UART 子系统层级架构

2.1 整体架构图

Linux UART 子系统采用严格的分层设计，从用户空间到物理硬件依次为：用户空间应用 → TTY 层（tty_struct / tty_driver）→ 线路规程（Line Discipline）→ Serial Core（uart_driver / uart_port）→ 底层硬件驱动（uart_ops）→ UART 硬件 FIFO。

图 2-1：Linux UART 子系统分层架构（基于 Linux Kernel 文档及源码）

2.2 用户空间层

用户程序通过标准 POSIX 文件接口访问串口设备节点。常见设备文件：

设备节点说明对应硬件
/dev/ttyS0 ~ ttyS3PC 板载 COM1~COM48250/16550 UART
/dev/ttyUSB0USB 转串口（通用）FTDI / CP210x / PL2303
/dev/ttyACM0USB CDC ACM 设备Arduino / modem
/dev/ttyAMA0ARM PL011 UARTRaspberry Pi 等
/dev/ttyMXC0i.MX 系列 UARTNXP i.MX SoC

串口配置由 termios 结构体描述，包含波特率、数据位、停止位、流控等参数，通过 tcgetattr() / tcsetattr() 或 ioctl(TCGETS) 等操作。

2.3 TTY 层

根据 Linux Kernel TTY 文档，TTY 层有以下核心结构：

📖 内核文档原文
"There are several major TTY structures. Every TTY device in a system has a corresponding struct tty_port. These devices are maintained by a TTY driver which is struct tty_driver. This structure describes the driver but also contains a reference to operations which could be performed on the TTYs. It is struct tty_operations. Then, upon open, a struct tty_struct is allocated and lives until the final close." — TTY — The Linux Kernel documentation
结构体作用源文件
struct tty_struct表示一个 TTY 设备实例（open 时分配，close 时释放）include/linux/tty.h
struct tty_driver描述一类 TTY 驱动，含 tty_operations 指针include/linux/tty_driver.h
struct tty_operationsTTY 驱动回调函数集合（open/close/write/ioctl...）include/linux/tty_driver.h
struct tty_port每个物理设备对应一个 port（跨 open/close 生存）include/linux/tty_port.h

2.4 线路规程层（Line Discipline）

每个字符在从设备到用户之间必须经过一个预先选定的线路规程（ldisc）。默认线路规程是 N_TTY，实现了字符回显、信号处理（Ctrl+C → SIGINT）、作业控制、特殊字符处理等功能。

📖 内核文档原文
"Every character received by the kernel (both from devices and users) is passed through a preselected TTY Line Discipline (in short ldisc; in C, struct tty_ldisc_ops). Its task is to transform characters as defined by a particular ldisc or by user too. The default one is n_tty, implementing echoes, signal handling, jobs control, special characters processing, and more." — TTY — The Linux Kernel documentation
规程名编号用途
N_TTY0默认规程，支持标准终端功能（正规模式 / 原始模式）
N_SLIP1SLIP 网络封装
N_MOUSE2鼠标输入
N_HDLC13HDLC 同步串行协议
N_GSM071021GSM 07.10 复用（多路复用串口）
N_PPS18PPS（精确时间源）

2.5 Serial Core 层

Serial Core 是连接 TTY 层和底层硬件驱动的中间框架层，其核心文件是 drivers/tty/serial/serial_core.c。

📖 内核文档原文
"Register a uart driver with the core driver. We in turn register with the tty layer, and initialise the core driver per-port state. We have a proc file in /proc/tty/driver which is named after the normal driver." — Low Level Serial API — The Linux Kernel documentation

驱动注册流程：

/* 1. 注册驱动 */
uart_register_driver(struct uart_driver *drv);

/* 2. 逐一注册串口 */
uart_add_one_port(struct uart_driver *drv, struct uart_port *port);

/* 3. 注销 */
uart_remove_one_port(drv, port);
uart_unregister_driver(drv);

核心数据结构 uart_driver 关键字段：

struct uart_driver {
    struct module       *owner;        /* 模块所有者，通常为 THIS_MODULE */
    const char          *driver_name;  /* 驱动名（出现在 /proc/tty/driver/） */
    const char          *dev_name;     /* 设备名（如 "ttyS"） */
    int                  major;        /* 主设备号 */
    int                  minor;        /* 起始次设备号 */
    int                  nr;           /* 支持的最大串口数 */
    struct console       *cons;        /* 串口控制台指针（可为 NULL） */
    /* 以下由 serial core 内部使用 */
    struct uart_state   *state;
    struct tty_driver   *tty_driver;
};

2.6 底层硬件驱动

底层驱动负责提供 uart_port（端口信息）和 uart_ops（控制方法）给 Serial Core，并处理硬件中断。

📖 内核文档原文
"The low level serial hardware driver is responsible for supplying port information (defined by uart_port) and a set of control methods (defined by uart_ops) to the core serial driver. The low level driver is also responsible for handling interrupts for the port, and providing any console support." — Low Level Serial API — The Linux Kernel documentation

数据传输机制：

📌 内核文档原文
"The data that is sent from the user to the tty layer through a call to write(2) is placed in a circular buffer by the serial driver layer, and it is up to the specific UART driver to pick up this data and send it out of the port. Usually, every time the UART interrupt happens, the driver checks the circular buffer to see if any more data is present to be sent." — Greg Kroah-Hartman, Linux Journal, 2002

2.7 uart_ops 回调接口

以下为 struct uart_ops 完整接口（摘自 内核官方文档）：

回调函数说明锁状态

set_mctrl(port, mctrl)设置调制解调器控制线（RTS/DTR/OUT1/OUT2/LOOP）port->lock

🔗 Kernel Doc: uart_ops 完整定义

📝 章节总结 · 第一性原理

  • 分层隔离：每层只向上提供抽象接口，向下调用下层服务。TTY 层不关心具体 UART 硬件，Serial Core 不关心具体线路规程。
  • 环形缓冲区（Circular Buffer）是 TX 路径的核心：Serial Core 写入，硬件驱动中断消费——实现了生产者-消费者解耦。
  • Flip Buffer 是 RX 路径的核心：中断处理程序写入 flip buffer，软中断（tasklet）将数据推送给线路规程。
  • uart_ops 定义了 Serial Core 与硬件驱动的"契约"——这是标准的策略模式（Strategy Pattern）设计。
  • 锁设计：port->lock（spinlock）保护发送/接收状态；tty_port->mutex（port_sem）保护端口生命周期操作。

📝 章节总结

  • /proc/tty/ 是内核运行时状态的只读快照窗口，无需特权即可读取，适合快速排查串口是否被内核识别。
  • /proc/tty/driver/serial 中 uart:unknown 意味着 UART 未被检测或被禁用，是硬件诊断的第一入口。
  • tx/rx 计数器永不重置（系统运行期间累计），可用差值计算吞吐量。
  • 第一性原理：/proc 的本质是"内核状态的只读投影"，内核通过注册 read_proc 回调将内部数据结构序列化为文本，供人类可读。

第四章/sys 文件系统中的 UART 信息

4.1 sysfs 概览

sysfs（/sys）是 Linux 2.6 引入的内核对象模型（kobject/kset）在用户空间的映射，提供设备、驱动、总线、类的结构化视图，支持属性读写。与 /proc 相比，sysfs 是设备模型驱动的，更适合程序化访问。

📖 内核文档原文
"There is only one valid place in sysfs where hierarchy can be examined and this is below: /sys/devices. It is planned that all device directories will end up in the tree below this directory." — Rules on how to access information in sysfs — The Linux Kernel documentation

图 4-1：/sys 中与 UART/Serial 相关的目录结构（基于内核 sysfs ABI 文档）

4.2 /sys/class/tty

/sys/class/tty/ 下包含系统中所有 TTY 设备的符号链接，每个链接指向 /sys/devices/ 树中对应的真实设备目录。

/sys/class/tty/ ├── console → ../../devices/virtual/tty/console │ └── active # 当前激活的控制台设备列表，如 "tty1 ttyS0" ├── tty → ../../devices/virtual/tty/tty │ └── active # 当前活动虚拟控制台，如 "tty1" ├── ttyS0 → ../../devices/pci0000:00/.../ttyS0 │ ├── uevent │ ├── uartclk # UART 时钟频率 = 16 × baud_base │ ├── type # 端口类型编号 │ ├── line # 行号（如 0 对应 ttyS0） │ ├── port # I/O 端口地址 │ ├── irq # 中断号 │ ├── flags # 端口状态标志 │ ├── xmit_fifo_size # 发送 FIFO 大小（字节） │ ├── close_delay # 关闭延迟（百分之一秒） │ ├── closing_wait # 等待关闭时长（百分之一秒，0=永久等待，65535=不等待） │ ├── custom_divisor # 自定义波特率除数 │ ├── io_type # I/O 访问类型 │ ├── iomem_base # MMIO 基地址 │ ├── iomem_reg_shift # 寄存器地址间距（移位值） │ ├── rx_trig_bytes # RX FIFO 中断触发字节数（可写） │ └── console # 是否作为内核控制台（可写，布尔值） ├── ttyS1 → ... └── ttyUSB0 → ../../devices/usb.../ttyUSB0/

4.4 /sys/devices 中的真实设备路径

sysfs 规则要求始终通过解析符号链接来获取真实路径。典型的串口设备路径结构：

# 板载 8250 串口（x86） /sys/devices/pci0000:00/0000:00:16.3/0000:02:00.0/ └── tty/ └── ttyS0/ ├── uartclk, irq, port, ... # ARM 平台串口（如 Raspberry Pi） /sys/devices/platform/fe201000.serial/ └── tty/ └── ttyAMA0/ ├── uartclk, irq, ... # USB 转串口 /sys/devices/pci0000:00/.../usb1/1-1/1-1.2/1-1.2:1.0/ └── ttyUSB0/ └── ... 
# 查看 ttyS0 的真实设备路径
$ readlink -f /sys/class/tty/ttyS0

# 查看 uartclk
$ cat /sys/class/tty/ttyS0/uartclk

# 查看 RX 触发字节数（16550A 支持 1/4/8/14）
$ cat /sys/class/tty/ttyS0/rx_trig_bytes

# 修改 RX 触发字节（需要 root 权限）
$ echo 8 > /sys/class/tty/ttyS0/rx_trig_bytes

# 动态挂载为控制台
$ echo 1 > /sys/class/tty/ttyS0/console

📝 章节总结

  • /sys 的本质是内核 kobject 树的可读写投影，与 /proc 不同：sysfs 属性是结构化的（每文件一属性），而 /proc 通常是人可读的汇总文本。
  • /sys/class/tty/ttyS*/ 下的属性通过 TIOCGSERIAL ioctl 接口暴露，等价于 setserial -g /dev/ttyS0 的输出。
  • rx_trig_bytes 是少数可写属性之一，可动态调整 FIFO 中断阈值以在延迟和 CPU 效率间取得平衡。
  • 第一性原理：sysfs 将内核对象模型（kobject）直接映射为文件系统层级，实现了"配置即文件、状态即文件"的 Unix 哲学。

第五章内核源码树结构

5.1 drivers/tty/serial 目录结构

Linux 串行驱动代码组织在 drivers/tty/serial/ 目录下，包含 Serial Core 框架及各厂商硬件驱动。

drivers/tty/ ├── tty_io.c # TTY 层核心（open/close/read/write） ├── tty_ioctl.c # TTY ioctl 处理 ├── n_tty.c # N_TTY 线路规程（默认） ├── tty_ldisc.c # 线路规程管理 ├── tty_buffer.c # Flip Buffer 管理 ├── tty_port.c # tty_port 生命周期管理 └── serial/ ├── serial_core.c # Serial Core 框架（uart_register_driver 等） ├── serial_base.c # 串行基础设施（ctrl/port device） ├── amba-pl011.c # ARM PL011 UART（参考实现） ├── imx.c # NXP i.MX UART ├── samsung.c # Samsung Exynos UART ├── msm_serial.c # Qualcomm MSM UART └── 8250/ ├── 8250_core.c # 8250 驱动核心 ├── 8250_port.c # 8250 端口操作（mapbase/membase） ├── 8250_pci.c # PCI 串口卡支持 ├── 8250_of.c # 设备树（Open Firmware）支持 └── 8250_omap.c # TI OMAP/AM33xx UART

"The reference implementation is contained within amba-pl011.c." — Low Level Serial API — The Linux Kernel documentation

📝 章节总结

  • 源码树将"框架"（serial_core.c）与"实现"（8250_core.c, amba-pl011.c）严格分离，遵循开闭原则（OCP）：添加新 UART 芯片只需实现 uart_ops，无需改动 serial core。
  • TX 路径核心是环形缓冲区（生产者=TTY层，消费者=UART中断）；RX 路径核心是Flip Buffer（中断上下文写，softirq 读）。
  • tty_insert_flip_char() 是 RX 路径的枢纽——驱动中断处理函数调用它，将数据从硬件 FIFO 搬运到 TTY 子系统。
  • ARM 平台参考实现 amba-pl011.c 是学习新驱动编写的最佳起点（内核文档明确推荐）。

6.1 stty — 串口参数查询与设置

stty（set tty）是 POSIX 标准工具，通过 termios 接口读写串口配置。

sttyCLI

查看 /dev/ttyS0 的当前所有设置

stty -F /dev/ttyS0 -a

stty 配置CLI

设置 115200 波特率 8N1 原始模式

stty -F /dev/ttyUSB0 115200 \
  cs8 -cstopb -parenb raw

stty 流控CLI

关闭硬件流控（RTS/CTS）

stty -F /dev/ttyS0 -crtscts

关键选项说明：

选项含义
-a显示所有当前设置
-F /dev/ttyS0指定设备（BSD: -f）
cs88 位数据位（cs7 = 7位）
-cstopb1 个停止位（cstopb = 2个）
-parenb无奇偶校验
raw原始模式（关闭所有 ldisc 处理）
-crtscts禁用 RTS/CTS 硬件流控
-ixon -ixoff禁用软件流控（XON/XOFF）

6.2 setserial — UART 硬件参数管理

setserial 允许获取或设置底层 UART 硬件参数（IRQ、IO 地址、UART 类型），通过 TIOCGSERIAL / TIOCSSERIAL ioctl 实现。

setserial -gCLI

显示所有 ttyS* 设备信息

setserial -g /dev/ttyS*

setserial 输出SYS

典型输出示例

/dev/ttyS0, UART: 16550A,
  Port: 0x03f8, IRQ: 4
/dev/ttyS1, UART: 16550A,
  Port: 0x02f8, IRQ: 3
/dev/ttyS2, UART: unknown,
  Port: 0x03e8, IRQ: 4
💡 提示
UART: unknown 说明该端口未被内核检测到或不存在真实硬件，此时端口不可用。可通过 /proc/tty/driver/serial 交叉验证。

6.3 终端仿真器

minicom终端

功能完整的串口终端，支持 Zmodem 文件传输

# 以 115200 8N1 连接
minicom -b 115200 -D /dev/ttyUSB0
# 配置界面
minicom -s

picocom终端

轻量级串口终端，嵌入式开发首选

6.4 socat — 高级串口桥接与调试

socat 是功能强大的双向字节流工具，可创建虚拟串口对、串口转网络等。

📖 ArchWiki 原文
"socat is a command line based utility that establishes two bidirectional byte streams and transfers data between them." — ArchWiki: Working with the serial console

socat 连接串口CLI

直接与串口交互（stdin/stdout）

socat - /dev/ttyACM0,\
  raw,echo=0,b115200

虚拟串口对调试

创建两个互连的虚拟 PTY（本地测试）

socat -d -d \
  pty,raw,echo=0 \
  pty,raw,echo=0 &

串口转 TCPCLI

将串口数据桥接到网络（远程调试）

socat TCP-LISTEN:4000,fork \
  /dev/ttyUSB0,b115200,raw

6.5 dmesg / 内核日志

内核在初始化时打印串口探测结果，是确认 UART 是否被识别的最早入口。

dmesg 串口过滤SYS

查看串口相关内核消息

dmesg | grep -i "serial\|tty\|uart"

USB 串口热插SYS

实时监控 USB 转串口设备插拔

dmesg -w | grep "ttyUSB\|ftdi\|cp210"

典型 dmesg 输出SYS

16550A 初始化信息

[ 0.547] Serial: 8250/16550 driver
[ 0.547] 0000:00:16.3: ttyS4
    at MMIO 0xfe03e000
    (irq=23, base_baud=115200)
    is a 16550A
💡 内核参数提示
可以使用 earlyprintk=serial,ttyS0,115200 内核启动参数在 console 初始化之前就通过串口输出早期内核消息，这对排查启动故障非常有效（参见 Kernel Parameters 文档）。

📝 章节总结 · 调试方法论

  • 第一步：硬件识别 — dmesg | grep tty + setserial -g /dev/ttyS*，确认内核是否探测到 UART 硬件。
  • 第二步：参数验证 — stty -F /dev/ttyS0 -a + cat /proc/tty/driver/serial，确认波特率/数据位匹配，检查 tx/rx 计数是否在增长。
  • 第三步：通信测试 — 使用 picocom 或 socat 的虚拟串口对进行环回（loopback）测试。
  • 第四步：深度诊断 — /sys/class/tty/ttyS0/ 属性检查；必要时使用 ftrace 追踪内核函数调用。
  • 第一性原理：所有调试工具本质上都是通过 termios ioctl（stty/setserial/minicom）或 sysfs/proc 文件接口（cat /sys/...）来读写内核串口驱动状态。理解层级架构后，调试路径自然清晰。
