🔌 Linux ss 命令深度指南

←→ 翻章
前言

概述：ss 是什么，为何取代 netstat？

ss（Socket Statistics）是 Linux 系统上用于转储套接字统计信息的命令行工具，属于 iproute2 工具集的一部分。它的功能与经典的 netstat 类似，但在速度、信息丰富度和内核接口上全面超越后者。

📖 官方定义

来自 man7.org ss(8) 手册页（2026-05-24）：
"ss is used to dump socket statistics. It allows showing information similar to netstat. It can display more TCP and state information than other tools."

核心差异：Netlink vs /proc

netstat 通过解析 /proc/net/tcp、/proc/net/udp 等伪文件系统文本文件获取数据；而 ss 直接通过内核 Netlink 套接字（AF_NETLINK / SOCK_DIAG_BY_FAMILY）与内核通信，绕过文本解析层。

图1：netstat 与 ss 的数据获取路径对比

功能对比表

📌 本章小结（第一性原理）

套接字（Socket）是操作系统内核中连接网络通信两端的抽象端点。内核维护一张"套接字表"，记录每个连接的状态、缓冲区使用、协议参数等。ss 的本质是：通过 Netlink 这条内核高速公路，直接读取内核的套接字表，而不是绕路通过 /proc 的文本表示。这是它一切优势的根源。

第二章

TCP 状态过滤与 TCP 状态机原理

来源：ss(8) 手册页 · RFC 793 - TCP 标准

TCP 状态机（RFC 793）

TCP 连接在其生命周期中经历一系列有限状态。ss 可精确过滤任意状态组合。

图2：TCP 状态机（基于 RFC 793）

所有可用状态标识符

来自 ss(8) STATE-FILTER 章节：

标识符含义触发场景
established连接已建立，数据可传输正常活跃连接
syn-sent已发 SYN，等待 SYN-ACK主动连接中
syn-recv收到 SYN 并已回 SYN-ACK，等待 ACK三次握手中（服务端）
fin-wait-1已发 FIN，等待 ACK 或对端 FIN主动关闭方
fin-wait-2FIN 已被确认，等待对端 FIN主动关闭方
time-wait等待 2×MSL，确保对端收到最终 ACK主动关闭方，约 60s
closed连接不存在（无 TCB）-
close-wait收到对端 FIN，等待本地应用关闭被动关闭方
last-ack发出 FIN 等待最终 ACK被动关闭方末态
listening监听中（等待连接请求）服务端端口
closing双方同时关闭，等待 ACK罕见
all所有状态-
connected除 listening 和 closed 外所有状态-
synchronizedconnected 中除 syn-sent 外-
bucket以 mini-socket 维护的状态：time-wait + syn-recv-
bigbucket 的对立面-

状态过滤示例

bash — 官方示例及扩展
# 查看所有已建立的 TCP 连接
ss -t state established

# 统计 TIME-WAIT 连接数（高并发短连接服务常见）
ss -Htn state time-wait | wc -l

# 查看 CLOSE-WAIT（通常表示应用未及时关闭连接——可能是 bug）
ss -tn state close-wait

# 查看 SYN-RECV（数量突增可能是 SYN Flood 攻击）
ss -Htn state syn-recv | wc -l

# 官方示例：显示所有 FIN-WAIT-1 的 HTTP/HTTPS 到特定网段，含计时器
ss -o state fin-wait-1 '( sport = :http or sport = :https )' dst 193.233.7/24

# 排除某个状态（exclude 关键字）
ss -t exclude time-wait

TIME-WAIT 原理深析

RFC 793 规定：主动关闭方在发出最终 ACK 后，必须在 TIME-WAIT 状态等待 2×MSL（Maximum Segment Lifetime，Linux 默认约 60 秒）。原因：

  1. 若最终 ACK 丢失，对端会重传 FIN，TIME-WAIT 允许重发 ACK；
  2. 防止旧连接的延迟数据包干扰新连接（相同四元组）。
⚠️ CLOSE-WAIT 积压告警

若 ss -tn state close-wait | wc -l 持续增大且不减少，几乎可以断定是应用层 Bug：对端已关闭连接（发了 FIN），但本地应用程序没有调用 close()，导致套接字永远停在 CLOSE-WAIT。

📌 本章小结（第一性原理）

TCP 是有状态协议，每个连接都是一台有限状态机。ss 可直接查询内核中每个 socket 的当前状态，这使得"连接卡在哪里"变得可观测。理解状态机是排查网络问题的基础：ESTABLISHED = 正常；TIME-WAIT 大量 = 短连接高吞吐；CLOSE-WAIT 累积 = 应用不关闭连接；SYN-RECV 爆炸 = 可能遭受攻击。

第三章

表达式过滤（EXPRESSION / FILTER）

来源：ss(8) EXPRESSION 章节

ss 支持强大的过滤表达式（FILTER），由一组谓词（predicates）通过布尔运算符组合而成。

语法规则

语法
ss [options] [ state STATE-FILTER ] [ EXPRESSION ]

# 布尔运算符（优先级从低到高）
or  (也可写 | 或 ||)
and (也可写 & 或 &&)
not (也可写 !)

# 谓词种类
{dst|src} [=] HOST          # 源/目的地址匹配
{dport|sport} [OP] PORT     # 源/目的端口比较，OP 可为 < <= = != >= >
dev [=|!=] DEVICE           # 网络设备匹配
fwmark [=|!=] MASK          # fwmark 值匹配
cgroup [=|!=] PATH          # cgroup 路径匹配
autobound                   # 源地址/路径是自动分配的

HOST 语法

来自 ss(8) HOST SYNTAX 章节：

通用格式：[FAMILY:]ADDRESS[:PORT]

  • inet/inet6：IP 地址（支持 CIDR 前缀）或 DNS 主机名，IPv6 地址须用 [ 和 ] 包围
  • unix：glob 模式（fnmatch），不区分大小写，支持路径和抽象名
  • link：以太网协议名 + 设备名/索引
  • * 可作为地址或端口的通配符

官方示例与扩展

bash — 来自 ss(8) 官方手册 USAGE EXAMPLES
# 官方示例1：所有已建立的 SSH 连接
ss -o state established '( dport = :ssh or sport = :ssh )'

State   Recv-Q  Send-Q  Local Address:Port  Peer Address:Port  Timer
ESTAB   0       36      192.168.1.100:22    203.0.113.5:54321   keepalive(22min)

# 官方示例2：连接到 X server 的本地进程
ss -x src /tmp/.X11-unix/*

# 官方示例3：FIN-WAIT-1 状态的 HTTP/HTTPS 到特定网段
ss -o state fin-wait-1 '( sport = :http or sport = :https )' dst 193.233.7/24

# 按目标 IP 过滤
ss -tn dst 10.0.0.1

# 按源端口范围过滤（大于 1024 的高端口）
ss -tn sport '> 1024'

# 查看连接到 443 端口的连接，显示进程
sudo ss -tnp dport = :443

# 复合过滤：来自 192.168.1.0/24 且目标端口是 443
ss -tn 'src 192.168.1.0/24 and dport = :443'

# 过滤除 22 端口外的所有已建立连接
ss -tn state established 'not ( sport = :22 or dport = :22 )'

📌 本章小结

EXPRESSION 过滤器让 ss 具备了"精确外科手术"能力——在数万条连接中定位到你关心的那几条。其底层是 ss 在 Netlink 请求中携带过滤条件，内核侧即可过滤，避免将海量数据传回用户空间再过滤，这是性能优异的关键。

拥塞控制可视化

图3：TCP 拥塞控制窗口（cwnd）变化曲线（CUBIC/Reno 示意）

实用场景：诊断网络延迟

bash
# 查看所有建立连接的 RTT，找出延迟最高的
ss -tni state established | grep rtt

# 查看拥塞窗口较小的连接（可能正在经历拥塞）
ss -tni state established | awk '/cwnd/{if($0~/cwnd:[0-9]+/){match($0,/cwnd:([0-9]+)/,a); if(a[1]+0<10) print}}'

# 查看当前 SSH 连接的 TCP 性能参数
ss -tni '( sport = :22 or dport = :22 )'

📌 本章小结（第一性原理）

TCP 拥塞控制的本质是：在不知道链路容量的情况下，通过探测丢包/ECN 动态调整发送速率。cwnd（拥塞窗口）是这个探测过程的核心变量，ssthresh 是慢启动与拥塞避免的分界线，rtt/rttvar 是判断链路质量的传感器。ss -i 直接暴露这些内核内部变量，让"黑盒"变成"白盒"。

第五章

套接字内存用量（-m）与计时器（-o）

来源：ss(8) -m --memory 和 -o --options

内存信息 (-m)

bash
ss -tm state established

State   Recv-Q  Send-Q  Local Address:Port  Peer Address:Port
ESTAB   0       0       192.168.1.100:22    203.0.113.5:54321
     skmem:(r0,rb131072,t0,tb87040,f0,w0,o0,bl0,d0)

内存格式：skmem:(r<rmem_alloc>,rb<rcv_buf>,t<wmem_alloc>,tb<snd_buf>,f<fwd_alloc>,w<wmem_queued>,o<opt_mem>,bl<back_log>,d<sock_drop>)

字段含义
r<rmem_alloc>已分配用于接收数据包的内存
rb<rcv_buf>接收缓冲区总容量上限
t<wmem_alloc>已分配用于发送（已交第三层）的内存
tb<snd_buf>发送缓冲区总容量上限
f<fwd_alloc>套接字缓存但尚未收/发的预分配内存
w<wmem_queued>已排队等待发送（尚未交第三层）的内存
o<opt_mem>套接字选项占用的内存（如 TCP MD5 密钥）
bl<back_log>sk backlog 队列占用的内存
d<sock_drop>在解多路复用前被丢弃的数据包数

图4：套接字内存字段与数据流向对应关系

计时器信息 (-o)

来自 ss(8) -o, --options，输出格式：

timer:(<timer_name>,<expire_time>,<retrans>)

timer_name含义
onTCP 重传定时器、early retrans 或 tail loss probe 定时器之一
keepaliveTCP keepalive 保活定时器
timewaitTIME-WAIT 阶段定时器
persist零窗口探测定时器（对端窗口为 0 时触发）
unknown以上均不是
bash
ss -to state established

State  Recv-Q  Send-Q  Local Address:Port  Peer Address:Port
ESTAB  0       0       192.168.1.100:22    203.0.113.5:54321  timer:(keepalive,22min,0)
ESTAB  0       204     192.168.1.100:443   10.0.0.5:32100     timer:(on,200ms,2)

# 第二行：正在重传（retrans=2），说明网络存在丢包

📌 本章小结

内存字段反映了套接字双向缓冲区的实时使用情况，sock_drop 非零意味着内核来不及处理就丢弃了数据包，通常是吞吐量过载的信号。计时器信息则是"TCP 当前在做什么"的快照：keepalive 表示连接空闲待保活，on + retrans>0 表示存在重传，persist 表示遭遇了接收方的零窗口公告（流量控制触发）。

安全上下文 (-Z/-z)

来自 ss(8)：-Z 类似 -p 但额外显示进程的 SELinux 安全上下文；-z 同时显示套接字上下文（取自关联的 inode）。

bash — 官方示例

sudo ss -t -a -Z

State   Recv-Q  Send-Q  Local Address:Port  Peer Address:Port  Process
LISTEN  0       128     0.0.0.0:22         0.0.0.0:*
        users:(("sshd",pid=1234,fd=4)) cgroup:/system.slice/sshd.service
        ctx:system_u:system_r:sshd_t:s0-s0:c0.c1023
📖 Netlink 套接字的上下文显示规则（官方文档原文）
  1. 如果有合法 PID，显示进程上下文；
  2. 如果目标是内核（pid=0），显示内核初始上下文；
  3. 如果内核或 netlink 用户分配了唯一标识符，显示 "unavailable"（通常意味着该进程有多个 netlink 套接字）。

📌 本章小结

-p 是"端口 → 进程"的反向映射。原理上，ss 通过 /proc/PID/fd/ 下的符号链接（指向 socket inode）建立套接字与进程的关联——这就是为什么需要 root 权限。-e 的 inode 可以与 /proc/net/* 或 lsof 交叉验证。-Z 在 SELinux 环境中对审计"哪个安全域的进程在监听敏感端口"极为有用。

第七章

Unix 域套接字（-x）

来源：ss(8) -x, --unix 选项

Unix 域套接字（Unix Domain Socket，UDS）是同一主机上进程间通信（IPC）的高效机制，比网络套接字快，因为数据不经过网络协议栈。

查看 Unix 套接字

bash
# 列出所有 Unix 域套接字
ss -x
# 等同于
ss -f unix

# 仅监听的 Unix 套接字
ss -xl

Netid  State   Recv-Q  Send-Q  Local Address:Port             Peer Address:Port
u_str  LISTEN  0       5       /tmp/mysql.sock 21279          * 0
u_str  LISTEN  0       128     /var/run/dbus/system_bus_socket * 0
u_dgr  UNCONN  0       0       /run/systemd/journal/syslog     * 0

# 官方示例：找到连接到 X Server 的本地进程
ss -x src /tmp/.X11-unix/*

# 查看带进程信息的 Unix 套接字
sudo ss -xlp

Netid 含义

Netid含义
u_strUnix Stream Socket（面向连接，类似 TCP）
u_dgrUnix Datagram Socket（无连接，类似 UDP）
u_seqUnix Sequential Packet Socket
nlNetlink 套接字
p_rawPacket Raw 套接字
p_dgrPacket Datagram 套接字

抽象命名空间套接字

Linux 特有的抽象命名空间（Abstract Namespace）Unix 套接字，地址以 @ 开头，不在文件系统中创建实体文件：

bash
ss -xp

Netid  State  Recv-Q  Send-Q  Local Address:Port          Peer Address:Port
u_dgr  ESTAB  0       0       @0002d 21279                * 21280
u_str  ESTAB  0       0       @/tmp/dbus-XYZ123456 24372  * 24371
# @ 前缀表示抽象命名空间套接字

📌 本章小结

Unix 域套接字的数据路径完全在内核内存中，无需网络协议栈处理（无校验和、无路由、无 IP 封装），因此延迟极低、吞吐极高。ss -x 可以揭示系统服务之间的 IPC 拓扑——如 D-Bus、数据库连接、X11、systemd 等都大量使用 Unix 套接字。

第八章

网络命名空间（-N）与容器排查

来源：ss(8) -N, --net=NSNAME

Linux 网络命名空间（Network Namespace）为每个容器/进程组提供隔离的网络栈。ss -N 允许查看指定命名空间的套接字。

使用示例

bash
# 查看指定网络命名空间的监听端口（如 Docker 容器网络）
sudo ss -N netns_name -tlnp

# 通过 ip netns 创建命名空间并检查
sudo ip netns add myns
sudo ip netns exec myns ss -tlnp

# 查看 Docker 容器的网络（先获取容器 PID）
PID=$(docker inspect --format='{{.State.Pid}}' my_container)
sudo nsenter -t $PID -n ss -tlnp

# 或使用 -N 选项（需命名空间已命名）
sudo ss -N /var/run/netns/myns -tlnp

图5：宿主机与容器各自独立的网络命名空间

📌 本章小结

网络命名空间是 Linux 容器技术（Docker/Podman/K8s）网络隔离的基础。每个命名空间有独立的路由表、iptables 规则和套接字表。ss -N 允许在不进入容器内部的情况下，从宿主机侧检查容器的网络状态，是容器化环境下的重要排查手段。

第九章

强制关闭套接字（-K）与事件监听（-E）

来源：ss(8) -K, --kill 和 -E, --events

强制关闭套接字 (-K)

来自官方文档："Attempts to forcibly close sockets. This option displays sockets that are successfully closed and silently skips sockets that the kernel does not support closing. It supports IPv4 and IPv6 sockets only."

bash
# 强制关闭所有到特定 IP 的连接（慎用！）
sudo ss -K dst 203.0.113.5

# 关闭特定端口上的异常连接
sudo ss -K src :80

# 关闭特定状态的连接（如清理 TIME-WAIT，实验性）
sudo ss -K state time-wait
🚨 危险操作警告

-K 会直接向内核发送关闭请求，可能导致正在传输的数据丢失。仅在紧急情况下（如处理僵尸连接、安全事件响应）使用，且需要 root 权限。

事件监听 (-E)

来自官方文档："Continually display sockets as they are destroyed"（持续显示被销毁的套接字）。

bash
# 持续监控 TCP 连接的销毁事件
ss -tE

Netid  State   Recv-Q  Send-Q  Local Address:Port  Peer Address:Port
tcp    CLOSE   0       0       192.168.1.100:22    203.0.113.5:54321   ← 连接断开瞬间显示
tcp    CLOSE   0       0       192.168.1.100:443   10.0.0.5:32100

# 结合 watch 监控 UDP 连接变化（-E 不支持 watch，用循环代替）
while true; do ss -uE | head -20; sleep 1; done

📌 本章小结

-K 利用的是 Linux 内核的 tcp_close() 机制（强制 RST）；-E 则利用了 Netlink 的异步通知能力——ss 订阅内核套接字事件流，在连接被销毁时实时通知，这正是 Netlink 相比轮询 /proc 的另一优势：支持事件驱动而非轮询。

第十章

实战综合场景：故障排查手册

场景一：服务器端口被占用

bash
# 问题：启动服务时报 "Address already in use"
sudo ss -tlnp sport = :8080
LISTEN  0  511  *:8080  *:*  users:(("python3",pid=12345,fd=3))
# 找到了！PID 12345 的 python3 进程占用了 8080 端口
kill 12345

场景二：连接数过多导致性能下降

bash
# 统计各状态连接数分布
ss -tan | awk 'NR>1 {print $1}' | sort | uniq -c | sort -rn
   8432 TIME-WAIT
   1204 ESTABLISHED
     56 CLOSE-WAIT
     12 SYN-RECV

# TIME-WAIT 过多：调整内核参数
# sysctl -w net.ipv4.tcp_tw_reuse=1  （谨慎使用）

# 查看连接最多的远端 IP
ss -tn state established | awk 'NR>1 {print $5}' | cut -d: -f1 | sort | uniq -c | sort -rn | head

场景三：怀疑 SYN Flood 攻击

bash
# 检查 SYN-RECV 连接数
ss -Htn state syn-recv | wc -l
# 正常应 < 100，若数万条则极可能是 SYN Flood

# 查看 SYN-RECV 的来源 IP 分布
ss -Htn state syn-recv | awk '{print $5}' | cut -d: -f1 | sort | uniq -c | sort -rn | head

# 查看 ss 统计摘要确认是否有大量半开连接
ss -s

场景四：诊断数据库连接泄漏

bash
# 查看连接到 PostgreSQL (5432) 的所有连接
sudo ss -tnp dst :5432

# 查看 CLOSE-WAIT 状态的数据库连接（应用未关闭连接的 bug）
ss -tn state close-wait dst :5432

# 查看当前连接数（与 max_connections 对比）
ss -Htn dst :5432 | wc -l

场景五：网络性能基线评估

bash
# 查看关键 TCP 连接的 RTT 和拥塞窗口，评估链路质量
ss -tni state established dport = :443

# 找出 RTT 最高（延迟最大）的连接
ss -tni state established | grep rtt | sort -t/ -k1 -n -r | head

# 查看是否有重传（retrans > 0 的连接）
ss -to state established | grep 'timer:(on'

场景六：IPv6 排查

bash
# 仅查看 IPv6 监听端口
ss -6tlnp

# 查看 IPv6 建立的连接
ss -6tn state established

# IPv6 地址在过滤器中需要用方括号
ss -6tn dst [2001:db8::1]:443

场景七：将原始数据转储到文件

bash — 来自 ss(8) -D, --diag 选项
# 将 TCP 套接字原始信息转储到文件（不显示，仅写入）
ss -D /tmp/tcp_diag.txt

# 从过滤文件读取过滤规则
cat /tmp/filter.txt
state established
dport = :443
ss -F /tmp/filter.txt -tnp

📌 综合总结（第一性原理回顾）

从第一性原理看，网络问题可归结为三类：连通性问题（能否建立连接）、状态问题（连接卡在哪个阶段）、性能问题（传输质量如何）。ss 的能力矩阵恰好覆盖这三类：

  • 连通性 → -tlnp 查看监听状态，-tn state established 确认连接建立
  • 状态问题 → state syn-recv/close-wait/time-wait 精确定位卡点
  • 性能问题 → -i 查 RTT/cwnd/重传，-m 查缓冲区，-o 查计时器

ss 的底层是 Netlink，其高效性与实时性保证了在高负载系统上诊断不会因工具本身的开销而影响被观测对象——这是优秀观测工具的核心原则。
