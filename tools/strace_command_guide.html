第零章
strace 简介与历史
了解 strace 的来历、定位与核心价值
原文：man7.org — strace(1) Linux manual page

什么是 strace？

根据 man7.org 官方手册页，strace 的定义如下：

"strace — trace system calls and signals"（追踪系统调用和信号）

在最简单的用例中，strace 运行指定的命令直到它退出，它拦截并记录进程发出的系统调用以及进程接收的信号。每个系统调用的名称、参数和返回值都会被打印到标准错误输出，或通过 -o 选项写入指定文件。

官方手册明确指出，strace 适用于以下几类人群：

  • 系统管理员、诊断人员、故障排查者：对那些没有源码的程序进行问题定位，无需重新编译即可追踪
  • 学生、黑客和好奇者：通过追踪普通程序可以学到大量关于系统和系统调用的知识
  • 程序员：系统调用和信号发生在用户态/内核态边界，对此边界的仔细审查对 bug 隔离、健全性检查和竞争条件捕捉非常有用

历史沿革

根据官方 man page 的 HISTORY 节：

  • 1991年：Paul Kranenburg 为 SunOS 编写了最初的 strace，灵感来自 SunOS 的 trace 工具
  • 1992年：Branko Lankester 将 SunOS 版本移植到 Linux，并编写了 Linux 内核支持
  • 1993年：Rick Sladkey 合并了两个版本，加入了 SVR4 的 truss(1) 特性，实现了跨平台支持
  • 1996–2002年：Wichert Akkerman 维护，迁移至 CVS，增加 FreeBSD 和多架构支持
  • 2002年起：Roland McGrath 接手维护，增加 AMD64、s390x 等支持，迁移至 Git
  • 2009年至今：Dmitry Levin 积极维护。2014 年加入栈追踪打印；2016 年实现系统调用篡改（tampering）

输出格式快速预览

官方手册给出的典型输出示例（追踪 cat /dev/null）：

open("/dev/null", O_RDONLY) = 3

每一行包含：系统调用名 → 括号内参数 → = 返回值。错误情况：

open("/foo/bar", O_RDONLY) = -1 ENOENT (No such file or directory)

信号输出示例（追踪并中断 sleep 666）：

sigsuspend([] 
--- SIGINT {si_signo=SIGINT, si_code=SI_USER, si_pid=...} ---
+++ killed by SIGINT +++
章节总结

strace 是一个无需源码、无需重编译即可观测进程行为的强力工具，诞生于 1991 年，历经多位维护者迭代，如今已支持数十种 Linux 架构。它的核心价值在于：将操作系统与用户程序之间那道不可见的"系统调用边界"完全透明化。

第一性原理

程序做任何有影响力的事情（读写文件、网络通信、创建进程、分配内存），必须通过系统调用进入内核。strace 监控的正是这一唯一必经之路，因此它能在不修改程序的前提下看到几乎所有行为。

第一章
底层原理：ptrace 机制
strace 如何在不修改程序的情况下拦截系统调用
原文：man7.org — ptrace(2) Linux manual page 原文：man7.org — strace(1) Linux manual page（SEE ALSO 节）

ptrace 系统调用

strace 的核心依赖是 Linux 内核提供的 ptrace(2) 系统调用。官方手册定义：

"The ptrace() system call provides a means by which one process (the 'tracer') may observe and control the execution of another process (the 'tracee'), and examine and change the tracee's memory and registers."

ptrace 主要用于实现断点调试器（如 gdb）和系统调用追踪器（如 strace）。

追踪的两种方式

方式一：从启动开始追踪（strace command）

strace 执行 fork() 创建子进程，子进程调用 ptrace(PTRACE_TRACEME, ...) 将自身标记为"被追踪"状态，随后执行 execve() 启动目标程序。内核检测到该进程处于被追踪状态后，在每次系统调用入口和出口处暂停它。

方式二：附加到已运行进程（strace -p PID）

strace 使用 ptrace(PTRACE_ATTACH, pid, ...) 或 PTRACE_SEIZE 附加到目标进程，目标进程收到 SIGSTOP 后暂停，strace 取得控制权。

系统调用拦截的完整流程

关键 ptrace 请求

ptrace 请求作用
PTRACE_TRACEME子进程声明自己被追踪（用于启动时追踪）
PTRACE_ATTACH附加到已运行进程，发送 SIGSTOP
PTRACE_SEIZE附加但不暂停（较新的接口）
PTRACE_GETREGS读取 tracee 的通用寄存器（含系统调用号、参数）
PTRACE_PEEKDATA读取 tracee 内存中的数据（用于读取字符串参数）
PTRACE_SYSCALL让 tracee 继续运行，在下一个系统调用入口或出口停止
PTRACE_CONT让 tracee 继续运行，不在系统调用处停止
PTRACE_DETACH解除附加，tracee 正常运行

x86-64 系统调用参数传递

在 x86-64 架构下，系统调用参数通过寄存器传递，strace 通过 PTRACE_GETREGS 读取：

寄存器含义
rax系统调用号（入口时）/ 返回值（出口时）
rdi第 1 个参数
rsi第 2 个参数
rdx第 3 个参数
r10第 4 个参数
r8第 5 个参数
r9第 6 个参数

seccomp-bpf 加速

官方手册指出，strace 支持 --seccomp-bpf 选项：利用内核的 seccomp BPF 过滤器，只让需要追踪的系统调用触发 ptrace 停止，大幅减少不必要的进程切换，降低性能开销。

⚠ 性能影响
官方手册明确指出："A traced process runs more slowly than a non-traced one."（被追踪进程比正常进程运行得更慢）。每次系统调用需要两次上下文切换（入口 + 出口），在系统调用密集型程序中可能造成数十倍的速度下降。
章节总结

strace 通过 ptrace(2) 系统调用实现非侵入式追踪：内核在每个系统调用的"入口"和"出口"处暂停 tracee，strace 读取寄存器获取调用号和参数，解码后打印，再让 tracee 继续运行。

第一性原理

ptrace 本质上是"让内核成为两个进程之间的中间人"：被追踪进程每次越过用户态/内核态边界时，内核都会把它"冻结"并通知 tracer，tracer 因此能在系统调用真正执行前后完全观测甚至修改其参数和返回值。

第二章
基本追踪与输出
最基础的 strace 用法：追踪命令的所有系统调用
原文：man7.org — strace(1) DESCRIPTION 节

场景：追踪一个命令的全部系统调用

这是 strace 最基本的用法，直接在命令前加上 strace：

strace ls -l /tmp

输出示例（摘自官方手册）：

execve("/bin/ls", ["ls", "-l", "/tmp"], 0x7fff... /* 21 vars */) = 0
openat(AT_FDCWD, "/etc/ld.so.cache", O_RDONLY|O_CLOEXEC) = 3
fstat(3, {st_mode=S_IFREG|0644, st_size=...}) = 0
lstat("/dev/null", {st_mode=S_IFCHR|0666, st_rdev=makedev(0x1, 0x3), ...}) = 0
write(1, "total 0\n"..., 8) = 8
+++ exited with 0 +++

输出格式详解

官方手册描述了输出格式的每一个细节：

格式元素说明示例
系统调用名内核系统调用的名称openat
参数列表括号内，C 语言风格，自动解码标志位O_RDONLY|O_CLOEXEC
= 返回值成功时为非负数，失败时为 -1= 3
errno 符号失败时附加的错误码符号名ENOENT
错误描述括号内的人类可读错误描述(No such file or directory)
结构体展开指针参数自动解引用并展示成员{st_mode=S_IFCHR|0666,...}

字符串截断控制 -s

默认只显示 32 个字节的字符串（超出部分显示 ...），使用 -s 可调整：

# 显示更长的字符串内容（最多 1024 字节）
strace -s 1024 curl https://example.com

打印非 ASCII 字符串

根据官方手册，-x 和 -xx 选项控制字符串打印方式：

# -x：非 ASCII 字节以十六进制显示
strace -x cat /etc/hostname
# -xx：所有字节都以十六进制显示
strace -xx cat /etc/hostname

详细结构体输出 -v

官方手册说明：默认情况下，结构体参数会被适当缩写（用 ...）。-v 选项打印完整结构体（等效于 -e abbrev=none）：

strace -v ls /

不加 -v：stat("/tmp", {st_mode=S_IFDIR|0777, ...}) = 0
加 -v：展示 stat 结构体的全部字段，包括 atime、mtime、ctime 等。

显示文件描述符路径 -y

官方手册 -y 选项：打印与文件描述符参数关联的路径名；-yy 额外显示套接字的 IP:port 信息：

strace -y cat /etc/passwd
# 输出中 fd 3 会显示为 3(/etc/passwd)
章节总结

strace 的基本用法极为简洁——只需在命令前加 strace。输出的每一行就是一次系统调用的完整快照：名称、参数（自动解码为人类可读形式）、返回值。通过 -s、-v、-y 等选项可以控制信息详细程度。

第三章
附加到运行中进程
使用 -p 选项追踪已在运行的进程，无需重启
原文：man7.org — strace(1) Startup 节：-p/--attach 选项

场景：诊断生产环境中的运行进程

官方手册对 -p 选项的描述：

"Attaches to the process with the process ID pid and begin tracing. The trace may be terminated at any time by a keyboard interrupt signal (CTRL-C). strace will respond by detaching itself from the traced processes, leaving them to continue running."

基本附加

# 找到目标进程的 PID
pidof nginx
12345

# 附加到该进程（需要 root 或适当权限）
strace -p 12345
# 输出：
Process 12345 attached
epoll_wait(5, ...) = 1
accept4(6, {sa_family=AF_INET, sin_port=htons(52341),...}, ...) = 7

同时附加多个进程

官方手册明确支持多 -p 选项，也支持逗号或空格分隔的 PID 列表：

# 方式 1：多个 -p 选项
strace -p 1234 -p 5678

# 方式 2：利用 pidof/pgrep 一次附加同名所有进程
strace -p "$(pidof nginx)"
# 等价于
strace -p "$(pgrep nginx)"

--daemonize 选项：减少对被追踪进程的影响

官方手册描述了三级 daemon 化选项，用于降低 strace 对 tracee 父子关系的影响：

选项等价形式效果
-D--daemonize=grandchildtracer 作为 tracee 的孙子进程运行，tracee 保持为调用进程的直接子进程
-DD--daemonize=pgrp在独立进程组中运行，防止被发给整个进程组的 kill 信号终止
-DDD--daemonize=session"真正的守护进程化"，在独立会话中运行，会话终止时 strace 不受影响
💡 使用场景
当你需要在不重启服务的情况下诊断它——比如 nginx 突然 CPU 飙高、Redis 响应变慢——附加模式是最直接的手段。按 Ctrl-C 后，strace 会自动解除附加，进程继续正常运行。

实战示例：诊断进程"卡住"的原因

# 发现某程序长时间无响应，附加并过滤阻塞性调用
strace -p $(pgrep myapp) -e trace=futex,epoll_wait,select,poll,read

# 如果输出显示持续的：
futex(0x7f..., FUTEX_WAIT_PRIVATE, 0, NULL) = ?
# → 说明进程在等待互斥锁，可能发生了死锁
章节总结

strace -p PID 实现了对运行中进程的"零停机诊断"。官方支持同时附加多个进程，也支持通过 -D/-DD/-DDD 减少追踪对进程关系的副作用。Ctrl-C 退出后，tracee 不受影响继续运行。

第四章
过滤系统调用
使用 -e 表达式精确控制追踪范围，聚焦关键调用
原文：man7.org — strace(1) Filtering 节：-e trace 选项

-e 表达式的完整格式

官方手册定义的表达式格式为：

[qualifier=][!]value[,value]...

其中 qualifier 可为：trace（默认）、abbrev、verbose、raw、signal、read、write、fault、inject、status 等。

按名称过滤

# 只追踪 open 系统调用（两种等价写法）
strace -e open ls
strace -e trace=open ls

# 追踪多个特定调用
strace -e trace=open,close,read,write ls

# 排除某个调用（! 取反）
strace -e trace=\!open ls

按分类过滤（官方预定义分类）

官方手册定义了以下系统调用类别：

分类关键字追踪范围示例场景
%file所有以文件名为参数的调用（open、stat、chmod...）查看程序访问了哪些文件
%process进程生命周期相关（fork、exec、wait、exit）观察进程创建/销毁流程
%network / %net所有网络相关调用（socket、bind、connect...）网络行为分析
%signal所有信号相关调用（sigaction、kill...）信号处理调试
%ipcIPC 相关（shmget、msgsnd、semop...）共享内存、消息队列分析
%desc文件描述符相关（read、write、close、dup...）I/O 操作分析
%memory内存映射相关（mmap、munmap、mprotect...）内存分配分析
%creds读写用户/组 ID 或 capability 的调用权限相关调试
# 只追踪文件相关系统调用
strace -e trace=%file python3 app.py

# 只追踪网络相关（附加到 nginx）
strace -e trace=%network -p $(pgrep nginx | head -1)

# 只追踪内存映射相关（查看 mmap 行为）
strace -e trace=%memory ./my_program

正则表达式过滤（官方支持）

官方手册说明，-e trace=/regex 支持 POSIX 扩展正则表达式：

# 追踪所有包含 "stat" 的系统调用
strace -e trace=/stat ls

# 输出可能包含：stat、lstat、fstat、statx、newfstatat 等
lstat("/dev/null", {st_mode=S_IFCHR|0666,...}) = 0
statx(3, "", AT_STATX_SYNC_AS_STAT|AT_EMPTY_PATH, ...) = 0

按路径过滤 -P

官方手册的 -P path / --trace-path 选项：只追踪涉及指定路径的系统调用：

# 只追踪访问 /etc/passwd 的系统调用
strace -P /etc/passwd cat /etc/passwd /etc/hosts

# 输出只会显示涉及 /etc/passwd 的调用，/etc/hosts 的不会出现
openat(AT_FDCWD, "/etc/passwd", O_RDONLY) = 3
fstat(3</etc/passwd>, {st_mode=S_IFREG|0644,...}) = 0

按返回状态过滤 --status

官方手册（较新版本）支持 --status 过滤，只显示特定返回状态的调用：

# 只显示失败（返回错误）的系统调用——排错利器
strace --status=failed ./my_program 2>&1 | head -30

# 只显示成功的系统调用
strace --status=successful ./my_program
章节总结

-e trace= 过滤器是 strace 最重要的选项之一。官方提供了按名称、分类（%file/%network 等）、正则表达式三种过滤方式，以及按路径（-P）和返回状态（--status）的过滤。合理使用过滤器可以将噪音减少 90%，直击关键系统调用。

第五章
性能分析与统计
使用 -c/-C 生成系统调用统计摘要，快速定位性能瓶颈
原文：man7.org — strace(1) Statistics 节：-c/-C/-w/-S/-U 选项

场景：找出哪个系统调用消耗了最多时间

官方手册对 -c 选项的描述：

"Counts time, calls, and errors for each system call and report a summary on program exit, suppressing the regular output. This shows system time (CPU time spent in the kernel), which is independent of wall clock time." 
strace -c find /usr -name "*.conf" 2>/dev/null

典型输出：

% time     seconds  usecs/call     calls    errors syscall
------- ----------- ----------- --------- --------- ----------------
  62.45    0.301234         45      6721           getdents64
  18.32    0.088432          8     10943      8221 newfstatat
   9.11    0.043981         15      2893           openat
   6.88    0.033214          6      5678           close
   3.24    0.015630         43       365       365 open
------- ----------- ----------- --------- --------- ----------------
100.00    0.482491                 26600      8586 total

-C 选项：统计同时显示实时追踪

官方手册：-C 类似 -c，但同时打印常规输出，即一边显示每个系统调用，最后再输出统计摘要。

strace -C ls /tmp

-w 选项：用挂钟时间（wall-clock）统计

官方手册：

"-w: Summarise the time difference between the beginning and end of each system call. The default is to summarise the system time."

-w 特别适合分析 I/O 密集型程序（如涉及网络等待的调用），因为等待时间不算 CPU 时间，默认的 -c 会低估它们。

# 用 wall-clock 统计时间，更能反映 I/O 等待
strace -c -w curl https://example.com

-S 排序选项

官方手册支持 -S sortby 按不同列排序统计结果：

-S 参数排序依据
time按累计时间排序（默认）
calls按调用次数排序
errors按错误次数排序
name按系统调用名称字母序排序
# 按调用次数排序，找出调用最频繁的系统调用
strace -c -S calls ./my_program
章节总结

strace -c 将系统调用追踪转化为一次"性能剖析"：无需 perf/eBPF 等复杂工具，即可快速看出哪个系统调用调用最频繁、耗时最长。-w 补充了挂钟时间视角，对 I/O 等待分析不可或缺。

第一性原理

程序的每一次内核交互都有计量代价。-c 把这些代价聚合成表格，让工程师能以"经济学视角"审视程序——找出"开销最大的交易"，即为性能优化的突破口。

第六章
追踪子进程与线程
使用 -f/-ff 选项追踪 fork/clone 创建的子进程和线程
原文：man7.org — strace(1) Tracing 节：-f/--follow-forks 选项

场景：追踪多进程程序

官方手册对 -f 选项的描述：

"Traces child processes as they are created by currently traced processes as a result of the fork(2), vfork(2) and clone(2) system calls. Note that if process PID is multi-threaded, using -f -p PID attaches to all of its threads, not just the one with threadid = PID." 
# 追踪 bash 及其所有子进程
strace -f bash -c "ls && echo hello"

# 输出中每行前有 [pid XXXXX] 前缀标识进程
[pid 12345] execve("/bin/ls", [...]) = 0
[pid 12346] execve("/bin/echo", [...]) = 0

-ff 选项：每个进程输出到独立文件

官方手册：-ff 结合 -o filename，将每个进程的追踪写入 filename.PID 文件，避免多进程输出交织。

# 每个进程的追踪分别写入 trace.PID 文件
strace -ff -o /tmp/trace nginx

# 之后可查看各进程单独的输出
ls /tmp/trace.*
# → /tmp/trace.12345  /tmp/trace.12346  /tmp/trace.12347

# 合并查看（官方提供 strace-log-merge 工具）
strace-log-merge /tmp/trace

多线程场景

官方手册明确指出：当目标进程是多线程进程时，-f -p PID 会附加到其全部线程，而不仅仅是 PID 对应的主线程。

# 追踪多线程 Python 程序（asyncio/threading）
strace -f -p $(pgrep python3) -e trace=%network

# 输出会看到多个 pid 的网络调用交织出现
[pid 5001] epoll_wait(3, ...) = 1
[pid 5002] recv(7, ...) = 1024
[pid 5003] send(8, ...) = 512

--detach-on=execve（-b execve）

官方手册描述的 -b syscall/--detach-on 选项：当被追踪进程执行到指定系统调用时，自动解除附加。当前只支持 execve：

# 追踪多线程程序但不追踪其子进程（用于降低复杂度）
strace -f --detach-on=execve make -j4
章节总结

-f 让 strace 自动追踪 fork/vfork/clone 创建的所有子进程和线程，是调试守护进程、Web 服务器、Make 构建等多进程程序的必备选项。-ff -o 组合可将各进程输出分文件保存，保持可读性。

第七章
文件与网络追踪
深度追踪文件访问行为和网络通信行为
原文：man7.org — strace(1) -e trace=%file / %network 节 参考：SUSE Linux Enterprise 调优文档 — strace 章节

场景一：追踪程序访问的所有文件

%file 类别涵盖所有以文件名为参数的系统调用（open、stat、chmod、unlink 等）：

# 看 nginx 启动时读取了哪些配置文件
strace -e trace=%file nginx -t 2>&1 | grep "openat\|open("

# 典型输出：
openat(AT_FDCWD, "/etc/nginx/nginx.conf", O_RDONLY) = 4
openat(AT_FDCWD, "/etc/nginx/conf.d/default.conf", O_RDONLY) = 5
openat(AT_FDCWD, "/etc/nginx/mime.types", O_RDONLY) = 6

只追踪失败的文件访问

# 找出程序尝试打开但不存在的文件（常见配置错误诊断）
strace -e trace=%file --status=failed ./my_app 2>&1 | grep ENOENT

openat(AT_FDCWD, "/etc/app/config.yaml", O_RDONLY) = -1 ENOENT (No such file or directory)
openat(AT_FDCWD, "/home/user/.config/app.conf", O_RDONLY) = -1 ENOENT (No such file or directory)

场景二：追踪读写内容（-e read= / -e write=）

官方手册描述 -e read=set 和 -e write=set：对指定文件描述符执行全量十六进制+ASCII dump：

"Perform a full hexadecimal and ASCII dump of all the data read from file descriptors listed in the specified set." 
# 查看 fd 3 和 fd 5 上的读取内容
strace -e read=3,5 cat /etc/hostname

# 输出（十六进制 + ASCII 双栏显示）：
 | 00000000  6d 79 73 65 72 76 65 72  0a                       |myserver.|

场景三：追踪网络相关系统调用

参考 SUSE 官方文档中的示例：

# 追踪 nginx 的网络系统调用
strace -e trace=%network -p 26520
Process 26520 attached - interrupt to quit
socket(PF_NETLINK, SOCK_RAW, 0) = 50
bind(50, {sa_family=AF_NETLINK, pid=0, groups=00000000}, 12) = 0
getsockname(50, {sa_family=AF_NETLINK, pid=26520,...}, ...) = 0
connect(4, {sa_family=AF_INET, sin_port=htons(80), sin_addr=inet_addr("93.184.216.34")}, 16) = 0

场景四：-yy 显示套接字 IP:port

官方手册的 -yy 选项：打印与套接字文件描述符关联的 IP:port 信息，使 fd 号更有意义：

strace -yy -e trace=%network curl https://example.com

# 输出中 fd 5 会显示为：
sendto(5<TCP:[127.0.0.1:54321->93.184.216.34:443]>, ...) = 512
章节总结

文件追踪（%file）是定位"程序找不到配置文件"等问题的最快方法；网络追踪（%network）可以在不用 tcpdump 的情况下看到连接建立过程；-e read= 和 -e write= 则直接展示 I/O 的原始内容，是替代 hexdump 的利器。

第八章
时间戳与延迟分析
为每个系统调用添加时间信息，精准定位延迟来源
原文：man7.org — strace(1) Output format 节：-t/-tt/-ttt/-r/-T/-w 选项

-t/-tt/-ttt：绝对时间戳

官方手册定义了三级时间戳精度：

选项格式精度示例
-tHH:MM:SS秒14:32:07 open(...
-ttHH:MM:SS.us微秒14:32:07.123456 open(...
-tttUnix 时间戳.us微秒1748534127.123456 open(...
# 带微秒时间戳，用于排查偶发性延迟
strace -tt -p $(pgrep redis-server) -e trace=%desc

# 输出：
14:32:07.012345 epoll_wait(5, ...) = 1
14:32:07.013210 read(8, "*3\r\n$3\r\nSET\r\n"..., 4096) = 24
14:32:07.019876 write(8, "+OK\r\n", 5) = 5

-r：相对时间戳（连续调用之间的间隔）

官方手册：

"Prints a relative timestamp upon entry to each system call. This records the time difference between the beginning of successive system calls." 
strace -r curl https://example.com

# 输出（每行前显示距上一次调用的间隔）：
     0.000000 execve(...) = 0
     0.001234 socket(AF_INET, ...) = 3
     0.000456 connect(3, ...) = 0
     0.124800 recv(3, ...) = 512  ← 这里等了 124ms，是网络延迟

-T：显示每个系统调用的耗时

官方手册：

"Show the time spent in system calls. This records the time difference between the beginning and the end of each system call." 
strace -T ls /var/log

# 每行末尾显示 <耗时秒数>
openat(AT_FDCWD, "/var/log", O_RDONLY|O_DIRECTORY) = 3 <0.000023>
getdents64(3, ...) = 256 <0.000178>
write(1, ...) = 512 <0.000045>

综合示例：定位慢查询

# 同时显示时间戳和每次调用耗时，找出慢系统调用
strace -tt -T -e trace=%file,read,write ./slow_app 2>&1 | \
  awk '/<0\.[0-9][0-9][0-9]/ {print}'  # 筛出耗时 > 100ms 的调用
章节总结

时间戳选项让 strace 从"行为观测器"升级为"性能诊断器"：-tt 精确到微秒的绝对时间便于与日志关联；-r 的相对间隔直接暴露"等待在哪里"；-T 的单次耗时揭示哪个调用本身最慢。三者结合，几乎可以取代轻量级的性能剖析工具。

第九章
信号追踪
观察进程接收和处理信号的完整过程
原文：man7.org — strace(1) DESCRIPTION 节（信号格式）和 -e signal 选项

strace 如何显示信号

官方手册的描述：

"Signals are printed as a signal symbol and a decoded siginfo structure. An excerpt from tracing and interrupting the command 'sleep 666' is:
sigsuspend([] <unfinished ...>
--- SIGINT {si_signo=SIGINT, si_code=SI_USER, si_pid=...} ---
+++ killed by SIGINT +++"

信号行格式：--- SIGNAL_NAME {siginfo 结构体} ---

场景：追踪程序接收的所有信号

# 追踪 sleep 进程并观察信号（另开终端 kill -SIGTERM 它）
strace -e trace=%signal sleep 100

# 正常等待时：
restart_syscall(<{tv_sec=100, tv_nsec=0}>) 

# 收到 SIGTERM 时：
--- SIGTERM {si_signo=SIGTERM, si_code=SI_USER, si_pid=9999, si_uid=1000} ---
+++ killed by SIGTERM +++

系统调用被信号中断并重启

官方手册描述了一种重要情况：可重启系统调用被信号打断的处理方式：

"The interruption of a (restartable) system call by a signal delivery is handled differently, as the kernel terminates the system call and arranges for its immediate re-execution after the signal handler completes." 
read(0, 0x7ffff72cf5cf, 1)            = ? ERESTARTSYS (To be restarted)
--- SIGALRM {si_signo=SIGALRM, si_code=SI_KERNEL} ---
rt_sigreturn({mask=[]}) = 0
read(0, "", 1) = 0

过滤特定信号

官方手册的 -e signal=set 选项，默认追踪所有信号：

# 只追踪 SIGSEGV 和 SIGBUS（调试崩溃）
strace -e signal=SIGSEGV,SIGBUS ./crashy_program

# 排除 SIGIO（I/O 频繁触发，过滤噪音）
strace -e signal=\!SIGIO -p $(pgrep myapp)

实战：追踪信号处理过程

# 追踪程序安装信号处理器的过程
strace -e trace=rt_sigaction,rt_sigprocmask ./server

# 可以看到程序注册了哪些信号处理器：
rt_sigaction(SIGTERM, {sa_handler=0x40a230, sa_mask=[TERM], sa_flags=SA_RESTORER,...}, NULL, 8) = 0
rt_sigaction(SIGINT,  {sa_handler=0x40a230, sa_mask=[INT],  sa_flags=SA_RESTORER,...}, NULL, 8) = 0
rt_sigaction(SIGHUP,  {sa_handler=0x40a350, ...}, NULL, 8) = 0
章节总结

strace 对信号的追踪揭示了进程与内核之间的异步通信机制：信号从发送方到目标进程的整个生命周期（接收 → 中断系统调用 → 执行处理器 → 恢复执行）都能被完整记录。这对调试进程无法正常退出、信号处理器逻辑错误等问题至关重要。

第十章
系统调用注入与故障模拟
使用 --inject/--fault 主动模拟系统调用失败，测试程序健壮性
原文：man7.org — strace(1) Filtering 节：--inject/--fault 选项（2016年新增）

功能简介

官方手册指出，系统调用注入（tampering）于 2016 年实现。--inject 选项可以：

  • 让指定系统调用返回错误（error injection）
  • 让指定系统调用返回自定义值（retval injection）
  • 向进程注入信号
  • 在调用前后添加延迟（delay injection）

--inject 完整语法

--inject=syscallset[:error=errno][:retval=value][:signal=sig][:delay_enter=usecs][:delay_exit=usecs][:when=expr]

场景一：模拟系统调用失败（错误注入）

# 让所有 open/openat 返回 ENOENT（测试程序的缺文件处理）
strace --inject=openat:error=ENOENT ./my_app

# 只让第 3 次及之后的 chdir 调用失败（when 语法）
strace --inject=chdir:error=ENOENT:when=3+ ./my_app

场景二：--fault 快捷方式

官方手册：--fault 是 --inject 的简写，默认使用 ENOSYS 错误码：

# 让 mmap 调用全部失败（测试内存分配失败路径）
strace --fault=mmap ./memory_heavy_app

# 等价于：
strace --inject=mmap:error=ENOSYS ./memory_heavy_app

场景三：自定义返回值注入

# 让 getpid() 始终返回 1234（欺骗程序认为自己的 PID 是 1234）
strace --inject=getpid:retval=1234 ./pid_sensitive_app

场景四：延迟注入（模拟慢 I/O）

# 在每次 read 前添加 100ms 延迟（模拟慢网络读取）
strace --inject=read:delay_enter=100ms ./network_app

# 在 write 系统调用退出后添加 50ms 延迟
strace --inject=write:delay_exit=50ms ./network_app

when 表达式语法

官方手册定义的 when 格式，格式为 first[..last][+[step]]：

when 表达式含义
3仅在第 3 次调用时注入
3..6在第 3 到第 6 次调用时注入
3+从第 3 次开始，每次都注入
1+2第 1、3、5、7... 次（步长为 2）
2..10+3第 2、5、8 次（2 到 10，步长为 3）
💡 混沌工程应用
--inject 与混沌工程（Chaos Engineering）的理念完全契合：在不修改代码的情况下，主动向运行中的程序注入各种系统级故障，验证程序在极端条件下的健壮性。
章节总结

strace 的注入能力（2016 年引入）将其从"被动观察者"变为"主动测试者"。--inject/--fault 可以在不修改程序代码的情况下，精确控制系统调用的失败时机和方式，是极为强大的健壮性测试工具。

第一性原理

所有程序都依赖系统调用成功才能正常运行。通过控制系统调用的返回值，strace 可以模拟任何"内核侧"的故障，从而在测试环境中还原生产环境中极难复现的边缘情况。

第十一章
输出到文件与格式控制
管理 strace 的输出目标、对齐方式及颜色
原文：man7.org — strace(1) Output format 节

-o：输出到文件

默认 strace 输出到 stderr，使用 -o 可重定向到文件：

# 输出到文件（不再和程序输出混在一起）
strace -o /tmp/strace.log ./my_app

# 追加模式（-A 选项，官方新版本支持）
strace -A -o /tmp/strace.log ./my_app

官方手册说明：如果 -o 的参数以 | 或 ! 开头，则作为命令管道处理：

# 直接将输出通过管道传给 grep（不影响程序自身的重定向）
strace -o '|grep ENOENT' ./my_app

-q/-qq/-qqq：抑制信息输出

官方手册定义了三级 quiet 模式：

选项抑制的内容
-q抑制 "attached/detached" 消息和 personality 变更消息
-qq在 -q 基础上还抑制进程退出状态
-qqq抑制所有可抑制消息
# 仅显示系统调用，不显示 attach/exit 提示
strace -q -p $(pgrep myapp)

-a：对齐列宽

官方手册：-a column 指定返回值对齐到第几列（默认第 40 列）：

# 将返回值对齐到第 60 列（参数较长时更美观）
strace -a 60 ls /etc

--color：彩色输出（官方 strace 7.x 新特性）

官方手册描述：

"Colorize the trace output. The default is --color=auto, which enables color only when the trace output stream is a terminal that supports at least 8 colors."

支持通过 STRACE_COLORS 环境变量自定义配色方案，可用的 key 有：syscall、argname、argval、const、comment、punct、retval、error。

# 始终启用颜色（即使重定向到文件）
strace --color=always ls | less -R

# 自定义颜色（syscall 显示黄色粗体，error 显示红色）
STRACE_COLORS="syscall=1;33:error=31" strace ls

-n：打印系统调用号

官方手册的 -n 选项（较新版本），在调用名旁边打印系统调用号：

strace -n ls
# 输出：
[ 59] execve("/bin/ls", ...) = 0
[257] openat(AT_FDCWD, ...) = 3
章节总结

strace 的输出控制选项使其输出可以适应不同的分析场景：-o 将输出与程序 I/O 分离；-q 减少干扰信息；-a 提升可读性；--color 让复杂输出一目了然。合理配置这些选项，是高效使用 strace 的必要技能。

第十二章
栈追踪与调试
使用 -k/-kk 在系统调用处打印调用栈，深入定位代码来源
原文：man7.org — strace(1) Output format 节：-k/--stack-trace 选项

-k：打印系统调用的调用栈

官方手册描述（2014 年加入）：

"-k --stack-trace[=symbol]: Prints the execution stack trace of the traced processes after each system call."
⚠ 前提条件
-k 需要 strace 编译时链接 libunwind 库。-kk（源码信息）还需要程序使用 -g（gcc）或 -g -gdwarf-aranges（clang）编译。 
# 追踪 malloc/mmap 并显示调用栈（定位内存分配来源）
strace -k -e trace=mmap,brk ./my_app

# 典型输出：
mmap(NULL, 135168, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANONYMOUS, -1, 0) = 0x7f...
 > /lib/x86_64-linux-gnu/libc.so.6(malloc+0x...)
 > /home/user/my_app(create_buffer+0x42) [0x401234]
 > /home/user/my_app(main+0x100) [0x401a00]

-kk：同时显示源码位置

# 需要程序带调试符号编译：gcc -g -o my_app my_app.c
strace -kk -e trace=write ./my_app

# 输出会包含源码文件名和行号：
write(1, "hello\n", 6) = 6
 > /lib/.../libc.so.6(write+0x14) [0x...]
 > my_app.c:23 my_app(log_message+0x2a) [0x401234]
 > my_app.c:45 my_app(process_request+0x80) [0x401400]
 > my_app.c:112 my_app(main+0x30) [0x401a20]

--stack-trace-frame-limit：限制栈帧数

官方手册（strace 7.x）的 --stack-trace-frame-limit=limit 选项，默认最多 256 帧：

# 只显示最深 5 个栈帧（避免输出过长）
strace -k --stack-trace-frame-limit=5 -e trace=open ./my_app

-i：打印指令指针

官方手册的 -i/--instruction-pointer 选项：在每个系统调用时打印指令指针（PC 寄存器值），可结合反汇编工具定位系统调用来自代码的哪一行：

strace -i ls

# 输出：
[00007f8a3b2c4a57] execve("/bin/ls", ...) = 0
[00007f8a3b2c5123] openat(AT_FDCWD, ...) = 3
章节总结

栈追踪（-k/-kk）将 strace 与调试器的能力结合：不仅知道"发生了什么系统调用"，还能看到"是代码中的哪个函数调用链触发的"。这对于大型代码库中定位意外的系统调用来源、排查第三方库的不当行为尤其有价值。

第十三章
常见实战案例
综合应用 strace 解决真实生产问题
原文：man7.org — strace(1)（基于官方手册的实战场景总结） 参考：SUSE Linux 调优指南 — strace 章节

案例一：程序启动失败，找不到原因

# 程序报错 "No such file or directory" 但不知道是哪个文件
strace -e trace=%file --status=failed ./broken_app 2>&1 | grep ENOENT

# 立刻看到缺失的文件：
openat(AT_FDCWD, "/etc/app/config.yaml", ...) = -1 ENOENT
openat(AT_FDCWD, "/usr/lib/app/plugins/auth.so", ...) = -1 ENOENT

案例二：程序"hang 住"，找出阻塞点

# 附加到卡住的进程，-T 显示每次调用的耗时
strace -p $(pgrep hung_app) -T

# 如果输出显示：
read(6, 
# → 进程在 fd 6 上等待读取（可能是 socket 或 pipe），没有数据到达

# 加 -y 参数可看到 fd 6 对应的文件/socket：
strace -p $(pgrep hung_app) -T -y
read(6<socket:[12345]>,

案例三：诊断 CPU 消耗异常

# 使用 -c 分析系统调用分布，找出 CPU 热点
strace -c -p $(pgrep cpu_heavy) --; sleep 10

# 结果显示 futex 调用占 80% 的 CPU 时间
% time     seconds  usecs/call     calls    errors syscall
  80.1    2.401234         12    200103           futex
# → 大量线程竞争互斥锁，存在锁争用问题

案例四：分析共享库加载过程

# 追踪动态链接库的搜索和加载过程
strace -e trace=openat ./my_app 2>&1 | grep ".so"

openat(AT_FDCWD, "/etc/ld.so.cache", O_RDONLY|O_CLOEXEC) = 3
openat(AT_FDCWD, "/lib/x86_64-linux-gnu/libm.so.6", ...) = 3
openat(AT_FDCWD, "/usr/lib/x86_64-linux-gnu/libssl.so.3", ...) = 3

案例五：权限问题诊断（EPERM/EACCES）

# 追踪权限相关的失败调用
strace --status=failed ./my_app 2>&1 | grep -E "EPERM|EACCES"

openat(AT_FDCWD, "/var/log/app.log", O_WRONLY|O_APPEND|O_CREAT, 0644) = -1 EACCES
# → 对 /var/log/app.log 没有写权限，需要检查文件权限或用户身份

案例六：使用 -ff 分析多进程 Web 服务器

# 追踪 Apache 的 prefork 模型，每个进程独立记录
strace -f -ff -o /tmp/apache_trace apache2 -X

# 分别查看各 worker 进程的追踪
ls /tmp/apache_trace.*
tail -f /tmp/apache_trace.12346  # 查看 worker 进程 12346 的追踪
章节总结

实战中 strace 的核心价值在于"不需要源码、不需要重编译、不需要重启服务"地回答以下问题：程序访问了哪些文件？为什么系统调用失败？卡在了哪个 I/O 操作上？哪类调用消耗了大部分 CPU？将各章节知识组合使用，几乎可以解决所有系统层面的黑盒问题。

第十四章
局限性与注意事项
了解 strace 的边界与使用陷阱，避免误判
原文：man7.org — strace(1) BUGS、NOTES、DIAGNOSTICS 节

官方已知问题（BUGS 节）

1. setuid 程序权限丢失

官方手册明确指出：

"Programs that use the setuid bit do not have effective user ID privileges while being traced."

即用 strace 追踪 sudo、passwd 等 setuid 程序时，其 EUID 不会提升，行为可能与正常运行不同。

2. 性能开销显著

官方手册：

"A traced process runs more slowly than a non-traced one. The performance impact can be mitigated by using the --seccomp-bpf option."

每次系统调用需要两次进程上下文切换（进入/退出），对系统调用密集型程序（如 I/O 密集型服务）影响尤为明显，可能导致数十倍的速度下降。

3. 子进程可能残留（CTRL-C 场景）

官方手册：

"When tracing a command, its descendant processes may be left running after strace is terminated by an interrupt signal (such as CTRL-C). This can be prevented by using the --kill-on-exit option."

4. EINTR 副作用

官方手册 NOTES 节：

"On some platforms, a process attached with the -p option may receive a spurious EINTR error from a non-restartable system call."

即 strace -p 附加时，目标进程正在执行的不可重启系统调用可能会以 EINTR 错误中断。

官方 NOTES 节的重要提醒

系统调用与 libc 函数不是一一对应的

官方手册指出：

"In some cases, a system call will differ from the documented behavior or have a different name. For example, the underlying faccessat(2) system call does not have a flags argument, and the setrlimit(2) library function is implemented using prlimit64(2) system call on modern (2.6.38+) kernels."

strace 只能观测系统调用层，不能观测用户态库函数

strace 追踪的是用户态与内核态的边界。用户态库函数（如 malloc、printf 的内部逻辑）不会出现在 strace 输出中，需要使用 ltrace 追踪库函数调用。

strace 不追踪 vDSO 系统调用

某些高频系统调用（如 gettimeofday、clock_gettime）被内核通过 vDSO（虚拟动态共享对象）在用户态实现，无需真正进入内核，因此 strace 无法追踪这些调用。

CLONE_UNTRACED 绕过

官方手册指出：

"A traced process can use the CLONE_UNTRACED flag with the clone system call to create a child process that is not traced by strace."

程序可以主动创建 strace 无法追踪的子进程，这是 --seccomp-bpf 安全保证的一个漏洞。

局限性影响场景缓解方法
性能开销大生产环境高并发服务使用 --seccomp-bpf 或 -e 过滤减少停止次数
setuid 权限丢失追踪 sudo/passwd 等程序以 root 运行 strace，或使用 setuid 安装的 strace
vDSO 调用不可见gettimeofday 等高频调用改用 perf 或 eBPF 追踪
只看系统调用层分析 libc 内部逻辑改用 ltrace 追踪库函数
EINTR 副作用-p 附加时影响目标进程附加前确认目标程序能正确处理 EINTR
章节总结

strace 是一把强大但有边界的工具。其局限性均有官方文档明确记录：性能开销是使用时必须考虑的现实约束；setuid 权限丢失是安全设计的必然结果；vDSO 和用户态库的不可见性则提示我们它只是观测层的一部分而非全貌。

第一性原理：工具的本质边界

strace 的能力边界与 ptrace 的能力边界完全重合——ptrace 只能观测"真正跨越用户态/内核态边界"的事件。凡是没有越过这条边界的事情（vDSO、纯用户态运算、库内部逻辑），strace 就无从得知。认识这条边界，是成为优秀系统工程师的必要认知。
