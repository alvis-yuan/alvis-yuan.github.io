TShark 简介与架构

🦈 什么是 TShark？

TShark 是 Wireshark 的终端版本，专为无 GUI 环境设计，用于捕获和分析网络数据包。 它支持与 Wireshark 相同的协议解析引擎，同样的显示过滤语法，以及强大的统计输出功能。

官方定位（来自 Wireshark 文档）：

"TShark is a terminal oriented version of Wireshark designed for capturing and displaying packets when an interactive user interface isn't necessary or available."

相比 tcpdump，TShark 能理解更多协议，支持多次扫描分析，并具备丰富的统计能力。

▸ TShark 核心架构与数据流
📋 两种工作模式
模式触发方式特点适用场景
实时抓包模式 -i eth0 从网络接口实时捕获，单遍扫描 生产环境监控、实时告警
离线分析模式 -r file.pcap 读取已有 pcap/pcapng，可两遍分析 取证分析、自动化处理
💡 提示
读取文件时（-r）支持使用 -2 两遍扫描，可填充"response in frame #"等前向引用字段。 实时抓包不支持两遍扫描（无法 seek 回流）。
🚀 最快速上手 
# 查看版本
tshark -v

# 列出所有可用网络接口
tshark -D

# 最简单的实时抓包（Ctrl+C 停止）
sudo tshark -i eth0

# 读取 pcap 文件并显示
tshark -r capture.pcap

▸ 接口选择与抓包流程
🔌接口相关命令
选项说明示例
-D列出所有可用网络接口并退出tshark -D
-i <接口>指定捕获接口（名称或编号）tshark -i eth0 / -i 1
-i any在所有接口上捕获（仅 Linux）tshark -i any

📏抓包大小与停止条件
# -s: 快照长度，默认抓取完整包（适当最大值）
sudo tshark -i eth0 -s 96   # 只抓头部，节省空间

# -c: 最多抓 N 个包后停止
sudo tshark -i eth0 -c 100

# -a: 自动停止条件（duration/filesize/files/packets）
sudo tshark -i eth0 -a duration:30       # 抓 30 秒
sudo tshark -i eth0 -a filesize:10240    # 文件达 10 MB 停止
sudo tshark -i eth0 -a packets:1000      # 匹配过滤器的包达 1000 停止

# -B: 内核缓冲区大小（MiB），丢包时调大
sudo tshark -i eth0 -B 64   # 64 MiB 缓冲区

# -r: 读取 pcap 文件（支持 stdin: -r -）
tshark -r capture.pcapng
cat capture.pcap | tshark -r -
⚠️ 注意
-s 0 在某些版本表示"最大快照长度"（65535 或系统最大），而新版本 TShark 默认即为适当最大值。 在高速网络环境中遇到丢包时，优先检查 -B 缓冲区大小。
💾写入文件
# -w: 写入 pcapng 文件（默认格式）
sudo tshark -i eth0 -w capture.pcapng

# -F: 指定输出文件格式
sudo tshark -i eth0 -w out.pcap -F pcap

# 列出所有支持的文件格式
tshark -F

# -P: 写文件的同时在终端打印摘要
sudo tshark -i eth0 -w capture.pcap -P

# --capture-comment: 添加文件注释
sudo tshark -i eth0 -w out.pcapng --capture-comment "生产故障排查 2024-01"

📖 第 3 章 · 基础入门
抓包过滤器（Capture Filter -f）
导读：抓包过滤器是"关口前移"——在数据进入 TShark 处理之前就将不需要的流量丢弃， 基于 BPF（Berkeley Packet Filter）语法，与 tcpdump 完全兼容。 用好它能大幅降低 CPU 开销和存储压力，但它的能力不如显示过滤器丰富。
📄 原文链接：tshark.dev Capture Filters 📄 原文链接：pcap-filter BPF 语法 man page
▸ 抓包过滤器 vs 显示过滤器
🔏BPF 捕获过滤器语法

捕获过滤器使用 BPF（libpcap）语法，用 -f 指定：

# 基本语法结构：
# [协议] [方向] [类型] [值]

# 按协议
tshark -i eth0 -f "tcp"
tshark -i eth0 -f "udp"
tshark -i eth0 -f "icmp"
tshark -i eth0 -f "arp"

# 按端口
tshark -i eth0 -f "port 80"
tshark -i eth0 -f "dst port 443"
tshark -i eth0 -f "portrange 8000-9000"

# 按 IP 地址
tshark -i eth0 -f "host 192.168.1.1"
tshark -i eth0 -f "src host 10.0.0.1"
tshark -i eth0 -f "net 192.168.0.0/24"

# 按网络
tshark -i eth0 -f "net 10.0.0.0/8"

# 组合（and / or / not）
tshark -i eth0 -f "icmp or tcp port 80"
tshark -i eth0 -f "tcp and not port 22"
tshark -i eth0 -f "host 192.168.1.1 and (port 80 or port 443)"

# 使用预定义过滤器（在 Wireshark GUI 中保存的）
tshark -i eth0 -f "predef:MyPredefinedFilter"
📊捕获过滤器 vs 显示过滤器 对比
特性捕获过滤器 (-f)显示过滤器 (-Y)
语法BPF / libpcapWireshark 显示过滤器语法
工作阶段捕获前（内核层）解析后（用户态）
性能极高一般
协议语义有限（仅字节级）完整（3000+ 字段）
能否读文件❌ 不能✅ 能
正则支持❌ 否✅ 是（matches）
典型用途减少抓包噪音，降低负载精确条件分析，字段查询
💡 最佳实践
先用 -f 粗筛（比如只抓 TCP），再用 -Y 精过滤（比如 HTTP 错误响应）。 两者组合使用可最大化效率。

🔍 第 4 章 · 过滤与分析
显示过滤器（-Y / -R）
导读：显示过滤器是 Wireshark/TShark 最强大的特性之一。它在协议解析之后工作， 可以访问任何协议的任何字段，支持比较运算、逻辑组合、正则匹配，甚至专家信息判断。 理解其语法结构，是精准抓取数据包的关键。

▸ 显示过滤器语法结构
🌐Layer 2-4 基础过滤

以下基于 tshark.dev 官方文档 的字段表格：

方向以太网 ethIP ipTCP tcpUDP udp
源eth.srcip.srctcp.srcportudp.srcport
目的eth.dstip.dsttcp.dstportudp.dstport
任意eth.addrip.addrtcp.portudp.port
# MAC 地址过滤
tshark -r cap.pcap -Y 'eth.src == 00:11:22:33:44:55'

# IP 地址过滤（双向）
tshark -r cap.pcap -Y 'ip.addr == 10.0.0.1'

# 端口过滤
tshark -r cap.pcap -Y 'tcp.dstport != 80'

# 协议过滤
tshark -r cap.pcap -Y 'icmp'
tshark -r cap.pcap -Y 'http'
tshark -r cap.pcap -Y 'dns'
⚙️逻辑运算符与组合
# and / && : 与
tshark -r cap.pcap -Y 'ip.addr == 8.8.8.8 and !(icmp or dns)'

# or / || : 或
tshark -r cap.pcap -Y 'http or dns'

# not / ! : 非
tshark -r cap.pcap -Y 'not arp'

# 括号分组
tshark -r cap.pcap -Y 'ip.addr==192.168.1.1 and (tcp.port==80 or tcp.port==443)'

# HTTP 响应状态码
tshark -r cap.pcap -Y 'http.response.code == 200'
tshark -r cap.pcap -Y 'http.response.code >= 400'

# DNS 查询
tshark -r cap.pcap -Y 'dns.flags.response == 0'  # 仅 DNS 请求
tshark -r cap.pcap -Y 'dns.flags.rcode != 0'     # DNS 错误响应
🔎两遍分析（-2 -R -Y）

官方文档（tshark.dev）说明：

"If you would like to optimize display filtering over 2 passes, you can specify the first and second with -R <filter> -2 -Y <2nd filter>." 
# 两遍分析：先粗筛 arp，再精确定位第 5 帧
tshark -r large.pcapng -R "arp" -2 -Y "frame.number == 5"

# 两遍比直接管道更快（实测约快 1.6 倍）
# 仅支持文件读取模式（-r），不支持实时抓包
💡 使用场景
两遍分析适合需要"前向引用"字段的场景，例如 frame.number、tcp.analysis.retransmission 等需要知道后续包才能确定的字段。
🔍查找协议字段
# -G: 输出所有协议和字段注册表
tshark -G | grep -E "sec_websocket_version"
# 输出: F  Sec-WebSocket-Version  http.sec_websocket_version  FT_STRING  ...

# 查找 http.response 的所有子字段
tshark -G | grep -E "http\.response\."

# 搜索特定协议的所有字段
tshark -G fields | grep "^F" | grep "dns\."

字段提取与输出格式（-T -e -E）
导读：TShark 的输出格式高度可定制——从人类可读的文本到机器友好的 JSON， 再到 CSV 风格的字段提取。本章是自动化流水线的核心：如何把 TShark 的分析结果 以正确的格式送入下游工具（awk、jq、Python 等）。
📄 原文链接：tshark(1) 官方 man page - Output 选项
📤-T 输出格式选项
格式说明适用场景
text默认人类可读文本（一行摘要）终端查看
fields按 -e 指定字段输出，Tab 分隔脚本处理、CSV
jsonJSON 格式，含所有解析字段jq 处理、API 接入
jsonrawJSON 格式，原始字节值底层调试
pdmlPacket Details XML 格式工具互操作
psmlPacket Summary XML 格式摘要导出

# JSON 输出（配合 jq 使用）
tshark -r cap.pcap -T json | jq '.[0]._source.layers.ip'

# 仅输出特定协议层的 JSON（-j 过滤子节点，-J 展开子节点）
tshark -r cap.pcap -T json -j "http tcp ip"
tshark -r cap.pcap -T json -J "http tcp"

# EK 格式输出（适合 Elasticsearch）
tshark -r cap.pcap -T ek > packets.ndjson
🎯-T fields 字段提取（最常用）
# 基础用法：提取 IP 源/目的和端口
tshark -r cap.pcap -T fields \
    -e frame.number \
    -e ip.src \
    -e ip.dst \
    -e tcp.port

# 加上表头
tshark -r cap.pcap -T fields \
    -e ip.src -e ip.dst -e tcp.dstport \
    -E header=y

# 逗号分隔（CSV）
tshark -r cap.pcap -T fields \
    -e ip.src -e ip.dst \
    -E separator=, -E quote=d

# 提取 HTTP 信息列（使用列类型）
tshark -r cap.pcap -T fields \
    -e frame.number \
    -e _ws.col.info

# 提取 DNS 查询名称和响应码
tshark -r cap.pcap -Y 'dns' -T fields \
    -e dns.qry.name \
    -e dns.flags.rcode \
    -E header=y -E separator=,

# 以 hex 格式输出字段（字段名前加 @）
tshark -r cap.pcap -T fields -e @ip.src
⚙️-E 字段格式控制选项
选项说明可选值
header=y|n输出字段名表头默认 n
separator=/t|/s|<char>字段分隔符/t=Tab（默认）, /s=空格
occurrence=f|l|a多值字段取第几个f=第一, l=最后, a=全部（默认）
aggregator=,|/s|<char>多值聚合符,（默认）
quote=d|s|n值引号样式d=双引号, s=单引号, n=无（默认）
bom=y|n输出 UTF-8 BOM默认 n
escape=y|n转义控制字符默认 y
🎨其他输出控制
# -V: 显示完整数据包详情（Packet Details）
tshark -r cap.pcap -V

# -O: 只显示指定协议的详情
tshark -r cap.pcap -O http,tcp

# -x: hex + ASCII dump
tshark -r cap.pcap -Y 'http' -x

# --hexdump: 更精细的 hex 控制（4.x 新增）
tshark -r cap.pcap --hexdump noascii

# -t: 时间戳格式
tshark -r cap.pcap -t a     # 绝对时间
tshark -r cap.pcap -t r     # 相对时间（默认）
tshark -r cap.pcap -t e     # Unix epoch
tshark -r cap.pcap -t ud    # UTC with date

# -n: 禁用名称解析（显示原始 IP/端口号）
tshark -r cap.pcap -n

# --color: 彩色输出（需终端支持 24-bit 色彩）
tshark -r cap.pcap --color

# --export-objects: 导出协议对象（如 HTTP 文件）
tshark -r cap.pcap --export-objects http,/tmp/http_files
tshark -r cap.pcap --export-objects help  # 列出支持的协议

🔍 第 6 章 · 过滤与分析
统计分析（-z）
导读：-z 是 TShark 的统计分析引擎，能在不逐包查看的情况下快速提炼关键指标： 协议分布、会话列表、响应时间、IO 速率、DNS/HTTP/SIP/SMB 协议专项统计…… 配合 -q（安静模式）使用，是自动化监控脚本的利器。
📄 原文链接：tshark(1) man page -z 章节 📄 原文链接：Oracle/tshark -z 详细说明
▸ -z 统计功能分类
📊IO 统计与协议层级
# 列出所有可用的 -z 统计类型
tshark -z help

# 协议层级统计（最常用！）
tshark -r cap.pcap -q -z io,phs
# 输出示例:
# ===Protocol Hierarchy Statistics===
# Filter:
# eth                      frames:1234 bytes:567890
#  ip                      frames:1200 bytes:560000
#   tcp                    frames:900 bytes:500000
#    http                  frames:200 bytes:200000

# IO 统计（每 1 秒的包数/字节数）
tshark -r cap.pcap -q -z "io,stat,1"

# IO 统计带过滤器（多列）
tshark -r cap.pcap -q \
    -z "io,stat,1,tcp,udp,icmp"

# 带 AVG/SUM 聚合函数
tshark -r smb.cap -q \
    -z "io,stat,0.1,AVG(smb.time)smb.time"
🔗会话与端点统计
# 会话统计（conv）：支持 eth/ip/ipv6/tcp/udp 等
tshark -r cap.pcap -q -z conv,tcp
tshark -r cap.pcap -q -z conv,ip
tshark -r cap.pcap -q -z conv,eth

# 带过滤器的会话统计
tshark -r cap.pcap -q -z "conv,tcp,ip.addr==192.168.1.1"

# 端点统计
tshark -r cap.pcap -q -z endpoints,ip
tshark -r cap.pcap -q -z endpoints,tcp

# IP 主机统计
tshark -r cap.pcap -q -z ip_hosts,tree

# IP 源/目统计（分开）
tshark -r cap.pcap -q -z ip_srcdst,tree

# 包长度分布
tshark -r cap.pcap -q -z plen,tree
⏱️响应时间（SRT/RTD）统计
# SMB 服务响应时间（Service Response Time）
tshark -r cap.pcap -q -z smb,srt
tshark -r cap.pcap -q -z smb2,srt

# HTTP 树状统计
tshark -r cap.pcap -q -z http,tree
tshark -r cap.pcap -q -z "http,stat"

# DNS 树状统计
tshark -r cap.pcap -q -z dns,tree

# SIP 统计（VoIP 场景）
tshark -r cap.pcap -q -z sip,stat

# RTP 流统计（音视频质量）
tshark -r cap.pcap -q -z rtp,streams

# NFS v3 服务响应时间
tshark -r cap.pcap -q -z rpc,srt,100003,3

# DCERPC 接口 SRT
tshark -r cap.pcap -q \
    -z dcerpc,srt,12345778-1234-abcd-ef00-0123456789ac,1.0

# 给 Info 列追加字段值
tshark -r cap.pcap -z proto,colinfo,nfs.fh.hash,nfs.fh.hash

⚙️ 第 7 章 · 高级功能
文件管理与环形缓冲（-b）
导读：在长时间或高速率抓包场景中，单个文件会无限增长。 TShark 的 -b（ring buffer，环形缓冲）和 -a（autostop）机制 提供了自动轮转与大小控制能力，是生产环境持续监控的基础设施选项。
📄 原文链接：tshark(1) man page - -b 与 -a 章节
▸ 环形缓冲（Ring Buffer）文件轮转机制
🔄-b 环形缓冲选项

官方文档（tshark man page）定义，-b 选项格式为 key:value，支持：

键说明
duration:N每 N 秒切换到下一个文件（支持浮点数，如 0.5）
filesize:N文件达到 N kB 时切换（最大 2TB）
files:N形成环形：文件数到 N 后覆盖最旧文件（需配合 duration/filesize）
packets:N每 N 个包切换文件
interval:N在 N 的整数倍时刻切换（如 3600 = 整点切换）
printname:FILE文件关闭后将文件名写入 FILE（可用 stdout/stderr）
# 官方示例：5个1MB文件的环形缓冲
sudo tshark -i eth0 -w out.pcapng \
    -b filesize:1000 -b files:5

# 每小时切换一个文件，保留最近 24 小时
sudo tshark -i eth0 -w /data/capture.pcapng \
    -b interval:3600 -b files:24

协议解码与 Decode As（-d）
导读：当流量使用非标准端口时（如 HTTP 跑在 8080，自定义协议跑在 9999）， TShark 无法自动识别。-d（Decode As）允许强制指定解码方式， 同时 --enable/disable-protocol 和 --only-protocols 提供精细的解析器开关控制。
📄 原文链接：tshark(1) man page - -d 章节
▸ Decode As 工作原理
🔧-d Decode As 命令

格式：-d <layer_type>==<selector>,<decode_as_protocol>

# 将 TCP 8080 端口强制解码为 HTTP
sudo tshark -i eth0 -d tcp.port==8080,http -Y 'http'

# 将 TCP 9999 解码为自定义协议
tshark -r cap.pcap -d tcp.port==9999,ssl

# 查看可用的 decode-as 选项
tshark -G decodes

# 多个 Decode As 规则
tshark -r cap.pcap \
    -d tcp.port==8080,http \
    -d tcp.port==8443,ssl

# 查看 Decode As 关联表
tshark -G decodes | head -20

高级语法与正则匹配
导读：本章汇集 TShark 最进阶的语法特性：matches（PCRE 正则）、 contains（字节序列搜索）、字段切片、比较函数，以及与管道、脚本集成的高级用法。 这些技术让 TShark 从"分析工具"升级为"数据挖掘引擎"。
📄 原文链接：tshark.dev - Filter with Regex: matches and contains
🔍matches：PCRE 正则匹配

官方文档说明（tshark.dev）： "matches will search with a regex while contains searches for exact byte sequences."

# 搜索包含俄语 .ru 域名的 HTTP GET（官方示例）
tshark -r cap.pcap \
    -Y 'http matches "https?.*?\.ru.*?worm"'

# 在 tshark 命令行中，内层引号需要转义
tshark -r cap.pcap \
    -Y "frame matches \"https?.*?\.ru.*?worm\""

# 搜索 User-Agent 中包含 curl 的请求
tshark -r cap.pcap \
    -Y 'http.user_agent matches "curl.*"'

# 搜索 DNS 查询中包含 "evil" 的域名
tshark -r cap.pcap \
    -Y 'dns.qry.name matches ".*evil.*"'

# 注意：matches 不能用于数字类型字段（int）
# 不能用：tcp.port matches "80"  ← 错误
# 应该用：tcp.port == 80           ← 正确

# 不能使用 \x00 空字节，改用 [^\x01-\xff]
🔎contains：字节序列搜索

官方文档（tshark.dev）："contains searches the text representation of a field."

# 搜索包含特定 OUI 的 MAC 地址（官方等价示例）
tshark -r cap.pcap -Y "eth.addr contains 00:16:e3"

# 以下三种方式等价（官方说明）
tshark -r cap.pcap -Y "eth.addr contains 00:16:e3"
tshark -r cap.pcap -Y "eth.addr[0:3] == 00:16:e3"
tshark -r cap.pcap -Y "eth.addr matches \"^[^\x01-\xff]\x16\xe3\""

# HTTP 响应体包含特定字符串
tshark -r cap.pcap -Y 'http contains "password"'

# TCP payload 包含特定字节
tshark -r cap.pcap -Y 'tcp contains "GET /"'
✂️字段切片（Slice）
# 语法：field[offset:length] 或 field[offset-end]

# MAC 地址前 3 字节（OUI）
tshark -r cap.pcap -Y "eth.src[0:3] == 00:16:e3"

# IP 地址最后一个字节
tshark -r cap.pcap -Y "ip.src[3:1] == 01"

# TCP payload 前 4 字节
tshark -r cap.pcap -Y "tcp.payload[0:4] == 47:45:54:20"
# 47:45:54:20 = "GET " in ASCII

🎯 第 11 章 · 总结
第一性原理总结
导读：第一性原理要求我们剥去所有习惯和惯例，回到最基本的事实重新推理。 本章用这一方法审视 TShark：它的本质是什么？所有命令背后的统一逻辑是什么？ 掌握这些根本原理，才能在遇到新问题时不查文档也能推导出正确命令。
▸ TShark 第一性原理 — 从本质到推论

🧱 原理一：一切皆字节流

本质：网络数据的终极形态是字节

TShark 的所有能力，本质上是对字节流施加的一系列变换。 无论是从网卡（-i）、从文件（-r）还是从管道（-r -）读取， 进入系统的都是原始字节。

推论：任何产生字节流的系统都可以作为 TShark 的输入源——tcpdump 管道、SSH 远程抓包、 甚至 netcat 转发的流量，都是合法的数据源。

# 推论验证：SSH远程抓包（任何字节流皆可输入）
ssh user@server "sudo tshark -i eth0 -w -" | tshark -r -

🚧 原理二：过滤是关门艺术——越早越便宜

本质：过滤是一种资源分配决策

TShark 的两类过滤器工作在完全不同的阶段： BPF 捕获过滤器（-f）在内核态、数据进入用户空间之前丢弃包，代价极低； 显示过滤器（-Y）在协议完整解析之后才执行，代价较高但功能完整。

推论：在已知"我不需要什么"时，应尽量用 -f 前置过滤； 在需要理解"协议语义"时，用 -Y 后置精过滤。两者并非互斥，而是互补。

# 最优实践：先粗筛后精过滤
sudo tshark -i eth0 -f "tcp port 80" -Y "http.response.code >= 400"

🔬 原理三：解析是语义赋予——协议是关于约定的共识

本质：协议解析器是字节到字段的映射函数

TShark 内置 3000+ 协议解析器。每个解析器将字节序列映射为有意义的字段名（如 http.response.code）。 -d（Decode As）的本质是：手动替换字节序列所绑定的解析函数。

推论：当某个协议跑在非标准端口时，问题不是"TShark 不支持"， 而是"TShark 使用了错误的解析器"——用 -d 纠正映射关系即可。

# 推论验证：强制用 HTTP 解析器处理 8080 端口的字节
tshark -r cap.pcap -d tcp.port==8080,http -Y 'http.request'

📤 原理四：输出是下游的接口——格式决定可用性

本质：TShark 的价值体现在输出的可消费性

TShark 的分析结果必须被人或机器"消费"才有价值。 -T text 面向人类阅读；-T fields 面向 awk/Python； -T json 面向 jq/API；-z 统计则是面向决策的聚合。

推论：在设计 TShark 命令时，应先确定"谁来消费这个输出"， 再选择对应的输出格式。这个选择比过滤条件的选择更影响实际效用。

# 面向人：文本
tshark -r cap.pcap -Y 'http' 
# 面向机器（awk/csv）：字段
tshark -r cap.pcap -T fields -e ip.src -e http.host -E separator=,
# 面向决策：聚合统计
tshark -r cap.pcap -q -z io,phs

🔄 原理五：两遍分析是时间折叠——以空间换知识

本质：网络协议中存在"前向引用"

某些协议字段（如"这个包的响应在第 N 帧"）只有在看完整个捕获后才能确定。 -2 两遍扫描的本质是：用额外的内存（缓冲输出）换取对未来信息的感知能力。

推论：当过滤条件涉及"上下文相关"字段时（重传检测、响应帧号、TCP 流重组）， 必须使用 -2。否则这些字段可能为空或不准确。

# 上下文相关字段示例
tshark -r cap.pcap -2 -R "tcp" -Y "tcp.analysis.retransmission"

⚡ 原理六：统计优于逐包——提升信噪比

本质：在噪声中找信号，聚合比遍历更高效

面对百万级数据包时，逐包查看是低效的。-z 统计的本质是： 用聚合函数将 N 个包压缩为 1 行有意义的统计量（均值、最大值、分布）。 这是信号处理中的降采样思想在网络分析中的应用。

推论：排查未知问题时，应先用 -z io,phs 建立协议分布基线， 再用 -z conv,tcp 确认问题会话，最后才用 -Y 精确定位问题包。 这是"从宏观到微观"的侦探方法论。

# 正确的侦探工作流
tshark -r cap.pcap -q -z io,phs          # 第一步：鸟瞰全局
tshark -r cap.pcap -q -z conv,tcp        # 第二步：锁定嫌疑会话
tshark -r cap.pcap -Y 'tcp.stream == 5'  # 第三步：深入单个流
🦈 终极公式

任何 TShark 命令都是以下公式的具体化：

tshark  \
  [数据来源: -i/-r/-r-]          \   # 原理一
  [捕获过滤: -f "BPF"]           \   # 原理二（早）
  [处理: -2/-n/-d/-o/-K]         \   # 原理三/五
  [显示过滤: -Y "语义过滤"]       \   # 原理二（晚）
  [输出: -T/-e/-E/-z/-V/-x/-w]       # 原理四/六

理解了这六条原理，无需死记硬背任何命令。 每次遇到新需求，只需回答：数据从哪来？怎么过滤？ 协议如何解码？谁来消费输出？答案自然推导出正确命令。
