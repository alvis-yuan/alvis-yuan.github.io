0Linux 事件驱动编程概述

在 Linux C 语言编程中，事件驱动（Event-Driven）是一种核心的并发模型：程序不主动阻塞等待某一个 I/O，而是向内核注册对多个文件描述符（fd）的兴趣，当其中任何一个就绪时，内核通知应用程序，再由应用程序对事件作出响应。

Linux 提供了三套 I/O 多路复用（I/O multiplexing）系统调用，它们都属于事件驱动的核心机制：

select()
最古老（4.2BSD起），POSIX标准。最多监控 FD_SETSIZE（1024）个 fd，每次调用均需重建集合。
已不推荐
poll()
解除了 1024 上限，使用 pollfd 数组，无需重建集合。POSIX.1-2001，Linux 2.1.23。
中等推荐
epoll
Linux 专属（2.5.44+），O(1) 事件返回，支持边沿触发，高并发首选。
强烈推荐
来源说明
本文所有技术内容均严格依据 Linux man-pages 6.18（2026-05-24）官方手册，原文链接： select(2)、 poll(2)、 epoll(7)、 epoll_ctl(2)。

§1 select()
1select() — 同步 I/O 多路复用

官方来源：man7.org/linux/man-pages/man2/select.2.html  ·  Linux man-pages 6.18, 2026-02-08

⚠ 官方警告
原文（man-pages 6.18）明确指出：select() 只能监控编号小于 FD_SETSIZE（1024）的文件描述符——这对很多现代应用来说是一个不合理的低限制，且这一限制不会改变。所有现代应用应改用 poll(2) 或 epoll(7)，它们不受此限制。

1.1 函数签名

#include <sys/select.h>

int select(int nfds,
           fd_set *readfds,
           fd_set *writefds,
           fd_set *exceptfds,
           struct timeval *timeout);

/* 操作文件描述符集合的宏 */
void  FD_ZERO(fd_set *set);           /* 清空集合 */
void  FD_SET(int fd, fd_set *set);   /* 添加 fd 到集合 */
void  FD_CLR(int fd, fd_set *set);   /* 从集合移除 fd */
int   FD_ISSET(int fd, fd_set *set); /* 测试 fd 是否在集合中 */

/* 超时结构体 */
struct timeval {
    time_t      tv_sec;    /* 秒 */
    suseconds_t tv_usec;   /* 微秒 */
};

1.2 参数说明

参数含义
nfds三个集合中最大文件描述符编号 + 1
readfds监控可读事件的 fd 集合；返回时被修改为"就绪"的子集
writefds监控可写事件的 fd 集合；返回时被修改为"就绪"的子集
exceptfds监控异常条件的 fd 集合；返回时被修改为"异常"的子集
timeout超时时长；NULL = 永久阻塞；{0,0} = 立即返回（轮询）
设计陷阱
每次调用后集合均被修改——因此在循环中使用 select() 时，必须在每次调用前重新初始化三个 fd 集合。这是 select() 设计上的主要缺陷，已被 poll(2) 和 epoll(7) 所规避。

1.3 三类事件集合

根据官方 man-pages，select() 内核实现中对 fd 事件的映射关系为：

/* 来自 Linux 内核源码，select() 与 poll/epoll 事件的对应关系 */
#define POLLIN_SET  (EPOLLRDNORM | EPOLLRDBAND | EPOLLIN |
                     EPOLLHUP | EPOLLERR)   /* readfds 就绪 */
#define POLLOUT_SET (EPOLLWRBAND | EPOLLWRNORM | EPOLLOUT |
                     EPOLLERR)              /* writefds 就绪 */
#define POLLEX_SET  (EPOLLPRI)              /* exceptfds 异常 */
readfds — 可读集合
监控 fd 是否可以进行 read(2) 操作而不阻塞；文件描述符到达 EOF 时也认为"可读"。
应用场景：
• TCP/UDP socket 有数据可接收
• 管道读端有数据
• 新连接到达（服务器 listen socket）
• stdin 有用户输入
• timerfd 定时器到期
writefds — 可写集合
监控 fd 是否可以进行 write(2) 操作而不阻塞。注意：即使返回可写，写入超过发送缓冲区的大数据仍可能阻塞。
应用场景：
• 非阻塞 connect() 完成
• socket 发送缓冲区有空间
• 管道写端不满
• 文件/设备可写
exceptfds — 异常集合
监控"异常条件"，对应 poll() 的 POLLPRI 事件（即 EPOLLPRI）。
应用场景：
• TCP 套接字带外数据（OOB）
• 伪终端主设备 packet 模式下从设备状态变化
• cgroup.events 文件被修改

1.4 返回值

  • 成功：返回三个集合中就绪 fd 的总位数（可为 0，表示超时）
  • 失败：返回 -1，设置 errno（EBADF / EINTR / EINVAL / ENOMEM）

1.5 pselect() — 信号安全变体

int pselect(int nfds,
            fd_set *readfds, fd_set *writefds, fd_set *exceptfds,
            const struct timespec *timeout,
            const sigset_t *sigmask);  /* 原子地替换信号屏蔽字 */

与 select() 的区别：超时使用纳秒精度的 timespec；增加 sigmask 参数，可原子地设置信号屏蔽字，避免"先检查全局标志，再调用 select()"之间的竞态条件。引入自 Linux 2.6.16。

1.6 select() 使用示例

#include <stdio.h>
#include <stdlib.h>
#include <sys/select.h>
#include <sys/time.h>

int main(void)
{
    int           retval;
    fd_set        rfds;
    struct timeval tv;

    /* 监控 stdin (fd=0) 是否有输入 */
    FD_ZERO(&rfds);
    FD_SET(0, &rfds);

    tv.tv_sec  = 5;   /* 等待最多 5 秒 */
    tv.tv_usec = 0;

    retval = select(1, &rfds, NULL, NULL, &tv);

    if (retval == -1)
        perror("select()");
    else if (retval)
        printf("有数据可读。\n");    /* FD_ISSET(0, &rfds) 为真 */
    else
        printf("5秒内无数据。\n");

    exit(EXIT_SUCCESS);
}

以上示例代码来自：man2/select.2 — EXAMPLES

§2 poll()
2poll() — 改进的 I/O 多路复用

官方来源：man7.org/linux/man-pages/man2/poll.2.html  ·  Linux man-pages 6.18, 2026-02-08  ·  Linux 2.1.23 引入

poll() 执行与 select(2) 类似的任务：等待一组文件描述符中某个就绪以执行 I/O。相较于 select()，poll() 使用 pollfd 结构体数组而非三个位图，消除了 FD_SETSIZE 的限制，且分离了"请求事件"与"返回事件"字段，避免了每次循环都重建集合的问题。

2.1 函数签名

#include <poll.h>

int poll(struct pollfd *fds, nfds_t nfds, int timeout);

struct pollfd {
    int   fd;       /* 文件描述符 */
    short events;   /* 请求监控的事件（输入） */
    short revents;  /* 内核返回的实际发生事件（输出） */
};

timeout 单位为毫秒；-1 = 永久阻塞；0 = 立即返回（非阻塞轮询）。 若 fd 字段为负数，该项被忽略，revents 返回 0——这是暂时禁用某个 fd 监控的简便方法。

2.2 poll 事件标志

POLLIN
有数据可读。
可读
应用场景： TCP/UDP socket 收到数据、管道写入数据、新客户端连接到达（accept），stdin 用户输入，timerfd 到期。
POLLPRI
文件描述符上存在某种异常条件。
带外/异常
应用场景： TCP 套接字上有带外数据（OOB）；伪终端主设备 packet 模式下从设备状态变化；cgroup.events 文件被修改。
POLLOUT
现在可以写入，但向 socket 或 pipe 写入超过可用空间的数据仍会阻塞（除非设置 O_NONBLOCK）。
可写
应用场景： 非阻塞 connect() 完成后确认可写、发送缓冲区腾出空间后继续发送数据。
POLLRDHUP Linux 2.6.17
流式 socket 对端关闭连接，或对端已关闭写入半连接。需要定义 _GNU_SOURCE 宏。
连接关闭
应用场景： 检测 TCP 连接对端关闭（与 POLLIN 配合使用），特别适合边沿触发模式下的连接断开检测。
POLLERR
错误条件。仅在 revents 中返回，events 中指定该位被忽略。当 pipe 读端关闭时，写端也会报告此事件。
错误
应用场景： socket 异步错误、管道读端已关闭时写端检测、连接错误检测（TCP RST 等）。
POLLHUP
挂起事件。仅在 revents 中返回，events 中指定被忽略。对于管道/流 socket，仅表示对端关闭了其端口，已有数据读完才会返回 0。
挂起/EOF
应用场景： 管道写端关闭后读端检测、串口/终端挂断（DCD 信号丢失）、socket 对端关闭。
POLLNVAL
无效请求：fd 未打开。仅在 revents 中返回，events 中指定被忽略。
无效fd
应用场景： 检测文件描述符已被关闭（编程错误检测）；多线程中 fd 意外关闭后的错误处理。
POLLRDNORM / POLLWRNORM
编译时定义 _XOPEN_SOURCE 才有效。POLLRDNORM 等同于 POLLIN，POLLWRNORM 等同于 POLLOUT。
兼容扩展
应用场景： XOPEN 兼容性代码，Linux 上不提供超出 POLLIN/POLLOUT 的额外信息。

2.3 ppoll() — 信号安全变体

#define _GNU_SOURCE
#include <poll.h>

int ppoll(struct pollfd *fds, nfds_t nfds,
          const struct timespec *tmop,
          const sigset_t *sigmask);

作用类似 pselect()：原子地替换信号屏蔽字并等待事件，避免信号与 poll() 之间的竞态条件。引入自 Linux 2.6.16。

§3 epoll
3epoll — Linux 高性能事件通知机制 Linux Only

官方来源：man7.org/linux/man-pages/man7/epoll.7.html  ·  man2/epoll_ctl.2.html  ·  Linux man-pages 6.18  ·  始于 Linux 2.5.44, glibc 2.3.2

epoll API 执行与 poll(2) 类似的任务：监控多个文件描述符以判断是否可进行 I/O。epoll API 既可作为边沿触发（ET）接口使用，也可作为水平触发（LT）接口使用，并能良好地扩展到大量被监控的文件描述符。

epoll API 的核心概念是 epoll 实例——一个内核数据结构，从用户空间角度看，可认为是两个列表的容器：

  • interest list（兴趣列表，也称 epoll set）：进程已注册兴趣监控的 fd 集合。
  • ready list（就绪列表）：当前"就绪"可 I/O 的 fd 集合，是 interest list 的子集，由内核根据 I/O 活动动态填充。

3.1 三个核心系统调用

#include <sys/epoll.h>

/* 1. 创建 epoll 实例，返回 epoll fd */
int epoll_create1(int flags);   /* flags=0 或 EPOLL_CLOEXEC */
int epoll_create(int size);    /* size 已被忽略但必须 > 0（旧接口）*/

/* 2. 管理 interest list（添加/修改/删除） */
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);

/* op 的合法值： */
EPOLL_CTL_ADD  /* 向 interest list 添加 fd */
EPOLL_CTL_MOD  /* 修改已注册 fd 的事件设置 */
EPOLL_CTL_DEL  /* 从 interest list 删除 fd（event 可为 NULL）*/

/* 3. 等待事件，阻塞直到有事件就绪或超时 */
int epoll_wait(int epfd, struct epoll_event *events,
               int maxevents, int timeout);
/* timeout: -1=永久阻塞, 0=立即返回, >0=毫秒超时 */

/* epoll_event 结构体 */
struct epoll_event {
    uint32_t events;    /* 事件位掩码（EPOLLIN | EPOLLET 等）*/
    union {
        void    *ptr;
        int      fd;
        uint32_t u32;
        uint64_t u64;
    } data;              /* 用户自定义数据（随事件一起返回）*/
};

3.2 epoll 事件类型（Event Types）

以下均来自 epoll_ctl(2) 官方手册，events 位掩码中可用的事件类型（由 epoll_wait 返回）：

EPOLLIN
关联文件可进行 read(2) 操作。
可读
应用场景：
• TCP 服务器：监听 socket 有新连接（accept()）
• 客户端连接：对端发送数据可读
• 管道/FIFO：写端写入数据后读端就绪
• timerfd：定时器超时，read() 读取到期次数
• signalfd：信号到达，read() 读取信号信息
• eventfd：计数器非零，可读取通知
EPOLLOUT
关联文件可进行 write(2) 操作。
可写
应用场景：
• 非阻塞 connect()：连接建立后触发，可调用 getsockopt() 检查结果
• 发送缓冲区：socket 发送缓冲区腾出空间，可继续发送
• 反压控制：与 EPOLLIN 组合，实现读写分离的背压机制
• 边沿触发优化：注册时同时指定 EPOLLIN|EPOLLOUT，避免频繁调用 epoll_ctl
EPOLLRDHUP Linux 2.6.17
流式 socket 对端关闭连接，或对端已关闭写入半连接。在边沿触发监控时，特别适合编写简洁的对端关闭检测代码。
半关闭
应用场景：
• TCP 连接管理：检测客户端调用 shutdown(SHUT_WR) 或 close()
• 边沿触发必备：ET 模式下配合 EPOLLIN 检测对端关闭，无需额外 read() 返回 0 判断
• 长连接心跳：减少不必要的心跳包，依赖此事件检测连接断开
EPOLLPRI
文件描述符上存在异常条件。对应 poll(2) 中的 POLLPRI。
带外/异常
应用场景：
• TCP OOB 数据：带外数据到达（MSG_OOB），需用 recv(MSG_OOB) 接收
• 伪终端：主设备 packet 模式下从设备状态变化
• cgroup.events：cgroup 事件文件被修改（容器管理）
EPOLLERR
关联文件描述符发生错误条件。当 pipe 读端关闭时，写端也会报告此事件。epoll_wait() 始终报告此事件，无需在 epoll_ctl() 时显式设置。
错误
应用场景：
• socket 异步错误：TCP RST、ICMP 错误通知，需用 getsockopt(SO_ERROR) 获取
• 管道写端：读端关闭后，写端自动收到 EPOLLERR
• 通用错误处理：关闭 fd，清理连接状态
EPOLLHUP
关联文件描述符发生挂起事件。epoll_wait() 始终等待此事件，无需在 epoll_ctl() 时显式设置。读取管道/流 socket 时，仅表示对端关闭其端，需读完已有数据后才返回 0（EOF）。
挂起
应用场景：
• 管道读端：写端关闭，读端后续 read() 返回 EOF
• 串口/终端：载波信号丢失（DCD）
• socket 关闭：配合 EPOLLIN 判断连接是否正常结束

3.3 epoll 输入控制标志（Input Flags）

以下标志在 epoll_ctl() 的 events 中设置，用于控制行为，不由 epoll_wait() 返回：

EPOLLET — 边沿触发
为关联 fd 请求边沿触发通知。默认行为是水平触发。
触发模式
应用场景：
• 高并发服务器：Nginx/Redis 等高性能服务器的标配模式
• 要求：必须配合非阻塞 fd（O_NONBLOCK）使用
• 要求：每次触发后需循环读写直到 EAGAIN
EPOLLONESHOT Linux 2.6.2
请求对关联 fd 的一次性通知。epoll_wait() 返回事件后，该 fd 在 interest list 中被禁用，不再报告事件。用户必须用 EPOLL_CTL_MOD 重新激活。
单次触发
应用场景：
• 多线程分发：防止多个线程同时处理同一 fd 的事件
• 线程池模型：任务分配给工作线程后禁用，处理完再重新注册
• 避免惊群：配合 EPOLLEXCLUSIVE 使用
EPOLLWAKEUP Linux 3.5
若 EPOLLONESHOT 和 EPOLLET 均未设置且进程具有 CAP_BLOCK_SUSPEND 能力，确保系统在该事件挂起或处理期间不进入"挂起"或"休眠"状态。
唤醒锁
应用场景：
• Android/嵌入式：系统处于 autosleep 模式时，保持设备唤醒直到事件被完整处理
• 电源管理：防止 I/O 事件被中断处理后系统立即重新休眠
EPOLLEXCLUSIVE Linux 4.5
为附加到目标 fd 的 epoll fd 设置独占唤醒模式。当唤醒事件发生且多个 epoll fd 通过 EPOLLEXCLUSIVE 附加到同一目标 fd 时，只有一个或若干（而非全部）epoll fd 会收到事件，用于避免"惊群"问题。仅可在 EPOLL_CTL_ADD 中使用。
惊群防护
应用场景：
• 多进程 accept()：多个 worker 进程监听同一套接字，避免所有进程同时被唤醒
• 线程池：共享 epoll 实例时减少不必要的线程唤醒

3.4 水平触发 vs 边沿触发

以下场景和解释均来自 epoll(7) 官方手册：

官方场景示例
假设以下顺序发生：
  1. 表示管道读端的文件描述符 rfd 被注册到 epoll 实例。
  2. 管道写端写入 2 kB 数据。
  3. 调用 epoll_wait()，返回 rfd 就绪。
  4. 管道读端从 rfd 读取 1 kB 数据。
  5. 再次调用 epoll_wait()。
图2：LT 与 ET 模式下 epoll 行为对比（数据来源：epoll(7) man page）
官方最佳实践（来自 epoll(7)）
当使用边沿触发（EPOLLET）接口时，为获得性能收益，可在 epoll 接口（EPOLL_CTL_ADD）中一次性指定 (EPOLLIN|EPOLLOUT)，从而无需反复调用 epoll_ctl(EPOLL_CTL_MOD) 在 EPOLLIN 和 EPOLLOUT 之间切换。

3.5 官方完整使用示例

来源：epoll(7) — Example for suggested usage

#define MAX_EVENTS 10
struct epoll_event ev, events[MAX_EVENTS];
int listen_sock, conn_sock, nfds, epollfd;

/* socket(), bind(), listen() 代码省略 */

epollfd = epoll_create1(0);
if (epollfd == -1) { perror("epoll_create1"); exit(EXIT_FAILURE); }

ev.events  = EPOLLIN;
ev.data.fd = listen_sock;
if (epoll_ctl(epollfd, EPOLL_CTL_ADD, listen_sock, &ev) == -1) {
    perror("epoll_ctl: listen_sock"); exit(EXIT_FAILURE);
}

for (;;) {
    nfds = epoll_wait(epollfd, events, MAX_EVENTS, -1);
    if (nfds == -1) { perror("epoll_wait"); exit(EXIT_FAILURE); }

    for (int n = 0; n < nfds; ++n) {
        if (events[n].data.fd == listen_sock) {
            /* 新连接到达 */
            conn_sock = accept(listen_sock,
                               (struct sockaddr *) &addr, &addrlen);
            if (conn_sock == -1) { perror("accept"); exit(EXIT_FAILURE); }
            setnonblocking(conn_sock);        /* 设置非阻塞 */
            ev.events  = EPOLLIN | EPOLLET;   /* 边沿触发 */
            ev.data.fd = conn_sock;
            if (epoll_ctl(epollfd, EPOLL_CTL_ADD, conn_sock, &ev) == -1) {
                perror("epoll_ctl: conn_sock"); exit(EXIT_FAILURE);
            }
        } else {
            do_use_fd(events[n].data.fd);    /* 处理已连接 fd */
        }
    }
}

6官方最佳实践与陷阱规避

以下内容均来自 epoll(7)、epoll_ctl(2)、select(2)、poll(2) 官方手册的 NOTES、BUGS 和 Pitfalls 章节。

6.1 边沿触发（ET）的正确用法

官方原文（epoll(7)）：使用 EPOLLET 的应用程序应使用非阻塞文件描述符，以避免阻塞的 read 或 write 使处理多个文件描述符的任务饿死。建议按以下方式使用 epoll 作为边沿触发接口：

// 正确的 ET 模式处理框架
while (1) {
    ssize_t n = read(fd, buf, sizeof(buf));
    if (n == -1) {
        if (errno == EAGAIN || errno == EWOULDBLOCK)
            break;  /* 数据读完，等待下次事件 */
        /* 其他错误：处理错误 */
        break;
    }
    if (n == 0) {
        /* EOF：对端关闭连接 */
        close(fd);
        break;
    }
    /* 处理读取到的数据... */
}

6.2 常见陷阱（官方来源）

陷阱 1：ET 模式下的饿死问题（epoll(7)）
若有大量 I/O 空间，尝试耗尽它可能导致其他文件描述符无法被处理，引发饿死。 解决方案：维护一个就绪列表，在数据结构中标记 fd 为就绪，让应用程序在所有就绪 fd 之间轮询（round robin）处理，并忽略已在就绪列表中的 fd 的后续事件。
陷阱 2：事件缓存中的悬空 fd（epoll(7)）
若使用事件缓存存储所有从 epoll_wait() 返回的 fd，则必须提供一种动态标记 fd 关闭的方式。例如：epoll_wait() 返回 100 个事件，处理第 47 个时导致第 13 个 fd 被关闭。若不处理，访问第 13 个事件时将引发混乱。 解决方案：处理关闭时先调用 epoll_ctl(EPOLL_CTL_DEL)，再 close()，并标记对应数据结构为"已移除"。
陷阱 3：dup() 导致 fd 未从 interest list 移除（epoll(7)）
文件描述符是对打开文件描述（open file description）的引用。通过 dup()/dup2()/fork() 复制的 fd 指向同一打开文件描述。只有当所有引用同一打开文件描述的 fd 均关闭后，interest list 中的条目才会被移除。 解决方案：关闭前先用 epoll_ctl(EPOLL_CTL_DEL) 显式移除，或确保关闭所有重复的 fd。
陷阱 4：select() fd 集合需每次重建（select(2)）
每次 select() 返回后，三个 fd_set 均被原地修改。在循环中使用时，必须在每次调用前重新初始化（FD_ZERO + FD_SET）。
陷阱 5：EPOLLWAKEUP 静默忽略（epoll_ctl(2)）
若在 flags 中指定 EPOLLWAKEUP，但调用者不具备 CAP_BLOCK_SUSPEND 能力，则该标志会被静默忽略（不报错）。健壮的应用程序应在使用 EPOLLWAKEUP 前验证自身拥有该能力。

6.3 /proc 内核参数

来自 epoll(7) 官方手册：

参数路径含义（引入版本）
/proc/sys/fs/epoll/max_user_watches 限制一个用户在所有 epoll 实例中可注册的 fd 总数（Linux 2.6.28）。按真实用户 ID 计。32位内核每个注册 fd 约消耗 90 字节，64位约 160 字节。默认值为可用低端内存的 1/25 (4%) 除以单个注册成本。

6.4 epoll 问答（来自 epoll(7) Q&A 章节）

  • Q：同一个 fd 能否注册两次？ → 返回 EEXIST。但可以将 dup() 后的 fd 添加到同一 epoll 实例，用不同事件掩码过滤事件。
  • Q：两个 epoll 实例能监控同一个 fd 吗？ → 可以，事件会同时报告给两个实例，但需谨慎编程。
  • Q：epoll fd 本身是否可被 poll/epoll/select 监控？ → 是的，若 epoll fd 有等待中的事件，它本身会显示为可读。
  • Q：能否将 epoll fd 添加到自己的 fd 集合中？ → epoll_ctl() 会返回 EINVAL。但可以将一个 epoll fd 嵌套到另一个 epoll 实例中。
  • Q：关闭 fd 是否会自动从所有 interest list 移除？ → 是，但需注意：若存在 dup() 副本，只有所有引用同一打开文件描述的 fd 都关闭后，才真正从 interest list 移除。
  • Q：两次 epoll_wait() 之间发生了多个事件，是合并还是分别报告？ → 合并（combined）报告。
  • Q：使用 EPOLLET 时是否需要持续读写直到 EAGAIN？ → 是的。必须认为 fd 就绪，直到下次非阻塞 read/write 返回 EAGAIN。

7完整代码示例

7.1 epoll 水平触发 TCP 服务器（推荐模式）

#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>
#include <fcntl.h>
#include <sys/epoll.h>
#include <sys/socket.h>
#include <netinet/in.h>
#include <errno.h>

#define MAX_EVENTS 64
#define PORT       8080

/* 设置 fd 为非阻塞 */
static void set_nonblocking(int fd) {
    int flags = fcntl(fd, F_GETFL, 0);
    fcntl(fd, F_SETFL, flags | O_NONBLOCK);
}

int main(void) {
    int listenfd, epollfd, nfds;
    struct epoll_event ev, events[MAX_EVENTS];
    struct sockaddr_in addr;

    /* 1. 创建监听 socket */
    listenfd = socket(AF_INET, SOCK_STREAM | SOCK_NONBLOCK, 0);
    int opt = 1;
    setsockopt(listenfd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));

    memset(&addr, 0, sizeof(addr));
    addr.sin_family      = AF_INET;
    addr.sin_addr.s_addr = INADDR_ANY;
    addr.sin_port        = htons(PORT);
    bind(listenfd, (struct sockaddr*)&addr, sizeof(addr));
    listen(listenfd, SOMAXCONN);

    /* 2. 创建 epoll 实例 */
    epollfd = epoll_create1(0);

    /* 3. 注册监听 socket（LT 模式，无需 EPOLLET）*/
    ev.events  = EPOLLIN;
    ev.data.fd = listenfd;
    epoll_ctl(epollfd, EPOLL_CTL_ADD, listenfd, &ev);

    printf("epoll server listening on port %d\n", PORT);

    for (;;) {
        /* 4. 等待事件（-1 = 永久阻塞）*/
        nfds = epoll_wait(epollfd, events, MAX_EVENTS, -1);
        if (nfds == -1 && errno == EINTR) continue;

        for (int i = 0; i < nfds; i++) {
            int fd = events[i].data.fd;

            if (events[i].events & (EPOLLERR | EPOLLHUP)) {
                /* EPOLLERR/EPOLLHUP：关闭连接 */
                close(fd);
                continue;
            }

            if (fd == listenfd) {
                /* EPOLLIN on 监听 socket：新连接 */
                int connfd = accept(listenfd, NULL, NULL);
                set_nonblocking(connfd);
                ev.events  = EPOLLIN | EPOLLRDHUP;
                ev.data.fd = connfd;
                epoll_ctl(epollfd, EPOLL_CTL_ADD, connfd, &ev);
            } else if (events[i].events & EPOLLRDHUP) {
                /* 对端关闭连接 */
                epoll_ctl(epollfd, EPOLL_CTL_DEL, fd, NULL);
                close(fd);
            } else if (events[i].events & EPOLLIN) {
                /* 有数据可读 */
                char buf[4096];
                ssize_t n = read(fd, buf, sizeof(buf));
                if (n > 0) {
                    write(fd, buf, n);  /* echo 回显 */
                } else if (n == 0) {
                    epoll_ctl(epollfd, EPOLL_CTL_DEL, fd, NULL);
                    close(fd);
                }
            }
        }
    }
    return 0;
}

7.2 timerfd + signalfd 与 epoll 集成（Linux 专属）

#include <sys/epoll.h>
#include <sys/timerfd.h>
#include <sys/signalfd.h>
#include <signal.h>
#include <stdint.h>
#include <stdio.h>
#include <unistd.h>

int main(void) {
    int epfd = epoll_create1(0);
    struct epoll_event ev, events[8];

    /* --- timerfd：1秒定时器 --- */
    int tfd = timerfd_create(CLOCK_MONOTONIC, TFD_NONBLOCK);
    struct itimerspec ts = {
        .it_interval = {.tv_sec = 1},
        .it_value    = {.tv_sec = 1}
    };
    timerfd_settime(tfd, 0, &ts, NULL);
    ev.events = EPOLLIN; ev.data.fd = tfd;
    epoll_ctl(epfd, EPOLL_CTL_ADD, tfd, &ev);

    /* --- signalfd：捕获 SIGINT/SIGTERM --- */
    sigset_t mask;
    sigemptyset(&mask);
    sigaddset(&mask, SIGINT);
    sigaddset(&mask, SIGTERM);
    sigprocmask(SIG_BLOCK, &mask, NULL);  /* 阻塞信号正常处理 */
    int sfd = signalfd(-1, &mask, SFD_NONBLOCK);
    ev.events = EPOLLIN; ev.data.fd = sfd;
    epoll_ctl(epfd, EPOLL_CTL_ADD, sfd, &ev);

    printf("事件循环启动，Ctrl+C 退出\n");

    for (;;) {
        int n = epoll_wait(epfd, events, 8, -1);
        for (int i = 0; i < n; i++) {
            if (events[i].data.fd == tfd) {
                uint64_t exp;
                read(tfd, &exp, sizeof(exp));   /* 读取到期次数 */
                printf("定时器到期 ×%lu\n", exp);
            } else if (events[i].data.fd == sfd) {
                struct signalfd_siginfo si;
                read(sfd, &si, sizeof(si));
                printf("收到信号 %u，退出\n", si.ssi_signo);
                close(tfd); close(sfd); close(epfd);
                return 0;
            }
        }
    }
}
