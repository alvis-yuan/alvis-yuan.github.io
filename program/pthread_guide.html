第 01 章 · 基础篇

POSIX 线程概述与编程模型

从操作系统原理出发，理解 Linux 线程的本质：什么是线程、线程与进程的关系、NPTL 实现机制，以及 pthreads API 的核心架构。

📄 原文来源：man7.org — pthreads(7) Linux manual page
📌 导读

本章建立最基础的心智模型：一个进程可以包含多个线程，它们共享同一地址空间但各自拥有独立的执行上下文（寄存器、栈、信号掩码）。理解这个"共享＋独立"的二元结构是掌握多线程编程的第一步。

什么是 POSIX 线程

POSIX.1 规定了一套线程编程接口，通称 POSIX threads (Pthreads)。一个进程可以包含多个线程，所有线程共同执行同一个程序，共享同一片全局内存（数据段和堆），但每个线程拥有独立的栈空间（局部变量）。

进程级共享资源 vs 线程私有资源

🗂 进程与线程内存结构示意

Linux 线程实现：NPTL

Linux 经历了两代线程实现：早期的 LinuxThreads（已废弃）和现代的 NPTL（Native POSIX Threads Library）。

  • NPTL 自 glibc 2.3.2 起提供，需要 Linux 2.6 内核支持
  • 采用 1:1 模型：每个线程对应一个内核调度实体
  • 使用 clone(2) 系统调用创建线程
  • 使用 futex(2) 实现高效的用户空间同步原语
  • 同一进程所有线程共享同一 PID（符合 POSIX.1）
💡

man7.org 原文："Both of these are so-called 1:1 implementations, meaning that each thread maps to a kernel scheduling entity. Both threading implementations employ the Linux clone(2) system call. In NPTL, thread synchronization primitives (mutexes, thread joining, and so on) are implemented using the Linux futex(2) system call."

pthreads 函数返回值约定

与传统系统调用不同，pthreads 函数成功返回 0，失败返回错误码（正整数），不设置 errno。这是重要的编程约定，切勿用 errno 检查 pthread 函数错误。

⚠️

man7.org 原文："Most pthreads functions return 0 on success, and an error number on failure. Note that the pthreads functions do not set errno."

编译方式

# 编译时必须链接 pthread 库
cc -pthread pthread_example.c -o example
# 注意：-pthread 优于 -lpthread（同时设置编译宏）
第 02 章 · 基础篇

线程生命周期管理

掌握线程的创建、终止、等待（join）和分离（detach）四个核心操作，理解线程属性配置与生命周期状态机。

📄 原文来源：CMU — Linux Tutorial: POSIX Threads
📌 导读

线程的生命周期是多线程编程的骨架。创建（create）→ 运行（running）→ 终止（exit）→ 回收（join/detach）构成完整闭环。漏掉 join 或 detach 会导致资源泄漏（"僵尸线程"）。

线程生命周期状态机

🔄 线程状态转换图

pthread_create — 创建线程

#include <pthread.h>

int pthread_create(
    pthread_t       *thread,        /* 输出：线程ID */
    const pthread_attr_t *attr,     /* NULL = 默认属性 */
    void *(*start_routine)(void *), /* 线程函数 */
    void *arg                       /* 传给线程的参数 */
);

关键参数说明

  • thread：返回新建线程的 ID（pthread_t，本质是无符号长整型）
  • attr：线程属性，NULL 使用默认值。可配置：分离状态、调度策略、栈大小等
  • start_routine：线程执行函数，接受 void* 参数，返回 void*
  • arg：传递给线程函数的参数指针；传多个参数时用结构体指针

pthread_exit — 线程终止

void pthread_exit(void *retval);
/* retval 不得指向局部变量（线程退出后栈即销毁）*/

线程可通过三种方式终止：① 调用 pthread_exit()；② start_routine 函数返回；③ 整个进程调用 exit()（所有线程随之终止）。

pthread_join — 等待线程

int pthread_join(pthread_t thread, void **retval);
/* 阻塞直到 thread 线程终止，回收其资源 */
/* 若 retval 非 NULL，接收线程的退出值 */
⚠️

CMU 原文提示："Unless we wait we run the risk of executing an exit which will terminate the process and all threads before the threads have completed." — 主线程必须 join 子线程，否则进程退出时子线程被强制杀死。

Joinable vs Detached

属性Joinable（默认）Detached
资源回收需要调用 pthread_join()线程结束自动释放
返回值可通过 join 获取无法获取
适用场景需要等待结果/协调后台独立任务
设置方式默认 / PTHREAD_CREATE_JOINABLEpthread_detach() 或属性设置

完整示例

#include <stdio.h>
#include <pthread.h>

void *worker(void *arg) {
    char *msg = (char *)arg;
    printf("%s\n", msg);
    return NULL;
}

int main() {
    pthread_t t1, t2;
    pthread_create(&t1, NULL, worker, "Thread 1");
    pthread_create(&t2, NULL, worker, "Thread 2");
    pthread_join(t1, NULL);  /* 等待 t1 完成 */
    pthread_join(t2, NULL);  /* 等待 t2 完成 */
    return 0;
}

🚨

第 04 章 · 同步机制篇

互斥锁 Mutex

互斥锁是最基础的线程同步原语，通过"互斥排他"机制保护临界区，确保同一时刻只有一个线程可以访问共享资源。

📄 原文来源：man7.org — pthread_mutex_init(3) Linux manual page
📌 导读

Mutex（互斥锁）是解决共享资源竞争问题的最直接工具。理解它的本质——"有人持有锁时其他人等待，持有者完成后释放"——有助于在更复杂场景中正确使用。记住：锁的粒度决定并发性能，粒度太粗降低并发，粒度太细容易死锁。

Mutex 工作原理

🔒 Mutex 互斥访问时序图

核心 API

#include <pthread.h>

/* 静态初始化（全局/静态变量）*/
pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;

/* 动态初始化 */
int pthread_mutex_init(pthread_mutex_t *mutex, const pthread_mutexattr_t *attr);

/* 加锁（阻塞等待）*/
int pthread_mutex_lock(pthread_mutex_t *mutex);

/* 尝试加锁（非阻塞）*/
int pthread_mutex_trylock(pthread_mutex_t *mutex);  /* 失败返回 EBUSY */

/* 解锁 */
int pthread_mutex_unlock(pthread_mutex_t *mutex);

/* 销毁（必须先解锁）*/
int pthread_mutex_destroy(pthread_mutex_t *mutex);

Mutex 类型

类型同线程重复加锁非Owner解锁适用场景
PTHREAD_MUTEX_NORMAL（默认）死锁未定义行为一般场景
PTHREAD_MUTEX_ERRORCHECK返回 EDEADLK返回错误调试阶段
PTHREAD_MUTEX_RECURSIVE允许（计数）返回错误递归函数

完整示例（保护计数器）

pthread_mutex_t mutex1 = PTHREAD_MUTEX_INITIALIZER;
int counter = 0;

void *functionC(void *arg) {
    pthread_mutex_lock(&mutex1);   /* 进入临界区 */
    counter++;
    printf("Counter: %d\n", counter);
    pthread_mutex_unlock(&mutex1); /* 离开临界区 */
    return NULL;
}
🚨

man7.org 原文警告："A mutex can never be owned by two different threads simultaneously. A thread attempting to lock a mutex that is already locked by another thread is suspended until the owning thread unlocks the mutex first." — 注意：加锁后必须解锁，任何代码路径（包括错误路径）都要保证最终解锁。

第 05 章 · 同步机制篇

条件变量 Condition Variable

条件变量允许线程在某个条件不满足时挂起等待，由其他线程在条件满足时唤醒，是实现"生产者-消费者"模式的核心机制。

📄 原文来源：man7.org — pthread_cond_init(3) Linux manual page
📌 导读

条件变量解决的是"我需要等待某个条件发生"的问题。如果只用 mutex，线程只能不断轮询（busy-wait，浪费 CPU）。条件变量让等待变成真正的休眠——零 CPU 消耗等待，被唤醒时精准继续。条件变量必须配合 mutex 使用，这是不可妥协的规则。

条件变量工作流程

⚡ 条件变量时序图（生产者-消费者）

核心 API

/* 静态初始化 */
pthread_cond_t cond = PTHREAD_COND_INITIALIZER;

/* 等待条件（原子释放锁 + 休眠）*/
int pthread_cond_wait(pthread_cond_t *cond, pthread_mutex_t *mutex);

/* 超时等待 */
int pthread_cond_timedwait(pthread_cond_t *cond, pthread_mutex_t *mutex,
                             const struct timespec *abstime);

/* 唤醒一个等待线程 */
int pthread_cond_signal(pthread_cond_t *cond);

/* 唤醒所有等待线程 */
int pthread_cond_broadcast(pthread_cond_t *cond);

正确使用模式（必须用 while，不能用 if）

/* ✅ 消费者线程正确写法 */
pthread_mutex_lock(&mutex);
while (condition_not_met) {          /* 必须是 while，不能是 if！*/
    pthread_cond_wait(&cond, &mutex); /* 原子释放锁并休眠 */
}                                    /* 唤醒后重新持锁，再次检查 */
/* --- 临界区：此时条件已满足，持有锁 --- */
do_work();
pthread_mutex_unlock(&mutex);

/* ✅ 生产者线程写法 */
pthread_mutex_lock(&mutex);
condition_not_met = 0;             /* 修改条件 */
pthread_cond_signal(&cond);        /* 发出信号 */
pthread_mutex_unlock(&mutex);
🚨

为什么必须用 while？ — man7.org 原文："A condition variable must always be associated with a mutex, to avoid the race condition where a thread prepares to wait on a condition variable and another thread signals the condition just before the first thread actually waits on it." 此外，虚假唤醒（spurious wakeup）是 POSIX 规范允许的：线程可能在没有信号的情况下被唤醒，while 循环可以再次检查，if 语句则会产生错误。

💡

man7.org 原文："pthread_cond_wait() atomically unlocks the mutex (as per pthread_mutex_unlock()) and waits for the condition variable cond to be signaled. The thread execution is suspended and does not consume any CPU time until the condition variable is signaled."

第 06 章 · 同步机制篇

信号量 POSIX Semaphore

信号量是一种计数型同步原语，可以控制对有限资源池的访问数量，并且是少数可以在信号处理函数中安全使用的同步机制之一。

📄 原文来源：man7.org — sem_overview(7) / CS 341 Illinois
📌 导读

信号量像一个"令牌池"：持有令牌才能进入，用完归还。二值信号量（初值为1）等效于 Mutex，计数信号量可以控制最大并发访问量（如连接池大小）。理解三者的等价性，选择最语义化的那个。

三种同步原语的等价关系

🔗 Mutex / 条件变量 / 信号量 等价关系

POSIX 无名信号量 API

#include <semaphore.h>

/* 初始化（pshared=0: 线程间; value: 初始值）*/
int sem_init(sem_t *sem, int pshared, unsigned int value);

/* 等待（原子性 value--，如果 value==0 则阻塞）*/
int sem_wait(sem_t *sem);

/* 非阻塞尝试 */
int sem_trywait(sem_t *sem);       /* 失败返回 EAGAIN */

/* 释放（原子性 value++，唤醒等待者）*/
int sem_post(sem_t *sem);         /* 可在信号处理函数中安全调用！*/

/* 销毁 */
int sem_destroy(sem_t *sem);

信号量作为 Mutex 使用（二值信号量）

/* 二值信号量 = Mutex（初值为 1）*/
sem_t semaphore;
sem_init(&semaphore, 0, 1);  /* 初值=1，最多1个线程进入 */

void *thread_body(void *arg) {
    int *shared = (int*)arg;
    sem_wait(&semaphore);    /* 进入临界区 */
    (*shared)++;
    sem_post(&semaphore);    /* 离开临界区 */
    return NULL;
}

信号量与 Mutex 的关键差异

特性MutexSemaphore
所有权谁加锁谁解锁任何线程可 post
信号处理函数安全❌ 不安全✅ sem_post 安全
计数能力❌ 二值✅ 任意非负整数
进程间共享需设置属性✅ pshared=1
主要用途保护临界区限流/资源池/事件通知
✅

CS 341 Illinois 原文："sem_post is one of a handful of functions that can be correctly used inside a signal handler; pthread_mutex_unlock is not." — 当需要在信号处理函数（SIGINT等）中唤醒等待线程时，只能使用 sem_post，而不是 pthread_mutex_unlock。

第 07 章 · 安全与陷阱篇

竞态条件与线程安全

竞态条件是多线程 Bug 中最隐蔽的一类——代码看起来正确，但结果依赖于线程调度顺序。本章深入分析其成因，并给出线程安全编程的系统性方法。

📄 原文来源：CMU POSIX Threads Tutorial + man7.org pthreads(7)
📌 导读

竞态条件的本质是：程序的行为取决于某些不受程序员控制的事件的相对顺序（如 OS 调度决策）。线程安全不是一个可以事后添加的属性，而是设计时就必须考虑的架构约束。

竞态条件根因分析

⚡ 竞态条件：counter++ 的危险执行序列

线程安全函数的定义

man7.org 原文："A thread-safe function is one that can be safely (i.e., it will deliver the same results regardless of whether it is) called from multiple threads at the same time."

线程安全的充分条件：

  • 不访问任何 static 或全局变量（或对其访问已受 mutex 保护）
  • 不返回指向 static 数据的指针（应返回动态分配或调用者提供的存储）
  • 函数是可重入的（Reentrant）

POSIX 标准中非线程安全函数（部分）

⚠️

man7.org 原文列出以下函数不保证线程安全（POSIX.1-2001/2008），使用时必须加锁或改用 _r（reentrant）版本：asctime(), ctime(), strtok(), gmtime(), localtime(), rand(), strerror(), readdir(), getenv(), ttyname() 等。安全替代：asctime_r(), ctime_r(), strtok_r(), gmtime_r(), localtime_r(), rand_r(), strerror_r(), readdir_r()。

线程安全实现策略

策略 1：消除共享状态

函数只使用栈上局部变量，不访问全局/静态数据。这是最彻底的线程安全方案。CMU 原文："any function that does not use static data or other shared resources is thread-safe."

策略 2：使用 _r 可重入版本

将内部 static 状态改为调用者提供的缓冲区。如 strtok_r()、strerror_r()。

策略 3：Mutex 保护临界区

对每个共享变量/数据结构关联一个 mutex，访问前加锁，访问后解锁。

策略 4：线程私有数据 (TSD)

将函数内部的全局/静态变量改为线程私有存储（pthread_key_t），每个线程拥有独立副本。

第 08 章 · 安全与陷阱篇

死锁：识别、分析与防范

死锁是多线程编程中最致命的 Bug——程序彻底停止响应，且通常无法从外部恢复。本章从 Coffman 四条件出发，建立系统性的死锁防范框架。

📄 原文来源：CMU — Linux Tutorial: POSIX Threads（Thread Pitfalls 节）
📌 导读

死锁的根源是循环等待。预防死锁的核心策略是破坏四个必要条件中的任意一个。最简单有效的方法是全局一致的加锁顺序——只要所有线程按相同顺序请求锁，循环等待就不可能发生。

死锁的四个必要条件（Coffman 条件）

🔴 死锁必要条件与破除方法

经典死锁场景

/* ❌ 死锁陷阱：两个线程以相反顺序加锁 */
void *function1(void *arg) {
    pthread_mutex_lock(&lock1);   /* 步骤1：获得lock1 */
    pthread_mutex_lock(&lock2);   /* 步骤3：等待lock2 → 死锁! */
    /* ... */
    pthread_mutex_unlock(&lock2);
    pthread_mutex_unlock(&lock1);
}

void *function2(void *arg) {
    pthread_mutex_lock(&lock2);   /* 步骤2：获得lock2 */
    pthread_mutex_lock(&lock1);   /* 步骤4：等待lock1 → 死锁! */
    /* ... */
    pthread_mutex_unlock(&lock1);
    pthread_mutex_unlock(&lock2);
}
🚨

CMU 原文："If function1 acquires the first mutex and function2 acquires the second, all resources are tied up and locked." — 这是最典型的ABBA死锁：Thread1持有A等待B，Thread2持有B等待A，双方永久等待。

死锁防范：trylock + 退避策略

/* ✅ CMU 推荐的 trylock 退避策略 */
pthread_mutex_lock(&mutex_1);
while (pthread_mutex_trylock(&mutex_2)) { /* 非阻塞尝试加 mutex_2 */
    pthread_mutex_unlock(&mutex_1);        /* 失败：释放已持有的锁 */
    /* 退避等待（避免活锁）*/
    usleep(1000);
    pthread_mutex_lock(&mutex_1);          /* 重新获取 mutex_1 */
}
/* 此处同时持有 mutex_1 和 mutex_2 */
count++;
pthread_mutex_unlock(&mutex_1);
pthread_mutex_unlock(&mutex_2);

死锁防范黄金法则

法则 1：全局一致的锁顺序（最重要）

所有线程按相同顺序申请多把锁。例如始终先锁 lock1 再锁 lock2，永不逆序。

法则 2：锁的粒度最小化

持有锁的时间越短越好。临界区内不做 I/O、不调用不可控函数。

法则 3：避免锁嵌套（能不嵌套就不嵌套）

持有锁时不申请其他锁，是完全避免死锁的最佳设计方案。

法则 4：使用超时等待代替无限阻塞

使用 pthread_mutex_timedlock() 或循环 trylock，设置超时以发现潜在死锁。

第 09 章 · 安全与陷阱篇

线程取消与信号处理

线程取消（cancellation）和信号（signal）处理在多线程环境下有特殊的语义和规则，处理不当会导致资源泄漏、数据损坏或未定义行为。

📄 原文来源：man7.org — pthreads(7)：Cancellation points & Async-cancel-safe
📌 导读

线程取消是外部强制终止线程的机制。POSIX 将其设计为"延迟取消"（在取消点处才实际取消），以保证资源能被正确清理。信号在多线程中有复杂的投递语义，需要建立专门的信号处理线程来统一管理。

线程取消机制

/* 向目标线程发送取消请求 */
int pthread_cancel(pthread_t thread);

/* 设置取消状态（线程可以禁用取消）*/
int pthread_setcancelstate(int state, int *oldstate);
/* state: PTHREAD_CANCEL_ENABLE (默认) 或 PTHREAD_CANCEL_DISABLE */

/* 设置取消类型 */
int pthread_setcanceltype(int type, int *oldtype);
/* type: PTHREAD_CANCEL_DEFERRED (默认) 或 PTHREAD_CANCEL_ASYNCHRONOUS */

/* 手动设置取消点（在禁用取消区间内设置检查点）*/
void pthread_testcancel(void);

取消点（Cancellation Points）

POSIX.1 规定了一系列"取消点"——当线程被设置为可取消且有延迟取消请求时，在调用这些函数时线程将被实际取消。

必须是取消点的函数（部分）：

read()    write()   sleep()   nanosleep()
recv()    send()    connect() accept()
open()    close()   wait()    waitpid()
pthread_cond_wait()  pthread_join()
sem_wait()  select()  poll()

取消清理处理器

/* 注册清理函数（线程取消或退出时自动调用）*/
pthread_cleanup_push(cleanup_func, arg);
/* ... 临界区代码（可能持有锁、打开文件）... */
pthread_cleanup_pop(1);  /* 1: 执行清理函数; 0: 仅移除 */

/* 清理函数示例：释放锁 */
void cleanup_handler(void *arg) {
    pthread_mutex_t *m = (pthread_mutex_t *)arg;
    pthread_mutex_unlock(m);  /* 防止被取消时锁未释放 */
}

多线程中的信号处理

📡 推荐的多线程信号处理架构
💡

man7.org 原文（NPTL）："With NPTL, all of the threads in a process are placed in the same thread group; all members of a thread group share the same PID." 信号可以发送给整个进程（由任意一个未屏蔽信号的线程处理）。推荐方案：主线程用 pthread_sigmask 屏蔽所有信号，创建专门的信号处理线程用 sigwait 同步等待，这样信号处理完全可控。

异步安全函数（Async-cancel-safe）

man7.org 原文："Only the following functions are required to be async-cancel-safe by POSIX.1-2001 and POSIX.1-2008：pthread_cancel(), pthread_setcancelstate(), pthread_setcanceltype()"

这意味着：使用异步取消（PTHREAD_CANCEL_ASYNCHRONOUS）时，几乎所有函数都不安全，实践中应坚持使用延迟取消（默认值）。

第 10 章 · 安全与陷阱篇

线程私有数据（TSD）

线程私有数据（Thread-Specific Data）提供了一种机制，让每个线程拥有全局键对应的独立数据副本，是将非线程安全的全局状态安全化的重要工具。

📄 原文来源：man7.org — pthread_key_create(3) Linux manual page
📌 导读

TSD 解决的问题是：某些库函数内部使用静态变量存储状态（如 errno、strtok 的分割指针），这在多线程中是危险的。TSD 让这种"全局键"在每个线程下指向独立的内存，像全局变量一样访问，但实际上是线程私有的。

TSD vs 全局变量 vs 局部变量

🗝 TSD 数据存储模型

TSD 核心 API

#include <pthread.h>

/* 创建线程私有数据键（全局一次性调用，通常在 pthread_once 中）*/
int pthread_key_create(pthread_key_t *key, void (*destructor)(void*));
/* destructor: 线程退出时自动调用，用于释放该线程的 TSD */

/* 设置当前线程的私有数据 */
int pthread_setspecific(pthread_key_t key, const void *value);

/* 获取当前线程的私有数据 */
void *pthread_getspecific(pthread_key_t key);

/* 删除键（通常在进程退出前调用）*/
int pthread_key_delete(pthread_key_t key);

实际应用：线程安全的错误上下文

static pthread_key_t  err_key;
static pthread_once_t key_once = PTHREAD_ONCE_INIT;

void free_err(void *ptr) { free(ptr); }

void make_key() {
    pthread_key_create(&err_key, free_err);  /* 全局只建一次 */
}

char *get_thread_error_buf() {
    pthread_once(&key_once, make_key);       /* 保证初始化一次 */
    char *buf = pthread_getspecific(err_key);
    if (!buf) {
        buf = malloc(256);
        pthread_setspecific(err_key, buf);    /* 首次使用时分配 */
    }
    return buf;
}

errno 的特殊处理

💡

man7.org 原文："As well as the stack, POSIX.1 specifies that various other attributes are distinct for each thread, including: the errno variable." — errno 已经被 glibc 实现为线程私有的宏，每个线程拥有独立的 errno 值，程序员无需为 errno 自行实现 TSD。这也解释了为什么 pthreads 函数不设置 errno 而直接返回错误码。

性能最佳实践

🏎 减少锁争用

缩短临界区时间；使用 读写锁（pthread_rwlock_t）在读多写少场景下大幅提升并发；考虑无锁数据结构（lock-free）。

🧵 合理设置线程数

CPU密集型任务：线程数 ≈ CPU核数；I/O密集型任务：线程数可以多于核数。过多线程导致上下文切换开销反而降低性能。

📦 使用线程池

频繁创建/销毁线程的开销可观。预先创建固定数量线程（线程池），通过任务队列分发工作，是高性能服务的标准模式。

🔄 避免伪共享（False Sharing）

不同线程访问同一缓存行（通常64字节）的不同变量，会导致缓存频繁失效。将各线程的数据结构做缓存行对齐（__attribute__((aligned(64)))）可以解决。

✅

每次代码审查时确认：
□ 所有全局/静态变量访问都有对应 mutex 保护
□ 所有 mutex_lock 对应 mutex_unlock，每条代码路径（含错误路径）
□ 条件变量等待使用 while 循环，而非 if
□ 不存在不同顺序的多重加锁（检查锁顺序一致性）

第 12 章 · 总结篇

第一性原理：多线程编程的战略性思维框架

回归本质，从最基本的公理出发，推导出多线程安全编程的核心原则，建立可以指导任何具体决策的战略性思维框架。

📌 导读

第一性原理思维要求我们不依赖类比或惯例，而是从最基础的事实出发，逐步推导出解决方案。多线程的所有问题都可以归结到一个根本矛盾：并发执行的需求 vs 共享状态的安全访问。下面从 5 条公理出发，重建整个知识体系。

公理层：不可违背的物理事实

公理 1

CPU 指令执行是唯一的原子单位

在硬件层面，只有单条机器指令（在特定条件下）是原子的。任何高级语言的"一行代码"都可能被分解为多条机器指令，从而被并发线程打断。

推论：所有非平凡的共享状态操作（读-改-写）都不是原子的，必须主动保护。
公理 2

操作系统调度器的决策对程序不可见

线程被调度的时机、顺序、运行时长完全由 OS 控制，程序无法预测和假设。代码的书写顺序不等于执行顺序。

推论：任何依赖特定线程执行顺序的程序都是脆弱的。必须用同步原语显式建立执行顺序约束。
公理 3

CPU 缓存与主内存存在可见性延迟

一个 CPU 核对内存的写入，不会立即对其他核可见。编译器和 CPU 都可能重排指令以优化性能。这是多核时代新增的复杂性。

推论：正确的同步原语（mutex, cond_wait, atomic）会建立 happens-before 关系，隐式解决可见性问题。不要绕过同步原语直接访问共享变量。

第一性原理推导层

原理 1

最小化共享状态

从公理1、2推导：共享状态越少，需要保护的临界区越少，出错概率越低，程序正确性越容易推理和验证。

战略指导：优先使用局部变量和函数参数传递数据；通过消息传递（管道、消息队列）代替共享内存；不得不共享时，最小化共享数据的范围和生命周期。

设计准则：一个函数需要访问的数据应该是参数，而不是全局变量。"我的函数使用了多少全局状态？"是线程安全审查的第一问题。
原理 2

所有共享可变状态必须有明确的所有权和访问协议

从公理1、2、3推导：每一个被多线程访问的可变数据，必须在设计时确定：谁"拥有"它？读取它需要什么？修改它需要什么？违反协议的代码路径是零容忍的。

战略指导：为每个共享数据结构关联一个 mutex，用注释或文档明确说明访问协议。代码审查时，每一个全局变量的每一次访问都应能指向对应的保护机制。

设计准则："这个数据是谁的，谁能改它，改的时候需要持有什么锁？" — 这三个问题的答案必须在代码中显而易见。
原理 3

锁的获取顺序必须全局一致

从公理2推导：死锁的唯一根本原因是循环等待。循环等待的充分必要条件是：存在两个或更多线程以不同顺序请求同一组锁。因此，消除不一致的加锁顺序就消除了死锁的可能性。

战略指导：建立全局锁层次（lock hierarchy）文档；高编号锁只能在低编号锁持有时申请；代码审查时验证所有加锁路径符合层次。

设计准则：系统中的所有锁应该有一个全局顺序编号。任何代码都不得在持有编号为 N 的锁时，去申请编号小于 N 的锁。
原理 4

用 while 检查条件，永远不信任唤醒本身

从公理2推导：线程被唤醒时，调度器不保证唤醒后条件仍然成立（其他线程可能在其获得锁之前改变了条件，或发生虚假唤醒）。唤醒只是"可能满足条件"的通知，而非"条件已满足"的断言。

战略指导：条件变量等待必须在 while 循环中，while 的条件就是线程继续执行所需的真实条件。

设计准则：pthread_cond_wait 调用必须写在 while(!condition) { ... } 中，这是不允许妥协的编码规范。
原理 5

线程的生死与资源释放必须完全闭合

从公理1、2推导：线程可以在任何取消点被取消，也可能因异常提前退出。如果资源释放依赖"正常执行到函数末尾"，则在异常路径下会泄漏。

战略指导：使用 pthread_cleanup_push/pop 注册清理函数；每个 mutex_lock 后立即安排对应的 cleanup handler；join 或 detach 每一个创建的线程。

设计准则：创建线程时同时确定其生命周期终止策略（join 还是 detach）。持有锁时进入可能阻塞的系统调用，必须注册清理处理器。

战略总结：多线程编程的思维层次

🏗 从第一性原理到工程实践的层次结构

终极判断标准

🎯

评判一段多线程代码是否正确，问自己三个问题：

1. 所有共享可变数据的每次访问，是否都在正确持有其对应保护锁的情况下进行？（对应原理1、2）

2. 所有锁的获取，是否遵循全局一致的顺序？（对应原理3）

3. 线程在任何退出路径上（正常、错误、取消），是否都能正确释放所有持有的资源？（对应原理5）

三个问题全部"是"，代码才算线程安全。
