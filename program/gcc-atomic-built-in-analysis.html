GCC __atomic 内置原子操作接口详解

基于 GCC 官方文档 · 内存模型感知的原子操作内置函数（Memory Model Aware Atomic Operations）

官方原文: gcc.gnu.org/onlinedocs/gcc/__atomic-Builtins.html

什么是 __atomic 内置函数？

GCC 从 4.7 版本起提供了 __atomic 前缀的内置原子操作函数，这些函数近似于 C++11 内存模型的要求，旨在取代旧的 __sync 系列内置函数。

新代码应始终使用 __atomic 内置函数而非 __sync 内置函数，主要区别在于内存序（memory order）可作为参数传入，从而精确控制同步语义。

✅
官方说明：这些函数假定程序符合 C++11 内存模型，特别是假定程序中不存在数据竞争（data race）。详见 C++11 标准。

支持的数据类型

__atomic 内置函数可用于任意长度为 1、2、4 或 8 字节的整型标量或指针类型。若架构支持 __int128，也允许使用 16 字节整型。

内存序模型（Memory Order）

6种内存序 · C++11 标准映射 · 约束强度说明

官方原文: gcc.gnu.org/onlinedocs/gcc/__atomic-Builtins.html

所有 __atomic 内置函数都接受一个 int memorder 参数（符号整数，仅低16位用于内存序）。以下 6 种内存序按约束强度升序排列。

__ATOMIC_RELAXED
无线程间顺序约束，仅保证操作本身原子性
__ATOMIC_CONSUME
当前以 ACQUIRE 实现（C++11语义缺陷所致）
__ATOMIC_ACQUIRE
建立 happens-before 约束；阻止代码提升到操作之前
__ATOMIC_RELEASE
建立 happens-before 约束；阻止代码下沉到操作之后
__ATOMIC_ACQ_REL
同时具备 ACQUIRE 和 RELEASE 的效果
__ATOMIC_SEQ_CST
全局全序，与所有 SEQ_CST 操作保持一致的顺序

x86 扩展内存序（HLE）

x86 架构额外支持硬件锁消除（Hardware Lock Elision）修饰符，可按位或叠加到标准内存序：

修饰符说明要求
__ATOMIC_HLE_ACQUIRE在锁变量上开始锁消除内存序须为 ACQUIRE 或更强
__ATOMIC_HLE_RELEASE在锁变量上结束锁消除内存序须为 RELEASE 或更强
 
// x86 HLE 示例（来自官方文档）
#include <immintrin.h>
int lockvar;

/* 使用 HLE 获取锁 */
while (__atomic_exchange_n(&lockvar, 1,
       __ATOMIC_ACQUIRE | __ATOMIC_HLE_ACQUIRE))
    _mm_pause(); /* 中止失败的事务 */

/* 使用 HLE 释放锁 */
__atomic_store_n(&lockvar, 0,
    __ATOMIC_RELEASE | __ATOMIC_HLE_RELEASE);

① 加载操作（Load）

原子读取内存中的值，保证读操作不被撕裂

官方原文链接
📖
加载操作（Load Operations）
共 2 个接口
type __atomic_load_n(type *ptr, int memorder)
原子加载操作（类型化版本）。返回 *ptr 的值。这是最常用的原子读取函数。
官方原文：This built-in function implements an atomic load operation. It returns the contents of *ptr.

有效内存序： __ATOMIC_RELAXED __ATOMIC_CONSUME __ATOMIC_ACQUIRE __ATOMIC_SEQ_CST 

// 示例：原子读取共享计数器
volatile int shared_counter = 0;

int val = __atomic_load_n(&shared_counter, __ATOMIC_ACQUIRE);
// val 保证是一个完整的读取结果，不会看到"撕裂"的值
void __atomic_load(type *ptr, type *ret, int memorder)
泛型版本的原子加载。将 *ptr 的内容存入 *ret。适用于任意数据类型（包括结构体等大型类型），当大小支持无锁操作时使用无锁指令，否则通过外部调用解决。
官方原文：This is the generic version of an atomic load. It returns the contents of *ptr in *ret.
// 示例：泛型版本，适用于非基础类型
struct Point { int x, y; } shared_pt = {0, 0};
struct Point local_pt;

__atomic_load(&shared_pt, &local_pt, __ATOMIC_SEQ_CST);
// local_pt 现在持有 shared_pt 的原子副本

使用场景

场景推荐内存序说明
读取统计计数器（不需要同步） RELAXED 仅需原子性，不需要顺序保证，性能最优
读取受保护的共享数据指针 ACQUIRE 配合写端的 RELEASE，建立 happens-before
读取全局状态标志（需要全局一致性） SEQ_CST 最强保证，确保所有线程看到相同顺序
读取依赖链数据（如 RCU） CONSUME 理论上是数据依赖顺序，目前以 ACQUIRE 实现
// 安全

② 存储操作（Store）

原子写入值到内存，保证写操作不被撕裂

官方原文链接
✍️
存储操作（Store Operations）
共 2 个接口
void __atomic_store_n(type *ptr, type val, int memorder)
原子存储操作（类型化版本）。将 val 写入 *ptr。
官方原文：This built-in function implements an atomic store operation. It writes val into *ptr.

有效内存序： __ATOMIC_RELAXED __ATOMIC_RELEASE __ATOMIC_SEQ_CST 

// 示例：发布共享数据
int data = 42;
volatile int ready = 0;

data = 42; // 先准备数据
__atomic_store_n(&ready, 1, __ATOMIC_RELEASE);
// RELEASE 保证：data=42 在 ready=1 之前对其他线程可见
void __atomic_store(type *ptr, type *val, int memorder)
泛型版本的原子存储。将 *val 的值存入 *ptr。适用于任意数据类型。
官方原文：This is the generic version of an atomic store. It stores the value of *val into *ptr.
// 示例：泛型版本存储结构体
struct Config { int timeout; int retries; } new_cfg = {30, 3};
struct Config shared_cfg;

__atomic_store(&shared_cfg, &new_cfg, __ATOMIC_RELEASE);

使用场景

场景推荐内存序说明
更新统计计数（不关心顺序）RELAXED最快，仅保证写操作原子
发布数据（通知其他线程数据就绪）RELEASE与读端 ACQUIRE 配合，保证数据可见
需要全局全序的状态更新SEQ_CST最安全，最保守，开销最大
初始化后发布指针RELEASE典型的发布-订阅模式
💡
最佳实践：在"生产者-消费者"模式中，生产者使用 __ATOMIC_RELEASE 写入标志，消费者使用 __ATOMIC_ACQUIRE 读取标志，形成同步对，是实现无锁通信的标准模式。

③ 交换操作（Exchange）

原子地写入新值并返回旧值，是读-改-写操作

官方原文链接
🔄
交换操作（Exchange Operations）
共 2 个接口
type __atomic_exchange_n(type *ptr, type val, int memorder)
原子交换操作（类型化版本）。将 val 写入 *ptr，并返回 *ptr 的原始值。这是一个原子的读-改-写（RMW）操作。
官方原文：This built-in function implements an atomic exchange operation. It writes val into *ptr, and returns the previous contents of *ptr. All memory order variants are valid.

有效内存序：所有内存序均有效（RELAXED / CONSUME / ACQUIRE / RELEASE / ACQ_REL / SEQ_CST）

// 示例1：实现自旋锁
int lock = 0;

// 加锁：原子地将 1 写入 lock，若返回 0 则获得锁
while (__atomic_exchange_n(&lock, 1, __ATOMIC_ACQUIRE) != 0) {
    /* 自旋等待 */
}

// 临界区...

// 解锁
__atomic_store_n(&lock, 0, __ATOMIC_RELEASE);

// 示例2：原子地替换指针
void *old_ptr = __atomic_exchange_n(&shared_ptr, new_ptr, __ATOMIC_ACQ_REL);
void __atomic_exchange(type *ptr, type *val, type *ret, int memorder)
泛型版本的原子交换。将 *val 存入 *ptr，原始的 *ptr 值复制到 *ret。
官方原文：This is the generic version of an atomic exchange. It stores the contents of *val into *ptr. The original value of *ptr is copied into *ret.
// 示例：泛型版交换
int new_val = 99;
int old_val;
__atomic_exchange(&shared_var, &new_val, &old_val, __ATOMIC_SEQ_CST);
// old_val 现在是 shared_var 的旧值

使用场景

场景推荐内存序说明
实现自旋锁（acquire）ACQUIRE确保锁后面的代码不被重排到锁之前
原子替换消息队列头指针ACQ_REL既需要看到之前的写，又需要发布自己的写
swap 操作（不关心同步）RELAXED仅需原子性，不关心线程间顺序
单次触发标志ACQ_REL保证设置者与消费者之间的同步

④ 比较并交换（Compare-and-Exchange / CAS）

最强大的原子原语，无锁算法的基石

官方原文链接
🔀
比较并交换操作（CAS）
共 2 个接口（各有 weak/strong 变体）
bool __atomic_compare_exchange_n(type *ptr, type *expected, type desired, bool weak, int success_memorder, int failure_memorder)
原子比较并交换操作（类型化版本）。比较 *ptr 与 *expected 的值：
  • 相等时：执行读-改-写，将 desired 写入 *ptr，返回 true，内存序按 success_memorder
  • 不等时：执行读操作，将 *ptr 当前值写入 *expected，返回 false，内存序按 failure_memorder
weak=true 为弱变体（可能伪失败），weak=false 为强变体（从不伪失败）。多数目标平台只提供强变体。
官方原文：weak is true for weak compare_exchange, which may fail spuriously, and false for the strong variation, which never fails spuriously. Many targets only offer the strong variation and ignore the parameter. When in doubt, use the strong variation.
⚠️
failure_memorder 不能是 __ATOMIC_RELEASE 或 __ATOMIC_ACQ_REL，且其强度不能超过 success_memorder。
// 示例1：无锁栈的 push 操作（CAS 循环）
typedef struct Node { int val; struct Node *next; } Node;
Node *stack_top = NULL;

void push(Node *node) {
    Node *old_top;
    do {
        old_top = __atomic_load_n(&stack_top, __ATOMIC_RELAXED);
        node->next = old_top;
    } while (!__atomic_compare_exchange_n(
        &stack_top,          // ptr: 目标
        &old_top,            // expected: 期望值（失败时被更新）
        node,                // desired: 新值
        false,               // weak=false: 强变体，不伪失败
        __ATOMIC_RELEASE,    // success: RELEASE 发布写
        __ATOMIC_RELAXED     // failure: 只需要读
    ));
}

// 示例2：原子地递增（仅当当前值匹配时）
int expected = __atomic_load_n(&counter, __ATOMIC_RELAXED);
while (!__atomic_compare_exchange_n(&counter, &expected,
       expected + 1, false,
       __ATOMIC_SEQ_CST, __ATOMIC_RELAXED)) {
    /* expected 已被更新为当前值，重试 */
}
bool __atomic_compare_exchange(type *ptr, type *expected, type *desired, bool weak, int success_memorder, int failure_memorder)
泛型版本的 CAS。与 __atomic_compare_exchange_n 几乎完全相同，区别在于 desired 也是指针形式。
官方原文：The function is virtually identical to __atomic_compare_exchange_n, except the desired value is also a pointer.

使用场景

场景推荐变体说明
无锁数据结构（链表、栈、队列）strong, RELEASE/RELAXED在 CAS 循环中实现无锁 push/pop
单例初始化（double-check）strong, ACQ_REL/ACQUIRE保证只有一个线程初始化

⑤ 算术与位运算（返回新值）

__atomic_op_fetch 系列：先操作后返回新值

官方原文链接
➕
返回新值的原子算术/位运算（op_fetch）
共 6 个接口：add / sub / and / xor / or / nand
ℹ️
语义： { *ptr op= val; return *ptr; }（nand 特殊：{ *ptr = ~(*ptr & val); return *ptr; }）
约束：第一个参数指向的对象必须是整型或指针类型，不能是 bool 类型。所有内存序均有效。
type __atomic_add_fetch(type *ptr, type val, int memorder)

原子地执行 *ptr += val，返回新值（加法后的结果）。

// 示例：原子递增，获取新值
int new_count = __atomic_add_fetch(&counter, 1, __ATOMIC_SEQ_CST);
// new_count 是递增后的值
type __atomic_sub_fetch(type *ptr, type val, int memorder)

原子地执行 *ptr -= val，返回新值（减法后的结果）。

// 示例：引用计数减 1，当返回 0 时触发释放
if (__atomic_sub_fetch(&ref_count, 1, __ATOMIC_RELEASE) == 0) {
    __atomic_thread_fence(__ATOMIC_ACQUIRE);
    free_object(); // 安全释放
}
type __atomic_and_fetch(type *ptr, type val, int memorder)

原子地执行 *ptr &= val，返回新值（AND 后的结果）。

// 示例：原子清除标志位，验证清除后的状态
unsigned int new_flags = __atomic_and_fetch(&flags, ~FLAG_BUSY, __ATOMIC_RELEASE);
type __atomic_xor_fetch(type *ptr, type val, int memorder)
type __atomic_or_fetch(type *ptr, type val, int memorder)
type __atomic_nand_fetch(type *ptr, type val, int memorder)

分别原子地执行 XOR、OR、NAND 操作，返回新值。

// XOR: 原子切换标志位（toggle），返回新状态
unsigned new_state = __atomic_xor_fetch(&flags, TOGGLE_BIT, __ATOMIC_SEQ_CST);

// OR: 原子设置标志位，返回新状态
__atomic_or_fetch(&status, STATUS_READY, __ATOMIC_RELEASE);

// NAND: *ptr = ~(*ptr & val)，返回新值（较少用）
__atomic_nand_fetch(&bits, mask, __ATOMIC_SEQ_CST);

⑥ 算术与位运算（返回旧值）

__atomic_fetch_op 系列：先获取旧值再操作

官方原文链接
🔢
返回旧值的原子算术/位运算（fetch_op）
共 6 个接口：fetch_add / fetch_sub / fetch_and / fetch_xor / fetch_or / fetch_nand
ℹ️
语义： { tmp = *ptr; *ptr op= val; return tmp; }（nand 特殊：{ tmp = *ptr; *ptr = ~(*ptr & val); return tmp; }）
约束：与 op_fetch 相同，所有内存序均有效。指针运算按 uintptr_t 处理，不按类型大小缩放。
type __atomic_fetch_add(type *ptr, type val, int memorder)

原子地执行 *ptr += val，返回旧值（加法前的值）。

// 示例1：为任务分配唯一 ID（返回的是"领到"的 ID）
int my_id = __atomic_fetch_add(&next_task_id, 1, __ATOMIC_RELAXED);
// my_id 是我领到的 ID，next_task_id 已经变为 my_id+1

// 示例2：实现信号量 V 操作（post）
__atomic_fetch_add(&semaphore, 1, __ATOMIC_RELEASE);
type __atomic_fetch_sub(type *ptr, type val, int memorder)

原子地执行 *ptr -= val，返回旧值。

// 示例：引用计数减 1（注意：sub_fetch 才能直接判断是否到 0）
int prev = __atomic_fetch_sub(&ref_count, 1, __ATOMIC_RELEASE);
if (prev == 1) {
    // 之前是 1，现在是 0，我是最后一个释放者
    __atomic_thread_fence(__ATOMIC_ACQUIRE);
    destroy_object();
}
type __atomic_fetch_and(type *ptr, type val, int memorder)
type __atomic_fetch_xor(type *ptr, type val, int memorder)
type __atomic_fetch_or(type *ptr, type val, int memorder)
type __atomic_fetch_nand(type *ptr, type val, int memorder)

分别原子地执行 AND、XOR、OR、NAND，返回旧值。

// fetch_and: 清除位并获取旧状态（判断位之前是否被设置）
unsigned old = __atomic_fetch_and(&flags, ~LOCK_BIT, __ATOMIC_ACQUIRE);
if (old & LOCK_BIT) { /* 之前有锁，现在解了 */ }

// fetch_or: 设置位并获取旧状态
unsigned old_status = __atomic_fetch_or(&status, READY_BIT, __ATOMIC_RELEASE);
if (!(old_status & READY_BIT)) { /* 我是第一个设置 READY 的 */ }

// fetch_xor: 切换位，获取旧值
unsigned prev = __atomic_fetch_xor(&toggle, BIT, __ATOMIC_SEQ_CST);

使用场景对比汇总

函数语义典型场景
fetch_add返回旧值后加唯一 ID 分配、信号量 post
fetch_sub返回旧值后减引用计数（判断旧值是否为1）
fetch_and返回旧值后 AND原子清除位，检查旧位状态
fetch_or返回旧值后 OR原子设置位，检查是否首次设置
fetch_xor返回旧值后 XOR原子切换位，获取切换前状态
fetch_nand返回旧值后 NAND特殊位操作（较少用）

⑦ 测试与清除（Test-and-Set / Clear）

原子标志位操作，用于实现自旋锁和简单互斥

官方原文链接
🚦
测试与清除（Test-and-Set / Clear）
共 2 个接口
bool __atomic_test_and_set(void *ptr, int memorder)
原子测试并设置操作。对 *ptr 处的字节执行原子 test-and-set：将该字节设置为某个实现定义的非零"已设置"值，返回 true 当且仅当之前的内容为"已设置"状态。
官方原文：This built-in function performs an atomic test-and-set operation on the byte at *ptr. The byte is set to some implementation defined nonzero "set" value and the return value is true if and only if the previous contents were "set". It should be only used for operands of type bool or char.
⚠️
仅应用于 bool 或 char 类型的操作数。对于其他类型，可能只有部分值被设置。所有内存序均有效。
// 示例：用 test_and_set 实现自旋锁
bool spinlock = false;

// 加锁：若返回 false（之前未设置），则成功获得锁
while (__atomic_test_and_set(&spinlock, __ATOMIC_ACQUIRE)) {
    /* 等待：让出 CPU 或 pause */
}

// 临界区代码...

// 解锁
__atomic_clear(&spinlock, __ATOMIC_RELEASE);
void __atomic_clear(bool *ptr, int memorder)
原子清除操作。对 *ptr 执行原子清除操作，操作完成后 *ptr 包含 0。应配合 __atomic_test_and_set 使用。对于非 bool 类型，可能只清除部分值，优先使用 __atomic_store。
官方原文：After the operation, *ptr contains 0. It should be only used for operands of type bool or char and in conjunction with __atomic_test_and_set. For other types it may only clear partially. If the type is not bool prefer using __atomic_store.

有效内存序： __ATOMIC_RELAXED __ATOMIC_RELEASE __ATOMIC_SEQ_CST 

// 示例：配合 test_and_set 的完整自旋锁实现
typedef bool spinlock_t;

static inline void spin_lock(spinlock_t *lock) {
    while (__atomic_test_and_set(lock, __ATOMIC_ACQUIRE))
        ; /* 自旋 */
}

static inline void spin_unlock(spinlock_t *lock) {
    __atomic_clear(lock, __ATOMIC_RELEASE);
}

⑧ 内存屏障（Memory Fence）

线程间与信号处理器间的同步栅栏

官方原文链接
🚧
内存屏障（Fence Operations）
共 2 个接口
void __atomic_thread_fence(int memorder)
线程间同步屏障。基于指定内存序在线程之间充当同步栅栏（fence）。所有内存序均有效。
官方原文：This built-in function acts as a synchronization fence between threads based on the specified memory order.
ℹ️
注意（来自官方）：在 C++11 内存模型中，栅栏（fence）与特定内存位置上的其他原子操作组合才能生效；仅凭栅栏本身不足以建立 happens-before 关系，还需配合特定内存位置上的原子操作使用。
// 示例1：独立的 ACQUIRE 屏障（配合 relaxed 加载）
int val = __atomic_load_n(&data, __ATOMIC_RELAXED);
__atomic_thread_fence(__ATOMIC_ACQUIRE);
// 屏障后面的代码不会被重排到屏障之前
use(val);

// 示例2：引用计数释放的典型模式（来自 Boost/Rust 惯用法）
if (__atomic_fetch_sub(&ref_count, 1, __ATOMIC_RELEASE) == 1) {
    // 确保看到所有之前的释放写入
    __atomic_thread_fence(__ATOMIC_ACQUIRE);
    destroy(); // 安全销毁
}

// 示例3：SEQ_CST 屏障保证全局顺序
__atomic_thread_fence(__ATOMIC_SEQ_CST);
void __atomic_signal_fence(int memorder)
线程与信号处理器间的同步屏障。在同一线程内的普通代码与信号处理器之间充当同步栅栏。所有内存序均有效。
官方原文：This built-in function acts as a synchronization fence between a thread and signal handlers based in the same thread.
💡
与 thread_fence 的区别：signal_fence 只在编译器层面阻止重排（不发出硬件屏障指令），因为信号处理器在同一线程中运行，无需跨 CPU 核心同步，开销更低。
// 示例：安全地在信号处理器与主线程间共享数据
volatile int signal_flag = 0;
int signal_data = 0;

// 在信号处理器中（写端）
void handler(int sig) {
    signal_data = 42;
    __atomic_signal_fence(__ATOMIC_RELEASE); // 编译器屏障
    __atomic_store_n(&signal_flag, 1, __ATOMIC_RELAXED);
}

// 在主线程中（读端）
while (!__atomic_load_n(&signal_flag, __ATOMIC_RELAXED))
    ;
__atomic_signal_fence(__ATOMIC_ACQUIRE);
use(signal_data); // 安全读取

无锁能力查询（Lock-Free Query）
共 2 个接口
bool __atomic_always_lock_free(size_t size, void *ptr)
编译期无锁查询。若目标架构对指定字节大小的对象始终生成无锁原子指令，返回 true。size 必须在编译期可确定为常量，结果也在编译期确定。
官方原文：This built-in function returns true if objects of size bytes always generate lock-free atomic instructions for the target architecture. size must resolve to a compile-time constant and the result also resolves to a compile-time constant.

ptr 是可选的对象指针，用于对齐判断。传 0 表示使用典型对齐，编译器也可能忽略该参数。

// 示例：编译期条件选择实现路径
if (__atomic_always_lock_free(sizeof(long long), 0)) {
    // 此平台 64 位原子操作保证无锁，可安全使用
    __atomic_fetch_add(&counter64, 1, __ATOMIC_RELAXED);
} else {
    // 降级为加锁保护
    pthread_mutex_lock(&mutex);
    counter64++;
    pthread_mutex_unlock(&mutex);
}

// 常见查询示例（来自官方）
if (__atomic_always_lock_free(sizeof(long long), 0))
    // ...
bool __atomic_is_lock_free(size_t size, void *ptr)
运行期无锁查询。若指定字节大小的对象始终生成无锁原子指令，返回 true。若编译器无法确定，则在运行时调用名为 __atomic_is_lock_free 的运行时例程。
官方原文：If the built-in function is not known to be lock-free, a call is made to a runtime routine named __atomic_is_lock_free.
// 示例：运行期根据对象对齐情况判断
struct MyData data;
if (__atomic_is_lock_free(sizeof(data), &data)) {
    // 此对象可以无锁原子访问
    __atomic_store_n((int*)&data.field, 42, __ATOMIC_RELAXED);
}

场景二：引用计数（Reference Counting）

int ref_count = 1;

// retain（增加引用）
void retain() {
    __atomic_fetch_add(&ref_count, 1, __ATOMIC_RELAXED);
    // RELAXED 足够：只需保证计数增加的原子性
}

// release（释放引用）
void release() {
    if (__atomic_fetch_sub(&ref_count, 1, __ATOMIC_RELEASE) == 1) {
        // 我是最后一个：之前的 RELEASE 保证了之前的所有写对我可见
        __atomic_thread_fence(__ATOMIC_ACQUIRE); // 与所有 RELEASE 同步
        destroy(); // 安全析构
    }
}

场景三：无锁栈（Lock-Free Stack）

typedef struct Node { int val; struct Node *next; } Node;
Node *top = NULL; // 栈顶（共享）

// push
void push(Node *node) {
    Node *old;
    do {
        old = __atomic_load_n(&top, __ATOMIC_RELAXED);
        node->next = old;
    } while (!__atomic_compare_exchange_n(&top, &old, node,
               false, __ATOMIC_RELEASE, __ATOMIC_RELAXED));
}

// pop（注意：存在 ABA 问题，实际需要 hazard pointer 等）
Node* pop() {
    Node *old, *next;
    do {
        old = __atomic_load_n(&top, __ATOMIC_ACQUIRE);
        if (!old) return NULL;
        next = old->next;
    } while (!__atomic_compare_exchange_n(&top, &old, next,
               false, __ATOMIC_ACQUIRE, __ATOMIC_RELAXED));
    return old;
}

场景四：单次初始化（Once Initialization）

int initialized = 0;
SomeObject *obj = NULL;

SomeObject* get_instance() {
    if (!__atomic_load_n(&initialized, __ATOMIC_ACQUIRE)) {
        int expected = 0;
        if (__atomic_compare_exchange_n(&initialized, &expected, 1,
                false, __ATOMIC_ACQ_REL, __ATOMIC_ACQUIRE)) {
            // 我赢了 CAS：负责初始化
            obj = create_object();
            __atomic_store_n(&initialized, 2, __ATOMIC_RELEASE);
        } else {
            // 等待初始化完成
            while (__atomic_load_n(&initialized, __ATOMIC_ACQUIRE) != 2)
                ; /* 自旋 */
        }
    }
    return obj;
}

内存序选择指南

场景描述推荐内存序理由
仅需原子性，不关心线程顺序（如统计计数）RELAXED最低开销，不产生硬件屏障
生产者写入数据后发布标志写用 RELEASE阻止写操作下沉到标志写之后
消费者读取标志后访问数据读用 ACQUIRE阻止读操作提升到标志读之前
既读又写（如 exchange 实现的锁）ACQ_REL同时具备 ACQUIRE 和 RELEASE 效果
需要所有线程看到全局一致顺序SEQ_CST最强保证，开销最大
CAS 成功路径任意合适的序无限制，按业务需求选择
CAS 失败路径≤成功序且非 REL/ACQ_REL失败仅是读，不需要写顺序
🎯
选择原则：从最弱的内存序开始考虑，只在需要时增强。过度使用 SEQ_CST 会在部分架构（如 ARM、POWER）产生显著性能损失。但在不确定时，SEQ_CST 始终是安全的选择。
