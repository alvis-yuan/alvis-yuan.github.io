多线程工程化决策框架

在C语言多线程工程中，锁（Lock）不是第一选择，而是最后手段。正确的工程决策应当优先消除共享状态，在不可避免时才逐步引入同步原语。

以下决策路径源自 nginx、memcached、Redis、Linux Kernel 等生产级开源项目的实践。

①
分析数据流
数据从哪里来？经过哪些处理？到哪里去？理解数据流是一切并发设计的起点。
②
切分状态 —— Share-Nothing
能否将数据分片，让每个线程独立处理自己的分片？彻底消除共享状态。
③
转移所有权 —— 指针传递
若必须共享，能否通过传递指针来转移所有权，而非多线程同时读写同一数据？
④
无锁数据结构 —— SPSC 环形队列
若必须通信，能否用单生产者单消费者的无锁环形队列解决？利用内存屏障代替锁。
⑤
批量 + 锁 —— 最后手段
若必须用锁，务必使用批量操作来降低锁粒度，并严格规定加锁层级以防止死锁。
场景 ①

Share-Nothing 数据分片

核心思想：将整个数据集按某个维度（如 socket fd、用户 ID、哈希桶）分片，每个线程/进程只处理属于自己分片的数据，彻底消除线程间的共享状态与竞争。

📦 典型案例：nginx Worker 进程模型

nginx 是 Share-Nothing 模式最经典的工程实践。每个 Worker 进程绑定特定 CPU 核，独立运行 epoll 事件循环，处理自己的网络连接，进程间不共享内存（配置除外，以只读方式继承自 master）。

📖 权威来源：AOSA Book — nginx Architecture nginx uses multiplexing and event notifications heavily, and dedicates specific tasks to separate processes. Connections are processed in a highly efficient run-loop in a limited number of single-threaded processes called workers. Within each worker nginx can handle many thousands of concurrent connections and requests per second. — aosabook.org/en/v2/nginx.html

🔧 完整源码示例

以下代码模拟 nginx Share-Nothing Worker 模型：每个线程绑定一组连接，使用独立的事件循环处理，完全无锁。

share_nothing_worker.c — 模拟 nginx Worker 分片模型 C 
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <pthread.h>
#include <unistd.h>
#include <stdint.h>

#define NUM_WORKERS   4
#define CONNS_PER_WKR 1000   /* 每个 Worker 独立处理的连接数 */
#define TOTAL_CONNS   (NUM_WORKERS * CONNS_PER_WKR)

/* ── 连接对象（每个 Worker 独自拥有，不跨 Worker 共享）── */
typedef struct {
    int      fd;
    uint64_t bytes_in;
    uint64_t bytes_out;
    char     buf[1024];
} conn_t;

/* ── Worker 上下文（完全独立，没有跨线程共享字段）── */
typedef struct {
    int     worker_id;
    int     cpu_core;           /* 绑定的 CPU 核 */
    conn_t *conns;              /* 此 Worker 独享的连接池 */
    int     conn_count;
    uint64_t total_requests;   /* 私有统计，无需加锁 */
} worker_ctx_t;

/* ── 模拟事件循环处理请求（实际中为 epoll_wait + 回调）── */
static void worker_event_loop(worker_ctx_t *ctx) {
    for (int round = 0; round < 10; round++) {
        for (int i = 0; i < ctx->conn_count; i++) {
            conn_t *c = &ctx->conns[i];
            /* 完全在本线程私有数据上操作，无需任何锁 */
            c->bytes_in  += 64;
            c->bytes_out += 32;
            ctx->total_requests++;
        }
    }
}

/* ── Worker 线程入口 ── */
static void *worker_thread(void *arg) {
    worker_ctx_t *ctx = (worker_ctx_t *)arg;

    /* 分配独享连接池（在本线程栈/堆上，无需共享） */
    ctx->conns = calloc(ctx->conn_count, sizeof(conn_t));
    for (int i = 0; i < ctx->conn_count; i++) {
        ctx->conns[i].fd = ctx->worker_id * CONNS_PER_WKR + i;
    }

    printf("[Worker %d] 启动，CPU核%d，负责连接 fd[%d..%d]\n",
           ctx->worker_id, ctx->cpu_core,
           ctx->conns[0].fd,
           ctx->conns[ctx->conn_count - 1].fd);

    /* 运行独立事件循环，不与任何其他 Worker 交互 */
    worker_event_loop(ctx);

    printf("[Worker %d] 完成，处理请求: %lu\n",
           ctx->worker_id, ctx->total_requests);

    free(ctx->conns);
    return NULL;
}

int main(void) {
    printf("=== Share-Nothing Worker 模型 (模拟 nginx) ===\n");
    printf("Worker数: %d，每Worker连接数: %d\n\n",
           NUM_WORKERS, CONNS_PER_WKR);

    pthread_t    threads[NUM_WORKERS];
    worker_ctx_t ctxs[NUM_WORKERS];

    for (int i = 0; i < NUM_WORKERS; i++) {
        ctxs[i].worker_id   = i;
        ctxs[i].cpu_core    = i;          /* 实际可用 sched_setaffinity */
        ctxs[i].conn_count  = CONNS_PER_WKR;
        ctxs[i].total_requests = 0;
        ctxs[i].conns       = NULL;

        pthread_create(&threads[i], NULL, worker_thread, &ctxs[i]);
    }

    for (int i = 0; i < NUM_WORKERS; i++) {
        pthread_join(threads[i], NULL);
    }

    uint64_t total = 0;
    for (int i = 0; i < NUM_WORKERS; i++)
        total += ctxs[i].total_requests;

    printf("\n✓ 全部 Worker 完成，总请求数: %lu\n", total);
    printf("✓ 整个过程未使用任何锁！\n");
    return 0;
}

⚖️ 原理说明

Share-Nothing 的本质是 "消除竞争条件的根源"：如果没有共享数据，就没有数据竞争，因此无需任何同步原语。nginx 通过 fork() 创建 Worker 进程（而非线程），进程间天然隔离内存空间。配置数据在 fork 之前由 master 读取，Worker 以只读方式继承，无需加锁。

在实际应用中，数据分片的维度可以是：连接 fd、用户 ID 取模、哈希桶编号、时间片等任何能将工作负载均匀划分的维度。

    ✓ 优势
  • 零锁开销，无竞争条件
  • 天然支持 CPU 亲和性，缓存命中率极高
  • 无死锁风险
  • 横向扩展线性增长
  • 调试简单，行为可预测
    ✗ 劣势
  • 负载不均衡时可能出现热点
  • 全局状态（如统计）需额外汇总
  • 数据重新分片代价高
  • 适合 CPU 密集 / 网络 I/O，不适合高度依赖共享状态的场景
🎯 何时使用 网络服务器（nginx）、数据分片处理（日志聚合、哈希分片缓存）、每核独立任务队列等场景。每个连接/请求的生命周期内只需单线程处理即可。
场景 ②

所有权转移（指针传递）

核心思想：当数据必须从一个线程传递到另一个线程时，通过转移指针所有权实现"接力棒"式传递——在任意时刻，只有一个线程拥有并操作该数据，彻底避免多线程并发读写同一对象。

📦 典型案例：memcached 调度线程 → Worker 线程

memcached 有一个主调度线程（dispatcher）和多个 Worker 线程。新连接到达时，dispatcher 创建一个 CQ_ITEM（连接描述符包装对象），将指针压入对应 Worker 的无锁队列，然后通过 pipe/eventfd 通知 Worker。之后，dispatcher 不再访问该对象，所有权已转移给 Worker。

📖 权威来源：memcached/thread.c dispatch_conn_new() 创建 CQ_ITEM 后，调用 cq_push(thread->new_conn_queue, item) 并通过 write(notify_send_fd) 通知 Worker。整个过程是指针的单向转移。 github.com/memcached/memcached/blob/master/thread.c

🔧 完整源码示例

ownership_transfer.c — 模拟 memcached 调度线程→Worker 所有权转移 C 
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <pthread.h>
#include <unistd.h>
#include <stdatomic.h>

#define NUM_WORKERS  3
#define TOTAL_CONNS  12
#define QUEUE_CAP    32

/* ── 连接项（memcached 中的 CQ_ITEM）─────────────────── */
typedef struct {
    int  fd;
    char remote_ip[32];
    int  proto;
} cq_item_t;

/* ── 极简单向锁队列（实际 memcached 使用无锁实现）────── */
typedef struct {
    cq_item_t   *items[QUEUE_CAP];
    volatile int head, tail;
    pthread_mutex_t lock;
} cq_t;

static void cq_init(cq_t *q) {
    q->head = q->tail = 0;
    pthread_mutex_init(&q->lock, NULL);
}

/* 压入指针（所有权转移给队列，调用方不再访问）*/
static void cq_push(cq_t *q, cq_item_t *item) {
    pthread_mutex_lock(&q->lock);
    q->items[q->tail % QUEUE_CAP] = item;
    q->tail++;
    pthread_mutex_unlock(&q->lock);
}

/* 取出指针（所有权转移给调用方）*/
static cq_item_t *cq_pop(cq_t *q) {
    pthread_mutex_lock(&q->lock);
    cq_item_t *item = NULL;
    if (q->head < q->tail) {
        item = q->items[q->head % QUEUE_CAP];
        q->head++;
    }
    pthread_mutex_unlock(&q->lock);
    return item;
}

/* ── Worker 上下文 ───────────────────────────────────── */
typedef struct {
    int   id;
    cq_t  queue;
    int   notify_pipe[2];  /* [0]=读端，[1]=写端（模拟 memcached pipe）*/
    int   done;            /* 调度结束标志 */
} worker_t;

/* ── Worker 线程：从队列取出 CQ_ITEM 并处理 ────────── */
static void *worker_thread(void *arg) {
    worker_t *w = (worker_t *)arg;
    char buf;

    while (1) {
        /* 阻塞等待 pipe 通知（模拟 libevent notify_receive_fd）*/
        if (read(w->notify_pipe[0], &buf, 1) <= 0) break;

        if (buf == 'q') break;  /* 退出信号 */

        /* 取出 CQ_ITEM — 此刻所有权归本 Worker */
        cq_item_t *item = cq_pop(&w->queue);
        if (!item) continue;

        printf("  [Worker %d] 处理连接 fd=%d ip=%s\n",
               w->id, item->fd, item->remote_ip);

        /* 处理完毕后释放（也可归还对象池）
         * Dispatcher 已无法访问此对象，无竞争 */
        free(item);
    }
    printf("  [Worker %d] 退出\n", w->id);
    return NULL;
}

/* ── Dispatcher：分配并转移所有权给 Worker ─────────── */
static void dispatch_conn(worker_t *workers, int fd,
                          const char *ip, int *last_wid) {
    int wid = (*last_wid + 1) % NUM_WORKERS;
    *last_wid = wid;

    /* 创建 CQ_ITEM（此时 dispatcher 拥有所有权）*/
    cq_item_t *item = malloc(sizeof(cq_item_t));
    item->fd = fd;
    snprintf(item->remote_ip, sizeof(item->remote_ip), "%s", ip);
    item->proto = 0;

    /* 将指针压入 Worker 队列 —— 所有权转移！
     * 从此 dispatcher 不再访问 item 指针 */
    cq_push(&workers[wid].queue, item);

    /* 通知 Worker（写 'c' 到 pipe）*/
    char c = 'c';
    write(workers[wid].notify_pipe[1], &c, 1);

    printf("[Dispatcher] fd=%d → Worker %d (所有权已转移)\n",
           fd, wid);
}

int main(void) {
    printf("=== 所有权转移模型 (模拟 memcached) ===\n\n");

    worker_t workers[NUM_WORKERS];
    pthread_t threads[NUM_WORKERS];

    for (int i = 0; i < NUM_WORKERS; i++) {
        workers[i].id = i;
        workers[i].done = 0;
        cq_init(&workers[i].queue);
        pipe(workers[i].notify_pipe);
        pthread_create(&threads[i], NULL, worker_thread, &workers[i]);
    }

    int last_wid = -1;
    const char *ips[] = {
        "192.168.1.1", "10.0.0.2", "172.16.0.3",
        "192.168.1.4", "10.0.0.5", "172.16.0.6",
        "192.168.1.7", "10.0.0.8", "172.16.0.9",
        "192.168.1.10","10.0.0.11","172.16.0.12"
    };

    for (int i = 0; i < TOTAL_CONNS; i++) {
        usleep(1000);
        dispatch_conn(workers, 1000 + i, ips[i % 12], &last_wid);
    }

    /* 发送退出信号 */
    usleep(50000);
    for (int i = 0; i < NUM_WORKERS; i++) {
        char q = 'q';
        write(workers[i].notify_pipe[1], &q, 1);
        pthread_join(threads[i], NULL);
        close(workers[i].notify_pipe[0]);
        close(workers[i].notify_pipe[1]);
    }

    printf("\n✓ 完成：指针所有权单向转移，Dispatcher 不会与 Worker 并发访问同一对象\n");
    return 0;
}

⚖️ 原理说明

所有权转移模型的核心不变式：任意时刻，每个对象只属于一个线程。Dispatcher 创建对象后，通过 cq_push() 传入指针，此后 Dispatcher 的局部变量指针失效（概念上），Worker 获得唯一访问权。

这与 C++ 中的 std::unique_ptr 语义一致。在 C 语言中通过代码规范和注释来约束此行为。memcached 源码中的注释明确写道传入队列后不再使用该指针。

    ✓ 优势
  • 无并发读写，无数据竞争
  • 与 Share-Nothing 结合使用效果极好
  • 仅队列操作需轻量级同步
  • 可配合对象池提升性能
    ✗ 劣势
  • 需要严格代码规范约束指针使用
  • 所有权语义在 C 中无法静态检查
  • 队列本身仍需同步（但仅此一处）
场景 ③

无锁环形队列（SPSC Lock-Free Ring Buffer）

核心思想：当一个生产者线程（Single Producer）需要持续向一个消费者线程（Single Consumer）传递数据时，可利用原子操作与内存屏障（Memory Barrier），在不使用任何互斥锁的情况下安全地共享环形缓冲区。

📦 理论基础：LMAX Disruptor 论文

📖 权威来源：LMAX Disruptor White Paper "In the case of having only one producer, and regardless of the complexity of the consumer graph, no locks or CAS operations are required. The whole concurrency coordination can be achieved with just memory barriers on the discussed sequences." lmax-exchange.github.io/disruptor/disruptor.html

🔧 完整源码示例（C11 原子操作实现）

spsc_ring.h + spsc_ring_demo.c — 无锁 SPSC 环形队列 C11 
/* === spsc_ring.h ===================================================
 * 无锁单生产者单消费者（SPSC）环形队列
 *
 * 原理：
 *   - 生产者只写 tail（使用 release 语义）
 *   - 消费者只写 head（使用 release 语义）
 *   - 双方读对方的计数器时使用 acquire 语义
 *   - 利用 acquire-release 配对形成 happens-before 关系
 *   - 无 CAS，无锁，无等待（wait-free）
 *
 * 参考：LMAX Disruptor Paper, lmax-exchange.github.io/disruptor/
 * ================================================================== */

#pragma once
#include <stdatomic.h>
#include <stdint.h>
#include <stdbool.h>
#include <stdlib.h>
#include <string.h>

/* 防止 false sharing：head 和 tail 放在不同缓存行 */
#define CACHE_LINE_SIZE 64

typedef struct {
    uint32_t  cap;       /* 容量（必须是2的幂）*/
    uint32_t  mask;      /* cap - 1，用于快速取模 */
    void    **buf;       /* 环形缓冲区（存指针）*/

    /* 消费者写 head，生产者读 head（检查是否已满）*/
    alignas(CACHE_LINE_SIZE) atomic_uint_fast32_t head;

    /* 生产者写 tail，消费者读 tail（检查是否有数据）*/
    alignas(CACHE_LINE_SIZE) atomic_uint_fast32_t tail;
} spsc_ring_t;

/* 初始化（cap 必须是2的幂，如 64、128、256）*/
static inline bool spsc_init(spsc_ring_t *r, uint32_t cap) {
    if (!r || cap == 0 || (cap & (cap - 1)) != 0) return false;
    r->cap  = cap;
    r->mask = cap - 1;
    r->buf  = calloc(cap, sizeof(void *));
    atomic_init(&r->head, 0);
    atomic_init(&r->tail, 0);
    return r->buf != NULL;
}

static inline void spsc_destroy(spsc_ring_t *r) {
    free(r->buf); r->buf = NULL;
}

/* ── 生产者：入队（只被一个线程调用）────────────────────
 *
 *  内存序解析：
 *  - head.load(acquire)：获取消费者的最新进度，确保看到正确的已消费位置
 *  - tail.store(release)：确保 buf[tail] = data 的写入在 tail 更新前对消费者可见
 *    （消费者用 acquire 读 tail 时，必然能看到 buf 的完整写入）
 * ─────────────────────────────────────────────────────── */
static inline bool spsc_push(spsc_ring_t *r, void *data) {
    uint32_t t = atomic_load_explicit(&r->tail, memory_order_relaxed);
    uint32_t h = atomic_load_explicit(&r->head, memory_order_acquire);

    if (t - h >= r->cap) return false; /* 队列已满 */

    r->buf[t & r->mask] = data;  /* 写入数据 */

    /* release 屏障：buf 写入 happens-before tail 更新 */
    atomic_store_explicit(&r->tail, t + 1, memory_order_release);
    return true;
}

/* ── 消费者：出队（只被一个线程调用）────────────────────
 *
 *  内存序解析：
 *  - tail.load(acquire)：获取生产者最新写入位置，确保看到完整的 buf 写入
 *  - head.store(release)：通知生产者此槽已被消费，可以复用
 * ─────────────────────────────────────────────────────── */
static inline void *spsc_pop(spsc_ring_t *r) {
    uint32_t h = atomic_load_explicit(&r->head, memory_order_relaxed);
    uint32_t t = atomic_load_explicit(&r->tail, memory_order_acquire);

    if (h == t) return NULL; /* 队列为空 */

    void *data = r->buf[h & r->mask];  /* 读取数据 */

    /* release 屏障：通知生产者此槽空闲 */
    atomic_store_explicit(&r->head, h + 1, memory_order_release);
    return data;
}

static inline bool     spsc_empty(spsc_ring_t *r) {
    return atomic_load_explicit(&r->head, memory_order_acquire)
        == atomic_load_explicit(&r->tail, memory_order_acquire);
}
static inline uint32_t spsc_size(spsc_ring_t *r) {
    return atomic_load_explicit(&r->tail, memory_order_acquire)
         - atomic_load_explicit(&r->head, memory_order_acquire);
}

/* ============================================================ */

#include <stdio.h>
#include <pthread.h>
#include <unistd.h>
#include <stdint.h>
#include <assert.h>

#define RING_CAP   64
#define NUM_ITEMS  10000

typedef struct {
    spsc_ring_t *ring;
    int          done;   /* 生产者结束标志 */
} demo_ctx_t;

static void *producer(void *arg) {
    demo_ctx_t *ctx = (demo_ctx_t *)arg;
    for (uintptr_t i = 1; i <= NUM_ITEMS; i++) {
        /* 忙等待直到队列有空位（实际项目中可加 yield 或 backoff）*/
        while (!spsc_push(ctx->ring, (void *)i)) {
            sched_yield();
        }
    }
    ctx->done = 1;  /* 标记结束 */
    printf("[Producer] 发送 %d 项完成\n", NUM_ITEMS);
    return NULL;
}

static void *consumer(void *arg) {
    demo_ctx_t *ctx = (demo_ctx_t *)arg;
    uintptr_t expected = 1, received = 0;

    while (1) {
        void *p = spsc_pop(ctx->ring);
        if (p) {
            uintptr_t val = (uintptr_t)p;
            assert(val == expected);  /* 验证顺序正确性 */
            expected++;
            received++;
        } else if (ctx->done && spsc_empty(ctx->ring)) {
            break;
        } else {
            sched_yield();
        }
    }
    printf("[Consumer] 接收 %lu 项，全部顺序正确 ✓\n",
           (unsigned long)received);
    return NULL;
}

int main(void) {
    printf("=== SPSC 无锁环形队列演示 ===\n");
    printf("容量: %d，传输: %d 项\n\n", RING_CAP, NUM_ITEMS);

    spsc_ring_t ring;
    spsc_init(&ring, RING_CAP);

    demo_ctx_t ctx = { .ring = &ring, .done = 0 };
    pthread_t tp, tc;
    pthread_create(&tp, NULL, producer, &ctx);
    pthread_create(&tc, NULL, consumer, &ctx);
    pthread_join(tp, NULL);
    pthread_join(tc, NULL);

    spsc_destroy(&ring);
    printf("\n✓ 无锁传输完成，未使用任何 mutex/spinlock\n");
    return 0;
}

⚖️ 内存屏障原理图解

    ✓ 优势
  • 完全无锁，极低延迟（纳秒级）
  • 无死锁、无优先级反转风险
  • cache 友好（head/tail 分开缓存行）
  • 高吞吐，适合实时系统
    ✗ 劣势
  • 严格限制：只能 1 个生产者 + 1 个消费者
  • 需要理解内存模型（容易出错）
  • 队列满时需要忙等（消耗 CPU）
  • 容量固定，运行时无法扩容
⚠️ 重要限制 SPSC 严格要求：push() 只被一个线程调用，pop() 只被另一个线程调用，不可混用。如需多生产者/消费者，需升级为 MPMC 方案（引入 CAS 操作）。
场景 ④

批量操作降低锁粒度

核心思想：当必须使用锁时，通过批量处理将多次锁操作合并为一次，大幅减少锁的争用次数和上下文切换开销。 临界区内做的事情越少越好，临界区外做的事情越多越好。

📦 典型案例：Redis I/O 线程批量处理

Redis 6.0 引入 I/O 多线程后，主线程并不是每处理一个客户端就加锁一次，而是将所有待处理客户端收集成批，统一分发给 I/O 线程，I/O 线程完成后再统一回收。整个协调过程通过原子计数器（而非互斥锁）来同步，极大减少了竞争。

📖 权威来源：sobyte.net — Redis 多线程网络模型 "主线程统一将 clients_pending_read 列表按轮询方式分配给各 I/O 线程，I/O 线程处理完毕后，主线程统一回收结果" — 批量分发而非逐个加锁。 sobyte.net/post/2022-03/redis-multi-threaded-network-model/

🔧 完整源码示例

batch_lock.c — 批量操作降低锁粒度（模拟 Redis I/O 线程调度） C 
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <pthread.h>
#include <unistd.h>
#include <stdint.h>
#include <time.h>

#define NUM_IO_THREADS 4
#define TOTAL_CLIENTS  200
#define BATCH_SIZE     32   /* 每批最多处理的客户端数 */

typedef struct {
    int  cid;
    char query[64];
    char response[64];
    int  processed;
} client_t;

/* ── 共享等待队列（模拟 clients_pending_read）── */
typedef struct {
    client_t   *items[1024];
    int         head, tail;
    pthread_mutex_t lock;
} pending_queue_t;

static pending_queue_t g_pending;
static volatile int  g_producer_done = 0;
static uint64_t g_lock_count = 0;  /* 统计加锁次数 */
static pthread_mutex_t g_stat_lock = PTHREAD_MUTEX_INITIALIZER;

/* ── 普通逐项入队（供生产者使用）── */
static void enqueue(client_t *c) {
    pthread_mutex_lock(&g_pending.lock);
    g_pending.items[g_pending.tail++ % 1024] = c;
    pthread_mutex_unlock(&g_pending.lock);
}

/* ── 批量出队（关键优化：一次锁取多个）── */
static int dequeue_batch(client_t **out, int max) {
    int cnt = 0;
    pthread_mutex_lock(&g_pending.lock);     /* 一次加锁 */

    /* 一次性取出多个，而非每次取一个都加锁 */
    while (cnt < max && g_pending.head < g_pending.tail) {
        out[cnt++] = g_pending.items[g_pending.head++ % 1024];
    }

    pthread_mutex_unlock(&g_pending.lock);   /* 一次解锁 */

    /* 统计加锁次数（证明批量减少了锁竞争）*/
    pthread_mutex_lock(&g_stat_lock);
    g_lock_count++;
    pthread_mutex_unlock(&g_stat_lock);

    return cnt;
}

/* ── I/O Worker：处理批次 ── */
typedef struct {
    int      id;
    uint64_t processed;
    uint64_t batches;
} io_worker_t;

static void *io_worker(void *arg) {
    io_worker_t *w = (io_worker_t *)arg;
    client_t *batch[BATCH_SIZE];

    while (1) {
        int n = dequeue_batch(batch, BATCH_SIZE);
        if (n == 0) {
            if (g_producer_done) break;
            usleep(100);
            continue;
        }

        /* 临界区外：处理整个批次，不持有锁 */
        for (int i = 0; i < n; i++) {
            client_t *c = batch[i];
            snprintf(c->response, sizeof(c->response),
                     "OK:cid=%d:worker=%d", c->cid, w->id);
            c->processed = 1;
        }

        w->processed += n;
        w->batches++;
    }
    printf("[IO Worker %d] 批次=%lu 处理=%lu\n",
           w->id, w->batches, w->processed);
    return NULL;
}

int main(void) {
    printf("=== 批量操作降低锁粒度 (模拟 Redis I/O 线程) ===\n");
    printf("I/O线程: %d，总客户端: %d，批大小: %d\n\n",
           NUM_IO_THREADS, TOTAL_CLIENTS, BATCH_SIZE);

    pthread_mutex_init(&g_pending.lock, NULL);
    g_pending.head = g_pending.tail = 0;

    /* 创建客户端对象 */
    client_t clients[TOTAL_CLIENTS];
    for (int i = 0; i < TOTAL_CLIENTS; i++) {
        clients[i].cid = i;
        snprintf(clients[i].query, sizeof(clients[i].query),
                 "GET key%d", i);
        clients[i].processed = 0;
    }

    /* 启动 I/O Workers */
    pthread_t   threads[NUM_IO_THREADS];
    io_worker_t workers[NUM_IO_THREADS];
    for (int i = 0; i < NUM_IO_THREADS; i++) {
        workers[i] = (io_worker_t){ .id = i, .processed = 0, .batches = 0 };
        pthread_create(&threads[i], NULL, io_worker, &workers[i]);
    }

    /* 生产者：将所有客户端入队 */
    for (int i = 0; i < TOTAL_CLIENTS; i++) {
        enqueue(&clients[i]);
        usleep(50);
    }
    g_producer_done = 1;

    for (int i = 0; i < NUM_IO_THREADS; i++)
        pthread_join(threads[i], NULL);

    /* 验证所有客户端均被处理 */
    int ok = 0;
    for (int i = 0; i < TOTAL_CLIENTS; i++)
        if (clients[i].processed) ok++;

    printf("\n结果：%d/%d 客户端处理成功\n", ok, TOTAL_CLIENTS);
    printf("批量出队加锁次数: %lu (平均每批处理 %.1f 项)\n",
           g_lock_count,
           g_lock_count > 0 ? (double)ok / g_lock_count : 0);
    printf("若逐项出队，将需加锁 %d 次，减少了 %.0f%%\n",
           TOTAL_CLIENTS,
           g_lock_count > 0
             ? (1.0 - (double)g_lock_count / TOTAL_CLIENTS) * 100
             : 0);
    return 0;
}

⚖️ Spinlock vs Mutex 选型原则

    ✓ 批量操作优势
  • 锁操作次数从 O(N) 降至 O(N/batch)
  • 减少上下文切换和系统调用开销
  • 提高缓存局部性（批次数据连续处理）
  • 降低其他线程等待时间
    ✗ 注意事项
  • 批次太大可能导致延迟增加
  • 临界区内禁止做 I/O 或长耗时操作
  • 需要权衡批量大小（通常 16~64）
场景 ⑤

锁层级防死锁

核心思想：死锁产生的根本原因是循环等待（Circular Wait）。通过为所有锁定义严格的全局获取顺序（Lock Hierarchy），消除循环等待的可能性，从根本上防止死锁。

📦 死锁产生条件（Coffman 四条件）

🔒
互斥（Mutual Exclusion）
资源同时只能被一个线程持有
🤝
持有并等待（Hold & Wait）
持有资源的线程还在等待更多资源
🚫
不可剥夺（No Preemption）
资源只能由持有者主动释放
🔄
循环等待（Circular Wait）
线程形成环形等待链 ← 打破此项！

lock_hierarchy.c — 锁层级机制 + 运行时检测违规 C 
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <pthread.h>
#include <unistd.h>
#include <assert.h>

/* ── 锁层级定义（全局规范，所有开发者必须遵守）──────────
 *
 *  Level 1 (最低层) — 哈希表 bucket 锁
 *  Level 2            — 全局计数器锁
 *  Level 3 (最高层) — 全局配置锁
 *
 *  规则：只能从低 Level 向高 Level 加锁，禁止逆序！
 * ─────────────────────────────────────────────────────── */
#define LOCK_LEVEL_BUCKET   1
#define LOCK_LEVEL_COUNTER  2
#define LOCK_LEVEL_CONFIG   3

/* ── 层级感知锁 ──────────────────────────────────────── */
typedef struct {
    pthread_mutex_t mu;
    int             level;
    const char     *name;
} hier_lock_t;

/* 线程本地：记录当前线程持有的最高层级 */
static pthread_key_t tls_max_level_key;
static pthread_once_t tls_init_once = PTHREAD_ONCE_INIT;

static void tls_init_key(void) {
    pthread_key_create(&tls_max_level_key, NULL);
}

static int get_thread_max_level(void) {
    void *val = pthread_getspecific(tls_max_level_key);
    return val ? (int)(intptr_t)val : 0;
}

static void set_thread_max_level(int lv) {
    pthread_setspecific(tls_max_level_key, (void *)(intptr_t)lv);
}

static void hier_lock_init(hier_lock_t *l, int level, const char *name) {
    pthread_once(&tls_init_once, tls_init_key);
    pthread_mutex_init(&l->mu, NULL);
    l->level = level;
    l->name  = name;
}

static void hier_lock(hier_lock_t *l) {
    int cur = get_thread_max_level();

    /* 检测层级违规（逆序加锁 → 潜在死锁）*/
    if (l->level <= cur) {
        fprintf(stderr,
            "[HIER VIOLATION] 尝试获取 %s(level=%d)，"
            "但当前最高持有 level=%d!\n"
            "  这会导致潜在死锁，终止程序。\n",
            l->name, l->level, cur);
        abort();
    }

    pthread_mutex_lock(&l->mu);
    set_thread_max_level(l->level);
    printf("  [TID=%lu] 获取 %s(L%d)\n",
           (unsigned long)pthread_self(), l->name, l->level);
}

static void hier_unlock(hier_lock_t *l, int prev_level) {
    set_thread_max_level(prev_level);
    pthread_mutex_unlock(&l->mu);
    printf("  [TID=%lu] 释放 %s(L%d)\n",
           (unsigned long)pthread_self(), l->name, l->level);
}

/* ── 业务锁对象 ─────────────────────────────────────── */
static hier_lock_t g_bucket_lock;   /* Level 1 */
static hier_lock_t g_counter_lock;  /* Level 2 */
static hier_lock_t g_config_lock;   /* Level 3 */
static int g_counter = 0;

/* ── 正确用法：按层级顺序加锁 ─────────────────────── */
static void *correct_thread(void *arg) {
    const char *name = (const char *)arg;
    printf("\n[%s] 开始正确顺序加锁 (L1→L2→L3):\n", name);

    int prev0 = get_thread_max_level();
    hier_lock(&g_bucket_lock);    /* Level 1 先 */
    int prev1 = LOCK_LEVEL_BUCKET;
    hier_lock(&g_counter_lock);   /* Level 2 次 */
    int prev2 = LOCK_LEVEL_COUNTER;
    hier_lock(&g_config_lock);    /* Level 3 后 */

    g_counter++;  /* 临界区操作 */
    usleep(1000);

    hier_unlock(&g_config_lock,  prev2); /* 逆序释放（可选）*/
    hier_unlock(&g_counter_lock, prev1);
    hier_unlock(&g_bucket_lock,  prev0);

    printf("[%s] 完成，counter=%d\n", name, g_counter);
    return NULL;
}

/* ── 错误用法：逆序加锁（触发层级检测）────────────── */
static void *wrong_thread(void *arg) {
    printf("\n[WrongThread] 尝试逆序加锁 (L2→L1)...\n");

    int prev0 = get_thread_max_level();
    hier_lock(&g_counter_lock);   /* Level 2 先 */

    /* 下一行将触发 abort，因为 Level 1 < current Level 2 */
    hier_lock(&g_bucket_lock);    /* Level 1 后 → 违规！*/

    hier_unlock(&g_bucket_lock,  LOCK_LEVEL_COUNTER);
    hier_unlock(&g_counter_lock, prev0);
    return NULL;
}

int main(void) {
    printf("=== 锁层级防死锁演示 ===\n");
    printf("Lock 层级：bucket(L1) < counter(L2) < config(L3)\n");

    hier_lock_init(&g_bucket_lock,  LOCK_LEVEL_BUCKET,  "bucket");
    hier_lock_init(&g_counter_lock, LOCK_LEVEL_COUNTER, "counter");
    hier_lock_init(&g_config_lock,  LOCK_LEVEL_CONFIG,  "config");

    /* 两个线程正确顺序加锁，不会死锁 */
    pthread_t t1, t2;
    pthread_create(&t1, NULL, correct_thread, (void *)"Thread-A");
    pthread_create(&t2, NULL, correct_thread, (void *)"Thread-B");
    pthread_join(t1, NULL);
    pthread_join(t2, NULL);
    printf("\n✓ 两线程均完成，无死锁\n");

    printf("\n--- 现在演示错误用法（程序将中止）---\n");
    printf("（若不希望 abort，注释掉 wrong_thread 调用）\n");
    /* 取消注释以演示层级违规检测：
    pthread_t tw;
    pthread_create(&tw, NULL, wrong_thread, NULL);
    pthread_join(tw, NULL);
    */
    return 0;
}

⚖️ 防死锁工程规范

规范 说明 来自项目
固定加锁顺序 所有代码路径中，相同锁组合必须以相同顺序获取 Linux Kernel
锁层级编号 为每把锁分配全局层级号，编号小的必须先获取 nginx
运行时检测 开发阶段用 TLS 记录当前线程最高持有层级，违规即 abort 调试实践
trylock + 回退 使用 pthread_mutex_trylock 检测潜在死锁，失败时释放所有锁重试 通用
最小临界区 临界区只包含必须原子执行的最少代码，I/O 操作移出临界区 memcached
ThreadSanitizer 编译时加 -fsanitize=thread，自动检测数据竞争和死锁风险 工具
    ✓ 锁层级优势
  • 从数学上消除循环等待条件
  • 无需运行时 deadlock detector 即可保证安全
  • 代码逻辑清晰，可审计
  • 与 sanitizer 工具结合效果极好
    ✗ 注意事项
  • 需要严格的代码审查文化
  • 层级设计需提前规划，重构成本高
  • 跨模块锁需统一文档化层级规范
🚫 绝对禁止 在持有锁 A 的情况下，调用任何可能去获取层级 ≤ A 的锁的外部函数（包括回调、第三方库）。这是死锁的最常见来源之一。
