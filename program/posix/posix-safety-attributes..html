📖 概述

Linux attributes(7)
POSIX 安全属性全解析

基于 GNU C 库官方手册"POSIX Safety Concepts"章节，系统阐释每种安全标注的含义与使用场景。

🔗 原文：man7.org/linux/man-pages/man7/attributes.7.html

什么是 attributes(7)？

Linux 手册页中大量函数包含 ATTRIBUTES 一节，该节通过一组标准化的安全标注（Safety Markings）描述函数在不同执行上下文中调用的安全性。 这些标注最初来源于 GNU C 库（glibc） 的"POSIX Safety Concepts"规范，用于指导开发者在多线程、异步信号处理、区域设置切换等复杂场景下正确使用库函数。

理解这些属性，是编写高质量、无竞态系统级程序的基础前提。

📌
版本说明：本文内容对应 man-pages-6.18（2026-02-08），由 man7.org 官方发布。 项目主页：kernel.org/doc/man-pages/

📋 章节概览

  • 基础安全属性：MT-Safe / MT-Unsafe — 函数是否可以在多线程程序中安全调用
  • 条件安全特性：init / race / const / sig / term — 在满足特定约束条件下可避免的安全问题
  • 其他安全备注：locale / env / hostid / sigintr / cwd — 不直接导致 MT-Unsafe，但需特别关注的行为
  • 标注语法：标识符（:identifier）与条件修饰（/condition）

🔒 基础安全属性

✅ MT-Safe — 线程安全

MT-Safe
多线程安全（Multi-Thread Safe）
可在存在其他线程的情况下安全调用的函数

MT-Safe（或称 Thread-Safe）函数可以在多线程程序中与其他线程并发执行时安全调用。其中 MT 代表 Multi Thread（多线程）。

📌 重要限定：MT-Safe 不等于原子性

官方原文明确指出，MT-Safe 并不意味着函数是原子的，也不意味着它使用了 POSIX 向用户暴露的任何内存同步机制。 甚至，顺序调用两个 MT-Safe 函数，也不能保证等同于原子地执行这两者的组合。

⚠️
组合不保证安全：一个线程连续调用两个 MT-Safe 函数，其他线程可能在两次调用之间进行破坏性的并发干预，导致不可预期的结果。

📌 内联优化警告

跨库接口进行内联的全程序优化（Whole-program optimization）可能暴露不安全的重排序。因此，不建议对 GNU C 库接口进行跨边界内联。 文档所记录的 MT-Safety 状态在全程序优化下不提供保证。但是，用户可见头文件中定义的函数被设计为可以安全内联。

🔧 适用场景

图 2：MT-Safe 函数的并发调用模型

📝 章节小结

  • MT-Safe 表示函数可在多线程环境下并发调用，不会产生内部竞争
  • 不保证原子性：单函数内部安全，但多函数组合仍需用户自己加锁
  • 避免对 glibc 接口进行跨边界全程序内联优化
  • 典型安全函数：malloc()、pthread_create()、memcpy() 等

🔒 基础安全属性

❌ MT-Unsafe — 线程不安全

MT-Unsafe
多线程不安全（Multi-Thread Unsafe）
不能在多线程程序中安全调用的函数

MT-Unsafe 函数在多线程程序中调用是不安全的。在多个线程并发运行的上下文中调用这类函数，可能导致数据竞争、状态损坏或未定义行为。

🚨 为何存在 MT-Unsafe 函数？

许多 MT-Unsafe 函数的历史早于 POSIX 线程标准，它们使用了静态内部缓冲区、全局状态或不可重入的算法。常见原因包括：

  • ✗ 使用静态（static）局部缓冲区返回结果，多线程共享同一内存区域
  • ✗ 依赖未受保护的全局变量（如 errno 的旧实现、strtok() 的内部指针）
  • ✗ 修改进程级的共享资源（如信号掩码、终端属性、区域设置）
🚫
规则：在多线程程序中，不得并发调用 MT-Unsafe 函数。若必须使用，需用互斥锁（mutex）将整个调用序列串行化，并确保其他所有可能访问相同内部状态的线程也受同一锁保护。

🔧 典型 MT-Unsafe 函数示例

/* 典型 MT-Unsafe 函数 — 使用静态缓冲区 */
char *ctime(const time_t *timep);   /* 返回指向静态字符串的指针 */
char *strtok(char *str, const char *delim);  /* 内部维护静态指针 */
struct hostent *gethostbyname(const char *name); /* 静态结构体 */

/* 安全替代方案（reentrant 版本） */
char *ctime_r(const time_t *timep, char *buf);
char *strtok_r(char *str, const char *delim, char **saveptr);

图 3：MT-Unsafe 函数中的静态缓冲区竞争示例

📝 章节小结

  • MT-Unsafe 函数绝对不能在多线程程序中无保护地并发调用
  • 根本原因：使用了线程间共享的内部状态（静态缓冲区、全局变量等）
  • 解决策略：①使用带 _r 后缀的可重入替代版本；②用 mutex 串行化调用；③仅在单线程阶段调用
  • 标准库提供了许多 _r 变体：strtok_r、ctime_r、localtime_r 等

⚠️ 条件安全特性

🔧 init — 初始化竞态

init
初始化时的 MT-Unsafe 特性
首次调用时执行非线程安全的初始化操作

标注了 init 作为 MT-Unsafe 特性的函数，在首次被调用时会执行 MT-Unsafe 的初始化操作。

解决方案：在单线程模式下至少调用该函数一次，即可消除这一特定的不安全因素。此后如果没有其他原因导致该函数不安全，则可以在启动其他线程之后安全调用。

📐 工作原理

某些函数在内部使用延迟初始化（lazy initialization）模式：第一次被调用时才初始化内部状态。 如果多个线程同时首次调用该函数，就会产生初始化竞态。

图 4：init 特性的工作原理与解决策略

🛠️ 使用场景与应对策略

💡
最佳实践：在 main() 函数创建任何线程之前，显式调用一次带 init 标注的函数，完成预初始化。之后在多线程环境下调用就是安全的（前提是无其他不安全原因）。 
int main(void) {
    /* 在创建线程之前，先在单线程环境调用一次，完成初始化 */
    (void)getpwuid(0);   /* 假设标注了 init */

    /* 现在可以安全创建线程并并发使用 */
    pthread_t t1, t2;
    pthread_create(&t1, NULL, worker, NULL);
    pthread_create(&t2, NULL, worker, NULL);
    /* ... */
}

📝 章节小结

  • init 表示函数在首次调用时执行非线程安全的内部初始化
  • 根本原因：延迟初始化（lazy init）未使用同步原语保护
  • 解决方法：在单线程阶段提前调用一次，之后可安全并发使用
  • 这是一种"可修复"的不安全——满足约束后即消除风险

⚠️ 条件安全特性

⚡ race — 数据竞争

race
数据竞争（Data Race）
函数以可能导致数据竞争或破坏性干扰的方式操作对象

标注 race 的函数在操作某些对象时，可能与并发执行产生数据竞争（data race）或其他形式的破坏性干扰。 涉及的对象有三种来源：

  • ① 由用户传入函数的参数对象
  • ② 函数用于向用户返回值的对象
  • ③ 用户完全不可见的内部对象

📐 数据竞争详解

数据竞争（Data Race）是指：两个或多个线程并发访问同一内存位置，其中至少一次是写操作，且没有使用同步机制保护。 这在 C11/C++11 标准下是未定义行为（Undefined Behavior）。

图 5：race 特性中的数据竞争场景

🛠️ 应对策略

💡
根据标识符分组保护：race 标注通常附带标识符（如 race:buf、race:tcattr(fd)）， 所有涉及同一标识符的函数调用需用同一个同步原语（mutex 或 rwlock）保护。 
pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;

void thread_worker(void) {
    pthread_mutex_lock(&lock);
    /* 保护所有 race:some_obj 的函数调用 */
    some_func_with_race();
    pthread_mutex_unlock(&lock);
}

📝 章节小结

  • race 标注指出函数操作某对象时会产生数据竞争风险，读者和写者均标记为 MT-Unsafe
  • 涉及对象可能是用户传入的参数、返回值缓冲区，或不可见的内部对象
  • 应对方法：用 mutex/rwlock 保护所有具有相同标识符的函数调用
  • 与 const 的区别：race 同时影响读者和写者；const 仅影响写者

⚠️ 条件安全特性

🔐 const — 非原子常量修改

const
非原子地修改内部"常量"对象
修改被视为常量的内部对象，影响大量未同步的读者

标注 const 的函数以非原子方式修改内部对象，而这些对象在逻辑上应被视为常量——因为 GNU C 库中有大量代码在没有同步的情况下访问它们。

与 race 不同，const 标注只施加于写者（修改者），而非读者。写者被标记为 MT-Unsafe，但由于写者被禁止在多线程运行时调用，对象实际上处于有效常量状态，因此读者可以被视为 MT-Safe（假设没有其他不安全原因）。

📐 const 与 race 的核心区别

图 6：race 与 const 标注的关键区别

🛠️ 绕过 const 安全问题的方案

官方文档提供了在确实需要调用写者（const 标注函数）的场景下，如何安全操作的方案： 使用与标识符关联的非递归读写锁（non-recursive read-write lock）。

📌
规则：
• 所有标注 const:X 的函数（写者）调用前必须持有 写锁
• 所有标注仅 X（不带 const 前缀）的函数（读者）调用前需持有 读锁 
pthread_rwlock_t locale_lock = PTHREAD_RWLOCK_INITIALIZER;

/* 写者：修改 locale（const:locale 标注） */
void change_locale(const char *loc) {
    pthread_rwlock_wrlock(&locale_lock);
    setlocale(LC_ALL, loc);   /* const:locale */
    pthread_rwlock_unlock(&locale_lock);
}

/* 读者：使用 locale（locale 标注，通常已是 MT-Safe） */
void use_locale(void) {
    pthread_rwlock_rdlock(&locale_lock);
    /* 调用依赖 locale 的函数 */
    printf("%s\n", nl_langinfo(CODESET));
    pthread_rwlock_unlock(&locale_lock);
}

📝 章节小结

  • const 只将写者标为 MT-Unsafe，读者因对象实际恒定而保持 MT-Safe
  • 该标注的核心假设：在多线程运行期间，没有人修改这些"常量"对象
  • 若确需在运行时修改，使用 pthread_rwlock：写者持写锁，读者持读锁
  • 典型例子：setlocale()（const:locale）、setenv()（const:env）

⚠️ 条件安全特性

📡 sig — 临时信号处理器

sig
临时安装信号处理器
函数可能为内部目的临时安装信号处理器，干扰其他信号用途

标注 sig 的函数可能在执行期间临时为内部目的安装某个信号的处理器（signal handler），这会干扰程序中对同一信号的其他用途。受影响的具体信号会在冒号后列出（如 sig:SIGPIPE）。

📐 信号处理器竞争场景

图 7：sig 特性中的信号处理器竞争窗口

🛠️ 应对策略

官方文档推荐的解决方案：

💡
① 持有非递归 mutex：保证所有使用同一临时信号的函数调用被串行化。
② 调用前阻塞该信号，调用后恢复：防止信号在函数执行窗口期间到达，然后重置信号处理器。 
pthread_mutex_t sig_mutex = PTHREAD_MUTEX_INITIALIZER;

void safe_call(void) {
    sigset_t old_mask, block_mask;
    sigemptyset(&block_mask);
    sigaddset(&block_mask, SIGPIPE);   /* 阻塞受影响的信号 */

    pthread_mutex_lock(&sig_mutex);
    pthread_sigmask(SIG_BLOCK, &block_mask, &old_mask);

    func_with_sig_sigpipe();   /* 标注了 sig:SIGPIPE */

    pthread_sigmask(SIG_SETMASK, &old_mask, NULL);   /* 恢复 */
    pthread_mutex_unlock(&sig_mutex);
}

📝 章节小结

  • sig 标注函数在执行时会临时修改特定信号的处理器，标识符指明影响哪个信号
  • 危险在于：在函数执行的"窗口期"内，该信号若到达则由错误的 handler 处理
  • 解决方案：调用期间阻塞目标信号 + 持 mutex 串行化所有相关调用
  • 典型函数：部分旧式 I/O 函数会临时安装 SIGPIPE 处理器

⚠️ 条件安全特性

💻 term — 终端属性竞态

term
终端设置竞态窗口（Terminal Settings Race）
以读取-修改-写回方式修改终端属性，存在竞态窗口

标注 term 的函数按照推荐方式修改终端设置：先调用 tcgetattr(3)，修改若干标志位，然后调用 tcsetattr(3)。 这三步操作之间存在一个窗口期，在此期间其他线程所做的更改会被覆盖（丢失）。因此，标注 term 的函数被认定为 MT-Unsafe。

📐 TOCTOU 竞态窗口

图 8：term 特性的 TOCTOU（检查时间-使用时间）竞态窗口

🛠️ 官方推荐应对方案

⚠️
建议：应用程序在使用终端时，应避免与终端进行并发或可重入的交互：
• 不在信号处理器中使用终端
• 阻塞可能使用终端的信号
• 调用相关函数及与终端交互时持锁
• 对于标注 race:tcattr(fd) 的函数，使用同一把锁（每个终端文件描述符一把锁，或全局一把）

📝 章节小结

  • term 由读取-修改-写回终端属性模式（tcgetattr → modify → tcsetattr）引发竞态窗口
  • 这是经典的 TOCTOU（Time-Of-Check Time-Of-Use）问题在终端场景的体现
  • 与 race:tcattr(fd) 标注共用同一把锁来保护所有终端操作
  • 实际场景：终端程序（shell、ncurses TUI、密码输入等）需特别注意

💡 其他安全备注

🌍 locale — 区域设置读取

locale
读取区域设置对象（无同步）
并发修改 locale 期间可能读到混合状态，但通常不标为 MT-Unsafe

标注 locale 的函数在没有任何形式同步的情况下读取 locale 对象。 如果在执行过程中并发地发生 locale 更改，函数可能以不对应任何单一 locale 的方式运行——而是反映某种不可预测的混合状态。

然而，修改 locale 对象的函数被标注为 const:locale 并被视为 MT-Unsafe。因此，locale 修改不应在多线程运行或异步信号启用时调用，这使得 locale 在这些上下文中实际上是"有效常量"，读者因此可以被视为 MT-Safe。

📌
关键推理：写者（setlocale）是 MT-Unsafe 的，不在多线程运行时调用 → locale 对读者来说是常量 → 读者是 MT-Safe 的。这是 const 机制的一个典型实例。

🔧 使用场景

  • → 国际化（i18n）程序：在程序启动时设置 locale，之后多线程只读
  • → 字符串排序、日期格式化、货币显示等依赖 locale 的操作
  • → 若需在运行时切换 locale，需在切换前停止所有线程或使用 rwlock 保护

📝 章节小结

  • locale 标注说明函数无同步地读 locale，理论上存在并发修改风险
  • 实践中因写者（setlocale）被标为 const:locale（MT-Unsafe），多线程阶段 locale 不被修改，故读者安全
  • 若要动态切换 locale，需用 pthread_rwlock 协调读写双方
  • 参考相关函数：nl_langinfo()、strcoll()、strftime() 等

💡 其他安全备注

🌿 env — 环境变量访问

env
无保护地访问环境变量
使用 getenv(3) 等接口访问环境，无并发修改保护

标注 env 的函数通过 getenv(3) 或类似接口访问环境变量，没有任何措施确保在并发修改存在时的安全性。

但这些函数不被标为 MT-Unsafe，原因与 locale 相同：修改环境的函数（如 setenv()、putenv()、unsetenv()）均被标注为 const:env 并视为 MT-Unsafe，因此在多线程运行或异步信号启用时不应调用，环境变量在这些上下文中实际上是"有效常量"。

🔧 使用场景

/* 读取环境变量 — 标注 env，通常是安全的 */
const char *path = getenv("PATH");   /* env 标注 */
const char *home = getenv("HOME");

/* 修改环境变量 — 标注 const:env，MT-Unsafe */
/* 只能在单线程阶段或停止所有线程后调用 */
setenv("MY_VAR", "value", 1);    /* const:env */
putenv("KEY=val");                /* const:env */
⚠️
注意：部分库函数内部会调用 getenv()（如 getaddrinfo() 读取 RES_OPTIONS），这些函数都有 env 标注。如果你在多线程程序中动态修改环境变量，可能会导致这些函数行为异常。

📝 章节小结

  • env 标注函数通过 getenv 读取环境变量，无同步保护
  • 安全性依赖于：环境修改函数（const:env）在多线程期间不被调用
  • 多线程程序应在启动线程之前完成所有环境变量设置
  • 涉及的写者：setenv()、putenv()、clearenv()、unsetenv()

💡 其他安全备注

🖥️ hostid — 主机 ID

hostid
读取系统主机 ID 数据结构
读取不能原子修改的系统级"主机 ID"数据

标注 hostid 的函数读取保存机器"主机 ID"（host ID）的系统级数据结构。这些数据结构通常不能以原子方式修改。

由于预期"主机 ID"不会经常改变，读取它的函数 gethostid(3) 被视为安全， 而修改它的函数 sethostid(3) 被标注为 const:hostid，表明如果要调用它，需要特别注意。

在这个具体案例中，特别注意意味着系统范围内（不仅仅是进程内）的协调——即需要确保整个系统上没有其他进程在同时读取主机 ID。

📌
特殊性：hostid 是少数需要跨进程、跨系统协调的情形，而不仅仅是进程内的线程协调。 这使得它与其他仅需 mutex 的标注有本质区别。

🔧 使用场景

函数标注操作安全性
gethostid(3) hostid 读取主机 ID MT-Safe（主机 ID 稳定）
sethostid(3) const:hostid 修改主机 ID MT-Unsafe（需系统级协调）

📝 章节小结

  • hostid 标注的函数读取系统主机 ID，因 ID 稳定而视为安全
  • 写者（sethostid）标注 const:hostid，需要系统范围协调（超出进程级 mutex）
  • 这是 const 机制的进阶案例：保护范围从进程内扩展到系统级
  • 实际使用：hostid 通常用于软件授权、集群标识等不频繁修改的场景

💡 其他安全备注

📶 sigintr — 信号中断内部结构

sigintr
访问 GNU C 库内部信号中断结构
无保护地访问 __sigintr 内部数据结构

标注 sigintr 的函数在没有任何保护措施的情况下访问 GNU C 库的 _sigintr 内部数据结构，该结构记录哪些信号会中断系统调用的行为。

这些函数同样不被标为 MT-Unsafe，因为修改该数据结构的函数均标注了 const:sigintr 并视为 MT-Unsafe。在多线程运行或异步信号启用期间，该数据结构被视为有效常量，读者因此是安全的。

🔧 相关函数

函数标注说明
siginterrupt(3) const:sigintr 修改信号中断系统调用的行为（写者，MT-Unsafe）
使用 sigintr 信息的函数 sigintr 读取内部结构（读者，实际为 MT-Safe）
📌
设计模式：sigintr 与 locale/env 遵循同样的安全推理模式：写者被禁止 → 数据结构稳定 → 读者安全。这是 GNU C 库中处理"伪常量"（pseudo-constant）的统一模式。

📝 章节小结

  • sigintr 标注表示函数访问 glibc 内部的 _sigintr 信号中断标志结构
  • 安全性依赖写者（siginterrupt，标注 const:sigintr）不在多线程时调用
  • 与 locale、env、hostid 共享相同的"有效常量"安全推理链

💡 其他安全备注

📁 cwd — 当前工作目录

cwd
临时改变当前工作目录
执行期间可能临时修改进程的当前工作目录

标注 cwd 的函数在执行期间可能临时更改当前工作目录（current working directory）。这会导致其他线程或异步信号/取消处理器中的相对路径解析出现意外结果。

这还不足以将这些函数标为 MT-Unsafe。但当该行为是可选的时候（例如 nftw(3) 搭配 FTW_CHDIR 选项）， 避免使用该选项，或改用绝对路径、文件描述符相对系统调用（如 openat(2)）是比较好的替代方案。

📐 cwd 竞态场景

图 9：cwd 标注引起的相对路径解析错误

🛠️ 解决方案

💡
替代方案：
① 避免使用触发 chdir 的选项（如 FTW_CHDIR）
② 使用绝对路径而非相对路径
③ 使用文件描述符相对的系统调用（openat(2)、fstatat(2)、mkdirat(2) 等），完全规避工作目录的影响 
/* 不推荐：使用 FTW_CHDIR 可能干扰其他线程的路径解析 */
nftw("/var/log", callback, 20, FTW_CHDIR);

/* 推荐：使用 openat 系列，无需依赖 cwd */
int dirfd = open("/var/log", O_RDONLY | O_DIRECTORY);
int fd = openat(dirfd, "syslog", O_RDONLY);
close(dirfd);

📝 章节小结

  • cwd 标注函数在执行过程中临时修改进程当前工作目录，影响其他线程的相对路径解析
  • 未被标为 MT-Unsafe，但在多线程程序中应尽量避免或用绝对路径替代
  • 最佳方案：使用 *at() 系列系统调用（openat、fstatat 等），彻底解耦路径解析与 cwd
  • 典型案例：nftw(3) + FTW_CHDIR、ftw(3)

🏷️ 标注语法

🔖 :identifier — 标识符注解

标注可以附加标识符（identifier），用于将若干具有相关联系的函数或参数分组，例如：以不安全方式访问相同数据结构（如 race 和 const），或提供更具体的信息（如在 sig 标注中命名特定信号）。

:identifier
标识符分组语法
将需要使用同一同步原语保护的函数和参数分组

标识符可以命名：

  • ▸一组函数（最常见）
  • ▸全局对象
  • ▸函数参数
  • ▸与参数关联的属性或逻辑组件

📐 标识符格式示例

标注示例含义
race:buf 函数以可能引发竞态的方式操作名为 buf 的对象/缓冲区
race:buf(arg) 与参数 arg 关联的缓冲区存在竞态风险
race:tcattr(fd) 文件描述符 fd 对应终端的属性存在竞态风险
const:locale 函数非原子地修改 locale 对象（写者）
locale（不带 const） 函数读取 locale 对象（读者，通常 MT-Safe）
sig:SIGPIPE 函数可能临时安装 SIGPIPE 的信号处理器
const:env 函数非原子地修改环境变量（写者）
📌
核心用途：标识符最重要的作用是标明哪些函数和参数需要使用同一把同步原语保护，才能在给定上下文中安全操作。 凡是带有同一标识符的标注，均应被同一个 mutex 或 rwlock 统一覆盖。

📝 章节小结

  • 标识符是标注的细化工具，将相关函数/对象按"风险组"分组
  • 同一标识符的所有函数应受同一同步原语保护
  • 可命名函数集合、全局对象、参数关联的缓冲区或逻辑组件
  • 未来可能扩展到 lock 和 corrupt 标注

🏷️ 标注语法

❓ /condition — 条件注解

某些安全标注可以是条件性的，即仅当某个布尔表达式（涉及函数参数、全局变量，甚至底层内核）求值为真时，该标注才适用。

/condition
条件化安全标注
仅在特定条件成立时，该标注的安全问题才适用

/condition 修饰符限制前面标注的适用范围。例如：

  • ▸race:ps/!ps：仅当参数 ps 为 NULL 时才适用
  • ▸/oneperline：仅当全局变量 oneperline 非零时才适用

当所有使函数不安全的标注都附加了条件修饰符，且没有任何命名的条件成立时，该函数可以被视为安全的。

📐 条件注解工作原理

图 10：条件注解的工作逻辑（以 race:ps/!ps 为例）

📌 实际应用

/* 以某个具有条件注解的函数为例
 * 标注：MT-Unsafe race:output/!ps
 * 含义：仅当 ps 参数为 NULL 时，output 对象存在竞态
 */
void example_func(struct obj *ps, FILE *output);

/* 当 ps != NULL 时，不涉及竞态，可视为 MT-Safe */
example_func(my_obj, stderr);   /* 安全 */

/* 当 ps == NULL 时，存在 race:output 问题，需加锁 */
pthread_mutex_lock(&output_lock);
example_func(NULL, stderr);     /* 需要保护 */
pthread_mutex_unlock(&output_lock);

📝 章节小结

  • /condition 使安全标注的适用范围受参数值或全局变量状态约束
  • 当所有不安全标注的条件均不成立时，函数可视为完全安全
  • 这让标注系统更精确，避免对"在某些参数组合下实际安全"的函数作过度限制
  • 开发者需要理解这些条件，针对实际调用场景判断是否需要同步保护

📊 综合总结

🔭 第一性原理综述

从第一性原理出发，attributes(7) 所描述的全部属性，本质上都是对同一个根本问题的不同维度的回答：

核心问题：共享可变状态（Shared Mutable State）

并发安全问题的根源，始终是多个执行单元（线程/信号处理器/进程）并发访问同一块可变状态，且缺乏适当的同步机制。 所有的不安全属性，都是这一根本矛盾在不同场景下的具体表现。

图 11：所有安全属性的第一性原理归因

🧩 三大设计模式

GNU C 库在 attributes(7) 中实际上展示了三种处理"共享可变状态"问题的设计模式：

模式一
有效常量化（Effective Constancy）

将写者标记为 MT-Unsafe（禁止在多线程时调用），使共享状态在多线程期间保持"有效常量"， 读者因此无需同步。适用于：locale、env、hostid、sigintr。

模式二
显式同步（Explicit Synchronization）

识别出共享对象，用 mutex 或 rwlock 保护所有访问者。适用于：race（mutex）、const（rwlock）、sig（mutex + signal block）、term（mutex）。

模式三
状态局部化（State Localization）

彻底消除共享状态，每个执行单元维护独立状态。适用于：使用 _r 可重入变体（thread-local 缓冲区）、使用 openat 消除对 cwd 的依赖。

🌐 第一性原理结论

🔭
并发安全的第一性原理：对于任意内存位置，若存在至少一个写操作，且该位置被多个执行单元可并发访问，则必须有同步机制。所有的 MT-Safe 标注，都是在声明"这一点已在库内部被满足"；所有的 MT-Unsafe 标注，都是在说明"这一点尚未满足，需要调用者来保证"。

attributes(7) 的本质，是 GNU C 库向用户精确声明责任边界的接口契约（interface contract）：库会保护什么，调用者需保护什么。

📝 全文总结

  • MT-Safe / MT-Unsafe：最顶层的安全判定，回答"能否并发调用"
  • init / race / const / sig / term：条件性不安全，明确了约束条件，满足后可消除风险
  • locale / env / hostid / sigintr / cwd：不影响 MT 判定，但指出调用者需注意的副作用
  • :identifier：细化分组，告诉调用者哪些操作需共享同一把锁
  • /condition：精确化适用范围，避免对特定参数场景的过度限制
  • 根本逻辑：所有属性都是对"共享可变状态"问题责任边界的精确声明
