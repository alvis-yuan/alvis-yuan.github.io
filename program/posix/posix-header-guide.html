1 导言：POSIX.1-2024 规范

IEEE Std 1003.1-2024（又称 POSIX.1-2024 / The Open Group Base Specifications Issue 8）是迄今最新的 POSIX 标准，定义了跨平台可移植操作系统接口，包含 86 个头文件，覆盖系统调用、线程、网络、国际化等各大领域。

📖 官方原文：XBD 目录（Base Definitions）
📌 规范来源 本文所有内容均严格基于 The Open Group Base Specifications Issue 8, IEEE Std 1003.1-2024 官方在线文档（pubs.opengroup.org/onlinepubs/9799919799/），原文版权归 IEEE 和 The Open Group 所有（Copyright © 2001-2024）。

📐 规范四大卷册

XBD — 基础定义卷

通用概念、术语、C 语言头文件定义（本文重点）

XSH — 系统接口卷

所有系统服务函数、C 语言系统服务、错误处理

XCU — Shell 与工具卷

Shell 命令解释器接口及通用工具程序定义

XRAT — 理由卷

历史信息、设计决策的扩展说明（资料性）

📝 第一章总结

  • POSIX.1-2024 是 IEEE Std 1003.1 的第八版（Issue 8），于 2024 年发布
  • 规范共定义 86 个头文件，分布于 XBD Chapter 14
  • 四大卷册各司其职：XBD 定义基础、XSH 定义接口、XCU 定义工具、XRAT 提供理由
  • 所有头文件均位于开发系统（application development system），不要求出现在目标运行系统上
  • 特性测试宏 _POSIX_C_SOURCE=202405L 用于启用完整 POSIX.1-2024 符号集

📝 第二章总结

  • POSIX 头文件不是平铺的——它们按功能内聚原则归属九大领域
  • ISO C 类占比最多（29/86），体现 POSIX 与 C 标准的紧密对齐关系
  • sys/ 前缀头文件均为 POSIX 特有扩展，用于系统级资源访问
  • 网络类头文件大多源自 BSD 4.4，POSIX.1g 将其纳入标准化
  • XSI (X/Open System Interfaces) 选项组的头文件（如 <sys/ipc.h>）是可选扩展集

3 线程安全体系

POSIX.1-2024 在 XBD 4.22 定义线程安全的基本概念，XSH 2.9 提供详细的线程安全规定。规范将函数按线程安全属性分为三个维度：MT-Safe、Async-Signal-Safe 和 Async-Cancel-Safe。

📖 XSH Chapter 2: General Information
📌 规范依据 POSIX.1-2024 XBD 4.22 Thread-Safety："Refer to XSH 2.9 Threads"；XSH 2.1.1 条款 6："A function shall not directly or indirectly access objects accessible by threads other than the current thread unless the objects are accessed directly or indirectly via the function's arguments."

🔐 三大线程安全属性

MT-Safe（多线程安全）
MT-Safe

函数可以在多个线程中同时调用而不产生数据竞争（data race）。POSIX 规定函数的默认义务是不访问其他线程可访问的对象（除非通过函数参数传递）。符合此义务的函数即为 MT-Safe。

适用场合 多线程应用中无需额外加锁即可直接调用。大多数数学函数、字符分类函数、内存操作函数属于此类。
MT-Unsafe（多线程不安全）
MT-Unsafe

函数内部使用了非线程安全的全局状态（如 errno、静态缓冲区、全局链表等），在多线程环境中调用会产生数据竞争。POSIX 为许多此类函数提供了线程安全的 _r 后缀替代版本（reentrant variants）。

适用场合 仅在单线程环境或持有适当互斥锁的前提下使用。示例：strtok()→使用 strtok_r()；getpwnam()→使用 getpwnam_r()。
Async-Signal-Safe（异步信号安全）
AS-Safe

根据 POSIX.1-2024 XSH 2.4，可以在信号处理函数中安全调用的函数。POSIX 提供了一张明确的 async-signal-safe 函数列表（约 180+ 个函数），不在列表中的函数在信号处理函数中调用会产生未定义行为。

适用场合 信号处理函数（signal handler）内部使用。典型安全函数：write()、read()、_exit()、kill()、sigprocmask()。

注：errno 的读写操作本身是 async-signal-safe 的，但信号处理函数应在入口保存、出口恢复 errno 值。

Async-Cancel-Safe（异步取消安全）
AC-Safe

根据 POSIX.1-2024 XBD 3.25，可以在线程取消处理函数（cancellation cleanup handler）中安全调用的函数。POSIX 规定 async-cancel-safe 函数集是 async-signal-safe 函数的严格子集。

适用场合 使用 pthread_cancel() 的异步取消场景。仅 pthread_cancel()、pthread_setcancelstate()、pthread_setcanceltype() 三个函数被明确列为 AC-Safe。

📋 典型函数线程安全速查

函数所在头文件MT-SafeAS-Safe说明
printf()<stdio.h>否*否使用内部 FILE 锁，但重入不安全
malloc()<stdlib.h>是否实现通常有内部锁；信号中不安全

strtok()<string.h>否否使用静态状态；用 strtok_r() 替代

📝 第三章总结

  • POSIX 线程安全是一个三层嵌套结构：AC-Safe ⊂ AS-Safe ⊂ MT-Safe
  • 规范的默认要求是：函数不得访问其他线程可见的外部状态（XSH 2.1.1§6）
  • 不安全的根本原因：静态存储（static buffer）、全局状态（global state）、非原子操作序列
  • POSIX 的 _r 后缀约定（如 localtime_r）是规范化的重入安全解决方案
  • Async-Signal-Safe 的本质约束：信号可以在任意点打断执行，不安全函数会导致死锁或状态损坏

📝 ISO C 类头文件总结

  • mmap() 是 AS-Safe 的少数高级 API 之一，信号处理中可安全调用
  • POSIX 共享内存（shm_open + mmap）是现代进程间大数据共享的推荐方式

  • <regex.h> 的 MT-Safe 设计：编译（regcomp）后的 regex_t 可并发 regexec，符合"只读共享安全"原则
  • 所有用户/组数据库函数（pwd.h/grp.h）都有对应的 _r 重入版本，多线程中必须使用 _r 版本
  • 国际化（i18n）框架优先级：GNU gettext（<libintl.h>）> POSIX 消息目录（<nl_types.h>）
  • <termios.h> 提供了对终端的完整控制，是所有 TUI/CLI 工具的底层基础
  • <endian.h> 中的函数/宏通常是内联的，因此是 AS-Safe 的——这是网络协议实现的最佳实践

🧠 第一性原理分析

原理一：最小惊讶原则（Principle of Least Astonishment）

POSIX 头文件的分类以功能内聚为核心——同一个功能域的所有 API 放在一个或少数几个头文件中。这使得程序员只需包含少数头文件就能完成一项任务，减少了意外的符号污染。

原理二：线程安全的本质——隔离与不变量

POSIX 线程安全体系的根本逻辑是：函数的不安全性等于其对外部可见状态的依赖程度。

  • 无外部状态 → AS-Safe（如 sin(), memcpy()）
  • 有内部锁保护的外部状态 → MT-Safe（如 malloc(), pthread_mutex_lock()）
  • 有未保护的外部状态 → MT-Unsafe（如 strtok(), localtime()）
  • 有外部状态且在信号中可能死锁 → 非 AS-Safe（如 printf(), malloc()）

解决方案是将状态外化——把静态内部状态变成调用者传入的参数（_r 后缀约定）。

原理三：抽象层次原则

POSIX 头文件形成了清晰的抽象层次：

ISO C 基础（stdint/stddef/stdlib） ↓ POSIX 系统调用封装（unistd/fcntl/sys/types） ↓ 进程/线程抽象（pthread/semaphore/spawn） ↓ 网络/IPC 抽象（socket/netdb/mqueue） ↓ 应用领域专用（termios/regex/iconv/dlfcn）

每一层只依赖下层的接口，上层应用可以自由选择从哪个抽象层次接入。

原理四：可移植性与性能的权衡

POSIX 选择提供标准接口，而非强制最优实现。这体现在：

  • poll() 而非 epoll（后者是 Linux 扩展）：标准接口保证可移植，平台特定接口提供最优性能
  • POSIX IPC（mqueue/semaphore）与 SysV IPC 并存：SysV 是历史接口，POSIX 是更清晰的替代
  • pthread.h 与 threads.h 并存：向后兼容与标准演进并重
  • _r 后缀重入版本：以额外 API 表面积换取线程安全，不破坏历史兼容性
