序言为什么以C89为基础？

📖 导读
本章阐明选择C89作为基准的根本原因，帮助读者理解后续内容的分析框架：C89是所有后续标准的起点，理解它才能真正读懂C99与C11的每一处改动背后的动机。

C89（正式名称 ANSI X3.159-1989，随后被ISO采纳为 ISO/IEC 9899:1990 即 C90）是C语言的第一个正式国际标准。在此之前，C程序员只能依赖 Kernighan & Ritchie 1978年出版的《The C Programming Language》（K&R C）作为事实标准——这本书优秀但并不精确，不同编译器厂商对语言行为的解读各异，严重阻碍了跨平台开发。

C语言标准化动机

C89的核心价值在于：它不是凭空设计的，而是对当时广泛使用的C代码进行编纂和规范，尽量不破坏已有代码。X3J11委员会从1983年开始工作，历经6年，确立了"保持向后兼容、精确定义语义、区分实现定义行为/未指定行为/未定义行为"的三大原则。

🔑 第一性原理

C89的存在解决了一个根本问题：没有精确规范，就没有可靠的跨平台编程。理解C89，就是理解C语言设计者在"表达能力、效率、可移植性"三者之间权衡的每一个决定。

参考来源： C89 Standard Draft (yodaiken.com) · iso-9899.info - The Standard

§1C语言演化史

📖 导读
本章用时间线梳理C语言从诞生到C11的完整演化路径，重点解释每次标准迭代的动机——是什么现实问题推动了新特性的引入，而不是孤立地列举特性。

1972 — C语言诞生

Dennis Ritchie 在贝尔实验室为 Unix 操作系统开发C语言，前身是B语言。目标：高效的系统编程语言，可移植汇编替代品。

1978 — K&R C

Kernighan & Ritchie 出版《The C Programming Language》第一版，成为事实标准。但不同编译器（MSVC、GCC 前身、Lattice C）各自扩展，互不兼容。

1983–1989 — X3J11委员会

ANSI 成立 X3J11 技术委员会，目标：「提供一个明确、机器无关的C语言定义」。历时6年，1989年通过，1990年被ISO采纳。

1989 — C89 / ANSI C

首个正式标准。引入函数原型、标准库、const/volatile、enum。确立32个关键字、类型系统、未定义行为体系。

1995 — C95 Amendment

小幅修正：引入宽字符库 <wchar.h>、<wctype.h>，以及提供C运算符替代拼写的 <iso646.h>（如 and、or、not 等）。

1999 — C99

重大更新。动机：现代编程实践（C++影响）、数值计算需求、嵌入式系统。新增：VLA、// 注释、混合声明、stdint.h、_Bool、inline、restrict、复数、柔性数组成员等。

2011 — C11

动机：多核时代、Unicode需求、安全编程。新增：线程模型（threads.h）、原子操作（stdatomic.h）、_Generic、_Static_assert、对齐控制、匿名结构体/联合体。

2017 — C17

仅修正C11中已知的缺陷报告（DR），无新语言特性。确立为当前最稳定标准。

C语言标准关键词增长 · 时间轴
标准年份关键词数核心动机重要机构
C891989/199032统一碎片化的C实现，提供可移植性基础ANSI X3J11 → ISO
C99199937 (+5)现代编程实践、数值计算、嵌入式优化ISO/IEC WG14
C11201144 (+7)多核并发、Unicode、安全边界检查ISO/IEC WG14
🔑 第一性原理

每一次标准迭代都是对「现实编程需求 vs. 语言能力缺口」的回应。C89解决了规范缺失，C99回应了90年代十年的工程实践积累，C11则是多核革命与Unicode普及的产物。标准演化不是设计者的偏好，而是工程现实的倒逼。

参考来源： C Standards History - TutorialsPoint · iso-9899.info - The Standard

3.2 基础类型与大小约束

C89 不规定类型的具体字节数，只规定最小范围和相互之间的大小关系（sizeof约束）。这是一个关键设计决策：允许编译器将int映射到最高效的本机字长。

3.3 派生类型

结构体（struct）

/* C89: struct成员必须全部声明完毕再使用 */
struct Point {
    int x;
    int y;
};  /* 注意此处分号是必须的 */

/* 自引用结构体（链表节点）*/
struct Node {
    int data;
    struct Node *next;  /* 可以使用指向自身的指针 */
};

/* C89 不支持匿名结构体，必须命名 */
struct Point p = {3, 4};  /* 按顺序初始化 */

联合体（union）

/* union 所有成员共享同一内存空间 */
union Data {
    int   i;
    float f;
    char  str[20];
};
/* sizeof(union Data) == 20 (最大成员大小) */

union Data d;
d.i = 42;
/* 之后访问 d.f 是未定义行为（C89严格意义上）*/

位域（Bit-fields）

struct Flags {
    unsigned int read    : 1;
    unsigned int write   : 1;
    unsigned int execute : 1;
    unsigned int padding : 29; /* 填充到32位 */
};
/* C89规定：位域只能是 int、unsigned int 或 signed int */
/* 具体布局是实现定义的（字节序、填充方式均未规范） */

3.4 类型转换规则（Arithmetic Conversions）

C89定义了整型提升（Integral Promotions）和通常算术转换（Usual Arithmetic Conversions）两套规则，决定二元运算时操作数如何转换：

C89 通常算术转换优先级（由高到低）
/* 整型提升示例 */
char a = 200, b = 100;
int  c = a + b;  /* a和b先提升为int，再相加，结果300不溢出 */

/* 通常算术转换示例 */
int    i = -1;
unsigned u = 1;
/* 混合 signed/unsigned: i 转换为 unsigned -1 → 很大的正数 */
/* 在C89中：i + u 的类型是 unsigned int，结果为 UINT_MAX */
if (i < u)  /* 此条件为 FALSE！因为 (unsigned)-1 > 1 */
    printf("逻辑错误：永远不会执行\n");
⚠️ 最常见的C陷阱之一：有符号数与无符号数混合运算时，有符号数会被提升为无符号数，可能导致逻辑错误。这在C89/C99/C11中均如此。
🔑 第一性原理

C89类型系统的根本目标是「效率优先，可移植性通过规范保证」。不规定int大小是为了让每个平台选择最高效的整数宽度；类型转换规则的存在是为了让混合类型运算有确定的语义，而不是抛出错误。

参考：C89 §3.1.2.5 Types, §3.2.1 Conversions

4.3 链接性（Linkage）

/* ── 文件作用域 + 外部链接 ── */
int global_var = 42;       /* 可被其他.c文件引用 */
void public_func(void);    /* 外部可见函数声明 */

/* ── 文件作用域 + 内部链接 ── */
static int module_var = 0; /* 只有本.c文件可见 */
static void private_func(void) { /* 模块私有函数 */ }

void example(void) {
    /* ── 块作用域 + 自动存储 ── */
    int local = 10;           /* 函数返回后消亡 */

    /* ── 块作用域 + 静态存储 ── */
    static int count = 0;    /* 跨调用保持值，初始化一次 */
    count++;

    /* ── register 提示 ── */
    register int i;           /* 不可以 &i，建议放寄存器 */
    for (i = 0; i < 100; i++) { /* 性能敏感循环 */ }
}

4.4 const 与 volatile

C89在K&R C基础上引入了这两个类型限定符（Type Qualifiers），解决了之前缺乏表达常量对象和硬件映射内存的问题：

/* const：对象的值不应被修改（但不一定放在ROM中） */
const int MAX = 100;
const char *str = "hello";   /* 指针指向const数据 */
char * const ptr = buf;       /* const指针（地址不变）*/
const char * const cptr = s; /* const指针指向const数据 */

/* volatile：告知编译器不要优化掉对该变量的访问 */
/* 典型用途：内存映射I/O、信号处理、多线程（C89时代无线程标准）*/
volatile int *port = (volatile int *)0xFFFF0000;
*port = 1;   /* 编译器必须实际执行此写操作 */
*port = 0;   /* 同上，不会被优化为只执行最后一次 */
💡 const 与 #define 的区别：const int MAX = 100; 是一个有类型的对象，有作用域和链接性，调试器可见。#define MAX 100 是预处理器文本替换，无类型，调试器不可见。C89 推荐对常量使用 const。
🔑 第一性原理

存储类和作用域系统的本质是：用静态分析（编译期）替代运行时开销，实现内存管理自动化。auto变量在栈上零开销分配/释放，static变量在BSS/data段，register给编译器最大优化空间——这正是C在系统编程中的核心竞争力。

参考：C89 §3.1.2.1-3.1.2.4, §3.5.1-3.5.3

5.2 序列点与未定义行为

C89规范（§2.1.2.3, §3.3）引入了序列点（Sequence Point）的概念：在序列点处，所有副作用都必须完成。序列点之间对同一对象的多次修改是未定义行为。

序列点位置
/* ─── 未定义行为示例 ─── */
int i = 5;

/* UB: i 在两个序列点之间被修改两次 */
i = ++i + 1;

/* UB: i 被修改，同时被读取（非为确定新值）*/
int a = i++ + i;

/* UB: 函数参数求值顺序未规定 */
printf("%d %d\n", i++, i++);

/* ─── 合法示例 ─── */
int b = i++;    /* OK: i只修改一次 */
int c = i + i;  /* OK: i只读不写 */
if (i++ > 0 && i > 5) {}  /* OK: &&是序列点 */

5.3 指针运算

int arr[5] = {10, 20, 30, 40, 50};
int *p = arr;     /* p 指向 arr[0] */

p++;              /* p 前进 sizeof(int) 个字节 */
*p == 20;        /* true */

p + 2;            /* 指向 arr[3]（即 p[2]）*/
arr[i] == *(arr + i);  /* C89规定：下标等价于指针+偏移的解引用 */

/* 合法指针范围：arr[0] 到 arr[5]（哨兵位，不可解引用）*/
int *end = arr + 5;  /* 指向数组末尾后一位是合法的 */
/* *end 是 UB! */
🔑 第一性原理

C89运算符体系的本质是：以最小的语义约束换取最大的编译器优化空间。"未定义行为"不是设计失误，而是允许编译器假设程序员不会触发UB，从而进行激进优化（如重排指令、消除冗余读写）。这是C的系统编程效率之源，也是安全编程的最大挑战。

参考：C89 §3.3 Expressions, §2.1.2.3 Sequence points

§6语句与控制流

📖 导读
本章解析C89规范§3.6的全部语句类型。重点关注C89的限制（声明必须在块开头）以及与C99的对比，同时深入分析switch语句的fall-through陷阱。

6.1 C89语句分类

C89 语句类型

6.2 C89 关键限制：声明必须在块头部

C89规范（§3.6.2）要求：复合语句（块）中，所有声明必须出现在任何语句之前。这是C89与C99最常被提及的差异之一：

/* ─── C89 合法写法 ─── */
{
    int i, sum;          /* 声明必须在所有语句之前 */
    int temp;

    sum = 0;             /* 语句从这里开始 */
    for (i = 0; i < 10; i++) {
        int sq;          /* 内嵌块开头可以声明 */
        sq = i * i;
        sum += sq;
    }
}

/* ─── C89 非法写法（C99才合法）─── */
int x = 10;
x++;              /* 语句 */
int y = x;       /* ❌ C89: 声明不能出现在语句之后 */

/* ─── C89非法但C99合法的for循环 ─── */
for (int i = 0; i < 10; i++) { /* ❌ C89 ✅ C99 */
}

6.3 switch 语句与 fall-through

switch (expr) {
    case 1:
        do_one();
        break;  /* break 阻止 fall-through */

    case 2:
    case 3:             /* case 2 和 3 共享处理逻辑 */
        do_two_three();
        break;

    case 4:
        step_one();
        /* 故意fall-through：无break时执行下一case */
    case 5:
        step_two();  /* case 4 和 5 都会执行此句 */
        break;

    default:
        handle_default();
        /* default 可在任意位置，但建议放最后 */
}
⚠️ C89的switch控制表达式类型必须是整型（integral type）。case标签必须是整型常量表达式。忘记break导致的fall-through是C中最常见的bug之一，C11及后续标准草案曾讨论添加警告机制。

6.4 goto 的合法用途

/* goto 在C89中的经典合法用途：跳出多重嵌套循环 */
int found = 0;
int i, j, k;
for (i = 0; i < N && !found; i++) {
    for (j = 0; j < M; j++) {
        for (k = 0; k < K; k++) {
            if (condition(i,j,k)) {
                found = 1;
                goto done;  /* 直接跳出三层循环 */
            }
        }
    }
}
done:
process_result(found);
🔑 第一性原理

C89语句系统设计的核心原则：语句是顺序执行的，控制流结构是显式的。"声明必须在块头部"这一限制简化了早期编译器的实现（单遍扫描），是技术条件的历史印迹，而非语言设计哲学。C99移除这个限制，是对编程体验的重大改善。

参考：C89 §3.6 Statements

7.2 隐式函数声明（C89的遗留危害）

C89（承继自K&R）允许在调用前未声明的函数：编译器会假设它返回 int 并接受任意参数。这被称为隐式函数声明，是臭名昭著的C陷阱：

/* C89中合法（但危险）*/
int main(void) {
    /* 未#include ，malloc被假设返回int */
    int *p = malloc(100);  /* 在64位系统：int截断为指针！ */
    return 0;
}
/* C99 起，隐式函数声明被删除，上述代码是编译错误 */

7.3 参数传递：值传递语义

/* C89（及所有C标准）函数参数均为值传递 */
void swap_wrong(int a, int b) {
    int tmp = a; a = b; b = tmp;
    /* 只修改了形参，调用者的实参不变 */
}

void swap_correct(int *a, int *b) {
    int tmp = *a; *a = *b; *b = tmp;
    /* 通过指针修改调用者的数据 */
}

/* 数组参数：名称退化为指向首元素的指针 */
void fill(int *arr, int n, int val) {
    int i;
    for (i = 0; i < n; i++) arr[i] = val;
}
🔑 第一性原理

函数原型的核心价值是将类型检查从运行时提前到编译时。这一变化使C程序员能够在不损失运行效率的前提下，获得编译器的类型安全保障。K&R C的"信任程序员"哲学通过函数原型演化为"让编译器帮你验证"哲学。

参考：C89 §3.5.4.3, §3.7.1 Function definitions

§9C99 新特性详解

📖 导读
C99（ISO/IEC 9899:1999）是C89发布10年后的重大更新，草案编号N1256。本章对每项新特性阐明：背景动机、语法规范、使用示例，以及与C89的明确对比。

9.1 单行注释 //

背景：C++从诞生起就支持 // 注释，C程序员在与C++混合编程时频繁使用这种语法。C89只允许 /* */，且不能嵌套。C99正式将单行注释纳入标准。

/* C89: 只有这种注释形式 */
// C99: 引入单行注释，从//到行尾
// 注意：不能跨行，续行符(\)使下一行也成为注释

int x = 10; // 赋值为10 — 行内注释很方便
C89 ✗C99 ✓

9.2 混合声明与语句

背景：C89要求块内所有声明在语句之前，这是为简化单遍编译器的历史限制。C99取消此限制，允许在需要时声明变量，提升代码可读性并减少无意义初始化。

/* C99: 声明可出现在任何语句之后的位置 */
void example(int n) {
    setup();          /* 先执行语句 */
    int result = compute(n);  /* ✅ C99: 声明在语句之后 */

    /* for循环变量声明（最常用场景）*/
    for (int i = 0; i < n; i++) {  /* ✅ C99 */
        /* i的作用域限于for循环 */
    }
    /* i 在此处不可见 */
}
C89 ✗C99 ✓

9.3 可变长数组（VLA）

背景：C89只允许编译期确定大小的数组（或动态malloc）。科学计算和矩阵操作需要运行时大小的栈上数组，VLA填补了这一需求。注意：C11将VLA改为可选特性。

/* C89: 必须用固定大小或动态分配 */
void process_c89(int n) {
    int *arr = (int*)malloc(n * sizeof(int));
    /* ...使用... */
    free(arr);
}

/* C99: 可变长数组，自动在栈上分配 */
void process_c99(int n) {
    int arr[n];    /* n在运行时确定，sizeof(arr)也在运行时计算 */
    int i;
    for (i = 0; i < n; i++) arr[i] = i;
    /* 无需free，离开作用域自动释放 */
}

/* VLA用于二维数组传递（非常实用）*/
void matrix_fill(int rows, int cols, int mat[rows][cols]) {
    int i, j;
    for (i = 0; i < rows; i++)
        for (j = 0; j < cols; j++)
            mat[i][j] = i * cols + j;
}
C89 ✗C99 ✓C11 可选
⚠️ VLA在栈上分配，过大的n可能导致栈溢出。内核代码（如Linux 4.20+）已禁止使用VLA，因其无法检测分配失败。

9.4 指派初始化器（Designated Initializers）

背景：C89只支持按顺序初始化聚合类型。当结构体成员较多或需要稀疏初始化数组时，代码可读性极差。C99引入指派语法，允许按名称/索引初始化。

/* 结构体指派初始化 */
struct Point { int x, y, z; };

/* C89: 必须按顺序 */
struct Point p89 = {1, 2, 3};

/* C99: 可按成员名初始化，顺序任意 */
struct Point p99 = {.z = 3, .x = 1, .y = 2};
struct Point partial = {.x = 5};  /* y,z 默认为 0 */

/* 数组指派初始化 */
int arr[10] = {[0]=1, [5]=10, [9]=99};
/* 其余元素为0：{1,0,0,0,0,10,0,0,0,99} */

/* 嵌套使用 */
struct Rect { struct Point tl, br; };
struct Rect r = {.tl = {.x=0, .y=0}, .br = {.x=100, .y=100}};
C89 ✗C99 ✓

9.5 复合字面量（Compound Literals）

/* 创建匿名的聚合对象，用于临时传递 */
void draw_point(struct Point p);

/* C89: 必须先声明再传递 */
struct Point tmp = {3, 4, 0};
draw_point(tmp);

/* C99: 直接在表达式中创建 */
draw_point((struct Point){.x=3, .y=4});

/* 复合字面量有自动存储期，地址可取 */
int *p = (int[]){1, 2, 3, 4, 5};
process(p, 5);

9.8 inline 函数

背景：C89时代用函数宏实现内联代码，但宏无类型检查且有副作用问题。C99引入 inline 关键字，让编译器优化掉函数调用开销同时保留类型安全。

/* C99: inline 函数建议编译器内联展开 */
inline int max(int a, int b) {
    return a > b ? a : b;
}

/* C99 inline 语义细节：
   - inline 只是建议，编译器可忽略
   - inline 函数在头文件中定义时，多个TU包含不会冲突
   - 需要 static inline 才能避免多定义问题（最常见用法）*/
static inline int clamp(int v, int lo, int hi) {
    return v < lo ? lo : (v > hi ? hi : v);
}

9.9 restrict 限定符

背景：编译器在优化指针操作时，必须保守地假设两个指针可能指向同一内存（别名问题）。restrict告诉编译器：在该指针的生命期内，只有这个指针能访问指向的内存，从而启用激进向量化优化。

/* 标准库函数使用 restrict 的例子 */
void *memcpy(void * restrict dst,
             const void * restrict src,
             size_t n);
/* restrict 承诺 dst 和 src 不重叠，memmove 无此承诺 */

/* 用户代码示例：向量加法 */
void vec_add(float * restrict out,
             const float * restrict a,
             const float * restrict b,
             int n) {
    int i;
    for (i = 0; i < n; i++)
        out[i] = a[i] + b[i];
    /* 编译器可安全自动向量化（SIMD）*/
}
C89 ✗C99 ✓

9.10 柔性数组成员（Flexible Array Member）

/* 结构体最后一个成员可以是未指定大小的数组 */

/* 动态分配包含数据的结构体 */

§10C11 新特性详解

📖 导读
C11（ISO/IEC 9899:2011，草案N1570，2011年12月8日发布）在多核时代背景下出现，核心主题是并发安全与类型安全增强。本章解析每项新特性的工程背景与使用方法。
参考文档：N1570 Draft

10.1 _Static_assert 静态断言

背景：C89/C99只有运行时 assert()。编译期条件检查只能靠复杂的宏技巧（如"数组负大小"trick）。C11提供原生的编译期断言，触发时产生清晰的编译错误信息。

#include <assert.h>   /* 提供 static_assert 别名 */

/* 编译时检查，失败则编译错误（不是运行时错误）*/
_Static_assert(sizeof(int) == 4,  "本代码要求int为4字节");
_Static_assert(sizeof(long) >= 8, "需要64位long");

/* 实际用途：检查结构体布局 */
struct Packet {
    uint8_t  type;
    uint8_t  flags;
    uint16_t length;
    uint32_t checksum;
};
_Static_assert(sizeof(struct Packet) == 8,
               "Packet结构体存在意外的填充字节");

/* 使用宏别名（需要assert.h）*/
static_assert(sizeof(size_t) >= 4, "size_t太小");
C89 ✗C99 ✗C11 ✓

10.2 _Generic 类型泛型选择

背景：C标准数学库有 sin()（double）、sinf()（float）、sinl()（long double）等变体。C99引入了<tgmath.h>用宏实现类型分发，但实现复杂且不可扩展。C11的 _Generic 提供语言层面的泛型选择机制。

/* _Generic(控制表达式, 类型1: 值1, 类型2: 值2, ..., default: 默认值) */
/* 根据控制表达式的类型，在编译期选择对应的值 */

#define type_name(x) _Generic((x), \
    int:         "int",   \
    double:      "double",\
    float:       "float", \
    char*:       "string",\
    default:     "unknown")

/* 实现类型安全的 abs 宏 */
#define my_abs(x) _Generic((x), \
    int:         abs(x),    \
    long:        labs(x),   \
    double:      fabs(x),   \
    float:       fabsf(x)   )

int main(void) {
    printf("%s\n", type_name(42));     /* int */
    printf("%s\n", type_name(3.14));   /* double */
    printf("%f\n", my_abs(-3.14));    /* 3.14 */
    return 0;
}
C89 ✗C99 ✗C11 ✓

10.3 多线程支持 <threads.h>

背景：C89/C99没有任何标准线程模型。在C11之前，C程序只能使用POSIX pthread（Unix）或Windows API，导致代码不可移植。C11提供平台无关的线程接口（此特性为可选）。

#include <threads.h>

/* 检查实现是否支持线程 */
#ifdef __STDC_NO_THREADS__
#error "此平台不支持C11线程"
#endif

int worker(void *arg) {
    int id = *(int*)arg;
    printf("线程 %d 运行中\n", id);
    return thrd_success;
}

int main(void) {
    thrd_t t1, t2;
    int id1 = 1, id2 = 2;

    thrd_create(&t1, worker, &id1);
    thrd_create(&t2, worker, &id2);
    thrd_join(t1, NULL);
    thrd_join(t2, NULL);

    /* 互斥锁 */
    mtx_t mutex;
    mtx_init(&mutex, mtx_plain);
    mtx_lock(&mutex);
    /* 临界区 */
    mtx_unlock(&mutex);
    mtx_destroy(&mutex);

    return 0;
}
C11 可选特性

10.4 原子类型 <stdatomic.h>

背景：多线程程序中，普通变量的读写不是原子的，需要互斥锁保护。对于简单的计数器、标志位，使用锁代价过高。C11引入原子类型，保证操作的原子性并提供内存序控制。

#include <stdatomic.h>

/* 原子整型，多线程安全访问 */
atomic_int counter = ATOMIC_VAR_INIT(0);

void increment(void) {
    atomic_fetch_add(&counter, 1);  /* 原子加1 */
}

int get_count(void) {
    return atomic_load(&counter);    /* 原子读取 */
}

/* 内存序控制（高级用法）*/
atomic_store_explicit(&counter, 0, memory_order_relaxed);
atomic_flag lock = ATOMIC_FLAG_INIT;

/* 自旋锁实现 */
while (atomic_flag_test_and_set(&lock)) {} /* 加锁 */
/* 临界区 */
atomic_flag_clear(&lock);  /* 释放锁 */
C11 可选特性

10.5 匿名结构体/联合体

背景：C89/C99中，联合体成员必须通过名称访问，即使联合体嵌套在结构体中也需要写出中间层名称，导致代码冗长。C11正式允许匿名的结构体和联合体成员。

/* C99: 必须命名内部联合体 */
struct Value99 {
    int type;
    union { int i; double d; } u;  /* 有名字 */
};
struct Value99 v; v.u.i = 42;  /* 需要 .u. */

/* C11: 匿名联合体，成员直接可见 */
struct Value11 {
    int type;
    union {           /* 匿名联合体 */
        int    i;
        double d;
        float  f;
    };                /* 无名称 */
};
struct Value11 v11; v11.i = 42;  /* ✅ 直接访问 */

/* 实用示例：向量数学库 */
typedef struct {
    union {
        struct { float x, y, z; };  /* 命名访问 */
        float v[3];                 /* 数组访问 */
    };
} Vec3;
Vec3 pos = {{1.0f, 2.0f, 3.0f}};
pos.x;     /* 等价于 pos.v[0] */
C89 ✗C99 ✗C11 ✓

10.6 对齐控制 _Alignas / _Alignof

#include <stdalign.h>  /* 提供 alignas, alignof 宏别名 */

/* _Alignof: 查询类型的对齐要求 */
size_t align = _Alignof(double);  /* 通常为8 */
size_t pa    = alignof(int);      /* stdalign.h别名 */

/* _Alignas: 指定变量/成员的对齐方式 */
_Alignas(64) char cache_line[64]; /* 64字节对齐（缓存行优化）*/
alignas(double) char buf[sizeof(double)]; /* 按double对齐 */

/* 对齐内存分配 */
void *ptr = aligned_alloc(64, 1024);  /* C11新增 */
free(ptr);
C11 ✓

10.7 _Noreturn 函数标注

#include <stdnoreturn.h>  /* 提供 noreturn 宏 */

/* 标注不会返回的函数，帮助编译器优化和诊断 */
_Noreturn void fatal_error(const char *msg) {
    fprintf(stderr, "致命错误: %s\n", msg);
    exit(1);
}

noreturn void panic(const char *msg);  /* 宏别名 */

int divide(int a, int b) {
    if (b == 0) fatal_error("除以零");  /* 编译器知道此后不会继续 */
    return a / b;  /* 编译器不会因为某路径无return而警告 */
}
C11 ✓

10.8 Unicode支持

#include <uchar.h>

/* C11 新增字符类型 */
char16_t c16 = u'中';     /* UTF-16 字符 */
char32_t c32 = U'😊';    /* UTF-32 字符（完整Unicode码点）*/

/* 字符串字面量前缀 */

C11 ✓
🔑 第一性原理

C11的核心驱动力是多核时代的到来。原子操作和线程支持不是锦上添花，而是让C程序在不依赖特定操作系统API的情况下，具备可移植的并发能力。_Generic是对"类型安全的泛型编程"这一需求的最小化语言支持。C11的每个特性都针对具体的工程需求，没有"为了有而有"的设计。

11.3 如何选择标准？

使用场景推荐标准理由
极端可移植性（老旧嵌入式、历史编译器）C89几乎所有C编译器都支持，工具链兼容性最佳
Linux内核及类Unix系统编程C89 / C99Linux内核使用GNU C（C89+扩展），用户态多用C99
嵌入式系统（中端MCU）C99stdint.h对嵌入式至关重要，GCC/Clang支持完善
现代应用程序开发C11静态断言、_Generic、匿名struct提升代码质量
多线程安全程序C11原子操作和线程支持是可移植并发的唯一标准选项
高性能数值计算C99 / C11restrict、VLA（C99）、_Alignas（C11）
🔑 第一性原理

三个标准之间的关系是完全向后兼容的超集：所有合法的C89代码都是合法的C99代码（除了删除的隐式规则），所有合法的C99代码都是合法的C11代码（除了VLA变为可选）。选择标准不是"放弃旧代码"，而是"选择在哪个能力边界上工作"。

🔑 整体第一性原理总结

C语言从K&R到C89，从C89到C99，从C99到C11，每一步都遵循同一个元原则：在不牺牲已有代码的前提下，用最小的语言改动，解决最真实的工程问题。

C不是一门追求优雅或表达力的语言，而是一门追求可预测的性能和精确的控制的语言。理解C标准，就是理解在计算机硬件约束下，人类工程师如何在"简单"与"强大"之间寻找永恒的平衡点。
