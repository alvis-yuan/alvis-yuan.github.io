1.3 本书定位与目标读者

本书聚焦于底层共享内存并行编程，语言为 C，目标场景是操作系统内核、并行数据库、底层库等。它不是分布式系统手册，也不是 GPU 编程指南。

导论
Introduction

为什么并行编程难？从历史、目标、挑战三个维度建立全局认知框架

🔗 官方原文：kernel.org/perfbook
⚠️
2.1 并行编程的历史困境

并行编程困难的历史由来已久：硬件的多样性（共享内存、消息传递、向量机等架构层出不穷）、工具链不成熟、以及开发者的顺序思维惯性，共同构成了并行编程的"原罪"。

Intel 于 2000 年在 Pentium 引入超线程，AMD/Intel 在 2005 年引入双核 CPU，自此多核成为主流。时钟频率的提升已到天花板，性能增长必须靠并行来实现。

🎯
2.2 并行编程的目标

McKenney 总结了并行编程的三大核心目标：

  • 性能（Performance）：Amdahl 定律告诉我们，串行部分是扩展性的天花板
  • 生产力（Productivity）：代码要让人类能够理解和维护，正确性比极致性能更重要
  • 通用性（Generality）：解决方案应当在不同场景下可复用
三角张力：这三个目标天然矛盾。最快的代码往往最难理解；最通用的接口往往牺牲性能。本书的核心主题之一，就是如何在这三者之间找到工程上的最优平衡点。

🔄
2.3 并行编程的替代方案

在投入并行编程之前，McKenney 建议先考虑：

  • 多进程模式：用多个独立进程（如 nginx worker），天然无共享内存难题
  • 复用已有并行软件：数据库、消息队列等已内置并行能力
  • 性能调优：算法改进往往比并行化效果更显著
工程原则：并行化是有成本的。如果顺序程序已经足够快，或者可以通过更好的算法解决问题，就没有必要引入并行编程的复杂性。
🔬 第一性原理解读
并行编程难的本质原因有两层：物理层面，现代 CPU 为了性能引入了乱序执行、缓存、流水线等机制，这些机制与直觉中的"同步、有序"世界观相悖；认知层面，人类习惯顺序推理，并行执行产生的组合状态空间指数级爆炸。McKenney 在这章建立的核心洞察是：并行编程困难不是宿命，而是工具与认知框架尚未成熟的表现。一旦建立了正确的心智模型并掌握了正确的工具，并行编程同样可以工程化。

3.1 概览：CPU 的真实工作方式

现代 CPU 并非简单执行指令的机器。它内部有多级缓存（L1/L2/L3）、乱序执行单元、存储缓冲区（store buffer）和失效队列（invalidation queue）。这些机制在提升单核性能的同时，制造了多核编程中所有复杂性的根源。

⏱️
3.2 各操作的实际开销

McKenney 给出了真实硬件上各种操作的延迟数据（以一个典型的多核系统为参考）：

/* 近似延迟参考（因硬件而异）*/ L1 缓存命中: ~1-4 时钟周期 L2 缓存命中: ~10-15 时钟周期 L3 缓存命中: ~40-75 时钟周期 DRAM 访问: ~200+ 时钟周期 核间缓存行转移: ~1000 时钟周期 互斥锁加锁（无竞争）: ~150 时钟周期 互斥锁加锁（有竞争）: ~数千 时钟周期
关键洞见：一次核间缓存行传输（cache line transfer）的开销相当于数百次 L1 命中。这意味着在并行程序中，共享数据的争用代价极其高昂，远超大多数程序员的直觉预期。
🍔
3.3 硬件的"免费午餐"？

引用 Herb Sutter 的著名论断：免费午餐已经结束（The Free Lunch Is Over）。单核性能的摩尔定律时代，程序员可以"躺赢"——什么都不做，等两年硬件升级，程序自动变快。多核时代，这条路已经走不通。

但硬件也在积极演进：NUMA 架构、硬件事务内存（HTM）、更大的 SIMD 宽度等技术为软件提供了新工具，但也引入了新的复杂性。

🛠️
3.4 对软件设计的影响

本章最重要的结论是：硬件结构直接决定软件设计决策。例如：

  • 频繁被多核写入的数据（如全局计数器）会引发大量 MESI 协议流量，性能极差
  • 每个 CPU 维护自己的计数器副本，最后汇总，可以消除大部分开销
  • 数据的内存布局（热/冷分离、False Sharing 的避免）对性能至关重要
🔬 第一性原理解读
从第一性原理看，现代多核系统的性能瓶颈根本上来自一个物理限制：光速。信号在芯片上的传播速度有限，远距离通信（跨核、跨 NUMA 节点、到内存）必然产生延迟。缓存存在的理由正是利用局部性（locality）来规避光速限制。因此，并行程序的设计哲学可以归结为：让每个 CPU 尽量只访问"靠近自己"的数据，减少远程通信。后续所有技术（分区、所有权、RCU 等）都是这个原则的工程实现。

并行工具箱
Tools of the Trade

掌握并行编程的基础工具：POSIX 线程、原子操作与 Linux 内核原语

🔗 官方原文：kernel.org/perfbook

4.1-4.2 脚本语言与 POSIX 多进程

本章首先介绍最高层的并行手段：Shell 脚本利用进程级并行（make -j 就是典型例子），完全避免共享内存问题。

POSIX 线程（pthreads）是本书主要使用的并行原语。核心 API 包括：

/* 线程创建 */ int pthread_create(pthread_t *tid, pthread_attr_t *attr, void *(*func)(void *), void *arg); /* 互斥锁 */ pthread_mutex_lock(&mylock); /* ... 临界区 ... */ pthread_mutex_unlock(&mylock); /* 条件变量 */ pthread_cond_wait(&mycond, &mylock); pthread_cond_signal(&mycond);
⚡
4.3 原子操作

原子操作是无锁编程的基石。现代 CPU 提供的原子指令包括：

  • Compare-And-Swap (CAS)：原子地比较并交换，是所有无锁算法的核心原语
  • Fetch-And-Add：原子地读取并递增，用于无锁计数器
  • Load-Linked/Store-Conditional (LL/SC)：ARM/RISC-V 上的等效机制
/* GCC 内建原子操作（C11 之前常见） */ int old = __sync_val_compare_and_swap(&x, expected, new_val); /* C11 标准原子 */ atomic_compare_exchange_strong(&x, &expected, new_val);
原子操作的代价：原子操作并非"免费"。它们会触发总线锁定或缓存一致性流量，在高竞争下性能可能不如预期。原子操作只是避免了锁的上下文切换，并不能消除核间通信延迟。
🐧
4.4-4.5 Linux 内核原语与工具选择

Linux 内核提供了与 POSIX 对应的一套原语，针对内核场景做了优化：

spinlock_t — 自旋锁
rwlock_t — 读写锁
atomic_t — 原子整型
per_cpu — Per-CPU 变量
rcu_read_lock — RCU 读锁
completion — 完成量

McKenney 的工具选择哲学：优先用最高层的工具，只有性能需求明确要求时才下降到底层。错误地使用低层工具（如手写无锁代码）往往带来比它解决的问题更多的麻烦。

🔬 第一性原理解读
所有并行工具的本质都是在解决同一个问题：如何在多个并发执行流之间协调对共享状态的访问。线程是执行流的抽象，锁是"谁拥有访问权"的协议，原子操作是无需外部协调的最小粒度操作。理解工具的选择，实质上是理解竞争程度与协调成本之间的权衡：竞争越激烈，协调成本越高；减少共享、增加独立性，才是根本出路。
📖 第 5 章

计数
Counting

以"并行计数器"为案例，深入演示并行设计的演进过程

🔗 官方原文：kernel.org/perfbook
为什么从"计数"开始？计数器看似是最简单的共享数据结构，但它完整地暴露了并行设计中的所有核心权衡：性能 vs 精度、可扩展性 vs 实现复杂度。这是一个绝佳的教学案例。
🔢
5.1 为什么并行计数不简单？

考虑最直觉的实现：全局变量 count++。在多核下，这是 race condition 的经典案例：

/* 危险！非原子操作，多核下结果不确定 */ long counter = 0; void increment() { counter++; } /* 读-改-写 非原子！ */ /* 解法一：用原子操作 */ atomic_long_inc(&counter); /* 问题：在 64 核系统上，高竞争导致性能崩溃 */
📊 不同计数器实现的可扩展性对比
📊
5.2 统计计数器（Statistical Counters）

关键思路：每个 CPU 维护自己的计数器副本，读取时汇总所有副本。

/* Per-CPU 计数器：写极快，读较慢（需汇总所有 CPU） */ DEFINE_PER_CPU(long, my_counter); void inc() { this_cpu_inc(my_counter); } /* 无锁！ */ long read() { long sum = 0; for_each_possible_cpu(cpu) sum += per_cpu(my_counter, cpu); return sum; /* 近似值！可能看到"过去"的状态 */ }

代价：读操作不再是 O(1)，而是 O(CPU数量)；读到的值是近似值，因为各 CPU 的计数没有全局同步点。

📉
5.3-5.4 近似限制计数器与精确限制计数器

当需要对计数值做上限控制时（如限速、资源配额），问题变得更复杂。本书介绍了两种方案：

  • 近似限制计数器：每个 CPU 持有局部配额，超过局部阈值才与全局同步。允许短暂超限，但性能极佳。
  • 精确限制计数器：每次接近限值时进行全局协调，保证绝不超限，但在临界区时性能有损。
工程权衡：很多"精确计数"的需求实际上并不需要绝对精确。一个请求限速系统允许短暂超出 1% 是完全可接受的，换来的是数量级的性能提升。
🔬 第一性原理解读
计数章节揭示了并行设计的核心定律：共享 = 串行化 = 瓶颈。任何被多个 CPU 共同写入的数据都必然成为性能瓶颈，因为维护它的一致性需要核间通信。解决办法只有一种根本路径：减少共享——复制（Replication）、分区（Partitioning）、局部积累后汇总（Local Accumulation）。精确性与可扩展性之间存在内禀的张力，工程师的任务是理解业务对精确性的真实需求，从而做出合理的折衷。

分区与同步设计
Partitioning and Synchronization Design

并行系统架构的顶层方法论：如何将问题拆分以最大化并行性

🔗 官方原文：kernel.org/perfbook
✂️
6.1 分区练习（Partitioning Exercises）

McKenney 用一系列实际案例（哲学家就餐、路由表等）展示分区的思考过程。关键原则：设计时先不考虑同步，先考虑如何让数据天然地"属于"某个线程。

思维反转：大多数程序员先写出共享数据结构，再想怎么加锁。McKenney 认为这是错误的顺序。应当先问：能否将数据设计成"不需要共享"的形式？
📐
6.2-6.3 设计准则与同步粒度

并行设计的核心准则可以用一个框架来描述：

数据分区：按数据域划分
功能分区：按功能职责划分
时间分区：按执行阶段划分
空间局部性：数据靠近使用它的 CPU
减少临界区长度：锁尽量细粒度
层次化同步：全局粗粒度+局部细粒度

同步粒度的选择是关键工程决策：粗粒度锁（一把大锁保护所有数据）实现简单但扩展性差；细粒度锁性能好但死锁风险高、代码复杂。

📊 分区策略对比（哈希表扩展性示意）
🚀
6.4 并行快路径（Parallel Fastpath）

一种重要的设计模式：让常见操作（快路径）完全并行、无锁，只在罕见情况（慢路径）时进行协调。

例如：读操作占 99% 时，用 RCU 让读操作完全无锁；写操作只占 1% 时允许它更昂贵。

🔬 第一性原理解读
分区设计的第一性原理来自 Amdahl 定律：程序的最大加速比由串行部分决定。同步 = 串行化，因此最大化并行度等价于最小化同步需求。最优的并行设计是：在设计阶段就将数据和任务切分为相互独立的单元，使得运行时根本不需要（或极少需要）协调。好的分区设计让并行成为自然的，而非强加的。

锁机制
Locking

并行编程的主力工具，同时也是最受误用的机制——深入理解锁的工作原理、实现与陷阱

🔗 官方原文：kernel.org/perfbook
⚠️
7.1 保持程序存活（Staying Alive）

锁使用不当会导致程序"死掉"。本节深入讨论三种致命问题：

  • 死锁（Deadlock）：两个线程互相等待对方持有的锁。预防：固定加锁顺序、使用锁层次（lock hierarchy）
  • 活锁（Livelock）：线程不断尝试加锁但总是失败，忙于重试而无法推进
  • 饥饿（Starvation）：某个线程长期无法获得锁，被高优先级线程或运气不好的调度持续抢占
/* 死锁经典场景 */ Thread A: lock(L1) → lock(L2) → ... Thread B: lock(L2) → lock(L1) → ... /* 预防：强制所有线程按 L1 → L2 的顺序加锁 */
🔐
7.2 锁的类型
互斥锁（Mutex）— 最基础
读写锁（rwlock）— 读并发
自旋锁（Spinlock）— 短临界区
顺序锁（Seqlock）— 读不阻塞写
公平锁（Fair Lock）— 防饥饿
MCS 锁 — 可扩展自旋

读写锁是一个常见的"伪优化"：在实际工作负载中，读者加锁与释放锁的开销往往高于预期，除非读操作极其耗时，否则不如直接用互斥锁。

🔧
7.3 锁的实现细节

一个正确的自旋锁实现需要依赖原子操作和内存屏障：

/* 简化的自旋锁实现（x86 test-and-set） */ typedef struct { int locked; } spinlock_t; void spin_lock(spinlock_t *l) { while (atomic_xchg(&l->locked, 1) == 1) cpu_relax(); /* PAUSE 指令，让出超线程资源 */ } void spin_unlock(spinlock_t *l) { smp_store_release(&l->locked, 0); /* 需要写屏障！ */ }
内存屏障的必要性：如果 unlock 不包含写屏障（store release），临界区内的内存操作可能在锁释放后才对其他 CPU 可见，导致数据竞争。锁的正确性依赖内存模型的保证。
🛡️
7.4-7.5 基于锁的存在性保证与锁的评价

锁不仅保护数据，也保证对象存在。当你在临界区内访问某个对象的指针时，锁确保了该对象在临界区期间不会被释放——这是内存安全的重要保证机制。

McKenney 对锁有著名的评价："锁是并行编程的主力工具，但也是并行编程最知名的反派。"锁的正确使用需要大量纪律：一致的加锁顺序、明确的保护关系文档、对 livelock/starvation 的考虑。

🔬 第一性原理解读
锁的本质是对互斥访问的协议实现：同一时刻只有一个执行者能持有锁，从而将并发访问串行化。锁的所有问题（死锁、活锁、性能损失）都根源于它将并行变串行的本质——这是必要的代价，但也是需要最小化的成本。理解锁的第一性原理意味着：每一把锁的存在都应该有明确的理由，保护明确的不变式（invariant），持有时间尽量短。不能用锁解决的问题不是锁的问题，而是设计问题。

最被忽视但最强大的同步策略：通过明确所有权彻底消除同步需求

🔗 官方原文：kernel.org/perfbook
核心洞见：最好的同步是不需要同步。如果数据只被一个线程访问，就根本不需要锁。数据所有权是实现这一理想的关键机制。
🏠
8.1-8.2 多进程与部分所有权

多进程模式是最彻底的所有权实现：进程间没有共享内存（除非显式创建共享内存段），每个进程完全拥有自己的地址空间。Unix 传统的 fork() 模型天然地实现了数据隔离。

线程局部存储（TLS / __thread）是线程内部的数据所有权机制：每个线程拥有自己的变量副本，无需任何同步。

/* 线程局部变量：每个线程一份，零同步开销 */ __thread int my_local_counter = 0; void increment() { my_local_counter++; /* 无锁！完全安全 */ }
📦
8.3-8.5 函数传送、指定线程与私有化

所有权的几种实现模式：

  • 函数传送（Function Shipping）：不共享数据，而是将操作"传送"到数据所在的线程执行（Actor 模型的核心思想）
  • 指定线程（Designated Thread）：某类数据只由特定线程负责，其他线程通过消息队列请求操作
  • 私有化（Privatization）：将原本共享的数据在需要时复制为私有副本，操作完成后再合并
📊 数据所有权模式对比
🔬 第一性原理解读
数据所有权的第一性原理是：竞争的根源是"共享"，消除共享就消除了竞争。如果每块数据都有明确的唯一所有者，那么所有者访问数据时根本不需要征得任何人的同意——世界上最快的锁，是不需要的锁。这一思想在现代编程语言中有深远影响：Rust 的所有权系统在编译时静态保证了数据所有权的唯一性，将运行时的并发错误转变为编译期错误，是这一原理的工程化极致。
📖 第 9 章

延迟处理
Deferred Processing

引用计数、危险指针、顺序锁与 RCU——并行编程最优雅的解法之一

🔗 官方原文：kernel.org/perfbook
延迟处理的核心思想：不是立即完成操作，而是将操作推迟到"安全"的时机执行。这一思想解决了读写并发中最棘手的问题：如何在读者还在访问数据时安全地修改或删除它。
🔢
9.1 引用计数（Reference Counting）

引用计数是最直觉的延迟释放机制：只有当计数归零时才真正释放内存。但并发下的引用计数面临"先有鸡还是先有蛋"的问题：

/* 问题：在拿到引用之前，对象可能已被释放！ */ obj = find_object(key); /* 拿到指针 */ /* ← 此时另一线程可能把 refcount 减到 0 并释放 obj */ refcount_inc(&obj->ref); /* 已经是悬空指针！ */

解决方案：加锁、RCU 保护、或使用 hazard pointer 来保证"拿到指针"和"增加引用"之间的原子性。

🔢
9.2 顺序锁（Sequence Locks / Seqlock）

Seqlock 是一种允许读操作不阻塞写操作的机制：写者递增序列号，读者在读前后各读一次序列号，如果不一致则重试。

unsigned seq; /* 写者 */ write_seqlock(&sl); /* seq 变为奇数 */ update_data(); write_sequnlock(&sl); /* seq 变回偶数 */ /* 读者 */ do { seq = read_seqbegin(&sl); read_data(); } while (read_seqretry(&sl, seq)); /* 奇数或变化则重试 */

适用场景：读多写少，且读操作不能修改共享数据，读操作可以廉价地重试。

⭐
9.3 读-复制-更新（RCU）

RCU 是本书最重要的技术之一，也是 McKenney 最著名的贡献（Linux 内核 RCU 的主要实现者）。其核心思想：

  • 读者完全无锁无等待（read-lock-free），只需简单地 rcu_read_lock()/rcu_read_unlock() 声明读临界区
  • 更新者创建新版本，原子地替换指针（publish-subscribe 模式）
  • 旧版本延迟回收：等所有在旧指针可见时就已经进入读临界区的读者退出后，才释放旧数据
/* RCU 更新流程 */ new_p = kmalloc(...); /* 创建新版本 */ *new_p = *old_p; /* 复制并修改 */ new_p->value = new_value; rcu_assign_pointer(gp, new_p); /* 原子发布 */ synchronize_rcu(); /* 等待宽限期（grace period）*/ kfree(old_p); /* 安全释放旧版本 */ /* RCU 读者 */ rcu_read_lock(); p = rcu_dereference(gp); /* 读取指针（加内存屏障）*/ use(p->value); rcu_read_unlock();
📊 RCU 宽限期（Grace Period）工作原理
🔬 第一性原理解读
延迟处理的第一性原理来自时间与因果关系：如果一个操作的后果在未来某个"安全点"之后才生效，那么在安全点到来之前，所有观察者看到的都是一致的旧状态。RCU 是这个原理的完美实现：读者活在"过去"的快照中，写者在"未来"建造新版本，只有确认没有读者停留在"过去"后，才销毁它。这正是 MVCC（多版本并发控制）的核心思想，也是 Git 分支模型、Copy-on-Write 文件系统的理论基础。
📖 第 10 章

数据结构
Data Structures

将前面所有技术综合运用于实际并行数据结构的设计：哈希表、链表与树

🔗 官方原文：kernel.org/perfbook
🗃️
10.1-10.2 可分区数据结构：并行哈希表

哈希表是最典型的可分区数据结构。设计步骤：

  1. 将哈希桶分组，每组有一把锁 → 不同桶的操作完全并行
  2. 进一步：读操作用 RCU 保护，写操作用锁保护 → 读者完全无锁
  3. 动态调整桶数量（resizing）时需要特别处理：使用两层哈希或"split-phase" resizing
/* 按 key 哈希确定使用哪把锁，不同桶完全并行 */ int bucket = hash(key) % NUM_BUCKETS; spin_lock(&locks[bucket]); /* 操作 bucket 中的链表 */ spin_unlock(&locks[bucket]);
📚
10.3-10.4 读多写少与非分区数据结构

读多写少数据结构（如路由表、配置表）的最佳实践：使用 RCU 包裹整个数据结构，读者无锁快速访问，写者创建新版本后原子替换。

非分区数据结构（如优先队列、并发跳表）更复杂，因为操作可能跨越多个节点，需要细粒度锁或无锁算法，而无锁算法的正确性证明极其困难。

⚡
10.6 微优化（Micro-Optimization）

在数据结构层面，缓存友好性（Cache Friendliness）是重要的性能因素：

  • False Sharing：两个不相关的变量恰好在同一缓存行，互相使对方的缓存失效
  • Cache Line Padding：在不同 CPU 的私有数据之间填充，避免 False Sharing
  • 热冷数据分离：频繁访问的字段（热）和很少访问的字段（冷）分开存储
/* 避免 False Sharing：为每个 CPU 的计数器对齐到缓存行 */ struct { long count; char pad[64 - sizeof(long)]; /* 填充至 64 字节缓存行 */ } __attribute__((aligned(64))) per_cpu_counter[MAX_CPUS];
🔬 第一性原理解读
并行数据结构设计的第一性原理是：数据结构的形态决定了并行的边界。哈希表天然可分区（不同的键互不干扰），而一个全局有序链表的头节点天然是竞争热点。好的并行数据结构设计，本质上是寻找数据之间的独立性边界（independence boundary），在边界内允许并行，跨越边界时才需要协调。微优化（缓存行对齐等）则是将软件结构与硬件物理结构对齐，让数据的"归属"不仅在逻辑上，也在物理上（缓存行级别）得到保证。
📖 第 11 章

验证
Validation

并行程序的正确性验证：测试、追踪、断言与概率性 Bug 的应对

🔗 官方原文：kernel.org/perfbook
并行验证的核心难点：并发 Bug 往往是概率性的（Heisenbug），在测试环境中可能从不出现，在生产环境中却频繁触发。传统的顺序调试方法对并行程序几乎无效。
🔍
11.2-11.3 追踪与断言

追踪（Tracing）：通过在关键路径插入日志，事后重建执行时序。但需要注意：添加追踪代码本身会改变执行时序（观察者效应），有可能让 Bug 消失。

断言（Assertions）：在代码中嵌入不变式检查，是最廉价的验证手段：

/* WARN_ON 在 Linux 内核中无处不在 */ WARN_ON(refcount_read(&obj->ref) == 0); BUG_ON(ptr == NULL && size > 0);
🔬
11.4-11.5 静态分析与代码审查

工具辅助的静态分析对并行程序尤为重要：

  • Sparse：Linux 内核专用的静态分析器，可检查 RCU 使用规范、锁注解等
  • ThreadSanitizer (TSan)：动态 race condition 检测器，在测试时插桩运行
  • Lockdep：Linux 内核内建的死锁检测器，维护完整的锁依赖图
  • 代码审查：对于底层并发代码，同行评审（peer review）是不可缺少的安全网
🎲
11.6 概率与 Heisenbugs

并发 Bug 的出现依赖于特定的线程调度时序。McKenney 介绍了几种提高 Bug 触发概率的技术：

  • 压力测试（Stress Testing）：长时间、高并发地运行，让罕见时序有机会出现
  • 随机调度（Random Scheduling）：在关键点随机插入 sleep，人工制造调度切换
  • 变种测试（Mutation Testing）：故意注入错误，验证测试是否能发现它们
Heisenbug 定律：观察本身会改变并发程序的行为。添加 printf/日志、使用调试器单步执行，都会彻底改变线程调度时序，使 Bug 消失。这使得并发调试极其困难。
🔬 第一性原理解读
并行程序验证的根本困难来自状态空间爆炸：N 个线程各有 M 个操作，可能的交错顺序是 (N×M)! / (M!)^N，这个数字随线程和操作数量指数级增长，任何有限时间的测试都无法穷举。因此，对并行程序的信心来源不能主要依靠测试，而要依靠：设计时的不变式（invariant）推理、形式化证明、以及工具辅助的系统模型检测（Model Checking）。本章的核心教训：正确的并行程序需要在设计阶段就建立并维护清晰的不变式，验证是对设计正确性的检验，而不是发现设计缺陷的主要手段。
📖 第 12 章

形式化验证
Formal Verification

🔗 官方原文：kernel.org/perfbook
🔭
12.1-12.4 Promela 与 Spin 模型检查器

Spin 是一个开源的模型检查工具，使用 Promela 语言描述并发系统，然后穷举所有可能的执行路径来验证属性。

/* Promela 语言示例：验证两线程互斥 */ bool lock = false; int count = 0; proctype thread() { atomic { /* 原子地取锁 */ lock == false -> lock = true; } count++; /* 临界区 */ lock = false; } /* Spin 会验证：count 是否可能 > 1 */

Spin 通过状态空间搜索（深度优先）穷举所有可能的线程交错，如果发现违反属性的路径则报告反例。这是对"测试无法穷举"的根本性解决方案。

📖
12.5-12.7 锁、QRCU 与 dynticks 的形式化验证

McKenney 用实际案例展示如何对复杂的并发协议进行验证：

  • 锁协议：验证自旋锁实现的互斥性和无死锁性
  • QRCU：验证 quiescent-state-based RCU 的内存安全性
  • dynticks 与抢占式 RCU：内核中最复杂的并发状态机之一，形式化验证帮助发现了实际的 Bug
真实案例：McKenney 描述了用 Spin 对内核 dynticks 代码进行验证的过程，最终发现了一个在代码审查中被所有人忽略的微妙竞争条件。形式化验证不只是理论工具。
⚖️
12.8 简洁性本身就是一种验证

McKenney 的重要观点：复杂的代码很难验证，简洁的代码几乎是自证明的。设计时追求简洁，可以降低验证负担。如果一段代码需要形式化验证才能确保正确，这往往是它太复杂的信号，应该重新考虑设计。

🔬 第一性原理解读
形式化验证的第一性原理是：程序是数学对象，其行为可以被严格推理。并发程序的正确性问题可以被转化为可达性问题（某个"坏"状态是否可以被达到）或时序逻辑公式的满足问题（某个属性是否在所有执行路径上成立）。模型检查是对这类问题的算法求解。其代价是状态空间爆炸，因此现实中的形式化验证必须在抽象（模型简化）和精确性之间折衷。关键洞见：简单的设计不仅工程上更安全，在验证理论上也更易处理——这两者是同一枚硬币的两面。

13.2 RCU 救援（RCU Rescues）

高级同步
Advanced Synchronization

🔗 官方原文：kernel.org/perfbook
🚫
14.1 避免锁（Avoiding Locks）

在某些极端场景下（NMI、实时调度、极致性能要求），连锁都不能使用。McKenney 介绍了几种替代手段：

  • 数据所有权：每个线程只访问自己的数据，根本不需要锁
  • RCU：读者完全无锁，写者只需与其他写者互斥
  • 原子操作：针对简单操作（计数、标志位）的无锁实现
  • 无锁数据结构：Michael-Scott 并发队列等算法保证 lock-free 或 wait-free 进展
🧲
14.2 内存屏障（Memory Barriers）

这是本书最难的内容之一。现代 CPU 和编译器会对内存操作进行重排序以提升性能，内存屏障是强制排序的硬件指令。

/* 没有内存屏障，Reader 可能先看到 data 后看到 flag */ /* 因为 CPU 可以乱序执行写操作 */ /* Writer (CPU 0): */ data = 42; smp_wmb(); /* 写屏障：确保 data 先于 flag 对其他核可见 */ flag = 1; /* Reader (CPU 1): */ if (flag == 1) { smp_rmb(); /* 读屏障：确保 flag 先于 data 被读 */ use(data); /* 现在保证看到 data=42 */ }

内存模型定义了不同 CPU 架构（x86/ARM/PowerPC/RISC-V）对内存操作可见性的保证。x86 是"强内存序"（TSO），ARM 和 PowerPC 是"弱内存序"，需要更多显式屏障。

📊 各 CPU 架构内存模型强度对比
🔓
14.3 非阻塞同步（Non-Blocking Synchronization）

非阻塞算法按进展保证分为三个层级：

  • Lock-Free：系统整体总在推进（某个线程总能完成操作），但个别线程可能被饿死
  • Wait-Free：每个线程在有限步内完成操作，最强保证，但通常性能不如 Lock-Free
  • Obstruction-Free：最弱，只保证没有其他线程干扰时能完成操作
无锁 ≠ 快：McKenney 明确指出，无锁算法并不总比有锁算法快。无锁算法消除了锁竞争，但引入了 CAS 循环、内存屏障和 ABA 问题等新挑战。在低竞争时，简单的有锁算法往往更快。
🔬 第一性原理解读
内存屏障的存在根源于性能与正确性的根本矛盾：硬件为了性能引入了乱序执行和多级缓存，但这破坏了程序员的"顺序一致性"直觉。内存屏障是程序员向硬件"喊话"的机制，代价是禁止某些乱序优化。非阻塞同步的意义不仅在于性能，更在于避免优先级反转（低优先级线程持锁而高优先级线程被阻塞）和提高容错性（线程崩溃不会导致全系统死锁）。这是实时系统和高可用系统的基本要求。
📖 第 15 章

并行实时计算
Parallel Real-Time Computing

在硬实时约束下进行并行编程：延迟确定性与抢占安全

🔗 官方原文：kernel.org/perfbook
实时 vs 高性能：实时计算的目标是延迟的确定性和上界，而不是平均性能最优。一个偶尔慢 100ms 的系统在实时约束下是不可接受的，即使平均延迟极低。
⏰
实时约束与并行编程的矛盾

实时系统面临独特的挑战：

  • 优先级反转：高优先级实时线程因等待低优先级线程持有的锁而被延迟 → 需要优先级继承（Priority Inheritance）
  • 内存分配：堆内存分配器本身包含锁，最坏情况延迟不可预测 → 实时系统需要实时内存分配器
  • 垃圾回收：GC 停顿是实时系统的噩梦，传统 GC 与实时约束天然对立
  • OS 调度：Linux PREEMPT_RT 补丁使内核几乎全程可抢占，为实时并行奠定基础
🛡️
RCU 在实时系统中的作用

RCU 天然适合实时场景，因为读者路径不持锁、不阻塞，延迟几乎是恒定的。但经典 RCU 的宽限期机制依赖线程调度，在实时系统中需要特殊处理：

  • 可睡眠 RCU（SRCU）：允许读者在临界区内睡眠，适合实时上下文
  • Tasks RCU：针对实时任务的 RCU 变种，宽限期不依赖上下文切换
🔬 第一性原理解读
实时计算的第一性原理是：系统的价值不仅取决于它做什么，还取决于它何时完成。一个迟到的正确答案在实时系统中可能等价于错误答案（安全气囊迟到 100ms 即失效）。因此，实时并行系统的设计约束比普通并行系统更严格：不仅要保证正确性和性能，还要保证时间上界（Temporal Guarantee）。这要求避免一切可能导致无界延迟的操作——包括传统锁（优先级反转）、动态内存分配（分配器延迟不定）和传统 GC（停顿不可预测）。

Ease of Use

15.1-15.2 什么是"易用"？Rusty 量表

McKenney 引用 Rusty Russell 的 API 设计量表来评估接口的易用性。这个量表从最好到最差：

  1. 不可能错误使用（编译器直接阻止错误）
  2. 错误使用会在编译期报错
  3. 错误使用会在运行期立即 crash（fail-fast）
  4. 正确使用比错误使用更简单
  5. 错误使用要多读文档才会犯
  6. ……（更差的层级）
  7. 任何使用方式都会搞坏整个系统
设计目标：好的并行 API 应当让"正确的用法"比"错误的用法"更自然、更简单。让错误变得困难，让正确变得容易。
🌀
15.3 剃须 Mandelbrot 集（Shaving the Mandelbrot Set）

这是本章最精彩的比喻。McKenney 将并行 API 的设计空间比作 Mandelbrot 集：边界极其复杂，微小的参数变化可能带来截然不同的结果。"剃须 Mandelbrot 集"意指：通过对 API 的精心约束，将这个复杂的边界"打磨"成更规则、更可预测的形状。

实践案例：

  • POSIX 线程：强大但易错（手动管理锁、死锁风险高）
  • OpenMP：通过编译指令让并行化更简单，但失去了细粒度控制
  • Rust 所有权系统：在编译期静态排除 data race，是目前最接近"不可能错误使用"的方案
🔬 第一性原理解读
易用性的第一性原理是：代码是人类与机器之间的接口，这个接口的质量由人类能否正确使用它来决定。并行编程困难的根本原因之一，是现有工具的抽象层次与人类认知模型不匹配——锁是机器级别的概念，不是人类思维的自然单元。提升易用性意味着设计更高层的抽象，将并发语义编码到类型系统或编译器中，把运行时错误转化为编译时错误。这是编程语言和运行时系统未来发展的核心方向。

16.1 CPU 技术的未来

Moore 定律进入"后时钟频率时代"。未来的 CPU 发展方向：

  • 更多核心：核心数量持续增长，但单核性能增速放缓
  • 异构计算：CPU + GPU + 专用加速器的混合系统
  • 3D 堆叠内存：HBM 等技术缩短内存访问延迟，改变软件设计权衡
  • 光子计算与量子计算：长远展望，现有编程模型可能根本性变革
💳
16.2-16.3 事务性内存（TM / HTM）

事务性内存（TM）是一个被寄予厚望的并行编程简化方案：用类似数据库事务的语义替代锁。

/* 事务性内存的理想愿景 */ __transaction_atomic { /* 整个块原子执行，冲突时自动重试 */ if (account[A] >= amount) { account[A] -= amount; account[B] += amount; } }

McKenney 对 TM 持批判态度，指出其已知限制：

  • I/O 操作无法回滚（事务中不能有副作用）
  • 硬件 HTM 的容量限制（Intel TSX 已被废弃）
  • 冲突检测和重试在高竞争下性能崩溃
  • TM 与现有系统软件（OS、锁）的互操作问题
λ
16.4 函数式编程与并行

纯函数式编程（Haskell、Erlang、Clojure 等）通过不可变数据和无副作用函数，从根本上消除了 data race 的可能性：

  • 优势：不可变数据无需同步，天然并行安全；引用透明性使推理变简单
  • 限制：性能开销（大量内存分配、GC 压力）；某些系统编程问题天然需要可变状态；与现有 C 系统生态的集成成本
McKenney 的结论：没有单一技术能解决所有并行编程问题。真正的进步来自多种技术的组合：更好的类型系统（Rust）、更高层的抽象（函数式编程）、更好的工具（形式化验证）、以及不断积累的工程智慧。

🔬 第一性原理解读
本章的终极第一性原理是：并行编程没有银弹（No Silver Bullet）。每种技术都是在特定的约束集合（硬件特性、应用场景、工程团队能力）下的最优解。真正的进步不来自某种"终极技术"的出现，而来自工具生态的整体成熟：更强的类型系统让错误难以犯、更好的形式化验证工具让正确性可以被机械证明、更高层的抽象让程序员无需面对底层细节。McKenney 这本书的最终贡献，不是提供了一种解法，而是建立了一套思考并行问题的框架——这个框架的生命力会超越任何具体技术。
