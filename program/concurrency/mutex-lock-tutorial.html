CHAPTER 01

互斥锁是什么

互斥锁（Mutex，Mutual Exclusion Lock）是多线程编程中用于保护临界区（Critical Section）的基本同步原语。从第一性原理出发，它本质上是对以下问题的回答：当多个执行单元需要访问同一共享资源时，如何保证同一时刻只有一个执行单元在访问？

📖 权威定义来源
根据 Kent 州立大学 POSIX 多线程编程教程（基于 Guy Kerens 原版），互斥锁保证以下三件事：
1. 原子性（Atomicity）— 加锁操作是原子的；
2. 单一性（Singularity）— 同一时刻只有一个线程持有锁；
3. 非忙等待（Non-Busy Wait）— 等待线程会被挂起，不消耗 CPU。

https://www.cs.kent.edu/~ruttan/sysprog/lectures/multi-thread/multi-thread.html

原子性：为什么普通变量不够

假设两个线程都执行 count++，这在 CPU 层面不是一条指令，而是三步：

; count++ 的真实 CPU 指令序列
LOAD   R1, [count]    ; 第1步：从内存读取 count 到寄存器
ADD    R1, R1, #1     ; 第2步：在寄存器中加 1
STORE  [count], R1   ; 第3步：写回内存

如果线程 A 在第 1 步之后、第 3 步之前被调度出去，线程 B 读到的是旧值，则两次 count++ 最终只加了 1。这就是竞态条件（Race Condition）。

图 1-1   竞态条件：count++ 的危险交错

三大核心属性详解

属性含义实现手段
原子性 加锁操作在其他线程看来是瞬间完成的 硬件原语 CAS / TAS 指令
单一性 任一时刻至多一个线程持有锁 内核等待队列 + 状态变量
非忙等待 等待线程不消耗 CPU，进入睡眠 futex / TASK_INTERRUPTIBLE
CHAPTER 02

第一性原理：互斥锁的实现

要从第一性原理理解互斥锁，必须理解两层：硬件原语（提供原子操作）和内核机制（提供睡眠/唤醒）。

第一层：硬件原语 — CAS 与 TAS

📖 来源：NPTL / attractivechaos 技术博客
在 NPTL（Native POSIX Thread Library）中，mutex 构建在 futex 之上，而 futex 的用户态部分使用 CAS（Compare-And-Swap）原子指令实现无锁快速路径。

https://attractivechaos.wordpress.com/2011/10/06/multi-threaded-programming-efficiency-of-locking/

CPU 提供的 CAS（Compare-And-Swap）指令：若内存地址的当前值等于期望值，则原子地将其替换为新值，并返回成功；否则失败。这是构建所有锁的基础原子操作。

// 用户态自旋锁的最简实现（spinlock）—— 来源：attractivechaos
volatile int lock = 0;

void *worker(void*) {
    // __sync_lock_test_and_set 原子地设为1并返回旧值
    while (__sync_lock_test_and_set(&lock, 1));  // 忙等待 = 自旋
    // 临界区
    __sync_lock_release(&lock);  // 原子地设为0
}
⚠️ 自旋锁 vs 互斥锁的本质区别
自旋锁（spinlock）：持续循环检测锁状态，浪费 CPU 周期，但延迟极低。
互斥锁（mutex）：等待失败时将线程放入睡眠，不消耗 CPU，但有系统调用开销。
互斥锁 = 自旋（极短重试）+ 系统调用睡眠（竞争激烈时）

第二层：futex — 睡眠与唤醒的内核原语

📖 来源：Linux kernel 官方源码 + futex internals
futex（Fast Userspace muTEX）是 Linux 内核提供的一种混合锁原语：无竞争时完全在用户态完成（无系统调用）；有竞争时通过系统调用进入内核，将线程放入等待队列睡眠。

https://github.com/torvalds/linux/blob/master/kernel/futex/waitwake.c https://kernel-internals.org/locking/futex/ 
// futex 的本质：一个 32 位整数 + 内核等待队列
// 三种核心状态（来自 futex 实现分析）：
0 = 未锁定（unlocked）
1 = 已锁定，无等待者（locked, no waiters）
2 = 已锁定，有等待者（locked, waiters exist）

// FUTEX_WAIT 系统调用 — 内核核心逻辑（简化版）
int futex_wait(u32 *uaddr, u32 val) {
    // 1. 计算 futex key（用虚拟/物理地址哈希定位等待桶）
    get_futex_key(uaddr, &q.key);

    // 2. 锁住等待桶（hash bucket）
    spin_lock(&hb->lock);

    // 3. 关键：在桶锁保护下重新检查 futex 值！
    //    防止"丢失唤醒"竞态：若值已变则直接返回
    get_user(uval, uaddr);
    if (uval != val) { spin_unlock(&hb->lock); return -EAGAIN; }

    // 4. 将当前线程加入等待队列
    plist_add(&q.list, &hb->chain);
    spin_unlock(&hb->lock);

    // 5. 设置线程状态为可中断睡眠，让出 CPU
    set_current_state(TASK_INTERRUPTIBLE);
    schedule();  // 调度器选择下一个线程运行

    // 6. 被唤醒后在此处恢复执行
    __set_current_state(TASK_RUNNING);
}

单锁场景：睡眠与唤醒时序

以两个线程 Thread A 和 Thread B 竞争同一互斥锁为例，完整分析睡眠/唤醒的时序过程。

📖 权威来源
"If a thread attempts to lock a thread that was locked by a second thread, the first thread will be suspended (and will not consume any CPU resources) until the lock is freed by the second thread. At this time, the first thread will wake up and continue execution, having the mutex locked by it."

Multi-Threaded Programming With POSIX Threads — cs.kent.edu

图 3-1   两线程单锁竞争完整时序（含内核状态）

睡眠唤醒的六步机制详解

  • ①
    Thread A 加锁成功：CAS 原子地将 mutex.state 从 0 改为 1，纯用户态操作，无系统调用，进入临界区。
  • ②
    Thread B 竞争失败：B 调用 CAS 尝试将 state 从 0 改为 1，但发现 state=1，CAS 失败。B 将 state 置为 2（表示"有等待者"），然后调用 FUTEX_WAIT(addr, 2) 系统调用。
  • ③
    内核将 B 加入等待队列：内核锁住 hash bucket，检查 state 仍为 2（防止丢失唤醒），将 B 的 task_struct 放入等待链表，解锁 bucket，调用 schedule() 让出 CPU，B 的状态变为 TASK_INTERRUPTIBLE。
  • ④
    Thread A 释放锁：A 调用 pthread_mutex_unlock()，原子地将 state 设为 0。发现 state 之前为 2（有等待者），则调用 FUTEX_WAKE(addr, 1) 系统调用。
  • ⑤
    内核唤醒 B：内核找到等待队列中的 B，将其 task_struct 状态置为 TASK_RUNNING，加入调度器的运行队列（run queue）。B 不一定立即运行，等待调度器选中。
  • ⑥
    Thread B 获锁：B 被调度执行，从 schedule() 返回，重新尝试 CAS 获取锁（成功），进入临界区。

线程状态机

图 3-2   线程在互斥锁操作中的状态转换
📌 关键：防止"丢失唤醒"（Lost Wakeup）竞态
内核在 FUTEX_WAIT 中，必须在持有 hash bucket 锁的情况下重新验证 futex 值：若值已变（说明 unlock 已发生），则直接返回不睡眠。这防止了以下竞态：线程 B 读到锁被占，但在调用 FUTEX_WAIT 之前，A 已经 unlock 并 FUTEX_WAKE——若无此检查，B 会永久睡眠。

https://kernel-internals.org/locking/futex/

双锁场景：死锁的本质

当多个线程需要同时持有两个互斥锁时，若加锁顺序不一致，就可能产生死锁（Deadlock）——所有线程都在等待对方释放锁，永久阻塞。

📖 来源：SEI CERT C 编码规范 POS51-C
"Sometimes, when locking mutexes, multiple threads hold each other's lock, and the program consequently deadlocks. There are four requirements for deadlock: Mutual exclusion, Hold and wait, No preemption, Circular wait."

https://wiki.sei.cmu.edu/confluence/display/c/POS51-C.+Avoid+deadlock+with+POSIX+threads+by+locking+in+predefined+order

典型死锁代码：银行转账场景

这是 SEI CERT C POS51-C 中给出的经典例子：两个线程互相转账，各自先锁自己账户，再锁对方账户，形成死锁。

// ❌ 不合规代码 — 来源：SEI CERT C POS51-C
void *deposit(void *ptr) {
    deposit_thr_args *args = ptr;

    // Thread 1: 先锁 ba1，再锁 ba2
    pthread_mutex_lock(&args->from->balance_mutex);  // 锁A
    pthread_mutex_lock(&args->to->balance_mutex);    // 等待锁B ← 可能死锁！

    args->from->balance -= args->amount;
    args->to->balance   += args->amount;

    pthread_mutex_unlock(&args->from->balance_mutex);
    pthread_mutex_unlock(&args->to->balance_mutex);
}

// Thread 1: deposit(ba1 → ba2)  锁顺序: A → B
// Thread 2: deposit(ba2 → ba1)  锁顺序: B → A  ← 逆序！必然死锁！

死锁预防：打破循环等待

📖 来源：SEI CERT C POS51-C 合规方案
"The solution to the deadlock problem is to use a predefined order for the locks in the deposit() function. Each thread will lock on the basis of the bank_account ID. This solution prevents the circular wait problem."

https://wiki.sei.cmu.edu/confluence/display/c/POS51-C.+Avoid+deadlock+with+POSIX+threads+by+locking+in+predefined+order
策略方法说明
固定加锁顺序 按资源 ID 排序后加锁 打破循环等待条件，最常用
trylock + 回退 pthread_mutex_trylock() 失败则释放已有锁 破坏"持有并等待"条件
超时锁 pthread_mutex_timedlock() 超时强制退出，避免永久阻塞
减少锁粒度 合并成一把大锁 牺牲并发度换安全性

双锁正确时序：有序加锁

采用固定顺序加锁（按账户 ID 升序），两个线程不再形成循环等待，死锁被彻底消除。

有序加锁睡眠唤醒时序

图 5-1   双锁有序加锁正确时序（Thread 1 先，Thread 2 后竞争）

实战代码：固定顺序加锁

// ✅ 合规代码 — 来源：SEI CERT C POS51-C
typedef struct {
    int balance;
    pthread_mutex_t balance_mutex;
    unsigned int id;  /* 不可变的唯一 ID */
} bank_account;

void *deposit(void *ptr) {
    deposit_thr_args *args = ptr;

    /* 关键：始终按 ID 升序加锁，打破循环等待 */
    if (args->from->id < args->to->id) {
        pthread_mutex_lock(&args->from->balance_mutex);  // 先锁 ID 小的
        pthread_mutex_lock(&args->to->balance_mutex);    // 再锁 ID 大的
    } else {
        pthread_mutex_lock(&args->to->balance_mutex);    // 先锁 ID 小的
        pthread_mutex_lock(&args->from->balance_mutex);  // 再锁 ID 大的
    }

    args->from->balance -= args->amount;
    args->to->balance   += args->amount;

    /* 解锁顺序任意，通常逆序 */
    pthread_mutex_unlock(&args->from->balance_mutex);
    pthread_mutex_unlock(&args->to->balance_mutex);
    free(ptr);
    return NULL;
}

互斥锁 vs 自旋锁：睡眠还是忙等

📖 权威来源：NPTL 实现分析
"The fundamental difference between spinlock and mutex is that spinlock keeps checking the lock (busy waiting), while mutex puts threads waiting for the lock into sleep (blocked). A busy-waiting thread wastes CPU cycles, while a blocked thread does not."

https://attractivechaos.wordpress.com/2011/10/06/multi-threaded-programming-efficiency-of-locking/
图 6-1   互斥锁 vs 自旋锁行为对比
维度自旋锁 (Spinlock)互斥锁 (Mutex)
等待方式忙等（循环检测）睡眠（进内核等待队列）
CPU 消耗等待期间 100% CPU等待期间 0% CPU
系统调用无（纯用户态）竞争时需要（futex）
唤醒延迟几乎无延迟需要调度器介入，有延迟
适用场景锁持有 < 几十 ns，单核禁用锁持有时间不确定或较长
