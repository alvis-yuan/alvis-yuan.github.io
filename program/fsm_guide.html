什么是有限状态机（FSM）

Finite State Machine · 基础概念
🧭

本章导读：建立对状态机的直觉认识。我们先从数学定义出发，明确"状态""事件""转移""动作"四要素，再用日常生活类比帮助建立直觉，最后通过 SVG 图解呈现 FSM 的结构骨架。

有限状态机（Finite State Machine，FSM）是计算机科学中的经典设计模型，被广泛用于嵌入式系统、通信协议、编译器词法分析等领域。它是一种行为型设计模式，将复杂的条件控制逻辑分解为一组明确定义的状态和状态之间的迁移规则。

来源：《State Machine Design in C》originally published at CodeProject，基于 Dr. Dobb's C/C++ Users Journal 原文。
"A common design technique in the repertoire of most programmers is the venerable finite state machine (FSM). Designers use this programming construct to break complex problems into manageable states and state transitions."

🔑 FSM 的四要素

要素 英文 说明 C语言表达
状态 State 系统在某一时刻所处的稳定模式，有限且互斥，同一时刻只能处于一个状态 typedef enum { ... } State;
事件 Event 驱动状态迁移的外部或内部刺激，如按键、超时、中断、消息到达等 typedef enum { ... } Event;
转移 Transition 当在状态 S 下收到事件 E 时，系统跳转到下一个状态 S' 的规则 转移表 / switch-case
动作 Action 伴随转移发生的操作，分为进入动作(Entry)、退出动作(Exit)、转移动作(Action) 函数指针回调

核心设计原理

内部事件 · 外部事件 · 转移逻辑 · 并发安全
⚙️

本章导读：深入理解状态机的运行机制。重点区分内部事件与外部事件，理解"运行至完成"（Run-to-Completion）语义，以及如何通过事务锁和条件队列处理并发场景下的状态竞争问题。

2.1 外部事件与内部事件

根据 GitHub 上 endurodave/C_StateMachine 的权威实现，事件分为两类：

类型 触发者 访问性 特点
外部事件 系统外部（任意线程/任务） 公开接口 需要验证状态转移合法性；进入时需加锁保护
内部事件 状态函数内部自生成 私有 无需验证，在当前外部事件的控制流内同步执行

图2：状态机事件处理流程 — Run-to-Completion 执行模型

2.2 并发与可重入性

AllPCB 文章指出，在嵌入式多任务系统中，若不加保护，并发的状态迁移请求会导致状态不一致：

竞态场景："task1 performs (s1, c1, a1 → s2) and is preempted during a1; task2 then sees the still-stale state s1 and processes c2, which may lead to the trap state instead of the intended sequence."
解决方案：引入 inTransaction 标志位 + 条件队列（ConditionQueue）。

图3：并发竞态问题与解决方案（事务标志 + 条件队列）

2.3 Guard / Entry / Exit 动作

扩展的 FSM 支持四类回调，来自 endurodave/C_StateMachine 的实现说明：

动作类型触发时机用途C函数签名
Guard（守卫）状态函数执行前条件检查，返回 FALSE 则取消转移BOOL (*GuardFunc)(...)
Entry（进入）进入新状态时初始化、资源分配void (*EntryFunc)(...)
Exit（退出）离开当前状态时清理、资源释放void (*ExitFunc)(...)
Action（动作）状态函数主体核心业务处理void (*StateFunc)(...)

Guard → Exit → Entry → Action 的执行顺序确保了状态生命周期管理的完整性。

C语言实现方法

switch-case · 查找表 · 函数指针结构体
🛠️

本章导读：对比三种主流 C 语言 FSM 实现方式的结构、优缺点和适用场景。通过代码骨架理解每种方式的数据组织形式，选择最适合项目约束的方案。

方法一：switch-case 条件分支

最简单直接的实现方式。来自 Aticleworld 和 endurodave 的权威文档都提到这是入门级实现，但随着状态增多会产生"状态爆炸"问题。

C · switch-case 实现 
#include <stdio.h>

// 状态枚举 — 来自 aticleworld.com 的 ATM 示例
typedef enum {
    Idle_State,
    Card_Inserted_State,
    Pin_Entered_State,
    Option_Selected_State,
    Amount_Entered_State
} eSystemState;

typedef enum {
    Card_Insert_Event,
    Pin_Enter_Event,
    Option_Selection_Event,
    Amount_Enter_Event,
    Amount_Dispatch_Event
} eSystemEvent;

int main() {
    eSystemState eNextState = Idle_State;

    while(1) {
        eSystemEvent eNewEvent = ReadEvent();

        switch(eNextState) {
        case Idle_State:
            if(Card_Insert_Event == eNewEvent)
                eNextState = InsertCardHandler();
            break;
        case Card_Inserted_State:
            if(Pin_Enter_Event == eNewEvent)
                eNextState = EnterPinHandler();
            break;
        /* ... 其他状态 ... */
        default: break;
        }
    }
}
缺点：endurodave 指出："There is no way to enforce the state transition rules. Any transition is allowed at any time." 且无法在多线程环境下安全使用。

方法二：二维函数指针查找表（推荐）

Aticleworld 称之为"最简单的查找表实现方式，未来增加状态或事件时无需大幅修改代码"。行为为当前状态，列为当前事件，表中存储对应的处理函数指针。

C · 2D 函数指针表实现 — 来源：aticleworld.com 
// 函数指针类型定义
typedef eSystemState (*pfEventHandler)(void);

// 二维查找表 [状态][事件] = 处理函数
typedef eSystemState (*const afEventHandler[last_State][last_Event])(void);

static afEventHandler StateMachine = {
    [Idle_State]           = {[Card_Insert_Event]      = InsertCardHandler  },
    [Card_Inserted_State]  = {[Pin_Enter_Event]         = EnterPinHandler    },
    [Pin_Eentered_State]   = {[Option_Selection_Event]  = OptionSelectionHandler},
    [Option_Selected_State]= {[Amount_Enter_Event]      = EnterAmountHandler },
    [Amount_Entered_State] = {[Amount_Dispatch_Event]   = AmountDispatchHandler},
};

// 主循环
while(1) {
    eSystemEvent eNewEvent = ReadEvent();
    // 边界检查 + NULL 检查
    if((eNextState < last_State) &&
       (eNewEvent  < last_Event) &&
       StateMachine[eNextState][eNewEvent] != NULL)
    {
        eNextState = (*StateMachine[eNextState][eNewEvent])();
    }
}
注意：Aticleworld 提醒："here table is sparse, if the states and events are increasing, this technique increases the wastage of the memory." 稀疏矩阵会浪费内存，状态/事件数量较多时应评估内存开销。

方法三：结构体数组（事件-状态-处理函数三元组）

Aticleworld 认为这是"一种优雅的方式"，将状态、事件、处理函数封装在结构体中，形成线性表，内存更紧凑。

C · 结构体数组实现 — 来源：aticleworld.com 
// 状态机条目结构体
typedef struct {
    eSystemState   eStateMachine;              // 当前状态
    eSystemEvent   eStateMachineEvent;          // 期望事件
    pfEventHandler pfStateMachineEventHandler;  // 处理函数
} sStateMachine;

// 初始化状态表
sStateMachine asStateMachine[] = {
    {Idle_State,            Card_Insert_Event,      InsertCardHandler      },
    {Card_Inserted_State,   Pin_Enter_Event,        EnterPinHandler        },
    {Pin_Eentered_State,    Option_Selection_Event, OptionSelectionHandler },
    {Option_Selected_State, Amount_Enter_Event,     EnterAmountHandler     },
    {Amount_Entered_State,  Amount_Dispatch_Event,  AmountDispatchHandler  },
};

// 派发逻辑
if((eNextState < last_State) && (eNewEvent < last_Event) &&
   (asStateMachine[eNextState].eStateMachineEvent == eNewEvent) &&
   (asStateMachine[eNextState].pfStateMachineEventHandler != NULL))
{
    eNextState = (*asStateMachine[eNextState].pfStateMachineEventHandler)();
}

电机状态函数与转移表

C · Motor.c 核心片段 — 来源：endurodave/C_StateMachine 
// 电机实例数据
typedef struct { INT currentSpeed; } Motor;
typedef struct { INT speed;        } MotorData;

// ── 状态枚举（顺序必须与 stateMap 一致）──
enum States {
    ST_IDLE, ST_STOP, ST_START, ST_CHANGE_SPEED, ST_MAX_STATES
};

// ── 状态函数定义 ──
STATE_DEFINE(Idle, NoEventData) {
    printf("Motor: ST_Idle\n");
    // 电机静止，无动作
}

STATE_DEFINE(Stop, NoEventData) {
    Motor* m = SM_GetInstance(Motor);
    m->currentSpeed = 0;
    printf("Motor: ST_Stop — 断电\n");
    SM_InternalEvent(ST_IDLE, NULL); // 触发内部事件 → ST_IDLE
}

STATE_DEFINE(Start, MotorData) {
    Motor* m = SM_GetInstance(Motor);
    m->currentSpeed = pEventData->speed;
    printf("Motor: ST_Start — 转速=%d\n", m->currentSpeed);
}

STATE_DEFINE(ChangeSpeed, MotorData) {
    Motor* m = SM_GetInstance(Motor);
    m->currentSpeed = pEventData->speed;
    printf("Motor: ST_ChangeSpeed — 新转速=%d\n", m->currentSpeed);
}

// ── 状态映射表 ──
BEGIN_STATE_MAP(Motor)
    STATE_MAP_ENTRY(ST_Idle)
    STATE_MAP_ENTRY(ST_Stop)
    STATE_MAP_ENTRY(ST_Start)
    STATE_MAP_ENTRY(ST_ChangeSpeed)
END_STATE_MAP

// ── SetSpeed 转移表 ──
EVENT_DEFINE(MTR_SetSpeed, MotorData) {
    BEGIN_TRANSITION_MAP            // 当前状态：
        TRANSITION_MAP_ENTRY(ST_START)        // ST_IDLE → ST_START
        TRANSITION_MAP_ENTRY(CANNOT_HAPPEN)   // ST_STOP → 不允许
        TRANSITION_MAP_ENTRY(ST_CHANGE_SPEED) // ST_START → ST_CHANGE_SPEED
        TRANSITION_MAP_ENTRY(ST_CHANGE_SPEED) // ST_CHANGE_SPEED → 自身
    END_TRANSITION_MAP(Motor, pEventData)
}

// ── Halt 转移表 ──
EVENT_DEFINE(MTR_Halt, NoEventData) {
    BEGIN_TRANSITION_MAP
        TRANSITION_MAP_ENTRY(EVENT_IGNORED)  // ST_IDLE → 忽略
        TRANSITION_MAP_ENTRY(CANNOT_HAPPEN)  // ST_STOP → 不允许
        TRANSITION_MAP_ENTRY(ST_STOP)        // ST_START → ST_STOP
        TRANSITION_MAP_ENTRY(ST_STOP)        // ST_CHANGE_SPEED → ST_STOP
    END_TRANSITION_MAP(Motor, pEventData)
}
设计亮点：转移表中使用 EVENT_IGNORED（静默忽略）和 CANNOT_HAPPEN（软件断言失败）两个特殊值，编译时通过 C_ASSERT() 宏验证转移表条目数与状态数是否匹配，将错误暴露在编译期。

优势与缺点

使用状态机前必须权衡的工程决策
⚖️

本章导读：从工程实践角度客观分析 FSM 的利弊。优势来自权威资料的原文提炼，缺点也以第一手文献为依据，避免对 FSM 过度神化或贬低。帮助读者在真实项目中做出恰当的技术决策。

✅ 优势

  • 可预测性与可靠性：行为明确无歧义，关键系统（医疗、汽车）首选
  • 强制合法转移：非法状态序列被转移表自动拦截，杜绝副作用
  • 调试与维护简单：状态和转移可独立追踪，日志友好
  • 代码组织清晰：状态逻辑和转移逻辑分离，单一职责
  • 无重复代码：查找表风格无 switch 的重复分支
  • 扩展性好：增加新状态只需扩充枚举和表项
  • 多实例支持：同一状态机类型可并行运行多个独立实例
  • 无堆内存依赖：固定块分配器可完全避免 malloc

❌ 缺点

  • 状态爆炸：状态数量指数级增长时，转移表和状态图都变得难以管理
  • 内存稀疏浪费：二维表中大量 NULL 项，状态/事件数量大时浪费可观
  • 编译单元增多：STATE 模式让代码更分散，对小项目是过度设计
  • 并发实现复杂：多线程下需要锁和队列，增加代码复杂度
  • 不适合简单场景：状态少时 if-else 更直接，FSM 反而是负担
  • 学习曲线：宏、函数指针、转移表的组合对初学者不友好

第一性原理总结

回归本质，重新审视状态机存在的根本原因
🔭

本章导读：用第一性原理（First Principles Thinking）的方法论，抛开所有技术术语，从"计算的本质"出发重新推导为何需要状态机，以及状态机在整个软件工程知识体系中的位置。

原理一：计算本质上是"记忆 + 变换"

一切程序都在做两件事：存储信息和按规则变换信息。当变换规则不仅依赖当前输入，还依赖历史时，系统就必须有"记忆"——这个记忆就是"状态"。

FSM 是对"带历史记忆的计算"的最小完备抽象：一个变量（currentState）存储全部历史的精华，一张映射表（transition table）定义未来。

原理二：复杂性来自"隐式状态"

没有显式状态机的系统，其状态分散在全局变量、标志位、嵌套 if-else 的控制流中——这就是隐式状态。隐式状态导致状态读取散落各处，任何地方都可能修改，任何人都无法掌握全局，BUG 随时可能从意外的状态组合中冒出。

FSM 强制将所有状态显式化、集中化，这不是技术层面的选择，而是信息组织方式的根本革命。

原理三：转移表是"规则的具象化"

业务规则的本质是："在什么情况下，允许发生什么，产生什么结果"。转移表（Transition Table）把这句话直接翻译为代码结构——行是"在什么状态"，列是"遇到什么事件"，单元格是"允许/禁止/执行什么"。

这意味着业务规则即代码，代码即文档，实现者和审查者看同一张表，无需翻译。

原理四：状态机的边界——"确定性"的代价

FSM 的威力来自确定性：给定状态+事件，结果唯一。但这也是它的固有局限：当状态空间连续或无限大时（如浮点参数空间、无限长历史），FSM 退化为无穷机，不再实用。

因此第一性原理告诉我们：FSM 适用且仅适用于"行为可离散化"的问题域。若问题需要模糊判断、概率推理或深度历史依赖，应转向其他模型（规则引擎、神经网络、事件溯源等）。

原理五：使用状态机是一种工程承诺

引入 FSM 意味着你向团队做出承诺：系统的所有行为模式已被穷举。这是纪律，不是技术。它要求你在设计阶段完成状态空间的完整分析，而不是在运行时"随机应变"。

这也是为什么嵌入式系统、安全关键系统（医疗设备、航空航天）大量采用 FSM：它将"我保证考虑了所有情况"这句话变成了可验证的结构性证明。

终极推论

一句话总结：状态机不是设计模式，而是对"行为随历史演化的系统"的最简正确建模方式。当你用 if-else 实现状态机时，你只是在用更混乱的形式做同一件事。FSM 唯一的贡献，是把隐藏在代码缝隙中的业务规则，用一张人和机器都能读懂的表格呈现出来。
