概览与调用顺序

libcurl 的使用涉及两个层级的初始化：全局环境初始化（整个进程唯一）和 Easy Handle 初始化（每次传输使用）。两者都有对应的去初始化函数，必须成对调用。

ℹ️
所有内容严格依据 curl.se 官方文档（curl_global_init、curl_easy_init、curl_global_sslset、curl_global_cleanup、curl_easy_cleanup、libcurl-thread 等 man page）整理。
🌍

curl_global_init — 全局环境初始化

INIT函数签名
#include <curl/curl.h>

CURLcode curl_global_init(long flags);

加入 curl 7.8 · 源文档：curl.se/libcurl/c/curl_global_init.html

目的与作用

官方原文："This function sets up the program environment that libcurl needs. Think of it as an extension of the library loader."

该函数设置 libcurl 所需的程序环境，相当于库加载器的延伸。libcurl 使用了大量内部常量（例如 GNU TLS 的协议描述树），这些常量过于复杂无法由库加载器自动完成，必须在运行时调用此函数完成设置。

  • 1
    初始化 SSL/TLS 层：加载 SSL 库所需的全局结构（如 GNU TLS 协议描述树、OpenSSL 全局状态等），避免 TLS 传输出现未定义行为。
  • 2
    初始化 Win32 Winsock（Windows 平台）：若未初始化 Winsock 就执行网络操作，将导致未定义行为。
  • 3
    设置默认内存分配器：将 malloc/free/realloc/strdup/calloc 指针绑定到系统默认实现。
  • 4
    初始化名称解析器、SSH、QUIC 等子系统（视编译选项而定）。
⚠️
官方规定："This function must be called at least once within a program before the program calls any other function in libcurl." 必须在调用任何其他 libcurl 函数之前调用，且整个进程中调用一次即可（多次调用效果等同一次，内部有引用计数）。

flags 标志位

标志常量含义备注
CURL_GLOBAL_ALL 初始化所有已知子系统（推荐） 除 CURL_GLOBAL_ACK_EINTR 外全部开启
CURL_GLOBAL_DEFAULT 等同 CURL_GLOBAL_ALL（当前实现） 同时初始化 SSL + Win32
CURL_GLOBAL_SSL 初始化 SSL 层 自 7.57.0 起此 flag 已无意义，SSL 始终被初始化
CURL_GLOBAL_WIN32 初始化 Win32 Socket 库 不设此位则应用程序自行初始化 Winsock
CURL_GLOBAL_NOTHING 不进行任何额外初始化 极少使用，需深入了解 libcurl 内部
CURL_GLOBAL_ACK_EINTR 连接/等待时应答 EINTR 信号 自 7.69.0 起为默认行为，此 flag 已无意义

返回值

返回 CURLcode，若为非零值则表示初始化失败，后续所有 libcurl 函数均不可使用。

示例代码

int main(void)
{
    CURLcode result;

    /* 在程序启动、仅单线程时调用 */
    result = curl_global_init(CURL_GLOBAL_DEFAULT);
    if(result != CURLE_OK) {
        fprintf(stderr, "全局初始化失败: %s\n", curl_easy_strerror(result));
        return 1;
    }

    /* ... 使用 libcurl ... */

    curl_global_cleanup();   /* 程序退出前调用 */
    return 0;
}

curl_global_init_mem — 带自定义内存分配器的初始化

INIT函数签名
#include <curl/curl.h>

CURLcode curl_global_init_mem(long flags,
                              curl_malloc_callback m,
                              curl_free_callback f,
                              curl_realloc_callback r,
                              curl_strdup_callback s,
                              curl_calloc_callback c);

加入 curl 7.12.0 · 源文档：curl.se/libcurl/c/curl_global_init_mem.html

目的与作用

官方原文："This function works exactly as curl_global_init with one addition: it allows the application to set callbacks to replace the otherwise used internal memory functions."

与 curl_global_init 完全相同，但额外允许应用程序替换 libcurl 内部的内存分配函数，适用于需要自定义内存管理的场景（如嵌入式系统、内存跟踪、内存池等）。

回调参数替换的标准函数回调原型
mmalloc()void *malloc_callback(size_t size)
ffree()void free_callback(void *ptr)
rrealloc()void *realloc_callback(void *ptr, size_t size)
sstrdup()char *strdup_callback(const char *str)
ccalloc()void *calloc_callback(size_t nmemb, size_t size)
⚠️
官方警告："If you are using libcurl from multiple threads or libcurl was built with the threaded resolver option then the callback functions must be thread-safe." 在多线程场景下回调必须是线程安全的。同时，所有回调参数均不得为 NULL。

curl_global_sslset — 选择 SSL 后端

SSL函数签名
#include <curl/curl.h>

CURLsslset curl_global_sslset(curl_sslbackend id,
                               const char *name,
                               const curl_ssl_backend ***avail);

加入 curl 7.56.0 · 源文档：curl.se/libcurl/c/curl_global_sslset.html

目的与作用

官方原文："This function configures at runtime which SSL backend to use with libcurl. This function can only be used to select an SSL backend once, and it must be called before curl_global_init(3)."

当 libcurl 被编译为支持多个 SSL 后端时，此函数用于在运行时选择具体使用哪个 SSL 实现。必须在 curl_global_init() 之前调用。

🚫
SSL 后端只能设置一次。若已设置，再次调用将返回 CURLSSLSET_TOO_LATE。若 curl_global_init() 已被调用，同样返回 CURLSSLSET_TOO_LATE。

支持的 SSL 后端

参数说明

参数类型说明
id curl_sslbackend 通过枚举值指定后端（如 CURLSSLBACKEND_OPENSSL）；若传 CURLSSLBACKEND_NONE 则通过 name 匹配
name const char* 通过名称（大小写不敏感）指定后端；id 和 name 同时指定时，name 被忽略
avail const curl_ssl_backend*** 若非 NULL，函数执行后指向可用后端列表（7.60.0+ 始终被设置）；可用于枚举所有可用后端

返回值

返回值含义
CURLSSLSET_OK后端选择成功
CURLSSLSET_UNKNOWN_BACKEND指定的后端不可用，avail 指向可用列表
CURLSSLSET_TOO_LATE已设置过 SSL 后端或 curl_global_init() 已被调用
CURLSSLSET_NO_BACKENDS此 libcurl 构建不支持任何 SSL 后端

示例代码

/* 必须在 curl_global_init() 之前调用 */
int i;
const curl_ssl_backend **list;

/* 方式1：通过枚举值选择 OpenSSL */
curl_global_sslset(CURLSSLBACKEND_OPENSSL, NULL, NULL);

/* 方式2：通过名称选择 wolfSSL */
curl_global_sslset(CURLSSLBACKEND_NONE, "wolfSSL", NULL);

/* 枚举所有可用的 SSL 后端 */
curl_global_sslset(CURLSSLBACKEND_NONE, NULL, &list);
for(i = 0; list[i]; i++)
    printf("后端 #%d: '%s' (ID: %d)\n", i, list[i]->name, list[i]->id);

curl_easy_init — 创建 Easy Handle

INIT函数签名
#include <curl/curl.h>

CURL *curl_easy_init(void);

加入 curl 7.1 · 源文档：curl.se/libcurl/c/curl_easy_init.html

目的与作用

官方原文："This function allocates and returns an easy handle. Such a handle is used as input to other functions in the easy interface."

分配并返回一个 easy handle（CURL*），作为 easy 接口所有其他函数的输入。每个 easy handle 代表一个独立的网络传输会话，持有所有相关状态和配置。

💡
官方建议："It is encouraged to reuse easy handles for repeated transfers." 重用 easy handle 可利用持久连接（persistent connections），是获得良好性能的关键。
⚠️
官方警告："If you did not already call curl_global_init before calling this function, curl_easy_init does it automatically. This can be lethal in multi-threaded cases for platforms where curl_global_init is not thread-safe." 强烈建议手动调用 curl_global_init()，不要依赖此自动行为。

返回值

若返回 NULL，则初始化失败，不能使用其他 curl 函数（通常由内存不足引起）。

SSL/TLS 选项设置 — curl_easy_setopt

通过 curl_easy_setopt() 设置 SSL/TLS 相关选项，是 easy handle 初始化后的必要配置步骤。以下为核心 SSL 选项（均有官方文档支撑）：

SSL CURLOPT_SSL_VERIFYPEER — 验证服务器证书真实性
CURLcode curl_easy_setopt(CURL *handle, CURLOPT_SSL_VERIFYPEER, long verify);

源文档：curl.se/libcurl/c/CURLOPT_SSL_VERIFYPEER.html

官方原文："This option determines whether curl verifies the authenticity of the peer's certificate."

  • 1L（默认）：验证服务器证书，基于 CA 证书链进行数字签名验证
  • 0L：跳过证书验证（不安全，仅测试用）
🚨
官方警告："disabling verification of the certificate allows bad guys to man-in-the-middle the communication without you knowing it."
SSL CURLOPT_SSL_VERIFYHOST — 验证证书主机名
CURLcode curl_easy_setopt(CURL *handle, CURLOPT_SSL_VERIFYHOST, long verify);

源文档：curl.se/libcurl/c/CURLOPT_SSL_VERIFYHOST.html

官方原文："Pass a long set to 2L to make libcurl verify the host in the server's TLS certificate."

  • 2L（默认）：证书中的 Common Name 或 Subject Alternate Name 必须与 URL 中的主机名匹配
  • 0L：不检查主机名（不安全）
  • 自 7.66.0 起：1 和 2 效果相同
SSL CURLOPT_CAINFO — 指定 CA 证书包路径
CURLcode curl_easy_setopt(CURL *handle, CURLOPT_CAINFO, char *path);

源文档：curl.se/libcurl/c/CURLOPT_CAINFO.html

官方原文："Pass a char pointer to a null-terminated string naming a file holding one or more certificates to verify the peer with."

  • 默认值由构建时确定（通常为系统 CA 证书路径）
  • 可通过 CURLINFO_CAINFO 查询当前默认路径
  • Schannel（Windows）默认使用 Windows 证书存储，此选项为可选

常用 SSL 选项汇总

CURLOPT_SSL_CTX_FUNCTION — SSL 上下文回调

CALLBACK回调原型
CURLcode ssl_ctx_callback(CURL *curl, void *ssl_ctx, void *clientp);

CURLcode curl_easy_setopt(CURL *handle, CURLOPT_SSL_CTX_FUNCTION, ssl_ctx_callback);

加入 curl 7.10.6 · 源文档：curl.se/libcurl/c/CURLOPT_SSL_CTX_FUNCTION.html

目的与作用

官方原文："This callback function gets called by libcurl immediately before the initialization of an SSL connection after having processed all other SSL related options to give a last chance to an application to modify the behavior of the SSL initialization."

在所有 SSL 选项处理完毕、SSL 连接建立之前，给应用程序最后一次机会直接修改底层 SSL 上下文（SSL_CTX）。典型用途：

  • 加载内存中的 CA 证书（无需写入文件）
  • 设置额外的证书验证逻辑
  • 配置 TLS 扩展（如 ALPN、SNI 细节）
  • 启用 OpenSSL 的异步证书验证（SSL_set_retry_verify，8.3.0+）
💡
ssl_ctx 参数说明：
• OpenSSL / wolfSSL：指向 SSL_CTX*
• mbedTLS：指向 mbedtls_ssl_config*
仅支持这三种 TLS 后端，其他后端返回 CURLE_NOT_BUILT_IN。
⚠️
官方注意："A connection that is set up with this callback can be put in the connection pool by libcurl and then reused in following transfers without the callback being called." 若需防止连接被复用到不同属性的传输，应同时设置 CURLOPT_FORBID_REUSE。

示例：从内存加载 CA 证书（OpenSSL）

/* OpenSSL 特定代码 */
static CURLcode sslctx_function(CURL *curl, void *sslctx, void *pointer)
{
    X509_STORE *store;
    X509 *cert = NULL;
    BIO *bio;
    char *mypem = pointer;

    bio = BIO_new_mem_buf(mypem, -1);
    PEM_read_bio_X509(bio, &cert, 0, NULL);

    store = SSL_CTX_get_cert_store((SSL_CTX *)sslctx);
    X509_STORE_add_cert(store, cert);

    X509_free(cert);
    BIO_free(bio);
    return CURLE_OK;
}

int main(void)
{
    CURL *curl;
    char *mypem = "-----BEGIN CERTIFICATE-----\n...-----END CERTIFICATE-----\n";

    curl_global_init(CURL_GLOBAL_ALL);
    curl = curl_easy_init();

    curl_easy_setopt(curl, CURLOPT_URL, "https://example.com/");
    curl_easy_setopt(curl, CURLOPT_SSL_VERIFYPEER, 1L);
    curl_easy_setopt(curl, CURLOPT_SSL_CTX_FUNCTION, *sslctx_function);
    curl_easy_setopt(curl, CURLOPT_SSL_CTX_DATA, mypem);

    curl_easy_perform(curl);
    curl_easy_cleanup(curl);
    curl_global_cleanup();
    return 0;
}

curl_easy_cleanup — 释放 Easy Handle

CLEANUP函数签名
#include <curl/curl.h>

void curl_easy_cleanup(CURL *handle);

加入 curl 7.1 · 源文档：curl.se/libcurl/c/curl_easy_cleanup.html

目的与作用

官方原文："This function is the opposite of curl_easy_init. It closes down and frees all resources previously associated with this easy handle."

是 curl_easy_init() 的逆操作，释放与 easy handle 关联的所有资源：

  • 关闭并释放所有此 handle 持有的连接（若未挂载到 multi handle）
  • 释放内部缓冲区、DNS 缓存、Session ID 缓存等
  • 可能触发协议级别的断开握手（FTP/POP3/IMAP 等），此时进度回调和 header 回调可能被调用
🚫
官方规定："Any use of the easy handle after this function has been called and have returned, is illegal." 调用此函数后，该 handle 指针立即失效，禁止任何访问。
⚠️
官方原文："To close an easy handle that has been used with the multi interface, make sure to first call curl_multi_remove_handle to remove it from the multi handle before it is closed." 与 multi 接口配合使用时，必须先调用 curl_multi_remove_handle()。
💡
传入 NULL 指针时，函数立即返回，不执行任何操作（安全设计）。
♻️

curl_easy_reset — 重置 Handle 选项

RESET函数签名
#include <curl/curl.h>

void curl_easy_reset(CURL *handle);

加入 curl 7.12.1 · 源文档：curl.se/libcurl/c/curl_easy_reset.html

目的与作用

官方原文："Re-initializes all options previously set on a specified curl handle to the default values. This puts back the handle to the same state as it was in when it was created with curl_easy_init."

将 handle 上设置的所有选项重置为默认值，但保留以下状态（官方明确列出）：

  • 活跃的 live connections（连接池）
  • Session ID 缓存
  • DNS 缓存
  • Cookies
  • Share 对象引用
  • alt-svc 缓存
✅
这是在同一 handle 上执行不同配置传输的轻量方式：无需销毁再重建 handle，可保留连接池实现连接复用，同时清除上一次传输的所有选项。

curl_global_cleanup — 全局资源释放

CLEANUP函数签名
#include <curl/curl.h>

void curl_global_cleanup(void);

加入 curl 7.8 · 源文档：curl.se/libcurl/c/curl_global_cleanup.html

目的与作用

官方原文："This function releases resources acquired by curl_global_init."

释放由 curl_global_init() 获取的全局资源，包括 SSL 库的全局结构（如 GNU TLS 协议树、OpenSSL 全局状态等）。

ℹ️
官方原文："You should call curl_global_cleanup once for each call you make to curl_global_init, after you are done using libcurl." 调用次数应与 curl_global_init() 相同（内部有引用计数，最后一次调用才真正释放）。

⚠️ 重要注意事项（官方文档）

⚠️
不等待线程终止：
官方原文："curl_global_cleanup does not block waiting for any libcurl-created threads to terminate (such as threads used for name resolving). If a module containing libcurl is dynamically unloaded while libcurl-created threads are still running then your program may crash or other corruption may occur."
建议不要在可动态卸载的模块中使用 libcurl。
⚠️
OpenSSL 多线程内存泄漏风险：
官方原文："libcurl may not be able to fully clean up after multi-threaded OpenSSL depending on how OpenSSL was built and loaded as a library. It is possible in some rare circumstances a memory leak could occur unless you implement your own OpenSSL thread cleanup."
例如在 Windows 上，若 libcurl 和 OpenSSL 均静态链接到 DLL 或可执行文件，则每个线程退出前需调用 OPENSSL_thread_stop()。
🧵

多线程安全注意事项

源文档：curl.se/libcurl/c/libcurl-thread.html

关键规则（官方文档）

规则官方原文处理方式
Handle 不可跨线程共享 "You must never share the same handle in multiple threads at any given time." 每个线程创建自己的 CURL* handle
全局函数（≤7.83.x）非线程安全 "curl_global_init is not thread safe on platforms without CURL_VERSION_THREADSAFE." 在主线程、单线程阶段调用
多线程中使用信号 "When using multiple threads you should set the CURLOPT_NOSIGNAL option to 1L for all handles." 设置 CURLOPT_NOSIGNAL=1L，注意 SIGPIPE
Share 对象需加锁 "you must provide your own locking and set CURLSHOPT_LOCKFUNC and CURLSHOPT_UNLOCKFUNC." 实现 lock/unlock 回调
全局函数 7.84.0+ 线程安全 "These functions are thread-safe since libcurl 7.84.0 if curl_version_info has the CURL_VERSION_THREADSAFE feature bit set." 检测版本特性位后决定是否需要手动保护

完整流程汇总

初始化与去初始化对应关系

初始化函数对应去初始化函数层级调用时机
curl_global_sslset() 无（无需清理） 全局·SSL选择 curl_global_init 之前，程序启动初期
curl_global_init() curl_global_cleanup() 全局 程序启动（单线程阶段）/ 程序退出前
curl_global_init_mem() curl_global_cleanup() 全局·自定义内存 同 curl_global_init
curl_easy_init() curl_easy_cleanup() Handle 每次传输前 / 传输完毕后（或复用）
curl_easy_init() + setopt curl_easy_reset()（软重置） Handle·选项 同一 handle 需切换配置时

最终完整示例（HTTPS）

#include <stdio.h>
#include <curl/curl.h>

int main(void)
{
    CURL     *curl;
    CURLcode  res;

    /* ─── 步骤0（可选）：选择 SSL 后端，必须最先调用 ─── */
    curl_global_sslset(CURLSSLBACKEND_OPENSSL, NULL, NULL);

    /* ─── 步骤1：全局初始化，单线程、程序启动时调用 ─── */
    res = curl_global_init(CURL_GLOBAL_DEFAULT);
    if(res != CURLE_OK) {
        fprintf(stderr, "全局初始化失败: %s\n", curl_easy_strerror(res));
        return 1;
    }

    /* ─── 步骤2：创建 easy handle ─── */
    curl = curl_easy_init();
    if(!curl) {
        curl_global_cleanup();
        return 1;
    }

    /* ─── 步骤3：配置 URL 与 SSL 选项 ─── */
    curl_easy_setopt(curl, CURLOPT_URL,            "https://example.com/");
    curl_easy_setopt(curl, CURLOPT_SSL_VERIFYPEER, 1L);   /* 验证证书（默认启用）*/
    curl_easy_setopt(curl, CURLOPT_SSL_VERIFYHOST, 2L);   /* 验证主机名（默认启用）*/
    curl_easy_setopt(curl, CURLOPT_CAINFO, "/etc/ssl/certs/ca-bundle.pem");
    curl_easy_setopt(curl, CURLOPT_CA_CACHE_TIMEOUT, 604800L); /* CA 缓存 1 周 */

    /* ─── 步骤4：执行传输 ─── */
    res = curl_easy_perform(curl);
    if(res != CURLE_OK)
        fprintf(stderr, "传输失败: %s\n", curl_easy_strerror(res));

    /* ─── 步骤5：释放 easy handle ─── */
    curl_easy_cleanup(curl);

    /* ─── 步骤6：全局清理，程序退出前调用 ─── */
    curl_global_cleanup();

    return (int)res;
}
