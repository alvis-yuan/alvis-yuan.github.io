C99 标准头文件安全分析指南

ISO/IEC 9899:1999 · 系统性安全分析 · 附使用场景与代码示例

本指南以 C99（ISO/IEC 9899:1999）标准所规定的 24 个标准头文件为基础，对每个头文件的接口进行分类分析：哪些接口是安全的、哪些存在安全风险，以及各自的适用场景。文档参考来源包括 cppreference.com、CERN安全指南、SEI CERT C 编码标准。

C99 的历史背景

C99 标准（ISO/IEC 9899:1999）于 1999 年由 ISO 正式发布，是继 C89/C90 之后十年来第一次重大语言修订。它引入了诸多关键特性，包括：可变长度数组（VLA）、restrict 指针限定、inline 函数、复数支持、指定初始化器、单行 // 注释、以及 6 个新标准头文件。

安全风险分级标准

🟢 安全
不依赖缓冲区大小、无竞争条件、行为明确定义
🟡 需谨慎
在特定条件下安全，需要使用者正确传递参数或理解语义
🔴 高风险
已知存在缓冲区溢出、格式字符串攻击、竞争条件或未定义行为

<stdio.h> — 标准输入/输出

stdio.h 是 C 标准库中使用最频繁、同时也是安全漏洞最集中的头文件。它提供了文件操作、格式化 I/O、字符读写等核心功能。其中 gets()、sprintf()、scanf("%s") 是已知最危险的 C 函数，是绝大多数缓冲区溢出漏洞的根源。参考: CERN C安全指南

典型漏洞示例

🔴 危险：gets() 缓冲区溢出

#include <stdio.h> int main() { char username[8]; int allow = 0; gets(username); // ❌ 攻击者输入 "AAAAAAAA\x01" 覆盖 allow if (allow != 0) privilegedAction(); // 被绕过！ }

🟢 修复：fgets() 安全替代

#include <stdio.h> #include <stdlib.h> int main() { char username[8]; fgets(username, sizeof(username), stdin); // ✅ 最多读 7 字符+'\0' // 去除尾部换行符 username[strcspn(username, "\n")] = '\0'; }

🔴 危险：格式字符串攻击

char *user_input = argv[1]; printf(user_input); // ❌ 若输入"%s%s%n"可泄露/写入内存 printf("%s", user_input); // ✅ 格式串硬编码，安全
🔴 关键规则
永远不要将用户可控字符串直接作为 printf 族函数的格式参数。格式字符串攻击可导致任意内存读写（%n 写入攻击）。

<string.h> — 字符串操作

string.h 提供 C 风格字符串（以 '\0' 结尾的 char 数组）操作函数。C 字符串没有内建的长度信息，这使得许多函数天然容易出错。根据 CERN 安全指南，strcpy、strcat、strcmp 整个函数族都存在缓冲区溢出风险。

函数风险等级风险原因安全替代

strcat(dst,src)🔴 高危在 dst 末尾追加，不检查剩余空间strncat(dst,src,remaining)
gets(via string)🔴 高危（stdio.h 函数，此处关联）fgets
sprintf🔴 高危（stdio.h 函数，此处关联）snprintf
strncpy(d,s,n)🟡 谨慎若 src≥n 则不补终止符，需手动补'\0'—
strncat(d,s,n)🟡 谨慎n 是追加限制，非目标剩余空间，易误用—
strtok()🟡 谨慎使用全局状态，不可重入，修改源字符串strtok_r()(POSIX)
strlen(s)🟢 安全只读，但要求 s 有效且以 \0 结尾—
strcmp/strncmp🟢 安全只读比较，strncmp 更安全—
strchr/strrchr🟢 安全只读搜索，结果需检查 NULL—
memcpy(d,s,n)🟡 谨慎区域重叠时 UB（用 memmove），n 需验证memmove()
memmove(d,s,n)🟢 安全正确处理内存重叠，但 n 仍需验证—
memset(s,c,n)🟢 安全需确保 n 不超过 s 的大小—
strstr(h,n)🟢 安全只读搜索，结果需 NULL 检查—

代码示例

🔴 strncat 常见误用

char dst[10] = "Hello"; // ❌ 错误: n 不是剩余空间，而是追加的最大字节数 strncat(dst, " World!", 10); // 总长 12 > 10，仍溢出! // ✅ 正确: 计算剩余空间 strncat(dst, " World!", sizeof(dst) - strlen(dst) - 1);

🟡 strncpy 需手动补终止符

char dst[5]; strncpy(dst, "Hello World", sizeof(dst)); // dst 可能没有 '\0'! 需手动补: dst[sizeof(dst) - 1] = '\0'; // ✅ 更好的方式: 用 snprintf snprintf(dst, sizeof(dst), "%s", src);

<stdlib.h> — 通用工具库

stdlib.h 是功能最混杂的头文件，涵盖内存管理、随机数、字符串转换、程序控制、排序搜索等。其中 system()、atoi()、rand() 是常见安全隐患。参考: cppreference stdlib.h

函数风险等级说明建议
system(cmd)🔴 高危调用 shell 执行命令，若 cmd 含用户输入则命令注入避免使用，改用 exec 族
atoi/atol/atof🔴 高危溢出时 UB，无错误检测，无法区分"0"和失败strtol/strtod + 错误检查
rand()🟡 谨慎不密码安全的 PRNG，不应用于安全场景OS随机源(/dev/urandom)
malloc(n)🟡 谨慎不归零，忘记检查 NULL；整数溢出时分配不足calloc() 或检查+初始化
calloc(n,sz)🟢 较安全自动归零，内部检测 n×sz 溢出（实现依赖）—
realloc(p,n)🟡 谨慎失败返回 NULL 但原指针未释放，需保存旧指针用临时变量接收返回值

bsearch()🟢 安全只读搜索，结果需 NULL 检查—
abort()/exit()🟢 安全行为明确，exit 调用 atexit 回调—

代码示例

🔴 危险：命令注入

char cmd[256]; snprintf(cmd, sizeof(cmd), "ping %s", user_ip); system(cmd); // ❌ 若 user_ip = "8.8.8.8; rm -rf /" → 灾难

🟢 安全：strtol 替代 atoi

#include <stdlib.h> #include <errno.h> long safe_parse(const char *s) { char *end; errno = 0; long val = strtol(s, &end, 10); if (errno == ERANGE) return /* 溢出处理 */; if (end == s || *end != '\0') return /* 非数字 */; return val; // ✅ 安全且可靠 }

<stdint.h> — 固定宽度整数类型 C99新增

stdint.h 是 C99 引入的重大改进，解决了 C 内置整数类型（int、long 等）大小依赖实现的问题。在不同平台上 int 可能是 16/32 位，long 可能是 32/64 位，这是嵌入式和跨平台代码的主要 bug 来源。参考: Stack Overflow: stdint.h and C99

<stdint.h> C99 新增 固定宽度整数类型定义
类型风险等级说明
int8_t ~ int64_t🟢 安全精确宽度有符号整数，跨平台一致
uint8_t ~ uint64_t🟢 安全精确宽度无符号整数，常用于协议/内存布局
intptr_t/uintptr_t🟢 安全能容纳指针的整数类型，用于指针运算
intmax_t/uintmax_t🟢 安全平台上最大整数类型
int_leastN_t🟢 安全至少 N 位宽，用于最大兼容性
int_fastN_t🟢 安全至少 N 位且最快的类型
INT8_MAX ~ UINT64_MAX🟢 安全对应宏常量，用于边界检查
🟢 stdint.h 是 C99 最重要的安全改进之一
使用固定宽度类型可以消除大量因类型大小假设错误导致的整数截断、溢出和跨平台不一致问题。在嵌入式、网络协议、文件格式处理中强烈推荐使用。

代码示例

#include <stdint.h> #include <stdio.h> // ❌ 不可移植: long 在 Windows(32位) 和 Linux(64位) 大小不同 long packet_id; // ✅ 可移植: 明确 32 位 int32_t packet_id; // 边界检查示例 void add_safe(int32_t a, int32_t b) { if (a > INT32_MAX - b) { /* 溢出处理 */ return; } int32_t result = a + b; // ✅ 安全 }

<inttypes.h> — 整数类型格式化 C99新增

inttypes.h 包含 stdint.h，并额外提供用于 printf/scanf 格式化固定宽度整数的宏（如 PRId32、SCNu64）。这解决了跨平台打印 uint64_t 时格式符不一致的问题（Linux 用 %lu，Windows 用 %llu）。参考: Stack Overflow: stdint vs inttypes

宏/函数风险等级说明
PRId8~PRId64🟢 安全printf 有符号格式宏，展开为平台正确的格式串
PRIu8~PRIu64🟢 安全printf 无符号格式宏
PRIx8~PRIx64🟢 安全printf 十六进制格式宏
SCNd8~SCNd64🟢 安全scanf 有符号格式宏
SCNu8~SCNu64🟢 安全scanf 无符号格式宏

imaxdiv()🟢 安全intmax_t 的整除，返回商和余数
strtoimax/strtoumax🟢 安全安全的字符串到 intmax_t 转换，带溢出检测
wcstoimax/wcstoumax🟢 安全宽字符版本

代码示例

#include <inttypes.h> #include <stdio.h> uint64_t big_num = 18446744073709551615ULL; // ❌ 不可移植: %lu 在 Windows 是 32 位，%llu 在部分 Linux 无效 printf("%lu\n", big_num); // ✅ 使用 inttypes 宏，自动适配平台 printf("%" PRIu64 "\n", big_num); printf("Hex: 0x%" PRIX64 "\n", big_num); // scanf 版本 int64_t val; scanf("%" SCNd64, &val);

<math.h> — 数学函数库

math.h 提供标准数学函数（三角、对数、幂等）。总体上安全性较高，主要风险来自定义域错误（如 sqrt(-1)、log(0)）和浮点精度问题导致的逻辑错误。C99 新增了大量函数并引入了错误处理机制。

<math.h> 数学函数 · C89 基础 · C99 大幅扩展
函数/宏风险等级说明
sin/cos/tan🟢 安全全定义域，结果范围 [-1,1]，无错误
asin/acos🟡 谨慎参数需在 [-1,1]，超范围返回 NaN+EDOM
sqrt(x)🟡 谨慎x<0 时返回 NaN，需提前检查
log/log2/log10🟡 谨慎x≤0 时分别返回 NaN 或 -∞
pow(x,y)🟡 谨慎x<0 且 y 非整数时 EDOM；溢出返回 ±HUGE_VAL
fmod(x,y)🟡 谨慎y==0 时返回 NaN
ceil/floor/round🟢 安全全定义域，但结果转 int 需检查范围
fabs(x)🟢 安全全定义域浮点绝对值
isnan(x) [C99]🟢 安全检测 NaN，宏实现
isinf(x) [C99]🟢 安全检测无穷大
isfinite(x) [C99]🟢 安全检测有限数
fpclassify [C99]🟢 安全分类浮点数（正常/次正规/零/无穷/NaN）
modf(x, *ip)🟢 安全分离整数和小数部分
frexp/ldexp🟢 安全浮点数分解/合成，科学计算中使用

代码示例

#include <math.h> #include <errno.h> #include <stdio.h> double safe_sqrt(double x) { if (x < 0.0) { fprintf(stderr, "sqrt domain error\n"); return 0.0; // 或返回 NAN } return sqrt(x); // ✅ 前置检查后安全 } // 检测浮点异常 double r = log(user_val); if (!isfinite(r)) { // C99 宏，检测 NaN 和 inf /* 错误处理 */ }

<complex.h> — 复数运算 C99新增

complex.h 是 C99 全新引入的头文件，提供完整的复数类型 _Complex（或通过宏定义的 complex）和对应的数学函数。背景：科学计算和信号处理领域长期需要复数支持，此前只能用结构体模拟。参考: POSIX tgmath.h 规范

类型/函数风险等级说明

creal/cimag🟢 安全提取实部/虚部
cabs(z)🟢 安全模（绝对值），等价实数的 fabs

代码示例

#include <complex.h> #include <stdio.h> int main() { // C99: 直接定义复数 double complex z = 3.0 + 4.0*I; // I 是虚数单位宏 printf("模: %.2f\n", cabs(z)); // 输出 5.00 printf("辐角: %.4f rad\n", carg(z)); // 复数开方（对负数有效） double complex neg = -1.0 + 0.0*I; double complex root = csqrt(neg); // 结果: 0+1i printf("sqrt(-1) = %.1f+%.1fi\n", creal(root), cimag(root)); return 0; }
💡 适用场景
信号处理（FFT）、电气工程仿真、量子力学计算、控制系统设计。complex.h 接口整体安全，主要注意 cpow 的边界条件。

<tgmath.h> — 类型泛型数学 C99新增

tgmath.h（Type-Generic Math）是 C99 引入的便利头文件，通过宏魔法统一了 math.h 和 complex.h 的函数命名。传统上 float/double/long double 需要分别调用 sqrtf/sqrt/sqrtl，现在统一用 sqrt() 宏即可根据参数类型自动选择。参考: POSIX tgmath.h

宏风险等级展开行为
sqrt(x)🟢 安全float→sqrtf, double→sqrt, _Complex→csqrt
sin/cos/tan🟢 安全根据参数类型自动分派
log/log2/log10🟡 谨慎继承底层函数的定义域限制
pow(x,y)🟡 谨慎继承 pow/cpow 的边界条件
fabs/cabs🟢 安全实数→fabs，复数→cabs
#include <tgmath.h> // 包含 math.h 和 complex.h #include <stdio.h> float a = sqrt(2.0f); // → sqrtf() double b = sqrt(2.0); // → sqrt() double complex c = sqrt(-1.0+0*I); // → csqrt() // 不需要记忆 sqrtf/sqrtl/csqrtf 等命名！

<fenv.h> — 浮点环境控制 C99新增

fenv.h 是 C99 引入的新头文件，提供对 IEEE 754 浮点环境（舍入模式、浮点异常标志）的程序化控制。背景：科学计算和数值分析需要精确控制浮点运算行为，如检测除以零或上溢事件。

<ctype.h> — 字符分类与转换

ctype.h 提供字符分类（是否字母、数字、空白等）和大小写转换函数。这些函数整体安全，但有一个关键的陷阱：参数必须是 unsigned char 值或 EOF，直接传入有符号 char 可能导致未定义行为。

#include <ctype.h> // ❌ 潜在 UB: char 可能是有符号的 char c = '\xff'; // 有符号时为 -1 if (isalpha(c)) { /* UB on many platforms */ } // ✅ 正确: 强制转为 unsigned char if (isalpha((unsigned char)c)) { /* 安全 */ } // ✅ 也可以: 读取时存为 int（fgetc 返回值） int ch = fgetc(fp); // int 可容纳 EOF(-1) if (ch != EOF && isalpha(ch)) { /* 安全 */ }

<locale.h> — 本地化设置

locale.h 提供区域设置（locale）管理，影响字符分类、数值格式、日期格式、货币符号等。主要安全风险是 setlocale 改变全局状态，影响多线程程序和库的行为。

<locale.h> 本地化设置 · 自 C89
函数风险等级说明
setlocale(cat, loc)🟡 谨慎修改全局 locale，非线程安全；影响 ctype/printf/scanf
localeconv()🟡 谨慎返回静态结构体指针，下次调用 setlocale 后失效
LC_ALL, LC_NUMERIC 等宏🟢 安全仅为常量，无副作用
⚠ 多线程注意
setlocale() 修改进程级全局状态。在多线程程序中，并发调用 setlocale 是未定义行为。POSIX 的 uselocale() 提供线程局部 locale 支持。

<wchar.h> — 宽字符操作

wchar.h（C95 引入）提供宽字符（wchar_t）版本的字符串操作和 I/O 函数。它与 string.h/stdio.h 类似，存在对应的安全风险。wchar_t 的大小在 Windows（16位 UTF-16）和 Linux（32位 UTF-32）上不同，是跨平台难点。

<wchar.h> 宽字符/宽字符串操作 · 自 C95
函数风险等级说明
wcscpy/wcscat🔴 高危宽字符版 strcpy/strcat，同样无边界检查
wcsncpy/wcsncat🟡 谨慎有长度限制，同 strncpy 的注意事项
wprintf/swprintf🟡 谨慎格式串需硬编码，swprintf 有大小限制（较安全）
fgetws/fputws🟢 安全带长度限制的宽字符行读写
wcslen🟢 安全宽字符串长度，要求有效 wchar_t*
wcscmp/wcsncmp🟢 安全宽字符串比较
mbrtowc/wcrtomb🟡 谨慎多字节与宽字符转换，有内部状态
wcsftime🟢 安全带大小限制的宽字符时间格式化

<wctype.h> — 宽字符分类

wctype.h 是 ctype.h 的宽字符版本，提供宽字符的分类函数（iswalpha、iswdigit 等）和大小写转换（towupper、towlower）。安全级别与 ctype.h 类似，相对安全。

<wctype.h> 宽字符分类与转换 · 自 C95
函数风险等级说明
iswalpha/iswdigit🟢 安全宽字符分类，接受 wint_t（可容纳 WEOF）
iswspace/iswpunct🟢 安全同上，受 locale 影响
towupper/towlower🟢 安全宽字符大小写转换，locale 影响结果
wctype/iswctype🟢 安全动态字符类型查询，扩展分类
wctrans/towctrans🟢 安全动态字符转换映射

<iso646.h> — ISO 646 替代运算符

iso646.h（C95 引入）为 C 运算符提供文字别名，这是为了在不支持 { } [ ] 等字符的 ISO 646 字符集终端上编程。现代编程中极少使用，C++ 将这些关键字提升为语言内建。

<iso646.h> 替代运算符写法 · 自 C95
宏等价符号风险等级
and&&🟢 安全
or||🟢 安全
not!🟢 安全
and_eq&=🟢 安全
or_eq|=🟢 安全
xor^🟢 安全
not_eq!=🟢 安全
bitand/bitor/compl&/|/~🟢 安全
💡 历史背景
ISO 646 是 1970 年代的 7 位字符集标准，某些国家变体中 { } [ ] # 等字符会被替换为本国字符。iso646.h 让这些环境中的程序员也能写 C 代码。现代环境均支持 ASCII，这些宏已基本废弃。

<signal.h> — 信号处理

signal.h 提供 UNIX 信号处理机制。信号处理器（signal handler）在异步的、不可预测的时机被调用，这带来了严重的安全和正确性挑战。信号处理器中只允许调用极少数"异步信号安全（async-signal-safe）"函数。参考: SEI CERT SIG00-C

<signal.h> 信号处理 · 自 C89 · 需谨慎使用
函数/宏风险等级说明
signal(sig, handler)🟡 谨慎注册信号处理器，处理器内受严格限制
raise(sig)🟡 谨慎向自身发送信号，行为依赖处理器设置
sig_atomic_t🟢 安全信号处理器中唯一可安全读写的类型
SIGINT/SIGTERM 等宏🟢 安全信号编号常量
SIG_DFL/SIG_IGN🟢 安全默认/忽略信号的特殊处理器值
处理器中调用 printf🔴 高危非异步信号安全，可能死锁或破坏 stdio 内部状态
处理器中调用 malloc/free🔴 高危堆操作非 async-signal-safe，可能导致堆损坏
处理器中修改全局变量🟡 谨慎需声明为 volatile sig_atomic_t

代码示例

#include <signal.h> #include <stdio.h> // ✅ 正确: 使用 volatile sig_atomic_t 标志 volatile sig_atomic_t g_interrupted = 0; void handler(int sig) { g_interrupted = 1; // ✅ 唯一安全操作 // ❌ 不要在这里: printf, malloc, free, longjmp ... } int main() { signal(SIGINT, handler); while (!g_interrupted) { /* 主循环 */ } printf("中断信号收到，清理退出\n"); // ✅ 在主线程中处理 }

<setjmp.h> — 非局部跳转

setjmp.h 提供 setjmp/longjmp 机制，实现跨函数的非局部跳转（类似于异常处理）。这是 C 中最危险的机制之一，因为它绕过了正常的栈展开，可能导致资源泄漏、局部变量值不确定等问题。参考: Wikipedia: setjmp.h

<setjmp.h> 非局部跳转 · 自 C89 · 高风险
函数/类型风险等级说明
setjmp(env)🟡 谨慎保存执行环境，返回0（初次）或非零（longjmp返回）
longjmp(env, val)🔴 高危恢复环境，绕过栈展开，需谨慎管理资源
jmp_buf🟢 安全环境缓冲区类型，不可复制到另一线程
longjmp 跨线程🔴 高危跳转到另一线程的 setjmp 是 UB
从信号处理器 longjmp🔴 高危仅在极严格限制下可行，一般应避免
longjmp 后未保存的 auto 变量🔴 高危未声明为 volatile 的局部变量值不确定

代码示例

#include <setjmp.h> #include <stdio.h> jmp_buf err_env; void risky_op() { // ❌ 如果这里有 malloc 并未 free，longjmp 后泄漏 if (error_condition) longjmp(err_env, 1); // 跳回 setjmp 处 } int main() { if (setjmp(err_env) != 0) { fprintf(stderr, "捕获到错误\n"); return 1; } // ✅ volatile 保证 longjmp 后值确定 volatile int saved = 42; risky_op(); return 0; }

<errno.h> — 错误码

errno.h 提供错误报告机制：宏 errno 在库函数失败时被设置为错误码。C99 要求 errno 是线程局部的（每线程独立），这修复了早期实现中的线程安全问题。

<errno.h> 错误码报告 · 自 C89 · C99 改进为线程局部
宏/类型风险等级说明
errno🟡 谨慎宏，展开为线程局部 int 左值，成功调用也可能改变它
EDOM🟢 安全定义域错误（数学函数）
ERANGE🟢 安全结果超范围
EILSEQ🟢 安全无效多字节序列（C99 新增）
#include <errno.h> #include <stdio.h> // ✅ 关键: 调用前先清零 errno errno = 0; double result = strtod(input, NULL); if (errno == ERANGE) { perror("转换溢出"); } // ❌ 错误做法: 不清零就检查（上次错误可能残留） double x = strtod(input, NULL); if (errno) { /* 可能是假阳性！ */ }

<assert.h> — 断言诊断

assert.h 提供 assert 宏，用于在调试阶段检测不变量违反。当 NDEBUG 宏定义时，所有 assert 展开为空操作（no-op），因此不应将 assert 用于正式的错误处理或安全检查。

<assert.h> 程序断言 · 自 C89
宏风险等级说明
assert(expr)🟡 谨慎调试检查，NDEBUG时禁用；不用于生产安全检查
NDEBUG 宏🟡 谨慎定义此宏会禁用所有断言，需注意副作用消失
static_assert [C11]🟢 安全编译期断言，不受 NDEBUG 影响
#include <assert.h> // ✅ 调试不变量 assert(ptr != NULL); // 仅调试时有效 assert(size > 0 && size < MAX_SIZE); // ❌ 不要这样做: assert 中有副作用 assert(fclose(fp) == 0); // NDEBUG时 fclose 不调用！资源泄漏！ // ✅ 正确的资源释放方式 int r = fclose(fp); assert(r == 0); // 副作用在 assert 外部

<stddef.h> — 基础类型与宏定义

stddef.h 定义了 C 程序中广泛使用的基础类型和宏。这是一个纯定义头文件，不包含任何函数实现，整体安全。

<stddef.h> 基础类型定义 · 自 C89
类型/宏风险等级说明
NULL🟢 安全空指针常量，解引用是 UB，使用前需检查
size_t🟡 谨慎无符号类型；整数溢出（如 n-1 当 n=0）是常见 bug
ptrdiff_t🟢 安全指针差值类型，有符号
wchar_t🟢 安全宽字符类型，大小实现定义
offsetof(type,member)🟢 安全结构体成员偏移量，常量表达式
#include <stddef.h> // size_t 下溢陷阱 size_t n = 0; if (n - 1 < MAX) { // ❌ n-1 = SIZE_MAX (巨大正数)，条件可能意外成立! } // ✅ 正确: 先检查 n > 0 if (n > 0 && n - 1 < MAX) { /* 安全 */ } // offsetof 用于序列化 struct Point { int x; int y; }; size_t off = offsetof(struct Point, y); // 4（通常）

<stdarg.h> — 可变参数

stdarg.h 提供访问可变参数函数（如 printf 内部实现）的机制。主要风险是类型安全性完全由调用者保证，错误的类型假设会导致未定义行为。

<stdarg.h> 可变参数访问 · 自 C89
宏风险等级说明
va_list🟢 安全参数列表类型
va_start(ap, last)🟡 谨慎初始化 va_list，last 必须是最后一个具名参数
va_arg(ap, type)🔴 高危提取下一个参数；类型必须与实参完全匹配，否则 UB

#include <stdarg.h> #include <stdio.h> double sum(int count, ...) { va_list ap; va_start(ap, count); double total = 0.0; for (int i = 0; i < count; i++) { // ✅ 必须确保调用者传入 double，否则 UB total += va_arg(ap, double); } va_end(ap); return total; } // sum(3, 1.0, 2.0, 3.0) ✅ | sum(3, 1, 2, 3) ❌ (int≠double → UB)

<stdbool.h> — 布尔类型 C99新增

stdbool.h 是 C99 引入的简单头文件，定义了 bool 类型（等价于 _Bool）和 true/false 宏。C89 没有布尔类型，程序员需自己 #define 或用 int，这是 bug 的常见来源。C23 起 bool/true/false 成为语言关键字，stdbool.h 已废弃。

<stdbool.h> C99 新增 布尔类型 · C23起内建关键字
宏风险等级说明

true🟢 安全展开为 1
false🟢 安全展开为 0
__bool_true_false_are_defined🟢 安全特性检测宏，值为 1
#include <stdbool.h> // C99 之前: 容易出错的自定义 // #define TRUE 1 // #define FALSE 0 (不同库可能不一致) // ✅ C99: 标准化的布尔类型 bool is_valid = true; bool result = (x > 0) && (x < MAX); // _Bool 的截断特性 _Bool b = 42; // 存储为 1（非零自动截断为1）

<time.h> — 时间日期

time.h 提供时间测量和格式化功能。主要安全风险包括：time_t 溢出（2038年问题）、缓冲区溢出（asctime/ctime）、以及静态缓冲区导致的非线程安全问题。

<time.h> 时间日期处理 · 自 C89

#include <time.h> #include <stdio.h> // ❌ 危险: 静态缓冲区，非线程安全 time_t t = time(NULL); char *s = ctime(&t); // 静态缓冲区，多线程中被覆盖 // ✅ 安全: strftime + 用户提供的缓冲区 struct tm *tm_info = localtime(&t); // 仍非线程安全，用 localtime_r 更好 char buf[64]; strftime(buf, sizeof(buf), "%Y-%m-%d %H:%M:%S", tm_info); printf("%s\n", buf);
⚠ 2038年问题
32位系统上 time_t 通常是 32位有符号整数，在 2038年1月19日 03:14:07 UTC 溢出。64位系统不受影响。确保 time_t 为 64位类型。
