引论（Introduction）

第1页 — 为什么需要线程？从同步到异步的思维跨越
异步并发并行 线程安全可重入POSIX

1.1 "救生筏程序员"（The Bailing Programmers）

Butenhof 以一个比喻开篇：一艘正在进水的船上，有人奋力舀水（同步单线程），有人则招来帮手分工合作（多线程）。这个比喻阐明了并发的本质：世界本身就是异步的，程序应该反映这种异步性，而不是人为地将其序列化。

作者认为，大多数程序员长期以来习惯于单线程的顺序执行模型，这是历史局限性，而非最优模型。引入线程，是让程序"跟上现实世界节奏"的自然进化。

1.2 定义与术语

术语定义关键区别
异步（Asynchronous）操作在发起者不等待其完成的情况下继续进行描述操作的时序关系
并发（Concurrency）多个执行流在时间上交叠，不一定同时运行逻辑上的"同时"
并行（Parallelism）多个执行流在物理上同时运行（需要多核/多处理器）物理上的"同时"
线程安全函数或数据结构在多线程并发访问时行为正确关注外部接口
可重入（Reentrant）函数可以被同时安全地多次调用（无静态/全局状态）实现层面的属性
💡
第一性原理：并发 vs 并行
软件"并发"= 英语"concurrency"（逻辑交叠）；软件"并行"= 英语"parallelism"（同时物理执行）。Butenhof 特别强调：在单处理器上可以有并发，但不能有并行；在多处理器上两者都可以存在。线程提供并发能力，多核才能实现真正并行。

1.3 为什么异步编程是直觉性的

因为 UNIX 本身是异步的： 信号（signal）、进程调度、I/O 多路复用——这些都表明异步是 UNIX 的基因。

因为现实世界是异步的： 厨师同时炒菜和烧水，文字处理器同时响应键盘和自动保存，网络服务器同时处理多个连接。用顺序程序建模异步世界是一种"阻抗失配"。

1.5 线程的收益与代价

收益

  • 并行性： 在多处理器系统上，线程可真正同时运行，提升计算吞吐量。
  • 并发性： 即使在单处理器，I/O 等待期间 CPU 可以执行其他线程，提升资源利用率。
  • 编程模型： 将复杂的异步状态机简化为多个简单的顺序流，代码更易理解和维护。

代价

  • 计算开销： 线程创建、上下文切换、同步操作都有开销。
  • 编程纪律： 必须正确使用互斥锁和条件变量，否则会产生竞态条件。
  • 调试困难： 线程交错执行是非确定性的，难以复现 bug。

1.6 POSIX 线程概念

体系结构概览： 一个进程包含一个或多个线程，所有线程共享进程的地址空间（代码、堆、全局变量、文件描述符），但每个线程有自己的栈和寄存器。

类型与接口： Pthreads 定义了 pthread_t（线程ID）、pthread_mutex_t（互斥量）、pthread_cond_t（条件变量）等核心类型。

错误检查： Pthreads 函数不使用 errno（以避免多线程冲突），而是通过返回值传递错误码。

/* Pthreads 的正确错误处理模式 */ int status; status = pthread_create(&thread, NULL, thread_func, NULL); if (status != 0) { fprintf(stderr, "pthread_create: %s\n", strerror(status)); return status; /* 不是 errno，是返回值！*/ }

线程（Threads）

第35页 — 线程的创建、生命周期与运行机制
pthread_createpthread_join pthread_detach线程生命周期

2.1 创建和使用线程

线程通过 pthread_create() 创建。新线程与调用者并发执行，共享进程地址空间但拥有独立的执行栈。

/* 最简单的线程创建示例 (hello.c 风格) */ #include <pthread.h> void *thread_routine(void *arg) { printf("Hello from thread!\n"); return NULL; } int main() { pthread_t thread; pthread_create(&thread, NULL, thread_routine, NULL); pthread_join(thread, NULL); /* 等待线程结束 */ return 0; }

2.2 线程的生命周期

Butenhof 将线程的完整生命周期分为五个阶段，这是理解线程行为的核心框架：

图2.1 线程完整生命周期状态图
  • 创建（Creation）： 调用 pthread_create()，分配线程 ID 和栈空间。
  • 启动（Startup）： 线程函数开始执行，但不保证何时开始（由调度器决定）。
  • 运行与阻塞（Running & Blocking）： 线程在就绪和等待状态之间切换，等待 I/O、锁或条件变量时阻塞。
  • 终止（Termination）： return、pthread_exit()、被取消，或进程退出。
  • 回收（Recycling）： 通过 pthread_join()（可获取返回值）或 pthread_detach()（自动回收资源）。
⚠️
僵尸线程： 若线程既未被 join 也未被 detach，其资源（包括返回值）将永久占用，类似进程中的僵尸进程。每个 pthread_create 必须有对应的 pthread_join 或 pthread_detach。

pthread_self()获取自身 ID可传递给其他线程

3.1 不变量、临界区与谓词

这是本书最重要的概念三角，也是正确多线程编程的哲学基础：

  • 不变量（Invariant）： 程序状态必须始终满足的约束条件（例如："链表的 head 指针要么为 NULL，要么指向有效节点"）。不变量在单线程中天然维护，在多线程中需要显式保护。
  • 临界区（Critical Section）： 必须以原子方式执行的代码区域，不允许被中断打断。通过互斥量来界定临界区。
  • 谓词（Predicate）： 线程等待的条件表达式（例如："队列不为空"）。谓词基于共享状态，必须在持有互斥量时检查。
🚨
竞态条件（Race Condition）：当不变量被破坏期间，另一个线程观察到了中间状态，就产生竞态条件。多线程编程的核心任务，就是防止任何线程观察到被破坏的不变量。

3.2 互斥量（Mutex）

互斥量（Mutual Exclusion Lock）是保护临界区的基本工具。同一时刻只有一个线程可以持有互斥量。

/* 互斥量的标准使用模式 */ pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER; /* 静态初始化 */ /* 或动态初始化 */ pthread_mutex_t *mutex = malloc(sizeof(pthread_mutex_t)); pthread_mutex_init(mutex, NULL); /* 使用 */ pthread_mutex_lock(&mutex); /* === 临界区：安全访问共享数据 === */ shared_counter++; /* ================================== */ pthread_mutex_unlock(&mutex); /* 销毁 */ pthread_mutex_destroy(&mutex);

互斥量的高级用法

  • 非阻塞尝试锁： pthread_mutex_trylock()——若锁已被占用则立即返回 EBUSY，不阻塞。
  • 锁的粒度： 粒度太粗降低并发性，粒度太细增加复杂性和开销。需根据数据访问模式合理设计。
  • 使用多个互斥量： 必须建立严格的锁层级（Lock Hierarchy），所有线程按相同顺序加锁，避免死锁。
  • 锁链（Lock Chaining）： 遍历数据结构时，持有当前节点的锁再获取下一节点的锁，然后释放前者。
图3.1 死锁与锁层级对比

3.3 条件变量（Condition Variables）

条件变量允许线程在某个条件不满足时高效地等待，而不是忙等待（spin）。它必须与互斥量配合使用。

/* 条件变量的标准使用模式 */ pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER; pthread_cond_t cond = PTHREAD_COND_INITIALIZER; int ready = 0; /* 等待方（消费者） */ pthread_mutex_lock(&mutex); while (!ready) { /* ⚠️ 必须用 while，不能用 if！*/ pthread_cond_wait(&cond, &mutex); /* 原子释放锁+等待 */ } /* 现在条件满足，处理数据 */ pthread_mutex_unlock(&mutex); /* 通知方（生产者） */ pthread_mutex_lock(&mutex); ready = 1; pthread_cond_signal(&cond); /* 唤醒一个等待者 */ /* 或 pthread_cond_broadcast(&cond) 唤醒所有等待者 */ pthread_mutex_unlock(&mutex);
⚠️
为什么必须用 while 而不是 if？
虚假唤醒（Spurious Wakeup）： POSIX 允许 pthread_cond_wait 在没有收到信号的情况下返回。用 while 循环可以在唤醒后重新检查条件，确保真正满足后再继续。这是 Pthreads 编程铁律。

3.4 内存可见性（Memory Visibility）

在现代多处理器系统中，每个 CPU 有自己的缓存。一个线程对共享变量的写入，不一定立即对其他线程可见。

Pthreads 保证：互斥量的加锁/解锁操作隐含内存屏障。即：线程 A 在持有锁时的所有写入，在线程 B 获取同一把锁之后，对 B 完全可见。这是内存模型的关键保证。

4.1 流水线（Pipeline）

将任务分解为多个连续处理阶段，每个阶段由一个线程处理，阶段间通过队列传递数据。类似工厂装配线。

图4.1 流水线模式：各阶段并行处理，提升吞吐量

优点： 各阶段并行执行，提高吞吐量；代码组织清晰，每个线程只做一件事。
适用场景： 编译器（词法分析→语法分析→代码生成）、多媒体处理、网络数据包处理。

4.2 工作组（Work Crew）

多个工作线程并行执行相同类型的任务，各自独立处理任务列表中的项目。类似于将一个大任务分配给多个工人同时完成。

书中示例（crew.c）是一个文件搜索工具：主线程建立工作组，工作线程并行遍历目录树搜索字符串。所有线程共享一个任务队列，自行取任务处理。

图4.2 工作组模式：多线程并行处理同类任务

4.3 客户/服务器（Client/Server）

一个或多个服务器线程处理来自客户端的请求。服务器线程等待请求，处理后返回结果。这是网络服务器最常见的多线程模型。

书中示例（server.c）展示了多个客户端线程向服务器线程提交请求，服务器顺序处理并返回，期间保证 I/O 不被其他线程干扰。

5.1 一次性初始化（One-Time Initialization）

pthread_once() 确保某段初始化代码在进程生命周期内只执行一次，无论多少线程并发调用。

pthread_once_t once_control = PTHREAD_ONCE_INIT; void init_routine() { /* 只执行一次的初始化 */ global_resource = create_resource(); } /* 每个线程都调用，但 init_routine 只执行一次 */ pthread_once(&once_control, init_routine);

5.2 属性对象（Attributes Objects）

Pthreads 使用属性对象（attr）来配置线程、互斥量和条件变量的非默认行为，实现策略与机制的分离：

属性对象可配置项典型用途
pthread_attr_t栈大小、栈地址、detach状态、调度策略/优先级嵌入式系统定制栈，设置实时优先级
pthread_mutexattr_t类型（普通/递归/纠错）、优先级协议防止死锁的递归互斥量
pthread_condattr_t进程共享属性跨进程共享条件变量

5.3 取消（Cancellation）

取消机制允许一个线程请求终止另一个线程。这是 Pthreads 中最复杂的特性之一，需要谨慎使用。

取消类型

  • 延迟取消（Deferred Cancelability）： 默认模式。取消请求被保留，直到线程到达"取消点"（如 pthread_testcancel()、sleep()、read() 等系统调用）才真正终止。安全可控。
  • 异步取消（Asynchronous Cancelability）： 线程可在任何时刻被强制终止。极度危险，几乎只有在执行纯计算（无系统调用、无锁操作）时才安全使用。

清理处理器（Cleanup Handlers）

使用 pthread_cleanup_push/pop 注册清理函数，确保线程被取消时能释放已持有的资源（锁、内存等），防止资源泄露。

pthread_cleanup_push(cleanup_mutex, &mutex); /* 注册清理：解锁 */ pthread_mutex_lock(&mutex); /* ... 临界区操作 ... */ pthread_mutex_unlock(&mutex); pthread_cleanup_pop(0); /* 注销清理（不执行）*/
🚨
取消安全规则： 每次 pthread_mutex_lock 之后，如果取消是启用的，必须注册相应的清理处理器来解锁。否则取消发生时，锁将永远保持，导致其他线程死锁。

5.4 线程私有数据（Thread-Specific Data, TSD）

TSD 允许每个线程拥有同一个"键（key）"对应的不同值，实现线程本地存储（类似全局变量，但每个线程独立）。最典型的例子是让 errno 在多线程中线程安全。

pthread_key_t key; /* 创建键（通常用 pthread_once 保护）*/ pthread_key_create(&key, destructor_function); /* destructor在线程退出时自动调用 */ /* 各线程设置自己的值 */ pthread_setspecific(key, thread_local_data); /* 各线程读取自己的值 */ void *data = pthread_getspecific(key);

5.5 实时调度（Realtime Scheduling）

POSIX 实时调度允许程序员控制线程的优先级和调度策略，适用于对时延敏感的系统。

调度策略行为适用场景
SCHED_FIFO先进先出，运行至自愿让出或被高优先级抢占实时控制系统
SCHED_RR轮转，有时间片限制实时任务中的公平调度
SCHED_OTHER系统默认，不保证优先级普通应用程序

优先级反转（Priority Inversion）： 高优先级线程等待低优先级线程释放锁，被中优先级线程间接阻塞。解决方案：优先级天花板（Priority Ceiling）和优先级继承（Priority Inheritance）互斥量。

引入线程后，POSIX 系统的许多原有语义发生了根本性变化。本章是"兼容性与陷阱"手册，对每个开发者都至关重要。

6.1 fork() 的陷阱

根本问题： fork() 创建子进程时，只复制调用线程，但复制所有互斥量的状态。若父进程中其他线程持有了某把锁，子进程会继承这个已锁定的互斥量，但那个持锁线程不存在于子进程中，导致死锁。

☠️
黄金法则： 在多线程程序中调用 fork() 后，子进程只应调用异步信号安全（async-signal-safe）的函数，或立即调用 exec()。绝不能在子进程中使用父进程的互斥量。

解决方案：fork 处理器（Fork Handlers）：使用 pthread_atfork(prepare, parent, child) 注册三个阶段的回调函数：

  • prepare： fork 之前调用，负责按正确顺序获取所有互斥量，防止 fork 时锁处于中间状态。
  • parent： fork 之后在父进程中调用，释放所有在 prepare 中获取的锁。
  • child： fork 之后在子进程中调用，重新初始化互斥量（而不是解锁），因为原始锁定线程已不存在。

6.2 exec() 的行为

exec() 替换进程的整个地址空间，包括所有线程——只有调用线程存活并成为新程序的主线程。Pthreads 要求所有线程相关的清理处理器在 exec 之前都被执行，资源得到正确释放。

6.3 信号（Signals）

信号在多线程环境中的行为极其复杂，是最容易犯错的领域之一：

  • 信号动作（Signal Actions）： 信号处理函数对整个进程有效，所有线程共享。但信号可以被单个线程的信号掩码屏蔽。
  • 信号掩码（Signal Masks）： 每个线程有独立的信号掩码。使用 pthread_sigmask()（而非 sigprocmask()）设置当前线程的掩码。
  • 推荐模式： 在主线程中屏蔽所有信号，创建一个专用信号线程，用 sigwait() 同步等待信号，然后在该线程中处理。这避免了信号处理函数的异步安全限制。
/* 推荐：专用信号处理线程 */ void *signal_thread(void *arg) { sigset_t sigset; int sig; sigemptyset(&sigset); sigaddset(&sigset, SIGINT); sigaddset(&sigset, SIGTERM); while (1) { sigwait(&sigset, &sig); /* 阻塞等待信号 */ handle_signal(sig); /* 普通函数，无异步安全限制 */ } }

6.4 标准 I/O（Stdio）的线程安全

POSIX 要求 printf、fgets 等标准 I/O 函数是线程安全的（内部使用锁）。但有时需要更细粒度的控制：

  • flockfile(FILE *fp) / funlockfile(FILE *fp)：手动锁定/解锁文件流，允许原子性地执行多个 I/O 操作。
  • getchar_unlocked() / putchar_unlocked()：不加锁的高性能版本，由调用者负责同步。

6.5 线程安全函数

POSIX 要求一批传统 C 库函数必须提供线程安全版本（带 _r 后缀）：

非线程安全版线程安全版问题根源
strtok()strtok_r()静态内部状态

7.1 扩展同步——屏障（Barriers）

屏障允许一组线程在某个执行点相互等待，直到所有线程都到达后，再一起继续执行。常见于并行计算（如矩阵运算的每一轮迭代结束后同步）。

读写锁优化了"读多写少"场景的并发性能：多个读线程可以同时持锁，写线程需要独占访问。

7.3 工作队列管理器（Work Queue Manager）

工作队列（workq.c/workq.h）是一个通用的线程池实现：主线程提交任务到队列，固定数量的工作线程从队列中取任务执行。工作线程在空闲超过2秒后自动退出，需要时再动态创建，实现弹性伸缩。

本节是工程实践的精华，讨论如何将非线程安全的旧代码改造为线程安全：

  • 改造库（Modifying Libraries）： 消除全局/静态变量，或用互斥量保护，提供 _r 版本。
  • 共存遗留库（Living with Legacy Libraries）： 当无法修改源码时，将对该库的所有调用串行化，使用一个专用互斥量（"库锁"）保护，或在专用线程中顺序执行。

这是全书最受读者推崇的章节，被誉为"多线程调试的圣经"。书中明言：最好的调试是永远不需要调试。

8.1 避免错误代码

① 避免依赖"线程惯性"（Thread Inertia）

永远不要假设刚刚创建的线程还没有运行，或者某个线程会在另一个线程之前先执行。调度是非确定性的，即使代码在某个平台运行正确，换个系统可能立即出错。

⚠️
反例： "我的 main 线程会在子线程访问数据之前初始化它" —— 这种假设在高负载或不同调度器下会崩溃。必须用同步原语（如屏障或信号量）强制保证执行顺序。

② 永远不要把线程竞争当赌注

程序正确性不能依赖于"线程A几乎总是先完成"。若代码不能在任意线程交错下正确运行，它就是有 bug 的代码。

③ 合作避免死锁

  • 始终按照固定的层级顺序获取多个互斥量。
  • 减少持有锁的时间（临界区尽量短）。
  • 避免在持有锁时调用回调函数或外部代码（可能引入未知的锁依赖）。

④ 谨防优先级反转

实时系统中使用互斥量必须考虑优先级反转。使用 PTHREAD_PRIO_INHERIT 或 PTHREAD_PRIO_PROTECT 属性的互斥量。

⑤ 永不跨谓词共享条件变量

不要用一个条件变量同时代表多个不同的谓词（条件）。否则 pthread_cond_broadcast 会唤醒所有等待线程，而它们各自检查不同的谓词，产生不可预测的行为和性能问题。每个逻辑条件应有独立的条件变量。

⑥ 栈溢出与共享内存破坏

每个线程都有独立的栈，默认大小通常为几 MB。深递归或大局部变量可能导致栈溢出，并悄无声息地覆盖相邻线程的栈——这类 bug 极难调试。需要大栈时应在线程属性中显式设置栈大小。

8.2 避免性能问题

① 警惕并发串行化（Concurrent Serialization）

如果所有线程大部分时间都在争抢同一把锁，多线程带来的并发优势将完全消失。表现为：线程数增加，但性能不提升甚至下降。解决：减小锁粒度，或用读写锁代替互斥量。

② 使用适当数量的互斥量

互斥量太少 → 竞争激烈，并发度低。互斥量太多 → 上下文切换开销大，死锁风险高，代码复杂。需要根据数据访问模式和实际测量来确定合理数量。

③ 永不争抢缓存行（Cache Lines）

当多个线程频繁写入位于同一 CPU 缓存行（通常 64 字节）的不同变量时，每次写入都会导致缓存行失效并在 CPU 之间传递，即伪共享（False Sharing）。这可能导致性能比单线程还差。

10.1 X/Open XSH5（UNIX98）扩展

X/Open 组织（后并入 The Open Group）在 UNIX98 规范中对 Pthreads 进行了扩展，Butenhof 本人深度参与了这项工作：

  • 互斥量类型（Mutex Type）： 定义了四种互斥量类型：
    • PTHREAD_MUTEX_NORMAL：普通，重复加锁会死锁。
    • PTHREAD_MUTEX_ERRORCHECK：检错，重复加锁返回错误（调试用）。
    • PTHREAD_MUTEX_RECURSIVE：递归，同一线程可多次加锁（需同等次数解锁）。
    • PTHREAD_MUTEX_DEFAULT：默认，行为由实现定义。
  • 设置并发级别： pthread_setconcurrency() 用于提示系统期望的并发级别（对多对少模型有意义）。
  • 栈守护大小（Stack Guard Size）： pthread_attr_setguardsize() 设置栈末尾的保护区域大小，用于检测栈溢出。
  • 并行 I/O： 对 AIO（异步 I/O）与线程的集成规范。

10.2 POSIX 1003.1j 草案

这个草案标准化了书中第7章提供的实现性原语，使其成为 POSIX 标准的正式组成部分：

  • 屏障（Barriers）： pthread_barrier_t、pthread_barrier_init/destroy/wait() 正式标准化。
  • 读写锁（Read-Write Locks）： pthread_rwlock_t、pthread_rwlock_rdlock/wrlock/unlock() 正式标准化（后纳入 POSIX 2001）。
  • 自旋锁（Spinlocks）： pthread_spinlock_t 用于高性能低延迟场景。
📌
历史印证： 书中提到的所有"未来"特性，最终都被纳入 POSIX.1-2001（又称 SUSv3），并在 POSIX.1-2008 中进一步完善。现代 Linux（glibc + NPTL）已全面支持这些特性。这证明了 Butenhof 作为标准制定者的深刻洞见。
