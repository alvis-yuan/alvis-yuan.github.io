01
调度概述与进程模型
Linux 调度的根本问题：稀缺资源的分配

调度器（Scheduler）是操作系统内核中最核心的子系统之一。它的根本职责是：在多个可运行进程之间，决定谁在哪个 CPU 上运行，运行多久。

1.1 为什么需要调度器

现代计算机系统中，CPU 核心数量远少于同时运行的进程数量。CPU 是"零和"资源——同一时刻，一个 CPU 只能运行一个任务。调度器的核心矛盾是：

  • 交互性（低延迟）vs 吞吐量（高 CPU 利用率）
  • 公平性（每个进程得到公平份额）vs 优先级（重要任务优先）
  • 实时性（硬截止期保证）vs 普通任务的灵活调度

1.2 进程的生命周期与状态

Linux 中每个进程（任务）用 task_struct 描述，调度器只关心处于 TASK_RUNNING（可运行）状态的任务。

图1-1：Linux 进程状态机（仅调度器可见 TASK_RUNNING 状态的进程）

1.3 Linux 调度器历史演进

版本调度器核心特性

第一性原理解读

调度的本质是资源分配的信息问题。 CPU 是时间维度上的稀缺资源。"谁该运行"这个问题，等价于"谁的等待代价最高"。从物理角度看，如果所有任务都同等重要，理想情况是 N 个任务各占 1/N 的 CPU——这正是 CFS 的出发点。历史上的各种启发式算法（判断是否交互性任务）本质上是在猜测用户意图；而 CFS 和 EEVDF 通过数学模型（虚拟时间）直接逼近理想状态，消灭了猜测。调度器的进化史，就是一部用精确数学替代模糊启发式的历史。

进程优先级与 Nice 值
Linux 双轨优先级体系详解

Linux 内核实现了两套独立的优先级体系：实时优先级和 普通进程 Nice 值，两者共同映射到内核内部的 prio 字段（范围 0–139）。

来源：kernel.org CFS 文档 参考书：Robert Love《Linux Kernel Development》第三版，第4章

2.1 优先级全景

图2-1：Linux 内核优先级全景（0最高，139最低）

2.2 Nice 值与权重映射

CFS 不直接使用 Nice 值进行调度，而是将其转换为权重（weight）。权重表采用每降低 1 个 Nice 级别、权重增加约 25% 的原则（公比约 1.25）：

// kernel/sched/core.c — Nice 值权重表（prio_to_weight）
static const int prio_to_weight[40] = {
 /* -20 */  88761, 71755, 56483, 46273, 36291,
 /* -15 */  29154, 23254, 18705, 14949, 11916,
 /* -10 */   9548,  7620,  6100,  4904,  3906,
 /*  -5 */   3121,  2501,  1991,  1586,  1277,
 /*   0 */   1024,   820,   655,   526,   423,
 /*   5 */    335,   272,   215,   172,   137,
 /*  10 */    110,    87,    70,    56,    45,
 /*  15 */     36,    29,    23,    18,    15,
};
核心原理 Nice 值 0 对应权重 1024。两个任务优先级相差 10 个 Nice 级别，CPU 时间比约为 10:1。具体地：vruntime 增量 = 实际运行时间 × (1024 / weight)，权重越高的任务 vruntime 增长越慢，从而获得更多 CPU 时间。

2.3 系统调用接口

系统调用功能适用策略
nice()调整当前进程 Nice 值SCHED_NORMAL
setpriority()设置进程 Nice 值SCHED_NORMAL
sched_setscheduler()设置调度策略和 RT 优先级所有策略
sched_setparam()设置实时优先级SCHED_FIFO/RR
sched_setattr()设置完整调度参数SCHED_DEADLINE / EEVDF
sched_getattr()读取调度参数（含动态 deadline）所有策略
第一性原理解读

优先级的本质是"时间资源的分配比例"。 Nice 值不是绝对的时间额度，而是相对比例系数。用公比 1.25 的等比数列作为权重映射，是因为人类对"优先级差距"的感知是对数的（韦伯定律）——每降低一个优先级感受到大约相同的性能下降。这种设计使得用户对 Nice 值调整的直觉符合实际效果。

Linux 调度器的模块化架构设计

Linux 内核从 2.6.23 开始引入调度类（Scheduling Classes）机制——这是一个可扩展的面向对象式调度器模块层次。调度核心（kernel/sched/core.c）不假设任何具体策略，而是通过 sched_class 结构体的函数指针调用各具体调度器。 官方文档

每个调度类通过 sched_class 结构体实现以下钩子函数：

3.3 __schedule() 调度主循环

// kernel/sched/core.c（简化版）
static void __sched notrace __schedule(int sched_mode)
{
    struct task_struct *prev, *next;
    struct rq *rq;

    rq = cpu_rq(smp_processor_id());
    prev = rq->curr;

    local_irq_disable();
    rq_lock(rq, &rf);
    update_rq_clock(rq);           // 更新运行队列时钟

    // 若任务主动放弃 CPU（阻塞），则 dequeue
    if (!(sched_mode & SM_MASK_PREEMPT) && prev->__state)
        try_to_block_task(rq, prev, &rf);

    // 核心：从最高优先级类开始，选择下一个任务
    next = pick_next_task(rq, prev, &rf);

    clear_tsk_need_resched(prev);   // 清除重调度标志

    if (likely(prev != next)) {
        context_switch(rq, prev, next, &rf);  // 执行上下文切换
    }
}
第一性原理解读

调度类是"策略与机制分离"原则的典范实现。 内核核心（机制）只知道如何选取最高优先级类中的最优任务，并执行切换；具体的"公平"或"实时"语义（策略）完全封装在各调度类中。这种设计让 Linux 能够在同一内核中同时运行硬实时任务（SCHED_DEADLINE）和普通交互任务（SCHED_NORMAL），且相互隔离——这是操作系统设计最重要的分层抽象之一。

04
CFS 完全公平调度器
虚拟运行时间与红黑树的精妙结合

CFS（Completely Fair Scheduler）由 Ingo Molnár 实现，于 Linux 2.6.23（2007）合入主线。其核心设计思想用一句话概括： 官方文档

CFS 核心理念（来源：Linux 官方文档） "CFS 基本上在真实硬件上模拟了一个'理想的、精确的多任务 CPU'。'理想多任务 CPU'是一个（不存在的）拥有 100% 物理算力、能以精确相同的速度并行运行每个任务的 CPU，每个任务以 1/nr_running 的速度运行。"

4.1 虚拟运行时间（vruntime）

由于真实 CPU 一次只能运行一个任务，CFS 引入了虚拟运行时间（vruntime）来模拟理想 CPU：

// 虚拟运行时间增量计算（kernel/sched/fair.c - update_curr）
delta_exec = now - curr->exec_start;         // 实际运行时间（纳秒）
vruntime += delta_exec * (NICE_0_LOAD / curr->load.weight);
// 权重越大（优先级越高），vruntime 增长越慢 → 获得更多 CPU 时间
图4-1：两个等权重任务的 vruntime 追踪示意

4.2 红黑树（RB-Tree）运行队列

CFS 用一棵以 vruntime 为 key 的红黑树组织所有可运行任务。调度决策极其简单：始终选最左节点（vruntime 最小的任务）。

图4-2：CFS 红黑树结构，蓝色节点为 pick_next_task 选中的最左（最小 vruntime）节点

4.3 调度延迟与最小粒度

CFS 保证每个可运行任务在调度延迟（sched_latency_ns）时间内至少运行一次。时间片 = 权重比例 × 调度延迟。

当任务数量过多时，CFS 引入最小粒度（min_granularity_ns）——每个任务至少运行 1ms，防止切换开销超过运行时间。

// /sys/kernel/debug/sched/ 关键调优参数（Linux 6.x）
base_slice_ns        // 基础时间片（默认约 3ms，旧版 latency_ns 默认 6ms）
min_granularity_ns   // 最小粒度（默认 0.75ms）
// 调整方向：
// → 降低 base_slice_ns：更低延迟，适合桌面/交互
// → 升高 base_slice_ns：更高吞吐，适合服务器/批处理

4.4 睡眠补偿与 min_vruntime

新任务或长期睡眠唤醒的任务，其 vruntime 会被设为 max(自身vruntime, min_vruntime)，避免"一觉醒来占满 CPU"的问题，同时给予一定唤醒优势（vruntime 会稍微减去一个时间片的惩罚量）。

第一性原理解读

vruntime 是"虚拟时间"的量化正义。 如果每个任务都应得到 1/N 的 CPU，那么一个理想系统中，所有任务的 vruntime 应该始终相等。CFS 的全部逻辑，就是尽量维持这个等式。红黑树是实现"找最亏欠的那个任务"的最优数据结构：插入/删除 O(log n)，查找最小值 O(1)（缓存 leftmost 指针）。

没有启发式，只有数学。 O(1) 调度器需要猜测"哪个任务是交互性的"，CFS 不需要——vruntime 自然地让 I/O 密集型任务（经常睡眠，vruntime 增长慢）在唤醒时获得更快响应，这不是特殊处理，而是公平性的自然结果。

EEVDF 调度器（Linux 6.6+）
最早合格虚拟截止期优先——CFS 的继任者
来源 EEVDF 于 1995 年在学术论文中首次提出，由 Peter Zijlstra 于 2023 年提议引入 Linux，并在内核 6.6 版本中开始过渡（2024 年作为新选项正式推出）。 官方文档

5.1 EEVDF 的核心改进

CFS 选择 vruntime 最小的任务，但这种简单策略无法区分"需要短时间片的延迟敏感任务"和"需要长时间片的批处理任务"。EEVDF 引入两个新概念：

  • lag（滞后值）：表示任务是否收到了应有的 CPU 时间份额。lag > 0 表示欠债（还没获得足够 CPU 时间），lag < 0 表示已超额使用。
  • 虚拟截止期（Virtual Deadline, VD）：类似实时调度中的截止期，让延迟敏感型任务通过设置短时间片来获得更高优先级。

5.3 睡眠任务的 lag 衰减

当任务睡眠时，它不会立即出队，而是被标记为"延迟出队"，其 lag 值随虚拟运行时间逐渐衰减归零。这防止了任务通过短暂睡眠来重置负 lag 值（即防止"系统套利"行为）。长期睡眠的任务最终 lag 会归零。

第一性原理解读

EEVDF 解决的是 CFS 的"单维度"问题。 CFS 的 vruntime 只追踪"公平性"（谁得到了更少的时间），但无法表达"紧迫性"（谁需要更快响应）。EEVDF 用 lag 衡量公平性，用 virtual deadline 衡量紧迫性，两者结合——先保证公平（lag≥0才eligible），再追求响应（选VD最早者）——从第一性原理上解决了"延迟敏感 + 公平"的二元矛盾，而不是像过去那样用启发式来猜测。

实时调度：SCHED_FIFO / SCHED_RR
硬优先级抢占的实时任务调度

Linux 实时调度器（kernel/sched/rt.c）实现了 POSIX 标准的两种实时调度策略： 官方文档

6.1 两种实时策略

策略行为使用场景
SCHED_FIFO 先来先服务，运行直到阻塞、主动让出或被更高优先级任务抢占。没有时间片限制。 硬实时任务、中断处理、驱动程序
SCHED_RR 与 FIFO 相同，但同优先级任务之间以时间片（默认 100ms）轮转。 同优先级实时任务共享 CPU

6.2 实时优先级队列

RT 调度器维护 100 个优先级队列（对应实时优先级 1–99），用位图快速找到最高优先级的非空队列，时间复杂度 O(1)：

图6-1：RT 调度器优先级位图 + FIFO 队列，始终选择最高优先级非空队列的队头任务

6.3 RT 限流（RT Throttling）

为防止实时任务饿死普通任务，Linux 引入 RT 限流机制：

// /proc/sys/kernel/
sched_rt_period_us  = 1000000   // 周期：1秒
sched_rt_runtime_us = 950000    // RT 任务最多使用 950ms/秒（95%）
// 剩余 5% 给 CFS 任务，防止死锁
// 设置为 -1 则禁用限流（危险！可能导致系统挂死）
警告 SCHED_FIFO 任务若出现死循环且没有 RT 限流，会完全饿死所有普通进程，可能导致系统无响应。建议在开发阶段保留 RT 限流。
第一性原理解读

实时调度的本质是"优先级即承诺"。 SCHED_FIFO/RR 的设计假设：实时任务知道自己要做什么，知道何时结束，并且相信高优先级任务比低优先级任务更重要。这种"信任模型"将调度责任从内核转移到了应用开发者——内核只负责"谁级别高谁先跑"，不做任何公平性判断。RT Throttling 是这种信任模型的安全网：承认开发者可能犯错，留一条"救命线"给系统管理员。

SCHED_DEADLINE 使用三个参数描述任务的时间需求：

7.2 CBS（恒定带宽服务器）算法

CBS 确保每个 SCHED_DEADLINE 任务得到且不超过其预定带宽，实现任务间的带宽隔离：

// 任务唤醒时，CBS 判断是否重置截止期：
if (remaining_runtime / (scheduling_deadline - now) > runtime / period):
    // 超额消耗，重置：
    scheduling_deadline = now + deadline
    remaining_runtime = runtime

// 任务运行时：remaining_runtime -= 运行时间
// 当 remaining_runtime <= 0：任务被"节流"（throttled）
// 直到 replenishment_time（= 旧 scheduling_deadline）时恢复

7.3 带宽准入控制

系统拒绝接受会导致超载的 SCHED_DEADLINE 任务：

// /proc/sys/kernel/
sched_rt_period_us  = 1000000  // 周期
sched_rt_runtime_us = 950000   // DEADLINE 任务最大总带宽（95%）
// 准入条件：Σ(runtime_i / period_i) ≤ N_cpu × 0.95
// 超出则 sched_setattr() 返回 EBUSY

7.4 GRUB 带宽回收

当任务提前完成（剩余时间不足），GRUB（Greedy Reclamation of Unused Bandwidth）算法允许其他任务回收这部分带宽，提高 CPU 利用率，同时不破坏实时保证。

任务状态含义带宽计入
ActiveContending可运行（正在竞争 CPU）计入 running_bw
ActiveNonContending刚阻塞，未到0-lag时间计入 running_bw
Inactive阻塞且超过0-lag时间带宽可被回收
第一性原理解读

SCHED_DEADLINE 是"合同式调度"。 任务向内核声明："我每 period 只需要 runtime 的 CPU 时间，且在 deadline 内完成"。内核在准入时验证合同可行性（带宽之和不超过1），执行时强制执行合同（CBS 节流），并允许多余资源被其他人使用（GRUB）。这种设计将实时调度从"相信高优先级"（SCHED_FIFO）提升到了"数学可证明的时间保证"——这是控制系统、音频/视频处理等硬实时场景的真正需要。

  2. arch_start_context_switch(prev)       // 架构相关准备

  // 地址空间切换（进程间切换）
  3. if (!next->mm)   // 内核线程：借用前任 mm
       next->active_mm = prev->active_mm;
     else             // 用户进程：切换 mm
       switch_mm_irqs_off(prev->active_mm, next->mm, next);

  // 寄存器/栈切换（架构汇编实现）
  4. switch_to(prev, next, prev);
     // x86: 保存/恢复通用寄存器、FPU状态（延迟恢复）
     // 切换内核栈指针（rsp）
     // 更新 TSS（任务状态段）

  5. finish_task_switch(prev)  // 清理，释放 rq 锁

8.4 上下文切换开销

典型开销（x86-64，现代 CPU） 同进程线程切换：约 1–2µs（无 TLB flush）；不同进程切换：约 3–10µs（含 TLB flush，视 PCID 支持情况）；NUMA 跨节点切换：额外数十µs（内存访问延迟增加）。
第一性原理解读

上下文切换的代价来自"状态的重建成本"。 CPU 执行一个任务时，处理器的所有状态（寄存器、TLB、L1/L2 cache、分支预测器状态）都是"属于"这个任务的。切换任务就是丢弃这一切，转而重建新任务的状态。这就是为什么频繁切换会降低性能——每次切换都需要"热身"时间。调度器的时间片设计，本质上是在"任务响应延迟"（时间片短）和"切换开销占比"（时间片长）之间找平衡点。

SMP 负载均衡与调度域
多核系统的任务分配与迁移策略
来源 kernel.org: sched-domains.rst Linux 官方调度域文档

9.1 调度域层次结构

Linux 的 SMP 负载均衡基于调度域（sched_domain）层次，自底向上反映 CPU 拓扑：

图9-1：调度域层次（SMT → SMP → NUMA），每层均衡策略不同

9.2 负载均衡触发机制

// kernel/sched/core.c
// 周期性负载均衡：每个 CPU 通过 sched_tick() 触发
sched_balance_trigger()        // 在 tick 中检查是否需要均衡
  → 软中断 SCHED_SOFTIRQ
  → sched_balance_softirq()
  → sched_balance_domains()   // 遍历调度域层次
  → sched_balance_rq()        // 核心均衡函数
      → 找最忙的调度组（busiest group）
      → 找最忙的运行队列（busiest rq）
      → 迁移任务（attach_task / detach_task）

9.3 idle 时均衡（新任务唤醒）

当 CPU 变空闲时，会立即触发 newidle_balance——从其他 CPU 拉取任务，而不等待周期性 tick。这是保证低延迟的关键机制。

9.4 CPU 亲和性（Affinity）

// 用户接口
sched_setaffinity(pid, cpusetsize, mask);  // 设置 CPU 亲和性掩码
sched_getaffinity(pid, cpusetsize, mask);  // 获取当前掩码

// 内核强制亲和性：
// p->cpus_mask 约束任务只能在特定 CPU 子集上运行
// SCHED_DEADLINE 任务不支持 sched_setaffinity，需通过 cpuset cgroup
第一性原理解读

负载均衡是"空间维度的调度"。 时间维度上，调度器决定"什么时候轮到谁运行"；空间维度上，负载均衡决定"哪个任务应该在哪个 CPU 上"。两个目标相互对立：理想均衡要求任务频繁迁移（但迁移有 cache miss 代价），理想局部性要求任务永远不迁移（但会导致负载不均）。调度域层次结构是这个矛盾的折中：SMT 兄弟间迁移代价低（共享 L1 cache），NUMA 跨节点迁移代价高——层次越高，均衡触发越稀疏，代价阈值越高。

10
组调度与 Cgroup CPU 控制
多层次公平与带宽限制
来源 官方文档：CFS 组调度扩展

10.1 组调度原理

普通 CFS 对每个任务公平，但在多用户场景下，用户 A 运行 10 个进程，用户 B 运行 1 个进程，B 只得到约 1/11 的 CPU——不公平。组调度允许将任务组成"组实体"，CFS 先在组之间公平分配，再在组内公平分配。

图10-1：组调度——先在用户组间公平分配（各50%），再在组内公平分配

10.2 CFS 带宽控制（CPU Throttling）

通过 cgroup v2 的 CPU 控制器，可以设置硬带宽上限：

# cgroup v2 CPU 控制
echo "50000 100000" > /sys/fs/cgroup/mygroup/cpu.max
# 格式：quota period（单位：微秒）
# 上例：每 100ms 周期内最多使用 50ms（50% CPU）

# cpu.weight：相对权重（替代 cpu.shares）
echo 200 > /sys/fs/cgroup/mygroup/cpu.weight   # 默认100，越大越优先

# 查看节流统计
cat /sys/fs/cgroup/mygroup/cpu.stat
# throttled_usec: 被节流的总时间（微秒）

10.3 CONFIG_FAIR_GROUP_SCHED 实现机制

启用组调度后，struct sched_entity 不仅代表单个任务，也可以代表整个 cgroup 的组实体。CFS 通过层次化的红黑树实现多级公平：

// 组调度挂载示例（官方文档）
mount -t cgroup -ocpu none /sys/fs/cgroup/cpu
mkdir /sys/fs/cgroup/cpu/multimedia  # 创建 multimedia 组
mkdir /sys/fs/cgroup/cpu/browser     # 创建 browser 组
echo 2048 > multimedia/cpu.shares    # multimedia 获 2倍份额
echo 1024 > browser/cpu.shares       # browser 获 1倍份额
echo $firefox_pid > browser/tasks
第一性原理解读

组调度是"公平性的层次化扩展"。 平等对待每个进程和平等对待每个用户是两种不同的公平定义。组调度通过引入"组实体"这一抽象层，让 CFS 的"最小 vruntime"原则在两个层次同时工作。这是"递归应用同一原则"的优雅设计——CFS 不需要知道组的内部结构，只需要将组实体视为普通任务进行调度，而组实体内部再次运行 CFS。

11
NUMA 感知调度
非统一内存访问架构下的调度优化

11.1 NUMA 拓扑与调度问题

在 NUMA 系统中，每个 CPU 有"本地"内存（访问延迟低，约 50–100ns）和"远端"内存（访问延迟高，约 100–300ns）。如果一个进程的内存页面在节点 0，但被调度到节点 1 的 CPU 上运行，每次内存访问都需要跨 NUMA 互联——严重影响性能。

图11-1：NUMA 跨节点内存访问问题与调度解决方案

11.2 NUMA 自动均衡（AutoNUMA）

Linux 的 NUMA 自动均衡（CONFIG_NUMA_BALANCING）通过以下机制将任务迁移到离其内存最近的节点：

  1. 页面扫描：定期将任务的内存页标记为 PROT_NONE（不可访问）
  2. 缺页 Fault 统计：当任务访问这些页时触发 fault，内核记录"哪个 CPU 访问了哪个节点的页面"
  3. 迁移决策：如果任务频繁访问远端节点的页，则迁移任务或迁移页面（取代价最小者）
// /proc/sys/kernel/numa_balancing 开关
echo 1 > /proc/sys/kernel/numa_balancing   // 启用 AutoNUMA

// /proc/sys/kernel/numa_balancing_*
numa_balancing_scan_period_min_ms   // 扫描最短周期（默认1000ms）
numa_balancing_scan_period_max_ms   // 扫描最长周期（默认60000ms）
numa_balancing_scan_size_mb         // 每次扫描大小（默认256MB）
第一性原理解读

NUMA 问题的本质是"计算与数据的距离"。 冯·诺伊曼架构的根本约束是：CPU 必须从内存取数据才能计算。在 NUMA 系统中，"距离"有了物理含义——跨越 QPI/UPI 总线的访问延迟是本地的 2–4 倍。AutoNUMA 的"按故障统计"机制是一种"懒惰求值"的工程智慧：不提前预测（难以精确），而是等到问题（缺页）发生时才收集证据，再做出统计驱动的迁移决策。

12.2 观测工具

# 查看进程调度信息
cat /proc/$PID/sched        # 调度统计（等待时间、切换次数等）

# 修改调度策略

12.4 典型调优场景

场景策略
低延迟交互（桌面）降低 base_slice_ns；使用 PREEMPT 内核
高吞吐批处理（服务器）升高 base_slice_ns；SCHED_BATCH 策略；关闭 NUMA balancing
音频/视频实时处理SCHED_FIFO 优先级 70–80；使用 PREEMPT_RT 内核
多租户隔离（容器/云）cgroup v2 cpu.max 限额 + cpu.weight 相对权重
工业控制硬实时SCHED_DEADLINE + PREEMPT_RT + CPU 隔离（isolcpus=）
第一性原理解读

调优是"让调度器的假设与你的工作负载匹配"。 内核默认参数针对"普通桌面工作负载"优化。任何偏离这个假设的场景——延迟比吞吐更重要、某个任务比其他任务更关键、任务的内存访问模式有 NUMA 局部性——都需要显式地告诉调度器。工具的核心价值是"量化问题"：在调优前，先用 perf/BPF 测量当前延迟分布，再针对瓶颈调整参数，最后验证改善效果。没有测量的优化是盲目的。
