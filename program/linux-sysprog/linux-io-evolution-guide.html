Chapter 00
I/O 演进总览
从阻塞到零拷贝的五十年历程

Linux I/O 子系统的演进是操作系统设计领域最典型的「以问题驱动创新」的案例。每一代新机制的诞生，都是为了解决前一代在吞吐量、延迟或 CPU 效率上的根本瓶颈。

Linux I/O 演进时间线

Unix 早期 / POSIX
阻塞 I/O — read()/write() 调用阻塞进程直到完成，最简单的模型
4.2BSD 1983
select() 首次出现，多路复用第一代，FD_SETSIZE=1024 限制
SVR3 1986
poll() — 突破 1024 FD 限制，用结构数组代替位图
Linux 2.2 (1999)
sendfile() — 首个零拷贝系统调用，file→socket 无需用户态中转
Linux 2.5 (2002)
epoll (kernel 2.5.44) + Linux 原生 AIO — O(1) 事件通知 + 真正内核态异步
Linux 2.6.17 (2006)
splice / tee / vmsplice — 通用零拷贝原语，sendfile 底层实现
Linux 5.1 (2019)
io_uring — 颠覆性创新，共享环形队列彻底消除系统调用开销，真正零拷贝 + 真异步
第一性原理 — I/O 演进的本质

I/O 操作的根本问题在于：CPU 速度与外设速度之间存在数量级差异（CPU ns 级 vs 磁盘 ms 级，网络 μs~ms 级）。所有 I/O 演进的本质都是在回答同一个问题：「当数据尚未就绪时，CPU 应该做什么？」

答案的演进：①等（阻塞）→ ②轮询（非阻塞）→ ③委托内核通知（多路复用）→ ④完全委托内核执行（异步 I/O）→ ⑤消除通信本身的开销（零拷贝 + io_uring）。

Chapter 01
阻塞 I/O（Blocking I/O）
POSIX.1 / SVr4 / 4.3BSD — 最原始、最简单的模型
📄 man2/read(2) 📄 man2/write(2) 📄 man2/open(2)

原理

当进程调用 read()/write() 等系统调用时，若数据尚未就绪（如网络数据未到达，磁盘 I/O 未完成），内核将进程放入等待队列并切换到其他进程运行，直到 I/O 完成后再唤醒该进程。整个等待过程中进程处于 TASK_INTERRUPTIBLE 睡眠状态，不消耗 CPU。

✅ 优点
  • 编程模型极其简单，符合线性思维
  • 等待期间 CPU 完全释放，不浪费算力
  • 内核实现成熟，性能可预测
  • 天然串行，无并发竞争问题
❌ 缺点
  • 一连接一线程，高并发时线程开销巨大
  • 单线程无法同时处理多个 I/O
  • 线程切换、内存（每线程栈 ~8MB）开销高
  • C10K 问题：万级并发无法用此模型

适用场景

低并发、单连接脚本（日志处理、文件转换）；多进程/多线程服务器（Apache prefork 模式）；连接数固定且少（如数据库连接池内部单连接通信）。

示例代码

/* 阻塞 I/O 最简读取 — man2/read(2) POSIX.1-2008 */
#include <unistd.h>
#include <fcntl.h>

int main(void) {
    char buf[4096];
    int  fd = open("/etc/hostname", O_RDONLY);  /* 默认阻塞模式 */

    /* 若数据未就绪，进程在此睡眠直到内核唤醒 */
    ssize_t n = read(fd, buf, sizeof(buf));

    /* 到达此处时数据已从内核页缓存拷贝到 buf */
    write(STDOUT_FILENO, buf, n);
    close(fd);
    return 0;
}
第一性原理

阻塞 I/O 的本质是「时间换空间」：用进程的等待时间换取编程简单性。当 I/O 速度远慢于 CPU 时，这是对 CPU 资源的浪费。其核心矛盾是：「一个进程一次只能等待一件事」，与服务器需要同时服务大量客户端的需求天然冲突。

Chapter 02
非阻塞 I/O（O_NONBLOCK）
POSIX.1 — 调用立即返回，通过轮询检测就绪
📄 man2/fcntl(2) 📄 man2/open(2) O_NONBLOCK

原理

通过 fcntl(fd, F_SETFL, O_NONBLOCK) 将文件描述符设置为非阻塞模式。此后，read()/write() 在数据未就绪时立即返回 -1 并设置 errno = EAGAIN（或 EWOULDBLOCK），而非挂起进程。应用需自行轮询重试。

注意：Linux man 页指出，POSIX.1-2001 允许 EAGAIN 或 EWOULDBLOCK 两种错误码均可返回，可移植程序应同时检查两者。

设置非阻塞的两种方式

/* 方式 1：open 时设置 */
int fd = open("/dev/ttyS0", O_RDWR | O_NONBLOCK);

/* 方式 2：对已存在 fd 用 fcntl 设置 — man2/fcntl(2) */
int flags = fcntl(fd, F_GETFL, 0);
fcntl(fd, F_SETFL, flags | O_NONBLOCK);

/* 非阻塞读取循环 */
char buf[4096];
for (;;) {
    ssize_t n = read(fd, buf, sizeof(buf));
    if (n > 0) {
        handle_data(buf, n);
        break;
    }
    if (n < 0 && (errno == EAGAIN || errno == EWOULDBLOCK)) {
        /* 数据未就绪，稍后重试 — 此处 CPU 空转！ */
        nanosleep(&(struct timespec){0, 1000}, NULL);
        continue;
    }
    break; /* 真正的错误 */
}
✅ 优点
  • 单线程可管理多个 FD（手动轮询）
  • 不会永久阻塞，可做超时控制
  • 与 I/O 多路复用配合使用效果极佳
❌ 缺点
  • 忙轮询浪费 CPU（如不加 sleep）
  • 轮询间隔难以把握：太短耗 CPU，太长增延迟
  • 单独使用很少见，通常配合 select/poll/epoll
第一性原理

非阻塞 I/O 将「等待」的职责从内核转移给应用程序。进程不再被动睡眠，而是主动反复询问。这解决了「单进程无法等待多 FD」的问题，但带来了 CPU 空转浪费。其本质矛盾是：「应用不知道何时数据会就绪」——这个矛盾催生了 I/O 多路复用机制。

Chapter 03
select — I/O 多路复用第一代
POSIX.1-2001 / 4.2BSD — 同步 I/O 多路复用
📄 man2/select(2) POSIX.1-2024 4.2BSD 起源

原理

来自 man2/select(2)：「select() allows a program to monitor multiple file descriptors, waiting until one or more of the file descriptors become "ready" for some class of I/O operation.」

select() 接受三个 fd_set 位图（读/写/异常），内核遍历所有被监控的 FD（O(n) 线性扫描），有就绪 FD 时返回，进程再遍历找出哪个 FD 就绪。每次调用都要重建位图，且有 FD_SETSIZE=1024 的硬性上限。

API 签名（来自 man2/select.2）

#include <sys/select.h>

int select(int nfds,               /* 最大FD+1 */
           fd_set *readfds,        /* 关注可读的FD集合 */
           fd_set *writefds,       /* 关注可写的FD集合 */
           fd_set *exceptfds,      /* 关注异常的FD集合 */
           struct timeval *timeout); /* NULL=永久等待 */

/* man2 警告：select() 只能监控 < FD_SETSIZE (1024) 的FD！
   现代应用应改用 poll(2) 或 epoll(7) */

示例：监控 stdin（来自 man2/select.2 官方示例）

#include <stdio.h>
#include <sys/select.h>
#include <sys/time.h>

int main(void) {
    fd_set         rfds;
    struct timeval tv;
    int            retval;

    FD_ZERO(&rfds);
    FD_SET(0, &rfds);  /* 监控 stdin (fd=0) */

    tv.tv_sec  = 5;    /* 超时 5 秒 */
    tv.tv_usec = 0;

    retval = select(1, &rfds, NULL, NULL, &tv);
    /* 注意：返回后必须重建 fd_set，因为内核修改了它 */

    if (retval == -1)       perror("select()");
    else if (retval > 0)  printf("Data is available now.\n");
    else                   printf("No data within five seconds.\n");
    return 0;
}
⚠ select 的已知缺陷（来自 man2/select.2 BUGS 节）
  • FD_SETSIZE=1024：glibc 将 fd_set 固定为 1024 位，无法监控超过此值的 FD
  • 每次重建位图：fd_set 作为 value-result 参数被内核修改，每次调用必须重建，是设计错误
  • O(n) 线性扫描：内核遍历所有 FD，应用也要遍历找就绪者
  • 用户↔内核拷贝：每次调用都要将 fd_set 复制到内核
第一性原理

select 的核心创新是「将等待权交给内核」：应用不再自己轮询，而是告诉内核「帮我盯着这些 FD，有动静叫我」。这是从主动轮询到被动通知的关键跃迁。但 select 用位图表达 FD 集合，决定了其 O(n) 的扫描复杂度和 1024 的上限——这是数据结构选择不当导致的根本性缺陷。

Chapter 04
poll — 移除 FD 数量限制
POSIX.1-2001 / SVR3 — 改进型 I/O 多路复用
📄 man2/poll(2) POSIX.1-2001

原理

来自 man2/poll(2)：「poll() performs a similar task to select(2): it waits for one of a set of file descriptors to become ready to perform I/O.」

poll 用结构体数组替代三个位图，每个 pollfd 独立描述一个 FD 及其关心的事件。无 FD 数量上限，但仍是 O(n) 扫描，仍需每次将数组从用户态复制到内核态。

#include <poll.h>

/* man2/poll.2 中的结构体定义 */
struct pollfd {
    int   fd;       /* 文件描述符（负值表示忽略该项） */
    short events;  /* 关注的事件 (POLLIN|POLLOUT|POLLPRI...) */
    short revents; /* 实际发生的事件（内核填充）*/
};

int poll(struct pollfd *fds, nfds_t nfds, int timeout);
poll 关键事件标志
  • POLLIN — 有数据可读
  • POLLOUT — 可写入数据
  • POLLPRI — 紧急/带外数据
  • POLLRDHUP — 对端关闭连接（Linux 2.6.17+）
  • POLLERR — 错误（仅 revents）
  • POLLHUP — 挂起（仅 revents）
select vs poll 对比
FD 上限 select: 1024 poll: 无限
数据结构 3个位图 结构体数组
每次重建 是 revents重置即可
扫描复杂度 O(n) O(n)
内核拷贝 每次 每次

示例：多 FD 监控

#include <poll.h>
#include <stdio.h>

int main(void) {
    struct pollfd fds[2];

    fds[0].fd = STDIN_FILENO;
    fds[0].events = POLLIN;

    fds[1].fd = some_socket_fd;
    fds[1].events = POLLIN | POLLOUT;

    int ret = poll(fds, 2, 5000); /* 5000ms 超时 */
    if (ret > 0) {
        if (fds[0].revents & POLLIN)
            handle_stdin();
        if (fds[1].revents & POLLIN)
            handle_recv();
        /* 优点：fd 负值可临时禁用，revents 只需清零，无需完整重建 */
    }
    return 0;
}
ppoll()：Linux 特有的 poll 变体，增加了信号掩码参数（类似 pselect），可原子性地在等待时屏蔽信号，避免 TOCTOU 竞态条件。定义需 _GNU_SOURCE。
第一性原理

poll 解决了 select 的数据结构问题（位图→结构体），但没有解决其算法问题（O(n) 扫描）。当连接数增大到数万级时，O(n) 扫描成为瓶颈——即使只有 1 个 FD 就绪，内核也要扫描全部 n 个。这揭示了一个第一性原理：「通知机制的效率不应与被监控FD的总数成正比」——这正是 epoll 要解决的根本问题。

Chapter 05
epoll — Linux 专有高性能事件通知
Linux 2.5.44 (2002) — O(1) 就绪通知，C10K 解决方案
📄 man7/epoll(7) 📄 epoll_create(2) 📄 epoll_ctl(2) 📄 epoll_wait(2) Linux 专有

原理

来自 man7/epoll(7)：「The epoll API can be used either as an edge-triggered or a level-triggered interface and scales well to large numbers of watched file descriptors.」

epoll 在内核维护一个兴趣列表（通过红黑树索引，O(logn) 注册/删除）和一个就绪列表。当被监控的 FD 发生事件时，内核主动将其加入就绪列表。epoll_wait() 只返回就绪的 FD，无需遍历全部。

三个核心系统调用（来自 man7/epoll.7）

#include <sys/epoll.h>

/* 1. 创建 epoll 实例 — Linux 2.5.44 */
int epollfd = epoll_create1(0);  /* 推荐用 epoll_create1，传 EPOLL_CLOEXEC */

/* 2. 注册/修改/删除兴趣 FD */
struct epoll_event ev;
ev.events  = EPOLLIN | EPOLLET;  /* 边缘触发模式 */
ev.data.fd = conn_fd;
epoll_ctl(epollfd, EPOLL_CTL_ADD, conn_fd, &ev);

/* 3. 等待事件 — 返回就绪 FD 数量，只含就绪项 */
struct epoll_event events[64];
int n = epoll_wait(epollfd, events, 64, -1); /* -1=无限等待 */
for (int i = 0; i < n; i++)
    do_use_fd(events[i].data.fd); /* 仅处理就绪的 FD，O(就绪数) */

边缘触发 vs 水平触发（来自 man7/epoll.7）

🔵 水平触发 LT（默认）

只要缓冲区还有数据，每次 epoll_wait 都会通知。类似 poll 的语义，更安全但略低效。

适合：大多数网络应用，编程更简单
⚡ 边缘触发 ET（EPOLLET）

仅在状态变化时通知一次。必须配合非阻塞 FD，循环读直到 EAGAIN。漏读则永远得不到再次通知。

适合：高性能服务器，减少系统调用次数

完整 Echo Server 示例（来自 man7/epoll.7 官方示例扩展）

#define MAX_EVENTS 10
struct epoll_event ev, events[MAX_EVENTS];
int listen_sock, conn_sock, nfds, epollfd;

epollfd = epoll_create1(0);

ev.events = EPOLLIN;
ev.data.fd = listen_sock;
epoll_ctl(epollfd, EPOLL_CTL_ADD, listen_sock, &ev);

for (;;) {
    nfds = epoll_wait(epollfd, events, MAX_EVENTS, -1);

    for (int n = 0; n < nfds; ++n) {
        if (events[n].data.fd == listen_sock) {
            conn_sock = accept(listen_sock,
                               (struct sockaddr *) &addr, &addrlen);
            setnonblocking(conn_sock);
            ev.events  = EPOLLIN | EPOLLET;  /* 边缘触发 */
            ev.data.fd = conn_sock;
            epoll_ctl(epollfd, EPOLL_CTL_ADD, conn_sock, &ev);
        } else {
            do_use_fd(events[n].data.fd);
        }
    }
}

指标selectpollepoll
FD 上限 1024 无限 无限 (/proc/sys/fs/epoll/maxuserwatches)

来自 man7/epoll.7 NOTES：类似机制在其他系统上的实现：FreeBSD 有 kqueue，Solaris 有 /dev/poll。epoll 是 Linux 专有接口（Linux 2.5.44，glibc 2.3.2）。
第一性原理

epoll 的本质突破是「反转通知方向」：select/poll 是应用定期询问内核「哪些 FD 就绪了？」，而 epoll 改为内核主动将就绪 FD 推入就绪列表。这将问题复杂度从「与 FD 总数相关」降为「与就绪 FD 数相关」。这正是推模型（push）优于拉模型（pull）的经典体现——当大多数 FD 无活动时，epoll 的优势极为显著。

Chapter 06
POSIX AIO — 标准异步 I/O
POSIX.1-2001 / glibc 2.1 — 用户态线程池模拟异步
📄 man7/aio(7) 📄 aio_read(3) POSIX.1-2008

原理

来自 man7/aio(7)：「The POSIX asynchronous I/O (AIO) interface allows applications to initiate one or more I/O operations that are performed asynchronously (i.e., in the background). The application can elect to be notified of completion of the I/O operation in a variety of ways: by delivery of a signal, by instantiation of a thread, or no notification at all.」

重要限制（来自 man7/aio.7 NOTES）：「The current Linux POSIX AIO implementation is provided in user space by glibc. This has a number of limitations, most notably that maintaining multiple threads to perform I/O operations is expensive and scales poorly.」——Linux 上的 POSIX AIO 本质是用户态线程池模拟，并非真正的内核异步。

aiocb 结构体（来自 man7/aio.7）

#include <aio.h>

struct aiocb {
    int             aio_fildes;    /* 文件描述符 */
    off_t           aio_offset;    /* 文件偏移量 */
    volatile void  *aio_buf;       /* 数据缓冲区 */
    size_t          aio_nbytes;    /* 传输字节数 */
    int             aio_reqprio;   /* 请求优先级 */
    struct sigevent aio_sigevent;  /* 完成通知方式 */
    int             aio_lio_opcode;/* lio_listio 操作码 */
};

/* 通知方式 (aio_sigevent.sigev_notify): */
/* SIGEV_NONE    — 不通知 */
/* SIGEV_SIGNAL  — 发送信号 (aio_sigevent.sigev_signo) */
/* SIGEV_THREAD  — 创建新线程调用回调 */

int aio_read  (struct aiocb *aiocbp);   /* 异步读 */
int aio_write (struct aiocb *aiocbp);   /* 异步写 */
int aio_error (const struct aiocb *aiocbp); /* 查询状态 */
ssize_t aio_return(struct aiocb *aiocbp); /* 获取结果 */
int aio_cancel(int fd, struct aiocb *aiocbp); /* 取消 */
int lio_listio(int mode, struct aiocb *const list[], ...); /* 批量提交 */
来自 man7/aio.7 NOTES：POSIX AIO 在 Linux 上由 glibc 用户态实现，通过维护多个线程执行 I/O，代价昂贵且扩展性差。真正的内核态异步 I/O 请参考 io_submit(2) 系列（Linux 原生 AIO）。
第一性原理

POSIX AIO 提供了正确的编程模型抽象（提交即返回，完成时通知），但 Linux 上的实现是「挂羊头卖狗肉」——底层仍是线程阻塞 I/O。其本质困境是：标准先于实现，glibc 用最朴素的方式满足 API 语义。线程池模型意味着 N 个并发 I/O 需要 N 个线程，扩展性天花板极低。真正的解决方案需要在内核中实现异步路径。

Chapter 07
Linux 原生 AIO（内核态异步 I/O）
Linux 2.5 / Linux 专有 — io_setup/io_submit/io_getevents
📄 man2/io_setup(2) 📄 man2/io_submit(2) 📄 man2/io_getevents(2) Linux 专有

原理

Linux 原生 AIO（libaio）是真正的内核态异步 I/O，通过 io_setup、io_submit、io_getevents 等系统调用实现。内核在完成 I/O 后将结果放入完成队列，应用通过 io_getevents 批量收割结果，无需线程池。

限制：仅对 O_DIRECT（绕过页缓存）的文件 I/O 真正异步，对普通文件和网络 socket 支持有限。这是其最大的设计缺陷，也是 io_uring 诞生的直接原因。
#include <linux/aio_abi.h>  /* Linux 专有头文件 */
#include <sys/syscall.h>

/* 1. 创建 AIO 上下文（内核资源）*/
aio_context_t ctx = 0;
syscall(__NR_io_setup, 128, &ctx); /* 最大未完成请求数=128 */

/* 2. 构造 I/O 控制块 */
struct iocb cb = {
    .aio_fildes     = fd,
    .aio_lio_opcode = IOCB_CMD_PREAD,
    .aio_buf        = (uint64_t)buf,
    .aio_nbytes     = 4096,
    .aio_offset     = 0,
};
struct iocb *cbp = &cb;

/* 3. 提交请求 — 立即返回，内核异步执行 */
syscall(__NR_io_submit, ctx, 1, &cbp);

/* 继续做其他事情... */

/* 4. 收割完成事件 */
struct io_event events[128];
int n = syscall(__NR_io_getevents, ctx, 1, 128, events, NULL);
for (int i = 0; i < n; i++) {
    ssize_t res = events[i].res;  /* 返回值，类似 read() 的返回值 */
    process_result(res);
}
✅ 相比 POSIX AIO 的优势
  • 真正的内核态异步，无线程池开销
  • 支持批量提交（一次 io_submit 多个请求）
  • 对 O_DIRECT 文件 I/O 真异步
  • 事件完成队列可与 eventfd 配合 epoll 使用
❌ 主要缺陷
  • 只对 O_DIRECT 文件真正异步
  • 普通 socket/pipe 退化为同步
  • API 设计复杂，仅通过 syscall 访问
  • 内存对齐要求严格（O_DIRECT 需 512B 对齐）
  • io_uring 出现后已逐渐淘汰
第一性原理

Linux 原生 AIO 迈出了「在内核中实现真正异步路径」的关键一步，证明了不依赖线程池可以实现真异步。但其 O_DIRECT 的限制揭示了一个更深的问题：内核的页缓存（page cache）本身就会引入同步等待。彻底的异步需要内核对所有 I/O 路径（含页缓存回写、网络 socket 等）都提供统一的异步原语——这正是 io_uring 的设计目标。

Chapter 08
io_uring — 新一代异步 I/O
Linux 5.1 (2019) — 共享环形队列，真正零开销异步
📄 man7/io_uring(7) 📄 io_uring_setup(2) 📄 io_uring_enter(2) 🔗 liburing Linux 5.1+ Linux 专有

原理

来自 man7/io_uring(7)：「io_uring is a Linux-specific API for asynchronous I/O. It allows the user to submit one or more I/O requests, which are processed asynchronously without blocking the calling process. io_uring gets its name from ring buffers which are shared between user space and kernel space.」

io_uring 使用两个共享内存环形队列（通过 mmap 在用户态与内核态之间共享）：提交队列（SQ）和完成队列（CQ）。应用向 SQ 尾部写入 SQE（提交队列项），内核从 SQ 头部读取并执行，完成后将 CQE（完成队列项）写入 CQ 尾部，应用从 CQ 头部读取结果——整个过程可以完全不产生系统调用（SQ Polling 模式）。

核心数据结构（来自 man7/io_uring.7）

/* SQE：提交队列项 — 描述一个 I/O 操作 */
struct io_uring_sqe {
    __u8  opcode;     /* 操作类型：IORING_OP_READ/WRITE/ACCEPT... */
    __u8  flags;      /* IOSQE_IO_LINK: 串联请求，强制顺序执行 */
    __s32 fd;         /* 目标文件描述符 */
    __u64 off;        /* 文件偏移 */
    __u64 addr;       /* 缓冲区地址 */
    __u32 len;        /* 缓冲区长度 */
    __u64 user_data;  /* 透传到 CQE，用于关联请求与完成 */
    ...
};

/* CQE：完成队列项 — 记录 I/O 结果 */
struct io_uring_cqe {
    __u64 user_data;  /* 与 SQE 的 user_data 一致 */
    __s32 res;        /* 返回值（类似 read/write 返回值，负值为 -errno）*/
    __u32 flags;      /* CQE 标志（IORING_CQE_F_MORE 等）*/
};

使用 liburing 简化接口示例

#include <liburing.h>   /* https://github.com/axboe/liburing */

struct io_uring ring;
io_uring_queue_init(32, &ring, 0);  /* 队列深度 32 */

/* 提交读请求 */
struct io_uring_sqe *sqe = io_uring_get_sqe(&ring);
io_uring_prep_read(sqe, fd, buf, sizeof(buf), 0);
sqe->user_data = (uint64_t)1234;  /* 自定义标识符 */
io_uring_submit(&ring);           /* 一次系统调用批量提交所有 SQE */

/* 等待并收割完成 */
struct io_uring_cqe *cqe;
io_uring_wait_cqe(&ring, &cqe);
if (cqe->res > 0)
    process_data(buf, cqe->res);
io_uring_cqe_seen(&ring, cqe);   /* 标记 CQE 已消费，推进 CQ 头 */

io_uring_queue_exit(&ring);
🏆 io_uring 关键特性

零拷贝机制

  • SQ/CQ 通过 mmap 共享，无需系统调用传递数据
  • Fixed buffers：预注册缓冲区，避免每次 I/O 的内存映射

批量操作

  • 一次 io_uring_enter() 可提交任意数量 SQE
  • SQPOLL 模式：内核线程自动消费 SQE，无需系统调用

高级特性

  • IOSQE_IO_LINK：强制请求顺序执行
  • Multi-shot：单次注册，多次返回（如 accept_multishot）
  • Provided buffers：内核自动选择缓冲区
  • 零拷贝网络：MSG_ZEROCOPY + io_uring

支持的操作（Linux 6.x）

  • 文件 I/O、网络、poll、定时器、splice、accept、connect...

第一性原理

io_uring 的根本创新是「将用户态与内核态的通信介质从系统调用换成共享内存」。传统 I/O 的瓶颈之一是系统调用本身的开销（尤其在 Spectre/Meltdown 补丁后，内核/用户态切换开销倍增）。通过共享环形队列，io_uring 使应用和内核像两个线程共享一个无锁队列一样工作，彻底消除了通信开销。这是 I/O 演进史上「减少层次」的终极体现。

Chapter 09
mmap — 内存映射 I/O
POSIX.1 / 4.4BSD — 零拷贝文件访问
📄 man2/mmap(2) POSIX.1-2001

原理

mmap 将文件（或设备）的一段内容映射到进程的虚拟地址空间。进程直接通过指针访问该区域，内核按需（缺页中断）将页从文件加载到物理内存（页缓存），并建立虚拟地址→物理页的映射。读写操作直接操作页缓存，无需 read()/write() 的用户↔内核数据拷贝。

#include <sys/mman.h>
#include <fcntl.h>

int fd = open("data.bin", O_RDWR);
fstat(fd, &sb);

/* 将整个文件映射到虚拟地址空间 */
char *data = mmap(NULL, sb.st_size,
                   PROT_READ | PROT_WRITE,
                   MAP_SHARED,   /* 修改写回文件 */
                   fd, 0);

/* 直接通过指针访问，无需 read()/write() */
memcpy(output, data + offset, length); /* 仅1次拷贝（到用户buffer）*/
data[100] = 'X';  /* 直接写页缓存，稍后内核脏页回写 */

munmap(data, sb.st_size);
close(fd);
✅ 适用场景
  • 大文件随机访问（数据库文件、索引）
  • 文件内容共享（多进程共享同一页缓存）
  • 可执行文件加载（动态库 .so）
  • 内存映射 I/O 设备（GPU framebuffer）
❌ 注意事项
  • 小文件映射开销大（页表建立成本）
  • 映射区域过大会消耗虚拟地址空间
  • SIGBUS 风险：文件缩小后访问越界区域
  • 不能直接用于 socket（用 sendfile/splice）
第一性原理

mmap 的本质是「消除用户态缓冲区这个中间层」——直接让应用的虚拟地址指向内核的页缓存。这利用了现代 CPU 的 MMU（内存管理单元）：通过页表映射，两个「地址」可以指向同一块物理内存。零拷贝的代价是：你现在直接在内核页上操作，任何并发/同步问题都需要自行处理。

sendfile — 文件到 Socket 零拷贝
Linux 2.2 (1999) — 内核态直接传输，无用户态中转
📄 man2/sendfile(2) Linux 专有风格 Linux 2.2+

原理

sendfile() 在 Linux 2.2 引入，允许在内核内部将数据从文件直接传输到 socket，完全绕过用户态缓冲区。在支持 Scatter-Gather DMA 的网卡上，连内核中的数据拷贝也可避免：只需将页缓存的页引用（指针+偏移）传给网卡 DMA，由硬件直接读取。

#include <sys/sendfile.h>

/* sendfile(2) — Linux 2.2+ */
ssize_t sendfile(int out_fd,   /* 输出：socket 或文件（≥2.6.33）*/
                 int in_fd,    /* 输入：必须支持 mmap（普通文件）*/
                 off_t *offset,/* NULL=使用文件当前偏移 */
                 size_t count); /* 传输字节数 */

/* 实用示例：HTTP 静态文件服务 */
int file_fd   = open("/var/www/html/index.html", O_RDONLY);
int client_fd = /* accept() 得到的 socket */;

struct stat st;
fstat(file_fd, &st);

/* 一次调用，内核完成全部传输，无用户态拷贝 */
off_t offset = 0;
ssize_t sent = sendfile(client_fd, file_fd, &offset, st.st_size);
/* 循环发送直到完成... */
第一性原理

sendfile 解答了一个问题：「如果我不需要在用户态查看数据，为什么要把数据搬到用户态？」对于文件服务器，数据从磁盘到网卡的路径中，用户进程只是一个「搬运工」而非「处理者」。sendfile 通过直接在内核态完成搬运，消除了这个不必要的中间角色。这是对「不做不必要的工作」这一第一性原理的直接应用。

Chapter 11
splice / tee / vmsplice — 通用零拷贝原语
Linux 2.6.17 (2006) — 管道作为中介的零拷贝传输
📄 man2/splice(2) 📄 man2/tee(2) 📄 man2/vmsplice(2) Linux 专有 Linux 2.6.17+

原理

splice() 是 sendfile 的底层实现和推广版本，通过管道（pipe）作为中介，在任意两个文件描述符之间移动数据（只要其中至少一个是管道）。数据以页引用（page reference）而非数据拷贝的形式在内核页缓存中传递，实现真正零 CPU 拷贝。

#include <fcntl.h>  /* splice, tee, vmsplice — Linux 2.6.17+ */

ssize_t splice(int fd_in, loff_t *off_in,
               int fd_out, loff_t *off_out,
               size_t len,
               unsigned int flags);  /* SPLICE_F_MOVE|SPLICE_F_NONBLOCK */

ssize_t tee(int fd_in, int fd_out, size_t len, unsigned int flags);

ssize_t vmsplice(int fd, const struct iovec *iov,
                  unsigned long nr_segs, unsigned int flags);

/* 示例：零拷贝 file→socket（等价于 sendfile，但更灵活）*/
int pfd[2];
pipe(pfd);

/* 第一步：文件→管道（零拷贝，传递页引用）*/
splice(file_fd, NULL, pfd[1], NULL, file_size, SPLICE_F_MOVE);

/* 第二步：管道→socket（零拷贝）*/
splice(pfd[0], NULL, sock_fd, NULL, file_size, SPLICE_F_MOVE);

/* tee 示例：日志分叉，同时写到文件和标准输出 */
tee(pipe_rd, log_pipe_wr, len, 0);  /* 复制到日志管道，不消耗原数据 */
splice(pipe_rd, NULL, stdout_fd, NULL, len, 0); /* 原数据写 stdout */

第一性原理

splice/tee 揭示了一个重要原理：「数据移动不等于数据拷贝」。通过传递「指向物理内存页的引用」代替复制字节，内核可以在不同 I/O 路径之间「移动」数据，而 CPU 完全不接触数据本体。管道在此作为统一的「引用传递中介」——这是将内核内部的写时复制（CoW）机制应用于 I/O 的创新。代价是：用户端的 API 复杂度增加，且需要管道作为中间媒介。

选型决策树

Linux 专有 vs POSIX/UNIX 通用

🐧 Linux 专有（高性能，不可移植）
  • epoll — Linux 2.5.44，O(1) 事件通知
  • io_uring — Linux 5.1，新一代异步 I/O
  • Linux 原生 AIO — io_submit/io_getevents
  • sendfile — Linux 2.2（BSD 也有类似实现）
  • splice/tee/vmsplice — Linux 2.6.17
  • eventfd/timerfd/signalfd — Linux 特有文件描述符
🌐 POSIX / UNIX 通用（可移植）
  • read/write — POSIX.1，所有 UNIX 系统
  • O_NONBLOCK — POSIX.1，所有 UNIX 系统
  • select — POSIX.1-2001，4.2BSD
  • poll — POSIX.1-2001，SVR3
  • POSIX AIO — POSIX.1-2001（实现质量不一）
  • mmap — POSIX.1-2001，4.4BSD
  • 等价替代：kqueue(FreeBSD/macOS)，IOCP(Windows)
第一性原理 — 演进的终极目标

回顾整个 Linux I/O 演进史，本质是在不断消除三类浪费：

① 等待浪费（阻塞→非阻塞→多路复用→异步）：让 CPU 在 I/O 等待期间做有用的工作。

② 拷贝浪费（mmap→sendfile→splice→io_uring 零拷贝）：让数据尽可能少经过 CPU，直接在内存和外设之间流动。

③ 系统调用浪费（批量提交→SQPOLL）：减少用户态/内核态上下文切换，尤其在 Spectre/Meltdown 修复后，这一开销被放大。

io_uring 是目前最接近「零浪费」的方案——零拷贝 + 零系统调用 + 真异步，但代价是仅在 Linux 5.1+ 上可用。最终的第一性原理是：「最快的 I/O 是不做 I/O；其次是不浪费任何一个 CPU 周期在搬运数据上」。
