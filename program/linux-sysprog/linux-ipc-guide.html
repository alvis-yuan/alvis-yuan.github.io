概述

Linux C 进程间通信
完全技术指南

Linux 操作系统的核心哲学之一是：进程彼此独立，各自拥有私有的虚拟地址空间。 这种隔离保证了稳定性与安全性——一个进程崩溃不会波及另一个。然而，现实系统中，多个进程必须协同工作。 进程间通信（IPC, Inter-Process Communication） 正是打破这道"隔离墙"的结构化机制集合，是 Linux 并发编程的基石。

📄 pipe(7) - Linux man-pages 📄 unix(7) - Linux man-pages 📄 ipc_namespaces(7) - Linux man-pages 📚 kernel.org man-pages 📖 IPC Overview - CircuitLabs

为什么需要 IPC？

现代 Linux 系统中，功能越来越分布在多个独立进程中——这种模块化设计提升了稳定性、安全性与可维护性。 考虑一个安防摄像头系统：

图1 · 多进程协作场景示意

各进程间的数据传递、事件通知都依赖 IPC 机制。选择合适的 IPC 方式， 直接影响系统的性能、可靠性和开发复杂度。

第一性原理：进程隔离的本质是操作系统对物理内存的虚拟化——每个进程看到的地址空间是幻象。 IPC 的本质就是在操作系统层面有控制地"打洞"，让两个或多个进程能访问同一块物理资源 （内存、文件描述符、内核缓冲区等），同时由内核保证访问的正确性与安全性。 所有 IPC 机制的差异，本质上都是"打洞方式"的不同： 是让数据经过内核缓冲区复制（管道/消息队列），还是直接共享同一物理内存（共享内存），还是仅传递一个数字通知（信号）。

第一性原理：所有 IPC 的本质都是两个进程如何共享同一份信息。 差异仅在于：① 数据经不经过内核缓冲区（有无拷贝开销）；② 通信是单向还是双向； ③ 接收方是主动查询还是被动通知；④ 是否需要建立连接。 从这四个维度出发，就能理解为什么会有这么多 IPC 机制，它们彼此并不冗余。

机制详解

1匿名管道 Anonymous Pipe

📄 pipe(7) - man7.org 📄 pipe(2) - man7.org

原理

管道是 Unix/Linux 最古老、最简单的 IPC 形式。调用 pipe() 时，内核分配一段环形缓冲区 并返回两个文件描述符：pipefd[0]（读端）和 pipefd[1]（写端）。 数据按字节流（byte stream）的方式流动，无消息边界概念。

根据 Linux man-pages pipe(7)： 自 Linux 2.6.11 起，管道缓冲区默认为 16 页（65,536 字节）； 自 Linux 2.6.35 起，可通过 fcntl(F_SETPIPE_SZ) 动态调整大小（最大值由 /proc/sys/fs/pipe-max-size 控制，默认 1 MiB）。

PIPE_BUF 在 Linux 上为 4096 字节：写入量小于等于 PIPE_BUF 的操作是原子的；超过则可能被内核分片插入，需应用层自行处理边界。

图3 · 匿名管道工作原理

适用场景

  • 父进程 fork 出子进程后，向子进程传递数据（Shell 管道 cmd1 | cmd2 即此）
  • 进程链式数据处理管线（日志收集、数据变换）
  • popen()/pclose() 底层实现

优劣分析

✅ 优点
  • API 极简：read()/write()
  • 内核自动管理缓冲与同步，读/写阻塞语义直观
  • 无需显式同步，内核保证数据有序
  • 写端关闭后，读端返回 EOF
❌ 缺点
  • 只能用于有亲缘关系的进程（需 fork 继承 fd）
  • 单向（双向需两个管道）
  • 无消息边界，需应用层协议
  • 缓冲区有限（默认 64KB）

代码示例

#include <unistd.h>
#include <stdio.h>
#include <string.h>
#include <sys/wait.h>

int main(void) {
    int pipefd[2];
    char buf[64];

    if (pipe(pipefd) == -1) { perror("pipe"); return 1; }

    pid_t pid = fork();
    if (pid == 0) {
        /* 子进程：读端 */
        close(pipefd[1]);                        // 关闭未用写端
        ssize_t n = read(pipefd[0], buf, sizeof(buf));
        write(1, buf, n);                        // 输出到 stdout
        close(pipefd[0]);
    } else {
        /* 父进程：写端 */
        close(pipefd[0]);                        // 关闭未用读端
        const char *msg = "Hello from parent!\n";
        write(pipefd[1], msg, strlen(msg));
        close(pipefd[1]);                        // 触发子进程 EOF
        wait(NULL);
    }
    return 0;
}

第一性原理：匿名管道的本质是内核维护的一段有界环形字节缓冲区， 两端各持一个文件描述符。进程隔离了地址空间，但文件描述符可以通过 fork() 继承——这就是"亲缘关系限制"的根本原因。管道没有消息边界，因为它模型上是 字节流，就像 TCP 一样；读写的原子性也仅在 PIPE_BUF 范围内有效。

机制详解

2命名管道 FIFO（Named Pipe）

📄 fifo(7) - man7.org 📄 mkfifo(3) - man7.org

原理

命名管道（FIFO，First-In-First-Out）是匿名管道的扩展：它在文件系统中创建一个 特殊类型文件（ls -l 显示类型为 p），任意具有文件权限的 进程都可以打开它进行读写，而无需亲缘关系。 一旦两端都打开，I/O 语义与匿名管道完全相同（字节流、阻塞语义、65536B 缓冲区）。

⚠️ 阻塞特性（来源：man7 fifo(7)）

默认情况下，open(path, O_RDONLY) 会阻塞，直到有进程以写方式打开同一 FIFO； open(path, O_WRONLY) 同理。这是经典的"握手"行为，可以通过 O_NONBLOCK 旗标改变。

适用场景

  • 日志收集器：多个应用进程写入同一 FIFO，日志服务进程从中读取
  • Shell 脚本与 C 程序之间的数据传递
  • 传感器生产者与数据展示消费者解耦

优劣分析

✅ 优点
  • 突破亲缘关系限制，任意进程均可通信
  • API 与普通文件相同，易于上手
  • 文件系统路径作为命名空间，无需共享 key
  • 内核负责同步，天然线程安全写入（<PIPE_BUF）
❌ 缺点
  • 仍然单向（双向需两个 FIFO）
  • 无消息边界，仍是字节流
  • 数据被读取后消失，不能"广播"给多个读者
  • FIFO 文件需手动 unlink() 清理

代码示例

/* === writer.c === */
#include <fcntl.h>
#include <sys/stat.h>
#include <unistd.h>
#include <stdio.h>

#define FIFO_PATH "/tmp/demo.fifo"

int main(void) {
    mkfifo(FIFO_PATH, 0666);           // 创建 FIFO 文件
    int fd = open(FIFO_PATH, O_WRONLY); // 阻塞直到读者打开
    dprintf(fd, "Temperature: %.2f C\n", 22.45);
    close(fd);
    unlink(FIFO_PATH);                  // 清理 FIFO 文件
    return 0;
}

/* === reader.c === */
#include <fcntl.h>
#include <unistd.h>
#include <stdio.h>

int main(void) {
    int fd = open("/tmp/demo.fifo", O_RDONLY);
    char buf[128];
    ssize_t n = read(fd, buf, sizeof(buf));
    write(1, buf, n);
    close(fd);
    return 0;
}

第一性原理：FIFO 解决了匿名管道的"亲缘关系"约束。 它的关键洞察是：两个进程需要一个"共同知道的名字"来汇合。 文件系统路径就是这个名字——任何进程只要有路径和权限，就能找到同一个内核缓冲区。 FIFO 文件本身不存储数据（I/O 不涉及磁盘读写），它仅是内核 IPC 对象的一个"门牌号"。

机制详解

3信号 Signal

📄 signal(7) - man7.org 📄 sigaction(2) - man7.org

原理

信号是 Linux 开销最低的 IPC 形式。它本质上是一个软件中断： 内核向目标进程递送一个整数编号，进程在"安全点"异步执行预注册的信号处理函数， 或执行内核默认行为（终止、忽略、产生 core dump 等）。

信号不能传递实质性数据（除了 sigqueue() 可携带一个 sigval_t 联合体， 通常为一个整数或指针）。常用信号如 SIGTERM（终止请求）、SIGKILL（强制终止）、 SIGCHLD（子进程状态变更）、SIGUSR1/SIGUSR2（用户自定义）。

图4 · 信号递送流程

适用场景

  • 进程控制：优雅关闭（SIGTERM）、重载配置（SIGHUP）
  • 父子同步：父进程感知子进程退出（SIGCHLD）
  • 超时机制：alarm() + SIGALRM
  • 最低成本通知：进程A通知进程B"数据已就绪"

优劣分析

✅ 优点
  • 开销极低，无需内核缓冲区
  • 可跨进程，只需目标 PID
  • 可广播（kill(-pgid, sig) 发给进程组）
  • 与内核事件深度集成
❌ 缺点
  • 不能传递数据（仅通知）
  • 信号处理函数限制严格，只能调用 async-signal-safe 函数
  • 信号会丢失：多个同类信号可能合并为一个
  • 调试困难，竞争条件难以排查

代码示例

#include <signal.h>
#include <stdio.h>
#include <unistd.h>

/* 信号处理函数（必须为 async-signal-safe）*/
static volatile sig_atomic_t reload_flag = 0;

static void on_hup(int sig) {
    reload_flag = 1;             // 设置标志，主循环中处理
}

int main(void) {
    struct sigaction sa = { .sa_handler = on_hup };
    sigemptyset(&sa.sa_mask);
    sa.sa_flags = SA_RESTART;
    sigaction(SIGHUP, &sa, NULL); // 注册 SIGHUP 处理（配置重载）

    printf("PID=%d, send: kill -HUP %d\n", getpid(), getpid());
    while (1) {
        if (reload_flag) {
            reload_flag = 0;
            puts("[INFO] Config reloaded!");
        }
        pause();               // 挂起等待信号
    }
}

第一性原理：信号的本质是一个整数编号的异步递送， 它利用了操作系统任务调度的"上下文切换"时机。信号不是进程间通道，而是 内核向进程注入的一次强制执行跳转。正因如此它开销极低——无需任何缓冲区， 一次 kill() 系统调用即完成；但也因此无法携带数据，因为没有数据缓冲区可写。

机制详解

4消息队列 Message Queue

📄 mq_overview(7) - man7.org 📄 sysvipc(7) - man7.org 📄 msgget(2) - man7.org 📄 mq_send(3) - man7.org

原理

消息队列是内核维护的有界消息链表，每条消息是一块数据 + 一个类型/优先级。 与管道不同，消息队列保留消息边界（每次 send 对应一次 recv）。 Linux 支持两套 API：

System V 消息队列

msgget() / msgsnd() / msgrcv() / msgctl()

用整数 key 标识；支持按消息类型选择性接收；API 较繁琐，全局 IPC 空间

POSIX 消息队列（推荐）

mq_open() / mq_send() / mq_receive() / mq_close()

用 /name 路径标识（挂载于 /dev/mqueue）；支持优先级；可与 select/poll/epoll 集成；可注册异步通知

📝 POSIX 消息队列关键特性（来源：man7 mq_overview(7)）
  • 消息按优先级排序（高优先级先被接收）
  • 可通过 mq_notify() 注册信号或线程通知
  • 队列属性：mq_maxmsg（最大消息数）、mq_msgsize（最大消息大小）
  • 挂载路径：/proc/sys/fs/mqueue/ 可查阅和调整限制

适用场景

  • 命令/控制消息分发（传感器→处理器、控制器→执行器）
  • 多生产者、多消费者的任务调度
  • 进程间异步通信（无需两端同时运行）
  • 需要优先级的事件队列

优劣分析

✅ 优点
  • 内核保留消息边界，无需应用层协议
  • POSIX 版支持消息优先级
  • 发送/接收进程无需同时运行（持久性）
  • 可多进程同时读写
❌ 缺点
  • 每条消息仍需两次拷贝（用户→内核→用户）
  • 最大消息大小和队列容量有限制
  • System V API 繁琐，资源需手动清理
  • 吞吐量低于共享内存

代码示例（POSIX 消息队列）

/* 发送方 */
#include <mqueue.h>
#include <stdio.h>
#include <string.h>

int main(void) {
    struct mq_attr attr = { .mq_maxmsg = 10, .mq_msgsize = 128 };
    mqd_t mq = mq_open("/demo_mq", O_CREAT | O_WRONLY, 0666, &attr);
    if (mq == (mqd_t)-1) { perror("mq_open"); return 1; }

    const char *msg = "CMD:start";
    mq_send(mq, msg, strlen(msg)+1, 5); // 优先级 5
    mq_close(mq);
    return 0;
}

/* 接收方 */
#include <mqueue.h>
#include <stdio.h>

int main(void) {
    mqd_t mq = mq_open("/demo_mq", O_RDONLY);
    char buf[128];
    unsigned prio;
    mq_receive(mq, buf, sizeof(buf), &prio);
    printf("Received[prio=%u]: %s\n", prio, buf);
    mq_close(mq);
    mq_unlink("/demo_mq");           // 清理
    return 0;
}
// 编译：gcc -lrt sender.c -o sender

第一性原理：消息队列解决了管道的两个根本缺陷：无边界 + 无法异步。 它的本质是内核维护的一个有优先级的链表——每个节点是一条完整消息。 两个进程可以在不同时间操作同一个链表，内核负责同步。 代价是每次通信仍有两次数据拷贝（用户→内核→用户），这是它与共享内存的根本差距。

机制详解

5共享内存 Shared Memory

📄 shm_overview(7) - man7.org 📄 mmap(2) - man7.org 📄 shmget(2) - man7.org 📄 shm_open(3) - man7.org

原理

共享内存是性能最高的 IPC 机制。其核心原理是：将同一块物理内存页映射到多个进程的虚拟地址空间， 建立映射后，进程之间的数据交换只需普通的指针读写，完全零拷贝，无系统调用。

图5 · 共享内存映射原理

Linux 提供两套 API（推荐使用 POSIX 版本）：

System V（较旧）

shmget() / shmat() / shmdt() / shmctl()
整数 key 标识，用 ipcs 查看

POSIX（推荐）

shm_open() + mmap() + munmap() + shm_unlink()
对象挂载于 /dev/shm/

⚠️ 关键警告：必须配合同步机制

共享内存没有任何内置同步。多个进程并发访问同一内存会产生竞争条件（race condition）。 必须配合信号量（Semaphore）或互斥锁（Mutex）使用来保护临界区。

代码示例（POSIX 共享内存）

/* === writer.c === 编译：gcc -lrt writer.c -o writer */
#include <fcntl.h>
#include <sys/mman.h>
#include <sys/stat.h>
#include <unistd.h>
#include <string.h>
#include <stdio.h>

typedef struct { int seq; double value; } shared_t;

int main(void) {
    // 1. 创建共享内存对象（/dev/shm/demo_shm）
    int fd = shm_open("/demo_shm", O_CREAT | O_RDWR, 0666);
    ftruncate(fd, sizeof(shared_t));     // 设置大小

    // 2. 映射到进程地址空间
    shared_t *shm = mmap(NULL, sizeof(shared_t),
                          PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
    close(fd);

    // 3. 直接写（零拷贝）
    shm->seq   = 42;
    shm->value = 3.14159;
    printf("Written: seq=%d value=%.5f\n", shm->seq, shm->value);

    munmap(shm, sizeof(shared_t));
    return 0;
}

/* === reader.c === */
int main(void) {
    int fd = shm_open("/demo_shm", O_RDONLY, 0);
    shared_t *shm = mmap(NULL, sizeof(shared_t),
                          PROT_READ, MAP_SHARED, fd, 0);
    close(fd);
    printf("Read:    seq=%d value=%.5f\n", shm->seq, shm->value);
    munmap(shm, sizeof(shared_t));
    shm_unlink("/demo_shm");            // 最后使用者清理
    return 0;
}

第一性原理：共享内存打破了进程隔离的根本手段——让内核的页表（Page Table） 将同一物理页同时映射给两个进程。一旦映射建立，进程间通信就退化为 普通内存读写，完全绕过内核。这就是"最快"的来源。代价是： 内核不再介入，也就意味着没有任何自动同步——这是不可能"免费"的， 必须由开发者通过信号量等机制手动保证正确性。

机制详解

6信号量 Semaphore

📄 sem_overview(7) - man7.org 📄 sem_wait(3) - man7.org 📄 sem_open(3) - man7.org

原理

信号量是同步原语，本身不传递数据，而是协调多个进程对共享资源的访问。 信号量本质上是一个由内核管理的整数计数器，支持两种原子操作：

sem_wait() — P 操作（等待/获取锁）

将计数器减 1。若减后 < 0，则当前进程进入睡眠等待，直到其他进程执行 post。

sem_post() — V 操作（释放锁）

将计数器加 1。若有进程在等待，唤醒其中一个。

二值信号量（初始值为 1）等价于互斥锁（Mutex）：只允许一个进程进入临界区。 计数信号量（初始值 N）可以限制并发访问数量（如连接池）。

⚠️ 注意：fork() + sem_init() 的陷阱

sem_init(sem, 0, 1) 第二个参数为 0 表示线程间共享； 若要跨进程共享，必须将信号量置于共享内存中，且第二个参数传 1。 单独调用 sem_init() + fork()，父子进程将不会共享同一个信号量。

两套 API 对比

特性System V (semget)POSIX (sem_open / sem_init)
标识方式整数 key（ftok）/name 路径 或 内存地址
跨进程使用是sem_open（命名）或共享内存中的 sem_init
API 复杂度高（semop 数组操作）低（sem_wait/sem_post）
推荐程度遗留，不推荐推荐

代码示例（POSIX 命名信号量保护共享内存）

#include <semaphore.h>
#include <fcntl.h>
#include <stdio.h>

int main(void) {
    // 创建命名信号量，初始值 1（二值/互斥）
    sem_t *sem = sem_open("/demo_sem", O_CREAT, 0666, 1);

    /* --- 临界区保护 --- */
    sem_wait(sem);         // 获取锁（P 操作）

    // ... 访问共享资源 ...
    printf("在临界区内安全操作共享内存\n");

    sem_post(sem);         // 释放锁（V 操作）
    /* ------------------- */

    sem_close(sem);
    sem_unlink("/demo_sem");
    return 0;
}

第一性原理：信号量解决的本质问题是"先后顺序"与"互斥访问"。 两个进程同时对一块内存写入会产生不确定结果，根本原因是 CPU 指令是非原子的。 信号量利用内核提供的原子性（硬件 CAS 指令 + 内核调度）， 将"我正在用"这一信息原子地记录下来，让其他进程知道"等一下"。 信号量本身不传数据，它只传递一个许可权。

机制详解

7Unix 域套接字

📄 unix(7) - man7.org 📄 socket(2) - man7.org 📄 socketpair(2) - man7.org

原理

Unix 域套接字（Unix Domain Socket，UDS）使用 AF_UNIX（又名 AF_LOCAL）地址族。 与网络套接字不同，UDS不经过网络协议栈（无 TCP/IP 封装），数据直接在内核缓冲区之间传递， 因此远比 TCP loopback 高效。

根据 Linux man-pages unix(7)，支持三种套接字类型：

SOCK_STREAM

面向连接、可靠字节流。类比 TCP，但无网络开销。最常用。

SOCK_DGRAM

无连接数据报，保留消息边界，本地可靠（不像 UDP 可能丢包）。

SOCK_SEQPACKET（Linux 2.6.4+）

面向连接 + 保留消息边界。兼顾 STREAM 的可靠性和 DGRAM 的边界性，使用较少。

🌟 UDS 独特能力：传递文件描述符（SCM_RIGHTS）

通过 sendmsg() + 辅助数据（ancillary data），Unix 域套接字可以将一个进程打开的 文件描述符传给另一个进程（甚至跨用户），实现共享文件、socket、管道等。 这是其他任何 IPC 机制都无法做到的。

三种地址类型（来源 unix(7)）

  • 路径名（pathname）：绑定到文件系统路径（如 /tmp/app.sock），需手动 unlink()
  • 匿名（unnamed）：未绑定或由 socketpair() 创建，自动释放
  • 抽象命名空间（abstract）：sun_path[0]='\0'，Linux 专有，不依赖文件系统，自动清理

适用场景

  • 微服务/守护进程的C/S 架构（如 nginx、systemd、dbus 均使用 UDS）
  • 需要双向通信的多客户端场景
  • 需要传递文件描述符的特权隔离设计
  • 框架级 IPC（Qt、DBus、containerd 等）

优劣分析

✅ 优点
  • 全双工，支持多客户端并发
  • 可传递文件描述符（SCM_RIGHTS）
  • 可获取对端凭据（SO_PEERCRED）用于认证
  • API 与网络编程一致，可复用经验
  • 可用 select/poll/epoll 实现事件驱动
❌ 缺点
  • API 最复杂（需 socket/bind/listen/accept/connect）
  • 数据仍有拷贝（不如共享内存快）
  • 需手动管理套接字文件清理
  • 抽象命名空间为 Linux 专有，不可移植

代码示例（STREAM 服务器/客户端）

/* === server.c（来自 man7 unix(7) 示例） === */
#include <sys/socket.h>
#include <sys/un.h>
#include <stdio.h>
#include <unistd.h>
#include <stdlib.h>

#define SOCK_PATH "/tmp/demo.sock"

int main(void) {
    struct sockaddr_un addr = { .sun_family = AF_UNIX };
    strncpy(addr.sun_path, SOCK_PATH, sizeof(addr.sun_path)-1);

    int srv = socket(AF_UNIX, SOCK_STREAM, 0);
    unlink(SOCK_PATH);                             // 清除残留
    bind(srv, (struct sockaddr*)&addr, sizeof(addr));
    listen(srv, 5);
    printf("Server listening on %s\n", SOCK_PATH);

    while (1) {
        int cli = accept(srv, NULL, NULL);
        char buf[128];
        ssize_t n = read(cli, buf, sizeof(buf));
        write(cli, buf, n);                        // echo 回显
        close(cli);
    }
    unlink(SOCK_PATH);
    return 0;
}

/* === client.c === */
int main(void) {
    struct sockaddr_un addr = { .sun_family = AF_UNIX };
    strncpy(addr.sun_path, SOCK_PATH, sizeof(addr.sun_path)-1);

    int fd = socket(AF_UNIX, SOCK_STREAM, 0);
    connect(fd, (struct sockaddr*)&addr, sizeof(addr));
    write(fd, "Hello UDS!\n", 11);
    char buf[128];
    ssize_t n = read(fd, buf, sizeof(buf));
    write(1, buf, n);
    close(fd);
    return 0;
}

第一性原理：Unix 域套接字将网络通信的抽象模型 （连接、双向流、消息边界）应用于同机进程间通信，并通过消除网络协议栈来优化性能。 它的本质是：一对内核缓冲区 + 连接状态机，地址是文件系统路径而非 IP 端口。 内核知道双方都在本机，因此可以跳过序列化/校验/路由等网络开销， 同时保留网络编程的全部语义灵活性。

决策指南

决如何选择最优 IPC 方式

面对一个需要进程间通信的需求，按以下步骤逐层决策，可以找到最合适的方案。

图6 · IPC 选型决策流程图

逐步决策详解

1
明确通信目的：纯通知 vs 数据传输

若只需通知对方"某事发生了"而无需传递数据（如：进程重载配置、告知退出），优先选择 Signal。开销最低，无需缓冲区。若需传递数据，继续往下。

2
确认进程关系：有亲缘 vs 无亲缘

若通信双方是父子/兄弟进程（通过 fork() 衍生），且通信为单向流式数据，优先使用 匿名管道。API 极简，内核自动同步。若为无关进程，继续。

3
判断消息模型：字节流 vs 结构化消息

若通信是连续字节流（如：日志文本、音视频数据），且只需单向，用 FIFO（命名管道）即可。若每次通信是一个有边界的独立"包"（如：命令、传感器读数结构体），则需要消息队列或套接字。

4
评估性能需求：高吞吐 vs 一般吞吐

若需要传输大量数据（视频帧、雷达点云、大型共享状态），且对延迟/吞吐敏感，选 共享内存 + 信号量（零拷贝）。否则，消息队列或套接字的拷贝开销可接受。

5
判断架构模式：点对点 vs C/S 多客户端

若需要一个"服务进程"对多个"客户端进程"提供服务、需要双向通信、需要传递文件描述符（SCM_RIGHTS）或验证进程身份（SO_PEERCRED），选 Unix 域套接字。

6
默认推荐：POSIX 消息队列

若上述条件均不完全匹配（需要保留消息边界、支持优先级、点对点、中等性能需求），POSIX 消息队列是最佳默认选择：API 清晰、边界明确、支持优先级。

典型场景 推荐方案 理由

混合使用策略

在实际系统中，多种 IPC 机制往往配合使用，各司其职：

🏗️ 典型组合：共享内存 + 信号量 + Signal
  • 共享内存存放高频更新的大型数据结构（如传感器数据）
  • 信号量保护共享内存读写（互斥锁）
  • Signal（SIGUSR1）通知消费进程"新数据已就绪"（避免忙等轮询）
🏗️ 典型组合：Unix Socket + 共享内存
  • Unix 域套接字用于控制通道：传递命令、配置、文件描述符
  • 共享内存用于数据通道：大批量数据零拷贝传输
  • 此模式是许多高性能框架（如 DPDK、视频处理框架）的标准做法

第一性原理总结：IPC 选型的核心矛盾是性能与复杂度的权衡。 性能最高的共享内存，需要开发者承担全部同步责任；最简单的管道，只能用于有限场景。 决策的本质是： ① 先确定数据的"形态"（通知 / 字节流 / 有边界消息 / 大块数据）， ② 再确定通信的"拓扑"（父子 / 任意进程 / 一对多 / 多对多）， ③ 最后评估"性能约束"（能否接受拷贝开销）。 三步之后，最优方案通常已经自然浮现。不存在"万能"的 IPC—— 选择最小够用的机制，往往是最好的工程决策。
