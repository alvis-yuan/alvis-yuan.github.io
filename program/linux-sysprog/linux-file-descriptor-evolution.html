Linux 内核 · 系统编程

一切皆文件
文件描述符进化史

在 Linux 哲学中，"Everything is a file"。几乎所有系统事件——I/O 就绪、信号、定时器、进程退出——在底层都被抽象为文件描述符（File Descriptor，FD）的状态变化，并可通过统一的 read/write/poll/epoll 接口消费。

📖 Linux man-pages 官方文档  ·  🐧 kernel.org

进化总览

📊 历史演进时间线

从 1983 年 4.2BSD 引入 select，到 2019 年 Linux 5.1 引入 io_uring，I/O 通知机制历经四十余年演进，驱动力始终是：更高并发、更低延迟、更少系统调用开销。

1983 · 4.2BSD
select(2) — 同步 I/O 多路复用诞生
首次提供"等待多个 FD 就绪"的能力，但受限于 FD_SETSIZE=1024
1986 · SVR3 Unix / Linux 1997
poll(2) — 突破 1024 限制
使用动态数组替代位集合，支持任意数量 FD，但仍是 O(n) 线性扫描
2002 · Linux 2.5.44
epoll(7) — O(1) 内核事件驱动
内核维护兴趣列表与就绪列表，只返回就绪的 FD，彻底解决 C10K 问题
2007 · Linux 2.6.22–2.6.27
signalfd / timerfd / eventfd — FD 化万物
信号、定时器、计数器均变为可 poll 的 FD，彻底消除特殊 API
2005 · Linux 2.6.13
inotify(7) — 文件系统事件 FD
替代 dnotify，通过 FD 读取文件系统变更事件流
2019 · Linux 5.1
io_uring — 零系统调用异步 I/O
共享内存环形队列，批量提交与收割，接近零开销的完整异步框架
2019 · Linux 5.3
pidfd(2) — 进程 FD 化
用 FD 表示进程，彻底解决 PID 复用竞态，可 poll 等待进程退出

⚛️ 第一性原理

文件描述符的本质是内核对象的整数句柄。"一切皆文件"并非哲学口号，而是工程决策：通过统一的 struct file_operations（含 .poll 钩子）将任意内核资源接入同一套 I/O 通知机制，减少 API 碎片，实现组合性（composability）。每一次进化，都是在降低「告知应用程序'某事发生了'」所需的 CPU 周期与系统调用次数。

UNIX 标准 · POSIX.1-2024

🔵 select(2) UNIX/POSIX

由 4.2BSD（1983）首次引入，是最早的同步 I/O 多路复用系统调用。select 允许程序同时监视多个文件描述符，等待其中一个或多个变为"就绪"状态。

📖 man7.org — select(2)

原理

应用程序将要监视的 FD 分别填入三个位图集合（readfds / writefds / exceptfds），内核在每次调用时线性扫描从 0 到 nfds-1 的所有 FD，将就绪的 FD 留在集合中，其余清零后返回。

图2：select(2) 工作原理 — 位图线性扫描

适用场景

  • 需要 POSIX 可移植性（BSD、macOS、Solaris、Linux 均支持）
  • 监视的 FD 数量极少（< 10）且值均小于 1024
  • 嵌入式或极简环境

代码示例（来自 man7.org）

#include <sys/select.h>

fd_set rfds;
struct timeval tv;
int retval;

FD_ZERO(&rfds);
FD_SET(0, &rfds);   /* 监视 stdin (fd 0) */

tv.tv_sec  = 5;
tv.tv_usec = 0;

retval = select(1, &rfds, NULL, NULL, &tv);
/* 返回后 rfds 已被修改，需在循环中重新 FD_SET */
if (FD_ISSET(0, &rfds))
    printf("stdin 有数据可读\n");
⚠️ 官方警告（man 2 select）

WARNING: select() can monitor only file descriptors numbers that are less than FD_SETSIZE (1024)—an unreasonably low limit for many modern applications—and this limitation will not change. All modern applications should instead use poll(2) or epoll(7).

⚛️ 第一性原理

select 的核心矛盾：它把"描述哪些 FD 需要监视"和"返回哪些 FD 已就绪"混用同一个位图，导致每次循环都要重建位图，且内核必须扫描 0~nfds 整段空间（即使其中大量 FD 不存在）。根本缺陷是无状态性——内核无法在调用间记忆兴趣列表。

POSIX.1-2001

🟡 poll(2) POSIX

1986 年由 SVR3 Unix 引入，1997 年进入 Linux libc。poll 用结构体数组替代位图，彻底解除了 1024 的 FD 上限，并分离了"请求事件"与"返回事件"字段，无需每次重建。

📖 man7.org — poll(2)

原理

应用程序传入 struct pollfd 数组，内核遍历每个元素，调用对应 FD 的 f_op->poll()，将实际发生的事件写入 revents 字段。时间复杂度仍为 O(n)，但不再有位图边界问题。

图3：poll(2) 工作原理 — 稀疏结构体数组扫描

适用场景

  • POSIX 可移植且 FD 数量超过 1024
  • FD 集合变化频繁（无需重建，只修改对应元素）
  • 并发连接数中等（数百到数千），性能可接受

代码示例

#include <poll.h>

struct pollfd fds[2];
fds[0].fd     = STDIN_FILENO;
fds[0].events = POLLIN;
fds[1].fd     = sock_fd;
fds[1].events = POLLIN | POLLOUT;

int ret = poll(fds, 2, 5000); /* 5秒超时 */
if (ret > 0) {
    if (fds[0].revents & POLLIN)
        handle_stdin();
    if (fds[1].revents & POLLIN)
        handle_socket_read();
}

⚛️ 第一性原理

poll 修复了 select 的"位图边界"问题，但根本矛盾未变：每次系统调用都是无状态的——内核不记得上次监视了哪些 FD。应用程序每次都要把完整兴趣列表重新传给内核（O(n) 数据拷贝），内核也必须逐一检查（O(n) 扫描）。随着连接数增长，CPU 时间线性增加，这正是 C10K 问题的核心瓶颈。

Linux 专有 · Linux 2.5.44

🟢 epoll(7) Linux Only

2002 年随 Linux 2.5.44 合入内核，glibc 2.3.2 提供封装。epoll 是解决 C10K 问题的关键创新：内核维护状态，只通知就绪事件，代价从 O(n) 降至 O(就绪事件数)。

📖 man7.org — epoll(7)

核心概念

epoll 实例包含两个内核数据结构：

  • 兴趣列表（Interest List / epoll set）：用红黑树存储已注册的 FD 及其关注事件
  • 就绪列表（Ready List）：内核在 FD 就绪时主动将其加入，epoll_wait 直接读取
图4：epoll 内部结构 — 红黑树兴趣列表 + 双向链表就绪列表

水平触发 vs 边缘触发

📊 LT (Level-Triggered) vs ET (Edge-Triggered)
特性水平触发（LT，默认）边缘触发（ET，EPOLLET）
触发时机FD 处于就绪状态时反复通知仅在状态变化（从非就绪→就绪）时通知一次
编程难度低，类似 poll高，需循环读至 EAGAIN
性能中高，减少 epoll_wait 唤醒次数
阻塞 FD可使用必须使用 O_NONBLOCK
惊群问题存在（多线程场景）EPOLLET+EPOLLONESHOT 可解决

代码示例（来自 man7.org epoll(7)）

#define MAX_EVENTS 10
struct epoll_event ev, events[MAX_EVENTS];
int epollfd = epoll_create1(0);

ev.events  = EPOLLIN;
ev.data.fd = listen_sock;
epoll_ctl(epollfd, EPOLL_CTL_ADD, listen_sock, &ev);

for (;;) {
    int nfds = epoll_wait(epollfd, events, MAX_EVENTS, -1);
    for (int n = 0; n < nfds; ++n) {
        if (events[n].data.fd == listen_sock) {
            int conn = accept(listen_sock, ...);
            ev.events  = EPOLLIN | EPOLLET; /* 边缘触发 */
            ev.data.fd = conn;
            epoll_ctl(epollfd, EPOLL_CTL_ADD, conn, &ev);
        } else {
            do_use_fd(events[n].data.fd);
        }
    }
}
📌
来源：Linux man-pages 6.18 — epoll(7)，HISTORY: Linux 2.5.44. glibc 2.3.2. FreeBSD 有类似的 kqueue，Solaris 有 /dev/poll。

⚛️ 第一性原理

epoll 的革命在于引入有状态的内核端兴趣列表。内核不再每次从用户空间重新学习"你关注哪些 FD"，而是通过 epoll_ctl 的一次性注册，借助 file_operations.poll 钩子将等待队列回调挂载到目标 FD 上。当 FD 就绪时，内核回调主动将其加入就绪列表。epoll_wait 的开销仅为 O(就绪事件数)，与监视的 FD 总量无关——这是架构层面的突破。

Linux 专有 · Linux 2.6.22

📡 signalfd(2) Linux Only

Linux 2.6.22 引入，将 UNIX 信号转化为可读的文件描述符。传统信号处理需要信号处理函数（异步、受 async-signal-safe 约束），signalfd 使信号可以像普通 I/O 一样被 read()、poll()、epoll() 处理。

📖 man7.org — signalfd(2)

原理

调用 signalfd(-1, &mask, flags) 创建一个与信号集绑定的 FD。当 mask 中的信号 pending 时，该 FD 变为可读；read() 返回 struct signalfd_siginfo，信号随即被消费。通常需配合 sigprocmask 阻塞对应信号以防止默认处理。

图5：signalfd 将信号统一接入 epoll 事件循环

代码示例

#include <sys/signalfd.h>
#include <signal.h>

sigset_t mask;
sigemptyset(&mask);
sigaddset(&mask, SIGTERM);
sigaddset(&mask, SIGINT);

/* 阻塞信号，避免默认处理 */
sigprocmask(SIG_BLOCK, &mask, NULL);

int sfd = signalfd(-1, &mask, SFD_NONBLOCK | SFD_CLOEXEC);

/* 加入 epoll，像普通 FD 一样处理信号 */
struct signalfd_siginfo si;
read(sfd, &si, sizeof(si));
printf("收到信号 %d，来自 PID %d\n", si.ssi_signo, si.ssi_pid);
✅
优势：信号处理可在主事件循环中同步完成，无需 async-signal-safe 约束，可安全使用 malloc、printf 等函数。

⚛️ 第一性原理

信号的本质是内核向进程发送的异步通知。传统信号处理函数打破了程序的顺序执行流，引入大量并发陷阱。signalfd 的设计思路是：将异步事件序列化为同步可读的字节流，回归"一切皆文件"的统一抽象，让信号成为事件循环中与网络 I/O 等价的普通事件源。

Linux 专有 · Linux 2.6.25

⏱️ timerfd(2) Linux Only

Linux 2.6.25 / glibc 2.8 引入，通过文件描述符交付定时器到期通知。相较 setitimer(2)（基于信号 SIGALRM）或 timer_create(2)（POSIX 定时器），timerfd 的优势在于可被 select/poll/epoll 监视，与其他 I/O 事件无缝集成。

📖 man7.org — timerfd_create(2)

支持的时钟

  • CLOCK_REALTIME — 可调整的系统时钟
  • CLOCK_MONOTONIC — 单调时钟（不受 settimeofday 影响）
  • CLOCK_BOOTTIME — 含系统挂起时间（Linux 3.15+）
  • CLOCK_REALTIME_ALARM / CLOCK_BOOTTIME_ALARM — 可唤醒系统（Linux 3.11+）

代码示例（来自 man7.org）

#include <sys/timerfd.h>

int fd = timerfd_create(CLOCK_REALTIME, 0);

struct itimerspec new_value = {
    .it_value    = { .tv_sec = 3, .tv_nsec = 0 },  /* 首次 3 秒后到期 */
    .it_interval = { .tv_sec = 1, .tv_nsec = 0 },  /* 之后每 1 秒触发 */
};
timerfd_settime(fd, 0, &new_value, NULL);

/* read 返回自上次 read 以来的到期次数（uint64_t）*/
uint64_t expirations;
read(fd, &expirations, sizeof(expirations));
printf("定时器到期 %llu 次\n", expirations);

/* 可加入 epoll 统一处理 */
/* timerfd 支持 select(2), poll(2), epoll(7) */

⚛️ 第一性原理

时间本质上是一种资源状态：定时器从"未到期"变为"已到期"，这是一种状态转变。timerfd 将此转变映射为 FD 可读，使时间事件与 I/O 事件在形式上完全等价。应用程序无需维护两套事件循环（一套处理 I/O，一套处理定时器），统一性带来了架构简洁性。

Linux 专有 · Linux 2.6.22

🔔 eventfd(2) Linux Only

Linux 2.6.22 引入，创建一个内核管理的 uint64_t 计数器，可用于线程/进程间事件通知。比 pipe 更轻量（单个 FD vs 两个 FD），语义更清晰。

📖 man7.org — eventfd(2)

语义（来自官方文档）

  • write(fd, val, 8) — 将 val 加到计数器（原子操作）
  • read(fd, buf, 8) — 读取并清零计数器（返回当前值）
  • 计数器 > 0 时 FD 可读（POLLIN）；写不会溢出时 FD 可写（POLLOUT）
  • 支持 EFD_SEMAPHORE 标志：每次 read 仅减 1

代码示例

#include <sys/eventfd.h>

int efd = eventfd(0, EFD_NONBLOCK | EFD_CLOEXEC);

/* 生产者线程：触发事件 */
uint64_t val = 1;
write(efd, &val, sizeof(val));

/* 消费者：通过 epoll 感知，read 清零 */
uint64_t cnt;
read(efd, &cnt, sizeof(cnt));
printf("收到 %llu 次事件\n", cnt);
💡
典型用途：io_uring 使用 eventfd 通知完成事件；线程池向主线程报告任务完成；KVM 虚拟机 IRQ 注入（ioeventfd/irqfd）。

⚛️ 第一性原理

eventfd 是最纯粹的"事件 = 文件"实现：它没有实际数据，只有一个计数器。其存在意义是将任意软件事件（线程通知、中断注入）纳入 FD 体系，用最小的内核资源（一个计数器 + 等待队列）实现可 poll 的事件通知原语。

Linux 专有 · Linux 2.6.13

👁️ inotify(7) Linux Only

Linux 2.6.13 / glibc 2.4 引入，提供文件系统事件监控机制，替代老旧的 dnotify。通过 inotify FD 可以监视文件或目录的创建、修改、删除、重命名等操作。

📖 man7.org — inotify(7)

核心事件类型（来自官方文档）

事件含义适用
IN_ACCESS文件被访问（read/execve）文件/目录
IN_MODIFY文件被修改（write/truncate）文件
IN_ATTRIB元数据变化（权限/时间戳/链接数）文件/目录
IN_CREATE目录内创建文件/子目录目录内
IN_DELETE目录内删除文件/子目录目录内
IN_MOVED_FROM / IN_MOVED_TO重命名（通过 cookie 关联）目录内
IN_CLOSE_WRITE / IN_CLOSE_NOWRITE文件关闭文件/目录
IN_UNMOUNT包含被监视对象的文件系统被卸载通知性

代码示例（基于 man7.org inotify(7)）

#include <sys/inotify.h>
#include <poll.h>

int fd = inotify_init1(IN_NONBLOCK);
int wd = inotify_add_watch(fd, "/tmp", IN_CREATE | IN_DELETE | IN_MODIFY);

/* 通过 poll 等待事件（也可用 epoll） */
struct pollfd pfd = { .fd = fd, .events = POLLIN };
poll(&pfd, 1, -1);

char buf[4096];
ssize_t n = read(fd, buf, sizeof(buf));

struct inotify_event *ev = (struct inotify_event *) buf;
printf("事件 mask=0x%x，文件名: %s\n", ev->mask, ev->name);

inotify_rm_watch(fd, wd);
close(fd);
⚠️
局限性（官方文档）：基于 inode，不跟踪网络文件系统事件；不报告 mmap 引起的访问；目录监控非递归；事件队列可能溢出（IN_Q_OVERFLOW）。

⚛️ 第一性原理

文件系统操作本质上是内核 VFS 层的状态转变。inotify 在 VFS 钩子点插入回调，将这些转变序列化为 FD 上可读的结构体流。这再次验证了核心模式：任何异步事件源，只要能被内核感知，就能被 FD 化，从而纳入统一的事件驱动编程模型。

Linux 专有 · Linux 5.3

🏷️ pidfd(2) Linux 5.3+

Linux 5.3（2019）引入，通过 pidfd_open(2) 或 clone(2) CLONE_PIDFD 创建指向进程的文件描述符。彻底解决了 UNIX 进程管理的经典竞态：PID 在 waitpid 与 kill 之间可能被复用。

📖 man7.org — pidfd_open(2)

pidfd 的能力（来自官方文档）

  • pidfd_send_signal(2) — 安全发送信号（无 PID 复用竞态）
  • poll/select/epoll — 进程变为 zombie 时 FD 变为可读（EPOLLIN），被 reap 后产生 EPOLLHUP
  • waitid(2) P_PIDFD — 以 FD 方式等待子进程
  • pidfd_getfd(2) — 获取另一进程的 FD 副本
  • setns(2) — 进入目标进程的命名空间
  • process_madvise(2) — 对目标进程提供内存建议

代码示例（来自 man7.org）

#define _GNU_SOURCE
#include <poll.h>
#include <sys/syscall.h>

int pidfd = syscall(SYS_pidfd_open, target_pid, 0);

struct pollfd pfd = { .fd = pidfd, .events = POLLIN };
poll(&pfd, 1, -1);  /* 阻塞直到进程退出 */

if (pfd.revents & POLLIN)
    printf("目标进程已退出（zombie）\n");

close(pidfd);
🔒
PIDFD_THREAD（Linux 6.9）：可创建指向特定线程（而非整个进程）的 pidfd，当该线程退出时（即使线程组不为空）FD 即变为可读。

⚛️ 第一性原理

PID 是时间维度上的弱引用——进程消亡后 PID 可被复用，基于 PID 的操作存在 TOCTOU 竞态。pidfd 提供的是进程的强引用（内核对象引用计数保护），操作 pidfd 等价于操作一个特定进程实例，而非一个可变的数字标识符。这是"一切皆文件"在进程生命周期管理上的最后一块拼图。

Linux 专有 · Linux 2.6

⚙️ Linux AIO (libaio) Linux Only

Linux 2.6 引入的内核原生异步 I/O，通过 io_submit / io_getevents 接口提交和收割 I/O 请求。主要适用于以 O_DIRECT 打开的文件（绕过页缓存）。

📖 man7.org — io_submit(2)

局限性

  • 仅对 O_DIRECT 文件可靠异步；普通文件可能退化为同步执行
  • 不支持 socket、pipe 等非文件 FD
  • 每次提交仍需系统调用，批量能力有限
  • 接口设计不够通用，已被 io_uring 取代
⚠️
现状：Linux AIO 已基本被 io_uring 取代。io_uring 支持所有 Linux AIO 的使用场景，并额外支持 socket、pipe、以及更多操作类型。

⚛️ 第一性原理

Linux AIO 是一次半成功的尝试：它正确识别了"避免阻塞"的需求，但接口设计过于专用（O_DIRECT + 块设备），且每次操作仍有系统调用开销。根本问题是没有提供统一的提交-完成解耦机制，这一问题由 io_uring 的环形队列架构彻底解决。

Linux 专有 · Linux 5.1

🚀 io_uring Linux 5.1+

由 Jens Axboe 设计，2019 年随 Linux 5.1 合入内核。io_uring 通过用户空间与内核共享的环形队列实现近乎零系统调用的异步 I/O，是 Linux I/O 发展史上最重要的架构突破。

📖 man7.org — io_uring_setup(2)

IORING_SETUP_IOPOLL忙等待完成，适合 NVMe 等低延迟存储（需 O_DIRECT）
IOSQE_IO_LINK请求链，前一个完成才提交下一个
固定缓冲区/FD预注册缓冲区和 FD，减少每次 I/O 的 pin/unpin 开销
eventfd 集成可配合 eventfd 通知 epoll，接入已有事件循环

代码示例（使用 liburing）

#include <liburing.h>

struct io_uring ring;
io_uring_queue_init(32, &ring, 0);  /* 32 项 SQ/CQ */

/* 提交异步读请求 */
struct io_uring_sqe *sqe = io_uring_get_sqe(&ring);
io_uring_prep_read(sqe, fd, buf, 4096, 0);
sqe->user_data = 42;
io_uring_submit(&ring);  /* 可批量多个再 submit */

/* 等待完成 */
struct io_uring_cqe *cqe;
io_uring_wait_cqe(&ring, &cqe);
printf("读取 %d 字节，user_data=%llu\n", cqe->res, cqe->user_data);
io_uring_cqe_seen(&ring, cqe);

io_uring_queue_exit(&ring);

io_uring_setup(2) 官方描述："The submission and completion queues are shared between userspace and the kernel, which eliminates the need to copy data when initiating and completing I/O." — man7.org

⚛️ 第一性原理

传统 I/O 的本质开销来自两处：① 系统调用的用户-内核态切换；② 参数数据的复制（用户→内核）。io_uring 用共享内存环形队列同时消除了这两个开销：用户直接写 SQE 到共享内存，内核直接读；内核写 CQE 到共享内存，用户直接读。在 SQPOLL 模式下，提交甚至不需要系统调用。这是计算机体系结构层面最优的跨特权级通信方式。

🧠 总结：第一性原理解读

站在第一性原理的角度，整个 Linux 文件描述符进化史可以归纳为三个核心命题：

🔷 命题一：事件的本质是状态转变

无论是网络数据到达（socket readable）、定时器到期（timerfd readable）、信号投递（signalfd readable）、文件修改（inotify readable）还是进程退出（pidfd readable），这些"事件"在内核层面都是某个数据结构从状态 A 变为状态 B。Linux 通过 struct file_operations.poll 钩子，将任意状态转变映射为 FD 的"可读/可写"语义，实现统一抽象。

🔷 命题二：进化的驱动力是减少无效工作

select → poll：消除位图边界，但仍 O(n) 扫描。
poll → epoll：消除无效扫描，引入内核有状态兴趣列表，O(n)→O(就绪数)。
epoll → io_uring：消除系统调用本身，用共享内存环形队列实现零拷贝、零切换。
每一代进化，都在识别并消除上一代中不必要的 CPU 循环。

🔷 命题三：统一性带来组合性

"一切皆文件"的真正价值不在于形式上的统一，而在于组合性（Composability）：任何可以产生 FD 的机制，都可以与 epoll 组合；任何 epoll 能处理的 FD，都可以接入 io_uring。这种正交组合使得复杂系统（如数据库引擎、高性能网络框架）可以用极少的代码实现极高的性能，而无需为每种事件类型维护专用的通知通道。
