C/C++ 多线程与并发编程：工程实践指南

本文档提炼自关于 C/C++ 多线程架构、底层原理及工程防坑经验的深度对话。旨在帮助开发者建立起从底层硬件逻辑到上层架构决策的完整并发编程体系。

1. 资源保护：多把锁 vs 一把大锁

误区纠正：如果共享资源间没有逻辑依赖，强行规定顺序获取多把锁，并不会把不相干的资源串行化。相反，使用“一把全局大锁”才会导致性能断崖式下跌。

工程黑科技：按内存地址排序法防死锁
如果必须同时获取两把动态的锁且无法在业务上定序，最经典的解法是按锁的内存地址大小进行排序：强制先锁地址小的，再锁地址大的。这能 100% 杜绝 AB-BA 死锁。

即使只有两个线程，怎么选？

  • 选多把锁： 如果两个线程频繁交叉访问各自独立的资源，为了极致并行，必须分开加锁。
  • 选一把大锁： 如果操作极快（如更新计数器）且冲突概率低，遵循 KISS 原则，把它们打包进一个 struct 用一把大锁保护，彻底消灭死锁风险。

2. 多线程工程架构“黄金准则”

按优先级从高到低排列：

  1. 消灭共享： 优先使用局部变量、Thread-Local 或只读数据。
  2. 消息传递： “不要通过共享内存来通信，而应该通过通信来共享内存”，使用队列、管道替代直接共享。
  3. 简单类型用原子操作： 计数器、标志位使用 <stdatomic.h>。
  4. 复杂数据先用一把锁： 打包结构体，普通 Mutex 保护，遇到瓶颈再优化。
  5. 读写分离： 如果 Profiling 发现读多写少，换成读写锁（RWLock）。
  6. 细粒度锁： 迫不得已时才拆分多把锁，并严格制定防死锁规范。

3. 互斥锁 (Mutex) vs 信号量 (Semaphore)

核心原则：互斥锁是为了“保护”，信号量是为了“通信/调度”。

维度 互斥锁 (Mutex) 信号量 (Semaphore)
底层概念 更衣室的钥匙（只有进去的人能开门） 停车场的显示牌/发令枪
所有权机制 有。 谁加锁，必须由谁解锁。 无。 线程 A Wait，可以由线程 B Post。
适用场景 保护数据防篡改（同一内存的并发读写）。 控制并发数（限流）、线程先后顺序协调。
致命禁忌：千万不要用“初始值为1的二值信号量”代替互斥锁！
信号量没有所有权保护，代码写错时可能被其他线程意外 Post 强行开门，导致数据崩溃且极难调试。且持有信号量时线程崩溃，系统无法自动回收，会导致永久死锁。

4. 通知机制：信号量 vs 条件变量 (CondVar)

场景：线程 A 干完活，告诉线程 B 接着干。

  • 信号量（有记忆的投币机）： 即使 B 还没开始等，A 提前发出的信号也会被记录。绝不会丢失信号。适用于简单的流水线、一对一通知。
  • 条件变量（无记忆的大喇叭）： 必须配合 Mutex 和 while 循环使用。如果 A 广播时 B 没在听，信号彻底丢失。适用于复杂的业务条件判断（如队列满/空）、一对多群发广播。

5. 深度：互斥锁与“内存可见性”

互斥锁不仅提供“排队干活”的互斥机制，底层还会触发内存屏障（Memory Barrier），强制 CPU 刷新 L1/L2 缓存，保证内存可见性。

致命陷阱：只读操作不加锁
如果线程 A 加锁修改了数据，线程 B 为了性能“不加锁”去读，B 极大概率读到的是自己 CPU 核心里的旧缓存！为了读到最新数据，读操作也必须加锁（或使用原子操作）。

6. volatile 的真实身份

在 C/C++ 中，volatile 是一封写给编译器的信，告诉它不要把变量优化到寄存器里。它无法阻止 CPU 硬件缓存，绝对不能用于多线程并发同步！

三大必需使用场景（且不需要锁）：

  1. 硬件驱动开发： 读取映射到内存的硬件寄存器状态。
  2. 信号处理函数： 进程被异步中断打断时共享的标志位（如 volatile sig_atomic_t keep_running）。
  3. 异常跳转： setjmp / longjmp 跨函数跳转时保护局部变量不被寄存器回滚。

7. 三大并发安全级别

从宽松到苛刻，操作系统级别的安全定义：

  1. 线程安全 (Thread-Safe)： 可以多线程并发调用。内部允许使用任何锁、全局变量、堆内存。（绝大多数业务代码只需达到此标准）。
  2. 可重入 (Reentrant)： 执行一半被挂起，再次进入执行依然正确。严禁使用互斥锁、静态/全局变量。全靠参数和局部变量。
  3. 异步信号安全 (Async-Signal-Safe)： 最苛刻。即使在任何机器指令处被异步硬件/系统信号打断，也能安全执行。只能使用 POSIX 规定的极少数白名单系统调用（不能用 printf，不能用 malloc）。

8. 业务架构 API 选型决策树

架构级防弹准则 (The Self-Pipe Trick)
不要在信号处理函数 (signal handler) 中写业务逻辑。在里面只用 write() 写一个字节到管道/EventFD。主业务线程用 epoll 监听该管道，将“异步信号”降维成普通的“网络事件”，从而摆脱极度受限的 API 约束。
