引言 · 为何借鉴内核设计

Linux Kernel · include/linux/ · lib/

Linux 内核历经 30 余年、数千位开发者的打磨，沉淀出大量经过极端场景验证的设计模式与数据结构。这些设计追求极致的性能与零额外开销，同时兼顾代码的可维护性与复用性。应用层开发者虽然不需要直接使用内核 API，但其背后的设计思想完全可以移植到用户空间，用 C/C++、Go、Rust 等语言重新实现。

🚀 极致性能

内核代码必须运行在最苛刻的条件下，避免一切不必要的间接层，每个字节、每个 CPU 周期都计较

🧩 泛型无 void*

用侵入式结构替代 void* 指针，保留编译期类型检查，获得类型安全的泛型能力

🔒 并发安全

内核并发模型远比用户态复杂，其无锁或低锁设计可直接应用于多线程应用场景

♻️ 零拷贝思维

侵入式设计让节点直接嵌入对象，消除额外的内存分配，改善缓存局部性

📌 原文参考
Linux 内核代码托管于 torvalds/linux，核心头文件位于 include/linux/，实现文件位于 lib/。每个小节均附有对应的内核源文件路径与官方文档链接。
①

侵入式双向循环链表 list.h

C 语言 零额外开销 侵入式设计
📄 include/linux/list.h

传统实现 vs 内核实现

传统链表把数据放在节点内部（或用 void* 指针指向数据），内核的做法则相反——把链表节点 嵌入 到数据结构中。这是"侵入式链表"（Intrusive Linked List）设计。

图1：传统链表（额外内存分配）vs 内核侵入式循环双向链表（节点嵌入对象内部）

核心数据结构

来源：include/linux/list.h，引入于 Linux 2.1.45

/* include/linux/list.h */
struct list_head {
    struct list_head *next, *prev;
};

/* 用法：把 list_head 嵌入你的数据结构中 */
struct fox {
    unsigned long  tail_length; /* 尾巴长度（厘米）*/
    unsigned long  weight;       /* 体重（千克）*/
    bool           is_fantastic;
    struct list_head list;       /* 链表节点，嵌入此处 */
};

关键 API

/* 初始化链表头 */
INIT_LIST_HEAD(&red_fox->list);
static LIST_HEAD(fox_list);          /* 静态声明全局链表头 */

/* O(1) 插入 */
list_add(&f->list, &fox_list);        /* 头插（实现栈）*/
list_add_tail(&f->list, &fox_list);   /* 尾插（实现队列）*/

/* O(1) 删除 */
list_del(&f->list);
list_del_init(&f->list);             /* 删除后重置为空链表 */

/* 遍历 */
struct fox *curr;
list_for_each_entry(curr, &fox_list, list) {
    /* curr 是指向 struct fox 的指针，自动通过 container_of 反算 */
}

/* 安全删除版（允许遍历时删除当前节点）*/
struct fox *n;
list_for_each_entry_safe(curr, n, &fox_list, list) {
    list_del(&curr->list);
}

为什么不用 void* ？

《Linux 内核开发》（Robert Love，第三版）Chapter 6 解释了这一问题：使用 void* 意味着每个节点需额外存储一个指针并做一次间接寻址（pointer chasing）。侵入式设计把链表节点直接放在数据内部，访问数据只需一次 container_of 地址偏移计算，没有额外内存分配，缓存命中率更高。

💡 应用层借鉴
在 C 语言应用层，可以直接把 list.h 复制出来使用（去掉内核相关宏）。在 C++ 中可实现 IntrusiveList<T, MemberPtr>。Nginx 的 ngx_queue_t 就是对这一模式的直接借鉴。

红黑树 rbtree

自平衡 BST O(log n) 侵入式嵌入
📄 官方文档

来源：lib/rbtree.c + include/linux/rbtree.h，版权归属 Andrea Arcangeli（1999）。

📌 内核中的实际使用场景
根据内核官方文档（kernel.org/doc/html/v5.9/core-api/rbtree.html）：I/O 调度器（deadline、CFQ）用 rbtree 跟踪请求；高分辨率定时器用 rbtree 组织 timer；ext3 文件系统用 rbtree 跟踪目录项；VMA（虚拟内存区域）、epoll 文件描述符、加密密钥、网络包调度（hierarchical token bucket）均使用 rbtree。

核心结构

/* include/linux/rbtree.h */
struct rb_node {
    unsigned long  __rb_parent_color; /* 父指针 + 颜色位（低2位）*/
    struct rb_node *rb_right;
    struct rb_node *rb_left;
} __attribute__((aligned(sizeof(long))));

struct rb_root {
    struct rb_node *rb_node;  /* 根节点 */
};

/* 用法：rb_node 嵌入用户数据结构 */
struct mytype {
    struct rb_node  node;       /* 嵌入节点 */
    char           *keystring;  /* 用户数据 */
};
图2：内核红黑树结构 —— 颜色信息复用父指针低2位（节省内存）

设计亮点：颜色位复用

官方文档指出，__rb_parent_color 字段同时存储了父节点指针和颜色信息。由于 rb_node 按 sizeof(long) 对齐，指针的低2位始终为0，因此可以把颜色（RED=1, BLACK=0）存储在低位，这是非常经典的"指针标记技巧"（pointer tagging）。

搜索与插入（用户自定义比较）

内核 rbtree 不提供比较回调函数，而是让用户自己写搜索和插入函数，然后调用底层旋转/重着色 API。这样避免了函数指针调用开销，编译器可以内联展开比较逻辑。

/* 搜索 */
struct mytype *my_search(struct rb_root *root, char *string)
{
    struct rb_node *node = root->rb_node;
    while (node) {
        struct mytype *data = container_of(node, struct mytype, node);
        int result = strcmp(string, data->keystring);
        if      (result < 0) node = node->rb_left;
        else if (result > 0) node = node->rb_right;
        else                 return data;
    }
    return NULL;
}

/* 插入 */
int my_insert(struct rb_root *root, struct mytype *data)
{
    struct rb_node **new = &(root->rb_node), *parent = NULL;
    while (*new) {
        struct mytype *this = container_of(*new, struct mytype, node);
        int result = strcmp(data->keystring, this->keystring);
        parent = *new;
        if      (result < 0) new = &((*new)->rb_left);
        else if (result > 0) new = &((*new)->rb_right);
        else                 return FALSE;
    }
    rb_link_node(&data->node, parent, new);
    rb_insert_color(&data->node, root);   /* 自动旋转+重着色 */
    return TRUE;
}
💡 应用层借鉴
rbtree 适合需要有序遍历 + 快速查找的场景（如定时器堆、区间树、进程调度）。相比 STL std::map，侵入式 rbtree 无需额外内存分配。C++ 标准库的 std::set/map 内部就是红黑树，但接口不支持侵入式使用。

静态哈希表 hashtable.h

编译期确定大小 O(1) 均摊 hlist + hlist_head
📄 include/linux/hashtable.h

由 Sasha Levin 于 2012 年贡献（Linux 3.7），基于 hlist_head（单头双向链表）实现，桶的数量通过位数在编译期确定为 2 的幂次方，用位运算替代取模。

关键设计：2的幂次方桶 + 位运算取模

/* include/linux/hashtable.h */
#define DEFINE_HASHTABLE(name, bits)                         \
    struct hlist_head name[1 << (bits)] =                 \
        { [0 ... ((1 << (bits)) - 1)] = HLIST_HEAD_INIT }

/* bits=8 → 256个桶；bits=10 → 1024个桶 */
DEFINE_HASHTABLE(htable, 8);   /* 等价于 struct hlist_head htable[256]; */

/* 添加：哈希值自动映射到桶 */
#define hash_add(hashtable, node, key)                        \
    hlist_add_head(node, &hashtable[hash_min(key, HASH_BITS(hashtable))])

/* 遍历某个 key 对应桶中的所有元素 */
hash_for_each_possible(htable, obj, member, key) {
    /* 处理哈希冲突链中的每个元素 */
}

/* 遍历整个哈希表 */
int bkt;
hash_for_each(htable, bkt, obj, member) { ... }
图3：内核静态哈希表结构 —— hlist_head 桶 + hlist_node 侵入式节点 + 位运算取模

hlist_head 的巧妙之处

哈希表的桶头用 hlist_head（只有 *first 一个指针），而节点用 hlist_node（有 *next 和 **pprev 两个指针）。这样桶数组只占常规双向链表的一半内存，同时节点因为持有指向前驱 next 域的指针，仍然可以实现 O(1) 删除。

💡 应用层借鉴
固定大小的哈希表（编译期确定桶数）+ 侵入式 hlist 节点，非常适合嵌入式、游戏引擎、网络服务等不希望动态扩容的场景。LRU 缓存可以把这两个结构组合使用：哈希表负责 O(1) 查找，双向链表负责 O(1) 淘汰。

无锁环形队列 kfifo

单读单写无锁 位运算取模 Power-of-2 设计
📄 include/linux/kfifo.h

kfifo 是内核中 FIFO 环形缓冲区的实现，设计极为精妙：单写者+单读者场景下完全无锁（Single Producer Single Consumer，SPSC）。

核心结构与无锁原理

/* lib/kfifo.c 中的核心结构 */
struct __kfifo {
    unsigned int  in;     /* 写指针（只有生产者修改）*/
    unsigned int  out;    /* 读指针（只有消费者修改）*/
    unsigned int  mask;   /* size - 1（用于位运算取模）*/
    unsigned int  esize;  /* 每个元素大小 */
    void         *data;   /* 数据缓冲区（大小必须是 2^n）*/
};

/* 关键：in 和 out 永不回绕（unsigned 自然溢出），真实下标用 & mask */
unsigned int len  = fifo->in - fifo->out;        /* 当前数据量 */
unsigned int avail = (fifo->mask + 1) - len;     /* 可写空间 */
unsigned int idx  = fifo->in & fifo->mask;       /* 写入位置 */
图4：kfifo 环形缓冲区 —— in/out 指针永不回绕，位运算取模，SPSC 无锁

为什么 in/out 永不回绕？

内核注释（lib/kfifo.c）指出：in 和 out 使用 unsigned int 类型，利用 C 语言无符号整数的自然溢出特性——当 in 溢出从 0xFFFFFFFF 变为 0 时，in - out 的差值依然正确。这样避免了任何分支判断，使得环形缓冲区逻辑极为简洁。

📌 来自内核注释的锁说明
"Note about locking: There is no locking required until only one reader and one writer is using the fifo and no kfifo_reset() will be called. For multiple writers, only need to lock the writer. And vice versa for multiple readers."
💡 应用层借鉴
用 std::atomic<uint32_t>（C++11）替代 unsigned int，配合 memory_order_acquire/release，即可在应用层实现完全无锁的 SPSC 队列。这在音频流、日志系统、网络 I/O 等高性能场景中非常实用。

container_of 反向寻址宏

C 宏技巧 面向对象模拟 零运行时开销
📄 内核设计模式文档

container_of 是贯穿整个内核的核心宏，定义在 include/linux/container_of.h（旧版在 include/linux/kernel.h）。它是所有侵入式数据结构能够工作的基础。

宏定义与原理

/**
 * container_of - 从成员指针反推出包含它的结构体指针
 * @ptr:    指向成员的指针
 * @type:   包含该成员的结构体类型
 * @member: 成员在结构体中的名称
 */
#define container_of(ptr, type, member) ({                \
    const typeof( ((type *)0)->member ) *__mptr = (ptr); \
    (type *)( (char *)__mptr - offsetof(type, member) ); })

/* offsetof 原理：通过空指针（地址0）计算成员偏移量 */
#define offsetof(TYPE, MEMBER) ((size_t) &((TYPE *)0)->MEMBER)
图5：container_of 通过偏移量从成员指针反推出包含它的结构体起始地址

实际应用：驱动设计模式

内核官方文档（Design Patterns）描述了 container_of 在驱动中的标准用法：

struct foo {
    spinlock_t            lock;
    struct workqueue_struct *wq;
    struct work_struct     offload;  /* 嵌入工作队列节点 */
};

/* work 回调中，通过 container_of 拿回 foo 实例 */
static void foo_work(struct work_struct *work)
{
    struct foo *foo = container_of(work, struct foo, offload);
    /* 现在可以访问完整的 foo 结构体 */
}

/* 这是一种典型的 C 语言"面向对象"继承实现 */
🔑 核心设计哲学

container_of 是内核实现"面向对象继承"的基础工具。通过把"基类"嵌入"派生类"，然后用 container_of 从基类指针反推派生类指针，实现了 C 语言中的多态。内核中 kobject、work_struct、rb_node、list_head 等所有侵入式结构都依赖这个宏。

通知链 notifier_chain

发布-订阅 多类型锁保护 观察者模式
📄 Linux Insides

定义于 include/linux/notifier.h，是内核内部各子系统之间通信的标准机制，实现了经典的发布-订阅（Publish-Subscribe）模型。

核心数据结构

/* include/linux/notifier.h */
struct notifier_block {
    /* 回调函数：返回 NOTIFY_OK / NOTIFY_STOP / NOTIFY_DONE */
    int (*notifier_call)(struct notifier_block *, unsigned long, void *);
    struct notifier_block __rcu *next; /* 链表下一个订阅者 */
    int   priority;                    /* 优先级，高者先被调用 */
};

/* 四种通知链类型（按锁机制区分）*/
struct atomic_notifier_head   { spinlock_t lock;     ... }; /* 中断/原子上下文 */
struct blocking_notifier_head { rw_semaphore rwsem;  ... }; /* 进程上下文，可阻塞 */
struct raw_notifier_head      {                      ... }; /* 自定义锁 */
struct srcu_notifier_head     { srcu_struct srcu;    ... }; /* 可睡眠 RCU */
图6：内核通知链 —— 发布者触发，优先级有序链表，依次回调所有订阅者

实际使用示例（USB 通知）

/* drivers/usb/core/notify.c（发布者：USB Core）*/
static BLOCKING_NOTIFIER_HEAD(usb_notifier_list);

void usb_register_notify(struct notifier_block *nb) {
    blocking_notifier_chain_register(&usb_notifier_list, nb);
}

/* 订阅者模块注册 */
static int my_usb_notify(struct notifier_block *nb,
                          unsigned long action, void *data) {
    if (action == USB_DEVICE_ADD)
        handle_new_device(data);
    return NOTIFY_OK;
}

static struct notifier_block my_nb = { .notifier_call = my_usb_notify };
usb_register_notify(&my_nb);   /* 注册订阅 */
💡 应用层借鉴
通知链本质是带优先级的观察者链表 + 函数指针回调。在应用层可实现为事件总线（Event Bus）、插件钩子系统、中间件拦截链（类似 Express.js 中间件）。相比简单的函数指针数组，链表结构支持动态注册/注销，优先级字段控制执行顺序。

引用计数 kref

原子操作 RAII 思想 生命周期管理
📄 LWN.net kref

kref 定义于 include/linux/kref.h，由 Greg Kroah-Hartman 设计，是内核对象生命周期管理的标准工具。

结构与 API

/* include/linux/kref.h */
struct kref {
    refcount_t refcount;  /* 原子引用计数（底层 atomic_t）*/
};

/* 初始化：计数置为 1 */
void kref_init(struct kref *kref);

/* 增加引用：计数 +1 */
void kref_get(struct kref *kref);

/* 减少引用：计数 -1，若到 0 则调用 release 函数 */
int kref_put(struct kref *kref, void (*release)(struct kref *));

/* 使用示例：嵌入到自定义对象 */
struct my_device {
    struct kref   kref;      /* 嵌入引用计数 */
    int           id;
    char         *name;
};

void my_device_release(struct kref *ref) {
    struct my_device *dev = container_of(ref, struct my_device, kref);
    kfree(dev->name);
    kfree(dev);   /* 计数为0时自动释放 */
}

/* 获取对象 */
kref_get(&dev->kref);

/* 释放对象（若是最后一个引用则销毁）*/
kref_put(&dev->kref, my_device_release);

为什么不直接用 atomic_t ？

来自 LWN.net 的报道（Kernel development, 2003）引用 Andrew Morton 的解释：

"I care more about being able to say 'ah, it uses kref. I understand that refcounting idiom, I know it's well debugged and I know that it traps common errors'. That's better than 'oh crap, this thing implements its own refcounting - I need to review it for the usual errors'."

kref 提供的是一种标准化的引用计数惯用法，让代码审查者能立刻识别并信任引用计数的正确性。

💡 应用层借鉴
C++ 的 std::shared_ptr 是 kref 的高级封装。在 C 语言应用层（如 GLib、libuv）可以直接仿照 kref 模式实现嵌入式引用计数，避免使用 shared_ptr 的额外堆分配（控制块）。Rust 的 Arc<T> 也是同类设计。

工作队列 work_struct

延迟执行 线程池 任务队列
📄 内核驱动模式文档

定义于 include/linux/workqueue.h，允许在进程上下文中延迟执行工作，是内核异步任务处理的核心机制。

设计模式：侵入式任务节点

work_struct 同样是侵入式设计——把工作节点嵌入业务结构体，通过 container_of 在回调中拿回业务数据，无需传递额外的 void* 参数。

struct foo {
    spinlock_t             lock;
    struct workqueue_struct *wq;
    struct work_struct     offload;  /* 嵌入工作节点 */
    int                    data;
};

static void foo_work(struct work_struct *work)
{
    /* 从 work 指针反推出 foo 实例 */
    struct foo *foo = container_of(work, struct foo, offload);
    /* 在工作队列线程中异步处理 */
    process_data(foo->data);
}

static int foo_probe(...)
{
    struct foo *foo = kzalloc(sizeof(*foo), GFP_KERNEL);
    foo->wq = create_singlethread_workqueue("foo-wq");
    INIT_WORK(&foo->offload, foo_work);  /* 绑定回调 */
}

/* 中断上下文中触发异步工作 */
static irqreturn_t foo_handler(int irq, void *arg)
{
    struct foo *foo = arg;
    queue_work(foo->wq, &foo->offload);  /* 提交到工作队列 */
    return IRQ_HANDLED;
}
图7：工作队列模式 —— 中断上下文提交任务，Worker 线程处理，container_of 获取业务数据
💡 应用层借鉴
这是线程池 + 任务队列的经典模式。应用层可以将任务上下文（类似 foo）嵌入一个通用任务节点（类似 work_struct），避免 void* 的强制转换，保留类型安全。Go 的 goroutine + channel、Rust 的 tokio task、C++ 的 std::packaged_task 都是这一思想的体现。

RCU：Read-Copy-Update

读者无锁 读密集型优化 延迟回收
📄 whatisRCU.txt

RCU 在 Linux 2.5.43（2002年10月）由 Dipankar Sarma 和 Paul E. McKenney 加入内核。它是一种读密集型场景下的无锁同步机制：读者完全不需要锁，写者通过"副本修改 + 原子指针替换 + 延迟回收"实现安全更新。

三步更新协议

来自内核官方文档 Documentation/RCU/whatisRCU.txt：

/* Step a：移除旧指针（读者无法再获取旧数据的引用）*/
struct data *newloc = kmalloc(...);
memcpy(newloc, curr, sizeof(*curr));
newloc->a = new_value;

/* Step b：原子发布新指针（读者从此刻起看到新数据）*/
rcu_assign_pointer(ptr, newloc);  /* 等同于 smp_store_release */

/* Step c：等待所有"旧时代"读者结束（宽限期）*/
synchronize_rcu();   /* 阻塞，直到所有 CPU 至少一次上下文切换 */
kfree(oldloc);        /* 此时可以安全释放旧数据 */

/* 读者侧：几乎零开销 */
rcu_read_lock();                         /* 禁止抢占（不是锁！）*/
struct data *curr = rcu_dereference(ptr); /* 读取指针（smp_load_acquire）*/
use_data(curr);
rcu_read_unlock();                        /* 恢复抢占 */
图8：RCU 时序 —— 写者发布新指针后等待宽限期，旧读者看旧数据，新读者看新数据，宽限期后安全释放旧数据

适用场景

适合 RCU ✓不适合 RCU ✗
读操作远多于写操作 写操作频繁（synchronize_rcu 代价高）
读路径性能至关重要（无原子操作） 数据原地修改（非指针替换）
通过指针访问数据 小型 per-CPU 数据（用 per-CPU 变量更好）
内核路由表、进程列表、模块链表 需要强一致性的计数器
💡 应用层借鉴
应用层可用 std::atomic<T*> + std::shared_ptr 模拟 RCU 的核心思想：读者用 atomic_load(acquire) 获取指针，写者 atomic_store(release) 发布新指针，通过引用计数延迟释放。适合配置热更新、路由表更新、黑名单/白名单等读多写少场景。

五大设计哲学

① 侵入式优于非侵入式

把管理节点（list_head、rb_node、work_struct）嵌入对象，避免额外内存分配，提升缓存局部性，消除 void* 类型擦除

② 零额外开销泛型

通过 container_of 在编译期计算偏移量，不依赖函数指针回调，让编译器有机会内联优化，实现类型安全的 C 语言泛型

③ 2的幂次方对齐

kfifo、hashtable 等数据结构强制要求大小为 2^n，用位运算 & 替代取模 %，节省数十纳秒的 CPU 周期

④ 标准化惯用法

kref、notifier_chain 提供标准化的引用计数和通知机制。规范化让代码审查者能立刻理解意图，减少 bug 引入

⑤ 读写分离无锁化

RCU 核心哲学：读路径不加锁，写路径用"副本修改 + 原子发布 + 延迟回收"。在读多写少场景比读写锁性能高出数倍

🔑 核心原则（引自内核文档 HackMD）
"Always reuse existing kernel infrastructure — don't reinvent the wheel." ——《Linux Kernel Development》Chapter 6

同样地，应用层开发者也应当借鉴这些经过极端场景验证的设计思想，而不是每次都从头实现链表、哈希表或引用计数。理解内核设计的"为什么"，比直接复制代码更有价值。
