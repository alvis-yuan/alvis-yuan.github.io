Overview

规范总览 & 危险等级

本手册依据 POSIX.1-2001 / POSIX.1-2008 标准、 Linux man-pages 6.18（2026-02-08）以及 Nginx、Redis、libuv 等大型 C 开源工程的实践， 系统梳理 C 语言多线程编程中的全量规范与禁用项。

危险等级说明

等级标签含义
🚫 绝对禁止 FORBIDDEN 在多线程环境下调用会导致数据竞争、UB 或崩溃，无论任何场景均不得使用。
⚠ 条件限制 CONDITIONAL 在特定场景（单线程初始化阶段、持有保护锁等）可安全使用，否则禁用。

ℹ 场景选择 SCENARIO 根据读写比率、持锁时长、CPU/核心数等选择最优同步原语。

Thread Model

线程模型与共享资源

POSIX 线程（Pthreads）采用 1:1 模型，每个用户线程对应一个内核调度实体。 Linux 通过 clone(2) 系统调用实现，同步原语由 futex(2) 支撑。 ↗ man7 pthreads(7)

线程间共享的资源（进程级）

共享资源说明并发风险
全局/堆内存data segment + heap数据竞争
打开的文件描述符open file descriptors位置竞争
信号处置signal dispositions处理器竞争
进程ID / 父进程IDPID / PPID只读安全
umask文件创建掩码条件安全

资源限制setrlimit(2) ≥ 2.6.10条件安全

Lifecycle

线程生命周期管理

线程从创建到销毁的全周期规范，涵盖属性设置、错误检查、取消点与清理处理器。

创建规范

/* ✅ 正确：检查返回值，明确指定属性 */
pthread_attr_t attr;
pthread_attr_init(&attr);
pthread_attr_setstacksize(&attr, 2 * 1024 * 1024); /* 2MB 栈 */

pthread_t tid;
int rc = pthread_create(&tid, &attr, worker_fn, arg);
if (rc != 0) {
    fprintf(stderr, "pthread_create: %s\n", strerror(rc));
    /* 注意：pthreads 函数返回错误码，不设置 errno */
}
pthread_attr_destroy(&attr); /* 必须销毁属性对象 */
🚫 禁止：不检查 pthread_create 返回值
  • pthreads 函数不设置 errno，直接返回错误号（EAGAIN、ENOMEM 等）
  • 系统线程数有限（/proc/sys/kernel/threads-max），创建失败必须处理

加入与分离

操作函数规范
等待线程结束 pthread_join(3) 每个可连接线程（joinable）必须 join 或 detach，否则资源泄露； 一个线程只能被一个线程 join，重复 join UB。
分离线程 pthread_detach(3) 分离后线程结束时自动回收资源；分离后不得 join。
非阻塞等待 pthread_tryjoin_np(3) Linux 扩展，非标准，注意可移植性。
⚠ 线程 ID 复用问题

POSIX 明确规定：已终止并被 join 的线程、或已分离线程终止后，其线程 ID 可被系统复用。 不得保存线程 ID 并在线程结束后使用，行为未定义。

↗ pthreads(7) Thread IDs 节

取消与清理

/* ✅ 注册清理处理器（资源保护）*/
void cleanup_handler(void *arg) {
    pthread_mutex_unlock((pthread_mutex_t *)arg);
}

void *worker(void *arg) {
    pthread_cleanup_push(cleanup_handler, &my_mutex);
    pthread_mutex_lock(&my_mutex);
    /* ... 临界区 ... */
    pthread_mutex_unlock(&my_mutex);
    pthread_cleanup_pop(0); /* 0 = 不执行，仅弹出 */
    return NULL;
}
🚫 禁止：在取消点前不注册清理处理器就持有锁

若线程在持有互斥锁时被取消（调用了取消点函数），且未注册清理处理器，互斥锁将永远不被释放，导致死锁。

Synchronization

互斥锁 Mutex

互斥锁是最基础的同步原语，确保同一时刻只有一个线程访问共享资源。 POSIX 提供 4 种类型，选择错误会导致死锁或竞争。

四种 Mutex 类型对比

类型（PTHREAD_MUTEX_*）同线程重复加锁其他线程解锁适用场景
NORMAL（默认） 死锁 UB 高性能，已知不会递归
ERRORCHECK 返回 EDEADLK 返回 EPERM 调试开发阶段
RECURSIVE 成功（计数） 返回 EPERM 递归调用（谨慎使用）
ROBUST 同 NORMAL 同 NORMAL 持有者进程/线程崩溃后自动恢复

完整使用规范

/* ✅ 初始化（静态）*/
pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;

/* ✅ 初始化（动态，带属性）*/
pthread_mutexattr_t attr;
pthread_mutexattr_init(&attr);
pthread_mutexattr_settype(&attr, PTHREAD_MUTEX_ERRORCHECK);
pthread_mutex_init(&lock, &attr);
pthread_mutexattr_destroy(&attr);

/* ✅ 加锁 / 解锁 */
pthread_mutex_lock(&lock);   /* 阻塞等待 */
/* 临界区 */
pthread_mutex_unlock(&lock);

/* ✅ 非阻塞尝试 */
if (pthread_mutex_trylock(&lock) == 0) {
    /* 成功获取锁 */
    pthread_mutex_unlock(&lock);
}

/* ✅ 超时加锁（避免永久阻塞）*/
struct timespec ts;
clock_gettime(CLOCK_REALTIME, &ts);
ts.tv_sec += 1; /* 1秒超时 */
int r = pthread_mutex_timedlock(&lock, &ts);
if (r == ETIMEDOUT) { /* 超时处理 */ }

/* ✅ 销毁（确保无线程等待中）*/
pthread_mutex_destroy(&lock);

绝对禁止项

🚫 禁止事项（FORBIDDEN）
  • 跨线程解锁：A 线程加锁，B 线程解锁（NORMAL 类型下行为未定义）
  • double-free mutex：对已销毁的 mutex 再次操作
  • 复制 mutex 对象：用 memcpy 复制 pthread_mutex_t，行为未定义
  • 持锁调用 exit()：其他等锁线程将永久阻塞
  • 信号处理器中调用 pthread_mutex_lock：非 async-signal-safe
  • NORMAL 类型下递归加锁：同一线程第二次加锁死锁
Synchronization

条件变量 Condition Variable

条件变量与互斥锁配合，实现线程等待特定条件成立，是生产者-消费者等模式的核心。

标准使用模式

pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t  cond = PTHREAD_COND_INITIALIZER;
int ready = 0;

/* ─── 等待方（Consumer）──────────────────── */
pthread_mutex_lock(&lock);
while (!ready) {          /* ✅ 必须用 while，不得用 if（防虚假唤醒）*/
    pthread_cond_wait(&cond, &lock);
}
/* 条件成立，处理数据 */
pthread_mutex_unlock(&lock);

/* ─── 通知方（Producer）──────────────────── */
pthread_mutex_lock(&lock);
ready = 1;
pthread_cond_signal(&cond);   /* 唤醒一个等待线程 */
/* pthread_cond_broadcast(&cond); 唤醒所有等待线程 */
pthread_mutex_unlock(&lock);
🚫 禁止：用 if 代替 while 等待条件变量

POSIX 明确允许虚假唤醒（spurious wakeup），即 pthread_cond_wait 在没有信号的情况下返回。必须用 while 循环重新检查条件。

⚠ signal vs broadcast 选择
  • pthread_cond_signal：唤醒至少一个等待线程，用于单生产者/单消费者，更高效
  • pthread_cond_broadcast：唤醒所有等待线程，用于条件更改影响多个线程时
  • 若不确定，优先用 broadcast（正确性优先于性能）

超时等待（防止永久阻塞）

struct timespec deadline;
clock_gettime(CLOCK_REALTIME, &deadline);
deadline.tv_sec += 5; /* 5秒超时 */

pthread_mutex_lock(&lock);
while (!ready) {
    int rc = pthread_cond_timedwait(&cond, &lock, &deadline);
    if (rc == ETIMEDOUT) { break; } /* 超时退出 */
}
pthread_mutex_unlock(&lock);
Synchronization

读写锁 RWLock

读写锁允许多个读者并发，写者独占。适用于读多写少的场景。 使用不当会导致写者饥饿。

适用场景判断

pthread_rwlock_t rwlock = PTHREAD_RWLOCK_INITIALIZER;

/* ✅ 读锁（多线程并发可读）*/
pthread_rwlock_rdlock(&rwlock);
/* 只读操作 */
pthread_rwlock_unlock(&rwlock);

/* ✅ 写锁（独占）*/
pthread_rwlock_wrlock(&rwlock);
/* 写操作 */
pthread_rwlock_unlock(&rwlock);
⚠ 写者饥饿（Writer Starvation）

若读者持续不断地获取读锁，写者可能永远无法获得写锁。 可通过 pthread_rwlockattr_setkind_np（Linux 扩展）设置 PTHREAD_RWLOCK_PREFER_WRITER_NONRECURSIVE_NP 来优先写者。

🚫 禁止：持有读锁时升级为写锁

在持有读锁的情况下直接请求写锁会死锁（因写锁要等待所有读锁释放，包括自己的读锁）。 必须先释放读锁，再获取写锁。

Synchronization

信号量 Semaphore

POSIX 信号量（<semaphore.h>）是计数型同步原语，适合限制并发访问数量或跨进程同步。

有名信号量 vs 无名信号量

类型创建适用场景注意
无名（匿名） sem_init(3) 同进程内线程间同步 pshared=0；不可在 fork 后的子进程中与父进程共享（除非放共享内存）
有名 sem_open(3) 跨进程同步 使用后必须 sem_close 和 sem_unlink 
/* ✅ sem_post 是 async-signal-safe，可在信号处理器中调用 */
sem_t sem;
sem_init(&sem, 0 /*进程内*/, 0 /*初始值*/);

/* 生产者 */
sem_post(&sem);   /* 计数+1，唤醒等待者 */

/* 消费者（阻塞等待）*/
sem_wait(&sem);   /* 计数-1，若0则阻塞 */

/* 超时等待 */
struct timespec ts;
clock_gettime(CLOCK_REALTIME, &ts);
ts.tv_sec += 2;
sem_timedwait(&sem, &ts);

sem_destroy(&sem);
✅ sem_post 可在信号处理器中使用

sem_post(3) 是 POSIX 标准规定的 async-signal-safe 函数， 是极少数可在信号处理器中安全调用的同步操作之一。 ↗ signal-safety(7)

Synchronization

自旋锁 Spinlock

自旋锁在等待时不睡眠，而是持续轮询（busy-wait）。 只适用于极短临界区且多核 CPU环境。

适用判断

条件推荐
临界区 < 几十纳秒，持锁时间极短适合自旋锁
多核 CPU，竞争不激烈适合自旋锁
临界区含 I/O、系统调用、sleep禁止使用自旋锁
单核 CPU禁止使用自旋锁（死锁）
中断/信号处理器中禁止使用自旋锁
pthread_spinlock_t spin;
pthread_spin_init(&spin, PTHREAD_PROCESS_PRIVATE);

pthread_spin_lock(&spin);
/* ⚠ 临界区必须极短，绝对不能 sleep/IO */
pthread_spin_unlock(&spin);

pthread_spin_destroy(&spin);
🚫 禁止：在自旋锁临界区内调用任何可能阻塞的函数

自旋锁持有者被调度出去时，其他等待该锁的线程将持续消耗 CPU， 严重降低系统性能。临界区内绝对不能调用：malloc、printf、文件 I/O、sleep、任何系统调用（除原子操作外）。

Atomics

原子操作 Atomics

C11 引入 <stdatomic.h> 提供标准原子类型和操作。 GCC 提供 __atomic_* 内置函数（推荐）和旧式 __sync_*。

C11 标准原子操作

/* C11 原子类型 */
_Atomic int counter = 0;       /* 或 atomic_int */
atomic_flag flag = ATOMIC_FLAG_INIT;

/* ✅ 原子加减 */
atomic_fetch_add(&counter, 1);  /* counter++ 的原子版本 */
atomic_fetch_sub(&counter, 1);

/* ✅ CAS（比较并交换）*/
int expected = 0;
int desired  = 1;
_Bool ok = atomic_compare_exchange_strong(&counter, &expected, desired);

/* ✅ 内存顺序（显式指定，减少不必要的屏障）*/
atomic_store_explicit(&counter, 42, memory_order_release);
int v = atomic_load_explicit(&counter, memory_order_acquire);

memory_order 选择指南

memory_order含义典型用途性能开销
relaxed 仅保证原子性，不保证顺序 统计计数器（不关心顺序） 最低
acquire 本操作后的读写不得提前 锁的获取端 中等
release 本操作前的读写不得延后 锁的释放端 中等
acq_rel acquire + release 读-修改-写操作（如 CAS） 较高
seq_cst 全局顺序一致（默认） 最强保证，简单正确但性能低 最高
🚫 禁止：用 volatile 替代原子操作实现同步

volatile 仅防止编译器优化掉对变量的读写，不提供任何内存顺序保证， 不防止 CPU 乱序执行。在 C11 之前 volatile 的线程同步用法是历史遗留，现代代码中禁止使用。

False Sharing（伪共享）

/* ❌ 错误：两个原子变量在同一缓存行，造成伪共享 */
struct { atomic_int a; atomic_int b; } counters;

/* ✅ 正确：对齐到缓存行（通常 64 字节）*/
struct {
    atomic_int a;
    char _pad[60]; /* 填充至 64 字节 */
} __attribute__((aligned(64))) counter_a;

struct {
    atomic_int b;
    char _pad[60];
} __attribute__((aligned(64))) counter_b;
API Safety

禁用函数（MT-Unsafe）

以下函数由 POSIX.1-2001 / POSIX.1-2008 标准明确标注为非线程安全（MT-Unsafe）， 在多线程程序中禁止直接调用。 ↗ pthreads(7) Thread-safe functions 节

🚫 禁用✅ 线程安全替代原因

asctime() asctime_r() 返回静态字符串缓冲区
ctime() ctime_r() 同上
basename() basename_r() 或手动实现 某些实现使用静态缓冲区
dirname() 手动实现 同上

🚫 禁用✅ 线程安全替代原因

随机数类

🚫 禁用✅ 线程安全替代原因

drand48() drand48_r() 同上
lrand48() lrand48_r() 同上
mrand48() mrand48_r() 同上

环境变量类

🚫 禁止：多线程运行期间调用 setenv / putenv / unsetenv / clearenv

getenv()、setenv()、putenv()、unsetenv()、clearenv() 均被 POSIX 标注为 MT-Unsafe（const:env 类）。
规范：所有环境变量操作必须在多线程启动前完成，运行期间禁止修改。

Signal Safety

异步信号安全函数（Async-Signal-Safe）

信号处理器（signal handler）中只能调用 async-signal-safe 函数。 非异步信号安全函数（包括几乎所有 stdio 函数）在信号处理器中调用会导致未定义行为。 ↗ signal-safety(7)

🚫 禁止：在信号处理器中调用以下函数
  • 所有 stdio 函数：printf、fprintf、fopen、fclose 等
  • malloc、free、realloc（使用内部锁，可能死锁）
  • pthread_mutex_lock（非 async-signal-safe）
  • 大多数 C 标准库函数

POSIX 要求必须是 async-signal-safe 的函数（精选）

类别可用函数
I/O（系统调用） read(2)、write(2)、open(2)、close(2)、send(2)、recv(2)
进程控制 _exit(2)、_Exit(2)、fork(2)（谨慎）、execve(2)
信号 sigaction(2)、sigprocmask(2)、sigsuspend(2)、kill(2)
时间 clock_gettime(2)、time(2)、alarm(2)
同步 sem_post(3)（唯一 async-signal-safe 的同步函数）
文件系统 stat(2)、fstat(2)、chmod(2)、chown(2)、unlink(2)
内存 memcpy(3)、memset(3)、memmove(3)（POSIX.1-2008 TC2 新增）
字符串（2008 TC2 新增） strlen(3)、strcpy(3)、strtok_r(3)、strcmp(3) 等
网络 socket(2)、bind(2)、listen(2)、accept(2)、connect(2)

errno 在信号处理器中的处理

/* ✅ 正确：保存和恢复 errno */
void signal_handler(int sig) {
    int saved_errno = errno;  /* 保存 */
    
    /* 只调用 async-signal-safe 函数 */
    write(STDERR_FILENO, "signal\n", 7);
    sem_post(&my_sem);
    
    errno = saved_errno;       /* 恢复 */
}
Cancellation

取消点函数（Cancellation Points）

当线程处于可延迟取消（PTHREAD_CANCEL_DEFERRED，默认）状态， 调用取消点函数时如果有待处理的取消请求，线程将在该点被取消。 ↗ pthreads(7) Cancellation points 节

⚠ 规范：持有资源时通过取消点必须注册清理处理器

任何线程在持有互斥锁、已打开文件、已分配内存等资源的情况下， 若调用了取消点函数，必须提前注册 pthread_cleanup_push 清理处理器。

必须是取消点的函数（POSIX 要求）

只有 3 个 async-cancel-safe 函数

ℹ POSIX 规定的 async-cancel-safe 函数（仅 3 个）
  • pthread_cancel()
  • pthread_setcancelstate()
  • pthread_setcanceltype()

其余函数在异步取消模式下均不安全。建议始终使用默认的延迟取消模式。

内存顺序 & 内存模型

C11 引入了正式的内存模型（ISO/IEC 9899:2011 §5.1.2.4）， 明确规定了编译器和 CPU 允许的内存重排序范围。

屏障类型代码作用

Deadlock

死锁预防规范

死锁是多线程程序最常见的致命 Bug，必须在设计阶段通过规范预防， 不能依赖运行时检测。

规则 1：全局固定加锁顺序（最重要）

/* ❌ 死锁：线程A加锁顺序 mutex_a→mutex_b，线程B反之 */

/* ✅ 正确：所有线程必须按相同顺序加锁 */
/* 文档化锁序：mutex_a 必须在 mutex_b 之前获取 */
pthread_mutex_lock(&mutex_a);   /* Level 1 */
pthread_mutex_lock(&mutex_b);   /* Level 2 */
/* ... */
pthread_mutex_unlock(&mutex_b); /* 逆序解锁 */
pthread_mutex_unlock(&mutex_a);

规则 2：trylock + 回退（CMU 推荐模式）

/* ✅ 正确：尝试加锁，失败则回退重试 */
pthread_mutex_lock(&mutex_a);
while (pthread_mutex_trylock(&mutex_b)) {
    pthread_mutex_unlock(&mutex_a); /* 释放已有锁，避免死锁 */
    sched_yield();                   /* 让出 CPU */
    pthread_mutex_lock(&mutex_a);   /* 重新获取 */
}
/* 两把锁都持有 */
pthread_mutex_unlock(&mutex_b);
pthread_mutex_unlock(&mutex_a);

规则 3：持锁最短时间原则

ℹ 最小持锁范围
  • 临界区内只做最少必要的工作
  • 不要在持锁时进行 I/O、网络操作、sleep
  • 可以先复制共享数据到局部变量，解锁后再处理
  • 避免在持锁时调用未知的第三方函数（可能内部加锁导致锁序冲突）
Signals

信号与线程交互

信号和线程的交互是 POSIX 编程中最复杂的领域之一。 NPTL（Native POSIX Threads Library，Linux ≥ glibc 2.3.2） 内部使用前两个实时信号（SIGRTMIN、SIGRTMIN+1），应用程序禁止使用。

核心规范

规范说明
🚫 禁止 在非主线程使用 sigprocmask 多线程程序中必须使用 pthread_sigmask(3)；sigprocmask 在多线程下行为未定义
✅ 推荐 信号专用线程模式 主线程和工作线程屏蔽所有信号，专门创建一个线程用 sigwait() 处理信号
🚫 禁止 NPTL 内部信号 不得安装 SIGRTMIN、SIGRTMIN+1 的处理器（NPTL 内部使用）
⚠ 注意 信号处理器与取消点 从异步信号处理器中调用取消点函数，可能触发等价于异步取消的效果，导致状态不一致

推荐：信号专用线程模式

/* ✅ 标准模式：信号专用线程 */
static void *signal_thread(void *arg) {
    sigset_t *set = (sigset_t *)arg;
    int sig;
    for (;;) {
        sigwait(set, &sig);  /* async-signal-safe 的等待 */
        switch (sig) {
            case SIGTERM: handle_shutdown(); break;
            case SIGHUP:  handle_reload();   break;
        }
    }
    return NULL;
}

int main() {
    sigset_t set;
    sigemptyset(&set);
    sigaddset(&set, SIGTERM);
    sigaddset(&set, SIGHUP);
    
    /* 屏蔽信号（在创建任何线程之前！子线程继承掩码）*/
    pthread_sigmask(SIG_BLOCK, &set, NULL);
    
    /* 创建工作线程（自动继承屏蔽掩码）*/
    /* ... pthread_create workers ... */
    
    /* 创建信号专用线程 */
    pthread_t sig_tid;
    pthread_create(&sig_tid, NULL, signal_thread, &set);
    /* ... */
}
TLS

线程局部存储（Thread Local Storage）

TLS 使每个线程拥有变量的独立副本，是消除共享状态、实现线程安全的重要手段。

三种 TLS 实现方式对比

方式语法特点适用
C11 标准 _Thread_local int x; C11 标准，可移植，支持静态初始化 推荐，现代编译器
GCC 扩展 __thread int x; GCC/Clang 扩展，广泛支持，性能与 C11 相同 兼容老项目
POSIX API pthread_key_create/pthread_setspecific 动态，可设置析构函数，数量有上限（PTHREAD_KEYS_MAX） 需要析构回调时 
/* ── C11 _Thread_local ──────────────────── */
_Thread_local int error_code = 0;
_Thread_local char buf[1024];

/* ── POSIX pthread key（带析构函数）────── */
pthread_key_t tls_key;

void tls_destructor(void *val) {
    free(val); /* 线程退出时自动释放 */
}

void init_tls() {
    pthread_key_create(&tls_key, tls_destructor);
}

void *get_tls_buf() {
    void *p = pthread_getspecific(tls_key);
    if (!p) {
        p = malloc(1024);
        pthread_setspecific(tls_key, p);
    }
    return p;
}
⚠ POSIX TLS Key 数量限制

POSIX 只保证至少 PTHREAD_KEYS_MIN（128）个 key， 实际上限为 PTHREAD_KEYS_MAX。析构函数循环调用上限为 PTHREAD_DESTRUCTOR_ITERATIONS（4）次， 析构函数中 setspecific 值最多重试 4 次。

Fork

fork() 与线程的陷阱

在多线程程序中调用 fork() 是极危险的操作， 子进程中只有调用 fork() 的线程存活，其他线程消失， 但所有互斥锁状态被复制。

🚫 禁止：多线程程序中在 fork 后的子进程中调用非 async-signal-safe 函数

fork() 后，子进程拥有父进程所有互斥锁的副本（包括其他线程持有的锁）。 若子进程调用任何使用这些锁的函数（如 malloc、printf）， 可能立即死锁。

安全规则：fork 后的子进程只能调用 async-signal-safe 函数， 然后立即 exec()（exec 会替换进程映像，消除所有锁状态）。

pthread_atfork：fork 前后钩子

/* ✅ 正确：使用 pthread_atfork 管理锁状态 */
void prepare() {
    pthread_mutex_lock(&my_mutex);   /* fork 前加锁 */
}
void parent() {
    pthread_mutex_unlock(&my_mutex); /* 父进程 fork 后解锁 */
}
void child() {
    pthread_mutex_unlock(&my_mutex); /* 子进程 fork 后解锁 */
}

pthread_atfork(prepare, parent, child);
ℹ 推荐模式：fork+exec（daemon 进程）
  • 多线程程序 fork 后，子进程应立即 exec，不做任何其他操作
  • Redis、Nginx 的持久化/重载均采用此模式
  • 确实需要 fork 子进程执行复杂逻辑时，应在多线程启动之前 fork
