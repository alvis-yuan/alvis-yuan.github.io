⚙️ 基础概念 · 01

信号量原理

来源：Linux man-pages 6.18 — sem_overview(7)

POSIX semaphores allow processes and threads to synchronize their actions. （POSIX 信号量允许进程和线程同步它们的行为。）

A semaphore is an integer whose value is never allowed to fall below zero. Two operations can be performed on semaphores: increment the semaphore value by one (sem_post(3)); and decrement the semaphore value by one (sem_wait(3)). If the value of a semaphore is currently zero, then a sem_wait(3) operation will block until the value becomes greater than zero.

核心思想：P / V 操作

信号量由荷兰计算机科学家 Edsger Dijkstra 提出（又称 P/V 操作或 Down/Up 操作）。 其本质是一个受保护的整数计数器，通过原子操作保证并发安全。

🔒 P 操作（sem_wait）—— 申请资源

将信号量值减 1：

  • 若值 > 0：立即减 1 并返回
  • 若值 = 0：调用者 阻塞，直到值 > 0

等价于：while(s==0) wait; s--;（原子操作）

🔓 V 操作（sem_post）—— 释放资源

将信号量值加 1：

  • 若有线程/进程在等待：唤醒其中一个
  • 否则：值 +1，直接返回

等价于：s++; if(waiters) wakeup_one();

📌
关键原则：信号量值永远不允许低于零（"an integer whose value is never allowed to fall below zero"）。 这是 POSIX 标准的核心保证，确保了原子语义。

二值信号量 vs 计数信号量

二值信号量（Binary Semaphore）

初始值为 1，值域为 {0, 1}。等同于互斥锁（Mutex）功能，用于互斥访问临界区。

pshared = 0 value = 1

计数信号量（Counting Semaphore）

初始值为 N（N > 1），表示资源数量。用于控制对有限数量资源的并发访问。

pshared = 0 value = N

API 函数参考

🔗
以下所有 API 均来自官方 Linux man-pages（POSIX.1-2008）： sem_init(3)、 sem_open(3)、 sem_wait(3)、 sem_post(3)
编译时须加 -pthread（链接 libpthread 库）

匿名信号量（Unnamed Semaphores）

函数签名 说明 返回值
int sem_init(sem_t *sem, int pshared, unsigned int value) 初始化匿名信号量。
pshared=0：线程间共享；pshared≠0：进程间共享（需放于共享内存）		 成功返回 0；失败返回 -1，设置 errno
int sem_destroy(sem_t *sem) 销毁匿名信号量，释放资源。销毁已初始化的信号量前须确保无线程等待其上 成功返回 0；失败返回 -1
int sem_wait(sem_t *sem) P 操作（decrement / lock）：若值 > 0 则减 1；否则阻塞直到值 > 0 成功返回 0；失败返回 -1（EINTR/EINVAL）
int sem_trywait(sem_t *sem) 非阻塞 P 操作：若值为 0 则立即返回错误（errno=EAGAIN） 成功返回 0；失败返回 -1
int sem_timedwait(sem_t *sem, const struct timespec *abstime) 带超时的 P 操作：超时后返回 -1（errno=ETIMEDOUT）；abstime 是绝对时间 成功返回 0；超时返回 -1
int sem_post(sem_t *sem) V 操作（increment / unlock）：值加 1；若有等待者则唤醒其中一个。async-signal-safe，可在信号处理函数中调用 成功返回 0；失败返回 -1（EINVAL/EOVERFLOW）
int sem_getvalue(sem_t *sem, int *sval) 获取当前信号量的值（存入 sval 指向的整数） 成功返回 0；失败返回 -1

命名信号量（Named Semaphores）

函数签名 说明 返回值
sem_t *sem_open(const char *name, int oflag, ...) 创建或打开命名信号量。名称格式为 /somename。
O_CREAT: 不存在则创建；O_EXCL: 已存在则报错。
创建时须提供 mode（权限）和 value（初值）		 成功返回 sem_t 指针；失败返回 SEM_FAILED
int sem_close(sem_t *sem) 关闭命名信号量描述符（不删除内核对象） 成功返回 0；失败返回 -1
int sem_unlink(const char *name) 从系统中删除命名信号量。具有内核持久性：不调用 sem_unlink 则信号量持续到系统关机 成功返回 0；失败返回 -1
⚠️
Linux 实现细节：命名信号量在 Linux 上创建于虚拟文件系统 /dev/shm， 以 sem.somename 形式命名。因此名称长度限制为 NAME_MAX-4（即最多 251 字符）。 POSIX 命名信号量具有内核持久性（kernel persistence）：若不调用 sem_unlink()，信号量将存在直到系统关机。
— Linux man-pages 6.18, sem_overview(7)

🔗 线程间使用 · 04

场景一：互斥访问（二值信号量）

🔐
场景描述：多线程安全递增共享计数器
使用匿名二值信号量（pshared=0, value=1）替代 mutex

问题：多个线程并发递增同一个共享变量，若无同步机制会产生竞态条件（Race Condition）， 导致计数结果错误。
方案：使用初始值为 1 的信号量（二值信号量）保护临界区，一次只允许一个线程进入。

图 4：二值信号量互斥访问时序（线程间，pshared=0, value=1）

C 线程间互斥 · 匿名信号量 · pshared=0 
#include <stdio.h>
#include <pthread.h>
#include <semaphore.h>   /* POSIX 信号量头文件 */

/* 全局变量：线程间共享同一地址空间，sem_t 可作为全局变量 */
sem_t sem;           /* 匿名信号量，pshared=0 时放全局/堆均可 */
volatile int counter = 0;
#define N_THREADS  4
#define N_ITER     100000

void *worker(void *arg) {
    for (int i = 0; i < N_ITER; i++) {
        sem_wait(&sem);   /* P 操作：进入临界区，S-- (S: 1→0) */

        counter++;        /* 临界区：安全地修改共享变量 */

        sem_post(&sem);   /* V 操作：离开临界区，S++ (S: 0→1) */
    }
    return NULL;
}

int main() {
    pthread_t tids[N_THREADS];

    /*
     * sem_init(sem, pshared, value)
     *   pshared = 0  → 线程间共享（thread-shared semaphore）
     *   value   = 1  → 二值信号量，同一时刻只有一个线程可进入临界区
     * 来源：Linux man-pages 6.18, sem_init(3)
     *   https://man7.org/linux/man-pages/man3/sem_init.3.html
     */
    sem_init(&sem, 0, 1);

    for (int i = 0; i < N_THREADS; i++)
        pthread_create(&tids[i], NULL, worker, NULL);

    for (int i = 0; i < N_THREADS; i++)
        pthread_join(tids[i], NULL);

    /* 期望结果：N_THREADS * N_ITER = 400000 */
    printf("counter = %d (expected %d)\n", counter, N_THREADS * N_ITER);

    sem_destroy(&sem);  /* 销毁信号量，释放资源 */
    return 0;
}
/* 编译：gcc -o mutex_sem mutex_sem.c -pthread */
✅
二值信号量 vs Mutex：两者功能相近，但信号量与 mutex 最大区别是 信号量可以跨线程释放（一个线程 wait，另一个线程 post 是合法的）， 而 mutex 要求同一线程加锁解锁。此特性使信号量更适合事件通知场景。

场景二：生产者-消费者（计数信号量）

⚙️
场景描述：有界缓冲区的生产者-消费者问题
使用三个信号量协调线程：mutex + buffer_count + item_count

经典并发问题：生产者往有界缓冲区放数据，消费者从中取数据。 需要保证：缓冲区满时生产者等待，缓冲区空时消费者等待，且读写操作互斥。
此示例来自 softprayog.in — POSIX Semaphores in Linux，使用匿名信号量（pshared=0）实现。

图 5：生产者-消费者三信号量协调模型（来源：softprayog.in POSIX Semaphores in Linux）

C 线程间 · 计数信号量 · 生产者-消费者 
#include <stdio.h>
#include <pthread.h>
#include <semaphore.h>
#include <unistd.h>

#define MAX_BUFFERS  10   /* 缓冲区槽位数量 */

/* 环形缓冲区 */
char buf[MAX_BUFFERS][100];
int  buffer_index = 0;
int  buffer_print_index = 0;

/*
 * 三个匿名信号量（pshared=0，线程间共享）：
 *   mutex_sem        : 二值信号量，保护对 buffer_index 的互斥访问
 *   buffer_count_sem : 计数信号量，追踪空闲槽位数（初始=MAX_BUFFERS）
 *   spool_signal_sem : 计数信号量，追踪待消费项数（初始=0）
 *
 * 来源：softprayog.in — POSIX Semaphores in Linux
 *   https://www.softprayog.in/programming/posix-semaphores
 */
sem_t mutex_sem, buffer_count_sem, spool_signal_sem;

void *producer(void *arg) {
    int id = *(int *)arg;
    for (int i = 0; i < 5; i++) {
        /* P(buffer_count_sem)：等待有空闲槽位 */
        sem_wait(&buffer_count_sem);

        /* P(mutex_sem)：多生产者竞争时保护索引 */
        sem_wait(&mutex_sem);
        int j = buffer_index;
        buffer_index = (buffer_index + 1) % MAX_BUFFERS;
        sem_post(&mutex_sem);    /* V(mutex_sem) */

        snprintf(buf[j], 100, "Producer-%d item-%d", id, i);

        /* V(spool_signal_sem)：通知消费者有新数据 */
        sem_post(&spool_signal_sem);
        sleep(1);
    }
    return NULL;
}

void *consumer(void *arg) {
    for (;;) {
        /* P(spool_signal_sem)：等待有数据可消费 */
        sem_wait(&spool_signal_sem);

        printf("Consumed: %s\n", buf[buffer_print_index]);
        buffer_print_index = (buffer_print_index + 1) % MAX_BUFFERS;

        /* V(buffer_count_sem)：归还空闲槽位 */
        sem_post(&buffer_count_sem);
    }
    return NULL;
}

int main() {
    pthread_t producers[3], consumer_tid;
    int ids[] = {1, 2, 3};

    /* 初始化三个匿名信号量（线程间共享，pshared=0） */
    sem_init(&mutex_sem,        0, 1);            /* 二值，互斥 */
    sem_init(&buffer_count_sem, 0, MAX_BUFFERS); /* 空槽数 */
    sem_init(&spool_signal_sem, 0, 0);          /* 已产出数（初始0）*/

    pthread_create(&consumer_tid, NULL, consumer, NULL);
    for (int i = 0; i < 3; i++)
        pthread_create(&producers[i], NULL, producer, &ids[i]);

    for (int i = 0; i < 3; i++)
        pthread_join(producers[i], NULL);

    pthread_cancel(consumer_tid);
    sem_destroy(&mutex_sem);
    sem_destroy(&buffer_count_sem);
    sem_destroy(&spool_signal_sem);
    return 0;
}

场景三：线程执行顺序控制

🔄
场景描述：强制线程 A 先于线程 B 执行
使用初始值为 0 的信号量作为"事件通知"机制

有时需要保证某个线程完成某操作后，另一个线程才能继续。 使用初始值为 0 的信号量，线程 B 调用 sem_wait() 会立即阻塞， 直到线程 A 完成任务并调用 sem_post() 发出信号。

关键特性（来源：sem_post(3) man page）：
"sem_post() is async-signal-safe: it may be safely called within a signal handler."
sem_post() 是异步信号安全的，可在信号处理函数中安全调用。

C 线程间 · 顺序控制 · value=0 信号通知 
#include <stdio.h>
#include <pthread.h>
#include <semaphore.h>

sem_t ready_sem;  /* 初始值=0：B 等待 A 完成 */

void *thread_A(void *arg) {
    printf("[A] 执行任务（初始化数据库连接）...\n");
    /* ... 耗时操作 ... */
    printf("[A] 任务完成，发送信号给 B\n");

    /*
     * sem_post()：V 操作，将 S 从 0 变为 1，唤醒正在等待的线程 B
     * sem_post() is async-signal-safe — sem_post(3) man page
     *   https://man7.org/linux/man-pages/man3/sem_post.3.html
     */
    sem_post(&ready_sem);
    return NULL;
}

void *thread_B(void *arg) {
    printf("[B] 等待 A 完成...\n");

    /*
     * sem_wait()：P 操作，S=0 时阻塞，直到 A 调用 sem_post()
     * sem_wait() decrements (locks) the semaphore...
     * If the semaphore's value is currently zero, then the call blocks
     * — sem_wait(3) man page
     *   https://man7.org/linux/man-pages/man3/sem_wait.3.html
     */
    sem_wait(&ready_sem);
    printf("[B] A 已完成，B 开始后续操作\n");
    return NULL;
}

int main() {
    pthread_t ta, tb;

    /* value=0：B 调用 sem_wait 时会立即阻塞 */
    sem_init(&ready_sem, 0, 0);

    pthread_create(&tb, NULL, thread_B, NULL); /* B 先创建也没关系 */
    pthread_create(&ta, NULL, thread_A, NULL);

    pthread_join(ta, NULL);
    pthread_join(tb, NULL);

    sem_destroy(&ready_sem);
    return 0;
}
/* 输出始终是：A 的打印在 B 的后续操作之前 */
💡
sem_trywait / sem_timedwait 变体： 当不希望无限等待时，可使用 sem_trywait()（非阻塞，值为 0 时返回 EAGAIN） 或 sem_timedwait()（带超时，超时返回 ETIMEDOUT）。 abstime 是绝对时间（相对 Epoch），通常用 clock_gettime(CLOCK_REALTIME, &ts); ts.tv_sec += N; 计算。 — sem_wait(3) man page

场景四：命名信号量跨进程同步

📛
场景描述：不相关进程间协调打印顺序
使用命名信号量（sem_open），任何知道名称的进程均可参与

命名信号量由名称（/sem-name 格式）在内核中标识， 任何具有权限的进程均可通过相同名称访问同一信号量，无需共享内存。 在 Linux 上，命名信号量物理存储于 /dev/shm/sem.sem-name。

内核持久性（来源：sem_overview(7)）： "POSIX named semaphores have kernel persistence: if not removed by sem_unlink(3), a semaphore will exist until the system is shut down."

图 6：命名信号量进程间同步（名称 /my-sem，存储于 /dev/shm）

C 进程间 · 命名信号量 · sem_open / sem_unlink 
/*
 * 进程 A（服务端/父进程）：创建命名信号量并发送信号
 *
 * sem_open() creates a new POSIX semaphore or opens an existing semaphore.
 * The semaphore is identified by name (format: /somename).
 * — sem_open(3) Linux man-pages 6.18
 *   https://man7.org/linux/man-pages/man3/sem_open.3.html
 */
#include <fcntl.h>
#include <sys/stat.h>
#include <semaphore.h>
#include <stdio.h>
#include <stdlib.h>

#define SEM_NAME  "/posix-demo-sem"  /* 格式：/somename */

int process_A() {
    /* 创建命名信号量（若不存在则创建，初始值=0）
     * mode = 0660：owner/group 读写权限
     * value = 0：B 调用 sem_wait 时立即阻塞 */
    sem_t *sem = sem_open(SEM_NAME,
                          O_CREAT | O_EXCL,  /* 必须是新建的 */
                          0660,             /* 权限 */
                          0);               /* 初始值 */
    if (sem == SEM_FAILED) { perror("sem_open"); exit(1); }

    printf("[A] 执行关键任务...\n");
    /* ... 执行初始化、文件写入等工作 ... */

    printf("[A] 完成，通知进程 B\n");
    sem_post(sem);   /* V 操作：S: 0 → 1，唤醒等待中的 B */

    sem_close(sem);       /* 关闭描述符（信号量对象仍在内核） */
    sem_unlink(SEM_NAME); /* 删除内核信号量对象 */
    return 0;
}

/* ─────────────────────────────────────────────────────── */

/* 进程 B（客户端/子进程）：打开同名信号量并等待 */
int process_B() {
    /* 打开已存在的命名信号量（不带 O_CREAT）*/
    sem_t *sem = sem_open(SEM_NAME, 0);
    if (sem == SEM_FAILED) { perror("sem_open"); exit(1); }

    printf("[B] 等待进程 A 完成...\n");
    sem_wait(sem);   /* P 操作：S=0 则阻塞，A 发信号后继续 */

    printf("[B] A 已完成，开始依赖 A 结果的工作\n");
    sem_close(sem);  /* 只关闭描述符，不删除（由 A 负责 unlink）*/
    return 0;
}
/* 编译：gcc process_a.c -o a -pthread
         gcc process_b.c -o b -pthread
   先运行 ./b（后台），再运行 ./a */
⚠️
必须调用 sem_unlink()：由于命名信号量具有内核持久性，若程序异常退出而未调用 sem_unlink()，信号量将残留至系统重启。可在 /dev/shm/ 目录查看： ls /dev/shm/sem.*。下次创建同名信号量时，若使用 O_EXCL 则会失败（EEXIST）。

场景五：匿名信号量 + 共享内存（父子进程）

🗄️
场景描述：父子进程通过共享内存中的匿名信号量同步
pshared=1，信号量须置于 mmap/shmget 共享内存中

当匿名信号量用于进程间时，必须将其放在共享内存区域。 官方说明（sem_init(3)）： "If pshared is nonzero, then the semaphore is shared between processes, and should be located in a region of shared memory (see shm_open(3), mmap(2), and shmget(2)). Since a child created by fork(2) inherits its parent's memory mappings, it can also access the semaphore."

图 7：匿名信号量 pshared=1 + mmap 共享内存（父子进程间）

C 进程间 · 匿名信号量 · pshared=1 · mmap共享内存 
#include <stdio.h>
#include <stdlib.h>
#include <semaphore.h>
#include <sys/mman.h>    /* mmap */
#include <sys/wait.h>    /* wait */
#include <unistd.h>

/* 将信号量和共享数据放在同一块共享内存结构体中 */
typedef struct {
    sem_t sem;      /* 匿名信号量，pshared=1 */
    int   data;     /* 父进程写入、子进程读取 */
} SharedMem;

int main() {
    /*
     * 创建匿名共享内存映射（MAP_SHARED + MAP_ANONYMOUS）
     * 父子进程通过 fork 共享同一物理内存页
     *
     * If pshared is nonzero, then the semaphore is shared between processes,
     * and should be located in a region of shared memory (see shm_open(3),
     * mmap(2), and shmget(2)).
     * — sem_init(3) Linux man-pages 6.18
     */
    SharedMem *shm = (SharedMem *)mmap(
        NULL,
        sizeof(SharedMem),
        PROT_READ | PROT_WRITE,
        MAP_SHARED | MAP_ANONYMOUS,  /* 匿名共享映射 */
        -1, 0
    );
    if (shm == MAP_FAILED) { perror("mmap"); exit(1); }

    /*
     * sem_init(sem, pshared=1, value=0)
     *   pshared=1：声明此信号量用于进程间共享
     *   value=0  ：子进程 sem_wait 时立即阻塞，等父进程 sem_post
     */
    sem_init(&shm->sem, 1, 0);
    shm->data = 0;

    pid_t pid = fork();

    if (pid == 0) {
        /* ── 子进程 ── */
        printf("[子] 等待父进程准备数据...\n");
        sem_wait(&shm->sem);  /* 阻塞，直到父进程 sem_post */
        printf("[子] 读取到共享数据：%d\n", shm->data);
        exit(0);

    } else {
        /* ── 父进程 ── */
        printf("[父] 准备数据...\n");
        shm->data = 42;       /* 写入共享数据 */
        printf("[父] 数据已就绪，通知子进程\n");
        sem_post(&shm->sem);  /* V 操作，唤醒子进程 */

        wait(NULL);           /* 等待子进程退出 */
        sem_destroy(&shm->sem);
        munmap(shm, sizeof(SharedMem));
    }
    return 0;
}
/* 编译：gcc -o shared_sem shared_sem.c -pthread */
⚠️
pshared=1 的系统支持：若系统不支持进程共享的匿名信号量， sem_init() 将返回 -1，并设置 errno = ENOSYS。 在完整 Linux 2.6 + NPTL glibc 环境下均受支持。 — sem_init(3)："ENOSYS: pshared is nonzero, but the system does not support process-shared semaphores"

注意事项与常见陷阱

1. 已初始化的信号量不可重复初始化

⚠️
来源：sem_init(3)： "Initializing a semaphore that has already been initialized results in undefined behavior." — 对同一 sem_t 变量重复调用 sem_init() 是未定义行为（UB）。

2. sem_wait 可被信号中断（EINTR）

C 正确处理 EINTR 中断 
/*
 * sem_wait() can be interrupted by a signal handler
 * Error: EINTR — The call was interrupted by a signal handler
 * — sem_wait(3) Linux man-pages
 *   https://man7.org/linux/man-pages/man3/sem_wait.3.html
 *
 * 正确做法：循环重试直到成功
 */
while (sem_wait(&sem) == -1 && errno == EINTR)
    continue;  /* 被信号打断则重试 */

/* 或使用 sem_timedwait 的官方示例同样采用此模式：
 * while ((s = sem_timedwait(&sem, &ts)) == -1 && errno == EINTR)
 *     continue;
 * — sem_wait(3) man page EXAMPLES section */

3. 命名信号量的残留问题

⚠️ 典型问题：sem_unlink 忘记调用

进程崩溃后 /dev/shm/sem.name 文件残留，下次运行时：

  • 使用 O_CREAT | O_EXCL：EEXIST 错误
  • 使用 O_CREAT（无 O_EXCL）：打开已存在的信号量，初始值可能不符预期

解决方案：使用信号处理函数（signal handler）或 atexit() 确保 sem_unlink 被调用； 或在程序启动时先调用 sem_unlink(name) 清理残留（忽略 ENOENT 错误）。

4. sem_post 是异步信号安全的

✅
来源：sem_post(3)： "sem_post() is async-signal-safe: it may be safely called within a signal handler." — sem_post() 可在信号处理函数中安全调用，这使得信号量可用于信号与线程之间的通知机制。 见 sem_wait(3) 的 EXAMPLES 章节官方示例（SIGALRM 信号处理函数中调用 sem_post）。

5. Linux 命名信号量的文件系统映射

在 Linux 上，命名信号量物理存储于 /dev/shm/ 虚拟文件系统中， 文件名前缀为 sem.（例如 /dev/shm/sem.my-sem）。 这也是名称长度限制为 NAME_MAX - 4（即 251 字符）而非 NAME_MAX 的原因。

自 Linux 2.6.19 起，可对 /dev/shm/ 下的文件设置 ACL， 以实现细粒度的访问控制（per-user/per-group）。

查看当前系统残留的命名信号量：ls /dev/shm/sem.*

6. POSIX 信号量 vs System V 信号量

📌
来源：sem_overview(7)： "System V semaphores (semget(2), semop(2), etc.) are an older semaphore API. POSIX semaphores provide a simpler, and better designed interface than System V semaphores; on the other hand POSIX semaphores are less widely available (especially on older systems) than System V semaphores."

POSIX 信号量接口更简洁、设计更好；System V 信号量历史更久、兼容性更广。 Linux 2.6 + NPTL glibc 环境下 POSIX 信号量已得到完整支持，新代码优先选用 POSIX 信号量。
