Linux ELF 文件结构深度解析

ELF（Executable and Linkable Format，可执行与链接格式）是 Linux 及绝大多数类 UNIX 系统中的标准二进制文件格式，用于：普通可执行文件、可重定位目标文件（.o）、共享对象（.so）以及核心转储（core dump）文件。

📄 原文参考：man7.org — elf(5) Linux manual page 📄 参考：Wikipedia — Executable and Linkable Format

ELF 文件的四种类型

ET_REL 0x01
可重定位文件

编译器生成的 .o 文件，地址待链接器填写

ET_EXEC 0x02
可执行文件

可直接运行，含固定加载地址

ET_DYN 0x03
共享目标文件

.so 动态库，也用于 PIE 可执行

ET_CORE 0x04
核心转储文件

进程崩溃时由内核生成，用于调试

文件整体布局（可视化）

⚡ 第一性原理

ELF 本质上是一种自描述的二进制容器：文件头告诉你"我是谁"，节区头表告诉链接器"数据在哪"，程序头表告诉操作系统"如何加载我"。三张表的设计让同一份数据可以同时服务于编译工具链（静态视图）和操作系统加载器（动态视图），这是 ELF 灵活性的根本所在。

① ELF 头（ELF Header）

ELF 头位于文件偏移 0x00，是整个文件的"目录"，大小固定：32 位为 52 字节，64 位为 64 字节。由 Elf64_Ehdr 结构体定义（源自 <elf.h>）。

📄 来源：man7.org elf(5) — ELF header (Ehdr) 
typedef struct {
    /* e_ident[16] — 魔数和基本属性 */
    unsigned char e_ident[EI_NIDENT]; /* 16 字节标识数组 */
    uint16_t      e_type;             /* 文件类型 (ET_EXEC / ET_DYN …) */
    uint16_t      e_machine;          /* 目标架构 (EM_X86_64 / EM_ARM …) */
    uint32_t      e_version;          /* ELF 版本，始终为 EV_CURRENT(1) */
    ElfN_Addr     e_entry;            /* 程序入口虚拟地址 */
    ElfN_Off      e_phoff;            /* 程序头表在文件中的偏移 */
    ElfN_Off      e_shoff;            /* 节区头表在文件中的偏移 */
    uint32_t      e_flags;            /* 处理器相关标志 */
    uint16_t      e_ehsize;           /* ELF 头本身大小（字节） */
    uint16_t      e_phentsize;        /* 程序头表每条目大小 */
    uint16_t      e_phnum;            /* 程序头表条目数量 */
    uint16_t      e_shentsize;        /* 节区头表每条目大小 */
    uint16_t      e_shnum;            /* 节区头表条目数量 */
    uint16_t      e_shstrndx;         /* 节名字符串表的节区索引 */
} Elf64_Ehdr;

⚡ 第一性原理

ELF 头是文件的"身份证"——它用 16 字节的 e_ident 数组把文件的类别、字节序、版本、ABI 等基本特征编码成独立于具体机器的统一格式，后续所有偏移量都从这里出发。操作系统、链接器和调试器读取这个头之后，才能决定如何解释后面的所有字节。

② 节区头表（Section Header Table）

节区头表（SHT）是一个 Elf64_Shdr 结构体数组，每个结构体描述文件中的一个节（section）。它是链接器、汇编器和调试器使用的静态视图。运行时加载时，节区头表本身并非必需（stripped 的二进制文件可以去掉它）。

📄 来源：man7.org elf(5) — Section header (Shdr) 
typedef struct {
    uint32_t   sh_name;      /* 节名在 .shstrtab 中的字符串偏移 */
    uint32_t   sh_type;      /* 节类型 (SHT_PROGBITS / SHT_SYMTAB …) */
    uint64_t   sh_flags;     /* 属性标志 (SHF_ALLOC / SHF_WRITE / SHF_EXECINSTR …) */
    Elf64_Addr sh_addr;      /* 若被加载，节在内存中的起始虚拟地址 */
    Elf64_Off  sh_offset;    /* 节在文件中的偏移 */
    uint64_t   sh_size;      /* 节的字节大小 */
    uint32_t   sh_link;      /* 关联节的索引（语义依 sh_type 而定） */
    uint32_t   sh_info;      /* 附加信息（语义依 sh_type 而定） */
    uint64_t   sh_addralign; /* 内存对齐要求（0/1 表示无要求） */
    uint64_t   sh_entsize;   /* 若节为固定大小条目的表，此为每条目大小 */
} Elf64_Shdr;

sh_type 常见类型

类型宏含义
SHT_NULL空节，第 0 号节始终为此类型
SHT_PROGBITS程序数据（代码、初始化数据等），由程序定义语义
SHT_SYMTAB符号表（链接期，含所有符号）
SHT_DYNSYM动态符号表（运行期）
SHT_STRTAB字符串表
SHT_RELA带显式加数的重定位条目
SHT_REL不带显式加数的重定位条目
SHT_NOBITS不占文件空间的节（如 .bss）
SHT_DYNAMIC动态链接信息
SHT_NOTE辅助信息（如 Build-ID）
SHT_HASH符号哈希表
SHT_GNU_HASHGNU 扩展哈希表（更快）

sh_flags 标志位

标志含义
SHF_WRITE节在运行时可写
SHF_ALLOC节需被加载到内存中
SHF_EXECINSTR节包含可执行机器指令
SHF_MERGE节中数据可被合并以节省空间
SHF_STRINGS节中数据为 null 结尾字符串
SHF_LINK_ORDER对链接顺序有特殊要求
SHF_TLS线程本地存储
⚡ 第一性原理

节区头表是给工具看的——它把文件切成有意义的逻辑块，让链接器知道哪些代码需要合并、哪些符号需要解析、哪些重定位需要处理。节的存在是因为"不同属性的数据需要不同的处理方式"这一根本需求。

③ 程序头表（Program Header Table）

程序头表（PHT）是一个 Elf64_Phdr 结构体数组，描述可执行文件或共享对象如何被操作系统映射到进程虚拟地址空间。它是运行时视图，链接时生成，仅可执行文件和共享对象才有。

📄 来源：man7.org elf(5) — Program header (Phdr) 
typedef struct {
    uint32_t   p_type;    /* 段类型 (PT_LOAD / PT_DYNAMIC / PT_INTERP …) */
    uint32_t   p_flags;   /* 段权限标志 PF_R | PF_W | PF_X */
    Elf64_Off  p_offset;  /* 段在文件中的偏移 */
    Elf64_Addr p_vaddr;   /* 段在内存中的虚拟地址 */
    Elf64_Addr p_paddr;   /* 段的物理地址（嵌入式系统用，Linux 忽略） */
    uint64_t   p_filesz;  /* 段在文件中的字节大小 */
    uint64_t   p_memsz;   /* 段在内存中的字节大小（≥ p_filesz） */
    uint64_t   p_align;   /* 对齐要求（页大小的整数倍，通常 0x1000） */
} Elf64_Phdr;

p_type 常见段类型

类型作用
PT_NULL未使用的条目，程序头表可含多个
PT_LOAD可加载段——只有此类型才被 mmap 进内存。按 p_vaddr 升序排列
PT_DYNAMIC动态链接元数据（含 DT_NEEDED 等标签）
PT_INTERP动态链接器路径，如 /lib64/ld-linux-x86-64.so.2
PT_NOTE辅助信息（如 NT_GNU_BUILD_ID）
PT_PHDR程序头表自身的位置与大小
PT_TLS线程本地存储模板
PT_GNU_STACKLinux 内核用此标志控制栈的可执行权限（NX 位）
PT_GNU_RELRO标记链接后可置为只读的区域（RELRO 安全加固）

p_flags 权限标志

PF_R (0x4)
可读
PF_W (0x2)
可写
PF_X (0x1)
可执行

代码段通常为 PF_R | PF_X（5），数据段通常为 PF_R | PF_W（6）。

p_filesz vs p_memsz

当 p_memsz > p_filesz 时，超出的部分在内存中被零初始化。这正是 .bss 节不占文件空间的机制：BSS 数据属于数据段，但仅在内存中存在，文件大小为 0。

⚡ 第一性原理

程序头表是给操作系统内核/加载器看的——它以"段"为粒度将文件内容映射到虚拟内存，赋予权限（R/W/X），并驱动 mmap 系统调用。只有 PT_LOAD 段才会被实际加载进内存，这个设计节省了物理内存，并通过不同权限位提供了基本的安全隔离。

④ 节（Section）vs 段（Segment）——两种视图

ELF 的一个核心设计是：同一份数据被两套表以不同角度描述。

📄 参考：linuxvox.com — Sections vs Segments in ELF
维度节（Section）段（Segment）
目的链接、调试、分析加载到内存、执行
描述结构节区头表（SHT）程序头表（PHT）
使用者链接器 ld、汇编器 as、调试器 gdbOS 加载器、内核
内容粒度逻辑分组（代码、数据、符号…）一个或多个节的集合

关系一个段可包含多个节一个节可属于一个或多个段

节→段 映射关系（SVG 可视化）

⚡ 第一性原理

节和段是同一份数据的两种投影。节服务于"如何构建程序"（静态），段服务于"如何运行程序"（动态）。这种双视图设计使得同一个 ELF 文件可以被链接器精细操作、也可以被加载器高效映射，而无需维护两份数据副本。

⑤ 核心节区详解

以下是 ELF 可执行文件中最重要的标准节区，它们的名称、类型和标志由 ABI 规范定义。

.bss 为什么不占文件空间？

C 标准规定：未初始化或零初始化的全局/静态变量在程序启动时值为 0。ELF 利用这一点：把这些变量放入 .bss 节（SHT_NOBITS），节头记录其大小但不在文件中分配字节。OS 加载时，将 .bss 对应的内存页清零（通过 mmap 匿名映射或 memset）。一个拥有 10 MB 全局数组的程序，其文件大小可能只有几 KB。

// 以下变量进入 .bss，不占文件空间
static int big_array[1024 * 1024];  // 4 MB，.bss
int counter;                         // 4 B，.bss

// 以下变量进入 .data，占文件空间
int initialized = 42;               // .data
⚡ 第一性原理

每个节区的存在都是因为"不同种类的数据需要不同的处理策略"。代码只读可执行，数据可读写，字符串可共享合并，重定位信息是临时的……ELF 将这些差异编码进节的类型和标志，而不是强迫所有内容遵循同一规则。

⑥ 符号表与重定位

📄 来源：man7.org elf(5) — Symbol table & Relocation entries

符号表结构（Elf64_Sym）

typedef struct {
    uint32_t      st_name;  /* 符号名在 .strtab 中的偏移 */
    unsigned char st_info;  /* 符号类型（低4位）+ 绑定属性（高4位） */
    unsigned char st_other; /* 符号可见性 */
    uint16_t      st_shndx; /* 所属节区索引 */
    Elf64_Addr    st_value; /* 符号值（通常是地址） */
    uint64_t      st_size;  /* 符号大小（字节） */
} Elf64_Sym;

符号绑定属性（st_info 高 4 位）

值宏名含义
0STB_LOCAL局部符号，文件内可见，链接时不导出
1STB_GLOBAL全局符号，跨目标文件可见
2STB_WEAK弱符号，优先级低于全局符号，可被覆盖

值宏名含义

4STT_FILE源文件名
6STT_TLS线程本地存储对象

重定位条目结构

重定位是连接"符号引用"与"符号定义"的桥梁。目标文件（.o）中的跨文件引用地址未知，重定位条目记录"哪里需要填地址、用哪个符号"。

/* 不带加数（.rel.* 节用） */
typedef struct {
    Elf64_Addr r_offset; /* 需修正位置的偏移/虚拟地址 */
    uint64_t   r_info;   /* 高32位=符号表索引，低32位=重定位类型 */
} Elf64_Rel;

/* 带加数（.rela.* 节用，x86-64 常用） */
typedef struct {
    Elf64_Addr r_offset;
    uint64_t   r_info;
    int64_t    r_addend; /* 计算时的常量加数 */
} Elf64_Rela;

重定位过程示意

⚡ 第一性原理

符号表是程序的"名字簿"，重定位条目是"待填空的账单"。编译器在不知道其他文件地址的情况下先写下"TODO: 这里需要 printf 的地址"，链接器汇总所有目标文件后统一"结账"——填入正确地址。这种分阶段的间接寻址设计使得独立编译（separate compilation）成为可能。

⑦ 动态链接节区

动态链接允许多个进程共享同一份库代码，节省内存。核心机制依赖 .dynamic、.plt、.got 三个节区协同工作。

📄 来源：man7.org elf(5) — Dynamic tags (Dyn)

.dynamic 节 — 动态标签数组

typedef struct {
    Elf64_Sxword d_tag;  /* 标签类型 (DT_NEEDED / DT_SYMTAB …) */
    union {
        Elf64_Xword d_val; /* 整数值 */
        Elf64_Addr  d_ptr; /* 虚拟地址 */
    } d_un;
} Elf64_Dyn;

常见 DT_ 标签

标签含义
DT_NEEDED所依赖的共享库名称（在 .dynstr 中的偏移），可出现多次
DT_SYMTAB动态符号表（.dynsym）地址
DT_STRTAB动态字符串表（.dynstr）地址
DT_RELA / DT_REL动态重定位表地址
DT_PLTGOTPLT/GOT 地址
DT_JMPRELPLT 专用重定位表地址
DT_INIT / DT_FINI初始化/终止函数地址
DT_SONAME共享库自身的名称
DT_RPATH / DT_RUNPATH运行时库搜索路径
DT_BIND_NOW立即绑定所有符号（关闭延迟绑定）
DT_NULL标记数组结束

PLT + GOT 延迟绑定机制

⚡ 第一性原理

PLT+GOT 的延迟绑定是"懒计算"思想的应用：只有当函数第一次被调用时才解析其地址，避免程序启动时解析所有动态符号的开销。GOT 作为一层间接层，使得地址可在运行时被修改，而 .text 代码（只读）始终不变——这是位置无关代码（PIC）的核心机制。

⑧ ELF 的编译链接生成全流程

一个 C 源文件变成可执行 ELF，要经历四个阶段。每个阶段都有专用工具，GCC 驱动程序（gcc）默认会依次调用它们。

📄 参考：Handcrafting x64 ELF — From Specification to Bytes 📄 参考：Columbia CS4157 — Introduction to ELF

GCC 分步调用示例

# ① 预处理（停在此阶段）
cpp main.c -o main.i
# 或
gcc -E main.c -o main.i

# ② 编译（停在此阶段，生成汇编）
gcc -S main.i -o main.s

# ③ 汇编（停在此阶段，生成可重定位 ELF）
as main.s -o main.o
# 或
gcc -c main.c -o main.o

# ④ 链接（生成可执行 ELF）
ld main.o /lib/x86_64-linux-gnu/crt1.o \
   /lib/x86_64-linux-gnu/crti.o \
   /lib/x86_64-linux-gnu/crtn.o \
   -lc -dynamic-linker /lib64/ld-linux-x86-64.so.2 \
   -o main
# 或直接用 gcc 驱动
gcc main.c -o main

链接时节区合并

⚡ 第一性原理

编译流水线的每一步都是一次语言翻译：源码→中间表示→汇编→机器码→链接。每步只做最小必要的工作，并将结果通过标准格式（ELF）传递给下一步。这种管道化设计使得工具链高度可替换（可以用 clang 替代 gcc，用 lld 替代 ld），这是 UNIX 哲学的具体体现。

⑨ ELF 的加载与执行

当你执行 ./a.out，从 shell 到程序跑起来，内核和动态链接器共同完成 ELF 的加载过程。

加载过程时序（详细）

内存布局（运行时）

⚡ 第一性原理

ELF 加载是"按需映射"的过程——内核通过 mmap 将文件内容懒加载到虚拟地址空间，只有真正被访问的页才触发物理内存分配（缺页异常）。动态链接器则负责在运行时完成编译期未能确定的最终地址绑定。整个过程是"编译期静态分析 + 链接期部分解析 + 运行期最终确定"的三阶段延迟求值策略。

⑩ 常用 ELF 分析工具

Linux 提供了完整的 ELF 分析工具链，全部来自 GNU binutils 和系统工具。

工具常用命令功能
readelf readelf -h/-S/-l/-s/-d file 读取 ELF 头、节区表、程序头表、符号表、动态节
objdump objdump -d/-D/-x/-t file 反汇编、显示各节内容、符号表
nm nm -n/-u file 列出符号表，按地址排序或仅显示未定义符号
ldd ldd ./a.out 列出程序的动态库依赖（DT_NEEDED）
file file ./a.out 识别 ELF 类型、架构、链接方式
size size ./a.out 显示 .text / .data / .bss 的大小
strip strip -s ./a.out 去除 .symtab/.debug_* 等调试信息
objcopy objcopy -O binary file.elf file.bin ELF 格式转换，提取裸二进制
patchelf patchelf --set-interpreter ... file 修改 ELF 的动态链接器路径、RPATH 等
gdb gdb ./a.out 结合 .debug_* DWARF 信息进行源码级调试

常用 readelf 命令速查

# 查看 ELF 头
readelf -h /bin/ls

# 查看所有节区（Section Header Table）
readelf -S ./a.out

# 查看程序头表（Segment Header Table）
readelf -l ./a.out

# 查看符号表
readelf -s ./a.out

# 查看动态节
readelf -d ./a.out

# 查看重定位条目
readelf -r ./a.out

# 查看全部信息
readelf -a ./a.out

# 查看 .text 反汇编
objdump -d -M intel ./a.out

# 查看动态库依赖
ldd /bin/ls

# 显示各节大小
size ./a.out
⚡ 第一性原理

这些工具都是同一个思想的具体实现：按照 ELF 规范的结构定义，解析二进制文件中特定偏移处的字节序列，并以人类可读的格式呈现。ELF 自描述的设计（文件头包含所有表的偏移和大小）使得任何工具都可以独立解析文件，无需外部元数据。这是"自包含"设计原则的胜利。
