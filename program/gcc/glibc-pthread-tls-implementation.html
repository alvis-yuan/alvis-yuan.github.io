Chapter 01

glibc Pthread TLS 实现原理深度分析

基于 glibc NPTL 源码 · x86-64 Linux · 权威技术文档

📖引言与背景

线程本地存储（Thread-Local Storage，TLS）提供了一种机制，使得每个线程都能拥有全局变量的独立副本。它是 GCC 扩展关键字 __thread、C11 _Thread_local 以及 C++11 thread_local 的底层实现基础。

POSIX 标准明确规定："每个线程拥有自己的线程 ID、调度优先级与策略、errno 值、浮点环境、线程特定键/值绑定……" TLS 正是实现这一特性的核心机制。

一个典型用例是 errno：不同线程调用系统调用时，errno 的值互不干扰。在 glibc 中，errno 通过 __errno_location() 函数返回一个线程局部变量的地址来实现线程隔离。

📌 权威参考
Ulrich Drepper 撰写的 《ELF Handling For Thread-Local Storage》 是 ELF TLS 的权威规范文档。各架构处理器的 psABI 文档通常会引用此文档。glibc 实现严格遵循该规范。

Linux ELF TLS 的协作层次

完整的 TLS 实现需要编译器、链接器、动态链接器、操作系统内核和语言运行时的协同配合。

图 1-1：TLS 实现的五层协作架构
⚗ 第一性原理
TLS 本质上解决的是共享代码 + 私有数据的矛盾：多个线程共享同一份可执行代码，但每个线程需要操作自己独立的变量副本。最简单的解法是给每个线程分配一块私有内存区域，并为每个线程维护一个"从变量名到私有地址"的映射。glibc 的 TLS 正是将此映射硬编码为寄存器偏移（静态TLS）或通过索引数组（DTV/动态TLS）来实现，在性能与灵活性之间取得平衡。
Chapter 02

🏗️TLS 整体架构

glibc 的 TLS 实现可分为两大部分：静态 TLS（Static TLS） 和 动态 TLS（Dynamic TLS）。

  • 静态 TLS：在程序启动时即确定偏移量，通过线程寄存器（FS）的固定偏移直接访问，性能最优。适用于主程序及 DT_NEEDED 依赖库中的 TLS 变量。
  • 动态 TLS：通过 dlopen 动态加载的共享库所引入的 TLS 变量，偏移量在运行时才确定，需通过 DTV 索引和 __tls_get_addr 函数访问。
图 2-1：glibc TLS 整体内存架构图（x86-64 Variant II）
⚗ 第一性原理
TLS 架构的本质是空间换时间的分层设计：对于确定性强（启动时可知偏移）的变量，使用最快的寄存器相对寻址；对于不确定性高（运行时才知道）的变量，使用多一层间接的动态数组查找。这完全遵循计算机体系结构的基本原则——已知信息越多，访问越快。
Chapter 03

🔩线程控制块 (TCB)

在 x86-64 Linux + glibc 中，线程控制块（Thread Control Block，TCB） 对应的核心数据结构是 tcbhead_t（定义在 sysdeps/x86_64/nptl/tls.h），它是 struct pthread 的第一个成员（通过 union 重叠）。

/* sysdeps/x86_64/nptl/tls.h — glibc 源码 */
typedef struct
{
  void *tcb;          /* 指向 TCB 自身（即 struct pthread *）*/
  dtv_t *dtv;         /* 动态线程向量指针 */
  void *self;         /* 指向线程描述符（同 tcb） */
  int multiple_threads;/* 是否多线程标志 */
  int gscope_flag;    /* 全局作用域标志 */
  uintptr_t sysinfo;  /* vDSO 系统调用入口 */
  uintptr_t stack_guard;   /* 栈保护金丝雀值（%fs:0x28）*/
  uintptr_t pointer_guard; /* 指针保护值 */
  unsigned long int unused_vgetcpu_cache[2];
  unsigned int feature_1; /* X86 特性标志（IBT/SHSTK） */
  int __glibc_unused1;
  void *__private_tm[4];  /* TM ABI 保留字段 */
  void *__private_ss;     /* GCC split stack 支持 */
  unsigned long long int ssp_base; /* 影子栈标记 */
  __128bits __glibc_unused2[8][4] __attribute__((aligned(32)));
  void *__padding[8];
} tcbhead_t;
⚠️ 重要说明
x86-64 使用 TLS_TCB_AT_TP=1, TLS_DTV_AT_TP=0（Variant II），即线程指针（FS）直接指向 TCB 的起始地址，静态 TLS 块位于 TCB 的低地址方向（负偏移处）。stack_guard 字段位于 %fs:0x28，是 GCC 栈保护金丝雀值的来源。
图 3-1：tcbhead_t 字段内存偏移布局（x86-64，来源：sysdeps/x86_64/nptl/tls.h）
⚗ 第一性原理
TCB 的本质是线程的私有控制平面：它将"这是哪个线程"（self/tcb 自引用）、"如何找到该线程的所有私有数据"（dtv 指针）和"如何保护该线程"（stack_guard/pointer_guard）三者集中在一块固定的内存中，由 FS 寄存器直接寻址。这是最小化寻址层级的设计——从任意 TLS 访问到找到目标，最多只需 FS→TCB→DTV→TLS块 三跳。
Chapter 04

📐动态线程向量 (DTV)

DTV（Dynamic Thread Vector）是 glibc TLS 的核心索引结构，定义在 sysdeps/generic/dl-dtv.h。

/* sysdeps/generic/dl-dtv.h */
typedef union dtv
{
  size_t counter;      /* dtv[-1]: DTV 数组元素总数
                           dtv[0]:  当前 DTV 版本（generation counter） */
  struct
  {
    void *val;           /* 指向该模块的 TLS 块地址 */
    bool is_static;      /* 是否为静态 TLS 块 */
  } pointer;
} dtv_t;
🔑 关键技巧：负索引
DTV 使用了 C 语言的负指针技巧：dtv 指针指向实际数组的第 1 号元素，而元素 dtv[-1] 存储的是数组长度（counter），dtv[0] 存储版本号（generation counter），dtv[1] 开始才是各模块的 TLS 块指针。
图 4-1：DTV 数组结构（dtv[-1]=长度, dtv[0]=版本号, dtv[1..N]=各模块TLS指针）

DTV Generation Counter（版本机制）

每次 dlopen 或 dlclose 调用都会递增全局计数器 dl_tls_generation。__tls_get_addr 在访问前会比较 dtv[0].counter 与全局计数器：若不一致则触发 DTV 更新（_dl_update_slotinfo），实现懒惰式 DTV 扩展。

⚗ 第一性原理
DTV 本质是以模块ID为下标的稀疏指针数组，解决了"N 个模块、每个线程需要 N 份私有数据"的索引问题。版本号机制（generation counter）让 DTV 的扩容变得懒惰——只有真正需要访问某动态模块的 TLS 时，该线程的 DTV 才会被扩展，避免了 dlopen 时同步更新所有线程 DTV 的高昂代价。
Chapter 05

🧵struct pthread 结构

struct pthread（定义在 nptl/descr.h）是 glibc 线程描述符，它与 tcbhead_t 通过 union 共享起始地址。在 x86-64 上（TLS_TCB_AT_TP=1），线程指针（FS）直接指向该结构的起始位置。

/* nptl/descr.h （精简版）*/
struct pthread
{
  union
  {
#if !TLS_DTV_AT_TP
    tcbhead_t header;  /* x86-64: TCB 与 struct pthread 共享起始地址 */
#else
    struct {            /* AArch64 等 Variant I 架构 */
      int multiple_threads;
      int gscope_flag;
    } header;
#endif
    void *__padding[24];
  };

  list_t list;             /* 线程链表节点 */
  pid_t tid;              /* 内核线程 ID */
  /* ... 调度、信号、取消等字段 ... */

  /* pthread_key 两级数组 */
  struct pthread_key_data
  {
    uintptr_t seq;         /* 序列号（用于 stale 检测）*/
    void *data;            /* 用户数据指针 */
  } specific_1stblock[PTHREAD_KEY_2NDLEVEL_SIZE]; /* 32个快速槽 */

  struct pthread_key_data *specific[PTHREAD_KEY_1STLEVEL_SIZE];
  bool specific_used;

  /* ... 更多字段 ... */
};
⚗ 第一性原理
struct pthread 与 tcbhead_t 共享同一起始地址的设计，使得"通过 FS 找线程描述符"和"通过 FS 找 DTV"成为零额外代价的操作——FS 就是 struct pthread*，也是 tcbhead_t*。这是将两个逻辑上相关的数据结构物理合并以消除间接寻址的经典手法。
Chapter 06

🗺️Variant II 内存布局

Ulrich Drepper 的《ELF TLS 处理》文档定义了两种 TLS 布局变体。x86-64 使用 Variant II：静态 TLS 块位于线程指针（TP）的负偏移方向（低地址侧），TCB 位于 TP 指向处。

图 6-1：Variant II TLS 内存布局（x86-64，TP=TCB起始，静态TLS在负偏移方向）
📌 非主线程的特殊之处
对于 pthread_create 创建的线程，allocate_stack() 在分配线程栈时，会将静态 TLS 块和 TCB（struct pthread）放置在栈空间的高地址末端（栈顶附近），整个区域统一分配，节省独立堆分配的开销。主线程的 TCB 则由动态链接器在堆上分配。
⚗ 第一性原理
Variant II 布局的根本逻辑是保持 TP 的语义稳定性：TP（FS）始终指向 TCB（线程描述符）的起始处，这让内核上下文切换代码可以简单地保存/恢复一个寄存器值（FS base）来切换线程上下文。TLS 数据"向下延伸"（负偏移），让各模块 TLS 偏移相互独立，且可以在程序启动时由链接器静态确定，无需运行时协调。
Chapter 07

⚙️FS 段寄存器机制

x86-64 上 Intel 80386 引入 FS 和 GS 作为用户自定义段寄存器。Linux 将 FS 用作线程指针寄存器（Windows/macOS 使用 GS），通过它访问每个线程的私有上下文。

设置 FS：TLS_INIT_TP 宏

在 x86-64 上，通过 arch_prctl(ARCH_SET_FS, tcb_addr) 系统调用来设置 FS 寄存器的基地址：

/* sysdeps/x86_64/nptl/tls.h */
#define TLS_INIT_TP(thrdescr) \
  ({ void *_thrdescr = (thrdescr);           \
     tcbhead_t *_head = _thrdescr;            \
     int _result;                             \
                                               \
     _head->tcb = _thrdescr;                   \
     _head->self = _thrdescr;                  \
                                               \
     asm volatile ("syscall"                  \
       : "=a" (_result)                        \
       : "0" ((unsigned long) __NR_arch_prctl),\
         "D" ((unsigned long) ARCH_SET_FS),   \
         "S" (_thrdescr)                       \
       : "memory", "cc", "r11", "cx");      \
     _result == 0;                            \
  })

读取 FS：THREAD_SELF 宏

/* GCC >= 6.0 版本的高效实现 */
#define THREAD_SELF \
  (*(struct pthread *__seg_fs *) offsetof(struct pthread, header.self))

/* 等效于（展开后）：*/
struct pthread *self;
asm ("mov %%fs:%c1,%0" : "=r" (self)
     : "i" (offsetof(struct pthread, header.self)));
/* 编译为单条指令：mov %fs:0x10, %rax */

内核上下文切换

FS 的基地址存储在 MSR（Model Specific Register）MSR_FS_BASE 中。当内核进行线程上下文切换时，会保存当前线程的 FS 基地址并恢复下一个线程的 FS 基地址，从而实现"每个线程看到不同的 FS 内容"。

图 7-1：FS 寄存器与线程上下文切换机制
⚗ 第一性原理
FS 寄存器的本质是一个硬件支持的"线程 ID → 私有数据基址"的映射。内核在上下文切换时切换 MSR_FS_BASE，等效于切换"当前线程私有数据的宇宙坐标系原点"。用户态的所有 TLS 访问都是在这个相对坐标系下进行的，绝对地址在不同线程间自动不同——这是最零开销的线程隔离手段。
Chapter 08

🧱静态 TLS 块

ELF 文件使用 PT_TLS 程序头段来描述 TLS 数据的初始镜像：

  • p_offset：TLS 初始镜像的文件偏移（.tdata 节数据）
  • p_filesz：有初始值的 TLS 数据大小（.tdata）
  • p_memsz：TLS 块总大小（.tdata + .tbss），后 p_memsz-p_filesz 字节初始化为零
  • p_align：对齐要求

动态链接器收集所有模块（主程序 + DT_NEEDED 库）的 PT_TLS 段，分配连续的静态 TLS 区域，并将初始化镜像复制进去。

TLS 符号的 st_value 表示相对于其所在模块 PT_TLS 段起始地址的偏移，而不是普通的虚拟地址。

图 8-1：PT_TLS 段的加载与静态 TLS 块初始化流程
⚗ 第一性原理
静态 TLS 的核心思想是将"运行时的每线程副本"映射为"编译期的固定偏移"。只要各线程的 TP 不同，相同的偏移指令就能访问不同的内存地址——即用位置相对性替代绝对地址寻址，这是实现线程隔离零运行时开销的关键。
Chapter 09

⚡Local Exec TLS 模型

最高性能 仅限可执行文件

Local Exec 模型适用于 TLS 变量定义在主可执行文件中（或具有非默认可见性的声明）的情形。变量相对于 TP 的偏移量在链接时即为编译期常量，无需任何运行时计算。

// C 源码
__thread int local_var = 42;

// x86-64 编译结果（本地执行模型）
// 重定位类型：R_X86_64_TPOFF32
movl  %fs:local_var@TPOFF, %eax
// 运行时展开为（假设 tlsoffset = -4）：
movl  %fs:-4, %eax         // 单条指令，最高效
图 9-1：Local Exec 模型访问流程（性能最优）
条件编译器选择 Local Exec 的时机
定义在可执行文件中-fno-pic/-fpie 模式下的 TLS 定义
非默认可见性声明__attribute__((visibility("hidden"))) 的外部声明
⚗ 第一性原理
Local Exec 是 TLS 访问的理想极限：当所有信息（模块偏移）在链接时已知时，访问退化为普通的寄存器相对寻址，与访问全局变量的代价完全相同。这是"信息越多，抽象层越薄，性能越高"原则的体现。
Chapter 10

🔍Initial Exec TLS 模型

次高性能 程序启动时加载的 DSO

当 TLS 变量定义在随程序启动加载的共享库（DT_NEEDED）中，其相对 TP 的偏移在程序启动时确定（不是链接时），但之后不再改变。此偏移通过 GOT 表存储（GOTTPOFF）。

// x86-64 Initial Exec 代码序列
// 重定位：R_X86_64_GOTTPOFF（GOT 中存 TP 偏移）
movq  ref@GOTTPOFF(%rip), %rax   // 从 GOT 加载 TP 偏移量
movl  %fs:(%rax), %eax           // 用 TP + offset 访问变量

// GOT 中动态重定位类型：R_X86_64_TPOFF64
// 动态链接器在启动时将 l_tls_offset 写入 GOT 对应槽位
图 10-1：Initial Exec 模型（GOT 存储 TP 偏移，启动时由动态链接器填写）
⚠️ DF_STATIC_TLS 限制
使用 Initial Exec 模型的共享库（包括使用 __attribute__((tls_model("initial-exec"))) 的代码）会被链接器设置 DF_STATIC_TLS 标志。这类库通常不能被 dlopen 安全加载（因为静态 TLS 区域大小在程序启动时就已固定）。glibc 保留了少量"TLS 剩余空间"以允许小型 IE 库的 dlopen，但 musl 等其他 libc 不支持此机制。
⚗ 第一性原理
Initial Exec 是"链接时不知道、启动时可以知道"的折中方案：用 GOT 表作为中间层，让动态链接器在启动时填写偏移量，之后访问退化为一次 GOT 读取加一次 TP 相对寻址。GOT 是实现"延迟绑定"的经典模式——将运行时才能确定的值用间接寻址来存储。
Chapter 11

🌐Global Dynamic TLS 模型

最通用 支持 dlopen

Global Dynamic（GD）是最灵活的模型，适用于 -fpic/-fPIC 编译、TLS 变量可能定义在任意共享库（包括 dlopen 加载）的情形。编译器生成对 __tls_get_addr 的调用。

// x86-64 Global Dynamic 代码序列
// 重定位：R_X86_64_TLSGD
data16 leaq  var@tlsgd(%rip), %rdi  // 加载 tls_index 指针到 %rdi
// data16 前缀为 NOP，占位使总长为 16 字节（便于链接器内联优化）
.value 0x6666
rex64 call   __tls_get_addr@PLT      // 调用获取 TLS 变量地址
movl  (%rax), %eax                   // 读取变量值

// GOT 中存储 tls_index 结构体（两个 GOT 条目）：
// GOT[n+0]: R_X86_64_DTPMOD64 → 运行时填入模块 ID
// GOT[n+1]: R_X86_64_DTPOFF64 → 运行时填入模块内偏移
图 11-1：Global Dynamic 模型通过 __tls_get_addr 间接访问 TLS 变量

__tls_get_addr 核心实现

/* elf/dl-tls.c */
void *
__tls_get_addr (tls_index *ti)
{
  dtv_t *dtv = THREAD_DTV();

  /* 快速路径：版本号匹配，DTV 是最新的 */
  if (__glibc_unlikely (dtv[0].counter != GL(dl_tls_generation)))
    return update_get_addr(ti);  /* 慢路径：更新 DTV */

  void *p = dtv[ti->ti_module].pointer.val;

  /* 懒加载：该模块 TLS 块尚未分配 */
  if (__glibc_unlikely (p == TLS_DTV_UNALLOCATED))
    return tls_get_addr_tail(ti->ti_offset, dtv, NULL);

  return (char *) p + ti->ti_offset;
}
⚗ 第一性原理
Global Dynamic 是 TLS 访问的"最大公约数"方案：用函数调用换取对所有模块的均匀支持能力。__tls_get_addr 包含版本检查和懒加载两种"慢路径"，但在热点代码中，DTV 通常是最新的（版本号匹配），函数只需做一次数组查找和加法。这是"乐观假设 + 慢路径兜底"的设计范式。
Chapter 12

🔧Local Dynamic TLS 模型

DSO 非可抢占符号 批量优化

Local Dynamic（LD）是 Global Dynamic 的优化变体：当一个函数需要访问多个属于同一模块的非可抢占 TLS 变量时，可以只调用一次 __tls_get_addr 获取模块 TLS 块的基址，然后用编译期常量偏移 dtpoff 来访问各变量：

// 示例：同一模块两个静态 TLS 变量
static __thread int def0, def1;
int f() { return ++def0 + ++def1; }

// 编译为 Local Dynamic：
leaq  def0@tlsld(%rip), %rdi   // R_X86_64_TLSLD: 模块 TLS 块基址请求
call  __tls_get_addr@PLT       // 只调用一次！返回模块 TLS 块首地址
movl  def0@dtpoff(%rax), %edx  // 编译期常量偏移访问 def0
movl  def1@dtpoff(%rax), %eax  // 编译期常量偏移访问 def1
图 12-1：Local Dynamic 通过共享一次 __tls_get_addr 调用优化多变量访问
⚗ 第一性原理
Local Dynamic 的优化逻辑是提取公共子表达式：同一模块的所有 TLS 变量共享同一个模块基址，只需获取一次，后续访问退化为加法运算。这是编译器常规的"循环不变量外提"（LICM）思想在 TLS 访问上的应用。
Chapter 13

🚀主线程 TLS 初始化

对于动态链接的 ELF 可执行文件，主线程的 TLS 初始化由动态链接器（ld.so，即 elf/rtld.c）完成，发生在 main() 函数执行之前。

图 13-1：主线程 TLS 初始化五步流程（elf/rtld.c → elf/dl-tls.c）

_dl_allocate_tls_init 核心逻辑

/* elf/dl-tls.c — _dl_allocate_tls_init() 精简版 */
void *
_dl_allocate_tls_init (void *result)
{
  dtv_t *dtv = GET_DTV(result);

  /* 检查并可能扩容 DTV 数组 */
  if (dtv[-1].counter < GL(dl_tls_max_dtv_idx))
    dtv = _dl_resize_dtv(dtv);

  /* 遍历所有已加载模块，初始化各模块的 TLS 块 */
  for each map in dl_tls_dtv_slotinfo_list:
    void *dest = (char *) result + map->l_tls_offset;
    dtv[map->l_tls_modid].pointer.val = dest;

    /* 复制 .tdata 初始化镜像 + 零初始化 .tbss */
    memset(mempcpy(dest, map->l_tls_initimage,
                         map->l_tls_initimage_size),
           '\0', map->l_tls_blocksize - map->l_tls_initimage_size);

  dtv[0].counter = maxgen; /* 设置 generation counter */
  return result;
}
⚗ 第一性原理
主线程 TLS 初始化的本质是将编译期的"模板数据"（PT_TLS 初始镜像）实例化为运行期的"线程私有副本"，并建立 DTV 这个索引结构。整个过程类似于"克隆一份带坐标系的数据空间"：每个线程克隆一份数据，但坐标系（偏移关系）是编译器和链接器协商好的，运行时只需填入绝对基址即可。

子线程将 TCB 和静态 TLS 放在栈顶，是空间局部性的极致利用：线程的控制元数据（struct pthread）与线程的执行栈物理相邻，减少了内存碎片和 TLB 压力。CLONE_SETTLS 标志让内核直接为新线程设置 FS，避免了新线程启动后再调用 arch_prctl 的额外开销——内核在线程诞生之初即"出厂配置完毕"。

🔗__tls_get_addr 完整实现

__tls_get_addr 是 glibc 动态 TLS 访问的统一入口，定义在 elf/dl-tls.c。它接收一个 tls_index 指针（存于 GOT 中），返回 TLS 变量的运行时地址。

/* sysdeps/x86_64/dl-tls.h — tls_index 结构体 */
typedef struct dl_tls_index
{
  uint64_t ti_module;  /* 模块 ID（由 R_X86_64_DTPMOD64 动态重定位填入）*/
  uint64_t ti_offset;  /* 模块内 TLS 偏移（R_X86_64_DTPOFF64 或编译期常量）*/
} tls_index;

/* elf/dl-tls.c — 完整实现（含 dtv 更新与懒加载） */
void *
__tls_get_addr (tls_index *ti)
{
  dtv_t *dtv = THREAD_DTV();

  /* ① 快速路径：generation 匹配，DTV 是最新版本 */
  if (__glibc_unlikely (dtv[0].counter != GL(dl_tls_generation)))
    return update_get_addr(ti);  /* 慢路径：更新 DTV */

  /* ② 用模块 ID 索引 DTV */
  void *p = dtv[ti->ti_module].pointer.val;

  /* ③ 懒加载路径：该 slot 尚未分配 */
  if (__glibc_unlikely (p == TLS_DTV_UNALLOCATED))
    return tls_get_addr_tail(ti->ti_offset, dtv, NULL);
    /* tls_get_addr_tail 调用 allocate_and_init() 堆分配该模块 TLS 块 */

  /* ④ 正常返回：TLS 块首地址 + 变量偏移 */
  return (char *) p + ti->ti_offset;
}

/* allocate_and_init() — 动态 TLS 块分配 */
static struct dtv_pointer
allocate_and_init (struct link_map *map)
{
  /* 按模块 TLS 对齐要求 memalign 分配 */
  struct dtv_pointer result = allocate_dtv_entry(
      map->l_tls_align, map->l_tls_blocksize);

  /* 复制初始化镜像 + 零化 .tbss */
  memset(mempcpy(result.val, map->l_tls_initimage,
                       map->l_tls_initimage_size),
         '\0', map->l_tls_blocksize - map->l_tls_initimage_size);
  return result;
}
图 15-1：__tls_get_addr 执行决策树（快速路径 vs 慢路径）
⚗ 第一性原理
__tls_get_addr 采用乐观快速路径 + 悲观慢路径的双层设计：在绝大多数情况下（DTV 最新、TLS 已分配），函数只做一次计数器比较和两次内存读取后返回。只有在真正需要时（dlopen 后首次访问、DTV 过旧）才进入代价高昂的更新/分配路径。这是"以常见情况为基准优化，以罕见情况为正确性兜底"的工程智慧。

🗄️两级数组实现（pthread_setspecific）

每个线程的 struct pthread 中存储线程私有的键值对数据，采用两级数组结构以节省内存：

  • specific_1stblock[32]：前 32 个键（PTHREAD_KEY_2NDLEVEL_SIZE）直接存储，无需额外分配
  • specific[32]：超出 32 个键时，使用二级指针数组（最多支持 1024 个键）
/* nptl/pthread_setspecific.c — 精简版 */
int ___pthread_setspecific(pthread_key_t key, const void *value)
{
  struct pthread *self = THREAD_SELF;

  if (__glibc_likely (key < PTHREAD_KEY_2NDLEVEL_SIZE)) {
    /* 快速路径：前 32 个键，直接用 specific_1stblock */
    struct pthread_key_data *level2 = &self->specific_1stblock[key];
    level2->seq = __pthread_keys[key].seq;
    level2->data = (void *) value;
  } else {
    /* 慢路径：key >= 32，使用二级数组 */
    unsigned int idx1st = key / PTHREAD_KEY_2NDLEVEL_SIZE; /* 一级索引 */
    unsigned int idx2nd = key % PTHREAD_KEY_2NDLEVEL_SIZE; /* 二级索引 */

    struct pthread_key_data *level2 =
        THREAD_GETMEM_NC(self, specific, idx1st);

    if (level2 == NULL) {  /* 按需分配二级数组 */
      level2 = (struct pthread_key_data *)
          calloc(PTHREAD_KEY_2NDLEVEL_SIZE, sizeof(*level2));
      THREAD_SETMEM_NC(self, specific, idx1st, level2);
    }
    level2[idx2nd].seq  = __pthread_keys[key].seq;
    level2[idx2nd].data = (void *) value;
  }
  return 0;
}
图 17-1：pthread_key 两级数组存储结构（前32个键快速访问，超出后按需分配二级数组）
⚗ 第一性原理
两级数组是稀疏存储的经典实现：多数程序使用的 pthread_key 数量远小于 1024（上限），直接分配 1024 槽会浪费内存。两级结构让前 32 个槽"零成本内联"（嵌在 struct pthread 中），更多键才触发按需分配。seq 字段的奇偶校验实现了"零额外开销的 key 有效性验证"——不需要单独的布尔标志位。
