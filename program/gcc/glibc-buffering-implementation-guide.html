Chapter 01

什么是 I/O 缓冲

从系统调用代价出发，理解缓冲层的存在意义

1.1 为什么需要缓冲？

在 Linux 系统中，每一次 read() 或 write() 都是一次系统调用——从用户态切换到内核态，这个切换本身就有数百至数千纳秒的开销。如果每输出一个字符都进行一次系统调用，代价是灾难性的。

glibc 的 stdio 层在应用程序与内核之间维护一块用户态缓冲区，将多次小规模写入积攒成一次大规模 I/O，极大地降低了系统调用次数。

1.2 缓冲层的位置

glibc 的 I/O 缓冲属于用户态缓冲，与下面几个缓冲层完全独立：

缓冲层位置管理者
glibc stdio 缓冲用户态进程内存程序员 / glibc
内核页缓存（Page Cache）内核内存Linux 内核
设备硬件缓冲磁盘控制器缓存硬件固件
官方原文

"Characters that are written to a stream are normally accumulated and transmitted asynchronously to the file in a block, instead of appearing as soon as they are output by the application program."
— The GNU C Library Manual §12.20

🔗 GNU C Library §12.20 Stream Buffering
第一性原理

系统调用是昂贵的

I/O 缓冲的本质是：用空间换时间。每次系统调用都会触发用户态→内核态的上下文切换（寄存器保存/恢复、TLB 刷新等），代价高昂。缓冲将 N 次小写入合并为 1 次大写入，使系统调用次数降至 1/N，这是所有缓冲机制的根本动力。

Chapter 02

三种缓冲模式

无缓冲、行缓冲、全缓冲的定义与触发条件

2.1 官方定义

GNU C Library 定义了三种缓冲策略（来自官方手册 §12.20.1）：

GNU C Library Manual §12.20.1 — 原文

• 无缓冲（Unbuffered）：Characters written to or read from an unbuffered stream are transmitted individually to or from the file as soon as possible.

• 行缓冲（Line Buffered）：Characters written to a line buffered stream are transmitted to the file in blocks when a newline character is encountered.

• 全缓冲（Fully Buffered）：Characters written to or read from a fully buffered stream are transmitted to or from the file in blocks of arbitrary size.

2.2 三种模式对比图

2.3 默认模式选择规则

glibc 在打开流时通过 isatty() 探测文件描述符是否连接到终端，据此自动选择缓冲模式：

流默认模式原因
stderr无缓冲（_IONBF）错误信息必须立即可见
stdout（连接终端）行缓冲（_IOLBF）交互式程序每行即时显示
stdout（重定向到文件/管道）全缓冲（_IOFBF）最大化吞吐量
普通文件（fopen）全缓冲（_IOFBF）最大化文件 I/O 效率
stdin（连接终端）行缓冲（_IOLBF）支持行编辑
重要陷阱

当 stdout 被重定向到管道或文件时（如 ./prog | grep xxx），它会从行缓冲切换为全缓冲！这是许多程序"输出不及时"的根本原因。

第一性原理

模式选择的本质是：交互性 vs 吞吐量

三种模式代表了对同一个矛盾的不同权衡：响应延迟与系统调用开销。终端是交互场景，要求低延迟，因此用行缓冲（或无缓冲）；文件/管道是批量场景，要求高吞吐，因此用全缓冲。isatty() 检测就是这个权衡的自动执行。

Chapter 03

FILE 结构体剖析

glibc 中 _IO_FILE 的内部布局与各字段含义

3.1 FILE 的真实身份

在 glibc 中，FILE 实际上是 struct _IO_FILE 的别名，定义于 libio/bits/types/struct_FILE.h。这是整个缓冲机制的核心数据结构。

/* glibc/libio/bits/types/struct_FILE.h（节选，附中文注释）*/
struct _IO_FILE
{
  int   _flags;          /* 高位是 _IO_MAGIC；低位是状态标志 */

  /* ── 读缓冲区指针组 ── */
  char *_IO_read_ptr;    /* 当前读位置（下一个待读字节） */
  char *_IO_read_end;    /* 读区域结束（已从内核取回的末尾） */
  char *_IO_read_base;   /* 读区域起始（含 putback 区） */

  /* ── 写缓冲区指针组 ── */
  char *_IO_write_base;  /* 写区域起始 */
  char *_IO_write_ptr;   /* 当前写位置（下一个写入位置） */
  char *_IO_write_end;   /* 写区域结束（控制何时触发 overflow） */

  /* ── 总缓冲区边界 ── */
  char *_IO_buf_base;    /* 分配的缓冲区起始地址 */
  char *_IO_buf_end;     /* 分配的缓冲区结束地址 */

  /* ── putback/备份区 ── */
  char *_IO_save_base;   /* 非当前读区域的起始 */
  char *_IO_backup_base; /* 备份区第一个有效字节 */
  char *_IO_save_end;    /* 非当前读区域的结束 */

  int   _fileno;         /* 底层文件描述符（fd） */
  char  _shortbuf[1];    /* 无缓冲模式使用的单字节缓冲 */
  /* ... 其他字段省略 ... */
};

3.2 缓冲区内存布局可视化

3.3 getc/putc 的快路径宏

glibc 使用宏实现 getc 和 putc 的快路径，直接操作指针，避免函数调用开销：

/* glibc/libio/bits/types/struct_FILE.h */

/* getc 快路径：read_ptr 未到 read_end 时直接返回字节 */
#define __getc_unlocked_body(_fp)  \
  (__glibc_unlikely ((_fp)->_IO_read_ptr >= (_fp)->_IO_read_end)  \
   ? __uflow (_fp)                      /* 慢路径：触发内核读 */  \
   : *(unsigned char *) (_fp)->_IO_read_ptr++)  /* 快路径：指针++ */

/* putc 快路径：write_ptr 未到 write_end 时直接写入 */
#define __putc_unlocked_body(_ch, _fp)  \
  (__glibc_unlikely ((_fp)->_IO_write_ptr >= (_fp)->_IO_write_end)  \
   ? __overflow (_fp, (unsigned char) (_ch))  /* 慢路径：触发刷新 */  \
   : (unsigned char) (*(_fp)->_IO_write_ptr++ = (_ch)))  /* 快路径 */
设计亮点

快路径（fast path）：指针比较 + 指针移动，纯用户态操作，无任何系统调用。
慢路径（slow path）：只有在缓冲区满（__overflow）或空（__uflow）时才触发，进而调用 write()/read() 系统调用。

第一性原理

缓冲的本质是指针状态管理

glibc 缓冲机制的核心是维护一组指针（buf_base/end、write_base/ptr/end、read_base/ptr/end），通过比较指针位置判断缓冲区状态，无需任何锁或原子操作即可完成"是否需要刷新/填充"的判断。这是高性能 I/O 的关键设计。

Chapter 05

全缓冲实现原理

_IOFBF 模式下的缓冲区管理与 BUFSIZ

5.1 全缓冲的工作方式

在全缓冲（_IOFBF）模式下，write_end 被设置为 buf_end，意味着只有当缓冲区被完全填满时，才会触发 __overflow，进而调用 write() 系统调用。

官方手册原文

"Characters written to or read from a fully buffered stream are transmitted to or from the file in blocks of arbitrary size."
— GNU C Library Manual §12.20.1

/* glibc/libio/oldfileops.c 注释（第62行）*/
/* We also have _IO_write_end == _IO_buf_end, but only in fully buffered mode. */

/* 全缓冲模式的指针初始化（setvbuf 设置 _IOFBF 时）*/
fp->_IO_write_base = buf;
fp->_IO_write_ptr  = buf;
fp->_IO_write_end  = buf + size;  /* == _IO_buf_end */
fp->_IO_buf_base   = buf;
fp->_IO_buf_end    = buf + size;

5.2 全缓冲写入过程图解

5.3 BUFSIZ 与最优缓冲大小

BUFSIZ 在大多数 Linux 系统上为 8192 字节。但官方手册建议使用 fstat 获取 st_blksize 作为更优的缓冲大小：

#include <stdio.h>
#include <sys/stat.h>

int main() {
    FILE *fp = fopen("data.bin", "wb");
    struct stat st;
    fstat(fileno(fp), &st);

    /* st_blksize 是文件系统的最优块大小，用它做缓冲更高效 */
    char *mybuf = malloc(st.st_blksize);
    setvbuf(fp, mybuf, _IOFBF, st.st_blksize);

    /* ... 写入操作 ... */
    fclose(fp);
    free(mybuf);
}
官方建议

"Actually, you can get an even better value to use for the buffer size by means of the fstat system call: it is found in the st_blksize field of the file attributes."
— GNU C Library Manual §12.20.3

第一性原理

全缓冲 = 最大化批处理规模

全缓冲的目标是使每次系统调用携带尽可能多的数据。缓冲区大小应匹配文件系统的块大小（st_blksize），使每次 write() 都能完整写入一个块，避免内核侧的读改写（read-modify-write）操作，这是全缓冲效率最大化的物理基础。

Chapter 06

行缓冲实现原理

_IOLBF 模式：newline 触发机制与特殊行为

6.1 行缓冲的核心机制

行缓冲模式下，write_end 被设置为 buf_base（即缓冲区起始），这迫使每次 putc 都走"慢路径"（__overflow），在慢路径中再检查字符是否为换行符。

glibc 源码注释（oldfileops.c 第81-87行）

"LINE BUFFERED OUTPUT:
During line buffered output, _IO_write_base==base() && epptr()==base().
However, ptr() may be anywhere between base() and ebuf().
This forces a call to filebuf::overflow(int C) on every put.
If there is more space in the buffer, and C is not a '\n',
then C is inserted, and pptr() incremented."

6.2 行缓冲写入流程图

6.3 行缓冲的特殊规则

来自 setbuf(3) Linux man page：

行缓冲触发条件（来自 man 手册原文）

"when it is line buffered, characters are saved up until a newline is output or input is read from any stream attached to a terminal device (typically stdin)."

即：从终端读取 stdin 时，也会触发所有行缓冲流的刷新！

flushlbf 专用函数

glibc 提供了 _flushlbf()（声明于 stdio_ext.h）来刷新所有行缓冲流，这在从终端读取前隐式调用，确保提示符先于读操作显示。

/* 行缓冲触发示例 */
#include <stdio.h>

int main() {
    /* stdout 连接终端时是行缓冲 */
    printf("请输入姓名：");   /* 无\n → 正常情况下不刷新 */
    /* 但读取 stdin 前，glibc 会自动调用 _flushlbf() */
    /* 所以提示符会正确显示 */
    char name[64];
    fgets(name, sizeof(name), stdin);  /* ← 此处触发 stdout 的行缓冲刷新 */
    printf("你好, %s\n", name);
}
第一性原理

行缓冲 = 以换行符为自然语义边界的批处理

人类与终端的交互以"行"为单位——输入一行命令、显示一行结果。行缓冲将这个语义边界（换行符 \n）映射为 I/O 批处理边界，在兼顾交互实时性的同时，将同一行内的多个字符合并为一次系统调用。这是对"终端是交互设备"这一物理事实的直接建模。

Chapter 07

无缓冲实现原理

_IONBF：_shortbuf 与立即传输

7.1 无缓冲的实现方式

无缓冲模式下，glibc 使用 _shortbuf[1]——一个嵌入在 _IO_FILE 结构体中的单字节数组——作为"临时中转"，但每次写入后立即调用 write()。

/* glibc/libio/oldfileops.c 注释（第88-90行）*/
/* UNBUFFERED STREAMS:
   If a filebuf is unbuffered(), the _shortbuf[1] is used as the buffer. */

/* 无缓冲模式下的指针设置（setvbuf 设置 _IONBF 时）*/
fp->_IO_buf_base  = fp->_shortbuf;
fp->_IO_buf_end   = fp->_shortbuf + 1;  /* 缓冲区大小 = 1 字节 */
fp->_IO_write_base = fp->_shortbuf;
fp->_IO_write_ptr  = fp->_shortbuf;
fp->_IO_write_end  = fp->_shortbuf;    /* write_end == write_base → 每次都 overflow */

7.2 无缓冲工作示意图

7.3 用 setvbuf 设置无缓冲

#include <stdio.h>

int main() {
    /* 将 stdout 设置为无缓冲，适用于实时监控场景 */
    setvbuf(stdout, NULL, _IONBF, 0);

    /* 等价于老式写法（不推荐） */
    /* setbuf(stdout, NULL); */

    printf("实时输出");  /* 立即显示，无需\n */
}
何时使用无缓冲？

• 错误日志：确保崩溃前消息已输出（stderr 默认就是）
• 实时监控：每个字符/字节需即时可见
• 交互终端：精确控制输出时机（通常行缓冲已足够）

第一性原理

无缓冲 = 以实时性换取吞吐量

无缓冲是"不积攒、立即发送"的极端策略，将系统调用开销最大化，但将延迟最小化至零。stderr 之所以默认无缓冲，是因为错误信息的可靠送达（不因缓冲区未满而丢失）比性能更重要。这体现了"安全优先"的设计原则。

缓冲区刷新机制

fflush、自动刷新条件与 __fpurge

8.1 自动刷新的五种场景

根据 GNU C Library 手册 §12.20.2，以下情况会自动刷新输出缓冲区：

8.2 fflush 函数

官方手册原文 §12.20.2

"This function causes any buffered output on stream to be delivered to the file. If stream is a null pointer, then fflush causes buffered output on all open output streams to be flushed."

#include <stdio.h>

int main() {
    printf("进度: 50%%");
    fflush(stdout);          /* 强制立即输出（无\n时） */
    do_long_computation();

    fflush(NULL);             /* 刷新所有打开的输出流 */

    /* 仅刷新行缓冲流（glibc 扩展，来自 stdio_ext.h）*/
    _flushlbf();
}

8.3 __fpurge：丢弃缓冲内容

#include <stdio_ext.h>

/* __fpurge：不写入底层文件，直接丢弃缓冲区内容 */
/* 对读模式：丢弃已读入但未消费的数据 */
/* 对写模式：丢弃未写出的数据（不会写入文件！） */
__fpurge(fp);

/* 查询缓冲状态（来自 stdio_ext.h，Solaris 引入，glibc 实现）*/
size_t bufsize = __fbufsize(fp);   /* 缓冲区总大小 */
size_t pending = __fpending(fp);   /* 待刷新字节数 */
int    is_lb   = __flbf(fp);       /* 是否行缓冲 */

8.4 刷新与丢弃的区别

第一性原理

刷新 = 将缓冲的承诺兑现给内核

缓冲区中的数据是"已承诺给程序员但尚未兑现给内核"的数据。fflush 是将承诺兑现的动作；__fpurge 是撤销承诺。两者都将指针重置为空状态，区别仅在于数据是否真正传达到底层文件描述符。

9.2 setvbuf 详解

#include <stdio.h>

/**
 * setvbuf - 设置流的缓冲模式
 * @stream: 目标流
 * @buf:    用户提供的缓冲区（NULL 则 glibc 用 malloc 分配）
 * @mode:   _IOFBF | _IOLBF | _IONBF
 * @size:   缓冲区大小（_IONBF 时忽略）
 * 返回：0=成功，非0=失败
 *
 * 注意：必须在流打开后、任何 I/O 操作之前调用！
 */
int setvbuf(FILE *stream, char *buf, int mode, size_t size);

/* ── 三种使用方式 ── */

/* 1. 全缓冲，glibc 自动分配缓冲区 */
setvbuf(fp, NULL, _IOFBF, BUFSIZ);

/* 2. 行缓冲，用户提供缓冲区 */
char mybuf[1024];
setvbuf(fp, mybuf, _IOLBF, sizeof(mybuf));

/* 3. 无缓冲 */
setvbuf(fp, NULL, _IONBF, 0);

9.3 调用时机的约束

官方手册警告（§12.20.3）

"After opening a stream (but before any other operations have been performed on it), you can explicitly specify what kind of buffering you want it to have using the setvbuf function."

必须在流打开后、首次 I/O 操作前调用。否则行为未定义。

9.4 buf 参数的生命周期陷阱

/* ❌ 错误示例：栈上 buf 先于 stream 失效 */
void bad_example() {
    char buf[BUFSIZ];          /* 栈上变量 */
    setvbuf(stdin, buf, _IOFBF, BUFSIZ);
    printf("Hello, world!\n");
    /* 函数返回后 buf 失效，但 stream 仍在使用它！→ 未定义行为 */
}  /* ← 此处 buf 被销毁，stream 留有悬空指针 */

/* ✅ 正确示例：静态或堆分配 */
static char static_buf[BUFSIZ];
setvbuf(fp, static_buf, _IOFBF, BUFSIZ);  /* OK */

char *heap_buf = malloc(BUFSIZ);
setvbuf(fp, heap_buf, _IOFBF, BUFSIZ);    /* OK（记得 free）*/
🔗 GNU C Library §12.20.3 Controlling Buffering 🔗 setbuf(3) Linux man page
第一性原理

API 设计 = 封装策略选择权

setvbuf 将三个维度的控制权交给程序员：模式（何时刷新）、内存（谁提供缓冲区）、大小（每次批处理多少）。这三个参数对应了 I/O 性能调优的三个正交维度，而 setbuf/setlinebuf 等简化接口则固定了其中若干维度，降低使用门槛。

Chapter 10

默认缓冲策略

isatty() 检测、stdin/stdout/stderr 的默认行为

10.1 glibc 如何决定默认缓冲模式

glibc 在首次 I/O 操作或 fopen 时，通过 isatty(fd) 系统调用确定流连接的是终端还是文件/管道，据此选择缓冲策略：

10.2 重定向时的行为变化

/* 以下两种运行方式，stdout 缓冲模式不同 */

/* 方式1：连接终端 → 行缓冲 */
$ ./myprogram
# stdout 是行缓冲，每 \n 立即输出

/* 方式2：重定向到文件 → 全缓冲 */
$ ./myprogram > output.txt
# stdout 变为全缓冲！输出可能不会立即写入 output.txt
# 直到缓冲区满或程序退出（exit()）才刷新

/* 解决方案1：每次输出后手动刷新 */
printf("progress\n"); fflush(stdout);

/* 解决方案2：强制行缓冲（不受重定向影响）*/
setvbuf(stdout, NULL, _IOLBF, 0);

/* 解决方案3：使用 stdbuf 工具（glibc 提供）*/
$ stdbuf -oL ./myprogram > output.txt
经典陷阱：管道导致输出延迟

./prog | grep keyword — prog 的 stdout 被管道连接，isatty() 返回 false，切换为全缓冲。若 prog 不主动 fflush，grep 可能长时间收不到数据。这是许多"实时日志"失效的根本原因。

第一性原理

默认策略 = 对 I/O 目标物理特性的自动感知

isatty() 检测本质上是在问："我的输出目标是人类实时在看（终端），还是机器批量处理（文件/管道）？"这个问题的答案决定了最优缓冲策略。glibc 将这个感知自动化，使程序员无需显式指定即可获得合理的默认行为。

11.3 性能优化建议

/* 建议1：大文件 I/O 使用 fstat 获取最优缓冲大小 */
struct stat st;
fstat(fileno(fp), &st);
setvbuf(fp, NULL, _IOFBF, st.st_blksize);  /* 匹配文件系统块大小 */

/* 建议2：批量写入优于多次小写入 */
/* ❌ 低效：多次 fprintf */
for (int i = 0; i < 10000; i++)
    fprintf(fp, "%d\n", i);  /* 每次都格式化 */

/* ✅ 高效：先拼接到内存缓冲，再一次 fwrite */
char *mbuf = malloc(BIG_SIZE);
int len = snprintf(mbuf, BIG_SIZE, ...);
fwrite(mbuf, 1, len, fp);

/* 建议3：fork 前务必 fflush */
fflush(NULL);   /* 清空所有输出缓冲区 */

第一性原理

调试缓冲问题的第一性原理思路

遇到 I/O 行为异常，先问三个问题：
① 这个流是什么缓冲模式？（用 __flbf/__fbufsize 查询）
② 刷新条件是否满足？（缓冲区满？有换行？有 fflush？）
③ 数据在哪一层"卡住"了？（glibc 缓冲？内核页缓存？硬件缓冲？）
逐层排查，每层都有独立的刷新机制（fflush/fsync/hdparm --flush）。
