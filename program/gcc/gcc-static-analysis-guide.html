概述 · Overview

背景与体系概览

GCC（GNU Compiler Collection）为 C99 代码提供了远超标准编译错误的丰富静态检查机制。 这些机制在编译阶段而非运行时发现问题，代价为零，是 Linux 内核、glibc、OpenSSL 等 大型 C 项目保证质量的核心工具。

GCC 的静态检查可分为四个层次： ① 编译期断言宏（利用 C99 语法特性与 GCC 扩展）， ② 函数/变量/类型属性（__attribute__）， ③ 内置函数（builtins）， ④ 静态分析器（-fanalyzer）。

第一章 · Chapter 1

BUILD_BUG_ON 系列宏

Linux 内核在 include/linux/build_bug.h 中定义了一套完整的编译期断言宏。 这是 GCC 扩展特性用于 C99 静态检查的最典型实现，被数千处内核代码引用。

原文来源：linux/include/linux/build_bug.h · torvalds/linux · GitHub

BUILD_BUG_ON — 核心宏

/**
 * BUILD_BUG_ON - break compile if a condition is true.
 * @condition: the condition which the compiler should know is false.
 *
 * If you have some code which relies on certain constants being equal, or
 * some other compile-time-evaluated condition, you should use BUILD_BUG_ON to
 * detect if someone changes it.
 */
/* 无优化时退化为负数组大小技巧 */
#ifndef __OPTIMIZE__
#define BUILD_BUG_ON(condition) ((void)sizeof(char[1 - 2*!!(condition)]))
#else
/* 开启优化时使用 __attribute__((error)) 获得更好的错误信息 */
#define BUILD_BUG_ON(condition) \
    BUILD_BUG_ON_MSG(condition, "BUILD_BUG_ON failed: " #condition)
#endif

完整 BUILD_BUG_ON 家族

BUILD_BUG_ON(cond)

条件为真时中断编译，最常用

BUILD_BUG_ON_MSG(cond, msg)

同上，但携带自定义错误消息

BUILD_BUG_ON_ZERO(e)

产生值为 0 的 int，可用于结构体初始化内

BUILD_BUG_ON_NOT_POWER_OF_2(n)

强制 n 必须是 2 的幂次

BUILD_BUG_ON_INVALID(e)

仅检查表达式合法性，不产生代码

BUILD_BUG()

无条件编译失败，用于标记永远不应到达的代码

内核中的典型用法

/* 确保结构体大小不变（协议兼容性保障）*/
BUILD_BUG_ON(sizeof(struct ethhdr) != 14);

/* 确保枚举值不超过数组边界 */
BUILD_BUG_ON(MAX_ERRNO >= 4096);

/* 哈希表桶数必须是 2 的幂次（用于位掩码优化）*/
BUILD_BUG_ON_NOT_POWER_OF_2(HASHTABLE_SIZE);

/* 检查结构体字段偏移量（ASSERT_STRUCT_OFFSET，内核 5.18+ 引入）*/
ASSERT_STRUCT_OFFSET(struct my_proto, field_x, 8);

/* 类型兼容性检查，配合 __builtin_types_compatible_p */
BUILD_BUG_ON(!__builtin_types_compatible_p(typeof(ptr), struct page *));
⚠️ 重要：优化依赖

当使用 __attribute__((error(...))) 实现的 BUILD_BUG_ON_MSG 时， GCC 需要 至少 -O1 优化级别（即 __OPTIMIZE__ 宏已定义）才能通过 死代码消除（DCE）触发编译错误。在 -O0 时内核退化为负数组大小方法。

static_assert / ASSERT_STRUCT_OFFSET（现代写法）

/* Linux 内核封装了 C11 _Static_assert，消息参数可选 */
#define static_assert(expr, ...) __static_assert(expr, ##__VA_ARGS__, #expr)
#define __static_assert(expr, msg, ...) _Static_assert(expr, msg)

/* 检查字段偏移量是否符合预期（原文注释："Compile time check that field
   has an expected offset"）*/
#define ASSERT_STRUCT_OFFSET(type, field, expected_offset)    \
    BUILD_BUG_ON_MSG(offsetof(type, field) != (expected_offset), \
        "Offset of " #field " in " #type " has changed.")
第一章 · 技术细节

负数组大小技巧

这是 GCC 静态检查最早的 C99 兼容实现，由 Pix Elbeat 在 2008 年提出，Linux 内核随即采用。 其本质利用了 GCC 不允许声明大小为负数的数组这一语义约束。

原文来源：pixelbeat.org — Compile time assertions in C 
/* 方法一：负数组大小（适用于全局作用域和局部作用域）*/
/* 当 e 为假 (0) 时，sizeof(char[-1]) → 编译错误 */
#define ct_assert(e) ((void)sizeof(char[1 - 2*!(e)]))

/* Linux 内核 BUILD_BUG_ON_ZERO 变体：产生表达式值（可嵌套在初始化器中）*/
/* 利用位域宽度不能为负的特性 */
#define BUILD_BUG_ON_ZERO(e, ...) \
    (sizeof(struct { int:(-!!(e)); }))
📌 与 _Static_assert 的区别

C11 的 _Static_assert 可在全局和局部作用域使用，错误信息更友好； 而负数组大小技巧是纯 C99 兼容的方案，且 BUILD_BUG_ON_ZERO 变体 能在表达式上下文（如结构体初始化器、宏参数中）使用，这是 _Static_assert 无法实现的。

第一章 · 实现核心

compiletime_assert 实现

内核 include/linux/compiler.h 的 compiletime_assert 结合了 __attribute__((error(...))) 与死代码消除，是 BUILD_BUG_ON_MSG 的实际底层。

原文来源：linux/include/linux/compiler-gcc.h 
/* 定义 GCC 版本感知的 compiletime_error 属性 */
#define __compiletime_error(message) \
    __attribute__((__error__(message)))
#define __compiletime_warning(message) \
    __attribute__((__warning__(message)))

/* 核心实现：利用 GCC 在优化时进行死代码消除（DCE）
   若条件为假，则 if (__cond) 体成为活代码，
   调用带 error 属性的外部函数 → 编译器报错 */
#ifdef __OPTIMIZE__
# define __compiletime_assert(condition, msg, prefix, suffix)    \
    do {                                                          \
        extern void prefix ## suffix(void)                       \
            __compiletime_error(msg);                             \
        if (!(condition))                                         \
            prefix ## suffix();                                   \
    } while (0)
#else
/* 无优化时退化为负数组大小方法（始终可用）*/
# define __compiletime_assert(condition, msg, prefix, suffix) \
    ((void)sizeof(char[1 - 2 * !(condition)]))
#endif

#define compiletime_assert(condition, msg) \
    _compiletime_assert(condition, msg, __compiletime_assert_, __LINE__)

#define BUILD_BUG_ON_MSG(cond, msg) \
    compiletime_assert(!(cond), msg)

MAYBE_BUILD_BUG_ON — 条件静态断言

内核还提供了一种混合模式宏：当条件可在编译期求值时，使用编译期断言；否则降级为运行时检查。 这利用了 __builtin_constant_p。

原文来源：Exploring GNU extensions in the Linux kernel — MaskRay 
#define MAYBE_BUILD_BUG_ON(cond)              \
    do {                                      \
        if (__builtin_constant_p((cond)))    \
            BUILD_BUG_ON(cond);             \
        else                                  \
            BUG_ON(cond);                   \
    } while (0)

/* 示例：nr 为常量时在编译期检查，否则运行时检查 */
BUILD_BUG_ON(__builtin_constant_p(nr) && nr != 1);
第二章 · Chapter 2

__attribute__((error)) / __attribute__((warning))

GCC 4.3+ 引入的函数属性，与死代码消除（DCE）和内联配合，实现可携带自定义消息的编译期错误。 这是 Linux 内核 compiletime_assert 的核心机制，也被 glibc 的 _FORTIFY_SOURCE 广泛使用。

GCC 官方：GCC Function Attributes: error, warning 
/* __attribute__((error("msg"))) 声明一个"如果被调用则报编译错误"的函数 */
extern void __invalid_argument_size(void)
    __attribute__((error("Invalid argument size, must be 1, 2, 4 or 8")));

static inline void check_size(int n)
{
    /* 仅当 n 为编译期常量且不满足条件时，编译器才会"看到"此调用 */
    if (__builtin_constant_p(n) &&
        n != 1 && n != 2 && n != 4 && n != 8)
        __invalid_argument_size();
}

/* glibc _FORTIFY_SOURCE 中的经典实现（openat 必须 4 参数）*/
extern int __openat_missing_mode(void)
    __attribute__((error("openat with O_CREAT or O_TMPFILE in third argument needs 4 arguments")));

/* __attribute__((warning("msg"))) 类似，但降级为警告而非错误 */
extern void __use_deprecated_api(void)
    __attribute__((warning("This API is deprecated, use new_api() instead")));
第二章 · 函数属性

__attribute__((nonnull)) — 非空指针检查

nonnull 属性声明某些函数参数不得为空指针。GCC 在检测到可证明为 NULL 的参数时 （通过 -Wnonnull，已包含于 -Wall）发出警告。

GCC 官方：GCC 4.7.2 Function Attributes — nonnull 
/* 声明参数 1 和 2 不得为 NULL */
extern void *my_memcpy(void *dest, const void *src, size_t len)
    __attribute__((nonnull (1, 2)));

/* 不指定参数索引：所有指针参数都不得为 NULL */
extern void *my_memcpy(void *dest, const void *src, size_t len)
    __attribute__((nonnull));

/* 调用时若传入字面 NULL，-Wnonnull 立即报警（-Wall 已包含）*/
my_memcpy(NULL, src, 16);
// warning: null argument where non-null required (argument 1) [-Wnonnull]
⚠️ 局限性

GCC 的 nonnull 仅能检测可静态证明为 NULL 的情况（如直接传入字面 NULL）。 对于间接赋值为 NULL 的指针变量（如 int *p = 0; f(p);），GCC 通常不警告， 而 Clang/Coverity 在此处表现更好。此外，该属性也被优化器用于假设参数非空，以消除不必要的检查。

第二章 · 函数属性

__attribute__((format)) — 格式字符串类型检查

format 属性让 GCC 像检查 printf/scanf 一样 检查自定义函数的格式字符串参数，捕获类型不匹配等错误。 已包含在 -Wall 中（通过 -Wformat=1）。 

/* archetype: printf/scanf/strftime/strfmon 之一
   string-index: 格式字符串是第几个参数（1-based）
   first-to-check: 可变参数列表从第几个参数开始 */
extern int my_printf(void *obj, const char *fmt, ...)
    __attribute__((format(printf, 2, 3)));

/* Linux 内核中的 printk */
__printf(1, 2)
int printk(const char *fmt, ...);

/* 内核封装宏 */
#define __printf(a, b)  __attribute__((format(printf, a, b)))
#define __scanf(a, b)   __attribute__((format(scanf, a, b)))

-Wformat 系列（-Wall 包含 -Wformat=1）

-Wformat=1 -Wall

检查 printf/scanf 类格式字符串与参数类型匹配

-Wformat=2 手动启用

更严格，包含 -Wformat-nonliteral 和 -Wformat-security

-Wformat-overflow=1 -Wall

检测 sprintf 类函数的输出缓冲区溢出

-Wformat-truncation=1 -Wall

检测 snprintf 类函数结果可能被截断

-Wformat-nonliteral 手动启用

格式字符串不是字符串字面量（潜在安全风险）

-Wformat-security 手动启用

格式字符串安全问题（如 printf(str) 无格式参数）

第三章 · Chapter 3

__builtin_constant_p — 编译期常量判断

__builtin_constant_p(exp) 在编译器能将 exp 折叠为常量时返回 1，否则返回 0。 这是 Linux 内核中最核心的内置函数之一，配合静态断言实现"条件编译期检查"。

原文来源：Exploring GNU extensions in the Linux kernel — __builtin_constant_p 
/* 1. 条件静态断言：仅对编译期常量参数进行静态检查 */
BUILD_BUG_ON(__builtin_constant_p(nr) && nr != 1);

/* 2. MAYBE_BUILD_BUG_ON 混合模式：
   常量 → 编译期检查  |  非常量 → 运行时检查 */
#define MAYBE_BUILD_BUG_ON(cond)               \
    do {                                       \
        if (__builtin_constant_p((cond)))     \
            BUILD_BUG_ON(cond);              \
        else                                   \
            BUG_ON(cond);                    \
    } while (0)

/* 3. 常量折叠优化路径选择（trace 宏中的用法）*/
#define __trace_if_var(cond) \
    (__builtin_constant_p(cond) ? (cond) : __trace_if_value(cond))

/* 4. 配合 __attribute__((error)) 的用法（glibc 风格）*/
static inline void function(const uint16_t len)
{
    if (!__builtin_constant_p(len)) {
        extern __attribute__((__error__("len must be a compile-time constant")))
        void compile_time_error(void);
        compile_time_error();
    } else if (len > 32768) {
        extern __attribute__((__error__("len too large")))
        void compile_time_error2(void);
        compile_time_error2();
    }
}
⚠️ -O0 的局限

__builtin_constant_p 在 -O0 下通常对所有表达式返回 0（不进行常量折叠）， 因此依赖它的模式（如 MAYBE_BUILD_BUG_ON）会退化为运行时检查。 这也是 Linux 内核无法用 -O0 编译的原因之一（原文明确提及）。

第三章 · 内置函数

__builtin_types_compatible_p — 类型兼容性检查

在 C99 中无法直接进行类型比较，__builtin_types_compatible_p(type1, type2) 填补了这一空白，常与 typeof 配合用于类型安全宏。

原文来源：Exploring GNU extensions — __builtin_types_compatible_p 
/* 内核辅助宏：检查两个变量/表达式类型相同 */
#define __same_type(a, b) \
    __builtin_types_compatible_p(typeof(a), typeof(b))

/* kunmap_atomic 中防止传入 struct page * 的保护：
   原文注释 "BUILD_BUG_ON(__same_type((__addr), struct page *));" */
#define kunmap_atomic(__addr)                              \
do {                                                       \
    BUILD_BUG_ON(__same_type((__addr), struct page *));   \
    __kunmap_atomic(__addr);                              \
} while (0)

/* 类型安全的 min/max 宏（防止整数/指针混用）*/
#define min(x, y) ({                                      \
    typeof(x) _x = (x);                                    \
    typeof(y) _y = (y);                                    \
    (void)(&_x == &_y);  /* 类型兼容检查技巧 */          \
    _x < _y ? _x : _y; })

/* jump_label.h 中模拟 C++17 if constexpr：
   类型不符时调用未定义函数 → 链接错误 */
if (__builtin_types_compatible_p(typeof(*x), struct static_key_true))
    branch = !arch_static_branch(&(x)->key, true);
else if (__builtin_types_compatible_p(typeof(*x), struct static_key_false))
    branch = !arch_static_branch_jump(&(x)->key, true);
else
    branch = ____wrong_branch_error();  /* 未定义，链接失败 */
第三章 · 内置函数

__builtin_object_size / _FORTIFY_SOURCE

__builtin_object_size(ptr, type) 在编译期估算指针所指对象的大小。 glibc 在 2004 年利用此特性引入 _FORTIFY_SOURCE；Linux 内核在 2017 年引入 CONFIG_FORTIFY_SOURCE，两者都通过内联函数 + BUILD_BUG_ON 在编译期捕获缓冲区溢出。

原文来源：Exploring GNU extensions — Object size checking 
/* __builtin_object_size(ptr, 0) 返回 ptr 所指对象大小的上界，
   编译器无法确定时返回 (size_t)-1 */
#define __compiletime_object_size(obj) \
    __builtin_object_size(obj, 0)

/* Linux 内核 strtomem 宏（include/linux/string.h）
   确保目标和源都有已知的编译期大小 */
#define strtomem(dest, src)                                           \
    do {                                                               \
        const size_t _dest_len = __builtin_object_size(dest, 1);    \
        const size_t _src_len  = __builtin_object_size(src,  1);    \
        /* 若大小为未知（-1），或不是常量，则编译失败 */              \
        BUILD_BUG_ON(!__builtin_constant_p(_dest_len) ||            \
                     _dest_len == (size_t)-1);                       \
        memcpy(dest, src, strnlen(src, min(_src_len, _dest_len))); \
    } while (0)

/* glibc _FORTIFY_SOURCE memcpy 检查原理 */
static __always_inline void *
__memcpy_chk(void *dest, const void *src, size_t n, size_t dest_len) {
    if (__builtin_constant_p(n) && __builtin_constant_p(dest_len)
        && n > dest_len) {
        /* 编译期可证明溢出 → error 属性触发编译错误 */
        __chk_fail();  /* extern void __chk_fail(void) __attribute__((error(...))); */
    }
    return __builtin___memcpy_chk(dest, src, n, dest_len);
}

第三章 · 内置函数

__builtin_*_overflow — 整数溢出检测

GCC 5+ 提供了一组内置函数，用于在算术运算发生溢出时返回 true。 Linux 内核的 include/linux/overflow.h 基于这些函数封装了完整的溢出检测工具集。

原文来源：Exploring GNU extensions — __builtin_*_overflow 
/* GCC 提供三种溢出检测内置函数：
   返回 true 表示溢出发生，结果存入 *result */
bool __builtin_add_overflow(type1 a, type2 b, type3 *result);
bool __builtin_sub_overflow(type1 a, type2 b, type3 *result);
bool __builtin_mul_overflow(type1 a, type2 b, type3 *result);

/* Linux 内核 overflow.h 封装（用于安全的内存分配大小计算）*/
#define check_add_overflow(a, b, d) \
    __builtin_add_overflow(a, b, d)

#define check_mul_overflow(a, b, d) \
    __builtin_mul_overflow(a, b, d)

/* 安全分配示例：防止 size * count 溢出导致 UAF/堆溢出 */
size_t total;
if (check_mul_overflow(count, sizeof(struct item), &total))
    return -EOVERFLOW;
ptr = kmalloc(total, GFP_KERNEL);
✅ 编译期常量折叠

当两个操作数都是编译期常量时，__builtin_*_overflow 在编译期求值， 溢出结果可被 BUILD_BUG_ON 等机制检测，实现零运行时开销的溢出断言。

第四章 · 超出范围

超出 -Wall 的重要检查（需手动启用）

这些警告由 GCC 官方文档明确说明"不被 -Wall 包含"，有些被 -Wextra 覆盖， 但相当多需要单独启用，在 Linux 内核、glibc 等项目中被显式加入构建标志。

GCC 官方：GCC Warning Options（"Note that some warning flags are not implied by -Wall"）
标志检测内容说明
-Wshadow 局部变量遮蔽外层同名变量 C 中常见的隐蔽 bug 来源；Linux 内核在 W=2 级别启用
-Wstrict-prototypes 函数声明未使用原型（K&R 风格） 确保所有参数类型明确声明（C99 推荐）
-Wmissing-prototypes 全局函数定义前无对应原型声明 防止接口暴露不当；Linux 内核使用
-Wcast-align 指针类型转换导致目标对齐要求更高 捕获潜在的内存对齐错误（ARM 等平台严苛）
-Wcast-qual 指针转换丢弃类型限定符（如丢弃 const） 保护 const 正确性
-Wconversion 隐式类型转换可能改变值 对整数截断、符号转换等隐患敏感
-Wsign-conversion 有符号/无符号整数隐式转换 -Wconversion 的子集，更聚焦符号问题
-Wfloat-equal 浮点数相等比较 浮点精度问题；应使用范围比较
-Wundef #if 中使用未定义的宏 防止宏名拼写错误被静默展开为 0
-Wpedantic / -pedantic ISO C99 标准以外的 GCC 扩展 确保代码严格符合 C99；可辅以 -pedantic-errors
-Wdeclaration-after-statement C90：在语句后声明变量 C99 允许，但用于保持与老代码兼容性
-Wmissing-declarations 全局函数声明前无头文件声明 类似 -Wmissing-prototypes，更宽泛
-Wredundant-decls 同一作用域内重复声明 检测冗余包含和重声明
-Wpadded 结构体因对齐产生填充字节 用于优化结构体布局（内存敏感场景）
-Wpacked packed 属性无效（结构体已对齐） 防止误用 packed 导致未对齐访问
-Winline 声明为 inline 但编译器无法内联的函数 检测 inline 关键字失效
-Wvla 使用了可变长数组（VLA） Linux 内核禁止 VLA；C11 中 VLA 变为可选
-Walloca 使用了 alloca() 可能导致栈溢出；内核使用 -Walloca
-Wstack-usage=N 函数栈帧超过 N 字节 防止深层递归栈溢出；内核使用 -Wframe-larger-than=
-Wformat=2 更严格的格式字符串检查 包含 -Wformat-nonliteral 和 -Wformat-security
-Woverlength-strings 字符串字面量超过 C99 最小最大长度（4095） 由 -Wpedantic 隐含
-Wnested-externs 函数内部出现 extern 声明 C99 允许但不推荐
-Wsuggest-attribute=pure/const/noreturn/malloc 建议为函数添加属性以便更好优化 辅助识别可以添加属性的函数

Linux 内核风格完整静态检查宏套件

/* 可移植的编译期断言套件（适用于 C99 + GCC）*/

/* 1. 通用编译期断言（表达式上下文友好）*/
#define BUILD_BUG_ON_ZERO(e) (sizeof(struct { int:(-!!(e)); }))

/* 2. 语句上下文断言 */
#ifdef __OPTIMIZE__
# define BUILD_BUG_ON(cond) \
    ({ extern void __build_bug_on_failed(void) \
        __attribute__((error("BUILD_BUG_ON failed"))); \
       if (cond) __build_bug_on_failed(); })
#else
# define BUILD_BUG_ON(cond) ((void)sizeof(char[1-2*!!(cond)]))
#endif

/* 3. 结构体大小断言 */
#define ASSERT_SIZE(type, size) \
    BUILD_BUG_ON(sizeof(type) != (size))

/* 4. 字段偏移量断言 */
#define ASSERT_FIELD_OFFSET(type, field, offset) \
    BUILD_BUG_ON(offsetof(type, field) != (offset))

/* 5. 2 的幂次断言 */
#define ASSERT_POWER_OF_2(n) \
    BUILD_BUG_ON((n) == 0 || (((n) & ((n)-1)) != 0))

/* 6. 类型兼容性断言 */
#define ASSERT_SAME_TYPE(a, b) \
    BUILD_BUG_ON(!__builtin_types_compatible_p(typeof(a), typeof(b)))
