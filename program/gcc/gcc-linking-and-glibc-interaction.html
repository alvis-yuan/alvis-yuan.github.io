GCC 编译链接深度解析

从你写下 int main() 的那一刻，到 Linux 内核最终跳转到你的第一行代码之前，中间究竟发生了什么？

本文系统拆解 GCC 四大编译阶段，并深入 glibc 源码层面，讲清楚一个孤立的 main.c 是如何与 glibc 的 CRT 启动对象文件（crt1.o、crti.o、crtn.o 等）进行拼接，最终产生可运行的 ELF 可执行文件的。

📚 官方来源：gcc.gnu.org · glibc · codebrowser.dev 🖥 目标平台：Linux x86-64 📦 工具链：GCC + GNU ld + glibc

图 0-1：GCC 四大编译阶段总览（来源：GCC 官方文档）

官方依据：GCC 官方文档（Overall Options）指出："Compilation can involve up to four stages: preprocessing, compilation proper, assembly and linking, always in that order."
⚛ 第一性原理

程序本质上是机器指令序列 + 数据 + 元信息。GCC 的四个阶段，是逐步将「人类可读的高层描述」转换为「处理器可直接执行的二进制」的过程。每一阶段都仅做一件事，层层解耦，使得每个步骤可独立替换、优化和缓存。链接阶段本质上是解决「多个独立编译单元之间的符号引用」——这是分离编译能够工作的根本前提。

阶段 01

① 预处理（Preprocessing）

预处理由 cpp（C Preprocessor）完成，GCC 在内部调用它。该阶段对源代码进行纯文本变换，不涉及任何语法分析。

主要工作

  • 1
    展开 #include
    将头文件内容完整替换到对应的 #include 指令位置，递归处理嵌套包含。
  • 2
    宏展开（#define / #undef）
    将所有宏替换为其定义的文本，包括带参数的函数式宏。
  • 3
    条件编译（#if / #ifdef）
    根据条件选择性地保留或删除代码块。
  • 4
    行号标注与注释删除
    插入 #line 指令，帮助编译器报错时还原源文件位置；删除所有注释。

命令示例

# 仅预处理，输出到 stdout（-P 去掉行号标注）
gcc -E main.c -o main.i
gcc -E -P main.c

输出产物

生成 main.i（仍是纯文本 C 代码），通常比原始文件大几十到几百倍（因为 stdio.h 等头文件内容被完整展开）。

⚛ 第一性原理

预处理器的本质是纯文本替换引擎。它不理解 C 语法，只做字符串操作。这样设计的原因是：将「代码复用（头文件/宏）」与「语法语义分析」两件事解耦——预处理器可以极快地完成文本展开，而真正的编译器只需要接收一个干净的、展开完毕的 C 文本。

阶段 02

② 编译（Compilation Proper）

编译阶段由 cc1（GCC 的内部编译器前端）完成。它将预处理后的 C 代码转换为目标架构的汇编代码。

内部流程

图 2-1：GCC 编译阶段内部流程

命令示例

# 编译并输出汇编（-S 停在汇编阶段）
gcc -S main.c -o main.s
gcc -S -masm=intel main.c    # Intel 语法

典型汇编输出（x86-64 AT&T 语法）

# main.s 片段（gcc -S -O0 main.c）
main:
    pushq   %rbp
    movq    %rsp, %rbp
    subq    $16, %rsp
    movl    $0, %eax
    call    printf@PLT      # 对 printf 的符号引用（尚未解析）
    movl    $0, %eax
    popq    %rbp
    ret
⚠ 关键点：此时 call printf@PLT 只是一个符号占位符，printf 的实际地址尚不存在于这个文件中。这正是为什么需要「链接」阶段的根本原因。
⚛ 第一性原理

编译器的本质是语义保持的等价变换——将一种形式化语言（C）中的语义，映射到另一种语言（汇编/机器码）中，同时保证行为等价。优化是在「保持语义」约束下，寻找「计算代价更小」的等价表达。此阶段产生的汇编中充满「未解析符号」，这些符号是跨编译单元协作的接口——分离编译的根本代价是符号解析必须推迟到链接阶段。

阶段 03

③ 汇编（Assembly）

GNU 汇编器 as（GNU Assembler）将汇编文本 .s 转换为可重定向目标文件（Relocatable Object File）.o，这是 ELF 格式的二进制文件。

命令示例

# 仅汇编，-c 停在目标文件阶段
gcc -c main.c -o main.o
# 或者直接调用汇编器
as main.s -o main.o

目标文件的 ELF Section 结构

.data已初始化全局/静态变量读写数据
.bss未初始化全局/静态变量仅占位，不占文件空间
.rodata只读数据（字符串字面量等）只读数据段

查看目标文件

# 查看符号表
nm main.o

# 查看 ELF section 信息
readelf -S main.o

# 查看重定位条目（可看到对 printf、__libc_start_main 等的未解析引用）
readelf -r main.o
可重定向（Relocatable）的含义：目标文件中的所有内存地址都是相对地址（从 0 开始），等待链接器分配最终的虚拟地址并填入正确的偏移量。.rela.text section 中的重定位条目就是告诉链接器「在哪个位置、填入什么符号的地址」。
⚛ 第一性原理

目标文件是一个「意图声明」：它声明了「我定义了哪些符号，我依赖哪些外部符号」。可重定向格式的本质是将地址绑定推迟——推迟到所有模块都已知的时候（链接时）。这是分离编译能够工作的技术基础：每个编译单元独立编译，互相之间只通过符号名称进行约定，链接器在最后负责兑现这些约定。

阶段 04

④ 链接（Linking）—— 核心拼合阶段

这是 main.c 与 glibc 入口函数真正「拼接」的阶段。GCC 驱动程序（driver）会调用 collect2（最终调用 GNU 链接器 ld），将多个目标文件和库文件合并成一个可执行 ELF 文件。

GCC 驱动自动注入的链接命令

当你执行 gcc main.c -o main 时，GCC 驱动在后台执行的链接命令（通过 gcc -v main.c 可看到）大致如下：

ld \
  -dynamic-linker /lib64/ld-linux-x86-64.so.2 \
  /usr/lib/x86_64-linux-gnu/crt1.o \
  /usr/lib/x86_64-linux-gnu/crti.o \
  /usr/lib/gcc/x86_64-linux-gnu/12/crtbegin.o \
  -L/usr/lib/gcc/x86_64-linux-gnu/12 \
  -L/usr/lib/x86_64-linux-gnu \
  main.o            # 用户目标文件
  -lgcc             # GCC 运行时库
  -lgcc_s           # GCC 共享运行时库
  -lc               # glibc（libc.so）
  -lgcc \
  -lgcc_s \
  /usr/lib/gcc/x86_64-linux-gnu/12/crtend.o \
  /usr/lib/x86_64-linux-gnu/crtn.o
重要：链接顺序至关重要！GNU ld 是单遍链接器，从左到右处理，CRT 启动对象必须在用户代码之前，CRT 结束对象必须在最后。

链接顺序总结表

⚛ 第一性原理

链接器的本质任务就两件事：①符号解析（把所有引用的符号名称绑定到它们的定义位置）和②重定位（将所有相对地址替换为最终的虚拟地址）。CRT 文件的「夹三明治」顺序（crt1+crti+crtbegin 在前，crtend+crtn 在后）是利用链接器的 Section 合并机制，将分散在多个目标文件中的 .init/.fini/.init_array 等片段按正确顺序拼合成完整函数。

CRT 文件

⑤ CRT 启动文件全解析

CRT（C RunTime）是一组在用户代码运行前后自动执行的汇编/C 目标文件。glibc 在 /usr/lib/x86_64-linux-gnu/ 目录下提供这些文件，GCC 也提供自己专属的 crtbegin.o/crtend.o。

⚛ 第一性原理

CRT 文件的存在是为了解决一个根本矛盾：程序运行需要运行时环境，但运行时环境本身也需要被初始化。操作系统只能保证将 ELF 加载到内存并跳转到入口点，它不负责初始化 C 标准库、准备 argc/argv、调用全局构造函数。CRT 正是填补「内核准备完毕」到「用户 main() 开始」之间的这段空白。多个 CRT 文件的设计是「关注点分离」原则的体现：每个文件只负责启动序列中的一个明确职责。

CRT 06

⑥ crt1.o 与 _start 符号

crt1.o 是整个链接链的「起点」。它由 glibc 的 csu/start.S（x86-64 为 sysdeps/x86_64/start.S）编译而来，是可执行文件中真正的程序入口点。

ELF 入口点的设定

链接器脚本中有 ENTRY(_start) 指令，链接器将 ELF 文件头的 e_entry 字段设置为 _start 符号的地址。内核执行 execve() 后，最终跳转到这个地址。

_start 的真实汇编代码（glibc x86-64）

以下来自 glibc 源码 sysdeps/x86_64/start.S：

/* sysdeps/x86_64/start.S  —— glibc 官方源码 */
.text
.globl _start
_start:
    /* 清除帧指针，标记调用栈底 */
    xorl    %ebp, %ebp

    /* rdx = rtld_fini（动态链接器的终止函数） */
    mov     %RDX_LP, %R9_LP

    /* 弹出 argc（内核放在栈顶）*/
    popq    %rsi

    /* rdx = argv（当前栈顶即 argv[0]）*/
    mov     %RSP_LP, %RDX_LP

    /* 将栈对齐到 16 字节（遵循 x86-64 SysV ABI）*/
    and     $~15, %RSP_LP

    /* 在寄存器中准备 __libc_start_main 的参数：
       rdi = main 函数指针
       rsi = argc
       rdx = argv
       rcx = 0（init，旧版为 __libc_csu_init）
       r8  = 0（fini，旧版为 __libc_csu_fini）
       r9  = rtld_fini
       stack = &stack_end */

    xorl    %r8d, %r8d     /* fini = 0 */
    xorl    %ecx, %ecx     /* init = 0 */

    /* 加载 main 函数地址（PIE 模式通过 GOT 间接加载）*/
    mov     $main, %RDI_LP

    /* 调用 glibc 的启动入口，永不返回 */
    call    *__libc_start_main@GOTPCREL(%rip)

    /* 如果意外返回，停机（防御性编程）*/
    hlt
原文来源： codebrowser.dev/glibc/glibc/sysdeps/x86_64/start.S.html

_start 向 __libc_start_main 传递的参数

寄存器/位置参数说明
%rdimain用户 main 函数指针
%rsiargc从栈弹出的参数个数
%rdxargv参数字符串数组指针
%rcxinit初始化函数（新版为 0）

版本演化（BZ #23323）：在较旧的 glibc 版本中，_start 会传入 __libc_csu_init 和 __libc_csu_fini（由 libc_nonshared.a 提供）。从 glibc 2.34+ 起，这两个参数被改为零，__libc_csu_init/__libc_csu_fini 的逻辑被移入 csu/libc-start.c，简化了接口。
⚛ 第一性原理

_start 存在的根本原因是：内核只提供原始的进程上下文（寄存器状态 + 栈布局），不提供 C 运行环境。内核在栈上按 Linux ABI 规定放好 argc/argv/envp，但 C 函数调用约定要求参数在寄存器中传递。_start 就是这个「从 ABI 格式转换为 C 调用约定」的最小胶水代码。它做的事情越少越好，因为此时 C 库尚未初始化，连最基本的内存分配都不可用。

CRT 07

⑦ crti.o 与 crtn.o —— .init/.fini 节的拼合

crti.o 和 crtn.o 由 glibc 的 csu/crti.S 和 csu/crtn.S 编译而来，它们共同构成 ELF 文件中的 .init 和 .fini 节。

Section 片段拼合机制

GNU 链接器将所有输入目标文件中的同名 section 按顺序合并。crti.o 提供 _init 函数的序言（prologue），crtn.o 提供尾声（epilogue），中间的 crtbegin.o/crtend.o 及用户代码填入函数体：

图 7-1：.init 节的分片拼合过程

crti.o 反汇编（glibc x86-64）

# crti.o — .init section（序言片段，无 ret）
_init:
    subq    $8, %rsp
    movq    (%rip), %rax    # R_X86_64_REX_GOTPCRELX __gmon_start__
    testq   %rax, %rax
    je      .Lskip
    callq   *%rax           # 若定义了 gmon 分析，调用之
.Lskip:
    # 注意：没有 addq / ret！由 crtn.o 补全

# crti.o — .fini section（序言片段）
_fini:
    subq    $8, %rsp
    # 没有 ret！由 crtn.o 补全
# crtn.o — .init section（尾声片段）
.init:
    addq    $8, %rsp
    retq

# crtn.o — .fini section（尾声片段）
.fini:
    addq    $8, %rsp
    retq
原文来源： startup-files-and-sequences.md (MaskRay / glibc)
⚛ 第一性原理

crti/crtn 的「分段拼合」设计是对链接器 Section 合并语义的一种创造性利用：在没有语言层面的「函数拼接」机制时，通过让每个目标文件提供函数的一部分（无 ret 的序言、函数体、有 ret 的尾声），依靠链接器的 Section 顺序合并，在二进制层面构造出完整函数。这是一种底层的「面向链接器编程」范式，代价是脆弱——顺序必须严格正确，否则 ret 指令对应不上栈帧，程序必然崩溃。

CRT 08

⑧ crtbegin.o / crtend.o —— C++ 全局构造/析构

crtbegin.o 和 crtend.o 由 GCC 自己提供（libgcc/crtstuff.c），不是 glibc 的一部分。它们专门处理 C++ 的全局对象构造和析构，以及 __attribute__((constructor)) 标注的函数。

现代 .init_array 方式（主流发行版）

用户 .o 文件全局构造函数指针填入全局析构函数指针填入

链接器脚本中的处理（摘自 GCC 默认脚本）

/* 链接器脚本片段（gcc -static -Wl,--verbose 可查看完整版本）*/
.init_array :
{
    PROVIDE_HIDDEN (__init_array_start = .);
    KEEP (*(SORT_BY_INIT_PRIORITY(.init_array.*)))
    KEEP (*(.init_array
            EXCLUDE_FILE (*crtbegin.o *crtbegin?.o
                          *crtend.o   *crtend?.o  ) .ctors))
    PROVIDE_HIDDEN (__init_array_end = .);
}
.ctors :
{
    KEEP (*crtbegin.o(.ctors))    /* ctors 列表头 */
    KEEP (*crtbegin?.o(.ctors))
    KEEP (*(EXCLUDE_FILE (*crtend.o *crtend?.o) .ctors))
    KEEP (*(SORT(.ctors.*)))
    KEEP (*(.ctors))              /* ctors 列表尾 */
}

⚛ 第一性原理

.init_array 机制的本质是：把「在 main 前执行哪些函数」这个问题，转化为「在特定内存区域存放函数指针列表」这个数据问题。启动代码只需要遍历这个列表逐一调用，无需知道每个构造函数的名称。这是一种「注册表（Registry）」模式的底层实现——每个需要初始化的模块向列表注册自己，启动代码统一驱动执行。

glibc 09

⑨ __libc_start_main —— 真正的启动编排者

__libc_start_main 定义在 glibc 的 csu/libc-start.c 中，是连接 _start 和用户 main() 之间真正做初始化工作的函数。

函数签名（历史版本）

/* glibc csu/libc-start.c 中的 __libc_start_main 签名 */
int __libc_start_main (
    int (*main)(int, char**, char**),  /* 用户 main 函数指针 */
    int argc,                            /* 参数个数 */
    char **argv,                         /* 参数列表 */
    void (*init)(void),                  /* 初始化函数（旧：__libc_csu_init）*/
    void (*fini)(void),                  /* 终止函数（旧：__libc_csu_fini）*/
    void (*rtld_fini)(void),             /* 动态链接器终止函数 */
    void *stack_end                      /* 栈底指针 */
);

__libc_start_main 的工作步骤（伪代码）

int __libc_start_main(...) {

    /* 1. 注册动态链接器的终止函数 */
    if (rtld_fini != NULL)
        __cxa_atexit(rtld_fini, NULL, NULL);

    /* 2. 初始化 TLS（线程本地存储）*/
    __libc_init_first(argc, argv, envp);

    /* 3. 注册 libc 的终止函数 */
    __cxa_atexit(fini, NULL, NULL);

    /* 4. 调用 preinit_array 中的函数 */
    call_init(&__preinit_array_start, &__preinit_array_end, argc, argv, envp);

    /* 5. 运行 .init_array（全局构造函数、__attribute__((constructor))）*/
    call_init(&__init_array_start, &__init_array_end, argc, argv, envp);

    /* 6. 终于——调用用户的 main() */
    result = main(argc, argv, envp);

    /* 7. 调用 exit()（运行 atexit 注册函数、.fini_array、析构函数）*/
    exit(result);
    /* 永不到达这里 */
}

完整的调用栈（main 前）

⚛ 第一性原理

__libc_start_main 是整个启动序列的「编排者（Orchestrator）」。它存在的根本原因是：将「平台相关的原始初始化」（_start 负责）与「标准库的初始化逻辑」彻底解耦。_start 是汇编实现的、平台强相关的；而 __libc_start_main 是 C 实现的、可跨架构复用的。这种两层设计使得 glibc 可以支持数十种 CPU 架构，只需要为每种架构编写极少量的 _start 汇编，其余初始化逻辑统一用 C 完成。

⑩ 动态链接可执行文件的启动序列

现代 Linux 系统上，gcc main.c -o main（不加 -static）默认生成动态链接的 ELF 文件。启动过程更复杂，涉及动态链接器 ld-linux.so。

图 10-1：动态链接可执行文件完整启动序列（来源：MaskRay startup-files / glibc 源码）

PLT/GOT 延迟绑定

动态链接下，_start 中的 call *__libc_start_main@GOTPCREL(%rip) 会触发 PLT（Procedure Linkage Table）跳转，第一次调用时通过 GOT（Global Offset Table）转发给动态链接器解析符号地址，之后直接调用。

⚛ 第一性原理

动态链接的本质是将「符号地址绑定」从编译时推迟到运行时，代价是一个额外的间接层（PLT/GOT），收益是多个进程可以共享同一份库的代码页（内存节省）并支持库的热更新。动态链接器（ld.so）是一个「在用户态自举的小型链接器」——它本身是一个可重定向程序，启动时先对自己进行重定位，然后才能执行加载其他库的工作。

⚛ 第一性原理

静态链接是「将所有依赖在链接时完全固化」的策略。它的本质是以空间（更大的二进制）换取简单性（无运行时依赖）和确定性（行为不受系统库版本影响）。静态链接下的启动序列更简单直接：没有 ld.so 的介入，_start → __libc_start_main → main 的链条完全在同一个进程内完成。

⚛ 第一性原理

链接器脚本是对「内存布局即程序架构」这一思想的直接体现。ELF 的 section/segment 二级结构将「逻辑分类」（section：.text/.data/.bss…）与「内存权限」（segment：R/W/X）解耦——链接器脚本定义如何从前者映射到后者。ENTRY(_start) 这一指令的本质是声明「可执行序列从哪个符号对应的地址开始」，是整个启动链条的起点。

⑬ 符号解析与重定位

链接器的核心工作：符号解析将每个引用绑定到唯一的定义；重定位则将目标文件中的「相对地址占位符」替换为最终的虚拟地址。

未定义符号本文件引用，但定义在其他地方U

单遍链接器（Single-pass Linker）的含义：GNU ld 从左到右扫描所有输入，当遇到一个 .o 文件中的未定义符号时，会向后查找（在后续出现的库 .a 文件中）找到定义。这就是为什么 -lc（libc）必须出现在用户 .o 文件之后——用户代码依赖 libc，如果 libc 在前面扫描完后才出现用户代码的未定义符号，链接器不会回头查找。
⚛ 第一性原理

符号解析的本质是解决一个命名空间中名称到定义的映射问题。重定位的本质是将「逻辑空间中的相对引用」转化为「物理（虚拟）地址空间中的绝对/相对偏移」。二者合称「链接」，其数学本质是：给定一个依赖图（符号引用关系），在满足约束的条件下（地址对齐、段权限），找到一个所有节点在地址空间中的合法布局，并将图中每条边（符号引用）替换为对应的数值（地址偏移）。

⚛ 第一性原理（全篇总结）

从第一性原理来看，main.c 与 glibc 入口拼接的全过程，本质上是在解决一个根本问题：「一段用高级语言写的函数，如何成为操作系统可以直接跳转执行的程序入口？」

这个问题分解为三层： ① 语言层：GCC 将 C 语义保持地翻译为机器指令（main.o），但 main() 只是一个普通函数，需要调用者来调用它。 ② 运行时层：glibc 的 CRT 文件（crt1.o 等）提供调用者（_start → __libc_start_main），在 main 之前完成 C 运行时的初始化，在 main 之后完成清理。 ③ 链接层：GNU ld 将所有目标文件按规则拼合，解析符号引用（main 的地址传给 __libc_start_main），填写 ELF 头的入口点（_start 的地址），最终生成操作系统可直接执行的二进制。

整个链条的设计哲学是关注点分离与职责最小化：内核只保证加载和跳转；_start 只做 ABI 格式转换；__libc_start_main 只做运行时编排；main() 只做业务逻辑。每一层都不需要知道其他层的实现细节——这正是「分层抽象」的力量。
