第 0 章

user_atomic.h

基于 GCC 内置原子操作的 C99 兼容封装参考手册

C99 兼容 GCC ≥ 4.1 (__sync) GCC ≥ 4.7 (__atomic) 无锁并发 多平台

介绍与背景

C11 标准在 <stdatomic.h> 中正式引入了原子类型与原子操作。 然而在嵌入式系统、内核代码、老旧工具链等场景中，开发者常常受限于 C99，无法直接使用 C11 原子操作。

GCC 编译器自 4.1 版本起提供了 __sync_* 系列内置函数，自 4.7 版本起进一步提供了功能更完整、语义更精确的 __atomic_* 系列内置函数。 这两套内置函数无需 C11 支持，可在 C99 模式下直接使用，是 GCC 平台上实现无锁并发编程的权威基础设施。

📖 官方文档来源

本手册所有 API 描述均严格依据 GCC 官方手册原文，不作任何臆测性扩展。

本章小结

  • C11 的 <stdatomic.h> 不可用于 C99 项目，GCC 内置函数是最佳替代方案。
  • GCC 提供两套内置函数：__sync_*（GCC ≥ 4.1，兼容/遗留）与 __atomic_*（GCC ≥ 4.7，推荐）。
  • user_atomic.h 统一封装两者，对外提供语义清晰、有中文注释的接口。
第 1 章

为什么需要原子操作

在多线程或多核并发环境中，普通的读-改-写操作（如 counter++）在机器层面通常是 三条指令的序列（Load → Add → Store），这三步不是原子的，存在数据竞争（Data Race）风险。

图 1-1：非原子 counter++ 导致的数据竞争场景

原子操作通过硬件（如 x86 的 LOCK 前缀、ARM 的 LDREX/STREX）保证 Load-Modify-Store 的不可分割性，从而在不使用互斥锁的情况下安全地访问共享变量，实现无锁（Lock-Free）算法。

原子操作 vs 互斥锁

特性原子操作互斥锁 (pthread_mutex)
系统调用开销❌ 无✅ 有（futex）
上下文切换❌ 无✅ 可能有
死锁风险❌ 无✅ 有
适用粒度单变量 RMW复杂临界区
内存序控制✅ 精细（__atomic）默认 SeqCst
C99 可用✅ GCC 内置✅ pthread

本章小结 — 第一性原理

  • 根本矛盾：CPU 的单个机器指令是原子的，但高级语言的复合操作不是。
  • 解决路径：通过硬件原子指令（或总线锁）保证 Load-Modify-Store 序列的不可分割性。
  • 软件视角：原子操作是介于普通变量访问和重量级锁之间的最小同步原语，性能接近普通访问，语义强于无保护访问。
第 2 章

内存序（Memory Order）

__atomic_* 系列函数接受一个 内存序（memory order） 参数，用于精确控制该原子操作与其他内存访问之间的排序约束。GCC 定义了六种内存序，与 C++11 标准一一对应（见 GCC 官方文档 §7.9.1）：

枚举值数值语义描述典型用途
__ATOMIC_RELAXED0 无线程间排序约束，仅保证本操作自身的原子性 统计计数（允许丢失顺序）
__ATOMIC_CONSUME1 当前 GCC 实现等同于 ACQUIRE（C++11 缺陷） 依赖链数据发布（不推荐直接使用）
__ATOMIC_ACQUIRE2 Load 屏障：本操作之后的读写不能被重排到之前 获取锁、读取共享数据
__ATOMIC_RELEASE3 Store 屏障：本操作之前的读写不能被重排到之后 释放锁、发布数据
__ATOMIC_ACQ_REL4 同时具备 ACQUIRE + RELEASE 语义 Read-Modify-Write（如 CAS）
__ATOMIC_SEQ_CST5 顺序一致性：与所有 SEQ_CST 操作形成全序 最强保证，默认安全选择
⚠ 重要说明

根据 GCC 官方文档，__ATOMIC_CONSUME 目前使用更强的 __ATOMIC_ACQUIRE 实现，原因是 C++11 对 memory_order_consume 的语义存在缺陷（deficiency）。在封装头文件中不暴露此选项。

图 2-1：GCC 内存序强度层级（越高约束越强）

本章小结 — 第一性原理

  • 根因：现代 CPU 与编译器都会对指令进行重排序以提升吞吐，内存序的本质是在性能与可见性保证之间做权衡。
  • 选择原则：能用 RELAXED 就用 RELAXED；需要同步（生产-消费）用 RELEASE+ACQUIRE 对；不确定时用 SEQ_CST。
  • __sync_* 系列相当于隐式的 __ATOMIC_SEQ_CST，是全屏障，性能有额外代价但使用最简单。
第 3 章

内存屏障原理

内存屏障（Memory Barrier / Memory Fence）是一类特殊指令，强制处理器和编译器不得将屏障两侧的内存操作越界重排。GCC 提供两类屏障函数（均来自 官方文档 §7.9.1）：

__atomic
void __atomic_thread_fence (int memorder)

在线程之间建立基于指定内存序的同步屏障。所有有效内存序均可使用。

__atomic
void __atomic_signal_fence (int memorder)

在线程与同一线程的信号处理函数之间建立同步屏障。仅影响编译器重排，不发出硬件指令。

图 3-1：内存屏障阻止两侧操作跨越重排
💡 compiler barrier（编译器屏障）

如果只需阻止编译器重排（不需要硬件指令），可使用 GCC 内联汇编： __asm__ __volatile__("" ::: "memory")。 这在单核系统或仅需保证编译顺序时比 __atomic_signal_fence 更轻量。

本章小结

  • __atomic_thread_fence：多线程间的硬件+编译器双重屏障，确保跨核可见性。
  • __atomic_signal_fence：仅针对编译器优化，不生成额外硬件指令，适用于信号处理场景。
  • 屏障本身不直接操作变量，需与原子 load/store 配合使用才能构成完整的同步关系。
第 4 章

原子加载 — __atomic_load_n

原子地从内存位置读取值，不会被其他线程的写操作撕裂。来源：GCC 官方文档 §7.9.1

__atomic · GCC≥4.7
type __atomic_load_n (type *ptr, int memorder)

原子加载操作，返回 *ptr 的内容。有效内存序：__ATOMIC_RELAXED、__ATOMIC_SEQ_CST、__ATOMIC_ACQUIRE、__ATOMIC_CONSUME。

{ return *ptr; } // 等价语义（原子保证）
__atomic · 泛型版
void __atomic_load (type *ptr, type *ret, int memorder)

泛型版本，将 *ptr 的值写入 *ret，支持任意大小数据类型（不一定无锁）。

封装后的用户接口

UA_LOAD(ptr, order) — 原子读取，参数：指针、内存序（默认可用 UA_SEQ_CST）

本章小结

  • 原子 Load 防止"撕裂读"（Torn Read），确保读到的值是某个时刻的完整写入结果。
  • 与 ACQUIRE 配合是生产-消费模式中消费端的标准用法。
  • 泛型版 __atomic_load 对大于 8 字节的类型可能退化为加锁实现。
第 5 章

原子存储 — __atomic_store_n

来源：GCC 官方文档 §7.9.1

__atomic · GCC≥4.7
void __atomic_store_n (type *ptr, type val, int memorder)

原子存储操作，将 val 写入 *ptr。有效内存序：__ATOMIC_RELAXED、__ATOMIC_SEQ_CST、__ATOMIC_RELEASE。

{ *ptr = val; } // 等价语义（原子保证）
__atomic · 泛型版
void __atomic_store (type *ptr, type *val, int memorder)

泛型版本，将 *val 的值存入 *ptr。

⚠ 注意

Store 操作不允许使用 __ATOMIC_ACQUIRE 或 __ATOMIC_ACQ_REL 内存序，否则为未定义行为。

本章小结

  • 原子 Store 防止"撕裂写"（Torn Write），确保其他线程只能看到写入前或写入后的完整值。
  • 与 RELEASE 配合是生产-消费模式中生产端的标准用法（先写数据，再 RELEASE 写标志）。
第 6 章

原子交换 — __atomic_exchange_n

来源：GCC 官方文档 §7.9.1

__atomic · GCC≥4.7
type __atomic_exchange_n (type *ptr, type val, int memorder)

原子交换操作，将 val 写入 *ptr，并返回 *ptr 的旧值。所有内存序均有效。

{ tmp = *ptr; *ptr = val; return tmp; } // 原子
__atomic · 泛型版
void __atomic_exchange (type *ptr, type *val, type *ret, int memorder)

泛型版本，将 *val 存入 *ptr，原始值写入 *ret。

图 6-1：__atomic_exchange_n 操作语义

本章小结

  • Exchange（原子交换）是实现自旋锁最常用的原语之一（写入1，返回旧值判断是否获得锁）。
  • 所有内存序均合法，获锁时建议用 ACQUIRE，释放时用 RELEASE。
第 7 章

比较并交换 — Compare and Exchange (CAS)

来源：GCC 官方文档 §7.9.1

__atomic · GCC≥4.7
bool __atomic_compare_exchange_n (type *ptr, type *expected, type desired, bool weak, int success_memorder, int failure_memorder)

原子比较并交换（CAS）。若 *ptr == *expected，则将 desired 写入 *ptr（Read-Modify-Write），返回 true，内存序为 success_memorder； 否则，将 *ptr 的当前值写入 *expected（Read），返回 false，内存序为 failure_memorder。

weak=true：弱 CAS，允许伪失败（Spurious Failure），多数目标忽略此参数但保留 strong 变体。不确定时请使用 weak=false（强 CAS）。

if (*ptr == *expected) { *ptr = desired; return true; }
else { *expected = *ptr; return false; }

约束：failure_memorder 不能为 __ATOMIC_RELEASE 或 __ATOMIC_ACQ_REL，且不能强于 success_memorder。

__atomic · 泛型版
bool __atomic_compare_exchange (type *ptr, type *expected, type *desired, bool weak, int success_mo, int failure_mo)

与 __atomic_compare_exchange_n 几乎相同，区别在于 desired 通过指针传入。

图 7-1：CAS 操作状态机

本章小结 — 第一性原理

  • CAS 是所有无锁数据结构的核心原语——它是"如果世界没变，我就改变它"的原子化表达。
  • 强 CAS（weak=false） 保证只要 *ptr == *expected 就必定成功，适合大多数场景。
  • 弱 CAS（weak=true） 允许伪失败，在循环重试场景中某些平台（如 ARM LDREX/STREX）性能更好。
  • failure_memorder 不能为 RELEASE/ACQ_REL：失败路径只做 Load，不需要 Release 语义。
第 8 章

原子算术与位运算

来源：GCC 官方文档 §7.9.1。所有内存序均有效。操作数需为整数或指针类型，不可为 bool 类型。

先操作后返回新值 (op_fetch)

这组函数执行操作并返回操作后的新值：{ *ptr op= val; return *ptr; }

函数操作语义返回值
__atomic_add_fetch(ptr, val, mo)*ptr += val新值
__atomic_sub_fetch(ptr, val, mo)*ptr -= val新值
__atomic_and_fetch(ptr, val, mo)*ptr &= val新值
__atomic_or_fetch(ptr, val, mo)*ptr |= val新值
__atomic_xor_fetch(ptr, val, mo)*ptr ^= val新值
__atomic_nand_fetch(ptr, val, mo)*ptr = ~(*ptr & val)新值

先返回旧值后操作 (fetch_op)

这组函数返回操作前的旧值：{ tmp = *ptr; *ptr op= val; return tmp; }

函数操作语义返回值
__atomic_fetch_add(ptr, val, mo)*ptr += val旧值
__atomic_fetch_sub(ptr, val, mo)*ptr -= val旧值
__atomic_fetch_and(ptr, val, mo)*ptr &= val旧值
__atomic_fetch_or(ptr, val, mo)*ptr |= val旧值
__atomic_fetch_xor(ptr, val, mo)*ptr ^= val旧值
__atomic_fetch_nand(ptr, val, mo)*ptr = ~(*ptr & val)旧值
图 8-1：fetch_op 与 op_fetch 返回值差异对比

本章小结

  • fetch_op（先取后改）用于需要知道修改前状态的场景，如：并发引用计数减 1 后判断是否为 0。
  • op_fetch（先改后取）用于需要确认新状态的场景，如：原子自增后立即使用新值。
  • NAND 实现：GCC ≥ 4.4 为 *ptr = ~(*ptr & val)，而非早期的 *ptr = ~*ptr & val，注意区别。

第 10 章

Test-and-Set / Clear

来源：GCC 官方文档 §7.9.1

__atomic · GCC≥4.7
bool __atomic_test_and_set (void *ptr, int memorder)

对 *ptr 处的字节执行原子 Test-and-Set。将该字节设置为某个实现定义的非零"已设置"值， 当且仅当之前内容为"已设置"时返回 true。 应仅用于 bool 或 char 类型。所有内存序均有效。

__atomic · GCC≥4.7
void __atomic_clear (bool *ptr, int memorder)

对 *ptr 执行原子 Clear 操作，将其置 0。有效内存序：__ATOMIC_RELAXED、__ATOMIC_SEQ_CST、__ATOMIC_RELEASE。 若类型非 bool，优先使用 __atomic_store。应与 __atomic_test_and_set 配合使用。

💡 自旋锁实现示例

__atomic_test_and_set + __atomic_clear 是实现极简自旋锁的标准手法： TAS 返回 false 表示锁从未设置（即成功获锁），返回 true 表示已被他人持有。

本章小结

  • TAS 是 "测试旧值并设置新值" 的原子组合，是实现标志位、自旋锁的最基本原语。
  • Clear 配套使用，确保释放操作的原子性与内存可见性。
第 11 章

锁自由检测 — is_lock_free

来源：GCC 官方文档 §7.9.1

__atomic · GCC≥4.7
bool __atomic_always_lock_free (size_t size, void *ptr)

若目标架构对 size 字节的对象总是能生成无锁原子指令，返回 true。 size 必须在编译时可确定，结果也是编译时常量。ptr 可为 0（使用典型对齐）。

示例：if (__atomic_always_lock_free(sizeof(long long), 0)) { ... }
__atomic · GCC≥4.7
bool __atomic_is_lock_free (size_t size, void *ptr)

运行时版本。若编译时无法确定，会调用名为 __atomic_is_lock_free 的运行时例程进行检测。

本章小结

  • 不同平台对不同大小类型的原子操作支持程度不同，在移植代码时应先用此函数检测。
  • 对于不支持无锁的类型，GCC 会回退到基于外部锁的实现，性能差但语义正确。
第 12 章

__sync 系列 — 先取后改（fetch_and_op）Legacy

来源：GCC 官方文档 §7.9.2。 自 GCC 4.1 起可用，是全屏障（Full Barrier）操作。官方文档明确建议：新代码应使用 __atomic 系列而非 __sync 系列。

⚠ 遗留接口说明

根据 GCC 官方文档，__sync_* 系列函数现在由 __atomic_* 内置函数实现，相当于隐式的 __ATOMIC_SEQ_CST。它们被保留是为了向后兼容，新代码不应使用。

以下函数返回操作前的旧值：

{ tmp = *ptr; *ptr op= value; return tmp; }
函数签名操作返回
__sync_fetch_and_add(ptr, value)*ptr += value旧值
__sync_fetch_and_sub(ptr, value)*ptr -= value旧值
__sync_fetch_and_or(ptr, value)*ptr |= value旧值
__sync_fetch_and_and(ptr, value)*ptr &= value旧值
__sync_fetch_and_xor(ptr, value)*ptr ^= value旧值
__sync_fetch_and_nand(ptr, value)*ptr = ~(*ptr & value)旧值

类型可为 1、2、4 或 8 字节的标量（int、long、long long 等），不含 _Bool。

来源：GCC 官方文档 §7.9.2。 以下函数返回操作后的新值：

{ *ptr op= value; return *ptr; }

__sync_add_and_fetch(ptr, value)*ptr += value新值

第 14 章

__sync 系列 — 比较并交换 Legacy

来源：GCC 官方文档 §7.9.2

__sync · GCC≥4.1
bool __sync_bool_compare_and_swap (type *ptr, type oldval, type newval, ...)

原子 CAS。若 *ptr == oldval，则写入 newval 并返回 true；否则返回 false。

__sync · GCC≥4.1
type __sync_val_compare_and_swap (type *ptr, type oldval, type newval, ...)

原子 CAS，返回操作前 *ptr 的值（不论成功与否）。

第 15 章

__sync 系列 — 全内存屏障 Legacy

来源：GCC 官方文档 §7.9.2

__sync · GCC≥4.1
void __sync_synchronize (...)

发出一个全内存屏障（Full Memory Barrier）。任何内存操作均不能跨越此屏障前向或后向移动。 等同于 __atomic_thread_fence(__ATOMIC_SEQ_CST)。

第 16 章

__sync 系列 — 锁原语 Legacy

来源：GCC 官方文档 §7.9.2

__sync · GCC≥4.1
type __sync_lock_test_and_set (type *ptr, type value, ...)

原子交换操作（非传统 Test-and-Set）：将 value 写入 *ptr，返回 *ptr 的旧值。 这是一个获取屏障（Acquire Barrier）：屏障后的操作不能被推测到屏障之前，但屏障前的 Store 可能尚未全局可见。

某些平台仅支持写入立即常量 1，实际写入值由实现定义。

__sync · GCC≥4.1
void __sync_lock_release (type *ptr, ...)

释放由 __sync_lock_test_and_set 获取的锁，通常写入常量 0 到 *ptr。 这是一个释放屏障（Release Barrier）：所有之前的 Store 均全局可见，但之后的读操作不保证不被推测到屏障之前。

第12-16章 小结（__sync 系列综述）

  • __sync_* 系列从 GCC 4.1 起可用，来源于 Intel Itanium ABI 规范，覆盖最广泛的 GCC 版本。
  • 大多数 __sync_* 操作是全屏障，无法像 __atomic_* 那样指定细粒度内存序，性能存在额外开销。
  • 官方已明确标记为遗留（Legacy），封装头文件中仅在 GCC < 4.7 时回退使用，否则优先使用 __atomic_*。

 *   1. 仅依赖 GCC 内置函数，无需 C11 ，完全 C99 兼容
 *   2. GCC >= 4.7 使用 __atomic_* 系列（支持细粒度内存序）
 *   3. GCC >= 4.1 且 < 4.7 回退使用 __sync_* 系列（全屏障语义）
 *   4. 不支持的平台/版本将在编译期报错，不静默降级
 *
 * 用法示例：
 *   int val = UA_LOAD(&my_var, UA_SEQ_CST);
 *   UA_STORE(&my_var, 42, UA_RELEASE);
 *   int old = UA_FETCH_ADD(&my_var, 1, UA_ACQ_REL);
 *   bool ok = UA_CAS_STRONG(&my_var, &expected, desired,
 *                           UA_ACQ_REL, UA_ACQUIRE);
 *
 * 版权声明：公共领域（Public Domain），无限制使用
 */

#ifndef USER_ATOMIC_H
#define USER_ATOMIC_H

#include <stddef.h>   /* size_t */

/* ================================================================
 * §0  GCC 版本检测宏
 *   __GNUC__           主版本号
 *   __GNUC_MINOR__     次版本号
 *   __GNUC_PATCHLEVEL__ 修订号
 * 参考：Fedora ARM Wiki 中对版本检测的权威示例
 *   https://fedoraproject.org/wiki/Architectures/ARM/GCCBuiltInAtomicOperations
 * ================================================================ */
#define UA_GCC_VERSION \
    (__GNUC__ * 10000 + __GNUC_MINOR__ * 100 + __GNUC_PATCHLEVEL__)

/* GCC 4.7.0+ 支持 __atomic_* 系列（内存序感知，推荐） */
#if UA_GCC_VERSION >= 40700
#  define UA_HAVE_ATOMIC_BUILTINS 1
/* GCC 4.1.0+ 支持 __sync_* 系列（全屏障，兼容回退） */
#elif UA_GCC_VERSION >= 40100
#  define UA_HAVE_SYNC_BUILTINS 1
#else
#  error "user_atomic.h: 需要 GCC >= 4.1.0 以支持内置原子操作"
#endif

/* ================================================================
 * §1  内存序常量
 *
 * 官方文档 §7.9.1 定义了 6 种内存序：
 *   __ATOMIC_RELAXED   - 无线程间顺序约束
 *   __ATOMIC_CONSUME   - 当前 GCC 等同于 ACQUIRE（C++11 缺陷）
 *   __ATOMIC_ACQUIRE   - 获取屏障：之后的操作不得重排到之前
 *   __ATOMIC_RELEASE   - 释放屏障：之前的操作不得重排到之后
 *   __ATOMIC_ACQ_REL   - 同时具备 ACQUIRE + RELEASE
 *   __ATOMIC_SEQ_CST   - 顺序一致（最强，全序保证）
 *
 * __sync_* 系列相当于隐式的 __ATOMIC_SEQ_CST（全屏障）。
 * 注意：UA_CONSUME 在当前 GCC 实现中等同于 UA_ACQUIRE。
 * ================================================================ */
#ifdef UA_HAVE_ATOMIC_BUILTINS

/* 无线程间排序约束，仅原子性 */
#  define UA_RELAXED    __ATOMIC_RELAXED

/* 依赖链消费（当前 GCC 实现为 ACQUIRE） */
#  define UA_CONSUME    __ATOMIC_CONSUME

/* 获取屏障：后续操作不得重排到此 Load 之前 */
#  define UA_ACQUIRE    __ATOMIC_ACQUIRE

/* 释放屏障：之前操作不得重排到此 Store 之后 */
#  define UA_RELEASE    __ATOMIC_RELEASE

/* 双向屏障：ACQUIRE + RELEASE，用于 RMW 操作 */
#  define UA_ACQ_REL    __ATOMIC_ACQ_REL

/* 顺序一致性：最强保证，与所有 SEQ_CST 操作形成全序 */
#  define UA_SEQ_CST    __ATOMIC_SEQ_CST

#else
/* __sync_* 回退模式：内存序参数被忽略（始终全屏障） */
#  define UA_RELAXED    0
#  define UA_CONSUME    1
#  define UA_ACQUIRE    2
#  define UA_RELEASE    3
#  define UA_ACQ_REL    4
#  define UA_SEQ_CST    5
#endif

/* ================================================================
 * §2  编译器屏障（Compiler Barrier）
 *
 * 仅阻止编译器重排，不生成任何硬件指令。
 * 适用于：单核系统，或仅需阻止编译器优化的场景。
 * 原理：GCC 内联汇编的 "memory" clobber 通知编译器内存已被修改。
 * ================================================================ */
/* 编译器屏障：阻止编译器将屏障两侧的内存访问重排 */
#define UA_COMPILER_BARRIER() \
    __asm__ __volatile__("" ::: "memory")

/* ================================================================
 * §3  线程内存屏障（Thread Fence）
 *
 * 官方文档 §7.9.1：
 *   __atomic_thread_fence(memorder)
 *     在线程之间建立基于指定内存序的同步屏障。所有内存序均有效。
 *   __atomic_signal_fence(memorder)
 *     在线程与同线程的信号处理函数之间建立屏障。
 *     仅影响编译器重排，不发出硬件指令。
 * ================================================================ */
#ifdef UA_HAVE_ATOMIC_BUILTINS

/* 线程间内存屏障：基于指定内存序，所有内存序均有效 */
#  define UA_THREAD_FENCE(order)   __atomic_thread_fence(order)

/* 线程-信号处理器屏障：仅编译器级别，无硬件指令 */
#  define UA_SIGNAL_FENCE(order)   __atomic_signal_fence(order)

#else

/* 回退：__sync_synchronize() 是全屏障（忽略 order 参数） */
#  define UA_THREAD_FENCE(order)   __sync_synchronize()
#  define UA_SIGNAL_FENCE(order)   UA_COMPILER_BARRIER()

#endif

/* ================================================================
 * §4  原子加载（Atomic Load）
 *
 * 官方文档 §7.9.1 __atomic_load_n:
 *   "This built-in function implements an atomic load operation.
 *    It returns the contents of *ptr."
 *   有效内存序：__ATOMIC_RELAXED, __ATOMIC_SEQ_CST,
 *               __ATOMIC_ACQUIRE, __ATOMIC_CONSUME
 *
 * UA_LOAD(ptr, order)：
 *   原子读取 *ptr 的值并返回，防止撕裂读（Torn Read）。
 *   典型用途：消费端读取共享标志（搭配 UA_ACQUIRE）。
 * ================================================================ */
#ifdef UA_HAVE_ATOMIC_BUILTINS

/* 原子加载：以指定内存序读取 *ptr 的值
 * @ptr:   指向目标变量的指针（1/2/4/8 字节整数或指针类型）
 * @order: 内存序（UA_RELAXED / UA_ACQUIRE / UA_SEQ_CST）
 * @返回:  *ptr 在此时刻的原子快照值 */
#  define UA_LOAD(ptr, order) \
    __atomic_load_n((ptr), (order))

#else

/* 回退：__sync_synchronize() + 普通读（全屏障保证顺序） */
#  define UA_LOAD(ptr, order) \
    ({ __sync_synchronize(); *(ptr); })

#endif

/* ================================================================
 * §5  原子存储（Atomic Store）
 *
 * 官方文档 §7.9.1 __atomic_store_n:
 *   "This built-in function implements an atomic store operation.
 *    It writes val into *ptr."
 *   有效内存序：__ATOMIC_RELAXED, __ATOMIC_SEQ_CST,
 *               __ATOMIC_RELEASE
 *   注意：不可使用 __ATOMIC_ACQUIRE 或 __ATOMIC_ACQ_REL！
 *
 * UA_STORE(ptr, val, order)：
 *   原子写入 val 到 *ptr，防止撕裂写（Torn Write）。
 *   典型用途：生产端写入数据后 RELEASE 标志（发布-订阅模式）。
 * ================================================================ */
#ifdef UA_HAVE_ATOMIC_BUILTINS

/* 原子存储：以指定内存序将 val 写入 *ptr
 * @ptr:   指向目标变量的指针
 * @val:   要写入的值
 * @order: 内存序（UA_RELAXED / UA_RELEASE / UA_SEQ_CST）
 *         注意：禁止使用 UA_ACQUIRE / UA_ACQ_REL */
#  define UA_STORE(ptr, val, order) \
    __atomic_store_n((ptr), (val), (order))

#else

/* 回退：普通写 + __sync_synchronize() 全屏障 */
#  define UA_STORE(ptr, val, order) \
    do { *(ptr) = (val); __sync_synchronize(); } while(0)

#endif

/* ================================================================
 * §6  原子交换（Atomic Exchange）
 *
 * 官方文档 §7.9.1 __atomic_exchange_n:
 *   "It writes val into *ptr, and returns the previous contents
 *    of *ptr. All memory order variants are valid."
 *   语义：{ tmp = *ptr; *ptr = val; return tmp; }
 *
 * UA_EXCHANGE(ptr, val, order)：
 *   原子将 val 写入 *ptr，返回旧值。所有内存序均有效。
 *   典型用途：实现自旋锁（写1返回旧值，旧值为0则获锁成功）。
 * ================================================================ */
#ifdef UA_HAVE_ATOMIC_BUILTINS

/* 原子交换：将 val 写入 *ptr，返回 *ptr 的旧值
 * @ptr:   指向目标变量的指针
 * @val:   要写入的新值
 * @order: 内存序（所有内存序均有效）
 * @返回:  操作前 *ptr 的旧值 */
#  define UA_EXCHANGE(ptr, val, order) \
    __atomic_exchange_n((ptr), (val), (order))

#else

/* 回退：__sync_lock_test_and_set 实现原子交换（仅保证写入1的可靠性）
 * 注意：部分平台 __sync_lock_test_and_set 只支持写入立即常量 1 */
#  define UA_EXCHANGE(ptr, val, order) \
    __sync_lock_test_and_set((ptr), (val))

#endif

/* ================================================================
 * §7  比较并交换（Compare and Exchange / CAS）
 *
 * 官方文档 §7.9.1 __atomic_compare_exchange_n:
 *   "Compares *ptr with *expected. If equal, writes desired into
 *    *ptr (RMW, returns true); otherwise, writes *ptr into
 *    *expected (Read, returns false)."
 *   weak=false（强CAS）：若 *ptr==*expected 则保证成功
 *   weak=true（弱CAS）：允许伪失败，性能更好（某些架构）
 *   约束：failure_memorder 不能为 RELEASE/ACQ_REL，且不能强于
 *         success_memorder
 *
 * UA_CAS_STRONG(ptr, expected, desired, succ_mo, fail_mo)：
 *   强 CAS：不会伪失败，适合大多数场景。
 *   返回 bool：true=成功写入，false=*ptr 已更新到 *expected。
 *
 * UA_CAS_WEAK(ptr, expected, desired, succ_mo, fail_mo)：
 *   弱 CAS：允许伪失败，需在循环中使用，适合高竞争 ARM 场景。
 * ================================================================ */
#ifdef UA_HAVE_ATOMIC_BUILTINS

/* 强比较并交换（推荐通用场景）
 * @ptr:     指向目标变量的指针
 * @expected: 指向期望值的指针，失败时被更新为 *ptr 当前值
 * @desired: 成功时写入 *ptr 的新值
 * @succ_mo: 成功时的内存序（无限制）
 * @fail_mo: 失败时的内存序（不可为 RELEASE/ACQ_REL，不可强于 succ_mo）
 * @返回:    true = 交换成功；false = *ptr != *expected，已更新 expected */
#  define UA_CAS_STRONG(ptr, expected, desired, succ_mo, fail_mo) \
    __atomic_compare_exchange_n((ptr), (expected), (desired), \
                                 0, (succ_mo), (fail_mo))

/* 弱比较并交换（循环重试场景，某些平台性能更优）
 * 允许伪失败：即使 *ptr == *expected 也可能返回 false */
#  define UA_CAS_WEAK(ptr, expected, desired, succ_mo, fail_mo) \
    __atomic_compare_exchange_n((ptr), (expected), (desired), \
                                 1, (succ_mo), (fail_mo))

#else

/* 回退：__sync_bool_compare_and_swap（全屏障，强语义）
 * 注意：无 expected 指针更新语义，失败时 expected 不变 */
#  define UA_CAS_STRONG(ptr, expected, desired, succ_mo, fail_mo) \
    __sync_bool_compare_and_swap((ptr), *(expected), (desired))

/* 弱 CAS 回退为强 CAS（__sync 系列不区分强弱） */
#  define UA_CAS_WEAK(ptr, expected, desired, succ_mo, fail_mo) \
    UA_CAS_STRONG((ptr), (expected), (desired), (succ_mo), (fail_mo))

#endif

/* 便捷宏：CAS 并返回旧值（val 版本）
 * 基于 __sync_val_compare_and_swap / __atomic_compare_exchange_n
 * 成功返回旧值==expected，失败返回不同的当前值（仅 __sync 回退有效） */
#ifdef UA_HAVE_ATOMIC_BUILTINS
/* 原子 CAS，返回操作前 *ptr 的旧值（不论成功与否）
 * @ptr:     指向目标变量的指针
 * @oldval:  期望的旧值（值语义，非指针）
 * @newval:  成功时写入 *ptr 的新值
 * @返回:    操作前 *ptr 的值 */
#  define UA_CAS_VAL(ptr, oldval, newval) \
    ({ \
        __typeof__(*(ptr)) _exp = (oldval); \
        __atomic_compare_exchange_n((ptr), &_exp, (newval), \
                                     0, __ATOMIC_SEQ_CST, __ATOMIC_SEQ_CST); \
        _exp; \
    })
#else
#  define UA_CAS_VAL(ptr, oldval, newval) \
    __sync_val_compare_and_swap((ptr), (oldval), (newval))
#endif

/* ================================================================
 * §8  原子算术与位运算 — 先取后改（fetch_op，返回旧值）
 *
 * 官方文档 §7.9.1 __atomic_fetch_add/sub/and/or/xor/nand:
 *   "perform the operation and return the value that had previously
 *    been in *ptr."
 *   语义：{ tmp = *ptr; *ptr op= val; return tmp; }
 *   所有内存序均有效。操作数须为整数或指针类型，非 bool。
 *
 * NAND 注意：GCC >= 4.4 实现为 *ptr = ~(*ptr & val)
 *            而非早期的 *ptr = ~*ptr & val（结果不同！）
 * ================================================================ */

#ifdef UA_HAVE_ATOMIC_BUILTINS

/* 原子加法（返回旧值）：old = *ptr; *ptr += val; return old
 * @典型用途：并发引用计数自增，检查增前的值 */
#  define UA_FETCH_ADD(ptr, val, order) \
    __atomic_fetch_add((ptr), (val), (order))

/* 原子减法（返回旧值）：old = *ptr; *ptr -= val; return old
 * @典型用途：引用计数自减，若旧值为1则触发释放 */
#  define UA_FETCH_SUB(ptr, val, order) \
    __atomic_fetch_sub((ptr), (val), (order))

/* 原子按位与（返回旧值）：old = *ptr; *ptr &= val; return old */
#  define UA_FETCH_AND(ptr, val, order) \
    __atomic_fetch_and((ptr), (val), (order))

/* 原子按位或（返回旧值）：old = *ptr; *ptr |= val; return old
 * @典型用途：原子设置标志位 */
#  define UA_FETCH_OR(ptr, val, order) \
    __atomic_fetch_or((ptr), (val), (order))

/* 原子按位异或（返回旧值）：old = *ptr; *ptr ^= val; return old
 * @典型用途：原子翻转标志位 */
#  define UA_FETCH_XOR(ptr, val, order) \
    __atomic_fetch_xor((ptr), (val), (order))

/* 原子 NAND（返回旧值）：old = *ptr; *ptr = ~(*ptr & val); return old
 * 注意：GCC >= 4.4 语义为 ~(*ptr & val)，早期版本行为不同 */
#  define UA_FETCH_NAND(ptr, val, order) \
    __atomic_fetch_nand((ptr), (val), (order))

#else  /* __sync_* 回退 */

/* 原子加法（返回旧值） —— __sync 回退版本 */
#  define UA_FETCH_ADD(ptr, val, order) \
    __sync_fetch_and_add((ptr), (val))

/* 原子减法（返回旧值） —— __sync 回退版本 */
#  define UA_FETCH_SUB(ptr, val, order) \
    __sync_fetch_and_sub((ptr), (val))

/* 原子按位与（返回旧值） —— __sync 回退版本 */
#  define UA_FETCH_AND(ptr, val, order) \
    __sync_fetch_and_and((ptr), (val))

/* 原子按位或（返回旧值） —— __sync 回退版本 */
#  define UA_FETCH_OR(ptr, val, order) \
    __sync_fetch_and_or((ptr), (val))

/* 原子按位异或（返回旧值） —— __sync 回退版本 */
#  define UA_FETCH_XOR(ptr, val, order) \
    __sync_fetch_and_xor((ptr), (val))

/* 原子 NAND（返回旧值） —— __sync 回退版本 */
#  define UA_FETCH_NAND(ptr, val, order) \
    __sync_fetch_and_nand((ptr), (val))

#endif

/* ================================================================
 * §9  原子算术与位运算 — 先改后取（op_fetch，返回新值）
 *
 * 官方文档 §7.9.1 __atomic_add_fetch/sub_fetch/and_fetch/...:
 *   "perform the operation and return the result of the operation."
 *   语义：{ *ptr op= val; return *ptr; }
 *   所有内存序均有效。
 * ================================================================ */

#ifdef UA_HAVE_ATOMIC_BUILTINS

/* 原子加法（返回新值）：*ptr += val; return *ptr
 * @典型用途：生成自增序列号并立即使用新值 */
#  define UA_ADD_FETCH(ptr, val, order) \
    __atomic_add_fetch((ptr), (val), (order))

/* 原子减法（返回新值）：*ptr -= val; return *ptr
 * @典型用途：引用计数自减，若新值为0则释放资源 */
#  define UA_SUB_FETCH(ptr, val, order) \
    __atomic_sub_fetch((ptr), (val), (order))

/* 原子按位与（返回新值）：*ptr &= val; return *ptr */
#  define UA_AND_FETCH(ptr, val, order) \
    __atomic_and_fetch((ptr), (val), (order))

/* 原子按位或（返回新值）：*ptr |= val; return *ptr */
#  define UA_OR_FETCH(ptr, val, order) \
    __atomic_or_fetch((ptr), (val), (order))

/* 原子按位异或（返回新值）：*ptr ^= val; return *ptr */
#  define UA_XOR_FETCH(ptr, val, order) \
    __atomic_xor_fetch((ptr), (val), (order))

/* 原子 NAND（返回新值）：*ptr = ~(*ptr & val); return *ptr */
#  define UA_NAND_FETCH(ptr, val, order) \
    __atomic_nand_fetch((ptr), (val), (order))

#else  /* __sync_* 回退 */

/* 原子加法（返回新值） —— __sync 回退版本 */
#  define UA_ADD_FETCH(ptr, val, order) \
    __sync_add_and_fetch((ptr), (val))

/* 原子减法（返回新值） —— __sync 回退版本 */
#  define UA_SUB_FETCH(ptr, val, order) \
    __sync_sub_and_fetch((ptr), (val))

/* 原子按位与（返回新值） —— __sync 回退版本 */
#  define UA_AND_FETCH(ptr, val, order) \
    __sync_and_and_fetch((ptr), (val))

/* 原子按位或（返回新值） —— __sync 回退版本 */
#  define UA_OR_FETCH(ptr, val, order) \
    __sync_or_and_fetch((ptr), (val))

/* 原子按位异或（返回新值） —— __sync 回退版本 */
#  define UA_XOR_FETCH(ptr, val, order) \
    __sync_xor_and_fetch((ptr), (val))

/* 原子 NAND（返回新值） —— __sync 回退版本 */
#  define UA_NAND_FETCH(ptr, val, order) \
    __sync_nand_and_fetch((ptr), (val))

#endif

/* ================================================================
 * §10  自增 / 自减便捷宏（基于 §8 / §9）
 *
 * 这些宏提供最常用的原子自增/自减操作。
 * 使用 SEQ_CST 以保证最强内存序（安全优先）。
 * 需要更细粒度控制时，直接使用 UA_FETCH_ADD / UA_ADD_FETCH。
 * ================================================================ */

/* 原子自增 1，返回旧值（等效于 val++） */
#define UA_INC(ptr)   UA_FETCH_ADD((ptr), 1, UA_SEQ_CST)

/* 原子自减 1，返回旧值（等效于 val--） */
#define UA_DEC(ptr)   UA_FETCH_SUB((ptr), 1, UA_SEQ_CST)

/* 原子自增 1，返回新值（等效于 ++val） */
#define UA_INC_FETCH(ptr)   UA_ADD_FETCH((ptr), 1, UA_SEQ_CST)

/* 原子自减 1，返回新值（等效于 --val） */
#define UA_DEC_FETCH(ptr)   UA_SUB_FETCH((ptr), 1, UA_SEQ_CST)

/* ================================================================
 * §11  Test-and-Set / Clear（测试并设置 / 清除）
 *
 * 官方文档 §7.9.1 __atomic_test_and_set:
 *   "performs an atomic test-and-set operation on the byte at *ptr.
 *    The byte is set to some implementation defined nonzero 'set'
 *    value and the return value is true if and only if the previous
 *    contents were 'set'. It should be only used for operands of
 *    type bool or char."
 *   所有内存序均有效。
 *
 * 官方文档 §7.9.1 __atomic_clear:
 *   "performs an atomic clear operation on *ptr. After the operation,
 *    *ptr contains 0."
 *   有效内存序：__ATOMIC_RELAXED, __ATOMIC_SEQ_CST, __ATOMIC_RELEASE
 *   "It should be only used for operands of type bool or char and
 *    in conjunction with __atomic_test_and_set."
 *
 * 典型用途：实现极简自旋锁
 *   while (UA_TAS(&lock, UA_ACQUIRE)) { /* 自旋等待 *\/ }
 *   /* 临界区 *\/
 *   UA_CLEAR(&lock, UA_RELEASE);
 * ================================================================ */
#ifdef UA_HAVE_ATOMIC_BUILTINS

/* 原子 Test-and-Set：将 *ptr 置位，返回置位前是否已被设置
 * @ptr:   指向 bool 或 char 类型的指针
 * @order: 内存序（所有内存序均有效）
 * @返回:  true = 之前已被设置（获锁失败）；false = 之前未设置（获锁成功） */
#  define UA_TAS(ptr, order) \
    __atomic_test_and_set((ptr), (order))

/* 原子 Clear：将 *ptr 清零（配合 UA_TAS 释放锁）
 * @ptr:   指向 bool 或 char 类型的指针
 * @order: 内存序（UA_RELAXED / UA_SEQ_CST / UA_RELEASE）
 *         注意：禁止使用 UA_ACQUIRE / UA_ACQ_REL */
#  define UA_CLEAR(ptr, order) \
    __atomic_clear((ptr), (order))

#else

/* 回退：__sync_lock_test_and_set（acquire barrier）
 * 仅保证写入 1，返回旧值非零表示已被设置 */
#  define UA_TAS(ptr, order) \
    (__sync_lock_test_and_set((ptr), 1) != 0)

/* 回退：__sync_lock_release（release barrier）
 * 将 *ptr 置 0，释放锁 */
#  define UA_CLEAR(ptr, order) \
    __sync_lock_release((ptr))

#endif

/* ================================================================
 * §12  锁自由检测（Lock-Free Detection）
 *
 * 官方文档 §7.9.1 __atomic_always_lock_free:
 *   "returns true if objects of size bytes always generate lock-free
 *    atomic instructions for the target architecture.
 *    size must resolve to a compile-time constant."
 *
 * 官方文档 §7.9.1 __atomic_is_lock_free:
 *   "returns true if objects of size bytes always generate lock-free
 *    atomic instructions for the target architecture.
 *    If the built-in function is not known to be lock-free, a call
 *    is made to a runtime routine named __atomic_is_lock_free."
 *
 * UA_ALWAYS_LOCK_FREE(size)：编译时常量检测，是否总是无锁
 * UA_IS_LOCK_FREE(size)：运行时检测（不确定时调用运行时函数）
 * ================================================================ */
#ifdef UA_HAVE_ATOMIC_BUILTINS

/* 编译时检测：给定字节数的对象在此平台上是否总是无锁原子操作
 * @size: 对象字节数（必须为编译时常量）
 * @返回: true = 保证无锁；false = 可能使用锁实现
 * 示例：UA_ALWAYS_LOCK_FREE(sizeof(int)) */
#  define UA_ALWAYS_LOCK_FREE(size) \
    __atomic_always_lock_free((size), 0)

/* 运行时检测：给定字节数的对象在此平台上是否无锁原子操作
 * @size: 对象字节数
 * @返回: true = 无锁；false = 使用锁实现（可能有性能开销） */
#  define UA_IS_LOCK_FREE(size) \
    __atomic_is_lock_free((size), 0)

#else

/* 回退：__sync_* 系列支持 1/2/4/8 字节，视为满足常见需求 */
#  define UA_ALWAYS_LOCK_FREE(size) \
    ((size) == 1 || (size) == 2 || (size) == 4 || (size) == 8)
#  define UA_IS_LOCK_FREE(size) \
    UA_ALWAYS_LOCK_FREE(size)

#endif

/* ================================================================
 * §13  全内存屏障便捷宏（对应 __sync_synchronize）
 *
 * 官方文档 §7.9.2 __sync_synchronize:
 *   "This built-in function issues a full memory barrier."
 *   等同于 __atomic_thread_fence(__ATOMIC_SEQ_CST)
 * ================================================================ */

/* 全内存屏障：阻止所有内存操作跨越此点重排（等同于 __sync_synchronize）
 * 适用于不关心内存序细节、需要最强同步保证的简单场景 */
#define UA_FULL_BARRIER()   UA_THREAD_FENCE(UA_SEQ_CST)

/* ================================================================
 * §14  自旋锁辅助宏（基于 UA_TAS / UA_CLEAR）
 *
 * 用法示例（类型为 char 或 bool 的锁变量）：
 *   static char my_lock = 0;
 *   UA_SPINLOCK_LOCK(&my_lock);
 *   // ... 临界区 ...
 *   UA_SPINLOCK_UNLOCK(&my_lock);
 *
 * 警告：自旋锁适用于临界区极短的场景。长临界区请使用 pthread_mutex。
 * ================================================================ */

/* 自旋加锁：循环等待直到获取锁
 * @lockptr: 指向 char/bool 类型锁变量的指针（初始值应为 0）
 * 注意：在单核系统上此宏会死锁，仅用于多核环境 */
#define UA_SPINLOCK_LOCK(lockptr) \
    do { \
        while (UA_TAS((lockptr), UA_ACQUIRE)) { \
            /* 自旋等待：CPU 退让，避免饥饿（可选 pause/yield） */ \
        } \
    } while (0)

/* 自旋解锁：释放锁
 * @lockptr: 指向 char/bool 类型锁变量的指针 */
#define UA_SPINLOCK_UNLOCK(lockptr) \
    UA_CLEAR((lockptr), UA_RELEASE)

#endif /* USER_ATOMIC_H */

本章小结

  • 头文件通过 UA_GCC_VERSION 宏自动选择 __atomic_*（GCC≥4.7）或 __sync_*（GCC≥4.1）。
  • 所有接口均以 UA_ 前缀统一命名，避免与用户代码及系统宏冲突。
  • 内存序常量、屏障、加载/存储、CAS、算术位运算、TAS/Clear、锁自由检测均有完整封装。
第 18 章

使用示例

示例 1：无锁引用计数

refcount.c无锁引用计数
#include "user_atomic.h"
#include <stdio.h>

typedef struct {
    int refcount;   /* 原子引用计数 */
    char data[64];
} MyObject;

/* 增加引用计数（返回旧值） */
static void obj_retain(MyObject *obj) {
    UA_FETCH_ADD(&obj->refcount, 1, UA_RELAXED);
    /* RELAXED：只需原子性，引用增加本身无需同步其他内存 */
}

/* 释放引用，返回释放后的计数 */
static int obj_release(MyObject *obj) {
    /* RELEASE：确保临界区内的所有写操作在减计数前全局可见 */
    int old = UA_FETCH_SUB(&obj->refcount, 1, UA_RELEASE);
    if (old == 1) {
        /* 最后一个引用：需要 ACQUIRE 屏障确保看到所有写操作 */
        UA_THREAD_FENCE(UA_ACQUIRE);
        printf("对象销毁\n");
        return 0;
    }
    return old - 1;
}

示例 2：无锁标志发布（生产-消费模式）

publish.c生产-消费同步
#include "user_atomic.h"

static int  shared_data;    /* 共享数据 */
static int  data_ready = 0; /* 就绪标志（原子变量） */

/* 生产者线程 */
void producer(void) {
    shared_data = 42;                            /* 1. 写入数据 */
    UA_STORE(&data_ready, 1, UA_RELEASE);       /* 2. RELEASE 发布标志
                                                        保证步骤1对消费者可见 */
}

/* 消费者线程 */
void consumer(void) {
    int ready;
    do {
        ready = UA_LOAD(&data_ready, UA_ACQUIRE); /* ACQUIRE 读取标志
                                                        与 RELEASE 配对，
                                                        保证看到生产者的写入 */
    } while (!ready);
    /* 此处可安全读取 shared_data */
}

示例 3：强 CAS 无锁链表头部插入

list.c无锁单链表头插入
#include "user_atomic.h"

typedef struct Node { int val; struct Node *next; } Node;
static Node *list_head = 0; /* 原子链表头（用指针大小对齐） */

/* 无锁头部插入（CAS 重试循环） */
void list_push(Node *node) {
    Node *expected;
    do {
        expected = UA_LOAD(&list_head, UA_RELAXED); /* 读取当前头 */
        node->next = expected;
        /* CAS：若 list_head 未被其他线程修改，则完成插入 */
    } while (!UA_CAS_STRONG(&list_head, &expected, node,
                              UA_RELEASE, UA_RELAXED));
}

示例 4：自旋锁的使用

spinlock.c极简自旋锁
#include "user_atomic.h"

static char g_lock = 0; /* 自旋锁变量，初始为未锁定状态（0） */
static int  g_counter = 0;

void thread_func(void) {
    UA_SPINLOCK_LOCK(&g_lock);   /* 获取锁（自旋等待） */
    g_counter++;                 /* 临界区 */
    UA_SPINLOCK_UNLOCK(&g_lock); /* 释放锁 */
}

/* 或者，检测平台是否支持无锁 int 操作 */
void check_platform(void) {
    if (UA_ALWAYS_LOCK_FREE(sizeof(int))) {
        /* int 类型的原子操作在此平台上保证无锁（编译时常量） */
    }
}

本章小结

  • 引用计数：增加用 RELAXED（无需同步），减少用 RELEASE，最后一个 Release 后用 ACQUIRE Fence。
  • 生产-消费：生产端 RELEASE Store，消费端 ACQUIRE Load，配对建立 happens-before 关系。
  • 无锁插入：CAS 重试循环是无锁算法的通用模板，成功用 RELEASE 发布，失败用 RELAXED。
  • 自旋锁：TAS 用 ACQUIRE 获锁，Clear 用 RELEASE 解锁，语义与 pthread_mutex 一致。

全局总结 — 第一性原理

  • 原子操作的本质：将多步骤的 Load-Modify-Store 序列压缩为单个不可分割的硬件操作，消除数据竞争的根本来源。
  • 内存序的本质：现代 CPU 和编译器的乱序执行是性能的来源，内存序是在"允许多少乱序"与"保证多少可见性"之间精确划定边界的语言机制。
  • __atomic vs __sync：__atomic_* 支持细粒度内存序，是对 CPU 硬件能力的直接表达；__sync_* 始终使用全屏障，语义简单但性能有代价，已被官方标记为遗留接口。
  • 封装价值：user_atomic.h 抹平了 GCC 版本差异，统一了接口命名，通过中文注释降低了使用门槛，是在 C99 项目中安全引入无锁并发的最小基础设施。
  • 使用建议：优先使用 UA_FETCH_ADD/UA_CAS_STRONG 等高层接口；需要精确控制性能时再选择适当的内存序；在不确定时始终选择 UA_SEQ_CST——正确性优先于性能。
