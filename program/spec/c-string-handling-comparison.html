引言 · 为何要关注字符串处理

C 语言没有内置的字符串类型，字符串本质上是以 \0（NUL 字节）结尾的 char 数组。这一设计简洁但危险——缓冲区溢出、未终止字符串、O(n) 的长度查询，这些问题伴随了 C 语言几十年。各大开源项目和操作系统内核都基于各自的设计哲学，提出了截然不同的字符串解决方案。

本文综合 GLib、APR（Apache Portable Runtime）、Redis SDS、libuv 及 Linux 内核 的官方源码与文档，逐一剖析其设计原理、实现细节、优缺点，最终给出一张适用于日常开发选型的 决策树。

C 标准库字符串 libc

C 标准库（POSIX / ISO C）提供的字符串函数是所有其他方案的基础，同时也是安全隐患的主要来源。理解它们的局限是理解高级方案存在意义的前提。

strcpy / strncpy 的设计缺陷

strcpy 完全不做边界检查，一旦源字符串超出目标缓冲区，就会发生经典的缓冲区溢出（Buffer Overflow）。内核文档将其列为 禁用函数：

📄 内核废弃接口文档 📄 man7 string_copying(7) 
/* 标准库的问题示例 */
char dst[8];
strcpy(dst, "Hello, World!"); /* 溢出! 写入 14 字节到 8 字节缓冲 */

/* strncpy 同样有陷阱：不保证 NUL 终止！*/
char buf[4];
strncpy(buf, "Hello", sizeof(buf));
/* buf = {'H','e','l','l'} ← 没有 '\0' 终止符！ */
/* 下次 strlen(buf) 会读越界 */

Linux 内核文档明确指出 strncpy() 的三大问题：① 不保证 NUL 终止；② 若源串短于 n，会用 NUL 填充剩余部分（不必要的性能开销）；③ 返回值为目标指针起始，无法判断是否截断。

⚠️ Linux 内核文档（deprecated.rst）原文："strncpy() does not guarantee that the destination buffer will be NUL terminated. This can lead to various linear read overflows and other misbehavior due to the missing termination."

strcat / strncat

/* strcat 的问题：每次都要先遍历 dst 找到末尾 → O(n²) 拼接 */
char buf[128] = "";
for (int i = 0; i < 100; i++)
    strcat(buf, "x"); /* 第 i 次调用遍历 i 个字符 → O(n²) */
优点
  • 零依赖，任何 C 环境可用
  • 编译器内置优化（memcpy/memset SIMD）
  • 开销极小，无额外分配
缺点
  • strcpy 无边界检查
  • strncpy 不保证 NUL 终止
  • strlen 是 O(n) 操作
  • strcat 连续调用为 O(n²)
  • 不支持二进制数据（含 NUL 字节）

GLib · GString LGPL-2.1+

📄 官方文档 GLib.String 💻 gstring.c 源码 💻 gstring.h 头文件

GLib 是 GNOME 桌面生态的基础库，GString 是其动态字符串实现。核心理念是：提供一个类似 C++ std::string 的可变长字符串，隐藏底层的内存管理复杂度。

结构设计

来自 glib/gstring.h（GNOME/glib main 分支，源码已验证）：

/* glib/gstring.h - GNOME/glib main branch */
struct _GString {
    gchar  *str;          /* 指向实际字符数据，始终 NUL 终止 */
    gsize   len;          /* 当前字符串长度（字节数，不含 NUL）*/
    gsize   allocated_len;/* 已分配的总字节数（含 NUL 终止符）*/
};

与 C 标准字符串不同，GString 始终记录 len，因此 sdslen 类似的操作是 O(1)。str 字段保持与 C 字符串的兼容性——可以直接传给 printf 等函数。

官方文档说明："A GString is similar to a standard C string, except that it grows automatically as text is appended or inserted. Also, it stores the length of the string, so can be used for binary data with embedded nul bytes."
（来源：GLib Strings）

动态扩容策略

来自 glib/gstring.c，扩容由 g_string_maybe_expand 触发： 

/* glib/gstring.c - GNOME/glib main branch */
static inline void
g_string_maybe_expand (GString *string, gsize len)
{
  if (G_UNLIKELY (len >= string->allocated_len - string->len))
    g_string_expand(string, len);
}

g_string_expand 内部使用 指数增长（幂次轮转分配）：找到最小的 2 的幂次方满足需求，避免频繁 realloc。

主要 API（来自官方文档）：

GString* g_string_new(const gchar *init);
GString* g_string_sized_new(gsize dfl_size); /* 预分配容量，减少 realloc */
GString* g_string_append(GString *string, const gchar *val);
GString* g_string_append_len(GString *string, const gchar *val, gssize len);
void     g_string_append_printf(GString *string, const gchar *format, ...);
GString* g_string_insert_len(GString *string, gssize pos,
                              const gchar *val, gssize len);
GString* g_string_erase(GString *string, gssize pos, gssize len);
gchar*   g_string_free(GString *string, gboolean free_segment);
优点
  • 自动扩容，无需手动管理缓冲区
  • O(1) 长度查询
  • 支持二进制数据（embedded NUL）
  • Unicode / UTF-8 感知 API
  • 丰富的操作（insert/erase/overwrite/printf）
  • 与 C 字符串兼容（str 字段）
缺点
  • 依赖 GLib 大型库
  • 每个 GString 需要独立 malloc（元数据+数据）
  • 非线程安全（需外部加锁）
  • 指数扩容可能浪费内存
  • 不适合内核或嵌入式场景

APR · 内存池字符串 Apache-2.0

📄 APR 官方字符串文档 💻 apr_strings.h 源码 💻 apr_strings.c 源码

APR（Apache Portable Runtime）是 Apache HTTP Server 的跨平台基础库。其字符串设计核心在于 内存池（Memory Pool）——所有字符串分配都依附于某个 apr_pool_t，池销毁时批量回收，彻底回避了单次 free 的复杂性。

内存池策略

来自 include/apr_pools.h 的注释（已验证原文）：

"designed so that we don't have to keep track of EVERYTHING so that it can be explicitly freed later (a fundamentally unsound strategy — particularly in the presence of die()). Instead, we maintain pools, and allocate items (both memory and I/O handlers) from the pools." 
/* 内存池使用示例 */
apr_pool_t *mp;
apr_pool_create(&mp, NULL);

/* 所有字符串分配都从池中申请，无需逐一释放 */
char *s1 = apr_pstrdup(mp, "hello");
char *s2 = apr_pstrcat(mp, "hello", " ", "world", NULL);
char *s3 = apr_psprintf(mp, "value=%d", 42);

/* 一次性销毁，回收所有字符串内存 */
apr_pool_destroy(mp);

APR 的池分配器内部维护一个 apr_allocator_t，当前活跃的内存节点（apr_memnode_t）有足够空间时直接线性分配： 

/* 池分配器核心逻辑（简化）*/
if (size < node_free_space(active)) {
    mem = active->first_avail;
    active->first_avail += size; /* 指针碰撞分配 O(1) */
    return mem;
}

核心 API 与实现

来自 strings/apr_strings.c（原文验证）：

/* apr_pstrdup: 从池中复制字符串 */
char *apr_pstrdup(apr_pool_t *a, const char *s) {
    char *res;
    apr_size_t len = strlen(s) + 1;
    res = apr_palloc(a, len);
    memcpy(res, s, len);
    return res;
}

/* apr_pstrcat: 多字符串拼接，先计算总长再一次 palloc */
char *apr_pstrcat(apr_pool_t *a, ...) {
    /* Pass 1: 计算所有参数总长度（缓存前 6 个）*/

    /* 以 NULL 哨兵结束 va_list */
}

apr_pstrcat 使用了 MAX_SAVED_LENGTHS = 6 的优化：缓存前 6 个参数的长度以避免重复 strlen。

安全函数 apr_cpystrn（对比 strncpy 的改进）：

官方文档原文三点区别：
1) strncpy() doesn't always NUL terminate; apr_cpystrn() does.
2) strncpy() pads the destination string with NULs; apr_cpystrn() does not.
3) strncpy() returns pointer to beginning of dst; apr_cpystrn() returns pointer to the NUL terminator of dst.
优点
  • 池分配 O(1) 均摊，吞吐极高
  • 批量回收，几乎零内存碎片
  • 请求级生命周期管理天然契合 Web Server 场景
  • 跨平台（Windows/Unix/macOS）
  • 线程安全池分配器
缺点
  • 字符串不可独立回收（只能等池销毁）
  • 不适合长生命周期的字符串对象
  • 需要理解池的生命周期才能正确使用
  • 池泄漏时内存占用高
  • 无 GString 式的动态追加 API

Redis SDS · Simple Dynamic Strings BSD-3

📄 Redis 官方 SDS 文档 💻 sds.h 源码 💻 sds.c 源码

SDS 由 Redis 作者 Salvatore Sanfilippo（antirez）设计，是 Redis 内部所有字符串的底层实现，同时也以独立库（antirez/sds）形式开源。SDS 最具创意的设计在于 "透明头部"——对外暴露的 sds 类型本质上是 char*，指向 buf 区域，通过负偏移访问元数据，对已有 C 代码完全透明。

多类型头部设计（SDSLib 2.0）

SDS 2.0（Redis 3.2+）引入了 5 种头部类型，根据字符串长度选择最小头部，减少内存占用（来自 sds.h 源码原文）：

#define SDS_TYPE_5  0  /* 最大 31 字节，flags 高 5 位存长度 */
#define SDS_TYPE_8  1  /* 最大 255 字节 */
#define SDS_TYPE_16 2  /* 最大 65535 字节 */
#define SDS_TYPE_32 3  /* 32 位平台最大 */
#define SDS_TYPE_64 4  /* 64 位平台超大字符串 */

struct __attribute__ ((__packed__)) sdshdr8 {
    uint8_t len;          /* 当前已用长度 */
    uint8_t alloc;        /* 总容量（不含头部和 NUL 终止符）*/
    unsigned char flags;  /* 低 3 位：类型标志 */
    char buf[];           /* 柔性数组，实际字符数据 */
};

/* sds 类型本质是 char*，指向 buf 区域 */
typedef char *sds;

/* 通过负偏移反向访问元数据（指针运算技巧）*/
static inline size_t sdslen(const sds s) {
    unsigned char flags = s[-1]; /* flags 字段在 buf 前一字节 */
    switch(flags & SDS_TYPE_MASK) {
        case SDS_TYPE_5:  return SDS_TYPE_5_LEN(flags);
        case SDS_TYPE_8:  return SDS_HDR(8, s)->len;
        case SDS_TYPE_16: return SDS_HDR(16, s)->len;
        /* ... */
    }
}

预分配与惰性释放策略

来自 sds.h 源码（antirez/sds master 分支）：

#define SDS_MAX_PREALLOC (1024 * 1024) /* 预分配上限：1 MiB */

/* sdsMakeRoomFor 扩容逻辑：
 * - 新长度 < 1MB：分配 2 倍所需空间（Prealloc 翻倍）
 * - 新长度 >= 1MB：每次多分配 1MB                        */
if (newlen < SDS_MAX_PREALLOC)
    newlen *= 2;   /* 小字符串：×2 预分配 */
else
    newlen += SDS_MAX_PREALLOC; /* 大字符串：+1MB */

惰性释放（lazy free）：sdsRemoveFreeSpace 不会立即归还多余内存，而是仅更新 len，保留 alloc 不变。这避免了短时间内频繁缩短后又扩展时的多次 realloc。

优点
  • O(1) 长度查询（len 字段）
  • 与 C 字符串完全兼容（sds = char*）
  • 二进制安全（可含 NUL）
  • 多类型头部节省小字符串内存
  • 预分配策略减少 realloc 频率
  • 可脱离 Redis 独立使用
缺点
  • 内存布局依赖 packed 属性（依赖编译器）
  • 指针运算技巧较隐晦，初学者难理解
  • 每次追加都需要 realloc（无池分配优势）
  • 非线程安全
  • 大字符串预分配可能浪费 1MB

libuv · uv_buf_t 与字符串工具 MIT

📄 libuv 官方文档 misc 💻 uv.h 源码

libuv 是 Node.js 的异步 I/O 底层库，其字符串处理哲学与其他库截然不同：libuv 不做字符串管理，只做字节缓冲区抽象。其核心字符串相关类型 uv_buf_t 只是一个薄薄的缓冲区描述符；真正的字符串工具函数专注于 Windows 与 UNIX 之间的 UTF-16 / WTF-8 互转。

uv_buf_t 缓冲区抽象

/* include/uv.h - libuv v1.x 分支 */
typedef struct uv_buf_t {
    char*  base;  /* 指向数据区的指针（由调用者管理生命周期）*/
    size_t len;   /* 数据区字节数 */
} uv_buf_t;

/* 使用示例（来自官方文档 utilities.html）*/
uv_buf_t buf;
buf.base = "Hello TTY\n";
buf.len  = strlen(buf.base);
uv_write(&req, (uv_stream_t*) &tty, &buf, 1, NULL);

uv_buf_t 的设计类似 POSIX 的 struct iovec，仅描述缓冲区，不持有所有权。生命周期由调用者负责。这与 libuv 的异步 I/O 模型高度契合：回调发生时，base 指针对应的内存必须在整个 I/O 操作期间有效。

Unicode 字符串工具函数（v1.47.0+）

来自 libuv 官方文档（String manipulation functions 章节）：

"These string utilities are needed internally for dealing with Windows, and are exported to allow clients to work uniformly with this data when the libuv API is not complete." 
/* UTF-16 ↔ WTF-8 互转（Windows 路径文件名处理）*/
size_t  uv_utf16_length_as_wtf8(const uint16_t* utf16, ssize_t utf16_len);
int     uv_utf16_to_wtf8(const uint16_t* utf16, ssize_t utf16_len,
                           char** wtf8_ptr, size_t* wtf8_len_ptr);
ssize_t uv_wtf8_length_as_utf16(const char* wtf8);
void    uv_wtf8_to_utf16(const char* wtf8, uint16_t* utf16,
                          size_t utf16_len);

WTF-8 是 UTF-8 的超集，可无损表示不合规的 UTF-16（包含孤立代理项的 Windows 文件名）。这解决了 libuv 跨平台文件名处理的历史遗留问题（Issue #2048）。

优点
  • 设计极简，零内存管理负担
  • 适合高频异步 I/O 场景
  • 与 scatter/gather I/O (iovec) 天然契合
  • UTF-16/WTF-8 互转解决跨平台问题
缺点
  • 不做字符串管理，需调用者自行维护生命周期
  • 无动态增长、拼接等高级功能
  • 仅适合 libuv I/O 上下文
  • Unicode 工具函数仅 v1.47.0+ 可用

Linux 内核字符串 GPL-2.0

📄 LWN: Ushering out strlcpy() 📄 LWN: strscpy() API 引入 📄 内核废弃接口文档 💻 lib/string.c 源码

内核的字符串需求与用户空间截然不同：不能使用动态分配（某些上下文），必须应对不可信数据（驱动、用户空间输入），且需要极致性能。内核经历了 strcpy → strncpy → strlcpy → strscpy 的演进。

strscpy 的演进与必要性

LWN.net 文章（2022）对内核 strlcpy 的分析（原文）：

"One obvious shortcoming is that this function will read the entire source string regardless of whether that data will be copied or not... bad things can happen if the source string is untrusted. If src is not NUL-terminated, then strlcpy() will continue merrily off the end until it does find a NUL byte."
— LWN.net, "Ushering out strlcpy()", 2022-08-25

内核 strlcpy 的实现（来自 lib/string.c 源码）：

/* lib/string.c - 内核原版 strlcpy（现已废弃）*/
size_t strlcpy(char *dest, const char *src, size_t size) {
    size_t ret = strlen(src); /* 问题：必须先读完整个 src */
    if (size) {
        size_t len = (ret >= size) ? size - 1 : ret;
        memcpy(dest, src, len);
        dest[len] = '\0';
    }
    return ret; /* 返回 src 长度 */
}

strscpy 的接口（2015 年，Chris Metcalf 引入，来自同一文件）：

/* lib/string.c - strscpy 定义 */
/**
 * Copy the string, or as much of it as fits, into the dest buffer.
 * The routine returns the number of characters copied (not including
 * the trailing NUL) or -E2BIG if the destination buffer wasn't big enough.
 * The behavior is undefined if the string buffers overlap.
 * The destination buffer is always NUL terminated, unless it's zero-sized.
 *
 * Preferred to strlcpy() since the API doesn't require reading memory
 * from the src string beyond the specified "count" bytes.
 */
ssize_t strscpy(char *dest, const char *src, size_t count);

字级别（word-at-a-time）优化

strscpy 在支持高效非对齐访问的架构（CONFIG_HAVE_EFFICIENT_UNALIGNED_ACCESS）上，采用逐字（long 大小）读取，通过位运算快速检测 NUL 字节，比逐字节慢循环快约 4 倍。 

/* lib/string.c - word-at-a-time 优化核心 */
while (max >= sizeof(unsigned long)) {
    unsigned long c, data;
    c = *(unsigned long *)(src + res); /* 一次读 8 字节 */
    if (has_zero(c, &data, &constants)) { /* 是否含 NUL 字节 */
        data = prep_zero_mask(c, data, &constants);
        data = create_zero_mask(data);
        *(unsigned long *)(dest + res) = c & zero_bytemask(data);
        return res + find_zero(data);
    }
    *(unsigned long *)(dest + res) = c;
    res += sizeof(unsigned long);
}
strscpy 优点
  • 始终 NUL 终止（除缓冲区为 0 时）
  • 不读 src 超过 count 字节（安全）
  • 返回值易检查截断（-E2BIG）
  • word-at-a-time 优化高效
  • 不受 src 竞争修改影响
内核字符串的通用限制
  • 静态缓冲区，无动态扩容
  • 不支持 Unicode 感知操作
  • 返回值语义与 strlcpy 不同，需注意兼容
  • strscpy_pad 为了 NUL 填充有额外开销

决策树 · 如何选型

根据项目的约束和需求，按以下流程做出最优选择：

🐧 内核 / 嵌入式

使用 strscpy() 替代所有 strcpy/strncpy/strlcpy。固定大小缓冲区，静态分配优先。

🌐 Web 服务器

APR 内存池模型。请求开始创建子池，结束时统一销毁，零碎片，零泄漏风险。

⚡ 高性能缓存

Redis SDS。O(1) 长度、二进制安全、与 C 字符串兼容，预分配策略减少 realloc。

🖥️ 桌面 / 通用

GLib GString。Unicode 感知，insert/erase/printf 全功能，适合 GNOME/GTK 生态。

🔌 异步 I/O

libuv uv_buf_t 描述符。生命周期由调用者管理，配合 scatter/gather I/O，零拷贝传输。

📦 通用安全复制

避免 strcpy/strncpy，改用 snprintf 或平台提供的 strlcpy，并检查返回值。

最佳实践总结

通用原则

1. 永远不用 strcpy，改用带长度的版本
内核文档（deprecated.rst）明确："strcpy performs no bounds checking on the destination buffer. This could result in linear overflows beyond the end of the buffer."

2. 避免 strncpy 的陷阱
若需安全复制，用 snprintf(dst, size, "%s", src) 或平台专用 API（strscpy/apr_cpystrn）。

3. 连续追加时预估容量
GString 用 g_string_sized_new(estimated_size)，SDS 用 sdsMakeRoomFor，避免多次 realloc。

4. 在请求处理路径上用池分配器
APR 的池模型能将请求级内存碎片降至零，大幅简化内存追踪。

5. 异步 I/O 中，缓冲区的生命周期必须覆盖整个操作
libuv 中，uv_buf_t.base 指向的内存必须在回调触发前保持有效。

各场景速查

一句话结论

没有"最好"的字符串方案，只有"最合适"的方案。

选择的核心维度：内存分配策略（手动 vs 池 vs 自动 GC）× 性能敏感度 × 生命周期模型 × 二进制安全需求。

若无特殊约束，用户空间通用开发推荐 GLib GString（功能最完整）或 Redis SDS（性能与轻量并重）；内核/系统编程优先 strscpy；高并发服务端考虑 APR Pool。
