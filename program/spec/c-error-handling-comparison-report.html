引言与研究背景

错误处理是系统级 C 程序设计中最基础也最容易被忽视的议题。 不同的开源项目在面对"如何向调用方传递错误信息"时，采用了截然不同的哲学和实现方式。 本文以权威文档和源码为依据，深度解析以下四种主流方案：

POSIXerrno 机制

  • POSIX.1 / C 标准原生
  • 线程局部存储全局变量
  • 所有 Unix/Linux 系统调用基础

GLibGError 体系

  • GNOME / GTK 生态核心库
  • 结构化错误对象 + 域/码/消息
  • 类异常的 C 错误传播

libuv负整数常量

  • Node.js 底层异步 I/O 库
  • 跨平台统一负整数错误码
  • 异步回调友好设计

Kernel多层返回约定

  • Linux 内核权威实践
  • 负整数 + ERR_PTR 双轨
  • 无 errno，无动态内存
📋 研究方法说明 本文所有结论均严格来源于官方文档和公开源码，包括： GLib 官方文档（docs.gtk.org）、 libuv 官方文档（docs.libuv.org）、 Linux 内核文档（kernel.org）及 Linux 内核源码（include/linux/err.h）、 POSIX 标准（man7.org/errno.3）。

POSIX errno 机制

errno 是 C 语言和 POSIX 标准定义的全局（线程局部）错误指示变量，是所有 POSIX 系统调用和 C 标准库函数报告错误的底层基础。 ↗ man7 errno(3) ↗ POSIX errno

工作原理

根据 POSIX 标准和 man7 文档，errno 是定义在 <errno.h> 中的可修改左值（modifiable lvalue），类型为 int。 在多线程环境中，它是线程局部变量，在一个线程中赋值不影响其他线程。

POSIX 官方原文（pubs.opengroup.org） "errno is thread-local; setting it in one thread does not affect its value in any other thread." — man7.org/errno.3
"The value of errno should only be examined when it is indicated to be valid by a function's return value." — POSIX errno

其核心工作流程如下：

  1. 调用者调用可失败函数（如 open()、malloc()）
  2. 函数通过返回值（通常是 -1 或 NULL）指示失败
  3. 失败时，函数将具体错误码写入 errno（正整数，如 ENOENT=2、EACCES=13）
  4. 调用者检查返回值后再读取 errno

标准使用规范

#include <errno.h>
#include <stdio.h>
#include <string.h>
#include <fcntl.h>

int main() {
    /* 关键：调用前若需精确判断，先清零 errno */
    errno = 0;
    int fd = open("/nonexistent", O_RDONLY);

    if (fd == -1) {
        /* 先检查返回值，再读 errno */
        fprintf(stderr, "Error %d: %s\n", errno, strerror(errno));
        /* errno == ENOENT (2): No such file or directory */
        if (errno == ENOENT) {
            /* 具体处理逻辑 */
        }
    }
    return 0;
}
⚠️ POSIX 官方警告 "The value of errno is significant only when the return value of the call indicated an error (i.e., -1 from most system calls; -1 or NULL from most library functions); a function that succeeds is allowed to change errno." — man7 errno(3)
绝不能仅凭 errno != 0 判断是否出错，必须先检查函数返回值。

优缺点分析

✅ 优点

  • 无额外内存分配，零开销
  • 所有平台普遍支持
  • 与系统调用天然集成
  • 线程安全（线程局部存储）
  • strerror() 提供可读描述

❌ 缺点

  • 只能携带单一整数，无上下文信息
  • 中间层调用会覆盖 errno
  • 容易被遗忘或错误读取
  • 错误码含义因平台而异
  • 不适合丰富的错误信息传递

GLib GError 结构化错误体系

GLib 是 GNOME 平台的核心工具库，其 GError 机制是 C 语言中最完善的结构化错误处理方案之一，相当于 C 语言的"异常系统"。 ↗ 官方文档 Error Reporting ↗ GError 结构

数据结构与核心原理

根据 GLib 官方文档，GError 结构体包含三个字段：

/* 来源: https://docs.gtk.org/glib/struct.Error.html */
struct GError {
    GQuark  domain;   /* 错误域，例如 G_FILE_ERROR */
    gint    code;     /* 错误码，例如 G_FILE_ERROR_NOENT */
    gchar  *message;  /* UTF-8 可读错误消息 */
};
GLib 官方原文（docs.gtk.org） "GError should only be used to report recoverable runtime errors, never to report programming errors. If the programmer has screwed up, then you should use g_warning(), g_return_if_fail(), g_assert(), g_error(), or some similar facility." — GLib Error Reporting

GQuark 错误域命名约定（来自官方文档）：

  • 错误域命名：<NAMESPACE>_<MODULE>_ERROR，如 G_SPAWN_ERROR、G_THREAD_ERROR
  • Quark 函数：<namespace>_<module>_error_quark()
  • 错误码枚举：<Namespace><Module>Error，如 GThreadError
  • 枚举成员：<NAMESPACE>_<MODULE>_ERROR_<CODE>
  • 通用失败码：<NAMESPACE>_<MODULE>_ERROR_FAILED

核心 API 使用模式

以下代码完全来自官方文档示例：

① 报告错误（被调用方）

/* 来源: https://docs.gtk.org/glib/error-reporting.html */
int
foo_open_file(GError **error)
{
    int fd;
    int saved_errno;

    /* 防御性检查：error 若非 NULL，则 *error 必须为 NULL */
    g_return_val_if_fail(error == NULL || *error == NULL, -1);

    fd = open("file.txt", O_RDONLY);
    saved_errno = errno;  /* 立即保存，防止被覆盖 */

    if (fd < 0) {
        g_set_error(error,
                    FOO_ERROR,               // 错误域
                    FOO_ERROR_BLAH,          // 错误码
                    "Failed to open file: %s",
                    g_strerror(saved_errno));
        return -1;
    }
    return fd;
}

② 消费错误（调用方）

/* 来源: https://docs.gtk.org/glib/error-reporting.html */
char *contents;
GError *err = NULL;  /* 必须初始化为 NULL */

g_file_get_contents("foo.txt", &contents, NULL, &err);
g_assert((contents == NULL && err != NULL) ||
         (contents != NULL && err == NULL));

if (err != NULL) {
    fprintf(stderr, "Unable to read file: %s\n", err->message);
    g_error_free(err);  /* 必须手动释放 */
} else {
    /* 使用 contents */
    g_free(contents);
}

③ 传播错误（中间层）

/* 来源: https://docs.gtk.org/glib/error-reporting.html */
gboolean
my_function_that_can_fail(GError **err)
{
    GError *tmp_error;

    g_return_val_if_fail(err == NULL || *err == NULL, FALSE);

    tmp_error = NULL;
    sub_function_that_can_fail(&tmp_error);

    if (tmp_error != NULL) {
        /* 将 tmp_error 移入 err（若 err 非 NULL），否则 free */
        g_propagate_error(err, tmp_error);
        return FALSE;
    }
    return TRUE;
}

④ 忽略错误（合法模式）

/* 传 NULL 表示"不关心错误详情"，等价于 try{} catch(...){} */
if (g_file_get_contents("foo.txt", &contents, NULL, NULL))
    /* 成功 */;
else
    /* 失败，但忽略具体原因 */;

官方规则摘要

GLib 官方强制规则（docs.gtk.org）
  • 不能用 GError 报告编程错误（参数错误、断言失败等）
  • GError* 传入前必须初始化为 NULL
  • GError 结构体不能在栈上分配
  • "Error pileup"（错误堆积）永远是 bug——每次错误后必须处理或向上传播
  • 返回 FALSE 时，error 必须被设置为非 NULL 值
  • 函数如果有 GError 参数，控制流不应依赖调用者是否传入 NULL
— GLib Error Reporting: Rules for use of GError
⚠️ 常见错误模式（官方明确标注为 bug） 以下代码是 错误的，因为 tmp_error 被第一次调用设置后，第二次调用又覆盖了它（错误堆积）： 
/* BUG: 错误堆积 - 来自官方文档反例 */
tmp_error = NULL;
sub_function_that_can_fail(&tmp_error);
other_function_that_can_fail(&tmp_error);  /* ← 若前一步失败则覆盖错误！ */

GLib 2.68+ 扩展 GError 域（Extended GError Domains）

自 GLib 2.68 起，可通过 G_DEFINE_EXTENDED_ERROR() 宏扩展 GError 类型，为特定域附加私有数据字段（如解析错误 ID、请求详情等），实现类似 C++ 异常子类的效果。 ↗ 官方文档

优缺点分析

✅ 优点

  • 携带域 + 码 + 消息，信息极丰富
  • gobject-introspection 可自动转为绑定语言的异常
  • g_error_matches() 实现精确分类处理
  • 传 NULL 可合法忽略错误
  • g_prefix_error() 支持上下文追加
  • 2.68+ 支持扩展私有数据

❌ 缺点

  • 每次错误需堆内存分配（性能）
  • 格式化消息字符串有额外开销
  • 不适合高频错误路径（如 EAGAIN 重试循环）
  • 调用链越深，分配次数越多
  • 需严格遵循使用规则，否则出现内存泄漏
官方对性能问题的说明 "GError has disadvantages though: it requires a memory allocation, and formatting the error message string has a performance overhead. This makes it unsuitable for use in retry loops where errors are a common case, rather than being unusual. For example, using G_IO_ERROR_WOULD_BLOCK means hitting these overheads in the normal control flow." — GLib Error Reporting

libuv 负整数错误码体系

libuv 是 Node.js 的底层异步 I/O 库，提供跨平台（Unix/Windows）统一的错误处理机制。 ↗ 官方文档 Error Handling

工作原理

libuv 官方文档原文（docs.libuv.org） "In libuv errors are negative numbered constants. As a rule of thumb, whenever there is a status parameter, or an API functions returns an integer, a negative number will imply an error. When a function which takes a callback returns an error, the callback will never be called." — libuv Error Handling

平台差异（官方实现细节）：

注： "Implementation detail: on Unix error codes are the negated errno (or -errno), while on Windows they are defined by libuv to arbitrary negative numbers." — libuv errors.html

从 libuv 0.10 到 1.0 的重要变迁（来自官方迁移指南）：

官方迁移说明（docs.libuv.org） "Error handling had a major overhaul in libuv 1.0. In general, functions and status parameters would get 0 for success and -1 for failure on libuv 0.10, and the user had to use uv_last_error to fetch the error code, which was a positive number. In 1.0, functions and status parameters contain the actual error code, which is 0 for success, or a negative number in case of error." — libuv 0.10 → 1.0 migration guide 
/* libuv 0.10 旧方式 */
r = uv_listen((uv_stream_t*) server, 511, NULL);
if (r == -1) {
    uv_err_t err = uv_last_error(uv_default_loop());
    /* err.code 包含 UV_EADDRINUSE */
}

/* libuv 1.x 新方式：返回值直接就是错误码 */
r = uv_listen((uv_stream_t*) server, 511, NULL);
if (r < 0) {
    /* r 直接是 UV_EADDRINUSE 等负值 */
    fprintf(stderr, "Listen error: %s\n", uv_strerror(r));
}

核心 API 设计

libuv 提供的错误处理 API（来自官方文档）：

/* 将平台原生错误码转换为 libuv 错误码 */

                  int events, int status) {

        /* status 为负数即 libuv 错误，callback 若出错也永不被调用 */
        fprintf(stderr, "Error: %s\n", uv_strerror(status));

}

优缺点分析

✅ 优点

  • 零内存分配，性能极高
  • 跨平台统一错误码（Unix/Windows）
  • 异步回调模型天然适配
  • 错误即返回值，不可忽视
  • uv_strerror_r 线程安全版本
  • uv_translate_sys_error 桥接系统错误

❌ 缺点

  • 无结构化上下文信息（只有整数）
  • 无法携带自定义错误消息
  • 负整数含义需查文档，可读性差
  • 无法自定义错误域
  • 调试时信息量有限

Kernel Linux 内核错误处理体系

Linux 内核没有 errno 变量（那是用户空间的），内核有自己成熟的多层错误返回约定体系。 ↗ 内核 Hacking 文档 ↗ include/linux/err.h

内核关键事实 "There is no errno variable in Linux kernel: this variable lives in user space only. If kernel function wants to report about the error and specify the error code, it encapsulates the error code into the returning value." — StackOverflow / 内核社区共识

约定一：负整数返回值（0/-errno）

内核官方文档原文（kernel.org） "For code called in user context, it's very common to defy C convention, and return 0 for success, and a negative error number (eg. -EFAULT) for failure. This can be unintuitive at first, but it's fairly widespread in the kernel." — Unreliable Guide To Hacking The Linux Kernel

内核函数有三种返回整数的约定（来自内核社区文档）：

  1. 0 / -err：成功返回 0，失败返回负错误码（最常见）
  2. +val / -err：成功返回正整数（如读取字节数），失败返回负错误码
  3. +val / 0 / -err：当 0 也是有效结果时
/* 内核函数典型模式 */
/* include/uapi/asm-generic/errno-base.h */
#define EPERM   1   /* Operation not permitted */
#define ENOENT  2   /* No such file or directory */
#define EINVAL  22  /* Invalid argument */
#define ENOMEM  12  /* Out of memory */

/* ioctl 内核代码示例 */
asmlinkage long sys_myioctl(int arg)
{
    if (!valid(arg))
        return -EINVAL;   /* 返回负错误码 */

    if (signal_pending(current))
        return -ERESTARTSYS;  /* 被信号中断，系统调用可重启 */

    return 0;  /* 成功 */
}

约定二：ERR_PTR / IS_ERR / PTR_ERR 体系

内核官方文档原文（kernel.org） "Using ERR_PTR() (include/linux/err.h) to encode a negative error number into a pointer, and IS_ERR() and PTR_ERR() to get it back out again: avoids a separate pointer parameter for the error number. Icky, but in a good way." — Unreliable Guide To Hacking The Linux Kernel

以下 API 来自 include/linux/err.h：

/*
 * Kernel pointers have redundant information, so we can use a
 * scheme where we can return either an error code or a normal
 * pointer with the same return value.
 * 来源: include/linux/err.h (torvalds/linux)
 */
#define MAX_ERRNO  4095

/* 判断是否为错误指针（值在 [-4095, -1] 范围内） */
#define IS_ERR_VALUE(x) \
    unlikely((unsigned long)(void *)(x) >= (unsigned long)-MAX_ERRNO)

/* 将负错误码编码进指针 */
static inline void * __must_check ERR_PTR(long error) {
    return (void *) error;
}

/* 从错误指针提取错误码 */
static inline long __must_check PTR_ERR(__force const void *ptr) {
    return (long) ptr;
}

/* 检测指针是否为错误指针 */
static inline bool __must_check IS_ERR(__force const void *ptr) {
    return IS_ERR_VALUE((unsigned long)ptr);
}

/* 同时检查 NULL 和错误指针 */
static inline bool __must_check IS_ERR_OR_NULL(__force const void *ptr) {
    return unlikely(!ptr) || IS_ERR_VALUE((unsigned long)ptr);
}

/* 便捷宏：若为错误指针则返回错误码，否则返回 0 */
static inline int __must_check PTR_ERR_OR_ZERO(__force const void *ptr) {
    if (IS_ERR(ptr))
        return PTR_ERR(ptr);
    else
        return 0;
}

典型使用模式：

/* 被调用方：返回有效指针或错误指针 */
struct mystruct *myfunc(void)
{
    struct mystruct *mys = kzalloc(sizeof(*mys), GFP_KERNEL);
    if (!mys)
        return ERR_PTR(-ENOMEM);  /* 内存分配失败 */

    if (some_setup_fails(mys)) {
        kfree(mys);
        return ERR_PTR(-EINVAL);  /* 初始化失败 */
    }
    return mys;  /* 成功返回真实指针 */
}

/* 调用方：用 IS_ERR 检测，PTR_ERR 提取错误码 */
struct mystruct *obj = myfunc();
if (IS_ERR(obj)) {
    long err = PTR_ERR(obj);   /* 获取负错误码，如 -ENOMEM */
    pr_err("myfunc failed: %ld\n", err);
    return err;
}

/* 便捷写法 */
return PTR_ERR_OR_ZERO(myfunc());
IS_ERR vs IS_ERR_OR_NULL 的选择 根据内核社区实践：IS_ERR_OR_NULL() 应仅用于返回值可能同时为 NULL（表示"功能已禁用"的合法状态）和错误指针的函数。 对于正常函数，应优先使用 IS_ERR()，滥用 IS_ERR_OR_NULL() 会掩盖 bug 并降低代码可读性。 ↗ 详细分析

printk(KERN_CRIT    "...");  /* 临界条件 */
printk(KERN_ERR     "...");  /* 错误条件 */
printk(KERN_WARNING "...");  /* 警告 */

优缺点分析

✅ 优点

  • 零动态内存分配
  • ERR_PTR 无需额外指针参数
  • __must_check 防止忽略错误
  • 极致性能，适合中断上下文
  • 错误码与 POSIX 标准对齐

❌ 缺点

  • 无上下文信息（只有整数码）
  • ERR_PTR 机制反直觉
  • 混淆 NULL / 错误指针易出 bug
  • 不适用于用户空间（无法直接复用）
  • 错误信息依赖 printk，不在返回值中

场景处方

🔧
场景 1：POSIX 系统编程 / 底层库（无外部依赖）
使用 errno + 负整数返回值。调用后立即保存 errno，先检查返回值再读 errno。
参考：open()、read()、pthread_create()（直接返回错误码，不用 errno）
🖥️
场景 2：GUI 应用 / GNOME 生态 / 需要语言绑定的库
使用 GError。函数签名末尾增加 GError **error，错误时 g_set_error()，成功时确保 *error == NULL。
参考：g_file_get_contents()、g_spawn_command_line_sync()
⚡
场景 3：异步事件驱动框架 / 网络 I/O 库
使用负整数错误码（libuv 风格）。回调函数的 status 参数 < 0 即错误，用 uv_strerror() 获取描述。
参考：libuv 所有异步 API、Node.js 底层 C++ 绑定
⚙️
场景 4：Linux 内核驱动 / 内核模块
整数函数用 return -ERRNO，指针函数用 ERR_PTR(-ERRNO) + IS_ERR() + PTR_ERR()。错误信息靠 printk(KERN_ERR ...)。
参考：所有 include/linux/err.h 用法
🏎️
场景 5：高频热路径 / 重试循环（如 EAGAIN）
避免使用 GError（每次错误都分配内存）。使用简单整数码或 errno，配合 g_set_error_literal() 减少格式化开销（如必须用 GLib）。
参考：GLib 官方文档对 G_IO_ERROR_WOULD_BLOCK 的明确警告。
📦
场景 6：独立 C 库（无 GLib 依赖，但需丰富错误信息）
自定义错误结构体，参考 GError 约定：输出参数 MyError **err，包含错误码和消息缓冲区，调用方负责清理。

总结与参考

核心结论

通过对四种主流 C 错误处理机制的深度研究，可以得出以下核心结论：

  1. 没有银弹：每种方案都有其特定的设计目标和适用场景。POSIX errno 适合系统调用，GError 适合应用层，libuv 负整数适合异步 I/O，内核双轨体系适合驱动开发。
  2. 性能 vs 信息量的权衡：零分配的整数错误码（errno、负整数、内核）性能最好但信息最少；GError 信息最丰富但有堆分配和格式化开销，GLib 官方文档明确说明其不适合高频错误路径。
  3. 错误绝不能被默默忽略：Linux 内核通过 __must_check 属性强制要求检查返回值；GLib 通过 GError** 输出参数的约定使错误明确可见；libuv 通过将错误码直接放在返回值中使其不可忽视。
  4. 边界处需要错误转换：不同层之间调用时，需要显式将错误表示从一种形式转换为另一种，如将 POSIX errno 转换为 GError，或用 uv_translate_sys_error() 桥接平台错误码。
  5. 文档先行：无论选择哪种机制，都必须在 API 文档中明确说明：什么情况下会出错、出错时返回什么、调用方如何处理和释放错误资源。
