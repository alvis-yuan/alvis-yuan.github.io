🧠 什么是设计模式

设计模式（Design Patterns）是软件工程中对反复出现的设计问题的通用、可复用的解决方案。它们不是直接可用的代码，而是描述了在某种情境下，解决某类问题的一种方案描述或模板。

👥
四人帮（Gang of Four，GoF）
Design Patterns: Elements of Reusable Object-Oriented Software, 1994

由 Erich Gamma、Richard Helm、Ralph Johnson、John Vlissides 合著，于1994年出版。此书提出了23种经典设计模式，迅速成为面向对象软件开发的基石，影响了数十年的软件工程实践。

🔗 原文参考：DigitalOcean GoF Patterns

⚙️ 为什么在C语言中使用设计模式

设计模式是与语言无关的架构概念。在C语言中，通过 结构体（struct）、函数指针（function pointer） 和 不透明指针（opaque pointer） 可以有效实现这些模式。C语言大型开源项目（Linux内核、Redis、Nginx、libevent等）中大量使用了设计模式思想。

✅
使用设计模式的好处
  • 代码复用：减少冗余，提升复用性
  • 可维护性：结构清晰、易修改
  • 共同词汇：团队沟通更高效
  • 解耦：降低模块间依赖
  • 可扩展：新增功能影响最小
⚠️
C语言实现要素
  • 函数指针：模拟多态与虚方法
  • 结构体：模拟类与接口
  • 不透明指针：实现封装
  • 回调函数：事件/观察者机制
  • 宏与泛型：增强代码复用

📂 三大模式分类

🏗️
创建型（Creational）

处理对象创建机制，将对象的实例化过程抽象化，使系统独立于对象的创建、组合和表示方式。

5种：Singleton、Factory Method、Abstract Factory、Builder、Prototype

🔗
结构型（Structural）

处理类和对象的组合，形成更大的结构，重点是如何将对象组合以获得更好的灵活性和效率。

7种：Adapter、Bridge、Composite、Decorator、Facade、Flyweight、Proxy

🎭
行为型（Behavioral）

处理对象之间的算法和职责分配，关注对象之间的通信方式，实现松耦合和灵活扩展。

11种：Observer、Strategy、Command、Chain、State、Template、Iterator、Mediator、Visitor、Memento、Interpreter

🔒 单例模式（Singleton）

确保一个类只有一个实例，并提供一个全局访问点。 — GoF《设计模式》

创建型全局实例线程安全

📐 原理

单例模式通过将构造函数私有化（在C中通过静态变量+访问函数实现），确保类只能被实例化一次，并提供一个全局访问该实例的方法。

图1：单例模式结构图 — 确保全局唯一实例

🎯 使用场景

  • 管理共享资源（如数据库连接池、线程池）
  • 全局配置管理器（应用程序配置只需一份）
  • 日志系统（统一写入日志）
  • 操作系统的设备驱动（硬件资源唯一）

✅ 优势

✅
控制全局状态全局只有一个实例，避免资源的多重占用和状态不一致
🔗
全局访问点无需传递对象引用，任何地方都可访问该实例
⚠️
注意：单例模式引入全局状态，多线程环境下需加锁保护。过度使用会导致强耦合，应谨慎使用。

💻 C语言实现示例（含线程安全）

以下为线程安全的单例模式C实现，使用 pthread_mutex 进行双重检查锁定（DCL）：

C — singleton.c
#include <stdio.h>
#include <stdlib.h>
#include <pthread.h>

typedef struct {
    int value;
} Singleton;

static Singleton* instance = NULL;
static pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;

/* 双重检查锁定（DCL）— 线程安全的懒汉式单例 */
Singleton* getInstance() {
    if (instance == NULL) {
        pthread_mutex_lock(&mutex);
        if (instance == NULL) {
            instance = (Singleton*)malloc(sizeof(Singleton));
            instance->value = 0;
        }
        pthread_mutex_unlock(&mutex);
    }
    return instance;
}

int main() {
    Singleton* s1 = getInstance();
    Singleton* s2 = getInstance();
    s1->value = 42;
    printf("s1==s2: %s, value=%d\n",
           s1 == s2 ? "true" : "false", s2->value);
    /* 输出：s1==s2: true, value=42 */
    free(instance);
    return 0;
}

🏭 工厂方法模式（Factory Method）

定义一个用于创建对象的接口，让子类决定实例化哪一个类。工厂方法使一个类的实例化延迟到其子类。 — GoF《设计模式》

创建型解耦创建多态

📐 原理

工厂方法将对象的创建逻辑集中到一个工厂函数中，客户端无需知道具体类的名字，只需调用工厂函数传入类型标识符即可获得正确的对象实例。在C语言中通过函数指针（vtable）实现产品的多态行为。

图2：工厂方法模式 — 由工厂函数决定创建哪种产品

🎯 使用场景

  • 当不知道该创建哪个具体类的对象时
  • 需要集中对象创建逻辑，便于维护
  • 希望通过配置或参数决定创建什么对象
  • 需要在运行时动态切换产品类型

✅ 优势

  • 解耦：客户端与具体类解耦，只依赖抽象接口
  • 开闭原则：新增产品类型只需扩展工厂，无需修改客户端
  • 集中创建：对象创建逻辑集中管理，易于维护

💻 C语言实现示例

C — factory_method.c（基于 mohitmishra786.github.io）
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

/* 产品接口（通过函数指针模拟多态） */
typedef struct {
    void (*operation)(void);
} Product;

/* 具体产品A */
void operationA(void) { printf("ProductA: 执行操作A\n"); }

/* 具体产品B */
void operationB(void) { printf("ProductB: 执行操作B\n"); }

/* 工厂函数 — 核心 */
Product* createProduct(const char* type) {
    Product* product = (Product*)malloc(sizeof(Product));
    if (strcmp(type, "A") == 0) {
        product->operation = operationA;
    } else if (strcmp(type, "B") == 0) {
        product->operation = operationB;
    } else {
        free(product); return NULL;
    }
    return product;
}

int main() {
    Product* pa = createProduct("A");
    Product* pb = createProduct("B");
    pa->operation();  /* 输出：ProductA: 执行操作A */
    pb->operation();  /* 输出：ProductB: 执行操作B */
    free(pa); free(pb);
    return 0;
}

🏛️ 抽象工厂模式（Abstract Factory）

提供一个接口，用于创建相关或相互依赖的对象族，而无需指定其具体类。 — GoF《设计模式》

创建型产品族工厂的工厂

📐 原理

抽象工厂是"工厂的工厂"。它提供创建一系列相关对象的接口，确保这些对象相互兼容。客户端使用工厂接口，不依赖任何具体产品实现。

图3：抽象工厂模式 — 创建相关产品族

🎯 使用场景 & ✅ 优势

使用场景

  • 需要创建一系列相关对象，确保相互兼容
  • 系统需要独立于产品创建、组合和表示
  • 需要一个产品族的多个变体（如跨平台UI组件）
  • Nginx 模块化配置解析不同协议族

优势

  • 确保同族产品相互兼容
  • 客户端与具体产品完全解耦
  • 更换产品族只需替换工厂
  • 符合开闭原则（对扩展开放）

🔨 建造者模式（Builder）

将一个复杂对象的构建与它的表示分离，使得同样的构建过程可以创建不同的表示。 — GoF《设计模式》

创建型分步构建复杂对象

📐 原理 & 使用场景

当对象的创建需要多个步骤，且各步骤可以有不同配置时，使用建造者模式。它将构建过程封装在 Builder 对象中，Director 控制构建顺序。

🎯 使用场景

  • 创建具有多个可选参数的复杂对象（避免"重叠构造函数"）
  • 同一构建过程可生成不同产品（如HTTP请求头构建）
  • Redis 响应报文构建（RESP协议格式化）
  • Nginx 配置块逐步解析构建

💻 C语言实现示例

C — builder.c（基于 mohitmishra786.github.io）
typedef struct {
    char type[50]; char size[50]; char topping[50];
} Pizza;

typedef struct PizzaBuilder {
    Pizza pizza;
    void (*setType)(struct PizzaBuilder*, const char*);
    void (*setSize)(struct PizzaBuilder*, const char*);
    Pizza* (*build)(struct PizzaBuilder*);
} PizzaBuilder;

/* 使用示例 */
PizzaBuilder* builder = newPizzaBuilder();
builder->setType(builder, "Margherita");
builder->setSize(builder, "Large");
Pizza* pizza = builder->build(builder);
/* 输出: Pizza order: Margherita, Large */

📋 原型模式（Prototype）

用原型实例指定创建对象的种类，并通过拷贝这些原型创建新对象。 — GoF《设计模式》

创建型克隆复制

📐 原理 & 使用场景

原型模式通过克隆已有对象来创建新对象，适用于对象初始化代价高昂或对象结构复杂时。

🎯 使用场景

  • 对象初始化开销大，克隆更高效
  • 需要创建与现有对象相似但略有不同的对象
  • Linux 进程 fork() 系统调用（写时复制）
  • Nginx 配置复制（master→worker进程）

✅ 优势

  • 减少重复初始化代价
  • 运行时动态增减产品种类
  • 支持深拷贝，完全隔离对象状态
  • 配合工厂模式使用效果更佳
🐧
Linux — fork() 写时复制（Copy-on-Write）

Linux 的 fork() 系统调用是原型模式的经典应用：子进程以父进程为原型创建，通过写时复制（COW）机制高效共享内存页，只在写入时才真正复制。

Unix — fork原型克隆示意
pid_t pid = fork();  /* 克隆当前进程为原型 */
if (pid == 0) {
    /* 子进程：从父进程原型克隆 */
} else {
    /* 父进程：原型保持不变 */
}

🔌 适配器模式（Adapter）

将一个类的接口转换成客户希望的另外一个接口。Adapter 模式使得原本由于接口不兼容而不能一起工作的那些类可以一起工作。 — GoF《设计模式》

结构型接口转换兼容

📐 原理

适配器（Adapter）包装一个已有对象（Adaptee），使其接口与客户期望的目标接口（Target）兼容。类似现实中的电源适配器。

图4：适配器模式 — 将 Adaptee 接口适配为 Target 接口

🔍 libevent — 多路复用后端适配器

libevent 是适配器模式的完美示例：它将 epoll/kqueue/select/poll 等不同的系统 I/O 接口统一适配为内部的 eventop 接口，对上层提供一致的 API。

libevent — event-internal.h（简化）
/* Target 接口：统一的事件操作接口 */
struct eventop {
    const char *name;
    void* (*init)(struct event_base*);
    int  (*add)(struct event_base*, evutil_socket_t, ...);
    int  (*del)(struct event_base*, evutil_socket_t, ...);
    int  (*dispatch)(struct event_base*, struct timeval*);
};

/* Adapter: epoll 后端适配实现 */
const struct eventop epollops = {
    "epoll",
    epoll_init,    /* 包装 epoll_create() */
    epoll_add,     /* 包装 epoll_ctl(EPOLL_CTL_ADD) */
    epoll_del,     /* 包装 epoll_ctl(EPOLL_CTL_DEL) */
    epoll_dispatch /* 包装 epoll_wait() */
};

🌉 桥接模式（Bridge）

将抽象部分与它的实现部分分离，使它们都可以独立地变化。 — GoF《设计模式》

结构型解耦抽象与实现独立变化

📐 原理 & 使用场景

桥接模式通过组合（而非继承）将抽象层和实现层分开，使两者可以独立演化。当抽象和实现都需要扩展时，避免类爆炸问题。

🎯 使用场景

  • 希望避免抽象和实现永久绑定
  • 抽象和实现都应通过子类扩展
  • 避免类的数量爆炸（M×N种组合）
  • Linux DRM 图形驱动：设备抽象与硬件实现分离

✅ 优势

  • 抽象与实现完全解耦
  • 两个维度独立扩展
  • 符合单一职责原则
  • 运行时可动态切换实现

🌳 组合模式（Composite）

将对象组合成树形结构以表示"部分-整体"的层次结构。Composite 使得用户对单个对象和组合对象的使用具有一致性。 — GoF《设计模式》

结构型树形结构统一处理

📐 原理 & 应用

组合模式允许客户端统一处理单个对象（叶节点）和对象的组合（容器），无需区分。

图5：组合模式 — 树形结构，叶节点与容器统一处理
🐧
Linux VFS — 虚拟文件系统

Linux VFS 是组合模式的典型应用：dentry（目录项）可以是文件（叶）或目录（组合），而 file_operations 接口对二者统一处理，客户端无需区分。

Linux VFS（简化）
/* 统一接口：目录和文件都实现此接口 */
struct file_operations {
    ssize_t (*read)(struct file*, char __user*, size_t, loff_t*);
    ssize_t (*write)(struct file*, const char __user*, size_t, loff_t*);
    int     (*open)(struct inode*, struct file*);
    /* 目录和文件都实现同一套接口 */
};

🎀 装饰器模式（Decorator）

动态地给一个对象添加一些额外的职责。就增加功能来说，Decorator 模式相比生成子类更为灵活。 — GoF《设计模式》

结构型动态扩展包装

📐 原理 & 使用场景

装饰器通过包装对象，在调用被包装对象的方法前后增加额外行为，形成装饰链。

🎯 使用场景

  • 需要在不修改类的情况下动态添加行为
  • 功能扩展需要任意组合
  • Nginx HTTP响应过滤器链
  • Linux I/O层叠加（加密/压缩/缓冲）

✅ 优势

  • 比继承更灵活（运行时组合）
  • 遵循单一职责原则
  • 装饰器可任意叠加
  • 对客户端透明
🌐
Nginx — HTTP响应过滤器链

Nginx 的 HTTP 模块使用装饰器链处理响应：gzip压缩过滤器→内容替换过滤器→header过滤器，每个过滤器包装上一个，依次增强输出。

Nginx 过滤链（ngx_http_filter.c 简化）
/* 每个过滤模块装饰（包装）上一个 */
static ngx_http_output_body_filter_pt ngx_http_next_body_filter;

ngx_int_t ngx_http_gzip_body_filter(ngx_http_request_t *r, ngx_chain_t *in) {
    /* 先执行自己的处理（压缩）*/
    do_gzip(in);
    /* 再调用下一个过滤器（装饰链） */
    return ngx_http_next_body_filter(r, in);
}

🪶 享元模式（Flyweight）

运用共享技术有效地支持大量细粒度的对象。 — GoF《设计模式》

结构型内存优化共享对象

📐 原理 & 使用场景

享元模式通过共享相同或相似状态来减少对象数量，将对象状态分为内部状态（可共享）和外部状态（不可共享）。

🎯 使用场景

  • 需要大量细粒度相似对象
  • 内存资源极度紧张
  • 对象状态大部分可外部化
  • Redis 共享小整数和短字符串
  • Linux 页缓存（Page Cache）

✅ 优势

  • 显著减少内存占用
  • 减少对象创建开销
  • 适合高频重复场景
  • 与工厂模式配合实现对象池
📦
Redis — 共享整数对象（Integer Flyweight）

Redis 预先创建 0~9999 的整数对象（shared.integers[]），当需要这些值时直接返回共享对象，不创建新对象，大幅减少内存分配次数。

Redis object.c — 享元整数共享
/* 预创建共享整数池（享元池） */
void createSharedObjects(void) {
    for (int j = 0; j < OBJ_SHARED_INTEGERS; j++) {
        shared.integers[j] = createObject(OBJ_STRING, ...);
    }
}

/* 获取整数对象时，优先返回共享对象 */
robj* createStringObjectFromLongLong(long long value) {
    if (value >= 0 && value < OBJ_SHARED_INTEGERS) {
        return shared.integers[value]; /* 返回享元 */
    }
    return createObject(...); /* 新建对象 */
}

代理对象与真实对象实现相同接口，客户端通过代理访问真实对象，代理可在访问前后执行额外操作（权限检查、日志、延迟初始化等）。

👁️ 观察者模式（Observer）

定义对象间的一种一对多的依赖关系，当一个对象的状态发生改变时，所有依赖于它的对象都得到通知并被自动更新。 — GoF《设计模式》

行为型发布-订阅事件驱动

📐 原理

Subject（主题）维护一个观察者列表，状态改变时自动通知所有注册的 Observer（观察者）。也称为"发布-订阅"模式。

📡
Redis — 发布/订阅（Pub/Sub）机制

Redis 的 Pub/Sub 是观察者模式的直接实现：发布者（Publisher）发布消息到频道，Redis服务器作为主题（Subject），所有订阅该频道的客户端（Observer）自动收到通知。

♟️ 策略模式（Strategy）

定义一系列的算法，把它们一个个封装起来，并且使它们可互换。本模式使得算法可独立于使用它的客户而变化。 — GoF《设计模式》

行为型算法封装可互换

📐 原理 & 使用场景

Context 持有策略接口的引用，在运行时可动态切换不同的算法实现。通过函数指针在C语言中实现。

💻 C语言实现

C — strategy.c（基于 mohitmishra786.github.io）
typedef struct { int (*execute)(int, int); } Strategy;
typedef struct { Strategy* strategy; } Context;

int add(int a, int b)      { return a + b; }
int multiply(int a, int b)  { return a * b; }

/* 运行时切换策略 */
Context ctx;
Strategy add_strat = { add };
ctx.strategy = &add_strat;
printf("%d\n", ctx.strategy->execute(10, 5)); /* 15 */

Strategy mul_strat = { multiply };
ctx.strategy = &mul_strat;
printf("%d\n", ctx.strategy->execute(10, 5)); /* 50 */

🔍 C开源项目应用

📦

Redis 对同一数据结构（如Hash、Set、List）根据数据量和元素大小动态选择编码策略：小数据用 ziplist（节省内存），大数据用 hashtable/skiplist（高性能）。

}

🔄 状态模式（State）

允许一个对象在其内部状态改变时改变它的行为。对象看起来似乎修改了它的类。 — GoF《设计模式》

行为型状态机行为随状态变化

📐 原理 & 应用

对象持有当前状态对象，行为委托给当前状态处理。状态改变时替换状态对象，而非修改对象本身。

🎯 使用场景

  • 对象行为随状态大量变化
  • 大量条件分支依赖对象状态
  • Redis客户端连接状态机
  • TCP/IP 协议状态机（LISTEN/SYN/ESTABLISHED）
  • 游戏角色AI状态机

✅ 优势

  • 消除大量if-else/switch条件语句
  • 状态相关代码集中管理
  • 新增状态无需修改已有代码
  • 状态切换显式清晰

📝 模板方法模式（Template Method）

定义一个操作中的算法的骨架，而将一些步骤延迟到子类中。TemplateMethod 使得子类可以不改变一个算法的结构即可重定义该算法的某些特定步骤。 — GoF《设计模式》

行为型算法骨架钩子方法

📐 原理 & 应用

父类（在C中为函数指针表）定义算法骨架（流程），具体步骤由子类（具体实现）填充。

🐧
Linux — file_operations 模板

Linux VFS 的 file_operations 是模板方法的典型应用：VFS 定义了 read/write/open/ioctl 等操作的调用框架（模板），具体文件系统（ext4/nfs/proc等）通过函数指针填充具体实现。

Linux VFS — file_operations 模板
/* 模板骨架：VFS定义调用流程 */
struct file_operations {
    ssize_t (*read)(...);  /* 步骤1：读 */
    ssize_t (*write)(...); /* 步骤2：写 */
    int     (*open)(...);  /* 步骤3：打开 */
    int     (*release)(...); /* 步骤4：关闭 */
};

/* ext4 具体实现（填充模板步骤）*/
const struct file_operations ext4_file_operations = {
    .read  = ext4_file_read_iter,
    .write = ext4_file_write_iter,
    .open  = ext4_file_open,
};

📡 中介者模式（Mediator）

用一个中介对象来封装一系列的对象交互。中介者使各对象不需要显式地相互引用，从而使其耦合松散，而且可以独立地改变它们之间的交互。 — GoF《设计模式》

🚶 访问者模式（Visitor）

表示一个作用于某对象结构中的各元素的操作。它使你可以在不改变各元素的类的前提下定义作用于这些元素的新操作。 — GoF《设计模式》

📐 原理 & 应用

访问者将数据结构与操作分离：新增操作只需新增访问者，不修改元素类。适合操作频繁变化而结构稳定的场景。

🎯 使用场景

  • 编译器：AST语法树遍历（代码生成、优化、类型检查）
  • Linux 内核设备树（DT）解析遍历
  • XML/JSON文档遍历处理

✅ 优势

  • 对元素不修改即可增加操作
  • 相关操作集中在访问者中
  • 符合开闭原则
  • 支持对复杂结构做多种处理

行为通信 运行时切换可互换算法？ Strategy Redis编码策略
行为通信 请求封装、支持撤销/排队？ Command Redis AOF
行为通信 请求依次经过多个处理者？ Chain of Resp. Nginx HTTP链
行为通信 对象行为随内部状态大量变化？ State Redis连接状态机
行为通信 固定算法骨架，步骤可定制？ Template Method Linux file_operations
行为通信 统一遍历方式，不暴露内部？ Iterator Redis dictIterator
行为通信 减少对象间直接依赖，集中协调？ Mediator Redis服务器中介
行为通信 不改元素类，增加新操作？ Visitor 编译器AST遍历
行为通信 捕获并恢复对象状态？ Memento Redis RDB快照
行为通信 定义语言文法并解释？ Interpreter Redis RESP解析

⚠️ 选择原则与注意事项

✅
最佳实践
  • 先识别核心问题类型（创建/结构/行为）
  • 明确主要约束（性能/内存/扩展性）
  • 模式可组合使用（如Factory+Singleton）
  • 优先使用简单方案，避免过度设计
⚠️
常见误区
  • 强行套用模式，过度工程化
  • 把 Singleton 用于所有全局变量
  • 混淆 Decorator 和 Proxy（目的不同）
  • 忽略C语言特性（无GC，需手动管理内存）
