引言

libuv 与 C 语言系统编程

为什么不引入 libuv，却还要学习它的设计思想？

libuv 是一个跨平台的 C 语言异步 I/O 支持库，最初为 Node.js 而生。它抽象了 Linux 的 epoll、macOS/BSD 的 kqueue、Solaris 的 event ports 以及 Windows 的 IOCP，提供统一的事件驱动异步 I/O 模型。

然而在许多 C 语言业务系统或嵌入式项目中，直接引入 libuv 成本太高：依赖链复杂、可移植性要求多样、内存占用敏感。但 libuv 的代码库中凝结了大量经过生产检验的设计模式，完全可以在不依赖库本身的前提下借鉴到日常开发中。

核心模式 · 01

RAII — 资源即初始化

libuv 的 Handle 生命周期管理与 C 语言 RAII 实践

📄 官方文档：Handles and requests — Design overview 📄 官方 API：uv_handle_t — Base handle

什么是 libuv 的 RAII 思想

官方文档明确规定了 Handle 的生命周期规则（Guidelines for dealing with handles and requests）：

  • 规则 1：如果 uv_foo_init() 成功初始化了 handle，则必须调用 uv_close()。若初始化失败则无需处理。
  • 规则 2：只有 handle 需要关闭，request 不需要。例如 uv_tcp_t 是 handle，uv_write_t 是 request。
  • 规则 3：handle 的内存只能在 uv_close_cb 内部或其返回之后才能被回收或重用，不能提前释放。
  • 规则 4：request 在完成时或被 uv_cancel() 取消时自动关闭，无需额外清理（除 uv_fs_t 和 uv_getaddrinfo_t 外）。
⚠️ 关键约束
官方文档原文："This MUST be called on each handle before memory is released. Moreover, the memory can only be released in close_cb or after it has returned."
（在释放内存前，每个 handle 都必须调用 uv_close。内存只能在 close_cb 中或其返回后释放。）

图2 · libuv Handle 生命周期状态机（官方设计规范）

C 语言中模拟 RAII 的实践方案

C++ 的 RAII 依赖析构函数，C 语言没有析构函数，但可以通过 init/cleanup 函数对 + 宏封装 模拟：

C · RAII 模拟/* === 借鉴 libuv 的 C 语言 RAII 模式 === */

/* 1. 资源结构体，内嵌状态标志 */
typedef struct {
    int    fd;          /* 文件描述符 */
    char  *buf;        /* 动态分配缓冲区 */
    int    initialized; /* 是否已初始化（对应 uv_foo_init 成功标志）*/
} my_handle_t;

/* 2. init 函数：成功则标记，失败则不标记 */
int my_handle_init(my_handle_t *h, const char *path) {
    h->initialized = 0;
    h->buf = (char*)malloc(4096);
    if (!h->buf) return -1;
    h->fd = open(path, O_RDWR);
    if (h->fd < 0) { free(h->buf); return -1; }
    h->initialized = 1;  /* 标记成功 */
    return 0;
}

/* 3. cleanup 函数：仅在 initialized 时执行 */
void my_handle_cleanup(my_handle_t *h) {
    if (!h->initialized) return;
    close(h->fd);
    free(h->buf);
    h->initialized = 0;
}

/* 4. 使用：通过 goto cleanup 统一出口 */
int process_file(const char *path) {
    my_handle_t h;
    int rc = -1;

    if (my_handle_init(&h, path) != 0) goto done;

    /* ... 使用 h ... */
    rc = 0;

cleanup:
    my_handle_cleanup(&h);  /* 安全：未初始化时是空操作 */
done:
    return rc;
}
✅ 核心借鉴点
libuv 的经验是：init 成功 → 必须 close，内存释放只在 close_cb 后。在 C 语言中翻译为：init 成功 → 必须 cleanup，cleanup 统一放在单一出口（goto cleanup）。这消除了中间 return 路径上的资源泄漏。
核心模式 · 02

Handle / Request 分离模式

长生命周期对象与短生命周期操作的解耦设计

📄 官方文档：Handles and requests — Design overview 📄 官方文档：Handles and Requests — Basics of libuv

官方定义

官方文档原文："Handles represent long-lived objects capable of performing certain operations while active. [...] Requests represent (typically) short-lived operations."

Handle（句柄）代表长生命周期对象，活跃期间能执行特定操作。Request（请求）代表通常是短生命周期的操作，可以作用于 handle（如写请求），也可以独立运行（如 getaddrinfo 不需要 handle，直接在 loop 上运行）。

特征Handle（句柄）Request（请求）
生命周期长期，active 期间持续运行短期，通常一个回调后结束
典型示例uv_tcp_t, uv_timer_t, uv_signal_tuv_write_t, uv_connect_t, uv_fs_t
关闭方式必须调用 uv_close()，异步回调完成或取消后自动关闭
内存释放时机close_cb 中或之后请求完成后（特殊请求需手动清理）
事件循环存活影响active + ref'd handle 使 loop 存活active request 使 loop 存活
📌 官方举例
"For example, an UDP socket is represented by a uv_udp_t, while individual writes to the socket use a uv_udp_send_t structure that is passed to the callback after the write is done."
UDP socket 是 Handle（持续存在），每次写操作是 Request（用完即弃）。

Handle/Request 分离的优势

  • 内存可预测：长期对象和短期对象分开分配，生命周期明确，不会因混用导致 use-after-free
  • 并发安全：每个 Request 携带独立上下文，无需全局锁
  • 可取消：Request 可通过 uv_cancel() 取消，Handle 则通过 uv_close() 关闭
  • 职责单一：Handle 管理连接/资源状态，Request 管理单次操作

图3 · Handle 与 Request 的职责分工（官方文档概念）

在 C 语言业务代码中的应用

C · Handle/Request 分离模式/* 借鉴 libuv 的"长期对象 vs 短期操作"分离 */

/* === 长期对象：连接 handle === */
typedef struct {
    int          fd;
    char         peer_ip[64];
    void        *userdata;
    void       (*on_close)(struct conn_handle *);
} conn_handle_t;

/* === 短期操作：写请求 === */
typedef struct {
    conn_handle_t *handle;   /* 关联的 handle */
    char          *data;     /* 待写数据 */
    size_t         len;
    void         (*on_done)(struct write_req *, int status);
} write_req_t;

/* 发起写操作：创建短期 request，不影响 handle 生命周期 */
int conn_write(conn_handle_t *h, char *data, size_t len,
               void (*cb)(write_req_t*, int)) {
    write_req_t *req = malloc(sizeof(*req));
    if (!req) return -1;
    req->handle  = h;
    req->data    = data;
    req->len     = len;
    req->on_done = cb;
    /* 提交到 I/O 子系统，完成后在 on_done 中 free(req) */
    return io_submit_write(req);
}

/* 写完成回调：释放 request，handle 继续存活 */
void on_write_done(write_req_t *req, int status) {
    if (status < 0) log_error("write failed: %d", status);
    free(req->data);
    free(req);  /* request 完成即释放，handle 不受影响 */
}
核心模式 · 03

事件循环架构

单线程非阻塞 I/O 的阶段化执行模型

📄 官方文档：The I/O loop — Design overview 📄 官方文档：Event loops — Basics of libuv

事件循环的本质

官方文档原文：

📌 官方伪代码
"while there are still events to process:
    e = get the next event
    if there is a callback associated with e:
        call the callback"

libuv 事件循环的核心特性：

  • 单线程：event loop 绑定单一线程，官方原文："it's meant to be tied to a single thread"
  • 非阻塞：所有网络 I/O 在非阻塞 socket 上执行，由平台最佳机制轮询
  • 平台最优：Linux 用 epoll，macOS/BSD 用 kqueue，Solaris 用 event ports，Windows 用 IOCP
  • 非线程安全："The libuv event loop is not thread-safe except where stated otherwise"

在 C 语言项目中的应用

即使不使用 libuv，事件循环的思想也可以自行实现。关键是将循环体拆分为明确的阶段：

C · 简化事件循环框架/* 借鉴 libuv 阶段化事件循环的精简版本 */

typedef struct {
    int  epfd;
    int  stop_flag;
    int  active_count;  /* active handles 计数 */
    int64_t now_ms;
    /* timer_heap, pending_queue, close_queue ... */
} my_loop_t;

void my_loop_run(my_loop_t *loop) {
    while (loop->active_count > 0 && !loop->stop_flag) {
        /* ① 更新当前时间（减少系统调用） */
        loop->now_ms = get_monotonic_ms();

        /* ② 运行到期定时器 */
        run_timers(loop);

        /* ③④ 运行 pending/idle/prepare 回调 */
        run_pending(loop);
        run_idle(loop);
        run_prepare(loop);

        /* ⑦ 计算 poll timeout */
        int timeout = calc_timeout(loop);

        /* ⑧ I/O 轮询（核心阻塞点） */
        io_poll(loop, timeout);   /* epoll_wait / kevent / ... */

        /* ⑨ check 回调 */
        run_check(loop);

        /* ⑩ close 回调 */
        run_closing(loop);
    }
}
⚠️ 重要提醒
官方特别强调："libuv uses a thread pool to make asynchronous file I/O operations possible, but network I/O is always performed in a single thread, each loop's thread."
文件 I/O 在线程池中运行，网络 I/O 永远在 loop 线程中。这是关键设计决策，借鉴时要注意区分。
核心模式 · 04

回调与上下文传递

void* data 字段与 Storing Context 模式

📄 官方文档：Storing context — Basics of libuv

官方上下文传递机制

官方文档原文：

📌 官方说明
"All handles and requests have a void* data member which you can set to the context and cast back in the callback. This is a common pattern used throughout the C library ecosystem. In addition uv_loop_t also has a similar data member."

所有 handle 和 request 都有一个 void* data 成员，用于在调用点和回调之间传递应用特定信息。这是 C 库生态中的通用模式。

为什么需要 void* data

在回调驱动的编程中，回调函数签名是固定的（由框架决定），无法通过函数参数直接传递业务上下文。void* data 是 C 语言中唯一安全的泛型上下文传递方式：

图5 · void* data 字段实现跨时间的上下文传递

实际代码示例

C · 上下文传递模式/* === 业务上下文结构体 === */
typedef struct {
    int          user_id;
    char         session_token[64];
    int          bytes_read;
    void        *write_queue;
} client_ctx_t;

/* 连接建立时，绑定上下文 */
void on_new_connection(uv_stream_t *server, int status) {
    uv_tcp_t *client = malloc(sizeof(uv_tcp_t));
    uv_tcp_init(loop, client);

    client_ctx_t *ctx = malloc(sizeof(client_ctx_t));
    ctx->user_id = allocate_user_id();
    ctx->bytes_read = 0;

    client->data = ctx;  /* ← 绑定到 handle 的 data 字段 */
    uv_accept(server, (uv_stream_t*)client);
    uv_read_start((uv_stream_t*)client, alloc_buffer, on_read);
}

/* 读回调中恢复上下文 */
void on_read(uv_stream_t *stream, ssize_t nread,
             const uv_buf_t *buf) {
    client_ctx_t *ctx = (client_ctx_t *)stream->data;  /* ← 取回 */
    if (nread > 0) {
        ctx->bytes_read += nread;
        process_data(ctx, buf->base, nread);
    } else if (nread < 0) {
        uv_close((uv_handle_t*)stream, on_close);
    }
    free(buf->base);
}

/* close 回调中释放上下文 */
void on_close(uv_handle_t *handle) {
    client_ctx_t *ctx = (client_ctx_t *)handle->data;
    free(ctx);    /* 先释放上下文 */
    free(handle); /* 再释放 handle（在 close_cb 中，符合 RAII 规范）*/
}
✅ 实践建议
在 C 语言中不管是否使用 libuv，凡是需要"回调 + 上下文"的场景（如 epoll 事件处理、定时器回调、线程任务），都应使用 void* data / void* userdata 字段模式，避免全局变量和静态变量导致的并发问题。

线程池与任务队列

阻塞操作异步化的官方设计与 C 语言移植

📄 官方文档：File I/O — Design overview 📄 官方文档：Threads — libuv guide

为什么需要线程池

官方文档原文：

📌 官方说明
"Unlike network I/O, there are no platform-specific file I/O primitives libuv could rely on, so the current approach is to run blocking file I/O operations in a thread pool."

"it is imperative to make sure that no function which runs periodically in the loop thread blocks when performing I/O or is a serious CPU hog, because this means that the loop slows down and events are not being handled at full capacity."

libuv 当前在全局线程池上运行以下 3 类操作：

  • 文件系统操作（File system operations）
  • DNS 函数（getaddrinfo / getnameinfo）
  • 通过 uv_queue_work() 提交的用户自定义代码

uv_queue_work 的工作模型

图6 · uv_queue_work 的线程池工作模型（官方 Threads 文档）

在 C 语言中实现类似机制

C · 简化线程池任务队列/* 借鉴 uv_queue_work 的 work + after 双回调模式 */

typedef struct task_t {
    void  (*work_fn)(struct task_t*);    /* 在工作线程执行（可阻塞）*/
    void  (*after_fn)(struct task_t*, int status); /* 在主线程执行 */
    void  *data;                        /* 用户数据（对应 uv_work_t.data）*/
    int    status;
} task_t;

/* 工作线程入口：执行阻塞操作，完成后通知主线程 */
static void* worker_thread(void *arg) {
    task_t *task = (task_t*)arg;
    task->work_fn(task);           /* 执行阻塞任务 */
    wakeup_main_thread(task);      /* 通知主线程（如写管道/eventfd）*/
    return NULL;
}

/* 主线程 event loop 中处理完成任务 */
void dispatch_completed_tasks(my_loop_t *loop) {
    task_t *task;
    while ((task = completed_queue_pop(loop)) != NULL) {
        if (task->after_fn)
            task->after_fn(task, task->status); /* 在主线程安全调用 */
    }
}

/* 使用示例：类比 uv_queue_work */
void do_file_read(task_t *t) {
    /* 此函数在工作线程运行，可以使用阻塞 read() */
    file_req_t *req = (file_req_t*)t->data;
    req->nread = read(req->fd, req->buf, req->len);
    t->status = (req->nread >= 0) ? 0 : -errno;
}

void after_file_read(task_t *t, int status) {
    /* 此函数在主线程运行，可以安全访问主线程数据结构 */
    file_req_t *req = (file_req_t*)t->data;
    if (status == 0) process_data(req->buf, req->nread);
    free(req);
    free(t);
}
高级技巧 · 06

引用计数与生命周期管理

uv_ref / uv_unref 与事件循环存活控制

📄 官方文档：Event loop reference count — Utilities

Loop 存活条件（官方定义）

官方文档原文："If a loop has active and ref'd handles, active requests or closing handles it's considered to be alive."

每个 handle 在 start 时自动增加 loop 的引用计数，在 stop 时减少。通过 uv_ref() 和 uv_unref() 可以手动控制：

  • uv_ref(handle)：增加引用，使 loop 保持存活（即使该 handle 是唯一活跃的）
  • uv_unref(handle)：减少引用，允许 loop 在该 handle 是唯一活跃时退出
📌 官方用例
"You might have a garbage collector which runs every X seconds [...] you don't want to have to stop them along all clean exit paths or error scenarios. [...] just unref the timer immediately after creation so that if it is the only watcher running then uv_run will still exit."
垃圾回收定时器、心跳定时器等"守护型"句柄，应 unref，使其不阻止程序正常退出。 
C · 引用计数模式（仿 libuv）/* 官方示例：垃圾回收定时器 unref，不阻塞 loop 退出 */

uv_timer_t gc_req;
uv_timer_t main_job_req;

int main() {
    uv_loop_t *loop = uv_default_loop();

    /* 初始化 GC 定时器，每 2 秒触发 */
    uv_timer_init(loop, &gc_req);
    uv_unref((uv_handle_t*) &gc_req);  /* ← unref：不影响 loop 存活 */
    uv_timer_start(&gc_req, gc_callback, 0, 2000);

    /* 主任务：9 秒后完成 */
    uv_timer_init(loop, &main_job_req);
    uv_timer_start(&main_job_req, job_done, 9000, 0);

    /* 9 秒后 main_job_req 结束，gc_req 虽活跃但 unref，loop 正常退出 */
    return uv_run(loop, UV_RUN_DEFAULT);
}

在 C 语言中实现引用计数管理

图7 · 引用计数控制 Loop 存活（官方 uv_ref/uv_unref 机制）

C · 简化引用计数实现typedef struct {
    int  active_count;  /* ref'd + active handles 数量 */
    int  stop_flag;
} my_loop_t;

typedef struct {
    my_loop_t *loop;
    int        active;
    int        ref;     /* 1=计入 loop 存活，0=不计入（对应 unref）*/
} my_handle_t;

void handle_start(my_handle_t *h) {
    h->active = 1;
    if (h->ref) h->loop->active_count++;
}

void handle_stop(my_handle_t *h) {
    if (!h->active) return;
    h->active = 0;
    if (h->ref) h->loop->active_count--;
}

void handle_unref(my_handle_t *h) {
    if (!h->ref) return;
    h->ref = 0;
    if (h->active) h->loop->active_count--;  /* 从计数中移除 */
}
高级技巧 · 07

Baton 数据传递模式

将 request 结构体嵌入业务结构体的一次性释放技巧

📄 官方文档：Passing data to worker thread — Utilities

什么是 Baton 模式

官方文档原文：

📌 官方定义
"A slight variation is to have the uv_work_t itself as the first member of this struct (called a baton). This allows cleaning up the work request and all the data in one free call."

Baton 模式将 uv_work_t（或任意 request 结构体）作为业务结构体的第一个成员，从而 request 的地址与业务结构体地址相同，可以用一次 free() 同时释放两者。

官方示例代码

C · Baton 模式（官方示例）/* 官方文档 Utilities - Passing data to worker thread */

struct ftp_baton {
    uv_work_t  req;        /* ← 必须是第一个成员！*/
    char      *host;
    int        port;
    char      *username;
    char      *password;
};

/* 分配 baton，req.data 指回 baton 自身 */
ftp_baton_t *baton = malloc(sizeof(ftp_baton_t));
baton->req.data = (void*) baton;   /* 自引用 */
baton->host = strdup("my.webhost.com");
baton->port = 21;

uv_queue_work(loop, &baton->req, ftp_session, ftp_cleanup);

/* 工作函数：从 req 恢复 baton */
void ftp_session(uv_work_t *req) {
    ftp_baton_t *baton = (ftp_baton_t*) req->data;
    fprintf(stderr, "Connecting to %s\n", baton->host);
    /* ... 阻塞的 FTP 操作 ... */
}

/* 清理函数：一次 free 搞定所有 */
void ftp_cleanup(uv_work_t *req, int status) {
    ftp_baton_t *baton = (ftp_baton_t*) req->data;
    free(baton->host);
    free(baton->username);
    free(baton);   /* baton == req，一次释放！*/
}

不依赖 libuv 的通用 Baton 模式

C · 通用 Baton 模式/* 通用原则：将"框架请求"作为业务结构体第一个成员 */

typedef struct {
    work_req_t  req;         /* ← 框架请求，必须第一个 */
    /* 业务数据 */
    int         user_id;
    char       *query;
    void       *result;
    int         result_len;
} db_query_baton_t;

void submit_db_query(int user_id, const char *sql) {
    db_query_baton_t *b = calloc(1, sizeof(*b));
    b->user_id = user_id;
    b->query   = strdup(sql);
    b->req.data = b;   /* 自引用 */
    threadpool_submit(&b->req, do_db_query, on_db_result);
}

void do_db_query(work_req_t *req) {              /* 工作线程 */
    db_query_baton_t *b = req->data;
    b->result = db_execute(b->query, &b->result_len);
}

void on_db_result(work_req_t *req, int status) {  /* 主线程 */
    db_query_baton_t *b = req->data;
    send_response(b->user_id, b->result, b->result_len);
    free(b->result);
    free(b->query);
    free(b);           /* 一次 free，释放 req 和所有业务数据 */
}

问题：线程不安全的事件循环

官方文档明确："The libuv event loop is not thread-safe except where stated otherwise." 工作线程不能直接调用任何 libuv API（除了 uv_async_send）。

⚠️ 官方警告
"It is important to realize that since the message send is async, the callback may be invoked immediately after uv_async_send is called in another thread, or it may be invoked after some time. libuv may also combine multiple calls to uv_async_send and invoke your callback only once."
多次 uv_async_send 可能只触发一次回调，不保证一对一。

C 语言中的跨线程通信方案

不使用 libuv 时，可用 eventfd（Linux）或管道实现类似机制：

C · 工作线程唤醒主线程（eventfd 方案）/* 借鉴 uv_async 的设计：工作线程写 eventfd，主线程 epoll 监听 */

typedef struct {
    int    event_fd;       /* eventfd 描述符 */
    void (*callback)(void *); /* 主线程执行的回调 */
    void  *data;
    _Atomic int pending;   /* 原子标志，避免回调合并问题 */
} my_async_t;

/* 初始化：创建 eventfd，注册到 epoll */
int my_async_init(my_loop_t *loop, my_async_t *a,
                   void (*cb)(void*), void *data) {
    a->event_fd = eventfd(0, EFD_NONBLOCK | EFD_CLOEXEC);
    a->callback = cb;
    a->data = data;
    atomic_store(&a->pending, 0);
    return epoll_add(loop->epfd, a->event_fd, a);
}

/* 工作线程调用：非阻塞，仅唤醒 loop（可合并多次调用）*/
void my_async_send(my_async_t *a) {
    atomic_store(&a->pending, 1);
    uint64_t v = 1;
    write(a->event_fd, &v, sizeof(v));  /* 唤醒 epoll_wait */
}

/* 主线程 epoll 事件处理：读取 eventfd，执行回调 */
void my_async_dispatch(my_async_t *a) {
    uint64_t v;
    read(a->event_fd, &v, sizeof(v));   /* 清除 eventfd */
    if (atomic_exchange(&a->pending, 0))
        a->callback(a->data);             /* 主线程安全回调 */
}

在 C 语言中建立统一错误处理风格

C · 统一错误处理约定/* 借鉴 libuv 的错误处理约定 */

/* 1. 定义错误码（类似 UV_E* 常量）*/
#define MY_ENOMEM    -1
#define MY_EINVAL    -2
#define MY_ECONNRESET -3
#define MY_EOF       -4

/* 2. 统一错误描述函数（类似 uv_strerror）*/
const char* my_strerror(int err) {
    switch (err) {
    case MY_ENOMEM:    return "out of memory";
    case MY_EINVAL:    return "invalid argument";
    case MY_ECONNRESET: return "connection reset by peer";
    case MY_EOF:       return "end of file";
    default:           return "unknown error";
    }
}

/* 3. 初始化函数：成功返回 0，失败返回负数 */
int conn_init(conn_t *c, const char *addr) {
    c->fd = socket(AF_INET, SOCK_STREAM, 0);
    if (c->fd < 0) return MY_ENOMEM;
    if (!addr) return MY_EINVAL;
    return 0;   /* 成功 */
}

/* 4. 异步回调：通过 status 参数传递错误 */
void on_read(conn_t *c, ssize_t nread, const char *buf) {
    if (nread == MY_EOF) {
        log_info("Connection closed normally");
        conn_close(c);
        return;
    }
    if (nread < 0) {
        log_error("Read error: %s", my_strerror(nread));
        conn_close(c);
        return;
    }
    process_data(buf, nread);
}

/* 5. 使用宏简化错误检查（类似 libuv 内部使用模式）*/
#define CHECK(r, msg) \
    do { if ((r) < 0) { fprintf(stderr, msg ": %s\n", my_strerror(r)); goto err; } } while(0)

int setup() {
    int r;
    conn_t c;
    r = conn_init(&c, "127.0.0.1");  CHECK(r, "conn_init failed");
    r = conn_bind(&c, 8080);          CHECK(r, "conn_bind failed");
    return 0;
err:
    conn_cleanup(&c);
    return -1;
}
✅ 错误处理最佳实践
  • 同步函数：返回值 0 = 成功，负数 = 具体错误码
  • 异步回调：status/nread 参数 < 0 表示错误，区分 EOF（正常结束）和真正的错误
  • 提供 my_strerror() 将错误码转为可读字符串
  • 错误路径统一通过 goto cleanup 处理，结合 RAII 模式

✅ 核心总结

libuv 最值得借鉴的思想，按优先级排序：
1. RAII（init/cleanup 配对 + goto 统一出口） — 消除内存泄漏，零成本引入
2. void* data 上下文字段 — 避免全局变量，适用所有回调场景
3. 负数错误码规范 — 使错误路径清晰可测试
4. Handle/Request 分离 — 长期资源与短期操作生命周期解耦
5. Baton 一次性释放 — 简化多层结构的内存管理
以上前 3 条可在任何 C 项目中立即采用，后两条在 I/O 密集型服务中价值最大。
