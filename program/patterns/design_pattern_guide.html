程序设计模式
— C语言实践指南

设计模式（Design Pattern）是软件工程领域针对反复出现的设计问题，经过实践检验的可复用解决方案模板。 它们不是可直接执行的代码，而是描述在特定上下文中如何解决问题的通用方案。

📖 导读：本章建立基础认知框架——"什么是模式、为什么需要模式"。 理解这一概念后，后续每种具体模式将在更清晰的坐标系中被理解。

定义与本质

设计模式是对软件设计中普遍存在（反复出现）的各种问题所提出的解决方案。 由 Erich Gamma、Richard Helm、Ralph Johnson 和 John Vlissides（合称 Gang of Four，GoF） 于 1994 年在著作 《Design Patterns: Elements of Reusable Object-Oriented Software》 中首次系统化整理， 共归纳了 23 种经典模式。

设计模式的本质——问题与解决方案之间的映射

为什么需要设计模式？

🗣️ 共同语言

开发者之间无需大量解释，说"用观察者模式"即可传递完整设计意图。

♻️ 避免重复发明

经过数十年实践验证的方案，避免从零设计带来的隐患和返工。

🔧 可维护性

清晰的关注点分离，新增功能无需大幅改动现有代码。

📐 架构思维

将思考层次从"怎么写代码"提升到"如何设计系统"。

C语言实现策略

C 语言没有类、继承、虚函数等 OOP 特性，但通过 结构体（struct）+ 函数指针 + 模块封装， 可以模拟所有设计模式的核心机制，同时保持 C 语言的高性能。

📖 导读：本章是理解所有 C 语言模式实现的"语法桥梁"。 掌握这三种核心技术，后续每个模式的代码将一目了然。

三大核心技术

① struct 模拟对象（Object-like Struct）

将数据字段和操作函数指针封装在同一个结构体中，模拟"类"的概念。

#include <stdio.h>

/* 用 struct 模拟"形状类" */
typedef struct {
    int x, y;
    void (*draw)(void*);   /* 函数指针 = 虚方法 */
} Shape;

void circle_draw(void* self) {
    Shape* s = (Shape*)self;
    printf("Draw circle at (%d,%d)\n", s->x, s->y);
}

② 函数指针模拟多态

通过在结构体中存储函数指针，实现运行时多态（类似 C++ 虚函数表 vtable）。

/* 接口定义 */
typedef struct {
    void (*sort)(int*, size_t);
} ISorter;

/* 具体实现 1 */
void bubble_sort(int* arr, size_t n) { /* ... */ }

/* 绑定：运行时可切换 */
ISorter sorter = { .sort = bubble_sort };
sorter.sort(data, len);   /* 多态调用 */

③ static + 不透明指针实现封装

用 static 关键字限制符号可见性，用不透明指针（opaque pointer）隐藏实现细节， 实现 C 版本的"私有成员"。

/* logger.h —— 只暴露接口 */
typedef struct Logger Logger;
Logger* logger_create(const char* path);
void    logger_write(Logger*, const char* msg);
void    logger_destroy(Logger*);

/* logger.c —— 实现完全隐藏 */
struct Logger {
    FILE* fp;
    int   level;   /* 外部无法访问 */
};

创建型 · 章节 3

单例模式 Singleton

确保某个对象（或结构体实例）在整个程序生命周期中只存在一个，并提供全局访问点。 适用于配置管理器、日志器、连接池等场景。

📖 导读："只有一个"是核心约束。通过 static 局部变量或文件级静态指针， C 语言可以优雅地实现这一约束，且无需加锁（单线程场景）。
单例模式结构

原理

  • 单一实例：通过文件级 static 指针保存唯一实例引用。
  • 延迟初始化：首次调用 get_instance() 时才分配内存。
  • 全局访问：通过公开函数（非变量）提供访问点，维持封装。

C语言实现（含注释）

#include <stdio.h>
#include <stdlib.h>

/* 单例结构体 */
typedef struct {
    int log_level;
    char config_path[256];
} AppConfig;

/* static 限制为文件私有，外部无法直接访问 */
static AppConfig* instance = NULL;

AppConfig* app_config_get() {
    if (instance == NULL) {
        instance = (AppConfig*)malloc(sizeof(AppConfig));
        instance->log_level = 1;
        snprintf(instance->config_path, 256, "/etc/app.conf");
    }
    return instance;
}

void app_config_destroy() {
    free(instance);
    instance = NULL;
}

int main() {
    AppConfig* cfg1 = app_config_get();
    AppConfig* cfg2 = app_config_get();
    /* cfg1 == cfg2，同一指针 */
    printf("同一实例：%s\n", (cfg1 == cfg2) ? "✓" : "✗");
    app_config_destroy();
}

C语言应用场景

全局日志器

程序中唯一的日志文件句柄，所有模块共享同一写入通道。

配置管理器

从配置文件读取一次，之后所有模块通过单例访问，避免重复解析。

GLib 主事件循环

GLib 的 g_main_loop_new() 在应用中通常只有一个实例，体现单例思想。

设备驱动句柄

嵌入式系统中，某个外设（如串口）只应被初始化一次，用单例控制。

创建型 · 章节 4

工厂方法模式 Factory Method

定义一个创建对象的接口，但让具体的创建逻辑由子模块（函数）来决定。 客户端不需要知道实际被创建的是哪种具体类型。

📖 导读：工厂模式是"把 new（malloc）藏起来"。 在 C 里，一个 switch/enum 驱动的工厂函数，返回统一接口的结构体指针，即可实现。
工厂方法模式 — 对象创建流程

C语言实现

#include <stdio.h>
#include <stdlib.h>

/* 统一"接口"：所有形状共享此基础结构 */
typedef struct {
    void (*draw)(void* self);
    void (*destroy)(void* self);
} Shape;

/* 圆形 */
typedef struct { Shape base; float radius; } Circle;
void circle_draw(void* s)   { printf("Circle r=%.1f\n", ((Circle*)s)->radius); }
void circle_destroy(void* s){ free(s); }

/* 矩形 */
typedef struct { Shape base; float w, h; } Rect;
void rect_draw(void* s)   { printf("Rect %.1fx%.1f\n", ((Rect*)s)->w, ((Rect*)s)->h); }
void rect_destroy(void* s){ free(s); }

typedef enum { SHAPE_CIRCLE, SHAPE_RECT } ShapeType;

/* 工厂函数：客户端只与 Shape* 交互 */
Shape* shape_factory(ShapeType type) {
    switch (type) {
        case SHAPE_CIRCLE: {
            Circle* c = malloc(sizeof(Circle));
            c->base.draw = circle_draw; c->base.destroy = circle_destroy;
            c->radius = 5.0f;
            return (Shape*)c;
        }
        case SHAPE_RECT: {
            Rect* r = malloc(sizeof(Rect));
            r->base.draw = rect_draw; r->base.destroy = rect_destroy;
            r->w = 4.0f; r->h = 3.0f;
            return (Shape*)r;
        }
        default: return NULL;
    }
}

int main() {
    Shape* s1 = shape_factory(SHAPE_CIRCLE);
    Shape* s2 = shape_factory(SHAPE_RECT);
    s1->draw(s1);   /* Circle r=5.0 */
    s2->draw(s2);   /* Rect 4.0x3.0 */
    s1->destroy(s1); s2->destroy(s2);
}
已知实际应用：libcurl 的 curl_easy_init() 是工厂模式的经典案例—— 客户端只需调用工厂函数，不需要了解内部如何分配和初始化 CURL 句柄。 参见：libcurl 官方文档
📚 参考来源：DEV.to — Design Patterns for C（Factory 章节）
创建型 · 章节 5

抽象工厂模式 Abstract Factory

提供一个创建一系列相关或相互依赖对象的接口，而不指定它们的具体类。 是"工厂的工厂"——管理产品族。

📖 导读：当系统需要创建多个相关对象（产品族），且这些对象必须保持一致（如"深色主题"的所有控件）， 抽象工厂比普通工厂更合适。C 语言用结构体中嵌套函数指针组成"工厂接口"。
抽象工厂 — 产品族概念

C语言实现

#include <stdio.h>

/* 抽象产品：按钮 */
typedef struct { void (*render)(); } Button;

/* 抽象工厂接口 */
typedef struct {
    Button* (*create_button)();
} UIFactory;

/* 浅色产品 */
void light_btn_render() { printf("[浅色按钮]\n"); }
Button* light_create_button() {
    Button* b = malloc(sizeof(Button));
    b->render = light_btn_render; return b;
}

/* 深色产品 */
void dark_btn_render() { printf("[深色按钮]\n"); }
Button* dark_create_button() {
    Button* b = malloc(sizeof(Button));
    b->render = dark_btn_render; return b;
}

/* 工厂实例 */
UIFactory light_factory = { .create_button = light_create_button };
UIFactory dark_factory  = { .create_button = dark_create_button  };

void render_ui(UIFactory* factory) {
    Button* btn = factory->create_button();
    btn->render();
    free(btn);
}

int main() {
    render_ui(&light_factory);  /* [浅色按钮] */
    render_ui(&dark_factory);   /* [深色按钮] */
}

应用场景

  • 跨平台 UI 工具包（GTK / Windows API 适配层）
  • 嵌入式系统中的硬件抽象层（HAL）——同一代码在不同 MCU 上运行
  • 数据库驱动工厂（SQLite / PostgreSQL 切换，接口一致）
📚 参考来源：DigitalOcean — Abstract Factory Pattern
创建型 · 章节 6

建造者模式 Builder

将复杂对象的构建过程与其最终表示分离，使同一构建过程可以创建不同的表示。 适合参数众多的复杂结构初始化。

📖 导读：建造者模式解决的是"初始化地狱"——当一个结构体有十几个字段， 其中许多是可选的时，链式构建函数比超长参数列表清晰得多。
建造者模式 — 分步构建流程

C语言实现（链式函数风格）

#include <stdio.h>
#include <string.h>
#include <stdlib.h>

/* 最终产品 */
typedef struct {
    char host[64];
    int  port;
    int  timeout_ms;
    int  use_ssl;
    int  max_connections;
} HttpConfig;

/* Builder 结构体 */
typedef struct HttpConfigBuilder {
    HttpConfig cfg;
    struct HttpConfigBuilder* (*set_host)    (struct HttpConfigBuilder*, const char*);
    struct HttpConfigBuilder* (*set_port)    (struct HttpConfigBuilder*, int);
    struct HttpConfigBuilder* (*set_timeout) (struct HttpConfigBuilder*, int);
    struct HttpConfigBuilder* (*set_ssl)     (struct HttpConfigBuilder*, int);
    HttpConfig*               (*build)       (struct HttpConfigBuilder*);
} HttpConfigBuilder;

static HttpConfigBuilder* impl_set_host(HttpConfigBuilder* b, const char* h) {
    strncpy(b->cfg.host, h, 63); return b;
}
static HttpConfigBuilder* impl_set_port(HttpConfigBuilder* b, int p) {
    b->cfg.port = p; return b;
}
static HttpConfigBuilder* impl_set_timeout(HttpConfigBuilder* b, int t) {
    b->cfg.timeout_ms = t; return b;
}
static HttpConfigBuilder* impl_set_ssl(HttpConfigBuilder* b, int s) {
    b->cfg.use_ssl = s; return b;
}
static HttpConfig* impl_build(HttpConfigBuilder* b) {
    HttpConfig* cfg = malloc(sizeof(HttpConfig));
    *cfg = b->cfg; return cfg;
}

HttpConfigBuilder* http_config_builder_new() {
    HttpConfigBuilder* b = malloc(sizeof(HttpConfigBuilder));
    b->cfg = (HttpConfig){ .port=80, .timeout_ms=3000, .max_connections=10 };
    b->set_host = impl_set_host; b->set_port = impl_set_port;
    b->set_timeout = impl_set_timeout; b->set_ssl = impl_set_ssl;
    b->build = impl_build; return b;
}

int main() {
    HttpConfigBuilder* b = http_config_builder_new();
    HttpConfig* cfg = b->set_host(b, "api.example.com")
                       ->set_port(b, 443)
                       ->set_ssl(b, 1)
                       ->set_timeout(b, 5000)
                       ->build(b);
    printf("Host: %s, Port: %d, SSL: %d\n", cfg->host, cfg->port, cfg->use_ssl);
    free(cfg); free(b);
}

创建型 · 章节 7

原型模式 Prototype

通过克隆（复制）现有对象来创建新对象，而不是从头实例化。 当对象创建代价昂贵（如大量初始化、I/O）时，克隆比重新创建效率更高。

📖 导读：原型模式在 C 中非常自然——memcpy 或自定义 clone 函数就是其核心。 关键是区分浅拷贝与深拷贝，指针成员需特殊处理。

C语言实现

#include <stdio.h>
#include <stdlib.h>
#include <string.h>

typedef struct {
    char* name;
    int   hp;
    float speed;
    /* clone 函数指针 */
    void* (*clone)(const void* self);
} Enemy;

void* enemy_clone(const void* self) {
    const Enemy* src = (Enemy*)self;
    Enemy* copy = malloc(sizeof(Enemy));
    *copy = *src;
    copy->name = strdup(src->name);  /* 深拷贝字符串 */
    return copy;
}

Enemy* enemy_new(const char* name, int hp, float spd) {
    Enemy* e = malloc(sizeof(Enemy));
    e->name = strdup(name); e->hp = hp; e->speed = spd;
    e->clone = enemy_clone;
    return e;
}

int main() {
    Enemy* proto = enemy_new("Goblin", 100, 2.5f);
    /* 批量克隆，无需重复初始化 */
    Enemy* e1 = proto->clone(proto);
    Enemy* e2 = proto->clone(proto);
    e1->hp = 80;  /* 独立修改，互不影响 */
    printf("%s HP=%d / %s HP=%d\n", e1->name, e1->hp, e2->name, e2->hp);
    free(proto->name); free(proto);
    free(e1->name); free(e1); free(e2->name); free(e2);
}

应用场景

  • 游戏中批量生成相似敌人或道具实体
  • 报文模板克隆（网络协议栈中预先构造的数据包头部）
  • 配置对象快照（保存当前配置后允许修改，失败时还原）
📚 参考来源：DigitalOcean — Prototype Pattern
结构型 · 章节 8

适配器模式 Adapter

将一个接口转换成客户端期望的另一个接口。使原本由于接口不兼容而无法协作的类/模块可以一起工作。 又称"包装器（Wrapper）"。

📖 导读：适配器是"翻译官"——它不改变现有代码，只是在外部包一层转换逻辑。 在 C 中，适配函数（包装函数）是最常见的实现方式。
适配器模式结构图

C语言实现

#include <stdio.h>

/* 旧接口（Adaptee）：只接受整数毫秒 */
void legacy_delay_ms(int ms) {
    printf("Legacy: delay %d ms\n", ms);
}

/* 新接口期望：接受秒（float） */
typedef struct {
    void (*delay_sec)(float sec);
} IDelay;

/* 适配器实现：转换单位 */
void adapter_delay_sec(float sec) {
    legacy_delay_ms((int)(sec * 1000));
}

IDelay delay_adapter = { .delay_sec = adapter_delay_sec };

int main() {
    /* 客户端使用新接口（秒），内部调用旧接口（毫秒） */
    delay_adapter.delay_sec(1.5f);  /* → Legacy: delay 1500 ms */
}
已知实际应用：GTK+ 的 config_data_to_combobox() 是适配器模式—— 将应用数据格式适配为 GtkComboBoxText 控件接口。 参见：GTK 官方文档
📚 参考来源：DEV.to — Design Patterns for C（Adapter 章节）
结构型 · 章节 9

外观模式 Facade

为复杂子系统提供一个简化的高层接口，隐藏子系统的复杂性。 客户端通过外观与子系统交互，而非直接操作各个组件。

📖 导读：外观是"简化入口"。想象一个家庭影院：你按一个"开始观影"按钮， 背后自动打开电视、DVD 机、音响——这就是外观模式。C 里用一组高层函数包装多个子系统调用即可。
外观模式 — 简化接口层

C语言实现（家庭影院）

#include <stdio.h>
#include <stdlib.h>
#include <string.h>

typedef struct { int is_on; } TV;
typedef struct { int is_on; char movie[64]; } DVDPlayer;
typedef struct { int is_on; } SoundSystem;

void tv_on(TV* tv)            { tv->is_on=1; printf("TV ON\n"); }
void dvd_on(DVDPlayer* d)     { d->is_on=1;  printf("DVD ON\n"); }
void sound_on(SoundSystem* s) { s->is_on=1;  printf("Sound ON\n"); }
void dvd_play(DVDPlayer* d, const char* m) {
    strncpy(d->movie, m, 63);
    printf("Playing: %s\n", d->movie);
}

/* ===== 外观 ===== */
typedef struct { TV tv; DVDPlayer dvd; SoundSystem sound; } HomeTheater;

void theater_watch_movie(HomeTheater* h, const char* movie) {
    tv_on(&h->tv);
    dvd_on(&h->dvd);
    sound_on(&h->sound);
    dvd_play(&h->dvd, movie);
}

int main() {
    HomeTheater ht = {0};
    theater_watch_movie(&ht, "The Matrix");
}
已知实际应用：GStreamer 的 gst_parse_launch("playbin ...") 是经典外观—— 一行代码隐藏了复杂的管道构建过程。 参见：GStreamer 文档
📚 参考来源：DEV.to — Design Patterns for C（Facade 章节）
结构型 · 章节 10

代理模式 Proxy

为另一个对象提供代理或占位符，以控制对它的访问。 代理可在转发请求前后添加额外逻辑（访问控制、懒加载、日志、缓存）。

📖 导读：代理是"中间人"——客户端以为在操作真实对象，实际是通过代理间接访问。 STM32 HAL 库就是硬件寄存器的代理。

C语言实现（带懒加载和日志的图片代理）

#include <stdio.h>
#include <stdlib.h>
#include <string.h>

typedef struct { void (*display)(void*); } IImage;

/* 真实对象 */
typedef struct { IImage base; char filename[64]; } RealImage;
void real_display(void* self) {
    printf("[Display] %s\n", ((RealImage*)self)->filename);
}

/* 代理对象：懒加载 + 日志 */
typedef struct {
    IImage base;
    RealImage* real;
    char filename[64];
} ProxyImage;

void proxy_display(void* self) {
    ProxyImage* p = (ProxyImage*)self;
    printf("[Proxy] 请求显示: %s\n", p->filename);
    if (!p->real) {                    /* 懒加载 */
        p->real = malloc(sizeof(RealImage));
        p->real->base.display = real_display;
        strncpy(p->real->filename, p->filename, 63);
        printf("[Proxy] 首次加载图片文件...\n");
    }
    p->real->base.display(p->real);
}

int main() {
    ProxyImage proxy = { .base={proxy_display}, .real=NULL };
    strncpy(proxy.filename, "photo.jpg", 63);
    proxy.base.display(&proxy);  /* 第一次：懒加载 */
    proxy.base.display(&proxy);  /* 第二次：复用 */
    free(proxy.real);
}
已知实际应用：STM32 HAL（如 HAL_GPIO_TogglePin()）是硬件寄存器的代理， 隐藏了直接读写 MMIO 地址的复杂性。 参见：STM32 HAL 文档

桥接模式 Bridge

将抽象与实现分离，使两者可以独立变化。 避免因继承导致的类爆炸问题。

📖 导读：桥接是"两条轨道"——抽象（功能逻辑）和实现（底层机制）各自演进， 通过指针桥接在一起。嵌入式 HAL 设计是最典型的场景。

C语言实现（跨平台渲染）

#include <stdio.h>

/* 实现层接口 */
typedef struct {
    void (*draw_circle)(int x, int y, int r);
} IRenderer;

/* 具体实现 A：OpenGL */
void opengl_draw_circle(int x, int y, int r) {
    printf("[OpenGL] Circle at (%d,%d) r=%d\n", x, y, r);
}

/* 具体实现 B：DirectX */
void directx_draw_circle(int x, int y, int r) {
    printf("[DirectX] Circle at (%d,%d) r=%d\n", x, y, r);
}

/* 抽象层：Shape 持有渲染器指针 */
typedef struct {
    IRenderer* renderer;  /* 桥 */
    int x, y;
} Shape;

typedef struct { Shape base; int radius; } Circle;

void circle_draw(Circle* c) {
    c->base.renderer->draw_circle(c->base.x, c->base.y, c->radius);
}

int main() {
    IRenderer ogl = { opengl_draw_circle };
    IRenderer d3d = { directx_draw_circle };
    Circle c1 = { .base={&ogl, 10, 20}, .radius=5 };
    Circle c2 = { .base={&d3d, 30, 40}, .radius=8 };
    circle_draw(&c1);   /* OpenGL */
    circle_draw(&c2);   /* DirectX */
}

应用场景

  • 嵌入式 HAL：同一驱动代码跑在 STM32F4 / STM32H7 两款 MCU 上
  • 数据库抽象层（ODBC / 自定义 DAL）
  • 日志系统：日志级别（抽象）× 输出目标（实现）独立组合

装饰者模式 Decorator

动态地为对象添加额外职责，提供比继承更灵活的功能扩展方式。 装饰者与被装饰对象实现相同接口，对客户端透明。

📖 导读：装饰者是"层层包裹"——每层装饰器加一点功能，又转发到下层。 C 语言中用相同接口的结构体嵌套实现，非常自然地表达"在前后加料"的语义。

C语言实现（日志装饰器）

#include <stdio.h>
#include <stdlib.h>
#include <string.h>

typedef struct IStream {
    int (*write)(struct IStream*, const char*, size_t);
} IStream;

/* 基础文件流 */
typedef struct { IStream base; FILE* fp; } FileStream;
int file_write(IStream* s, const char* buf, size_t n) {
    return fwrite(buf, 1, n, ((FileStream*)s)->fp);
}

/* 装饰器：加压缩标记（模拟） */
typedef struct { IStream base; IStream* inner; } CompressStream;
int compress_write(IStream* s, const char* buf, size_t n) {
    printf("[Compress] %zu bytes\n", n);
    return ((CompressStream*)s)->inner->write(((CompressStream*)s)->inner, buf, n);
}

/* 装饰器：加加密标记（模拟） */
typedef struct { IStream base; IStream* inner; } EncryptStream;
int encrypt_write(IStream* s, const char* buf, size_t n) {
    printf("[Encrypt] %zu bytes\n", n);
    return ((EncryptStream*)s)->inner->write(((EncryptStream*)s)->inner, buf, n);
}

int main() {
    FileStream fs     = { .base={file_write}, .fp=stdout };
    CompressStream cs = { .base={compress_write}, .inner=(IStream*)&fs };
    EncryptStream  es = { .base={encrypt_write},  .inner=(IStream*)&cs };

    const char* data = "Hello";
    /* 调用链：Encrypt → Compress → File */
    es.base.write((IStream*)&es, data, strlen(data));
}

应用场景

  • I/O 流管道：BufferedStream → GZipStream → FileStream
  • 中间件链：HTTP 请求的认证、限流、日志装饰
  • 嵌入式：CRC 校验装饰在 UART 发送流上

组合模式 Composite

将对象组织成树形结构，表示"部分—整体"层次关系， 使客户端能以统一方式处理单个对象和对象组合。

📖 导读：组合模式让"叶子"和"容器"长相一致。文件系统（文件 vs 文件夹）、 UI 控件树（按钮 vs 面板）都是典型场景。C 用递归结构体 + 统一函数指针接口实现。

C语言实现（文件系统树）

#include <stdio.h>
#include <stdlib.h>

typedef struct FSNode {
    char name[64];
    int  is_dir;
    void (*print)(struct FSNode*, int depth);
    struct FSNode** children;
    int  child_count;
} FSNode;

void file_print(FSNode* n, int d) {
    for(int i=0;i<d;i++) printf("  ");
    printf("📄 %s\n", n->name);
}

void dir_print(FSNode* n, int d) {
    for(int i=0;i<d;i++) printf("  ");
    printf("📁 %s/\n", n->name);
    for(int i=0;i<n->child_count;i++)
        n->children[i]->print(n->children[i], d+1);
}

FSNode* make_file(const char* name) {
    FSNode* n = calloc(1, sizeof(FSNode));
    snprintf(n->name, 64, "%s", name);
    n->print = file_print; return n;
}
FSNode* make_dir(const char* name) {
    FSNode* n = calloc(1, sizeof(FSNode));
    snprintf(n->name, 64, "%s", name);
    n->is_dir = 1; n->print = dir_print;
    n->children = malloc(8*sizeof(FSNode*)); return n;
}
void dir_add(FSNode* d, FSNode* child) {
    d->children[d->child_count++] = child;
}

int main() {
    FSNode* root = make_dir("project");
    FSNode* src  = make_dir("src");
    dir_add(src,  make_file("main.c"));
    dir_add(src,  make_file("util.c"));
    dir_add(root, src);
    dir_add(root, make_file("README.md"));
    root->print(root, 0);
}

享元模式 Flyweight

通过共享尽可能多的相同数据，来支持大量细粒度对象，最小化内存使用。 将对象状态分为"内在状态（共享）"和"外在状态（独有）"。

📖 导读：享元是"共用模板"——想象文字编辑器中每个字符都是独立对象会有多少内存浪费。 享元让所有'A'共用同一个字形数据，只有位置不同。C 里用全局字典（数组或哈希表）缓存共享部分。

C语言实现（粒子系统）

#include <stdio.h>
#include <string.h>

/* 内在状态（共享）：重量级数据 */
typedef struct {
    char texture[32];  /* 大纹理，共享 */
    char color[16];
} ParticleType;

/* 享元池 */
static ParticleType pool[8];
static int pool_size = 0;

ParticleType* get_type(const char* tex, const char* col) {
    for(int i=0;i<pool_size;i++)
        if(strcmp(pool[i].texture, tex)==0) return &pool[i];
    strncpy(pool[pool_size].texture, tex, 31);
    strncpy(pool[pool_size].color,   col, 15);
    printf("[Flyweight] 创建类型: %s\n", tex);
    return &pool[pool_size++];
}

/* 外在状态（独有）：轻量 */
typedef struct {
    ParticleType* type;  /* 引用共享数据 */
    float x, y, vx, vy; /* 独有坐标和速度 */
} Particle;

int main() {
    Particle particles[1000];
    /* 1000 个粒子，只有 2 种类型对象 */
    for(int i=0;i<1000;i++) {
        particles[i].type = (i%2) ?
            get_type("fire.png","orange") :
            get_type("smoke.png","gray");
        particles[i].x = i; particles[i].y = i*0.5f;
    }
    printf("池中类型数: %d（vs 1000 对象）\n", pool_size);
}

观察者模式 Observer

定义对象间的一对多依赖关系：当主题（Subject）状态改变时， 所有依赖它的观察者（Observer）都会自动收到通知并更新。又称"发布—订阅"模式。

📖 导读：观察者让主题与观察者完全解耦——主题不知道谁在监听，只管通知。 C 语言用函数指针数组存储观察者回调，实现事件驱动系统。
观察者模式 — 通知流程

C语言实现（天气传感器）

#include <stdio.h>
#include <string.h>
#define MAX_OBSERVERS 8

typedef struct WeatherSensor WeatherSensor;

/* 观察者回调类型 */
typedef void (*Observer)(WeatherSensor*);

/* 主题 */
struct WeatherSensor {
    double    temperature;
    Observer  observers[MAX_OBSERVERS];
    int       count;
};

void ws_subscribe(WeatherSensor* ws, Observer obs) {
    if(ws->count < MAX_OBSERVERS)
        ws->observers[ws->count++] = obs;
}
void ws_notify(WeatherSensor* ws) {
    for(int i=0;i<ws->count;i++) ws->observers[i](ws);
}
void ws_set_temp(WeatherSensor* ws, double t) {
    ws->temperature = t;
    ws_notify(ws);
}

/* 具体观察者 */
void thermostat_update(WeatherSensor* ws) {
    printf("[恒温器] 温度: %.1f°C\n", ws->temperature);
}
void alarm_update(WeatherSensor* ws) {
    if(ws->temperature > 35.0)
        printf("[警报] 过热预警！%.1f°C\n", ws->temperature);
}

int main() {
    WeatherSensor ws = {0};
    ws_subscribe(&ws, thermostat_update);
    ws_subscribe(&ws, alarm_update);
    ws_set_temp(&ws, 22.5);
    ws_set_temp(&ws, 37.0);
}
已知实际应用：GLib 的 signal/slot 机制（g_signal_connect()） 是 Observer 模式的工业级实现。参见：GLib 文档

策略模式 Strategy

定义一系列算法，将每个算法封装起来，并使它们可以相互替换。 策略模式让算法可以独立于使用它的客户端而变化。

📖 导读：策略是"可换的算法插槽"。C 语言天然支持函数指针， 直接把函数指针当作策略传入上下文即可——这是最纯粹的策略模式实现。

C语言实现（排序策略）

#include <stdio.h>
#include <stdlib.h>

/* 策略接口：函数指针类型 */
typedef void (*SortStrategy)(int*, size_t);

/* 策略 A：冒泡排序 */
void bubble_sort(int* a, size_t n) {
    for(size_t i=0;i<n-1;i++)
        for(size_t j=0;j<n-i-1;j++)
            if(a[j]>a[j+1]){ int t=a[j]; a[j]=a[j+1]; a[j+1]=t; }
    printf("[BubbleSort] done\n");
}

/* 策略 B：标准库 qsort */
int cmp_int(const void* a, const void* b) { return *(int*)a - *(int*)b; }
void quick_sort(int* a, size_t n) {
    qsort(a, n, sizeof(int), cmp_int);
    printf("[QuickSort] done\n");
}

/* 上下文：不关心具体策略 */
void do_sort(int* arr, size_t n, SortStrategy strategy) {
    strategy(arr, n);
}

int main() {
    int data[] = {5,2,8,1,9};
    size_t n = 5;

    /* 运行时切换策略 */
    SortStrategy s = (n < 100) ? bubble_sort : quick_sort;
    do_sort(data, n, s);
}
已知实际应用：zlib 的 compress2(..., Z_BEST_SPEED) vs compress2(..., Z_BEST_COMPRESSION) 是策略模式—— 同一接口，不同压缩算法策略。参见：zlib 官方文档
📚 参考来源：DEV.to — Design Patterns for C（Strategy 章节）

状态模式 State

允许对象在内部状态改变时改变其行为，看起来如同改变了类。 将状态相关行为封装到独立的状态结构中，用状态机替代庞大的 if-else 链。

📖 导读：状态模式将"状态 → 行为"的映射变成"状态对象 → 行为实现"。 嵌入式固件中的设备状态机、游戏中的玩家生命状态，都是绝佳场景。
状态模式 — 状态转换图（玩家示例）

C语言实现（Hunt the Wumpus 游戏状态机）

#include <stdio.h>
#include <stdlib.h>

typedef struct Player Player;

typedef struct {
    void (*move)(Player*, int room);
} PlayerState;

struct Player {
    PlayerState* state;
    int          room;
};

extern PlayerState alive_state, dead_state, won_state;

void alive_move(Player* p, int room) {
    p->room = room;
    printf("移动到房间 %d\n", room);
    if(room == 2) { printf("遇到了 Wumpus！死亡\n"); p->state = &dead_state; }
    else if(room == 4) { printf("找到黄金！胜利！\n"); p->state = &won_state; }
}
void dead_move(Player* p, int room) {
    (void)p; (void)room;
    printf("已死亡，无法移动\n");
}
void won_move(Player* p, int room) {
    (void)p; (void)room;
    printf("已获胜，无需移动\n");
}

PlayerState alive_state = { alive_move };
PlayerState dead_state  = { dead_move };
PlayerState won_state   = { won_move };

int main() {
    Player p = { .state=&alive_state, .room=0 };
    p.state->move(&p, 1);
    p.state->move(&p, 2);  /* → 死亡 */
    p.state->move(&p, 3);  /* → 已死亡 */
}
已知实际应用：GStreamer 用状态模式管理媒体管道状态： NULL → READY → PAUSED → PLAYING，每个状态对应不同行为。 参见：GStreamer 文档

命令模式 Command

将请求封装成对象，从而允许以参数形式传递请求、将请求排队、记录请求日志， 以及支持可撤销操作。

📖 导读：命令模式将"做什么"与"何时做/谁来做"解耦。 C 语言中的命令就是一个结构体，包含函数指针（execute/undo）和必要的参数。 宏队列、事务系统、撤销/重做功能都是典型应用。

C语言实现（带撤销的文本编辑器命令）

#include <stdio.h>
#include <stdlib.h>
#include <string.h>

/* 命令接口 */
typedef struct Command {
    void (*execute)(struct Command*);
    void (*undo)   (struct Command*);
} Command;

/* 接受者：编辑器 */
char doc[512] = "";

/* 具体命令：追加文字 */
typedef struct {
    Command base;
    char    text[64];
    size_t  old_len;
} AppendCommand;

void append_execute(Command* cmd) {
    AppendCommand* c = (AppendCommand*)cmd;
    c->old_len = strlen(doc);
    strncat(doc, c->text, sizeof(doc)-strlen(doc)-1);
    printf("执行追加: \"%s\" → \"%s\"\n", c->text, doc);
}
void append_undo(Command* cmd) {
    AppendCommand* c = (AppendCommand*)cmd;
    doc[c->old_len] = '\0';
    printf("撤销追加 → \"%s\"\n", doc);
}

AppendCommand* make_append(const char* text) {
    AppendCommand* c = malloc(sizeof(AppendCommand));
    c->base.execute = append_execute;
    c->base.undo    = append_undo;
    strncpy(c->text, text, 63); return c;
}

/* 命令队列（历史） */
Command* history[16]; int hist_top = 0;
void exec(Command* c) { c->execute(c); history[hist_top++] = c; }
void undo_last()       { if(hist_top) history[--hist_top]->undo(history[hist_top]); }

int main() {
    AppendCommand* c1 = make_append("Hello");
    AppendCommand* c2 = make_append(" World");
    exec((Command*)c1);
    exec((Command*)c2);
    undo_last();   /* 撤销 "World" */
    free(c1); free(c2);
}

应用场景

  • 嵌入式：RTOS 消息队列中的命令对象（事件驱动固件）
  • 宏录制系统：记录用户操作序列，支持回放
  • 事务型数据库操作的提交/回滚
📚 参考来源：DigitalOcean — Command Pattern

责任链模式 Chain of Responsibility

将请求沿着处理链传递，每个处理者决定是否处理该请求， 或将其传递给链中的下一个处理者。实现请求发送者与处理者的解耦。

📖 导读：责任链是"传球"——处理者形成链条，请求沿链流动直到被处理。 Linux 内核的中断处理、HTTP 中间件管道都是经典案例。 C 里用链表或函数指针数组实现处理链。

C语言实现（日志级别过滤链）

#include <stdio.h>
#include <stdlib.h>

typedef enum { LOG_DEBUG=0, LOG_INFO, LOG_WARN, LOG_ERROR } LogLevel;

typedef struct Handler {
    LogLevel            min_level;
    struct Handler*     next;
    void (*handle)(struct Handler*, LogLevel, const char*);
} Handler;

void base_handle(Handler* h, LogLevel lvl, const char* msg) {
    if(lvl >= h->min_level)
        printf("[%s] %s\n",
               lvl==LOG_DEBUG?"DBG":lvl==LOG_INFO?"INF":lvl==LOG_WARN?"WRN":"ERR",
               msg);
    else if(h->next)
        h->next->handle(h->next, lvl, msg);
}

int main() {
    Handler err_h  = { LOG_ERROR, NULL,   base_handle };
    Handler warn_h = { LOG_WARN,  &err_h, base_handle };
    Handler info_h = { LOG_INFO,  &warn_h,base_handle };
    Handler dbg_h  = { LOG_DEBUG, &info_h,base_handle };

    dbg_h.handle(&dbg_h, LOG_DEBUG, "调试信息");
    dbg_h.handle(&dbg_h, LOG_WARN,  "磁盘空间不足");
    dbg_h.handle(&dbg_h, LOG_ERROR, "数据库连接失败");
}

迭代器模式 Iterator

提供顺序访问聚合对象中各元素的方法，而无需暴露其内部结构。 将遍历逻辑与数据集合分离。

📖 导读：迭代器让"如何访问"与"存储什么"解耦。 C 语言中通过在结构体中嵌入 next()、has_next() 函数指针， 可以为任何自定义集合提供统一遍历接口。

C语言实现（链表迭代器）

#include <stdio.h>
#include <stdlib.h>

/* 链表节点 */
typedef struct Node { int val; struct Node* next; } Node;

/* 迭代器 */
typedef struct {
    Node* current;
    int  (*has_next)(struct Iterator*);
    int  (*next_val)(struct Iterator*);
} Iterator;
typedef struct Iterator Iterator;

int it_has_next(Iterator* it) { return it->current != NULL; }
int it_next_val(Iterator* it) {
    int v = it->current->val;
    it->current = it->current->next;
    return v;
}

Iterator list_iterator(Node* head) {
    return (Iterator){ .current=head, .has_next=it_has_next, .next_val=it_next_val };
}

int main() {
    /* 构建链表 1→2→3 */
    Node n3 = {3, NULL};
    Node n2 = {2, &n3};
    Node n1 = {1, &n2};

    Iterator it = list_iterator(&n1);
    while(it.has_next(&it))
        printf("%d ", it.next_val(&it));  /* 1 2 3 */
    printf("\n");
}

模板方法模式 Template Method

在操作中定义算法骨架，将某些步骤的实现延迟到子实现中， 子实现可以重定义某些步骤，但不改变算法整体结构。

📖 导读：模板方法是"固定流程，可变步骤"。C 语言中， 模板函数包含固定的执行顺序，通过函数指针回调注入可变步骤， 非常适合数据处理管道（读取→校验→处理→存储）。

C语言实现（数据处理管道）

#include <stdio.h>

/* 算法步骤接口 */
typedef struct {
    void* (*read_data)  ();
    int   (*validate)   (void*);
    void* (*process)    (void*);
    void  (*save_result)(void*);
} DataPipeline;

/* 模板方法：固定骨架 */
void run_pipeline(const DataPipeline* dp) {
    void* raw = dp->read_data();
    if(!dp->validate(raw)) { printf("验证失败\n"); return; }
    void* result = dp->process(raw);
    dp->save_result(result);
}

/* 具体实现 A：CSV 数据管道 */
void* csv_read()       { printf("读取 CSV\n"); return (void*)1; }
int   csv_validate(void* d) { printf("校验 CSV\n"); return d!=NULL; }
void* csv_process(void* d) { printf("处理 CSV\n"); return d; }
void  csv_save(void* d)    { printf("存储结果\n"); }

int main() {
    DataPipeline csv_pipeline = {
        .read_data=csv_read, .validate=csv_validate,
        .process=csv_process, .save_result=csv_save
    };
    run_pipeline(&csv_pipeline);
}

中介者模式 Mediator

定义一个封装一组对象交互方式的对象（中介者）， 通过中介者协调对象间通信，降低耦合度，避免对象间直接相互引用。

📖 导读：中介者是"控制中心"——原本 N 个对象两两通信（O(N²)条连线）， 改为所有对象只和中介者通信（N 条连线）。适用于 UI 组件协调、总线系统等。

C语言实现（聊天室中介者）

#include <stdio.h>
#include <string.h>
#define MAX_USERS 8

typedef struct User User;
typedef struct {
    User* users[MAX_USERS];
    int   count;
    void (*broadcast)(struct { User* users[MAX_USERS]; int count; void*;}*, User*, const char*);
} ChatRoom;

struct User {
    char     name[32];
    ChatRoom* room;
};

typedef struct {
    User* users[MAX_USERS];
    int   count;
} Room;

void room_join(Room* r, User* u) { r->users[r->count++] = u; }
void room_send(Room* r, User* from, const char* msg) {
    for(int i=0;i<r->count;i++)
        if(r->users[i] != from)
            printf("[%s → %s]: %s\n", from->name, r->users[i]->name, msg);
}

int main() {
    Room room = {0};
    User alice = { "Alice" };
    User bob   = { "Bob"   };
    User carol = { "Carol" };
    room_join(&room, &alice);
    room_join(&room, &bob);
    room_join(&room, &carol);
    room_send(&room, &alice, "大家好！");
}

备忘录模式 Memento

在不破坏封装的前提下，捕获并外部化对象的内部状态， 以便之后可以将对象恢复到此状态。支持撤销/重做、快照等功能。

📖 导读：备忘录是"时光胶囊"——快照对象状态，需要时恢复。 与命令模式配合可实现完整的撤销/重做历史。

C语言实现（编辑器快照）

#include <stdio.h>
#include <string.h>
#include <stdlib.h>

/* 备忘录：快照 */
typedef struct {
    char content[512];
} Memento;

/* 原发器：编辑器 */
typedef struct {
    char content[512];
} Editor;

Memento editor_save(const Editor* e) {
    Memento m; strncpy(m.content, e->content, 511); return m;
}
void editor_restore(Editor* e, const Memento* m) {
    strncpy(e->content, m->content, 511);
}

int main() {
    Editor ed = { "" };
    Memento history[4]; int h = 0;

    strncpy(ed.content, "版本一", 511);
    history[h++] = editor_save(&ed);     /* 快照1 */

    strncpy(ed.content, "版本二（错误修改）", 511);
    printf("当前: %s\n", ed.content);

    editor_restore(&ed, &history[0]);     /* 还原 */
    printf("还原: %s\n", ed.content);
}

访问者模式 Visitor

表示一个作用于某对象结构中各元素的操作，使你可以在不改变元素类的前提下 定义作用于这些元素的新操作。将算法与对象结构分离。

📖 导读：访问者解决"对固定结构添加新操作"的问题。 C 语言通过为每种元素类型提供访问函数，配合函数指针分发，模拟双重分发机制。

C语言实现（AST 节点访问）

#include <stdio.h>

typedef struct ASTNode ASTNode;
typedef struct Visitor Visitor;

struct ASTNode {
    const char* type;
    double      value;
    ASTNode*    left;
    ASTNode*    right;
    void (*accept)(ASTNode*, Visitor*);
};

struct Visitor {
    void (*visit_num)  (Visitor*, ASTNode*);
    void (*visit_add)  (Visitor*, ASTNode*);
};

void num_accept(ASTNode* n, Visitor* v) { v->visit_num(v, n); }
void add_accept(ASTNode* n, Visitor* v) { v->visit_add(v, n); }

/* 访问者实现：计算表达式值 */
double result = 0;
void eval_num(Visitor* v, ASTNode* n) { result = n->value; }
void eval_add(Visitor* v, ASTNode* n) {
    n->left->accept(n->left, v);  double l = result;
    n->right->accept(n->right, v); double r = result;
    result = l + r;
    printf("%.1f + %.1f = %.1f\n", l, r, result);
}

int main() {
    ASTNode n1 = { "num", 3.0, NULL, NULL, num_accept };
    ASTNode n2 = { "num", 4.0, NULL, NULL, num_accept };
    ASTNode add = { "add", 0, &n1, &n2, add_accept };
    Visitor eval = { eval_num, eval_add };
    add.accept(&add, &eval);  /* 3.0 + 4.0 = 7.0 */
}

总结 · 章节 25

第一性原理总结

从最基本的公理出发，重新推导设计模式的本质——抛开所有"应该这样做"的规则， 回答"为什么软件需要设计模式，C 语言又为何同样适用"。

📖 导读：本章不是模式列表的复述，而是从第一性原理出发， 找到所有模式背后共同的根因，给出可以独立推导出设计模式的基础逻辑。

第一性原理：从公理开始

公理 1：软件的唯一不变性是"变化"

需求会变、硬件会变、团队会变。任何抵抗变化的设计最终都会崩溃。 因此软件设计的根本目标不是"做正确的事"，而是"使未来的改变代价最小"。

公理 2：人类大脑的工作记忆有限

人同时能理解的概念数量约为 7±2 个。 大型系统必须被分解为可独立理解的模块。这直接推导出"模块化"和"抽象"的必要性。

公理 3：代码是写给人看的，顺便被机器执行

程序员花在读代码上的时间远多于写代码。可读性、可维护性是第一生产力， 而非运行效率（在大多数场景下）。

从公理推导设计模式

1
公理1 → 封装变化点

找出系统中"可能变化"的部分，用接口隔离它。这是工厂、策略、桥接、状态模式的共同动机。 在 C 里，typedef 的函数指针就是"变化的封装"。

2
公理2 → 单一职责与分层

每个模块只做一件事。外观、中介者、责任链都是在不同维度上"让每层只关注自己的职责"。 C 的 .h/.c 文件分离天然支持这种分层。

3
公理3 → 显式意图，而非魔法

设计模式给"代码结构"命名，使意图可见。 当你读到 shape_factory(SHAPE_CIRCLE)，立即知道这是工厂模式，而不是看一堆 malloc。 命名即文档。

4
组合推论 → 依赖倒置

高层模块不应依赖低层实现，两者都应依赖抽象。 在 C 中，"抽象"就是函数指针类型（typedef），这是所有模式中最核心的技术武器。

最终结论

📌 设计模式的第一性原理结论

设计模式不是面向对象的专利，而是人类应对复杂系统的认知工具。 它的本质是：在"易变"与"稳定"之间划定边界，让变化的部分可替换、稳定的部分可复用。

C 语言通过 函数指针（接口）+ struct（聚合）+ static（封装）， 可以无损地表达所有 GoF 模式的核心意图，同时保持机器级别的执行效率。

真正的设计感不来自语言特性，而来自程序员对"变化的预判"和"职责的清晰界定"。 一个设计良好的 C 程序，比一个乱用设计模式的 Java 程序拥有更高的可维护性。

工具不创造设计，思考创造设计。设计模式只是思考结果的语言。
