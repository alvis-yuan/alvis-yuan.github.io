C语言FSM设计与高性能事件驱动架构

在C语言系统（尤其是嵌入式和高性能网络服务器）中，有限状态机（FSM）与事件驱动（epoll）是构建复杂业务逻辑的基石。本文档整理了FSM的核心设计原则、状态/事件的边界定义，以及FSM与epoll结合的现代Reactor并发架构。

一、 状态机三要素：状态、事件与动作

如果把FSM比作一部电影：状态是场景，事件是推动剧情的转折点，动作是演员具体的表演。设计的核心是保证它们之间的边界清晰。

维度 状态 (State) 事件 (Event) 动作 (Action)
本质属性 环境、模式、历史记忆的总结（容器） 外部刺激、条件触发（信使） 实质性的响应逻辑（干活的）
时间特性 线段（占据一段时间，静止等待） 点（瞬间发生） 无阻塞的瞬间（Run-to-Completion）
语法命名 形容词 / V-ing（例：Heating, Idle） 名词 + 过去式（例：Btn_Pressed） 动词 + 名词（例：TurnOn_Heater）
设计原则 MECE原则（互斥且穷尽），避免状态爆炸 业务抽象（非硬件级），独立且原子 不能包含阻塞代码（如delay），多用Entry/Exit

实战案例：微波炉状态机 (UML图)

二、 C语言规范代码实现 (表驱动法)

当状态和事件定义得高度正交时，C语言中最优雅的实现是结构体数组（状态转移矩阵）。这倒逼你必须把状态和事件定义为连续的枚举值。

// 1. 定义事件与状态 (前缀区分)
typedef enum {
    EVENT_START_PRESSED = 0,
    EVENT_CANCEL_PRESSED,
    EVENT_TIMER_EXPIRED,
    EVENT_MAX
} FsmEvent_t;

typedef enum {
    STATE_IDLE = 0,
    STATE_HEATING,
    STATE_MAX
} FsmState_t;

// 2. 状态转移表定义
typedef struct {
    FsmState_t currentState;
    FsmEvent_t event;
    void (*actionFunc)(void);
    FsmState_t nextState;
} FsmTransition_t;

// 3. 路由表映射
static const FsmTransition_t fsmTable[] = {
  // 当前状态      触发事件               执行动作              下一状态
  {STATE_IDLE,    EVENT_START_PRESSED,  TurnOn_Magnetron,   STATE_HEATING},
  {STATE_HEATING, EVENT_CANCEL_PRESSED, TurnOff_Magnetron,  STATE_IDLE},
  {STATE_HEATING, EVENT_TIMER_EXPIRED,  TurnOff_Magnetron,  STATE_IDLE},
};

三、 概念辨析：epoll（事件驱动） vs 状态机（FSM）

在宏观代码结构上，它们都是一个 while(1) 循环等待回调，但本质属于不同维度的技术。

  • epoll (底层的“邮差”)： 操作系统级别的 I/O 多路复用器。它是无状态的，不关心业务流程，只负责通知“FD现在可读/可写”。
  • FSM (顶层的“大脑”)： 应用层的业务逻辑编排器。它是强状态的，维护上下文记忆，基于历史状态决定拿到数据后该做什么。
总结： epoll 提供的是一种“通知机制”（什么时候去执行）；而状态机提供的是一种“组织机制”（在回调里到底应该执行什么逻辑）。

四、 核心架构：Reactor + FSM 的并发模型

初学者的常见误区： 认为状态机需要一个专门的线程去 while(1) 等待事件。这是错误的。
状态机本质是一个被动调用的数据结构。在大多数高性能场景下，epoll 和 FSM 是在同一个线程里串行执行的。

模式 1：最经典的单线程模式 (Single-Threaded Reactor + FSM)

Redis、早期Node.js采用的模式。没有锁，没有线程切换，性能极高。

模式 2：现代C/C++服务器模式 (Multiple Reactors / One Loop Per Thread)

这是 Nginx 和 Muduo 网络库倡导的模式。为了利用多核CPU，同时避免加锁，系统会创建多个 Worker 线程。每个线程拥有属于自己的独立 epoll 和独立分配的 FSM 实例集合。
