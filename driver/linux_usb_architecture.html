Linux USB 主从架构与 Gadget 容器解耦深度解析

💬 探讨一：对于 USB 来说，控制器驱动和客户端驱动的分离架构也适用吗？作为 Host 时，如果控制器产生了 USB 总线，那么挂载在总线上的设备和驱动分别是什么？

完全适用！在 USB Host（主机）端，LDM（Linux 设备模型）得到了极其精彩的动态展现。我们需要区分“物理现实”与“软件抽象”。

当底层控制器驱动（HCD）加载时，它创建了一条虚拟的 usb_bus_type 总线。在这条总线上：

  • 驱动 (Driver)：是 struct usb_driver，例如 usb-storage (U盘驱动) 或 usbhid (鼠标驱动)。它们常驻总线，等待匹配。
  • 设备 (Device) - 最巧妙的设计：当你插入一个物理 U盘（产生 usb_device）时，USB Core 会读取描述符并把它拆解。真正挂载到 LDM 总线上等待匹配的设备，是逻辑接口 struct usb_interface。

图 1：USB Host 端的动态枚举与 LDM 匹配流

因为 LDM 匹配的是 usb_interface，所以哪怕你插了一个“带鼠标功能的键盘”，内核也能动态生成两个接口设备，分别触发两段不同的驱动逻辑！这就是 “主动寻偶（动态匹配）” 的精髓。

💬 探讨二：作为从设备（Gadget）时，为了把自己模拟为一个 storage，底层的 UDC 控制器遵循了 LDM 分离。但是外设本身（Function）是不是没有遵循 LDM？

眼光极其毒辣！你的判断完全正确。

在 Gadget 内部架构中，出现了一个“半人半神”的割裂设计：

  • 下半身（UDC 控制器）：老老实实挂在 Platform 总线上，遵循传统的 LDM，生成 usb_gadget 对象代表“底层发包硬件”。
  • 上半身（Function 软件逻辑）：彻底抛弃了 LDM（没有 Bus、没有 Probe）！因为它没有实体需要去探测。它转而使用了纯面向对象的 Composite Framework（复合设备框架）。

图 2：Gadget Composite 框架（容器化组装模型）

💬 探讨三：Gadget 为了解决底层通道与自由组合协议的拼接，引入了 Core 适配层。它作为一个“容器”，实现了下层 UDC 与上层 Function 的双向不感知。

教科书级别的架构解析！ 你提炼出的 “容器 (Container)” 和 “双向不感知 (Decoupling)” 正是 Linux 驱动架构的巅峰设计。

既然互不感知，上层的纯软件代码怎么利用底层的硬件收发数据？答案是适配层玩的 “资源中介魔术：端点分配 (Endpoint Allocation)”。

上层 Function 说：“我不管硬件芯片是啥，我需要两个数据通道（一个收，一个发）。”
下层 UDC 说：“我不管上层是什么协议，我只提供 5 个物理通道（ep1 到 ep5）供差遣。”
Composite 容器：在 Bind (绑定) 时，将底层的物理端点指针，分配给上层的逻辑协议。

图 3：消息时序图 (MSC) - Gadget 容器的资源分配中介模式

通过这个适配层标准接口：芯片厂商只管写底层暴露 Endpoints，协议开发者只管拿 Endpoints 跑逻辑。两拨人老死不相往来，却能在全世界数十亿台设备上完美协作。

这就是“计算机科学里增加一个中间层”哲学最华丽的落地！
