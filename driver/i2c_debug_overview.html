第一章

I2C 协议简介

从协议规范出发，理解 Linux I2C 子系统的设计基础

🔗 原文：Introduction to I2C and SMBus — The Linux Kernel documentation

I²C（读作 "I squared C"，在内核文档中写作 I2C）是 Philips 公司开发的协议。它是一种双线低速协议（可变速，最高 400 kHz），具有高速扩展（3.4 MHz）。I2C 为连接多种低频率或低带宽通信需求的设备提供了低成本总线，在嵌入式系统中被广泛使用。

某些系统使用不符合品牌要求的变体，因此不以 I2C 名称销售，而采用不同名称，例如 TWI（Two Wire Interface，两线接口） 和 IIC。

📖 官方规范

I2C 官方规范是 NXP Semiconductors 发布的《I2C-bus specification and user manual》（UM10204）。

SMBus（系统管理总线）

SMBus（System Management Bus）基于 I2C 协议，是 I2C 协议和信令的子集。许多 I2C 设备可在 SMBus 上工作，但某些 SMBus 协议增加了超出 I2C 品牌要求的语义。现代 PC 主板依赖 SMBus，通过 SMBus 连接的最常见设备是使用 I2C EEPROM 配置的 RAM 模块以及硬件监控芯片。

Linux 内核中的术语

术语I2C 规范Linux 内核称呼驱动位置
主机芯片（Master） 发起通信的节点 适配器（Adapter）/ 总线 drivers/i2c/busses/
算法（Algorithm） — 可用于实现整类 I2C 适配器的通用代码 drivers/i2c/algos/
从机芯片（Slave） 响应主机通信的节点 客户端（Client） drivers/media/i2c/ 等

图1：I2C 总线拓扑 — 一个主机（Master/Adapter）连接多个从机（Slave/Client）共享 SDA 和 SCL 两根线

I2C 信号特性

  • I2C 只需要 两根信号线（SCL 时钟线、SDA 数据线），节省 PCB 空间并最大限度减少信号质量问题。
  • 大多数 I2C 设备使用 7 位地址，总线速度最高 400 kHz；存在尚未广泛使用的高速扩展（3.4 MHz）。
  • I2C 是一种多主机总线；使用漏极开路信令在主机之间进行仲裁，以及握手和同步时钟。

📝 本章总结

  • I2C 是 NXP（原 Philips）设计的两线低速串行协议，官方规范为 UM10204。
  • Linux 内核将 I2C 主机（Master）抽象为 Adapter（适配器），将 从机（Slave）抽象为 Client（客户端）。
  • SMBus 是 I2C 的子集协议，被现代 PC 广泛用于硬件监控和 EEPROM 配置。

🔬 第一性原理

I2C 的根本设计动机是：用最少的硬件引脚（仅2根）实现多设备互联。为此它选择了共享漏极开路总线（天然"线与"），任何设备都可以将线拉低，从而实现多主机仲裁、时钟同步和从机 ACK 应答，这正是整个 Linux I2C 子系统分层设计的基础出发点。

第二章

Linux I2C 层级架构

内核如何分层抽象 I2C 硬件与驱动

🔗 原文：I2C and SMBus Subsystem — The Linux Kernel documentation

Linux I2C 编程接口支持总线交互的主机端（master side）和从机端（slave side）。编程接口围绕两种驱动和两种设备构建：

  • I2C 适配器驱动（Adapter Driver）：抽象控制器硬件；绑定到物理设备（例如 PCI 设备或 platform_device），并暴露 struct i2c_adapter，代表其管理的每个 I2C 总线段。
  • I2C 客户端驱动（Client Driver）：每个 I2C 总线段上的 I2C 设备由 struct i2c_client 表示，这些设备会绑定到 struct i2c_driver，后者遵循标准的 Linux 驱动模型。

核心数据结构

数据结构角色说明
struct i2c_adapter 总线/适配器 代表一条 I2C 总线，包含算法指针、总线编号、设备列表等
struct i2c_algorithm 传输方法 定义 xfer()、smbus_xfer()、functionality() 等回调
struct i2c_client 从机设备 代表总线上的一个 I2C 设备，含地址、名称、所属 adapter 等
struct i2c_driver 设备驱动 含 probe()、remove()、id_table 等，遵循 Linux 驱动模型
struct i2c_board_info 设备描述 用于构建 I2C 设备信息表（类型、地址、IRQ 等），在静态注册场景使用
struct i2c_msg 消息 一次 I2C 读或写传输的基本单元，含地址、标志、数据缓冲区

图2：Linux I2C 子系统层级架构图

适配器驱动（Adapter Driver）

I2C "适配器驱动" 抽象控制器硬件；它绑定到物理设备（可能是 PCI 设备或 platform_device），并暴露 struct i2c_adapter，代表其管理的每个 I2C 总线段。Linux 内核在源目录 drivers/i2c/busses 下提供了多种 I2C 总线驱动，用于将内核 I2C API 转换为针对不同系统的寄存器操作。

struct i2c_algorithm — 传输方法

来源：docs.kernel.org/driver-api/i2c.html
struct i2c_algorithm {
    union {
        int (*xfer)(struct i2c_adapter *adap, struct i2c_msg *msgs, int num);
        int (*master_xfer)(...);  /* 已废弃，使用 xfer */
    };
    int (*smbus_xfer)(struct i2c_adapter *adap, u16 addr,
                      unsigned short flags, char read_write,
                      u8 command, int size, union i2c_smbus_data *data);
    u32 (*functionality)(struct i2c_adapter *adap);
};

struct i2c_client — 从机设备

来源：docs.kernel.org/driver-api/i2c.html
struct i2c_client {
    unsigned short flags;       /* I2C_CLIENT_PEC | I2C_CLIENT_TEN | ... */
    unsigned short addr;        /* I2C 总线上使用的地址 */
    char           name[I2C_NAME_SIZE]; /* 设备类型/芯片名称 */
    struct i2c_adapter *adapter; /* 管理该设备所在总线段的适配器 */
    struct device  dev;          /* 驱动模型设备节点 */
    int            irq;          /* 该设备产生的 IRQ（如有）*/
    struct dentry  *debugfs;     /* I2C core 为该 client 创建的 debugfs 子目录 */
};

struct i2c_driver — 设备驱动

来源：docs.kernel.org/driver-api/i2c.html
struct i2c_driver {
    unsigned int class;
    int  (*probe)(struct i2c_client *client);   /* 设备绑定回调 */
    void (*remove)(struct i2c_client *client); /* 设备解绑回调 */
    void (*shutdown)(struct i2c_client *client);
    struct device_driver       driver;
    const struct i2c_device_id *id_table; /* 驱动支持的 I2C 设备列表 */
    int  (*detect)(struct i2c_client *client, struct i2c_board_info *info);
    const unsigned short      *address_list; /* 探测地址列表（用于 detect）*/
};

设备实例化方式

🔗 原文：How to instantiate I2C devices — The Linux Kernel documentation

与 PCI 或 USB 设备不同，I2C 设备不在硬件层面枚举。软件必须知道每个 I2C 总线段上连接了哪些设备及其地址。实例化 I2C 设备有几种方式：

方法场景机制
方法1：静态声明 嵌入式系统、已知总线号 设备树（DTS）、ACPI 或 board files；总线注册时自动实例化
方法2：新风格驱动 枚举设备（DT/ACPI） 驱动模型自动绑定
方法3：探测（Probing） 无其他手段时 使用 I2C_SMBUS_QUICK 探测，须在总线设置 class 位域
方法4：用户空间实例化 调试/开发 写入 /sys/bus/i2c/devices/i2c-N/new_device
📌 关键说明

如有可能，应首选方法 1 和方法 2。方法 3（探测）仅在没有其他方式时才使用，因为它可能带来不希望的副作用。
通过 sysfs 的 new_device 和 delete_device 两个只写属性文件，可在用户空间动态实例化和删除 I2C 设备。

📝 本章总结

  • Linux I2C 子系统分为：用户空间工具层 → i2c-dev → I2C Core → 客户端驱动 / 总线驱动 → 硬件。
  • 核心抽象：i2c_adapter（总线）、i2c_algorithm（传输方法）、i2c_client（从机设备）、i2c_driver（驱动）。
  • I2C 设备不可自动枚举，依赖 DTS/ACPI 静态声明或用户空间显式实例化。
  • 总线驱动位于 drivers/i2c/busses/，客户端驱动按功能分布在各子系统目录下。

🔬 第一性原理

Linux I2C 子系统分层的根本原因：硬件多样性 + 设备多样性 = 需要两个正交的抽象维度。适配器层隔离 SoC 差异（寄存器操作各不相同），客户端层隔离设备差异（传感器、EEPROM、GPIO 扩展器语义各不相同）。i2c_algorithm 是两层之间的接口契约，使得任意适配器都能驱动任意客户端。

第三章

/sys/bus/i2c 目录结构

Sysfs 是理解 Linux I2C 拓扑的主要入口

🔗 原文：Linux I2C Sysfs — The Linux Kernel documentation

Linux Sysfs 文件系统通常挂载在 /sys 目录。I2C Sysfs 位于 /sys/bus/i2c/devices，可以直接 cd 进入。该目录下有一系列符号链接：

  • 以 i2c- 开头的链接是 I2C 总线，可以是物理或逻辑总线。
  • 以数字开头、以数字结尾的链接是 I2C 设备，格式为 ${总线号}-${设备地址}，其中总线号为十进制，地址为十六进制。

以 Google Pixel 3 手机为例：

示例：/sys/bus/i2c/devices 目录内容（来源：docs.kernel.org/i2c/i2c-sysfs.html）
blueline:/sys/bus/i2c/devices $ ls
0-0008  0-0061  1-0028  3-0043  4-0036  4-0041  i2c-1  i2c-3
0-000c  0-0066  2-0049  4-000b  4-0040  i2c-0   i2c-2  i2c-4

其中 i2c-2 是编号为 2 的 I2C 总线，2-0049 是总线 2 地址 0x49 处绑定了内核驱动的 I2C 设备。

完整 /sys/bus/i2c 目录结构

图3：/sys/bus/i2c 完整目录结构关系图

关键路径说明

路径含义
/sys/bus/i2c/devices/所有 I2C 总线和设备的符号链接列表
/sys/bus/i2c/devices/i2c-N/第 N 号逻辑 I2C 总线目录
/sys/bus/i2c/devices/N-XXXX/总线 N 地址 0xXXXX 处的 I2C 设备目录
/sys/bus/i2c/devices/i2c-N/new_device只写文件，用于从用户空间实例化 I2C 设备
/sys/bus/i2c/devices/i2c-N/delete_device只写文件，用于从用户空间删除 I2C 设备
/sys/bus/i2c/drivers/已注册的 I2C 驱动目录
/sys/class/i2c-dev/字符设备（i2c-dev）接口，含 Major/Minor 号
/dev/i2c-N用户空间访问接口，字符设备 Major=89

查看适配器信息

通过 sysfs 查看 i2c-7 的信息
~$ cd /sys/bus/i2c/devices/i2c-7
/sys/bus/i2c/devices/i2c-7$ ls

delete_device  i2c-73         new_device     uevent
device         i2c-86         of_node

# 读取总线名称
/sys/bus/i2c/devices/i2c-7$ cat name

# 读取 device 符号链接，判断是否为物理总线
/sys/bus/i2c/devices/i2c-7$ readlink device

# 链接指向 .i2c 结尾的目录 → 物理 I2C 总线

📝 本章总结

  • /sys/bus/i2c/devices/ 是 Linux I2C Sysfs 的主入口，包含 I2C 总线（i2c-N）和设备（N-XXXX）的符号链接。
  • new_device / delete_device 文件允许在用户空间动态实例化/删除 I2C 设备。
  • 通过读取总线目录下的 device 符号链接可以判断是物理总线还是逻辑（MUX）总线。
  • /sys/class/i2c-dev/ 和 /dev/i2c-N 由 i2c-dev 模块维护，提供字符设备访问接口。
第四章

逻辑 I2C 总线与 MUX 拓扑

理解物理总线、逻辑总线与 I2C MUX 的关系

🔗 原文：Linux I2C Sysfs — The Linux Kernel documentation

I2C 拓扑可能很复杂，因为存在 I2C MUX（I2C 多路复用器）。Linux 内核将 MUX 通道抽象为逻辑 I2C 总线编号。

术语定义

术语定义
（物理）I2C 总线控制器 Linux 运行的硬件系统可能有多个物理 I2C 总线控制器，控制器是硬件和物理的，系统在内存空间中定义了多个寄存器来操控控制器。Linux 内核在 drivers/i2c/busses 下有 I2C 总线驱动将内核 I2C API 转换为针对不同系统的寄存器操作。
I2C 总线物理编号 对于每个物理 I2C 总线控制器，系统厂商可能为每个控制器分配一个物理编号。例如寄存器地址最低的第一个 I2C 总线控制器可能被称为 I2C-0。
逻辑 I2C 总线 在 Linux I2C Sysfs 中看到的每个 I2C 总线号都是一个已分配编号的逻辑 I2C 总线，类似于软件代码通常基于虚拟内存地址而非物理内存地址编写。每个逻辑 I2C 总线可能是物理 I2C 总线控制器的抽象，或是 I2C MUX 背后某个通道的抽象。
物理 I2C 总线 如果逻辑 I2C 总线是物理 I2C 总线控制器的直接抽象，则称为物理 I2C 总线。
⚠️ 注意事项（Caveat）

在设备树（DTS）的 aliases 段中，可以将 I2C 总线物理编号重命名为不同的逻辑 I2C 总线编号。最佳实践是：保持物理 I2C 总线编号与对应逻辑 I2C 总线编号相同，而不是重命名，以避免混淆。参见 arch/arm/boot/dts/nuvoton-npcm730-gsj.dts。

MUX 拓扑示例

以下是文档中使用的复杂 I2C 拓扑示例（来源：docs.kernel.org/i2c/i2c-sysfs.html）：

i2c-7 (物理 I2C 总线控制器 7)
└── 7-0071 (4 通道 I2C MUX，地址 0x71)
    ├── i2c-60 (channel-0)
    ├── i2c-73 (channel-1)
    │   ├── 73-0040 (I2C 传感器，带 hwmon 目录)
    │   ├── 73-0070 (I2C MUX 0x70，存在于 DTS 但 probe 失败)
    │   └── 73-0072 (8 通道 I2C MUX，地址 0x72)
    │       ├── i2c-78 (channel-0)
    │       ├── ... (channel-1…6, i2c-79…i2c-84)
    │       └── i2c-85 (channel-7)
    ├── i2c-86 (channel-2)
    └── i2c-203 (channel-3)

图4：I2C MUX 拓扑结构 — 物理总线通过 MUX 扇出为多个逻辑总线（来源：docs.kernel.org/i2c/i2c-sysfs.html）

区分物理与逻辑 I2C 总线

区分的简单方法是读取 I2C 总线目录下的 device 符号链接：

来源：docs.kernel.org/i2c/i2c-sysfs.html
# 物理总线示例：device 链接指向以 .i2c 结尾的目录
$ readlink /sys/bus/i2c/devices/i2c-7/device
../../f0087000.i2c
$ ls /sys/bus/i2c/devices/i2c-7/mux_device
ls: /sys/bus/i2c/devices/i2c-7/mux_device: No such file or directory
# 没有 mux_device → 物理总线

# 逻辑总线示例：device 链接指向另一 I2C 总线
$ readlink /sys/bus/i2c/devices/i2c-73/device
../../i2c-7
$ readlink /sys/bus/i2c/devices/i2c-73/mux_device
../7-0071
# mux_device 存在 → 逻辑总线（由 7-0071 MUX 扇出）

查找逻辑 I2C 总线号的完整步骤

来源：docs.kernel.org/i2c/i2c-sysfs.html — 查找 0x72 MUX channel-3 的逻辑总线号
# 步骤1：进入物理总线目录
~$ cd /sys/bus/i2c/devices/i2c-7
/sys/bus/i2c/devices/i2c-7$ ls
7-0071  i2c-60  i2c-73  i2c-86  i2c-203  name  device  ...

# 步骤2：进入 MUX 设备目录
/sys/bus/i2c/devices/i2c-7$ cd 7-0071/
/sys/bus/i2c/devices/i2c-7/7-0071$ readlink channel-1
../i2c-73
# channel-1 对应逻辑总线 i2c-73

# 步骤3：进入 i2c-73，找到第二级 MUX
$ cd /sys/bus/i2c/devices/i2c-73
/sys/bus/i2c/devices/i2c-73$ cd 73-0072
/sys/bus/i2c/devices/i2c-73/73-0072$ readlink channel-3
../i2c-81
# 找到！0x72 MUX 的 channel-3 是逻辑总线 i2c-81

固定逻辑总线号（Pinned Logical I2C Bus Number）

若未在 DTS 中指定，当 I2C MUX 驱动应用且 MUX 设备成功 probe 后，内核将根据当前最大逻辑总线号递增分配 MUX 通道的逻辑总线号。例如，若系统的最高逻辑总线号为 i2c-15，且成功应用 4 通道 MUX，则 MUX channel 0 为 i2c-16，以此类推直到 i2c-19 为 channel 3。

内核软件开发者可以在 DTS 中将扇出 MUX 通道固定到静态逻辑 I2C 总线号。参见 arch/arm/boot/dts/aspeed-bmc-facebook-wedge400.dts。

💡 提示

可以设计一个易于人工记忆或按算术规律计算的逻辑 I2C 总线号方案。例如，将总线 3 上 MUX 的扇出通道固定为从 30 开始，则 30 为总线 3 上 MUX channel 0 的逻辑总线号，37 为 channel 7 的逻辑总线号。

📝 本章总结

  • Linux 内核将 I2C MUX 通道抽象为逻辑 I2C 总线，每次访问逻辑总线时，内核自动切换 MUX 到对应通道。
  • 通过读取总线目录下 device 链接（指向 .i2c → 物理总线，指向 i2c-X → 逻辑总线）和 mux_device 链接来区分。
  • 逻辑总线号默认动态分配（按当前最大号递增），也可通过 DTS aliases 固定。
  • 查找 MUX 通道对应的逻辑总线号：进入 MUX 设备目录 → 读取 channel-N 符号链接。

🔬 第一性原理

MUX 逻辑总线的本质是透明代理：对上层（用户空间、驱动）完全隐藏 MUX 的存在，访问逻辑总线 81 和访问物理总线 7 在接口上完全一致，内核在传输前自动完成 MUX 通道切换。这正是操作系统抽象层的价值所在——消灭硬件差异，提供统一接口。

第五章

I2C 设备目录详解

深入了解每个 I2C 设备目录的内容

🔗 原文：Linux I2C Sysfs — The Linux Kernel documentation（I2C Devices 章节）

每个 I2C 设备目录名称格式为 ${bus}-${addr}，${bus} 是十进制逻辑总线号，${addr} 是十六进制 I2C 地址。

设备目录基本文件

文件/目录类型说明

subsystem符号链接指向 /sys/bus/i2c

📝 本章总结

  • I2C 设备目录以 ${bus}-${addr} 命名，name 文件记录芯片型号，driver 链接记录所用驱动。
  • driver 链接缺失 → probe 失败，结合 dmesg 可定位错误原因。
  • MUX 设备额外包含 channel-* 符号链接指向各逻辑子总线。
  • 传感器设备若被 hwmon 驱动绑定，则包含 hwmon/ 子目录，其中的 sysfs 文件遵循标准命名约定（in0_input、curr1_input 等）。
第六章

/dev/i2c-N 用户空间接口

通过字符设备直接访问 I2C 适配器

🔗 原文：Implementing I2C device drivers in userspace — The Linux Kernel documentation

通常，I2C 设备由内核驱动控制。但也可以通过 /dev 接口从用户空间访问适配器上的所有设备。需要加载 i2c-dev 模块。

设备文件规范

  • 每个已注册的 I2C 适配器从 0 开始分配编号。
  • I2C 设备文件是字符设备文件，主设备号为 89，次设备号对应总线编号。
  • 文件命名为 i2c-%d（i2c-0、i2c-1……i2c-10……）。
  • 所有 256 个次设备号都为 I2C 保留。
  • 可检查 /sys/class/i2c-dev/ 来查看编号与适配器的对应关系，或运行 i2cdetect -l。
⚠️ 适配器编号注意

适配器编号是动态分配的，不能做过多假设，甚至可能在两次启动之间发生变化。应通过 /sys/class/i2c-dev/ 或 i2cdetect -l 来确定实际编号。

C 语言访问示例

来源：docs.kernel.org/i2c/dev-interface.html
#include <linux/i2c-dev.h>
#include <i2c/smbus.h>

int file;
int adapter_nr = 2; /* 通常动态确定 */
char filename[20];

/* 步骤1：打开设备文件 */
snprintf(filename, 19, "/dev/i2c-%d", adapter_nr);
file = open(filename, O_RDWR);
if (file < 0) { /* ERROR HANDLING */ exit(1); }

/* 步骤2：指定从机地址 */
int addr = 0x40; /* I2C 从机地址 */
if (ioctl(file, I2C_SLAVE, addr) < 0) { exit(1); }

/* 步骤3a：使用 SMBus 命令（推荐）*/
__u8 reg = 0x10;
__s32 res = i2c_smbus_read_word_data(file, reg);
if (res < 0) { /* 错误处理 */ }

/* 步骤3b：使用原始 I2C 写（等效于 SMBus write_word_data）*/
char buf[3];
buf[0] = reg;   buf[1] = 0x43;  buf[2] = 0x65;
if (write(file, buf, 3) != 3) { /* 错误处理 */ }

支持的 ioctl 命令

ioctl 命令说明
I2C_SLAVE 设置从机地址；若地址已被其他驱动使用则失败
I2C_SLAVE_FORCE 强制设置从机地址（即使已被使用），危险！
I2C_TENBIT 启用/禁用 10 位地址模式（需要 I2C_FUNC_10BIT_ADDR）
I2C_PEC 启用/禁用 SMBus 数据包错误校验（PEC）
I2C_FUNCS 查询适配器支持的功能（I2C_FUNC_* 标志）
I2C_RDWR 执行联合读/写事务（无中间 STOP），仅当适配器支持 I2C_FUNC_I2C 时有效
I2C_SMBUS 执行 SMBus 事务
📖 i2c-dev 模块说明

i2c-dev 驱动负责校验从用户空间传入的所有参数的有效性。在此之后，通过 i2c-dev 从用户空间发起的调用与内核 I2C 芯片驱动直接发起的调用没有区别。这意味着 I2C 总线驱动无需实现任何特殊逻辑来支持用户空间访问。可以将 i2c-dev 视为一个可以从用户空间编程的通用 I2C 芯片驱动。

📝 本章总结

  • /dev/i2c-N 是字符设备（主设备号 89），由 i2c-dev 内核模块提供，允许用户空间程序直接访问 I2C 适配器。
  • 访问流程：open("/dev/i2c-N") → ioctl(I2C_SLAVE, addr) → read()/write() 或 SMBus 函数。
  • SMBus 命令优于原始 I2C 读写，条件允许时应优先使用。
  • i2c-dev 模块完成参数验证，对总线驱动完全透明，无需特殊适配。

i2cget — 读取 I2C/SMBus 寄存器

🔗 i2cget 手册页

i2cget 用于读取 I2C/SMBus 芯片寄存器的值。

基本语法

i2cget [-f] [-y] [-a] i2cbus chip-address [data-address [mode [length]]]

常用示例

# 从 i2c-1 总线地址 0x68 的设备读取寄存器 0x00（字节模式）
# i2cget -y 1 0x68 0x00 b

# 读取字（2字节）
# i2cget -y 1 0x68 0x00 w
0x006a

# 读取 EEPROM 前 8 字节（I2C 块读取，来源：kernel.org patch）
# i2cget -y 4 0x50 0x00 i 8
0x00 0x01 0x02 0x03 0x04 0x05 0x06 0x07

# 读/写字节事务（先写寄存器地址，再读返回值）
# i2cget -y 1 0x68 0x00 c

mode 参数说明

mode说明
b读取字节数据（read byte data）
w读取字数据（read word data）
c写字节/读字节事务（write byte/read byte）
iI2C 块读取（read I2C block）；需附加 length 参数
p 后缀追加 p 启用 SMBus PEC（不适用于 I2C 块事务）

i2cset — 写入 I2C/SMBus 寄存器

🔗 i2cset 手册页

i2cset 用于向 I2C/SMBus 芯片寄存器写入值。

⚠️ 危险警告

不当使用 i2cset 可能损坏您的 I2C 总线、导致数据丢失，或产生更严重的后果。向内存 DIMM 上的串行 EEPROM（芯片地址 0x50 到 0x57 之间）写入数据可能损毁内存，导致系统无法启动！请极其谨慎地使用此程序。

基本语法

i2cset [-f] [-y] [-m mask] [-r] [-a] i2cbus chip-address data-address [value] ... [mode]

常用示例

# 向 i2c-1 总线 0x2d 设备的寄存器 0x11 写入值 0xaa（字节模式）
# i2cset -y 1 0x2d 0x11 0xaa b

# 写入字（2字节）
# i2cset -y 1 0x2d 0x11 0x1234 w

# 带掩码写（-m 0x0f 只修改低4位）
# i2cset -y -m 0x0f 1 0x2d 0x11 0xaa b

# 写后读回验证（-r 选项）
# i2cset -y -r 1 0x2d 0x11 0xaa b

主要选项

选项说明
-y禁用交互式确认
-f强制访问，即使设备正被驱动使用（危险！）
-m mask位掩码，只修改与掩码对应的位
-r写入后立即读回并显示，验证写入结果
工具详解

i2ctransfer — 自定义 I2C 消息传输

🔗 i2ctransfer 手册页

i2ctransfer 是 Wolfram Sang 开发的工具，允许在一次传输中发送用户自定义的多条 I2C 消息，没有 SMBus 降级。主要用于测试 I2C 主控制器。

⚠️ 危险警告

i2ctransfer 若使用不当可能极其危险。它可能混乱您的 I2C 总线、导致数据丢失，或产生更严重的后果。向内存 DIMM 上的串行 EEPROM 写入可能损毁内存，导致系统无法启动！

基本语法

i2ctransfer [-f] [-y] [-v] [-V] [-a] i2cbus desc [data] [desc [data]] ...

消息描述格式：{r|w}Length@Address，读消息用 r，写消息用 w。

官方使用示例

来源：i2ctransfer 手册页（patchwork.kernel.org）
# 从 i2c-0 总线地址 0x50 的 EEPROM，从偏移 0x64 读取 8 字节
# （第一条消息写1字节设置内存指针，第二条消息读取数据）
# i2ctransfer 0 w1@0x50 0x64 r8

# 向同一 EEPROM 的偏移 0x42 写入 0xff 0xfe ... 0xf0
# （一条写消息：第一字节设置指针为 0x42，后续字节从 0xff 递减）
# i2ctransfer 0 w17@0x50 0x42 0xff-

数据后缀说明

后缀含义示例
=保持值不变直到消息结束0= → 0, 0, 0, ...
+每字节递增 10+ → 0, 1, 2, ...
-每字节递减 10xff- → 0xff, 0xfe, 0xfd, ...
p以该值为种子生成 8 位伪随机序列0p → 0x00, 0x50, 0xb0, ...

🔬 总体第一性原理

Linux I2C 子系统的整体设计遵循一条核心原则：「每一层只解决一个问题，通过接口契约与其他层解耦」。

  • 硬件差异由适配器驱动（struct i2c_algorithm）屏蔽 → 上层代码不关心是哪款 SoC 的 I2C 控制器。
  • 设备差异由客户端驱动屏蔽 → 上层代码不关心是温度传感器还是 EEPROM。
  • MUX 复杂性由 Core 的逻辑总线抽象屏蔽 → 用户代码访问 i2c-81 和 i2c-7 完全等价。
  • 内核/用户边界由 i2c-dev 模块和 sysfs 屏蔽 → 用户空间工具（i2c-tools）无需了解内核内部结构。

这种分层解耦策略使得任意硬件组合（SoC × MUX × 设备）都能被统一的 Linux I2C 框架支持，而无需改动上层代码。
