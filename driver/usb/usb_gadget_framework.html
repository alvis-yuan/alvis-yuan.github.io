1. 框架概述

Linux USB Gadget 子系统让嵌入式设备作为 USB 外设（Peripheral / Device） 工作， 与 Host 控制器（EHCI/OHCI/xHCI）相对。本目录实现从硬件 UDC 控制器驱动到协议功能驱动的完整栈。

Kconfig 入口：CONFIG_USB_GADGET 开启 Gadget 支持； CONFIG_USB_LIBCOMPOSITE 提供 Composite 基础设施； CONFIG_USB_CONFIGFS 允许通过 ConfigFS 在用户态动态组装 Gadget。 Gadget 支持不依赖 Host 侧 CONFIG_USB。

典型数据流：Host 发起 USB 事务 → UDC 硬件中断 → UDC 驱动 → Gadget 框架回调 → Composite 分发 → Function 驱动 → 用户态（可选）。

2. 目录与模块组成

drivers/usb/gadget/
├── libcomposite.ko          ← 核心框架（见 Makefile）
│   ├── composite.c          Composite 设备逻辑、EP0、set_config
│   ├── config.c             描述符组装辅助
│   ├── configfs.c           ConfigFS 用户态配置入口
│   ├── functions.c          Function 驱动注册/查找
│   ├── epautoconf.c         端点自动分配
│   ├── usbstring.c          字符串描述符
│   └── u_f.c                Function 通用辅助
├── udc/                     ← 各 SoC 的 UDC 控制器驱动 + core.c
├── function/                ← 可复用 usb_f_*.ko Function 模块
└── legacy/                  ← 单体 g_*.ko 驱动（内部直接 composite）

顶层 Makefile 将三部分统一纳入 CONFIG_USB_GADGET：

obj-$(CONFIG_USB_LIBCOMPOSITE) += libcomposite.o
obj-$(CONFIG_USB_GADGET)       += udc/ function/ legacy/

3. 分层架构

自底向上共四层：硬件 UDC → UDC 驱动与 Gadget API → Composite 框架 → Function / 用户态。

4. 核心对象关系

关键结构体定义于 include/linux/usb/gadget.h 与 include/linux/usb/composite.h，运行时由 Composite 层串联。

5. libcomposite 详解

5.1 composite.c — 中枢逻辑

函数/机制职责
usb_composite_probe()填充 composite_driver_template，调用 usb_gadget_probe_driver()
composite_bind()分配 usb_composite_dev，调用驱动 bind(cdev) 注册配置
composite_setup()EP0 标准请求处理：GET_DESCRIPTOR、SET_CONFIGURATION 等
set_config()Host 选择配置后，遍历 interface 调用各 f->set_alt(f, intf, 0)
reset_config()切换/清除配置时调用各 f->disable(f)
usb_add_config()注册一个 configuration，执行其 bind 回调添加 functions
usb_add_function()将 function 挂入 configuration，分配 interface ID
function_descriptors()按当前 speed 选择 fs/hs/ss/ssp 描述符表
config_ep_by_speed_and_alt()为 endpoint 设置与速率匹配的 descriptor

5.2 辅助模块

  • config.c — usb_descriptor_fillbuf()、usb_gadget_config_buf() 组装配置描述符
  • epautoconf.c — usb_ep_autoconfig() / usb_ep_autoconfig_ss() 自动匹配空闲端点
  • usbstring.c — 字符串 ID 分配与 UTF-16LE 描述符生成
  • functions.c — Function 驱动注册表，支持 request_module("usbfunc:%s") 按需加载
  • u_f.c — ConfigFS 属性、OS Descriptor 等 Function 侧共用代码

6. UDC 核心层

udc/core.c 实现 UDC 类驱动框架，统一管理所有 peripheral controller 驱动注册的 usb_gadget。

6.1 本目录 UDC 驱动示例

各平台实现在 udc/ 子目录，均通过 usb_add_gadget_udc() 注册。常见包括：

  • dummy_hcd.c — 虚拟 UDC，开发调试首选
  • fsl_udc_core.c / snps_udc_core.c — NXP / Synopsys DWC3
  • aspeed-vhub/ — BMC 虚拟 HUB 型 UDC
  • s3c2410_udc.c、at91_udc.c 等 — 各 ARM SoC 历史驱动

7. Function 驱动与 Legacy 驱动

现代推荐方式是将每个 USB 类协议实现为独立 Function 模块，通过 Composite 或 ConfigFS 组合；Legacy 驱动则在单模块内完成全部 composite 逻辑。

模式位置配置方式典型模块
Function + ConfigFS function/f_*.c 用户态写 ConfigFS，写 UDC 绑定 usb_f_printer.ko、usb_f_iap2.ko
Legacy 单体 legacy/*.c modprobe + 模块参数 g_printer.ko、g_serial.ko

7.1 Function 驱动注册模式

/* function/f_printer.c 等 */
DECLARE_USB_FUNCTION_INIT(printer, printer_alloc_instance, printer_alloc_func);
/* 展开为 usb_function_register() + MODULE_ALIAS("usbfunc:printer") */

ConfigFS 通过 usb_get_function_instance("printer") 查找并实例化；内核也会 request_module("usbfunc:printer") 自动加载对应 .ko。

7.2 Function 生命周期回调

回调调用时机
bind(config, f)配置绑定时：分配 interface ID、usb_ep_autoconfig、注册 cdev
unbind(config, f)配置移除或驱动卸载
set_alt(f, intf, alt)Host SET_CONFIGURATION / SET_INTERFACE 后启用端点
disable(f)配置切换或 disconnect 时禁用端点、清理请求
setup(f, ctrl)类/vendor 控制请求（非标准设备级）
suspend/resume(f)USB 总线挂起/恢复

8. ConfigFS 动态配置

configfs.c 在 /sys/kernel/config/usb_gadget/ 下暴露可配置 Gadget。核心私有结构 gadget_info 嵌入 usb_composite_dev 与 usb_composite_driver。

# 典型 ConfigFS 配置流程（printer function 示例）
mkdir /sys/kernel/config/usb_gadget/g1
cd g1
echo 0x0483 > idVendor && echo 0x5740 > idProduct
mkdir strings/0x409 && echo "MyPrinter" > strings/0x409/product
mkdir functions/printer.0
mkdir configs/c.1 && mkdir configs/c.1/strings/0x409
ln -s functions/printer.0 configs/c.1/
echo "My Config" > configs/c.1/strings/0x409/configuration
echo <udc_name> > UDC          # 绑定 UDC，开始枚举

10. EP0 控制请求分发

所有标准 USB 控制传输由 UDC 收到后调用 gadget_driver->setup()，Composite 层实现为 composite_setup()。

11. Bulk/Interrupt 数据传输

Function 驱动在 set_alt() 中调用 usb_ep_enable() 启用端点，随后通过 usb_ep_alloc_request() + usb_ep_queue() 发起传输。 UDC 完成 DMA 后调用 req->complete(ep, req)，驱动在回调中处理数据并可能提交下一个 request。
