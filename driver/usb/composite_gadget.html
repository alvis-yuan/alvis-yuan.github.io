USB Gadget Composite 框架

基于 composite.c · config.c · functions.c · include/linux/usb/composite.h · 内核官方 Gadget API 文档

1. 概述与原理

Composite Device（复合设备）是 USB 规范 Chapter 9 中的设备模型： 一个物理 USB 设备可暴露一个或多个 Configuration，每个 Configuration 包含若干 Interface。在 Linux Gadget 栈中，每个 Interface（或 IAD 组）通常由一个 usb_function 驱动实现。

Composite 框架（libcomposite.ko）是可选的中间层，位于 Gadget API（usb_gadget_driver）之上、各 Function 驱动之下， 负责把多个 Function「粘合」为单一 Gadget 驱动，统一处理 EP0 标准请求与配置切换。 官方说明见 USB Gadget API — Composite Device Framework。

单配置多功能 vs 多配置

模式Host 视角典型场景
单 Configuration + 多 Function枚举后同时使用多个 Interface（如 RNDIS + ADB）Android USB、多功能打印机
多 ConfigurationHost 通过 SET_CONFIGURATION 切换整组 Interface同一硬件在不同配置下暴露不同功能组合

composite.h 头文件注释中的例子：同一 Configuration 内同时提供网络与 Mass Storage， Host 可并行使用；若拆成两个 Configuration，则 Host 一次只能选其一。

Kconfig：CONFIG_USB_LIBCOMPOSITE 构建 libcomposite.ko； CONFIG_USB_CONFIGFS 依赖 libcomposite 并在 configfs.c 中提供用户态组装入口。 Gadget 栈不依赖 Host 侧 CONFIG_USB。

2. 架构与模块

libcomposite-y := usbstring.o config.o epautoconf.o
libcomposite-y += composite.o functions.o configfs.o u_f.o
源文件职责
composite.c核心：usb_composite_probe、composite_bind/setup、set_config、字符串/OS 描述符、suspend/resume
config.cusb_gadget_config_buf() — 按当前 speed 组装 Configuration 描述符
epautoconf.cusb_ep_autoconfig() / usb_ep_autoconfig_ss() — 为 Function 分配硬件端点
usbstring.c字符串 ID 分配、UTF-16LE 描述符生成
functions.cusb_function_register 全局表；request_module("usbfunc:%s") 按需加载
configfs.c用户态 Gadget 配置；configfs_composite_bind/unbind/setup
u_f.cFunction 侧 ConfigFS 属性、OS Descriptor 共用代码

composite_driver_template（composite.c）定义标准 Gadget 驱动回调： bind、unbind、setup、reset/disconnect、suspend/resume。 usb_composite_probe() 将其填入 usb_composite_driver.gadget_driver 后调用 usb_gadget_probe_driver()。

3. 核心数据结构

结构体定义位置作用

/* 典型 Legacy 驱动骨架 — composite.h 注释与 g_ffs.c 等 */
static struct usb_composite_driver my_driver = {
    .name       = "my_gadget",
    .dev        = &device_desc,
    .strings    = gadget_strings,
    .max_speed  = USB_SPEED_HIGH,
    .bind       = my_bind,      /* usb_add_config() + usb_add_function() */
    .unbind     = my_unbind,
};
module_init(() => usb_composite_probe(&my_driver));

4. 注册与绑定流程

无论 Legacy 还是 ConfigFS，最终都通过 UDC Core 调用 composite_bind()：

  1. 分配 usb_composite_dev（ConfigFS 路径使用内嵌在 gadget_info 中的 cdev）
  2. composite_dev_prepare() — 分配 EP0 请求与 4KB 缓冲区（USB_COMP_EP0_BUFSIZ）
  3. 调用驱动 bind(cdev) — 注册 configuration 与 function
  4. 可选 OS 描述符准备；更新设备描述符字段
  5. 返回后 UDC 侧 usb_gadget_connect()，Host 开始枚举

4.1 Legacy 驱动路径

legacy/g_ffs.c、legacy/serial.c、legacy/hid.c 等在 module_init 中调用 usb_composite_probe()。 bind() 内通常： 

static int my_bind(struct usb_composite_dev *cdev)
{
    return usb_add_config(cdev, &my_config, config_bind);
}

static int config_bind(struct usb_configuration *c)
{
    return usb_add_function(c, f_printer);  /* 或 usb_get_function + usb_add_function */
}

4.2 ConfigFS 路径

用户写 UDC 属性 → usb_gadget_probe_driver(&gi->composite.gadget_driver) → composite_bind() → configfs_composite_bind()（非通用 driver->bind，由 gadget_info 专用模板注册）。

  • 校验至少一个 configuration，且每个 configuration 至少一个 function（否则 pr_err 并失败）
  • 附加字符串描述符（usb_gstrings_attach）
  • 对每个 configuration 调用 usb_add_function(c, f)
  • usb_gadget_check_config() 验证端点资源是否满足 UDC 能力
ConfigFS 详细目录树与 shell 示例见专题文档 ConfigFS Gadget。

5. EP0 与 set_config

composite_setup() 是 Gadget 驱动 EP0 入口（composite_driver_template.setup）。标准请求处理摘要：

请求Composite 行为
GET_DESCRIPTORDEVICE / CONFIG / STRING / BOS / QUALIFIER 等；config_desc() 动态组装
SET_CONFIGURATIONset_config(cdev, ctrl, wValue)
GET_CONFIGURATION返回当前 bConfigurationValue 或 0
SET_INTERFACE / GET_INTERFACE路由到对应 f->set_alt / f->get_alt
类/厂商请求按 interface 或 endpoint 位图路由到 f->setup，或 config->setup

set_config()（composite.c）核心步骤：

  1. 若已有配置 → reset_config() → 各 f->disable(f)
  2. 遍历 c->interface[]，根据当前 speed 的描述符记录 endpoint 位图
  3. 对每个 interface 调用 f->set_alt(f, intf, 0)；失败则 rollback
  4. 设置 usb_gadget_vbus_draw、USB_STATE_CONFIGURED
延迟状态：若 set_alt 返回 USB_GADGET_DELAYED_STATUS， Function 需在准备好后调用 usb_composite_setup_continue(cdev) 完成控制传输。 见 composite.h 中 USB_GADGET_DELAYED_STATUS 定义。

6. 描述符、字符串与端点

多速率描述符

function_descriptors() 按当前 speed 选择 ssp_descriptors → ss_descriptors → hs_descriptors → fs_descriptors（逐级 fallback）。 Function 在 bind() 中通过 usb_interface_id() 分配 interface 号， 通过 usb_ep_autoconfig() 绑定硬件端点。

usb_add_function 约束

  • 必须提供 set_alt 与 disable（否则 usb_add_function 返回 -EINVAL）
  • bind() 失败时从 configuration 链表移除 function
  • 根据各 function 提供的描述符设置 config->fullspeed/highspeed/superspeed 标志

配置描述符组装

Host 读取 Configuration 描述符时，config_desc() 调用 usb_gadget_config_buf()（config.c）遍历各 function 在当前 speed 下的描述符链， 填入 interface、endpoint、IAD 等结构。

7. 使用方式

7.1 ConfigFS 组装（推荐）

用户态在 /sys/kernel/config/usb_gadget/ 创建 gadget、function、configuration 并 symlink，最后写 UDC。Composite 逻辑对用户透明，但底层仍走 usb_add_function。

G=/sys/kernel/config/usb_gadget/g1
mkdir -p $G/functions/printer.0 $G/configs/c.1
ln -sf $G/functions/printer.0 $G/configs/c.1/
echo "your_udc_name" > $G/UDC    # 触发 configfs_composite_bind

完整步骤与多功能示例见 ConfigFS Gadget 专题。

7.2 Legacy 内核 Composite 驱动

本树 legacy/ 目录保留经典「单模块固定组合」驱动，均基于 usb_composite_probe()：

模块文件说明
g_ffslegacy/g_ffs.cFunctionFS — 用户态通过 /dev/usb-ffs/ 提供协议
g_seriallegacy/serial.cUSB 串口 Gadget
usb_f_hid / hid gadgetlegacy/hid.cHID 复合
g_iap2legacy/iap2.cApple iAP2（本树定制）

新平台开发优先使用 ConfigFS + 独立 usb_f_* 模块，Legacy 驱动适合固定产品形态或参考 bind 写法。

7.3 Function 注册与按需加载

functions.c 维护全局 usb_function_driver 链表。ConfigFS mkdir functions/printer.0 时：

  1. usb_get_function_instance("printer")
  2. 未找到则 request_module("usbfunc:printer")
  3. alloc_inst() 创建可配置实例；symlink 到 config 时 usb_get_function() → usb_add_function()

8. 调试

8.1 排查清单

现象可能原因 / 检查
usb_add_function 返回负值dmesg 中 adding 'xxx' --> -NN；常见：端点不足、缺少 set_alt/disable、bind 失败
ConfigFS bind 失败Need at least one configuration / needs at least one function（configfs_composite_bind）
usb_gadget_check_config 失败UDC 端点数量/类型不满足所有 function 需求；检查 HS/FS 描述符是否一致
Host 枚举后立刻 disconnectset_config 中 set_alt 失败触发 reset_config；查看 interface 编号冲突
GET_DESCRIPTOR 异常字符串 ID 未分配；configuration 为空；speed 与描述符表不匹配
类请求无响应f->setup 未实现或 interface 路由错误；用 usbmon 抓 EP0
delayed status 卡死Function 未调用 usb_composite_setup_continue()

8.3 无硬件调试

使用 dummy_hcd 虚拟 UDC 与 Host 侧 usbip 或 loopback 测试（见 USB Gadget Testing）：

modprobe libcomposite
modprobe usb_f_printer
modprobe dummy_hcd
# ConfigFS 配置 gadget 后：
echo dummy_udc.0 > /sys/kernel/config/usb_gadget/g1/UDC

9. 本仓库要点

  • libcomposite.ko 由顶层 Makefile 构建，与 function/ 下各 usb_f_* 模块配合
  • configfs.c 含 Android 兼容扩展（android_usb state、uevent 上报），包装 composite_setup/disconnect
  • function/f_printer.c、function/f_iap2.c 等通过 DECLARE_USB_FUNCTION_INIT 注册到 functions.c
  • 分层速览：Layer 3 Composite · Function 层 Layer 2
