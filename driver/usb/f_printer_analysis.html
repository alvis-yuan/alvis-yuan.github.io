f_printer.c 架构分析

USB Printer Class Gadget Function · 用户态交互 · Composite / ConfigFS 集成

1. 整体定位与分层

f_printer.c 实现 Linux USB Gadget 框架下的 Printer Class Function （接口类 USB_CLASS_PRINTER，子类 1，协议 2 双向）。 编译为独立模块 usb_f_printer.ko（CONFIG_USB_F_PRINTER）， 通过 DECLARE_USB_FUNCTION_INIT(printer, …) 向 Composite 层注册函数名 "printer"。

它在协议栈中处于「类功能驱动」层：向下对接 UDC 的 Bulk IN/OUT 端点，向上暴露字符设备供用户态打印服务读写。

2. 向上：用户态交互接口

Function 驱动在 printer_func_bind() 中完成字符设备注册，用户态通过标准文件 I/O 与 USB 打印数据通路交互。 这是与 legacy 单体驱动 g_printer.ko 相同的用户可见 ABI。

2.1 设备节点与 sysfs

接口创建时机说明
/dev/g_printerN printer_func_bind() N = minor（IDA 分配，最多 4 个）。cdev_add + device_create(usb_gadget_class, …, "g_printer%d")
sysfs class usb_printer_gadget gprinter_setup() 首次 alloc_inst class_create("usb_printer_gadget")，alloc_chrdev_region 分配主设备号

2.2 file_operations 语义

系统调用函数行为摘要
open() printer_open 单开（printer_cdev_open）；设置 PRINTER_SELECTED 状态位；需 interface 已激活
read() printer_read 从 Host 经 OUT 端点收到的打印作业数据拷贝到用户缓冲；支持阻塞 / 非阻塞；无数据时 wait_event(rx_wait)
write() printer_write 用户数据经 IN 端点发给 Host（状态回传、双向通信）；按 8KB 分片；必要时发 ZLP
poll() printer_poll EPOLLIN 当 rx 有数据；EPOLLOUT 当 tx 有空闲 req
fsync() printer_fsync 等待所有 in-flight TX 完成（tx_flush_wait）
ioctl() printer_ioctl 见下表
close() printer_close 清除 PRINTER_SELECTED；kref_put

2.3 ioctl 命令（include/uapi/linux/usb/g_printer.h）

GADGET_GET_PRINTER_STATUS  _IOR('g', 0x21, unsigned char)
GADGET_SET_PRINTER_STATUS  _IOWR('g', 0x22, unsigned char)

/* 状态位 */
PRINTER_NOT_ERROR   0x08
PRINTER_SELECTED    0x10   /* open() 时置位，供 Host GET_PORT_STATUS 查询 */
PRINTER_PAPER_EMPTY 0x20

Host 侧通过 USB 类请求 GET_PORT_STATUS 读取的是内核中 dev->printer_status； 用户态可通过 ioctl 读写同一字段，从而向 Host 报告「缺纸」等状态。

3. 数据通路（读 / 写）

3.1 接收路径（Host → 用户态 read）

  1. setup_rx_reqs() 从 rx_reqs 空闲链表取 req，usb_ep_queue(out_ep) 预提交接收。
  2. Host 发送 Bulk OUT 数据 → UDC 完成 → rx_complete()。
  3. 成功时将 req 挂到 rx_buffers（已完成、待用户读取），wake_up(rx_wait)。
  4. printer_read() 从 rx_buffers 取数据，copy_to_user()；若一次读不完，暂存于 current_rx_*。
  5. req 读完后归还 rx_reqs，再次 setup_rx_reqs() 继续接收。

3.2 发送路径（用户态 write → Host）

  1. 从 tx_reqs 取空闲 req，copy_from_user() 填入最多 USB_BUFSIZE（8192）字节。
  2. 挂入 tx_reqs_active，usb_ep_queue(in_ep)。
  3. tx_complete() 完成后归还 tx_reqs，唤醒 tx_wait / tx_flush_wait。

4. 融入 Gadget / Composite 框架

f_printer.c 不是独立 Gadget 驱动，而是可组合的 usb_function 插件。 模块加载时注册 function driver；Composite 层（ConfigFS 或 legacy）按需实例化并绑定到 configuration。

4.1 注册与模块入口

DECLARE_USB_FUNCTION_INIT(printer, gprinter_alloc_inst, gprinter_alloc);

/* 等价于：
 *   name = "printer"
 *   alloc_inst → 创建 f_printer_opts + ConfigFS 属性组
 *   alloc_func → 创建 printer_dev + 填充 usb_function 回调
 *   module_init → usb_function_register()
 */

4.2 两套核心对象

6. ConfigFS 配置示例

典型手工启用流程（需已加载 UDC 与 libcomposite）：

modprobe usb_f_printer          # 注册 function "printer"
modprobe libcomposite           # 若尚未加载

mkdir -p /config/usb_gadget/g1
cd /config/usb_gadget/g1

echo 0x0525 > idVendor          # 示例 VID（legacy 默认 NetChip）
echo 0xa4a8 > idProduct         # 示例 PID
mkdir -p strings/0x409
echo "serial" > strings/0x409/serialnumber

mkdir -p functions/printer.gs0
echo "MFG:linux;MDL:g_printer;CLS:PRINTER;SN:1;" > functions/printer.gs0/pnp_string
echo 10 > functions/printer.gs0/q_len

mkdir -p configs/c.1
ln -s functions/printer.gs0 configs/c.1/
echo "musb-hdrc.0" > UDC        # 替换为实际 UDC 名称

# 此时出现 /dev/g_printer0，用户态可 open 读写
与单体模块对比： g_printer.ko（legacy/printer.c）在模块参数里设置 PNP 串和 q_len， 内部同样调用 usb_get_function_instance("printer")。 f_printer.c 的优势是可与其他 function（如 ACM、Mass Storage）组成复合设备。

7. 核心数据结构与缓冲队列

struct printer_dev {
    spinlock_t       lock;           /* 保护链表、状态 */
    struct mutex     lock_printer_io; /* 序列化 read/write/poll */
    struct usb_gadget *gadget;
    struct usb_ep    *in_ep, *out_ep;

    /* RX: Host → 用户 */
    struct list_head rx_reqs;         /* 空闲，可再次 queue */
    struct list_head rx_reqs_active;  /* 已提交 UDC，等待完成 */
    struct list_head rx_buffers;      /* 已完成，待 read() 消费 */
    wait_queue_head_t rx_wait;

    /* TX: 用户 → Host */
    struct list_head tx_reqs;
    struct list_head tx_reqs_active;
    wait_queue_head_t tx_wait, tx_flush_wait;

    /* 部分 read 缓冲 */
    struct usb_request *current_rx_req;
    size_t current_rx_bytes;
    u8 *current_rx_buf;

    u8 printer_status;               /* Port status / ioctl 共享 */
    int interface;                     /* -1 = 未激活 */
    struct cdev printer_cdev;          /* 用户态入口 */
    struct usb_function function;      /* 嵌入 Composite */
};

同步策略：

  • spinlock_t lock — 保护链表与短临界区；usb_ep_queue 前会短暂解锁避免死锁。
  • mutex lock_printer_io — 同一时刻仅一个 read/write/poll 路径操作缓冲（open 仍用 spinlock 保护单开标志）。
  • kref — open 时 get，close 时 put，与 function 生命周期解耦。

8. 本仓库定制点

相对上游 Linux f_printer.c，本文件增加了全局变量 connect_status， 在 set_interface() 置 1、printer_func_disable() 置 0。 setup_rx_reqs()、printer_write() 仅在 connect_status 为真时才 usb_ep_queue()，否则返回 -ENODEV。

printer_write() 中当 TX 空闲链表为空时，上游会 wait_event(tx_wait) 阻塞等待； 本版本改为：若 tx_reqs_active 非空则对队尾 req 调用 usb_ep_dequeue() 尝试回收， 失败则直接返回 -EAGAIN。这改变了背压语义，用户态非阻塞写或 poll 行为可能与上游不同，集成时需注意。

gprinter_alloc() 末尾也将 connect_status = 0，与 disable 路径配合。
