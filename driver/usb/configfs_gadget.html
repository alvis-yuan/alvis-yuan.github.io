USB Gadget ConfigFS

基于 configfs.c · functions.c · f_*.c · Documentation/usb/gadget_configfs.rst

1. 概述与原理

ConfigFS 是一种用户态驱动内核对象配置的文件系统接口。 对 USB Gadget 而言，它把「设备由哪些配置（configuration）组成、每个配置包含哪些功能（function）」 这一决策从编译期/模块参数转移到运行时：用户在 VFS 中创建目录、写属性、建立符号链接， 最后写 UDC 属性触发与硬件控制器的绑定与 Host 枚举。

官方定义： Gadget 在 Host 眼中是一组 configuration；每个 configuration 包含若干 interface， 在 Gadget 侧称为 function（如串口、U 盘、打印机）。 详见 Linux USB gadget configured through configfs。

内核依赖关系：

  • CONFIG_USB_CONFIGFS → 选中 USB_LIBCOMPOSITE → 选中 CONFIGFS_FS
  • 实现代码在 libcomposite.ko 的 configfs.c 中，module_init(gadget_cfs_init) 注册子系统
  • 挂载点通常为 /sys/kernel/config（需 mount -t configfs none /sys/kernel/config）

2. ConfigFS 通用机制

ConfigFS 将内核对象暴露为目录（group/item）和文件（attribute）。 用户可创建/删除目录；文件只读或读写，对应 show/store 回调。 详见 Configfs — Userspace-driven Kernel Object Configuration。

3. 实现架构（configfs.c）

3.1 子系统注册

static struct configfs_subsystem gadget_subsys = {
    .su_group = {
        .cg_item = {
            .ci_namebuf = "usb_gadget",   /* → /config/usb_gadget/ */
            .ci_type = &gadgets_type,
        },
    },
};
/* gadget_cfs_init(): configfs_register_subsystem + android_usb class */

gadgets_make()：用户 mkdir usb_gadget/g1 时分配 struct gadget_info，并创建默认子组：

ConfigFS 目录内核类型make/drop 回调
functions/functions_typefunction_make / function_drop
configs/config_desc_typeconfig_desc_make / config_desc_drop
strings/gadget_strings_strings_type按语言 ID 创建字符串组
os_desc/os_desc_typeWindows OS 描述符扩展

3.2 核心数据结构

struct gadget_info {
    struct config_group group;          /* g1 根目录 */
    struct config_group functions_group, configs_group, strings_group, os_desc_group;
    struct mutex lock;
    struct list_head available_func;    /* 已创建的 function instance */
    struct usb_composite_driver composite;
    struct usb_composite_dev cdev;      /* 嵌入 composite 运行时 */
    bool use_os_desc;
    spinlock_t spinlock;
    bool connected;                     /* Host 连接状态 */
};

struct config_usb_cfg {
    struct config_group group;
    struct usb_configuration c;
    struct list_head func_list;         /* symlink 后暂存的 usb_function */
};

3.3 Function 创建与关联

创建 — function_make()（mkdir functions/printer.0）：

  1. 解析名称 FUNC.INSTANCE（必须含 .）
  2. usb_get_function_instance("printer") → functions.c 查找 / request_module("usbfunc:printer")
  3. 加入 gi->available_func 链表

关联 — config_usb_cfg_link()（ln -s functions/printer.0 configs/c.1/）：

  1. 校验 fi 属于本 gadget 的 available_func
  2. usb_get_function(fi) → 调用各模块 alloc_func()
  3. function 暂存于 cfg->func_list（尚未 bind 到 UDC）

解除 — config_usb_cfg_unlink()：若已绑定 UDC，先 unregister_gadget()，再 usb_put_function()。

3.4 UDC 绑定触发

/* gadget_dev_desc_UDC_store() */
echo "" > UDC          → unregister_gadget() → usb_gadget_unregister_driver()
echo "my_udc" > UDC    → usb_gadget_probe_driver(&gi->composite.gadget_driver)
                         → configfs_composite_bind()
                         → composite_dev_prepare + 遍历 usb_add_function

configfs_driver_template 使用 match_existing_only = 1，必须指定已存在的 UDC 名称（见 /sys/class/udc/）。

4. ConfigFS 目录树

/sys/kernel/config/usb_gadget/g1/
├── idVendor          idProduct          bcdDevice
├── bDeviceClass      bDeviceSubClass    bDeviceProtocol
├── bMaxPacketSize0   bcdUSB             max_speed
├── UDC               ← 写 UDC 名启用 / 空字符串禁用
├── strings/
│   └── 0x409/
│       ├── manufacturer    product    serialnumber
├── configs/
│   └── c.1/                    # 名称.配置编号
│       ├── MaxPower            bmAttributes
│       ├── strings/0x409/configuration
│       └── printer.0 → ../../functions/printer.0
├── functions/
│   └── printer.0/              # 函数名.实例名
│       ├── pnp_string          q_len   (printer 特有)
│       └── os_desc/…           (部分 function 提供)
└── os_desc/
    ├── use    b_vendor_code    qw_sign
    └── c.1 → ../configs/c.1    (OS 描述符关联配置)

属性完整列表见内核 ABI： Documentation/ABI/testing/configfs-usb-gadget 及 configfs-usb-gadget-* 各 function 文件。

5. 使用指南

5.1 前置条件

modprobe libcomposite
# 按需加载 function 模块，或依赖 auto modprobe：
# modprobe usb_f_printer

mount -t configfs none /sys/kernel/config   # 若尚未挂载
ls /sys/class/udc/                          # 确认 UDC 名称

5.2 示例：Printer Gadget

G=/sys/kernel/config/usb_gadget/g1
UDC=$(ls /sys/class/udc | head -1)

mkdir -p $G
cd $G
echo 0x0525 > idVendor      # NetChip (示例)
echo 0xa4a7 > idProduct
echo 0x0200 > bcdUSB
echo 0x0100 > bcdDevice
echo 0x00 > bDeviceClass

mkdir -p strings/0x409
echo "1234567890"   > strings/0x409/serialnumber
echo "My Corp"      > strings/0x409/manufacturer
echo "USB Printer"  > strings/0x409/product

mkdir -p functions/printer.0
echo "MDL:Generic Printer;" > functions/printer.0/pnp_string
echo 10 > functions/printer.0/q_len

mkdir -p configs/c.1/strings/0x409
echo "Printer Config" > configs/c.1/strings/0x409/configuration
echo 250 > configs/c.1/MaxPower
ln -sf $G/functions/printer.0 configs/c.1/

echo $UDC > UDC             # 启用；Host 开始枚举
# 用户态：open /dev/g_printer0

Printer function 属性见 configfs-usb-gadget-printer （本树 Documentation/ABI/testing/configfs-usb-gadget-printer）：pnp_string、q_len。

5.3 禁用与清理

echo "" > UDC
rm configs/c.1/printer.0
rmdir configs/c.1/strings/0x409 configs/c.1
rmdir functions/printer.0
rmdir strings/0x409
cd .. && rmdir g1
注意：禁用 gadget 后 function 的 .ko 模块不会自动卸载（官方文档明确说明）。 修改配置前应先 echo "" > UDC；unlink function 时若仍绑定，configfs 会强制 unbind。

5.4 多功能复合设备

同一 configuration 可 symlink 多个 function；也可创建 c.1、c.2 多配置。每个 function 实例名唯一（如 acm.0、printer.0）。

7. Function 侧的 ConfigFS（f_*.c）

每个 Function 模块在 alloc_instance() 中初始化自己的 config_group 与属性。 以 f_printer.c 为例：

  • f_printer_opts 嵌入 struct usb_function_instance func_inst
  • printer_func_type 注册 pnp_string、q_len 的 show/store
  • q_len 在 instance 已被引用（refcnt）时写入返回 -EBUSY

Function 名与模块别名对应关系（DECLARE_USB_FUNCTION_INIT）：

8. 调试

8.1 排查清单

现象检查点
UDC store 失败dmesg：UDC 名是否正确、是否已被占用；ls /sys/class/udc/
mkdir functions/xxx 失败模块是否加载：modprobe usbfunc:xxx；名称格式必须为 func.inst
bind 报错 “Need at least one configuration”是否创建 configs/c.N 且 symlink 至少一个 function
bind 报错 “needs at least one function”configuration 内是否有 symlink 到 function
Host 不枚举UDC 驱动是否 probe；usb_gadget_connect；电缆/Role
枚举但无 /dev 节点Function 的 bind/set_alt 是否成功；查看 function 驱动日志
修改 max_speed 失败绑定 UDC 后不可改（gadget_dev_desc_max_speed_store 检查 udc_name）

8.2 工具与接口

# UDC 列表与状态
ls -la /sys/class/udc/
cat /sys/class/udc//state    # 部分 UDC 驱动提供

# ConfigFS 当前绑定
cat /sys/kernel/config/usb_gadget/g1/UDC

# 本树扩展：Android 兼容 state  sysfs（configfs.c android_device_create）
cat /sys/class/android_usb/android0/state
# 可能值：DISCONNECTED / CONNECTED / CONFIGURED

# 内核日志
dmesg -w | grep -iE 'composite|configfs|udc|gadget|printer'

# 可选 debugfs（需 CONFIG_USB_GADGET_DEBUG_FS）
ls /sys/kernel/debug/usb/

# 可选 /proc（需 CONFIG_USB_GADGET_DEBUG_FILES）
cat /proc/driver/udc

8.3 相关 Kconfig 调试选项

  • CONFIG_USB_GADGET_DEBUG / VERBOSE — 驱动 printk 调试信息
  • CONFIG_USB_GADGET_DEBUG_FS — debugfs 节点
  • CONFIG_USB_GADGET_DEBUG_FILES — /proc/driver/udc
  • CONFIG_CONFIGFS_UEVENT_REPORT — 本树 uevent（USB_STATE=CONNECTED 等）

8.4 无硬件调试

使用 dummy_hcd 虚拟 UDC（见 gadget-testing.rst）：

modprobe dummy_hcd
echo dummy_udc.0 > /sys/kernel/config/usb_gadget/g1/UDC

9. 本仓库定制点

configfs.c 在本树中包含 Android 相关扩展：

  • android_class / android_device — 创建 /sys/class/android_usb/android0
  • state 属性 — 反映 DISCONNECTED / CONNECTED / CONFIGURED
  • CONFIG_CONFIGFS_UEVENT_REPORT — 通过 uevent 上报 USB 状态（兼容旧 Android 方案）
  • configfs_composite_setup/disconnect — 包装 composite_setup 并更新连接状态
