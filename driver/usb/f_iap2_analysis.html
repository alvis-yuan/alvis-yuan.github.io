f_iap2.c 架构分析

USB iAP2 Gadget Function · MFi 数据通道 · 用户态交互 · Composite / ConfigFS 集成

1. 整体定位与分层

f_iap2.c 实现 Linux USB Gadget 框架下的 iAP2 数据通道 Function， 编译为 usb_f_iap2.ko（CONFIG_USB_F_IAP2）， 向 Composite 注册函数名 "iap2"。

该驱动不解析 iAP2 协议帧（认证、EA 会话、Link 协商等）， 仅提供与 f_hid / f_printer 类似的 Bulk 双向字节流管道： 用户态 MFi / iAP2 协议栈通过字符设备读写，内核负责 USB 端点调度与缓冲。

2. USB 描述符与 iAP2 角色

2.1 接口描述符（Vendor Specific）

/* 当前启用 */
.bInterfaceClass    = 0xFF   /* Vendor Specific */
.bInterfaceSubClass = 0xF0   /* Apple MFi 常用子类 */
.bInterfaceProtocol = 0x00

/* 注释掉的备选（与 Printer 相同布局，未使用） */
/* .bInterfaceClass = 7, .bInterfaceSubClass = 1, .bInterfaceProtocol = 2 */

.iInterface = "iAP Interface"   /* usb_string */

端点为一对 Bulk IN + Bulk OUT，缓冲大小 USB_BUFSIZE = 8192， 支持 Full / High / Super Speed 三套描述符（512 / 1024 max packet）。

协议边界： Apple iAP2 规范中的 Link 同步、检测消息、EA 会话、External Accessory Protocol 等 均在本驱动之上的用户态实现。 内核只保证 Bulk 字节可靠（尽力）传输；MFi 认证芯片、序列号、Bundle ID 等也不在此文件内。

2.2 方向约定（Gadget 视角）

端点用户态 API数据方向典型含义
out_ep (Bulk OUT)read()iPhone → 配件Host 下发的 iAP2 帧 / 控制数据
in_ep (Bulk IN)write()配件 → iPhone配件回复的 iAP2 帧 / 响应

3. 向上：用户态交互接口

3.1 设备节点

接口创建时机说明
/dev/iap2N iap2_bind() N = minor（IDA，最多 4）。device_create(iap2_class, …, "iap2%d")
sysfs class iap2 iap2_mod_init() class_create("iap2") + alloc_chrdev_region(..., "iap2")

3.2 file_operations

系统调用函数行为摘要
open() f_iap2_open 单开（cdev_open）；必须 enabled==true（Host 已 set_alt）否则 -ENODEV
read() f_iap2_read 从 completed_out_req 取 Host OUT 数据；支持 partial read（f_iap2_req_list.pos）；读完后 re-queue OUT req
write() f_iap2_write 经 IN 端点发往 iPhone；8KB 分片；ZLP 处理；无空闲 TX 时可 dequeue 或阻塞
poll() f_iap2_poll EPOLLIN 当 completed_out 非空；EPOLLOUT 当 tx_reqs 非空；未 enabled 返回 ERR/HUP
fsync() f_iap2_fsync 等待 tx_reqs_active 清空
close() f_iap2_release 清除 cdev_open；kref_put
与 f_printer 的重要区别：无 ioctl，无 GADGET_GET/SET_PRINTER_STATUS 类控制接口。 端口状态、协议状态全部由用户态 iAP2 栈自行维护。

4. 数据通路（读 / 写）

4.1 接收路径（iPhone → read）

  1. iap2_set_alt() 成功后在 OUT 端点预提交 qlen 个 8KB 接收 req（iap2_set_report_complete）。
  2. Host 发送 Bulk OUT → complete 回调分配 f_iap2_req_list，挂入 completed_out_req，wake_up(read_queue)。
  3. f_iap2_read() 取队首 list 项，按 pos 偏移 copy_to_user；支持一次 read 只消费部分数据。
  4. 当 pos == req->actual 时，删除 list 包装，将 req 重新 usb_ep_queue(out_ep) 继续接收。

4.2 发送路径（write → iPhone）

  1. iap2_bind() 预分配 qlen 个 TX req 放入 tx_reqs 空闲链表。
  2. f_iap2_write() 取 req → copy_from_user → 挂 tx_reqs_active → usb_ep_queue(in_ep)。
  3. f_iap2_req_complete() 完成后归还 tx_reqs，唤醒 write_queue / tx_flush_wait。

8. 核心数据结构与同步

struct f_iap2_req_list {
    struct usb_request *req;
    unsigned int        pos;    /* partial read 偏移 */
    struct list_head    list;
};

struct f_iap2 {
    struct list_head    completed_out_req;  /* RX：已完成，待 read */
    spinlock_t          spinlock;
    wait_queue_head_t   read_queue;

    struct mutex        lock;               /* 序列化 write */
    bool                enabled;            /* set_alt 后置 true */
    struct list_head    tx_reqs;
    struct list_head    tx_reqs_active;
    wait_queue_head_t   write_queue;
    wait_queue_head_t   tx_flush_wait;

    int minor;
    struct cdev cdev;
    struct kref kref;
    bool cdev_open;
    struct usb_function func;
    struct usb_ep *in_ep, *out_ep;
    unsigned int qlen;
};

10. 本仓库定制点

  • connect_status 全局变量：与 f_printer 相同，set_alt 成功置 1，disable / alloc 置 0；read re-queue 与 write queue 均检查。
  • write 背压：TX 空闲链表空时，若 connect_status 且 tx_reqs_active 非空，对队尾 req 调用 usb_ep_dequeue，否则阻塞在 write_queue（与 f_printer 类似定制）。
  • Vendor 描述符：使用 0xFF/0xF0 而非注释中的 Printer Class，适配 MFi iAP Interface 枚举。
  • open 日志：printk("open usb mfi device success\n") 调试输出。
  • Remote wakeup：宏 ENABLE_WAKEUP_SUSPEND_HOST 默认关闭；开启后 suspend 时置 flag，write 路径调用 usb_gadget_wakeup() 唤醒 Host。
  • iap2_set_report_complete：若 kzalloc(req_list) 失败则静默丢包（req 未 re-queue），高负载下需注意。
