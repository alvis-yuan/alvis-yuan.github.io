01

总览与第一性原理分析

从第一性原理出发：串口（Serial Port / UART）的本质是将并行数据转化为时序化的单线位流进行传输。处理器内部以字节/字为单位并行操作数据，而长距离传输需要极少的物理线路。UART 解决了这一矛盾——用时间换空间，用时钟精度换取通信线路数量。

图1：从通信需求到Linux驱动的第一性原理推导
🎯 核心矛盾

处理器内部并行数据总线 vs 远距离传输需要的最少导线数目

⚙️ 解决机制

移位寄存器将并行字节转为串行位流，双方约定波特率代替共享时钟

📐 帧同步策略

起始位（低电平）触发采样，停止位（高电平）复位，奇偶位检验完整性

🔌 物理层扩展

UART仅定义数字逻辑，RS-232/RS-485/RS-422定义了电气特性和总线拓扑

物理特性

📖 官方来源：kernel.org RS485 Serial Communications

2.1 信号帧格式

UART帧由起始位、数据位、可选奇偶校验位和停止位构成。发送方与接收方必须预先约定波特率（符号率），不共享时钟线。

图2：UART帧格式（8N1: 8数据位，无奇偶校验，1停止位）

2.2 物理电气标准对比

UART仅定义逻辑层。RS-232、RS-485、RS-422等标准定义了电气特性，决定了传输距离、抗干扰能力和连接设备数量。

标准 模式 逻辑"1"电平 逻辑"0"电平 最大距离 最大设备数 最大速率
TTL UART 单端 +3.3V / +5V 0V ~1m 点对点 ~4 Mbps
RS-232 单端 -3V ~ -15V (MARK) +3V ~ +15V (SPACE) ~15m 点对点 ~20 Kbps
RS-485 差分 A-B > +200mV A-B < -200mV ~1200m 32~256节点 10 Mbps
RS-422 差分 A-B > +200mV A-B < -200mV ~1200m 1发10收 10 Mbps
注意（来自embeddedsecurity.io）：UART 是协议层，不定义物理层。TTL UART 中正电压为逻辑"1"；RS-232 反转——负电压为逻辑"1"（MARK），正电压为逻辑"0"（SPACE）。RS-485是差分信号，A-B > +200mV 为逻辑"1"，极强的共模抑制使其适合工业噪声环境。

2.4 常见波特率

波特率(bps)每位时间(μs)典型应用场景
9600104.2调试、低速传感器、Modbus
1152008.68嵌入式调试串口（最常见）
2304004.34较高速设备通信
9216001.08高速模块（蓝牙/WiFi模组）
30000000.33高性能UART（如RPi与WiFi芯片）
波特率计算原理：UART波特率生成器内部分频公式为：Divisor = UART_CLK / (16 × BaudRate)。例如1.8432 MHz晶振，波特率115200，则 Divisor = 1843200/(16×115200) = 1（整除，无误差）。这也是嵌入式系统选择1.8432 MHz晶振的原因。

偏移DLAB名称缩写功能
0x000(写)发送保持寄存器THR待发送数据写入此寄存器，装入TX FIFO

0x03-线路控制寄存器LCR数据位长、停止位数、奇偶校验、DLAB位

3.4 FIFO 工作机制

16550相比16450的核心改进是加入了16字节深度的TX/RX FIFO，显著减少CPU中断频率。

  • RX FIFO触发阈值：可配置为1、4、8、14字节，达到阈值触发RX中断
  • RX超时中断：数据低于触发阈值时，等待4个字符时间后触发超时中断，防止少量数据被遗留
  • TX FIFO空中断：TX FIFO完全为空时触发，驱动继续填充数据
  • FIFO溢出：RX FIFO满且新数据到来，LSR[OE]置位，数据丢失

Linux串口驱动框架总览

📖 官方来源：kernel.org Low Level Serial API 📖 官方来源：kernel.org TTY Layer

Linux串口驱动采用三层架构。内核官方文档（Documentation/driver-api/serial/driver.rst）明确指出："The low level serial hardware driver is responsible for supplying port information (defined by uart_port) and a set of control methods (defined by uart_ops) to the core serial driver."

图9：Linux串口驱动三层架构（用户空间→TTY层→Serial Core→低层驱动→硬件）

5.1 驱动注册流程

根据官方文档，驱动注册分两步：先注册驱动，再逐个注册端口：

/* Step 1: 定义并注册 uart_driver */
static struct uart_driver my_uart_driver = {
    .owner       = THIS_MODULE,
    .driver_name = "my_uart",
    .dev_name    = "ttyMY",
    .major       = 0,   /* 自动分配 major */
    .minor       = 0,
    .nr          = MY_UART_NR_PORTS,
    .cons        = &my_console,
};

/* module_init 中 */
uart_register_driver(&my_uart_driver);

/* Step 2: probe 时注册每个端口 */
uart_add_one_port(&my_uart_driver, &my_port.port);

/* module_exit 中逆序注销 */
uart_remove_one_port(&my_uart_driver, &my_port.port);
uart_unregister_driver(&my_uart_driver);

关键数据结构

📖 官方来源：kernel.org v6.1 Serial Driver API

6.1 struct uart_driver

代表一类串口设备的驱动，注册后创建 /proc/tty/driver/<name> 条目。

struct uart_driver {
    struct module      *owner;
    const char         *driver_name;  /* 驱动名（日志/proc中显示）*/
    const char         *dev_name;     /* 设备节点前缀，如 "ttyS" */
    int                 major;        /* 主设备号，0=自动分配 */
    int                 minor;
    int                 nr;           /* 端口总数 */
    struct console     *cons;        /* 串口控制台，可为NULL */
    /* 以下为内核内部使用 */
    struct uart_state  *state;
    struct tty_driver  *tty_driver;
};

6.2 struct uart_port

描述单个UART端口的所有属性，是低层驱动与Serial Core之间最核心的数据结构。驱动通常将其嵌入自定义结构体：

/* 嵌套扩展惯用法（来自官方文档）*/
struct my_port {
    struct uart_port  port;   /* 必须是第一个成员 */
    int               my_custom_field;
    void __iomem     *base;
};

/* uart_port 关键字段 */
struct uart_port {
    spinlock_t         lock;        /* 端口锁，保护 mctrl/icount/xmit */
    unsigned long      iobase;      /* I/O 基地址（PIO） */
    unsigned char __iomem *membase; /* MMIO 虚拟基地址 */
    unsigned int       irq;         /* 中断号 */
    unsigned long      irqflags;    /* 中断标志 */
    unsigned int       uartclk;     /* UART 时钟频率（Hz） */
    unsigned int       fifosize;    /* TX FIFO 深度 */
    unsigned int       line;        /* 端口编号（从0开始）*/
    unsigned int       mctrl;       /* 当前 modem 控制状态 */
    struct uart_icount icount;      /* 统计计数（tx/rx/error）*/
    const struct uart_ops *ops;    /* 操作函数集（必须实现）*/
    struct serial_rs485 rs485;      /* RS485 配置 */
    struct device     *dev;
};

6.3 锁机制（来自官方文档）

官方说明（kernel.org）：存在两种锁：
1. port->lock（per-port spinlock）：保护 port->mctrl、port->icount、port->state->xmit.head/tail（环形缓冲区）
2. port_sem（tty_port->mutex）：防止端口在不合时宜的时候被添加/删除/重配置。自v2.6.27起，此信号量就是 tty_port 结构体的 mutex 成员，通常称为"port mutex"。

6.4 struct uart_icount 统计字段

字段含义对应 /proc/tty/driver/ 列

rx已接收字节数rx
frame帧错误次数fe

uart_ops 回调接口详解

📖 官方来源：kernel.org v6.1 uart_ops定义

官方文档原文："The uart_ops structure is the main interface between serial_core and the hardware specific driver. It contains all the methods to control the hardware."

struct uart_ops {
    unsigned int  (*tx_empty)(struct uart_port *);
    void          (*set_mctrl)(struct uart_port *, unsigned int mctrl);
    unsigned int  (*get_mctrl)(struct uart_port *);
    void          (*stop_tx)(struct uart_port *);
    void          (*start_tx)(struct uart_port *);
    void          (*throttle)(struct uart_port *);
    void          (*unthrottle)(struct uart_port *);
    void          (*send_xchar)(struct uart_port *, char ch);
    void          (*stop_rx)(struct uart_port *);
    void          (*enable_ms)(struct uart_port *);
    void          (*break_ctl)(struct uart_port *, int ctl);
    int           (*startup)(struct uart_port *);
    void          (*shutdown)(struct uart_port *);
    void          (*flush_buffer)(struct uart_port *);
    void          (*set_termios)(struct uart_port *, struct ktermios *new,
                                  const struct ktermios *old);
    void          (*pm)(struct uart_port *, unsigned int state, unsigned int old);
    const char   *(*type)(struct uart_port *);
    void          (*release_port)(struct uart_port *);
    int           (*request_port)(struct uart_port *);
    void          (*config_port)(struct uart_port *, int);
    int           (*verify_port)(struct uart_port *, struct serial_struct *);
    int           (*ioctl)(struct uart_port *, unsigned int, unsigned long);
};
回调调用时机锁状态关键说明（来自官方文档）
tx_empty 查询TX FIFO/移位寄存器是否为空 无锁 空返回TIOCSER_TEMT，否则返回0。不可睡眠
set_mctrl 设置RTS/DTR/OUT1/OUT2/LOOP port->lock held 局部中断禁用，不可睡眠
get_mctrl 读取DCD/CTS/DSR/RI状态 port->lock held 只返回输入信号，不可睡眠
stop_tx CTS失效或XOFF时停止发送 port->lock held 尽快停止，局部中断禁用，不可睡眠
start_tx 有数据需要发送时启动TX port->lock held 局部中断禁用，不可睡眠
startup 端口首次打开时 port_sem held 申请IRQ、初始化硬件；禁止在此激活RTS/DTR
shutdown 端口最后一次关闭时 port_sem held 释放IRQ，完成后不得访问port->state
set_termios 参数变更（波特率/数据位/校验） tty_port->mutex held 更新read_status_mask和ignore_status_mask，不可睡眠
pm 电源状态变更（suspend/resume） 无锁 控制台端口也会调用，即使未启用CONFIG_PM
config_port 自动配置端口类型/IRQ 无锁 UART_CONFIG_TYPE：探测并设置port->type；UART_CONFIG_IRQ：探测IRQ
flush_buffer TX环形缓冲区被清空时 port->lock held 清除DMA状态，停止DMA传输，不可睡眠

6.2 set_termios 中 cflag/iflag 处理

官方文档明确列出驱动必须处理的 termios 位：

c_cflag 位
  • CSIZE：数据位宽（CS5~CS8）
  • CSTOPB：2个停止位
  • PARENB：使能奇偶校验
  • PARODD：奇校验（PARENB置位时）
  • CREAD：使能接收
  • CRTSCTS：硬件流控（RTS/CTS）
  • CLOCAL：忽略Modem状态变化
c_iflag 位
  • INPCK：帧错误/奇偶错误上报给TTY
  • BRKINT/PARMRK：BREAK事件上报
  • IGNPAR：忽略奇偶/帧错误
  • IGNBRK：忽略BREAK（若IGNPAR也设置，则忽略溢出）

RS485 / RS422 内核支持

📖 官方来源：kernel.org RS485 Serial Communications

官方文档原文："EIA-485, also known as TIA/EIA-485 or RS-485, is a standard defining the electrical characteristics of drivers and receivers for use in balanced digital multipoint systems. This standard is widely used for communications in industrial automation because it can be used effectively over long distances and in electrically noisy environments."

7.1 struct serial_rs485 结构体

struct serial_rs485 {
    __u32 flags;
/* 标志位定义（官方kernel.org）*/
#define SER_RS485_ENABLED          _BITUL(0)  /* 使能RS485模式 */
#define SER_RS485_RTS_ON_SEND      _BITUL(1)  /* 发送时RTS=1 */
#define SER_RS485_RTS_AFTER_SEND   _BITUL(2)  /* 发送后RTS=1 */
#define SER_RS485_RX_DURING_TX     _BITUL(4)  /* 发送时也接收（全双工RS485）*/
#define SER_RS485_TERMINATE_BUS    _BITUL(5)  /* 使能总线终端电阻 */
#define SER_RS485_ADDRB            _BITUL(6)  /* 多点寻址模式 */
#define SER_RS485_ADDR_RECV        _BITUL(7)  /* 使能接收地址过滤 */
#define SER_RS485_ADDR_DEST        _BITUL(8)  /* 设置目标地址 */
#define SER_RS485_MODE_RS422       _BITUL(9)  /* RS422模式（需SER_RS485_ENABLED）*/
    __u32 delay_rts_before_send;  /* RTS置位到发送前延迟（ms）*/
    __u32 delay_rts_after_send;   /* 发送完成后RTS复位延迟（ms）*/
};

7.2 用户空间配置RS485（官方示例）

#include <linux/serial.h>
#include <sys/ioctl.h>

int fd = open("/dev/mydevice", O_RDWR);
struct serial_rs485 rs485conf = {0};

/* 使能RS485模式 */
rs485conf.flags |= SER_RS485_ENABLED;

/* 发送时RTS=1，发送后RTS=0 */
rs485conf.flags |= SER_RS485_RTS_ON_SEND;
rs485conf.flags &= ~(SER_RS485_RTS_AFTER_SEND);

/* 发送前延迟1ms，发送后延迟2ms */
rs485conf.delay_rts_before_send = 1;
rs485conf.delay_rts_after_send  = 2;

ioctl(fd, TIOCSRS485, &rs485conf);  /* 设置 */
ioctl(fd, TIOCGRS485, &rs485conf);  /* 读回 */

7.3 驱动侧 RS485 实现要点

官方要求：任何支持RS485的驱动必须在 struct uart_port 中实现 rs485_config 回调并提供 rs485_supported 字段。serial core 在响应 TIOCSRS485 ioctl 时调用 rs485_config；TIOCGRS485 可读回当前配置。驱动还需调用 uart_get_rs485_mode() 以从设备树读取RS485启动参数。

驱动开发关键要点

8.1 中断处理流程

正确的中断服务程序是串口驱动稳定性的核心。以下是标准模式：

static irqreturn_t my_uart_irq(int irq, void *dev_id)
{
    struct uart_port *port = dev_id;
    unsigned int status = read_status_reg(port);

    spin_lock(&port->lock);

    /* 处理 RX：从硬件读数据存入 tty flip buffer */
    if (status & RX_DATA_READY) {
        unsigned char ch = read_rx_reg(port);
        uart_insert_char(port, status, MY_OE_BIT, ch, TTY_NORMAL);
        port->icount.rx++;
    }

    /* 处理 TX：从 xmit 环形缓冲区取数据写入硬件 */
    if (status & TX_FIFO_EMPTY) {
        struct circ_buf *xmit = &port->state->xmit;
        if (!uart_circ_empty(xmit)) {
            write_tx_reg(port, xmit->buf[xmit->tail]);
            xmit->tail = (xmit->tail + 1) & (UART_XMIT_SIZE - 1);
            port->icount.tx++;
        }
        if (uart_circ_chars_pending(xmit) < WAKEUP_CHARS)
            uart_write_wakeup(port);  /* 通知上层继续填充 */
    }

    spin_unlock(&port->lock);
    tty_flip_buffer_push(&port->state->port);
    return IRQ_HANDLED;
}

8.2 波特率计算

驱动中使用内核提供的辅助函数，避免手动计算：

static void my_set_termios(struct uart_port *port,
                             struct ktermios *termios,
                             const struct ktermios *old)
{
    unsigned int baud, divisor;

    /* 1. 用 serial_core 辅助函数获取合法波特率 */
    baud = uart_get_baud_rate(port, termios, old,
                               port->uartclk / 16, port->uartclk);

    /* 2. 计算分频值：divisor = baud_base / baud */
    divisor = uart_get_divisor(port, baud);

    /* 3. 更新超时（FIFO drain时间）*/
    uart_update_timeout(port, termios->c_cflag, baud);

    /* 4. 更新错误掩码 */
    port->read_status_mask  = MY_OE | MY_DR;
    if (termios->c_iflag & INPCK)
        port->read_status_mask |= MY_FE | MY_PE;

    /* 5. 写硬件寄存器 */
    write_divisor(port, divisor);
}

8.3 DMA支持要点

⚠️ flush_buffer

TX环形缓冲区被清空时内核调用此接口。驱动必须：停止正在进行的DMA传输、重置DMA状态、清空TX FIFO。

🔒 锁与DMA

DMA完成回调中访问 xmit 缓冲区需持 port->lock。DMA映射应在 startup() 时申请，shutdown() 时释放。

🔧 GPIO Modem控制

内核提供 mctrl_gpio_init() 从设备树获取CTS/RTS等GPIO并自动设置IRQ，无需驱动手动处理GPIO变化中断。

🖥️ 早期控制台

实现 console_write/console_setup，使用 uart_console_write() 辅助函数，调试阶段内核启动日志即通过此接口输出。

8.4 设备树配置示例

/* arch/arm64/boot/dts/... */
uart0: serial@12340000 {
    compatible = "vendor,my-uart";
    reg = <0x12340000 0x1000>;

oe（overrun error）：接收溢出计数，是FIFO溢出的直接证据，频繁出现说明CPU未能及时读取数据，需检查中断延迟或使用DMA。

9.2 stty — 查询与设置串口参数

# 查看当前参数（完整模式）
stty -F /dev/ttyS0 -a

# 设置 115200 8N1，无流控
stty -F /dev/ttyS0 115200 cs8 -cstopb -parenb -crtscts

# 设置原始模式（raw mode，无行规处理）
stty -F /dev/ttyS0 raw

# 常见参数说明:
# cs8      → 8数据位
# -cstopb  → 1停止位（加-号表示关闭）
# -parenb  → 无奇偶校验
# crtscts  → 使能RTS/CTS硬件流控
# -ixon    → 禁用XON/XOFF软件流控

9.3 minicom / picocom — 终端仿真器

# picocom（轻量，嵌入式常用）
picocom -b 115200 --databits 8 --parity n --stop 1 --flow n /dev/ttyS0

# minicom（功能丰富）
minicom -b 115200 -o -D /dev/ttyS0

# screen（快速测试）
screen /dev/ttyS0 115200

# 快速发送测试数据
echo -ne "Hello UART\r\n" > /dev/ttyS0

# 接收数据
cat /dev/ttyS0

9.4 setserial — 串口硬件参数配置

# 查看端口硬件信息
setserial -g /dev/ttyS0

# 设置低延迟模式（减少中断到数据可读的延迟）
setserial /dev/ttyS0 low_latency

# 设置自定义分频（用于非标准波特率）
setserial /dev/ttyS0 spd_cust divisor 10

9.5 dmesg — 驱动初始化日志

# 过滤串口相关内核日志
dmesg | grep -i "ttyS\|uart\|serial"

# 典型成功输出:
# [    1.234567] 12340000.serial: ttyMY0 at MMIO 0x12340000 (irq = 45, ...
# [    1.235000] console [ttyS0] enabled

9.6 /sys/class/tty — sysfs 接口

# 查看所有 tty 设备
ls /sys/class/tty/

# 查看串口的 udev 属性
udevadm info -a -n /dev/ttyS0

# 查看串口驱动目录
find /sys -type d -name '*serial*'

9.7 逻辑分析仪 + sigrok（硬件调试）

# 使用 sigrok-cli 解码UART（需逻辑分析仪硬件）
sigrok-cli -d fx2lafw -c samplerate=1m \
  --protocol-decoders uart \
  --protocol-decoder-options uart:baudrate=115200:format=ascii

console=ttyS0,115200 kgdboc=ttyS0,115200 kgdbwait

# 主机端 gdb 连接

常见问题排查

现象可能原因排查方法
无数据接收 TXD/RXD未交叉连接；波特率不匹配；驱动未加载 检查连线；dmesg确认驱动；stty对比两端参数
数据乱码 波特率、数据位、停止位、奇偶校验不匹配 两端用 stty -a 对比所有参数
数据丢失（oe计数增加） RX FIFO溢出，CPU处理不及时 cat /proc/tty/driver/serial查看oe计数；使用DMA；降低波特率
RS485发送无响应 DE/RE方向控制错误；RTS极性设置错误 检查 SER_RS485_RTS_ON_SEND 标志；示波器测量DE引脚
流控死锁 CTS/RTS接线错误或未接；软件流控与硬件流控冲突 stty -crtscts 先禁用流控测试；检查MCR/MSR寄存器
高延迟 线路规程缓冲；low_latency未设置 setserial /dev/ttyS0 low_latency；检查中断合并设置
驱动probe失败 设备树compatible不匹配；时钟/IRQ未配置 dmesg查看错误；检查DTS compatible字符串
