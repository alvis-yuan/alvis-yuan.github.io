00

架构总览：从用户态到 Flash 硬件

在 Linux 嵌入式系统中，用户态程序对 Flash 的每一次读写请求，都要经历多个软件层的转换、翻译与保护。与普通块设备（如 eMMC）不同，裸 Flash（Raw Flash）没有内置 FTL（Flash 转换层），操作系统必须直接管理擦除块、坏块、磨损均衡和 ECC 纠错。

Linux 为此提供了一套完整的软件栈——MTD（Memory Technology Device）子系统，它是连接用户空间与 Flash 物理硬件的核心桥梁。

📌 官方来源
MTD 子系统由 Linux 内核维护，官方文档位于： linux-mtd.infradead.org kernel.org MTD API kernel.org UBIFS
⚡ 第一性原理

Flash 不是磁盘。它不能原地改写，只能先擦再写；擦写次数有限，坏块可能增多。因此 Linux 必须设计一个专用子系统（MTD）来屏蔽这些物理限制，让上层软件像操作普通文件一样操作 Flash，同时在底层默默完成坏块管理、磨损均衡和 ECC 纠错。整个软件栈的存在，本质上是对 Flash 物理约束的抽象与补偿。

01

用户应用 & 系统调用入口

用户态程序通过 POSIX 标准 C 函数（open、read、write、ioctl）与 Flash 交互。这些函数由 glibc/musl 封装为系统调用指令（syscall），触发 CPU 从用户模式切换到内核模式（trap），进入内核的系统调用入口。

两种访问路径

访问方式设备节点说明
open/read/write/dev/mtd0MTD 字符设备，支持任意大小读，写必须对齐最小写单元
ioctl(MEMERASE)/dev/mtd0擦除操作必须用 ioctl，不能用 write 直接擦
挂载文件系统/dev/mtdblock0MTD 块设备，供 ext2 等块文件系统使用（不推荐裸 NAND）
UBIFS 挂载ubi0:rootfs经过 UBI 层的卷名访问，最推荐的 NAND 使用方式

系统调用过程（以 read 为例）

# 用户态 C 代码
int fd = open("/dev/mtd0", O_RDONLY);
read(fd, buf, 512);
// ↑ glibc 将此转为:
// mov rax, __NR_read   ; 系统调用号
// syscall              ; 触发中断/异常，进入内核
⚠️ NAND 写入限制
直接对 /dev/mtd0 写入时：
• 写大小必须是设备 最小写单元（writesize） 的整数倍（通常 512B / 2048B / 4096B）
• 写之前对应块必须已被擦除（全为 0xFF）
• 读可以是任意大小（内核会对齐后拷贝所需字节）
⚡ 第一性原理

系统调用是用户态与内核态的唯一合法边界。用户程序无法直接访问硬件寄存器，必须通过系统调用请求内核代劳。这个边界的本质是 CPU 特权级切换（Ring 3 → Ring 0），内核在此验证参数合法性，防止用户程序破坏系统。

02

VFS 虚拟文件系统层

VFS（Virtual File System）是 Linux 内核中的统一文件系统抽象层。它定义了一套通用的数据结构和操作接口，使得应用程序无需关心底层是 ext4、UBIFS 还是 NFS，统一使用相同的 POSIX API。

核心数据结构

结构体作用
struct superblock描述整个已挂载文件系统的元数据（块大小、总块数、文件系统类型）
struct inode描述单个文件/目录的元信息（权限、大小、块指针）
struct dentry目录项缓存，将路径名映射到 inode（dentry cache）
struct file进程打开文件的上下文（当前偏移、标志位、file_operations 指针）

VFS 如何分发到 MTD

每种文件系统注册自己的 file_operations 和 address_space_operations。当 VFS 收到读写请求时，通过函数指针调用具体文件系统的实现。对于 MTD 字符设备（/dev/mtdX），分发路径如下：

sys_read()
  └─ vfs_read()
       └─ file->f_op->read()          ← MTD 字符驱动注册的 .read
            └─ mtd_read_oob() / mtd_read()
                 └─ 具体 NAND/NOR 驱动
📌 官方参考
VFS 设计参考：kernel.org VFS 文档 MTD 字符设备注册：drivers/mtd/mtdchar.c，MTD 块设备：drivers/mtd/mtdblock.c
⚡ 第一性原理

VFS 的本质是接口与实现的解耦。它用函数指针表（file_operations）定义"应该做什么"，具体文件系统实现"怎么做"。这和面向对象中的"接口/多态"完全等价。对于 Flash，VFS 是所有上层请求的第一个分拣员，决定请求该去哪条路径。

03

页缓存（Page Cache）

页缓存是 Linux 内核中以内存页（4KB）为单位的数据缓冲区。读操作优先查页缓存，命中则直接返回；未命中才触发实际 Flash 读取并填充缓存。写操作先写入页缓存（脏页），由内核定期刷写（writeback）。

Flash 文件系统的特殊性

⚠️ 重要差异
UBIFS / JFFS2 等 Flash 文件系统不完全依赖标准页缓存路径。原因是：
• Flash 擦除块远大于 4KB（通常 128KB~256KB），不能像磁盘一样原地覆写
• JFFS2 完全绕过页缓存，直接管理 Flash
• UBIFS 使用自己的 TNC（Tree Node Cache）缓存索引，写操作通过 journal 机制批量提交
• MTD 字符设备直接读写，也绕过页缓存（O_DIRECT 语义）

直接访问（mtd-utils）与 FS 访问对比

访问方式是否经过页缓存说明
read /dev/mtd0❌ 否直接到 MTD 核心，绕过页缓存
UBIFS 文件读写△ 部分UBIFS 有自己的 TNC 缓存，特定路径会用 page cache
JFFS2 文件读写✅ 是（读）读取后填充页缓存；但扫描挂载时全量读取 Flash
nandwrite 工具❌ 否直接 ioctl/write 到 MTD 字符设备
⚡ 第一性原理

页缓存的存在是为了弥补 CPU（GHz）与存储（慢得多）之间的速度鸿沟。但 Flash 的写入物理约束（必须先擦再写、块粒度大）使得通用页缓存的 writeback 策略不够高效，Flash 文件系统因此各自实现了更适合 Flash 特性的缓冲机制。

04

Flash 文件系统层（UBIFS / JFFS2）

Flash 文件系统工作在 MTD 层之上，负责文件的组织、索引、压缩和崩溃恢复。主流选择是 UBIFS（推荐）和 JFFS2（较老）。

📌 官方来源
UBIFS 官方文档 kernel.org UBIFS.txt

UBIFS vs JFFS2 核心区别

UBIFS 内部数据结构

组件作用存储位置
Index (B+树)文件系统索引，叶节点存文件数据Flash 上
Journal先于索引更新写入，降低写放大Flash 上（固定大小）
TNCB+树的内存镜像，加速查找RAM 中
LPT每个 LEB 的空间账本（空闲/脏）Flash 上

UBIFS 写入流程

用户 write(fd, data, len)
  → VFS → UBIFS ubifs_write_begin/end()
     → 数据写入 Journal（追加写，不原地改）
     → 定期 commit：Journal → B+树索引
     → 通过 UBI API 写 LEB
        → MTD mtd_write()
           → NAND 驱动
⚡ 第一性原理

UBIFS 的设计核心是日志结构（Log-Structured）——所有写操作以追加方式写入 Journal，避免频繁随机写。这契合 Flash 的物理特性：追加顺序写比随机写效率高得多，且延迟了擦除操作。B+树提供 O(log n) 查找，克服了 JFFS2 扫描式设计的可扩展性瓶颈。

05

UBI 卷管理层（Unsorted Block Images）

UBI 是 Linux 内核中的裸 Flash 卷管理系统，由 Artem Bityutskiy 开发，自 Linux 2.6.22 起并入主线。UBI 工作在 MTD 层之上、文件系统之下，提供逻辑卷（volume）抽象，对上层屏蔽坏块和磨损均衡细节。

📌 官方来源
UBI 官方文档（linux-mtd.infradead.org） 源码目录：drivers/mtd/ubi/

UBI 核心功能

UBI 每个 PEB 的两个头部（64 字节）

每个物理擦除块（PEB）开头存放两个 64 字节头：

EC Header（擦除计数头）：
  magic | version | padding_type | erase_count | crc32

VID Header（卷标识头）：
  magic | version | vol_type | copy_flag | compat | vol_id
  lnum  | leb_ver | data_size | used_ebs | data_pad
  data_crc | sqnum | crc32
✅ UBI 操作命令
ubiattach /dev/ubi_ctrl -m 0      # 将 mtd0 附加到 UBI
ubimkvol /dev/ubi0 -N rootfs -s 64MiB  # 创建逻辑卷
ubiinfo /dev/ubi0                  # 查看 UBI 设备信息
cat /sys/class/ubi/ubi0/volumes_count  # sysfs 查询卷数量
⚡ 第一性原理

UBI 的本质是一个中间映射层：逻辑擦除块（LEB）到物理擦除块（PEB）的动态映射表。通过在写时选择磨损最轻的 PEB，UBI 实现全局磨损均衡；通过维护备用 PEB 池，透明替换坏块。这与 SSD 内部 FTL 的原理相同，只是在软件层实现而非硬件层。

06

MTD 核心抽象层

MTD（Memory Technology Device）是 Linux 内核中专门为裸 Flash 设备设计的抽象层，提供统一 API 屏蔽不同 Flash 类型（NAND、NOR、OneNAND 等）的差异。

📌 官方来源
API 定义：include/linux/mtd/mtd.h MTD API 文档

MTD 核心 API（内核内部调用）

/* 定义于 include/linux/mtd/mtd.h */

/* 读取数据（自 from 偏移起读 len 字节） */
int mtd_read(struct mtd_info *mtd, loff_t from, size_t len,
             size_t *retlen, u_char *buf);

/* 写入数据（写之前该块必须已擦除） */
int mtd_write(struct mtd_info *mtd, loff_t to, size_t len,
              size_t *retlen, const u_char *buf);

/* 擦除一个擦除块（整块擦除） */
int mtd_erase(struct mtd_info *mtd, struct erase_info *instr);

/* 读取 OOB 区域（NAND 专用，含 ECC 信息） */
int mtd_read_oob(struct mtd_info *mtd, loff_t from,
                 struct mtd_oob_ops *ops);

/* 写入 OOB 区域 */
int mtd_write_oob(struct mtd_info *mtd, loff_t to,
                  struct mtd_oob_ops *ops);

struct mtd_info 关键字段

struct mtd_info {
    u_char   type;          /* MTD_NANDFLASH / MTD_NORFLASH 等 */
    uint32_t flags;         /* MTD_WRITEABLE / MTD_NO_ERASE 等 */
    uint64_t size;          /* 设备总大小（字节） */
    uint32_t erasesize;     /* 擦除块大小（如 131072 = 128KiB） */
    uint32_t writesize;     /* 最小写单元（NAND：页大小） */
    uint32_t oobsize;       /* OOB 区域大小（每页） */
    /* ... 函数指针：_read / _write / _erase / _read_oob ... */
    int (*_read)(struct mtd_info *, loff_t, size_t, size_t *, u_char *);
    int (*_write)(struct mtd_info *, loff_t, size_t, size_t *, const u_char *);
    int (*_erase)(struct mtd_info *, struct erase_info *);
    /* ... */
};

mtd_read() 返回值说明

返回值含义
0成功，无任何位翻转
-EUCLEAN有位翻转但已被 ECC 纠正（严重程度超过阈值，建议迁移数据）
-EBADMSG不可纠正的 ECC 错误（数据已损坏）
-ENODEV设备不存在
📌 内核 3.4 之后的变化
在 Linux 3.4 之前，上层代码直接调用 mtd_info 中的函数指针（mtd->read()）。
3.4 之后，统一改为调用 mtd_read() 等封装函数，由 MTD 核心代码中转后再调用驱动的函数指针，增加了统一检查和钩子。
⚡ 第一性原理

MTD 核心层的本质是接口标准化：无论下面是并行 NAND、SPI NOR 还是 OneNAND，上层代码只需调用统一的 mtd_read/write/erase。这是软件工程中"依赖抽象而非依赖实现"原则的直接体现。函数指针表（struct mtd_info 中的 _read/_write）就是 C 语言实现的虚函数表。

Flash 驱动层（NAND / SPI-NOR）

Flash 驱动是 MTD 子系统与物理硬件之间的桥接层。不同接口的 Flash 有对应的驱动框架：

SPI-NOR 驱动软件栈（来自 Analog Devices 官方文档）

         MTD
------------------------
    SPI NOR framework        ← drivers/mtd/spi-nor/core.c
------------------------
        m25p80               ← 厂商芯片驱动（JEDEC ID 识别）
------------------------
     SPI bus driver          ← drivers/spi/spi.c
------------------------
      SPI NOR chip           ← 物理芯片（硬件）

NAND 读操作（驱动层）关键步骤

/* drivers/mtd/nand/core.c 简化流程 */
nand_do_read_ops()
  1. 等待芯片就绪（等待 R/B# 引脚拉高）
  2. 发送 READ 命令序列到 NAND 控制器：
       CMD_READ0 (0x00) | 列地址 | 行地址 | CMD_READSTART (0x30)
  3. 等待芯片内部操作完成（tR 时间）
  4. 从数据寄存器读取页数据 + OOB
  5. 调用 ECC 引擎校验数据
  6. 拷贝有效数据到用户 buffer
  7. 如有 bitflip 超阈值，返回 -EUCLEAN
⚡ 第一性原理

Flash 驱动的核心工作是将逻辑地址转换为物理命令序列，发送给硬件控制器。NAND 的"命令-地址-数据"三阶段协议和 SPI NOR 的"操作码+地址+数据"协议，都是硬件厂商定义的协议，驱动代码忠实实现这些协议并向上层提供统一的 read/write/erase 语义。

09

ECC 校验机制

ECC（Error-Correcting Code，纠错码）是 Flash（尤其是 NAND）数据可靠性的关键保障。NAND 在存储和传输过程中会产生位翻转（bit-flip），ECC 用于检测和纠正这些错误。

📌 官方来源
kernel.org NAND ECC 文档 源码：drivers/mtd/nand/ecc-sw-hamming.c

ECC 存储位置：OOB 区域

Linux 内核 ECC 实现方式

方式说明配置
软件 Hamming ECCCPU 计算汉明码，256B 数据生成 3B ECC，可纠 1-bit 错误NAND_ECC_ENGINE_TYPE_SOFT
软件 BCH ECCBCH 码，可纠正更多位错误（4/8/16-bit），适合 MLC NANDNAND_ECC_ENGINE_TYPE_SOFT + BCH
硬件 ECCSoC NAND 控制器内置 ECC 引擎，CPU 不参与计算NAND_ECC_ENGINE_TYPE_ON_HOST
片内 ECCNAND 芯片内部自带 ECC（如某些 eMMC/UFS）NAND_ECC_ENGINE_TYPE_ON_DIE

ECC 工作流程

写入时：
  1. 对每 256 字节（或 512B/页）数据计算 ECC 码
  2. 将 ECC 码写入同一页的 OOB 区域
  3. 数据和 ECC 一起写入 Flash

读取时：
  1. 读取页数据 + OOB 中的 ECC 码
  2. 对读出数据重新计算 ECC 并与存储值比对
  3. 无差异 → 数据正确，返回 0
  4. 有差异 → 尝试纠正（1-bit → 可纠正，返回 -EUCLEAN）
           → 超出纠正能力 → 返回 -EBADMSG（数据损坏）
⚠️ -EUCLEAN 阈值
Linux 3.4 之后，-EUCLEAN 表示"纠正的位翻转数量超过 bitflip_threshold"。
该阈值可通过 sysfs 调整：/sys/class/mtd/mtd0/bitflip_threshold
建议值：≤ ECC 强度的 75%，触发后应将该擦除块数据迁移。
⚡ 第一性原理

ECC 的本质是冗余信息换可靠性：牺牲少量存储空间（OOB 区域）存储校验码，获得在数据发生少量位翻转时的自我修复能力。汉明码/BCH 码都是信息论中"最优前向纠错（FEC）码"的工程实现。Flash 越来越小的制程使位翻转率增加，ECC 强度也随之提升（从 1-bit 到 64-bit 纠错）。

10

总线控制器 & 物理硬件层

最底层是 Flash 控制器和物理芯片之间的电气信号交互。不同 Flash 类型使用不同总线协议：

NAND Flash 命令序列（以读页为例）

NAND 控制器发出的信号序列（并行接口）：

CLE=1, WE↓  → 发送命令字节 0x00 (READ0)
ALE=1, WE↓  → 发送列地址低字节 (column[7:0])
ALE=1, WE↓  → 发送列地址高字节 (column[11:8])
ALE=1, WE↓  → 发送行地址 byte0 (row[7:0])
ALE=1, WE↓  → 发送行地址 byte1 (row[15:8])
ALE=1, WE↓  → 发送行地址 byte2 (row[23:16])  ← 大容量 NAND
CLE=1, WE↓  → 发送命令字节 0x30 (READ START)

等待 R/B# 引脚变高（芯片内部页读取完成，约 25μs）

RE↓ × 2112  → 循环从 I/O[7:0] 读取 2048+64 字节数据

SPI-NOR Flash 命令序列

SPI 控制器发出的序列（4 线 SPI）：

CS 低电平 → 开始通信
MOSI → 0x03 (READ 操作码 SPINOR_OP_READ)
MOSI → 地址字节 [23:16]
MOSI → 地址字节 [15:8]
MOSI → 地址字节 [7:0]
MISO → 连续读取 N 字节数据
CS 高电平 → 结束通信

写入前必须发送：
0x06 (WREN)  → 写使能
0x20 (SE)    → 扇区擦除（必须先擦）
等待 WIP bit（状态寄存器）清零（约 100ms~1s）
0x02 (PP)    → 页编程，最多 256 字节

设备树（Device Tree）中的 Flash 描述

/* QSPI Flash 设备树示例（来自 Petalinux/Xilinx 官方文档） */
&qspi {
    flash0: flash@0 {
        compatible = "micron,n25q512a", "jedec,spi-nor";
        reg = <0x0>;
        spi-tx-bus-width = <1>;
        spi-rx-bus-width = <4>;
        spi-max-frequency = <50000000>;    /* 50 MHz */

        partition@0 {
            label = "boot";
            reg = <0x00000000 0x00500000>;
        };
        partition@1 {
            label = "kernel";
            reg = <0x00520000 0x00a80000>;
        };
    };
};
⚡ 第一性原理

物理层的本质是电信号协议：控制器通过时序精确的高低电平序列告诉 Flash 芯片"执行什么操作、操作哪个地址、传输多少数据"。设备树则是操作系统在启动时了解硬件拓扑的机制——相当于硬件的"说明书"，驱动根据它知道 Flash 连在哪条总线、什么型号、支持哪些参数。

⚡ 第一性原理

调试的本质是在软件层次结构中精确定位问题所在的层。每一层都有其专属的观测接口：用户态用 strace，内核态用 ftrace/dmesg，硬件层用示波器/逻辑分析仪。分层调试的核心策略是"分而治之"——先确认上层数据是否正确到达本层，再验证本层输出是否符合预期，从而将问题缩小到具体的一层。
