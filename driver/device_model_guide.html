Linux 内核设备模型

深入解析 Linux 内核统一设备模型的设计背景、演化历程、实现原理与核心抽象，基于官方内核文档与权威来源编写。

内核文档 kernel.org LWN.net 权威解析 Linux 6.x 内核

什么是 Linux 设备模型

Linux 内核驱动模型（Linux Kernel Driver Model）是对内核中以前各种分散驱动模型的统一整合。它旨在通过将一组数据和操作整合到全局可访问的数据结构中，来增强各种总线和设备的桥接驱动。

官方定义
"The Linux Kernel Driver Model is a unification of all the disparate driver models that were previously used in the kernel. It is intended to augment the bus-specific drivers for bridges and devices by consolidating a set of data and operations into globally accessible data structures."

— Patrick Mochel，内核文档，2002年8月26日
kernel.org — The Linux Device Model

模型的核心目标

  • 为描述总线及其下设备提供统一的通用数据模型
  • 支持无缝的设备即插即用（Plug and Play）
  • 提供通用的电源管理能力
  • 实现完整的热插拔（Hot Plug）机制
  • 向用户空间输出完整的设备层级视图（通过 sysfs）

设计背景与驱动力

要理解 Linux 设备模型为何以今天的方式设计，必须回顾 2.4 内核时代的困境。

Linux 2.4 时代的三大痛点

根据 Wikipedia sysfs 条目及内核文档记载，在 2.5 开发周期引入设备模型之前，2.4 内核存在如下根本性缺陷：

问题描述影响
没有统一的驱动-设备关系表示 每个总线子系统各自维护独立的数据结构和设备列表，没有跨总线的统一模型 代码重复，设备管理混乱
没有通用热插拔机制 USB、PCI 等不同总线各自实现独立的热插拔逻辑，无法统一 热插拔支持参差不齐，难以维护
procfs 被非进程信息污染 /proc 中充斥着大量设备、驱动、总线等非进程信息 接口混乱，信息难以结构化访问
Wikipedia — sysfs：历史背景
核心矛盾
传统驱动模型为它们控制的设备实现了某种树状结构（有时只是列表），但不同总线类型之间没有任何统一性。随着现代计算机对设备即插即用、热插拔、精细电源管理的需求不断增长，这种碎片化的架构已无法胜任。

2.6 之前的问题详解

各自为政的子系统

在 Linux 2.4 中，PCI 子系统有 pci_dev 列表，USB 子系统有 usb_device 树，SCSI 有 Scsi_Host 链表……每个子系统都是孤岛，电源管理信号无法跨越边界传递。

电源管理困境

当系统需要执行 suspend/resume 操作时，内核必须依次通知每个设备，但因缺乏统一的设备树，无法保证按正确的依赖顺序关闭/唤醒设备（必须先关闭子设备，再关闭父设备）。

driverfs 的诞生（2.5.45 前身）

Patrick Mochel 在 2.5 开发周期开始设计统一的设备抽象。他首先创建了 driverfs（后来更名为 sysfs），用于向用户空间导出设备树。

历史记录
当 Patrick Mochel 在 2.5.45 内核中加入 kobject 类型时，他的描述是："This is not meant to be fancy; just something simple for which we can control the refcount and other common functionality using common code."（这不是要搞得很花哨；只是一个简单的东西，我们可以用公共代码控制其引用计数和其他公共功能。）

— LWN.net，《The debut of kref》，2004年
LWN.net — The debut of kref（kobject 起源）

kobject — 设备模型的基石

理解驱动模型以及构建于其上的 kobject 抽象的难点在于没有明显的切入点。要处理 kobject，需要理解几种相互引用的不同类型。

官方定义（Greg Kroah-Hartman）
"A kobject is an object of type struct kobject. Kobjects have a name and a reference count. A kobject also has a parent pointer (allowing objects to be arranged into hierarchies), a specific type, and, usually, a representation in the sysfs virtual filesystem."
kernel.org — Everything you never wanted to know about kobjects, ksets, and ktypes

kobject 的核心能力

  • 引用计数：作为嵌入结构的引用计数器，确保对象在所有引用释放前不被销毁
  • 层级结构：parent 指针构建对象树，映射到 sysfs 目录结构
  • sysfs 表示：每个 kobject 在 sysfs 中有对应目录
  • uevent 生成：kobject 的添加/删除触发热插拔事件

struct kobject 定义

struct kobject {
    const char              *name;       /* sysfs 目录名 */
    struct list_head        entry;       /* kset 中的链表节点 */
    struct kobject          *parent;     /* 父 kobject，构建层级 */
    struct kset             *kset;       /* 所属的 kset 容器 */
    struct kobj_type        *ktype;      /* 类型描述与回调 */
    struct kernfs_node      *sd;         /* sysfs 目录项 */
    struct kref             kref;        /* 引用计数（原子操作） */
    unsigned int state_initialized:1;   /* 已初始化 */
    unsigned int state_in_sysfs:1;      /* 已注册到 sysfs */
    unsigned int state_add_uevent_sent:1;/* ADD uevent 已发出 */
    unsigned int state_remove_uevent_sent:1;
    unsigned int uevent_suppress:1;     /* 抑制 uevent */
};

嵌入模式：面向对象的 C 实现

内核代码极少单独创建一个独立的 kobject。相反，kobject 被嵌入到更大的特定领域结构中。可以将 kobject 看作顶层抽象基类，其他结构从中"派生"。C 语言不支持直接表达继承，因此使用结构嵌入技术：

/* UIO 驱动中的嵌入示例 */
struct uio_map {
    struct kobject kobj;   /* 嵌入的 kobject — 必须是唯一一个！ */
    struct uio_mem *mem;
};

/* 从 kobject 指针反向获取包含结构（类似 C++ 的 upcast） */
struct uio_map *map = container_of(kp, struct uio_map, kobj);

/* 通常定义便捷宏 */
#define to_map(map) container_of(map, struct uio_map, kobj)
重要规则
任何结构都绝不能嵌入超过一个 kobject。否则对象的引用计数必然出错，代码将充满 bug。
— 官方内核文档，Greg Kroah-Hartman

kobject 生命周期 API

/* 初始化 kobject（必须指定 ktype） */
void kobject_init(struct kobject *kobj, const struct kobj_type *ktype);

/* 注册到 sysfs（指定父节点和名称） */
int kobject_add(struct kobject *kobj, struct kobject *parent,
                const char *fmt, ...);

/* 初始化并注册的便捷函数 */
int kobject_init_and_add(struct kobject *kobj, const struct kobj_type *ktype,
                         struct kobject *parent, const char *fmt, ...);

/* 引用计数操作 */
struct kobject *kobject_get(struct kobject *kobj);  /* 增加引用 */
void kobject_put(struct kobject *kobj);             /* 减少引用，可能触发 release */

/* 发送 uevent 事件 */
int kobject_uevent(struct kobject *kobj, enum kobject_action action);

ktype — 类型描述符

ktype 是嵌入 kobject 的对象的类型。每个嵌入 kobject 的结构都需要一个对应的 ktype，它控制 kobject 创建和销毁时的行为。

struct kobj_type {
    void (*release)(struct kobject *kobj);       /* 引用归零时的清理回调 */
    const struct sysfs_ops *sysfs_ops;           /* sysfs 属性读写操作 */
    const struct attribute_group **default_groups; /* 默认属性组 */
    const struct kobj_ns_type_operations *(*child_ns_type)(...);
    const void *(*namespace)(struct kobject *kobj);
    void (*get_ownership)(struct kobject *kobj, kuid_t *uid, kgid_t *gid);
};

release 方法的重要性

每个 kobject 都必须有一个 release() 方法，且 kobject 必须在该方法被调用之前保持一致状态。这个通知机制解决了一个关键问题：创建 kobject 的代码通常不知道最后一个引用何时被释放。

/* release 方法的标准实现模式 */
static void my_object_release(struct kobject *kobj)
{
    /* container_of 从 kobject 指针回溯到包含结构 */
    struct my_object *mine = container_of(kobj, struct my_object, kobj);

    /* 执行额外清理工作，然后释放内存 */
    kfree(mine);
}

static struct kobj_type my_ktype = {
    .release   = my_object_release,
    .sysfs_ops = &my_sysfs_ops,
    .default_groups = my_groups,
};

kset — kobject 容器

kset 是一组想要彼此关联的 kobject 的集合。它们不必是相同的 ktype，但要谨慎混用。

kset 的三大功能

  • 对象容器：内核用 kset 追踪"所有块设备"或"所有 PCI 设备驱动"等集合
  • sysfs 目录：kset 在 sysfs 中是一个子目录，其关联的 kobject 在其中展示；sysfs 的顶级目录层次结构就是通过这种方式构建的
  • uevent 控制：kset 可以支持 kobject 的"热插拔"，并影响 uevent 事件向用户空间的报告方式
kernel.org — kobject/kset/ktype 官方文档
/* 创建并注册 kset */
struct kset *kset_create_and_add(const char *name,
                                 const struct kset_uevent_ops *uevent_ops,
                                 struct kobject *parent_kobj);

/* 销毁 kset */
void kset_unregister(struct kset *k);

/* uevent 操作回调 */
struct kset_uevent_ops {
    /* filter: 返回 0 则阻止该 kobject 的 uevent 发出 */
    int (* const filter)(struct kobject *kobj);
    /* name: 覆盖发送给用户空间的 kset 名称 */
    const char *(* const name)(struct kobject *kobj);
    /* uevent: 在发送 uevent 前添加自定义环境变量 */
    int (* const uevent)(struct kobject *kobj, struct kobj_uevent_env *env);
};

引用计数机制

kobject 的关键功能之一是作为其嵌入对象的引用计数器。只要对象存在引用，该对象（及其支撑代码）就必须继续存在。

/* kref：纯粹的引用计数，无需完整的 kobject */
struct kref {
    refcount_t refcount;
};

/* kobject_init() 将引用计数设为 1 */
/* 增加引用：防止对象被过早释放 */
struct kobject *kobject_get(struct kobject *kobj);

/* 减少引用：当计数降到 0 时，调用 ktype->release() */
void kobject_put(struct kobject *kobj);

/* 警告：注册后绝不能直接 kfree()，只能通过 kobject_put() */
设计原则
引用计数不在创建 kobject 的代码直接控制之下——sysfs 的其他部分可以持有任何已注册 kobject 的引用。因此，当最后一个引用消失时，必须通过 release() 回调异步通知对象的创建者。

总线 (Bus) — 连接的枢纽

总线是处理器与输入/输出设备之间的通信通道。为确保模型的通用性，所有设备都通过某种总线连接到处理器——即便是没有物理对应硬件的虚拟总线。

kernel.org — Bus Types 官方文档

struct bus_type

struct bus_type {
    const char      *name;          /* 总线名称，如 "pci", "usb" */
    const char      *dev_name;
    struct device   *dev_root;

    /* 属性组 */
    const struct attribute_group **bus_groups;
    const struct attribute_group **dev_groups;
    const struct attribute_group **drv_groups;

    /* match：新设备或新驱动加入总线时调用，进行匹配 */
    int (*match)(struct device *dev, struct device_driver *drv);
    /* uevent：热插拔事件发生前调用，添加环境变量 */
    int (*uevent)(struct device *dev, struct kobj_uevent_env *env);
    int (*probe)(struct device *dev);
    int (*remove)(struct device *dev);
    void (*shutdown)(struct device *dev);

    int (*suspend)(struct device *dev, pm_message_t state);
    int (*resume)(struct device *dev);

    const struct dev_pm_ops *pm;
    struct subsys_private *p;  /* 管理设备/驱动列表的私有数据 */
};

总线注册示例

/* 定义并注册自定义总线 */
static int my_match(struct device *dev, struct device_driver *driver)
{
    /* 最简单的匹配：比较设备名与驱动名 */
    return !strncmp(dev_name(dev), driver->name, strlen(driver->name));
}

static int my_uevent(struct device *dev, struct kobj_uevent_env *env)
{
    add_uevent_var(env, "DEV_NAME=%s", dev_name(dev));
    return 0;
}

struct bus_type my_bus_type = {
    .name   = "mybus",
    .match  = my_match,
    .uevent = my_uevent,
};

static int __init my_bus_init(void)
{
    return bus_register(&my_bus_type);
}

设备 (Device)

系统中的任何设备都有一个关联的 struct device 结构。设备通过不同的内核方法被发现（热插拔、驱动初始化、系统启动）并注册到系统中。每个内核中存在的设备在 /sys/devices 下都有一个条目。

kernel.org — The Basic Device Structure

struct device

struct device {
    struct device        *parent;    /* 父设备（通常是控制器） */
    struct device_private *p;        /* 内部私有数据 */
    struct kobject        kobj;      /* 嵌入的 kobject */

    const char           *init_name; /* 设备初始名称 */
    const struct device_type *type;

    struct bus_type      *bus;       /* 所属总线类型 */
    struct device_driver *driver;   /* 已绑定的驱动（NULL=未绑定） */

    void                 *platform_data;
    struct dev_pm_info   power;     /* 电源管理信息 */
    struct dev_pm_domain *pm_domain;

    struct device_node  *of_node;   /* 设备树节点 */
    struct acpi_dev_node acpi_node; /* ACPI 节点 */

    dev_t                 devt;      /* 主/次设备号（字符/块设备） */
    struct class         *class;    /* 功能分类（input、net 等） */

    const struct attribute_group **groups;
    void (*release)(struct device *dev); /* 引用归零时的释放回调 */
};

设备属性

/* 定义设备属性宏 */
#define DEVICE_ATTR(name, mode, show, store)

/* 简化宏 */
static DEVICE_ATTR_RO(type);         /* 只读属性 */
static DEVICE_ATTR_RW(power);        /* 读写属性 */

/* 注册前必须设置属性组，以确保 uevent 发出前属性已存在 */
dev->groups = dev_groups;
device_register(dev);

总线特定设备结构（嵌入模式）

/* PCI 设备：struct device 被嵌入到 struct pci_dev */
struct pci_dev {
    /* PCI 特定字段 ... */
    unsigned short  vendor;
    unsigned short  device;
    /* ... */
    struct device   dev;   /* 通用设备接口（不一定在开头！） */
    /* ... */
};
设计意图
struct device dev 故意不放在 pci_dev 结构的开头。这是为了让开发者在总线驱动和通用驱动之间切换时认真思考，防止无意义且错误的强制类型转换。

驱动 (Driver)

设备驱动是静态分配的结构。尽管一个驱动可能支持系统中的多个设备，struct device_driver 代表的是整个驱动（而不是某个特定的设备实例）。

kernel.org — Device Drivers 官方文档
struct device_driver {
    const char          *name;      /* 驱动名称（sysfs 中显示） */
    struct bus_type    *bus;       /* 所属总线 */

    struct module      *owner;
    const char         *mod_name;

    /* 核心回调函数 */
    int  (*probe)   (struct device *dev);   /* 绑定设备 */
    void (*sync_state)(struct device *dev); /* 所有消费者探测完毕后调用 */
    int  (*remove)  (struct device *dev);   /* 解绑设备 */
    void (*shutdown)(struct device *dev);   /* 系统关机 */
    int  (*suspend) (struct device *dev, pm_message_t state); /* 休眠 */
    int  (*resume)  (struct device *dev);   /* 唤醒 */

    const struct attribute_group **groups;
    const struct dev_pm_ops *pm;
    struct driver_private *p;
};

probe() 的语义

probe() 在任务上下文中调用，持有总线的读写信号量，驱动部分绑定到设备。驱动通常使用 container_of() 将 dev 转换为总线特定类型。

-EPROBE_DEFER
如果驱动依赖尚未可用的资源（例如，由尚未初始化的驱动提供的资源），probe() 可以返回 -EPROBE_DEFER。驱动核心会将设备放入延迟探测列表，稍后重新尝试。但如果 probe() 已经创建了子设备，则不得返回 -EPROBE_DEFER，否则会导致无限循环。

总线特定驱动（嵌入模式）

/* PCI 驱动：struct device_driver 嵌入 struct pci_driver */
struct pci_driver {
    const struct pci_device_id *id_table; /* 支持的设备 ID 表 */
    struct device_driver driver;           /* 嵌入的通用驱动结构 */
};

/* eepro100 驱动声明示例 */
static struct pci_driver eepro100_driver = {
    .id_table = eepro100_pci_tbl,
    .driver   = {
        .name    = "eepro100",
        .bus     = &pci_bus_type,
        .probe   = eepro100_probe,
        .remove  = eepro100_remove,
        .suspend = eepro100_suspend,
        .resume  = eepro100_resume,
    },
};

设备类 (Class)

设备类是 Linux 设备模型的高层视图，它抽象了实现细节。例如，有 SCSI 驱动和 ATA 驱动，但都属于磁盘类。类根据功能对设备进行分组，而不是根据它们的连接方式或工作方式。

类的两大核心作用

  • 功能抽象：屏蔽不同总线/硬件的差异，为用户空间呈现统一接口（如 /dev/sda 对应一个 block device）
  • 自动创建 /dev 节点：配合 udev，当类设备注册时，自动在 /dev 创建对应的设备文件
/* 注册一个类 */
struct class my_class = {
    .name  = "myclass",
    .owner = THIS_MODULE,
};
class_register(&my_class);

/* 在类下创建设备（自动在 /dev 创建节点）*/
struct device *my_classdev;
my_classdev = device_create(&my_class, NULL, cdev.dev, &dev, "myclass0");
/* → 生成 /dev/myclass0 */

/* 销毁类设备 */
device_destroy(&my_class, cdev.dev);

sysfs — 内核模型的用户空间镜像

sysfs 是一个由 Linux 内核提供的伪文件系统，通过虚拟文件向用户空间导出内核设备模型中的各种内核子系统、硬件设备和相关设备驱动的信息。它通常挂载在 /sys 目录下。

kernel.org — User Interface（sysfs）
# 通常通过 /etc/fstab 自动挂载
none  /sys  sysfs  defaults  0 0

# 手动挂载
# mount -t sysfs sysfs /sys

sysfs 顶级目录结构

目录内容
/sys/block系统中所有可用的块设备（磁盘、分区）
/sys/bus物理设备所连接的总线类型（pci, ide, usb...）
/sys/class驱动类别（net, sound, usb...）
/sys/devices连接到系统的设备的层级结构（物理视图）
/sys/firmware系统固件信息（ACPI）
/sys/fs已挂载文件系统的信息
/sys/kernel内核状态信息
/sys/module当前加载的模块列表
/sys/power电源管理子系统相关信息
/sys/dev按主次设备号索引的设备（block/char）

kobject 与 sysfs 的对应关系

当设备被插入设备树时，会为其创建一个目录。这个目录可以在发现的每一层填充内容——全局层、总线层或设备层。

重要区别
内核设备模型和 sysfs 文件系统是两个不同的概念。内核设备模型不依赖 sysfs 就可以工作，但反之不成立——sysfs 依赖设备模型。两者之间的相似性容易造成混淆，但它们服务于不同目的：前者是内核内部的数据结构，后者是向用户空间导出这些结构的接口。

标准 sysfs 属性

  • dev：主/次设备号标识符，可用于自动创建 /dev 条目
  • device：指向包含设备目录的符号链接
  • driver：指向驱动目录的符号链接（位于 /sys/bus/*/drivers）

热插拔 (Hotplug) 机制

热插拔描述了在系统运行期间添加或移除设备而无需重启的机制。热插拔事件是内核在系统配置发生变化时向用户空间发出的通知。

电源管理

统一设备模型的一个关键价值是为系统级电源管理提供了结构化基础。在 2.4 时代，没有统一的设备树，电源管理无法按正确的依赖顺序操作设备。

有序关机的保证

驱动模型维护一个设备有序链表用于电源管理。此链表的顺序由已注册设备的父子关系决定，从而保证设备在其物理父设备之前关闭，反之亦然。

sync_state 的设计意图

sync_state() 是一个典型用例：内核从 bootloader 接管设备管理。如果 bootloader 将某设备保持开启并处于特定硬件配置，设备驱动可能需要保持该配置，直到所有该设备的消费者都完成探测后，才将硬件状态同步为所有消费者请求的软件状态的聚合。

绑定机制 (Binding)

将设备与其对应驱动连接称为绑定（Binding）。总线在设备驱动程序员的背后做了大量工作来绑定设备和驱动。

触发绑定的两类事件

  • 设备注册时：通过 device_register() 将设备注册到总线时，内核尝试将该设备与该总线上的每个已注册驱动进行绑定
  • 驱动注册时：通过 driver_register() 将驱动注册到总线时，内核尝试将该驱动与该总线上的每个未绑定设备进行绑定

第一性原理分析

从第一性原理出发，Linux 设备模型解决的本质问题是：如何在一个不断变化的异构硬件世界中，以统一、可组合、可扩展的方式管理设备的生命周期？

1. 为什么需要统一模型？

第一原理：复杂性的必然性
现代计算机系统连接着数十种不同类型的设备，通过 PCI、USB、I²C、SPI、ACPI 等多种总线协议。每种设备的发现方式、资源需求、电源特性各不相同。没有统一抽象，每个子系统都必须重复实现相同的基础设施（注册、引用计数、生命周期管理、用户空间接口），这违背了"不重复自己"（DRY）原则，且维护代价极高。

2. kobject：最小化的公共基础设施

kobject 的设计体现了最小必要原则：它只提供所有设备模型实体都需要的最小公共功能集：

  • 引用计数（kref）：解决"谁最后一个离开就关灯"的问题——保证对象在所有使用者释放后才被销毁
  • 父子关系（parent 指针）：解决"依赖顺序"问题——镜像真实世界的设备拓扑
  • sysfs 节点（sd 指针）：解决"可见性"问题——内核状态的标准化用户空间接口
  • uevent 通知：解决"响应变化"问题——连接内核事件与用户空间策略

3. 嵌入模式：C 语言的面向对象

由于 C 没有继承机制，Linux 采用结构嵌入 + container_of 宏实现了类似继承的效果：

  • struct kobject 等价于抽象基类
  • struct device 等价于从 kobject 派生的中间类
  • struct pci_dev 等价于从 device 派生的具体类
  • container_of() 等价于 C++ 中的 static_cast
图6：通过结构嵌入实现的 C 语言"继承"层级

4. 三元分离：Bus / Device / Driver

设备模型的核心架构决策是将硬件世界分解为三个正交关注点：

抽象职责回答的问题
Bus 枚举设备、提供匹配逻辑 设备如何被发现？设备和驱动如何配对？
Device 表示硬件实例，持有设备状态 这是什么硬件？它在哪里？它的资源是什么？
Driver 提供操作设备的软件逻辑 如何初始化这个设备？如何响应 I/O 请求？

5. 开放/封闭原则的体现

设备模型是对开放/封闭原则（Open/Closed Principle）的完美诠释：

  • 对扩展开放：添加新的总线类型（如 Thunderbolt、NVMe）只需实现 bus_type 接口，无需修改核心代码
  • 对修改封闭：kobject/kset 核心基础设施保持稳定，所有设备子系统共享相同的生命周期管理代码

6. 为什么 sysfs 是正确的接口

sysfs 的设计遵循了单一职责原则和最小特权原则：

  • 每个文件对应一个属性，而不是多个属性聚合（不同于 /proc 中混乱的格式）
  • 树状目录结构直接镜像设备拓扑，关系一目了然
  • 读写接口统一（show/store 回调），用户空间工具可以通用处理

7. 总结：设计的本质权衡

第一性原理总结
Linux 设备模型的核心本质是一个分层的、基于组合的设备抽象框架，它通过以下关键决策解决了复杂性管理问题：

① 用 kobject 提供最小化公共基础（引用计数 + 层级 + sysfs），而不是为每个子系统重新发明轮子
② 用结构嵌入模拟面向对象继承，在不引入虚拟机开销的前提下实现多态
③ 将总线、设备、驱动三个关注点正交分离，通过 match/probe 回调动态组合
④ 用 sysfs 将内核内部状态标准化暴露给用户空间，打破内核/用户空间之间的信息屏障
⑤ 用 uevent 机制将内核事件异步传达给用户空间策略层（udev），实现策略与机制的分离
