01

GPIO 概述与物理特性

官方定义（Linux Kernel 文档）
"A 'General Purpose Input/Output' (GPIO) is a flexible software-controlled digital signal. They are provided from many kinds of chips, and are familiar to Linux developers working with embedded and custom hardware. Each GPIO represents a bit connected to a particular pin, or 'ball' on Ball Grid Array (BGA) packages."
kernel.org — GPIO Introduction

什么是 GPIO？

GPIO（General Purpose Input/Output，通用输入/输出）是嵌入式系统中最基础的数字信号接口。每个 GPIO 表示连接到芯片特定引脚（或 BGA 封装焊球）的一个可编程数字信号位。SoC 处理器高度依赖 GPIO——某些芯片上几乎所有非专用引脚都可配置为 GPIO，大多数芯片至少有几十个 GPIO。

GPIO 的典型用途包括：监测 MMC/SD 卡插入/拔出、检测写保护状态、驱动 LED、配置收发器、位拆解串行总线、触发硬件看门狗、感测开关等。

📌 SoC 集成 GPIO
处理器内部直接集成，通过内存映射寄存器控制，速度最快，通常可在自旋锁持有期间访问
🔌 GPIO 扩展器
通过 I2C/SPI 串行总线连接的独立芯片，访问可能需要睡眠，不可在中断上下文直接使用
🔲 FPGA GPIO
可编程逻辑器件提供的 GPIO，可灵活配置为各种接口和功能，电气特性可编程
🎛 多功能芯片
电源管理芯片、音频编解码器等通常也集成少量 GPIO，弥补 SoC 引脚不足

GPIO 核心能力（Linux 官方定义）

来源：kernel.org GPIO Introduction
  • 输出值可写（high=1, low=0），部分芯片支持配置驱动方式，如"线或"和"开漏"模式
  • 输入值可读（1, 0），部分芯片支持对配置为输出的引脚进行回读
  • GPIO 控制器可内置去毛刺/去抖逻辑，有时可由软件控制
  • 输入常可用作 IRQ 信号，通常为边沿触发，有时为电平触发，可配置为系统唤醒事件
  • 通常可配置为输入或输出，部分为单方向；大多数可在持锁时访问

内部逻辑电路构造

GPIO 的核心是由 PMOS 和 NMOS 晶体管构成的推挽输出级（称为"图腾柱"结构），Linux Kernel 文档对此有精确描述：

来源：Linux Kernel GPIO Driver Interface 文档
"Integrated electronics often have an output driver stage in the form of a CMOS 'totem-pole' with one N-MOS and one P-MOS transistor where one of them drives the line high and one of them drives the line low. This is called a push-pull output."
docs.kernel.org — GPIO Driver Interface

推挽（Push-Pull）电路结构

图2：GPIO 图腾柱电路结构（Linux Kernel 文档原文描述）
Linux Kernel 原文说明
"The desired output signal (e.g. coming directly from some GPIO output register) arrives at IN. The switches named 'OD' and 'OS' are normally closed, creating a push-pull circuit. Consider the little 'switches' named 'OD' and 'OS' that enable/disable the P-MOS or N-MOS transistor right after the split of the input. As you can see, either transistor will go totally numb if this switch is open. The totem-pole is then halved and give high impedance instead of actively driving the line high or low respectively. That is usually how software-controlled open drain/source works."
docs.kernel.org — GPIO Driver: Open Drain/Source Support

开漏 vs 开源输出

Linux Kernel 原文定义
"Open drain (CMOS) or open collector (TTL) means the line is not actively driven high: instead you provide the drain/collector as output, so when the transistor is not open, it will present a high-impedance (tristate) to the external rail."
docs.kernel.org — GPIO Driver Interface: Open Drain/Source
🔓 开漏（Open Drain）
  • 禁用 PMOS，仅保留 NMOS
  • 能主动拉低（LOW），无法主动拉高
  • 输出 HIGH 时呈高阻抗（Hi-Z）
  • 需要外部上拉电阻提供高电平
  • 应用：I2C、One-Wire、线与逻辑
⬆ 开源（Open Source）
  • 禁用 NMOS，仅保留 PMOS
  • 能主动拉高（HIGH），无法主动拉低
  • 输出 LOW 时呈高阻抗（Hi-Z）
  • 需要外部下拉电阻提供低电平
  • 应用：较少见，某些电源开关场景
💡 开漏的两个核心用途（Linux Kernel 原文）
"This configuration is normally used as a way to achieve one of two things:
1. 电平移位：通过外部上拉至更高电压，达到比芯片 VDD 更高的逻辑电平。
2. 线或逻辑：多个开漏设备共享同一信号线，任何一个设备都可拉低，其他设备不会冲突——I2C 的 SCL/SDA 线即为此原理。"
docs.kernel.org

工作模式PMOSNMOS上下拉施密特典型用途

上下拉电阻

Linux Kernel 官方定义
"A GPIO line can support pull-up/down using the .set_config() callback. This means that a pull up or pull-down resistor is available on the output of the GPIO line, and this resistor is software controlled. The .set_config() callback can only turn pull up or down on and off, and will have no semantic knowledge about the resistance used."
docs.kernel.org — GPIO lines with pull up/down resistor support

为什么需要上下拉电阻？

当 GPIO 引脚配置为输入且没有外部驱动信号时，引脚处于高阻抗（Hi-Z）浮空状态。浮空引脚极易受噪声影响，会随机读取到 0 或 1，导致系统行为不可预测，还会引起额外功耗（因施密特触发器在噪声环境下随机翻转）。

图4：上拉电阻（默认高电平）与下拉电阻（默认低电平）电路对比

内部 vs 外部上下拉

来源：QuecPython GPIO 文档
"Pull-up resistor: It sets the status of GPIO pin in IDLE or in condition of invisible input signal. The Pull-up resistor will connect the GPIO pin to the positive to set it as the default high-level status. While the Pull-down resistor will connect the GPIO pin to the GND to set it as the default low-level status. Both of them will be controlled by register configured by SW."
QuecPython GPIO Documentation
特性内部上下拉外部上下拉
阻值40～100 kΩ（典型）1 kΩ ～ 100 kΩ（按需选择）
控制方式寄存器软件配置（PUPDR）PCB 焊接固定
灵活性高，可随时切换低，需重新焊接
驱动能力弱（< 100 μA）强（由阻值决定，可达数 mA）
功耗低取决于阻值（V²/R）
适用场景简单按键、一般信号线I2C 总线、长走线、高速信号

⚠️ 浮空输入的危害
"When no external signal is present, the Schmitt trigger randomly toggles between the logical levels induced by the external noise. This increases the overall consumption."
来源：ST AN4899 GPIO Hardware Settings

浮空输入引脚极易受电磁干扰影响，导致随机中断、状态不确定，以及因施密特触发器频繁翻转引起的额外动态功耗。未使用的 GPIO 应配置上拉、下拉或输出固定电平。

Linux 内核中的上下拉配置

/* 通过 pinctrl 配置上拉 (Device Tree) */
&gpio_pin {
    pinctrl-0 = <&pinctrl_gpio>;
    pinctrl-names = "default";
};

&iomuxc {
    pinctrl_gpio: gpiogrp {
        fsl,pins = <
            /* 引脚, 配置: 上拉100kΩ, 施密特触发使能 */
            MX6UL_PAD_GPIO1_IO00__GPIO1_IO0  0x1b020
        >;
    };
};

/* GPIO Driver 中通过 .set_config() 配置上下拉 */
/* kernel: drivers/gpio/gpio-xxx.c */
static int my_gpio_set_config(struct gpio_chip *gc,
                               unsigned offset,
                               unsigned long config)
{
    /* config 由 PIN_CONF_PACKED() 宏打包 */
    enum pin_config_param param = pinconf_to_config_param(config);
    switch (param) {
    case PIN_CONFIG_BIAS_PULL_UP:
        /* 使能上拉寄存器 */
        break;
    case PIN_CONFIG_BIAS_PULL_DOWN:
        /* 使能下拉寄存器 */
        break;
    case PIN_CONFIG_BIAS_DISABLE:
        /* 禁用上下拉 */
        break;
    }
}

施密特触发器（Schmitt Trigger）

来源：QuecPython GPIO 文档
"TTL Schmitt trigger: Process noise or unstable input data. In most cases, one TTL Schmitt trigger will be embedded to eliminate the noise of digital signal."
QuecPython GPIO Documentation

什么是施密特触发器？

施密特触发器是一种具有滞回特性的比较电路，内嵌于 GPIO 输入通道。它通过设置两个不同的阈值电压（上升阈值 VT+ 和下降阈值 VT-），使得信号在上升和下降时拥有不同的切换点，从而有效抑制噪声引起的误触发。

图6：施密特触发器传输特性曲线与噪声抑制效果（滞回窗口防止噪声误触发）

施密特触发器的工作原理

📈 上升过程

输入从低电平上升时，只有当电压超过上升阈值 VT+，输出才翻转为高电平。在此之前即使信号已越过 VDD/2，输出仍保持低电平。

📉 下降过程

输入从高电平下降时，只有当电压低于下降阈值 VT-，输出才翻转为低电平。两个阈值之间的"死区"就是滞回带（Hysteresis）。

💡 Hysteresis 数值参考（权威来源）
来自 Raspberry Pi GPIO 电气规格：
  • VT-（下降阈值）：1.09 V ～ 1.16 V
  • VT+（上升阈值）：2.24 V ～ 2.74 V
  • 迟滞电压 Vhys：> 0.25 V

各工作模式下施密特触发器状态

GPIO 配置模式施密特触发器原因
浮空输入使能抑制浮空时的随机噪声
上拉/下拉输入使能提供滞回，稳定输入采样
推挽输出使能（回读）允许读回引脚实际电平
开漏输出使能（回读）开漏输出可能需要读取总线状态
模拟模式禁用模拟信号不需要数字整形，禁用以降低功耗
⚠️ 模拟模式注意事项（ST AN4899）
"When the pin is configured in Analog mode, the Schmitt Trigger and digital input path are physically disconnected. This reduces power consumption and improves analog measurement accuracy."
在模拟模式下，施密特触发器和数字输入路径被物理断开，读取输入数据寄存器 (IDR) 将始终返回 0。

去抖动（Debounce）

机械开关按下时会产生多次快速抖动。Linux 内核 GPIO 子系统提供了软件去抖动支持：

来源：Linux Kernel GPIO Driver Interface
"Debouncing is a configuration set to a pin indicating that it is connected to a mechanical switch or button, or similar that may bounce. Bouncing means the line is pulled high/low quickly at very short intervals for mechanical reasons. This can result in the value being unstable or irqs firing repeatedly unless the line is debounced. Debouncing in practice involves setting up a timer when something happens on the line, wait a little while and then sample the line again, so see if it still has the same value (low or high)."
docs.kernel.org — GPIO lines with debounce support 
/* 在 Device Tree 中配置去抖动时间 */
button-gpio {
    compatible = "gpio-keys";
    debounce-interval = <20>;  /* 20ms 去抖动 */
};

/* 或通过 gpio_chip .set_config() 回调设置 */
/* config = PIN_CONFIG_INPUT_DEBOUNCE, argument = debounce_ms */

驱动能力与速率控制

驱动能力（Drive Strength）

驱动能力定义了 GPIO 引脚在维持有效逻辑电平（VOH/VOL规范范围内）的前提下，能够提供或吸收的最大电流。

来源：Infineon PSoC6 GPIO PDL 手册
"Drive strength can be defined to select how much of the IO driver is enabled to provide additional control of the rising and falling edge slew rates. Drive strength options include: Full (Default), Half, Quarter, Eighth. The Full, Half, Quarter and Eighth setting is for Drive strength that determines how hard to drive parallel circuit capacitance. Full has the most cap drive hence the fastest slew rate."
Infineon Community — GPIO Driver Strength
图7：不同驱动能力配置下的输出电压-电流特性（驱动能力不限制实际电流，仅保证规格范围）
⚠️ 重要：驱动能力的真实含义（Raspberry Pi 官方数据）
"Source/sink current capability does not limit the current into or out of the pin, but only specifies the maximum current for which the output signal high/low voltage specifications will be met. If misused, output pins can be damaged by excessive current irrespective of the source/sink current programmed."
Mosaic Industries — RPi GPIO Electrical Specifications

驱动电流配置不是电流限制器，是电压规格保证。超出配置仍然会输出电流，但会损坏引脚！

典型驱动电流规格

平台最小驱动电流最大驱动电流默认配置粒度
Raspberry Pi (BCM283x)2 mA16 mA8 mA2 mA 步进，按组配置
STM32（通用 GPIO）约 4 mA约 25 mA（单引脚）低速4 档：低/中/高/超高速

压摆率（Slew Rate）控制

压摆率定义了信号从低电平到高电平（或反之）的切换速度，单位为 V/ns 或通过输出速度档位（Speed Grade）间接控制。

来源：GitHub periph 项目讨论（含硬件厂商参考）
"Slew rate is the duration for an output GPIO pin to go from one level to another. It is usually specified in nanosecond. Fast slew rate can generate EMI interference due to overshoot and can interfere with neighbouring pads (like SDCard and DRAM). A slow slew rate limits the frequency at which communication can happen. It is essentially a low-pass filter."
google/periph GitHub Issue #239
图8：快速压摆产生过冲和 EMI，慢速压摆改善信号质量但限制最高频率
💡 选择压摆率的原则
"The rule-of-thumb: go as low and slow as your design can tolerate. If you need to drive more capacitance and/or at a faster output clock rate you want to favor a faster slew rate otherwise you risk your output looking like a pattern of triangle waves."
来源：Infineon PSoC6 GPIO 驱动能力文档

实践规则：选择能满足时序要求的最低速度档位，以减少 EMI 发射、降低相邻走线串扰。

STM32 速度配置寄存器

/* STM32 OSPEEDR 寄存器配置 GPIO 速度 */
/* 每引脚占 2 位 */

/* 低速：最大 ~2MHz，低驱动能力，低 EMI */
GPIOx->OSPEEDR &= ~(0x3 << (pin * 2));  /* 00 */

/* 高速：最大 ~100MHz，高驱动能力，需关注 EMI */
GPIOx->OSPEEDR |= (0x3 << (pin * 2));   /* 11 */

/* STM32 HAL 库写法 */
GPIO_InitStruct.Speed = GPIO_SPEED_FREQ_LOW;      /* 2 MHz  */
GPIO_InitStruct.Speed = GPIO_SPEED_FREQ_MEDIUM;   /* 25 MHz */
GPIO_InitStruct.Speed = GPIO_SPEED_FREQ_HIGH;     /* 50 MHz */
GPIO_InitStruct.Speed = GPIO_SPEED_FREQ_VERY_HIGH;/* 100 MHz */
⚠️ 总电流限制（RPi 官方警告）
"Current sourced by the outputs is drawn from the 3.3 V supply, which can supply only 50 mA maximum. Consequently, the maximum you can source from all the GPIO outputs simultaneously is less than 50 mA."
来源：Mosaic Industries — Raspberry Pi GPIO Electrical Specifications

驱动多个 GPIO 时需注意整体功耗，单引脚限制 × 引脚数 ≠ 总限制！

gpio_chip 结构体（核心抽象）

来源：Linux Kernel GPIO Driver Interface
"In the gpiolib framework each GPIO controller is packaged as a 'struct gpio_chip' with members common to each controller of that type, these should be assigned by the driver code: methods to establish GPIO line direction, methods used to access GPIO line values, method to set electrical configuration for a given GPIO line, method to return the IRQ number associated to a given GPIO line..."
docs.kernel.org 
/* linux/gpio/driver.h — gpio_chip 核心成员 */
struct gpio_chip {
    const char         *label;           /* 调试标签 */
    struct device      *parent;          /* 父设备 */

    /* 方向控制 */
    int    (*get_direction)(struct gpio_chip *gc, unsigned offset);
    int    (*direction_input)(struct gpio_chip *gc, unsigned offset);
    int    (*direction_output)(struct gpio_chip *gc, unsigned offset, int value);

    /* 读写值 */
    int    (*get)(struct gpio_chip *gc, unsigned offset);
    void   (*set)(struct gpio_chip *gc, unsigned offset, int value);
    void   (*set_multiple)(struct gpio_chip *gc, unsigned long *mask,
                           unsigned long *bits);

    /* 电气配置（上下拉、开漏、去抖动等）*/
    int    (*set_config)(struct gpio_chip *gc, unsigned offset,
                         unsigned long config);

    /* IRQ 映射 */
    int    (*to_irq)(struct gpio_chip *gc, unsigned offset);

    /* 是否可睡眠（I2C/SPI 扩展器必须设为 true）*/
    bool   can_sleep;

    /* GPIO 数量 */
    unsigned int  ngpio;

    /* 引脚名称数组（可选）*/
    const char * const *names;

    /* IRQ 芯片支持（可选）*/
    struct gpio_irq_chip irq;
};

Consumer API — 描述符接口

Linux Kernel 官方：推荐使用描述符接口
"All the functions that work with the descriptor-based GPIO interface are prefixed with gpiod_. The gpio_ prefix is used for the legacy interface. No other function in the kernel should use these prefixes. The use of the legacy functions is strongly discouraged, new code should use <linux/gpio/consumer.h> and descriptors exclusively."
docs.kernel.org — GPIO Descriptor Consumer Interface 
#include <linux/gpio/consumer.h>

/* 获取 GPIO 描述符（推荐：设备管理版本）*/
struct gpio_desc *led = devm_gpiod_get(dev, "led", GPIOD_OUT_LOW);
if (IS_ERR(led))
    return PTR_ERR(led);

/* gpiod_get flags 选项 */
// GPIOD_ASIS          — 不初始化，需后续设置方向
// GPIOD_IN            — 初始化为输入
// GPIOD_OUT_LOW       — 初始化为输出，初值 0
// GPIOD_OUT_HIGH      — 初始化为输出，初值 1
// GPIOD_OUT_LOW_OPEN_DRAIN  — 开漏输出，初值 0
// GPIOD_OUT_HIGH_OPEN_DRAIN — 开漏输出，初值 1

/* 设置方向 */
gpiod_direction_input(gpio_desc);
gpiod_direction_output(gpio_desc, 0);

/* 读写值（操作逻辑电平，自动处理 active-low）*/
int val = gpiod_get_value(gpio_desc);   /* 原子上下文，不可睡眠 */
gpiod_set_value(gpio_desc, 1);

/* 可睡眠版本（用于 I2C/SPI GPIO 扩展器）*/
int val = gpiod_get_value_cansleep(gpio_desc);
gpiod_set_value_cansleep(gpio_desc, 1);

/* GPIO 作为 IRQ 使用 */
int irq = gpiod_to_irq(gpio_desc);
request_irq(irq, handler, IRQF_TRIGGER_RISING, "my-gpio-irq", dev);

Active-Low 语义（逻辑 vs 物理电平）

Linux Kernel 原文
"The GPIO library has internal logic to handle lines that are active low, such as indicated by overstrike or #name in a schematic, and the driver should not try to second-guess the logic value of a line. Low and high values are defined as physical low on the line in/out to the connector such as a physical pad, pin or rail."
docs.kernel.org — GPIO level semantics
函数调用line 属性物理引脚电平
gpiod_set_value(desc, 0)active high（默认）LOW
gpiod_set_value(desc, 1)active high（默认）HIGH
gpiod_set_value(desc, 0)active lowHIGH（反转）
gpiod_set_value(desc, 1)active lowLOW（反转）
gpiod_set_value(desc, 0)open drainLOW（主动拉低）
gpiod_set_value(desc, 1)open drain高阻抗（Hi-Z）

IRQ 子系统集成

GPIO 控制器驱动通常同时充当中断控制器，通过 gpio_irq_chip 结构实现：

Linux Kernel 原文
"It is custom that GPIO drivers (GPIO chips) are also providing interrupts, most often cascaded off a parent interrupt controller, and in some special cases the GPIO logic is melded with a SoC's primary interrupt controller. The IRQ portions of the GPIO block are implemented using an irq_chip..."
docs.kernel.org — GPIO drivers providing IRQs
链式级联 GPIO IRQ
SoC 内置 GPIO 控制器常用。GPIO 中断处理程序从父 IRQ 直接调用，IRQ 禁用期间执行，不可使用 I2C 等慢速总线。
嵌套线程 GPIO IRQ
I2C/SPI GPIO 扩展器必须使用。通过线程处理中断，设置 can_sleep = true，天然 RT 兼容。
层次化中断控制器
每条 GPIO 线对应父中断控制器的一条专用 IRQ 线，无需查询 GPIO 硬件判断触发源。

Linux 驱动开发关键要点

✅ 核心开发规范

1. 始终使用描述符接口（gpiod_*）

官方文档明确废弃了基于整数的 gpio_* 旧接口，所有新代码必须使用 <linux/gpio/consumer.h> 中的描述符接口。

/* ❌ 旧接口（废弃，禁止在新代码中使用）*/
gpio_request(gpio_num, "my-gpio");
gpio_direction_output(gpio_num, 0);

/* ✅ 新接口（推荐）*/
struct gpio_desc *gpio = devm_gpiod_get(dev, "my", GPIOD_OUT_LOW);
2. 区分原子上下文与可睡眠上下文
Linux Kernel 官方说明
"Most GPIO controllers can be accessed with memory read/write instructions. Those don't need to sleep, and can safely be done from inside hard (non-threaded) IRQ handlers... Some GPIO controllers must be accessed using message based buses like I2C or SPI. Commands to read or write those GPIO values require waiting to get to the head of a queue to transmit a command and get its response. This requires sleeping, which can't be done from inside IRQ handlers."
docs.kernel.org — GPIO Consumer Interface 
/* 检查 GPIO 是否需要睡眠 */
if (gpiod_cansleep(desc)) {
    /* 必须在可睡眠上下文中使用 */
    gpiod_set_value_cansleep(desc, 1);
} else {
    /* 可以在中断上下文中使用 */
    gpiod_set_value(desc, 1);
}
3. 必须先设置方向再使用
官方警告
"Be aware that there is no default direction for GPIOs. Therefore, using a GPIO without setting its direction first is illegal and will result in undefined behavior!"
docs.kernel.org
4. 实时内核（RT）注意事项
Linux Kernel RT 合规要求
"The GPIO driver should not use spinlock_t or any sleepable APIs (like PM runtime) in its gpio_chip implementation (.get/.set and direction control callbacks) if it is expected to call GPIO APIs from atomic context on realtime kernels (inside hard IRQ handlers and similar contexts)."
docs.kernel.org — GPIO Driver Interface: Realtime considerations 
/* RT 内核中 GPIO IRQ 驱动检查清单：
   ✅ 使用 raw_spinlock_t 替换 spinlock_t
   ✅ 不在 .get/.set 回调中使用 PM Runtime 等可睡眠 API
   ✅ 链式 IRQ 处理程序中不使用 spinlock_t / 可睡眠 API
   ✅ 使用 IRQCHIP_IMMUTABLE 标志声明 irq_chip
*/
static const struct irq_chip my_gpio_irq_chip = {
    .name   = "my_gpio_irq",
    .irq_mask   = my_gpio_mask_irq,
    .irq_unmask = my_gpio_unmask_irq,
    .flags  = IRQCHIP_IMMUTABLE,
    GPIOCHIP_IRQ_RESOURCE_HELPERS,
};
5. Device Tree 中正确声明 GPIO
Linux Kernel GPIO Mappings 文档
"GPIOs mappings are defined in the consumer device's node, in a property named <function>-gpios, where <function> is the function the driver will request through gpiod_get()."
docs.kernel.org — GPIO Mappings 
/* Device Tree 声明 GPIO */
foo_device {
    compatible = "acme,foo";
    /* 格式：<&gpio控制器 引脚号 极性标志> */
    led-gpios   = <&gpio 15 GPIO_ACTIVE_HIGH>; /* 高有效 */
    power-gpios = <&gpio 1  GPIO_ACTIVE_LOW>;  /* 低有效 */
    /* 引脚控制：上拉、驱动能力、施密特触发 */
    pinctrl-0 = <&pinctrl_foo>;
    pinctrl-names = "default";
};

/* 驱动代码获取 GPIO */
struct gpio_desc *led   = devm_gpiod_get(dev, "led",   GPIOD_OUT_HIGH);
struct gpio_desc *power = devm_gpiod_get(dev, "power", GPIOD_OUT_HIGH);
/* power 为 active-low，但驱动设 1 = 逻辑有效 = 物理 LOW，透明处理 */

GPIO IRQ 开发完整示例

/* 完整的 GPIO 中断驱动示例（基于 Linux Kernel 官方文档示例）*/
#include <linux/gpio/driver.h>
#include <linux/irq.h>

struct my_gpio {
    struct gpio_chip gc;
    void __iomem *base;
};

static void my_gpio_mask_irq(struct irq_data *d)
{
    struct gpio_chip *gc = irq_data_get_irq_chip_data(d);
    irq_hw_number_t hwirq = irqd_to_hwirq(d);
    /* 1. 通知 gpiolib IRQ 状态变化 */
    gpiochip_disable_irq(gc, hwirq);
    /* 2. 操作硬件寄存器屏蔽中断 */
    /* writel(...) */
}

static void my_gpio_unmask_irq(struct irq_data *d)
{
    struct gpio_chip *gc = irq_data_get_irq_chip_data(d);
    irq_hw_number_t hwirq = irqd_to_hwirq(d);
    /* 1. 通知 gpiolib 先 */
    gpiochip_enable_irq(gc, hwirq);
    /* 2. 操作硬件寄存器使能中断 */
}

/* 必须为 const + IRQCHIP_IMMUTABLE（RT 内核要求）*/
static const struct irq_chip my_gpio_irq_chip = {
    .name             = "my-gpio-irq",
    .irq_mask         = my_gpio_mask_irq,
    .irq_unmask       = my_gpio_unmask_irq,
    .irq_set_type     = my_gpio_set_irq_type,
    .flags            = IRQCHIP_IMMUTABLE,
    GPIOCHIP_IRQ_RESOURCE_HELPERS,  /* 自动填充资源回调 */
};

static int my_gpio_probe(struct platform_device *pdev)
{
    struct my_gpio *g;
    struct gpio_irq_chip *girq;
    int irq;

    g = devm_kzalloc(&pdev->dev, sizeof(*g), GFP_KERNEL);

    /* 初始化 gpio_chip */
    g->gc.label           = "my-gpio";
    g->gc.parent          = &pdev->dev;
    g->gc.ngpio           = 32;
    g->gc.direction_input = my_gpio_direction_input;
    g->gc.direction_output= my_gpio_direction_output;
    g->gc.get             = my_gpio_get;
    g->gc.set             = my_gpio_set;
    g->gc.set_config      = gpiochip_generic_config; /* 委托给 pinctrl */
    g->gc.can_sleep       = false;  /* MMIO 寄存器，不需睡眠 */

    /* 配置 IRQ 支持 */
    irq = platform_get_irq(pdev, 0);
    girq = &g->gc.irq;
    gpio_irq_chip_set_chip(girq, &my_gpio_irq_chip);
    girq->parent_handler  = my_gpio_irq_handler;  /* 链式 IRQ 处理 */
    girq->num_parents     = 1;
    girq->parents         = devm_kcalloc(&pdev->dev, 1,
                                          sizeof(*girq->parents), GFP_KERNEL);
    girq->parents[0]      = irq;
    girq->default_type    = IRQ_TYPE_NONE;
    girq->handler         = handle_level_irq;

    return devm_gpiochip_add_data(&pdev->dev, &g->gc, g);
}

关键注意事项速查表

场景注意要点依据来源
IRQ 上下文使用 GPIO 仅可用 gpiod_get/set_value()（不可睡眠），扩展器 GPIO 不允许在硬 IRQ 中使用 kernel.org
GPIO 作 IRQ 输入 必须先 gpiod_direction_input()，用 gpiochip_lock_as_irq() 防止方向被改 kernel.org
RT 实时内核 用 raw_spinlock_t，irq_chip 声明 IRQCHIP_IMMUTABLE kernel.org
开漏/线与总线 用 GPIOD_OUT_LOW_OPEN_DRAIN，或 DT 中配置，不要在驱动中手动反转 kernel.org
多 GPIO 批量操作 使用 gpiod_get_array() + gpiod_set_array_value()，硬件支持时可原子操作整个 bank kernel.org
模块卸载资源释放 优先使用 devm_gpiod_get()，设备释放时自动归还，避免手动 gpiod_put() 泄漏 kernel.org
电气配置（上下拉/驱动力） 通过 DT pinctrl 节点配置，不要在 GPIO 驱动中硬编码；set_config() 可委托给 gpiochip_generic_config() kernel.org
未使用 GPIO 引脚 配置为上拉或下拉（避免浮空），降低功耗与 EMI ST AN4899 / embetronicx
