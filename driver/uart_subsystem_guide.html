① 第一性原理概述：为什么需要 UART 子系统？

1.1 通信的本质约束

从物理世界出发，两个设备之间的串行通信只有一条时间轴。在这条时间轴上，双方需要就三件事达成共识：

UART（Universal Asynchronous Receiver-Transmitter）的核心问题只有一个：如何让操作系统的上层应用，通过统一接口，访问千变万化的硬件串口控制器？

1.2 分层抽象的必然性

若不做抽象，每款 SoC 的 UART 寄存器地址不同、FIFO 深度不同、时钟配置不同，应用程序将与硬件紧耦合。Linux 内核的解法是三层分离：

🔬 第一性原理总结 §1

UART 子系统分层的根本原因是：物理世界的硬件多样性 + 软件世界的接口稳定性需求之间存在结构性矛盾。Linux 通过"策略与机制分离"原则，将"如何控制硬件"（uart_ops 回调）与"何时控制硬件"（serial_core 状态机）解耦。这是所有优秀系统软件设计的共同规律。

② TTY 层架构：从 /dev/ttyS0 到内核

2.1 TTY 是什么

TTY（Teletypewriter，电传打字机）层是 Linux 内核对所有终端设备的统一抽象。当用户程序 open("/dev/ttyS0") 时，VFS 层将调用转发给 TTY 驱动。

参考来源：The Linux Kernel TTY Documentation

2.2 核心数据结构

2.3 行规程（Line Discipline）

行规程是 TTY 层最重要的设计——它在驱动和用户空间之间插入一层"数据处理器"。默认的 N_TTY 行规程实现了：

  • 规范模式（Canonical）：按行处理，支持退格、Ctrl+C 等
  • 原始模式（Raw）：直接透传，串口通信常用此模式（清除 ICANON 标志）
  • 回显（Echo）控制
  • XON/XOFF 软件流控
📌 关键路径

接收数据：硬件中断 → 驱动 ISR → tty_insert_flip_char() → tty_flip_buffer_push() → 工作队列 → 行规程 receive_buf() → 用户 read()

🔬 第一性原理总结 §2

TTY 层的存在解决了一个根本矛盾：硬件 UART 产生的是原始字节流，而用户程序期望的是经过处理的文本行（或精确控制的原始字节）。行规程就是这个"契约"的实现——它让同一个串口硬件既能用于调试终端（规范模式），也能用于 GPS 模块通信（原始模式），无需修改驱动。

③ Serial Core 框架：三大数据结构

Serial Core（drivers/tty/serial/serial_core.c）是 Linux UART 子系统的核心，定义于 include/linux/serial_core.h。它通过三个关键结构体将 TTY 层与硬件驱动解耦。

参考：torvalds/linux serial_core.c

3.1 struct uart_driver — 驱动描述符

/* include/linux/serial_core.h */
struct uart_driver {
    struct module        *owner;
    const char           *driver_name;   /* 驱动名，如 "serial" */
    const char           *dev_name;      /* 设备名，如 "ttyS" */
    int                   major;         /* 主设备号 */
    int                   minor;         /* 次设备号起始 */
    int                   nr;            /* 该驱动管理的端口数 */
    struct console       *cons;          /* 关联控制台（可选）*/

    /* 以下字段由 serial_core 内部使用 */
    struct uart_state    *state;        /* 每端口状态数组 */
    struct tty_driver    *tty_driver;   /* 关联的 tty_driver */
};

3.2 struct uart_port — 端口描述符

struct uart_port {
    spinlock_t           lock;          /* 端口自旋锁 */
    unsigned long        iobase;        /* I/O 端口基址（x86 ISA）*/
    unsigned char        __iomem *membase; /* 内存映射基址（ARM/MIPS）*/
    resource_size_t      mapbase;       /* 物理地址 */
    unsigned int         irq;           /* 中断号 */
    unsigned int         uartclk;       /* UART 时钟（Hz）*/
    unsigned int         fifosize;      /* TX FIFO 大小 */
    unsigned char        x_char;        /* xon/xoff 字符 */
    unsigned int         mctrl;         /* 当前 modem 控制状态 */
    upf_t                flags;         /* UPF_* 标志位 */
    unsigned int         type;          /* PORT_8250, PORT_16550A 等 */
    unsigned int         iotype;        /* UPIO_MEM, UPIO_PORT 等 */
    unsigned int         regshift;      /* 寄存器地址移位（Ingenic=2）*/
    const struct uart_ops *ops;        /* ← 硬件操作回调 */
    struct uart_state   *state;        /* 关联的 uart_state */
    struct device       *dev;          /* 父设备（platform_device）*/
    uart_icount          icount;        /* 统计计数器 */
    /* serial_in / serial_out 可被厂商覆写 */
    unsigned int         (*serial_in)(struct uart_port*, int);
    void                 (*serial_out)(struct uart_port*, int, int);
};

🔬 第一性原理总结 §3

Serial Core 的设计本质是依赖倒置原则（DIP）：高层模块（TTY 层）不依赖低层模块（具体 UART 芯片驱动），两者都依赖于抽象（uart_ops）。uart_port.serial_in/out 的可覆写设计更进一步，允许厂商在不修改通用逻辑的情况下处理寄存器访问的差异性（如地址移位、字节序等）。

④ 通用寄存器：NS16550 标准

NS16550（1987年，National Semiconductor）是 PC 时代串口控制器的工业标准，至今几乎所有 SoC 厂商（包括君正）的 UART 都兼容此寄存器集。参考：16550 UART — Grokipedia

⚠️ 注意：君正 JZ4780 的地址步长

标准 16550 的寄存器间隔是 1 字节（regshift=0）。君正 JZ4780 UART 寄存器按 4 字节对齐（regshift=2），即偏移 0 对应物理地址 0x00，偏移 1 对应物理地址 0x04，依此类推。驱动中通过 uart.port.regshift = 2 告知 serial_core。

波特率计算

/* 波特率 = UART时钟 / (16 × 除数) */
divisor = DIV_ROUND_CLOSEST(port->uartclk, 16 * baud);

/* 写入: 先置 LCR.DLAB=1，再写 DLL/DLM，最后清除 DLAB */

early_out(port, UART_DLL, divisor & 0xff);
early_out(port, UART_DLM, (divisor >> 8) & 0xff);
early_out(port, UART_LCR, UART_LCR_WLEN8);  /* 清除 DLAB */

/* 例：JZ4780 外部晶振 24MHz，目标波特率 115200 */
/* divisor = 24000000 / (16 × 115200) = 13.02 ≈ 13 (0x000D) */
/* 实际波特率 = 24000000 / (16 × 13) = 115384 bps（误差 0.16%）*/
🔬 第一性原理总结 §4

NS16550 寄存器集的精妙之处在于：用 DLAB（除数锁存访问位）这一"地址复用"技巧，在同一物理偏移上区分了"运行时数据寄存器"和"初始化配置寄存器"。这以最小的寄存器数量支持了全部功能。FCR 的 FIFO 使能设计也体现了"向后兼容"原则：上电默认关闭 FIFO，让旧软件无需修改即可工作。

6.4 驱动源码关键路径分析

① 寄存器读写适配（处理 regshift=2）

/* 源码：drivers/tty/serial/8250/8250_ingenic.c */

/* 君正特有：FCR 需要额外置位 UME（UART Module Enable）*/
#define UART_FCR_UME    BIT(4)   /* 君正 UFCR 第4位 = 模块使能 */
#define UART_MCR_MDCE   BIT(7)   /* 调制解调器控制使能 */
#define UART_MCR_FCM    BIT(6)   /* 流控模式 */

static void ingenic_uart_serial_out(struct uart_port *p,
                                    unsigned int offset, u32 value)
{
    u32 ier;
    switch (offset) {
    case UART_FCR:
        /* 每次写 FCR 都必须同时置 UME 位，否则模块被关闭 */
        value |= UART_FCR_UME;
        break;
    case UART_IER:
        /* 接收超时中断与行状态中断绑定（bit2 = RLSIE 时同时开 RTOIE）*/
        value |= (value & 0x4) ? UART_IER_RTOIE : 0;
        break;
    case UART_MCR:
        /* Modem 流控：依据 IER.MSIE 决定是否使能 MDCE+FCM */
        ier = p->serial_in(p, UART_IER);
        if (ier & UART_IER_MSI)
            value |= UART_MCR_MDCE | UART_MCR_FCM;
        else
            value &= ~(UART_MCR_MDCE | UART_MCR_FCM);
        break;
    }
    /* regshift=2：offset 左移2位得到字节地址 */
    writeb(value, p->membase + (offset << p->regshift));
}

static u32 ingenic_uart_serial_in(struct uart_port *p, unsigned int offset)
{
    u8 value = readb(p->membase + (offset << p->regshift));
    /* 屏蔽上层不可见的君正私有位，维护16550兼容性 */
    switch (offset) {
    case UART_FCR: value &= ~UART_FCR_UME;          break;
    case UART_MCR: value &= ~(UART_MCR_MDCE|UART_MCR_FCM); break;
    }
    return value;
}

② probe 函数：驱动与端口注册

static int ingenic_uart_probe(struct platform_device *pdev)
{
    struct uart_8250_port uart = {};
    struct ingenic_uart_data *data;
    const struct ingenic_uart_config *cdata;

    cdata = of_device_get_match_data(&pdev->dev);

    /* 配置 uart_port */
    uart.port.type      = PORT_16550A;   /* 声明兼容类型 */
    uart.port.flags     = UPF_SKIP_TEST | UPF_IOREMAP | UPF_FIXED_TYPE;
    uart.port.regshift  = 2;             /* 4字节对齐！ */
    uart.port.fifosize  = cdata->fifosize;  /* jz4780=64 */
    uart.tx_loadsz      = cdata->tx_loadsz; /* jz4780=32 */
    uart.capabilities   = UART_CAP_FIFO | UART_CAP_RTOIE;

    /* 覆写 I/O 访问函数 */
    uart.port.serial_out = ingenic_uart_serial_out;
    uart.port.serial_in  = ingenic_uart_serial_in;

    /* 双时钟：module clk（控制器门控）+ baud clk（波特率源）*/
    data->clk_module = devm_clk_get(&pdev->dev, "module");
    data->clk_baud   = devm_clk_get(&pdev->dev, "baud");
    clk_prepare_enable(data->clk_module);
    clk_prepare_enable(data->clk_baud);

    uart.port.uartclk = clk_get_rate(data->clk_baud);

    /* 向 8250 core 注册，它会再调用 uart_add_one_port */
    data->line = serial8250_register_8250_port(&uart);
    return 0;
}

/* 早期控制台不依赖设备模型，直接操作寄存器 */
static int __init ingenic_earlycon_setup_tail(
    struct earlycon_device *dev, const char *opt)
{
    unsigned int divisor = DIV_ROUND_CLOSEST(
        port->uartclk, 16 * baud);

    early_out(port, UART_IER, 0);                    /* 关闭所有中断 */
    early_out(port, UART_LCR, UART_LCR_DLAB|UART_LCR_WLEN8);
    early_out(port, UART_DLL, 0); early_out(port, UART_DLM, 0);
    early_out(port, UART_LCR, UART_LCR_WLEN8);
    /* 使能 FIFO + UME（君正专属位）*/
    early_out(port, UART_FCR,
        UART_FCR_UME | UART_FCR_CLEAR_XMIT |
        UART_FCR_CLEAR_RCVR | UART_FCR_ENABLE_FIFO);
    early_out(port, UART_MCR, UART_MCR_RTS | UART_MCR_DTR);
    /* 重新设置波特率 */
    early_out(port, UART_LCR, UART_LCR_DLAB|UART_LCR_WLEN8);
    early_out(port, UART_DLL, divisor & 0xff);
    early_out(port, UART_DLM, (divisor >> 8) & 0xff);
    early_out(port, UART_LCR, UART_LCR_WLEN8);
    return 0;
}

6.5 双时钟设计原理

💡 君正双时钟机制

baud clk：来自外部晶振（ext，通常 24MHz），是波特率计算的频率基准。uartclk = clk_get_rate(clk_baud)

module clk：来自 CGU（时钟管理单元），用于 AHB/APB 总线访问门控。系统休眠时可单独关闭，降低功耗。

这种分离确保了：即使总线时钟变化，波特率（依赖 ext 晶振）仍然精确稳定。

🔬 第一性原理总结 §6

君正 JZ4780 驱动的核心洞察：它不是"另起炉灶"写一个全新的串口驱动，而是继承 8250 核心框架，仅覆写差异点（serial_in/out 函数指针 + FCR 的 UME 位 + 双时钟管理）。这一策略最小化了维护代价：通用的 FIFO 管理、中断处理、波特率计算逻辑全部复用，只有"哪里不同就改哪里"。这正是开源内核驱动开发的最佳实践。

🔬 第一性原理总结 §7

UART 数据流的设计体现了中断驱动 + 缓冲解耦的核心思想。TX 方向：用环形缓冲区（xmit）解耦用户写入速度与硬件发送速度，THRE 中断做"水位控制"。RX 方向：flip buffer 解耦中断上下文（快速存储）与行规程处理（可慢速）。IIR 的优先级编码确保了多类中断同时发生时的确定性处理顺序——这是硬件设计者为软件工程师提前解决并发问题的体现。

🔬 最终总结：第一性原理全局视角

问题本质：如何用一套软件框架，让成百上千种 UART 硬件对用户程序呈现统一接口？

解法层次：① 物理层（16550 寄存器标准化） → ② 驱动层（uart_ops 接口规范化） → ③ 核心层（serial_core 状态管理） → ④ TTY 层（行规程标准化） → ⑤ VFS 层（文件接口统一化）

君正的贡献：在保持 16550 兼容性的同时，通过 regshift=2、FCR_UME、双时钟管理三个最小化修改点，将定制硬件无缝接入标准框架——这是"最小有效差异"设计原则的完美案例。
