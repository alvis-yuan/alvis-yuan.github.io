BLE 控制器 Link Layer 报文解析与分流

对照 Bluetooth Core v4.2 — Controller Volume（Link Layer / Vol 6 Part B） 与 honor50_sunmi_ble_ota 抓包，说明控制器在收到空口比特后如何解析并分流。 结构参考 经典蓝牙基带分流图，将 BR/EDR 的「基带头 TYPE / 载荷 L_CH」 映射为 LE 的「信道类型 + Data PDU 头 LLID / LLCP Opcode」。

参考材料
· 规范：Core_v4.2-Core System Package [Low Energy Controller volume].pdf（Link Layer 第 2 章 PDU、第 2.4 节 LLCP）
· 抓包：honor50_sunmi_ble_ota.csv（与 .btt 同场景；工作区未检出 .btt 时以 CSV 中 Data 列为准）
· 图示范式：baseband_packet_parse_flow.html

1. 与经典蓝牙（baseband_parse）的分层对照

阶段BR/EDR 控制器BLE 控制器（本页）
空口同步 72-bit Access Code + 18-bit 基带头 Advertising：Access Address 固定；Connected：Access Address 由 CONNECT 分配（如 0x93F4E4CB）
一级分流 基带头 TYPE（NULL/SCO/ACL） 物理信道：Advertising Channel vs Data Channel（由当前状态机决定监听哪类信道）
二级分流 ACL 载荷头 L_CH（LMP / L2CAP） Data Channel PDU 头 LLID（LL Data vs LL Control）
控制面 LMP（固件内部，HCI 不可见或仅事件） LLCP（LL Control PDU，Opcode 分流；Wireshark 称 LLCP）
数据面 L2CAP → Host LL Data PDU → L2CAP/ATT → Host（HCI ACL Data / LE Meta）

抓包工具中的 LE-C Packet = 已在控制器侧完成 CRC 校验后的 Data Channel PDU（含 2 字节 LL 头 + 载荷 + 3 字节 CRC）； LLCP xxx 行通常仅显示 LL Control 载荷（无 LL 头与 CRC）。

字段说明（对应分流树内报文头）

说明框 A · 广播信道 PDU Header（2 字节）
  • PDU Type（4 bit）一级分流字段；如 ADV_IND、SCAN_REQ、CONNECT_IND 等。
  • RFU / ChSel / TxAdd / RxAdd按 PDU 类型解释；CONNECT_IND 中 TxAdd/RxAdd 标识 InitA/AdvA 地址类型。
  • Length（6 bit）载荷长度（字节）；CONNECT_IND 后随固定格式连接参数。
CONNECT_IND 载荷主要字段
  • InitA / AdvA发起方与广播方设备地址（各 6 字节）。
  • AA（Access Address）连接态 4 字节接入地址，数据信道 thereafter 使用。
  • CRCInit连接态 CRC 初始化（3 字节）。
  • WinOffset / Interval / Latency / Timeout首个连接事件窗口、间隔、从机延迟、监督超时。
  • ChM / Hop / SCA信道图、跳频增量、睡眠时钟精度。
  • 解析结果建立连接态，跳转到 Data Channel 调度。
说明框 B · 数据信道 PDU Header（LLID 分流）
  • LLID = 11LL Control PDU → 首字节 Opcode 二级分流（LLCP），在链路层状态机内处理。
  • LLID = 10LL Data 起始分片 → 上送 L2CAP（可能经 HCI 重组）。
  • LLID = 01LL Data 延续分片；若 Length=0 则为 Empty PDU，仅维持 ACK/序列号。
  • LLID = 00保留。
  • NESN / SN / MD序列号与 More Data；配合硬件 ACK 与分片重组。
  • Length载荷长度（字节），不含 2 字节 LL 头与 3 字节 CRC。
说明框 C · 常见 LLCP Opcode（honor50 首次连接可见）
  • 0x08 / 0x09LL_FEATURE_REQ / LL_FEATURE_RSP — 特性交换。
  • 0x14 / 0x15LL_LENGTH_REQ / LL_LENGTH_RSP — 最大 PDU 长度协商。
  • 0x00LL_CONNECTION_PARAM_REQ / UPDATE — 连接参数更新。
  • 0x18–0x1APHY REQ / RSP / UPDATE（BLE 5.x）。
  • 0x01LL_CHANNEL_MAP_IND — 信道图更新。
  • 0x02LL_TERMINATE_IND — 断开（如原因 0x13）。

5.3 Empty LE（LLID=01, Length=0）

0D 00 ← LLID=01 (Continuation), Length=0 80 AF 8E ← 可能含 CRC / 或分析器附加字段

连接保持期大量 Empty LE（约每 15ms 连接事件）：控制器只跑 ARQ/流控，不上报 ATT。 第二次连接抓包几乎全是此类 PDU，故 Wireshark 无 ATT 行。

5.4 从 LL 头解析 LLID/Length 的位运算（与 BR/EDR 读 TYPE 同级）

/* 设 Octet0 = B0, Octet1 = B1 */ LLID = B0 & 0x03; NESN = (B0 >> 2) & 1; SN = (B0 >> 3) & 1; MD = (B0 >> 4) & 1; Length = ((B0 >> 5) & 0x07) | ((B1 & 0x1F) << 3); /* 共 8 bit，最大 255 */ if (LLID == 3) → 解析 Payload[0] 为 Opcode，查表分发 LLCP； if (LLID == 2) → L2CAP 片起始；if (LLID == 1) → 延续或空载荷。

7. 小结

  • BLE 控制器与 BR/EDR 一样，核心都是用头部位域做硬件/固件分流；LE 用 LLID 代替 ACL 的 L_CH。
  • LLCP 仅在 LLID=11 时进入，在控制器内终结；ATT 只在 LL Data 路径出现。
  • 分析 honor50_sunmi_ble_ota 时：看到 LE-C 先拆 2 字节 LL 头，再看 Opcode 或 L2CAP；LLCP xxx 行已是控制载荷视图。
  • 规范细节（CTE、CIS、扩展长度等）以 Core v4.2 Controller PDF 第 2 章为准；本页覆盖 honor50 OTA 场景中最常见的 Adv + LLCP + Empty + ATT 路径。
