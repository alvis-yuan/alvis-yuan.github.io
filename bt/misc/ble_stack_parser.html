📶 BLE 协议栈自底向上报文解析流程 (Core v5.2)

当一个 BLE 空口射频信号抵达天线时，它经历了怎样的层层“剥壳”与“路由”，最终变成应用层的业务数据？
以下按照自底向上（Bottom-Up）的顺序详细解析。

第一层：物理层 (PHY) - 空口信号捕获 Controller
前导码
1-2 Bytes
接入地址 (Access Address)
4 Bytes
PDU + CRC
传递给上层
🎯 PHY 层解析与过滤逻辑：
射频前端捕获电磁波，通过检查 前导码(Preamble) 完成频率同步与 AGC（自动增益控制）。
提取 接入地址 进行第一道硬件过滤：广播包匹配固定地址 `0x8E89BED6`；连接包匹配协商好的 32 位随机地址。
若地址匹配，剥离 PHY 头部，将后续比特流推送到链路层。
字段作用说明
Preamble交替的 0/1 序列，用于接收端硬件时钟同步。1M PHY 固定 1 字节，2M PHY 固定 2 字节。
Access Address区分不同物理信道的核心标识。决定了该报文是发给自己的还是旁路的干扰包。
⬆
第二层：链路层 (LL) - 链路维护与逻辑路由 Controller
PDU 报头
(含 LLID, Length)
2 Bytes
载荷 (Payload)
[ Opcode+Data | L2CAP PDU ]
0 ~ 251 Bytes
MIC
4 Bytes (加密)
CRC
3 Bytes
🔀 LL 层解析与路由逻辑：
执行硬件 CRC 校验 和 MIC 解密校验，失败则丢弃并请求重传 (ARQ)。
读取报头中的 LLID (逻辑链路标识符) 决定路由方向：
➔ 若 LLID = 0b11：识别为 LL 控制包。直接读取载荷第1字节的 Opcode（如连接更新），本层截留消化，不上报。
➔ 若 LLID = 0b01/0b10：识别为 用户数据 (ACL-U)。剥离 LL 头尾，将载荷封装通过 HCI 上送。
字段作用说明
LLID2 bits。`01`: 延续分片；`10`: 起始分片；`11`: 控制 PDU。
SN / NESN各 1 bit。序列号与预期下一序列号，用于底层包的确认与重传机制。
Opcode (仅控制PDU)载荷首字节。定义具体的链路层控制操作（如 `LL_FEATURE_REQ`）。
⬆
第三层：主机控制器接口 (HCI) - 物理总线桥梁 Bridge (UART/USB)
HCI Packet Type
1 Byte (如 0x02 ACL)
Handle + Flags(PB,BC)
2 Bytes
Data Length
2 Bytes
载荷 (L2CAP 帧分片)
Variable
🌉 HCI 层解析与组装逻辑 (Host 端)：
Host 接收到串口/USB数据，首先读取包类型（指令/事件/ACL数据/同步数据）。对于用户数据，通常是 ACL Data (0x02)。
提取 Connection Handle 匹配对应的逻辑链路。
检查 PB Flag (封包边界标志)：若为首包(First packet)，准备接收；若为延续包(Continuation)，则将载荷与前包拼接。
依据 Data Length 接收完毕后，将完整的 L2CAP 报文上抛。
字段作用说明
Connection Handle12 bits。控制器分配给主机的连接句柄，标识数据属于哪个设备的哪条链路。
PB Flag2 bits。标识 L2CAP PDU 的边界（起始分片、延续分片），用于 Host 端的重组。
⬆

PDU Length
2 Bytes
通道 ID (CID)
2 Bytes
控制字段(可选)
流控模式存在
载荷 (ATT / SMP Data)
Variable
🛤️ L2CAP 层解析与分发逻辑：
提取报头 CID (Channel ID) 进行协议解复用（路由多路分离）：
➔ 若 CID = 0x0004：将载荷送往 ATT (属性协议) 模块。
➔ 若 CID = 0x0006：将载荷送往 SMP (安全管理器) 处理配对加密。
若处于增强重传模式 (ERTM) 或基于信用的流控模式，还会解析可选的 控制字段 处理序列号及 SDU 重组。
字段作用说明
Length指示后续载荷（不含 Header）的字节总长度。
CID通道终点标识。0x0004(ATT), 0x0006(SMP), 0x0005(L2CAP信令通道)。
⬆
第五层：属性协议 (ATT) - 数据交互方法 Host
操作码 (Opcode)
1 Byte
属性句柄 (Handle)
2 Bytes
属性值 (Value) / 错误码
Variable
⚙️ ATT 层解析与执行逻辑：
解析首字节 Opcode，确定客户端发来的指令意图。
➔ 如 `0x0A` (Read Req)：提取 Handle，在本地数据库查找对应数据并组装 Response 发回。
➔ 如 `0x12` (Write Req)：提取 Handle 和 Value，覆盖本地数据库，触发上层应用回调。
➔ 如 `0x1B` (Notify)：客户端接收到通知，直接提取数据抛给 GATT/App。
字段作用说明
Opcode定义六种 PDU 类型：请求(REQ)、响应(RSP)、命令(CMD)、通知(NTF)、指示(IND)、确认(CFM)。
Handle0x0001 - 0xFFFF，属性在数据库中的唯一内存索引地址。
⬆
第六层：应用层 (GATT / Profile) - 业务语义赋予 Application
(逻辑结构无固定包头，完全依赖 Profile 规范解析二进制 Value)
💡 业务解析逻辑（以心率服务为例）：
ATT 层上报 Handle 0x0012 更新了数据 [0x06, 0x4B]。
GATT 查表得知 Handle 0x0012 对应 UUID 0x2A37 (心率测量特征)。
App 按照《心率 Profile 规范》解析原始字节：
- Byte 0 (0x06 -> `0000 0110`)：Flags 字段，指示格式为 uint8，且传感器接触正常。
- Byte 1 (0x4B -> `75`)：心率值为 75 bpm。
UI 界面更新心率数字，完成整个从物理电磁波到用户视觉感知的闭环流转。
