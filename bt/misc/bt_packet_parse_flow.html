🚀 第二部分：自下而上的解析与过滤流水线 (Bottom-Up)

1. 基带层 (Baseband)

物理跳频与链路控制
微网门卫 & 纠错引擎

🛡️ 过滤条件：我该收这个包吗？

  • 匹配 Access Code (访问码)： 根据 Master 的 MAC 地址生成。如果访问码不对，说明是隔壁微网的包，直接丢弃（防串台）。
  • 匹配 LT_ADDR (逻辑传输地址)： 占 3 个比特（1-7）。一台 Master 最多连 7 个 Slave。Slave 检查这个地址，如果是发给自己的（或者广播地址 0），才唤醒后续处理电路。

⚙️ 本层处理：包类型解析大作战

  • 检查 TYPE (4 Bits) 决定报文类型：
    - NULL / POLL： 没数据的空包，仅用于 ACK 或主设备轮询，直接硬件消化。
    - SCO (HV1/HV3)： 同步语音包！不需要重传，直接把 Payload 剥离，通过 PCM 硬件总线 送给声卡解码（不走上面的蓝牙协议栈）！
    - ACL (DH1/DH3/DH5)： 异步数据包！DH3/DH5 代表这是一个占用多时隙的高速长包。
  • 提取 ARQN 与 SEQN 进行应答和重传控制；校验通过后，分离出 ACL 数据，送至上层 HCI。

2. HCI 接口

主机控制接口
总线拼图站

🛡️ 过滤条件：连接映射

  • 从底层串口或 USB 收到 `HCI ACL Data` 包。
  • 提取 Connection Handle (12 Bits)，映射到 OS 内存中的物理设备上下文（“哦，这是那台打印机发来的”）。

⚙️ 本层处理：大包组装 (Reassembly)

  • 检查 HCI 头部的 PB Flag (Packet Boundary)。
  • 因为经典蓝牙受限于时隙，一个大的 L2CAP 包可能被切成了几十个 DH5 包。HCI 层负责将这些分片缓存，直到收到完整的结束包，拼成一个完整的 L2CAP PDU，再往上投递。

3. L2CAP 层

逻辑链路控制与适配层
多路复用大都会

🛡️ 过滤条件：分发给哪个协议？

  • 读取 2 字节的 CID (Channel ID)。经典蓝牙不像 BLE 只有几个固定的 CID，它是动态分配的。
  • - 若 CID = `0x0001`：底层信令（协商 MTU）。
    - 若 CID = `0x0040` (举例)：SDP 服务发现请求。
    - 若 CID = `0x0041` (举例)：RFCOMM 数据！
    - 若 CID = `0x0042` (举例)：AVDTP 音乐数据！

⚙️ 本层处理：拆头转发

  • 校验 L2CAP `Length` 字段。
  • 剥离 4 字节的 L2CAP 头部。
  • 根据 CID 的指示，将纯净的 Payload 注入到对应的上层协议处理机（例如本例中的 RFCOMM 引擎）。

4. RFCOMM 层

虚拟串口协议
RS-232 模拟器

🛡️ 过滤条件：去哪个虚拟串口？

  • 一台手机可能同时开着多个虚拟串口通信。读取 Address 字段 提取 DLCI (数据链路连接标识)。
  • DLCI = 0 是串口控制通道；DLCI > 0 对应着特定的 SPP 端口（例如 Server Channel 1）。

⚙️ 本层处理：解析串口控制帧

  • 读取 Control 字段：
    - 如果是 `SABM` 或 `UA`：代表打开或确认串口，更改底层状态机。
    - 如果是 `UIH` (Unnumbered Info with Header Check)：代表这是真实的串口 Payload 数据！
  • 校验包尾的 FCS (帧校验序列)，如果正确，提取 `UIH` 里的流式数据。

5. 应用层 (SPP)

手机 App (Serial Port Profile)
字节流终点站

🛡️ 过滤条件：Socket 绑定

  • 操作系统将 RFCOMM 提取出的数据流，直接推入到开发者创建的 BluetoothSocket 的缓冲区中。
  • 对于 App 开发者来说，底层的跳频、重传、通道标识全部被屏蔽。App 只需要认准这个连接对应的 Socket 对象。

⚙️ 本层处理：流式读取 (Stream)

  • App 触发异步读取事件：`InputStream.read(buffer)`。
  • 将获取到的字节流（例如 `"Hello World"` 或者串口通信的 JSON 指令）交给业务逻辑处理，打印日志或渲染在屏幕上。
  • 至此，一段经历跳频斩棘的串口数据，安全抵达目的地。
