SDP 数据库组织结构

Service Discovery Protocol 在设备本地维护一张服务记录表。远端通过 L2CAP PSM=0x0001 查询，得到每条服务的 Handle、类别、协议栈与 Profile 信息，再据此建立 HID、RFCOMM 等业务通道。

第一层：SDP 数据库 Database

整个数据库 = 多条服务记录（Service Record）的集合。每条记录有唯一的 32 位 Service Record Handle，由设备在注册服务时分配（通常从 0x00010001 起递增）。

要点：Handle 只是数据库内的索引，不是 L2CAP CID。客户端先通过 Service Search 按 UUID 找到 Handle，再用 Service Attribute Request 按 Handle 拉取完整属性列表。

第二层：单条服务记录 Service Record

一条服务记录 = 一组 (Attribute ID, Attribute Value) 键值对。Attribute ID 为 16 位无符号整数；Value 必须是符合 SDP 规范的数据元素（Data Element）。

Attribute ID名称作用
0x0000ServiceRecordHandle本条记录在库中的 Handle（必须与查询用的 Handle 一致）
0x0001ServiceClassIDList服务类别 UUID 列表，用于 Service Search 匹配
0x0004ProtocolDescriptorList连接该服务需走的协议栈（L2CAP → RFCOMM/HIDP 等）
0x0005BrowseGroupList浏览组，常含 PublicBrowseGroup
0x0006LanguageBaseAttributeIDList多语言字符串的基址偏移
0x0009BluetoothProfileDescriptorList使用的 Profile 及版本号
0x0100+ServiceName 等用户可见名称（依赖 LanguageBase 偏移）
0x0200+HID 扩展属性仅 HID 服务：描述符、子类、报告等

第三层：属性值的编码 Data Element

每个 Attribute Value 都是一个数据元素：首字节的高 5 位表示类型，低 3 位与后续字节表示大小。复杂结构用 Data Element Sequence (DES) 嵌套，形成树形 TLV。

0x35 DES — 元素序列（最常用，包一组子元素）
0x19 UUID — 16 位短 UUID（如 0x1124 = HID）
0x08 UINT — 无符号整数（通道号、版本等）
0x25 STRING — UTF-8 字符串（服务名）
0x28 BOOLEAN — 布尔值
0x09 UINT16 — 16 位属性 ID 在 PDU 里也会出现

抓包十六进制示例：09 00 01 35 03 19 11 24 → 属性 ID 0x0001，值为 DES，内含 UUID 0x1124（HID）。

组织方式小结

逻辑结构

  • 扁平多记录：无文件夹，靠 Handle 区分服务
  • 记录内树形：DES 嵌套表达协议栈、Profile 列表
  • 类型化值：UUID/UINT/STRING 等统一 Data Element 编码
  • Profile 扩展：HID/PnP 等在通用属性外增加 0x02xx 属性

与实现的对应

  • 协议栈在启动时 sdp_register_service() 类 API 写入各 Record
  • 修改服务需更新 Record 并保证 Handle 稳定或重新搜索
  • MTU 限制可能导致大 Attribute（HID Descriptor）分多次查询
  • 抓包中多次 L2CAP 建断 = 主机分轮查询，非数据库本身结构
