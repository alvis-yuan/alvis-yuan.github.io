🔑第一性原理：安全通信的两个根本问题

对比 TLS 握手协商，任何安全通信协议在本质上都必须解决且仅需解决两个根本问题：

🪪
问题一：身份认证（Who are you?）
确认通信对端是我所期望的那个设备，而非冒充者。在 TLS 中通过证书链完成；在 BLE 中因无 PKI 基础设施，依赖用户辅助（PIN、数字比较）或带外信道（OOB）完成。
🤝
问题二：密钥协商（What do we share?）
在不安全的无线信道上安全地建立共享密钥，用于后续对称加密。TLS 使用 ECDHE；BLE Legacy 使用 TK→STK；BLE Secure Connections 使用 ECDH→LTK。
🕵️
威胁一：被动窃听（Passive Eavesdropping）
攻击者记录所有配对数据包并离线破解密钥。BLE Legacy（TK=0 或 6 位数字）极易被暴力破解；LESC 的 ECDH 即使在完全透明的信道上也提供防窃听保护。
👾
威胁二：中间人攻击（MITM）
攻击者主动插入两端之间，分别与双方建立独立会话。必须通过用户参与（确认数字、输入 PIN）或安全带外信道来检测和阻断。Just Works 不提供此保护。
BLE 配对 vs TLS 握手 —— 第一性原理对照

🏛BLE 安全体系概览

BLE 的安全功能由 安全管理器（Security Manager, SM）定义，运行于固定 L2CAP 通道 0x0006 上的 安全管理协议（SMP）完成通信。BLE 安全架构包含五个特性：

📌 Bluetooth Core Specification v5.4 — Part H §2.1 原文
The Security Manager (SM) uses a key distribution approach to perform identity and encryption functionalities in radio communication. Pairing is performed to establish keys which can then be used to encrypt a link. A transport specific key distribution is then performed to share the keys which can be used to encrypt a link in future reconnections, verify signed data and random address resolution.
  • 配对（Pairing）：创建一个或多个共享密钥的过程
  • 绑定（Bonding）：将配对过程中产生的密钥持久化存储，以便后续重连
  • 设备认证（Device Authentication）：验证两个设备拥有相同的密钥
  • 加密（Encryption）：提供消息机密性（AES-CCM）
  • 消息完整性（Message Integrity）：防止消息被伪造

📅版本演进时间线

Bluetooth 4.0 · 2010年
LE Legacy Pairing 引入
BLE 首次引入，SMP 协议定义三种配对方法：Just Works、Passkey Entry、OOB。密钥协商基于对称加密（AES-128），临时密钥（TK）→短期密钥（STK）→长期密钥（LTK）分发。Just Works 的 TK=0，Passkey 最多仅 20 位熵，对被动窃听脆弱。
Bluetooth 4.1 · 2013年
小幅改进，Legacy 配对体系不变
主要引入拓扑改进（多角色）和 LE 重连优化。安全配对机制与 4.0 基本相同，仍使用 Legacy Pairing，被动窃听漏洞依然存在。
Bluetooth 4.2 · 2014年
LE Secure Connections (LESC) 引入 — 重大安全升级
引入基于 ECDH（P-256 曲线）的密钥交换，新增数字比较（Numeric Comparison）配对方法。LTK 由双方独立计算（而非分发），密钥生成函数升级为 AES-CMAC（f4, f5, f6, g2）。即使 Just Works 也可抵抗被动窃听。完全向后兼容 4.0/4.1。
Bluetooth 5.0 · 2016年 — 5.4 · 2023年
LESC 体系持续加固
速率提升（2M PHY、Coded PHY）、广播扩展等物理层改进，安全配对机制以 4.2 的 LESC 为基础持续完善。5.1+ 增加方向寻找特性。跨传输密钥导出（BR/EDR ↔ LE）机制标准化。
BLE 配对安全能力演进对比

📻BT 4.0/4.1 — LE Legacy Pairing

Legacy Pairing 是 BLE 最初的配对机制（Bluetooth 4.0，2010年），定义于 Core Spec Vol 3, Part H。其核心思路是：通过某种带外或用户辅助方式在双方建立一个临时密钥（Temporary Key, TK），再利用双方各自生成的随机数和 TK 推导出短期密钥（Short Term Key, STK）用于加密初始连接，然后在加密信道上分发真正的长期密钥（Long Term Key, LTK）用于重连。

⚠️ 关键安全缺陷
Legacy Pairing 不使用非对称加密。所有密钥交换信息（Confirm值、Random值）均在空中明文传输。若攻击者在配对期间抓取全部数据包，即可离线暴力破解 TK（Just Works：只有一个可能值 0；Passkey：仅 10^6 = 1,000,000 种可能）从而恢复 STK，解密整个会话。

🔐BT 4.2 — LE Secure Connections (LESC)

Bluetooth 4.2（2014年）引入 LE Secure Connections，这是 BLE 安全史上最重大的升级。核心变化是引入 ECDH（椭圆曲线 Diffie-Hellman，P-256 曲线）密钥交换，从根本上解决了 Legacy Pairing 对被动窃听的脆弱性。

✅ 核心改进
  • 引入 ECDH 公钥交换：即使攻击者捕获所有空中数据，也无法计算 LTK（离散对数难题）
  • 新增数字比较（Numeric Comparison）联想模型，提供用户可验证的 MITM 防护
  • LTK 由双方各自计算（非分发），使用 f5 函数
  • 使用 AES-CMAC 替代裸 AES 作为密码学基础（f4, f5, f6, g2 函数）
  • Passkey Entry 升级为逐比特确认流程，极大增强 MITM 抵抗力

The building block for the cryptographic functions f4, f5, f6, g2, h6, and h7 is the security function AES-CMAC. [RFC-4493 defines the Cipher-based Message Authentication Code (CMAC) that uses AES-128 as the block cipher function]

🔄三阶段配对架构（通用）

根据 Bluetooth Core Specification，无论 Legacy 还是 LESC，BLE 配对均分为三个阶段：

📌 Core Spec §2.1 原文
Pairing is a three-phase process. The first two phases are always used and may be followed by an optional transport specific key distribution phase: Phase 1: Pairing Feature Exchange · Phase 2 (LE legacy pairing): Short Term Key (STK) Generation · Phase 2 (LE Secure Connections): Long Term Key (LTK) Generation · Phase 3: Transport Specific Key Distribution
三阶段配对总体流程（Initiator = Central，Responder = Peripheral）

I/O 能力与配对方法选择

在阶段1，双方交换 I/O 能力，由规范定义的查表逻辑决定配对方法。I/O 能力包括：NoInputNoOutput、DisplayOnly、DisplayYesNo、KeyboardOnly、KeyboardDisplay。

⚡Legacy Pairing — Just Works

Just Works 是最简单的配对方法，专为无 I/O 能力的设备（如 NoInputNoOutput）设计。其核心特征是 TK = 0（全零 128 位值）。

第一性原理分析

🪪
身份认证
❌ 不提供任何身份认证。任何设备均可声称自己是目标设备。用户只需接受连接提示，无需验证任何信息。
🤝
密钥协商
TK=0，STK = s1(0, Srand, Mrand)。所有交换的 Confirm 和 Rand 值均明文传输，攻击者可直接计算 STK。
Legacy Pairing — Just Works MSC（消息序列图）

✅ 优点

  • 无需用户交互，零摩擦配对
  • 适用于无 I/O 能力的设备（传感器、信标等）
  • 配对速度最快

❌ 缺点

  • TK=0，STK 对被动窃听毫无保护
  • 无任何 MITM 防护
  • 任何人均可伪装成合法设备
  • 安全级别：Mode 1, Level 2（无认证）

🔢Legacy Pairing — Passkey Entry

Passkey Entry 适用于一方有显示屏、另一方有键盘的场景（例如 PC 与蓝牙键盘配对）。TK 是用户在两端输入的 6 位数字（0~999999），即 TK = passkey（范围 0x000000 ~ 0x0F423F）。

第一性原理分析

🪪
身份认证
⚠️ 有限的 MITM 防护。用户通过输入/确认 6 位数字来间接验证对端，但攻击者若未在线监听配对过程，无法得知 Passkey 是否被截获。理论上存在 MITM 攻击向量。
🤝
密钥协商
TK 仅有 10^6 种可能。攻击者若抓包，可离线暴力枚举所有 1,000,000 个 TK 值，验证 Confirm 值后恢复 STK。熵值仅约 20 位，对被动窃听脆弱。
Legacy Pairing — Passkey Entry MSC

✅ 优点

  • 提供基本 MITM 防护（用户输入验证）
  • 适用于键盘+显示屏组合设备
  • 安全级别高于 Just Works

❌ 缺点

  • TK 仅 20 位熵，对被动窃听脆弱
  • 抓包后可离线暴力破解
  • 用户体验较差（需手动输入）
  • 存在理论 MITM 攻击向量

📡Legacy Pairing — Out of Band (OOB)

OOB 配对通过 BLE 以外的信道（如 NFC）传输 TK。TK 可以是完整的 128 位随机数，提供最高的加密强度。安全性完全取决于 OOB 信道本身的安全性。

第一性原理分析

🪪
身份认证
✅ 依赖 OOB 信道的物理属性。NFC 的通信距离极短（几厘米），物理上难以实施中间人攻击。若 OOB 信道提供 MITM 防护，则 BLE 连接也获得同等保护。
🤝
密钥协商
✅ TK 可达 128 位随机数（充分熵值）。在加密强度上是三种 Legacy 方法中最强的，被动窃听的保护程度等同于 OOB 信道的安全性。
Legacy Pairing — OOB MSC

✅ 优点

  • TK 最多 128 位，加密强度最高
  • 若 OOB 信道安全则 MITM 防护最强
  • NFC 物理距离限制提供天然安全屏障
  • 无需用户手动输入数字

❌ 缺点

  • 需要额外 OOB 硬件（如 NFC 模块）
  • 增加设备成本和设计复杂度
  • 安全性依赖 OOB 信道，不可独立保证
  • 普及率较低

🛡LE Secure Connections — 总体流程

LESC 于 Bluetooth 4.2（2014年）引入，根本性地改变了 BLE 的密钥协商机制。引入 ECDH（P-256 曲线）公钥密码学，使得即使攻击者获取所有空中数据也无法推算出 LTK（离散对数问题的困难性）。

📌 Core Spec §2.3.5.6 — LE Secure Connections 阶段2总体流程
MacKey || LTK = f5(DHKey, Nc, Np, BD_ADDR_C, BD_ADDR_P)
其中 DHKey 是 ECDH 共享密钥，Nc/Np 是双方各自生成的随机 nonce，f5 基于 AES-CMAC 构造。
LE Secure Connections Phase 2 总体流程（以 Numeric Comparison / Just Works 为例）

⚡LESC — Just Works

LESC Just Works 与 Legacy Just Works 的用户体验相同（无需用户交互），但底层安全性有根本区别：ECDH 使得被动窃听者即使抓获全部数据包也无法恢复 LTK。

🪪
身份认证
❌ 仍不提供 MITM 防护。用户不参与验证任何信息，主动攻击者可替换公钥。但 Confirm/Random 交换确保双方计算了相同的 DHKey。
🤝
密钥协商
✅ ECDH 完全保护被动窃听。LTK 由 f5(DHKey,...) 派生，攻击者无法从公开的 PKa、PKb 推算 DHKey（P-256 ECDLP）。即使抓包也无法解密。
⚠️ MITM 仍是弱点
LESC Just Works 中，若攻击者能替换公钥（主动 MITM），仍可成功攻击。用户没有机会验证对端身份。因此安全级别为 Mode 1 Level 2（加密但无认证）。

🔢LESC — 数字比较（Numeric Comparison）

数字比较是 BLE 4.2 新引入的联想模型，仅适用于 LESC。双方独立计算相同的 6 位数字（由 g2 函数从公钥和 nonce 派生），用户确认两端显示的数字一致。这是目前 BLE 中 MITM 防护最强的方法之一。

🪪
身份认证
✅ 强 MITM 防护。6 位数字由双方公钥和 nonce 联合派生，中间人若替换公钥，将导致双方显示不同的数字，用户拒绝后配对终止。成功概率为 1/10^6。
🤝
密钥协商
✅ 同 LESC Just Works，ECDH 完全保护被动窃听。f5 函数从 DHKey 和双方 nonce、地址派生 LTK，攻击者无法离线推算。
📌 Core Spec §2.3.5.6.4 — 数字比较值计算
g2(PKax, PKbx, Na, Nb) = AES-CMAC_PKax(PKbx || Na || Nb) mod 10^6
该值由双方公钥 x 坐标和双方 nonce 共同决定，确保任何公钥篡改都会改变输出值。
LESC Numeric Comparison — 完整 MSC

✅ 优点

  • ECDH 完全防止被动窃听
  • 6 位数字验证提供强 MITM 防护
  • g2 函数将公钥绑定到显示值，篡改立即可检测
  • 最高安全级别：Mode 1, Level 4
  • 用户体验好（只需确认是/否）

❌ 限制

  • 仅 LESC 支持（BT 4.2+）
  • 双方均需显示屏和确认输入能力
  • 用户若不核对而直接点 YES 则防护失效
  • 比 Just Works 有额外的用户交互

🔑LESC — Passkey Entry（升级版）

LESC 中的 Passkey Entry 与 Legacy 版本用户体验类似（输入6位数字），但安全性有根本提升。不再将 Passkey 作为 TK 直接用于加密，而是将其逐比特嵌入 Confirm 函数的计算中，共执行 20 轮（每轮对应 passkey 的一位）。

📌 Core Spec §2.3.5.6.3 原文
In the Passkey Entry protocol, the most significant bit of Z is set equal to one and the least significant bit is made up from one bit of the passkey: if the passkey bit is 1, then Z = 0x81; if the passkey bit is 0, then Z = 0x80. The process iterates 20 times, once for each bit of the passkey.
LESC Passkey Entry — 核心流程（20轮逐比特确认）

✅ 优点（相比 Legacy Passkey）

  • 逐比特确认：无法离线暴力破解，必须在线猜测每位
  • ECDH 防止被动窃听（Legacy 版本不具备）
  • 安全级别达到 Mode 1, Level 4
  • 与 Legacy Passkey 相同用户体验

❌ 缺点

  • 20 轮通信增加配对延迟
  • 用户仍需手动输入数字
  • 仅 BT 4.2+ 支持

📲LESC — OOB

LESC OOB 通过带外信道传递的不是 TK，而是 公钥、nonce 和 Confirm 值，将身份验证与 ECDH 密钥交换深度结合。

📌 Core Spec §2.3.5.6.2 原文
For LE Secure Connections OOB, the OOB data contains the device's Confirm value Ca = f4(PKax, PKax, ra, 0) and random value ra, and these are transferred to the peer. If the Initiator has the Responder's OOB data, and vice versa, both can verify each other's identity.

LESC OOB 要求至少一方拥有对端的 OOB 数据即可启动（而 Legacy OOB 要求双方均有）。两者结合 ECDH，即使 OOB 信道仅提供单向认证，也能显著提升安全性。

✅ 优点

  • ECDH + OOB 双重防护，安全性最高
  • OOB 信道传递 Confirm+nonce，可验证公钥真实性
  • 最高安全级别（Mode 1, Level 4）

❌ 缺点

  • 需要额外 OOB 硬件
  • 实现复杂度最高
  • 普及率低

STK Short Term Key 最多128 bit 加密初始连接（仅 Legacy） Phase 2 结束，s1(TK, Srand, Mrand)
LTK Long Term Key 最多128 bit 重连时直接加密，是持久化的主密钥 Legacy: Phase 3 分发；LESC: Phase 2 由 f5 派生
IRK Identity Resolving Key 128 bit 解析可解析私有地址（RPA），识别设备身份 Phase 3 分发
CSRK Connection Signature Resolving Key 128 bit 数据签名验证（无加密时） Phase 3 分发
DHKey Diffie-Hellman Key 256 bit ECDH 共享密钥，用于派生 LTK（仅 LESC） LESC Phase 2 公钥交换后本地计算，不在空中传输

✅总结与建议

从第一性原理视角的演进脉络

BLE 配对机制的演进，本质上是围绕两个根本问题的逐步完善：

BT 4.0 Legacy · 2010
问题一（身份认证）：用户辅助 PIN/OOB
问题二（密钥协商）：对称 TK→STK，安全性与 TK 熵值强相关。Just Works TK=0 完全不安全；Passkey 20位熵可被暴力破解；OOB 最强但需硬件。整体被动窃听保护极弱。
BT 4.2 LESC · 2014
密钥协商革命：ECDH 替代对称 TK
引入 P-256 ECDH，将"密钥协商"问题从"如何安全传递 TK"升级为"如何利用公钥密码学消除 TK 传递需求"。即使攻击者获取全部空中数据，也无法推算 LTK。身份认证方面新增 Numeric Comparison，将公钥指纹（6位数字）绑定到用户交互验证。
设计哲学对比 TLS
无 PKI 体系下的信任建立
TLS 依赖 CA 证书链实现无需人类介入的身份认证；BLE 因无法建立全局 PKI，采用"首次使用信任（TOFU）"模型——将认证的负担转移给用户（PIN 确认、数字比较）或特殊物理信道（OOB/NFC）。这是有意为之的设计权衡，而非技术缺陷。

实践建议

🏆
最佳选择：LESC Numeric Comparison
双方有显示屏和确认按钮时首选。ECDH+用户验证，同时防止被动窃听和 MITM。Mode 1, Level 4。
🥈
次优：LESC OOB（如 NFC）
有 NFC 硬件时使用。物理距离限制+ECDH，安全性最高，但成本较高。
⚠️
受限场景：LESC Just Works
无 I/O 能力设备使用。防被动窃听，但不防 MITM。至少比 Legacy Just Works 强很多。
❌
避免：Legacy Just Works
TK=0，无任何安全性，攻击者可即时解密。除非设备真的无其他选择，否则不应使用。
✅ 核心结论

BLE 配对安全演进的根本驱动力：Legacy Pairing 将密钥协商安全性寄托于"带外传递"（TK），在空中可见的信息足以离线破解；LESC 用 ECDH 从根本上解决了这一问题——即使攻击者看到全部空中数据，也无法在计算上可行的时间内推算出 LTK。身份认证方面，Numeric Comparison 通过将 ECDH 公钥绑定到可人工验证的 6 位数字，优雅地解决了"无 CA 体系下的 MITM 防护"难题，是 BLE 安全设计中最精妙的创新之一。
