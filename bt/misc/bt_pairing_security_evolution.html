FIRST PRINCIPLES

经典蓝牙配对安全演进

从第一性原理出发，任何安全通信协议的配对阶段无非解决两个核心问题：身份认证（你是谁？）与密钥协商（我们用什么密钥加密？）。以下内容基于 NIST SP 800-121r2、蓝牙核心规范及学术论文，系统梳理经典蓝牙（BR/EDR）三代配对机制的演进历程。

第一性原理：安全通信的两大核心问题

配对的终极目标：双方完成身份认证，并安全地建立 128-bit 链路密钥，供后续数据加密使用

问题一：身份认证
Authentication
证明对端是合法设备，防止仿冒与中间人攻击。类比 TLS 中的证书验证。
问题二：密钥协商
Key Exchange
在不安全信道上安全地建立共享密钥，防止窃听者获取密钥材料。类比 TLS 中的 ECDHE。
结果
Link Key
128-bit 链路密钥，是后续 E0/AES-CCM 加密会话的根密钥。存储后即为 Bonding。
EVOLUTION TIMELINE

经典蓝牙配对安全演进总览

从蓝牙 1.0（1999）到蓝牙 4.1（2013），经典蓝牙配对机制历经三次重大变革，每次均在前一代基础上针对已知安全缺陷进行改进。

1999 — 蓝牙 1.0/1.1/1.2/2.0

阶段一：Legacy PIN 配对（传统 PIN 配对）

基于 SAFER+ 算法族（E21/E22/E1），用户输入或设备内置 PIN 码作为共享秘密，派生初始化密钥再交换组合密钥。无公钥密码学，无法抵御被动窃听，短 PIN 可被暴力破解。

2007 — 蓝牙 2.1 + EDR

阶段二：简单安全配对 SSP（Secure Simple Pairing）

引入椭圆曲线 Diffie-Hellman（ECDH，P-192 曲线），公钥密码学取代 PIN 作为密钥协商机制。提供四种关联模型：Numeric Comparison、Just Works、Passkey Entry、Out-of-Band，对抗被动窃听。引入 Security Mode 4。

2013 — 蓝牙 4.1

阶段三：安全连接 Secure Connections（SC）

升级 ECDH 至 P-256 曲线，设备认证算法升级为 HMAC-SHA-256 + AES-CTR，加密升级为 AES-CCM（含完整性保护），全面采用 FIPS 认可算法。新增 Secure Connections Only Mode（FIPS 模式）。

PHASE 1 · 1999–2004

阶段一：Legacy PIN 配对（传统 PIN 配对）

适用于蓝牙规范 v1.0B 至 v2.0 + EDR（部分设备延用至 v2.1 的 Security Mode 1/2/3）。NIST SP 800-121r2 称之为 "Personal Identification Number (PIN) Pairing"，亦称 "Legacy Pairing" 或 "Classic Pairing"。

第一性原理对应： 身份认证通过「双方均知道同一个 PIN」来隐式完成（对称秘密 = 身份证明）；密钥协商通过 SAFER+ 算法族从 PIN 派生，无公钥密码学参与。

协议流程与密钥派生

Legacy PIN 配对分为四个主要步骤，整个过程无需公钥密码学，所有密钥材料均从用户输入或内置的 PIN 码派生。

算法细节：E21 / E22 / E1

以下内容来源于 NIST SP 800-121r1/r2 及学术文献：

  • E22（初始化密钥生成）： 输入 PIN（最长 16 字节，不足时用发起方 BD_ADDR 补齐）、IN_RAND（128-bit 随机数）和 PIN 长度 L，输出 128-bit K_INIT。该密钥仅用于保护后续密钥交换，应在初始化后丢弃。
  • E21（组合密钥分量生成）： 输入本地随机数 LK_RAND 和蓝牙设备地址 BD_ADDR，输出 128-bit 密钥分量。两个设备分别计算 K1=E21(LK_RAND_A, BD_ADDR_A) 和 K2=E21(LK_RAND_B, BD_ADDR_B)，最终链路密钥 = K1 ⊕ K2。
  • E1（认证函数）： 基于 SAFER+ 算法，输入 BD_ADDR、Link Key 和挑战随机数 AU_RAND，输出 128-bit。高 32 位用于认证响应 SRES，低 96 位为 ACO（Authenticated Ciphering Offset，用于加密密钥派生）。

安全分析与已知漏洞

优点

  • 实现简单，计算量小，适合低功耗早期设备
  • 通过相互认证（E1 Challenge-Response）可验证双方持有相同密钥
  • 支持长 PIN（最长 16 字节 = 128-bit），理论上足够安全
  • 加密密钥由 Link Key + 随机数派生，会话密钥刷新有基本隔离

缺点（关键缺陷）

  • 被动窃听：无公钥密码学，攻击者监听整个配对流程后可用 PIN 计算出 Link Key
  • 短 PIN 爆破：耳机等设备内置 "0000"/"1234"，6 位 PIN 仅有 10⁶ 种可能，可暴力破解
  • 无 MITM 防护：协议设计中无法真正防止中间人攻击（CVE-2020-26555）
  • 无消息完整性：仅 CRC，不提供密码学完整性保证
  • 密钥流重复：E0 密钥流约 23.3 小时后重复，长连接存在解密风险
  • 单元密钥问题：v1.2 之前使用的静态单元密钥可被重放用于冒充其他设备

攻击场景：被动窃听 PIN 配对

CVE-2020-26555（NIST/CERT 披露）： 蓝牙核心规范 v1.0B 至 v5.2 中，BR/EDR PIN 配对协议存在仿冒漏洞。攻击者可欺骗目标设备的蓝牙地址（BD_ADDR），反射加密后的随机数，在不知道 PIN 的情况下完成配对。要求攻击设备在无线范围内且目标设备处于可连接、可绑定状态。（来源：CERT/CC VU#799380）
PHASE 2 · 2007

阶段二：简单安全配对 SSP（Secure Simple Pairing）

蓝牙 2.1 + EDR 引入 SSP，对应 Security Mode 4。这是经典蓝牙安全史上最重大的变革：引入了椭圆曲线 Diffie-Hellman（ECDH，P-192 曲线），将密钥协商从对称秘密（PIN）升级为公钥密码学。

第一性原理对应： 密钥协商通过 ECDH 实现，解决了被动窃听问题——即使攻击者抓取全部报文，没有私钥也无法还原共享密钥；身份认证通过四种「关联模型」的用户交互实现，提供不同级别的 MITM 防护。

四种关联模型（Association Models）

SSP 根据设备 IO 能力（Display/Keyboard/NoInput/NoOutput）自动协商使用哪种关联模型，每种模型提供不同级别的身份验证强度。以下来源于 NIST SP 800-121r2 §3.1.1.2 和蓝牙核心规范。

Numeric Comparison 双方均有显示屏和是/否输入 双方屏幕各显示 6 位数字，用户确认是否相同 ✓ ECDH ✓ 有 6 位数字不参与密钥生成，仅用于用户确认。安全性最强。
Just Works 至少一方无显示或无输入 无任何用户交互 ✓ ECDH ✗ 无 等同于 Numeric Comparison 但用固定值 "000000" 跳过确认。对蓝牙耳机等设备。
Passkey Entry 一方有键盘，另一方有显示 一方显示 6 位数字，用户在另一方键盘输入 ✓ ECDH △ 理论有 6 位 Passkey 参与密钥承诺计算。存在理论 MITM 风险（CVE-2020-26558）。

SSP ECDH 密钥交换完整流程（以 Numeric Comparison 为例）

f1, f2, g, f3 函数说明（蓝牙核心规范定义）：
f1（承诺函数）： 基于 HMAC-SHA-256，接受两个公钥、随机数和判别符，输出承诺值，防止随机数被篡改。 g（数字验证函数）： 基于 SHA-256，从两个公钥和两个随机数计算出 6 位数字，用于 Numeric Comparison 用户确认。该数字本身不参与密钥生成。 f2（密钥派生函数）： 基于 HMAC-SHA-256，从 DHKey（ECDH 共享密钥）、两个随机数、标签和设备地址派生出 MacKey 和 Link Key。 f3（DHKey 检验函数）： 基于 HMAC-SHA-256，用 MacKey 对所有配对参数进行认证，双方发送并验证对方的 Check 值，确认密钥材料一致。

安全分析

SSP 的改进

  • ECDH（P-192）提供被动窃听防护：攻击者抓包后无法计算 DHKey
  • Numeric Comparison 提供完整 MITM 防护（用户确认数字）
  • HMAC-SHA-256 用于密钥派生，取代弱 SAFER+ 派生
  • 用户体验改善：不再需要手动输入 PIN，随机数自动生成
  • Security Mode 4 要求所有服务声明安全需求
  • Numeric Comparison 中 6 位数字不参与密钥生成，窃听者无法利用

SSP 的局限

  • Just Works 无 MITM 防护：耳机等无界面设备强制 Just Works，无法防主动攻击
  • Passkey Entry 漏洞（CVE-2020-26558）：攻击者可通过构造响应逐位猜测 Passkey
  • 认证和加密未升级：E1（SAFER+）认证、E0 加密沿用 Legacy，非 FIPS 算法
  • 无消息完整性：仍仅有 CRC
  • 降级攻击：与不支持 SSP 的老设备配对时降回 Legacy PIN 模式
  • P-192 安全边界：随量子计算发展，192-bit 椭圆曲线安全性受质疑
PHASE 3 · 2013

阶段三：安全连接 Secure Connections（SC）

蓝牙 4.1 正式引入 BR/EDR Secure Connections，对 SSP 的密码学组件进行全面升级，实现 FIPS 完全合规。在 Security Mode 4 的框架下新增 Level 4（安全连接必须使用 FIPS 算法）。

第一性原理对应： 密钥协商升级至 P-256 椭圆曲线，提供更高安全边界；身份认证和加密算法全面替换为 FIPS 批准的现代密码原语（HMAC-SHA-256、AES-CCM），同时增加消息完整性保护（MIC）。

P-256 升级与 FIPS 算法全景

AES-CCM 认证加密的意义

AES-CCM（Counter with CBC-MAC）是一种 AEAD（Authenticated Encryption with Associated Data）算法，同时提供：

  • 机密性（Confidentiality）： AES-CTR 模式加密，128-bit 密钥，远强于 E0 流加密
  • 完整性（Integrity）： CBC-MAC 生成消息完整性校验码（MIC），防止攻击者篡改密文
  • 认证性（Authenticity）： MIC 绑定了密钥，只有持有正确密钥的一方才能生成有效 MIC，提供数据来源认证
KNOB 攻击（CVE-2019-9506）：即使在 Secure Connections 模式下，攻击者可通过 MITM 将加密密钥长度强制降至 1 字节（7-bit 暴力破解），从而解密通信。蓝牙 SIG 建议厂商强制最小密钥长度为 7 字节（56-bit）。此攻击源于加密密钥长度协商未受认证保护。

安全分析

SC 的改进

  • P-256 椭圆曲线，128-bit 安全强度（业界标准）
  • HMAC-SHA-256 认证：密码学上可证明安全，替换弱 SAFER+
  • AES-CCM 认证加密：同时保证机密性 + 完整性，解决 Legacy/SSP 的完整性缺失
  • Secure Connections Only Mode：强制设备拒绝降级到非 FIPS 算法
  • FIPS 全面合规：符合政府/企业级安全要求
  • 前向保密性（Forward Secrecy）：ECDH 密钥对每次配对重新生成

SC 的局限

  • 向后兼容降级：与旧设备互连时可能退回 SSP 或 Legacy
  • KNOB 攻击：加密密钥长度协商未受保护，可被强制降级（CVE-2019-9506）
  • BLUFFS 攻击（CVE-2023-24023）：会话密钥派生机制存在缺陷，攻击者可影响会话密钥熵
  • Just Works 仍无 MITM 防护：无显示设备仍然脆弱
  • Passkey Entry 仍存在理论漏洞：CVE-2020-26558 影响范围包括 SC
  • 重放认证：重连时的认证仍基于 Challenge-Response，非 PFS 会话

核心设计差异：为何蓝牙不用证书？

蓝牙的核心设计约束是无公共基础设施：两台全新设备在无任何预共享秘密、无证书机构的前提下，需要在物理接近的情况下建立信任。这与 TLS 的互联网模型根本不同——TLS 依赖预先存在的 PKI，而蓝牙必须自举信任（Bootstrap Trust），因此"用户眼睛"成为了身份验证的最后防线。
