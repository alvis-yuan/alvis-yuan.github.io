§ 1

简介与目标（Introduction）

RFCOMM 是蓝牙的串行端口仿真协议，全称 Radio Frequency Communication。它基于 ETSI 标准 GSM TS 07.10，运行于 L2CAP 之上，目的是让原本依赖 RS-232（EIA/TIA-232-E）串口的应用程序，无需修改即可通过蓝牙无线传输数据。

协议目标

  • 仿真 RS-232 的 9 个电路信号
  • 让传统串口应用无缝迁移至蓝牙
  • 支持两设备间最多 60 条并发串口连接
  • 提供可靠、有序的数据传输

适用场景

  • PC ↔ 手机数据同步（SPP 串口 Profile）
  • 蓝牙耳机拨号（HSP / HFP）
  • 打印机、PDA 无线连接
  • PPP / 拨号上网（DUN Profile）

§ 2

RFCOMM 参考模型（Reference Model）

规范定义了 RFCOMM 参考模型，描述了两台设备间"虚拟串口通信段"的完整架构。通信路径包含两端的应用程序（通信端点）和它们之间的蓝牙无线链路（通信段）。RFCOMM 只关注设备间直连情形，或设备与调制解调器之间的连接。
RFCOMM 参考模型（含多虚拟串口示意）

通信端点（Endpoints）

位于两个不同设备上的应用程序，通过虚拟 COM 口与 RFCOMM 交互，完全感知不到底层无线传输细节。

通信段（Communication Segment）

两设备之间的蓝牙链路。RFCOMM 仅关注此直连段，不处理端到端路由或中继场景。

§ 4

支持的帧类型（Frame Types）

RFCOMM 使用 TS 07.10 基本选项（Basic Option），支持 5 种帧类型。注意：TS 07.10 中的 UI（Unnumbered Information）帧在 RFCOMM 中不支持。所有帧均通过 L2CAP 传输，无需 RFCOMM 层重传。
帧类型 全称 控制字段值 方向 用途 需要响应
SABM Set Asynchronous Balanced Mode 0x2F 命令 建立连接（启动复用器 / DLC） ✓ UA 或 DM
UA Unnumbered Acknowledgement 0x63 响应 确认 SABM / DISC，表示接受 ✗
DM Disconnected Mode 0x0F 响应 拒绝连接，表示处于断连状态 ✗
DISC Disconnect 0x43 命令 断开 DLC 或关闭复用器 ✓ UA
UIH Unnumbered Information with Header check 0xEF 命令/响应 传输数据（DLCI≠0）或多路控制命令（DLCI=0） ✗（异步）

§ 5

帧结构（Frame Structure）

RFCOMM 采用 TS 07.10 基本选项帧格式（去掉开头/结尾 Flag 字节）。每个 L2CAP 包只含一个 RFCOMM 帧。帧由 Address、Control、Length、Data（仅 UIH）、FCS 五个字段组成。
RFCOMM 帧字段布局（UIH 数据帧）

Address 字段为 1 字节，包含 4 个子字段，决定帧的寻址与命令/响应性质：

Control 字段标识帧类型，P/F（Poll/Final）位用于命令/响应握手：

帧类型Control 值（P/F=1）Control 值（P/F=0）
SABM0x3F0x2F
UA0x730x63
DM0x1F0x0F
DISC0x530x43
UIH0xFF（带 Credit）0xEF

P/F=1 的 UIH 帧用于信用（Credit）传递，是流量控制的核心机制。

Length 字段使用 EA 扩展机制，允许表示更大的数据负载：

// 单字节 Length（EA=1，最大 127 字节） [L7 L6 L5 L4 L3 L2 L1 EA=1] // EA=1 表示只有这一字节 // 双字节 Length（EA=0，最大 32767 字节） [L7 L6 L5 L4 L3 L2 L1 EA=0] // 低字节，EA=0 表示还有下一字节 [L14 L13 L12 L11 L10 L9 L8 0] // 高字节 // 默认 MTU = 127 字节，最大 = 32767 字节

FCS（Frame Check Sequence）使用 CRC-8 算法，多项式为 x⁸+x²+x+1（0x07）：

SABM / DISC / UA / DM

FCS 计算覆盖：Address + Control + Length 三个字段

UIH 帧

FCS 仅计算：Address + Control 两个字段（不含 Data，性能优化）

§ 6

多路复用控制命令（Multiplexer Commands）

控制命令通过 DLCI=0 的 UIH 帧传输，用于管理 RFCOMM 连接。规范定义了 7 类命令/响应，每条命令都有对应的响应格式。若收到不支持的命令，必须回复 NSC（Non-Supported Command）。

PN — 参数协商

Parameter Negotiation。建立 DLC 前协商帧大小、优先级、信用数等参数。
必须

MSC — 调制解调器状态

Modem Status Command。传递 RS-232 控制信号（FC、RTC、RTR、IC、DV、Break）。
必须

RPN — 远程端口协商

Remote Port Negotiation。协商虚拟串口参数（波特率、数据位、停止位、校验位、流控方式）。
可选

RLS — 远程线路状态

Remote Line Status。通知对端线路错误（过载、奇偶校验错误、成帧错误）。
可选（但必须能响应）

Test — 测试命令

发送测试数据并要求对端原样回应，用于测试链路连通性与延迟。
可选

FCon / FCoff — 流控

Flow Control On/Off。聚合流控命令，暂停/恢复所有 DLC 的数据流（已被信用机制补充）。
已弱化

NSC — 不支持命令响应

Non-Supported Command。收到任何不识别的命令类型时，必须以 NSC 响应，不得忽略。
必须

DLCI6待协商的数据链路连接标识符标识目标 DLC
I1~I44帧类型标识RFCOMM 中必须为 0（使用 UIH）

§ 7

连接建立与断开（Connection Establishment）

RFCOMM 连接分为两个阶段：① 先建立 L2CAP 信道（PSM=3），再建立 RFCOMM 会话（复用器，DLCI=0）；② 在已建立的会话上建立 DLC（具体虚拟串口，DLCI≠0）。所有这些步骤都需要精确的帧交互。

§ 8

流量控制机制（Flow Control）

RFCOMM v1.1 强制要求实现信用（Credit）流控机制，这是相较于原始 TS 07.10 的核心扩展。信用机制在单个 DLC 级别精细控制数据流，替代了仅能全局暂停的 FCon/FCoff 方式。
信用（Credit）流控工作原理

FCon / FCoff（全局流控）

老式聚合流控，通过 DLCI=0 的控制命令对所有信道同时生效，粒度粗，现已被信用机制取代。

遗留机制

Credit 信用流控（强制）

每个 DLC 独立维护收发信用计数。发送方每发一帧消耗 1 信用，信用耗尽则停止发送，直到对端通过 UIH(P/F=1) 补充信用。

强制实现
§ 9

DLCI 编址与服务器信道（DLCI & Server Channel）

DLCI（Data Link Connection Identifier，数据链路连接标识符）是一个 6 位字段，由方向位 D 和 5 位服务器信道号（Server Channel，1~30）组合构成。DLCI 在两设备之间的 RFCOMM 会话中唯一标识一条虚拟串口连接。
DLCI 值域分配图

DLCI 计算规则

// 发起方（Initiator, D=1） DLCI = ServerChannel × 2 + 1 // 示例：Server Channel = 9 DLCI = 9×2 + 1 = 19 (0x13) // 响应方（Responder, D=0） DLCI = ServerChannel × 2 + 0 // 示例：Server Channel = 9 DLCI = 9×2 + 0 = 18 (0x12)

服务器信道（Server Channel）

  • 范围：1 ~ 30（共 30 个）
  • 服务器信道号需在 SDP 数据库注册
  • 客户端通过 SDP 查询获取目标服务器信道号
  • 多个 Profile 可在同一会话中并行使用不同信道

§ 10

RS-232 串口信号仿真（Port Emulation）

RFCOMM 通过 MSC（Modem Status Command）命令传递 RS-232 的 9 个控制信号，并通过 RPN（Remote Port Negotiation）协商虚拟串口参数（如波特率、数据位、奇偶校验），使传统串口应用无感知地运行于蓝牙之上。

§ 11

错误处理与链路丢失（Error Handling）

RFCOMM 依赖 L2CAP 提供最大可靠性的信道（有序无重复）。RFCOMM 本身不重传，但定义了在链路丢失、FCS 错误、非预期帧等情况下的处理规则，以及各状态下收到 SABM/DISC/UA/DM 的响应行为。
RFCOMM 错误场景处理决策图
当前状态收到 SABM收到 DISC收到 UA收到 DM
已断开（Disconnected） 回 UA / DM 回 UA 忽略 忽略
已连接（Connected/ABM） 重复 SABM → 回 UA 关闭 DLC，回 UA 忽略 通知上层断开
等待 UA（W4_UA） 忽略 忽略 连接成功 连接被拒
§ 12

服务发现（Service Discovery via SDP）

RFCOMM 服务必须在蓝牙 SDP（Service Discovery Protocol）数据库中注册，客户端在建立 RFCOMM 连接之前，先通过 SDP 查询获取目标服务的 Server Channel 号。这是 RFCOMM 连接建立的第一步（零步骤）。

设备类型说明

规范定义两种设备类型影响信号映射方向：

  • Type 1 设备（DTE）：设备本身是通信终端（如 PC、打印机），虚拟 COM 口直接对接 RFCOMM
  • Type 2 设备（含端口代理）：设备有专用端口驱动层，在非 RFCOMM 侧执行流控，RFCOMM 层透传
Null Modem 配置（两 DTE 直连 = RFCOMM 典型场景）

协议核心要点汇总

帧类型（5种）

  • SABM 建立连接
  • UA 确认接受
  • DM 拒绝/断开态
  • DISC 断开请求
  • UIH 数据传输

DLCI 编址

  • 6位字段，值域 0~63
  • DLCI=0：控制信道
  • DLCI=1,62,63：保留
  • 可用：2~61（共60个）
  • 公式：DLCI=SC×2+D

流量控制

  • 强制：Credit 信用流控
  • PN CMD CL=0xF 开启
  • UIH P/F=1 传递信用
  • 遗留：FCon/FCoff
  • 每 DLC 独立控制

控制命令（7类）

  • PN：参数协商（必须）
  • MSC：调制解调器状态（必须）
  • RPN：远程端口协商
  • RLS：远程线路状态
  • Test：链路测试
  • FCon/FCoff：流控
  • NSC：不支持命令响应

关键参数

  • L2CAP PSM = 0x0003
  • 默认帧长 = 127 bytes
  • 最大帧长 = 32767 bytes
  • 最大 DLC 数 = 60
  • Server Channel 范围：1~30

连接建立顺序

  1. SDP 查询 Server Channel
  2. 建立 L2CAP 信道
  3. SABM(0) → 会话
  4. UIH+PN → 参数协商
  5. SABM(N) → DLC
  6. MSC → 信号交换
  7. UIH 数据传输
