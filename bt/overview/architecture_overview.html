Part A

架构概述 Architecture

§1总体描述 General Description

蓝牙无线技术是一种用于替代有线连接的短距离通信系统，核心特性包括：抗干扰性强、低功耗、低成本。规范中大量特性是可选的，以允许产品差异化。

蓝牙系统存在两种形态：BR/EDR（基本速率/增强数据率）和 LE（低功耗），两者均包含设备发现、连接建立机制，但底层实现互不兼容。

1.1 BR/EDR 概述

BR/EDR 物理层工作在 2.4 GHz ISM 频段，使用 FHSS 跳频收发器对抗干扰与衰落。基本速率使用 1 Msym/s GFSK 调制，EDR 使用 π/4-DQPSK（2 Mb/s）或 8-DPSK（3 Mb/s）。

一组设备同步到同一个中心设备（Central）的时钟和跳频序列后，构成一个 Piconet（微微网）。Central 控制时隙调度，Peripheral 与其通信。数据以 TDD（时分双工）方式在时隙中传输。

1.2 LE（低功耗蓝牙）概述

LE 同样工作在 2.4 GHz ISM 频段，使用 FDMA + TDMA 双重多址接入方式。40 个物理信道间隔 2 MHz；用于距离估算时，还使用 72 个间隔 1 MHz 的信道。

LE 设备角色：广播者（Advertiser）在广播信道发送广播包；扫描者（Scanner）监听不打算连接的广播；发起者（Initiator）发现并建立连接。连接建立后，Initiator 成为 Central，Advertiser 成为 Peripheral。

§2主机与控制器 Host & Controller

蓝牙核心系统由一个 Host（主机）和一个或多个 Controller（控制器）组成，二者通过 HCI（主机控制器接口）交互。

Host / Controller 分层与三种配置
ℹ️
一个蓝牙核心实现 只能有一个控制器，该控制器可以是纯 LE、纯 BR/EDR 或二者的组合。Host 侧实现 HCI 以上的所有层（L2CAP、SMP、GATT、GAP 等）。

§3信道与链路层次体系 Channel & Link Hierarchy

从物理信道向上，蓝牙定义了五层抽象：物理信道 → 物理链路 → 逻辑传输 → 逻辑链路 → L2CAP 信道。每一层都在下一层之上提供更高级别的服务。

五层信道/链路层次图

3.1 物理信道 (Physical Channels)

物理信道是 RF 层的基本单元。BR/EDR 使用 79 个 1 MHz 间隔的信道，LE 使用 40 个 2 MHz 间隔的信道（其中 3 个为广播信道）。设备使用相同的跳频序列以实现同步。

3.2 物理链路 (Physical Links)

物理链路建立在 Central 与 Peripheral 之间。一个 Central 可同时维护多个 Peripheral 的物理链路，但 Peripheral 之间不直接建立物理链路。特殊情况：广播物理链路是单向的。

3.3 逻辑传输 (Logical Transports)

逻辑传输在物理链路上复用，支持不同 QoS 属性（可靠/同步/等时）。主要类型如下：

类型方向可靠性适用
ACL (LE ACL)双向可靠重传通用数据、控制信令
SCO / eSCO双向同步，有限重传BR/EDR 语音
CIS双向等时，无重传LE Audio 耳机
BIS单向广播等时，无重传LE Audio 广播
ADVB单向广播不可靠设备发现、广播

3.4 逻辑链路 (Logical Links)

逻辑链路映射在逻辑传输之上，定义数据流的 QoS 特性。不同逻辑链路可以共享同一逻辑传输，通过调度函数管理资源。控制协议（LMP/LL）也作为特殊逻辑链路承载。

3.5 L2CAP 信道

L2CAP（逻辑链路控制和适配协议）在逻辑链路上提供信道抽象，负责协议复用、分包重组，以及多个高层协议（ATT、SMP、RFCOMM 等）的数据复用。

§4网络拓扑 Communication Topologies

蓝牙支持多种网络拓扑，从点对点到散射网（Scatternet），设备可同时扮演多种角色。

§5等时传输 Isochronous Channels (BT 5.2+)

等时信道专为时延敏感的音频流设计，保证固定间隔（ISO_Interval）的数据传输，分为连接等时（CIS）和广播等时（BIS）两种。

5.1 CIS — 连接等时流

CIS 建立在 ACL 连接之上，提供 Central ↔ Peripheral 之间的双向等时数据流。每个 CIS 事件包含若干子事件（subevent），Central 先发，Peripheral 后应答。通道由信道选择算法决定。

5.2 BIS — 广播等时流 & BIG

BIS 无需连接即可广播等时数据。一个设备可发送多个同步 BIS，称为 BIG（广播等时组）。发送端还会在 BIG 结尾插入 Control subevent 广播控制信息。接收端通过同步到周期性广播的 BIG 同步信息后，才能接收 BIS 流。

§7无线共存 Wireless Coexistence (MWS)

蓝牙 BR/EDR 设备可能与 LTE/Wi-Fi 等同频段技术共存于同一设备（例如手机主板）。规范定义了 MWS（移动无线系统）共存信令机制，使蓝牙控制器能感知外部帧结构，减少互扰。

机制说明
FRAME_SYNC 信号外部系统（LTE）的帧同步信号，蓝牙对齐自己的时隙
时钟粗调 (CCA)使用 LMP_CLK_ADJ PDU 移动蓝牙 CLK，用于 MWS 接入时对齐
时钟拖拽 (Clock Dragging)缓慢调整时钟相位（不超过自然漂移速率），持续微调
SAM（时隙可用性掩码）两设备互告可用时隙，优化调度以减少与外部无线技术的冲突
💡
建议将与 MWS 共存的蓝牙设备设为 Central，因为 Central 可更快地响应时序偏差并发起纠正措施。

§8方向探测 Direction Finding (AoA / AoD)

BLE 5.1 引入方向探测功能，允许设备估算信号的到达角（AoA）或出发角（AoD），公式为：

φ = (2π × d × cos θ) ÷ λ   ⟹   θ = arccos( φλ ÷ 2πd )

8.1 到达角（AoA）

发射端使用单天线发送方向探测包（CTE 常音扩展）。接收端拥有天线阵列，在接收时切换天线，采集 I/Q 样本，通过不同天线元素接收到的相位差估算信号到达角。适用于实时定位系统（RTLS）。

8.2 出发角（AoD）

发射端拥有天线阵列，在发送时切换天线。接收端使用单天线接收，采集 I/Q 样本，利用发射端天线阵列的布局信息计算出发角。适用于基础设施广播的室内定位。

§9信道测音 Channel Sounding (BT 6.0)

Channel Sounding（CS）是蓝牙 6.0 引入的核心新特性，通过在多个 RF 信道上双向交换信号，对两设备间的传播路径进行特性化测量，进而估算距离。

🔑
CS 同时支持 RTT（往返时间）和 PBR（相位测距）两种方法，二者可联合使用以提升精度和安全性。

9.1 CS 层次结构

CS 的时频结构分为四个层次：CS 过程 → CS 事件 → CS 子事件 → CS 步骤。每个步骤（Step）从频率切换时间段 T_FCS 开始，是 Initiator 与 Reflector 之间一次双向交换。

CS 层次关系图
步骤模式用途
Mode-0频率和时序校准（每个子事件开头）
Mode-1交换 RTT 包（CS_SYNC），测量到达/出发时间
Mode-2交换 CS 音调，测量相位和幅度（PBR）
Mode-3同时进行 RTT 测量和 PBR 音调测量

9.2 基于相位的测距（PBR）

在多个频率 f 上测量相位，则距离 x 与相位随频率的变化率成正比：

φCH(f) = −4πxf/c   ⟹   x = −(dφ/df) × c / (4π)

当频点间隔为 1 MHz 时，相位模糊距离 dwrap = 150 m，超出此距离需结合 RTT 消歧。

9.3 基于往返时间的测距（RTT）

两设备交换 CS_SYNC 包，记录各自的出发时间（ToD）和到达时间（ToA），估算飞行时间（ToF）：

x = (Tinitiator − Treflector) / 2 × c

支持三种精度等级：32 位伪噪声接入地址（基础）、96 位探测序列（中等）、128 位随机序列（最高）。

9.4 安全特性

CS 内置了防距离欺骗（Distance Spoofing）机制：RTT 与 PBR 在同一子事件中提供两路独立距离估算；随机序列可检测中间人攻击；DRBG 密码随机化跳频序列、步骤模式、音调调制和 CS_SYNC 内容。密钥材料在加密链路下交换，不对外暴露。

通用术语与解释 General Terminology & Interpretation

§1语言约定 Language Conventions

规范使用一套严格定义的关键词来表达不同程度的需求强制性，开发者必须严格区分：

shall
必须 — 规范要求，必须原样实现，不得偏差（强制）
shall not
禁止 — 规范明令禁止的行为
should
建议 — 推荐但不强制，不满足也不算违规
should not
不建议 — 不推荐但未明令禁止
may
可以 — 在规范范围内允许，实现可自行选择
must
必然 — 陈述无条件事实，或满足某条件后的必然结果
can
能够 — 表达一种可能性或能力，不具有规范约束力
⚠️
混淆 shall 与 should、may 与 must 是规范理解中最常见的错误。shall 是实现的硬性要求，违反将导致不合规。

1.7 实现替代方案

规范中示例所展示的某种实现方式，并不限制其他符合规范的替代实现。只要满足 shall 要求，实现者可自由选择技术路线。

1.9 适当语言（Appropriate Language）

规范已将某些被认定为不适当的术语替换为中性用语（如 master→Central，slave→Peripheral），并相应调整了 HCI 命令/事件名称。完整映射表见 Bluetooth SIG 官网。

§2通用解释规则 General Interpretation Rules

2.1 二进制与十六进制

二进制前缀 0b（如 0b1101 = 13），十六进制前缀 0x（如 0x42 = 66）。可用下划线分隔数字以增强可读性（如 0b0010_1011），不影响数值。未带前缀的数为十进制。

2.2 位编号与位域

位始终从 0 开始编号，0 为最低有效位（LSB）。位域写法：下标连字符（CLK4-2）或方括号加冒号（CLK[4:2]），端点均包含，最高位写在前。

2.4 保留字段处理规则

标记发送方行为接收方行为
RFU（Reserved for Future Use）必须置 0必须忽略（不得因此拒绝结构）
Previously used按 RFU 处理按 RFU 处理（不用于新特性）
Reserved for spec dev不得在产品中使用按 RFU 处理
ℹ️
RFU 字段用于 CRC/加密等计算时，必须使用实际收到的原始值，而非将其替换为 0。

2.7 响应无效行为

当设备收到规范不允许的包或 PDU（状态错误、内容非法）时，可采取以下之一应对：

  • 忽略该包或 PDU
  • 恢复：发送拒绝响应 / 忽略超长内容 / 使用最近合法值 / 使用默认值
  • 终止连接：立即或在合适层面断开（如安全过程中检测到攻击时）
💡
选择哪种响应方式影响用户体验。例如，Peripheral 发出的 POLL 包可能是 NULL 包的 TYPE 字段被破坏，忽略比断连更合理；但安全过程中的无效行为则应立即断连。

控制器错误码 Controller Error Codes

§1错误码概述 Overview

错误码用于 HCI 命令失败或 LMP/LL 消息传递失败时，指示失败原因。错误码大小为 1 字节（0x00–0xFF）。

  • 0x00 = Success（成功）
  • 0x01–0xFF = 各类失败原因
  • Host 若收到未识别的错误码，应将其等同于 0x1F Unspecified Error
ℹ️
错误码在 HCI 命令返回、LMP PDU 和 LL Control PDU 中均可能出现。部分错误码在特定协议层有特定语义，不可随意混用。
