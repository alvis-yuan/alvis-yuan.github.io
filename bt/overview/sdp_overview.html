§1引言 (Introduction)

本部分定义了一种协议，用于发现蓝牙设备所提供或可通过蓝牙设备访问的服务。SDP（Service Discovery Protocol，服务发现协议）是蓝牙协议栈的重要组成部分，解决了"如何知道对方设备支持哪些服务"的根本问题。

🔍
服务发现
发现附近蓝牙设备上可用的服务类型
📋
属性查询
获取服务的详细属性，如协议栈、访问方式
🔗
连接准备
为后续建立服务连接提供必要信息
🌐
动态拓扑
适应蓝牙设备动态进入/离开的无线环境

§1.1总体描述

SDP 为应用程序提供了一种机制，用于发现哪些服务是可用的，并确定这些可用服务的特征。服务的属性包括：服务的类型或类别，以及利用该服务所需的机制或协议信息。

⚠️ 注意： SDP 仅负责发现服务及其属性（包括相关服务访问协议），而不负责实际利用这些服务（例如不传输服务数据）。实际的服务连接需要在 SDP 查询完成后，通过独立的连接来建立。

§1.5约定 (Conventions)

位与字节顺序约定

在单字节内包含多个位字段时，更高有效位（高位）在左侧，低有效位在右侧。多字节字段按网络字节序（大端序）传输，高字节在左，低字节在右。

§2总览 (Overview)

§2.1SDP 客户端-服务器架构

SDP 采用经典的客户端-服务器模型。服务器维护一个 SDP 数据库，包含描述其所关联服务特征的服务记录列表。客户端通过发出 SDP 请求，从服务器检索服务记录中的信息。

图 2.1 · 2.2 — SDP 客户端-服务器交互模型

图 2.1/2.2：SDP 客户端通过 L2CAP 连接向 SDP 服务器发送请求，服务器从服务记录数据库中查询并返回结果

关键特性： SDP 服务器集合相对于 SDP 客户端是动态变化的——当服务器变为可用时，需由 SDP 以外的机制（如询问）通知客户端。当服务器离开范围时，SDP 不提供显式通知，但客户端可通过轮询推断服务器是否可用。

§2.2服务记录 (Service Record)

服务是指能够代表另一个实体提供信息、执行动作或控制资源的任何实体，可通过软件、硬件或软硬件组合实现。SDP 服务器中关于某服务的所有信息都包含在单个服务记录中，服务记录仅为服务属性的列表。

服务记录数据库结构

SDP 数据库包含多条服务记录，每条记录由唯一的 32 位句柄标识，Handle 0x00000000 保留给 SDP 服务器自身

服务记录句柄规则： 句柄在单个 SDP 服务器内唯一。在 L2CAP 连接期间，服务器不得重用句柄值。若某服务被移除，使用其旧句柄的请求将收到"无效服务记录句柄"错误。

§2.3服务属性 (Service Attribute)

每个服务属性描述服务的一个单独特征，由两部分组成：属性 ID（Attribute ID）和属性值（Attribute Value）。

属性名属性含义
ServiceClassIDList标识服务记录所代表的服务类型（服务所属的类列表）
ServiceID唯一标识某个服务的特定实例
ProtocolDescriptorList指定可用于利用该服务的协议栈
ProviderName提供服务的个人或组织的文字名称
IconURL指向可用于代表该服务的图标图像的 URL
ServiceName包含服务的人类可读名称的文本字符串
ServiceDescription描述服务的文本字符串

§2.3.1 属性 ID

属性 ID 是一个 16 位无符号整数，用于在服务记录中区分各个服务属性，并标识相关属性值的语义。属性 ID 仅在每个服务类内部唯一。在 SDP 中，属性 ID 以数据元素的形式表示。

§2.3.2 属性值

属性值是一个可变长度字段，其含义由与之关联的属性 ID 以及包含该属性的服务记录的服务类决定。在 SDP 中，属性值以数据元素的形式表示，通常可以是任何类型的数据元素，受服务类定义中规定的约束限制。

§2.4服务类 (Service Class)

每个服务都是某个服务类的实例。服务类定义提供了服务记录中所有属性的定义。每个服务类都被分配一个唯一标识符（UUID），包含在 ServiceClassIDList 属性中。

服务类继承层次示例（打印机服务）

子类服务类保留父类的所有属性定义，并可添加特定于子类的新属性

§2.5搜索服务 (Searching for Services)

服务搜索事务允许客户端根据服务记录属性中所包含的 UUID 值来检索特定服务记录的句柄。SDP 仅支持基于 UUID 值的属性搜索，不支持任意属性值搜索。

§2.5.1 UUID（通用唯一标识符）

UUID 是一个 128 位的值，预期在所有空间和时间上是唯一的（独立生成的 UUID 相同的概率可忽略不计）。无需中央注册机构即可分布式创建。

为减少存储和传输 128 位 UUID 值的负担，蓝牙预分配了一段 UUID 范围，已注册的常用用途使用 16 位或 32 位别名。

§2.5.2 服务搜索模式 (Service Search Patterns)

服务搜索模式是一个 UUID 列表，用于定位匹配的服务记录。当且仅当服务搜索模式中每一个 UUID 都包含在某服务记录的任意属性值中，该模式才与该服务记录匹配。

搜索匹配逻辑示意

搜索模式是 UUID 的子集匹配：服务记录必须包含搜索模式中的所有 UUID 才算匹配

§2.6浏览服务 (Browsing for Services)

浏览是指在没有任何关于服务的先验信息的情况下，发现 SDP 服务器服务记录所描述的服务类型。SDP 的浏览机制基于所有服务类共享的 BrowseGroupList 属性。

每个 UUID 代表一个浏览组，服务通过将根浏览组的 UUID（PublicBrowseRoot）放入自己的 BrowseGroupList 属性，使自身出现在顶级浏览中。SDP 服务器可定义分层浏览组结构，每个 BrowseGroupDescriptor 服务记录通过 GroupID 属性定义一个新的浏览组。

§3数据表示 (Data Representation)

SDP 使用统一的数据元素格式来表示所有的属性 ID 和属性值，这是 SDP 协议中信息编码的基础。

§3.1数据元素 (Data Element)

数据元素是 SDP 中用于表示信息的基本构建块。每个数据元素由两个主要字段组成：类型描述符（Type Descriptor）和数据（Data）。类型描述符由类型（type）和大小（size）两个子字段构成。

数据元素结构（类型描述符字节）

§3.2数据元素类型 (Data Element Types)

类型值 (高5位)数据类型说明
0b00000 (0)Nil空类型，仅有一个可能的值（大小索引必须为 0）
0b00001 (1)Unsigned Integer无符号整数（1/2/4/8/16 字节）
0b00010 (2)Signed Integer有符号二进制补码整数（1/2/4/8/16 字节）
0b00011 (3)UUID通用唯一标识符（2/4/16 字节）
0b00100 (4)Text String文本字符串（可变长度）
0b00101 (5)Boolean布尔值（1 字节，0=false，非0=true）
0b00110 (6)Data Element Sequence数据元素序列（其他数据元素的有序序列）
0b00111 (7)Data Element Alternative数据元素替代（互斥的数据元素集合）
0b01000 (8)URL统一资源定位符（可变长度）
0b01001–0b11111保留未来扩展使用

§3.3 Nil 类型

Nil 类型仅有一个可能的值（null），大小索引必须为 0，表示一个单字节的数据元素（仅类型描述符字节，无后续数据）。

§3.4 无符号/有符号整数

整数可以是 1、2、4、8 或 16 字节。大于 1 字节的整数按网络字节序（大端序）存储。有符号整数使用二进制补码表示法。

§3.5 UUID 类型

UUID 有三种长度：16 位（2字节）、32 位（4字节）、128 位（16字节）。在比较前，较短的 UUID 必须扩展到相同长度。

§3.6 文本字符串

文本字符串为字节序列，使用大小索引 5、6 或 7（后跟 1、2 或 4 字节的长度字段）。字符串编码（UTF-8 等）由服务属性的 LanguageBaseAttributeIDList 属性指定。

§3.7 布尔值

布尔值占用 1 字节，0 表示 FALSE，任何非零值表示 TRUE。

§3.8 数据元素序列 / 替代

序列（Sequence）：多个数据元素的有序列表，用于表示 ServiceClassIDList、ProtocolDescriptorList 等复合属性值。替代（Alternative）：表示互斥的选项集合，客户端选择其中一个即可。

§3.9 URL 类型

URL 类型存储统一资源定位符，编码格式遵循 RFC 2396，使用可变长度字段（大小索引 5、6、7）。

§4协议描述 (Protocol Description)

本节描述 SDP 协议的完整技术细节，包括字节序规定、PDU（协议数据单元）格式，以及各类请求/响应的具体定义。

§4.1传输字节顺序

除特别说明外，SDP 中所有多字节字段均采用网络字节序（大端序）传输，即最高有效字节（MSB）最先传输。

§4.2PDU 格式 (Protocol Data Unit)

所有 SDP 消息都封装在 PDU 中。每个 PDU 都具有相同的头部格式，后跟特定于该 PDU 类型的参数。

SDP PDU 通用头部格式
PDU ID8 位 (1字节)
TransactionID16 位 (2字节)
ParameterLength16 位 (2字节)
Parameters可变长度

PDU ID: 标识 PDU 类型  |  TransactionID: 请求/响应匹配用的事务 ID（客户端生成，服务器在响应中原样返回） |  ParameterLength: Parameters 字段的字节长度

SDP PDU ID 汇总
PDU IDPDU 名称方向作用
0x01SDP_ERROR_RSP服务器 → 客户端错误响应，包含错误码
0x02SDP_SERVICE_SEARCH_REQ客户端 → 服务器请求匹配搜索模式的服务记录句柄
0x03SDP_SERVICE_SEARCH_RSP服务器 → 客户端返回服务记录句柄列表
0x04SDP_SERVICE_ATTR_REQ客户端 → 服务器请求特定服务记录的属性值
0x05SDP_SERVICE_ATTR_RSP服务器 → 客户端返回请求的属性值
0x06SDP_SERVICE_SEARCH_ATTR_REQ客户端 → 服务器合并搜索+属性查询：搜索匹配服务并返回属性
0x07SDP_SERVICE_SEARCH_ATTR_RSP服务器 → 客户端返回所有匹配服务的指定属性

§4.3错误响应 (SDP_ERROR_RSP)

当服务器无法处理请求时，返回 SDP_ERROR_RSP。参数为 16 位错误码。

常见错误码
错误码含义
0x0001无效/不支持的 SDP 版本
0x0002无效的服务记录句柄
0x0003无效的请求语法
0x0004无效的 PDU 大小
0x0005无效的续传状态
0x0006请求因资源不足被拒绝

§4.7服务搜索属性请求 (SDP_SERVICE_SEARCH_ATTR)

此事务将"服务搜索"和"属性获取"合二为一。客户端指定搜索模式和所需属性 ID，服务器返回所有匹配服务记录的指定属性。这是实际应用中最常用的 SDP 事务类型。

🔍
ServiceSearchPattern
UUID 列表，用于筛选服务（同 SERVICE_SEARCH_REQ）
📋
AttributeIDList
请求的属性 ID 或 ID 范围（同 SERVICE_ATTR_REQ）
📦
AttributeLists
返回：所有匹配服务的属性列表的数据元素序列

§4.8续传状态 (Continuation State)

当服务器的响应数据超过单个 PDU 所能承载的大小时，使用续传机制将大响应分成多个 PDU 传输。续传状态参数允许客户端在后续请求中通知服务器从上次中断处继续。

§6安全 (Security)

在安全模式 4（Security Mode 4）中，SDP 默认不使用安全机制，但可以使用安全（经过认证或未认证的链路密钥加上加密）。具体安全策略见 [Vol 3] Part C, Section 5.2.2。

🔓
默认：无安全
SDP 通常以明文运行，不加密，因为其服务信息本身是公开的
🔐
可选：加密链路
可使用已认证或未认证的链路密钥 + 加密来保护 SDP 通信
🛡️
安全模式 4
服务级别强制安全，在 SDP 发现后的实际服务连接中执行认证

附录 BSDP 事务示例

以下是典型 SDP 事务的简单示例，用于说明 SDP 协议的实际消息流程。示例假设已建立 L2CAP 连接，不考虑缓存、服务可用性、SDP 版本和错误响应。

B.1示例 1 — ServiceSearchRequest（服务搜索）

场景：SDP 客户端搜索通用打印服务。服务器上有两个打印服务可用。

ServiceSearch 消息流
/* 客户端 → 服务器: 搜索打印服务 */
SDP_SERVICE_SEARCH_REQ[15] {
  PDUID[1]         { 0x02 }              /* ServiceSearch 请求 */
  TransactionID[2] { 0xtttt }
  ParameterLength[2] { 0x000A }
  ServiceSearchPattern[7] {
    DataElementSequence[7] {
      0b00110 0b101 0x05  /* 序列，5字节 */
      UUID[5] {
        0b00011 0b010      /* 32位UUID */
        0xpppppppp         /* PrinterServiceClassID */
      }
    }
  }
  MaximumServiceRecordCount[2] { 0x0003 }
  ContinuationState[1] { 0x00 }         /* 无续传 */
}

/* 服务器 → 客户端: 返回两个打印服务句柄 */
SDP_SERVICE_SEARCH_RSP[18] {
  PDUID[1]              { 0x03 }
  TransactionID[2]      { 0xtttt }
  ParameterLength[2]    { 0x000D }
  TotalServiceRecordCount[2]   { 0x0002 }
  CurrentServiceRecordCount[2] { 0x0002 }
  ServiceRecordHandleList[8] {
    0xqqqqqqqq  /* 打印服务 1 句柄 */
    0xrrrrrrrr  /* 打印服务 2 句柄 */
  }
  ContinuationState[1] { 0x00 }
}

B.2示例 2 — ServiceAttributeRequest（属性查询）

场景：使用示例 1 获得的句柄 (qqqq)，查询该打印服务的 ProtocolDescriptorList 属性（属性 ID = 0x0004）。

/* 客户端 → 服务器: 查询打印服务1的协议描述列表 */
SDP_SERVICE_ATTR_REQ[17] {
  PDUID[1]         { 0x04 }
  TransactionID[2] { 0xuuuu }
  ParameterLength[2] { 0x000C }
  ServiceRecordHandle[4]        { 0xqqqqqqqq }
  MaximumAttributeByteCount[2]  { 0x0080 }
  AttributeIDList[5] {
    DataElementSequence[5] {
      0b00110 0b101 0x03
      AttributeID[3] { 0b00001 0b001 0x0004 }  /* ProtocolDescriptorList */
    }
  }
  ContinuationState[1] { 0x00 }
}

/* 服务器 → 客户端: 返回协议栈 (L2CAP → RFCOMM(ch=2) → PostscriptStream) */
SDP_SERVICE_ATTR_RSP[38] {
  PDUID[1]             { 0x05 }
  TransactionID[2]     { 0xuuuu }
  AttributeListByteCount[2] { 0x001E }
  AttributeList[30] {
    DataElementSequence {
      Attribute { ID=0x0004, Value=ProtocolDescriptorList[
        [L2CAP_UUID],
        [RFCOMM_UUID, ServerChannel=2],
        [PostscriptStream_UUID]
      ]}
    }
  }
  ContinuationState[1] { 0x00 }
}
