链路管理协议
Link Manager Protocol (LMP)

LMP 是蓝牙 BR/EDR 控制器中负责 链路建立与控制 的核心协议层，运行于两个设备的链路管理器（Link Manager）之间，用于协商和管理物理链路、逻辑传输及逻辑链路的一切方面。LMP 消息不会向上层传播，完全由 Link Manager 解释处理。

📡承载于 ACL-C / APB-C 逻辑链路
🔐功能 认证 · 加密 · 配对
⏱超时 LMP 响应超时 30s
📦最大包长 DM1 = 17 字节
🔢Opcode 7 位 / 15 位
§ 1
引言 — Introduction

链路管理协议（LMP）用于控制和协商两台蓝牙设备之间连接运作的所有方面，包括逻辑传输（Logical Transport）、逻辑链路（Logical Link）和物理链路（Physical Link）的建立与控制。LMP 消息在两台设备的链路管理器（Link Manager）之间传输，不会被传递到更高的协议层。

RF（射频层）2.4 GHz ISM Band 物理层
🔗

物理链路控制

管理两设备间的 RF 物理链路，含功率控制、时钟同步等基础参数协商。

🚌

逻辑传输管理

建立和控制 ACL、SCO、eSCO 等不同类型的逻辑传输通道。

🔐

安全与认证

实现配对、认证和加密的完整安全流程，保障链路数据安全。

§ 2
通用规则 — General Rules

本节定义 LMP 的基本运作规则，包括消息传输方式、同步机制、报文格式、事务管理、错误处理等核心机制。

2.1 消息传输 — Message Transport

LMP 消息通过 ACL-C 或 APB-C 逻辑链路传输，这两种链路专用于控制信令，与承载 L2CAP 用户数据的 ACL-U / APB-U 链路通过包头中的 LLID 字段加以区分。控制逻辑链路的优先级高于其他流量。

LMP 消息本身不包含额外的错误检测字段，依赖基带 ACL 逻辑传输的纠错能力以及对消息内容的合理性检查（sanity check）来保障正确性。

🔌 逻辑链路分类与优先级

🔵 控制链路（Control）

ACL-C：承载 LMP 控制消息，基于默认 ACL 逻辑传输
APB-C：承载 LMP 控制消息，基于 APB 逻辑传输
高优先级，优先于用户数据传输

🟢 用户数据链路（User）

ACL-U：承载 L2CAP 及用户数据
APB-U：承载 APB 广播的用户数据
通过 LLID 字段与控制链路区分

ℹ️
FLOW 位在 LMP 数据包头中始终为 1，接收方应忽略该位。APB-C 逻辑链路上的 LMP 消息有特殊规则，不受 LMP 响应超时约束。
2.2 同步机制 — Synchronization

由于 LMP 消息承载在不保证投递时间与确认时间的链路上，因此需要精确的同步机制来保证两端状态的一致性。例如：

  • 逻辑传输地址（LT_ADDR）的复用时间点通过协议协商确定
  • Hold 模式、角色切换等操作以蓝牙时钟值作为参数来定义固定同步点
  • Sniff 模式的进出通过过渡模式（transition mode）保护
  • Central 以不少于每 T_poll 时隙一次的频率与 Peripheral 通信

2.3 报文格式 — Packet Format

每个 PDU 被分配一个 7 位或 15 位的操作码（Opcode） 来唯一标识 PDU 类型。

  • 7 位 Opcode：最常用的 PDU，Opcode 的高 7 位 + 事务 ID (TID) 位于 Payload 第 1 字节
  • 15 位 Opcode：当初始 7 位为特殊转义值（124~127）时，第 2 字节包含扩展 Opcode
  • 参数：紧跟在 Opcode 之后存放，7 位 Opcode 时从第 2 字节开始，15 位 Opcode 时从第 3 字节开始
  • LMP 消息使用 DM1 包传输；若存在 HV1 SCO 链路且 Payload ≤ 9 字节，可使用 DV 包
📦 LMP PDU Payload 格式

7 位 Opcode 格式（最常用）：

bit 0 TID
bits 7:1 Opcode (7-bit)
bytes 2…N Parameters
1 bit
7 bits
可变长度

15 位 Opcode 格式（Escape 扩展）：

bit 0 TID
bits 7:1 = 124~127 Escape Opcode
byte 2 Extended Opcode (8-bit)
bytes 3…N Parameters
💡
Escape 值 124 ~ 127（0x7C~0x7F）是保留的扩展标志，用于指示后续跟随 8 位扩展 Opcode，两者合并构成 15 位 Opcode。
2.4 事务机制 — Transactions

LMP 以事务（Transaction）为单位运作。一个事务是为实现特定目的而组成的一组消息交换，属于同一事务的所有 PDU 具有相同的事务 ID（TID）。

  • TID = 0：由 Central（主设备）发起的事务
  • TID = 1：由 Peripheral（从设备）发起的事务
  • 配对和加密等过程的所有序列共享同一个事务 ID
  • LMP_SETUP_COMPLETE 是独立 PDU，自成一个事务
🔄 事务流程示意 — 以连接建立为例
2.4.1 LMP 响应超时 — LMP Response Timeout

从接收到含 LMP PDU 的基带数据包，到发送有效响应 PDU 的时间间隔，必须小于 LMP 响应超时（30 秒）。超时或检测到链路丢失时，等待方应判定该过程已失败终止。

⏰

超时时限

固定为 30 秒，每次发送需要回复的 LMP PDU 时重置计时器。

🔁

适用范围

适用于 §4 中定义的所有过程序列，也适用于加密等期间禁用 ACL-U 流量时的一系列事务。

🚫

例外

APB-C 逻辑链路上的 LMP 消息有特殊规则，不受此超时约束。

2.5 错误处理 — Error Handling

LM 针对各种异常情况有明确的响应规则：

⚠️ 错误类型与对应处理方式
错误类型错误码处理方式
未知 Opcode 0x19 Unknown LMP PDU 回复 LMP_NOT_ACCEPTED[_EXT]，Opcode 参数为未识别的操作码
参数后含多余数据 0x1E Invalid LMP Parameters 可忽略多余数据，或回复 LMP_NOT_ACCEPTED[_EXT]
PDU 过短（参数缺失） 0x1E Invalid LMP Parameters 可以实现特定值继续，或回复 LMP_NOT_ACCEPTED[_EXT]
参数无效（长度正确） 0x1E Invalid LMP Parameters 必须回复 LMP_NOT_ACCEPTED[_EXT]
PDU 当前不被允许 0x24 LMP PDU Not Allowed 若该 PDU 需要回复则回复拒绝；若不需要回复则忽略
可选但不支持的特性 0x1A Unsupported LMP Feature 若需要回复则拒绝，否则忽略
响应超时 / 链路丢失 — 判定该过程失败终止
2.5.1 事务冲突解决 — Transaction Collision Resolution

当两端 LM 同时发起相同或不同且互不兼容的过程时，需要解决冲突：

⚡

同类事务冲突

双方发起相同过程 → Central 拒绝 Peripheral 发起的过程
错误码：0x23 LMP Error Transaction Collision
随后完成 Central 发起的过程。

💥

不同事务冲突

双方发起不同且不兼容的过程 → Central 拒绝 Peripheral 发起的过程
错误码：0x2A Different Transaction Collision
随后完成 Central 发起的过程。

2.7 通用响应消息 — General Response Messages

LMP 定义了四个通用响应 PDU，用于对各种请求 PDU 进行确认或拒绝：

📨 通用响应 PDU 一览
M/OPDU 名称适用 Opcode内容
M LMP_ACCEPTED 7 位 Opcode Opcode（被接受的操作码）
M LMP_NOT_ACCEPTED 7 位 Opcode Opcode + Error_Code（拒绝原因）
O LMP_ACCEPTED_EXT 15 位 Opcode Escape_Opcode + Extended_Opcode
O LMP_NOT_ACCEPTED_EXT 15 位 Opcode Escape_Opcode + Extended_Opcode + Error_Code
💡
规则：当被响应的 PDU 使用 7 位 Opcode 时，必须使用 LMP_ACCEPTED/NOT_ACCEPTED；当 PDU 使用 15 位 Opcode 时，必须使用 _EXT 版本。
2.8 LMP 消息约束 — Message Constraints

LMP 的设计遵循以下原则：

📏

最大长度约束

任何 LMP 消息不超过单个 DM1 包的最大 Payload 长度，即 17 字节。

📐

固定长度

所有 LMP 消息均为固定长度，便于解析和校验，无需长度字段。

🔖

版本号不指示功能

LMP 版本号不用于指示功能的存在与否，功能支持通过特性掩码（Feature Mask）声明。

§ 3
设备特性 — Device Features
3.1 特性概述 — General Description

每个 PDU 在规范中标注为强制（M）或可选（O）。可选 PDU 的支持情况通过特性掩码（Features Mask）来声明和读取。

  • LM 不必能够发送可选 PDU——只有在双方特性掩码均支持时才使用
  • 设备支持的特性集合在有连接期间不得改变
  • 对端可能会缓存本端的特性掩码，本端也可能缓存对端的特性掩码
  • 特性掩码可通过 §4.3.4 定义的过程来读取
🎯 特性协商流程

① 特性读取（Feature Exchange）

连接建立后，双方交换各自的 Features Mask，了解对端支持哪些可选功能。

② 扩展特性读取（Extended Features）

对于扩展特性（Feature Page 1/2 等），通过专用 PDU 进一步交换特性页信息。

③ 兼容性判定

LM 根据双方特性掩码的交集，判断哪些可选 PDU 可以使用，避免发送对端不支持的 PDU。

④ 特性缓存

连接期间可缓存对端特性掩码，但若特性集发生变化须重新协商（连接存在期间不允许改变）。

§ 4
过程规则 — Procedure Rules

本节是 LMP 规范的核心，详细定义了所有链路管理过程（Procedure）的消息序列和规则。主要分为链路控制、安全、连接控制、SCO/eSCO 音频和功率控制等类别。

4.1 链路控制过程 — Link Control Procedures
🔗 连接建立过程 — Connection Establishment (§4.1.1)

连接建立涉及主机连接请求、特性交换和连接完成等步骤，以下为典型序列：

🔄

角色切换 (§4.1.14) — Role Switch

允许 Central 和 Peripheral 互换角色。通过 LMP_SWITCH_REQ 发起，包含 Switch_Instant（蓝牙时钟锚点）以确保双方同步切换。切换后原 Peripheral 成为新的 Central。

📊

服务质量 QoS (§4.1.8)

通过 LMP_QOS_REQ / LMP_QOS 协商 Poll_Interval（轮询间隔）和 N_BC（广播重传次数），以满足应用层的 QoS 需求。

🌐

AFH 自适应跳频 (§4.1.15)

通过 LMP_SET_AFH 激活/停用 AFH，协商 AFH 信道图（Channel Map）和生效时刻（AFH_Instant），以规避固定频段干扰。

🔌

链路断开 — Detach

通过 LMP_DETACH 通知对端断开连接，附带 Error_Code 说明断开原因（如用户请求、认证失败等）。

4.2 安全过程 — Security Procedures

LMP 安全过程涵盖设备配对、认证和加密三大部分，共同保障蓝牙链路安全。

🔐 认证过程 — Authentication (§4.2.1)

认证基于质询-响应机制（Challenge-Response），使用共享链路密钥（Link Key）进行 E1 算法计算。

🔑 安全简单配对 SSP — Secure Simple Pairing (§4.2.2)

SSP 基于椭圆曲线 Diffie-Hellman（ECDH）密钥交换，使用 P-192 或 P-256 曲线，极大提升了安全性并简化了用户操作体验。

🔢

① 公钥交换

通过 LMP_ENCAPSULATED_HEADER/PAYLOAD 交换 ECDH 公钥（P-192 或 P-256）

🎲

② 认证阶段

根据 IO 能力选择关联模型：Numeric Comparison / Passkey Entry / Just Works / OOB

🔐

③ DHKey 计算

双方各自计算 DHKey = ECDH(私钥, 对端公钥)，作为后续密钥派生基础

🗝

④ 链路密钥生成

通过 H3/H4 等 KDF 派生链路密钥（Link Key），用于后续认证和加密

4.3 连接控制过程 — Connection Control Procedures
📛

名称读取 (§4.3.1) — Name Request

通过 LMP_NAME_REQ 请求对端设备的友好名称，对端以 LMP_NAME_RES 分片返回（最长 248 字节），含 Name_Offset 和 Name_Length 字段。

🕒

时钟偏移 (§4.3.3) — Clock Offset

通过 LMP_CLKOFFSET_REQ / LMP_CLKOFFSET_RES 获取对端时钟偏移，用于加快后续重新连接时的寻呼（Paging）过程。

📋

特性读取 (§4.3.4) — Supported Features

通过 LMP_FEATURES_REQ / LMP_FEATURES_RES 及 LMP_FEATURES_REQ_EXT / LMP_FEATURES_RES_EXT 读取基本和扩展特性掩码（最多 3 页）。

🔢

版本信息 (§4.3.6) — Version

通过 LMP_VERSION_REQ / LMP_VERSION_RES 交换 LMP 版本号（Version）、公司标识符（Company_Identifier）和子版本号（Subversion）。

4.4 SCO / eSCO 音频过程

SCO（同步面向连接）和 eSCO（扩展 SCO）逻辑传输用于传输同步音频数据（如语音通话）。LMP 负责这些音频通道的建立、修改和释放。

4.5 多时隙与功率控制
📶

最大时隙数 (§4.5.1) — Max Slots

通过 LMP_MAX_SLOT / LMP_MAX_SLOT_REQ 协商单个数据包可占用的最大时隙数（1、3 或 5 时隙），影响吞吐量和延迟。

⚡

功率控制 (§4.5.4) — Power Control

通过 LMP_POWER_CONTROL_REQ / LMP_POWER_CONTROL_RES 请求对端调整发射功率（增大/减小/保持），以优化链路质量和功耗。

📊

数据速率 (§4.5.2) — Data Rate

通过 LMP_PREFERRED_RATE 建议首选数据速率，支持标准速率（BR）和增强数据速率（EDR 2Mbps/3Mbps）之间的动态切换。

🔢

时钟调整 (§4.5.6) — Clock Adjust

通过 LMP_CLK_ADJ 系列 PDU 实现精确的时钟微调（Clk_Adj_Offset），用于 Peripheral 广播（APB）等需要高精度同步的场景。

4.6 低功耗模式 — Low Power Modes

LMP 定义了两种 BR/EDR 低功耗工作模式，让设备在不活跃时节省能量：

😴 Sniff 模式与 Hold 模式对比
💤

Hold 模式 (§4.6.1)

PDU：LMP_HOLD_REQ / LMP_HOLD
参数：Hold_Instant, Hold_Time
特点：设备在指定时隙后暂停 ACL 通信并进入低功耗状态，Hold 时间结束后自动恢复。期间 LT_ADDR 保留，可去执行 Inquiry 等操作。

🔕

Sniff 模式 (§4.6.2)

PDU：LMP_SNIFF_REQ / LMP_ACCEPTED
参数：TSniff, DSniff, Sniff_Attempt, Sniff_Timeout
特点：Peripheral 只在固定的 Sniff 时隙窗口内监听，其他时间可休眠。支持子速率（Sniff Subrating）进一步降低功耗。

5.3 LMP 封装数据 — LMP Encapsulated (Table 5.4)

LMP 封装机制用于在 LMP 帧内传输较大的二进制数据块，主要用于 SSP 配对时传输 ECDH 公钥。

🔑 LMP 封装数据类型
🔐

P-192 公钥

Major Type: 1，Minor Type: 1
Payload 长度：48 字节
格式：X 坐标 (Bytes 23:0) + Y 坐标 (Bytes 47:24)
字节序：小端（Little-endian）

🔒

P-256 公钥

Major Type: 1，Minor Type: 2
Payload 长度：64 字节
格式：X 坐标 (Bytes 31:0) + Y 坐标 (Bytes 63:32)
字节序：小端（Little-endian）

ℹ️
由于 LMP 单包最大 17 字节，公钥通过多个 LMP_ENCAPSULATED_PAYLOAD PDU 分片传输，由 LMP_ENCAPSULATED_HEADER 描述总长度和类型，LMP_ENCAPSULATED_PAYLOAD 携带实际数据（每次 16 字节）。
