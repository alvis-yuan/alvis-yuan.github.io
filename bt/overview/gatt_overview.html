第 1 章 · 引言 Introduction
定义 GATT 协议的背景与适用范围

1.1 范围 Scope

通用属性协议（Generic Attribute Profile，GATT）基于属性协议（ATT）定义了一套服务框架，描述了服务（Service）及其特征（Characteristic）的格式与过程，包括：

  • 🔍 发现（Discovering）：发现服务和特征
  • 📖 读取（Reading）：读取特征值
  • ✏️ 写入（Writing）：写入特征值
  • 🔔 通知（Notifying）：服务端主动推送更新
  • 📣 指示（Indicating）：带确认的服务端推送
  • 📡 广播（Broadcasting）：配置特征广播
GATT 核心功能全景
🔍
服务发现
客户端查询服务端上有哪些 Primary/Secondary Service 及其 Characteristic
📖
特征读取
通过 ATT_READ_REQ 读取服务端的特征值，获取传感器等数据
✏️
特征写入
通过 ATT_WRITE_REQ 或 ATT_WRITE_CMD 向服务端写入控制命令
🔔
通知与指示
服务端主动推送数据变化，Notification 无需确认，Indication 需要确认
📡
广播配置
允许设备通过广播包发送特征值（Server Characteristic Configuration）
⚡
ATT Bearer
基于 L2CAP 信道承载，可建立多条 ATT Bearer 提升并发性能

1.2 Profile 依赖关系

GATT 位于蓝牙协议栈的 Profile 层，依赖于通用访问协议（GAP）和属性协议（ATT）。

1.5 约定 Conventions

  • 过程名、PDU、Opcode 等专有名称使用 斜体 表示
  • «Primary Service» 双尖括号表示 UUID 类型
  • ATT 错误码使用斜体并附带十六进制数值，如 Database Out Of Sync (0x12)
  • 多字节字段采用小端序（LSB 优先）传输，Characteristic Value 字段除外

2.5 Attribute 协议 (ATT)

2.5.1 ATT PDU 结构

GATT 使用 ATT PDU 在设备间传输数据。每个 PDU 的结构如下：

图 2.3：ATT PDU 格式
Opcode
1 字节
Attribute Parameters
可变长度
Auth Signature
12 字节（可选）
Opcode：包含具体的操作码（命令/请求/响应/指示/通知/确认）以及认证标志位
Attribute Parameters：针对具体操作的数据内容或响应结果
Authentication Signature：可选的 12 字节签名，用于命令认证（ATT_SIGN_FLAG=1 时有效）

服务端上的每个 Attribute（属性）由四个部分组成：

图 2.4：Attribute 逻辑结构

2 字节，唯一索引
0x0001–0xFFFF

UUID（16位或128位）
描述属性类型
Attribute Value
可变长度
实际数据内容
Attribute Permissions

💡 Attribute Handle：服务端上属性的唯一标识索引，Handle 值随地址递增排列（允许有间隙）
💡 Attribute Type：UUID 描述该属性的类型（如 Primary Service = 0x2800）
💡 Attribute Permissions：读/写权限控制，不能通过 ATT 协议直接读写，由 GATT 或高层 Profile 定义

2.5.2 属性缓存 Attribute Caching

属性缓存允许 Client 在发现 Attribute 信息后，在重新连接时复用缓存数据，无需重复发现，大幅节省时间和数据包交换。

缓存有效性状态机
缓存有效
Change-Aware
→
服务发生变化
（添加/修改/删除服务）
→
缓存失效
Change-Unaware
1. 服务变化时
Server 发送 ATT_HANDLE_VALUE_IND（包含受影响的 Handle 范围），通知 Client 缓存失效范围
2. Client 收到指示
Client 将受影响范围的缓存标记为无效，发送 ATT_HANDLE_VALUE_CFM 确认
3. 恢复缓存
Client 重新执行服务发现，或读取 Database Hash 特征以更新缓存（使用 Robust Caching 特性）
4. Bonded 设备
对于有信任关系（Bond）的设备，断开连接后缓存保持有效；非 Bond 设备仅在单次连接期间有效
⚠️ Robust Caching（强健缓存）：若 Server 支持 Database Hash + Service Changed 特征，则必须支持 Robust Caching。当 Client 处于 Change-Unaware 状态时，Server 返回 ATT_ERROR_RSP（Error Code = Database Out Of Sync = 0x12），直到 Client 读取 Database Hash 后重新变为 Change-Aware 状态。

2.5.3 属性分组 Attribute Grouping

GATT 将属性按服务和特征进行逻辑分组，形成有层次的数据库结构：

GATT 属性数据库层次结构
Primary Service «Temperature Service» [Handle: 0x0010–0x001F]
  │
  ├─ Characteristic Declaration [Handle: 0x0011]
  │   Properties: Read|Notify
  │   Value Handle: 0x0012
  │   UUID: «Temperature Measurement»
  │
  ├─ Characteristic Value [Handle: 0x0012]
  │   Value: 0x028A (= 6.5°C)
  │
  ├─ CCCD Descriptor [Handle: 0x0013]
  │   UUID: 0x2902 (Client Characteristic Config)
  │
  └─ User Description [Handle: 0x0014]
      "Outside Temperature"

2.6 服务定义 Service Definition

GATT 定义了两类服务：

🔵
Primary Service（主服务）
UUID = 0x2800。设备对外暴露的主要功能服务。Client 通过发现主服务来了解设备能力。
🟣
Secondary Service（次级服务）
UUID = 0x2801。不直接对外，由其他服务通过 Include 引用，提供共享功能模块。

服务可以通过 «Include»（UUID = 0x2802）引用其他服务，实现服务复用。

2.7 特征定义 Characteristic Definition

每个特征（Characteristic）由以下属性组成：

组成部分 UUID 说明
Characteristic Declaration 0x2803 包含 Properties（属性标志位）、Value Handle、UUID
Characteristic Value 特征专属 UUID 实际存储特征数据的属性
Descriptor（描述符） 各种 0x29xx 补充元数据，如 CCCD(0x2902)、格式(0x2904)、用户描述(0x2901)
Characteristic Properties 标志位
Broadcast (0x01) Read (0x02) Write Without Response (0x04) Write (0x08) Notify (0x10) Indicate (0x20) Authenticated Signed Writes (0x40) Extended Properties (0x80)

第 3 章 · 服务交互要求
Client 与 Server 在交互过程中必须遵守的行为规范

本章规定了 GATT Client 和 Server 的基本行为要求：

Client / Server 行为规范
📱 Client 要求
  • 每次连接只能发出一个未完成的请求事务
  • 必须处理来自 Server 的错误响应
  • 必须支持 MTU 交换过程
  • 对于绑定设备，应缓存 Attribute 信息避免重复发现
  • 收到 Service Changed 指示后必须执行服务重发现
🌡️ Server 要求
  • 对于不支持的操作返回 ATT_ERROR_RSP
  • Handle 值一经分配不应随意改变
  • 服务变化时须存储 Service Changed 信息（针对已绑定设备）
  • 支持 Robust Caching 时须正确管理 Change-Aware 状态
  • Database Hash 变化时须更新对应特征值
📋 如果服务端的 GATT 服务在设备生命周期内不会改变，则不需要包含 Service Changed 特征，Client 也无需在初次服务发现后重复发现。

第 5 章 · 认证与授权
GATT 操作的安全访问控制机制

GATT 的安全访问控制分为三个层次：

🔑
授权 Authorization
特定属性需要应用层明确授权才能访问，与配对/绑定无关
🛡️
认证 Authentication
要求设备已完成认证配对（使用 MITM 保护），可信身份访问
🔒
加密 Encryption
要求连接已建立加密链路，防止数据被窃听（LE 可选，BR/EDR 必选）
安全访问决策流程
Client 发送 ATT 请求
尝试读取或写入某个 Attribute
Server 检查 Attribute Permissions
验证当前连接安全等级是否满足该 Attribute 的访问权限
权限不足 → 返回错误码
Insufficient Authorization (0x08)、Insufficient Authentication (0x05)、Insufficient Encryption (0x0F)
权限满足 → 正常执行
执行读/写操作，返回正常响应
💡 ATT 认证签名（Authentication Signature）允许在未加密链路上对 ATT 命令进行签名验证，使用 CSRK（Connection Signature Resolving Key）和签名计数器，防止重放攻击。

7.2 Client Supported Features（客户端支持特性）

UUID = 0x2B29，Properties = Read/Write (0x0A)。Client 向 Server 声明自己支持的可选 GATT 特性：

Bit特性名称含义
0Robust CachingClient 支持强健缓存机制，接受 Database Out Of Sync 错误响应
1Enhanced ATT BearerClient 支持 EATT（增强型 ATT Bearer）并发传输
2Multiple Handle Value NotificationsClient 支持在单个 NTF PDU 中接收多个 Handle 的值

7.3 Database Hash（数据库哈希）

UUID = 0x2B2A，Properties = Read (0x02)。128 位（16字节）AES-CMAC 哈希值，反映当前 GATT 数据库的状态：

Database Hash 计算方法

将所有属性的 [Handle + Type + Value]（按规则选择性包含）拼接为字节流 m，然后：

// Hash key k = 固定 128-bit 全零密钥
Database Hash = AES-CMAC(k = 0x00...00, m = 属性数据拼接)

// 示例结果（见附录B）：
Hash = F1 CA 2D 48 EC F5 8B AC 8A 88 30 BB B9 FB A9 90
Client 可以通过比较缓存的 Hash 与当前读取到的 Hash 来判断数据库是否变化，无需重新完整发现。
🔏 Database Hash 中不包含特征值本身（只包含声明和描述符），也不包含 Service Changed 特征值和 Client Supported Features 特征值。Hash 变化时 Server 必须更新此特征。

7.4 Server Supported Features（服务端支持特性）

UUID = 0x2B3A，Properties = Read (0x02)。Server 声明自己支持的可选 GATT 特性：

Bit特性名称含义
0Enhanced ATT BearerServer 支持 EATT 并发 ATT Bearer

附录 B · Database Hash 计算示例
演示如何从 GATT 数据库计算 AES-CMAC Database Hash 值

Database Hash 将数据库中的特定属性数据按规则拼接为字节流 m，再计算 AES-CMAC：

哪些属性字段参与 Hash 计算
属性类型 H（Handle） T（Type/UUID） V（Value）
Primary/Secondary Service 声明✅✅✅
Characteristic 声明✅✅✅
Include 声明✅✅✅
Characteristic Value❌❌❌
CCCD 描述符✅✅❌（不含值）
Extended Properties 描述符✅✅✅

🎯 GATT 协议核心要点总结
🏗️ 基础架构
基于 ATT，分为 Client/Server 角色，运行于 L2CAP ATT Bearer
📂 数据模型
服务→特征→描述符三层结构，每个属性由 Handle+Type+Value+Permission 构成
⚡ 高效通信
支持属性缓存、Robust Caching、EATT 并发、MTU 协商等优化手段
🔔 数据推送
Notification（无确认）和 Indication（需确认）实现服务端主动推送
🔒 安全机制
三层安全（授权/认证/加密），LE 可选，BR/EDR 强制加密
🔄 缓存管理
Database Hash 快速验证数据库变化，Service Changed 指示精确通知范围
