OVERVIEW

蓝牙 L2CAP 协议完全指南

L2CAP（逻辑链路控制与适配协议）是蓝牙协议栈中的核心层，位于 HCI 之上、应用/高层协议之下。 本文档严格按照蓝牙核心规范 6.0 的章节顺序，逐节解读 L2CAP 的设计目标、数据结构与状态机行为，并配以可视化图表辅助理解。

图0-1：蓝牙协议栈总览与 L2CAP 位置

🔗 连接复用

在同一 ACL 逻辑链路上，通过 CID 标识多个独立的 L2CAP 信道，实现多协议共存。

✂️ 分段与重组

将上层大数据包（SDU，最大 64KB）切分为适合 Controller 传输的小帧（PDU）并在接收端重组。

🚦 流量与错误控制

提供基于窗口的流控机制和 PDU 重传，确保高可靠性数据传输。

SECTION 1

1. 介绍（Introduction）

L2CAP 是蓝牙主机协议栈中的核心层，为高层协议和应用提供 面向连接 和 无连接 的数据服务， 并具备 协议复用、分段重组（SAR） 能力，允许传输最大 64KB 的 SDU。 L2CAP 位于一个或多个 Controller 逻辑链路之上，提供 L2CAP 信道（逻辑信道），并被高层协议（如 RFCOMM、SDP、ATT）所使用。

图1-1：L2CAP 架构模块（L2CAP Architectural Blocks）

1.1 L2CAP 功能特性（L2CAP Features）

L2CAP 的功能需求覆盖以下七大核心特性，缺一不可：

🔀 协议/信道复用

在同一 Controller 之上，通过协议复用将连接路由到正确的上层协议（如 RFCOMM、ATT 等），通过逻辑信道复用区分多个上层实体。

ℹ️L2CAP 信道只能在一个 Controller 上运行，不能在 Controller 之间迁移。
✂️ 分段与重组（SAR）

Resource Manager 允许应用控制传输帧长度，L2CAP 控制 PDU 长度带来以下好处：

1

交错传输不同应用数据以满足时延需求

2

更容易进行内存和缓冲区管理

3

提升重传效率，减少损失数据量

🚦 每信道流控

当多个数据流共享同一 Controller 时，每个 L2CAP 信道需要独立流控。L2CAP 提供 基于窗口（Window-based） 的流控方案。

🔁 错误控制与重传

L2CAP 提供 PDU 级别的错误检测和重传，保护传输免受 Controller 伪接收错误和刷新丢包的影响。

🎵 流媒体支持

音频等流媒体应用以固定速率传输数据，不需要流控干预。通过刷新超时（Flush Timeout）和流媒体模式（Streaming Mode）绕过流控机制。

🧩 分片与重组合

低层（HCI、Controller）可能进一步分片 L2CAP PDU，接收方 L2CAP 层将所有分片重组为原始 PDU。

⚠️不同层次、不同设备的分片大小可能不同，但最终 PDU 保持完整。
📊 服务质量（QoS）

L2CAP 连接建立过程允许两端交换 QoS 期望信息。通过设置 HCI ACL 数据包的 Packet_Boundary_Flag，可同时支持：

等时（Guaranteed）

可自动刷新的 L2CAP 包，按照 ACL 链路的自动刷新超时进行刷新，适用于实时流媒体。

异步（Best Effort）

不可自动刷新的 L2CAP 包，不受自动刷新超时影响，可用 HCI_Flush 命令手动刷新。

1.2 协议假设（Assumptions）

1

有序交付

Controller 提供有序数据包交付，允许单包损坏或重复。BR/EDR/LE 设备间任意两设备间最多只有一条 ACL-U 和一条 LE-U 逻辑链路。

2

全双工信道

Controller 始终提供全双工通信的表象。单向流量不要求双向信道，但接口始终具备双向能力。

3

有限可靠性

L2CAP 提供一定程度的可靠性，通过分段、错误检测和重传增强。某些 Controller 会在超时后丢弃数据（flush）。

4

四种流控/错控模式

增强重传模式、重传模式、流控模式、流媒体模式——涵盖从高可靠到低延时的各种需求。

1.3 范围（Scope）

🚫
L2CAP 不处理以下内容：
• 不传输用于 SCO 或 eSCO 逻辑传输的同步数据（语音/实时音频由 Baseband 直接处理）
• 不支持可靠广播信道（Connectionless Data Channel 不提供可靠广播）

SECTION 2

2. 信道标识符（Channel Identifiers, CID）

CID 是 16 位无符号整数，用于在对等设备的 L2CAP 层之间标识一个逻辑信道端点。 CID 是本地端点标识符，两端的 CID 可以不同（在动态分配信道中几乎总是不同）。

图2-1：CID 空间分布

2.1 空标识符（Null Identifier, 0x0000）

🚫 CID 0x0000 为空标识符（Null Identifier），不可用于任何 L2CAP 连接。任何包含此 CID 的包均为非法包，应丢弃。

2.2 固定信道（Fixed Channels）

CID描述适用于
0x0001L2CAP 信令信道（L2CAP Signaling Channel）BR/EDR
0x0002无连接数据信道（Connectionless Data Channel）BR/EDR
0x0003AMP 管理协议（AMP Manager Protocol）BR/EDR
0x0004属性协议（ATT）信道LE
0x0005LE L2CAP 信令信道LE
0x0006安全管理协议（SMP）信道LE
0x0007安全管理协议（SMP）信道（BR/EDR）BR/EDR
0x0008–0x003E保留—
0x003FAMP 测试管理器BR/EDR

2.3 动态分配信道（Dynamically Allocated Channels）

动态信道在连接建立时按需分配，连接释放后 CID 归还。BR/EDR 的动态范围是 0x0040–0xFFFF， LE 的动态范围是 0x0040–0x007F。动态信道需要通过信令包（L2CAP_CONNECTION_REQ / RSP）建立。

ℹ️ 发起方和响应方各自独立分配本地 CID，通过信令包将对方的 CID（Destination CID）告知本地，形成双向信道映射。
SECTION 3

3. 操作模式（Operational Modes）

L2CAP 支持多种操作模式，每种模式具有不同的流控、错误控制和分段能力，适用于不同场景。 模式在信道配置阶段通过协商确定。

图3-1：L2CAP 操作模式对比

3.1 基本 L2CAP 模式（Basic L2CAP Mode）

最简单的操作模式，无分段、无流控、无错误控制。PDU 直接以 B-frame 格式发送， 数据可靠性完全依赖 Controller 层。所有 BR/EDR 设备必须支持此模式作为默认模式。

图3-2：Basic L2CAP Mode — B-frame 结构

3.2 无连接数据信道（Connectionless Data Channel）

使用固定 CID 0x0002，支持从一对多的广播传输（BR/EDR）。 无连接数据信道不提供可靠传输保证，数据包头包含 PSM 字段用于上层协议标识。

⚠️ L2CAP 不支持可靠广播信道。无连接信道中的广播是不可靠的。

3.3 流量控制模式（Flow Control Mode）

在基本模式基础上增加了分段和流控功能，使用 I-frame 发送数据，接收方通过 RR/RNR 帧控制数据流。 不提供重传：如果包丢失或损坏，信道将关闭。

3.4 重传模式（Retransmission Mode）

在流量控制模式基础上增加了 PDU 重传能力。使用滑动窗口和 REJ（拒绝）机制请求重传。 适用于需要可靠传输但不要求高效率的场景（已被增强重传模式取代）。

3.5 增强重传模式（Enhanced Retransmission Mode, ERTM）

最完整的可靠传输模式，支持 REJ（Go-Back-N）和 SREJ（选择性拒绝）两种重传策略， 可更精确地只重传丢失的帧。配合监控定时器（MonitorTimer）和重传定时器（RetransmissionTimer）使用。

图3-3：增强重传模式 — SREJ 选择性重传示意

3.6 流媒体模式（Streaming Mode）

专为音频等流媒体应用设计，发送方持续发送 I-frame，接收方不发送 RR/RNR/REJ/SREJ， 不进行重传。丢失的帧被标记为丢失后报告给重组函数。 必须为发送方设置有限的 Flush Timeout。

🎵 流媒体模式保持数据流的稳定性，即使有丢包也不中断，适合对时延敏感而对可靠性要求相对低的场景（如音频流）。

3.7 LE 信用流控模式（LE Credit Based Flow Control Mode）

专为 LE 设计的连接导向信道模式，使用信用（Credit）机制控制数据流。 发送方每发送一个 K-frame 消耗一个信用，接收方通过 L2CAP_FLOW_CONTROL_CREDIT_IND 归还信用。

图3-4：信用流控机制示意

3.8 增强信用流控模式（Enhanced Credit Based Flow Control Mode）

扩展了 LE 信用流控模式，支持在 LE 和 BR/EDR 上使用，并允许单次请求建立最多 5 个信道（通过 L2CAP_CREDIT_BASED_CONNECTION_REQ）。ACL 逻辑传输必须设置无限自动刷新超时。

vs LE 信用流控

支持 BR/EDR + LE；单次可建立多个信道；CID 范围相同（0x0040–0x007F for LE）。

安全要求

在 LE 传输上收到连接请求时，需验证认证、授权和加密密钥长度，否则拒绝连接。

SECTION 4

4. 信令包格式（Signaling Packet Formats）

信令包在固定信道 0x0001（BR/EDR）或 0x0005（LE）上传输， 用于信道管理操作。每个信令包由一个或多个命令组成，每个命令包含 Code、Identifier、Length 字段。

图4-1：信令命令通用格式

4.1 命令拒绝（L2CAP_COMMAND_REJECT_RSP, Code 0x01）

当收到无法理解或无法处理的信令命令时，发送此响应。包含拒绝原因（Reason）和可选数据。

Reason 值含义
0x0000不支持的命令（Command not understood）
0x0001信令 MTU 超出（Signaling MTU exceeded）
0x0002无效的 CID（Invalid CID in request）

4.2 连接请求/响应（L2CAP_CONNECTION_REQ / RSP）

用于在两个 BR/EDR 设备之间建立 L2CAP 动态信道。

图4-2：L2CAP 连接建立信令流程
Result 值含义
0x0000连接成功（Connection successful）
0x0001连接待定（Connection pending）
0x0002PSM 不支持（Connection refused – PSM not supported）
0x0003安全块（Connection refused – security block）
0x0004无资源可用（Connection refused – no resources available）

4.3 配置请求/响应（L2CAP_CONFIGURATION_REQ / RSP）

连接建立后，双方通过配置命令协商信道参数（MTU、刷新超时、QoS、重传/流控模式等）。 配置过程是双向的：双方各自发起配置请求，独立协商本端的接收参数。

ℹ️ 配置响应的 Result 为 0x0001（Failure - unacceptable parameters）时，响应中包含对方可接受的参数值，发起方应重新发送配置请求。

4.4 断开连接请求/响应（L2CAP_DISCONNECTION_REQ / RSP）

用于释放一条已建立的 L2CAP 动态信道。任意一方均可发起断连请求， 收到断连请求的一方必须响应 L2CAP_DISCONNECTION_RSP。

图4-3：L2CAP 断连流程

4.5 信息请求/响应（L2CAP_INFORMATION_REQ / RSP）

用于查询对端 L2CAP 的扩展功能支持情况，如：

扩展功能掩码（Extended Features Mask） 固定信道支持（Fixed Channels Supported） 无连接 MTU

4.6 LE 信用连接请求（L2CAP_LE_CREDIT_BASED_CONNECTION_REQ / RSP）

用于在 LE 设备间建立信用流控信道，包含 PSM、SCID、MTU、MPS 和初始信用（Credits）字段。 响应方分配 DCID 并返回自身的 MTU、MPS 和初始信用。

4.7 流量控制信用指示（L2CAP_FLOW_CONTROL_CREDIT_IND）

在信用流控模式中，接收方处理完 K-frame 后归还信用，告知对端可继续发送更多 K-frame。 Credit 字段值为本次归还的信用数量，累积信用不得超过 65535，否则断开信道。

SECTION 5

5. 配置参数选项（Configuration Parameter Options）

配置参数在 L2CAP_CONFIGURATION_REQ/RSP 中以 TLV（Type-Length-Value）格式携带， 双方协商信道的工作参数。

5.1 MTU 选项（Maximum Transmission Unit）

MTU 指定了信道可接收的最大 SDU 字节数。若对方发送的 L2CAP PDU 超过本端 MTU， 行为未定义（由实现决定是否拒绝或截断）。默认 MTU 为 48 字节（BR/EDR）。

图5-1：MTU 与 MPS 的关系

5.2 刷新超时选项（Flush Timeout Option）

配置自动刷新超时（Automatic Flush Timeout），单位为毫秒（0.625ms 粒度）。 超时后，Controller 自动丢弃未成功发送的 PDU。

值含义
0x0000不可刷新（Non-flushable，不受超时影响）
0x0001–0xFFFE具体超时值（单位：0.625ms 粒度）
0xFFFF无限超时（Infinite flush timeout，等价于不刷新）

5.3 服务质量（Quality of Service, QoS）

QoS 选项定义了信道对资源的需求，包含以下关键参数：

参数含义
Service TypeNo Traffic / Best Effort / Guaranteed
Token Rate令牌桶填充速率（bytes/sec）
Token Bucket Size令牌桶大小（bytes）
Peak Bandwidth峰值带宽（bytes/sec）
Latency最大时延（μs）
Delay Variation最大时延抖动（μs）

5.4 重传与流控选项（Retransmission and Flow Control Option）

指定信道的工作模式及相关参数：

参数含义
ModeBasic / Retransmission / Flow Control / ERTM / Streaming
TxWindow Size发送窗口大小（帧数），1–63（ERTM 标准），1–16383（扩展窗口）
MaxTransmit最大重传次数
Retransmission Timeout重传超时（ms）
Monitor Timeout监控超时（ms），P-bit 响应等待
MPS最大 PDU 负载大小（octets）
SECTION 6

6. 状态机（State Machine）

L2CAP 为每个动态信道维护一个状态机，定义了信道从创建到关闭的完整生命周期。 每个状态下，只有特定的事件会触发状态转移或执行动作。

6.1 信道状态（Channel States）

图6-1：L2CAP 信道状态机（BR/EDR）

6.2 状态说明与转换条件

状态描述进入条件
CLOSED信道未建立，初始状态初始 / 断连完成
W4_L2CAP_CONNECT_RSP已发送连接请求，等待对端响应上层发起连接请求

W4_L2CA_DISCONNECT_RSP已响应断连请求，等待上层确认收到 DISCONNECTION_REQ
SECTION 7

7. PDU 格式规范（PDU Format Specifications）

L2CAP 定义了多种 PDU 类型，每种类型服务于不同的操作模式。所有 PDU 均以 Basic L2CAP Header（4字节）开头。

7.1 B-frame（Basic Information Frame）

用于基本 L2CAP 模式和无连接数据信道的数据帧格式，结构最简单。

图7-1：B-frame 格式

7.2 I-frame（Information Frame）

用于增强重传模式和流媒体模式的数据帧，携带控制字段（Control Field）。标准控制字段 2 字节，扩展控制字段 4 字节。

图7-2：I-frame 格式（增强重传模式，标准控制）

7.3 S-frame（Supervisory Frame）

监督帧，用于增强重传模式中的流控和错误控制信令（确认、流控、错误请求），不携带数据。 S-frame 类型由 Control Field 中的 Supervisory 字段（2 bit）决定。

S 值类型作用
0b00RR（Receiver Ready）确认接收，告知对方可继续发送
0b01REJ（Reject）Go-Back-N 重传请求：从 ReqSeq 开始重传所有帧
0b10RNR（Receiver Not Ready）告知对方本端缓冲已满，暂停发送
0b11SREJ（Selective Reject）选择性重传请求：只重传指定的 ReqSeq 帧

7.4 K-frame（Credit-Based Frame）

用于 LE 信用流控模式和增强信用流控模式的数据帧。 K-frame 包含 SDU Length 字段（仅首段）和 SDU 数据，每发送一个 K-frame 消耗一个信用。

图7-3：K-frame 格式
SECTION 8

8. 增强重传与流控细节（Enhanced Retransmission Details）

增强重传模式（ERTM）是 L2CAP 最复杂也最可靠的传输模式， 使用滑动窗口 + 选择性重传（SREJ）+ 监控/重传定时器实现高效可靠的双向数据传输。

8.1 发送 I-frame

发送方维护以下关键变量：

NextTxSeq

下一个待发送帧的序号，发送后递增 1（mod 64）。

TxWindow

发送窗口大小，限制未被确认的 I-frame 数量（最大 63 或 16383）。

UnackedFrames

当前未收到确认的 I-frame 数量，小于 TxWindow 时才可发送新帧。

RetransmissionTimer

每发送一个 I-frame 后启动，超时时触发 P-bit 查询（Poll）。

8.2 接收 I-frame

接收方维护 ExpectedTxSeq（期望接收的下一帧序号）。收到 I-frame 后：

1

序号正确

TxSeq == ExpectedTxSeq：将帧送往重组函数，ExpectedTxSeq++，发送 RR 确认。

2

序号超前（乱序）

TxSeq > ExpectedTxSeq：发送 SREJ 请求缺失帧，记录缺失帧列表（SrejList）。

3

序号无效

TxSeq 超出有效范围（TxWindow 一半以上）：关闭信道。

4

缓冲已满

无可用缓冲：发送 RNR 告知对方暂停发送，待缓冲可用后发送 RR。

8.3 确认机制（Acknowledgment）

RR/RNR/REJ 的 ReqSeq 字段确认 0 到 ReqSeq-1 的所有 I-frame（累积确认）。 P-bit（Poll bit）/ F-bit（Final bit）用于 MonitorTimer 超时后的健康检查：

图8-1：P/F-bit 交互机制

8.4 SREJ_SENT 状态

当接收到乱序 I-frame 并发送 SREJ 后，进入 SREJ_SENT 状态。 此状态下，所有收到的新 I-frame 将被缓存，等待缺失帧到达后按序提交重组。 SREJ 列表维护所有待重传的缺失帧序号。

⚠️ SREJ_SENT 状态下，MaxTransmit 次重传失败后信道将关闭（Close Channel）。

8.5 流媒体模式细节（Streaming Mode Specifics）

流媒体模式使用与 ERTM 相同的 I-frame 格式，但行为大相径庭：

发送方

设置 TxSeq = NextTxSeq，ReqSeq = 0（始终），NextTxSeq++；F-bit 始终为 0；不使用任何定时器。

接收方

收到乱序帧时，缺失帧标记为丢失，ExpectedTxSeq 直接跳到 TxSeq+1；不发送 RR/RNR/REJ/SREJ；无重传。

SECTION 10

10. 信用流控过程（Credit Based Flow Control Procedures）

信用流控是 LE L2CAP 面向连接信道的核心机制，通过信用（Credit）精确控制接收方缓冲。 有两种变体：LE 信用流控（L2CAP_LE_CREDIT_BASED_CONNECTION）和增强信用流控（L2CAP_CREDIT_BASED_CONNECTION）。

10.1 LE 信用流控模式规则

1

初始信用

信道建立时，双方在连接请求/响应中各自指定初始信用数量（Credits 字段）。

2

发送 K-frame

仅在信用 > 0 时可发送 K-frame，每发一个 K-frame 信用减 1。

3

归还信用

接收方任意时刻可发送 L2CAP_FLOW_CONTROL_CREDIT_IND 归还信用，发送方累加信用计数。

4

信用溢出保护

信用总数不得超过 65535。收到信用归还后若超出 65535，或信用为 0 时收到 K-frame，必须断开信道。

5

安全检查

收到连接请求时须验证认证、授权和加密密钥长度，不足时拒绝连接并返回对应错误码。

图10-1：LE 信用流控完整连接与数据传输流程

10.2 增强信用流控模式

增强版本的主要改进：支持同时建立最多 5 个信道（通过 L2CAP_CREDIT_BASED_CONNECTION_REQ）， 同时支持 BR/EDR 和 LE。ACL 逻辑传输须配置无限自动刷新超时。
