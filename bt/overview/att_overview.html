Part F · 属性协议（ATT）

本文档完整解析蓝牙核心规范 6.0 Part F —— 属性协议（Attribute Protocol, ATT）。ATT 定义了一种协议，使设备能够发现、读取和写入对端设备上的属性。本文严格遵循原始规范章节顺序，并配以图表辅助理解。

1. 简介（Introduction）

1.1 范围（Scope）

属性协议（ATT）允许一个被称为服务端（Server）的设备向被称为客户端（Client）的对端设备公开一组属性及其关联值。服务端公开的属性可由客户端进行发现、读取和写入，也可由服务端指示（Indication）和通知（Notification）。

🔷 图1-1 · ATT 客户端 / 服务端角色模型

1.3 约定（Conventions）

在本规范中，PDU 名称以斜体显示。错误码在 Table 3.4 中定义，以斜体加数字形式表示（例如 Attribute Not Found (0x0A)）。其他名称（如参数）以正常字体显示。

属性协议定义了服务端和客户端两种角色。每个属性由以下三个核心属性唯一标识：

ATT 协议提供通知（Notification）和指示（Indication）能力，无需客户端主动读取即可高效推送属性值。

3. 协议要求（Protocol Requirements）

3.1 基本介绍

每个属性都有一个属性类型（UUID），用于标识其含义；一个属性句柄，用于在服务端上访问该属性；以及一个属性值。属性句柄由客户端使用 ATT PDU 发现。同一服务端上可以存在多个具有相同属性类型的属性。

3.2 基本概念（Basic Concepts）

3.2.1 属性类型（Attribute Type）

每种属性类型都通过 UUID（通用唯一标识符）标识。UUID 在全时空中被视为唯一的，可由任何人独立创建。蓝牙 SIG 定义的属性类型在 Assigned Numbers 中给出，非 SIG 类型也可自定义。

🔷 图3-1 · UUID 位宽格式

3.2.2 属性句柄（Attribute Handle）

属性句柄是一个 16 位值，由每个服务端分配给其属性，供客户端在读/写请求中引用。

句柄值含义
0x0000保留，不可使用
0x0001–0xFFFE有效属性句柄范围（唯一、非零）
0xFFFF最大属性句柄（Maximum Attribute Handle）
注意：属性可在 ATT Bearer 建立期间被添加或移除；但已移除的属性，在任何 ATT Bearer 存在期间，不能用相同句柄替换。
🔷 图3-2 · 属性句柄地址空间示意

3.2.3 属性句柄分组（Attribute Handle Grouping）

分组由放在一组属性起始位置的特定属性定义（由上层规范决定）。客户端可以请求获取某个属性组的第一个和最后一个句柄，从而高效地批量发现服务。

🔷 图3-3 · 属性句柄分组示例

3.2.4 属性值（Attribute Value）

属性值是一个八位字节数组，可以是固定长度或可变长度。属性值对协议本身是不透明的——其编码由属性类型定义。

规则说明
最大长度属性值最长为 512 字节
长度隐含大多数 PDU 中不传输属性值长度，由数据包长度隐含
单一可变长每个 PDU 中只能有一个可变长度属性值字段
多值读取多属性读取响应中包含多个值，各属性类型必须为固定大小

3.2.5 属性权限（Attribute Permissions）

每个属性都有一组权限，决定是否可读/写，以及所需的安全级别。权限由上层规范定义，不可通过属性协议直接发现。

🔓 访问权限
  • 可读（Readable）
  • 可写（Writable）
  • 可读可写（Read & Write）
🔒 加密权限
  • 需要加密（Encryption Required）
  • 无需加密（No Encryption）
🔑 认证权限
  • 需要认证（Authentication Required）
  • 无需认证（No Authentication）
✅ 授权权限
  • 需要授权（Authorization Required）
  • 无需授权（No Authorization）
🔷 图3-4 · 权限检查错误响应流程

3.2.6 控制点属性（Control-Point Attributes）

无法读取、只能被写入、通知或指示的属性称为控制点属性。上层协议利用它们启用特定设备过程，例如写入命令或在某一设备过程完成时发送指示。

3.2.7 协议方法（Protocol Methods）

ATT 使用 Section 3.4 定义的方法来发现、读取、写入、通知和指示属性。每种方法属于以下类别之一：

类别后缀方向说明
CommandCMDClient → Server不触发响应的命令
RequestREQClient → Server触发响应的请求
ResponseRSPServer → Client对请求的应答
NotificationNTFServer → Client主动推送，无需确认
IndicationINDServer → Client主动推送，需要确认
ConfirmationCFMClient → Server对指示的确认

3.2.8 交换 MTU 大小（Exchanging MTU Size）

ATT_MTU 是客户端与服务端之间任何数据包的最大尺寸。使用固定 CID 的 L2CAP 信道时，双方可选择性地通过 ATT_EXCHANGE_MTU_REQ / ATT_EXCHANGE_MTU_RSP 交换各自接收能力，并使用两者中的最小值。ATT_MTU 是每个 ATT Bearer 独立的值。

🔷 图3-5 · MTU 交换过程

使用动态分配 CID 的 L2CAP 信道时，ATT_MTU 等于 L2CAP MTU 大小，无需额外交换。

3.2.9 长属性值（Long Attribute Values）

单个数据包最多可携带 (ATT_MTU - 1) 字节的属性值。超出此限制的属性称为长属性。

操作PDU长度限制
读取（首段）ATT_READ_REQ最多 ATT_MTU-1 字节
读取（完整长属性）ATT_READ_BLOB_REQ多次调用，每次偏移不同
写入（首段）ATT_WRITE_CMD最多 ATT_MTU-3 字节
写入（完整长属性）ATT_PREPARE_WRITE_REQ + ATT_EXECUTE_WRITE_REQ分段写入，最大 512 字节
注意：属性值最大长度为 512 字节。使用多个 PDU 读取属性值时，防止值在读取过程中发生变化是上层规范的责任。

3.2.10 原子操作（Atomic Operations）

服务端须将每个请求或命令视为原子操作，不受同时来自另一个 ATT Bearer 的请求或命令影响。长属性的读写不能在单次原子操作中完成。若 ATT Bearer 因任何原因终止，已修改属性值的处理由上层规范负责。

3.2.11 ATT Bearer

ATT Bearer 是用于发送 ATT PDU 的信道，使用 L2CAP 信道实现。一台设备可以与对端设备建立多个 ATT Bearer。

类型LE 支持BR/EDR 支持说明
固定信道 Bearer✅ 1个（ACL建立即可用）❌Unenhanced ATT Bearer
动态信道 Bearer✅ 多个✅ 一个或多个通过 L2CAP 建立
Enhanced ATT Bearer✅（ECBFC 模式）✅（增强重传模式）增强型，行为与标准 Bearer 相同（除非特别说明）
🔷 图3-6 · ATT Bearer 架构图

3.3 属性 PDU（Attribute PDU）

ATT PDU 分为六种类型，由 PDU 名称后缀表示。所有 ATT PDU 都至少包含一个字节的操作码（Opcode）。

类型后缀方向是否需要响应说明
Command_CMDC→S❌不触发响应
Request_REQC→S✅ RSP触发响应或错误
Response_RSPS→C—对请求的应答
Notification_NTFS→C❌主动推送无需确认
Indication_INDS→C✅ CFM主动推送需要确认
Confirmation_CFMC→S—对指示的确认

4. 安全考量（Security Considerations）

ATT 协议可用于访问可能需要授权和已认证加密物理链路的信息。

🔷 图4-1 · 服务端安全检查顺序（必须严格遵守）
重要：属性列表（ATT_FIND_INFORMATION_REQ）不被视为私密信息，因此始终允许执行，服务端不得对此请求返回 Insufficient Authorization (0x08) 或 Insufficient Authentication (0x05) 错误。

安全注意事项摘要：

场景服务端行为错误码
客户端未被授权访问属性拒绝访问0x08 Insufficient Authorization
物理链路未认证拒绝访问0x05 Insufficient Authentication
链路未加密但属性要求加密拒绝访问0x0F Insufficient Encryption
加密密钥长度不足拒绝访问0x0C Encryption Key Size Too Short
多个 Bearer，不同安全等级各 Bearer 独立检查服务端不可假设跨 Bearer 认证共享

一个属性可能允许任何设备读取，但仅允许已认证设备写入。实现者不应假设"可读"意味着"可写"，反之亦然。
