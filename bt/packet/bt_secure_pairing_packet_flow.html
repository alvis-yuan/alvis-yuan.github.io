BR/EDR 安全配对过程

基于 honor50_sunmi_bt_ota.csv：HONOR 50 Pro（Central）与 CloudPrint_0159（Peripheral）在第二次 ACL 建链后的 SSP / Secure Connections 配对与链路加密。

能力前提 IO Capability 公钥交换 Simple Pairing Confirm 随机数揭示 DH Key Check 链路认证 加密协商 密钥关系

抓包时间线总览

配对发生在 LMP Setup Complete 之后、SDP 查询之前。从 IO Capability 到 Start Encryption 约 2.9 秒（28.524s → 31.427s）。

28.391 Extended Features (SSP/SC) 28.524 IO Capability 28.53–29.05 LMP Public Key ×2 (P-256) Simple Pairing Confirm ×2 29.050 Simple Pairing Number ×2 31.374 DH Key Check ×2 31.403 Secure Auth 31.413–31.427 加密启动

0 能力前提：SSP / Secure Connections 28.391s

配对前双方已通过 LMP Extended Features 声明支持 SSP Host 与 SC Host（Secure Connections），控制器侧还有 SC Controller。

时间报文含义
28.391412LMP Extended Features Exchange (SSP Host, LE Host, SC Host …)主机栈支持 Secure Simple Pairing 与 BR/EDR Secure Connections
28.395162… (SC Controller, Ping …)控制器支持 SC 配对算法与 Ping
因此后续走 ECDH + DHKey Check 的 SC 路径，而不是老式 PIN + 仅 4 位配对码的 Legacy 流程。

1 IO Capability：选择配对关联模型 28.523913s

LMP IO Capability Transaction 交换双方的 IO 能力、OOB、MITM 需求与 Bonding 类型，用于确定 SSP 关联模型（Just Works / Numeric Comparison / Passkey Entry / OOB）。

2 公钥交换：LMP Public Key 28.524s 之后 · IO 与随机数之间

IO Capability 确定关联模型后，双方各发送一条 LMP Public Key。本抓包为 BR/EDR Secure Connections，使用 P-256 曲线，公钥点为 X‖Y 共 64 字节（各 32 字节大端）。私钥不出空口。

顺序LMP 信令方向载荷作用
1LMP_public_keyHONOR → CloudPrintP-256 公钥 64 字节Central 公布己方公钥点
2LMP_public_keyCloudPrint → HONORP-256 公钥 64 字节Peripheral 公布己方公钥点
/* LMP Public Key — BR/EDR Secure Connections，曲线 P-256 */ 7C ← LMP opcode：public_key（Wireshark: “LMP Public Key”） X[0..31] ← 点 X，32 字节 Y[0..31] ← 点 Y，32 字节 公钥载荷 64 字节 + LMP 头；HONOR / CloudPrint 各一条

收到对端公钥后本地计算：DHKey = ECDH(本地私钥, 对端 P-256 公钥)。
下一步先交换 Commitment（不暴露随机数），再 揭示 Na/Nb。

与 CSV：公钥两条信令在原始 pcap 的 28.53s～29.05s 段；honor50_sunmi_bt_ota.csv 未单独导出时可按 “Public Key” / opcode 0x7C 过滤。曲线以 SC 协商为准，本连接为 P-256（非 P-192）。

3 承诺值：Simple Pairing Confirm 公钥之后 · Number 之前

公钥交换后进入 SSP 认证阶段 1：双方各发一条 LMP Simple Pairing Confirm（Wireshark 亦可能写作 Confirmation / Commitment）， 载荷为 16 字节承诺值 Ca / Cb。此时尚未发送随机数 Na/Nb，防止中途篡改公钥或 IO 参数。

顺序LMP 信令方向载荷含义
1LMP_simple_pairing_confirmHONOR → CloudPrint16 字节Ca = f(Na, Pub, IOcap, …) 的承诺，不暴露 Na
2LMP_simple_pairing_confirmCloudPrint → HONOR16 字节Cb = f(Nb, Pub, IOcap, …) 的承诺
/* LMP Simple Pairing Confirm — 典型结构 */ 0B ← LMP opcode：simple_pairing_confirm（部分版本显示为 Confirmation） C[0..15] ← 16 字节 Commitment（Ca 或 Cb） 计算绑定：本地随机数 + 双方公钥 + IO Capability + BD_ADDR 等 对端在收到 Simple Pairing Number 后用同样公式校验 C 是否匹配
阶段顺序（规范）：Public Key ×2 → Confirm ×2 → Number ×2（揭示 Na/Nb）→ 本地校验承诺 → DH Key Check。

4 随机数揭示：Simple Pairing Number 29.050–29.053s

认证阶段 2：双方发送 LMP Simple Pairing Number，各 16 字节，即揭示 Na / Nb。 对端用收到的随机数重算 Commitment，必须与先前 Confirm 一致，否则配对失败。

时间方向（推断）报文内容（16 字节）结果

29.053297对向 Simple Pairing Number87 8A 64 4B 77 20 C5 B5 1E 1B 6D B8 11 44 28 FFAccepted
/* 报文 1 — honor50_sunmi_bt_ota.csv 第 33 行 */ B3 60 03 B8 AB B3 CC E7 6C BF E6 0F 13 03 6E A8 揭示 Na（HONOR 侧随机数，16 字节） /* 报文 2 — 第 34 行 */ 87 8A 64 4B 77 20 C5 B5 1E 1B 6D B8 11 44 28 FF 揭示 Nb（CloudPrint 侧随机数）

收到对端 Number 后：用 Na/Nb + P-256 公钥 + IOcap 等重算 Commitment，与步骤 3 的 Confirm 比对。 通过后结合 DHKey 进入 DH Key Check；DHKey 仍不在空口明文传输。

5 DH Key Check：验证共享密钥一致 31.374–31.400s

LMP DH Key Check 发送的是基于 DHKey 与配对上下文计算的 校验值 Ea / Eb（MAC 思路），对端本地重算后比对，不是“加密一段数据让对方解密”。

时间校验值（16 字节）结果
31.37395216 29 E3 04 79 83 FE 30 62 C3 31 8A 1C DD E2 75Accepted
31.399579E8 1D EE 69 EC 7D 9F 50 AD BB 30 0B 16 8D E8 76Accepted
CheckValue = f(DHKey, Na, Nb, BD_ADDR_A, BD_ADDR_B, IOcap, AuthReq, …) HONOR → CloudPrint: 16 29 E3 04 79 83 FE 30 62 C3 31 8A 1C DD E2 75 CloudPrint → HONOR: E8 1D EE 69 EC 7D 9F 50 AD BB 30 0B 16 8D E8 76 双向 Accepted ⇒ 双方 DHKey 一致，SSP 认证阶段成功
通过后从 DHKey 派生 link key（BR/EDR 长期配对密钥，供认证与加密密钥派生使用）。

6 Secure Authentication：用 Link Key 认证当前链路 31.402702s

DH Key Check 解决的是“配对是否成功”；LMP Secure Authentication Transaction 解决的是“双方现在是否持有同一个 link key”，为开启加密做准备。

时间报文抓包字段
31.402702LMP Secure Authentication Transaction0xCBF367D8（认证响应/挑战相关值，非 link key 本身）
/* 一方发随机挑战，另一方用 link key 计算响应 */ Response = g(link_key, AU_RAND, BD_ADDR, …) → 抓包可见 0xCBF367D8 对端用本地 link key 计算期望值并比较，一致则认证通过

7 加密协商与启动 31.413–31.427s

认证通过后，协商加密能力、密钥长度，再发 Start Encryption Request 用 link key 派生会话加密密钥并进入加密态。

时间LMP 报文抓包内容解读
31.412702Encryption Key Size Mask0xFFC0 — 双方支持的密钥长度位图（1～16 字节各 bit）
31.416452Encryption ModeEncryption · Accepted — 同意对本 ACL 启用加密
31.421453Encryption Key Size16 bytes · Accepted — 选用最大常规强度（128 bit 有效密钥材料）
31.426453Start Encryption Request见下方 Data 列
/* Start Encryption Request — CSV Data 列 第 44 行 */ 22 17 FB A1 CB A3 6E 74 1B E0 13 22 44 82 16 E2 E7 22 → LMP opcode（enc_req / Start Encryption 相关） 后 16 字节 → EN_RAND，用于与 link key 派生本次链路的 encryption key (Kc) 双方按协商的 16-byte key size 截断/使用，空口切换为 E0（或 SC 路径下的相应算法）加密
31.435829 起出现 L2CAP Configure 与 SDP 查询，说明业务已在加密链路上进行。31.422079 的 LMP Name Transaction ("HONOR 50 Pro") 发生在加密启动前后，属配对后信息交换。

安全结论（本抓包）：链路已加密（16 字节密钥长度），但 IO 组合为 DisplayYesNo + NoInputNoOutput，实际为 Just Works，无法满足 HONOR 侧声明的 MITM Required 的用户确认语义。高安全场景需在应用层或 OOB 补足设备身份绑定。
