引言与背景

Linux 5.4 蓝牙 HCI UART 驱动深度解析

基于 Linux kernel v5.4 官方源码，深入分析蓝牙 Host Controller Interface (HCI) 在 UART 物理层上的完整实现，涵盖设计原理、分层架构到代码落地。

kernel v5.4 drivers/bluetooth/ hci_ldisc.c hci_h4.c / hci_h5.c hci_bcsp.c
📄 hci_ldisc.c (v5.4) 📄 hci_uart.h (v5.4) 📄 hci_h4.c (v5.4) 📄 hci_h5.c (v5.4) 📄 hci_bcsp.c (v5.4)

为什么需要 HCI UART 驱动？

蓝牙规范（Bluetooth Core Specification）定义了 HCI（Host Controller Interface） 作为主机软件与蓝牙控制器硬件之间的标准接口。当蓝牙控制器通过 UART（通用异步收发器） 连接时，需要一套专门的驱动将 Linux 蓝牙协议栈（BlueZ）与串行物理层对接。

这在嵌入式系统（ARM SoC、IoT 设备）中极为普遍——蓝牙芯片（BCM、QCA、Intel、MTK 等）往往通过 UART 接口与主机 SoC 相连。Linux 5.4 中的 HCI UART 驱动版本为 2.3，由 Qualcomm、Maxim Krasnyansky 和 Marcel Holtmann 等人持续演进。

📌 源码版本说明： 本文档严格基于 Linux kernel v5.4 tag 的官方源码（drivers/bluetooth/ 目录），所有代码引用均可在 GitHub torvalds/linux @ v5.4 得到验证。
⚡ 第一性原理概述

蓝牙控制器是独立的硬件实体，它只"说" HCI 协议。UART 只是一根串行管道。问题的本质是：如何把一个面向消息的协议（HCI packet）安全、高效地塞进一根连续字节流的管道（UART），再从另一端完整取出？ HCI UART 驱动存在的全部理由，就是解决这一个问题。

01

整体分层架构

Linux 蓝牙 HCI UART 子系统采用严格的分层设计，从用户空间到物理线路，共跨越五个层次：

各层职责划分

层次文件职责关键接口

⚡ 第一性原理

分层的本质是关注点隔离（Separation of Concerns）。每一层只解决一类问题：TTY 层解决字节的物理搬运；线路规程层解决协议切换的可配置性；协议插件层解决不同芯片厂商的帧格式差异；HCI接口层解决与上层协议栈的标准对接。层与层之间通过明确定义的接口（函数指针表）通信，修改任一层不影响其它层。

02

设计原理与模式

核心设计模式

🔌 策略模式（Strategy Pattern）

hci_uart_proto 结构体定义了协议操作的抽象接口（函数指针表）。H4、H5、BCSP 等各自实现这些接口，运行时通过 IOCTL HCIUARTSETPROTO 动态切换。上层代码无需感知具体协议。

🏭 注册-发现模式（Registry）

全局数组 hup[HCI_UART_MAX_PROTO] 作为协议注册表。各协议模块在 module_init 时调用 hci_uart_register_proto() 注册自身，系统可动态加卸载。

🔀 线路规程（Line Discipline）

TTY 线路规程是 Linux 的标准扩展机制。N_HCI=15 这个 ldisc ID 让内核知道：当此 TTY 设备被设置为蓝牙模式时，所有收发数据都由 HCI 驱动接管，而无需修改任何底层串口驱动。

🔒 生产者-消费者模式

发送路径：HCI 核心层（生产者）将 sk_buff 通过 proto->enqueue() 放入协议队列，工作队列（消费者）hci_uart_write_work 异步从队列取出并写入 TTY。

关键设计决策分析

图2：HCI UART 驱动六大核心设计决策

HCI 数据包类型

HCI 协议定义了四种基本数据包类型，所有 UART 协议都必须支持其区分和路由：

包类型标识字节头部大小长度字段用途
HCI_COMMAND_PKT0x013 字节1 字节（offset 2）主机→控制器命令
HCI_ACLDATA_PKT0x024 字节2 字节（offset 2）ACL 数据包（L2CAP等）
HCI_SCODATA_PKT0x033 字节1 字节（offset 2）SCO 语音数据
HCI_EVENT_PKT0x042 字节1 字节（offset 1）控制器→主机事件
⚡ 第一性原理

所有设计决策都源于同一个约束：UART 是串行、无帧边界的字节流。因此必须解决：①如何区分包类型（包类型字节）；②如何知道包的边界（头部长度字段）；③如何处理部分接收（状态机+sk_buff）；④如何不阻塞调用方（工作队列+异步发送）；⑤如何安全地关闭（proto_lock 保护）。每一个设计选择都直接对应着这些约束。

03

TTY 线路规程（Line Discipline）

TTY 线路规程（ldisc）是 Linux 内核提供的 TTY 子系统扩展机制。每个 TTY 设备在同一时刻只能关联一个线路规程。HCI UART 驱动将自身注册为 N_HCI = 15 号线路规程，名为 "n_hci"。

线路规程注册（module_init）

// hci_ldisc.c - hci_uart_init()
static int __init hci_uart_init(void)
{
    static struct tty_ldisc_ops hci_uart_ldisc;
    int err;

    BT_INFO("HCI UART driver ver %s", VERSION);  /* VERSION = "2.3" */

    memset(&hci_uart_ldisc, 0, sizeof(hci_uart_ldisc));
    hci_uart_ldisc.magic        = TTY_LDISC_MAGIC;
    hci_uart_ldisc.name         = "n_hci";
    hci_uart_ldisc.open         = hci_uart_tty_open;
    hci_uart_ldisc.close        = hci_uart_tty_close;
    hci_uart_ldisc.read         = hci_uart_tty_read;    /* 返回0，不提供用户态读 */
    hci_uart_ldisc.write        = hci_uart_tty_write;   /* 返回0，不提供用户态写 */
    hci_uart_ldisc.ioctl        = hci_uart_tty_ioctl;
    hci_uart_ldisc.compat_ioctl = hci_uart_tty_ioctl;
    hci_uart_ldisc.poll         = hci_uart_tty_poll;    /* 返回0 */
    hci_uart_ldisc.receive_buf  = hci_uart_tty_receive; /* 核心：接收数据 */
    hci_uart_ldisc.write_wakeup = hci_uart_tty_wakeup;  /* 核心：可以继续写 */
    hci_uart_ldisc.owner        = THIS_MODULE;

    err = tty_register_ldisc(N_HCI, &hci_uart_ldisc);
    if (err) {
        BT_ERR("HCI line discipline registration failed. (%d)", err);
        return err;
    }
    /* 注册各协议子模块 */
#ifdef CONFIG_BT_HCIUART_H4
    h4_init();
#endif
#ifdef CONFIG_BT_HCIUART_3WIRE
    h5_init();
#endif
    /* ... 其他协议 ... */
    return 0;
}
MODULE_ALIAS_LDISC(N_HCI);

线路规程的打开（tty_open）

当用户态程序（如 btattach）通过 ioctl(fd, TIOCSETD, &ldisc) 将 TTY 设备切换为 N_HCI 线路规程时，内核调用 hci_uart_tty_open()：

static int hci_uart_tty_open(struct tty_struct *tty)
{
    struct hci_uart *hu;

    /* 若 TTY 没有 write 操作，拒绝打开（安全校验） */
    if (tty->ops->write == NULL)
        return -EOPNOTSUPP;

    hu = kzalloc(sizeof(struct hci_uart), GFP_KERNEL);
    if (!hu) {
        BT_ERR("Can't allocate control structure");
        return -ENFILE;
    }

    tty->disc_data = hu;    /* 将 hci_uart 挂在 tty 的私有数据 */
    hu->tty = tty;
    tty->receive_room = 65536;  /* 告知 TTY 层接收缓冲大小 */

    hu->alignment = 1;   /* 默认无对齐要求 */
    hu->padding   = 0;

    INIT_WORK(&hu->init_ready, hci_uart_init_work);   /* 异步注册 hci_dev */
    INIT_WORK(&hu->write_work, hci_uart_write_work);  /* 异步发送工作 */
    percpu_init_rwsem(&hu->proto_lock);

    tty_driver_flush_buffer(tty);  /* 清除残留字节 */
    return 0;
}

线路规程激活流程

图3：TTY 线路规程激活流程（用户态到内核态）
注意： 线路规程的 read()、write()、poll() 回调均返回 0 或空操作。这是刻意设计的——HCI UART 不提供用户态直接读写接口，所有数据流都通过蓝牙协议栈（HCI 核心层）进行，避免绕过协议栈的不安全直接访问。
⚡ 第一性原理

线路规程的本质是在不改变硬件驱动的前提下改变数据的处理方式。同一个 UART 硬件，切换 ldisc 前是终端（n_tty），切换后变成蓝牙控制器接口。这体现了 Unix "一切皆文件" 的哲学——通过配置而非代码改变行为。

核心数据结构

hci_uart_proto — 协议操作表

这是协议插件系统的核心接口，类似于面向对象语言中的"接口（Interface）"或"虚函数表（vtable）"：

/* hci_uart.h - v5.4 */
struct hci_uart_proto {
    unsigned int  id;            /* 协议 ID，如 HCI_UART_H4=0 */
    const char   *name;          /* 协议名称字符串，如 "H4" */
    unsigned int  manufacturer;  /* 制造商 ID（用于 VND_DETECT） */
    unsigned int  init_speed;    /* 初始化波特率（0=使用默认） */
    unsigned int  oper_speed;    /* 工作波特率 */

    /* --- 生命周期回调 --- */
    int  (*open)(struct hci_uart *hu);   /* 分配协议私有数据 */
    int  (*close)(struct hci_uart *hu);  /* 释放协议私有数据 */
    int  (*flush)(struct hci_uart *hu);  /* 清空发送队列 */
    int  (*setup)(struct hci_uart *hu);  /* 硬件初始化（固件下载等） */

    /* --- 波特率协商 --- */
    int  (*set_baudrate)(struct hci_uart *hu, unsigned int speed);

    /* --- 数据收发 --- */
    int  (*recv)(struct hci_uart *hu, const void *data, int len);
    int  (*enqueue)(struct hci_uart *hu, struct sk_buff *skb);
    struct sk_buff *(*dequeue)(struct hci_uart *hu);
};

hci_uart — 连接上下文

每个已激活 N_HCI 线路规程的 TTY 设备，都有一个对应的 hci_uart 实例（存于 tty->disc_data）：

struct hci_uart {
    /* --- 传输层绑定 --- */
    struct tty_struct         *tty;      /* 关联的 TTY（传统模式）*/
    struct serdev_device      *serdev;   /* 关联的 serdev（新模式）*/

    /* --- HCI 设备 --- */
    struct hci_dev            *hdev;     /* 上层 HCI 设备抽象 */
    unsigned long              flags;    /* HCI_UART_PROTO_SET 等 */
    unsigned long              hdev_flags; /* HCI_UART_RAW_DEVICE 等 */

    /* --- 异步工作 --- */
    struct work_struct         init_ready;  /* 注册 hci_dev 的工作 */
    struct work_struct         write_work;  /* 异步发送工作 */

    /* --- 协议管理 --- */
    const struct hci_uart_proto *proto;      /* 当前绑定的协议 */
    struct percpu_rw_semaphore  proto_lock;  /* 保护协议操作的 percpu 锁 */
    void                       *priv;       /* 协议私有数据指针 */

    /* --- 发送状态 --- */
    struct sk_buff             *tx_skb;    /* 当前未发送完的 skb */
    unsigned long               tx_state;  /* HCI_UART_SENDING 等 */

    /* --- 速率配置 --- */
    unsigned int  init_speed;   /* 初始波特率 */
    unsigned int  oper_speed;   /* 工作波特率 */

    /* --- 对齐支持（某些芯片要求字节对齐） --- */
    u8  alignment;  /* 对齐字节数（0或1=不对齐）*/
    u8  padding;    /* 当前包的填充字节计数 */
};

⚡ 第一性原理

数据结构的设计反映了系统的信息模型。hci_uart 把"物理传输"（tty/serdev）、"协议逻辑"（proto）和"HCI抽象"（hdev）三者捏合在一起——这三样东西的结合，正好是"在串行线上跑HCI"这件事的完整定义。h4_recv_pkt 则是将"如何解析一种包"这个过程用数据而非代码表达，让同一段解析逻辑可以服务所有包类型。

协议插件系统

协议注册表

内核中维护一个全局静态数组，以协议 ID 为索引：

/* hci_ldisc.c */
static const struct hci_uart_proto *hup[HCI_UART_MAX_PROTO];  /* [12] */

int hci_uart_register_proto(const struct hci_uart_proto *p)
{
    if (p->id >= HCI_UART_MAX_PROTO)  return -EINVAL;
    if (hup[p->id])                     return -EEXIST;  /* 防重复注册 */
    hup[p->id] = p;
    BT_INFO("HCI UART protocol %s registered", p->name);
    return 0;
}

int hci_uart_unregister_proto(const struct hci_uart_proto *p)
{
    if (p->id >= HCI_UART_MAX_PROTO)  return -EINVAL;
    if (!hup[p->id])                    return -EINVAL;
    hup[p->id] = NULL;
    return 0;
}

⚡ 第一性原理

协议插件系统的存在是因为：蓝牙芯片厂商众多，各有不同的串行协议方言，但 HCI 命令/事件的语义是统一的。插件系统把"如何搬运字节"和"字节代表什么意思"彻底分离。只要符合 hci_uart_proto 接口，任何新厂商都可以不改动通用代码而接入系统。

发送路径（TX Path）

发送路径从 HCI 核心层向下直到 TTY 写入，核心是异步、非阻塞的工作队列机制。

// 部分写入时保存剩余 if skb->len: hu->tx_skb=skb; break if TX_WAKEUP: goto restart // 发送中有新数据 hci_uart_tx_wakeup() 详细逻辑 // 可在 IRQ 上下文调用，使用 trylock if (!percpu_down_read_trylock(&hu->proto_lock)) return 0; // tty 正在关闭，忽略 if (!test_bit(HCI_UART_PROTO_READY, &hu->flags)) goto no_schedule; set_bit(HCI_UART_TX_WAKEUP, &hu->tx_state) if (test_and_set_bit(HCI_UART_SENDING, ...)) goto no_schedule; // 已在发送，TX_WAKEUP 已设 schedule_work(&hu->write_work); // 调度工作队列

hci_uart_send_frame — 入口函数

/* HCI 核心层调用此函数发送一帧 */
static int hci_uart_send_frame(struct hci_dev *hdev, struct sk_buff *skb)
{
    struct hci_uart *hu = hci_get_drvdata(hdev);

    BT_DBG("%s: type %d len %d", hdev->name,
           hci_skb_pkt_type(skb), skb->len);

    percpu_down_read(&hu->proto_lock);

    if (!test_bit(HCI_UART_PROTO_READY, &hu->flags)) {
        percpu_up_read(&hu->proto_lock);
        return -EUNATCH;  /* 协议未就绪 */
    }

    hu->proto->enqueue(hu, skb);  /* 交给协议层处理并入队 */
    percpu_up_read(&hu->proto_lock);

    hci_uart_tx_wakeup(hu);  /* 触发异步发送 */
    return 0;
}

hci_uart_tty_receive — 接收入口

static void hci_uart_tty_receive(struct tty_struct *tty,
                                   const u8 *data, char *flags, int count)
{
    struct hci_uart *hu = tty->disc_data;

    if (!hu || tty != hu->tty) return;

    percpu_down_read(&hu->proto_lock);

    if (!test_bit(HCI_UART_PROTO_READY, &hu->flags)) {
        percpu_up_read(&hu->proto_lock);
        return;
    }

    /* TTY 互斥保护，无需额外锁 */
    hu->proto->recv(hu, data, count);

    percpu_up_read(&hu->proto_lock);

    if (hu->hdev)
        hu->hdev->stat.byte_rx += count;  /* 统计字节数 */

    tty_unthrottle(tty);  /* 通知 TTY 可以继续接收 */
}

h4_recv_buf — 通用帧重组引擎

这是 H4 接收缓冲函数，也被 BCM、QCA、Intel 等厂商协议复用，是核心接收状态机：

struct sk_buff *h4_recv_buf(struct hci_dev *hdev, struct sk_buff *skb,
                             const unsigned char *buffer, int count,
                             const struct h4_recv_pkt *pkts, int pkts_count)
{
    struct hci_uart *hu = hci_get_drvdata(hdev);
    u8 alignment = hu->alignment ? hu->alignment : 1;

    if (IS_ERR(skb)) skb = NULL;  /* 清除上次错误 */

    while (count) {
        int i, len;

        /* 跳过对齐填充字节 */
        for (; hu->padding && count > 0; hu->padding--) { count--; buffer++; }

        if (!skb) {
            /* 状态1: 等待包类型字节 */
            for (i = 0; i < pkts_count; i++) {
                if (*buffer != (&pkts[i])->type) continue;
                skb = bt_skb_alloc((&pkts[i])->maxlen, GFP_ATOMIC);
                if (!skb) return ERR_PTR(-ENOMEM);
                hci_skb_pkt_type(skb) = (&pkts[i])->type;
                hci_skb_expect(skb)   = (&pkts[i])->hlen;  /* 期待头部 */
                break;
            }
            if (!skb) return ERR_PTR(-EILSEQ);  /* 未知包类型 */
            count--; buffer++;  /* 消耗类型字节 */
        }

        /* 状态2/3: 追加数据直到期待量 */
        len = min_t(uint, hci_skb_expect(skb) - skb->len, count);
        skb_put_data(skb, buffer, len);
        count -= len; buffer += len;

        if (skb->len < hci_skb_expect(skb)) continue;  /* 包不完整 */

        /* 头部接收完毕，解析数据长度 */
        if (skb->len == (&pkts[i])->hlen) {
            u16 dlen;
            switch ((&pkts[i])->lsize) {
            case 0: dlen = 0; break;                          /* 无数据体 */
            case 1: dlen = skb->data[(&pkts[i])->loff]; break;  /* 单字节长度 */
            case 2: dlen = get_unaligned_le16(                  /* 双字节长度 */
                         skb->data + (&pkts[i])->loff); break;
            }
            hci_skb_expect(skb) += dlen;  /* 更新期待总量 */
            if (!dlen) { (&pkts[i])->recv(hdev, skb); skb = NULL; }
        } else {
            /* 完整包，计算对齐填充，上报 */
            hu->padding = (skb->len - 1) % alignment;
            hu->padding = (alignment - hu->padding) % alignment;
            (&pkts[i])->recv(hdev, skb);
            skb = NULL;
        }
    }
    return skb;  /* 返回未完成的 skb（跨调用保持）*/
}
EXPORT_SYMBOL_GPL(h4_recv_buf);

接收路径的本质问题是：UART 传来的是字节流，但 HCI 需要的是完整的帧。h4_recv_buf 用保持状态的 sk_buff 指针跨越多次 receive_buf 调用，像"拼图"一样逐步拼出完整的帧。这个设计精妙之处在于：它不需要额外的状态变量，skb 本身（NULL 或非NULL）就是状态，hci_skb_expect 就是期待量——状态与数据合一。

static int h4_recv(struct hci_uart *hu, const void *data, int count)
{
    struct h4_struct *h4 = hu->priv;

    if (!test_bit(HCI_UART_REGISTERED, &hu->flags))
        return -EUNATCH;

    h4->rx_skb = h4_recv_buf(hu->hdev, h4->rx_skb, data, count,
                               h4_recv_pkts, ARRAY_SIZE(h4_recv_pkts));

        int err = PTR_ERR(h4->rx_skb);
        bt_dev_err(hu->hdev, "Frame reassembly failed (%d)", err);
        h4->rx_skb = NULL;
        return err;
    }
    return count;
}

/* hci_h5.c - 头部字段位域解析宏 */
#define H5_HDR_SEQ(hdr)       ((hdr)[0] & 0x07)         /* 发送序列号 [2:0] */
#define H5_HDR_ACK(hdr)       (((hdr)[0] >> 3) & 0x07)  /* 确认序列号 [5:3] */
#define H5_HDR_CRC(hdr)       (((hdr)[0] >> 6) & 0x01)  /* CRC 存在标志 */
#define H5_HDR_RELIABLE(hdr)  (((hdr)[0] >> 7) & 0x01)  /* 可靠包标志 */
#define H5_HDR_PKT_TYPE(hdr)  ((hdr)[1] & 0x0f)          /* HCI 包类型 */
#define H5_HDR_LEN(hdr)       ((((hdr)[1] >> 4) & 0xff) + ((hdr)[2] << 4))
#define H5_HDR_CHECKSUM(hdr)  ((hdr)[3])                 /* ~(hdr[0]+hdr[1]+hdr[2]) */

#define SLIP_DELIMITER  0xc0  /* SLIP 帧边界 */
#define SLIP_ESC        0xdb  /* SLIP 转义字符 */
#define SLIP_ESC_DELIM  0xdc  /* 转义后的 0xC0 */
#define SLIP_ESC_ESC    0xdd  /* 转义后的 0xDB */

H5 接收状态机（字节驱动）

/* H5 用函数指针实现状态机，每收一字节调用当前状态处理函数 */
static int h5_rx_3wire_hdr(struct hci_uart *hu, unsigned char c)
{
    struct h5 *h5 = hu->priv;
    const unsigned char *hdr = h5->rx_skb->data;

    /* 校验头部 checksum: 所有4字节之和必须为 0xFF */
    if (((hdr[0] + hdr[1] + hdr[2] + hdr[3]) & 0xff) != 0xff) {
        bt_dev_err(hu->hdev, "Invalid header checksum");
        h5_reset_rx(h5);
        return 0;
    }

    /* 验证序列号（可靠包）*/
    if (H5_HDR_RELIABLE(hdr) && H5_HDR_SEQ(hdr) != h5->tx_ack) {
        bt_dev_err(hu->hdev, "Out-of-order packet arrived (%u != %u)",
                   H5_HDR_SEQ(hdr), h5->tx_ack);
        set_bit(H5_TX_ACK_REQ, &h5->flags);
        hci_uart_tx_wakeup(hu);  /* 触发 NAK 回复 */
        h5_reset_rx(h5);
        return 0;
    }

    h5->rx_func    = h5_rx_payload;   /* 切换到载荷接收状态 */
    h5->rx_pending = H5_HDR_LEN(hdr); /* 期待载荷长度 */
    return 0;
}

static int h5_rx_crc(struct hci_uart *hu, unsigned char c)
{
    struct h5 *h5 = hu->priv;
    const unsigned char *hdr = h5->rx_skb->data;
    u16 crc;
    __be16 crc_be;

    crc    = crc_ccitt(0xffff, hdr, 4 + H5_HDR_LEN(hdr));
    crc    = bitrev16(crc);
    crc_be = cpu_to_be16(crc);

    if (memcmp(&crc_be, hdr + 4 + H5_HDR_LEN(hdr), 2) != 0) {
        bt_dev_err(hu->hdev, "Received packet with invalid CRC");
        h5_reset_rx(h5);
    } else {
        skb_trim(h5->rx_skb, 4 + H5_HDR_LEN(hdr));
        h5_complete_rx_pkt(hu);  /* CRC 通过，上报包 */
    }
    return 0;
}

BCSP 重传机制（定时器驱动）

/* 重传所有未确认包 */
static void bcsp_timed_event(struct timer_list *t)
{
    struct bcsp_struct *bcsp = timer_container_of(bcsp, t, tbcsp);
    struct hci_uart    *hu   = bcsp->hu;
    struct sk_buff     *skb;
    unsigned long       flags;

    BT_DBG("hu %p retransmitting %u pkts", hu, bcsp->unack.qlen);

    /* 将 unack 队列中所有包倒序放回 rel 队列首部（重发）*/
    spin_lock_irqsave_nested(&bcsp->unack.lock, flags, SINGLE_DEPTH_NESTING);
    while ((skb = __skb_dequeue_tail(&bcsp->unack)) != NULL) {
        bcsp->msgq_txseq = (bcsp->msgq_txseq - 1) & 0x07;
        skb_queue_head(&bcsp->rel, skb);
    }
    spin_unlock_irqrestore(&bcsp->unack.lock, flags);

    hci_uart_tx_wakeup(hu);  /* 触发重新发送 */
}

static int hci_uart_register_dev(struct hci_uart *hu)
{
    struct hci_dev *hdev;
    int err;

    hdev = hci_alloc_dev();   /* 从 HCI 核心分配设备 */
    if (!hdev) { ... return -ENOMEM; }

    hu->hdev = hdev;
    hdev->bus = HCI_UART;          /* 总线类型标识 */
    hci_set_drvdata(hdev, hu);    /* hdev → hu 的反向查找 */

        set_bit(HCI_QUIRK_RESET_ON_CLOSE, &hdev->quirks);

    /* 先 open 协议，再注册 hci_dev */
    err = hu->proto->open(hu);  /* 分配协议私有数据 */
    if (err) { ... }

    /* INIT_PENDING: 等待异步初始化（如固件下载） */
    if (test_bit(HCI_UART_INIT_PENDING, &hu->hdev_flags))
        return 0;  /* 延迟注册，由 hci_uart_init_ready() 完成 */

    err = hci_register_dev(hdev);
    if (err) { ... }

    set_bit(HCI_UART_REGISTERED, &hu->flags);

hci_uart_setup — 波特率与厂商初始化

static int hci_uart_setup(struct hci_dev *hdev)
{
    struct hci_uart *hu = hci_get_drvdata(hdev);
    struct hci_rp_read_local_version *ver;
    unsigned int speed;

    /* 1. 设置初始化波特率（优先级：hu > proto > 默认）*/
    if      (hu->init_speed)        speed = hu->init_speed;
    else if (hu->proto->init_speed) speed = hu->proto->init_speed;
    else                            speed = 0;
    if (speed) hci_uart_set_baudrate(hu, speed);

    /* 2. 协商工作波特率（厂商专用命令）*/
    if      (hu->oper_speed)        speed = hu->oper_speed;
    else if (hu->proto->oper_speed) speed = hu->proto->oper_speed;
    else                            speed = 0;
    if (hu->proto->set_baudrate && speed) {
        err = hu->proto->set_baudrate(hu, speed);  /* 发 HCI 命令给控制器 */
        if (!err) hci_uart_set_baudrate(hu, speed); /* 同步更新主机波特率 */
    }

    /* 3. 厂商初始化（固件下载等）*/
    if (hu->proto->setup)
        return hu->proto->setup(hu);

    /* 4. VND_DETECT: 通过 Read Local Version 自动检测厂商 */
    if (!test_bit(HCI_UART_VND_DETECT, &hu->hdev_flags))
        return 0;

    skb = __hci_cmd_sync(hdev, HCI_OP_READ_LOCAL_VERSION,
                         0, NULL, HCI_INIT_TIMEOUT);
    ver = (struct hci_rp_read_local_version *)skb->data;
    switch (le16_to_cpu(ver->manufacturer)) {
#ifdef CONFIG_BT_HCIUART_INTEL
    case 2:  /* Intel */
        hdev->set_bdaddr = btintel_set_bdaddr;
        btintel_check_bdaddr(hdev); break;
#endif
#ifdef CONFIG_BT_HCIUART_BCM
    case 15: /* Broadcom */
        hdev->set_bdaddr = btbcm_set_bdaddr;
        btbcm_check_bdaddr(hdev); break;
#endif
    }
    return 0;
}

percpu_rw_semaphore（proto_lock）

这是 v5.4 中最关键的并发控制机制。proto_lock 是一个 per-CPU 读写信号量，专门为以下场景设计：

场景锁操作原因
接收数据（hci_uart_tty_receive） percpu_down_read() 读锁：多个接收可并发
发送数据（hci_uart_send_frame） percpu_down_read() 读锁：发送与接收可并发
TX wakeup（可能在 IRQ 中） percpu_down_read_trylock() 非阻塞 trylock，防止 IRQ 死锁
关闭协议（hci_uart_tty_close） percpu_down_write() 写锁：排他，等所有读完成后关闭 
/* 关闭时的锁操作序列（保证安全关闭）*/
if (test_bit(HCI_UART_PROTO_READY, &hu->flags)) {
    percpu_down_write(&hu->proto_lock);      /* 取写锁：等待所有读操作完成 */
    clear_bit(HCI_UART_PROTO_READY, &hu->flags); /* 清就绪标志 */
    percpu_up_write(&hu->proto_lock);        /* 释放写锁 */
    cancel_work_sync(&hu->write_work);       /* 等 write_work 完成 */
    ...
    hu->proto->close(hu);                    /* 此后不会有新的收发操作 */
}

⚡ 第一性原理

并发设计的根本是：读多写少，读可并发，写必独占。正常工作时（发送/接收）都是读锁，可以在多 CPU 上并发执行，性能最优。关闭是稀少的一次性操作，用写锁确保安全。per-CPU 读写信号量相比普通互斥锁，在读路径上几乎无竞争开销——这正是高频收发所需要的。

硬件流量控制（RTS/CTS）

void hci_uart_set_flow_control(struct hci_uart *hu, bool enable)
{
    struct tty_struct *tty = hu->tty;
    struct ktermios ktermios;

    if (hu->serdev) {
        serdev_device_set_flow_control(hu->serdev, !enable);
        serdev_device_set_rts(hu->serdev, !enable);
        return;
    }

    if (enable) {
        /* 禁用硬件流控（波特率协商期间，避免 CTS 阻塞发送）*/
        ktermios = tty->termios;
        ktermios.c_cflag &= ~CRTSCTS;
        tty_set_termios(tty, &ktermios);
        /* 清 RTS，防止控制器发送 */
        tty->driver->ops->tiocmset(tty, 0, TIOCM_RTS | TIOCM_OUT2);
    } else {
        /* 重新启用硬件流控 */
        tty->driver->ops->tiocmset(tty, TIOCM_RTS | TIOCM_OUT2, 0);
        ktermios = tty->termios;
        ktermios.c_cflag |= CRTSCTS;
        tty_set_termios(tty, &ktermios);
    }
}

hci_ldisc.c
⚠️ 波特率切换时序： 厂商协议在切换到高速波特率时，必须先通过 hci_uart_set_flow_control(hu, true) 暂时关闭硬件流控，发送波特率切换命令，然后在控制器应答（或超时）后重新开启流控。这个时序如果出错，会导致控制器接收到乱码。
⚡ 第一性原理

波特率协商的本质是一个鸡生蛋问题：想用高速传输，但高速传输要先协商，而协商本身要用低速。所以必须两阶段：先用低速（所有设备通用的"普通话"）建立联系，再切换到双方都支持的最高速度（"方言"）。流控的暂时关闭是因为：切换中间有一段时间主机和控制器波特率不同步，这段时间必须没有数据传输。

/* 在 hci_ldisc.c 中，serdev 和 tty 路径的统一处理 */
bool hci_uart_has_flow_control(struct hci_uart *hu)
{
    /* serdev: 始终认为支持流控（由 serdev 驱动保证）*/
    if (hu->serdev)
        return true;

    /* TTY 路径: 检查底层驱动是否实现了 tiocmget/tiocmset */
    if (hu->tty->driver->ops->tiocmget &&
        hu->tty->driver->ops->tiocmset)
        return true;

    return false;
}

static inline void host_set_baudrate(struct hci_uart *hu, unsigned int speed)
{

        serdev_device_set_baudrate(hu->serdev, speed);
    else
        hci_uart_set_baudrate(hu, speed);
}

⚡ 第一性原理

serdev 的引入是消除"手工配置"中间状态的过程。传统 TTY ldisc 方式需要用户态程序（btattach）来"宣布"这个 UART 是蓝牙设备，这是因为内核无法从 UART 硬件本身判断连接了什么。设备树让硬件连接关系成为内核可知的信息，serdev 则是把这个信息转化为自动枚举的机制——回到"内核知道一切，自动做一切"的理想状态。

设计总结

全局架构回顾

图14：Linux 5.4 HCI UART 完整数据流全景图

关键设计原则汇总

① 分层解耦

5层架构，每层通过明确定义的函数指针接口通信，修改任一层不影响其它层。新协议、新芯片只需实现接口。

② 异步非阻塞

发送路径全程异步：send_frame 入队即返回，write_work 在内核工作队列中独立执行，TTY 缓冲满时自动背压。

③ 状态机驱动接收

h4_recv_buf 用 sk_buff 本身承载状态，跨越多次 receive_buf 调用逐步重组帧，无额外状态变量，零拷贝友好。

④ 精细化锁策略

percpu_rw_sem 允许读并发，只在关闭时独占。IRQ 上下文用 trylock。原子位操作保护发送状态，无需互斥锁。

⑤ 数据驱动解析

h4_recv_pkt 数组描述包类型，同一解析函数服务所有包类型。厂商协议只需扩展此数组即可添加私有包类型。

⑥ 两种接入路径

TTY ldisc（传统/灵活）和 serdev（现代/嵌入式）并存，统一的 hci_uart 结构抽象两者差异，上层代码透明。

内核版本重要变化

⚡ 终极第一性原理

整个 Linux 5.4 HCI UART 驱动系统，从最根本的角度看，是在回答一个问题： 如何让一根串行线成为蓝牙控制器与主机协议栈之间的透明管道？

答案是：①用"帧协议"（H4/H5/BCSP）给无边界的字节流加上边界；②用"线路规程"在不改变硬件的前提下改变字节的语义；③用"协议插件"隔离不同芯片厂商的差异；④用"异步工作队列"让协议栈不被硬件限速阻塞；⑤用"精细锁"让收发并发而关闭安全。每一个机制都有且仅有一个根本原因，不多也不少。
