Linux 5.4 蓝牙 SMP 实现深度解析

基于 Linux Kernel v5.4 · Bluetooth Core Specification 5.x · BlueZ 协议栈

⬡ 源码：net/bluetooth/smp.c (v5.4) ⬡ 官方规范：Bluetooth Core Spec Part H — Security Manager

文档说明

本文以 Linux 内核 5.4 版本（net/bluetooth/smp.c）为基准，结合蓝牙官方核心规范（Bluetooth Core Specification, Part H — Security Manager），从设计原理出发，逐步深入到具体的 C 语言实现，系统性讲解蓝牙低功耗（LE）安全管理协议（SMP）的全貌。

SMP 在蓝牙协议栈中的位置
图 0.1：SMP 在 Linux 蓝牙协议栈中的位置（L2CAP 固定信道 0x0006）
⚛ 第一性原理

SMP 存在的根本原因：无线信道天然是公开的广播介质，任何人都能监听。要在公开信道上建立"只有通信双方能解读"的加密通道，必须先通过某种形式的带外（OOB）或可验证的方式协商出共同的秘密。SMP 就是这个"协商过程"在蓝牙 LE 上的标准化定义——它解决的核心问题只有一个：在不可信信道上安全地生成、交换、存储加密密钥。

🏗 架构

01 · 设计原理与总体架构

⬡ Bluetooth Core Spec Part H § 2.1 Introduction

蓝牙规范将安全管理器（Security Manager，SM）定义为：一套管理 LE 设备之间配对（Pairing）、认证（Authentication）和加密（Encryption）的协议与行为规范。官方规范原文指出：

官方规范核心设计原则
  • 密钥分发模型（Key Distribution）：每台设备负责生成并控制自己分发的密钥，没有任何其他设备能影响这些密钥的生成过程。密钥的强度完全由分发设备内部实现的算法决定。
  • 非对称内存/计算需求：安全架构特意设计为响应设备（Responding Device）的内存和计算需求低于发起设备（Initiating Device）。
  • 三阶段配对：配对是一个三阶段的过程，前两个阶段始终执行，第三阶段（密钥分发）可选。

三阶段配对模型

图 1.1：SMP 三阶段配对模型（源自 Bluetooth Core Spec Part H §2.1）

SMP 提供的安全服务

安全服务描述依赖机制
设备认证验证通信对端是否为声称的设备Passkey / OOB / Numeric Comparison
数据保密性防止第三方读取通信内容AES-CCM 链路层加密
数据完整性防止数据被篡改CSRK 签名机制
隐私保护防止设备被追踪（Tracking）IRK + 可解析私有地址（RPA）
前向安全历史会话密钥不因当前密钥泄露而暴露LE Secure Connections ECDH
⚛ 第一性原理

三阶段设计的根本逻辑：阶段一解决"我们能做到什么安全等级"（能力协商），阶段二解决"如何在不可信信道上生成双方共知的秘密"（密钥建立），阶段三解决"下次如何跳过繁琐配对直接恢复安全连接"（密钥持久化）。缺少任何一个阶段，整个安全体系都无法完整运作。

🔐 密码学

02 · 密码学基础函数

⬡ Bluetooth Core Spec Part H § 2.2 Cryptographic Toolbox ⬡ net/bluetooth/smp.c — 密码函数实现

SMP 的密码学工具箱以 AES-128 为基础构建，规范定义了两套函数体系：Legacy 体系用 AES-128 直接加密，Secure Connections 体系用 AES-CMAC（RFC 4493）作为构建块。

Legacy 体系函数

安全函数 e — 底层 AES-128 加密

所有 Legacy 函数的构建基础。可在 Host 实现或通过 HCI_LE_Encrypt 命令在 Controller 执行。

/* 规范定义 */
encryptedData = e(key, plaintextData)
/* AES-128 块加密，key 和 plaintextData 均为 128 bit */

/* Linux 5.4 内核实现（smp.c）*/
static int smp_e(struct crypto_cipher *tfm, const u8 *k, u8 *r)
{
    int err;
    crypto_cipher_setkey(tfm, k, 16);
    err = crypto_cipher_encrypt_one(tfm, r, r);
    return err;
}
函数 c1 — Legacy 确认值生成

用于 LE Legacy 配对过程中生成 Confirm 值。

/* 规范定义：
   p1 = pres || preq || rat' || iat'
   p2 = padding || ia || ra
   c1(k,r,preq,pres,iat,rat,ia,ra) = e(k, e(k, r XOR p1) XOR p2) */

static int smp_c1(struct smp_chan *smp, const u8 k[16],
               const u8 r[16], const u8 pres[7], const u8 preq[7],
               u8 _iat, const bdaddr_t *ia,
               u8 _rat, const bdaddr_t *ra, u8 res[16])
函数 s1 — Legacy STK 生成
/* 规范定义：
   r1' = r1 的低 64 位，r2' = r2 的低 64 位
   r' = r1' || r2'
   s1(k, r1, r2) = e(k, r') */

static int smp_s1(struct smp_chan *smp, const u8 k[16],
               const u8 r1[16], const u8 r2[16], u8 _r[16])

随机地址哈希函数 ah

函数 ah — 可解析私有地址哈希

用于生成和解析 Resolvable Private Address（RPA）。IRK 持有方可以对 RPA 的 prand 部分运行 ah(IRK, prand) 并与 hash 部分比对，以识别设备身份。

/* ah(k, r) = e(k, r') mod 2^24
   其中 r' = 0x000...0 || r  (r 为 24 位，pad 到 128 位) */
static int smp_ah(struct crypto_cipher *tfm, const u8 irk[16],
               const u8 r[3], u8 res[3])
⚛ 第一性原理

Legacy 体系的本质缺陷：c1 和 s1 依赖 TK（Temporary Key）作为种子，而 TK 的熵值取决于配对方法——Just Works 下 TK 为全零（128 位均为 0），攻击者只需被动抓包即可离线暴力破解 STK。这是 Legacy Pairing 最根本的安全弱点，Secure Connections 体系正是为解决此问题而设计。

🔑 SC密钥推导

03 · 安全连接密钥推导函数

⬡ Bluetooth Core Spec Part H § 2.2.5–2.2.9

LE Secure Connections（LESC）引入了基于 AES-CMAC 的五个高级密码函数（f4、f5、f6、g2、h6/h7）和 ECDH P-256 密钥交换，彻底消除了 Legacy Pairing 的被动监听漏洞。

AES-CMAC — SC 体系核心构建块

/* RFC 4493 — Cipher-based MAC using AES-128
   Linux 5.4 使用内核 crypto API */
/* smp_chan_create() 中分配 CMAC 上下文 */
smp->tfm_cmac = crypto_alloc_shash("cmac(aes)", 0, 0);

/* 通用 AES-CMAC 调用封装 */
static int aes_cmac(struct crypto_shash *tfm, const u8 k[16],
                    const u8 *m, size_t mlen, u8 mac[16])

函数 f4 — SC Confirm 值生成

f4 详细说明
参数大小含义
U256 bitPKax（发起方公钥 x 坐标）
V256 bitPKbx（响应方公钥 x 坐标）
X128 bit随机数 Na 或 Nb
Z8 bitNumeric/Just Works=0；Passkey=0x80/0x81（bit 为 passkey 位）
 
/* f4(U, V, X, Z) = AES-CMAC_X(U || V || Z) */
static int smp_f4(struct crypto_shash *tfm_cmac,
               const u8 u[32], const u8 v[32],
               const u8 x[16], u8 z, u8 res[16])
{
    u8 m[65];   /* 32 + 32 + 1 = 65 bytes */
    memcpy(m, u, 32);
    memcpy(m + 32, v, 32);
    m[64] = z;
    return aes_cmac(tfm_cmac, x, m, sizeof(m), res);
}

函数 f5 — LTK 与 MacKey 生成

f5：最关键的密钥推导函数

f5 从 ECDH 共享秘密（DHKey）推导出 LTK 和 MacKey。这是安全连接安全性的核心。

/* 规范定义：
   T = AES-CMAC_SALT(W)    SALT = 0x6C888391_AAF5A538_60370BDB_5A6083BE
   MacKey || LTK = f5(DHKey, Nc, Np, BD_ADDR_C, BD_ADDR_P)
   LTK  = AES-CMAC_T(Counter=1 || keyID || N1 || N2 || A1 || A2 || Length=256)
   MacKey = AES-CMAC_T(Counter=0 || keyID || ...) */
static int smp_f5(struct crypto_shash *tfm_cmac, const u8 w[32],
               const u8 n1[16], const u8 n2[16],
               const u8 a1[7], const u8 a2[7],
               u8 mackey[16], u8 ltk[16])

函数 f6 / g2 — 认证检查与数字比对

函数用途关键输入输出
f6 生成 DHKey Check 值（认证阶段 2） MacKey, Na, Nb, R, IOcap, A1, A2 16 字节 Ea/Eb
g2 生成 6 位数字比对值（Numeric Comparison） PKax, PKbx, Na, Nb g2 mod 2³² → 6 位十进制
h6/h7 跨传输（LE↔BR/EDR）密钥转换 LTK 或 BR/EDR Link Key 转换后的等效密钥

ECDH P-256 密钥交换

/* Linux 5.4 使用内核 crypto KPP API */
smp->tfm_ecdh = crypto_alloc_kpp("ecdh-nist-p256", 0, 0);

/* 公私钥对生成 (ecdh_helper.c) */
int generate_ecdh_keys(struct crypto_kpp *tfm, u8 public_key[64]);

/* DHKey 计算 */
int compute_ecdh_secret(struct crypto_kpp *tfm, const u8 remote_pk[64],
                        u8 secret[32]); /* 输出 DHKey (只用 x 坐标) */
调试密钥（Debug Key）：规范定义了一对固定的调试公私钥（debug_pk / debug_sk），在 Linux 5.4 的 smp.c 中硬编码。若双方均使用调试密钥，连接可被 Wireshark 等工具解密。内核通过 SMP_FLAG_DEBUG_KEY 标志跟踪此状态并在上层告警。
⚛ 第一性原理

ECDH 的本质是离散对数难题在椭圆曲线上的应用：给定点 G 和 kG，计算 k 在计算上不可行。两方各持私钥 a/b，交换公钥 aG/bG，各自计算 a(bG) = b(aG) = abG。第三方即使看到 aG 和 bG，也无法在多项式时间内还原 abG。配合 f5 的 SALT 和 keyID 混入，最终推导出的 LTK 具有密码学意义上的不可伪造性。

内核实现：get_auth_method()

/* net/bluetooth/smp.c — Linux 5.4 */
/* Legacy 方法查表 */
static const u8 gen_method[5][5] = {
    { JUST_WORKS, JUST_WORKS, JUST_CFM, JUST_WORKS, JUST_CFM },
    { JUST_WORKS, JUST_WORKS, JUST_CFM, JUST_WORKS, JUST_CFM },
    { CFM_PASSKEY, CFM_PASSKEY, REQ_PASSKEY, JUST_WORKS, CFM_PASSKEY },
    { JUST_WORKS, JUST_WORKS, JUST_CFM, JUST_WORKS, JUST_CFM },
    { CFM_PASSKEY, CFM_PASSKEY, REQ_PASSKEY, JUST_WORKS, JUST_CFM },
};

/* SC 方法查表（增加 Numeric Comparison） */
static const u8 sc_method[5][5] = {
    { JUST_WORKS, JUST_WORKS, REQ_PASSKEY, JUST_WORKS, REQ_PASSKEY },
    { JUST_WORKS, CFM_PASSKEY, REQ_PASSKEY, JUST_WORKS, CFM_PASSKEY },
    { DSP_PASSKEY, DSP_PASSKEY, REQ_PASSKEY, JUST_WORKS, DSP_PASSKEY },
    { JUST_WORKS, JUST_WORKS, JUST_WORKS,  JUST_WORKS, JUST_WORKS  },
    { DSP_PASSKEY, CFM_PASSKEY, REQ_PASSKEY, JUST_WORKS, CFM_PASSKEY },
};

static u8 get_auth_method(struct smp_chan *smp, u8 local_io, u8 remote_io)
{
    /* SC 标志已设置则使用 sc_method，否则使用 gen_method */
    if (test_bit(SMP_FLAG_SC, &smp->flags))
        return sc_method[remote_io][local_io];
    return gen_method[remote_io][local_io];
}
⚛ 第一性原理

配对方法选择的本质：用通道外（Out-of-Band）的人类认知能力来弥补无线信道缺乏身份验证的缺陷。Just Works 不借助人类判断（零带外信道），无法防御 MITM；Passkey 借助人类读数/输数（6 位数字带外信道，约 20 位熵）；Numeric Comparison 让人类在双屏上对照相同数字（视觉带外信道）；OOB 完全依赖独立安全信道（NFC、二维码等）。安全强度正比于带外信道的安全性和熵值。

🔄 Legacy配对

05 · LE Legacy 配对流程

⬡ Bluetooth Core Spec Part H § 2.3.5 LE Legacy Pairing

LE Legacy 配对在 Bluetooth 4.0–4.1 时代被广泛使用，其核心是基于 TK 的 Confirm/Random 交换机制。

图 5.1：LE Legacy Pairing 完整消息流（Just Works / Passkey Entry）

TK 值的决定规则

方法TK 值安全性
Just Works0x000...000（全零 128 bit）无 MITM 保护，被动监听可破解
Passkey Entry0x000...PPPPPP（6位PIN，0~999999）约 20 bit 熵，有限保护
OOB带外信道传递的 128 bit 随机数取决于 OOB 信道安全性
⚛ 第一性原理

Legacy Pairing 的 Confirm-then-Random 流程是一个"承诺-揭示"（Commit-Reveal）协议：先承诺（发送加密后的 Confirm），再揭示（发送 Random），对方在收到 Random 后才能验证 Confirm 是否一致，从而防止一方在看到另一方随机数后伪造自己的 Confirm。然而该协议的安全性完全依赖 TK 的熵——TK 为零时，整个承诺体系形同虚设。

🔒 安全连接

06 · LE Secure Connections 配对

⬡ Bluetooth Core Spec Part H § 2.3.5.6 LE Secure Connections

LE Secure Connections（引入自 Bluetooth 4.2）彻底重构了 Phase 2，引入 ECDH P-256 密钥交换，消除了被动监听攻击面。

图 6.1：LE Secure Connections 完整配对流程（Numeric Comparison / Just Works）

SC 与 Legacy 的关键差异

维度LE LegacyLE Secure Connections
密钥交换无（依赖 TK）ECDH P-256
生成密钥STK（临时密钥）LTK（直接生成长期密钥）
前向安全❌ 无✅ 有（每次配对独立 DHKey）
被动监听❌ 可破解（Just Works TK=0）✅ 无法破解
MITM 保护Passkey 有限保护Numeric Comparison + Passkey
加密算法AES-128（c1/s1）AES-CMAC（f4/f5/f6/g2）
内核标志—SMP_FLAG_SC
⚛ 第一性原理

Secure Connections 的安全边界在于 ECDH 的计算不可逆性：即使攻击者完整录制了所有空口报文（包括 PKa、PKb、Ca、Cb、Na、Nb、DHKey Check），在当前计算能力下也无法推导 DHKey，从而无法计算 LTK。这是 Legacy Pairing 无法做到的根本改变——Legacy 的安全性是"信息论意义上的不安全"（TK 熵值有限），而 SC 的安全性是"计算复杂度意义上的安全"（ECDH 难题）。

⚙ 内核结构

07 · 内核数据结构

⬡ net/bluetooth/smp.c — struct smp_chan / smp_dev

Linux 5.4 的 SMP 实现围绕两个核心数据结构展开：smp_dev（每适配器设备级状态）和 smp_chan（每连接会话状态）。

smp_dev — 设备级状态

/* net/bluetooth/smp.c — Linux 5.4 */
struct smp_dev {
    /* Secure Connections OOB 数据 */
    bool  local_oob;           /* 是否有本地 OOB 数据 */
    u8    local_pk[64];        /* 本地 ECDH 公钥 (x||y 各32B) */
    u8    local_rand[16];      /* OOB 随机数 */
    bool  debug_key;           /* 是否使用规范定义的调试密钥对 */

    struct crypto_shash *tfm_cmac;  /* AES-CMAC 变换上下文 */
    struct crypto_kpp  *tfm_ecdh;  /* ECDH P-256 变换上下文 */
};

smp_chan — 连接级会话状态

struct smp_chan {
    struct l2cap_conn      *conn;          /* 关联的 L2CAP 连接 */
    struct delayed_work    security_timer; /* 30 秒超时看门狗 */
    unsigned long          allow_cmd;      /* 位图: 当前允许接收的命令 */

    /* ---- 配对协商数据 ---- */
    u8  preq[7];  /* Pairing Request 原始报文 */
    u8  prsp[7];  /* Pairing Response 原始报文 */
    u8  prnd[16]; /* 本地随机数 (Na 或 Nb) */
    u8  rrnd[16]; /* 远端随机数 */
    u8  pcnf[16]; /* 远端 Confirm 值 */
    u8  tk[16];   /* Temporary Key (Legacy) */
    u8  rr[16];   /* Remote OOB ra/rb */
    u8  lr[16];   /* Local OOB ra/rb */

    /* ---- 密钥信息 ---- */
    u8   enc_key_size;      /* 协商的加密密钥长度 (7–16 字节) */
    u8   remote_key_dist;   /* 远端待分发密钥位图 */
    bdaddr_t  id_addr;      /* 远端身份地址 */
    u8   id_addr_type;
    u8   irk[16];           /* 远端 IRK */

    struct smp_ltk  *ltk;            /* 本地 LTK */
    struct smp_ltk  *slave_ltk;      /* 远端 LTK (Legacy) */
    struct smp_csrk *csrk;           /* 本地 CSRK */
    struct smp_csrk *slave_csrk;     /* 远端 CSRK */
    struct smp_irk  *remote_irk;     /* 远端 IRK 数据库条目 */
    u8              *link_key;        /* BR/EDR 链路密钥 (跨传输) */

    /* ---- 状态标志 ---- */
    unsigned long  flags;            /* SMP_FLAG_* 位图 */
    u8             method;           /* 当前配对方法 */
    u8             passkey_round;    /* SC Passkey 轮次计数 (0–19) */

    /* ---- SC 专用字段 ---- */
    u8  local_pk[64];   /* 本次连接本地公钥 */
    u8  remote_pk[64];  /* 远端公钥 */
    u8  dhkey[32];      /* ECDH 共享秘密 */
    u8  mackey[16];     /* f5 输出的 MacKey */

    struct crypto_shash *tfm_cmac;   /* 每连接 CMAC 上下文 */
    struct crypto_kpp  *tfm_ecdh;   /* 每连接 ECDH 上下文 */
};

SMP_FLAG_* 状态标志详解

标志含义触发条件
SMP_FLAG_TK_VALIDTK 已就绪（Legacy）用户输入 Passkey 或 Just Works 直接置位
SMP_FLAG_CFM_PENDINGConfirm 等待 TKTK 尚未就绪时收到对端 Confirm
SMP_FLAG_MITM_AUTH需要 MITM 保护AuthReq.MITM 位为 1
SMP_FLAG_COMPLETE配对完成密钥分发完毕
SMP_FLAG_INITIATOR本端是发起方由本端发起 Pairing Request
SMP_FLAG_SC使用 SC 模式双方 AuthReq.SC=1 且均支持
SMP_FLAG_REMOTE_PK已收到远端公钥处理 SMP_CMD_PUBLIC_KEY 后
SMP_FLAG_DEBUG_KEY使用调试密钥对远端发送规范定义的 debug_pk
SMP_FLAG_WAIT_USER等待用户输入/确认Passkey 或 Numeric Comparison
SMP_FLAG_DHKEY_PENDINGDHKey 计算中异步计算 ECDH 共享秘密期间
SMP_FLAG_CT2使用 h7 进行跨传输转换双方均支持 CT2
图 7.1：Linux 5.4 SMP 核心数据结构关系
⚛ 第一性原理

smp_dev vs smp_chan 的分离体现了"设备级共享 vs 连接级独立"的设计哲学：ECDH 密钥对的预生成（smp_dev）可以跨连接复用，降低每次配对的延迟；而每条连接的随机数、Confirm 值、密钥必须完全隔离（smp_chan），以防止跨会话的密钥混用。allow_cmd 位图则是一个轻量级状态机，确保 SMP 消息只能按协议规定的顺序到达——这是防止状态跳跃攻击的关键防线。

接收路径：smp_recv_cb → smp_sig_channel

static int smp_recv_cb(struct l2cap_chan *chan, struct sk_buff *skb)
{
    int err = smp_sig_channel(chan, skb);
    if (err) {
        struct smp_chan *smp = chan->data;
        if (smp)
            cancel_delayed_work_sync(&smp->security_timer);
        hci_disconnect(chan->conn->hcon, HCI_ERROR_AUTH_FAILURE);
    }
    return err;
}

/* smp_sig_channel: 根据命令码分发处理 */
static int smp_sig_channel(struct l2cap_chan *chan, struct sk_buff *skb)
{
    u8 code = skb->data[0];
    /* 白名单检查：命令必须在 allow_cmd 位图中 */
    if (!test_and_clear_bit(code, &smp->allow_cmd))
        return -EOPNOTSUPP;

    switch (code) {
    case SMP_CMD_PAIRING_REQ:    return smp_pairing_req(conn, skb);
    case SMP_CMD_PAIRING_RSP:    return smp_pairing_rsp(conn, skb);
    case SMP_CMD_PAIRING_CONFIRM:return smp_confirm(conn, skb);
    case SMP_CMD_PAIRING_RANDOM: return smp_random(conn, skb);
    case SMP_CMD_PUBLIC_KEY:     return smp_cmd_public_key(conn, skb);
    case SMP_CMD_DHKEY_CHECK:    return smp_cmd_dhkey_check(conn, skb);
    case SMP_CMD_IDENT_INFO:     return smp_cmd_ident_info(conn, skb);
    /* ... 其他命令 ... */
    }
}

超时机制

#define SMP_TIMEOUT msecs_to_jiffies(30000)  /* 30 秒 */

static void smp_timeout(struct work_struct *work)
{
    struct smp_chan *smp = container_of(work, struct smp_chan,
                                        security_timer.work);
    /* 超时直接断开连接，返回 AUTH_FAILURE */
    hci_disconnect(smp->conn->hcon, HCI_ERROR_AUTH_FAILURE);
}
⚛ 第一性原理

allow_cmd 位图是整个 SMP 状态机安全性的关键防线。在任意时刻，只有协议规定"下一步应当收到"的命令码才会被设置在位图中。任何乱序到来的命令都会被直接拒绝（返回错误，触发断连）。这防止了攻击者通过发送乱序 SMP 命令来跳过认证步骤或触发未定义的状态转换。30 秒超时则防止了挂起攻击（对端停止响应导致资源耗尽）。

🔨 配对实现

09 · 配对流程内核实现

⬡ net/bluetooth/smp.c — smp_pairing_req / smp_pairing_rsp / sc_send_public_key

smp_chan_create — 会话初始化

static struct smp_chan *smp_chan_create(struct l2cap_conn *conn)
{
    struct smp_chan *smp = kzalloc(sizeof(*smp), GFP_ATOMIC);
    if (!smp) return NULL;

    /* 分配每连接 AES-CMAC 上下文 */
    smp->tfm_cmac = crypto_alloc_shash("cmac(aes)", 0, 0);
    if (IS_ERR(smp->tfm_cmac)) goto zfree_smp;

    /* 分配每连接 ECDH P-256 上下文 */
    smp->tfm_ecdh = crypto_alloc_kpp("ecdh-nist-p256", 0, 0);
    if (IS_ERR(smp->tfm_ecdh)) goto free_shash;

    smp->conn = conn;
    chan->data = smp;

    /* 初始只允许接收 PAIRING_FAIL（任意时刻均可） */
    SMP_ALLOW_CMD(smp, SMP_CMD_PAIRING_FAIL);

    /* 启动 30 秒看门狗 */
    INIT_DELAYED_WORK(&smp->security_timer, smp_timeout);
    hci_conn_hold(hcon);
    return smp;
}

smp_pairing_req — 响应方处理配对请求

static u8 smp_pairing_req(struct l2cap_conn *conn, struct sk_buff *skb)
{
    struct hci_dev *hdev = conn->hcon->hdev;
    struct smp_cmd_pairing *req = (void *) skb->data;
    struct smp_cmd_pairing rsp;
    u8 key_size, auth, sec_level;

    /* 构建 Pairing Response */
    build_pairing_cmd(conn, req, &rsp, auth);

    /* 双方均置 SC 标志则启用 Secure Connections */
    if (rsp.auth_req & SMP_AUTH_SC)
        set_bit(SMP_FLAG_SC, &smp->flags);

    /* 协商加密密钥长度 (7-16 字节) */
    key_size = min(req->max_key_size, rsp.max_key_size);
    if (check_enc_key_size(conn, key_size))
        return SMP_ENC_KEY_SIZE;

    /* 生成本地随机数 */
    get_random_bytes(smp->prnd, sizeof(smp->prnd));

    /* 发送 Pairing Response */
    memcpy(&smp->prsp[1], &rsp, sizeof(rsp));
    smp_send_cmd(conn, SMP_CMD_PAIRING_RSP, sizeof(rsp), &rsp);

    if (test_bit(SMP_FLAG_SC, &smp->flags)) {
        /* SC 模式：等待对端公钥 */
        SMP_ALLOW_CMD(smp, SMP_CMD_PUBLIC_KEY);
        return 0;
    }

    /* Legacy 模式：发起 TK 请求（用户输入或 Just Works） */
    return tk_request(conn, 0, auth, rsp.io_capability, req->io_capability);
}

sc_send_public_key — SC 发送公钥

static u8 sc_send_public_key(struct smp_chan *smp)
{
    struct hci_dev *hdev = smp->conn->hcon->hdev;

    /* 若有预生成的本地 OOB 公钥，直接使用 */
    if (test_bit(SMP_FLAG_LOCAL_OOB, &smp->flags)) {
        struct smp_dev *smp_dev = SMP_DEV(hdev)->data;
        memcpy(smp->local_pk, smp_dev->local_pk, 64);
        memcpy(smp->lr, smp_dev->local_rand, 16);
    } else {
        /* 否则实时生成 ECDH 密钥对 */
        if (generate_ecdh_keys(smp->tfm_ecdh, smp->local_pk))
            return SMP_UNSPECIFIED;
        /* 检测是否使用了调试密钥 */
        if (!memcmp(smp->local_pk, debug_pk, 64))
            set_bit(SMP_FLAG_DEBUG_KEY, &smp->flags);
    }

    smp_send_cmd(smp->conn, SMP_CMD_PUBLIC_KEY, 64, smp->local_pk);
    SMP_ALLOW_CMD(smp, SMP_CMD_PUBLIC_KEY);
    return 0;
}

sc_dhkey_check — DHKey Check 阶段

static u8 sc_dhkey_check(struct smp_chan *smp)
{
    u8 a[7], b[7], *local_addr, *remote_addr;
    struct smp_cmd_dhkey_check check;

    /* 构造地址参数 (type || BD_ADDR) */
    build_local_addr(hcon, a);
    build_remote_addr(hcon, b);

    /* 计算 Ea = f6(MacKey, Na, Nb, 0, IOcapA, A, B) */
    smp_f6(smp->tfm_cmac, smp->mackey,
           smp_is_le_initiator(hcon) ? smp->prnd : smp->rrnd,
           smp_is_le_initiator(hcon) ? smp->rrnd : smp->prnd,
           r, io_cap, a, b, check.e);

    smp_send_cmd(smp->conn, SMP_CMD_DHKEY_CHECK,
                 sizeof(check), &check);
    return 0;
}
⚛ 第一性原理

DHKey Check（f6 输出的 Ea/Eb）是整个 SC 配对的"最终防线"：它将 MacKey（由 DHKey 推导）、双方随机数、IO 能力、设备地址绑定在一起，生成一个双方都能独立计算并验证的值。任何一个输入的篡改（包括 MITM 替换了某个公钥）都会导致 f6 结果不一致，从而检测到攻击。这是"承诺-揭示-验证"三步协议的最终验证步骤。

🗝 密钥分发

10 · 密钥分发实现

⬡ net/bluetooth/smp.c — smp_distribute_keys

Phase 3 的密钥分发在链路加密后进行。Linux 5.4 的 smp_distribute_keys() 按固定顺序分发：EncKey(LTK) → IdKey(IRK) → SignKey(CSRK)。

密钥类型说明

密钥全称大小用途
LTKLong Term Key128 bit未来重连时直接加密链路（免重新配对）
EDIVEncrypted Diversifier16 bitLegacy: 与 Rand 一起用于在数据库中定位 LTK
RandRandom Number64 bitLegacy: 与 EDIV 配合使用
IRKIdentity Resolving Key128 bit解析可解析私有地址（RPA），识别设备身份
CSRKConnection Signature Resolving Key128 bit对 ATT 层数据进行签名，验证完整性
 
static void smp_distribute_keys(struct smp_chan *smp)
{
    struct l2cap_conn *conn = smp->conn;
    u8 *keydist;

    /* 确定本端应分发的密钥集合 */
    keydist = test_bit(SMP_FLAG_INITIATOR, &smp->flags)
              ? &smp->local_key_dist : &smp->remote_key_dist;

    /* 1. 分发 LTK + EDIV + Rand (EncKey) */
    if (*keydist & SMP_DIST_ENC_KEY) {
        struct smp_cmd_encrypt_info enc;
        u16 ediv; u64 rand;

        get_random_bytes(enc.ltk, sizeof(enc.ltk));
        get_random_bytes(&ediv, sizeof(ediv));
        get_random_bytes(&rand, sizeof(rand));

        smp_send_cmd(conn, SMP_CMD_ENCRYPT_INFO, sizeof(enc), &enc);
        smp_send_cmd(conn, SMP_CMD_INITIATOR_IDENT, ...);
        *keydist &= ~SMP_DIST_ENC_KEY;
    }

    /* 2. 分发 IRK + 身份地址 (IdKey) */
    if (*keydist & SMP_DIST_ID_KEY) {
        struct smp_cmd_ident_info idinfo;
        memcpy(idinfo.irk, hdev->irk, sizeof(idinfo.irk));
        smp_send_cmd(conn, SMP_CMD_IDENT_INFO, sizeof(idinfo), &idinfo);
        smp_send_cmd(conn, SMP_CMD_IDENT_ADDR_INFO, ...);
        *keydist &= ~SMP_DIST_ID_KEY;
    }

    /* 3. 分发 CSRK (SignKey) */
    if (*keydist & SMP_DIST_SIGN) {
        struct smp_cmd_sign_info sign;
        get_random_bytes(sign.csrk, sizeof(sign.csrk));
        smp_send_cmd(conn, SMP_CMD_SIGN_INFO, sizeof(sign), &sign);
        *keydist &= ~SMP_DIST_SIGN;
    }

    /* 等待对端分发完毕 */
    if (smp->remote_key_dist & KEY_DIST_MASK) {
        smp_allow_key_dist(smp);
        return;
    }

    /* 双方均完成 → 通知上层，销毁 smp_chan */
    set_bit(SMP_FLAG_COMPLETE, &smp->flags);
    smp_notify_keys(conn);
    smp_chan_destroy(conn);
}

SC 模式下密钥分发的特殊处理

SMP_SC_NO_DIST 限制
/* SC 模式下，LTK 和 Link Key 已通过 f5 生成，不再通过 Phase 3 分发 */
#define SMP_SC_NO_DIST (SMP_DIST_ENC_KEY | SMP_DIST_LINK_KEY)

/* 在协商完成后清除这些位 */
smp->remote_key_dist &= ~SMP_SC_NO_DIST;

SC 模式下，LTK 由 f5 直接推导，双方各自计算相同的 LTK，无需在 Phase 3 中传输 LTK 明文。只有 IRK 和 CSRK 仍通过 Phase 3 分发。

⚛ 第一性原理

密钥分发的本质是"密钥绑定"（Key Binding）：将本次配对生成的密钥与设备身份（IRK + BD_ADDR）绑定，写入持久化存储，使未来的重连不需要重新配对。IRK 的作用尤为关键——它使接收方能够"认出"对方使用了可解析私有地址（RPA），而不暴露真实地址，在保护隐私的同时维持身份的可验证性。这是隐私保护与便利性之间的工程权衡。

static inline u8 authreq_to_seclevel(u8 authreq)
{
    if (authreq & SMP_AUTH_SC) {
        if (authreq & SMP_AUTH_MITM)
            return BT_SECURITY_FIPS;   /* Level 4 */
        else
            return BT_SECURITY_MEDIUM; /* Level 2 (SC Just Works) */
    }
    if (authreq & SMP_AUTH_MITM)
        return BT_SECURITY_HIGH;   /* Level 3 */
    return BT_SECURITY_MEDIUM;     /* Level 2 */
}

跨传输密钥推导（Cross-Transport Key Derivation, CTKD）

当设备同时支持 LE 和 BR/EDR（双模设备），可以从一种传输的密钥推导另一种传输所需的密钥，无需分别配对。Linux 5.4 通过 h6/h7 函数实现此功能。

CTKD 推导方向
图 11.1：CTKD 跨传输密钥推导方向（h6/h7 函数）
/* net/bluetooth/smp.c — h6 实现 */
static int smp_h6(struct crypto_shash *tfm_cmac,
               const u8 w[16], const u8 key_id[4], u8 res[16])
{
    return aes_cmac(tfm_cmac, w, key_id, 4, res);
}

/* h7: 用于 CT2=1 时的中间密钥推导 */
static int smp_h7(struct crypto_shash *tfm_cmac,
               const u8 w[16], const u8 salt[16], u8 res[16])
{
    return aes_cmac(tfm_cmac, salt, w, 16, res);
}

/* BR/EDR Link Key → LE LTK (CT2=1 路径) */
static int sc_derive_link_key(struct smp_chan *smp)
{
    u8 tmp1[16], tmp2[16];
    if (test_bit(SMP_FLAG_CT2, &smp->flags)) {
        /* CT2: h7(SALT="tmp1", LTK) → h6(result, "brle") */
        smp_h7(smp->tfm_cmac, smp->tk, salt, tmp1);
        smp_h6(smp->tfm_cmac, tmp1, "brle", link_key);
    } else {
        /* CT2=0: 直接 h6(LTK, "brle") */
        smp_h6(smp->tfm_cmac, smp->tk, "brle", link_key);
    }
}
⚛ 第一性原理

安全级别体系的本质是"最低可接受安全保证"的形式化：应用程序声明它所需的最低安全等级，内核负责确保连接满足该等级后才允许数据流通。CTKD 则解决了双模设备的"配对冗余"问题——从数学上证明，如果 LE SC 配对是安全的，则从其 LTK 推导出的 BR/EDR Link Key 具有等价安全性，无需重新在 BR/EDR 上配对。

1. KNOB 攻击（Key Negotiation Of Bluetooth）

攻击者在空口篡改密钥长度协商，强迫双方使用 1 字节加密密钥，极易暴力破解。

默认最大密钥长度SMP_MAX_ENC_KEY_SIZE = 16
 
static u8 check_enc_key_size(struct l2cap_conn *conn, u8 enc_key_size)
{
    struct hci_dev *hdev = conn->hcon->hdev;
    if (enc_key_size > SMP_MAX_ENC_KEY_SIZE ||
        enc_key_size < hdev->le_min_key_size)
        return SMP_ENC_KEY_SIZE;  /* 拒绝并返回错误码 */
    return 0;
}
2. 调试密钥泄露

规范定义的调试公钥（debug_pk）是已知固定值，若设备使用调试密钥，连接可被任何人解密。

/* 检测到调试密钥时置位，并通过 mgmt 层向用户空间告警 */

    set_bit(SMP_FLAG_DEBUG_KEY, &smp->flags);

/* HCI_FORCE_DEBUG_KEYS: 调试模式标志，生产环境绝不应启用 */

某些实现错误地从 Security Request 中复制了 RFU 位，导致对端误以为支持 SC，但实际不支持。Linux 5.4 有专门的 workaround：

/* fixup_sc_false_positive() — 仅在 Responder 角色触发 */
static int fixup_sc_false_positive(struct smp_chan *smp)
{
    /* 如果我们发送了 SC 位但对端又没发公钥，
       尝试降级回 Legacy 模式重新协商 */
    clear_bit(SMP_FLAG_SC, &smp->flags);
    return tk_request(...);  /* 重新启动 Legacy 流程 */
}
4. 状态机防护 — allow_cmd 位图

任何乱序命令（如在未完成 Feature Exchange 前收到 DHKey Check）都会被 allow_cmd 检查拦截，触发断连。这防止了各类状态机跳跃攻击。

⚛ 第一性原理

SMP 的安全边界来自三个层次：协议层（规范定义的密码学函数和消息顺序）、实现层（内核的状态机白名单、密钥长度检查）、使用层（用户选择的配对方法和 IO 能力）。任何一层的弱点都可能导致整体安全性下降。最根本的教训是：安全性不能依赖单点，必须在每一层都有独立的检查和防护。Just Works 模式的存在表明，有时系统会显式地用安全性换取便利性——理解这种取舍，比盲目追求最高安全配置更重要。
