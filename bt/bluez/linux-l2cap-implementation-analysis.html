Linux 5.4 蓝牙 L2CAP 深度解析

📌 基准版本：Linux Kernel v5.4 📁 源码路径：net/bluetooth/l2cap_core.c 📋 规范来源：Bluetooth Core Spec 5.x Vol.3 Part A

L2CAP（Logical Link Control and Adaptation Protocol，逻辑链路控制与适配协议）是蓝牙协议栈中介于底层 HCI（Host Controller Interface）与上层应用协议之间的核心中间件层。它提供多路复用、数据包分段与重组（SAR）、逐信道流量控制和错误控制四大核心能力。

🎯 第一性原理：L2CAP 存在的根本原因 无线信道本质上是不可靠、带宽受限的共享介质。蓝牙基带只提供原始的 ACL 数据包传输，不区分上层协议，也不支持多路复用。L2CAP 解决的核心问题是：如何在单条物理链路上，可靠地承载多个相互独立的逻辑数据流？ 答案是：用 CID 做多路复用，用 SAR 适配 MTU，用可选的 ERTM/流控模式按需提供可靠性保障。
设计原理

L2CAP 设计原理

1.1 核心功能目标

依据 Bluetooth Core Specification Vol.3 Part A Section 1.1，L2CAP 的功能需求包含：

  • 协议/信道多路复用（Protocol/Channel Multiplexing）：单条 ACL 逻辑链路上同时承载多个上层协议（RFCOMM、AVDTP、ATT、SMP 等）
  • 分段与重组（SAR）：将上层 SDU（最大 64KB）切分成适合 HCI 缓冲区的 PDU 并在接收端重组
  • 逐信道流控（Per-channel Flow Control）：防止接收方缓冲区溢出
  • 错误控制与重传（Error Control & Retransmission）：补偿基带误码，提供比 Controller 更高的可靠性
  • QoS 协商：在连接建立阶段交换质量期望参数

1.2 架构假设

规范原文 Section 1.2 — Assumptions 规范明确设计依赖以下前提：Controller 保证数据包有序投递，但可能存在单包损坏和重复；Controller 始终呈现全双工信道；部分应用需要比 Controller 更好的错误控制能力（因此 L2CAP 提供 ERTM 等可选模式）。

1.3 L2CAP 功能架构块

规范将 L2CAP 分为三个内部功能块（Section 1.1 Figure 1.1）：

图1-1：L2CAP 功能架构块（基于规范 Section 1.1 Figure 1.1）

🎯 第一性原理：分层解耦 L2CAP 的三个功能块体现了"关注点分离"原则：Channel Manager 管理控制平面（状态机/信令），Retransmission & Flow Control 管理数据平面可靠性，Resource Manager 负责与硬件接口的适配调度。这三者的清晰分工使得不同可靠性需求的信道可以共存于同一物理链路上，而互不干扰。

信道标识符 CID

3.1 CID 概念与作用域

根据规范 Section 2.1，CID（Channel Identifier）是设备端对逻辑信道端点的本地名称。CID 的作用域与逻辑链路绑定，即同一设备上不同 ACL 连接的 CID 空间相互独立。

3.3 PSM（Protocol/Service Multiplexer）

对于动态信道，连接请求需要携带 PSM 值来路由到正确的上层协议。PSM 分两个范围：

PSM 范围分配方典型值
0x0001–0x00FF（奇数）Bluetooth SIG 分配SDP=0x0001, RFCOMM=0x0003, AVDTP=0x0019, AVCTP=0x0017
0x1001–0xFFFF（奇数）厂商自定义各厂商私有协议
🔍 内核实现：PSM 注册 在 Linux 5.4 中，上层协议通过 l2cap_add_psm(chan, psm) 注册 PSM，内核维护一个全局的 chan_list 链表，当收到 L2CAP_CONN_REQ 时在该链表中查找匹配的 PSM 处理信道。
🎯 第一性原理：用数字代替名字 CID 和 PSM 本质上都是解决"如何在共享信道上区分不同会话"的问题。CID 是连接建立后动态协商的本地端口号（类比 TCP 端口），PSM 是服务发现阶段的协议标识（类比 TCP 端口的固定知名端口）。固定信道（0x0001–0x003F）则是"永远在线"的特殊服务，无需握手。

工作模式

Basic Mode（基本模式）

5.1 定义

Basic Mode 是 L2CAP 的默认和最简单模式，对应规范 Section 3.1 和 Section 7。在此模式下：

  • 没有重传机制，没有 ACK
  • 上层 SDU 直接映射为 B-frame PDU
  • 如果接收缓冲区满，内核直接丢包（规范称为"bad"，但别无选择）
  • MTU 默认值：672 字节（L2CAP_DEFAULT_MTU）
  • 最小 MTU：48 字节（L2CAP_DEFAULT_MIN_MTU）

5.2 内核实现（l2cap_core.c）

/* Basic Mode 数据发送路径 (l2cap_chan_send) */
case L2CAP_MODE_BASIC:
    /* 直接分配 skb，写入 L2CAP 基本头部 */
    skb = l2cap_create_basic_pdu(chan, msg, len);
    if (IS_ERR(skb))
        return PTR_ERR(skb);
    l2cap_do_send(chan, skb);  /* 调用 hci_send_acl */
    err = len;
    break;

/* Basic Mode 接收路径 (l2cap_data_channel) */
case L2CAP_MODE_BASIC:
    if (chan->imtu < skb->len)
        goto drop;  /* 超过 MTU 直接丢弃 */
    if (!sock_queue_rcv_skb(sk, skb))
        goto done;
    break;

5.3 适用场景

✅ 适合使用

  • SDP 服务发现
  • ATT 属性协议（BLE GATT）
  • SMP 安全管理
  • 对延迟敏感、丢包可容忍的场景

❌ 不适合

  • 需要数据完整性保证（如文件传输）
  • 基带错误率较高的环境
  • 大数据量传输（无法利用 SAR）
🎯 第一性原理：够用原则 Basic Mode 体现了"不为不需要的功能付费"的设计哲学。对于 SDP、ATT 这类"查询-响应"型协议，底层蓝牙基带的 CRC 校验已经提供了足够可靠性，额外的 ERTM 重传机制只会增加延迟和实现复杂度。

ERTM — 增强重传模式

6.1 设计目标

ERTM（Enhanced Retransmission Mode）在规范 Section 8 中详细定义。其目标是在 BR/EDR 链路上提供接近 TCP 的可靠性，通过滑动窗口 + 选择性重传（SREJ）实现高效数据传输。

6.5 定时器机制

Linux 5.4 实现了三个 ERTM 定时器（均用 delayed_work 实现）：

/* include/net/bluetooth/l2cap.h */
struct l2cap_chan {
    ...
    struct delayed_work retrans_timer;   /* 重传定时器：2000ms */
    struct delayed_work monitor_timer;   /* 监控定时器：12000ms */
    struct delayed_work ack_timer;       /* ACK 延迟定时器：200ms */
    ...
};
定时器启动时机超时动作
RetransTimer发送 I-frame 后，无已发送的 S-frame(P=1)重传所有未 ACK 的帧，发 RR(P=1)，启动 MonitorTimer
MonitorTimerRetransTimer 超时后，或发 RR/RNR(P=1) 后重新发 RR(P=1)；超时次数超限则断开连接
AckTimer收到 I-frame 但暂时不想立刻发 S-frame发送延迟的 RR ACK（批量确认优化）

6.6 内核 ERTM 初始化（l2cap_ertm_init）

static int l2cap_ertm_init(struct l2cap_chan *chan)
{
    chan->next_tx_seq = 0;
    chan->expected_tx_seq = 0;
    chan->expected_ack_seq = 0;
    chan->unacked_frames = 0;
    chan->buffer_seq = 0;
    chan->tx_win = min_t(u16, chan->tx_win,
                         L2CAP_DEFAULT_TX_WINDOW);
    chan->tx_win_max = L2CAP_DEFAULT_EXT_WINDOW;

    /* 初始化 SREJ 队列和列表 */
    skb_queue_head_init(&chan->srej_q);
    l2cap_seq_list_init(&chan->srej_list, chan->tx_win);

    /* 初始化三个定时器 */
    INIT_DELAYED_WORK(&chan->retrans_timer, l2cap_retrans_timeout);
    INIT_DELAYED_WORK(&chan->monitor_timer, l2cap_monitor_timeout);
    INIT_DELAYED_WORK(&chan->ack_timer, l2cap_ack_timeout);

    return 0;
}
🎯 第一性原理：可靠性的代价 ERTM 的本质是：用内存（重传缓冲区、SREJ 队列）和时间（等待 ACK）换取可靠性。SREJ 比 REJ（Go-Back-N）更高效，因为它只重传真正丢失的帧，而非整段重传。但 SREJ 要求接收方有乱序缓冲区，增加了内存需求。这是典型的"时间-空间-可靠性"三角权衡。
工作模式

Streaming Mode（流模式）

7.1 设计哲学

规范 Section 8.7 定义了 Streaming Mode。它使用与 ERTM 相同的帧格式（I-frame），但完全不确认（No ACK），不使用 RR/REJ/RNR/SREJ，不启动 RetransTimer 和 MonitorTimer。

这是专为音频/视频流设计的模式——对于这类应用，延迟比完整性更重要：一个迟到的音频帧比没有音频帧更差。

7.2 Streaming Mode 与 ERTM 对比

特性ERTMStreaming Mode
帧格式I-frame + S-frameI-frame only（无 S-frame）
序列号（TxSeq）有，用于确认和重传有，但仅用于检测丢帧
ACK 机制RR/SREJ 确认无
重传支持（RetransTimer）不支持
F-bit用于 Poll-Final 握手始终为 0（忽略）
丢帧处理SREJ 请求重传标记为丢失，继续处理后续帧
FlushTimeout无穷大（不冲刷）有限值（保证数据流动）
典型用途OBEX 文件传输、RFCOMMA2DP 音频（AVDTP 承载）

7.3 内核实现

/* Streaming 发送路径（l2cap_chan_send） */
case L2CAP_MODE_STREAMING:
    __skb_queue_head_init(&seg_queue);
    err = l2cap_segment_sdu(chan, &seg_queue, msg, len);
    if (err) break;
    l2cap_streaming_send(chan, &seg_queue);  /* 直接发出，不等 ACK */
    err = len;
    break;

/* l2cap_streaming_send 关键逻辑 */
static void l2cap_streaming_send(struct l2cap_chan *chan,
                                  struct sk_buff_head *skbs)
{
    struct sk_buff *skb;
    struct l2cap_ctrl *control;

    skb_queue_splice_tail_init(skbs, &chan->tx_q);
    while ((skb = skb_dequeue(&chan->tx_q))) {
        control = &bt_cb(skb)->l2cap;
        control->txseq = chan->next_tx_seq;
        chan->next_tx_seq = __next_seq(chan, chan->next_tx_seq);
        l2cap_do_send(chan, skb);  /* 立即发送，不存入重传队列 */
    }
}
🎯 第一性原理：实时性优先 Streaming Mode 的本质逻辑是：对于音频/视频这类时效性数据，重传会导致时序混乱（新数据必须等待重传的旧数据），这比直接丢帧更有害。因此放弃重传，用 TxSeq 仅作丢包检测（通知上层哪些帧丢了），让上层 codec 进行错误隐藏（PLC）。这是"信息论最优"策略：不浪费带宽和时间在对实时性无益的重传上。

LE Credit Based Flow Control

8.1 背景与目标

BLE 引入了全新的流控模式（规范 Section 10.1），专门为 LE L2CAP 面向连接信道设计，使用信用（Credit）机制替代 ERTM 的确认机制，更适合低功耗场景。

8.4 Enhanced Credit Based（ECOC，Linux 5.4 引入）

ℹ️ ECOC in Linux 5.4 Linux 5.4 通过 enable_ecred 模块参数（CONFIG_BT_LE_L2CAP_ECRED）支持 Enhanced Credit Based，允许一次请求同时建立最多 5 条信道（L2CAP_CREDIT_BASED_CONNECTION_REQ），并支持信道重配置。
🎯 第一性原理：推模型 vs. 拉模型 传统 ERTM 是"推模型"：发送方不断发送，接收方用 RNR 刹车。Credit 机制是"拉模型"：接收方主动给出许可配额，发送方用完就停。拉模型在低功耗场景更优，因为接收方可以精确控制自己的缓冲区消耗，不需要频繁的 RNR/RR 往返，更节能。
连接管理

信令通道与命令

9.1 信令信道

L2CAP 使用固定信道 CID 0x0001（BR/EDR）和 CID 0x0005（LE）传输信令命令（C-frame）。一个 C-frame 在 BR/EDR 上可以携带多个命令（批处理）；而 LE 信令信道每个 C-frame 只携带一个命令。

14.1 核心入口函数

ERTM 的核心是两个状态机驱动函数，由规范中的事件表直接映射而来：

/* 发送状态机入口 */
static void l2cap_tx(struct l2cap_chan *chan,
                     struct l2cap_ctrl *control,
                     struct sk_buff_head *skbs,
                     u8 event)
{
    /* event: L2CAP_EV_DATA_REQUEST / RECV_REQSEQ_AND_FBIT /
              RECV_FBIT / RETRANS_TO / MONITOR_TO /
              EXPLICIT_POLL / LOCAL_BUSY_DETECTED / CLEAR */
    l2cap_tx_state_machine[chan->tx_state][event](chan, control, skbs);
}

/* 接收状态机入口 */
static int l2cap_rx(struct l2cap_chan *chan,
                    struct l2cap_ctrl *control,
                    struct sk_buff *skb,
                    u8 event)
{
    /* event: RECV_IFRAME / RECV_RR / RECV_REJ / RECV_RNR / RECV_SREJ */
    return l2cap_rx_state_machine[chan->rx_state][event](chan, control, skb);
}

14.2 关键事件处理（接收路径）

/* 接收到 I-frame 的处理（RECV 状态）*/
static int l2cap_rx_state_recv(struct l2cap_chan *chan,
                               struct l2cap_ctrl *control,
                               struct sk_buff *skb, u8 event)
{
    switch (l2cap_classify_txseq(chan, control->txseq)) {
    case L2CAP_TXSEQ_EXPECTED:
        /* 期望的序号 → 直接送达上层 */
        l2cap_pass_to_tx(chan, control);
        chan->expected_tx_seq = __next_seq(chan, chan->expected_tx_seq);
        l2cap_reassemble_sdu(chan, skb, control);
        break;

    case L2CAP_TXSEQ_UNEXPECTED:
        /* 乱序帧 → 发送 SREJ，进入 SREJ_SENT 状态 */
        l2cap_send_srej(chan, control->txseq);
        chan->rx_state = L2CAP_RX_STATE_SREJ_SENT;
        break;

    case L2CAP_TXSEQ_DUPLICATE:
        /* 重复帧 → 发 RR 确认，忽略数据 */
        l2cap_pass_to_tx(chan, control);
        break;

    case L2CAP_TXSEQ_INVALID:
        /* 无效序号 → 断开连接 */
        l2cap_send_disconn_req(chan, ECONNRESET);
        break;
    }
    return 0;
}

14.3 TxSeq 分类（l2cap_classify_txseq）

分类条件动作
TXSEQ_EXPECTEDtxseq == expected_tx_seq正常接收，更新 expected_tx_seq
TXSEQ_EXPECTED_SREJ在 SREJ_SENT 状态中，txseq 是 srej_list.headSREJ 请求的帧已补收
TXSEQ_UNEXPECTEDtxseq 在窗口内但不是期望值发 SREJ 请求丢失帧
TXSEQ_UNEXPECTED_SREJSREJ_SENT 中，收到不在 srej_list 中的帧发 SREJ
TXSEQ_DUPLICATEtxseq <= last_acked_seq重复帧，忽略数据
TXSEQ_INVALIDtxseq 超出有效范围断开连接
TXSEQ_INVALID_IGNORE窗口超过一半时的无效序号静默忽略（双重 Poll 场景）

14.4 SREJ 重传实现

static void l2cap_send_srej(struct l2cap_chan *chan, u16 txseq)
{
    struct l2cap_ctrl control;
    u16 seq;

    memset(&control, 0, sizeof(control));
    control.sframe = 1;
    control.super = L2CAP_SUPER_SREJ;

    /* 对 [expected_tx_seq, txseq) 中每个未请求的序号发一个 SREJ */
    for (seq = chan->expected_tx_seq; seq != txseq;
         seq = __next_seq(chan, seq)) {
        if (!l2cap_ertm_seq_in_queue(&chan->srej_q, seq)) {
            control.reqseq = seq;
            l2cap_send_sframe(chan, &control);
            l2cap_seq_list_append(&chan->srej_list, seq);
        }
    }
    chan->expected_tx_seq = __next_seq(chan, txseq);
}
🎯 第一性原理：状态机即规范的代码化 ERTM 的内核实现是规范 Section 8 状态表的直接映射：每个 (state, event) 对应一个处理函数。这种设计保证了实现与规范的一致性，任何偏差（bug）都可以通过对照规范状态表来发现。l2cap_classify_txseq() 是整个接收状态机的"判断门"，它将连续的序号空间切分为有限的、语义明确的类别，极大简化了后续处理逻辑。

15.3 重组实现

static int l2cap_reassemble_sdu(struct l2cap_chan *chan,
                                 struct sk_buff *skb,
                                 struct l2cap_ctrl *control)
{
    switch (control->sar) {
    case L2CAP_SAR_UNSEGMENTED:
        /* 未分段，直接回调上层 recv */
        chan->ops->recv(chan, skb);
        break;

    case L2CAP_SAR_START:
        /* 提取 SDU 总长，创建重组缓冲区 */
        chan->sdu_len = get_unaligned_le16(skb->data);
        skb_pull(skb, L2CAP_SDULEN_SIZE);  /* 去掉 2 字节 SDU Length */
        chan->sdu = skb;
        break;

    case L2CAP_SAR_CONTINUE:
        /* 追加到重组缓冲区 */
        skb_copy_from_linear_data(skb, skb_put(chan->sdu, skb->len),
                                  skb->len);
        break;

    case L2CAP_SAR_END:
        /* 追加末段，SDU 完整，送达上层 */
        skb_copy_from_linear_data(skb, skb_put(chan->sdu, skb->len),
                                  skb->len);
        chan->ops->recv(chan, chan->sdu);
        chan->sdu = NULL;
        break;
    }
    return 0;
}
🎯 第一性原理：流切包 SAR 解决了一个根本问题：L2CAP SDU 可达 64KB，而蓝牙基带 ACL 数据包通常仅有几百字节。必须切片传输（分段），且接收方必须能还原原始边界（重组）。SAR 字段（2 bits）是最经济的边界标记方案——4 个状态恰好够用：不需要切/开始切/继续切/切完了。SDU Length 仅在首段携带，避免了每帧重复携带元信息的冗余。

}

17.3 FCS 帧校验

ERTM 模式下，L2CAP 在每个 I-frame/S-frame 末尾附加 CRC-16 校验（可协商关闭）：

/* CRC-16 计算（使用内核 crc16 模块）*/
static inline u16 l2cap_fcs_crc(const u8 *data, size_t len)
{
    return crc16(0, data, len);   /* linux/crc16.h */
}

/* 发送时追加 FCS */
static void l2cap_do_send(struct l2cap_chan *chan, struct sk_buff *skb)
{
    if (chan->fcs == L2CAP_FCS_CRC16) {
        u16 fcs = crc16(0, skb->data, skb->len);
        put_unaligned_le16(fcs, skb_put(skb, 2));
    }
    hci_send_acl(chan->conn->hchan, skb, flags);
}
🎯 第一性原理：纵深防御 L2CAP 安全是多层的：蓝牙基带提供 FHSS 防窃听；链路层提供 AES-128 加密；L2CAP 层提供 PSM 级别的访问控制和密钥长度验证；FCS 提供数据完整性校验。每一层解决不同的威胁模型，共同构成纵深防御体系。

19.3 从第一性原理回望 L2CAP

🎯 最终第一性原理归纳

L2CAP 的所有设计都源于一个根本约束：蓝牙物理链路是单一的、共享的、不可靠的无线信道。面对这个约束，L2CAP 给出了以下"最小充分"解答：

  • 多路复用问题 → CID（数字标识符，零额外协商）
  • MTU 适配问题 → SAR（4 状态位标记切片边界）
  • 可靠性需求多样 → 可插拔模式（Basic/ERTM/Stream/Credit），按需启用
  • 流控问题 → 滑动窗口（ERTM）或 Credit（LE），两种范式各适其场
  • 互操作性问题 → 配置协商（TLV 选项 + UNACCEPT 协商机制）
  • 安全性问题 → PSM 级别访问控制 + 密钥长度验证 + FCS 完整性校验

Linux 5.4 的实现将规范状态机一一映射为代码（l2cap_tx/rx），用 kref 管理生命周期，用 ops 模式解耦上层，用 mutex 保护并发。这是一个教科书级的"协议规范→内核实现"范例。
