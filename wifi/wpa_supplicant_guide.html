01概述与背景

📄 来源：hostap.epitest.fi/wpa_supplicant/ & linux.org man page

wpa_supplicant 是 Linux、BSD、macOS 和 Windows 平台上的 WPA/WPA2/WPA3 客户端认证守护进程，由芬兰开发者 Jouni Malinen 编写，以 GPLv2/BSD 双重许可证发布。它实现了 IEEE 802.11i 标准中"Supplicant（请求者）"的角色——负责与接入点（AP/Authenticator）协商密钥、完成认证并控制无线驱动的关联行为。

为什么需要 wpa_supplicant？

无线网络与有线网络最大的不同在于：任何人都可以在物理范围内被动监听所有帧。IEEE 802.11 最初的 WEP 加密已被证明存在严重缺陷（RC4 密钥调度问题、IV 空间过小、无重放保护等）。wpa_supplicant 正是为了实现更安全的 WPA/WPA2 协议（基于 AES-CCMP）而诞生。

图1：Wi-Fi 安全协议演进时间线

核心能力

特性说明
WPA/WPA2-Personal（PSK）使用预共享密钥认证，适合家庭/小型办公网络
WPA/WPA2-Enterprise（EAP）与 RADIUS 服务器联动的企业级认证
WPA3-SAE同等认证（Simultaneous Authentication of Equals），抵御离线字典攻击
IEEE 802.1X 有线认证支持有线网卡的端口级访问控制
多驱动支持nl80211（现代标准）、wext（传统）、wired 等
RSN 预认证 / PMKSA 缓存漫游时加速重关联，减少切换延迟

📝 本章小结

  • wpa_supplicant 是无线网络客户端的"认证大脑"，运行于用户空间，以守护进程形式工作
  • 它解决了 WEP 时代遗留的无线网络安全问题，实现了 WPA/WPA2/WPA3 三代协议
  • 物理设备+驱动就绪后，wpa_supplicant 完成认证，之后才能进行 DHCP 等上层配置

🔬 第一性原理

无线网络的本质问题是：如何在不可信的广播介质上实现只有合法双方才能读取的通信。wpa_supplicant 的答案是：先通过密钥协商协议证明双方持有相同的"秘密"（PSK 或 EAP 凭证），再用由此派生的临时密钥对数据加密。

# 自动尝试 nl80211，失败则回退 wext
wpa_supplicant -Dnl80211,wext -iwlan0 -c/etc/wpa_supplicant.conf

📝 本章小结

  • 驱动层通过 nl80211（现代）或 wext（传统）与内核 mac80211 层交互

03核心协议原理

📄 来源：hostap.epitest.fi — WPA & IEEE 802.11i/RSN 章节

WPA vs WPA2 vs WPA3

协议标准加密算法密钥交换安全级别
WEPIEEE 802.11 原始RC4 + CRC32静态密钥❌ 已破解
WPA802.11i Draft 3TKIP + RC4（每包密钥）4次握手⚠️ 过渡方案
WPA2IEEE 802.11iCCMP + AES-1284次握手✅ 主流安全
WPA3Wi-Fi Alliance 2018CCMP + AES-256SAE Dragonfly🔒 最强安全

密钥层次结构

官方文档明确指出，WPA 实现了全新的密钥握手机制（4-Way Handshake + Group Key Handshake）用于在认证方（AP）和请求方（客户端）之间生成和交换数据加密密钥。

图3：WPA2 密钥层次结构（PMK → PTK → TK/GTK）

两种认证模式

🏠 WPA-Personal (PSK 模式)

PMK 直接由用户密码+SSID 通过 PBKDF2-SHA1（4096次迭代）派生，无需认证服务器。Wi-Fi Alliance 称之为"WPA-Personal"。

PMK = PBKDF2(HMAC-SHA1, passphrase, SSID, 4096, 32字节)

两端（AP 和客户端）各自用相同密码计算 PMK，PMK 本身不在网络上传输，4次握手只是证明双方持有相同的 PMK。

🏢 WPA-Enterprise (EAP 模式)

PMK 来自 EAP 认证过程产生的主会话密钥（MSK）。客户端通过 AP 代理与 RADIUS 认证服务器交换 EAP 消息，认证成功后 RADIUS 将 MSK 传给 AP，客户端本地也派生出 MSK，双方从中截取 PMK。

  • 支持 EAP-TLS（证书）、EAP-PEAP（用户名+密码）、EAP-TTLS 等多种方法
  • 每个用户有独立的 PMK，一个用户密钥泄露不影响其他用户
  • 需要部署 RADIUS 服务器（如 FreeRADIUS）

📝 本章小结

  • WPA2 用 AES-CCMP 替代 WEP 的 RC4，从根本上解决了加密强度问题
  • 密钥不直接传输，而是通过 PRF 从双方共知的材料（PMK + Nonce + MAC）独立派生 PTK
  • PSK 模式简单但所有用户共享密码；Enterprise 模式按用户独立认证，安全性更高

🔬 第一性原理

密钥协商的核心挑战：如何让两方在不安全信道上协商出相同的密钥，同时第三方无法获知。WPA2 的答案是：各自用已知的共同秘密（PMK）和公开交换的随机数（Nonce）运行相同的 PRF，得到相同结果——本质上是对称密码体系中的密钥派生函数（KDF）应用。

04四次握手详解

📄 来源：hostap.epitest.fi — Using wpa_supplicant 节 & WPA2 4-Way Handshake Breakdown

官方文档描述的关联步骤：wpa_supplicant 扫描→选择 BSS→关联→（WPA-EAP 或 WPA-PSK 获得 PMK）→完成4次握手→配置加密密钥→正常传输数据。

图4：WPA2 四次握手 EAPOL-Key 消息交换时序

四条消息详解

  • M1
    AP → 客户端：ANonce（认证方随机数） AP 生成 32 字节随机数 ANonce，通过 EAPOL-Key 帧明文发送。此时无 MIC 保护，攻击者可见但无法利用（没有 PMK 就无法派生 PTK）。客户端收到 ANonce 后，生成自己的 SNonce 并立即派生 PTK。
  • M2
    客户端 → AP：SNonce + MIC（消息完整性码） 客户端发送 SNonce 和用 KCK 计算的 MIC。AP 收到后，用 ANonce、SNonce、双方 MAC 地址和 PMK 同样派生 PTK，验证 MIC——MIC 一致说明客户端密码正确。这是认证的核心机制。
  • M3
    AP → 客户端：GTK（加密）+ Install 标志 + MIC AP 用 KEK 加密 GTK（组临时密钥，用于广播/多播）并携带 Install 标志和 MIC 发给客户端。客户端验证 AP 的 MIC——完成双向认证（mutual authentication）。
  • M4
    客户端 → AP：确认 ACK + MIC 客户端确认已安装密钥。AP 收到并验证后，双方同时安装 TK（临时密钥），从此用 AES-CCMP 加密所有数据帧。握手完成。

💡 为什么 PMK 不在网络上传输？ 安全的密钥协商协议不传输秘密本身，而是让双方独立推导出相同结果并互相验证（通过 MIC）。这样即使攻击者抓取全部握手包，也只能离线暴力枚举密码（PMKID 攻击利用的正是 M1 帧）。

📝 本章小结

  • 四次握手通过 EAPOL-Key 帧在 AP 和客户端之间完成密钥协商，全程不传输 PMK
  • M2 的 MIC 验证密码正确性，M3 的 MIC 实现双向认证，M4 确认密钥安装
  • PTK = KCK（128bit）+ KEK（128bit）+ TK（128bit），分别用于 MIC 计算、GTK 加密、数据加密

🔬 第一性原理

四次握手本质是一种零知识证明变体：双方通过交换随机数（Nonce），用共享秘密（PMK）各自独立计算出临时密钥（PTK），再互相出示用 PTK 衍生的 KCK 签名的 MIC 来证明"我知道 PMK"——而不暴露 PMK 本身。

05命令行参数

📄 来源：wpa_supplicant(8) — COMMAND LINE OPTIONS

SYNOPSIS

wpa_supplicant [ -BddfhKLqqsTtuvW ] [ -iifname ] [ -cconfig file ] 
               [ -Ddriver ] [ -PPID_file ] [ -foutput_file ]

常用参数说明

参数说明示例
-B后台运行（fork into background）-B
-i指定网络接口名-i wlan0
-c配置文件路径（每接口）-c /etc/wpa_supplicant.conf
-D驱动类型，可逗号分隔多个回退-D nl80211,wext
-d增加调试详细度（-dd 更详细）-d
-f日志输出到文件（需编译 CONFIG_DEBUG_FILE）-f /var/log/wpa.log
-K调试输出中包含密钥/密码（慎用！）-K
-P写入 PID 文件-P /run/wpa_supplicant.pid
-s日志输出到 syslog-s
-t调试消息包含时间戳-t
-u启用 D-Bus 控制接口-u
-v显示版本号-v
-N多接口分隔符，后接新接口的参数组-N -i wlan1 -c ...
-Cctrl_interface socket 路径（不使用 -c 时）-C /run/wpa_supplicant
-b可选的桥接接口名-b br0
-W等待控制接口监视器后再启动-W

官方示例命令

# 最常用：后台运行，指定接口和配置文件
wpa_supplicant -B -c/etc/wpa_supplicant.conf -iwlan0

# 前台运行 + 调试（排查问题首选）
wpa_supplicant -c/etc/wpa_supplicant.conf -iwlan0 -d

# 自动选择驱动（nl80211 失败则用 wext）
wpa_supplicant -Dnl80211,wext -c/etc/wpa_supplicant.conf -iwlan0

# 同时管理两个接口（官方 man page 示例）
wpa_supplicant \
    -c wpa1.conf -i wlan0 -D nl80211 -N \
    -c wpa2.conf -i ath0  -D wext

📝 本章小结

  • -B -i <iface> -c <conf> 是生产环境最常用的启动三件套
  • -d 调试模式不要在生产中长期使用，-K 会泄露密钥绝对仅限测试
  • 多接口用 -N 分隔，一个进程可以管理多个无线网卡

06配置文件详解

📄 来源：wpa_supplicant.conf(5) man page

wpa_supplicant 通过文本配置文件管理网络列表和安全策略。官方说明："配置文件中的所有文件路径应使用绝对路径"，"通过发送 SIGHUP 信号可让 wpa_supplicant 重新读取配置文件"。

全局配置项

# 控制接口路径（wpa_cli 需要此项）
ctrl_interface=DIR=/var/run/wpa_supplicant GROUP=wheel

# 允许 wpa_cli 写回配置
update_config=1

# AP 扫描模式：1=自动扫描（无线），0=不扫描（有线/手动）
ap_scan=1

# 国家代码（影响信道和功率限制）
country=CN

# 快速重认证（EAP 场景）
fast_reauth=1

network 块结构

network={
    ssid="网络名称"          # SSID
    scan_ssid=1            # 1=主动扫描（隐藏SSID时必须）
    key_mgmt=WPA-PSK       # 密钥管理：WPA-PSK|WPA-EAP|NONE|SAE
    psk="密码明文"           # 或用 wpa_passphrase 生成的哈希值
    pairwise=CCMP TKIP     # 单播加密套件
    group=CCMP TKIP        # 组播加密套件
    priority=5             # 优先级，越大越优先
}

配置重载与热更新

# 发送 SIGHUP 重读配置（官方文档说明）
killall -HUP wpa_supplicant

# 或通过 wpa_cli
wpa_cli reconfigure

⚠️ 设置 update_config=1 允许 wpa_supplicant 覆写配置文件。在多用户系统中，请确保配置文件权限为 chmod 600，否则密码可能被其他用户读取。（来源：ArchWiki）

📝 本章小结

  • 配置文件支持多个 network 块，自动按优先级和信号强度选择最优网络
  • ctrl_interface 是启用 wpa_cli 交互控制的前提条件
  • 可通过 SIGHUP 或 wpa_cli reconfigure 热重载配置，无需重启进程

07场景一：WPA/WPA2-PSK 个人网络

📄 来源：ArchWiki — wpa_supplicant & wpa_supplicant.conf(5)

这是最常见的使用场景，适用于家庭或小型办公室 Wi-Fi，使用预共享密钥（PSK）进行认证。

原理

客户端和 AP 通过各自用密码+SSID计算 PMK，无需认证服务器。完成四次握手后使用 AES-CCMP（WPA2）或 TKIP（WPA）加密数据。

图5：WPA2-PSK 认证与数据流

方法一：使用 wpa_passphrase（官方推荐）

# 步骤1: 生成配置（SSID: MyHome, 密码: mysecret）
wpa_passphrase MyHome mysecret > /etc/wpa_supplicant/myhome.conf

# 生成的配置内容（密码已哈希，原文注释保留）：
# network={
#     ssid="MyHome"
#     #psk="mysecret"      ← 注释中保留明文（可手动删除）
#     psk=8a9bcdef...      ← 256-bit PBKDF2 哈希
# }

# 步骤2: 启动（后台）
wpa_supplicant -B -i wlan0 -c /etc/wpa_supplicant/myhome.conf

# 步骤3: 获取 IP（DHCP）
dhcpcd wlan0
# 或
dhclient wlan0

方法二：进程替换（快速测试）

# 一行命令（需 root shell）
wpa_supplicant -B -i wlan0 -c <(wpa_passphrase MyHome mysecret)

配置文件示例（官方格式）

ctrl_interface=DIR=/var/run/wpa_supplicant GROUP=wheel

network={
    ssid="MyHome"
    scan_ssid=1
    key_mgmt=WPA-PSK
    psk="very secret passphrase"   # 明文密码（建议用哈希）
}

📝 场景小结

  • PSK 模式无需服务器，配置简单，适合家庭/个人网络
  • 建议使用 wpa_passphrase 生成哈希存储，避免明文密码写入配置文件
  • 连接后需单独运行 dhcpcd/dhclient 获取 IP 地址

08场景二：WPA-Enterprise 企业网络

📄 来源：wpa_supplicant.conf(5) Quick Examples & hostap.epitest.fi — EAP Methods

原理

WPA-Enterprise 使用 IEEE 802.1X + EAP 框架。客户端通过 AP 代理与后端 RADIUS 服务器进行 EAP 交换，认证成功后 RADIUS 向 AP 派发 MSK（主会话密钥），AP 和客户端各自从 MSK 中截取 PMK，随后进入标准的四次握手。

图6：WPA-Enterprise 三方认证架构（Supplicant - AP - RADIUS）

EAP-TLS（证书认证）配置示例

官方 wpa_supplicant.conf(5) 提供的示例：使用 EAP-TLS 配合 WPA，仅允许 CCMP 和 TKIP 加密。

network={
    ssid="CorpNet"
    scan_ssid=1
    key_mgmt=WPA-EAP
    pairwise=CCMP TKIP
    group=CCMP TKIP
    eap=TLS
    identity="user@example.com"
    # EAP-TLS 需要客户端证书和 CA 证书
    ca_cert="/etc/cert/ca.pem"
    client_cert="/etc/cert/user.pem"
    private_key="/etc/cert/user.prv"
    private_key_passwd="password"
}

EAP-PEAP/MSCHAPv2（用户名+密码）配置示例

官方 wpa_supplicant.conf(5) 提供的 EAP-PEAP 示例，适合大多数企业内网和高校 Eduroam：

network={
    ssid="Eduroam"
    key_mgmt=WPA-EAP
    eap=PEAP
    identity="user@university.edu"
    password="foobar"
    ca_cert="/etc/cert/ca.pem"
    phase1="peaplabel=0"
    phase2="auth=MSCHAPV2"
}

💡 官方文档指出：EAP-TLS 需要服务器端和客户端双向证书；EAP-PEAP 和 EAP-TTLS 只需服务器端证书，客户端用用户名/密码认证。wpa_supplicant 支持 X.509 PEM 和 DER 格式证书。

支持的 EAP 方法（官方列表）

EAP-TLS EAP-PEAP/MSCHAPv2 EAP-PEAP/TLS EAP-TTLS/MSCHAPv2 EAP-TTLS/TLS EAP-SIM EAP-AKA EAP-FAST EAP-PSK EAP-PAX LEAP EAP-IKEv2 EAP-SAKE

📝 场景小结

  • Enterprise 模式每用户独立认证，一个账号泄露不影响其他用户的加密密钥
  • EAP-TLS 安全性最高（双向证书），EAP-PEAP 最常见（无需客户端证书）
  • 所有证书路径必须使用绝对路径；PKCS#12 格式需先转换为 PEM/DER

09场景三：开放网络（无加密）

📄 来源：ArchWiki — wpa_supplicant Configuration

连接无密码的公开 Wi-Fi（如咖啡馆、机场热点）时，无需加密协商，但 wpa_supplicant 仍负责处理 IEEE 802.11 关联过程。

原理

设置 key_mgmt=NONE 后，wpa_supplicant 跳过 WPA/WPA2 协商，直接完成 802.11 开放认证（Open Authentication）和关联（Association），不进行任何加密密钥协商。数据以明文传输。

配置示例

# 开放网络（无密码，如公共 WiFi）
network={
    ssid="CoffeeShop_Free"
    key_mgmt=NONE
}

# 作为最低优先级的回退网络（ArchWiki 示例）
network={
    key_mgmt=NONE
    priority=-999   # 最低优先级，其他网络不可用时才连接
}

通过 wpa_cli 连接开放网络

# 根据 ArchWiki 说明：无密码网络须显式设置 key_mgmt
> add_network
0
> set_network 0 ssid "CoffeeShop_Free"
> set_network 0 key_mgmt NONE
> enable_network 0

🚨 安全警告：开放网络不进行任何加密，所有流量对网络中任何人可见。强烈建议在开放网络上使用 VPN 或 HTTPS-only 策略。

📝 场景小结

  • 开放网络用 key_mgmt=NONE，wpa_supplicant 仅完成 802.11 关联，不协商密钥
  • 可设置极低 priority 作为"最后手段"回退网络
  • 开放网络下数据完全明文，务必使用上层加密（VPN/TLS）保护

10场景四：WPA3-SAE 新一代安全

📄 来源：ArchWiki — Connections to pure WPA3-SAE access points

WPA3 于 2018 年由 Wi-Fi Alliance 发布，引入 SAE（Simultaneous Authentication of Equals，同等认证）替代 PSK，彻底解决了 WPA2-PSK 的离线字典攻击弱点并提供前向保密。

SAE 与 PSK 的根本区别

特性WPA2-PSKWPA3-SAE
密钥交换算法PBKDF2（静态派生）Dragonfly（Diffie-Hellman 变体）
PMK 派生密码+SSID 静态计算每次连接动态协商
离线字典攻击❌ 可行（抓取握手包后离线爆破）✅ 防御（需在线交互）
前向保密❌ 无（密码泄露则历史流量可解密）✅ 有（临时密钥，互相独立）
管理帧保护 MFP可选必须（ieee80211w=2）

纯 WPA3 网络配置

# ArchWiki 官方示例：纯 WPA3-SAE 接入点
network={
    ssid="MySecureNet"
    key_mgmt=SAE
    sae_password="the.literal.wifi.password"
    ieee80211w=2   # MFP 必须（WPA3 强制要求）
}

混合 WPA2-PSK/WPA3-SAE 网络配置

# ArchWiki 示例：兼容 WPA2/WPA3 混合模式 AP
network={
    ssid="MyNet"
    key_mgmt=WPA-PSK-SHA256
    psk="mypassword"
    ieee80211w=2
}

💡 部分 Intel Wi-Fi 6 网卡在全局配置（非 network 块）中还需添加 sae_pwe=1，以启用 H2E（Hash to Element）方式的 SAE 密码派生。（来源：ArchWiki）

图7：SAE 握手 vs 传统 PSK 握手的安全性对比

📝 场景小结

  • WPA3-SAE 使用 Dragonfly 密钥交换，PMK 每次连接都不同，根本上消除了离线字典攻击
  • 必须设置 ieee80211w=2（管理帧保护），这是 WPA3 强制要求
  • 混合 WPA2/WPA3 模式用 key_mgmt=WPA-PSK-SHA256 保持向下兼容

🔬 第一性原理

WPA2-PSK 的根本缺陷：PMK 是密码的确定性函数，且握手过程完全公开，攻击者可以离线枚举所有密码。SAE 的解决方案是将密钥协商变为交互式零知识证明——必须在线参与，使得离线攻击失去意义。

11场景五：隐藏 SSID 网络

📄 来源：ArchWiki — Tip: scan_ssid for hidden SSID

原理

隐藏 SSID 的 AP 不在 Beacon 帧和 Probe Response 中广播 SSID。客户端无法通过被动扫描（passive scan）发现它，必须主动发送携带目标 SSID 的 Probe Request（主动扫描，active scan）才能找到该 AP。

图8：被动扫描与主动扫描对比

配置示例

network={
    ssid="HiddenNetwork"
    scan_ssid=1          # 关键：启用主动扫描以发现隐藏 SSID
    key_mgmt=WPA-PSK
    psk="mypassword"
}

⚠️ 官方 WPS 文档明确指出："Hidden SSIDs are explicitly disallowed with WPS (any version of WPS), the protocol just does not work with hidden SSIDs." 隐藏 SSID 与 WPS 不兼容，请勿同时使用。

📝 场景小结

  • scan_ssid=1 使 wpa_supplicant 发送携带 SSID 的定向 Probe Request 来发现隐藏网络
  • 隐藏 SSID 并不是强安全机制，仅增加一点发现难度，仍需配合 WPA2/WPA3 加密
  • 隐藏 SSID 与 WPS 协议不兼容

12场景六：多接口管理

📄 来源：wpa_supplicant(8) man page — EXAMPLES

原理

官方文档说明：wpa_supplicant 可以通过两种方式管理多个接口：为每个接口单独运行一个进程，或在命令行用 -N 参数分隔在一个进程中管理多个接口。

官方示例：单进程管理两个接口

# 官方 man page 原文示例：同时管理 wlan0 和 ath0
wpa_supplicant \
    -c wpa1.conf -i wlan0 -D nl80211 -N \
    -c wpa2.conf -i ath0  -D wext

多进程方式

# 接口一：Intel 网卡
wpa_supplicant -B -i wlan0 -D nl80211 \
    -c /etc/wpa_supplicant/wlan0.conf

# 接口二：USB 无线网卡
wpa_supplicant -B -i wlan1 -D nl80211,wext \
    -c /etc/wpa_supplicant/wlan1.conf

官方说明："If the specific driver wrapper is not known beforehand, it is possible to specify multiple comma separated driver wrappers on the command line. wpa_supplicant will use the first driver wrapper that is able to initialize the interface."

💡 ArchWiki 补充：多个逗号分隔的驱动（如 -Dnl80211,wext）可自动适配不同硬件，特别适合热插拔 USB 无线设备场景。

📝 场景小结

  • -N 参数允许单进程管理多接口，每组接口参数互相独立
  • 多进程方式更易于单独管理和重启特定接口
  • 逗号分隔驱动（nl80211,wext）可自动选择最佳驱动，适合混合硬件环境

13场景七：有线 802.1X 认证

📄 来源：ArchWiki — 802.1x/radius for wired adapter

原理

IEEE 802.1X 不仅适用于无线网络，也可用于有线以太网的端口级访问控制。交换机（Authenticator）阻断未认证设备的流量，客户端（Supplicant）需完成 EAP 认证后才能访问网络。wpa_supplicant 通过 wired 驱动支持此场景，常见于企业有线内网准入控制（NAC）。

图9：有线 802.1X 网络准入控制架构

配置示例（ArchWiki 原文）

# /etc/wpa_supplicant/wpa_supplicant-wired-eth0.conf
ctrl_interface=/run/wpa_supplicant
ap_scan=0          # 有线模式：禁用 AP 扫描

network={
    key_mgmt=IEEE8021X
    eap=PEAP
    identity="username"
    password="userpassword"
    phase2="autheap=MSCHAPV2"
}

启动命令

# 先将网卡 down（ArchWiki 要求）
ip link set eth0 down

# 启动（指定 wired 驱动）
wpa_supplicant -B -D wired -i eth0 \
    -c /etc/wpa_supplicant/wpa_supplicant-wired-eth0.conf

# 或使用 systemd 服务
systemctl enable wpa_supplicant-wired@eth0.service
systemctl start  wpa_supplicant-wired@eth0.service

💡 ArchWiki 提示：若想用于无线 802.1X（非有线），将 IEEE8021X 改为 WPA-EAP，去掉 ap_scan=0 即可。密码建议使用 NT-Hash 存储：password=hash:<hashvalue>

密码哈希生成（ArchWiki 方法）

# 生成 NT-Hash（避免明文存储）
echo -n "密码" | iconv -t utf16le | openssl dgst -md4 -provider legacy

📝 场景小结

  • 有线 802.1X 需设置 ap_scan=0（无需 AP 扫描）和 -D wired（有线驱动）
  • key_mgmt=IEEE8021X 用于有线，WPA-EAP 用于无线企业认证
  • 启动前需先 ip link set <iface> down，让 wpa_supplicant 接管接口

15进阶：systemd 开机自启

📄 来源：ArchWiki — At boot (systemd)

17进阶：漫游与 PMKSA 缓存

📄 来源：ArchWiki — Roaming & kernel.org — RSN preauthentication

漫游原理

当 Wi-Fi 覆盖区域有多个 AP 时，wpa_supplicant 负责在 AP 之间切换（漫游）。切换需要重新完成关联和四次握手，会产生短暂中断。ArchWiki 指出："默认配置下，wpa_supplicant 对漫游相当保守：只有在与当前 AP 彻底断开时才重新扫描。"

背景扫描（bgscan）主动漫游

# 全局配置（在 network 块之外）
bgscan="simple:30:-70:3600"
#        ↑格式: 模块:短间隔:信号阈值:长间隔
# 解读：信号低于 -70dBm 时每 30 秒扫描一次
#       信号好时每 3600 秒（1小时）扫描一次

PMKSA 缓存与 RSN 预认证

图10：RSN 预认证加速漫游切换

OKC（机会性密钥缓存）配置

# kernel.org 文档说明：
# OKC=1 让同 SSID 的 AP 共享 PMK，进一步加速漫游

network={
    ssid="CorpNet"
    key_mgmt=WPA-EAP
    eap=PEAP
    identity="user@corp.com"
    password="password"
    phase2="auth=MSCHAPV2"
    okc=1       # 开启机会性密钥缓存
    bgscan="simple:30:-70:3600"   # 主动背景扫描
}

kernel.org 官方文档说明：wpa_supplicant 对 WPA2 网络默认启用 RSN 预认证。PMKSA 候选列表由 rsn_preauth_scan_result() 在每次扫描完成后更新，要求候选 AP 满足：

  • 不能是当前已关联的 AP（BSSID 不同）
  • SSID 相同
  • AP 必须在 IE（信息元素）中声明支持预认证能力，或开启了 OKC

📝 本章小结

  • bgscan 让 wpa_supplicant 主动监控信号并扫描，而非等待断开才漫游
  • PMKSA 缓存保存 PMK，漫游时跳过耗时的 EAP 全量认证，直接进行四次握手
  • RSN 预认证 + OKC 是企业 Wi-Fi 漫游优化的标准方案

🔬 第一性原理

漫游延迟的根本来源是认证时间（EAP 往返数百毫秒）而非关联时间。PMKSA 缓存的本质是用空间（存储 PMK）换时间（跳过 EAP），将漫游切换从"冷启动"变为"热切换"，是典型的以预计算换实时延迟的工程优化。
