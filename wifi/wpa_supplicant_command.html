第 1 章 · 基础篇

概述与背景

wpa_supplicant 是 Wi-Fi Protected Access（WPA）客户端的实现，即运行在客户端设备上的 请求方（Supplicant） 组件。它同时实现了 IEEE 802.1X 请求方协议，可对接企业级认证服务器。

来源：Debian Manpages — wpa_supplicant(8)

官方定义（原文）

"wpa_supplicant is an implementation of the WPA Supplicant component, i.e., the part that runs in the client stations. It implements WPA key negotiation with a WPA Authenticator and EAP authentication with Authentication Server. In addition, it controls the roaming and IEEE 802.11 authentication/association of the wireless LAN driver."

— Arch Linux Man Page — wpa_supplicant(8)

为什么需要 wpa_supplicant？

无线网络与有线网络最根本的区别在于：攻击者无需物理接触网络设备就能被动监听所有帧，或非授权接入网络。早期的 WEP 加密被证明存在严重缺陷，IEEE 802.11i（WPA2）引入了基于 AES-CCMP 的加密机制，才能真正称为安全。wpa_supplicant 正是在客户端实现这套安全机制的守护进程。

历史演进

协议标准主要缺陷wpa_supplicant 支持

📌 本章小结

wpa_supplicant 解决的核心问题是：在无线介质的开放物理环境中，如何让客户端安全地向接入点证明自己的身份，并协商出只有双方知晓的加密密钥。这是整个无线安全体系的逻辑起点。

第一性原理：无线信道是广播信道，任何人都能收听。因此必须在链路层建立一个"只有合法双方才能推导出"的对称密钥，且这个密钥永远不能在信道上明文传输。wpa_supplicant 的所有机制都服务于这一基本约束。

第 2 章 · 基础篇

系统架构原理

wpa_supplicant 被设计为一个后台守护进程（daemon），通过控制接口（ctrl_interface）与前端程序通信，并通过驱动接口与内核无线驱动交互。

官方组件定义

根据官方 man page（Debian），wpa_supplicant 系统由以下组件构成：

组件角色说明
wpa_supplicant.conf配置文件描述用户希望计算机连接的所有网络及安全策略
wpa_supplicant核心守护进程直接与网络接口交互的主程序
wpa_cli命令行前端为守护进程功能提供高级命令行接口
wpa_passphrase密钥生成工具用于生成含加密口令的 wpa_supplicant.conf

📌 本章小结

wpa_supplicant 采用前后端分离架构：守护进程掌管所有低级无线操作，前端通过 Unix socket 发出高级命令。这种设计使得图形界面、命令行工具、NetworkManager 等可以共用同一个后端引擎，且可以热插拔前端而不影响连接状态。

第 3 章 · 基础篇

核心组件详解

wpa_supplicant.conf 配置文件

配置文件列出所有可接受的网络和安全策略。wpa_supplicant 会自动选择最优网络，选择依据是：配置文件中的网络块顺序、网络安全级别（WPA/WPA2 优先）、信号强度。

来源：man.cx — wpa_supplicant.conf(5)

配置文件路径说明

  • 所有文件路径必须使用绝对路径（因为守护进程可能更改工作目录）
  • 通过发送 SIGHUP 信号（killall -HUP wpa_supplicant）可热重载配置
  • 也可用 wpa_cli reconfigure 触发重载

ctrl_interface — 控制接口

# 允许 wheel 组用户通过 wpa_cli 控制连接
ctrl_interface=DIR=/var/run/wpa_supplicant GROUP=wheel
update_config=1

wpa_passphrase 工具

将明文密码转换为 256-bit PSK（PBKDF2 推导），避免配置文件中存储明文：

# 生成 PSK 并追加到配置文件
$ wpa_passphrase MySSID MyPassword >> /etc/wpa_supplicant/wpa_supplicant.conf

# 输出示例
network={
    ssid="MySSID"
    #psk="MyPassword"
    psk=8a9da8...  # 64位十六进制 PSK
}

wpa_cli 控制工具

wpa_cli 提供交互式和批量模式两种使用方式，通过 Unix socket 与守护进程通信：

# 交互模式
$ wpa_cli -i wlan0

# 批量执行单条命令
$ wpa_cli -i wlan0 status

# 常用命令
> status          # 获取当前 WPA/EAPOL/EAP 状态
> scan            # 触发 BSS 扫描
> scan_results    # 显示扫描结果
> list_networks   # 列出已配置网络
> reconnect       # 断开后重连
> reconfigure     # 重新读取配置文件
> pmksa           # 显示 PMKSA 缓存

来源：linux.die.net — wpa_cli(8)

📌 本章小结

三个核心工具形成完整闭环：wpa_passphrase 安全生成密钥 → wpa_supplicant.conf 描述网络策略 → wpa_supplicant 守护进程执行 → wpa_cli 运行时管理。配置文件中使用 PSK 哈希而非明文密码，是安全最佳实践。

驱动接口层

wpa_supplicant 通过驱动抽象层与内核通信，支持多种驱动后端。具体使用哪个驱动在编译时确定，运行时通过 -D 参数指定。

macsec_linux MACsec MACsec 加密以太网 特定
none 无 仅 RADIUS/WPS ER 特殊

nl80211 与 wext 的技术对比

驱动自动检测

官方文档说明：如果不确定用哪个驱动，可以用逗号分隔列出多个，wpa_supplicant 会使用第一个能成功初始化接口的驱动：

# 自动回退：先尝试 nl80211，失败则用 wext
$ wpa_supplicant -Dnl80211,wext -c/etc/wpa_supplicant.conf -iwlan0

来源：Arch Linux Man Page — wpa_supplicant(8) SUPPORTED DRIVERS

📌 本章小结

nl80211 是当前 Linux 无线网络的标准接口，基于 Netlink 协议族，功能完整、性能更好。wext 是遗留接口，逐渐被弃用。对于所有现代系统，应优先使用 -D nl80211。驱动层的抽象使 wpa_supplicant 能跨不同硬件保持一致的行为。

第 5 章 · 原理篇

WPA 关联完整流程

以下是官方 man page 定义的使用 WPA 关联接入点的完整步骤：

来源：Arch Linux Man Page — wpa_supplicant(8) OVERVIEW

1
扫描邻近 BSS
wpa_supplicant 请求内核驱动扫描附近的基本服务集（BSS）。驱动通过 nl80211 发送探测请求帧（Probe Request）并收集响应。
2
BSS 选择
wpa_supplicant 根据配置文件中的网络块顺序、安全级别（WPA2 > WPA > Open）、信号强度（RSSI）综合评分，选择最优 BSS。
3
关联请求
wpa_supplicant 请求内核驱动与所选 BSS 进行 IEEE 802.11 认证和关联（Authentication + Association）。此时尚未建立加密通道。
4
EAP 认证（仅 WPA-EAP）
集成的 IEEE 802.1X 请求方与认证服务器（由 AP 代理）完成 EAP 认证。支持 EAP-TLS、EAP-PEAP、EAP-TTLS 等多种方法。
5
获取主密钥（PMK）
WPA-EAP：从 IEEE 802.1X 请求方获取 MSK（主会话密钥），派生出 PMK。
WPA-PSK：直接用 PSK 作为主会话密钥（PMK = PBKDF2(passphrase, SSID, 4096, 32)）。
6
四次握手 + 组密钥握手
wpa_supplicant 与认证方（AP）完成 WPA 四次握手（4-Way Handshake）和组密钥握手（Group Key Handshake），派生单播和组播加密密钥。
7
安装加密密钥
wpa_supplicant 将单播密钥（PTK）和广播密钥（GTK）配置到驱动，启用链路层加密。
8
正常数据传输
加密通道建立完成，上层协议（如 DHCP）可以开始运行，正常数据包可以收发。

📌 本章小结

关联流程体现了"认证在前、加密在后"的分层设计。DHCP 等上层协议只有在 wpa_supplicant 完成加密密钥配置后才能运行。对 WPA-PSK，整个过程无需外部认证服务器；对 WPA-EAP，认证服务器参与但只影响 PMK 来源，之后的四次握手流程完全相同。

第 6 章 · 原理篇

四次握手（4-Way Handshake）详解

四次握手是 WPA/WPA2 安全机制的核心，其目的是让双方在不传输密钥本身的前提下，通过数学证明双方都持有相同的 PMK，并协商出会话密钥。

密钥层次体系

四条消息详解

M1：AP → Client（ANonce）

AP 发送 32 字节随机数（ANonce）。此消息无加密保护——nonce 的作用是提供"新鲜性"以防重放攻击，而非保密。

客户端收到 M1 后，已具备推导 PTK 所需的所有参数：ANonce + 自己生成的 SNonce + AP/Client MAC + PMK。

M2：Client → AP（SNonce + MIC）

客户端生成 SNonce，推导 PTK，用 KCK 计算 MIC（消息完整性码），连同 RSN 信息元素发送给 AP。

MIC 是关键：这是客户端在证明"我拥有正确的 PMK"，AP 此时也可推导 PTK 并验证 MIC。若 MIC 不匹配 = 密码错误，握手失败。

M3：AP → Client（GTK + MIC，Install PTK）

AP 用 KEK 加密 GTK 后发送，并设置 Install 标志（通知客户端安装 PTK）。AP 的 MIC 是反向认证：证明 AP 也持有正确的 PMK。

M4：Client → AP（ACK）

客户端确认已安装 PTK 和 GTK。AP 收到并验证 M4 后，双方安装密钥，开始用 TK 加密数据帧。

⚠ 安全要点

PMK 本身从不在空口传输。握手只交换随机数（nonce）和 MIC。离线破解攻击需要截获握手包后对 MIC 进行字典/暴力攻击，这也是为什么弱密码是 WPA2-PSK 的主要安全威胁。

📌 本章小结

四次握手的第一性原理是零知识证明思想的工程实现：双方通过交换随机数和哈希值，证明彼此持有相同的秘密（PMK），而无需公开这个秘密。PTK = PRF(PMK, ANonce, SNonce, MAC_AP, MAC_STA) 的设计保证每次会话密钥都是独一无二的，即使 PMK 相同（防止重放）。

第 7 章 · 使用场景

场景一：WPA-PSK 家用/个人网络

WPA-Personal 最常见的使用场景，路由器和客户端共享同一个预共享密钥（PSK）。

背后原理

PSK 由用户密码通过 PBKDF2-HMAC-SHA1（4096 次迭代，SSID 为盐）派生为 256-bit PMK，无需认证服务器。安全强度取决于密码强度。

步骤 1：生成 PSK

# 使用 wpa_passphrase 将密码转换为 PSK（推荐，避免明文存储）
$ wpa_passphrase "MyHomeWifi" "my_secret_pass" | sudo tee /etc/wpa_supplicant/wpa_supplicant.conf

步骤 2：配置文件

# /etc/wpa_supplicant/wpa_supplicant.conf
# 来源示例：man.cx/wpa_supplicant.conf(5)
ctrl_interface=DIR=/var/run/wpa_supplicant GROUP=netdev
update_config=1

network={
    ssid="home"
    scan_ssid=1
    key_mgmt=WPA-PSK
    psk="very secret passphrase"
    # 或使用哈希形式：
    # psk=8a9da83c0b30...
}

来源：man.cx — wpa_supplicant.conf(5) QUICK EXAMPLES

步骤 3：启动守护进程

# 前台调试模式（推荐初次使用）
$ sudo wpa_supplicant -iwlan0 -c/etc/wpa_supplicant.conf -d

# 后台运行（生产模式）
$ sudo wpa_supplicant -B -iwlan0 -c/etc/wpa_supplicant.conf

# 获取 IP 地址
$ sudo dhclient wlan0

来源：Debian Manpages — wpa_supplicant(8) QUICK START

步骤 4：验证连接状态

$ wpa_cli -i wlan0 status
bssid=aa:bb:cc:dd:ee:ff
freq=2412
ssid=home
id=0
mode=station
pairwise_cipher=CCMP
group_cipher=CCMP
key_mgmt=WPA2-PSK
wpa_state=COMPLETED
多网络优先级

可在配置文件中定义多个 network 块，wpa_supplicant 自动按安全级别和信号选择最优，也可用 priority 字段手动指定优先级（数字越大优先级越高）。

📌 本章小结

WPA-PSK 场景下，所有安全性依赖密码强度。使用 wpa_passphrase 将密码哈希后存储是最佳实践。守护进程启动后，DHCP 等上层操作才能进行。-B 标志使进程后台运行，-d 用于调试。

第 8 章 · 使用场景

场景二：WPA-Enterprise 企业级认证

WPA-Enterprise 通过 IEEE 802.1X + EAP 与 RADIUS 服务器进行用户级认证，每个用户有独立的凭证，PMK 由认证服务器动态生成。

背后原理

与 WPA-PSK 的核心区别：PMK 不是从固定密码派生，而是由 RADIUS 服务器在 EAP 认证成功后动态生成并分发给 AP 和客户端（MSK → PMK）。这意味着每次连接都可以有不同的 PMK，大幅提升安全性。

EAP-TLS（证书双向认证）

最安全的 EAP 方式，客户端和服务器均需证书，无需密码：

# 工作网络：EAP-TLS + WPA2 — 仅 CCMP 和 TKIP
# 来源：man.cx/wpa_supplicant.conf(5) QUICK EXAMPLES
network={
    ssid="work"
    scan_ssid=1
    key_mgmt=WPA-EAP
    pairwise=CCMP TKIP
    group=CCMP TKIP
    eap=TLS
    identity="user@example.com"
    ca_cert="/etc/cert/ca.pem"
    client_cert="/etc/cert/user.pem"
    private_key="/etc/cert/user.prv"
    private_key_passwd="password"
}

EAP-PEAP/MSCHAPv2（用户名+密码，常见于企业）

# EAP-PEAP with MSCHAPv2 — 来源：man.cx/wpa_supplicant.conf(5)
network={
    ssid="example"
    scan_ssid=1
    key_mgmt=WPA-EAP
    eap=PEAP
    identity="user@example.com"
    password="foobar"
    ca_cert="/etc/cert/ca.pem"
    phase1="peaplabel=0"
    phase2="auth=MSCHAPV2"
}

EAP-TTLS/MD5（匿名外层身份）

# EAP-TTLS with anonymous outer identity
# 真实身份仅在 TLS 隧道内传输，外层使用匿名身份
network={
    ssid="example"
    scan_ssid=1
    key_mgmt=WPA-EAP
    eap=TTLS
    identity="user@example.com"
    anonymous_identity="anonymous@example.com"
    password="foobar"
    ca_cert="/etc/cert/ca.pem"
    phase2="auth=MD5"
}

来源：man.cx — wpa_supplicant.conf(5) QUICK EXAMPLES

证书格式转换

# PKCS#12/PFX 转 PEM 格式（wpa_supplicant 支持 PEM/DER）
$ openssl pkcs12 -in example.pfx -out user.pem -clcerts
$ openssl pkcs12 -in example.pfx -out ca.pem -cacerts -nokeys
EAP 方法客户端证书服务端证书典型场景
EAP-TLS✓ 必需✓ 必需高安全企业
EAP-PEAP/MSCHAPv2✗✓ 必需eduroam、企业 Wi-Fi
EAP-TTLS✗✓ 必需用户名密码认证
EAP-SIM / EAP-AKASIM 卡✓运营商热点

📌 本章小结

企业模式下每个用户有独立身份和动态 PMK，吊销单个用户凭证不影响其他用户，且单个用户的流量不能被其他用户解密（即使他们在同一 AP 下）。这是企业 Wi-Fi 相比家用 PSK 的根本安全优势——密钥材料不再是共享的静态秘密，而是每次会话动态生成。

第 9 章 · 使用场景

场景三：WPA3 / SAE（同步认证等值）

WPA3 使用 SAE（Simultaneous Authentication of Equals，同步认证等值）替代 WPA2-PSK 的预共享密钥交换，基于 Dragonfly 密钥交换协议，提供前向保密。

背后原理

SAE 使用椭圆曲线 Diffie-Hellman 风格的密钥交换（Dragonfly），双方各自随机生成临时密钥并推导 PMK，密码在任何时候都不直接用于计算最终密钥，而是作为输入生成椭圆曲线上的一个点（Password Element，PWE）。即使攻击者截获到交换帧，也无法进行离线字典攻击（前向保密）。

ieee80211w=2 启用管理帧保护（MFP），这是 WPA3 要求的必选项。

# WPA3-Personal (SAE)
# 来源：Void Linux Handbook / wpa_supplicant.conf 文档
network={
    ssid="MYSSID"
    key_mgmt=SAE
    sae_password="YOUR AP PASSWORD"
    ieee80211w=2          # 管理帧保护（必选）
}

来源：Void Linux Handbook — wpa_supplicant

WPA2/WPA3 混合模式

# 同时支持 WPA2-PSK 和 WPA3-SAE（过渡模式）
network={
    ssid="MYSSID"
    key_mgmt=SAE WPA-PSK
    psk="password"
    sae_password="password"
    ieee80211w=1          # 可选管理帧保护
}

📌 本章小结

SAE 的核心创新是：密码不再直接作为 PMK，而是通过椭圆曲线运算成为 PMK 推导的起点（PWE），且每次握手引入新的随机性。即使攻击者录制了握手帧，也无法离线猜测密码（因为没有足够的数学依据）。这是 WPA3 相比 WPA2-PSK 的根本性安全提升。

第 10 章 · 使用场景

场景四：连接隐藏 SSID 网络

部分 AP 不广播 SSID（隐藏网络），需要客户端发送含目标 SSID 的定向探测请求（Directed Probe Request）才能发现。

背后原理

通常 AP 在信标帧（Beacon）中广播 SSID，wpa_supplicant 被动扫描即可发现。隐藏 SSID 的 AP 发空 SSID 信标，只有收到含正确 SSID 的 Probe Request 才会回应。scan_ssid=1 告知 wpa_supplicant 为该网络发送定向 Probe Request。

来源：MIT/FreeBSD — wpa_supplicant.conf 官方示例文件（scan_ssid 字段说明）

# 连接隐藏 SSID 网络
network={
    ssid="HiddenNetwork"
    scan_ssid=1          # 使用 SSID 特定的 Probe Request 帧
    key_mgmt=WPA-PSK
    psk="my_secret"
}
⚠ 隐藏 SSID 并非真正的安全措施

隐藏 SSID 只是防止普通用户看到网络名，任何被动扫描工具（如 airodump-ng）都能通过监听客户端的 Probe Request 发现真实 SSID。此外，scan_ssid=1 会增加扫描延迟（需要额外的 Probe 交换），不建议在非必要场景启用。

按 BSSID 锁定特定 AP

在多 AP 同 SSID 的网络中，可以锁定到特定 AP（BSSID）：

network={
    ssid="MyNetwork"
    bssid="aa:bb:cc:dd:ee:ff"  # 只与此 AP 关联
    key_mgmt=WPA-PSK
    psk="password"
}

📌 本章小结

scan_ssid=1 是 wpa_supplicant 应对隐藏网络的机制，它改变了扫描阶段的探测帧类型（广播 → 定向）。但安全本质上不依赖 SSID 隐藏，而依赖加密协议本身。

第 11 章 · 使用场景

场景五：开放网络与 WEP

已废弃 WEP 和开放网络（无加密）配置说明，仅供历史参考和兼容旧设备使用。

开放网络（无加密）

# 连接无加密开放网络（如酒店/咖啡馆免费 Wi-Fi）
network={
    ssid="FreeWifi"
    key_mgmt=NONE
}

WEP 网络（不推荐，已被攻破）

# WEP 配置 — 来源：Void Linux Handbook
# ⚠ WEP 已于 2004 年被证明不安全，应避免使用
network={
    ssid="MYSSID"
    key_mgmt=NONE
    wep_key0="YOUR AP WEP KEY"
    wep_tx_keyidx=0
    auth_alg=SHARED
}

来源：Void Linux Handbook — wpa_supplicant

⚠ 安全警告

WEP（Wired Equivalent Privacy）使用 RC4 流密码，存在 IV 重用等根本性缺陷，数秒内可被破解。IEEE 802.11i 已将其废弃，wpa_supplicant 编译时默认关闭 WEP 支持（需要 wep USE flag）。生产环境请勿使用。

📌 本章小结

开放网络下 wpa_supplicant 仅完成 802.11 关联，不执行任何加密/认证。数据帧明文传输，极易被截获。对于需要在开放网络上保护隐私的场景，应配合 VPN 使用，而非依赖无线层加密。

第 12 章 · 使用场景

场景六：有线 802.1X 认证

wpa_supplicant 不仅支持无线网络，也支持有线以太网的 IEEE 802.1X 认证，常见于企业内网准入控制（NAC）。

背后原理

使用 -D wired 驱动，ap_scan=0 禁用无线扫描（因为有线无需 BSS 扫描），直接通过以太网帧进行 EAPOL 交换与认证服务器通信。

来源：man.cx — wpa_supplicant.conf(5) QUICK EXAMPLES（示例 6）

# 有线 802.1X 配置 — 使用 wired 或 roboswitch 驱动
ctrl_interface=DIR=/var/run/wpa_supplicant GROUP=netdev
ap_scan=0   # 禁用无线 AP 扫描

network={
    key_mgmt=IEEE8021X
    eap=MD5
    identity="user"
    password="password"
    eapol_flags=0   # 有线模式，无需动态 WEP 密钥
}

有线 EAP-TLS（证书认证）

# 有线 EAP-TLS — 来源：Gentoo Wiki
ap_scan=0
network={
    ssid="1x-test"
    key_mgmt=IEEE8021X
    eap=TLS
    identity="user@example.com"
    ca_cert="/etc/cert/ca.pem"
    client_cert="/etc/cert/user.pem"
    private_key="/etc/cert/user.prv"
    private_key_passwd="password"
    eapol_flags=3
}

启动命令

# 启动有线 802.1X 认证（Arch Linux Wiki 建议先将接口置为 DOWN）
$ sudo ip link set eth0 down
$ sudo wpa_supplicant -B -Dwired -ieth0 -c/etc/wpa_supplicant-wired.conf

来源：Arch Wiki — wpa_supplicant

📌 本章小结

有线 802.1X 是企业网络准入控制（NAC）的常见实现。wpa_supplicant 通过 -D wired + ap_scan=0 跳过无线特有的扫描/关联流程，直接在以太网上运行 EAPOL 认证协议，与无线 WPA-Enterprise 共用相同的 EAP 引擎。

第 13 章 · 使用场景

场景七：多接口并发管理

wpa_supplicant 可以单进程管理多个无线接口，通过 -N 参数分隔每个接口的配置。

背后原理

每个接口对应独立的配置文件和驱动，守护进程内部为每个接口维护独立的状态机，互不影响。这比为每个接口运行独立进程更高效。

# 同时管理两个无线接口 — 来源：Debian Manpages wpa_supplicant(8)
$ wpa_supplicant \
    -c wpa1.conf -i wlan0 -D nl80211 -N \
    -c wpa2.conf -i ath0  -D wext

来源：Debian Manpages — wpa_supplicant(8) ARCHITECTURE

全局控制接口

# 使用全局 ctrl_interface，接口定义可省略
$ wpa_supplicant -g /run/wpa_supplicant/global

# 通过 wpa_cli 动态添加接口
$ wpa_cli -g /run/wpa_supplicant/global interface_add wlan0 /etc/wpa.conf nl80211

systemd 服务管理（单接口）

# 启用/启动 systemd 服务
$ sudo systemctl enable wpa_supplicant@wlan0.service
$ sudo systemctl start  wpa_supplicant@wlan0.service

# 查看状态
$ systemctl status wpa_supplicant@wlan0.service

📌 本章小结

多接口管理体现了 wpa_supplicant 的"一个后端，多个实例"设计。-N 参数优雅地解决了多接口场景，而全局控制接口则允许运行时动态增减接口，适合嵌入式路由器、网络管理器等复杂场景。

事件驱动脚本（wpa_cli -a）

wpa_cli 可以在守护模式下执行脚本，响应连接/断开事件：

#!/bin/sh — /etc/wpa_supplicant/action.sh
# 来源：Arch Wiki — wpa_supplicant
case "$2" in
    CONNECTED)
        notify-send "WPA supplicant: 连接已建立"
        dhclient "$1"
        ;;
    DISCONNECTED)
        notify-send "WPA supplicant: 连接已断开"
        ;;
esac
# 使用 -a 参数绑定事件脚本
$ wpa_cli -i wlan0 -a /etc/wpa_supplicant/action.sh

📌 本章小结

wpa_cli 的动态操作能力使 wpa_supplicant 可以在运行时"热配置"，无需重启守护进程。这是网络管理工具（如 NetworkManager）集成 wpa_supplicant 的核心机制：通过 Unix Socket 向守护进程发送命令，让用户无感知地完成网络切换。

第 15 章 · 进阶篇

漫游机制（Roaming）

当客户端处于有多个接入点的无线网络（同一 SSID，不同 BSSID）中时，wpa_supplicant 负责管理漫游行为。

来源：Arch Wiki — wpa_supplicant Roaming

官方说明（Arch Wiki）

"When connected to a wireless network with multiple access points, wpa_supplicant is typically responsible for roaming between access points. Choosing a new access point requires wpa_supplicant to perform a scan of available networks, which causes a brief interruption in connectivity to the current access point while the wireless radio scans other frequencies."

默认漫游行为（保守策略）

官方文档指出：默认配置下，wpa_supplicant 只会在与当前 AP 的关联完全丢失时才重新扫描。这意味着客户端即使距离当前 AP 较远（信号弱），只要连接未断，也不会主动漫游到更近的 AP。

PMKSA 缓存（快速漫游）

WPA2/WPA3 支持 PMKSA（Pairwise Master Key Security Association）缓存，当漫游到曾经连接过的 AP 时，可以跳过完整的 EAP 认证，直接使用缓存的 PMK 进行四次握手，大幅减少重连延迟。

# 查看 PMKSA 缓存
$ wpa_cli -i wlan0 pmksa
Index / AA                / PMK-ID (缓存的认证结果)
1     aa:bb:cc:dd:ee:ff  xxx...  2024-01-01 00:00:00

BSS 选择优先级

📌 本章小结

wpa_supplicant 的漫游策略相对保守，以减少不必要的扫描（每次扫描需要暂时切换信道，会影响数据传输）。PMKSA 缓存是企业网络漫游速度优化的关键，避免了重复的 EAP 认证往返延迟。如需更激进的漫游行为，可配合 bgscan 插件实现后台扫描。

调试模式启动

# 输出到日志文件
$ sudo wpa_supplicant -c/etc/wpa_supplicant.conf -iwlan0 -d -f /tmp/wpa_debug.log

# 查看当前连接状态

$ wpa_cli -i wlan0 scan
$ wpa_cli -i wlan0 scan_results

常见问题排查

症状可能原因排查方法
守护进程立即退出接口不存在或驱动未加载ip link show 确认接口存在
wpa_state=SCANNING 卡住驱动问题或 SSID 不存在加 -d 看扫描结果，尝试 -D wext

📌 本章小结

调试 wpa_supplicant 最有效的方式是前台 + -d 输出，结合 wpa_cli status 实时监控状态机。状态机路径 DISCONNECTED → SCANNING → ASSOCIATING → 4WAY_HANDSHAKE → COMPLETED 可以精准定位问题出现在哪个阶段。

总结与第一性原理

全局知识图谱

🔑 第一性原理总结

从最根本的物理约束出发，理解 wpa_supplicant 的设计逻辑：

  1. 无线信道是广播介质 → 所有人都能收听 → 必须加密。
  2. 加密密钥不能在信道上传输 → 必须让双方独立推导出相同密钥 → PBKDF2（PSK）或 EAP（Enterprise）或 SAE（WPA3）。
  3. 密钥推导后需验证双方一致 → 四次握手通过 MIC（HMAC）进行零知识证明，PMK 永远不出现在信道上。
  4. 每次会话密钥需唯一 → ANonce + SNonce 引入随机性，即使 PMK 相同，每次 PTK 也不同 → 防止重放攻击。
  5. 加密完成后，上层协议（DHCP 等）才能运行 → 链路层安全是网络层的前提。

wpa_supplicant 的所有功能特性，都是这五条物理/数学约束在工程层面的具体实现。理解了这些约束，所有配置选项和行为逻辑就自然清晰了。
