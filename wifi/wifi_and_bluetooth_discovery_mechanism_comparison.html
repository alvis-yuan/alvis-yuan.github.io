§0 · 引言

引言与设计背景

在无线通信领域，"如何让设备相互发现"是一个基础而关键的问题。经典蓝牙、BLE 和 WiFi 三种技术，由于其物理层特性、应用场景和设计哲学的不同，发展出了截然不同的发现机制。

核心问题：无线设备发现本质是在共享射频介质中，让"询问者"找到"应答者"。三种技术面临相同挑战：效率、功耗、抗干扰、兼容性，但给出了不同答案。
三种无线技术发现机制全景对比

理解这三种机制的差异，有助于工程师在物联网、消费电子、工业控制等场景中做出正确的技术选型，也有助于理解无线协议设计背后的工程权衡。

参考规范：IEEE 802.11-2020 Standard；Bluetooth Core Specification v5.4；CWAP Official Study Guide
§1 · 经典蓝牙

经典蓝牙：Inquiry 查询扫描机制

经典蓝牙（BR/EDR）的设备发现基于 Inquiry（查询）子状态，使用通用查询访问码 GIAC 在 79 个跳频信道上广播 ID 包，被发现设备以 FHS 包应答。

1.1 物理层基础

经典蓝牙工作于 2.4 GHz ISM 频段，采用 FHSS（跳频扩频）技术，将频段划分为 79 个 1 MHz 宽的信道（2402 MHz ~ 2480 MHz），以 1600 跳/秒的速率（每时隙 625 µs）在信道间高速跳变，抵抗窄带干扰和多径衰落。

关键规范（Bluetooth Core Specification v5.4, Part B §2.6）：
访问码（Access Code）由设备地址的 LAP（Lower Address Part）字段派生。Inquiry 专用的 通用查询访问码 GIAC（General Inquiry Access Code） 使用保留 LAP 值 0x9E8B33，对应 DCI（Default Check Initialization）值 0x00。
一个共 64 个连续 LAP 值（0x9E8B00 ~ 0x9E8B3F）保留用于查询操作，任何蓝牙设备地址均不得使用这些值。

1.2 Inquiry 子状态（主动方）

发起 Inquiry 的设备（即"主设备"角色）进入 Inquiry 子状态。根据 Bluetooth Core Spec Part B §8.4 及实际观测：

  • 1
    从 32 个跳频序列（均匀分布于 79 个信道）中选取，该序列由 GIAC 的 28 位地址和当前时钟派生。
  • 2
    每 312.5 µs（半个时隙）在当前频率上发送一次 Inquiry ID 包，随后在下一时隙等待 FHS 应答。
  • 3
    跳频序列分为 A、B 两列（各 16 个频率），持续以伪随机方式切换，每 10 ms（16 个时隙）完成一轮。
  • 4
    Inquiry 总时长为 N × 1.28 s，默认推荐最少 10.24 s（N=8）以确保覆盖所有 Inquiry Scan 设备。
经典蓝牙 Inquiry 时序图（主设备视角）

1.3 Inquiry Scan 子状态（被动方）

希望被发现的设备进入 Inquiry Scan（查询扫描）子状态，根据 Bluetooth Core Spec Part C（GAP）及实测数据：

关键时间参数（Bluetooth Core Spec v5.4 / hcitool 文档）

• 设备至少每 2.56 秒（4096 时隙）进入一次 Inquiry Scan 状态
• 每次监听至少 10ms（16 时隙）
• 每 1.28 秒（2048 时隙）切换一个新的监听信道
• 监听信道与跳频序列均由 GIAC（0x9E8B33）对应的 32 频率集合派生

收到 ID 包的被扫描设备，以 FHS（Frequency Hop Synchronization）包作为 Inquiry Response 应答，其中包含设备的 BD_ADDR、时钟偏移、设备类别（CoD）等信息，供主设备识别。

参考来源：Bluetooth Core Spec v5.4 Part B §2.6 Hop selection； Part C GAP §4.1 Inquiry procedure； hcitool 文档（Inquiry Scan 参数）； Peterson et al., "Bluetooth Inquiry Time Characterization and Selection", IEEE 2006
§2 · 经典蓝牙

经典蓝牙：状态机与发现流程

经典蓝牙的发现过程涉及严格定义的状态机，设备在 Standby、Inquiry、Inquiry Scan 等子状态之间切换，形成主被动配合的发现协议。

经典蓝牙设备发现完整状态机与信号流
GIAC 与 DIAC（Bluetooth Core Spec Part C，GAP）：
• GIAC（General Inquiry Access Code）：用于通用发现，所有处于可发现模式的设备均应响应。
• DIAC（Dedicated IAC）：共 63 个，用于发现具有特定特征（如特定服务类别）的设备，各对应预定义的 LAP 值。
处于 General Discoverable Mode 的设备不应响应 LIAC（Limited IAC）查询。

根据 Peterson 等人 2006 年 IEEE 论文研究，使用 v1.1 规范默认 11.25ms 扫描窗口（每 1.28s 开启一次），单个发起方在 5.12 秒内可以 99% 概率发现范围内所有扫描中设备（而非规范推荐的 10.24s）。采用 v1.2 交错扫描（interlaced scan）可进一步缩短至 1.28 秒。

参考来源：Bluetooth Core Spec v5.4 Part C GAP §4.1； Peterson B. et al. "Bluetooth Inquiry Time Characterization and Selection" (2006)； WPG Bluetooth Device Discovery and Synchronization
§3 · BLE

BLE：广播与扫描机制

BLE（低功耗蓝牙）彻底革新了发现机制：外设（Peripheral）周期性广播 ADV 包，中心设备（Central）扫描侦听，无需同步跳频。其设计核心是极致的低功耗。

3.1 GAP 角色定义

根据 BLE 规范（Bluetooth Core Spec, Part C, GAP）以及 TI 等厂商实现文档，BLE 定义了 4 种 GAP 角色：

GAP 角色 说明 发现中的作用
Broadcaster 仅发送广播事件 类似 Beacon，只发不连
Observer 仅接收广播事件 被动扫描方
Peripheral 接受 LE 物理链路建立 广播方，可被连接
Central 发起物理连接建立 扫描方，发起连接

3.2 广播流程

BLE 外设以固定的 广播间隔（AdvInterval） 向外广播，每次广播事件在 3 个广播信道（37、38、39）上依次发送相同的 ADV 包：

  • 1
    广播间隔范围：20ms ~ 10.24s，步进 0.625ms；每次广播事件还会叠加 0~10ms 随机延迟，以降低不同设备间发生同步碰撞的概率。
  • 2
    外设在信道 37 → 38 → 39 上依次发送 ADV_IND（可连接、非定向广播）或其他类型的广播 PDU，每帧间隔 TIFS = 150µs。
  • 3
    Central 在每次扫描窗口（Scan Window）内监听指定信道，每过一个扫描间隔（Scan Interval）切换一次信道。
BLE 广播与扫描时序图

3.3 主动扫描 vs 被动扫描

来自 TI BLE 文档（Texas Instruments）：
当设备未连接时，可通过发送广播包（advertise）或扫描附近设备（scan）进行设备发现。
• 主动扫描（Active Scanning）：扫描方可向广播方发送 SCAN_REQ，请求额外 31 字节数据，广播方以 SCAN_RSP 应答。
• 被动扫描（Passive Scanning）：扫描方仅接收 ADV 数据，不发送任何请求。
参考来源：TI BLE Scanning and Advertising 文档； Argenox BLE Advertising Primer (2025)； Sensors MDPI: BLE Discovery Process (Bluetooth 5.0)； ESP-IDF BLE Device Discovery
§4 · BLE

BLE 广播信道设计背景

BLE 的 3 个广播信道（37、38、39）并非随意选取，而是精心设计以最大限度避免 WiFi 干扰，这体现了共存设计的工程智慧。

2.4 GHz 频段：BLE 广播信道与 WiFi 信道共存示意图
来自 Argenox BLE Advertising Primer（2025）及 Circuit Cellar / Nordic 文档：
BLE 的 40 个 1 MHz 信道（2402~2480 MHz）中，37、38、39 号（尽管编号连续，但频率并不相邻）专用于广播发现：
• 信道 37：2402 MHz（频段起始端）
• 信道 38：2426 MHz（位于 WiFi Ch1 与 Ch6 之间）
• 信道 39：2480 MHz（频段末端）
三个信道横跨整个 2.4 GHz 频段，间距较大。若任一信道被干扰（多为窄带干扰源），其余两个仍大概率可用，显著提升广播成功率。

Apple 公司推荐外设在所有 3 个广播信道上发送广播包，而不是节省功耗仅发 1~2 个信道，原因正是为了保障抗干扰冗余性。

参考来源：Argenox: BLE Advertising Primer (2025)； Nordic Developer Academy: BLE Advertising Process； Circuit Cellar: No Blues with Bluetooth Part 3
§5 · WiFi

WiFi 802.11：发现过程概述

IEEE 802.11 定义了两种网络发现方式：被动扫描（Passive Scanning）和主动扫描（Active Scanning），均基于管理帧（Management Frames）实现。

IEEE 802.11 标准是一个基础设施驱动的网络体系：AP（接入点）是网络的固定基础设施，STA（站点/客户端）需要找到 AP 才能接入。WiFi 的发现过程本质上是客户端发现 AP的过程，而非设备互相发现。

IEEE 802.11 发现机制两条路径

被动扫描（Passive Scanning）：STA 在每个信道上静默等待，侦听 AP 定期广播的 Beacon 帧，从中提取网络信息（SSID、能力、速率等）。

主动扫描（Active Scanning）：STA 在每个信道上主动广播 Probe Request 帧，AP 收到后以 Probe Response 帧应答，提供完整的 BSS 能力信息。这种方法效率更高，是大多数操作系统的默认行为。

§6 · WiFi

被动扫描：Beacon 帧机制

Beacon 帧是 WiFi 网络的"心跳信号"，AP 以固定间隔（默认 102.4ms）广播，是被动扫描的核心载体。

6.1 Beacon 帧结构

Beacon 帧结构（IEEE 802.11-2012 §8.3.3.2）

6.2 TBTT 与 Beacon 间隔

根据 IEEE 802.11 标准（stackoverflow.com 对 802.11 spec §11.1 的引用）以及 NDSS 2025 论文《CHAOS》的分析：

TSF（Timing Synchronization Function）机制（IEEE 802.11 §11.1）：
• AP 为 BSS 的时序主控，维护一个精度微秒级的 64 位 TSF 计时器。
• Beacon 帧以 dot11BeaconPeriod 参数（单位 TU，1 TU = 1024 µs）为间隔定时发送，默认值为 100 TU ≈ 102.4 ms。
• 每次 Beacon 发送时刻称为 TBTT（Target Beacon Transmission Time）。
• 由于 CSMA/CA 信道竞争，AP 无法保证恰好在 TBTT 发送（实际可能延迟），但 Beacon 帧中携带的 TSF 时间戳使 STA 能够重新校正本地时钟。
• STA 收到 Beacon 后，若 TSF 计时器与 Beacon 中的时间戳不符，立即更新本地 TSF。

6.3 被动扫描过程

  • 1
    STA 将射频调谐到目标信道，进入静默监听状态。
  • 2
    等待接收来自该信道上任意 AP 的 Beacon 帧（默认间隔 100ms）。
  • 3
    从 Beacon 中提取 SSID、BSSID、支持速率、安全能力、信道信息等。
  • 4
    在所有待扫信道上重复以上步骤，汇总结果后选择合适 AP 发起关联。
被动扫描的主要缺陷（my80211.com / networkengineering.stackexchange.com）：
每个信道需等待至少 100ms（可能更长）才能确保听到 Beacon。如果 2.4 GHz 有 11 个信道，最坏情况下总扫描时间可达 数秒（WiFi Explorer Pro 测量约 8 秒）。被动扫描速度慢，且容易因等待时间不足而错过 Beacon，因此大多数客户端优先使用主动扫描。
参考来源：mrncciew.com: 802.11 Mgmt Beacon Frame（source: IEEE 802.11-2012）； StackOverflow: 802.11 spec §11.1 TSF 机制讨论； NDSS 2025: CHAOS (TSF & TBTT 详细分析)； NXP Community: 802.11 Connection Process； Intuitibits: WiFi Explorer Pro 扫描模式
§7 · WiFi

主动扫描：Probe Request/Response 详解

主动扫描是 WiFi 网络发现的主流方式，STA 广播 Probe Request 帧，AP 以 Probe Response 单播应答，整个过程在每信道约 10ms 内完成。

7.1 Probe Request 帧

根据 CWAP 官方学习指南（引用 IEEE 802.11-2012 标准）及 mrncciew.com 的帧捕获分析：

Probe Request 帧地址字段

• 地址字段 1（RA/DA）：ff:ff:ff:ff:ff:ff（广播地址）
• 地址字段 2（TA/SA）：发送方（STA）自身 MAC 地址
• 地址字段 3（BSSID）：ff:ff:ff:ff:ff:ff（面向所有 AP）
• 发送速率：最低速率（2.4 GHz 通常为 1 Mbps 或 6 Mbps，以最大覆盖范围）

7.2 两种 Probe Request 类型

定向 Probe 与广播 Probe 对比示意

7.3 Probe Response 帧

AP 收到 Probe Request 后，若 SSID 匹配（或为广播 SSID），以单播方式向 STA 发送 Probe Response。

Probe Response 与 Beacon 的区别（mrncciew.com，引用 IEEE 802.11-2012）：
1. Beacon 帧包含 TIM（Traffic Indication Map），Probe Response 不含 TIM
2. Beacon 帧包含 QoS 能力信息元素
3. Probe Response 可包含 STA 在 Probe Request 中明确请求的信息元素
Probe Response 是单播帧，STA 需要以 ACK 确认收到（与 Beacon 的广播+无需 ACK 不同）。

7.4 完整 Probe 时序

  • 1
    STA 切换到目标信道，发送 Probe Request（广播地址，最低速率）。
  • 2
    启动 ProbeTimer（IEEE 802.11 标准未规定具体值，典型实现约 10ms）倒计时，同时等待 Probe Response。
  • 3
    该信道上所有 SSID 匹配的 AP 以 Probe Response 单播应答（AP 以当前 BSS 最低强制速率发送）。
  • 4
    STA 向每个 Probe Response 发送 ACK 确认。
  • 5
    ProbeTimer 到期后，STA 处理收到的所有应答，然后切换到下一信道重复。
主动扫描速率优势（networkengineering.stackexchange.com）：
主动扫描每信道等待时间远少于 100ms，通常仅需 10ms 以内。对于 2.4 GHz 11 个信道，总扫描时间约 110ms；5 GHz 更多信道也可在合理时间内完成。这比被动扫描快约 10~100 倍，是操作系统默认优先采用的模式。
参考来源：mrncciew.com: CWAP 802.11 Probe Request/Response（IEEE 802.11-2012）； my80211.com: 802.11 Active and Passive Scanning（Cisco VOIP 4.1 Design Guide）； NetworkEngineering SE: How do WiFi clients detect SSIDs from APs； IEEE 802.11 TGk DCN: Probe Request Frame Format（§7.2.3.8）
§8 · WiFi

WiFi 发现机制的设计背景

IEEE 802.11 的发现机制并非偶然形成，而是在特定技术约束和应用场景下，经过多轮工程权衡演进而来的。

8.1 基础设施中心化设计

WiFi 的设计目标是为固定基础设施（AP）提供无线接入，而非对等发现（这与蓝牙 piconet 的设计哲学不同）。AP 是已知的、静止的、功率充足的基础设施节点，由 AP 承担主动广播（Beacon）的职责是自然选择——AP 不受电池限制，可以持续广播。

8.2 多信道挑战

WiFi 在 2.4 GHz 有最多 13 个信道（不同国家地区不同），5 GHz 有 20+ 个信道，6 GHz（WiFi 6E）更多。STA 必须逐信道扫描，因为 802.11 无线电一次只能调谐到一个频率。这是 WiFi 与蓝牙发现机制的核心结构差异之一。

信道扫描设计权衡（Cisco Voice Over WLAN 4.1 Design Guide，引用自 my80211.com）：
• 子集扫描：利用后台扫描信息预先确定哪些信道有 AP，仅扫描这些信道
• 提前终止：语音通话场景下，找到第一个可接受 AP 即停止扫描
• 调整计时器：语音场景下缩短 ProbeTimer，减少切换中断
• 后台扫描：在不传输数据的空闲时段扫描备选信道，为漫游提前准备

8.3 为何不能像蓝牙一样"广播发现"？

蓝牙通过跳频扩展到 79 个信道扫描，BLE 固定 3 个广播信道。WiFi 能否类似地在全部信道上"广播"呢？答案是不能，原因在于：

📡
信道宽度差异

WiFi 信道宽度 20/40/80/160 MHz，远宽于蓝牙 1/2 MHz 信道，无法在不同频率上同时传输

⚡
OFDM 物理层

802.11 使用 OFDM 多子载波调制，无法像蓝牙 GFSK 那样在 625µs 内快速切换信道广播

🔧
基础设施模型

AP 一直在线广播 Beacon，无需客户端做全频段扫描；Probe 机制是对 Beacon 的效率优化

8.4 隐藏 SSID 与定向扫描

由于 Beacon 帧携带明文 SSID，部分 AP 配置为"隐藏 SSID"（即 Beacon 中 SSID 字段为空）。在这种情况下：
• 被动扫描无法发现该网络（因为 Beacon 不含 SSID）
• 主动扫描通过定向 Probe Request（携带目标 SSID）可以发现并连接

这也揭示了一个隐私问题：STA 在主动扫描时广播的定向 Probe Request 中携带历史连接的 SSID，可被周边任何设备截获，从而暴露用户的历史 WiFi 使用记录（如 Stanford EE26n 课程材料所指出）。

参考来源：my80211.com（引用 Cisco Voice Over WLAN 4.1 Design Guide）； Stanford EE26n Assignment 7: WiFi 隐私问题； CWNP: 802.11 MAC Series – Scanning
§9 · 对比

三种机制横向对比

从物理层、时序、功耗、角色分工等维度系统对比经典蓝牙 Inquiry、BLE 广播扫描与 WiFi 主被动扫描的异同。

维度 经典蓝牙 BR/EDR BLE WiFi 802.11
发现模式 主动 Inquiry（双向跳频同步） 外设广播 + Central 扫描（异步） 被动：侦听 Beacon；主动：Probe Request/Response
频率使用 79 个 1MHz 信道（2402~2480MHz），FHSS 跳频 3 个固定广播信道（37/38/39），不跳频 逐信道扫描（2.4GHz 最多 13 信道，5GHz 20+信道），OFDM
核心帧/包 ID 包（含 GIAC 访问码）→ FHS 应答包 ADV_IND → SCAN_REQ → SCAN_RSP Beacon（100ms 周期）；Probe Request → Probe Response → ACK
发起方角色 主动发送 ID 包（Inquirer） 扫描方（Observer/Central）被动收包 主动：STA 发 Probe Request；被动：STA 等待 Beacon
被发现方角色 周期进入 Inquiry Scan（每 1.28s 切换信道） 持续广播 ADV 包（外设/Broadcaster） AP 持续发送 Beacon；收 Probe 后单播应答
最短发现时间 ≥ 1.28s（推荐 10.24s，v1.2 可达 1.28s） 取决于广播间隔（最小 20ms）+ 扫描窗口 主动：每信道约 10ms；2.4GHz 11 信道总计 ~110ms
功耗特征 Inquiry 子状态功耗高，阻断正常通信 广播功耗极低，外设可长期广播；扫描方功耗较高 扫描期间无法正常收发数据；AP 功耗不受限
抗干扰机制 79 信道 FHSS，天然抗窄带干扰 3 信道分散于频段两端与中间，随机延迟避免碰撞 固定信道，CSMA/CA 避免冲突；多信道选择分散负载
身份识别 FHS 包含 BD_ADDR（48 位设备地址） ADV 包含广播地址（可为随机地址） Beacon/Probe Response 含 BSSID（AP MAC）+ SSID
可发现性控制 General/Limited Discoverable Mode（GIAC/LIAC） 广播类型（ADV_IND/ADV_NONCONN_IND 等）+ 白名单过滤 隐藏 SSID（Beacon 不播 SSID）；仅响应定向 Probe
隐私风险 FHS 暴露 BD_ADDR 可用随机 MAC；但广播内容仍可被收集 Probe Request 暴露 STA MAC 及历史 SSID 列表
三种机制发现时延对比（典型值，对数坐标感知）

§10 · 第一性原理

第一性原理总结

剥去协议细节，从最基本的物理和信息论约束出发，理解三种发现机制的内在逻辑。

核心命题：无线发现的本质是在时间×频率二维资源中，让"发问者"与"应答者"在同一时刻、同一频率上相遇。三种机制的差异，归根结底是对"如何组织这次相遇"的不同工程选择。

约束决定设计

🎯
经典蓝牙：强同步，高代价

FHSS 物理层要求双方跳频同步。发现=找到共同的频率时隙。解决方案：主动方覆盖所有频率，被动方周期性"露头"。代价：慢（秒级），阻塞通信。

⚡
BLE：异步广播，极致省电

解耦发现与连接：固定 3 个广播信道，外设持续发包（被动角色的极限转换为主动角色），扫描方按自己节奏监听。牺牲灵活性换取超低功耗。

🏗️
WiFi：基础设施主导，效率优先

AP 不受电池限制，由 AP 持续广播心跳（Beacon）。客户端主动 Probe 是对被动等待的速度优化。多信道是 OFDM 宽信道物理层的必然结构代价。

三个关键设计权衡

① 谁主动？

• BT Classic：发起方主动（Inquiry），被发现方被动响应（Scan）
• BLE：被发现方主动（广播），发起方被动（扫描）— 角色倒置！
• WiFi：AP 主动广播（Beacon），STA 可选主动（Probe）或被动（侦听）— 灵活双模

② 频率如何组织？

• BT Classic：79 信道全覆盖跳频，时间换空间（慢但可靠）
• BLE：3 信道固定，空间换简单性（快速同步，冗余抗干扰）
• WiFi：单信道顺序扫描，多信道序列化遍历（工业实用主义）

③ 功耗如何分摊？

• BT Classic：对称分摊（双方均消耗资源），且阻断数据通信
• BLE：外设高频广播（低功耗 GFSK，短包），扫描方承担更大功耗；通过广播间隔调节
• WiFi：AP 承担全部广播代价（有线供电），STA 扫描时暂停数据；功耗代价由基础设施吸收

最简一句话概括

经典蓝牙：两个流浪者在夜晚举着火把跑遍 79 条街道互相找对方，总要花几秒钟。

BLE：小设备每隔几百毫秒在路口闪一下手电筒（固定 3 个路口），手机扫一圈就能看见。

WiFi：路灯（AP）一直亮着（Beacon），或者你（STA）主动喊一声"有人吗？"（Probe Request），路灯立刻答复——因为路灯接着电，不需要省电。

这三种机制的演进，折射出无线通信技术从对称耦合（经典蓝牙）→ 极致低功耗异步（BLE）→ 基础设施主导高效率（WiFi）的范式迁移，核心驱动力始终是：在物理约束下最大化用户体验与能量效率的乘积。
