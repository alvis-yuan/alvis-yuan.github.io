第一章

Linux iw 命令详解

官方来源：Linux Wireless Wiki — iw · Kali Linux iw Tools · kernel.org iw releases

概述与背景

iw 是 Linux 下新一代无线设备配置工具，用于配置和查询无线网络设备（802.11 WiFi）。它基于 nl80211 内核接口，取代了使用已废弃 Wireless Extensions（WEXT）接口的旧工具 iwconfig。

📌 官方说明

"iw is a new nl80211 based CLI configuration utility for wireless devices. It supports all new drivers that have been added to the kernel recently. The old tool iwconfig, which uses Wireless Extensions interface, is deprecated and it's strongly recommended to switch to iw and nl80211."
— Linux Wireless Wiki

iw 的发布版本可从 http://kernel.org/pub/software/network/iw/ 获取，源码仓库位于 git.kernel.org。

安装方式

sudo apt install iw       # Debian/Ubuntu/Kali
sudo yum install iw       # RHEL/CentOS
sudo pacman -S iw         # Arch Linux

# 查看版本
iw --version
iw version 6.17

依赖关系

依赖项作用说明
libnl-3Netlink 套接字抽象库提供 Netlink 消息构建/收发 API
libnl-genl-3通用 Netlink 扩展nl80211 基于 Generic Netlink (genl)
nl80211 内核模块无线配置内核接口随内核 cfg80211 一起提供

本章小结

iw 是 Linux 下无线配置的现代标准工具，依托 nl80211/cfg80211 内核接口，支持所有现代无线硬件驱动。官方已明确废弃 iwconfig，建议全面迁移到 iw。

第二章

架构原理

整体架构层次

iw 的运作涉及用户空间、内核空间两大层次，通过 Generic Netlink 套接字进行通信。以下是完整的调用链路：

图1：iw 工具的完整调用链路（基于 Linux Wireless Wiki）

核心组件详解

① Generic Netlink (genl) — 用户与内核通信桥梁

iw 通过 libnl 库，使用 Generic Netlink 套接字与内核的 nl80211 子系统通信。Netlink 是一种专为内核与用户空间进程间通信设计的套接字类型，相比 ioctl 具备以下优势：

  • 支持异步事件通知（内核主动推送），而 ioctl 只能同步轮询
  • 消息格式灵活，基于 TLV（Type-Length-Value）属性，易于扩展
  • 支持多播组，多个进程可同时订阅同一事件
图2：iw 发送的 Generic Netlink 消息结构（TLV 属性）

② nl80211 — 内核无线配置入口

nl80211 是注册在内核中的 Generic Netlink 家族名称（family name: "nl80211"），实现于 net/wireless/nl80211.c。它定义了所有无线操作命令（NL80211_CMD_*）和属性（NL80211_ATTR_*），例如：

内核命令对应 iw 操作说明
NL80211_CMD_TRIGGER_SCANiw dev scan trigger触发无线扫描
NL80211_CMD_GET_SCANiw dev scan dump获取扫描结果
NL80211_CMD_SET_WIPHYiw phy set channel设置物理层参数
NL80211_CMD_CONNECTiw dev connect发起连接
NL80211_CMD_NEW_INTERFACEiw phy interface add创建虚拟接口
NL80211_CMD_GET_STATIONiw dev station dump获取站点统计

③ cfg80211 — 内核无线配置核心

cfg80211 是 Linux 内核中无线配置的通用 API 层（include/net/cfg80211.h），它：

  • 接收来自 nl80211 转发的用户命令
  • 通过 struct cfg80211_ops 回调函数，调用具体的驱动程序
  • 管理 struct wiphy（物理设备）和 struct wireless_dev（虚拟接口）

④ mac80211 — 软件 MAC 层

对于使用 mac80211 框架的驱动（如 ath9k、iwlwifi、mt76），mac80211 实现了 802.11 MAC 层逻辑（帧封装、ACK、加密等），驱动只需实现硬件寄存器操作。

图3：Linux 无线子系统完整层次栈

本章小结

iw 的核心工作原理：用户调用 iw 命令 → iw 通过 libnl 构建 Generic Netlink 消息 → 经 AF_NETLINK 套接字发送至内核 → nl80211 解析命令 → cfg80211 调用驱动回调 → mac80211/硬件驱动执行操作。整个链路设计上将"用户意图"与"驱动实现"彻底解耦，支持多进程并发订阅事件，相比 ioctl 有本质性改进。

第三章

与 iwconfig 对比

⚠️ 废弃警告

iwconfig 使用的 Wireless Extensions (WEXT) 接口已被废弃。官方建议全面迁移到 iw + nl80211。

对比维度 iwconfig（旧） iw（新）
内核接口 废弃 Wireless Extensions (WEXT / ioctl) 现代 nl80211 / Generic Netlink
通信机制 ioctl 系统调用（同步，不支持事件推送） Netlink 套接字（异步，支持事件订阅）
驱动支持 仅支持实现了 WEXT 的旧驱动 支持所有现代 cfg80211 驱动
802.11n/ac/ax 支持 ❌ 基本不支持 HT/VHT/HE 参数 ✅ 完整支持 HT/VHT/HE/EHT MCS
虚拟接口管理 ❌ 不支持 ✅ 支持创建多类型虚拟接口
事件监听 ❌ 不支持 ✅ iw event 实时监听内核事件

命令对照表

以下对照来自 Linux Wireless Wiki — Replacing iwconfig with iw：

# 获取 wlan0 状态
iwconfig wlan0                        # 旧
iw dev wlan0 link                     # 新

# 连接开放网络
iwconfig wlan0 essid foo              # 旧
iw wlan0 connect foo                  # 新

# 连接指定频率的开放网络
iwconfig wlan0 essid foo freq 2432M   # 旧
iw wlan0 connect foo 2432             # 新

# WEP 加密连接
iwconfig wlan0 key s:abcde            # 旧
iw wlan0 connect foo keys 0:abcde d:1:0011223344  # 新（注：使用0-based密钥编号）

本章小结

iw 在功能、接口质量、现代协议支持上全面超越 iwconfig。其底层从 ioctl 迁移到 Netlink，从根本上解决了 ioctl 无法异步推送事件的缺陷，是 Linux 无线配置的现代标准。

关键字段说明

字段含义
phy#0物理无线设备编号（对应实际 WiFi 芯片）
ifindexLinux 网络接口索引号
wdev无线设备标识符（wiphy index + per-wiphy unique ID）
type managed工作模式：managed=STA客户端，monitor=监控等
channel / center1当前信道号及中心频率
txpower当前发射功率（dBm）

背后原理

执行 iw dev 时，iw 向内核发送 NL80211_CMD_GET_INTERFACE 命令，并设置 NLM_F_DUMP 标志，内核 nl80211 遍历所有注册的无线接口，通过 Netlink 多条消息依次返回。iw phy 则发送 NL80211_CMD_GET_WIPHY，返回物理设备信息。

图4：iw dev 命令执行流程

本章小结

iw dev 和 iw phy 是无线管理的"侦察命令"，分别对应虚拟接口（wireless_dev）和物理设备（wiphy）两个层次。phy 是硬件抽象，一个 phy 可以对应多个 dev（虚拟接口）。

背后原理

iw dev wlan0 link 发送 NL80211_CMD_GET_STATION 命令（针对当前连接的 AP MAC）。内核通过 mac80211 的站点表（station table）返回统计信息，包括 RSSI（由驱动从射频芯片读取）和最近 TX/RX 速率（由速率控制算法记录）。

本章小结

iw dev wlan0 link 是快速诊断连接质量的首选命令。信号强度（RSSI）反映无线传播条件，TX 速率反映链路自适应算法的当前判断，MCS 索引则直接揭示当前采用的调制编码方案。

场景：无线扫描

来源：Linux Wireless Wiki — Scanning

常用命令

# 完整扫描并显示结果（需 root）
sudo iw dev wlan0 scan

# 只触发扫描（不等待结果）

sudo iw dev wlan0 scan ssid "MyNetwork"

扫描类型对比

图6：主动扫描 vs 被动扫描

背后原理

iw dev wlan0 scan 向内核发送 NL80211_CMD_TRIGGER_SCAN，内核通过 mac80211 驱动在每个信道上：

  1. 主动扫描：发送 802.11 Probe Request 帧（携带 SSID 或通配符），等待 AP 响应 Probe Response
  2. 被动扫描（使用 passive 参数）：在信道上静默等待，收集 AP 定期广播的 Beacon 帧
  3. 扫描完成后，内核发送 NL80211_CMD_NEW_SCAN_RESULTS 事件，iw 随后发送 NL80211_CMD_GET_SCAN（带 NLM_F_DUMP）获取结果
💡 实用提示

iw dev wlan0 scan dump 返回内核缓存的扫描结果，无需重新扫描，适合快速查看且不中断当前连接。

本章小结

无线扫描是连接的前提。iw 的扫描机制通过 nl80211 触发内核驱动进行主动/被动探测，结果缓存在内核，可按需获取。调度扫描（sched_scan）允许在省电模式下后台持续扫描，是 NetworkManager 实现漫游检测的底层机制。

第五章 · 场景五

场景：连接与断开网络

来源：Linux Wireless Wiki — Establishing a basic connection

⚠️ 重要限制

iw 只能连接 开放网络 或 WEP 加密网络。WPA/WPA2/WPA3 网络必须使用 wpa_supplicant。

连接开放网络

# 连接 SSID 为 foo 的开放网络
iw wlan0 connect foo

# 连接指定频率（2432MHz）的网络（当多个 AP 同名时）
iw wlan0 connect foo 2432

# 连接指定 BSSID 的网络
iw wlan0 connect foo 0 aa:bb:cc:dd:ee:ff

连接 WEP 加密网络

# 使用 ASCII 密钥（key index 0）和十六进制密钥（key index 1）
iw wlan0 connect foo keys 0:abcde d:1:0011223344
# 注意：iw 使用 0-based 密钥编号（与 iwconfig 的 1-based 不同）

断开连接

iw dev wlan0 disconnect

WPA/WPA2 连接（通过 wpa_supplicant）

# wpa_supplicant 底层也使用 nl80211 驱动
wpa_supplicant -D nl80211 -i wlan0 -c /etc/wpa_supplicant.conf
# 配置文件示例：
network={
    ssid="MyWPA2Network"
    psk="mypassword"
    key_mgmt=WPA-PSK
}

802.11 连接流程

图7：802.11 开放网络关联流程（Probe → Auth → Association）

背后原理

iw wlan0 connect foo 向内核发送 NL80211_CMD_CONNECT。对于开放网络，mac80211/cfg80211 自动执行 802.11 认证（Open System Authentication）和关联（Association）状态机。连接结果通过内核事件 NL80211_CMD_CONNECT 返回。

对于 WPA/WPA2，4-way handshake（密钥协商）需要 wpa_supplicant 参与，因为它需要处理复杂的密钥材料（PMK/PTK/GTK），iw 本身不实现这些协议。

本章小结

iw 的连接功能适用于测试和开放网络场景。其底层通过 NL80211_CMD_CONNECT 驱动 802.11 状态机（Probe → Authenticate → Associate）。现代加密网络（WPA2/WPA3）需要 wpa_supplicant，后者同样通过 nl80211 驱动与内核交互，两者设计上是互补关系。

场景：站点统计信息

# 列出所有已知站点统计（AP 模式下显示所有客户端）

sudo iw dev wlan1 station get aa:bb:cc:dd:ee:ff

	rx packets:	75

	tx retries:	3
	tx failed:	0
	signal:  	-29 dBm
	signal avg:	-30 dBm
	tx bitrate:	54.0 MBit/s
	rx bitrate:	54.0 MBit/s
	authorized:	yes
	authenticated:	yes
	associated:	yes
	preamble:	long
	WMM/WME:	yes
	MFP:		no

关键统计字段

字段含义诊断价值
inactive time站点最后活跃至今时间检测"僵尸"站点
tx retries发送重试次数高重试 → 链路质量差
tx failed发送失败次数持续失败 → 链路断开迹象
signal avg平均信号强度比瞬时 signal 更稳定
authorized是否已授权（通过 802.1X 或 PSK）认证状态确认
WMM/WMEWi-Fi 多媒体扩展（QoS）是否支持优先级队列

背后原理

station dump 发送 NL80211_CMD_GET_STATION（带 NLM_F_DUMP）。内核 mac80211 维护着每个已关联站点的 struct sta_info，其中记录了详细的收发统计、信号强度（由驱动从射频芯片读取）、速率等信息。

本章小结

station dump/get 是无线链路诊断的核心工具。在 AP 模式下可监控所有客户端状态，在 STA 模式下可详细了解与 AP 的连接质量。tx retries 和 tx failed 是判断链路质量下降的最直接指标。

第六章 · 场景七

场景：虚拟接口创建与删除

来源：Linux Wireless Wiki — Adding interfaces with iw / iw vif 页面

接口类型

类型描述典型用途
managedSTA 客户端模式（也称 station）连接到 AP
ibssIBSS/Ad-hoc 点对点模式无 AP 的设备直连
monitor监控模式，接收所有帧抓包、无线调试

创建接口

# 创建监控模式接口 moni0
iw phy phy0 interface add moni0 type monitor

# 创建 managed 模式接口
iw phy phy0 interface add wlan10 type managed

# 创建带 4addr 模式的 managed 接口（用于 WDS 桥接）
iw phy phy0 interface add sta0 type managed 4addr on

# 创建 Mesh 接口
iw phy phy0 interface add mesh0 type mp mesh_id mymesh

# 启用接口
ip link set moni0 up

删除接口

iw dev moni0 del

修改接口类型

# 先关闭接口
ip link set wlan0 down
# 修改类型
iw dev wlan0 set type monitor
# 重新启动
ip link set wlan0 up
图8：一个物理设备(phy0)可以创建多个不同类型的虚拟接口

背后原理

iw phy phy0 interface add 发送 NL80211_CMD_NEW_INTERFACE。内核在 cfg80211 中创建一个新的 struct wireless_dev，并关联到对应的 struct wiphy（phy0）。mac80211 驱动负责在硬件层实现多接口并发（例如同时运行 AP 和 monitor）。

多接口限制：硬件支持的并发接口组合受 iw list 中的 "valid interface combinations" 约束，不同芯片限制不同。

本章小结

虚拟接口机制是 mac80211 的核心能力之一，允许同一块物理 WiFi 芯片同时扮演多个角色（如同时作为 STA 连接上游 AP，又作为 AP 为下游设备提供服务）。iw 通过 NL80211_CMD_NEW_INTERFACE 调用这一能力，是构建无线路由器、蜜罐、中继器的基础工具。

场景：监控模式接口（抓包）

监控接口 Flags

Flag含义

Radiotap 帧结构

图9：监控接口捕获帧的结构（Radiotap + 802.11 原始帧）

背后原理

监控模式接口使 WiFi 芯片进入"混杂"接收状态，接收射频上所有可见的 802.11 帧（不论目标 MAC 是否是本机）。mac80211 将每个捕获的帧加上 Radiotap 头——包含信号强度（RSSI）、速率、信道、时间戳等元数据——通过 netif_receive_skb() 传递给用户空间（tcpdump/Wireshark）。

官方文档特别指出：使用 flags none 创建监控接口，可以在不中断原有 managed 接口运行的情况下进行无线调试，这是 mac80211 多接口虚拟化的典型应用。

本章小结

监控模式是无线安全审计和调试的核心工具。通过 flags none 创建的"轻量"监控接口可与 managed 模式接口共存。Radiotap 头携带了丰富的射频元数据，是 Wireshark 等工具分析无线信号质量的数据来源。从第一性原理看，监控模式本质上是将 WiFi 芯片的 MAC 地址过滤器旁路，允许内核收到所有空口帧。

常用 WiFi 信道频率对照

第七章 · 场景十二

场景：发射功率管理

来源：Linux Wireless Wiki — Setting TX power

使用命令

# 查看当前发射功率
iw dev wlan0 info  # 查看 txpower 字段

# 设置功率（单位 mBm = 100 * dBm）
# 设为 15 dBm = 1500 mBm
iw dev wlan0 set txpower fixed 1500

# 设置功率上限（允许驱动在此之下自动调整）
iw dev wlan0 set txpower limit 2000  # 上限 20 dBm

# 自动功率控制（让驱动/固件决定）
iw dev wlan0 set txpower auto

# 也可通过 phy 设置
iw phy phy0 set txpower fixed 1500
⚠️ 单位说明

iw 使用 mBm（毫贝分毫瓦），而非常用的 dBm。换算：mBm = dBm × 100。 例如：20 dBm → 2000 mBm，15 dBm → 1500 mBm。

功率模式说明

模式含义适用场景
auto驱动/固件自动决定功率日常使用（默认）
fixed固定到指定功率值测试、精确控制
limit设置功率上限，驱动可在此以下自动调整满足法规要求时

背后原理

发射功率由 RF 前端（功率放大器）控制。iw set txpower 发送 NL80211_CMD_SET_WIPHY，携带 NL80211_ATTR_WIPHY_TX_POWER_SETTING 和 NL80211_ATTR_WIPHY_TX_POWER_LEVEL 属性。驱动将此值转换为硬件寄存器设置。

功率受到两个维度的限制：①驱动/硬件最大值（由 iw list 中频率列表的 dBm 标注），②监管域允许值（由 CRDA 根据国家/地区规定）。两者取更严格的约束。

本章小结

发射功率是无线覆盖范围与干扰的核心参数。从第一性原理看，功率越高覆盖越远，但同时对邻居网络的干扰越大，且受各国法规硬性约束。iw 通过 nl80211 统一接口控制功率，最终受到监管域和硬件能力的双重约束。

802.11n MCS 速率索引表（单流，20MHz）

图13：802.11n MCS 0-7（单流，20MHz，长GI）速率

背后原理

iw set bitrates 发送 NL80211_CMD_SET_TX_BITRATE_MASK。内核将此转化为速率掩码，传递给速率控制算法（如 minstrel_ht）。速率控制算法在掩码允许的范围内自适应选择最优速率，实际速率仍由算法动态决定，不是强制固定到某个值（除非只设置一个速率）。

场景：省电模式

来源：Linux Wireless Wiki — Power save

使用命令

# 启用省电模式（mac80211 会启用动态省电 Dynamic Power Save）
sudo iw dev wlan0 set power_save on

# 关闭省电模式
sudo iw dev wlan0 set power_save off

# 查询当前省电设置
iw dev wlan0 get power_save
Power save: on

802.11 省电机制原理

图14：802.11 动态省电模式（Dynamic PS）工作时序

背后原理

官方文档说明：对于 mac80211 驱动，启用省电模式意味着 Dynamic Power Save (DPS) 被激活。DPS 的工作机制：

  1. 空闲时 STA 通知 AP 进入 PS（Power Save） 状态（通过帧控制字段的 Power Management 位）
  2. AP 缓存发往该 STA 的帧，在 Beacon 帧的 TIM（Traffic Indication Map） 字段中标记有待接收帧
  3. STA 在每个 DTIM Beacon 时唤醒，若 TIM 指示有帧则发送 PS-Poll 请求 AP 发送缓存帧
  4. 接收完毕后重回睡眠状态

本章小结

省电模式是 WiFi 节能的核心机制，对移动设备续航至关重要。iw 通过 nl80211 控制 mac80211 的 DPS 开关。从第一性原理看，省电的本质是"让射频芯片大部分时间处于低功耗睡眠状态，仅在必要时唤醒"，代价是引入额外延迟（需等待 Beacon/DTIM 周期）。

iw phy phy0 wowlan show

# 启用 WoWLAN：收到 Magic Packet 时唤醒
iw phy phy0 wowlan enable magic-packet

# 启用 WoWLAN：断开连接时唤醒
iw phy phy0 wowlan enable disconnect

# 启用 WoWLAN：GTK 重协商失败时唤醒（安全监控）
iw phy phy0 wowlan enable gtk-rekey-failure

# 启用 WoWLAN：后台发现指定 SSID 时唤醒（需网络检测）
iw phy phy0 wowlan enable net-detect interval 5000 delay 30 freqs 2412 2437 matches ssid HomeWifi

iw phy phy0 wowlan enable patterns ff:ff:ff:ff:ff:ff

# 禁用 WoWLAN
iw phy phy0 wowlan disable

WoWLAN 触发器类型

触发器描述适用场景
any任意网络活动触发唤醒调试
disconnectWiFi 连接断开时唤醒高可用需求
magic-packet收到 Magic Packet 时唤醒远程开机（WoL）

背后原理

WoWLAN 由 WiFi 固件在主机 CPU 睡眠期间独立监控网络流量。iw wowlan enable 发送 NL80211_CMD_SET_WOWLAN 将触发条件下发给固件。当条件满足时，固件通过 PCIe/USB 的唤醒信号（WoL）将系统从睡眠中唤醒，实现"固件主机分离的轻量网络监控"。

本章小结

WoWLAN 是 WiFi 固件能力的典型应用：主机 CPU 深度睡眠，WiFi 固件独立工作，监测特定网络事件后发出硬件中断唤醒整机。这是笔记本电脑"即时唤醒"、NAS 远程开机等功能的底层实现机制。iw 通过 nl80211 将唤醒规则精确配置到固件。

第一性原理总结

📐 第一性原理

iw 的本质：一个将人类可读的无线配置意图，通过结构化的 Netlink 消息传递给 Linux 内核无线子系统（nl80211/cfg80211），最终驱动射频硬件执行的命令行工具。

从第一性原理分解 iw 的设计哲学：

  1. 解耦原则：iw（用户意图）→ libnl（消息构建）→ nl80211（命令协议）→ cfg80211（内核抽象）→ 驱动（硬件控制），每层职责单一，彼此通过标准接口解耦。驱动程序无需了解用户界面，用户工具无需了解硬件细节。
  2. 无线的物理本质：所有 iw 命令最终操纵的是射频参数（频率、功率、调制方式）和 802.11 协议状态机（扫描→认证→关联→传输→断开）。理解了无线电波的物理特性（频率/干扰/衰减）和 802.11 帧交换流程，就理解了 iw 每条命令的作用。
  3. 事件驱动设计：nl80211 基于 Netlink 多播的事件推送，是相对于旧版 ioctl 的根本性改进。无线网络本质上是异步事件驱动的（连接/断开/扫描完成/雷达检测都是异步事件），Netlink 的设计天然适配这一特性。
  4. 合规约束：所有无线操作都受监管域约束，这反映了无线频谱作为公共资源的法律监管现实。iw 的监管域机制（CRDA + wireless-regdb）是技术对法律约束的实现。

最终总结

iw 是 Linux 无线管理的统一命令行入口，通过 nl80211/cfg80211 内核接口实现了对现代 WiFi 硬件的完整控制。它的价值在于：

  • 标准化：统一的 Netlink 接口取代了碎片化的 ioctl 接口
  • 现代化：全面支持 802.11n/ac/ax 特性（MCS、VHT、HE）
  • 可观测性：事件监听、站点统计、信道调查提供完整的无线状态可见性
  • 可扩展性：虚拟接口机制使一块网卡能同时承担多种角色
