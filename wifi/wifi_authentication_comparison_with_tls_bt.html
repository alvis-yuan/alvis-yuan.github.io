第 0 章

引言：为什么要比较这三个协议？

本文以 WiFi 认证流程为主线，通过对比 TLS 1.3 握手（RFC 8446）和蓝牙安全配对（Bluetooth Core 5.4 Part H）两个同类协议，帮助读者从多维度理解无线认证的本质。每个协议都面临相同的根本问题：如何在不安全的空中接口上，让两个设备证明彼此身份，并安全地协商出一把会话密钥？

现代无线协议都需要在开放媒介（空气）上解决 身份认证（Authentication）、密钥协商（Key Establishment）和数据保密（Confidentiality）三大问题。尽管应用场景迥异，TLS、蓝牙配对和 WiFi 认证在解题思路上高度趋同——它们都使用了 Diffie-Hellman 或其椭圆曲线变体，通过承诺-揭示（Commit-Reveal）或证书链建立信任。

🔵 TLS 1.3

  • 标准：RFC 8446（IETF，2018）
  • 场景：互联网客户端/服务器
  • 信任根：X.509 CA 证书
  • RTT：1-RTT（或 0-RTT 恢复）

🟣 蓝牙 SSP/SC

  • 标准：Bluetooth Core 5.4 Part H
  • 场景：近距离点对点配对
  • 信任根：带外信道/用户确认
  • 方式：ECDH + 用户交互

🟢 WiFi WPA2/WPA3

  • 标准：IEEE 802.11i / Wi-Fi Alliance
  • 场景：无线局域网接入认证
  • 信任根：预共享密钥或 RADIUS
  • 方式：四次握手 / SAE

三者共同问题

  • 如何抵御被动窃听？
  • 如何抵御中间人攻击？
  • 如何实现前向安全？
  • 如何证明身份？

TLS 1.3 — 第 1 章

TLS 1.3 协议概述

本章从 RFC 8446 原文出发，介绍 TLS 1.3 的设计目标与核心架构。理解 TLS 有助于把握后续 WiFi 认证的"参照系"——两者在密钥推导思想上高度相似。

RFC 8446 定义了 TLS 1.3，其首要目标是在两个通信对端之间提供安全信道，具体保障三项属性（引自 RFC 8446 第 1 节）：

A

Authentication（身份认证）

服务端总是被认证；客户端可选认证。可通过非对称密码（RSA、ECDSA、EdDSA）或对称预共享密钥（PSK）实现。

C

Confidentiality（机密性）

信道建立后发送的数据仅对端点可见。TLS 不隐藏数据长度，但端点可以填充记录以对抗流量分析。

I

Integrity（完整性）

信道建立后发送的数据攻击者无法在不被察觉的情况下篡改。

TLS 1.3 相较 TLS 1.2 的主要改进

RFC 8446 第 1.2 节列出了若干重要变化：

· 移除所有遗留对称加密算法，仅保留 AEAD（Authenticated Encryption with Associated Data）算法。
· 增加 0-RTT 模式，以牺牲部分安全性为代价节省连接建立时的一个往返。
· 移除静态 RSA 和 Diffie-Hellman 密码套件；所有基于公钥的密钥交换机制均提供前向安全。
· ServerHello 之后的所有握手消息均加密。
· 密钥推导函数重新设计，引入基于 HMAC 的 HKDF 作为基础原语。
· 握手状态机大幅简化，去除 ChangeCipherSpec 等冗余消息。

TLS 的两个核心组件

握手协议（Section 4）

  • 认证通信方
  • 协商加密模式与参数
  • 建立共享密钥材料
  • 设计目标：抵御篡改，主动攻击者无法强迫对端协商更弱参数

记录协议（Section 5）

  • 使用握手协议建立的参数
  • 保护通信对端之间的流量
  • 将流量划分为独立记录
  • 每条记录使用流量密钥独立保护
原文引用： RFC 8446, Section 1 — "The primary goal of TLS is to provide a secure channel between two communicating peers; the only requirement from the underlying transport is a reliable, in-order data stream." — E. Rescorla, Mozilla
链接：https://datatracker.ietf.org/doc/html/rfc8446#section-1

TLS 1.3 — 第 2 章

TLS 1.3 握手流程详解

本章依据 RFC 8446 第 2 节与第 4 节，逐步拆解 TLS 1.3 的完整握手过程。图表还原了 RFC 8446 Figure 1 的 MSC（消息序列图），帮助你看清每条消息的时序与加密边界。

📊 图 2-1 — TLS 1.3 完整握手 MSC（基于 RFC 8446 Figure 1）

三个握手阶段（RFC 8446 §2）

1

密钥交换（Key Exchange）

建立共享密钥材料，选择加密参数。此阶段之后所有内容均加密。Client 发送 ClientHello（含支持的密码套件、key_share 扩展）；Server 响应 ServerHello（含选定参数和自己的 key_share）。双方通过 (EC)DHE 独立计算出相同的共享秘密。

2

服务端参数（Server Parameters）

建立其他握手参数，包括是否需要客户端认证、应用层协议支持等。Server 发送加密的 EncryptedExtensions、可选的 CertificateRequest。

3

认证（Authentication）

Server 发送 Certificate、CertificateVerify（用私钥对握手记录签名）、Finished（HMAC 校验）。若需客户端认证，Client 也发送对应消息。Finished 消息保证握手记录的完整性。

关键设计：TLS 1.3 在 ServerHello 之后立即切换到加密通信，Certificate 和 CertificateVerify 均在加密信道内传输，服务端身份受到机密性保护，被动攻击者无法获知服务端使用的证书。

原文引用： RFC 8446, Section 2 — "The handshake can be thought of as having three phases: Key Exchange, Server Parameters, Authentication."
链接：https://datatracker.ietf.org/doc/html/rfc8446#section-2

TLS 1.3 — 第 3 章

TLS 1.3 密钥调度与前向安全

密钥调度（Key Schedule）是 TLS 1.3 最精妙的设计之一。本章依据 RFC 8446 第 7 节，解析 HKDF 驱动的密钥层级如何实现前向安全，并为后续理解 WiFi PTK 推导提供参照。

📊 图 3-1 — TLS 1.3 密钥调度（HKDF-based，RFC 8446 §7.1）

前向安全（Forward Secrecy）

RFC 8446 Appendix E.1 明确指出：

若长期密钥材料（证书签名密钥或外部 PSK）在握手完成后泄露，只要会话密钥本身已被擦除，就不会危及会话密钥的安全性。——"Forward secret with respect to long-term keys"

这是通过每次握手生成新鲜的临时 (EC)DHE 密钥对实现的。TLS 1.3 移除了静态 RSA 密钥传输，确保所有密钥交换都具备前向安全。

原文引用： RFC 8446, Section 7.1 — Key Schedule 与 Appendix E.1 — Handshake Security Properties
链接：https://datatracker.ietf.org/doc/html/rfc8446#section-7.1

蓝牙 — 第 4 章

蓝牙安全配对概述

本章基于 Bluetooth Core 5.4 Vol 3 Part H（Security Manager Specification）和 NIST SP 800-121r2，介绍蓝牙配对的整体框架。蓝牙配对与 TLS 最大的不同在于：TLS 使用 CA 证书建立信任，而蓝牙依赖用户物理在场（或带外信道）来对抗中间人攻击。

安全管理器（Security Manager）的定位

蓝牙 Core 5.4 Part H 第 1 节指出：安全管理器（SM）定义了 LE 设备（含 BR/EDR/LE 复合设备）之间进行配对、认证和加密的协议与行为。其核心职责包括：密钥分发方法、配对协议，以及供其他协议层使用的安全工具箱。

蓝牙配对演进历史

📊 图 4-1 — 蓝牙配对技术演进时间线

三阶段配对框架（Bluetooth Core 5.4 Part H §2.1）

Bluetooth Core 5.4 Part H 第 2 节明确定义，配对是一个三阶段过程：

1

Phase 1：配对特性交换（Pairing Feature Exchange）

两台设备交换认证需求和 IO 能力（IOcapability）。这决定了在第二阶段使用哪种密钥生成方法：Just Works、Numeric Comparison（仅 LE Secure Connections）、Passkey Entry 或 Out of Band。

2

Phase 2：密钥生成（STK 或 LTK）

LE Legacy 配对生成短期密钥 STK；LE Secure Connections 生成长期密钥 LTK。此阶段包含 ECDH 公钥交换、承诺值验证和用户交互（依关联模型而定）。

3

Phase 3：传输特定密钥分发（可选）

在链路加密后，设备可分发 IRK（身份解析密钥）、CSRK（签名密钥）等传输特定密钥，用于后续重连的随机地址解析和数据签名验证。

原文引用： Bluetooth Core 5.4, Vol 3 Part H, Section 2.1 — "Pairing is a three-phase process."
链接：Bluetooth Core 5.4 Part H — Security Manager Specification
NIST：NIST SP 800-121r2 — Guide to Bluetooth Security

蓝牙 — 第 5 章

LE Secure Connections 详细流程

LE Secure Connections（引入于 Bluetooth 4.2）是蓝牙当前最高安全等级的配对机制。本章以数字比较（Numeric Comparison）关联模型为例，依据 Bluetooth.com 官方博客与 Core 5.4 Part H，逐步拆解完整的 ECDH + 承诺-揭示流程。

📊 图 5-1 — 蓝牙 LE Secure Connections 数字比较 MSC（基于 Bluetooth Core 5.4 Part H 图 C.8）

关键承诺-揭示机制

Bluetooth Core 5.4 Part H 第 2.2.6 节定义了确认值生成函数 f4：

Cb = f4(PKbx, PKax, Nb, 0) = AES-CMACNb(PKbx ‖ PKax ‖ 0)
响应方在发送 Nb 之前必须先发送 Cb（承诺），发起方收到 Nb 后校验 Cb，若不一致说明存在攻击者或传输错误，立即终止配对（Step 6.a）。这是承诺方案（Commitment Scheme）的典型应用，防止响应方在看到 Na 后才构造 Nb。

蓝牙 — 第 6 章

关联模型与密钥层级

蓝牙配对提供四种关联模型（Association Models），根据设备的 IO 能力自动选择。NIST SP 800-121r2 对各模型的安全等级有明确分析。理解这些模型有助于理解为什么 WiFi 中的"预共享密钥"思路需要演进到 SAE（WPA3）。

四种关联模型对比

关联模型 适用场景 防窃听 防 MITM 前向安全 用户交互
Just Works 无显示/键盘设备（耳机等） ✓ SC ✗ ✓ SC 无
Numeric Comparison 双方均有显示屏 ✓ ✓ ✓ 确认 6 位数
Passkey Entry 一方有键盘 有限 有限 ✓ SC 输入 6 位数
Out of Band (OOB) NFC/QR 等带外信道 ✓ 取决于 OOB 信道 ✓ SC OOB 交互

NIST SP 800-121r2 指出： Passkey Entry 在 LE Legacy 模式下，TK 值空间仅 10⁶（百万），对主动窃听者提供的保护十分有限。LE Secure Connections 通过 ECDH 大幅改善了这一问题。

WiFi — 第 7 章

WPA2 四次握手（4-Way Handshake）

四次握手是 WPA2 认证的核心，定义于 IEEE 802.11i（现并入 IEEE 802.11-2020 Clause 12），通过四条 EAPOL-Key 帧完成双向身份证明与会话密钥派生。本章对比 TLS Finished 消息的设计思路，解析 PTK 推导与 MIC 校验机制。

WPA2 的已知漏洞：
① KRACK（2017）：通过重放消息 3 强制 nonce 复用，允许解密流量。根本原因是 STA 收到重复消息 3 时会重装已安装密钥并重置 nonce。
② 离线字典攻击：攻击者捕获四次握手后可离线暴力破解弱 PSK，因为 PMK 直接依赖密码（PBKDF2）且不变。WPA3-SAE 通过引入前向安全密钥交换彻底解决了这个问题。

WiFi — 第 8 章

WPA3-SAE：同等级身份认证

WPA3-Personal（2018）引入 SAE（Simultaneous Authentication of Equals）替代 WPA2 的 PSK 直接推导方式，SAE 也称 Dragonfly 密钥交换，定义于 IEEE 802.11-2020 Section 12.4 及 RFC 7664。本章结合官方定义，对比 TLS 1.3 的 (EC)DHE，理解 SAE 如何同时实现前向安全和抵御离线字典攻击。

📊 图 8-1 — WPA3-SAE（Dragonfly）认证流程 MSC

WPA3-SAE 相较 WPA2-PSK 的根本改进

❌ WPA2-PSK 的问题

  • PMK = PBKDF2(密码, SSID, 4096) — 静态不变
  • 捕获握手即可离线暴力破解
  • 无前向安全：PMK 泄露 → 所有历史会话均可解密
  • KRACK 攻击可强制 nonce 复用

✅ WPA3-SAE 的改进

  • PMK 通过 ECDH 风格的 Dragonfly 交换得出
  • 每次连接的 PMK 均不同（临时密钥）
  • 不可离线字典攻击（需在线交互猜测）
  • 前向安全：过去会话密钥不可追溯

与 TLS 1.3 的类比：WPA3-SAE 的 Dragonfly 密钥交换 ≈ TLS 1.3 的 ECDHE；WPA2-PSK 的静态 PMK ≈ TLS 1.2 中弃用的静态 RSA 密钥传输。两者都通过引入临时密钥（Ephemeral Key）来实现前向安全。

权威参考： IEEE 802.11-2020 Section 12.4（SAE）；WPA3-SAE Mode, mrncciew.com（实测报文分析）；Wi-Fi Security Overview, Smallstep；Dragonfly Key Exchange — RFC 7664

WiFi — 第 9 章

企业级认证：802.1X 与 EAP

企业 WiFi（WPA2/WPA3-Enterprise）通过 802.1X 端口访问控制 + EAP（可扩展认证协议）+ RADIUS 服务器实现每用户独立认证。EAP-TLS 是最安全的企业认证方式，本质上是在 WiFi 认证流程内嵌套了一次 TLS 握手，形成了两层安全机制的叠加。

第一性原理总结

回归本质：无论是 TLS、蓝牙配对还是 WiFi 认证，它们都在尝试解决同一个古老问题——在不可信的公共媒介上，如何让两个陌生设备建立一个只有它们自己知道的秘密？本章从第一性原理出发，归纳出三条核心公理和一个统一设计框架。

公理一：任何安全协议都必须解决"信任引导"问题

在密码学中，密钥交换（如 ECDH）仅能在数学上保证两方推导出相同的秘密，但它本身无法区分合法对端和中间人。三个协议各自用不同的方式绑定了密钥交换的结果与"身份"：

🔵

TLS 1.3

用 CA 签名的 X.509 证书绑定服务端公钥与域名。信任来自于社会性机构（证书颁发机构）的背书，是一种机构信任模型。

🟣

蓝牙配对

用用户的眼睛和双手（确认匹配的 6 位数字）来对抗中间人。信任来自于用户的物理在场和感知，是一种人类感知信任模型。

🟢

WiFi SAE

用双方共同知晓的秘密（密码）作为信任锚，SAE 确保双方无需传输密码本身即可证明知识。是一种共享秘密信任模型。

公理三：前向安全是现代协议的最低要求

三个协议的安全演进都遵循同一条主线：从静态长期密钥 → 临时密钥（Ephemeral Key）。

阶段TLS蓝牙WiFi
早期/不安全 静态 RSA 密钥传输（TLS 1.2 前） PIN 配对（v1.x） WEP 静态密钥
过渡 DHE（可选，TLS 1.2） SSP P-192（v2.1） WPA2-PSK（静态 PMK）
现代/安全 强制 ECDHE（TLS 1.3） 强制 P-256（LE SC v4.2） SAE 临时 PMK（WPA3）

统一设计框架：任何无线认证协议 = 以下四步

A

能力协商（Capability Negotiation）

双方宣告自己支持的算法套件、安全级别、IO 能力。TLS → ClientHello/ServerHello；蓝牙 → Pairing Request/Response；WiFi → RSN IE in Beacon/Association。

B

临时密钥协商（Ephemeral Key Agreement）

基于 DH 或其变体，双方交换公钥（或标量/元素），独立推导出相同的共享秘密（DHE 共享值 / DHKey / Dragonfly K）。此步骤保证前向安全的数学基础。

C

身份绑定（Identity Binding）

将共享秘密与"身份"绑定，防止中间人。TLS → 服务端证书签名握手记录；蓝牙 → 用户确认承诺值；WiFi SAE → Confirm 值互相校验（Dragonfly 的 Commit + Confirm）。

D

密钥推导与安装（Key Derivation & Installation）

用伪随机函数（HKDF / AES-CMAC / PRF）从共享秘密推导出分层的会话密钥（握手密钥 → 应用密钥），分别用于认证（KCK/MacKey）、密钥加密（KEK）和数据加密（TK/LTK）。最后双方相互发送 Finished/MIC/DHKey-Check 来证明密钥推导的一致性。

一句话总结（第一性原理）：
所有无线认证协议的本质都是：用一个可信的先验秘密（CA 证书 / 物理在场 / 预共享密码）来绑定一次临时的 Diffie-Hellman 密钥交换，使得只有合法的双方能推导出相同的会话密钥，而任何窃听者或中间人——即便截获了所有空中数据包——也无法重建该密钥。前向安全则进一步保证：即使未来的先验秘密泄露，也无法追溯解密过去的会话。
